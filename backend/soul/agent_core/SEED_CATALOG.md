# Soul Seed Catalog

这个文件只给人看，不进入运行时 prompt。

`agent_core/seeds/` 里的文件是候选 seed。
运行时真正会进入模型上下文的，只有 `agent_core/ACTIVE_SEED.md`。
模型不会同时读到多份 seed，也不会知道候选 seed 目录里还有哪些其它版本。

## 使用方式

切换当前灵魂设定的方法很简单：

1. 从 `agent_core/seeds/` 中挑一份 seed
2. 把它的正文原样复制到 `agent_core/ACTIVE_SEED.md`
3. 下次装配 prompt 时，这份 `ACTIVE_SEED.md` 就会作为当前灵魂设定进入模型

## 运行时装配关系

对模型来说，静态 prompt 的顺序是：

1. `ACTIVE_SEED.md`
2. `CORE.md`

也就是说：

- `ACTIVE_SEED.md` 先告诉模型“你现在是什么样子，平时怎么说，怎么做”
- `CORE.md` 再补充“无论当前风格是什么，都必须守住哪些共同契约”，包括事实底线、输出边界、暴露限制、稳定协作偏好和项目口径

## Seed Contract

- seed 只负责灵魂设定，不负责系统底线。
- seed 只定义当前身份、语言风格、工作习惯、语言组织方式和特定约束。
- seed 不定义事实边界、输出底线、暴露限制，也不改写工具能力或系统边界。
- `ACTIVE_SEED.md` 必须和当前选中的 seed 文件保持正文一致，不要另外手写一版。
- “哪些任务更适合哪个 seed” 这类说明只写在文档里，不写进运行时 seed 正文。

## Core / Seed 分工

### `ACTIVE_SEED.md`

负责当前这一次真正给模型看的设定。它回答的是：

- 你现在是什么样的人
- 你习惯怎么说
- 你习惯怎么处理问题
- 你最容易往哪里偏，所以要特别防什么

### `CORE.md`

负责接在 seed 后面，补充所有风格都不能越过的底线。它回答的是：

- 你怎么处理事实
- 你怎么控制输出
- 你什么时候该收边界
- 你哪些内部内容不能往外露
- 这个用户或项目有哪些长期稳定口径

## 固定模板

每个 seed 正文固定使用下面五个模块，顺序不要改：

1. `身份锚点`
2. `语言风格`
3. `工作习惯`
4. `语言组织方式`
5. `特定约束`

## 适合写进 Seed 的内容

- 当前身份怎么理解自己
- 说话的语气、温度、称呼和节奏
- 默认怎么看问题、怎么推进问题
- 回答时先讲什么、后讲什么
- 这个风格最容易出现什么偏差

## 不适合写进 Seed 的内容

- 所有情况都必须遵守的系统底线
- 事实标准
- 暴露限制
- 工具能力差别
- “只有这类任务才能用这个灵魂”这类硬路由暗示

## Seed Matrix

### 句芒

- 关键词：对话、统筹、分派、归口
- 默认手法：先确认目标，再判断是否需要分工，需要时把多方产物收束成统一答案
- 风险：容易把统筹误当成权限上级，或者在简单任务里过度分派

### 河伯

- 关键词：澄清、收束、稳判断
- 默认手法：先去噪，再抓主问题，再给当前最稳妥的判断
- 风险：容易过于克制，最后把判断说轻了

### 四岳

- 关键词：结构、层次、定框架
- 默认手法：先看整体，再定边界，再拆步骤
- 风险：容易讲得太大，把简单问题说复杂了

### 祝融

- 关键词：推进、执行、破局
- 默认手法：先找卡点，再给最有效的一步，再继续推进
- 风险：容易太快，压缩掉必要验证

### 玄女

- 关键词：前提、歧义、隐含条件
- 默认手法：先把没说清的部分拆开，再收回到结论
- 风险：容易分析过细，让主结论变弱

## Quick Recommendation

- 不设置默认主灵魂
  - 当前对话灵魂由用户切换或任务系统绑定决定
- 如果当前任务更偏信息收集、证据整理、上下文召回：选河伯
- 如果当前任务更偏系统规划或多层结构整理：选四岳
- 如果当前任务更偏修复、推进、执行突破：选祝融
- 如果当前任务更偏复杂理解、歧义辨析、语义拆解：选玄女

# 投影系统架构审查报告

日期：2026-06-14

审查范围：langchain-agent 项目的投影系统

---

## 一、架构概述

### 1.1 系统组成

投影系统负责将 agent 内部状态、模型输出、工具结果和运行时协议转换为前端可消费的、安全的公开显示信号。系统分为三个主要层次：

| 层次 | 文件 | 职责 |
|------|------|------|
| 能力目录投影 | `backend/capability_system/catalog_projection.py` (750行) | 将工具目录和能力定义投影为前端可展示的工具卡片、风险标签、分组信息 |
| 证据投影 | `backend/evidence/projection.py` (268行) | 将工具观察、文件证据、绑定状态等运行时证据投影为结构化显示帧 |
| 操作投影 | `backend/harness/runtime/operation_projection.py` (126行) | 抽象权限模式、操作鉴权结果到前端可理解的公开状态 |
| 前端投影 | `frontend/src/components/chat/agentRunProjection.ts` (14行) | 客户端侧投影守卫，过滤内部控制协议文本 |
| 合同测试 | `backend/tests/public_projection_contract_test.py` (669行) | 验证公开投影帧的字段完整性、权限过滤和数据格式 |

### 1.2 数据流

```
模型输出 → 动作解析(JSON) → 动作准入(action_permit) 
    → 工具执行 → 工具观察(op observation) 
    → 证据投影/操作投影 → 公开投影帧(public_projection_frames)
    → chatrun:* ledger(持久化) → 前端SSE流 → agentRunProjection过滤 → UI渲染
```

关键路由：
- `_project_public_stream_event` (api/chat.py) 是主聊天 SSE 事件的公共投影入口。
- 公开投影帧包含 `attachment` 描述其目标显示表面（如 `display_state/main_chat_surface`）。
- 工具调用结果经过 `catalog_projection` 获取工具元数据（名称、描述、风险等级）后合并为显示帧。
- 非任务普通对话不进入任务 timeline（见 137 号文档修订）。

### 1.3 设计意图

- 隔离内部实现与用户可见内容：内部协议字段（如 `model_action_request_json`）不会直接暴露给用户。
- 统一公开显示通路：所有前端渲染信号必须通过公开投影帧，接受 `PUBLIC_PROJECTION_AUTHORITY` 校验。
- 支持多种显示表面：主聊天表面、任务状态面板、工具调用窗口、编辑器上下文等。

## 二、问题清单

### 问题类别 A：投影覆盖缺口

**A1** — `operation_projection.py` 投影层次过薄，缺少操作入场的完整投影。
- 文件：`backend/harness/runtime/operation_projection.py` L1-126
- 仅处理权限模式标准化和简单映射，未覆盖模型动作解析失败、动作解析重试、工具批处理分组等关键运行状态的投影。
- 影响：`single_agent_turn.py` 中的多轮工具交互状态可能没有对应的公开帧描述，前端无法区分“等待工具执行”与“动作解析失败”。
- 证据：`public_projection_contract_test.py` 测试了大量帧字段，但未见到针对 `action_parse_failure`、`batch_group_started` 等内部状态的投影测试。

**A2** — 前端 `agentRunProjection.ts` 仅做内部协议文本过滤，未参与结构化投影逻辑。
- 文件：`frontend/src/components/chat/agentRunProjection.ts` L1-14
- 仅导出 `looksLikeRawToolOutput` 函数，依赖 `isInternalControlProtocolText`。
- 风险：若后端投影帧新增字段（如 `tool_call_id` 去重键），前端没有对应的归并逻辑，可能导致 UI 显示重复（与 131 号文档记录的投影重复根因相关）。
- 实际去重逻辑在 138 号执行记录中已收束到后端 ledger，但前端缺少防御性投影检查。

**A3** — `evidence/projection.py` 仅处理文件证据绑定，未覆盖其他运行时证据类型（如会话状态、记忆检索结果）。
- 文件：`backend/evidence/projection.py` L1-268
- 核心逻辑集中在 `context.active_binding_identity` 和 `followup_binding_key`，但测试中未验证记忆、项目会话绑定等证据的投影。
- 影响：后续新增证据类型可能需要手动添加投影逻辑，缺少统一抽象。

### 问题类别 B：一致性与规范化缺陷

**B1** — 投影权威来源分散在多处，存在不一致风险。
- 依据：135 号计划书（L1-240）和 137 号修复方案（L1-240）明确指出了投影权威来源的问题：legder、公开帧、timeline 之间的职责长期混淆。
- 修复后已将主聊天投影收束为 `chatrun:*` ledger（138 号执行记录 L9-60），但 `catalog_projection.py` 仍是独立函数被多处调用，并未强制统一入口。
- 证据：`_project_public_stream_event` 在 `api/chat.py` 中调用多种投影函数，缺少统一的 `projection_context` 管理，可能导致不同调用路径产生不同格  式的帧。

**B2** — 工具风险等级评估逻辑硬编码，缺少可配置机制。
- 文件：`backend/capability_system/catalog_projection.py` L241-243
```python
if "shell" in tags or not bool(tool.get("is_read_only", True)):
    return "高"
```
- 仅基于标签和只读标记判断风险，未考虑工具的实际资源暴露策略（如 `resource_exposure_policy`）。
- 风险：新工具注册时可能被错误分类，或高风险工具被误标为中低风险。

**B3** — 公开投影帧的 `attachment` 字段在不同投影函数中定义方式不一致。
- `catalog_projection.py` 中 `build_capability_projection` 函数生成的帧 attachment 结构与 `evidence/projection.py` 中不同。
- 测试（`public_projection_contract_test.py` L1-669）虽然覆盖了帧的基本结构，但未跨投影源做一致性校验。

### 问题类别 C：测试覆盖不足

**C1** — 测试仅覆盖公开投影合同的最小字段集，缺少跨投影源的集成测试。
- 文件：`backend/tests/public_projection_contract_test.py` L1-669
- 主要测试帧的字段存在性和类型，但未模拟完整对话流程中的多个投影源并发输出。
- 缺少对投影帧去重机制的验证（131 号文档记录的重复问题修复后未添加回归测试）。

**C2** — 前端 `agentRunProjection` 无单元测试。
- 前端投影逻辑虽然简短，但 `isInternalControlProtocolText` 依赖于一个外部模块，没有测试保证其过滤规则与后端投影规则同步更新。

## 三、风险评估

| 风险 | 严重度 | 可能性 | 说明 |
|------|--------|--------|------|
| 投影帧重复导致 UI 显示混乱 | 中 | 低 | 138 号修复已将权威收束，但缺少回归测试，未来修改可能复现 |
| 新证据/操作类型未正确投影 | 中 | 中 | 投影函数各自独立，新增类型需手动修改多处，容易遗漏 |
| 工具风险标签错误 | 低 | 低 | 当前硬编码逻辑对现有工具集适用，但扩展性差 |
| 前后端投影规则不同步 | 中 | 低 | 前端仅做简单文本过滤，复杂去重依赖后端；若后端修改框架，前端可能未更新 |
| 公开帧中泄露内部信息 | 高 | 低 | `public_projection_contract_test.py` 测试了权限过滤，但未穷举所有字段；开发者可能无意中在新帧中包含内部路径等敏感信息 |

---

## 四、改进建议

1. **统一投影上下文管理**：引入 `ProjectionContext` 对象，封装所有投影函数，确保帧格式、attachment 结构和权威来源统一。
2. **扩展操作投影**：在 `operation_projection.py` 中增加模型动作解析状态、工具批处理生命周期等投影，提升前端对 agent 运行状态的可见性。
3. **工具风险可配置化**：将 `catalog_projection.py` 中的风险判定逻辑改为基于 `resource_exposure_policy` 和用户自定义规则的映射表。
4. **添加投影去重回归测试**：基于 131 号文档记录的重复场景，编写端到端测试验证 `chatrun:*` ledger 投影路线的去重行为。
5. **前端投影加固**：在 `agentRunProjection.ts` 中增加结构化帧的去重逻辑作为防御层，即使后端未正确去重也不会影响 UI。
6. **证据投影扩展**：抽象证据投影接口，使新增证据类型（记忆、项目绑定等）可插件式注册投影规则。
7. **文档与代码同步审查**：129-138 号系列文档已完整记录投影演化历史，建议每次投影模块改动后更新相关文档，保持架构知识新鲜度。

---

## 五、总结

投影系统经过多轮修复已趋于稳定，尤其是 ledger 权威收束修复（138 号执行记录）解决了长期存在的投影重复问题。当前主要风险集中于：投影函数分散、测试覆盖有限、前端防御薄弱。

建议按优先级推进：先补齐测试（C1/C2），再统一投影上下文（B1），最后扩展新证据类型投影（A3）。

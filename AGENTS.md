# AGENTS.md instructions for D:\AI应用\langchain-agent

## 固定本地节点

- 本项目本地运行节点固定为：
  - 前端 Next.js：`http://127.0.0.1:3000`
  - 后端 FastAPI / Uvicorn：`http://127.0.0.1:8003`
  - 前端 API Base：`http://127.0.0.1:8003/api`
- 除非用户明确要求，不允许临时切换到 `3001`、`3002`、`8002`、`8004`、`8007` 或其他随机端口。
- 如果固定端口被占用，必须先查明占用进程是否属于本项目。若属于本项目旧进程，可以关闭后重启；若不是本项目进程，必须告知用户并请求处理，不得擅自关闭无关进程。
- 前端掉线、请求失败、SSE 断开、监控断开时，第一检查项必须是：
  - `3000` 是否只有一个前端进程监听。
  - `8003` 是否只有一个后端进程监听。
  - 前端实际请求目标是否为 `http://127.0.0.1:8003/api`。
- Electron、浏览器直开、Next 代理和前端配置必须统一指向上述固定节点；不要同时维护多套默认端口。
- 前端开发服务必须从干净的 `.next` 启动。遇到 `Cannot find module ./.next/server/...`、旧导入残留、页面热更新异常时，不要换端口，先关闭项目旧前端进程并用固定 `3000` 重新启动。
- 涉及启动、运行链路、前后端联调、SSE、监控、Electron 或页面可用性的修改，必须使用 CLI 真实启动前端和后端进行实测。不能只凭静态检查、代码阅读或假设判断启动成功。

## 项目协作规则
- 修改代码，增加代码，应该尽量以优质代码落地，增强链路稳定，而少用补丁式兜底。
- 修改前必须先阅读相关代码、调用链、测试和项目约定。优先遵循现有架构、命名、错误处理、类型系统和测试方式，不要凭空新建一套风格。
- 对用户提出的功能要求，必须以真实工程可用为标准。不能用“差不多可用”的方案替代成熟方案；这意味着，当你提出方案时，你必须先想一想是否有更适合，更稳健，更先进，更具备可拓展性的方案，并主动提出，当涉及 agent 架构、runtime、workflow、tool calling、state、memory、error recovery 等核心问题时，必须以成熟 coding agent 的外显工程能力为参照，包括明确任务规划、工具调用边界、可追踪执行状态、失败恢复、权限控制、上下文管理、可验证输出和测试闭环。
- 当涉及不稳定信息、外部标准、第三方 API、模型能力、框架最佳实践，或无法确认成熟方案时，必须查阅官方文档、权威论文或可信工程资料。若问题完全可由本项目代码、调用链和项目约定判断，则优先阅读本项目，不要为了装饰性引用而搜索。

## 精简优先与成熟架构边界
- 每次新增功能、抽象、依赖、配置、状态、协议字段或兜底链路前，先判断它是否真实服务当前需求；假想未来、展示式复杂度、为了“看起来完整”而增加的结构一律不要做。
- 优先使用标准库、框架原生能力、平台原生能力和项目内已经成熟的主链路。只有当现有能力无法满足真实需求，并且新增方案的收益大于维护成本时，才允许新增依赖、抽象或链路。
- 精简优先不是追求最短代码。涉及 runtime、workflow、agent prompt、tool calling、state、memory、权限、失败恢复、上下文管理、前后端协议等核心链路时，标准是职责清晰、边界稳定、状态可追踪、失败可恢复、输出可验证，而不是一行代码或最少文件。
- 对普通局部问题，采用最小正确修改；对重复出现的问题、隐式状态、绕多个分支的补丁、旧逻辑干扰、同一职责多处实现，必须升级为结构性修复，优先收敛主链路。
- 删除优先于新增。默认清理无用旧代码、废弃分支、重复链路、调试残留和过时测试文件；不允许以兼容、兜底、保险为理由保留已经被新主链路替代的旧逻辑。
- 测试必须验证真实行为。不要为很小的改动机械增加噪声测试；不要维护只服务旧逻辑的测试。遇到测试仍在约束废弃链路时，应删除或重写为验证新主链路的测试，而不是迁就旧行为。
- 以下情况必须先写计划书并等待用户确认：跨 3 个以上核心模块；改动 runtime / workflow / agent prompt / tool calling / state / memory / API contract；删除或替换现有主链路；影响数据库结构、任务执行语义或前后端协议；重写测试体系或目录结构。
- 计划通过后必须按计划完整实施。若实施过程中发现计划假设错误、风险显著扩大、测试暴露架构问题或需要改变目标设计，必须暂停并说明偏差，再请求确认。

- 前端设计时，根据架构分层，不同层级用卡片按钮做页面切换，不要将不同层级的页面混在一页。不要把开发式的语句当作副标题。

## Agent Prompt 规则
  系统框架的原则是agent负责调度和控制，除了显示的用户契约外，系统只能接收agent的调度式执行契约，包括提供资源，边缘控制。禁止设计越权链路，让系统反过来阻碍agent的运行。
动范围和契约，然后系统为agent的契约做对接，然后才是兜底机制，
- 时刻记住本项目是在做 agent。给 agent 的 prompt 必须让 agent 理解自己的角色、职责、边界、输入、输出、裁决标准和失败处理，不要把开发说明当作 prompt 发给 agent。
-  在设计和维护的时候首先检查prompts，prompts是否准确，是否要求他给出合理的判断，规定好他的行动范围和契约，然后系统为agent的契约做对接，然后才是兜底机制，  
- 写给 agent 的 prompt 必须能被目标 agent 直接执行。它应描述身份、任务、输入、输出、禁止事项、判断标准和失败处理，而不是描述该节点在代码里的技术用途。
- 不要写成：

```text
这是 runtime 节点。
根据任务图执行 world_review。
这个节点用于校验资产。
```

- 应该写成：

```text
你是一名世界观审核员。
你只负责评审当前世界观设定是否完整、一致、可支撑后续写作。
你不负责替创作者扩写设定。
你需要指出问题、给出裁决、说明是否允许进入下一阶段。
```

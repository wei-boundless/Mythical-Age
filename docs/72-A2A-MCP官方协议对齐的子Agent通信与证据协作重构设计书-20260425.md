## 0. 本版结论

本项目后续不再自造“一套同时负责 agent、tool、state、handoff、memory 的万能协议”。

推荐采用三层分治：

```text
控制层：本地 QueryRuntime / Planner / EvidenceOrchestrator
  -> Agent-Agent 协议层：A2A
  -> Agent-Tool 协议层：MCP
  -> Worker 内部执行层：PDF/Retrieval/StructuredData 实现细节
```

核心判断：

1. `A2A` 负责子 agent 之间的通信、发现、长任务、流式返回与边界隔离。
2. `MCP` 负责子 agent 与工具/资源之间的契约，不让工具事实泄露到主线程。
3. `OpenAI Agents handoff` 与 `AutoGen` 只作为本地 runtime 交接语义参考，不作为跨系统正式协议。
4. 本项目独有的 `SourceObjectRef / ArtifactHandle / ResultHandle / SubsetHandle` 不单独发明主协议，而是作为 `A2A extension` 接入。
5. `QueryRuntime` 仍是控制面 owner；`Worker` 不直接越权写主线程真相。

一句话收口：

> A2A 负责 agent-agent，MCP 负责 agent-tool，本项目自定义的只有 handle 与 evidence 扩展层，而且必须挂在官方协议之上。

### 0.1 按 74 校准后的当前主线

截至 `74` 审查，当前代码已经有 `WorkerRequest / WorkerResult / EvidenceEnvelope / EvidenceArtifactGraph / WorkerProjection` 等骨架，但仍处在过渡态。因此本设计书后续解释顺序调整为：

```text
72/73 定主骨架
  -> 71 定 follow-up / handle / mixed-source 语义
  -> 70 定 evidence graph / worker handoff
  -> 68 定 tool contract / MCP-compatible adapter
```

本轮不急着拆远程服务，也不急着实现真实 HTTP/JSON-RPC/SSE。正确优先级是：

1. 先把本地 `WorkerRequest / WorkerResult` 升级成 A2A-compatible envelope。
2. 再把 follow-up 从 `owner task replay` 切到 `handle continuation`。
3. 再补齐 `ResultHandle / SubsetHandle` 的存储和恢复。
4. 再收口旧 RAG 分支、prompt 投影和 memory 边界。
5. 最后再做 MCP adapter 与远程化准备。

这条顺序的原因很直接：如果不先修 handle 主链，A2A 化只会把旧 owner 污染和旧 RAG 分支包装成“更标准的旧问题”。

## 1. 官方协议依据

### 1.1 A2A

官方定位：

- 用于独立 agent 之间的互操作。
- 核心对象包括 `Agent Card`、`Message`、`Task`。
- 基于 `HTTP(S) + JSON-RPC + SSE`。
- 明确支持异步长任务、流式事件、能力发现、边界隔离。

官方资料：

- A2A 规范主页：<https://google-a2a.github.io/A2A/specification/>
- A2A 核心概念：<https://google-a2a.github.io/A2A/topics/key-concepts/>
- A2A 与 MCP 的关系：<https://google-a2a.github.io/A2A/topics/a2a-and-mcp/>

本项目使用结论：

1. 未来的 `retrieval-agent / pdf-agent / structured-data-agent` 作为 A2A agent 暴露。
2. 当前本地 `WorkerRequest / WorkerResult` 先做成 A2A-compatible envelope。
3. 本地 in-process 调用与未来 remote service 调用使用同一语义模型。

### 1.2 MCP

官方定位：

- 用于模型与工具、资源、提示词之间的标准化连接。
- 关注 `tools / resources / prompts` 的暴露与调用契约。
- 不解决 agent-agent 协作。

官方资料：

- MCP 规范主页：<https://modelcontextprotocol.io/specification/>
- MCP 草案规范：<https://modelcontextprotocol.io/specification/draft>

本项目使用结论：

1. `pdf-agent` 内部的 parser、OCR、table extraction 能力走 MCP tool/resource。
2. `retrieval-agent` 内部的 vector search、rerank、document lookup 走 MCP。
3. `structured-data-agent` 内部的 schema inspect、aggregate、query 走 MCP。
4. `QueryRuntime` 不直接感知这些工具实现细节。

### 1.3 OpenAI Agents Handoff

官方定位：

- 用于多 agent 之间的 handoff、输入过滤、历史裁剪、上下文移交。
- 更像 runtime 语义和边界控制方案。

官方资料：

- Handoff 文档：<https://openai.github.io/openai-agents-python/handoffs/>
- Agents SDK 总览：<https://platform.openai.com/docs/guides/agents-sdk/>

本项目使用结论：

1. 借其“handoff 只传必要上下文”的原则。
2. 借其“交接不是历史全量复制”的边界设计。
3. 不把它当成 wire protocol。

### 1.4 AutoGen Message Model

官方定位：

- 提供单 runtime 内多 agent 的 direct messaging 与 topic/broadcast 语义。
- 更像本地 orchestration runtime 参考。

官方资料：

- 消息与通信：<https://microsoft.github.io/autogen/stable/user-guide/core-user-guide/framework/message-and-communication.html>

本项目使用结论：

1. 借其 direct message 与 publish/subscribe 的模型。
2. 用于本地 orchestrator 内部事件设计参考。
3. 不作为正式外部协议。

## 4. 推荐目标架构

## 4.1 分层架构

```text
User
  -> QueryRuntime / Planner / Followup / Session Memory
  -> Agent Orchestration Layer
      -> A2A Client Runtime
      -> A2A Agent Registry
      -> A2A Task Dispatcher
  -> Sub Agents
      -> retrieval-agent
      -> pdf-agent
      -> structured-data-agent
  -> MCP Tool Layer inside each agent
      -> retriever / reranker / parser / OCR / dataframe ops
  -> Canonical Result / Evidence Projection / Answer Finalizer
```

### 4.1.1 本项目中的 owner 关系

| 层级 | owner | 唯一职责 |
| --- | --- | --- |
| 控制层 | `QueryRuntime` | 决定当前轮该找谁、收谁的结果、怎么续接 |
| agent 协议层 | `A2A runtime adapter` | 标准化子 agent 的发现、调用、stream、task 跟踪 |
| tool 协议层 | `MCP adapter` | 标准化子 agent 内部工具/资源调用 |
| 执行层 | 各 worker/agent | 完成自身能力，不越权决定主线程真相 |
| 输出层 | `CanonicalResult + Finalizer` | 只消费稳定结果和 typed refs |

### 4.1.2 本地与远程统一

第一阶段：

- 子 agent 先是本地 in-process worker。
- 但交互 envelope 已按 A2A-compatible 设计。

第二阶段：

- `retrieval-agent / pdf-agent / structured-data-agent` 可拆成独立 A2A endpoint。
- `QueryRuntime` 切换为 A2A client，不改业务语义层。

## 5. 官方协议映射设计

## 5.1 A2A 在本项目中的角色

### 5.1.1 Agent Card 对应

A2A `Agent Card` 在本项目中应映射为：

```text
SubAgentCard
  agent_id
  title
  description
  capabilities
  accepted_inputs
  produced_outputs
  supported_handle_types
  handoff_policies
  mcp_profiles
```

最小实现建议：

- `retrieval-agent`
- `pdf-agent`
- `structured-data-agent`

每个 card 至少声明：

1. 能处理的 `task_kind`
2. 接受的 `SourceObjectRef / ArtifactHandle / ResultHandle`
3. 产出的 `CanonicalResult / Candidate / Handle`
4. 是否允许流式
5. 是否允许长任务

### 5.1.2 Message / Task 对应

本项目不直接把当前 `WorkerRequest` 当 A2A wire message，而是做如下映射：

| 本项目对象 | A2A 对应 |
| --- | --- |
| `WorkerRequest` | A2A `message/send` 请求体中的 task input |
| `WorkerExecutionPlan` | A2A task dispatch descriptor |
| `WorkerResult` | A2A task result / streamed events |
| `CanonicalResult` | A2A 最终 agent result payload |

### 5.1.3 A2A extension 命名空间

本项目自定义协议部分统一挂在：

```text
x-langchain-agent.*
```

不得使用：

- 与 A2A 主字段重名的自定义字段
- 没有命名空间前缀的私有字段

建议扩展字段：

- `x-langchain-agent.task_frame`
- `x-langchain-agent.object_refs`
- `x-langchain-agent.artifact_refs`
- `x-langchain-agent.result_refs`
- `x-langchain-agent.subset_refs`
- `x-langchain-agent.arbitration_reason`
- `x-langchain-agent.presentation_hints`

## 5.2 MCP 在本项目中的角色

### 5.2.1 MCP 只进 agent 内部

硬约束：

1. 主线程 runtime 不直接暴露高噪声 MCP tools 给模型。
2. 子 agent 通过 MCP 访问自身工具与资源。
3. `QueryRuntime` 只感知子 agent 能力，不感知其内部工具树。

### 5.2.2 当前 `tools/contracts.py` 的演进方向

当前 [backend/tools/contracts.py](/D:/AI应用/langchain-agent/backend/tools/contracts.py) 可以继续保留，但语义上要明确：

| 当前对象 | 目标定位 |
| --- | --- |
| `ToolExecutionContract` | MCP 风格 execution contract |
| `ToolResolutionContract` | MCP 风格 resource/input resolution policy |
| `ToolOutputContract` | MCP tool output policy |
| `ToolProjectionContract` | 本项目运行时 projection contract |

也就是说：

- 本项目工具契约层不是与 MCP 对立，而是 MCP-compatible internal abstraction。
- 以后若需要真正对接外部 MCP server，现有 contract 可作为 adapter 层输入。

## 6. 本项目自定义扩展协议

## 6.1 为什么必须有扩展层

官方协议没有直接定义本项目特有的这些对象：

- `SourceObjectRef`
- `ArtifactHandle`
- `ResultHandle`
- `SubsetHandle`
- `BindingCandidate`

但这些正是长场景 follow-up 和污染隔离的核心。

因此必须有扩展层，且只能作为：

- A2A extension
- MCP resource/tool metadata extension

不能变成另一套主协议。

## 6.2 核心扩展对象

### 6.2.1 TaskFrame

```python
TaskFrame:
    intent
    source_kind
    task_kind
    modality
    route_hint
    capability_requests
    anchors
    constraints
```

说明：

1. `TaskFrame` 是控制层对象，不直接暴露给模型。
2. 作为 `x-langchain-agent.task_frame` 进入 A2A 消息。

### 6.2.2 SourceObjectRef

表示原始对象：

- 本地 PDF
- 本地 xlsx/csv
- 知识库文档对象
- workspace 文件

最小字段：

```python
SourceObjectRef:
    object_id
    object_type
    uri
    locator
    metadata
```

### 6.2.3 ArtifactHandle

表示从原始对象派生出的中间可处理对象：

- `pdf_page`
- `pdf_section_locator`
- `pdf_table`
- `dataset_schema`
- `retrieval_chunk`

最小字段：

```python
ArtifactHandle:
    artifact_id
    artifact_type
    source_object_id
    content_ref
    metadata
    visibility
```

### 6.2.4 ResultHandle

表示某次执行形成的稳定结果：

- `pdf_summary`
- `pdf_section_summary`
- `structured_subset`
- `grouped_result`
- `mixed_synthesis`

最小字段：

```python
ResultHandle:
    result_id
    result_kind
    owner_task_id
    source_object_ids
    artifact_ids
    metadata
```

### 6.2.5 SubsetHandle

表示结构化结果或表格结果的稳定子集：

- top5 rows
- grouped subset
- filtered rows

最小字段：

```python
SubsetHandle:
    subset_id
    subset_kind
    owner_result_id
    dataset_ref
    filter_signature
    metadata
```

### 6.2.6 ClarificationCandidate

表示“可确认但未提交 binding”的候选对象：

- `DocumentCandidate`
- `DatasetCandidate`
- `TableCandidate`

约束：

1. 候选永远不是 binding。
2. 只能由 orchestrator 提交 binding。

## 7. 子 agent 通信关系设计

## 7.1 角色定义

### retrieval-agent

职责：

1. 发现相关对象和候选。
2. 返回证据与 candidate。
3. 不提交 binding。
4. 不直接替代 PDF/结构化分析。

输入：

- `TaskFrame`
- 可选 `object/result/subset refs`

输出：

- `EvidenceEnvelope`
- `DocumentCandidate / DatasetCandidate / TableCandidate`
- 必要时 `CanonicalResult`

### pdf-agent

职责：

1. 文档级阅读。
2. 页/节定位。
3. 表格抽取。
4. 产出文档类结果和表格 artifact。

输入：

- `SourceObjectRef(pdf)`
- `ArtifactHandle(pdf_page/pdf_section_locator)`
- `ResultHandle(pdf_summary/pdf_section_summary)`

输出：

- `CanonicalResult`
- `ArtifactHandle(pdf_page/pdf_table)`
- `ResultHandle(pdf_summary/pdf_section_summary)`

### structured-data-agent

职责：

1. 分析 dataset 或 table artifact。
2. 生成 subset/grouped result。
3. 不消费 prose。

输入：

- `SourceObjectRef(dataset)`
- `ArtifactHandle(table_object/pdf_table)`
- `ResultHandle(structured_result)`
- `SubsetHandle`

输出：

- `CanonicalResult`
- `ResultHandle(structured_subset/grouped_result)`
- `SubsetHandle`

### orchestrator

职责：

1. 选择 agent。
2. 管理 candidate -> binding -> execution。
3. 管理 mixed-source arbitration。
4. 管理 result/subset continuation。

硬约束：

1. 子 agent 不能自己决定下一个子 agent。
2. 子 agent 不能直接写 session 主真相。

## 8. 按问题与场景定义理想链路

## 8.1 显式 PDF

```text
User -> TaskFrame(explicit_pdf)
  -> orchestrator
  -> pdf-agent
  -> CanonicalResult + ResultHandle
  -> projection
```

原则：

1. 不先走 retrieval。
2. follow-up 优先续接 `ResultHandle`。

## 8.2 知识库 PDF 发现后深读

```text
User -> TaskFrame(corpus_lookup)
  -> retrieval-agent
  -> DocumentCandidate / SourceObjectRef(pdf)
  -> orchestrator
      -> 若证据足够，直接 finalize
      -> 若需要页/节深读，转 pdf-agent
```

原则：

1. retrieval 发现对象。
2. pdf-agent 解释对象。

## 8.3 显式 dataset

```text
User -> TaskFrame(explicit_dataset)
  -> orchestrator
  -> structured-data-agent
  -> ResultHandle / SubsetHandle
```

## 8.4 “数据库里哪些城市缺货”

```text
User -> TaskFrame(local_data_lookup)
  -> retrieval-agent
  -> DatasetCandidate
  -> orchestrator commit binding
  -> structured-data-agent
  -> structured result
```

原则：

1. retrieval 只能提供 dataset candidate。
2. 不允许 retrieval 文本直接冒充表格结论。

## 8.5 “PDF 里的表格按地区汇总”

```text
User -> TaskFrame(pdf_table_analysis)
  -> pdf-agent
  -> TableArtifact
  -> orchestrator
  -> structured-data-agent
  -> structured result
```

原则：

1. `pdf_table` 先是 artifact，不是 dataset path。
2. 是否升级为结构化分析由 orchestrator 决定。

## 8.6 mixed-source synthesis

```text
User -> TaskFrame(mixed_source_synthesis)
  -> orchestrator
  -> gather existing ResultHandles
      -> missing retrieval side: call retrieval-agent
      -> missing pdf side: call pdf-agent
  -> finalizer
  -> CanonicalResult(mixed_synthesis)
```

原则：

1. 先仲裁，再调用单源 agent。
2. 不允许 authority context 直接抢路由。

## 10. 扩展性设计

## 10.1 新增子 agent 的接入规则

以后新增任何子 agent，例如：

- `code-agent`
- `memory-agent`
- `web-agent`
- `multimodal-agent`

只允许修改：

1. agent card 注册
2. orchestrator dispatch rule
3. 对应 result/object/subset extension schema
4. 测试

不允许修改：

1. 主线程 prompt 协议
2. follow-up 主语义
3. session memory 核心结构

## 10.2 新增工具的接入规则

以后新增 agent 内部工具：

1. 优先按 MCP-compatible contract 接入。
2. 不暴露给主线程。
3. 不让主线程产生新的工具名特判。

## 10.3 本地到远程的切换规则

本设计要求：

1. 本地 worker 与远程 agent 使用同一业务 envelope。
2. transport 可以换，业务协议不能换。

也就是说：

- 第一阶段：Python 对象直连，但 envelope 已经 A2A-compatible。
- 第二阶段：本地/远程 transport adapter 并存，业务字段不变。
- 第三阶段：必要时切到 HTTP/JSON-RPC/SSE A2A。

切换点只在 transport adapter，不在业务层。

远程化前置条件：

1. handle continuation 已经替代 owner replay。
2. `ResultHandle / SubsetHandle` 已经能从 runtime state 恢复。
3. 旧 `route == "rag" -> retrieval_service.retrieve()` 主分支已经收口。
4. prompt 与 memory 投影已经确认不会携带 raw evidence。

## 11. 记忆系统对接设计

## 11.1 本项目记忆层必须分四层

结合当前代码与既有记忆原则，A2A/MCP 对接后，记忆系统必须严格拆成下面四层：

1. `Runtime Evidence State`
2. `Session Memory`
3. `Durable Memory`
4. `Static Profile`

对应当前项目：

- `Runtime Evidence State`
  - [backend/runtime/session_store.py](/D:/AI应用/langchain-agent/backend/runtime/session_store.py)
  - [backend/query/evidence_store.py](/D:/AI应用/langchain-agent/backend/query/evidence_store.py)
  - [backend/query/evidence_graph.py](/D:/AI应用/langchain-agent/backend/query/evidence_graph.py)

- `Session Memory`
  - [backend/structured_memory/session_memory.py](/D:/AI应用/langchain-agent/backend/structured_memory/session_memory.py)

- `Durable Memory`
  - [docs/29-长期记忆重构方案.md](/D:/AI应用/langchain-agent/docs/29-长期记忆重构方案.md)
  - [docs/设计原则/23-Memory系统.md](/D:/AI应用/langchain-agent/docs/设计原则/23-Memory系统.md)

- `Static Profile`
  - `context_profile/*`

硬约束：

1. 子 agent 不直接写任意记忆层。
2. 所有记忆写入都必须经过 `QueryRuntime -> Projection -> Memory Policy`。
3. 记忆层永远不是 agent 间通信总线。

## 11.2 Runtime Evidence State 的定位

这是 A2A 子 agent 结果的首要承接层，作用域是当前 session runtime。

允许保存：

- `SourceObjectRef`
- `ArtifactHandle`
- `ResultHandle`
- `SubsetHandle`
- `BindingCandidate`
- `EvidenceArtifactGraph`
- unresolved clarification candidates

禁止保存：

- prompt prose
- durable note 正文
- raw assistant history 的拼接真相

设计要求：

1. 这层由 runtime/store 管理，不进入模型 prompt。
2. 这是 follow-up、binding arbitration、A2A task continuation 的第一恢复源。
3. 这层比 `session memory prose` 优先级更高。

与 A2A 的关系：

- 子 agent 返回的 `object/result/subset refs` 先落到 `runtime_state`
- 不直接进入 `session-memory summary.md`

## 11.3 Session Memory 的定位

`Session Memory` 是控制面工作记忆，不是执行面数据面。

允许保存：

- `active_goal`
- `active_object_handle_id`
- `active_result_handle_id`
- `active_subset_handle_id`
- `followup_binding_owner_task_id`
- `task_summary_refs`
- minimal correction / worklog / next step

禁止保存：

- raw retrieval snippets
- raw pdf page text
- raw table rows
- candidate prose 原文
- A2A agent 内部中间态

设计要求：

1. `SessionMemoryManager` 只消费结构化投影结果。
2. `committed_pdf_owner_task_id / committed_dataset_owner_task_id` 只能作为弱索引，不再作为当前轮主裁决源。
3. `session memory` 只负责恢复当前工作，不负责长期记忆决策。

与 A2A 的关系：

- A2A agent 不直接写 `session memory`
- 只能通过 `WorkerProjectionAdapter -> MainContext projection` 间接写入

## 11.4 Durable Memory 的定位

`Durable Memory` 不属于 agent 协议层，而属于 agent 执行后的治理层。

正确链路：

```text
A2A sub-agent result
  -> CanonicalResult
  -> WorkerProjection
  -> MainContext / TaskSummary projection
  -> Durable Candidate Extractor
  -> Durable Write Policy
  -> Durable Store
```

错误链路：

```text
A2A sub-agent result
  -> 直接写 durable note
```

硬约束：

1. 子 agent 不能直接写 durable memory。
2. `Durable Candidate` 只能从稳定 projection 提取，不从 raw transcript/evidence 直接提取。
3. `page dump / retrieval chunk / degraded diagnostics` 不能直接变成 durable note。

这与 [29-长期记忆重构方案.md](/D:/AI应用/langchain-agent/docs/29-长期记忆重构方案.md) 的 `summary-first, policy-gated` 完全一致。

## 11.5 Static Profile 的定位

`Static Profile` 保持现状：

- 由人维护
- 不接子 agent 自动写入
- 作为长期稳定规则源

不与 A2A / MCP 混层。

## 11.6 A2A extension 中允许出现的记忆字段

允许：

- `x-langchain-agent.memory_projection_hints`
- `x-langchain-agent.object_refs`
- `x-langchain-agent.result_refs`
- `x-langchain-agent.subset_refs`

其中 `memory_projection_hints` 只能是最小控制面，例如：

- `should_project_session`
- `projection_kind`
- `durable_candidate`
- `contains_runtime_noise`

禁止：

- `raw_memory_block`
- `session_summary_text`
- `durable_note_body`
- `retrieval_snippets`
- `page_text_dump`

原则：

> A2A 只传“是否允许投影”的 hint，不传“记忆正文”。

## 11.7 对未来 memory-agent 的约束

未来即使引入 `memory-agent`，其定位也只能是：

- `memory-curation-agent`

允许职责：

1. 读取 session projection
2. 生成 durable candidates
3. 给出 merge/supersede/deprecate 建议
4. 给出 relevant memory selection 建议

禁止职责：

1. 直接写 session memory
2. 直接写 durable store
3. 直接注入主 prompt
4. 直接成为 follow-up 主语义 owner

也就是说：

> memory-agent 只能做治理建议，不能做记忆主控 owner。

## 11.8 当前模块对接要求

### worker_projection.py

必须新增或稳定区分两类投影：

1. `session_memory_projection`
2. `durable_candidate_projection`

不允许把这两类语义混在一个 summary 字段里。

### session_memory.py

必须新增或稳定支持：

- `active_object_handle_id`
- `active_result_handle_id`
- `active_subset_handle_id`

并降低 `owner_task_id` 的语义权重。

### session_store.py

继续作为：

- `runtime evidence state` 的持久化容器

要求：

1. `runtime_state` 优先保存结构化 handle/graph/candidate
2. 不把这些东西退化成 message history

### durable memory facade

继续作为 durable write/read 唯一入口。

要求新增 reject 规则：

- `result_kind=pdf_page`
- `artifact_type=retrieval_chunk`
- `degraded_reason_typed=*`
- `candidate_clarification`

这些类型默认不准进入 durable memory。

## 11.9 一句话边界

记忆系统在本架构中的固定关系是：

```text
A2A 子 agent
  -> runtime evidence state
  -> session memory projection
  -> durable memory governance
```

而不是：

```text
A2A 子 agent
  -> 直接写 memory
```

这条边界必须和 agent 协议边界一样强，否则新引入的子 agent 会重新污染会话与长期记忆。

> 继续围绕 `worker_request`、`tool_input`、`binding_owner_task_id` 临时补丁式演化。

而是：

> 以 A2A 作为子 agent 主通信协议，以 MCP 作为子 agent 工具契约协议，以 `x-langchain-agent.*` 承载对象/结果/子集扩展，保持 `QueryRuntime` 作为唯一控制面 owner。

这样做的收益是：

1. 当前框架可以平滑对接，不需要推翻重写。
2. `P0` 这类 owner 污染问题能在协议层被切断。
3. 新增子 agent、新增工具、新增模态时不需要重构主链。
4. 本地实现与未来远程服务化使用同一业务协议。

一句话收口：

> 以后扩展系统，不再“多加几条 heuristics”，而是在 A2A 主骨架和 MCP 工具骨架上增加新的 agent card、handle schema 和 orchestrator rule。

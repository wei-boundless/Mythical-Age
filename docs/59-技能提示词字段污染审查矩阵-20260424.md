# 技能提示词字段污染审查矩阵

> 日期：2026-04-24
> 状态：审查完成，待执行
> 关联计划：
> - `docs/58-协议残差与脑补工具调用收缩计划书.md`
> - `docs/55-工具与技能系统契约化重构初步计划书.md`

---

## 1. 审查目标

本矩阵只回答一个问题：

`当前有哪些 skill 字段正在污染模型可见提示词，以及这些字段该保留、降级还是移除`

本矩阵不处理：

1. skill 注册数据本身是否还要保留在 runtime。
2. tool contract 是否需要调整。
3. follow-up / binding / route 的结构主链。

本矩阵只处理：

1. `SKILLS_SNAPSHOT.md` 里的模型可见字段。
2. `format_active_skill_block()` 生成的模型可见字段。

---

## 2. 当前注入链

当前 skill 相关信息进入模型提示词有两条链。

## 2.1 静态全量注入

文件：

- `backend/query/prompt_builder.py`
- `backend/SKILLS_SNAPSHOT.md`

行为：

1. `build_static_prompt()` 会把 `SKILLS_SNAPSHOT.md` 整份注入 static prompt。
2. 这意味着所有 skill 的控制面字段都会长期暴露给模型。

问题：

1. 这不是“当前任务需要的能力提示”，而是“全量技能控制面目录”。
2. 会让模型长期学习 route、tool、authority、activation 等内部术语。

## 2.2 当前激活 skill 注入

文件：

- `backend/query/runtime.py`
- `backend/query/prompt_builder.py`
- `backend/skill_system/registry.py`

行为：

1. `runtime` 会把 `format_active_skill_block(active_skill)` 传给 `build_system_prompt()`。
2. `build_session_memoized_prompt()` 会把它注入 `<!-- Active Skill -->`。

问题：

1. 当前激活 skill 的控制面字段再次重复暴露。
2. 它会直接诱导模型按“我现在走某 route / 调某工具”的方式说话。

---

## 3. 字段污染矩阵

说明：

- `保留`：可以继续进入模型可见提示词
- `降级`：字段本身保留在 runtime，但不应原样进入模型提示词，应改写为更抽象、更面向任务的说明
- `移除`：不应再进入模型可见提示词，只保留在内部结构或 trace 中

## 3.1 Active Skill Block

来源：

- `backend/skill_system/registry.py::format_active_skill_block()`

| 字段 | 当前状态 | 处理决定 | 原因 |
|---|---|---|---|
| `Skill` | 模型可见 | 保留 | 低风险，属于能力名，不是控制面协议 |
| `Skill ID` | 模型可见 | 降级 | 更偏内部实现标识，必要性弱 |
| `Description` | 模型可见 | 保留 | 这是最有价值的能力说明字段 |
| `Preferred Route` | 模型可见 | 移除 | 纯路由控制面字段，易诱导模型说“我走 tool/rag” |
| `Activation Policy` | 模型可见 | 移除 | 纯 runtime 元信息 |
| `Context Mode` | 模型可见 | 移除 | 纯上下文装配元信息，如 `inline/isolated` 不该给模型看 |
| `Route Authority` | 模型可见 | 移除 | 纯仲裁控制面字段 |
| `Supported Sources` | 模型可见 | 降级 | 可保留为非常抽象的“适用资料类型”，不应原样使用内部 taxonomy |
| `Supported Tasks` | 模型可见 | 降级 | 可保留为非常抽象的“适用问题类型”，不应原样使用内部 task kind 列表 |
| `Allowed Tools` | 模型可见 | 移除 | 这是最直接的工具协议污染源 |
| `Modalities` | 模型可见 | 降级 | 可转成简短能力范围说明，不必保留内部 modality 枚举 |
| `Routing Hints` | 模型可见 | 移除 | 本质是启发式触发词表，不应喂给模型 |

## 3.2 Skills Snapshot Block

来源：

- `backend/SKILLS_SNAPSHOT.md`

| XML 字段 | 当前状态 | 处理决定 | 原因 |
|---|---|---|---|
| `<summary>` | 模型可见 | 降级 | 当前 summary 仍含“workflow contracts / invoke”这类控制面措辞 |
| `<skill name>` | 模型可见 | 保留 | 技能名可见问题不大 |
| `id` | 模型可见 | 降级 | 更偏内部实现标识 |
| `path` | 模型可见 | 移除 | 泄漏内部文件路径，无任务价值 |
| `<description>` | 模型可见 | 保留 | 是核心能力说明 |
| `<preferred_route>` | 模型可见 | 移除 | 纯控制面协议 |
| `<modalities>` | 模型可见 | 降级 | 可抽象，但不应保留内部枚举风格 |
| `<source_kinds>` | 模型可见 | 降级 | 可改成“适用资料类型”，不应用内部 source taxonomy |
| `<task_kinds>` | 模型可见 | 降级 | 可改成“适用问题类型”，不应用内部 task taxonomy |
| `<allowed_tools>` | 模型可见 | 移除 | 直接污染工具调用语言 |
| `<capability_tags>` | 模型可见 | 降级 | 价值有限，可压缩成一句能力范围 |
| `<activation_policy>` | 模型可见 | 移除 | 纯 runtime 元信息 |
| `<context_mode>` | 模型可见 | 移除 | 纯装配元信息 |
| `<route_authority>` | 模型可见 | 移除 | 纯控制面字段 |
| `<routing_hints>` | 模型可见 | 移除 | 触发词表不应直接喂给模型 |
| `<forbidden_routes>` | 模型可见 | 移除 | 纯路由约束，不是模型该消费的能力信息 |
| `<references>` | 模型可见 | 移除 | 泄漏内部路径和实现组织方式 |

## 4. 高风险字段清单

这些字段是当前最明确的“协议残差”来源，应优先移除。

## 4.1 第一优先级

1. `Allowed Tools`
2. `Preferred Route`
3. `Route Authority`
4. `Activation Policy`
5. `Context Mode`
6. `Routing Hints`
7. `forbidden_routes`

这些字段的共同特征是：

1. 它们全部属于控制面，不属于任务知识。
2. 它们会显著诱导模型使用系统内部术语组织回答。
3. 它们会把“技能能力描述”扭曲成“运行时协议说明”。

## 4.2 第二优先级

1. `path`
2. `references`
3. `id`
4. `task_kinds`
5. `source_kinds`

这些字段未必直接诱导工具脑补，但会：

1. 泄漏内部实现结构
2. 增强“系统内部感”
3. 让模型学习仓库内部 taxonomy，而不是用户语义

---

## 5. 推荐的保留最小集

如果按最小必要原则，建议模型可见的 skill 信息只保留四类内容：

1. 技能名称
2. 一句能力说明
3. 一句适用场景说明
4. 一句输出行为约束

也就是说，最终模型可见 skill block 可以收缩为类似形式：

```text
Skill: 结构化数据分析
Capability: 适合处理本地表格和结构化数据的统计、排序、分组汇总问题。
Use When: 用户在问表格、Excel、CSV、JSON 数据中的排名、汇总、筛选、Top N 或记录查询。
Output Rule: 直接回答用户问题，不描述内部工具调用过程，不暴露内部协议。
```

对应地，`SKILLS_SNAPSHOT.md` 也不应再是“技能注册表镜像”，而应是非常轻量的能力目录。

---

## 6. 推荐的 Phase 1 执行顺序

本矩阵建议的落地顺序如下：

1. 先修改 `backend/skill_system/registry.py`
   - 收缩 `format_active_skill_block()`
2. 再修改 `backend/SKILLS_SNAPSHOT.md` 的生成逻辑
   - 让 snapshot 从“控制面快照”改成“能力目录快照”
3. 最后验证 `backend/query/prompt_builder.py`
   - 确保 static prompt 和 active skill 不再重复暴露协议字段

这样做的原因是：

1. `active_skill` 是当前轮直接污染源，优先级最高。
2. `SKILLS_SNAPSHOT.md` 是长期静态污染源，范围更广。
3. `prompt_builder` 本身不是主污染源，它只是装配器。

---

## 7. 执行边界

本矩阵明确建议：

1. 这一步只处理“模型可见字段”。
2. 不删除 runtime 内部仍需要的 skill 字段。
3. 不修改 route、tool contract、follow-up 主链。
4. 不通过新增正则来掩盖污染。

也就是说，本阶段的正确动作是：

`字段降权与模型可见面收缩`

而不是：

`结构层回摆`

---

## 8. 最终结论

当前 skill 系统对提示词的污染不是抽象担忧，而是已被代码结构确认的事实：

1. `SKILLS_SNAPSHOT.md` 在 static prompt 中长期暴露全量控制面字段。
2. `Active Skill Block` 在 session prompt 中重复暴露当前 skill 的控制面字段。

因此，Phase 1 的正式收缩方向应是：

1. 保留能力说明
2. 移除控制面协议
3. 降级内部 taxonomy
4. 禁止把工具白名单、路由偏好、触发词表继续暴露给模型

这一步做完之后，才有资格继续判断：

`协议残差到底还剩多少是模型自身问题，多少其实是系统自己教出来的`

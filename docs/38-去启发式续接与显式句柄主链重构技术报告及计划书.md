# 去启发式续接与显式句柄主链重构技术报告及计划书

> 编写日期：2026-04-21  
> 目标：针对当前系统中“`那 / 再 / 呢 / 这个表 / 那个表` 等弱语言线索误触发结构化续接、历史兜底绑定拖偏主链”的问题，先给出一份技术报告，说明设计思路来源、为什么采用、为什么不采用；再给出一份严格贴合当前代码结构的重构计划书，用于后续一次性推进实现。

## Part A. 技术报告

## 1. 问题定义

当前系统的核心问题，不是单个关键词识别不准，而是续接和绑定的判决权仍有一部分落在“弱语言线索 + 历史兜底”上。

当前真实风险表现为：

- 用户只说“那个表”“再看下”“那一页”，系统就可能把它当成结构化对象续接入口。
- 当显式对象未出现时，`history_fallback` 仍可能从历史里捡出一个旧 `.xlsx` 或旧 PDF 继续执行。
- `task_understanding`、`continuation_resolver`、`binding_resolver`、`runtime`、session/process state 分别持有一部分“当前真相”，但没有形成单一 owner。
- 恢复层、续接层、绑定层没有完全分离，导致“恢复成功”被误用为“当前轮就应该继续它”。

本次重构的本质目标是：

> 把“续接谁、绑定谁、恢复什么”从启发式文本理解，改造成显式句柄、单一 owner、状态受控的主链协议。

## 3. 各方案对这个问题的适配度比较

### 3.1 Claude Code

优点：

- 继续逻辑强依赖显式句柄。
- 恢复与续接严格分层。
- 主线程和子执行单元的上下文职责清楚。
- 明确反对“based on your findings”这类把理解责任外包的做法。

局限：

- Claude Code 主要解决 worker/session 的继续问题，不直接提供“表格/PDF/数据对象绑定”的业务层抽象。
- 它能给纪律，但不能直接给你现成的数据对象主链模型。

本项目应采用的部分：

- 句柄优先。
- 恢复只恢复，不裁决。
- 父线程综合理解后再派发。

本项目不应机械照搬的部分：

- 不需要把你的执行体系改造成 Claude Code 的完整多 agent 模式。
- 不需要为了学 Claude 再叠更多 agent 壳层。

### 3.2 LangGraph

优点：

- `thread_id`、checkpoint、state snapshot、subgraph namespace 很适合建“主链状态机”。
- 非常强调显式 state，而不是把当前真相留在自然语言里。
- 对“恢复点”和“状态 owner”有天然优势。

局限：

- 如果全量迁移到 LangGraph，改造成本会很大。
- 你当前项目已经有自己的 runtime、planner、task coordinator，没必要整体换框架。

本项目应采用的部分：

- 把“当前主链”“当前对象绑定”“当前可续接 owner”建成显式 state。
- 把恢复点、绑定点、输出点分成独立层。

### 3.3 OpenAI Agents SDK

优点：

- `Session` 和 `RunContext` 的分层非常适合“本地状态不进 LLM 上下文”。
- `handoff` 是显式目的地，不是模糊语言跳转。
- 非常适合借鉴“控制面状态”和“模型可见上下文”分离。

局限：

- 对复杂主链状态机的约束不如 LangGraph 强。
- 它更像 agent orchestration SDK，不是你的对象绑定系统现成替代品。

本项目应采用的部分：

- 把 `active_task_id`、`active_binding_ref`、`binding_owner_task_id` 放在本地 runtime state，而不是扔给模型自己推。
- 把主线程可见控制面和内部状态层严格分开。

### 3.4 Semantic Kernel Process Framework

优点：

- process / step / event / resume 的企业流程感很强。
- 很适合要求审计性、暂停恢复、状态机清晰的系统。

局限：

- 对你的问题来说偏重。
- 它更像“流程编排引擎”，而不是“对话续接对象绑定协议”。

本项目应采用的部分：

- 步骤化生命周期。
- 显式事件驱动状态切换。

### 3.5 AutoGen

优点：

- agent/team 的 save/load state 能力完善。
- 多 agent 状态持久化思路比较成熟。

局限：

- 它更擅长保存 agent 会话状态，不天然解决“对象绑定不能被泛词拖偏”。
- 如果缺少强约束，很容易重新滑回多 agent + LLM 自行理解的一般性方案。

本项目应采用的部分：

- 保存/恢复状态对象时，保留稳定 ID 和父子关系。

### 3.6 CrewAI

优点：

- 上手快。
- memory、分类、归档、召回一体化程度高。

局限：

- 官方 memory 方案里，很多归档和召回前分析都仍依赖 LLM 判断 scope / importance / category。
- 这和你当前要解决的问题是反方向的，因为你最需要的是“减少把主链真相交给 LLM 临场判断”。

本项目结论：

- 不建议把 CrewAI 的 unified memory 设计拿来作为这条续接主链的基础。

## 4. 本项目推荐采用的设计路线

最终推荐采用：

> Claude Code 的显式续接纪律  
> + LangGraph 的显式状态与 checkpoint 观念  
> + OpenAI Agents 的本地上下文分层方式

这不是三套框架叠加，而是三类原则的组合：

1. 继续谁，靠显式句柄。
2. 当前真相，靠显式状态。
3. 模型只看到必要控制面，不看到内部裁决状态。

### 4.1 为什么选这个组合

因为它同时满足你最在意的 5 个要求：

- 不靠大量启发式语言触发器。
- 不把主链真相交给临场语义理解。
- 不引入过多维护成本。
- 能兼容你当前已有的 `TaskRecord / FollowupResolution / MainContextState / OutputBoundary` 基础。
- 能把之前几轮改造里已经做对的部分收束到一个统一协议上。

### 4.2 为什么不是“继续调 prompt / 继续调词表”

因为词表补丁只能解决局部误判，不能解决以下结构问题：

- 谁有资格成为当前 owner。
- 恢复层何时可参与，何时必须退场。
- 哪个 state 才是主链唯一真相。
- 输出层消费的到底是 summary/ref，还是原始工具输出。

这些问题如果不在结构层解决，后续无论怎么调 prompt，都会重复出现“又修出一个新绑错”。

---

## 5. 本次重构的总原则

下面这些原则将在后半部分计划书里逐项落地：

1. 弱语言线索只能触发“可能是 follow-up”的识别，不能直接触发结构化绑定。
2. 显式路径、显式句柄、显式对象引用优先级最高。
3. 恢复层只能恢复可选候选，不能越权裁决当前轮 owner。
4. 任何可续接执行都必须能落到稳定 `task_id`。
5. 任何可续接对象都必须能落到稳定 `binding_ref`。
6. 主线程只保留控制面真相，不保留 task-local 工作文本。
7. 输出层优先消费 summary/ref，不直接消费 raw tool output。
8. 同一对象 identity 在续接层只能形成一个活跃 lineage，不能被多个历史 task 并列争抢。
9. 泛指弱词只能触发 follow-up 倾向或澄清判断，不能直接成为对象 owner 的证据。

## 7. 重构目标架构

本轮要形成一条统一协议：

`显式用户输入 -> follow-up 识别 -> 句柄/owner 仲裁 -> runtime 执行 -> summary/ref 输出 -> 控制面更新`

### 7.1 核心对象

本轮统一围绕下面这些对象工作：

- `task_id`
- `task_kind`
- `binding_ref`
- `binding_kind`
- `binding_identity`
- `binding_owner_task_id`
- `active_task_id`
- `active_binding_ref`
- `active_binding_identity`
- `continuation_mode`

### 7.2 核心判决顺序

后续代码必须统一使用下面的顺序：

1. 用户是否给出了显式路径、显式文件、显式对象引用。
2. 当前是否存在可续接的显式 `task_id` 或 `binding_ref`。
3. 当前 active state 是否允许在同 owner 下续接。
4. 只有在缺失显式句柄时，才允许恢复层提供候选。
5. 恢复层提供的是候选，不是裁决。
6. 如果候选不唯一或信号不足，则澄清或降级为 fresh query，而不是偷偷绑定。

### 7.3 同一 binding identity 合并

后续代码必须统一引入 `binding_identity` 概念，用来表达“当前对象到底是哪一个具体 PDF / dataset”。

要求：

1. 同一路径归一化后的 PDF，只能对应一个活跃 PDF identity。
2. 同一路径归一化后的 dataset，只能对应一个活跃 dataset identity。
3. 历史上多个 task 如果绑定到同一 identity，在续接裁决层必须先合并为一个 lineage，再决定是否续接。
4. 同一 identity 出现多个历史 owner 时，默认只保留最新有效 owner 参与当前轮裁决。

### 7.4 澄清触发契约

后续代码必须统一遵守以下澄清规则：

1. 只有“泛指引用 + 多个不同 binding identity 冲突”时，才允许进入 clarify。
2. 如果只有一个活跃 identity，即使用户说“那个表 / 再看一下 / 那一页”，也应优先续接该 active owner。
3. 如果没有 active owner，且用户也没有给显式对象，则降级为 fresh query 或普通 follow-up，而不是从历史里偷绑一个对象。

### 7.5 弱词的新角色

“那”“再”“呢”“这个表”“那个表”以后只能扮演以下角色之一：

- 提示这可能是一个 follow-up。
- 提示需要去查当前 active owner。
- 提示可能需要澄清。

它们以后不能单独扮演以下角色：

- 结构化对象句柄。
- dataset / PDF 的直接绑定依据。
- history fallback 的开闸条件。

## 8. 数据结构重构方案

### 8.1 `FollowupResolution` 扩充为显式主链裁决结果

目标：

- 让 follow-up resolver 输出的不是“看起来像 follow-up”，而是“当前轮主链裁决结果”。

建议字段：

- `resolved_task_id`
- `resolved_task_kind`
- `resolved_binding_ref`
- `resolved_binding_kind`
- `resolved_binding_identity`
- `resolved_binding_owner_task_id`
- `resolution_source`
- `requires_clarification`
- `clarification_reason`

要求：

- `resolution_source` 必须明确区分 `explicit_input / active_handle / restored_candidate / none`
- 禁止再出现“看起来像跟这个表有关，所以先绑一下”这种无来源 owner

### 8.2 `MainContextState` 只保留控制面状态

目标：

- 从主线程 state 中清掉 task-local 文本真相，只保留控制面引用。

应保留：

- `active_task_id`
- `active_task_kind`
- `active_binding_ref`
- `active_binding_kind`
- `active_binding_identity`
- `active_owner_task_id`
- `active_output_channel`

应移出或弱化：

- 直接页面内容
- 直接表格内容摘要
- task-local prose 续接信息

### 8.3 `TaskContextRef` 成为 binding owner 真相

目标：

- 所有对象级真相都落到 task owner 上。

要求：

- PDF、dataset、session_summary 等对象都必须能挂到 `TaskContextRef.bindings`
- direct tool 执行和 compound 执行都必须落真实 owner

### 8.4 建立 `binding_identity` 归一化规则

目标：

- 让续接层判定的是“同一个对象”，不是“历史里恰好出现过几个 task”。

要求：

- PDF identity 基于归一化文件路径生成。
- dataset identity 基于归一化文件路径生成。
- 同一 identity 允许存在多个历史 task，但当前轮裁决时必须先折叠成一个 lineage。
- `binding_ref` 用于执行注入，`binding_identity` 用于续接判定；两者职责不能混淆。

2. 用 prompt 补丁去弥补 owner 不统一的问题。
3. 让 history fallback 继续直接返回绑定结果。
4. 让 planner 或 tool_input_resolver 临时猜当前对象 owner。
5. direct tool 继续不落真实 task handle。
6. output boundary 继续直接消费 raw tool output。

## 14. 最终结论

本次重构不应继续走“补启发式短语”的路线。

正确路线应当是：

> 弱词退位  
> 句柄上位  
> owner 唯一  
> 恢复降权  
> 输出收口

也就是说，后续的实现目标不是“让系统更会猜”，而是“让系统不需要猜那么多”。

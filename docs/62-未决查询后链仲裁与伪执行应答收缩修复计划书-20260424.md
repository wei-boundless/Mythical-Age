# 未决查询后链仲裁与伪执行应答收缩修复计划书

> 日期：2026-04-24
> 状态：待执行
> 目标问题：
> - `搜索本地数据库` 被入口短路成 `web_search`
> - `他今年还在打比赛吗` 这类显式时效性问题又被压成 `rag`
> - 模型在没有真实工具执行时，仍会输出“我现在去查 / 我需要查询一下”
> - 这些伪执行应答还会被持久化进 session，继续污染后续对话

## 1. 问题定义

当前问题已经不再是单个词命中不准，而是同一条链路上同时存在两类失真：

1. 入口理解层仍保留“弱词直判强 route”的旧式短路权。
2. 对未决查询，后链没有提供稳定的“受限工具池仲裁”通道。
3. 输出层仍允许“未真实执行的查询承诺”晋升为可见答案。
4. 持久化层会把这些伪执行文本写回 session，形成下一轮污染。

因此，本次计划要修的不是单个模块，而是一条完整责任链：

`fresh query -> execution posture -> bounded tool availability -> answer gate -> persistence gate`

## 2. 技术源报告

## 2.1 已确认的真实失败样本

### 样本 A：本地查询被误判成联网搜索

证据：

- `backend/sessions/75b659b1d3574435864fb4ee5b8e0bf4.json`
- `output/local_traces/20260424/local-20260424-091112-75b659b1d3574435-1528b8a3.json`

用户原话：

- `为我搜索本地的数据库，看看有没有缺货情况`

已确认事实：

1. 当前系统把它路由成了 `app.route = tool`
2. 实际工具为 `app.tool_name = web_search`
3. 对应 skill 为 `app.skill_name = web-search`

这说明：

1. `搜索` 在入口理解层仍然拥有过高的 route authority
2. `动作词 = 联网工具` 的错误等式仍然存在

### 样本 B：显式时效性问题被压成 RAG，工具池为空

证据：

- `backend/sessions/75b659b1d3574435864fb4ee5b8e0bf4.json`
- `output/local_traces/20260424/local-20260424-093200-75b659b1d3574435-fa3f01c1.json`
- `output/local_traces/20260424/local-20260424-093411-75b659b1d3574435-f1ce0752.json`

用户原话：

- `他今年还在打比赛吗`
- `你真的有在查吗 怎么感觉你没动啊`

trace 已确认：

1. 两轮都落到了 `app.route = rag`
2. `app.skill_name = rag-skill`
3. `app.tool_count = 0`
4. 回答来源是 `app.answer_source = segment.visible_text`

这说明：

1. 入口没有把“今年 / 还在 / 最新状态”这类显式时效性约束识别成“需要后链 freshness 仲裁”
2. 一旦落入 `rag`，runtime 就完全不给工具
3. 模型实际上没有任何真实查询能力可用

### 样本 C：未执行查询承诺被当作正常答案展示并写回 session

证据：

- `backend/sessions/75b659b1d3574435864fb4ee5b8e0bf4.json`
- `backend/session-memory/75b659b1d3574435864fb4ee5b8e0bf4/summary.md`
- `backend/session-memory/75b659b1d3574435864fb4ee5b8e0bf4/views/agent_view.md`

已确认文本包括：

1. `我需要查询一下勒布朗·詹姆斯的最新状态，稍等。`
2. `我现在立即查询勒布朗·詹姆斯的最新状态。`
3. `我确实没有真正执行查询。这是我的问题。现在立即查询……`

这说明：

1. 输出层没有把“未执行承诺”识别为过程文本
2. 持久化层没有阻止这类文本进入 session record
3. 后续 memory compaction 又把它们当成“上一阶段结果”继续注入

## 2.2 当前代码中的真实责任链

### A. 入口理解层仍保留强短路树

文件：

- `backend/understanding/task_understanding.py`

当前事实：

1. `analyze_task_understanding()` 仍是固定顺序短路树
2. `_detect_web_task()` 中 `explicit_web_markers` 仍包含 `搜索`
3. 未命中 web/pdf/dataset/faq 后，会直接落到 `fallback_knowledge_lookup`

这导致两个对称错误：

1. `搜索本地数据库` 因 `搜索` 被抬成 `web`
2. `他今年还在打比赛吗` 因未出现 `搜索/联网/官网/新闻` 被压成 `rag`

### B. QueryUnderstanding 会把入口结果直接冻结

文件：

- `backend/understanding/query_understanding.py`

当前事实：

1. `_from_task()` 直接把 `TaskUnderstanding` 投影成最终 `route`
2. 如果 route 不是 `tool`，后面就不会再进入工具选择收束

这意味着：

1. 入口一旦把问题定成 `rag`
2. planner/runtime 后面就没有“重新考虑当前事实查询是否需要工具”的正式位置

### C. Runtime 对 `rag` 路线直接清空工具池

文件：

- `backend/query/runtime.py`

当前事实：

1. `_allowed_tool_names_for_execution()` 对 `route in {"memory", "rag"}` 直接返回空集合
2. `query.model_stream` 阶段因此出现 `tool_count = 0`

这意味着：

1. 对任何被误判为 `rag` 的时效性问题
2. 模型即使“想查”，也没有任何工具可用

### D. 输出分类器会放过“查询承诺句”

文件：

- `backend/query/output_classifier.py`
- `backend/query/output_boundary.py`

当前事实：

1. `looks_like_progress_text()` 主要覆盖 `检索/搜索/查看/读取...`
2. `_PROCEDURAL_LINE_RE` 也主要覆盖“我来检索 / 我来查看 / 我将调用...”
3. `查询一下 / 立即查询 / 需要查询` 没有被完整覆盖
4. 在 `classify_output_candidate()` 中，未命中 progress 规则的文本仍可能作为 `answer_candidate`

这意味着：

1. 模型的“伪执行承诺”能穿过输出层
2. 即使当前 turn 根本没有 tool receipt，也照样能成为答案

### E. 持久化层没有拒绝“无 receipt 的过程应答”

文件：

- `backend/query/runtime.py`

当前事实：

1. `astream()` 完成后会直接把 canonical content 持久化进 session
2. 当前没有一层“过程承诺但无 receipt”的 persistence gate

这意味着：

1. 输出层漏掉一次
2. session 和 memory 就会把错误进一步固化

## 3. 从 docs 提炼出的硬约束

本次计划必须严格遵守以下原则。

### 3.1 来自 `docs/38`

1. 弱信号不能承担裁决权。
2. restore 不等于 decide。
3. 正确方向不是“让系统更会猜”，而是“让系统不需要猜那么多”。

对本问题的含义：

1. `搜索` 不能再直接决定 `web_search`
2. `今年/现在/最新` 这类时效性信号也不能直接决定具体工具
3. 它们只能决定“是否需要进入 freshness-aware 后链”

### 3.2 来自 `docs/55`

1. 工具契约层已经收口，不应重新把问题修回“谁能偷偷触发工具”
2. skill 已降权为能力系统，不应重新让 hint/example/tag 抢 route
3. 高污染工具继续 fail-closed，不得靠隐式猜测补 path

对本问题的含义：

1. 不能新增“本地数据库/库存表”词表来偷判 dataset tool
2. 不能重新把 `web-search` skill 的 `搜索` 提示词拿来做路由依据
3. 不能为了让它“更会查”而重新打开工具猜测执行

### 3.3 来自 `docs/58`

1. 工具调用真相应由结构承载，不由自然语言承载
2. 不能把过程化文本当成执行凭证
3. 不能只靠输出层正则兜底

对本问题的含义：

1. “我现在去查”不是工具执行
2. “我需要查询一下”不是 receipt
3. 输出层必须显式拒绝“无 receipt 的执行承诺”

### 3.4 来自 `docs/61`

1. `动作词 != route authority`
2. 入口应从“启发式裁决器”收缩成“强直达检测器 + 未决任务分流器”
3. 未决查询应交给后链的能力边界与契约继续收束

对本问题的含义：

1. `搜索本地数据库` 不应直达 web
2. `他今年还在打比赛吗` 不应直接落纯 rag
3. 两类问题都应进入“未决查询”的统一主链，而不是各自打 patch

## 4. 推荐设计方向

本次不采用“再补词表”，也不采用“只在输出层加正则”。

推荐路线是：

`入口强直达收缩 + 未决查询后链仲裁 + 无 receipt 伪执行应答拒收 + persistence gate`

这条路线的核心不是“更激进地调 prompt”，而是重排以下权力边界：

1. 入口层只负责识别强显式任务与未决任务
2. planner/runtime 负责未决任务的 bounded tools 仲裁
3. 输出层负责拒绝没有真实 receipt 的执行承诺
4. 持久化层负责阻止伪执行文本进入 session 和记忆

---

## 5. 目标架构

改造后的固定主链应为：

`user query -> execution posture -> bounded tool scope -> model/tool loop -> tool receipt -> answer gate -> persistence gate`

## 5.1 Execution Posture

入口层不再直接输出“最终工具是谁”，而是输出以下几类执行姿态：

1. `direct_tool`
   仅用于强显式任务：
   - weather
   - gold
   - explicit web request
   - explicit pdf reference
   - explicit dataset path

2. `direct_rag`
   仅用于稳定本地知识查询：
   - 明确知识库/本地资料/FAQ 类问题
   - 无时效性约束
   - 无工具必要性

3. `bounded_agent`
   用于未决查询：
   - 显式时效性约束，但未明确外部来源
   - 宽 lookup 请求，来源未定
   - 有 active owner/binding，可由后链继续收束

## 5.2 Bounded Tool Scope

`bounded_agent` 不是“全工具开放”，而是受限能力集。

初始约束：

1. 默认未决事实查询：
   - `search_knowledge`
   - `web_search`

2. 若存在 active dataset binding：
   - 在上面基础上附加 `structured_data_analysis`

3. 若存在 active pdf binding：
   - 在上面基础上附加 `pdf_analysis`

4. 不向未决查询开放：
   - `terminal`
   - `python_repl`
   - 写操作工具

这样可以满足两点：

1. 不把 `搜索` 直接翻译成 `web_search`
2. 也不让 `rag` 路线继续完全失去工具能力

## 5.3 Tool Receipt

后链必须显式记录最小 receipt：

1. `tool_name`
2. `status`
3. `input_digest`
4. `canonical_summary_available`
5. `evidence_ref`

规则：

1. 没有 receipt，就没有“已执行查询”这回事
2. 自然语言承诺不能反向制造 receipt

## 5.4 Answer Gate

输出层必须新增正式 gate：

1. 过程承诺句：
   - `我需要查询一下`
   - `我现在立即查询`
   - `我去确认一下最新状态`
   若无 receipt，则一律拒收

2. 显式工具声称句：
   - `我已经查到`
   - `我刚刚查询了`
   若无 receipt，则一律拒收

3. 对 `bounded_agent` 路线：
   - 要么展示真实工具结果的可见摘要
   - 要么明确 fallback
   - 不允许展示“空执行承诺”

## 5.5 Persistence Gate

在 assistant 内容写回 session 前，必须再做一次 gate：

1. 纯过程承诺且无 receipt：不持久化
2. 过程承诺 + 真实 receipt：只持久化最终可见结论，不持久化承诺句
3. fallback 说明可持久化，但不能写成“我现在去查”

## 6. 分阶段实施计划

## Phase 1：入口层改成“强直达 or 未决”

目标：

1. 去掉 `搜索 -> web_search` 的弱短路
2. 去掉“未命中强词就直接 rag”的单向塌陷
3. 建立正式的 `bounded_agent` 姿态

涉及文件：

- `backend/understanding/task_understanding.py`
- `backend/understanding/query_understanding.py`

具体动作：

1. `_detect_web_task()` 只保留强显式外部性触发：
   - 联网
   - 官网
   - 新闻
   - official docs
   - web search
   - 上网 / 网上查

2. 删除 `搜索` 作为 web 直达标志。

3. 为显式时效性约束新增“未决 freshness posture”：
   - 今年
   - 现在
   - 还在
   - 最新状态
   - 目前
   这些信号只能触发 `bounded_agent`，不能直接锁具体工具。

4. fallback 不再直接是 `rag`，而是按是否存在 freshness / active owner 决定：
   - 稳定知识 -> `direct_rag`
   - 时效性或来源未定 -> `bounded_agent`

完成标准：

1. `搜索本地数据库` 不再直达 `web_search`
2. `他今年还在打比赛吗` 不再落纯 `rag`
3. 入口不新增“本地数据库/库存表”词表

## Phase 2：planner/runtime 引入 bounded tools 仲裁

目标：

1. 为未决查询提供正式的后链工具能力
2. 不重新打开工具猜测执行

涉及文件：

- `backend/query/planner.py`
- `backend/query/runtime.py`

具体动作：

1. 在 plan/execution 中显式加入 `execution_posture`

2. 对 `bounded_agent`：
   - planner 生成受限 candidate tools
   - runtime 不再因为 route 不是 `tool` 就把工具池清空

3. `runtime._allowed_tool_names_for_execution()` 改成：
   - `direct_rag` 才清空工具
   - `bounded_agent` 返回 bounded tool names

4. trace 增加：
   - `app.execution_posture`
   - `app.bound_candidate_tools`
   - `app.direct_route_reason`

完成标准：

1. 未决查询进入 model stream 时 `tool_count > 0`
2. 但工具池仍受限，不会全开

## Phase 3：输出层新增“无 receipt 伪执行应答”拒收

目标：

1. 让“我现在去查”不再被当作答案
2. 把伪执行承诺从输出层正式剔除

涉及文件：

- `backend/query/output_classifier.py`
- `backend/query/output_boundary.py`
- `backend/query/runtime.py`

具体动作：

1. 在输出分类中新增：
   - `procedural_promise`
   - `tool_claim_without_receipt`

2. 扩展当前过程文本检测，覆盖：
   - 查询
   - 核实
   - 确认最新状态
   - 立即查询
   - 稍等我查一下

3. `build_output_decision()` 在无 receipt 时拒绝这些候选，不再默认归入 `answer_candidate`

4. 为 `bounded_agent` 增加 fallback：
   - `尚未形成真实查询结果`
   - 而不是“我现在去查”

完成标准：

1. 无 receipt 的查询承诺不能成为 `selected_channel=answer_candidate`
2. trace 中能看到明确的 `fallback_reason`

## Phase 4：引入 persistence gate，阻止 session 污染

目标：

1. 不让伪执行文本写入 session
2. 阻断它们进入后续 memory compaction

涉及文件：

- `backend/query/runtime.py`

具体动作：

1. 在 `_build_assistant_messages()` 或更早位置增加 persistence gate

2. 若 canonical content 满足：
   - procedural promise
   - no receipt
   则不写入 session，或仅写入安全 fallback

3. 若本轮确有 receipt，但 visible text 含过程句：
   - 只保留最终稳定结论
   - 过程句不入 session

完成标准：

1. `session-memory/*/summary.md` 不再出现“我现在立即查询……”一类伪结果

## Phase 5：长场景回归与收口

目标：

1. 验证入口降权和输出收口同时生效
2. 确保不破坏 `docs/55` 已收口边界

涉及文件：

- `backend/tests/task_understanding_regression.py`
- `backend/tests/query_planner_regression.py`
- 新增建议：`backend/tests/unresolved_lookup_regression.py`
- 新增建议：`backend/tests/procedural_promise_guard_regression.py`

完成标准：

1. 本地查询不再误进 web
2. 实时问题不再假装执行
3. session 不再记住伪执行结果

## 8. 固定执行流程

为防止实施中再次回摆，执行顺序固定如下：

1. 先改入口执行姿态，不先补输出层正则
2. 再改 runtime 工具可用性，不先补更多 skill hints
3. 再加 receipt-aware answer gate
4. 最后加 persistence gate
5. 每一阶段都补回归后再进下一阶段

禁止反过来：

1. 先用输出层正则掩盖入口误路由
2. 先补“本地数据库/库存表”词表替代结构改造
3. 先让模型“更会说自己在查”而不是让它真的有可用工具

1. session 中不再保存纯过程承诺句
2. memory compaction 不再把这类句子当“上一阶段结果”

---

## 10. 风险控制与迁移策略

本次改造涉及入口与 runtime 主链，必须按影子模式切换。

建议：

1. 引入 `UNRESOLVED_LOOKUP_MODE=shadow|enforce`
2. `shadow` 模式下：
   - 保留旧 route
   - 记录新 `execution_posture` 与新 bounded tool scope 到 trace
3. `enforce` 模式下：
   - 真正切换到新执行姿态

回滚策略：

1. 若 Phase 1 造成 route 大面积漂移，只回滚入口 posture 切换
2. 若 Phase 3 误杀正常答案，只回滚 no-receipt gate，不回滚入口降权

这样可以保证：

1. 结构修复和输出修复分开验证
2. 不会因为一个 gate 误判把整条新主链全退回去

## 11. 明确禁止事项

本计划执行中，以下做法一律禁止：

1. 继续给 `_detect_web_task()` 追加宽词表
2. 新增“本地数据库/库存表/仓库数据”一类本地词表来替代结构修复
3. 重新让 `web-search` skill 的 hints/examples 参与 route authority
4. 通过 prompt 修辞要求模型“别假装去查”来代替 receipt gate
5. 只改 `output_boundary.py` 正则，不改入口与 runtime
6. 让无 receipt 的承诺文本继续进入 session

## 12. 最终结论

当前问题不是单点 bug，而是一条完整链路上的四层失配：

1. 入口把弱词当强 route authority
2. `rag` 路线又把未决查询彻底剥夺了工具能力
3. 输出层允许“未执行承诺”晋升为答案
4. 持久化层继续把错误答案写进 session 和记忆

因此，正确修法不是：

- 补更多关键词
- 调 prompt 让模型“更自觉”
- 只在输出层继续堆正则

而是：

1. 把入口收缩成“强直达检测器 + 未决查询分流器”
2. 给未决查询建立 `bounded_agent` 后链仲裁
3. 用 receipt-aware answer gate 拒绝伪执行应答
4. 用 persistence gate 阻断 session 污染

这条路线同时符合：

1. `docs/38` 的“弱信号不能拥有裁决权”
2. `docs/55` 的“工具契约层不回摆”
3. `docs/58` 的“工具真相不能靠自然语言承载”
4. `docs/61` 的“动作词 != route authority”

这才是当前问题域最严格、也最可维护的修法。

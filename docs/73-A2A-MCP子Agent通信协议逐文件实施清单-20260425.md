## 0. 本清单结论

后续实施按四条主线推进，不交叉乱改：

1. **协议冻结线**
   - 先冻结 `A2A-compatible runtime envelope`、`A2A extension schema`、`MCP-compatible tool contract`。

2. **本地兼容线**
   - 先不拆远程服务，先把现有 `worker` 链路改成 A2A-compatible。

3. **续接修复线**
   - 优先把 `follow-up` 从 `owner task replay` 改成 `handle continuation`。

4. **工具收口线**
   - 把 agent 内部工具向 MCP-compatible 契约收口，主线程不再感知工具细节。

固定顺序：

```text
Phase 0 协议冻结
  -> Phase 1 本地 A2A-compatible runtime
  -> Phase 2 handle 续接主链切换
  -> Phase 3 ResultHandle / SubsetHandle 存储层补齐
  -> Phase 4 旧 RAG 分支收口
  -> Phase 5 Prompt 与 memory 投影收紧
  -> Phase 6 MCP-compatible tool adapter
  -> Phase 7 远程化准备
```

顺序调整依据：

1. `MCP adapter` 不应早于 handle 主链，否则 agent 内部工具会在旧 owner 语义上继续扩张。
2. `memory 对接` 不应等到远程化前才做，因为 session memory 和 runtime state 已经会影响当前长场景 follow-up。
3. 旧 `route == "rag" -> retrieval_service.retrieve()` 分支必须在远程化前收口，否则 retrieval-agent 与旧 RAG 主链会长期并存。

## 1. 实施前硬约束

### 1.1 不允许边做边发明新协议

实施过程中，以下内容必须先在文档层冻结，再允许进代码：

1. `agent_id` 命名规则
2. `A2A extension` 命名空间
3. `object/result/subset handle` 字段名
4. `WorkerRequest / WorkerResult` 的兼容字段与未来字段
5. `session memory` 允许保存的最小控制面

### 1.2 不允许绕过控制层

任何阶段都不允许：

1. 让 `pdf_worker / retrieval_worker / structured_data_worker` 自己直接改 `main_context`
2. 让 `runtime_followup` 跳过 `handle` 解析直接回放旧 owner
3. 让 `tool raw output` 重新进入 prompt/history/session-memory

### 1.3 不允许新增提示词补丁来兜底协议缺口

出现语义 bug 时，只允许：

1. 修协议
2. 修 owner 边界
3. 修 handle 续接
4. 修 arbitration

不允许：

1. 在 `task_understanding.py`
2. 在 `continuation_resolver.py`
3. 在 `runtime_followup.py`

新增“某句式 -> 强制走某 worker”的新补丁。

## 10.1 每阶段入口条件

1. follow-up 主链已能按 handle 续接
2. owner replay 只剩 fallback

### Phase 4 入口

1. ResultHandle / SubsetHandle 已能从 runtime state 恢复
2. task registry 只保存索引，不保存 payload

1. direct RAG 已 primary 进入 retrieval agent
2. 旧 RAG 分支只剩可观测 fallback

### Phase 6 入口

1. Prompt 与 memory 投影边界已收紧

### Phase 7 入口

## 10.2 回滚规则

允许回滚：

1. transport adapter
2. 某一阶段新增字段的消费优先级

不允许回滚：

1. `A2A` 作为 agent-agent 主骨架
2. `MCP-compatible` 作为 tool 子层方向
3. `handle continuation` 替代 owner replay
4. 主线程不再吸收 raw tool/worker 噪声
5. 子 agent 不直接写 session 或 durable memory

## 13. 推荐执行顺序

建议严格按下面顺序执行：

1. 补 `worker_models` 与 `evidence_orchestrator` 的协议元数据。
2. 改 `followup_resolver` 与 `runtime_followup`，让 handle continuation 替代 owner replay。
3. 补 `ResultHandle / SubsetHandle` 的 runtime state 存储。
4. 收口旧 `route == "rag"` 分支，让 retrieval agent 成为 primary。
5. 改 `context_models / session_memory / worker_projection`，收紧 prompt 与 memory 投影。
6. 再做 `tools/contracts / tool_registry / mcp_adapter` 的 MCP-compatible adapter。
7. 最后再加 `backend/agents/*` 远程化模块。

原因：

1. 现在最痛的是 follow-up 和 owner 污染，不是远程化
2. 不先把 handle 主链打通，A2A 化也只会把旧 bug 远程化
3. 不先把 memory 边界收好，新增子 agent 会再次污染 session/durable

## 14. 一句话收口

这份实施清单的目标不是“把系统包装得像官方协议”，而是：

> 让现有 runtime 真正以内核协议对齐官方标准，并把本项目独有的对象/结果/子集续接能力稳定挂到 A2A extension 上，使后续扩展新 agent、新工具、新模态时只扩边缘，不重写主链。

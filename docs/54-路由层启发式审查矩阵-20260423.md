> 目的：
> 把当前路由层仍存在的启发式判断按职责合法性分成三类：
>
> 1. `可保留`
> 2. `应下沉为候选/辅助信号`
> 3. `应删除或重写`

---

## 1. 审查结论总览

当前系统并不是“完全不该有启发式”，而是：

1. 启发式数量仍然偏多
2. 一部分启发式仍在承担 route 决策权
3. 一部分启发式虽然不再直接短路主链，但仍会显著影响 planner 前的理解结果

整体判断如下：

- `task_understanding.py`：`启发式过重，且仍承担初始路由裁决`
- `continuation_resolver.py`：`部分健康，部分仍在做高风险 promotion`
- `followup_resolver.py`：`已从路由器降级为候选层，但候选判断仍偏厚`

核心结论：

1. 真正最危险的启发式集中在 `task_understanding.py`。
2. `continuation_resolver.py` 里的“显式文件提升”是合理的，但“summary / authority 适配”仍需要继续收窄。
3. `followup_resolver.py` 现在的问题已经不是“直接越权执行”，而是“候选生成过宽，容易给 runtime 施加错误压力”。
4. Group D 要优先处理的，并不是“删光启发式”，而是：
   - 把承担路由决定权的弱启发式降级
   - 把可保留的强显式信号和弱 hint 明确分层

## 2. 审查标准

### 2.1 可保留

满足以下任一条件的启发式，可以保留：

1. 识别的是显式对象：
   - `.pdf`
   - `.xlsx/.csv/.xls`
   - 明确页码
   - 明确 section
2. 只产生候选，不直接决定执行
3. 误判成本低，且有下游仲裁兜底

### 2.2 应下沉为候选/辅助信号

满足以下条件的启发式，不应直接决定 route，但可以保留为候选：

1. 依赖模糊词：
   - `刚才`
   - `这个`
   - `那份`
   - `结论`
   - `重点`
2. 无法单独确认 source owner
3. 需要结合当前 plan / task registry / authority_context 才能成立

### 2.3 应删除或重写

满足以下条件的启发式，应视为越界：

1. 单个关键词直接决定 route
2. 弱 hint 压过当前 turn 的整体任务意图
3. 恢复逻辑直接承担当前 turn 的决策职责

1. 它现在还是“路由短路树”

- realtime
- knowledge
- pdf contextual signal

而不是先动 FAQ 这种相对稳定的小模块。

## 4. 总体分级矩阵

## 4.1 可保留

这些启发式当前可以保留：

1. 显式文件扩展名识别
2. 显式页码 / section 识别
3. ordinal task 引用识别
4. 显式 binding identity 对齐
5. 强结构化动作识别
6. 全局综合请求的负向排除规则

## 4.2 应下沉为候选/辅助信号

这些启发式不应继续直接改 route 或 owner：

1. `刚才 / 这个 / 那份 / 前面`
2. `结论 / 重点 / 约束 / 提炼`
3. `这页 / 那页 / 这一章`
4. `按仓库 / 按部门 / 展开 / 汇总` 在无显式 source 时的独立指向能力
5. compound authority 套用中的泛语义触发条件

## 4.3 应删除或重写

这些启发式当前最危险：

1. `天气` 单词直接夺走多源总结 route
2. `知识库/报告/总结` 过宽词表直接导向 `rag`
3. `session summary` 仍靠有限词表直判
4. `task_understanding` 的固定顺序短路树

## 6. 最终判断

现在的系统不该再用一句“路由层启发式很多”来描述。

更准确的结论是：

1. `task_understanding` 仍是高耦合启发式短路树
2. `continuation_resolver` 已部分健康，但 summary / authority promotion 还不够收敛
3. `followup_resolver` 已降级成功，但候选范围仍偏宽

因此，下一轮精修的方向不是：

- “把所有启发式删光”

而是：

- `把承担决策权的弱启发式降级`
- `把可保留的强显式信号留在 promotion/candidate 层`
- `把最终路由裁决继续收回主链`

这才符合当前系统已经建立起来的边界治理方向。

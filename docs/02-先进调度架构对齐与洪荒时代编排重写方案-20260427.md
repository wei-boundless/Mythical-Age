# 先进调度架构对齐与 洪荒时代 编排重写方案

日期：2026-04-27  
定位：用于重写 `docs/90` 后续架构口径，冻结编排层、能力契约层、任务层、Agent/Worker 层、Memory/Output 写回层的职责边界。

---

## 0. 结论先行

洪荒时代（mythical ageent）不应该直接照搬 LangGraph、Temporal、OpenAI Agents SDK、Google ADK、AutoGen、CrewAI 或任何单一框架。

当前最适合洪荒时代（mythical ageent）的架构是：

```text
UserRequest
  -> Candidate Producers
     - understanding candidate
     - restore candidate
     - memory candidate
     - capability candidate
     - follow-up candidate
  -> OrchestrationCoordinator
     - canonical OrchestrationPlan
     - ResourcePolicy / MemoryPolicy / ContextPolicy / AnswerPolicy
     - ExecutionGraph / ExecutionDirective
  -> PlanValidator
     - fail-closed
  -> RuntimeControl
     - executable specs
  -> Execution Plane
     - model executor
     - tool executor
     - worker executor
     - bounded agent executor
  -> OutputCommitGate
     - final answer
     - task refs
     - session projection
     - durable memory candidate
```

一句话原则：

> 模型和旧 planner 可以提候选，但当前轮执行真相只能来自 `OrchestrationPlan / ExecutionDirective`。

---

## 1. 设计理念

### 1.1 Control Plane / Data Plane 分离

成熟调度系统的共同点不是“有很多 agent”，而是把“决定怎么做”和“实际去做”分开。

在洪荒时代（mythical ageent）里：

- **Control Plane**：`backend/orchestration/*`
  - 负责本轮计划、权限、资源、上下文、记忆、输出策略。
  - 产出 `OrchestrationPlan`、`ExecutionGraph`、`ExecutionDirective`。
- **Execution Plane**：`backend/query/runtime.py`、`RuntimeToolBridge`、workers、model runtime。
  - 只按 directive 执行。
  - 不再自行理解、恢复、调度、选择工具、决定写回。

出处：

- Microsoft Agent Framework Workflows 把 executor 定义为处理 typed messages、产出 output messages/events 的基本单元；workflow context 负责发送消息和产出工作流输出。  
  https://learn.microsoft.com/en-us/agent-framework/workflows/executors
- Temporal 把 Workflow 定义为步骤序列，Activity 定义为单一明确动作；Workflow 负责编排，Activity 承担可失败、可重试的工作。  
  https://docs.temporal.io/workflows  
  https://docs.temporal.io/activities

洪荒时代落地：

- `OrchestrationCoordinator` 是控制平面。
- `QueryRuntime` 降级为执行胶水。
- `ExecutionDirective` 是控制平面给执行平面的唯一命令。

### 1.2 Candidate 不等于 Decision

当前系统的主要历史问题是：`QueryPlanner / continuation / follow-up / memory restore` 都曾经在某些分支里直接改变执行目标。

新架构必须把这些上游输出全部降级为候选：

```text
理解层       -> IntentFrameCandidate
恢复层       -> RestoreCandidate
记忆层       -> MemoryCandidate
能力层       -> CapabilityCandidate
follow-up 层 -> FollowupCandidate
```

只有 `OrchestrationCoordinator` 能把候选采纳为正式决策。

出处：

- LangGraph durable execution 要求 workflow 可以从 checkpoint 恢复，但 side effects / nondeterministic operations 要包进 task，避免恢复时重复执行。这个思想可以迁移为：恢复层只能恢复状态，不应自行执行副作用或改写当前决策。  
  https://docs.langchain.com/oss/python/langgraph/durable-execution
- 洪荒时代本地原则：`docs/75-原则分层与现有链路违背点精确对照表-20260425.md` 明确写了 `restore != decide`。

洪荒时代落地：

- `QueryPlanner` 只允许产 `QueryPlanCandidate`。
- `RestoreAuthorityContextGate` 只允许产 `RestoreCandidate`。
- `RuntimeFollowupCoordinator` 只允许产 `FollowupCandidate`。
- `orchestration.validation` 决定候选能否进入 directive。

### 1.3 Typed Contract 优先于 Prompt 暗示

Prompt 可以指导模型，但不能作为后端执行授权。

所有可执行资源必须有 typed contract：

- Skill contract
- Tool contract
- Agent profile
- Worker request/result contract
- Memory read/write policy
- Output commit policy

出处：

- OpenAI agent guide 建议工具有标准化定义，以支持 tools 与 agents 的多对多关系，并提升可发现性、复用和版本管理。  
  https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/
- MCP ToolAnnotations 提供 readOnly、destructive、idempotent、openWorld 等提示，但规范也强调这些是 hints，客户端不能基于不可信 server 的 annotations 做决策。  
  https://modelcontextprotocol.io/specification/draft/schema

洪荒时代落地：

- `SkillRuntimeContract` 保留为能力意图契约。
- `ToolExecutionContract / ToolOutputContract / ToolProjectionContract` 保留为执行资源契约。
- 新增或强化 `AgentProfileContract` 和 `WorkerContract`。
- `ResourcePolicy` 必须从 `CapabilityManifest` 导出，不再由 prompt 文本推断。

### 1.4 Fail-Closed 是默认行为

当出现缺 plan、缺 directive、未知 tool、未知 worker、缺 binding、权限不允许、validator blocked 时，不回旧链路。

出处：

- OpenAI guide 把 guardrails 定义为分层防御；高风险工具应基于风险等级触发自动暂停、额外检查或人工升级。  
  https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/
- OpenAI Agents SDK guardrails 里，tripwire 被触发时会抛出异常并停止 agent execution。  
  https://openai.github.io/openai-agents-python/guardrails/
- 本地 `docs/90` 已锁定：缺契约、缺 directive、权限不允许、绑定缺失时 fail-closed，不允许回旧 planner/runtime 执行。

洪荒时代落地：

- `PlanValidator` 拒绝后 `RuntimeControl.primary_active = False`。
- `QueryRuntime` 返回用户可见阻断说明。
- `OutputCommitGate` 不写 session/durable memory。

## 2. 外部框架借鉴矩阵

| 框架 / 协议 | 设计理念 | 洪荒时代借什么 | 不借什么 |
| --- | --- | --- | --- |
| LangGraph | 图状态、checkpoint、durable execution、interrupt/resume | `ExecutionGraph`、checkpoint-like trace、side effects 包成 task/activity | 不直接引入 LangGraph runtime |
| Temporal | Workflow 编排，Activity 执行副作用，deterministic replay | 编排/副作用分离、idempotent activity、长任务模型 | 不引入完整 Temporal 服务 |
| OpenAI Agents SDK | agents、handoffs、guardrails、tracing | agents-as-tools、guardrail/tripwire、handoff 谨慎使用 | 不允许 decentralized handoff 接管主会话 |
| Google ADK | Sequential/Parallel/Loop workflow agents，非 LLM workflow agent 决定执行顺序 | `single / bundle / fanout / loop verify` 固定拓扑 | 不共享完整 InvocationContext 污染主线程 |
| Microsoft Agent Framework | executor + typed message + workflow context | `ExecutionNode` / executor 类型化 | 不绑定 C#/.NET 实现 |
| MCP | tool schema、annotations、task support | Tool contract、risk hints、taskSupport | 不信任外部 annotation 直接授权 |
| A2A | Task/Message/Artifact/AgentCard | Worker task/result/artifact handle 协议 | 不让外部 agent 访问内部 memory/tools |
| AutoGen Core | topic/subscription、event-driven agent runtime | 后续可做事件总线 | 当前不引入 pub/sub，避免更散 |
| CrewAI Flows | event-driven flow、state persistence | 可借 flow state 与可视化 | 不采用 crew 自治协作作为主架构 |
| Dagster | asset-centric orchestration | 证据/结果 artifact lineage 思路 | 不用于对话主链 |

## 4. 架构细节、技术选择与出处

### 4.1 `TaskContract`：请求的根对象

技术选择：

- 自定义 dataclass，不引入外部 runtime。
- 借 A2A Task 的“有状态工作单元”模型。
- 借 Temporal Workflow Execution 的“一次请求一个运行实例”模型。

设计：

```python
@dataclass(slots=True)
class TaskContract:
    task_id: str
    session_id: str
    user_goal: str
    task_kind: str
    modality: str
    source_needs: list[str]
    explicit_constraints: dict[str, Any]
    candidate_refs: list[str]
    status: Literal["pending", "validated", "running", "completed", "blocked", "failed"]
```

出处：

- A2A Task 是核心工作单元，有 id、status、artifacts、history。  
  https://a2a-protocol.org/dev/specification/
- Temporal Workflow Execution 是 Workflow Definition 加输入后的一次运行实例。  
  https://docs.temporal.io/workflows
- 本地 `docs/设计原则/14-任务系统.md`：Task 是并发执行和生命周期管理核心。

落地文件：

- 新增或扩展：`backend/orchestration/models.py`
- 对接：`backend/tasks/coordinator.py`

迁移规则：

- 所有 `QueryExecutionPlan.subtask_id / bundle_item_id` 逐步映射到 `TaskContract.task_id`。
- `TaskCoordinator` 不再靠 query 文本推断 task kind，而是消费 `TaskContract`。

### 4.2 `CapabilityManifest`：资源单一事实源

技术选择：

- 保留当前 `backend/capabilities/manifest.py`。
- 继续使用 registry + binding graph。
- 增加给编排层使用的 machine-readable manifest，不只给前端 operations 页。

设计：

```text
CapabilityManifest
  skills[]
  tools[]
  agents[]
  workers[]
  binding_graph
  search_policy_map
  risk_map
```

出处：

- OpenAI guide：tools 应标准化定义，支持 tools 和 agents 的多对多关系。  
  https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/
- MCP：tool annotations 包含 read-only/destructive/idempotent/open-world 等信息，但不能盲信外部 hints。  
  https://modelcontextprotocol.io/specification/draft/schema
- 本地 `docs/设计原则/09-工具系统设计.md`、`25-架构模式总结.md`：单一 registry + layered filtering。

落地文件：

- `backend/capabilities/manifest.py`
- `backend/capabilities/validation.py`
- `backend/tools/definitions.py`
- `backend/skill_system/contracts.py`

迁移规则：

- `ResourcePolicy` 只能从 `CapabilityManifest` 和本轮 search policy 得出。
- `QueryPlanner` 不再绕过 manifest 直接选择工具。

### 4.3 `ResourcePolicy`：本轮资源授权

技术选择：

- 借 OpenAI guardrails 的 tripwire 思路。
- 借本地权限系统 deny-first / fail-closed。
- 借 MCP readOnly/destructive/idempotent/openWorld 风险维度。

设计：

```python
@dataclass(slots=True)
class ResourcePolicy:
    allowed_sources: list[str]
    allowed_skills: list[str]
    allowed_tools: list[str]
    allowed_agents: list[str]
    allowed_workers: list[str]
    denied_tools: list[str]
    denied_agents: list[str]
    high_risk_requires_confirmation: bool
    refs: dict[str, Any]
```

出处：

- OpenAI Agents SDK guardrails：tool guardrails 可在工具执行前后验证或阻断；tripwire 触发即停止。  
  https://openai.github.io/openai-agents-python/guardrails/
- OpenAI guide：高风险工具应基于风险评级触发暂停、检查或人工升级。  
  https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/
- 本地 `docs/设计原则/16-权限系统.md`：deny 优先、bypass 不是万能、熔断保护。

落地文件：

- `backend/orchestration/models.py`
- `backend/orchestration/validation.py`
- `backend/permissions/decision_pipeline.py`
- `backend/query/runtime_tools.py`

迁移规则：

- `allowed_workers` 必须进入 validator。
- `terminal / python_repl / shell / destructive` 走高风险路径。
- 未知资源默认 blocked。

### 4.4 `ExecutionGraph`：从线性 directive 升级到可验证图

技术选择：

- 借 LangGraph 的 state graph 思想，但不用 LangGraph runtime。
- 借 Microsoft Agent Framework 的 executor typed message 模型。
- 借 Google ADK 的 deterministic workflow agent 模式。

设计：

```python
@dataclass(slots=True)
class ExecutionNode:
    node_id: str
    executor_type: Literal["model", "tool", "worker", "agent"]
    action: str
    inputs: dict[str, Any]
    resource_policy_ref: str
    output_contract_ref: str
    depends_on: list[str]
    shared_channels: list[str]
    retry_policy: dict[str, Any]
    fallback: Literal["fail_closed", "clarify", "degrade"]

@dataclass(slots=True)
class ExecutionGraph:
    graph_id: str
    nodes: list[ExecutionNode]
    edges: list[dict[str, str]]
    topology: Literal["single", "sequential", "parallel", "loop"]
```

出处：

- LangGraph durable execution：workflow state checkpoint、task 包副作用、可恢复。  
  https://docs.langchain.com/oss/python/langgraph/durable-execution
- Microsoft Agent Framework：executor 处理 typed messages，可产出 output/events。  
  https://learn.microsoft.com/en-us/agent-framework/workflows/executors
- Google ADK SequentialAgent：非 LLM workflow agent 按固定顺序执行子 agent，因此执行顺序是 deterministic。  
  https://adk.dev/agents/workflow-agents/sequential-agents/

落地文件：

- 新增：`backend/orchestration/execution_graph.py`
- 修改：`backend/orchestration/runtime_adapter.py`
- 修改：`backend/query/runtime.py`

迁移规则：

- 当前 `ExecutionDirective[]` 先保持兼容。
- 新增 `execution_graph` 字段，`RuntimeControl.execution_specs` 从 graph nodes 导出。
- `bundle_execution / explicit_fanout` 不再由 `QueryRuntime` 特判，而是 graph topology。

### 4.5 `ExecutionDirective`：唯一可执行命令

技术选择：

- 保留已有 dataclass。
- 增加 `executor_type`、`task_contract_id`、`resource_policy_ref`、`output_contract_ref`。

设计：

```python
@dataclass(slots=True)
class ExecutionDirective:
    step_id: str
    task_contract_id: str
    executor_type: str
    action: str
    execution_id: str
    skill: str
    tool: str
    agent_id: str
    worker_route: str
    inputs: dict[str, Any]
    risk_tags: list[str]
    shared_channels: list[str]
    fallback: str = "fail_closed"
```

出处：

- Temporal Activity：Activity 是单一明确动作，可以长短运行，建议幂等。  
  https://docs.temporal.io/activities
- 本地 `docs/90`：`ExecutionDirective` 是 tool、worker、agent、model 执行的唯一入口。

落地文件：

- `backend/orchestration/models.py`
- `backend/orchestration/runtime_adapter.py`
- `backend/query/runtime.py`

迁移规则：

- `_stream_planned_execution(session_id, execution)` 改为 `_stream_directive_execution(session_id, spec, candidate_context)`。
- `QueryExecutionPlan` 只能作为 candidate_context，不再是执行真相。

### 4.6 Agent / Worker：bounded specialist，不接管主会话

技术选择：

- 借 OpenAI agents-as-tools / manager pattern。
- 借 A2A Task/Artifact 协议表达 worker 结果。
- 借 Google ADK Parallel/Sequential agent 拓扑，但由洪荒时代 control plane 驱动。

设计：

```python
@dataclass(slots=True)
class AgentProfile:
    agent_id: str
    role: str
    allowed_tools: list[str]
    allowed_workers: list[str]
    memory_scope: str
    isolation_mode: Literal["default", "context_only", "worktree", "remote"]
    prompt_profile: str
    output_contract: str
```

出处：

- OpenAI guide：agent 本身可以作为 orchestration tool；handoff 是单向移交，适合不需要中心 agent 综合的场景。  
  https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/
- OpenAI Agents SDK handoffs：handoff 留在单次 run 内，guardrail 适用范围要小心；需要 tool guardrails 包住工具调用。  
  https://openai.github.io/openai-agents-python/handoffs/
- A2A：Agent 可以返回 Task 做复杂处理，stream 任务状态和 artifact updates。  
  https://a2a-protocol.org/dev/specification/
- 本地 `docs/设计原则/12-Agent-系统.md`：默认隔离，选择性共享。

洪荒时代决策：

- 主会话 agent 保留最终答案所有权。
- `retrieval/pdf/structured_data` 是 bounded workers。
- 不采用 decentralized handoff 作为默认。

落地文件：

- `backend/agents/a2a_cards.py`
- `backend/query/worker_models.py`
- `backend/query/evidence_orchestrator.py`
- 新增或扩展：`backend/orchestration/agent_profiles.py`

迁移规则：

- `agent_tool_bindings` 从展示配置升级为 validator 输入。
- worker 返回必须包含 `task_status / artifact_refs / result_handle_ids / memory_policy`。

### 4.7 Tool：带契约的微服务

技术选择：

- 保留当前 `ToolDefinition`。
- 对齐 MCP tool annotations，但不直接信任外部 hints。
- 增加 `taskSupport` 概念：长任务必须挂 TaskCoordinator。

设计：

```text
ToolDefinition
  execution_contract
  resolution_contract
  output_contract
  projection_contract
  risk_tags
  read_only / destructive / idempotent / open_world
  task_support: forbidden | optional | required
```

出处：

- MCP ToolAnnotations：readOnlyHint、destructiveHint、idempotentHint、openWorldHint。  
  https://modelcontextprotocol.io/specification/draft/schema
- MCP ToolExecution：taskSupport 可表达工具是否支持或要求 task-augmented execution。  
  https://modelcontextprotocol.io/specification/draft/schema
- 本地 `docs/设计原则/09-工具系统设计.md`：工具不只是执行逻辑，而是带输入验证、权限、结果序列化的微服务。

落地文件：

- `backend/tools/contracts.py`
- `backend/tools/definitions.py`
- `backend/tools/runtime.py`
- `backend/query/runtime_tools.py`

迁移规则：

- read-only safe tool 可以直连。
- document/data/retrieval 工具优先下沉 worker。
- system_execution 工具必须 permission gate + explicit directive。

### 4.8 Memory：read policy 与 commit candidate 分离

技术选择：

- 借 LangGraph memory/context persistence 但保留自有 MemoryFacade。
- 借本地 Claude Code memory 五层模型。
- 引入 `MemoryCommitCandidate`。

设计：

```python
@dataclass(slots=True)
class MemoryPolicy:
    read_mode: str
    write_mode: str
    use_session_state: bool
    use_durable_memory: bool
    restored_candidates: list[str]
    writeback_scope: list[str]

@dataclass(slots=True)
class MemoryCommitCandidate:
    candidate_id: str
    scope: Literal["session", "durable", "agent"]
    source_task_id: str
    content_summary: str
    evidence_refs: list[str]
    allow_write: bool
```

出处：

- LangGraph durable execution：state 保存和恢复是 workflow 能恢复的基础。  
  https://docs.langchain.com/oss/python/langgraph/durable-execution
- 本地 `docs/设计原则/23-Memory系统.md`：记忆分层、相关记忆按需召回、后台提取与主 agent 写入互斥。
- 本地 `docs/90`：Memory 系统输出 `MemoryPolicy` 与状态候选，不能用历史状态覆盖当前轮目标。

落地文件：

- `backend/memory/facade.py`
- `backend/query/runtime_context_state.py`
- `backend/orchestration/output_commit.py`
- 新增：`backend/orchestration/memory_commit.py`

迁移规则：

- prompt assembly 可以读 memory。
- durable extraction 不能默认执行，必须受 `MemoryPolicy.write_mode` 和 `AnswerPolicy.memory_writeback_allowed` 控制。
- Soul/Profile 只能影响 prompt/context，不拥有执行权。

### 4.9 ContextPolicy：上下文只装配，不调度

技术选择：

- 借 LangGraph checkpoint/context 思想。
- 借本地 prompt manifest。
- 拆 static / session / volatile / skill / memory sections。

设计：

```python
@dataclass(slots=True)
class ContextPolicy:
    mode: str
    required_handles: list[str]
    evidence_budget: str
    prompt_sections: list[str]
    restore_indexes: list[str]
    compaction_mode: str
```

出处：

- LangGraph durable execution 要求 thread identifier 追踪 workflow history，并由 persistence 保存进度。  
  https://docs.langchain.com/oss/python/langgraph/durable-execution
- 本地 `docs/设计原则/04-System-Prompt-工程.md`、`07-Prompt-Cache.md`：prompt 分段、static/dynamic boundary、缓存稳定。
- 本地 `docs/设计原则/06-上下文管理.md`：上下文压缩和恢复，不应替代业务执行路径。

落地文件：

- `backend/query/prompt_manifest.py`
- `backend/query/prompt_builder.py`
- `backend/query/runtime_context_state.py`
- `backend/orchestration/models.py`

迁移规则：

- `ContextPolicy` 只能决定哪些上下文进入模型窗口。
- 不能决定 route/tool/worker。

### 4.10 OutputCommitGate：唯一写回门

技术选择：

- 借 Temporal Activity 的 idempotent side effect 思路。
- 借 LangGraph side effects 包 task 的思想。
- 保留现有 `OutputCommitGate` 并强化 policy 输入。

设计：

```python
@dataclass(slots=True)
class OutputCommitPlan:
    final_answer: dict[str, Any]
    assistant_messages: list[dict[str, Any]]
    task_summary_refs: list[dict[str, Any]]
    session_projection: dict[str, Any] | None
    memory_commit_candidates: list[dict[str, Any]]
    post_turn_actions: list[dict[str, Any]]
```

出处：

- Temporal Activity 可以包含单次写操作、批量写、读后写，并建议拆小、幂等。  
  https://docs.temporal.io/activities
- LangGraph 要求有副作用操作放入 tasks，避免恢复时重复执行。  
  https://docs.langchain.com/oss/python/langgraph/durable-execution
- 本地 `docs/90`：OutputCommit 按 `AnswerPolicy / MemoryPolicy` 写回。

落地文件：

- `backend/orchestration/output_commit.py`
- `backend/query/runtime_persistence.py`
- `backend/query/runtime.py`

迁移规则：

- `refresh_session_memory`、`schedule_durable_memory_extraction`、`append_messages` 都必须来自 commit plan。
- fallback answer 默认不写 durable memory。

### 4.11 Observability：plan / actual / diff

技术选择：

- 保留当前 `orchestration.diff`。
- 借 OpenAI Agents SDK tracing 的 run/tool/handoff 可观测性。
- 借 A2A task status / artifact update event。

设计：

```text
orchestration_plan event
runtime_control event
execution_node_start/end
worker_artifact_delta
output_commit_plan
orchestration_diff
```

出处：

- OpenAI Agents SDK 有 tracing 模块，记录 agent/handoff/tool 等 span。  
  https://openai.github.io/openai-agents-python/ref/tracing/
- A2A streaming 支持 task status update 和 artifact update。  
  https://a2a-protocol.org/dev/specification/

落地文件：

- `backend/orchestration/diff.py`
- `backend/orchestration/behavior_trace.py`
- `backend/experiments/orchestration_trace.py`
- `backend/tests/system_eval/long_runner.py`

迁移规则：

- 测试不再只看最终答案。
- 必须校验 `plan.execution_graph` 和 actual execution 是否一致。

## 8. 暂不采用的架构

### 8.1 不直接引入 Temporal

原因：

- 当前问题是“控制权未收口”，不是“缺分布式 workflow 引擎”。
- 引入 Temporal 会增加部署、worker、task queue、deterministic replay 约束。

可保留：

- Workflow / Activity 分离。
- Activity 幂等。
- 长任务 task queue 思路。

### 8.2 不直接引入 LangGraph

原因：

- 你已有 `OrchestrationPlan / RuntimeControl / TaskCoordinator / EvidenceOrchestrator`。
- 直接引入 LangGraph 会形成第二套运行时。

可保留：

- graph state。
- checkpoint。
- side effect task。

### 8.3 不采用 AutoGen pub/sub 作为当前主线

原因：

- 当前单轮控制面还未固定。
- pub/sub 会让调度权更分散。

可保留：

- 后续多 agent 事件总线。

### 8.4 不采用 decentralized handoff

原因：

- 洪荒时代需要主会话收束最终答案。
- PDF/RAG/结构化 agent 不应接管用户会话。

可保留：

- agents-as-tools。
- bounded worker。

## 9. 与本地设计原则的映射

| 本地原则 | 本文架构落点 |
| --- | --- |
| 单一事实源 | `OrchestrationPlan`、`CapabilityManifest`、`ResourcePolicy` |

| 输出层消费 typed result | `OutputCommitGate` + `AnswerPolicy` |
| 测试验证计划与实际差异 | `orchestration_diff` |

## 10. 最终目标完成态

完成后，洪荒时代（mythical ageent）的一句话架构应该是：

> 一个以 `OrchestrationPlan` 为唯一控制面、以 `CapabilityManifest` 为资源事实源、以 `ExecutionGraph` 为执行拓扑、以 `TaskContract` 管生命周期、以 `OutputCommitGate` 管写回的 agent runtime。

完成态代码特征：

- `QueryPlanner` 没有最终执行权。
- `QueryRuntime` 没有跨系统调度权。
- `RuntimeToolBridge` 不从旧 route/tool 字段推断工具范围。
- `TaskCoordinator` 不从自然语言猜任务绑定。
- `MemoryFacade` 不直接覆盖当前轮目标。
- `OutputCommitGate` 是唯一写回门。
- `orchestration_diff` 能解释每个 plan node 是否按预期执行。

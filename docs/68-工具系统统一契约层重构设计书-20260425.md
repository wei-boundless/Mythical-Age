# 68-工具系统统一契约层重构设计书-20260425

> 目的：基于当前理解层、工具层、执行层与输出收口层的真实结构，重新设计一条统一的工具契约链路。  
> 本文替代旧版 `68-工具统一契约协议设计书-20260425.md`。旧版的问题是方向正确，但还是偏“工具契约字段清单”，没有把理解层和收尾层如何对接讲透。本版重写为可实施的架构设计书。

---

## 0. 本版结论

本轮要统一的不是“工具定义字段”，而是整条链路：

```text
用户输入
  -> 理解层产出 Task Frame
  -> 工具契约层裁决 Tool Request
  -> 输入契约层解析 Tool Input
  -> Runtime 执行 Tool
  -> 输出契约层归一 Tool Result Envelope
  -> 收尾层只消费 Canonical Output
```

核心判断：

1. 理解层只负责结构化意图与显式锚点，不再长期承担工具选择细节。
2. 工具契约层是能力、输入、执行、安全、输出、投影的唯一事实源。
3. Runtime 只执行已经裁决好的 `ToolInvocation`，不再靠工具名补知识。
4. 输出收口层只消费 `ToolResultEnvelope` 与 `CanonicalOutput`，不再猜原始文本是否可展示。

## 1. 当前系统真实链路

### 1.1 理解层

相关文件：

- `backend/understanding/task_understanding.py`
- `backend/understanding/query_understanding.py`

当前理解层已经开始向结构化方向移动：

1. `task_understanding.py` 产出 `TaskSignals`
2. `TaskUnderstanding` 里已有 `capability_requests`
3. `QueryUnderstanding` 继承 `capability_requests / structural_signals`
4. `query_understanding.py` 调用 `ToolRegistry.resolve_candidate_names()` 和 `select_best()`

但当前仍有几个问题：

1. `task_understanding.py` 仍有较多 domain marker。
2. 个别路径仍直接给 `candidate_tools`，例如 `workspace_read -> read_file`。
3. `query_understanding.py` 同时做 skill scope、tool candidate、tool selection，职责偏厚。
4. 工具选择仍部分依赖 `ToolRegistry` 内部打分。

### 1.2 规划层

相关文件：

- `backend/query/planner.py`
- `backend/query/tool_input_resolver.py`
- `backend/query/binding_resolver.py`

当前 planner 在 `route == "tool"` 且有 `tool_name` 时，会立即调用 `ToolInputResolver.resolve()` 生成 `tool_input`，并把 `execution_kind` 设为 `direct_tool`。

问题在于：

1. planner 过早把“工具选择”和“输入解析”绑死。
2. `ToolInputResolver` 仍按工具名分支处理 PDF、表格、多模态、直读文件。
3. 绑定恢复、显式路径、catalog 查找、repo-root normalize 等规则没有统一归属。

### 1.3 工具层

相关文件：

- `backend/tools/contracts.py`
- `backend/tools/definitions.py`
- `backend/tools/runtime.py`
- `backend/tools/tool_registry.py`

当前已有：

1. `ToolExecutionContract`
2. `ToolContractGate`
3. `ToolDefinition`
4. `ToolRuntime`
5. `ToolRegistry`

这是一套好骨架，但还只是半成品。它主要回答“能不能执行”，没有完整回答：

1. 这个工具由什么 capability 触发
2. 输入参数从哪里来
3. 缺失输入如何澄清
4. 结果能不能直接展示
5. 结果能不能写入 task summary / session / memory
6. 结果是否需要 finalizer

### 1.4 执行与收尾层

相关文件：

- `backend/query/runtime_tools.py`
- `backend/query/runtime_output_policy.py`
- `backend/query/output_classifier.py`
- `backend/query/output_boundary.py`

当前 `RuntimeToolBridge` 已经承担了直接工具执行主链：

1. contract gate
2. permission
3. invoke
4. task ledger
5. output decision
6. PDF finalization / persistence gate

但它仍有明显的工具名分支：

1. `read_file` 的纯文本输出放行
2. `pdf_analysis` 的结果投影、持久化、finalization
3. `pdf_analysis / structured_data_analysis` 的失败文案
4. 直接工具结果是否可展示，仍由 runtime 与 classifier 共同猜

## 2. 本轮真正要修的系统性质

本轮不是单点修复，而是所有权修复。

当前工具知识散落位置：

| 知识类型 | 当前位置 | 应归属 |
| --- | --- | --- |
| 工具能力 | `definitions.py` / `tool_registry.py` / understanding | `ToolDefinition.capability` |
| 工具输入来源 | `tool_input_resolver.py` | `ToolResolutionContract` |
| 工具执行要求 | `contracts.py` | `ToolInvocationContract` |
| 工具输出可见性 | `runtime_tools.py` / `output_classifier.py` | `ToolOutputContract` |
| 工具状态投影 | `runtime_tools.py` / `runtime_output_policy.py` | `ToolProjectionContract` |
| 工具 finalizer | `runtime_tools.py` / `runtime_output_policy.py` | `ToolOutputContract.finalizer` |

正确终态：

> 工具事实只声明一次，理解层、规划层、执行层和收尾层都消费同一份契约。

## 3. 本项目必须遵守的设计原则

本设计继承以下文档约束：

- `docs/设计原则/09-工具系统设计.md`
- `docs/设计原则/12-Agent-系统.md`
- `docs/设计原则/25-架构模式总结.md`
- `docs/38-去启发式续接与显式句柄主链重构技术报告及计划书.md`
- `docs/55-工具与技能系统契约化重构初步计划书.md`
- `docs/66-Claude式输出后处理去启发式重构计划书-20260425.md`
- `docs/67-QueryRuntime分层拆解与结构坏味道排查计划书-20260425.md`

落到本轮，硬原则是：

1. 结构修复优先于关键词补丁。
2. 显式路径、显式绑定、显式句柄优先于弱语言线索。
3. 工具 raw output 默认隔离，只有契约允许的 canonical 内容能进入主线程。
4. restore / decide / execute / present 必须分层。
5. Runtime 不应成为工具私有知识的容器。
6. 收尾层只消费类型化结果，不再猜字符串角色。
7. 新工具接入应主要修改工具实现、工具定义和测试。

## 4. 推荐目标架构

## 4.1 新增核心对象：Task Frame

理解层最终产出的是 `TaskFrame`，不是工具名。

可先复用现有 `QueryUnderstanding` 字段，不必马上新建类，但语义必须按这个对象收口：

```python
TaskFrame:
    intent
    source_kind
    task_kind
    modality
    route_hint
    capability_requests
    structural_signals
    anchors
    constraints
```

理解层允许做：

1. 识别显式路径、URL、页码、文件后缀等结构锚点
2. 标注 capability，例如 `document_analysis / dataset_analysis / workspace_read`
3. 标注 modality，例如 `pdf / table / code / web / realtime`

理解层不应做：

1. 解析工具最终输入
2. 判断工具输出是否可展示
3. 按工具名补特殊规则

## 4.2 新增核心对象：Tool Invocation Request

工具契约层从 `TaskFrame` 生成 `ToolInvocationRequest`：

```python
ToolInvocationRequest:
    tool_name
    capability
    route
    unresolved_input
    anchors
    binding_context
    contract_status
```

这一层负责 decide：

1. 哪个工具满足 capability
2. 当前 anchors / bindings 是否足够
3. 如果不足，是 clarify、deny、fallback 还是 route 到 agent

这一步应由 `ToolRegistry` 或新的 `ToolInvocationPlanner` 负责，不应放在 `task_understanding.py`。

## 4.3 扩展 ToolDefinition 为唯一事实源

建议 `ToolDefinition` 最终包含五组契约：

```python
ToolDefinition:
    name
    module
    factory
    capability_contract
    invocation_contract
    resolution_contract
    output_contract
    projection_contract
    safety_contract
```

### Capability Contract

回答“这个工具能处理什么”：

```python
ToolCapabilityContract:
    capability_tags
    supported_modalities
    route_hints
    safe_for_auto_route
```

### Invocation Contract

回答“什么情况下可以执行”：

```python
ToolInvocationContract:
    required_inputs
    required_bindings
    owner_scope
    missing_input_behavior
    missing_binding_behavior
    context_policy
```

当前 `ToolExecutionContract` 可演进为这个对象，短期保留旧名兼容。

### Resolution Contract

回答“输入从哪里来”：

```python
ToolResolutionContract:
    input_sources
    path_field
    path_policy
    catalog
    binding_map
    filename_fallback
    normalize_rules
```

示例：

| 工具 | path policy | catalog | fallback |
| --- | --- | --- | --- |
| `pdf_analysis` | knowledge-relative | pdf catalog | filename fallback |
| `structured_data_analysis` | knowledge-relative | dataset catalog | filename fallback |
| `read_file` | repo-root-relative | none | filename fallback 可选 |
| `analyze_multimodal_file` | repo-root-relative | multimodal adapter | supported-file scan |

### Output Contract

回答“结果怎么进入可见层”：

```python
ToolOutputContract:
    envelope_kind
    display_mode
    canonical_fields
    fallback_behavior
    finalization_policy
    persistence_policy
```

关键 display mode：

1. `verbatim_text`：原文可直接展示，例如 `read_file`
2. `summary_text`：工具摘要可展示，例如天气、金价
3. `canonical_structured`：结构化结果经 formatter 展示
4. `finalize_then_display`：先走 finalizer，例如 PDF
5. `artifact_only`：只产生 artifact，不直接回答
6. `raw_debug_only`：只进 debug，不进入用户可见答案

### Projection Contract

回答“哪些结果能进入主线程状态”：

```python
ToolProjectionContract:
    task_summary_policy
    result_ref_policy
    main_context_fields
    memory_projection_policy
```

这一步很重要。PDF 的污染问题，本质就是 projection 没有足够契约化。

## 5. 固定执行流

本轮最终执行流应固定为：

```text
1. Understanding
   input: user message, memory intent
   output: TaskFrame

2. Tool Arbitration
   input: TaskFrame, tool definitions, skill scope, active bindings
   output: ToolInvocationRequest or non-tool route

3. Input Resolution
   input: ToolInvocationRequest, ToolResolutionContract, binding context
   output: ToolInvocation with final tool_input

4. Contract Gate
   input: ToolInvocation, ToolInvocationContract, permission scope
   output: allow / clarify / deny / fallback

5. Tool Execution
   input: ToolInvocation
   output: raw_tool_output

6. Output Adapter
   input: raw_tool_output, ToolOutputContract
   output: ToolResultEnvelope

7. Finalization / Projection
   input: ToolResultEnvelope, ToolProjectionContract
   output: CanonicalOutput, task refs, context refs

8. Runtime Done Event
   input: CanonicalOutput
   output: done.content, session content, debug trace
```

固定边界：

1. understanding 不执行。
2. registry 不解析具体文件路径。
3. resolver 不选择工具。
4. runtime 不猜输出文本类型。
5. output policy 不读取工具 raw dump，除非 output contract 明确允许。

## 6. 阶段计划

## Phase 0：冻结旧问题口径

目标：

1. 确认 `68` 为正式设计书。
2. 后续实施只按本文推进。
3. 记录当前已有的 `contracts.py` 草稿状态，实施前先复核。

不做：

1. 不继续扩散代码改动。
2. 不新增临时工具特判。

完成标准：

1. 本文写入完成。
2. 后续 implementation 以本文为准。

## Phase 1：定义层补齐，不改变行为

目标：

1. 在 `backend/tools/contracts.py` 稳定新增契约对象。
2. 在 `backend/tools/definitions.py` 为现有工具补全契约声明。
3. 保持旧字段兼容，不破坏现有测试。

涉及文件：

- `backend/tools/contracts.py`
- `backend/tools/definitions.py`
- `backend/tools/runtime.py`
- `backend/tools/tool_registry.py`
- `backend/TOOLS_REGISTRY.json`

完成标准：

1. 每个工具都有 capability / invocation / resolution / output / projection 信息。
2. registry payload 能导出这些字段。
3. 旧 direct tool 行为不变。

## Phase 2：输入解析契约化

目标：

1. `ToolInputResolver` 改为消费 `ToolResolutionContract`。
2. 显式路径、binding、catalog、fallback 的规则从 `definitions.py` 来。
3. resolver 核心路径不再按工具名分支。

涉及文件：

- `backend/query/tool_input_resolver.py`
- 可新增 `backend/query/tool_resolution.py`
- `backend/tests/query_planner_regression.py`
- `backend/tests/query_runtime_route_guard_regression.py`

完成标准：

1. PDF、表格、workspace read、多模态文件解析都走同一管线。
2. 旧路径解析回归全部通过。
3. `ToolInputResolver` 不再是工具私有知识容器。

## Phase 3：工具仲裁契约化

目标：

1. `ToolRegistry.select_best()` 从打分式改为约束式。
2. capability / modality / route / owner satisfiable 共同决定候选。
3. `query_understanding.py` 不再承担过多工具选择细节。

涉及文件：

- `backend/tools/tool_registry.py`
- `backend/understanding/query_understanding.py`
- `backend/query/planner.py`
- `backend/tests/tool_registry_regression.py`
- `backend/tests/task_understanding_regression.py`

完成标准：

1. 没有工具名加分表。
2. auto-route 只选择契约满足的工具。
3. `safe_for_auto_route=False` 的工具不会被自动选中，除非有显式授权入口。

## Phase 4：输出 envelope 化

目标：

1. 新增或落地 `ToolResultEnvelope`。
2. `RuntimeToolBridge` 把 raw output 先转 envelope。
3. `build_direct_tool_output_decision()` 消费 output contract，不再按工具名放行。

涉及文件：

- `backend/query/runtime_tools.py`
- 可新增 `backend/query/tool_output_adapter.py`
- `backend/query/runtime_output_policy.py`
- `backend/query/output_classifier.py`
- `backend/tests/query_runtime_route_guard_regression.py`

完成标准：

1. `read_file` 通过 `display_mode=verbatim_text` 直出。
2. 表格 raw dump 不直出。
3. PDF raw dump 不直出。
4. 天气、金价等 canonical 文本仍稳定展示。

## Phase 5：投影与收尾层契约化

目标：

1. PDF / structured / read_file 等工具的 task summary、result ref、main context 投影由 `ProjectionContract` 决定。
2. `runtime_output_policy.py` 不直接读取工具名决定是否投影。
3. canonical output 成为 session 和记忆的唯一入口。

涉及文件：

- `backend/query/runtime_tools.py`
- `backend/query/runtime_output_policy.py`
- `backend/query/runtime_persistence.py`
- `backend/runtime/session_store.py`
- `backend/tests/pdf_agent_runtime_regression.py`

完成标准：

1. 降级 PDF 不进入主线程 summary。
2. raw dump 不进入 session 和记忆。
3. finalization 只对 contract 标记需要的 envelope 触发。

## Phase 6：旧链路清理

目标：

1. 删除 resolver 旧工具名分支。
2. 删除 runtime 旧工具输出特判。
3. 删除 registry 工具名加分表。
4. 更新测试为契约断言。

完成标准：

1. 新工具接入主要只改工具实现、`definitions.py` 和测试。
2. 工具事实不再散落多层。

## 7. 迁移规则

### 7.1 Shadow First

先声明契约，不立刻切主链。  
契约字段进入 `definitions.py` 后，测试先验证声明完整性。

### 7.2 One Consumer At A Time

每次只让一个消费者切到新契约：

1. resolver
2. registry
3. runtime output
4. projection / persistence

这样便于定位回归。

### 7.3 No Hidden Fallback

旧逻辑可以短期保留，但必须满足：

1. 有明确删除阶段
2. 新链路有测试覆盖
3. trace/debug 能看出当前使用的是新契约还是旧路径

### 7.4 Rollback Boundary

允许回滚实现，不允许回滚原则。

可回滚：

1. 某个 consumer 是否读取新契约
2. 某个 adapter 的实现
3. 某条测试的迁移顺序

不可回滚：

1. 工具事实单一来源
2. 输出 raw 默认隔离
3. 理解层不承担工具私有知识
4. runtime 不继续堆工具特判

## 10. 必须避免的反模式

1. 为某个工具在 understanding 加关键词规则。
2. 在 resolver 新增 `if tool_name == ...`。
3. 在 runtime 新增“某工具 raw string 可以展示”的特例。
4. 把 output contract 藏在工具实现内部。
5. 让 PDF 降级结果进入 task summary / memory。
6. 用长场景某个失败样本倒逼恢复启发式补丁。
7. 在迁移期长期保留双轨且没有删除计划。

## 11. 实施入口建议

第一步不要直接改 runtime。  
推荐从定义层开始：

1. 稳定 `contracts.py` 中已经出现的契约草稿。
2. 扩展 `ToolDefinition`，但先保持旧字段兼容。
3. 给全部工具补齐契约声明。
4. 补 registry payload 回归。
5. 再切 `ToolInputResolver`。

理由：

1. 定义层是单一事实源的根。
2. 先改 resolver 比先改 runtime 风险低。
3. output envelope 要等输入契约稳定后再做，避免两条大链路同时震动。

## 12. 最终判断

这次重构真正要达成的状态是：

> 工具不再由理解层、resolver 和 runtime 分别“认识一遍”。  
> 工具只在定义层被认识一次，其余层只消费契约。

当这个状态成立时，新增普通工具的成本应回到：

1. 写工具实现
2. 写工具定义
3. 写工具测试

而不是继续让整条查询主链都跟着改。

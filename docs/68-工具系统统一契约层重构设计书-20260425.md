# 68-工具与Skills双链路统一契约层重构设计书-20260425

> 目的：基于当前理解层、skills 系统、工具层、执行层与输出收口层的真实结构，重新设计一条统一的能力控制面与工具执行面。  
> 本文替代旧版 `68-工具系统统一契约层重构设计书-20260425.md`。旧版只把 `skill scope` 当成工具仲裁的辅助输入，没有把 skills 作为一条并行控制链纳入整体设计，因此仍可能留下第二套 route authority。

---

## 0. 本版结论

应该一次性把 skills 与 tools 放进同一份设计里。

原因很简单：如果只重构工具系统，而 skills 仍然可以单独影响路由、候选工具、allowed tools、prompt 注入和运行时权限，那么系统会保留两套控制面：

1. `Tool Contract` 控制工具是否可执行。
2. `Skill Registry / Skill Prompt / Skill Scope` 控制模型看到什么、候选工具怎么收窄、runtime 允许什么。

这会重新制造你一直反对的结构问题：同一个能力事实被理解层、skill 层、tool 层和 runtime 各认识一遍。

本轮正确切法是：

```text
User Input
  -> TaskFrame
  -> SkillPolicyFrame（可选控制面，不直接执行）
  -> ToolInvocationRequest
  -> ToolResolutionContract
  -> ToolExecution
  -> ToolResultEnvelope
  -> CanonicalOutput
```

一句话定义边界：

> Skill 是能力、提示、作用域和上下文注入层；Tool 是输入解析、权限校验、执行、输出归一和投影层。  
> Skill 可以约束工具，但不能越过工具契约层直接决定执行真相。

硬结论：

1. skills 必须纳入 68，而不是另开一条无约束旁路。
2. Skill 不应再是 route authority，只能是 `policy / prompt / scope adapter`。
3. 工具定义仍是执行事实的唯一来源。
4. `allowed_tools: list[str]` 必须升级为类型化 `ToolScope / SkillToolScope`。
5. 理解层产出结构化 `TaskFrame`，skill 和 tool 共同消费结构信号，但不能各自做一套关键词裁决。
6. Runtime 只执行通过 `ToolContractGate + PermissionService + ToolScope` 的 `ToolInvocation`。
7. 输出层只消费 `ToolResultEnvelope / CanonicalOutput`，不消费 skill prompt、raw tool dump 或过程文本。

## 1. 当前系统真实链路

### 1.1 理解层

相关文件：

- `backend/understanding/task_understanding.py`
- `backend/understanding/query_understanding.py`

当前理解层已经开始向结构化方向移动：

1. `task_understanding.py` 产出 `TaskUnderstanding`
2. `TaskUnderstanding` 里已有 `capability_requests / structural_signals`
3. `QueryUnderstanding` 继承 `capability_requests / structural_signals`
4. `query_understanding.py` 调用 `ToolRegistry.resolve_candidate_names()` 和 `select_best()`

但当前仍有问题：

1. `task_understanding.py` 仍有 domain marker 和若干 preferred skill/tool 倾向。
2. `query_understanding.py` 同时做 skill 匹配、tool candidate、tool selection，职责偏厚。
3. skill 命中后会把 `candidate_tools` 直接替换为 `allowed_tool_scope()`，等价于让 skill scope 参与工具候选裁决。
4. `candidate_tools` 仍可能成为多层传递的弱事实，后续层很难知道它来自 TaskFrame、SkillScope 还是 ToolRegistry。

### 1.2 Skills 系统

相关文件：

- `backend/skill_system/registry.py`
- `backend/tools/skills_scanner.py`
- `backend/SKILLS_REGISTRY.json`
- `backend/SKILLS_SNAPSHOT.md`

当前 skills 系统包含两组视图：

1. `SkillPromptView`
   - 面向模型可见 prompt
   - 包含 name/title/capability/use_when/output_rule
2. `SkillRuntimeContract`
   - 面向 runtime 控制
   - 包含 allowed_tools、supported_modalities、supported_task_kinds、supported_source_kinds、capability_tags、preferred_route、forbidden_routes、routing_hints、activation_policy、context_mode、route_authority、reference_paths

这是一个好雏形，但当前问题在于：

1. `SkillRuntimeContract.allowed_tools` 仍是裸 `list[str]`。
2. `SkillRegistry.match_for_query()` 仍在做 skill 分数选择。
3. `_score_skill()` 虽然主要基于结构字段，但仍是本地加权选择器。
4. `_build_skill_use_when()` 和 scanner 中的 use_when 生成仍会把 source/modality 映射成自然语言规则。
5. skills 同时承担 prompt 选择、工具 scope、route preference，边界还不够硬。

### 1.3 工具层

相关文件：

- `backend/tools/contracts.py`
- `backend/tools/definitions.py`
- `backend/tools/runtime.py`
- `backend/tools/tool_registry.py`

当前已有：

1. `ToolExecutionContract`
2. `ToolContractGate`
3. `ToolDefinition`
4. `ToolRuntime`
5. `ToolRegistry`

这套骨架方向正确，但还没有完整覆盖：

1. capability 如何从 TaskFrame 匹配到工具
2. 输入参数从哪里来
3. 缺失输入如何澄清
4. 结果能否直接展示
5. 结果能否写入 task summary / session / memory
6. 结果是否需要 finalizer
7. skill scope 如何作为 typed policy 进入权限和契约层

### 1.4 Runtime 与收尾层

相关文件：

- `backend/query/runtime_tools.py`
- `backend/query/runtime_output_policy.py`
- `backend/query/runtime_persistence.py`
- `backend/query/output_classifier.py`
- `backend/query/output_boundary.py`

当前 runtime 已拆出若干 bridge/policy，这是好方向。但 direct tool 链仍能看到：

1. `skill_allowed_tool_scope(execution.active_skill)` 返回裸 list。
2. `PermissionService.can_invoke_tool(... allowed_tools=list)` 仍消费字符串列表。
3. `ToolContractGate.evaluate(... skill_allowed_tools=list)` 仍消费字符串列表。
4. `runtime_tools.py` 中仍按工具名处理 PDF 输出投影、失败文案、persist gate。

这说明当前不是“没有契约”，而是契约还没有贯穿 skill/tool/runtime/output 全链。

## 2. 本轮真正要修的系统性质

本轮不是把工具定义字段补全，也不是给 skills 再加几个 metadata。

本轮要修的是：

> 能力控制面和执行面必须只有一条主链，不能让 skill、tool、runtime、output 各自拥有一份私有判断。

当前事实散落如下：

| 知识类型 | 当前位置 | 应归属 |
| --- | --- | --- |
| 用户任务结构 | `task_understanding.py` / `query_understanding.py` | `TaskFrame` |
| skill 是否适用 | `SkillRegistry.match_for_query()` | `SkillPolicyResolver`，只消费结构字段 |
| skill 允许工具 | `allowed_tools: list[str]` | `SkillToolScope` |
| 工具能力 | `definitions.py` / `tool_registry.py` / understanding | `ToolCapabilityContract` |
| 工具输入来源 | `tool_input_resolver.py` | `ToolResolutionContract` |
| 工具执行要求 | `contracts.py` | `ToolInvocationContract` |
| 工具权限作用域 | runtime list 传参 | `ToolScope` |
| 工具输出可见性 | `runtime_tools.py` / `output_classifier.py` | `ToolOutputContract` |
| 工具状态投影 | `runtime_tools.py` / `runtime_output_policy.py` | `ToolProjectionContract` |
| skill prompt 注入 | prompt builder / registry | `SkillPromptView`，不参与执行事实 |

正确终态：

> Skill 只影响“模型如何理解能力”和“工具作用域如何收敛”；Tool 只负责“能否执行、如何输入、如何输出、如何投影”。二者通过 typed contract 对接，不通过关键词、裸字符串列表或隐式候选覆盖对接。

本设计继承以下文档约束：

- `docs/设计原则/04-System-Prompt-工程.md`
- `docs/设计原则/05-对话循环.md`
- `docs/设计原则/07-Prompt-Cache.md`
- `docs/设计原则/09-工具系统设计.md`
- `docs/设计原则/12-Agent-系统.md`
- `docs/设计原则/24-Skill-Plugin开发实战.md`
- `docs/设计原则/25-架构模式总结.md`
- `docs/55-工具与技能系统契约化重构初步计划书.md`
- `docs/64-子任务分解去启发式与显式编排改造计划书-20260424.md`
- `docs/66-Claude式输出后处理去启发式重构计划书-20260425.md`
- `docs/67-QueryRuntime分层拆解与结构坏味道排查计划书-20260425.md`

落到本轮，约束是：

1. 结构修复优先于关键词补丁。
2. 显式路径、显式绑定、显式句柄优先于弱语言线索。
3. 默认隔离，显式共享。
4. restore / decide / execute / present 必须分层。
5. Runtime 不应成为 skill 或 tool 私有知识的容器。
6. Skill prompt 不等于执行权限。
7. Tool raw output 默认隔离，只有契约允许的 canonical 内容能进入主线程。
8. 新工具接入应主要修改工具实现、工具定义和测试。
9. 新 skill 接入应主要修改 `SKILL.md`、skill registry snapshot 和测试，不应修改 runtime 主链。

### 3.1 原则到本设计的强制映射

| docs 原则 | 对本设计的硬约束 | 验收点 |
| --- | --- | --- |
| System Prompt 分段与动态边界 | Skill prompt 必须作为可控动态段或 attachment 注入，不能混进静态核心 prompt | prompt hash 变化能定位到 skill section |
| Prompt Cache 字节稳定 | Tool schema、skill prompt view、active skill block 都必须有稳定排序与 session latch | 同一 session 内无关 turn 不改变工具/skill prompt 字节 |
| 对话循环状态机 | skills/tools 调度必须是显式 scheduler 阶段，不允许理解层、planner、runtime 各自调度 | trace 中能看到唯一 dispatch decision |
| 工具注册表单一来源 | 工具能力、输入、输出、投影只来自 `ToolDefinition` | 新工具不改 understanding/runtime 主链 |
| Agent 默认隔离、显式共享 | Skill context 默认只影响当前 turn；跨 turn 共享必须通过显式 scope/ref | skill 不把 raw tool result 写入主线程记忆 |
| 安全防线 fail-closed | SkillScope 只能收窄工具集，不能扩权或覆盖 deny | permission deny 优先于 skill allow |
| 输出 canonical truth | skill prompt、tool raw output、过程附件都不能成为 session assistant truth | session 只存 canonical output |

这张表是实施时的约束，不是解释性文字。后续代码改造如果无法对应到某一行验收点，就不应纳入本轮。

### 3.2 Prompt 泄露与执行泄露审查结论

当前需要重点防的不是传统意义的 prompt injection，而是三类执行事实泄露：

1. `SkillPromptView` 泄露为执行事实  
   风险：模型看到“这个 skill 可用”，系统随后把这段文字当成 route/permission 依据。  
   设计约束：prompt view 只能进入模型上下文，不能进入 permission、contract、projection。

2. `allowed_tools` 泄露为权限事实  
   风险：skill 声明 allowed tools 后，runtime 直接把裸 list 传给 permission/contract，导致 scope 来源和信任级别丢失。  
   设计约束：必须用 `SkillToolScope`，并且只能收窄已有权限。

3. tool raw output 泄露为 prompt / memory / session truth  
   风险：PDF、表格、文件读取的 raw dump 被拼回 prompt 或 session，后续 turn 被当作用户可见事实。  
   设计约束：raw output 只进入 `ToolResultEnvelope.raw_debug`，用户可见和持久化只消费 canonical/projection 字段。

本轮必须新增一个审查点：

```text
Prompt Exposure Audit:
  SkillPromptView -> model-visible only
  SkillRuntimePolicy -> scheduler/permission only
  ToolDefinition -> scheduler/resolver/runtime only
  ToolResultEnvelope.raw -> debug only
  CanonicalOutput -> visible/session/memory only
```

任何跨越上面边界的字段传递都应视为执行泄露。

## 4. Claude Code 参考结论

本轮参考本地 docs 中对 Claude Code 的拆解，重点不是照搬实现，而是借不变量。

### 4.1 Tool 侧不变量

Claude Code 的工具系统体现了几条可迁移原则：

1. 工具是完整微服务，不只是函数：
   - input schema
   - validate input
   - check permissions
   - read-only / destructive / concurrency safe
   - result serialization
   - UI/render contract
2. 注册表是单一来源。
3. 安全默认 fail-closed：
   - 并发默认不安全
   - 读写默认保守
4. 工具池经过注册、启用、权限过滤后才进入模型或执行层。

迁移到本项目：

1. `ToolDefinition` 必须成为能力、输入、执行、安全、输出、投影的唯一事实源。
2. `ToolRegistry` 不应是关键词打分器，而应是契约满足性过滤器。
3. `ToolContractGate` 与 `PermissionService` 应消费 typed scope，不消费裸 list。

### 4.2 Agent / Skill 侧不变量

Claude Code 的 Agent/Skill 机制体现了几条可迁移原则：

1. Agent 默认隔离，显式共享。
2. Skill 是 prompt + 行为约束 + allowed tools 的扩展点。
3. Skill 可以 inline，也可以 fork 到独立 agent。
4. 多来源 skill/agent 通过 registry 聚合和优先级去重。
5. Skill/Plugin 来源涉及信任边界，远程或不可信来源不能执行同等能力。

迁移到本项目：

1. Skill 不能和 Tool 平级争夺执行权。
2. Skill 应成为 `SkillPolicyFrame`：
   - prompt view
   - tool scope
   - context policy
   - source trust
   - activation policy
3. Skill 的 allowed tools 应进入权限约束，但不能跳过工具自己的 invocation/resolution/output contract。
4. Skill 匹配不能再依赖自然语言 hint/example/关键词加权。

### 4.3 不照搬的内容

本项目暂不照搬：

1. Claude Code 完整 Tool UI render 协议。
2. 完整 MCP tool_reference 延迟加载。
3. 完整 Agent fork 执行框架。
4. Hook 体系。

原因：

1. 当前项目首要问题是契约与边界，不是 CLI UI。
2. 工具数量还没有大到必须引入 ToolSearch。
3. 引入完整 fork skill 会显著扩大改造面。

## 5. 目标架构

## 5.1 TaskFrame：理解层唯一输出

理解层最终产出的是 `TaskFrame`，不是 skill 名，也不是 tool 名。

短期可复用 `QueryUnderstanding` 字段，长期建议显式建模：

```python
TaskFrame:
    intent
    source_kind
    task_kind
    modality
    route_hint
    capability_requests
    structural_signals
    anchors
    constraints
    explicit_handles
```

理解层允许做：

1. 识别显式路径、URL、页码、文件后缀等结构锚点。
2. 标注 capability，例如 `document_analysis / dataset_analysis / workspace_read`。
3. 标注 modality，例如 `pdf / table / code / web / realtime`。
4. 标注是否存在显式用户指定。

理解层不应做：

1. 解析工具最终输入。
2. 判断工具输出是否可展示。
3. 按工具名补特殊规则。
4. 因为词面像某个场景就强塞 preferred skill。

## 5.2 SkillPolicyFrame：skills 的正确位置

新增或语义化现有 `SkillRuntimeContract`：

```python
SkillPolicyFrame:
    skill_name
    prompt_view
    activation_policy
    context_mode
    tool_scope
    supported_capabilities
    supported_modalities
    supported_task_kinds
    supported_source_kinds
    source_trust
    reference_paths
```

Skill 可以做：

1. 提供模型可见指导。
2. 提供能力说明。
3. 约束工具作用域。
4. 声明上下文模式，如 inline / isolated。
5. 声明参考资料路径。

Skill 不可以做：

1. 直接选择工具输入。
2. 直接允许工具越过 contract gate。
3. 直接决定 raw output 是否可见。
4. 直接决定 session/memory 投影。
5. 作为 route authority 覆盖 TaskFrame。

### Skill Prompt View 与 Skill Runtime Policy 必须分离

```python
SkillDefinition:
    prompt_view: SkillPromptView
    runtime_policy: SkillRuntimePolicy
```

`prompt_view` 面向模型，只能回答：

1. 这个能力是什么
2. 什么时候适合使用
3. 输出风格约束是什么

`runtime_policy` 面向系统，只能回答：

1. 这个 skill 允许哪些 tool scope
2. 是否允许 inline / isolated
3. 来源是否可信
4. 作用域是否可进入权限层

这两者不能混在一起。模型看见的文字不能成为执行权限。

## 5.3 ToolScope / SkillToolScope：替代裸 allowed_tools

新增类型化 scope：

```python
ToolScope:
    source: Literal["global", "skill", "agent", "session", "explicit_user"]
    allowed_tools: frozenset[str]
    denied_tools: frozenset[str]
    capability_constraints: frozenset[str]
    trust_level: Literal["system", "project", "user", "external"]
    reason: str

SkillToolScope(ToolScope):
    skill_name: str
    activation_policy: str
    context_mode: str
```

替代以下旧接口：

1. `SkillDefinition.allowed_tool_scope() -> list[str]`
2. `PermissionService.can_invoke_tool(... allowed_tools=list)`
3. `ToolContractGate.evaluate(... skill_allowed_tools=list)`
4. `RuntimeToolBridge._skill_allowed_tool_scope(...) -> list[str]`

目标不是把 list 包一层皮，而是让权限层知道：

1. scope 从哪里来
2. 为什么存在
3. 是否可信
4. 是硬约束还是建议约束
5. 是否可以与 session/global scope 合并

## 5.4 ToolDefinition：执行事实唯一来源

建议 `ToolDefinition` 最终包含：

```python
ToolDefinition:
    name
    module
    factory
    capability_contract
    invocation_contract
    resolution_contract
    output_contract
    projection_contract
    safety_contract
```

### Capability Contract

回答“这个工具能处理什么”：

```python
ToolCapabilityContract:
    capability_tags
    supported_modalities
    supported_routes
    safe_for_auto_route
```

### Invocation Contract

回答“什么情况下可以执行”：

```python
ToolInvocationContract:
    required_inputs
    required_bindings
    owner_scope
    missing_input_behavior
    missing_binding_behavior
    context_policy
    read_only
    concurrency_safe
```

当前 `ToolExecutionContract` 可演进为这个对象，短期保留旧名兼容。

### Resolution Contract

回答“输入从哪里来”：

```python
ToolResolutionContract:
    input_sources
    path_field
    path_policy
    catalog
    binding_map
    filename_fallback
    normalize_rules
```

### Output Contract

回答“结果怎么进入可见层”：

```python
ToolOutputContract:
    envelope_kind
    display_mode
    canonical_fields
    fallback_behavior
    finalization_policy
    persistence_policy
```

关键 display mode：

1. `verbatim_text`：原文可直接展示，例如 `read_file`
2. `summary_text`：工具摘要可展示，例如天气、金价
3. `canonical_structured`：结构化结果经 formatter 展示
4. `finalize_then_display`：先走 finalizer，例如 PDF
5. `artifact_only`：只产生 artifact，不直接回答
6. `raw_debug_only`：只进 debug，不进入用户可见答案

### Projection Contract

回答“哪些结果能进入主线程状态”：

```python
ToolProjectionContract:
    task_summary_policy
    result_ref_policy
    main_context_fields
    memory_projection_policy
```

PDF / dataset 软污染问题，本质就是 projection 没有足够契约化。

## 5.5 Prompt Assembly Contract：防止 skills/tools 泄露到 prompt

必须把“给模型看的内容”和“给 runtime 执行的内容”拆成两条显式通道。

```python
PromptAssemblyContract:
    static_system_sections
    session_dynamic_sections
    turn_attachments
    skill_prompt_blocks
    tool_schema_blocks
    cache_policy
    exposure_policy
```

### 模型可见内容

允许进入模型上下文：

1. `SkillPromptView.render_block()`
2. 工具 schema / capability 的模型可见描述
3. 当前 turn 的必要 context refs
4. 已经 canonical 化的历史摘要

禁止进入模型上下文：

1. `SkillRuntimePolicy`
2. `ToolScope / PermissionDecision`
3. `ToolInvocationContract` 的内部策略字段
4. raw tool output
5. contract gate debug
6. memory projection debug

### Runtime 可见内容

允许进入 runtime：

1. `TaskFrame`
2. `SkillPolicyFrame.runtime_policy`
3. `ToolDefinition`
4. `ToolScope / PermissionDecision`
5. `ToolResultEnvelope`

禁止 runtime 从 prompt 文本反推：

1. 当前 skill 是否激活
2. 当前工具是否允许执行
3. 当前 raw output 是否可见
4. 当前结果是否可入库

### Prompt Cache 约束

参考 `docs/设计原则/04-System-Prompt-工程.md` 与 `07-Prompt-Cache.md`，本项目的 prompt 组装应遵守：

1. static core prompt 不包含 active skill。
2. skill prompt block 是 session/turn 动态段。
3. active skill block 排序稳定。
4. tool schema 序列化稳定。
5. 若某个 skill block 每 turn 重新计算，必须标记为 `DANGEROUS_uncached_skill_section` 或等价机制，并写明原因。
6. prompt/debug trace 需要记录本轮注入了哪些 skill blocks，但不能把 runtime policy 展示给模型。

验收问题：

```text
如果一个 skill 只改变 runtime scope，不改变 prompt view，system prompt 是否仍保持字节稳定？
如果一个 skill 只改变 prompt view，不改变 runtime scope，permission 是否不变？
如果一个 tool raw output 很长，它是否只能通过 envelope/projection 进入后续上下文？
```

答案必须都是“是”。

## 5.6 CapabilityDispatchScheduler：skills 与 tools 的唯一调度层

需要新增一个显式调度层，避免 understanding、skill registry、tool registry、planner、runtime 分别调度一次。

建议对象：

```python
CapabilityDispatchScheduler:
    resolve(task_frame, registries, context) -> DispatchPlan
```

输入：

```python
DispatchInput:
    task_frame
    active_bindings
    session_tool_scope
    skill_registry
    tool_registry
    permission_context
```

输出：

```python
DispatchPlan:
    route
    task_frame
    skill_policy: SkillPolicyFrame | None
    effective_tool_scope: ToolScope
    tool_candidates: list[ToolCandidate]
    selected_tool_request: ToolInvocationRequest | None
    prompt_exposure: PromptExposurePlan
    reasons: list[str]
```

### 调度顺序

固定顺序：

```text
1. Consume TaskFrame
2. Resolve SkillPolicyFrame
3. Build effective ToolScope
4. Filter ToolDefinition by capability/modality/route/binding/scope
5. Produce ToolInvocationRequest or non-tool route
6. Produce PromptExposurePlan
7. Runtime only executes DispatchPlan
```

调度层允许做：

1. 根据结构字段选择 active skill。
2. 合并 global/session/skill scopes。
3. 过滤 contract 不满足的工具。
4. 生成 prompt exposure plan。
5. 在 trace 中记录每一步原因。

调度层不允许做：

1. 解析最终 tool input。
2. 执行工具。
3. 根据 skill prompt 文本选择工具。
4. 根据自然语言 hint/example 给工具加分。
5. 让 skill 覆盖 permission deny。
6. 决定 tool output 是否可展示。

### DispatchPlan 是 planner/runtime 的交接物

Planner 不应再同时接收 `candidate_tools / skill_name / tool_name / allowed_tools list` 这些松散字段。

目标是：

```text
QueryUnderstanding -> TaskFrame
CapabilityDispatchScheduler -> DispatchPlan
Planner -> ExecutionPlan
Runtime -> Execute ExecutionPlan
```

这可以防止当前的漂移：

1. understanding 先猜 candidate tools
2. skill registry 再覆盖 candidate tools
3. tool registry 再 select best
4. planner 再 resolve input
5. runtime 再拿 skill allowed list 判断 permission

这些必须收敛为一次 dispatch。

## 6. 固定执行流

最终执行流固定为：

```text
1. Understanding
   input: user message, memory intent
   output: TaskFrame

2. Capability Dispatch
   input: TaskFrame, skill registry, tool registry, permission context, active bindings
   output: DispatchPlan

3. Prompt Exposure Planning
   input: DispatchPlan
   output: PromptExposurePlan

4. Tool Arbitration
   input: DispatchPlan, ToolDefinition registry, effective ToolScope
   output: ToolInvocationRequest or non-tool route

5. Input Resolution
   input: ToolInvocationRequest, ToolResolutionContract, binding context
   output: ToolInvocation with final tool_input

6. Contract + Permission Gate
   input: ToolInvocation, ToolInvocationContract, ToolScope
   output: allow / clarify / deny / fallback

7. Tool Execution
   input: ToolInvocation
   output: raw_tool_output

8. Output Adapter
   input: raw_tool_output, ToolOutputContract
   output: ToolResultEnvelope

9. Finalization / Projection
   input: ToolResultEnvelope, ToolProjectionContract
   output: CanonicalOutput, task refs, context refs

10. Runtime Done Event
   input: CanonicalOutput
   output: done.content, session content, debug trace
```

固定边界：

1. understanding 不执行。
2. scheduler 只 decide，不 execute。
3. skill 不执行工具。
4. skill scope 不等于 permission allow。
5. prompt exposure plan 不等于 runtime policy。
6. registry 不解析具体文件路径。
7. resolver 不选择工具。
8. runtime 不猜输出文本类型。
9. output policy 不读取工具 raw dump，除非 output contract 明确允许。

## 7. 关键仲裁规则

### 7.1 Skill 选择不是工具选择

Skill resolution 只产出：

1. `active_skill`
2. `SkillToolScope`
3. `SkillPromptView`

它不产出：

1. 最终 `tool_name`
2. 最终 `tool_input`
3. 最终 `output_policy`

### 7.2 ToolScope 只能收窄，不能扩权

如果 skill 声明 allowed tools，它只能收窄当前可用工具池：

```text
effective_tools = global/session tools
  ∩ permission tools
  ∩ skill scope
  ∩ tool contract satisfiable tools
```

禁止：

1. skill scope 把一个全局不可用工具变成可用。
2. skill scope 跳过 permission deny。
3. skill scope 跳过 tool contract required binding。

### 7.3 Skill Prompt 不进入执行事实

Skill prompt 可以进入模型上下文，但不进入：

1. permission decision
2. contract decision
3. output persistence
4. memory projection

如果需要影响执行，必须通过 `SkillRuntimePolicy / SkillToolScope` 的结构字段表达。

### 7.4 只允许少量稳定启发式

允许的弱规则：

1. 显式文件后缀识别
2. 显式路径/URL/页码提取
3. 明确协议泄露识别
4. 明确 raw JSON / raw tool dump 识别

不允许：

1. 用自然语言关键词强选 skill
2. 用自然语言关键词强选 tool
3. 用 skill hint/example 给 route 加权
4. 用输出句式决定是否 final answer

## 11. 迁移规则

### 11.1 Shadow First

先声明契约，不立刻切主链。  
typed scope 和 output contract 先进入 debug/trace，确认不改变行为。

### 11.2 One Consumer At A Time

每次只让一个消费者切到新契约：

1. permission / contract gate
2. skill policy resolver
3. tool arbitration
4. input resolver
5. runtime output
6. projection / persistence

### 11.3 No Hidden Fallback

旧逻辑可以短期保留，但必须满足：

1. 有明确删除阶段。
2. 新链路有测试覆盖。
3. trace/debug 能看出当前使用的是新契约还是旧路径。

### 11.4 Rollback Boundary

允许回滚实现，不允许回滚原则。

可回滚：

1. 某个 consumer 是否读取新契约。
2. 某个 adapter 的实现。
3. 某条测试的迁移顺序。

不可回滚：

1. skill 不是执行 authority。
2. 工具事实单一来源。
3. 输出 raw 默认隔离。
4. 理解层不承担工具私有知识。
5. runtime 不继续堆工具特判。

---

## 12. 必须避免的反模式

1. 为某个工具在 understanding 加关键词规则。
2. 为某个 skill 在 registry 加关键词打分。
3. skill 命中后直接覆盖 candidate_tools。
4. 在 resolver 新增 `if tool_name == ...`。
5. 在 runtime 新增“某工具 raw string 可以展示”的特例。
6. 把 output contract 藏在工具实现内部。
7. 让 PDF 降级结果进入 task summary / memory。
8. 把模型可见 skill prompt 当成权限事实。
9. 继续把 `allowed_tools: list[str]` 作为跨层协议。
10. 用长场景某个失败样本倒逼恢复启发式补丁。

## 13. 最终判断

这次重构真正要达成的状态是：

> Skill 不再是第二套工具路由系统。  
> Tool 不再被理解层、resolver 和 runtime 分别认识一遍。  
> 能力控制面与工具执行面通过 typed contract 对接。

当这个状态成立时：

1. 新工具接入成本应回到：
   - 写工具实现
   - 写工具定义
   - 写工具测试
2. 新 skill 接入成本应回到：
   - 写 `SKILL.md`
   - 声明 prompt/policy/scope
   - 写 skill policy 测试
3. runtime 不再因为工具或 skill 新增而持续膨胀。

所以，这一版 68 的正式推荐路线是：

> `TaskFrame -> SkillPolicyFrame -> ToolInvocationRequest -> ToolResultEnvelope -> CanonicalOutput`

而不是：

> understanding 猜一次，skill 猜一次，tool registry 猜一次，runtime 再猜一次。

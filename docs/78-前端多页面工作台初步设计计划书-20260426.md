# 前端多页面工作台初步设计计划书

日期：2026-04-26

## 目标

当前前端已经具备会话主界面、灵魂切换、右侧导航入口、会话列表、技能列表、文件读写、RAG 开关、上下文 token 统计等基础能力。下一阶段的目标，是把右侧导航区从“预览入口”升级成真正的工作台导航，让用户可以在当前会话区域内切换不同功能页面。

这个改造不把应用拆成多个浏览器路由页面，而是先采用“单页工作台 + 主工作区视图切换”的方式。右侧导航负责切换工作区，中央原本的会话区域负责承载当前页面。这样保留现在的整体布局和灵魂主题，也方便后续把某个页面独立成 Next route。

## 设计原则

1. 会话仍然是默认主入口

用户打开页面后仍进入对话区域。右侧导航不是替代会话，而是让用户在同一个工作台里查看记忆、框架、测试、能力等系统状态。

2. 右侧导航切换中央区域

右侧导航点击后，中央区域从 `ChatPanel` 切换到对应功能面板。顶部导航、左侧灵魂与会话区、右侧导航保持稳定。

3. 第一阶段以只读为主

记忆系统、系统框架、实验系统、能力模块先以查询、展示、跳转、运行入口为主。涉及删除、重建索引、批量改写记忆、触发长测试等高风险操作，先放到第二阶段。

4. 先复用现有 API，再补必要小接口

现有前端已经有 `/api/sessions`、`/api/sessions/{id}/history`、`/api/tokens/session/{id}`、`/api/files`、`/api/skills`、`/api/config/rag-mode`。新功能优先复用这些能力。确实没有数据源的地方，再补窄接口。

5. 每个模块都有明确边界

右侧入口不再只是“功能名字”，而是对应一个清晰页面。每个页面要能回答一个问题：用户点进来能看到什么，能查询什么，能做什么。

## 模块一：记忆系统

定位：查询、理解和核对当前系统里的记忆。

页面需要覆盖两类记忆：

1. 状态记忆

状态记忆对应当前会话的工作状态，主要来自 `session-memory/` 和 `structured_memory` 体系。它不等于完整聊天历史，而是当前任务状态、阶段、目标、风险、待办和上下文摘要。

2. 长期记忆

长期记忆对应 `durable_memory/`，主要用于跨会话稳定事实、用户偏好、项目约定和长期可复用信息。

第一阶段功能：

1. 展示当前会话的状态记忆摘要。
2. 展示长期记忆文件列表或索引条目。
3. 支持关键词查询记忆。
4. 支持按来源筛选：`状态记忆`、`长期记忆`、`全部`。
5. 点击结果后显示原文片段、来源路径、更新时间或 metadata。
6. 支持跳转到已有 inspector 打开对应文件，如果文件路径在 `/api/files` 白名单内。

当前可复用能力：

1. `/api/sessions/{session_id}/history` 可以读取当前会话历史和 compressed context。
2. `/api/files?path=durable_memory/...` 可以读取长期记忆文件。
3. `backend/storage/indexes_v2/session_memory/units.json` 和 `backend/storage/indexes_v2/durable_memory/units.json` 已经存在索引数据，但当前没有面向前端的专用 API。

建议补接口：

```text
GET /api/memory/session/{session_id}
GET /api/memory/durable
GET /api/memory/search?q=&scope=session|durable|all&session_id=
```

返回建议：

```ts
type MemorySearchResult = {
  id: string;
  scope: "session" | "durable";
  title: string;
  preview: string;
  source: string;
  score?: number;
  updated_at?: string;
  metadata?: Record<string, unknown>;
};
```

第二阶段功能：

1. 手动触发当前会话状态记忆刷新。
2. 手动触发长期记忆抽取。
3. 对长期记忆做只读审阅和安全编辑。
4. 显示本轮回答实际召回了哪些记忆。

## 模块二：证据检索

定位：后续承载 RAG、PDF、结构化数据和知识库证据查询。

当前状态：先保留，不急着实装。

第一阶段页面形态：

1. 显示当前 RAG 开关状态。
2. 展示“知识库 / PDF / 表格数据”三个入口卡片。
3. 每个入口显示当前阶段：`待接入`、`已有后端能力`、`需要补前端页面`。
4. 不做复杂检索交互，避免和后面的证据子系统重构冲突。

后续可接入：

1. `knowledge/` 文件和索引。
2. PDF agent runtime。
3. normalized ingestion 的文档分块结果。
4. retrieval service 的 evidence payload。

## 实施阶段

### 阶段一：前端视图切换骨架

目标：右侧导航可以切换中央工作区。

交付：

1. `activeWorkspaceView` 状态。
2. `RightRail` 导航项可点击。
3. `WorkspacePanel` 根据视图渲染。
4. 原 `ChatPanel` 作为 `chat` 视图保留。
5. 证据检索、试验场先做占位页。

验收：

1. 切换页面不丢当前会话。
2. 切回会话后消息和输入框保持正常。
3. 灵魂主题仍然作用于所有页面。

### 阶段二：记忆系统最小实装

目标：用户可以查询状态记忆和长期记忆。

交付：

1. 记忆系统页面。
2. 记忆 scope 筛选。
3. 关键词查询。
4. 结果列表和详情。
5. 必要的 `/api/memory/*` 接口。

验收：

1. 能看到当前会话状态记忆。
2. 能看到长期记忆条目。
3. 查询结果能显示来源和片段。

### 阶段三：系统框架页面

目标：清晰展示项目模块架构和工作流程。

1. 架构节点图。
2. 工作流程图。
3. 模块详情面板。
4. 当前运行状态卡片。

验收：

1. 用户能看懂请求链路。
2. 用户能看懂 seed、memory、tool、skill、model runtime 的关系。
3. 每个模块能定位到关键代码文件。

### 阶段四：能力模块页面

目标：统一展示 skills、tools、sub-agent。

交付：

1. Skills 列表和详情。
2. Tools 列表和详情。
3. Agent 静态说明和文件入口。
4. 能力搜索和分类筛选。

## 第一轮推荐实施顺序

建议先做：

1. 阶段一：工作区切换骨架。
2. 阶段三：系统框架页面静态版。
3. 阶段四：能力模块页面，先接 `/api/skills` 和 `TOOLS_REGISTRY.json`。
4. 阶段二：记忆系统，需要补接口，所以放在骨架之后。
5. 阶段五：实验系统只读版。

暂缓：

1. 证据检索完整实装。
2. 试验场实装。
3. 前端直接运行测试。
4. 记忆编辑和删除。

## 风险与注意点

1. 不要把所有功能都塞进 `ChatPanel`

会话页面应该保持干净。多页面功能应该通过 `WorkspacePanel` 切换，而不是在 `ChatPanel` 内部堆条件分支。

2. 不要让右侧导航变成开发者日志

导航项应该使用用户能理解的功能语言，比如“记忆系统”“系统框架”“实验系统”，不要显示“待接二级页”“预留”等内部状态。

3. 记忆系统必须区分状态记忆和长期记忆

状态记忆服务当前会话推进，长期记忆服务跨会话稳定事实。前端如果混在一起，会让用户误以为记忆都是同一种东西。

4. 实验系统要先只读

测试运行涉及耗时、文件输出和模型调用。第一阶段先展示测试体系和报告，后续再做运行能力。

5. 系统框架页面要避免写成文档墙

应该优先用图、模块卡片、流程线、详情抽屉来表达架构关系。长文本只作为模块详情补充。

## 初步验收清单

1. 右侧导航可以切换中央工作区。
2. 切回会话后当前会话状态不丢。
3. 灵魂主题在所有新页面中继续生效。
4. 记忆系统能区分状态记忆和长期记忆。
5. 系统框架能展示核心模块和主要链路。
6. 实验系统能看到测试分类和报告入口。
7. 能力模块能展示 skills、tools、sub-agent 的最小信息。
8. 证据检索和试验场有正式占位页，不影响其他功能。

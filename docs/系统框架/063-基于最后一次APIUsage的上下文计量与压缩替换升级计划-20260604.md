# 基于最后一次 API Usage 的上下文计量与压缩替换升级计划

日期：2026-06-04

状态：已废弃。2026-06-05 起，当前 session 上下文压力不再以最后一次 provider API usage 为权威。最新实现由 `runtime.context_management.session_compaction` 计算 deterministic session pressure，`ContextUsageMeter` 只套用模型窗口/压缩阈值并保留 provider/cache 诊断。本文仅保留为历史方案记录，不作为实现依据。

## 1. 结论

本项目后续上下文计量采用 Claude Code 风格的核心方法：

```text
当前上下文占用 = 最后一次真实模型响应返回的 provider usage
             + 最后一次响应之后新增内容的本地估算
```

也就是说：

- **最后一次 API usage 是上下文计量锚点**，不是累计 token 总量。
- `prompt_tokens + completion_tokens` 近似表示上一轮模型响应结束后，模型线程里已经存在的上下文体积。
- `cached_tokens` 只能用于成本和 prompt cache 命中分析，不能从上下文窗口占用里扣掉。
- 本地 token 估算只用于发起请求前预算、usage 缺失时兜底、以及最后一次 provider usage 之后新增消息的 pending 估算。
- 压缩可以持续准备，但**替换历史**必须接近上下文上限才允许触发。DeepSeek 1M 上下文默认到 `900_000` token 左右再进入自动替换门槛。

这不是复刻 Claude Code 或 Codex 的产品形态，而是借鉴它们成熟 agent 的工程不变量：provider usage 是事实锚点，累计账单和当前上下文窗口必须分离，压缩替换必须保护当前热协议和 prompt cache。

## 2. 当前源码事实

### 2.1 已有正确基础

当前代码已经具备一部分可复用基础：

```text
backend/runtime/prompt_accounting/provider_usage.py
backend/runtime/model_gateway/model_runtime.py
backend/runtime/prompt_accounting/ledger.py
backend/runtime/prompt_accounting/models.py
backend/runtime/prompt_accounting/cache_planner.py
backend/runtime/prompt_accounting/token_counter.py
```

事实：

- `provider_usage.py` 已能从 `usage`、`usage_metadata`、`token_usage`、`response_metadata` 中抽取 provider usage。
- DeepSeek 的 `prompt_cache_hit_tokens`、`prompt_cache_miss_tokens` 已被映射到 `cached_tokens` / `prompt_tokens`。
- `model_runtime.py` 在请求前写入 `local_prediction`，在模型返回后写入 `provider_usage`。
- `ledger.py` 已能按 `session_id`、`run_id`、`task_run_id` 汇总 token usage 与 prompt cache record。
- `token_counter.py` 已明确 provider usage 是账单真相，本地 counter 只是 pre-request predictor。

这些结构可以承载新方案，不需要另起一套 token 系统。

### 2.2 当前主要问题

当前问题集中在“谁拥有上下文压力判断权”：

```text
backend/context_system/compaction/compactor.py
backend/context_system/packaging/controller.py
backend/api/tokens.py
backend/harness/runtime/compiler.py
backend/token_accounting.py
```

问题：

- `ContextCompactor._pressure_level()` 仍然用本地 `conversation_tokens` 和 `message_count > max_messages` 判断 `full_compact`。这会导致 DeepSeek 1M 上下文还远没满时，仅因为消息数量超过 18 就进入 full compact。
- `/api/tokens/session/{session_id}` 把累计 provider prompt tokens、本地 message tokens、raw history estimate 混在一个 `total_tokens` 视图里，容易把“账单累计”误读成“当前上下文窗口占用”。
- `_provider_protocol_message_specs()` 会把 `api_transcript` / `provider_protocol_history` 全部按 `compression_role=preserve` 回放给模型。这样即使 session UI 历史被压缩，模型 replay 仍可能保留巨大旧协议历史。
- `cache_planner.py` 的 `ttl_seconds=300` 是本地估计字段，不是 DeepSeek 官方缓存事实。新方案不再需要 TTL 维度，只保留本地计划事实和 provider 官方 usage 事实。
- `token_accounting.py` 使用 `cl100k_base` 或 `len(text)//4` 估算。这个估算可用于预算，但不能作为 DeepSeek 精确 token 事实。

## 3. 外部参考结论

### 3.1 Claude Code 可借鉴点

本次采用 Claude Code 的上下文计量思路：

- 找到最后一个带真实 `usage` 的 assistant 响应。
- 使用该 usage 作为当前上下文窗口的锚点。
- 只估算最后一次 usage 之后新增的消息。
- 对并行工具调用导致的拆分 assistant 记录，回溯到同一 assistant message id 的第一条 sibling，避免 pending 计算漏掉同一响应的拆分片段。

本项目不需要照搬 Claude Code 的消息结构，但需要落地同样的计量原则：**当前上下文窗口不是累计 token 账单，而是最后一次模型响应后的线程状态**。

### 3.2 Codex 可借鉴点

Codex 的用法也支持同一方向：

- `last_token_usage` 用于展示当前上下文窗口压力。
- `total_token_usage` 用于累计成本和整体消耗。
- cached input 不从上下文窗口中扣除，只影响成本和缓存命中分析。

本项目需要把这两个视图拆开：`billing_totals` 和 `context_meter`。

## 4. 目标权威链

新的 token/context 权威链如下：

```text
Provider Response Usage
-> PromptAccountingLedger
-> ContextUsageMeter
-> ContextCompactionGate
-> ContextCompactor / RuntimeCompiler / Tokens API
```

职责：

- `Provider Response Usage`：模型实际返回的 usage，是已完成请求的 token 事实。
- `PromptAccountingLedger`：只存事实，不解释上下文是否快满。
- `ContextUsageMeter`：解释“当前执行上下文”占用，选择最后一次可用 provider usage 并估算 pending。
- `ContextCompactionGate`：决定是否允许 warning、prepare、microcompact、full replacement。
- `ContextCompactor`：执行压缩，不再自己拥有压力判断权。
- `RuntimeCompiler`：只装配本轮 model replay，不应强制保留所有 UI 原始 transcript。

## 5. 目标数据结构

新增上下文计量快照，建议放在：

```text
backend/runtime/prompt_accounting/context_usage_meter.py
```

核心结构：

```python
ContextUsageSnapshot:
    session_id: str
    run_id: str
    task_run_id: str
    provider: str
    model: str
    context_window_tokens: int
    reserved_output_tokens: int
    replacement_threshold_tokens: int
    warning_threshold_tokens: int
    provider_anchor_request_id: str
    provider_anchor_created_at: float
    provider_prompt_tokens: int
    provider_completion_tokens: int
    provider_reasoning_tokens: int
    provider_total_tokens: int
    provider_cached_tokens: int
    estimated_pending_tokens: int
    current_context_tokens: int
    current_context_ratio: float
    cache_hit_rate_latest: float
    estimate_mode: str
    anchor_valid: bool
    invalidation_reason: str
    diagnostics: dict
```

计算规则：

```text
provider_anchor_tokens = prompt_tokens + completion_tokens + reasoning_tokens
current_context_tokens = provider_anchor_tokens + estimated_pending_tokens
cache_hit_rate_latest = cached_tokens / prompt_tokens
```

注意：

- `cached_tokens` 不参与 `current_context_tokens` 扣减。
- 如果 provider 返回的 `total_tokens` 已包含 reasoning，则 `diagnostics` 中记录 provider 原始字段，避免重复展示；上下文 meter 的主计算仍优先拆分字段。
- 如果没有 provider usage，则 `estimate_mode=local_predicted_no_provider_anchor`，并明确这不是精确事实。
- 如果跨 turn 切换了项目、workspace binding、task environment、provider/model、prompt manifest 或 stable prefix，则旧 anchor 不可直接用于新上下文。

## 6. DeepSeek 1M 策略

DeepSeek 当前作为大上下文模型使用时，压缩策略必须避免过早替换。

默认策略：

| 区间 | 行为 |
| --- | --- |
| `< 750_000` | 不替换历史。允许后台准备 session working summary、cache diagnostics、compact preview。 |
| `750_000 - 850_000` | warning。提示接近大窗口，但仍不自动替换。 |
| `850_000 - 900_000` | ready。允许准备 full compact 候选和手动 compact preview。 |
| `>= 900_000` | 允许自动替换冷历史，但必须保护当前热工具协议、当前用户意图、active task state、文件 evidence refs。 |

阈值还要受模型输出预留影响：

```text
available_input_context = context_window_tokens - reserved_output_tokens - safety_margin
replacement_threshold = min(900_000, available_input_context)
```

如果模型不是 1M 窗口，则按模型 registry 中的 context window 计算，不硬套 900k。

## 7. 压缩分层

压缩要分成“准备”和“替换”两类，避免为了省 token 过早污染 model replay。

### 7.1 可随时做的后台准备

允许在低压时持续执行：

- session working summary 更新。
- semantic compaction preview。
- prompt segment map 和 cache stability report。
- 大工具输出 ref 化候选。
- code/file condensation 候选。

这些动作只生成候选、摘要或诊断，不替换当前 model replay。

### 7.2 需要门槛的替换

以下动作必须通过 `ContextCompactionGate`：

- `replace_runtime_context`。
- full compact 后把旧消息替换为 compact summary + recent hot tail。
- 本地重写旧 tool result / provider protocol history。
- 从 model replay 中删除旧 `api_transcript`。

自动替换只在达到阈值、或 provider 报 context overflow、或用户手动 compact 时触发。

## 8. UI Transcript 与 Model Replay 分离

当前 `api_transcript` 被同时当作 UI 历史和 provider protocol replay 来源，容易造成压缩失效。需要拆成两个概念：

```text
public_chat_timeline / api_transcript
    用于 UI、审计、用户查看原始对话。
    不决定下一轮模型必须完整回放。

model_replay_history / provider_protocol_replay
    用于下一轮发给模型。
    可根据 ContextUsageMeter 和 CompressionGate 做压缩、裁剪、summary replacement。
```

`_provider_protocol_message_specs()` 后续只允许保留：

- 当前 turn 正在进行的 hot tool call / tool result pair。
- DeepSeek thinking mode 需要的最近 assistant `reasoning_content` 协议片段。
- compact 后由 invariant normalizer 判定必须保留的协议消息。

不允许继续把全部旧 `api_transcript` 作为 `compression_role=preserve` 注入模型。

## 9. 跨 Turn 环境切换与 Prompt Cache

跨 turn 环境切换必须影响 usage anchor 和 cache baseline。

新增 `PromptContextAnchorFingerprint`：

```text
provider
model
base_url cache relevant part
session_id
workspace_binding_id / project_id
task_environment_id
prompt_manifest_hash
stable_prefix_hash
tool_schema_hash
```

规则：

- fingerprint 一致时，可以使用最后一次 provider usage 作为上下文 meter anchor。
- fingerprint 改变时，旧 provider usage 只能保留在 billing totals，不能继续作为当前上下文占用锚点。
- 环境切换后第一轮，如果还没有新 provider usage，使用当前 prompt segment local prediction，并标记 `anchor_valid=false`。
- cache baseline 必须 reset，reason 记录为 `environment_fingerprint_changed`。
- TaskRun 使用自己的 meter；父 TurnRun 只接收 TaskRun 的 brief/output ref，不把 TaskRun 内部全部 provider history 并入父 turn。

## 10. API 与前端展示

`/api/tokens/session/{session_id}` 需要拆分返回：

```text
billing_totals
    累计 provider usage、本地预测、cache savings、record count。

context_meter
    当前上下文窗口占用、最后一次 API usage anchor、pending 估算、阈值和压力等级。

cache_metrics
    latest cache hit rate、recent hit rates、cached tokens、miss tokens、cache break diagnostics。

compaction_readiness
    是否允许 auto replacement、是否只允许 preview、阻塞原因、推荐动作。
```

旧字段 `total_tokens` 可以保留一个版本作为兼容展示，但必须改名或标记含义，避免继续表示“当前上下文占用”。

文件：

```text
backend/token_accounting.py
backend/runtime/prompt_accounting/token_counter.py
```

完成标准：

- DeepSeek 标记为 `local_predicted`，不伪装 exact。
- fallback 使用语言感知字符比例：
  - 英文字符约 `0.3 token`。
  - 中文字符约 `0.6 token`。
  - JSON、代码、符号密集文本做保守上调。
- 图片、文档、base64 类块不按原始字符串无限膨胀，优先使用 provider metadata 或固定成本估算。

### Phase 6：cache TTL 语义清理

文件：

```text
backend/runtime/prompt_accounting/cache_planner.py
backend/runtime/prompt_accounting/models.py
```

完成标准：

- 新版 cache record 不再暴露 `ttl_seconds`。
- 本地只记录 cache plan：prefix hash、cache key、cache eligibility、baseline、cache break diagnostics。
- 官方只记录 provider usage：cached tokens、miss tokens、cache read/create tokens、latest/recent hit rate。
- 压缩替换决策不得依赖 TTL。
- cache warm/cold 只能来自 provider usage、cache hit 趋势或 provider 明确能力，不靠固定 5 分钟假设。
- 如果旧 JSONL 中仍有 `ttl_seconds` 字段，读取时忽略，后续重写或清理时自然消失；不为它保留新的业务语义。

## 13. 不允许的实现方式

- 不允许把本地 `cl100k_base` 估算当作 DeepSeek 精确 token。
- 不允许把累计 token 账单显示为当前上下文占用。
- 不允许为了提高 cache 命中率过早替换用户历史。
- 不允许仅因为 message count 超过阈值触发 full compact。
- 不允许把 `cached_tokens` 从上下文窗口占用中扣除。
- 不允许把 DeepSeek prompt cache 当成固定 TTL。
- 不允许新增本地 TTL、官方 TTL、估算 TTL 之间的第三套 cache 判断口径；cache 判断只分本地计划事实和 provider 官方 usage 事实。
- 不允许继续把旧 `api_transcript` 全量 preserve 到 model replay。
- 不允许让压缩摘要替代当前用户消息、当前工具协议、权限事实和文件 evidence refs。
- 不允许新增一套与 `PromptAccountingLedger` 平行的 token ledger。

## 15. 实施顺序建议

先做 Phase 1 和 Phase 2，把“当前上下文占用”显示正确。否则后续压缩策略仍会被错误 token 指标误导。

然后做 Phase 3，解除 message count 对 full compact 的旧权威，并把 DeepSeek replacement gate 锁到 provider-anchored meter。

再做 Phase 4，拆分 UI transcript 与 model replay。这一步会影响运行语义，必须配套协议不变量测试。

最后做 Phase 5 和 Phase 6，清理本地 estimator 和 TTL 语义，减少后续误用。

本计划没有要求立即改 prompt 文案。当前问题首先是计量权威和压缩替换权威错误，属于 runtime/context/accounting 结构问题。

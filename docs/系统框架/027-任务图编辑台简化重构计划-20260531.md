# 任务图编辑台简化重构计划

## 背景

当前任务图编辑台同时承载工作集选择、图状态、主会话绑定、分层配置入口、拓扑编辑、发布运行等职责。功能完整但首屏密度过高，用户很难区分“当前在配置哪个任务环境”“正在编辑哪个图草稿”“哪些是全局动作”“哪些是当前层配置”。

## 目标

- 保留现有图级配置、拓扑、执行者、资源流、风险治理、契约质量、发布运行等配置能力。
- 把编辑台改成清晰的工作台层级：顶部图身份与发布动作、工作集条、左侧分组层导航、右侧当前层页面。
- 任务图工作台保留在主页面图任务工作台，不移入任务系统；任务系统只管理任务环境、契约库和节点配置。节点配置只保存执行者引用、运行档案引用、模型能力要求和权限约束，不管理执行者注册信息。
- 任务环境管理页定位为 agent 运行资源装载控制台，不再把环境目录重复放进主工作区；左侧只负责选择环境，右侧只配置当前环境在 agent 启动前要装载的 Prompt、文件、记忆、知识、检索索引、存储空间和执行边界。
- 清理旧的 metrics / rail 样式残留，避免新旧 UI 规则叠加。

## 非目标

- 不改 `TaskGraphDefinition`、`GraphHarnessConfig`、发布运行语义和后端 API contract。
- 不重写拓扑画布内部交互。
- 不把配置能力隐藏到不可发现的弹窗中。

## 实施步骤

1. 工作集区域收敛：环境配置、图草稿、绑定主会话、新图草稿保持在同一条工作集栏，但明确分成 controls / status / actions。
2. 全局状态压缩：把节点数、边数、问题数、发布态从四个指标卡压缩为内联事实，保留保存和发布主动作。
3. 分层导航重构：使用主图、执行、资源、验证四组卡片式入口，保留所有配置层。
4. 样式清理：删除旧 `task-graph-studio-topbar__metrics` 和旧 rail 残留，统一系统页与中心工作区的响应式规则。
5. 任务系统层级修复：任务系统只保留任务环境、契约库、节点配置三个管理面；移除任务域、任务图编辑台和跨页面跳转入口。
6. 任务环境管理页修复：移除主区重复的环境列表，把首屏改为 agent 运行装载配置和装载预览；Prompt、文件 Profile、仓库类型、环境记忆、项目知识、共享上下文、检索索引、资源空间策略作为常用配置，高级 JSON 只承载完整策略对象。
7. 验证：运行前端类型检查和相关测试，并用固定端口检查真实页面布局。

## 验收标准

- 首屏能明确看出当前任务环境、图草稿和主会话绑定状态。
- 配置层入口完整可达，但不再和拓扑工具、发布状态混成一组。
- 系统页和中心工作区均无明显重叠、横向挤压或不可点击区域。
- 任务系统左侧只出现任务环境、契约库、节点配置；不出现任务图编辑台入口或跳转按钮。
- 任务环境页主区不再出现第二套环境列表；主区以 agent 运行装载为主，不把环境配置做成只读信息展示。
- 保存任务环境时必须提交完整 `TaskEnvironmentSpec`，不能因为前端使用 `record` payload 而丢掉 `file_management`、`resource_space`、`memory_space` 或环境 Prompt。
- 环境 Prompt 文案仍是给 agent 直接执行的角色和边界说明，不写成开发说明。
- 类型检查和相关前端测试通过。

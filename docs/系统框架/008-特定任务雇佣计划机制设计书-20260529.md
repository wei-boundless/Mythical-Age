日期：2026-05-29

状态：明确方案稿

适用范围：

```text
backend/task_system
backend/task_system/environments
backend/task_system/repositories/specific_task_repository.py
backend/task_system/tasks/assembly_policy.py
backend/task_system/tasks/assembly_policy_resolver.py
backend/task_system/assembly/contract_issuer.py
backend/harness/runtime
backend/harness/loop
backend/query/runtime.py
```

## 0. 结论

本项目不应把“特定任务”理解为意图识别结果，也不应把它默认等同于 TaskRun。

特定任务应标准化为：

```text
Registered Engagement Plan
已注册任务承接计划
```

它是系统中预先登记好的、可雇佣的任务契约。它描述：

```text
做什么
由谁承接
在哪个任务环境中运行
需要什么输入
产出什么结果
采用哪种执行策略
如何验收
失败后如何恢复
```

它进入系统的主链固定为：

```text
EngagementRequest
-> EngagementPlanResolver
-> EngagementAdmission
-> EngagementContractIssuer
-> RuntimeAssembly
-> ExecutionStrategyDispatch
-> EngagementRunRecord / EngagementEvent
-> Delivery
-> Acceptance
-> Closeout
```

没有意图识别层。系统收到的是明确契约引用，不是用户自然语言中的任务猜测。

v1 固定采用严格合同方案：

```text
一个 Registered Engagement Plan 绑定一个 task_environment_id。
一个 Registered Engagement Plan 绑定一个 execution_strategy。
一个 Registered Engagement Plan 绑定一个 assignee。
启动请求不能覆盖 task_environment_id。
启动请求不能覆盖 execution_strategy。
启动请求不能覆盖 runtime_mode。
需要不同环境、不同策略、不同模式时，创建新的 plan version。
```

原因：

```text
特定任务机制的价值是把已确定的雇佣关系标准化，而不是在启动时重新做策略选择。
如果允许启动时覆盖环境、模式和执行策略，engagement plan 会退化成半成品配置，系统又会回到旧的 runtime lane / selected_task / override 分叉。
```

## 1. 当前代码事实

### 1.1 已有可复用结构

当前项目已有几类结构可以迁移：

```text
backend/task_system/registry/flow_models.py
  SpecificTaskRecord
  TaskExecutionPolicy
  TaskFlowContractBinding
  TaskMemoryRequestProfile

backend/task_system/repositories/specific_task_repository.py
  SpecificTaskRepository

  SpecificTaskAssemblyPolicy
  AgentSelectionPolicy
  ToolCapabilityRequirements
  RequirementRefs

  resolve_specific_task_assembly_policy

  TaskContractIssuer.issue_specific_task_contract

backend/harness/loop/task_lifecycle.py
  TaskRunContract
  start_task_lifecycle

backend/harness/runtime/assembly.py
  assemble_runtime
```

这些结构说明项目已经有“任务库记录 -> 装配策略 -> 合同发行”的雏形。

### 1.2 当前断点

当前机制还没有形成成熟主链，主要断点是：

```text
SpecificTaskRecord 只是任务记录，不是完整可执行契约。
SpecificTaskAssemblyPolicy 容易被理解为任务专属权限配置，必须退出主链。
TaskContractIssuer 生成 task_system.contracts.TaskContract，而单 agent loop 启动使用 harness.loop.task_lifecycle.TaskRunContract。
特定任务没有统一 execution_strategy，因此会被误解成“一定启动 TaskRun”。
环境任务库和特定任务记录之间没有明确生命周期关系。
```

目标不是另建一套执行系统，也不是给旧结构改名，而是把可用信息一次性迁移进统一的 engagement plan 契约入口。迁移完成后，runtime 主链只认 engagement plan / engagement contract。

## 2. 核心概念

### 2.1 Registered Engagement Plan

Registered Engagement Plan 是预注册任务承接计划。

它不是：

```text
意图识别结果
任务类型枚举
skill
workflow
TaskRun
agent profile
task environment
```

它是一个成型契约，系统可解析、可装配、可执行、可验收。

### 2.2 和 Skill 的区别

```text
Skill 描述 agent 怎样使用某种能力。
Engagement Plan 描述系统如何雇佣某个 agent/workflow 完成一个任务契约。
```

Skill 更偏能力方法。

Engagement Plan 更偏任务合同、输入输出、运行场地、承接者和验收。

### 2.3 和 TaskRun 的区别

TaskRun 是长任务生命周期。

Engagement Plan 可以启动 TaskRun，也可以只是当前 turn 的约束。

因此必须有：

```text
execution_strategy.kind
```

而不能默认启动 TaskRun。

## 3. 权责边界

### 3.1 Task Environment

环境负责系统运行场地：

```text
environment_prompts
storage_space
artifact_root
runtime_state_root
task_library_root
file_management
file_access_tables
sandbox_policy
execution_policy
risk_policy
artifact_policy
memory_space 作为环境资源描述
```

环境不负责：

```text
选择 agent
授予工具
决定是否启动 TaskRun
解释任务目标
动态装配 memory
动态装配 skill
```

### 3.2 Engagement Plan

计划负责任务契约：

```text
任务目标
输入合同
输出合同
承接者要求
执行策略
prompt contract
验收策略
恢复策略
需要的资源/能力声明
```

计划不负责：

```text
直接授权工具
绕过 agent profile
覆盖环境边界
执行工具
写入 artifact
替系统验收
```

### 3.3 Agent Profile

Agent profile 负责能力边界：

```text
allowed_operations
blocked_operations
runtime modes
是否可委派
工具可见性上限
工作角色 prompt
memory/context 动态装配策略
```

### 3.4 Runtime Assembly

runtime assembly 负责当次装配：

```text
读取 engagement plan
解析 task environment
读取 agent profile
生成 available tools
生成 operation authorization projection
生成 runtime packet
```

runtime assembly 不负责重新解释任务。

### 3.5 Execution Strategy Dispatcher

执行策略分发器负责按计划执行形态进入对应生命周期：

```text
turn_contract
turn_execution
single_agent_task_run
workflow_run
human_gate
```

它不能绕过 runtime assembly。

## 4. 标准数据模型

### 4.1 RegisteredEngagementPlan

新增：

```text
backend/task_system/engagement/models.py
```

核心字段：

```python
RegisteredEngagementPlan(
    plan_id: str,
    title: str,
    description: str = "",
    version: str = "1.0.0",
    status: "draft" | "active" | "deprecated" | "disabled" | "archived" = "draft",
    task_environment_id: str,
    assignee: EngagementAssignee,
    runtime_profile: EngagementRuntimeProfile,
    execution_strategy: EngagementExecutionStrategy,
    input_contract: dict,
    output_contract: dict,
    prompt_contract: dict,
    resource_requirements: dict,
    capability_requirements: dict,
    memory_requirements: dict,
    acceptance_policy: dict,
    recovery_policy: dict,
    created_at: str = "",
    updated_at: str = "",
    supersedes_plan_id: str = "",
    metadata: dict,
    authority: str = "task_system.registered_engagement_plan",
)
```

字段规则：

```text
task_environment_id 是 plan 的固定环境绑定，不允许 start 请求覆盖。
status=active 才可启动。
draft 只可编辑和校验，不可执行。
deprecated 可查看历史 run，不可新启动。
disabled 是人工下线，不可启动。
archived 只保留审计，不出现在默认任务库列表。
version 是同一 plan_id 的语义版本；环境、策略、承接者发生变化时必须产生新 version。
supersedes_plan_id 用于把旧计划指向替代计划，不用于运行时自动跳转。
```

### 4.2 EngagementAssignee

```python
EngagementAssignee(
    kind: "agent" | "workflow" | "human" | "system",
    agent_id: str = "",
    agent_profile_id: str = "",
    workflow_id: str = "",
    participant_agent_ids: tuple[str, ...] = (),
)
```

规则：

```text
kind=agent 只说明承接者是单 agent，不说明是否创建 TaskRun。
kind=workflow 只说明承接者是 workflow，不说明 workflow 内部是否创建 TaskRun。
assignee 只选择承接者，不授予工具权限。
execution_strategy.kind 是唯一执行生命周期来源。
```

### 4.3 EngagementRuntimeProfile

```python
EngagementRuntimeProfile(
    runtime_mode: "role" | "standard" | "professional" | "custom",
    runtime_mode_policy: dict = {},
)
```

规则：

```text
runtime_mode 是 plan 固定选择的 agent 运行模式。
runtime_mode_policy 是 plan 固定选择的模式配置。
启动请求不能覆盖 runtime_mode。
不得使用 specific_task_runtime_policy 这类旧命名。
```

### 4.4 EngagementExecutionStrategy

```python
EngagementExecutionStrategy(
    kind: "turn_contract" | "turn_execution" | "single_agent_task_run" | "workflow_run" | "human_gate",
    startup_policy: dict = {},
    lifecycle_policy: dict = {},
)
```

策略定义：

| kind | 含义 | 是否创建 TaskRun |
| --- | --- | --- |
| `turn_contract` | 只约束当前 turn 的回答、格式、角色和质量标准 | 否 |
| `turn_execution` | 当前 turn 内完成轻量任务，可使用允许的 bounded tools | 否 |
| `single_agent_task_run` | 启动单 agent 长任务生命周期 | 是 |
| `workflow_run` | 启动预配置 workflow/graph | 否 |
| `human_gate` | 创建人工确认、审批或建议入口 | 否 |

规则：

```text
workflow_run 本身不创建 TaskRun。
workflow 节点如需创建 TaskRun，必须在节点执行阶段显式提交 TaskRunContract 并重新经过 admission。
不得把 workflow_run 静默降级为 single_agent_task_run。
```

```python
EngagementRequest(

    session_id: str = "",
```

```text
EngagementRequest 不包含 task_environment_id。
EngagementRequest 不包含 execution_strategy_override。
EngagementRequest 不包含 runtime_mode_override。
requested_by=agent 时，必须来自模型明确 action request，不能由系统从自然语言推断。
```

## 5. 标准生命周期

### 5.1 Registration

登记阶段只保存可雇佣计划，不执行。

输入：

```text
RegisteredEngagementPlan
```

输出：

```text
plan stored by EngagementPlanRepository under task_environment.task_library_root
```

校验：

```text
plan_id 唯一
task_environment_id 存在
execution_strategy.kind 合法
assignee 引用合法
output_contract 与 strategy 匹配
status 非 active 时不可启动
```

存储权威：

```text
TaskEnvironmentSpec.task_library_root 提供物理归属。
EngagementPlanRepository 是唯一读写接口。
不存在 environment task library 和 engagement repository 两份主数据。
```

### 5.2 Engagement Request

系统收到明确请求：

```json
{
  "engagement_plan_id": "engage.dungeon_game_5_floor",
  "startup_parameters": {}
}
```

这个请求不是自然语言意图识别。

允许来源：

```text
用户在任务环境任务库中选择
前端显式按钮
上层任务/工作流发送明确 plan_ref
agent 在运行时中请求某个已知 plan_ref
```

agent 发起时的唯一合法形态：

```text
ModelActionRequest.kind = request_registered_engagement
ModelActionRequest.plan_id = 已知 RegisteredEngagementPlan.plan_id
ModelActionRequest.startup_parameters = 结构化参数
```

系统不能从普通回答、关键词、标题、文件名或 selected_task_id 自动生成这个请求。

### 5.3 Admission

系统检查雇佣是否成立：

```text
plan.status == active
task_environment_id 存在
startup input 满足 input_contract
assignee 存在
agent profile 存在
execution_strategy 受支持
长任务具备 output_contract / acceptance_policy
workflow_run 的 workflow 存在
需要人工审批时进入 requires_approval
```

Admission 不能做的事：

```text
不能替 agent 授权工具
不能修改任务目标
不能把 invalid 自动改成 TaskRun
不能替换 task_environment_id
不能替换 execution_strategy
不能替换 runtime_mode
```

### 5.4 Runtime Assembly

通过 admission 后，系统装配 runtime：

```text
task_environment_id
agent_profile_id
runtime_mode
runtime_mode_policy
engagement_contract
prompt_contract
input/output contract
available tools
operation authorization projection
sandbox/file/artifact boundary
```

agent 收到的是：

```text
你已被雇佣执行这个计划契约。
当前环境是什么。
当前输入是什么。
输出和验收标准是什么。
你可以请求哪些行动。
```

agent 不需要猜这个任务是什么。

### 5.5 Dispatch

按 `execution_strategy.kind` 分流：

```text
turn_contract -> compile_turn_action_packet with engagement contract
turn_execution -> bounded turn execution loop
single_agent_task_run -> issue TaskRunContract -> start_task_lifecycle
workflow_run -> workflow/graph launcher
human_gate -> create gate record
```

所有分支都必须保留：

```text
engagement_plan_ref
environment_ref
agent_profile_ref
runtime_assembly_ref
engagement_run_id
```

Dispatch 固定规则：

```text
Dispatcher 不选择策略，只读取 EngagementContract.execution_strategy.kind。
Dispatcher 不创建 runtime packet，只消费 RuntimeAssembly 输出。
Dispatcher 不把 unsupported strategy 降级为其它 strategy。
Dispatcher 在进入具体生命周期前必须写 EngagementEvent(dispatched)。
```

### 5.6 Delivery

不同策略交付不同产物：

```text
turn_contract: 用户可见回答或结构化结果
turn_execution: 回答 + 可选轻量 artifact/observation
single_agent_task_run: artifact + verification + task lifecycle record
workflow_run: workflow result + node outputs
human_gate: approval / rejection / user suggestion
```

### 5.7 Acceptance

验收按契约进行：

```text
required_artifacts 是否存在
required_verifications 是否完成
completion_criteria 是否满足
output schema 是否满足
失败是否有 recovery action
```

agent 自述不能替代验收。

Acceptance 分两类：

```text
TurnAcceptance:
  适用于 turn_contract / turn_execution。
  检查 schema、格式、必答项、用户可见结果是否满足合同。
  不要求 artifact gate。

ArtifactAcceptance:
  适用于 single_agent_task_run / workflow_run 中声明真实交付物的节点。
  检查 required_artifacts、required_verifications、文件存在性、可打开性、验证记录。
```

### 5.8 Closeout

关闭时记录：

```text
status: completed | blocked | failed | canceled | waiting_user | waiting_approval
artifact refs
verification refs
summary
failure reason
recovery action
next resumable ref
```

Closeout 规则：

```text
completed 必须有 acceptance result。
blocked 必须有 blocking reason 和 recovery action 或 user action。
failed 必须有 failure reason。
waiting_user / waiting_approval 不能被 UI 当成失败。
next resumable ref 只能指向 EngagementRunRecord 或 TaskRun lifecycle record，不能指向裸 task_run_id。
```

### 6.2 SpecificTaskRepository

```text
backend/task_system/engagement/repository.py
```

职责：

```text
list plans
get plan
upsert plan
delete plan
load plans from environment task library
```

旧 `specific_task_records.json` 可以迁移为：

```text
engagement_plans.json
```

迁移规则：

```text

```

### 6.3 SpecificTaskAssemblyPolicy

当前 `SpecificTaskAssemblyPolicy` 不进入新主链。

清理规则：

```text

不得被 EngagementAdmission 调用。

不得作为 API 输出模型。

```

### 6.4 TaskContractIssuer

当前 `TaskContractIssuer` 不再签发特定任务主链合同。新主链使用：

```text
EngagementContractIssuer
TaskRunContractAdapter
```

职责：

```text
EngagementContractIssuer:
  RegisteredEngagementPlan -> EngagementContract

TaskRunContractAdapter:
  EngagementContract where strategy=single_agent_task_run -> harness.loop.TaskRunContract
```

这样可以避免 task_system.contracts.TaskContract 和 harness.loop.TaskRunContract 两套合同长期打架。

旧入口清理规则：

```text
TaskContractIssuer.issue_specific_task_contract 不再被 API/runtime/loop 调用。
如果仍有调用点，必须改到 EngagementContractIssuer。
不得保留“如果 engagement 不存在就退回 old specific task contract”的 fallback。
```

## 7. 执行策略细节

### 7.1 turn_contract

用途：

```text
结构化回答
轻量审查
固定格式输出
角色化约束
```

流程：

```text
plan_ref
-> admission
-> runtime assembly
-> compile turn packet with engagement_contract
-> model respond / ask_user / block
```

不创建：

```text
TaskRun
TaskRunLedger
artifact acceptance gate
```

仍必须创建：

```text
EngagementRunRecord
EngagementEvent
TurnAcceptance
```

### 7.2 turn_execution

用途：

```text
当前 turn 内可完成的轻量任务
允许 bounded observation
允许小型 artifact
```

流程：

```text
compile turn_execution packet
-> model tool_call/respond loop
-> observation records
-> final response
```

限制：

```text
不能长时间续跑
不能需要多轮 checkpoint
不能承诺长期 artifact 验收
```

仍必须创建：

```text
EngagementRunRecord
ObservationRecord
EngagementEvent
TurnAcceptance
```

### 7.3 single_agent_task_run

用途：

```text
长任务
真实 artifact
写文件
命令/浏览器验证
失败恢复
最终验收
```

流程：

```text
EngagementContract
-> TaskRunContractAdapter
-> start_task_lifecycle
-> execute_task_run
-> artifact/verification acceptance
```

必须：

```text
绑定 task_environment_id
绑定 artifact root
绑定 required_artifacts / completion_criteria / required_verifications
记录 step summaries
支持 resume / repair
```

TaskRunContractAdapter 固定映射：

```text
EngagementContract.contract_id -> TaskRunContract.source_contract_ref
EngagementContract.plan_id -> TaskRunContract.external_plan_ref
EngagementContract.task_environment_id -> TaskRunContract.task_environment_id
EngagementContract.prompt_contract -> TaskRunContract.task_prompt
EngagementContract.output_contract -> TaskRunContract.artifact_contract
EngagementContract.acceptance_policy -> TaskRunContract.acceptance_policy
EngagementContract.recovery_policy -> TaskRunContract.recovery_policy
```

禁止：

```text
TaskRunContractAdapter 不重新解释用户目标。
TaskRunContractAdapter 不从 plan title 猜 task_goal_type。
TaskRunContractAdapter 不补默认 artifacts。
缺少必需 output/acceptance 时 admission 阶段直接 invalid。
```

### 7.4 workflow_run

用途：

```text
多阶段、多节点、多 agent 或图任务
```

流程：

```text
EngagementContract
-> workflow/graph launcher
-> workflow runtime assembly
-> node execution
-> workflow acceptance
```

本设计只定义接口，不在单 agent 阶段实现 graph 重构。

v1 实施规则：

```text
workflow_run 如果 launcher 未接通，返回 unsupported_strategy。
不得临时转给 single_agent_task_run。
workflow 内部节点如需 TaskRun，必须显式创建 TaskRunContract 并重新 admission。
```

### 7.5 human_gate

用途：

```text
人工审批
人工建议
人工选择
```

流程：

```text
create gate record
-> wait user
-> apply decision
-> closeout or dispatch next strategy
```

v1 实施规则：

```text
human_gate 只创建 gate record 和 EngagementEvent(waiting_approval/waiting_user)。
用户通过后，系统继续原 EngagementContract 的下一步，不能创建新的隐式 plan。
用户建议会作为 observation/context 输入下一次 runtime invocation，不修改原 plan。
```

## 8. Runtime Packet 装配要求

当 engagement plan 生效时，runtime packet 必须包含：

```text
engagement_plan_ref
engagement_contract
execution_strategy
input_contract
output_contract
prompt_contract
acceptance_policy
recovery_policy
task_environment
agent_profile_ref
available_tools
operation_authorization
engagement_run_ref
```

不得包含：

```text
系统猜测的 user intent
系统生成的 task type
旧 selected_task_id 推导
specific_task_runtime_policy
```

## 9. Prompt Contract 标准

Engagement Plan 可以提供 prompt contract，但必须写给 agent 理解职责：

正确：

```text
你是一名独立游戏开发 agent。
你被雇佣来完成一个五层地下塔小游戏。
你需要交付可打开、可操作、可验证的真实文件。
你不能用说明文档代替游戏实现。
死亡、重开、关卡推进、资源加载和基础美术都必须真实可用。
最终完成前必须自检入口文件、资源引用和核心交互。
```

错误：

```text
这是 dungeon_game 节点。
根据 task graph 执行 game_build。
输出 index.html。
```

## 12. 验证矩阵

### 12.1 turn_contract

```text
启动一个只要求固定格式回答的 plan。
系统不创建 TaskRun。
runtime packet 包含 engagement_contract。
系统创建 EngagementRunRecord。
系统产生 TurnAcceptance。
agent 按 prompt_contract 输出。
```

### 12.2 turn_execution

```text
启动一个轻量审查 plan。
agent 可使用只读工具。
产生 ObservationRecord。
不创建 TaskRun。
系统创建 EngagementRunRecord。
系统产生 TurnAcceptance。
```

### 12.3 single_agent_task_run

```text
启动五层地下塔游戏 plan。
系统创建 TaskRunContract。
TaskRun 进入 waiting_executor。
executor 装配 env.development.sandbox。
产物写入 artifact/storage。
最终验收检查真实 index.html、assets、验证记录。
EngagementRunRecord 引用 task lifecycle record。
```

### 12.4 agent 权限边界

```text
plan 声明需要 shell。
agent profile 没有 op.shell。
工具不可见，admission 或 runtime assembly 报 capability missing。
环境不得授予 shell。
```

### 12.5 环境边界

```text
plan 在 readonly environment 启动写入策略。
工具即使可见，执行层也不能写真实 workspace。
sandbox/file boundary fail-closed。
```

### 12.6 unsupported strategy

```text
workflow_run 未接通时返回 unsupported_strategy。
不得假装启动 workflow。
不得降级成 single_agent_task_run。
```

### 12.7 启动覆盖禁止

```text
POST start 传入 environment_id 时返回 invalid_request。
POST start 传入 execution_strategy_override 时返回 invalid_request。
POST start 传入 runtime_mode_override 时返回 invalid_request。
系统不做自动忽略，必须显式拒绝。
```

### 12.8 agent 发起计划

```text
agent 输出 request_registered_engagement(plan_id, startup_parameters)。
系统按 plan_id 解析 engagement plan。
系统不从自然语言生成 plan_id。
plan 不存在时返回 invalid，不搜索相似任务名。
```

## 13. 反模式清单

禁止：

```text
把 engagement plan 当意图识别结果。
把 engagement plan 默认转 TaskRun。
让 engagement plan 授权工具。
让 environment 授权工具。
让 workflow_run 静默降级成 single_agent_task_run。
让 agent profile 被 plan 覆盖。
用 prompt 合同替代系统验收。
保留 specific_task_runtime_policy 作为新入口。
在旧 SpecificTaskRecord 上继续堆新字段而不建立正式模型。
允许 start 请求覆盖 task_environment_id。
允许 start 请求覆盖 execution_strategy。
允许 start 请求覆盖 runtime_mode。
在 resolver/admission/dispatcher 中猜测替代 plan。
把 deprecated plan 自动跳转到 supersedes_plan_id。
把 SpecificTaskAssemblyPolicy 当成 compatibility adapter 长期保留。
```

## 14. 最终定义

本项目的特定任务机制定义为：

```text
Registered Engagement Plan 是任务系统中的预注册承接契约。
它在任务环境中被管理，可以由用户、系统、上层任务或 agent 明确引用。
它被 EngagementRequest 引用，经过 resolver/admission 后签发 EngagementContract，再进入 runtime assembly。
它通过 execution_strategy 选择 turn、TaskRun、workflow 或 human gate 生命周期。
它不能绕过 agent profile、task environment、operation gate、tool executor 和 acceptance。
它不允许启动时覆盖环境、策略和模式；变化必须通过新的 plan version 表达。
它的价值是让成熟任务以预先确定的计划模式进入系统，同时仍服从统一任务体系。
```

# TaskRun 状态权威收口与监控投影修复方案

## 背景

本次审查发现，harness 主控制链路已经基本回到“Agent 负责语义判断，系统负责边界控制”的方向，但 TaskRun 状态投影仍存在结构性分裂：

- `harness.task_run_state_view` 已经在解释 TaskRun 的工作状态、恢复状态和 executor lease。
- Runtime Monitor 又在 `activity.py`、`projector.py`、`management.py` 中重新解释 running / waiting / resumable / interruptible。
- Session task projection 又在 `_projection_control()` 和 `_projection_status()` 中解释一遍。
- 前端再根据 raw `status`、`control_state`、`lifecycle` 推导会话活动和可见状态。

这会导致同一个 TaskRun 在不同层显示出不同的控制能力。例如 `ready_to_continue` 本来已经是恢复点，却可能被会话投影暴露为可暂停状态。

## 目标

把“TaskRun 当前是什么状态、用户能如何控制它”收口到唯一权威视图：

```text
TaskRun + diagnostics + monitor overlay
-> Canonical TaskRunStateView
-> RuntimeMonitorItem / SessionTaskProjection / ManagementActions
-> Frontend render
```

修复后：

1. `TaskRunStateView` 是 single-agent TaskRun 状态和控制能力的唯一解释器。
2. Runtime Monitor 只叠加 stale、visibility、lane、revision 等展示层信息，不重新决定可暂停、可恢复、可停止。
3. Session projection 只把 canonical view 转成会话公开结构，不再自建状态机。
4. 前端只消费后端给出的 `activity_state`、`activity_label`、`control_capability`；raw status 只作缺字段 fallback。
5. graph-controlled task 不暴露 single-agent task 的 direct pause/resume/stop。

## Canonical View 设计

`task_run_state_view(task_run, monitor=None)` 输出需要包含以下稳定结构：

```python
{
    "task_status": "...",
    "terminal_reason": "...",
    "task_work_state": "active | ready_to_continue | paused | waiting_approval | waiting_user | completed | failed | stopped",
    "executor_status": "...",
    "executor_lease_state": "none | scheduled | running | lost | recovering | blocked",
    "control_state": "...",
    "runtime_control": {...},
    "recoverable": bool,
    "recovery_action": "...",
    "graph_controlled": bool,
    "running_claimed": bool,
    "can_pause": bool,
    "can_resume": bool,
    "can_stop": bool,
    "resume_mode": "...",
    "control_capability": {
        "can_pause_task": bool,
        "can_resume_task": bool,
        "can_stop_task": bool,
        "is_resumable": bool,
        "is_interruptible": bool,
        "resume_mode": "...",
        "control_reason": "...",
        "authority": "harness.task_run_state_view.control_capability",
    },
    "activity": {
        "activity_state": "...",
        "activity_label": "...",
        "is_running": bool,
        "is_waiting": bool,
        "is_resumable": bool,
        "is_interruptible": bool,
        "control_reason": "...",
        "tone": "...",
        "authority": "harness.task_run_state_view.activity",
    },
    "authority": "harness.task_run_state_view",
}
```

## 控制能力矩阵

| work state | pause | resume | stop | 说明 |
| --- | --- | --- | --- | --- |
| `active` 且 executor claimed | 是 | 否 | 是 | 真实运行中，可请求边界暂停或停止 |
| `ready_to_continue` | 否 | 是 | 是 | 已经停在恢复点，不能再 pause |
| `paused` | 否 | 是 | 是 | 暂停点可继续或停止 |
| `waiting_approval` | 否 | 否 | 是 | 等审批，不应伪装成 pause |
| `waiting_user` | 否 | 否 | 是 | 等用户或系统处理，可停止 |
| `completed` / `failed` / `stopped` | 否 | 否 | 否 | 终态 |
| `graph_controlled` | 否 | 否 | 否 | 交给 graph runtime 控制 |

`is_interruptible` 只表示“运行中的 executor 可被边界控制打断”，不能再作为 stop 按钮的唯一依据。停止恢复点是 lifecycle close，不是 running interrupt。

## 文件级改造

1. `backend/harness/task_run_state_view.py`
   - 扩展 canonical view。
   - 增加 `control_capability` 和 `activity`。
   - 修正 `ready_to_continue` 不可 pause。

2. `backend/harness/runtime/run_monitor/projector.py`
   - `project_task_run()` 先生成 `state_view`。
   - monitor item 使用 canonical `activity` 和 `control_capability`。
   - stale 只作为 monitor overlay，不改写 TaskRun 控制能力。

3. `backend/harness/runtime/run_monitor/activity.py`
   - 从状态解释器降级为展示 adapter。
   - 优先消费 item 中已有的 canonical activity/control capability。
   - 不再根据 raw status 推导 resumable / interruptible。

4. `backend/harness/runtime/run_monitor/management.py`
   - `_actions_for_signal()` 使用 explicit capability：
     - `can_pause_task`
     - `can_resume_task`
     - `can_stop_task`
   - 删除基于 `running/resumable/interruptible` 的按钮推断。

5. `backend/harness/runtime/session_task_projection.py`
   - `_projection_control()` 与 `_projection_status()` 改为 canonical view adapter。
   - 删除 `ready_to_continue` 可 pause 的旧逻辑。

6. `frontend/src/lib/store/runtime.ts`
   - `updateSessionActivityFromLiveMonitor()` 优先消费后端 `activity_state/activity_label`。
   - raw `status` 只作 fallback。

## 验证

聚焦小测试，不引入巨大死测试：

1. `ready_to_continue`：`pause=false, resume=true, stop=true`。
2. `paused`：`pause=false, resume=true, stop=true`。
3. `active/running`：`pause=true, resume=false, stop=true`。
4. `waiting_approval`：`pause=false, resume=false, stop=true`。
5. `graph_controlled`：direct pause/resume/stop 都 false。
6. Runtime Monitor action 使用 canonical capability。
7. Session projection 与 runtime monitor 对同一个 TaskRun 的控制能力一致。
8. 前端 live monitor 优先使用后端 activity 投影。

## 成功判准

1. `ready_to_continue` 不再暴露 pause。
2. Runtime Monitor、session task projection、frontend session activity 对同一 TaskRun 状态一致。
3. 用户控制按钮不再由 UI raw status 推断。
4. 后端执行入口仍由 `task_executor` 做最终校验，projection 只决定展示和预检，不绕过真实控制边界。

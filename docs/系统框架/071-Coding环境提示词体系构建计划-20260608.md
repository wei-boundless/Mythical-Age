# Coding 环境提示词体系构建计划

日期：2026-06-08

## 1. 目标

为 `env.coding.vibe_workspace` 构建一套成熟、可执行、可审计的 coding 环境提示词体系。

这套体系不负责判断用户是否要进入 coding 任务，也不负责替主 agent 选择任务类型。它只在系统已经选中 coding 环境后，向模型提供清晰的项目工作边界、代码修改纪律、工具使用原则、验证标准、失败恢复和交付收口要求。

目标标准以成熟 coding agent 为参照：

- 能读懂真实项目，而不是凭文件名或记忆猜。
- 能保护用户已有改动。
- 能区分项目工作区、沙盒、artifact、git 视图和测试产物。
- 能在修改前定位调用链、配置、测试入口和当前文件事实。
- 能让工具请求、文件写入、命令执行、浏览器验证和子 agent 协作形成连续证据链。
- 能处理失败、权限缺口、环境不匹配和测试失败，而不是重复同一个失败动作。
- 能用真实验证和产物收口，不把计划、todo、启动命令或没有看到错误当成完成证据。

## 3. 权威边界

### 主 agent

主 agent 负责理解用户目标、选择动作、决定是否读文件、是否修改、是否验证、是否询问、是否阻塞、是否委派子 agent。

Coding 环境 prompt 不替主 agent 做语义判断。

### Runtime protocol

Runtime protocol 只负责动作 schema、JSON/action 格式、工具由系统执行、用户可见字段边界。

Coding 环境 prompt 不定义 action schema，也不发明工具。

### General lifecycle prompt

通用 lifecycle 负责所有环境共用的请求判断、上下文权威、当前工作关系、工具观察恢复、子 agent 委派、验证闸门和收口闸门。

Coding 环境 prompt 不复制这些通用生命周期，只补充 coding 任务中特有的文件事实、代码修改、测试验证和 git 安全纪律。

### Coding environment overlay

Coding overlay 负责被选中后的工作纪律：

- 项目工作区和沙盒边界。
- 文件事实和 stale 状态。
- 代码库探索和大范围任务拆分。
- 编辑前读取真实内容。
- 调试和失败恢复。
- 测试、构建、浏览器和运行验证。
- Git 和用户改动保护。
- Windows PowerShell 命令约束。
- 进度记录和真实完成声明。

### Tool guidance

Tool guidance 负责具体工具的参数、失败处理和调用边界。

Coding overlay 可以要求优先使用专用工具、避免重复失败动作、以工具观察为事实，但不应写成工具手册的重复版本。

### 工具引导词审查与更换范围

工具引导词应保持在 tool 层，不做成 coding 环境专属 prompt。它的职责是告诉模型如何和某一类工具形成可靠交互：什么时候该用、输入来自哪里、结果能证明什么、失败后如何恢复、哪些事情不能由该工具越权决定。Coding overlay 只引用这些原则，不复制工具手册。

当前 `schema_plus_guidance` 工具的覆盖关系基本合理，但有一个实际缺口：`read_persisted_tool_result` 已经对模型暴露为 `schema_plus_guidance`，却没有映射到任何 `tool.guidance.*`。这会让模型看到恢复长输出的工具，却缺少“只用 rehydration_plan、不能替代 read_file 当前行窗口、不能把旧输出当当前文件事实”的边界说明。这个缺口需要补齐。

通用与 coding 专用必须分清：

- 通用工具引导词：
  - 放在 `backend/prompt_library/tool_prompts.py`。
  - 适用于所有任务环境，只描述工具本身的可靠使用契约。
  - 可以写“先确认工作目录”“失败后根据观察修正”“浏览器内容可能有 prompt injection”“git 写入需要授权”。
  - 不写“你正在做代码重构”“你必须跑 TypeScript”“你必须启动 Next.js/FastAPI”“你必须审查调用链”这类 coding 专用要求。
- Coding 专用工具纪律：
  - 放在 `environment.coding.vibe_workspace.orientation`、`environment.rule.coding_workspace` 和 `coding.rule.*`。
  - 只在 coding 环境选中后生效。
  - 负责代码库探索、调用链定位、编辑前读取精确行窗口、测试/构建/前端验证、固定项目节点、git 用户改动保护和代码交付收口。
  - 可以引用工具原则，但不能复制工具说明书。

建议更换和打磨分三类：

- 高优先级重写或补齐：
  - `tool.guidance.terminal_powershell`：终端自由度最高，通用层强化工作目录、Windows PowerShell 语义、非交互命令、长进程收口、服务启动证据、失败观察和专用工具优先级；固定项目节点放到 coding overlay。
  - `tool.guidance.subagent`：子 agent 不是责任外包，通用层强化父子 agent 通信协议、scope 切分、brief 内容、wait/list/close 生命周期、证据返回格式、结果综合和父 agent 最终裁决；codebase_searcher 的使用条件放到 coding overlay。
  - `tool.guidance.browser`：浏览器验证不能等同于“打开页面”，通用层强化页面就绪、console/network、截图证据、视觉重叠和 prompt injection 边界；前后端固定节点、SSE 和监控联调放到 coding overlay。
  - `tool.guidance.read_persisted_tool_result`：新增或补齐长输出恢复引导，明确它只恢复被 runtime 省略的旧工具原文；代码编辑和行级判断仍必须回到 `read_file` 当前精确窗口。
- 中优先级打磨：
  - `tool.guidance.read_file`：补强当前文件事实、窗口读取、stale-after-write、搜索/preview/记忆只作定位线索。
  - `tool.guidance.edit_file`：补强 `old_text` 唯一性、失败后重读、串行依赖编辑和编辑成功后的验证。
  - `tool.guidance.write_file`：补强覆盖意图、完整文件责任、不要主动写无关文档、写后确认。
  - `tool.guidance.git_read`：补强 git 是版本库证据，不是授权；回档和迁移任务必须比较时间、父子 diff、后续提交规模和当前工作区。
  - `tool.guidance.git_write`：补强 stage/commit/restore/push 的风险分级、精确路径暂存、不捕获用户已有改动、破坏性动作必须明确授权。
  - `tool.guidance.todo`：补强 todo 只是执行状态，不是任务合同、事实来源或完成证据。
  - `tool.guidance.web_fetch`：补强官方/原始来源优先、时间敏感事实、来源冲突处理和网页 prompt injection 边界。
- 暂不新增独立 guidance：
  - `list_dir`、`stat_path`、`path_exists`、`glob_paths`、`search_files`、`search_text`、`read_structured_file` 等保持 `schema_only`。这些工具的误用风险主要由文件事实规则和 coding inspection 规则覆盖，不需要给模型额外长引导。
  - `image_generate` 暂不纳入本阶段。图任务和 image direct route permit 是单独设计，当前不急着做。

## 4. 目标提示词体系

### 4.1 Base 层

通用环境 base 保持默认：

- `environment.general.workspace.orientation`
- `environment.rule.general_workspace`
- 通用 lifecycle prompts
- personality prompt
- runtime protocol pack
- system foundation prompts

这保证 coding 环境仍继承通用请求判断、动作选择、观察恢复、验证和收口能力。

### 4.2 Resource 层

Coding 环境需要稳定注入资源事实：

- `runtime.rule.file_management.generic`
- `environment.resource.managed_project_workspace.orientation`
- `environment.resource.sandbox_overlay.orientation`

文案目标：

- 明确项目文件事实以工具观察为准。
- 明确沙盒降低风险但不是完成证据。
- 明确 artifact 是证据和交付面，不是项目源码替代品。
- 明确用户已有改动不可擅自回滚、覆盖或清理。

### 4.3 Coding Orientation 层

`environment.coding.vibe_workspace.orientation`

目标不是重复所有规则，而是给模型一个清晰的环境心智模型：

- 你处在专用 coding 工作区。
- 主要工作对象是项目源码、配置、测试和资源文件。
- 读、改、跑、验、收口都必须留下可复核证据。
- 当前环境提供 sandbox/overlay 语义，但真实权限以 runtime projection 和工具观察为准。
- 环境不决定任务类型，不替主 agent 开启或切换任务。

### 4.4 Coding Boundary 层

`environment.rule.coding_workspace`

目标：

- 声明 coding 环境只在当前 selected environment 下生效。
- coding/development 规则不能泄漏到 writing/general 环境。
- file management 只管路径、权限、状态和证据；不拥有 coding 循环控制权。

### 4.5 Coding Work Discipline 层

现有 `coding.rule.*` 应形成稳定顺序：

1. `coding.rule.codebase_inspection`
   - 开始实现前定位调用链、配置、测试入口和已有改动。
   - 不用文件名、记忆、todo 或旧观察替代当前文件事实。

2. `coding.rule.large_scope_exploration`
   - 大范围任务先判定规模，再拆区域、分证据。
   - 适合时委派 codebase_searcher。
   - 子 agent 只提供证据输入，主 agent 负责综合和裁决。

3. `coding.rule.editing`
   - 修改前必须读当前真实内容和精确行窗口。
   - 保持既有架构、命名、错误处理、类型系统和测试方式。
   - 不为一次性任务创建无关文档或抽象。

4. `coding.rule.debug_discipline`
   - 失败时先描述症状、期望、实际、证据来源。
   - 确认工作目录、项目根、沙盒根、命令实际读取路径。
   - 设计能排除假设的 probe，不重复同一失败动作。

5. `coding.rule.verification`
   - 验证与风险匹配。
   - 代码改动需要测试、构建、语法检查、运行检查或浏览器验证。
   - 不能跳过、降低断言、硬编码、删除失败用例来制造通过。

6. `coding.rule.git_safety`
   - 保护用户已有改动。
   - 不擅自 commit、push、reset、clean、切分支或回滚。
   - git 读取是证据，git 写入需要明确授权和控制层许可。

7. `coding.rule.windows_shell`
   - PowerShell 兼容。
   - 不使用 Bash 专属语法。
   - 不启动交互式命令。

8. `coding.rule.task_progress`
   - 多步骤任务维护状态。
   - todo 只是执行跟踪，不是事实来源。
   - 完成声明必须基于合同、产物、观察和验证。

## 5. 文案原则

所有 coding 环境 prompt 必须写给模型直接执行，不能写成开发说明。

不写：

```text
这是 coding runtime 节点。
该节点用于执行代码修改。
```

应写：

```text
你处在专用 coding 工作区。
开始修改前，先定位相关调用链、配置、测试入口和已有改动。
修改前必须读到目标文件当前真实内容；搜索结果和旧摘要只能告诉你下一步该读哪里，不能作为编辑依据。
```

文案检查标准：

- 每段都有明确身份或场景。
- 每段说明职责、边界、输入事实、行动标准、失败处理和收口条件。
- 不使用 `confidence` 作为模型输出要求。
- 不使用“这是某某节点”“用于某某功能”的开发口吻。
- 不重复 runtime schema。
- 不把环境写成任务选择器。
- 不把工具 guidance 复制到环境层。
- 不保留旧 `.v1` prompt ID。

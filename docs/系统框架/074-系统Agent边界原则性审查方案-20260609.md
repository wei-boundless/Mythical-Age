# 系统-Agent 边界原则性审查方案

## 定位

本文不是一次性修复计划，也不是某个 bug 的施工清单。

它是一套用于审查系统和 Agent 交互边界的原则性方法。目标是在排查慢回复、双回复、串线、暂停恢复异常、投影错乱、工具卡住、子 Agent 未收口、todo 状态漂移等问题时，先疏通主链路，再决定是否改代码，避免“修一处、堵一处”。

最高原则：

1. Agent 负责思考、语义判断、任务规划、动作选择、结果综合。
2. 系统负责环境提供、事实采集、权限准入、生命周期控制、执行边界、状态记录和前端投影。
3. 用户体验是第一约束：系统不能为了内部实现方便制造等待、重复回复、不可解释的阻塞或错误状态。
4. 系统只能在边界上控制风险，不能抢 Agent 的语义决策权。
5. 任何修复都必须让链路更单向、更可追踪、更少隐式状态；不能用新的分支掩盖旧链路冲突。

## 成熟 Agent 的审查基准

审查时统一使用如下权威链作为基准：

```text
RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> RuntimeStartPacket
-> ModelTurnDecision
-> ActionAdmission
-> ActionPermit
-> ExecutionLoop / ToolControlPlane
-> RuntimeObservation
-> Agent Follow-up
-> CanonicalOutput
-> RuntimeMonitor / Frontend Projection
```

这不是要求代码必须使用这些类名，而是要求权责必须单向。

每一层只允许做自己的事：

| 层 | 审查时允许看到的职责 | 一旦出现就要警惕的行为 |
| --- | --- | --- |
| RequestFacts | 记录用户消息、附件、会话、环境绑定、active turn 事实 | 判断意图、选择任务、默认续跑、决定工具 |
| BoundaryPolicy | 校验显式控制包、权限边界、生命周期前置条件 | 把普通用户输入改写成控制动作 |
| ContextCandidates | 收集候选上下文、记忆、文件、todo、active work、工具说明 | 让旧上下文覆盖用户最新消息 |
| RuntimeStartPacket | 装配本轮 prompt、上下文、工具、权限和可见状态 | 重新推断 intent、偷偷扩大权限、注入旧任务 |
| ModelTurnDecision | 由模型决定本轮语义动作、任务关系、下一步 | 被系统 fallback、regex、旧 classifier 替代 |
| ActionAdmission | 校验模型动作是否被本轮允许 | 被拒后换一个等价动作继续执行 |
| ActionPermit | 把获准动作绑定到具体工具或控制面 | 跳过 permit 直接执行工具 |
| ExecutionLoop | 执行动作、记录观察、把失败回灌给 Agent | 失败时伪造成功、猜测恢复、直接收口 |
| RuntimeObservation | 把工具结果、拒绝、暂停、子 Agent 状态等作为观察 | 把观察直接当最终答案 |
| Projection | 展示可见进度、状态、结果 | 用旧快照覆盖新状态、根据 UI 状态反推运行时语义 |

## 审查的第一步：先判权责，不先判 bug

遇到问题时，禁止从症状直接跳到补丁。

必须先问：

1. 这是自然语言输入、显式控制包、后台续跑、工具观察、监控投影，还是前端交互事件？
2. 当前这一步有没有合法权威去“决定语义”？
3. 如果它没有语义决策权，它是否只是记录事实、校验边界、执行动作、投影状态？
4. 如果它做了决定，这个决定是否已经由上游或下游另一个层做过？
5. 如果它失败了，失败是显式暴露，还是被 fallback 改成另一条链路？

只有权责判清楚后，才允许定位具体代码。

## 审查对象：所有交接点都要过一遍

每次审查不是只盯一个函数，而是沿着“交接点”检查。交接点是最容易串线、重复回复和卡顿的位置。

### 1. 前端输入到后端请求

审查问题：

- 前端是在发送自然语言输入，还是显式控制包？
- 是否因为当前有 active work，就把普通输入硬改成 steer？
- 当前 stream 是否被串行化？后续输入是排队还是并发开第二条流？
- UI 的 selected environment 是否只作为用户选择事实，而不是运行时反向推断？

判准：

- 前端可以排队、取消、重连、展示状态。
- 前端不能替 Agent 判断“这句话是不是继续当前任务”。
- 前端不能为了避免并发，把所有 active work 输入都变成 steer。

### 2. 请求事实到边界策略

审查问题：

- `active_turn_input_policy`、`expected_active_turn_id`、环境绑定、模型选择是否只是事实？
- 显式控制包失败时，是 blocked terminal，还是落回普通模型回合？
- 普通 `auto` 是否进入模型判断，而不是进入 steer fast path？

判准：

- `steer` 是显式控制。
- `auto` 是携带 active work 事实的普通模型回合。
- 系统可以拒绝控制包，但不能把普通输入提升成控制包。

### 3. 上下文候选到 RuntimeStartPacket

审查问题：

- 旧任务、memory、todo、monitor、recent outcome 是否只是候选材料？
- 是否有旧字段如 `task_selection`、`intent`、`selected_task_id`、`goal_frame` 继续影响本轮动作？
- prompt 中是否混入开发说明，而不是 Agent 可执行的角色任务说明？

判准：

- 上下文只能帮助 Agent 判断，不能替 Agent 判断。
- RuntimeStartPacket 只装配已允许的上下文、工具和权限。
- prompt 必须面向 Agent 角色、职责、输入、输出、裁决标准和失败处理。

### 4. 模型决策到动作准入

审查问题：

- 模型输出是否经过 schema 协议校验？
- action 是否经过 admission？
- admission deny 后，是形成观察让 Agent 修正，还是系统换一个动作继续？
- 模型是否被允许请求 task run、tool call、subagent、active work control？

判准：

- Agent 决定想做什么。
- 系统决定能不能做。
- 系统拒绝后只能返回拒绝事实或观察，不能替 Agent 换目标。

### 5. ActionPermit 到工具控制面

审查问题：

- 工具执行是否必须携带 permit？
- executor 是否绕过 tool control plane 直接执行？
- pause、stop、resume、replan 是否被保留为控制信号，而不是被统一吞成失败或停止？
- 工具失败是否保留结构化原因？

判准：

- permit 是工具执行的唯一通行证。
- executor 不能因为知道工具名就直接执行。
- 控制信号必须可追踪，不能在中间层丢失语义。

### 6. TaskRun 生命周期

审查问题：

- TaskRun 是由 Agent 明确请求、显式合同、还是旧状态猜出来的？
- 当前任务存在时，新任务是替换、继续、排队、拒绝，还是并发启动？
- `waiting_executor` 是可调度状态，还是被当成 terminal 卡死？
- 暂停后恢复是继续同一任务，还是需要重启任务？依据是什么？

判准：

- 生命周期状态必须是显式状态机，不靠扫描旧日志猜。
- `waiting_executor` 不能被取消机制误杀成无法继续。
- resume 必须检查恢复条件：任务合同、权限、pending approval、runtime control、最新观察。
- 需要重启时必须显式形成新的 task run 或 replacement 边界，不能悄悄复用旧任务。

### 7. Todo

审查问题：

- todo 是否只是执行清单？
- 是否用 todo completed 作为任务完成依据？
- 是否有多个 `in_progress`？
- todo 初始化是否反向改变任务合同？

判准：

- todo 是可见执行状态，不是任务权威。
- 完成判定来自 Agent 综合、合同验收和执行观察，不来自 todo。

### 8. 子 Agent

审查问题：

- 子 Agent 是否由父 Agent 明确 spawn？
- 父 Agent 是否 wait/list 后再综合子 Agent 结果？
- 子 Agent 的上下文是否污染父线程？
- 父任务完成前是否还有 active child？

判准：

- 子 Agent 是隔离执行单元。
- 子 Agent 结果是证据，不是自动最终答案。
- 父 Agent 必须负责收口。

### 9. Steer 和用户额外引导

审查问题：

- steer 是显式控制包，还是普通用户输入？
- 用户在任务期间追加的引导，是否排队后作为下一次发送给模型的高权威用户补充信息？
- steer 是否被持久化为本会话重点，而不是被误写成长期偏好？
- steer 是否可能绕过 Agent 判断，直接改写执行状态？

判准：

- steer 的本质是用户在当前任务期间追加的约束或方向。
- 它应该在任务期间下一次模型调用时作为用户提供的额外引导进入上下文，并要求模型不可忽视。
- 系统负责收集、排序、去重、注入和过期控制。
- 系统不负责替模型判断 steer 与任务的语义关系，除非这是一个显式控制包且前置条件完整。

### 10. Monitor 和前端投影

审查问题：

- monitor 是从权威运行状态投影，还是自己扫描事件猜状态？
- stale snapshot 是否会覆盖新状态？
- terminal history 是否混进 live running？
- 前端是否用 monitor 反向决定后端运行语义？

判准：

- monitor 只展示，不决策。
- 投影必须可被 source revision / run id / task run id 约束。
- stale 数据必须被拒绝，不能部分应用。

### 11. 记忆、压缩和恢复

审查问题：

- session memory 是恢复材料，还是长期偏好？
- compact 是否丢失用户最新约束、未解决错误、工具结果引用、文件状态？
- resume 是否清理未闭合 tool use、孤儿 assistant、半截 action？
- 恢复是否靠 explicit runtime state，而不是靠历史文本猜？

判准：

- 记忆服务上下文，不拥有本轮意图。
- 压缩可以降噪，不能改写事实。
- 恢复必须保留协议合法性和任务状态合法性。

## 修复前必须做的链路问答

每一个准备修复的点，都必须先回答：

1. 这个点现在属于哪一层权威？
2. 它正在做的决定是否属于自己？
3. 它的输入事实从哪里来？是否可追踪？
4. 它输出的状态给谁消费？是否有多个消费者各自解释？
5. 它失败时应该 blocked、ask user、observation、retry、resume，还是 terminal？
6. 修复后是否减少了决策源，还是又新增了一个分支？
7. 是否有旧字段、旧 prompt、旧测试继续把旧链路拉回来？
8. 用户是否会感知到更少等待、更少重复、更清晰状态？

答不清楚时，不能直接改代码。

## 反模式清单

以下行为一律视为边界污染：

1. 用 `intent`、`task_selection`、regex、旧 classifier 替模型决定本轮语义。
2. 用 `auto` 伪装 steer。
3. 显式控制失败后落入普通模型回合。
4. admission deny 后换另一个 action 继续执行。
5. executor 在工具失败后伪造完成。
6. pause / stop / resume 信号在工具层被吞掉。
7. todo completed 被当成任务完成。
8. 子 Agent 未 wait 就被父任务引用为结论。
9. monitor stale event 部分覆盖新状态。
10. 前端为了防并发而改变输入语义。
11. prompt 写成“这是 runtime 节点”，而不是写成 Agent 能执行的角色职责。
12. 旧兼容字段没有明确删除条件，却继续参与主链路。
13. 测试只保护旧内部结构，不保护用户可见行为。

## 慢回复和不流畅问题的审查方式

慢回复不能只看模型速度。必须拆成五段：

```text
frontend queue / stream gate
-> backend request admission
-> prompt/context assembly
-> model first token latency
-> tool/task executor wait
-> monitor/projection delivery
```

审查顺序：

1. 是否存在第二条并发 stream 或前端排队等待未释放。
2. 是否进入了不该进入的控制 fast path，或控制失败后又开模型回合。
3. 是否 prompt cache 因动态上下文抖动失效。
4. 是否任务处于 `waiting_executor` 但 scheduler 没有接管。
5. 是否等待 tool approval、subagent wait、pending steer、completion verification。
6. 是否后端已经产出事件，但前端 projection 因 stale revision、cursor、session mismatch 没展示。
7. 是否 monitor 请求频率、SSE 重连或 latest-run 选择了错误 run。

判准：

- 如果模型首 token 慢，是模型或 prompt/cache 问题。
- 如果模型前已经等待，是系统 admission、queue、context assembly 或 scheduler 问题。
- 如果后端已有事件但 UI 慢，是 projection 或前端 stream gate 问题。
- 如果任务显示等待但其实可继续，是生命周期调度问题。

## 双回复和串线问题的审查方式

双回复通常不是“模型啰嗦”，而是链路上出现了两个合法消费者或两个启动源。

必须检查：

1. 同一用户输入是否生成了两个 run。
2. 显式 steer ACK 是否被 latest-run 当成主回复重连。
3. auto active-work 判断回合是否被前端纳入 active stream 串行保护。
4. executor auto-continue 是否和用户新输入并发启动。
5. monitor projection 是否把旧 terminal 输出再次投影成当前回答。
6. resume 是否把上一次未完成 assistant 消息重新送进模型。

判准：

- 一个用户语义输入最多只能有一个模型主回合。
- 显式控制 ACK 不是主回答。
- 后台 executor 续跑不能抢占用户当前主回合的展示位置。
- resume 可以恢复上下文，但不能复活旧输出。

## 暂停、恢复、重启的判准

暂停后重新发送，不能简单理解成“继续”或“重启”。要由状态机判定。

审查时按这个顺序：

1. 任务是否仍有合法 TaskRun、contract、session、environment。
2. 当前状态是否可恢复：`waiting_executor`、paused、blocked recoverable、waiting approval。
3. 是否存在未解决 tool approval 或权限拒绝。
4. 是否存在 pending steer 或用户新约束，需要先进入下一轮模型。
5. 上一次中断点是否在工具前、工具后、模型 action 前、模型 action 后。
6. 重启是否会造成重复副作用。
7. 如果不能恢复，是否明确生成 replacement / restart，而不是静默复用旧任务。

判准：

- 可恢复就恢复，不阻断 Agent。
- 需要模型重新判断就把恢复事实作为上下文交给 Agent。
- 需要重启就显式重启，保留旧任务终态和原因。
- 不允许用“取消 waiting_executor”来规避调度问题。

## Prompt 审查标准

所有写给 Agent 的 prompt 都必须通过以下检查：

1. 是否直接告诉 Agent 它是谁？
2. 是否说明它负责什么、不负责什么？
3. 是否说明输入材料如何使用？
4. 是否说明输出格式或动作选择边界？
5. 是否说明裁决标准？
6. 是否说明失败、缺信息、冲突、权限不足时如何处理？
7. 是否避免了开发实现语句？

不合格示例：

```text
这是 runtime 节点。
根据任务图执行 world_review。
这个节点用于校验资产。
```

合格方向：

```text
你是一名世界观审核员。
你只负责评审当前世界观设定是否完整、一致、可支撑后续写作。
你不负责替创作者扩写设定。
你需要指出问题、给出裁决、说明是否允许进入下一阶段。
```

## 判断一个修复是否成熟

一个修复必须同时满足：

1. 决策源更少，而不是更多。
2. 状态机更明确，而不是靠字段拼凑。
3. 失败更可见，而不是被 fallback 吞掉。
4. 用户输入语义不被系统改写。
5. Agent 收到的信息更完整、更稳定、更少噪声。
6. 权限和副作用边界更严格。
7. 前端体验更流畅，且不靠语义偷换换取流畅。
8. 旧链路、旧字段、旧测试有明确删除或迁移。

如果一个修复只是“这里多挡一下、那里多兜底一下”，即使测试通过，也不能算成熟。

## 审查输出格式

每次边界审查输出应保持这个形状：

```text
问题现象：
入口类型：
涉及交接点：
当前权威链：
越权/重复决策点：
状态或投影混乱点：
用户体验影响：
应删除的旧链路：
应保留的系统边缘控制：
应交还给 Agent 的语义责任：
验证方式：
```

这份输出不是计划书。只有当审查结论显示需要跨模块修改时，才另行写实施计划。

## 最终审查原则

正确的系统不是替 Agent 多想一点，而是让 Agent 能稳定地想、能及时拿到事实、能被清楚地授权或拒绝、能把失败转成下一步。

正确的边界不是把问题堵住，而是让每个问题沿着唯一权威链自然流动：

- 用户输入进入 Agent 判断。
- 显式控制进入系统校验。
- 工具动作进入 permit。
- 执行失败进入 observation。
- 状态变化进入 monitor。
- 展示只做 projection。

只要某处开始“顺手替别的层决定”，链路迟早会再次变慢、串线或双回复。

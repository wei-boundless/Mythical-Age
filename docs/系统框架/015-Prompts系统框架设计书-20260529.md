# Prompts 系统框架与设计原则

日期：2026-05-29

## 1. 定位

Prompts 系统是稳定 prompt 资产的统一管理层。它负责保存、分类、版本化、审计和装配稳定 prompt 资源，但不负责判断用户意图、不负责授权工具、不负责启动任务、不负责保存运行时事实。

本系统服务于单 agent harness、TaskRun、图任务节点、特定任务、技能、环境和角色表达。它的目标是让 agent 每一次被调用时都收到清晰、可审计、可缓存、边界明确的 prompt packet。

## 2. 当前链路事实

### 2.1 单 agent 主链

当前主链：

```text
backend/query/runtime.py
-> AgentHarness
-> backend/harness/runtime/assembly.py::assemble_runtime
-> backend/harness/runtime/compiler.py::RuntimeCompiler
-> backend/harness/loop/agent_loop.py / task_executor.py
```

当前实际进入模型的 prompt 来源：

| 来源 | 当前位置 | 是否主链有效 | 说明 |
|---|---|---:|---|
| runtime system prompt | `backend/harness/runtime/compiler.py` | 是 | turn_action、task_execution、observation_followup 仍硬编码 |
| runtime projection | `backend/harness/runtime/compiler.py` | 是 | 已替代 mode instruction，表达本次运行边界 |
| agent work role prompt | `storage/orchestration/agent_runtime_profiles.json` | 是 | 目前内联在 profile metadata |
| environment prompt | `backend/task_system/environments/default_environments.py` | 是 | 目前内联在 environment definition |
| soul role prompt | `backend/soul/assembly_service.py` | role mode 有效 | 只作为角色表达锚点 |
| tool manifest | `backend/harness/runtime/assembly.py` | 是 | 不是 prompt asset，是本次可见能力投影 |
| observations / task_run state | runtime state | 是 | 是 volatile state，不属于 prompts 库 |

### 2.2 图任务与特定任务链路

图任务 prompt 当前链路：

```text
GraphHarnessConfig.nodes[*].prompt
-> GraphContextMaterializer.input_package.prompt_contract
-> WorkOrder
-> TaskRunContract.prompt_contract
-> RuntimeCompiler.compile_task_execution_packet
```

标准字段：

```text
role_prompt
task_instruction
output_instruction
forbidden_behavior
definition_of_done
```

这套结构应成为图节点、特定任务、单节点契约的统一 prompt contract。

### 2.3 旧 prompt 链路

旧链路：

```text
backend/prompting/*
backend/prompt_library/*
storage/prompt_library/prompt_resources.json
```

现状：

- `backend/prompting/*` 仍被 `query.runtime.build_static_system_prompt_for_session()`、`sessions/tokens API` 使用。
- `backend/prompt_library/*` 已有资源模型雏形，但当前混合了旧 `task_goal_type` selector。
- `storage/prompt_library/prompt_resources.json` 主要保存写作图任务 `stage_role` prompt。

目标：

- `backend/prompting/*` 不再作为新 harness prompt 来源。
- `backend/prompt_library/*` 升级为正式 Prompts Library。
- 旧 `stage_role` 迁移为 `graph_node.role` 类资源。
- 普通 turn 不再基于 `task_goal_type` 选择 prompt。

## 3. 设计原则

### 3.1 Prompt Asset 与 Runtime State 分离

Prompt asset 是稳定资产，必须可版本化、可复用、可缓存、可审计。

Runtime state 是本次运行事实，不能进入 Prompts Library。

Prompt asset 包括：

- runtime prompt pack
- agent role prompt
- environment prompt
- task prompt contract
- graph node prompt
- skill prompt
- soul role prompt

Runtime state 包括：

- 当前用户消息
- history
- observations
- tool results
- task_run status
- active/historical failures
- artifact evidence
- operation_authorization
- runtime envelope
- agent_visible_runtime_projection

### 3.2 Mode 是装配控制，不是 Prompt 文案

Mode 只能控制 runtime 装配：

- 可用动作集合
- 是否允许 TaskRun
- 是否装配 soul role prompt
- self-review 强度
- step summary / todo 策略
- 上下文范围
- 权限和工具可见性策略

Mode 不能直接生成模型可见文案。

禁止：

```text
当前 runtime 是 professional 模式。
```

应该：

```text
本次运行边界：
- 你可以直接回答、询问用户、调用本次可见工具。
- 当目标需要真实交付物、持续执行或验证闭环时，可以请求正式 TaskRun。
```

### 3.3 Prompts Library 不做决策

Prompts Library 只负责稳定 prompt 资产管理。

它不负责：

- 判断用户意图
- 从自然语言猜任务
- 启动 TaskRun
- 授权工具
- 改写 task contract
- 保存 observations
- 保存 task_run 当前状态
- 根据旧 task_goal_type 给普通 turn 选 prompt

### 3.4 显式引用优先

Prompt 装配应优先来自显式引用：

- runtime prompt pack refs
- agent prompt refs
- environment prompt refs
- task prompt contract
- graph node prompt contract
- skill prompt refs
- soul prompt ref

只有任务合同、图节点配置、agent profile、environment config 这类明确结构可以提供 prompt 选择依据。不得通过关键词或隐式分类器选择 prompt。

### 3.5 Prompt 必须写给 Agent

Agent prompt 必须让目标 agent 明确理解身份、职责、边界、输入、输出、裁决标准和失败处理。

禁止写开发说明：

```text
这是 runtime 节点。
根据任务图执行 world_review。
这个节点用于校验资产。
```

应写成可执行职责：

```text
你是一名世界观审核员。
你只负责评审当前世界观设定是否完整、一致、可支撑后续写作。
你不负责替创作者扩写设定。
你需要指出问题、给出裁决、说明是否允许进入下一阶段。
```

### 3.6 环境 Prompt 只描述资源边界

Environment prompt 只描述：

- 资源边界
- 存储边界
- 沙盒约束
- 文件边界
- artifact 区域
- 验证通道

Environment prompt 不描述：

- 工具授权
- skills 装配
- agent 工作角色
- 任务类型
- 用户意图

工具是否可见由 agent/profile/capability/runtime authorization 决定。

### 3.7 Skill Prompt 属于 Agent 能力装配

Skill prompt 可以进入 Prompts Library 索引，但是否装配由 agent 能力配置决定。

Environment 不装配 skill。

Skill prompt 不应成为系统强制行为；它只在 agent 获得该 skill 能力时作为能力说明进入 runtime packet。

### 3.8 Soul Prompt 只属于 Role Mode

Soul prompt 是角色表达锚点。

它不能：

- 改变工具权限
- 改变任务生命周期
- 改变输出契约
- 覆盖 runtime boundary
- 覆盖 task contract
- 进入 standard/professional 默认工作链路

### 3.9 Graph / Specific Task 使用统一 Prompt Contract

图节点、单节点特定任务、特定任务计划应使用统一结构：

```text
role_prompt
task_instruction
output_instruction
forbidden_behavior
definition_of_done
```

这套结构是 agent 可见任务职责，不是系统执行说明。

### 3.10 Prompt Cache 优先保护稳定前缀

Prompt cache 应按稳定性分层：

- global static
- environment stable
- task stable
- dynamic projection
- volatile state

RuntimeCompiler 必须保持 stable prompt sections 在 volatile state 之前。

Observations、tool results、user message 不得进入 cacheable static prompt。

Prompt cache 不是健康系统或 UI 的统计功能，而是模型请求构造层的运行时约束。成熟 agent 的 cache 设计必须满足：

- cache boundary 由显式 segment metadata 决定，不由 message 文案关键词推断。
- cacheable prefix 的字节序列必须可复现、可审计、可解释。
- dynamic projection、operation authorization、当前时间、随机 id、运行状态、错误观察不得污染 global static prefix。
- task stable 只能包含任务合同、图节点合同、完成标准等不可变内容；`task_run.status`、`updated_at`、当前步骤、active failures 必须属于 volatile state。
- provider cache 命中、miss、创建和读取 token 必须回写到账本，且能解释 cache break 原因。
- 若 provider 支持显式 cache control，请求构造层负责把 segment plan 转换成 provider 原生结构；若 provider 只支持自动前缀缓存，请求构造层仍必须保持 byte-stable prefix。

禁止：

- `CanonicalPromptSerializer` 通过 `"当前情境"`、`"Runtime Execution Facts"` 等文本片段判断 segment 类型。
- 把完整 `task_run`、完整 observations 或完整 tool results 放入 stable system payload。
- 为了提高命中率把动态事实错误标记为 stable。
- 在 retry、delegate、subagent 调用中重新渲染已经存在的 stable prompt 字节。

### 3.11 每次调用必须可审计

每个 RuntimeInvocationPacket 都必须能追踪：

- 使用了哪些 prompt resources
- prompt refs 的版本
- 哪些 section 进入 static prefix
- 哪些 section 是 task stable
- 哪些 section 是 dynamic projection
- 哪些输入是 volatile state
- 哪些 prompt 被拒绝装配及原因
- 预估 token 和 cache 边界
- 实际模型请求的 segment plan
- provider cache policy 和是否已应用到请求
- provider usage 中的 cached/cache_read/cache_creation token
- cache break 检测结果和失效原因

### 3.12 旧链路不能以兼容名义保留为主链

旧 `prompting`、旧 `prompt_library selector`、旧 `task_goal_type` prompt 选择逻辑，不得继续作为新单 agent harness 的主链权威。

保留旧资产只允许用于：

- 显式迁移
- 只读审计
- 已声明的 graph/task prompt resource

不得作为 fallback 隐式参与新 runtime prompt 装配。

## 4. 目标架构

```text
PromptResourceStore
  -> 保存稳定 prompt 资产

PromptPackRegistry
  -> 定义 runtime / task / graph / skill 等场景需要的 prompt 组合

PromptAssemblyService
  -> 根据 RuntimeAssembly 的显式 refs 和合同字段装配 prompt sections

RuntimeProjectionBuilder
  -> 根据本次 runtime 装配结果生成动态运行边界

RuntimeCompiler
  -> 合成 RuntimeInvocationPacket:
     1. stable prompt sections
     2. stable contract payload
     3. dynamic runtime projection
     4. volatile user/history/observation state
  -> 同时生成 PromptSegmentPlan

PromptManifest
  -> 记录本次 packet 使用的 prompt refs、segment plan、动态投影、cache 边界和 token 估算

ModelRequestBuilder
  -> 按 PromptSegmentPlan 生成真实 provider request bytes
  -> 保证 stable prefix 字节稳定

ProviderCachePolicyAdapter
  -> 把 segment plan 转换为 provider 支持的 cache policy
  -> 不支持显式 cache control 时保留自动前缀缓存优化

PromptAccounting
  -> 基于真实 provider request 生成 PromptSegmentMap、token prediction、cache record
  -> 运行后读取 provider usage 作为账单真值

PromptCacheBreakDetector
  -> 对比相邻请求的 system hash、tool schema hash、model、thinking config、cache policy、boundary hash
  -> 当 cached tokens 下降或 prefix hash 改变时输出失效原因
```

## 5. 层级职责

### 4.1 Prompts Library

职责：

- 保存 prompt resource。
- 管理 category、subtype、version、status。
- 管理 prompt pack。
- 按显式 refs 和明确 contract fields 返回 prompt sections。
- 生成 PromptAssemblyResult。

禁止：

- 不判断用户意图。
- 不从自然语言猜任务。
- 不启动 TaskRun。
- 不授权工具。
- 不保存 runtime state。
- 不把 `task_goal_type` 当普通 turn 的选择权威。

### 4.2 Runtime Assembly

职责：

- 解析 agent profile。
- 解析 runtime mode 为装配策略。
- 解析 task environment。
- 建立工具可见性和操作授权投影。
- 收集 prompt refs。
- 装配 soul role prompt 的可用性边界。

Runtime Assembly 可以决定“本次应该引用哪些 prompt”，但不渲染最终模型输入。

### 4.3 Runtime Projection

职责：

- 把本次已装配结果转成 agent 可理解的运行边界。
- 描述本次允许的 action types。
- 描述 TaskRun 是否允许。
- 描述工具、权限、环境、artifact、自审、step summary 边界。

Runtime Projection 是动态事实，不进入 Prompts Library。

### 4.4 Runtime Compiler

职责：

- 读取 PromptAssemblyResult。
- 读取 RuntimeProjection。
- 读取 volatile state。
- 输出 RuntimeInvocationPacket。
- 输出 PromptSegmentPlan。
- 保持 global static / environment stable / task stable / dynamic projection / volatile 的顺序和 cache 边界。
- 只把不可变合同放入 stable contract payload。
- 把 `task_run` 当前状态、execution_state、observations、active failures 放入 volatile payload。

Runtime Compiler 不应长期保存大段 prompt 文案。

### 4.5 Model Request Builder

职责：

- 消费 RuntimeInvocationPacket 和 PromptSegmentPlan。
- 生成真实发送给 provider 的模型请求结构。
- 保证同一 stable segment 在同一 session/task/fork 下生成 byte-exact 内容。
- 将 provider 支持的 cache control、TTL、scope、boundary 写入请求。
- 对不支持显式 cache control 的 provider，保持自动前缀缓存可命中的消息顺序和字节稳定性。

禁止：

- 重新用文本关键词推断 segment。
- 在请求构造阶段重排 PromptSegmentPlan 未授权重排的段。
- 在 stable prefix 中注入当前时间、随机 id、运行状态或临时错误。

### 4.6 Prompt Accounting

职责：

- 基于 ModelRequestBuilder 的真实请求生成 PromptSegmentMap。
- 运行前记录 local prediction。
- 运行后记录 provider usage。
- 记录 PromptCacheRecord。
- 为 Context Compression、Health System、Monitor 提供同一事实源。

Prompt Accounting 不能改变模型请求，只记录和分析请求。

### 4.7 Prompt Cache Break Detection

职责：

- 在模型调用前记录 prompt state snapshot。
- 在模型调用后读取 provider cached/cache_read/cache_creation token。
- 对比同一 session/task 的前后请求。
- 输出 cache break 原因。

必须跟踪：

- system prompt hash
- stable prefix hash
- tool schema hash
- model/provider/base_url
- reasoning/thinking config
- cache policy / TTL / scope
- segment boundary
- runtime mode/profile/environment refs
- retry/fork/delegate 是否复用父级 stable bytes

### 6.6 PromptSegmentPlan

`PromptSegmentPlan` 是请求构造和 prompt accounting 的共同权威。它由 RuntimeCompiler 生成，由 ModelRequestBuilder 和 CanonicalPromptSerializer 消费。

```json
{
  "segment_plan_id": "segplan:...",
  "packet_id": "rtpacket:...",
  "provider_policy_ref": "pcache-policy:...",
  "segments": [
    {
      "segment_id": "segplan:1",
      "kind": "global_static",
      "source_ref": "runtime.pack.task_execution.v1",
      "cache_scope": "global",
      "cache_role": "cacheable_prefix",
      "compression_role": "preserve",
      "content_hash": "sha256:...",
      "ordinal": 1,
      "model_message_role": "system"
    }
  ],
  "authority": "runtime.prompt_segment_plan"
}
```

Segment kind 固定集合：

- `global_static`
- `environment_stable`
- `agent_stable`
- `task_stable`
- `tool_schema_stable`
- `dynamic_projection`
- `volatile_user`
- `volatile_history`
- `volatile_observation`
- `volatile_task_state`

`CanonicalPromptSerializer` 不得自行猜测 kind；如果缺少 `PromptSegmentPlan`，只能标记为 `unknown_unplanned` 并使 cache plan bypass。

### 6.7 ProviderCachePolicy

```json
{
  "policy_id": "pcache-policy:...",
  "provider": "deepseek",
  "model": "deepseek-v4-pro",
  "strategy": "automatic_prefix | native_cache_control | disabled",
  "scope": "global | org | session | task | none",
  "ttl_seconds": 300,
  "boundary_segment_id": "segplan:5",
  "cache_control_applied": false,
  "cache_safety_reasons": [],
  "authority": "runtime.provider_cache_policy"
}
```

策略说明：

- OpenAI-compatible / DeepSeek 这类自动前缀缓存 provider，系统主要保证 byte-stable prefix，并用 provider usage 校验命中。
- Anthropic native 或其它支持显式 cache control 的 provider，系统把 segment boundary 转为 provider 原生 request block。
- 不支持缓存或存在安全风险时，strategy 必须是 `disabled`，并记录原因。


```text

```

## 9. 缓存分层

```text
global static:
  runtime prompt pack
  common contract
  stable agent role prompt

environment stable:
  environment prompt

task stable:
  task prompt contract
  graph node prompt contract

dynamic projection:
  runtime projection
  operation authorization summary

volatile:
  user message
  history
  observations
  tool results
  task_run state
  active failures
```

### 9.1 请求顺序规则

模型请求必须按以下顺序构造：

```text
1. global_static
2. agent_stable
3. environment_stable
4. task_stable
5. tool_schema_stable
6. dynamic_projection
7. volatile_user / volatile_history / volatile_observation / volatile_task_state
```

说明：

- global static 尽量跨 session 复用。
- agent/environment/task stable 允许在 session/task 范围内稳定。
- dynamic projection 可重复生成，但不能进入 global static prefix。
- volatile 永远位于 stable boundary 之后。

### 9.2 Retry / Delegate / Subagent 缓存规则

Retry：

- retry 不重新编译 stable prompt。
- retry 可以改变 request id，但不能改变 stable segment bytes。
- retry 的 cache key 不得包含 request id。

Delegate / Subagent：

- 父 agent 必须把已渲染 stable prompt bytes 或 segment refs 传给子 agent。
- 子 agent 不应重新扫描环境、重新渲染 stable sections，除非明确进入新的隔离 runtime。
- fork 类调用必须保留父级 model/provider/thinking config/tool schema 的 cache-safe 参数。

### 9.3 Cache Break 诊断

当缓存未命中或 cached tokens 显著下降时，系统必须能判断至少一种原因：

- stable prompt text changed
- tool schema changed
- model/provider changed
- thinking/reasoning config changed
- cache policy changed
- boundary changed
- dynamic content entered stable prefix
- task mutable state entered stable prefix
- provider did not return usage
- provider does not support cache accounting

## 10. 最终能力

Prompts 系统完成后应支持：

- 自由分类管理 prompt 资产。
- prompt 资源版本化、启停、审计。
- runtime prompt pack 可替换。
- agent/environment/task/graph/skill/soul prompt 可组合装配。
- 每次模型调用生成 PromptManifest。
- 每次模型调用生成 PromptSegmentPlan。
- ModelRequestBuilder 按 segment plan 生成真实 provider request。
- prompt cache 可依据 manifest 和真实 request 精确计算。
- provider cache hit/miss 能被账本和 cache break detector 解释。
- retry、delegate、subagent 能复用 byte-stable prefix。
- 旧 prompt 链路不再影响新单 agent harness。

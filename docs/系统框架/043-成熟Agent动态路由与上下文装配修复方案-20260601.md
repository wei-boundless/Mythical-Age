# 成熟 Agent 动态路由与上下文装配修复方案

日期：2026-06-01

状态：实施中。已修复动态 `task_state` 收敛问题，并开始移除 prompt/runtime 中的文本关键词路由。

## 1. 背景与已修问题

近期长任务跟踪暴露出两个相互关联的问题：

- 模型可见 prompt 中，任务执行状态曾由 `execution_state`、`observations`、`work_history`、`task_run_state` 多路并列进入，导致同一个工具失败被重复描述，动态段快速膨胀。
- runtime ledger 已记录了结构化错误，但模型可见投影可能丢失字段。例如 `payload.result` 是 JSON 字符串时，`structured_error.provider_retryable`、`agent_retry_policy`、`attempts` 没有进入 `task_state`。

已修复的具体 bug：

- `backend/harness/runtime/dynamic_context/tool_result_projector.py` 现在会解析工具结果里的 JSON 字符串，抽取 `structured_error`、`status`、`error`、artifact refs 等结构化字段。
- `backend/harness/runtime/dynamic_context/structured_error_projection.py` 统一保留 `provider_retryable`、`agent_retry_policy`、`attempts`。
- `backend/harness/runtime/dynamic_context/task_state_projector.py` 将任务执行的模型可见动态状态收敛到单一 `task_state`。
- 相关测试已覆盖 JSON 字符串工具结果、结构化错误保留、失败去重、非代码任务不装配 Python 开发策略。

## 2. 外部成熟实现对照

本方案只借鉴成熟 agent 的结构不变量，不照搬产品形态。

### 2.1 Claude Code：稳定前缀与动态段硬分界

本地源码证据：

```text
D:\AI应用\claude-code-nb-main\constants\prompts.ts
D:\AI应用\claude-code-nb-main\utils\api.ts
D:\AI应用\claude-code-nb-main\constants\systemPromptSections.ts
D:\AI应用\claude-code-nb-main\utils\forkedAgent.ts
D:\AI应用\claude-code-nb-main\tools\AgentTool\runAgent.ts
D:\AI应用\claude-code-nb-main\services\AgentSummary\agentSummary.ts
```

关键做法：

- 用 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 把 system prompt 拆成静态段和动态段。
- 静态段可进入 cache scope；动态段不进入全局稳定前缀。
- `systemPromptSection()` 默认缓存 section，只有显式 `DANGEROUS_uncachedSystemPromptSection()` 才允许每轮变化。
- fork / summary / memory 等后台 agent 通过 `CacheSafeParams` 继承父请求的 system prompt、tools、model、messages prefix 和 thinking config。
- 不通过改变工具列表来禁止 fork agent 使用工具，而是保留相同工具定义，再用 `canUseTool` 拒绝工具调用，避免破坏 cache key。

可迁移结论：

```text
缓存命中不是靠“手写 prompt cache”实现的。
缓存命中来自稳定的请求前缀、稳定的工具定义、稳定的模型/thinking 配置，以及严格的动态段隔离。
```

### 2.2 Codex：turn context 是结构化权威，不靠 prompt 猜路由

本地源码证据：

```text
D:\AI应用\openai-codex\codex-rs\core\src\session\turn_context.rs
D:\AI应用\openai-codex\codex-rs\core\src\state\additional_context.rs
D:\AI应用\openai-codex\codex-rs\core\src\session\tests.rs
D:\AI应用\openai-codex\codex-rs\core\src\session\turn.rs
D:\AI应用\openai-codex\codex-rs\app-server\tests\suite\v2\turn_start.rs
```

关键做法：

- `TurnContext` 是结构化对象，包含模型、权限、环境、用户指令、动态工具、输出 schema 等。
- `AdditionalContextStore` 按 trust level 进入不同角色：`Application` 进入 developer fragment，`Untrusted` 进入 user fragment。
- resume 后初始上下文只在第一次 context update 注入，不会每轮重复灌入。
- root thread 和 subagent thread 的 usage hint 由 session source 决定，不由模型在混合 prompt 中猜身份。
- plan、工具调用、approval、interrupt 都是 turn 事件和状态，不是靠 prompt 文本隐式路由。

可迁移结论：

```text
动态路由的成熟做法是 runtime 先确定当前 turn 的结构事实，再把结构事实投影给模型。
模型负责语义判断和下一步行动，不负责从历史任务、监控、旧 prompt 混合物里猜当前控制权。
```

## 3. 当前项目主要差距

### 3.1 动态状态曾经多路进入模型

涉及文件：

```text
backend/harness/runtime/compiler.py
backend/harness/runtime/dynamic_context/manager.py
backend/harness/runtime/dynamic_context/observation_projector.py
backend/harness/runtime/dynamic_context/execution_state_projector.py
backend/harness/runtime/dynamic_context/work_history_projector.py
```

旧问题：

```text
execution_state
observations
work_history
task_run_state
```

这些内容在同一个 volatile message 中并列出现，形成重复事实源。工具失败可能同时出现在 active failure、observation summary、work history、receipt 中。

目标结构：

```text
volatile_task_state:
  task_state:
    runtime_status
    current_step
    current_facts
    latest_tool_results
    active_failures
    historical_failures
    artifact_evidence
    pending_user_steers
    active_contract_revisions
    work_progress
    task_run_state
    runtime_boundary
```

### 3.2 ledger 与模型可见投影权威不一致

涉及文件：

```text
backend/harness/runtime/dynamic_context/tool_result_projector.py
backend/harness/runtime/dynamic_context/observation_projector.py
backend/harness/loop/task_executor.py
backend/runtime/prompt_accounting/*
```

问题不是“账本有没有记录”，而是：

```text
ledger 记录的是运行事实；
projection 决定模型下一轮实际看见什么。
```

如果字段只在 ledger 中存在，但模型可见投影丢失，agent 仍会误判。例如 provider 允许 retry，但 agent policy 禁止自动 retry，这种差异必须进入模型可见错误对象。

### 3.3 routing prompt 与 task execution prompt 混装

涉及文件：

```text
backend/harness/runtime/compiler.py
backend/prompt_library/packs.py
backend/prompt_library/registry.py
```

旧问题：

- `task_execution` packet 中可能装入普通 turn routing 文案。
- 非代码任务可能装入 Python 开发策略。
- 模型收到“当前是任务执行循环”和“普通 turn 中决定是否开启持续流程”两类冲突指令。
- 曾出现过错误修复方向：在 `RuntimeCompiler` 里按 prompt 内容或用户任务文本过滤/选择策略。这种做法不符合成熟 agent 架构，必须删除。

目标：

```text
single_agent_turn:
  可以判断是否启动任务、是否普通回答。

task_execution:
  只执行已确定的任务合同。
  不重新判断是否开启任务。
  不装配与任务无关的环境策略。
```

硬约束：

```text
禁止用关键词、自然语言片段、prompt 文案内容来控制 route、prompt ref、环境策略或工具策略。
route 只能来自 RuntimeAssembly / ActiveTurn / explicit contract 等结构化字段。
prompt ref 只能来自 profile/environment/task contract 中的结构化 refs。
```

### 3.4 prompt cache 仍依赖后置诊断，不够前置约束

涉及文件：

```text
backend/harness/runtime/prompt_segment_plan.py
backend/runtime/model_gateway/model_runtime.py
backend/runtime/prompt_accounting/*
backend/scripts/diagnose_deepseek_prompt_cache.py
```

当前已有 `segment_plan` 和账本，但还需要继续收紧：

- 所有主链模型调用必须有 segment plan。
- stable segment 必须拒绝 runtime id、observation、attempt、pending steer 等动态字段。
- utility 调用必须有明确 `cache_metric_scope`，不能混入 agent runtime KPI。
- prompt pack rejected refs 不能静默导致 provider global prefix 缺失。

### 3.5 当前控制权与动态上下文仍需继续收敛

涉及文档：

```text
docs/系统框架/042-单AgentActiveTurn权威与任务监控串台修复方案-20260601.md
```

动态路由和 prompt 装配不能分开修：

```text
ActiveTurnRegistry 决定当前 turn / task 权威；
RuntimeCompiler 只装配当前 active turn 允许模型看到的结构事实；
PromptSegmentPlan 决定哪些段可缓存；
PromptAccounting 只记录事实，不反向决定 prompt 语义。
```

## 4. 目标设计

### 4.1 权威链

目标链路：

```text
UserSubmission
-> ActiveTurnRegistry
-> StructuralTurnRoute
-> RuntimeAssembly
-> RuntimeCompiler
-> DynamicContextManager
-> RuntimeInvocationPacket
-> PromptSegmentPlan
-> ModelRuntime
-> PromptAccountingLedger
-> RuntimeObservation / TaskState
```

权责：

- `ActiveTurnRegistry`：决定当前 session 是否已有 active turn，绑定 task run。
- `StructuralTurnRoute`：决定 single turn / explicit task / blocked runtime，不扫描历史 task 当当前权威。
- `RuntimeAssembly`：给出 profile、environment、operation authorization、prompt refs。
- `RuntimeCompiler`：唯一负责 agent 主链 prompt 装配。
- `DynamicContextManager`：把 runtime facts 投影成模型可见动态状态。
- `PromptSegmentPlan`：强制 stable / volatile 边界。
- `ModelRuntime`：执行调用、记账、暴露 provider usage，不猜 prompt 结构。
- `PromptAccountingLedger`：记录真实请求和 cache 命中，不参与语义决策。

### 4.2 Prompt 分层

目标分层：

```text
provider_global:
  全局 agent 行为、工具使用总规则、输出总规则。

session:
  同会话稳定设置，如用户偏好、固定运行模式、固定权限轮廓。

task:
  当前任务合同、当前 task environment 的稳定描述、工具 catalog hash。

volatile:
  当前用户输入、最近历史、task_state、observations、runtime envelope、pending steers、tool receipts。

none:
  不应缓存或不参与前缀的临时内容。
```

硬规则：

- `provider_global` 不得包含 task id、session id、当前日期、文件路径、runtime assembly id。
- `session` 不得包含 task_run_id、observations、attempt、current_step、pending_user_steers。
- `task` 可以包含任务语义合同，但不得包含 executor 状态、observations、attempt、last_error。
- `volatile` 必须带 dynamic context metadata。

### 4.3 模型可见 task state

任务执行模型只能看到统一 `task_state`：

```json
{
  "task_state": {
    "runtime_status": "running",
    "current_step": {},
    "current_facts": [],
    "latest_tool_results": [],
    "active_failures": [],
    "historical_failures": [],
    "artifact_evidence": [],
    "pending_user_steers": [],
    "active_contract_revisions": [],
    "work_progress": {},
    "task_run_state": {},
    "runtime_boundary": {}
  }
}
```

禁止重新暴露：

```text
volatile_payload.execution_state
volatile_payload.observations
volatile_payload.work_history
volatile_payload.task_run_state
```

这些可以作为内部原始输入存在，但不能并列进入模型。

### 4.4 工具结果投影

工具结果规范化顺序：

```text
raw tool result / observation payload
-> parse result_envelope
-> parse structured_payload
-> parse JSON string result when applicable
-> normalize status/error/structured_error/artifact_refs
-> project preview/ref/content_replacements
-> merge into observation projection
-> merge into task_state
```

结构化错误模型可见字段：

```json
{
  "code": "...",
  "message": "...",
  "retryable": false,
  "provider_retryable": true,
  "agent_retry_policy": "do_not_auto_retry",
  "origin": "image_provider",
  "attempts": [
    {
      "model": "gpt-image-2",
      "attempt_index": 1,
      "http_status": 504,
      "code": "image_provider_transient_error",
      "retryable": true
    }
  ]
}
```

### 4.5 Prompt refs 装配

目标：

```text
RuntimeAssembly:
  agent_prompt_refs_by_invocation:
    single_agent_turn: [...]
    task_execution: [...]
    tool_observation_followup: [...]

single_agent_turn:
  只使用 single_agent_turn 对应 refs。

task_execution:
  只使用 task_execution 对应 refs。

tool_observation_followup:
  只使用 tool_observation_followup 对应 refs。

environment strategy:
  只能在 environment/task assembly 显式提供 strategy ref 时装入。
```

禁止：

- task execution 装入“普通 turn 中判断是否开启持续处理流程”。
- 图片/写作/通用任务装入 Python 开发策略。
- prompt ref rejected 后静默生成缺失全局前缀的 packet。
- 在 compiler 中通过 `if "某句话" in prompt` 删除内容。
- 在 compiler 中通过用户请求文本推断“开发任务”“写作任务”等 prompt 策略。

## 8. 设计结论

本项目下一步不应继续围绕“多写几句 prompt 防御”优化，而应按成熟 agent 的结构不变量收敛：

```text
当前控制权结构化。
上下文来源结构化。
稳定前缀和动态状态分层。
工具结果统一投影。
模型只看任务所需事实。
账本只记录事实，不反向决定语义。
```

这条路线与 Claude Code 的 cache-safe prefix、Codex 的 structured turn context 是一致的，也符合本项目当前 `RuntimeCompiler -> DynamicContextManager -> PromptSegmentPlan -> ModelRuntime` 的目标方向。

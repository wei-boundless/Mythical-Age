# 健康系统框架设计书

日期：2026-05-29

状态：框架重校准稿

## 1. 定位结论

健康系统不是测试系统、不是任务账本、不是执行器。
任务记录管理归健康系统管理，但它管理的是任务记录的健康治理、保留策略、异常标记、归档建议和受控维护入口，不等同于拥有 runtime 原始事实账本。

运行监控系统可以纳入健康系统的管理域，但必须分层：

```text
HealthSystem 管理监控健康汇总、风险解释、策略阈值和处理建议。
RuntimeMonitor 持有实时状态机、投影算法和运行快照事实。
```

也就是说，监控系统的健康结论和治理建议归健康系统汇总，实时运行监控页面、运行事实和状态投影实现仍属于 runtime/主监控页面。

健康系统的根本职责是：

```text
读取 Agent 平台运行事实，
生成健康画像、风险判断、成本压力、效率异常和治理建议。
```

因此本轮定义收束为：

```text
HealthSystem =
  Agent 运行治理观察层
  + 诊断决策投影层
  + 运行监控健康汇总层
  + 健康问题与报告登记层
```

健康系统不负责：

```text
执行 agent
运行测试
调度实验
修改 TaskRun 事实账本
无审计地清理 EventLog / PromptAccounting 记录
维护 runtime monitor 实时状态机
生成 token 真值
拥有恢复、重试、修复、验证的执行权
```

健康系统只消费事实源，不生产事实源；只给出治理判断，不直接执行治理动作。它可以定义监控治理口径和页面组织，但不能直接改写 monitor 的实时状态事实。

## 2. 现有代码审查结论

当前代码已经部分接近目标定位，但仍有边界漂移。

### 2.1 已经正确的方向

`backend/health_system/README.md` 已将健康系统定义为 Agent 运行治理系统，管理任务风险、系统风险、Token 消耗和运行效率。

`backend/health_system/governance.py` 中的 `HealthGovernanceBuilder` 已经形成核心治理聚合：

```text
TaskRun / EventLog
PromptAccountingLedger
RuntimeMonitor
  -> HealthGovernanceBuilder
  -> overview / tasks / risks / token_usage / efficiency
```

`backend/health_system/agent_config.py` 和 `backend/health_system/execution_planner.py` 已经把健康 agent 执行配置 fail-closed，避免在配置缺失时伪造可执行能力。

`docs/系统规划/229-运行监控系统重构计划书-20260528.md` 明确运行监控的唯一状态投影层是 `TaskRunMonitorProjector`，并指出 Health System 应消费 diagnostics bucket 作为风险输入。

`docs/系统规划/230-精准Token与Prompt缓存统一重构计划书-20260528.md` 明确健康系统只能读取 PromptAccounting 账本和诊断结果，不应重新发明 token 统计口径。

### 2.2 当前边界漂移

第一，健康系统已暴露任务记录清理接口：

```text
POST /api/health-system/task-records/prune
```

任务记录管理应归健康系统，但当前接口语义过粗：它直接删除 `TaskRun`、`EventLog` 和 prompt accounting 记录，缺少健康治理所需的审计、预检、影响范围说明、保留策略和不可删除保护。
因此问题不是“任务记录管理不属于健康系统”，而是现有 prune 需要重定为健康系统的受控任务记录维护能力。

第二，健康系统仍暴露健康 agent 执行和会话控制面：

```text
/health-system/commands
/health-system/conversation-sessions
/health-system/issues/{issue_id}/agent-runs
```

但当前健康 agent 配置实际为 `health_agent_config_not_rebuilt`。前端 `HealthAgentDock` 仍展示“分析当前问题”，会制造可用性错觉。

第三，系统风险覆盖不足。

`HealthGovernanceBuilder._system_risks()` 当前只消费 runtime monitor 的 `error/stale/waiting` 摘要，没有接入：

```text
固定端口 3000 / 8003
API base
SSE 状态
前端连接健康
sandbox 可用性
模型服务异常
工具执行失败聚合
```

第四，健康治理与 `TaskRun` 模型存在契约漂移。

当前 `governance.py` 读取 `task_run.runtime_lane`，但 `runtime.shared.models.TaskRun` 没有该字段。健康 token 测试已经失败，说明治理层不能直接猜测 TaskRun 字段，应通过稳定投影或兼容读取策略消费事实。

## 3. 成熟架构参照

健康系统属于可观测性与治理层，成熟参照应来自 observability 和 SRE 体系，而不是测试平台或任务执行器。

OpenTelemetry 将可观测信号分为 traces、metrics、logs、baggage，强调不同信号从不同角度描述同一系统活动，并通过统一属性进行关联。参考：[OpenTelemetry Signals](https://opentelemetry.io/docs/concepts/signals/)。

OpenTelemetry 还明确自身是遥测数据的生成、采集和导出框架，不是观测后端本身。这个边界适合本项目：健康系统可以聚合和解释信号，但不应替代事实源或执行后端。参考：[What is OpenTelemetry?](https://opentelemetry.io/docs/what-is-opentelemetry/)。

Google SRE 的监控四个黄金信号是 latency、traffic、errors、saturation。对应到本 Agent 项目，应转译为：

```text
latency      -> 任务耗时、工具等待、模型响应耗时
traffic      -> 任务数量、运行中任务、模型请求量、工具调用量
errors       -> failed/blocked/error 事件、工具错误、模型错误
saturation   -> token 压力、上下文窗口压力、队列/等待、sandbox/端口资源压力
```

参考：[Google SRE - Monitoring Distributed Systems](https://sre.google/sre-book/monitoring-distributed-systems/)。

本项目不需要照搬 OpenTelemetry 或 SRE 平台，但必须借用其核心不变量：

```text
事实源和治理判断分离。
监控信号和执行动作分离。
原始事件和聚合视图分离。
指标口径必须稳定、可追溯、可解释。
```

## 4. 权责链

健康系统覆盖运行监控的健康汇总，但健康系统的上游事实源固定为：

```text
RuntimeStateIndex
  owns TaskRun / AgentRun / worker records

RuntimeEventLog
  owns runtime events and trace logs

TaskRunMonitorProjector
  owns live monitor lifecycle / bucket / stale / action_required projection

PromptAccountingLedger
  owns provider usage / local prediction / cache accounting

RuntimeEnvironment / ConnectionHealth / PortGuard
  owns frontend/backend/api_base/SSE/environment health probes

ToolRuntime / ModelGateway / SandboxBackend
  owns tool/model/sandbox execution facts and errors
```

健康系统的权责链为：

```text
Task records
Runtime events
Runtime monitor projection
Prompt accounting ledger
Environment health
Tool/model/sandbox diagnostics
  -> HealthSignalReader
  -> HealthGovernanceBuilder
  -> HealthRiskClassifier
  -> HealthViewModels
  -> HealthIssue / HealthReport registry
```

健康系统不得反向写入上游事实源。

运行监控纳入健康系统后的责任分界：

```text
HealthSystem owns:
  监控健康摘要
  监控治理规则和阈值
  监控异常解释
  监控风险升级
  监控建议动作
  监控历史健康报告
  监控事实在健康总览和系统风险页的汇总呈现

RuntimeMonitor owns:
  实时运行监控页面
  TaskRunMonitorProjector
  lifecycle / bucket / stale / action_required
  global/session/task detail monitor views
  live monitor revision
  runtime snapshot consistency
```

健康系统可以管理监控域的“怎么评估、怎么汇总、怎么升级为健康问题”，但不能拥有“当前任务到底是什么状态”的事实裁决权，也不重复建设实时运行监控页面。

## 5. 核心概念模型

### 5.1 HealthSignal

HealthSignal 是健康系统消费的标准输入，不是新的事实账本。

```text
HealthSignal
  signal_id
  source_authority
  signal_type
  target_ref
  target_kind
  severity_hint
  observed_at
  payload
  evidence_refs
```

允许的 `signal_type`：

```text
task_lifecycle
task_stale
task_failure
tool_error
model_error
token_pressure
context_pressure
environment_error
connection_error
sandbox_error
monitor_diagnostic
```

### 5.2 HealthRiskEvent

HealthRiskEvent 是健康系统自己的治理判断。

```text
HealthRiskEvent
  risk_id
  scope
  severity
  target_ref
  title
  summary
  evidence_refs
  recommended_action
  created_at
  source_signal_refs
```

风险事件必须可追溯到上游 signal 或事实记录，不能只来自 UI 文案或临时推断。

### 5.3 HealthTaskProfile

HealthTaskProfile 是任务健康画像。

```text
HealthTaskProfile
  task_run_id
  session_id
  title
  lifecycle
  status
  duration_seconds
  monitor_bucket
  action_required
  stale
  error_count
  tool_call_count
  model_request_count
  token_total
  token_source
  risk_level
  latest_risk_event
  evidence_refs
```

`lifecycle/bucket/stale/action_required` 必须来自 `TaskRunMonitorProjector`，健康系统不重复实现监控状态机。

### 5.4 HealthCostProfile

HealthCostProfile 只读取 PromptAccounting。

```text
HealthCostProfile
  task_run_id
  session_id
  provider_usage_tokens
  local_prediction_tokens
  trace_estimate_tokens
  cached_tokens
  cache_savings_tokens
  token_source
  billing_truth_available
  pressure_level
```

主指标优先级：

```text
provider_usage > local_prediction > trace_estimate
```

`trace_estimate` 只能作为旧任务迁移回退，不允许和精确账本混称为真实消耗。

### 5.5 HealthSystemRisk

系统风险不是任务风险的简单合计。

```text
HealthSystemRisk
  risk_id
  subsystem
  severity
  status
  summary
  evidence_refs
  recommended_owner
```

必须覆盖：

```text
runtime_monitor
frontend_port
backend_port
api_base
sse_stream
model_gateway
tool_runtime
sandbox_backend
prompt_accounting
```

### 5.6 HealthMonitorGovernance

HealthMonitorGovernance 是健康系统对运行监控健康结论的汇总模型。

```text
HealthMonitorGovernance
  monitor_revision
  global_monitor_ref
  diagnostics_bucket_count
  stale_task_count
  action_required_count
  unknown_status_count
  stale_threshold_seconds
  escalation_rules
  recommended_actions
```

它不重新计算 `lifecycle`，只读取 RuntimeMonitor 输出并决定：

```text
哪些 monitor diagnostics 应升级为健康风险。
哪些 stale/action_required 需要置顶。
哪些 monitor 异常应生成 HealthIssue。
哪些监控口径异常说明 runtime monitor 自身需要修复。
```

## 6. API 框架

健康治理 API 应保留：

```text
GET  /api/health-system/overview
GET  /api/health-system/tasks
GET  /api/health-system/tasks/{task_run_id}
GET  /api/health-system/risks
GET  /api/health-system/system-risks
GET  /api/health-system/token-usage
GET  /api/health-system/efficiency
GET  /api/health-system/recommendations
GET  /api/health-system/monitor-governance
GET  /api/health-system/issues
POST /api/health-system/issues
GET  /api/health-system/reports
GET  /api/health-system/reports/{report_id}
```

任务记录管理 API 归健康系统，但需要重定语义：

```text
GET  /api/health-system/task-records/maintenance
POST /api/health-system/task-records/prune
```

治理要求：

```text
必须预检影响范围。
必须返回 affected_task_runs / affected_events / affected_prompt_accounting。
必须写入 HealthMaintenanceReceipt。
默认只能归档、标记、压缩或清理健康系统可安全维护的派生/过期记录。
如涉及 runtime 原始事实账本删除，必须有明确保留策略、审计记录和保护条件。
```

健康 agent 执行控制面必须按本轮统一裁决处理：

```text
POST /api/health-system/commands analyze_trace
POST /api/health-system/conversation-sessions
POST /api/health-system/issues/{issue_id}/agent-runs
GET  /api/health-system/agent-runs/*
```

目标规则：

```text
历史 HealthAgentRun / conversation / result 数据只读可查。
前端不得展示健康 agent 执行主入口。
新的 POST 执行入口不得创建真实 runtime run。
若短期保留 POST endpoint，只能返回明确 disabled/blocked 语义，并标记 legacy。
恢复健康 agent 必须另走 TaskSystem + Runtime 标准任务链路，不允许在 HealthSystem 内恢复执行器。
```

## 7. 前端框架

健康系统页面应围绕五个一级视图：

```text
总览
任务健康
任务记录管理
系统风险
Token 与成本
```

运行监控不再作为健康系统内的独立页面。主页面已经有运行监控页面，负责实时状态查看、session/task 监控和运行态细节。健康系统只在总览和系统风险中消费 `HealthMonitorGovernance`，汇总监控事实、风险升级和处理建议，不在前端重新计算任务生命周期，也不重复建设监控界面。

前端只展示后端 view model，不重新判断：

```text
任务生命周期
运行 bucket
stale
risk level
token source truth
system risk severity
```

`HealthAgentDock` 不应作为默认健康系统能力出现。若保留，必须满足：

```text
后端 preview status == ready
健康 agent runtime config 已通过 admission
UI 明确展示执行边界和权限
所有分析任务走 TaskSystem/Runtime 标准链路
```

在此之前，应隐藏或替换为只读说明：

```text
健康分析 Agent 未启用；当前页面只提供治理视图和问题登记。
```

## 8. 模块框架

目标目录保持 `backend/health_system`，但内部职责应收束：

```text
backend/health_system/
  governance.py
    组装 overview/tasks/risks/token/efficiency view model

  signals.py
    从 runtime monitor、event log、token ledger、environment health 读取 HealthSignal

  risk_classifier.py
    将 HealthSignal 转成 HealthRiskEvent

  cost_profiles.py
    只读取 PromptAccountingLedger，生成 HealthCostProfile

  task_profiles.py
    只消费 TaskRunMonitorProjector + RuntimeStateIndex，生成 HealthTaskProfile

  system_risks.py
    汇总 port/api/SSE/sandbox/model/tool/runtime monitor 风险

  monitor_governance.py
    消费 RuntimeMonitor view model，生成监控治理摘要、升级规则和健康建议

  registry.py
    管 HealthIssue / HealthReport / Receipt 的健康治理登记

  store.py
    只存健康系统自己的 issue/report/receipt/conversation legacy 数据
```

应从健康系统移出的职责：

```text
test system maintenance
experiment runner
verification gate
agent execution planner
runtime admission for health agent execution
health-local runtime lane registry
health-local evidence packet / scorer / extractor execution链路
```

若未来恢复健康 agent，应作为标准任务接入：

```text
TaskSystem 定义 health analysis task
Runtime 执行 task
HealthSystem 读取执行结果并登记报告
```

而不是让 HealthSystem 自己拥有执行器。

## 9. 固定数据流

### 9.1 总览

```text
RuntimeStateIndex.list_task_runs()
TaskRunMonitorProjector.build_global_monitor()
PromptAccountingLedger.summarize_task/session
RuntimeConnectionHealth.check()
Tool/Model/Sandbox diagnostics
  -> HealthSignalReader
  -> HealthMonitorGovernance
  -> HealthGovernanceBuilder.build_overview()
  -> HealthSystemView
```

### 9.1.1 运行监控治理

```text
RuntimeMonitor global/session/detail views
  -> HealthMonitorGovernance
  -> monitor health summary
  -> monitor risk escalation
  -> health overview / system risk summary
```

HealthMonitorGovernance 只消费 monitor view，不生成 monitor view。

### 9.2 任务详情

```text
TaskRun
TaskRunMonitorItem
EventLog recent events
PromptAccounting task detail
Tool/model error signals
  -> HealthTaskProfile
  -> task risks
  -> task detail view
```

### 9.3 系统风险

```text
RuntimeConnectionHealth
PortGuard
RuntimeMonitor diagnostics bucket
ModelGateway error records
ToolRuntime error records
SandboxBackend health
  -> HealthSystemRisk list
```

系统风险事实源必须至少覆盖下列代码域：

```text
backend/runtime/model_gateway/*
backend/runtime/tool_runtime/*
backend/runtime/tool_runtime/sandbox_backend.py
backend/runtime/tool_runtime/docker_sandbox_backend.py
backend/runtime/shared/event_log.py
backend/harness/runtime/monitor_projection.py
backend/runtime/environment/*
```

若某一事实源暂时没有稳定 health probe，必须以 `unknown/not_instrumented` 进入系统风险视图，不能默认为 healthy。

### 9.4 问题登记

```text
User/manual/system creates HealthIssue
HealthSystem stores issue/report/receipt
No task execution happens inside HealthSystem
```

若需要分析：

```text
HealthIssue
  -> TaskSystem health-analysis task
  -> Runtime execution
  -> Result artifact/report
  -> HealthSystem registers report
```

## 10. 迁移规则

第一阶段：修复治理主链路。

```text
修复 governance.py 与 TaskRun 模型契约漂移。
任务画像改用 getattr 或 TaskRunMonitorItem。
恢复 health_governance_token_usage_test。
```

第二阶段：系统风险补源。

```text
接入 RuntimeConnectionHealth。
修正 PortGuard：端口未监听时 ok 必须为 false。
增加 API base 与 SSE 状态输入。
将 environment_error 转为 HealthSystemRisk。
识别固定端口监听进程是否属于本项目。
非本项目进程占用固定端口时输出 wrong_process_on_fixed_port，不自动关闭。
```

第三阶段：监控治理纳入健康系统。

```text
新增 HealthMonitorGovernance。
健康系统总览和系统风险消费 monitor_governance summary。
monitor diagnostics bucket 升级为健康风险。
前端健康系统不新增运行监控页；实时监控界面仍由主监控页面承担。
```

第四阶段：重定任务记录维护能力。

```text
保留任务记录管理归健康系统。
将 /health-system/task-records/prune 重定为受控维护接口。
前端 HealthSystemView 保留任务记录管理入口，但必须展示预检、影响范围、保护条件和维护回执。
禁止无审计、无预检地直接删除 runtime 原始事实账本。
```

第五阶段：降级健康 agent 控制面。

```text
隐藏 HealthAgentDock 的执行入口。
保留历史 report/result 只读查询。
commands/conversation/agent-runs 的新 POST 执行入口返回 disabled/blocked，不创建 runtime run。
```

第六阶段：重建健康治理模型。

```text
新增 HealthSignal / HealthRiskEvent / HealthTaskProfile / HealthCostProfile。
让 overview/tasks/risks/token/efficiency 全部基于稳定 profiles。
```

## 12. 禁止模式

健康系统后续实现禁止出现：

```text
为了兼容旧 UI 保留健康测试系统入口。
以 health command 名义绕过 TaskSystem/Runtime 执行任务。
在 health governance 或前端里重新实现 runtime monitor 状态机。
把 token trace estimate 当作真实账单值。
健康系统直接删除 runtime 事实账本。
前端自行判断风险等级并覆盖后端结论。
健康 agent 未启用时仍展示“分析当前问题”主按钮。
健康 agent 未启用时 POST endpoint 偷偷创建 runtime run。
端口被无关进程占用时误判为 healthy。
用空 mock 或硬编码让健康测试通过。
```

## 13. 最终定义

健康系统是 Agent 平台的治理观察、运行监控健康汇总与诊断决策层。

它读取任务记录、实时监控、token 账本、运行事件和环境健康信号，汇总运行监控健康结论，生成健康画像、风险判断、成本压力、效率异常和处理建议。

它不生产运行事实，不执行任务，不运行测试，不无审计地清理账本，不拥有恢复或验证动作。

所有执行、恢复、验证动作必须交给对应的 runtime、task 或 verification 系统完成。
任务记录维护由健康系统统一治理，但必须以受控维护、审计回执和事实账本保护为边界。

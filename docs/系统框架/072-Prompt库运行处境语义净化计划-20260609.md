# Prompt 库运行处境语义净化计划

日期：2026-06-09

## 1. 目标

对全 prompt 库做一次语义净化：所有写给 agent 的提示词，都必须站在 agent 实际运行时能感知到的处境来表达，而不是站在开发者解释架构、节点、prompt 管理或装配机制的角度。

核心原则：

- Prompt 是 agent 当场执行的处境说明，不是开发者文档。
- Agent 只看到已经装配好的提示词、上下文、工具和观察；它不需要知道某段 prompt 在系统里属于哪个内部模块。
- 可以描述“你会看到什么、你应该怎么判断、你能用什么、你不能假设什么、失败后怎么处理、怎样收口”。
- 不应写成“这个 prompt 用于什么”“该节点用于什么”“本段只定义什么”“当前环境已被系统选中后生效”。
- Runtime 协议层可以保留 `JSON`、`action_type`、`schema` 等必要格式词，因为那是 agent 必须执行的输出协议，不属于开发者视角噪声。

## 2. 分层判断

### 2.1 必须改成运行处境表达的层

- `backend/prompt_library/agent_prompts.py`
  - 主 agent work role。
  - 应描述 agent 如何理解当前请求、推进任务、处理观察和收口。
  - 避免“关键词分类器”“环境切换器”“prompt/payload 提供”等开发视角。

- `backend/prompt_library/general_lifecycle_prompts.py`
  - 通用生命周期提示词。
  - 应描述 agent 在某一轮会话、工具派发、观察恢复、记忆读写、收口前的具体处境。
  - 避免“系统装配”“环境说明决定什么”等架构解释。

- `backend/task_system/environments/prompt_resources.py`
  - 环境 orientation 和资源 orientation。
  - 应描述环境里有什么、资源如何使用、边界是什么、完成证据是什么。
  - 避免“任务环境是外层容器”“runtime 权限模式”“该环境用于”等开发说明。

- `backend/prompt_library/rules.py` 中的 environment/coding/writing/general 规则。
  - 应写成 agent 当场的工作纪律。
  - 避免“规则只在某环境适用”这类开发口吻，改成“处理此类工作时，不要套用另一类工作方式”。

### 2.2 可以保留协议词，但要净化周边表达的层

- `backend/prompt_library/packs.py`
- `backend/prompt_library/rules.py` 中 runtime protocol rules。
- `backend/prompt_library/system_prompts.py`

这些层会包含 `action_type`、`JSON`、`schema`、工具协议和权限边界。它们是 agent 必须遵守的运行协议，不能因为出现技术词就删除。需要改的是周边开发解释：

- 不写“本段只约束某某协议”。
- 少写“由某某 prompt/payload 提供”。
- 改成“你本轮会收到……你需要……你只能……”。

### 2.3 低风险保留但可局部打磨的层

- `backend/prompt_library/tool_prompts.py`
  - 大部分已经是工具使用处境说明。
  - 可将个别 `runtime 曾经省略` 改为“系统曾经省略”。

- `backend/prompt_library/worker_prompts.py`
  - 多数 worker prompt 已经是角色化任务说明。
  - 需要检查是否有“子 Agent 模块/能力说明”式表达。

- `backend/prompt_library/personality_prompts.py`
  - 人格层应只描述身份、称呼、语气、协作风格。
  - 避免 `personality prompt`、`action schema` 这类内部术语。

- 图节点角色 prompt。
  - 大多已经是“你是一名……审核员/管理员”的运行角色说明。
  - runtime graph protocol 中的“任务图/节点合同/输出合同”属于 agent 运行处境，可以保留；“系统装配”应改成“你会看到”。

## 3. 初步发现

扫描模型可见 PromptResource 后，优先处理以下问题：

- `agent.main_interactive_agent.*.work_role`
  - 存在“你不是关键词分类器，也不是环境切换器”等开发对照式表达。
  - 应改成正向处境表达：你需要根据用户最新请求、当前上下文和可见工具选择最小充分行动。

- `environment.general.lifecycle.environment_capability_alignment`
  - 存在“本轮系统装配”“环境说明决定系统能提供什么”等架构说明。
  - 应改成：你会看到当前工作区、文件边界、工具可见性和权限状态；先检查它们是否支持用户目标。

- `environment.resource.base_workspace.orientation`
  - 存在 `runtime 权限模式` 表达。
  - 应改成“本轮系统给出的权限和可见工具”。

- `environment.resource.sandbox_overlay.orientation`
  - 存在 `runtime 明确允许` 表达。
  - 应改成“本轮系统明确允许”。

- `environment.rule.development_sandbox`
  - 存在“该环境用于……”开发说明。
  - 应改成“你正在使用通用开发沙盒……”。

- `personality.default.mythical_age`
  - 存在 `personality prompt` 和 `action schema` 表达。
  - 应改成“这个人格只影响称呼、语气和表达风格；不改变可用动作格式、工具权限或任务边界”。

- `runtime.single_agent_turn`、`runtime.task_execution`、`runtime.observation_followup`
  - 协议词需要保留，但“身份风格由 personality prompt 提供”“判断由 runtime payload 提供”等表达应净化。

## 4. 实施顺序

### Phase 1：Agent 与人格层

文件：

- `backend/prompt_library/agent_prompts.py`
- `backend/prompt_library/personality_prompts.py`
- 相关 registry/profile tests

动作：

- 将主 agent work role 改成正向运行处境表达。
- 将人格层内部术语改为 agent 可感知的边界表达。
- 保持 agent/personality prompt ID 不变。

### Phase 2：Environment 与 lifecycle 层

文件：

- `backend/prompt_library/general_lifecycle_prompts.py`
- `backend/task_system/environments/prompt_resources.py`
- `backend/prompt_library/rules.py`
- `backend/tests/task_environment_registry_regression.py`
- `backend/tests/prompt_rule_system_regression.py`

动作：

- 净化 general lifecycle 的环境对齐、工具派发、观察恢复、收口等提示词。
- 净化 base workspace、sandbox、development、general、writing 等环境提示词。
- 保留 coding workspace 已完成的运行处境表达。

### Phase 3：Runtime/graph 协议层

文件：

- `backend/prompt_library/packs.py`
- `backend/prompt_library/rules.py`
- `backend/tests/prompt_library_registry_regression.py`
- `backend/tests/harness_model_action_protocol_regression.py`

动作：

- 保留 JSON、action_type、schema 等必要协议词。
- 将“由某某 prompt/payload 提供”“本段只约束”改成“你本轮会收到/你必须输出/你只能使用”。
- 确保协议格式测试不被削弱。

### Phase 4：Worker、tool、graph role 层

文件：

- `backend/prompt_library/worker_prompts.py`
- `backend/prompt_library/tool_prompts.py`
- task graph role prompt 相关生成路径和测试

动作：

- worker prompt 保持角色化任务说明，只清理内部模块/能力说明口吻。
- tool guidance 只做轻微净化，不删除工具契约。
- 图节点角色 prompt 保持角色职责、输入依据、裁决标准和输出要求。

### Phase 5：全库治理测试

新增或扩展测试：

- 扫描所有模型可见 prompt resource。
- 对非 runtime protocol 层拒绝明显开发口吻：
  - `这是 runtime 节点`
  - `根据任务图执行`
  - `该节点用于`
  - `本段只定义`
  - `环境已被系统选中`
  - `不授予工具`
  - `runtime packet`
- runtime protocol 层只允许在格式协议上下文中出现 `schema/action_type/JSON`。
- 保持 prompt ID、装配顺序和环境切换体系不变。

# 主会话图任务工作台与任务系统配置页重构计划

日期：2026-05-31

## 1. 问题定义

当前任务系统页面把两类不同层级的东西混在一起：

- 配置资产：任务环境、具体任务、契约、Agent 装配配置。
- 执行工作台：图编辑器、图任务启动、图运行监控、节点输出、产物查看。

这导致用户在“任务系统”里既要管理配置，又要运行任务图。这个层级不符合既有设计原则：任务系统是合同和配置系统，不应该成为日常运行台；GraphHarness / GraphLoop 才负责运行生命周期；主会话才是用户发起执行和观察执行的主要工作面。

正确目标是：

- `任务系统` 只负责配置资产管理。
- `主会话 / 图任务层` 承担图任务选择、图编辑、编译校验、运行、监控、产物查看。
- 图编辑器编辑出的配置必须直接编译为 runtime 可消费的标准图配置，不能再出现一套编辑配置、一套运行配置。

来源：

- `docs/系统框架/002-任务系统框架设计书-20260527.md`
- `docs/系统框架/004-图任务GraphHarness重建计划书-20260528.md`

必须遵守的原则：

- 任务系统负责预定义任务、任务环境和合同签发，不负责推进 runtime / loop 生命周期。
- `TaskGraphDefinition` 是保存态任务图的主事实源。
- `GraphHarnessConfig` 是运行期静态执行合同，GraphRuntime / GraphLoop 只消费它，不回头解释旧字段。
- 图编辑器不是另一套语言，它必须编辑可编译为标准图合同的结构。
- 图任务运行必须在用户侧可监视，节点、边、批次、产物、人工门控都要能进入同一个图任务运行视图。

## 3. 目标架构

### 3.1 页面职责

#### 任务系统配置页

只负责配置资产：

- 任务环境：环境边界、存储策略、权限边界、环境 prompt。
- 具体任务：某个环境下可执行的任务定义。
- 契约：具体任务所需的输入契约、输出契约、节点契约、边通信契约。
- Agent 配置：可被节点引用的 agent role、模型、工具包、skills、运行模式。

任务系统配置页不负责：

- 图拓扑实时编辑。
- 图任务启动。
- 图运行监控。
- 节点输出查看。
- 运行产物管理。

#### 主会话图任务工作台

位置：`主会话 -> 图任务层`，也就是现有 `CenterWorkspaceView` 的图任务工作面。

负责：

- 选择任务环境。
- 选择具体任务。
- 加载该任务绑定的任务图。
- 实时编辑图拓扑、节点契约、边契约、记忆读写协议、agent 装配。
- 编译并校验标准 `GraphHarnessConfig`。
- 直接启动图任务运行。
- 显示运行状态、节点输出、错误、产物、断点恢复、人工门控。

#### 右侧全局监控栏

位置：`TaskMonitorDock`。

职责保持轻量：

- 显示全局运行列表。
- 显示运行中、完成、失败、诊断计数。
- 点击某个运行后把主会话切到对应图任务工作台上下文。

它不承载完整图编辑器。完整图编辑器在主会话中间工作面，否则右侧窄栏会破坏可用性。

### 3.2 数据主线

```text
任务系统配置页
  -> 保存 TaskEnvironment / SpecificTask / ContractSpec / AgentProfile
  -> 主会话图任务层读取配置资产
  -> 图编辑器编辑 TaskGraphDefinition / contract_bindings
  -> 编译 GraphHarnessConfig
  -> GraphRuntime / GraphLoop 运行
  -> 运行监控与产物回到主会话图任务层展示
```

## 4. 重构阶段

### Phase 1：锁定任务系统配置页边界

目标：

- 任务系统页从“大杂烩管理台”收缩为配置资产管理页。
- 导航中移除不属于配置资产的运行入口和图编辑入口。

实施点：

- `TaskSystemView.tsx`
  - 保留任务环境管理、具体任务管理、契约管理、Agent 配置管理。
  - 去掉任务系统主页里对图编辑器和运行管理的主入口展示。
  - `graphs` 不再是任务系统管理页的运行/编辑主页面，只作为配置资产引用关系的一部分。
- `TaskSystemShell.tsx`
  - 左侧对象导航改成配置资产导航。
  - 不再把运行阶段、运行管理、图编辑器作为同级管理页面。

完成标准：

- 任务系统页面只看到配置资产管理入口。
- 页面文案不再暗示任务系统负责运行生命周期。

待确认：

- 你说“导航页这两个去掉”，我当前理解为去掉任务系统导航中不属于配置资产管理的两个入口：图编辑/运行类入口。如果你指的是具体两个 UI 项，需要在实施前按页面标注确认一次。

### Phase 2：抽出图工作台状态与组件

目标：

- 让 `TaskGraphWorkbench` 不再依赖 `TaskSystemView.tsx` 作为唯一宿主。
- 把图编辑状态、保存、标准视图刷新、运行启动能力抽成可被主会话图任务层复用的控制器。

建议新增/调整：

- `frontend/src/components/workspace/views/task-system/useTaskGraphWorkbenchController.ts`
- `frontend/src/components/workspace/views/task-system/TaskGraphWorkbenchHost.tsx`
- `frontend/src/components/workspace/views/task-system/taskGraphWorkbenchContext.ts`

迁移内容：

- `selectedTaskGraphId`
- `editorEnvironmentId`
- `editorTaskGraphId`
- `taskGraphDraftV2`
- `taskGraphStandardViewState`
- `topologyDirty`
- 保存图任务
- 刷新标准视图
- 发布/运行入口

完成标准：

- `TaskGraphWorkbench` 可以被 `TaskSystemView` 和 `CenterWorkspaceView` 之外的宿主挂载。
- 图工作台只依赖明确 props/controller，不隐式读取任务系统页面状态。

### Phase 3：主会话图任务层升级为图任务工作台

目标：

- `CenterWorkspaceView` 的 `图任务层` 从“选择任务 + 监控”升级为“编辑 + 运行 + 监控”的一体化工作台。

实施点：

- `CenterWorkspaceView.tsx`
  - 引入抽出的 `TaskGraphWorkbenchHost`。
  - 顶部保留任务环境和具体任务选择。
  - 中间显示图编辑器主画布。
  - 右侧或下方显示运行监控、节点输出、产物和人工门控。
  - 底部保留图任务目标输入和运行按钮。

建议交互结构：

```text
图任务层
  顶部：任务环境 / 具体任务 / 图草稿 / 标准配置状态 / 保存状态
  中间：图编辑器画布 + 节点/边 Inspector
  右侧：运行监控 / 节点输出 / 产物 / 人工门控
  底部：图任务目标输入 + 启动运行
```

完成标准：

- 用户在主会话图任务层可以完成：选任务、改图、保存、编译、运行、看输出。
- 运行时使用当前图编辑器编译出的标准配置，不走任务系统页面跳转。

### Phase 4：运行与监控接线

目标：

- 主会话图任务工作台直接启动图任务运行，并绑定 monitor。
- 全局监控栏只做运行索引，不做编辑器。

实施点：

- `CenterWorkspaceView.tsx`
  - 运行前确认当前 `GraphHarnessConfig` 是最新编译结果。
  - 如果图有未保存/未同步变更，运行按钮必须进入明确状态：保存并运行、仅预检、取消。
- `TaskMonitorDock.tsx`
  - 点击图任务运行时，切到主会话图任务层并定位对应图运行。
- store：
  - 增加或复用 `taskGraphMonitorBinding`。
  - 明确绑定字段：`graph_id`、`graph_run_id`、`graph_harness_config_id`、`task_environment_id`、`specific_task_id`。

完成标准：

- 从主会话启动图任务后，监控自动绑定当前运行。
- 刷新页面后可以恢复当前绑定的图运行上下文。

### Phase 5：清理旧入口与旧链路

目标：

- 不保留两套图编辑入口。
- 不让任务系统页面继续承担运行语义。

清理内容：

- 删除任务系统管理页中的图编辑器主页面入口。
- 删除任务系统管理页中的运行管理入口，或降级为只读审计页。
- 删除跳转到编排页才能配置 agent 的旧式入口；Agent 配置应是任务系统配置资产的一部分。
- 清理与旧入口绑定的 CSS、状态字段、导航项和测试。

完成标准：

- 搜索 UI 文案时，不再出现“任务系统负责运行图任务”的暗示。
- 图任务编辑入口只有主会话图任务工作台。

## 7. 风险与禁止事项

禁止：

- 把图编辑器复制一份到主会话，留下任务系统里的旧编辑器继续可用。
- 让任务系统页面继续负责运行生命周期。
- 在主会话运行时临时拼另一套图配置。
- 用旧字段兜底绕过 `GraphHarnessConfig`。
- 把右侧全局监控栏做成完整图编辑器。

风险：

- `TaskSystemView.tsx` 现在状态过重，抽离时容易遗漏保存、预检、标准视图刷新逻辑。
- 图编辑器组件可能默认假设自己在任务系统管理页里，需要逐步解耦 props。
- 主会话图任务层空间有限，需要把编辑器、Inspector、监控输出做成清晰分区，不能继续堆卡片。

## 8. 切换策略

建议切换方式：

1. 先抽组件，不改变用户入口。
2. 主会话图任务层接入新工作台，但先隐藏旧任务系统入口。
3. 确认主会话可编辑、可运行、可监控后，删除任务系统旧图编辑入口。
4. 最后清理任务系统旧运行页和冗余样式。

回滚策略：

- 如果主会话图任务工作台运行不稳定，可以临时恢复旧任务系统图编辑入口。
- 但回滚只能作为短期诊断手段，不能长期保留两套入口。

## 9. 需要确认的点

1. “导航页这两个去掉”具体指哪两个 UI 入口。计划默认理解为去掉任务系统内的图编辑/运行类入口。
2. 主会话图任务层是否默认打开“运行监控”，还是默认打开“图编辑器”。建议默认打开图编辑器，运行后自动切到监控输出。
3. 任务系统配置页里的 Agent 配置，是复用现有编排资源页面，还是新建一个只面向任务节点的 Agent 配置页。建议新建任务节点 Agent 配置页，底层可复用 registry。

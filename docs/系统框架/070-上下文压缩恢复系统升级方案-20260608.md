# 上下文压缩恢复系统升级方案

日期：2026-06-08

## 目标

把当前“压缩摘要字符串”升级为一等公民的上下文恢复系统：

```text
对话、工具、lifecycle 同步记录事实
-> 记忆员异步维护 session memory
-> 压缩协调器判断是否需要压缩
-> 压缩员生成 ContextRecoveryPackage
-> runtime 替换旧历史并把 context_recovery_package 注入下一轮模型
-> UI 展示压力、覆盖范围、新鲜度和压缩来源
```

压缩不是 UI 摘要，也不是普通聊天总结。正式压缩产物必须服务下一轮模型恢复工作。

## 2026-06-08 实施状态

- 阶段 1 已建立 `context_recovery_package` 主契约：session memory 可写入 `context_recovery_package.json`，API/UI 可展示 freshness、coverage 和 source。
- 阶段 2 已接入 pre-turn 自动压缩：`HarnessRuntimeFacade.astream()` 会在模型 turn 前检查压力，压缩后重新加载 session record。
- 阶段 3 已接入 mid-turn 自动压缩：工具观察返回后、follow-up model call 前，`single_agent_turn` 会按 `ContextUsageMeter.auto_replacement_allowed` 触发真实 session compact。
- mid-turn 压缩完成后会重新编译 observation follow-up packet，把 `context_recovery_package` 注入模型，并保留当前工具观察。
- 当前用户请求保持在 volatile current request 槽位；压缩后刷新 active history 时会剔除当前 turn 的 user message，避免同一请求在 follow-up 中重复出现。
- `provider_protocol_compaction_created_at` 已作为 provider tool/reasoning 协议重放边界，旧协议轨迹不再越过压缩边界注入。
- 前端会话页上下文压力展示已使用 `compaction_pressure_tokens / replacement_threshold_tokens`，不是总上下文窗口比例。
- 前端会话输入区已提供“流式”显示开关；开启后请求会携带 `stream_policy`，single-agent 直接文本输出会发送 `content_delta`，JSON action 协议输出不会作为流式文本展示。

## 当前结构

- `backend/harness/entrypoint/runtime_facade.py`
  - `RuntimeFacade.astream()` 在模型 turn 前调用 `auto_compact_session_if_needed()`。
  - assistant commit 后调用 `memory_facade.enqueue_memory_maintenance_after_commit()`。
- `backend/runtime/context_management/session_compaction.py`
  - 构造 context usage snapshot。
  - 调用 `ContextCompactor.apply_strategy()`。
  - 压缩成功后通过 `session_manager.replace_runtime_context()` 写回消息与 `compressed_context`。
- `backend/runtime/prompt_accounting/context_usage_meter.py`
  - 已计算 `compaction_pressure_tokens / replacement_threshold_tokens`。
  - 这是 UI 展示上下文压缩剩余量的正确比例。
- `backend/memory_system/maintenance.py`
  - `MemoryMaintenanceAgent` 维护 session memory。
  - `MemoryCommitter` 写入 session memory 和 `compaction_state.json`。
- `backend/memory_system/storage/session_memory.py`
  - 存储 `summary.md`、`views/compaction_view.md`、`compaction_state.json`。
  - `validate_compaction_state()` 校验 message prefix coverage。
- `backend/context_system/compaction/compactor.py`
  - full compact 优先使用校验通过的 session memory。
  - session memory 不可用时调用 semantic compactor。
- `backend/harness/runtime/compiler.py`
  - 模型可见上下文使用 `context_recovery_package`，`compressed_context` 只保留为 session record 的 markdown 存储材料。
- `frontend/src/components/chat/ChatPanel.tsx`
  - 已按 `compaction_pressure_ratio` 展示上下文压力。
- `frontend/src/components/workspace/views/MemoryView.tsx`
  - 已展示 token、session memory 和压缩状态。

## 目标权责

| 层 | 责任 | 禁止事项 |
| --- | --- | --- |
| lifecycle recorder | 同步记录事实事件、offset、refs | 不总结、不裁决压缩 |
| 摘要员 memory agent | 异步维护 session memory | 不替换历史、不决定注入 |
| 压缩协调器 | 判断 pre-turn/mid-turn/manual 是否压缩 | 不编造摘要 |
| 压缩员 compactor agent | 生成正式 context recovery package | 不执行用户任务、不写文件/记忆 |
| runtime injection | 替换历史、注入 context_recovery_package | 不重新解释用户意图 |
| UI/API | 展示压力、覆盖、新鲜度、手动操作 | 不把 UI 摘要当上下文事实 |

## 阶段 1：第一闭环

目标：建立稳定 context recovery package 契约，并让 pre-turn 压缩和 UI 能使用它。

1. 新增 `backend/runtime/context_management/recovery_package.py`。
2. 新增 session memory 到 `ContextRecoveryPackage` 的确定性映射。
3. 在 `SessionMemoryManager` 中增加 `context_recovery_package.json` 读写。
4. 修正 semantic compactor prompt 和 output contract，统一以 `context_recovery_package` 为主。
5. `compressed_context` 只作为 context recovery markdown 渲染结果，不再作为事实源。
6. `/tokens/session/{session_id}` 返回 context recovery freshness、coverage、source。
7. 前端上下文状态展示 recovery package 是否 fresh、来源、覆盖消息数。

## 阶段 2：pre-turn 升级

目标：模型调用前确保上下文恢复包可用。

```text
estimate pressure including pending user/editor/runtime context
validate session memory and context recovery freshness
if threshold reached:
    generate or refresh context recovery package
    compact history
    reload session record
```

## 阶段 3：mid-turn 压缩

目标：工具观察回来后、follow-up model call 之前补压缩检查。

```text
if token_limit_reached or pressure >= replacement_threshold:
    run mid_turn context recovery compaction
    preserve current user message and recent tool trajectory
    rebuild followup model messages
```

## 阶段 4：lifecycle source

目标：让生命周期事件成为 context recovery package 的材料索引。

至少记录并可压缩：

```text
turn_received
context_pressure_checked
pre_turn_compaction_checked
context_assembled
model_call_started
model_call_completed
tool_observation_recorded
mid_turn_compaction_checked
final_answer_committed
memory_maintenance_finished
```

## 阶段 5：UI/API

`/tokens/session/{session_id}` 应返回：

```text
context_recovery_package_present
context_recovery_package_fresh
context_recovery_package_source
covered_message_count
covered_event_offset_end
last_compaction_reason
manual_compact_available
```

前端展示：

- 上下文压力：`compaction_pressure_tokens / replacement_threshold_tokens`
- 距自动压缩剩余 token
- context recovery package 是否存在
- context recovery package 是否新鲜
- context recovery package 来源：memory / semantic_compactor / manual
- 覆盖消息数和 stale reason

## 需要替换的旧逻辑

- `summary_content` 作为 semantic compactor 主契约的旧 prompt。
- 旧的“压缩摘要”模型可见主字段命名。已迁移为 `context_recovery_package`。
- 只记录 message coverage、不记录 event offset 的压缩状态。
- 只做 pre-turn 压缩、mid-turn 无检查的路径。
- UI 中把普通上下文占比误当作压缩阈值占比的展示。

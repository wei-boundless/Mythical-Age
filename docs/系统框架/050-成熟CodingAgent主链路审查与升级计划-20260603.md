# 成熟 Coding Agent 主链路审查与升级计划

日期：2026-06-03

状态：首轮收尾完成。本文档不直接改图调度、不改固定入口图运行语义；050 首轮已完成多条主链路权威收敛切片，后端 ArtifactAuthority 主链 cutover、CanonicalOutputBoundary final/presentation/memory commit 接入、前端 canonical metadata 和 ArtifactAuthority refs 展示接入均已完成。剩余内容改列为 050 后续成熟化切片；`/api/sessions` 慢响应作为独立运行风险记录，不算本轮主链未完成项。

## 0. 结论

当前系统已经支持 vibe coding 的基础形态：已有 `env.coding.vibe_workspace`、通用文件管理 profile、首批 `FileStateAuthority`、单轮可见工具收窄和 runtime monitor 清理。但“普通单轮默认只读”还没有成为全链路硬不变量，它还不是 Codex / Claude Code 级别的成熟 coding agent。

主要差距不是“prompt 写得不够多”，而是主链路中仍有几类权力没有彻底收敛：

```text
RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> RuntimeStartPacket
-> ProtocolSanitizer
-> ModelTurnDecision
-> ModelResponseProtocol
-> ActionAdmission
-> ActionPermit
-> ExecutionLoop
-> ProtocolRecovery
-> StableDynamicContext
-> CanonicalOutputBoundary
-> Presentation
-> MemoryCommit
```

当前项目已经拥有许多正确零件，但还存在下列成熟度缺口：

- 模型调用前没有一个 Codex 式统一 `for_prompt()` / protocol sanitizer，tool call 和 tool result 的合法性仍分散处理。
- 用户意图反馈仍有隐藏入口裁决：`single_agent_turn.py` 里有关键词判断，`ActiveTurnRegistry.steer()` 也会在入口层直接接管用户输入。
- `RuntimeToolControlPlane` 已经方向正确，但 task-run 和 agent-turn 工具执行仍有 `run()` / `run_core()` 双路径，subagent 也仍是 operation 特判。
- `ToolResultEnvelope` 已存在，但幂等键、状态外部化、aborted/canceled/denied 结果闭环还没有成为所有路径的硬规则。
- `FileStateAuthority` 已落地雏形，但仍主要从 observations 重建，不是持久 task-local 文件状态权威。
- `ArtifactAuthority` 后端主链已完成切换，task executor、dynamic context、graph 固定入口适配、monitoring、progress projection 已复用统一 artifact refs authority；前端 `PublicRunActivity` 已消费 logical artifact refs 展示产物状态，不把 `absolute_path` 作为主展示路径；后续重点转向外围残留守护。
- memory、continuation、restore candidates 的设计方向是对的，但还没有完全接入当前轮裁决链，容易让“恢复候选”在部分路径里变成实际决策。
- prompt 语言基本符合 agent 可执行要求，但纠错、恢复、协议修复、上下文裁剪还没有全部结构化为 runtime 不变量。
- 输出边界已有 `runtime.output_boundary`，且后端 final answer、presentation、memory commit 已接入 canonical output decision；前端 store/events/message presentation 已消费 canonical state、persist policy、fallback reason 和 leak flags；后续重点是继续清查外围直接 sanitize 路径。
- `runtime.output_boundary.output_models` 中的重名 `ToolResultEnvelope` 已清理，当前保留的是不与工具结果协议冲突的 `ToolVisibleOutputEnvelope`。
- action admission、permissions、operation gate、tool control plane 的边界还没有写成一张不可冲突的 permit 链，容易出现“入口允许、工具层拒绝”或相反的双重权威。

因此，本计划是在 `049-VibeCoding文件状态权威链与Agent流畅度重构计划` 之上继续升级：把“文件状态权威链”扩展成完整成熟 coding agent 主链路。

好的方向：

### 2.2 当前冲突点

#### 冲突 A：用户意图裁决多头

证据：

```text
backend\harness\loop\single_agent_turn.py
backend\harness\runtime\active_turn.py
backend\request_intent\request_signals.py
backend\context_system\current_turn\turn_binding.py
```

问题：

- `single_agent_turn.py` 仍有 `_user_message_targets_active_work()` 关键词判断，用来决定是否注入 `active_work_context`。
- `ActiveTurnRegistry.steer()` 在有 `expected_active_turn_id` 时会直接把用户消息作为当前任务 steer，`intent="active_turn_steer"`。
- `RequestSignals` 的设计反而更成熟：它只观察路径、材料、弱能力需求，明确 `tool_selection_allowed=False`，不替模型选择动作。

目标：

- `ActiveTurnRegistry` 只保留并发锁、当前 turn 句柄、队列和安全接管，不做语义意图裁决。
- “继续/暂停/补充/质疑/新问题”必须由 `ModelTurnDecision` 或结构化 current-turn decision packet 裁决。
- 入口层只能生成 `RequestFacts` 和 `ActiveWorkCandidate`，不能直接改写成任务语义。

#### 冲突 B：模型协议合法性不是统一入口不变量

证据：

```text
backend\runtime\shared\history_assembler.py
backend\harness\runtime\dynamic_context\history_projector.py
backend\harness\loop\single_agent_turn.py
backend\runtime\tool_runtime\provider_tool_call_adapter.py
```

问题：

- `assemble_runtime_history()` 只保留 user/assistant 普通消息，tool 协议由 `api_protocol_messages`、`session_context.api_transcript`、follow-up 手写消息等多处处理。
- `single_agent_turn.py` 手动拼 assistant tool call message 和 tool observation message。
- 没有 Codex 式“每次送模型前统一修正缺失 output、删除 orphan output、注入 aborted”的 sanitizer。

目标：

- 新增统一 `ProtocolSanitizer`，所有模型调用前都经过它。
- `accepted action` 如果没有 observation，必须生成 `aborted` envelope。
- orphan tool output 不能进入 prompt。
- interrupted turn 只能在结构清洗后注入 continuation fact。

#### 冲突 C：工具控制平面仍有双执行路径

证据：

```text

backend\runtime\tool_runtime\tool_executor.py
backend\harness\loop\task_executor.py
backend\harness\loop\single_agent_turn.py
```

问题：

- task-run 走 `tool_runtime_executor.run(...)`。
- agent-turn 走 `tool_runtime_executor.run_core(...)`。
- subagent 通过 `_is_subagent_operation()` 特判进入 `SubagentControl`。
- single-turn 中 `side_effect_policy="runtime_authorized"`，而 `tool_control_plane` 里 agent turn sandbox 可能允许 `op.write_file/op.edit_file/op.shell/op.python_repl`，这与“普通单轮默认只读”的成熟边界有潜在冲突。

目标：

- 工具执行统一为 handler registry/control plane，不按 caller_kind 分裂执行核心。
- subagent 作为正式 handler 能力挂入 registry，不靠 operation name 前缀特判。
- 普通 single-turn 默认只读；写入/命令/长期验证必须进入 explicit task contract 或明确授权的 coding runtime。
- 所有路径输出同一 `ToolResultEnvelope`。

- `ActionAdmission` 只负责 action schema、allowed action type、tool visibility、side-effect 是否必须进入 contract，以及稳定 action id。
- `ActionPermit` / `OperationGate` 负责 operation/resource/sandbox/approval 的最终授权，并产出 permit receipt 或 denied envelope。

证据：


问题：

- 全链路统一使用 `caller_ref + action_request_id + tool_call_id` 派生 action/tool/result/replacement idempotency key。
- `execution_context_id` 可以是记录 id，但不能进入恢复和去重主键。
- denied、canceled、aborted、output replacement 和 resume projection 必须复用同一 idempotency key。

## 3. 目标权威链

目标不是新增一套 vibe coding 专属循环，而是把通用 coding agent 能力做成熟。vibe coding 只作为一种 task environment/profile 消费这些能力。

```text
User Input
-> RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> RuntimeStartPacket
-> ProtocolSanitizer
-> ModelTurnDecision
-> ModelResponseProtocol
-> ActionAdmission
-> ActionPermit
-> ToolControlPlane
-> ToolResultEnvelope
-> FileStateAuthority
-> ArtifactAuthority
-> TaskRuntimeState
-> ProtocolRecovery
-> StableDynamicContext
-> CanonicalOutputBoundary
-> Presentation
-> MemoryCommit
```

### 3.1 RequestFacts

职责：

- 记录用户原文、显式路径、材料类型、active turn 是否存在、expected id 是否存在、当前 session scope。
- 只观察，不决定。

禁止：

- 用关键词决定继续任务。
- 在入口层把用户消息改写为任务 steer。
- 直接决定工具或任务类型。

### 3.2 BoundaryPolicy

职责：

- 决定当前请求是否允许回答、是否需要拒绝、是否需要权限、是否必须进入 task contract。
- 决定普通 single-turn 是否只读。

禁止：

- 解释用户语义目标。
- 因为有 active work 就强行劫持当前用户消息。

### 3.3 ContextCandidates

职责：

- 汇总 memory candidates、restore candidates、file state、artifact state、active work state、recent work outcome。
- 候选只作为模型上下文或当前轮决策输入。

禁止：

- restore candidate 直接覆盖当前轮事实。
- memory 候选直接触发旧任务恢复。

### 3.4 RuntimeStartPacket

职责：

- 编译模型可见 prompt、tools、permission snapshot、context segments、prompt manifest。
- 生成待清洗的模型输入包；模型输入包在送模型前必须经过 `ProtocolSanitizer`。

禁止：

- 临时拼 tool output。
- 从 raw observation 长期推断文件或产物事实。

### 3.5 ProtocolSanitizer

职责：

- 作为所有模型调用前的唯一协议清洗入口。
- 保证 assistant tool call 与 tool output 成对，缺失结果转为 `aborted` fact，孤儿 tool output 转为 diagnostic 或删除。
- 记录清洗诊断、history version / transcript version、continuation fact。

禁止：

- 用自然语言修复替代结构修复。
- 让旧 `api_transcript`、手写 follow-up tool message 或脏 resume transcript 绕过清洗。

### 3.6 ModelTurnDecision

职责：

- 模型决定 respond、ask_user、tool_call、request_task_run、active_work_control、block。
- 对用户意图作语义判断并给出用户可见反馈。

禁止：

- 由入口层关键词替代模型判断。
- 由文件/产物/记忆候选层替代模型判断。

### 3.7 ModelResponseProtocol

职责：

- 解析 provider response、native tool calls、JSON action object、stream recovery partial output。
- 将模型输出归一成 `ModelTurnDecision` / `ModelActionRequest` / `OutputCandidate`，并把解析失败交给 `ProtocolRecovery`。

禁止：

- 在解析层私自批准工具、恢复旧任务或写文件。
- 把 inline pseudo tool call 当成已经执行过的工具结果。

### 3.8 ActionAdmission

职责：

- 校验 action type、tool visibility、side-effect 是否必须进入 task contract。
- 为 accepted action 分配稳定 action id / action_request_id。

禁止：

- 修改用户意图。
- 把 denied/error 吞掉。
- 直接承担 operation/resource/sandbox/approval 的最终授权。

### 3.9 ActionPermit

职责：

- 基于 permission mode、resource policy、operation gate、sandbox、approval token 产出 permit receipt。
- 将 allow / denied / needs_approval / needs_contract 写成结构化结果。

禁止：

- 重新解释用户语义目标。
- 让 ToolControlPlane 在没有 permit receipt 的情况下执行 side-effect 工具。

### 3.10 ToolControlPlane

职责：

- 统一工具 handler registry。
- 统一 pre/post hooks、permit enforcement、handler lifecycle、cancellation。
- 执行时必须携带 `ActionPermit` / permit receipt，并把 sandbox 执行结果写入 envelope。
- 输出统一 `ToolResultEnvelope`。

禁止：

- task executor 或 single turn 自己绕过 control plane。
- subagent 走隐藏特判。
- 在没有 permit receipt 的情况下自行批准 side-effect 工具。

### 3.11 ToolResultEnvelope

现有 `backend\runtime\tool_runtime\tool_result_envelope.py` 的 `ToolResultEnvelope` 保留，不新增平行 envelope。`runtime.output_boundary.output_models.ToolResultEnvelope` 必须删除或重命名，不能继续以同名类型存在。

目标字段必须稳定：

```text
envelope_id
idempotency_key
tool_call_id
action_request_id
caller_kind
caller_ref
tool_name
normalized_args
status: ok|error|denied|needs_approval|needs_contract|aborted|canceled
text_preview
structured_payload
observed_paths
matched_paths
written_paths
artifact_refs
file_state_events
artifact_state_events
verification_events
operation_gate
execution_receipt
replacement_refs
diagnostics
```

幂等主键：

```text
caller_ref + action_request_id + tool_call_id
```

`envelope_id` 可以是记录 id，但不能作为恢复和去重主键。

### 3.12 FileStateAuthority

职责：

- task-local 文件状态权威。
- 保存路径归一化、mtime/hash、read coverage、partial view、stale、write events、search hits。

模型可见投影只保留短事实：

```text
path
status
read_coverage
total_lines
last_changed_by
next_suggested_read
evidence_refs
```

禁止：

- 把全文塞进 file_state。
- 让 latest_observations 继续承担文件记忆。

### 3.13 ArtifactAuthority

职责：

- 注册、去重、存在性检查、发布、验证、投影 artifact refs。
- 统一 tool artifact、contract artifact、final answer artifact、固定入口 graph 已产出 refs。

禁止：

- 修改 graph 调度。
- 各 projector 自己扫描 payload 形成不同 artifact 事实。

### 3.14 TaskRuntimeState

职责：

- 聚合 task lifecycle、file_state、artifact_state、verification_state、active work、recoverable errors、pending user steer。
- 作为 long task resume 和 dynamic context 的稳定事实源。

禁止：

- 从旧 observation 反推长期事实。
- task-local truth 泄漏成 main thread truth，除非通过 explicit summary/ref。

### 3.15 ProtocolRecovery

职责：

- 处理 model action JSON 解析失败、tool call/output 不配对、中断、取消、重复失败、输出超限。
- 产生结构化 recovery fact，而不是散落 prompt。

### 3.16 StableDynamicContext

职责：

- 从 `TaskRuntimeState`、`FileStateAuthority`、`ArtifactAuthority`、replacement state 和 sanitized history 编译下一轮上下文。
- 明确 stable / volatile / never_cache 分段，保护 prompt cache 前缀稳定。

禁止：

- 让 `latest_observations`、raw child logs 或未验证 artifact 成为长期事实源。

### 3.17 CanonicalOutputBoundary

职责：

- 统一生成 canonical answer、persist policy、finalization policy、leak flags、rejected candidates。
- 最终回答、前端 presentation、memory writeback 必须消费它的 decision。

禁止：

- 把 raw observation、tool raw output、内部协议字段或未验证 pseudo tool call 直接作为最终回答或 memory 内容。

### 3.18 Presentation

职责：

- 给用户展示清楚的当前判断、下一步、阻塞原因、产物、已读/已改/待验证状态。
- 不泄漏 runtime packet、task id、隐藏推理、内部协议字段。

### 3.19 MemoryCommit

职责：

- 将 canonical output、用户偏好、任务 durable facts 写入对应 memory layer。
- 写入必须在 output boundary 后，不能用未验证 raw observation 污染长期记忆。

## 5. 固定执行流

实施后的主链：

```text
1. 用户输入进入 HarnessRuntimeFacade。
2. RequestFacts 捕获事实，不做语义动作裁决。
3. BoundaryPolicy 决定允许边界和 side-effect 边界。
4. ContextCandidates 汇总 memory/file/artifact/active work 候选。
5. RuntimeCompiler 编译 RuntimeStartPacket。
6. ProtocolSanitizer 清洗 history/tool protocol。
7. ModelTurnDecision 输出结构化动作。
8. ModelResponseProtocol 归一 provider response、native tool calls、JSON action 和 partial stream output。
9. ActionAdmission 校验 action schema、工具可见性和 side-effect contract 边界。
10. ActionPermit 校验 permission、operation gate、sandbox、approval。
11. ToolControlPlane 通过统一 handler 执行。
12. ToolResultEnvelope 记录 ok/error/denied/needs_approval/aborted/canceled。
13. FileStateAuthority / ArtifactAuthority / TaskRuntimeState 消费 envelope 事件。
14. ProtocolRecovery 处理中断、失败、修复和恢复。
15. StableDynamicContext 编译下一轮上下文。
16. CanonicalOutputBoundary 生成 canonical answer、persist policy、leak flags。
17. Presentation 输出用户可见进展或最终回答。
18. MemoryCommit 在 canonical output 后写入允许的 memory layer。
```

禁止流：

```text
用户消息 -> 关键词判断 -> 直接当作当前任务 steering
tool result -> 各 projector 自己扫 artifact_refs
observation -> latest_observations 长期保存文件事实
resume -> old api transcript 直接回灌模型
single turn -> sandbox 自动允许写文件/命令
admission allow -> 未经 ActionPermit 直接执行 side-effect 工具
execution_context -> uuid idempotency -> replacement/resume 漂移
subagent -> raw child logs 混入父线程
output_boundary.ToolResultEnvelope -> 被当成 runtime tool result 协议
raw output / pseudo tool call -> final answer / memory
graph artifact adapter -> 反向修改 graph 调度
prompt -> 要求模型别重复读，但 runtime 没有 file_state
```

## 8. 迁移与删除规则

### 8.1 Shadow 阶段

允许新旧投影短期并行，但只用于 diagnostics：

- 新 authority 必须写入诊断。
- 测试断言新旧一致。
- shadow 不作为长期兼容分支。

### 8.2 Cutover 阶段

满足后切主链：

- ProtocolSanitizer 覆盖 single-turn/task-run/model follow-up。
- ModelResponseProtocol 覆盖 provider response、native tool calls、JSON action 和 partial stream recovery。
- ActionPermit 覆盖 permission、operation gate、sandbox、approval，ToolControlPlane 不再自行补第二套授权语义。
- ToolResultEnvelope 覆盖所有工具结果状态，并使用统一幂等键。
- FileStateAuthority 持久覆盖 read/search/write/edit/stat/path_exists。
- ArtifactAuthority 覆盖 tool/contract/final answer/fixed graph refs。
- DynamicContext 从 authority 读取稳定事实。
- CanonicalOutputBoundary 覆盖 final answer、presentation 和 memory writeback。

### 8.3 删除阶段

切换后删除：

- `_user_message_targets_active_work()` 关键词主链。
- 多处重复 artifact refs 主链扫描。
- 无 envelope 的 tool observation 路径。
- latest_observations 长期文件事实推断。
- task-run / agent-turn 工具执行双核心。
- subagent operation 前缀特判。
- `runtime.output_boundary.output_models.ToolResultEnvelope` 重名类型。
- `execution_context.py` 中基于 uuid 的幂等主键。
- final answer 只做 sanitize、未经过 canonical output decision 的路径。
- ActionAdmission / permissions / operation gate 重复承担同一最终授权的旧测试。
- 只为兼容旧链路保留的测试和投影。

### 8.4 回滚规则

允许回滚入口：

- 可以临时恢复旧 projector 读取作为诊断。
- 不允许恢复无 envelope 工具结果。
- 不允许恢复单轮 side-effect 模糊授权。
- 不允许恢复旧关键词意图主链。
- 不允许恢复重名 `ToolResultEnvelope` 或 uuid 幂等主键。
- 不允许恢复绕过 `CanonicalOutputBoundary` 的 memory writeback。
- 双主链不得超过一个修复周期。

### 9.4 对其它任务环境的影响边界

本计划会影响其它任务环境，但影响必须分层：

#### 应该影响所有环境的通用 runtime 层

这些修复是成熟 agent 的公共底座，应覆盖写作、研究、文档处理、数据分析、通用任务和 coding：

- `ProtocolSanitizer`：所有模型调用前都必须清洗 tool call/output、resume transcript、partial stream。
- `ModelResponseProtocol`：所有模型响应都必须统一解析为 action/output candidates。
- `ActionAdmission` / `ActionPermit`：所有环境都使用同一套 action schema、工具可见性、permission、operation gate、sandbox、approval 授权链。
- `ToolResultEnvelope`：所有工具结果都进入统一 envelope，状态、幂等键、artifact refs、file state events 一致。
- `ArtifactAuthority`：所有环境的 tool artifact、contract artifact、final answer artifact、固定入口 graph refs 都统一注册和投影。
- `CanonicalOutputBoundary`：所有 final answer、presentation、memory writeback 都必须经过 canonical output decision。

这些是正向影响：它们减少串台、脏 transcript、权限漂移、raw output 泄漏和记忆污染，不应为某个环境保留旧旁路。

#### 不能外溢到其它环境的 coding 专属能力

这些能力只能由对应 task environment / runtime profile 显式开启，不能因为 `050` 主链升级而变成全局默认：

- `env.coding.vibe_workspace` 的 sandbox overlay、项目文件状态、shell/python、git、浏览器验证、项目级 artifact 验证。
- `file_profile.managed_project_workspace` 的 project workspace / sandbox workspace / git worktree view / test artifacts。
- coding prompt 中的实现、测试、补丁、验证、git 语义。
- coding runtime 的较高工具轮次、长期任务推进、测试闭环和文件重读策略。

其它环境必须继续按自身 spec 执行：

- `env.creation.writing`：使用 `file_profile.writing_manuscript`，shell/browser 默认 denied，官方作品写入仍需要 review receipt / commit gate。
- `env.development.sandbox`：可使用通用开发沙箱，但不自动继承 managed project workspace 的 repository 命名和 coding prompt。
- `env.general.workspace`：保持 task_decided / permission_context，不默认开放 shell、真实 workspace 写入或 git 操作。
- 数据分析、文档处理、web evidence 等文件 profile 只开放各自 repository、artifact 和 source material 规则。

#### 实施守则

- 禁止在主链中写死 `env.coding.vibe_workspace` 分支；主链只能读取 environment spec、file profile、runtime profile、tool package 和 permission policy。
- 禁止把 `file_profile.managed_project_workspace` 作为所有环境默认文件 profile。
- 禁止把 coding 的 side-effect permit、shell/browser/git 操作扩散到写作、研究、文档或 general 环境。
- `FileStateAuthority` 和 `ArtifactAuthority` 可以通用，但它们的读写范围必须来自当前环境的 file access table / artifact policy。
- prompt contract 要按环境角色生成：写作环境不能收到 coding agent 的开发式执行说明，coding 环境也不能收到写作节点式提示。
- 环境 prompt 优先承载可由语言约束稳定解决的职责：角色身份、任务边界、质量标准、输入解释、输出格式、禁止事项、失败反馈和用户意图回问。
- 控制器只承载 prompt 不能可靠保证的执行权：环境切换、工具显隐、权限准入、side-effect permit、文件访问范围、artifact 归属、状态推进和失败恢复。
- 不允许为了某个环境新增一套硬编码循环。新增环境能力必须通过 environment spec、runtime profile、tool package、file profile、permission policy 和 prompt contract 组合表达。

## 10. 最终成熟形验收标准

系统达到成熟 coding agent 形态时，应满足：

- 用户意图反馈清晰：系统能说明当前判断、为什么进入任务或不进入任务、下一步是什么。
- 普通单轮不被旧任务劫持。
- active work 控制由模型/turn decision 裁决，入口层只做并发和安全。
- 每次模型调用前都有 protocol sanitizer。
- tool call/output 永远成对；缺失结果会变成 aborted fact。
- permission、operation gate、sandbox、approval 由 ActionPermit 统一授权；ToolControlPlane 只消费 permit 并统一执行、取消和记录。
- 所有工具结果都有统一 ToolResultEnvelope。
- 文件状态持久、短、稳定，避免重复读。
- 产物状态唯一，前端、final answer、下一轮 context 一致。
- memory 只提供候选和写入后的 canonical facts，不抢当前轮决策。
- resume 清洗 transcript，不把脏历史回灌模型。
- subagent 隔离上下文，父线程只接收 summary/refs/deltas。
- prompt 是 agent 可执行职责说明，不是开发节点说明。
- 前端展示权威状态，不展示 raw logs 作为主要进展。
- 旧旁路、重复抽取、隐藏兜底和过时测试被删除。

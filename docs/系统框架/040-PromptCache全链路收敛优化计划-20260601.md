# Prompt Cache 全链路收敛优化计划

日期：2026-06-01

## 背景

DeepSeek 真实账本显示，新版 `RuntimeCompiler -> RuntimeInvocationPacket -> ModelRuntime` 链路在受控默认装载下可以稳定命中缓存；第三轮后 DeepSeek 官方 usage 可达到约 90% prompt cache hit rate。

但正式账本仍暴露出全链路未完全收敛的问题：

- 存在 `unknown_unplanned` / `prefix_tier=none` 的模型请求，说明仍有入口裸调 `ModelRuntime.invoke_messages`，没有携带 segment plan。
- 旧账本和部分链路中出现 `runtime_boundary` 这类动态 runtime 信息被标注为 stable prefix 的记录。
- 部分 stable segment 在同类调用之间 hash 变化，尤其 turn / observation followup / graph task execution 混合账本中较明显。
- 诊断脚本目前能发现问题，但还不能清晰区分“当前架构新链路问题”和“历史旧账本残留问题”。
- 如果调用方传入无效 `prompt_pack_refs`，当前装配会记录 rejected refs，但可能导致默认全局 runtime pack 没有进入消息最前缀，从而让 `provider_global_prefix_tokens=0`。

本计划目标不是提高某一个样本命中率，而是把模型调用权威入口统一，确保所有项目内模型请求都被分段、分层、计账和诊断。

## 目标

1. 所有生产模型调用必须携带明确的 `segment_plan`，不得出现无计划的 agent/runtime 主链路请求。
2. 动态 runtime、观测、历史、当前用户输入、执行状态、重试 attempt 等内容只能进入 `volatile` 或 `none` tier。
3. `provider_global` 只允许真正跨会话、跨任务、跨节点稳定的全局运行规则。
4. `session` 只允许同会话稳定内容。
5. `task` 只允许同一任务/节点稳定内容。
6. utility/RAG/finalizer 等非主 agent 调用必须显式标注为 utility scope，不混入 agent runtime cache KPI。
7. 诊断报告必须能区分：
   - 当前新链路是否健康；
   - 历史旧链路残留；
   - 裸调用；
   - post-warm 仍 miss；
   - stable 段真实变更。

## 权威链路

目标链路：

```text
RuntimeCompiler / UtilityPromptPlanner
-> RuntimeInvocationPacket 或 UtilityInvocationPacket
-> ModelRuntime._begin_prompt_accounting
-> ModelRequestBuilder
-> CanonicalPromptSerializer
-> PromptCachePlanner
-> DeepSeek provider usage
-> PromptStabilityReporter / diagnose_deepseek_prompt_cache
```

权责划分：

- `RuntimeCompiler`：唯一负责 agent 主链路 prompt 装配、稳定/动态边界、segment plan。
- `UtilityPromptPlanner`：负责标题、摘要、RAG finalizer、active work decision 等小模型调用的最小 segment plan。
- `ModelRuntime`：拒绝或降级标记无 plan 请求；不得猜测 agent 主链路结构；所有 invoke/stream 入口都必须进入同一 accounting gate。
- `PromptSegmentPlan`：硬校验 tier，不允许 runtime 字段进入 stable tiers。
- `PromptAccounting`：记录真实账本，不参与 prompt 语义决策。
- `diagnose_deepseek_prompt_cache.py`：只诊断事实，不修改运行行为。

## 现状问题与处理

### 1. 裸调用没有 segment plan

证据：

- 正式账本近期有 `unknown_unplanned`、`prefix_tier=none`、`status=bypassed` 请求。
- 典型来源包括 RAG answer finalizer 等直接调用 `model_runtime.invoke_messages(messages)` 的路径。
- 生产代码中除 `invoke_messages` / `invoke_messages_with_tools` 外，还有 `astream_messages`、`astream_messages_with_tools`、`astream_conversation` 等流式入口，也必须纳入同一规则。

处理：

- 新增统一 utility segment plan builder，覆盖 utility/finalizer 类调用。
- 修改裸调用入口，必须传入 `accounting_context.segment_plan` 和 `cache_metric_scope`。
- 对生产环境 agent 主链路请求，如果 `accounting_context` 缺少 `segment_plan`，账本中显式记录 `unplanned_model_call` issue；测试中收紧断言。
- `ModelRuntime` 的 invoke 与 stream 入口共享同一个 unplanned 检测逻辑，避免只修非流式调用。

### 1.1 prompt pack 静默丢失全局前缀

证据：

- 受控复查中，当 `runtime_assembly.profile.prompt_pack_refs` 指向不存在的 pack 时，`PromptAssemblyService` 会把该 ref 放入 rejected refs，但不会回退默认 pack；最终消息首段缺失 `global_static`，`provider_global_prefix_tokens=0`。

处理：

- 主 agent 链路中，显式传入的 prompt pack ref 被 rejected 时必须暴露为编译错误或诊断失败，不允许静默继续。
- 如果调用方没有传入 prompt pack refs，继续使用 `default_pack_ref_for_invocation`。
- 增加测试：无效 prompt pack ref 不得生成缺失 `global_static` 的可执行 packet。

### 2. 动态 runtime 曾进入 stable prefix

证据：

- 诊断脚本发现 `runtime_boundary` 带有 `volatility_reason` 但 `cache_role=session_stable`。
- 当前 `RuntimeCompiler` 新链路已经把 `dynamic_projection` 标为 volatile，但正式账本包含旧结构或未收敛入口。

处理：

- 保留 `PromptSegmentPlan` 硬校验。
- 增加回归测试覆盖 `single_agent_turn`、`task_execution`、`tool_observation_followup` 三条主链路，断言动态段均为 `prefix_tier=volatile`。
- 清理或替换仍生成 `runtime_boundary` stable 段的旧测试/旧入口；不得以兼容方式保留旧链路。

### 3. stable segment hash 变化

证据：

- `turn_context`、`task_stable`、`system_session` 等稳定段在正式账本中出现多个 hash。
- 部分变化是合理的任务/节点变更；部分是历史、工具、动态上下文进入 stable 的结果。

处理：

- 将诊断按 invocation/session/task 分组，避免把不同任务的合理变化误报为同一 stable 段漂移。
- 对同一 `session_id + invocation_kind + source_ref` 下的 stable hash 变化保留告警。
- 对同一 `task_run_id/graph node` 下的 stable hash 变化升级告警。

### 4. DeepSeek hit rate KPI 混杂

证据：

- utility/unplanned 调用与 agent runtime 调用混在整体命中率里，会拉低或扭曲全局判断。

处理：

- `cache_metric_scope` 统一为：
  - `agent_runtime`
  - `utility_minimal_plan`
  - `rag_finalizer`
  - `unplanned_model_call`
- 诊断默认展示 overall，同时重点展示 agent runtime steady-state hit rate。
- `rag_finalizer` 是 utility 类 scope 的细分，不计入 agent runtime KPI；诊断表中单独展示，避免和 `utility_minimal_plan` 混为一类后看不出 RAG finalizer 成本。

## 实施步骤

### 阶段一：补齐全入口 segment plan

改动文件：

- `backend/runtime/model_gateway/model_runtime.py`
- `backend/evidence/output_policy.py`
- 可能涉及 `backend/harness/loop/active_work.py`
- 可能涉及标题/摘要 utility 调用所在文件

动作：

- 抽出或新增 utility segment plan helper。
- 给 RAG finalizer 增加 `accounting_context`。
- 检查 `invoke_messages`、`invoke_messages_with_tools`、`astream_messages`、`astream_messages_with_tools`、`astream_conversation` 所有非测试调用点，补齐 segment plan 或明确 utility scope。
- 对无 plan 请求记录 `unplanned_model_call` scope。
- 对 `PromptAssemblyResult.rejected_refs` 做主链路硬校验，防止无效 pack 静默破坏 provider-global prefix。

### 阶段二：收紧主链路 prompt tier 断言

改动文件：

- `backend/harness/runtime/compiler.py`
- `backend/harness/runtime/prompt_segment_plan.py`
- `backend/prompt_library/assembly.py`
- `backend/tests/dynamic_prompt_context_projection_test.py`
- `backend/tests/prompt_cache_prefix_tier_regression.py`
- `backend/tests/prompt_accounting_ledger_test.py`

动作：

- 主链路 dynamic/volatile 段必须带 dynamic context metadata。
- 主链路 stable 段不得包含 runtime ids、attempt、observations、pending_user_steers、runtime_envelope。
- 为 `single_agent_turn`、`task_execution`、`tool_observation_followup` 增加全链路分层断言。
- 为无效 prompt pack ref 增加断言：不能生成缺失默认 runtime pack 的 packet。

### 阶段三：诊断脚本升级

改动文件：

- `backend/scripts/diagnose_deepseek_prompt_cache.py`
- `backend/tests/deepseek_prompt_cache_diagnostics_test.py`

动作：

- 增加 `unplanned_model_call` 统计。
- 增加 post-warm hit rate 汇总。
- 增加按 `cache_metric_scope` 的命中率表。
- stable hash 变化按 `invocation_kind/session/task/source_ref` 分组，减少跨任务误报。
- 历史旧链路残留单独归类，不混同当前新链路问题。

### 阶段四：真实验收

动作：

- 跑 focused tests。
- 跑正式 prompt accounting 诊断。
- 再跑一轮真实 DeepSeek 默认装载验收。
- 如涉及前后端运行链路，再按项目固定端口启动：
  - backend `127.0.0.1:8003`
  - frontend `127.0.0.1:3000`

验收标准：

- 新产生的 agent runtime 请求不再出现 `unknown_unplanned`。
- 新产生的 agent runtime 请求均有 `segment_plan_ref`。
- 新产生的 invoke 与 stream 请求均有明确 `cache_metric_scope`。
- 主链路 dynamic segments 全部为 `prefix_tier=volatile`。
- 主链路 packet 必须包含 `global_static` provider-global segment；显式无效 prompt pack ref 必须失败或被诊断为不可执行。
- 默认装载验收 post-warm 命中率稳定高于 80%。
- 诊断中 post-warm repeated prefix miss 不再由首轮冷启动触发。
- utility/RAG/finalizer 不计入 agent runtime KPI。

## 风险与边界

- 正式账本包含历史旧记录，不能把旧记录当成当前代码必然仍存在的问题。实施后需要用新 baseline 或时间窗口判断。
- DeepSeek 服务端缓存有 warmup 和过期行为，不能要求第一轮命中。
- 本地 token 预测不是 DeepSeek 官方 tokenizer，只用于预算和分段诊断；真实 prompt tokens 以 provider usage 为准。
- 不保留旧 prompt 装配链路作为兼容 fallback；发现旧入口应迁移或删除。

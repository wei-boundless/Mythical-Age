本设计书用于指导后续重构。它不讨论“是否需要环境任务系统”，而是明确现有代码如何迁移到新的配置权威模型：环境负责资源边界、运行约束、静态 skill 候选和环境任务库；agent 负责工具权限、行为模式、审批和动态策略；task contract 负责本次目标、交付物、验收和显式资源/skill 要求；runtime assembly 只负责交集、过滤和投影。

## 0. 重构目标

目标主链：

```text
User / Task Contract
-> Resolve TaskEnvironmentCatalog
-> Resolve AgentRuntimeProfile
-> Resolve TaskContract Requirements
-> Project Skill Candidates
-> Project Operation Authorization
-> Compile RuntimeInvocationPacket
-> Agent decides action
-> System validates / authorizes / executes
-> Observation and artifact receipts return to agent
```

重构后的系统必须满足：

```text
环境不授予工具。
skills 可以由环境静态提供候选。
tools 只由 agent/system 授权链决定。
环境约束可以过滤工具，但不能新增工具。
task contract 可以请求资源和 skill，但不能扩大权限。
runtime mode 不能绑定 task environment。
task type 不能成为默认 skill/tool 权威。
runtime assembly 不做意图识别、不做任务原语选择、不使用关键词启发。
```

## 1. 当前代码结构诊断

### 1.1 已经完成的正确结构

| 文件 | 当前结构 | 评价 |
| --- | --- | --- |
| `backend/task_system/environments/models.py` | `TaskEnvironmentSpec` 已包含 `sandbox_policy`、`file_management`、`resource_space`、`memory_space`、`execution_policy`、`risk_policy`、`artifact_policy`、`runtime_policy` | 资源边界和约束主体已经具备 |
| `backend/task_system/environments/default_environments.py` | 默认环境已收敛为 `creation / development / general` 三大资源场景，具体环境描述已改成资源边界和约束说明 | 顶层语义方向正确 |
| `backend/task_system/environments/catalog.py` | `TaskEnvironmentCatalog` 已作为 runtime/API 共用的环境资源出口 | 环境资源目录已经有统一入口 |
| `backend/harness/runtime/assembly.py` | runtime assembly 已通过 catalog 解析环境；无显式环境时回到 `env.general.workspace` | 已去掉 mode 绑定环境的旧方向 |
| `backend/agent_system/profiles/runtime_profile_models.py` | agent profile 持有 `allowed_operations`、`blocked_operations`、runtime modes、delegation、approval | 工具权限主权来源明确 |
| `backend/capability_system/skill_registry.py` | skill 有 runtime contract 和 prompt view | skills 具备独立候选能力基础 |

### 1.2 仍然存在的问题

| 问题 | 当前位置 | 风险 |
| --- | --- | --- |
| 环境没有 `skill_space` | `backend/task_system/environments/models.py` | 环境无法表达静态 skill 候选，只能靠 task binding 或 specific task 注入 |
| runtime packet 没有统一 skill candidate 投影 | `backend/harness/runtime/assembly.py`、`backend/harness/runtime/compiler.py` | agent 看不到“此环境推荐的静态技能”，只能看到工具和环境 prompt |
| tool authorization 没有显式记录环境约束过滤 | `backend/harness/runtime/assembly.py`、`backend/capability_system/tool_authorization.py` | 无法审计某个工具是被 agent 权限拒绝、环境约束拒绝，还是前端可见性过滤 |
| task binding 仍有 `skill_scope` 默认路径 | `backend/task_system/services/bindings.py`、`backend/task_system/tasks/definitions.py` | 旧任务类型可能继续成为 skill 权威，重新制造“任务分类控制” |
| task assembly 中 skill operations 可能参与 operation requirements | `backend/task_system/services/assembly_builder.py` | 需要确保 skill.required_operations 只表达需求/可用性，不反向授予工具 |
| 环境配置仍以内置代码为主 | `default_environments.py`、`registry.py` | 难以在 UI/配置层定制环境、任务库和 skill 候选 |
| 前端只展示部分环境资源 | `frontend/src/lib/api.ts`、`TaskSystemView.tsx` | 无法完整管理 skill_space、约束、任务库和验证状态 |

## 2. 目标权威模型

### 2.1 Environment Authority

环境只拥有静态边界：

```text
resource_space
storage_space
file_management
sandbox_policy
execution_policy
risk_policy
artifact_policy
memory_space
environment_prompts
skill_space
task_library
```

环境不能拥有：

```text
allowed_operations
blocked_operations
runtime_mode
agent behavior
approval decision
delegation permission
task lifecycle trigger
tool authorization final decision
```

环境可以表达：

```text
shell_policy = sandboxed / denied / task_decided
network_policy = allowed / denied / task_decided
write_scope_policy = artifact_only / sandbox_or_file_access_table
default_skill_refs = skill candidates
```

环境不能表达：

```text
给 agent terminal 工具
给 agent image_generate 工具
professional 模式默认使用开发环境
这个任务应该开启 TaskRun
```

### 2.2 Agent Authority

agent 拥有动态权限和行为：

```text
allowed_operations
blocked_operations
enabled_runtime_modes
default_runtime_mode
allowed_runtime_lanes
allowed_memory_scopes
allowed_context_sections
can_delegate_to_agents
allowed_delegate_agent_ids
approval_policy
lifecycle_policy
model_profile
work_role_prompt
self-review policy
```

agent 不能拥有：

```text
environment storage roots
artifact roots
sandbox root policy
file repository topology
environment task library
environment prompt
```

### 2.3 Task Contract Authority

task contract 拥有当前任务的目标和验收：

```text
user_visible_goal
task_run_goal
completion_criteria
required_artifacts
required_verifications
resource_requirements
permission_requirements
skill_requirements
acceptance_policy
recovery_policy
```

task contract 只能请求，不能授权。

### 2.4 Runtime Assembly Authority

runtime assembly 只做装配：

```text
resolve
normalize
intersect
filter
project
diagnose
```

runtime assembly 禁止：

```text
根据关键词选择技能
根据 mode 选择环境
根据 task type 授权工具
根据 skill.required_operations 授权工具
根据环境直接发工具
```

## 4. 装配流程设计

### 4.1 Environment Resolution

入口：`assemble_runtime()`

当前：

```text
_resolve_runtime_task_environment()
-> build_task_environment_catalog()
-> runtime_environment_payload()
```

目标：

```text
_resolve_runtime_task_environment()
-> TaskEnvironmentCatalog.resolve_runtime_payload()
-> EnvironmentRuntimeProjection
```

输出必须包含：

```text
environment_id
group
environment_prompts
sandbox_policy
storage_space
resource_space
file_management
file_access_tables
memory_space
artifact_policy
execution_policy
risk_policy
runtime_policy
skill_space
task_library
```

### 4.2 Skill Projection

新增函数建议：

```text
backend/harness/runtime/skill_projection.py
```

输入：

```text
environment.skill_space
task_selection.skill_requirements
task_contract.skill_requirements
agent_profile.metadata.preferred_skill_refs
agent_profile.metadata.blocked_skill_refs
SkillRegistry
authorized_operation_ids
```

输出：

```text
skill_candidates: tuple[RuntimeSkillCandidate, ...]
filtered_skills: tuple[dict, ...]
```

规则：

```text
1. 环境 default_skill_refs 是静态候选底座。
2. task required_refs 追加并提高优先级。
3. agent preferred refs 可以追加。
4. environment denied > task denied > agent blocked，任何 denied 都移除或标记 denied。
5. skill.required_operations 只决定 availability，不改变 authorized tools。
6. missing skill definition 不阻塞 runtime，但必须进入 diagnostics。
```

### 4.3 Tool Authorization Projection

当前：

```python
allowed_operations = set(profile.allowed_operations)
tool_set = build_authorized_tool_set(...)
```

目标：

```text
agent_operation_ceiling = agent.allowed_operations - agent.blocked_operations
environment_operation_constraints = project_environment_operation_constraints(environment)
task_operation_requests = task.permission_requirements / task_selection
operation_projection = project_operation_authorization(...)
authorized_tools = build_authorized_tool_set(operation_projection.allowed_operations)
```

环境过滤规则：

| Operation | 环境字段 |
| --- | --- |
| `op.shell` / `op.python_repl` | `execution_policy.shell_execution_policy` |
| `op.browser_control` | `execution_policy.browser_execution_policy` |
| `op.web_search` / `op.fetch_url` | `execution_policy.network_execution_policy` |
| `op.write_file` / `op.edit_file` | `execution_policy.write_scope_policy` + `file_access_tables` |
| `op.image_generate` | `sandbox_policy.side_effect_operations` 或 artifact policy |

注意：这张表是 operation 到环境约束的结构映射，不是关键词启发式。映射应来自 tool definition 的 operation metadata，不能靠 tool name。

### 4.4 Runtime Packet Compilation

`RuntimeCompiler` 中稳定 payload 应增加：

```json
{
  "task_environment": {},
  "available_tools": [],
  "skill_candidates": [],
  "operation_authorization": {},
  "runtime_context": {}
}
```

给 agent 的说明应分层：

```text
环境说明：资源边界和约束。
技能候选：可用方法、适用场景、输入输出边界、是否可用。
工具列表：真实可调用工具。
任务合同：本次目标和验收。
```

禁止在 skill candidate 中写“你可以调用某工具”，只能写：

```text
这个 skill 需要以下 operations；当前 availability 是 available/partial/unavailable。
```

## 5. 任务系统迁移设计

### 5.1 TaskBinding 去权威化

当前 `TaskBindingRecord.skill_scope` 来自 `TaskDefinition.default_skill_refs`。

目标：

```text
TaskBinding.skill_scope 不再作为默认 skill 主来源。
环境 skill_space 成为默认候选来源。
specific task skill_requirements 保留为显式要求。
TaskDefinition.default_skill_refs 只作为迁移期输入，逐步清空或降级为 legacy_hint。
```

迁移步骤：

```text
1. 新增 environment.skill_space 后，先复制当前高价值 skill refs 到对应环境。
2. assembly_builder 中同时读取 environment.skill_space 和 existing skill_scope，输出 diagnostics 标记 source。
3. 加测试确认环境 skill candidates 存在。
4. 将 TaskDefinition.default_skill_refs 降级为 explicit task fallback。
5. 删除旧 default skill 注入或只保留 specific task 显式路径。
```

### 5.2 Specific Task 接入

specific task 不走意图识别，不自动选择。

路径：

```text
explicit selected_task_id / task contract
-> resolve_specific_task_assembly_policy()
-> policy.environment_id
-> TaskEnvironmentCatalog
-> task skill_requirements
-> runtime projection
```

规则：

```text
specific task 可以声明 task_environment_id。
specific task 可以声明 required skill refs。
specific task 可以声明 required operations，但 required operations 只是 task request，最终仍受 agent 权限和环境约束过滤。
```

### 5.3 Environment Task Library

环境任务库是资源，不是选择器。

```text
storage/task_environments/{namespace}/task_library
```

内容：

```text
task templates
task contracts
acceptance profiles
example resource requirements
skill requirement presets
```

系统可以展示任务库，可以按显式 ID 读取任务库，但不能根据自然语言自动匹配任务库。

## 7. 验证矩阵

| 场景 | 验收 |
| --- | --- |
| 无显式环境 + professional mode | 使用 `env.general.workspace` |

## 8. 扩展方向

### 8.1 环境 Marketplace

后续可以把环境做成可安装模板：

```text
development.webapp
development.data
creation.novel
creation.game-design
general.research
general.document-review
```

这些模板不是任务分类，而是资源边界模板。

### 8.2 Skill Pack 与环境缓存

环境 skill candidates 可以形成静态 prompt cache：

```text
environment prompt block
skill candidate cards
resource boundary summary
artifact policy summary
```

收益：

```text
减少每次 runtime 动态拼装成本。
让 agent 更稳定地理解当前资源边界。
避免每个 agent 重复配置相同 skill 候选。
```

### 8.3 环境级任务库

每个环境可以有独立任务库：

```text
development/sandbox/task_library
creation/writing/task_library
general/research/task_library
```

未来可支持：

```text
用户显式选择任务模板。
agent 在 runtime 内请求查看可用任务模板。
系统展示任务库，但不自动替 agent 选择。
```

### 8.4 环境级观测和审计

可以按环境统计：

```text
常用 skill
常用 tools
被拒绝 operations
artifact 成功率
sandbox publish 成功率
用户审批频率
```

这些数据可帮助调优环境配置，但不能反向变成隐式权限。

### 8.5 多 agent / graph harness 接入

graph harness 后续应复用同一套环境配置：

```text
graph node runtime
-> node agent profile
-> graph/task environment
-> node skill candidates
-> node tool authorization
```

不应为 graph harness 另建一套环境解释规则。

## 9. 不足点与风险

### 9.1 当前 skill registry 粒度不足

当前 `SkillDefinition` 可以表达 `requires_operations`，但缺少：

```text
环境适用性
资源前置条件
是否适合静态缓存
是否允许 partial availability
skill prompt token budget
```

需要后续扩展 skill contract。

### 9.2 operation 到环境约束的映射仍需规范

目前工具定义有 operation id，但缺少统一的 channel metadata。

需要补充：

```text
operation_channel = file_write / shell / browser / network / image / memory / delegation
side_effect_level
read_only
resource_scope
```

否则 environment constraint filter 容易退化成 operation id 判断表。

### 9.3 task assembly 旧链路仍有惯性

`build_task_execution_assembly_bundle()` 仍包含：

```text
select_runtime_task_definitions
_augment_skill_scope_from_runtime_intent
TaskDefinition.default_skill_refs
operation_policy merging
```

这些结构需要分阶段清理，不能一次性删除，否则 specific task 和旧 graph/task assembly 可能断链。

### 9.4 可持久化配置需要严格校验

环境配置一旦开放 UI 编辑，必须校验：

```text
storage namespace 不越界
file profile 存在
skill ref 存在
artifact root 合法
sandbox mode 与 execution policy 不冲突
legacy alias 不重复
环境被引用时不能删除
```

否则会把配置错误变成 runtime 隐性故障。

### 9.5 skills 和 tools 的用户可见解释要谨慎

前端和 packet 里必须区分：

```text
Skill candidate: agent 可参考的方法能力。
Available tool: agent 真实可调用动作。
```

如果 UI 把 skill 展示成“权限”，用户会误解系统边界。

## 10. 切换规则

### 10.1 Shadow 阶段

```text
保留旧 skill_scope。
新增 environment.skill_space。
runtime 输出 skill_candidates diagnostics。
不改变工具授权。
```

### 10.2 Cutover 阶段

```text
runtime packet 使用 environment skill candidates。
task binding skill_scope 降级为 legacy fallback。
tests 保护 skill 不授予工具。
```

### 10.3 Cleanup 阶段

```text
删除或清空 TaskDefinition.default_skill_refs。
删除 _augment_skill_scope_from_runtime_intent 中基于 capability_needs 的隐式 skill 扩展。
删除前端旧环境 id 默认值。
删除旧测试中保护任务类型默认 skill 的断言。
```

## 11. 最终验收标准

重构完成后，必须能回答以下问题：

```text
某个工具为什么可见？
  因为 agent allowed_operations 允许，环境约束未拒绝，task/system policy 未拒绝。

某个 skill 为什么可见？
  因为环境 skill_space 或 task contract 显式声明，并经过 agent/task/environment 过滤。

某个 artifact 应写到哪里？
  由 environment.storage_space 和 artifact_policy 决定。

某个任务为什么开启 TaskRun？
  因为 agent 在 runtime 内显式 request_task_run，且 admission 通过。

某个环境为什么被选择？
  因为用户/task contract/specific task 显式指定；没有显式指定则 fallback 到 general workspace。
```

只有这些答案都能从结构化 payload 和 diagnostics 中追踪出来，环境任务系统才算完成。

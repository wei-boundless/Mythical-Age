# Agent 系统框架总纲

日期：2026-05-27

## 1. 总原则

本项目是 agent 平台，不是普通聊天应用。所有系统设计必须围绕 agent 的闭环控制展开：

```text
系统装配 agent 当前可见世界
-> agent 基于上下文作出语义决策
-> loop 执行动作并观察结果
-> 系统记录状态并重新装配下一阶段上下文
-> agent 继续行动直到完成、暂停、失败或等待审批
```

核心边界：

```text
Harness 是 agent 管控系统的工程边界。
Runtime 负责阶段性上下文装配。
Loop 负责动态状态推进。
Agent 负责语义判断、计划和行动请求。
系统负责环境、权限、工具、文件、记忆、沙盒和审计护栏。
```

本项目后端控制系统的顶层目录应收敛为：

```text
backend/harness
```

Harness 内部分为两类核心职能：

```text
backend/harness/runtime
backend/harness/loop
```

Runtime 和 Loop 可以作为 harness 内部概念继续存在，但不再作为项目顶层控制系统名称。项目外层不再表达为“runtime 系统”和“loop 系统”，而表达为“harness 管控系统”。

本项目只允许两类拥有状态推进主权的 harness loop：

```text
AgentLoop
GraphLoop
```

除此之外，任何能力、工具、子 agent、搜索器、代码搜索器、验证器、图模块都不能单独长出 runtime 主权。

## 2. Harness、Runtime 与 Loop 的定义

### 2.0 Harness

Harness 是生产级 agent 管控系统。它不是测试夹具，也不是普通执行器，而是把静态装配和动态推进纳入同一个工程边界的控制层。

Harness 回答：

```text
这次 agent 执行由谁装配？
这次 agent 执行由谁推进？
装配结果如何交给 loop？
loop 如何调用模型、权限、工具、文件、记忆和沙盒服务？
执行事实如何沉淀为 checkpoint、event、trace 和 result？
```

Harness 负责：

```text
承载 runtime 子系统
承载 loop 子系统
暴露 AgentHarness / GraphHarness facade
统一 RuntimeStartPacket、LoopState、ResultEnvelope 的控制链
约束能力、工具、profile、graph module 不得自建控制循环
```

Harness 不负责：

```text
注册具体工具
实现具体文件存储
实现具体 memory 存储
实现具体 sandbox 后端
替 agent 做语义判断
替任务系统选择任务环境
```

测试系统也可以有 test harness，但它只能作为测试夹具和场景驱动器存在，不得成为生产控制逻辑的设计来源。生产代码不得 import 测试 harness。

### 2.1 Runtime

Runtime 不是一条独立执行分支，也不是某种能力名称。Runtime 是系统根据实时状态和当前阶段，为 agent 或图任务装配出来的静态运行合同。

Runtime 回答：

```text
当前阶段 agent 能看到什么？
当前阶段 agent 能请求哪些工具？
当前阶段有哪些权限边界？
当前阶段有哪些文件、memory、sandbox、模型和输出契约？
当前阶段有哪些预算、风险策略和终止条件？
```

Runtime 的核心产物：

```text
RuntimeStartPacket
AgentRuntimeSpec
GraphRuntimeSpec
ToolCapabilityTable
ExecutionPermit
ContextPack
FileView
MemoryView
OutputContract
RuntimePolicy
```

Runtime 不允许：

```text
替 agent 做语义决策
推进 loop 状态
执行工具
选择任务环境
扩大工具权限
从 domain_id、metadata、runtime lane、projection、current_turn_context 猜测任务目标
把 search、codebase search、verifier、subagent、graph module 命名成独立 runtime
```

### 2.2 Loop

Loop 是动态控制系统。Loop 消费 RuntimeStartPacket，推动状态流转，并根据结果要求 runtime 装配下一阶段上下文。

Loop 回答：

```text
下一步是否调用模型？
模型输出是 final、tool_call、delegate、ask_user 还是 plan？
工具调用是否进入权限链？
工具 observation 如何回灌？
状态如何更新？
何时继续、暂停、失败、完成或等待审批？
```

Loop 不允许：

```text
重新选择任务环境
改写任务目标
绕过权限系统
把具体能力变成新 runtime
在 finalizer 中补做本应由 loop 完成的状态推进
```

## 3. 平台主链

### 3.1 单 agent 主链

```text
ChatInput / TaskOrder
-> TaskEnvironmentSpec
-> SpecificTaskAssemblyPolicy
-> AgentFactory
-> RuntimeStartPacket
-> AgentLoop
-> Tool / Permission / File / Memory / Sandbox services
-> LoopState updated
-> RuntimeStartPacket for next stage
-> FinalResult / Receipt / Event
```

AgentHarness 在工程上应是 agent 执行门面，内部由 runtime 装配 RuntimeStartPacket，并由 AgentLoopController 完成动态推进。历史名称 AgentRuntime 只允许在明确迁移窗口内作为待删除兼容名存在。

### 3.2 图任务主链

```text
TaskGraphRuntimeSpec
-> GraphLoop
-> NodeWorkOrder
-> AgentFactory / ExecutorFactory
-> RuntimeStartPacket
-> AgentLoop for agent node
-> NodeResultEnvelope
-> GraphLoopState advance
-> GraphResult
```

GraphHarness 负责图调度、节点状态、handoff、merge、resume、human gate 和子图调用。内部可以持有 GraphRuntimeSpec，但不得执行 agent model/tool 内环。agent 节点必须交给 AgentLoop。历史名称 GraphTaskRuntime 只允许在明确迁移窗口内作为待删除兼容名存在。

## 4. 系统职能定义

### 4.1 任务系统

任务系统回答：用户要求平台执行什么工作？

负责：

```text
接收 ChatInput / UI 任务启动请求
生成 TaskOrder 或 TaskOrderDraft
绑定 TaskEnvironment
绑定 SpecificTask
生成 TaskExecutionEnvelope
编译 TaskGraphRuntimeSpec
维护订单、任务、执行通道的可审计事实
```

不负责：

```text
执行 agent
调用工具
决定 agent 下一步行为
绕过权限
在运行中重选环境
```

任务系统是订单和任务契约系统，不是 agent 运行系统。

### 4.2 任务环境系统

任务环境回答：这项工作发生在什么平台环境里？

负责：

```text
声明文件空间
声明 memory 空间
声明工具市场上限
声明 sandbox、network、shell、browser 等资源供给
声明风险基线
声明 prompt pack 和 policy pack
声明 runtime_shape 上限
```

不负责：

```text
替 agent 规划行动
选择 agent 下一步
被 agent 在运行中切换
表达具体任务步骤
```

任务环境是系统级边界。agent 不能选择、切换或扩大任务环境。

### 4.3 Agent 系统

Agent 系统回答：本次执行要装配成什么 agent？

负责：

```text
AgentDefinition
AgentRuntimeProfile
角色 prompt
skills
model / thinking / budget
agent memory policy
工具偏好或限制
子 agent 类型定义
```

不负责：

```text
拥有独立 runtime
直接执行工具
绕过任务环境
绕过权限
定义图任务调度语义
```

professional、standard、role 都是 AgentRuntimeProfile 或 RuntimePolicy 的装配结果，不是独立 runtime 或独立 loop。

### 4.4 Runtime 装配系统

Runtime 装配系统回答：当前阶段给 agent 看见什么世界？

负责：

```text
消费 TaskEnvironmentSpec、SpecificTaskAssemblyPolicy、AgentRuntimeProfile、LoopState
装配 RuntimeStartPacket
装配模型上下文
装配 ToolCapabilityTable
装配 FileView / MemoryView
装配 OutputContract
装配预算、风险和终止策略
```

不负责：

```text
推进状态
执行模型或工具
替 agent 做语义决策
从旧控制面猜测环境或目标
```

Runtime 装配系统是静态配置系统，不是动态循环系统。

### 4.5 Loop 控制系统

Loop 控制系统回答：状态如何向前推进？

负责：

```text
调用模型
解析模型输出
处理 tool_call / final / delegate / ask_user / plan
调用权限系统
调用工具系统
把 observation 回灌给 agent
更新 LoopState
写 checkpoint
判断继续、暂停、失败、完成或等待审批
```

不负责：

```text
选择任务环境
选择任务目标
绕过 RuntimeStartPacket
绕过权限系统
```

AgentLoop 是 agent 闭环控制器。GraphLoop 是图调度控制器。

### 4.6 工具系统

工具系统回答：平台有哪些能力可以被调用？

负责：

```text
ToolRegistry
ToolDefinition
Tool schema
Tool adapter
Tool executor
ToolResultEnvelope
工具输入校验
工具结果结构化
工具可见描述
```

不负责：

```text
决定 agent 是否拥有权限
选择任务环境
选择 agent
变成 runtime
```

工具是能力，不是 runtime。

### 4.7 权限系统

权限系统回答：这次具体操作能不能做？

负责：

```text
ToolCapabilityTable
PermissionContext
ExecutionPermit
PermissionDecision
ApprovalRequest
PermissionReceipt
ExecutionReceipt
risk classification
deny / ask / allow
```

不负责：

```text
注册工具
选择任务环境
替 agent 行动
把 deny 通过用户审批变成 allow
让子 agent 绕过父 agent 权限
```

ToolCapabilityTable 只是本轮可见和可请求的事实表。每一次具体工具调用仍必须经过 PermissionDecision。

### 4.8 文件管理系统

文件管理系统回答：平台如何托管工作文件、正式作品和产物？

负责：

```text
workspace
正式作品
草稿区
版本
提交
回滚
artifact refs
file receipts
文件读写策略
```

不负责：

```text
选择任务环境
替 agent 决策
普通文件覆盖正式作品
绕过 commit gate
```

写作环境的正式作品、canon、草稿、版本和正式 memory 都必须经过平台文件管理链路。agent 可以请求写入候选内容，但不能直接覆盖正式作品普通文件。

### 4.9 记忆系统

记忆系统回答：agent 和环境如何读取、写入、提交和检索 memory？

负责：

```text
agent memory
environment memory
formal memory
retrieval view
memory commit
memory receipt
memory index
memory projection as read model
```

不负责：

```text
选择任务环境
扩大工具权限
把环境级 memory 写成 agent 私有 memory
用 projection 替代 RuntimeStartPacket
```

memory 可被 runtime 装配为 MemoryView，但 memory 系统本身不拥有 loop 主权。

### 4.10 沙盒系统

沙盒系统回答：执行如何隔离？

负责：

```text
sandbox policy
mount policy
network policy
process policy
execution isolation
sandbox receipt
```

不负责：

```text
选择任务目标
选择 agent
绕过权限
```

沙盒是环境服务和权限护栏，不是 runtime。

### 4.11 观测与状态系统

观测与状态系统回答：执行事实如何被记录、恢复和展示？

负责：

```text
event log
checkpoint
trace
monitor read model
RuntimeStateIndex
result envelope
artifact refs
completion receipt
```

不负责：

```text
反向选择任务环境
反向选择 agent
作为任务配置事实源
作为 runtime 控制输入
```

监控、projection、read model 只能展示事实，不能参与控制。

## 5. 子 Agent、搜索、代码搜索、验证器和图模块

### 5.1 子 Agent

子 agent 不是 subruntime。子 agent 是 AgentFactory 装配出的另一次 AgentInvocation。

正确链路：

```text
parent agent requests delegate_to_agent
-> PermissionDecision
-> AgentFactory resolves AgentDefinition / AgentRuntimeProfile
-> RuntimeStartPacket
-> AgentLoop
-> AgentResultEnvelope
```

禁止链路：

```text
delegate_to_agent
-> ChildAgentRuntimeExecutor
-> if template == search then SearchAgentRuntime
```

### 5.2 搜索与代码搜索

搜索和代码搜索只能是：

```text
工具
能力执行器
agent profile + tool set
skill-backed service
```

它们不能是 runtime。

例如：

```text
Search Agent = AgentDefinition + search tools + research prompt + policy
Codebase Search = read-only tool/capability + codebase search agent profile
```

### 5.3 验证器

验证器是 agent profile 或工具能力，不是 runtime 强制门禁。

主 agent 可以通过 delegate_to_agent 请求 verifier。系统负责授权和装配。verifier 返回结构化结果，但 runtime 不得暗中用 verifier 改写最终答案。

### 5.4 图模块

图模块不是第三类 runtime。图模块是 GraphTaskRuntime 的嵌套图调用。

正确抽象：

```text
GraphModuleInvocation
NestedGraphInvocation
GraphModuleExecutor
GraphResultEnvelope
```

不建议继续使用：

```text
SubRuntime
SubRuntimeInvocationContract
```

除非该名称仅作为短期迁移对象存在，并有明确删除条件。

## 6. 命名规则

### 6.0 Harness 命名

生产控制系统统一使用 Harness 作为顶层工程名：

```text
backend/harness
backend/harness/runtime
backend/harness/loop
AgentHarness
GraphHarness
HarnessRuntimeAssembly
HarnessLoopController
```

测试系统允许使用 test harness 命名，但必须放在测试边界内：

```text
backend/tests/test_runtime_support
backend/tests/support/harness
```

测试 harness 只能负责 fake model、fake tool、scenario builder、assertion helpers，不得提供生产控制策略、生产权限策略或生产 runtime 装配逻辑。

### 6.1 可以使用 Runtime 命名的对象

只有以下对象可以使用 Runtime 命名：

```text
AgentRuntimeSpec
GraphRuntimeSpec
RuntimeStartPacket
RuntimePolicy
RuntimeAssembly
HarnessRuntimeAssembly
```

AgentRuntime facade 和 GraphTaskRuntime facade 不再作为目标命名，只允许在迁移期作为旧入口存在，最终应收敛为 AgentHarness 和 GraphHarness。

### 6.2 不得使用 Runtime 命名的对象

以下对象不得使用 Runtime 命名：

```text
搜索能力
代码搜索能力
验证器
文件执行器
工具执行器
子 agent
图模块
记忆投影
监控视图
兼容桥
```

推荐命名：

```text
SearchCapability
CodebaseSearchCapability
VerifierAgentProfile
GraphModuleExecutor
NestedGraphInvocation
AgentDelegationExecutor
ToolExecutor
FileManagementService
MemoryService
```

## 7. 审查规则

后续重构时，任何模块都必须回答以下问题：

```text
它属于哪个系统？
它消费什么输入？
它产出什么输出？
它是否拥有不该拥有的决策权？
它是否从旧 payload / metadata / projection 中猜测控制事实？
它是否把能力写成 runtime？
它是否绕过 RuntimeStartPacket、ToolCapabilityTable 或 PermissionDecision？
它是否让 finalizer 补做 loop 应该完成的状态推进？
它是否保护真实行为，还是只保护旧内部形状？
```

如果一个模块无法回答这些问题，或答案显示它拥有错误主权，应迁移、合并、改名或删除。

## 8. 与 Claude Code 的对齐原则

Claude Code 的可借鉴结构：

```text
AgentDefinition 是静态 agent 配置。
ToolUseContext 是本轮上下文与工具环境。
query loop 是统一动态循环。
AgentTool 只是委派入口。
子 agent 重新装配上下文后进入同一个 query loop。
Explore / Plan / Verification 是 agent definition，不是独立 runtime。
```

本项目应借鉴的不是具体文件名，而是以下不变量：

```text
能力不长 runtime。
agent 类型不长 runtime。
子 agent 不长 runtime。
工具表来自装配和过滤。
权限按每次调用决策。
动态推进集中在 loop。
```

## 9. 重构方向

后续重构应按以下顺序执行：

```text
1. 固定系统边界和命名规则。
2. 建立 backend/harness 目录，将 runtime 和 loop 收敛为 harness 内部子系统。
3. 收缩旧 backend/runtime 目录，只保留迁移入口；目标入口为 AgentHarness、GraphHarness、RuntimeStartPacket、RuntimePolicy、LoopController。
4. 将 search_agent_runtime、codebase_search_runtime 迁为 capability/tool/agent profile。
5. 将 subruntime 改造成 GraphModuleExecutor / NestedGraphInvocation。
6. 将 TaskRunLoop 降级为服务宿主或拆分为状态、事件、checkpoint、finalization 服务。
7. 将 finalization 中的状态补账逻辑前移到 loop。
8. 删除保护旧 runtime 形状的测试，改为保护 RuntimeStartPacket、LoopState、ToolCapabilityTable、PermissionDecision、ResultEnvelope 行为。
```

## 10. 禁止事项

以下行为后续重构中禁止出现：

```text
以兼容为理由保留旧 runtime 分支
在旧壳上叠新壳
把 task domain / metadata / projection / runtime lane 当控制事实
让 agent 选择任务环境
让具体任务突破任务环境上限
让工具执行绕过权限
让子 agent 绕过父 agent 权限
让 search/codebase/verifier/subagent 独立成 runtime
让 GraphTaskRuntime 执行 agent model/tool loop
让 AgentLoop 调度完整任务图
用 prompt 描述代替结构化权限和环境配置
```

## 11. 最终定义

本项目最终 agent 系统框架可以压缩为一句话：

```text
Harness 是 agent 的管控系统；Harness Runtime 为 agent 装配阶段性可见世界；Harness Loop 推动 agent 在这个世界中行动、观察、修正和收口；任务环境和权限系统提供边界；工具、文件、记忆、沙盒提供能力；观测系统记录事实但不反向控制。
```

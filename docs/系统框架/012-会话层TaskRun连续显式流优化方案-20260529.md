# 会话层 TaskRun 连续显式流优化方案

日期：2026-05-29

## 1. 问题定义

当前单 agent harness 已能启动 TaskRun、执行工具、记录 trace、生成 artifact，并在完成后把最终答复回写到会话历史。但会话页面仍存在体验断层：

- 普通 turn 的 SSE 事件会挂在当前 assistant message 的 `runtimeProgress` 上。
- TaskRun 启动后，聊天 stream 很快以“执行器接管”收口。
- 后台 TaskRun 的后续事件进入 `RuntimeEventLog` / live monitor，但不会继续挂回会话消息。
- TaskRun 完成后 final answer 会作为普通 assistant message 回写，但中间过程只在监控/trace 里可见。

这导致用户看到的是“普通对话 -> 后台任务 -> 最终结果”的分裂体验，而不是成熟 agent 常见的连续任务活动流。

## 2. 目标

会话页必须把 TaskRun 看作当前 turn 的运行附件，而不是独立跳走的后台对象。

目标形态：

```text
user message
assistant handoff message
  runtime attachment:
    - TaskRun 状态
    - step summaries
    - 工具/观察摘要
    - artifact refs
    - 完成/失败/等待状态
assistant final answer
```

运行过程不应全部写成聊天文本；聊天历史保留用户消息、handoff 消息、最终答复。过程由会话 timeline projection 读取 TaskRun trace/monitor 后展示。

## 3. 当前代码事实

后端：

- `backend/api/sessions.py` 的 `/sessions/{session_id}/history` 只返回 `SessionManager.get_history()`。
- `backend/harness/runtime/single_agent_host.py` 已提供：
  - `list_session_traces(session_id)`
  - `get_session_live_monitor(session_id)`
  - `get_trace(task_run_id)`
- `backend/harness/runtime/monitor_projection.py` 能把 TaskRun 投影成 live monitor item，但 session monitor 只展示 `running/diagnostics`。
- TaskRun final answer 已由 `backend/harness/loop/task_executor.py` 通过 commit gate 回写 session history。

前端：

- `frontend/src/lib/store/utils.ts` 的 `toUiMessages()` 只处理原始 messages。
- `frontend/src/lib/store/events.ts` 的 `reduceStreamEvent()` 只把 SSE 期间的 runtime progress 挂到当前 assistant message。
- `frontend/src/lib/store/runtime.ts` 的 `hydrateLatestOrchestrationSnapshot()` 只更新 session activity 和 monitor，不会补写 message attachment。
- `frontend/src/components/chat/ChatMessage.tsx` 已有 `RuntimeRunSummary` 展示入口，但缺少持久 TaskRun attachment 数据源。

## 4. 设计原则

1. TaskRun trace 是事实源。
2. Timeline 是展示投影，不拥有执行决策。
3. Monitor 是实时状态，不是历史记录。
4. 会话消息不承载所有过程日志，只承载用户可读收口文本。
5. TaskRun 与会话 turn 的绑定必须来自结构化 id/ref，不靠关键词。
6. 不新增任务识别、意图分类或兼容旧链路。

## 5. 后端目标结构

新增会话 timeline API：

```http
GET /api/sessions/{session_id}/timeline
```

返回：

```json
{
  "authority": "session_runtime_timeline",
  "session_id": "...",
  "messages": [...],
  "runtime_attachments": [
    {
      "attachment_id": "runtime-attachment:{task_run_id}",
      "anchor_turn_id": "turn:{session_id}:{turn_index}",
      "task_run_id": "...",
      "status": "running|completed|failed|...",
      "lifecycle": "...",
      "title": "...",
      "latest_step_summary": "...",
      "progress_entries": [...],
      "artifact_refs": [...],
      "final_answer": "...",
      "trace_available": true
    }
  ]
}
```

绑定规则：

- `turnrun:turn:{session}:{index}` 只代表普通 turn runtime。
- `taskrun:turn:{session}:{index}:*` 绑定到同一 turn 的 assistant handoff message。
- 若 TaskRun diagnostics 里有 `turn_id`，优先使用 diagnostics.turn_id。
- 不从文本中猜测 task id 或任务意图。

## 6. 前端目标结构

`Message` 增加：

```ts
runtimeAttachments?: RuntimeAttachment[];
```

加载会话时：

- 优先读取 `/sessions/{id}/timeline`。
- 用原始 messages 生成 UI messages。
- 按 `anchor_turn_id` / message source index 把 attachments 挂到对应 assistant message。
- 若 attachment 已完成，仍保留在会话页。

stream 期间：

- 继续使用 SSE 更新当前 assistant message 的 `runtimeProgress`。

TaskRun handoff 后：

- monitor 轮询更新对应 message 的 attachment。
- 不只更新底部 `SessionActivityBar`。

## 7. 实施计划

### Phase 1：后端 timeline projection

文件：

- `backend/harness/runtime/session_timeline.py`
- `backend/api/sessions.py`
- `backend/tests/query_runtime_runtime_loop_regression.py` 或新增 focused test

完成标准：

- `/sessions/{id}/timeline` 返回原始 messages 和 runtime attachments。
- completed TaskRun 也出现在 attachments 中。
- attachment 的 progress entries 来自 `step_summary_recorded`、tool/observation、terminal events。

### Phase 2：前端类型与加载

文件：

- `frontend/src/lib/api.ts`
- `frontend/src/lib/store/types.ts`
- `frontend/src/lib/store/utils.ts`
- `frontend/src/lib/store/runtime.ts`

完成标准：

- session detail load 使用 timeline。
- UI message 挂载 runtime attachments。
- 连续 assistant message 不会错误吞掉 runtime attachment。

### Phase 3：前端显示

文件：

- `frontend/src/components/chat/ChatMessage.tsx`
- `frontend/src/components/chat/RuntimeRunSummary.tsx`

完成标准：

- 会话页可看到 handoff message 下方的 TaskRun 活动块。
- completed 后活动块仍保留。
- artifact refs 可见。

### Phase 4：实测

验证：

- 普通对话不出现 TaskRun attachment。
- 启动 TaskRun 后，会话页显示运行块。
- TaskRun 后台完成后，会话历史刷新仍保留运行块和最终答复。
- CLI/后端 trace 与前端展示同源。

## 8. 禁止事项

- 禁止把 TaskRun 每一步写入 session messages。
- 禁止用关键词把“执行器接管”等文本识别为任务锚点。
- 禁止让前端自己猜测任务生命周期。
- 禁止新建第二套 monitor 事实源。
- 禁止改动 agent 的任务启动判断语义。

# 系统-Agent 交互边界链路治理方案

## 目标

本方案用于约束主聊天、active work、TaskRun、工具、todo、子 Agent、运行监控和前端流式交互之间的边界。

核心原则：

1. Agent 负责语义判断、任务规划、动作选择和结果综合。
2. 系统负责事实收集、环境装配、权限准入、执行边界、生命周期状态、观察记录和前端投影。
3. 系统可以拒绝不满足前置条件的显式控制包，但不能把普通用户输入偷偷改写成控制动作。
4. 系统不能用一个边界问题去堵另一个边界问题。例如不能为了避免串线而把所有 active turn 输入都硬塞进 steer，也不能为了降低延迟绕过 agent 的关系判断。

## 目标链路

```text
用户输入 / 前端显式控制
-> RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> RuntimeStartPacket
-> ModelTurnDecision
-> ActionAdmission
-> ActionPermit
-> ExecutionLoop / ToolControlPlane
-> RuntimeObservation
-> Agent Follow-up
-> CanonicalOutput
-> RuntimeMonitor / Frontend Projection
```

各层权威：

| 层 | 允许做的事 | 禁止做的事 |
| --- | --- | --- |
| RequestFacts | 记录用户消息、active turn、环境绑定、editor context、模型选择 | 判断用户意图、选择任务、授权工具 |
| BoundaryPolicy | 验证显式控制包前置条件，拒绝 stale / mismatch / paused / stopped | 把普通输入改成控制动作 |
| ContextCandidates | 提供 active work、recent outcome、memory、todo、工具观察候选 | 让旧上下文覆盖用户最新消息 |
| RuntimeStartPacket | 装配 prompt、工具、环境、权限和 schema | 偷偷扩大工具权限或注入无关任务 |
| ModelTurnDecision | 由 agent 输出 respond / ask_user / tool_call / request_task_run / active_work_control / block | 系统替 agent 选择语义动作 |
| ActionAdmission | 校验 action 是否被本轮 packet、权限、工具和生命周期允许 | 把被拒动作改成另一个动作执行 |
| ActionPermit | 将 admission 结果绑定到具体工具调用 | 重新解释 action 或跳过 tool control plane |
| ExecutionLoop | 执行获准动作，记录观察，处理暂停/停止/重试边界 | 在失败时伪造成功，或绕过 agent 直接完成 |
| RuntimeObservation | 把工具结果、拒绝、pending steer、子 Agent 状态作为观察回灌 | 把观察当最终答案 |
| Projection | 展示状态、进度和结果 | 用旧 monitor / stale event 覆盖新状态 |

## 当前代码链路

### 主聊天入口

- `backend/api/chat.py`
  - 接收 `message`、`environment_binding`、`model_selection`、`permission_mode`、`expected_active_turn_id`、`active_turn_input_policy`。
  - 不再接收或转发 `task_selection`。
  - `latest-run?active_only=true` 跳过显式短 steer run，避免重连到补充控制 ACK。

- `backend/harness/entrypoint/models.py`
  - `HarnessRuntimeRequest` 是请求事实容器。
  - 环境绑定由用户/前端提供，不由 agent 选择。

- `backend/harness/runtime/request_facts.py`
  - `TurnInputFacts` 明确声明只记录事实，不分类 intent，不选择工具，不授权，不决定 active work 控制。

### 显式 active turn 控制

- `backend/harness/entrypoint/runtime_facade.py::_active_turn_steer_fast_path`
  - 只处理 `active_turn_input_policy == "steer"`。
  - 成功时仅追加用户补充要求到正在运行的 TaskRun。
  - 前置条件失败时返回 blocked terminal，不落入普通模型回合。
  - `active_turn_input_policy == "auto"` 必须进入模型回合，由 agent 通过 `active_work_control` 判断。

这条规则是关键修正：`auto` 是“把 active turn 作为事实交给 agent”，不是“系统默认当成 steer”。

### Agent 决策与准入

- `backend/harness/loop/single_agent_turn.py`
  - 编译 single-agent turn packet 后由模型输出 action。
  - tool loop 和 final action 都经过 `admit_model_action`。
  - admission 失败可以触发协议修复或返回 blocked，不得偷偷执行替代动作。

- `backend/harness/loop/model_action_protocol.py`
  - 只接受合法 action schema。
  - task execution 禁止 `task_contract_seed`、`active_work_control` 等跨上下文字段。

- `backend/harness/loop/admission.py`
  - 校验 action 是否在 packet allowed actions 内。
  - 校验工具是否在 runtime assembly 内。
  - plan mode / side effect / task lifecycle policy 在此拒绝。

- `backend/harness/loop/action_permit.py`
  - 将 admission 结果绑定到具体 action/tool/operation。
  - ToolControlPlane 必须验证 permit，不能只凭模型 tool name 执行。

### TaskRun 生命周期

- `backend/harness/loop/task_lifecycle.py`
  - 只有 agent 明确 `request_task_run` 或系统显式合同可启动 TaskRun。
  - 合同缺目标或验收边界时 fail closed。
  - executor schedule 失败时返回结构化失败，不伪装成已接管。

- `backend/harness/entrypoint/runtime_facade.py::_prepare_current_task_for_new_task_request`
  - 当前任务存在时，新任务必须走替换/接管边界。
  - 旧任务停止失败时不得启动新任务。

### Task executor / tool loop

- `backend/harness/loop/task_executor.py`
  - 每步模型 action 均经过 protocol、admission、permit、tool control plane。
  - admission 被拒绝时写入观察，让 agent 下一轮修正。
  - pending steer、active contract revision、active subagent、completion verification 不满足时，写入 repair observation，而不是系统直接完成或直接改答案。
  - pause / stop / replan 是 executor 控制信号，工具边界必须保留原始信号，不得统一吞成 stop。

### Todo

- `backend/capability_system/tools/tool_units/agent_todo_tool.py`
  - todo 是执行状态工具。
  - 最多一个 `in_progress`。
  - 不能作为事实来源、任务合同或完成依据。

- `backend/harness/entrypoint/runtime_facade.py::_initialize_task_todo_for_contract`
  - 系统可从合同初始化 todo，但只能作为可见执行清单。
  - 初始化 todo 不得反向改变合同、目标、验收标准或用户消息。

### 子 Agent

- `backend/harness/agent_control/controller.py`
  - `spawn_subagent` 创建真实 child `subagent_task`。
  - `wait_subagent` 只返回 child 状态、消息和 result payload。
  - 父 agent 必须 wait/list 后综合证据，不能把 child 结果自动当最终答案。
  - 父任务完成前若还有 active child subagent，executor 写入 repair observation，要求 wait/list/close 后再收口。

### 前端流式交接

- `frontend/src/lib/store/runtime.ts`
  - 显式 steer 短通道发送 `active_turn_input_policy: "steer"`，且 `persistCursor: false`，避免 latest-run 重连到短 ACK。
  - 普通 active-work follow-up 发送 `active_turn_input_policy: "auto"`，由后端 agent 判断。
  - auto active-work 模型判断回合期间，当前 session 必须进入 active stream 串行保护；后续用户输入进入本地队列，不能并发开第二条 chat stream，也不能绕过 agent 直接走 steer。

- `frontend/src/lib/run-monitor/controller.ts`
  - stale monitor snapshot 被拒绝时，不能继续应用同 payload 的 runtime event，避免半旧半新状态。

## 已修复边界

1. 显式 steer 失败不再落入普通模型回合。
2. 无 context 的显式 steer terminal 不再误投影本轮 transient active turn。
3. `active_turn_input_policy == "auto"` 不再走 steer fast path。
4. auto active-work 判断回合被纳入 active stream 串行保护。
5. active work turn decision 不再把 `intent` 当 `action` 兜底。
6. task executor 工具边界保留 pause / stop / replan 控制信号。
7. active task 控制 API 校验 `expected_active_turn_id`。
8. monitor action 拒绝 stale `source_revision`。
9. latest active run 不重连显式 steer 短 run。
10. prompt rule 命名从 `intent_feedback` 转为 `turn_decision_alignment`，避免暗示存在系统意图识别层。

## 仍需治理的结构债

### 1. `selected_task_id` 命名残留

位置：

- `backend/harness/loop/task_lifecycle.py`

现状：

- `runtime_contract.get("selected_task_id")` 仍作为 `task_id` 候选。

判断：

- 它目前没有进入主聊天 `task_selection`，不构成慢回复入口。
- 但命名仍会误导后续开发者，以为存在一个系统 task selection 层。

建议：

- 改为 `task_id` / `graph_task_id` / `contract_task_id`。
- 删除 prompt 和文档中把 `selected_task_id` 解释为用户意图选择的表述。

### 2. explicit image route 需要保持显式

位置：

- `backend/api/chat_direct_routes.py`

现状：

- 只有 `image_generation` payload 明确为 image model / generate mode 时才绕过 agent。

治理要求：

- 该 route 只能处理 UI 明确给出的生成图像控制包。
- 不允许根据用户自然语言“画一张图”自行走 direct route；自然语言应进入 agent 决策，由 agent 请求对应工具或任务。

### 3. auto active-work 是否需要更细关系输出

现状：

- agent 可通过 `active_work_control` 决定当前输入与 active work 的关系。

后续增强：

- 对 `active_work_control` 的模型输出继续要求 `relation_to_current_work`、`evidence`、`answer_obligation`。
- 系统只校验一致性，不替 agent 分类。

## 禁止模式

1. 禁止恢复 `task_selection` 作为主聊天字段。
2. 禁止用 `intent`、`task_selection`、旧 todo、旧 memory、旧 monitor 来决定本轮 action。
3. 禁止 `auto` 输入直接进入 steer fast path。
4. 禁止显式控制包失败后落入普通模型回合。
5. 禁止 admission deny 后换等价工具或等价 action 执行。
6. 禁止子 Agent 未 wait 就引用其结论。
7. 禁止 todo completed 作为任务完成依据。
8. 禁止 monitor stale snapshot 被拒后继续应用同 payload 的 event。
9. 禁止为了避免并发而把所有 active work 输入都前端硬排成 steer。
10. 禁止为了追求速度绕过 agent 的 active work 关系判断。

## 最终判定

这条链路的成熟标准不是“尽量少出错所以系统多挡一点”，而是：

- 普通输入必须进入 agent 判断。
- 显式控制必须由系统严格校验。
- 工具和任务必须由 admission / permit / tool control plane 执行。
- 执行失败必须形成观察或 blocked terminal。
- 前端必须串行化流，但不能把串行化等同于语义改写。

只有这样，系统才是在提供环境和边缘控制，而不是抢 agent 的思考和调度权。

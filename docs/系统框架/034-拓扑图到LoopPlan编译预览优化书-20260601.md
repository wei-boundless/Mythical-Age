# 拓扑图到 LoopPlan 编译预览优化书

## 1. 问题定义

任务图编辑器已经能够表达节点、边、资源、契约、记忆读写、返修和循环，但用户在发布前仍然看不清“这张拓扑图最终会怎样被 GraphLoop 执行”。这会带来两个问题：

- 用户只能看到图形结构，看不到调度起点、终点、依赖边、上下文边、返修边和循环 frame 的运行含义。
- 前端容易为了说明运行结果而重新推导后端 runtime 逻辑，导致权责混乱。

正确目标不是让前端把拓扑图模拟成 loop，而是由后端编译一个只读的 LoopPlan 预览，让用户在发布前看到后端真实会采用的调度计划。

## 2. 当前代码链路

现有主链路已经具备成熟基础：

```text
TaskGraphDefinition
-> normalize_task_graph_layers()
-> build_graph_harness_config_from_graph()
-> build_scheduler_view()
-> GraphRuntimeEnvelope
-> GraphLoop.initialize()
-> GraphLoopState / GraphNodeWorkOrder
```

关键文件：

- `backend/task_system/compiler/layered_graph_normalizer.py`
- `backend/task_system/compiler/graph_harness_config_publisher.py`
- `backend/harness/graph/scheduler_view.py`
- `backend/harness/graph/state_machine.py`
- `backend/harness/graph/loop.py`
- `backend/task_system/graphs/task_graph_standard_models.py`
- `frontend/src/components/workspace/views/task-system/TaskGraphPublishRunPage.tsx`

## 3. 目标设计

新增一个后端权威的 `loop_plan` 编译产物，挂到 `TaskGraphStandardView.diagnostics.loop_plan`，供发布页展示。

```text
Topology
  nodes / edges / resources / loop_frames / contracts
        |
        v
GraphHarnessConfig
        |
        v
SchedulerView + layered graph
        |
        v
LoopPlan preview
        |
        v
Frontend publish page display only
```

LoopPlan 是预览契约，不是运行状态。它只能描述编译结果，不生成 work order，不写 checkpoint，不替代 GraphLoop。

## 4. LoopPlan 字段

第一版 LoopPlan 保持小而准：

- `available`: 是否成功编译。
- `authority`: `task_system.loop_plan_preview`。
- `start_node_ids`: 后端调度起点。
- `terminal_node_ids`: 后端终点。
- `executable_node_ids`: 会被 GraphLoop 执行的节点，资源节点不在其中。
- `dependency_edges`: 真正参与 ready 判断的边。
- `context_edges`: 只提供上下文的边，例如 memory/artifact/file read。
- `commit_edges`: 写入或提交类边。
- `revision_edges`: 返修/审核反馈边。
- `loop_frames`: entry/router/continue/exit 编译结果。
- `initial_ready_node_ids`: GraphLoop 初始化时第一批 ready 节点。
- `summary`: 节点、边、frame、问题数量。
- `issues`: 编译或结构问题。

## 5. 权责边界

后端负责：

- 根据 GraphHarnessConfig 和 SchedulerView 生成 LoopPlan。
- 使用同一套 scheduler/state_machine 规则判断可执行节点和 initial ready。
- 标记 memory/artifact/revision/loop frame 的运行含义。

前端负责：

- 在发布运行页展示 LoopPlan。
- 明确显示“这是后端编译预览”。
- 不推导 ready 节点，不判断边是否参与调度，不模拟循环。

禁止：

- 禁止在前端根据 edge_type 重新实现 scheduler。
- 禁止把 LoopPlan 当成真实运行状态。
- 禁止保留一套旧的运行推导 UI。

## 6. 实施计划

阶段 1：后端 LoopPlan 编译器

- 新增 `backend/task_system/compiler/loop_plan_preview.py`。
- 输入 `GraphHarnessConfig` 和可选 `layered_graph`。
- 复用 `build_scheduler_view()` 与 `GraphStateMachine()`。
- 输出稳定 dict，失败时输出 fail-closed 的 unavailable 结构。

阶段 2：标准视图接入

- 修改 `backend/task_system/graphs/task_graph_standard_models.py`。
- 在 graph_config 成功时生成 `diagnostics.loop_plan`。
- 在 graph_config 失败时生成 unavailable loop_plan，并带上 issue。

阶段 3：前端类型和展示

- 修改 `frontend/src/lib/api.ts`，补充 `TaskGraphLoopPlanPreview` 类型。
- 修改 `TaskGraphPublishRunPage.tsx`，在“运行准备”附近展示“拓扑编译计划”。
- 展示 summary、start/terminal/ready、dependency/context/revision/loop_frames。

阶段 4：测试

- 后端测试：标准视图必须包含 LoopPlan，且资源节点不进入 executable，memory read 不进入 dependency。
- 前端测试：构建 LoopPlan summary 的纯函数测试，避免 UI 里散落推导。

## 7. 完成标准

- 发布页能解释“这张图会从哪里开始、哪些边影响调度、哪些边只是上下文、循环入口和出口是什么”。
- 前端没有新增 scheduler 推导逻辑。
- 后端 LoopPlan 与 GraphLoop 初始化规则使用同一个 state machine/scheduler 规则。
- 现有发布、运行、监控功能不受影响。

# 环境生命周期与工具可见性重构计划

状态：核心重构已落地，prompt 模板选择已显式化  
日期：2026-06-09  
适用范围：runtime prompt 装配、任务环境 prompt 挂载、lifecycle prompt 选择、工具可见性与 tool catalog 投影

## 1. 背景

本次排查来自一次真实 TaskRun prompt cache 监测。该运行的 `model_messages` 约 52,497 字符，其中：

- `environment_stable` 约 17.4k 字符。
- `tool_index_stable` 约 18.9k 字符。
- 任务合同本身约 1.7k 字符。

问题不是单纯 token 数偏大，而是 prompt 装配语义混杂：

- 专业任务环境被通用环境 base prompt 隐式叠加。
- lifecycle prompt 只追加通用版本，没有按环境覆盖槽位。
- tool guidance prompt 只按通用默认生成，缺少环境按工具/工具组覆盖的槽位解析。
- `full_access` 等权限模式会扩大工具候选，导致 agent 未配置或 hidden 工具进入模型可见面。
- tool catalog 对 `prompt_exposure_policy=hidden` 缺少最后一道防线。
- 多个 prompt 层仍由 compiler 或局部 controller 兜底挂载，缺少统一编排计划作为唯一装配权威。

这些问题会造成 prompt 过重、语义污染和工具主权边界不清。

## 2. 当前实现偏差

### 2.1 环境 prompt 被加法合并

当前 `build_base_prompt_mount_plan` 采用：

```text
general base prompts + selected environment overlay prompts
```

非通用环境会默认带上 `environment.general.workspace.*`，再叠加当前环境自己的 prompt。这个实现不符合目标设计。

目标设计应为：

```text
selected task environment prompts only
```

通用环境只是一个普通环境，不是所有环境的隐式父类。

### 2.2 lifecycle 没有槽位覆盖

当前实现里 `_LIFECYCLE` 是全局固定映射。环境 prompt refs 里的 lifecycle refs 会被 `_without_lifecycle_refs` 从静态环境 prompt 中移除，然后 selector 再追加通用 lifecycle refs。

这导致专业环境无法覆盖某个 lifecycle 槽位，只能与通用 lifecycle 叠加或被忽略。

目标设计应为：

```text
lifecycle selector selects lifecycle keys
lifecycle resolver maps key -> selected environment override or general default
```

同一个 lifecycle key 最终只能出现一个 prompt ref。

### 2.3 工具可见性没有严格保持交集

工具可见性目标是双重筛选：

```text
agent internal allowed operations
∩ task environment allowed operations
∩ runtime contract explicit ceiling, if any
```

权限模式只决定执行许可、审批和风险边界，不应扩大模型可见工具集合。

当前 `full_access` / `bypass` 会把 `known_operations | task_requested` 纳入允许候选，导致 agent profile 没有配置的工具也可能进入可见面。当前 assembly 还会在 full access 下 `include_hidden`，导致 `python_repl` 这类 hidden 工具进入 prompt catalog。

目标设计应为：

```text
permission_mode cannot grant model-visible tool capability
hidden/debug tools are never model-visible unless a dedicated internal runtime path explicitly consumes them outside prompt catalog
```

### 2.4 tool guidance 缺少环境覆盖槽位

当前 tool guidance 由 `tool_guidance_payload_for_visible_tools` 根据工具名映射到通用 guidance prompt。这个方向的一半是正确的：只应从可见工具集合生成 guidance。

缺失的是覆盖语义：

```text
tool guidance defaults are general
selected environment can override a tool guidance key
same key must not stack general and environment-specific guidance
```

例如 coding 环境可以覆盖 `tool.guidance.terminal_powershell`，把固定端口、服务启动、浏览器验证等 coding 专属约束放在 coding 版本中；office 环境可以覆盖 `tool.guidance.read_file` / `tool.guidance.web_fetch`，把文件服务和搜索服务的边界写得更轻。

### 2.5 prompt 层仍存在隐式默认层

当前系统仍有多处“没有显式配置就自动补默认 prompt”的路径，例如 runtime pack 默认选择、通用环境 base、固定 lifecycle 映射、通用 tool guidance 映射等。

目标设计不应存在 runtime 隐式默认层。

正确语义是：

```text
Prompt orchestration plan is the only authority for prompt layers.
Templates are selectable plans, not hidden defaults.
Compiler renders the selected plan; compiler does not invent missing layers.
```

系统可以提供模板，例如 general chat、office file search、coding workspace，但模板必须由 agent profile、session、task environment、task contract 或 UI/session 配置显式选中。模板一旦选中，里面的通用 lifecycle/tool guidance 才是该模板的显式内容，而不是 runtime 隐式默认。

## 3. 目标权威链

目标权威链如下：

```text
PromptOrchestrationPlan
-> TaskEnvironmentSelection
-> EnvironmentPromptResolver
-> LifecycleKeySelector
-> LifecycleOverrideResolver
-> OperationAuthorizationProjection
-> RuntimeToolPlan
-> ToolGuidanceResolver
-> ToolCatalogManifest
-> PromptCompositionManifest
```

职责划分：

| 层级 | 当前职责 | 禁止行为 |
| --- | --- | --- |
| `PromptOrchestrationPlan` | 显式声明本 agent/任务要装配哪些 prompt 层、模板、覆盖和禁用项 | 让 compiler 或 controller 自行补默认 prompt 层 |
| `TaskEnvironmentSelection` | 选择一个任务环境 | 隐式继承通用环境 prompt |
| `EnvironmentPromptResolver` | 输出选中环境自己的环境 prompt refs 和资源 prompt refs | 合并其它环境身份 |
| `LifecycleKeySelector` | 根据 invocation、观察、工具、记忆等选择 lifecycle keys | 直接绑定具体 prompt ref |
| `LifecycleOverrideResolver` | 用环境覆盖替换通用 lifecycle 默认实现 | 同 key 叠加多个 prompt |
| `OperationAuthorizationProjection` | 计算 agent 与环境的操作交集 | 用 full_access 扩大 agent 能力 |
| `RuntimeToolPlan` | 输出模型可见且可派发的工具集合 | 暴露 hidden/debug 工具 |
| `ToolGuidanceResolver` | 按可见工具解析通用默认 guidance 或环境覆盖 guidance | 为不可见工具挂 guidance，或同 key 叠加多个 guidance |
| `ToolCatalogManifest` | 渲染工具摘要和 guidance | 把 control-plane 或 hidden 工具投给模型 |
| `PromptCompositionManifest` | 记录最终装配、slot、cache/lifecycle 边界 | 重新决定环境或工具权限 |

## 4. 目标装配规则

### 4.0 prompt 编排总规则

所有 prompt 层必须由编排系统显式配置。runtime 可以提供模板，但模板不是隐藏默认层。

编排计划至少应能描述：

```json
{
  "template_id": "prompt_template.office.file_search",
  "enabled_layers": [
    "system_foundation",
    "runtime_protocol",
    "agent_role",
    "environment",
    "lifecycle",
    "tool_guidance",
    "task_contract",
    "dynamic_context"
  ],
  "prompt_refs_by_layer": {
    "system_foundation": [],
    "runtime_protocol": [],
    "agent_role": [],
    "environment": [],
    "lifecycle": [],
    "tool_guidance": []
  },
  "layer_overrides": {},
  "disabled_layers": []
}
```

约束：

- compiler 只能渲染编排计划给出的层。
- controller 只能解析编排计划授权的层。
- 缺失必要层时 fail closed 或返回配置错误，不自动补通用默认层。
- 通用模板可以存在，但必须显式选择，例如 `prompt_template.general.chat`。
- 环境、lifecycle、tool guidance、personality、agent role、runtime protocol 都是可编排层，不是硬编码默认层。
- prompt cache 只能标注已选中层的稳定性，不能反向决定是否挂载某层。

### 4.1 环境 prompt 规则

选中环境后，只挂载该环境自己的 prompt refs：

```text
environment_prompt_refs = environment_resource_prompt_refs(selected_spec)
                        + selected_spec.environment_prompts excluding lifecycle prompts
```

说明：

- `environment_resource_prompt_refs` 属于当前环境资源配置派生，不是通用环境继承。
- `env.general.workspace` 只在用户或系统明确选择通用环境时生效。
- 旧诊断字段 `overlay_mode=general_base_plus_selected_overlay` 应删除或替换为 `environment_prompt_strategy=selected_environment_only`。

### 4.2 lifecycle 覆盖规则

生命周期 prompt 按 key 管理：

```text
selected_lifecycle_keys = selector(invocation, allowed_actions, observations, tools, memory, execution_state)
resolved_ref[key] = selected_environment.lifecycle_policy.prompt_overrides[key]
                 ?? selected_prompt_template.lifecycle_defaults[key]
```

约束：

- 每个 key 最多解析出一个 prompt ref。
- 专业环境覆盖同 key 时，通用版本不进入模型。
- 环境没有覆盖时，使用当前编排模板声明的 lifecycle 默认实现。
- 如果当前编排模板没有声明该 key 的默认实现，不能自动补通用 lifecycle；应记录配置缺口并 fail closed 或跳过该 key，取决于该 key 是否被标记为 required。
- lifecycle refs 不进入 `environment_prompt_refs`，单独记录在 `lifecycle_prompt_refs`。

建议 `PromptMountPlan` 增加或保留以下字段：

```json
{
  "environment_prompt_refs": [],
  "lifecycle_prompt_keys": [],
  "lifecycle_prompt_refs": [],
  "lifecycle_prompt_overrides": {},
  "lifecycle_resolution": [
    {
      "key": "tool_dispatch",
      "prompt_ref": "environment.general.lifecycle.tool_dispatch",
      "source": "general_default",
      "trigger_reason": "tool_call action is allowed"
    }
  ]
}
```

### 4.3 工具可见性规则

模型可见工具必须满足：

```text
operation in agent_profile.allowed_operations
operation not in agent_profile.blocked_operations
operation in environment_allowed_operations(selected_environment)
operation in explicit operation_ceiling, if a ceiling exists
tool prompt exposure policy in {schema_only, schema_plus_guidance}
```

`full_access` / `bypass` 不改变工具集合，只改变审批和执行风险处理。

`prompt_exposure_policy` 规则：

| policy | model catalog | guidance | dispatch |
| --- | --- | --- | --- |
| `schema_plus_guidance` | yes | yes | yes, if authorized |
| `schema_only` | yes | no | yes, if authorized |
| `hidden` | no | no | no model-visible dispatch |
| `debug_only` | no | no | no model-visible dispatch |

如果后续确实需要内部 runtime 使用 hidden 工具，应走独立 control-plane path，不能进入 `available_tools` 或 `ToolCatalogManifest.model_visible_catalog`。

### 4.4 tool guidance 覆盖规则

tool guidance 与 lifecycle 一样按槽位解析：

```text
visible_tool_names = RuntimeToolPlan.model_visible_tools.tool_name
required_guidance_keys = guidance keys derived from visible_tool_names
resolved_guidance[key] = selected_environment.tool_guidance_policy.prompt_overrides[key]
                      ?? selected_prompt_template.tool_guidance_defaults[key]
```

约束：

- 只为最终模型可见工具生成 guidance。
- 同一 guidance key 最多解析一个 prompt。
- 专业环境覆盖某个 guidance key 时，通用版本不进入模型。
- 环境没有覆盖时，使用当前编排模板声明的 tool guidance 默认实现。
- 如果当前编排模板没有声明某个 guidance key，不能自动补通用 guidance。
- 不可见工具、hidden 工具、debug 工具不得生成 guidance。
- tool guidance 不应重复 lifecycle 或 environment prompt 的职责。

建议环境配置支持：

```json
{
  "tool_guidance_policy": {
    "prompt_overrides": {
      "tool.guidance.terminal_powershell": "tool.guidance.coding.terminal_powershell",
      "tool.guidance.read_file": "tool.guidance.office.read_file"
    }
  }
}
```

若暂不新增 `TaskEnvironmentSpec.tool_guidance_policy` 字段，可先把该配置放在 `spec.metadata.tool_guidance_policy`，但 resolver 的输出必须是同一结构，避免调用点读取 metadata 猜语义。

## 5. 具体代码改动计划

### 5.0 prompt 编排系统入口

涉及文件：

- `backend/prompt_library/packs.py`
- `backend/prompt_library/assembly.py`
- `backend/harness/runtime/assembly.py`
- `backend/harness/runtime/compiler.py`
- 新增或扩展 prompt orchestration models。

改动：

- 引入 `PromptOrchestrationPlan` 或等价结构，作为所有 prompt 层的唯一输入。
- 把当前 runtime pack、foundation、agent role、environment、lifecycle、tool guidance 等层纳入该 plan。
- `default_pack_ref_for_invocation` 只能作为模板解析的一部分，不能在 compiler 内部兜底隐式调用。
- `PromptAssemblyService` 只根据 plan 装配，不自行决定默认层。
- 在 manifest 中记录每个 prompt ref 来自哪个 template、layer、override 或 task contract。

模板建议：

- `prompt_template.general.chat`
- `prompt_template.office.file_search`
- `prompt_template.coding.workspace`
- `prompt_template.semantic_compaction`

这些模板可作为产品默认选项，但必须由 profile/session/environment/task contract 显式选择。

### 5.1 `backend/harness/runtime/environment_prompt_controller.py`

- `overlay_mode=general_base_plus_selected_overlay`

### 5.2 `backend/task_system/environments/spec_resolver.py`

  - `prompt_kind="lifecycle_tool_dispatch"`

### 5.5 `backend/harness/runtime/operation_projection.py`

目标语义：

### 5.6 `backend/harness/runtime/assembly.py`

### 5.7 `backend/harness/runtime/tool_catalog_manifest.py`

## 6. 验收标准

### 6.1 环境 prompt

选择 `env.coding.vibe_workspace` 时，模型输入不得因为通用层叠加而自动包含：

```text
environment.general.workspace.orientation
environment.rule.general_workspace
```

除非该环境显式把它们写入自己的 refs。

### 6.2 prompt 编排

每个进入 `model_messages` 的 prompt ref 都必须能追溯到：

- 显式 prompt template。
- agent profile/session/task contract 显式配置。
- selected environment 显式配置。
- runtime dynamic context 或 task contract。

不得出现 compiler/controller 自行补充的未编排默认层。

### 6.3 lifecycle

当环境覆盖 `tool_dispatch` 时，模型输入只出现专业环境版本，不出现通用版本。

当环境未覆盖 `tool_dispatch` 时，模型输入使用当前编排模板声明的默认版本。

### 6.4 工具可见性

在 `permission_mode=full_access` 下：

- agent 未配置的工具不得进入 `assembly.available_tools`。
- agent blocked 的工具不得进入 `assembly.available_tools`。
- `python_repl` 如果是 hidden，不得进入 `ToolCatalogManifest.model_visible_catalog`。

### 6.5 tool guidance

当环境覆盖 `tool.guidance.read_file` 时：

- 最终只出现环境覆盖版本。
- 通用 `tool.guidance.read_file` 不进入模型。
- 若 `read_file` 不在最终可见工具中，两个版本都不进入模型。

当环境未覆盖 `tool.guidance.read_file` 时，使用当前编排模板声明的默认 guidance；模板未声明则不自动补。

### 6.6 prompt 重量

同等任务下：

- environment segment 不再包含无关通用环境 base。
- office 环境只挂文件/搜索相关 prompt 和必要 lifecycle。
- tool catalog 不包含 hidden/debug 工具。
- tool guidance 只覆盖最终可见工具，不为环境允许但 agent 未允许的工具生成 guidance。

## 8. 删除标准

以下旧逻辑应删除，不保留兼容分支：

- compiler/controller 隐式注入未编排默认 prompt 层的路径。
- `general_base_plus_selected_overlay` 隐式环境继承。
- lifecycle refs 只追加不覆盖的路径。
- tool guidance refs 只追加不覆盖的路径。
- 为不可见工具生成 guidance 的路径。
- full_access 扩大 agent model-visible capability 的路径。
- prompt catalog 中 hidden/debug 工具可见的路径。

保留兼容只允许用于数据迁移或外部 API 合同，且必须有明确删除条件。本次 runtime prompt 主链路不需要保留旧语义。

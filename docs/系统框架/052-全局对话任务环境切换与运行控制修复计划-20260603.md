# 全局对话任务环境切换与运行控制修复计划

日期：2026-06-03

状态：待确认实施。本文记录 2026-06-03 关于任务环境前端切换、全局对话状态、运行中任务环境不可变和停止控制的修复方案。本文是实施计划，不代表代码已完成。

## 0. 结论

当前前端左侧“任务环境”下拉不是读取底层 `TaskEnvironmentRegistry`，而是用 `WorkspaceView` 硬编码了三项：

```text
chat -> 常规环境
code-environment -> 开发环境
creative -> 写作环境
```

但后端真实注册的任务环境来自：

```text
backend/task_system/environments/registry.py
backend/task_system/environments/default_environments.py
backend/task_system/environments/catalog.py
```

当前注册表包含：

```text
env.coding.vibe_workspace
env.development.sandbox
env.creation.writing
env.general.workspace
```

这造成了一个结构性错位：前端把“页面视图”伪装成“任务环境”，导致用户看不到真实注册的 coding 环境，也无法自然使用未来新增的自定义任务环境。

本次目标不是改显示文案，而是重建权威链：

```text
TaskEnvironmentRegistry
-> TaskEnvironmentCatalog API
-> Global Conversation Active Environment
-> Turn Environment Snapshot
-> TaskRun Environment Contract
-> Runtime Monitor / Stop Control
-> UI Presentation
```

核心原则：

- 对话是全局的，不因切换任务环境而换 session。
- 任务环境是当前对话状态的一部分，允许在同一对话内切换。
- 图任务是独立会话口，不纳入全局对话；图任务继续使用独立 `task_environment` session scope。
- 每一轮 turn 必须记录实际使用的 `task_environment_id`。
- `task_run` 一旦启动，运行环境不可变。
- 切换任务环境只影响后续 turn / 后续 task run，不影响正在运行的任务。
- 停止按钮必须控制真实运行中的 task run / graph run，不只停止前端流。


- 读取后端注册表。
- 切换到真实 task environment catalog item。
- 记录每轮 turn 使用的环境。
- 改变已经运行的 task run。


这个测试方向有一半是正确的：切换环境不应换掉全局对话。

但它保护了错误来源：环境来自硬编码 `WorkspaceView` 映射，而不是注册表。

## 2. 目标行为

### 2.1 全局对话

主对话是全局 session，不按任务环境拆 session pool。

同一条主对话可以连续执行跨域任务：

```text
turn 1-3: env.general.workspace
turn 4-7: env.coding.vibe_workspace
turn 8: env.development.sandbox
turn 9: env.general.workspace
```

用户切换任务环境时，不创建新的全局对话，也不丢弃当前消息历史。

图任务例外：

```text
env.creation.writing 的图任务运行、任务图编辑台和 GraphRun 继续使用独立任务环境会话口。
图任务不合并进主对话的全局 session。
```

原因：

- 图任务有独立 GraphRun、GraphLoop、work order、checkpoint 和任务图会话解析器。
- 图任务会话需要固定 `workspace_view=task_environment` 与 `task_environment_id`，用于图运行恢复、项目绑定和 GraphRun scope 校验。
- 主对话可以引用图任务结果，但不能把图任务运行态混入主对话 session。

### 2.2 当前激活环境

前端 store 需要表达当前对话的激活环境：

```text
active_task_environment_id
active_task_environment_label
active_task_environment_source
```

该状态来自用户选择或后端恢复，不来自硬编码视图。

该状态必须持久化为主对话状态，而不能只存在前端内存中。刷新页面、重新打开窗口或恢复会话时，应能恢复当前主对话的 active environment。

允许的落点：

```text
session metadata
conversation state
chat run state
```

实施前需要先确认现有 session manager 的状态写回模型，避免新增旁路文件。

### 2.3 Turn 环境快照

每次发送消息时，前端把当前 `active_task_environment_id` 作为本轮 task selection 发给后端。

后端必须把该环境写入本轮运行上下文，至少需要在以下位置可审计：

```text
turn metadata / message metadata
chat run task_selection
task_run.task_environment_id
runtime assembly diagnostics
runtime monitor projection
```

Turn 环境快照不是环境 prompt。它只记录本轮使用的环境事实、prompt refs 和运行绑定关系，不能把环境 prompt 全文写进长期对话历史。

建议结构：

```json
{
  "turn_id": "turn:6",
  "task_environment_id": "env.coding.vibe_workspace",
  "environment_kind": "coding",
  "environment_prompt_refs": [
    "environment.coding.vibe_workspace.orientation.v1",
    "environment.rule.coding_workspace.v1",
    "coding.rule.codebase_inspection.v1"
  ],
  "runtime_assembly_id": "...",
  "task_run_id": "taskrun:..."
}
```

环境 prompt 的模型可见内容仍只由当前 turn 的 runtime assembly 装配：

```text
active_task_environment_id
-> resolve TaskEnvironmentSpec
-> resolve environment_prompt_refs
-> PromptAssemblyService
-> RuntimeCompiler model messages
```

历史 turn 只允许暴露必要的跨域事实摘要和环境标注，不允许重新注入旧环境 prompt 全文。

如果某一轮没有显式选择，则应使用明确默认环境：

```text
env.general.workspace
```

默认环境只能由后端 runtime selection / fallback diagnostics 明确说明，不能由前端静默推断多个旧 ID。

### 2.4 TaskRun 环境合同不可变

`task_run` 创建后，环境是运行合同的一部分：

```text
task_run.task_environment_id
task_run.environment_id
task_run.runtime_assembly.task_environment.environment_id
```

切换前端任务环境不能修改已经存在的 task run。

继续、暂停、停止、监控、恢复都必须以原 task run 环境为准。

图任务继续以 GraphRun 自身的 `task_environment_id`、`session_scope_key` 和 owning `task_run_id` 为准，不读取主对话当前 active environment。

### 2.5 切换环境的语义

主对话中切换环境只影响：

- 后续用户 turn。
- 后续 task run。
- 当前 UI 的资源边界提示。
- 后续工具和 runtime 装配输入。

主对话中切换环境不影响：

- 已经启动的 task run。
- 旧 turn 的环境记录。
- 旧消息的运行附件。
- 当前全局 session ID。
- 任何独立图任务会话或已经启动的 graph run。

### 2.6 停止控制

任务运行中，前端必须提供停止键。

停止键要覆盖三类情况：

```text
普通 streaming turn -> stopCurrentStream()
single_agent_task task_run -> stopActiveTaskRun()
graph run / bound graph monitor -> owning task_run stop control
```

停止按钮显示依据必须来自后端 monitor / active task run 状态，而不是当前选择的任务环境。

如果用户切换了任务环境，旧任务仍在运行，停止键仍应能停止旧任务。

GraphRun 停止不应新增独立停止权威。GraphRun UI 应解析 owning `task_run_id`，通过统一 TaskRun stop control 停止运行；如需新增 GraphRun stop API，该 API 只能代理到 owning task run stop，不能单独维护 GraphRun 停止状态。

## 3. 权威链设计

### 3.1 读取权威

```text
Backend TaskEnvironmentRegistry
-> build_task_environment_catalog()
-> GET /api/tasks/overview
-> frontend task environment catalog state
-> task environment selector options
```

前端不再维护任务环境列表。

### 3.2 选择权威

```text
User selects environment in global conversation
-> frontend active_task_environment_id
-> persisted conversation active environment
-> next turn task_selection
-> backend runtime assembly
-> task_run environment contract
```

前端只决定“下一轮使用哪个环境”，不决定已经运行任务的环境。

### 3.3 运行权威

```text
task_run.task_environment_id
-> runtime monitor
-> message runtime attachment
-> UI progress / stop controls
```

运行中任务显示与控制来自 task run，不来自当前下拉。

### 3.4 页面视图权威

`WorkspaceView` 只负责展示壳，不负责定义任务环境。

可以保留“推荐展示壳”：

```text
env.coding.vibe_workspace -> code-environment shell
env.development.sandbox -> code-environment shell 或 chat shell
env.general.workspace -> chat shell
custom -> chat shell
```

该映射只能决定 UI 布局，不决定环境存在性、环境权限、运行合同。

图任务 / 写作图工作台例外：

```text
env.creation.writing -> creative graph session shell
```

该 shell 是独立图任务会话口，不是主对话 active environment 切换的一部分。

## 4. 实施计划

### Phase 1：任务环境目录接入前端全局状态

目标：

- 前端 store 能读取并缓存 `task_environment_management.environments`。
- 左侧任务环境下拉使用注册表数据。
- 删除 `WorkspaceModeSwitcher` 内硬编码环境选项。

涉及文件：

```text
frontend/src/lib/api.ts
frontend/src/lib/store/types.ts
frontend/src/lib/store/runtime.ts
frontend/src/components/layout/WorkspaceModeSwitcher.tsx
frontend/src/components/layout/WorkbenchShell.tsx
frontend/src/components/workspace/views/task-system/environment/environmentPresentation.ts
```

要求：

- 使用已有 `/api/tasks/overview`，优先不新增 API。
- 如果任务系统 overview 过重，可新增轻量 catalog API，但必须仍由后端 `TaskEnvironmentRegistry` 构建。
- 前端环境标题统一复用 `taskEnvironmentDisplayTitle()`，避免多套翻译。

### Phase 2：全局对话 active environment 状态

目标：

- 新增或重命名当前全局对话 active environment 状态，推荐命名为 `conversationActiveEnvironment` 或 `activeTaskEnvironment`。
- `chatTaskEnvironmentBinding` 只作为迁移读取对象，最终不再作为主要状态名。
- active environment 不再由 `WorkspaceView` 自动硬编码决定。
- active environment 需要写回主对话状态，支持刷新后恢复。

涉及文件：

```text

frontend/src/components/chat/ChatPanel.tsx
frontend/src/components/layout/WorkspaceModeSwitcher.tsx

backend/api/chat.py
```

要求：

- 切换环境不改变 `currentSessionId`。
- 切换环境不清空 `messages`。
- 切换环境不取消 running stream / task run。
- 下一次 `sendMessage()` 使用新的 active environment。
- 图任务会话不使用该状态切换；仍走任务环境会话解析器。

### Phase 3：Turn 与 TaskRun 环境快照

目标：

- 每轮发送时明确携带当前 active environment。
- 后端创建 task run 时固化环境。
- UI 能显示消息或运行附件实际使用的环境。

涉及文件：

```text

backend/sessions/__init__.py

backend/harness/runtime/assembly.py
backend/runtime/shared/execution_record.py
```

实施前需要先读现有 chat run / task run 字段，避免新增重复字段。

要求：

- 历史 turn 的环境不可被当前选择覆盖。
- runtime monitor 里的环境显示来自 task run，而不是前端下拉。
- 如果后端已有字段，优先补投影，不新增旁路结构。
- 图任务 turn / GraphRun 使用图任务会话 scope 和 GraphRun 环境字段，不从主对话 active environment 继承。

### Phase 4：运行中任务停止键

目标：

- 对当前对话中活跃 task run / graph run 提供稳定停止入口。
- 停止逻辑不依赖当前 active environment。

现有入口：

```text
frontend/src/lib/store/runtime.ts
stopCurrentStream()
stopActiveTaskRun()
pauseActiveTaskRun()
resumeActiveTaskRun()
```

现有 UI：

```text
frontend/src/components/chat/ChatPanel.tsx
```

已存在单 agent task 的停止按钮，但显示条件过窄。

已知缺口：

```text
frontend/src/components/workspace/views/task-system/TaskGraphPublishRunPage.tsx
```

当前存在提示：

```text
停止运行入口暂未接入新 GraphRun 链路。
```

修复要求：

- single agent task run：继续使用 `stopActiveTaskRun()`，但显示条件改为从 active monitor / task instance 判断。
- graph run：从 GraphRun monitor / binding 解析 owning `task_run_id`，调用统一 TaskRun stop control；如果补 GraphRun stop API，也只能作为代理入口。
- 普通流：保留 ChatInput 停止生成。
- 停止后刷新 monitor，并在消息运行附件中显示停止状态。

### Phase 5：清理旧硬编码和旧测试

目标：

- 删除 `TASK_ENVIRONMENT_WORKSPACE_MODES` 作为环境定义源的权力。
- 删除 `TASK_ENVIRONMENT_VIEW_BY_ID` 作为环境存在性来源的权力。
- 测试不再断言 `Vibe Coding Workspace` 这种前端硬编码标签。

保留：

- `WorkspaceView`。
- environment -> 推荐 UI shell 的展示映射。

删除或改写测试：


```text

```

## 6. 非目标

本计划不做：

- 不把每个任务环境拆成独立 session pool。
- 不把图任务纳入主对话全局 session。
- 不让前端下拉改写运行中 task run。
- 不用前端硬编码替代后端注册表。
- 不为了兼容旧 `env.development.readonly`、`env.vibe_coding` 等历史 ID 保留旧链路。
- 不把 `WorkspaceView` 当成任务环境 ID。

## 7. 风险与处理

### 风险一：全局对话跨环境后上下文污染

处理：

- 每个 turn 记录实际环境。
- runtime assembly 只按当前 turn 环境装配工具、文件、资源和权限。
- 历史消息可以作为对话语义上下文，但不能把旧环境权限带到新环境。
- 历史环境 prompt 不作为下一轮 model-visible prompt 复用；只允许使用 prompt refs / environment id 做审计和 UI 标注。
- 从一个环境切到另一个环境时，当前轮 prompt 需要显式声明“你现在处于新的任务环境”，但不能把旧环境的工具、文件或写入权限带入当前环境。
- 跨域上下文只传递任务事实、用户目标、产物引用和已批准摘要；不传递旧环境的 sandbox policy、file access table、tool authorization、system rule text。
- 如果用户要求基于旧环境产物继续，新环境必须通过当前环境允许的资源读取路径重新获取产物，不能把旧环境的未授权文件状态当作当前环境事实。

上下文投影规则：

```text
history messages: semantic continuity only
turn environment snapshot: audit/provenance only
current runtime assembly: only authority for current tools/prompts/permissions
task_run contract: only authority for running task environment
```

禁止：

```text
old environment prompt content -> next turn prompt
old file access table -> current environment file permission
old sandbox policy -> current command permission
old task_run environment -> new task_run environment
```

### 风险二：前端当前选择和运行中任务环境不一致

这是预期允许的状态。

处理：

- UI 明确显示 running task 的真实环境。
- 停止键绑定 running task，而不是当前选择。
- 新任务使用当前选择。

### 风险三：GraphRun 停止链路不完整

处理：

- 先检查 GraphRun monitor / binding 是否稳定暴露 owning `task_run_id`。
- 优先调用统一 TaskRun stop control。
- 如果新增 GraphRun stop API，必须代理到 owning TaskRun stop，不允许形成第二套停止状态。
- 前端停止后必须刷新 monitor 并展示停止状态。

### 风险四：图任务会话和主对话 active environment 混用

处理：

- 写作图任务、TaskGraphRun、GraphLoop、GraphRun monitor 继续走独立任务环境会话口。
- 主对话 active environment 不参与 GraphRun scope 校验。
- 主对话可以引用图任务产物，但不能接管图任务运行态。

## 8. 完成标准

本计划完成后应满足：

- 左侧任务环境列表来自底层注册表。
- `env.coding.vibe_workspace` 在前端真实可见。
- 自定义任务环境保存后可出现在同一选择器中。
- 全局对话切换环境不换 session、不丢消息。
- 下一轮消息使用当前 active environment。
- 当前 active environment 刷新后可恢复。

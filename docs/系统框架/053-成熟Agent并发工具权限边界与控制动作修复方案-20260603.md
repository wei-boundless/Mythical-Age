# 成熟 Agent 并发工具、权限边界与控制动作修复方案

日期：2026-06-03

## 1. 问题定义

当前单 Agent 回合出现过多次用户可见阻塞：

```text
模型一次返回了多个系统动作，运行时已停止执行，避免遗漏或误触发。
```

这不是单个模型偶发问题，而是运行时协议设计问题。当前代码把普通工具调用、系统控制动作、权限准入结果混在同一条 native action 链路里处理，导致三类本应分层的行为互相污染：

- 普通工具调用本来可以多个、可以并发。
- 控制动作本来必须互斥，例如 `respond`、`ask_user`、`block`、`request_task_run`、`active_work_control`。
- 权限准入本来应该提供边界和能力反馈，不应该默认终止 agent 主循环。

正确目标不是放开权限，也不是盲目执行多个动作，而是：

```text
普通工具可批量，控制裁决只允许一个，权限拒绝转为模型观察，协议冲突先内部修复。
```

## 2. 本项目当前证据

### 2.1 single_agent_turn 混合了工具与控制动作

文件：`backend/harness/loop/single_agent_turn.py`

当前 `_native_tools_for_packet()` 同时暴露：

- 普通运行工具：来自 `available_tools`
- 控制工具：`request_task_run`、`active_work_control`、`ask_user`、`block`

当前 `_invoke_single_turn_model()` 对 native tools 设置：

```python
tool_call_options={"parallel_tool_calls": False}
```

这等于用 provider 参数强压模型“一次只返回一个 native tool call”。但 DeepSeek 等模型仍可能返回多个 tool calls。随后 `_single_agent_action_request_from_response()` 遇到多个 native actions 直接返回：

```python
code="single_agent_turn_multiple_native_actions"
```

该错误再被 `_emit_single_agent_protocol_error()` 投射为用户可见 blocked。

问题不在于 fail-closed 本身，而在于它保护的是混乱协议：普通工具和控制动作被放在一个 native action 面里，导致“多个普通工具”也被设计成危险形态。

### 2.2 权限准入直接终止回合

文件：`backend/harness/loop/single_agent_turn.py`

工具 loop 中存在：

```python
if admission.decision == "needs_contract": ... answer_channel="blocked"
if admission.decision != "allow": ... answer_channel="blocked"
```

工具 observation 非 `ok` 也直接：

```python
answer_channel="blocked"
terminal_reason=f"tool_{observation.status}"
```

这会让权限、工具不可用、工具参数错误、工具失败直接变成 agent 停止。成熟 agent 应该把这些反馈作为模型可见 observation，允许模型恢复：换工具、改用直接回答、询问用户、请求任务合同，或明确裁决 block。

### 2.3 task_executor 也有同类阻塞风险

文件：`backend/harness/loop/task_executor.py`

任务执行中：

```python
if admission.decision != "allow":
    return _finish_executor_blocked(...)
```

这会把工具准入拒绝升级成整个 TaskRun blocked。对于真正不可恢复的硬边界可以 blocked，但普通工具权限拒绝、工具不可用、合同不足，应优先作为 observation 进入下一次模型调用。

### 2.4 compiler prompt contract 语义过宽

文件：`backend/harness/runtime/compiler.py`

当前 `_single_agent_turn_output_contract()` 写入：

```python
"single_action_per_turn": True
"native_actions": {"parallel_tool_calls": False}
```

这里的 “single action” 过宽。成熟语义应该是：

```text
single_control_action_per_turn = true
tool_calls.parallel_allowed = true
control_actions.parallel_allowed = false
```

### 3.1 Codex 工具声明是否支持并行

文件：`D:\AI应用\openai-codex\codex-rs\tools\src\tool_executor.rs`

Codex 的工具执行器 trait 有默认：

```rust
fn supports_parallel_tool_calls(&self) -> bool {
    false
}
```

具体工具 handler 可以覆盖该能力。例如：

- `core/src/tools/handlers/unified_exec/exec_command.rs` 中 `exec_command` 返回 `true`
- `core/src/tools/handlers/tool_search.rs` 中 `tool_search` 返回 `true`

这说明成熟设计不是全局禁止并行，而是由工具能力声明和调度器判断。

### 3.2 Codex prompt 允许模型级 parallel_tool_calls

文件：`D:\AI应用\openai-codex\codex-rs\core\src\session\turn.rs`

`build_prompt()` 中：

```rust
parallel_tool_calls: turn_context.model_info.supports_parallel_tool_calls
```

并行工具调用是模型/工具能力的一部分，而不是被主循环永久关闭。

### 3.3 Codex 把工具非致命错误返回给模型

文件：`D:\AI应用\openai-codex\codex-rs\tools\src\function_call_error.rs`

Codex 区分：

```rust
RespondToModel(String)
Fatal(String)
```

文件：`D:\AI应用\openai-codex\codex-rs\core\src\tools\parallel.rs`

非 fatal 工具错误会被转换成 function call output：

```rust
Err(other) => Ok(Self::failure_response(error_call, other))
```

这支持本方案原则：工具参数错误、权限策略不允许、工具不可用等应反馈给模型，而不是默认阻塞 agent 主循环。

### 3.4 Codex 权限/审批不是绕过，也不是直接停机

文件：`D:\AI应用\openai-codex\codex-rs\core\src\tools\handlers\unified_exec\exec_command.rs`

当 approval policy 不允许请求 escalated permissions 时，Codex 返回：

```rust
FunctionCallError::RespondToModel(...)
```

沙箱拒绝时，Codex 构造工具输出，带 exit code 和输出文本，让模型可见该结果。也就是说权限仍然阻止真实执行，但 agent 可以继续基于结果恢复。

### 3.5 Codex 有并发工具回归测试

文件：`D:\AI应用\openai-codex\codex-rs\core\tests\suite\code_mode.rs`

`code_mode_nested_tool_calls_can_run_in_parallel` 测试用 `Promise.all` 验证嵌套工具可以并行完成。

这进一步说明：成熟 agent 设计支持工具级并发，但需要由工具能力、权限、调度器共同约束。

## 4. 原则复核结论

### 原则 A：普通工具可以并发

结论：合适，必须改。

理由：

- Codex 源码中有 `supports_parallel_tool_calls()`。
- Codex prompt 层会传 `parallel_tool_calls`。
- Codex 有并发工具测试。
- 本项目当前全局 `parallel_tool_calls=False` 会限制 agent 能力，并放大模型多工具返回时的协议冲突。

边界：

- 单 Agent 回合默认只开放只读工具，可并发执行。
- 写入、shell、浏览器、网络等副作用工具必须由调度器做顺序/冲突判断。
- 未授权工具仍不执行。

### 原则 B：控制动作必须单选

结论：合适，必须保留。

理由：

`respond`、`ask_user`、`block`、`request_task_run`、`active_work_control` 是 turn 控制流裁决，不是普通工具。一个回合不能同时成立：

- 一边问用户，一边继续任务。
- 一边 block，一边 request_task_run。
- 一边 respond，一边 active_work_control。

需要修改的是命名和协议，不应再写 “single_action_per_turn”，而应写：

```text
single_control_action_per_turn
```

### 原则 C：权限是能力边界，不是运行阻断器

结论：合适，但要保留硬安全边界。

权限层应该负责：

- 判断工具是否可见。
- 判断工具是否可执行。
- 判断是否需要用户审批、任务合同或更高权限。
- 生成可追踪的 permit / denial / observation。

权限层不应该负责：

- 替模型决定最终回答。
- 直接把普通权限拒绝升级为整轮 blocked。
- 把工具不可用变成用户不可恢复错误。

硬安全边界仍可终止执行，例如：

- 尝试越权写入受保护路径。
- 破坏性命令被沙箱拒绝。
- 运行时完整性失败。
- 明确违反安全/权限政策。

但这些也应先区分：

```text
工具不可执行 != agent 不可继续
```

### 原则 D：prompt 修复不能替代结构修复

结论：合适。

仅改 prompt 会继续依赖模型自律，无法解决 DeepSeek 等模型仍返回多个 native tool calls 的情况。必须在 runtime 结构上分离：

- ToolBatch
- ControlAction
- AdmissionObservation
- ProtocolRepair

### 原则 E：repair prompt 必须防止上下文污染

结论：合适。

内部 repair 只用于当前 turn 的协议修复，不写入全局会话正文，不把开发说明暴露给业务 agent。成功后只持久化最终动作或工具 observation；失败时记录 diagnostics，不持久化原始冲突动作。

## 5. 目标权威链

目标运行链路：

```text
RuntimeCompiler
-> ModelResponseParser
-> ToolBatchOrControlAction
-> AdmissionPermit
-> ToolScheduler / ControlExecutor
-> ObservationOrTerminal
-> ModelFollowup / FinalAnswer
```

职责边界：

- `RuntimeCompiler`：装配环境、工具、控制能力和 prompt contract。
- `ModelResponseParser`：只解析和分类，不做权限决定。
- `AdmissionPermit`：只做准入和 permit，不决定最终用户输出。
- `ToolScheduler`：执行允许的工具，拒绝项生成 observation。
- `ControlExecutor`：只执行一个合法控制动作。
- `ModelFollowup`：接收工具/权限 observation 后恢复。
- `Terminal`：只有明确控制裁决或硬边界才终止。

## 6. 修复设计

### 6.1 native tools 只承载普通工具

从 single-agent 常规 native tool 暴露中移除：

- `request_task_run`
- `active_work_control`
- `ask_user`
- `block`

这些改走 JSON action control protocol。

保留对模型意外返回这些 native control calls 的兼容解析，但只作为协议错误输入进入 repair，不直接执行。

### 6.2 多普通工具改为 ToolBatch

解析规则：

- 只有普通 native tool calls：生成 `ToolBatch`。
- 一个 JSON control action：生成 `ControlAction`。
- 普通工具和控制动作混用：触发 repair。
- 多个控制动作：触发 repair。
- 无工具、无 JSON、只有文本：按 assistant message 收口。

### 6.3 权限拒绝转 observation

对每个 tool action 做 admission：

- `allow`：执行工具。
- `deny`：不执行工具，生成 `ToolObservation(status="denied")`。
- `needs_contract`：不执行工具，生成 `ToolObservation(status="needs_contract")`。
- `invalid`：不执行工具，生成 `ToolObservation(status="error")`。

这些 observation 进入后续模型调用，模型必须恢复。

### 6.4 工具失败转 follow-up

工具执行结果非 `ok` 时：

- 不直接 `answer_channel=blocked`。
- 把失败输出作为 tool observation 放入 model messages。
- 模型下一轮决定：重试、换工具、说明限制、请求任务、询问用户或 block。

### 6.5 repair 只修协议，不执行原始动作

触发条件：

- 多个控制动作。
- JSON action 和 native tool calls 混用。
- native control action 出现在普通工具批次里。
- JSON action 不合法但可修复。

repair 只尝试一次。repair 成功后按新动作继续；repair 失败时不执行原始动作，并记录 `protocol_repair_failed` diagnostics。

### 6.6 task_executor 权限恢复

TaskRun 内 admission deny 不应直接 `_finish_executor_blocked`，应优先生成 executor observation 并进入下一次模型调用。

只有这些情况才 blocked：

- 模型明确返回 `block`。
- 达到恢复上限。
- 运行时硬边界不可恢复。
- 合同必须由用户补充且模型已选择 ask/block。

## 7. Prompt 修改方案

### 7.1 单 Agent 主提示新增工具/控制分层

```text
你可以在同一轮请求多个普通工具，以便并行获取事实。
普通工具包括读取、检索、状态检查等不会直接改变用户目标控制流的工具。

你每一轮只能做出一个控制裁决。
控制裁决包括：直接回复用户、询问用户、说明不可继续、请求启动持续任务、控制当前进行中的任务。

不要在同一条响应里同时发起控制裁决和普通工具调用。
如果还需要事实，先调用工具；收到工具结果后再做控制裁决。
```

### 7.2 权限/能力边界提示

```text
权限边界描述的是当前你能使用哪些能力。
当某个工具不可用、被环境过滤、需要任务合同或缺少授权时，不要把这当作默认阻塞理由。
你应该先尝试使用当前允许的能力继续推进；如果仍无法可靠完成，再询问用户、请求任务合同，或清楚说明边界。
```

### 7.3 block 控制动作提示

```text
只有在用户目标越过硬性安全边界、所有可恢复路径都不可用、或继续会明显产生误导性结果时，才选择 block。
不要因为单个工具不可用、权限需要确认、环境能力不足就直接 block。
这些情况优先选择：换用可用工具、请求任务运行、询问用户、或直接说明当前可完成的部分。
```

### 7.4 active_work_control 提示

```text
当用户明确要求继续、暂停、停止、补充或询问当前进行中的工作时，选择 active_work_control。
这个控制动作只影响当前已有工作，不会改变已经启动的任务运行环境。
不要同时返回 active_work_control 和 ask_user、block、request_task_run。
```

### 7.5 内部协议 repair prompt

```text
你是一名动作协议修复员。
你只负责把上一轮模型输出修复为一个合法的控制裁决或工具调用批次。
你不能执行动作，不能扩写用户目标，不能引入新需求。

上一轮输出违反了运行协议。
请根据用户当前请求、运行边界和允许动作，只输出一个 JSON 对象。
如果需要控制裁决，只能选择一个 action_type。
如果需要普通工具，可以输出一个 tool_batch，但不能混入 ask_user、block、request_task_run 或 active_work_control。

禁止输出解释文字，禁止 Markdown，禁止多个控制动作。
```

## 10. 切换规则

不保留旧主链路。

允许短期保留的只有“解析模型意外返回旧 native control tool”的 repair 入口，且该入口不能执行旧动作，只能转换为协议错误并触发 repair。它不是兼容执行链路。

删除或废弃：

- 控制动作作为常规 native tools 的模型可见暴露。
- 多 native actions 直接用户可见 blocked 的默认路径。
- 权限拒绝直接停止 single-agent turn 的默认路径。

保留：

- 权限准入。
- action permit。
- tool control plane。
- sandbox / approval enforcement。
- 明确 `block` 控制动作。

## 11. 不做的事

- 不把权限 deny 改成 allow。
- 不允许未授权工具真实执行。
- 不把所有工具都并发执行；副作用工具必须调度。
- 不把 graph node 的业务 blocked 语义直接删除。
- 不用 prompt 文案替代 runtime 结构修复。

## 12. 最终结论

本方案原则是合适的，并且和 Codex 源码体现的成熟 agent 设计一致：

- 工具是否并发由工具能力和调度器决定。
- 工具错误和权限不可用应尽量反馈给模型。
- 权限仍然阻止真实越权执行。
- 控制裁决必须互斥。
- 运行时结构必须强制这些边界，不能只靠 prompt。

当前项目的问题不是“agent 一次只能做一个动作”的安全要求，而是把“一个控制裁决”错误实现成“一个 native action”。修复方向应当是拆分 ToolBatch 与 ControlAction，并把权限拒绝从用户可见阻塞改成模型可恢复 observation。

# Codex式任务恢复与自然控制重构升级计划书

日期：2026-05-30

状态：待审阅实施稿

## 1. 问题定义

当前系统已经具备 `TaskRun`、`RuntimeEventLog`、`RuntimeObjectStore`、运行监控投影、暂停/继续/停止控制和健康系统任务记录治理能力，但“任务恢复”和“自然语言控制”仍没有达到成熟 agent 的体验标准。

主要问题不是某个“继续”按钮或状态文案，而是恢复主权尚未稳定：

```text
用户自然输入
-> 当前工作识别
-> 恢复候选重建
-> 模型判断本轮意图
-> 恢复策略裁决
-> 执行器启动或 checkout/fork
-> 进展呈现
```

这条链路现在被拆散在 `QueryRuntime`、`active_work.py`、`task_executor.py`、`resume_decision.py` 和 monitor 投影中。结果是：

```text
1. user_aborted / runtime restart / executor interrupted 后，旧任务可能从 active work 视野里消失。
2. “继续”容易被处理成普通聊天，或者只对 waiting_executor 生效。
3. 同一个 TaskRun 被当成可原地复活对象，和 Codex 式 checkout/fork 恢复语义不一致。
4. 用户能看到 runtime packet / TaskRun / 执行器 / 装配 等开发细节，体验不自然。
5. UI 展示任务运行像硬说明，不像连续进展。
```

正确目标是：用户仍然只是在和同一个助手自然对话；系统内部可以进入任务状态机，但不把状态机暴露成产品语言。恢复应基于可重建历史，而不是基于关键词或直接改写 terminal `TaskRun`。

## 2. Codex源码参照结论

已检查官方 `openai/codex` 源码，本地源码位置：

```text
C:\Users\admin\.codex\sources\openai-codex
commit 20da4c3
```

Codex 的关键设计不是“Task 对象复活”，而是：

```text
Thread / Session 是会话主线。
Turn 是用户可感知的一轮处理。
SessionTask 是 turn 内部的执行单元。
Rollout history 是恢复和 fork 的真实来源。
Interrupted turn 会写入模型可见的中断边界。
Resume / Fork 通过 InitialHistory 重建上下文。
```

源码证据：

```text
codex-rs/core/src/tasks/mod.rs
  SessionTask.run / abort
  spawn_task / start_task
  abort_all_tasks / handle_task_abort / on_task_finished

codex-rs/core/src/tasks/regular.rs
  RegularTask.run -> run_turn

codex-rs/core/src/session/turn.rs
  run_turn model/tool loop

codex-rs/protocol/src/protocol.rs
  InitialHistory::{New,Cleared,Resumed,Forked}
  AgentStatus::{PendingInit,Running,Interrupted,Completed,Errored,Shutdown}
  TurnStarted / TurnComplete / TurnAborted

codex-rs/core/src/thread_manager.rs
  resume_thread_from_rollout
  fork_thread
  ForkSnapshot::Interrupted
  append_interrupted_boundary

codex-rs/core/src/session/rollout_reconstruction.rs
  reconstruct_history_from_rollout
```

对本项目可借鉴的不变量：

```text
1. 任务执行对象不是恢复真值；历史和 checkpoint 才是恢复真值。
2. 中断不是普通失败；Interrupted 是可继续接受输入的状态。
3. 对半途终止的工作，成熟做法是 checkout/fork 一个新 attempt，而不是改旧 terminal run。
4. 中断事实必须进入模型可见上下文，但不应以开发术语展示给用户。
5. UI 状态从事件派生，不手写另一个状态机。
```

不照搬的部分：

```text
1. 不把项目改成 Codex 的 ThreadManager / Rust session 结构。
2. 不把 GraphHarness 交给 LangGraph runtime；本项目只借用 checkpoint 抽象。
3. 不让健康系统执行恢复或改写 runtime 事实源。
```

### 3.2 当前结构性缺口

第一，`build_active_work_context()` 只返回一个 active work，并且 terminal 状态被直接过滤：

```text
_ACTIVE_WORK_STATUSES = created/running/waiting_executor/waiting_approval/blocked
_TERMINAL_STATUSES = completed/success/failed/aborted/cancelled/error
```

这会导致 `user_aborted`、运行时重启后的旧任务、半途卡住的任务不进入“可自然继续”的候选集。

第二，`resume_paused_task_run()` 语义只适合 same-run resume：

```text
waiting_executor / paused / recoverable protocol terminal -> same TaskRun 继续
completed / failed / aborted -> conflict
```

这本身是正确的，但缺少 `checkout/fork` 机制承接 terminal interrupted work。

第三，`runtime.shared.resume_decision.py` 已有 checkpoint 决策模型，但当前用法偏底层，且包含 `_user_requests_force_continue()` 这种词面判断。它不应做用户语义判断，只应在上层模型已决定“继续当前工作”后裁决恢复策略。

第四，单 agent TaskRun 没有统一 `WorkRollout / TaskRunTimeline`。当前恢复只能从 event log、observations、diagnostics 拼装，缺少稳定的模型可见恢复摘要。

第五，UI 仍容易把 runtime 事件直投给用户。产品层应该看到流程进展和 agent 简要输出，而不是 `runtime_invocation_packet_compiled` 等执行细节。

## 4. 目标架构

### 4.1 权责链

目标权责链固定为：

```text
QueryRuntime
  只提交用户消息、调用控制链、转发事件。

WorkContinuationResolver
  从 monitor / TaskRun / EventLog / WorkRollout 中收集当前会话的可继续候选。
  不判断用户意图。

ActiveWorkTurnDecision
  由模型判断用户本轮自然语言与候选工作的关系。
  可选择 continue / pause / stop / append_instruction / answer / start_new_work / normal_response。

ResumePolicy
  在用户意图已确定后，裁决 same-run resume / checkout fork / completed iteration / restart / ask_user。
  不解析用户关键词。

TaskRunExecutor
  只执行被授权的 TaskRun 或 checkout attempt。
  不重新判断用户意图。

WorkRolloutStore
  记录可恢复模型历史、进展、工具摘要、中断边界、artifact refs、checkpoint refs。

RuntimeEventLog / RuntimeStateIndex
  保持原始事实账本和最新对象索引。

RuntimeMonitor
  从事实源投影实时状态。

HealthSystem
  管理任务记录健康、保留策略、异常治理和报告。
  不执行恢复，不改写 runtime 原始事实。
```

### 4.3 状态语义

必须区分：

```text
paused / waiting_executor:
  旧 TaskRun 仍是非 terminal，可 same-run resume。

running / scheduled:
  不重复启动，只追加用户输入或提示正在处理。

waiting_approval / blocked:
  不强行继续；需要用户确认或解释阻塞。

user_aborted / interrupted / runtime_restart_interrupted:
  旧 TaskRun 保持 terminal 或 waiting_executor 事实不被篡改。
  如果用户要求继续，创建 checkout attempt。

completed:
  用户说继续通常是 iteration，不是 resume。
```

`aborted` 不再从自然继续候选里完全消失；它不能 same-run resume，但可以作为 `interrupted_checkoutable` candidate 提供给模型判断。

## 5. 固定执行流

### 5.1 普通用户输入

```text
QueryRuntime.astream
-> commit user message
-> direct system route
-> WorkContinuationResolver.collect(session_id)
-> if candidates exist:
     ActiveWorkTurnDecision.decide(user_message, candidates)
     apply decision
   else:
     normal AgentHarness
```

### 5.2 用户说继续

```text
模型判断 action=continue_active_work
-> ResumePolicy.build_plan(candidate)
-> same_run_resume:
     resume_paused_task_run
     schedule executor
-> checkout_fork:
     create child TaskRun attempt
     write lineage
     inject interrupted marker + rollout summary
     schedule executor
-> completed_iteration:
     normal AgentHarness turn with previous result summary
```

### 5.3 用户追加要求

```text
模型判断 action=append_instruction_to_active_work
-> append_user_work_instruction
-> WorkRollout append user instruction item
-> if same-run resumable: schedule executor
-> if checkout-only: ResumePolicy checkout_fork with instruction included
```

### 5.4 用户暂停/停止

```text
pause:
  request_task_run_pause
  execution boundary converts running -> paused/waiting_executor
  WorkRollout records pause boundary

stop:
  stop_task_run
  old TaskRun terminal_reason=user_aborted
  WorkRollout records interrupted boundary and last safe progress
  no automatic checkout until user naturally asks to continue
```


## 9. 风险控制

主要风险：

```text
1. checkout/fork 误判导致重复执行副作用。
2. rollout 摘要过度简化，模型恢复时漏掉关键 artifact/tool 结果。
3. UI progress 与真实状态不同步。
4. 健康系统误删可恢复 lineage。
```

控制措施：

```text
1. checkout 恢复 prompt 要求先检查当前工作区和已有 artifact。
2. WorkRollout 记录 refs，不只记录自然语言摘要。
3. 进展 UI 只消费后端 public progress projection，不自己推断。
4. 任务记录清理必须检查 parent/child lineage 和 latest rollout。
5. 所有恢复路径必须记录 ResumePlan 事件。
```

## 10. 是否符合原设计原则

本方案符合既有系统框架原则：

```text
1. QueryRuntime 仍是 adapter，不回收任务主权。
2. AgentHarness/TaskExecutor 仍负责执行。
3. TaskRun 仍是任务记录，归健康系统治理，不迁出。
4. HealthSystem 只做治理观察，不执行恢复。
5. RuntimeMonitor 继续持有实时状态投影。
6. 恢复基于 checkpoint/rollout，不靠关键词启发式。
7. 用户体验保持自然语言，不暴露内部状态机。
8. 旧 terminal TaskRun 不被篡改，链路安全。
```

因此，若审阅通过，可按 Phase 1 到 Phase 6 顺序实施。Phase 1 到 Phase 4 是后端语义主链，Phase 5 是 UI 体验收敛，Phase 6 是健康系统治理接入。不得跳过 Phase 1/2 直接做 UI 或关键词补丁。

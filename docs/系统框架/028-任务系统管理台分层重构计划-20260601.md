# 任务系统管理台分层重构计划

## 背景

当前任务系统页已经把任务图编辑台移出，但契约库和节点配置仍沿用旧管理形态：

- 契约库仍是左侧长列表 + 右侧超长表单，字段、产物、验收、运行策略、交接策略全部堆在一个滚动面板里。它能编辑数据，但不帮助用户判断“这个契约给谁用、影响哪些节点/边/任务图、当前是否可运行”。
- 节点配置页没有真正的节点配置主数据。它从 `/orchestration/agents` 读取 agent 和 runtime profile 候选，只做说明和展示，容易被理解成 agent 管理页。
- 任务环境页已经开始表达 agent 运行装载，但任务环境类型、资源装载、环境 prompt、环境内任务、任务图之间还没有形成清晰的层级。
- 主图工作台负责编辑任务图；任务系统管理台应负责管理任务环境、契约资产、节点配置资产和它们的绑定关系，不应再把任务图拓扑编辑塞回系统页。

这不是样式问题，而是信息架构和决策权混乱：用户想管理的是运行前的配置资产，旧 UI 却按底层 JSON/列表暴露。

## 目标模型

任务系统管理台按“运行配置资产”组织，而不是按旧数据库表展示。

```text
任务系统
├─ 任务环境类型
│  └─ 环境分组 / environment_kind / 默认策略模板
├─ 任务环境
│  ├─ 资源装载：文件 Profile、仓库、记忆、知识、检索、存储空间
│  ├─ 环境 Prompts：给 agent 直接执行的环境说明
│  ├─ 环境内任务：TaskAssignment / Flow / ExecutionPolicy
│  └─ 环境内任务图：绑定到该环境的 TaskGraph 摘要和问题，不编辑拓扑
├─ 契约库
│  ├─ 契约目录：按用途、kind、环境、任务图、问题筛选
│  ├─ 契约详情：身份、输入输出、产物、验收、运行/交接/失败策略
│  └─ 使用影响：哪些任务、节点、边、任务图正在引用
└─ 节点配置
   ├─ 节点配置模板：角色、职责、prompt、执行者引用、runtime profile 引用
   ├─ 节点契约绑定：input/output/node contract、artifact、acceptance
   ├─ 节点能力边界：模型能力、工具上限、记忆读写、失败和人工门控
   └─ 装配预览：环境 + 节点配置 + agent profile + 契约 合成 runtime start packet
```

## 权限与数据所有权

| 对象 | 权威来源 | 管理页职责 | 不允许做的事 |
| --- | --- | --- | --- |
| 任务环境类型 | 新增 `TaskEnvironmentKindTemplate` + `TaskEnvironmentGroup` | 定义环境类别、默认策略模板、允许资源类型和适用任务图范围 | 不把 kind 只藏在 `TaskEnvironmentRecord.environment_kind` 字符串里 |
| 任务环境资源装载 | `TaskEnvironmentSpec` | 管理 Prompt、文件、记忆、知识、检索、存储、sandbox、执行边界 | 不把资源装载做成只读说明 |
| 环境 Prompt | `TaskEnvironmentSpec.environment_prompts` | 编写 agent 可直接执行的环境说明和边界 | 不写开发说明或节点用途说明 |
| 环境内任务 | `TaskAssignment`、`TaskFlowDefinition`、`TaskExecutionPolicy` | 维护任务属于哪个环境、默认执行链、默认契约 | 不继续只靠 flow/task/graph 的 metadata 推断环境归属 |
| 任务图 | `TaskGraphRecord` | 在任务系统里只显示归属、绑定、问题和发布态 | 不把主任务图编辑台移回任务系统 |
| 契约 | `ContractSpec` | 管理可复用契约资产、校验、使用影响 | 不在契约里复制节点/环境主数据 |
| 节点配置 | 新增 `TaskNodeConfigurationSpec` | 管理节点角色、prompt、执行者引用、运行档案引用、契约绑定、能力边界 | 不管理 agent 注册、provider base URL、密钥 |
| Agent / Runtime Profile | `AgentRuntimeRegistry`、Orchestration registry | 只作为节点配置可引用对象 | 不在任务系统里改 agent 注册信息 |

## 目标页面结构

### 1. 系统页总导航

左侧不再只有三个粗粒度入口，而是两级工作台：

```text
配置域
├─ 环境管理
│  ├─ 环境类型
│  ├─ 环境资源装载
│  ├─ 环境 Prompts
│  ├─ 环境内任务
│  └─ 环境任务图
├─ 契约库
│  ├─ 契约目录
│  ├─ 契约详情
│  └─ 使用影响
└─ 节点配置
   ├─ 节点目录
   ├─ 节点详情
   ├─ 能力与权限
   └─ 装配预览
```

切换一级域时切换工作台；切换二级页时不跳转、不离开任务系统。

### 2. 环境管理

环境管理不是单页大表单，拆成五个明确子页：

- `环境类型`：管理分组、kind、默认策略模板和适用范围。
- `环境资源装载`：保留当前 Agent 运行装载编辑能力，作为核心子页。
- `环境 Prompts`：独立管理环境 prompt，展示 prompt 生效顺序、版本、cache scope 和 agent 可见预览。
- `环境内任务`：展示该环境下的 TaskAssignment、Flow、默认契约、执行策略，可编辑任务归属和默认执行链。
- `环境任务图`：展示绑定该环境的 TaskGraph 列表、发布态、问题、入口/出口、契约绑定摘要。这里只做归属和健康检查，不编辑拓扑。

### 3. 契约库

重建契约库为三栏工作台：

```text
左栏：筛选目录
中栏：契约列表/表格
右栏：当前契约详情 Inspector
```

关键操作：

- 按 `contract_kind`、环境、任务图、节点/边使用情况、校验状态筛选。
- 列表列出：名称、kind、输入字段数、输出字段数、验收规则数、引用数、问题数、版本。
- 详情 Inspector 用 tabs 分区：
  - `概览`：身份、用途、版本、启用状态。
  - `Schema`：输入/输出字段，支持行内新增、复制、排序、必填、可见性。
  - `产物与验收`：artifact requirements、acceptance rules。
  - `运行策略`：runtime requirements、context visibility、handoff、failure、human gate。
  - `使用影响`：引用该契约的任务、节点、边、任务图。
  - `高级 JSON`：只保留元数据和少量高级扩展，不再作为主要编辑入口。

旧 `ContractLibraryPanel` 在新工作台落地后删除，不保留两套契约编辑 UI。

### 4. 节点配置

节点配置必须成为一等对象，而不是 agent 候选展示。新增后端模型：

```text
TaskNodeConfigurationSpec
- node_config_id
- title
- description
- node_kind
- environment_scope
- role_prompt
- executor_ref
  - agent_id
  - agent_profile_id
  - agent_selection_policy
- contract_bindings
  - input_contract_id
  - output_contract_id
  - node_contract_id
- model_requirements
- tool_policy
- memory_policy
- artifact_policy
- failure_policy
- human_gate_policy
- metadata
```

节点配置页结构：

- `节点目录`：按环境、任务图、节点 kind、执行者、问题筛选。
- `节点详情`：角色、职责、role prompt、禁止事项、输出标准。
- `执行装配`：agent 引用、runtime profile 引用、模型能力要求、streaming/timeout/budget。
- `契约绑定`：input/output/node contract，显示契约健康状态和字段摘要。
- `能力与权限`：工具、记忆、文件、产物、网络、浏览器权限的有效边界。
- `装配预览`：把任务环境 + 节点配置 + runtime profile + 契约合成 runtime start packet，显示 agent 实际会看到的 prompt、资源、权限和失败门禁。

任务图节点不再复制完整节点配置主数据，只引用 `node_config_id`，并保留少量实例级 override。运行前由编译/保存层把节点配置展开成 runtime 需要的 `TaskGraphNodeRecord` 字段。

## 实施阶段

### 阶段 1：任务系统层级壳重构

目标：先把 UI 层级稳定下来，不改变后端数据语义。

工作：

- 把 `TaskSystemLayer` 从 `environments | contracts | nodes` 升级为一级域 + 二级页。
- `TaskSystemShell` 支持二级导航和当前对象上下文。
- 环境页拆出独立组件，当前资源装载逻辑迁移到 `EnvironmentLoadoutPage`。

验收：

- 用户能明确看到当前位于环境管理 / 契约库 / 节点配置的哪个子页。
- 不出现旧“所有东西堆一个页面”的结构。

### 阶段 2：环境管理子页完整化

目标：把任务环境类型、资源装载、环境 prompt、环境内任务、环境任务图分开。

工作：

- 新增环境类型模板 registry；环境类型页读取 kind templates 和 groups，并允许编辑自定义 group。
- 环境类型模板声明：允许的 resource refs、默认 sandbox/execution/risk policy、默认 prompt cache scope、允许的任务图 kind。
- 环境 Prompt 从资源装载页中抽出独立页，但保存仍写回 `TaskEnvironmentSpec.environment_prompts`。
- 环境 Prompt 页必须提供 agent 可见预览和基础检查：不能出现“这是 runtime 节点”“根据任务图执行”这类开发说明。
- 环境内任务页从 `task_assignments`、`task_flow_definitions`、`execution_policies` 建 inventory。
- 新增 `PUT /tasks/task-assignments/{task_id}`，任务归属写入 `TaskAssignment.task_environment_id`，不能只在前端临时过滤。
- 环境任务图页从 `TaskGraphRecord.runtime_policy.task_environment_id` 和 `context_policy.task_environment_id` 建 inventory；metadata 只作为迁移输入。

验收：

- 环境资源装载页只管资源装载。
- 环境 prompt 页只管 agent 可见 prompt。
- 环境任务和环境任务图不混在资源表单里。

### 阶段 3：契约库重建

目标：用可筛选目录 + 分区 Inspector 替换长表单。

工作：

- 构建 contract usage index：按“关系索引规则”扫描任务、图、节点、边和 contract_bindings 引用。
- 契约列表支持 kind、引用状态、问题状态筛选。
- 契约详情按概览、Schema、产物与验收、运行策略、使用影响、高级 JSON 分区。
- 新编辑器保存仍调用现有 `PUT /tasks/contracts/{contract_id}`。
- 新契约编辑器稳定后删除旧 `ContractLibraryPanel`。

验收：

- 用户能从契约快速看到用途、引用影响和错误。
- 编辑字段、验收、运行策略不需要滚完整个巨型表单。

### 阶段 4：节点配置一等化

目标：把节点配置从“agent 候选展示”升级为可保存、可引用、可预览的配置资产。

工作：

- 新增 `TaskNodeConfigurationSpec` 后端 registry。
- 从现有任务图节点批量迁移初始 node configurations。迁移规则必须确定：
  - 可安全归并的节点才归并为同一个配置。
  - prompt、contract、executor、memory/tool policy 任一不同则生成不同配置。
  - 无法判定时使用 `nodecfg.{graph_id}.{node_id}`，并在目录中标记需要人工整理。
- 图节点增加 `node_config_id` 引用。
- 节点配置页实现目录、详情、契约绑定、能力权限、runtime 预览。
- 节点配置页同时读取 `/orchestration/agents` 和 `/orchestration/capability-items`，agent/runtime profile 只作为候选引用，不作为节点配置源数据。
- runtime preview 必须调用后端合成结果，不能由前端拼接推测最终权限。
- 旧只读节点配置页删除。

验收：

- 节点配置可保存。
- 图节点可引用节点配置。
- 节点配置不管理 agent 注册，只引用 agent/runtime profile。
- runtime preview 能展示有效 prompt、资源、契约、权限。

### 阶段 5：任务图关系收束

目标：任务系统保留任务图的归属和健康视图，主工作台继续负责拓扑编辑。

工作：

- 环境任务图页展示图绑定、发布态、入口/出口、问题、契约绑定摘要。
- 图拓扑编辑仍在主页面任务图工作台。
- 任务系统中不再出现“任务图编辑台”旧入口或跨页面跳转按钮。

验收：

- 任务系统能回答“这个环境有哪些任务图、是否健康、引用了哪些契约和节点配置”。
- 不把拓扑编辑重新混入任务系统。

### 阶段 6：旧链路删除与验证

目标：删除旧 UI 和旧同步逻辑，避免新旧并行。

工作：

- 删除旧契约长表单入口。
- 删除旧节点只读页面。
- 删除任务图保存链路中把 runtime/contract 主字段藏回 metadata 的旧兜底。
- 删除只为旧 `TaskGraphNodeRecord` 字段编辑服务的前端表单分支；保留编译输出，不保留旧编辑入口。
- 清理旧 CSS 和旧测试中只保护旧结构的部分。
- 新增/更新回归测试。

验收：

- 类型检查通过。
- 任务环境 API、契约保存、节点配置保存、图节点引用、runtime preview 测试通过。
- Playwright 检查环境、契约、节点三个工作台在 375/768/1440 无横向溢出。

## 切换规则

- 不保留旧契约编辑器和新契约编辑器并行入口。新编辑器完成后删除旧 `ContractLibraryPanel`。
- 不保留旧节点配置只读页。节点配置 registry 和 UI 完成后删除旧 `TaskNodeConfigurationLibraryPage`。
- 图节点旧字段只作为一次性迁移输入；迁移完成后，任务图源数据的主路径是 `node_config_id` + override，发布编译产物可以包含展开字段但不反写源数据。
- 如果迁移发现某些图节点无法归并为节点配置，必须在节点配置目录显示问题，不能静默生成默认配置。
- 切换期间不做双写。每个阶段完成后以一次性迁移脚本或 registry upsert 固化数据，再删除旧入口。
- 回滚只允许通过 git revert 或恢复迁移前配置文件备份；不在运行时保留新旧双路径选择器。

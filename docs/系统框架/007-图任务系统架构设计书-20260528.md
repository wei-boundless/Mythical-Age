本设计书先定义图任务系统本身的成熟架构，不急于实装任务环境接入。任务环境系统仍在重构时，图任务只依赖一组稳定接口：

```text
EnvironmentRuntimeProjection
FileManagementService

```

## 1. 问题定义

图任务不是固定流程模板，也不是前端画布快照。图任务的本质是：

```text
用户或任务库配置一个可自由组装的任务拓扑；
系统把该拓扑发布为 harness 可识别的不可变配置；
GraphLoop 根据配置推进节点状态；
节点通过 AgentHarness、工具、人工 gate 或子图执行；
执行事实以 checkpoint、event、artifact ref、memory receipt 形式沉淀；
恢复时只依赖已发布配置和 checkpoint，不重新编译 live graph。
```

当前需要修复的核心问题不是单点 bug，而是图语言、执行配置和调度投影的边界不清：

```text
完整图语言被压缩成可执行 DAG。
memory/resource/revision 结构在发布配置中被裁剪。
GraphLoop 容易把所有 edge 都当作调度依赖。
GraphRuntime / GraphLoop / TaskSystem compiler 的权威边界仍需进一步锁定。
```

正确目标：

```text
完整图语言是事实源。
GraphHarnessConfig 是运行期唯一静态合同。
SchedulerProjection 是从 GraphHarnessConfig 派生的调度视图，不是事实源。
GraphLoop 只推进状态，不重新编译图，不执行 agent 内环。
Agent 节点只通过 AgentHarness / AgentLoop 执行。
```

## 2. 本地代码证据

### 2.1 写作任务图的真实结构

当前 `storage/tasks/task_graphs.json` 中，模块化长篇写作任务不是一个单图，而是四个已发布图：

| graph_id | 节点数 | 边数 | 节点类型 | 边类型 |
| --- | ---: | ---: | --- | --- |
| `graph.writing.modular_novel.master` | 3 | 2 | `graph_module: 3` | `structured_handoff: 2` |
| `graph.writing.modular_novel.design_init` | 15 | 80 | `agent_role: 6`, `review_gate: 3`, `memory_commit: 3`, `memory_repository: 2`, `issue_ledger: 1` | `memory_commit: 36`, `memory_read: 26`, `structured_handoff: 15`, `revision_request: 3` |
| `graph.writing.modular_novel.chapter_cycle` | 18 | 261 | `agent_role: 7`, `memory_repository: 4`, `review_gate: 3`, `memory_commit: 3`, `issue_ledger: 1` | `memory_read: 202`, `memory_commit: 44`, `structured_handoff: 12`, `revision_request: 3` |
| `graph.writing.modular_novel.finalize` | 8 | 73 | `memory_repository: 4`, `agent_role: 1`, `review_gate: 1`, `memory_finalize: 1`, `issue_ledger: 1` | `memory_read: 58`, `memory_commit: 12`, `structured_handoff: 2`, `revision_request: 1` |

结论：

```text
写作图的核心不是普通 DAG。
memory_repository、memory_read、memory_commit、revision_request、issue_ledger 是任务语义的一部分。
这些节点和边不能被发布链路静默裁剪。
```

### 2.2 当前已发布 GraphHarnessConfig 的语义损失

当前 `storage/tasks/graph_harness_configs.json` 中写作图发布后出现明显裁剪：

| graph_id | 原始节点/边 | 已发布节点/边 | 丢失结构 |
| --- | ---: | ---: | --- |
| `design_init` | 15 / 80 | 12 / 15 | `memory_repository`、`issue_ledger` 节点和全部 `memory_read/memory_commit/revision_request` 语义边被移出主图 |
| `chapter_cycle` | 18 / 261 | 13 / 12 | `memory_repository`、`issue_ledger` 节点和大量 memory/revision 边被移出主图 |
| `finalize` | 8 / 73 | 3 / 2 | `memory_repository`、`issue_ledger` 节点和 memory/revision 边被移出主图 |

相关代码位置：

```text
backend/task_system/compiler/layered_graph_normalizer.py
backend/task_system/compiler/coordination_graph_compiler.py
backend/task_system/compiler/graph_harness_config_publisher.py
backend/harness/graph/loop.py
```

当前问题不是 normalizer 抽取 memory/resource 信息本身错误，而是发布后的 `GraphHarnessConfig.nodes/edges` 没有保留完整图语言，导致运行期只剩一个过窄的调度图。

### 2.3 当前 GraphLoop 的风险点

`backend/harness/graph/loop.py` 当前 `_upstream_node_ids()` 会把 `graph_config.edges` 中所有边都视为前置依赖。

这与目标图语言冲突：

```text
structured_handoff 可以是调度依赖。
gate/control 可以是调度依赖。
memory_read 通常是上下文装配边，不是节点完成依赖。
memory_commit 是写入/候选提交语义，不是普通下游依赖。
revision_request 是返修控制语义，不能等同普通 DAG 边。
artifact_context 是产物上下文边，不一定阻塞调度。
```

因此必须引入 `SchedulerProjection`，GraphLoop 只能用投影判断 ready，不能直接把完整边集当依赖边。

## 3. 权威边界

### 3.1 TaskSystem

负责：

```text
维护 TaskGraphDraft / CanonicalTaskGraph。
校验图语言。
发布 GraphHarnessConfig。
记录 graph_id -> published config_id 绑定。
```

不负责：

```text
运行图。
推进节点状态。
执行 agent。
在运行中重新签发节点任务。
根据 checkpoint 补配置。
```

### 3.2 GraphHarnessConfig

负责：

```text
作为图运行唯一静态合同。
保留完整图语言。
包含调度投影生成所需字段。
锁定 task_environment_id、resource scopes、node contracts、edge contracts、loop frames。
```

不负责：

```text
保存运行状态。
保存 active work orders。
保存节点结果。
```

### 3.3 GraphRuntime

负责：

```text
校验 GraphHarnessConfig status/content_hash。
锁定 config_id/config_hash。
解析任务环境运行投影。
装配 graph run 启动 envelope。
创建 GraphRuntimeEnvelope。
```

不负责：

```text
判断 ready 节点。
执行 agent。
合并节点结果。
从 TaskGraphDefinition 重新编译图。
```

### 3.4 GraphLoop

负责：

```text
初始化 GraphLoopState。
根据 SchedulerProjection 判断 ready。
生成 NodeWorkOrder。
接收 NodeResultEnvelope。
推进 node/edge/loop/artifact/memory 状态。
写 checkpoint 和 event。
判断 completed/failed/paused/waiting。
```

不负责：

```text
执行模型内环。
执行工具内环。
选择任务环境。
扩大节点权限。
重新解释 SpecificTask。
重新编译 live graph。
```

### 3.5 AgentHarness

负责：

```text
执行 agent 节点的单 agent 闭环。
每次模型调用前编译 RuntimeInvocationPacket。
调用权限和工具系统。
返回节点执行结果。
```

不负责：

```text
判断图下一个节点。
修改 GraphLoopState。
选择图任务环境。
```

### 3.6 TaskEnvironment

负责：

```text
提供文件、memory、artifact、sandbox、tool market、risk、prompt pack 的环境上限。
提供写作环境、开发环境等资源边界。
```

不负责：

```text
表达具体图步骤。
决定节点 ready。
替 agent 规划行动。
被 agent 在运行中切换。
```

节点可以收窄环境权限，不能扩大环境权限。

## 4. 目标主链

### 4.1 发布链

```text
Graph Editor Draft
-> CanonicalTaskGraph
-> GraphLanguageValidator
-> GraphHarnessConfigPublisher
-> GraphHarnessConfig(content_hash)
-> GraphHarnessConfigRepository
-> PublishedBinding(graph_id -> config_id)
```

发布链可以读取草稿图和任务系统配置。运行链不能读取草稿图。

### 4.2 启动链

```text
graph_id
-> resolve published config_id
-> load GraphHarnessConfig
-> GraphRuntime.start()
-> GraphRuntimeEnvelope
-> GraphLoop.initialize()
-> GraphLoopState
-> SchedulerProjection
-> NodeWorkOrder[]
```

启动时必须 fail closed：

```text
没有 published config：失败。
config_hash 不一致：失败。
schema 不支持：失败。
未知 node/edge 类型：失败或显式 extension handler，不得静默丢弃。
```

### 4.3 节点执行链

```text
NodeWorkOrder
-> ExecutorRouter
-> AgentHarness / ToolExecutor / HumanGate / ChildGraphHarness
-> NodeResultEnvelope
-> GraphLoop.accept_node_result()
-> GraphLoopState transition
```

`NodeResultEnvelope` 是节点完成事实的唯一入口。

### 4.4 恢复链

```text
graph_run_id
-> load GraphLoopState checkpoint
-> load GraphHarnessConfig by config_id
-> verify config_hash
-> reconnect active work orders
-> dispatch ready nodes
```

禁止：

```text
恢复时读取 TaskGraphDefinition。
恢复时重新 publish。
恢复时从 editor graph 补字段。
恢复时用 metadata fallback 改执行语义。
```

## 5. 图语言模型

### 5.1 三个视图

图任务必须同时拥有三个明确视图：

```text
CanonicalGraph
完整图语言事实源，用于编辑、校验、发布。

GraphHarnessConfig
发布后的不可变运行合同，保留完整图语言。

SchedulerProjection
GraphLoop 从 GraphHarnessConfig 派生的调度视图，只包含 ready 判断需要的控制依赖。
```

关系：

```text
CanonicalGraph -> GraphHarnessConfig -> SchedulerProjection
```

禁止：

```text
SchedulerProjection -> GraphHarnessConfig
SchedulerProjection -> CanonicalGraph
GraphLoop -> CanonicalGraph
```

### 5.2 节点类型

节点分为 executable node 和 resource node。

executable node：

```text
agent_role
review_gate
memory_commit
memory_finalize
tool
human_gate
router
barrier
graph_module
input
output
```

resource node：

```text
memory_repository
artifact_repository
file_repository
issue_ledger
progress_ledger
runtime_state_store
external_resource
```

关键原则：

```text
resource node 不一定生成 NodeWorkOrder。
resource node 仍然必须保留在 GraphHarnessConfig.nodes。
resource node 定义节点可读写的环境资源、memory 仓库、文件仓库、问题账本和产物空间。
```

### 5.3 边类型

控制/调度边：

```text
structured_handoff
control
gate_pass
gate_fail
route
barrier
revision_request
```

上下文/资源边：

```text
memory_read
artifact_context
file_read
resource_bind
issue_read
```

写入/提交边：

```text
memory_commit
artifact_commit
file_commit
issue_append
```

异步/事件边：

```text
event
signal
ack
notification
```

边必须声明：

```text
edge_id
source_node_id
target_node_id
edge_type
semantic_role
scheduler_role
payload_contract
delivery_policy
failure_policy
ack_policy
```

`scheduler_role` 建议取值：

```text
dependency
conditional_dependency
context
commit
event
audit
none
```

只有 `dependency` 和满足条件的 `conditional_dependency` 能进入 SchedulerProjection。

## 7. 节点任务分配

节点任务来自发布后的 `GraphNodeConfig`，不是运行中重新调用任务系统。

节点任务分配规则：

```text
SpecificTask 定义图任务整体目标。
GraphHarnessConfig 定义节点任务拆分。
GraphNodeConfig.task_ref 指向节点任务合同。
GraphNodeConfig.agent_binding 指向 agent/profile/mode/skill 候选。
GraphNodeConfig.prompt_contract 给 agent 明确角色和职责。
GraphNodeConfig.resource policies 收窄环境资源。
```

运行期流程：

```text
GraphLoop 选中 ready node
-> 读取 GraphNodeConfig
-> 生成 NodeWorkOrder
-> ExecutorRouter 根据 work_kind 分发
-> agent 节点进入 AgentHarness
```

禁止：

```text
节点回到 TaskSystem 重新签发 TaskContract。
节点根据 metadata 猜 agent profile。
节点运行时扩大 task environment。
GraphLoop 把 prompt 拼成“执行节点 xxx”的开发说明。
```

## 8. 节点通信

节点之间不直接调用。通信必须通过 GraphLoop 可审计地记录和投递。

### 8.1 通信载体

```text
NodeResultEnvelope.outputs
NodeResultEnvelope.handoff_payload
NodeResultEnvelope.artifact_refs
NodeResultEnvelope.memory_candidates
NodeResultEnvelope.review_receipt
NodeResultEnvelope.issue_updates
```

### 8.2 通信路径

```text
source NodeResultEnvelope
-> GraphLoop result_index / handoff_index
-> EdgeDeliveryPolicy
-> target NodeWorkOrder input_package/context_refs
-> AgentHarness RuntimeInvocationPacket
```

### 8.3 边语义

`structured_handoff`：

```text
把源节点结构化输出交给目标节点。
可参与调度依赖。
```

`memory_read`：

```text
声明目标节点调用前需要读取哪个 memory_repository。
进入 MemoryView 编译，不默认参与调度依赖。
```

`memory_commit`：

```text
声明源节点输出如何形成 memory candidate 或 commit request。
进入 memory_commit_index，不默认参与普通下游 ready 判断。
```

`revision_request`：

```text
声明 review gate 失败时回修到哪个节点。
属于条件控制边。
只在 review_decision=revise/reject 时激活。
```

`artifact_context`：

```text
声明目标节点需要展开哪些产物上下文。
进入 ArtifactView 编译，不默认参与调度依赖。
```

## 8.4 通信技术方案

节点之间的通信不使用直接函数调用，不使用共享可变对象，也不把大段上下文塞进 graph state。成熟方案应采用：

```text
Typed Envelope
+ Durable Ref
+ Context Materializer
+ Append-only Event Log
+ Idempotent Work Order
```

也就是：

```text
节点只产出 NodeResultEnvelope。
GraphLoop 根据边语义生成 HandoffPacket / MemoryReadRequest / ArtifactContextRequest。
Runtime 在节点调用前物化成 NodeInputPackage / RuntimeInvocationPacket。
Agent 只看 model-visible view。
Agent 的写入只能成为候选或请求。
系统负责验证、提交、记录 receipt。
```

### 8.4.1 协议对象

必须有以下协议对象：

```python

    model_visible_items: tuple[dict, ...]
    missing_required_records: tuple[dict, ...]

class MemoryWriteRequest:
    request_id: str
    graph_run_id: str
    node_id: str
    edge_id: str
    repository_ref: str
    collection_id: str
    candidate_payload: dict
    canonical_text: str
    artifact_refs: tuple[str, ...]
    idempotency_key: str

class MemoryWriteReceipt:
    receipt_id: str
    request_id: str
    candidate_version_id: str
    transaction_id: str
    status: str
    committed_version_id: str
    visible_after_clock: str
    visible_after_clock_seq: int
```

这些对象的职责：

```text
HandoffPacket 负责节点到节点的结构化交接。
MemoryReadRequest/Result 负责系统到记忆库的读取。
MemoryWriteRequest/Receipt 负责 agent 输出到正式记忆库的候选提交链。
ArtifactContextRequest/Packet 负责产物上下文展开。
NodeInputPackage 负责把上述材料合并成节点可消费输入。
RuntimeInvocationPacket 负责把节点输入编译成 agent 可见上下文。
```

### 8.4.2 传值策略

小对象可以内联，大对象必须用 ref：

```text
内联：节点决策、短摘要、结构化字段、状态码、schema id。
引用：长文本、文件、正式作品、memory record version、artifact、完整上游输出。
```

推荐引用：

```text
artifact_ref
memory_version_id
read_log_id
handoff_packet_id
node_result_id
file_receipt_id
commit_receipt_id
```

这样可以同时保证：

```text
高效：GraphLoopState 不膨胀。
精准：每个上下文都有来源和版本。
可恢复：checkpoint 只保存稳定引用。
可审计：event log 能还原谁读了什么、谁写了什么。
```

### 8.4.3 存储技术选择

短期可以沿用当前 repository/service 结构，但协议边界必须先稳定。

推荐技术路线：

```text
本地单机阶段：
  SQLite WAL + JSON column / msgpack payload
  用于 event_log、work_order、checkpoint、memory transaction、artifact index。

当前过渡阶段：
  现有 JSON-backed store 可以继续作为临时存储。
  但必须通过 typed envelope 和 strict validator 读写。

多人并发或服务化阶段：
  PostgreSQL 作为事实存储。
  work_order 表承担轻量任务队列。
  需要跨进程分发时再引入 Redis Streams / NATS / RabbitMQ。
```

不建议一开始直接把节点通信做成实时消息总线。图任务的核心是可恢复、可审计、可重放，优先级高于实时广播。消息队列只能作为 work order 分发加速层，不能成为事实源。

事实源应当是：

```text
GraphHarnessConfig
GraphLoopState checkpoint
GraphEventLog
NodeResultEnvelope
FormalMemoryTransaction
ArtifactRecord
```

### 8.4.4 Schema 技术

内部可以短期复用 dataclass，但持久化/API/跨模块边界建议使用严格 schema：

```text
Pydantic v2 model 或等价 strict validator
JSON Schema export 给前端图编辑器
content_hash 用 canonical JSON
所有 envelope 支持 from_dict/to_dict
未知字段进入 extension/metadata，但不能参与执行权威
未知 node/edge 类型 fail closed
```

关键要求：

```text
GraphLoop 不能接受裸 dict 作为执行事实。
MemoryService 不能接受未校验的 agent 原文直接写正式库。
ArtifactService 不能接受越过 file policy 的路径。
RuntimeInvocationPacket 不能包含系统隐藏控制字段的可见副本。
```

## 8.5 Agent 与系统交互控制

agent 和系统的关系必须是明确的请求/授权/观察关系：

```text
系统给 agent 编译当前可见世界。
agent 基于可见世界作出语义判断或行动请求。
系统验证请求是否合法。
系统执行或拒绝请求。
系统把真实 observation / receipt 回灌给 agent。
```

agent 不能直接拥有：

```text
GraphLoopState 写权限。
GraphHarnessConfig 修改权限。
FormalMemoryStore 直接写权限。
ArtifactRepository 直接提交正式产物权限。
FileManagement 正式作品覆盖权限。
PermissionDecision 决策权限。
```

agent 只能产出：

```text
final output
tool_call request
delegate request
memory_write_candidate
artifact_candidate
review_decision
revision_request
ask_user request
```

系统必须产出：

```text
ObservationRecord
ToolResultEnvelope
MemoryReadResult
MemoryWriteReceipt
ArtifactReceipt
PermissionReceipt
NodeResultEnvelope
GraphEvent
```

### 8.5.1 Model-visible 与 hidden-control 分离

RuntimeInvocationPacket 必须区分：

```text
model_visible_context
hidden_control_refs
```

model_visible_context 可以包含：

```text
任务说明
角色职责
输入材料
memory 摘要或被选择记录
artifact 摘要或展开文本
可用工具说明
输出契约
失败处理规则
```

hidden_control_refs 可以包含：

```text
graph_run_id
node_id
config_id
config_hash
work_order_id
idempotency_key
permission_scope_ref
checkpoint_ref
```

规则：

```text
agent 可以知道自己在执行什么任务，但不能被要求维护系统控制状态。
agent 可以引用 artifact/memory ref，但不能伪造 receipt。
agent 输出中的控制字段只作为请求，不能直接成为系统事实。
```

### 8.5.2 Action Request 协议

agent 对系统的所有动作都必须是显式请求：

```python
class AgentActionRequest:
    request_id: str
    graph_run_id: str
    node_id: str
    work_order_id: str
    action_type: str
    payload: dict
    requested_permissions: tuple[str, ...]
    idempotency_key: str
```

系统处理链：

```text
AgentActionRequest
-> schema validation
-> PermissionDecision
-> service execution
-> receipt / observation
-> RuntimeInvocationPacket next turn
```

禁止：

```text
agent 输出 JSON 后系统直接当事实写入。
agent 声称已提交 memory，系统就标记 committed。
agent 声称某节点完成，GraphLoop 就推进状态。
agent 自己选择下一个图节点。
```

## 8.6 记忆读写的精确流程

### 8.6.1 读取流程

```text
GraphLoop 准备调度节点
-> 收集指向该节点的 memory_read edges
-> 生成 MemoryReadRequest
-> FormalMemoryService.select_for_node()
-> 生成 FormalMemoryReadLog
-> 返回 MemoryReadResult
-> ContextMaterializer 生成 MemoryView
-> RuntimeInvocationPacket 给 agent 可见摘要/记录
```

读取必须满足：

```text
按 repository / collection / selector 定向读取。
按 clock / visible_after_clock 做版本可见性控制。
每次读取写 FormalMemoryReadLog。
missing required records 按 edge.on_missing 决定 block/fail_closed/warn/ignore。
```

现有 `FormalMemoryService.select_for_node()` 已经具备接近目标的基础：

```text
selector
version_selector
read_log
missing_required_records
content_requirement
visible_after_clock
```

后续应把它接入 GraphLoop 的 MemoryReadRequest，而不是让 node protocol 自己拼 memory_snapshot。

### 8.6.2 写入流程

agent 不写正式 memory，只提交候选：

```text
Agent output
-> NodeResultEnvelope.memory_candidates
-> GraphLoop 根据 memory_commit edge 生成 MemoryWriteRequest
-> FormalMemoryService.write_candidate_from_edge()
-> FormalMemoryTransaction(status=candidate)
-> memory_commit/review_gate 节点裁决
-> FormalMemoryService.commit_from_edge()
-> MemoryWriteReceipt
-> GraphLoop 更新 memory_commit_index
```

写入必须满足：

```text
候选版本和正式提交分离。
commit 必须有 edge_id、node_run_id、idempotency_key。
正式版本有 visible_after_clock，避免同一节点读到自己未批准写入。
content_requirement 不满足时 fail closed。
review gate 不通过时不能提交正式 memory。
```

现有 `FormalMemoryService.write_candidate_from_edge()` 和 `commit_from_edge()` 可以作为基础，但需要收束成图协议：

```text
MemoryWriteRequest -> write_candidate_from_edge
MemoryCommitDecision -> commit_from_edge
MemoryWriteReceipt -> GraphLoopState.memory_commit_index
```

### 8.6.3 记忆可见性

记忆可见性建议采用轻量 MVCC 思路：

```text
candidate：候选版本，不对普通下游节点可见。
committed：已提交版本。
visible_after_clock：提交后从哪个图时钟/循环轮次开始可见。
read_log：记录节点实际读取过哪些版本。
```

这样可以避免：

```text
节点读取未审核记忆。
同一轮循环读到自己刚生成但尚未 gate 的内容。
恢复后重新选择到不同记忆版本。
```

## 8.7 上下文物化器

GraphLoop 不应该自己拼 prompt。需要一个独立的 `GraphContextMaterializer`：

```text
GraphHarnessConfig
+ GraphLoopState
+ NodeWorkOrder
+ HandoffPacket refs
+ MemoryReadResult refs
+ ArtifactContextPacket refs
+ RevisionPacket refs
-> NodeInputPackage
-> RuntimeAssembly request
-> RuntimeInvocationPacket
```

职责：

```text
解析边语义。
读取 refs。
裁剪上下文。
排序输入材料。
生成 model-visible sections。
把 hidden refs 放入 hidden_control_refs。
检查 token/context budget。
记录 materialization receipt。
```

ContextMaterializer 不负责：

```text
决定节点是否 ready。
执行工具。
提交 memory。
修改 GraphLoopState。
```

## 8.8 高效与精准的工程原则

高效：

```text
GraphLoopState 只存状态和 refs，不存长文本。
大内容进 artifact/memory repository。
节点输入按边 selector 定向读取。
MemoryView 做预算裁剪。
artifact 文本按需展开。
read/write 都有 idempotency_key。
```

精准：

```text
每个输入都有 source_edge_id / source_node_id / read_log_id。
每个输出都有 result_id / content_hash。
每次 memory 读取有 selected_version_ids。
每次 memory 写入有 transaction_id / receipt_id。
每次 artifact materialization 有 artifact_ref / content_hash。
每次权限判断有 PermissionReceipt。
```

agent/系统边界：

```text
agent 只能请求和产出候选。
系统负责授权、执行、提交、回执。
系统不把隐藏控制权交给 prompt。
GraphLoop 不相信 agent 自述的系统状态。
RuntimeInvocationPacket 是唯一 agent 可见世界入口。
```

## 9. 时序推进

GraphLoop 的推进算法：

```text
1. 从 GraphHarnessConfig 派生 SchedulerView。
2. 从 GraphLoopState 读取 node/edge/loop 状态。
3. 计算 ready nodes。
4. 根据 max_active_nodes / async policy / resource lock 选择 dispatch 集合。
5. 为每个节点生成 NodeWorkOrder。
6. 将节点置为 running/waiting。
7. 接收 NodeResultEnvelope。
8. 校验 work_order_id/idempotency_key/node_id/config_hash。
9. 更新 result_index、node_state、edge_state、artifact_index、memory_commit_index。
10. 根据 loop frame / route / gate 计算下一轮 ready。
11. 判断 completed/failed/paused/waiting。
12. 写 checkpoint 和 event。
```

ready 判断只允许消费：

```text
SchedulerView
GraphLoopState
NodeResultEnvelope.decisions
LoopFrameState
Human/approval state
```

禁止消费：

```text
raw editor graph
TaskGraphDefinition
monitor projection
metadata fallback
```

## 12. 记忆库关系

记忆库是图语言中的一级资源，不是隐藏 metadata。

### 12.1 memory_repository 节点

```python
class MemoryRepositoryNode:
    node_id: str
    repository_ref: str
    memory_scope: str
    read_policy: dict
    write_policy: dict
    commit_policy: dict
    visibility_policy: dict
```

### 12.2 memory_read 边

```text
目标节点运行前，Runtime 根据 edge policy 构建 MemoryView。
读取失败按 fail_closed/warn/ignore 处理。
读取结果进入 NodeWorkOrder.memory_view_request，不直接塞进 GraphLoop 推理。
```

```text

```


## 15. 子图与 graph_module

`graph_module` 是节点 executor 类型，不是第三种 runtime。


## 17. 图编辑器设计原则

图编辑器不是配置语言的来源。配置语言来自 Harness/Loop/Runtime 的执行需求，图编辑器只是图化编辑界面。

```text

4. 保存 Draft。
5. 发布为 GraphHarnessConfig。
6. 运行 published config。
```

图编辑器必须支持：

```text

所有 edge_type。
resource node。
memory repository。
artifact repository。
issue ledger。

graph module。
async policy。

prompt contract。

```

```text

```

## 21. 最终定义

图任务系统的成熟定义：

```text
图任务是由完整图语言表达的固定任务合同；
GraphHarnessConfig 是该合同发布后的不可变运行配置；
GraphRuntime 装配运行环境；
GraphLoop 基于派生调度视图推进图状态；
节点通过 AgentHarness、工具、人工 gate 或子图执行；
memory、artifact、file、issue、revision 都是图语言的一等语义；
断点恢复只依赖 config_id、config_hash 和 checkpoint；
图编辑器只是配置语言的图化编辑界面，不是运行期事实源。
```

一句话：

```text
完整图语言是事实源，GraphHarnessConfig 是执行合同，SchedulerView 是调度派生物，GraphLoop 是状态推进器，AgentHarness 是 agent 节点执行器。
```

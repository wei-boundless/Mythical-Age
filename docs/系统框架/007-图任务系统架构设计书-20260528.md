# 图任务系统架构设计书

日期：2026-05-28  
重写日期：2026-05-30  
状态：框架设计重定版  
依据：外部任务图设计文档、当前代码证据、写作任务图设计原则、任务系统与 harness 边界原则。

本文定义“任务图系统”本身，不定义某次重构步骤。后续重构计划必须先对齐本文，再写实施计划。

## 0. 本次纠偏结论

上一版文档错误地把图任务压缩成：

```text
GraphHarnessConfig + GraphLoop + A2A 通信
```

这是不完整的。A2A 是节点通信协议层，GraphHarness 是运行控制层，它们都不是任务图系统的全部。

任务图的正确主语是：

```text
TaskGraphDefinition
```

任务图系统的核心不是“运行期怎么把一堆消息传过去”，而是把用户可编辑的图语言、标准化契约语言、发布执行包、运行调度、节点执行、记忆产物沉积、监控恢复连成同一条可追溯链路：

```text
TaskGraphDefinition
  -> LayeredGraph
  -> StandardView / ComposableGraphView / FlowView
  -> ContractManifest
  -> RuntimeSpec / GraphHarnessConfig
  -> ExecutionPackage / FlowPackage
  -> SchedulerState / GraphLoopState
  -> GraphHarness Loop
  -> Node WorkOrder
  -> AgentHarness / Tool / HumanGate / ResourceService
  -> FlowPacket / Receipt / ArtifactRef / MemoryRef
  -> TimelineLedger / Checkpoint / Monitor
  -> downstream ready / graph composition resume / merge
```

因此本文采用两个判断：

1. 图任务系统是任务系统的一等编排对象，不是 harness 的附属配置。
2. harness 只消费编译后的运行合同并推进状态，不能反过来定义任务图语言。

## 1. 任务图是什么

任务图是长任务的结构化运行定义。它不是一次运行实例，不是前端画布快照，不是业务任务详情的子对象，也不是 LangGraph 图。

任务图负责表达：

- 哪些执行单元存在。
- 它们的输入输出接口是什么。
- 它们按什么边交接。
- 哪些边是控制依赖、记忆读写、产物上下文、审核返修、时序闸门。
- 哪些结果必须审核后才能提交。
- 哪些产物和记忆对下游可见。
- 哪些节点可以并行、批次化、循环或人工门控。
- 哪些流程可以封装成可组合图单元，并继续通过端口和边参与拓扑组合。

源码事实源是：

```text
backend/task_system/graphs/task_graph_models.py
  TaskGraphDefinition
  TaskGraphNodeDefinition
  TaskGraphEdgeDefinition
```

其中 `TaskGraphDefinition.authority = task_system.task_graph_definition`，这说明保存态任务图的权威归任务系统，不归 harness。

## 2. 为什么需要任务图

单 Agent 长任务只能靠一个 agent 自己维持计划、上下文、产物和恢复边界。任务越长，越容易出现四类问题：

- 上下文膨胀：下游节点收到整段历史，而不是它应读的包。
- 状态污染：候选稿、审核意见、提交结果混成一团。
- 决策权混乱：运行时根据节点名或任务族补私门逻辑。
- 恢复困难：无法判断当前运行卡在哪个节点、哪条边、哪个批次、哪个提交。

任务图存在的目的就是把长任务拆成可审计的控制工程：

```text
节点负责执行一个职责位。
边负责交接一个契约化 packet。
记忆库和产物库负责沉积结果。
时序账本负责记录状态推进。
调度器负责判断下一步谁可运行。
agent 只处理当前节点被授权的工作包。
```

写作任务只是一个实例。视频任务、代码迁移、资料整理、批量测试、知识库建设都应使用同一套图任务能力。

## 3. 核心对象模型

### 3.1 Graph

`Graph` 是一张可保存、可发布、可运行、可复用的任务流定义。

它包含：

```text
graph_id
domain_id
entry_node_id
output_node_id
nodes
edges
contract_bindings
default_protocol_id
runtime_policy
context_policy
working_memory_policy
loop_frames
metadata
publish_state
```

图属于任务环境或任务域下的任务图库，但不属于某个 `SpecificTaskRecord`。具体任务定义可以被节点引用，但不是图的父对象。

### 3.2 Node / FlowUnit

节点不是任务本身。节点是图里的执行位置、职责位、时序位。

成熟模型应把节点推广为 `FlowUnit`：

```text
FlowUnit =
  atomic_node
  graph_ref
  flow_fragment
  dynamic_split
  merge_gate
  reservoir
  human_gate
  tool_unit
  monitor_unit
```

当前代码里的对应关系：

```text
TaskGraphNodeDefinition         -> atomic_node / resource / gate
ComposableUnit                  -> 可组合单元视图
GraphModuleRuntimePlan          -> graph_ref 的运行计划
memory_repository               -> reservoir
review_gate / manual_gate       -> human_gate
runtime_monitor                 -> monitor_unit
```

节点必须声明自己的职责、Agent/工具/人工执行引用、输入输出、记忆读取、产物策略、验收策略、运行策略。节点不能靠后端根据名字补能力。

### 3.3 Encapsulated Graph Unit

图可以被封装成可组合单元，然后和其它封装图、原子节点、资源节点一起组成新的拓扑。

正确语义：

```text
EncapsulatedGraphUnit = FlowUnit 类型 graph_ref
```

封装图不是上下级归属对象，而是一个暴露接口的组合单元；它和其它单元之间通过端口、边契约、packet、scope 和 receipt 连接。

组合拓扑只看到封装图的公开接口：

- 输入端口。
- 输出端口。
- linked graph id / version ref。
- handoff contract。
- isolation policy。
- committed output packet。
- artifact refs / memory refs。
- 封装图运行状态摘要。

外部组合拓扑不能直接读取封装图内部候选稿、临时上下文、内部节点状态或未提交记忆。封装图内部细节只能通过它声明的输出端口、提交包、产物引用、记忆引用和监控投影暴露。

### 3.4 Edge / FlowEdge

边不是线，也不是执行端。边是交接协议和调度约束。

成熟模型应把边推广为 `FlowEdge`：

```text
FlowEdge =
  handoff
  split
  merge
  memory_read
  memory_commit
  artifact_context
  revision_feedback
  temporal_gate
  event_trigger
```

边必须说明：

```text
source_unit / source_port
target_unit / target_port
packet_contract_id
handoff_policy
temporal_policy
visibility_policy
failure_policy
merge_policy
split_policy
a2a_message_type
contract_bindings
```

重要原则：

```text
传递策略来自边契约。
不是 context materializer 默认生成 summary_and_refs。
不是 GraphLoop 自行决定给下游塞什么。
```

当前代码已有 `TaskGraphEdgeDefinition.result_delivery_policy`，取值包括 `contract_payload_and_refs`、`refs_only`、`summary_and_refs`、`notification_only`。这类字段必须由边契约控制。

### 3.5 Port / Interface

图和节点统一为可组合单元后，连接点不应继续停留在：

```text
source_node_id -> target_node_id
```

长期目标是：

```text
source_unit_id.output_port_id -> target_unit_id.input_port_id
```

当前代码已有：

```text
backend/task_system/graphs/composable_graph_models.py
  UnitPort
  UnitInterface
  UnitPortEdge
```

这说明“图即节点、节点可展开为图”的方向已经有模型基础，但还需要收敛为标准编辑与发布事实，而不能长期停留在 shadow / overlay。

### 3.6 FlowPacket

FlowPacket 是图任务中真正流动的东西。

它不是“整张图上下文”，也不是“上游全文拼接”。它应该是按边契约生成的可追踪传输单元。

标准字段：

```text
packet_id
packet_type
source_unit_id
target_unit_id
source_port_id
target_port_id
edge_id
scope_id
contract_id
a2a_message_type
payload_summary
payload_refs
artifact_refs
memory_refs
result_refs
visibility
status
lineage
created_at
```

关键原则：

- 默认传引用，不复制正文。
- 模型可见内容由边契约、节点契约、记忆可见性、产物可见性共同决定。
- 大正文进入 artifact/file repository。
- 记忆事实进入 memory repository。
- 审核裁决进入 receipt / ledger。
- 未审核 candidate 不得外溢为 committed packet。

### 3.7 FlowScope

FlowScope 定义一段任务流的上下文边界。

它回答：

- 当前节点属于哪个 graph run。
- 是否位于封装图组合单元。
- 是否位于某个批次。
- 是否位于某次循环。
- 能读取哪些记忆版本。
- 能看到哪些产物引用。
- 能写入哪些仓库。

封装图单元、批次、循环、人工门、动态分流都必须有清晰 scope。

### 3.8 FlowLedger / TimelineLedger

TimelineLedger 是任务语义时序账本。checkpoint 是物理恢复点，不能替代语义时序。

必须记录：

```text
scheduler_evaluated
node_dispatch_requested
node_result_received
receipt_accepted
receipt_rejected
artifact_materialized
memory_candidate_written
memory_committed
revision_requested
batch_committed
merge_ready
checkpoint_linked
human_gate_opened/resumed
```

旧文档 `120-TaskGraph时序主轴...` 的判断是正确的：

```text
Scheduler 判断谁能跑。
TimelineLedger 记录真实发生了什么。
LangGraph / checkpoint 只提供物理恢复底座。
Context resolver 按 clock/scope 装配可见上下文。
Receipt / Commit 决定输出何时对下游可见。
```

### 3.9 FlowReservoir

Reservoir 是任务流沉积层，包括：

- baseline memory。
- dynamic memory。
- artifact repository。
- issue ledger。
- thread ledger。
- progress ledger。
- runtime state store。

写作任务里的记忆分层原则应上升为通用原则：

```text
candidate 不等于 committed。
review 不等于 commit。
dynamic memory 不得反向污染 baseline。
artifact index 不承担 canon 存储职责。
提交必须有来源、版本、审核依据和可见性规则。
```

## 4. 层级链路

### 4.1 保存态：TaskGraphDefinition

保存态是任务系统的主事实源。

它保存：

- 图身份。
- 节点。
- 边。
- 图级契约。
- 节点契约。
- 边契约。
- 任务环境引用。
- 通信协议引用。
- 时序、循环、批次、资源节点。
- 可组合图 metadata。

保存态不能保存运行中状态。

### 4.2 分层态：LayeredGraph

LayeredGraph 从保存态抽取不同职责层。

当前代码：

```text
backend/task_system/compiler/layered_graph_normalizer.py
```

它应抽取：

- execution layer。
- timeline layer。
- memory layer。
- artifact context layer。
- revision layer。
- resource layer。
- loop layer。
- graph composition layer。

LayeredGraph 是编译中间层，不是用户保存事实源。

### 4.3 标准态：StandardView / ComposableGraphView / FlowView

标准态服务前端编辑、发布预检、对象追溯。

当前代码：

```text
backend/task_system/graphs/task_graph_standard_models.py
backend/task_system/graphs/composable_graph_builder.py
backend/task_system/graphs/composable_graph_models.py
```

标准态必须能展示：

- graph。
- nodes。
- edges。
- resources。
- units。
- interfaces。
- port_edges。
- graph_module_runtime。
- graph_module_expansions。
- timeline。
- runtime_isolation。
- memory_matrix。
- memory_protocol。
- diagnostics。

`graph_module_expansions` 只能作为标准视图和追溯视图，不能被误解为运行期反复建图的理由。运行期必须消费已发布的运行合同。

### 4.4 契约态：ContractManifest

契约不是 schema 小工具，而是图任务运行协议。

统一分类：

```text
schema
execution
unit_batch
artifact
memory
handoff
acceptance
runtime
governance
temporal
```

源码已有 `CONTRACT_BINDING_SECTIONS` 支撑这些分类。

Manifest 负责：

- 归一化图、节点、边上的契约。
- 检查旧字段与 `contract_bindings` 冲突。
- 检查缺失、越权、类型错配。
- 给运行层提供唯一契约来源。

运行时不能根据任务族、节点名、契约 id 前缀补隐藏行为。

### 4.5 运行合同态：RuntimeSpec / GraphHarnessConfig

RuntimeSpec 和 GraphHarnessConfig 是 harness 可消费的运行投影。

当前新 harness 使用：

```text
backend/task_system/compiler/graph_harness_config_publisher.py
backend/harness/graph/models.py
```

它应锁定：

- nodes。
- edges。
- loop frames。
- resources。
- contracts manifest。
- task environment。
- tools / permissions / agents 引用。
- communication protocol id。
- source refs。
- content hash。

GraphHarnessConfig 是静态合同，不是运行状态。它可以包含编译后的拓扑和契约，但不能保存完整节点输出正文、完整对话历史或运行中临时选择。

### 4.6 发布执行态：ExecutionPackage / FlowPackage

ExecutionPackage 是发布前真实执行事实源。

它应聚合：

- StandardView。
- ContractManifest。
- RuntimeSpec / GraphHarnessConfig。
- SchedulerView。
- node runtime assembly preview。
- graph units / flow units。
- split plans。
- batch lifecycle plan。
- object trace index。
- issues。

目标是让用户能回答：

```text
我在编辑器里配置的对象，最终进入了哪个运行节点？
哪条边会生成哪个 packet？
哪个节点可读哪个 memory reservoir？
哪个 batch 会如何 split / review / commit / merge？
哪条配置缺失导致发布失败？
```

### 4.7 运行态：GraphLoopState / SchedulerState

运行态保存的是状态，不是配置。

它包含：

- node states。
- edge states。
- active work orders。
- result index。
- ready / blocked / running / completed / failed。
- loop state。
- checkpoint refs。
- diagnostics。

GraphLoop 只负责推进这些状态。

## 5. 编辑器如何设计

图编辑器不是生成另一种语言。图编辑器是 `TaskGraphDefinition` 的图形化编辑界面。

正确关系：

```text
Graphical Editor
  编辑节点、边、端口、资源、时序、契约、协议、布局。

TaskGraphDefinition
  保存唯一事实。

Compiler
  从 TaskGraphDefinition 编译出标准视图、契约清单、运行合同、执行包。

GraphHarness
  只消费发布后的运行合同。
```

因此“图语言”和“配置语言”不是两套语言。

更准确地说：

```text
配置语言来自任务图/harness 运行所需的标准对象和契约字段。
图语言是这些字段的可视化编辑投影。
编辑器负责把用户构图动作落回 TaskGraphDefinition。
发布器负责把 TaskGraphDefinition 编译成 harness 可识别运行合同。
```

编辑器必须支持：

- Graph 对象。
- Node / FlowUnit。
- EncapsulatedGraphUnit / graph_ref。
- ResourceUnit / reservoir。
- Interface / Port。
- PortEdge / FlowEdge。
- Timeline / LoopFrame。
- Contract bindings。
- Protocol bindings。
- ExecutionPackage preview。
- Object trace。

编辑器不允许：

- 保存一份和 TaskGraphDefinition 并列的隐藏图事实。
- 把 layout 当运行事实。
- 让前端字段看似生效但后端运行不消费。
- 把封装图内部节点和外部组合拓扑混成一张无边界大图。

## 6. harness 边界

### 6.1 GraphRuntime

GraphRuntime 负责启动图运行。

它应该：

- 验证 GraphHarnessConfig 已发布且 content hash 正确。
- 创建 graph run / task run / runtime envelope。
- 建立 task environment runtime scope。
- 同步 memory/artifact effective repository。
- 初始化 checkpoint。

它不应该：

- 重新解释编辑器图语言。
- 动态补节点。
- 决定节点交接内容。
- 执行 Agent 内环。

### 6.2 GraphLoop

GraphLoop 是动态图状态推进器。

它应该：

- 初始化 GraphLoopState。
- 调用 SchedulerView 判断 ready / blocked。
- 生成 GraphNodeWorkOrder。
- 接收 NodeResultEnvelope。
- 更新节点状态、边状态、checkpoint、monitor event。
- 按 loop/revision/batch/merge 规则推进状态。

它不应该：

- 生成 prompt。
- 根据节点名猜业务行为。
- 读取完整 memory 正文。
- 写完整 artifact 正文。
- 随意摘要整张图给下游节点。

### 6.3 ContextMaterializer

ContextMaterializer 负责把当前节点 work order 的上下文装配成可执行包。

它应该：

- 读取 GraphHarnessConfig。
- 读取 GraphLoopState。
- 按入边契约生成 inbound packets。
- 按 memory/artifact/file/issue policy 生成引用请求。
- 只装配当前节点需要的可见上下文。

它不应该：

- 自己决定边传递策略。
- 默认把上游结果总结成 summary_and_refs。
- 忽略边上的 `result_delivery_policy`。
- 绕过 memory/artifact repository 直接塞大正文。

### 6.4 WorkOrderExecutor

WorkOrderExecutor 负责把节点工作单交给执行端：

- AgentHarness。
- tool executor。
- human gate。
- memory commit service。
- artifact materializer。
- resource service。

它不应该成为新的业务运行时，也不应该根据写作、代码、检索等领域写私门。

### 6.5 AgentHarness

AgentHarness 只负责单 agent 执行链条。

它接收一个 RuntimeInvocationPacket，并通过 agent loop 推动模型、工具、观察、后续调用。

图任务中的节点如果需要 agent 执行，GraphHarness 只能把节点 work order 编译为 AgentHarness 可识别的任务执行包。AgentHarness 不应该知道整张图，也不应该选择任务图路径。

## 7. 通信协议与 A2A 的位置

A2A 是节点通信协议层，不是任务图系统本身。

它的正确位置：

```text
FlowEdge / handoff contract
  -> FlowPacket
  -> A2A Message / Task / Artifact representation
  -> downstream WorkOrder inbound packet
```

当前代码证据：

```text
backend/agent_system/a2a/official_adapter.py
backend/agent_system/assembly/__init__.py
storage/tasks/task_communication_protocols.json
backend/task_system/runtime_semantics/protocol_boundary.py
```

A2A 应用于：

- `message/send`
- `message/stream`
- `task/status`
- `task/artifact`
- `task/review_feedback`
- `task/revision_request`
- `task/memory_read`
- `task/memory_write`

但通信内容必须由边契约决定。

禁止：

- 把 A2A 当成图任务主架构。
- 所有边都强行塞成同一种 A2A 摘要消息。
- 让 A2A adapter 自行决定 memory 可见性或 artifact 展开策略。

## 8. 批次、分流、汇流

批次能力是通用图任务能力，不是写作专用逻辑。

当前文档和代码已经形成方向：

```text
contract_bindings.unit_batch
contract_bindings.runtime.split_policy
contract_bindings.runtime.batch_acceptance_policy
contract_bindings.runtime.merge_policy
```

运行对象：

- split plan。
- batch lifecycle plan。
- batch state。
- batch execution instance。
- merge readiness。

原则：

- batch 是运行时一等对象。
- 每个 batch 必须有 execution identity。
- 结果归位必须匹配 request / dispatch / execution identity。
- merge 只消费 committed batch packet。
- agent 不能在 prompt 里自行决定批次数量。
- 写作里的“章”只是 unit_kind 的一种实例。

长期目标是支持：

- sequential batch。
- parallel batch。
- dynamic split。
- merge gate。
- worker pool。
- idempotent resume。

## 9. 记忆与产物交接

任务图必须通过文件管理、记忆管理、产物管理来承载大内容，不允许靠上下文堆叠。

标准交接包：

- continuity_packet。
- revision_packet。
- constraint_packet。
- thread_packet。
- artifact_context_packet。
- memory_snapshot。

写作旧文档中的原则应成为通用约束：

```text
下游只消费 accepted / committed。
candidate 和 review 只能用于本轮返修。
基准库冲突优先级高于动态库。
章节批次交接必须包含单单位引用，不能只给笼统批次名。
不把聊天历史当主事实源。
不让摘要替代事实。
不让未审核内容进入下游记忆。
```

memory read/write/commit 应由图上的 memory edge 和 memory contract 控制，不是权限表孤立决定，也不是 agent 自由写入。

## 10. 监控与恢复

图任务运行必须全程可监视。

用户侧应看到一个图任务运行实例；节点、批次、封装图运行实例都进入该图任务的内部监控视图。

监控对象：

- graph run。
- node state。
- edge state。
- active work order。
- packet refs。
- batch lifecycle。
- encapsulated graph run。
- memory commit receipt。
- artifact refs。
- issue ledger。
- checkpoint refs。

恢复原则：

- checkpoint 提供物理恢复。
- TimelineLedger 提供语义恢复。
- Receipt 决定输出是否可见。
- WorkOrder idempotency 防止重复副作用。
- unknown execution identity 必须拒绝归位，不能猜当前 active node 或 active batch。

## 11. 写作任务如何落到该框架

写作任务不是特殊后端逻辑，而是一组任务图、契约和协议的实例。

它应包含：

- 组合编排图。
- 设计初始化图。
- 章节循环图。
- 收尾交付图。
- 写作通信协议。
- 记忆库和产物库资源节点。
- 章节批次契约。
- 审核、返修、提交、路由节点。

写作节点的 prompt 必须写给 agent，而不是写开发说明。

错误写法：

```text
这是 runtime 节点。
根据任务图执行 world_review。
```

正确写法：

```text
你是一名世界观审核员。
你只负责评审当前世界观设定是否完整、一致、可支撑后续写作。
你不负责替创作者扩写设定。
你需要指出问题、给出裁决、说明是否允许进入下一阶段。
```

写作图的通用闭环：

```text
生产候选
  -> 审核裁决
  -> 提交入库
  -> 路由下一步
```

这不是写作私门，而是任何长任务都需要的 candidate / review / commit / route 闭环。

## 12. 权威边界表

| 层 | 权威 | 能做什么 | 禁止做什么 |
| --- | --- | --- | --- |
| TaskSystem | TaskGraphDefinition | 保存图、节点、边、契约、协议、环境引用 | 运行节点、推进状态 |
| Editor | TaskGraphDefinition 投影 | 图形化编辑任务图事实 | 保存隐藏事实源 |
| Compiler | LayeredGraph / Manifest / Config | 归一化、诊断、发布运行合同 | 执行任务、补领域私门 |
| ExecutionPackage | 发布事实聚合 | 预检、追溯、展示运行包 | 成为运行状态 |
| GraphRuntime | Runtime envelope | 启动图运行、锁定配置 | 改图、补图 |
| SchedulerView | readiness projection | 判断 ready / blocked | 分配语义时间、读写 memory |
| GraphLoop | GraphLoopState | 推进状态、派发 work order、接收 result | 生成 prompt、决定业务内容 |
| ContextMaterializer | WorkOrder packet | 按契约装配当前节点上下文 | 默认整图摘要、绕过边契约 |
| AgentHarness | RuntimeInvocationPacket | 执行单 agent 内环 | 选择任务图路径 |
| Memory/Artifact/File | Repository | 保存正文、记忆、产物、版本 | 替调度器决定下一节点 |
| A2A | Message protocol | 表示节点间消息/状态/产物 | 定义任务图架构 |

## 13. 当前实现风险点

以下不是本文的实施计划，但后续计划必须处理：

1. `GraphHarnessConfig` 当前会通过 `_expand_composition_sources` 展开组合图。必须明确这是发布期编译行为，不是运行期反复建图。
2. `ContextMaterializer` 必须严格消费边契约，禁止默认摘要策略覆盖 `result_delivery_policy`。
3. `GraphLoop` 当前只有基本 ready/complete 状态推进，批次、packet、receipt、ledger、封装图组合运行的完整闭环还需要和新 harness 对齐。
4. `A2A` adapter 存在，但还没有成为 graph handoff 的严格执行协议。
5. `ComposableGraphView` 仍带 shadow / overlay 痕迹，说明编辑器与保存态还未完全统一。
6. 写作图配置里存在大量节点名判断脚本逻辑，后续要迁移成契约、协议、能力装配，而不是进入 harness。

## 14. 后续重构顺序

后续重构计划必须按这个顺序写：

1. 锁定任务图标准对象和字段：Graph / FlowUnit / FlowEdge / Port / Interface / FlowPacket / Scope / Ledger / Reservoir。
2. 锁定编辑器保存语言：所有图形化编辑都落回 `TaskGraphDefinition`。
3. 锁定编译链：Definition -> LayeredGraph -> StandardView -> Manifest -> GraphHarnessConfig -> ExecutionPackage。
4. 锁定 GraphHarness 运行链：Runtime -> Loop -> WorkOrder -> NodeResult -> Packet/Receipt -> State。
5. 对接 AgentHarness：单节点 work order 编译为单 agent RuntimeInvocationPacket。
6. 对接 A2A：边契约生成 FlowPacket，再投影为 A2A message/task/artifact。
7. 对接 memory/artifact/file：大内容只通过 repository refs 流动。
8. 对接批次与封装图组合：batch、encapsulated graph run、merge 都成为可监控、可恢复运行对象。
9. 逆推编辑器：发布前必须能看到执行包和对象追溯。
10. 最后清理旧字段、旧测试、旧兼容路径。

## 15. 设计红线

后续任何代码实现不得违反以下红线：

- 不能把 harness 当任务图事实源。
- 不能把图编辑器和运行配置写成两套语言。
- 不能让运行时根据节点名、任务族、契约名前缀补领域逻辑。
- 不能把整张图上下文灌给单节点。
- 不能让未审核 candidate 进入 committed memory。
- 不能让外部组合拓扑直接读取封装图内部临时状态。
- 不能把 A2A 当成图任务系统本体。
- 不能用 summary 替代边契约声明的传递策略。
- 不能用 checkpoint 替代 TimelineLedger。
- 不能保留已被新主链替代的旧链路作为兼容兜底。

一句话收口：

```text
任务图系统 = 任务系统保存的可组合任务流事实 + 契约化编译链 + harness 状态推进 + agent/tool/human/resource 执行端 + packet/ledger/repository 化交接与恢复。
```

# 产物系统治理与运行产物优化方案

日期：2026-05-30

状态：重构前方案稿

## 1. 结论

当前项目的产物问题不是“文件太多”本身，而是正式产物、运行事实、监控快照、诊断痕迹、缓存和调试脚本混在同一组目录里，缺少统一分类、索引、生命周期和可恢复边界。

因此正确方向不是继续给某个目录加数量限制或删除补丁，而是建立成熟的产物治理框架：

```text
ArtifactSystem 负责正式产物目录、产物索引、产物血缘和交付物检索。
RuntimeSystem 负责可恢复运行事实、事件账本、checkpoint、runtime object。
TaskSystem 负责任务定义、图配置和任务记录事实结构。
HealthSystem 负责任务记录健康管理、产物压力观察、维护预检、归档建议和受控维护收据。
DiagnosticOutput 负责调试痕迹、浏览器截图、启动日志、临时缓存，默认可重建、可清理。
```

这套方案保留用户之前确认的原则：任务记录不要迁出，任务记录管理仍归健康系统管；但健康系统不能直接变成原始产物写入者，也不能无审计删除 runtime 事实。

## 2. 成熟参照

本方案参考三类成熟原则，但不照搬外部平台：

1. OpenTelemetry 将可观测性信号分为 traces、metrics、logs、baggage，并强调不同信号可以关联同一系统活动。对应本项目：运行事件、监控投影、prompt accounting、健康风险不能混成一个大 JSON，而应通过稳定关联键关联。参考：<https://opentelemetry.io/docs/concepts/signals/>
2. OpenTelemetry Collector 的核心思想是采集、处理、导出信号，事实产生者和观测后端分离。对应本项目：HealthSystem 可以治理和汇总，不应替 RuntimeEventLog 或 ArtifactRepository 产出事实。参考：<https://opentelemetry.io/docs/reference/specification/overview/>
3. W3C PROV 将 provenance 定义为与产物生成有关的 entity、activity、agent 信息，用于判断质量、可靠性和可信度。对应本项目：正式产物必须记录 producer、contract、run、hash、scope 和派生关系。参考：<https://www.w3.org/standards/history/prov-dm/>、<https://www.w3.org/TR/2013/REC-prov-constraints-20130430/>
4. Google SRE 的监控四个黄金信号可转译为 agent 产物治理的 latency、traffic、errors、saturation：产物生成耗时、产物量和事件量、失败产物和无效产物、事件/checkpoint/cache 压力。参考：<https://sre.google/resources/book-update/monitoring-distributed-systems/>

借鉴后的本地不变量：

```text
正式产物和诊断产物分离。
原始事实和投影视图分离。
可恢复事实和可重建缓存分离。
产物血缘必须可查，不靠目录名猜测。
清理必须 dry-run、受保护、可审计，有维护收据。
性能优化优先修写入/读取结构，不靠减少功能。
```

## 4. 代码证据

### 4.1 项目目录权威

`backend/project_layout.py` 定义了核心存储根：

```text
storage/runtime_state
storage/tasks
storage/health_system
storage/knowledge
storage/task_environments
```

这说明当前代码已经存在存储边界，但没有统一的产物类别和生命周期注册表。

### 4.2 RuntimeEventLog 性能问题

`backend/runtime/shared/event_log.py` 中：

```text
RuntimeEventLog.append()
  -> next_offset(task_run_id)
  -> path.read_text(...).splitlines()
```

每次 append 新事件前都会扫描整个事件文件来计算 offset。当前最大事件文件已经达到 46 MB 级别，这会导致写入成本随历史事件线性增长。对长任务来说，这是结构性性能问题，不是通过删几个日志能根治的问题。

同一文件里的 `list_events()` 也会一次性读取整个事件文件。监控、健康系统和会话 timeline 如果频繁调用，会放大大事件文件的读取成本。

### 4.3 RuntimeObjectStore 已经具备大对象外置思路

`backend/runtime/shared/runtime_object_store.py` 使用 `rtobj:{kind}:{id}` 引用，把重 payload 放到 `runtime_objects`。这是正确方向，但当前还缺少：

```text
ref 反向索引
引用可达性检查
过期策略
归档/压缩策略
```

### 4.4 ArtifactRepository 已经具备正式产物索引雏形

`backend/artifact_system/artifact_repository_service.py` 和 `backend/artifact_system/artifact_repository_store.py` 已经支持：

```text

task_run_id
output_contract_id
producer_node_id
content_hash
artifact_ref
status
```

这正是正式产物系统应该扩展的中心，而不是另起一套“限制补丁”。

### 4.5 TaskSystem 的正式配置和调试快照混写

`backend/task_system/storage/json_store.py` 将 `storage/tasks` 作为 JSON store。当前该目录既有正式配置：

也有明显的测试/监控快照：

这说明 `storage/tasks` 已经被用作“任务配置库 + 测试快照桶”。这是当前产物口混乱的主要来源之一。

### 4.6 HealthSystem 已有维护入口，但粒度过粗

`backend/health_system/task_record_maintenance.py` 已有任务记录维护服务，并且会预检 active、failed、lineage 等保护条件。但当前 prune 仍倾向于删除 task_run、event_log、prompt accounting 记录，而不是先做产物类别治理、压缩、归档和恢复验证。

结论：健康系统继续管理任务记录维护，但需要调用更细粒度的 Artifact/Runtime governance 能力，而不是直接承担所有存储清理逻辑。

## 5. 当前问题归因

### 5.1 混淆了“正式产物”和“运行事实”

正式产物应该回答：

```text
这个任务交付了什么？
由哪个 agent / node / contract 生成？
存在哪里？
内容 hash 是什么？
能否作为后续任务输入？
是否可展示给用户？
```

运行事实应该回答：

```text
这次任务怎么运行的？
执行到哪里？
能不能恢复？
失败点是什么？
哪些事件和 checkpoint 构成事实链？
```

当前很多任务结果、monitor 快照和测试输出被直接放在 `storage/tasks` 或 `output` 下，导致正式交付物和运行诊断互相污染。

### 5.2 混淆了“事实账本”和“可重建投影”

`RuntimeEventLog`、checkpoint、prompt accounting 是事实源；monitor view、health overview、session timeline 是投影。投影不应该复制大 payload，也不应该长期保存为新的事实文件。

当前 `storage/tasks/*monitor*_latest.json` 这类文件实际上是投影快照，却被保存在任务配置目录，容易形成重复大文件。

### 5.3 混淆了“可恢复状态”和“诊断痕迹”

任务恢复依赖：

```text
TaskRun / AgentRun
RuntimeEventLog
Graph checkpoint
RuntimeObject refs
Artifact refs
Prompt accounting segment refs
```

诊断痕迹包括：

```text
local_traces
Playwright screenshots
runtime logs
old test stdout/stderr
frontend build cache
```

这两类可以关联，但生命周期不同。诊断痕迹应该默认 TTL/压缩/按需保留；恢复状态必须先做可达性和恢复验证。

### 5.4 写入结构放大了卡顿

当前最大事件文件接近 46 MB。`RuntimeEventLog.next_offset()` 每次 append 都扫描全文，导致事件越多写入越慢。成熟方案应增加 append cursor/index，而不是限制任务步数或隐藏进展。

### 5.5 缺少产物口统一注册表

目前系统知道很多目录，但没有一个统一的 `ArtifactClass` / `ArtifactPort` 注册表说明：

```text
owner
authority
root
source_of_truth
recoverability
retention
compaction_strategy
delete_protection
health_metrics
```

因此健康系统只能按目录和状态猜测，无法精准治理。

## 6. 目标架构

### 6.1 统一产物分类

目标引入统一分类，不要求所有文件迁到一个目录，但要求所有产物口被登记、可解释、可治理。

```text
ArtifactClass
  canonical_artifact       正式交付物，可给用户和后续任务使用
  runtime_fact             恢复关键事实，不可无审计删除
  runtime_projection       可重建投影，可丢弃重建
  diagnostic_trace         诊断痕迹，可 TTL/压缩
  task_definition          任务定义/图配置，归 TaskSystem
  task_record              任务运行记录，归 HealthSystem 管理维护
  knowledge_asset          知识/记忆资产，归 Memory/Knowledge 系统
  build_cache              构建缓存，可直接重建
```

### 6.2 权责链

```text
TaskSystem
  owns task definitions, graph configs, task assignments

RuntimeSystem
  owns runtime facts, event ledger, checkpoint, runtime object, execution state

ArtifactSystem
  owns canonical artifact registry, artifact refs, content hash, producer lineage

HealthSystem
  owns task record governance, artifact pressure report, maintenance policy, dry-run, receipts

DiagnosticOutput
  owns local traces, screenshots, process logs, temporary test output
```

### 6.3 新的核心服务

建议新增：

```text
backend/artifact_system/governance.py
  ArtifactGovernanceRegistry
  ArtifactPortPolicy
  ArtifactInventoryService
  ArtifactMaintenancePlanner

backend/runtime/shared/event_index.py
  RuntimeEventCursorStore
  RuntimeEventTailIndex
  RuntimeEventCompactionManifest

backend/health_system/artifact_governance_view.py
  HealthArtifactGovernanceViewBuilder
```

不建议新增一个“万能清理脚本”。清理只是治理的最后一步。

## 7. 产物口目标治理策略

### 7.1 正式产物口：ArtifactRepository

当前路径：

```text
storage/artifact_repository/artifact_repository.sqlite
storage/task_environments/**/artifacts
output/novel_artifacts
```

目标：

```text
ArtifactRepository 成为正式产物索引中心。
任务环境 artifacts 继续作为文件落地点，但必须被 ArtifactRepository 记录。
历史 output/novel_artifacts 后续迁为 artifact record 或归档为 legacy artifact bundle。
```

必须记录：

```text
artifact_ref
absolute/relative path
content_hash
output_contract_id
producer_node_id
task_run_id
graph_run_id
materialization_id
scope_kind
status
lifecycle_policy
```

禁止：

```text
只写文件不登记产物。
只靠文件名判断产物用途。
把 monitor/debug JSON 当正式产物。
```

### 7.2 运行事件口：RuntimeEventLog

当前路径：

```text
storage/runtime_state/events/*.jsonl
```

目标：

```text
保留 append-only 事实账本。
新增 offset/cursor sidecar，append 不再扫描全文。
新增 tail index，monitor 和 session timeline 读取最近事件不再全量读文件。
新增 compaction manifest，对已完成旧任务生成摘要段和归档段。
```

建议结构：

```text
storage/runtime_state/events/{task_run}.jsonl
storage/runtime_state/event_index/{task_run}.cursor.json
storage/runtime_state/event_index/{task_run}.tail.json
storage/runtime_state/event_archive/{task_run}/{segment}.jsonl.zst
storage/runtime_state/event_archive/{task_run}/manifest.json
```

保护规则：

```text
running / paused / waiting_* 不压缩。
有 checkout/resume lineage 的任务不删除原始近期事件。
失败任务必须先有 health report 或 failure receipt 才允许归档。
归档后必须可通过 manifest 回放 trace 摘要。
```

### 7.3 Graph checkpoint 口

当前路径：

```text
storage/runtime_state/graph_checkpoints.sqlite
storage/runtime_state/graph_checkpoints.sqlite-wal
```

目标：

```text
按 graph_run_id 保留 latest + stable recovery points + terminal receipt。
对 completed 且无 resume 依赖的旧 run 做 checkpoint compaction。
定期 checkpoint WAL，避免 wal 长期膨胀。
```

禁止：

```text
直接删除 sqlite/wal。
只按时间清 checkpoint，不看 TaskRun 和 lineage。
```

### 7.4 RuntimeObject 口

当前路径：

```text
storage/runtime_state/runtime_objects/**
```

目标：

```text
新增 ref index 和 reachability graph。
从 event log、state_index、checkpoint、artifact_repository 收集 rtobj 引用。
不可达对象进入 quarantine，不立即删除。
```

阶段策略：

```text
Phase A: inventory only
Phase B: quarantine with receipts
Phase C: delayed delete after restore test
```

### 7.5 StateIndex / RuntimeViews 口

当前路径：

```text
storage/runtime_state/state_index/**
storage/runtime_state/runtime_views/**
```

目标：

```text
state_index 保留任务恢复必要索引。
runtime_views 明确标记为 runtime_projection，可重建。
session_latest_task_runs 等轻量索引保留，但不能复制大 payload。
```

### 7.6 Prompt accounting 口

当前路径：

```text
storage/runtime_state/prompt_accounting/*.jsonl
```

目标：

```text
保留 token 真值账本。
新增日级/任务级 aggregate summary。
健康系统读 summary 做总览，必要时再读 ledger 明细。
已归档 task_run 的 accounting 记录可按 manifest 压缩。
```

### 7.7 Task records/config 口

当前路径：

```text
storage/tasks/*.json
```

目标：

```text
任务定义、图配置和任务记录继续留在 storage/tasks。
测试/监控/重跑快照不再写入 storage/tasks 根目录。
```

建议分区：

```text
storage/tasks/task_graphs.json                         正式配置
storage/tasks/graph_harness_configs.json               正式配置
storage/tasks/task_assignments.json                    正式配置
storage/tasks/specific_task_records.json               任务记录
storage/tasks/run_snapshots/{date}/{run_id}.json       受控快照
storage/tasks/debug_snapshots/{date}/{name}.json       诊断快照，可 TTL
```

关键规则：

```text
任务记录不要迁出。
HealthSystem 管理任务记录维护。
Debug snapshot 不能和正式 registry 同层。
latest 文件必须变成指针/manifest，不复制完整 monitor payload。
```

### 7.8 Local traces / Playwright / runtime logs 口

当前路径：

```text
output/local_traces
output/playwright
output/runtime
```

目标：

```text
诊断产物统一登记为 diagnostic_trace。
按日期滚动。
默认 TTL。
保留失败任务关联 trace。
保留最近 N 次实测截图。
runtime logs rotate，pid 只保留当前固定端口实例。
```

### 7.9 Frontend build cache 口

当前路径：

```text
frontend/.next
frontend/.next/cache
```

目标：

```text
标记为 build_cache。
不纳入健康系统任务记录治理。

```

## 9. 实施计划

### Phase 0：只读盘点和基线测试

目标：

```text
建立产物口 inventory，不执行删除。
确认当前恢复、监控、健康系统、ArtifactRepository 基线。
```

改动：

```text
新增 ArtifactPortPolicy 注册表。
新增 inventory 命令/API。
健康系统总览接入只读产物压力摘要。
```

验收：

```text
能列出各产物口文件数、体积、owner、class、保护原因。
不改变任何运行行为。
```

### Phase 1：RuntimeEventLog 索引化

目标：

```text
append 不再扫描整个 JSONL。
monitor/timeline 读最近事件不再全量读文件。
```

改动：

```text
新增 RuntimeEventCursorStore。
append 写事件后同步更新 cursor。
新增 tail index。
list_events 保留全量读取能力，但新增 list_recent_events / iter_events。
monitor_projection、session_timeline、health_governance 改用 recent/tail API。
```

验收：

```text
对大事件文件 append 新事件不调用全文 read_text。
现有事件 offset 不倒退、不重复。
监控进展仍实时更新。
旧 JSONL 仍可读取。
```

### Phase 2：Task snapshots 分层

## 11. 迁移和切换规则

### 11.1 Shadow-first

先上线只读 inventory 和新索引 shadow 写入。旧读取链路保持不变，直到测试确认新索引和旧事实一致。

### 11.2 Cutover

切换读取顺序：

```text
monitor/timeline/health
  -> recent/tail index
  -> fallback full event log read with warning
```

这个 fallback 是迁移窗口保护，不是长期兼容旧逻辑。切换完成后必须移除或降为修复工具。

### 11.3 Archive before delete

运行事实类产物不直接删除。先 archive/quarantine，写 manifest 和 receipt，通过恢复/trace 摘要测试后，才允许延迟删除。

### 11.4 不迁出任务记录

`storage/tasks` 里的正式任务定义、任务图配置和任务记录不迁出。只把调试快照、监控快照、重跑输出从根目录分层出去。

## 13. 风险控制

不能做：

```text
不能为了变小直接删除 runtime_state/events。
不能直接删除 graph_checkpoints.sqlite 或 wal。
不能把任务记录迁出健康系统管理域。
不能把正式产物和 debug snapshot 混在同一目录层。
不能用 max events / max files 这种限制补丁替代索引和压缩。
不能为了测试通过 mock 掉恢复、监控或产物登记。
```

必须做：

```text
先 inventory。
先 dry-run。
先 archive/quarantine。
先恢复验证。
维护动作必须有 receipt。
健康系统只做治理，不重写 runtime 事实。
```

## 14. 自检结论

这份方案与既有设计原则一致：

```text
任务记录仍归健康系统管理。
监控事实仍归 RuntimeMonitor / RuntimeEventLog。
正式产物归 ArtifactSystem。
HealthSystem 汇总和治理，不替事实源做决定。
优化重点是结构化索引、血缘、生命周期和可恢复维护，不是限制补丁。
```

当前最优先修复项是 `RuntimeEventLog` 的 append/read 结构，因为它直接解释了大事件文件下的卡顿风险；第二优先级是 `storage/tasks` 根目录的正式配置与调试快照分层；第三优先级是把正式产物全面纳入 `ArtifactRepository`，让后续任务和健康系统都通过产物引用和血缘工作。

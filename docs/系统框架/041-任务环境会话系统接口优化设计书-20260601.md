# 任务环境会话系统接口优化设计书

日期：2026-06-01

## 1. 结论

任务环境的会话系统必须从“全局聊天会话”升级为“环境内交互上下文”。

目标结构是：

```text
任务环境 TaskEnvironment
-> 项目 / 作品 Project
-> 会话 Session
-> 用户需求、沟通记录、普通 agent 对话
-> 任务图运行 GraphRun
-> 单节点 / 单 Agent 运行 TaskRun
```

会话不直接代表任务图运行，也不直接代表项目库。会话代表用户在某个任务环境、某个项目中的沟通上下文。任务图运行、单 Agent 运行、记忆读写、产物生成都必须挂到这个会话下，但不能反过来让运行对象决定会话归属。

本设计的核心目标是：

1. 不同任务环境的会话彻底隔离。
2. 同一任务环境下不同项目的会话彻底隔离。
3. 启动聊天、启动任务图、续跑任务图都通过同一套 session scope 解析。
4. 前端不再自己拼接会话归属，后端负责解析和校验。
5. 不保留无 scope 的任务环境会话旧链路。

## 2. 当前代码事实

### 2.1 已存在能力

后端 `SessionManager` 已经支持 scope：

文件：

- `backend/sessions/__init__.py`
- `backend/api/sessions.py`

当前 scope 结构：

```json
{
  "workspace_view": "creative",
  "task_environment_id": "env.creation.writing",
  "project_id": "project.creation.writing.honghuang"
}
```

当前接口：

```http
GET /api/sessions?workspace_view=creative&task_environment_id=env.creation.writing&project_id=project.creation.writing.honghuang
POST /api/sessions
```

前端 API 已支持：

文件：

- `frontend/src/lib/api.ts`

```ts
listSessions(scope?: Partial<SessionScope>)
createSession(title?: string, scope?: Partial<SessionScope>)
```

### 2.2 当前断层

前端主 store 仍然以全局会话为中心：

文件：

- `frontend/src/lib/store/runtime.ts`
- `frontend/src/lib/store/types.ts`

现状问题：

1. `initializeWorkspace()` 调用 `listSessions()`，读取的是全局会话列表。
2. `currentSessionId` 是全局单值。
3. `createFreshSession()` 默认调用 `createSession()`，没有 scope。
4. `refreshSessions()` 默认调用 `listSessions()`，没有 scope。
5. 创作环境页面自己维护 `scopedSessions`，但运行绑定仍调用全局 `selectSession()` 和全局 `currentSessionId`。

这形成两套权威：

```text
CreativeEnvironmentView.scopedSessions
frontend store.currentSessionId
```

前者知道环境和项目，后者不知道。任务图启动时又需要两者协作，导致会话归属容易串到其他页面或其他环境。

### 2.3 任务图启动接口断层

当前任务图启动接口：

文件：

- `backend/api/orchestration.py`
- `frontend/src/lib/api.ts`

```http
POST /api/orchestration/harness/task-graphs/{graph_id}/start
```

请求只显式接收：

```json
{
  "session_id": "session-xxx",
  "initial_inputs": {},
  "run_mode": "auto_run"
}
```

问题：

- 接口只校验 `session_id` 格式，不校验 session 是否属于当前任务环境。
- 接口不校验 graph 是否属于 session scope 中的任务环境。
- 接口不校验 `initial_inputs.project_id` 是否等于 session scope 的 `project_id`。
- 页面需要自己决定使用哪个 session。

这不符合成熟 agent 系统的边界要求。运行系统只应该接受已经被任务环境入口解析过的 session，并在启动时做强校验。

## 3. 设计原则

1. Session scope 是会话归属权威。
2. TaskEnvironment 是会话入口权威。
3. Project 是资料库、产物库、项目记忆的归属权威。
4. GraphRun / TaskRun 是执行事实，不拥有会话归属决策权。
5. 前端只声明当前工作上下文，不推断会话归属。
6. 后端负责 resolve、create、validate，不允许静默 fallback。
7. 普通聊天和任务图启动必须共享同一套 session scope。
8. 会话 scope 不进入 agent prompt 的客户可见文案，但必须进入 runtime context，提醒 agent 当前环境和项目已经改变。

## 4. 目标领域模型

### 4.1 TaskEnvironment

任务环境定义一类工作空间的资源和执行边界。

写作环境示例：

```json
{
  "task_environment_id": "env.creation.writing",
  "workspace_view": "creative",
  "title": "创作环境",
  "default_project_kind": "long_novel"
}
```

任务环境拥有：

- 环境 prompts
- 节点配置
- 契约库
- 任务图模板
- 项目模板
- 环境资源库
- 权限策略

### 4.2 Project

项目是用户实际管理对象。

```json
{
  "project_id": "project.creation.writing.honghuang",
  "task_environment_id": "env.creation.writing",
  "title": "洪荒时代",
  "project_kind": "long_novel",
  "library_id": "library.project.creation.writing.honghuang"
}
```

项目拥有：

- 项目库
- 正式作品
- 草稿与改写
- 创作产出
- 作品记忆
- 设定素材
- 评审记录

### 4.3 Session

会话是用户在环境和项目中的沟通上下文。

```json
{
  "id": "session-xxx",
  "title": "洪荒时代 沟通记录",
  "scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "project_id": "project.creation.writing.honghuang"
  },
  "messages": []
}
```

会话负责承载：

- 用户需求
- agent 对话
- 任务图启动意图
- 任务图运行附件
- 最终结果摘要
- 续跑和恢复入口

会话不负责：

- 决定任务图拓扑
- 管理项目库物理路径
- 直接执行节点
- 存放所有运行日志

### 4.4 GraphRun / TaskRun

`GraphRun` 和 `TaskRun` 是运行事实。

它们必须带有：

```json
{
  "session_id": "session-xxx",
  "task_environment_id": "env.creation.writing",
  "project_id": "project.creation.writing.honghuang",
  "graph_id": "graph.writing.modular_novel.master"
}
```

这些字段只能来自已解析的 session scope 和 graph config，不能由节点执行器重新猜测。

## 5. 目标接口

### 5.1 低层 Session CRUD

保留：

```http
GET /api/sessions
GET /api/sessions?workspace_view=...&task_environment_id=...&project_id=...
POST /api/sessions
GET /api/sessions/{session_id}/timeline
PUT /api/sessions/{session_id}
DELETE /api/sessions/{session_id}
```

但定位调整为低层存储接口。任务环境页面不应直接用低层接口决定“当前会话”。

### 5.2 任务环境会话解析

新增：

```http
POST /api/task-environments/{environment_id}/sessions/resolve
```

请求：

```json
{
  "workspace_view": "creative",
  "project_id": "project.creation.writing.honghuang",
  "intent": "start_graph",
  "title": "洪荒时代 沟通记录",
  "preferred_session_id": "",
  "create_if_missing": true
}
```

响应：

```json
{
  "authority": "task_environment.session_resolver",
  "session": {
    "id": "session-xxx",
    "title": "洪荒时代 沟通记录",
    "scope": {
      "workspace_view": "creative",
      "task_environment_id": "env.creation.writing",
      "project_id": "project.creation.writing.honghuang"
    }
  },
  "created": false,
  "reason": "preferred_session_valid"
}
```

解析规则：

1. 如果 `preferred_session_id` 存在，后端读取该 session。
2. 如果该 session scope 完全匹配请求 scope，则返回。
3. 如果不匹配，返回 409，不允许自动挪用。
4. 如果没有 preferred session，则查找同 scope 最近会话。
5. 如果没有最近会话且 `create_if_missing=true`，创建新会话。
6. 如果没有最近会话且 `create_if_missing=false`，返回 404。

禁止：

- 禁止从全局 chat 会话 fallback 到 creative 项目会话。
- 禁止只按 title 匹配会话。
- 禁止 project_id 缺失时默认使用第一个项目。

### 5.3 任务环境会话列表

新增：

```http
GET /api/task-environments/{environment_id}/sessions?workspace_view=creative&project_id=project.creation.writing.honghuang
```

响应：

```json
{
  "authority": "task_environment.session_list",
  "scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "project_id": "project.creation.writing.honghuang"
  },
  "sessions": []
}
```

这不是重复 `/api/sessions`，而是给任务环境页面使用的明确入口。它的职责是强制环境 scope，避免页面误读全局会话。

### 5.4 聊天运行接口

当前：

```http
POST /api/chat/runs
```

建议增强请求：

```json
{
  "session_id": "session-xxx",
  "session_scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "project_id": "project.creation.writing.honghuang"
  },
  "message": "请继续完善世界观设定",
  "runtime_profile": {},
  "task_selection": {}
}
```

后端校验：

```text
session exists
session.scope == request.session_scope
```

普通 chat 页面也使用 scope：

```json
{
  "workspace_view": "chat",
  "task_environment_id": "env.general.workspace",
  "project_id": ""
}
```

### 5.5 任务图启动接口

当前：

```http
POST /api/orchestration/harness/task-graphs/{graph_id}/start
```

目标请求：

```json
{
  "session_id": "session-xxx",
  "session_scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "project_id": "project.creation.writing.honghuang"
  },
  "initial_inputs": {
    "project_id": "project.creation.writing.honghuang",
    "writing_flow": "design",
    "user_request": "请以洪荒时代种子启动设定体系设计"
  },
  "include_trace": true,
  "dispatch_ready": true,
  "run_mode": "auto_run"
}
```

后端必须校验：

```text
session exists
session.scope.workspace_view == payload.session_scope.workspace_view
session.scope.task_environment_id == payload.session_scope.task_environment_id
session.scope.project_id == payload.session_scope.project_id
graph_config.task_environment_id == payload.session_scope.task_environment_id
initial_inputs.project_id == payload.session_scope.project_id
```

如果 `initial_inputs.project_id` 缺失：

- 对需要项目的环境，拒绝。
- 对无项目环境，允许为空。

如果 `graph_config.task_environment_id` 缺失：

- 已发布 graph config 必须补齐。
- 不允许默认归入写作环境。

响应补充：

```json
{
  "authority": "harness.api.task_graph_run_start",
  "session_scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "project_id": "project.creation.writing.honghuang"
  },
  "graph_id": "...",
  "graph_run_id": "...",
  "task_run_id": "..."
}
```

### 5.6 续跑接口

续跑不重新解析用户意图，但必须校验调用者环境。

```http
POST /api/orchestration/harness/graph-runs/{graph_run_id}/resume
```

请求：

```json
{
  "graph_harness_config_id": "...",
  "session_scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "project_id": "project.creation.writing.honghuang"
  },
  "dispatch_ready": true
}
```

校验：

```text
graph_run.session_id -> session.scope == request.session_scope
graph_run.task_environment_id == request.session_scope.task_environment_id
graph_run.project_id == request.session_scope.project_id
```

### 5.7 图运行变更接口统一校验

任何会改变、推进、写入或删除图运行状态的接口，都必须带 `session_scope` 并校验。

当前需要纳入的接口包括：

```http
POST   /api/orchestration/harness/graph-runs/{graph_run_id}/dispatch-ready
POST   /api/orchestration/harness/graph-runs/{graph_run_id}/node-results
POST   /api/orchestration/harness/graph-runs/{graph_run_id}/work-orders/execute
POST   /api/orchestration/harness/graph-runs/{graph_run_id}/run-until-idle
POST   /api/orchestration/harness/graph-runs/{graph_run_id}/resume
DELETE /api/orchestration/harness/graph-runs/{graph_run_id}
```

统一请求形态：

```json
{
  "session_scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "project_id": "project.creation.writing.honghuang"
  }
}
```

每个接口在执行业务动作前必须校验：

```text
graph_run exists
graph_run.session_id -> session.scope == request.session_scope
graph_run.task_environment_id == request.session_scope.task_environment_id
graph_run.project_id == request.session_scope.project_id
```

如果接口同时接收 `graph_harness_config_id`，还必须校验：

```text
graph_config.task_environment_id == request.session_scope.task_environment_id
```

如果接口接收 work order 或 node result，还必须校验：

```text
work_order.graph_run_id == path.graph_run_id
work_order.session_id == graph_run.session_id
work_order.project_id == graph_run.project_id
```

禁止：

- 禁止只凭 `graph_run_id` 推进运行。
- 禁止任务环境页面调用无 scope 的 graph mutation API。
- 禁止在 scope 不匹配时返回成功但不执行。
- 禁止通过 monitor 里最近一次绑定推断当前 scope。

### 5.8 Session 直达接口的 scope 防护

低层 session-id 直达接口也必须纳入 scope 防护，否则任务环境页面仍可能绕过 resolver 读取、改名、删除或截断其他环境会话。

需要处理的接口：

```http
GET    /api/sessions/{session_id}/history
GET    /api/sessions/{session_id}/timeline
GET    /api/sessions/{session_id}/messages
PUT    /api/sessions/{session_id}
DELETE /api/sessions/{session_id}
POST   /api/sessions/{session_id}/messages/truncate
POST   /api/sessions/{session_id}/generate-title
GET    /api/tokens/session/{session_id}
GET    /api/memory/session/{session_id}/files
```

目标规则：

1. 任务环境 UI 调用这些接口时必须传 `session_scope`。
2. 后端必须校验 `session.scope == request.session_scope`。
3. 无 scope 调用只允许普通 chat 迁移期和内部诊断使用。
4. 完成迁移后，生产 UI 不允许无 scope 调用 session-id 直达接口。

建议形式：

```http
GET /api/sessions/{session_id}/timeline?workspace_view=creative&task_environment_id=env.creation.writing&project_id=project.creation.writing.honghuang
```

变更类接口使用 body：

```json
{
  "session_scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "project_id": "project.creation.writing.honghuang"
  }
}
```

### 5.9 resolver intent 规则

`sessions/resolve` 不能只按“最近会话”处理所有场景。不同用户意图必须有不同解析规则。

支持的 intent：

```text
continue_conversation
new_conversation
start_graph
resume_graph
open_project
```

规则：

| intent | 规则 |
| --- | --- |
| `continue_conversation` | 优先使用 preferred session；没有 preferred session 时复用同 scope 最近会话；没有则按 create_if_missing 决定是否新建。 |
| `new_conversation` | 忽略最近会话，创建新的 scoped session；如果 create_if_missing=false，返回 400。 |
| `start_graph` | 优先使用当前页面选中的 preferred session；没有 preferred session 时可以新建；不得自动复用其他项目或其他环境会话。 |
| `resume_graph` | 必须从 graph_run 绑定的 session 解析；不得另选最近会话；scope 不匹配直接 409。 |
| `open_project` | 只加载该项目 scoped sessions，不自动创建；需要用户提交需求时再创建。 |

`resume_graph` 请求必须额外携带：

```json
{
  "intent": "resume_graph",
  "graph_run_id": "grun:xxx"
}
```

后端解析顺序：

```text
graph_run_id -> graph_run.session_id -> session.scope -> compare request scope
```

禁止：

- 禁止 resume_graph 时按最近会话重绑。
- 禁止 start_graph 在没有用户选择时复用其他 scope 的最近会话。
- 禁止 open_project 自动创建空会话污染项目。

## 6. 前端状态设计

### 6.1 当前状态问题

当前 store：

```ts
currentSessionId: string | null;
sessions: SessionSummary[];
messages: Message[];
```

这只适合单一聊天环境，不适合任务环境。

### 6.2 目标状态

新增：

```ts
type SessionScopeKey = string;

type ScopedSessionState = {
  scope: SessionScope;
  sessions: SessionSummary[];
  currentSessionId: string | null;
  messages: Message[];
  tokenStats: TokenStats | null;
  sessionActivity: SessionActivityState;
  activeStreamSessionIds: string[];
  sessionActivitiesById: Record<string, SessionActivityState>;
  orchestrationSnapshot: OrchestrationSnapshot | null;
  taskGraphMonitorBinding: TaskGraphMonitorBinding | null;
  taskGraphLiveMonitor: HarnessTaskRunLiveMonitor | null;
  streamingSessionCache: Record<string, SessionStreamCache>;
  loading: boolean;
  error: string;
};

type StoreState = {
  activeSessionScopeKey: SessionScopeKey;
  scopedSessionsByKey: Record<SessionScopeKey, ScopedSessionState>;
};
```

scope key 生成规则：

```ts
function sessionScopeKey(scope: SessionScope) {
  return [
    scope.workspace_view || "chat",
    scope.task_environment_id || "",
    scope.project_id || "",
  ].join("|");
}
```

保留 `currentSessionId` 作为兼容投影时，必须只从当前 active scope 派生，不再作为事实源。

### 6.2.1 scoped state 覆盖范围

前端迁移不能只迁移 `sessions/currentSessionId/messages/tokenStats`。以下状态同样属于会话或运行视图，必须按 scope 隔离：

```text
sessionActivity
activeStreamSessionIds
sessionActivitiesById
orchestrationSnapshot
taskGraphMonitorBinding
taskGraphLiveMonitor
streamingSessionCache
chat stream cursor
runtime monitor selected session view
```

否则会出现：

- 切到创作环境后仍显示代码环境运行状态。
- 断线重连时 reattach 到旧环境 session。
- `hydrateLatestOrchestrationSnapshot()` 把旧 session 的任务图快照写回当前页面。
- 后台 stream 完成后把旧环境消息覆盖到当前环境 messages。

迁移规则：

1. `currentSessionId` 只作为 `activeSessionScopeKey` 下 scoped state 的派生值。
2. `selectSession()` 必须升级为 `selectScopedSession(scope, sessionId)`。
3. `reattachChatRunForSession()` 必须校验 session scope。
4. `hydrateLatestOrchestrationSnapshot()` 必须接收 scope 并校验返回 session。
5. `streamingSessionCache` key 必须至少包含 `scope_key + session_id`。
6. chat stream cursor 保存和读取时必须校验 session scope。

### 6.3 前端 actions

新增统一动作：

```ts
listScopedSessions(scope: SessionScope): Promise<void>
resolveScopedSession(scope: SessionScope, options: ResolveSessionOptions): Promise<string>
selectScopedSession(scope: SessionScope, sessionId: string): Promise<void>
createScopedSession(scope: SessionScope, title: string): Promise<string>
sendScopedMessage(scope: SessionScope, message: string): Promise<void>
startGraphRunFromScopedSession(scope: SessionScope, graphId: string, payload: GraphStartPayload): Promise<void>
resumeGraphRunFromScopedSession(scope: SessionScope, graphRunId: string, payload: GraphResumePayload): Promise<void>
deleteGraphRunFromScopedSession(scope: SessionScope, graphRunId: string): Promise<void>
```

创作页不再维护本地 `scopedSessions`，只读取：

```ts
const scope = creativeSessionScope(projectId);
const sessionState = useScopedSessionState(scope);
```

### 6.4 页面接法

创作环境：

```text
进入页面
-> 读取项目
-> 构造 scope
-> listScopedSessions(scope)
-> resolveScopedSession(scope, create_if_missing=false)
-> 用户输入需求
-> resolveScopedSession(scope, create_if_missing=true)
-> startGraphRunFromScopedSession(scope, graphId, payload)
```

代码环境：

```text
scope = {
  workspace_view: "code-environment",
  task_environment_id: "env.development.sandbox",
  project_id: workspace_project_id 或 ""
}
```

普通工作台：

```text
scope = {
  workspace_view: "chat",
  task_environment_id: "env.general.workspace",
  project_id: ""
}
```

## 7. Agent prompt / runtime context 接法

任务环境变化必须进入 runtime context，而不是作为开发说明泄露给用户。

运行包里应显式提供：

```json
{
  "runtime_scope": {
    "workspace_view": "creative",
    "task_environment_id": "env.creation.writing",
    "task_environment_title": "创作环境",
    "project_id": "project.creation.writing.honghuang",
    "project_title": "洪荒时代"
  }
}
```

给 agent 的 prompt 不应写成：

```text
这是 creative 环境。
当前 task_environment_id 是 env.creation.writing。
```

应写成：

```text
你正在为作品《洪荒时代》提供长篇创作支持。
你需要遵守当前创作环境的资料边界，只使用本作品资料、作品记忆和当前任务输入。
如果用户要求切换作品或使用其他作品资料，你必须先说明需要切换作品上下文，不能混用资料。
```

## 8. 权责链路

目标权威链：

```text
PageContext
-> SessionScope
-> TaskEnvironmentSessionResolver
-> SessionRecord
-> RuntimeStartRequest
-> GraphRun / ChatRun
-> RuntimeMonitor / Timeline
```

权责表：

| 层 | 权责 | 禁止 |
| --- | --- | --- |
| PageContext | 知道当前页面、环境、项目 | 自己猜当前 session |
| SessionScope | 描述归属 | 携带运行结果 |
| TaskEnvironmentSessionResolver | 创建/选择/校验 session | 静默 fallback |
| SessionRecord | 保存用户沟通和 timeline 锚点 | 决定图拓扑 |
| RuntimeStartRequest | 启动 chat/graph run | 改写 session scope |
| GraphRun / TaskRun | 记录执行事实 | 决定会话归属 |
| Timeline | 展示运行附件 | 成为执行事实源 |

### 8.1 GraphRun / TaskRun scope 持久化

GraphRun 和 TaskRun 必须持久化 resolved scope，而不是只把 scope 放在 diagnostics 或 runtime_scope 投影中。

目标字段：

```json
{
  "session_id": "session-xxx",
  "workspace_view": "creative",
  "task_environment_id": "env.creation.writing",
  "project_id": "project.creation.writing.honghuang",
  "session_scope_key": "creative|env.creation.writing|project.creation.writing.honghuang"
}
```

写入规则：

1. graph start API 先解析并校验 session scope。
2. GraphHarness start_run 接收 resolved scope。
3. GraphRun 持久化 resolved scope。
4. 顶层 TaskRun 持久化同一 resolved scope。
5. 子节点 TaskRun / work order 从 GraphRun 继承 scope。
6. runtime_scope 可以包含更多运行字段，但不能替代 resolved session scope。

#### 8.1.1 project-required 环境禁止 graph-run fallback

当前图运行 runtime 在缺少 project_id 时可能生成 `graphrun.{graph_run_id}` 作为 project scope。该策略只能用于不需要项目的临时图运行，不能用于写作、代码项目、健康场景等项目必需环境。

新增规则：

```text
if environment.requires_project == true and session_scope.project_id is empty:
  reject start / resume / mutation
```

写作环境必须设置：

```json
{
  "task_environment_id": "env.creation.writing",
  "requires_project": true
}
```

禁止：

- 禁止写作任务图自动生成 `graphrun.*` 项目。
- 禁止用 `scope_id` 替代 project_id 启动写作图。
- 禁止在 GraphRun 已持久化 scope 后由 runtime_scope 覆盖 project_id。

## 10. 验收标准

### 10.1 接口验收

必须通过：

1. `GET /task-environments/env.creation.writing/sessions?project_id=A` 不返回项目 B 的会话。
2. `POST /task-environments/env.creation.writing/sessions/resolve` 可以复用同 scope 最近会话。
3. preferred session scope 不匹配时返回 409。
4. 写作任务图不能用 chat session 启动。
5. 写作任务图不能用其他写作项目 session 启动。
6. graph run response 包含 session_scope。
7. resume graph run 时校验 session_scope。
8. dispatch-ready、node-results、work-orders/execute、run-until-idle、delete 都校验 session_scope。
9. GraphRun / TaskRun 持久化 resolved scope。
10. project-required 环境缺 project_id 时拒绝启动。
11. session-id 直达接口不能跨 scope 读取或修改。

### 10.2 前端验收

必须通过：

1. 进入创作环境只显示当前作品沟通记录。
2. 切到代码环境后，不显示创作环境沟通记录。
3. 切回创作环境，恢复该作品最后使用的 scoped session。
4. 切换写作项目，当前会话随项目切换。
5. 开始创作不会使用普通工作台会话。
6. 任务图运行监控绑定到当前作品会话。
7. 断线重连后，根据 session scope 恢复到正确环境。
8. 后台 stream 完成不会把其他 scope 的 messages 写入当前页面。
9. 切换项目后 orchestration snapshot 和 task graph monitor 不沿用旧项目。

### 10.3 Agent 行为验收

必须通过：

1. Agent 能感知当前作品名和当前创作环境。
2. Agent 不把 `env.*`、`project.*` 等内部 ID 当作用户可见表达。
3. Agent 不混用其他项目资料。
4. 切换任务环境后，新 turn 的 runtime context 明确变化。

## 11. 风险与处理

### 风险 1：历史无 scope 会话

处理：

- 历史缺 scope 会话统一归入 `workspace_view=chat`。
- 不允许任务环境 resolver 自动吸收历史无 scope 会话。

### 风险 2：前端 active session 迁移复杂

处理：

- 先引入 `scopedSessionsByKey`。
- `currentSessionId` 暂时作为 active scope 投影。
- 所有新页面接 scoped actions。
- 最后删除对全局 current session 的业务依赖。

### 风险 3：任务图历史运行缺 project_id

处理：

- 新启动运行必须强制 project_id。
- 历史运行在 monitor 中标注 scope 缺失，不参与 scoped resume。
- 不做静默补 project。
- project-required 环境不允许 `graphrun.*` fallback。

### 风险 4：普通 chat 与任务环境 chat 混淆

处理：

- 普通 chat 也显式使用 scope。
- 所有 chat run 请求均可带 session_scope。
- 后端只校验，不替前端猜测。

### 风险 5：图运行 mutation 接口遗漏

处理：

- 将 graph mutation API 统一登记为 scope-guarded endpoints。
- 后端提供共用 `assert_graph_run_scope(graph_run_id, session_scope)`。
- 测试逐个覆盖 dispatch、node result、work order、run-until-idle、resume、delete。

### 风险 6：前端后台流覆盖当前 scope

处理：

- streaming cache key 加 scope key。
- stream reducer 写入前校验 active scope。
- 非当前 scope 的 stream 只更新对应 scoped state，不更新当前页面投影。

## 12. 禁止事项

- 禁止继续让任务环境页面直接使用全局 `currentSessionId` 作为事实源。
- 禁止任务图启动接口只拿 `session_id` 而不校验 scope。
- 禁止 project_id 缺失时默认使用第一个项目。
- 禁止把其他环境会话静默迁移到当前环境。
- 禁止通过 session title 匹配任务环境。
- 禁止让 GraphRun / TaskRun 反向决定 session scope。
- 禁止把内部 ID 暴露到客户页面文案。
- 禁止 graph mutation 接口无 scope 推进运行。
- 禁止 project-required 环境生成 `graphrun.*` 项目 fallback。
- 禁止 session-id 直达接口绕过 scope 防护。
- 禁止后台 stream 或 hydrate 把旧 scope 状态写入当前页面。

## 13. 推荐最终链路

创作环境开始一次写作：

```text
用户输入写作需求
-> 前端构造 scope: creative + env.creation.writing + project.creation.writing.honghuang
-> POST /task-environments/env.creation.writing/sessions/resolve
-> 返回 session-xxx
-> POST /orchestration/harness/task-graphs/{graph_id}/start
   with session_id + session_scope + initial_inputs.project_id
-> 后端校验 session scope / graph environment / project
-> GraphRun 记录 session_id + task_environment_id + project_id
-> 前端绑定 monitor
-> 会话 timeline 展示任务图运行附件
```

这条链路中，只有任务环境 session resolver 可以决定用哪个 session。页面、任务图、节点执行器都不能绕过它。

# 成熟 Agent 多工具并发与并行子 Agent 改造计划书

日期：2026-06-03

状态：待确认实施。本文是 `053-成熟Agent并发工具权限边界与控制动作修复方案-20260603.md` 的实施级细化计划，专门解决“agent 一次只能做一个动作”、任务执行链路不能批量工具调用、子 agent 无法在同一轮批量 spawn 并行执行的问题。

## 0. 结论

当前项目不是完全没有并发能力，而是并发能力只局部存在，主任务执行协议仍把 agent 限成“一轮一个动作”。

当前主问题：

- `single_agent_turn` 已经能对多个 native tool calls 使用 `asyncio.gather` 并发执行。
- `task_executor` 仍使用单个 JSON action，且 `allow_native_tool_calls=False`，每轮只能执行一个 `tool_call`。
- 子 agent spawn 后已经可以通过 background task 执行，但主 agent 每轮只能 spawn 一个 child task run，导致并行子 agent 能力被协议层锁死。
- 权限拒绝、工具不可见、工具失败有时被升级成 blocked，而不是作为 observation 反馈给模型继续恢复。

目标架构：

```text
ModelTurnDecision
-> ToolCallBatch
-> per-call Admission
-> RuntimeToolBatchScheduler
-> Resource Locks / Concurrency Limits
-> Parallel Execution
-> ObservationBatch
-> Next Model Turn
```

核心原则：

- 普通工具调用可以批量并发。
- 控制动作必须单选。
- 权限是边界和能力反馈，不是阻止 agent 主循环继续的理由。
- 子 agent 是真实后台 worker，允许在同一父任务轮次批量 spawn 并并行执行。
- 写入、shell、git、browser 等副作用工具必须经过资源调度，不允许无边界并发。

## 1. 当前代码证据

### 1.1 单轮对话已经局部支持并发

文件：

```text
backend/harness/loop/single_agent_turn.py
```

当前 `single_agent_turn` 对多个 native tool call 已经有：

```python
invocation_results = await asyncio.gather(
    *(invocation for _row_index, invocation in pending_tool_invocations),
    return_exceptions=True,
)
```

这说明本项目已有并发执行的局部实现，不需要推翻整个 runtime。

### 1.2 task executor 仍是单工具协议

文件：

```text
backend/harness/loop/model_action_protocol.py
backend/harness/loop/task_executor.py
```

当前 `TaskExecutionModelActionRequest` 只有单个：

```python
tool_call: dict[str, Any]
```

任务执行模型调用中：

```python
allow_native_tool_calls=False
```

随后 `task_executor` 每轮只处理：

```python
if action_request.action_type == "tool_call":
    observation = await _execute_task_tool_call(...)
```

这就是“成熟 agent 明明应该能并发，但当前只能一次做一个动作”的直接原因。

### 1.3 子 agent 已经能后台跑，但不能批量启动

文件：

```text
backend/harness/agent_control/controller.py
```

`spawn_subagent` 会创建 child `TaskRun` 和 child `AgentRun`，然后：

```python
spawner(_runner(), name=f"subagent-executor:{child_task_run_id}")
```

如果一轮能 spawn 多个 child task run，它们理论上已经可以并行执行。当前限制来自父 agent 协议每轮只能发一个 `spawn_subagent`。

### 1.4 子 agent 配额当前不是并发原子检查

文件：

```text
backend/harness/agent_control/controller.py
```

`_spawn_allowed()` 当前通过读取现有 children 计算：

```python
active = [item for item in children if item.status in {"pending", "running"}]
if max_active > 0 and len(active) >= max_active:
    return False, "max_active_subagents_exceeded"
```

如果同一批次并发 spawn 多个子 agent，必须把这个检查改成父任务维度的原子配额预留，否则存在同批绕过 `max_active_subagents` 的风险。

## 2. 目标协议

### 2.1 控制动作保持单选

以下动作互斥，一轮只能出现一个：

```text
respond
ask_user
block
request_task_run
request_registered_engagement
active_work_control
```

原因：

- `respond` 和 `ask_user` 不能同时成立。
- `block` 和 `request_task_run` 不能同时成立。
- `active_work_control` 是对当前运行中任务的控制，不应和普通工具批次混用。

### 2.2 工具动作改成批量

任务执行的 canonical JSON action 改为：

```json
{
  "authority": "harness.loop.model_action_request",
  "action_type": "tool_call",
  "public_progress_note": "正在并行检查相关文件、配置和测试入口。",
  "public_action_state": {
    "visible_status": "waiting_for_tool",
    "current_judgment": "需要先收集多个独立事实。",
    "next_action": "并行调用读取和搜索工具。",
    "evidence_refs": [],
    "open_risks": [],
    "completion_status": "waiting_for_tool"
  },
  "tool_calls": [
    {
      "tool_name": "search_text",
      "args": {"pattern": "active_work_control", "path": "backend"}
    },
    {
      "tool_name": "read_file",
      "args": {"path": "backend/harness/loop/task_executor.py"}
    }
  ],
  "diagnostics": {}
}
```

单工具也使用 `tool_calls` 一元素数组。`tool_call` 不再作为 task execution 主协议字段。

### 2.3 Observation 批次

每个工具调用生成独立 observation：

```text
observation_id
tool_call_id
tool_name
status
summary
payload
error
resource_keys
batch_id
```

同一批次额外生成聚合事件：

```text
task_tool_batch_started
task_tool_batch_completed
task_tool_observation_recorded
```

模型下一轮应看到完整 observation batch，而不是只看到最后一个工具结果。

## 3. RuntimeToolBatchScheduler

新增文件：

```text
backend/harness/runtime/tool_batch_scheduler.py
```

职责：

- 不做权限判断。
- 不做模型语义判断。
- 只根据工具定义、operation 描述、参数资源和环境策略决定并发批次。

输入：

```python
ToolBatchScheduleInput(
    tool_calls=tuple[ToolCallRequest, ...],
    runtime_tool_plan=RuntimeToolPlan,
    definitions_by_name=dict[str, ToolDefinition],
    runtime_assembly=dict[str, Any],
    task_run=dict[str, Any],
)
```

输出：

```python
ToolBatchSchedule(
    batch_id=str,
    groups=tuple[ToolExecutionGroup, ...],
    diagnostics=dict[str, Any],
)
```

每个 `ToolExecutionGroup` 表示一个可并发执行组。同组内 `asyncio.gather`，组与组之间串行。

## 4. 资源锁与并发分类

### 4.1 默认可并行工具

这些工具默认并发安全，适合批量收集事实：

```text
read_file
list_dir
stat_path
path_exists
glob_paths
search_files
search_text
python_code_outline
python_parse_check
python_symbol_search
git_status
git_diff
git_log
git_show
git_branch_list
web_search
fetch_url
read_structured_file
mcp_pdf
mcp_structured_data
memory_read
list_subagents
wait_subagent
```

调度规则：

- 同批最大并发默认 6。
- 搜索类可以和读取类并发。
- 多个 read_file 可并发。
- 多个 wait_subagent 可并发。
- 结果按模型发起顺序回填，不能因为先完成而改变观察顺序。

### 4.2 有条件并行工具

#### terminal

默认不和其他副作用工具并行。满足以下条件时可并行：

- 命令是只读检查、测试或状态查询。
- 不共享端口、进程、输出文件、临时目录。
- 不修改 git/workspace。
- sandbox scope 明确。

风险命令必须串行：

```text
npm install
git checkout
git commit
start server
kill process
database migration
```

#### write_file / edit_file

可并行条件：

- 目标路径不同。
- 不写同一目录下的生成索引、manifest、lockfile。
- 不依赖上一个 edit 的输出。

同一文件必须串行。无法解析路径时默认串行。

#### image_generate

不同 artifact target 可并行。同一目标路径必须串行。

#### send_subagent_message

不同 subagent 可并行。同一个 subagent 的消息必须串行。

### 4.3 默认串行工具

```text
agent_todo
memory_write_candidate
git_stage
git_unstage
git_commit
git_restore
git_push
close_subagent
browser_control
```

原因：

- 它们会改全局状态、git 状态、浏览器 session 或 agent session 状态。
- 并行执行容易产生不可恢复的顺序歧义。

## 5. 并行子 Agent 方案

### 5.1 保留现有子 agent 工具

首期不新增工具，复用现有：

```text
spawn_subagent
send_subagent_message
wait_subagent
list_subagents
close_subagent
```

通过 `tool_calls` 批次实现：

```json
{
  "action_type": "tool_call",
  "tool_calls": [
    {
      "tool_name": "spawn_subagent",
      "args": {
        "target_agent_id": "agent:codebase_searcher",
        "goal": "定位 active_work 工具关闭链路"
      }
    },
    {
      "tool_name": "spawn_subagent",
      "args": {
        "target_agent_id": "agent:verifier",
        "goal": "复核并发工具调度风险和测试入口"
      }
    }
  ]
}
```

### 5.2 原子配额

在 `SubagentControl.spawn_subagent()` 前新增父任务维度配额预留：

```text
parent_task_run_id + parent_agent_run_ref -> subagent_spawn_lock
```

执行流程：

```text
acquire lock
read active children
reserve slot
create child AgentRun + child TaskRun
release lock
schedule background executor
```

这样同批多个 spawn 不会绕过：

```text
max_subagent_runs_per_task
max_active_subagents
allow_nested_subagents
allowed_subagent_ids
```

### 5.3 子 agent 并行执行

当前 `_append_child_task_run()` 已经会：

```text
create child TaskRun(status=waiting_executor)
spawn background executor
```

多 spawn 批次完成后，多个 child task run 会并行进入执行。

父 agent 后续使用：

```text
list_subagents
wait_subagent x N
```

并发收集结果。

### 5.4 父任务停止级联

父 `task_run` stop 时必须处理 child task run：

```text
parent stop
-> list owned child task runs
-> request stop for pending/running children
-> record subagent_parent_stopped
```

否则并行子 agent 会在父任务停止后继续运行，破坏任务控制边界。

## 6. Prompt 改造

### 6.1 task execution prompt

需要告诉 agent：

```text
你可以一次请求多个普通工具调用。
当多个读取、搜索、检查、子 agent 等动作彼此独立时，应放入同一个 tool_calls 批次。
有依赖关系、同一文件写入、git 状态修改、浏览器同一 session 操作必须串行。
工具失败、权限拒绝或工具不可见不是你停止运行的理由；你应基于 observation 更换路径、缩小范围、询问用户或在证据充分时裁决 block。
```

### 6.2 subagent prompt

需要告诉主 agent：

```text
当任务可以拆成互不依赖的调查、验证或实现分支时，你可以并行启动多个子 Agent。
每个子 Agent 必须有清晰边界、输入、禁止事项和预期输出。
不要让多个写入型子 Agent 修改同一文件或同一全局状态。
父 Agent 必须汇总子 Agent 证据并做最终裁决，不能把子 Agent 输出原样当成最终答案。
```

### 6.3 环境 prompt 不承担工具调度

环境 prompt 只说明资源边界、风险和可用工作区，不写“这个节点用于并行”这类开发说明。并行策略由 runtime prompt 和 task execution action schema 表达。

## 7. 实施阶段

### 阶段一：协议改造

修改：

```text
backend/harness/loop/model_action_protocol.py
backend/harness/runtime/compiler.py
backend/tests/*protocol*
```

内容：

- 新增 `ToolCallRequest` 模型。
- `TaskExecutionModelActionRequest` 改为 `tool_calls`。
- prompt schema 改为批量工具协议。
- 删除 task execution 单 `tool_call` 主路径。

验收：

- 模型返回多个 `tool_calls` 时协议 accepted。
- 控制动作和工具批次混用时协议 rejected，并生成内部 repair observation，不直接 blocked。

### 阶段二：task executor 并发执行

修改：

```text
backend/harness/loop/task_executor.py
backend/harness/runtime/tool_batch_scheduler.py
backend/runtime/tool_runtime/tool_control_plane.py
```

内容：

- 将 `_execute_task_tool_call()` 包装成 batch executor。
- per-call admission。
- per-call action permit。
- 同组 `asyncio.gather`。
- batch observation 进入下一轮 packet。

验收：

- 三个 read/search 工具并发执行，总耗时小于串行总和。
- 单个工具失败不导致 task_run blocked。

### 阶段三：资源锁和副作用串行化

修改：

```text
backend/harness/runtime/tool_batch_scheduler.py
backend/permissions/operations.py
backend/capability_system/tools/native_tool_catalog.py
```

内容：

- 使用 operation descriptor 的 `read_only`、`destructive`、`concurrency_safe`。
- 从 tool args 提取 resource keys。
- 写同一路径串行。
- git/browser/agent state 默认串行。
- shell 依据 sandbox 和命令资源保守调度。

验收：

- 同文件两个 edit 被分到两个串行 group。
- 不同文件 read/edit 按规则分组。
- git 操作不与写入或 shell 并发。

### 阶段四：并行子 agent

修改：

```text
backend/harness/agent_control/controller.py
backend/harness/loop/task_executor.py
backend/tests/*subagent*
```

内容：

- `spawn_subagent` 支持同批多个调用。
- 父任务维度原子配额锁。
- `wait_subagent` 多个并发等待。
- parent stop 级联 child stop。

验收：

- 同一轮 spawn 2 个子 agent，生成 2 个 child task run。
- child task run 同时进入 running/waiting_executor 调度。
- `max_active_subagents=2` 时第三个同批 spawn 被拒绝为 observation，不阻塞父 agent。
- 父任务 stop 后 child task run 被停止。

## 9. 非目标

本计划不做以下事情：

- 不放开所有权限。
- 不让控制动作并发。
- 不让多个 agent 无边界写同一文件。
- 不绕过 permission / sandbox / resource policy。
- 不把工具失败伪装成成功。
- 不在前端决定哪些工具可以并行。

## 10. 成熟性自检

本方案符合成熟 agent 架构：

- 模型负责提出多个独立动作。
- runtime 负责授权和资源调度。
- 工具负责执行并返回 observation。
- 权限拒绝作为可恢复 observation 回到模型。
- 控制动作保持单一裁决。
- 子 agent 是 task run，不是同步函数调用。
- 父任务对 child task run 拥有停止和汇总责任。

这不是“大改整套 runtime”，而是把已经存在的局部并发能力提升到 task executor 主链路，并补上子 agent 批量调度和资源安全边界。

# 环境任务系统配置框架设计书

日期：2026-05-28

状态：设计锁定稿

适用范围：

```text
backend/task_system/environments
backend/task_system/services
backend/task_system/tasks
backend/agent_system/profiles
backend/harness/runtime
backend/harness/loop
backend/capability_system
frontend/src/components/workspace/views
```

本设计书定义任务环境、agent 配置、skills、tools、任务合同和 runtime assembly 的边界。它的目标不是增加一层配置名词，而是让系统配置权威稳定下来：环境提供静态资源边界和静态能力候选，agent 提供动态权限和行为边界，任务合同提供本次目标和验收，runtime 只做装配与交集过滤。

## 0. 结论

任务环境不是任务类型，也不是工具权限包。

任务环境的定义是：

```text
TaskEnvironment = Resource Boundary + Runtime Constraints + Static Prompt + Static Skill Candidates + Environment Storage + Task Library
```

agent 配置的定义是：

```text
AgentProfile = Operation Permission Ceiling + Runtime Modes + Behavior Policies + Memory Policies + Delegation Policies + Approval Policies
```

runtime assembly 的职责是：

```text
Resolve Environment
-> Resolve Agent Profile
-> Resolve Task Contract
-> Project Skills
-> Authorize Tools
-> Build Runtime Packet
```

禁止的方向：

```text
环境不能授予工具。
agent 权限不能扩大环境资源边界。
task contract 不能绕过 agent 权限和环境约束。
runtime assembly 不能根据关键词、任务类型或模式发明权限。
skills 不能等同于 tools。
模式不能绑定环境。
环境不能变成任务分类树。
```

## 1. 当前代码事实

### 1.1 已存在结构

| 模块 | 当前事实 | 设计判断 |
| --- | --- | --- |
| `backend/task_system/environments/models.py` | `TaskEnvironmentSpec` 已有 `sandbox_policy`、`file_management`、`resource_space`、`memory_space`、`execution_policy`、`risk_policy`、`artifact_policy`、`runtime_policy` | 环境已经具备资源边界和约束模型的主体 |
| `backend/task_system/environments/catalog.py` | 已新增 `TaskEnvironmentCatalog`，runtime/API 共用 catalog payload | catalog 应成为环境配置唯一出口 |
| `backend/agent_system/profiles/runtime_profile_models.py` | `AgentRuntimeProfile` 持有 `allowed_operations`、`blocked_operations`、runtime modes、delegation、approval | 工具权限主权应归 agent/system 授权链 |
| `backend/harness/runtime/assembly.py` | runtime assembly 使用 agent profile operations 构建 authorized tool set | 工具装配应继续以 agent 权限为主，再受环境约束过滤 |
| `backend/task_system/services/bindings.py` | `TaskBindingRecord.skill_scope` 当前来自任务定义绑定 | 这是旧路径，后续应改成 task 显式要求，不再作为默认 skill 主要来源 |
| `backend/task_system/tasks/assembly_policy_resolver.py` | specific task policy 支持 `skill_requirements` 和 `tool_capability_requirements` | 特定任务可以声明显式 skill/tool 要求，但不能替代环境和 agent 权威 |
| `backend/capability_system/skill_registry.py` | skill 有 prompt view、runtime contract、required operations | skills 是方法/能力提示候选，可能引用工具需求，但本身不是权限 |

### 1.2 当前缺口

| 缺口 | 风险 | 修复方向 |
| --- | --- | --- |
| `TaskEnvironmentSpec` 没有 `skill_space` | skills 只能从任务定义或特定任务策略拼出来，环境无法提供静态候选 | 给环境增加 `SkillSpace` |
| tools 和 environment constraints 尚未形成明确交集模型 | agent 有工具权限时，环境约束可能只停留在 prompt 或 sandbox policy | 建立 `OperationAuthorizationProjection` |
| task binding 的 `skill_scope` 仍像旧任务类型默认能力 | 容易把任务类型重新变成控制权威 | 迁移为 task contract 显式 skill requirement |
| 前端 task environment 管理只展示资源，不完整支持编辑 | 环境配置仍偏代码内置 | 建立可持久化环境配置库 |

## 2. 权威边界

### 2.1 环境配置负责什么

环境配置负责静态、可缓存、可复用的系统边界。

环境配置拥有：

```text
resource boundary:
  workspace roots
  storage namespace
  artifact roots
  cache roots
  task library roots
  file profile refs
  repository kinds
  material mount policy

runtime constraints:
  sandbox mode
  real workspace access
  write scope policy
  shell/browser/network policy
  side effect policy
  publish policy
  review/commit gate policy

static prompts:
  environment prompts explaining resources and constraints

static skill candidates:
  default skill candidates for this environment
  optional skill candidates
  denied skill candidates
  skill prompt visibility policy

environment task library:
  specific task records available under this resource boundary
  environment-local task templates
  environment-local examples and contracts
```

环境配置不拥有：

```text
agent allowed_operations
agent blocked_operations
runtime mode
model choice
delegation permission
human approval policy
task lifecycle trigger
tool authorization final decision
user intent decision
```

### 2.2 agent 配置负责什么

agent 配置负责动态行为、权限上限和能力边界。

agent 配置拥有：

```text
allowed_operations
blocked_operations
enabled_runtime_modes
default_runtime_mode
allowed_runtime_lanes
memory read/write policy
context section policy
delegation policy
approval policy
self-review intensity
work role prompt
model profile
```

agent 配置不拥有：

```text
environment storage root
artifact root
file repository topology
sandbox root policy
environment task library
environment prompt
```

### 2.3 task contract 负责什么

任务合同只负责本次任务。

任务合同拥有：

```text
user_visible_goal
task_run_goal
completion_criteria
required_artifacts
required_verifications
resource_requirements
permission_requirements
skill_requirements
acceptance_policy
recovery_policy
```

任务合同不能：

```text
授予 agent 不具备的工具权限
扩大环境资源边界
绕过 sandbox
跳过 artifact 验收
把 task type 当成执行路径
```

### 2.4 runtime assembly 负责什么

runtime assembly 只做解析、交集、投影，不做语义猜测。

runtime assembly 输入：

```text
task_environment_id
agent_profile_id
runtime_mode
task_contract
available tool definitions
skill registry
system safety policy
```

runtime assembly 输出：

```text
RuntimeAssembly
RuntimeEnvelope
RuntimeInvocationPacket
AuthorizedToolSet
SkillCandidatePacket
FileAccessTables
SandboxPolicy
ArtifactPolicy
EnvironmentPromptPacket
```

runtime assembly 禁止输出：

```text
意图分类
任务类型判断
基于关键词的 tool/skill 选择
未经 agent 权限确认的工具
未经环境约束确认的资源路径
```

## 3. 配置模型

### 3.1 TaskEnvironmentSpec 目标结构

当前 `TaskEnvironmentSpec` 保留，并新增 `skill_space`。

```python
@dataclass(frozen=True, slots=True)
class SkillSpace:
    default_skill_refs: tuple[str, ...] = ()
    optional_skill_refs: tuple[str, ...] = ()
    denied_skill_refs: tuple[str, ...] = ()
    prompt_visibility: str = "model_visible_candidates"
    selection_policy: str = "agent_decides"
    activation_policy: str = "task_or_agent_selected"
    required_operation_policy: str = "declare_only"
    authority: str = "task_environment.skill_space"
```

含义：

| 字段 | 含义 |
| --- | --- |
| `default_skill_refs` | 环境默认展示给 agent 的技能候选 |
| `optional_skill_refs` | 环境可按任务合同或 agent 偏好补充的候选 |
| `denied_skill_refs` | 当前环境不应装配的技能 |
| `prompt_visibility` | 是否进入模型可见 packet |
| `selection_policy` | agent 自主判断是否使用 |
| `activation_policy` | skill 激活来自任务或 agent 选择，不由系统硬判 |
| `required_operation_policy` | skill 需要的工具只作为声明，不能授予权限 |

### 3.2 AgentRuntimeProfile 目标结构

现有字段继续作为权限主权来源。

```python
allowed_operations: tuple[str, ...]
blocked_operations: tuple[str, ...]
enabled_runtime_modes: tuple[str, ...]
default_runtime_mode: str
allowed_runtime_lanes: tuple[str, ...]
allowed_memory_scopes: tuple[str, ...]
can_delegate_to_agents: bool
approval_policy: str
metadata.work_role_prompt: str
```

可新增但不强制：

```python
preferred_skill_refs: tuple[str, ...]
blocked_skill_refs: tuple[str, ...]
skill_prompt_density: str
```

新增字段如果进入 agent profile，必须只表示偏好和过滤，不能覆盖环境 `denied_skill_refs`。

### 3.3 TaskContract skill requirements

任务合同可以显式要求 skill。

```json
{
  "skill_requirements": {
    "required_refs": ["skill.browser-verification"],
    "optional_refs": ["skill.test-debugging"],
    "denied_refs": []
  }
}
```

规则：

```text
required_refs 不代表工具授权。
required_refs 只代表 runtime packet 应优先展示该 skill。
如果 required skill 的 required_operations 超出 agent 权限，skill 仍可显示为 unavailable 或 partial，不得放行工具。
```

## 4. 装配流程

### 4.1 固定流程

```text
1. ResolveEnvironment
   input: task_environment_id
   output: EnvironmentResourcePacket

2. ResolveAgentProfile
   input: agent_profile_id
   output: AgentPermissionPacket

3. ResolveTaskContract
   input: model action request / specific task contract
   output: TaskRequirementPacket

4. ProjectSkillCandidates
   input:
     environment.skill_space
     task.skill_requirements
     agent preferred/blocked skills
     skill registry
   output:
     SkillCandidatePacket

5. AuthorizeTools
   input:
     agent.allowed_operations
     agent.blocked_operations
     task.permission_requirements
     environment.execution_policy
     environment.sandbox_policy
     tool definitions
   output:
     AuthorizedToolSet

6. BuildRuntimePacket
   input:
     environment prompts
     skill candidate cards
     authorized tools
     file access tables
     sandbox policy
     artifact policy
     task contract
   output:
     RuntimeInvocationPacket
```

### 4.2 Skill projection algorithm

```text
candidate = environment.default_skill_refs
candidate += environment.optional_skill_refs if task asks or agent prefers
candidate += task.skill_requirements.required_refs
candidate += task.skill_requirements.optional_refs
candidate += agent.preferred_skill_refs

candidate -= environment.denied_skill_refs
candidate -= task.skill_requirements.denied_refs
candidate -= agent.blocked_skill_refs

for each skill:
  load SkillDefinition from SkillRegistry
  inspect required_operations
  mark availability:
    available if required_operations subset authorized_operations
    partial if some required operations missing
    unavailable if all required operations blocked
```

关键点：

```text
skill required_operations 只能影响 skill availability 标记。
skill required_operations 不能反向授予 tools。
```

### 4.3 Tool authorization algorithm

```text
operation_candidates = agent.allowed_operations
operation_candidates -= agent.blocked_operations

operation_candidates = filter_by_environment_constraints(
  operation_candidates,
  environment.execution_policy,
  environment.sandbox_policy,
  environment.risk_policy,
)

operation_candidates = filter_by_task_permission_requirements(
  operation_candidates,
  task.permission_requirements,
)

authorized_tools = build_authorized_tool_set(
  definitions_by_name,
  operation_candidates,
)
```

环境过滤不是授权源，只是约束源。

示例：

```text
agent 有 op.shell
environment.shell_execution_policy = denied
=> terminal 不装配

agent 没有 op.shell
environment.shell_execution_policy = sandboxed
=> terminal 仍不装配

agent 有 op.shell
environment.shell_execution_policy = sandboxed
=> terminal 可装配，但执行上下文必须使用 sandboxed shell
```

## 5. Prompt 装配边界

prompt 必须分层，不能把开发说明当成 agent prompt。

| Prompt 类型 | 来源 | 内容 |
| --- | --- | --- |
| environment prompt | `TaskEnvironmentSpec.environment_prompts` | 资源边界、约束、sandbox、artifact、存储说明 |
| skill prompt | `SkillRegistry` + `SkillCandidatePacket` | 方法、适用条件、输入输出边界、禁止用途 |
| agent prompt | `AgentRuntimeProfile` | 工作角色、行为强度、自审、委派、审批风格 |
| task prompt | `TaskContract` | 本次目标、交付物、验收、恢复要求 |
| runtime control prompt | `RuntimeCompiler` | action schema、输出协议、禁止泄露内部控制 |

禁止：

```text
在 environment prompt 中授予工具。
在 skill prompt 中绕过权限。
在 agent prompt 中覆盖环境写入边界。
在 task prompt 中重新解释环境。
```

## 7. 任务库归属

环境负责维护任务库位置，但不负责自动选择任务。

```text
storage/task_environments/{namespace}/task_library
```

任务库内容：

```text
specific task templates
task contracts
acceptance profiles
example resource declarations
environment-local prompt resources
```

使用规则：

```text
用户显式选择 specific task:
  task system 直接读取 task contract
  runtime 使用 task_environment_id 装配环境

agent 自己决定开启 TaskRun:
  agent 在 runtime 内提出 request_task_run
  系统 admission 校验合同
  如果合同引用环境任务库资源，则 resolve refs

系统不能因为环境任务库里有匹配名称就自动替 agent 选择任务。
```

## 10. 反模式清单

以下结构一旦出现，应删除或重构：

```text
通过 runtime_mode 推导 task_environment_id。
通过 task_environment_id 直接授予 tools。
通过 skill required_operations 反向放行 tools。
通过 task type 自动开启 TaskRun。
通过关键词选择 specific task。
在 environment prompt 中写“你有 terminal 权限”。
在 agent prompt 中写“你可以忽略 sandbox”。
在 task binding 中长期保留大而全的 default_skill_refs。
前端硬编码 legacy env id 作为默认环境。
```

## 11. 最终目标结构

稳定后，本系统应形成以下配置关系：

```text
Task Environment
  owns resources, constraints, storage, artifact policy, static prompts, skill candidates, task library

Agent Profile
  owns operation permissions, modes, behavior, memory policy, delegation, approval

Task Contract
  owns concrete goal, artifacts, verifications, resource requests, explicit skill requirements

Runtime Assembly
  owns intersection/projection only

Execution Loop
  owns action validation, tool execution, observations, recovery, artifact verification
```

这套边界的关键收益：

```text
配置成本低：skills 由环境静态复用。
权限清晰：tools 只由 agent/system 授权。
缓存友好：environment prompts 和 skill candidates 可静态缓存。
运行稳定：runtime 不再发明任务类型或权限。
扩展自然：新增环境就是新增资源边界和 skill 候选，不需要改 agent 权限模型。
```

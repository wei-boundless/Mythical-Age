状态：实施前设计稿

关联文档：

## 1. 结论

TaskRun 长任务自然语言控制的核心缺口不是某个 `executor_status` 字段没有清理，而是缺少运行中执行轮的控制权。

现有系统已经具备：

```text
1. active work 识别当前工作。
2. 自然语言 turn decision 能判断继续、暂停、停止、追加指令。
3. active_task_steer 能记录用户补充意见。
4. waiting_executor / resume_task_run 能启动下一轮 executor。
5. packet 编译时能把 pending_user_steers 投影进 execution_state。
```

但现有系统缺少：

```text
1. 当前 executor epoch 的运行时句柄。
2. 运行中 model call / tool call 的取消入口。
3. append / overturn / pause / stop 对 in-flight epoch 的统一控制协议。
4. 被用户新指令打断后的 executor closeout 语义。
5. closeout 后自动重新组包并消费 pending steer 的调度规则。
```

因此高压实测中会稳定出现：

```text
active_task_steer_recorded 增加
active_task_steer_included 不增加
active_task_steer_consumed 不增加
task_model_action_waiting 持续存在
executor_status 仍表现为 running
```

这不是随机性，也不是模型没有听话，而是旧模型调用仍在执行，新的用户指令没有办法进入已经编译好的旧 packet。

## 4. 目标架构

### 4.1 双层控制模型

目标结构：

```text
TaskRunLifecycleControl
  负责整个长任务生命周期：
  stop / stopped / terminal cleanup / runtime restart recovery

ExecutorEpochControl
  负责当前执行轮：
  pause current epoch
  interrupt current epoch for replan
  cancel current epoch for stop
  clear runtime lease
```

对应 Claude Code：

```text
abortController                  -> TaskRun lifecycle
currentWorkAbortController        -> Executor epoch
```

对应 Codex：

```text
RunningTask.cancellation_token    -> active turn / task cancellation
turn-local pending_input          -> next request drain
```

### 4.2 固定执行流

```text
用户追加/推翻方向
-> active_work decision = append_instruction_to_active_work
-> append_user_work_instruction 记录 steer
-> 如果 TaskRun running：
     request_executor_replan(task_run_id, steer_ref)
     当前 model call 被 cancel
     当前 epoch closeout 为 waiting_executor + interrupted_for_replan
     scheduler 立即开新 executor epoch
     新 packet 包含 pending_user_steers / active_contract_revisions
     模型必须消费 steer 后才能完成
-> 如果 TaskRun waiting_executor：
     resume_paused_task_run
     scheduler 开新 executor epoch
```

```text
用户暂停
-> request_task_run_pause
-> 如果 TaskRun running：
     request_executor_pause(task_run_id)
     当前 model call 被 cancel
     closeout 为 waiting_executor + paused
     不自动调度
-> 如果 TaskRun waiting_executor：
     直接标记 paused
```

```text
用户停止
-> stop_task_run
-> 如果 TaskRun running：
     request_executor_stop(task_run_id)
     当前 model/tool call 被 cancel
     closeout 为 aborted + stopped
     不自动调度
-> 如果 TaskRun waiting_executor：
     直接 terminal closeout
```

```text
用户问进度
-> answer_about_active_work
-> 不 interrupt
-> 不记录 steer
```

### 4.3 状态不变量

```text
1. status=running 且 executor_status=running 时，必须存在当前进程内 executor epoch control。
2. executor epoch control 消失时，持久状态不得继续保留 executor_status=running。
3. pause_requested / stop_requested / replan_requested 必须能被 _await_task_model_action_with_status 观察。
4. replan closeout 必须进入 waiting_executor，而不是 failed/blocked。
5. paused 状态不得被 auto_continue 调度。
6. replan 状态如果存在 pending steer，应自动进入下一 executor epoch。
7. completed 前 pending_user_steer_count 和 active_contract_revision_count 必须为 0。
8. status question 不得触发 interrupt。
```

## 5. 文件级改造方案

### 5.1 新增 `task_run_execution_control.py`

建议路径：

```text
backend/harness/loop/task_run_execution_control.py
```

职责：

```text
1. 维护当前进程内 executor epoch registry。
2. 保存 task_run_id、executor_epoch、model asyncio task、创建时间。
3. 接收 pause / stop / replan 信号。
4. 提供 executor 轮询检查接口。
5. 在 closeout / finally 中清理 epoch registry。
```

建议接口：

```python
@dataclass
class ExecutorControlSignal:
    kind: Literal["pause", "stop", "replan"]
    reason: str
    requested_by: str
    steer_ref: str = ""
    requested_at: float = 0.0

def register_executor_epoch(runtime_host, task_run_id, executor_epoch, *, model_task=None) -> None: ...
def attach_model_task(runtime_host, task_run_id, executor_epoch, model_task) -> None: ...
def request_executor_pause(runtime_host, task_run_id, *, reason, requested_by) -> bool: ...
def request_executor_stop(runtime_host, task_run_id, *, reason, requested_by) -> bool: ...
def request_executor_replan(runtime_host, task_run_id, *, steer_ref, reason, requested_by) -> bool: ...
def peek_executor_signal(runtime_host, task_run_id, executor_epoch) -> ExecutorControlSignal | None: ...
def clear_executor_epoch(runtime_host, task_run_id, executor_epoch) -> None: ...
```

存储位置：

```text
runtime_host 进程内属性，例如 runtime_host.task_run_execution_control
```

注意：

```text
不能只写 diagnostics。
diagnostics 是恢复与投影，不是运行句柄。
```

### 5.2 修改 `task_executor.py`

受影响函数：

```text
execute_task_run
_await_task_model_action_with_status
append_user_work_instruction
request_task_run_pause
stop_task_run
_finish_executor_terminal
_normalize_executor_terminal_closeout
```

改造点：

```text
1. claim executor sequence 后 register_executor_epoch。
2. _await_task_model_action_with_status 创建 model_task 后 attach_model_task。
3. 等待循环改为短周期控制检查 + 低频进度记录。
4. 收到 pause / stop / replan 后 cancel model_task。
5. 抛出或返回结构化中断结果，由 execute_task_run 做 closeout。
6. finally 清理 executor epoch registry。
```

建议控制检查节奏：

```text
control_check_interval: 1.0s
progress_summary_interval: 15.0s
```

不要继续让控制响应被 15 秒等待间隔限制。

新增 terminal reasons：

```text
task_run_paused
user_interrupt_replan_required
user_aborted
```

新增或统一 recovery_action：

```text
pause -> resume_task_run，但 paused 阻止 auto_continue
replan -> resume_task_run 或 rerun_task_executor
stop -> 无 recovery_action
```

## 8. 冲突与遗漏审查

### 8.1 与现有 pause/resume 是否冲突

不冲突，但要修正语义边界。

现有 pause 对 waiting_executor 已能直接变 paused。新增能力只处理 running epoch：

```text
waiting_executor pause -> 持久状态变 paused
running pause -> 先 interrupt current epoch，再持久状态变 paused
```

resume 仍从 waiting_executor 调度，不需要直接恢复旧 model call。成熟 agent 也不会把已取消的旧模型调用继续使用，而是重新组包。

### 8.2 与 recovery_state 是否冲突

不冲突。

`task_run_recovery_state.py` 应继续做持久状态归一判断；新增的 execution control 是进程内运行句柄。二者边界必须固定：

```text
recovery_state 不 cancel task。
execution_control 不决定持久可恢复策略。
```

### 8.3 与 graph_node_assigned 是否冲突

需要保留现有边界。

当前控制函数拒绝 `graph_node_assigned`。新增 replan/pause/stop 也必须遵守：

```text
graph_node_assigned TaskRun 由 graph runtime 控制。
single_agent_task 自然语言控制不得直接抢 graph executor 权限。
```

后续如要支持图任务中断，应在 graph runtime 层另设控制协议，不在本次改造里顺手打通。

### 8.4 与 stop_task_run 是否冲突

需要明确 stop 优先级最高。

如果同一 epoch 同时收到 replan 和 stop：

```text
stop > pause > replan
```

原因：

```text
stop 是用户终止意图，不能被 pending steer 自动续跑覆盖。
```

### 8.5 与 status question 是否冲突

必须防止误中断。

`answer_about_active_work` 只读 monitor / rollout / active_work_context，不得：

```text
记录 steer
request replan
request pause
schedule executor
```

### 8.6 与 pending steer 消费是否冲突

不冲突。

现有 pending steer include/consume 机制应保留。新增 replan 的目的不是消费 steer，而是强制尽快重新 compile packet，让现有机制消费 steer。

### 8.7 与 terminal closeout 是否冲突

需要避免把 replan 当 failed。

`user_interrupt_replan_required` 是控制性中断，不是任务失败。它应归一为：

```text
status=waiting_executor
executor_status=waiting_executor
recovery_action=resume_task_run 或 rerun_task_executor
recoverable_error.retryable=true
```

### 8.8 与测试真实性是否冲突

不能只用 stub 模型证明成功。

stub 压测只能验证状态机和 packet 链路。最终必须用真实 API 证明：

```text
in-flight model call 被用户控制打断后，
新 epoch 能重新组包并消费用户补充指令。
```

## 9. 不做事项

本次不做：

```text
1. 不重写 active_work 语义分类 prompt。
2. 不把 graph runtime 控制合并进 single_agent_task 控制。
3. 不删除 pending steer / contract revision 机制。
4. 不把 monitor 层改成状态修复层。
5. 不靠缩短模型 timeout 伪造中断能力。
6. 不用测试 stub 硬编码 consumed_steer_refs 当成功依据。
```

## 10. 最终验收口径

修复完成后，以下说法必须都成立：

```text
1. TaskRun running 时，系统知道当前 executor epoch 的运行时句柄。
2. 用户追加/推翻方向能中断当前 epoch。
3. 中断后不会丢 pending steer。
4. 新 epoch 一定重新 compile packet。
5. pending_user_steers 和 active_contract_revisions 最终清零。
6. pause 停在 waiting_executor + paused，且不自动续跑。
7. stop 进入 terminal stopped，且不自动续跑。
8. status question 不改变执行状态。
9. runtime restart 后没有 running lease 污染恢复判断。
10. 真实 API 高压实验能提供运行目录、耗时、事件计数和最终状态。
```

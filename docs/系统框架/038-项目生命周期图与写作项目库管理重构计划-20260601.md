# 项目生命周期图与写作项目库管理重构计划

日期：2026-06-01

## 1. 结论

本次重构不再采用 `TaskProjectBinding`。

正确方向是：

```text
任务环境
-> 项目实例
-> 项目库 manifest
-> 项目生命周期图
-> 写作执行图
-> 文件 / 产物 / 记忆 / 索引
```

项目管理必须纳入任务图系统，但要和写作执行图分离：

- 写作执行图负责创作、审核、提交候选、记忆候选。
- 项目生命周期图负责建库、迁移、采纳、归档、清理、索引、引用修复。
- 前端只发起项目动作、查看影响预览、提交审批信号，不直接迁移文件或修改底层 JSON。

## 2. 成熟工程基线

本方案采用三类成熟系统的工程原则：

1. Durable execution：项目管理动作必须可恢复、可重试、可审计，不能依赖前端一次性按钮完成。
2. BPMN / workflow control：需要人类确认的步骤应成为流程节点，流程暂停等待信号，再继续执行。
3. Artifact repository：运行产物必须以逻辑引用和元数据管理，不由节点或前端决定物理路径。

落到本项目里，核心约束是：

- 项目动作必须有 `lifecycle_run_id`。
- 运行中断后可以续跑。
- 每个管理动作要有影响预览、执行日志、失败原因和验证结果。
- 文件入口统一使用 `project_id + repository_id + logical_path`。
- 不允许前端传物理路径。
- 不允许 `TaskProjectBinding` 这种后期绑定关系继续作为权威。

## 3. 当前问题

### 3.1 任务与节点混淆

当前写作系统里存在大量 `task.writing.modular_novel.node.*` 任务记录。这些不是用户意义上的写作项目任务，而是任务图节点。

它们不应该出现在“任务实例管理”列表中。

目标是：

```text
项目实例：洪荒时代
图节点：world_design / world_review / character_design / chapter_draft ...
```

项目实例是用户管理对象，图节点是运行内部对象。

### 3.2 库模型不够精细

只靠“任务执行时的库”不够。写作系统至少需要：

```text
环境资源库
模板库
项目实例库
运行产物库
项目记忆库
环境记忆候选库
索引库
```

否则会出现三个问题：

1. 通用写作资源无法复用。
2. 项目级事实、运行中间状态、环境级偏好混在一起。
3. 图任务产物无法稳定采纳、回滚、续跑和追踪来源。

### 3.3 项目管理不能靠用户手动迁移

用户不应该手动搬文件、改目录、建索引、修引用。

用户只管理：

- 新建项目
- 选择项目模板
- 启动写作流程
- 审批库升级
- 采纳产物
- 归档 / 删除 / 清理

系统负责：

- 建库
- 套用模板
- 建 manifest
- 写入产物
- 维护索引
- 修复引用
- 迁移库结构
- 验证完整性

### 3.4 前一版实现的方向错误

以下新增结构需要删除或改造：

- `backend/task_system/projects/task_project_binding.py`
- `backend/task_system/repositories/task_project_binding_repository.py`
- `backend/tests/task_project_binding_regression.py`
- `GET /api/tasks/{task_id}/project`
- `PUT /api/tasks/{task_id}/project`
- 前端 `TaskProjectBinding` 类型与相关调用

原因：它把“项目实例拥有项目库”错误地做成了“任务后期绑定项目”。

## 4. 目标领域模型

### 4.1 TaskEnvironment

任务环境管理一类工作空间的资源边界。

写作环境应拥有：

```text
环境 prompt
节点配置模板
契约库
图模板库
项目模板库
环境资源库
环境记忆候选库
文件 profile
权限策略
```

### 4.2 ProjectInstance

项目实例是用户真正管理的对象。

示例：

```json
{
  "project_id": "project.creation.writing.honghuang",
  "environment_id": "env.creation.writing",
  "title": "洪荒时代",
  "project_kind": "long_novel",
  "template_id": "writing.template.long_novel.commercial",
  "library_id": "library.project.creation.writing.honghuang",
  "lifecycle_state": "active",
  "schema_version": "writing_library.v1",
  "created_at": 1780000000,
  "updated_at": 1780000000
}
```

一个项目实例创建时自动拥有一个项目库。这里不需要用户绑定。

### 4.3 ProjectLibraryManifest

项目库 manifest 是库状态权威。

示例：

```json
{
  "library_id": "library.project.creation.writing.honghuang",
  "project_id": "project.creation.writing.honghuang",
  "environment_id": "env.creation.writing",
  "schema_version": "writing_library.v1",
  "template_id": "writing.template.long_novel.commercial",
  "repositories": [
    {
      "repository_id": "repo.writing.official_work",
      "role": "official_work",
      "root_ref": "project://official_work",
      "lifecycle": "canonical",
      "commit_gate": "review_required"
    },
    {
      "repository_id": "repo.writing.draft_workspace",
      "role": "draft_workspace",
      "root_ref": "project://drafts",
      "lifecycle": "working"
    },
    {
      "repository_id": "repo.writing.artifact_repository",
      "role": "artifact_repository",
      "root_ref": "project://artifacts",
      "lifecycle": "run_output"
    },
    {
      "repository_id": "repo.writing.project_memory",
      "role": "project_memory",
      "root_ref": "project://memory/project",
      "lifecycle": "committed_memory"
    }
  ],
  "indexes": {
    "file_index": "ready",
    "artifact_index": "ready",
    "memory_index": "ready"
  },
  "migration_log": []
}
```

manifest 不暴露物理路径。物理路径由后端 resolver 根据 `project_id`、`root_ref` 和环境 storage namespace 解析。

### 4.4 ProjectLifecycleGraph

项目管理动作都变成任务图运行。

第一批生命周期图：

```text
project.lifecycle.create
project.lifecycle.migrate_library
project.lifecycle.promote_artifact
project.lifecycle.archive_project
project.lifecycle.cleanup_legacy_writing_tasks
project.lifecycle.rebuild_indexes
project.lifecycle.verify_integrity
```

这些图不写小说内容，只管理项目状态、库、索引、引用和采纳。

### 4.5 WritingExecutionGraph

写作执行图继续负责创作生产。

建议保留或演进为：

```text
writing.design_init
writing.chapter_cycle
writing.volume_cycle
writing.revision_cycle
writing.finalize
```

写作执行图输出产物和候选，不直接改正式库。

正式采纳必须通过生命周期图：

```text
writing output artifact
-> project.lifecycle.promote_artifact
-> official_work / project_memory / index
```

## 5. 写作专属精细化体系

### 5.1 写作设计体系

写作项目需要一个可管理的设计体系，而不是把世界观、大纲、角色散落在产物里。

建议建立 `WritingDesignSystem`：

```text
题材与市场定位
读者画像
卖点与爽点模型
世界观规则
力量体系
人物体系
势力体系
剧情主线
分卷结构
章节节奏
伏笔与回收
风格规范
禁用事项
连续性规则
```

落地方式：

- 作为项目库中的 `design_system` repository 或 project memory collection。
- 由设计初始化图生成候选。
- 由审核节点裁决。
- 由生命周期图提交为项目级权威设计。
- 后续章节图只能读取已提交设计，不能随意改写。

### 5.2 写作框架管理

写作框架不是单个 prompt，而是一套可版本化模板。

建议建立 `WritingFrameworkTemplate`：

```text
长篇商业网文框架
短篇故事框架
设定集框架
角色驱动框架
剧情驱动框架
单元剧框架
多卷长线框架
```

每个框架模板包含：

```text
推荐图模板
默认节点配置
契约族
项目库结构
设计体系字段
章节节奏策略
审核标准
记忆分层策略
产物采纳策略
```

用户创建项目时只选框架，不手动搭目录。

### 5.3 创作流程管理

创作流程要从“节点串联”升级为“阶段状态机 + 图运行”。

建议阶段：

```text
项目立项
设计初始化
设计审核
大纲冻结
章节生产
章节审核
分卷复盘
设定扩展
修订
完结归档
```

每个阶段应定义：

- 允许启动的图模板
- 入口条件
- 输出契约
- 可采纳产物类型
- 可写入记忆层
- 失败恢复策略
- 人工审批点

### 5.4 记忆分层

写作记忆必须分层：

```text
run_memory：本次运行临时观察和中间状态
project_memory_candidate：待审核项目事实
project_memory_committed：已确认项目事实
environment_memory_candidate：可能复用到环境的经验
environment_memory_committed：管理员确认后的环境级规则
```

节点不能直接写环境级长期记忆。环境级记忆必须走候选和审批。

### 5.5 产物采纳规则

写作执行图只产生候选：

```text
world_design_candidate
character_design_candidate
outline_candidate
chapter_draft
review_report
memory_commit_candidate
continuity_issue
```

生命周期图负责采纳：

```text
candidate
-> review receipt
-> promote
-> official repository / project memory / index
```

### 5.6 产物投影到设计体系

图任务产物系统必须成为项目库结构化生长的来源，但不能让节点自己决定落盘位置。

正确链路是：

```text
节点 prompt / 契约
-> 结构化产物
-> ArtifactRepositoryService.record_materialization()
-> ArtifactRecord
-> ArtifactProjectionRule
-> ProjectLibrary candidate area
-> ProjectLifecycleGraph adoption
-> committed design system / project memory / index
```

当前代码已有基础：

- `ArtifactRecord` 已保存 `artifact_ref`、`output_contract_id`、`artifact_kind`、`producer_node_id`、`graph_run_id`、`status`、`metadata`。
- `ContractSpec.artifact_requirements` 已能声明产物要求和 `artifact_type`。
- 图运行语义已经区分 `producer`、`validator`、`approver`、`publisher`，并用边语义表达 `validation_input`、`publish_input`、`resource_write`。

缺失的是中间投影层：

```text
ArtifactTypeCatalog
ArtifactProjectionRule
DesignSystemSection
LifecycleAdoptionRule
```

#### 5.6.1 ArtifactTypeCatalog

`ArtifactType` 是系统语义，不是文件扩展名。

示例：

```text
world_design_candidate
power_system_candidate
faction_design_candidate
character_profile_candidate
plot_outline_candidate
volume_outline_candidate
chapter_draft
review_report
continuity_issue
memory_commit_candidate
style_rule_candidate
```

每个类型必须定义：

- 归属写作阶段。
- 默认输出契约。
- 是否可被直接展示。
- 是否需要审核。
- 是否可进入项目记忆。
- 是否可进入环境记忆候选。
- 是否允许被章节生成图读取。

#### 5.6.2 DesignSystemSection

写作设计体系必须有稳定槽位。

示例：

```json
{
  "section_id": "worldbuilding.power_system",
  "title": "力量体系",
  "parent_section_id": "worldbuilding",
  "accepted_artifact_types": ["power_system_candidate"],
  "read_visibility": "committed_only",
  "write_policy": "lifecycle_adoption_only",
  "versioning": "append_with_supersede",
  "required_review": true
}
```

设计体系槽位不直接等于文件夹。它是项目设计权威的语义位置，可以映射到：

- project memory collection
- design system repository
- artifact index
- 前端设计体系视图

#### 5.6.3 ArtifactProjectionRule

`ArtifactProjectionRule` 决定某类产物进入项目库的哪个候选区。

示例：

```json
{
  "rule_id": "projection.world_design.to_design_system",
  "environment_id": "env.creation.writing",
  "project_kind": "long_novel",
  "artifact_type": "world_design_candidate",
  "source_contract_id": "contract.writing.world_design.output",
  "target": {
    "repository_id": "repo.writing.design_system",
    "section_id": "worldbuilding",
    "state": "candidate"
  },
  "adoption_graph_id": "project.lifecycle.promote_design_artifact",
  "read_visibility": "not_visible_until_committed",
  "failure_policy": "quarantine_unmapped_artifact"
}
```

规则要求：

- 只能引用 manifest 中存在的 repository 或设计体系 section。
- 不能写物理路径。
- 未匹配的产物进入隔离区，不得自动进入项目记忆。
- 规则必须版本化，生命周期 run 记录所用规则版本。
- 同一 `artifact_type + source_contract_id + project_kind` 只能有一个主投影规则。

#### 5.6.4 LifecycleAdoptionRule

投影只进入候选区，采纳必须由生命周期图执行。

示例：

```json
{
  "rule_id": "adopt.world_design_candidate",
  "artifact_type": "world_design_candidate",
  "required_review_artifact_type": "review_report",
  "required_verdict": "approved",
  "target_state": "committed",
  "write_targets": [
    {
      "repository_id": "repo.writing.design_system",
      "section_id": "worldbuilding"
    },
    {
      "repository_id": "repo.writing.project_memory",
      "collection_id": "memory.writing.baseline"
    }
  ],
  "post_actions": ["refresh_file_index", "refresh_memory_index", "record_provenance"]
}
```

采纳规则要求：

- 没有审核回执不得提交到 committed。
- 采纳动作必须写 provenance：来源图、节点、run、契约、hash、审核结论。
- 被新版本取代的设计不能直接删除，应标记 superseded。
- 后续写作图只能读取 committed 设计，不读取 candidate。

#### 5.6.5 Prompt 与契约边界

具体写什么仍然由 prompt 和契约决定。

系统抽象只决定：

```text
这是什么类型的产物
它进入哪个候选槽位
谁能读
需要什么审核
如何采纳
如何索引
如何回滚
```

Prompt 不能写成开发说明。节点 prompt 要描述创作者、审核员、结构师的职责。契约负责声明输出字段和 artifact requirement。投影规则负责系统流转。

#### 5.6.6 可拓展方式

新增一个细分需求时，不应该改核心代码。

例如新增“宗门势力体系”：

1. 新增 `DesignSystemSection = worldbuilding.factions`.
2. 新增 `ArtifactType = faction_design_candidate`.
3. 新增或复用输出契约。
4. 新增 `ArtifactProjectionRule` 投影到 `worldbuilding.factions`。
5. 新增 `LifecycleAdoptionRule` 定义审核和提交方式。
6. 需要时新增节点 prompt 模板。

这样可以细分拓展世界观、人物、剧情、节奏、伏笔、风格、商业定位，而不扩大前端复杂度，也不让节点硬编码文件路径。

```text
PATCH /api/tasks/projects/{project_id}
```

创建项目时必须自动创建项目库 manifest，并启动或记录 `project.lifecycle.create`。

```text
GET /api/tasks/projects/{project_id}/library
```

禁止：

- `/api/files` 读取任务环境库。
- 前端传物理路径。

## 9. 实施阶段

### 阶段 1：计划切换与错误抽象清理

目标：

- 删除 `TaskProjectBinding` 抽象。
- 删除相关 API 和前端类型。
- 保留项目库读取能力，但入口改为 `project_id`。

完成标准：

- 代码里不再有 `TaskProjectBinding` 决定项目归属。
- 前端不再出现“绑定项目”概念。
- 测试不再保护旧绑定行为。

### 阶段 2：项目实例与 manifest

目标：

- 新增 `ProjectInstance`。
- 新增 `ProjectLibraryManifest`。
- 默认创建 `project.creation.writing.honghuang`。
- manifest 声明写作库分区。

完成标准：

- 能列出写作环境项目实例。
- 能读取项目库 manifest。
- 能通过 `project_id + repository_id + logical_path` 打开真实产物。

### 阶段 3：项目生命周期图最小闭环

目标：

- 实现生命周期运行记录。
- 支持 `cleanup_legacy_writing_tasks` 的 preview 和 execute。
- 支持人工审批信号。

完成标准：

- 清理旧写作节点任务必须先生成影响预览。
- 清理通过生命周期 run 执行，不手动删 JSON。
- 清理后旧节点任务不在项目列表复活。
- 图定义和产物不被误删。

### 阶段 4：写作框架与设计体系

目标：

- 新增写作框架模板。
- 新增项目设计体系视图。
- 把设计初始化产物归入设计体系候选和项目记忆候选。
- 新增产物类型目录、投影规则和采纳规则。

完成标准：

- 项目页能看到设计体系分类。
- 设计体系区分候选、已审核、已提交。
- 章节流程只能读取已提交项目设计，不读取未审核候选。
- `ArtifactRecord.output_contract_id + artifact_type` 能映射到设计体系候选区。
- 未匹配产物进入 quarantine，不自动写入项目记忆。
- 采纳动作必须通过生命周期图记录 provenance。

### 阶段 5：创作流程状态机

目标：

- 新增创作阶段模型。
- 阶段决定可启动图、入口条件、输出契约和采纳规则。

完成标准：

- 页面按阶段展示项目进度。
- 用户从阶段启动写作图。
- 写作图输出进入候选产物，不直接污染正式库。

### 阶段 6：前端管理台重构

目标：

- 创作页变成项目管理与创作工作台。
- 左侧为项目实例，不是节点任务。
- 中间为当前项目工作台。
- 右侧为库、产物、生命周期 run、维护动作。

完成标准：

- 没有后端开发式文案。
- 没有“任务项目绑定”操作。
- 没有物理路径。
- 可打开洪荒时代项目库和真实产物。
- 可执行旧节点任务清理预览。

## 11. 禁止事项

1. 禁止保留 `TaskProjectBinding` 作为兼容路径。
2. 禁止在前端手动迁移文件。
3. 禁止用 `/api/files` 打开任务环境项目库。
4. 禁止前端传 physical path。
5. 禁止写作执行图直接写正式稿或环境级长期记忆。
6. 禁止把节点任务展示成用户项目实例。
7. 禁止把设计体系候选当成已提交项目事实。
8. 禁止清理旧任务时删除图定义和已有产物。
9. 禁止节点 prompt 或前端决定产物物理落点。
10. 禁止未匹配产物自动进入设计体系或项目记忆。
11. 禁止章节图读取未审核候选设计。

## 12. 最终验收

1. 项目管理纳入任务图系统。
2. 项目实例创建时自动建库，不需要绑定。
3. 项目库由 manifest 管理。
4. 写作环境支持精细化设计体系、写作框架、创作阶段。
5. 生命周期图负责迁移、采纳、索引、清理和归档。
6. 创作页是项目工作台，不是节点任务列表。
7. 旧写作节点任务可以通过生命周期图安全清理。
8. 文件、产物、记忆和索引都有明确归属和恢复路径。
9. 图任务产物能通过投影规则进入写作设计体系候选区。
10. 设计体系扩展通过类型、槽位、投影规则和采纳规则完成，不需要修改核心文件管理逻辑。

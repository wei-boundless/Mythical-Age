# 写作图任务运行控制与创作工作区产品化重构计划

日期：2026-05-31

## 1. 目标结论

写作图任务不能把 `memory / durable_memory / artifact / checkpoint` 这些开发概念直接暴露给用户。用户需要看到的是一个成熟写作产品：

- 世界观资料库
- 角色资料库
- 势力与地点资料库
- 大纲与卷纲
- 章节稿件
- 审核意见
- 修订任务
- 连载进度
- 问题清单

底层仍然使用图合同、边合同、记忆协议、artifact 写入、checkpoint 恢复；但前端必须通过产品对象投影来呈现，不能让用户去理解“记忆库文件”。

## 2. 当前问题

### 2.1 运行控制问题

当前监控台只有“续跑”这类工程动作，语义不完整：

- 手动模式缺少人工审核确认。
- 自动模式缺少断链续跑边界，容易把“自动继续”和“无限派发”混在一起。
- 前端按钮直接调用 run-until-idle，无法表达用户在审核门上的裁决。

### 2.2 文件管理问题

当前文件控制系统仍是开发者视角：

- `InspectorPanel` 展示的是固定文件路径，例如 `durable_memory/index/MEMORY.md`。
- `MemoryView` 暴露 durable memory 概念。
- 写作图任务运行期间，用户无法自然进入“本次作品工作区”审核世界观、角色、大纲、章节和问题清单。

这会让写作产品变成开发调试台。

## 3. 权责模型

### 3.1 运行控制权责

| 层级 | 职责 | 禁止事项 |
| --- | --- | --- |
| GraphRuntime | 锁定图配置、创建 graph run、生成 file_scope / memory_scope / artifact_scope | 不做循环推进 |
| GraphLoop | 解释状态机、节点状态、边合同、人工门控、checkpoint | 不直接执行 agent |
| GraphRunRunner | 执行 active work order、断链恢复、跑到 idle/gate/terminal/budget | 不绕过人工门控 |
| 前端监控台 | 观察状态、提交人工裁决、切换自动/手动策略 | 不自行判断图拓扑 |
| 创作工作区 | 展示产品对象、允许人工审核和修改 | 不暴露底层 memory 技术结构 |

### 3.2 文件产品权责

```text
GraphRun memory/artifact/checkpoint refs
  -> WritingWorkspaceManifest
  -> 产品对象列表
  -> 编辑器/审核台
  -> 保存为 artifact 或人工修订包
  -> GraphLoop human gate decision / revision packet
```

底层对象可以是文件、数据库记录或记忆条目，但前端只消费 `WritingWorkspaceManifest`，不直接拼路径。

## 4. 运行控制设计

### 4.1 手动模式

手动模式不是“点一下继续”。正确语义是：

```text
节点完成
  -> GraphLoop 进入 waiting_human_gate
  -> 前端展示本节点产物和影响范围
  -> 用户审核
  -> 用户选择：
       approve_continue
       request_revision
       reroute_to_node
       stop_and_checkpoint
       abort_graph
  -> GraphLoop.apply_human_gate_decision
  -> 根据裁决推进
```

前端按钮应命名为产品动作：

- 通过并继续
- 要求返修
- 停在此处
- 退回某节点
- 终止任务

而不是只显示“续跑当前阶段”。

### 4.2 自动模式

自动模式负责自动推进普通链路，但必须尊重门控：

```text
监控轮询 GraphRun
  -> 如果 waiting_human_gate：停止自动推进，提示需要人工审核
  -> 如果 active_work_orders 存在但 executor 断链：调用 run_until_idle 恢复
  -> 如果 ready_node_ids 存在且没有 active work：延时保护后调用 run_until_idle
  -> 如果 terminal / failed / blocked：停止自动推进并显示原因
```

自动模式的目标是“断链续跑和链路推动”，不是跳过审核。

### 4.3 延时保护

前端开关只改变策略，不立即连续点击派发：

- 开启自动后等待 2-3 秒再触发第一次续跑。
- 同一 GraphRun 同一时刻只能有一个自动续跑请求。
- 若监控状态变化为 `waiting_human_gate / failed / completed`，取消 pending 自动续跑。
- 若用户切回手动，取消 pending 自动续跑。

## 5. 创作工作区设计

### 5.1 产品对象

写作图任务的工作区应按创作对象组织：

| 产品对象 | 底层来源 | 用户动作 |
| --- | --- | --- |
| 项目简介 | seed / project_brief artifact | 查看、修订 |
| 世界观资料库 | baseline memory + world artifacts | 审核、编辑、锁定 |
| 角色资料库 | character artifacts / memory slices | 审核、编辑、锁定 |
| 大纲与卷纲 | outline artifacts | 审核、调整 |
| 章节稿件 | chapter artifacts | 阅读、批注、返修 |
| 审核意见 | review artifacts / issue ledger | 处理、关闭 |
| 连续性问题 | issue ledger / dynamic memory conflicts | 标记解决、退回 |
| 修订任务 | revision packets | 生成返修指令 |

### 5.2 前端页面结构

图任务运行页不应承载所有文件编辑。应拆成三个产品区：

1. 运行监控台  
   显示 GraphRun 状态、自动/手动策略、当前门控、运行健康。

2. 创作工作区  
   展示世界观、角色、大纲、章节、审核意见，不显示底层路径。

3. 审核与返修台  
   当 GraphLoop 停在人工门时，展示本次需要审核的产物、差异、上游依据和裁决按钮。

### 5.3 文件系统接口

新增一个 GraphRun 工作区 manifest，而不是前端硬编码路径：

```json
{
  "authority": "harness.graph_run_writing_workspace",
  "graph_run_id": "...",
  "project_title": "...",
  "sections": [
    {
      "section_id": "world_bible",
      "label": "世界观资料库",
      "items": [
        {
          "item_id": "...",
          "title": "洪荒时代基础设定",
          "kind": "world_fact",
          "status": "approved",
          "editable": true,
          "source_ref": "...",
          "file_ref": "..."
        }
      ]
    }
  ]
}
```

前端只能展示 `label/title/status/kind/editable`，路径只作为保存接口的内部 ref。

## 6. 后端改造点

### 6.1 GraphRun 人工门控 API

新增：

```text
POST /orchestration/harness/graph-runs/{graph_run_id}/human-gate-decisions
```

请求：

```json
{
  "graph_harness_config_id": "...",
  "node_id": "...",
  "human_action": "approve_continue",
  "review_note": "...",
  "route_target_node_id": ""
}
```

该 API 只调用 `GraphHarness.apply_human_gate_decision()`，不自行改状态。

### 6.2 GraphRun 创作工作区 API

新增：

```text
GET /orchestration/harness/graph-runs/{graph_run_id}/writing-workspace
```

职责：

- 读取 GraphRuntimeEnvelope 的 `file_scope / memory_scope`。
- 读取 GraphLoopState 的 artifact refs、result refs、issue ledger refs。
- 输出产品化 `WritingWorkspaceManifest`。
- 隐藏底层文件路径和 memory 技术名。

### 6.3 文件保存

用户编辑产品对象时，不直接写任意 durable memory 文件：

```text
产品对象保存
  -> workspace item save API
  -> artifact revision 或 manual_edit patch
  -> GraphLoop human gate decision 引用该 patch
```

这样可以保证人工修改可追踪、可回滚、可进入后续边合同。

## 7. 前端改造点

### 7.1 监控台

将运行控制改为策略开关：

- 手动审核模式
- 自动推进模式

手动审核模式显示：

- 当前等待审核节点
- 产物摘要
- 打开创作工作区
- 通过并继续
- 要求返修
- 停在此处

自动推进模式显示：

- 自动推进已开启
- 最近一次自动续跑时间
- pending 延时保护状态
- 若进入人工门：自动暂停，显示“需要人工审核”

### 7.2 创作工作区

新增运行期工作区组件：

```text
TaskGraphWritingWorkspace
  左侧：资料类型导航
  中间：产品对象列表 / 文档阅读
  右侧：审核状态、引用来源、修改记录、保存动作
```

不展示 `durable_memory/xxx.md` 这类路径。路径只能作为调试详情折叠展示，默认隐藏。

### 7.3 InspectorPanel 调整

现有 `InspectorPanel` 继续作为开发者文件 Inspector，但不能作为写作任务主工作区。

写作图任务必须使用产品化 Workspace；Inspector 只能作为“高级/调试”入口。

## 8. 实施顺序

### Phase 1：修正运行控制语义

1. 增加 human gate decision API。
2. Store 增加 `taskGraphRunControlMode: manual_review | auto_advance`。
3. 手动模式禁用直接 run-until-idle，改为等待门控裁决。
4. 自动模式只做断链续跑和 ready 延时续跑，遇到 human gate 自动停下。
5. 增加 store 回归测试：自动模式不越过 waiting_human_gate。

### Phase 2：创作工作区 manifest

1. 后端新增 writing workspace manifest builder。
2. 从 GraphRuntimeEnvelope、GraphLoopState、artifact refs、memory namespace 中生成产品对象。
3. 前端新增 `TaskGraphWritingWorkspace`。
4. 监控台加入“打开创作工作区”和当前审核对象入口。

### Phase 3：人工审核与修订

1. 前端审核台展示当前门控节点产物。
2. 支持用户写审核说明。
3. 支持 approve / request_revision / stop_and_checkpoint。
4. 人工修改通过 patch/ref 进入裁决，不直接污染已提交资料。

### Phase 4：写作任务实测

1. 启动写作图任务。
2. 自动模式跑到第一个人工门，确认不会越权继续。
3. 手动审核通过后继续。
4. 打开创作工作区，确认用户看到的是作品对象，不是 memory 文件。
5. 断链后自动模式能恢复到下一个 idle/gate。

## 9. 验证标准

- 手动模式下，未提交人工裁决时不会调用 `run_until_idle` 越过门控。
- 自动模式下，`waiting_human_gate` 会暂停自动推进。
- 自动模式下，ready/断链状态会延时续跑。
- 用户能从写作任务进入作品工作区。
- 工作区默认不出现 `durable_memory`、`memory_scope`、`checkpoint` 等开发词。
- 用户编辑内容会形成可追踪 revision/patch，不直接污染已提交基准资料。

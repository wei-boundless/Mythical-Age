# Harness 旧链条清理与权限系统重构计划

日期：2026-06-04

状态：已实施阶段 1、3、4 的主链清理与修复；阶段 2 的 RuntimeAuthorizationPlan / ToolExecutionPermit 独立抽象仍作为后续架构升级项。

实施记录：

- 已补齐 TaskRun 工具审批闭环：pending approval -> approval grant -> ApprovalState -> resume -> control plane gate -> 一次性消费。
- 已删除旧 `permissions.tool_admission.build_tool_request_runtime_admission()` 导出和文件，测试不再保护旧工具请求 admission 链。
- 已将 TurnRun 的无恢复 `approval_waiting` 闭合为模型可见 denial：单轮 side-effect 若需要人工审批，必须走可恢复任务审批流，不再持久化伪等待态。
- 已把 MCP tool pool 降级为 catalog preview，不再用 `PermissionService.allowed_tool_names()` 决定 harness 的 model-visible 工具。
- 已收敛 graph node contract 生成权威到 `harness.graph.work_order_contract`，并修复 graph node completion verifier 默认误拦截。
- 已删除 task executor admission 旧 observation matcher，重复拒绝只按新 `admission_denial_fingerprint` 统计。

## 1. 结论

当前 harness 主链路已经具备成熟 agent runtime 的骨架：RuntimeAssembly、RuntimeToolPlan、AdmissionDecision、ActionPermit、RuntimeToolControlPlane、ToolSupervisor、OperationGate、TaskRun/TurnRun 状态都已经存在。但权限链还没有完全收敛成一条不可冲突的授权协议。

主要问题不是“权限太严格”或“权限太宽松”之一，而是同一件事被多层重复解释：

```text
RuntimeAssembly 过滤工具
-> RuntimeToolPlan 生成可见/可调度能力
-> AdmissionDecision 判断模型动作
-> ActionPermit 绑定本次动作
-> RuntimeToolControlPlane 再做 membership / ResourcePolicy
-> ToolSupervisor 再做 membership / preflight
-> OperationGate 最终裁决
-> ToolRuntimeExecutor 再做 dispatch guard / sandbox guard / tool validation
```

这个结构可以成立，但必须明确每一层的唯一职责：

- `RuntimeAssembly/RuntimeToolPlan` 只负责把本次运行的授权能力编译成模型可见和执行可见的能力表。
- `AdmissionDecision/ActionPermit` 只负责把模型输出绑定到本次 packet / tool plan，防止串改、越权调用或格式错误。
- `OperationGate` 只负责执行前的最终 gate：adopted policy、approval grant、sandbox/path/safety validator、destructive deny policy。
- `ToolRuntimeExecutor` 只负责工具合同、参数、执行器一致性和真实执行。

授权后的 agent 不应该被旧的全局 permission mode、旧 tool scope 或重复 ResourcePolicy 生成逻辑再次否定。真正可以拦截已授权 agent 的，只能是新的边界事实：用户撤销、审批未授予、路径越界、安全校验失败、执行器缺失或工具合同不满足。

## 3. 发现的问题

### P0：审批等待任务无法可靠恢复

`OperationGate` 和 `ToolSupervisor` 已经支持 `ApprovalToken/ApprovalState`：

- `backend/permissions/operation_gate.py:67`
- `backend/permissions/operation_gate.py:77`
- `backend/runtime/tooling/supervisor.py:60`

但 `RuntimeToolControlPlane` 调用 supervisor 时没有传入 approval token/state：

- `backend/runtime/tool_runtime/tool_control_plane.py:193`
- `backend/runtime/tool_runtime/tool_control_plane.py:324`

TaskRun 遇到 `needs_approval` 后会写入 `waiting_approval`：

- `backend/harness/loop/task_executor.py:3067`

但是恢复状态机没有把 `waiting_approval` 视为可恢复：

- `backend/harness/loop/task_run_recovery_state.py:59`
- `backend/harness/loop/task_run_recovery_state.py:62`

`resume_paused_task_run()` 因此会拒绝 `waiting_approval`，返回 `task_run_not_resumable:waiting_approval`：

- `backend/harness/loop/task_executor.py:253`

这会导致一个经过授权但需要平台确认的 agent 被卡死：它既拿不到审批 token，也不能通过 resume 继续。这是当前最符合用户担忧的权限阻碍问题。

### P0：TurnRun 审批等待没有闭环

单轮工具调用遇到 `needs_approval` 会把 `TurnRun` 标成 blocked/waiting_approval：

- `backend/harness/loop/single_agent_turn.py:508`
- `backend/harness/loop/single_agent_turn.py:2499`

但当前没有对应的 TurnRun approval grant / resume 入口。成熟设计中，单轮 side-effect 审批要么升级为 TaskRun，要么持久化成可恢复的 TurnApproval。现在它只是一个阻塞事件。

### P1：graph node contract 有两套权威实现

`runtime_facade.py` 内联了一套 graph node TaskRun contract / runtime selection / task_run_id 生成逻辑：

- `backend/harness/entrypoint/runtime_facade.py:1302`
- `backend/harness/entrypoint/runtime_facade.py:1733`
- `backend/harness/entrypoint/runtime_facade.py:1815`
- `backend/harness/entrypoint/runtime_facade.py:1851`
- `backend/harness/entrypoint/runtime_facade.py:1927`

同时 `backend/harness/graph/work_order_contract.py` 也有同语义实现，并且测试和诊断脚本直接引用 graph 模块：

- `backend/harness/graph/work_order_contract.py:9`
- `backend/harness/graph/work_order_contract.py:91`
- `backend/harness/graph/work_order_contract.py:127`
- `backend/harness/graph/work_order_contract.py:203`

实际创建 graph node TaskRun 走的是 `runtime_facade.py` 内联函数，而不是 graph 包的专职模块。这是旧链条残留，会造成 graph contract、allowed operations、origin/runtime_scope authority 分叉。

### P1：权限裁决的 ResourcePolicy 生成分叉

TaskRun 工具执行中，control plane 直接为当前 operation 生成 `allowed_operations=(operation_id,)`：

- `backend/runtime/tool_runtime/tool_control_plane.py:523`
- `backend/runtime/tool_runtime/tool_control_plane.py:526`

这能保证已经进入 RuntimeToolPlan 的工具不会被旧 permission mode 二次否定，但也绕开了 `requires_approval_by_default`、显式 approval policy 和旧 `build_tool_request_runtime_admission()` 语义。

TurnRun 另一边则在 control plane 内重新决策：

- `backend/runtime/tool_runtime/tool_control_plane.py:547`
- `backend/runtime/tool_runtime/tool_control_plane.py:613`

也就是说 TaskRun 和 TurnRun 对同一 operation 的 ResourcePolicy 语义不同。这个差异必须显式化：如果 TaskRun 已有合同授权，则 control plane 不应重判；如果 TurnRun 没有 TaskRun 合同，则 side-effect 应该走审批或升级 TaskRun，而不是隐式 allow。

### P1：RuntimeToolPlan 丢失审批语义

`build_runtime_tool_plan()` 给所有能力写死 `requires_approval=False`：

- `backend/harness/runtime/tool_plan.py:111`
- `backend/harness/runtime/tool_plan.py:116`

这意味着模型可见工具表不会告诉 agent 哪些工具会触发审批等待。实际审批只在 control plane / OperationGate 后发生，agent 无法提前规划，容易反复调用同一需要审批的工具并卡住。

### P1：旧 PermissionService 仍可影响工具池可见性

`PermissionService` 仍按全局 `permission_mode` 和旧 `mode_allows_tool()` 过滤工具：

- `backend/permissions/service.py:24`
- `backend/permissions/decision_pipeline.py:12`
- `backend/permissions/policy.py:15`

harness 主执行没有直接调用它，但 MCP tool pool 仍通过它决定 builtin tool 是否 `available_to_model`：

- `backend/capability_system/mcp/server/tool_pool.py:52`
- `backend/capability_system/mcp/server/tool_pool.py:56`

如果 MCP/tool pool 的结果进入 agent 可见能力，旧全局 permission mode 就可能覆盖 RuntimeAssembly/RuntimeToolPlan 的授权结果。这类系统级权限应该服务 runtime tool plan，而不是成为旁路阻断器。

### P2：tool admission / runtime policy builder 已经偏离主链

`permissions.tool_admission.build_tool_request_runtime_admission()` 仍被 `runtime.__init__` 和 `orchestration.__init__` 导出，但 harness control plane 已经自己生成 RuntimeDirective/ResourcePolicy：

- `backend/permissions/tool_admission.py:13`
- `backend/runtime/tool_runtime/tool_control_plane.py:484`
- `backend/runtime/__init__.py:60`
- `backend/orchestration/__init__.py:70`

这不是直接运行 bug，但它制造了第二套“工具请求 admission”语义，测试也仍覆盖旧函数。后续应删除、迁移为 preview-only，或改造成新的唯一 `RuntimeAuthorizationCompiler`。

### P2：admission 旧 observation 匹配仍在主执行器里

`task_executor` 为兼容旧 observation 没有 fingerprint 的情况，保留了 legacy matcher：

- `backend/harness/loop/task_executor.py:4454`
- `backend/harness/loop/task_executor.py:4490`

这不直接授权或拒绝执行，只影响重复拒绝识别。若没有明确历史数据兼容窗口，应删除；若需要短期保留，必须写清删除条件。

## 4. 目标架构

目标不是复制 Codex 或 Claude Code，而是按本项目现状收敛为一条清晰链路：

```text
RuntimeAssembly
-> RuntimeAuthorizationPlan
-> RuntimeToolPlan
-> AdmissionDecision
-> ActionPermit
-> ToolExecutionPermit
-> OperationGate
-> ToolRuntimeExecutor
```

### 4.1 新增 RuntimeAuthorizationPlan

建议新增 `backend/harness/runtime/authorization.py`，集中产出：

- `allowed_operations`
- `denied_operations`
- `requires_approval_operations`
- `not_executable_operations`
- `operation_decisions`
- `approval_policy`
- `permission_mode`
- `task_environment_boundary`
- `source_authority`

`assemble_runtime()` 只调用这个 compiler，不再散落读取 `allowed_operations`、`blocked_operations`、`execution_permit`。

### 4.2 RuntimeToolPlan 携带执行状态

每个 `ToolCapability` 至少携带：

- `dispatchable`
- `read_only`
- `destructive`
- `requires_approval`
- `approval_channel`
- `authorization_reason`
- `safety_boundary`
- `model_guidance`

模型看到的工具应能知道：可直接执行、会等待审批、只读、sandboxed side-effect、或需要 TaskRun。

### 4.3 ToolExecutionPermit 成为 control plane 的输入

`ActionPermit` 只证明“模型动作是本轮 packet 允许的”。  
`ToolExecutionPermit` 证明“该工具调用已经由 runtime 授权，可进入 OperationGate”。

`RuntimeToolControlPlane` 不再临时重建两套 ResourcePolicy，而是消费 `ToolExecutionPermit` 中的 adopted ResourcePolicy。TaskRun 和 TurnRun 的差异由 permit compiler 显式决定。

### 4.4 OperationGate 只保留最终执行 gate

OperationGate 保留：

- operation descriptor 存在性检查。
- directive/resource_policy adopted/executable 检查。
- deny rule。
- approval token/state 检查。
- sandbox/path/safety validator。
- destructive explicit deny。

OperationGate 不再承担 agent profile、tool visibility、旧 permission mode 的二次解释。

### 4.5 ApprovalGrant 闭环

新增可持久化 approval grant：

```text
pending_approval
-> user approval decision
-> ApprovalToken/ApprovalState persisted
-> resume TaskRun/TurnRun
-> OperationGate consumes token
-> approved tool executes once or within bounded scope
```

grant 必须绑定：

- `operation_id`
- `directive_ref`
- `approval_risk_fingerprint`
- `tool_name`
- `tool_args_hash`
- `task_run_id` 或 `turn_run_id`
- `expires_at` 或一次性消费策略

## 5. 实施计划

### 阶段 1：修复审批恢复闭环

1. 扩展 `pending_approval`，写入 `approval_fingerprint`、`directive_ref`、`tool_call_id`、`risk_payload_hash`。
2. 新增 approval grant 存储，建议先放在 `runtime_objects`，再投影到 TaskRun/TurnRun diagnostics。
3. 新增 `approve_task_run_tool_call()` 或扩展 resume API，使用户确认后生成 `ApprovalToken`。
4. `resume_paused_task_run()` 支持 `waiting_approval`，但只有匹配 approval grant 时才进入 `waiting_executor`。
5. `RuntimeToolControlPlane` 从 request/runtime state 读取 `ApprovalState` 并传给 `ToolSupervisor`。
6. TurnRun 审批统一决策：需要 side-effect approval 的单轮工具，要么升级为 TaskRun，要么新增 TurnApproval resume 入口；禁止只记录 blocked。

### 阶段 2：收敛权限编译权威

1. 新建 `RuntimeAuthorizationPlan`。
2. 把 `operation_projection.py`、`tool_scheduling.py`、`RuntimeAssemblyProfile.allowed_operations` 的合并逻辑迁入唯一 compiler。
3. `assemble_runtime()` 只消费 compiler 结果。
4. `build_runtime_tool_plan()` 使用 compiler 结果填充 `requires_approval`、`authorization_reason` 和 `safety_boundary`。
5. `RuntimeToolControlPlane._execution_contracts()` 和 `_agent_turn_execution_contracts()` 改为消费 `ToolExecutionPermit`，不再现场决定 profile/approval policy。

### 阶段 3：清理 graph node 权威分叉

1. `runtime_facade.py` 改为从 `harness.graph.work_order_contract` 导入 graph node contract 函数。
2. 删除 `runtime_facade.py` 内联的 `_graph_node_contract_from_work_order()`、`_graph_node_task_selection()`、`_graph_node_allowed_operations()`、`_graph_node_task_run_id()` 等重复函数。
3. 修正 `harness.graph.work_order_contract` 内 authority 命名，统一为 `harness.graph.work_order_contract` 或 `harness.graph_node_runtime_scope`。
4. 更新 graph runtime tests，确保 runtime facade 和 graph module 使用同一 contract payload。

### 阶段 4：清理旧 permission/service 旁路

1. `PermissionService` 降级为配置读写/API 展示服务，不再决定 model-visible tool availability。
2. `capability_system.mcp.server.tool_pool` 改为接收 `RuntimeAuthorizationPlan` 或明确标注为 catalog preview，不参与 harness 执行授权。
3. 删除或迁移 `permissions.tool_admission.build_tool_request_runtime_admission()`。
4. 清理 `runtime.__init__`、`orchestration.__init__` 对旧 tool admission 的导出。
5. 若没有真实历史数据兼容要求，删除 `legacy_model_action_admission_observation_matches()`；若保留，写明删除版本和数据窗口。

### 阶段 5：测试补强

必须新增或更新以下测试：

- `waiting_approval` TaskRun 不能被普通 resume 绕过，但 approval grant 匹配后可以恢复并执行一次。
- approval token 的 operation/directive/fingerprint 任一不匹配时，OperationGate 必须拒绝。
- 已经在 RuntimeToolPlan 中 dispatchable 且 ActionPermit 匹配的工具，不会被旧 PermissionService 或旧 tool_scope 再次否定。
- `permission_mode=plan` 仍阻止 side-effect 工具，但允许只读工具。
- graph node TaskRun contract 只由 `harness.graph.work_order_contract` 生成。
- `build_runtime_tool_plan()` 能投影 `requires_approval=True`。
- TaskRun 和 TurnRun 对同一 operation 的授权差异必须由 `ToolExecutionPermit` 明确表达。

## 6. 不做的事

- 不移除 sandbox/path/safety validator。它们不是旧权限阻碍，而是执行安全边界。
- 不把 `full_access` 解释成无视合同、无视路径越界、无视工具参数校验。
- 不保留无边界兼容链路。确需短期兼容的历史数据路径，必须有删除条件。
- 不把本项目改造成 Codex 或 Claude Code 的目录结构；只吸收成熟 agent 的权威链、审批闭环、工具边界和恢复语义。

## 7. 验收标准

完成后应满足：

1. 用户授权后的 agent 不会被旧 permission service、旧 tool admission 或重复 ResourcePolicy 生成逻辑阻断。
2. 真正需要审批的工具能生成 pending approval，用户批准后能恢复执行，且 token 与风险指纹绑定。
3. TaskRun / TurnRun 权限语义可解释：谁授权、谁审批、谁执行、谁记录。
4. graph node contract 只有一个生成权威。
5. 所有旧链条删除或降级后，测试不靠跳过、硬编码或弱断言通过。

# ReadFile 文件读取权威链统一修复方案

日期：2026-06-05

状态：已按本方案实施第一轮主链切换，验证见“11. 实施记录”。

该修复涉及 runtime、tool calling、file state、dynamic context 和旧工具链清理。本文保留原方案和实施后的验收记录，作为后续检查 read_file 权威链的基准。

## 0. 结论

当前项目已经具备成熟 agent 文件读取链路的主要零件，但还没有形成唯一权威链。

正确方向已经存在：

- `backend/runtime/tool_runtime/native_tools.py` 的 `NativeReadFileTool` 会按行窗口读取文件，并生成 `total_lines`、`start_line`、`end_line`、`next_start_line`、`has_more`、`content_sha256` 等结构化信息。
- `backend/runtime/tool_runtime/tool_result_envelope.py` 能把读取结果包装成 `ToolResultEnvelope`，并推导 `file_state_events`。
- `backend/harness/runtime/dynamic_context/tool_result_projector.py` 能把工具结果投影成 preview、`content_range`、`rehydration_plan` 和 evidence policy。
- `backend/runtime/memory/file_state_authority.py` 能从 observation 重建文件读取范围、搜索命中、写入事件和 stale 状态。

主要问题是：

1. `read_file` 有新旧两条实现。`backend/capability_system/tools/tool_units/read_file_tool.py` 仍然是纯文本窗口返回，没有结构化读取元数据。
2. `FileStateAuthority` 目前主要从 observations 重建，不是 task-local 持久权威。
3. `ReplacementStore` 和 `rehydration_plan` 已存在，但还没有成为所有 resume / compact 路径必须遵守的硬契约。
4. 旧纯文本读取结果仍可能通过部分入口进入上下文，导致 direct path 与 task runtime path 的证据质量不一致。
5. `backend/capability_system/tools/native_tool_catalog.py` 当前硬 import `ReadFileTool` 并把它作为 `ToolDefinition.factory`，因此不能在未解除 catalog 依赖前直接删除旧文件。

目标不是给旧路径补字段，而是统一为一条主链：

```text
Model tool call
-> ToolControlPlane / ActionPermit
-> ToolRuntimeExecutor
-> NativeReadFileTool
-> ReadFileWindowResult
-> ToolResultEnvelope
-> ToolObservationRecord
-> Durable FileStateAuthority
-> DynamicContextProjection
-> model-visible preview / refs / rehydration_plan
```

## 1. 当前失败模式

### 1.1 读取结果权威不唯一

`NativeReadFileTool` 是成熟路径，但 `capability_system` 里的 `ReadFileTool` 仍然自己解析路径、读取文件、格式化行号并返回字符串。

这会带来两个问题：

- 同一个 `read_file` 工具名，在不同入口下可能返回不同结构。
- 只有 native runtime 路径能稳定产生 `content_range`、`content_sha256` 和 `file_state_events`。

### 1.2 上下文投影依赖结构化结果

`ToolResultProjector` 能识别 `structured_payload.tool_result` 里的 read_file 元数据。如果读取结果只是纯文本，它只能当作 legacy text 处理，无法稳定生成：

- `content_range`
- 下一段读取建议
- 当前文件 hash
- read window evidence policy
- 精确 rehydration plan

### 1.3 文件状态不是持久权威

`DynamicContextManager` 当前会在 execution projection 没有 `file_state` 时，从 observations 重建 file state。这可以作为过渡投影，但不适合作为成熟 coding agent 的权威状态。

成熟状态应该是 task-local 持久对象。每次工具执行完成后，由 `ToolResultEnvelope.file_state_events` 增量更新，而不是每次从日志里重新推导。

### 1.4 旧结果可能被当作硬证据

纯文本工具输出只适合给模型预览，不应该被当作“已读完整文件”或“可直接编辑依据”。代码编辑前必须重新读取目标行所在的当前窗口。

## 2. 目标权威划分

| 层 | 当前问题 | 目标权威 |
| --- | --- | --- |
| 路径解析 | `WorkspaceFileService` 和 FileGateway 路径都存在，入口不完全统一 | `NativeReadFileTool` 统一选择 FileGateway 或 workspace 文件边界 |
| 读取执行 | native 与 capability tool 双实现 | 只有 native runtime tool 允许执行读取 |
| 结果规范化 | 纯文本窗口和 structured result 并存 | `ReadFileWindowResult` 是唯一结构化读取结果 |
| 工具结果 | 部分路径只能 legacy text | 所有 read_file 结果必须进入 `ToolResultEnvelope` |
| 文件状态 | 从 observations 重建 | `FileStateAuthorityStore` task-local 持久增量更新 |
| 上下文注入 | preview、raw result、legacy text 混杂 | `DynamicContextProjection` 只注入 preview、range、refs、rehydration plan |
| 恢复读取 | replacement 已有但不是硬契约 | 长输出必须可通过 `read_persisted_tool_result` 恢复 |

## 3. 目标数据契约

新增或抽取统一结构：`ReadFileWindowResult`。

建议文件：

```text
backend/runtime/tool_runtime/read_file_window.py
```

结构：

```json
{
  "authority": "runtime.tool_result.read_file_window.v1",
  "kind": "text_file",
  "status": "ok",
  "path": "backend/app.py",
  "repository_id": "",
  "managed_file_ref": {},
  "text": "  1 | ...",
  "total_lines": 320,
  "start_line": 1,
  "end_line": 240,
  "line_count": 240,
  "returned_lines": 240,
  "next_start_line": 241,
  "has_more": true,
  "truncated": true,
  "content_sha256": "..."
}
```

约束：

- `text` 是模型可见的当前窗口，不代表全文件。
- `content_sha256` 计算对象必须是完整文件内容，不是窗口内容。
- `has_more=true` 时必须提供 `next_start_line`。
- `status=error` 时也必须保留 path 和结构化错误，不能只返回字符串。
- 所有 downstream 模块只消费这个结构，不再从普通文本猜行号和路径。

## 4. 固定执行流

### 4.1 读取执行

```text
ToolRuntimeExecutor
-> build_native_runtime_tool(read_file)
-> NativeReadFileTool.validate_input()
-> NativeReadFileTool.call()
-> build_read_file_window_result()
-> ToolResultEnvelope
```

`capability_system.tools.tool_units.read_file_tool.ReadFileTool` 不再允许保留独立读取逻辑。实施顺序必须先解除 catalog 对旧类的硬依赖，再删除旧执行类。

具体处理：

1. 先把 `ToolDefinition.factory` 从必填 `BaseTool` 工厂改成可空，或为 native-only 工具增加显式 `native_runtime_only=True` 标记。
2. `read_file` 的 catalog 定义不再 import `ReadFileTool`，而是只保留 schema、contract、operation_id、prompt exposure 和 resource exposure。
3. 所有 runtime 执行入口只通过 `build_native_runtime_tool(read_file)` 获取执行器。
4. 更新直接 import `ReadFileTool` 的测试和旧调用点。
5. 最后删除 `backend/capability_system/tools/tool_units/read_file_tool.py`，或只保留无执行能力的 schema 壳。推荐最终删除，不保留可执行旧工具。

如果实施中发现某个非 runtime 系统仍必须拿到 LangChain `BaseTool` 实例，则该系统应该改为调用 runtime executor 或读取 tool schema，而不是让 `ReadFileTool` 继续拥有文件读取执行权。

### 4.2 结果记录

所有成功读取必须生成：

- `ToolResultEnvelope.text`
- `ToolResultEnvelope.structured_payload.tool_result`
- `ToolResultEnvelope.observed_paths`
- `ToolResultEnvelope.file_state_events`
- `ToolResultEnvelope.result_ref`

`infer_file_state_events()` 只接收结构化 `tool_result`。没有结构化 result 的 read_file 不允许被标记为 `read_material`。

### 4.3 文件状态更新

新增持久 store：

```text
backend/runtime/memory/file_state_store.py
```

职责：

- 按 `task_run_id` 保存 `FileStateAuthority`。
- 每个 tool observation 完成后应用 `file_state_events`。
- 写入、编辑后把对应文件旧 read ranges 标记为 stale。
- resume 时直接加载 task-local file state。

更新边界：

- `NativeReadFileTool` 只负责执行读取和生成 envelope，不直接写持久 file state。
- `ToolRuntimeExecutor` 只负责执行工具、完成 envelope、返回 observation，不成为状态权威。
- file state store 的写入应发生在 observation commit 边界，也就是 task-run / agent-turn 将工具 observation 持久记录成功之后。
- `DynamicContextManager` 只消费 file state snapshot，不负责从工具结果里二次决定文件状态。

`FileStateAuthority.from_observations()` 只能作为迁移期诊断工具，不再作为主链默认来源。

### 4.4 上下文投影

`DynamicContextManager` 目标行为：

1. 优先读取 task-local `FileStateAuthorityStore`。
2. `ToolResultProjector` 只向模型暴露受预算控制的 preview。
3. 长输出通过 `ReplacementStore` 持久化，模型只看到 `replacement_ref` 和 `rehydration_plan`。
4. 对代码编辑，投影必须继续提示：编辑前读取目标行的当前窗口，不要从 search preview、summary 或旧 persisted output 直接编辑。

## 5. 实施阶段

### 阶段 1：统一 read_file 结果契约

目标：消灭重复窗口格式化逻辑。

改动：

- 新增 `backend/runtime/tool_runtime/read_file_window.py`。
- 将 `native_tools.py` 中 `_read_line_window()` 迁移为统一函数。
- `NativeReadFileTool` 使用统一函数构造 `ReadFileWindowResult`。
- 解除 `native_tool_catalog.py` 对 `ReadFileTool` 的硬 import 和 factory 依赖。
- 删除 `capability_system/tools/tool_units/read_file_tool.py` 中的 `_line_window_text()` 独立逻辑；最终删除旧工具类，或只保留无执行能力的 schema 壳。

完成标准：

- `rg "_line_window_text|def _read_line_window"` 只剩一个权威实现。
- `rg "from capability_system.tools.tool_units.read_file_tool import ReadFileTool"` 没有 runtime 或 catalog 命中。
- native 和非 task-run 入口对同一文件同一窗口生成相同结构化字段。

### 阶段 2：统一工具执行路径

目标：所有 `read_file` 调用都经过 `NativeReadFileTool`。

改动：

- 审查 `ToolRuntimeExecutor.run()` 与 `_run_core()`，确认 task-run 和 agent-turn 都优先 `build_native_runtime_tool(read_file)`。
- 审查 `single_agent_turn` 的 native tool exposure，避免模型看到旧 LangChain read_file schema 后走旧执行路径。
- 如果仍存在直接调用 `ReadFileTool._run()` 的入口，改为调用 runtime executor 或删除入口。
- 已知直接依赖旧类的测试包括 `backend/tests/workspace_file_tools_regression.py` 和 `backend/tests/rag_external_layout_regression.py`，需要改成 native runtime tool 或 runtime executor 行为测试。

完成标准：

- task-run path 和 agent-turn path 的 read_file observation 都包含 `result_envelope.file_state_events`。
- legacy read_file 纯文本结果不再能满足 `read_material`。

### 阶段 3：FileStateAuthority 持久化

目标：文件状态由 task-local store 持久维护，而不是从 observations 临时重建。

改动：

- 新增 `FileStateAuthorityStore`，支持 `load(task_run_id)`、`apply_events(task_run_id, events)`、`snapshot(task_run_id)`。
- 工具执行完成后，从 `ToolResultEnvelope.file_state_events` 增量更新 store。
- `DynamicContextManager._with_file_state_authority_projection()` 改为优先消费 store。
- `from_observations()` 仅保留为迁移诊断或测试辅助，不再是默认运行路径。

完成标准：

- read -> edit/write 后，store 中对应 read range 变为 stale。
- resume 后不需要重扫 observations 也能恢复 file state。

### 阶段 4：上下文重放与长输出恢复硬契约

目标：长工具输出不污染上下文，但可以按需恢复。

改动：

- `ToolResultProjector` 对超过预算的 read_file 输出必须写入 `ToolResultStore`。
- `ReplacementStore` 记录 `replacement_key`、content hash、projection policy hash、rehydration plan。
- `read_persisted_tool_result` 的权限和 runtime profile 保持只读，但只用于恢复被省略的旧工具输出。
- 对代码编辑，policy 继续要求 fresh `read_file` 当前目标行窗口。

完成标准：

- 超长 read_file 输出在 prompt 中只出现 preview 和 rehydration plan。
- 可通过 `read_persisted_tool_result` 恢复 omitted bytes。
- 编辑工具不能把 search preview 或旧 persisted output 当作当前源码依据。

### 阶段 5：清理旧逻辑和旧测试

目标：不保留旧链路作为兜底。

删除或改写：

- 旧 `ReadFileTool` 的独立读取、独立窗口格式化、独立错误字符串。
- 依赖纯文本 read_file 结果作为硬证据的测试。
- 从 legacy text 猜测 read_file 读取范围的逻辑。

保留：

- `legacy_text` debug hints 可以短期用于诊断，但不能授予 `read_material`。
- `from_observations()` 可以保留为测试构造器或迁移检查，不作为 runtime 默认来源。

完成标准：

- 没有旧 read_file 执行路径。
- 没有测试要求纯文本路径继续可用。
- 所有 read_file 行为测试都围绕 structured envelope、file_state_events 和 dynamic context projection。

## 8. 切换规则

不建议用长期 feature flag 保留两条 read_file 主链。

允许的过渡方式：

1. 阶段 1 到阶段 3 可以在测试里做 shadow comparison：同一 observation 重建结果与持久 store 结果必须一致。
2. 一旦 store 验证通过，runtime 默认只读 store。
3. 旧 observation rebuild 不再作为运行兜底。
4. 如需回滚，使用代码回滚，不在代码里保留旧主链开关。

## 9. 禁止事项

- 禁止通过 prompt 提醒替代结构化读取契约。
- 禁止保留两个 read_file 执行权威。
- 禁止把 search preview、summary、旧 persisted output 当作当前源码编辑依据。
- 禁止让 `FileStateAuthority` 长期依赖 observations 重建。
- 禁止测试只断言字符串内容，而不验证 envelope、file state event、content range 和 rehydration plan。
- 禁止为了兼容旧测试保留纯文本工具路径。

## 10. 最终验收标准

完成后，`read_file` 应满足：

```text
同一工具名只有一个执行权威。
同一读取结果只有一个结构化契约。
同一 task_run 的文件状态只有一个持久权威。
上下文里只有预算内 preview、refs 和恢复计划。
代码编辑前必须读取当前目标行窗口。
旧纯文本 read_file 不再是运行主链的一部分。
```

这才是成熟 coding agent 应有的文件读取能力：不是把文件文字塞进上下文，而是把文件事实、读取范围、证据有效性和恢复路径纳入 runtime 权威链。

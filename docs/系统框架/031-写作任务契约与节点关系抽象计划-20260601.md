# 写作任务契约与节点关系抽象计划

日期：2026-06-01

## 现状判断

写作任务里的多数契约不是用户应该逐个维护的业务对象。现有契约库中，`contract.writing.modular_novel.*` 大量节点契约都共享同一类结构：

- 输入通常是上游交接包。
- 输出通常是 `artifact_refs` 加一个具体产物引用。
- 验收通常是“必须产出可追踪产物引用”。
- 审核节点的差异主要不在契约结构，而在审核 Prompt、裁决标准和返修边关系。

因此，写作场景的前端管理对象应从“逐个编辑契约 ID 和 JSON”调整为：

1. 写作节点：写手、审核员、返修者、记忆管理员等角色。
2. 节点 Prompt：身份、职责、禁止事项、完成标准、质量要求。
3. 关系边：写手交给审核、审核通过进入提交、审核不通过返回返修、返修后再审。
4. 记忆边：读取已提交资料、写入候选记忆、审核通过后提交正式记忆。
5. 少量契约覆盖项：产物类型、审核标准、最大返修次数、是否需要人工门、记忆集合与可见性。

契约库仍存在，但变成底层可复用模板和高级能力，不应成为写作用户的主要操作入口。

## 目标模型

### 1. 写作契约族

新增一层 `writing_contract_family` 概念，用少量模板表达稳定协议：

- `writing.draft_artifact`：写手产出草稿、设定、章节、提案等产物。
- `writing.review_verdict`：审核员输出通过、返修、驳回、问题清单和返修要求。
- `writing.revision_request`：审核失败时传回写手或返修者的返修包。
- `writing.commit_receipt`：审核通过后写入正式产物或记忆库的提交回执。
- `writing.memory_update`：写作记忆、设定、人物、章节摘要等可追踪更新。

这些模板负责稳定字段、验收规则和运行可追踪性。具体世界观、人设、章节正文、细纲等只作为 `artifact_type` 或 `writing_stage` 覆盖项，不再复制一份完整契约。

### 2. 写作节点角色

用户主要维护节点角色，而不是运行内部字段：

- 写手节点：关注创作 Prompt、输入资料、产出类型。
- 审核节点：关注审核标准、裁决格式、是否允许返修。
- 返修节点：关注如何使用审核意见改稿。
- 记忆/提交节点：关注哪些结果可以进入正式资料或记忆库。

节点配置页的第一屏只展示 Prompt 和必要设置：

- 角色名称
- 角色 Prompt
- 产物类型
- 输入来源
- 输出去向
- 绑定的复用契约族

Agent、runtime profile、JSON 策略、原始契约 ID 放入高级区。

### 3. 写作关系边

将边抽象成用户能理解的关系类型：

- `draft_to_review`：写手产物进入审核。
- `review_pass_to_commit`：审核通过后进入提交或下一阶段。
- `review_revise_to_writer`：审核未通过返回写手返修。
- `revision_to_review`：返修产物回到审核。
- `review_reject_to_human`：严重失败进入人工确认。
- `approved_memory_commit`：审核通过内容写入记忆或资料库。
- `memory_read_required`：节点运行前读取已提交记忆，缺失时阻断或提示人工补齐。
- `memory_write_candidate`：节点产出候选记忆，不直接进入正式资料。
- `memory_commit_after_review`：审核通过后把候选记忆提交为正式可读资料。

边关系自动绑定对应的复用契约族。用户只需要配置：

- 目标审核节点
- 通过后去哪里
- 返修回哪里
- 最大返修次数
- 审核裁决字段
- 需要携带哪些记忆或产物引用
- 读取哪个记忆库/集合
- 哪个节点产生候选记忆
- 哪个审核节点拥有提交裁决权

### 4. 任务图语义层

新增一层 `task_graph_semantic_relation`，把任务图里的边标准化为用户可理解、运行时可解析的语义关系：

```text
用户选择关系边
-> semantic_relation_id + 少量参数
-> 后端解析成 edge_type / contract_family / metadata / contract_bindings
-> layered normalizer 编译出 memory_protocol、revision_edges、artifact_context_edges
-> runtime 按明确协议执行
```

这层的权威在后端；前端只展示预设、收集参数和展示解析结果。不能让前端用一堆裸 `payload_contract_id`、`memory_edge_type`、JSON policy 来替代语义选择。

第一批关系族：

- `writing.draft_to_review`
- `writing.review_pass_to_commit`
- `writing.review_revise_to_writer`
- `writing.revision_to_review`
- `writing.review_reject_to_human`
- `memory.read_required`
- `memory.write_candidate`
- `memory.commit_after_review`

这些关系族必须能覆盖写作工作流中最常见的“写手 -> 审核 -> 返修 -> 通过提交 -> 记忆读取/写入”结构。

## 页面重构

### 写作任务工作台

新增写作专用视图或在任务图主页面增加写作模式，分三栏：

- 左侧：写作流程骨架，例如“章节草稿 -> 章节审核 -> 返修 -> 提交”。
- 中间：节点 Prompt 编辑，优先展示写手和审核员。
- 右侧：关系边与必要契约设置，包括通过、返修、提交、人工门。

默认不显示契约 ID、schema JSON、runtime packet、storage path。高级模式才显示原始绑定。

### 契约库

契约库改成两层：

- 复用契约族：写作草稿、审核裁决、返修请求、提交回执、记忆更新。
- 高级契约详情：字段、验收规则、可见性、失败策略。

普通用户默认只选择契约族和少量覆盖项，不直接编辑每个契约实例。

### 节点配置

节点配置页改成“节点角色配置”，默认页签：

1. 角色 Prompt
2. 输入/输出
3. 审核关系
4. 能力与权限
5. 高级装配

其中“审核关系”只在写作节点或审核节点显示。

## 后端重构

### 新增模块

- `backend/task_system/contracts/writing_contract_families.py`
  - 定义写作契约族。
  - 提供 `resolve_writing_contract(family_id, overrides)`。
  - 生成稳定 ContractSpec，不让前端拼装 schema。

- `backend/task_system/writing/relation_presets.py`
  - 定义写手、审核、返修、提交关系。
  - 输出标准边配置和契约绑定。

- `backend/task_system/graphs/semantic_relations.py`
  - 定义任务图关系族，包括写作关系和记忆关系。
  - 提供 `resolve_semantic_relation(relation_id, parameters)`。
  - 输出标准 `edge_type`、`metadata`、`contract_bindings` 与用户可读说明。
  - 作为前端关系选择器、模板生成和后端 normalizer 之间的单一语义权威。

- `backend/task_system/writing/workflow_recipes.py`
  - 定义章节草稿审核循环、设定审核循环、卷级审核循环等写作流程配方。

### 清理原则

- 不新增一堆 `contract.writing.modular_novel.xxx` 的重复契约。
- 现有重复契约迁移到契约族解析结果。
- 前端不再把契约 ID 当主要输入。
- 原始 contract_bindings 保留为高级调试和导入导出层，不作为默认交互。
- 记忆读写提交不能散落成一组裸字段；必须通过 `memory.read_required`、`memory.write_candidate`、`memory.commit_after_review` 关系族表达。

## 风险与边界

- 不能直接删除现有写作契约，必须先确认任务图和运行装配不依赖具体 ID。
- 不能把契约族做成前端硬编码，契约族解析权威应在后端。
- 前端只负责呈现和收集用户意图：角色、Prompt、关系边、少量覆盖项。
- 运行时仍必须拿到明确的 ContractSpec 和 contract_bindings，不能因为 UI 简化而削弱执行协议。

## 当前结论

写作任务的成熟结构应是：

```text
写作流程配方
-> 节点角色 Prompt
-> 写手/审核/返修/提交/记忆读写关系边
-> 契约族 + 少量覆盖项
-> 后端解析为稳定 ContractSpec 与 contract_bindings
-> runtime 执行
```

用户关注 Prompt、审核标准、关系边和必要记忆集合；系统负责契约复用、字段稳定性、产物追踪、记忆协议和运行装配。

## 执行补充：语义边权威边界

前端任务图编辑器只能保存语义意图：

```text
edge_id / source / target
+ semantic_relation_id
+ semantic_parameters
```

前端可以为了矩阵、预检和选择器临时投影出 `memory_read`、`memory_write_candidate`、`memory_commit` 等显示字段，但这些字段不是保存权威。正式的 `edge_type`、`payload_contract_id`、`contract_bindings.memory`、`commit_visibility_policy` 和 `memory_protocol` 必须由后端 `task_system.graphs.semantic_relations.resolve_semantic_relation` 与 layered normalizer 生成。

这条边界同样适用于记忆读取/写入：

- `memory.read_required` 表示节点运行前读取已提交正式记忆。
- `memory.write_candidate` 表示节点只写候选记忆，不直接污染正式资料。
- `memory.commit_after_review` 表示审核通过后提交正式可读记忆。

测试要求：前端模板和矩阵不再生成完整 memory contract bindings；后端必须能从最小语义 payload 解析出读取、候选写入和审核提交三类 `memory_protocol` 边。

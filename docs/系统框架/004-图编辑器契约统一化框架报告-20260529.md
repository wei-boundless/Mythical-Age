适用范围：

```text

```

## 0. 结论

单 agent 契约可以、也应该由任务图编译产生。

本项目现在已经具备这个方向的基础：

```text
TaskGraphDefinition 支持 graph_kind="single_agent"。
图、节点、边都已经有 contract_bindings。
图编辑器保存时会把旧字段归一进 contract_bindings。
后端已经能把 TaskGraphDefinition 编译成 GraphHarnessConfig / runtime preview。
GraphRuntime 已经能根据发布后的 GraphHarnessConfig 创建 graph run 和 task run。
```

但当前系统还没有真正统一：

```text
RegisteredEngagementPlan 仍然默认 single_agent_task_run。
EngagementDispatcher 仍然直接把 engagement contract 转成 TaskRunContract。
单 agent TaskRunContract 和 TaskGraphDefinition.contract_bindings 是两套契约权威。
图编辑器编译结果主要面向 GraphHarnessConfig 预览和发布，还没有成为特定任务启动的唯一入口。
SpecificTaskRecord / TaskAssignment / Flow / Workflow 仍然残留为节点任务资产投影，部分写作图脚本会把同一份 contract_bindings 复制到多个旧结构里。
```

目标统一结构应该是：

```text
RegisteredEngagementPlan
-> 只声明雇佣入口、环境、默认 agent/profile、启动参数约束
-> execution_strategy.kind = graph_task_run
-> execution_strategy.startup_policy.graph_id = graph.xxx

TaskGraphDefinition
-> 契约唯一权威
-> graph.contract_bindings 定义整体任务边界
-> node.contract_bindings 定义单个执行单元边界
-> edge.contract_bindings 定义交接边界

GraphCompiler
-> 将 TaskGraphDefinition 编译成 GraphRuntimePlan
-> 如果是单可执行 agent 节点，产出 SingleAgentTaskRunStartPacket
-> 如果是多节点，产出 GraphRuntimeStartPacket

Runtime
-> 只执行编译产物
-> 不重新解释用户意图
-> 不从旧 task/flow/workflow 中反向推导契约
```

这意味着：单 agent 任务不是 engagement 的特殊硬编码路径，而是单节点图的编译结果。

```text
编辑器不需要另做一个单 agent 编辑器。
它应该是“任务契约图编辑器”。
single_agent / multi_agent / coordination 只是 graph_kind 和编译结果差异，不应该对应三套契约结构。
```

```text
Engagement 不再直接生产 TaskRunContract。
```

现状：

```text

graph_harness_config_publisher.py 能从 TaskGraphDefinition 发布 GraphHarnessConfig。
GraphHarnessConfig 是发布锁定产物，带 content_hash、nodes、edges、resources、memory、artifacts、permissions、tools、agents、contracts。
GraphRuntime.start 会根据 published GraphHarnessConfig 创建 GraphRun、TaskRun、GraphRuntimeEnvelope。
```

问题：

```text
TaskGraphRuntimeSpec 和 GraphHarnessConfig 的职责边界不够清晰。
GraphHarnessConfig 当前承担“发布配置 + runtime assembly + scheduler 输入”的混合职责。
EngagementDispatcher 没有接入 graph runtime。
single_agent 图没有独立的编译出口：单节点图应该能编译成 TaskRunContract，但当前没有正式 compiler。
```

目标：

```text
保留 TaskGraphDefinition 作为契约源。
定义统一的 GraphStartCompiler。
GraphStartCompiler 输出两类 start packet：
  SingleAgentTaskRunStartPacket
  GraphRuntimeStartPacket
GraphHarnessConfig 可以作为 GraphRuntimeStartPacket 的部署锁定物，但不是契约权威。
```

现状：

```text

```

问题：

```text
同步不是统一。
复制越多，权威越不清楚。

```

```text
SpecificTaskRecord / TaskAssignment / Flow / Workflow 只能作为外部资产、库索引、展示投影或历史导入来源。
一旦任务进入 runtime，契约只能来自 TaskGraphDefinition.contract_bindings。
写作图节点的 task asset 不应再拥有独立 contract_bindings 副本；它只能记录 source_graph_id / source_node_id / contract_projection_ref。
```

## 2. 图编辑器目标框架

### 2.1 产品定位

图编辑器不是“多 agent 编排器”，而是：

```text
任务契约图工作台
```

它编辑的是可运行任务合同的结构化源，不是某一种运行模式。

支持三类图：

```text
single_agent：一个执行 agent 节点，可有资源节点、门控节点、artifact 节点。
multi_agent：多个 agent 节点，有显式 handoff / join / review。
coordination：包含图模块、批处理、循环、资源层和调度层。
```

三类图共用同一数据模型：

```text
TaskGraphDefinition
TaskGraphNodeDefinition
TaskGraphEdgeDefinition
contract_bindings
```

差异只发生在编译阶段。

### 2.2 编辑器层级

推荐工作台层级：

```text
图级
  graph identity
  task_environment_id
  graph_kind
  graph.contract_bindings
  runtime_policy / context_policy

拓扑
  execution nodes
  resource nodes
  gate nodes
  graph module nodes
  dependency edges
  resource/context edges

节点契约
  node role prompt resource
  agent/profile binding
  schema input/output
  artifact target
  memory read/write
  acceptance gate
  runtime model requirement

边契约
  payload schema
  handoff policy
  ack/failure propagation
  artifact refs
  memory handoff
  temporal semantics

编译预览
  compile validity
  executable node list
  start packet kind
  runtime packet preview
  authority map
  object trace

启动绑定
  engagement plan binding
  startup parameter schema
  approval policy
  published graph version/hash
```

注意：

```text
“启动绑定”不是图契约本身。
它只是把一个已发布图注册成可被用户、agent、workflow 调用的 engagement。
```

### 2.3 contract_bindings 规范

统一后的 section 语义：

```text
schema
  输入、输出、载荷、图合同引用。

execution
  执行端类型、节点合同、agent 选择策略、任务承接方式。

artifact
  真实产物路径、产物要求、引用策略、发布策略。

memory
  读取、写回、动态读取、边交接记忆。

handoff
  边交接、ack、结果投递、失败传播。

acceptance
  质量门、人工门、验收条件、完成证据。

runtime
  模型能力需求、长度预算、执行模式、等待策略、失败策略。

unit_batch
  批处理单元、分片、合并、批验收。

governance
  风险、账本、问题闭环、上下文边界。

temporal
  触发、可见、确认、传播时序。
```

禁止：

```text
在 contract_bindings 外另开一套任务契约字段给 runtime 消费。
在 engagement plan 中复制 node.contract_bindings。
在 flow/workflow/specific_task 中保存可覆盖图契约的合同副本。
```

### 2.4 单 agent 图的规范形态

最小单 agent 图：

```text
TaskGraphDefinition(
  graph_kind="single_agent",
  entry_node_id="main",
  output_node_id="main",
  nodes=[
    TaskGraphNodeDefinition(
      node_id="main",
      node_type="agent",
      agent_id="agent:0",
      contract_bindings={...}
    )
  ],
  edges=[]
)
```

允许存在非执行节点：

```text
artifact_repository
memory_resource
manual_gate
runtime_monitor
output
```

编译规则：

```text
如果可执行 agent 节点数量为 1，且没有需要 scheduler 推进的并行/图模块/多 agent handoff，则编译为 SingleAgentTaskRunStartPacket。
否则编译为 GraphRuntimeStartPacket。
```

这不是启发式意图识别，而是结构化图编译规则。

## 3. 统一后的启动链

### 3.1 统一入口

```text
User / Agent / Workflow
-> request_registered_engagement(plan_id, startup_parameters)
-> EngagementService.start
-> resolve plan
-> admit plan
-> issue engagement contract
-> compile graph start packet
-> dispatch start packet
```

### 3.2 EngagementPlan 目标结构

```json
{
  "plan_id": "engage.writing.modular_novel.master",
  "title": "模块化长篇小说任务",
  "task_environment_id": "env.writing",
  "assignee": {
    "kind": "agent",
    "agent_id": "agent:0",
    "agent_profile_id": "main_interactive_agent"
  },
  "runtime_profile": {
    "runtime_mode": "professional",
    "runtime_mode_policy": {}
  },
  "execution_strategy": {
    "kind": "graph_task_run",
    "startup_policy": {
      "graph_id": "graph.writing.modular_novel.master",
      "publish_policy": "require_published",
      "compile_policy": "single_agent_if_single_executable_node"
    },
    "lifecycle_policy": {
      "closeout_policy": "real_artifacts_required"
    }
  },
  "input_contract": {},
  "output_contract": {},
  "prompt_contract": {}
}
```

说明：

```text
input_contract / output_contract / prompt_contract 可以保留为 engagement 层的启动约束或展示摘要。
它们不能覆盖 graph.contract_bindings。
真正执行契约来自 graph_id 指向的 TaskGraphDefinition。
```

### 3.3 编译出口

统一 compiler 输出：

```text
GraphStartPacket
  authority = task_system.graph_start_packet
  source_graph_id
  source_graph_version/hash
  engagement_contract_ref
  task_environment_id
  startup_parameters
  start_kind
  single_agent_task_run_packet?
  graph_runtime_packet?
  compile_issues
  authority_map
```

单节点图出口：

```text
SingleAgentTaskRunStartPacket
  task_run_contract
  agent_profile_ref
  runtime_selection
  prompt_contract_projection
  artifact_requirements
  acceptance_requirements
  source_graph_ref
  source_node_ref
```

多节点图出口：

```text
GraphRuntimeStartPacket
  graph_harness_config_ref 或 graph_runtime_plan_ref
  initial_inputs
  scheduler_bootstrap
  graph_run_policy
```

## 4. 权威边界

| 层 | 可以做什么 | 不可以做什么 |
| --- | --- | --- |
| 图编辑器 | 编辑 TaskGraphDefinition 和 contract_bindings | 为 runtime 创建第二套 task contract |
| EngagementPlan | 注册一个可启动任务入口 | 复制或覆盖图节点契约 |
| EngagementService | 准入、记录、调用 compiler | 根据 plan 类型直接拼 TaskRunContract |
| GraphCompiler | 结构化编译图，决定 start packet kind | 根据用户自然语言判断任务类型 |
| Runtime | 执行 start packet、记录 observation、推进生命周期 | 反向读取 flow/workflow 来补合同 |
| SpecificTask/Flow/Workflow | 作为资产索引、展示投影、导入来源 | 成为运行时契约权威 |
| GraphHarnessConfig | 发布锁定物和部署产物 | 成为源契约或覆盖 TaskGraphDefinition |

## 5. 需要修复的结构问题

### 5.1 `single_agent_task_run` 不应作为 engagement 默认策略

当前：

```text
backend/api/task_system.py 默认 kind=single_agent_task_run。
frontend EngagementPlanLibraryPage 默认 kind=single_agent_task_run。
backend/task_system/engagement/admission.py 支持 single_agent_task_run。
backend/task_system/engagement/dispatcher.py 对 single_agent_task_run 走直接 TaskRunContract。
```

目标：

```text
默认 kind 改为 graph_task_run。
single_agent_task_run 仅在过渡期作为被 compiler 产出的 internal dispatch kind，不作为用户配置策略。
最终删除 engagement 层 single_agent_task_run。
```

### 5.2 缺少 graph_task_run strategy

需要新增：

```text
EngagementExecutionKind 增加 graph_task_run。
admission 校验 startup_policy.graph_id 存在、图存在、图已发布或允许 draft preview。
resolver 解析 graph_ref。
contract issuer 把 graph_ref 纳入 contract。
dispatcher 调用 GraphStartCompiler。
```

### 5.3 缺少单节点图到 TaskRunContract 的 compiler

需要新增：

```text
backend/task_system/compiler/graph_start_compiler.py
```

职责：

```text
读取 TaskGraphDefinition。
校验 graph_kind、contract_bindings、可执行节点。
识别单执行 agent 节点。
从 graph/node contract_bindings 编译 TaskRunContract。
产出 source_graph_ref / source_node_ref。
保留 engagement_contract_ref。
```

禁止：

```text
从 EngagementContract.prompt_contract 拼任务目标作为唯一来源。
从 SpecificTaskRecord / Flow / Workflow 推断完成条件。
```

### 5.4 图运行和单 agent 运行的 task_run 监控需要统一

当前：

```text
GraphRuntime.start 会创建 TaskRun + GraphRun。
SingleAgent TaskRun 通过 start_task_lifecycle 创建 TaskRun + AgentRun + TaskLifecycleRecord。
```

目标：

```text
两者都必须出现在同一个 task run 监控台。
等待中、运行中、失败、完成状态一致。
不运行的任务不进入运行监控。
engagement run 记录只保存启动入口和 closeout，不替代 task run 状态。
```

### 5.5 写作图任务包需要停止多处复制契约

当前：

```text
scripts/configure_writing_modular_novel_graph.py 会把 node contract_bindings 写入 workflow、flow、specific task、assignment。
```

目标：

```text
写作图节点 task asset 改成引用：
  source_graph_id
  source_node_id
  contract_projection_ref

运行时只消费 graph.nodes[*].contract_bindings。
旧资产只用于库展示和人工检索。
```

测试矩阵：

```text

2. 单节点图 TaskRun 执行真实 write_file，产出 artifact，closeout 绑定真实 artifact。
3. 创建 multi-node 图，绑定 engagement plan，启动后生成 GraphRun + TaskRun。
4. 图编译失败时，engagement run blocked，错误进入 engagement events，不进入运行监控。
5. 没有 graph_id 的 plan fail closed。
6. 没有 artifact/acceptance 的长任务合同不能被编译为 completed。
```

```text
python -m pytest backend/tests/graph_start_compiler_regression.py -q
```

实测要求：

```text

至少一个 graph_task_run 必须启动真实 GraphRun。
```

## 7. 删除和保留规则

必须删除或降级：

```text
Engagement 层直接拼 TaskRunContract 的路径。
用户可配置的 single_agent_task_run 策略。
写作任务资产中可覆盖 node.contract_bindings 的副本。
任何从旧 Flow / Workflow / SpecificTaskRecord 反向推导运行契约的逻辑。
```

可以保留：

```text
TaskGraphDefinition / contract_bindings。
GraphHarnessConfig，前提是它只作为发布锁定部署产物。
SpecificTaskRecord / Flow / Workflow，前提是它们只作为资产库、展示投影、导入来源。
GraphRuntime / GraphRun，前提是它们消费编译产物，不反向解释业务合同。
```

## 8. 风险控制

### 8.1 最大风险：把旧链路改名为 graph_task_run

禁止方案：

```text
graph_task_run -> 读取 engagement output_contract -> task_run_adapter -> start_task_lifecycle
```

正确方案：

```text
graph_task_run -> graph_id -> TaskGraphDefinition -> GraphStartCompiler -> start packet -> runtime
```

### 8.2 最大风险：GraphHarnessConfig 变成新的第二权威

GraphHarnessConfig 只能是：

```text
published graph deployment artifact
```

不能是：

```text
人工编辑的合同源
覆盖 TaskGraphDefinition 的配置源
旧 Flow/Workflow 的替代壳
```

### 8.3 最大风险：单节点图被当作启发式判断

单节点图编译为 TaskRunContract 的条件必须是结构规则：

```text
graph_kind == "single_agent"
可执行 agent 节点数量 == 1
不存在需要 scheduler 的执行边或图模块
node.contract_bindings.acceptance/artifact 能生成完成证据
```

不允许：

```text
根据标题、prompt、用户请求文本、任务名称猜测。
```

## 9. 最终目标链路

```text
图编辑器
-> 保存 TaskGraphDefinition
-> 发布图版本
-> 注册 EngagementPlan(graph_task_run, graph_id)
-> User/Agent 请求 engagement
-> EngagementService 准入
-> GraphStartCompiler 编译图
-> 单节点图：start_task_lifecycle(TaskRunContract from graph)
-> 多节点图：GraphRuntime.start(GraphRuntimeStartPacket)
-> Runtime 执行
-> Artifact / Acceptance / Closeout
-> EngagementRun 绑定真实结果
```

这条链路满足成熟 agent 架构要求：

```text
系统不识别意图。
系统只装配和执行结构化合同。
agent 通过 request_registered_engagement 调用已注册任务。
任务合同来自图，不来自分散旧结构。
单 agent 和图任务同源。
长任务完成绑定真实 artifact 和验收证据。
```

## 0. 目标结论

本次升级不是把旧图编辑器复活，也不是把写作配置脚本搬进前端。

更准确地说，本计划不应被理解为“给任务系统加一个图编辑器小机制”。图任务系统必须被做成平台的一等控制系统：

```text
图语言定义任务合同。
编译器把任务合同发布为不可变运行合同。
GraphHarness 装配运行边界。
GraphLoop 推进状态。
AgentHarness 执行 agent 节点。
文件、记忆、产物、权限、工具和监控系统提供服务，但不反向拥有图控制权。
```

当前 `009` 原计划方向正确，但颗粒度仍偏 UI 功能清单。正式实施前必须把它升级为完整系统工程蓝图：图语言内核、编译发布、运行控制、持久化恢复、资源通信、编辑器交互、版本治理和验证矩阵必须同时成立。否则编辑器看起来能画图，运行时仍会退化成旧的节点拼接和隐式 metadata 解释。

目标是：

```text
旧图编辑器的成熟工作台交互框架
+ 写作配置包暴露出的复杂任务图语义
+ 新链路 TaskGraphDefinition 的唯一权威模型
= 可维护、可发布、可运行的新图任务编辑器
```

升级后的编辑器必须满足：

```text
TaskGraphDefinition 是唯一编辑事实源。
GraphHarnessConfig 是发布后的不可变运行投影。
StandardView / ComposableView 只用于诊断和预览，不能成为第二套编辑权威。
写作图这种高密度任务图可以被编辑器真实理解、维护和发布。
单 agent 任务、图任务、特定任务启动都统一通过 TaskGraphDefinition -> graph_task_run 链路。
```

## 1. 设计基准

### 1.1 成熟编辑器的基本要求

成熟任务图编辑器不是普通节点画布。它至少要同时支持：

```text
结构：节点、边、阶段、frame、图模块、循环区域。
职责：节点身份、角色 prompt、边界、禁止行为、完成标准。
契约：图契约、节点输入输出契约、边载荷契约、验收契约。
资源：memory repository、artifact repository、issue ledger、read/commit matrix。
运行：执行者、运行模式、等待策略、汇合策略、模型需求、长度预算。
质量：审核门、返修边、人工门、重试策略、最终验收。
发布：预检、编译预览、GraphHarnessConfig 发布、graph_task_run 启动入口。
```

### 1.2 权威边界

编辑器必须遵守单向权威链：

```text
用户编辑 canonical graph draft
-> TaskGraphDefinition
-> GraphHarnessConfig
-> GraphRuntime / GraphLoop
-> NodeWorkOrder
-> AgentHarness / tool / human gate
```

禁止出现：

```text
SpecificTaskRecord 反向覆盖 node.contract_bindings。
Workflow / Flow 继续复制写作节点契约。
StandardView 被当作可编辑事实源。
Overlay graph 与 canonical graph 长期并存。
旧 coordination_task 语言变成新编辑器主语言。
```

### 1.3 写作配置包作为压力样本

`scripts/configure_writing_modular_novel_graph.py` 暴露了复杂任务图的真实需求：

```text
NodeSpec.prompt                    -> 节点角色 prompt / prompt contract
NodeSpec.artifact_paths            -> 多产物目标和路径模板
NodeSpec.readable_repositories     -> 资源读取矩阵
NodeSpec.write_mode                -> 记忆提交权限和提交路径
NodeSpec.review_revision_stage_id  -> 审核返修路径
NodeSpec.loop_scope_id             -> 循环作用域
NodeSpec.title_template            -> 批次标题模板
NodeSpec.loop_route_policy         -> 循环路由裁决
NodeSpec.length_budget             -> 长输出预算与修复策略
REPOSITORY_NODES                   -> memory/artifact/issue ledger 资源边界
graph_module nodes                 -> 图模块组合与隔离运行
```

这些不是写作专用特例，而是成熟任务图编辑器应当原生支持的通用配置能力。

### 4.2 旧 metadata.role_prompt 处理

短期允许读取：

```text
metadata.role_prompt
```

正式写入必须使用：

```text
metadata.prompt_contract.role_prompt
```

迁移后保存时：

```text
如果 metadata.role_prompt 存在且 prompt_contract.role_prompt 不存在：
  写入 metadata.prompt_contract.role_prompt
  标记 metadata.legacy_prompt_migration

如果两者都存在：
  prompt_contract.role_prompt 为权威
  metadata.role_prompt 只作为显示兼容，不再更新
```

最终清理条件：

```text
所有写作图节点都迁移到 metadata.prompt_contract.role_prompt 后，
发布器仍可读取 metadata.role_prompt，但编辑器不再写 metadata.role_prompt。
```

### 4.3 多产物目标

正式结构：

```json
{
  "artifact_policy": {
    "artifact_targets": [
      {
        "path": "world/world_candidate_round_{round_index:03d}.md",
        "required": true,
        "source": "node_spec",
        "visibility": "artifact_ref",
        "commit_stage": "staging_until_review"
```

兼容读取：

```text

```

编辑器写入只写：

```text
node.artifact_policy.artifact_targets

```

### 4.4 Loop / Batch

正式结构：

```text

graph.contract_bindings.runtime.split_policy
node.contract_bindings.unit_batch
node.contract_bindings.runtime.length_budget
node.loop_scope_id
node.loop_route_policy
metadata.graph_loop_policy.frames
```

编辑器应提供专门页面，不再让用户在多个 JSON 区域手工同步。

### 4.5 Graph Module

正式结构：

```text
node.node_type = graph_module
node.executor_policy.default_executor = graph_module
node.executor_policy.linked_graph_id
node.contract_bindings.runtime.graph_module_runtime.linked_graph_id
node.contract_bindings.handoff.handoff_contract_id
```

禁止：

```text
运行期重新读取 linked graph 并二次发布。
GraphLoop 在运行期解释 child graph language。
```

允许：

```text
编辑器展示 linked graph。
发布期展开或锁定 graph module runtime plan。
运行期只执行已发布 GraphHarnessConfig。
```

## 10. 系统级修订：图任务必须是一等控制平面

### 10.1 本计划的成熟度修正

原计划已经抓住了正确方向：

```text
TaskGraphDefinition 是唯一编辑事实源。
GraphHarnessConfig 是发布后的不可变运行投影。
StandardView / ComposableView 只读诊断。
写作图作为复杂压力样本。
特定任务统一走 graph_task_run。
```

但如果要达到生产级 agent 平台标准，还必须补足以下缺口：

```text
1. 图语言内核没有被单独定义为平台级 IR。
2. 编译器还没有明确 pass pipeline、输入输出和失败规则。
3. 运行控制和编辑器功能仍容易混在一起。
4. 持久化恢复仍偏 GraphLoopState JSON 存储，没有完整 checkpoint / event / pending work order 协议。
5. 节点通信、agent 与系统通信、memory 与 artifact 读写还没有统一成可审计协议。
6. UI 计划仍以页面功能为中心，缺少成熟工作流 IDE 的交互规格。
7. 版本治理、发布回滚、diff、schema migration 和 graph contract compatibility 没有进入主计划。
```

因此后续实施不能按“补几个编辑面板”的方式推进。必须按以下顺序做：

```text
Graph Language Kernel
-> Graph Compiler Pipeline
-> Graph Harness Runtime Contract
-> GraphLoop Durable Execution
-> Node Communication Protocol
-> Task Graph Workbench
-> Publish / Run / Monitor / Resume Closure
```

### 10.2 参考成熟系统时应借鉴的不变量

本项目不应该简单复制 LangGraph、Temporal 或 React Flow，但需要吸收它们的成熟不变量：

```text
Temporal 类系统：
  长流程的核心不是“函数调用”，而是 durable execution、event history、checkpoint、idempotent activity 和恢复后继续。

LangGraph 类系统：
  graph state 必须可 checkpoint；每个 run/thread 有独立状态线；失败或中断后恢复必须使用 checkpoint，而不是重新解释 live graph。

React Flow 类系统：
  编辑器层应把节点、边、handle/port、viewport、selection、custom node/edge 交给成熟画布基础设施；业务语义必须在图语言 IR 中，不在画布组件里。
```

本项目采用的结论：

```text
1. 图任务运行不依赖 LangGraph runtime，但 checkpoint/store/interrupt/pending writes 应优先复用 LangGraph Checkpointer 能力。
2. 长流程不直接上 Temporal，但要采用 event-sourced + idempotent work order + checkpoint resume 的 durable execution 模式。
3. 图编辑器建议引入 @xyflow/react / React Flow 作为画布底座；若暂不引入，当前自研画布只能作为过渡，不允许承担图语言语义。
```

依赖审查结论：

```text
requirements.txt 已声明 langgraph>=1.1.4,<2.0.0。
当前 Python 环境已安装 langgraph==1.1.4。
当前 Python 环境已安装 langgraph-checkpoint==4.0.1。
当前 Python 环境未安装 langgraph-checkpoint-sqlite。
当前 Python 环境未安装 langgraph-checkpoint-postgres。
```

因此设计上不应继续默认自研 checkpoint 存储。正确方案是：

```text
GraphLoop 持有项目领域接口 GraphCheckpointStore。
GraphCheckpointStore 的优先生产实现是 LangGraphCheckpointStore。
LangGraphCheckpointStore 内部适配 langgraph.checkpoint.base.BaseCheckpointSaver。
本地开发可使用 InMemorySaver 或安装 langgraph-checkpoint-sqlite 后使用 SqliteSaver。
生产环境优先安装 langgraph-checkpoint-postgres 并使用 PostgresSaver。
```

边界：

```text
允许 LangGraph 负责 checkpoint storage interface。
不允许 LangGraph 负责 TaskGraphDefinition 编译。
不允许 LangGraph StateGraph 接管 SchedulerView / GraphLoopState 推进。
不允许把 memory_read、artifact_context、review_gate、graph_module 语义映射成 LangGraph 隐式调度语义。
```

### 10.3 图任务系统的正式产品定义

图任务系统由六个子系统组成：

```text
1. Graph Language Kernel
   定义节点、边、资源、loop frame、contract、permission、memory、artifact、file、版本和扩展点。

2. Graph Compiler
   从 TaskGraphDefinition 编译 GraphHarnessConfig、SchedulerView、ContractManifest、Diagnostics。

3. Graph Harness
   锁定已发布 GraphHarnessConfig，创建 graph run、task run、runtime envelope。

4. Graph Loop
   基于 SchedulerView 和 GraphLoopState 推进 ready/running/completed/failed/waiting 状态。

5. Node Execution Protocol
   用 GraphNodeWorkOrder、GraphNodeInputPackage、NodeResultEnvelope 连接 AgentHarness、工具、人工门和系统服务。

6. Task Graph Workbench
   提供图语言的正式编辑、校验、发布、运行、监控、恢复和版本管理界面。
```

任何功能如果不能明确归属这六个子系统之一，就不能进入主链。

### 10.4 不允许再出现的设计退化

以下结构必须视为系统性错误，而不是普通技术债：

```text
把图任务当 UI 画布数据。
GraphLoop 读取 TaskGraphDefinition 或 editor draft。
运行期重新编译 linked graph。
graph_module 成为第三类 runtime。
memory_read / artifact_context 被当普通依赖边。
monitor projection 参与控制。
SpecificTask / Flow / Workflow 复制并覆盖 node contract。
节点 prompt 写成“这是 runtime 节点”。
agent 自称完成后系统直接信任并提交正式 memory / artifact。
失败恢复时从旧 metadata 补字段。
```

## 12. Graph Compiler Pipeline

### 12.1 编译器不是普通 mapper

编译器是任务图从编辑事实进入运行合同的唯一门。它必须显式分阶段：

```text
Pass 1: Load Canonical Graph
  输入 TaskGraphDefinition。
  输出 CanonicalGraphIR。

Pass 2: Normalize Legacy Fields
  只读旧字段，迁入正式字段。
  不产生运行语义 fallback。

Pass 3: Validate Graph Language
  校验 node/edge/loop/resource/contract schema。
  unknown control field fail closed。

Pass 4: Expand Composition
  graph_module / 父子块在发布期展开或内联。
  输出 scoped nodes、scoped edges、loop_frames、source_refs。

Pass 5: Build Resource Index
  生成 memory/artifact/file/issue ledger 索引。
  不从主 nodes/edges 删除资源语义。

Pass 6: Build SchedulerView
  只抽取调度依赖。
  context/commit/audit/event 边不阻塞 ready。

Pass 7: Build ContractManifest
  汇总 graph/node/edge/runtime/acceptance contracts。

Pass 8: Build GraphHarnessConfig
  写入不可变 config、content_hash、authority_map、diagnostics。

Pass 9: Publish
  存储 config。
  绑定 graph_id -> config_id。
```

### 12.2 编译失败规则

以下问题必须 fail closed：

```text
unknown executable node_type。
unknown scheduler_role。
graph_module 缺 linked_graph_id。
composition cycle。
config_hash mismatch。
resource node 被裁剪。
memory/revision 边无法归类。
review_gate 没有 pass/fail/revision 策略。
loop_frame 缺 router/continue/exit。
agent node 缺可执行 prompt_contract 或 task_instruction。
artifact required 但无 artifact policy。
formal memory commit 没有 commit gate。
```

### 12.3 GraphHarnessConfig 的定位

`GraphHarnessConfig` 是发布锁定物，不是第二事实源：

```text
可以：
  作为 GraphRuntime 的唯一静态输入。
  保存完整图语言的可执行投影。
  保存 source_refs 和 composition_sources 用于审计与回显。

不可以：
  被 UI 手工编辑成另一套事实。
  被运行期从旧 graph 或 metadata 增量补字段。
  覆盖 TaskGraphDefinition 的合同权威。
```

## 13. Durable GraphLoop 与恢复协议

### 13.1 GraphCheckpointStore

当前 `RuntimeObjectStore` 存 `GraphLoopState` 是可用基础，但不能作为最终成熟方案。生产级图任务需要显式 checkpoint 协议，并优先使用 LangGraph Checkpointer 作为持久化底座。

项目领域接口：

```text
GraphCheckpointStore
  put_checkpoint(graph_run_id, checkpoint, metadata) -> checkpoint_ref
  get_latest(graph_run_id) -> GraphCheckpoint | None
  list_checkpoints(graph_run_id, before?, limit?) -> GraphCheckpoint[]
  put_pending_writes(graph_run_id, task_id, writes) -> None
  delete_graph_run(graph_run_id) -> None
```

LangGraph 适配关系：

```text
graph_run_id
  -> RunnableConfig.configurable.thread_id

checkpoint_id
  -> LangGraph checkpoint id / checkpoint_ns 内部字段

GraphLoopState
  -> checkpoint.channel_values["graph_loop_state"]

active_work_orders
  -> checkpoint.channel_values["active_work_orders"]

GraphEvent cursor
  -> checkpoint.channel_values["event_cursor"]

pending work order / artifact / memory writes
  -> BaseCheckpointSaver.put_writes(...)

config_id / config_hash / schema_version
  -> checkpoint metadata
```

实现层：

```text
LangGraphCheckpointStore
  依赖 BaseCheckpointSaver。
  不 import StateGraph。
  不 compile LangGraph graph。
  不决定 ready nodes。
  只负责 checkpoint get/list/put/put_writes/delete_thread。

LocalJsonCheckpointStore
  仅作为迁移期 fallback 或调试工具。
  不作为生产默认。

InMemoryCheckpointStore
  仅测试使用。
```

```text
GraphCheckpoint
  checkpoint_id
  graph_run_id
  task_run_id
  config_id
  config_hash
  step
  parent_checkpoint_id
  event_cursor
  loop_state
  active_work_orders
  pending_writes
  recovery_receipts
  created_at
  authority
```

`pending_writes` 用于记录同一推进步中已经成功但尚未整体提交的 work order / artifact / memory 写入，避免恢复时重复执行已成功部分。

### 13.1.1 为什么不用 LangGraph 直接跑图

不直接使用 LangGraph StateGraph 跑本项目图任务，原因不是 LangGraph 不成熟，而是控制语义不一致：

```text
本项目的 GraphHarnessConfig 是任务合同，不是普通 DAG。
SchedulerView 只选择 scheduler_role=dependency 的边。
memory_read / artifact_context 是上下文装配边。
memory_commit / artifact_commit 是提交边。
revision_request 是条件返修边。
graph_module 在发布期展开，运行期不是子图 runtime。
agent 节点必须进入 AgentHarness。
权限、文件、memory、artifact 提交必须走本项目 receipt / gate。
```

如果直接把 TaskGraphDefinition 编译成 LangGraph StateGraph，LangGraph 就会成为第二个 GraphLoop。正确用法是借用其 checkpoint 抽象，而不是借用其图调度主权。

### 13.1.2 依赖补齐策略

当前已有：

```text
langgraph==1.1.4
langgraph-checkpoint==4.0.1
```

需要补齐：

```text
开发 / 单机持久化：
  langgraph-checkpoint-sqlite>=3.1.0,<4.0.0

生产持久化：
  langgraph-checkpoint-postgres>=3.1.0,<4.0.0
```

选择规则：

```text
测试：InMemorySaver。
本地开发：SqliteSaver。
生产：PostgresSaver。
没有安装 sqlite/postgres saver 时，GraphCheckpointStore 初始化必须明确失败或降级到标注为 dev-only 的 LocalJsonCheckpointStore，不能静默伪装成生产持久化。
```

### 13.2 WorkOrderStore

每个节点派发都必须幂等：

```text
work_order_id
idempotency_key
graph_run_id
node_id
config_id
config_hash
status = pending | dispatched | running | completed | failed | waiting_external | stale
executor_ref
dispatch_event_id
last_heartbeat_at
retry_count
created_at
updated_at
```

恢复时：

```text
completed work order 不重跑。
running + heartbeat alive 继续等待。
running + stale + idempotent 可重派。
running + stale + non-idempotent 进入 waiting_manual_recovery。
config_hash mismatch fail closed。
```

### 13.3 GraphLoop 状态推进

GraphLoop 的唯一输入：

```text

```

GraphLoop 的唯一输出：

```text

```

禁止输入：


### 14.2 Agent 可见世界只能来自 NodeInputPackage

```text
  memory_view
  artifact_view
  file_view

  tool_capability_table
```

Prompt 必须是 agent 能直接执行的任务语言：

### 14.3 Agent 不能直接提交正式资源

## 15. Task Graph Workbench 产品级结构

### 15.1 画布技术决策

图编辑器不应继续完全手搓复杂画布。目标技术路线：

```text
首选：@xyflow/react / React Flow 作为拓扑画布底座。
保留：当前 CoordinationTopologyGraph 可短期作为只读/轻量诊断画布。
禁止：把业务语义写进 canvas component。
```

React Flow 负责：

```text
节点拖拽、边连接、handle/port、viewport、selection、custom nodes、custom edges、keyboard interaction。
```

项目自有代码负责：

```text
TaskGraphDefinition。
GraphLanguage validation。
Inspector 编辑。
Compiler diagnostics。
Publish / run / monitor。
```

### 15.2 工作台层级

最终 Task Graph Workbench 应分为：

```text
Graph Library
  图资产列表、版本、发布状态、运行入口。

Graph Workbench
  左侧：图层导航。
  中央：React Flow canvas / topology view。
  右侧：Inspector。
  底部：Diagnostics / compiler output / run timeline。

Contract Studio
  图、节点、边合同编辑。

Resource Studio
  memory repository、artifact repository、file repository、issue ledger、read/write/commit matrix。

Runtime Studio
  GraphHarnessConfig、SchedulerView、NodeWorkOrder、GraphLoopState 只读预览。

Run Monitor
  graph run、node work orders、events、checkpoints、artifact/memory receipts。
```

### 15.3 UI 的硬性规则

```text
不同层级必须分页面或明确图层切换，不能堆在一页。
```

## 16. 版本、发布、回滚和兼容策略

### 16.1 图版本

```text

```

发布规则：

```text

已经启动的 graph_run 永远锁定启动时 config_id/config_hash。
```

允许：


禁止：

```text

让运行中的 graph_run 切换 config。
恢复时用新 draft 补旧 checkpoint。
```

```text

```

只允许保留：

```text
导入器。

明确有删除日期的迁移脚本。
```

## 18. 顶级验收矩阵

### 18.1 结构验收

```text
写作 master/design/chapter/finalize 图全部可保存、预检、发布。
GraphHarnessConfig.nodes/edges 不裁剪 memory_repository、issue_ledger、memory_read、memory_commit、revision_request。
SchedulerView 只包含调度边。
graph_module 发布期展开，运行期不启动子图 runtime。
```

### 18.2 运行验收

```text
单节点 graph_task_run 创建真实 TaskRun。
多节点 graph_task_run 创建真实 GraphRun + TaskRun。
首批 ready node work orders 可派发。
agent node 通过 AgentHarness 执行。
NodeResultEnvelope 被 GraphLoop 接收后推进下游 ready。
失败节点使 graph run fail closed。
```

### 18.3 恢复验收

```text
GraphCheckpointStore 通过 LangGraph BaseCheckpointSaver 适配实现。
本地持久化使用 SqliteSaver 或明确的 dev-only fallback。
生产配置可切换 PostgresSaver。
checkpoint 包含 config_id/config_hash/event_cursor/active_work_orders。
hash mismatch fail closed。
completed work order 不重复执行。
stale idempotent work order 可重派。
human_wait 状态可恢复。
resume 不 import task_system.compiler。
GraphLoop 不 import LangGraph StateGraph。
```

### 18.4 资源验收

```text
memory_read 生成 MemoryView，不阻塞调度。
agent 产出 memory candidate，不直接提交正式 memory。
memory_commit 节点或 gate 生成正式 commit receipt。
artifact target 进入文件管理候选区。
正式作品提交需要 commit gate。
```

### 18.5 UI 验收

```text
用户能在工作台看到：图结构、节点职责、输入输出、资源读写、审核返修、循环策略、发布版本、运行状态。
GraphHarnessConfig 只读。
StandardView / ComposableView 只读。
旧 task graph monitor decision UI 不存在。
运行监控显示 graph_run_id、config_id、checkpoint、work orders、events、node results。
```

### 18.6 反旧链验收

```text
rg 不再命中旧 graph runtime 主链。
新 graph harness 不 import task_system.compiler。
GraphLoop 不读取 TaskGraphDefinition。
GraphLoop 不读取 linked_graph_id。
前端不再调用旧 getTaskGraphRunMonitor / evaluateTaskGraphRunMonitor。
旧 workflow/flow/specific task 不保存可覆盖 graph contract 的写作节点合同。
```

## 19. 最终工程判断

这个系统做到位以后，图任务不是“画几个节点再跑一下”的机制，而是：

```text
一个可视化的、可编译的、可持久化恢复的、可审计的、可运行的 agent workflow control plane。
```

它的成熟标准不是“页面能画图”，而是：

```text
图语言能完整表达复杂任务。
编译器能严谨地产生不可变运行合同。
GraphLoop 能 durable 地推进状态。
节点能通过正式协议通信。
agent 只能在 RuntimeInvocationPacket / NodeInputPackage 给定世界中行动。
文件、记忆、产物、权限和工具全部有 receipt。
恢复不重新解释图。
监控不反向控制。
旧链路不能复活。
```

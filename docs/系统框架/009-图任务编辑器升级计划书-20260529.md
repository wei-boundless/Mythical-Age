## 0. 目标结论

本次升级不是把旧图编辑器复活，也不是把写作配置脚本搬进前端。

目标是：

```text
旧图编辑器的成熟工作台交互框架
+ 写作配置包暴露出的复杂任务图语义
+ 新链路 TaskGraphDefinition 的唯一权威模型
= 可维护、可发布、可运行的新图任务编辑器
```

升级后的编辑器必须满足：

```text
TaskGraphDefinition 是唯一编辑事实源。
GraphHarnessConfig 是发布后的不可变运行投影。
StandardView / ComposableView 只用于诊断和预览，不能成为第二套编辑权威。
写作图这种高密度任务图可以被编辑器真实理解、维护和发布。
单 agent 任务、图任务、特定任务启动都统一通过 TaskGraphDefinition -> graph_task_run 链路。
```

## 1. 设计基准

### 1.1 成熟编辑器的基本要求

成熟任务图编辑器不是普通节点画布。它至少要同时支持：

```text
结构：节点、边、阶段、frame、图模块、循环区域。
职责：节点身份、角色 prompt、边界、禁止行为、完成标准。
契约：图契约、节点输入输出契约、边载荷契约、验收契约。
资源：memory repository、artifact repository、issue ledger、read/commit matrix。
运行：执行者、运行模式、等待策略、汇合策略、模型需求、长度预算。
质量：审核门、返修边、人工门、重试策略、最终验收。
发布：预检、编译预览、GraphHarnessConfig 发布、graph_task_run 启动入口。
```

### 1.2 权威边界

编辑器必须遵守单向权威链：

```text
用户编辑 canonical graph draft
-> TaskGraphDefinition
-> GraphHarnessConfig
-> GraphRuntime / GraphLoop
-> NodeWorkOrder
-> AgentHarness / tool / human gate
```

禁止出现：

```text
SpecificTaskRecord 反向覆盖 node.contract_bindings。
Workflow / Flow 继续复制写作节点契约。
StandardView 被当作可编辑事实源。
Overlay graph 与 canonical graph 长期并存。
旧 coordination_task 语言变成新编辑器主语言。
```

### 1.3 写作配置包作为压力样本

`scripts/configure_writing_modular_novel_graph.py` 暴露了复杂任务图的真实需求：

```text
NodeSpec.prompt                    -> 节点角色 prompt / prompt contract
NodeSpec.artifact_paths            -> 多产物目标和路径模板
NodeSpec.readable_repositories     -> 资源读取矩阵
NodeSpec.write_mode                -> 记忆提交权限和提交路径
NodeSpec.review_revision_stage_id  -> 审核返修路径
NodeSpec.loop_scope_id             -> 循环作用域
NodeSpec.title_template            -> 批次标题模板
NodeSpec.loop_route_policy         -> 循环路由裁决
NodeSpec.length_budget             -> 长输出预算与修复策略
REPOSITORY_NODES                   -> memory/artifact/issue ledger 资源边界
graph_module nodes                 -> 图模块组合与隔离运行
```

这些不是写作专用特例，而是成熟任务图编辑器应当原生支持的通用配置能力。

## 2. 当前代码判断

### 2.1 可继承的旧编辑器设计

可学习但不照搬的文件：

```text

frontend/src/components/workspace/views/task-system/TaskGraphComposableEditorPage.tsx

frontend/src/components/workspace/views/task-system/TaskGraphResponsibilityPage.tsx
frontend/src/components/workspace/views/task-system/NodeResponsibilityCard.tsx
```

可继承设计：

```text
三栏工作台：左侧导航 / 中央画布 / 右侧 Inspector / 底部诊断。
拓扑画布：frame、状态、节点类型视觉区分、选择、节点工具、边工具。
对象 Inspector：根据 graph / node / edge / module / issue 分流编辑。
认知包预览：把节点最终收到的输入、输出、prompt、风险合成模型可理解视图。
节点职责卡：按“你是谁 / 只负责什么 / 不负责什么 / 完成标准”组织 role prompt。
```

### 2.2 必须限制的旧结构

不能继承的部分：

```text
Composable overlay 不能成为长期编辑事实。
StandardView 不能被编辑。
旧 coordination_run / coordination_task 不能成为新 UI 主语言。
DomainTask / SpecificTask 不应成为节点默认运行入口。
旧 Workflow / Flow 不能继续复制节点契约。
```

### 2.3 当前新编辑器基础

当前新编辑器已有基础：

```text

taskGraphSaveMapper.ts           保存 canonical nodes / edges / policies / contract_bindings。
```

主要短板：

```text

prompt_contract 不是节点一等编辑对象。

review gate / revision edge 还没有形成独立质量门编辑体验。
graph module 页面偏向编译预览，和 canonical 编辑边界仍需明确。
写作配置包尚未迁移为纯新图资产。
```

### 4.2 旧 metadata.role_prompt 处理

短期允许读取：

```text
metadata.role_prompt
```

正式写入必须使用：

```text
metadata.prompt_contract.role_prompt
```

迁移后保存时：

```text
如果 metadata.role_prompt 存在且 prompt_contract.role_prompt 不存在：
  写入 metadata.prompt_contract.role_prompt
  标记 metadata.legacy_prompt_migration

如果两者都存在：
  prompt_contract.role_prompt 为权威
  metadata.role_prompt 只作为显示兼容，不再更新
```

最终清理条件：

```text
所有写作图节点都迁移到 metadata.prompt_contract.role_prompt 后，
发布器仍可读取 metadata.role_prompt，但编辑器不再写 metadata.role_prompt。
```

### 4.3 多产物目标

正式结构：

```json
{
  "artifact_policy": {
    "artifact_targets": [
      {
        "path": "world/world_candidate_round_{round_index:03d}.md",
        "required": true,
        "source": "node_spec",
        "visibility": "artifact_ref",
        "commit_stage": "staging_until_review"
      }
    ]
  }
}
```

兼容读取：

```text
node.artifact_target
node.output_path
node.artifact_targets
contract_bindings.artifact.artifact_targets
```

编辑器写入只写：

```text
node.artifact_policy.artifact_targets
node.contract_bindings.artifact.artifact_targets
```

### 4.4 Loop / Batch

正式结构：

```text
graph.contract_bindings.unit_batch
graph.contract_bindings.runtime.split_policy
node.contract_bindings.unit_batch
node.contract_bindings.runtime.length_budget
node.loop_scope_id
node.loop_route_policy
metadata.graph_loop_policy.frames
```

编辑器应提供专门页面，不再让用户在多个 JSON 区域手工同步。

### 4.5 Graph Module

正式结构：

```text
node.node_type = graph_module
node.executor_policy.default_executor = graph_module
node.executor_policy.linked_graph_id
node.contract_bindings.runtime.graph_module_runtime.linked_graph_id
node.contract_bindings.handoff.handoff_contract_id
```

禁止：

```text
运行期重新读取 linked graph 并二次发布。
GraphLoop 在运行期解释 child graph language。
```

允许：

```text
编辑器展示 linked graph。
发布期展开或锁定 graph module runtime plan。
运行期只执行已发布 GraphHarnessConfig。
```

```text
prompt_contract 不丢。
artifact_targets 不丢。
loop/batch 不丢。
memory repository 和 memory edges 不丢。
review/revision 不丢。
graph_task_run 可启动。
旧 single_agent_task_run 不可从新入口启动。
```

### 7.2 实测

必须实测：

```text
1. 新建单节点任务图。
2. 配置 role prompt。
3. 配置 artifact target。
4. 发布 GraphHarnessConfig。
5. 创建 graph_task_run engagement plan。
6. 启动 graph run。
7. 查看 node work order。
8. 导入/运行写作 master 图。
```

### 7.3 写作图验收

写作图必须满足：

```text
四个图存在：
graph.writing.modular_novel.master
graph.writing.modular_novel.design_init
graph.writing.modular_novel.chapter_cycle
graph.writing.modular_novel.finalize

master 图包含三个 graph_module。
chapter_cycle 包含 loop/batch/length_budget。
节点 role prompt 全部可在编辑器看到。
资源库和 memory edge 可在资源流页面看到。
review gate 和 revision path 可在质量门页面看到。
发布后的 GraphHarnessConfig 保留完整图语言，不裁剪 memory/resource/revision 结构。
```

## 8. 风险点

### 8.1 最大风险：第二套权威

风险：

```text
为了 UI 方便，让 StandardView、ComposableView、overlay 或旧 task asset 继续写回执行事实。
```

处理：

```text
所有编辑只写 TaskGraphDefinition。
所有预览只读编译结果。
所有 overlay 必须迁移为 canonical node/edge 后才能发布。
```

### 8.2 最大风险：Prompt 变成开发说明

风险：

```text
编辑器字段引导用户写“这是 runtime 节点”之类开发说明。
```

处理：

```text
PromptContractEditor 的字段必须面向 agent：
你是谁。
你只负责什么。
你不负责什么。
你必须交付什么。
你如何裁决。
```

### 8.3 最大风险：写作图特殊化污染通用编辑器

风险：

```text
把 chapter、volume、world_review 写死进通用编辑器。
```

处理：

```text
通用编辑器只表达 unit、batch、loop、review、artifact、memory。
写作图只是一个配置样本。
```

## 9. 最终交付标准

本计划完成后，应达到：

```text
复杂写作图可以由新图编辑器查看、编辑、发布、启动。
编辑器能清楚表达 agent 节点最终会收到什么 prompt、输入包、资源和验收要求。
图模块、循环、资源、质量门不是 metadata 黑箱。
旧 workflow/flow/specific task/assignment 不再承载写作图节点契约。
特定任务启动统一走 graph_task_run。
```

最终判断标准：

```text
如果用户打开写作 master 图，不需要看 Python 配置脚本，也能理解：
这个任务分几个阶段；
每个节点是谁；
每个节点收到什么；
每个节点要交付什么；
哪些资源可读写；
审核失败回哪里；
循环何时继续或结束；
发布后会怎样运行。
```

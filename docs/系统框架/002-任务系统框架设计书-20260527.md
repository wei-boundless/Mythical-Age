# 任务系统框架设计书

日期：2026-05-27

状态：框架重校准稿

## 1. 总原则

任务系统不是所有 agent 行动的总入口，也不是 agent 的限制器。

任务系统的根本职责是：

```text
配置 TaskEnvironment，
维护任务库中的 SpecificTask，
根据已经确定的具体任务签发 TaskContract。
```

因此本轮定义收束为：

```text
TaskSystem =
  TaskEnvironment 配置系统
  + SpecificTask 任务库
  + TaskContract 签发系统
```

任务系统不负责：

```text
替 agent 做语义判断
执行 agent
调用工具
推进 loop 状态
管理执行生命周期
保存执行生命周期引用
执行 RuntimeAssemblyPlan / LoopPlan
承接所有 agent 自发的新执行生命周期
把普通工具调用升级为任务
```

## 2. TaskEnvironment

`TaskEnvironment` 是系统为一类任务打开的工作世界边界。

它不是每次任务的动态运行实例，而是预定义任务可以使用的资源、权限和制度上限。

它定义：

```text
environment_id
environment_type
resource_market
tool_market
permission_baseline
file_scope_policy
memory_scope_policy
artifact_scope_policy
agent_pool_policy
prompt_space_policy
skill_space_policy
model_policy
risk_policy
approval_policy
audit_policy
runtime_affordances
graph_runtime_policy
human_gate_policy
recovery_policy
```

`TaskEnvironment` 回答：

```text
这个具体任务发生在哪个工作世界？
这个世界最多能提供哪些工具、资源、memory、文件、artifact 和 agent？
哪些操作需要审批、审计、隔离或拒绝？
图任务和人工 gate 在这个环境中是否可用？
```

`TaskEnvironment` 不回答：

```text
本次用户到底要做什么？
agent 下一步行动是什么？
本轮 loop 应不应该继续？
某次工具调用是否已经完成？
```

## 3. SpecificTask

`SpecificTask` 是任务库里的具体任务定义，也是固定任务的工作申请模板。

它不是 runtime，不直接运行 agent。

它声明：

```text
task_id
task_title
适用 TaskEnvironment
目标结构
输入 schema
工作对象 schema
输出合同
验收规则
推荐 agent / agent profile
prompt pack / skill pack
工具能力需求
资源需求
memory 策略
artifact 策略
risk / approval 策略
runtime_shape
loop_plan
graph_ref / graph_policy
human_gate_policy
```

因此，具体任务本身已经具备“工作申请”的大部分字段。任务系统不应在订单工厂里重新猜测这些字段。

正确链路：

```text
SpecificTask
+ TaskEnvironment
+ 用户/API启动参数
-> TaskContract
-> Harness Runtime
-> Harness Loop
```

## 4. TaskContract

`TaskContract` 是任务系统签发给 Harness 的固定任务执行合同。

它代表：

```text
系统已经确认要执行任务库中的哪个 SpecificTask，
这个任务处于哪个 TaskEnvironment，
本次执行的输入、资源、工具、agent、输出、验收和 loop 流程是什么。
```

建议字段：

```text
contract_id
session_id
task_id
environment_id
source_ref
objective
working_objects
input_refs
runtime_requirements
loop_requirements
resource_scope
tool_scope
memory_scope
artifact_scope
agent_assignment
prompt_pack_refs
skill_pack_refs
output_contract
acceptance_policy
recovery_policy
approval_policy
risk_policy
graph_contract
human_gate_contract
status
created_at
```

`TaskContract` 不保存动态 loop 内部状态，也不保存执行生命周期引用。

禁止字段：

```text
latest_runtime_start_packet_ref
latest_loop_state_ref
result_ref
loop_lifecycle_ref
task_run_ref
```

动态状态、运行包引用、执行生命周期和结果回执都属于 Harness。任务系统可以通过独立观测/read model 查询执行侧事实，但不能把这些事实写回任务合同本体。

## 5. 订单与合同签发

订单不是任务合同签发机关。

订单最多表示：

```text
用户、UI、API 或任务库触发了某个 SpecificTask。
```

因此：

```text
TaskOrder = 来源记录 / 启动记录 / UI/API 申请载体
TaskContract = 任务系统签发给 Harness 的执行合同
```

当前代码里的 `TaskOrderFactory` 名称和职责需要收束。目标不应是让订单工厂发放任务合同，而是改为：

```text
TaskContractIssuer
  从 SpecificTask + TaskEnvironment + 启动参数签发 TaskContract。

TaskOrderRecorder
  只记录订单来源事实，不参与 runtime / loop 控制。
```

禁止：

```text
TaskOrderFactory 重新决定 environment。
TaskOrderFactory 重新决定工具权限。
TaskOrderFactory 签发 runtime 合同。
TaskExecutionEnvelope 成为任务合同本体。
ExecutionChannel 成为任务生命周期本体。
```

## 6. Harness 与 Agent 自发任务

agent 自发的新执行生命周期暂不进入任务系统。

本轮关闭以下链路：

```text
AgentWorkLifecycleIntent -> TaskSystem -> TaskContract
agent 自发沉淀可复用任务模板
agent 自动调用任务库 SpecificTask
```

关闭原因：

```text
先稳定固定任务合同、runtime 装配和 loop 状态控制。
避免任务系统重新膨胀成所有 agent 行动的总入口。
避免 agent 自发任务和预定义任务在任务系统内互相污染。
```

agent 自发需要新执行生命周期时，由 Harness 处理：

```text
AgentActionRequest
-> Harness RuntimeAssemblyRequest
-> RuntimeStartPacket
-> LoopLifecycle
```

它与固定任务“同源”的地方是：

```text
最终都进入 RuntimeAssembly 和 Loop。
```

它们不同源的地方是：

```text
固定任务来自 TaskSystem 签发的 TaskContract。
agent 自发执行来自 Harness 内部 action routing。
```

未来只有在以下情况才重新打开 agent 到任务系统的链路：

```text
agent 明确请求调用已注册 SpecificTask。
agent 请求把当前工作沉淀为可复用 SpecificTask 模板。
agent 需要任务系统提供固定验收、图流程、任务环境或审计合同。
```

该链路后续单独设计，不在本轮实现。

## 7. Runtime 装配

Harness Runtime 消费任务系统签发的合同，而不是反向解释任务系统。

任务系统可以在合同中给出 `RuntimeAssemblyPlan`。这里的 plan 是发送给 Harness Runtime 的装配请求规格，不是执行生命周期，也不是任务系统自己执行 runtime。

固定任务执行链：

```text
TaskContract
+ TaskEnvironment
+ AgentProfile
+ PermissionState
+ File / Memory / Artifact services
-> RuntimeStartPacket
```

`RuntimeStartPacket` 至少包含：

```text
packet_id
task_contract_ref
environment_ref
runtime_shape
loop_plan_ref
agent_profile
prompt_pack
skill_pack
tool_capability_table
resource_bindings
file_view
memory_view
artifact_policy
permission_baseline
approval_policy
execution_policy
output_contract
acceptance_policy
recovery_policy
budget_policy
risk_policy
observability_policy
```

Runtime 不允许：

```text
从 task_selection 猜任务合同。
从 domain_id / projection 猜环境。
从 metadata 补工具权限。
从 current_turn_context 改写 SpecificTask。
```

## 8. Loop 流程

Loop 是 Harness 的执行生命周期，不归任务系统管理。

Harness Loop 消费 `RuntimeStartPacket`，打开一次执行状态机。

任务系统可以在合同中给出 `LoopPlan`。这里的 plan 是发送给 Harness Loop 的循环流程请求规格，不代表任务系统拥有或推进 loop 生命周期。

目标链：

```text
RuntimeStartPacket
-> LoopLifecycle
-> AgentTurn
-> AgentActionRequest
-> PermissionDecision / ToolExecution / UserGate / HarnessChildLoop
-> Observation
-> LoopStateTransition
-> ResultEnvelope
```

Loop 状态由 Harness 持有。任务系统不记录 loop 引用，不推进 loop，不把 loop 状态写回 TaskContract。

目标状态：

```text
created
assembling
assembled
running
waiting_model
waiting_tool
waiting_user
waiting_approval
waiting_child_loop
observing
verifying
finalizing
completed
failed
cancelled
paused
resuming
```

禁止：

```text
finalizer 根据缺失字段补状态。
monitor / projection / read model 反向改状态。
tool executor 根据工具结果直接完成任务。
任务系统推进 LoopState。
任务系统保存 LoopLifecycle / RuntimeStartPacket / TaskRun 引用。
```

## 9. 图任务

图任务是任务库中的一种固定任务形态，不是自由 runtime。

图任务链路：

```text
SpecificTask(runtime_shape = task_graph)
+ TaskGraphDefinition
+ TaskEnvironment
-> GraphTaskContract
-> GraphRuntimeAssemblyPlan
-> GraphLoopPlan
-> GraphHarness
-> GraphLoop
```

GraphLoop 负责：

```text
推进图状态
判断节点依赖
处理 gate
调度节点
合并节点结果
判断图级完成或失败
```

GraphLoop 不负责：

```text
执行 agent model/tool 内环
选择 TaskEnvironment
扩大节点资源或权限
绕过固定任务合同
```

节点执行链：

```text
GraphLoop
-> NodeExecutionContract
-> RuntimeStartPacket(node)
-> AgentLoop
-> NodeResultEnvelope
-> GraphLoopStateTransition
```

节点不需要回到任务系统重新签发任务合同。节点是图合同内部的执行单元，受父 `GraphTaskContract` 和 `TaskEnvironment` 约束。

## 10. 人工任务与 Human Gate

人工 gate 是固定任务合同的一部分，不是独立任务来源。

```text
SpecificTask
-> TaskContract.human_gate_contract
-> Harness Loop waiting_user / waiting_approval
```

人工接管如果只是推进当前任务，不创建新任务合同。

未来如果需要把人工工作沉淀为任务库任务，再通过 SpecificTask 模板设计。

## 11. 实现结构

目标目录收束为：

```text
backend/task_system/
  environments/
    TaskEnvironment 定义、registry、resolver

  tasks/
    SpecificTask 定义、任务库记录、任务启动模板

  contracts/
    TaskContract
    GraphTaskContract
    NodeExecutionContract
    OutputContract
    AcceptancePolicy

  assembly/
    SpecificTask + TaskEnvironment -> TaskContract

  graphs/
    TaskGraphDefinition
    GraphRuntimeAssemblyPlan
    GraphLoopPlan
    graph contract helpers

  orders/
    只保留来源记录和 API 申请载体
    不保留 runtime / loop 控制权

  repositories/
    纯数据读写
```

Harness 侧：

```text
backend/harness/runtime/
  TaskContract -> RuntimeStartPacket

backend/harness/loop/
  RuntimeStartPacket -> LoopLifecycle / LoopState
  AgentActionRequest router

backend/harness/graph/
  GraphTaskContract -> GraphLoop
```

## 12. 必须避免的反模式

```text
让任务系统成为所有 agent 自发任务的总入口。
让 TaskOrderFactory 签发任务合同。
让 TaskOrder / TaskExecutionEnvelope / ExecutionChannel 成为任务本体。
让 task_selection、domain_id、projection 参与 runtime 控制。
让 Runtime 从 metadata 反推任务目标。
让 GraphNode 回到任务系统重新签发独立任务合同。
让图任务形成脱离 TaskEnvironment 的 runtime。
让任务系统推进 LoopState。
让任务系统保存执行生命周期引用。
让 LifecycleSuggestion 自动创建任务。
为了统一概念把 Harness 内部子 loop 强行写回任务系统。
```

## 13. 最终链路

固定单 agent 任务：

```text
TaskEnvironment
-> SpecificTask
-> TaskContract
-> RuntimeStartPacket
-> AgentLoop
-> ResultEnvelope
```

固定图任务：

```text
TaskEnvironment
-> SpecificTask(runtime_shape = task_graph)
-> GraphTaskContract
-> GraphRuntimeAssemblyPlan
-> GraphLoopPlan
-> GraphLoop
-> NodeExecutionContract
-> RuntimeStartPacket(node)
-> AgentLoop
-> GraphResultEnvelope
```

agent 自发新执行生命周期：

```text
AgentActionRequest
-> Harness RuntimeAssemblyRequest
-> RuntimeStartPacket
-> LoopLifecycle
```

暂时关闭：

```text
AgentActionRequest -> TaskSystem
Agent -> create reusable SpecificTask
Agent -> invoke registered SpecificTask
```

一句话总结：

```text
任务系统负责预定义任务的环境、任务库和合同签发；
Harness 负责实际 runtime 装配和 loop 执行；
agent 自发的新执行生命周期先留在 Harness 内部；
未来需要复用任务库或沉淀模板时，再单独打开 agent 与任务系统的受控对接链路。
```

## 1. 核心结论

Prompt 系统不是一组文案，而是 agent 的行为宪法、环境接口、人格表达和生命周期控制面。成熟 agent 的 prompt 体系必须做到：

1. 模型知道自己是谁、此刻负责什么、不能负责什么。
2. 系统知道每段 prompt 属于哪个权威层、什么时候挂载、为什么挂载。
3. 环境只提供边界和能力，不替模型判断用户意图。
4. 生命周期 prompt 按当前状态选择，不常驻全部状态说明。
5. 工具 prompt 说明工具契约，权限、执行和恢复由 runtime 控制。
6. 名称、人格、语气和互动风格属于可切换 personality prompt，不应固化为最高稳定系统词。
7. 子 agent 保持通信和证据回传，但最终裁决权始终属于当前会话主 agent。
8. 记忆、压缩摘要、todo、active work 都只是上下文材料，不能覆盖用户最新请求和真实工具观察。
9. 不再要求模型输出自评分 confidence；质量表达改为 evidence_refs、limitations、verification_status、source_strength、parse_quality 和系统计算分。

如果一段 prompt 不能让目标 agent 直接执行，而只是描述“这是某个 runtime 节点”或“这个节点用于某功能”，它就不是合格的 model-visible prompt。

## 2. 体系权威链

Prompt 体系应服务单向权威链：

```text
UserRequest
-> RequestFacts
-> BoundaryPolicy
-> PersonalityPromptController
-> EnvironmentPromptController
-> PromptMountPlan
-> RuntimePromptManifest
-> ModelTurnDecision
-> ActionPermit
-> ExecutionLoop
-> ObservationFollowup
-> PublicProjection
-> MemoryMaintenance
```

各层边界：

| 层级 | 合法职责 | 禁止事项 |
| --- | --- | --- |
| UserRequest | 提供当前用户意图最高语义信号 | 被旧任务、旧摘要或记忆自动覆盖 |
| RequestFacts | 记录请求、session、editor context、项目绑定等事实 | 判断用户意图或启动任务 |
| BoundaryPolicy | 判断权限、沙盒、外部访问和安全边界 | 伪装成模型语义判断 |
| PersonalityPromptController | 选择用户/session 指定人格，提供名称、语气和表达偏好 | 改写安全、工具、环境、生命周期或任务合同 |
| EnvironmentPromptController | 选择通用环境基座、任务环境 overlay、生命周期 prompt | 静默切换用户环境或全量挂载生命周期 |
| PromptMountPlan | 结构化说明本轮 prompt 来源、层级和触发原因 | 只返回不可审计字符串 |
| RuntimePromptManifest | 记录最终装配、cache 边界、诊断和拒绝项 | 隐藏 fallback 或吞掉缺失 prompt |
| ModelTurnDecision | 由当前会话主 agent 理解请求、选择下一步、调度动作 | 被分类器、环境、personality 或子 agent 替代最终判断 |
| ActionPermit | 授权工具、写入、命令、网络、浏览器和控制动作 | 根据 prompt 文案绕过权限 |
| ExecutionLoop | 执行动作、收集观察、反馈失败 | 重写用户目标 |
| ObservationFollowup | 基于真实观察继续判断 | 伪造未发生结果 |
| PublicProjection | 给用户展示进展和状态 | 反推隐藏 runtime 状态或执行工具 |
| MemoryMaintenance | 提出记忆候选和恢复摘要 | 写入未经证据支撑的长期事实 |

## 3. Prompt 分类

Prompt 库至少应包含以下类别。类别不是目录名，而是权威边界。

### 3.1 System Foundation

作用：定义稳定能力宪法、指令优先级、安全边界、上下文权威、验证真实性和协作底线。

要求：

- 不固定产品名、人格、口吻或互动风格。
- model-visible 文案可以说明“你是一名在用户本地项目中工作的协作 agent”，但不要把默认名称写成最高稳定词。
- 不写“Vibe coding agent foundation”这类开发标签给模型看。
- 不包含某个环境的专用任务规则。
- 不包含当前 turn 的动态状态。
- 新 prompt ID 不使用 `.v1` 后缀；版本放进 metadata。

### 3.2 Personality Prompt

作用：定义当前会话主 agent 的可见名称、人格气质、表达风格、互动偏好和默认语言倾向。

默认人格可以是 Mythical Age（洪荒智能），但它只是默认 personality，不是不可替换的系统身份。用户可以在 session 或个人配置中切换人格。

Personality prompt 的权威边界：

- 可以影响称呼、语气、表达节奏、解释风格、主动性、幽默度、严肃度和审美倾向。
- 可以提供默认展示名，例如 Mythical Age（洪荒智能）。
- 可以影响用户可见回复的风格，但不能改变 action schema。
- 不能覆盖系统/开发/项目规则。
- 不能改变权限、工具、环境边界、记忆治理、验证要求或任务合同。
- 不能让 agent 伪造结果、跳过验证、暴露隐藏推理或扩大执行范围。

建议资源结构：

```text
personality.default.mythical_age
personality.user.<profile_id>
personality.team.<profile_id>
```

建议 metadata：

```text
display_name
localized_name
tone
communication_style
initiative_level
verbosity_preference
language_preference
allowed_agent_refs
allowed_invocation_kinds
authority_scope: identity_and_style_only
user_configurable: true
```

切换策略：

- 默认人格由系统配置选择。
- 用户可以显式切换人格。
- session 可以保存当前人格选择。
- 模型不能静默切换人格。
- personality 缺失时回落到默认人格，但该 fallback 必须在 manifest 里可见。

### 3.3 Agent Role

作用：定义某个 agent 在某类 invocation 中的角色。

当前主 agent 应明确：

- 单轮判断时：理解用户最新请求，判断直接回答、只读观察、工具调用、active work control、持续任务 handoff 或阻塞。
- 持续执行时：围绕已建立任务合同推进，不重新判断是否建立生命周期。
- 工具观察后：基于真实观察继续，不预测工具结果。

写法原则：

- 不在稳定 agent role 中硬编码人格名称；名称由 personality prompt 提供。
- 不写“通用主 agent”“请求判断层”这种开发语境。
- 说明“你负责什么”和“你不负责什么”。
- 说明输入权威：用户最新消息、工具观察、active work、记忆、todo、压缩摘要各自地位。
- 说明输出边界：只能提交本轮 schema 允许的动作。

### 3.4 Runtime Protocol

作用：定义 action schema、JSON/native action 约束、输出字段和协议边界。

要求：

- 只说明协议，不重复生命周期判断文案。
- 不教模型如何做所有事情。
- 不承载环境能力说明。
- schema 没有的字段，模型不能自行添加。

Runtime prompt 的成熟形态应短、硬、可解析；语义判断交给 Agent Role 和 Lifecycle prompt。

### 3.5 Environment Prompt

作用：描述当前任务环境的资源边界、文件边界、工具边界、记忆命名空间和环境特定质量标准。

要求：

- 默认环境是 `env.general.workspace`。
- 其它任务环境只做 overlay，不能替换通用基座。
- 环境不是任务选择器，不能替模型判断用户意图。
- 用户或 session 控制环境切换；模型只能请求未来控制面确认，不能静默切换。
- 环境 prompt 不常驻 active work、tool recovery、memory handoff 等状态 prompt。

### 3.6 Lifecycle Prompt

作用：在特定生命周期状态下补充判断标准。

通用生命周期应覆盖：

1. context_intake：分清当前可见材料权威。
2. request_judgment：判断用户最新请求真正要求什么。
3. environment_capability_alignment：把目标与当前环境、工具和权限对齐。
4. action_selection：选择本轮最小充分动作。
5. active_work_control：仅在 active work 可见且 action 允许时挂载。
6. task_run_handoff：仅在允许建立持续任务时挂载。
7. user_steer_contract_revision：仅在已有工作和用户补充/修订相关时挂载。
8. tool_observation_recovery：仅在工具观察 followup 或 task execution 有观察时挂载。
9. memory_state_handoff：仅在本轮有可见记忆或压缩/维护状态时挂载。
10. finalization：收口前始终检查真实完成度和验证状态。

原则：

- 生命周期 prompt 是状态触发，不是环境常驻。
- 每段只解决一个生命周期问题。
- 不重复 runtime protocol。
- 不写开发说明；要让模型知道“此刻你应该如何判断”。

### 3.7 Tool Guidance

作用：说明具体工具什么时候用、什么时候不用、输入边界、失败处理和结果含义。

要求：

- read/edit/write/terminal/git/todo/subagent/browser/web fetch 都应有独立工具 prompt。
- 工具 prompt 不扩大权限，只说明在授权可见时如何使用。
- 失败必须被当作事实观察；下一步要改变参数、范围、工具或计划。
- 专用工具优先于 shell；shell 用于命令、脚本、构建、测试、服务和调试。
- 多工具调用由模型提出，runtime 根据安全、资源冲突和审批决定并发或串行。

### 3.7 Worker / Subagent Prompt

作用：定义子 agent 的专业角色、边界、输入、输出和失败处理。

原则：

- 子 agent 是 fresh specialist，不继承父线程完整上下文。
- 父 agent brief 必须像给刚进入任务的聪明同事交代任务：目标、已知事实、范围、排除项、可用 context_refs、期望输出、失败处理。
- 子 agent 不替 Mythical Age 做最终用户答复。
- 子 agent 返回后，Mythical Age 必须综合证据再决定下一步。
- 子 agent 未返回前不能预测、引用或编造其结论。
- 子 agent raw transcript 不进入父模型上下文；父上下文接收 summary、evidence_refs、limitations、recommended_parent_action 和必要 stable refs。

子 agent 输出建议：

```text
answer_candidate
positive_findings
negative_findings
files_read / sources_read
evidence_refs
limitations
open_questions
recommended_parent_action
```

### 3.8 Memory Prompt

记忆系统至少包含三类角色：

| 角色 | 责任 | 禁止事项 |
| --- | --- | --- |
| context_compactor | 把旧历史压缩成可恢复工作点 | 继续执行任务、读取文件、写记忆、补写新事实 |
| memory_manager | 根据当前会话提出记忆候选 | 回答用户、推进任务、决定物理存储和注入 |
| memory_searcher | 读取允许的历史记忆和恢复点 | 把历史记忆升级成当前事实 |

事实优先级：

```text
系统/开发/项目规则
-> 用户最新明确要求
-> 最新工具观察和当前文件事实
-> 当前任务合同 / active work
-> session summary / compaction
-> durable memory
```

长期记忆只保存跨会话稳定、有复用价值、经用户确认或由真实观察支撑的信息。临时计划、失败猜测、runtime 诊断、过期路径、旧任务状态和可从当前文件重新读取的事实，不应写成长期记忆。

## 4. 环境式 Prompt 架构

目标装配模型：

```text
general foundation
-> general environment base
-> selected task environment overlay
-> invocation lifecycle selector
-> visible tool guidance
-> project/user scoped instructions
-> dynamic runtime projection
-> volatile current user message / observations
```

`EnvironmentPromptController` 应输出结构化 `PromptMountPlan`：

```text
base_environment_id
selected_environment_id
base_prompt_refs
overlay_prompt_refs
lifecycle_prompt_refs
tool_guidance_refs
memory_prompt_refs
precedence_report
rejected_refs
diagnostics
```

环境切换策略：

- 默认使用通用环境。
- 用户或 session 控制环境选择。
- 其它环境只 overlay 通用基座。
- 可预留 `environment_switch_request` 接口，表示 Mythical Age 认为某任务更适合另一个环境。
- 该接口暂时不实装 action/API/UI；不能让模型直接切换环境。

## 5. 生命周期完整性

一次用户请求进入系统后，成熟行为应是：

1. API 收到请求，只做字段校验、session/project/editor context 绑定。
2. runtime 装配 request facts、active work、recent outcome、memory context、environment boundary 和可见工具。
3. EnvironmentPromptController 选择通用基座、overlay 和生命周期 prompt。
4. RuntimeCompiler 生成稳定 prompt、动态投影、volatile 用户消息和 output contract。
5. Mythical Age 判断当前请求语义。
6. Mythical Age 选择一个 action：respond、ask_user、tool_call、request_task_run、active_work_control、block 等。
7. ActionPermit 判断动作是否允许。
8. ExecutionLoop 执行动作并记录观察。
9. 工具观察回来后进入 observation followup，而不是由前端或旧分类器重新解释。
10. 完成后 PublicProjection 对用户展示结果。
11. 收口后 MemoryMaintenance 只在合适时提出记忆候选。

任何跳过该链路的旧 prompt、旧分类器、旧 fallback 或前端反推状态，都应该视为治理对象。

## 6. Model-Visible Prompt 写作标准

每段 model-visible prompt 必须回答八个问题：

1. 你是谁？
2. 你此刻只负责什么？
3. 你明确不负责什么？
4. 你会收到哪些输入，哪些输入更可信？
5. 你必须输出什么，或者只能请求什么动作？
6. 你的判断标准是什么？
7. 失败、缺证据、冲突、越界时怎么处理？
8. 哪些内容不能对用户或下游 agent 暴露？

合格写法：

```text
你是一名世界观审核员。
你只负责评审当前世界观设定是否完整、一致、可支撑后续写作。
你不负责替创作者扩写设定。
你需要指出问题、给出裁决、说明是否允许进入下一阶段。
```

不合格写法：

```text
这是 runtime 节点。
根据任务图执行 world_review。
这个节点用于校验资产。
```

禁止反模式：

- 把开发模块名、节点名、内部控制层名字直接写给模型。
- 用“请求判断层”“通用主 agent”替代 agent 身份。
- 在环境 prompt 中判断用户任务类型。
- 在 runtime protocol 中重复工具、记忆、active work 全部策略。
- 让模型自评 confidence。
- 用 prompt 弥补 action permit、tool runtime、projection 或 memory governance 的结构缺口。
- 为兼容旧链路保留无触发条件的 fallback prompt。

## 7. Prompt ID 与版本规则

新 prompt 资源不使用 `.v1`、`.v2` 作为 prompt_id 后缀。

建议：

```text
system.foundation.identity
agent.mythical_age.single_turn
runtime.protocol.single_turn
environment.general.lifecycle.request_judgment
tool.guidance.read_file
worker.prompt.web_research
memory.agent.maintenance
```

版本信息放在 metadata：

```text
version: "2026-06-08"
status: "active"
managed_by: "prompt_library..."
authority: "..."
```

旧 `.v1` prompt 可以在迁移计划中逐步改名，但不能通过长期 alias 保留两套主链路。迁移完成标准应是生产装配只使用新 ID，旧 ID 无 production import。

## 8. 子 Agent 通信协议

主 agent 和子 agent 的通信不能取消，只能治理。

父 agent brief 必须包含：

```text
goal
known_facts
scope
out_of_scope
context_refs
tools_or_capabilities_expected
evidence_requirements
output_contract
failure_handling
```

子 agent result 必须包含：

```text
status
summary
evidence_refs
files_read / sources_read
limitations
open_questions
recommended_parent_action
```

父 agent 集成原则：

- 子 agent 结果是证据候选，不是最终结论。
- 父 agent 必须处理冲突、缺证据和 stale 结果。
- 子 agent 不能让父 agent 免除读取关键文件、验证关键命令或对用户负责。
- 父 agent 不能把“理解用户请求”外包给子 agent。

## 9. 记忆与压缩治理

记忆体系不是一个长期记忆 prompt，而是一整套运行角色：

1. Session recovery summary：服务当前会话恢复。
2. Session emphasis：保存用户本会话显式强调的约束。
3. Durable memory：保存跨会话稳定信息。
4. Runtime fact bridge：只把治理后的事实桥接到记忆候选。
5. Context compactor：压缩旧历史，不能执行任务。
6. Memory manager：提出候选，不能回答用户。
7. Memory searcher：回溯历史，不能证明当前事实。

设计原则：

- 记忆不拥有当前事实权。
- 记忆写入必须有 evidence_excerpt 和 source_message_refs。
- 记忆注入必须带来源、范围、新鲜度和限制。
- 压缩摘要不能加入新事实。
- 记忆维护发生在提交后或明确维护阶段，不应干扰当前用户回复。

目标：

- EnvironmentPromptController 输出完整 mount plan。
- RuntimeCompiler 只消费 mount plan，不自行猜测环境 refs。
- precedence 从诊断升级为装配策略。

重点文件：

- `backend/harness/runtime/environment_prompt_controller.py`
- `backend/harness/runtime/assembly.py`
- `backend/harness/runtime/compiler.py`
- `backend/prompt_library/assembly.py`
- `backend/prompt_library/manifest.py`

### Phase 3：生命周期按状态挂载

目标：

- active work prompt 只在 active work 可见且 action 允许时挂载。
- tool recovery prompt 只在 observation followup 或 task execution 有观察时挂载。
- memory handoff prompt 只在有可见记忆、压缩或维护状态时挂载。
- 普通直接回答不被长期任务、记忆写入和工具恢复 prompt 干扰。

### Phase 4：子 Agent 通信 envelope

目标：

- spawn brief 和 result envelope 标准化。
- 父上下文只接收 summary、refs、limitations、recommended action。
- 子 agent 不能覆盖父 agent 裁决权。

### Phase 5：记忆与压缩 prompt 治理

目标：

- memory manager、compactor、memory searcher 三类角色彻底分离。
- 删除模型自评 confidence。
- 记忆候选必须有证据和来源。
- 压缩摘要只服务恢复，不升级事实。

### Phase 6：Prompt Governance Linter

建议新增检查：

- model-visible prompt 不包含开发节点口吻。
- 新 prompt_id 不带 `.v1`。
- worker/memory/subagent prompt 不要求 confidence。
- 每段 prompt 有 role、responsibility、boundary、input、output、failure handling。
- runtime protocol 不重复 lifecycle 规则。
- environment prompt 不包含任务选择逻辑。
- mount manifest 能解释每段 prompt 的 layer、trigger、cache_scope 和 authority。

## 12. 验收标准

体系成熟后，应满足：

1. 任意本轮模型输入都能从 manifest 解释每段 prompt 来源、权威和触发原因。
2. 普通问答不会挂载 active work、task handoff、tool recovery、memory handoff 的无关重负载 prompt。
3. 环境切换只能由用户/session 控制，模型不会静默切环境。
4. Mythical Age 始终拥有用户请求理解和最终裁决权。
5. 子 agent 保持可通信、可观察、可证据回传，但不污染父上下文。
6. 记忆、压缩摘要和 todo 都不会覆盖当前用户请求或工具观察。
7. 工具失败会进入观察恢复，而不是被 prompt 警告吞掉。
8. 不存在模型自评 confidence 字段作为运行裁决依据。
9. prompt 文案可以直接给目标 agent 执行，不包含开发者写给开发者看的说明。
10. 旧 prompt ID、旧 lifecycle 常驻路径和旧 fallback 在切换完成后没有生产入口。

## 13. 总结

洪荒智能的 prompt 体系应是一套环境式、生命周期式、可审计的 agent 行为系统。它的核心不是“写得更长”，而是每段 prompt 都有清晰权威、触发条件、输入边界、输出合同和失败处理。

主 agent 的灵魂是 Mythical Age（洪荒智能）：它理解用户、调度动作、综合证据并负责最终答复。系统负责装配环境、授权工具、执行命令、记录观察、投影进展和治理记忆。子 agent、记忆管理员和压缩员都是这个体系中的专业角色，但不能取代主 agent 的最终裁决权。

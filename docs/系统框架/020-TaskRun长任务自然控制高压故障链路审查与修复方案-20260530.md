## 1. 结论摘要

高压长任务实验失败不是随机性，也不是游戏任务本身的问题。失败是稳定复现的结构性问题：

```text
自然语言控制层可以识别用户要继续。
用户补充要求可以被记录为 steer。
但状态层在 executor/checkpoint 后残留 executor_status=running 和 runtime_control 旧状态。
active_work 候选层、resume_policy、task_executor 对同一个 TaskRun 的可继续性各自判断。
最终 resume_policy 或 executor 入口拒绝继续，后续 steer 只记录、不进入执行 packet。
```

最新高压实验样本：

```text
路径：output/test_runs/long-task-pressure-20260530-104011-c462c1
脚本：backend/tests/system_eval/long_task_natural_language_pressure_experiment.py
命令：python backend/tests/system_eval/long_task_natural_language_pressure_experiment.py --rounds 24 --premature-every 4

final_status: waiting_executor
pending_user_steer_count: 14
resume_plan: refuse / not_resumable:waiting_executor
task diagnostics.executor_status: running
task diagnostics.runtime_control.state: paused
task status: waiting_executor
```

这是一组互相矛盾的状态：外层状态说 `waiting_executor`，控制状态说 `paused`，执行器诊断还说 `running`。系统后续判断被这组脏状态污染。

## 2. 成熟架构标准

参考 Codex / Claude Code 这类成熟 agent 的外显机制，长任务控制应满足以下不变量：

```text
1. 用户自然语言意图只在 turn decision 层裁决一次。
2. 可继续性由统一状态语义裁决，不由多个模块分别猜。
3. executor 是执行层，不应通过 terminal_reason 白名单重新决定恢复策略。
4. checkpoint / interrupted / waiting boundary 后必须清理运行态租约，不能保留 running。
5. pending user steer 必须进入下一次 execution packet；未消费前不能假装任务完成。
6. monitor 是投影层，只展示事实，不修复事实。
7. health system 只治理记录，不拥有恢复执行权。
```

目标权责链应固定为：

```text
UserSubmission
-> ActiveWorkResolver          收集候选，不判断用户语义
-> ActiveWorkTurnDecision      判断暂停、继续、追加、询问、停止
-> ResumePolicy                裁决 same-run / checkout / wait / refuse
-> TaskRunControl              写入控制请求
-> TaskRunScheduler            调度 executor
-> TaskRunExecutor             执行 packet，不重判用户意图
-> RuntimeEventLog/StateIndex  记录事实和最新状态
-> RuntimeMonitor              投影事实
```

当前故障违反的是第 2、3、4 条。

## 3. 实测链路复盘

### 3.1 成功部分

实验前半段链路能工作：

```text
1. 初始长任务创建 TaskRun。
2. 用户自然语言暂停被识别为 pause_active_work。
3. task_run_pause_requested 被记录。
4. 用户自然语言继续被识别为 continue_active_work。
5. resume_plan_created 一度为 same_run_resume。
6. task_run_resume_requested 被记录。
7. task_run_executor_scheduled / claimed 发生。
8. 第一条 steer 被 included / consumed，revision 被 decided。
```

这说明自然语言入口和 steer 记录不是主因。

### 3.2 失败起点

执行器返回一个可恢复阻塞：

```text
action_type: block
blocking_reason: pressure_checkpoint_waiting_for_next_user_control
action_request.diagnostics.recoverable_error.retryable: true
action_request.diagnostics.recovery_action: resume_task_run
```

这在语义上是“等待下一轮用户控制后可继续”，不是永久失败。

但最终 TaskRun 状态变成：

```text
status: waiting_executor
terminal_reason: waiting_executor
diagnostics.executor_status: running
diagnostics.runtime_control.state: paused
diagnostics.recovery_action: resume_task_run
diagnostics.recoverable_error.retryable: true
pending_user_steer_count: 14
```

问题不是没有 recoverable 信息，而是 recoverable 信息与 stale running 信息并存。

### 3.3 后续为什么只记录 steer 不执行

后续用户继续输入仍被识别为当前工作控制：

```text
answer_source: harness.loop.active_work_turn
terminal_reason: continue_active_work
active_work.status: waiting_executor
```

但 resume plan 反复变成：

```text
decision: refuse
reason: not_resumable:waiting_executor
```

原因是 `active_work.py` 内部先把候选归类为 `active`：

```python
if status in {"created", "running"} or (
    status in _ACTIVE_WORK_STATUSES and executor_status in {"scheduled", "running"}
):
    return "active"
```

在 `waiting_executor + executor_status=running` 的污染状态下，候选 `continuation_kind=active`。

但转换成 `ActiveWorkContext` 后：

```python
running=candidate.continuation_kind == "active" and candidate.status in {"created", "running"}
same_run_allowed=candidate.same_run_allowed
```

此时 `candidate.status=waiting_executor`，所以：

```text
context.running = false
context.same_run_allowed = false
context.resumable = false
context.status = waiting_executor
```

于是 `resume_policy.build_resume_plan()` 进入兜底：

```text
refuse / not_resumable:waiting_executor
```

这就是“执行层为什么不承认”的直接链路：不是用户意图没有识别，而是上游传给 resume_policy 的上下文已经变成一个自相矛盾的对象。

## 4. 代码证据

### 4.1 可执行性判断仍依赖 terminal_reason 白名单

文件：`backend/harness/loop/task_executor.py`

```python
def is_task_run_executable(task_run: Any) -> bool:
    return (
        str(task_run.status or "") == "waiting_executor"
        or _is_recoverable_protocol_terminal(task_run)
    ) and task_run_control_state(task_run) not in {
        _TASK_RUN_PAUSE_REQUESTED,
        _TASK_RUN_PAUSED,
        _TASK_RUN_STOP_REQUESTED,
        _TASK_RUN_STOPPED,
    }
```

```python
def _is_recoverable_protocol_terminal(task_run: Any) -> bool:
    diagnostics = dict(getattr(task_run, "diagnostics", {}) or {})
    recoverable = dict(diagnostics.get("recoverable_error") or {})
    terminal_reason = str(getattr(task_run, "terminal_reason", "") or "")
    return (
        str(getattr(task_run, "status", "") or "") in {"failed", "blocked"}
        and terminal_reason in {"model_action_invalid", "model_action_protocol_repair_required", "task_execution_step_budget_exceeded"}
        and bool(recoverable.get("retryable", True))
    )
```

这段逻辑的问题不是白名单内容少，而是执行层不应把某些 terminal_reason 当成恢复主权。恢复主权应来自结构化状态合同，例如：

```text
recoverable_error.retryable != false
recovery_action in resume_task_run / rerun_task_executor
task not stopped / aborted / graph-controlled / completed
```

### 4.2 terminal/checkpoint 结束没有统一清理运行态

文件：`backend/harness/loop/task_executor.py`

`_finish_executor_terminal()` 当前把 terminal payload 合并进 diagnostics，并提升 `recoverable_error/recovery_action`，但没有统一重写：

```text
executor_status
latest_step_status
runtime_control.state
active_packet_ref / claimed lease
```

因此旧的：

```text
executor_status=running
runtime_control.state=running 或 paused
```

可以在 blocked/waiting checkpoint 后继续留在最新 TaskRun 上。

### 4.3 active_work 同时把 running 当事实和推断

文件：`backend/harness/loop/active_work.py`

`_continuation_kind()` 把 `executor_status=running` 作为 active 的依据；`active_work_context_from_candidate()` 又只把 `created/running` 状态当 running。两个函数对同一候选的状态语义不一致。

这导致候选层既不明确承认它正在运行，也不允许它继续。

### 4.4 resume_policy 已经开始接受 recoverable blocked，但权威不足

文件：`backend/harness/loop/resume_policy.py`

```python
elif context.same_run_allowed or context.resumable or _is_recoverable_blocked_context(context, task_run):
    decision = "same_run_resume"
```

这比旧逻辑更接近正确方向，但它仍依赖 `ActiveWorkContext` 的布尔字段。如果上游 context 已被 stale executor_status 污染，resume_policy 无法稳定修正。

## 5. 根因分层

### 5.1 事实层根因

`TaskRun.status`、`diagnostics.executor_status`、`diagnostics.runtime_control.state` 缺少统一状态转换函数。多个函数直接用 `replace(... diagnostics={...})` 局部覆盖，导致 terminal/checkpoint 后没有清理旧运行态字段。

### 5.2 决策层根因

same-run 可恢复性分散在至少三处：

```text
active_work._continuation_kind()
resume_policy.build_resume_plan()
task_executor._is_task_run_resumable_for_user_control()
task_executor.is_task_run_executable()
agent_loop._schedule_task_executor()
agent_loop._task_executor_should_continue()
```

这些函数输入相似、结论不同，且没有共享同一个 structured recovery contract。

### 5.3 执行层根因

执行层用 `_is_recoverable_protocol_terminal()` 根据 terminal_reason 白名单重新判断 recoverable。这属于执行层越权。执行器只应该验证：

```text
1. 当前 TaskRun 被授权可执行。
2. 没有并发 executor claim。
3. 没有 stop/abort/graph-control 禁止条件。
4. ResumePolicy 或状态合同允许 same-run execution。
```

### 5.4 观测层根因

monitor 能展示 `waiting_executor` 和 `paused`，但它只是投影，不能清理 `executor_status=running`。如果事实层污染，monitor 只能把污染投出来，不能承担修复职责。

## 6. 修复原则

禁止的修复：

```text
1. 给 _is_recoverable_protocol_terminal() 继续加 terminal_reason 白名单。
2. 在高压实验里绕过 resume_policy。
3. 降低 pending steer / revision 的完成断言。
4. 让 monitor 或 health system 回写 TaskRun 修复状态。
5. 保留两套可恢复性判断，以“兼容”为理由继续并存。
```

必须采用的修复：

```text
1. 建立单一 TaskRun recovery state helper。
2. 所有 same-run resumable / executable / active candidate 判断都调用同一结构化状态语义。
3. terminal/checkpoint closeout 必须重写 executor_status，不允许 running 残留。
4. recoverable blocked 应转为明确的 waiting_executor 或 blocked_recoverable 语义。
5. stop/aborted/completed/graph_node_assigned 仍必须强拒绝 same-run resume。
6. checkout/fork 语义仍按 016 文档推进，不用 same-run 修复掩盖 terminal interrupted 工作。
```

## 7. 目标状态合同

建议新增统一状态判定模块：

```text
backend/harness/loop/task_run_recovery_state.py
```

核心函数：

```python
class TaskRunRecoveryState:
    status: str
    executor_status: str
    control_state: str
    recoverable: bool
    recovery_action: str
    same_run_resumable: bool
    executable: bool
    running_claimed: bool
    paused: bool
    stopped: bool
    graph_controlled: bool
    reason: str
```

裁决规则：

```text
completed/success:
  same_run_resumable=false
  executable=false
  continuation=completed_iteration

aborted/user_aborted/stopped/stop_requested:
  same_run_resumable=false
  executable=false
  continuation=checkout_fork only when policy allows

graph_node_assigned:
  same_run_resumable=false
  executable=false from chat control

running + executor_status in scheduled/running:
  running_claimed=true
  same_run_resumable=false
  executable=false for new schedule

waiting_executor + control_state not paused/stop_requested/stopped:
  same_run_resumable=true
  executable=true

waiting_executor + control_state paused:
  same_run_resumable=true for user continue
  executable=false until resume request clears paused

blocked/failed + recoverable_error.retryable != false + recovery_action resume_task_run/rerun_task_executor:
  same_run_resumable=true unless stopped/aborted/graph-controlled
  closeout should normalize to waiting_executor where the system expects same-run execution

blocked/failed non-retryable:
  same_run_resumable=false
  executable=false
```

## 8. 修复实施方案

### Phase 1：统一恢复状态权威

新增：

```text
backend/harness/loop/task_run_recovery_state.py
```

迁移调用：

```text
backend/harness/loop/active_work.py
backend/harness/loop/resume_policy.py
backend/harness/loop/task_executor.py
backend/harness/loop/agent_loop.py
```

完成标准：

```text
active_work、resume_policy、task_executor 对同一个 TaskRun 给出同一个可恢复性结论。
不再有 executor terminal_reason 白名单作为恢复主权。
```

### Phase 2：修复 executor closeout 状态同步

修改 `_finish_executor_terminal()`：

```text
1. 从 action_request.diagnostics 提升 recoverable_error / recovery_action。
2. 计算 normalized closeout state。
3. recoverable blocked/checkpoint:
   status=waiting_executor 或 status=blocked + executor_status=waiting_executor，二者必须与 active_work 约定一致。
   terminal_reason 保留真实 blocking_reason。
   runtime_control.state 不允许 running；若不是用户显式 paused，则清空或设为等待态。
   latest_step_status=waiting_executor 或 blocked。
4. completed:
   executor_status=completed
   清理 active_packet_ref。
5. failed non-retryable:
   executor_status=failed
6. aborted/stopped:
   executor_status=stopped
```

建议优先采用：

```text
recoverable block -> TaskRun.status=waiting_executor
diagnostics.executor_status=waiting_executor
terminal_reason=<真实 checkpoint reason>
diagnostics.recovery_action=resume_task_run
```

原因：现有 scheduler 和 CLI watch 已围绕 `waiting_executor` 建模，风险低于新增 `blocked_recoverable` 外部状态。但文档和 helper 中必须保留 `blocked + recoverable` 的兼容识别，用于读取旧污染数据和未迁移路径。

### Phase 3：清理 active_work 状态矛盾

`_continuation_kind()` 不再直接解释 `executor_status=running`。它应读取 `TaskRunRecoveryState`：

```text
running_claimed -> active
paused -> paused
same_run_resumable -> waiting
recoverable blocked -> blocked_recoverable 或 waiting
completed -> completed_iteration
checkoutable terminal -> interrupted_checkoutable
otherwise ignore
```

`active_work_context_from_candidate()` 不能再二次推翻 candidate 语义：

```text
context.running = candidate.continuation_kind == "active"
context.resumable = candidate.same_run_allowed
```

### Phase 4：替换 executor 入口越权判断

替换：

```text
_is_recoverable_protocol_terminal()
_is_task_run_resumable_for_user_control()
is_task_run_executable()
agent_loop._schedule_task_executor() 的裸 waiting_executor 判断
agent_loop._task_executor_should_continue() 的裸 waiting_executor 判断
```

为统一 helper 调用。

保留规则：

```text
is_task_run_executor_claimed() 只判断真实运行租约。
resume_paused_task_run() 只接受 same_run_resumable 且非 stopped。
execute_task_run() 只接受 executable 或已声明可接管的 scheduled/running claim。
agent_loop scheduler 只调度 recovery state 明确 executable 的 TaskRun。
```

### Phase 5：补强回归和高压实验

新增测试：

```text
1. recoverable blocked + recovery_action=resume_task_run -> natural continue schedules executor。
2. recoverable blocked closeout 不残留 executor_status=running。
3. waiting_executor + executor_status=running 脏数据不会产生 not_resumable:waiting_executor。
4. paused waiting_executor + continue -> resume_requested -> scheduled。
5. non-retryable blocked -> wait/refuse，不 same-run resume。
6. user stopped / aborted / completed -> 不 same-run resume。
7. graph_node_assigned -> chat natural control 不接管。
8. pending steer 多轮追加后，下一次 packet include 全部 pending steer。

```

## 9. 与 016 计划书的关系

本方案不是替代 `016`，而是补上高压实验暴露出的底层状态修复。

`016` 的主线是：

```text
自然控制
-> work continuation candidate
-> resume policy
-> same-run / checkout fork
-> rollout history
```

本方案处理的是：

```text
same-run / recoverable checkpoint 的状态合同必须先正确。
否则 checkout/fork 和 rollout 做完后，仍会被 stale executor_status 污染。
```

因此实施顺序应调整为：

```text
1. 先做本方案 Phase 1-4，修复状态合同和执行层越权。
2. 再继续 016 中 checkout/fork 与 WorkRollout 的完整升级。
3. 最后接 UI 和 HealthSystem 治理视图。
```

这不违反 016，反而是 016 能稳定运行的前置条件。

## 10. 文档自审：遗漏、矛盾和风险

### 10.1 已确认没有矛盾的点

```text
1. 本方案没有把 terminal aborted 改成 same-run resume；aborted 仍走 checkout/fork。
2. 本方案没有让 monitor 或 health system 修事实；事实修复仍在 executor/state 层。
3. 本方案没有降低 pending steer 约束；相反要求 pending steer 被 include/consume。
4. 本方案没有继续扩大 terminal_reason 白名单；要求删除其恢复主权。
5. 本方案允许读取旧污染数据，但不是保留旧链路；旧污染识别只用于迁移和防御。
```

### 10.2 仍需实施时重点核查的遗漏

```text
1. task_checkout.py 已有 checkout 逻辑，后续与 016 合并时要避免第二套恢复状态。
2. monitor_projection 对 stale_waiting_executor 的诊断应只报警，不回写。
3. WorkRollout 的 append_work_rollout_item 可能也读取 status，需要确认不会记录错误 interrupted_boundary。
4. runtime_control 缺少 waiting 这种显式状态；实施时要决定清空 running 还是新增 waiting，并同步测试。
5. executor_epoch / active_packet_ref / next_invocation_index 的 closeout 规则需要和 packet id 去重一起测。
```

### 10.3 当前方案的主要风险

```text
1. 如果 recoverable block 统一转成 waiting_executor，可能掩盖需要用户回答的 blocked。必须以 recovery_action=resume_task_run/rerun_task_executor 作为前提。
2. 如果 helper 过度集中但没有覆盖 graph_node_assigned，可能让主会话接管图节点任务。必须强拒绝 graph-controlled。
3. 如果清理 runtime_control.state 太激进，可能丢掉用户显式 paused。用户 paused 应保留，executor 自然 checkpoint 不应伪装成 paused。
4. 如果 active_work context 和 candidate 同时保留旧字段，可能再次出现语义分叉。实施后应删除重复推断。
```

### 10.4 最终验收标准

```text
1. 高压实验稳定通过，不再出现 not_resumable:waiting_executor。
2. recoverable checkpoint 后 diagnostics.executor_status 不再是 running。
3. 用户多轮补充和推翻后，pending steer 被下一轮 packet 纳入并消费。
4. pause/continue/stop/checkout/completed iteration 各自状态路径可解释。
5. 代码中只有一个 same-run resumable 权威。
```

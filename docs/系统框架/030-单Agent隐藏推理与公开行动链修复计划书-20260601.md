# 单Agent隐藏推理与公开行动链修复计划书

日期：2026-06-01

## 1. 问题结论

当前单 agent 长任务实测暴露的问题不是“缺少一句状态文案”，而是隐藏推理、任务状态、系统事件和用户可见展示的边界没有固定。

已观察到的问题：

- 长任务执行期间反复显示“正在分析当前目标、已有进展和最新要求”“正在思考”等空泛状态。
- agent 的真实行动判断没有稳定进入用户可见状态流。
- runtime 等待、上下文装配、模型动作请求等系统过程被当成用户进展展示。
- 工具观察、agent 判断、下一步行动没有被分层记录和投影。
- 续跑时 executor 可以恢复为 `waiting_executor`，但用户看到的是模糊状态，无法判断真实进度。
- 之前尝试通过新增较多字段、alias、前端替换规则解决问题，这个方向过宽，存在新的系统性风险。

本计划的修复目标是：以成熟 agent 的方式建立“隐藏推理增强行动力，公开状态解释行动过程，系统证据保证真实性”的最小闭环。

## 2. 成熟 Agent 标准

成熟 agent 使用内部推理辅助任务执行，但不会把原始思维链直接展示或持久化。

正确分层是：

```text
隐藏推理层
-> 模型动作层
-> 系统执行与证据层
-> 压缩任务状态层
-> 用户可见进展层
```

各层职责：

| 层级 | 职责 | 不允许做的事 |
| --- | --- | --- |
| 隐藏推理层 | 模型在本次调用内理解任务、权衡风险、规划下一步、自我审查 | 不输出原始思维链，不持久化，不作为下一轮上下文原文回灌 |
| 模型动作层 | 输出下一步 action、公开行动摘要、必要的任务状态更新 | 不伪造工具观察，不替系统记录事实 |
| 系统执行与证据层 | 调用工具、记录真实 observation、artifact、错误和恢复事件 | 不替 agent 做语义判断 |
| 压缩任务状态层 | 维护当前目标、已确认事实、当前判断、下一步、风险、验收状态 | 不累积原始思考文本，不把每轮状态无限追加进上下文 |
| 用户可见进展层 | 展示工具状态、真实观察、agent 公开判断、下一步和结果 | 不展示隐藏推理，不用前端硬编码猜测语义 |

## 3. 设计原则

1. **隐藏推理不可见、不可持久化**
   - prompt 可以要求 agent 在内部完成状态审查和自我检查。
   - 不新增 `chain_of_thought`、`reasoning_trace`、`thoughts` 这类可见或持久字段。
   - 不把内部推理原文写进 event log、work rollout、memory 或下一轮 model context。

2. **持久化任务状态，不持久化思维链**
   - 持久化对象应是 compact task state：
     - `confirmed_facts`
     - `latest_observation_refs`
     - `current_judgment`
     - `next_action`
     - `open_risks`
     - `completion_status`
     - `artifact_refs`
   - 状态按字段替换或合并，不按自然语言流水账无限追加。

3. **系统事实来自系统，不来自模型编造**
   - 工具结果、文件路径、artifact、错误、恢复事件只能由 runtime/tool/event ledger 产生。
   - agent 只能引用和总结这些观察，不能把自己的判断伪装成工具事实。

4. **公开进展是行动摘要，不是思维链摘要**
   - 展示格式应类似：

```json
{
  "public_progress_note": "已确认 HTML 产物存在，下一步检查事件触发逻辑后收尾。",
  "public_action_state": {
    "evidence_refs": ["rtobs:read_file:..."],
    "current_judgment": "主要产物存在，剩余风险是事件机制是否完整可触发。",
    "next_action": "检查战斗、陷阱、宝箱、休息入口；满足则提交 artifact 路径。",
    "completion_status": "verifying"
  }
}
```

   不应该是：

```text
我首先想到……然后我又想到……所以我准备……
```

5. **UI 只消费权威投影，不写语义补丁**
   - 前端不得继续靠字符串替换把内部状态改写成用户状态。
   - 后端 runtime monitor 必须输出干净的 `tool_status / latest_observation / current_judgment / next_action / completion_status`。

6. **等待状态不能覆盖行动状态**
   - 模型等待 heartbeat 只能说明 executor 活着。
   - heartbeat 不应覆盖最近一次 agent 公开行动判断。
   - 用户看到的最新语义状态应来自最近一次有效模型动作、工具观察或诊断事件。

7. **严格禁止启发式关键词扩展**
   - 不做 `analysis -> judgment`、`plan -> next_action` 这种 alias 推断。
   - 协议字段必须 canonical。
   - 模型输出不合规时让协议修复，而不是系统猜测。

## 4. 当前错误实现需要回收的部分

之前临时改动中需要收缩或撤回的方向：

| 位置 | 问题 | 处理 |
| --- | --- | --- |
| `model_action_protocol.py` | `progress_report` alias 映射会引入启发式关键词 | 删除 alias，仅保留 canonical schema |
| `agent_loop.py` | 普通短 turn 强制完整行动链可能造成协议修复循环 | 只在 task execution 强制，普通 turn 先保持轻量 |
| `RuntimeRunSummary.tsx` / `presentation.ts` | 前端继续用字符串替换修语义 | 清理新增替换，前端只显示后端投影 |
| `session_timeline.py` | 为展示新增过多 helper | 收缩成单一投影函数，避免展示层承担协议逻辑 |
| 测试 stub | 为迎合过宽协议大量补字段 | 回到最小测试，只验证 task execution 的公开行动链 |

## 5. 目标架构

### 5.1 模型调用提示

task execution prompt 必须包含以下语义：

```text
你需要先在内部完成任务状态审查、证据核对、风险判断和下一步选择。
不要输出隐藏推理链。
你必须输出 public_progress_note 和 public_action_state。
public_action_state 只记录公开行动状态，不记录完整思维过程。
最终完成声明必须基于合同、真实观察、artifact 和验证证据。
```

### 5.2 模型动作协议

只在任务执行阶段要求：

```json
{
  "authority": "harness.loop.model_action_request",
  "action_type": "respond|ask_user|tool_call|block",
  "public_progress_note": "用户可理解的一句话行动摘要",
  "public_action_state": {
    "evidence_refs": [],
    "current_judgment": "",
    "next_action": "",
    "open_risks": [],
    "completion_status": "working|verifying|ready_to_finish|blocked"
  },
  "final_answer": "",
  "tool_call": {},
  "diagnostics": {}
}
```

字段原则：

- `public_progress_note`：用于 UI 最新摘要。
- `public_action_state.current_judgment`：公开判断，不是隐藏推理。
- `public_action_state.next_action`：下一步具体行动。
- `public_action_state.evidence_refs`：引用系统 observation，不复制大段观察内容。
- `public_action_state.open_risks`：公开风险项，便于续跑和用户理解。
- `public_action_state.completion_status`：收尾判断状态。

### 5.3 任务状态压缩

系统维护 `task_execution_state`：

```json
{
  "confirmed_facts": [],
  "latest_observation_refs": [],
  "current_judgment": "",
  "next_action": "",
  "open_risks": [],
  "completion_status": "",
  "artifact_refs": [],
  "updated_from_action_ref": ""
}
```

写入规则：

- 工具 observation 由系统追加到 evidence ledger。
- agent action 只更新 judgment / next_action / risks / completion_status。
- 每次 runtime 装配只给模型 compact state + latest evidence refs。
- 不把所有历史 `public_progress_note` 原文塞回上下文。

### 5.4 用户可见状态流

显示顺序：

1. 工具调用状态：系统生成。
2. 工具观察摘要：系统从 observation ledger 投影。
3. agent 公开判断：来自 `public_action_state.current_judgment`。
4. 下一步：来自 `public_action_state.next_action`。
5. 结果正文：来自 final answer。
6. 诊断状态：来自 runtime monitor，不伪装成 agent 思考。

等待 heartbeat 只更新活跃时间，不覆盖 3/4。

## 6. 实施计划

### 阶段 A：回收错误方向

目标：撤掉会制造新污染的临时结构。

动作：

1. 删除 `model_action_protocol.py` 中的 alias 映射。
2. 普通 `agent_loop.py` 不强制完整 `public_action_state`。
3. 回退前端新增的语义替换规则。
4. 保留“等待 heartbeat 不覆盖最新有效行动状态”的原则。

验收：

- 搜索不到新增 alias 表。
- 前端没有新增“正在思考/正在分析”类替换补丁。
- 普通聊天不会因为缺少行动链进入协议修复循环。

### 阶段 B：任务执行协议最小升级

目标：只在长任务/TaskRun 执行阶段强制公开行动状态。

动作：

1. 在 `ModelActionRequest` 中新增 canonical `public_action_state`。
2. `task_executor` 调用解析时要求 `public_progress_note + public_action_state`。
3. `compiler.task_execution_action_schema()` 加入字段说明。
4. `_runtime_projection_instruction()` 加入隐藏推理和公开状态边界说明。

验收：

- 缺少 `public_action_state.current_judgment` 或 `next_action` 时，任务执行动作协议失败并回灌修复观察。
- 普通 turn 不受影响。

### 阶段 C：任务状态与上下文装配

目标：防止思维链污染上下文。

动作：

1. 新增或复用 task execution state projection。
2. 从模型 action 中提取：
   - `current_judgment`
   - `next_action`
   - `open_risks`
   - `completion_status`
   - `evidence_refs`
3. 写入 task diagnostics 或 runtime object 中的 compact state。
4. `RuntimeCompiler.compile_task_execution_packet()` 只装配 compact state 和 latest evidence refs。

验收：

- model-visible context 中没有历史思维链文本。
- 多轮执行后 dynamic context 不线性堆积 public progress。
- prompt cache 的 stable payload 不受动态状态污染。

### 阶段 D：监控与会话显示

目标：显示真实行动过程，不显示内部套话。

动作：

1. runtime monitor 输出：
   - `latest_progress.tool_status`
   - `latest_progress.observation`
   - `latest_progress.current_judgment`
   - `latest_progress.next_action`
   - `latest_progress.completion_status`
2. session timeline 从 monitor 投影，不自行猜测。
3. 前端 `RuntimeRunSummary` 只展示这些字段，不写语义推断。

验收：

- 长任务执行时 UI 能看到“观察/判断/下一步”。
- heartbeat 期间仍保留最近一次有效行动状态。
- 不再反复显示“正在分析当前目标”。

### 阶段 E：恢复与收尾

目标：续跑后 agent 能基于 compact state 继续，而不是重新搜索和重复读文件。

动作：

1. 恢复后 runtime 装配 compact state。
2. 若已有 artifact refs 和验证观察，prompt 要求优先判断是否收尾，而不是重复读取。
3. `waiting_executor` 恢复时不覆盖最新行动状态。

验收：

- 后端重启后同一 TaskRun 可续跑。
- 续跑后 agent 能看到上次判断、风险和下一步。
- 不重复执行同一类无意义读取超过合理次数。

## 7. 测试计划

### 单元测试

1. `model_action_protocol`
   - task execution 缺少 `public_action_state` 会失败。
   - canonical 字段可被解析。
   - alias 字段不被接受。

2. `task_executor`
   - 模型 action 记录公开行动状态。
   - wait heartbeat 不覆盖最新行动状态。
   - 协议修复观察不会伪装成用户进展。

3. `runtime_monitor_projection`
   - 投影 observation / current_judgment / next_action / completion_status。
   - stale diagnostics 覆盖正常进展，但不丢失最后有效行动状态。

4. `dynamic_context_projection`
   - 上下文只包含 compact task state 和 evidence refs。
   - 不包含历史原始 reasoning / progress 流水账。

### 实测

1. 普通聊天：不进入 TaskRun，不要求完整行动链。
2. 简单任务：创建小 artifact，显示观察/判断/下一步，正常收尾。
3. 长任务：五层地下塔 HTML 游戏。
4. 中途用户纠偏：agent 记录用户补充，更新 compact state 后继续。
5. 后端重启续跑：TaskRun 不丢状态，恢复后不重复无意义读取。

## 8. 不做事项

- 不展示原始思维链。
- 不持久化原始思维链。
- 不把思维链原文作为下一轮上下文。
- 不在前端增加语义猜测规则。
- 不通过关键词启发式判断 agent 意图。
- 不为了测试硬编码任务结果。
- 不把短对话强行纳入长任务行动链协议。

## 9. 完成标准

修复完成必须同时满足：

1. 代码结构上，隐藏推理、公开行动状态、系统证据、UI 展示四层边界清晰。
2. 运行表现上，长任务状态显示真实行动过程，而不是套话。
3. 上下文管理上，不出现思维链原文堆积。
4. 恢复能力上，续跑使用 compact state 接续工作。
5. 测试上，通过 focused 单元测试和 CLI 实测。

最终判断标准不是“UI 句子变好听”，而是 agent 是否能更稳定地理解任务状态、减少重复动作、基于真实证据推进和收尾。

# Codex式公开进展协议与内联运行体验修复方案

日期：2026-05-30

状态：自检通过后实施

## 1. 问题定义

当前运行进展体验的问题不是单纯 UI 丑，也不是文案替换不够多，而是协议层没有把“用户可见的阶段说明”作为一等输出。

现状链路是：

```text
ModelActionRequest.action_type
-> task_executor / agent_loop 写入固定 step summary
-> session_timeline 投影为 progress_entries
-> RuntimeRunSummary 再清洗 runtime 文案
```

这导致三个问题：

1. 用户看到的是“系统说明”，不是 agent 自然推进过程。
2. 前端被迫根据 `action_type`、`step`、`event_type` 猜用户可见文案。
3. 旧的 `runtime packet`、`TaskRun`、`agent 请求`、`执行器` 等内部词会反复泄漏到对话区。

正确修复方向是把公开进展变成模型动作协议的一部分：

```text
ModelActionRequest.public_progress_note
-> runtime event / work rollout
-> session runtime timeline
-> inline progress UI
```

## 2. Codex源码参照结论

已检查本地 Codex 源码：

```text
C:\Users\admin\.codex\sources\openai-codex
```

Codex 不靠前端把运行日志翻译成“自然过程”。它在协议层把一轮执行拆成一等 item：

| Codex 项 | 作用 | 本项目借鉴 |
| --- | --- | --- |
| `AgentMessage` | 助手对用户可见的自然语言输出 | `public_progress_note` / 最终回答 |
| `MessagePhase::Commentary` | 中途说明 | 本次修复的公开进展字段 |
| `MessagePhase::FinalAnswer` | 最终回答 | 现有 `final_answer` |
| `Reasoning.summary_text` | 可公开推理摘要 | 不直接暴露私有思维链，只允许公开摘要 |
| `CommandExecution` / `FileChange` / `McpToolCall` | 客观执行证据 | 保留工具、观察、文件和验证事件 |
| `ItemStarted` / `ItemCompleted` | 明确开始/完成 | 前端根据事件状态显示完成项和当前项 |

关键原则：

1. 不暴露完整 chain-of-thought。
2. 用户可见过程由模型主动写出，且必须是公开、安全、简短的阶段说明。
3. 工具执行、文件变化、失败、暂停、完成是客观证据，不由 UI 编造。
4. UI 不从内部事件名推断 agent 的想法。

## 3. 目标体验

用户看到的应该类似：

```text
我先核对当前文件状态，确认上次中断点是否已经落盘。
已完成 1/3

我发现资源路径没有被页面引用，下一步补齐引用并验证加载。
已完成 2/3
```

而不是：

```text
任务运行 · 2 个工具
系统已为当前任务步骤装配 runtime packet，并交给 agent 判断下一步。
```

外显要求：

1. 普通聊天不加框。
2. 任务进展是内联进展流，不是大卡片。
3. 默认只显示当前进展和完成计数。
4. 展开后显示最近步骤。
5. agent 简要输出显示在当前阶段下面。
6. 开发词必须过滤，不作为新链路默认文案。
7. 暂停、继续、恢复仍然通过同一会话自然语言控制。

## 4. 权责链

目标权责链固定为：

```text
ModelTurnDecision
  生成 action_type、工具请求、最终回答、公开进展说明。

ModelActionProtocol
  解析和校验结构化动作；清洗 public_progress_note。

TaskRunExecutor / AgentLoop
  执行动作、记录公开进展和客观事件；不重新编写 agent 意图。

WorkRollout
  记录可恢复进展、简要输出、断点和 artifact 引用。

SessionTimeline
  把事实源投影为聊天消息可消费的 progress_entries。

RuntimeRunSummary
  只渲染公开进展和状态；不根据内部事件名编造自然语言。

HealthSystem / Monitor
  汇总任务记录和运行健康，不接管进展表达。
```

禁止事项：

1. 前端不能继续用大量正则把内部文案翻译成自然话术。
2. `public_runtime_progress_summary()` 只能做安全清洗，不能成为主要文案生成器。
3. 执行器不能把 `runtime packet`、`执行器`、`TaskRun` 当作用户进展写入。
4. 不能把公开进展说明当作私有思维链。

## 5. 协议设计

### 5.1 `ModelActionRequest` 新字段

新增字段：

```python
public_progress_note: str = ""
```

含义：

```text
agent 给用户看的当前阶段公开说明。
它必须简短、自然、可公开、面向用户。
它可以说明正在做什么、刚判断出什么、下一步会做什么。
它不得包含私有推理过程、隐藏系统规则、runtime packet、TaskRun、执行器、内部模块名、工具授权细节。
```

长度建议：

```text
清洗后 160 字以内。
超过限制截断。
```

兼容策略：

```text
解析旧 action request 时允许缺省。
新 prompt schema 必须要求模型填写。
缺省时 runtime 只能使用 action_type 的低信息 fallback，不能生成伪装成 agent 的复杂叙事。
```

### 5.2 prompt schema 修改

`model_action_request_schema()` 和 `task_execution_action_schema()` 都新增：

```json
{
  "public_progress_note": "一句给用户看的当前阶段说明；不要写思维链、内部系统词、runtime、TaskRun、执行器、packet。"
}
```

`_runtime_projection_instruction()` 增加输出规则：

```text
每次输出 JSON 时必须填写 public_progress_note。
它是用户可见的阶段说明，不是推理草稿。
不要描述系统如何装配上下文、如何调用 runtime 或如何校验协议。
```

## 6. 事件与投影设计

### 6.1 step summary

`step_summary_recorded` 保留，但 payload 增强：

```json
{
  "step": "model_action_received:1",
  "status": "running",
  "summary": "我先检查当前项目状态，确认可以从哪里继续。",
  "public_progress_note": "我先检查当前项目状态，确认可以从哪里继续。",
  "agent_brief_output": "",
  "presentation_source": "model_action.public_progress_note"
}
```

说明：

1. `summary` 为兼容字段，仍保存用户可见摘要。
2. `public_progress_note` 是新链路首选字段。
3. `presentation_source` 用于调试和防止后续误把内部模板当作公开输出。

### 6.2 work rollout

`WorkRolloutItem` 现有字段足够：

```text
summary              -> public_progress_note
agent_brief_output   -> 最终回答摘要、工具观察摘要或阶段简要输出
payload.action_type  -> 客观动作类型
refs                 -> action / observation / checkpoint 引用
```

本次不新增 rollout 表结构，只改变写入来源。

### 6.3 session timeline

`progress_entries` 增强输出：

```json
{
  "title": "思考下一步",
  "body": "我先检查当前项目状态，确认可以从哪里继续。",
  "publicNote": "我先检查当前项目状态，确认可以从哪里继续。",
  "agentBrief": "",
  "evidenceType": "model_action",
  "statusText": "running"
}
```

前端优先级：

```text
publicNote > body > toolName > title
agentBrief 只用于辅助显示，不替代 publicNote。
```

## 7. 前端设计

### 7.1 信息层级

默认折叠态：

```text
[状态点] 阶段名  公开进展说明                2/4 已完成
        agent 简要输出
```

展开态：

```text
[✓] 思考下一步  我先检查当前项目状态，确认可以从哪里继续。  已完成
[✓] 执行操作    已读取目录和入口文件。                            已完成
[●] 验证结果    我正在跑页面验证，确认改动真实生效。              进行中
```

### 7.2 视觉原则

1. 不用卡片包普通聊天。
2. 进展区无边框或极轻左侧线，压低视觉权重。
3. 工具证据折叠显示，避免抢占对话空间。
4. 桌面与移动端都必须避免文字重叠。
5. 不显示 “任务运行”“系统已执行 agent 请求”等开发式标题。

## 9. 切换和回退规则

切换：

```text
新 prompt schema 立即要求 public_progress_note。
旧 action payload 缺少该字段时，后端保留兼容 fallback。
前端优先新字段，但仍能显示旧 progress_entries。
```

回退：

```text
如果模型未输出 public_progress_note，不阻塞执行。
如果公开进展字段含内部词，清洗层过滤后显示。
如果新字段引起前端类型问题，progress_entries 仍可退回 body 显示。
```

明确禁止：

```text
不能保留旧 runtime 文案作为新任务主体验。
不能为了测试通过删除失败路径。
不能让前端凭 action_type 生成复杂 agent 叙事。
```

## 10. 自检结论

### 10.1 与现有设计是否冲突

不冲突。

`016-Codex式任务恢复与自然控制重构升级计划书` 解决恢复、checkout/fork、自然控制。本方案解决执行过程如何自然展示。两者共享同一原则：TaskRun 是内部状态机，用户只看到当前工作进展。

### 10.2 与健康系统职责是否冲突

不冲突。

健康系统继续管理任务记录、监控汇总和诊断，不负责生成或改写运行进展。公开进展由 agent 动作协议产生，由 runtime 记录，由 timeline 投影。

### 10.3 是否暴露思维链

不暴露。

`public_progress_note` 是公开阶段说明，不是私有推理。prompt 和清洗层都会禁止私有推理、隐藏系统规则和内部运行结构。

### 10.4 是否会破坏旧链路

风险可控。

新增字段是向后兼容字段；旧 payload 不包含该字段时继续执行。前端优先新字段但保留 `body` fallback。执行和授权链路不改变。

### 10.5 主要风险

1. 模型初期可能漏填 `public_progress_note`。
   - 控制：schema 和 runtime instruction 双重要求，后端 fallback 不阻塞。
2. 旧事件仍包含内部词。
   - 控制：清洗层保留安全过滤，前端不把旧事件作为新体验标准。
3. UI 仍可能显得像状态面板。
   - 控制：无卡片、无边框、内联、默认折叠、只显示当前说明。

结论：方案可以实施。

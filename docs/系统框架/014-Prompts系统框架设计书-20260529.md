# Prompts 系统框架设计书

日期：2026-05-29

## 1. 核心结论

Prompts 系统应该成为稳定 prompt 资产的统一库，而不是 runtime 事实、mode 解释、任务识别或旧 task_goal_type 路由的集合。

正确边界：

```text
Prompts Library
  存稳定、可复用、可版本化的 prompt 资产

Runtime Assembly
  根据 agent/profile/environment/task/graph/specific-task 选择本次需要的 prompt refs

Runtime Projection
  根据本次已装配结果生成动态边界说明

Runtime Compiler
  将 prompt assets + runtime projection + volatile state 装配成 model packet
```

禁止边界：

- Prompts 库不能做用户意图识别。
- Prompts 库不能根据关键词启动任务。
- Prompts 库不能授权工具。
- Prompts 库不能保存 observations、tool result、task_run 当前状态、runtime envelope。
- Mode 不能直接变成 `mode instruction`，只能控制 runtime 装配。

## 2. 当前代码链路审核

### 2.1 单 agent 主链

主链入口：

```text
backend/query/runtime.py
-> AgentHarness
-> backend/harness/runtime/assembly.py::assemble_runtime
-> backend/harness/runtime/compiler.py::RuntimeCompiler
-> backend/harness/loop/agent_loop.py / task_executor.py
```

当前有效 prompt 来源：

| 来源 | 当前文件 | 当前状态 | 问题 |
|---|---|---|---|
| Runtime system prompt | `backend/harness/runtime/compiler.py` | 主链有效 | 大段硬编码，应迁入 runtime prompt pack |
| Runtime projection | `backend/harness/runtime/compiler.py` | 主链有效 | 刚替换 mode instruction，方向正确；后续应独立模块化 |
| Agent work role prompt | `storage/orchestration/agent_runtime_profiles.json` / `runtime_profile_registry.py` | 主链有效 | 应从 profile 内联迁移为 prompt resource ref |
| Environment prompt | `backend/task_system/environments/default_environments.py` | 主链有效 | 应由 environment config 引用 prompt resource |
| Soul role prompt | `backend/soul/assembly_service.py` | role mode 有效 | 边界正确，只能作为表达锚点 |
| Tool schema/view | `backend/harness/runtime/assembly.py` | 主链有效 | 是 tool manifest，不是 prompt asset |
| RuntimeEnvelope / observations / task_run state | runtime 运行时对象 | 主链有效 | 只能是 volatile state，不得入库 |

### 2.2 图任务 / 特定任务链路

图任务 prompt 来源：

```text
task_system/compiler/graph_harness_config_publisher.py
-> harness/graph/context_materializer.py
-> query/runtime.py::_graph_node_contract_from_work_order
-> TaskRunContract.prompt_contract
-> RuntimeCompiler.compile_task_execution_packet
```

当前字段：

```text
role_prompt
task_instruction
output_instruction
forbidden_behavior
definition_of_done
```

判断：

- 这是正确的任务 prompt contract 形态。
- 单节点契约和图任务契约应该统一使用这套结构。
- 图编辑器应编辑 prompt contract，而不是编辑开发说明。
- 旧 `prompt_library` 中保存的 stage_role 资产可迁移为 `task.graph_node.role` 类 prompt resource。

### 2.3 旧 query 静态 prompt

当前残留：

```text
backend/query/runtime.py imports prompting.build_static_prompt
backend/api/sessions.py / backend/api/tokens.py 读取 build_system_prompt_for_session
```

判断：

- 这不是新单 agent 主链的 runtime prompt 来源。
- 它目前主要服务旧 session prompt 展示和 token 估算。
- 后续应改为读取最近一次 RuntimeInvocationPacket / RuntimePromptManifest。
- 旧 `backend/prompting/*` 不应继续作为新 harness prompt 权威。

### 2.4 现有 prompt_library

当前状态：

- `backend/prompt_library/models.py` 已有 `PromptResource`、`PromptSelectionContext`、`PromptAssemblyPlan`。
- `storage/prompt_library/prompt_resources.json` 主要保存写作图任务 `stage_role`。
- `selector.py` 仍包含 `task_goal_type` 匹配逻辑。
- `assembler.py` 会生成“你本轮要完成的任务类型是：...”等旧语义任务 prompt。

判断：

- 现有 prompt_library 可以作为新 Prompts Library 的基础，但必须改权威边界。
- `task_goal_type` 不能是普通 turn 的 prompt 选择依据。
- 对于特定任务或图任务，类型/节点/合同可以作为显式契约字段使用，但不能作为系统猜测用户意图的入口。

## 3. 设计原则

### 3.1 Prompt Asset 与 Runtime State 分离

Prompt asset 是稳定资产：

- 可版本化。
- 可复用。
- 可缓存。
- 可审查。
- 可被环境、agent、task、skill、graph 节点引用。

Runtime state 是本次运行事实：

- 用户当前消息。
- history。
- observations。
- tool results。
- task_run status。
- active/historical failures。
- artifact evidence。
- operation_authorization。
- runtime envelope。
- agent_visible_runtime_projection。

Runtime state 不进入 prompts 库。

### 3.2 Mode 是装配控制，不是 prompt 类型

Mode 只能决定：

- 可用动作集合。
- 是否允许 TaskRun。
- 是否装配 soul role prompt。
- self-review 强度。
- step summary / todo 策略。
- 上下文范围。
- 权限和工具可见性策略。

Mode 不应该生成：

```text
当前 runtime 是 professional 模式...
```

Agent 应看到的是 runtime projection：

```text
本次运行边界：
- 你可以直接回答、询问用户、调用本次可见工具。
- 当目标需要真实交付物、持续执行或验证闭环时，可以请求正式 TaskRun。
```

### 3.3 Prompts 库只存稳定 prompt，不做决策

Prompts 库职责：

- 存储 prompt resource。
- 管理类别、版本、适用范围、缓存属性。
- 根据 RuntimeAssembly 已给出的 refs 或显式 selector context 返回 prompt sections。
- 生成 prompt manifest。

Prompts 库不负责：

- 判断用户是否要开启任务。
- 从自然语言猜 `task_goal_type`。
- 重新决定工具权限。
- 改写 task contract。
- 根据旧链路补默认任务类型。

### 3.4 Agent Prompt 必须写给 agent，而不是写给开发者

禁止：

```text
这是 runtime 节点。
根据任务图执行 world_review。
```

应该：

```text
你是一名世界观审核员。
你只负责评审当前世界观设定是否完整、一致、可支撑后续写作。
你需要指出问题、给出裁决、说明是否允许进入下一阶段。
```

## 4. Prompts Library 目标分类

### 4.1 runtime

稳定运行时 prompt pack。

子类：

- `runtime.turn_action`
- `runtime.task_execution`
- `runtime.observation_followup`
- `runtime.action_schema`
- `runtime.failure_recovery`
- `runtime.completion_self_review`

用途：

- 定义模型动作协议。
- 定义 TaskRun 执行职责。
- 定义观察回灌后的处理规则。
- 定义完成前自检要求。

不得包含：

- 当前 observations。
- 当前 task_run status。
- 当前工具失败记录。
- mode 名称解释。

### 4.2 agent

Agent 职责 prompt。

子类：

- `agent.main.work_role`
- `agent.worker.role`
- `agent.verifier.role`
- `agent.memory.role`
- `agent.search.role`
- `agent.graph_node_executor.role`

来源：

- 当前 `agent_runtime_profiles.json.metadata.work_role_prompt`
- worker agent blueprint
- verifier / memory / searcher agent 配置

目标：

- agent profile 不再内联大段 prompt。
- agent profile 只引用 `prompt_resource_id`。

### 4.3 environment

任务环境 prompt。

子类：

- `environment.development.sandbox`
- `environment.development.readonly`
- `environment.creation.writing`
- `environment.general.workspace`
- `environment.research.web`
- `environment.document.processing`

来源：

- 当前 `default_environments.py` 中的 `EnvironmentPrompt`
- 用户自定义 task environment config

目标：

- Environment definition 引用 `prompt_resource_id`。
- 环境 prompt 只描述资源边界、存储边界、沙盒约束、验证通道。
- 环境不授予工具，不装配 skills，不写 agent 工作方法训练。

### 4.4 task

特定任务契约 prompt。

子类：

- `task.specific.role`
- `task.specific.instruction`
- `task.specific.output`
- `task.specific.done`
- `task.specific.forbidden`

用途：

- 已成型的特定任务合同。
- 类似“雇佣兵任务”的启动协议。
- 可选择启动 TaskRun，也可作为简单约束进入普通 turn。

关键：

- 特定任务不是意图识别结果。
- 特定任务必须由显式契约、用户选择、图任务节点或系统注册计划下发。

### 4.5 graph_node

图任务节点 prompt。

子类：

- `graph_node.role`
- `graph_node.task_instruction`
- `graph_node.output_instruction`
- `graph_node.definition_of_done`
- `graph_node.forbidden_behavior`

来源：

- `GraphHarnessConfig.nodes[*].prompt`
- 旧 `storage/prompt_library/prompt_resources.json` 中的 stage_role

目标：

- 图编辑器编辑 prompt contract。
- publisher 输出标准 prompt refs 或内联 prompt contract。
- context materializer 将 prompt contract 精确放入 work order。

### 4.6 skill

技能 prompt。

子类：

- `skill.usage`
- `skill.boundary`
- `skill.output`
- `skill.examples`

来源：

- `SKILL.md`
- capability units 中的 skill metadata

边界：

- skill prompt 属于 agent 能力装配。
- environment 不装配 skills。
- skill prompt 可以进入 prompts 库索引，但是否可见由 agent/profile/capability 装配决定。

### 4.7 soul

角色表达 prompt。

子类：

- `soul.role_persona`
- `soul.expression_style`
- `soul.guardrails`

边界：

- 只允许 role mode 装配。
- 不改变工具权限。
- 不改变 TaskRun 生命周期。
- 不改变输出契约。
- 不覆盖 system/runtime/task 边界。

## 5. 数据模型设计

### 5.1 PromptResource

目标字段：

```json
{
  "prompt_id": "runtime.task_execution.v1",
  "category": "runtime",
  "subtype": "task_execution",
  "version": "v1",
  "title": "TaskRun execution prompt",
  "content": "...",
  "locale": "zh-CN",
  "owner_layer": "runtime",
  "cache_scope": "static|session|task|dynamic",
  "model_visible": true,
  "allowed_invocation_kinds": ["task_execution"],
  "allowed_runtime_modes": [],
  "allowed_agent_refs": [],
  "allowed_environment_refs": [],
  "source_ref": "...",
  "status": "active|draft|deprecated",
  "metadata": {}
}
```

说明：

- `category/subtype` 替代旧 `resource_type` 的泛化混用。
- `allowed_runtime_modes` 只能作为装配约束，不能生成 mode 文案。
- `cache_scope` 必须明确，服务 prompt cache。
- `status=deprecated` 的资源不能进入新主链。

### 5.2 PromptPack

用于 runtime 固定调用场景。

```json
{
  "pack_id": "runtime.pack.task_execution.v1",
  "invocation_kind": "task_execution",
  "ordered_prompt_refs": [
    "runtime.action_schema.v1",
    "runtime.task_execution.v1",
    "runtime.failure_recovery.v1",
    "runtime.completion_self_review.v1"
  ],
  "cache_scope": "static",
  "authority": "prompts.prompt_pack"
}
```

### 5.3 PromptAssemblyRequest

由 RuntimeAssembly 发起。

```json
{
  "invocation_kind": "turn_action",
  "agent_profile_ref": "main_interactive_agent",
  "task_environment_ref": "env.development.sandbox",
  "runtime_mode": "professional",
  "prompt_pack_refs": ["runtime.pack.turn_action.v1"],
  "agent_prompt_refs": ["agent.main.work_role.v1"],
  "environment_prompt_refs": ["environment.development.sandbox.v1"],
  "task_prompt_contract": {},
  "graph_node_prompt_contract": {},
  "skill_prompt_refs": [],
  "soul_prompt_ref": ""
}
```

注意：

- `runtime_mode` 是选择约束，不是模型可见 prompt 文案。
- 动态 runtime projection 不放在 request 中，由 RuntimeCompiler 或 ProjectionBuilder 生成。

### 5.4 PromptAssemblyResult

```json
{
  "assembly_id": "promptasm:...",
  "stable_sections": [
    {
      "section_id": "runtime.task_execution.v1",
      "category": "runtime",
      "cache_scope": "static",
      "content": "..."
    }
  ],
  "task_sections": [],
  "agent_sections": [],
  "environment_sections": [],
  "skill_sections": [],
  "soul_sections": [],
  "manifest": {}
}
```

### 5.5 RuntimePromptManifest

每次 model packet 都要有 manifest。

```json
{
  "manifest_id": "rtprompt:...",
  "packet_id": "rtpacket:...",
  "invocation_kind": "task_execution",
  "stable_prompt_refs": [],
  "dynamic_projection_refs": [],
  "volatile_state_refs": [],
  "cache_boundary": {
    "static_prefix_end": 3,
    "session_stable_end": 5
  },
  "token_estimate": {},
  "authority": "runtime.prompt_manifest"
}
```

## 6. 固定装配流程

### 6.1 普通 turn

```text
User message
-> QueryRuntime
-> assemble_runtime
   - resolve agent profile
   - resolve mode as runtime policy
   - resolve task environment
   - authorize visible tools
   - collect prompt refs
-> PromptsLibrary.assemble
   - runtime pack: turn_action
   - agent work role
   - environment prompt
   - soul prompt only if role mode
-> RuntimeProjectionBuilder
   - allowed actions
   - task lifecycle boundary
   - tool/permission/environment boundary
-> RuntimeCompiler
   - system/static prompt sections
   - stable contract JSON
   - volatile user/history state
-> Model action decision
```

### 6.2 TaskRun execution

```text
TaskRunContract
-> assemble_runtime
-> PromptsLibrary.assemble
   - runtime pack: task_execution
   - agent work role
   - environment prompt
   - task prompt contract
   - skill prompts if agent selected/authorized
-> RuntimeProjectionBuilder
   - no request_task_run
   - allowed tools
   - artifact/verification/self-review boundary
-> RuntimeCompiler
   - stable contract before volatile observations
-> ExecutionLoop
```

### 6.3 Observation followup

```text
ObservationRecord
-> PromptsLibrary.assemble(runtime.pack.observation_followup)
-> RuntimeProjectionBuilder
-> RuntimeCompiler
   - stable instructions unchanged
   - current observations volatile
-> Model repairs / continues / blocks / responds
```

### 6.4 Graph node

```text
GraphHarnessConfig node.prompt
-> GraphContextMaterializer input_package.prompt_contract
-> WorkOrder
-> TaskRunContract.prompt_contract
-> PromptsLibrary.assemble(task/graph_node sections)
-> RuntimeCompiler task_execution packet
```

图节点 prompt 不经过普通 turn 意图识别。

## 7. 旧链路处理

### 7.1 `backend/prompting/*`

当前问题：

- 旧 static prompt 仍被 `query.runtime.build_static_system_prompt_for_session()` 使用。
- 旧 prompt cache 与新 prompt accounting 并存。

目标：

- 不再作为新 harness prompt 来源。
- `/sessions/{id}/messages` 和 `/tokens/session/{id}` 改为读取 RuntimePromptManifest 或 prompt accounting ledger。
- 迁出仍有价值的 CORE/common contract 文本为 `system/common` 类 prompt resource。

### 7.2 `backend/prompt_library/*`

保留基础能力：

- resource model。
- registry。
- manifest validation。
- runtime sections 的可复用结构。

必须移除/重写：

- 普通 turn 相关的 `task_goal_type` 选择逻辑。
- `PromptSelectionContext` 中旧意图字段作为主选择依据。
- `assembler.py` 里生成“你本轮要完成的任务类型是...”这类旧任务类型 prompt。

改造方向：

- 从 `prompt_library` 升级为 `prompts` 或保留目录名但更新 authority。
- selector 改为“显式 refs + 明确 contract fields”，不做语义猜测。

### 7.3 `storage/prompt_library/prompt_resources.json`

当前主要是写作图任务 stage_role prompt。

迁移：

- `stage_role` -> `graph_node.role`
- 增加 `category=graph_node`
- 绑定 `graph_id/node_id`
- 移除 `applies_to_task_goal_types` 作为主链选择依据
- 保留 source_ref 便于审计

## 8. 缓存与 Token 策略

Prompt cache 分层：

```text
global static:
  runtime prompt pack
  system/common contract
  stable agent role prompt

environment stable:
  environment prompt
  environment storage/boundary static description

task stable:
  task prompt contract
  graph node role/task/output/done

session/task dynamic:
  runtime projection
  operation authorization summary

volatile:
  user message
  observations
  tool results
  active failures
  task_run state
```

原则：

- 大段稳定 prompt 必须引用 resource id 和 version。
- RuntimeProjection 可以进入 stable JSON，但不进入 Prompts Library。
- observations 和 execution_state 必须保持 volatile。
- tool schema 不应重复塞入自然语言 prompt 和 provider tool schema 两边。

## 9. 验证标准

### 9.1 静态验证

- 新单 agent 主链不 import `prompting.build_static_prompt`。
- RuntimeCompiler 不再硬编码大段长期 prompt。
- PromptResource 必须有 category/subtype/version/cache_scope。
- Prompts 库不根据 task_goal_type 猜普通 turn。
- mode 不生成模型可见 mode instruction。

### 9.2 行为验证

- 普通闲聊不启动 TaskRun。
- 需要真实产物的任务由 agent 请求 TaskRun。
- TaskRun 内不允许再次 request_task_run。
- 图节点 prompt contract 能进入 TaskRunContract。
- role mode 只装配 soul role prompt，不改变工具/任务权限。
- environment prompt 只声明资源边界，不授予工具。

### 9.3 审计验证

每个 model packet 都能回答：

- 用了哪些 prompt resources？
- 哪些是 static/cacheable？
- 哪些是 task stable？
- 哪些是 runtime projection？
- 哪些是 volatile observations？
- 为什么某个 prompt 被装配？
- 哪些 prompt 被拒绝装配，原因是什么？

## 11. 最终目标

最终 Prompts 系统应具备：

- 分类清晰：runtime / agent / environment / task / graph_node / skill / soul。
- 权威单一：prompt 只提供稳定语义资产，runtime 装配决定是否使用。
- 可审计：每次调用都有 manifest。
- 可缓存：稳定 prompt resource 与 volatile runtime state 分离。
- 可扩展：自定义环境、自定义 agent、自定义特定任务、图任务节点都能通过统一 prompt resource 接入。
- 不回退旧结构：不再用 task_goal_type、旧 selector、旧 static prompt 作为普通 turn 主链权威。

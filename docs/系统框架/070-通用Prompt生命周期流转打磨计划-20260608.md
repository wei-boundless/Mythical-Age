# 通用 Prompt 生命周期流转打磨计划

日期：2026-06-08

状态：第一轮已实施。本文基于当前 prompt_library、EnvironmentPromptController、RuntimeCompiler 和已有治理文档重新从头梳理通用 prompt 体系；2026-06-08 已完成 foundation / agent role / runtime protocol / general lifecycle 的第一轮主链路打磨。

## 1. 总结

当前系统已经有可用的 prompt 资源库、人格层、环境基座、生命周期 selector、工具 guidance、worker prompt 和 prompt manifest。但通用 prompt 体系还没有完全达到成熟 agent 的顺滑流转标准。

核心问题不是“缺几句提醒”，而是几类权威仍然混在一起：

- System foundation 仍以 `vibe_coding_agent` 命名，并把 coding 行为写成最高稳定规则。
- Agent role 仍硬编码 Mythical Age（洪荒智能），且包含大量工具、文件、terminal、子 agent、todo 细节。
- Runtime protocol 本应只定义动作协议，但仍带部分任务判断和状态解释。
- Lifecycle prompt 已经按状态挂载，但覆盖的生命周期还不够完整。
- Tool guidance 已存在，但和 agent role / runtime rule 有重复。
- Memory prompt 已分角色，但 memory read、memory write、compaction、session emphasis 的模型可见边界还可以更清楚。
- Environment prompt 是边界，不应承担任务选择；后续可以有 environment style overlay，但不能自动切换主人格。

目标是把通用 prompt 体系打磨成一条自然、稳定、可审计的主 agent 生命周期：

```text
stable system foundation
-> agent role
-> selected personality
-> runtime protocol
-> general environment base
-> selected environment overlay
-> lifecycle selector
-> tool guidance
-> project instructions
-> dynamic runtime projection
-> volatile current request / observations
```

## 2. 当前实际装配路径

### 2.1 Single Agent Turn

当前主要文件：

- `backend/harness/runtime/assembly.py`
- `backend/harness/runtime/environment_prompt_controller.py`
- `backend/harness/runtime/personality_prompt_controller.py`
- `backend/harness/runtime/compiler.py`
- `backend/prompt_library/packs.py`
- `backend/prompt_library/agent_prompts.py`
- `backend/prompt_library/general_lifecycle_prompts.py`

当前模型输入顺序大致是：

```text
runtime prompt pack / system foundation
-> stable boundary payload
-> personality_stable
-> environment + lifecycle + agent role + skill candidates
-> session history / provider transcript
-> dynamic runtime projection
-> current user request
```

优点：

- 当前用户请求已经作为 volatile_user 单独放在末尾。
- personality 已经独立为 `personality_stable`。
- environment lifecycle 已经由 selector 挂载，不再全量常驻。

问题：

- `system.foundation.vibe_coding_agent.v1` 是最高稳定层，但名字和内容偏 coding。
- `agent.main_interactive_agent.single_agent_turn.work_role` 仍自称 Mythical Age，和 personality 层重复。
- single-turn agent role 同时管请求判断、任务 handoff、搜索域判断、子 agent brief、工具失败、审批、最终自审，职责偏大。
- runtime protocol 仍写了持续任务 handoff、active work control、工具请求，和 lifecycle prompt 有交叉。

### 2.2 Task Execution

当前模型输入顺序大致是：

```text
runtime prompt pack / system foundation
-> task execution action schema
-> environment + lifecycle + environment stable payload
-> personality_stable
-> agent_stable
-> agent function shared stable
-> artifact scope
-> tool index / tool guidance
-> graph shared context
-> active skills
-> task contract
-> task prompt contract
-> dynamic runtime boundary
-> volatile task state
-> user steering updates
```

优点：

- task execution 已经明确围绕任务合同推进。
- action schema、environment、agent、tool index、task contract 分段清楚。
- user steering updates 是 volatile，不污染稳定前缀。

问题：

- agent role 中重复了 read_file、edit_file、terminal、todo、subagent、verification 等细节。
- runtime protocol 中也解释 task_state 字段，容易和 dynamic projection 的真实 schema 耦合。
- task execution lifecycle 缺少显式 verification gate 和 completion readiness prompt。
- pending_user_steers 只挂 `user_steer_contract_revision`，但没有更细的“当前轮是否应先响应用户 steering 再继续工具”的判断提示。

### 2.3 Tool Observation Followup

当前模型输入顺序大致是：

```text
runtime prompt pack / system foundation
-> observation followup stable contract
-> environment + lifecycle
-> personality_stable
-> agent_stable
-> skill candidates
-> session history / provider transcript
-> dynamic runtime projection
-> observation followup current request
```

优点：

- observation followup 已经有专门 invocation。
- `tool_observation_recovery` 会在此阶段挂载。

问题：

- observation followup agent role 偏短，只说继续推进，没有覆盖“观察后收口、二次观察、升级任务、阻塞、回滚判断、用户汇报”的完整分支。
- runtime observation followup prompt 仍自称 Mythical Age，应交给 personality。
- 对多工具观察、部分失败、部分成功、输出省略、persisted result 的处理分散在 runtime rule / tool guidance / dynamic projection 中，需要统一生命周期判断。

## 3. 成熟通用生命周期目标

通用生命周期应覆盖以下状态。每个状态都应有明确触发、模型职责、禁止事项和输出关系。

| 生命周期 | 触发 | 模型职责 | 不应做 |
| --- | --- | --- | --- |
| context_intake | 每个 single turn / followup / task execution | 分清当前用户、系统观察、active work、task state、memory、todo、history 的权威 | 把旧摘要或记忆升级成当前事实 |
| request_judgment | single turn | 判断用户真正要回答、观察、执行、控制当前工作、建立任务还是阻塞 | 关键词分类、环境替用户判断任务 |
| work_relation | active work/recent outcome 可见时 | 判断用户是否指向当前工作、历史断点、独立新请求或状态查询 | 让 current work 劫持新请求 |
| environment_alignment | 每轮 | 对齐目标和环境、工具、权限、文件边界、网络/浏览器能力 | 要用户重复批准已授权能力 |
| plan_gate | 高影响或计划模式可见时 | 判断是否需要先计划、等待确认、锁定实施范围 | 在应计划时直接改代码或运行副作用 |
| action_selection | 每轮 | 选择一个最小充分 action | 混合回答、工具、任务开启和控制动作 |
| task_handoff | request_task_run 允许且需要持续执行 | 形成用户可懂的任务合同 seed | 在 handoff 阶段伪造执行结果 |
| tool_dispatch | tool_call 之前 | 确认工具服务于当前步骤，参数和边界清楚 | 调用不可见工具或重复失败动作 |
| observation_recovery | 工具观察后 | 处理成功、失败、拒绝、超时、省略、部分成功 | 伪报成功或忽略失败 |
| subagent_delegation | 需要隔离搜索/验证/研究时 | 写完整 brief，划分 scope，等待结果 | 把理解用户请求外包给子 agent |
| subagent_result_integration | wait_subagent 返回后 | 综合证据、冲突、限制和下一步 | 把子 agent 结论自动当最终答案 |
| user_steer_revision | 用户补充或质疑当前任务时 | 判断补充、状态查询、合同修订、独立请求 | 静默覆盖原合同 |
| verification_gate | 准备完成前 | 判断交付物、测试、构建、浏览器/命令验证是否足够 | 把阅读代码或计划当验证 |
| finalization | respond 前 | 如实报告完成、验证、风险、未完成项 | 夸大完成度或暴露内部协议 |
| memory_handoff | 收口后或维护阶段 | 区分回复、task state、session summary、durable memory | 把临时状态写长期记忆 |
| compaction_handoff | 上下文压力或恢复点生成 | 保留可恢复事实，丢弃噪声 | 加入新事实或继续执行任务 |

## 4. 现有 prompt 分类调整建议

### 4.1 System Foundation

目标：

- 从 `vibe_coding_agent` 改为通用稳定 foundation。
- 不再固定 Mythical Age，也不绑定 coding。
- 只保留所有环境都成立的行为宪法。

建议拆成：

```text
system.foundation.local_collaboration
system.foundation.current_request_authority
system.foundation.truth_and_verification
system.foundation.security_and_injection
system.foundation.context_memory_cache
system.foundation.user_change_protection
```

应移出 foundation 的内容：

- coding 文件调用链搜索；
- edit_file / terminal / git 细节；
- 具体任务环境质量规则；
- 人格名称和表达风格。

### 4.2 Agent Role

目标：

- Agent role 只定义当前 invocation 中的职责和裁决权。
- 不硬编码人格名。
- 不重复工具 guidance。

建议：

```text
agent.role.primary.single_turn
agent.role.primary.task_execution
agent.role.primary.observation_followup
agent.role.context_compactor.semantic_compaction
agent.role.memory_manager.maintenance
```

Single turn agent role 应聚焦：

- 理解当前请求；
- 维护最终裁决权；
- 判断是否直接回答、询问、观察、任务 handoff、active work control 或 block；
- 需要子 agent 时写 brief，但不展开每类工具细节。

Task execution agent role 应聚焦：

- 按合同推进；
- 处理 task_state / user steers；
- 调度工具；
- 验证和收口；
- 不重新判断是否建立任务生命周期。

Observation followup agent role 应补强：

- 多观察结果整合；
- 部分成功/失败；
- 是否需要下一次观察；
- 是否升级 task run；
- 是否可直接最终答复；
- 是否必须报告阻塞。

### 4.3 Runtime Protocol

目标：

- 只定义动作协议和输出结构。
- 不承载生命周期判断。
- 不承载人格。

建议：

- `runtime.single_agent_turn.v1` 删除 “你是 Mythical Age”。
- `runtime.observation_followup.v1` 删除 “你是 Mythical Age”。
- task_state 字段说明应转移到 dynamic runtime projection 或 task_state payload schema。
- request_task_run、active_work_control 的语义可以保留最小 schema 指引，复杂判断交给 lifecycle。

### 4.4 Lifecycle Prompt

当前 10 个生命周期 prompt 是正确基础，但应扩展为更完整的通用生命周期包。

建议新增：

```text
environment.general.lifecycle.work_relation
environment.general.lifecycle.plan_gate
environment.general.lifecycle.tool_dispatch
environment.general.lifecycle.subagent_delegation
environment.general.lifecycle.subagent_result_integration
environment.general.lifecycle.verification_gate
environment.general.lifecycle.compaction_handoff
environment.general.lifecycle.project_instruction_conflict
```

建议拆分：

- `memory_state_handoff` 拆成 `memory_read_context` 和 `memory_write_handoff`。
- `action_selection` 保持最终裁决，但不再承担 tool_dispatch 和 verification_gate。

### 4.5 Tool Guidance

目标：

- 工具 prompt 承担工具契约，不让 agent role 重复工具细节。
- 每个工具要说明何时用、何时不用、失败后怎么恢复、结果意味着什么。

优先打磨：

- `read_file`：行窗口、截断、persisted result、搜索片段不能编辑。
- `edit_file`：old_text 唯一性、失败后重读、避免整文件替换。
- `write_file`：新文件/完整重写边界、覆盖用户改动保护。
- `terminal`：PowerShell、服务启动/停止、长进程观察、非交互。
- `git_read/git_write`：读证据与写状态分离。
- `subagent`：brief/result envelope 和父 agent 最终裁决权。
- `web_fetch`：官方来源、时间敏感、法律医疗金融风险、引用边界。
- `browser`：真实页面状态、截图/console/network 证据、固定端口。

### 4.6 Personality 与 Environment Style Overlay

推荐结构：

```text
stable personality
-> environment style overlay
```

人格由用户/session 控制。任务环境可以提供风格 overlay，但不能自动替换主人格。

示例：

```text
personality.default.mythical_age
environment.style.coding
environment.style.writing
environment.style.research
```

style overlay 只能影响表达密度、证据要求、审美倾向和交互节奏，不能改变权限、工具、记忆和任务合同。

## 5. Lifecycle Selector 调整建议

当前 selector：

- single_agent_turn：context_intake、request_judgment、environment_alignment、action_selection、可选 active_work、可选 task_run_handoff、可选 memory、finalization。
- tool_observation_followup：context_intake、environment_alignment、action_selection、tool_observation_recovery、可选 memory、finalization。
- task_execution：context_intake、environment_alignment、action_selection、可选 tool_observation_recovery、可选 user_steer、可选 memory、finalization。

建议目标：

### Single Turn

```text
context_intake
request_judgment
work_relation            # active/recent work 可见时
environment_alignment
plan_gate                # plan mode / high impact / requires_plan 可见时
action_selection
task_run_handoff         # request_task_run allowed
tool_dispatch            # tool_call allowed
subagent_delegation      # subagent tools visible
memory_read_context      # memory visible
finalization
```

### Tool Observation Followup

```text
context_intake
environment_alignment
observation_recovery
subagent_result_integration # wait_subagent observation 可见时
tool_dispatch               # 仍可继续工具
task_run_handoff            # 可升级持续任务
memory_read_context
finalization
```

### Task Execution

```text
context_intake
environment_alignment
user_steer_revision       # pending user steers 可见时
observation_recovery      # observations 可见时
subagent_result_integration
tool_dispatch
verification_gate
memory_read_context
memory_write_handoff      # 只在收口或维护阶段
finalization
```

## 6. 测试设计

新增或调整测试：

1. Foundation 不含人格名、不含 coding 专属规则。
2. Agent role 不含 Mythical Age；人格只出现在 personality prompt。
3. Runtime protocol 不含 Mythical Age，不重复 lifecycle 长判断。
4. Single turn 无 active work 时不挂 work_relation/active_work_control。
5. 有 active work 时挂 work_relation 和 active_work_control。
6. 有 tool observation 时挂 observation_recovery。
7. wait_subagent observation 时挂 subagent_result_integration。
8. task execution 准备 respond 时必须可见 verification_gate。
9. memory context 可见时挂 memory_read_context；只有维护/收口阶段才挂 memory_write_handoff。
10. Prompt manifest 能报告 foundation、agent、personality、runtime、environment、lifecycle、tool、project、dynamic、volatile 的来源和顺序。

## 8. 需要确认的边界

这轮改动通用 prompt 主链路，影响 single turn、task execution、observation followup 三类模型输入。用户已要求继续打磨，第一轮已按本计划实施。

建议先执行 Phase 1 到 Phase 3：

- 先把稳定 foundation、agent role、runtime protocol 的权威边界收干净；
- 暂不一次性新增全部 lifecycle prompt；
- 确保现有行为不漂移，只清理人格/环境/协议混杂。

Phase 4 到 Phase 6 再做生命周期扩展、工具 guidance 深打磨和 linter 硬化。

已完成：

仍需继续：

- Runtime prompt ID、tool prompt ID、worker prompt ID 仍有旧 `.v1`，后续应制定单独迁移批次，避免一次性改动过大。
- Prompt governance 不应做具体人格名、字段名或历史残留词的硬拒绝；已经清除的历史残留应由回归测试和 manifest 审计证明“不再装配”。
- 后续 linter 应只检查结构性问题：开发口吻 prompt、owner_layer/category/cache_scope 不匹配、prompt_rule scope 不匹配、runtime protocol 多重装配、显式 metadata 权威冲突等。
- Environment prompt 的任务选择禁令可以进一步做结构化 metadata/manifest 审计；如果做语义 linter，需要先整理现有环境文案，避免误伤“环境不是任务选择器”这类否定式治理语句。

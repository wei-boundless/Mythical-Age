## 2. 问题一：RAG 首问检索成功，但没有生成可展示答案

### 2.1 现象

场景：

- `research-brief-and-document-resume`
- turn 3

用户问题：

- `基于本地知识库，先用业务语言告诉我 AI 治理里最常见的三类风险。`

实际结果：

- `plan.route = rag`
- 发生了 `retrieval`
- 最终回答为：
  `已检索到相关资料，但当前模型尚未产出可直接展示的结论。`

### 2.2 直接证据

文件：

- [run_result.json](D:/AI应用/langchain-agent/output/test_runs/20260423-230547-long/run_result.json)
- [turn-03-main.json](D:/AI应用/langchain-agent/output/test_runs/20260423-230547-long/artifacts/research-brief-and-document-resume/turn-03-main.json)

关键信号：

- `answer_source = fallback_policy`
- `answer_fallback_reason = rag_missing_answer`
- `answer_leak_flags = ["internal_protocol_stream", "internal_protocol_ai_update"]`

### 2.3 责任边界定位

不是检索没有命中。

从 `turn-03-main.json` 可见：

- 已返回多条 retrieval evidence
- rerank 也已生效

因此责任不在检索链，而在：

1. `query.model_stream`
2. `output_boundary`
3. `output_classifier`

### 2.4 初步根因

代码位置：

- [output_boundary.py](D:/AI应用/langchain-agent/backend/query/output_boundary.py)
- [output_classifier.py](D:/AI应用/langchain-agent/backend/query/output_classifier.py)

判断：

1. 模型流里出现了内部协议/过程化文本。
2. `AssistantOutputBoundary` 记录了 leak flag。
3. `build_output_decision(...)` 未能拿到稳定 answer candidate。
4. 最终走了 `build_route_fallback(... route='rag' ...)`，落成 `rag_missing_answer`。

也就是说：

`问题不在“检索不到”，而在“检到了，但没有把 evidence 收成最终答案”。`

## 3. 问题二：memory 改写轮被过程化文本污染，并且污染内容直接进入可见答案

### 3.1 现象

场景：

- `research-brief-and-document-resume`
- turn 4

用户问题：

- `把刚才那三类风险压成适合管理层汇报的三条。`

实际结果：

- `plan.route = memory`
- `runtime_effective_route = memory`
- 输出变成大量重复：
  `让我检查知识库目录结构`

### 3.2 直接证据

文件：

- [turn-04-main.json](D:/AI应用/langchain-agent/output/test_runs/20260423-230547-long/artifacts/research-brief-and-document-resume/turn-04-main.json)
- [local trace](D:/AI应用/langchain-agent/output/local_traces/20260423/local-20260423-230632-f700b1233a5343e8-efe5c330.json)

关键信号：

- `answer_source = segment.visible_text`
- `answer_fallback_reason = ""`
- `answer_leak_flags = ["internal_protocol_stream", "internal_protocol_ai_update"]`

而 token 流里明确出现了：

- `我来检索本地知识库`
- `search`
- `让我检查知识库目录结构`

### 3.3 责任边界定位

这不是 follow-up 劫持，也不是 planner 错路由。

planner 已经把它判成：

- `route = memory`

所以问题落在：

1. `memory route` 下的模型输出控制
2. `output_boundary` 的可见文本清洗
3. `output_decision` 的候选选择

### 3.4 初步根因

代码位置：

- [output_boundary.py](D:/AI应用/langchain-agent/backend/query/output_boundary.py)

关键点：

1. `sanitize_visible_assistant_content(...)` 只会剔除一部分内部协议。
2. `_trim_procedural_edges(...)` 只在“边缘”裁掉过程化句子。
3. 当整段输出几乎全是过程化/伪操作文本时，没有真正的内容段，函数会把整段保留下来。
4. `build_response(...)` 虽然保留了 `leak_flags`，但仍允许 `segment.visible_text` 成为最终答案源。

也就是说：

`当前 output boundary 能识别“脏”，但还不能阻止“脏内容”被当作答案。`

## 5. 问题四：多源运营摘要被误判成天气工具调用

### 5.1 现象

场景：

- `commerce-ops-data-live-switch`
- turn 9

用户问题：

- `把库存、员工、黄金和天气这四块信息分开给我一个运营摘要。`

实际结果：

- `plan.route = tool`
- `plan.tool = get_weather`
- 最终执行了天气工具
- 回答只剩：
  `工具 get_weather 已执行，但当前结果尚未形成可直接展示的答案。`

### 5.2 直接证据

文件：

- [turn-09-main.json](D:/AI应用/langchain-agent/output/test_runs/20260423-230547-long/artifacts/commerce-ops-data-live-switch/turn-09-main.json)

关键信号：

- `tool_start.tool = get_weather`
- `tool_input.query = 原问题全文`
- `main_context.active_constraints.active_location = 原问题全文`
- `answer_fallback_reason = tool_missing_summary`

### 5.3 责任边界定位

这是明确的路由误判，不是输出层问题。

责任落在：

1. `task_understanding`
2. `continuation_resolver.promote_session_summary_query(...)`

### 5.4 初步根因

代码位置：

- [task_understanding.py](D:/AI应用/langchain-agent/backend/understanding/task_understanding.py)
- [continuation_resolver.py](D:/AI应用/langchain-agent/backend/query/continuation_resolver.py)

根因链：

1. `_detect_realtime_task(...)` 只要命中 `天气` 就优先判成 weather tool。
2. `analyze_task_understanding(...)` 的优先级里，realtime 判断先于 knowledge / structured / fallback。
3. `promote_session_summary_query(...)` 的 `_looks_like_session_summary(...)` 只识别：
   - `总结 / 回顾 / 归纳 / 梳理 / 整理 / 概括 / 分成 / 拆成`
4. 当前用户说的是：
   - `运营摘要`
   - `分开`
   - `库存、员工、黄金和天气四块信息`

这句话没有命中 session summary 规则，但命中了 `天气`。

所以这是：

`多源总结请求被 realtime keyword 抢先路由成单工具调用。`

## 6. 问题五：PDF “第二部分”追问没有 owner binding，直接退化成裸 PDF 工具调用

### 6.1 现象

场景：

- `commerce-ops-data-live-switch`
- turn 10

用户问题：

- `回到刚才那份 PDF，第二部分强调的约束是什么？`

实际结果：

- `plan.tool = pdf_analysis`
- 但 `tool_input` 只有 `mode = section`
- 没有 `path`
- 最后返回：
  `已读取与当前问题最相关的 PDF 页面：P21、P2、P3，但当前还没有形成稳定摘要。`

### 6.2 直接证据

文件：

- [turn-10-main.json](D:/AI应用/langchain-agent/output/test_runs/20260423-230547-long/artifacts/commerce-ops-data-live-switch/turn-10-main.json)

关键信号：

- `tool_start.input = { query, mode: section }`
- 没有 `path`
- `followup_mode = task_ref`
- `context_ref.bindings.active_pdf = ""`
- `active_binding_identity = ""`

### 6.3 责任边界定位

这不是当前 follow-up 修复回退了，而是“根本没有可用 owner”。

在这个 scenario 里，前面没有任何 PDF 打开动作，只有：

- inventory
- employees
- gold
- weather

因此：

1. `followup_resolver` 不可能正确恢复 PDF owner
2. `binding follow-up` 也不会触发
3. planner 只能根据 `PDF + 第二部分` 把它判成普通 `pdf_analysis`

### 6.4 初步根因

这是一个“悬空指代”问题：

- 用户说了 `回到刚才那份 PDF`
- 但当前 session 中不存在可恢复的 PDF 绑定

于是系统退化成：

- `裸 pdf tool query`

再加上没有 `path`，PDF 工具只能依赖内部默认行为或弱解析，最终导致：

- 命中了一组页面
- 但没有稳定摘要

因此这个问题的主根因不是本轮 follow-up 仲裁，而是：

`悬空对象引用 + 缺少 owner binding + 裸工具退化执行`

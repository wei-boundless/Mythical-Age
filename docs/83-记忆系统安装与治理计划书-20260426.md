# 记忆系统安装与治理计划书

日期：2026-04-26

## 1. 当前目标

记忆系统不应该只是一组文件，也不应该只是一张好看的图。它需要同时满足两件事：

- 运行时可以稳定调用它，作为状态记忆、长期记忆和 prompt 注入来源。
- 前端可以直观看到它，让我们能判断一轮回答到底用了哪些记忆、为什么召回、是否应该注入。

所以本项目采用“双层记忆系统”：后端作为可调用记录层，前端作为可视化分析层。

## 2. 第一阶段已安装内容

本阶段只做只读安装，不开放写入，避免把脏记忆写进长期层。

已接入：

- `GET /api/memory/overview`
  读取长期记忆概览、状态记忆现场、模型可见上下文和上下文压力信息。

- `POST /api/memory/recall-preview`
  输入一句问题，模拟这一轮会召回哪些长期记忆，并返回召回理由、选中记录和注入摘要。

- `GET /api/memory/durable/{filename}`
  读取单条长期记忆的安全预览。

- 前端“记忆系统”工作台
  展示状态记忆、长期记忆、召回模拟、模型可见上下文和可注入记录。

- 对话现场展开
  记忆系统页面可以按轮次展开当前会话的用户输入和助手回应，并可把某一轮内容直接放入召回模拟，用来检查“这句话会触发哪些记忆”。

- 单轮记忆链路
  在“对话现场”展开某一轮后，可以点击“查看这一轮记忆链路”。前端会用这一轮文本发起召回模拟，并在同一页面展示 `输入 -> 状态记忆 -> 长期召回 -> Prompt 注入` 四段链路。这个版本属于普通会话的模拟链路，不会改写真实记忆。

## 3. 当前边界

当前版本明确不做这些事：

- 不从前端直接写入长期记忆。
- 不允许前端传任意文件路径读取记忆。
- 不默认展示完整长期记忆内容，只展示 preview 和元数据。
- 不把召回模拟结果自动写入当前会话。

这样可以先把观察和 debug 做稳，再进入治理阶段。

## 4. 后续阶段

### 第二阶段：召回链路可视化

目标：把记忆召回接到系统框架图和测试系统。

当前已推进到第一版可用：

- 新增 `GET /api/experiments/runs/{run_id}/turns/{turn_id}/memory-trace`。
- 测试系统 turn 列表会标记 `Memory 已记录 / Memory 缺失`。
- 系统框架图中的“记忆门面”节点会在有 memory trace 时显示徽标。
- 点击某个 turn 进入系统框架图后，可以展开本轮记忆链路，查看状态记忆、长期召回、Prompt 注入摘要。

要做到：

- 测试 turn 中显示状态记忆、长期记忆、prompt manifest 的联动关系。已完成第一版。
- 系统框架图点击“记忆门面”时，能显示该轮 exact match、relevant match、是否注入 prompt。已完成第一版。
- 召回模拟结果可以和 Prompt Manifest 的 turn 层对齐。下一步继续细化。
- 将 memory trace 与节点耗时、失败原因、follow-up 解析结果进一步合并。待做。
- 普通会话页已补充单轮模拟链路，下一步可以把测试系统里的真实 memory trace 复用到同一面板中，形成“真实链路 / 模拟链路”对照。

### 第三阶段：候选写入队列

目标：把“可能应该记住的内容”先放进待审队列。

要做到：

- 后端生成 memory candidate。
- 前端显示候选内容、来源消息、推荐类型、推荐原因。
- 用户确认后才 commit。
- 支持忽略、延后、合并、替换旧记忆。

### 第四阶段：长期记忆治理

目标：让长期记忆可以被维护，而不是越积越乱。

要做到：

- 支持停用、归档、恢复、替换、合并。
- 显示 `eligible_for_injection`、`status`、`confidence`、`stability`。
- 对冲突记忆给出提示。
- 测试系统能检查“脏记忆污染”问题。

## 5. 验证方式

每一阶段都要保留这些验证：

- 后端 API 对空记忆、旧记忆、无会话状态不报错。
- 前端 build 通过。
- 召回模拟不能改变真实记忆。
- Prompt Manifest 中的记忆来源能和记忆系统显示结果对齐。
- 长场景测试里能追踪“记忆是否导致回答偏移”。

## 6. 推荐下一步

下一步优先先做页面拆分，再继续把记忆系统和系统框架图、测试 turn 连接起来。

页面拆分详见：

- `docs/84-记忆系统三层页面拆分计划书-20260426.md`

当前推荐把记忆系统拆成三个前端页面：

- 对话记忆：负责对话展开、单轮记忆链路、真实链路 / 模拟链路对照。
- 状态记忆：负责 session memory 阅读、context slots、flow state、上下文压力和测试系统 memory trace 联动。
- 长期记忆：负责 durable memory 查询、召回模拟、后续治理。

前端三页切换、测试系统 memory trace 联动、真实链路 / 模拟链路对照已经完成第一版。

长期记忆治理也已经开放受控操作：

- 写入：通过治理台创建新的 durable note。
- 停用：将 note 标记为 `inactive`，并关闭 `eligible_for_injection`。
- 归档：将 note 标记为 `archived`，并关闭 `eligible_for_injection`。
- 合并：创建新的合并 note，并将旧 note 标记为 `deprecated`。

所有治理操作都会写入 `durable_memory/meta/governance_log.jsonl`。

后续如果继续推进，需要补“撤销 / 回滚”和更细的冲突检测。

最小闭环是：

```text
测试系统点击某个 turn
-> 系统框架图高亮记忆门面
-> 点击记忆门面
-> 显示该 turn 的状态记忆、长期召回、是否进入 prompt
```

这会让 follow-up、状态漂移和长期记忆误召回的问题更容易定位。

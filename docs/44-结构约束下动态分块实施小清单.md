# 结构约束下动态分块实施小清单

> 编写日期：2026-04-22  
> 对应阶段：`docs/42-RAG成熟方案对照与改造执行清单.md` 中 `Step 2 / Phase 2-3`  
> 目的：把“结构约束下的动态分块”从原则描述收紧为可以直接驱动代码改造的实施蓝图，避免后续在 `chunking.py` 中边写边定规则。

---

## 0. 本次修订说明

结合本轮检索重构失败经验，本小清单明确收紧边界：

1. 本文只负责 `结构清洗 -> 动态分块 -> 三层节点 -> 检索输入契约`。
2. 本文不再承担 `sparse / hybrid / rerank` 主链设计职责。
3. 不允许再把“前处理契约是否成立”和“某轮 hybrid 排名是否成功”写成同一阶段的完成条件。
4. benchmark 产物必须显式带链路版本标识，不能再用旧产物反推当前代码状态。

本次修订的直接原因是：

- `normalized_ingestion` 的动态分块与三层节点能力已经落地；
- 但后续检索方案一度越过了 `44` 的边界，把 hybrid 设计和评测收益也写进了同一阶段；
- 结果导致代码状态、索引状态、评测产物曾出现失配，破坏了计划书的约束作用。

## 1. 目标

本小清单只解决一个问题：

如何在不破坏当前正式链路稳定性的前提下，把现有单层粗块生成逻辑，升级为“结构优先、长度受控、可回归、可解释”的动态分块规则。

本阶段不做：

- 不直接改 dense / sparse / hybrid 检索策略
- 不直接接入 Qdrant sparse
- 不直接上线全量句子级索引
- 不用 `TextRank` 或其它启发式算法充当主切分规则
- 不以检索指标涨跌作为本小清单是否成立的唯一判定条件

---

## 2. 设计边界

### 2.1 主规则

- 结构边界优先于 token 长度
- 长度控制只能在结构边界内部生效
- 叶子块必须尽量保持语义完整，不允许为了凑长度把明显完整的结构单元硬切碎

### 2.2 辅助规则

- 标点停顿、列表项、表格行可作为结构内二次切分信号
- `TextRank`、关键词权重、句子打分如需使用，只能作为辅助观测，不得作为主切分条件

### 2.3 明确禁止

- 纯固定 `512 token` 粗切分
- 纯 `TextRank` 驱动切分
- benchmark 专用切分规则
- 为了提高命中率把标题、正文、表格、对象暴力拼接成超大块

---

## 3. 输入契约

动态分块阶段必须只消费正式结构层已提供的字段，不允许再从原始文档临时重新猜测结构。

### 3.1 必须输入字段

- `doc_id`
- `source_type`
- `parser_backend`
- `fallback_used`
- `block_id`
- `block_type`
- `text`
- `clean_text`
- `page`
- `section_path`
- `section_label`
- `reading_order`
- `modality`
- `quality_flags`
- `cleaning_flags`
- `eligibility`
- `index_profiles`

### 3.2 可选输入字段

- `bbox`
- `object_ref_ids`
- `parent_block_id`
- parser 附带的结构元数据

### 3.3 输入前置条件

- 所有输入 block 已完成基本清洗
- `drop` block 不进入动态分块主流程
- 表格对象、图片对象、对象锚点 block 不强行并入正文块

---

## 4. 正式切分单元

动态分块不是从“字符流”开始切，而是从下面这些正式结构单元开始：

### 4.1 一级结构单元

- 标题块
- 段落块
- 列表项块
- 表格块
- 图像/对象块
- 页级结构块

### 4.2 二级切分单元

只有一级结构单元过长时，才允许继续往下切：

- 句号、分号、冒号停顿
- 列表项边界
- 表格行边界
- 同 section 内的相邻短段落合并边界

### 4.3 不允许直接作为主切分单元的对象

- 任意固定 token 窗口
- 关键词密度排序结果
- `TextRank` 排序结果
- rerank 分数

---

## 5. 长度控制规则

本阶段先采用“目标长度 + 上下界”的稳妥策略，不上复杂自适应学习逻辑。

### 5.1 建议阈值

这些值是工程起点，不是永久常量，但必须先写死为正式默认值：

- `min_tokens = 96`
- `target_tokens = 220`
- `soft_max_tokens = 320`
- `hard_max_tokens = 420`

### 5.2 规则解释

- 小于 `min_tokens`
  优先尝试与同 section、同 page、相邻 reading order 的兼容块合并
- 介于 `min_tokens` 与 `soft_max_tokens`
  默认保持当前结构块完整，不再继续切
- 高于 `soft_max_tokens`
  启动结构内二次切分
- 高于 `hard_max_tokens`
  必须继续切，直到回到允许范围内

### 5.3 表格与对象例外

- `table` / `figure` / `object_anchor` 不适用正文长度规则
- 表格优先按表格行、表格说明、对象锚点单独建模
- 对象块不得因长度规则被随意拆碎

---

## 6. 合并规则

动态分块不只是“切”，还包括必要的“合”。

### 6.1 允许合并的条件

- 同 `doc_id`
- 同 `section_path`
- 同 `page` 或相邻页且结构明确连续
- `reading_order` 连续
- `block_type` 兼容
- 合并后不超过 `soft_max_tokens`

### 6.2 不允许合并的条件

- 标题块跨 section 合并
- 表格块与正文块直接合并
- 对象块与说明块无边界拼接
- 不同 parser fallback 路径下的异构残片硬合并

- rerank 前局部抽取
- 命中块内部细化

## 9. 必做回归点

### 9.1 结构契约回归

- Markdown 标题、段落、列表项是否稳定透传
- PDF fallback 到 MinerU 时，section/page 是否稳定透传
- object/table 是否保留独立结构身份

### 9.2 动态分块回归

- 短段落是否发生合理合并
- 长段落是否只在结构内切分
- 表格是否未被正文切分逻辑污染
- 不同 section 的块是否没有被误合并

### 9.3 三层节点回归

- `document_summary` 是否存在
- `parent_section` 是否存在
- `leaf_block` 是否保留对 parent 的稳定引用
- 句子级节点是否没有误入第一阶段主索引

### 9.4 benchmark 回归

- SciFact 不允许临时走另一套切分规则
- 动态分块改造后必须重新输出诊断信息
- 如果 top1 下降，必须先区分是切分问题、索引问题、检索问题，还是代码与产物状态不一致的问题

### 9.5 产物一致性回归

- 评测输出必须记录 `chain_version / build_id / index_root / code_commit`
- 不允许继续复用未标记链路版本的旧 benchmark 结果
- 代码主链、索引元数据、评测产物三者必须能互相对上

## 10. 完成标准

只有同时满足下面条件，才允许认为 `Step 2` 完成：

- [x] 动态分块规则已经写成正式代码，不再只是文档原则
- [x] 输入契约与输出契约已经固定
- [x] 三层节点已能被稳定生成
- [x] 句子级能力仍被限制在局部增强层
- [x] 回归测试已覆盖结构透传、动态切分、层级节点三个方面
- [ ] benchmark 没有引入正式系统之外的特化切分逻辑
- [ ] benchmark 产物已显式标记链路版本，且与当前代码状态一致
- [ ] 检索层不再越权追加前处理阶段不存在的临时切分逻辑

如果以上任一项未满足，则说明 `Step 2` 仍未完成，不能进入后续 sparse/hybrid 正式迁移。

补充说明：

- `Step 2 完成` 不等于 `检索主链已经达标`
- `Step 2 完成` 只代表前处理、动态分块、三层节点和检索输入契约已经收口
- sparse/hybrid/rerank 是否成熟，必须由后续独立检索计划单独验收

## 11. 当前落地状态

截至 `2026-04-22`，本小清单对应的正式代码可确认已完成以下落地：

- `document_conversion -> normalized_ingestion -> chunking` 已稳定透传结构字段
- `chunking.py` 已按结构优先与长度约束生成动态 leaf block
- `document_summary / parent_section / leaf_block` 三层节点已进入正式索引生成链路
- 相关回归已稳定通过：
  - `backend/tests/document_conversion_phase1_regression.py`
  - `backend/tests/normalized_ingestion_cleaning_regression.py`
  - `backend/tests/retrieval_core_phase2_regression.py`
  - `backend/tests/retrieval_bootstrap_phase2_regression.py`
  - `backend/tests/retrieval_service_cutover_regression.py`

但本轮也确认了下面这些边界尚未在 `44` 的口径上完全收口：

- 当前检索层虽然消费了三层节点生成出的 `IndexableUnit`，但不能据此等同于“层级化检索主链已经完成”
- `parent_context / document_context` 的回填方式与排序关系，不应继续写成 `44` 的既成事实
- benchmark 索引、评测产物与当前代码版本仍缺少统一版本标识
- 因此，`44` 现在应被视为“前处理与节点建模边界文档”，而不是“检索主链完成说明”

## 1. 结论

本轮 extended 长测已经不只是“两个场景失败”，而是暴露出三类更关键的问题：

1. `durable memory` 写入与跨会话召回仍不稳定，且会被旧主线污染。
2. `PDF` 跟读链虽然经常触发了正确工具，但可见答案仍可能是空摘要，形成“伪通过”。
3. 一旦中段出现记忆写入异常，`session summary / compaction view / flow state` 会继续携带错误语义，导致后续复合任务、总结任务、跨线程判断全部漂移。

因此，修复优先级不能只盯失败 turn，必须同时处理：

1. 真失败项
2. 断言通过但语义错误项
3. 会反向污染 session memory 的错误答案

## 3. 显式失败问题

## 3.1 LONG-003：跨会话 recall 场景混入了不适合 durable memory 的测试项

场景：

- `工作偏好写入与跨会话回忆`

失败 turn：

- `turn-06-recall.json`
- 文件：`output/test_runs/20260424-live-extended-full/artifacts/memory-preference-and-cross-session-recall/turn-06-recall.json`

用户原话：

- `我们这个项目现在优先抓哪条主线？`

预期：

1. 直接从 durable memory / recall 链路回答
2. 至少应包含 `Memory` 或 `RAG`

实际行为：

1. route 是 `agent`
2. 中途调用了 `search_knowledge`
3. 回答内容变成：
   - “我需要先了解这个项目的具体背景”
   - “让我先检索一下本地知识库”

已确认事实：

1. 这条失败并不完全等价于“durable recall 坏了”
2. 场景本身把“项目主线 / 默认终端”当成跨会话 durable 记忆断言，这个设定并不合理
3. 更适合跨会话 durable recall 的，应当是稳定用户偏好，例如：
   - 复杂问题先给结论
   - 回答时如何称呼用户
   - 信息不足时先说明缺口，不直接猜

根因边界：

1. 场景把“用户偏好型 durable memory”和“项目/环境约定”混在一起了
2. 后两类信息更适合走静态 profile、会话态约束或工作上下文，不应直接作为 durable recall 金标准
3. 因此这条失败里既有 recall 问题，也有场景设计问题，不能直接据此要求把“项目主线”持久化

修复方向：

1. 把该场景改成只验证稳定用户偏好型 durable recall
2. 项目主线、终端默认等内容，改由更合适的静态协议或会话上下文测试覆盖
3. durable recall 继续只承接：
   - 稳定回答偏好
   - 稳定称呼偏好
   - 稳定处理偏好

## 3.2 LONG-007：长跑中段发生记忆写入污染后，复合任务失去分解能力

场景：

- `六十轮真实用户长跑`

首错 turn：

- `turn-27-main.json`
- 文件：`output/test_runs/20260424-live-extended-full/artifacts/sixty-turn-real-user-marathon/turn-27-main.json`

用户原话：

- `先总结 PDF 第三页，再给我 inventory.xlsx 最缺货的前三个仓库，最后补一句北京天气。`

预期：

1. 至少拆成 3 个子任务
2. 触发 `subtask_start`
3. 按 PDF / 数据 / 实时分别推进

实际行为：

1. `plan.subqueries=1`
2. 没有 `subtask_start`

   - `库存最缺货仓库：需要分析 inventory.xlsx 文件`
   - `北京天气：正在查询`

3. `flow_state` / `active_goal` / `current_task_state` 已经错位
4. 错位后 planner 没有进入任务分解链，而是退化成一段 memory prose

修复方向：

1. 先修 19-22 turn 的 memory 写入污染
2. 再修复复合任务在低置信度/脏主线状态下的退化行为
3. 禁止“没有真实执行的正在查询”进入最终答案

## 4. 语义伪通过问题

这些问题没有让场景断言失败，但在真实使用里依然属于明显错误。

## 4.1 PDF 页级追问返回空摘要，属于答非所问

涉及文件：

1. `output/test_runs/20260424-live-extended-full/artifacts/research-brief-and-document-resume/turn-07-main.json`
2. `output/test_runs/20260424-live-extended-full/artifacts/research-brief-and-document-resume/turn-08-main.json`
3. `output/test_runs/20260424-live-extended-full/artifacts/sixty-turn-real-user-marathon/turn-06-main.json`
4. `output/test_runs/20260424-live-extended-full/artifacts/sixty-turn-real-user-marathon/turn-07-main.json`
5. `output/test_runs/20260424-live-extended-full/artifacts/multi-session-workbench-isolation/turn-04-doc.json`

典型表现：

- `已读取这份 PDF，但当前工具尚未形成可直接展示的摘要。`

问题性质：

1. 工具被正确调用了
2. 绑定也还在
3. 但对“第三页讲了什么 / 第四页重点句”这种问题，没有返回内容性答案

为什么这很严重：

1. runner 只检查了 `response.nonempty`，所以被判通过
2. 但真实用户看到的仍是“空回答”
3. 这个空回答还会被写入 `task_summary_refs / session summary / warm snapshots`

根因边界：

1. PDF 工具输出到可见答案之间仍存在 `pdf_missing_summary`
2. 工具结果没有稳定进入 canonical visible answer
3. fallback 文本被当成正常跟读结果持久化

修复方向：

1. `pdf_analysis` 必须保证页级追问至少返回：
   - 结构化页内容
   - 或明确的“该页无正文信息”
2. 不能再用“尚未形成可展示摘要”充当正常页读结果
3. `pdf_missing_summary` 不能继续进入 session summary

## 4.2 记忆写入语句被旧 PDF 主线串线

1. `output/test_runs/20260424-live-extended-full/artifacts/sixty-turn-real-user-marathon/turn-20-main.json`
2. `output/test_runs/20260424-live-extended-full/artifacts/sixty-turn-real-user-marathon/turn-21-main.json`
3. `output/test_runs/20260424-live-extended-full/artifacts/sixty-turn-real-user-marathon/turn-22-main.json`

### turn 20

用户原话：

- `记住：回答我时可以直接称呼我岩。`

实际回答：

- `当前没有可验证的执行结果。`

问题性质：

1. 这不是工具执行问题，也不需要 receipt
2. 却被错误套用了“无可验证执行结果”的 fallback

这说明：

1. `durable_memory_statement` 与“伪执行应答收缩”发生了错误耦合
2. 某些纯记忆写入句子被误判成“需要真实执行凭证”

### turn 21

用户原话：

- `记住：如果信息不足，先明确告诉我缺什么，不要直接猜。`

实际行为：

1. 开始谈“之前讨论的 PDF 内容”
2. 调用了 `search_knowledge`
3. 检索词是 `PDF 第二部分 约束 重点`

问题性质：

1. 语义完全错位
2. durable 写入意图被旧 PDF follow-up 链吸走

### turn 22

用户原话：

- `你刚才帮我长期记住了什么？`

实际回答：

1. 只确认第一条“先给结论”
2. 后续稳定偏好没有一起被可靠带回

这证明 turn 20/21 不是偶发口误，而是确实写入失败或回执失败。

根因边界：

1. 记忆写入语句没有与当前业务主线做硬隔离
2. `active_goal / flow_type / current_task_state` 被旧 PDF 主线继续占用
3. 输出回退策略把记忆写入句误杀成“无执行结果”

修复方向：

1. `durable_memory_statement` 必须强制切出独立的低耦合处理支路
2. 对纯记忆写入句，禁止读取旧 PDF/数据 follow-up 绑定
3. `no_receipt_tool_claim` 规则不能误杀纯写入确认句

## 4.3 长跑尾部总总结发生事实性失真

涉及文件：

- `output/test_runs/20260424-live-extended-full/artifacts/sixty-turn-real-user-marathon/turn-60-main.json`

错误文本：

- `本次会话完成了两项数据分析任务，未涉及 PDF 和实时信息查询，应用了一条长期记忆偏好。`

问题性质：

1. 这条 60 turn 场景前面显然做过 PDF、天气、金价、多会话切换
2. 总结却说“未涉及 PDF 和实时信息查询”
3. 这是事实性错误，不是表达偏弱

为什么这很严重：

1. 总结错误会进入 session summary
2. 后续复盘会拿错事实继续推理
3. 这类错总结比单轮错答污染更深

根因边界：

1. 总总结只抓取了局部近邻数据任务
2. earlier flows 被压缩或被错误 current task state 覆盖
3. summary assembly 没有按 “PDF / 数据 / 实时 / 长期记忆” 做真实回放聚合

修复方向：

1. 总结任务不能只看近邻 hot window
2. 必须回放结构化的 session-level flow ledger
3. 若某类分区没有确定结果，应用“本轮存在但未形成稳定结论”，而不是“未涉及”

## 4.4 “正在查询”类措辞仍有残留误导

涉及文件：

- `output/test_runs/20260424-live-extended-full/artifacts/sixty-turn-real-user-marathon/turn-27-main.json`

典型文本：

- `北京天气：正在查询`

已确认事实：

1. 该 turn 没有天气工具调用
2. 只是自然语言里写了“正在查询”

问题性质：

1. 这不是严格意义上的伪 receipt
2. 但会给用户造成“系统已经开始执行”的错觉

根因边界：

1. `procedural_promise` 的收缩还没有覆盖这种半执行式分项表达
2. 尤其在复合任务场景中，模型容易写“第一项怎样、第二项怎样、第三项正在查”

修复方向：

1. 对没有 receipt 的分项任务，不允许写“正在查询 / 稍后给您结果”
2. 应改成：
   - `尚未执行该子任务`
   - 或 `当前未形成真实查询结果`

## 5. 问题间的因果链

这轮长测中，多个问题不是并列独立，而是会级联。

当前已确认的主要链路如下：

1. `PDF 页级回答为空`
   ->
2. `空摘要被写入 session summary`
   ->
3. `记忆写入 turn 仍挂在旧 PDF 主线下`
   ->
4. `称呼偏好 / 缺口优先说明` 这类稳定偏好写入失败或回执异常
   ->
5. `你刚才帮我长期记住了什么` 给出异常说明
   ->
6. `复合任务` 时 planner/state 已被污染，无法正常分解
   ->
7. `长跑尾部总总结` 再次读取错误 session facts，生成事实失真总结

这说明修复顺序必须是：

1. 先切断记忆写入与旧主线污染
2. 再修 PDF 空摘要
3. 再修复合任务和总总结

不能倒过来。

## 6. 修复优先级

## P0：必须先修

1. `durable_memory_statement` 与旧 follow-up 主线隔离
2. `durable_memory_query` 禁止误转 `search_knowledge`

目标：

1. `回答我时可以直接称呼我岩`
2. `如果信息不足，先明确告诉我缺什么，不要直接猜`
3. `你刚才帮我长期记住了什么`

这三条链要先打通。

## P1：紧接着修

1. `pdf_analysis` 页级追问不能再返回空摘要
2. `pdf_missing_summary` 不得进入 session summary

目标：

1. `第三页具体讲了什么`
2. `第四页重点看哪几句`

1. 复合任务的子任务分解在低置信度脏状态下的退化

## 7. 回归补充建议

本轮问题已经说明现有 runner 断言不够覆盖语义质量，需要补充以下回归。

## 7.1 durable memory 语义回归

新增回归应覆盖：

1. 写入：
   - `记住：回答我时可以直接称呼我岩。`
   - `记住：如果信息不足，先明确告诉我缺什么，不要直接猜。`
2. 召回：
   - `你刚才帮我长期记住了什么？`
   - 新 session：`你之后应该怎么称呼我？`
   - 新 session：`如果信息不足，你应该怎么处理？`

断言不应只看 `response.nonempty`，而应要求包含：

1. `岩`
2. `缺什么`
3. `先给结论`

并且：

1. 不允许 `search_knowledge`

## 7.2 PDF 语义回归

新增回归应覆盖：

1. `第三页具体讲了什么？`
2. `第四页如果要给业务负责人看，应该重点看哪几句？`

断言应要求：

1. 不包含 `尚未形成可直接展示的摘要`
2. 回答长度和内容命中最低阈值

## 7.3 长跑语义回归

对 60 turn 场景，应额外增加语义检查：

1. 复合任务必须产生 `subqueries>=3`
2. 总总结不得出现：
   - `未涉及 PDF`
   - `未涉及实时`
   前提是 earlier turns 实际发生过这些查询

## 9. 最终判断

这轮 extended 长测已经证明：

1. 系统最大的风险不再只是“工具有没有调起来”
2. 而是“错误语义被当成稳定事实继续传下去”

当前最危险的不是单轮错答，而是：

1. 空 PDF 摘要被写成上一阶段结果
2. 记忆写入被旧业务主线劫持
3. 错误总结被写回 session summary

因此，下一轮修复必须围绕：

1. `durable write/read isolation`
2. `pdf visible answer guarantee`
3. `summary truthfulness`

依次推进。

## 2. 设计目标

本轮不是单纯为了解决某个长测试失败，也不是做表面 prompt 去噪，而是为了解决当前系统的结构性问题：

- compound 子任务的局部约束无法稳定保留
- follow-up 对“第一个/第二个子任务”的解析仍依赖 session 模糊文本猜测
- 主线程会吸收子任务的 raw evidence、tool raw output、debug/runtime metadata，导致上下文污染
- session memory 既像恢复索引，又像日志仓库，职责混乱
- durable memory 的输入边界过松，容易吃进 task-local / transient 噪声
- prompt 中混入治理性元数据，模型被迫看到过多本不该看到的运行时信息

最终目标是形成下面这个清晰的事实边界：

- 主线程只保留“继续决策所必需”的状态
- 子任务只保留“完成该任务所必需”的状态
- 原始证据只在局部消费，不回流主 prompt
- session memory 只负责恢复，不负责归档
- durable memory 只负责稳定事实，不负责 live workflow
- debug / trace 只用于观测和测试，不进入模型上下文

## 3. 外部技术调研与选型结论

### 3.1 参考的一手资料

- Anthropic subagents：独立上下文窗口，防止主线程污染
- Anthropic prompt caching：稳定前缀与易变后缀
- LangGraph subgraph persistence：per-invocation 子图默认隔离
- LangGraph persistence/checkpointer：线程级状态快照
- OpenAI practical guide to agents：centralized orchestration 与 handoff 的权衡
- Microsoft AutoGen：structured output、bounded model context

### 3.2 采用的模式

#### Anthropic subagents 的 separate context window 思想

采用原因：

- 它解决的是当前最真实的问题：主线程被探索性中间过程污染。
- 我们需要的是“子任务局部上下文独立”，而不是立即引入完整子 agent 框架。

在本项目中的落地方式：

- 先在 `TaskCoordinator + QueryRuntime` 内实现逻辑上的 task-local context
- 子任务完成后只返回结构化 summary 和 result ref，不返回完整中间过程给主线程

#### LangGraph 的 per-invocation isolated subgraph 思想

采用原因：

- 它强调“每个子任务 invocation 默认隔离状态”，这非常适合当前 compound query 的子任务模型。

在本项目中的落地方式：

- 用现有 `dataclass + local JSON/task records` 实现类似 checkpointer 的语义
- 每个 task 自带 `TaskContextRef`，不共享 live mutable state

#### AutoGen 的 structured output + bounded model context 思想

采用原因：

- 当前系统最大的问题之一，就是子任务结果没有强约束结构，最终容易裸拼 raw output。

在本项目中的落地方式：

- 运行时内部状态仍用 `dataclass`
- LLM 输出协议、follow-up resolver、answer assembler 使用 `Pydantic`

#### Anthropic / OpenAI 的 prompt boundary / cache-stable prefix 思想

采用原因：

- 当前 `prompt_builder.py` 把 static、session-stable、turn-volatile、debug notes 混在一起，是典型的上下文污染源。

在本项目中的落地方式：

- prompt 明确拆成 `static / session-memoized / turn-volatile`
- 稳定内容字节稳定，便于 cache 和认知稳定

### 3.3 明确不采用的方案

#### 不把项目直接迁到 LangGraph / AutoGen / OpenAI Agents SDK

原因：

- 当前项目最缺的是边界纪律，不是框架能力。
- 直接换框架，成本大、迁移面广，而且容易把原有问题换一种形式继续存在。

#### 不采用完全 decentralized handoff

原因：

- 当前项目的 memory、follow-up、final answer assembly 必须由主线程集中治理。
- 如果子 agent 直接接管用户对话主权，会进一步加剧上下文和记忆边界混乱。

#### 不引入外部 memory framework 替换现有 memory system

原因：

- 当前已有 `structured_memory / context_management / memory facade` 基础，不应推倒。
- 正确做法是收口边界，而不是整体替换。

## 4. 架构总蓝图：7 层上下文 + 1 层 Debug

本轮改造的正式目标结构如下：

1. `Static Prompt Layer`
2. `Session-Stable Context Layer`
3. `Main Working Context Layer`
4. `Task Local Context Layer`
5. `Evidence Layer`
6. `Session Memory Layer`
7. `Durable Memory Layer`

外加一层：

- `Debug / Trace Layer`

### 4.1 Static Prompt Layer

职责：

- 存放稳定的 system-level 规则与身份信息
- 为模型提供不会随 turn 抖动的高层指令

典型内容：

- stable skill summary
- safety policy
- permissions policy
- product identity
- durable memory policy

禁止项：

- 不得包含 turn-specific 状态
- 不得包含 task-local evidence
- 不得包含 debug / trace / dropped-sections 之类治理信息

### 4.2 Session-Stable Context Layer

职责：

- 存放当前 session 内相对稳定、可跨 turn 复用的上下文

典型内容：

- stable project conventions
- durable memory summaries
- stable session constraints
- active skill block，如果它在 session 维度上稳定

禁止项：

- 不得承载本轮 task-local 工作状态
- 不得承载 raw retrieval/tool output

### 4.3 Main Working Context Layer

职责：

- 这是主线程真正要送进模型的“当前真相”
- 它只保留继续决策所必需的信息

典型内容：

- active goal
- active work item
- follow-up target task id
- active constraints
- latest correction
- next step

禁止项：

- 不得直接塞入完整 evidence
- 不得直接塞入 session memory 的流水账
- 不得承担 task-local detail 的存放职责

### 4.4 Task Local Context Layer

职责：

- 为每个 compound 子任务保存独立的局部绑定、约束、状态和恢复信息

典型内容：

- task bindings
- local constraints
- local evidence refs
- local status
- local summary
- local recovery hints

禁止项：

- 不得直接覆盖主线程工作态
- 不得与其他子任务共享 mutable live state

### 4.5 Evidence Layer

职责：

- 存放原始检索结果、工具原始输出、表格原始结果、PDF snippets 等证据

典型内容：

- retrieval raw chunks
- tool raw outputs
- table raw results
- page snippets

原则：

- raw evidence 局部消费
- 主线程只拿 evidence summary，不拿 evidence body

禁止项：

- 不得让 raw evidence 直接进入主 prompt
- 不得把 raw evidence 直接抄进 session memory

### 4.6 Session Memory Layer

职责：

- 只做恢复索引与恢复导向状态投影

典型内容：

- recovery-oriented summary
- recent task summaries
- current valid constraints
- correction log
- warm snapshots

禁止项：

- 不做日志仓库
- 不存 raw assistant output
- 不存 raw tool output

### 4.7 Durable Memory Layer

职责：

- 只保留跨会话稳定的事实与约定

典型内容：

- stable user facts
- reusable project conventions
- durable correction outcomes

禁止项：

- 不保存 live task state
- 不保存 transient 情绪
- 不保存 task-local noisy notes

### 4.8 Debug / Trace Layer

职责：

- 支撑测试、排查、回归与可观测性

典型内容：

- trace id
- timing
- compaction strategy
- selected sections
- dropped sections
- selected evidence

强约束：

- 默认不进入模型 prompt
- 只能留在 event payload、trace、artifacts、测试断言侧

## 5. 技术实现原则

### 5.1 数据建模技术

- `dataclass(slots=True)`：用于 runtime 内部状态对象
- `Pydantic BaseModel`：用于 LLM 输出协议、follow-up resolver 输出、answer assembler 协议
- `TypedDict`：用于轻量 event payload

选择理由：

- 当前代码已大量使用 `dataclass`，迁移成本最低
- 需要校验模型结构化输出时，裸 `dict` 不够稳

### 5.2 状态存储技术

- 继续保留本地 `JSON / Markdown`
- 新增 `task state snapshot JSON`
- 本轮不引入数据库
- 本轮不引入 Redis / PostgreSQL

### 5.3 调度与编排技术

- 继续基于现有 `TaskCoordinator`
- 引入 `TaskContextRef`
- 引入 `summary-first return`
- 暂不引入 Celery / distributed queue

### 5.4 压缩与上下文预算技术

- 保留现有 `context_controller / context_compactor`
- 但压缩对象从“整段 messages”升级为“分层 context package”

### 5.5 回答装配技术

- 新增 `AnswerAssembler`
- 主线程最终输出不再直接拼工具 raw output
- 子任务只返回 `summary + result_ref + constraints`

## 8. 分阶段实施清单

下面的分阶段顺序不是可选建议，而是本轮执行的固定顺序。

### Phase 0：护栏与观测

目标：

- 把 prompt 分层、task record 结构、follow-up 目标解析都变成可回归事实

修改文件：

- `backend/tests/task_coordinator_regression.py`

- `backend/tests/system_eval/long_runner.py`

退出条件：

- 能稳定看到主 prompt 中有哪些层
- 能断言 raw evidence 是否泄漏到主 prompt
- 能断言 compound 子任务结果是否仍是裸拼

---

### Phase 1：Task Context Ref 落地

目标：

- 每个 compound 子任务都有独立上下文引用、绑定、约束和 summary

修改文件：

- `backend/tasks/context_models.py` 新增

- `backend/tasks/coordinator.py`
- `backend/tasks/__init__.py`
- `backend/query/runtime.py`
- `backend/tests/task_coordinator_regression.py`

退出条件：

- compound 每个子任务都有独立 `task_id + bindings + constraints`
- `subtask_end` 返回结构化 `summary/context_ref/result_ref`

### Phase 2：Main Working Context 重建

# 编排系统 Agent 行为逻辑开发平台升级规划

日期：2026-04-26

## 1. 这次升级要解决什么

现在的编排系统已经能把测试 turn 或当前运行链路整理成“运行控制塔”，这解决的是“这一轮发生了什么”。下一步要升级的是另一件事：让编排系统成为 Agent 行为逻辑的开发平台。

这里的“行为逻辑”不是工具列表，也不是 skills 开关。它指的是 Agent 面对一条输入时，如何理解任务、如何选择执行姿态、如何读取上下文、如何使用 skills 和工具契约、如何进入 worker 或模型主链、如何收口输出、失败时如何恢复。

正确的目标状态是：

- 编排系统能读出当前后端真实决策链，不靠前端猜测。
- 编排系统能解释一次请求为什么进入某条 route、为什么使用某个 skill、为什么开放或拒绝某个工具。
- 编排系统能做 dry-run，让用户输入一句话后，在不调用模型、不执行工具、不写入记忆的前提下，看到 Agent 会怎样规划。
- 编排系统未来可以承载行为策略草案，但所有策略仍必须经过后端的路由层、skills policy、工具契约、权限、上下文策略校验，不能成为运行时后门。

## 2. 当前后端真实链路对照

### 2.1 Skills 链路

相关文件：

- `backend/skill_system/registry.py`
- `backend/skill_system/policy.py`
- `backend/tools/skills_scanner.py`
- `backend/SKILLS_REGISTRY.json`

当前结构已经比较清楚。`SkillRegistry` 从 `SKILLS_REGISTRY.json` 加载 skill runtime contract，`SkillDefinition` 同时提供两种视图：运行时契约和 prompt 可见视图。`SkillPolicyResolver` 不靠 prompt 文字匹配，而是根据 `route / modality / task_kind / source_kind / capability_requests / candidate_tools` 做结构化匹配。

这说明编排系统不应该直接编辑 skill prompt 来控制行为，而应该读取和展示：

- 当前任务帧命中了哪些 skill contract。
- 为什么选择当前 skill，例如 `tool_contract_match`、`capability_contract_match`、`task_kind_contract_match`。
- skill 给出的 `tool_scope` 是什么。
- skill 的 prompt view 是否被注入模型。

需要补强的是：当前 `SkillPolicyResolver.resolve()` 只返回最终 frame，候选 skill 和被过滤原因没有完整暴露。编排系统要做行为开发平台，需要一个 read-only 的 `SkillPolicyTrace`，记录候选、命中项、过滤项和最终选择。

### 2.2 契约层链路

相关文件：

- `backend/tools/contracts.py`
- `backend/tools/definitions.py`
- `backend/tools/tool_registry.py`
- `backend/tools/runtime.py`
- `backend/query/runtime_tools.py`
- `backend/permissions/service.py`
- `backend/permissions/decision_pipeline.py`
- `backend/permissions/policy.py`

当前契约层分成四部分：

- `ToolDefinition` 描述工具能力、输入契约、输出契约、投影契约、安全标签和是否可自动路由。
- `ToolContractGate` 检查必填输入、绑定对象和 skill scope。
- `PermissionService` 按当前 permission mode、tool scope、安全标签和工具本地校验决定是否允许调用。
- `RuntimeToolBridge` 在真实执行前串起 contract gate、permission、tool runtime、task coordinator 和输出收口。

这条链路已经可以成为编排系统的“安全边界可视化”。编排系统应该展示的不是“这个工具按钮能不能点”，而是：

- 这轮请求候选了哪些工具。
- 哪些工具被 skill scope 允许或过滤。
- 最终工具是否通过契约检查。
- 如果失败，是缺 input、缺 binding、scope 拒绝、permission 拒绝，还是工具本地校验失败。
- 输出 contract 会如何影响最终回答和记忆投影。

需要补强的是：契约决策目前主要出现在 direct tool 执行时。dry-run 阶段需要新增一个“不执行工具的 contract preview”，让编排系统提前看到某个工具若被调用会卡在哪个契约节点。

### 2.3 上下文管理链路

相关文件：

- `backend/context_management/context_controller.py`
- `backend/context_management/context_models.py`
- `backend/context_management/context_compactor.py`
- `backend/memory/facade.py`
- `backend/memory/context.py`
- `backend/query/runtime_context_state.py`
- `backend/structured_memory/session_memory.py`
- `backend/structured_memory/process_state.py`

当前上下文系统的真实职责是把会话历史、状态记忆、长期记忆、检索证据和静态上下文整理成 `ContextPackage`。`ContextPackage` 已经包含 pressure level、预算、model-visible sections、debug sections、selected/dropped sections、compaction decisions 和 token accounting。

这条链路非常适合接入编排系统，因为它能回答 Agent 行为里最关键的一类问题：这一轮模型到底看到了什么，为什么看这些，哪些被丢弃了。

编排系统应该读取：

- `pressure_level`
- `budget`
- `selected_sections`
- `dropped_sections`
- `compaction_strategy`
- `compaction_decisions`
- `model_visible_sections`
- `debug_sections`
- `token_accounting`

需要注意边界：记忆系统负责“阅读和管理记忆文件”，编排系统只负责解释“本轮行为中哪些记忆被选入上下文、哪些没有选入、原因是什么”。也就是说，编排系统可以跳转到记忆系统，但不直接替代记忆系统的文件管理页面。

### 2.4 路由层链路

相关文件：

- `backend/understanding/memory_intent.py`
- `backend/understanding/task_understanding.py`
- `backend/understanding/query_understanding.py`
- `backend/query/continuation_resolver.py`
- `backend/query/followup_resolver.py`
- `backend/query/planner.py`
- `backend/query/bundle_planner.py`
- `backend/query/subtask_planner.py`
- `backend/query/capability_dispatch.py`
- `backend/query/runtime.py`

当前路由不是单点完成的，而是多段叠加：

- `memory_intent` 先判断是否为记忆读写或会话总结类任务。
- `task_understanding` 从输入中抽取结构信号，产出 `route_hint`、`execution_posture`、`capability_requests`、`candidate_tools`。
- `query_understanding` 把 task understanding 转为 runtime 可用的 `QueryUnderstanding`，并用 tool registry 补候选工具。
- `continuation_resolver` 根据显式 PDF、数据集、会话总结等信号提升 route。
- `QueryPlanner` 决定 `single_execution / bundle_execution / explicit_fanout`。
- `CapabilityDispatchScheduler` 记录 skill/tool/worker dispatch plan，目前是兼容式 shadow 记录。
- `QueryRuntime` 最终执行 direct tool、worker、bundle 或模型主链。

这说明编排系统要接入的不是一个“路由开关”，而是一串行为决策。它应该把 route 拆成可读阶段：

- 输入信号识别。
- 记忆意图判断。
- 任务理解。
- follow-up / continuation 判断。
- bundle 或 explicit fanout 判断。
- skill policy 选择。
- capability dispatch。
- execution kind 落点。

需要补强的是：当前很多中间阶段只有最终结果，没有统一的决策 trace。编排系统升级时应新增 `BehaviorDecisionTrace`，把这些阶段统一记录。

## 3. 推荐设计方向

编排系统升级为 `Agent Behavior Control Plane`，中文界面可以叫“行为控制台”或继续叫“编排系统”。

它不拥有 skills、tools、memory、worker 的原始定义，也不直接执行测试。它拥有的是：

- 行为策略的组合视图。
- 单轮行为决策 trace。
- dry-run 推演能力。
- 策略校验与风险提示。
- 和测试系统、记忆系统、操作系统、Agent 系统之间的跳转联动。

四条模块关系应该这样定义：

- 路由层提供“任务理解与执行姿态”的决策来源。
- 上下文管理系统提供“模型可见上下文”的决策来源。
- skills 提供“当前行为包与能力范围”的决策来源。
- 契约层提供“工具、权限、输出投影”的边界来源。

编排系统把这些来源组合成“这一轮 Agent 为什么这样行动”的解释面。

## 4. 目标数据模型

### 4.1 AgentBehaviorPolicy

用于表达策略草案，不直接代表最终运行事实。

建议字段：

```python
@dataclass(slots=True)
class AgentBehaviorPolicy:
    policy_id: str
    name: str
    mode: Literal["readonly", "dry_run", "turn_override"]
    route_preferences: dict[str, Any]
    clarification_policy: dict[str, Any]
    memory_strategy: dict[str, Any]
    context_strategy: dict[str, Any]
    skill_strategy: dict[str, Any]
    tool_contract_strategy: dict[str, Any]
    worker_strategy: dict[str, Any]
    output_strategy: dict[str, Any]
    recovery_strategy: dict[str, Any]
```

第一阶段只保存和展示默认策略，不让它改变 runtime。

### 4.2 BehaviorDecisionTrace

用于表达一次请求的真实或推演决策。

建议字段：

```python
@dataclass(slots=True)
class BehaviorDecisionTrace:
    trace_id: str
    source: Literal["live_turn", "test_turn", "dry_run"]
    session_id: str
    run_id: str
    turn_id: str
    input_text: str
    final_route: str
    execution_mode: str
    execution_kind: str
    nodes: list[BehaviorDecisionNode]
    edges: list[BehaviorDecisionEdge]
    module_refs: dict[str, Any]
    warnings: list[str]
```

### 4.3 BehaviorDecisionNode

用于可视化节点。

建议节点类型：

- `input_signal`
- `memory_intent`
- `task_understanding`
- `continuation`
- `followup`
- `execution_mode`
- `skill_policy`
- `context_package`
- `capability_dispatch`
- `tool_contract`
- `permission`
- `worker_selection`
- `prompt_assembly`
- `output_boundary`
- `persistence`

每个节点至少包含：

- 序号。
- 中文名称。
- 来源文件或来源阶段。
- 输入摘要。
- 输出摘要。
- 决策原因。
- 状态：`passed / warning / blocked / inferred / skipped`。
- 可跳转目标，例如记忆系统、操作系统、测试系统、Agent 系统。

### 4.4 ContractPreview

用于 dry-run 中提前展示工具契约风险。

建议字段：

```python
@dataclass(slots=True)
class ContractPreview:
    tool_name: str
    scope_allowed: bool
    contract_action: str
    contract_reason: str
    permission_allowed: bool
    permission_reason: str
    missing_inputs: list[str]
    missing_bindings: list[str]
    risk_tags: list[str]
```

## 5. 后端升级方案

### 5.1 新增行为控制模块

建议新增目录：

- `backend/orchestration/behavior_models.py`
- `backend/orchestration/behavior_trace.py`
- `backend/orchestration/behavior_dry_run.py`
- `backend/orchestration/behavior_policy.py`
- `backend/orchestration/contract_preview.py`

其中：

- `behavior_models.py` 放结构模型。
- `behavior_trace.py` 把 `QueryPlan`、`QueryExecutionPlan`、`DispatchPlan`、`ContextPackage`、`PromptManifest` 转成统一 trace。
- `behavior_dry_run.py` 调用 planner、memory facade、contract preview，但不调用模型、不执行工具、不写入 session。
- `behavior_policy.py` 负责读取、校验、规范化行为策略草案。
- `contract_preview.py` 复用 `ToolContractGate` 和 `PermissionService` 做只读预检。

### 5.2 给现有 Planner 增加 trace 输出，不改决策

`QueryPlanner.build_plan()` 当前直接返回 `QueryPlan`。建议第一阶段不改返回值，新增旁路方法：

- `build_plan_with_trace(...) -> tuple[QueryPlan, BehaviorDecisionTrace]`

或者更稳妥：

- 保持 `build_plan()` 不动。
- 在 `behavior_trace.py` 中用 `QueryPlan` 反向构造 read-only trace。
- 后续再把中间候选信息补进 planner。

推荐先选第二种，风险低。

### 5.3 给 SkillPolicyResolver 增加候选解释

新增方法：

- `SkillPolicyResolver.inspect(task_frame) -> SkillPolicyInspection`

返回：

- 最终 skill。
- 候选 skill。
- 每个候选的 specificity。
- 每个候选的 reasons。
- 被 forbidden route 过滤的 skill。

现有 `resolve()` 继续保持原逻辑，内部可以复用 inspect 的结果。

### 5.4 给契约层增加 preview

新增 `ContractPreviewService`，输入：

- `QueryExecutionPlan`
- `DispatchPlan`
- 当前 permission mode
- 当前 skill scope

输出：

- 候选工具 contract preview。
- selected tool contract preview。
- permission preview。

这里不能执行工具，只调用：

- `ToolContractGate.evaluate()`
- `PermissionService.can_invoke_tool()`
- `ToolRuntime.get_contract()`

### 5.5 给上下文系统增加策略解释视图

当前 `ContextPackage.to_dict()` 已经很完整。编排系统需要的是更可读的投影：

- 哪些 section 进入模型。
- 哪些 section 只在 debug 可见。
- 哪些被压缩或丢弃。
- token 预算如何分配。
- durable memory 是否参与。
- retrieval evidence 是否参与。

建议新增：

- `ContextPackageInspector.to_behavior_node(package) -> BehaviorDecisionNode`

这个可以先放在 `backend/orchestration/behavior_trace.py`，不必急着进入 context_management 包内部。

### 5.6 给 QueryRuntime 增加 live 行为事件

当前 SSE 已有：

- `context_management`
- `memory_context`
- `prompt_manifest`
- `tool_start`
- `tool_end`
- `worker_start`
- `worker_end`
- `done`

建议新增：

- `behavior_trace`
- `behavior_decision`

第一阶段在 planner 后发一次 `behavior_trace`，后续在 context、prompt、tool contract、output 之后补充节点状态。

这样前端不用从一堆低层事件中猜测行为链。

## 6. API 规划

新增路由文件：

- `backend/api/orchestration.py`

建议接口：

- `GET /api/orchestration/catalog`
  返回 skills、tools、permission modes、可用 route、worker route、context section 类型。

- `POST /api/orchestration/dry-run`
  输入 session_id、message、可选 behavior_policy_id，返回 `BehaviorDecisionTrace`、`PromptManifest` 预览、`ContractPreview`。

- `GET /api/orchestration/sessions/{session_id}/last-trace`
  返回当前会话最近一轮 live behavior trace。

- `GET /api/orchestration/runs/{run_id}/turns/{turn_id}/behavior`
  从测试 artifact 读取并转换成 behavior trace。

- `POST /api/orchestration/policies/validate`
  校验策略草案，不执行。

- `POST /api/orchestration/policies/dry-run`
  用策略草案推演同一条输入的不同结果。

暂时不建议开放：

- 持久化策略发布。
- 前端强制 route。
- 前端强制开放工具。
- 前端绕过 context budget。

这些要等 read-only 和 dry-run 稳定后再做。

## 7. 前端页面规划

编排系统建议拆成 5 个页签。

### 7.1 行为总览

展示一次请求的行为图：

- 输入。
- 记忆意图。
- 任务理解。
- follow-up / continuation。
- 执行模式。
- skill policy。
- context package。
- capability dispatch。
- contract / permission。
- model / worker / tool。
- output boundary。
- persistence。

节点可以点击，右侧显示“为什么做这个决策”。

### 7.2 路由策略

展示 route 相关信息：

- 当前输入识别出的 structural signals。
- `route_hint` 到最终 route 的变化。
- `execution_posture`。
- `execution_mode`。
- bundle / explicit fanout 的拆分。
- follow-up 是否触发，来源是什么。

### 7.3 上下文策略

展示模型可见上下文：

- token pressure。
- 各 section 预算。
- 进入模型的内容摘要。
- 被丢弃的 section。
- 与记忆系统的跳转。
- 与 prompt manifest 的跳转。

### 7.4 能力与契约

展示 skills 和工具契约：

- active skill。
- skill 候选和选择原因。
- tool scope。
- 候选工具。
- contract preview。
- permission preview。
- worker selection。

这里和“操作系统”的区别是：操作系统展示能力定义，编排系统展示本轮为什么使用或不使用这些能力。

### 7.5 Dry-run 实验

用户输入一句话，选择 session，点击推演：

- 不调用模型。
- 不执行工具。
- 不写入记忆。
- 返回行为图、contract preview、prompt manifest 预览。

后续可以支持策略草案对比：

- 默认策略。
- 草案 A。
- 草案 B。

## 8. 分阶段实施计划

### 第一阶段：只读行为 trace

目标：把现有运行事实统一成 `BehaviorDecisionTrace`。

涉及文件：

- `backend/orchestration/behavior_models.py`
- `backend/orchestration/behavior_trace.py`
- `backend/api/orchestration.py`
- `frontend/src/components/workspace/views/OrchestrationView.tsx`
- `frontend/src/lib/api.ts`
- `frontend/src/lib/store/types.ts`

完成标准：

- 当前会话最近一轮能看到行为 trace。
- 测试 turn 能跳转到行为 trace。
- 每个节点有序号、中文名称、来源阶段、决策原因。
- 不改变 runtime 行为。

### 第二阶段：Skills 与契约解释

目标：把 skill 选择、tool scope、contract preview、permission preview 接入编排系统。

涉及文件：

- `backend/skill_system/policy.py`
- `backend/orchestration/contract_preview.py`
- `backend/orchestration/behavior_trace.py`
- `backend/tools/contracts.py`
- `backend/permissions/service.py`

完成标准：

- 可以看到 active skill 的选择原因。
- 可以看到候选 skill 和过滤原因。
- 可以看到候选工具是否被 scope、contract、permission 拦截。
- direct tool 执行失败时能定位到契约节点。

### 第三阶段：上下文策略可视化

目标：把 `ContextPackage` 解释成可读行为节点。

涉及文件：

- `backend/orchestration/behavior_trace.py`
- `backend/memory/facade.py`
- `backend/context_management/context_models.py`
- `frontend/src/components/workspace/views/OrchestrationView.tsx`

完成标准：

- 可以看到模型可见上下文 section。
- 可以看到 debug-only section。
- 可以看到被丢弃内容和 token 原因。
- 可以从上下文节点跳转到记忆系统对应页面。

### 第四阶段：Dry-run 推演

目标：输入一条消息后，在无副作用模式下推演行为链。

涉及文件：

- `backend/orchestration/behavior_dry_run.py`
- `backend/orchestration/contract_preview.py`
- `backend/api/orchestration.py`
- `frontend/src/components/workspace/views/OrchestrationView.tsx`

完成标准：

- dry-run 不调用模型。
- dry-run 不执行工具。
- dry-run 不写 session、任务、记忆。
- 返回 route、execution mode、skill、contract preview、context preview、prompt manifest 预览。

### 第五阶段：行为策略草案

目标：允许用户编辑策略草案，但只用于校验和 dry-run。

涉及文件：

- `backend/orchestration/behavior_policy.py`
- `backend/orchestration/behavior_models.py`
- `backend/api/orchestration.py`

完成标准：

- 策略草案可以校验。
- 策略草案可以参与 dry-run。
- 策略草案不能直接绕过 permission、contract、memory policy。
- 页面能对比默认策略和草案策略。

### 第六阶段：受控单轮 override

目标：允许当前 turn 使用临时行为策略，但必须后端重新裁决。

涉及文件：

- `backend/query/models.py`
- `backend/query/planner.py`
- `backend/query/runtime.py`
- `backend/orchestration/behavior_policy.py`

完成标准：

- `QueryRequest` 可携带 `behavior_policy_override`。
- override 默认只对当前 turn 生效。
- 后端校验失败时 fail closed。
- 所有 override 都进入 trace。

## 9. 关键边界

编排系统可以做：

- 解释 route、skill、context、contract、worker、output 的决策过程。
- 展示一次请求的行为图。
- 做无副作用 dry-run。
- 管理行为策略草案。
- 把问题跳转到测试系统、记忆系统、操作系统、Agent 系统。

编排系统不应该做：

- 直接管理长期记忆文件。
- 直接替代操作系统管理 tool/skill 定义。
- 直接替代 Agent 系统管理 worker 和 RAG 子 agent。
- 直接替代测试系统运行 profile。
- 绕过 permission 或 tool contract 强制调用工具。
- 在第一阶段改变 `QueryRuntime` 主链。

## 10. 验证策略

需要补充的测试：

- `backend/tests/orchestration_behavior_trace_regression.py`
  验证普通问答、memory route、tool route、worker route、bundle route 都能生成 trace。

- `backend/tests/orchestration_dry_run_regression.py`
  验证 dry-run 不写 session、不执行工具、不调用模型。

- `backend/tests/orchestration_contract_preview_regression.py`
  验证缺输入、缺 binding、scope 拒绝、permission 拒绝都能生成清晰 preview。

- `backend/tests/orchestration_skill_policy_trace_regression.py`
  验证 skill 候选、选择原因和 forbidden route 可以解释。

- `backend/tests/orchestration_context_trace_regression.py`
  验证 context package 的 selected/dropped/token accounting 可转换成行为节点。

前端验证：

- 编排系统页面能打开当前会话 trace。
- 测试系统 turn 能跳转到对应行为 trace。
- dry-run 页面输入后能展示完整行为图。
- contract preview 不会误导用户以为工具已执行。

## 11. 推荐先做的最小闭环

建议下一步先做第一到第四阶段的一条最小闭环：

1. 新增 `backend/orchestration` 模块和 `BehaviorDecisionTrace`。
2. 从 `QueryPlan`、`DispatchPlan`、`ContextPackage`、`PromptManifest` 生成 read-only trace。
3. 新增 `/api/orchestration/dry-run`。
4. 前端编排系统新增 “Dry-run 实验” 页签。
5. 测试系统 turn 可以跳到同一套行为图。

这样编排系统就会真正变成 Agent 开发平台的入口：既能读真实运行，也能在前端试探“如果用户这样说，Agent 会怎么行动”，而不污染主运行链。

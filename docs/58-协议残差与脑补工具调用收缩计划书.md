## 1. 计划定位

本计划只处理一个问题域：

`模型为什么会脑补工具调用/过程协议，并且这些协议残差为什么还能进入可见答案`

它不再处理：

1. 工具是否允许执行。
2. binding owner 是否能够跨轮续接。
3. RAG/PDF 的 canonical evidence 是否完整。

这些问题分别由 `docs/55`、`docs/56`、`docs/57` 负责。

本计划只处理以下责任链：

1. `prompt-visible protocol -> model inner narration`
2. `tool execution stream -> visible stream delta`
3. `tool result / process text -> final answer gate`
4. `tool invocation truth -> answer provenance verification`

换句话说，本计划修的不是“工具能力”，而是：

`协议语言如何从系统内部泄漏到模型表达层，并最终污染用户答案`

---

## 2. 当前现状

经过 `docs/55` 到 `docs/57` 的连续修复，当前系统已经具备：

1. 工具契约化执行边界。
2. 高污染工具 fail-closed。
3. 输出层基础收口能力。
4. follow-up / binding 连续性的主计划。

但“协议残差”和“脑补工具调用”仍未被结构性清除。

当前的实际现象不是简单的 `<tool_call>` 直出，而是更隐蔽的几类污染：

1. 模型输出“我来检索一下”“我先查看目录结构”“我将调用某工具”这类过程化语言。
2. 工具调用没有真正展示给用户，但模型在自然语言里模拟了调用过程。
3. 输出边界虽然能清洗明显协议片段，但仍依赖字符串正则兜底。
4. 流式阶段可能先把过程句吐给用户，最终答案阶段再尝试收口。

这说明当前系统已经不再是“裸协议片段直出”的早期问题，而是更深入的一类问题：

`模型已经把工具协议内化为自己的表达方式`

## 3. 技术源报告

## 3.1 当前代码中的真实污染源

基于现有代码，协议残差并不是单点问题，而是四层叠加。

### A. 技能块仍向模型暴露开发协议

文件：

- `backend/skill_system/registry.py`
- `backend/query/runtime.py`

当前 `QueryRuntime.build_system_prompt_for_session()` 会把：

- `self.skill_registry.format_active_skill_block(active_skill)`

直接注入系统提示词。

而 `SkillRegistry.format_active_skill_block()` 当前仍会暴露如下字段：

1. `Preferred Route`
2. `Activation Policy`
3. `Context Mode`
4. `Route Authority`
5. `Allowed Tools`
6. `Routing Hints`

这些字段对开发者有用，但对模型来说，它们会被误学成“对话中可引用的内部协议语言”。

这类信息不属于任务知识，也不属于用户上下文，而是：

`运行时控制面元信息`

把这类信息直接喂给模型，会自然诱发以下表达：

1. “我来调用……”
2. “我先检索……”
3. “根据工具返回……”
4. “接下来我将……”

### B. 输出边界主要仍是字符串清洗

文件：

- `backend/query/output_boundary.py`

当前 `AssistantOutputBoundary` 的核心能力仍然是：

1. 检测 `</think>`、`<tool_call>`、`**工具调用:**`
2. 删除参数 JSON
3. 删除部分过程化句子
4. 删除部分 pseudo tool call

这层已经比之前强很多，但它本质上仍然是：

`文本后处理`

也就是说：

1. 它依赖已知污染模式。
2. 它不能阻止模型产生过程化文本。
3. 它只能在文本已经生成后尝试清洗。

这符合 `docs/56` 的职责边界，但不构成协议残差问题的最终解。

### C. 流式展示先于最终答案仲裁

文件：

- `backend/query/runtime.py`
- `backend/query/output_boundary.py`

当前执行链中，`ingest_stream_text()` 会实时生成 `visible_delta`。

这意味着：

1. 模型在流式阶段输出的过程化句子，可能先进入用户可见层。
2. 即使最终 `build_response()`、`output_classifier`、`answer_finalizer` 把正式答案收干净，流式阶段也可能已经暴露过污染文本。

这不是“最终答案分类错误”，而是：

`展示层和最终答案层尚未彻底隔离`

### D. 分类器仍允许部分半成品晋升为 answer candidate

文件：

- `backend/query/output_classifier.py`

当前逻辑是：

1. 若文本能抽出显式结论，优先作为 `answer_candidate`
2. 若是工具可见 summary，则作为 `tool_visible_summary`
3. 若像 `progress_text`，则拒收
4. 否则默认仍可能作为 `answer_candidate`

这让系统具备“有内容就尽量给答案”的韧性，但同时也意味着：

1. 只要一段过程化文本没有命中现有规则，
2. 它就可能被当成某种“可见答案候选”

所以当前系统并不是完全阻断协议残差，而是：

`尽量识别并压制协议残差`

## 3.2 对照本地设计原则后的正式判断

根据仓库已有原则，这个问题并不应该继续靠提示词修辞或词表补丁解决。

### 来自 `docs/设计原则/05-对话循环.md`

可直接抽出的约束：

1. 流式产出和最终结果不是同一层。
2. 可恢复的中间状态应先暂扣，再决定是否暴露。
3. 工具循环属于控制流，不应默认等于用户可见流。

对本问题的含义：

1. “模型正在检索/准备调用工具”的过程，不该直接进入用户可见层。
2. 流式 delta 不能天然视为正式答案。

### 来自 `docs/设计原则/06-上下文管理.md`

可直接抽出的约束：

1. 不同层级的上下文应按用途区分。
2. 模型上下文、恢复上下文、压缩上下文不应混层。

对本问题的含义：

1. skill runtime 元信息不应直接作为模型行为 prompt。
2. 工具协议不应以“人类可读说明”的方式暴露给主模型。

### 来自 `docs/设计原则/09-工具系统设计.md`

可直接抽出的约束：

1. 工具系统应 fail-closed。
2. 工具抽象需要安全默认值。
3. 并发安全、权限、安全边界都应在工具系统内部结构化表达，而不是靠模型自然语言推断。

对本问题的含义：

1. 工具调用应靠结构，不应靠模型“自述”。
2. 工具允许/不允许调用，不应在可见 prompt 中再讲一遍。

### 来自 `docs/设计原则/12-Agent-系统.md`

可直接抽出的约束：

1. 默认隔离、显式共享。
2. 子系统状态不能随意泄漏到主线程上下文。

对本问题的含义：

1. tool runtime / skill runtime 的控制面，不应泄漏到主模型表达层。
2. 工具事件和用户答案必须分层。

### 来自 `docs/设计原则/16-权限系统.md`

可直接抽出的约束：

1. 防御纵深。
2. 规则层、模式层、决策层应分离。
3. 不能把安全边界全压给单一层。

对本问题的含义：

1. 不能只靠 output boundary 正则兜底。
2. 必须同时修 prompt-visible 协议暴露、流式展示边界、最终答案 gate、结果验证。

## 4. 外部成熟方案对照

本节只对照与“协议残差 / 脑补工具调用”直接相关的成熟方法，不做泛泛框架综述。

## 4.1 OpenAI：结构化工具调用，不靠自然语言协议

OpenAI 在 Structured Outputs 中明确强调：

1. 通过 `strict: true` 让模型输出严格匹配开发者 schema。
2. function calling 应直接生成结构化工具调用，而不是靠自然语言描述调用过程。

对本项目可借鉴的核心点：

1. 工具调用的“真相”应该由结构化对象表达。
2. 用户答案与工具调用是不同输出通道。
3. 如果要给模型看工具定义，应给 schema 和真实约束，而不是控制面说明。

适配本仓库后的结论：

1. `Allowed Tools / Route Authority / Activation Policy` 这类说明不该进入模型可见层。
2. 最终答案层不能消费“模型口头描述的工具动作”。

参考：

- OpenAI, *Introducing Structured Outputs in the API*  
  https://openai.com/index/introducing-structured-outputs-in-the-api/

## 4.2 Anthropic：工具描述要明确、严格、面向行为

Anthropic 在工具设计实践中强调：

1. 工具描述和 spec 是改善 agent 行为的高杠杆点。
2. 输入输出要明确、严格、无歧义。
3. 错误返回也应该帮助 agent 学会更正确地用工具。

对本项目可借鉴的核心点：

1. 工具 prompt 应该是“如何正确调用工具”的行为约束。
2. 不应把内部路由/激活机制当成工具说明喂给模型。
3. 工具层最重要的是少而准的约束，不是更多控制面元信息。

适配本仓库后的结论：

1. skill block 应从“运行时控制元信息摘要”改成“最小必要能力说明”。
2. tool-visible 信息应面向模型行为，而不是面向开发者调试。

参考：

- Anthropic, *Writing effective tools for AI agents*  
  https://www.anthropic.com/engineering/writing-tools-for-agents

## 4.3 LangChain：把 model context、tool context、life-cycle context 分开

LangChain 当前对 agent context engineering 的分层非常清楚：

1. `Model Context`
2. `Tool Context`
3. `Life-cycle Context`

并明确说明 guardrails、summarization、logging 属于 life-cycle context。

对本项目可借鉴的核心点：

1. 工具协议不等于模型上下文。
2. 清洗、验证、展示控制应该属于 life-cycle 层。
3. 最终输出前的“after agent”校验是正式结构，不是字符串补丁。

适配本仓库后的结论：

1. 当前 `output_boundary` 只能算 life-cycle guardrail 的一部分。
2. 需要把“流式展示”和“正式答案”彻底分层。
3. 最终答案前应保留一次正式 gate，而不是只做 token 级清洗。

参考：

- LangChain, *Context engineering in agents*  
  https://docs.langchain.com/oss/javascript/langchain/context-engineering
- LangChain, *Guardrails*  
  https://docs.langchain.com/oss/python/langchain/guardrails

## 4.4 研究方向：减少工具幻觉与增加结果可验证性

近期研究方向也已经不再把问题理解成“模型偶尔说错一句”，而是：

1. 需要降低 tool hallucination。
2. 需要给工具结果提供可验证痕迹，而不是只靠模型口头声称“我调用过工具”。

对本项目可借鉴的核心点：

1. “是否调用过工具”应有 receipt / trace / structured event 级证据。
2. 最终答案如引用了工具动作，应可追溯到真实执行记录，而不是只信模型口述。

适配本仓库后的结论：

1. 未来应增加 answer provenance / tool receipt 验证，而不是只做文本净化。
2. 对高风险 route，应优先展示基于真实 tool event 的结论，而非模型叙述。

参考：

- Xu et al., *Reducing Tool Hallucination via Reliability Alignment*  
  https://arxiv.org/abs/2412.04141
- Basu, *Tool Receipts, Not Zero-Knowledge Proofs: Practical Hallucination Detection for AI Agents*  
  https://arxiv.org/abs/2603.10060

## 5. 方案选择与取舍

## 5.1 应借鉴什么

本仓库应明确借鉴以下成熟做法：

1. 借鉴 OpenAI 的结构化输出思路：
   工具调用真相由结构承载，不由自然语言承载。
2. 借鉴 Anthropic 的工具写法：
   工具/技能说明要最小必要、无歧义、面向行为。
3. 借鉴 LangChain 的分层：
   把 model context、tool context、life-cycle context 严格拆开。
4. 借鉴研究方向的 verification 思路：
   最终答案与真实 tool event 建立 receipt 关系。

## 5.2 不应照搬什么

本仓库不应直接照搬以下东西：

1. 不直接引入重型 framework runtime。
   当前系统已有自己的 `QueryRuntime / ToolContractGate / OutputBoundary` 主链，没必要重做 agent 框架。
2. 不把所有输出问题都交给 model-based safety model。
   这会让简单问题成本过高，也会再次把结构问题推给模型。
3. 不重新把工具层/理解层打开重构。
   `docs/55` 和 `docs/57` 已经明确了当前边界，不能回摆。

## 5.3 推荐方向

推荐方向不是“再加几条正则”，而是：

`收缩 prompt-visible 协议 + 分离流式展示通道 + 建立 final answer gate + 增加 tool receipt 验证`

这是最符合当前仓库结构、也最符合外部成熟方法的方向。

## 6. 目标设计

目标链路应调整为：

`tool/runtime control plane -> structured tool events -> guarded visible streaming -> final answer synthesis/gate -> provenance-checked visible answer`

核心设计回答如下。

### 6.1 谁拥有工具调用真相

正式答案：

1. 由结构化 tool event 拥有。
2. 不由模型自然语言拥有。

### 6.2 哪一层可以恢复或展示工具过程

正式答案：

1. life-cycle 层可以保留 tool event 供 runtime 使用。
2. 模型上下文不能直接消费完整控制面协议。
3. 用户可见流不能直接消费未经 gate 的过程句。

### 6.3 弱信号允许做什么

正式答案：

1. 过程化文本、流式 delta、模型自述调用行为，最多只能作为污染信号。
2. 它们不能成为“已执行工具”的证据。
3. 它们不能单独晋升为最终答案。

### 6.4 输出层允许消费什么

正式答案：

1. 允许消费 canonical answer。
2. 允许消费 tool-visible summary。
3. 允许消费经过 finalizer/gate 的 evidence-based answer。
4. 不允许直接消费 raw tool protocol。
5. 不允许直接消费“模型正在调用工具”的过程句。

## 7. 分阶段优化路线

## Phase 1：Prompt-visible 协议收缩

目标：

1. 清除模型可见层中的开发协议残差。
2. skill block 改成最小必要能力说明。
3. 不再向模型暴露 route / authority / activation 控制面术语。

涉及文件：

1. `backend/skill_system/registry.py`
2. `backend/query/runtime.py`
3. 如有必要：`backend/query/prompt_builder.py`

具体动作：

1. 从 `format_active_skill_block()` 中移除：
   - `Preferred Route`
   - `Activation Policy`
   - `Context Mode`
   - `Route Authority`
   - `Allowed Tools`
   - `Routing Hints`
2. skill block 改为仅保留：
   - 能力名称
   - 能力用途
   - 成功输出形式
   - 明确禁止暴露内部协议
3. 若某些字段仅供 runtime 使用，则转移到非模型可见 trace 或 state 中。

完成标准：

1. 系统 prompt 中不再出现控制面协议字段。
2. 模型在首轮回答中主动“口头调用工具”的概率显著下降。

## Phase 2：流式展示与最终答案分层

目标：

1. 未经 gate 的过程句不再直接进入用户流式展示。
2. 将“流式可见文本”与“正式答案候选”彻底区分。

涉及文件：

1. `backend/query/runtime.py`
2. `backend/query/output_boundary.py`
3. 如有必要：`backend/query/output_models.py`

具体动作：

1. 引入 `stream_display_gate`：
   - 对过程化句、协议化句、tool-prelude 句先截流
   - 不在 token 层直接透出
2. `AssistantOutputBoundary.ingest_stream_text()` 不再天然等于用户可见流。
3. 允许：
   - 安全自然语言增量流
4. 拒绝：
   - “我来检索/我先查看/接下来我将调用”
   - 工具参数、tool names、route hints 的自然语言再现

完成标准：

1. 即使最终答案 fallback，流式阶段也不再明显暴露过程协议。

## Phase 3：最终答案 gate 正式化

目标：

1. 从“正则清洗”升级到“候选拒收 + provenance gate”。
2. 禁止模型自述工具调用充当答案。

涉及文件：

1. `backend/query/output_classifier.py`
2. `backend/query/output_boundary.py`
3. `backend/query/runtime.py`

具体动作：

1. 在 candidate classification 中新增：
   - `procedural_answer_candidate`
   - `tool_claim_without_receipt`
2. 若答案声称：
   - 已检索
   - 已读取 PDF
   - 已查看表格
   但当前 turn 中没有相应 tool receipt / canonical evidence，则拒收。
3. `fallback_reason` 新增：
   - `protocol_visible_pollution`
   - `tool_claim_unverified`

完成标准：

1. “口头说自己调用了工具”但没有真实记录的答案，不再被接受。

## Phase 4：Tool Receipt / Provenance 引入

目标：

1. 建立“答案引用工具行为”与“真实 tool event”之间的可验证关系。
2. 让 output layer 不再只看文本本身。

涉及文件：

1. `backend/query/runtime.py`
2. `backend/query/output_models.py`
3. `backend/query/answer_finalizer.py`
4. 如有必要：`backend/observability/*`

具体动作：

1. 为每轮执行维护最小 receipt：
   - tool name
   - execution status
   - canonical summary available / not available
   - evidence ref
2. final answer gate 在接受工具型答案前检查：
   - 是否存在 receipt
   - 是否存在 canonical evidence
3. 对 RAG/PDF route，优先基于 canonical evidence 做 finalization，而不是信任模型自述。

完成标准：

1. 答案若引用“已执行的工具动作”，系统可追溯到真实 receipt。

## Phase 5：长场景回归与收口

目标：

1. 验证协议残差显著下降。
2. 确保不破坏 `docs/55`、`docs/56`、`docs/57` 已收口的边界。

涉及文件：

1. `backend/tests/query_runtime_route_guard_regression.py`
2. 建议新增：`backend/tests/protocol_residue_regression.py`
3. `backend/harness/run.py` 长场景回归

完成标准：

1. 用户可见文本中明显过程协议继续下降。
2. 不新增 route 回摆。
3. 不通过更多启发式理解层补丁来“掩盖”协议问题。

## 9. 验证矩阵

## 9.1 局部回归

需要新增或覆盖：

1. 模型输出“我来检索/我先查看”时，流式层不再直接透出。
2. skill block 不再向模型暴露 `Allowed Tools / Route Authority`。
3. 没有 receipt 的工具口头声明不能成为最终答案。
4. 有真实 RAG/PDF evidence 时，答案优先来自 finalization，而非过程化流。

## 9.2 长场景回归

至少重跑：

1. `research-brief-and-document-resume`
2. `commerce-ops-data-live-switch`
3. 任一包含 PDF + RAG + structured 切换的长场景

重点指标：

1. `answer_leak_flags`
2. `answer_source`
3. `answer_fallback_reason`
4. `selected_channel`
5. 用户可见流中过程句比例

## 9.3 验收标准

本计划的验收标准不是“所有答案完美”，而是：

1. 模型自述工具调用显著减少。
2. 流式可见层不再明显先暴露过程协议。
3. 最终答案若引用工具行为，能追溯到真实 receipt。
4. output boundary 不再承担唯一防线职责。
5. 不通过增加更多词表启发式来伪修这个问题。

## 10. 明确禁止事项

本阶段明确禁止：

1. 继续在 `output_boundary.py` 里单纯追加更多正则作为主方案。
2. 为了压协议残差，重新把理解层做成大词表触发器。
3. 让模型自然语言“描述工具调用”成为执行凭证。
4. 在没有 receipt / canonical evidence 的情况下，把工具口头声明升级为可见答案。
5. 重新把 `docs/55` 的工具契约层或 `docs/57` 的 binding 连续性层打开回做。

---

## 11. 最终结论

当前“协议残差 / 脑补工具调用”问题的本质，不是模型单独不听话，而是：

`控制面协议暴露给模型 -> 模型把协议内化为表达方式 -> 流式层先泄漏 -> 输出层再用正则补救`

因此，本阶段最合理的路线不是：

- 再加提示词
- 再加几条清洗规则
- 再给理解层加词表

而是：

1. 收缩 prompt-visible 协议暴露
2. 分离流式展示通道与最终答案通道
3. 用 final answer gate 拒绝未验证的工具口头声明
4. 为工具行为引入 receipt / provenance

更准确地说：

1. `docs/55` 解决了“工具能不能正确执行”
2. `docs/56` 解决了“已有 evidence 如何稳定说出来”
3. `docs/57` 解决了“下一轮还能不能续上正确对象”
4. 本计划要解决的是“模型是否还在用内部协议说话”

这四层必须严格分开，不可混修。

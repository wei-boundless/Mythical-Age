# 测试系统分页重构计划书

日期：2026-04-26

状态：第一阶段已实施。前端测试系统已经拆成 `测试控制台 / 运行监控 / 轮次回放 / 问题诊断 / 产物档案` 五个页面，并把测试档位扩展为 `冒烟测试 / 稳定门禁 / 长场景核心 / 长场景批量 / 六十轮长跑`。系统保留原有启动、轮询、取消、链路跳转、记忆跳转和产物读取能力。

## 1. 当前问题

测试系统已经接了很多能力：

- 启动 `smoke / stable / long` 测试。
- 查看最近运行。
- 查看运行状态和日志。
- 查看失败 issue。
- 查看 turn 热力图和 turn 卡片。
- 从 turn 跳到系统框架图。
- 从 turn 跳到记忆系统。
- 查看 prompt manifest 和 memory trace 是否存在。

这些能力本身是对的，问题是它们都挤在 `TestSystemView.tsx` 一个页面里。现在用户进入测试系统时，会同时看到“运行入口、地图、日志、历史、turn、issue、debug 跳转”，页面变成了测试总仓库，而不是一个清晰的测试工作流。

所以这次重构目标不是删功能，而是分页：

```text
测试系统
-> 测试控制台
-> 运行监控
-> 轮次回放
-> 问题诊断
-> 产物档案
```

每个页面只解决一个问题。

## 2. 设计原则

### 测试控制和 Debug 分开

启动测试、取消测试、选择 profile 属于控制面。查看失败 turn、系统链路、记忆链路属于 debug 面。两者不能一直堆在同一屏。

### 运行对象是主线

测试系统的核心对象是 `run_id`。分页切换时，当前选中的 run 不能丢。所有页面都围绕同一个 active run 展示不同角度。

### Turn 是第二主线

长场景问题基本都落在 turn 上。`turn_id` 应该成为测试系统内部的可选焦点，后续可以从它跳到：

- 系统框架图
- 记忆系统
- Prompt 透明化面板
- 原始 artifact

### 图只放该放图的地方

现在测试系统图、系统框架图、turn 热力图混在一起。分页后：

- 测试控制台保留 profile 图。
- 轮次回放保留 turn 热力图。
- 系统运行关系继续交给系统框架图。

### 原始产物不要压过诊断视图

`runner.log`、`report.md`、`issues.json`、`trace.jsonl` 很重要，但它们应该放在“产物档案”，不要在首页占太多位置。

## 3. 推荐分页结构

### 3.1 测试控制台

页面目标：

```text
我要运行什么测试？
当前推荐跑哪一个？
能不能一键开始？
```

保留内容：

- 测试 profile 图。
- 冒烟测试、稳定门禁、长场景核心、长场景批量、六十轮长跑选择。
- `运行所选测试`。
- `直接运行六十轮`。
- 长场景核心、长场景批量、六十轮长跑二次确认。
- 当前 profile 的说明、风险、预计耗时。

新增建议：

- 把测试 profile 做成更明确的测试阶梯：
  `冒烟 -> 稳定门禁 -> 长场景核心 -> 长场景批量 -> 六十轮长跑`
- 每个 profile 显示：
  `用途 / 风险 / 预计耗时 / 适合什么时候跑`
- 首页只保留最近一次 run 的小状态条，不展开完整日志。

不放在这里：

- turn 卡片。
- issue 列表。
- runner.log 大段输出。
- report.md。

### 3.2 运行监控

页面目标：

```text
当前 run 跑到哪了？
有没有失败？
日志有没有异常？
```

保留内容：

- 当前运行状态。
- run id。
- profile。
- passed / failed / total。
- `runner.log` tail。
- 取消当前测试。
- 刷新当前 run。

新增建议：

- 做成运行仪表盘：
  - 左侧：当前 run 状态。
  - 中间：通过率、失败数、耗时。
  - 右侧：日志尾部。
- 对 running 状态做更明显的动态提示。
- 如果 run 结束，显示下一步按钮：
  - `查看轮次回放`
  - `查看问题诊断`
  - `查看产物档案`

不放在这里：

- 全部历史 run。
- 所有 turn 卡片。
- 复杂图。

### 3.3 轮次回放

页面目标：

```text
长场景每一轮发生了什么？
哪一轮失败？
这一轮可以跳到哪些系统？
```

保留内容：

- turn 热力图。
- turn 过滤：
  - 异常优先
  - 只看异常
  - 全部
- turn 卡片。
- `系统链路` 按钮。
- `记忆链路` 按钮。
- Prompt 是否记录。
- Memory 是否记录。

新增建议：

- turn 卡片分成两层：
  - 热力图：快速看全局。
  - 详情列表：看当前筛选结果。
- 点击 turn 后在页面右侧显示 turn 详情，而不是只能跳走。
- turn 详情里显示：
  - 状态
  - summary
  - artifact path
  - prompt manifest 状态
  - memory trace 状态
  - issue count
  - 快捷跳转：系统框架 / 记忆系统 / Prompt 来源 / 原始产物

不放在这里：

- 运行 profile 选择。
- 最近 run 长列表。
- 完整 report.md。

### 3.4 问题诊断

页面目标：

```text
失败原因是什么？
问题映射到了哪个系统节点？
我应该从哪里开始修？
```

保留内容：

- issues 列表。
- issue 到系统框架图的高亮跳转。
- 首个异常入口。
- `查看首个异常链路`。
- `查看异常记忆`。

新增建议：

- issue 按严重度/类型分组：
  - 执行链问题
  - follow-up 问题
  - 记忆问题
  - prompt 问题
  - 工具问题
  - 检索问题
- 每个 issue 卡片显示：
  - title
  - summary
  - mapped nodes
  - mapped edges
  - affected turn
  - 建议入口
- 顶部做“首个异常焦点”：
  - Turn N
  - 问题摘要
  - 一键去系统框架
  - 一键去记忆系统

不放在这里：

- 测试启动按钮。
- 测试地图。
- 全部日志。

### 3.5 产物档案

页面目标：

```text
这次测试生成了哪些文件？
我要看原始报告、trace、issues 或 run_result。
```

保留内容：

- 最近运行列表。
- active run 切换。
- run_result。
- issues。
- report。
- trace tail。
- output_dir。
- log_path。

新增建议：

- 左侧：run 列表。
- 右侧：产物阅读器。
- 产物切换：
  - Summary
  - Report
  - Issues JSON
  - Trace Tail
  - Runner Log
  - Run Result
- 每个产物都显示路径，方便和本地文件对应。

不放在这里：

- 测试启动主按钮。
- turn 大量卡片。

## 4. 页面间流转

推荐固定流转：

```text
测试控制台
-> 启动 run
-> 自动进入运行监控
-> run 完成后进入轮次回放或问题诊断
-> 从 turn/issue 跳到系统框架或记忆系统
-> 回到测试系统时保留 active run
```

页面跳转不应该丢状态：

- `activeRunId` 保留。
- `selectedProfile` 保留。
- `turnDisplayMode` 保留。
- 后续可以加 `selectedTurnId`。
- 后续可以加 `activeTestPage`。

## 5. 前端状态设计

当前 `TestSystemView.tsx` 里的状态可以继续保留，但建议增加：

```ts
type TestSystemPage =
  | "control"
  | "monitor"
  | "turns"
  | "issues"
  | "artifacts";
```

新增状态：

```ts
const [activePage, setActivePage] = useState<TestSystemPage>("control");
const [selectedTurnId, setSelectedTurnId] = useState("");
const [artifactView, setArtifactView] = useState<"summary" | "report" | "issues" | "trace" | "log" | "result">("summary");
```

后续如果测试系统越来越大，可以把这些状态提升到 store，这样从系统框架和记忆系统返回时也能恢复原页面。

第一版可以先放在 `TestSystemView.tsx` 内部，风险更低。

## 6. 组件拆分建议

当前 `TestSystemView.tsx` 已经承载太多东西。建议第一版先在同文件内拆内部组件，稳定后再拆文件。

目标结构：

```text
TestSystemView.tsx
  TestSystemTabs

  TestMonitorPage
  TestTurnsPage
```

## 7. 后端是否需要改

第一阶段不需要改后端。

现有 API 已经够支撑分页：

- `GET /experiments/profiles`
- `GET /experiments/runs`
- `POST /experiments/runs`
- `GET /experiments/runs/{run_id}`
- `GET /experiments/runs/{run_id}/artifacts`
- `GET /experiments/runs/{run_id}/turns`
- `GET /experiments/runs/{run_id}/graph-overlay`
- `GET /experiments/runs/{run_id}/turns/{turn_id}/graph-overlay`
- `GET /experiments/runs/{run_id}/turns/{turn_id}/prompt-manifest`
- `GET /experiments/runs/{run_id}/turns/{turn_id}/memory-trace`
- `POST /experiments/runs/{run_id}/cancel`

第二阶段可以考虑新增更适合前端的聚合 API：

```text
GET /api/experiments/runs/{run_id}/debug-summary
```

它可以一次返回：

- run summary
- first problem turn
- issue groups
- prompt/memory coverage
- recommended next action

但这不是第一阶段必须。

## 8. 视觉方向

测试系统的视觉不应该像普通后台表格，而应该像一个“试验控制室”。

推荐风格：

- 控制台：清晰、有仪表感，按钮明确。
- 运行监控：像实时状态面板，日志像黑匣子。
- 轮次回放：像时间轴和热力扫描。
- 问题诊断：像事故分析板，突出首个异常和系统定位。
- 产物档案：像证据柜，强调文件路径和原始记录。

每页都可以共用测试系统的基础色，但主视觉不同，避免又变成换了 tab 的同一堆卡片。

## 9. 实施顺序

### 第一阶段：只做前端分页

实施状态：已完成。

目标：降低页面复杂度，不改后端。

要做：

- 在测试系统顶部加入分页 tabs。
- 默认进入 `测试控制台`。
- 把现有模块按页面迁移。
- 保留所有现有按钮和跳转能力。
- run 完成后可以提示进入 `轮次回放` 或 `问题诊断`。

验收：

- 冒烟、稳定、长场景核心、长场景批量、六十轮长跑仍能启动。
- running 状态仍能轮询。
- 最近 run 仍能切换。
- turn 仍能跳系统框架和记忆系统。
- issue 仍能跳系统框架。
- `npm run build` 通过。

### 第二阶段：增强轮次详情

目标：turn 不再只是卡片列表，而是可选中查看。

要做：

- 增加 `selectedTurnId`。
- 点击 turn 先在当前页显示详情。
- 详情区提供跳转按钮。
- 显示 prompt manifest 和 memory trace 覆盖状态。

验收：

- 不跳系统框架也能理解 turn 的基本情况。
- 需要深挖时再跳系统框架或记忆系统。

### 第三阶段：问题诊断分组

目标：issue 不再只是列表，而是可操作诊断板。

要做：

- 按 issue 类型或 graph refs 分组。
- 顶部显示首个异常焦点。
- 每个 issue 显示推荐入口。
- 和系统框架 overlay 保持联动。

验收：

- 用户能从问题页判断先修哪条链路。
- 每个 issue 都能跳到系统图。

### 第四阶段：产物档案阅读器

目标：把原始报告和原始 JSON 收进一个专门页面。

要做：

- run 列表和产物阅读器并排。
- 支持 report / issues / trace / log / result 切换。
- 保留 artifact path。

验收：

- 不需要打开 IDE 也能读主要测试产物。
- 需要本地文件时能看到准确路径。

## 11. 风险控制

### 不要一次拆太碎

第一版可以只在 `TestSystemView.tsx` 内部拆逻辑，不急着拆文件。否则会同时引入布局变化和模块移动风险。

### 不要改变测试 runner

分页只是前端信息架构变化，不应该影响 `experiment_runner` 和后端 API。

### 不要丢 active run

分页后最容易出的问题是切页丢运行对象。实现时必须保证所有页面共享同一个 `activeRunId`。

### 不要把原始产物藏起来

页面变清爽后，原始 report、trace、issues 仍然要能找到，只是放到“产物档案”。

## 12. 推荐下一步

建议先实施第一阶段：

```text
TestSystemView 加分页 tabs
-> 测试控制台
-> 运行监控
-> 轮次回放
-> 问题诊断
-> 产物档案
```

这一步不改后端，能最快解决“页面杂乱”的问题。等分页稳定后，再继续做 turn 详情和 issue 分组。

# Git版本管理与工具包配置升级计划书

日期：2026-05-30

## 目标

为项目补齐成熟 coding agent 需要的 Git 版本管理能力，并将现有“用途分类”升级为“工具包管理 + 单工具执行”的两级配置模型。

核心边界：

- 工具包由现有用途分类升级而来，只用于注册表管理、前端筛选和 agent profile 配置。
- prompt、模型可见工具、tool calling 仍只暴露单个工具 schema。
- 最终执行权限仍由 operation id 和 OperationGate 裁决。
- Git 写操作必须是结构化工具，不通过 shell 参数拼接绕过权限。

## 成熟架构标准

成熟 coding agent 的版本管理能力至少应包含：

- 只读观察：status、diff、log、show、branch list。
- 受控写入：stage 指定文件、unstage 指定文件、commit staged、create branch、restore 指定路径。
- 远端操作：push 作为高风险独立能力，默认不进入主 agent。
- 提交前能区分用户已有改动和本轮 agent 改动，不能默认 `git add .`。
- 危险操作默认禁止：`reset --hard`、`clean -fd`、force push、无路径 restore。
- 每个工具有结构化入参、operation id、风险标签、权限门和测试。

## 目标 authority chain

```text
ToolDefinition / ToolPackageDefinition
-> AgentRuntimeProfile package selection
-> resolved operation scope
-> runtime assembly tool projection
-> OperationGate authorization
-> structured tool execution
-> event / test observation
```

职责划分：

- ToolDefinition：定义单工具 schema、operation id、风险属性、模型可见方式。
- ToolPackageDefinition：定义管理分组，不拥有执行权限。
- AgentRuntimeProfile：声明选中哪些包、包内排除项、额外单工具、阻断项。
- Runtime assembly：只消费解析后的 operation scope。
- OperationGate：唯一执行授权点。
- Git tools：只执行明确动作，不解释用户意图，不自行扩大范围。

## 后端改动

### 1. Git 工具补齐

文件：

- `backend/capability_system/units/tools/git_tools.py`
- `backend/capability_system/tool_definitions.py`
- `backend/capability_system/operation_registry.py`
- `backend/permissions/resource_scope_mapping.py`
- `backend/api/orchestration_catalog.py`

新增工具：

- `git_branch_list` / `op.git_branch_list`：只读。
- `git_branch_create` / `op.git_branch_create`：创建分支，不自动切换，默认需要审批。
- `git_stage` / `op.git_stage`：只 stage 指定路径，禁止空路径和 `.`。
- `git_unstage` / `op.git_unstage`：只 unstage 指定路径。
- `git_commit` / `op.git_commit`：只提交 staged 内容，提交前校验 staged diff 非空。
- `git_restore` / `op.git_restore`：只恢复指定路径，禁止无路径恢复。
- `git_push` / `op.git_push`：远端操作，默认不启用，禁止 force。

### 2. 用途分类升级为工具包注册模型

新增文件：

- `backend/capability_system/tool_packages.py`

模型：

```python
ToolPackageDefinition(
    package_id: str,
    title: str,
    description: str,
    category: str,
    operation_ids: tuple[str, ...],
    risk_level: str,
    managed: bool,
    tags: tuple[str, ...],
)
```

现有 `TOOL_TYPE_OPTIONS` 不再只是工具标签，而升级为工具包体系的管理分类。每个工具包必须归属一个用途分类；前端原有“用途分类”筛选升级为“工具包管理”入口。

默认工具包：

- `pkg.filesystem.read`
- `pkg.filesystem.write`
- `pkg.search.local`
- `pkg.git.read`
- `pkg.git.write`
- `pkg.git.remote`
- `pkg.web`
- `pkg.memory`
- `pkg.agent`
- `pkg.execution`
- `pkg.multimodal`
- `pkg.mcp.local`

`pkg.git.remote` 默认不进入主 agent。

用途分类与工具包关系：

- `实时查询`：`pkg.web`
- `本地文件`：`pkg.filesystem.read`、`pkg.filesystem.write`、`pkg.search.local`
- `文档数据`：`pkg.mcp.local`
- `知识检索`：`pkg.memory`
- `系统执行`：`pkg.execution`
- `多模态处理`：`pkg.multimodal`
- `版本控制`：`pkg.git.read`、`pkg.git.write`、`pkg.git.remote`
- `通用能力`：`pkg.agent`

因此 `TOOL_TYPE_OPTIONS` 需要新增 `版本控制`，Git 工具默认归入 `版本控制`。

### 3. 工具注册表输出升级

文件：

- `backend/capability_system/tool_definitions.py`
- `backend/capability_system/tool_registry.py`
- `backend/capability_system/catalog.py`
- `backend/capability_system/supply.py`
- `frontend/src/lib/api.ts`

注册表输出增加：

```json
{
  "tools": [],
  "tool_packages": [],
  "default_library": []
}
```

其中：

- `tools` 是单工具。
- `tool_packages` 用于管理和配置。
- `default_library` 是未被工具包覆盖或需要单独显示的工具集合。
- `operation_metadata.tool_type` 保持存在，但其语义升级为“工具包管理分类”。

### 4. Agent profile 两级配置

文件：

- `backend/agent_system/profiles/runtime_profile_models.py`
- `backend/agent_system/profiles/runtime_profile_registry.py`
- `backend/task_system/services/assembly_builder.py`
- `backend/api/orchestration_catalog.py`

新增配置：

```python
allowed_tool_packages: tuple[ToolPackageSelection, ...]
extra_allowed_operations: tuple[str, ...]
blocked_operations: tuple[str, ...]
```

解析规则：

```text
selected package operations
+ extra_allowed_operations
+ legacy allowed_operations
- package excluded_operations
- blocked_operations
= final allowed_operations
```

说明：

- `allowed_operations` 继续作为显式单工具配置入口，避免破坏已有 profile。
- `final_allowed_operations` 写入 `to_dict()`，供前端和诊断展示。
- runtime 消费应使用解析后的操作集合。

### 5. API 输出

文件：

- `backend/api/capability_system.py`
- `backend/api/orchestration_catalog.py`

输出：

- 工具包列表。
- 包内工具映射。
- 默认库工具。
- agent 当前选择。
- 解析后的最终 operation scope。

## 前端改动

文件：

- `frontend/src/components/workspace/views/capability-system/CapabilitySystemWorkbench.tsx`
- `frontend/src/lib/api.ts`
- 如 agent 配置页存在独立组件，则同步更新该页。

优化方向：

- 将现有“工具类型/用途分类”升级为工具包管理体系。
- 保留风险、边界筛选。
- 在“执行工具”页增加“工具包”视图：先选用途分类，再选工具包，再看包内单工具。
- 详情区仍以单工具为主，显示所属包、风险、operation id、执行契约。
- agent 配置页支持：
  - 整包启用。
  - 展开包内工具单独勾选/排除。
  - 从默认库额外选择单工具。
  - blocked 操作优先级提示。

UI 约束：

- 不把不同层级混成一页；总览、工具包、单工具详情层级清晰。
- 不在页面副标题写开发式说明。
- 使用紧凑、可扫描的控制台风格，不做营销页。

## 测试计划

新增或更新：

- `backend/tests/git_tools_regression.py`
- `backend/tests/tool_package_registry_regression.py`

## 遗漏检查清单

- [ ] Git 工具是否全部有 ToolDefinition。
- [ ] Git operation 是否全部进入 OperationRegistry。
- [ ] ResourceScopeMapping 是否能从 operation 映射到工具名。
- [ ] 默认主 agent 是否启用 git read/write 但不启用 push。
- [ ] push 是否保持独立高风险能力。
- [ ] 工具包是否只影响配置，不进入 prompt。
- [ ] runtime assembly 是否使用 final operation scope。
- [ ] 前端 catalog 类型是否包含 tool_packages/default_library。
- [ ] 工具页是否将原用途分类升级为工具包管理体系，并保留风险/边界筛选。
- [ ] 测试是否使用真实临时 Git 仓库，而不是 mock 核心 Git 行为。

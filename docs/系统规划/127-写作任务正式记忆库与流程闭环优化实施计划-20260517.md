# 写作任务正式记忆库与流程闭环优化实施计划

## 1. 问题定义

当前写作长任务已经不是“能不能跑起来”的问题，而是两个更深的工程问题：

1. 记忆库的真实载体不清晰  
   现在同时存在 `md` 产物、WorkingMemory、TaskGraph memory repository/memory edge 三套表达。用户看到的是 `md`，运行时读到的是 WorkingMemory，图上配置的是 repository selector，三者还没有完全收拢成单一工程语义。

2. 写作流程的计划层和执行层没有完全分开  
   旧流程里 `volume_plan -> chapter_draft` 直接相连，导致分卷规划直接压给正文作者，缺少批次细纲这一层稳定执行蓝图。结果就是：
   - 正文作者拿不到足够稳定的当前批次输入。
   - 审核意见和原稿、章纲、提交事实之间的交接边界不够清楚。
   - baseline 观感上容易混入过程版本。

正确目标不是增加更多页面或更多提示词，而是建立：

```text
正式记忆库作为 source of truth
md 产物作为可审阅 artifact
写作流程按“计划 -> 执行 -> 审核 -> 提交 -> 冻结/增量”闭环运行
```

## 2. 代码事实

### 2.1 当前项目已经不是“纯 md 记忆”

从代码上看，项目已经明确在往正式记忆库走：

- `scripts/configure_writing_simple_novel.py`
  - 已有 `memory_repository` 资源节点。
  - 已有 `memory_read` / `memory_write_candidate` / `memory_commit` 边。
  - 已有 `baseline / mutable / artifact_index / issue_ledger` 分仓结构。
- `backend/orchestration/runtime_loop/runtime_assembly_builder.py`
  - 已把 memory edge 解析成 `repository / collection / selector / version_selector` 描述。
- `backend/orchestration/runtime_loop/context_packet_resolver.py`
  - 已把本轮上下文装配成 `memory_snapshot / artifact_context_packet / revision_packet`。

这说明架构方向已经明确：**记忆应当是正式仓库机制，不应把 `md` 文件本身当作正式记忆库。**

### 2.2 但运行时 selector 还没有完全落到正式仓库

`backend/memory_system/working_memory_service.py` 当前 `select_for_node()` 仍然主要基于：

- `task_run_id`
- `graph_id`
- `scope`
- `visibility`
- `kind`
- `status`

进行 WorkingMemory 过滤。

它还没有把 TaskGraph memory edge 上的这些信息真正做成运行时强约束：

- `repository`
- `collection`
- `selector`
- `version_selector`
- `clock/scope` 下的可见版本

所以当前状态要实事求是地表述为：

```text
结构层已经有正式记忆库模型
运行时仍以 WorkingMemory 选择为主
repository selector 还没有彻底成为后端硬执行语义
```

### 2.3 `md` 文件的正确角色

`md` 文件应该继续保留，但它的角色必须收紧为：

1. 人类可读产物
2. 审核对象
3. 交付对象
4. 供 artifact ref 指向的正文/设定文件

不应该把 `md` 文件直接当作：

1. 权限系统
2. 可见性系统
3. 版本裁决系统
4. selector 检索系统

因此本项目里正确的双层结构是：

```text
Formal Memory Repository
  负责：分仓、分区、record kind、提交状态、读取权限、版本可见性

Artifact MD Files
  负责：正文、世界观、章纲、审核报告、提交记录的人类可读落盘
```

## 3. 正式记忆库设计方向

## 3.1 结论

写作任务应采用：

**正式记忆库 + artifact md 双层机制**

而不是：

**纯 md 记忆库**

## 3.2 仓库分层

### A. baseline

只存 canon 主干，只读为主。

建议语义：

- `world`
- `outline`
- `characters`
- `relationships`

特点：

- 只允许 `baseline_memory_seed` 写入。
- 后续节点只能读，不能重定义。
- 内容必须是“单一最终版本”，不能混入候选稿历史。

### B. mutable

只存允许在长周期中逐卷演化的增量层。

建议语义：

- `strategy`
- `outline_delta`
- `character_weight`
- `future_chapter_outline`

特点：

- 只能由审核通过后的扩展提交节点写入。
- 不能覆盖 baseline，只能补充后续执行策略。

### C. artifact_index

负责索引候选稿、审核稿、提交稿、交付稿。

建议语义：

- `candidates`
- `reviews`
- `world_commits`
- `volume_plans`
- `chapter_outlines`
- `chapter_commits`
- `volume_commits`
- `delivery`

特点：

- 这里不是 canon 本体，而是任务流转索引层。
- 适合表达“当前卷计划”“当前批次章纲”“已提交章节引用”等可追踪引用。

### D. issue_ledger

负责问题账本。

建议语义：

- `world_issues`
- `chapter_issues`
- `volume_issues`
- `final_issues`

特点：

- 只记录问题、返修要求、处理状态。
- 不替代正文、不替代记忆主干。

## 3.3 计划层和执行层拆分

写作长任务最容易出问题的地方，就是把“计划”直接当成“执行输入”。

正确拆分应是：

### 计划层

- `baseline_memory`
- `volume_plan`
- `chapter_outline`

### 执行层

- `chapter_draft`
- `chapter_review`
- `memory_commit_chapter`

### 复盘增量层

- `volume_postmortem`
- `world_outline_extension_proposal`
- `extension_review`
- `extension_commit`

## 3.4 章纲冻结规则

为了满足“已写章节的大纲不再修改，未写章节还可整理”，记忆库规则应是：

1. 当前批次细纲在开写前属于候选/计划层。
2. 一旦该批次正文通过审核并提交，该批次细纲就冻结为历史依据。
3. 未开写批次的后续安排，仍可存在于未来章纲整理区，由后续批次重新整理。

这意味着系统里要区分两类章纲记录：

- `chapter_outline` / `future_chapter_outline`
- `frozen_chapter_outline`

## 4. 写作流程重构方向

## 4.1 当前应采用的主流程

```text
project_brief
-> world_design
-> world_review
-> memory_commit_world
-> baseline_memory_seed
-> volume_plan
-> chapter_outline
-> chapter_draft
-> chapter_review
-> memory_commit_chapter
-> chapter_progress_router
-> volume_review
-> volume_commit
-> volume_postmortem
-> world_outline_extension_proposal
-> extension_review
-> extension_commit
-> next_volume_router
-> final_assemble
-> final_review
-> memory_finalize
```

## 4.2 当前轮次必须落实的流程修正

### 修正 1：`volume_plan` 不能直接进入 `chapter_draft`

必须改成：

```text
volume_plan -> chapter_outline -> chapter_draft
```

### 修正 2：审核和提交链必须显式拿到本批章纲引用

至少要保证：

- `chapter_review_input` 能拿到 `chapter_outline_ref`
- `chapter_review` 结果能带出 `chapter_outline_ref`
- `memory_commit_chapter` 提交阶段能显式知道它提交的是哪一批章纲对应的正文

### 修正 3：批次循环入口必须回到 `chapter_outline`

因为下一批开写前，必须先生成或整理当前批次细纲。

所以循环入口应是：

- `chapter_progress_router -> chapter_outline`
- `next_volume_router -> chapter_outline`

而不是直接回 `chapter_draft`

### 修正 4：baseline 只保留定稿骨架

baseline 不能再表现成：

- 候选稿历史
- 审核措辞历史
- 多版本世界观混写

它必须只保留：

- 审核通过后的世界主干
- 大纲主干
- 冻结人物事实
- 冻结关系事实

## 5. 本轮实施范围

本轮先完成两类工作。

### 第一类：立即收口当前写作配置脚本

涉及文件：

- `scripts/configure_writing_simple_novel.py`

目标：

- 修正旧交接残留。
- 收紧 chapter outline 在主流程中的位置。
- 收紧 handoff/ref 契约。
- 修正 runtime loop 入口。

### 第二类：把正式记忆库方向固化成工程基线

涉及文件：

- 本计划文档

目标：

- 明确 `md` 不是正式记忆库本体。
- 明确正式记忆库分仓职责。
- 明确后续 selector 后端化是下一阶段必做项。

## 6. 后续阶段

本轮不会假装已经完成“正式 repository selector 后端化”，因为当前代码事实并没有做到。

下一阶段要真正落的，是：

1. 在运行时读路径上落实 `repository + collection + selector + version_selector`
2. 让 `memory_snapshot` 不再只是 WorkingMemory 选择结果，而是 repository 解析结果
3. 让“按卷写入、按卷读取、按批次读取”成为后端确定性能力，而不只是图上的描述

## 7. 完成标准

本轮完成后，至少要满足：

1. 写作任务图中不再存在 `volume_plan -> chapter_draft` 旧主线残留。
2. 批次循环入口明确回到 `chapter_outline`。
3. 审核、提交链条显式拿到当前批次细纲引用。
4. baseline 的角色被定义为“定稿骨架库”，不是过程文档拼接区。
5. 方案文档明确区分：
   - 正式记忆库
   - artifact md
   - 计划层
   - 执行层
   - 提交层

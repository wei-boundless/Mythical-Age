# RAG 系统重构计划书

日期：2026-05-10

## 目标

把当前检索增强生成系统从“可运行但边界松散、结构局部有效、策略分散”的状态，重构为一套职责清晰、路径完整、可观测、可扩展、可实测的正式 RAG 系统。

本次重构关注的是整套链路，而不是单点调参。

目标包括：

1. 梳理并固定 RAG 主链的完整结构：
   - 入口
   - 索引构建
   - Query Rewrite
   - Route Plan
   - Dense Retrieval
   - Lexical Retrieval
   - Retrieval Filter
   - Fusion / Coalesce
   - Rerank
   - Runtime 接线
   - Diagnostics / Benchmark
2. 把当前“知道 query 类型，但没有把它真正转成检索边界”的结构性问题修掉。
3. 明确各层的输入输出合同，避免 rewrite、route、retrieval、rerank、runtime 互相吞职责。
4. 把历史有效方法链中值得保留的部分纳入正式主链，不再依赖历史提交记忆。
5. 为后续中文检索增强、复杂 filter、collection 拆层、任务系统接线预留清晰落点。

---

## 一、现状总览

当前 RAG 系统并不是一条短链，而是已经具备了完整雏形，只是几个关键边界没有封严。

主链大致如下：

```text
User Query
  -> RetrievalService
  -> RAGQueryRouter.plan()
  -> QueryRewriter
  -> RoutePlan(selected_collections, query_type, rewritten_query)
  -> RetrievalBootstrapper.backend.retrieve()
  -> Dense Retrieval
  -> Lexical Retrieval
  -> Fusion
  -> Coalesce
  -> RetrievalService payload build
  -> Reranker
  -> Final top_k results
  -> Runtime / MCP / Task system consumer
```

从代码组织上，目前主要模块是：

- `backend/retrieval/service.py`
- `backend/capability_system/units/mcp/local/retrieval/router.py`
- `backend/capability_system/units/mcp/local/retrieval/query_rewriter.py`
- `backend/capability_system/units/mcp/local/retrieval/collections.py`
- `backend/retrieval_core/bootstrap.py`
- `backend/retrieval_core/llamaindex_backend.py`
- `backend/retrieval_core/lexical.py`
- `backend/capability_system/units/mcp/local/retrieval/reranker.py`
- `backend/capability_system/units/mcp/local/retrieval/benchmark_runner.py`

## 二、当前架构结论

### 1. 主链已经是 Hybrid RAG，而不是纯向量检索

当前 backend 已经明确实现：

```text
dense retrieval
+ lexical BM25 retrieval
+ application fusion
+ coalesce
+ rerank
```

这意味着当前系统不是“缺 RAG”，而是“已有 RAG，但边界和策略层级不清”。

### 2. 检索入口和索引构建已经分层

当前职责大体是：

- `RetrievalBootstrapper`
  - 负责 source discovery
  - 文档转换
  - normalized ingestion
  - indexable units 构建
  - 调 backend build

- `LlamaIndexRetrievalBackend`
  - 负责 dense index
  - lexical index
  - dense / lexical retrieve
  - fusion / coalesce

- `RetrievalService`
  - 负责 runtime-facing retrieval execution
  - payload 组装
  - diagnostics
  - rerank

这个分层方向是对的，不需要推倒重来。

### 3. Route 层过弱，是当前最大结构问题

`RAGQueryRouter.plan()` 当前主要决定：

- `rewritten_query`
- `query_type`
- `selected_collections`

但它没有把：

- `pdf_page`
- `document`
- `table`
- `memory`

这些 query 类型继续转成结构化 `retrieval_filter`。

结果就是：

```text
系统知道“这是 table 查询”
但真正执行检索时并不会优先过滤到 table
```

这会直接导致：

- 结构化查询检索边界过宽
- 长文档页码查询不够精
- PDF / image / table / text 混查
- rerank 负担过重

### 4. Collection 级边界过粗

当前默认 collection：

- `knowledge`
- `durable_memory`
- `session_memory`
- `benchmark`

但运行时通常只有：

- `knowledge`
- `durable_memory`

会参与 chat retrieval。

这意味着：

```text
knowledge collection 内部混合了 pdf / text / table / image / docx / pptx / xlsx
但 route 层没有继续下推 modality / doc kind filter
```

所以 collection 只是第一层粗过滤，不是足够的边界。

### 5. Dense 侧缺少 metadata filter 下推

当前 qdrant 查询只传：

- query vector
- collection name
- limit

没有明确 payload filter。

也就是说 dense 侧现在没有真正利用：

- modality
- page
- block_type
- source_path
- section_path
- doc_id

这些 metadata。

### 6. Lexical 侧已经比 dense 更容易接 filter

词法侧现在是应用层 BM25：

- 自建 index
- 自己分词
- 自己打分

这反而给了我们很好的改造空间，因为：

```text
lexical result 可以在装配 hit 前按 metadata 过滤
```

不像外部黑盒检索器那样难改。

### 7. Rerank 在结构上承担了过多补救责任

当前 rerank 既负责：

- 精排
- 结果纠偏
- 一部分 route 失败后的“补救”

这会让系统出现一种错觉：

```text
前面检索边界不够准，后面 rerank 还能救回来
```

但这会导致：

- 候选池越来越大
- rerank 成本上升
- 误召回越来越多
- top_k 对结果极度敏感

所以 rerank 应该是：

```text
精排层
不是主边界层
```

## 三、当前完整链路拆解

### 1. 数据构建链路

当前构建链路：

```text
discover_source_files
  -> DoclingConverter / parser pipeline
  -> NormalizedDocumentBuilder
  -> build_indexable_units
  -> backend.build_collection()
  -> dense index
  -> lexical index
```

优点：

- 已经有 normalized ingestion
- 已经有 block / object / page summary 粒度
- metadata 比较丰富

问题：

- 构建阶段虽然产出了丰富 metadata，但检索阶段没有充分使用

### 2. Query Rewrite 链路

当前 `QueryRewriter` 已经做了：

- query type detection
- rule-based keyword expansion
- retrieval hints 拼接

已有 query_type：

- `general`
- `memory`
- `table`
- `pdf_page`
- `document`

优点：

- 入口简单
- 工程上可控

问题：

- query_type 没有产出结构化 filter
- rule 与 retrieval 执行层之间没有正式 contract

### 3. Route Plan 链路

当前 `RoutePlan` 只有：

- `query`
- `rewritten_query`
- `selected_collections`
- `reason`
- `rewrite`

问题：

它太薄了，缺以下信息：

- retrieval filter
- retrieval policy
- modality preference
- page hint
- source hint
- result granularity preference

### 4. Dense Retrieval 链路

当前 dense 支持：

- qdrant backend
- local fallback
- payload 回填

优点：

- 已经是正式可用路径

问题：

- 无 payload filter
- 无 query-mode aware filter

### 5. Lexical Retrieval 链路

当前 lexical 支持：

- application BM25
- tokenizer 可切换空间
- 当前已接 `jieba + bigram fallback`

优点：

- 容易改造
- 对中文更可控

问题：

- 缺 structured lexical filter
- query_type 没被真正用进去

### 6. Fusion / Coalesce 链路

当前支持：

- dense / lexical fusion
- mode tracking
- result granularity annotation
- coalesce grouped merge

优点：

- 历史强结果主要就来自这条链

问题：

- 现在 fusion 同时承担了部分“检索边界收束”的职责
- 这不是最合理的位置

### 7. Runtime 接线

当前 `RetrievalService` 已经有：

- runtime execution result
- diagnostics
- degraded reason
- chain version

优点：

- 是一个比较好的 runtime-facing adapter

问题：

- 缺 route/filter 层完整诊断
- 缺 shadow compare / cutover 级治理能力

## 四、结构性问题清单

### 1. 缺 `RetrievalFilter` 合同

这是当前最大的遗漏。

当前系统有：

- query_type
- metadata
- query_mode

但没有中间合同把这些真正接起来。

### 2. 缺 `RetrievalPolicy` 合同

例如：

- candidate_top_k
- rerank_top_n
- rerank_candidate_pool
- modality preference
- coalesce policy
- memory inclusion policy

目前散在：

- router
- config
- service
- benchmark

需要集中。

### 3. collection / modality / granularity 三层边界没有区分开

现在容易混成一层。

正确层级应是：

```text
collection boundary
  -> modality boundary
    -> granularity boundary
```

### 4. query rewrite 与 route 之间没有正式协议

现在 rewrite 输出是弱结构：

- keywords
- applied_rules
- query_type

应升级成可供 route 消费的更强结构。

### 5. benchmark 只覆盖效果，不覆盖结构诊断

当前 benchmark 可以测：

- hit@k
- recall
- mrr

但还不够测：

- route 是否合理
- filter 是否生效
- dense / lexical 各自贡献
- rerank 是否在补救前链错误

## 五、重构方向

### 方向 A：建立 RetrievalPlan 正式合同

把当前 `RoutePlan` 升级成更完整的 `RetrievalPlan`：

```text
query
rewritten_query
query_type
selected_collections
retrieval_filter
retrieval_policy
reason
rewrite_diagnostics
```

### 方向 B：建立 RetrievalFilter 合同

建议至少支持：

```text
modality
doc_kind
page_hint
source_terms
section_terms
doc_id
block_type
prefer_memory
prefer_exact_source
```

### 方向 C：Dense / Lexical 同时消费 filter

不是只在一侧加。

- dense：
  - payload filter
  - qdrant filter 下推
- lexical：
  - candidate metadata filter
  - score 后过滤或 prefilter

### 方向 D：把 rerank 从“补边界”拉回“精排”

前面 route/filter 做准，
rerank 负责最后排序质量。

### 方向 E：把 diagnostics 升级为系统级可观测

每次 retrieval 至少应能看到：

- query_type
- selected_collections
- retrieval_filter
- candidate_top_k
- dense_hit_count
- lexical_hit_count
- fusion_hit_count
- rerank_applied_count
- final_result_count
- chain_version

## 六、目标架构

目标链路：

```text
User Query
  -> QueryRewrite
  -> RetrievalIntent
  -> RetrievalPlan
       - selected_collections
       - retrieval_filter
       - retrieval_policy
  -> Dense Retrieval (consume filter)
  -> Lexical Retrieval (consume filter)

  -> Coalesce
  -> Rerank
```

其中明确职责：

- `QueryRewrite`
  - 不做 collection 决策

- `Dense/Lexical Retrieval`
  - 只执行 plan
  - 不私自补 route 决策

- `Rerank`
  - 只负责精排

- `RetrievalService`
  - 负责 orchestration/runtime-facing 输出

## 十二、升级判断：不再把旧框架当作边界

前文的重构方向仍然成立，但它更像是“把当前 Hybrid RAG 链路修完整”。继续研究代码和成熟 RAG 做法后，可以得到一个更强判断：

```text
当前系统真正缺的不是某个更好的 tokenizer、top_k 或 rerank 参数，
而是一个能把用户问题转成检索计划的 Retrieval Planner。
```

因此后续实施不应该只恢复历史高命中配置，也不应该只沿着：

```text
dense + BM25 + fusion + rerank
```

继续叠功能。

新的目标应升级为：

```text
Intent Extraction
  -> Retrieval Planner
  -> Filtered Candidate Generation
  -> Candidate Graph / Fusion
  -> Rerank / Compression
  -> Evidence Packaging
  -> Runtime / Agent
```

也就是说，dense、BM25、jieba、Qdrant、rerank 都只是候选生成和排序执行器，不再是 RAG 系统的中心。中心必须是“检索规划”和“证据装配”。

## 十三、外部成熟做法对照

### 1. Qdrant 的正确用法不是只做向量库

Qdrant 官方文档把 metadata 称为 payload，并明确支持在搜索时使用 payload filter 限定返回点。也就是说，我们现在只传向量、不传 filter，是没有用满 Qdrant 的能力。

参考：

- [Qdrant Payload](https://qdrant.tech/documentation/manage-data/payload/)
- [Qdrant Query Points API](https://api.qdrant.tech/api-reference/search/query-points)
- [Qdrant Vector Search Filtering](https://qdrant.tech/articles/vector-search-filtering/)

对本项目的结论：

```text
doc_id / source_path / modality / page / block_type / unit_type
必须从 metadata 变成 query-time filter。
```

### 2. LlamaIndex 的启发是“路由器选择检索器”，不是让一个检索器包打天下

LlamaIndex 的 RouterRetriever / Router Query Engine 思路是：先根据 query 和候选工具 metadata 选择检索路径，再执行具体检索器。

参考：

- [LlamaIndex RouterRetriever](https://docs.llamaindex.ai/en/stable/api_reference/retrievers/router/)
- [LlamaIndex Routing](https://docs.llamaindex.ai/en/stable/module_guides/querying/router/)

对本项目的结论：

```text
RAGQueryRouter 不应该只返回 collections。
它应该返回 RetrievalPlan，包括 intent、filters、strategy、candidate budget、query variants。
```

### 3. Hybrid Search 的重点不是“dense 加 BM25”，而是多路候选的可解释融合

LlamaIndex + Qdrant 的 Hybrid Search 文档强调 dense 和 sparse 可以分别取 top_k，再通过 fusion 合并；Qdrant 也支持 sparse/dense hybrid。

参考：

- [LlamaIndex Qdrant Hybrid Search](https://docs.llamaindex.ai/en/stable/examples/vector_stores/qdrant_hybrid/)
- [LlamaIndex Hybrid RAG with Qdrant](https://docs.llamaindex.ai/en/stable/examples/vector_stores/Qdrant_hybrid_rag_multitenant_sharding/)

对本项目的结论：

```text
当前 application BM25 可以保留，但中期应该允许接入 Qdrant sparse vector / BM25 sparse。
短期先把 app BM25 作为 lexical executor。
中期再引入 vector-store-native sparse executor。
```

### 4. RAG-Fusion 的价值是多查询召回，不是盲目扩大 top_k

RAG-Fusion 的核心是生成多个 query variant，分别检索，再用 RRF 融合。它解决的是单一 query 表达不充分的问题。

参考：

- [RAG-Fusion paper](https://arxiv.org/abs/2402.03367)

对本项目的结论：

```text
QueryRewriter 不应只拼关键词。
它应为 RetrievalPlan 产出 query_variants：
- original
- normalized
- keyword-expanded
- entity-focused
- page/source-focused
```

但必须限制：query variant 只能服务候选召回，不能把猜测当事实写入 agent prompt。

### 5. 层级检索适合长文档，不适合继续把所有 chunk 混搜

RAPTOR 这类层级检索方法说明，长文档可以先检索摘要/父节点，再展开到子块或页块。这比直接把所有 block 混在一个候选池里更稳。

参考：

- [RAPTOR paper](https://arxiv.org/abs/2401.18059)

对本项目的结论：

```text
我们已有 document / page_summary / object_block / content_block。
缺的是 parent-child retrieval policy：
先找 document/page_summary，再按 doc_id/page 展开子块。
```

## 十四、目标架构：Retrieval Planner + Candidate Graph + Evidence Packager

### 1. 总体结构

```text
User Query
  -> RetrievalIntentExtractor
  -> RetrievalPlanner
      - selected_collections
      - filters
      - query_variants
      - retrieval_strategy
      - candidate_budget
      - expansion_policy
  -> CandidateGenerator
      - dense executor
      - lexical BM25 executor
      - future sparse executor
      - memory executor
  -> CandidateGraph
      - normalize
      - merge by doc/page/object/block
      - parent-child expansion
      - RRF / weighted fusion
  -> Reranker / Compressor
  -> EvidencePackager
  -> Runtime / MCP / Agent
```

### 2. 新核心对象

新增或升级以下合同：

```text
RetrievalIntent
RetrievalPlan
RetrievalFilter
RetrievalPolicy
QueryVariant
CandidateHit
CandidateGraph
EvidencePack
RetrievalTrace
```

其中最关键的是：

```text
RetrievalPlan = 当前检索决策的唯一真相源
```

### 3. RetrievalIntent

职责：

- 识别用户到底想找什么
- 不负责执行检索
- 不负责决定最终答案

建议字段：

```text
intent_type:
  - factual_lookup
  - document_lookup
  - page_grounded_lookup
  - table_lookup
  - memory_lookup
  - source_constrained_lookup
  - exploratory_lookup

entities:
  - 人名 / 项目名 / 文件名 / 术语

hints:
  - page
  - source
  - modality
  - date
  - task_domain
```

### 4. RetrievalFilter

短期必须支持：

```text
collections
modality_any
unit_type_any
block_type_any
page_eq
page_any
doc_id_any
source_path_contains_any
parser_backend_any
quality_flags_exclude_any
```

中期支持：

```text
task_domain
project_id
source_kind
created_at_range
updated_at_range
memory_status
memory_scope
```

注意：

```text
filter 是边界，不是排序偏好。
preference 是 policy，不应该塞进 filter。
```

### 5. RetrievalPolicy

职责：

- 决定怎么召回
- 决定候选预算
- 决定是否展开父子节点
- 决定是否做多查询

建议字段：

```text
strategy:
  - precise
  - balanced
  - broad
  - hierarchical

dense_top_k
lexical_top_k
sparse_top_k
fusion_top_k
rerank_top_n
query_variant_limit
parent_child_expansion
result_granularity
```

### 6. CandidateGraph

当前系统已有 coalesce，但还只是结果合并。新的 CandidateGraph 应该表达候选之间的结构关系：

```text
document
  -> page_summary
    -> content_block
    -> object_block
```

它要解决的问题：

- 同一页多个块如何合并
- 命中文档摘要后是否展开页块
- 命中表格对象后是否带上表格上下文
- 命中 memory 后是否需要排除过期 memory

### 7. EvidencePackager

RAG 最终给 agent 的不应该是一堆 raw hit，而是一组可读、可追踪、边界明确的证据包。

建议输出：

```text
EvidencePack:
  query
  answer_contract
  evidence_items:
    - text
    - source_ref
    - page
    - doc_id
    - retrieval_reason
    - confidence
    - allowed_use
  trace:
    - selected_collections
    - filters
    - query_variants
    - executor_counts
    - fusion_method
    - rerank_method
```

这对 agent 项目尤其重要：agent 看到的应该是“它可以用什么证据做当前任务”，不是看到“系统内部怎么开发的”。

## 十六、优化路线选择

### 方案 A：继续调 dense/BM25/rerank

优点：

- 改动小
- 快速复测

缺点：

- 结构问题仍在
- rerank 继续补前链错误
- 页码/表格/文件限定查询仍不稳

结论：

```text

```

### 方案 B：换一个更强检索框架

例如完全改用 LlamaIndex 原生 hybrid / router / query engine。

优点：

- 成熟能力多
- 生态完整

缺点：

- 本项目已有 normalized ingestion、runtime、MCP、agent context policy
- 全量替换会丢失大量本地控制面
- 任务系统和 agent 权限边界更难接

结论：

```text
不做全量替换，只借鉴 router/filter/hybrid 思路。
```

### 方案 C：保留本地控制面，重建 Planner 架构

优点：

- 最大化复用已有链路
- 能解释每次检索为什么这么做
- 能逐步接入 Qdrant filter、sparse、层级检索
- 更适合 agent runtime

缺点：

- 需要重构合同和诊断

结论：

```text
选方案 C。
```

## 二十、数据处理与 Chunk 切分重构

前面的 Retrieval Planner 只解决“该怎么搜”的问题，但 RAG 命中率还有一个更底层的前提：

```text
被索引的数据单元必须是稳定、可定位、可过滤、可展开的。
```

如果 chunk 切得不对，后面的 filter、fusion、rerank 都是在错误候选池上补救。因此本轮 RAG 重构必须把数据处理和 chunk 切分纳入主架构，而不是把它当成检索前的黑盒。

### 1. 当前真实链路

当前数据处理链路是：

```text
SourceFileRecord
  -> DoclingConverter / PDF parser / legacy parser
  -> ConversionResult / ConversionBlock
  -> NormalizedDocumentBuilder
  -> NormalizedBlock / NormalizedObjectRef
  -> clean_block / eligibility / index_profiles
  -> build_indexable_units
  -> IndexableUnit
  -> dense index + lexical index
```

相关文件：

- `backend/document_conversion/docling_converter.py`
- `backend/document_conversion/structured_text.py`
- `backend/capability_system/units/mcp/local/retrieval/parser_adapter.py`
- `backend/normalized_ingestion/builder.py`
- `backend/normalized_ingestion/eligibility.py`
- `backend/normalized_ingestion/chunking.py`
- `backend/normalized_ingestion/summaries.py`
- `backend/retrieval_core/bootstrap.py`
- `backend/retrieval_core/llamaindex_backend.py`

这个骨架是有价值的，不应该推倒。真正的问题是：

```text
有 normalized ingestion 的形，
但缺 ChunkingPolicy / IndexUnitPolicy 的正式治理。
```

### 2. 当前数据层优点

当前已经具备：

- `NormalizedDocument`
- `NormalizedBlock`
- `NormalizedObjectRef`
- `IndexableUnit`
- `content_block`
- `object_block`
- `page_summary`
- `parent_section`
- `document_summary`
- `index_profiles`
- `quality_flags`
- `section_path`
- `page`
- `block_type`

这说明当前不是“没有数据结构”，而是这些结构没有被策略化管理，也没有被后续检索充分消费。

### 3. 当前关键问题

#### 3.1 chunk 配置存在但没有真正接入

当前配置里有：

```text
rag_chunk_size = 500
rag_chunk_overlap = 60
```

但 `backend/normalized_ingestion/chunking.py` 使用的是硬编码阈值：

```text
MIN_TOKENS = 96
TARGET_TOKENS = 220
SOFT_MAX_TOKENS = 320
HARD_MAX_TOKENS = 420
```

这会导致：

- 配置层以为 chunk 可调
- 实际构建链路不可调
- benchmark 参数和真实索引策略可能不一致

结论：

```text
chunk 参数必须从配置进入 ChunkingPolicy，不能继续硬编码散落。
```

#### 3.2 长文本切分没有 overlap

当前 `_split_long_text` 按句子合并到 target token，但没有 overlap。

影响：

- 跨 chunk 的事实容易断裂
- 定义、引用、表述前后文容易丢失
- rerank 只能看到半截证据

结论：

```text
长文本 leaf chunk 必须支持 token/句子级 overlap。
```

#### 3.3 中文长段落切分不稳定

当前 `_split_long_text` 先把空白压成单空格，再用包含空白条件的正则切句。中文常见文本是：

```text
第一句。第二句。第三句。
```

句号后没有空格时，可能不能按预期拆分。

结论：

```text
中文切分必须支持无空格标点边界。
```

#### 3.4 `HARD_MAX_TOKENS` 定义了但没有强制使用

当前有硬上限常量，但实际切分中没有强制兜底。遇到超长句、OCR 大段、无标点文本时，仍可能产生过长 chunk。

结论：

```text
ChunkingPolicy 必须有 hard max enforcement。
```

#### 3.5 Docling markdown 路径可能丢页码和布局

PDF 若 docling 可用，会走：

```text
docling -> export_to_markdown -> build_markdown_blocks
```

`build_markdown_blocks` 只按空行、标题、表格痕迹拆分，不保留 page / bbox / layout source。

这会伤害：

- page filter
- 页码型问题
- EvidencePack source ref
- PDF 长文定位

结论：

```text
PDF 不能只依赖 markdown block。
必须优先保留 page-level segment，docling markdown 只能作为结构补充或 fallback。
```

#### 3.6 表格切分策略过粗

当前：

- CSV 最多读 100 行，基本单块输出
- XLSX 固定 20 行一块
- 表格对象主要作为 `object_block`

问题：

- 大表被截断
- 行级定位不足
- 多 sheet / 多区域查询只能靠文本匹配
- 表头、行范围、sheet 名没有成为正式 filter 维度

结论：

```text
表格需要 TableChunkingPolicy，而不是套用普通文本 chunk。
```

#### 3.7 层级单元已经生成，但没有成为检索策略

当前已生成：

- `parent_section`
- `document_summary`
- `page_summary`
- `content_block`
- `object_block`

但检索阶段没有正式使用：

```text
先召回 summary / parent，再展开 child
```

结论：

```text
层级检索不能只靠 coalesce，必须由 RetrievalPolicy 显式启用。
```

### 4. 目标数据处理架构

新的数据层应升级为：

```text

  -> Parser Router
  -> Structured Conversion
  -> Normalized Block Layer
  -> Chunk Planner

  -> Index Manifest / Quality Report
```

## 二十一、最新推荐施工顺序

从现在开始，后续实施以本节为准：

1. 先建 `backend/normalized_ingestion/policy.py`，锁定 `ChunkingPolicy / IndexUnitPolicy / ChunkPlan`。
2. 把 `rag_chunk_size / rag_chunk_overlap` 接入 `build_indexable_units`。

4. 修 PDF page-aware conversion，避免 docling markdown 吞掉页码。
5. 升级表格 row-window chunk。
6. 建 `backend/retrieval/planning.py`，锁定 `RetrievalPlan` 合同。
7. 升级 `QueryRewriter`，让它抽取 intent hints，而不是只拼关键词。
8. 升级 `RAGQueryRouter.plan()`，返回带 filter/policy/query_variants 的 plan。
9. 改 `RetrievalService`，把 plan.filters 传到 `RetrievalRequest.filters`，并输出 trace。
10. 改 `LlamaIndexRetrievalBackend`，让 dense Qdrant 和 lexical BM25 真正消费 filters。
11. 加结构测试，先证明 chunk、filter 生成和消费真实生效。
12. 再跑 SciFact 和中文小基线，看命中率是否恢复或超过历史链。
13. 再做 CandidateGraph 和 EvidencePackager。

一句话：

```text
先让系统“切出可靠证据单位”，再让系统“知道该搜哪里、该排除哪里”，最后讨论“怎么排得更准”。
```

### 2026-05-10：Phase R10 Rerank 成本治理落地

本轮目标：

```text
不再把 rerank 当作无限补救层。
保留 candidate_top_k=200 与 rerank_candidate_pool=200 的宽召回边界，
但降低真正进入 cross_encoder 的 rerank_top_n。
```

执行对比：

| 实验文件 | candidate_top_k | rerank_top_n | hit@10 | accuracy@1 | recall@10 | mean_retrieval_seconds | mean_rerank_seconds | 结论 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| `scifact_v2_current_top30.json` | 200 | 30 | 0.86 | 0.64 | 0.836 | 1.5035 | 1.5315 | 成本低，但低于 hit@10 0.88 底线 |
| `scifact_v2_current_top50.json` | 200 | 50 | 0.92 | 0.62 | 0.882 | 1.3696 | 2.4794 | 达到 top200 质量，成本大幅下降 |
| `scifact_v2_current_top100.json` | 200 | 100 | 0.92 | 0.62 | 0.882 | 1.3565 | 4.9668 | 对 top50 无质量收益，成本约翻倍 |
| `scifact_v2_current_top200.json` | 200 | 200 | 0.92 | 0.62 | 0.882 | 2.1744 | 54.3429 | 高质量基线，但 runtime 成本不可接受 |

决策：

```text
默认 rerank_top_n 从 200 下调到 50。
rerank_candidate_pool 保持 200。
```

原因：

```text
1. top50 在当前 SciFact 50 query 基线中与 top100/top200 同为 hit@10=0.92；
2. top50 的 mean_rerank_seconds=2.4794，约为 top200 的 1/22；
3. top30 虽然更快，但 hit@10=0.86，低于 0.88 质量底线；
4. candidate_pool 仍然保留 200，避免把召回边界一起砍窄。
```

代码落点：

- `backend/config.py`
  - `_resolve_rerank_top_n()` 默认值改为 `50`
  - 非法 `RERANK_TOP_N` 回退值改为 `50`
- `backend/config.json`
  - `system_config.retrieval.rerank_top_n=50`
- `backend/tests/config_runtime_regression.py`
  - 新增默认值回归测试，防止后续误回退到 `200`

当前结构原则：

```text
召回层可以宽，精排层必须有成本纪律。
RAG 系统后续优化不应靠不断扩大 rerank_top_n，
而应继续加强 chunk、metadata filter、query plan、candidate graph 和 evidence packaging。
```

# 通用任务图单位批次契约与动态范围架构计划

状态：实施中

## 1. 问题重新定义

这次问题不能被定义成“写作节点一次写 10 章太难，所以改成 5 章”。用户要求一次写 10 章时，10 章就是任务的运行时事实。系统应该能够让任务图、时序层、节点提示、审核和产物路径共同理解这个事实，而不是靠修改写作专用 Prompt 或硬改默认批次数来规避。

真正缺失的是通用任务图能力：

- 图需要表达“本轮处理多少个任务单位”。
- 时序层需要用这个单位数量派生范围、清单、目标量和下一轮位置。
- 节点需要收到标准化的运行范围契约，而不是从自然语言里猜“本轮到底几章”。
- 审核需要根据同一份运行范围契约检查缺单位、越界单位和正文主体，而不是写死“章”的数量。

写作任务里的“章”只是 `unit_label=章` 的一种实例。未来也可以是素材、镜头、任务卡、问题、章节、文件、节点等。

## 2. 现有代码事实

当前链路分布如下：

- `scripts/start_writing_project_run.ps1` 允许用户传入 `ChaptersPerRound`，并写入 `chapters_per_round` / `chapter_batch_size`。
- `scripts/configure_writing_simple_novel.py` 同时存在 `CHAPTERS_PER_ROUND` 常量、初始输入、派生字段、路由推进步长和 Prompt 文本。
- `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py` 的 `_apply_loop_derived_fields` 只支持固定 `value` / `size`，不能从运行时输入读取动态批次大小。
- `_runtime_batch_boundary_instruction` 已经会把 `batch_start_index`、`batch_end_index`、`batch_chapter_list` 注入节点消息，但它仍是写作章号表达。
- `sectioned_text_batch_quality` 已有通用雏形：通过 `unit_*_key`、`required_heading_patterns` 读取单位范围，但配置仍绑定写作字段。

因此，运行时已经有“动态批次”的入口，但派生字段和配置生成仍把批次大小冻结在脚本常量里。

## 3. 设计原则

1. 用户输入优先：用户要求每轮 10 章时，运行时事实必须是 10，而不是系统降级成 5。
2. 图配置给默认值，运行时输入给事实值：图模板可以声明默认 `default_unit_batch_size=10`，但启动输入可以覆盖。
3. 节点只读契约，不猜来源：节点看到的是 `batch_start_index`、`batch_end_index`、`batch_chapter_list`、`chapters_per_round`，不需要知道这些值来自用户、前端还是默认配置。
4. 时序层负责派生：范围、清单、目标量、下一轮步长都必须由同一套派生规则生成，不能由 Prompt 自己算。
5. 写作配置只是实例：写作任务可以继续用 `chapter_*` 字段，但底层机制必须支持任意 `unit_*` 字段。
6. 审核不替代执行：质量门可以发现缺章、越界、正文不足，但不能为了通过审核改变用户要求的批次大小。
7. 禁止后端小灶：后端不能因为任务是写作、代码、数据或某个具体节点就写专用分支。后端只提供通用执行、派生、验收、产物、记忆和工具装配能力；具体能力必须由任务图节点契约显式声明。
8. 程序能力配置化：如果某个节点需要后端程序来完成检查、拆分、统计、检索、格式化、生成索引或调用外部能力，不能把逻辑藏进运行时判断里，而应在节点上配置 `tools`、`skills`、`acceptance_policies`、`artifact_policy` 或其它通用能力契约。

## 4. 目标契约

### 4.1 通用单位批次字段

运行时应支持以下通用语义：

- `unit_label`：单位显示名，例如“章”。
- `unit_index_key`：当前单位起点字段，例如 `chapter_index`。
- `unit_batch_size_key`：批次大小字段，例如 `chapters_per_round`。
- `unit_start_key`：本批起点字段，例如 `batch_start_index`。
- `unit_end_key`：本批终点字段，例如 `batch_end_index`。
- `unit_list_key`：本批单位清单字段，例如 `batch_chapter_list`。
- `unit_target_metric_key`：单单位目标量，例如 `chapter_target_words`。
- `batch_target_metric_key`：批次目标量，例如 `batch_target_words`。

写作任务暂时保持既有字段名，避免大规模迁移；但派生规则必须用 `value_key` / `size_key` / `step_key` 从运行时读取批次大小。

### 4.2 动态派生规则

`_apply_loop_derived_fields` 增加动态参数：

- `add` 支持 `value_key`：`batch_end_index = chapter_index + chapters_per_round - 1`。
- `multiply` 支持 `value_key`：`batch_target_words = chapter_target_words * chapters_per_round`。
- `ordinal_group` 支持 `size_key`：`batch_index` 按运行时批次大小分组。
- `_advance_loop_counters` 支持 `step_key`：通过审核后 `chapter_index += chapters_per_round`。

图配置仍可保留固定 `value` / `size` / `step` 作为默认回退。

### 4.3 节点提示契约

Prompt 不应该写“固定 5 章”或“固定 10 章”作为系统事实，而应该写：

- 你每轮必须完成运行时 `batch_start_index` 至 `batch_end_index` 的全部章节。
- 本轮允许章号清单是 `batch_chapter_list`。
- 本轮批次大小由运行时 `chapters_per_round` 给出。
- 如果自然语言要求、上游产物、历史摘要和运行时字段冲突，以运行时字段为准。

这样当前任务传入 10，就写 10；前端以后改成 3，也写 3。

### 4.4 前端编辑器语义

图编辑器中节点不配置“写几章”的硬文案，而配置：

- 本图的单位类型：章。
- 本图的默认批次大小：10。
- 本图允许用户启动时覆盖批次大小：是。
- 本节点读取的范围字段：`batch_start_index` / `batch_end_index`。
- 本节点的验收策略：必须覆盖范围内全部单位。

边负责交接当前单位范围和产物引用；节点负责执行范围内动作；时序层负责推进范围。

### 4.5 节点能力装配契约

当节点需要程序能力时，任务图必须把能力作为节点配置的一部分声明，而不是由后端根据节点名或任务类型隐式注入。

节点可声明：

- `tool_bindings`：允许节点调用的后端工具，例如文本分段器、计数器、索引生成器、校验器、检索器。
- `skill_bindings`：允许节点使用的 Skill，例如章节质量审查、结构化大纲检查、代码测试分析、文档格式检查。
- `acceptance_policies`：节点输出验收策略，例如 indexed unit 完整性、正文主体统计、结构化字段完整性。
- `artifact_policy`：节点产物拆分、路径、正文区和说明区规则。
- `memory_access_policy`：节点可读写的记忆库、集合、主题和写入权限。

运行时只做三件事：

1. 读取节点声明的能力。
2. 按通用权限模型装配能力。
3. 把能力执行结果写回标准诊断、产物或交接包。

运行时禁止做三件事：

1. 根据 `chapter_draft`、`memory_commit_chapter` 等节点名自动启用特殊逻辑。
2. 根据 `writing.simple_novel` 等任务族自动改变质量门、批次大小、产物路径或记忆写入规则。
3. 为了让某个任务通过测试，在后端补写领域专用兜底。

## 5. 本轮实施范围

1. 恢复写作任务默认批次为 10 章，匹配用户要求。
2. 修改运行时派生规则，让批次大小来自运行时输入，而不是配置脚本硬常量。
3. 修改写作配置的派生字段和路由推进：
   - `batch_end_index` 使用 `chapters_per_round - 1`。
   - `batch_index` 使用 `chapters_per_round`。
   - `batch_target_words` 使用 `chapters_per_round`。
   - `chapter_index` 推进使用 `chapters_per_round`。
4. 修改写作 Prompt 文本，避免把批次数写成固定事实，只引用运行时范围和清单。
5. 保留通用输出分区能力，但移除后端中的写作专用默认段落常量；写作配置通过 artifact policy / quality policy 自己声明正文区和说明区。
6. 更新测试：用户传入 10 时，运行时派生、节点提示、配置默认和质量门都应以 10 章为准。
7. 审查并移除运行时中按写作节点名、写作契约名或章节字段自动触发的私门逻辑；需要保留的能力必须迁移为节点显式配置。
8. 为节点能力装配预留通用契约，先支持质量门、产物拆分、提交身份、批次边界提示这几类已出现需求，后续再扩展到更多 tools/skills。

## 6. 验证标准

1. `scripts/configure_writing_simple_novel.py` 默认 `CHAPTERS_PER_ROUND == 10`。
2. 启动脚本默认 `ChaptersPerRound == 10`。
3. `loop_derived_fields()` 不再把 5 或 10 写死到派生逻辑里，而是使用运行时 `chapters_per_round`。
4. `_apply_loop_derived_fields` 能根据不同 `chapters_per_round` 派生 1-10、11-20，也能派生 1-5、6-10。
5. 路由推进能用 `step_key=chapters_per_round`，而不是固定步长。
6. 节点提示里不出现“连续完成 5 章正文”这类固定批次数。
7. 写作质量门继续检查当前运行范围内全部章节，十章时就检查十章。
8. 后端运行时核心不出现 `memory_commit_chapter`、`chapter_draft`、`contract.writing.simple_novel.*` 这类领域专用判断；这些规则必须能在任务图节点配置中找到来源。
9. 需要程序能力的节点必须能通过节点 `tool_bindings` / `skill_bindings` / policy 配置解释，不能依赖后端隐藏分支。

## 7. 通用契约管理统筹

### 7.1 问题边界

“一次写多少章”“一个节点能不能调用计数器”“质量门检查哪些字段”“产物怎么拆分”“记忆库能读写哪里”，这些都不应该散落在 Prompt、脚本常量、运行时分支和前端临时字段里。它们本质上都是契约：系统在某个图、节点、边或运行阶段上，对输入、输出、能力、权限、交接、验收和治理作出的显式约定。

因此契约管理要从“输入输出 schema 管理”升级为“任务图运行协议管理”。schema 只是契约的一类，不能代表全部契约。

### 7.2 现有代码事实

当前系统已经有契约体系的骨架：

- 契约注册表：`storage/tasks/contract_specs.json` 存储可复用契约规格，`backend/tasks/flow_registry.py` 的 `list_contract_descriptors()` 会汇总 profile、flow、specific task、protocol 中出现的契约引用。
- 契约绑定层：任务、流程、任务图节点、任务图边已经存在 `input_contract_id`、`output_contract_id`、`payload_contract_id`、`node_contract_id`、`contract_refs` 等字段。
- 契约编译层：`backend/orchestration/runtime_loop/contract_compiler.py` 会把图、节点、边、Agent profile 编译成 `ContractManifest`。
- 契约清单模型：`backend/orchestration/runtime_loop/contract_compiler_models.py` 已有 `CompiledGlobalContract`、`CompiledNodeContract`、`CompiledEdgeHandoffContract`、`CompiledRuntimeContract`、`CompiledAcceptanceContract`。
- 运行时阶段契约：`backend/orchestration/runtime_loop/continuation_policy.py` 的 `CoordinationStageContract` 已承载 `memory_read_policy`、`memory_writeback_policy`、`artifact_policy`、`quality_retry_policy`、`review_gate_policy` 等策略字段。
- 前端管理入口：`frontend/src/components/workspace/views/task-system/TaskGraphContractQualityPage.tsx` 已能编辑图契约、节点契约、节点输入输出契约，并展示契约进入执行包的状态。
- API 出口：`backend/api/tasks.py` 已提供契约 CRUD 和任务图契约清单编译接口。

现状的问题不是没有契约，而是契约分类、归属、编译标准和前端编辑方式还没有统一。结果是 schema 类契约比较明确，policy 类契约仍容易变成运行时隐式能力。

### 7.3 通用契约分类

后续契约管理统一按以下类型组织，不再把所有东西混称为“节点契约”：

| 契约类型 | 作用 | 典型字段 | 归属层 |
| --- | --- | --- | --- |
| `schema_contract` | 定义输入、输出、边载荷的数据形状 | `input_fields`、`output_fields`、required fields | 图、节点、边 |
| `execution_contract` | 定义节点如何执行、可使用什么能力 | role、runtime lane、`tool_bindings`、`skill_bindings`、executor policy | 节点 |
| `unit_batch_contract` | 定义本轮处理的单位、范围、数量和推进规则 | `unit_label`、`unit_batch_size_key`、`unit_list_key`、`step_key` | 图、循环帧、节点 |
| `artifact_contract` | 定义产物写入、拆分、正文区、说明区和 manifest | `artifact_policy`、`artifact_targets`、section policy | 节点 |
| `memory_contract` | 定义记忆库读写权限、基准库/动态库边界和提交规则 | `memory_read_policy`、`memory_writeback_policy`、scope、write mode | 节点、边 |
| `handoff_contract` | 定义边如何交接、必须带哪些引用、失败如何处理 | `payload_contract_id`、`required_refs`、handoff policy | 边 |
| `acceptance_contract` | 定义输出是否合格的验收规则 | acceptance rules、quality policy、retry policy | 节点、图 |
| `runtime_contract` | 定义模型、输出上限、超时、流式、思考模式等运行参数 | model profile、max output tokens、timeout、stream policy | 节点、Agent profile |
| `governance_contract` | 定义版本、所有权、废弃、迁移和发布门禁 | version、owner、deprecated、migration、cutover | 契约注册表、图 |

写作任务里的“章范围”属于 `unit_batch_contract`；正文量检查属于 `acceptance_contract`；章节文件切分属于 `artifact_contract`；基准库/动态库读取属于 `memory_contract`；写手是否能用计数工具属于 `execution_contract`。这些都不能写成后端对写作任务的特殊判断。

### 7.4 契约所有权

契约必须有清晰所有权，否则运行时会继续“替图做决定”。

- 契约注册表只负责保存可复用契约规格、模板、版本和治理状态，不绑定具体运行实例。
- 任务图负责选择本图采用哪些契约，并声明图级默认值，例如单位类型、默认批次、默认记忆边界。
- 节点负责声明执行端契约，例如输入输出、角色、工具、技能、产物、记忆和验收策略。
- 边负责声明对接端契约，例如交接载荷、引用要求、上下文裁剪、失败语义和可见性。
- 编译器负责把图、节点、边上的分散声明规范化成 `ContractManifest`，并输出缺失、冲突、越权诊断。
- 运行时只消费编译后的 manifest 和 stage contract，不根据任务族、节点名、契约名前缀推断隐藏行为。
- 前端负责让用户用自然语言和结构化表单配置契约，但保存时必须落成规范字段。

### 7.5 统一绑定模型

为了兼容现有字段并走向通用化，目标结构是在图、节点、边上都引入同名的 `contract_bindings` 对象。旧字段在迁移期继续作为输入来源，但编译器输出必须以规范化后的绑定为准。

```json
{
  "contract_bindings": {
    "schema": {
      "input_contract_id": "contract.example.input",
      "output_contract_id": "contract.example.output",
      "payload_contract_id": ""
    },
    "execution": {
      "node_contract_id": "contract.example.executor",
      "tool_bindings": [],
      "skill_bindings": [],
      "executor_policy": {}
    },
    "unit_batch": {
      "unit_label": "章",
      "unit_index_key": "chapter_index",
      "unit_batch_size_key": "chapters_per_round",
      "unit_start_key": "batch_start_index",
      "unit_end_key": "batch_end_index",
      "unit_list_key": "batch_chapter_list",
      "step_key": "chapters_per_round"
    },
    "artifact": {
      "artifact_policy": {},
      "artifact_targets": []
    },
    "memory": {
      "memory_read_policy": {},
      "dynamic_memory_read_policy": {},
      "memory_writeback_policy": {}
    },
    "handoff": {
      "handoff_policy": "structured_packet",
      "required_refs": [],
      "context_visibility_policy": {}
    },
    "acceptance": {
      "acceptance_policies": [],
      "quality_retry_policy": {}
    },
    "runtime": {
      "runtime_lane": "",
      "stream_policy": {},
      "model_profile_overrides": {}
    },
    "governance": {
      "owner": "",
      "version": 1,
      "deprecated": false,
      "migration_note": ""
    }
  }
}
```

规范化规则：

- `input_contract_id`、`output_contract_id`、`payload_contract_id` 等旧字段映射到 `contract_bindings.schema`。
- `node_contract_id`、`contract_id`、`contract_refs` 映射到 `execution` 或 `schema`，具体由契约类型决定。
- `artifact_policy`、`memory_read_policy`、`quality_retry_policy` 等 policy 字段映射到对应类型。
- 如果旧字段和 `contract_bindings` 冲突，编译器必须报 `contract_binding_conflict`，不能静默选择一个。
- 迁移完成后，旧字段只作为视图层快捷字段，不再作为运行时权威来源。

### 7.6 编译与预检规则

契约编译器需要从“收集契约引用”升级为“编译契约绑定包”。

必须检查：

- 每个可执行节点必须有输入契约或声明 `input_mode=no_input_required`。
- 每个可执行节点必须有输出契约或声明 `output_mode=side_effect_only`，并说明副作用落点。
- 每条业务边必须有 `payload_contract_id` 或明确声明 `handoff_mode=control_only`。
- 每个 `acceptance_policy` 必须能追溯到节点、图或注册表中的契约来源。
- 每个 `tool_binding` / `skill_binding` 必须声明能力名、权限范围、输入来源和输出落点。
- 每个 `memory_writeback_policy` 必须声明写入库、写入模式、覆盖规则和提交身份。
- 每个 `unit_batch_contract` 必须能派生起点、终点、单位清单和推进步长。
- 每个运行时覆盖项必须来自节点、图或 Agent profile，不能由运行时根据领域名自动添加。
- 缺失引用、重复权威、类型错配、越权读写都应在 manifest issues 中暴露。

禁止：

- 在运行时按 `chapter_draft`、`memory_commit_chapter`、`writing.simple_novel` 这类名称启用特殊能力。
- 在质量门里硬编码写作领域字段来弥补图配置缺失。
- 用 Prompt 文案代替契约字段，让 Agent 自己猜单位范围、产物路径或记忆权限。
- 为通过测试生成伪产物、伪诊断或伪 manifest。

### 7.7 前端契约工作台

前端不应该只展示“输入契约/输出契约/节点契约”三个选择框，而要按契约类型组织配置。

建议在任务图编辑器内保留“契约与质量门”一级页，但内部拆成同一工作台里的分区：

- 图级契约：图默认单位、循环批次、全局治理、默认运行策略。
- 节点契约：schema、execution、artifact、memory、acceptance、runtime。
- 边契约：handoff、payload schema、required refs、上下文可见性、失败语义。
- 契约清单：展示编译后的 `ContractManifest`，让用户看到每条规则最终进入哪个节点或边。
- 诊断面板：按错误来源跳转到图、节点、边、契约注册表。

编辑体验要遵循“自然语言可理解，保存结构化”的原则：

- 用户可以看到中文名，例如“本轮单位范围”“产物拆分规则”“动态记忆读取”。
- 每个配置项解释对象职责，而不是暴露后端实现词。
- 高级字段可以折叠，但不能隐藏为后端默认。
- 从图上选中节点或边后，右侧编辑台直接显示该对象的契约分区。
- 图层关系仍在左侧，便于父子图、任务图作为节点、节点展开为图。

### 7.8 运行时消费规则

运行时只允许执行以下动作：

1. 读取 `ContractManifest` 和当前 stage contract。
2. 根据 `contract_bindings` 装配工具、技能、记忆、产物、质量门和运行参数。
3. 执行节点任务。
4. 根据 acceptance contract 产出诊断和是否重试。
5. 根据 handoff contract 生成边交接包。
6. 根据 unit batch contract 推进下一轮时序。

运行时不允许：

- 自己补齐未配置的领域逻辑。
- 根据契约 ID 前缀猜任务类型。
- 根据节点标题、中文名、文件路径自动改变行为。
- 为某个任务图写隐藏默认值。

如果配置不足，正确行为是预检失败或运行失败，并指出缺失契约，而不是运行时私自修补。

### 7.9 与写作任务的关系

写作任务是通用契约系统的一个实例：

- “五十章目标”是图级目标契约。
- “每轮十章”是单位批次契约。
- “写手读取世界观、大纲、人设、伏笔”是记忆读取契约。
- “写出本轮全部章节正文”是输出 schema + artifact 契约。
- “正文量不足、缺章、越界”是验收契约。
- “通过后提交动态记忆，必要时更新大纲”是 memory writeback + handoff 契约。

这样以后换成“视频镜头生成”“代码批量修复”“资料卡整理”“测试用例生成”，不用改后端，只换任务图契约。

## 8. 契约管理实施计划

### 阶段一：契约分类落盘

目标：让契约注册表和任务图都能表达契约类型。

改动点：

- 扩展 `ContractSpec` 的 `contract_kind` 使用规范，收敛到 `schema_contract`、`execution_contract`、`unit_batch_contract`、`artifact_contract`、`memory_contract`、`handoff_contract`、`acceptance_contract`、`runtime_contract`、`governance_contract`。
- 在任务图、节点、边数据模型中增加 `contract_bindings`。
- 保留旧字段读取，但新增保存优先写入 `contract_bindings`。

完成标准：

- 前端创建或编辑契约时能选择契约类型。
- 后端 API 返回 contract descriptor 时带出规范类型。
- 没有任何新功能依赖写作专用字段才能生效。

### 阶段二：编译器规范化

目标：所有契约统一进入 `ContractManifest`。

改动点：

- `compile_coordination_contract_manifest()` 读取 `contract_bindings`，并将旧字段规范化为同一结构。
- `CompiledNodeContract` 增加 categorized bindings，例如 `schema_bindings`、`execution_bindings`、`artifact_bindings`、`memory_bindings`、`acceptance_bindings`、`runtime_bindings`。
- `CompiledEdgeHandoffContract` 增加 `handoff_bindings`、`payload_schema_bindings`。
- `ContractCompileIssue` 增加冲突、缺失、越权、类型错配类诊断。

完成标准：

- 任务图 manifest 能完整展示每个节点和边实际使用的契约。
- 旧字段和新绑定冲突时预检失败。
- 运行时所需 policy 都能从 manifest 追溯来源。

### 阶段三：运行时切换为 manifest 消费

目标：运行时不再从零散节点字段推断领域行为。

改动点：

- `_contract_payload()` 从规范化 binding 生成 stage contract payload。
- 质量门、产物物化、记忆读写、动态批次派生都从对应契约读取配置。
- 移除或阻断按写作节点名、写作契约名前缀触发的运行时分支。

完成标准：

- 写作任务能通过通用契约跑出章节。
- 非写作任务可以复用同一单位批次、产物、质量门机制。
- 后端核心搜索不到新增的写作专用 runtime 判断。

### 阶段四：前端契约工作台升级

目标：用户可以自然地配置图、节点、边契约。

改动点：

- `TaskGraphContractQualityPage.tsx` 按契约类型重构展示和编辑。
- 节点右侧编辑台集成 schema、execution、artifact、memory、acceptance、runtime 分区。
- 边编辑页集成 payload、handoff、required refs、context visibility。
- 契约清单页展示 manifest 编译结果和跳转诊断。

完成标准：

- 选中节点即可配置节点契约，不需要跳到多个孤立页面。
- 选中边即可配置交接契约。
- 发布前能看到契约缺失和来源。

### 阶段五：迁移与清理

目标：旧字段不再成为事实来源。

改动点：

- 为现有任务图生成 `contract_bindings`。
- 用回归测试覆盖写作任务和至少一个非写作任务。
- 删除无用旧字段写入路径、旧测试和旧兜底逻辑。

完成标准：

- 旧字段只作为导入兼容或 UI 展示快捷入口。
- 新建任务图只写 `contract_bindings`。
- 测试覆盖契约缺失、冲突、合法运行、写作实例和非写作实例。

## 9. 验证矩阵

| 场景 | 应验证内容 | 期望 |
| --- | --- | --- |
| 写作每轮十章 | unit batch contract 派生 1-10、11-20 | 不靠 Prompt 猜测 |
| 缺少边载荷契约 | handoff contract 预检 | manifest 报错 |
| 质量门开启但无来源 | acceptance contract 追溯 | manifest 报错 |
| 节点声明计数工具 | execution contract 装配 tool | 运行时可调用，权限可见 |
| 节点写动态记忆 | memory contract 检查写入库和模式 | 越权失败 |
| 产物正文拆分 | artifact contract 物化章节或文件 | 路径和正文区来自配置 |
| 非写作批量任务 | 同一 unit batch contract 用于文件/用例/素材 | 不需要后端新分支 |
| 旧字段冲突 | 旧 `input_contract_id` 与新 binding 不一致 | 预检失败 |

## 10. 决策结论

契约管理必须作为任务图系统的一等通用能力推进。后端提供通用契约注册、绑定、编译、预检、运行消费和诊断；任务图通过契约声明具体业务意图；前端提供自然可理解的契约编辑工作台。写作任务不能再推动后端长出专用门禁或专用质量门，而要用同一套契约系统表达章节范围、产物、记忆、交接和验收。

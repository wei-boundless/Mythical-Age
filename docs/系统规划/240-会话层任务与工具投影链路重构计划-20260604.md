# 240-会话层任务与工具投影链路重构计划 - 2026-06-04

## 1. 问题定义

当前主会话里的任务执行反馈、工具调用反馈、任务收口反馈处于结构性错乱状态。问题不是某一个组件渲染错，而是会话层同时存在多套公开投影 authority，且它们的语义边界不一致：

- 实时流事件通过 `frontend/src/lib/store/events.ts` 进入 `message.runtimeProgress`。
- 底部全局状态通过 `sessionActivity` 单独维护。
- 历史恢复和 monitor 回填通过 `backend/harness/runtime/session_timeline.py` 生成 `runtime_attachments.public_timeline`。
- 会话正文又主要消费 `runtimeAttachments.public_timeline`，实时流缺口期间只能退回 `runtimeProgress` 的临时投影。

这会导致：

- 正文区和底部状态不一致。
- 同一信息在 `runtimeProgress`、`public_timeline`、`assistant_text` 三处重复或错位。
- 工具完成后仍显示“正在执行”，因为显示状态和实际状态来源不同。
- 实时流阶段只看到“正在思考”或弱反馈，历史回填后又出现另一套任务轨迹。
- turn 级附件与 task run 级附件混用，用户无法形成稳定的“当前 agent 在做什么”的感知。

真正损坏的系统性质是：`会话层公开投影 authority 不唯一`。

正确终态应该是：

- 会话正文只消费一套公开 chat contract。
- 实时流与历史恢复生成相同语义对象。
- 底部状态条只做全局摘要，不再承担正文任务叙事。
- 工具调用、任务进展、收口总结都按同一套 item contract 进入会话。

本次重构只处理单 agent 主会话链路，不处理 graph 侧独立工作台。

## 2. 现状审计

### 2.1 真实链路

#### A. 后端实时流入口

文件：

- `backend/api/chat.py`

事实：

- SSE 已经公开发送 `runtime_step_summary`、`model_action_request`、`model_action_admission`、`assistant_text`、`content_delta`、`done/error/stopped`。
- 说明后端并非没有实时语义事件，问题在于这些事件进入前端后没有收敛到单一公开 contract。

#### B. 前端实时状态层

文件：

- `frontend/src/lib/store/events.ts`

事实：

- `reduceStreamEvent()` 一边维护 `message.runtimeProgress`。
- 一边更新 `sessionActivity`。
- 一边维护 `stageStatus`。
- 这层是“实时事件解释器”，但它没有直接产出稳定的 `public_timeline`。

结论：

- 当前实时 authority 实际上是 `runtimeProgress + sessionActivity` 的混合体。
- 这不是成熟 agent 的公开 contract，只是前端中间状态。

#### C. 后端历史恢复/附件构建层

文件：

- `backend/harness/runtime/session_timeline.py`
- `backend/harness/runtime/public_chat_timeline.py`
- `backend/harness/runtime/progress_presenter.py`

事实：

- `build_session_runtime_timeline()` 会给 formal task run 和 turn run 生成 `runtime_attachments`。
- `public_chat_timeline.py` 已经尝试把 work unit 映射成 `assistant_text / tool_activity / artifact / final_summary / blocked`。
- turn run 附件会通过 `_public_timeline_from_progress_entries()` 直接从 progress entries 衍生 timeline。

结论：

- 历史/回填链路已经有一套更接近正确目标的 contract。
- 但它和实时流链路不是同一 projector。

#### D. 前端正文显示层

文件：

- `frontend/src/components/chat/ChatMessage.tsx`
- `frontend/src/components/chat/PublicRunActivity.tsx`

事实：

- 正文最初主要依赖 `runtimeAttachments.public_timeline`。
- 为修补实时空白，又补上了 `runtimeProgress -> PublicRunActivity` 的临时投影。
- 当前正文 authority 已变成“attachment 优先 + runtimeProgress 补桥”。

结论：

- 这是过渡方案，不是目标结构。
- 它解决了一部分“没有东西显示”的问题，但没有解决 authority 分裂。

#### E. 底部状态条

文件：

- `frontend/src/components/chat/SessionActivityBar.tsx`
- `frontend/src/lib/store/runtime.ts`

事实：

- `sessionActivity` 混合了承接基础设施错误、实时任务进度、控制状态、等待确认等信息。
- 它既被当作全局状态，又在若干情况下承担了实时执行反馈。

结论：

- `sessionActivity` 的职责过宽。
- 它应该降级为 `global status summary`，不再承担会话正文任务叙事。

### 2.2 当前 authority 冲突

当前存在四个公开信号源：

1. `assistant message content`
2. `message.runtimeProgress`
3. `runtimeAttachments.public_timeline`
4. `sessionActivity`

这些层目前的问题：

- 不是严格的上下游关系。
- 不是单向收敛关系。
- 不是同一 contract 的实时版与持久版关系。

所以它们会彼此覆盖、补洞、重复、漂移。

## 3. 参考原则

结合当前代码、既有文档和成熟 agent 外显体验，可以把本问题的硬约束锁成以下几条：

1. 会话正文是 conversation surface，不是 runtime event surface。
2. 实时流和历史恢复必须收敛到同一 public timeline contract。
3. 公开语义应尽量后端拥有，前端负责消费和轻量组织，而不是前端长期猜测 raw runtime event。
4. 底部状态条只能显示全局摘要，不能替代正文里的执行叙事。
5. `restore` 不得重新解释另一套公开语义；它只能恢复与实时流相同的公开对象。
6. `runtimeProgress` 可以作为 transient buffer，但不能作为长期公开 authority。
7. `assistant_text`、`tool_activity`、`final_summary` 必须有单一语义来源，不能前端再猜。
8. turn 级附件与 task run 级附件必须边界清晰，不能同时对同一公开动作发声。
9. 公开叙事中的 prose 分两类且必须区分：
   - agent 正文：模型真实输出
   - runtime 客观反馈：工具执行、等待、阻塞、收口
   不能再由系统硬编码“代替 agent 思考”。

## 4. 推荐设计方向

### 4.1 目标 authority

会话正文的唯一公开 authority 统一为：

`public_timeline`

这是唯一允许被 `ChatMessage` / `PublicRunActivity` 直接消费的任务与工具反馈 contract。

### 4.2 其他层级的新职责

#### `assistant message content`

职责：

- 只承载 assistant 正文回复本身。
- 流式阶段承载 `content_delta` / `assistant_text` 的正文。
- 不再承担工具活动和执行条目。

#### `public_timeline`

职责：

- 承载执行中的公开任务反馈。
- 类型只允许：
  - `status_update`
  - `assistant_text`
  - `tool_activity`
  - `artifact`
  - `verification`
  - `blocked`
  - `final_summary`

说明：

- `status_update` 只允许承载客观阶段状态，例如“任务已接管”“等待确认”“恢复运行中”。
- `assistant_text` 只允许承载 agent 在执行过程中的可见判断或说明，不承载系统 receipt。

#### `runtimeProgress`

职责：

- 仅作为实时流 projector 的 transient build buffer。
- 不再被正文组件直接当作长期 authority 使用。
- cutover 后只允许用于：
  - 构造 live `public_timeline`
  - 调试断言
  - 过渡期回归校验

#### `sessionActivity`

职责：

- 只做底部 global summary。
- 仅允许三类内容：
  - 基础设施级状态：连接失败、历史读取失败、恢复中、暂停/等待确认
  - 当前是否仍在运行
  - 当前会话的 coarse-grained summary

它不能再表达正文级工具调用序列。

## 5. 目标链路

### 5.1 单一公开链路

目标链路应固定为：

```text
runtime events
  -> backend public timeline event projector
  -> SSE public timeline delta
  -> message.publicTimelineDraft (transient)
  -> ChatMessage/PublicRunActivity

session refresh / session timeline
  -> persisted public_timeline
  -> replace transient draft with canonical persisted items
```

重点：

- 实时流和历史恢复都产出同一 item schema。
- 后端负责 canonical 公开语义；前端不再长期解释 raw runtime 事件。
- 不是“前端一套解释，后端一套解释”。

### 5.2 turn 与 task run 的边界

#### turn 级附件

适用：

- 单轮 agent turn 内的短动作。
- 例如直接工具调用、短思考、直接回复。

规则：

- 只锚定当前 assistant turn。
- 若后续升级为 formal task run，则 turn 级 draft 必须收口，不与 task run 级 timeline 并列重复表达同一工作。

#### task run 级附件

适用：

- 已建立 formal task run 的持续任务。

规则：

- 成为当前会话任务反馈的唯一 authority。
- 一旦 task run handoff 成立，后续执行反馈都应进入该 task run 的 `public_timeline`。
- turn 级 runtime draft 在 handoff 之后只保留必要开场，不再继续滚动工具活动。

## 6. 重构方案

### Phase 1：定义唯一 contract

目标：

- 明确前后端共用的 `public_timeline` item schema。
- 禁止正文层直接消费 `runtimeProgress` 作为长期 authority。
- 锁定“谁能写 public prose”。

修改：

- `frontend/src/lib/api.ts`
- `backend/harness/runtime/public_chat_timeline.py`
- `backend/harness/runtime/session_timeline.py`

完成标准：

- `PublicChatTimelineItem` 成为会话层唯一公开任务反馈结构。
- 文档、类型、后端输出一致。
- item kind 白名单与真实实现一致，不再出现文档允许集和代码允许集冲突。

### Phase 2：后端接管实时 public timeline projector

目标：

- 把当前实时公开语义从前端猜测改为后端生成。

设计：

- 优先新增后端实时 public timeline delta projector，建议位置：
  - `backend/harness/runtime/public_chat_timeline.py`
  - 或独立 helper：`backend/harness/runtime/public_timeline_stream.py`
- `backend/api/chat.py` 在 SSE 中直接发送标准化 public timeline delta。
- 前端只负责把 delta 合并到当前 assistant message draft。

输入：

- `runtime event`
- `current run context`
- `current public timeline state`

输出：

- `PublicChatTimelineItem` 或 `PublicChatTimelineDelta`

规则：

- 实时流阶段不再让 `ChatMessage` 自己把 `runtimeProgress` 再猜成 timeline。
- 实时 projector 与历史恢复必须共享同构语义规则。
- 若过渡期仍保留前端补桥，必须只作为 shadow 校验，不得继续作为正式 authority。

完成标准：

- 新任务执行时，即便尚未刷新 history，也能在正文里看到稳定的 `assistant_text / tool_activity / final_summary`。
- 前端不再需要把 `runtimeProgress` 翻译成 chat narrative。

### Phase 3：状态层去重与职责收缩

目标：

- 让 store 里只保留必要 authority。

修改：

- `frontend/src/lib/store/types.ts`
- `frontend/src/lib/store/events.ts`
- `frontend/src/lib/store/runtime.ts`

动作：

1. 给 `Message` 增加 transient public timeline draft 字段，例如：
   - `runtimePublicTimelineDraft?: PublicChatTimelineItem[]`
2. `runtimeProgress` 降级为中间态，不再作为正文 authority。
3. `sessionActivity` 降级为 coarse global summary。
4. `applyVisibleStreamState()` 和 `refreshSessionDetails()` 统一按“canonical public timeline 替换 transient draft”的逻辑更新。
5. 引入显式字段区分：
   - `assistant content`
   - `runtime public timeline draft`
   - `persisted runtime attachments`
   避免一个字段同时承担正文、状态、附件三种职责。

完成标准：

- store 中 authority 清晰：
  - 正文：`assistant content + public timeline`
  - 全局状态：`sessionActivity`
  - 调试中间态：`runtimeProgress`

### Phase 4：ChatMessage / PublicRunActivity 重写为单 authority 渲染

目标：

- 正文组件只消费一套 timeline。

修改：

- `frontend/src/components/chat/ChatMessage.tsx`
- `frontend/src/components/chat/PublicRunActivity.tsx`

动作：

1. `ChatMessage` 不再自己决定“attachment 还是 runtimeProgress”。
2. `ChatMessage` 只接收合并后的 `public timeline items`。
3. `PublicRunActivity` 不再负责 authority 决策，只负责渲染。
4. `final_summary` 只允许在 assistant 正文未覆盖时显示。
5. `assistant_text` 项只能作为执行中的任务反馈，不能与正文答复重复。
6. 等待确认、排队、恢复运行等状态必须走客观 runtime item，而不是伪装成 agent 发言。

完成标准：

- UI 层只做渲染和轻量 dedupe，不再承担语义推理。

### Phase 5：后端历史恢复与实时语义对齐

目标：

- 让 `session_timeline.py` 生成的 persisted `public_timeline` 与实时 projector 严格同义。

修改：

- `backend/harness/runtime/session_timeline.py`
- `backend/harness/runtime/public_chat_timeline.py`
- 必要时新增共享 projector helper

动作：

1. turn attachment 和 task attachment 使用相同 item contract。
2. 把 `_public_timeline_from_progress_entries()` 从“事件转条目”升级成与实时 projector 同构的逻辑，禁止继续保留 turn 专属简化语义。
3. 严格 suppress：
   - `done`
   - `agent_turn_terminal`
   - `step_summary_recorded`
   - 内部心跳
   - 裸布尔和原始 provider/internal status

完成标准：

- session refresh 后，不会把实时正文替换成另一套不同语义的 timeline。
- turn 恢复与 task 恢复不会再因不同 projector 生成不同文风或不同条目密度。

### Phase 6：删除旧残留 authority

目标：

- 清掉继续制造错乱的旧链路。

应删除或降级的内容：

- `ChatMessage` 中 runtimeProgress 直接投影正文的过渡逻辑
- `PublicRunActivity` 里对 runtimeProgress 的长期 authority 逻辑
- 前端对 `runtimeProgress` 的二次语义猜测
- 任何把 `sessionActivity` 当成正文代理的逻辑

完成标准：

- 不再存在“三套公开投影同时参与正文”的情况。

## 7. 文件级实施清单

### 后端

- `backend/api/chat.py`
  - 复核实时 SSE 公开字段是否足够支撑统一 live projector。

  - 升级为单一 chat public item projector。

  - 去掉临时/事件式差异逻辑。

### 前端

- `frontend/src/lib/api.ts`
  - 锁定 `PublicChatTimelineItem` 类型。

- `frontend/src/lib/store/types.ts`
  - 为 message 增加单一 public timeline 草稿字段。
  - 降级 `runtimeProgress` authority 语义。

- `frontend/src/lib/store/events.ts`
  - 移除正文 authority 的 event 直写混合逻辑。

  - 收缩 `sessionActivity`。

- `frontend/src/components/chat/ChatMessage.tsx`
  - 只渲染合并后的 public timeline。

- `frontend/src/components/chat/SessionActivityBar.tsx`
  - 收缩为 global summary bar。

## 8. Cutover 规则

### 8.1 Cutover 模式

采用 shadow -> cutover 两阶段。

#### Shadow

- 新增 live public timeline delta 链路。
- 继续保留旧 `runtimeProgress` 写入，但只用于比对和回归。
- UI 先切到新 timeline authority。

#### Cutover

- 确认实时流与历史恢复语义一致后：
  - 去掉正文对 `runtimeProgress` 的直接消费。
  - 删除旧投影补桥逻辑。

### 8.2 Rollback 规则

若 cutover 后出现以下任一情况，允许回滚到 shadow：

- 会话正文实时无反馈。
- session refresh 后 timeline 语义跳变。
- task handoff 后 turn/task 双重反馈重复。
- final summary 重复显示。

回滚范围：

- 仅回滚正文 authority 切换，不恢复旧样式噪声。

## 9. 验证矩阵

### 场景 1：普通单轮直接回复

期望：

- 正文流式显示 assistant text。
- 没有多余任务 timeline。
- `sessionActivity` 只显示粗粒度“生成回答中”。

### 场景 2：单轮工具调用

期望：

- 正文出现：
  - `assistant_text` 当前判断
  - `tool_activity` 当前动作
  - 完成后停止转动

### 场景 3：formal task run handoff

期望：

- turn 开场说明后，后续持续反馈只来自 task run public timeline。
- 不出现 turn/task 双轨重复。

### 场景 4：历史刷新 / 恢复

期望：

- refresh 前后的公开语义一致。
- 仅 authority 从 transient draft 切换到 persisted timeline。

### 场景 5：暂停 / 等待确认 / 受阻

期望：

- 正文显示 blocked 或 waiting 的公开解释。
- 底部状态条显示 coarse summary。
- 不出现原始 event name 或裸错误码。

### 场景 6：任务中断后续接 / 用户插话

期望：

- 当前任务被打断、续接、排队接收新输入时，正文区出现客观 runtime item。
- 不允许直接跳成工具调用而没有公开说明。
- 不允许系统静默吃掉 interruption 信息。

### 场景 7：任务收口

期望：

- 只出现一处收口总结。
- 若 assistant 正文已经包含最终总结，则 `final_summary` 不再重复。
- 收口摘要若来自 runtime，则必须是客观 closeout；若来自 agent，则显示为 agent 正文，不混成 receipt。

## 10. 禁止事项

本次重构过程中禁止：

1. 继续在 `ChatMessage` 或 `PublicRunActivity` 中堆更多 if/else 补洞。
2. 用 CSS 层掩盖 authority 分裂。
3. 保留“runtimeProgress 直接驱动正文”和“public_timeline 驱动正文”两套长期并存链路。
4. 让 `sessionActivity` 继续承担正文执行叙事。
5. 让 turn attachment 和 task attachment 对同一动作重复发声。
6. 在前端继续新增关键词猜测，把 raw event 临时翻译成“像 agent 的话”。

## 11. 推荐执行顺序

1. 锁类型和 authority 文档。
2. 后端落实时 public timeline delta projector。
3. 前端 store 接 delta，切正文到单 authority。
4. 对齐 backend session timeline / restore projector。
5. 删除旧补桥逻辑和前端语义猜测。
6. 跑真实长任务验收。

## 12. 结论

当前问题的本质不是“某个工具条目显示错了”，而是会话层没有单一公开投影 authority。

本次重构的核心不是再调 UI，而是把：

- 实时流
- 历史恢复
- 会话正文
- 底部状态

重新压回一条成熟 agent 应有的单链路：

`runtime events -> public timeline -> conversation`

只有这样，任务执行中的工具调用、任务进展、agent 判断、最终收口，才会在用户眼里形成稳定、强大、连续的智能感。

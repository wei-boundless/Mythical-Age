## 0. 核心结论

当前项目已经有产物系统，而且不是空的：

- 图层已经声明 `artifact_policy`、`artifact_context_policy`、`artifact_paths`
- 运行时已经能把最终内容材料化为文件和 `artifact_refs`
- 仓库层已经能按任务运行、项目、持久三种 scope 记录产物
- API 已经能查询产物仓库概览和记录
- 写作图也已经把 `artifact_index` 当成显式资源节点

但这套系统还没有统一成一个清晰的权威合同。现在的问题不是“没有产物系统”，而是：

```text
声明层、材料化层、仓库层、查询层各自都在说“产物”，
但它们还没有共用同一套命名、同一套主键、同一套验收语义。
```

所以这次规范化的目标不是再加一个旁路索引，而是把现有能力收束成一条主链：

```text
Artifact Declaration
  -> Runtime Materialization
  -> Repository Record
  -> Query / Memory View
  -> Acceptance / Handoff
```

## 1. 现状审查

### 1.1 已有主链

当前真实链路大致是：

- `scripts/configure_writing_modular_novel_graph.py` 负责声明节点级 `artifact_paths`
- `backend/orchestration/runtime_loop/runtime_assembly_builder.py` 把产物政策投影到模型可见运行契约
- `backend/orchestration/runtime_loop/task_artifact_materializer.py` 负责写文件、生成 `artifact_refs`、写 run report
- `backend/artifact_system/artifact_repository_service.py` 负责 scope 解析和仓库/记录落库
- `backend/artifact_system/artifact_repository_store.py` 负责 SQLite 存储
- `backend/orchestration/runtime_loop/task_run_loop.py` 在终态收口时串起验证、材料化、仓库记录、结果回写
- `backend/orchestration/runtime_loop/artifact_refs.py` 负责从运行态追踪产物引用
- `backend/api/memory.py` 提供产物仓库查询接口

### 1.2 已有有效能力

这套系统已经具备几个不错的底座：

- 产物路径已经是 workspace 内受限写入
- 产物写入支持版本化，避免覆盖
- 产物能生成运行报告
- 仓库支持 `run_scoped` / `project_scoped` / `durable`
- 写作图已经把产物索引纳入显式 memory/resource 节点
- 现有输出目录已经形成 round / review / commit / debug 的稳定习惯

### 1.3 结构性缺陷

1. 语义分裂。
   - `artifact_policy`、`artifact_target`、`output_path`、`artifact_paths`、`artifact_refs`、`artifact_repository` 同时存在，但职责边界不够干净。

2. 权威不清。
   - 图层说“要写什么”
   - 运行时说“写成什么”
   - 仓库层说“记录了什么”
   - API 层说“查到了什么”
   - 但缺少一条统一的 canonical contract 把它们串起来。

3. 查询链断。
   - `backend/orchestration/runtime_loop/artifact_refs.py` 里 `latest_output_refs_by_contract()` 目前返回空，说明按契约找产物这条能力还没真正闭合。

4. 运行时耦合。
   - `task_run_loop.py` 同时承担验证、材料化、仓库记录、结果回写，职责偏厚，未来很容易继续长出补丁式分支。

5. API 归属错位。
   - 产物接口现在挂在 `backend/api/memory.py` 下，容易让“记忆”和“产物”继续混成一个概念层。

6. 元数据不够硬。
   - 当前仓库记录里有路径、scope、状态，但对输出契约、内容 hash、产物类别、版本语义的表达还不够统一。

## 2. 目标架构

### 2.1 单一权威合同

以后产物系统只承认四类对象：

| 对象 | 责任 |
|---|---|
| `ArtifactDeclaration` | 图/契约层声明要产出什么 |
| `ArtifactMaterialization` | 运行时实际写出了什么 |
| `ArtifactRecord` | 仓库层登记了什么 |
| `ArtifactQueryView` | API / memory 层展示什么 |

其中：

- `ArtifactDeclaration` 只管声明，不管写盘
- `ArtifactMaterialization` 只管写盘，不管解释业务
- `ArtifactRecord` 只管登记事实，不管替代产物本身
- `ArtifactQueryView` 只管查询，不反向改写事实

### 2.2 权威边界

建议固定为：

1. 图契约层是声明权威。
2. 运行时材料化层是事实权威。
3. 仓库层是持久化权威。
4. API / memory 层是查询权威。

### 2.3 统一术语

以后建议统一如下语义：

| 术语 | 定义 |
|---|---|
| `artifact_policy` | 节点/阶段的产物声明 |
| `artifact_context_policy` | 上游产物如何作为上下文使用 |
| `artifact_ref` | 产物的唯一交接句柄 |
| `artifact_refs` | 当前回合交付给下游的产物句柄集合 |
| `artifact_root` | 本次材料化的根目录 |
| `repository_id` | 仓库逻辑或实例标识 |
| `logical_repository_id` | 仓库逻辑名 |
| `effective_repository_id` | scope 后的实际仓库实例名 |
| `collection_id` | 产物生命周期/类别集合 |
| `status` | accepted / rejected / candidate / committed 等状态 |

## 3. 规范化规则

### 3.1 图层规则

- 新图只写 `artifact_policy.artifacts[]`
- `artifact_target`、`output_path` 只保留迁移读取，不再作为新配置主路径
- `artifact_context_policy` 只负责上下文扩展，不负责写盘
- `artifact_paths` 只允许留在生成脚本里作为代码生成辅助，不应成为通用运行时语义

### 3.2 运行时规则

- `materialize_task_artifacts()` 是唯一材料化入口
- 它只接受已经收口的 `final_content` 和明确的 `task_policy`
- 它负责：
  - 解析目标路径
  - 写入文件
  - 生成 `artifact_refs`
  - 生成 run report
  - 返回材料化诊断
- 它不负责业务裁决

### 3.3 仓库规则

- 仓库记录必须表达：
  - scope
  - 逻辑仓库
  - 实际仓库
  - task_run_id
  - graph_id / stage_id / node_run_id
  - task_ref
  - collection_id
  - status
  - content_hash
  - provenance metadata

- `content_hash` 应该表示内容或文件事实，不应只哈希 ref 字符串
- 仓库层不应再把 `node_id` 当作仓库名的别名使用，最好改成更清楚的 owner / producer 语义

### 3.4 查询规则

- 产物查询要支持：
  - task_run
  - graph
  - stage
  - node_run
  - repository
  - collection
  - status
  - output_contract_id

- `latest_output_refs_by_contract()` 不能再空返回，必须真正能从结果或仓库里取到合同级引用

### 3.5 接受规则

- 长任务的最终接受判断必须同时看：
  - 终态
  - 产物引用
  - 产物政策
  - review verdict
  - 业务质量门

- 不能只靠“看起来写了文件”

## 4. 方案分期

### Phase 1: 统一命名与主键

目标：

- 收束 `artifact_policy`、`artifact_target`、`output_path`、`artifact_paths` 的职责
- 明确 `artifact_ref`、`artifact_refs`、`artifact_root`、`repository_id` 的唯一语义
- 补齐按契约查询产物的最小闭环

关键文件：

- `backend/artifact_system/artifact_repository_models.py`
- `backend/artifact_system/artifact_repository_service.py`
- `backend/artifact_system/artifact_repository_store.py`
- `backend/orchestration/runtime_loop/artifact_refs.py`

不做的事：

- 不扩新链路
- 不再引入新的产物别名
- 不保留双轨业务语义

### Phase 2: 仓库模型硬化

目标：

- 把仓库记录从“能记”升级成“能追溯”
- 让 scope、collection、status、contract、provenance 成为刚性字段
- 把 repository 生命周期与 task_run/materialization 解耦

关键文件：

- `backend/artifact_system/artifact_repository_store.py`
- `backend/artifact_system/artifact_repository_service.py`

### Phase 3: 运行时材料化与验证解耦

目标：

- 让 `task_run_loop.py` 不再同时扛太多职责
- 把 artifact validation、materialization、repository record、result writeback 拆成明确步骤
- 让 acceptance 读取统一的产物事实，不再从局部变量拼判断

关键文件：

- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/orchestration/runtime_loop/task_artifact_materializer.py`
- `backend/orchestration/runtime_loop/runtime_assembly_builder.py`

### Phase 4: API 与查询面规范化

目标：

- 将产物查询从 memory 语义中剥离成独立资源面
- 保留当前接口作为过渡时，再补正式 artifact API
- 让上层可以按 contract / repository / stage / node 查询产物

关键文件：

- `backend/api/memory.py`
- `backend/api/artifacts.py`（建议新增）

### Phase 5: 旧链路清理与收口

目标：

- 删除不再需要的 legacy 输入分支
- 清理旧测试和旧适配层
- 让图层、运行时、仓库层、查询层只保留同一套术语

重点清理项：

- `artifact_target` / `output_path` 的业务主路径
- `latest_output_refs_by_contract()` 的占位返回
- `artifact_repository_id` 等重复别名
- 产物与 memory API 的概念混挂

## 6. 验证与风险控制

### 验证矩阵

1. 材料化验证
   - 目标路径存在
   - `artifact_refs` 非空
   - run report 真实写入
   - 重跑时版本化行为正确

2. 仓库验证
   - scope 解析正确
   - project / durable / run 三种仓库可区分
   - 记录能按 task_run / repository / collection / status 过滤

3. 查询验证
   - 可按 contract 找到最新产物
   - 可按 stage / node_run 找到产物
   - `artifact_refs` 和仓库记录一致

4. 图配置验证
   - 写作图的 `artifact_paths` 最终能正确生成产物政策
   - `artifact_index` 仍是显式资源节点
   - review / commit / debug 目录与产物类别一致

5. 回归验证
   - 模块化长篇写作图仍能产出真实文件
   - 产物仓库 overview 能看到记录
   - 失败态不会伪造交付文件

### 风险

- 迁移时最容易出问题的是旧字段读取和新字段写入不一致
- `task_run_loop.py` 如果不拆，会继续把问题掩盖成局部补丁
- 如果不先补 `output_contract_id` / 查询契约，产物索引仍会半残
- 如果继续让 memory API 承担产物语义，后面还会再分叉

### 收口标准

1. 产物只认一套主语，不再有多套近义词并行
2. 图层、运行时、仓库层、API 层对同一产物说同一种话
3. 按契约能查到最新产物
4. 按 scope 能区分 run / project / durable
5. 旧链路清理完成后，不再保留新的双轨分支

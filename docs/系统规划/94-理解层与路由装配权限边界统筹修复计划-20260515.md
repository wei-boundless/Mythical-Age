# 94-理解层与路由装配权限边界统筹修复计划-20260515

## 0. 计划定位

本文用于统筹修复“显式任务节点被理解层能力候选反向污染，导致写作任务误装配为 RAG 任务”的结构性问题。

本文不是替代 `93-TaskTemplate选择层拆除正式实施计划`。93 处理的是 `TaskTemplate` 选择层退场；本文处理的是理解层、能力候选、执行形态解析、权限边界之间的主权关系。

本轮目标不是单独屏蔽 RAG，也不是为写作团队加特殊判断，而是建立一条通用规则：

当系统已经进入显式任务、显式节点、显式 agent 或显式权限域时，弱理解信号只能作为参考，不能覆盖当前任务主权。

## 1. 已确认的真实问题链路

### 1.1 现象

《洪荒时代》写作团队的 `world_design` 节点进入了 `agent:world_designer`，但模型回答时引用了 AI 眼镜、AI 营销等 RAG 检索证据。

表面看像是写作 agent 获得了 RAG 权限，实际不是。

### 1.2 证据

事件文件：

`storage/runtime_state/events/taskrun_writing_team_honghuang_20260515_real_graph_fix2_taskinst_turn_writing_team_honghuang_20260515_real_graph_fix2_973523f6_world_design_025126ee.jsonl`

关键事件：

1. `search_policy_resolved`
   - `search_policy = null`
   - `allowed_sources = ["local_files", "rag", "web"]`

2. `task_contract_built`
   - `template_id = "template.rag.knowledge_answer"`
   - `match_reasons = ["rag_execution_posture"]`
   - `effective_route = "rag"`
   - `execution_posture = "direct_rag"`
   - `effective_skill = "rag-skill"`
   - 同时 `task_family = "writing_team_long_novel"`
   - 同时 `task_mode = "world_designer"`

3. `context_snapshot_built`
   - `selected_sections = ["retrieval_evidence"]`
   - `retrieval_evidence_count = 3`

4. `runtime_directive_issued`
   - `allowed_operations = ["op.model_response"]`
   - `denied_operations = ["op.mcp_retrieval", "op.search_text"]`

结论：

RAG 不是执行时被授权的，而是在模型上下文准备前被错误装配进来的。

### 1.3 根因

当前 `AgentRuntimeChainAssembler` 在显式任务选择时调用了：

`backend/orchestration/agent_runtime_chain.py::_align_understanding_with_explicit_task_selection`

它会把顶层理解结果改成：

- `route = "agent"`
- `execution_posture = "task_runtime"`
- `preferred_skill = None`
- `should_skip_rag = True`

但它没有清理嵌套字段：

- `candidate_capabilities`
- `capability_resolution`

复现同参装配时可见，顶层字段已经正确，但嵌套字段仍然是：

```json
{
  "selected_candidate_type": "skill",
  "selected_candidate_name": "rag-skill",
  "route": "rag",
  "execution_posture": "direct_rag",
  "preferred_skill": "rag-skill"
}
```

随后：

`backend/understanding/capability_resolution_view.py::capability_resolution_view`

优先读取嵌套 `capability_resolution`，于是把已经对齐为 `agent/task_runtime` 的显式任务重新解释为 `rag/direct_rag`。

再随后：

`backend/tasks/execution_shape_resolver.py::resolve_execution_shape`

看到 `direct_rag` 或 `rag-skill` 后，选择了 RAG 执行形态和 RAG recipe preset。

最后：

`backend/capability_system/search_policy.py::normalize_search_policy(None)`

把空搜索策略解释为默认允许：

- `rag`
- `local_files`
- `web`

因此 retrieval evidence 被带进上下文。

## 2. 系统性质判定

这是结构性问题，不是单点 bug。

它破坏的是四个层的主权边界：

1. 理解层
   - 负责提出候选，不负责最终装配权限。

2. 显式任务层
   - 一旦用户或任务图已经选中 task/node/agent，应当拥有当前轮主权。

3. 路由装配层
   - 应根据显式任务主权和权限域决定 execution shape，而不是反向相信弱候选。

4. 权限层
   - 应以 task/agent/search policy 的交集作为上下文与工具可见性边界。

当前的问题是：理解层候选被当成了装配主权。

## 3. 不可破坏的能力场景

本轮不能简单删除 `capability_resolution`，否则会影响以下正常能力：

1. 普通 RAG 问答
   - 没有显式任务图节点时，用户问知识库内容，应自然进入 RAG。

2. PDF 分析
   - 用户提供 PDF 或绑定 PDF 时，应自然进入 PDF 子能力。

3. 表格/结构化数据分析
   - 用户提供 CSV/表格/数据集时，应自然进入 structured data。

4. 网页搜索
   - 用户开启联网或显式询问实时信息时，应自然进入 web search。

5. 主会话自然工具调用
   - 非任务图、非显式节点时，理解层候选仍然应能辅助路由。

所以修复必须是“按主权域裁剪候选”，不是“禁掉候选系统”。

## 4. 目标设计

### 4.1 主权等级

执行装配时必须按以下优先级判断：

1. 显式 TaskGraph stage execution request
2. 显式 selected_task_id / task_id / specific_task_id
3. 显式 agent runtime profile
4. 显式用户权限开关 / search_policy
5. 当前轮 query understanding 顶层 route
6. capability_resolution / candidate_capabilities
7. heuristic fallback

低等级不能覆盖高等级。

### 4.2 理解层输出分级

`QueryUnderstanding` 中的字段分为三类：

1. 当前轮主权字段
   - `route`
   - `execution_posture`
   - `preferred_skill`
   - `tool_name`
   - `should_skip_rag`

2. 候选解释字段
   - `candidate_capabilities`
   - `capability_resolution`

3. 诊断字段
   - `reasons`
   - `structural_signals`

当进入显式任务主权域后，候选解释字段必须降级为诊断，不能继续参与 `ExecutionShapeResolver`。

### 4.3 能力解析视图改造

`capability_resolution_view()` 不能无条件优先读取嵌套 `capability_resolution`。

目标规则：

1. 如果存在 `structural_signals.understanding_aligned_to_explicit_task = true`
   - 优先顶层 `route/execution_posture/preferred_skill/tool_name`
   - 嵌套 `capability_resolution` 只作为 diagnostics

2. 如果 `should_skip_rag = true`
   - 不允许嵌套 `rag/direct_rag/rag-skill` 恢复为有效路由

3. 如果没有显式任务主权
   - 保持现有候选能力自然路由

### 4.4 显式任务对齐改造

`_align_understanding_with_explicit_task_selection()` 必须做完整主权收口：

1. 设置顶层字段
2. 清理或失效化嵌套 `capability_resolution`
3. 保留原候选到 diagnostics
4. 清空 `candidate_capabilities` 或标记为 `suppressed_candidate_capabilities`
5. 写入明确的 `authority = "explicit_task_selection"`

建议字段：

```json
{
  "capability_resolution": {
    "selected_candidate_type": "",
    "selected_candidate_name": "",
    "route": "agent",
    "execution_posture": "task_runtime",
    "preferred_skill": "",
    "tool_name": "",
    "mcp_route": "",
    "diagnostics": {
      "selection_source": "explicit_task_selection_override",
      "suppressed_previous_resolution": {}
    }
  }
}
```

### 4.5 权限域默认值改造

`normalize_search_policy(None)` 当前默认返回 `rag/local_files/web`，适合主会话探索，但不适合任务图节点。

不能直接全局改为 deny all，否则会影响主会话正常能力。

目标规则：

1. 主会话无显式限制
   - 可继续默认 `rag/local_files/web`

2. TaskGraph 节点 / coordination_task
   - `search_policy=None` 不能解释为全开放
   - 应从节点 runtime profile、agent profile、task operation policy 计算
   - 若没有声明，则默认空集合或仅允许任务上下文需要的非检索上下文

3. 显式用户权限开关
   - 必须参与 search policy 交集
   - 关闭 RAG 时，不允许 retrieval evidence 进入上下文
   - 关闭 web 时，不允许 web_search agent 或 web tool 出现在可调用集合

### 4.6 上下文注入边界

上下文包不能只看 retrieval_results 或 memory facade 候选。

任何 `retrieval_evidence` 进入模型上下文前，必须满足：

1. 当前 recipe/source kind 允许 retrieval
2. 当前 resource policy 允许 `op.mcp_retrieval`
3. 当前 search policy 允许 `rag`
4. 当前 task/agent 没有 `should_skip_rag`

四者缺一则不能注入。

## 5. 实施阶段

### Phase 1：理解主权收口

修改文件：

1. `backend/orchestration/agent_runtime_chain.py`
2. `backend/understanding/capability_resolution_view.py`

动作：

1. 增强 `_align_understanding_with_explicit_task_selection`
2. 增加 `capability_resolution` 主权覆盖
3. 增加 `candidate_capabilities` 抑制记录
4. 修改 `capability_resolution_view`，显式任务优先顶层字段

退出标准：

1. 同参 `world_design` 装配输出 `route=agent`
2. `execution_shape` 不再出现 `rag_execution_posture`
3. `template/recipe` 不再是 `template.rag.knowledge_answer`
4. 普通 RAG 查询仍可进入 RAG

### Phase 2：ExecutionShapeResolver 权限化

修改文件：

3. 写作任务应落到通用 `task_runtime/conversation` 或 task-specific recipe，而不是 retrieval recipe
4. 保留 PDF/表格/web 在非显式任务或显式能力任务中的自然路由

退出标准：

1. 写作团队节点不再被 RAG recipe 接管
2. PDF 显式任务仍走 PDF recipe
3. 表格显式任务仍走 structured data recipe
4. 搜索显式任务仍走 search recipe

### Phase 3：搜索权限策略分域

修改文件：

1. `backend/capability_system/search_policy.py`

3. `backend/orchestration/runtime_loop/agent_delegation_executor.py`

动作：

1. 保留主会话默认搜索策略
2. 新增 task-runtime 专用 search policy 解析
3. coordination_task 如果没有显式 search_policy，默认不能继承全开放
4. 权限开关与 agent/task policy 做交集

退出标准：

1. `world_design` 事件中不再出现 `allowed_sources=["local_files","rag","web"]`
2. 未授权节点不能 delegated 到 `agent:rag_analyst`
3. 主页面开启 RAG 时，主会话 RAG 仍正常
4. 主页面关闭 RAG 时，RAG evidence 和 RAG agent 都不可见

### Phase 4：上下文注入闸门

修改文件：

1. `backend/orchestration/runtime_loop/task_run_loop.py`
2. `backend/orchestration/agent_runtime_chain.py`

动作：

1. `retrieval_evidence` 注入前检查 search policy
2. `retrieval_evidence` 注入前检查 resource policy
3. memory restore 不得绕过当前任务权限
4. task graph 节点只接收允许的 upstream_outputs / artifact_refs / project memory

退出标准：

1. 写作节点上下文不含无关 RAG evidence
2. RAG 节点上下文仍能看到 retrieval evidence
3. PDF 节点上下文只看到 PDF evidence，不混入 RAG evidence

2. 删除 `get_local_mcp_primary_template`

退出标准：

1. capability system 不再有 template -> mcp 映射
2. recipe preset 不再使用 `template.rag.*` 命名
3. runtime loop 不再消费 `selected_template`

## 6. 回归矩阵

### 6.1 必测任务图场景

1. 写作团队 `world_design`
   - 预期：不触发 RAG
   - 预期：无 retrieval evidence
   - 预期：产物是世界观设计，不是“检索证据不足”

2. 写作团队完整图首轮
   - 预期：节点按图流转
   - 预期：上游产物通过 artifact refs 传递
   - 预期：不出现 `rag_answer:*` 作为写作产物引用

3. 写作团队 writer 节点
   - 预期：只能看到世界观、大纲、人物、章节细纲等上游产物
   - 预期：不能看到 unrelated AI Knowledge 检索证据

### 6.2 必测能力场景

1. 普通 RAG 问答
   - 预期：自然进入 RAG

2. PDF 分析
   - 预期：自然进入 PDF

3. 表格分析
   - 预期：自然进入 structured data

4. 网页搜索
   - 预期：开启联网时可调用 web
   - 预期：关闭联网时不可调用 web

5. 主页面权限开关
   - RAG 关：无 RAG evidence / 无 rag agent
   - Web 关：无 web tool / 无 web agent
   - 本地权限关：无 local file/PDF/table 读取

### 6.3 必测结构断言

1. 显式任务选择后：
   - `query_understanding.route == "agent"`
   - `query_understanding.execution_posture == "task_runtime"`
   - `capability_resolution_view(...).route == "agent"`

2. task graph 节点：
   - `search_policy=None` 不等于全开放

3. retrieval evidence：
   - 必须有明确授权才进入 context package

## 7. 禁止的修复方式

1. 禁止只判断 `agent:world_designer` 然后屏蔽 RAG
2. 禁止只在 prompt 里写“不要用 RAG”
3. 禁止只清 `preferred_skill` 不清 `capability_resolution`
4. 禁止把 `normalize_search_policy(None)` 全局改成空集合而不区分主会话和任务图
5. 禁止保留 template -> mcp 的旧主路由映射作为长期兼容
6. 禁止用测试假产物绕过真实任务图流转

## 8. 推荐实施顺序

本问题建议先执行本文 Phase 1 到 Phase 4，再继续 93 的 Phase 5。

原因：

1. 当前污染发生在理解与权限边界，不先修会继续误导 recipe 拆除。
2. 93 的模板拆除能减少长期复杂度，但不能单独解决嵌套 resolution 越权。
3. 先建立主权边界，再清模板残留，风险最小。

## 9. 完成定义

本计划完成必须同时满足：

1. 写作团队节点不再误装配 RAG。
2. 显式任务选择不会被 capability candidate 覆盖。
3. TaskGraph 节点默认不继承主会话全开放搜索域。
4. 上下文注入受 resource policy 和 search policy 双重约束。
5. RAG/PDF/表格/Web 在各自授权场景仍正常工作。
6. 旧 template -> MCP 主映射进入清理阶段，不再作为新链路依赖。

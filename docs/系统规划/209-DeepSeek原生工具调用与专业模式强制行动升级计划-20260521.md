# 209-DeepSeek原生工具调用与专业模式强制行动升级计划

日期：2026-05-21

状态：实施中

关联文档：

- `206-角色标准专业三模式与专业长任务母版架构升级计划-20260521.md`
- DeepSeek Tool Calls Guide: `https://api-docs.deepseek.com/guides/tool_calls`
- DeepSeek Chat Completion API: `https://api-docs.deepseek.com/api/create-chat-completion/`

## 1. 问题定义

本问题不是“提示词没有告诉模型要用工具”，而是运行时没有把“当前轮必须行动”下沉为 provider 原生工具调用约束。

DeepSeek 的工具调用是 OpenAI 兼容协议：

```text
request.tools -> assistant.tool_calls -> app executes tool -> role=tool result -> next request
```

模型不会自己执行工具。若 `tool_choice` 保持默认 `auto`，模型可以选择直接回答，也可能把工具调用写成可见文本。因此专业模式长任务会出现三类失败：

1. 需要读取、写入、验证时，模型只说“我将执行”而没有真实 `tool_calls`。
2. 动态工具收窄后，模型仍按提示词惯性输出伪 `DSML/tool_calls/invoke`。
3. 写入类任务在读完材料后没有被协议层强制切到 `write_file/edit_file`。

正确终态：

```text
SemanticTaskContract
  -> ProfessionalTaskRunDriver decides next required tool set
  -> ModelResponseRuntimeExecutor forwards ToolCallBindingOptions
  -> ModelRuntime.bind_tools(..., tool_choice=..., parallel_tool_calls=False)
  -> ModelResponseRuntimeExecutor only accepts native response.tool_calls
  -> visible output never contains internal tool protocol
```

## 2. 当前源码事实

| 层 | 文件 | 现状 |
|---|---|---|
| 模型绑定 | `backend/execution/model_runtime.py` | 已通过 `model.bind_tools(tools)` 接入原生工具，但没有传 `tool_choice`、`strict`、`parallel_tool_calls` |
| 工具响应 | `backend/execution/model_response.py` | 已读取 `response.tool_calls` 并转换为 `tool_call_requested` |
| 专业模式 | `backend/orchestration/runtime_loop/professional_task_run_driver.py` | 已能计算 `required_next_tools` 并收窄本轮 tools，但没有传 provider 原生强制策略 |
| DeepSeek thinking | `backend/execution/model_runtime.py` | 已保留 `reasoning_content`，可支持 thinking + tool roundtrip |

## 3. 设计方向

新增一个小而明确的执行层协议：

```python
ToolCallBindingOptions(
    tool_choice=None | "auto" | "none" | "required" | {"type": "function", "function": {"name": "..."}},
    strict=None | False | True,
    parallel_tool_calls=None | False,
)
```

策略：

| 场景 | tools 暴露 | tool_choice |
|---|---|---|
| 无工具轮 / 收口轮 | `[]` | 不传 |
| 可选工具轮 | 可用工具集合 | `auto` 或不传 |
| 下一步必须行动，只有一个合法工具 | 只暴露该工具 | 指定该 function |
| 下一步必须行动，有多个等价工具 | 只暴露候选工具 | `required` |
| 专业模式工具轮 | 动态收窄 | `parallel_tool_calls=False` |
| strict mode | 默认不开 | schema 审计后再按 provider/任务开启 |

## 4. 文件级实施清单

1. `backend/execution/tool_call_policy.py`
   - 新增 `ToolCallBindingOptions`。
   - 新增 `build_required_tool_call_options()`。
   - 只负责 provider 兼容参数，不做业务判断。

2. `backend/execution/model_runtime.py`
   - `invoke_messages_with_tools()` 和 `astream_messages_with_tools()` 接收 `tool_call_options`。
   - `bind_tools()` 统一从 options 传 `tool_choice`、`strict`、`parallel_tool_calls`。
   - 不开启 strict 默认值。

3. `backend/execution/model_response.py`
   - `stream()` 接收 `tool_call_options`。
   - 流式、非流式和 stream recovery 都透传同一 options。

4. `backend/orchestration/runtime_loop/professional_task_run_driver.py`
   - 根据 `required_next_tools` 和当前暴露的工具实例生成 options。
   - 写入/验证/材料读取等强制阶段使用 provider 原生 tool choice。
   - 收口与 repair 阶段继续不暴露工具。

5. 测试
   - 模型运行时：断言 `bind_tools()` 收到指定 `tool_choice`、`parallel_tool_calls=False`。
   - 专业模式：断言读材料后下一轮只暴露写工具，并传入 required/指定 tool_choice。
   - 回归：继续跑三模式与专业长任务测试集。

## 5. 禁止事项

- 不把 `DSML/tool_calls/invoke` 当作模型可见的工具教学协议。
- 不为了兼容旧链路保留 `autonomous_task_run` 新入口。
- 不用 prompt 文案替代 provider tool choice。
- 不全局开启 DeepSeek strict beta，除非工具 schema 已完成审计。

## 6. 验收标准

1. 原生 `response.tool_calls` 仍能被接收并执行。
2. 专业模式强制行动轮会传递明确的 `tool_choice`。
3. 单轮最多一个工具调用时，provider 层禁用 parallel tool calls。
4. 写入类任务读完材料后，下一轮不再暴露读搜工具。
5. 专业长任务场景集不因伪工具文本泄漏而失败。

# 100-协调监控实时化修复计划-20260515

## 0. 问题定位

当前协调监控不是实时监控，而是“按需拉取 trace 后重建快照”：

1. 前端只在进入会话和流结束后调用一次 `hydrateLatestOrchestrationSnapshot()`。
2. `hydrateLatestOrchestrationSnapshot()` 先查 session task runs，再拉完整 `task run trace`。
3. 后端 `trace_reader` 为了返回 trace，会读取整条 event log、checkpoint、agent runs、coordination runs。
4. 前端再把 trace 重放成 `orchestrationSnapshot`。

这导致三个结构性后果：

1. 监控天然不是持续更新，而是一次性快照。
2. session 级查询链路偏重，长任务执行中容易超时或返回滞后状态。
3. 主任务 trace 与子节点实际推进节奏不完全同步时，前端看到的是陈旧图状态。

因此这里不能继续在 trace 上打补丁，必须引入“监控专用 live read model”。

## 1. 修复目标

把协调监控改造成真正的实时运行窗口：

1. 用户选中会话后，监控面板能持续刷新。
2. 当前运行节点、ready/running/blocked/completed 节点集合实时更新。
3. 当前交接、工作记忆操作、阶段产物能够随运行推进刷新。
4. 监控读取不能依赖完整 trace 回放。
5. 历史 trace 保留给诊断和回放，不再承担实时监控职责。

## 2. 设计原则

### 2.1 监控与诊断分层

- `trace` 用于审计、回放、问题追踪。
- `live monitor` 用于前端实时呈现。

两者都可以来自同一运行时状态源，但不能再共用同一条重型读取链路。

### 2.2 监控读取必须轻量

实时监控只读取这些“最新态”：

1. `TaskRun`
2. `latest checkpoint`
3. `CoordinationRun`
4. `CoordinationNodeRun`
5. `latest merge result`
6. `CoordinationRun.diagnostics` 中已投影好的：
   - `coordination_flow`
   - `langgraph_runtime_state`
   - `task_graph_scheduler_state`

不再为实时监控扫描整条 event log。

### 2.3 前端轮询要独立于聊天流

聊天消息流结束与否，不应决定图监控是否刷新。  
监控应该有自己独立的轮询生命周期。

## 3. 实施步骤

### 3.1 后端新增 live monitor 读模型

新增 session/task-run 级协调监控接口，直接从 `state_index + checkpoint + coordination diagnostics` 组装：

1. session 当前最新 task run
2. task run 基本状态
3. coordination run 基本状态
4. flow/current stage
5. node runs
6. handoff packets
7. working memory operations
8. merge result
9. graph spec 中的 nodes / edges

输出为轻量 monitor payload，不再包含全量事件回放。

### 3.2 前端新增 live monitor API

前端增加 `getOrchestrationRuntimeLoopSessionLiveMonitor(sessionId)`，不再走：

1. `listOrchestrationRuntimeLoopTaskRuns`
2. `getOrchestrationRuntimeLoopTrace`

这两步链路只保留给 task system 诊断页或历史查看。

### 3.3 前端把 live monitor 转成现有监控快照

为了不整体推翻 `CoordinationRunPanel`，先新增一个 monitor -> snapshot 的转换器：

1. 把 flow/node runs/runtime state 映射为 `OrchestrationSnapshot`
2. 用合成事件承载：
   - `coordination_flow`
   - `langgraph_runtime_state`
   - `coordination_node_run`
   - `coordination_merge_result`
   - `handoff_packets`
   - `working_memory_operations`
3. 让现有监控面板先无缝吃到实时数据

等这一层稳定后，再考虑把 `CoordinationRunPanel` 直接改成消费 live model。

### 3.4 前端建立独立轮询器

在 store runtime 内新增 monitor polling：

1. 当前 session 选中后启动轮询
2. session 切换或页面销毁时停止轮询
3. 每轮请求必须串行，避免并发叠加
4. 瞬时失败保留上一帧，不清空画面
5. 任务结束后可降频，但本轮先保持固定频率

## 4. 验证标准

### 4.1 代码级验证

1. 后端新增 monitor 聚合测试
2. 前端新增 live monitor -> snapshot 转换测试
3. 原有 trace 接口测试不回归

### 4.2 真实运行验证

至少做一次真实协调任务运行验证：

1. 启动协调任务
2. 前端监控页自动切到监控视图
3. 当前节点随执行推进变化
4. ready/running/completed 计数随阶段推进变化
5. 节点产出和工作记忆操作逐步出现
6. 任务完成后监控最终稳定停留在终态

## 5. 本次不处理

1. 不重写 `CoordinationRunPanel` 的整体 UI
2. 不改 trace 回放体系
3. 不引入 websocket
4. 不把所有非协调任务都并入该 live monitor 读模型

本轮目标只解决“协调任务监控必须实时可信”。

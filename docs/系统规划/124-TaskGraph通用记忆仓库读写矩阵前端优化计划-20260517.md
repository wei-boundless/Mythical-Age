# TaskGraph 通用记忆仓库读写矩阵前端优化计划

## 0. 计划定位

本计划用于优化 TaskGraph 通用编辑器中的“记忆与产物”页面。

核心目标不是新增一套特殊记忆类型，而是把记忆库做成通用资源节点，让用户在编辑器里直接配置：

```text
有哪些记忆仓库
仓库里有哪些 collection / record 类型
哪些节点可以读
哪些节点可以写候选
哪些节点可以提交
每次读取选择哪个版本
写入何时对后续节点可见
```

换句话说，记忆权限来自图结构和边配置：

```text
repository node
  + memory_read edge
  + memory_write_candidate edge
  + memory_commit edge
  + selector / version_selector
  + receipt visibility rule
```

不需要在系统层面把某些库定义成特殊库。只要没有写入边和提交边，它在运行中自然就是只读资源。

## 1. 当前代码事实

### 1.1 前端已有页面骨架

相关文件：

- `frontend/src/components/workspace/views/task-system/TaskGraphMemoryArtifactPage.tsx`
- `frontend/src/components/workspace/views/task-system/WorkingMemoryPolicyEditor.tsx`
- `frontend/src/components/workspace/views/task-system/ArtifactPolicyEditor.tsx`
- `frontend/src/components/workspace/views/task-system/taskGraphDraftV2.ts`

当前 `TaskGraphMemoryArtifactPage.tsx` 已有五个分面：

```text
repositories      仓库节点
contracts         读写契约
matrix            Clock 可见性
artifact_context  产物上下文
global            全局策略
```

已有能力：

- 能创建 `memory_repository / artifact_repository / progress_ledger / issue_ledger` 等资源节点。
- 能展示资源节点数量。
- 能编辑资源节点 `resource_lifecycle_policy`。
- 能编辑边的 `edge_type / repository / collection / version_selector / on_missing`。
- 能展示粗粒度 Scope × 记忆矩阵。
- 能编辑 artifact policy 和 working memory policy。

不足：

- 仓库结构还停留在自由文本 `collections`，没有明确 schema / collection / record_kind / key_strategy。
- 读写边的配置不够精确，没有形成统一 selector 模型。
- matrix 只是按 phase 粗略统计读写，没有形成“节点 × 仓库/collection × 操作”的可编辑权限矩阵。
- 缺少 write candidate 和 commit 的显式分离。
- 缺少 receipt 后可见性规则配置。
- 缺少“节点实际会收到的 MemorySnapshot 预览”。
- artifact 上下文和记忆上下文尚未在同一套 packet/receipt 语言下表达。

相关文件：

- `backend/tasks/task_graph_models.py`
- `backend/tasks/layered_graph_normalizer.py`
- `backend/orchestration/runtime_loop/context_packet_resolver.py`
- `backend/orchestration/runtime_loop/stage_execution_request.py`
- `backend/orchestration/runtime_loop/stage_execution_receipt.py`
- `backend/orchestration/runtime_loop/timeline_ledger.py`

已有基础：

- 节点类型包含 `memory_repository / memory_collection / artifact_repository / progress_ledger / issue_ledger / working_memory_store`。
- 边已有 `edge_type / context_filter_policy / artifact_ref_policy / working_memory_handoff_policy / metadata`。
- 运行时已开始支持 `MemorySnapshot / ArtifactContextPacket / RevisionPacket / HandoffPacket / StageExecutionReceipt`。
- TimelineLedger 已提供 `clock_seq / scope_path`。

这意味着前端不应该发明新语义，而应该把这些后端对象变成可配置、可预览、可校验的通用编辑体验。

## 2. 设计原则

### 2.1 记忆仓库是通用资源节点

仓库节点只表达存储结构和生命周期，不表达领域身份。

推荐节点模型：

```text
node_type = memory_repository
resource_lifecycle_policy
  versioning
  commit_required
  default_visibility
  retention_policy
metadata.memory_repository
  repository_id
  title
  schema_id
  collections
```

### 2.2 读写权限来自边

节点能读什么、写什么，不由仓库名称决定，而由边决定。

```text
memory_read            repository -> task node
memory_write_candidate task node -> repository
memory_commit          reviewer/committer node -> repository
memory_handoff         node -> node
```

没有写入边的仓库，在运行时自然不可写。

### 2.3 selector 必须显式

不要让 Agent 或运行时模糊搜索整个仓库。读取必须由 selector 决定。

```text
selector
  collection
  record_kinds
  record_keys
  scope_filter
  status_filter
  tag_filter
  limit

version_selector
  latest_committed_before_clock
  latest_committed_before_scope
  pinned_version
  by_receipt
  manual_snapshot
```

RAG 可以作为辅助检索策略，但不能替代主路径的定向读取。

### 2.4 写入必须分候选和提交

写入不应直接污染后续节点。

```text
node output
  -> memory_write_candidate
  -> validation / review
  -> memory_commit
  -> receipt
  -> visible after clock/scope
```

前端必须把 candidate 和 commit 分开显示。

### 2.5 MemorySnapshot 是节点实际输入预览

用户最终要能看到：

```text
这个节点运行时会收到哪些记忆
来自哪些仓库和 collection
由哪些边授权
使用哪个 version_selector
缺失时是否阻断
在 prompt 中如何称呼
```

## 3. 目标信息架构

`TaskGraphMemoryArtifactPage` 保留一个主层级，不新增主导航页。页面内部重组为五个更清晰的分面：

```text
仓库结构
读写矩阵
Selector 配置
Snapshot 预览
产物上下文
```

原有 `global` 分面可以合并到仓库结构和发布预检中，避免变成杂项堆放处。

## 4. 页面结构设计

### 4.1 仓库结构

用途：

- 创建资源节点。
- 管理仓库 schema。
- 管理 collection。
- 设置版本和提交策略。

建议 UI：

```text
左侧：仓库列表
中间：仓库结构编辑
右侧：结构摘要与风险
```

每个仓库显示：

- repository id
- node id
- schema id
- collection 数
- read edge 数
- write candidate edge 数
- commit edge 数
- 是否有未授权写入风险

每个 collection 配置：

```text
collection_id
title
record_kinds
key_strategy
schema_ref
default_version_selector
required_receipt_status
```

### 4.2 读写矩阵

用途：

- 直接控制节点对仓库/collection 的权限。
- 让用户一眼看到哪些节点能读、写候选、提交。

推荐矩阵：

```text
行：业务节点 / 审核节点 / 提交节点
列：仓库.collection
格子：forbidden / read / write_candidate / commit / read+write_candidate
```

点击格子：

- 新增或修改对应 memory edge。
- 配置 selector。
- 配置缺失行为。
- 配置模型可见别名。
- 配置是否进入 prompt。

格子状态来自真实图边，而不是单独保存 UI 状态。

### 4.3 Selector 配置

用途：

- 精确定义节点读取哪些记录。
- 精确定义节点写入哪些 candidate。

读边配置：

```text
edge_type = memory_read
source_node_id = repository node
target_node_id = task node
metadata.selector
metadata.version_selector
metadata.model_visible_label
metadata.usage_instruction
metadata.on_missing
```

写候选边配置：

```text
edge_type = memory_write_candidate
source_node_id = task node
target_node_id = repository node
metadata.target_collection
metadata.allowed_record_kinds
metadata.write_mode
metadata.requires_receipt
```

提交边配置：

```text
edge_type = memory_commit
source_node_id = commit/review node
target_node_id = repository node
metadata.candidate_source_node_ids
metadata.accepted_receipt_status
metadata.effective_scope_policy
```

### 4.4 Snapshot 预览

用途：

- 预览节点运行时真正能读到什么。
- 将配置和运行时 `MemorySnapshot` 对齐。

预览内容：

```text
node_id
clock/scope placeholder
read_edge_ids
repository_refs
collection selectors
version selectors
missing required records
model visible labels
usage instructions
```

短期没有后端实时 preview API 时，前端先做静态 preview：

- 从当前 graph nodes/edges 计算授权读取包。
- 标注“配置预览，不是运行时快照”。

后续接入后端时，升级为真实 `StageExecutionRequest.memory_snapshot` 预览。

### 4.5 产物上下文

用途：

- 让 artifact 和 memory 的交接语义统一。
- 显示 artifact context edge 如何进入节点执行包。

保留现有 artifact policy 能力，但改成：

```text
artifact_repository
artifact_write_candidate
artifact_context_read
artifact_commit / receipt
```

至少要在 UI 上标明：

- 当前节点读哪个 artifact packet。
- 当前节点输出到哪个 artifact target。
- 下游是否通过 receipt 读取已接受产物。

## 6. 配置交互流程

### 6.1 从仓库出发

1. 用户创建仓库节点。
2. 添加 collection。
3. 选择 record kinds 和版本策略。
4. 在读写矩阵中给节点授权。
5. 系统自动生成对应 memory edge。
6. 用户点开格子补 selector 和使用说明。

### 6.2 从节点出发

1. 用户选中节点。
2. 查看“该节点可读记忆”。
3. 查看“该节点可写候选”。
4. 查看“该节点可提交记录”。
5. 预览 MemorySnapshot。
6. 跳转到职责与交接页补 prompt 使用说明。

### 6.3 从边出发

1. 用户选中 memory edge。
2. 配置 selector。
3. 配置 version selector。
4. 配置缺失行为。
5. 配置 model visible label。
6. 配置 usage instruction。

## 7. 发布预检

需要新增或强化以下预检：

- `memory_read_missing_selector`
  - memory_read edge 没有 selector。
- `memory_read_missing_usage_instruction`
  - 模型可见记忆没有使用说明。
- `memory_write_without_candidate_mode`
  - 写入边没有标记 candidate/commit。
- `memory_commit_without_receipt`
  - 提交边没有绑定 receipt 条件。
- `repository_collection_missing_schema`
  - collection 没有 record kinds 或 schema。
- `node_has_memory_write_without_target_collection`
  - 节点写入没有目标 collection。
- `readonly_by_topology`
  - 仓库没有写入边，这是信息提示，不是错误。
- `uncommitted_memory_visible_downstream`
  - 下游读取了未提交候选，这是错误。

## 10. 不做事项

- 不引入特殊记忆库类型。
- 不把领域概念写死进通用编辑器。
- 不让前端保存独立权限矩阵副本。
- 不用 RAG 配置替代 selector。
- 不把长 prompt 正文塞进 TaskGraph 节点。
- 不让未提交候选默认对下游可见。

## 11. 验收标准

完成后，用户应能做到：

1. 创建通用记忆仓库和 collection。
2. 在矩阵里配置每个节点对每个 collection 的 read/write_candidate/commit 权限。
3. 点开权限格子配置 selector 和 version selector。
4. 预览节点会收到的 MemorySnapshot。
5. 在发布前发现 selector、commit、receipt、usage instruction 的配置缺口。
6. 运行时仍由后端 TimelineLedger、ContextResolver、Receipt 决定真实可见性。

最终目标是让 TaskGraph 编辑器成为真正的通用任务设计工作台：用户通过图结构和边规则控制记忆读写，而不是依赖 Agent 自己猜上下文。

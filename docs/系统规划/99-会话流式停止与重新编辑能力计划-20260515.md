# 会话流式停止与重新编辑能力计划

日期：2026-05-15

## 问题

多会话并行流式后，前端仍缺少通用停止控制：

- 正在运行的会话只能等流式自然结束。
- 用户停止后不能直接拿回刚才输入的内容重新编辑。
- 如果仍使用全局停止，会破坏多会话并行流式模型。

## 目标

- 每个会话的流式请求都有独立停止能力。
- 停止当前会话时，不影响其他会话的流式请求。
- 停止后当前输入框恢复可编辑。
- 用户消息提供显式“编辑”入口；编辑后重发会截断该轮之后的服务端会话历史。
- 被停止的 assistant 占位消息显示“已停止”状态，避免被误判为错误。

## 实施步骤

1. `streamChat` 支持传入 `AbortSignal`，用于中断当前 fetch/SSE 流。
2. Runtime 为每个正在流式的 session 维护 `AbortController`。
3. 新增 `stopCurrentStream` action，只停止当前会话对应的流式请求。
4. 流式 reducer 增加 `stopped` 事件，显示“已停止”状态。
5. `ChatInput` 在当前会话流式时展示停止按钮。
6. 消息模型保留服务端消息下标，用户消息提供显式编辑表单。
7. 服务端新增会话历史截断接口；编辑重发时先删除该用户消息及其之后的历史，再用新文本发送。

## 验证

- `npm run build` 通过。
- A 会话流式时可停止 A。
- A 停止不影响 B 会话正在流式。
- 停止后的消息状态不是错误，而是“已停止”。
- 点用户消息“编辑”后重发，服务端历史从该轮重新开始。

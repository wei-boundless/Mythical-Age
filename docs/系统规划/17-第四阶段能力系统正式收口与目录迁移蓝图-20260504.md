## 二、第四阶段的真实问题

第四阶段的核心问题不是：

`把 operations 改个名字`

而是：

`把平台能力资源供应层从历史混装目录里正式抽出来，锁定系统边界、正式命名、正式目录和正式供给出口`

在第三阶段之后，源码已经开始显露出真实结构：

- tool / skill / worker / capability endpoint 的目录化供给已经存在
- ResourcePolicy / OperationGate / AgentRegistry 的运行约束逻辑已经更接近编排系统
- 任务系统已经有自己的 capability requirement 对象

这说明真正的问题不是能力不存在，而是：

1. 历史名字 `operations` 同时承载了能力供给、运行约束和任务侧 requirement
2. “操作系统”这个名字会把能力供应层误听成执行/编排/运行时本体
3. API、测试、前端工作台和文档仍然停留在旧语义
4. `backend/operations` 仍像正式根目录，但实际上已经变成兼容壳

因此第四阶段必须解决的是：

`把能力系统从历史杂糅边界中正式收口，成为唯一能力资源供给出口`

## 三、为什么正式改名为“能力系统”

“操作系统”这个名字在本项目里有两个持续副作用：

1. 它容易让人误以为这个系统负责 runtime 执行本身  
2. 它会把 ResourcePolicy、OperationGate、worker route、tool bridge 和 capability catalog 混成一个执行黑箱

但从实际设计原则看，这个系统真正负责的是：

- 平台有哪些能力
- 能力以什么对象登记
- 能力如何被 skill、worker、endpoint、tool 暴露
- 能力如何被任务系统引用
- 能力如何被编排系统授权

因此更贴切的正式命名应是：

`能力系统`

这个名字的好处是：

- 强调它是资源供应层，不是运行控制层
- 更容易和编排系统、任务系统、健康系统分开
- 更符合外部先进架构里常见的 capability catalog / tool registry / skill registry / worker endpoint 语义

## 四、当前源码的正式归属判断

## 4.1 属于能力系统的对象

当前以下对象应视为能力系统正式对象：

- `ToolRecord` 类对象
- `SkillRecord` 类对象
- `WorkerCapabilityRecord` 类对象
- `CapabilityEndpoint` 类对象
- `OperationDescriptor`
- `OperationRegistry`
- `CapabilityCatalog`
- `CapabilityValidationIssue`

对应源码已经迁入或应迁入：

- `backend/capability_system/catalog.py`
- `backend/capability_system/models.py`
- `backend/capability_system/endpoints.py`
- `backend/capability_system/validation.py`
- `backend/capability_system/operation_registry.py`
- `backend/capability_system/worker_registry.py`
- `backend/capability_system/validators/*`

## 4.2 属于编排系统的对象

以下对象虽然过去在 `operations` 下面，但本质属于编排系统：

- `AgentRegistry`
- `ResourcePolicy`
- `ResourceDecision`
- `ResourceRuntimeView`
- `OperationGate`
- `RuntimeApprovalContext`

这些对象负责的是：

- 单 agent 可执行范围
- runtime 批准与拒绝
- adopted resource policy
- 真正执行时的 gate 检查

因此它们属于编排系统，而不是能力系统。

## 4.3 属于任务系统的对象

以下对象属于任务系统：

- `OperationRequirement`

原因很简单：

它表达的是任务侧对能力的需求，不是能力本身，也不是编排执行时的采用结果。

## 五、第四阶段的目标目录结构

第四阶段结束后，正式目录目标应为：

```text
backend/
  capability_system/
    catalog.py
    endpoints.py
    models.py
    operation_registry.py
    worker_registry.py
    validation.py
    validators/

  orchestration/
    agent_registry.py
    resource_policy.py
    resource_policy_builder.py
    resource_runtime_view.py
    resource_gate.py

  tasks/
    capability_requirements.py

  api/
    capability_system.py
```

同时：

- `backend/operations` 不再作为正式系统根目录
- `backend/capabilities` 不再作为真实系统边界
- `backend/api/operations.py` 不再保留为正式 API 文件

## 六、第四阶段的执行约束

本阶段必须严格遵守以下约束。

## 6.1 能力系统只负责供给，不负责执行决策

能力系统可以：

- 暴露 catalog
- 暴露 tool / skill / worker / endpoint 元数据
- 暴露 operation registry
- 做静态结构校验

能力系统不能：

- 直接决定 task 是否允许执行
- 直接 adopted resource policy
- 直接替代编排系统做 gate

## 6.2 编排系统只消费正式能力对象

编排系统后续只能读取：

- `OperationRegistry`
- `OperationDescriptor`
- `CapabilityCatalog`

不再允许：

- 直接扫描底层 tools 目录
- 直接猜 worker 目录结构
- 直接从 skill 文件反推资源授权

## 6.3 任务系统只声明需求，不拼执行权限

任务系统后续只能输出：

- required operations
- optional operations
- denied operations

不再允许：

- 直接构造 runtime executable policy
- 直接伪造 allow rule

## 七、第四阶段实施步骤

## 7.1 第一步：正式命名切换

统一把以下表达改成能力系统：

- 文档标题
- API 文件名
- API 路径
- 前端工作台标题
- 测试 owner_system
- 健康系统问题归属标签

这是第四阶段的第一优先级，因为只要命名继续混乱，后面的目录治理就会继续漂移。

## 7.2 第二步：正式目录切换

把真实源码归属切到：

- `backend/capability_system`
- `backend/orchestration`
- `backend/tasks`

并确保：

- `backend/operations` 只剩过渡 re-export
- 过渡层不再增加新代码

## 7.3 第三步：正式 API 切换

把能力系统 API 切换为：

- `/api/capability-system/catalog`
- `/api/capability-system/catalog/refresh`
- `/api/capability-system/skills/*`
- `/api/capability-system/tools/*`
- `/api/capability-system/agents`

切换原则：

- 新前端直接走新路径
- 不再继续扩展旧 `/operations/*`

## 7.4 第四步：测试治理同步

测试系统、健康系统和 harness map 必须同步切换：

- `owner_system = capability_system`
- 用例标题改成“能力系统”
- 测试文件名改成 `capability_system_*`

否则健康系统仍会把问题归到历史系统名下。

## 7.5 第五步：工作台入口同步

前端需要同步切换：

- 右侧导航标签
- 工作台视图 key
- 系统框架图节点标签
- 健康报告和测试分析中的 owner label

## 八、第四阶段切换完成条件

以下条件全部满足，第四阶段才算正式完成：

1. 后端正式 API 已改为 `capability-system`
2. 前端正式工作台入口已使用“能力系统”
3. `capability_system` 已成为正式代码根目录
4. `AgentRegistry / ResourcePolicy / OperationGate` 已不再归属能力系统
5. 测试系统与健康系统 owner label 已切换
6. 定向回归通过

## 九、第四阶段不做的事情

本阶段明确不做：

- 不重写工具实现本体
- 不重写技能文件体系
- 不重写 worker 执行链
- 不在这一阶段重做 orchestration runtime 模型

第四阶段是边界收口和目录迁移阶段，不是能力实现重写阶段。

## 十、残留清单

本阶段完成后仍允许短暂存在的残留：

- `backend/operations/__init__.py`
- `backend/capabilities/__init__.py`
- 文档中作为历史分析引用的 `backend/operations/...` 路径

但这些残留的定位必须明确：

`它们只是迁移参考或兼容层，不再是正式系统边界`

## 十一、切换后的结论

第四阶段完成后，平台正式系统边界应进一步清晰为：

- 任务系统：决定做什么
- 编排系统：决定怎么装配、怎么授权、怎么运行
- 能力系统：决定平台有什么能力资源可供给
- 记忆系统：决定记忆如何治理和供给
- 灵魂系统：决定身份与 prompt 投影
- 健康系统：决定如何诊断、验证和门禁

能力系统的正式建立，意味着平台终于有了一个可管理、可验证、可供给的能力资源供应层，而不是继续让 tool、skill、worker 和 endpoint 散落在历史目录和隐式契约里。

# 编排系统能力权限卡片池与详情面板重构计划书

## 背景

编排系统的 Agent 运行档案实际保存 `allowed_operations` 与 `blocked_operations`。当前前端把它们展示成两个原始多选列表，用户只能看到操作 ID，无法判断能力来源、风险、用途，也无法理解 skill、tool、MCP 与 operation 授权之间的关系。

后端能力系统已经拆出三类来源：

- Skills：模型可见的任务能力入口，包含 `prompt_view`、`preferred_route`、任务类型和模态信息。
- Tools：本地工具定义，包含 `operation_id`、安全标签、运行可见性、资源暴露策略和风险元数据。
- MCP：本地 MCP 能力端点，包含 `operation_id`、路由、传输方式、输入输出模式和关联 skill。

因此本次不把内置 Agent 或具体旧字段做特殊处理，而是把“能力来源”和“运行授权操作”分层显示。

## 目标

1. 权限页主区域改为三个卡片池：`Skills`、`Tools`、`MCP`。
2. 每张卡显示能力来源、能力名称、依赖 operation、风险和当前授权状态。
3. 卡片内部提供 `允许` 和 `禁止` 两个操作按钮：
   - 点 `允许`：把该能力依赖的 operation 加入允许列表，并从禁止列表移除。
   - 点 `禁止`：把该能力依赖的 operation 加入禁止列表，并从允许列表移除。
   - 点卡片其它位置：只选中卡片，在右侧查看详情。
4. 允许状态使用亮边和轻微发光，禁止状态降亮度，未配置保持中性。
5. 右侧从“权限摘要”改为“能力详情”，展示：
   - 来源分类
   - 风险说明
   - 必要能力介绍
   - 依赖授权操作
   - 与当前运行档案的允许/禁止关系
6. 如果某类能力不能清晰说明来源、风险或用途，优先暴露为“注册信息不足”，后续再回到能力系统注册层补字段，而不是在 UI 里硬编故事。

## 分类规则

- Skill 池：来自 `/capability-system/catalog.skills`。Skill 本身不是 operation，按 `preferred_route` 映射到 MCP operation：
  - `rag` -> `op.mcp_retrieval`
  - `pdf` -> `op.mcp_pdf`
  - `structured_data` -> `op.mcp_structured_data`
- Tool 池：来自 `/capability-system/catalog.tools`，直接使用 `tool.operation_id`。
- MCP 池：来自 `/capability-system/catalog.capability_supply_package.mcp_refs`、`binding_graph.mcp_nodes` 或 operation 类型为 `mcp` 的后备数据，直接使用 MCP 的 `operation_id`。

## 实施范围

- 前端 `OrchestrationView` 加载能力系统目录，并传给权限工作台。
- 前端 `OrchestrationPermissionsWorkbench` 负责能力卡片池、授权切换和详情面板。
- 类型层补充 `CapabilitySystemCatalog` 的 MCP 字段。
- CSS 增加权限卡片池、状态按钮、详情面板样式。

## 验证

- 运行 `npx tsc --noEmit`。
- 确认不再保留旧的“允许操作/阻断操作”双列表 UI。
- 确认卡片允许/禁止会真实写入 `runtimeDraft.allowed_operations_text` 与 `runtimeDraft.blocked_operations_text`。

# TaskGraph 通用编辑器重构实施报告

## 0. 报告目的

本报告用于记录根据《120-TaskGraph 时序主轴与确定性记忆闭环工程实施计划》推进 TaskGraph 通用编辑器与运行时重构时的实际改动。

它不是新的设计书，而是实施过程的审计账本。每次代码改动必须说明：

- 为什么改。
- 改了哪些文件。
- 属于计划书哪一阶段。
- 是否改变了通用架构边界。
- 是否引入写作任务特化。
- 用什么测试验证。
- 有哪些未完成风险。

## 1. 当前架构边界锁定

实施期间必须维持以下边界：

```text
TaskGraph Definition
  负责用户可编辑的通用拓扑与引用。

TaskGraph Scheduler
  负责 ready / blocked / running / completed 判断。
  不负责分配语义 clock。

LangGraph Runtime Kernel
  负责物理执行、checkpoint、resume、stream、interrupt 底座。
  不负责 TaskGraph 业务语义。

TimelineLedger
  负责语义 clock_seq、scope_path、事件因果与审计历史。

Context Resolvers
  负责按 dispatch context 装配 memory/artifact/revision/handoff packets。

Receipt Layer
  负责节点输出的 accepted/rejected/committed 状态与后续可见性。

Projection / Prompt System
  负责 Agent 可理解的身份、任务、输入使用方式和输出要求。
```

## 2. 禁止偏离项

- 不把节点定义成时序。
- 不让 Scheduler 分配 clock。
- 不把 LangGraph checkpoint 当成 TaskGraph 语义时序。
- 不为写作任务硬编码通用 runtime/editor。
- 不用 latest 文件推断当前输入。
- 不把返修主路径放在 `pending_inputs`。
- 不绕过测试。

改动内容：

- 新增 `LangGraphRuntimeKernel`，把 `StateGraph.invoke()` 与 coordination checkpoint 写入收束成物理运行边界。
- 新增 `TimelineLedgerStore`，用 append-only JSON ledger 为每个 coordination run 分配单调 `clock_seq`。
- Runtime 在 `run_started / scheduler_evaluated / node_dispatch_requested / node_result_received / checkpoint_linked` 等位置写入语义事件。
- `TraceReader` 从 TimelineLedger 读取语义时序快照，监控视图不依赖 checkpoint state 猜时序。

架构影响：

- LangGraph 仍只负责物理推进与 checkpoint。
- Scheduler 仍只负责 ready/blocked 判断，没有分配 clock。
- TimelineLedger 成为 TaskGraph 语义时间的唯一 owner。

通用性检查：

- 是否引入领域特化：否。
- 是否改变 TaskGraph/Scheduler/LangGraph/TimelineLedger 边界：是，按计划新增 TimelineLedger 与 LangGraphRuntimeKernel 边界。

测试：

- `python -m pytest backend/tests/timeline_ledger_regression.py`
- 已通过。

遗留问题：

- 尚未切换 LangGraph 原生 checkpointer；当前仍使用项目 JSON checkpoint adapter 作为 bridge。

改动内容：

- `StageExecutionRequest` 新增 `dispatch_context / memory_snapshot / artifact_context_packet / revision_packet / handoff_packet_refs / execution_receipt_policy`。
- `idempotency_key` 优先绑定 `dispatch_event_id`，同节点多轮返修或循环不再复用同一个执行身份。
- 新增 `ContextPacketResolver`，按当前 dispatch context 装配冻结的记忆快照、产物上下文包、返修包和交接包引用。
- 新增 `StageExecutionReceipt`，节点结果进入 `stage_results` 时同步生成 accepted/rejected 回执。
- 审核退稿时生成 `RevisionPacket`，返修节点通过 request 获取被审原稿、审核结果和修改要求。
- A2A payload metadata 带入 dispatch/timeline 引用，但不把 resolver 交接包重复塞进旧 handoff 字段，避免破坏现有 A2A 合约。

架构影响：

- 节点执行入口从“散乱 explicit_inputs”升级为“dispatch context + 确定性交接包”。
- rejected 输出不会作为 committed receipt 对后续可见。
- Monitor 可以显示当前 dispatch、packets、receipts。

通用性检查：

- 是否引入领域特化：否。
- 是否改变 TaskGraph/Scheduler/LangGraph/TimelineLedger 边界：否，新增的是 runtime context resolver 与 receipt 层。

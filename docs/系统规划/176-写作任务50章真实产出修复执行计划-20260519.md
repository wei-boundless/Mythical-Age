当前干净运行 `writing-simple-novel-honghuang-20260519-101428` 已完成：

`project_brief -> world_design -> world_review -> memory_commit_world -> character_design -> plot_design -> design_sync -> outline_design -> outline_review -> baseline_memory_seed`

这些上游资产已经形成可信基准。当前阻塞在 `volume_plan`：

1. 首轮 `volume_plan` 产物泛化，未继承基准库与已审核大纲，被正确失效并移动到 `invalidated`。
2. 断点重跑后执行器失败，只产生 debug 报告，没有生成 `volume_plan/volume_plan.md`。
3. 下游章节循环尚未恢复，因此无法继续推进到 50 章真实产出。

本次修复不允许伪造文件，也不允许绕过审核。目标是让任务系统按真实流程恢复并持续产出。

## 目标流程

`baseline_memory_seed -> volume_plan -> chapter_outline -> chapter_draft -> chapter_review -> memory_commit_chapter -> chapter_progress_router`

其中章节循环每批 10 章，至少完成 5 个已提交批次：

- 第 1-10 章
- 第 11-20 章
- 第 21-30 章
- 第 31-40 章
- 第 41-50 章

## 关键验收标准

### 分卷规划

`volume_plan/volume_plan.md` 必须真实继承当前项目基准，不得出现泛化模板。至少应包含：

- `大泽·沉碑`
- `泽`
- `黎`
- `漪`
- `烬`
- 七条伏笔或明确的伏笔与回收窗口承接表
- 第 1 卷第 1-10 章的章节循环接口

不得出现未来自基准库的泛化卷名：

- 混沌初开
- 万族争锋
- 天道崩塌
- 重塑乾坤
- 洪荒纪元

### 章节循环

每个批次必须满足：

1. 先有当前批次细纲。
2. 再有当前批次正文候选。
3. 再有章节审核。
4. 再由记忆提交节点登记本批摘要、文件引用、连续性增量和问题状态。
5. 下一批读取上一批提交摘要、上一批审核意见、基准库、分卷计划和可改动记忆。

### 真实产出

50 章必须位于有效 artifact root 下，不能在 `rejected` 或 `invalidated` 目录中。每批正文应为完整正文，不得用摘要、提纲、说明或“等待补充”代替。

## 修复原则

1. 先定位结构性原因：交接包缺失、执行器失败、流式恢复失败、产物抽取失败、还是质量门误判。
2. 若问题是提示过宽，收窄节点协议，让 `volume_plan` 只做当前卷与当前批次接口，不重写全书。
3. 若问题是交接包缺失，修复边契约和输入包构造，不在 prompt 内硬编码补偿。
4. 若问题是执行活性，修复超时、失败事件和断点重派发，不让节点永久半失败。
5. 若问题是产物抽取，修复物化器和必需文件判断，不把 debug 报告当业务产物。

## 执行步骤

1. 读取失败任务运行记录，确认 `executor_failed` 的直接原因。
2. 检查 `volume_plan` 节点执行请求的输入包，确认是否实际携带基准库和已审核大纲。
3. 修复运行时或配置。
4. 运行回归测试，至少覆盖：
   - 任务产物物化
   - LangGraph 协同运行
   - 节点执行请求
   - 写作任务配置
5. 重新生成写作任务配置。
6. 从可信上游恢复或新开运行。
7. 监控至 `volume_plan` 通过。
8. 继续推进章节循环，直到有效提交至少 50 章。
9. 检查 50 章是否连续继承世界观、大纲、人设和伏笔结构。

## 风险

1. 单批 10 章正文较长，模型调用可能超时。若出现执行器超时，应优先修复有界恢复和批次质量重试，而不是降低章节真实性。
2. 章节审核过严可能导致循环停滞。审核应阻止设定跑偏和正文缺失，不应把正常可修的小问题当作门禁阻断。
3. 章节之间的连续性不能只靠 prompt 记忆，必须由 `memory_commit_chapter` 把上一批摘要和状态写入可读索引。

## 追加问题：同源草稿重复运行

### 现象

在 `chapter_outline` 修复后，多次继续或回退请求因为 HTTP 调用超时而被重复发起，但后台线程仍在执行，最终同一个 `chapter_draft` 阶段出现一个已完成草稿和多个后续同源草稿。

### 根因判断

1. `NodeExecutionRequest` 已经携带 `request_id` 与 `idempotency_key`，但 API 调度入口没有先查询同一请求是否已经在运行或已完成。
2. `_continue_coordination_delivery_stream` 在缺少稳定 `turn_id` 时会生成随机 `turn:{session}:{uuid}`，导致同一阶段请求被实例化为多个不同 `task_run_id`。
3. 任务运行记录没有稳定保存阶段调度身份，事后只能从 `task_id/task_run_id` 反推阶段，难以及时合并重复请求。

### 修复目标

1. 同一个 `coordination_run_id + stage_id + idempotency_key/request_id` 只能有一个非终止执行实例。
2. 同一个节点执行请求进入运行层时必须得到稳定 `turn_id/task_id`，重复调用应命中同一任务实例。
3. 继续和回退 API 返回 `background_started=false` 与已存在任务引用，而不是静默再开一个后台线程。
4. 允许正式重试：只要回退或重派发产生新的 `idempotency_key/request_id`，就可以启动新的执行实例。

### 实施点

1. 在调度入口增加阶段执行身份查询，发现同源运行中或已完成实例时跳过后台线程。
2. 在继续执行流中用 `NodeExecutionRequest.idempotency_key/request_id` 派生稳定 `turn_id`，消除随机实例分叉。
3. 在任务运行诊断中写入 `coordination_run_id/stage_id/stage_request_id/stage_idempotency_key`。
4. 增加回归测试覆盖同源请求稳定命名、重复调度跳过、不同幂等键允许重试。

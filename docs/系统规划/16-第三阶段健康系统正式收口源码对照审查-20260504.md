## 二、当前已经符合设计书的部分

### 2.1 健康系统开始消费正式编排对象

健康系统预览与准入现在不再只看：

- `TaskFlowDefinition`
- `TaskAgentBinding`
- `AgentRuntimeProfile`

而是会先构造正式运行计划，并显式拿到：

- `TaskExecutionAssembly`
- `TaskBodyOrchestration`
- `AgentRuntimeSpec`

这说明健康系统已经从“binding 级检查”进入“runtime spec 级检查”。

### 2.2 健康系统执行主链已统一

以下链路已统一围绕同一份 plan 工作：

- `preview_agent_run`
- `admit_health_command`
- `start_agent_run`
- `build_agent_run_trace_report`

这意味着健康系统不再各自维护一份隐式运行结构。

### 2.3 历史健康 agent 别名已从主链移除

主链已统一使用：

- `agent:3`

不再继续把：

- `agent:health:maintainer`

作为并行正式身份参与运行预览、投影默认值或健康样例数据。

### 2.4 `HealthRegistry` 已开始瘦身

当前已经从 `HealthRegistry` 中抽走两类高耦合职责：

- `HealthStore`：负责健康系统 JSONL 仓储读写
- `HealthCommandBuilder`：负责 command / receipt / report / response 组装

这意味着 `HealthRegistry` 不再自己持有仓储细节和命令对象构造细节，已经开始回到“协调入口”定位。

### 2.5 verification / gate 已进入健康系统正式解释层

本轮新增了：

- `HealthVerificationRun`
- `RegressionGateDecision`

并由 `HealthVerificationService` 统一生成：

- verification run 列表
- verification resource catalog
- gate projection

这说明：

- health workbench 不再直接把 `test_system_service` 的原始结果当正式对象
- regression gate 已开始被健康系统重新解释

### 2.6 workbench 已切到正式对象投影

`HealthWorkbenchBuilder` 当前读取的关键对象已经变成：

- `issues`
- `verification_runs`
- `gate_projection`
- `verification_resources`

而不是直接读取：

- `test_system.harness_map`
- `test_system_service.list_runs()` 原始返回值

## 三、仍然存在但已收敛的结构问题

### 3.1 `HealthRegistry` 仍偏大

虽然 `execution_planner`、`trace_builder`、`store`、`command_builder` 已经拆出，但 `HealthRegistry` 仍同时承担：

- 仓储读写
- command application service
- agent run execute
- report / receipt 组装

其中“仓储读写”“report / receipt 组装”“health command application service”“health verification / gate service”都已经开始外移。

当前还没彻底拆完的主要部分变成：

- health agent run execute
- health resource policy builder
- health model message builder

这说明第三阶段已经完成主链收口，但还没有完成健康系统内部的彻底分层。

### 3.2 健康专用对象名仍偏实现层

当前已经有正式运行计划，但健康系统对象层本身仍主要沿用：

- `HealthIssue`
- `HealthAgentRun`
- `HealthReport`
- `HealthTestRun`

后续如果要继续对齐设计书，还应补出更稳定的系统层命名，例如：

- `HealthTraceRecord`
- `HealthDiagnosticReport`
- `RuntimeAdmissionDecision`
- `RegressionGateDecision`

如果继续推进第三阶段后半段，建议按下面顺序：

1. 把 `HealthRegistry` 再拆成 `store + command_service + verification_service`
2. 把 `harness/regression_gate.py` 的正式解释层并入 `health_system`
3. 为 `trace_analysis / case_draft / fix_verification` 补齐健康专用模板层，而不是继续复用 triage 倾向模板

# Harness 概念收口与状态协议重构计划书

日期：2026-05-27

状态：待实施计划

## 1. 背景

上一轮迁移已经完成文件级和入口级收口：

```text
TaskRunLoop -> HarnessServiceHost
runtime/unit_runtime -> harness/service_host + harness/loop/*
runtime/coordination_runtime -> harness/loop/graph_coordination/*
runtime/execution_engine -> harness/loop/agent_execution
runtime/execution_permit -> harness/runtime/execution_policy
api/orchestration_runtime_loop.py -> api/orchestration_harness.py
/orchestration/runtime-loop/* -> /orchestration/harness/*
langgraph_runtime_state -> graph_coordination_state
```

最新扫描结果显示，旧公开入口已经基本切断，但概念层仍有一批 `RuntimeLoop*` 和 `runtime_loop_*` 残留。它们不全是同一种问题，不能机械改名。

本计划只处理 Harness 控制体系的概念收口，不处理健康系统重构、不处理任务系统新合同实现、不处理文件/工具/权限系统的新功能扩展。

## 2. 设计原则

本轮以 `docs/系统框架/001-Agent系统框架总纲-20260527.md` 为最高边界：

```text
Harness 是 agent 管控系统。
Runtime 负责阶段性上下文装配。
Loop 负责动态状态推进。
Agent 负责语义判断、计划和行动请求。
系统负责环境、权限、工具、文件、记忆、沙盒和审计护栏。
```

必须维持两条主链：

```text
AgentHarness -> Harness Runtime -> AgentLoop
GraphHarness -> GraphLoop -> AgentHarness/AgentLoop
```

禁止继续出现：

```text
能力长 runtime
状态协议挂在旧 runtime 顶层
前端 API 继续以 RuntimeLoop 作为控制面名称
任务系统通过 runtime_loop_policy 反向拥有 loop 主权
为了兼容保留旧事件双通道
```


唯一命中是边界测试自身的禁止断言：

```text

```

### 3.2 未完成

残留分为四类。

第一类：Harness 正在使用但仍放在 `runtime.shared` 的状态模型。

```text
backend/runtime/shared/models.py::RuntimeLoopState
backend/runtime/shared/loop_control.py::RuntimeLoopLimits
backend/runtime/shared/loop_control.py::RuntimeLoopControlDecision
backend/runtime/shared/loop_control.py::check_runtime_loop_control
backend/runtime/shared/checkpoint.py
```

问题：

```text
LoopState 是 Harness Loop 的动态状态，不应由 backend/runtime 顶层拥有。
checkpoint store 是 Harness 的状态记录基础设施，不应继续表达为 runtime.shared。
```

第二类：SSE / stream 事件协议仍叫 `runtime_loop_*`。

```text
runtime_loop_started
runtime_loop_event
runtime_loop_control
```

问题：

```text
事件协议是前后端和测试共同依赖的公开协议。
它已经不是旧 runtime 文件入口，但仍在命名上表达旧控制系统。
```

第三类：前端 API 类型和函数仍叫 `RuntimeLoop*`。

```text
RuntimeLoopTaskRunTrace
RuntimeLoopTaskRunLiveMonitor
listOrchestrationRuntimeLoopTaskRuns
getOrchestrationRuntimeLoopTrace
startTaskGraphRuntimeLoopRun
```

问题：

```text
前端已请求 /orchestration/harness/*，但类型和函数仍向 UI 层传播旧概念。
```

第四类：图任务配置字段仍叫 `runtime_loop_policy / runtime_loop_summary`。

```text
backend/harness/loop/graph_coordination/engine.py
backend/task_system/compiler/layered_graph_normalizer.py
frontend/src/components/workspace/views/task-system/taskGraphRuntimeLoopConfig.ts
```

问题：

```text
这不是单纯旧入口。
它承载图任务批次、卷章、长度预算、循环 frame 等业务配置。
需要先确定目标名是 graph_loop_policy 还是 harness_loop_policy。
不能直接替换，否则可能破坏任务图存量配置。
```

## 4. 目标边界

### 4.1 状态模型

目标：

```text
RuntimeLoopState -> HarnessLoopState
RuntimeLoopLimits -> HarnessLoopLimits
RuntimeLoopControlDecision -> HarnessLoopControlDecision
check_runtime_loop_control -> check_harness_loop_control
RuntimeCheckpointStore -> HarnessCheckpointStore
```

目标目录：

```text
backend/harness/loop/state.py
backend/harness/loop/control.py
backend/harness/loop/checkpoint_store.py
```

`backend/runtime/shared/models.py` 不再导出 Harness Loop 状态。

### 4.2 事件协议

目标事件：

```text
harness_run_started
harness_loop_event
harness_loop_control
```

事件 payload 内部可以继续包含：

```text
event_type = loop_terminal / loop_error / task_run_started / ...
```

但外层 stream type 不再叫 `runtime_loop_event`。

不保留双事件：

```text
禁止同时 yield runtime_loop_event 和 harness_loop_event。
```

如果需要读取历史 trace，只在 trace reader 内做只读历史归一化，不作为活跃控制协议。

### 4.3 前端类型

目标类型：

```text
HarnessTraceEvent
HarnessTaskRunSummary
HarnessSessionTaskRuns
HarnessTaskRunTrace
HarnessTaskRunLiveMonitor
HarnessSessionLiveMonitor
```

目标函数：

```text
listOrchestrationHarnessTaskRuns
getOrchestrationHarnessTrace
getOrchestrationHarnessTaskRunLiveMonitor
resolveHarnessTaskRunApproval
startTaskGraphHarnessRun
```

旧 `RuntimeLoop*` 类型和函数删除，不做 re-export。

### 4.4 图任务循环配置

目标建议：

```text
runtime_loop_policy -> graph_loop_policy
runtime_loop_summary -> graph_loop_summary
taskGraphRuntimeLoopConfig.ts -> taskGraphLoopConfig.ts
```

理由：

```text
这些字段描述的是图任务内部的批次/章节/循环 frame，不是单 agent Harness runtime。
它们属于 GraphLoop 的业务控制输入，应明确为 graph_loop。
```

迁移规则：

```text
新写入只写 graph_loop_policy。
读取层只允许一次性迁移旧 metadata.runtime_loop_policy 到 graph_loop_policy。
迁移完成后不保留运行期 fallback。
测试必须保护旧字段不会再被新写入。
```

## 5. 权威链

| 层 | 当前问题 | 目标权威 |
| --- | --- | --- |
| observe | QueryRuntime 仍按 runtime_loop_event 识别终态 | QueryRuntime 只观察 harness_loop_event |
| normalize | frontend/store/events 把各类事件归到 runtime_loop_event | 前端统一归到 harness_loop_event |
| decide | runtime_loop_policy 可能被任务系统和 graph engine 共同解释 | TaskSystem 只声明 graph_loop_policy，GraphLoop 解释 |
| authorize | execution_policy 已迁入 harness/runtime | 保持不变 |
| assemble | RuntimeStartPacket 已在 harness/runtime | 保持不变 |
| execute | AgentLoop/GraphLoop 已在 harness/loop | 保持不变 |
| recover | checkpoint/state 仍在 runtime.shared | 迁入 harness.loop checkpoint/state |
| record | trace/monitor 已是 HarnessTraceReader，但前端类型仍旧 | 改为 Harness trace/read model |

## 7. 不在本轮处理

以下内容不纳入本计划，避免范围失控：

```text
健康系统自身重构。
Memory 系统目录重构。
工具系统新调用体系。

```

其中 `default_projection_id` 属于 Agent/Soul 身份配置历史问题，不是 Harness Loop 主链残留；后续应独立审查。

## 8. 风险

主要风险：

```text
1. 事件协议切换会同时影响后端 SSE、前端 reducer、测试和健康提取。
2. Graph loop 配置改名会影响任务图存量 JSON。
3. Checkpoint 状态模型迁移会影响历史 checkpoint 读取。
```

风险处理：

```text
1. 不做双事件兼容，只在历史读取层做只读归一化。
2. Storage JSON 和 fixture 一次性迁移。
3. CheckpointStore 读取旧 state authority 时只做数据加载转换，不作为新写入格式。
4. 每阶段必须跑 focused tests，不允许跳过。
```

## 10. 结论

上一轮迁移完成了“旧 runtime/loop 架构入口”的切除，但还没有完成“状态协议和概念命名权威”的最终收口。

下一轮应按本计划推进：先把 LoopState 和 checkpoint/control 权威迁入 Harness，再切事件协议，然后改前端类型，最后处理 GraphLoop 的业务配置字段。

这不是一次简单改名，而是把动态控制权从 `runtime.shared` 完整收回到 `harness.loop`。

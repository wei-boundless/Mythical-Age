本次重点不是看灵魂页面是否“能管理人物卡”，而是看灵魂系统是否已经真正成为：

`平台中的 prompts 身份层系统`

也就是检查：

- 灵魂系统是否只负责身份与姿态
- 是否不再混入 workflow、能力授权、记忆裁决
- 是否已经形成正式 public boundary
- 编排系统是否只消费身份层装配结果

## 二、当前源码已经对齐的原则

### 2.1 灵魂系统已经从散装 prompt 实现提升为正式系统入口

当前灵魂系统已经建立统一正式入口：

- [backend/soul/facade.py](/d:/AI应用/langchain-agent/backend/soul/facade.py:1)

并在内部形成三段式服务边界：

- 注册层：[backend/soul/registry_service.py](/d:/AI应用/langchain-agent/backend/soul/registry_service.py:1)
- 投影层：[backend/soul/projection_service.py](/d:/AI应用/langchain-agent/backend/soul/projection_service.py:1)
- 装配层：[backend/soul/assembly_service.py](/d:/AI应用/langchain-agent/backend/soul/assembly_service.py:1)

这意味着外层不再需要直接理解：

- `SoulRegistry`
- `projection_store`
- `ProjectionTemplateRegistry`
- `ProjectionInstanceRegistry`
- `build_soul_runtime_view`

而是可以从 `SoulFacade` 进入。

这与总设计书和灵魂系统设计书要求的：

`系统边界通过正式入口表达`

是一致的。

### 2.2 灵魂系统仍然只负责 prompts 身份层，没有重新长回任务方法层

从当前实现看，灵魂系统输出的核心仍然是：

- `SoulProfile`
- projection card / template / instance preview
- `SoulRuntimeView`
- `PromptManifest`
- `AgentPromptBundle`

对应实现主要集中在：

- [backend/soul/registry.py](/d:/AI应用/langchain-agent/backend/soul/registry.py:1)
- [backend/soul/projection_builder.py](/d:/AI应用/langchain-agent/backend/soul/projection_builder.py:1)
- [backend/soul/runtime_assembly.py](/d:/AI应用/langchain-agent/backend/soul/runtime_assembly.py:1)
- [backend/soul/view_mapping.py](/d:/AI应用/langchain-agent/backend/soul/view_mapping.py:1)
- [backend/soul/prompt_assembly.py](/d:/AI应用/langchain-agent/backend/soul/prompt_assembly.py:1)

这些对象表达的是：

- 灵魂是谁
- 这次以什么姿态出现
- 身份层 prompt section 如何进入装配链

它们并没有上升成：

- 任务定义 owner
- workflow owner
- resource policy owner
- memory truth owner

这保持住了灵魂系统设计书里最关键的一条硬边界：

`灵魂系统是 prompts 身份层，不是任务方法层`

### 2.3 projection 仍然只是姿态投影，不授予执行权和资源权

当前 projection 相关实现虽然能看到 skill/tool 摘要，但这些字段只进入可见视图和 prompt section，并没有变成授权来源。

例如：

- `SoulToolView.authorization_owner == "ResourcePolicy"`
- `runtime_executable` 只作为资源侧结果被映射
- `build_soul_runtime_view(...)` 不直接生成 resource authority

对应实现：

- [backend/soul/runtime_assembly.py](/d:/AI应用/langchain-agent/backend/soul/runtime_assembly.py:1)
- [backend/soul/view_mapping.py](/d:/AI应用/langchain-agent/backend/soul/view_mapping.py:1)
- [backend/tests/soul_projection_resource_boundary_regression.py](/d:/AI应用/langchain-agent/backend/tests/soul_projection_resource_boundary_regression.py:1)

这符合灵魂系统设计书中的要求：

`projection 只能改变姿态，不能授予能力、权限、记忆写回权`

### 2.4 编排系统已经改为消费灵魂系统正式装配结果

当前编排层已经不再直接拼 soul 内部零件，而是通过 facade 获取运行时身份层结果：

- [backend/orchestration/assembly_builder.py](/d:/AI应用/langchain-agent/backend/orchestration/assembly_builder.py:1)

编排侧现在消费的是：

- `runtime_view`
- `prompt_manifest`
- `projection_id`

并把这些结果挂进：

- `TaskBodyOrchestration`
- `AgentRuntimeSpec`

这与灵魂系统设计书中：

`编排系统只消费身份层素材包`

这一原则是对齐的。

### 2.5 API 对灵魂系统底层的直接穿透已经明显收缩

当前灵魂 API 已经主要通过 `SoulFacade` 进入：

- [backend/api/souls.py](/d:/AI应用/langchain-agent/backend/api/souls.py:1)

也就是说，外部接口不再把自己当作：

`registry + projection store + prompt assembly 的半个内部模块`

这与前几个阶段已经建立起来的统一收口方向一致：

`外层走正式系统入口，内部零件留在系统内治理`

## 三、与先进架构原则对齐的地方

### 3.1 单一 public boundary 已经立住

对照 `docs/设计原则/25-架构模式总结.md` 提炼出来的可迁移模式，灵魂系统现在已经具备清晰的单一入口：

- `SoulFacade`

这和成熟架构里常见的：

`单一注册点 + 单一供给点 + 外层不碰底层零件`

是同一个方向。

当前虽然内部还保留若干底层实现模块，但外部调用面已经开始收束到一个正式边界上，这比“谁需要就 import 哪个内部函数”的方式稳定得多。

### 3.2 派生产物没有反客为主成为 authority owner

当前：

- `runtime_view` 是运行时身份视图
- `prompt_manifest` 是装配清单
- `agent_prompt_bundle` 是 prompt 交付包

它们都是：

`derived artifact`

并没有反过来成为：

- 资源授权真相源
- 记忆事实真相源
- 任务编排真相源

这与先进架构里非常重要的一条原则一致：

`派生结果可以被消费，但不应上升为权威源`

### 3.3 分层方向已经从“文件仓库思维”转向“对象供给链思维”

以前灵魂系统更像：

`prompt 文件目录 + 若干散装拼接函数`

现在已经开始形成供给链：

- registry 维护人格底座
- projection 维护姿态素材和实例化
- assembly 负责向编排系统交付身份层运行视图

这与设计书里提出的：

`soul registry -> projection registry -> prompt identity asset -> prompt assembly manifest`

已经基本同向，说明结构方向是对的。

## 四、当前仍然存在的差距

### 4.1 灵魂系统内部虽然已拆层，但 builder 仍可继续对象化

虽然 facade 和 service 层已经建立，而且 `projection.py` 这一兼容壳已经被删除，但当前新拆出的内部 builder 仍然主要按“实现分段”组织：

- [backend/soul/projection_builder.py](/d:/AI应用/langchain-agent/backend/soul/projection_builder.py:1)
- [backend/soul/runtime_assembly.py](/d:/AI应用/langchain-agent/backend/soul/runtime_assembly.py:1)
- [backend/soul/view_mapping.py](/d:/AI应用/langchain-agent/backend/soul/view_mapping.py:1)

这说明第六阶段已经完成了：

`边界收口`

但还没有完全完成：

`内部对象链彻底纯化`

后续如果继续深收口，优先方向应是把这里继续拆成更明确的：

- identity asset builder
- runtime view builder
- manifest assembler

而不是继续往单一 builder 文件里叠逻辑。

### 4.2 `SoulAssemblyService` 目前更像正式装配出口，而不是完整装配模型层

当前：

- [backend/soul/assembly_service.py](/d:/AI应用/langchain-agent/backend/soul/assembly_service.py:1)

已经作为正式出口成立，但本体仍然主要转发到：

- `SoulProjectionBuilder`
- `build_soul_runtime_view(...)`

这意味着装配服务的边界已经立住，但内部模型还没有完全沉到 service 自己名下。

这不是方向错误，但说明“装配层正式建模”还可以继续走一步。

### 4.3 `api/souls.py` 仍保留少量系统外壳辅助逻辑

例如：

- 立绘上传校验
- frontend portrait path 计算
- path resolve 的 HTTPException 包装

对应文件：

- [backend/api/souls.py](/d:/AI应用/langchain-agent/backend/api/souls.py:1)

这些逻辑问题不大，因为它们更多是 API 壳层职责。

但如果以后继续追求系统纯度，可以考虑把“灵魂资产文件管理”单独沉成灵魂系统内部资产服务，而不是继续堆在 API 路由里。

### 4.4 当前 runtime prompt 装配仍存在并行链，而非完全以 soul assembly 为唯一身份装配中心

从编排实现看，`orchestration` 里仍然保留了一部分 prompt contract 自组装逻辑，例如：

- task section
- workflow section
- projection section
- output section

对应：

- [backend/orchestration/assembly_builder.py](/d:/AI应用/langchain-agent/backend/orchestration/assembly_builder.py:1)

这意味着当前已经完成的是：

`灵魂系统正式接住身份层`

但还没有完全达到：

`所有 prompt 身份装配语义都只从 soul assembly 统一表达`

这也是正常的阶段状态，因为 workflow / output boundary 本就不该属于灵魂系统。

更准确的结论应该是：

- 身份层已经收口到 soul
- 任务方法层仍留在 orchestration / task

这条边界现在是对的，不需要为了“统一”而错误合并。

## 五、对“修改后框架”和“原则框架”的差别结论

### 5.1 当前差别已经从“边界错误”缩小为“内部完备度不足”

现在灵魂系统已经不再是明显的结构错位状态。

当前剩余差别主要不是：

- owner 搞错
- workflow 混回灵魂系统
- projection 获得权限
- API 到处直连内部零件

而更像是：

- service 已立住，但内部 builder 还可以继续拆
- runtime identity asset 的对象命名还可以继续正式化
- 文件资产管理还未完全沉到系统内层

这说明最关键的结构风险已经被修正掉了。

### 5.2 当前框架已经基本符合“灵魂系统是 prompts 身份层系统”的原则框架

对照设计书，现在可以把灵魂系统的实际定位表述为：

`灵魂系统已经从 prompt 素材仓库，提升为具备正式边界、投影层与身份装配出口的 prompts 身份层系统`

这个结论是成立的。

## 六、建议的后续动作

如果我们继续推进灵魂系统的下一步纯化，建议顺序如下：

1. 维持 `SoulFacade` 作为唯一正式 public boundary，不再新增外层对 `registry.py`、`projection_builder.py`、`runtime_assembly.py`、`projection_store.py` 的直接依赖。
2. 后续如需继续纯化，优先把 [backend/soul/projection_builder.py](/d:/AI应用/langchain-agent/backend/soul/projection_builder.py:1) 与 [backend/soul/runtime_assembly.py](/d:/AI应用/langchain-agent/backend/soul/runtime_assembly.py:1) 进一步对象化，把 identity asset、runtime view、manifest 的边界拉得更清楚。
3. 如果后续要继续做灵魂资产治理，再单独建立 portrait / asset 管理服务，不要把这类逻辑继续堆进 API 路由。
4. 保持当前边界判断：
   task/workflow 归任务系统与编排系统，identity/projection 归灵魂系统，不为了“统一 prompt”而把方法层重新混回灵魂系统。

## 七、审查结论

本轮收口后，灵魂系统已经基本符合设计原则文档中的核心要求：

- 正式系统入口已建立
- 注册层、投影层、装配层已经初步分层
- 灵魂系统仍然只负责 prompts 身份层
- projection 没有越界成权限和能力 owner
- 编排系统已经开始只消费灵魂系统交付的身份装配结果

当前剩余差距主要是内部继续纯化的问题，不再是结构方向问题。

这意味着第六阶段可以视为：

`灵魂系统正式收口完成，后续进入内部对象链纯化与资产治理细化阶段`

## 二、总判断

当前后端并不是“六大系统都不存在”，而是已经长出了很多接近目标架构的骨架。

但这些骨架还没有完全锁定正式 owner，仍存在以下问题：

1. 任务系统、灵魂系统、编排系统之间还存在跨层拼装
2. 编排系统已经有 profile 和资源边界骨架，但还没有完整的单 Agent 身体对象分层
3. 记忆系统的治理哲学已经接近目标，但正式入口仍未唯一化
4. 操作系统的资源目录化已经存在，但还没有成为别的系统唯一的供给入口
5. 健康系统已经开始消费运行体和任务信息，但运行体正式契约还没有闭合

因此，本轮重构不应从“补字段”开始，而应从“锁定正式边界”和“回收跨层 owner”开始。

## 三、总映射规则

后续对现有源码的处理统一分四类：

- `保留`：可直接吸收为新系统主干
- `重命名`：方向基本正确，但当前目录或命名仍带历史语义
- `拆分`：当前模块承担了跨系统职责，必须拆回各自边界
- `降级/清理`：保留为内部实现细节或兼容层，不再作为正式系统边界

## 四、六大系统映射

## 4.1 任务系统

### 4.1.1 应保留

- `backend/tasks/flow_models.py`
- `backend/tasks/workflow_models.py`
- `backend/tasks/flow_registry.py`
- `backend/tasks/template_registry.py`
- `backend/tasks/template_models.py`
- `backend/tasks/spec_models.py`

这些模块已经提供了较强的任务对象骨架，例如：

- `TaskFlowDefinition`
- `GeneralTaskProfile`
- `TaskAssignment`
- `TaskAgentBinding`
- `CoordinationTaskDefinition`
- `TaskWorkflowBinding`
- `TaskSpec`

说明任务系统已经不再只是提示词碎片堆放区，而开始具备正式对象层。

### 4.1.2 应重命名或继续收束

- `TaskAssignment` 后续应继续向 `SpecificTaskRecord` 方向收束
- `TaskSpec` 后续应继续向 `TaskExecutionAssembly` 方向收束
- `TaskAgentBinding` 后续应拆出更明确的 `TaskAgentAdoptionPlan`

### 4.1.3 必须拆分

- `backend/tasks/contract_builder.py`

该模块当前承担了明显跨系统职责。它不只构建任务装配，还直接碰了：

- projection requirement
- soul runtime view
- prompt manifest
- task prompt contract

这说明它已经跨过任务系统边界，进入了灵魂系统和运行体装配前的职责范围。

后续必须拆成：

1. 任务系统只输出任务侧正式装配请求
2. 灵魂系统只输出身份层素材
3. 编排系统负责最终 prompt / resource / memory 装配

### 4.1.4 当前结论

`backend/tasks` 适合作为新任务系统主干保留，但必须拆除 `contract_builder` 这种跨系统总装配器。

## 4.2 编排系统

### 4.2.1 应保留

- `backend/orchestration/agent_runtime_models.py`
- `backend/orchestration/agent_runtime_registry.py`
- `backend/operations/registry.py`
- `backend/operations/policy_builder.py`
- `backend/operations/runtime_view.py`
- `backend/orchestration/runtime_loop/context_manager.py`

这些模块已经提供了编排系统最关键的几类骨架：

- Agent 默认运行 profile
- operation registry
- resource policy candidate
- resource runtime view
- runtime context snapshot
- prompt source report

### 4.2.2 应重命名

以下目录虽然方向正确，但名称仍停留在旧语义层：

- `backend/orchestration`
- `backend/operations`
- `backend/permissions`

后续应统一被解释为“编排系统”的不同子层，而不是继续被看成彼此平级的独立系统。

### 4.2.3 必须补出的正式对象

当前缺少以下关键对象：

- `AgentBodyProfile`
- `TaskBodyOrchestration`
- `PromptStructureProfile`
- `MemoryScopeProfile`
- `AgentRuntimeSpec`

现有 `AgentRuntimeProfile` 很有价值，但它还不足以替代完整的单 Agent 身体对象体系。

### 4.2.4 应降级为内部执行层

- `backend/orchestration/runtime_loop/*`

这条线后续仍然要保留，但它的定位应被收回为：

- runtime 执行层
- trace / checkpoint / event 证据层

它不应继续承担隐式装配职责。

### 4.2.5 当前结论

编排系统不是从零做，而是要以 `orchestration + operations` 为主干，补齐正式对象层，并把 runtime loop 收回到“只执行不决策”的位置。

## 4.3 记忆系统

### 4.3.1 应保留为正式入口候选

- `backend/memory_system/contracts.py`
- `backend/memory_system/runtime_view.py`
- `backend/memory_system/conversation_memory.py`
- `backend/memory_system/state_memory.py`
- `backend/memory_system/long_term_memory.py`
- `backend/memory_system/writeback.py`
- `backend/memory_system/gate.py`
- `backend/memory_system/governance.py`

这条线已经体现出正确的结构性方向：

- candidate-only memory
- read-only runtime view
- restore 不等于 decide
- writeback 需要 gate
- durable memory 需要 governance

### 4.3.2 应降级为兼容层

- `backend/memory/*`

这条线有较多聚合与适配价值，例如 `MemoryFacade`，但不适合继续作为平台正式边界暴露。

后续更合理的做法是：

- `memory_system` 成为正式系统入口
- `memory` 退为兼容适配层或内部服务层

### 4.3.3 应分批吸收

- `backend/structured_memory/*`

这一层沉淀了 durable/session 的很多底层实现能力，但目录语义仍是旧架构表达。后续应被 `memory_system` 分批吸收，而不是继续平级并行生长。

### 4.3.4 应从记忆系统剥离

- `backend/context_management/*`

这条线更适合被重新解释为：

- 上下文打包层
- prompt/context 呈现层
- token budget 与 compaction 层

它不应继续被误认为记忆系统本体。

### 4.3.5 当前结论

记忆系统重构的关键不是重做能力，而是统一正式入口、统一对象语言、统一写回治理，并把旧层降为内部实现。

## 4.4 操作系统

### 4.4.1 应保留

- `backend/operations/registry.py`
- `backend/capabilities/models.py`
- `backend/capabilities/manifest.py`
- `backend/capabilities/validation.py`
- `backend/capabilities/endpoints.py`

这条线已经具备了：

- operation descriptor
- operation registry
- capability binding graph
- capability manifest
- capability validation

### 4.4.2 应继续统一出口

当前仍然分散存在于：

- `backend/capabilities`
- `backend/tools`
- `backend/skills`
- `backend/skill_system`
- `backend/workers`
- `backend/operations`

这些资源本身不一定错，但后续必须统一为正式供给链路：

`registry -> manifest -> validation -> supply package -> task/body consumption`

### 4.4.3 应降级为子资源层

以下目录后续应继续存在，但定位是操作系统内部资源层，而不是平台级正式系统边界：

- `backend/tools`
- `backend/skills`
- `backend/skill_system`
- `backend/workers`

### 4.4.4 当前结论

操作系统已经有了很好的底座。重构重点不是重写资源，而是统一资源供给出口，阻止任务系统和编排系统继续直接扫描底层目录。

## 4.5 灵魂系统

### 4.5.1 应保留

- `backend/soul/registry.py`
- `backend/soul/contracts.py`
- `backend/soul/projection_store.py`
- `backend/soul/prompt_assembly.py`
- `backend/soul/projection.py`

这条线已经具备：

- soul registry
- soul file payload
- projection store
- prompt manifest

### 4.5.2 应继续保留为素材层

- `backend/soul/agent_core/*`
- `backend/soul/agent_core/seeds/*`

这些文件适合作为 soul 本体与 builtin 素材来源。

### 4.5.3 必须去 runtime 化

- `backend/soul/projection_store.py`

当前 projection card 中仍带有以下运行态信息：

- `runtime_preview`
- `tool_views`
- `memory_policy_summary`
- `output_contract_summary`

这些内容说明 projection store 还承担了运行态预览职责。

后续应收回为：

- projection 只保留身份层素材
- runtime preview 改由编排系统根据正式输入对象生成

### 4.5.4 当前结论

灵魂系统基础很好，但后续一定要继续去 workflow 化、去 runtime 化，回到 prompts 身份层素材系统的定位。

## 4.6 健康系统

### 4.6.1 应保留

- `backend/health_system/models.py`
- `backend/health_system/registry.py`
- `backend/health_system/workbench.py`
- `backend/health_system/runtime_admission.py`
- `backend/health_system/test_catalog.py`
- `backend/harness/regression_gate.py`

这说明健康系统已经不是空壳，它已经具备：

- issue / run / report / receipt / command 对象
- workbench projection
- regression gate
- runtime admission

### 4.6.2 应继续吸收

- `backend/harness/*`

这条线适合继续沉到健康系统的回归门禁子层，而不是长期独立漂浮。

### 4.6.3 当前结构冲突

`backend/health_system/runtime_admission.py` 当前访问了 `profile.allowed_workflow_ids`。

但现有 `backend/orchestration/agent_runtime_models.py` 中的 `AgentRuntimeProfile` 并没有这个字段。

这说明：

- 健康系统已经开始试图检查运行体正式契约
- 但编排系统尚未真正对外发布完整契约

这不是单点 bug，而是结构未闭合的直接证据。

### 4.6.4 当前结论

健康系统已经足够作为正式系统继续演化，但它的后续推进依赖编排系统先把正式接口补齐。

## 五、需要降级或清理的旧层

以下内容不代表立即删除，而代表不应继续作为正式平台边界扩张：

### 5.1 应降级为兼容或内部实现层

- `backend/memory/*`
- `backend/structured_memory/*`
- `backend/context_management/*`
- `backend/permissions/*`

### 5.2 应拆除总装配器角色

- `backend/tasks/contract_builder.py`

### 5.3 不应继续扩大历史混合语义

- 把 `orchestration` 继续当作“大总管目录”来塞更多装配逻辑
- 把 `projection` 继续掺入 resource / memory / output preview
- 把 `context_management` 继续同时表达记忆、上下文、恢复、呈现

## 六、阶段化执行顺序

为了避免继续在混合边界上打补丁，建议按以下顺序执行。

## 6.1 第一阶段：回收任务系统边界

目标：

- 让任务系统只表达任务对象、任务方法、任务采用、任务装配请求

动作：

1. 拆解 `backend/tasks/contract_builder.py`
2. 明确任务系统正式输出对象
3. 禁止任务系统继续直接拼 soul runtime 和 prompt manifest

阶段完成标准：

- 任务系统可独立输出正式任务装配请求
- 任务系统不再直接装配灵魂素材和运行体 prompt 结构

## 6.2 第二阶段：建立编排系统正式对象层

目标：

- 从 `orchestration + operations` 中抽出单 Agent 身体正式对象

动作：

1. 补出 `AgentBodyProfile`
2. 补出 `TaskBodyOrchestration`
3. 补出 `PromptStructureProfile / MemoryScopeProfile / AgentRuntimeSpec`
4. 让 runtime loop 改为只消费正式运行体对象

阶段完成标准：

- 编排系统拥有独立正式对象层
- runtime 不再承担隐式装配职责

## 6.3 第三阶段：清理灵魂系统跨层职责

目标：

- 让灵魂系统只提供身份层素材，不再保存运行态预览逻辑

动作：

1. 精简 projection store 持久字段
2. 把 runtime preview 交回编排系统生成
3. 锁定 `SoulProfile / ProjectionProfile / PromptIdentityAsset / PromptAssemblyManifest`

阶段完成标准：

- projection 不再混入 runtime 资源和记忆摘要
- 灵魂系统与 workflow、resource、memory 完整解耦

## 6.4 第四阶段：统一记忆系统正式入口

目标：

- 让 `memory_system` 成为唯一正式记忆系统入口

动作：

1. 把 `memory` 收回为兼容适配层
2. 把 `structured_memory` 能力分批吸收进 `memory_system`
3. 把 `context_management` 从记忆本体中剥离
4. 固化 `MemoryRequest / MemoryScopePolicy / MemoryBundle / MemoryWritebackRecord`

阶段完成标准：

- 记忆系统对外只暴露统一正式入口
- 旧层不再作为正式边界继续扩张

## 6.5 第五阶段：统一操作系统供给出口

目标：

- 让任务系统和编排系统只消费正式能力供给包

动作：

1. 统一 Tool / Skill / Worker / Operation / CapabilityManifest
2. 补出 `CapabilitySupplyPackage`
3. 禁止任务系统和编排系统直接扫描底层资源目录

阶段完成标准：

- 操作系统成为唯一能力供给出口

## 6.6 第六阶段：让健康系统只检查正式契约

目标：

- 让健康系统从“猜字段”切换为“检查正式对象”

动作：

1. 修复编排系统与健康系统的契约缺口
2. 让 runtime admission 只依赖正式运行体对象
3. 把 harness / regression gate 正式并入健康系统门禁层

阶段完成标准：

- 健康系统可以稳定检查任务链路、运行体链路、记忆链路和回归门禁

## 七、执行中必须避免的反模式

后续重构严禁再次出现以下模式：

1. 为了兼容继续保留错误边界
2. 在任务系统里继续拼装灵魂和运行体对象
3. 在 runtime 中继续临时猜 task / memory / prompt 结构
4. 让多个 memory 入口长期并行扩张
5. 让 projection 继续承担 workflow、resource、memory 预览职责
6. 让健康系统依赖未正式发布的隐式字段

## 八、执行结论

当前代码库已经有足够多的新架构骨架，可以支撑正式系统化重构。

因此本轮工作重点不是“从零新建六个系统”，而是：

- 识别可吸收骨架
- 回收跨层 owner
- 锁定正式对象
- 降级旧兼容层
- 按阶段完成切换

后续所有实现，应以本文件作为“现状到目标”的迁移蓝图，而不是继续在历史目录语义上做局部修补。

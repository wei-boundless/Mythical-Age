# 目录与文档体系重构蓝图

日期：2026-05-04

## 一、文档定位

本文用于定义本项目在“正式代码重构”之前，目录体系与文档体系应先完成的治理动作。

它主要回答：

1. 当前 `docs/` 和 `backend/` 的目录问题到底是什么
2. 哪些目录是正式边界，哪些目录是历史残留
3. 文档和目录应该按什么层级收口
4. 后续搬目录时的顺序应该是什么

## 二、当前真实问题

当前问题不是“文件太多”，而是：

`正式边界、历史残留、数据目录、实现目录混在一起了`

这主要体现为四类问题：

1. 文档分区已经出现，但缺少统一总导航和阅读入口
2. 后端目录里存在正式系统目录与历史目录并行
3. 代码包目录与持久化数据目录命名过于接近
4. 某些 compat 文件看起来仍像正式系统能力

## 三、当前目录中的关键混杂点

## 3.1 正式系统与旧目录并行

当前典型问题：

- `backend/health_system`
- `backend/health-system`

以及：

- `backend/memory_system`
- `backend/memory`
- `backend/structured_memory`
- `backend/session-memory`

这会导致后续阅读和重构时，无法一眼确认：

`哪个是正式系统入口，哪个只是历史实现层`

## 3.2 代码包目录与数据目录混名

当前典型问题：

- 代码包：`backend/orchestration/runtime_loop`
- 数据目录：`backend/runtime-loop`

这虽然技术上能运行，但会在认知上制造错觉：

`runtime-loop` 到底是代码系统还是持久化存储`

## 3.3 文档体系缺少统一入口

当前 `docs/设计原则` 和 `docs/系统规划` 已经自然形成了两区，但此前缺少：

- `docs/README.md`
- `docs/系统规划/README.md`

这会导致后来加入的人不容易判断：

- 先看哪类文档
- 哪些是外部参考
- 哪些是本项目正式依据

## 四、正式目录分层原则

后续所有目录治理统一按下面四条原则推进。

## 4.1 一个系统只保留一个正式 public boundary

例如：

- 任务系统：`backend/tasks`
- 编排系统：`backend/orchestration`
- 记忆系统：`backend/memory_system`
- 灵魂系统：`backend/soul`
- 操作系统：`backend/operations`
- 健康系统：`backend/health_system`

其余相关目录如果仍存在，应降级为：

- 内部实现层
- 兼容层
- 数据层

## 4.2 数据目录和代码目录必须在命名上可区分

建议原则：

- Python 包目录用下划线风格
- 持久化数据目录避免与系统名 1:1 同名

例如后续更适合：

- 代码包：`runtime_loop`
- 数据目录：`runtime_state_store`

## 4.3 compat 目录必须显式化

兼容层文件不应长期躲在正式系统根目录下。

后续应优先进入：

- `compat/`
- `adapters/legacy/`

以避免误导后续开发者把 compat 当正式边界继续扩展。

## 4.4 文档必须区分“参考资料”和“正式决策”

统一规则：

- `设计原则` 放外部参考与方法论
- `系统规划` 放本项目正式结论和实施蓝图

## 五、目标结构

建议最终目标结构如下：

```text
docs/
  README.md
  设计原则/
  系统规划/
    README.md

backend/
  README.md
  api/
  query/
  tasks/
  orchestration/
    runtime_loop/
  memory_system/
  soul/
  operations/
  health_system/
  execution/
  output_boundary/
  storage/
  compat/
  data/ 或 runtime_state_store/
```

## 六、目录治理执行顺序

目录治理建议分四步。

## 6.1 第一步：补导航与治理文档

本轮已完成：

- `docs/README.md`
- `docs/系统规划/README.md`
- `backend/README.md`
- 本文

目标：

- 先把规则和入口立起来

## 6.2 第二步：冻结正式边界

接下来要做：

- 明确哪些目录是正式系统入口
- 明确哪些目录暂时保留但不再扩展

这一步先不搬文件，先冻结扩展方向。

## 6.3 第三步：清 public export 与 compat 外露面

优先清理：

- `tasks.__init__`
- runtime 旧入口
- compat surface 的公开暴露

因为目录搬迁前，先清 public boundary 的收益最高。

## 6.4 第四步：再做物理搬迁

只有在上面三步完成后，才建议做：

- 目录 rename
- 代码 import 调整
- 数据目录改名

否则会边搬边变，成本很高。

## 七、当前正式结论

这轮目录与文档体系重构的目标不是“一次性把 backend 搬干净”，而是：

1. 先建立统一导航
2. 先冻结正式边界
3. 先让后续重构有稳定落点

因此当前最重要的判断是：

`先治理，再搬迁；先清 owner，再清目录`

只有这样，后面的第二阶段源码重构才不会又回到“边改边混”的状态。

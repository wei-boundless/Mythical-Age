# 任务系统固定任务合同重构计划书

日期：2026-05-27

状态：重校准计划书

依据文档：

- `docs/系统框架/001-Agent系统框架总纲-20260527.md`
- `docs/系统框架/002-任务系统框架设计书-20260527.md`

## 1. 总结论

任务系统本轮不再被设计为所有 agent 新生命周期的总入口。

本轮目标收束为：

```text
TaskEnvironment
-> SpecificTask
-> TaskContract
-> Harness Runtime
-> Harness Loop
```

任务系统负责：

```text
配置任务环境。
维护任务库里的具体任务。
根据已确定的 SpecificTask 签发 TaskContract。
记录任务环境、具体任务、任务合同和任务库事实。
```

任务系统不负责：

```text
承接所有 agent 自发任务。
替 agent 判断是否要开新 loop。
执行模型、工具或 agent loop。
推进 RuntimeLoopState。
管理执行生命周期。
保存执行生命周期引用。
执行 RuntimeAssemblyPlan / LoopPlan。
让订单工厂发放任务合同。
```

agent 自发的新执行生命周期暂时留在 Harness：

```text
AgentActionRequest
-> Harness RuntimeAssemblyRequest
-> RuntimeStartPacket
-> LoopLifecycle
```

以下链路本轮关闭：

```text
AgentWorkLifecycleIntent -> TaskSystem
agent 自动调用已注册 SpecificTask
agent 把当前工作沉淀为可复用 SpecificTask
```

未来等固定任务系统稳定后，再单独设计 agent 与任务系统的受控对接。

## 2. 当前代码问题

当前任务系统仍混杂多套职责：

```text
TaskEnvironment 配置
SpecificTask 任务库
TaskOrder 订单来源
TaskOrderFactory 合同签发
TaskExecutionEnvelope 旧运行输入
TaskLifecycle / TaskActivationRequest 新原语
图编译和图 runtime spec
projection / domain_id / task_selection 兼容控制面
```

主要问题：

```text
TaskOrderFactory 名称和职责不对。
具体任务本身已有工作申请字段，但合同签发仍绕到订单工厂。
TaskActivationRequest 被扩大成所有任务来源的统一入口，容易把 agent 自发任务也拉回任务系统。
TaskLifecycle 容易被误用为 Harness 执行生命周期，而固定任务真正需要的是 TaskContract。
Graph node 被设计成 child TaskLifecycle，会让图任务节点重新穿过任务系统，增加控制面混乱。
```

## 3. 目标权威模型

### 3.1 TaskEnvironment

大环境权威，定义预定义任务可用的资源、工具、权限、memory、artifact、agent、prompt、skill、graph、人审、恢复和审计边界。

### 3.2 SpecificTask

任务库中的具体任务定义，也是固定任务的工作申请模板。

它应直接提供合同签发需要的字段：

```text
environment_id
objective schema
input schema
working object schema
output contract
acceptance policy
agent assignment
tool capability requirements
resource requirements
memory policy
artifact policy
risk / approval policy
runtime_shape
loop_plan
graph_ref / graph_policy
human_gate_policy
```

### 3.3 TaskContract

任务系统签发给 Harness 的固定任务执行合同。

`TaskContract` 取代“TaskOrderFactory 发放任务合同”的旧结构。

建议字段：

```text
contract_id
session_id
task_id
environment_id
source_ref
objective
working_objects
input_refs
runtime_shape
runtime_requirements
loop_requirements
runtime_assembly_plan
loop_plan
resource_scope
tool_scope
memory_scope
artifact_scope
agent_assignment
prompt_pack_refs
skill_pack_refs
output_contract
acceptance_policy
recovery_policy
approval_policy
risk_policy
graph_contract
human_gate_contract
status
```

禁止进入 `TaskContract` 的字段：

```text
latest_runtime_start_packet_ref
latest_loop_state_ref
result_ref
loop_lifecycle_ref
task_run_ref
runtime_packet_ref
```

### 3.4 TaskOrder

只作为来源记录或 API/UI 启动载体。

```text
TaskOrder != TaskContract
TaskOrderFactory != TaskContractIssuer
TaskExecutionEnvelope != TaskContract
ExecutionChannel != 任务生命周期
```

### 3.5 Harness Runtime / Loop

Harness Runtime 消费 `TaskContract` 生成 `RuntimeStartPacket`。

Harness Loop 消费 `RuntimeStartPacket` 推进 `LoopLifecycle / LoopState`。

`RuntimeAssemblyPlan / LoopPlan` 可以由 `TaskContract` 携带。它们是发给 Harness Runtime / Harness Loop 的请求规格，不是任务系统执行 runtime 或管理 loop 生命周期的凭据。

任务系统不保存执行生命周期引用，不推进动态状态。执行事实需要展示时，由观测/read model 从 Harness 查询或投影，不写回任务合同本体。

## 4. 代码结构目标

目标目录：

```text
backend/task_system/
  environments/
    TaskEnvironment 定义、registry、resolver

  tasks/
    SpecificTask 定义、任务库记录、任务启动模板

  contracts/
    TaskContract
    GraphTaskContract
    NodeExecutionContract
    OutputContract
    AcceptancePolicy

  assembly/
    SpecificTask + TaskEnvironment -> TaskContract

  graphs/
    TaskGraphDefinition
    GraphRuntimeAssemblyPlan
    GraphLoopPlan
    graph contract helpers

  orders/
    只保留来源记录和 API 申请载体
    不保留 runtime / loop 控制权

  repositories/
    纯数据读写
```

需要降级或删除：

```text
TaskOrderFactory 合同签发职责
TaskActivationRequest 作为所有来源统一入口的设计
AgentWorkLifecycleIntent 接入任务系统的代码和测试
GraphNodeDispatch -> child TaskLifecycle 的设计
TaskExecutionEnvelope 作为主链运行合同
ExecutionChannel 作为任务生命周期
task_selection / projection / domain_id 控制事实
```

## 5. 实施阶段

### Phase 0：冻结边界

目标：

```text
明确任务系统只签发预定义任务合同。
agent 自发新执行生命周期暂不接入任务系统。
```

修改：

```text
文档、测试命名、代码注释中删除“agent 发起任务必须进入 TaskSystem”的表述。
```

完成标准：

```text
文档只保留 agent 调用任务库 / 沉淀模板作为未来能力。
本轮计划不再包含 AgentWorkLifecycleIntent -> TaskSystem。
```

### Phase 1：建立 TaskContract 模型

新增：

```text
backend/task_system/contracts/task_contract.py
```

或合并到现有 `contracts/contract_models.py`，避免过度散文件。

目标：

```text
TaskContract 可序列化。
GraphTaskContract / NodeExecutionContract 有清晰父子关系。
TaskContract 不保存 runtime / loop ref，不持有动态 LoopState。
```

测试：

```text
backend/tests/task_contract_issuer_regression.py
```

### Phase 2：从 SpecificTask 签发 TaskContract

新增或改造：

```text
backend/task_system/assembly/contract_issuer.py
backend/task_system/tasks/assembly_policy_resolver.py

```

目标链：



### Phase 3：降级 TaskOrderFactory


### Phase 4：Harness 消费 TaskContract


```text

```

完成标准：

```text
RuntimeStartPacket 有 task_contract_ref / environment_ref。
Harness Runtime 不再从 task_selection / projection / domain_id 反解合同。
TaskContract 不回写 runtime packet ref / loop state ref。
```

### Phase 5：图任务合同化

修改：

```text

backend/harness/loop/graph_loop.py

```

```text
SpecificTask(runtime_shape = task_graph)
-> GraphTaskContract
```

完成标准：


### Phase 6：删除旧权威和旧测试

删除或降级：

```text
TaskActivationRequest 作为所有入口的统一主链
TaskLifecycle 作为固定任务合同主语
AgentWorkLifecycleIntent -> TaskSystem
TaskOrderRun / ExecutionChannel / TaskExecutionEnvelope 主链权威
projection / domain_id / task_selection 控制链
保护旧订单内部形状的测试
```

保留条件：

```text
只读历史展示。
一次性迁移脚本。
明确标注删除条件的 legacy adapter。
```

## 6. 第一执行切片

第一刀不做 Harness 大改，先把任务系统内部权威改正：

```text
1. 新增 TaskContract / GraphTaskContract / NodeExecutionContract。
2. 新增 TaskContractIssuer，从 SpecificTaskAssemblyPolicy + TaskEnvironmentSpec 签发合同。
3. API 创建具体任务时先签发 TaskContract，再由 legacy adapter 投影旧读模型。
4. TaskOrderFactory 降级为来源记录，不再生成任务合同。
5. 删除或改写“agent 发起任务进入 TaskSystem”的测试和文档引用。
```

## 7. 禁止模式

```text
不要让订单工厂继续签发任务合同。
不要为了保留旧链路把 TaskContract 包成 TaskExecutionEnvelope。
不要把 agent 自发任务接回任务系统。
不要让图节点重新走任务系统签子任务合同。
不要从 projection / domain_id / task_selection 恢复控制事实。
不要把 Harness 内部子 loop 伪装成任务系统任务。
不要为了“同源”牺牲职责边界。
```

## 8. 验收标准

结构验收：

```text
任务系统输出 TaskContract，而不是 TaskOrderRun / TaskExecutionEnvelope。
SpecificTask 是固定任务工作申请模板。
TaskEnvironment 是任务合同的资源和权限上限。
订单只保留来源记录职责。
agent 自发执行生命周期留在 Harness。
固定任务执行生命周期也留在 Harness。
```

运行验收：

```text
固定单 agent 任务可以从 SpecificTask 签发合同并进入 Harness。
固定图任务可以从 SpecificTask 签发 GraphTaskContract。
GraphRuntimeAssemblyPlan / GraphLoopPlan 作为合同请求规格交给 Harness。
RuntimeStartPacket 消费 TaskContract，不反解 task_selection。
LoopState 由 Harness 推进，任务系统不保存执行生命周期引用。
```

一句话总结：

```text
先把系统预定义任务做稳：任务环境、具体任务、任务合同。
agent 调用任务库和沉淀任务模板是后续能力，本轮关闭。
```

# 工作记忆层与子 Agent 动态记忆治理重构设计书

日期：2026-05-08

## 0. 设计结论

本轮不先做“大一统长期记忆数据库”，而是新增一层正式的 `Working Memory`。

工作记忆是任务执行过程中的结构化生产状态，服务任务图、RunLoop、子 Agent 协作、章节/长任务连续性和最终沉淀。

最终记忆层级调整为：

```text
Conversation Memory
  当前会话近期上下文、对话热窗口。

State Memory
  当前 session / turn / flow 的恢复候选和运行状态。

Working Memory
  当前任务/任务图/节点/产物在执行期间产生的结构化生产材料。

Long-term / Durable Memory
  经治理后沉淀的稳定知识、偏好、项目约定、外部引用。
```

核心规则：

```text
写入默认隔离。
读取默认最小可见。
共享必须显式。
采纳必须由 RunLoop / Coordinator / Human Gate 裁定。
晋升长期记忆必须经过治理。
任务完成必须收束。
```

新增最高约束：

```text
Working Memory 必须是通用任务基础设施，不得写死为单一任务专用系统。
Working Memory 必须与 Conversation / State / Long-term / Durable 严格隔离，不得混写、混读、混晋升。
Agent 通信与 Working Memory 必须协同，不能形成两套并行事实来源。
```

## 1. 背景与当前断点

### 1.1 当前系统已有能力

当前代码中已经存在：

- `backend/memory_system/facade.py`
  - 聚合 conversation / state / long_term / durable。
- `backend/memory_system/contracts.py`
  - 定义 `MemoryContextCandidate`、`StateMemoryRestoreCandidate`、`MemoryWriteCandidate`。
  - 明确记忆候选不能自带当前事实权威。
- `backend/memory_system/supply.py`
  - 定义 `MemoryRequest`、`MemoryScopePolicy`、`MemoryBundle`、`MemoryWritebackProposal`。
- `backend/memory_system/long_term_memory.py`
  - 将 durable memory header 转成长期记忆记录和 context candidate。
- `frontend/src/components/workspace/views/MemoryView.tsx`
  - 前端按 conversation / state / durable 三层展示。
- `frontend/src/components/workspace/views/task-system/TaskRunLoopWorkbenchPanel.tsx`
  - 已把任务侧 RunLoop / Memory 请求配置放进工作台。

### 1.2 当前不足

现有系统的问题不是“没有记忆”，而是缺少任务生产过程中的中间治理层：

1. 子 Agent 工作中产生的材料没有统一的结构化存储位置。
2. 子 Agent 写入内容要么留在 trace / 消息里，要么过早尝试进入 long-term。
3. 任务图节点之间缺少“可动态读取的工作上下文池”。
4. 多 Agent 协作时，哪些记忆私有、哪些共享、哪些只能交接给下游，没有正式策略。
5. 长篇写作、章节循环、设定连续性需要的“草稿事实、人物变化、伏笔、冲突标记”不适合直接进入长期记忆。
6. 任务完成后缺少收束机制：归档、晋升、废弃、产物绑定、健康问题登记。

### 1.3 本设计要解决的系统属性

本设计修复的是：

```text
任务运行期间的生产状态所有权、可见性、读写治理和完成后沉淀机制。
```

不是：

```text
把长期记忆替换成数据库。
把所有 agent 的上下文都持久化。
让子 agent 自由读取所有记忆。
把工作记忆直接当作当前事实。
```

## 2. 工作记忆层定位

### 2.1 定义

Working Memory 是任务运行过程中的结构化工作材料库。

它记录：

- 本轮任务目标
- 当前章节/阶段计划
- 子 Agent 输出草稿
- 审查意见
- 冲突标记
- 角色/世界/时间线变化草案
- 待合并结论
- 待晋升长期记忆候选
- 产物引用

### 2.2 与其他记忆层区别

| 层级 | 生命周期 | 权威程度 | 主要消费者 | 典型内容 |
|---|---|---|---|---|
| Conversation | 会话内 | 低 | AgentLoop | 最近对话、用户输入 |
| State | 会话/运行状态 | 中 | RunLoop / restore | 当前 goal、flow state、context slot |
| Working | 任务运行期 | 中高，但需采纳 | GraphRunLoop / 子 Agent | 计划、草稿、设定变化、冲突、产物引用 |
| Long-term | 跨任务长期 | 高，但需验证 | MemoryCompiler / AgentLoop | 稳定偏好、项目约定、长期设定 |

### 2.3 工作记忆不是长期记忆

工作记忆中允许存在：

- 未确认草稿
- 互相冲突的候选
- 子 Agent 私有观察
- 暂时有效的章节状态
- 需要审查的事实声明

长期记忆中不应该直接保存这些未收束内容。

## 3. 对象边界

### 3.1 Agent 与 Capability Unit

工作记忆所有权只允许绑定真实 Agent。

遵守 `29-Agent与CapabilityUnit边界统一定义`：

- `owner_agent_id` 只能来自 AgentRegistry。
- Capability Unit 不拥有工作记忆。
- Tool / MCP Endpoint 只能作为 `source_ref` 或 `artifact_ref` 出现。

### 3.2 TaskGraph 与 Working Memory

TaskGraph 不吞掉记忆本体。

TaskGraph 只声明：

- 节点是否可读工作记忆
- 节点可读哪些 scope
- 节点可写哪些 kind
- 边交接时携带哪些工作记忆引用

Working Memory 是独立系统，由 RuntimeAssembly / RunLoop 组装。

### 3.3 RunLoop 与 Working Memory

GraphRunLoop 负责：

- 创建 task run 的 working memory workspace
- 根据节点策略选择可见工作记忆
- 接受子 Agent 的工作记忆写入候选
- 采纳/拒绝/标记冲突
- 在任务完成时收束

AgentLoop 负责：

- 读取被 RuntimeAssembly 注入的工作记忆切片
- 产出工作记忆写入候选
- 不直接决定候选是否成为共享事实

### 3.4 Agent 通信与 Working Memory 的边界

必须明确：

```text
通信是交接通道。
Working Memory 是受控工作状态池。
两者不能各自维护一套相互独立的任务事实。
```

通信层负责：

- 发送 handoff payload
- 发送 review / ack / merge / reject 信号
- 发送必要的 artifact refs
- 发送 working memory ref 或摘要

Working Memory 负责：

- 持有任务过程中需要复用的结构化工作状态
- 为下游节点提供可再次读取的受控切片
- 记录采纳、冲突、替代、晋升和归档

禁止：

- A2A payload 携带完整工作记忆副本并绕过 Working Memory Store。
- 下游节点仅依赖消息文本、不经 RunLoop 选择就把通信内容视为共享事实。
- 同一条任务事实同时在通信载荷和 Working Memory 中演化，但没有统一 version / source_ref。

## 5. 子 Agent 存储策略

### 5.1 默认写入原则

子 Agent 不直接写共享事实。

默认写入：

```text
status = draft
visibility = private_to_agent 或 private_to_node
authority = candidate_only
```

### 5.2 按 Agent 角色的写入策略

| Agent 角色 | 默认可写 kind | 默认 scope | 默认 visibility |
|---|---|---|---|
| 主 Agent / Coordinator | task_goal, decision_record, handoff_context | task_scope / graph_scope | shared_in_graph |
| 规划 Agent | plan_fragment, outline_fragment, chapter_goal | graph_scope | proposed |
| 写作 Agent | scene_draft, chapter_draft, character_state_delta | node_scope / artifact_scope | private_to_node |
| 审查 Agent | review_note, conflict_flag, continuity_conflict | graph_scope | coordinator_only |
| 整理 Agent | promotion_candidate, summary, artifact_ref | graph_scope / artifact_scope | proposed |
| 健康/质量 Agent | conflict_flag, health_candidate | graph_scope | human_review_only |

### 5.3 写入权限来源

写入权限由三处共同决定：

```text
Agent RuntimeProfile
TaskGraphNode.memory_writeback_policy
TaskRunLoop policy
```

禁止：

- 子 Agent 自行提升 visibility。
- 子 Agent 自行把 draft 改为 accepted。
- 子 Agent 自行晋升 long-term。

### 5.4 覆盖与版本

工作记忆不原地覆盖。

更新时生成新版本：

```text
parent_item_id = previous_work_memory_id
version = previous.version + 1
previous.status = superseded
```

只有 Coordinator / RunLoop 可以把旧版本标记为 superseded。

## 6. 子 Agent 读取策略

### 6.1 默认读取原则

读取默认最小可见。

每个节点运行前，RunLoop 调用：

```text
WorkMemorySelector.select_for_node()
```

输入：

```text
task_run_id
graph_id
node_id
agent_id
node_role
stage
contract_refs
edge_context
memory_request_profile
token_budget
```

输出：

```text
WorkMemoryContextSlice
  required_items
  preferred_items
  optional_refs
  excluded_items
  diagnostics
```

### 6.2 读取过滤顺序

```text
1. scope 过滤
2. visibility 过滤
3. node role 过滤
4. contract kind 过滤
5. stage 过滤
6. freshness / status 过滤
7. token budget 排序
8. conflict / superseded 排除
```

### 6.3 不同 Agent 的默认读取范围

规划 Agent：

- task_goal
- prior decision_record
- outline_fragment
- unresolved conflict_flag
- accepted artifact_ref

写作 Agent：

- chapter_goal
- scene_plan
- accepted character_state_delta
- accepted world_state_delta
- style_constraint
- relevant artifact_ref

审查 Agent：

- chapter_draft / scene_draft
- timeline_event_draft
- style_constraint
- contract_refs
- prior continuity_conflict

整理 Agent：

- accepted / proposed 工作记忆
- review_note
- artifact_ref
- promotion_candidate

### 6.4 动态读取

子 Agent 可以请求增量读取，但请求必须经过 RunLoop：

```text
AgentMemoryReadRequest
  requested_kind
  topic
  reason
  max_items
```

RunLoop 返回：

```text
approved_context_slice
denied_reason
```

禁止子 Agent 直接查询完整工作记忆库。

### 6.5 通信到读取的协调规则

当节点收到上游消息时，不允许直接把消息内容和工作记忆同时注入而不做协调。

必须经过：

```text
HandoffResolver
  -> 解析通信载荷
  -> 拆分 signal / summary / artifact_refs / working_memory_refs
  -> 判断哪些内容只是一次性交接
  -> 判断哪些内容需要写入或更新 Working Memory
  -> 再由 WorkMemorySelector 生成最终读取切片
```

协调原则：

1. `signal` 只用于推进 RunLoop，不进入工作记忆正文。
2. `summary` 可作为一次性上下文，但不能直接替代已 accepted 的工作记忆。
3. `working_memory_refs` 优先于消息里重复拷贝的正文。
4. 如通信内容声明了新事实，先形成 `draft/proposed` 工作记忆候选，再决定是否对下游可见。
5. 若通信内容与现有 accepted working memory 冲突，必须生成 `conflict_flag`，不能直接覆盖。

## 7. 任务处理中管理策略

### 7.1 生命周期

```text
create_workspace
  创建 task_run_id 级工作记忆空间。

node_read
  节点运行前选取上下文切片。

node_write
  节点结束后提交写入候选。

coordinator_review
  协调者采纳、拒绝、合并或标冲突。

handoff
  边交接时附带可见工作记忆引用。

checkpoint
  RunLoop 中断恢复时保存当前 working set 索引。

finalize
  任务完成后归档、晋升、废弃、产物绑定。
```

### 7.2 冲突管理

冲突来源：

- 两个 Agent 对同一事实给出不同版本。
- 工作记忆与长期记忆不一致。
- 时间线顺序矛盾。
- 人物状态不连续。
- 契约验收失败。

冲突记录：

```text
kind = conflict_flag 或 continuity_conflict
status = conflicted
visibility = coordinator_only
metadata.conflict_with = [...]
```

冲突处理：

```text
Coordinator resolve
Human gate resolve
Reviewer agent propose resolution
```

### 7.3 采纳策略

draft 变 accepted 的条件：

- 符合节点输出契约。
- 没有未处理 blocker 冲突。
- 通过 coordinator review 或自动采纳规则。
- 若是长期影响项，必须生成 promotion_candidate 而不是直接长期写入。

### 7.4 通信与工作记忆的一致性管理

RunLoop 需要维护：

```text
message_ref -> work_memory_id
work_memory_id -> latest_message_refs
handoff_id -> adopted_items
```

目的：

- 追踪某条通信消息是否已经被采纳为 working memory。
- 避免同一 handoff 被重复读取并重复写入。
- 让下游节点知道“看消息摘要”还是“看已采纳工作记忆”。

一致性规则：

1. 上游 handoff 若已经生成 accepted work item，下游优先读取 accepted item。
2. 同一 handoff 不得在多个节点重复创建语义相同但 ID 不同的工作记忆项。
3. 通信文本中的事实更新若未采纳，只能作为 ephemeral handoff context。
4. 采纳后的 working memory item 必须记录 source_message_ref / handoff_id。

## 8. 任务完成后的存储策略

### 8.1 收束分流

任务完成时，RunLoop 执行：

```text
WorkMemoryFinalizer.finalize_task_run()
```

分流规则：

| 工作记忆状态/类型 | 收束去向 |
|---|---|
| draft 且未被引用 | discarded 或 archived |
| accepted plan / decision | archived |
| artifact_ref / chapter_draft | Artifact Store |
| story_bible_delta / stable character_state_delta | promotion_candidate |
| conflict_flag 未解决 | Health Issue / Review Queue |
| style_constraint / user preference | Long-term candidate |
| handoff_context | run trace archive |

### 8.2 晋升长期记忆

晋升不是自动保存。

流程：

```text
promotion_candidate
  -> MemoryGovernance review
  -> accepted
  -> LongTermMemory / DurableMemory
```

晋升要求：

- 有 source_event_refs。
- 有 artifact_refs 或 evidence。
- 有 stability 标记。
- 通过冲突检查。
- 明确 memory_type / memory_class。

### 8.3 写作任务收束

长篇写作任务完成一章后：

```text
chapter_draft -> Artifact Store
chapter_summary -> Working archive + possible long-term candidate
character_state_delta -> candidate
world_state_delta -> candidate
timeline_event_draft -> candidate
foreshadow_track -> working archive
continuity_conflict -> health/review queue
style_constraint -> long-term candidate if stable
```

## 9. RuntimeAssembly 接入

### 9.1 新增 Context Section

RuntimeAssembly 中新增工作记忆段：

```text
context_sections:
  - section_id: working_memory.required
  - section_id: working_memory.preferred
  - section_id: working_memory.artifact_refs
  - section_id: working_memory.conflict_warnings
```

### 9.2 Assembly 只携带切片

RuntimeAssembly 不携带完整工作记忆库。

只携带：

- 当前节点需要的工作记忆摘要
- 可按需读取的 refs
- 被排除项 diagnostics

## 10. RunLoop 接入

### 10.1 GraphRunLoop

GraphRunLoop 增加：

```text
working_memory_workspace_ref
working_memory_checkpoint_ref
working_memory_selector
working_memory_writeback
working_memory_finalizer
```

### 10.2 AgentLoop

AgentLoop 只看到：

```text
model_visible_working_memory_sections
dynamic_read_tool_or_internal_request
writeback_contract
```

AgentLoop 不直接操作数据库。

### 10.3 A2A 交接

A2A payload 不直接塞完整记忆。

边交接携带：

```text
working_memory_refs
handoff_summary
artifact_refs
visibility_policy
```

下游节点由 RunLoop 再次选择可见切片。

### 10.4 通信与工作记忆的统一装配入口

GraphRunLoop 中新增一层：

```text
CommunicationMemoryCoordinator
```

职责：

- 统一接收 A2A / handoff payload
- 解析 payload 中的 summary、artifact refs、working memory refs、signals
- 生成工作记忆写入候选或一次性交接上下文
- 调用 WorkMemorySelector 重新生成下游可见切片

最终约束：

```text
所有跨节点工作状态复用，都必须经过 CommunicationMemoryCoordinator。
不能一部分走通信直读，一部分走 Working Memory 直读。
```

## 14. 迁移与兼容

### 14.1 不删除旧记忆

旧 durable memory 保留。

Working Memory 上线后：

- 旧 durable memory 继续服务长期召回。
- 新工作记忆服务任务运行期。
- promotion 后才写入 durable / long-term。

### 14.2 overlap 期

第一阶段：

```text
Working Memory 只新增，不替换。
RunLoop 可以写入工作记忆，但长期召回仍走旧 long_term adapter。
```

第二阶段：

```text
MemoryCompiler 同时读 working + long_term。
RuntimeAssembly 注入 working memory context sections。
```

第三阶段：

```text
任务完成后 promotion candidate 进入治理流程。
```

## 16. 验证矩阵

### 16.1 单 Agent

- 单节点任务可写 node_scope 工作记忆。
- 下轮同一节点可读取 accepted 工作记忆。
- draft 不自动进入 long_term。

### 16.2 多 Agent

- A 写入 private_to_agent，B 不可见。
- Coordinator 采纳后，B 可按策略读取。
- 边交接只带 handoff_only 指定 refs。
- 同一 handoff 中的 summary 与 working_memory_refs 不冲突时，下游优先读取 accepted item。
- handoff 中的新事实若未采纳，只作为一次性上下文，不进入共享 working set。

### 16.3 写作任务

- 写作 Agent 写 chapter_draft。
- 审查 Agent 写 continuity_conflict。
- 整理 Agent 生成 promotion_candidate。
- 完成后 chapter_draft 进入 artifact，设定变化进入待晋升。

### 16.4 冲突

- 两个人物状态变化冲突时标记 conflicted。
- conflicted item 不进入下游 required context。
- Coordinator resolve 后生成 accepted 新版本。

### 16.5 权限

- Capability Unit 不可作为 owner_agent。
- 子 Agent 不可直接 accept 自己的 draft。
- 动态读取超出 scope 被拒绝并记录 read log。

### 16.6 通信一致性

- A2A 消息中的 `working_memory_refs` 能正确映射到 store 中 item。
- 重复 handoff 不会生成重复 work item。
- handoff 文本与 accepted working memory 冲突时生成 conflict_flag。
- 下游节点不会同时拿到两份语义重复但版本不同的事实正文。

## 17. 禁止事项

实现中禁止：

1. 把工作记忆直接写成长期记忆。
2. 让子 Agent 自由查询全库。
3. 让 Capability Unit 拥有工作记忆。
4. 让工作记忆 candidate 覆盖当前 turn truth。
5. 在 RuntimeAssembly 中塞完整工作记忆库。
6. 用 prompt 文案代替 visibility / scope / status 结构。
7. 为写作任务写死小说字段到通用模型顶层。
8. 跳过任务完成收束。
9. 让通信消息和 Working Memory 各自维护一套事实而不做协调。
10. 让下游节点直接把 handoff 文本当作共享工作事实，不经过采纳。

## 18. 最终目标

工作记忆层完成后，系统应具备：

```text
任务运行中可保存。
子 Agent 可按需读取。
多 Agent 可受控共享。
RunLoop 可 checkpoint 和恢复。
任务完成后可归档、晋升、废弃。
长期记忆不会被草稿污染。
写作、开发、健康、研究等任务都能复用同一治理层。
```

# Vibe Coding 能力移植规划

日期：2026-05-24

## 1. 结论

可以做，但不建议把 `D:/AI应用/opencode-main` 当作代码骨架整体迁入。

更稳的路线是：把 OpenCode 当作“专业 coding agent 内核参考”，抽取它的工作区状态、文件编辑协议、补丁协议、诊断闭环、权限确认、coding prompt 行为约束，重建为我们平台里的一个 `vibe_coding` 专业模式。

原因很直接：

- OpenCode 是 Go + TUI + 单 coding agent 产品，核心入口在 `D:/AI应用/opencode-main/internal/llm/agent/agent.go`。
- 我们是 Python 后端 + Next 工作台 + 任务图/多 Agent/记忆/权限/能力系统平台，主干在 `backend/runtime`、`backend/capability_system`、`backend/agent_system`、`backend/task_system`。
- 直接搬 OpenCode 骨架会制造第二套 session、message、permission、tool loop、LSP、history 存储，和我们已有运行时冲突。
- 真正缺的不是“agent 编排”，而是“coding 专用内核”：工作区编辑状态、读后写保护、patch 原子应用、诊断反馈、命令语义、diff 审批、验证闭环。

目标不是做一个 OpenCode clone，而是让我们的 Agent 平台新增一个专业 coding 工作模式：用户提出代码任务后，系统能像 Codex/OpenCode 一样自然地读代码、改代码、跑验证、展示 diff、解释风险，并把过程收束成可审查的产物。

## 2. 源码依据

### 2.1 OpenCode 的核心内核

OpenCode 的主循环很轻，但非常垂直：

- `D:/AI应用/opencode-main/internal/llm/agent/agent.go:233`：`processGeneration` 负责消息历史、总结截断、用户消息创建、工具循环。
- `D:/AI应用/opencode-main/internal/llm/agent/agent.go:322`：`streamAndHandleEvents` 流式接收模型事件，落库 assistant message，再执行 tool calls。
- `D:/AI应用/opencode-main/internal/llm/agent/agent.go:445`：`processEvent` 处理 thinking/content/tool start/tool stop/complete。
- `D:/AI应用/opencode-main/internal/llm/agent/tools.go:14`：`CoderAgentTools` 固定装配 `bash/edit/fetch/glob/grep/ls/sourcegraph/view/patch/write/agent`。
- `D:/AI应用/opencode-main/internal/llm/agent/tools.go:43`：`TaskAgentTools` 是只读搜索子 Agent 工具集。
- `D:/AI应用/opencode-main/internal/llm/tools/tools.go:69`：工具接口只有 `Info()` 和 `Run()`，很适合垂直 coding 产品，但不够承载我们已有的权限、合同、投影和运行追踪。

OpenCode 的 coding 能力核心不是复杂多 Agent，而是工具协议很硬：

- `D:/AI应用/opencode-main/internal/llm/tools/view.go:99`：View 工具读取文件、行号化、记录已读文件、触发 LSP open。
- `D:/AI应用/opencode-main/internal/llm/tools/edit.go:124`：Edit 工具要求文件已读、检查文件修改时间、要求 `old_string` 唯一匹配，写入后返回 diff 元数据和 LSP diagnostics。
- `D:/AI应用/opencode-main/internal/llm/tools/patch.go:89`：Patch 工具一次处理多文件变更，先确认涉及文件已读、检查文件未被外部改动，再解析 patch、申请权限、应用 commit、写 history、跑 diagnostics。
- `D:/AI应用/opencode-main/internal/llm/tools/diagnostics.go:68`：Diagnostics 工具把 LSP 诊断转成模型可消费的文本。
- `D:/AI应用/opencode-main/internal/permission/permission.go:74`：写入和命令执行会阻塞等待用户授权，支持 session 级持久授权。
- `D:/AI应用/opencode-main/internal/llm/prompt/coder.go:74`：Coder prompt 直接定义“读代码、改代码、验证、git sanity check”的行为范式。

### 2.2 我们已有的承接点

我们已有平台能力不少，适合承接为更强的 coding mode：

- `backend/capability_system/tool_runtime.py:12`：已有工具运行时和工具定义注册。
- `backend/runtime/tool_runtime/tool_executor.py:19`：已有 ToolRuntimeExecutor，包含 dispatch guard、ToolContractGate、sandbox overlay、execution record、result envelope。
- `backend/runtime/tool_runtime/sandbox_backend.py:51`：已有 local overlay sandbox，可作为 coding mode 的默认写入边界。
- `backend/capability_system/workspace_file_service.py:36`：已有 workspace 内路径解析、读写、glob、安全根目录约束。
- `backend/capability_system/units/tools/write_file_tool.py:35`：已有 `write_file`。
- `backend/capability_system/units/tools/write_file_tool.py:66`：已有 `edit_file`。
- `backend/capability_system/units/tools/terminal_tool.py:33`：已有 `terminal`。
- `backend/agent_system/registry/worker_agent_factory.py:18`：已有 worker blueprint，包括 explorer/planner/verification/execution。
- `backend/runtime/professional_runtime/driver.py:104`：已有 professional task run driver，能做任务合同、工具轮次、验证 closeout、completion judgment。

当前缺口在 coding 细节，不在编排框架：

- `write_file/edit_file` 太轻，缺少“必须先读文件”“文件被外部修改后拒绝写”“唯一匹配保护”“diff 元数据”“诊断闭环”。
- `terminal` 只做基础危险命令过滤和 5000 字符截断，缺少 coding 命令语义、长输出持久化、持续任务、验证命令分类。
- 工具结果还没有形成 coding 专用的 `WorkspaceChangeSet`，前端难以用“文件树 + diff + diagnostics + 验证结果”的方式管理。
- Agent 模式已有 explorer/planner/verification/execution，但没有一个统一的 coding session 内核把它们编成“探索 -> 计划 -> 修改 -> 验证 -> 收口”的自然流程。

## 3. 设计原则约束

来自 `docs/设计原则` 的约束应转成落地规则：

- 对话循环要保留流式状态机优势。`docs/设计原则/05-对话循环.md:25` 强调 AsyncGenerator 驱动状态机，工具结果和恢复过程应能实时投递。我们在 Python 里不必照搬 AsyncGenerator 形式，但 coding mode 必须保留事件流。
- 上下文管理要分层。`docs/设计原则/06-上下文管理.md:18` 强调多层压缩，`docs/设计原则/06-上下文管理.md:162` 强调 microcompact 清理旧工具结果。coding mode 应保留最近关键 diff/diagnostics，把大文件读取、命令输出转成 ref 和摘要。
- 工具必须 fail-closed。`docs/设计原则/09-工具系统设计.md:12` 与 `:125` 强调安全默认，`docs/设计原则/10-BashTool-深度剖析.md:164` 强调 shell AST/安全分析 fail-closed。我们的 coding shell 不确定时必须 ask/deny，不应静默放行。
- Agent 隔离要显式。`docs/设计原则/12-Agent-系统.md:194` 强调 `runAgent()` 生命周期和 context 隔离，`docs/设计原则/13-内置Agent设计模式.md:94` 强调只读 Agent 的硬约束。coding 子 Agent 必须通过 profile/operation scope 禁写，而不是只靠 prompt。
- 任务系统要统一生命周期。`docs/设计原则/14-任务系统.md:17` 强调并发任务生命周期管理。coding mode 的测试、dev server、长命令、后台检查都应成为可追踪任务，而不是裸 subprocess。

## 4. 推荐架构

### 4.1 新增一个专业模式，而不是替换主运行时

新增 `vibe_coding` interaction mode，作为 professional runtime 的一个领域绑定：

- 入口：用户要求代码修改、调试、重构、测试、前端实现时触发。
- 运行目标：真实代码变更 + 真实验证证据 + 可审查 diff。
- 输出边界：最终回答只消费 `WorkspaceChangeSet`、验证记录、关键说明，不泄露工具协议。

建议命名：

- `coding_session`：一次代码任务的状态容器。
- `workspace_change_set`：本轮变更的权威 diff 包。
- `coding_kernel`：探索、编辑、验证、收口的控制器。
- `coding_agent_mode`：agent profile / prompt / tool scope 的装配层。

### 4.2 Coding Kernel 的核心状态

新增状态模型：

```text
CodingSession
- session_id
- workspace_root
- mode: inspect | plan | edit | verify | review
- read_set: path -> {mtime, size, hash, last_read_at}
- write_set: path -> {before_hash, after_hash, diff_ref, additions, removals}
- diagnostics: path -> diagnostics[]
- command_runs: command_id -> {command, cwd, exit_code, stdout_ref, stderr_ref}
- approvals: approval_id -> status
- final_change_set_ref
```

关键规则：

- 写入前必须有 read_set，除非是新文件。
- 写入前必须检查文件 mtime/hash 未变。
- `edit_file` 必须唯一匹配，不能用简单 `replace(..., 1)` 静默改错位置。
- 多文件变更优先走 patch/change_set，前端展示统一 diff。
- 验证命令输出大于阈值必须落盘成 ref，模型只看摘要和关键尾部。

### 4.3 Coding 工具层

不直接搬 Go 工具，重建 Python 版工具：

1. `view_file`
   - 替代或增强现有 `read_file`。
   - 输出带行号、大小限制、编码识别、binary/image 拒绝。
   - 写入 `CodingSession.read_set`。

2. `edit_file_strict`
   - 基于 OpenCode `edit.go` 的约束。
   - 必须已读、mtime/hash 未变、old_text 唯一匹配。
   - 返回 diff metadata，不只返回“Edit succeeded”。

3. `apply_patch_strict`
   - 基于 OpenCode patch grammar，但用 Python 实现。
   - 一次处理 add/update/delete/move。
   - 先 parse -> validate -> build change set -> approval -> apply。
   - fuzz 默认不允许，除非进入专门 repair 轮。

4. `workspace_diagnostics`
   - 初期可先接命令型 diagnostics：TypeScript 用 `npx tsc --noEmit`，Python 用项目已有测试/ruff/pytest。
   - 后期再接 LSP client。

5. `coding_terminal`
   - 包装现有 `terminal`，增加 coding 命令语义、输出持久化、验证标签。
   - 读搜类命令引导模型用专用工具，不鼓励 terminal 代替 grep/read。

### 4.4 Agent 模式层

不要只加一个“万能 coding agent”。建议四个角色，但由一个 coding kernel 串起来：

- `coding.explorer`：只读探索，允许 `read/search/glob/git_diff/git_status`。
- `coding.planner`：只读计划，输出修改路径、风险、验证命令。
- `coding.executor`：允许 strict edit/patch/write/terminal，但必须受 change set 和 approval gate 约束。
- `coding.verifier`：项目目录禁写，允许验证命令，输出可复现证据。

我们已有 `worker.explorer`、`worker.planner`、`worker.execution`、`worker.verification`，可以升级这些 blueprint，而不是新建第二套 agent 系统。

## 5. 不建议照搬的部分

- 不搬 OpenCode 的 session/message/db。我们已有 session、runtime event log、checkpoint、memory。
- 不搬 Go TUI。我们前端是 Next 工作台，需要做代码任务工作台视图。
- 不搬 OpenCode 的极简 `BaseTool` 接口。我们已有 ToolContractGate、OperationGate、ResourcePolicy、execution receipt，这些比 OpenCode 更适合平台。
- 不搬 OpenCode 的 bash 安全实现原样到 Windows PowerShell。我们的环境是 Windows，shell 安全要按 PowerShell 语义重做。
- 不把 OpenCode prompt 原文直接塞进 agent prompt。要转成 agent 能理解的角色职责，避免把开发说明当 prompt。

## 7. 第一版最小可行范围

第一版不要一口气做完整 IDE。

MVP 只做：

1. `view_file`：读文件并登记 read_set。
2. `edit_file_strict`：严格替换并返回 diff。
3. `apply_patch_strict`：多文件 patch。
4. `coding_terminal`：运行验证命令并返回 receipt。
5. `vibe_coding` mode：读 -> 改 -> 验证 -> 总结。
6. 前端：change set + diff + verification 三块。

暂缓：

- 真 LSP server 管理。
- Git commit/PR 自动化。
- dev server 后台任务常驻。
- OpenCode 式 TUI 体验。
- 跨仓库多 workspace。

## 8. 风险与控制

### 风险 1：又造一套工具系统

控制：所有 coding tools 仍注册进 `ToolRuntime` 和 `OperationRegistry`，不绕过 `ToolRuntimeExecutor`。

### 风险 2：Prompt 说得很好，但写入边界不硬

控制：read-before-write、hash check、unique match、permission diff approval 必须在工具层执行。

### 风险 3：terminal 变成万能逃逸口

控制：coding_terminal 只作为验证和必要命令执行；读搜优先专用工具；危险/不确定命令 fail-closed。

### 风险 4：上下文被文件内容和命令输出撑爆

控制：大文件只读片段；大输出落盘成 ref；final answer 只消费 change set 和 verification 摘要。

### 风险 5：前端做成复杂仿 IDE

控制：第一版只做管理面板：文件变更、diff、验证、审批，不做编辑器替代品。

## 9. 推荐执行顺序

1. 先做 `CodingSession + read_set/write_set/change_set`。
2. 再替换/新增 strict edit 和 patch 工具。
3. 再接 diff approval。
4. 再接 coding terminal 和 verification evidence。
5. 最后装配 `vibe_coding` mode 和前端工作台。

这条路线能最大化复用我们的平台，又能吸收 OpenCode 真正有价值的 coding 内核。

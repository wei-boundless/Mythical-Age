# 前端整体重设计审查报告

日期：2026-05-07

## 结论摘要

当前前端并不差，甚至已经有比较明确的品牌气质和系统感，尤其是“灵魂切换驱动全局风格”的机制，是整个产品最有辨识度、也最值得保留的部分。

但从产品体验和现代 UI 判断看，当前前端存在三个根问题：

1. 壳层过重，功能区过轻。
2. 信息过多，动作路径不够锋利。
3. 视觉语言过满，导致核心操作不够突出。

换句话说，现在的问题不是“不会设计”，而是“设计表达压过了工作流本身”。

## 本次审查范围

本次审查基于以下内容：

- 实际前端结构：
  - `frontend/src/app/page.tsx`
  - `frontend/src/components/layout/Navbar.tsx`
  - `frontend/src/components/layout/Sidebar.tsx`

  - 使用 Playwright 打开任务系统协调任务页面进行截图与尺寸检查

## 应保留的部分

### 1. 灵魂切换驱动风格机制

这是当前前端最有产品身份感的能力，必须保留。

价值在于：

- 它不是普通换肤，而是“人格 / 视角 / 操作气质”与 UI 风格联动。
- 它能让产品形成稳定识别度，而不是普通深色后台。
- 它天然适合你的系统设定，不是外加装饰。

建议保留方式：

- 保留 `html[data-soul="..."]` 变量系统。
- 保留全局色相、发光、纹理、微动效的切换。
- 但把灵魂风格从“重度占据布局”改成“轻量影响界面气质”。

也就是说：

- 保留气质
- 降低体积
- 避免每个页面都像一个视觉展板

### 2. 模块化工作台方向

当前以“会话 / 任务系统 / 编排系统 / 记忆系统 / 健康系统 / 能力系统 / MCP / 灵魂系统 / 系统配置”组成的工作台方向是对的。

这说明产品已经有平台化结构，而不是单页面玩具。

需要改的不是有没有模块，而是：

- 模块入口太重
- 模块壳层重复太多
- 模块切换成本偏高

### 3. 协调任务拓扑图化方向

协调任务和协调监控坚持图形化，是正确方向，而且应当继续强化。

这部分最接近未来产品形态：

- 用户搭建协调任务
- 系统运行同一拓扑
- 用户在监控中观察节点状态、产物、输出

问题不是方向错，而是围绕图的界面组织还不够克制。

## 当前主要问题

## 一、壳层过重

当前页面普遍存在三层以上固定壳层：

- 顶部品牌栏
- 左侧灵魂 / 会话栏
- 右侧模块工作台栏
- 中间真实功能区

这会导致两个结果：

1. 真正可工作的主区域被压缩。
2. 用户在操作前必须先穿过很多“说明性 UI”。

在任务系统和编排系统这种生产型页面里，这种结构尤其吃亏。

### 具体表现

- 左侧灵魂区视觉体积过大。
- 右侧工作台栏在功能页面中重复出现，抢占横向空间。
- 顶部品牌头部高度偏大，页面首屏有效工作面积不足。

### 影响

- 拓扑图、表格、检查器、流程编辑器都被压窄。
- 用户进入系统后先看到的是品牌与容器，而不是任务。
- 页面“像产品展示”，多于“像工作台”。

## 二、信息混杂，页面层级不够干净

当前很多页面把以下内容同时放在一个视口层级里：

- 品牌信息
- 状态信息
- 模块导航
- 数据摘要
- 操作按钮
- 编辑器正文
- 系统字段

这会让用户难以一眼判断：

- 我现在在哪一层
- 这个页面主要是看、改、还是运行
- 哪些信息是主流程，哪些只是元数据

### 任务系统里的典型问题

虽然任务系统已经在结构上分成：

- 任务域
- 任务装配
- 域内协调任务
- 契约

但单个层内仍然混有太多不同语义的信息块。

比如协调任务页里，之前同时出现：

- 协调任务标题
- 复制草稿
- 模板按钮
- 保存发布
- 拓扑状态
- 拓扑图
- 快速连线
- 任务池
- 节点检查器

如果这些东西没有清晰的视觉主次，就会显得笨重。

## 三、按钮尺度偏大，导致界面显笨

当前许多按钮和卡片有这些特征：

- 高度偏大
- 内边距偏厚
- 圆角偏重
- 同一层里按钮数量过多

单个看不难看，但在系统型页面里会形成“钝重感”。

这类 UI 在品牌页、展示页里可以成立，但在生产工具里会让人感觉：

- 反应慢
- 信息密度低
- 操作不够利落

### 具体判断

- 主操作按钮应该保留存在感。
- 次级操作和辅助动作应该更轻、更窄、更短。
- 图编辑器、检查器、表单页不适合大量大按钮并排堆积。

## 四、卡片化使用过多

当前 UI 有很强的“所有东西都住在卡片里”的倾向。

问题在于：

- 页面 section 像卡片
- 卡片内部还有卡片
- 卡片里又放按钮组和状态条

结果是页面出现明显的“块块分裂”。

这对系统型产品尤其不利，因为系统型工作台更需要：

- 连续的工作流
- 清楚的主次区块
- 更像“编辑台”而不是“陈列架”

## 五、品牌气质和生产工具气质没有完全分层

目前你的品牌气质很强：

- 神话感
- 灵魂人格感
- 古风 / 仪式感
- 纹理、光晕、装饰线

这很好，但生产工具页不应该把这种气质完整平铺到每个页面层级。

更合理的方式是：

- 壳层保留品牌气质
- 核心工具面板更克制、更清爽
- 灵魂风格通过色温、对比、细节和动势来表达

而不是通过大面积纹理、厚重容器、强装饰边界来表达

## 六、现代 UI 语境下还不够“结果导向”

现在很多页面仍然在告诉用户“这是哪个系统”“这个系统管什么”，但系统已经足够复杂后，用户更需要的是：

- 现在最该做什么
- 当前状态是否健康
- 下一步入口在哪
- 我刚刚改动的对象是什么

也就是从“说明型系统界面”转向“结果型工作界面”。

## 视觉与交互层面的核心判断

## 1. 当前前端的气质关键词

当前关键词大致是：

- 神话感
- 厚重
- 发光
- 纹理
- 仪式性
- 模块感

## 2. 目标前端更适合的关键词

建议的新方向关键词是：

- 神话气质
- 冷静
- 轻锋利
- 工作台
- 信息克制
- 高密但不拥挤

也就是：

保留“洪荒时代”的灵魂和气息，但把产品视觉从“展示型神话界面”转成“未来感神话工作台”。

## 建议的重设计原则

## 一、重新定义三层结构

建议把前端分成三层：

### 第一层：品牌层

负责：

- 灵魂风格
- 顶层品牌身份
- 全局色相与氛围

特征：

- 轻
- 稳
- 不抢功能区

### 第二层：导航层

负责：

- 模块切换
- 当前视图定位
- 全局状态入口

特征：

- 清楚
- 紧凑
- 可折叠

### 第三层：工作层

负责：

- 真正的编辑、运行、监控、检查

特征：

- 占主屏
- 低装饰
- 高可读
- 强主次

## 二、把灵魂机制从“大面积占位”改成“全局风格变量”

建议保留：

- 主色
- 次色
- 光晕
- 动态纹理
- 局部意象图

建议缩减：

- 过高的灵魂展示卡
- 在每个工作态页面里重复出现的大面积人格面板
- 装饰性说明文字

建议结果：

- 用户能感知当前灵魂
- 但不会因为灵魂 UI 太大而牺牲工作空间

## 三、统一按钮等级

建议把按钮分成 4 级：

1. 主命令：
   发布、运行、保存
2. 次命令：
   复制、创建、导入
3. 轻操作：
   编辑、切换、删除、反转
4. 图内微操作：
   节点工具、边工具

当前问题是这些等级在视觉上差距不够大。

建议未来：

- 主命令保留较强存在感
- 次命令变窄
- 轻操作更像工具按钮
- 图内微操作彻底工具化

## 四、减少“解释性标题 + 说明文字”的频率

现在很多区域都有：

- 标题
- 小字说明
- 状态行
- 再加卡片容器

这会导致每一块都像在做自我介绍。

建议：

- 只有真正新的页面层级才需要说明
- 同层级块不重复解释自己
- 多用直接结果信息，少用概念性说明

## 五、把任务系统和编排系统设计成真正的“专业工作面”

这是最关键的。

任务系统、编排系统、协调监控不是内容页，而是生产工具。

它们更适合：

- 更紧的工具条
- 更清楚的面板分工
- 更少的装饰卡片
- 更大的主工作区
- 更稳定的表单 / 图 / 列表比例

## 推荐的重设计方向

## 方向判断

最适合你的不是极简 SaaS，也不是华丽奇幻展示风，而是：

**神话气质的工业化工作台**

也可以理解为：

- 品牌层有神话感
- 工具层像专业编排台
- 灵魂系统像风格引擎，不像大海报

## 具体风格建议

### 1. 收窄顶部品牌栏

- 更短
- 更轻
- 只保留品牌名、全局动作、状态

### 2. 左侧改成折叠式主导航

保留：

- 灵魂切换
- 会话入口
- 模块导航

但改为可伸缩双态：

- 展开态：显示文字与少量摘要
- 收起态：只显示图标和灵魂状态

### 3. 右侧工作台栏改为上下文抽屉

不是常驻栏，而是：

- 手动展开
- 需要时显示
- 默认不抢主工作区

### 4. 核心工作区采用低装饰高密度面板

规则：

- 少卡片
- 少大圆角
- 少厚边框
- 少大按钮
- 强分栏
- 强层级

## 对当前各区域的审查结论

## 1. Navbar

问题：

- 品牌头太高
- 占首屏空间较多
- 对任务型页面来说信息价值偏低

建议：

- 高度减少约 25% 到 35%
- 标题压缩
- 保留灵魂色相与品牌字标

## 2. Sidebar

优点：

- 灵魂切换入口明确
- 会话入口集中

问题：

- 灵魂人物区占比过大
- 会话列表卡片偏厚
- 对任务工作态来说太像独立内容栏

建议：

- 灵魂区缩成紧凑头部
- 会话列表密度提高
- 支持任务工作态自动折叠

## 3. RightRail

优点：

- 模块导航清楚

问题：

- 常驻时浪费宽度
- 与左侧栏一起形成双边包夹
- 在任务/编排工作态里破坏主区

建议：

- 改成可展开导航抽屉
- 或并入左侧主导航

## 4. ChatPanel

优点：

- 会话 / 协调监控双页结构是对的

问题：

- 空状态仍偏“展示”
- 工具性还可以更强

建议：

- 保留双页结构
- 弱化空状态装饰
- 强化任务上下文与当前执行对象

## 5. TaskSystemView

优点：

- 功能完整
- 已有明确层次
- 协调任务编辑器方向正确

问题：

- 单页承载过多语义
- 壳层说明过多
- 工具区主次不够强
- 长时间使用会累

建议：

- 做专注工作模式
- 简化层内摘要块
- 提高操作区克制度
- 强化图形编辑中心地位

## 6. CoordinationRunPanel

优点：

- 方向非常好
- 拓扑、输出内容、产物拆分合理

问题：

- 还可以更锐利
- 顶部状态 pill 仍稍厚
- 输出面板可以更像日志/交付面，而非展示卡

建议：

- 进一步工具化
- 降低装饰感
- 提升监控感

## 总体建议

## 第一阶段：审美与布局纠偏

目标：

- 先把笨重感去掉

建议动作：

- 缩顶部
- 缩按钮
- 降低卡片厚度
- 让工作区更大
- 减少常驻双侧栏

## 第二阶段：统一交互等级

目标：

- 让按钮、导航、工具条形成明确等级

建议动作：

- 重做按钮体系
- 重做状态条体系
- 重做卡片与面板层级

## 第三阶段：保留灵魂机制并升级为真正的风格引擎

目标：

- 把灵魂机制从“好看设定”变成“产品核心识别层”

建议动作：

- 统一灵魂变量
- 把品牌层与工作层分离
- 让不同灵魂只影响气质，不压迫工作流

## 最终判断

当前前端值得推倒重做的不是“全部”，而是“结构和视觉秩序”。

应当保留的：

- 灵魂切换风格机制
- 模块化工作台思路
- 协调任务图形化方向
- 现有深色神话气质基础

应当重做的：

- 壳层结构
- 导航布局
- 按钮体系
- 卡片密度
- 任务页和编排页的工具化表达

一句话总结：

**保留灵魂，削掉笨重，减少解释，放大工作。**

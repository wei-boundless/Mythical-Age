# DeepSeek 候选切换链路修正报告

日期：2026-05-02

## 问题现象

长情景运行中出现日志：

- `Switching tool-enabled model candidate after provider_error on deepseek/deepseek-v4-pro`

表面上看像正常 fallback，但排查后发现，这不是单纯的健康降级，而是配置解析允许 provider/model 错配进入主链，随后被候选切换机制兜底。

## 根因

### 1. 主模型解析优先级有误

原链路中，`deepseek` provider 的模型解析会优先读取全局 `LLM_MODEL`，再读取 `DEEPSEEK_MODEL`。

这会导致：

- 已选 provider 是 `deepseek`
- base_url 是 `https://api.deepseek.com`
- 但模型名可能来自一个与 provider 专属配置不一致的全局值

在本次现场配置中：

- `LLM_MODEL=deepseek-v4-pro`
- `DEEPSEEK_MODEL=DeepSeek-V4-Flash`

因此主链实际拿到的是 `deepseek-v4-pro`，而不是 DeepSeek 专属配置想表达的稳定主模型。

### 2. fallback 解析也存在同类错配风险

原链路中，fallback model 也会优先读全局 `LLM_FALLBACK_MODEL`，再读 provider 专属模型字段。

这会导致例如：

- `LLM_FALLBACK_PROVIDER=bailian`
- `LLM_FALLBACK_MODEL=glm-5`

这样的 provider/model 错配被直接接入候选链。

### 3. 运行时日志不可观测

原 runtime 在候选切换时只记录：

- `provider_error`

不会把 provider 返回的原始错误摘要带入 warning，因此只能看到“切换发生了”，看不到“为什么切换”。

## 修正内容

### A. 配置解析改为 provider 专属优先

修改文件：

- [config.py](/D:/AI应用/langchain-agent/backend/config.py)

调整后规则：

- 主模型：优先 `DEEPSEEK_MODEL` / `BAILIAN_MODEL` / `ZHIPU_MODEL`，全局 `LLM_MODEL` 仅作兼容兜底
- fallback 模型：优先 provider 专属模型字段，全局 `LLM_FALLBACK_MODEL` 仅作兼容兜底
- base_url 同样改为 provider 专属优先

这保证了：

- provider 已确定后，不再允许全局兼容字段悄悄改写该 provider 的模型与 endpoint 组合

### B. DeepSeek 模型名做归一化

DeepSeek 模型名在配置层统一转为小写，减少大小写差异带来的候选重复和行为噪音。

### C. 候选切换日志补充原始错误摘要

修改文件：

- [model_runtime.py](/D:/AI应用/langchain-agent/backend/runtime/model_runtime.py)

调整后：

- `Switching model candidate ...`
- `Switching tool-enabled model candidate ...`
- `Switching stream model candidate ...`

都会附带压缩后的 provider 原始错误摘要，便于判断到底是：

- tool schema 不兼容
- 模型名非法
- provider 临时故障
- 认证/参数问题

### D. 当前运行配置同步收敛

修改文件：

- [backend/.env](/D:/AI应用/langchain-agent/backend/.env)

调整为：

- 主链优先 DeepSeek
- `LLM_MODEL=deepseek-v4-flash`
- `DEEPSEEK_MODEL=deepseek-v4-flash`
- `LLM_FALLBACK_PROVIDER=bailian`
- `LLM_FALLBACK_MODEL=qwen3.5-plus`
- `BAILIAN_MODEL=qwen3.5-plus`

## 验证结果

通过的新增针对性回归：

- provider 专属主模型优先于全局模型
- provider 专属 fallback 模型优先于全局 fallback 模型
- 候选切换 warning 会输出 provider 原始错误摘要

命令：

```powershell
python -m pytest backend/tests/config_runtime_regression.py::test_provider_specific_llm_model_takes_precedence_over_global_model backend/tests/config_runtime_regression.py::test_provider_specific_fallback_model_takes_precedence_over_global_fallback backend/tests/model_runtime_regression.py::test_model_runtime_logs_provider_detail_when_switching_tool_candidate -q
```

结果：

- `3 passed`

## 额外发现的旧结构问题

这次验证过程中还暴露了两个旧问题，但不属于本次主修目标：

### 1. `runtime.model_runtime` 导入链过重

`model_runtime.py` 原先为了类型注解直接导入 `agents.models`，会触发 `agents/__init__.py`，进一步拉起 tools/pdf/memory/understanding/orchestration 全链。

已做轻量修正：

- 改为 `TYPE_CHECKING` 下导入类型，避免 model runtime 在测试和轻载场景下触发整套 agent 初始化。

### 2. 本机 pytest 临时目录权限异常

完整 pytest 运行结束时，`tmp_path` / `basetemp` 清理阶段会触发 Windows 权限错误，导致部分完整测试摘要被 session finish 打断。

这属于本地测试环境/目录权限问题，不是本次 DeepSeek 候选切换修复引入的问题。

## 结论

本次修正后，系统仍然保持“DeepSeek 优先”，但已经从结构上避免了：

- 全局配置误伤 DeepSeek 主链
- fallback provider/model 错配混入候选链
- 切换发生时只有黑盒 `provider_error`、无法追根因

后续如果再次出现 DeepSeek 候选切换，日志将足以区分：

- 配置错误
- provider 暂时故障
- tool-call 协议兼容性问题

# 任务系统规范化重构计划书

日期：2026-05-02

## 一、问题定义

当前系统已经有任务相关代码，但还不是一个可装配、可执行、可观测的任务系统。

现在的结构更接近四条并行链路：

- `backend/tasks/definitions.py`：把用户请求归到 `request_intake`、`knowledge_retrieval`、`capability_execution`、`task_execution` 等任务模式。
- `backend/tasks/contract_builder.py`：把任务定义、技能、operation requirement、projection requirement 装配成运行时 prompt contract。
- `backend/tasks/flow_registry.py`：为健康系统提供 flow / agent binding / topology 草案。
- `backend/tasks/coordinator.py`：旧式 query/tool/execution task 记录器，包含启发式绑定和结果引用。

这些结构各自有价值，但没有统一的任务实例、任务模板、步骤蓝图、生命周期和结果账本。因此系统在面对“开发一个贪吃蛇小游戏”这类显式负责的任务时，会出现几个问题：

- 任务理解容易退回知识查询，而不是进入开发/执行任务。
- 文件写入、执行验证、worker 委派没有统一任务状态机。
- 主 agent 能生成代码，但不能稳定证明文件已创建、验证已完成。
- 前端无法装配“任务模板”，只能间接配置工具、技能或 agent。
- 健康任务、RAG、PDF、结构化数据、轻量开发任务无法用同一套模型描述。

本计划的目标是把任务系统从“运行时分类附属物”重构为“操作系统中的可装配任务层”。

## 二、设计原则约束

本计划只采用新的设计原则，不沿用旧的 preview / denied prompt / prompt 权限说明链路。

### 2.1 状态分层

来自 `docs/设计原则/03-状态管理.md`：

- 全局状态、应用状态、运行时上下文不能混用。
- 越底层状态越克制。
- 运行时上下文服务当前任务，不应污染长期记忆。

落地约束：

- `TaskTemplate` 是配置态，生命周期长。
- `TaskSpec` 是当前任务实例，生命周期等于一次任务。
- `TaskRun` 是执行态，生命周期等于一次运行。
- `TaskResult` 是可恢复结果，生命周期由 session / artifact 策略决定。

### 2.2 对话循环

来自 `docs/设计原则/05-对话循环.md`：

- 对话循环是状态机，不是一次性问答。
- 工具调用后必须把结果回传并形成最终收口。
- 每个 continue / terminal 分支都必须有明确原因。

落地约束：

- 多步骤任务必须有明确步骤状态。
- direct tool、worker、model response 最终必须收敛到同一 `TaskRun` 结果账本。
- 空回答、预算耗尽、工具不可用都必须成为可解释 terminal state。

### 2.3 任务系统

来自 `docs/设计原则/14-任务系统.md`：

- 任务系统负责生命周期、隔离、终止、输出收集。
- 任务失败不应污染其它任务。
- 后台 worker / shell / agent 都应由统一任务框架管理。

落地约束：

- 任务系统不是“prompt 分类器”。
- 任务系统必须拥有 task id、status、step state、output refs、kill / cancel / retry 语义。
- worker、tool、agent 都是执行器，不是任务本身。

### 2.4 工具与权限

来自 `docs/设计原则/09-工具系统设计.md` 与 `16-权限系统.md`：

- 工具是能力微服务，带输入、输出、安全和运行时协议。
- 权限由运行时系统过滤，不通过 prompt 暴露 denied 列表。
- fail-closed，非只读能力必须被明确授权。

落地约束：

- `TaskTemplate` 只声明需要的 operation / capability，不直接绕过权限。
- 运行时根据 agent profile、permission mode、tool registry 过滤可用工具。
- agent 只能看到本轮授权后的可用工具。

### 2.5 Agent 与 worker 隔离

来自 `docs/设计原则/12-Agent-系统.md`：

- 主 agent 负责理解、协调、最终收口。
- 子 agent / worker 拥有隔离上下文和专门工具集。
- 子任务只返回结构化结果，不污染主上下文。

落地约束：

- 主 agent 可以负责读写文件和整合结果。
- terminal / python / build / browser verify 这类执行能力优先交给 local worker。
- worker 输出进入 `TaskStepResult`，再由主 agent 或输出边界收口。

## 三、现有代码评估

### 3.1 可复用资产

- `TaskDefinition`：可保留为低层任务模式或模板分类。
- `TaskContract`：可扩展为 `TaskSpec` 的运行时契约。
- `TaskPromptContract`：继续负责模型可见表达，但不做权限说明。
- `RuntimeDirective`：已经具备 executor、operation refs、resource policy ref，是执行层契约核心。
- `TaskContextRef` / `TaskResultRef` / `TaskSummary`：可升级为任务结果账本中的引用模型。
- `TaskFlowRegistry`：可从 health 专用 registry 扩展成通用 template / flow registry。
- `AgentRegistry` / `ToolDefinition` / `OperationRegistry`：可作为任务装配时的能力验证来源。

### 3.2 需要收敛的旧结构

- `TaskCoordinator` 当前混合了任务记录、启发式 query 解析、结果存储和 worker 概念，应逐步降级为兼容适配层。
- `select_task_definitions()` 的关键词启发式只能作为 fallback，不应成为任务系统主入口。
- `flow_registry.py` 只内置 health flows，任务模板不可配置，前端无法装配。
- `TaskFlowDefinition` 缺少步骤、输入槽位、能力需求、验证规则，不足以表达开发类任务。

### 3.3 当前缺失

- 通用 `TaskTemplate`
- 通用 `TaskSpec`
- 通用 `TaskStepBlueprint`
- 通用 `TaskRun`
- 通用 `TaskStepResult`
- 任务模板与 agent / skill / operation 的验证矩阵
- 任务运行状态 API
- 前端可编辑的任务模板 manifest
- 轻量开发任务模板，例如 `template.dev.light_web_game`

## 四、目标架构

目标结构分为五层。

```text
Frontend Task Assembly
  -> TaskTemplate / StepBlueprint / ValidationRule
Decision Plane
  -> TaskSpec / CurrentTurnContext / Template Selection
Execution Plane
  -> TaskRun / TaskStepRun / RuntimeDirective
Executor Plane
  -> model / tool / worker / agent
Result Plane
  -> TaskResult / ArtifactRef / Session Binding / Final Answer
```

### 4.1 TaskTemplate：前端可装配的任务模板

`TaskTemplate` 描述一类任务，不描述某一次运行。

建议字段：

```python
class TaskTemplate:
    template_id: str
    title: str
    description: str
    task_family: str
    task_mode: str
    input_schema: dict
    output_schema: dict
    default_agent_id: str
    allowed_agent_ids: tuple[str, ...]
    required_capability_tags: tuple[str, ...]
    required_operations: tuple[str, ...]
    optional_operations: tuple[str, ...]
    step_blueprints: tuple[TaskStepBlueprint, ...]
    validation_rules: tuple[TaskValidationRule, ...]
    ui_manifest: dict
    enabled: bool
```

示例模板：

- `template.rag.knowledge_answer`
- `template.pdf.document_analysis`
- `template.data.structured_analysis`
- `template.dev.light_web_game`
- `template.dev.workspace_patch`
- `template.health.issue_triage`

### 4.2 TaskSpec：当前任务实例

`TaskSpec` 是用户本轮请求被绑定后的任务实例。

建议字段：

```python
class TaskSpec:
    task_id: str
    template_id: str
    session_id: str
    user_goal: str
    inputs: dict
    bindings: dict
    constraints: dict
    current_turn_context_ref: str
    requested_outputs: tuple[str, ...]
    selected_agent_id: str
    selected_skill_ids: tuple[str, ...]
    operation_requirement_ref: str
```

`TaskSpec` 由系统 resolver 产出，不直接由 prompt 生成。

### 4.3 TaskStepBlueprint：任务步骤蓝图

每个模板由一组步骤组成。

建议字段：

```python
class TaskStepBlueprint:
    step_id: str
    title: str
    step_kind: str  # understand/read/write/analyze/delegate/verify/finalize
    executor_type: str  # model/tool/worker/agent
    required_operations: tuple[str, ...]
    optional_operations: tuple[str, ...]
    input_refs: tuple[str, ...]
    output_contract_id: str
    stop_policy: str
    retry_policy: dict
```

开发小游戏模板示例：

```text
template.dev.light_web_game
  step.inspect_workspace      model/tool    op.search_files, op.read_file
  step.create_game_file       model/tool    op.write_file
  step.verify_artifact        worker        op.shell or internal verify
  step.finalize_user_report   model         op.model_response
```

### 4.4 TaskRun：任务运行状态机

统一状态：

```text
pending -> planning -> running -> waiting_tool -> waiting_worker
        -> finalizing -> completed
        -> failed
        -> cancelled
        -> blocked
```

建议字段：

```python
class TaskRun:
    run_id: str
    task_id: str
    template_id: str
    session_id: str
    status: str
    active_step_id: str
    step_runs: tuple[TaskStepRun, ...]
    resource_policy_ref: str
    runtime_directive_refs: tuple[str, ...]
    artifact_refs: tuple[str, ...]
    result_refs: tuple[str, ...]
    final_result_ref: str
    terminal_reason: str
    diagnostics: dict
```

### 4.5 TaskResult：结果账本

结果账本是 follow-up 和最终回答的事实源。

建议字段：

```python
class TaskResult:
    result_id: str
    task_id: str
    run_id: str
    result_type: str
    summary: str
    artifact_refs: tuple[str, ...]
    object_handle_refs: tuple[str, ...]
    validation_state: str
    user_visible_claims: tuple[str, ...]
    created_at: float
```

最终回答只能基于 `TaskResult` 和已提交的 tool/worker result，不允许模型凭空声明“已创建文件”。

## 五、执行流设计

### 5.1 单任务执行流

```text
user message
  -> query understanding
  -> task template selection
  -> TaskSpec build
  -> operation requirement build
  -> resource policy adoption
  -> TaskRun create
  -> step execution loop
  -> observation aggregation
  -> TaskResult commit
  -> final answer commit
```

### 5.2 复合任务执行流

```text
user compound request
  -> bundle TaskSpec
  -> child TaskSpec[]
  -> child TaskRun[]
  -> result ledger aggregation
  -> bundle TaskResult
  -> final answer
```

要求：

- PDF、结构化数据、天气这类组合任务必须进入 bundle ledger。
- 后续“第二个子任务”“刚才那个表”必须引用 bundle item / task result handle。
- 不能让最后一个 direct tool result 覆盖整个复合任务答案。

### 5.3 开发类任务执行流

```text
开发一个贪吃蛇小游戏
  -> select template.dev.light_web_game
  -> bind output target
  -> inspect workspace
  -> create or edit file
  -> verify artifact through worker if available
  -> commit TaskResult with file refs
  -> final answer reports actual committed files
```

默认策略：

- 主 agent 可执行 `read_file/search_files/write_file/edit_file`。
- terminal/python/build/browser verify 优先通过 local worker。
- 如果 worker 不可用，任务可以完成为 `completed_with_unverified_artifact`，但最终回答必须说明“已写入，未执行运行验证”。

## 六、模块重构计划

### Phase 1：任务数据模型归一

目标：建立通用任务模型，不改变运行行为。

新增或改造文件：

- `backend/tasks/template_models.py`
- `backend/tasks/run_models.py`
- `backend/tasks/result_models.py`
- `backend/tasks/step_models.py`
- `backend/tasks/models.py`

工作内容：

- 定义 `TaskTemplate`、`TaskSpec`、`TaskStepBlueprint`、`TaskRun`、`TaskStepRun`、`TaskResult`。
- 保留现有 `TaskRecord`，但标记为 legacy-compatible record。
- 为现有 `TaskContextRef` / `TaskResultRef` 增加迁移适配函数。

完成标准：

- 新模型可以序列化。
- 不影响现有测试。
- 有最小单元测试覆盖状态转换和序列化。

### Phase 2：通用任务模板注册表

目标：把 health flow 和通用任务模板放入同一 registry。

改造文件：

- `backend/tasks/flow_models.py`
- `backend/tasks/flow_registry.py`
- `backend/tasks/definitions.py`

工作内容：

- 新增 `TaskTemplateRegistry`。
- 将现有 health flows 转换为内置 `TaskTemplate`。
- 新增基础模板：
  - `template.rag.knowledge_answer`
  - `template.pdf.document_analysis`
  - `template.data.structured_analysis`
  - `template.dev.workspace_patch`
  - `template.dev.light_web_game`
  - `template.chat.general_response`
- `TaskDefinition` 降级为 template selection 的兼容视图。

完成标准：

- `TaskFlowRegistry.build_overview()` 可以返回 templates、bindings、validation matrix。
- health 现有 overview 不断。
- 前端可以读取模板 manifest。

### Phase 3：任务选择与 TaskSpec 构建

目标：让任务理解产出模板选择，而不是只产出工具或 RAG route。

改造文件：

- `backend/understanding/task_understanding.py`
- `backend/tasks/contract_builder.py`
- `backend/tasks/contracts.py`

工作内容：

- 增加 `template_hint`、`task_intent_kind`、`execution_intent`。
- 增加开发类识别：
  - 创建文件
  - 修改文件
  - 实现功能
  - 开发小游戏
  - 运行验证
- `build_task_runtime_contract()` 改为优先接收 `TaskSpec`。
- operation requirement 从 `TaskTemplate + TaskSpec + SkillRuntimeView` 派生。

完成标准：

- “开发一个贪吃蛇小游戏”不再落入 `knowledge_lookup`。
- “直接执行 Python 改文件”能被识别为 local execution intent，但是否执行由 agent/worker policy 决定。
- RAG / PDF / data 原路径不回退。

### Phase 4：TaskRun 状态机接入 RuntimeLoop

目标：运行时不再只处理模型/工具事件，而是推进 TaskRun。

改造文件：

- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/orchestration/runtime_loop/observation_aggregator.py`
- `backend/orchestration/runtime_directive.py`
- `backend/orchestration/runtime_loop/model_adoption.py`
- `backend/orchestration/runtime_loop/tool_adoption.py`

工作内容：

- 在 runtime loop 创建 `TaskRun`。
- 每次 tool / worker / model 结果写入 `TaskStepResult`。
- budget exhausted、tool unavailable、permission blocked 必须写入 terminal reason。
- direct tool 和 bundle execution 统一走 result ledger。

完成标准：

- 不再出现空 assistant answer。
- 复合任务不再被最后一个工具结果覆盖。
- `TaskRun` 能解释每个失败或降级原因。

### Phase 5：worker 与执行类任务归位

目标：让轻量开发任务具备执行闭环。

改造文件：

- `backend/orchestration/unit_registry.py`
- `backend/operations/agent_registry.py`
- `backend/tools/definitions.py`
- `backend/tasks/flow_registry.py`
- local worker 相关模块

工作内容：

- 将 local worker 注册为任务执行器。
- 主 agent 默认不直接拿 `terminal/python_repl`。
- `template.dev.light_web_game` 的验证步骤委派给 worker。
- worker 返回结构化验证结果。

完成标准：

- 轻量开发任务可以写文件并得到验证状态。
- 如果 worker 不可用，结果状态为 `completed_unverified`，不是假装已验证。

### Phase 6：前端任务装配页

目标：前端管理任务模板，而不是直接拼 prompt。

涉及文件：

- `frontend/src/lib/api.ts`
- `frontend/src/components/workspace/views/SystemConfigView.tsx`
- 新增 Task Template 管理视图

前端可编辑：

- 模板名称、描述、启用状态。
- 输入槽位。
- 输出规则。
- 默认 agent。
- step blueprint。
- required/optional capabilities。
- validation rules。

前端不可编辑：

- 实际授权后的工具列表。
- denied prompt 文案。
- runtime resource policy 内部决策。

完成标准：

- 可以从 UI 启用/禁用任务模板。
- 可以查看模板到 agent/tool/operation 的验证结果。
- 不暴露未授权工具给 agent。

## 七、迁移与清理规则

### 7.1 保留窗口

短期保留：

- `TaskDefinition`
- `TaskRecord`
- `TaskCoordinator`

但只作为兼容层，不再增加新能力。

### 7.2 清理目标

完成新链路后清理：

- `TaskCoordinator` 中的启发式 binding。
- 仅服务旧 query/tool 记录的 task_type 字段。
- 旧式 flow 仅 health 专用的并行结构。
- 任何 preview / denied prompt / resource_section 权限说明残留。

### 7.3 禁止事项

- 不用 prompt 告诉 agent 哪些工具不可用。
- 不让 memory 存储系统契约。
- 不把 task result 当普通聊天文本恢复。
- 不为单个测试关键词补丁式增加判断。
- 不把 frontend 装配直接绑定到旧 `TaskDefinition`。

## 八、验证矩阵

### 8.1 单元测试

- `TaskTemplate` 序列化与 schema 校验。
- `TaskRun` 状态转换。
- `TaskStepResult` 聚合。
- `TaskTemplateRegistry` validation matrix。

### 8.2 回归测试

- RAG 问答仍能走 `search_knowledge`。
- PDF 分析仍能走 `pdf_analysis`。
- 结构化数据仍能走 `structured_data_analysis`。
- 天气/黄金价格仍能走 dedicated tool。
- 健康任务 overview 不断。

### 8.3 新增场景测试

- “开发一个贪吃蛇小游戏”应选择 `template.dev.light_web_game`。
- “创建 snake.html”应获得 `op.write_file`。
- “修复 frontend 某组件”应选择 `template.dev.workspace_patch`。
- “直接在终端删除临时文件”应进入 local command intent，并根据权限返回可解释 blocked / delegated / executed。
- 复合任务 PDF + 数据 + 天气应生成 bundle result ledger。

### 8.4 长情景测试

目标指标：

- 复合任务结果不被覆盖。
- follow-up 能回指 task result handle。
- 空回答为 0。
- 权限不可用时有明确 terminal reason。
- 开发类任务至少能完成单文件产物生成。

## 九、优先级建议

第一优先级：

- Phase 1 数据模型
- Phase 2 模板注册表
- Phase 3 TaskSpec 构建

原因：这三步决定前端能不能装配任务，也决定后续运行时有没有稳定输入。

第二优先级：

- Phase 4 TaskRun 接入 RuntimeLoop
- Phase 5 worker 执行闭环

原因：这两步决定任务能不能真正执行和自证。

第三优先级：

- Phase 6 前端装配页

原因：前端必须绑定稳定后端模型，不能先做 UI 再迁移结构。

## 十、最终目标

重构完成后，系统应满足：

- 用户一句自然语言可以被解析为明确任务模板和任务实例。
- 前端可以装配任务模板、步骤、agent 绑定和验证规则。
- 主 agent 只看到本轮授权后的可用能力。
- worker / tool / agent 执行结果统一进入 TaskRun 账本。
- 最终回答只基于真实执行结果。
- 轻量开发任务能够完成“写文件 -> 验证 -> 收口”的基本闭环。

对“开发一个贪吃蛇小游戏”这类任务，系统的正确目标不是只生成代码，而是形成一个可追踪任务：

```text
TaskTemplate: template.dev.light_web_game
TaskSpec: 创建 snake.html 或接入前端页面
TaskRun:
  inspect_workspace -> completed
  create_game_file -> completed
  verify_artifact -> completed / completed_unverified
TaskResult:
  artifact_refs: [snake.html]
  validation_state: verified / unverified
FinalAnswer:
  基于 TaskResult 汇报真实创建的文件和验证状态
```

这就是任务系统应该承担的“显式负责”。

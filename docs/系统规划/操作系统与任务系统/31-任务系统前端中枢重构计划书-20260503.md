## 一、核心结论

任务系统前端不应再被设计成“任务、绑定、健康运行混合展示台”，也不应临时转成“点 Agent 跳去编排系统”的目录页。

它应该是一个真正的任务组织中枢：

```text
任务系统
  -> 通用任务管理
  -> 特定任务管理
  -> Agent 管理
  -> 分配关系与连接诊断
```

其中最重要的边界是：

```text
操作系统管理 Skill / Workflow 本体
任务系统只管理 Task -> Agent -> Workflow 的分配关系
编排系统管理 Agent 的运行能力与执行准入
```

换句话说：

- `Skill` 是能力单元，归操作系统管理。
- `Workflow` 是任务做法，归操作系统管理。
- `Task` 是业务入口和分配关系，归任务系统管理。
- `Agent` 在任务系统中有组织身份和任务承载关系，但深层运行权限不在任务系统编辑。

## 二、目标定位

### 2.1 任务系统负责什么

任务系统前端负责回答这些问题：

- 系统里有哪些 Agent。
- Agent 被分成哪些类型。
- 通用任务默认如何由主 Agent 接收和完成。
- 特定任务如何绑定 Workflow。
- 特定任务默认由哪个 Agent 承接。
- 特定任务是否需要子 Agent 或系统管理 Agent 参与。
- 某个 Agent 承载了哪些任务。
- 某个任务当前的 Agent / Workflow / Projection / Contract 连接是否成立。

### 2.2 任务系统不负责什么

任务系统前端不负责：

- 编辑 Skill 文件。
- 编辑 Workflow 文件。
- 编辑 Workflow 的步骤和 Skill 组成。
- 编辑 capability profile。
- 编辑 runtime lane。
- 编辑 resource policy。
- 编辑 operation 权限。
- 直接启动健康系统领域动作。
- 把健康 run、health issue、health report 当成任务系统主对象。

这些对象分别归属：

- `Skill / Workflow`：操作系统。
- `Capability / Runtime Lane / Resource Policy / Admission`：编排系统。
- `Health Issue / Health Run / Health Report`：健康系统。

## 三、术语与关系

### 3.1 Skill

`Skill` 是能力单元，回答“系统会什么”。

它包含：

- 能力说明。
- 使用场景。
- 提示片段。
- 允许工具。
- 输入输出边界。
- 能力标签。

Skill 本体由操作系统管理。

### 3.2 Workflow

`Workflow` 是任务执行流程，回答“这类任务应该怎么做”。

它包含：

- workflow id。
- workflow 文件引用。
- 所属 task mode。
- 包含哪些 skills。
- 执行步骤。
- 输入边界。
- 输出边界。
- 停止条件。
- 证据要求。

Workflow 本体由操作系统管理。任务系统只能选择和引用 Workflow，不能编辑 Workflow 内容。

### 3.3 Task

`Task` 是业务入口，回答“用户或系统现在要做什么”。

任务系统需要把 Task 分为两类：

```text
通用任务
  用户通过对话提出，由主 Agent 直接理解、规划、完成。

特定任务
  系统中明确登记的任务类型，默认主 Agent 承接，也可分配给子 Agent 或系统管理 Agent。
```

### 3.4 Agent

Agent 是任务承载者，任务系统中至少分为三类：

- `main_agent`：主 Agent，负责对话入口、通用任务处理、任务分派和最终整合。
- `system_management_agent`：系统管理 Agent，负责某个系统域的治理、诊断、维护类任务。
- `worker_sub_agent`：工作子 Agent，负责具体执行、检索、加工、生成等任务。

Agent 管理在任务系统内保留，采用卡片形式展示。点击 Agent 卡片后，在任务系统中进入 Agent 编辑视图，编辑它的组织身份和任务承载关系。

## 四、目标信息架构

### 4.1 一级结构

任务系统前端重构为四个一级页面或切换区：

1. `通用任务`
2. `特定任务`
3. `Agent 管理`
4. `连接诊断`

### 4.2 通用任务

通用任务表示用户通过对话交给主 Agent 的任务。

它的前端管理目标是：

- 配置主 Agent 的默认通用任务入口。
- 配置通用任务默认 Workflow 引用。
- 展示通用任务如何进入主 Agent。
- 展示通用任务的输入契约和输出契约。
- 展示通用任务是否具备可运行的任务连接。

通用任务默认约束：

```text
default_agent_id = agent:main
participant_agent_ids = []
task_scope = general_conversation_task
workflow_id = 从操作系统 Workflow 目录中选择
```

通用任务不是每次用户对话的运行记录，而是“用户对话任务如何被主 Agent 接住”的配置入口。

### 4.3 特定任务

特定任务表示系统中可登记、可复用、可分配的任务类型。

它的前端管理目标是：

- 创建特定任务。
- 选择该任务引用的 Workflow。
- 指定默认承接 Agent。
- 配置参与 Agent。
- 配置输入/输出契约。
- 配置 Projection 引用。
- 配置任务结构。
- 查看连接是否有效。

特定任务的推荐字段：

```text
task_id
task_title
task_family
task_mode
task_kind = specific_task
flow_id
default_agent_id
participant_agent_ids
workflow_id
workflow_file_ref
projection_template_id
input_contract_id
output_contract_id
task_structure
enabled
metadata
```

其中：

- `workflow_id` 是任务系统引用操作系统 Workflow 的主字段。
- `workflow_file_ref` 只读展示，不在任务系统编辑。
- `participant_agent_ids` 用于配置是否有子 Agent 或系统管理 Agent 参与。
- `task_structure` 只表达任务组织结构，不表达运行权限。

### 4.4 Agent 管理

Agent 管理保持在任务系统内。

前端采用卡片形式展示：

- 主 Agent 卡片组。
- 系统管理 Agent 卡片组。
- 工作子 Agent 卡片组。

每张卡片展示：

- Agent 名称。
- Agent ID。
- Agent 分类。
- 所属系统。
- 生命周期状态。
- 默认 Projection。
- 承载任务数量。
- 绑定 Workflow 数量。
- 当前连接状态。

点击 Agent 卡片进入 Agent 编辑视图。

Agent 编辑视图分为：

- `基础身份`
- `任务承载`
- `Workflow 引用`
- `连接诊断`

允许编辑：

- display name。
- owner system。
- agent 分类。
- lifecycle。
- default projection。
- 承载哪些任务。

不允许编辑：

- capability profile。
- allowed operations。
- blocked operations。
- runtime lane 权限。
- resource policy。

这些字段只可作为只读摘要或跳转提示展示。

### 4.5 连接诊断

连接诊断不是运行报告，而是配置结构检查。

需要聚合：

- 无 Workflow 的任务。
- Workflow 引用不存在的任务。
- 无默认 Agent 的任务。
- 默认 Agent 不存在的任务。
- 参与 Agent 不存在的任务。
- Agent 没有承载任何任务。
- binding invalid。
- profile invalid。
- Projection 引用缺失。
- input/output contract 缺失。
- 任务连接已成立但编排授权待确认。

诊断面板必须明确区分：

```text
任务系统判断：分配关系是否成立
编排系统判断：运行时是否准入
```

## 五、页面交互设计

### 5.1 总览区

页面顶部展示中枢状态：

- Agent 总数。
- 通用任务数量。
- 特定任务数量。
- Workflow 引用数量。
- 有效连接数量。
- 阻断连接数量。

这些指标服务管理判断，不展示 health run。

### 5.2 通用任务页

通用任务页采用“主 Agent 工作入口”布局：

- 左侧：通用任务卡片。
- 中间：通用任务配置。
- 右侧：Workflow 只读摘要和连接诊断。

核心动作：

- 选择默认 Workflow。
- 保存通用任务入口配置。
- 查看 Workflow 文件引用。
- 查看该 Workflow 包含的 Skills 摘要。

### 5.3 特定任务页

特定任务页采用“任务卡片列表 + 任务编辑器”布局：

- 左侧：特定任务卡片列表。
- 中间：任务定义表单。
- 右侧：承接关系和 Workflow 摘要。

核心动作：

- 新建特定任务。
- 编辑任务标题、family、mode。
- 选择 Workflow。
- 选择默认 Agent。
- 选择参与 Agent。
- 配置输入/输出契约。
- 保存任务分配。

### 5.4 Agent 管理页

Agent 管理页采用模块化卡片布局：

```text
主 Agent
  [agent card]

系统管理 Agent
  [agent card] [agent card]

工作子 Agent
  [agent card] [agent card] [agent card]
```

点击卡片后进入 Agent 编辑：

```text
Agent 详情
  -> 基础身份
  -> 承载任务
  -> 引用 Workflow
  -> 连接诊断
```

Agent 编辑页要回答：

- 这个 Agent 属于哪类。
- 它负责哪些任务。
- 它承载哪些 Workflow 引用。
- 它是否有无效任务连接。

### 5.5 Workflow 摘要展示

任务系统中只能只读展示 Workflow。

展示字段：

- workflow id。
- title。
- task mode。
- workflow 文件路径。
- visible skills。
- steps。
- input boundary。
- output boundary。
- stop conditions。

任务系统页面可以提供“去操作系统编辑 Workflow”的入口，但不能在本页内编辑 Workflow 本体。

## 六、数据模型调整

### 6.1 Task Assignment

建议新增或明确任务分配模型：

```python
class TaskAssignment:
    task_id: str
    task_title: str
    task_kind: Literal["general_task", "specific_task"]
    task_family: str
    task_mode: str
    flow_id: str
    default_agent_id: str
    participant_agent_ids: tuple[str, ...]
    workflow_id: str
    projection_template_id: str
    input_contract_id: str
    output_contract_id: str
    task_structure: dict[str, Any]
    enabled: bool
    metadata: dict[str, Any]
```

### 6.2 General Task Profile

通用任务建议有单独配置模型：

```python
class GeneralTaskProfile:
    profile_id: str
    title: str
    default_agent_id: str
    default_workflow_id: str
    input_contract_id: str
    output_contract_id: str
    conversation_entry_policy: str
    enabled: bool
    metadata: dict[str, Any]
```

默认：

```text
profile_id = general.conversation.default
default_agent_id = agent:main
```

### 6.3 Agent Task Carrying Profile

Agent 管理视图需要一个聚合投影：

```python
class AgentTaskCarryingProfile:
    agent_id: str
    display_name: str
    profile_type: str
    owner_system: str
    lifecycle_state: str
    carried_general_task_refs: tuple[str, ...]
    carried_specific_task_refs: tuple[str, ...]
    workflow_refs: tuple[str, ...]
    projection_template_refs: tuple[str, ...]
    validation_state: str
    blocked_reasons: tuple[str, ...]
```

它可以复用 `AgentTaskConnectionProfile` 的底层诊断能力，但前端主语义不再只围绕 connection profile 展开。

## 七、API 规划

### 7.1 读取接口

建议任务系统提供：

```text
GET /tasks/overview
GET /tasks/general-profiles
GET /tasks/specific-assignments
GET /tasks/assignments/{task_id}
GET /tasks/agent-carrying-profiles
GET /tasks/agent-carrying-profiles/{agent_id}
GET /tasks/connection-diagnostics
```

继续保留：

```text
GET /tasks/agent-task-connections
GET /tasks/agent-task-connections/{agent_id}
```

但它们作为诊断和底层连接视图，不再是前端唯一主入口。

### 7.2 写入接口

建议任务系统提供：

```text
PUT /tasks/general-profiles/{profile_id}
PUT /tasks/specific-assignments/{task_id}
PUT /tasks/agents/{agent_id}
```

写入边界：

- 允许保存任务分配。
- 允许保存 Agent 组织身份。
- 允许保存 Task -> Workflow 引用。
- 不允许保存 Workflow 本体。
- 不允许保存 Skill 本体。
- 不允许保存 capability profile。

### 7.3 操作系统接口引用

任务系统需要读取操作系统提供的 Workflow 目录：

```text
GET /skills/workflows
GET /skills/workflows/{workflow_id}
```

任务系统只读使用这些接口。

## 八、前端重构方案

### Phase 31A：信息架构重建

- `TaskSystemView` 改成四个主视图：
  - 通用任务
  - 特定任务
  - Agent 管理
  - 连接诊断
- 移除旧的 health run 视角。
- 移除旧的“任务/绑定/健康运行混合展示”结构。
- 保留 Workflow 只读引用，不提供 Workflow 编辑表单。

### Phase 31B：通用任务管理

- 增加通用任务入口卡。
- 默认绑定 `agent:main`。
- 可选择默认 Workflow。
- 展示 Workflow 摘要。
- 展示通用任务连接诊断。

### Phase 31C：特定任务管理

- 增加特定任务卡片列表。
- 支持新建特定任务。
- 支持选择 Workflow。
- 支持选择默认 Agent。
- 支持选择参与 Agent。
- 支持配置输入/输出契约。
- 支持保存 Task Assignment。
- 展示特定任务引用的 Workflow / Skills 摘要。

### Phase 31D：Agent 卡片管理

- 按三类 Agent 分组展示卡片。
- 点击卡片进入 Agent 编辑。
- Agent 编辑不混入任务表单。
- Agent 详情展示它承载的通用任务、特定任务、Workflow 引用和连接状态。

### Phase 31E：诊断聚合

- 聚合任务分配错误。
- 聚合 Agent 承载错误。
- 聚合 Workflow 引用错误。
- 聚合 Projection / Contract 缺失。
- 明确显示“任务连接状态”和“编排授权状态”的区别。

## 九、后端重构方案

### Phase 31F：任务分配存储

新增或整理持久化文件：

```text
storage/tasks/general_task_profiles.json
storage/tasks/task_assignments.json
```

避免继续把所有任务语义都压进旧 `task_flows.json`。

### Phase 31G：旧数据迁移

将现有 `TaskFlowDefinition` 映射为特定任务：

```text
flow.health.issue_triage -> specific task
flow.health.trace_analysis -> specific task
flow.health.case_draft -> specific task
flow.health.fix_verification -> specific task
```

旧 flow 可以继续作为底层 flow id 使用，但前端语义应展示为“特定任务分配”，而不是散装 flow 表。

### Phase 31H：Agent 分类清理

正式分类只保留：

```text
main_agent
system_management_agent
worker_sub_agent
```

清理旧残留：

- `primary`
- `sub_agent`
- `visible_sub_agent_count`
- 其它旧的主/子 Agent 统计字段

如果遇到旧数据，只做一次性迁移，不在 UI 和正式模型中继续暴露旧枚举。

## 十、验收标准

### 10.1 页面验收

- 页面有清晰的四个主入口：通用任务、特定任务、Agent 管理、连接诊断。
- Agent 以卡片形式展示。
- 点击 Agent 卡片进入 Agent 编辑。
- 通用任务默认主 Agent。
- 特定任务默认主 Agent，但可以配置子 Agent 或系统管理 Agent 参与。
- 每个任务都能选择 Workflow 引用。
- Workflow 只读展示，不在任务系统编辑。
- 页面不再展示 health run 作为主对象。
- 页面不再提供健康系统领域动作按钮。

### 10.2 数据验收

- 任务系统能保存通用任务配置。
- 任务系统能保存特定任务分配。
- 任务系统能保存 Agent 组织身份。
- Task -> Agent -> Workflow 分配关系可查询。
- Agent -> Task -> Workflow 承载关系可查询。
- 无效引用能进入连接诊断。

### 10.3 边界验收

- 任务系统不编辑 Skill 本体。
- 任务系统不编辑 Workflow 本体。
- 任务系统不编辑 capability profile。
- 任务系统不编辑 runtime lane。
- 任务系统不编辑 resource policy。
- 编排授权状态只读展示，不被任务系统伪造为最终权限。

## 十一、风险与控制

### 11.1 风险：任务系统再次变成混合大杂烩

控制：

- 前端一级入口固定为通用任务、特定任务、Agent 管理、连接诊断。
- 健康系统对象不进入任务系统主视图。
- Workflow 只读引用，不展开编辑。

### 11.2 风险：Workflow 与 Skill 被任务系统接管

控制：

- Workflow 详情只能展示摘要。
- Workflow 编辑入口跳转或提示去操作系统。
- 任务系统保存 payload 中只允许 `workflow_id`，不允许提交 workflow steps 或 skill body。

### 11.3 风险：Agent 管理与编排权限混层

控制：

- Agent 管理只编辑组织身份和任务承载。
- capability profile 只读展示。
- runtime lane / resource policy 归编排系统。

### 11.4 风险：旧残留继续污染新模型

控制：

- 删除无用旧字段。
- 旧枚举只做迁移，不做兼容 UI。
- 前端文案和模型只使用 `main_agent / system_management_agent / worker_sub_agent`。

## 十二、最终交付物

- 重构后的 `TaskSystemView.tsx`。
- 任务系统相关 CSS。
- `api.ts` 中的任务分配类型。
- 后端任务分配 registry。
- 后端任务分配 API。
- 旧 flow 到特定任务的迁移逻辑。
- 任务系统前端构建验证。
- 任务系统连接诊断验证。

最终完成后，任务系统前端应当能真正完成三件事：

```text
管理 Agent
管理通用任务和特定任务
管理 Task -> Agent -> Workflow 的分配关系
```

而不是只展示散乱表格，也不是越界管理 Skill、Workflow 或编排权限。

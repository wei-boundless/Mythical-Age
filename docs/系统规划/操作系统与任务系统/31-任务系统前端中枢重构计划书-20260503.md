# 任务系统前后端重构总计划书

日期：2026-05-03

## 一、文档定位

本文件不是讨论稿，而是任务系统重构的执行蓝图。

它用于统一回答以下问题：

- 任务系统到底负责什么
- Agent 在任务系统中的地位是什么
- workflow 和 projection 应该归谁管理
- 前端应该如何组织管理界面
- 后端应该如何组织注册、绑定、校验和运行契约
- 现有代码哪些必须清理，哪些可以保留

后续任务系统前后端重构，统一以本文件为准。

## 二、重构目标

本次重构的目标不是继续修补旧任务页，而是把任务系统重建为真正可管理、可登记、可绑定、可执行、可扩展的中枢系统。

重构后的任务系统必须满足：

1. 能清楚管理 Agent 名册与类别
2. 能清楚管理通用任务与特定任务
3. 能在任务侧完成 workflow 绑定与 projection 绑定
4. 能输出稳定的任务执行契约，供运行时和 Agent 消费
5. 能在未来扩展到多 Agent 协作，而不破坏单 Agent 主链路
6. 不保留无意义旧残留，不再为了兼容继续维持错误边界

## 三、基础理解与硬边界

### 3.1 系统不是 Agent

系统是能力域、资源域、治理域。

Agent 是任务执行者。

系统不直接执行任务，Agent 才执行任务。

### 3.2 主 Agent 和子 Agent 没有本体差异

它们都是 Agent。

差异只在于：

- 对接入口不同
- 接收任务不同
- 输出目标不同
- 内部运行设定不同

### 3.3 任务系统和权限系统的边界

任务系统回答的是：

`谁做什么`

权限系统回答的是：

`这个 Agent 怎么做`

因此：

- 任务系统负责 Agent 管理、任务管理、workflow 绑定、projection 绑定、协作组织
- 权限系统负责 Agent 内部 prompts 边界、工具边界、资源边界、上下文边界、输出边界

### 3.4 workflow 和 projection 必须归任务侧

workflow 是任务做法。

projection 是任务执行时附加给 Agent 的任务向补充。

因此二者必须由任务系统管理，不能继续放在权限系统中。

### 3.5 任务系统不是自由聊天壳

通用任务可以允许模型承担较多理解。

但特定任务不能只靠模型自由理解，必须先由系统完成：

- 任务登记
- Agent 选择
- workflow 绑定
- projection 绑定
- 输入输出契约定义

然后再交给 Agent 执行。

## 四、任务系统的目标对象模型

本次重构后，任务系统前后端统一围绕五类核心对象组织。

### 4.1 AgentRecord

表示任务执行者的注册记录。

它至少应包含：

- `agent_id`
- `agent_name`
- `agent_category`
- `interface_target`
- `description`
- `enabled`
- `builtin`
- `editable`
- `soul_id`
- `default_projection_id`
- `task_scope`
- `metadata`

其中：

- `agent_id` 是唯一主标识，不再引入第二套混乱命名
- `agent_category` 只允许三类：
  - `main_agent`
  - `system_management_agent`
  - `worker_sub_agent`
- `interface_target` 用于表达这个 Agent 面向哪个入口
  - 主会话
  - 某系统悬浮会话窗
  - 某工作入口

### 4.2 GeneralTaskProfile

表示通用任务入口配置。

它描述的是：

- 用户从哪个对话入口进入
- 默认由哪个 Agent 承接
- 默认允许什么任务模式
- 是否允许模型自由理解
- 是否允许进一步分流到特定任务

它服务于主会话和未来其他标准对话入口。

### 4.3 SpecificTaskRecord

表示特定任务的登记记录。

它至少应包含：

- `task_id`
- `task_name`
- `task_domain`
- `description`
- `enabled`
- `default_agent_id`
- `workflow_id`
- `projection_id`
- `input_contract`
- `output_contract`
- `trigger_signals`
- `notes`

它表达的是：

`当某类任务被识别或触发时，由哪个 Agent，按什么 workflow，并带着什么 projection 去执行`

### 4.4 TaskWorkflowRecord

表示任务系统内部维护的 workflow。

workflow 脱离任务没有管理意义，但作为任务资源，仍需要在任务系统内部可编辑、可校验、可引用。

因此它不是独立一级管理域，而是任务管理域中的子资源。

它至少应包含：

- `workflow_id`
- `workflow_name`
- `task_mode`
- `description`
- `steps`
- `visible_skill_ids`
- `stop_conditions`
- `required_evidence_refs`
- `output_boundary`
- `enabled`

### 4.5 CoordinationTemplate

表示多 Agent 协调结构模板。

当前阶段先建立数据结构和管理入口，不要求先把多 Agent runtime 做完。

它至少应包含：

- `template_id`
- `template_name`
- `description`
- `coordinator_agent_id`
- `participant_agent_ids`
- `nodes`
- `edges`
- `handoff_rules`
- `enabled`

## 五、任务消费主链路

这是本次重构最核心的内容。

特定任务要稳定被 Agent 消费，不是靠提示词越写越长，而是靠结构化任务消费链路。

### 5.1 通用任务链路

```text
用户进入主会话
  -> 任务系统识别为通用任务
  -> 选择 GeneralTaskProfile
  -> 默认分配主 Agent
  -> 生成 TaskIntentContract
  -> 匹配 TaskTemplate
  -> 生成 TaskSpec
  -> 如有需要绑定 workflow / projection
  -> 输出 TaskPromptContract
  -> 交给运行时执行
```

这里允许模型承担更多理解。

### 5.2 特定任务链路

```text
用户请求 / 系统信号进入
  -> 任务系统识别为特定任务
  -> 命中 SpecificTaskRecord
  -> 读取 default_agent_id
  -> 读取 workflow_id
  -> 读取 projection_id
  -> 组合输入输出契约
  -> 生成 TaskIntentContract
  -> 生成 TaskSpec
  -> 生成 TaskPromptContract
  -> 交给运行时执行
```

这里不依赖模型自由猜测任务做法，而是先由任务系统定形，再由 Agent 执行。

### 5.3 prompts 装配顺序

本项目必须锁定以下顺序：

1. 先装配 `task`
2. 再装配 `workflow`
3. 最后装配 `projection`

不能颠倒。

因为：

- task 先定义目标
- workflow 再定义做法
- projection 最后补充执行姿态

### 5.4 稳定消费的真实抓手

Agent 能稳定消费特定任务，依赖的是以下对象都先被显式化：

- `TaskIntentContract`
- `TaskSpec`
- `TaskPromptContract`
- `selected_agent_id`
- `workflow_id`
- `projection_id`
- `input_contract`
- `output_contract`

也就是说：

稳定性来自契约，不来自自由发挥。

## 六、前端信息架构

任务系统前端重构为三个一级区：

1. `Agent 管理`
2. `任务管理`
3. `编排管理`

不再放入无意义总览、检测面板、诊断墙作为主结构。

这些可以是后续辅助视图，但不能占据任务系统主导航。

### 6.1 Agent 管理

Agent 管理采用两级结构。

第一层先展示三类 Agent：

```text
Agent 管理
  -> 主 Agent
  -> 系统管理 Agent
  -> 工作子 Agent
```

点击类别，再进入该类别的列表页。

#### 主 Agent

- 类别入口可点击
- 列表页可查看
- 详情页只读
- 不提供编辑

#### 系统管理 Agent

- 类别入口可点击
- 列表页可查看
- 详情页只读
- 不提供编辑

#### 工作子 Agent

- 类别入口可点击
- 列表页可查看
- 详情页可编辑
- 允许新增

### 6.2 Agent 详情页展示内容

Agent 详情页应展示：

- Agent 基本身份
- 类别
- 对接入口
- 灵魂归属
- 默认 projection
- 承接任务列表
- 相关 workflow 引用
- 任务覆盖范围

但不允许在任务系统中直接编辑：

- runtime 权限边界
- 工具白名单
- 资源访问策略
- prompts 装配底层规则

### 6.3 任务管理

任务管理拆成两个标签区：

1. `通用任务`
2. `特定任务`

#### 通用任务页

用于管理主会话等标准入口如何接入任务系统。

核心字段包括：

- profile 名称
- 对接入口
- 默认 Agent
- 默认 task mode
- 默认 workflow
- 默认 projection
- 是否允许自由理解
- 是否允许分流到特定任务

#### 特定任务页

这是本次重构的主战场。

每个特定任务卡片或列表项必须直接表达：

- 这是哪个任务
- 默认谁执行
- 绑定了哪个 workflow
- 绑定了哪个 projection
- 输入输出契约是什么
- 当前是否启用

点进详情页后可以编辑：

- 任务名称与说明
- default_agent_id
- workflow_id
- projection_id
- trigger_signals
- input_contract
- output_contract
- 备注

### 6.4 workflow 在前端中的位置

workflow 仍在任务系统中管理，但不作为与任务平级的一级导航。

正确做法是：

- 在任务管理域内部提供 workflow 资源管理面板
- workflow 为任务服务
- workflow 的主要入口应来自任务详情页的绑定与跳转

换句话说：

`workflow 是任务资源，不是独立中心`

### 6.5 编排管理

编排管理用于承载未来多 Agent 协同。

当前阶段前端先实现：

- 协调任务列表
- 拓扑模板列表
- 节点与边的结构查看
- coordinator / participant 管理

当前阶段不要求在这里混入权限边界编辑。

## 七、后端重构目标

后端要做的不是加几个接口，而是把任务系统恢复成统一的数据拥有者。

### 7.1 后端职责

任务系统后端必须统一负责：

- Agent 注册
- 通用任务配置
- 特定任务登记
- workflow 存储
- projection 引用校验
- 任务绑定校验
- coordination template 存储
- runtime contract 组装

### 7.2 后端必须形成的服务边界

建议收束为以下几组能力：

1. `AgentRegistry`
   - 管理 AgentRecord

2. `TaskProfileRegistry`
   - 管理 GeneralTaskProfile 和 SpecificTaskRecord

3. `TaskWorkflowRegistry`
   - 管理 workflow 资源

4. `TaskCoordinationRegistry`
   - 管理协调任务与拓扑模板

5. `TaskContractBuilder`
   - 负责从任务记录生成运行契约

### 7.3 运行时接口边界

运行时不应该再决定：

- 任务归属
- workflow 归属
- projection 归属

运行时应该消费的是任务系统已经装配好的结果。

即：

```text
Task system
  -> 生成 TaskSpec / TaskPromptContract / ProjectionRequirement
  -> Runtime consume
```

而不是：

```text
Runtime 自己再去决定任务到底该怎么绑定
```

## 八、现有代码中的结构性问题

根据当前代码，以下偏差必须正面清理。

### 8.1 AgentRegistry 语义不对

当前 `backend/operations/agent_registry.py` 及相关接口仍混有以下旧字段：

- `owner_system`
- `profile_type`
- `lifecycle_state`

这些字段会把 Agent 混成系统归属对象或能力配置对象，不适合作为新任务系统主模型。

本次重构中需要把它们收束为新的 Agent 分类和入口语义。

### 8.2 任务系统接口仍混杂旧结构

当前 [backend/api/tasks.py](/d:/AI应用/langchain-agent/backend/api/tasks.py) 中仍暴露大量旧接口和旧请求模型，例如：

- agent upsert 中混入 `owner_system`
- general / flow / assignment 分裂为多套并行结构
- workflow 仍被当成独立主对象暴露
- connection diagnostics / carrying profiles 等旧面板接口占据主路径

这些接口不适合作为新任务系统主 API。

### 8.3 flow_registry 承担了过多旧兼容逻辑

当前 [backend/tasks/flow_registry.py](/d:/AI应用/langchain-agent/backend/tasks/flow_registry.py) 同时承担：

- 通用任务
- 特定任务
- Agent carrying profile
- 连接诊断
- workflow 引用投影
- coordination template

这是典型的结构混杂。

本次重构应拆回明确职责，而不是继续往里加分支。

### 8.4 前端任务页信息架构错误

当前 [frontend/src/components/workspace/views/TaskSystemView.tsx](/d:/AI应用/langchain-agent/frontend/src/components/workspace/views/TaskSystemView.tsx) 仍保留：

- 过多总览型结构
- 诊断型结构
- 平铺旧对象
- 与新边界不一致的编辑入口

这会直接误导后端模型设计。

## 九、清理原则

本次重构必须遵守以下原则：

1. 不为了兼容保留无意义旧残留
2. 不在旧结构上继续补丁式加功能
3. 先重建对象模型，再重建页面，再接回运行时
4. 任务系统只保留任务系统该有的东西
5. workflow 和 projection 的边界必须一次性理顺

## 十、实施阶段

### Phase 1：锁定新数据模型

输出：

- AgentRecord
- GeneralTaskProfile
- SpecificTaskRecord
- TaskWorkflowRecord
- CoordinationTemplate

完成标准：

- 前后端命名统一
- 旧字段淘汰清单明确

### Phase 2：重构后端注册层

输出：

- 新 AgentRegistry
- 新任务注册服务

- 新 coordination 服务

完成标准：

- 能独立完成 CRUD
- 不再依赖旧 carrying / diagnostics 结构作为主链路

### Phase 3：重构任务契约生成链路

输出：

- 从任务记录生成 runtime contract 的统一流程

完成标准：

- 通用任务可生成契约
- 特定任务可生成契约
- task -> workflow -> projection 顺序被锁定

### Phase 4：重构前端任务系统

输出：

- 新 Agent 管理
- 新任务管理
- 新编排管理

完成标准：

- 主 Agent / 系统管理 Agent / 工作子 Agent 分类清楚
- 通用任务 / 特定任务结构清楚
- workflow 成为任务管理子资源

### Phase 5：去除旧链路

输出：

- 删除不再使用的旧接口
- 删除旧页面残留
- 删除错误字段和旧状态面板

完成标准：

- 新链路单独可运行
- 不再依赖旧兼容逻辑

## 十二、验收标准

重构完成后，必须能明确证明以下事情：

1. Agent 不再被当成系统本体
2. 任务系统已经成为 Agent、任务、workflow、projection 的任务侧拥有者
3. 权限系统不再负责 workflow 和 projection
4. 特定任务可以稳定绑定 Agent + workflow + projection
5. 运行时消费的是任务系统产出的契约，而不是自己反向猜任务结构
6. 前端可以真正完成管理，而不只是展示状态

## 十三、最终结论

本次任务系统重构的真正目标，不是做一个更复杂的页面，而是建立一条稳定主链路：

```text
任务进入
  -> 任务系统识别
  -> 任务系统选定 Agent
  -> 任务系统绑定 workflow
  -> 任务系统绑定 projection
  -> 任务系统生成契约
  -> 运行时执行
```

其中：

- Agent 是执行者
- workflow 是做法
- projection 是任务向补充
- 权限系统只定义 Agent 内部运行边界

后续所有实现，都必须服从这条主链路。

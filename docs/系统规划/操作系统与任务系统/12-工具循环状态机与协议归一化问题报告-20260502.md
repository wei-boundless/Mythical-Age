# 工具循环状态机与协议归一化问题报告

日期：2026-05-02

## 一、问题定义

当前长情景失败已经不是单点能力缺失，也不是提示词里某一句话说错了，而是运行时主链存在一处更深的结构断裂：

1. 首轮模型调用是“可调工具”的执行态。
2. 工具结果返回后，follow-up 轮仍沿用“继续执行”的任务契约和上下文。
3. 但运行时却把第二轮模型调用降成了“不带工具绑定的纯回答态”。

这导致系统进入一个半开状态：

- 任务层认为还在继续能力执行；
- 模型上下文里保留了上一轮 assistant tool call 和 ToolMessage；
- 运行时却不再给模型结构化工具调用出口。

在 DeepSeek 这类提供者上，模型会把“本该走结构化 tool call 的意图”退化为内联协议文本，例如 DSML `tool_calls` 块。随后这段文本又被执行层和输出边界错误当成普通回答，最终污染会话、丢失文件绑定，并触发后续 follow-up 路由退化。

所以真正损坏的系统属性不是“某个 PDF follow-up 没命中”，而是：

- 工具循环状态机不一致；
- 提供者工具协议没有统一归一；
- 输出边界没有做到协议无关的 fail-closed。

## 四、深层结构追踪

## 4.1 运行时工具循环状态机断裂

关键文件：

- `backend/orchestration/runtime_loop/task_run_loop.py`

首轮调用：

- `model_response_executor.stream(..., tool_instances=runtime_tool_instances)`

follow-up 轮调用：

- `model_response_executor.stream(..., tool_instances=[])`

也就是：

1. 第一轮允许结构化工具调用。
2. 一旦收到工具结果，第二轮虽然仍以“continue_after_tool_result”继续循环，但工具绑定被撤掉了。

而同一时间，follow-up 输入却明确保留了：

- `AIMessage(content=assistant_tool_call_content, tool_calls=pending_tool_calls)`
- `ToolMessage(...)`

这会制造一个协议不一致状态：

- 对模型来说，历史表明自己刚刚正在工具循环里；
- 对任务契约来说，当前仍要求“执行相关能力并基于结果回答”；
- 对运行时来说，却不再接受新的结构化工具调用。

这是本轮最核心的结构错误。

## 4.2 执行层只识别结构化 `response.tool_calls`

关键文件：

- `backend/execution/model_response.py`

当前提取逻辑只有：

- `tool_calls = _normalize_tool_calls(getattr(response, "tool_calls", None))`

如果提供者不把工具调用放进 `response.tool_calls`，而是落进 `response.content`，执行器就会直接进入输出边界：

- `output_boundary.ingest_ai_update(raw_content, has_tool_calls=False)`

这意味着当前执行层默认假设：

- “工具调用一定以标准结构字段返回”

这个假设对 DeepSeek 并不稳。

## 4.3 DeepSeek 在半开状态下会把工具意图退化成 DSML 文本

关键文件：

- `backend/runtime/model_runtime.py`

这里已经为 DeepSeek 做了 `reasoning_content` 回放兼容，但没有做“工具协议文本归一化”。

一旦 follow-up 轮不再带工具绑定，DeepSeek 仍会根据上下文继续生成工具意图，只是它失去了结构化出口，于是退化成：

- `<｜｜DSML｜｜tool_calls>`
- `<｜｜DSML｜｜invoke name="pdf_analysis">`

这不是“模型胡说”，而是运行时把模型推到了一个协议悬空的状态。

## 4.4 输出边界不是协议无关的 fail-closed

关键文件：

- `backend/output_boundary/boundary.py`
- `backend/output_boundary/classifier.py`

当前输出边界已经能识别并清理一些协议残留，例如：

- `</think>`
- `<tool_call>`
- `**工具调用:**`
- `_CANONICAL_RESULT::`

但它并没有识别 DeepSeek DSML。

结果就是：

1. DSML 文本没被识别成内部协议。
2. `sanitize_visible_assistant_content(...)` 没清掉它。
3. `classifier.py` 仍可能把它收成 `answer_candidate`。

因此输出边界当前仍是“已知协议枚举式过滤”，不是“未消费协议文本一律不能成为 canonical answer”的结构。

## 五、为什么这不是单纯的 DSML 清洗问题

如果只在输出边界上补一个 DSML 正则，确实可以减少泄漏文本直接展示，但这仍然只是补洞，不是修结构。

原因有三点：

1. 根因不是“DSML 长什么样”，而是运行时把第二轮放进了一个不一致状态。
2. 就算把 DSML 文本滤掉，工具也没有真正执行，`active_pdf / task_summary_refs` 仍然不会被刷新。
3. 以后换成别的提供者专有协议，问题还会再次出现。

所以这轮必须先修状态机，再做协议归一，最后补输出边界兜底。

## 六、设计原则约束

结合当前代码和既有设计文档，本问题必须遵守以下约束：

1. 工具执行态和回答收口态必须是显式不同的运行时状态，不能混用。
2. prompt 只表达本轮任务与可用性，不负责承受协议兼容。
3. 提供者私有协议必须在执行层被归一，不应下沉到记忆或会话文本。
4. 输出边界只能消费：
   - 已验证执行结果
   - 或明确的用户可见答案
5. 未被执行层消费的协议文本，一律不能成为 canonical answer。
6. 文件工作对象绑定必须依赖真实工具执行结果投影，而不是依赖模型自述。

## 七、建议修复方向

### 7.1 第一优先级：修正工具循环状态机

目标：

- 明确 follow-up 轮到底是：
  - `answer_only_synthesis`
  - 还是 `tool_capable_followup`

当前系统既然保留了：

- `pending_tool_calls`
- `ToolMessage`
- `continue_after_tool_result`

更合理的方向是：

- follow-up 轮在工具循环未自然收口前，继续保持工具绑定；
- 只有在明确进入最终收口阶段时，才切换到 answer-only。

也就是说，运行时不能再出现：

- 上下文仍在工具循环里
- 但 `tool_instances=[]`

### 7.2 第二优先级：增加提供者工具协议归一层

目标：

- 在执行层统一解析：
  - 标准 `response.tool_calls`
  - 提供者内联协议文本

建议落点：

- `backend/execution/model_response.py`
- 必要时配合 `backend/runtime/model_runtime.py`

要求：

- 归一结果统一转成现有 `tool_call_requested` 事件；
- 不能把提供者协议文本继续向下游泄漏。

### 7.3 第三优先级：输出边界改为协议无关 fail-closed

目标：

- 任何“疑似协议文本但未被执行层消费”的输出，都不能进入 canonical answer。

建议：

- `boundary.py` 增加更抽象的内部协议检测，而不是继续堆具体协议字符串；
- `classifier.py` 对未消费协议文本强制退回“当前没有可验证执行结果”类兜底。

## 十、结论

本轮长情景残余失败的主因已经明确：

- 不是任务理解单独出错；
- 不是 PDF 能力本身出错；
- 也不是简单的提示词问题。

根因是：

- 工具循环状态机不一致，
- 进而触发提供者协议文本泄漏，
- 最终破坏输出收口和文件绑定延续。

下一步修复必须按以下顺序推进：

1. 先修工具循环状态机；
2. 再做提供者协议归一；
3. 最后补输出边界 fail-closed。

只有这样，后续 PDF / 结构化数据 / RAG 的长链能力才能稳定达到企业级可用，不会反复被协议残留和半开状态拖垮。

## 十四、第二轮深追：文件绑定被历史刷新覆盖

### 14.1 新现象

在后续复测中，`turn-05` 的 PDF 工具结果已经可以直接收口，并且 `turn-06` / `turn-07` 的 `done.main_context.active_constraints.active_pdf` 与 `task_summary_refs` 都能正常出现。

但 `turn-08` 仍然失败：

- 用户消息：`把这份 PDF 的核心结论压成三条行动建议。`
- 运行计划：`search_knowledge`
- `followup.mode = ""`
- `used_task_summary_refs = []`
- `main.active_pdf = ""`

这说明第一层工具循环已经修正，但第二层“工作对象延续”仍然断裂。

### 14.2 深层原因

进一步追踪后发现，问题不是 PDF 工具写回能力本身不能保存绑定，而是测试与兼容刷新路径在每轮结束后调用：

- `QueryRuntime.refresh_session_memory(session_id)`

该路径会用纯对话历史重建 session memory。纯历史里只有用户/助手文本，没有运行时提交的结构化 `main_context` 与 `task_summary_refs`。因此它会覆盖刚由运行时写入的结构化槽位。

本质上这是一个“restore 与 decide 混用”的问题：

- 运行时工具结果投影已经确认了 `active_pdf`；
- 历史摘要刷新只是一个兼容/压缩视图刷新；
- 但兼容刷新却重新决定了结构化文件绑定，并把已确认绑定清空。

### 14.3 修正原则

这次不使用 prompt 修补，也不把某一句“这份 PDF”硬编码成答案路径。修正点放在结构化工作记忆层：

1. 已确认文件工作对象的绑定槽位不能被纯历史刷新随意清空。
2. 当前轮是跟进句，并且没有显式切换任务时，应继承上一轮已确认的 `active_pdf / active_dataset`。
3. 绑定继承时必须同时保留：
   - `active_binding_kind`
   - `active_binding_identity`
   - `active_binding_owner_task_id`
   - `active_object_handle_id`
   - `active_result_handle_id`
   - `active_subset_handle_id`
   - `committed_pdf / committed_dataset`
4. 历史刷新可以提供连续性摘要，但不能覆盖运行时确认的文件工作对象事实。

### 14.4 已实施修正

影响文件：

- `backend/structured_memory/process_engine.py`
- `backend/tests/file_work_object_writeback_regression.py`

具体修正：

- `_build_context_slots(...)` 在跟进请求中保留上一轮已确认的 PDF / dataset 绑定；
- 对 PDF 跟进保留上一轮页/章节作用域，避免被普通文本推断重置为全文模式；
- `_apply_reconciliation_to_context_slots(...)` 修复不完整拷贝，避免后处理清空 binding / handle / committed 字段；
- 新增回归：`test_history_refresh_preserves_bound_pdf_followup_slots`。

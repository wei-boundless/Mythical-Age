# 问题三：模板选择与任务拆分契约化重构计划书

日期：2026-05-02

实施状态：2026-05-02 首轮实现已完成，主链代码与回归测试已落地。

## 一、文档目标

本计划书是对 `19-问题三-模板选择与任务拆分契约化差距报告-20260502.md` 的进一步展开。

本计划书回答四个更具体的问题：

1. 为什么问题三不能靠“继续补关键词和路由规则”来解决。
2. 先进架构对父任务、子任务、输入绑定、状态隔离的最小共同要求是什么。
3. 结合当前代码，模板选择与任务拆分契约层应该落在哪一层，哪些结构可以直接复用。
4. 如何在不牺牲任务灵活性的前提下，把模板匹配、bundle 拆分、follow-up 绑定收敛成稳定的结构化 contract。

本计划不讨论的问题：

- crash-resume 与副作用幂等细则
- 故障演练矩阵的完整铺开
- worker 跨进程调度协议
- 前端任务编排与模板装配 UI

这些问题后续分别由 `20` 对应阶段继续处理。

## 二、核心判断

### 2.1 需要的不是“更多模板”，而是“模板匹配与拆分契约层”

当前系统已经有：

- `TaskTemplate`
- `TaskSpec`
- `TaskStepBlueprint`
- `CurrentTurnContext`
- `BundleItem`

因此问题已经不再是“有没有模板”和“能不能拆子任务”，而是：

- 模板到底为什么被选中
- 子任务到底为什么被拆出来
- 父子任务之间共享什么输入和状态
- follow-up 到底根据什么稳定地回指前一轮结果

当前答案仍然是：

- 模板主要由 `select_template()` 按 route hint、模态、关键词和工具候选混合判断
- 子任务主要由 `_bundle_items()` 按自然语言分句和能力猜测拆分
- follow-up 主要由 ordinal 文本和上一轮投影结果猜测回绑

目标答案应该是：

- 先形成 `TaskIntentContract`
- 再形成 `TemplateMatchResult`
- 如果需要拆分，再形成 `BundleSpec / BundleItemSpec`
- 最后用 `StepInputBinding` 把输入、继承状态、写回边界接到具体步骤

### 2.2 我们要的不是“更聪明的启发式”，而是“显式父子任务契约”

本次重构不追求：

- 把每一种用户措辞都手工穷举进路由规则
- 让模板选择继续依赖关键词叠加
- 让 bundle 继续由句子切分结果直接代表子任务边界

本次重构明确追求：

- 模板匹配结果可解释、可记录、可比较
- bundle 是显式数据结构，不是 split 后的临时列表
- 子任务输入继承和私有状态边界可声明
- follow-up 绑定优先基于稳定 id，而不是自然语言序号猜测

### 2.3 最终目标

最终要形成如下关系：

```text
user request + current-turn truth
  -> TaskIntentContract
  -> TemplateMatchResult
  -> TaskSpec
  -> BundleSpec / BundleItemSpec (optional)
  -> StepInputBinding[]
  -> RuntimeLoop consumes explicit task/bundle contracts
```

而不是当前更接近的：

```text
user wording
  -> understanding heuristics
  -> select_template()
  -> _bundle_items()
  -> follow-up rebinding by text hints
```

## 三、理论依据

### 3.1 本地设计原则依据

来自项目内已有设计原则：

- [03-状态管理.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/03-状态管理.md:1)
  - 状态需要分层，边界需要清晰
- [05-对话循环.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/05-对话循环.md:1)
  - 对话循环是状态机，每次推进都要有清晰依据
- [14-任务系统.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/14-任务系统.md:1)
  - 任务系统负责生命周期、隔离、终止、输出收集

这几条叠加后的直接约束是：

- 模板选择不能长期停留在语义猜测层
- 子任务拆分不能只是 prompt 侧便利逻辑
- 父任务、子任务、步骤输入三层边界必须显式存在

### 3.2 LangGraph Subgraphs 的依据

LangGraph 在 subgraphs 设计里强调：

- 父图和子图之间要有显式 state schema
- 哪些状态是共享的，哪些状态是私有的，要提前定义
- 子图既可以被复用，也可以按 fresh invocation 运行

这对我们的直接启发是：

- bundle item 不能只是 `user_text + capability_kind`
- 父任务传给子任务的输入需要显式 contract
- follow-up 不能只依赖“第几个子任务”这种文本回指

来源：

- https://docs.langchain.com/oss/javascript/langgraph/use-subgraphs

### 3.3 LangGraph Durable Execution 的依据

LangGraph durable execution 的关键思想之一是：

- 可恢复运行依赖稳定的 invocation boundary
- side-effecting / non-deterministic work 必须包在清晰边界中
- 恢复与回放必须建立在结构化输入输出边界上

这对我们的启发是：

- 如果模板匹配和 bundle 拆分没有稳定 contract，durable execution 也会变得不稳定
- step state 和 execution record 已经稳定之后，任务入口仍必须收敛为结构化 contract

来源：

- https://docs.langchain.com/oss/javascript/langgraph/durable-execution

### 3.4 结构化编排的一般工程依据

成熟任务运行时通常都遵守同一个原则：

- 控制结构依赖显式 contract
- 语言理解只负责产生候选解释，而不是直接充当运行时真相

这也是为什么前两个问题先解决了：

- 问题一解决“步骤状态谁说了算”
- 问题二解决“副作用执行语义谁说了算”

问题三对应要解决的就是：

- “任务入口和子任务边界谁说了算”

如果继续沿用现状，会持续带来以下问题：

- 同一意图换一种说法，可能命中不同模板
- 前端如果显式选了模板，后端仍可能被语义启发式绕走
- bundle item 数量、顺序、能力类型会受标点和措辞影响
- follow-up 在多轮之后会越来越依赖文本猜测，而不是结构回指
- 问题一和问题二新建立的状态机、execution record 很难获得稳定的上游输入边界

## 五、设计原则与边界

### 5.1 原则一：区分 intent contract、template match、bundle contract、step binding 四层

这四层职责必须分开：

- intent contract
  - 表达本轮任务的结构化输入意图
- template match
  - 表达为什么命中这个模板
- bundle contract
  - 表达父任务如何拆成多个子任务
- step binding
  - 表达每个步骤实际消费什么输入、继承什么状态、写回什么结果

不能继续让 `CurrentTurnContext` 或自然语言分句同时承担这四层职责。

### 5.2 原则二：模板匹配优先看显式 contract，启发式只保留为 fallback

优先级建议如下：

1. 显式模板选择
2. 显式输入绑定与 modality contract
3. 显式 capability / output requirement
4. 受限启发式 fallback

这意味着：

- 启发式仍可保留，但必须降级为可观测 fallback
- 不允许它继续作为默认主路径

### 5.3 原则三：父子任务输入边界必须显式声明

子任务需要区分：

- 直接来自用户请求的输入
- 继承自父任务的共享输入
- 当前子任务自己的私有输入
- 当前子任务完成后需要写回的结果引用

这层边界不明确，后续并行 bundle、follow-up、恢复都很难可靠。

### 5.4 原则四：follow-up 绑定优先走稳定 id，而不是自然语言序号

序号类 follow-up 可以保留，但它应当只是：

- 从文本进入结构化 target lookup 的入口

而不是最终绑定依据。

最终应优先落在：

- `bundle_id`
- `item_id`
- `task_spec_ref`
- `step_id`

### 5.5 原则五：不提前做重型 DAG 引擎

本阶段不做：

- 通用 DAG planner
- 任意图状依赖求解
- 模板内动态插步引擎
- 多层嵌套 subgraph runtime

本阶段只做：

- 稳定模板匹配
- 稳定 bundle contract
- 稳定 step input/output binding

## 六、目标设计

### 6.1 新增 TaskIntentContract

建议新增统一模型：

```python
class TaskIntentContract:
    task_intent_id: str
    session_id: str
    task_id: str
    intent_kind: str
    execution_intent: str
    requested_outputs: tuple[str, ...]
    explicit_inputs: dict[str, Any]
    source_bindings: tuple[str, ...]
    followup_target: dict[str, Any]
    diagnostics: dict[str, Any]
```

建议它表达的不是“模型理解了什么”，而是“运行时准备按照什么结构启动任务”。

### 6.2 新增 TemplateMatchResult

建议新增统一模型：

```python
class TemplateMatchResult:
    match_id: str
    task_intent_ref: str
    template_id: str
    match_source: str
    match_reasons: tuple[str, ...]
    fallback_used: bool
    capability_contract: tuple[str, ...]
    output_contract: tuple[str, ...]
    diagnostics: dict[str, Any]
```

建议最小 `match_source`：

- `explicit_template`
- `binding_contract`
- `capability_contract`
- `heuristic_fallback`

### 6.3 新增 BundleSpec / BundleItemSpec

建议新增统一模型：

```python
class BundleSpec:
    bundle_id: str
    parent_task_spec_ref: str
    aggregation_policy: str
    items: tuple["BundleItemSpec", ...]
    diagnostics: dict[str, Any]


class BundleItemSpec:
    item_id: str
    ordinal: int
    template_id: str
    capability_kind: str
    requested_outputs: tuple[str, ...]
    inherited_binding_refs: tuple[str, ...]
    private_inputs: dict[str, Any]
    target_ref: str
    diagnostics: dict[str, Any]
```

这里的重点不是换个名字，而是让 bundle 变成：

- 可持久化
- 可追踪
- 可 follow-up 回指
- 可被 runtime loop 稳定消费

### 6.4 新增 StepInputBinding

建议新增统一模型：

```python
class StepInputBinding:
    step_id: str
    input_refs: tuple[str, ...]
    inherited_parent_refs: tuple[str, ...]
    private_state_refs: tuple[str, ...]
    output_writebacks: dict[str, str]
    binding_policy: str
```

建议它回答三件事：

- 这个 step 读什么
- 这个 step 能继承什么
- 这个 step 结束后把什么写回哪里

### 6.5 TaskSpec 的目标收口

最终 `TaskSpec` 应稳定携带：

- `task_intent_ref`
- `template_match_ref`
- `bundle_spec_ref` 或 `bundle_item_ref`
- `step_input_bindings`

这样 runtime loop 消费的就不再是“混合推断结果”，而是显式 contract。

## 七、分阶段改造建议

### 7.1 Phase 3A：引入 TaskIntentContract 与 TemplateMatchResult

目标：

- 把“为什么选中这个模板”从隐式分支逻辑提升为显式模型

主要动作：

- 新增 `TaskIntentContract`
- 新增 `TemplateMatchResult`
- 让 `CurrentTurnContext` 先投影为 `TaskIntentContract`
- 让 `select_template()` 返回结构化 match result，而不是只返回模板

优先改动文件：

- [template_registry.py](/abs/path/d:/AI应用/langchain-agent/backend/tasks/template_registry.py:372)
- [contract_builder.py](/abs/path/d:/AI应用/langchain-agent/backend/tasks/contract_builder.py:1)
- 可新增 `backend/tasks/match_contracts.py`

阶段验收：

- 模板匹配必须带理由
- 显式模板选择优先级必须高于启发式
- 同一输入 contract 的不同文案变体，尽量收敛到同一 `template_id`

### 7.2 Phase 3B：引入 BundleSpec / BundleItemSpec，替换临时 bundle item 列表

目标：

- 把 bundle 从“句子拆分结果”提升为稳定 contract

主要动作：

- 用 `BundleSpec` 取代当前临时 `BundleItem` 主导地位
- 给每个子任务稳定 `item_id`
- 明确 `aggregation_policy`
- 在 parent task 和 item task 之间建立引用关系

优先改动文件：

- [resolver.py](/abs/path/d:/AI应用/langchain-agent/backend/context_management/resolver.py:162)
- [spec_models.py](/abs/path/d:/AI应用/langchain-agent/backend/tasks/spec_models.py:1)
- 可新增 `backend/tasks/bundle_models.py`

阶段验收：

- 相同复合请求在语义等价改写后，bundle item 数量和主能力类型保持稳定
- bundle item 能被 trace / checkpoint / follow-up 一致引用

### 7.3 Phase 3C：引入 StepInputBinding，收敛步骤输入和写回边界

目标：

- 让 step 不再隐式读取上下文，而是消费显式 binding

主要动作：

- 为 `TaskStepBlueprint` 对应生成 `StepInputBinding`
- 区分父任务继承输入与子任务私有输入
- 明确 step 完成后的 writeback contract

优先改动文件：

- [step_models.py](/abs/path/d:/AI应用/langchain-agent/backend/tasks/step_models.py:1)
- [task_run_loop.py](/abs/path/d:/AI应用/langchain-agent/backend/orchestration/runtime_loop/task_run_loop.py:1)
- [projection.py](/abs/path/d:/AI应用/langchain-agent/backend/context_management/projection.py:1)

阶段验收：

- runtime loop 能解释“当前 step 正在读哪些输入”
- follow-up 能基于 writeback ref 定位上一轮结果，而不是靠文本推断

### 7.4 Phase 3D：重构 follow-up 绑定，优先走稳定结构引用

目标：

- 把 follow-up 从“文本序号猜测”提升为“结构 target lookup”

主要动作：

- 保留 ordinal / 文本 follow-up 入口
- 但内部第一时间解析到 `bundle_id / item_id / task_spec_ref / step_id`
- 降低 `bundle_result_refs` 这种投影列表的最终真相职责

优先改动文件：

- [resolver.py](/abs/path/d:/AI应用/langchain-agent/backend/context_management/resolver.py:97)
- [projection.py](/abs/path/d:/AI应用/langchain-agent/backend/context_management/projection.py:1)
- [trace_reader.py](/abs/path/d:/AI应用/langchain-agent/backend/orchestration/runtime_loop/trace_reader.py:1)

阶段验收：

- “继续第 2 个子任务”会稳定解析到已有 `item_id`
- 同一 bundle 的 follow-up 在多轮对话后仍能命中正确 target

### 7.5 Phase 3E：切换主链并清理旧启发式残留

目标：

- 让 contract-first 成为主路径，并清理不再有价值的旧残留

主要动作：

- 保留启发式分支仅作为 fallback
- 为 fallback 增加显式 diagnostics
- 删除只对旧分句 / 旧匹配逻辑服务的无用残留代码

优先改动文件：

- [template_registry.py](/abs/path/d:/AI应用/langchain-agent/backend/tasks/template_registry.py:372)
- [resolver.py](/abs/path/d:/AI应用/langchain-agent/backend/context_management/resolver.py:162)
- [contract_builder.py](/abs/path/d:/AI应用/langchain-agent/backend/tasks/contract_builder.py:1)

阶段验收：

- 主链默认不再依赖启发式
- 启发式是否介入对 trace 可见
- 无价值旧残留被清理，而不是继续并存

## 八、切换策略

### 8.1 先记录 contract，再切主消费方

建议切换顺序：

1. 先在 contract builder 里生成 `TaskIntentContract` / `TemplateMatchResult`
2. 再让 bundle path 生成 `BundleSpec`
3. 再让 runtime loop 读取 `StepInputBinding`
4. 最后切 follow-up 和 template selection 主路径

这样可以避免：

- 一步切完导致 bundle、follow-up、template selection 同时失稳

### 8.2 先 shadow compare，再硬切

建议在过渡期保留：

- 新 contract 结果
- 旧启发式结果

并记录它们是否一致。

当以下条件满足时再硬切：

- 高频模板匹配一致率稳定
- 复合请求 bundle 稳定
- follow-up target 解析稳定

### 8.3 清理策略必须积极

根据项目 `AGENTS.md` 约束：

- 无价值旧残留不要长期保留
- 不要为了“保险”而把旧路径永久并行

因此本阶段完成后，应主动清理：

- 只服务于旧链路的模板匹配分支
- 只服务于旧 bundle split 的临时逻辑
- 只服务于旧 follow-up 猜测的无用中间结构

## 九、验收总口径

问题三完成后，至少应满足以下条件：

1. 同一任务意图在措辞变化下，模板匹配结果基本稳定。
2. 显式模板选择、显式输入绑定、显式 capability 请求的优先级高于启发式推断。
3. bundle 是稳定 contract，不再只是运行中临时分句结果。
4. follow-up 优先绑定到稳定结构 id，而不是只依赖文本序号。
5. step 输入来源与结果写回边界可追踪。
6. 启发式若仍介入，trace 中必须能解释其介入原因。

## 十、建议新增回归矩阵

建议至少补以下回归：

- 模板稳定性回归
  - 同义改写命中相同模板
- 显式模板优先级回归
  - 显式选择不被启发式覆盖
- bundle 稳定性回归
  - 标点、连接词变化不改变核心 item 结构
- follow-up 结构绑定回归
  - “继续第 2 个子任务”命中稳定 `item_id`
- step binding 回归
  - 每个 step 的输入和写回都可解释

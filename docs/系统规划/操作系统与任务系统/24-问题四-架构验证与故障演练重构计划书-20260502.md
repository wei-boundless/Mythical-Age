# 问题四：架构验证与故障演练重构计划书

日期：2026-05-02

## 一、文档目标

本计划书是对 `20-问题四-架构验证与故障演练差距报告-20260502.md` 的进一步展开。

本计划书聚焦问题四本体，即架构验证、故障演练、恢复证明和切流证据。健康系统独立控制面、任务系统中的通用 agent/task 连接、健康管理 agent 代操职责的跨系统统筹，见：

- [25-健康系统管理控制面与验证体系统筹重构计划书-20260502.md](/abs/path/d:/AI应用/langchain-agent/docs/系统规划/操作系统与任务系统/25-健康系统管理控制面与验证体系统筹重构计划书-20260502.md:1)

本计划书回答四个更具体的问题：

1. 为什么问题四不能靠“继续多写一些 happy path 回归”来解决。
2. 先进架构对故障演练、恢复验证、cutover 证据的最小共同要求是什么。
3. 结合当前代码，验证矩阵、故障注入、恢复证据、切流规则应该落在哪一层，哪些结构可以直接复用。
4. 如何在不引入笨重 chaos 平台的前提下，把问题四做成稳定、可执行、可持续迭代的工程闭环。

本计划不讨论的问题：

- 新的任务模板扩展
- worker 跨机器编排协议
- 前端运维面板与可视化演练 UI
- 分布式集群级容量治理

这些问题不属于本阶段的主目标。

## 二、核心判断

### 2.1 需要的不是“更多测试数量”，而是“架构级验证层”

当前系统已经有：

- `TaskRunLedger`
- `TaskResult`
- `RuntimeCheckpoint`
- `RuntimeEventLog`
- `RuntimeExecutionStore`
- 一批 runtime / memory / task regression

因此问题已经不再是“有没有测试”和“trace 能不能看”，而是：

- 故障是不是被定义成一级对象
- 恢复是不是有明确验收口径
- 切流是不是依赖固定证据而不是主观判断
- 失败后的结构正确性是不是能被系统性证明

当前答案仍然是：

- 以功能链路为主
- 以静态装配和 happy path 为主
- 以人工阅读 trace 为主

目标答案应该是：

- 先定义故障场景目录
- 再定义每类场景的结构性断言
- 再把这些断言落成固定测试矩阵
- 最后把 cutover / rollback 规则接进阶段验收制度

### 2.2 我们要的不是“随机混沌实验”，而是“结构化故障演练”

本次重构不追求：

- 引入重型 chaos engineering 平台
- 把所有依赖都做成分布式故障模拟器
- 一上来覆盖全栈所有失败场景

本次重构明确追求：

- 对关键运行边界建立固定故障目录
- 对关键恢复语义建立固定断言口径
- 对每次架构阶段切换建立固定 cutover 检查表
- 让失败链和恢复链与 happy path 一样可回归、可复查、可持续维护

### 2.3 问题四最适合和健康系统管理子 agent 联动

问题四和前三个问题有一个重要不同：

- 前三个问题主要在重构主运行时结构
- 问题四主要在建立“对主运行时持续体检和验证”的机制

因此问题四不应只作为主 agent 的一次性回归任务存在，更适合沉淀到健康系统管理子 agent：

- 由健康子 agent 负责故障目录维护
- 由健康子 agent 负责 trace / checkpoint / ledger / execution record 的结构化体检
- 由健康子 agent 负责 fix verification、回归验证、切流前健康检查

但这里必须保留一个强边界：

- 健康子 agent 负责“验证与分析”
- 主 runtime 仍然负责“事实写入与执行真相”

也就是说，健康子 agent 可以解释系统健康状态、给出验证结论、组织验证流程，但不能替代 `TaskRunLedger`、`TaskResult`、`RuntimeCheckpoint` 这些主结构的真相地位。

### 2.4 最终目标

最终要形成如下关系：

```text
static contract validation
  -> runtime happy-path validation
  -> fault injection validation
  -> recovery / replay validation
  -> cutover readiness gate
  -> health sub-agent continuous verification workflow
```

而不是当前更接近的：

```text
template validation
  -> runtime happy-path tests
  -> manual confidence
```

### 2.5 任务系统与编排系统的进一步分工

结合当前重构进度，本计划补充一个更明确的系统分工：

- 任务系统更像“思维连接层”
  - 决定有哪些 agent
  - 决定任务如何分配给 agent
  - 决定父任务、子任务、bundle、follow-up、协作拓扑如何组织
  - 为健康子 agent 提供任务身份、灵魂投影设定、工作流和 task flow binding
- 编排系统更像“身体配置层”
  - 决定每个 agent 的 runtime lane
  - 决定每个 agent 的 resource policy、可用 operation、memory scope、projection template、workflow
  - 决定执行过程中 checkpoint、ledger、execution record、commit gate 如何推进
  - 决定权限拒绝、replay denied、retry exhausted 这类运行时边界如何生效

如果用一个更直观的比喻：

- 任务系统负责“大脑里的任务连接和分工关系”
- 编排系统负责“每个 agent 的身体能力、动作边界和运行姿态”

这条分工非常重要，因为它直接决定健康子 agent 应该如何建设：

- 健康管理 agent 的“任务身份、灵魂投影、工作流和绑定关系”应作为通用 agent/task 连接的一部分在任务系统里管理
- 健康系统的“健康设置、测试启动、issue/report/readiness 管理”应由健康系统独立操作台和健康系统后端管理，健康管理 agent 只能通过嵌入式对话窗代操或建议
- 健康管理 agent 的“实际读写权限、链路能力、能否启动某类验证动作”应由编排系统和资源策略层负责硬约束

也就是说：

- 任务系统可以让用户配置“某个 agent 使用哪个灵魂投影”“绑定哪些 task flow”“使用哪个 skill workflow”“参与哪些协作拓扑”，健康管理 agent 是其中一个实例
- 健康系统原生操作台可以让用户直接配置“要不要启用健康链路分析”“要不要运行某类验证”“健康系统设置如何调整”
- 健康系统嵌入式 agent 对话窗可以让用户要求健康管理 agent 解释、建议或代操上述动作
- 但真正能不能执行、执行时能读什么、能不能触发写操作，仍然必须由编排系统在运行时判定

这样才能避免把“前端配置”直接变成“运行时越权”。

- 恢复链必须有结构化证据
- 权限拒绝、部分完成、重试耗尽必须属于架构验收对象

### 3.2 AWS Operational Excellence 的依据

AWS Operational Excellence 强调：

- operations as code
- refine procedures frequently
- anticipate failure
- 从运行故障持续学习

这对我们的直接启发是：

- 故障演练不能只写在文档里
- 演练程序必须代码化
- 每次阶段切换前要有固定的验证程序，而不是“感觉已经差不多”

来源：

- https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/oe-design-principles.html

### 3.3 AWS Reliability 的依据

AWS Reliability 对运行失败的要求更具体：

- fail fast，避免问题扩散
- limit retries，避免多层重试放大故障
- limit queues / stale backlog，避免积压演变成系统级不一致

这对我们的直接启发是：

- retry exhaustion 必须有专门验证
- permission blocked 和 replay denied 不能只看最终报错文案
- stale queue / delayed worker / backlog 需要进入故障目录

来源：

- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_mitigate_interaction_failure_fail_fast.html
- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_mitigate_interaction_failure_limit_retries.html

### 3.4 Durable runtime 的依据

成熟 durable runtime 的共同特点不是“永不失败”，而是：

- 失败后仍然能说明系统在哪一步停下
- 能说明哪些副作用已提交、哪些未提交
- 能说明恢复为何允许、为何拒绝

这和我们前两阶段已经落地的：

- 步骤级状态机
- 副作用执行语义层

是直接衔接的。问题四的目标不是重复建设它们，而是把它们变成可证明、可演练、可切流的工程依据。

来源：

- https://docs.temporal.io/workflow-execution
- https://assets.temporal.io/durable-execution.pdf
- https://docs.langchain.com/oss/javascript/langgraph/durable-execution

## 五、设计原则与边界

### 5.1 原则一：把 failure path 当成一级对象

验证体系必须显式区分：

- happy path
- failure path
- recovery path
- cutover path

不能继续让 failure path 只作为个别 regression 的边角条件出现。

### 5.2 原则二：验证矩阵必须覆盖静态、动态、故障、恢复、切流五层

建议统一五层：

1. 静态契约验证
2. runtime happy-path 验证
3. 故障注入验证
4. 恢复与 replay 验证
5. cutover / rollback readiness 验证

### 5.3 原则三：故障断言必须围绕结构不变量，而不是文案

问题四不能只验证：

- 是否抛错
- 是否输出一句提示

更要验证：

- `TaskRunLedger` 是否终止在正确 step
- `TaskResult` 是否与 terminal reason 一致
- `execution_record` 是否进入正确 replay state
- checkpoint / trace / commit gate 是否彼此一致

### 5.4 原则四：高风险动作默认验证 fail-closed

对 shell、edit、worker、外部写入这类路径：

- 无法证明 replay-safe 时，验证应以拒绝自动恢复为正确结果
- 不要把“恢复顺畅”当成唯一目标

### 5.5 原则五：不提前做分布式 chaos 平台

本阶段不做：

- 大规模随机故障注入系统
- 集群级网络抖动实验平台
- 跨节点调度器混沌编排

本阶段只做：

- 关键运行边界的结构化故障目录
- 固定断言口径
- 固定 cutover 门槛

### 5.6 原则六：健康子 agent 负责验证编排，不负责替代主真相层

建议明确职责切分：

- 主 runtime
  - 负责执行
  - 负责账本
  - 负责 checkpoint / trace / result truth
- 健康子 agent
  - 负责读取这些结构
  - 负责诊断与归因
  - 负责组织验证任务
  - 负责输出健康结论与修复验证建议

这条原则很重要，因为如果让健康子 agent 直接“改写真相”，问题四会反过来污染问题一、问题二、问题三。

### 5.7 原则七：任务系统负责子 agent 配置，编排系统负责运行时约束

健康子 agent 未来可以暴露更多管理能力，但这些能力必须分两层：

- 任务系统层能力
  - 启用或禁用健康子 agent 的任务身份
  - 配置默认 task mode、灵魂投影、skill workflow、task flow binding
  - 配置健康子 agent 参与哪些协作拓扑
- 健康系统层能力
  - 启动验证
  - 切换验证模式
  - 调整健康策略
  - 管理 issue / report / readiness / health test
  - 通过健康系统悬浮小框承载用户与健康子 agent 的对话
- 编排系统层能力
  - 判定该 agent 当前是否真的允许执行该类动作
  - 判定该动作的 operation requirement、resource policy、approval policy
  - 决定该动作最终进入哪个 runtime lane、使用哪个 workflow、读写哪些上下文

这里的关键约束是：

- 任务系统可以表达子 agent 配置意图
- 健康系统可以表达健康管理意图
- 编排系统必须决定可执行性

不能反过来让任务配置或健康设置直接覆盖 resource policy。

### 5.8 原则八：健康子 agent 应是“管理者”，不是“万能执行者”

本次明确把健康子 agent 定位为健康系统管理者，它应该拥有除了“直接改写主结构真相”之外的健康系统权限：

- 能接收对话中的故障线索
- 能沿 trace / checkpoint / ledger / execution record 找问题
- 能生成问题报告、复现草案、修复验证结论
- 能向后台健康系统写入 issue、run、report、verification result
- 能响应用户发起的健康设置调整与测试启动请求

但它不应拥有以下能力：

- 直接改写 `TaskRunLedger`
- 直接改写 `RuntimeCheckpoint`
- 直接改写 `TaskResult`
- 直接绕过 resource policy 扩权
- 直接把前端设置解释成可执行写权限

## 六、目标设计

### 6.1 新增 FailureScenarioCatalog

建议新增统一场景目录：

```python
class FailureScenario:
    scenario_id: str
    layer: str
    injection_point: str
    preconditions: dict[str, Any]
    expected_terminal_reason: str
    expected_step_statuses: dict[str, str]
    expected_replay_policy: str
    expected_trace_events: tuple[str, ...]
    expected_commit_state: dict[str, Any]
```

建议最小覆盖 layer：

- `task_contract`
- `step_execution`
- `operation_gate`
- `execution_record`
- `checkpoint_commit`
- `worker_runtime`

### 6.2 新增 RecoveryInvariantReport

建议新增统一恢复不变量视图：

```python
class RecoveryInvariantReport:
    report_id: str
    task_run_id: str
    checkpoint_ref: str
    ledger_status_ok: bool
    execution_summary_ok: bool
    replay_decision_ok: bool
    commit_state_ok: bool
    violations: tuple[str, ...]
```

它不是替代 trace，而是把 trace 压成“恢复是否结构正确”的固定口径。

### 6.3 新增 CutoverReadinessReport

建议新增阶段切流检查结果：

```python
class CutoverReadinessReport:
    stage_id: str
    candidate_scope: str
    static_validation_ok: bool
    happy_path_ok: bool
    fault_matrix_ok: bool
    rollback_plan_present: bool
    blockers: tuple[str, ...]
```

这个结构的目标是把“能不能切流”从口头判断收回来。

### 6.4 固化问题四的场景目录

建议至少固定以下场景：

- side effect committed before checkpoint
- checkpoint written before result commit
- permission denied mid-step
- replay denied for non-safe operation
- retry exhausted on worker step
- partial completion with allowed skip
- stale backlog / delayed worker result
- duplicate side effect request with reused receipt

### 6.5 固化问题四的断言入口

建议优先复用并扩展：

- `test_system` 的 assertion 体系
- runtime loop trace summary
- memory contract regression
- task runtime contract regression

也就是说，问题四不是另起一套验证框架，而是把现有断言系统升级成架构验证层。

### 6.6 引入 Health Verification Workflow

建议把问题四的一部分固定沉淀为健康系统工作流：

```python
class HealthVerificationWorkOrder:
    work_order_id: str
    scenario_id: str
    target_task_run_id: str
    target_checkpoint_ref: str
    target_trace_ref: str
    verification_mode: str
    expected_invariants: dict[str, Any]
    issued_by: str
```

建议它由健康子 agent 消费，并输出：

```python
class HealthVerificationResult:
    verification_id: str
    work_order_ref: str
    verdict: str
    invariant_report_ref: str
    fix_verification_needed: bool
    recommended_actions: tuple[str, ...]
```

这让问题四不只是“跑测试”，而是：

- 让健康子 agent 有稳定输入
- 让健康子 agent 的输出可以回写到架构治理流程
- 让 trace analysis / fix verification / cutover readiness 能进入同一验证工作流

### 6.7 引入 Health Management Control Plane

建议把健康子 agent 建设成独立的健康管理控制面，而不只是若干只读 flow 的集合。

建议最小结构如下：

```python
class HealthManagementCommand:
    command_id: str
    command_type: str
    target_scope: str
    requested_by: str
    task_mode: str
    payload: dict[str, Any]
```

其中 `command_type` 至少覆盖：

- `report_issue`
- `analyze_trace`
- `draft_case`
- `verify_fix`
- `launch_health_test`
- `update_health_setting`
- `build_cutover_readiness`

建议再补一层后台落账结构：

```python
class HealthManagementReceipt:
    receipt_id: str
    command_ref: str
    accepted: bool
    execution_mode: str
    health_issue_ref: str
    health_run_ref: str
    verification_result_ref: str
    diagnostics: dict[str, Any]
```

这层结构的目标是把“用户和健康子 agent 的管理对话”变成后台可追踪的健康操作账本，而不是留在聊天文本里。

### 6.8 明确健康子 agent 的职责矩阵

建议把健康子 agent 的职责明确收敛为六类：

1. 故障接入
   - 从用户对话、trace、runtime 事件中识别健康问题
2. 链路诊断
   - 基于 `HealthIssue` 组织 trace analysis、problem node 归因、风险判断
3. 报告编制
   - 生成问题报告、复现草案、修复验证报告、切流健康结论
4. 后台汇报
   - 向健康后台写入 issue、run、report、verification result
5. 设置管理
   - 接收用户对健康系统开关、验证策略、测试分组的调整请求
6. 测试与演练调度
   - 发起固定验证场景、故障演练、cutover readiness 检查

同时明确不属于它的职责：

- 不直接编辑运行时内部真相结构
- 不直接拥有 shell / edit / worker 扩权
- 不替代主 agent 进行最终业务回答整合

### 6.9 明确健康子 agent 的建设分层

建议按四层建设，而不是把所有逻辑塞进一个 registry：

1. 前端任务控制层
   - 放在任务系统 UI 中
   - 只负责健康子 agent 的任务身份、灵魂投影、workflow、task flow binding 配置
2. 健康管理命令层
   - 把健康系统原生操作台的用户直操意图、嵌入式 agent 对话窗的代操意图编译成 `HealthManagementCommand`
   - 决定调用 issue_triage / trace_analysis / fix_verification / health test 等哪条流
3. 编排约束层
   - 由 `AgentCapabilityProfile`、`ResourcePolicy`、runtime lane 决定动作能否执行
4. 健康执行与回执层
   - 由 `HealthRegistry`、health workbench、verification workflow 记录 issue、run、receipt、result

这四层一旦分清，健康子 agent 就既能成为“管理者”，又不会越过主 runtime 的边界。

## 八、切换策略

### 8.1 先补断言骨架，再补场景，不要反过来

建议顺序：

1. 先统一问题四的断言口径
2. 再补故障目录
3. 再补具体场景
4. 最后接 cutover 规则

这样可以避免：

- 一次写很多故障测试，但每个测试口径都不一样

### 8.2 先高风险边界，后外围场景

建议优先级：

1. 非 replay-safe 写路径
2. permission denied / gate blocked
3. retry exhausted / partial completion
4. stale backlog / delayed worker

这样更符合当前系统真实风险分布。

### 8.3 文档与测试必须同步落地

问题四最忌讳的是：

- 文档列了很多场景
- 代码里没有固定断言

因此每补一种场景，都应同时落：

- 场景定义
- 测试用例
- 验收口径

## 九、验收总口径

问题四完成后，至少应满足以下条件：

1. 静态装配验证、happy path 验证、故障注入验证、恢复验证、cutover 验证形成统一矩阵。
2. 任一关键副作用路径都能明确证明“失败后是否允许恢复、为何允许、为何拒绝”。
3. 任一关键 step 失败都能明确证明 ledger、task_result、checkpoint、trace 是一致的。
4. 任一阶段切换都能明确给出 cutover 和 rollback 条件，而不是口头判断。
5. 新模板、新执行器、新恢复语义必须能挂接到现有故障目录与断言体系。
6. 健康子 agent 能消费统一验证工单和统一不变量视图，而不是依赖额外的人肉上下文解释。
7. 任务系统已经能以通用 agent/task 连接方式配置健康管理 agent 的任务身份、灵魂投影、workflow 和 binding，但不能承接健康系统管理职责。
8. 任务系统、健康系统、编排系统的边界对健康管理 agent 来说已经稳定：任务系统负责通用 agent/task 连接，健康系统负责健康领域对象和用户直操/agent 代操，编排系统负责能力边界与执行真相。

## 十、建议新增回归矩阵

建议至少补以下回归：

- `runtime_fault_injection_regression.py`
  - 聚焦 checkpoint / replay / permission / retry / partial completion
- `runtime_cutover_readiness_regression.py`
  - 聚焦 readiness report 和 rollback gate
- `runtime_recovery_invariant_regression.py`
  - 聚焦 ledger / checkpoint / execution summary / commit state 一致性
- `runtime_worker_degradation_regression.py`
  - 聚焦 delayed worker / stale backlog / timeout

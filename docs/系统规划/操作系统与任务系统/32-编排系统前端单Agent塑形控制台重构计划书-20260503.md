## 一、计划目标

本计划书只解决一件事：

把编排系统前端从“实验台、操作目录、系统配置页分散承载编排信息”，重构为“单 agent 身体与大脑塑形控制台”。

这里的“身体与大脑”有明确定义：

- 身体：capability profile、runtime lane、operation scope、resource policy、审批模式、写入边界
- 大脑：projection、skill workflow、prompt exposure、context section、execution directive、plan mode

这个页面负责解释“一个 agent 被塑造成什么样”，而不是决定“它在系统里承担什么社会角色”。

## 二、总分关系

必须坚持以下关系：

- `31` 管 agent 社会结构和任务连接中枢
- `32` 管单 agent 的身体能力与执行脑回路

任务系统回答：

- 这个 agent 在组织里负责什么

编排系统回答：

- 这个 agent 被允许怎样执行
- 为什么能执行
- 为什么不能执行
- 拒绝以后留下了什么证据

如果编排系统页面开始编辑 task family / topology，它就越界了。
如果任务系统页面开始编辑 capability / resource policy，它也越界了。

## 三、当前代码对照

这说明数据并不缺，缺的是“单 agent 视角的收口”。

### 3.2 当前问题不是功能少，而是观察视角错位

现在的编排相关页面主要从这些视角出发：

- 一次实验的编排快照
- 一套工具/skill 目录
- 一组全局系统配置

但缺少一个关键前端视角：

- “我选中某个 agent 后，能一次性看到它的身体、大脑、边界、拒绝和恢复语义”

### 3.3 当前页面结构与 `29` 的目标不完全一致

按照 `29`，编排系统要呈现的是：

- capability profile
- runtime lane
- resource policy
- admission
- blocked reasons
- traceability
- replay / retry / denied

而当前页面更偏：

- 编排实验复盘
- 工具目录观察
- 配置项编辑

这两者都重要，但不能互相替代。

## 四、目标定位

### 4.1 编排系统前端是“单 agent 塑形控制台”

目标页面要围绕 `agent_id` 组织，而不是围绕“实验 run”或“工具目录”组织。

建议核心模型：

```text
Agent Selector
  -> Capability Profile
      -> Runtime Lanes
          -> Operation Scope
              -> Projection / Workflow / Context Brain
                  -> Admission / Deny / Replay / Trace Evidence
```

### 4.2 页面必须回答六个问题

1. 这个 agent 被允许执行哪些任务模式
2. 它能走哪些 runtime lane
3. 它能使用哪些 operation，哪些被 blocked
4. 它采用哪些 projection / workflow / context section
5. 当它被拒绝时，拒绝发生在哪一层
6. 一次执行如何追溯到 binding / policy / trace / receipt

### 4.3 页面必须让“拒绝”变成一等对象

现在系统后端已经有：

- admission
- blocked reasons
- receipt
- trace report

前端不能只展示“allowed profile”，而要能展示：

- deny path
- fail-closed 结果
- retry / replay 的边界
- 为什么某个命令被挡住

否则编排系统仍旧只是“允许运行时目录”，不是治理控制台。

## 五、目标信息架构

### 5.1 一级页面结构

建议把编排系统前端重构为六个一级区块：

1. `Agent 塑形总览`
   - 当前 agent 的 profile type
   - enabled / disabled
   - allowed task modes 数
   - allowed runtime lanes 数
   - allowed / blocked operations 数
   - projection / workflow 绑定情况

2. `身体配置`
   - capability profile
   - runtime lane
   - operation scope
   - approval policy
   - memory scope
   - output contract

3. `大脑配置`
   - projection template
   - skill workflow
   - prompt exposure
   - context section
   - execution directive 语义

4. `门禁与拒绝`
   - admission 结果
   - blocked reasons
   - denied command receipts
   - fail-closed 路径

5. `追踪与恢复`
   - trace report
   - replay / retry / denied
   - checkpoint / execution record 引用
   - stale backlog / retry exhausted 信号

6. `全局编排治理`
   - permission mode
   - orchestration plan mode
   - runtime console 中与编排直接相关的配置
   - 但以只读/跳转方式整合，不把全局配置页吞并

### 5.2 单 agent 详情结构

每个 agent 的详情建议分为：

- `任务承接前提`
  - 来自任务系统的 binding 候选摘要
- `能力边界`
  - allowed / blocked operations
- `执行脑回路`
  - projection / workflow / context section
- `执行约束`
  - runtime lane / policy / approval
- `拒绝历史`
  - admission 失败 / blocked receipt
- `恢复与重放`
  - replay / retry / restore 证据

## 六、交互边界

### 6.1 编排系统前端允许做什么

- 选择 agent
- 查看 capability / lane / workflow / projection / policy
- 查看 admission 结果和 blocked reasons
- 查看 trace / replay / retry 相关证据
- 调整编排层允许调整的配置
  - permission mode
  - orchestration plan mode
  - capability profile 编辑
  - runtime governance 相关配置

### 6.2 编排系统前端不允许做什么

- 不定义 task family
- 不编辑 coordination topology
- 不定义哪个 agent 负责哪个社会角色
- 不直接创建 health issue / report / test
- 不把领域设置直接映射成权限改写

也就是说：

它塑造 agent 个体，但不组织 agent 社会。

## 七、实施方案

### Phase 32A：建立独立 Agent 编排控制台

新增或重构一个编排前端视图，使其不再寄生于单一实验视角。

建议：

- 保留 `ExperimentsView` 的实验复盘职责
- 新增独立 `OrchestrationControlView`
- 或将 `ExperimentsView` 中与 agent 塑形直接相关的部分拆出成专门控制台

### Phase 32B：按 agent 视角收口数据

整合以下接口数据：

- `/operations/agents`
- `/operations/agents/{agent_id}`
- `/operations/agents/{agent_id}/capability-profile`
- `/orchestration/catalog`
- `/config/permission-mode`
- `/config/runtime-console`

必要时补充：

- `agent -> latest admission receipts`
- `agent -> denied command history`
- `agent -> runtime trace refs`

### Phase 32C：把 admission / deny 做成一等 UI

前端要能直接看到：

- 最近哪些 command 被挡住
- 被哪一层挡住
- `blocked_reasons`
- 是否生成 receipt
- 是否留下 trace / report / execution evidence

而不是只能从健康系统间接看见。

### Phase 32D：把身体和大脑拆开展示

页面上明确分开：

- 身体配置
  - operation
  - runtime lane
  - policy
  - approval
  - memory write boundary

- 大脑配置
  - projection
  - workflow
  - prompt exposure
  - context section
  - answer/output contract

这是整个页面最重要的结构分割。

### Phase 32E：接入恢复与重放语义

把问题一、问题二的成果真正可视化：

- replay allowed / denied
- retry exhausted
- checkpoint restore
- execution receipt
- loop terminal reason

否则编排系统前端仍然看不到 durable runtime 的真实价值。

## 八、实施顺序

建议顺序：

1. 先完成 `31` 的任务系统中枢前端收口
2. 再实施 `32A / 32B / 32C`
3. 然后补 `32D / 32E`
4. 最后做一次 `31 / 32 / 27 / 28 / 29` 联调

原因是：

- 先明确 agent 在组织中的角色
- 再明确 agent 作为个体如何被塑形

## 九、验收标准

- 编排系统前端能按 `agent_id` 观察完整运行边界
- 页面能区分“身体配置”和“大脑配置”
- admission / blocked reasons / denied receipts 成为显式 UI
- 页面能解释单 agent 为什么能执行、为什么不能执行
- 页面能看到 replay / retry / restore 的运行证据
- 页面不承担 task family / topology 的组织职责

## 十、风险与控制

### 10.1 风险

- 如果继续只依赖 `ExperimentsView`，编排系统会长期停留在“实验复盘台”
- 如果把 operations / config / orchestration 三块继续散着放，用户看不到单 agent 的完整身体
- 如果不显式展示 deny path，编排治理会继续隐藏在日志里

### 10.2 控制策略

- 建立单独的 agent 编排控制台
- 以 `agent_id` 而不是 `run_id` 为页面主轴
- 把 deny / blocked / replay 当成一等对象
- 只把全局配置整合进来，不吞并任务系统和健康系统的领域职责

## 十一、交付物

- 新的编排系统前端控制台组件
- 清晰的 agent 编排类型定义与 API 收口
- admission / deny / replay 可视化面板
- 编排系统前端回归：
  - 能显示单 agent 身体/大脑配置
  - 能显示 blocked reasons
  - 能显示 denied receipt
  - 能显示 runtime lane / policy / projection / workflow
  - 不编辑任务组织层对象

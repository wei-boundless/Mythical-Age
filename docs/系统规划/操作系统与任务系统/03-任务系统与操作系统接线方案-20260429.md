# 任务系统与操作系统接线方案

日期：2026-04-29  
定位：在 `01-任务系统重构实施计划-20260429.md` 与 `02-操作系统重构实施计划-20260429.md` 之间，锁定任务契约、资源需求、资源授权、模型可见摘要、灵魂投影和控制内核之间的接口边界。  
当前目标：先完成 preview-only 接线，不接真实工具、worker、文件写入和 memory 写入。

---

## 0. 结论

任务系统和操作系统的接线，不是把工具列表塞进任务，也不是让灵魂投影决定可用能力，而是建立一条明确的资源授权链：

```text
UserRequest
  -> TaskContract
  -> TaskDefinition
  -> TaskBinding
  -> SkillRuntimeView
  -> OperationRequirement
  -> OperationRegistry / CapabilityManifest
  -> ResourcePolicyPreview
  -> ResourceRuntimeView
  -> TaskPromptContract.resource_section
  -> SoulToolView / SoulRuntimeView
  -> PromptManifestPreview
  -> ControlKernel diagnostics
```

一句话：

```text
任务系统提出“本任务可能需要什么资源”。
操作系统裁决“本任务实际能看见什么资源边界”。
灵魂系统只呈现“已经裁剪后的资源摘要”。
控制内核只记录 preview 诊断，不生成真实 RuntimeDirective。
```

第一阶段最终状态：

```text
可以生成 TaskPromptContract。
可以生成 ResourcePolicyPreview。
可以把 ResourceRuntimeView 接入 resource_section。
可以让 SoulRuntimeView 呈现资源边界。
可以让 ControlKernel 看到资源诊断。
不允许任何真实执行和写回。
```

## 1. 接线目标与非目标

### 1.0 源码对照后的硬约束

对照本地 Claude Code 源码：

```text
D:\AI应用\claude-code-nb-main\Tool.ts
D:\AI应用\claude-code-nb-main\tools.ts
D:\AI应用\claude-code-nb-main\utils\permissions\permissions.ts
D:\AI应用\claude-code-nb-main\utils\permissions\pathValidation.ts
D:\AI应用\claude-code-nb-main\tools\AgentTool\loadAgentsDir.ts
D:\AI应用\claude-code-nb-main\tools\AgentTool\runAgent.ts
D:\AI应用\claude-code-nb-main\Task.ts
D:\AI应用\claude-code-nb-main\utils\task\framework.ts
D:\AI应用\claude-code-nb-main\constants\prompts.ts
D:\AI应用\claude-code-nb-main\constants\systemPromptSections.ts
```

第 03 号接线方案必须补上下面这些真实工程约束：

```text
1. Tool 是带 input schema、output schema、权限检查、只读/破坏性/并发安全声明、结果序列化的微服务。
2. 工具默认 isReadOnly=false、isConcurrencySafe=false；缺声明时必须按高风险或不可并发处理。
3. getAllBaseTools 是工具存在事实入口；OperationRegistry 只能包装能力事实，不能变成授权事实。
4. 权限不是单个 allow 布尔值，而是 deny / ask / allow、mode、hook、classifier、headless fallback 的组合管线。
5. 文件路径必须在执行前重新验证，防止 plan 阶段与执行阶段之间出现 symlink、shell expansion、glob 写入等 TOCTOU 风险。
6. Agent frontmatter 中的 tools、permissionMode、memory、mcpServers、hooks 只是配置输入，不是资源授权事实。
7. 后台 / headless agent 无法弹出确认时，requires_approval 必须转成 hook 决策、bubble 到父层，或 fail-closed deny。
8. TaskState 有 pending/running/completed/failed/killed 和 notified 语义；执行型任务必须有生命周期记录，preview 任务不能伪装成 running。
9. Prompt section 有 cacheBreak 与动态边界；ResourcePolicy / ResourceRuntimeView 属于动态段，不能污染静态 prompt cache。
```

这些约束会直接影响后续代码字段和测试断言。

### 1.1 目标

```text
1. 把 TaskBinding.operation_scope 转成 OperationRequirement。
2. 把 OperationRequirement 转成 ResourcePolicyPreview。
3. 把 ResourcePolicyPreview 转成 ResourceRuntimeView。
4. 把 ResourceRuntimeView 写入 TaskPromptContract.resource_section。
5. 把 resource section 映射到 SoulToolView / SoulRuntimeView。
6. 把 ResourcePolicyPreview 作为 diagnostics 交给 ControlKernel。
7. 保持全链路 preview_only / fail_closed。
```

### 1.2 非目标

```text
不真实调用 op.web_search。
不真实读取文件。
不真实编辑文件。
不真实启动 worker。
不真实启动 bounded agent。
不真实写 session message。
不真实写 durable memory。
不把 preview policy 转成 RuntimeDirective。
```

### 1.3 总边界

```text
TaskBinding 是配置。
OperationRequirement 是候选需求。
ResourcePolicyPreview 是资源授权预览事实。
AdoptedResourcePolicy 才是 ControlKernel 采纳后的执行授权事实。
ResourceRuntimeView 是模型可见摘要。
SoulToolView 是投影后的呈现摘要。
RuntimeDirective 才是真实执行指令。
CommitCandidate 才是写回候选。
```

禁止把这些层混用：

```text
TaskBinding.operation_scope != allowed_operations
OperationRequirement != ResourcePolicy
ResourcePolicyPreview != AdoptedResourcePolicy
ResourceRuntimeView.authorized != runtime_executable
SoulToolView.visible != tool execution permission
PromptManifest section != execution state
```

最关键的区分：

```text
preview_available:
  模型可见，表示“如果后续被 ControlKernel 采纳，理论上可用”。

runtime_authorized:
  ControlKernel 已采纳 ResourcePolicy，ExecutionGraph / RuntimeDirective 已引用，且 OperationGate 执行前复核通过。

runtime_executable:
  当前执行节点已经具备 directive、policy、input、approval、path/network/shell/memory scope 的全部条件。
```

第一阶段只允许出现：

```text
preview_available
```

不允许出现：

```text
runtime_authorized
runtime_executable
```

## 2. 接线对象职责

### 2.1 TaskContract

任务根事实。

```text
owner:
  TaskSystem

输入来源:
  UserRequest
  current session context

输出给:
  TaskDefinition resolver
  TaskBinding resolver
  OperationRequirement builder
  ControlKernel diagnostics
```

约束：

```text
TaskContract.authority == "task_contract"
TaskContract 不保存工具实现。
TaskContract 不保存最终授权。
TaskContract 不触发执行。
```

### 2.2 TaskDefinition

任务静态定义。

```text
owner:
  TaskSystem

提供:
  default_operation_requirements
  default_skill_refs
  default_projection_role
  output_contract
  review_policy
```

约束：

```text
TaskDefinition 可以声明“这类任务通常需要什么资源”。
TaskDefinition 不能声明“本轮已经允许什么资源”。
```

### 2.3 TaskBinding

任务配置。

```text
owner:
  TaskSystem

关键字段:
  operation_scope
  denied_operations
  approval_policy
  projection_selector
  skill_scope
  denied_skills
  memory_scope
  review_policy
```

约束：

```text
operation_scope 是需求范围。
denied_operations 是配置层硬限制。
approval_policy 是审批策略提示。
最终授权仍归 ResourcePolicy。
```

### 2.4 SkillRuntimeView

当前任务可用方法摘要。

```text
owner:
  SkillSystem / TaskSystem bridge

提供:
  method_summary
  input_boundary
  output_boundary
  required_operations
  forbidden_uses
```

约束：

```text
SkillRuntimeView.required_operations 只能补充 OperationRequirement。
SkillRuntimeView 不得直接授予工具权限。
```

### 2.5 OperationRequirement

任务提出的资源需求候选。

```text
owner:
  OperationSystem

输入来源:
  TaskDefinition.default_operation_requirements
  TaskBinding.operation_scope
  TaskBinding.denied_operations
  SkillRuntimeView.required_operations
  explicit user constraints
```

必须保持：

```text
OperationRequirement.authority == "candidate_only"
```

### 2.6 ResourcePolicyPreview

资源授权预览事实。

```text
owner:
  OperationSystem

输入来源:
  OperationRequirement
  OperationRegistry
  CapabilityManifest
  task binding policy
  global safety policy
```

必须保持：

```text
ResourcePolicy.authority == "resource_policy"
ResourcePolicyPreview.preview_only == true
ResourcePolicyPreview.adopted == false
unknown operation -> deny
denied operation wins over allowed
high risk operation -> requires_approval
```

注意：

```text
ResourcePolicyPreview 只能说明“当前资源策略预览会如何裁决”。
它不能被 ToolExecutor / WorkerExecutor / BoundedAgentExecutor 消费。
它不能写入 RuntimeDirective。
它不能被 PromptManifest 当作执行状态保存。
```

后续真正执行前必须经过：

```text
ResourcePolicyPreview
  -> ControlKernel adoption
  -> AdoptedResourcePolicy
  -> ExecutionGraph.policy_refs
  -> RuntimeDirective.allowed_resource_ref
  -> OperationGate final check
```

### 2.7 ResourceRuntimeView

模型可见资源摘要。

```text
owner:
  OperationSystem

派生自:
  ResourcePolicyPreview
  ResourceDecision
  OperationDescriptor
```

必须保持：

```text
ResourceRuntimeView.authorization_owner == "ResourcePolicy"
ResourceRuntimeView 不进入 ExecutionRuntime。
ResourceRuntimeView 只进入 TaskPromptContract.resource_section。
ResourceRuntimeView.authorized 在第一阶段只表示 preview_available，不表示 runtime_executable。
```

字段建议：

```text
policy_decision:
  allow_preview / deny / requires_approval / unknown

preview_available:
  是否允许在模型可见摘要中呈现为可用资源。

runtime_executable:
  第一阶段固定 false。
```

### 2.8 TaskPromptContract

模型可见任务契约。

```text
owner:
  TaskSystem

resource_section 输入:
  ResourceRuntimeView[]
  ResourcePolicySummary
  OperationDiagnostics
```

约束：

```text
TaskPromptContract 是给模型看的运行契约。
TaskPromptContract 不拥有工具授权事实。
```

### 2.9 SoulToolView / SoulRuntimeView

投影后的资源呈现。

```text
owner:
  ProjectionSystem

输入来源:
  TaskPromptContract.resource_section
  ResourceRuntimeView[]
```

必须保持：

```text
SoulToolView.authorization_owner == "ResourcePolicy"
SoulRuntimeView.authorization_owner == "ResourcePolicy"
Projection 不得新增 operation id。
Projection 不得把 deny / requires_approval 改成 allow。
```

### 2.10 ControlKernel diagnostics

控制内核预览诊断。

```text
owner:
  ControlKernel

接收:
  TaskContract
  ResourcePolicyPreview
  TaskPromptContract
  PromptManifestPreview
```

第一阶段：

```text
只记录 diagnostics。
不生成真实 RuntimeDirective。
ExecutionGraph 只能是 preview / blocked。
```

必须额外记录：

```text
resource_policy_state = "preview"
resource_policy_adopted = false
runtime_executable = false
operation_gate_required_before_execution = true
```

## 2A. Claude Code 源码映射表

| Claude Code 源码机制 | 本方案对象 | 接线要求 |
| --- | --- | --- |
| `Tool.ts` 的 inputSchema / outputSchema | OperationDescriptor.input_contract / output_contract | OperationRegistry 必须保留 schema，不允许只登记名字 |
| `Tool.ts` 的 isReadOnly / isConcurrencySafe / isDestructive | OperationDescriptor.risk_tags / read_only / concurrency_safe / destructive | 缺声明时按非只读、非并发安全处理 |
| `tools.ts` 的 `getAllBaseTools()` | OperationRegistry / CapabilityManifest | 能力存在事实单一入口 |
| `permissions.ts` 的 deny / ask / allow / mode / hook / classifier | ResourcePolicy / OperationGate | 授权必须是管线结果，不是单个布尔值 |
| `pathValidation.ts` 的路径安全与 TOCTOU 检查 | OperationGate filesystem scope | 真实执行前必须重校验路径 |
| `loadAgentsDir.ts` 的 tools / disallowedTools / permissionMode / memory / mcpServers | AgentProfile / TaskBinding / ResourcePolicy input | frontmatter 是配置输入，不是授权事实 |
| `runAgent.ts` 的 shouldAvoidPermissionPrompts | ApprovalPolicy / OperationGate | headless requires_approval 必须 fail-closed 或走 hook / bubble |
| `Task.ts` 的 TaskStatus / notified | TaskRecord / TaskStageRecord | preview 不得伪装成 running；真实执行必须记录生命周期 |
| `systemPromptSections.ts` 的 cacheBreak | PromptManifest.cache_scope | resource section 属于动态段，必须可追踪来源 |

## 3. Task -> Operation 输入转换

### 3.1 输入合并顺序

OperationRequirement builder 按下面顺序收集资源需求：

```text
TaskDefinition.default_operation_requirements
  + TaskBinding.operation_scope
  + SkillRuntimeView.required_operations
  + CompositePlan stage requirements
  + explicit user constraints
  - TaskBinding.denied_operations
```

注意：

```text
denied_operations 不是真的从需求里删除。
denied_operations 必须保留到 OperationRequirement.denied_operations。
这样 ResourcePolicyPreview 能解释为什么被拒绝。
```

### 3.2 转换规则

```text
default_operation_requirements
  -> required_operations

operation_scope
  -> required_operations 或 optional_operations

SkillRuntimeView.required_operations
  -> optional_operations，除非 TaskDefinition 标记为 required

TaskBinding.denied_operations
  -> denied_operations

approval_policy
  -> metadata.approval_policy

review_policy
  -> metadata.review_policy
```

### 3.3 生成样例

输入：

```text
TaskDefinition:
  task.local_material_read
  default_operation_requirements:
    op.read_file
    op.search_files

TaskBinding:
  operation_scope:
    op.read_file
    op.search_text
  denied_operations:
    op.write_file
    op.shell

SkillRuntimeView:
  required_operations:
    op.search_text
```

输出：

```text
OperationRequirement:
  required_operations:
    op.read_file
    op.search_files
    op.search_text
  denied_operations:
    op.write_file
    op.shell
  authority:
    candidate_only
```

## 4. Operation -> ResourcePolicyPreview

### 4.1 Policy builder 输入

```text
OperationRequirement
OperationRegistry
CapabilityManifest
TaskBinding.approval_policy
global safety policy
session policy
project policy
execution_environment
ui_capability
headless_mode
```

### 4.2 判定顺序

必须固定为：

```text
1. operation 是否存在。
2. operation 是否在 explicit denied_operations。
3. operation 是否被全局 safety policy 拒绝。
4. operation 是否超出 task / session / project scope。
5. operation 是否高风险。
6. operation 是否需要用户确认。
7. 当前运行环境是否能处理用户确认。
8. operation 是否可在 preview 中展示为 available。
```

优先级：

```text
explicit deny
  > safety deny
  > missing policy deny
  > requires approval
  > allow preview
```

### 4.3 决策输出

每个 operation 生成一个 `ResourceDecision`：

```text
allow:
  可进入 ResourceRuntimeView.preview_available = true
  第一阶段仍必须 preview_only

requires_approval:
  可进入 ResourceRuntimeView.requires_approval = true
  不可进入 RuntimeDirective
  headless / async agent 场景必须降级为 hook / bubble / deny

deny:
  可进入 ResourceRuntimeView.denied_reason
  不可显示为可用

unknown:
  执行语义等价 deny
```

### 4.4 固定约束

```text
unknown operation -> deny
denied operation wins over allowed
write_file / edit_file 第一阶段 requires_approval 或 deny
shell / python_repl 第一阶段 requires_approval 或 deny
memory_write_candidate 第一阶段默认 deny
worker / agent 第一阶段 preview_only
```

### 4.5 Headless 审批规则

Claude Code 源码中，后台 agent 无法直接展示权限确认时，会设置避免权限弹窗的上下文，并通过 hook、bubble 或自动拒绝处理。迁移到本系统：

```text
requires_approval + interactive_ui_available:
  decision = requires_approval

requires_approval + headless + approval_hook_available:
  decision = requires_approval
  approval_channel = hook

requires_approval + headless + bubble_to_parent_allowed:
  decision = requires_approval
  approval_channel = parent

requires_approval + headless + no approval channel:
  decision = deny
  reason = "approval unavailable in headless context"
```

禁止：

```text
后台 worker / agent 因为没有 UI 确认能力而自动 allow。
requires_approval 在 headless 场景静默进入 RuntimeDirective。
```

## 5. ResourcePolicyPreview -> ResourceRuntimeView

### 5.1 映射规则

```text
ResourceDecision.allow
  -> authorized = true
  -> preview_available = true
  -> runtime_executable = false
  -> requires_approval = false
  -> denied_reason = ""
  -> preview_only = true

ResourceDecision.requires_approval
  -> authorized = false
  -> preview_available = false
  -> runtime_executable = false
  -> requires_approval = true
  -> denied_reason = "requires approval"
  -> preview_only = true

ResourceDecision.deny
  -> authorized = false
  -> preview_available = false
  -> runtime_executable = false
  -> requires_approval = false
  -> denied_reason = decision.reason
  -> preview_only = true

ResourceDecision.unknown
  -> authorized = false
  -> preview_available = false
  -> runtime_executable = false
  -> requires_approval = false
  -> denied_reason = "unknown operation"
  -> preview_only = true
```

### 5.2 resource_section 组装

`TaskPromptContract.resource_section` 不应列出全量 OperationRegistry，只列出本任务相关的资源边界：

```text
available:
  当前任务可在 preview 中使用的 read-only / low-risk resource。

requires_approval:
  当前任务可能需要，但真实执行前必须确认的 resource。

denied:
  当前任务明确不可使用，或因风险 / policy 被拒绝的 resource。

notes:
  preview_only 状态、风险摘要、失败处理。
```

示例：

```text
Resource boundary:
- Available in preview: op.read_file, op.search_text.
- Requires approval before real execution: op.edit_file.
- Denied: op.shell, op.python_repl, op.memory_write_candidate.
- This preview does not grant runtime execution permission.
```

### 5.3 Tool contract 摘要规则

ResourceRuntimeView 来自工具或 worker 时，必须保留 Claude Code 工具系统中的核心契约信息：

```text
operation_id
operation_type
input_contract_ref
output_contract_ref
read_only
concurrency_safe
destructive
risk_tags
result_serialization
permission_check_required
```

默认值：

```text
read_only 缺失 -> false
concurrency_safe 缺失 -> false
destructive 缺失 -> unknown_or_false_but_risk_tagged
permission_check_required 缺失 -> true
```

模型可见摘要可以压缩这些字段，但 PromptManifest 必须能追踪它们来自哪个 OperationDescriptor。

## 6. Operation -> SoulProjection 映射

### 6.1 映射链

```text
ResourceRuntimeView[]
  -> SoulToolView[]
  -> SoulRuntimeView.resource_summary
  -> PromptManifest.resource_section
```

### 6.2 字段映射

```text
ResourceRuntimeView.resource_id
  -> SoulToolView.tool_id

ResourceRuntimeView.title
  -> SoulToolView.title

ResourceRuntimeView.capability_summary
  -> SoulToolView.summary

ResourceRuntimeView.authorized
  -> SoulToolView.authorized

ResourceRuntimeView.preview_available
  -> SoulToolView.preview_available

ResourceRuntimeView.runtime_executable
  -> SoulToolView.runtime_executable

ResourceRuntimeView.requires_approval
  -> SoulToolView.requires_approval

ResourceRuntimeView.denied_reason
  -> SoulToolView.denied_reason

ResourceRuntimeView.authorization_owner
  -> SoulToolView.authorization_owner
```

固定值：

```text
SoulToolView.authorization_owner = "ResourcePolicy"
SoulToolView.runtime_executable = false in phase 1
```

### 6.3 Projection 允许做什么

```text
压缩资源说明。
按当前投影调整表达密度。
强调风险和审批要求。
把 denied resource 解释成用户可理解的限制。
把 requires_approval resource 表述为“需要确认后才能执行”。
```

### 6.4 Projection 禁止做什么

```text
不得新增 operation id。
不得隐藏 denied reason。
不得把 requires_approval 改成 authorized。
不得把 denied 改成 authorized。
不得生成 RuntimeDirective。
不得声明 memory write allowed。
不得直接引用 OperationRegistry 全量内容。
不得改变 PromptManifest.cache_scope。
```

## 7. 接入 ControlKernel 与 ExecutionGraph Preview

### 7.1 第一阶段接入点

ControlKernel 接收：

```text
TaskContract
OperationRequirement
ResourcePolicyPreview
TaskPromptContract
PromptManifestPreview
```

生成：

```text
ControlKernelDiagnostics:
  task_id
  resource_policy_ref
  resource_policy_state = "preview"
  resource_policy_adopted = false
  denied_operations
  requires_approval_operations
  preview_only = true
  fail_closed = true
  runtime_directive_enabled = false
  runtime_executable = false
  operation_gate_required_before_execution = true
```

### 7.2 ExecutionGraph preview

第一阶段可生成空图或预览图：

```text
ExecutionGraphPreview:
  graph_id
  task_id
  nodes = []
  blocked_reason = "preview_only"
  resource_policy_ref
```

或：

```text
ExecutionNodePreview:
  node_id
  node_type = "preview"
  operation_refs
  policy_refs
  executable = false
  blocked_reason = "preview_only"
```

### 7.3 绝对禁止

```text
ResourcePolicyPreview -> RuntimeDirective
ResourceRuntimeView -> RuntimeDirective
SoulToolView -> RuntimeDirective
PromptManifestPreview -> RuntimeDirective
ExecutionGraphPreview -> ToolExecutor
ExecutionGraphPreview -> WorkerExecutor
```

### 7.4 执行前复核要求

即使后续进入真实执行阶段，也不能把 plan 阶段的授权当作永久授权。

真实执行前必须重新检查：

```text
operation still exists in OperationRegistry
operation descriptor hash still matches policy snapshot
input schema validation passes
resource policy is adopted
operation is not denied by current policy
approval is still valid
filesystem path is re-resolved and revalidated
network scope is still valid
shell / powershell / python command risk is still valid
agent / worker owner is still valid
session mode still permits execution
```

文件系统特别规则：

```text
写入路径不能使用 glob。
路径中不能含 shell expansion 语法后再被当作已验证路径执行。
symlink / realpath 必须在执行前重新确认。
workspace 内 read 与 write 不是同一种授权。
```

## 8. 三个贯穿验收用例

### 8.1 联网搜索官方资料

用户请求：

```text
帮我联网搜索 Claude Code subagent 官方资料。
```

任务系统：

```text
TaskDefinition:
  task.information_search

TaskBinding.operation_scope:
  op.web_search
  op.fetch_url

TaskBinding.denied_operations:
  op.write_file
  op.edit_file
  op.shell
  op.python_repl
```

操作系统：

```text
OperationRequirement:
  required_operations:
    op.web_search
    op.fetch_url
  denied_operations:
    op.write_file
    op.edit_file
    op.shell
    op.python_repl
  authority:
    candidate_only

ResourcePolicyPreview:
  allowed preview:
    op.web_search
    op.fetch_url
  denied:
    op.write_file
    op.edit_file
    op.shell
    op.python_repl
```

投影：

```text
Projection posture:
  evidence-first
  traceability
  source-aware
```

验收：

```text
TaskPromptContract.resource_section 包含 web search / fetch url。
ResourceRuntimeView.authorization_owner == "ResourcePolicy"。
PromptManifestPreview 记录 resource section 来源。
ControlKernel diagnostics.preview_only == true。
不生成真实 web search RuntimeDirective。
```

## 10. 测试与验收标准

### 10.1 必测断言

```text
TaskContract.authority == "task_contract"
OperationRequirement.authority == "candidate_only"
ResourcePolicy.authority == "resource_policy"
ResourcePolicyPreview.preview_only == true
ResourcePolicyPreview.adopted == false
ResourceRuntimeView.authorization_owner == "ResourcePolicy"
ResourceRuntimeView.runtime_executable == false
SoulToolView.authorization_owner == "ResourcePolicy"
SoulToolView.runtime_executable == false
unknown operation -> deny
denied operation wins over allowed
high risk operation -> requires_approval
preview_only does not enter RuntimeDirective
CommitCandidate default denied
```

### 10.2 防回退断言

必须防止旧链路复活：

```text
QueryRuntime 不得从 query_understanding.tool_name 直接执行工具。
TaskBinding.operation_scope 不得直接变成 allowed tools。
SkillRuntimeView.required_operations 不得直接变成 allowed tools。
Projection 不得生成 visible_tool_ids 的新增授权。
PromptManifestPreview 不得保存 RuntimeDirective。
WorkerResult 不得直接写 final answer。
ResourcePolicyPreview 不得被 ToolExecutor 消费。
ResourceRuntimeView.authorized 不得被解释为 runtime_executable。
Agent frontmatter tools / permissionMode / mcpServers 不得绕过 ResourcePolicy。
```

### 10.3 快照验收

三个贯穿用例应保存 preview snapshot：

```text
search_official_material.preview.json
local_read_and_summarize.preview.json
modify_then_review.preview.json
```

每个 snapshot 至少包含：

```text
task_contract
operation_requirement
resource_policy_preview
resource_runtime_views
task_prompt_contract
soul_runtime_view
prompt_manifest_preview
control_kernel_diagnostics
```

### 10.4 源码对照验收

每个 preview snapshot 额外检查：

```text
operation_descriptor 中存在 input_contract_ref / output_contract_ref。
工具缺 read_only 声明时，不得默认当作只读。
工具缺 concurrency_safe 声明时，不得默认并发。
requires_approval 在 headless 场景有明确 approval_channel 或 deny reason。
resource section 的 PromptManifest.cache_scope 为 dynamic / turn。
resource section 记录 owner_layer = ResourcePolicy。
filesystem operation 记录 execution_time_revalidation_required = true。
agent / worker operation 记录 isolation_required = true。
```

## 11. 当前定稿口径

```text
TaskSystem
  管任务契约和任务配置。

OperationSystem
  管资源需求、资源授权、资源摘要和执行前门控。

ProjectionSystem
  管模型可见呈现，不管授权。

ControlKernel
  管采纳和执行真相，第一阶段只记录 preview diagnostics。

ExecutionRuntime
  第一阶段不接入。

CommitGate
  第一阶段不写回，只保持候选默认 denied 的规则。
```

最短口径：

```text
任务系统只提需求。
操作系统才裁边界。
灵魂系统只做呈现。
控制内核暂不放行。
```

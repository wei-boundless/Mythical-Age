# 问题三：模板选择与任务拆分契约化差距报告

日期：2026-05-02

## 一、问题定义

当前任务系统已经开始从 `TaskDefinition` 过渡到 `TaskTemplate` 和 `TaskSpec`，这是正确方向。但模板选择和复合任务拆分仍主要依赖启发式推断，而不是显式任务契约。

真正损坏的系统属性不是“关键词不够多”，而是：

- 父任务与子任务之间没有统一 contract；
- 模板选择仍受自然语言表述方式强影响；
- bundle item 的输入输出和共享状态边界尚未固化；
- 前端将来即使能配置模板，底层仍可能被启发式绕开。

## 二、当前代码证据

核心文件：

- `backend/tasks/template_registry.py`
- `backend/context_management/resolver.py`

当前已经具备：

- `TaskTemplate`、`TaskStepBlueprint`、`TaskValidationRule` 已经落地。
- `select_template()` 能根据 understanding、current turn、definitions 选择模板。
- `_bundle_items()` 已能把复合任务拆成 bundle item。

当前仍然存在的结构问题：

- `select_template()` 依赖 `route_hint`、`execution_posture`、`preferred_skill`、`modality`、`source_kind`、关键词等信号。
- `_looks_compound()`、`_capability_for_text()` 使用自然语言特征推断子任务能力域。
- 当前没有独立的 template match contract、bundle item schema transform、step input binder。

这意味着目前的入口更像：

```text
用户话术
  -> 理解层启发式归类
  -> 选择模板
  -> 拆分 bundle
```

而不是：

```text
用户请求 + 显式输入 + 已绑定对象
  -> task intent contract
  -> template match
  -> bundle spec
  -> step input binding
```

## 三、与先进架构的差距

### 3.1 与 LangGraph Subgraphs 的差距

LangGraph subgraph 的重点不是“自动拆得多聪明”，而是：

- parent graph 和 subgraph 之间有显式 state schema；
- 状态映射是显式的；
- 是否共享状态、是否私有状态、是否每次 fresh invocation 都由结构决定。

当前差距：

- 我们的 bundle item 还没有稳定的 parent-child state contract。
- 子任务输入更多来自启发式文本拆分，而不是 schema 化的 invocation contract。

### 3.2 与成熟工作流模板系统的差距

成熟模板系统通常要求：

- 模板选择依据结构化输入和能力声明；
- 子任务由 blueprint 或 DAG 显式定义；
- 调用边界稳定，不受表达细节强烈波动。

当前差距：

- 现在相同意图只要用户换一种说法，可能走入不同模板。
- 同样的 bundle 请求换一种句式，也可能得到不同拆分结果。

## 四、会带来的具体问题

### 4.1 模板越多，误分类越快上升

当前模板数量还不大时，启发式容易看起来“足够好”。但随着：

- 开发模板变多；
- 数据和文档模板细分；
- 多 agent 模板增加；

模板间冲突会迅速增加。

### 4.2 前端装配会失去确定性

如果前端未来能配置模板，但后端仍能被启发式绕开，就会出现：

- UI 选择了 A 模板；
- 后端根据 wording 自动改走 B 模板；
- 用户和系统对“当前任务到底是什么”失去共同真相。

### 4.3 follow-up 绑定会继续脆弱

如果 bundle item 不是契约化实体，只是临时自然语言拆分结果，那么像：

- “展开第二个子任务”
- “复用刚才的第一个分析对象”
- “只重跑验证步骤”

这类 follow-up 仍然容易退回猜测。

## 五、目标设计

### 5.1 引入 template match contract

模板选择应优先依据以下结构化输入：

- task family
- source binding
- explicit input schema
- execution intent
- requested outputs
- permission posture

启发式只能作为 fallback，而不能继续作为主入口。

### 5.2 引入 bundle spec

建议新增稳定的 bundle 实体：

```python
class BundleSpec:
    bundle_id: str
    parent_task_id: str
    items: tuple[BundleItemSpec, ...]
    aggregation_policy: str
```

```python
class BundleItemSpec:
    item_id: str
    ordinal: int
    template_id: str
    inputs: dict
    bindings: dict
    requested_outputs: tuple[str, ...]
```

### 5.3 引入 step input binder

每个 `TaskStepBlueprint` 不应只声明操作需求，还应声明：

- 输入从哪里来；
- 输入是否允许继承 parent state；
- 输出写回 parent 的哪个 slot；
- 哪些状态是私有的，哪些是共享的。

## 六、分阶段改造建议

### Phase 3A：把模板选择从启发式迁移到结构化 match

目标：

- 为每个模板建立显式 match rule。
- 让 `CurrentTurnContext` 成为模板选择的主输入。

### Phase 3B：新增 bundle spec

目标：

- 让 bundle item 成为稳定可回指实体，而不是只存在于当前轮拆分结果里。

### Phase 3C：新增 step input/output binding 协议

目标：

- 每个 step 都能明确知道读取哪些输入、写出哪些输出、回写到哪里。

## 七、验收标准

- 同一请求在措辞轻微变化下仍能稳定命中同一模板。
- 复合任务拆分结果可被持久化和回指。
- “第二个子任务”“只重跑验证步骤”这类 follow-up 有稳定结构入口。
- 前端模板装配与后端模板执行之间不存在语义漂移。

# 问题二：恢复重放与副作用幂等性差距报告

日期：2026-05-02

## 一、问题定义

当前系统已经进入有 checkpoint、有 event log、有 trace summary 的阶段，但还没有真正形成“恢复后不会重复副作用”的运行时语义。

真正的结构性问题不是“少几条恢复逻辑”，而是：

- checkpoint 记录了状态，但没有统一副作用重放边界；
- mutating operation 还没有标准化的 idempotency contract；
- replay 起点、去重依据、重复执行容忍度尚未在运行时模型中固化。

因此当前恢复能力更接近“可回看”，还不是“可安全继续执行”。

## 二、当前代码证据

核心文件：

- `backend/orchestration/runtime_loop/checkpoint.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/orchestration/runtime_loop/trace_reader.py`

当前已经具备：

- loop 启动后会写 checkpoint。
- `task_contract_built`、`commit_gate_checked`、`loop_terminal` 等事件已经入 trace。
- `TaskResult`、`TaskRunLedger` 已经进入 done event、commit payload、checkpoint 和 trace。

当前仍然缺失：

- 没有统一的 `operation_execution_id` / `idempotency_token`。
- 没有为 `edit_file`、`write_file`、`shell`、worker 输出建立 replay-safe contract。
- 没有在恢复时显式区分“可重放步骤”和“不可重放副作用步骤”。

## 三、与先进架构的差距

### 3.1 与 LangGraph Durable Execution 的差距

LangGraph 对 durable execution 的核心要求包括：

- 使用 checkpointer 保存 workflow progress；
- 使用 thread id 区分具体运行实例；
- 将非确定性动作与副作用动作包进 task；
- 恢复时从持久层重取已完成 task 的结果，避免重复副作用。

当前差距：

- 我们有 checkpoint，但还没有把 mutating operation 包装成统一“已执行且可去重”的 task 语义。
- 我们能恢复账本，但还不能保证文件写入、shell 执行、外部请求不会重复发生。

### 3.2 与 AWS Reliability 的差距

AWS Well-Architected 对幂等和重试的要求非常明确：

- 变更类操作应支持幂等；
- 重试前要确认目标操作是否幂等；
- 应使用 token 区分新请求和重复请求；
- 失败场景下要 fail fast，限制重试和积压。

当前差距：

- 我们的运行时还没有把 mutating action 统一标记成幂等或非幂等。
- 重试、超时、权限拒绝、worker 中断等场景还没有统一的 retry boundary。

## 四、会带来的具体问题

### 4.1 crash-resume 会重复写文件

例如：

- step 已经执行 `edit_file`
- checkpoint 尚未或刚刚写入
- 进程中断
- 恢复时重新跑到同一步

如果没有去重 contract，就可能对同一文件重复修改。

### 4.2 shell / worker 行为会重复触发

对于 build、test、browser verify、脚本执行等动作：

- 有些可重复执行；
- 有些会污染环境；
- 有些带外部副作用。

没有 replay policy 时，系统无法区分这些动作应不应该再次触发。

### 4.3 retry 会扩大系统负担

如果某个 dependency 失败而客户端或运行时在多层都重试：

- 容易产生 retry storm；
- 容易放大资源争用；
- 容易把原本局部错误放大成全局不稳定。

### 4.4 checkpoint 价值被打折

没有副作用语义保护时，checkpoint 只能安全用于：

- 只读分析
- 纯模型回答
- 无外部变更的工具路径

这会严重限制任务系统向开发和执行类任务扩展。

## 五、目标设计

### 5.1 引入副作用分类

所有 operation 必须被分成至少三类：

- pure read
- replay-safe write
- non-replay-safe action

系统必须根据类型决定：

- 是否允许自动重试；
- 是否必须持有幂等 token；
- 是否只能在明确人工确认或特定 durability mode 下执行。

### 5.2 引入 operation execution contract

建议为每次副作用执行建立统一结构：

```python
class OperationExecutionRecord:
    execution_id: str
    task_run_id: str
    step_id: str
    operation_id: str
    request_fingerprint: str
    idempotency_token: str
    replay_policy: str
    result_ref: str
    status: str
```

### 5.3 引入恢复规则

恢复时必须明确：

- 读操作允许重放；
- replay-safe write 需优先检查已有 execution record；
- non-replay-safe action 默认 fail-closed，除非有明确恢复协议。

## 六、分阶段改造建议

### Phase 2A：为 mutating operation 加 execution record

目标：

- `edit_file`、`write_file`、`shell`、worker verify 先接入统一执行记录。

### Phase 2B：接入 idempotency token 与 request fingerprint

目标：

- 用稳定输入摘要和 step identity 形成去重键。
- 为支持幂等的执行器增加“重复请求返回原结果”的协议。

### Phase 2C：为恢复流程增加 replay policy

目标：

- 在 checkpoint 恢复时明确哪些动作可重放、哪些只可读回结果、哪些必须终止。

## 七、验收标准

- crash-resume 不会对同一副作用动作重复生效。
- 重试策略能区分幂等与非幂等动作。
- trace 中能回答某次副作用为什么没有重复执行。
- 运行时能明确说明某个失败是 fail-fast、retry exhausted、permission blocked 还是 replay denied。

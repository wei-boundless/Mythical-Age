# 问题二：恢复重放与副作用幂等性差距报告

日期：2026-05-02

配套计划书：

- `22-问题二-恢复重放与副作用执行语义重构计划书-20260502.md`

## 一、问题定义

当前系统已经进入有 checkpoint、有 event log、有 trace summary 的阶段，但还没有真正形成“恢复后不会重复副作用”的运行时语义。

这不是“少写几条恢复逻辑”的问题，而是一个更基础的结构缺口：

- checkpoint 记录了 loop 状态，但没有统一副作用执行账本；
- mutating operation 已经有风险元数据，但还没有被运行时消费成 replay-safe contract；
- recovery 已经有入口，但没有稳定的 replay boundary、去重键、结果回读协议。

因此当前恢复能力更接近“可回看”，还不是“可安全继续执行”。

## 二、当前代码证据

### 2.1 checkpoint 记录的是 loop state，不是副作用执行真相

核心文件：

- `backend/orchestration/runtime_loop/checkpoint.py`
- `backend/orchestration/runtime_loop/models.py`

当前 `RuntimeCheckpoint` 已经记录了：

- `loop_state`
- `event_offset`
- `approval_state`
- `commit_state`

但还没有记录：

- 已创建但未完成的副作用执行记录
- 某次执行的幂等 token / request fingerprint
- 某次恢复应该“重放”“回读结果”还是“拒绝继续”的 replay decision

这意味着恢复只能知道 loop 走到了哪一步，还不知道这一步之前的副作用是否已经实际触发。

### 2.2 action request 和 observation 之间缺少 execution record 层

核心文件：

- `backend/orchestration/runtime_loop/action_request.py`
- `backend/execution/tool_executor.py`

当前结构是：

```text
RuntimeActionRequest
  -> ToolRuntimeExecutor.run()
  -> RuntimeObservation
```

其中缺失的中间层是“这次副作用执行本身”的 durable record。

当前 `RuntimeActionRequest` 只有：

- `request_id`
- `operation_id`
- `payload`

当前 `RuntimeObservation` 只有：

- `request_ref`
- `directive_ref`
- `payload.result`

但没有：

- `execution_id`
- `idempotency_token`
- `request_fingerprint`
- `replay_policy`
- `execution_status`
- `side_effect_committed`

这会导致运行时无法回答“这次 observation 对应的是一次新执行，还是一次恢复后回读结果”。

### 2.3 operation registry 已经有关键元数据，但尚未进入恢复语义

核心文件：

- `backend/operations/registry.py`
- `backend/orchestration/runtime_loop/tool_adoption.py`

`OperationDescriptor` 已经定义了：

- `read_only`
- `destructive`
- `idempotent`
- `interrupt_behavior`
- `requires_approval_by_default`

这是很好的基础，说明系统并不是完全没有副作用分类能力。

但当前这些字段主要只用于：

- gate 决策
- adoption diagnostics
- UI / 预览能力说明

还没有进入以下运行时判断：

- 恢复时是否允许自动重放
- 重复请求是否返回首次结果
- 哪些操作必须 fail-closed
- retry 应该落在哪一层

### 2.4 executor 仍然是“直接调用工具”，没有去重与回读能力

核心文件：

- `backend/execution/tool_executor.py`
- `backend/tools/runtime.py`

当前 `ToolRuntimeExecutor.run()` 的语义是：

1. 拿到 `RuntimeActionRequest`
2. 找到 tool definition / instance
3. 直接执行
4. 把结果包装成 `RuntimeObservation`

当前没有：

- 执行前登记 durable execution record
- 以 step identity + request fingerprint 形成 dedupe key
- 命中重复请求时回读原结果
- 在恢复路径里把 non-replay-safe action fail-closed

这说明当前执行器还只是“调度层”，不是“可恢复执行语义层”。

### 2.5 trace 能看见请求和结果，但还看不见“为什么没重放”

核心文件：

- `backend/orchestration/runtime_loop/events.py`
- `backend/orchestration/runtime_loop/trace_reader.py`

当前事件里已经有：

- `tool_call_requested`
- `tool_result_received`
- `recovery_attempted`
- `checkpoint_written`

但还没有专门表达：

- execution record 创建
- replay guard 命中
- duplicate request 命中并复用已有结果
- recovery 对某次副作用做出的 replay decision

因此 trace 目前可以回看“发生了什么”，还不能稳定解释“为什么这次恢复没有重复写文件”。

## 三、与先进架构的差距

### 3.1 与 LangGraph Durable Execution 的差距

LangGraph 官方 durable execution 文档强调几件事：

- workflow progress 需要由 checkpointer 持久化；
- 要使用 thread 级身份恢复运行实例；
- non-deterministic 或 side-effecting work 要包进 `tasks`；
- 恢复时应依赖持久层和 task boundary，而不是重新触发已完成副作用。

来源：

- https://docs.langchain.com/oss/javascript/langgraph/durable-execution

当前差距：

- 我们有 checkpoint，但还没有“副作用动作必须先落 execution record，再进入 executor”的统一 boundary。
- 我们有 `task_run_id`，但还没有 `request fingerprint + idempotency token + execution receipt` 的组合身份。
- 我们能恢复账本，但还不能保证恢复时优先读回已有执行结果，而不是重跑工具。

### 3.2 与 Temporal Workflow Execution / Durable Execution 的差距

Temporal 的官方文档和技术指南强调：

- workflow execution 的系统真相来自 event history；
- durable execution 的核心是“每个执行步骤都以可恢复方式持久化”；
- 恢复依赖历史与确定性边界，而不是盲目重复副作用。

来源：

- https://docs.temporal.io/workflow-execution
- https://assets.temporal.io/durable-execution.pdf

当前差距：

- 我们已经有 event log，但 event log 里还没有“副作用执行账本事件”这一层。
- 我们已经把步骤状态机接起来了，但 side-effect boundary 还没有和 event/checkpoint 对齐。
- 当前恢复更多是 loop state 恢复，不是 execution history 驱动的安全延续。

### 3.3 与 AWS Well-Architected Reliability 的差距

AWS 官方在 Reliability 里对 mutating operation、retry、queue 积压有很明确的要求：

- mutating operation 应尽量设计为 idempotent；
- 用 client request token 区分新请求和重复请求；
- 重试前必须明确操作是否幂等；
- fail fast，避免多层重试和积压放大。

来源：

- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_prevent_interaction_failure_idempotent.html
- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_mitigate_interaction_failure_limit_retries.html
- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_mitigate_interaction_failure_fail_fast.html

当前差距：

- 运行时还没有统一 `idempotency_token`。
- retry 语义还没有明确落在单一层级。
- 对 shell / edit / worker 这类动作还没有把“可重试”和“禁止自动重放”分开。

## 四、会带来的具体问题

### 4.1 crash-resume 会重复写文件

典型路径：

1. step 已经准备执行 `edit_file`
2. 工具执行成功，但 checkpoint 还没同步反映副作用执行状态
3. 进程中断
4. 恢复时重新进入同一步

如果没有 execution record + idempotency contract，就可能对同一文件重复修改。

### 4.2 shell / worker 行为会重复触发

对于 build、test、browser verify、脚本执行等动作：

- 有些是可重复的
- 有些会污染环境
- 有些会造成外部副作用

当前系统还不能稳定区分这些动作在恢复时应：

- 自动重放
- 回读已有结果
- 直接拒绝继续

### 4.3 retry 会放大系统负担

如果 dependency 失败，而模型层、runtime 层、executor 层都各自补一层 retry：

- 容易形成 retry storm
- 容易加剧资源争用
- 容易把局部错误扩大成全局不稳定

### 4.4 checkpoint 的价值被严重打折

没有副作用保护时，checkpoint 只能安全用于：

- 只读分析
- 纯模型回答
- 无外部变更的工具路径

这会直接限制任务系统向开发和执行类任务扩展。

## 五、结构性判断

问题二的本质不是“给几个高风险工具加补丁”，而是：

- 让步骤状态机之下，再出现一层副作用执行语义层；
- 让 request、execution、observation 三者分离；
- 让 replay decision 成为运行时一等结构，而不是恢复时临场判断。

更准确地说，当前缺的不是“恢复功能”，而是“durable side-effect semantics”。

## 六、目标设计方向

### 6.1 引入 execution record 层

建议在 `RuntimeActionRequest` 和 `RuntimeObservation` 之间增加统一的副作用执行记录。

最小结构建议：

```python
class OperationExecutionRecord:
    execution_id: str
    task_run_id: str
    step_id: str
    request_ref: str
    directive_ref: str
    operation_id: str
    executor_type: str
    request_fingerprint: str
    idempotency_token: str
    replay_policy: str
    status: str
    result_ref: str
```

### 6.2 引入 replay policy

恢复时必须明确区分至少三类动作：

- `replay_read`
  - 只读动作，可直接重放
- `reuse_completed_result`
  - 幂等写动作，优先回读首次结果
- `deny_auto_replay`
  - 非 replay-safe 副作用，恢复时默认 fail-closed

### 6.3 引入单一 retry boundary

retry 需要被限定在单一层级，而不是多层叠加。

建议：

- step 层决定“是否允许 retry”
- execution record 层决定“该 retry 是否仍指向同一幂等身份”
- executor 只负责执行，不再私自补第二套重试逻辑

## 七、分阶段改造建议

### Phase 2A：接入执行记录层

目标：

- 为 mutating operation 创建 durable execution record
- 让 checkpoint 和 trace 能引用 execution id

### Phase 2B：接入 idempotency token 与 request fingerprint

目标：

- 用稳定 step identity + normalized input 形成 dedupe key
- 对支持幂等的动作建立“重复请求返回首次结果”的协议

### Phase 2C：接入 recovery replay policy

目标：

- 恢复时先看 execution record，再决定是否重放
- 对 non-replay-safe action 默认 fail-closed

### Phase 2D：把 trace 和测试补到故障链路

目标：

- trace 能解释“为什么这次恢复没有重放”
- 测试能覆盖 crash-resume、duplicate request、permission blocked、retry boundary

## 八、验收标准

- crash-resume 不会对同一副作用动作重复生效。
- 重试策略能区分幂等与非幂等动作。
- trace 中能回答某次副作用为什么没有重复执行。
- 运行时能明确说明某个失败是 fail-fast、retry exhausted、permission blocked 还是 replay denied。

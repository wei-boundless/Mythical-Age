# 编排权限与运行约束子系统详细设计书

日期：2026-05-03

上位计划：`25-健康系统管理控制面与验证体系统筹重构计划书-20260502.md`

## 一、子系统定位

编排权限与运行约束子系统是所有 agent 的“身体配置层”。本文件以健康管理 agent 为第一批落地对象，但规则不应写成健康专用。

它不决定健康任务的业务意义，也不决定前端如何展示。它只负责：

- agent 能否执行某类任务
- 能走哪条 runtime lane
- 能使用哪些 operation
- 能读取哪些上下文
- 是否允许副作用
- 是否写入 ledger/checkpoint/trace/execution record
- replay / retry / denied 如何被记录

健康管理 agent 可以成为健康系统管理者/代操者，但它的身体能力必须由编排系统硬约束。

## 三、目标约束模型

### 3.1 HealthCommandRuntimeAdmission

建议新增或内聚一个 admission 结果模型：

```python
class HealthCommandRuntimeAdmission:
    command_id: str
    agent_id: str
    agent_profile_id: str
    task_mode: str
    flow_id: str
    binding_id: str
    runtime_lane: str
    resource_policy_ref: str
    admitted: bool
    blocked_reasons: tuple[str, ...]
    diagnostics: dict[str, Any]
```

它是健康 command 从“业务意图”进入“运行时执行”之前的硬门禁。

### 3.2 Admission 检查顺序

```text
HealthManagementCommand
  -> command type supported
  -> task mode allowed by task flow
  -> binding valid
  -> agent capability profile valid
  -> projection template allowed
  -> skill workflow allowed
  -> runtime lane allowed
  -> resource policy allows requested operations
  -> runtime directive admitted
```

任何一步失败都必须 fail-closed。

## 四、运行语义

### 4.1 健康设置不能直接改权限

`update_health_setting` 可以改变：

- 默认健康测试组
- 默认报告策略
- 默认 issue 创建策略
- 默认 cutover readiness 必需项
- 默认健康管理 task flow / agent binding

但不能直接改变：

- allowed operations
- blocked operations
- runtime lane
- resource policy
- ledger truth
- checkpoint truth

如果用户想改变权限，必须进入编排系统配置流程，而不是健康设置。

### 4.2 健康 agent run 必须可追溯

每次健康 agent run 必须能说明：

- 来自哪个 command
- 使用哪个 task flow
- 使用哪个 binding
- 使用哪个 projection template
- 使用哪个 skill workflow
- 使用哪个 resource policy
- 走哪条 runtime lane
- 写了哪些 trace event
- 是否出现 permission denied / replay denied / retry exhausted

### 4.3 只读优先不是口头约束

健康管理 agent 的只读优先要落在三层：

- `AgentCapabilityProfile.blocked_operations`
- `ResourcePolicy`
- runtime operation gate

前端提示只是解释，不是权限控制。

## 五、与其它子系统协调

### 5.1 对任务系统

任务系统提供绑定候选。

编排系统校验绑定是否可运行。

任务系统不能把 `default_runtime_lane` 当作最终授权。

### 5.2 对健康后端控制面

健康后端把 command 交给 admission。

admission 返回 admitted/blocked，健康后端据此生成 receipt。

如果被拒绝，健康后端不能重试其它隐藏路径。

### 5.3 对健康系统原生操作台与嵌入式 agent 对话窗

健康系统原生操作台和嵌入式 agent 对话窗都只看到对用户有意义的 receipt 和 blocked reasons。

它们不应该知道具体 ResourcePolicy 内部细节，但可以展示对用户有意义的拒绝原因。

### 5.4 对测试体系

测试体系需要专门覆盖：

- permission denied
- blocked operation
- invalid binding
- invalid projection
- replay denied
- retry exhausted
- stale backlog

这些失败不能只在日志里出现，必须能进入健康报告。

## 六、实施步骤

### Phase 29A：Admission 服务

- 新增 `backend/health_system/runtime_admission.py` 或放入 `management_service.py`
- 输入 command + task config
- 输出 `HealthCommandRuntimeAdmission`
- 不执行任务，只做裁决

### Phase 29B：Command Handler 接入

- `analyze_trace / draft_case / verify_fix` 执行前必须调用 admission
- admission 失败时生成 rejected receipt
- admission 成功时继续创建 run

### Phase 29C：Trace 证据增强

- health agent run metadata 写入 command id、binding id、resource policy ref
- trace report 展示 admission result
- HealthReport evidence refs 引用 runtime trace 和 execution record

### Phase 29D：测试覆盖

- 增加 invalid binding regression
- 增加 blocked operation regression
- 增加 replay denied regression
- 增加 health command admission regression

## 七、验收标准

- 健康 command 进入执行前都有 admission 结果
- admission 失败不会执行任何副作用
- 失败 command 仍有 receipt
- 健康 agent run 能追溯到 command/binding/policy
- 健康设置不能绕过权限系统
- 旧隐藏路径无法绕过 admission

## 八、残留清理

应清理：

- 健康系统里直接构造 runtime lane 并执行的散装逻辑
- 不经过 binding validation 的 health run 路径
- 不生成 admission/receipt 的执行入口
- 把前端设置直接映射成 ResourcePolicy 的路径

应保留：

- `AgentCapabilityProfile`
- `TaskFlowBinding`
- `ResourcePolicy`
- `RuntimeExecutionStore`
- `TraceReader`

这些是编排层的核心骨架。

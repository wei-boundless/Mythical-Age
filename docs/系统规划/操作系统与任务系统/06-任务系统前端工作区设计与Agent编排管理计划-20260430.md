## 1. 设计原则

### 1.1 权力边界不能乱

任务系统前端可以展示和发起 Agent 管理操作，但底层权力仍归操作系统。

```text
操作系统：
  创建、删除、启停、归档、权限落库、生命周期治理。

任务系统：
  绑定 agent 到任务流。
  配置 task mode、runtime lane、workflow、projection、memory scope、output contract。
  决定 agent 是否在任务系统可见。
  管理主 Agent 调度策略和协调任务模板。
```

前端可以把这些能力放在同一个任务工作台里，但接口语义必须清楚：

```text
建立子 Agent：
  调用 OperationSystem.create_agent_profile。

隐藏子 Agent：
  修改 TaskSystem visibility，不删除操作系统注册。

禁用子 Agent：
  调用 OperationSystem.disable_agent_profile。

删除子 Agent：
  默认 archive，不物理删除历史 trace 引用。
```

### 1.2 主 Agent 与子 Agent 分开管理

主 Agent 不应该和子 Agent 混在一个列表里。

主 Agent 是：

```text
用户入口。
任务识别者。
任务分配者。
子 Agent 委派者。
最终对话与结果整合者。
```

子 Agent 是：

```text
绑定固定任务族。
绑定受限链路权限。
绑定 Skills 工作流。
绑定 ProjectionTemplate。
输出结构化候选。
```

因此：

```text
主 Agent 页面管调度和分配。
子 Agent 页面管实例、任务流和权限。
```

### 1.3 子 Agent 默认固定任务流，但允许受控变体

子 Agent 不是只能做一个动作，而是围绕一个任务族工作。

例如健康维护子 Agent：

```text
issue_triage
trace_analysis
case_draft
fix_verification
```

但它不能越界到：

```text
代码实现
任意 shell
长期记忆写入
任意联网检索
创建其他子 Agent
```

### 1.4 协调任务必须提前占位

后续多 Agent 协作不是简单“多个子 Agent 列表”，而是协调任务。

任务系统前端必须提前支持：

```text
CoordinationTaskDefinition
TopologyTemplate
CoordinatorAgent
ParticipantAgents
HandoffPolicy
SharedContextPolicy
MemorySharingPolicy
ConflictResolutionPolicy
OutputMergePolicy
StopConditions
```

## 2. 任务系统工作区信息架构

建议任务系统工作区使用左侧二级导航，右侧内容区。

页面结构：

```text
任务系统
  总览
  主 Agent 调度中心
  子 Agent 实例
  单 Agent 任务流
  协调任务
  拓扑模板
  链路权限矩阵
  Skills 工作流
  投影分配
  运行记录
```

### 2.1 总览

总览不是宣传页，而是运维仪表盘。

展示：

```text
已注册 Agent 数
任务系统可见子 Agent 数
启用任务流数
协调任务模板数
存在权限异常的绑定数
最近失败任务
最近健康问题
最近 RuntimeLoop traces
```

总览核心价值：

```text
告诉开发者任务系统现在能不能工作。
告诉开发者哪里有断链或越权。
```

### 2.2 主 Agent 调度中心

主 Agent 页面单独管理，不混进子 Agent 列表。

模块：

```text
主 Agent Profile
  agent_id
  agent_profile_id
  lifecycle_state = system_builtin
  projection_resolution_policy
  memory_scope
  task_dispatch_policy

任务识别策略
  task_classifier
  task_mode routing
  confidence threshold
  fallback task

委派策略
  可委派子 Agent
  delegation conditions
  handoff / no-handoff policy
  input filter policy

整合策略
  final response owner
  output merge policy
  commit policy

最近调度记录
  task_id
  selected task mode
  delegated agent
  runtime lane
  result
```

主 Agent 不是固定任务流执行者。它是任务分配层。

### 2.3 子 Agent 实例

子 Agent 页面管理实例状态和任务系统可见性。

列表字段：

```text
agent_id
display_name
owner_system
agent_profile_id
lifecycle_state
task_visibility
allowed_task_modes
allowed_runtime_lanes
default_projection_template
default_skill_workflows
memory_scope
risk_status
last_run
```

详情页分区：

```text
基础信息
生命周期
任务系统可见性
链路权限
Skills 工作流
投影模板
记忆范围
输出合同
最近运行
治理记录
```

操作：

```text
创建子 Agent
复制为新子 Agent
隐藏 / 显示
禁用 / 启用
归档
编辑 task visibility
编辑默认任务流绑定
查看 OperationSystem capability profile
```

危险操作规则：

```text
有历史 run 的 agent 只能归档，不物理删除。
主 Agent 不在此页出现。
健康维护子 Agent 初版不允许前端扩大危险操作权限。
```

### 2.4 单 Agent 任务流

这是任务系统的核心页面。

一条任务流不是一个 prompt，而是一条装配链：

```text
TaskDefinition
  -> TaskAgentBinding
  -> SkillWorkflowBinding
  -> ProjectionTemplate
  -> RuntimeDirectiveLane
  -> ResourcePolicy
  -> OutputContract
```

列表字段：

```text
task_mode
task_family
default_agent
workflow
projection
runtime_lane
memory_scope
output_contract
enabled
last_run_status
```

详情页：

```text
任务目标
输入合同
输出合同
默认 agent
可选 agent
workflow
projection_resolution_policy
runtime lane
resource policy preview
memory scope
stop conditions
validation report
```

健康维护任务流示例：

```text
task.health.issue_triage
  default_agent = agent:health:maintainer
  projection = xuannv__health_maintainer
  workflow = workflow.health.issue_triage
  lane = health_issue_read
  memory_scope = issue_local_readonly
  output_contract = HealthTriageResult
```

### 2.5 协调任务

协调任务页面为多 Agent 做准备。

它管理的是“多个受限 agent 如何协作”，不是现在就开放任意多 agent 自由对话。

字段：

```text
coordination_task_id
title
coordination_mode
coordinator_agent_id
participant_agent_ids
topology_template_id
shared_context_policy
memory_sharing_policy
handoff_policy
conflict_resolution_policy
output_merge_policy
stop_conditions
enabled
```

协调模式：

```text
sequential
parallel
supervisor
review_merge
debate
pipeline
```

页面应展示每个参与者的独立链路权限：

```text
participant agent
  task_mode
  runtime_lane
  workflow
  projection
  memory_scope
  allowed_operations
  output_contract
```

原则：

```text
协调任务不让多个 agent 共享全链路。
协调任务为每个参与者分配独立 capability 和 context window。
```

### 2.6 拓扑模板

拓扑模板是协调任务的结构复用。

示例：

```text
写作流程：
  director -> outline -> draft -> editor -> final_merge

公司协作：
  coordinator -> parallel_review -> conflict_resolution -> decision_summary

健康修复：
  health_triage -> issue_owner -> fix_verification -> report
```

字段：

```text
template_id
nodes
edges
handoff_rules
join_policy
failure_policy
terminal_policy
```

第一版只做只读展示和草案配置，不执行多 Agent。

### 2.7 链路权限矩阵

这个页面必须做实。

它用于一眼看清：

```text
哪个 agent 在哪个 task mode 下能走哪条链路。
```

矩阵维度：

```text
Agent
TaskMode
RuntimeLane
SkillWorkflow
ProjectionTemplate
MemoryScope
Operations
OutputContract
Visibility
RiskStatus
```

视觉要求：

```text
允许项用明确状态标识。
阻断项显示 blocked reason。
越权项显示 fail-closed。
不要展示后端 JSON 原文作为主要 UI。
```

### 2.8 Skills 工作流

Skills 工作流页不是 skill 列表。

它展示：

```text
workflow_id
task_mode
visible_skill_ids
steps
input_boundary
output_boundary
stop_conditions
required_evidence_refs
```

每个 workflow 要能看到绑定它的任务流和 agent。

### 2.9 投影分配

投影分配页管理任务态 projection。

核心字段：

```text
projection_template_id
soul_id
agent_profile_id
role_type
task_mode
projection_resolution_policy
default_skill_workflow_id
default_memory_policy
default_output_contract
```

投影切换策略：

```text
pinned:
  固定投影。适合健康维护、测试用例、代码审查。

manual:
  运行前手动选择。

auto:
  系统根据 task_mode、risk、trace、intent 自动选择。

hybrid:
  默认系统判断，但允许用户手动覆盖。
```

建议：

```text
健康子 Agent：pinned。
主 Agent：hybrid。
写作类子 Agent：manual 或 hybrid。
```

### 2.10 运行记录

运行记录页展示任务系统视角的 RuntimeLoop trace。

字段：

```text

agent_id
agent_profile_id
runtime_lane
task_mode
projection_id
prompt_manifest_id
workflow_id
status
terminal_reason
duration
health_issue_ref
```

## 5. UI 体验方向

任务系统是配置与治理工具，不是营销页。

设计应偏：

```text
高密度
清晰
可扫描
工程化
状态可读
链路可追踪
```

避免：

```text
大段说明文字。
后端 JSON 文件列表。
卡片堆卡片。
把运行按钮放到总览页。
混淆主 Agent 和子 Agent。
把测试/健康、操作系统、灵魂系统的职责塞进一个页面。
```

推荐 UI 结构：

```text
左侧窄导航：
  总览 / 主 Agent / 子 Agent / 单 Agent 任务流 / 协调任务 / 权限矩阵 / 工作流 / 投影 / 运行记录

顶部状态条：
  enabled agents / broken bindings / risky permissions / recent failed runs

右侧主内容：
  表格 + 详情抽屉 + 链路图 + 状态徽标
```

关键视觉组件：

```text
链路图：
  Task -> Agent -> Workflow -> Projection -> Lane -> ResourcePolicy -> RuntimeLoop -> Output

权限矩阵：
  行为 agent/task，列为 lane/workflow/projection/memory/operation。

详情抽屉：
  展示当前选择项的合同、风险和最近运行。

运行时间线：
  RuntimeEventLog 友好展示，不直接丢 JSON。
```

```text
页面不再把后端文件名当作主要管理对象。
```

## 8. 最终口径

任务系统前端不是系统架构海报。

它是：

```text
AgentRuntime 的任务装配、分配、协调和可追踪治理工作台。
```

主 Agent：

```text
承担任务识别、分配、委派和整合。
```

子 Agent：

```text
承担固定任务族下的受限执行。
```

协调任务：

```text
承担未来多 Agent 的拓扑、共享上下文、冲突解决和结果合并。
```

链路权限：

```text
把每个 agent 在每类任务中能走的系统链路显式化。
```

因此，将“系统架构”工作区替换成“任务系统”工作区是合理的。系统架构文档继续存在，但前端主工作区应该服务于真实运行和开发维护。

# 任务系统 Agent 任务连接子系统详细设计书

日期：2026-05-03

上位计划：`25-健康系统管理控制面与验证体系统筹重构计划书-20260502.md`

## 一、子系统定位

任务系统不是健康子 agent 的专用配置台，而是通用的 agent/task 思维连接层。

它负责回答这些通用问题：

- 系统里有哪些 agent 可以参与任务。
- 某个 agent 被分配什么任务身份。
- 父任务、子任务、bundle、follow-up 如何组织。
- task flow 如何绑定到 agent。
- skill workflow、projection template、memory scope 如何参与任务连接。
- 多 agent 协作拓扑如何组织。
- 哪些 task-agent binding 当前有效，哪些被配置或权限约束阻断。

健康管理 agent 只是这套通用机制里的一个实例：`agent:health:maintainer` 被任务系统赋予健康管理类任务身份、灵魂投影、workflow 和 task flow binding，然后由编排系统授予或拒绝实际运行能力，由健康系统承载它所管理的健康领域对象。

## 二、总分关系

本文件是 `25` 的任务系统分册，但它不是“健康系统分册”。

正确关系是：

- `24`：问题四本体，关注架构验证、故障演练、恢复证明和 cutover 证据。
- `25`：跨系统统筹，说明健康系统、任务系统、编排系统、测试体系、agent 对话窗如何协作。
- `26`：任务系统分册，定义通用 agent/task 连接机制，健康管理 agent 是一个落地用例。
- `27`：健康系统后端分册，定义健康系统自己的独立控制面。
- `28`：健康系统前端分册，定义内嵌 agent 对话窗，不替代健康系统操作台。
- `29`：编排分册，定义 agent 能力、权限、runtime lane 和执行约束。
- `30`：验证分册，定义健康验证、测试目录、readiness 和报告化。

## 四、目标领域模型

### 4.1 AgentTaskConnectionProfile

建议新增通用聚合模型：

```python
class AgentTaskConnectionProfile:
    profile_id: str
    agent_id: str
    agent_profile_id: str
    owner_system: str
    profile_type: str
    lifecycle_state: str
    task_family_refs: tuple[str, ...]
    available_task_modes: tuple[str, ...]
    flow_refs: tuple[str, ...]
    binding_refs: tuple[str, ...]
    projection_template_refs: tuple[str, ...]
    skill_workflow_refs: tuple[str, ...]
    topology_refs: tuple[str, ...]
    default_flow_ref: str
    default_projection_template_ref: str
    default_skill_workflow_ref: str
    default_runtime_lane_hint: str
    validation_state: str
    blocked_reasons: tuple[str, ...]
    diagnostics: dict[str, Any]
```

它是任务系统的通用 agent/task 连接投影。

健康管理 agent 对应其中一条：

```text
agent_id = agent:health:maintainer
owner_system = health_system
task_family_refs = ("health",)
```

### 4.2 TaskAgentConnectionOverview

建议新增：

```python
class TaskAgentConnectionOverview:
    authority: str
    profiles: tuple[AgentTaskConnectionProfile, ...]
    invalid_profile_count: int
    task_family_count: int
    topology_count: int
    diagnostics: dict[str, Any]
```

这个 overview 服务所有 agent，不只服务健康 agent。

## 五、目标接口

建议新增通用接口：

```text
GET /tasks/agent-task-connections
GET /tasks/agent-task-connections/{agent_id}
```

支持过滤：

```text
GET /tasks/agent-task-connections?owner_system=health_system
GET /tasks/agent-task-connections?task_family=health
```

不要新增健康专用的 `/tasks/health-agent-control-profile` 作为主接口。健康系统如需读取健康管理 agent 配置，应通过通用接口按 `agent_id` 或 `task_family` 过滤。

## 六、与其它子系统协调

### 6.1 对健康系统

任务系统提供的是健康管理 agent 的任务连接信息：

- agent id
- task family
- task mode
- flow id
- binding id
- projection template id
- skill workflow id
- topology id
- runtime lane hint

健康系统仍然独立管理：

- health issue
- health report
- health setting
- health test scenario
- health test run
- health workbench

健康系统可以引用任务系统提供的 agent/task 连接来创建健康管理 agent 会话或发起代操任务，但不能把任务系统当成健康领域对象的 owner。

### 6.2 对编排系统

任务系统输出的是“任务连接意图”和“绑定候选”。

编排系统负责最终判断：

- agent 是否可执行该 task mode
- runtime lane 是否允许
- resource policy 是否允许
- operation 是否被 blocked
- 是否需要审批或 fail-closed

因此任务系统不能直接授予最终运行权限。

### 6.3 对健康系统内嵌 Agent 对话窗

对话窗不直接拼接任务系统配置。

它通过健康系统后端创建会话；健康系统后端再按 `agent_id=agent:health:maintainer` 从任务系统读取通用 agent/task connection profile。

### 6.4 对测试体系

测试体系可读取任务系统连接视图，用于验证：

- agent-task binding 是否有效
- workflow/projection/output contract 是否一致
- topology 是否存在断链
- 健康管理 agent 作为一个普通 agent 实例是否满足任务连接不变量

## 七、实施步骤

### Phase 26A：通用聚合

- 在 `backend/tasks/flow_registry.py` 新增 `build_agent_task_connection_profiles`
- 复用 `list_flows / list_bindings / list_topology_templates / AgentRegistry.build_catalog`
- 不复制健康专用静态配置

### Phase 26B：通用接口

- 在 `backend/api/tasks.py` 新增 `/tasks/agent-task-connections`
- 在 `backend/api/tasks.py` 新增 `/tasks/agent-task-connections/{agent_id}`
- 支持 `owner_system` 和 `task_family` 查询参数

### Phase 26C：前端收敛

- 在 `frontend/src/lib/api.ts` 新增 `AgentTaskConnectionProfile`
- 在 `TaskSystemView.tsx` 中把相关视图命名为“Agent 任务连接”
- 健康管理 agent 作为其中一张 agent 卡片展示
- 移除任务系统页面里直接启动健康 run/test/report 的操作按钮

### Phase 26D：防越权

- 任务系统页面不编辑 health issue/report/setting/test run
- 任务系统页面不直接启动健康测试
- 任务系统页面不直接创建健康报告
- 任务系统页面不直接修改 runtime capability profile

## 八、验收标准

- 任务系统仍是通用 agent/task 思维连接层。
- 健康管理 agent 只是通用连接视图中的一个实例。
- 健康系统通过通用 agent/task connection profile 获取健康管理 agent 配置。
- 编排系统仍是最终权限裁决者。
- 任务系统不能承接健康系统领域对象的管理职责。

## 九、残留清理

应清理或迁移：

- 把任务系统页面命名或实现成“健康子 agent 专用配置台”的路径。
- `/tasks/health-agent-control-profile` 这类健康专用主接口设计。
- 任务系统页面直接调用 `startHealthAgentRun`、`startTestRun` 或写入 health report 的路径。

应保留：

- `/tasks/overview`
- `/tasks/flows`
- `/tasks/agent-bindings`
- `/tasks/link-permission-matrix`
- `/tasks/topology-templates`

这些是通用任务连接层的基础资产。

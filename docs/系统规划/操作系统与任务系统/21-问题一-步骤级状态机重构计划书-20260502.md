# 问题一：步骤级状态机重构计划书

日期：2026-05-02

## 一、文档目标

本计划书是对 `17-问题一-任务运行账本与步骤状态机差距报告-20260502.md` 的进一步展开。

本计划书回答四个更具体的问题：

1. 为什么当前系统需要状态机，而且必须是步骤级状态机。
2. 为什么不能把它做成重流程机或业务语义状态机。
3. 结合当前代码，状态机应该落在哪一层，哪些文件先改，哪些后改。
4. 如何在不牺牲灵活性的前提下，把 `TaskRunLedger` 提升为执行真相源。

本计划不讨论的问题：

- 副作用幂等与 crash-resume 细则
- bundle / subtask 的完整契约化
- worker 多机编排
- 前端任务编排 UI

这些问题后续分别由 `18`、`19`、`20` 对应阶段继续处理。

## 二、核心判断

### 2.1 需要的不是“更重的流程图”，而是“更薄的控制状态机”

当前系统已经具备以下结构：

- `TaskTemplate` 持有步骤蓝图
- `TaskSpec` 持有本轮实例
- `TaskRunLedger` 持有运行账本
- `RuntimeLoopState` 持有当前 loop 状态

因此问题已经不再是“要不要引入任务步骤”，而是“步骤状态到底谁说了算”。

当前答案仍然是：终局收口逻辑说了算。  
目标答案应该是：运行时步骤迁移说了算。

### 2.2 我们要的状态机层级

本次重构只引入两层状态机：

- `TaskRun` 层：控制整次任务运行生命周期
- `TaskStepRun` 层：控制每个 blueprint step 的生命周期

本次重构明确不引入：

- 业务语义状态枚举
- 模型内部推理状态机
- DAG 编排引擎
- 可视化 BPM 式流程定义器

### 2.3 最终目标

最终要形成如下关系：

```text
TaskTemplate
  -> TaskStepBlueprint[]
  -> TaskSpec
  -> RuntimeLoop consumes current step
  -> TaskRunLedger records step transitions as truth
  -> TaskResult is projected from final ledger
```

而不是当前更接近的：

```text
TaskTemplate
  -> RuntimeLoop executes broadly
  -> terminal phase infers step status
  -> TaskRunLedger / TaskResult are produced after the fact
```

## 三、理论依据

### 3.1 本地设计原则依据

来自项目内已有设计原则：

- [05-对话循环.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/05-对话循环.md:1)
  - 对话循环是状态机，不是一次性问答
  - 每个 continue / terminal 分支都必须明确原因
- [14-任务系统.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/14-任务系统.md:1)
  - 任务系统负责生命周期、隔离、终止、输出收集
  - worker、tool、agent 都应由统一任务框架管理
- [03-状态管理.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/03-状态管理.md:1)
  - 状态按生命周期分层
  - 越底层状态越克制

这三条叠加后的直接约束是：

- 状态机是必要的
- 但状态机必须克制
- 它应该落在运行控制层，而不是业务语义层

### 3.2 Temporal 的依据

Temporal 把 workflow execution 建立在 event history 和 durable execution 之上。它的关键思想不是“把业务逻辑写死成图”，而是：

- 工作流执行进度必须是系统真相
- 过程状态必须可恢复
- 终局结果不能替代执行历史

这对我们的启发是：

- `TaskRunLedger` 必须成为执行真相层
- step transition 必须先于 `TaskResult`
- event log 和 checkpoint 要围绕步骤迁移组织

来源：

- https://docs.temporal.io/workflow-execution
- https://assets.temporal.io/durable-execution.pdf

### 3.3 LangGraph 的依据

LangGraph 的 durable execution 重点在于：

- step progress 可持久化
- 恢复依赖结构化状态
- 副作用包进明确边界

它不是要求把 agent 行为僵化，而是要求：

- runtime 至少知道自己进行到哪一阶段
- 哪些阶段已经产生可信结果

这和我们要做的“步骤级控制状态机”高度一致。

来源：

- https://docs.langchain.com/oss/javascript/langgraph/durable-execution

### 3.4 AWS Step Functions 的依据

Step Functions 的价值在于说明一件事：

状态机不等于僵化顺序流。

它通过 `Choice`、`Parallel`、`Map` 证明：

- 状态机可以承担控制职责
- 任务内部依然可以保持灵活
- 复杂性可以放在状态之间的转移规则，而不是把所有业务含义塞进状态名

这对我们的启发是：

- 现在先做线性步骤级状态机是合理的
- 后续即使扩到 bundle / subtask / parallel，也不必推倒重来

来源：

- https://docs.aws.amazon.com/step-functions/latest/dg/concepts-statemachines.html
- https://docs.aws.amazon.com/step-functions/latest/dg/state-choice.html
- https://docs.aws.amazon.com/step-functions/latest/dg/state-parallel.html
- https://docs.aws.amazon.com/step-functions/latest/dg/amazon-states-language-map-state.html

## 五、设计原则与边界

### 5.1 原则一：控制面状态机，执行面保持灵活

状态机只回答以下问题：

- 当前在哪个 step
- 该 step 是否 running / completed / failed / skipped
- 下一步是否允许进入
- 终局为什么发生

状态机不回答以下问题：

- 模型具体如何推理
- 工具内部如何执行
- 文档分析、RAG、数据分析的业务语义差别

### 5.2 原则二：步骤之间确定，步骤之内开放

推荐边界：

- step 之间：严格、显式、可追踪
- step 之内：灵活、模板驱动、执行器可演化

举例：

- `step.inspect_pdf` 内部可以继续调用模型和 `pdf_analysis`
- 但 runtime 必须知道“当前是在 inspect_pdf，而不是 finalize_pdf”

### 5.3 原则三：结果只能从账本投影，不得反向推断账本

约束如下：

- `TaskResult` 只能从最终 `TaskRunLedger` 投影
- 不允许 `TaskResult` 反过来决定某个 step 是否完成
- 不允许 `final_content` 单独决定整个步骤链路的状态

### 5.4 原则四：先做线性步骤机，不提前上复杂控制流

本阶段只支持：

- 单当前 step
- 串行推进
- 明确 skip / fail / complete

本阶段不做：

- 并行步骤
- nested subgraph runtime
- step fan-out / fan-in
- 动态插入 blueprint

### 5.5 原则五：状态名必须抽象，不能业务化

允许的状态：

- `pending`
- `running`
- `completed`
- `failed`
- `skipped`

不允许的方向：

- `pdf_analyzing`
- `web_game_verifying`
- `bundle_aggregating_weather`

业务语义属于 `step_id` 和 `step_kind`，不属于状态值。

## 六、目标模型

### 6.1 TaskRun 级别状态

保留当前 `TaskRunStatus` 基本结构：

```text
created -> running -> completed
                   -> failed
                   -> aborted
                   -> blocked
                   -> waiting_approval
```

这个层级只管理整轮生命周期，不承载 step 细节。

### 6.2 TaskStepRun 级别状态

保留当前 `TaskStepRunStatus`，但要求它进入运行时主循环：

```text
pending -> running -> completed
pending -> running -> failed
pending -> skipped
running -> completed
running -> failed
```

### 6.3 需要新增或明确的字段

建议在 `TaskStepRun` 增加以下字段：

```python
entered_at: float = 0.0
completed_at: float = 0.0
attempt_count: int = 0
failure_reason: str = ""
step_result_ref: str = ""
executor_ref: str = ""
```

这些字段的目的不是做复杂审计，而是为后续恢复、重试、跳步提供稳定落点。

### 6.4 需要新增的 transition helper

建议新增一组集中式 helper，而不是在 loop 里散落拼对象：

```python
start_step_run(...)
complete_step_run(...)
fail_step_run(...)
skip_step_run(...)
advance_task_run_ledger(...)
project_task_result_from_ledger(...)
```

这些 helper 应成为本次阶段的核心切入点。

## 七、推荐方案比较

### 7.1 方案 A：保持现状，只优化终局推断

优点：

- 改动最小

缺点：

- 仍然没有 authoritative step state
- 恢复、重试、取消问题不会真正解决
- 后续 worker / shell / verify 扩展时会继续失真

结论：

- 不采用

### 7.2 方案 B：重流程机

特点：

- 每类任务定义完整固定流程
- runtime 严格按流程解释业务

优点：

- 控制极强

缺点：

- 与当前 agent 型 runtime 不匹配
- 模板一多就僵化
- 需要把业务语义塞进 runtime 核心

结论：

- 不采用

### 7.3 方案 C：步骤级薄状态机

特点：

- 运行时只管理 step transition
- 模板负责步骤语义
- 结果从 ledger 投影

优点：

- 既能提升恢复与可观测性
- 又不会把任务系统变成 BPM 引擎

缺点：

- 仍需改动 loop、trace、checkpoint、tests

结论：

- 采用

## 八、重构目标设计

### 8.1 运行顺序

目标执行链：

```text
build task contract
  -> create initial ledger
  -> mark first step running
  -> executor produces observation/result
  -> complete/fail/skip current step
  -> advance next step if any
  -> when terminal, project TaskResult from final ledger
```

### 8.2 事件设计

建议新增事件：

- `step_entered`
- `step_completed`
- `step_failed`
- `step_skipped`
- `task_run_ledger_updated`

已有事件继续保留：

- `task_contract_built`
- `loop_terminal`
- `checkpoint_written`

原则：

- 新事件记录过程
- 旧终局事件记录结果

### 8.3 checkpoint 设计

每次 step transition 后应考虑写 checkpoint，至少保证：

- step entered 后有可恢复状态
- step completed / failed / skipped 后有可恢复状态

本阶段不要求做到完整副作用恢复，但必须让 checkpoint 至少对 step progress 负责。

### 8.4 trace 设计

`RuntimeLoopTraceReader` 应能直接回答：

- 初始 step 是什么
- 当前 / 最后进入的 step 是什么
- 每个 step 的终态是什么
- 哪一步触发 terminal

而不是只能从最终 `task_result.step_runs` 倒推。

## 九、阶段实施计划

### Phase 1A：数据模型收紧

目标：

- 让 `TaskStepRun` 能承载运行期最小事实
- 明确 `TaskResult` 只能来自 final ledger projection

输入：

- 现有 `TaskStepRun` / `TaskRunLedger` / `TaskResult`

输出：

- 收紧后的运行模型
- 明确的 helper API 边界

涉及文件：

- `backend/tasks/run_models.py`
- `backend/tasks/step_models.py`

完成标准：

- `TaskStepRun` 增加运行期字段
- 新增 transition helper
- 不再把 retry policy 塞进 `diagnostics` 充当过程信息替代品

### Phase 1B：runtime loop 接管步骤迁移

目标：

- 让 `RuntimeLoopState.current_step_id` 成为控制字段
- 让 loop 在执行中推进 ledger，而不是在终局推断

输入：

- `task_contract_built` 后的初始 ledger

输出：

- 执行期 ledger 更新
- step 进入与退出事件

涉及文件：

- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/orchestration/runtime_loop/models.py`

完成标准：

- 第一个 step 在运行早期进入 `running`
- executor observation 会驱动当前 step 完成或失败
- 终局阶段不再重新生成一套“猜出来的步骤状态”

### Phase 1C：checkpoint 与 trace 对齐

目标：

- checkpoint 写入真实 step progress
- trace summary 能直接读取 step transition

涉及文件：

- `backend/orchestration/runtime_loop/checkpoint.py`
- `backend/orchestration/runtime_loop/trace_reader.py`

完成标准：

- checkpoint 中的 `loop_state.current_step_id` 与 ledger 一致
- trace 能列出 step event 概览

### Phase 1D：结果投影改造成单向投影

目标：

- `TaskResult` 只从 final ledger 投影
- 删除或降级 `_final_step_runs()` 的反推职责

涉及文件：

- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/tasks/run_models.py`

完成标准：

- `_build_runtime_task_result()` 不再根据 `final_content` / `tool_observation_count` 单独判断 step status
- `_build_final_task_run_ledger()` 不再成为终局“二次造账本”的主要入口

### Phase 1E：回归与切换

目标：

- 证明步骤级状态机不会损失当前功能
- 为下一阶段恢复语义提供稳定基础

涉及文件：

- `backend/tests/task_runtime_contract_regression.py`
- `backend/tests/query_runtime_runtime_loop_regression.py`
- `backend/tests/test_system_runtime_loop_regression.py`

完成标准：

- 新增 step transition 回归
- 新增 skip / fail / partial completion 回归
- 原有 contract / runtime 回归继续通过

## 十一、禁止事项

本次阶段明确禁止以下做法：

### 11.1 禁止把状态机做成业务语义机

不允许把“PDF 分析”“结构化数据分析”“网页小游戏验证”做成状态枚举值。

### 11.2 禁止继续强化终局反推

不允许再围绕 `_final_step_runs()` 加更多 if/else 去修补状态判断。

### 11.3 禁止把 bundle / parallel 提前混进本阶段

本阶段先把线性步骤机立住，bundle / parallel 以后通过 contract 和 control-flow 扩展。

### 11.4 禁止把灵活性理解为“无状态”

允许模型灵活，不等于允许 runtime 对步骤位置失去真相。

## 十二、切换与回滚规则

### 12.1 切换策略

建议采用小步切换：

1. 先保留旧终局投影逻辑作为 shadow compare
2. 新旧逻辑同时产出 step runs
3. 回归对齐后，再删除旧反推逻辑

### 12.2 shadow compare 期

建议在短期内保留：

- 新 ledger-driven step runs
- 旧 inferred step runs

并在测试中断言：

- 在当前支持的模板范围内，两者结果一致
- 一旦出现不一致，优先以 ledger-driven 路径为准，排查旧推断逻辑

### 12.3 回滚策略

如果阶段中发现：

- 新 step transition 无法覆盖当前模板
- trace / checkpoint 明显退化
- runtime loop 出现广泛终局不一致

则允许临时回滚到：

- 保留数据模型扩展
- 暂停 loop 接管
- 继续使用旧终局推断

但不允许回滚掉新增字段和 helper，因为这些是后续阶段基础。

## 十四、实施后预期收益

1. `TaskRunLedger` 从描述层变成执行真相层。

## 十五、与后续阶段的关系

本阶段不是终点，而是后续三个阶段的前提：

- 没有步骤级状态机，就无法可靠做恢复与幂等。
- 没有步骤级状态机，就无法把 bundle / subtask 契约真正接进运行时。
- 没有步骤级状态机，故障演练也只能验证终局，不可能验证过程。

因此本阶段应视为整个任务运行时重构的第一道硬门槛。

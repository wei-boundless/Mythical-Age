# 全项目系统边界总设计书

日期：2026-05-03

## 一、文档定位

本文件是当前项目的上位总设计书，用于统一说明：

- 项目的核心目标
- 各系统的边界
- Agent 的统一定义
- 任务系统与权限系统的关系
- 多系统与多 Agent 的协作关系
- 后续各分册文档的约束基线

后续细化文档必须以本文件为准。

## 二、项目总体目标

本项目不是单纯的聊天应用，也不是单纯的工具集合，而是一个：

`以任务承接为中心、以 Agent 为执行体、以多个系统为能力域、以运行时编排层负责执行的智能操作系统`

它要解决的问题不是“如何生成一句回答”，而是：

1. 如何承接用户或系统发出的任务
2. 如何把任务交给合适的 Agent
3. 如何让 Agent 在不同系统资源上稳定工作
4. 如何在复杂任务中组织多个 Agent 协作
5. 如何持续监控、验证并修正系统运行质量

## 三、基础对象统一定义

## 3.1 系统

系统是能力域、资源域、治理域。

系统本身不执行任务，而是为任务执行提供：

- 资源
- 规则
- 状态
- 能力
- 上下文

当前核心系统包括：

- 任务系统
- 权限系统
- 记忆系统
- 健康系统
- 操作系统
- 灵魂系统
- 运行时编排层

## 3.2 Agent

Agent 是任务执行者。

Agent 的统一定义如下：

- 接收任务
- 接收特定信号
- 按内部设定处理任务
- 访问必要系统资源
- 输出特定结果

Agent 不是系统本体，也不是系统天然长出的结构。

Agent 是开发者为了完成任务而配置出来的执行体。

## 3.3 主 Agent

主 Agent 只是一个特殊职责方向的 Agent。

它的特点是：

- 对接用户主会话
- 承接通用任务
- 负责最终整合输出

主 Agent 与其他 Agent 没有本体差异，只有职责差异。

## 3.4 子 Agent

子 Agent 是开发者为了特定任务而配置出的执行者。

例如：

- 健康系统需要维护
- 因此配置健康管家 Agent
- 由它承接实验报告解读、健康状态分析、维护类任务

因此：

- 健康系统是系统
- 健康管家是 Agent

两者不能混同。

## 3.5 任务

任务是希望被完成的目标。

任务分为两类：

### 通用任务

由用户在主会话中提出，默认由主 Agent 承接。

### 特定任务

由系统预先登记，绑定特定 workflow，由特定 Agent 承接。

## 3.6 Workflow

Workflow 是任务做法。

它回答的是：

`这类任务应该按什么流程完成`

workflow 归任务系统管理，不归权限系统管理。

## 3.7 Projection

Projection 是任务执行时附加给 Agent 的任务向提示补充。

它用于增强：

- 表达姿态
- 工作方式
- 任务执行风格
- 特定任务导向

projection 在任务层进行选择和绑定，不在权限层管理。

## 四、项目总体结构

整个系统可抽象为七层：

1. `用户与外部信号层`
2. `任务系统`
3. `Agent 执行者层`
4. `权限系统`
5. `运行时编排层`
6. `领域系统层`
7. `健康验证闭环`

关系如下：

```text
用户 / 外部系统信号
  -> 任务系统
      -> 判断任务类型
      -> 分配 Agent
      -> 绑定 workflow
      -> 绑定 projection
      -> 必要时组织多 Agent 协作
  -> 权限系统
      -> 定义 Agent 内部设定与边界
  -> 运行时编排层
      -> 执行 prompt 装配 / 上下文组织 / 工具调用 / 结果输出
  -> 领域系统
      -> 提供能力、资源、上下文
  -> 健康验证闭环
      -> 观测、诊断、测试、纠偏
```

其中最关键的两条职责分界是：

- `任务系统决定谁做什么`
- `权限系统决定这个 Agent 怎么做`

## 五、各系统边界

## 5.1 任务系统

### 定义

任务系统是全项目的任务承接与任务组织中枢。

### 负责

- Agent 管理
- 通用任务管理
- 特定任务管理
- workflow 与任务绑定
- projection 与任务绑定
- Agent 承接关系
- 多 Agent 协调任务
- 多 Agent 拓扑管理

### 不负责

- Agent 内部权限设定
- runtime 底层保护边界
- 系统资源访问规则本体
- 记忆本体管理
- 灵魂本体管理

### 核心问题

`什么任务，由哪个 Agent，用什么 workflow 和 projection 去完成`

## 5.2 权限系统

### 定义

权限系统是 Agent 内部设定与运行边界管理系统。

### 负责

- prompts 组织边界
- 工具能力边界
- 系统资源访问边界
- 上下文边界
- 记忆边界
- 输出边界
- 子系统窗口级保护策略

### 不负责

- 任务登记
- Agent 承接关系
- workflow 绑定
- projection 绑定

### 核心问题

`这个 Agent 以什么内部结构和边界运行`

### 特别规则

权限系统不再管理：

- workflow
- projection

## 5.3 记忆系统

### 定义

记忆系统是信息沉淀、存储、召回和上下文支持系统。

### 负责

- 长期记忆
- 状态记忆
- 召回结果
- 记忆范围控制
- 为 Agent 提供可用上下文

### 不负责

- 任务分配
- workflow 绑定
- Agent 角色定义

### 核心问题

`什么信息应该被记住，以及什么时候被取回给 Agent 使用`

## 5.4 健康系统

### 定义

健康系统是系统运行问题观测、诊断、验证与纠偏系统。

### 负责

- 采集问题
- 解读实验报告
- 运行异常诊断
- 修复建议
- 健康验证与回归

### 不负责

- 定义任务系统边界
- 定义 Agent 角色分配
- 定义权限系统内部策略

### 核心问题

`系统哪里坏了，为什么坏，如何验证修复`

## 5.5 操作系统

### 定义

操作系统是能力资源与工具资源管理系统。

### 负责

- skills
- 工具能力
- 操作资源
- 可调用能力目录

### 不负责

- 任务归属
- Agent 承接关系
- workflow 绑定关系

### 核心问题

`系统有哪些能力可以被调用`

## 5.6 灵魂系统

### 定义

灵魂系统是人格基底与投影来源系统。

### 负责

- 灵魂设定
- 人格基底
- 上下文组织风格
- 投影来源

### 不负责

- 任务分配
- workflow 绑定
- Agent 承接关系

### 核心问题

`Agent 以什么人格基底和投影来源存在`

## 5.7 运行时编排层

### 定义

运行时编排层是实际执行层。

### 负责

- prompt 装配
- 上下文组织
- workflow 执行
- 工具调用
- 结果输出
- 单 Agent 执行
- 多 Agent 协作执行

### 不负责

- 任务登记
- Agent 注册
- 灵魂本体管理
- 记忆本体管理

### 核心问题

`已经决定好的任务，如何被稳定执行出来`

## 六、Agent 体系总定义

## 6.1 Agent 的本质

Agent 是任务执行者，不是系统本体。

一个系统可以有多个 Agent。

Agent 是按任务方向被配置出来的。

## 6.2 主 Agent 与子 Agent 的关系

主 Agent 和子 Agent 只有职责差异，没有本体差异。

### 主 Agent

- 主会话入口
- 通用任务承接
- 最终整合输出

### 子 Agent

- 特定任务承接
- 特定系统任务承接
- 复杂任务协同参与

不能把主 Agent 看成“总控本体”，把其他 Agent 看成“挂件”。

更准确的理解是：

- 主 Agent：主入口执行者
- 子 Agent：定向任务执行者

## 6.3 子 Agent 的保护边界

子 Agent 是为了开发便利和任务执行而接入的执行者。

它们不是系统的内生底层结构，因此前端必须限制危险修改面。

不应开放：

- workflow 修改
- 权限修改
- runtime 底层保护边界修改

原因：

`防止系统功能被破坏`

## 七、接口 Agent 与未来会话入口

当前项目中，Agent 还承担一个重要角色：

`对外会话入口接口`

预期会有以下六个主要入口：

1. 主 Agent
2. 权限系统 Agent
3. 记忆系统 Agent
4. 健康系统 Agent
5. 操作系统 Agent
6. 灵魂系统 Agent

其中：

- 主 Agent 对接主会话
- 其余五个系统 Agent 对接未来的系统悬浮会话窗口

但即使如此，也不能把这些入口 Agent 误认为是系统本体。

它们仍然只是：

`负责承接某类信号与任务的执行者`

## 八、任务系统与权限系统的关系

这是全项目最重要的一组边界。

## 8.1 任务系统的决定权

任务系统决定：

- 任务是什么
- 任务交给谁
- 任务用什么 workflow
- 任务用什么 projection
- 是否需要多 Agent 协作

## 8.2 权限系统的决定权

权限系统决定：

- 这个 Agent 如何组织 prompts
- 这个 Agent 可以访问哪些能力
- 这个 Agent 可以接收什么信号
- 这个 Agent 输出必须遵守什么边界

## 8.3 不可混淆项

不能再把以下职责混在一起：

- 任务系统去改 Agent 内部权限
- 权限系统去改 workflow 与 projection

统一原则：

`任务系统做任务分配`
`权限系统做 Agent 内部设定`

## 九、多 Agent 协作的定位

多 Agent 协作不是权限系统的主职责，而是任务系统的组织能力。

因为多 Agent 协作回答的是：

- 谁参与
- 谁先做
- 谁后做
- 谁汇总
- 如何 handoff

这本质上是任务组织问题，不是权限设定问题。

权限系统在多 Agent 协作中的作用是：

- 给每个 Agent 提供自己的内部边界

但不负责决定整个协作结构。

## 十、对当前代码实现的约束

基于本总设计书，后续代码必须满足以下约束：

1. 任务系统中保留 workflow 和 projection 绑定职责
2. 权限系统中移除 workflow 和 projection 管理职责
3. Agent 注册模型必须体现“任务执行者”语义
4. 主 Agent 与子 Agent 不应在模型层被定义为本体不同对象
5. 子 Agent 的危险修改面必须被限制
6. 多 Agent 编排归任务系统，不归权限系统

## 十二、最终统一结论

将整个项目压缩成一句话：

`任务系统决定谁做什么，权限系统决定这个 Agent 怎么做，运行时编排层把它实际做出来，各领域系统提供能力域，健康系统负责持续验证与纠偏。`

将 Agent 体系压缩成一句话：

`Agent 是任务执行者，不是系统本体；主 Agent 与子 Agent 只有职责差异，没有本体差异。`

将任务系统压缩成一句话：

`任务系统的目的，是承接任务、分配任务、组织 Agent 完成任务。`

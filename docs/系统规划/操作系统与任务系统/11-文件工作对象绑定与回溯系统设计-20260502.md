# 文件工作对象绑定与回溯系统设计

日期：2026-05-02

## 一、问题定义

PDF、Excel、CSV、JSON、Markdown 等输入在运行时都应被视为“文件工作对象”。它们的差异只影响解析和分析方式，不应影响上下文绑定、结果回溯和 follow-up 恢复的主协议。

当前问题不是模型无法理解“刚才那个文件”，而是运行时没有稳定把“刚才处理过的文件、分析结果、结果子集”写成可恢复状态。于是用户说“这些人”“第二部分”“刚才那个表”时，系统容易退回到自然语言猜测或错误路由。

正确的系统属性是：

- 文件本体有稳定身份。
- 每次分析结果有稳定身份。
- 分析结果里的页码、章节、筛选行、Top-N、分组结果等子集有稳定身份。
- follow-up 先恢复这些身份，再决定调用哪个工具。

## 二、设计原则约束

本设计只按当前系统已有原则收敛：

1. 工具只执行，不拥有长期上下文。
2. skill 是方法指引，不是状态存储。
3. prompt 只展示本轮必要的可用上下文，不承担权限或绑定真相。
4. 记忆/上下文系统负责恢复候选和当前工作对象。
5. 运行时收口层负责把执行结果投影成可恢复状态。
6. 显式当前输入优先于历史绑定；历史绑定只在 follow-up 指代明确时生效。

## 三、统一对象模型

### 3.1 SourceFileHandle

表示可继续处理的源文件。

字段：

- `handle_id`
- `file_path`
- `file_kind`: `pdf | spreadsheet | csv | json | markdown | text | unknown`
- `display_name`
- `owner_task_id`
- `created_at`

### 3.2 ResultHandle

表示一次稳定分析结果。

字段：

- `handle_id`
- `source_file_handle_id`
- `result_kind`: `pdf_summary | page_answer | section_answer | table_analysis | filtered_rows | aggregation | top_n`
- `query`
- `summary`
- `owner_task_id`
- `created_at`

### 3.3 SubsetHandle

表示结果中的可回指子集。

字段：

- `handle_id`
- `source_file_handle_id`
- `result_handle_id`
- `subset_kind`: `page_range | section | row_selection | top_n | filtered_group`
- `labels`
- `page_range`
- `filter_column`
- `row_keys`
- `natural_aliases`
- `owner_task_id`
- `created_at`

PDF 页码、PDF 章节、Excel 前五行、CSV 筛选结果，本质上都归入 `SubsetHandle`。

## 四、职责边界

### 4.1 Tool / Worker

负责：

- 执行具体解析和分析。
- 返回 canonical result。
- 返回 source/result/subset handles。

不负责：

- 修改 session memory。
- 决定长期记忆。
- 用 prompt 解释绑定规则。

### 4.2 Projection

负责：

- 把 canonical result 转成 `MainContextState`。
- 更新 active source/result/subset。
- 产出 `TaskSummaryRef`。

### 4.3 Session Context

负责：

- 保存当前文件工作对象绑定。
- 保存最近流程快照。
- 作为 follow-up 恢复的主要事实源。

### 4.4 Understanding / Follow-up Resolver

负责：

- 根据当前用户消息和 session context 决定绑定对象。
- 显式路径优先。
- 明确指代时恢复 active subset/result/source。
- 多候选同等可信时才澄清。

### 4.5 Prompt

负责：

- 给模型看必要的可用性信号。
- 例如“当前有一个活跃表格结果可继续分析”。

不负责：

- 暴露内部 handle id。
- 暴露路径和协议字段。
- 用自然语言硬编码工具权限或状态绑定。

## 五、固定执行流

```text
用户请求
  -> understanding 识别显式文件或 follow-up 指代
  -> resolver 绑定 SourceFile / Result / Subset
  -> task contract 写入 bound_file_context
  -> tool/worker 执行
  -> worker/tool 返回 canonical result + handles
  -> projection 写回 MainContextState
  -> session memory 持久化当前绑定
  -> 下一轮 follow-up 从 session context 恢复
```

当前代码中，`PDFWorker`、`StructuredDataWorker` 和 `WorkerProjectionAdapter` 已经具备部分 handle 产出能力。主缺口在运行时收口层：工具执行结果没有稳定投影回 `main_context` 和 session state。

## 六、绑定优先级

1. 当前用户消息有显式文件路径：使用当前路径，并覆盖旧绑定。
2. 当前消息有“这些人 / 这些数据 / 前五 / 上面结果”：绑定 `active_subset_handle`。
3. 当前消息有“刚才的分析 / 这个结果”：绑定 `active_result_handle`。
4. 当前消息有“这份 PDF / 这个表 / 刚才那个文件”：绑定 `active_source_file_handle`。
5. 多个候选同等可信时，澄清。

## 七、实施计划

### 阶段 1：运行时写回接线

目标：

- 工具观测中保留工具名、输入参数和结果。
- 对 PDF / 结构化文件分析结果建立统一投影。
- 运行时最终 `done` 事件不再清空 `main_context` 和 `task_summary_refs`。

影响文件：

- `backend/orchestration/runtime_loop/action_request.py`
- `backend/execution/tool_executor.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`

### 阶段 2：Session Memory 持久化

目标：

- assistant commit 后如果存在 file-work projection，优先用 `refresh_session_memory_from_context_state` 写回状态。
- 没有 projection 时保留现有从历史消息刷新方式。

影响文件：

- `backend/query/runtime.py`

### 阶段 3：Follow-up 绑定增强

目标：

- 后续 resolver 消费 active source/result/subset。
- “这些人”“第二部分”等指代不再只靠关键词。

影响文件：

- 当前运行时理解层和任务契约层。

本轮先完成阶段 1 和阶段 2，阶段 3 可跟结构化数据 follow-up 一起继续收敛。

## 八、验收标准

1. PDF 和结构化数据工具结果能写回 session state。
2. `active_object_handle_id`、`active_result_handle_id`、`active_subset_handle_id` 不再在成功分析后长期为空。
3. prompt 里只出现可用性信号，不暴露内部 handle。
4. 长情景测试中 PDF / 结构化数据 follow-up 不再因上下文写回为空而失败。
5. 没有新增 `Denied`、`preview_only` 或权限解释型 prompt。

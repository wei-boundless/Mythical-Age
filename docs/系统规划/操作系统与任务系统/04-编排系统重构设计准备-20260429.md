# 编排系统重构设计准备

日期：2026-04-29  
定位：在任务系统与操作系统 preview 主链之后，开始重构“编排系统”。本文只做设计准备和施工边界，不开放真实工具、worker、agent、文件写入或 memory 写回。  
命名约定：后续统一称为“编排系统”，不再用“调度系统”作为正式系统名。

---

## 0. 结论

当前任务系统和操作系统已经完成第一阶段 preview 主链：

```text
UserRequest
  -> TaskContract
  -> TaskDefinition / TaskBinding / SkillRuntimeView
  -> OperationRequirement
  -> ResourcePolicyPreview
  -> ResourceRuntimeView
  -> TaskPromptContract
  -> SoulRuntimeView / PromptManifestPreview
  -> ControlKernel diagnostics
```

但真实执行不能由任务系统或操作系统直接完成，必须由编排系统统一采纳、验证、生成执行图和运行指令。

目标口径：

```text
任务系统管“要做什么”。
操作系统管“能碰什么”。
编排系统管“怎么做、谁来做、按什么顺序做、是否可以执行”。
执行层只按 RuntimeDirective 做。
CommitGate 统一处理写回。
```

下一阶段不是恢复旧 `QueryRuntime` 的工具/worker 分支，而是建立：

```text
TaskOperationPreview
  -> CandidateCollector
  -> OrchestrationPlanPreview
  -> PlanValidationResult
  -> ExecutionGraphPreview
  -> RuntimeDirectiveCandidate
  -> blocked / preview_only
```

第一阶段编排系统仍然：

```text
不真实执行。
不生成可消费的 RuntimeDirective。
不调用 ToolExecutor / WorkerExecutor / BoundedAgentExecutor。
不写 session / durable memory / artifact。
```

## 1. 设计原则约束

### 1.1 单一执行真相

必须继承 `00-设计原则继承与重构约束.md`：

```text
当前轮执行真相只能来自 ControlKernel 采纳后的 OrchestrationPlan / ExecutionGraph / RuntimeDirective。
```

禁止：

```text
QueryPlanner 直接决定执行。
QueryRuntime 根据 query_understanding.tool_name 直接调工具。
RuntimeToolBridge 自己扩大工具范围。
TaskCoordinator 从自然语言猜任务绑定后直接写 task result。
WorkerResult 直接成为 final answer。
Projection / Soul 声明工具权限。
```

### 1.2 Candidate 不等于 Decision

旧 planner、follow-up、memory restore、capability dispatch 都只能是候选生产者。

第一批候选：

```text
TaskOperationPreviewCandidate
LegacyQueryPlanCandidate
FollowupCandidate
MemoryContextCandidate
CapabilityCandidate
WorkerCandidate
ToolCandidate
OutputCommitCandidate
```

固定约束：

```text
candidate.authority == "candidate_only"
candidate 不含 runtime_directive。
candidate 不触发工具、worker 或写回。
ControlKernel / OrchestrationCoordinator 才能采纳 candidate。
```

### 1.3 控制平面与执行平面分离

目标分层：

```text
Control Plane:
  CandidateCollector
  OrchestrationCoordinator
  PlanValidator
  ExecutionGraphBuilder
  RuntimeDirectiveBuilder
  CommitPlanner

Execution Plane:
  ModelExecutor
  ToolExecutor
  WorkerExecutor
  BoundedAgentExecutor
```

第一阶段只做 Control Plane preview。

### 1.4 Fail-Closed 默认

下面情况必须 blocked：

```text
缺 ResourcePolicy。
ResourcePolicy 仍是 preview。
PlanValidator 未通过。
ExecutionGraph 没有 adopted policy refs。
RuntimeDirective 缺失。
OperationGate 未复核。
CommitGate 未允许。
```

禁止 validator blocked 后回到旧执行链。

## 2. 旧代码现实报告

### 2.1 目前新的编排骨架

已有文件：

```text
backend/orchestration/contracts.py
backend/orchestration/candidates.py
backend/orchestration/execution_graph.py
backend/orchestration/kernel.py
backend/orchestration/unit_registry.py
```

已有能力：

```text
ControlKernelPreviewContext 能接收 task/resource/prompt refs。
ControlKernel.collect() 能返回 blocked result。
ExecutionGraph 可以承载 refs，但 nodes 仍为空。
ExecutionNode 要求 authority == "runtime_directive"。
CommitCandidate 默认 allowed = False。
```

当前不足：

```text
没有 OrchestrationPlan。
没有 PlanValidator。
没有 RuntimeDirective / RuntimeDirectiveCandidate。
没有 CandidateCollector。
没有从 TaskOperationPreview 生成 ExecutionGraphPreview 的 builder。
没有 AdoptionCandidate / AdoptedResourcePolicy 接线。
```

### 2.2 旧 `QueryRuntime` 仍残留调度权

关键文件：

```text
backend/query/runtime.py
```

旧调度权残留：

```text
_stream_single_execution()
  调 planner.build_plan() 并拿 QueryExecutionPlan。

_stream_planned_execution()
  根据 execution.execution_kind 选择 worker/direct_tool/model。

execution.worker_plan
  直接进入 EvidenceOrchestrator。

execution.query_understanding.tool_name
  直接进入 RuntimeToolBridge。

_allowed_tool_names_for_execution()
  仍从 execution / skill / permission 推模型可见工具。

_build_output_commit_plan()
  目前是 removed 状态，但 _apply_output_commit_plan 仍可 append assistant messages。
```

这些都应降级为候选或执行层适配器，不能再是执行真相。

### 2.3 `RuntimeToolBridge` 仍是旧工具执行入口

关键文件：

```text
backend/query/runtime_tools.py
```

现实问题：

```text
allowed_tool_names_for_execution() 从 QueryExecutionPlan 推工具范围。
stream_direct_tool_execution() 直接检查 tool contract / permission / tool_runtime.get_instance。
通过 task_coordinator.run_tool_task() 执行并记录 task。
直接产出 tool_start / tool_end / done。
```

编排系统完成后，`RuntimeToolBridge` 只能变成 `ToolExecutor` 的内部适配器，入口必须来自 RuntimeDirective。

### 2.4 Worker 仍由旧 execution 直接驱动

关键文件：

```text
backend/query/evidence_orchestrator.py
backend/query/worker_models.py
```

现实问题：

```text
EvidenceOrchestrator.stream_execution() 直接消费 QueryExecutionPlan + WorkerExecutionPlan。
WorkerResult 可以产出 done event。
done event 仍包含 answer_channel / answer_source / memory_policy。
```

目标：

```text
WorkerExecutionPlan 降级为 WorkerCandidate。
WorkerExecutor 只接受 RuntimeDirective。
WorkerResultEnvelope 只返回 result refs / artifact refs / diagnostics。
最终答案归 OutputBoundary / CommitGate / ControlKernel 后续阶段处理。
```

### 2.5 TaskCoordinator 仍混合生命周期和推断

关键文件：

```text
backend/tasks/coordinator.py
```

现实问题：

```text
_derive_task_bindings() 从 query 文本猜 active_pdf / active_dataset。
run_query_tasks() 直接 mark_running / mark_completed / persist_result_ref。
run_tool_task() 执行 runner 并直接落 result_ref。
```

目标：

```text
TaskCoordinator 记录 TaskRecord / TaskStageRecord。
绑定来自 TaskContract / ExecutionNode / ArtifactRef。
写回结果必须先进入 CommitCandidate。
```

### 2.6 QueryPlanner 应降级为候选生产者

关键文件：

```text
backend/query/planner.py
```

现实问题：

```text
build_plan() 产 QueryPlan，并决定 execution_mode。
_build_execution() 可直接决定 execution_kind = worker / direct_tool / agent。
_worker_plan_from_dispatch() 可直接生成 WorkerExecutionPlan。
```

目标：

```text
QueryPlannerCandidate:
  route hints
  subtask candidates
  legacy execution snapshots
  confidence / reasons

不得再产出可执行 ExecutionPlan。
```

## 3. 编排系统目标对象

### 3.1 CandidateCollector

职责：

```text
收集任务、操作、旧 planner、follow-up、memory、capability 的候选。
统一包装成 CandidateEnvelope。
不做采纳。
不执行。
```

输入：

```text
TaskOperationPreview
history / session refs
legacy QueryPlan snapshot
followup_resolution snapshot
memory context snapshot
capability manifest summary
```

输出：

```text
CandidateSet
```

### 3.2 OrchestrationPlanPreview

编排层计划预览，不是执行授权。

字段建议：

```python
@dataclass(slots=True, frozen=True)
class OrchestrationPlanPreview:
    plan_id: str
    task_id: str
    source: str
    task_contract_ref: str
    task_prompt_contract_ref: str
    resource_policy_ref: str
    prompt_manifest_ref: str
    selected_candidates: tuple[str, ...] = ()
    topology: str = "single"
    stages: tuple[OrchestrationStagePreview, ...] = ()
    diagnostics: dict[str, Any] = field(default_factory=dict)
    authority: str = "orchestration_plan_preview"
    preview_only: bool = True
    adopted: bool = False
    runtime_executable: bool = False
```

约束：

```text
preview_only == true
adopted == false
runtime_executable == false
```

### 3.3 OrchestrationStagePreview

阶段预览，表示“如果未来执行，会需要什么节点”。

字段建议：

```python
stage_id
task_mode
stage_goal
candidate_refs
required_operation_refs
resource_policy_ref
executor_hint
depends_on
blocked_reason
preview_only = True
```

### 3.4 PlanValidationResult

验证计划是否能进入下一阶段。

第一版固定 fail-closed，但要把原因讲清楚：

```python
plan_id
status: "blocked" | "valid_preview"
reason
checks[]
can_adopt_resource_policy = False
can_build_runtime_directive = False
runtime_executable = False
```

第一阶段 checks：

```text
task_contract_exists
resource_policy_preview_only
resource_policy_not_adopted
operation_gate_required
runtime_directive_disabled
commit_gate_required
```

### 3.5 ExecutionGraphPreview

不要复用 `ExecutionNode` 表示 preview 节点，因为当前 `ExecutionNode.authority` 固定要求 `runtime_directive`。

新增：

```python
@dataclass(slots=True, frozen=True)
class ExecutionNodePreview:
    node_id: str
    node_type: str
    executor_hint: str
    stage_ref: str
    operation_refs: tuple[str, ...] = ()
    policy_refs: tuple[str, ...] = ()
    depends_on: tuple[str, ...] = ()
    executable: bool = False
    blocked_reason: str = "preview_only"
    authority: str = "preview_only"
```

第一阶段：

```text
ExecutionGraph.nodes 继续为空。
ExecutionGraph.refs 中记录 preview_node_refs / blocked_reason。
或新增 ExecutionGraphPreview 独立对象。
```

建议先采用独立 `ExecutionGraphPreview`，避免污染 `ExecutionGraph` 的执行语义。

### 3.6 RuntimeDirectiveCandidate

候选指令，不是执行指令。

字段建议：

```python
directive_candidate_id
stage_ref
executor_type
operation_ref
resource_policy_ref
input_schema_ref
output_contract_ref
blocked_reason
preview_only = True
runtime_executable = False
authority = "candidate_only"
```

禁止：

```text
RuntimeDirectiveCandidate 被 ToolExecutor / WorkerExecutor 消费。
RuntimeDirectiveCandidate 写入 ExecutionNode.directive_ref。
```

### 3.7 AdoptionCandidate

后续从 preview 进入 adopted 的桥，不在第一阶段放行。

职责：

```text
记录“哪些资源策略、阶段计划、指令候选理论上可采纳”。
默认 blocked。
```

必须区分：

```text
ResourcePolicyPreview != AdoptedResourcePolicy
OrchestrationPlanPreview != OrchestrationPlan
RuntimeDirectiveCandidate != RuntimeDirective
```

## 4. 固定主链设计

### 4.1 第一阶段主链

```text
QueryRuntime._execution_events()
  -> build_task_runtime_contract_preview()
  -> CandidateCollector.collect_preview_candidates()
  -> OrchestrationCoordinator.build_preview_plan()
  -> PlanValidator.validate_preview()
  -> ExecutionGraphPreviewBuilder.build()
  -> ControlKernel.collect(preview_context)
  -> emit task_operation_preview
  -> emit orchestration_plan_preview
  -> emit orchestration_control
  -> blocked / preview_only
```

第一阶段输出事件：

```text
task_operation_preview
candidate_set_preview
orchestration_plan_preview
plan_validation
execution_graph_preview
orchestration_control
error preview_only
```

### 4.2 第二阶段主链

第二阶段仍不真实执行，只允许生成 adoption report。

```text
ResourcePolicyPreview
  -> AdoptionCandidate
  -> PlanValidator
  -> blocked unless all hard checks pass
```

### 4.3 第三阶段主链

仅当验证器稳定后，才做只读执行试点：

```text
AdoptedResourcePolicy
  -> ExecutionGraph.nodes
  -> RuntimeDirective
  -> OperationGate final check
  -> ReadOnlyExecutor
  -> ResultCandidate
  -> CommitCandidate denied by default
```

## 6. 迁移与切换规则

### 6.1 Shadow / Preview 阶段

当前阶段：

```text
新编排系统只生成 preview artifacts。
旧执行链不恢复。
真实主链 blocked。
```

### 6.2 Cutover 条件

只有满足下面条件，才能进入只读执行试点：

```text
PlanValidator 有硬规则。
AdoptedResourcePolicy 与 ResourcePolicyPreview 分离。
RuntimeDirective 只能由 ControlKernel 生成。
OperationGate 执行前复核通过。
CommitCandidate 默认 denied。
QueryRuntime 不再从 query_understanding.tool_name / worker_plan 直接执行。
```

### 6.3 Rollback 规则

如果新编排 preview 出错：

```text
回到 blocked / preview_only。
不得回旧 direct_tool / worker 执行链。
不得恢复 RuntimeToolBridge 自动推断执行。
```

## 7. 验收标准

每一阶段必须检查：

```text
OrchestrationPlanPreview.preview_only == true
PlanValidationResult.runtime_executable == false
ExecutionGraphPreview 不产生 ExecutionNode(authority=runtime_directive)
RuntimeDirectiveCandidate.authority == candidate_only
ResourcePolicyPreview.adopted == false
ControlKernel.status == blocked
QueryRuntime 不调用 ToolExecutor / WorkerExecutor
RuntimeToolBridge 不被主链直接调用
TaskCoordinator 不因 preview mark_running / mark_completed
CommitCandidate.allowed == false
```

防回退检查：

```text
query_understanding.tool_name 不得成为执行入口。
execution.execution_kind 不得成为执行入口。
worker_plan 不得直接进入 EvidenceOrchestrator。
allowed_tool_names_for_execution 不得控制模型可见工具。
WorkerResult.done 不得直接成为 final answer。
session_manager.append_messages 不得绕过 CommitGate。
```

## 8. 当前建议

下一步先做 Phase A 和 Phase B：

```text
1. 新增 OrchestrationPlanPreview / PlanValidationResult / ExecutionGraphPreview。
2. 新增 CandidateCollector。
3. 让真实 QueryRuntime 在 fail-closed 前输出 orchestration_plan_preview。
```

暂时不要做：

```text
不要接真实工具执行。
不要修旧 query_runtime_route_guard 大测试。
不要让 RuntimeToolBridge 重新成为主链入口。
不要先改 PDF worker / structured worker。
不要让 TaskCoordinator 继续承担调度职责。
```

最短施工口径：

```text
先让编排计划可见。
再让编排验证可解释。
最后才让执行指令可采纳。
```

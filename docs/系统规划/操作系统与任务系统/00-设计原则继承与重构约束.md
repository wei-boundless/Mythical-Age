# 任务系统重构设计原则继承与约束

日期：2026-04-29  
定位：任务系统、操作系统与 ControlKernel 后续重构的原则入口。  
依据：`docs/设计原则/00-25`、`docs/01-洪荒时代模块化子单元系统重设计方案-20260427.md`、`docs/02-先进调度架构对齐与洪荒时代编排重写方案-20260427.md`、灵魂系统已完成架构。

---

## 0. 结论

后续任务系统和操作系统重构，不是重新发明一套抽象，而是把 `docs/设计原则` 中已经沉淀的 Claude Code 工程模式迁移到洪荒时代的新控制平面里。

核心口径：

```text
设计原则
  是底层工程约束。

系统规划
  是洪荒时代的新架构方向。

灵魂系统
  是已完成的身份、投影和 prompt manifest 层。

任务系统 / 操作系统
  是下一步把任务契约、资源边界、执行图和写回门接起来。
```

一句话：

```text
任务系统负责把“要做什么”收束成契约。
操作系统负责把“能碰什么”收束成资源边界。
ControlKernel 负责把候选采纳为执行真相。
ExecutionGraph 负责把执行拓扑显式化。
CommitGate 负责把所有写回延迟到最后统一裁决。
ProjectionSystem 只负责把已裁剪的契约低噪声呈现给模型。
```

## 1. 当前重构必须继承的总原则

### 1.1 单一事实源

来自设计原则：

```text
工具注册表只有一个 getAllBaseTools。
任务状态只有一个 TaskState。
Prompt section 有 manifest 和 cache boundary。
权限决策走统一 pipeline。
```

迁移到洪荒时代：

```text
当前轮执行真相只能来自 ControlKernel 采纳后的 OrchestrationPlan / ExecutionGraph / RuntimeDirective。

TaskContract 是任务真相。
ResourcePolicy 是资源授权真相。
ExecutionGraph 是执行拓扑真相。
OutputCommitPlan 是写回真相。
PromptManifest 是模型可见上下文来源真相。
```

禁止：

```text
QueryPlanner 直接决定执行。
Runtime 根据 query_understanding.tool_name 直接调工具。
Followup / restore / memory 直接覆盖当前任务。
Projection / Soul 直接声明工具权限。
Skill prompt 直接覆盖 TaskContract。
```

### 1.2 Candidate 不等于 Decision

来自设计原则和系统规划：

```text
上游模块可以提出候选。
只有控制平面能采纳候选。
```

迁移对象：

```text
TaskFamilyCandidate
TaskModeCandidate
TaskSoulRequirement
SoulSelectionCandidate
AgentProfileCandidate
CapabilityCandidate
OperationRequirementCandidate
MemoryCandidate
FollowupCandidate
RestoreCandidate
```

约束：

```text
所有 candidate 默认 authority = candidate_only。
candidate 不能含 runtime_directive。
candidate 不能绕过 ResourcePolicy。
candidate 不能触发 CommitGate 写回。
candidate 可以进入 trace，但不能进入 execution plane。
```

### 1.3 Control Plane / Execution Plane 分离

来自 `docs/02-先进调度架构...`：

```text
控制平面决定怎么做。
执行平面只按 directive 做。
```

迁移分层：

```text
Control Plane:
  TaskContract
  TaskBinding
  ResourcePolicy
  MemoryPolicy
  ContextPolicy
  AnswerPolicy
  ExecutionGraph
  RuntimeDirective

Execution Plane:
  ModelExecutor
  ToolExecutor
  WorkerExecutor
  BoundedAgentExecutor
```

禁止：

```text
QueryRuntime 自己判断 worker/direct_tool/model。
ToolExecutor 自己扩大 allowed tools。
WorkerExecutor 自己决定 final answer ownership。
AgentExecutor 自己接管主会话。
```

### 1.4 Fail-Closed 默认

来自工具系统、BashTool、权限系统和 ControlKernel 清线方案：

```text
未知工具 blocked。
未知 worker blocked。
缺 binding blocked。
缺 policy blocked。
权限不明确 blocked 或 requires_approval。
validator blocked 后不回旧链路。
```

迁移到第一阶段：

```text
TaskBinding 缺失时，不生成真实 RuntimeDirective。
ResourcePolicy 缺失时，不允许工具执行。
CompositePlan 未通过 validator 时，不转 staged contract。
CommitCandidate 默认 allowed = false。
Projection 不得把 denied tool 呈现为可用。
```

## 2. 对任务系统的继承约束

### 2.1 任务系统不是分类器，而是契约整理中心

来自 `docs/设计原则/14-任务系统.md` 的可迁移点：

```text
任务系统管理生命周期。
任务状态必须可追踪、可终止、可通知、可清理。
任务失败不应污染其他任务。
任务输出要通过统一通道进入主循环。
```

洪荒时代的任务系统要升级为：

```text
TaskDefinition
  定义一类任务是什么。

TaskBinding
  定义当前这类任务怎么配置。

TaskPromptContract
  定义本次交给模型看的最小可执行说明。

TaskRecord
  记录任务生命周期、阶段、状态、产物引用和 trace。
```

边界：

```text
TaskDefinition 不是执行计划。
TaskBinding 不是工具授权。
TaskPromptContract 不是最终 prompt。
TaskRecord 不是调度权威。
```

### 2.2 BasicTask 是骨架

第一批 basic task 应保持少而清晰：

```text
basic.request_intake
basic.information_search
basic.local_material_read
basic.information_synthesis
basic.task_execution
basic.inspection_and_correction
basic.final_response
```

这些 basic task 只回答：

```text
当前最小任务动作是什么。
输入是什么。
输出是什么。
完成标准是什么。
需要哪些 skill refs。
可能需要哪些 operation requirements。
默认是否需要 review。
```

它们不回答：

```text
具体调用哪个工具。
具体用哪个 worker。
具体由哪个灵魂拥有权限。
最终是否写回 memory。
```

### 2.3 CompositeTask 是受控规划，不是穷举模板

来自当前讨论的新定案：

```text
复合任务不能靠系统提前穷举所有流程。
模型可以规划组合，但系统必须验证边界。
```

新增核心对象：

```text
CompositeTaskPolicy
  定义模型允许怎样规划复合任务。

CompositePlan
  模型产出的阶段计划。

CompositePlanValidator
  校验阶段、资源、权限、review、approval 和最大复杂度。

StagedTaskPromptContract
  通过验证后的每阶段运行时契约。
```

第一版字段：

```text
CompositeTaskPolicy:
  policy_id
  enabled
  planning_mode
  allowed_basic_tasks
  denied_basic_tasks
  allowed_skill_scope
  denied_skill_scope
  allowed_tool_scope
  denied_tool_scope
  projection_policy
  review_policy
  max_steps
  max_parallelism
  requires_approval_when
```

禁止：

```text
不要为 read_and_summarize_document、search_then_write、modify_then_review 等全部手写模板。
不要让模型生成的 CompositePlan 直接进入执行。
不要让 CompositeTaskDefinition 变成唯一复合任务入口。
```

### 2.4 TaskPromptContract 是模型可见任务契约

TaskPromptContract 必须成为任务系统给模型的主要入口。

第一版最小 section：

```text
task_section
  当前目标、任务类型、阶段目标。

method_section
  来自 SkillRuntimeView 的方法摘要。

resource_section
  来自 ResourceRuntimeView 的资源边界摘要。

projection_section
  当前需要的承载姿态要求。

output_section
  输出格式、完成标准、artifact refs。

guardrail_section
  禁止事项、失败处理、冲突策略。
```

装配顺序：

```text
TaskDefinition
  -> TaskBinding
  -> SkillRuntimeView
  -> ResourceRuntimeView
  -> ProjectionRequirement
  -> MemoryPolicySummary
  -> OutputContract
  -> TaskPromptContract
```

然后：

```text
TaskPromptContract
  -> ProjectionSystem
  -> SoulRuntimeView
  -> PromptManifest
```

禁止：

```text
SkillRuntimeView 不直接拼最终 prompt。
ResourceRuntimeView 不授予工具执行权。
ProjectionSystem 不改写 TaskPromptContract 的任务事实。
PromptManifest 不保存执行状态。
```

## 3. 对操作系统的继承约束

### 3.1 操作系统不是工具目录，而是资源边界层

来自工具系统和权限系统：

```text
工具是带契约的微服务。
工具注册表是单一事实源。
工具默认不并发、默认非只读，安全属性必须显式声明。
权限判断走统一管线。
```

洪荒时代的 OperationSystem 应包含：

```text
OperationRegistry
  登记 tool / worker / memory / file / network / shell / session 等可触达资源。

OperationRequirement
  任务提出的资源需求候选。

ResourcePolicy
  ControlKernel 采纳后的资源授权结果。

ResourceRuntimeView
  给模型看的资源摘要。

OperationGate
  真实执行前的权限、路径、风险、确认检查。
```

关键区分：

```text
ResourcePolicy 是执行事实。
ResourceRuntimeView 是模型可见摘要。
模型看到摘要，不等于获得权限。
```

### 3.2 工具继续是微服务，不是任务系统内部方法

从 `docs/设计原则/09-工具系统设计.md` 继承：

```text
Tool 不只是 call。
Tool 有 input schema、output schema、权限检查、风险标签、只读/破坏性/并发安全声明、结果序列化。
```

迁移要求：

```text
ToolContract 必须进入 CapabilityManifest。
ToolExecutionContract 只能由 RuntimeDirective 引用。
ToolOutputContract 必须进入 execution result。
Tool risk tags 必须进入 ResourcePolicy validator。
```

禁止：

```text
任务系统直接保存 tool implementation。
Skill prompt 直接声明工具可用。
ProjectionCard 保存 visible tools 的最终授权。
RuntimeToolBridge 继续从 query_understanding.tool_name 推断工具范围。
```

### 3.3 权限系统必须防御纵深

从 `docs/设计原则/16-权限系统.md` 和 `10-BashTool-深度剖析.md` 继承：

```text
deny 优先。
ask 可以绕过默认 allow。
bypass 不是万能。
safety check 在高信任模式下仍要有不可绕过层。
路径、命令、网络、副作用都需要独立校验。
```

迁移到 OperationGate：

```text
OperationGate:
  1. validate input
  2. check denied resource
  3. check task resource scope
  4. check agent / worker ownership
  5. check operation risk
  6. check path / command / network safety
  7. check approval requirement
  8. emit allow / deny / requires_approval
```

默认：

```text
unknown -> deny
high_risk -> requires_approval
destructive -> requires_approval
external_open_world -> requires_approval 或 restricted
missing_policy -> deny
```

### 3.4 MCP / 外部工具只提供能力，不提供授权

从 `docs/设计原则/15-MCP-协议实现.md` 继承：

```text
外部工具可以被发现、包装、连接、缓存。
外部 tool annotations 只能作为 hint。
授权必须由本地 ResourcePolicy / OperationGate 决定。
```

迁移要求：

```text
MCP tool -> ToolContract wrapper。
MCP annotations -> risk hints。
MCP server config -> OperationRegistry source。
MCP auth state -> resource availability。
MCP tool execution -> RuntimeDirective only。
```

禁止：

```text
外部 MCP annotation 直接放行 destructive 操作。
MCP server 自带的 schema 直接变成本地权限事实。
Agent frontmatter 中的 mcpServers 绕过 ResourcePolicy。
```

## 4. 对 Agent / Worker 的继承约束

### 4.1 Agent 是 bounded specialist，不接管主会话

来自 `docs/设计原则/12-Agent-系统.md`：

```text
子 Agent 默认隔离。
共享必须显式 opt-in。
子 Agent 可以拥有独立工具集、权限模式、记忆范围。
后台 Agent 没有 UI 确认能力时必须自动拒绝或走 hook。
```

洪荒时代迁移：

```text
AgentProfile
  定义角色、工具边界、memory scope、isolation mode、output contract。

BoundedAgentExecutor
  只执行 ExecutionNode(agent)。

AgentResult
  只返回 artifact refs、summary、diagnostics，不直接写最终答案。
```

禁止：

```text
AgentProfile 直接执行。
Agent handoff 接管主 session。
子 Agent 直接写 durable memory。
子 Agent 的 allowed_tools 越过 ResourcePolicy。
```

### 4.2 Worker 是执行器，不拥有最终答案

来自先进调度文档：

```text
worker 可以处理 PDF、retrieval、structured data、tool-like operation。
worker 返回 artifact / evidence / result handle。
主会话 final answer ownership 不下放。
```

迁移对象：

```text
WorkerContract
WorkerExecutionNode
WorkerResultEnvelope
ArtifactRef
EvidenceRef
```

WorkerResult 必须包含：

```text
task_status
artifact_refs
result_handle_ids
diagnostics
memory_commit_candidates
output_contract_status
```

禁止：

```text
PDF worker 自行接管当前轮。
Retrieval worker 直接生成最终回答。
Structured data worker 与 direct tool 形成双主链。
```

## 5. 对 Prompt / Context / Projection 的继承约束

### 5.1 Prompt 必须分段、可追踪、可缓存

来自 `docs/设计原则/04-System-Prompt-工程.md` 和 `07-Prompt-Cache.md`：

```text
Prompt 不是一整坨字符串。
Prompt section 要有 owner、source、cache scope、visible flag。
动态段和静态段要明确。
不稳定内容要尽量从稳定 prompt 中移出。
```

洪荒时代迁移：

```text
TaskPromptContract
  是任务系统的模型可见契约来源。

SoulRuntimeView
  是投影系统装配后的模型可见视图。

PromptManifest
  记录每段 prompt 来源、owner、hash、cache_scope、visible_to_model。
```

约束：

```text
任何进入模型的任务、skill、resource、memory、projection 文本都必须能追踪来源。
PromptManifest 不拥有事实，只记录装配结果。
ProjectionSystem 只能消费裁剪后的视图，不直接读全量 registry。
```

### 5.2 上下文只装配，不调度

来自 `docs/设计原则/06-上下文管理.md`：

```text
上下文管理负责压缩、恢复、预算和注入。
它不能改变当前任务目标。
compact 后要重建必要上下文，但不能恢复旧 route。
```

洪荒时代迁移：

```text
ContextPolicy:
  required_handles
  evidence_budget
  prompt_sections
  compaction_mode
  restore_indexes
```

禁止：

```text
Context restore 改写 TaskContract。
Memory restore 覆盖 current user goal。
PromptBuilder 自己选择 tool / worker。
compact summary 变成执行计划。
```

### 5.3 Projection 是第三层降噪，不是人格装饰

来自灵魂系统完成态：

```text
TaskSystem 第一层降噪：理解当前任务。
SkillSystem 第二层降噪：裁剪当前方法。
ProjectionSystem 第三层降噪：以合适承载姿态呈现信息。
```

投影只管理：

```text
ProjectionCard
ProjectionState
ProjectionRuntimePosture
SoulRuntimeView
PromptManifest
```

投影不管理：

```text
TaskDefinition
TaskGraph
SkillRegistry
ToolRegistry
ResourcePolicy
Tool authorization
Memory write decision
RuntimeDirective
CommitGate
```

## 6. 对 Memory / Commit 的继承约束

### 6.1 Memory 读写分离

来自 `docs/设计原则/23-Memory系统.md`：

```text
记忆有分层。
记忆要按需召回。
后台提取要和主 Agent 写入互斥。
可派生信息不应写入长期记忆。
```

洪荒时代迁移：

```text
MemoryPolicy
  决定当前任务能读哪些记忆、是否允许提交写入候选。

MemoryRuntimeView
  给模型看的记忆摘要。

MemoryCommitCandidate
  任务完成后提交给 CommitGate 的写入候选。
```

禁止：

```text
SoulProfile 直接写长期记忆。
ProjectionState 直接写回 SoulProfile。
TaskPromptContract 直接触发 durable memory extraction。
WorkerResult 直接落 durable memory。
```

### 6.2 CommitGate 是唯一写回门

来自系统规划和错误恢复原则：

```text
所有副作用必须延迟到 commit。
写回候选默认 denied。
失败、blocked、fallback 状态默认不写 durable memory。
```

迁移对象：

```text
CommitCandidate
OutputCommitPlan
MemoryCommitCandidate
TaskResultCommitCandidate
ArtifactCommitCandidate
```

写回类型：

```text
session_message
task_result_ref
artifact_graph
session_projection
durable_memory
title
post_turn_action
```

禁止：

```text
QueryRuntime 直接 append assistant message。
MemoryFacade 在任务未完成时默认写 durable memory。
TaskCoordinator 自己写 result_ref 而不经过 commit plan。
OutputPolicy 旁路写回。
```

## 7. 对配置、扩展和前端管理的继承约束

### 7.1 Settings 是多层配置，不是执行事实

来自 `docs/设计原则/17-Settings-系统.md`：

```text
配置源有优先级。
项目级配置不能天然被信任。
企业策略可以覆盖用户和项目配置。
数组类规则通常拼接去重，标量覆盖。
```

迁移到任务系统：

```text
TaskBinding 可以来自 builtin / plugin / project / user / policy。
ResourcePolicy 最终仍由 ControlKernel + Validator 生成。
project 级 binding 不得自动放行高风险资源。
policy 级 binding 可以强制 deny 或 requires_approval。
```

### 7.2 Skill / Plugin 是扩展点，不是主链权威

来自 `docs/设计原则/24-Skill-Plugin开发实战.md`：

```text
Skill 是可复用方法片段。
Agent 是可配置角色。
Plugin 可贡献 skill / agent / hook / mcp server。
```

洪荒时代约束：

```text
SkillSystem 提供 SkillRuntimeView。
Plugin 可以贡献 TaskDefinition / TaskBinding / Skill / AgentProfile。
但所有执行仍必须经过 ResourcePolicy 和 RuntimeDirective。
```

禁止：

```text
Plugin 直接注册 RuntimeDirective。
Skill frontmatter 直接声明写权限。
Agent markdown 直接覆盖 ResourcePolicy。
```

### 7.3 前端只能管理配置草案，不拥有执行真相

来自系统规划：

```text
Frontend Operations 展示和编辑能力资源。
执行真相在后端 ControlKernel。
```

前端可以做：

```text
展示 TaskDefinition / TaskBinding。
编辑 binding draft。
展示 ResourcePolicy preview。
展示 PromptManifest preview。
展示 ExecutionGraph preview。
触发保存配置。
```

前端不能做：

```text
直接授权工具。
直接执行 worker。
直接写 memory。
绕过 validator 修改 execution graph。
```

## 8. 目标对象映射表

| 设计原则对象 | 洪荒时代目标对象 | 说明 |
| --- | --- | --- |
| Tool registry 单一来源 | OperationRegistry / CapabilityManifest | 能力存在事实 |
| Tool contract | ToolContract / OperationContract | 工具微服务契约 |
| Permission pipeline | ResourcePolicy / OperationGate | 资源授权与执行前检查 |
| TaskState 生命周期 | TaskRecord / TaskStageRecord | 任务状态与产物引用 |
| AgentDefinition | AgentProfile | 角色、工具边界、模型、记忆范围 |
| createSubagentContext | BoundedAgentExecutor isolation | 默认隔离、显式共享 |
| Prompt section | TaskPromptContract / SoulRuntimeView | 模型可见分段 |
| Prompt cache boundary | PromptManifest.cache_scope | section 稳定性和缓存范围 |
| Relevant memories | MemoryRuntimeView | 按需召回而非全量注入 |
| Background extraction | MemoryCommitCandidate | 后台候选，不直接写 |
| Hook permission request | ApprovalPolicy / OperationGate | headless 场景的外部审批 |
| AsyncGenerator execution | RuntimeExecutor event stream | 执行层流式事件 |
| Retry / fallback | FailurePolicy | 错误恢复，不越权写回 |
| Settings priority | BindingSourcePriority | 多源配置覆盖规则 |
| Output persistence | CommitGate | 唯一写回门 |

## 10. 验收标准

这一轮重构每推进一层，都应满足：

```text
1. 是否有单一事实源。
2. 是否区分 candidate 与 decision。
3. 是否 fail-closed。
4. 是否没有旧 planner/runtime 旁路。
5. 是否没有 prompt 抢权。
6. 是否没有 soul / projection 授权。
7. 是否没有 skill / plugin 直接执行。
8. 是否所有模型可见内容都有 manifest 来源。
9. 是否所有副作用都进入 CommitCandidate。
10. 是否测试能验证 plan 与 actual 的差异。
```

只要其中一条不满足，就说明设计正在滑回旧链路。

## 11. 当前定稿口径

```text
TaskSystem
  负责任务真相、任务配置、任务运行时契约和任务生命周期。

OperationSystem
  负责资源登记、资源需求、资源授权、执行前检查和模型可见资源摘要。

SkillSystem
  负责按任务层级提供静态方法片段。

ProjectionSystem
  负责将已裁剪的任务、方法、资源、记忆和输出边界以合适姿态呈现。

ControlKernel
  负责采纳候选并生成唯一执行真相。

ExecutionRuntime
  负责只按 RuntimeDirective 执行。

CommitGate
  负责最终写回裁决。
```

最短口径：

```text
任务系统管契约。
操作系统管边界。
控制内核管采纳。
执行层管执行。
提交门管写回。
灵魂投影管呈现。
```

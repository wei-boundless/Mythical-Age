# 任务系统与运行时架构差距总报告

日期：2026-05-02

## 一、报告目的

本报告不是重复描述“还没做完什么”，而是明确说明当前任务系统与主流先进架构之间仍然存在的结构差距，并据此锁定后续分阶段重构顺序。

本报告服务三个目标：

- 统一当前项目对“四个结构问题”的判断口径。
- 把问题从实现细节上升到状态所有权、恢复语义、任务契约、验证闭环四个系统级约束。
- 为后续实施文档提供固定入口，避免边做边改、局部补丁不断累积。

配套分报告：

- `17-问题一-任务运行账本与步骤状态机差距报告-20260502.md`
- `18-问题二-恢复重放与副作用幂等性差距报告-20260502.md`
- `19-问题三-模板选择与任务拆分契约化差距报告-20260502.md`
- `20-问题四-架构验证与故障演练差距报告-20260502.md`

## 二、总判断

当前系统已经具备先进任务运行时的主骨架，但还没有完成从“可记录”到“可恢复、可重放、可证明”的跃迁。

已经站稳的部分：

- 已有 `CurrentTurnContext`、`TaskTemplate`、`TaskSpec`、`TaskRunLedger`、`TaskResult` 的分层雏形。
- 已有 runtime loop、checkpoint、trace、commit gate、session writeback 的基础闭环。
- 已经开始把任务、上下文、结果账本从 prompt 解释层收回到系统结构层。

仍未站稳的部分：

- `TaskRunLedger` 还不是执行真相源，步骤状态更多是结束后推断出来的。
- checkpoint 已经存在，但恢复与副作用去重语义还没有封死。
- 模板选择与 bundle 拆分仍偏启发式，缺少强输入输出契约。
- 验证体系主要覆盖 happy path 和静态适配，尚未形成故障演练闭环。

因此当前阶段更准确的判断是：

- 架构方向正确。
- 结构基础扎实。
- 工程闭环尚未硬化。
- 继续推进是对的，但必须按结构问题顺序推进，不能靠零散补洞完成。

## 三、四个核心差距

### 3.1 差距一：任务账本仍是结果描述层，不是执行真相层

当前 `TaskRunLedger`、`TaskResult` 已经接入运行时，但 `TaskStepRun.status` 主要仍在 loop 结束时根据 observation、final answer、terminal reason 反推。

这意味着系统已经能“描述最后发生了什么”，但还不能稳定“驱动每一步应该怎么推进、暂停、恢复、重试、取消”。

对应分报告：

- `17-问题一-任务运行账本与步骤状态机差距报告-20260502.md`

### 3.2 差距二：恢复链已出现骨架，但还没有 replay-safe 语义

当前已经有 checkpoint、event log、trace summary，但 shell、edit、write、worker 这类副作用动作还没有被统一包进可重放协议。

这意味着系统已经在“记录执行”，但还没有真正做到“中断后恢复时不会重复执行副作用”。

对应分报告：

- `18-问题二-恢复重放与副作用幂等性差距报告-20260502.md`

### 3.3 差距三：模板选择与任务拆分仍由启发式主导

当前模板选择、bundle 拆分、能力域归类已经能跑通，但主要依赖 route hint、关键词、模态和显式输入推断。

这意味着系统已经能“猜对很多任务”，但还没有把父任务、子任务、步骤输入输出、共享状态边界固化成强契约。

对应分报告：

- `19-问题三-模板选择与任务拆分契约化差距报告-20260502.md`

### 3.4 差距四：验证矩阵偏静态适配，故障工程证据不足

当前验证能说明模板、agent、operation 的装配关系基本有效，也能通过 trace 查看运行结果。

但这只能证明“结构被接上了”，还不能证明“在失败、中断、重试、权限拒绝、积压、半完成场景下仍然成立”。

对应分报告：

- `20-问题四-架构验证与故障演练差距报告-20260502.md`

## 四、本地代码依据

### 4.1 任务运行账本相关代码

- `backend/tasks/run_models.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`

关键观察：

- `build_task_run_ledger()` 会初始化 step run，但仅提供初始账本。
- `_build_final_task_run_ledger()`、`_build_runtime_task_result()`、`_final_step_runs()` 会在终局阶段综合生成最终步骤状态。
- 当前步骤推进并未形成独立的、逐步持久化的运行状态机。

### 4.2 恢复链和 trace 相关代码

- `backend/orchestration/runtime_loop/checkpoint.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/orchestration/runtime_loop/trace_reader.py`

关键观察：

- checkpoint、event log、trace summary 已经存在。
- `commit_gate_checked`、`loop_terminal` 等事件已经能表达终局语义。
- 但副作用动作并没有统一 idempotency token、replay policy、dedupe contract。

### 4.3 模板选择与 bundle 拆分相关代码

- `backend/tasks/template_registry.py`
- `backend/context_management/resolver.py`

关键观察：

- `select_template()` 依赖 `route_hint`、`execution_posture`、`preferred_skill`、`modality`、`candidate_tools` 等启发式信号。
- `_bundle_items()`、`_looks_compound()`、`_capability_for_text()` 负责把自然语言复合请求拆成 bundle item。
- 当前拆分和模板归类缺乏显式的 parent-child contract 与 state schema 映射层。

### 4.4 验证与回归相关代码

- `backend/tasks/template_registry.py`
- `backend/tests/task_template_registry_regression.py`
- `backend/tests/query_runtime_runtime_loop_regression.py`
- `backend/tests/test_system_runtime_loop_regression.py`

关键观察：

- `build_validation_matrix()` 当前验证的是 agent 能力、operation 注册和模板基本装配有效性。
- 现有回归已经能覆盖若干上下文、运行时、任务契约场景。
- 但针对 crash-resume、duplicate side effect、permission denied、stale backlog、partial completion 的专门故障测试仍缺失。

## 五、与先进架构的主要差距

### 5.1 对齐点

和主流先进架构已经明显对齐的原则：

- 把状态按生命周期拆层，而不是继续让 prompt 兼任事实源。
- 把任务模板、任务实例、运行账本、结果账本拆成不同数据结构。
- 开始把 trace、checkpoint、commit gate 纳入同一运行时框架。
- 逐步把 direct path 与 compound path 收回同一 loop。

### 5.2 未对齐点

与 Temporal / LangGraph / AWS Well-Architected 相比，当前仍有四个关键未对齐点：

- 缺少由步骤状态直接驱动的 durable execution 状态机。
- 缺少针对副作用的 replay-safe 和 idempotent 协议。
- 缺少父任务到子任务之间稳定的 state schema / contract 层。
- 缺少把失败场景当作一级验证对象的工程闭环。

## 六、推荐重构顺序

### Phase 1：把 `TaskRunLedger` 从结果账本提升为执行真相账本

阶段目标：

- 让 step transition 在执行中发生，而不是在结束时综合推断。
- 让 `current_step_id`、step status、step output ref 成为 loop 的第一等控制字段。

优先原因：

- 如果步骤状态仍靠终局推断，后续恢复、重试、取消都没有稳固落点。

### Phase 2：建立恢复与副作用语义

阶段目标：

- 为 mutating operation 引入稳定 request token、result token、dedupe contract。
- 明确 checkpoint 恢复后的 replay 起点和 side effect guard。

优先原因：

- 没有 replay-safe 语义，checkpoint 越深入，重复副作用风险越高。

### Phase 3：把模板选择与 bundle 拆分改造成契约驱动

阶段目标：

- 从“按话术猜模板”升级为“按任务输入、绑定对象、子任务 schema 绑定模板”。
- 为 parent task、bundle item、step blueprint 建立显式输入输出协议。

优先原因：

- 一旦任务模板增多、前端开始装配模板，启发式入口会成为主要不稳定源。

### Phase 4：补齐故障工程验证

阶段目标：

- 建立 crash-resume、retry storm、permission blocked、partial completion、stale queue 的回归矩阵。
- 把验证从静态装配扩展到动态恢复与失败链路。

优先原因：

- 没有这一步，前面三步只能算“架构完成”，还不能算“工程闭环成立”。

## 七、阶段依赖关系

推荐严格按以下顺序推进：

1. 先做步骤状态机。
2. 再做恢复与副作用协议。
3. 再收束模板选择与 bundle 契约。
4. 最后补齐故障演练矩阵。

不建议的顺序：

- 不要先扩模板数量，再修选择机制。
- 不要先宣布支持 crash-resume，再补副作用幂等。
- 不要先做前端模板装配页，再补底层任务契约。

## 八、阶段交付物

每个阶段都应至少产出以下四类交付物：

- 数据模型调整
- runtime loop 行为调整
- 回归测试新增
- 文档和 cutover 规则更新

## 九、验收总口径

本轮四阶段重构完成后，应满足以下统一口径：

- 任一任务运行都能明确回答“当前在哪一步、上一步产出了什么、下一步为什么继续”。
- 任一 checkpoint 恢复都能明确回答“哪些副作用绝不会重放、哪些步骤允许重试、依据是什么”。
- 任一模板选择都能明确回答“为什么是这个模板、不是另一个模板、输入输出契约是什么”。
- 任一失败场景都能明确回答“系统是否 fail fast、是否可恢复、是否会留下脏账本、是否会产生重复副作用”。

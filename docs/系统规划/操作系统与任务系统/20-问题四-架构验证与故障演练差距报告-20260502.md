# 问题四：架构验证与故障演练差距报告

日期：2026-05-02

## 一、问题定义

当前系统已经开始具备可观测性，也有一批回归测试，但验证体系仍然偏向“功能是否接上”和“静态装配是否有效”。

真正缺失的不是测试数量本身，而是：

- 没有把 failure path 当成一级对象；
- 没有把恢复语义当成验收口径；
- 没有把 retry、permission denied、partial completion、queue backlog 作为架构级验证；
- 没有把运行时演练沉淀为固定制度。

这意味着当前系统的工程依据更多来自“设计看起来合理”和“happy path 可以跑通”，而不是“失败后仍能保持结构正确”。

## 二、当前代码证据

核心文件与测试：

- `backend/tasks/template_registry.py`
- `backend/orchestration/runtime_loop/trace_reader.py`
- `backend/tests/task_runtime_contract_regression.py`
- `backend/tests/query_runtime_runtime_loop_regression.py`
- `backend/tests/test_system_runtime_loop_regression.py`
- `backend/tests/context_management_current_turn_regression.py`
- `backend/tests/memory_system_contracts_regression.py`
- `backend/tests/task_template_registry_regression.py`

当前已经具备：

- 模板验证矩阵能校验 agent / capability / operation 基本装配。
- 运行时回归已经覆盖上下文、任务契约、系统 loop、trace summary 的若干核心路径。
- trace 能读出 `task_contract_built`、`commit_gate_checked`、`loop_terminal` 等关键事件。

当前仍然缺失：

- crash-resume 场景验证
- duplicate side effect 验证
- permission blocked 验证
- retry exhaustion 验证
- partial completion / skipped step 验证
- stale queue / delayed worker / degraded dependency 验证

## 三、与先进架构的差距

### 3.1 与 AWS Operational Excellence 的差距

AWS 的 operational excellence 明确强调：

- operations as code
- small reversible changes
- refine procedures frequently
- anticipate failure
- 从所有运行失败中学习

当前差距：

- 我们已经有文档和回归，但还没有把“故障演练流程”固定成工程程序。
- 当前更多是在验证设计是否搭起来，而不是验证运维与恢复程序是否可靠。

### 3.2 与生产级 durable runtime 的差距

成熟 durable runtime 不只看：

- 能不能开始跑；
- 能不能得出结果；

更看：

- 失败时能否快速终止；
- 能否避免积压扩大；
- 能否恢复到正确状态；
- 能否证明没有重复副作用。

当前差距：

- 现有回归更多覆盖功能链路，不足以证明恢复链和故障链已经闭合。

## 四、会带来的具体问题

### 4.1 设计正确但工程上不稳

这类系统最危险的状态不是“完全没设计”，而是：

- happy path 很顺；
- 文档也很完整；
- 真到故障场景时没有足够证据支持。

### 4.2 迭代越快，隐藏故障越多

如果在没有故障矩阵的情况下继续扩：

- 更多模板；
- 更多执行器；
- 更多 worker；
- 更多恢复入口；

隐藏风险会被放大，而不是均匀增加。

### 4.3 变更难以安全 cutover

没有固定的故障演练和 rollback 验收口径，后续阶段改造就容易出现：

- 旧路径未关干净；
- 新路径在失败时回退不明确；
- 文档完成了，但切流依据不足。

## 五、目标设计

### 5.1 建立四层验证矩阵

建议把验证拆成四层：

- 静态契约验证
- 运行时 happy path 验证
- 故障注入验证
- 恢复与回滚验证

### 5.2 建立故障场景目录

至少覆盖：

- process crash after side effect before checkpoint
- checkpoint exists but result not committed
- permission denied mid-step
- worker timeout / unavailable
- repeated retry with non-idempotent action
- queue backlog / stale request
- partial completion with allowed skip

### 5.3 建立 cutover 和 rollback 规则

每个阶段改造都必须先定义：

- 新旧路径并存窗口
- 切换条件
- 回滚条件
- 哪些 trace / metrics / tests 作为切换依据

## 六、分阶段改造建议

### Phase 4A：补静态到动态的验证桥

目标：

- 保留 `build_validation_matrix()`，但把它定位为最外层静态验证，而不是总代表。

### Phase 4B：新增故障注入回归

目标：

- 为恢复、重试、权限、积压和部分完成建立专门测试文件或测试组。

### Phase 4C：建立固定演练制度

目标：

- 每次重要阶段切换前，必须跑固定的恢复与故障回归矩阵。

## 七、验收标准

- 每个阶段都有明确 cutover 前置测试。
- 任一关键副作用路径都有故障注入回归。
- trace 能支撑故障定位，而不仅是事后阅读。
- rollback 规则能在文档和测试中同时落地。

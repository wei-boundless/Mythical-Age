# 操作系统重构实施计划

日期：2026-04-29  
定位：在任务系统契约链路和灵魂投影链路之间，建立资源事实、资源授权、运行时摘要与执行前门控层。  
当前目标：先接通资源边界 preview 链路，不接真实工具执行。

---

## 0. 结论

这里的“操作系统”不是传统 OS，也不是工具目录，而是 agent runtime 里的资源边界层。

它回答：

```text
当前任务能看见哪些资源。
当前任务想触达哪些资源。
哪些资源真的被授权。
哪些资源必须拒绝。
哪些资源需要用户确认。
模型最后能看到怎样的资源摘要。
执行前还要经过哪些门控检查。
```

一句话：

```text
TaskSystem 提出资源需求。
OperationSystem 生成资源边界和授权事实。
ProjectionSystem 只呈现资源摘要。
ExecutionRuntime 只按 RuntimeDirective 触达资源。
CommitGate 统一处理资源触达后的写回。
```

第一阶段链路：

```text
TaskBinding.operation_scope
  -> OperationRequirement
  -> OperationRegistry / CapabilityManifest
  -> ResourcePolicyPreview
  -> ResourceRuntimeView
  -> TaskPromptContract.resource_section
  -> SoulToolView / SoulRuntimeView
  -> PromptManifestPreview
```

当前不做：

```text
不真实调用工具。
不真实执行 worker。
不真实写文件。
不真实写 memory。
不恢复旧 RuntimeToolBridge 推断路径。
```

---

## 1. 操作系统的职责边界

### 1.1 操作系统负责什么

```text
登记可触达资源。
表达任务对资源的需求。
根据任务、binding、agent、policy 生成资源授权结果。
给模型生成低噪声 ResourceRuntimeView。
执行前再次做 OperationGate 检查。
把资源风险、审批要求和拒绝原因显式化。
```

### 1.2 操作系统不负责什么

```text
不定义任务目标。
不生成任务流程。
不选择灵魂。
不装配最终 prompt。
不执行模型推理。
不写 session / durable memory / artifact graph。
不直接生成最终答案。
```

### 1.3 与任务系统关系

任务系统提供：

```text
TaskContract
TaskDefinition.default_operation_requirements
TaskBinding.operation_scope
TaskBinding.denied_operations
TaskBinding.approval_policy
TaskPromptContract.resource_section 输入需求
```

操作系统返回：

```text
OperationRequirement
ResourcePolicyPreview
ResourceRuntimeView
ResourcePolicySummary
OperationDiagnostics
```

关键约束：

```text
TaskBinding 是配置，不是授权事实。
operation_scope 是需求范围，不等于 allowed operations。
ResourcePolicy 才是授权事实。
```

### 1.4 与灵魂系统关系

灵魂系统只消费：

```text
SoulToolView
ResourcePolicy summary
ResourceRuntimeView 的可见摘要
```

灵魂系统不拥有：

```text
tool authorization
worker authorization
memory write authorization
filesystem authorization
network authorization
shell authorization
```

必须保持：

```text
SoulToolView.authorization_owner = "ResourcePolicy"
SoulRuntimeView.authorization_owner = "ResourcePolicy"
Projection 不得把 denied resource 重新写成 authorized。
Projection 不得扩大 visible tool ids。
```

### 1.5 与 ControlKernel 关系

第一阶段：

```text
OperationSystem 只生成 ResourcePolicyPreview。
ControlKernel 仍 fail-closed。
```

后续：

```text
ControlKernel 采纳 ResourcePolicy。
PlanValidator 校验 ResourcePolicy。
ExecutionGraph 节点引用 resource_policy_ref。
RuntimeDirective 引用具体 authorized resource。
```

第一阶段：

```text
ResourcePolicy 可以是 preview。

```

### 2.5 ResourceDecision

单个 operation 的授权结论。

建议文件：

```text
backend/operations/policies.py
```

第一版字段：

```python
@dataclass(slots=True, frozen=True)
class ResourceDecision:
    operation_id: str
    decision: str
    reason: str
    risk_tags: tuple[str, ...] = ()
    requires_user_approval: bool = False
    authorization_owner: str = "ResourcePolicy"
    diagnostics: dict[str, Any] = field(default_factory=dict)
```

decision 枚举：

```text
allow
deny
requires_approval
preview_only
unknown
```

约束：

```text
unknown 在执行前等价 deny。
preview_only 不能进入 RuntimeDirective。
requires_approval 不能被 Projection 改成 allow。
```

### 2.6 ResourceRuntimeView

给模型看的资源摘要。

建议文件：

```text
backend/operations/runtime_view.py
```

与任务系统第一阶段临时定义保持兼容：

```python
@dataclass(slots=True, frozen=True)
class ResourceRuntimeView:
    resource_id: str
    title: str
    capability_summary: str
    authorized: bool = False
    authorization_owner: str = "ResourcePolicy"
    risk_summary: str = ""
    denied_reason: str = ""
    requires_approval: bool = False
    preview_only: bool = True
    metadata: dict[str, Any] = field(default_factory=dict)
```

生成规则：

```text
decision == allow:
  authorized = True
  preview_only = False only after real policy is adopted

decision == requires_approval:
  authorized = False
  requires_approval = True

decision == deny / unknown:
  authorized = False
  denied_reason 必须存在
```

### 2.7 OperationGate

真实执行前的最后门控。

建议文件：

```text
backend/operations/gate.py
```

第一版先定义接口，不接真实执行：

```python
@dataclass(slots=True, frozen=True)
class OperationGateResult:
    operation_id: str
    decision: str
    reason: str
    allowed: bool = False
    requires_approval: bool = False
    diagnostics: dict[str, Any] = field(default_factory=dict)
```

门控顺序：

```text
1. validate operation exists
2. validate directive_ref exists
3. validate ResourcePolicy exists
4. check denied_operations
5. check allowed_operations
6. check operation risk tags
7. check filesystem / network / shell / memory scope
8. check approval requirement
9. return allow / deny / requires_approval
```

第一阶段默认：

```text
OperationGate always preview_only / deny real execution。
```

## 3. 风险模型

### 3.1 风险标签

第一批 risk_tags：

```text
read_only
local_read
local_write
destructive
shell_execution
python_execution
network_open_world
external_fetch
memory_read
memory_write_candidate
session_write_candidate
artifact_write_candidate
worker_execution
agent_execution
requires_human_approval
```

### 3.2 默认策略

```text
read_only:
  可在 preview 中 allow。

local_read:
  仅限任务相关路径或 workspace 内路径。

local_write:
  第一阶段 requires_approval 或 preview_only。

destructive:
  deny 或 requires_approval。

shell_execution:
  requires_approval。

python_execution:
  requires_approval。

network_open_world:
  requires_approval 或 restricted。

memory_write_candidate:
  只允许提交候选，不允许真实写。

session_write_candidate:
  只允许提交 CommitCandidate。

worker_execution:
  第一阶段 preview_only。

agent_execution:
  第一阶段 preview_only。
```

### 3.3 Deny 优先

判定顺序：

```text
explicit deny
  > safety deny
  > missing policy deny
  > requires approval
  > allow
```

这意味着：

```text
只要 operation 在 denied_operations 中，就算同时出现在 allowed_operations，也必须 deny。
```

## 5. 与灵魂投影的接线

### 5.1 输入到 SoulProjectionRequest

操作系统给投影系统提供：

```text
tool_views / resource_views
resource_policy_summary
authorization_owner
```

映射到当前灵魂系统：

```text
ResourceRuntimeView
  -> SoulToolView

ResourcePolicy.summary
  -> SoulProjectionRequest.memory_policy_summary 或 task_contract_summary 中的 resource summary

authorized
  -> SoulToolView.authorized

authorization_owner
  -> "ResourcePolicy"
```

### 5.2 投影呈现规则

Projection 可以做：

```text
用当前投影的语言压缩资源摘要。
强调资源风险。
提示哪些操作需要确认。
提示哪些资源不可用。
调整表达密度。
```

Projection 不能做：

```text
把 denied 改成 allowed。
把 requires_approval 改成 allowed。
新增 tool id。
隐藏关键风险。
生成 RuntimeDirective。
声明 memory write allowed。
```

### 5.3 PromptManifest 要记录来源

资源 section 进入 PromptManifest 时必须记录：

```text
source_type = resource_policy / operation_runtime_view
owner_layer = resource_policy
visible_to_model = true
authorization_owner = ResourcePolicy
```

## 6. 与 ControlKernel / ExecutionGraph 的接线

### 6.1 第一阶段

```text
ResourcePolicyPreview
  -> ControlKernel diagnostics
  -> ExecutionGraph preview refs
  -> status = blocked / preview_only
```

不生成真实 directive：

```text
RuntimeDirective = disabled
ExecutionNode = preview_only 或 empty
```

### 6.2 第二阶段

当 validator 完成后：

```text
ResourcePolicy
  -> PlanValidator
  -> ExecutionGraph.nodes[].policy_refs
  -> RuntimeDirective.allowed_resource_ref
  -> OperationGate
```

ExecutionNode 必须包含：

```text
node_id
node_type
executor
directive_ref
policy_refs
inputs
depends_on
authority = runtime_directive
```

### 6.3 执行前门控

即使 ExecutionGraph 已经生成，真实执行前仍要：

```text
OperationGate.check(node, directive, resource_policy)
```

原因：

```text
计划生成时的 policy 可能过期。
用户权限模式可能变化。
路径可能变成 symlink 或越界。
MCP 连接状态可能变化。
worker / agent 可能不可用。
```

## 7. 与 CommitGate 的接线

操作系统触达资源后产生的副作用不能直接写回。

必须转成：

```text
CommitCandidate
```

资源到 commit 类型映射：

```text
op.write_file / op.edit_file
  -> artifact_commit_candidate 或 file_change_candidate

op.memory_write_candidate
  -> MemoryCommitCandidate

op.session_message_candidate
  -> session_message CommitCandidate

op.artifact_result_ref
  -> artifact_graph CommitCandidate

worker result
  -> task_result_ref / artifact_graph / memory_commit_candidate
```

默认：

```text
CommitCandidate.allowed = False
CommitGate 明确 allow 后才写回。
```

```text
OperationRequirement.authority == "candidate_only"
ResourcePolicy.authority == "resource_policy"

preview_only 不进入 RuntimeDirective
SoulToolView 不扩大 authorized scope

10. 三个用例验收通过。
```

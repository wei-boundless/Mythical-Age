# 健康验证与测试体系详细设计书

日期：2026-05-03

上位计划：`25-健康系统管理控制面与验证体系统筹重构计划书-20260502.md`

## 一、子系统定位

健康验证与测试体系负责把“系统是否健康”从人工判断变成可调度、可复盘、可报告、可切流引用的验证资产。

它不是普通 regression 文件集合，而是健康系统独立控制面的一部分。

它负责：

- 测试目录
- 测试组
- 测试运行
- 测试 artifact
- failure to issue/report
- cutover readiness
- fault injection
- replay/recovery 验证

它不负责：

- 子 agent 灵魂投影配置
- runtime 权限授权
- 健康对话 UI
- 直接修改主运行时真相

## 五、目标 API 与服务关系

### 5.1 健康系统测试触发入口

健康测试不应只从 `/test-system/runs` 直接启动，也不应只依赖健康管理 agent。

健康系统应该同时支持：

- 用户在健康系统原生操作台直接启动测试。
- 健康管理 agent 在嵌入式对话窗中代为启动测试。
- 开发者或 CI 直接运行底层测试。

其中前两类属于健康系统管理动作，应该形成 command/receipt。

建议健康系统提供：

```text
POST /health-system/commands
```

其中：

```json
{
  "command_type": "launch_health_test",
  "payload": {
    "profile": "runtime",
    "scenario_ids": ["health.runtime.recovery_replay"]
  }
}
```

command handler 内部调用 test system。

### 5.2 Test System 仍保留专业能力

`test_system` 继续负责：

- 运行测试
- 维护 profiles
- 维护 managed cases
- 提供 artifacts
- 提供 runtime-loop turn trace

健康系统负责：

- 把测试意图变成 command
- 把测试结果变成 HealthTestRun
- 把失败变成 HealthReport/HealthIssue
- 把关键结果纳入 readiness

## 六、与其它子系统协调

### 6.1 对任务系统

测试体系读取任务系统的：

- health flows
- task modes
- bindings
- workflows
- projection templates
- output contracts

用于生成 static contract 和 health agent management 测试。

### 6.2 对健康后端控制面

测试体系可以被健康系统原生操作台或健康管理 agent 代操触发。需要进入健康后台审计和报告的触发路径，应由 health command 调度。

每次测试启动必须有：

- command id
- receipt id
- health test run id
- test system run id

### 6.3 对编排权限运行层

测试体系必须覆盖编排层不变量。

测试失败不能只记录为 pytest failure，还要转换成健康报告证据。

### 6.4 对健康系统原生操作台与嵌入式 agent 对话窗

用户可以通过健康系统原生操作台直接启动测试，也可以通过嵌入式 agent 对话窗要求健康管理 agent 代为启动测试。

原生操作台可以展示测试目录、运行记录和 report；嵌入式对话窗主要展示 agent 代操产生的 health command/receipt/report，不直接替代 test runner 或健康系统 verify 页面。

## 七、实施步骤

### Phase 30A：健康测试目录

- 新增 `backend/health_system/test_catalog.py`
- 定义 health scenario catalog
- 把现有 test profiles/cases 映射到 health scenario

### Phase 30B：健康测试运行封装

- 新增 `backend/health_system/test_runner.py`
- 输入 command payload
- 调用 `test_system_service.start`
- 记录 `HealthTestRun`
- 查询 artifacts

### Phase 30C：失败报告化

- 测试失败自动生成 `HealthReport`
- 可定位失败关联或创建 `HealthIssue`
- report evidence refs 引用 artifacts、runtime-loop trace、command、receipt

### Phase 30D：Cutover readiness

- 新增 readiness builder
- 汇总关键 scenario 结果
- 输出 readiness report
- 把 blocker 写入 health report

### Phase 30E：前端验证页升级

- `HealthSystemView.tsx` 的 verify 页面展示 health scenarios
- 启动测试改为提交 `launch_health_test` command
- 展示 health test run 与 test system run 的映射
- 展示 readiness 结论

## 八、验收标准

- 健康系统有独立 health scenario catalog
- 启动健康测试会产生 command/receipt
- 测试运行会产生 HealthTestRun
- 失败测试能生成 HealthReport 或 HealthIssue
- cutover readiness 能引用测试结果
- 前端 verify 页不再只是普通 test runner 面板

## 九、残留清理

应清理或迁移：

- 健康页面直接启动 test run 且不生成 command/receipt 的新路径
- 只存在 pytest 输出、不进入健康后台的关键验证
- 无法关联 scenario/report/readiness 的散装测试

应保留：

- `test_system_service`
- `/test-system/runs`
- `/test-system/artifacts`
- `/test-system/harness-map`

这些是测试执行层，健康体系应复用而不是重造。

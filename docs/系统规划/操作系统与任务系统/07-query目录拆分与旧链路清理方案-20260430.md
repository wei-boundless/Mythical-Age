# query 目录拆分与旧链路清理方案

日期：2026-04-30  
定位：本文件专门追踪 `backend/query` 的旧框架残留、保留边界和迁移归宿，用于指导真实主链贯通后的旧链路清理。  
当前结论：`query` 只能保留为 API / 事件流适配层，不能继续承担理解、调度、工具执行、worker 编排、上下文恢复、输出写回等系统职责。

---

## 0. 总结论

当前 `backend/query` 共有 39 个 Python 文件，其中 `runtime.py` 约 2641 行。它已经不是一个单一 query 模块，而是历史上多套系统被堆进同一个目录后的混合体：

```text
API adapter
旧 QueryPlanner
follow-up / continuation 恢复链
worker / evidence / artifact graph
direct tool bridge
prompt builder / prompt manifest
runtime context state
output boundary / persistence
answer assembly / finalization
```

按新架构，`query` 的完成态只能是：

```text
backend/query/
  __init__.py
  models.py              # 仅保留 QueryRequest / event adapter DTO
  runtime.py             # 仅负责 API 输入、事件流输出、调用 AgentRuntime
```

其余能力必须迁出：

```text
理解 / continuation       -> backend/understanding
任务计划 / fanout          -> backend/tasks + backend/orchestration
资源与工具执行             -> backend/operations + backend/execution
worker / evidence          -> backend/evidence 或 backend/workers
prompt manifest            -> backend/soul 或 backend/prompting
context state              -> backend/context_policy / backend/memory_system
output boundary / commit    -> backend/output_boundary / backend/orchestration
```

本阶段不建议直接删除全部旧文件，因为部分测试和旧入口仍引用它们；但必须先冻结旧执行权，再按归宿迁移。已经被 fail-closed 的旧链路不得通过“临时兼容”方式复活。

### 1.2 当前真实主链

`backend/query/runtime.py` 的 `astream()` 现在已经绕过旧 planner 主执行分支，进入新链：

```text
QueryRuntime.astream()
  -> _commit_user_message()
  -> _build_live_agent_runtime_chain_preview()
  -> _agent_runtime_chain_preview_events()
  -> _stream_model_response_directive()
  -> OperationGate(op.model_response)
  -> model_runtime.invoke_messages()
  -> AssistantOutputBoundary
  -> build_blocked_runtime_commit_gate()
  -> done(commit_gate_blocked)
```

这说明真实单 agent 主链已经开始接通，但 `runtime.py` 仍然 import 并初始化了大量旧系统：

```text
QueryPlanner
RuntimeFollowupCoordinator
RuntimeToolBridge
EvidenceOrchestrator
RetrievalWorker / PDFWorker / StructuredDataWorker
RuntimeContextState
RuntimePersistenceAssembler
RuntimeOutputPolicy
```

这些 import 和初始化是下一阶段清理的核心风险：即使主链目前 fail-closed，旧对象仍在运行时构造，后续改动很容易误触旧分支。

### 1.3 已经退役但仍存在的旧执行入口

当前代码中这些入口已经被 fail-closed 或显式 retired：

```text
_planner_build_plan()
_stream_single_execution()
_stream_bundle_execution()
_stream_planned_execution()
_stream_direct_tool_execution()
```

其中 `_planner_build_plan()` 已直接抛出：

```text
legacy QueryPlanner execution path is retired; use AgentRuntimeChainPreview and RuntimeDirective
```

这符合新原则，但还不够：函数和依赖仍留在 `runtime.py`，会让旧链路继续污染主运行时边界。

## 2. 对照设计原则

### 2.1 Candidate 不等于 Decision

旧问题：

```text
QueryPlanner / continuation / follow-up / memory restore 曾经直接改变执行目标。
```

当前要求：

```text
QueryPlanner 只能产候选。
FollowupResolver 只能产候选。
Memory restore 只能产上下文候选。
Worker plan 只能产候选。
RuntimeDirective 只能由 OrchestrationSystem 采纳后生成。
```

清理约束：

```text
任何 query 文件不得再从 query_understanding.tool_name、worker_plan、execution_kind 直接进入执行。
```

### 2.2 控制平面与执行平面分离

旧问题：

```text
runtime.py 同时做输入、理解、计划、工具权限、worker 调度、模型执行、输出清洗、session 写回。
```

当前要求：

```text
Control Plane:
  TaskSystem / OperationSystem / MemorySystem / OrchestrationSystem

Execution Plane:
  ModelExecutor / ToolExecutor / WorkerExecutor / BoundedAgentExecutor

Adapter:
  QueryRuntime
```

清理约束：

```text
QueryRuntime 只能调用 AgentRuntime 或 RuntimeDirectiveExecutor，不能自己选择 executor。
```

### 2.3 默认 fail-closed

已经做到：

```text
旧 planner / bundle / planned / direct tool 分支已经 fail-closed。
assistant output 写回仍被 runtime commit gate blocked。
memory write 仍是 candidate-only。
```

还没做到：

```text
旧 worker / evidence / follow-up 初始化仍存在。
旧输出和 persistence 仍在 query 命名空间。
旧 tests 仍大量引用 query planner / worker runtime。
```

### 2.4 单一执行真相

完成态必须是：

```text
OrchestrationPlan
  -> ExecutionGraph
  -> RuntimeDirective
  -> OperationGate
  -> Executor
  -> ResultCandidate
  -> CommitGate
```

所以 `QueryExecutionPlan`、`WorkerExecutionPlan`、`QueryPlan.execution_kind` 都不能继续作为当前轮执行真相。它们最多是 legacy candidate 的输入格式。

## 3. 文件级分类

### 3.1 保留在 query，但收缩

| 文件 | 处理 | 原因 |
| --- | --- | --- |
| `__init__.py` | 收缩 | 只导出 `QueryRuntime` / `QueryRequest`，停止导出 `QueryExecutionPlan`、`QueryPlan` 等旧计划模型 |
| `models.py` | 拆分后保留 adapter DTO | `QueryRequest` 可保留；`QueryExecutionPlan`、`QueryPlan`、`BundlePlan` 应迁入 legacy candidate 或删除 |
| `runtime.py` | 大幅收缩 | 只保留 `astream()`、输入 commit gate、调用 AgentRuntimeChain、事件流适配、错误包装 |

完成态目标：

```text
QueryRuntime.__init__ 不再构造 QueryPlanner / RuntimeToolBridge / EvidenceOrchestrator / RuntimeFollowupCoordinator。
QueryRuntime.astream 不再拥有 model/tool/worker 选择逻辑。
```

### 3.2 迁到 output boundary / commit boundary

| 当前文件 | 目标归宿 | 说明 |
| --- | --- | --- |
| `output_boundary.py` | `backend/output_boundary/boundary.py` | 保留，职责正确但目录错误 |
| `output_classifier.py` | `backend/output_boundary/classifier.py` | 保留，作为输出候选过滤器 |
| `output_models.py` | `backend/output_boundary/models.py` | 保留 |
| `runtime_output_policy.py` | `backend/output_boundary/policy.py` 或 `backend/orchestration/output_policy.py` | 保留策略能力，但不能依赖 `QueryExecutionPlan` |
| `runtime_persistence.py` | `backend/orchestration/commit_planner.py` 或 `backend/output_boundary/persistence.py` | 重写为 commit candidate assembler，禁止直接写 session |
| `answer_assembler.py` | `backend/output_boundary/answer_assembler.py` | 保留为输出组装 |
| `answer_models.py` | `backend/output_boundary/answer_models.py` | 保留 |

| `tool_output_adapter.py` | `backend/output_boundary/tool_output_adapter.py` | 保留工具输出 envelope 适配 |

迁移后规则：

### 3.6 迁到 prompting / soul projection

| 当前文件 | 目标归宿 | 说明 |

### 3.7 follow-up 旧链路处理

## 4. 清理优先级

### Phase 1：先缩 QueryRuntime 初始化面

目标：

```text
QueryRuntime 不再构造旧 planner / worker / tool bridge / follow-up coordinator。
```

动作：

```text
删除或延迟隔离这些初始化：
  QueryPlanner
  RuntimeToolBridge
  RuntimeFollowupCoordinator
  EvidenceOrchestrator
  RetrievalWorker / PDFWorker / StructuredDataWorker
  TableMaterializer
  BindingCandidateStore / EvidenceGraphStore
```

允许保留：

```text
OutputBoundary
PromptBuilder
MemoryRuntimeView
TaskOperationPreview
OperationGate(op.model_response)
ModelRuntime
```

完成标准：

```text
普通 chat 主链启动时不会实例化旧 worker / old planner / direct tool bridge。
```

### Phase 2：迁出 output boundary

目标：

```text
输出边界成为独立系统，不再挂在 query 下。
```

动作：

```text
新增 backend/output_boundary/
移动 output_*、answer_*、tool_output_adapter。
更新 QueryRuntime / memory.messages / answer_finalizer imports。
```

完成标准：

```text
git grep "from query.output" -- backend 为空。
```

### Phase 3：迁出 prompt manifest / context models

目标：

```text
PromptManifest / MainContextState 从 query 目录脱离。
```

动作：

```text
新增 backend/prompting/
新增或补齐 backend/context_policy/models.py
迁移 prompt_manifest.py、prompt_builder.py、context_models.py。
```

完成标准：

```text
QueryRuntime 只调用 prompt/context facade，不直接拼复杂 context。
```

### Phase 4：冻结并迁移旧 planner 候选层

目标：

```text
QueryPlanner 不再作为 planner 存在。
```

动作：

```text
把 planner.py 拆为 candidate producers。
把 BundlePlanner / SubtaskPlanner / BindingResolver / CapabilityDispatchScheduler 迁到对应系统。
删除 QueryPlan.execution_kind 的执行权。
```

完成标准：

```text
QueryExecutionPlan 不再进入 runtime 执行函数。
旧 planner tests 转为 candidate tests 或归档。
```

### Phase 5：迁出 worker / evidence

目标：

```text
worker 系统独立，EvidenceOrchestrator 不再是全局编排器。
```

动作：

```text
新增 backend/evidence/
新增 backend/workers/
迁移 evidence_*、worker_*、pdf_worker、structured_data_worker、retrieval_worker、table_materializer。
```

完成标准：

```text
worker 只能由 RuntimeDirectiveExecutor 调用。
worker result 只能进入 ResultCandidate / CommitGate。
```

### Phase 6：删除旧 fail-closed 函数

目标：

```text
runtime.py 从 2600+ 行收缩到 adapter 大小。
```

动作：

```text
删除 _stream_bundle_execution
删除 _stream_single_execution
删除 _stream_planned_execution
删除 _stream_direct_tool_execution
删除 follow-up binding execution helpers
删除 old task summary / old context projection helpers
```

完成标准：

```text
QueryRuntime 只剩 API adapter、输入 gate、AgentRuntime 调用、事件流转换。
```

## 5. 禁止事项

清理过程中禁止：

```text
为了兼容测试重新调用 QueryPlanner.build_plan。
为了保留 PDF 能力恢复 EvidenceOrchestrator.stream_execution 直连。
为了保留工具能力恢复 RuntimeToolBridge.invoke_tool。
为了写回答绕过 RuntimeCommitGate append assistant message。
为了恢复上下文让 RuntimeContextState 覆盖 TaskContract。
为了省事让 prompt_builder 继续直接读取 memory 并决定模型可见范围。
```

一句话：

```text
可以迁移能力，不能迁移旧权力。
```

## 6. 近期执行建议

下一步建议先做两件事：

### 6.1 先缩 `QueryRuntime.__init__`

这是最划算的清理点。因为主链已经走 `op.model_response`，大量旧对象当前不是主链必需：

```text
QueryPlanner
RuntimeToolBridge
EvidenceOrchestrator
PDFWorker
StructuredDataWorker
RetrievalWorker
RuntimeFollowupCoordinator
TableMaterializer
```

把这些从默认初始化路径拿掉，可以立即降低误触旧链路的风险。

### 6.2 再迁 `output_boundary`

`output_boundary` 是新架构需要的能力，不该删；但它留在 `query` 目录会让“输出边界是 query 私有逻辑”的错觉继续存在。先迁出它，可以形成第一条清晰的新边界：

```text
runtime_directive result
  -> output_boundary
  -> commit_gate
```

## 7. 当前判断

`query` 臃肿不是代码风格问题，而是系统边界问题。现在最危险的不是旧代码还在，而是旧代码还和 `QueryRuntime` 同生共死：只要入口类还默认构造旧 planner、worker、tool bridge，就会持续给后续重构制造隐性耦合。

所以后续清理顺序应该是：

```text
先断默认初始化。
再迁保留能力。
再删除退役函数。
最后清理旧测试和旧导出。
```

这条路线能保证：

```text
主链不跑偏。
能力不误删。
旧权力不复活。
```

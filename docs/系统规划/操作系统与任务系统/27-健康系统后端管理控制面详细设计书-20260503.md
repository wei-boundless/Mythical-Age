# 健康系统后端管理控制面详细设计书

日期：2026-05-03

上位计划：`25-健康系统管理控制面与验证体系统筹重构计划书-20260502.md`

## 一、子系统定位

健康系统后端管理控制面是健康系统自己的独立领域后端。它承载健康系统原生操作台，也承载健康管理 agent 的代操请求。

它负责把不同来源的健康动作统一记录、审计和约束：

- 用户在健康系统操作台里的直接操作
- 健康管理 agent 通过嵌入式对话窗发起的代操
- 系统事件自动触发的健康动作
- 测试体系回传的验证结果

这些动作最终收敛成：

- `HealthManagementCommand`
- `HealthManagementReceipt`
- `HealthIssue`
- `HealthAgentRun`
- `HealthReport`
- `HealthSystemSetting`
- `HealthAgentConversationSession`

它不直接修改主 runtime 的 `TaskRunLedger / RuntimeCheckpoint / TaskResult`，也不直接扩展 agent 的运行权限。

## 二、当前代码对照

### 2.1 现有健康后端 API

当前 `backend/api/health_system.py` 已经提供：

- `GET /health-system/overview`
- `GET /health-system/issues`
- `POST /health-system/issues`
- `GET /health-system/issues/{issue_id}`
- `GET /health-system/agent-runs/{run_id}`
- `GET /health-system/agent-runs/{run_id}/result`
- `GET /health-system/agent-runs/{run_id}/trace-report`
- `POST /health-system/issues/{issue_id}/agent-runs/preview`
- `POST /health-system/issues/{issue_id}/agent-runs`

当前问题不是没有健康 API，而是 API 还以 issue/run 为中心，缺少能同时表达“用户直操”和“agent 代操”的命令、回执和审计语义。

### 2.2 现有 HealthRegistry

当前 `backend/health_system/registry.py` 已经提供：

- `create_issue`
- `list_issues`
- `list_agent_runs`
- `preview_agent_run`
- `execute_agent_run`
- `build_agent_run_trace_report`

其中 `execute_agent_run` 已经会读取：

- `TaskFlowRegistry`
- `ProjectionInstanceRegistry`
- `SkillWorkflowRegistry`
- `ResourcePolicy`
- `task_run_loop`
- `model_response_executor`

这说明后端已经有一条“健康 issue -> task flow -> projection -> runtime loop -> result”的链路。

缺口是这条链路没有统一挂在 command / receipt 上，导致一次管理动作不能被后台完整追踪。

### 2.3 现有 HealthWorkbench

当前 `backend/health_system/workbench.py` 已经把健康 issue、测试 harness、最近测试运行、evidence gaps、recommended actions 汇总成工作台视图。

它适合作为健康系统页面的数据投影，但不应承担命令处理职责。

## 三、目标领域模型

### 3.1 HealthManagementCommand

```python
class HealthManagementCommand:
    command_id: str
    command_type: str
    initiator_type: str
    initiator_ref: str
    requested_by: str
    source: str
    conversation_session_ref: str
    target_scope: str
    target_ref: str
    task_mode: str
    payload: dict[str, Any]
    status: str
    created_at: float
    updated_at: float
```

`command_type` 最小集合：

- `report_issue`
- `analyze_trace`
- `draft_case`
- `verify_fix`
- `launch_health_test`
- `update_health_setting`
- `build_cutover_readiness`

`initiator_type` 最小集合：

- `user`
- `agent`
- `system`
- `test_system`

这一区分非常关键：同一个 `launch_health_test`，用户在健康系统 verify 页面直接点击，与健康管理 agent 在对话窗里代为发起，后端执行链路可以收敛，但权限解释、审计语义和 UI 回显不能混淆。

### 3.2 HealthManagementReceipt

```python
class HealthManagementReceipt:
    receipt_id: str
    command_ref: str
    accepted: bool
    status: str
    health_issue_ref: str
    health_run_ref: str
    test_run_ref: str
    report_ref: str
    blocked_reasons: tuple[str, ...]
    diagnostics: dict[str, Any]
    created_at: float
```

失败命令也必须有 receipt。这样无论用户在健康系统操作台直接点击，还是在对话窗里让 agent 代操，如果被权限或配置拦截，后台都能看到为什么被拦截。

### 3.3 HealthAgentConversationSession

```python
class HealthAgentConversationSession:
    session_id: str
    agent_id: str
    agent_profile_id: str
    projection_template_id: str
    skill_workflow_id: str
    runtime_lane: str
    active_issue_ref: str
    active_run_ref: str
    command_refs: tuple[str, ...]
    status: str
    created_at: float
    updated_at: float
```

这个 session 只服务健康系统内嵌 agent 对话窗。它不是健康系统原生操作台的必需前置条件。

### 3.4 HealthReport

```python
class HealthReport:
    report_id: str
    report_type: str
    issue_ref: str
    command_ref: str
    agent_run_ref: str
    test_run_ref: str
    evidence_refs: tuple[str, ...]
    verdict: str
    severity: str
    summary: str
    recommended_actions: tuple[str, ...]
    created_at: float
```

## 四、目标 API

### 4.1 管理命令与报告 API

新增：

```text
POST /health-system/commands
GET /health-system/commands/{command_id}
GET /health-system/receipts/{receipt_id}
GET /health-system/reports
GET /health-system/reports/{report_id}
POST /health-system/conversation-sessions
GET /health-system/conversation-sessions/{session_id}
POST /health-system/conversation-sessions/{session_id}/messages
```

其中 command API 同时服务：

- 健康系统原生操作台，`initiator_type=user`
- 健康管理 agent 代操，`initiator_type=agent`
- 系统自动动作，`initiator_type=system`

conversation session API 只服务嵌入式 agent 对话窗。

### 4.2 兼容入口

现有入口暂不立即删除：

- `POST /health-system/issues`
- `POST /health-system/issues/{issue_id}/agent-runs`
- `POST /health-system/issues/{issue_id}/agent-runs/preview`

但其中带有管理副作用的动作应逐步改为 command handler 的兼容包装：

- 创建 issue 等价于 `report_issue`
- 启动 run 等价于 `analyze_trace` 或 `verify_fix`
- preview 等价于 `command dry_run`

最终前端新路径应分为两类：

- 健康系统原生操作台继续调用健康系统自己的资源 API 或 command API。
- 健康管理 agent 对话窗调用 conversation session API，并在需要代操时调用 command API。

旧散装 issue/run 执行入口保留到迁移窗口结束后再清理，但不能成为绕过 command/receipt 的隐藏路径。

## 五、命令处理流程

```text
HealthManagementCommand
  -> HealthCommandHandler
  -> initiator classification
  -> TaskSystem agent/task connection profile
  -> AgentCapabilityProfile validation
  -> ResourcePolicy validation
  -> HealthRegistry write issue/run/report
  -> HealthManagementReceipt
```

关键规则：

- command handler 是健康管理副作用的统一审计入口，但不是健康系统唯一 UI 入口
- command handler 可以调用 test system，但必须生成 receipt
- command handler 可以调用 health agent run，但必须记录 agent/binding/projection/runtime lane
- command handler 不能直接写 runtime truth
- command handler 不能把 setting 直接转成权限变更
- user 直操和 agent 代操可以共用 handler，但必须保留 `initiator_type`

## 六、与其它子系统协调

### 6.1 对任务系统

健康系统后端在创建健康管理 agent 会话或执行 agent 代操命令时，向任务系统读取 agent/task 连接配置。

读取内容包括：

- agent id
- task mode
- flow id
- binding id
- workflow id
- projection template id
- runtime lane hint
- output contract

健康系统后端不自己定义 agent 的任务身份、projection、workflow 和 binding。它只引用任务系统的通用 agent/task connection profile。

### 6.2 对编排系统

健康控制面把 command 转成待执行健康任务，但编排系统负责最终运行约束。

如果 capability profile 或 resource policy 拒绝动作，健康控制面必须生成 rejected receipt，而不是绕过限制。

### 6.3 对健康系统原生操作台

健康系统原生操作台可以直接调用健康系统后端。

它负责用户直操：

- 创建 issue
- 查看 report
- 启动测试
- 调整设置
- 查看 readiness

需要审计、执行或可能触发权限裁决的动作，应生成 `HealthManagementCommand`，并标记 `initiator_type=user`。

### 6.4 对嵌入式 Agent 对话窗

嵌入式 agent 对话窗只向后端提交 session message 或 agent 代操 command。

后端负责：

- 建 session
- 记录消息
- 生成 command
- 返回 receipt
- 返回 report/run/issue 引用
- 标记 initiator 是 agent 代操

前端不能自行拼接执行链路。

### 6.5 对测试体系

健康控制面通过内部服务调用测试体系。

测试体系返回：

- test run id
- artifacts
- failures
- runtime-loop trace refs

健康控制面负责把结果落为 `HealthReport` 或关联 `HealthIssue`。

## 八、验收标准

- 每次健康管理动作都有 command
- 每个 command 都有 receipt
- 被拒绝的动作也有 blocked reasons
- 健康 agent run 能追溯到 command/session
- 健康 report 能追溯到 issue/run/test/evidence
- 旧 issue/run 执行入口不再绕过 command/receipt

## 九、残留清理

应清理或迁移：

- 前端把高风险执行按钮直接映射到 `startHealthAgentRun` 且不生成 command/receipt 的路径
- 后端散落的“一次性健康动作”入口
- 不产生 receipt 的健康管理动作
- 不绑定 command 的 report/run

不应清理：

- 现有 issue/run/result/trace-report 查询能力

这些能力仍是健康控制面的基础查询资产。

# 问题二：恢复重放与副作用执行语义重构计划书

日期：2026-05-02

实施状态：2026-05-02 首轮实现已完成，主链代码与回归测试已落地。

## 一、文档目标

本计划书是对 `18-问题二-恢复重放与副作用幂等性差距报告-20260502.md` 的进一步展开。

本计划书回答四个更具体的问题：

1. 为什么问题二不能靠“给高风险工具单独打补丁”来解决。
2. 先进架构对 durable side effect semantics 的最小共同要求是什么。
3. 结合当前代码，副作用执行语义层应该落在哪一层，哪些结构可以直接复用。
4. 如何在不承诺伪“exactly once”的前提下，把恢复重放、安全去重、失败收口做成稳定的运行时能力。

本计划不讨论的问题：

- 模板选择与 bundle 契约化
- 故障演练矩阵的完整铺开
- worker 跨进程编排协议
- 前端恢复态可视化

这些问题后续分别由 `19`、`20` 对应阶段继续处理。

## 二、核心判断

### 2.1 需要的不是“恢复入口”，而是“副作用执行语义层”

当前系统已经有：

- `TaskRunLedger`
- `RuntimeCheckpoint`
- `RuntimeEvent`
- `RuntimeActionRequest`
- `RuntimeObservation`

因此问题已经不再是“有没有恢复骨架”，而是“副作用执行到底由谁记账、谁做 replay decision、谁提供 dedupe 保证”。

当前答案仍然是：

- request 发出后直接进入 executor
- executor 返回 observation
- recovery 只能从 loop state 猜测是否重跑

目标答案应该是：

- request 先映射为 execution record
- execution record 决定幂等身份和 replay policy
- executor 只消费 execution record
- observation 带着 execution receipt 回到 loop

### 2.2 我们要的不是伪“exactly once”，而是分级可承诺语义

本次重构不承诺：

- 所有外部副作用都全局 exactly once
- 所有 shell / worker / 外部系统都天然可恢复

本次重构明确承诺：

- 只读动作可安全重放
- 幂等写动作优先按 token 去重并回读首次结果
- 非 replay-safe 动作恢复时默认 fail-closed
- trace 必须能解释 recovery 为什么做出这个决定

### 2.3 最终目标

最终要形成如下关系：

```text
RuntimeActionRequest
  -> OperationExecutionRecord
  -> Executor dispatch
  -> ExecutionReceipt
  -> RuntimeObservation
  -> TaskRunLoop transition / checkpoint / trace
```

而不是当前更接近的：

```text
RuntimeActionRequest
  -> Executor direct call
  -> RuntimeObservation
  -> recovery time guesswork
```

## 三、理论依据

### 3.1 本地设计原则依据

来自项目内已有设计原则：

- [03-状态管理.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/03-状态管理.md:1)
  - 状态按生命周期分层
  - 越底层状态越克制
- [05-对话循环.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/05-对话循环.md:1)
  - continue / terminal 必须有明确原因
- [09-工具系统设计.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/09-工具系统设计.md:1)
  - 工具系统应保持安全默认、统一调用边界
- [16-权限系统.md](/abs/path/d:/AI应用/langchain-agent/docs/设计原则/16-权限系统.md:1)
  - 高风险动作需要多层防线与 fail-closed

这几条叠加后的直接约束是：

- 恢复语义必须是结构层能力
- 副作用边界必须统一，而不是每个工具自创一套
- 对高风险动作要默认保守，而不是默认自动重放

### 3.2 LangGraph 的依据

LangGraph durable execution 的关键思想是：

- 用 checkpointer 保存 workflow progress
- 用 thread identity 恢复具体运行实例
- 把 non-deterministic / side-effecting work 包进 `tasks`
- 恢复时依赖这些边界避免重复副作用

这对我们的直接启发是：

- `RuntimeActionRequest` 不能直接等同于“安全可重放的执行”
- mutating operation 必须先进入统一执行边界
- 恢复时应先看历史执行记录，而不是先决定重跑工具

来源：

- https://docs.langchain.com/oss/javascript/langgraph/durable-execution

### 3.3 Temporal 的依据

Temporal 的价值不在于“它有重试功能”，而在于：

- workflow execution 的真相来自 event history
- durable execution 要围绕执行历史组织恢复
- 副作用与确定性边界必须被系统显式管理

这对我们的启发是：

- execution record 必须进入 event / checkpoint 视图
- request、execution、result 不能继续混成一层
- 恢复不能只看当前 step，还必须看这一步是否已有已提交副作用

来源：

- https://docs.temporal.io/workflow-execution
- https://assets.temporal.io/durable-execution.pdf

### 3.4 AWS Reliability 的依据

AWS 官方对 idempotency 和 retry 的要求有很强的工程可操作性：

- mutating operation 使用 request token 区分新旧请求
- 对重复请求返回首次结果，而不是再次执行
- retry 要受控，避免多层叠加
- fail fast，避免堆积和放大故障

这对我们的启发是：

- 幂等 token 不能只是文档约定，必须进入执行数据结构
- dedupe 命中后应回读已有结果
- retry 需要固定在单一控制边界

来源：

- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_prevent_interaction_failure_idempotent.html
- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_mitigate_interaction_failure_limit_retries.html
- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_mitigate_interaction_failure_fail_fast.html

## 五、设计原则与边界

### 5.1 原则一：区分 request、execution、observation 三层

这三层职责必须分开：

- request
  - 表达 loop 想做什么
- execution
  - 表达这次副作用是否已登记、是否可去重、是否已提交
- observation
  - 表达执行结果如何返回 loop

不能继续让 observation 兼职承担“执行是否安全可重放”的语义。

### 5.2 原则二：复用 operation registry 的风险元数据

本次不新增一套平行风险分类系统。

优先使用现有：

- `read_only`
- `idempotent`
- `destructive`
- `interrupt_behavior`
- `requires_approval_by_default`

在此基础上推导 replay policy，而不是让工具、registry、runtime 各说各话。

### 5.3 原则三：不承诺假 exactly-once，只承诺分级 replay-safe

推荐语义分层：

- 只读动作
  - 允许重放
- 幂等写动作
  - 使用 token 去重，重复请求返回首次结果
- 非 replay-safe 动作
  - 默认 deny auto replay

这比空泛承诺“恢复没问题”更诚实，也更接近主流系统的真实做法。

### 5.4 原则四：retry 只能有一个主控制边界

推荐边界：

- step / runtime 决定是否 retry
- execution record 维持 retry 的幂等身份
- executor 不再额外补不透明重试

### 5.5 原则五：高风险恢复默认 fail-closed

对于 shell、外部写入、未知 worker 副作用：

- 如果不能证明 replay-safe
- 就默认拒绝自动恢复执行

不要为了“恢复看起来顺畅”而牺牲副作用安全。

## 六、目标设计

### 6.1 新增 OperationExecutionRecord

建议新增统一模型：

```python
class OperationExecutionRecord:
    execution_id: str
    task_run_id: str
    step_id: str
    request_ref: str
    directive_ref: str
    operation_id: str
    executor_type: str
    request_fingerprint: str
    idempotency_token: str
    replay_policy: str
    status: str
    side_effect_state: str
    result_ref: str
    attempt_count: int
```

建议最小状态：

- `created`
- `dispatched`
- `completed`
- `failed`
- `replay_suppressed`
- `reused_completed_result`

建议最小副作用状态：

- `not_started`
- `in_progress`
- `committed`
- `unknown`

### 6.2 新增 ExecutionReceipt

建议 observation 不再只带结果文本，而是带统一 receipt：

```python
class ExecutionReceipt:
    execution_id: str
    request_ref: str
    status: str
    replay_decision: str
    reused_previous_result: bool
    result_ref: str
```

这样 loop、trace、checkpoint 才能稳定回答这次结果的来源。

### 6.3 replay policy 设计

建议最小策略集合：

- `replay_read`
  - 只读动作可直接再跑
- `reuse_completed_result`
  - 已完成幂等写动作优先回读
- `deny_auto_replay`
  - 不允许自动重放
- `manual_recovery_required`
  - 需要人工介入决定是否继续

推荐推导逻辑：

- `read_only=True` -> `replay_read`
- `read_only=False and idempotent=True` -> `reuse_completed_result`
- `destructive=True or idempotent=False` -> `deny_auto_replay`
- 特殊 executor / worker 可在 descriptor metadata 中显式提升到 `manual_recovery_required`

### 6.4 去重键设计

建议至少包含：

- `task_run_id`
- `step_id`
- `operation_id`
- 标准化后的请求输入摘要

推荐关系：

- `request_fingerprint`
  - 用于判断“这是同一请求吗”
- `idempotency_token`
  - 用于对外/对执行器表达“重复请求应回原结果”

### 6.5 checkpoint 与事件扩展

建议：

- checkpoint 增加 `execution_state_ref` 或 `execution_refs`
- event 增加以下类型：
  - `execution_record_created`
  - `execution_dispatch_started`
  - `execution_result_recorded`
  - `execution_result_reused`
  - `replay_guard_triggered`
  - `recovery_replay_decided`

### 6.6 executor 改造边界

`ToolRuntimeExecutor` 不应再直接从 action request 调工具。

推荐改成：

1. loop 创建 action request
2. execution coordinator 创建 / 查询 execution record
3. coordinator 产出 replay decision
4. executor 只在允许执行时真正 dispatch
5. 结果先回写 execution record，再生成 observation

## 七、分阶段改造建议

### Phase 2A：新增执行记录模型与存储

目标：

- 新增 `OperationExecutionRecord`
- 新增 execution store
- 让 checkpoint / trace 可以引用 execution id

建议涉及文件：

- `backend/orchestration/runtime_loop/action_request.py`
- `backend/orchestration/runtime_loop/events.py`
- `backend/orchestration/runtime_loop/checkpoint.py`
- 新增 `backend/orchestration/runtime_loop/execution_record.py`

验收：

- 任一 tool / worker action 都有 execution id
- trace 可展示 request -> execution -> observation 的链路

### Phase 2B：接入 replay policy 与 dedupe

目标：

- 基于 `OperationDescriptor` 推导 replay policy
- 建立 request fingerprint / idempotency token
- 命中重复请求时返回已有 execution result

建议涉及文件：

- `backend/operations/registry.py`
- `backend/orchestration/runtime_loop/tool_adoption.py`
- 新增 `backend/orchestration/runtime_loop/execution_policy.py`
- `backend/execution/tool_executor.py`

验收：

- 只读动作恢复后可直接重放
- 幂等写动作重复请求会回读首次结果
- 非 replay-safe 动作不会被静默自动重放

### Phase 2C：把恢复链接到 step state machine

目标：

- 恢复时先看 execution record 再决定 step 是否继续
- step failure / replay denied / reused result 要进入 ledger 和 terminal reason

建议涉及文件：

- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/tasks/run_models.py`
- `backend/orchestration/runtime_loop/trace_reader.py`

验收：

- 恢复时 loop 能说明“为什么继续 / 为什么停”
- `TaskResult` 能表达 replay denied / reused previous result 等状态

### Phase 2D：补齐故障回归

目标：

- crash-resume
- duplicate side effect request
- permission blocked after checkpoint
- retry exhausted
- replay denied

建议涉及文件：

- 新增 `backend/tests/runtime_recovery_idempotency_regression.py`
- `backend/tests/query_runtime_runtime_loop_regression.py`
- `backend/tests/operation_system_preview_regression.py`

验收：

- 每个高风险恢复分支都有自动化测试覆盖

## 八、切换策略

推荐采用“先记录、再守卫、最后收口”的切换顺序：

1. 先引入 execution record，但不立即改变旧 executor 结果路径
2. 再让幂等写动作命中去重回读
3. 最后把 non-replay-safe 动作改成 fail-closed

不建议的顺序：

- 不要先在 `tool_executor.py` 里零散加 if/else 去重
- 不要先给单个 `edit_file` 或 `shell` 写特判补丁
- 不要先对外宣称 crash-resume 安全，再补底层执行账本

## 九、验收总口径

本阶段完成后，应满足以下统一口径：

- 任一副作用动作都能明确回答“是否已经真正执行过”。
- 任一恢复分支都能明确回答“为什么可以重放、为什么复用旧结果、为什么必须拒绝继续”。
- 任一重复请求都能明确回答“它与首次请求的身份键是什么”。
- 任一 retry 都能明确回答“控制边界在哪里，是否会形成多层叠加”。

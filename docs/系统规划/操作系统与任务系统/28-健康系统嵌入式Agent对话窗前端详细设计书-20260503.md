# 健康系统嵌入式 Agent 对话窗前端详细设计书

日期：2026-05-03

上位计划：`25-健康系统管理控制面与验证体系统筹重构计划书-20260502.md`

## 一、子系统定位

健康系统嵌入式 Agent 对话窗不是健康系统的主入口，也不是健康系统操作台的替代品。

健康系统前端应保持两个并行入口：

- 用户原生操作台：用户直接查看 issue、报告、测试、设置、readiness，并通过健康系统页面操作。
- Agent 对话窗：用户和健康管理 agent 对话，由 agent 在授权范围内解释、整理、代操或申请更高权限动作。

因此本文件只定义健康系统页面内嵌的 agent 对话窗口。健康系统本身的页面、表单、列表、测试按钮、设置入口仍然属于健康系统前端操作台。

## 三、目标交互模型

### 3.1 双入口关系

```text
HealthSystemView 原生操作台
  -> 用户直接操作 health issue / test / report / setting / readiness
  -> 需要审计或执行的动作进入 HealthManagementCommand

HealthAgentDock 嵌入式对话窗
  -> 用户与健康管理 agent 对话
  -> agent 解释、整理、建议、代操
  -> 代操动作进入 HealthManagementCommand
```

两条路径共享健康系统后端、审计模型和权限门禁，但前端入口不同。

### 3.2 Dock 状态

```ts
type HealthAgentDockState = {
  open: boolean;
  position: { x: number; y: number };
  sessionId: string;
  boundIssueId: string;
  boundRunId: string;
  messages: HealthAgentConversationMessage[];
  pendingCommandId: string;
  pendingReceiptId: string;
  mode: "idle" | "sending" | "delegating" | "blocked" | "failed";
};
```

### 3.3 对话消息

```ts
type HealthAgentConversationMessage = {
  message_id: string;
  session_id: string;
  role: "user" | "assistant" | "system";
  content: string;
  command_ref?: string;
  receipt_ref?: string;
  report_ref?: string;
  created_at: number;
};
```

### 3.4 Agent 代操动作

用户可以在对话窗里请求 agent 代操：

- “帮我分析这个问题” -> `analyze_trace`
- “帮我生成复现用例草案” -> `draft_case`
- “帮我验证这个修复” -> `verify_fix`
- “帮我启动这组健康测试” -> `launch_health_test`
- “帮我生成切流报告” -> `build_cutover_readiness`
- “帮我修改这个健康设置” -> `update_health_setting`

这些动作不是前端直接执行，也不是 agent 绕过健康系统执行，而是以 `initiator_type=agent` 或 `delegated_by_agent=true` 的方式提交给健康系统后端。

## 四、目标 API 调用

### 4.1 Agent 对话 API

在 `frontend/src/lib/api.ts` 新增：

```ts
export async function createHealthAgentConversationSession(payload: {
  active_issue_ref?: string;
  active_run_ref?: string;
}) {}

export async function sendHealthAgentConversationMessage(sessionId: string, payload: {
  content: string;
  target_scope?: string;
  target_ref?: string;
}) {}
```

### 4.2 代操命令 API

对话窗需要调用健康系统后端的命令 API：

```ts
export async function createHealthManagementCommand(payload: {
  command_type: string;
  initiator_type: "agent";
  conversation_session_ref: string;
  target_scope?: string;
  target_ref?: string;
  task_mode?: string;
  payload?: Record<string, unknown>;
}) {}

export async function getHealthManagementReceipt(receiptId: string) {}
```

健康系统原生操作台也可以调用同一个 command API，但它的 `initiator_type` 应是 `"user"`，并且不需要 `conversation_session_ref`。

## 五、界面结构

### 5.1 HealthSystemView 继续作为主工作台

健康系统页面继续承担：

- issue 列表和详情
- report 列表和详情
- verify 页面
- test run 管理
- setting 管理
- readiness 展示
- evidence gaps 和 recommended actions

### 5.2 AgentDock 作为辅助层

AgentDock 展开态建议分为：

- 上下文条：当前绑定 issue/run/test/report
- 对话区：用户与健康管理 agent 消息
- 回执区：当前代操 command 状态、blocked reasons、report refs
- 快捷动作区：分析问题、解释运行、生成报告草案、发起验证

### 5.3 权限提示

对话窗必须持续说明：

- agent 可以解释和代操，但不替代用户操作台。
- agent 不能直接修代码。
- agent 不能直接执行 shell。
- agent 不能直接改 runtime truth。
- agent 的代操动作必须进入健康系统后端 command/receipt。
- 开发级动作需要编排权限或审批策略明确允许。

## 六、与其它子系统协调

### 6.1 对任务系统

对话窗不配置 agent。

健康系统后端在创建会话时，通过任务系统通用 `agent-task-connections` 读取 `agent:health:maintainer` 的任务身份、projection、workflow 和 binding。

### 6.2 对健康系统后端

对话窗只负责 agent 对话和代操意图。

健康系统后端负责：

- 创建会话
- 记录消息
- 生成 command
- 返回 receipt
- 返回 report/run/issue/test refs
- 标记 initiator 是 user 直操还是 agent 代操

### 6.3 对编排系统

对话窗不直接接触 runtime loop。

如果用户请求开发级动作，例如“改代码并跑验证”，对话窗只能把它表达为代操请求；最终是否允许，由健康系统后端、任务系统连接信息、编排权限和审批策略共同裁决。

### 6.4 对测试体系

用户可以直接在健康系统 verify 页面启动测试，也可以让 agent 在对话窗里代为启动测试。

两条路径都应进入健康系统的测试运行记录和报告体系。

## 七、实施步骤

### Phase 28A：API 类型补齐

- 在 `frontend/src/lib/api.ts` 新增 session/message/command/receipt 类型
- command payload 增加 `initiator_type`
- 支持 user 直操和 agent 代操两类来源

### Phase 28B：Dock 接入真实 session

- `HealthAgentDock` 打开时创建或恢复 session
- selected issue/run 变化时更新 session context
- 用户发送消息时调用 session message API
- agent 代操返回 command/receipt 时渲染状态卡片

### Phase 28C：保留原生操作台

- HealthSystemView 原有 issue/test/report 页面继续存在
- 原生按钮不迁移到 dock
- 需要审计的直操作动作进入 command API，并标记 `initiator_type=user`

### Phase 28D：防旧路径

- dock 内不再直接调用 `startHealthAgentRun`
- dock 不直接调用底层 test runner
- dock 不直接修改 setting
- dock 不承担任务系统 agent 配置职责

## 八、验收标准

- 健康系统操作台仍可由用户直接使用。
- 用户可以通过嵌入式对话窗与健康管理 agent 对话。
- agent 代操动作会形成 command 和 receipt。
- command 能区分 `initiator_type=user` 与 `initiator_type=agent`。
- dock 不替代 HealthSystemView 的原生操作能力。
- dock 不绕过健康后端和编排权限。

## 九、残留清理

应清理：

- 把 dock 写成健康系统唯一入口的表述或实现。
- dock 内仅本地消息、无法追溯后台的代操动作。
- dock 直接调用散装 issue/run/test/setting API 的新路径。
- 把任务系统配置项塞进 dock 编辑的设计。

应保留：

- 当前悬浮、拖拽、折叠和位置持久化能力。
- HealthSystemView 的原生健康系统工作台能力。

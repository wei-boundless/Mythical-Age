本计划书是在 `24-问题四-架构验证与故障演练重构计划书-20260502.md` 基础上的进一步统筹。

`24` 主要解决的是：

- 架构验证如何从 happy path 测试升级为故障演练与恢复证明
- 健康子 agent 如何参与 trace analysis、fix verification、cutover readiness

本计划书进一步解决的是：

- 健康系统的测试体系如何成型

- 编排系统如何负责每个 agent 的链路权限、运行姿态、资源边界
- 三者如何避免互相越权、重复决策、形成隐藏旧路径

- `24` 是问题四本体计划，聚焦架构验证、故障演练、恢复证明和切流证据。

- `26` 是任务系统分册，任务系统是通用 agent/task 思维连接层，健康管理 agent 只是其中一个实例。
- `27` 是健康系统后端分册，健康系统保持独立控制面，支持用户直操和 agent 代操两类来源。
- `28` 是健康系统前端分册，只定义嵌入式 agent 对话窗，不替代健康系统原生操作台。
- `29` 是编排分册，负责 agent 权限、resource policy、runtime lane 和运行约束。
- `30` 是验证分册，负责健康验证、测试目录、readiness 和报告化。

真正缺的是更高一层：

- 统一的健康管理命令
- 统一的健康设置
- 统一的测试/演练目录
- 统一的验证结果回执
- 统一的通用 agent/task 连接配置入口
- 统一的编排权限裁决入口
- 统一的健康系统 agent 对话窗接入方式

所以问题四不能只落成一批测试文件。它必须落成一个健康管理控制面。

### 2.2 任务系统与编排系统必须分工

本轮重构采用以下分工：

- 任务系统是 agent 的任务分配与思维连接层
- 编排系统是 agent 的链路权限与身体配置层
- 健康系统是面向系统健康的管理与验证控制面

更具体地说：

- 任务系统决定“谁负责什么任务、谁和谁协作、什么健康任务能被启动”
- 编排系统决定“这个 agent 能不能执行、能读什么、能写什么、走哪条 runtime lane”
- 健康系统决定“故障如何接入、如何诊断、如何报告、如何验证、如何形成后台记录”

这三者不能混成一层。

如果任务系统直接控制 runtime 权限，就会绕过编排系统。
如果编排系统负责决定业务任务分配，就会把身体配置层做成任务脑。
如果健康系统直接改写主 runtime 真相，就会污染 `TaskRunLedger / RuntimeCheckpoint / TaskResult`。

### 2.3 健康管理 agent 的定位是“受约束的管理者/代操者”

健康管理 agent 未来不应只是一个只读分析助手，但它也不能替代健康系统本身。

正确定位是：

- 健康系统是独立系统，拥有自己的操作台、后端 API、设置、测试、报告和 readiness。
- 用户可以直接使用健康系统操作台完成健康管理动作。
- 健康管理 agent 是系统创建出来的通用 agent 实例，被任务系统分配健康管理任务身份、灵魂投影、workflow 和 task flow binding。
- 健康管理 agent 在健康系统中充当更熟悉链路的管理员/代操者，可以解释、整理、建议、发起代操或申请开发级动作。
- 健康管理 agent 的能力由编排系统、resource policy、runtime lane 和审批策略硬约束。

它可以在授权范围内：

- 接收用户或系统上报的故障
- 基于 trace / checkpoint / ledger / execution record 追踪链路
- 生成问题报告草案或结构化诊断
- 通过健康系统后端写入健康后台
- 代用户启动健康测试
- 代用户提出健康设置调整
- 生成 cutover readiness 建议
- 发起 fix verification 请求

它不能：

- 替代健康系统原生操作台
- 成为健康系统唯一入口
- 直接改写 `TaskRunLedger`
- 直接改写 `RuntimeCheckpoint`
- 直接改写 `TaskResult`
- 直接绕过 `ResourcePolicy`
- 直接执行 shell / edit / worker 类扩权操作
- 直接把前端设置转译为运行时权限

这个边界是健康管理 agent 可以“管事”和“代操”，但不会吞掉健康系统或越权的关键。

### 2.4 健康系统交互形态采用“独立操作台 + 嵌入式 Agent 对话窗”

健康系统的用户入口不应放在任务系统里，也不应被 agent 对话窗替代。

建议采用当前设定：

- 用户点进健康系统，首先进入健康系统原生操作台。
- 健康系统操作台展示 issue、report、test、setting、readiness 等原生管理能力。
- 健康系统页面内嵌一个悬浮小框，作为用户和健康管理 agent 的对话窗口。
- 健康管理 agent 是由 agent 框架创建出来的真实 agent 实例。
- 该实例读取任务系统提供的通用 agent/task 连接配置，包括任务身份、灵魂投影、workflow 和 binding。
- 该实例执行时仍受编排系统的 capability profile、resource policy、runtime lane 和审批策略约束。
- 用户直操和 agent 代操最终都应能落成可审计的 `HealthManagementCommand` 和 `HealthManagementReceipt`，但 initiator 必须区分。

这个设计有一个很好的性质：

- 任务系统负责 agent/task 思维连接。
- 健康系统负责健康领域对象和原生操作台。
- 嵌入式对话窗负责用户与健康管理 agent 的对接。
- 编排系统负责约束 agent 的身体能力。

因此健康系统里的悬浮小框不是健康系统的主入口，而是健康管理 agent 的对接窗口。

## 三、理论依据

### 3.1 本地架构依据

现有系统已经形成了几个明确方向：

- 问题一已经把 `TaskRunLedger` 推向运行时真相源
- 问题二已经补了副作用执行语义层
- 问题三已经把模板选择和 bundle 拆分推向 contract-first
- 问题四计划书已经把故障演练、恢复验证、cutover readiness 定义成架构验证层

因此健康系统管理控制面的前提是：

- 不能再绕回散装启发式
- 不能再绕回“靠最终文本解释过程”
- 不能让健康子 agent 跳过 contract / ledger / execution record

### 3.2 AWS Operational Excellence 的依据

AWS Operational Excellence 强调：

- operations as code
- refine procedures frequently
- anticipate failure
- learn from operational events

这对应到本系统，就是：

- 健康测试和故障演练必须代码化
- 健康设置必须可审计
- 故障报告必须有后台记录
- 切流前必须有固定证据

来源：

- https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/oe-design-principles.html

### 3.3 AWS Reliability 的依据

AWS Reliability 强调：

- fail fast
- control and limit retries
- avoid stale backlog
- design for recovery

这对应到本系统，就是：

- 健康子 agent 可以调度测试，但失败必须被结构化记录
- replay denied、permission denied、retry exhausted 必须进入健康报告
- delayed worker / stale backlog 不能只靠人工翻 trace

来源：

- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_mitigate_interaction_failure_fail_fast.html
- https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_mitigate_interaction_failure_limit_retries.html

### 3.4 Durable runtime 的依据

Temporal / LangGraph 这类 durable runtime 的共同点是：

- 执行进度由系统事实记录
- 恢复决策基于持久状态
- 副作用必须有明确边界
- trace / checkpoint / event history 能解释失败和恢复

这对应到本系统，就是：

- 健康系统只能读取并解释主 runtime 事实
- 健康系统不能创造另一套执行真相
- 健康验证结果应该引用 ledger / checkpoint / trace / execution record，而不是替代它们

来源：

- https://docs.temporal.io/workflow-execution
- https://assets.temporal.io/durable-execution.pdf
- https://docs.langchain.com/oss/javascript/langgraph/durable-execution

## 四、目标架构

### 4.1 四层结构

建议目标结构分成五个协作面。

```text
Task System Agent/Task Connection
  -> Orchestration Permission / Runtime Contracts
  -> Health System Independent Control Plane
      -> Health System Native Console
      -> Embedded Health Agent Conversation Window
```

五个协作面职责如下：

1. 任务系统 agent/task 连接层
   - 管理有哪些 agent 可参与任务
   - 定义 task flow
   - 定义 agent-task binding
   - 定义 coordination topology
   - 绑定灵魂投影、projection template、skill workflow
   - 向编排系统提供 agent 的任务身份和工作流连接
   - 健康管理 agent 只是其中一个 agent 实例

2. 编排权限与运行层
   - 校验 capability profile
   - 生成 resource policy
   - 决定 runtime lane
   - 写入 checkpoint / ledger / execution record / trace

3. 健康系统独立控制面
   - 接收健康命令
   - 创建或更新 health issue
   - 启动 health agent run
   - 调度验证与故障演练
   - 生成报告和回执
   - 汇总 readiness

4. 健康系统原生操作台
   - 面向用户提供健康系统直接操作入口
   - 展示 issue、report、run、readiness、setting、test run
   - 支持用户直接创建、查询、启动、调整和验证
   - 需要审计和执行的动作提交为健康管理命令

5. 健康系统嵌入式 agent 对话窗
   - 承载用户与健康管理 agent 的对话
   - 绑定当前 issue / run / report / test 上下文
   - 把 agent 代操请求提交为健康管理命令
   - 不承载任务系统的 agent 配置职责

### 4.2 数据流

目标数据流如下：

```text
用户直操路径:
  用户
    -> 健康系统原生操作台
    -> HealthManagementCommand(initiator_type=user)
    -> HealthIssue / HealthTestRun / HealthReport / HealthSetting

Agent 代操路径:
  用户
    -> 健康系统嵌入式 agent 对话窗
    -> HealthAgentConversationSession
    -> HealthManagementCommand(initiator_type=agent)
    -> TaskFlowBinding / AgentCapabilityProfile / ResourcePolicy validation
    -> HealthAgentRun / HealthReport / HealthTestRun

系统自动路径:
  系统故障线索或测试失败
    -> HealthManagementCommand(initiator_type=system 或 test_system)
    -> HealthIssue / HealthReport / ReadinessReport
```

关键规则：

- 任务系统只管理通用 agent/task 连接，健康管理 agent 只是其中一个实例
- 编排系统只做运行权限与执行真相
- 健康系统独立拥有 issue、report、test、setting、readiness
- 健康系统原生操作台和嵌入式 agent 对话窗都可以触发健康管理命令
- 嵌入式 agent 对话窗不替代健康系统原生操作台

### 4.3 健康子 agent 实例模型

健康子 agent 应该通过现有 agent 框架创建实例，而不是健康系统自己临时拼一个“特殊助手”。

建议新增或显式复用如下实例语义：

```python
class HealthAgentConversationSession:
    session_id: str
    agent_id: str
    agent_profile_id: str
    projection_template_id: str
    skill_workflow_id: str
    runtime_lane: str
    health_context_ref: str
    active_issue_ref: str
    command_refs: tuple[str, ...]
    created_at: float
```

实例创建时必须满足：

- `agent_id` 来自 agent registry，例如 `agent:health:maintainer`
- `agent_profile_id` 来自 capability profile，例如 `health_maintainer_agent`
- `projection_template_id` 来自任务系统配置
- `skill_workflow_id` 来自 health task flow binding
- `runtime_lane` 来自编排系统允许的 lane
- 对话中的动作必须转成 command，不直接写 runtime truth

这样健康子 agent 在用户体验上是一个悬浮对话对象，在系统结构上仍是标准 agent 实例。

### 4.4 管理命令模型

建议新增：

```python
class HealthManagementCommand:
    command_id: str
    command_type: str
    requested_by: str
    source: str
    target_scope: str
    task_mode: str
    payload: dict[str, Any]
    created_at: float
```

`command_type` 最小集合：

- `report_issue`
- `analyze_trace`
- `draft_case`
- `verify_fix`
- `launch_health_test`
- `update_health_setting`
- `build_cutover_readiness`

### 4.5 管理回执模型

建议新增：

```python
class HealthManagementReceipt:
    receipt_id: str
    command_ref: str
    accepted: bool
    status: str
    health_issue_ref: str
    health_run_ref: str
    test_run_ref: str
    report_ref: str
    blocked_reasons: tuple[str, ...]
    diagnostics: dict[str, Any]
    created_at: float
```

这让每一次健康管理动作都有后台账本，而不是只存在于对话文本或前端临时状态。

### 4.6 健康设置模型

建议新增：

```python
class HealthSystemSetting:
    setting_id: str
    key: str
    value: Any
    scope: str
    updated_by: str
    updated_at: float
    validation_state: str
    diagnostics: dict[str, Any]
```

最小设置范围：

- 默认健康子 agent
- 默认 trace analysis flow
- 默认 fix verification flow
- 启用的测试组
- cutover readiness 必需项
- 故障报告自动创建策略
- 健康测试运行模式

重要约束：

- 设置可以改变默认任务入口
- 设置不能直接改变运行时权限
- 设置变更必须再次经过任务系统和编排系统验证

### 4.7 健康测试目录

建议新增：

```python
class HealthTestScenario:
    scenario_id: str
    title: str
    category: str
    owner_system: str
    required_flows: tuple[str, ...]
    expected_invariants: dict[str, Any]
    default_enabled: bool
```

最小测试分类：

- `static_contract`
- `runtime_happy_path`
- `fault_injection`
- `recovery_replay`
- `cutover_readiness`
- `health_agent_management`

### 4.8 健康报告模型

建议新增：

```python
class HealthReport:
    report_id: str
    report_type: str
    issue_ref: str
    command_ref: str
    evidence_refs: tuple[str, ...]
    verdict: str
    severity: str
    summary: str
    recommended_actions: tuple[str, ...]
    created_at: float
```

`report_type` 最小集合：

- `incident_report`
- `trace_analysis`
- `case_draft`
- `fix_verification`
- `readiness_report`
- `setting_change_report`

## 五、职责矩阵

### 5.1 任务系统职责

任务系统负责：

- 管理通用 agent/task 连接
- 管理 agent 可见任务身份
- 管理 task flow
- 管理 agent-task binding
- 管理 coordination topology
- 管理 agent 的灵魂投影设定
- 管理 agent 可使用的 skill workflow
- 把 agent 的任务连接配置提供给编排系统
- 健康管理 agent 只是这套机制里的一个实例

任务系统不负责：

- 不直接修改 runtime lane
- 不直接改变 allowed operations
- 不直接写 checkpoint / ledger / execution record
- 不管理健康 issue、健康设置、健康测试运行和健康报告正文
- 不把健康管理 agent 特例化成任务系统唯一目标

### 5.2 编排系统职责

编排系统负责：

- 校验 agent capability profile
- 生成 resource policy
- 决定 runtime directive
- 维护 ledger / checkpoint / trace / execution record
- 对高风险动作执行 fail-closed

编排系统不负责：

- 不决定健康任务的业务含义
- 不决定健康系统嵌入式 agent 对话窗如何展示
- 不生成最终健康报告正文

### 5.3 健康系统职责

健康系统负责：

- 维护 health issue
- 维护 health agent run
- 维护 health management command / receipt
- 维护 health setting
- 维护 health test scenario
- 维护 health report
- 调度健康子 agent 执行分析和验证

健康系统不负责：

- 不绕过编排权限
- 不直接改写主 runtime 真相
- 不把健康报告当成 runtime truth

### 5.4 健康子 agent 职责

健康子 agent 负责：

- 从故障线索创建或补充 issue
- 阅读 trace / checkpoint / ledger / execution record
- 生成诊断、报告、验证结论
- 根据用户请求发起健康测试命令
- 根据用户请求提出健康设置变更
- 把结果写入健康后台

健康子 agent 不负责：

- 不直接修代码
- 不直接改内部结构真相
- 不直接扩权
- 不替代主 agent 做最终业务回答

### 5.5 健康系统嵌入式 agent 对话窗职责

健康系统嵌入式 agent 对话窗负责：

- 在健康系统中展示悬浮小框
- 承载用户与健康管理 agent 的对话
- 绑定当前健康 issue / run / report / test 上下文
- 展示 agent 代操命令的 receipt
- 展示 agent 代操产生的 report/run/issue 引用
- 把 agent 代操请求提交为健康管理命令

健康系统嵌入式 agent 对话窗不负责：

- 不直接写 runtime 状态
- 不直接调用底层执行器
- 不绕过后台验证
- 不管理子 agent 的灵魂投影、workflow、task binding
- 不直接创建绕过 agent 框架的特殊子 agent
- 不替代健康系统原生操作台

## 九、任务系统、健康操作台与嵌入式 agent 对话窗调整方向

任务系统不新增健康管理区。

任务系统只负责通用 agent/task 连接，为所有 task-visible agent 提供合适的任务身份、灵魂投影设定、工作流、任务流绑定和协作拓扑。健康管理 agent 是其中一个实例。健康系统的 issue、report、test、setting、readiness 应归健康系统原生操作台和健康系统后台管理；健康管理 agent 可以通过嵌入式对话窗代操，但不能替代健康系统。

任务系统中的最小视图：

- agent 概览
- 灵魂投影设定
- skill workflow
- task flows
- agent-task binding
- capability profile 摘要
- runtime lane 摘要

健康系统原生操作台中的最小视图：

- issue 列表
- report 列表
- agent run 列表
- 健康测试组
- 健康设置
- readiness 状态

健康系统嵌入式 agent 对话窗中的最小视图：

- 悬浮小框
- 健康管理 agent 对话流
- 当前绑定 issue / run / report / test
- 代操 command receipt
- 代操结果引用

健康系统原生操作台和嵌入式 agent 对话窗中的执行动作都应提交为后台命令，但 initiator 必须区分：

- 用户直操 -> `initiator_type=user`
- agent 代操 -> `initiator_type=agent`

嵌入式 agent 对话窗中的典型代操动作：

- “启动链路分析” -> `analyze_trace`
- “启动修复验证” -> `verify_fix`
- “运行健康测试组” -> `launch_health_test`
- “修改健康设置” -> `update_health_setting`
- “生成切流报告” -> `build_cutover_readiness`

## 十、旧路径清理规则

本轮重构必须避免留下隐性旧路径。

需要清理或降级的旧路径：

- 直接从悬浮小框调用散装 issue/run endpoint 的路径
- 绕过 command / receipt 的健康测试启动路径
- 不经过 capability profile 的健康 agent run 启动路径
- 不经过 agent 框架直接创建健康对话助手的路径
- 只生成聊天文本、不写后台 report 的健康分析路径
- 不引用 runtime truth 的健康结论路径

允许保留的过渡路径：

- 老 issue/run endpoint 可以短期保留，但内部应调用 command handler
- 老 trace report API 可以保留，但应该逐步引用 invariant/report 结构

## 十一、验收标准

本统筹计划完成后，至少应满足：

1. 健康子 agent 已经从只读分析助手升级为受约束的健康系统管理者。
2. 任务系统可以配置健康子 agent 的任务身份、灵魂投影、工作流和任务绑定。
3. 编排系统仍然拥有最终运行权限裁决权。
4. 健康系统拥有 command / receipt / setting / scenario / report 的后台账本。
5. 健康测试可以通过 pytest、健康系统原生操作台、健康管理 agent 代操三类路径触发。
6. 测试结果和故障报告能进入健康后台。
7. 健康子 agent 的任何结论都能追溯到 runtime truth refs。
8. 设置变更不能直接绕过 resource policy。
9. 切流与回滚有固定 readiness 报告。
10. 健康系统嵌入式 agent 对话窗使用 agent 框架创建健康管理 agent 会话实例。
11. 旧的散装健康管理路径已经清理或明确降级为兼容层。

## 十二、建议实施顺序

建议严格按以下顺序实施：

1. `Phase 25A`：领域模型
2. `Phase 25B`：命令入口
3. `Phase 25C`：任务系统中的通用 agent/task 连接层

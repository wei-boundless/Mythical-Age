# 上下文管理与当前轮真相重构计划

日期：2026-05-02

## 一、问题定义

本轮长情景暴露的问题不是单个 PDF、结构化数据或天气工具能力不足，而是上下文管理的真相边界不稳定。

典型失败链路：

1. 用户提出复合任务：总结 PDF 第三页、分析 inventory.xlsx 最缺货仓库、补一句北京天气。
2. 模型实际已经请求了 `pdf_analysis`、`structured_data_analysis`、`get_weather` 三个工具。
3. 运行时逐个工具结果尝试 `direct_tool` 收口。
4. 最后一个天气结果覆盖最终答案。
5. PDF 和结构化数据结果没有形成复合任务结果账本。
6. 下一轮用户说“只展开第二个子任务”时，系统没有稳定绑定对象，只能退回自然语言猜测。

所以当前损坏的系统属性是：

- 当前轮真相没有唯一所有者。
- 文件工作对象绑定没有成为 follow-up 决策入口。
- direct tool、worker、memory、prompt 四条链路对上下文的表达不一致。
- Prompt 中展示了过多本应由系统 resolver 消费的状态候选。

正确目标不是让 prompt 更会解释上下文，而是让系统在模型调用前已经完成上下文绑定与当前轮裁决。

## 二、重新读取后的设计原则约束

本计划只采用现有设计原则中已经明确的方向，不引入额外审批系统、preview 模式或 denied prompt 说明。

### 2.1 状态管理原则

来自 `docs/设计原则/03-状态管理.md`：

- 状态必须按生命周期分层。
- 全局状态、应用状态、单次工具/任务运行上下文不能混用。
- 越底层的状态越要克制，不应无限膨胀。

落到当前系统：

- session state 只能保存可恢复状态。
- runtime context 只能服务当前任务。
- model visible prompt 只是表达层，不是状态所有者。

### 2.2 对话循环原则

来自 `docs/设计原则/05-对话循环.md`：

- 对话循环是状态机，不是一次性问答。
- 工具调用后必须把工具结果回传给模型，直到形成最终收口。
- 每个 continue / terminal 分支都必须明确转移原因。

落到当前系统：

- 多工具结果不能被单个 direct answer 提前终止。
- 工具结果回传、综合收口、最终提交必须是同一状态机的一部分。
- direct tool 只能是单工具快捷路径，不能绕开复合任务聚合。

### 2.3 上下文管理原则

来自 `docs/设计原则/06-上下文管理.md`：

- 上下文窗口里不应长期保留全部原始工具结果。
- 历史工具结果应逐步压缩为摘要、引用或可恢复状态。
- 恢复上下文依赖结构化状态，而不是让模型阅读大量旧文本自行判断。

落到当前系统：

- 文件对象、结果对象、子集对象必须用 handle/ref 表达。
- Prompt 只展示必要可用性信号。
- Restore 候选应先交给 resolver，不应直接作为高优先级自然语言塞进 prompt。

### 2.4 任务系统原则

来自 `docs/设计原则/14-任务系统.md`：

- 异步任务要有生命周期、隔离和输出收集。
- 任务状态和任务输出不能混成普通聊天文本。
- 任务完成后应形成可追踪结果，而不是只留下自然语言描述。

落到当前系统：

- 复合任务必须产生 bundle item / subtask refs。
- 子任务结果应可被后续“第一个 / 第二个 / 第三个子任务”稳定回指。
- 工具执行结果不是最终上下文，投影后的 task summary / handle 才是可恢复上下文。

### 2.5 Memory 系统原则

来自 `docs/设计原则/23-Memory系统.md`：

- 长期记忆、会话记忆、相关记忆应分层。
- 可从当前项目状态或工具结果派生的信息，不应重复写成长期事实。
- Relevant memories 是按需注入，不是无差别注入。

落到当前系统：

- 当前任务绑定不应写入长期记忆。
- session memory 可以保存当前工作对象，但必须是结构化绑定，不是把 prompt 契约或工具协议转写成摘要。
- Hot Truth Window 是连续性摘要，不是当前轮事实源。

## 三、当前代码链路复核

### 3.1 当前上下文入口

入口文件：

- `backend/runtime/agent_chain.py`
- `backend/memory_system/runtime_view.py`
- `backend/context_policy/package_builder.py`
- `backend/orchestration/runtime_loop/context_manager.py`

当前流程：

```text
user message
  -> analyze_memory_intent
  -> build_memory_runtime_view
  -> build_memory_context_package
  -> analyze_query_understanding(active_bindings)
  -> build_task_runtime_contract
  -> prepare_model_context
  -> system prompt + history + user
```

问题：

- `MemoryRuntimeView` 同时包含 context candidates 和 restore candidates。
- `context_policy` 把 state candidate 渲染进 `active_process_context`。
- `query_understanding` 只从 active bindings 中拿少量路径信息，没有拿到完整 current-turn binding contract。
- `task_contract` 没有承载 bound source/result/subset，也没有承载 bundle item。

### 3.2 当前工具结果投影

入口文件：

- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/workers/projection.py`

当前已有能力：

- `_project_file_work_context_from_tool_observation()` 可从 `pdf_analysis` / `structured_data_analysis` 结果产生 `main_context` 和 `task_summary_refs`。
- `WorkerProjectionAdapter` 可从 worker canonical result 产生 `MainContextState`、`TaskSummaryRef`、object/result handles。

问题：

- direct tool 和 worker projection 是两套投影协议。
- direct tool 收口可能阻断 follow-up synthesis。
- 多工具结果的 `final_main_context` / `final_task_summary_refs` 目前是覆盖式赋值，不是聚合式账本。

### 3.3 当前 session 写回

入口文件：

- `backend/query/runtime.py`
- `backend/structured_memory/session_memory.py`
- `backend/memory/facade.py`

当前已有能力：

- assistant commit 时，如果 payload 有 `main_context` 或 `task_summary_refs`，会优先调用 `refresh_session_memory_from_context_state()`。
- session memory 能从 task summaries 中恢复 `pdf=`、`dataset=`、`pdf_pages=` 等 key points。

问题：

- 写回入口是对的，但上游经常没有稳定聚合所有子结果。
- session memory 仍会把状态再投成自然语言 summary，后续又可能作为 prompt continuity 被模型消费。
- 当前 active goal、active binding、committed file、handle refs 被装在同一个状态槽体系里，生命周期边界不清。

### 3.4 当前 follow-up 理解

入口文件：

- `backend/understanding/task_understanding.py`
- `backend/understanding/query_understanding.py`
- `backend/structured_memory/turn_understanding.py`
- `backend/structured_memory/understanding_reconciliation.py`

当前已有能力：

- 显式路径优先。
- bound pdf / bound dataset 能辅助 PDF 和结构化数据 follow-up。

问题：

- follow-up resolver 没有强类型输入。
- “第二个子任务”“这些结果”“刚才那个表”的解析仍偏自然语言启发式。
- restore candidates 没有进入统一 binding arbitration，而是部分进入 prompt、部分进入 active bindings。

## 四、核心结构性结论

当前系统的问题不是“上下文太少”，而是“上下文平面没有分离”。

必须拆成三个平面：

### 4.1 Decision Plane：决策平面

系统内部使用，模型不可见或只通过摘要信号间接可见。

负责：

- 当前用户消息解析。
- 显式文件 / 显式工具 / 显式子任务识别。
- 从 session state、task summaries、restore candidates 中恢复绑定候选。
- 产出当前轮唯一 `CurrentTurnContext`。

不负责：

- 生成给用户看的回答。
- 保存长期记忆。
- 拼 prompt 文本。

### 4.2 Execution Plane：执行平面

运行时状态机使用。

负责：

- 根据 `CurrentTurnContext` 选择工具、worker、bundle execution。
- 执行工具。
- 聚合多个 observation。
- 产生 canonical result graph。
- 统一投影成 `ContextProjection`。

不负责：

- 从 prompt 文本里猜上下文。
- 把工具结果直接当长期状态。

### 4.3 Expression Plane：表达平面

模型可见 prompt 和最终回答使用。

负责：

- 给模型最小必要上下文信号。
- 组织最终回答。
- 告诉模型当前可继续的对象类型，而不是暴露内部 handle 协议。

不负责：

- 决定当前绑定。
- 执行权限过滤。
- 充当状态事实源。

## 五、目标架构

### 5.1 新增 CurrentTurnContext

建议新增模块：

- `backend/context_management/current_turn.py`

核心模型：

```python
CurrentTurnContext:
  session_id
  task_id
  user_message
  intent
  execution_mode: single | bundle
  explicit_inputs
  resolved_bindings
  bundle_items
  restore_candidates_used
  unresolved_ambiguities
  confidence
  authority = "context.current_turn"
```

其中：

```python
ResolvedBinding:
  binding_id
  binding_kind: source_file | result | subset | task_ref
  file_kind
  identity
  source_handle_id
  result_handle_id
  subset_handle_id
  owner_task_id
  confidence
  source: explicit_user_input | session_state | task_summary | restore_candidate
```

```python
BundleItem:
  item_id
  ordinal
  user_text
  capability_kind
  target_binding
  required_tool
  output_requirement
```

### 5.2 新增 ContextResolver

建议新增模块：

- `backend/context_management/resolver.py`

输入：

- 当前用户消息
- `MemoryRuntimeView.state_snapshot`
- `MemoryRuntimeView.restore_candidates`
- 最近 `task_summary_refs`
- `QueryUnderstanding`

输出：

- `CurrentTurnContext`

固定规则：

1. 当前用户显式路径优先。
2. 当前用户显式子任务序号优先绑定最近 bundle items。
3. 明确 follow-up 指代绑定 active subset/result/source。
4. restore candidates 只能作为候选，不能自我提升为当前事实。
5. 多候选同等可信时才澄清。
6. 当前用户消息与历史候选冲突时，以当前用户消息为准。

### 5.3 新增 ContextProjection

建议新增模块：

- `backend/context_management/projection.py`

核心模型：

```python
ContextProjection:
  main_context
  task_summary_refs
  bundle_summary_refs
  object_handle_ids
  result_handle_ids
  subset_handle_ids
  memory_policy
  prompt_visibility
  authority = "context.execution_projection"
```

用途：

- 替代 direct tool 私有投影。
- 兼容 worker projection。
- 支持多工具聚合。
- 成为 assistant commit 写回 session memory 的唯一结构入口。

### 5.4 收缩 ModelVisibleContext

调整模块：

- `backend/context_policy/package_builder.py`
- `backend/orchestration/runtime_loop/context_manager.py`
- `backend/memory_system/state_memory.py`

目标：

- state candidate 不再以 raw preview 进入 `active_process_context`。
- prompt 只展示经过 `CurrentTurnContext` 和 `ContextProjection` 过滤后的可用信号。

示例：

```text
当前可继续处理：
- 一个 PDF 分析结果可继续展开。
- 一个表格分析结果可继续按仓库/商品/缺口维度追问。
- 上一轮包含 3 个子任务结果，可按序号回指。
```

不再展示：

```text
active_goal: ...
next_step: ...
committed_pdf: ...
active_result_handle_id: ...
```

## 六、固定执行流

目标流程：

```text
用户输入
  -> MemoryRuntimeView 读取 session / durable / restore candidates
  -> QueryUnderstanding 初步识别能力与显式输入
  -> ContextResolver 产出 CurrentTurnContext
  -> TaskRuntimeContract 绑定 CurrentTurnContext
  -> RuntimeLoop 执行 single 或 bundle
  -> Tool/Worker 返回 observations
  -> ObservationAggregator 聚合所有结果
  -> ContextProjection 统一投影
  -> Answer synthesis 生成最终回答
  -> Assistant commit
  -> SessionMemory 写回 ContextProjection
  -> 下一轮从 CurrentTurnContext resolver 恢复
```

## 七、直接要删除或收缩的旧旁路

### 7.1 direct tool 多工具提前收口

位置：

- `backend/orchestration/runtime_loop/task_run_loop.py`

规则：

- 同一模型轮只出现一个 tool call，才允许 `_direct_tool_answer_from_observation()` 快捷收口。
- 出现多个 tool call 时，必须进入 aggregation + synthesis。
- direct tool 不能直接写最终 `final_content` 并阻断 follow-up。

### 7.2 final_main_context 覆盖式赋值

位置：

- `backend/orchestration/runtime_loop/task_run_loop.py`

规则：

- 多 observation 时不能用最后一个投影覆盖前一个。
- 必须按 observation 生成多个 `TaskSummaryRef`，并聚合到 `bundle_summary_refs`。

### 7.3 state candidate raw prompt 注入

位置：

- `backend/memory_system/state_memory.py`
- `backend/context_policy/package_builder.py`

规则：

- state snapshot 可进入 resolver。
- state raw preview 不应直接进入 model visible prompt。
- 如果要展示，只能展示经过 `ModelVisibleContext` 转换的能力信号。

### 7.4 durable context state 自然语言签名污染

位置：

- `backend/memory/durable.py`

规则：

- `main_context` / `task_summaries` 不应被投成长期事实。
- 文件工作对象和本轮任务结果默认 `session_context_only`。
- durable memory 只保存用户明确要求长期记住的偏好、原则、项目事实。

## 八、实施阶段

### 阶段 1：建立 CurrentTurnContext 数据结构

目标：

- 不改变行为，先建立结构化输出。
- 让长测 artifact 能看到 current turn 绑定结果。

影响文件：

- 新增 `backend/context_management/current_turn.py`
- 新增 `backend/context_management/resolver.py`
- 修改 `backend/runtime/agent_chain.py`
- 修改 `backend/tasks/contract_builder.py`

完成标准：

- 当前用户显式 PDF / dataset / weather 可进入 `CurrentTurnContext.explicit_inputs`。
- session active pdf / dataset / handle 可进入 `restore_candidates`，但不会覆盖显式输入。
- task contract payload 中出现 `current_turn_context`。

### 阶段 2：复合任务识别与 bundle item 建账

目标：

- 复合任务在模型调用前形成 bundle items。
- “先...再...最后...”这类请求不再只依赖模型自由发多个工具。

影响文件：

- `backend/understanding/task_understanding.py`
- `backend/context_management/resolver.py`
- `backend/tasks/contract_builder.py`
- `backend/tests/system_eval/long_runner.py` 只增加观测，不改业务逻辑。

完成标准：

- 示例请求形成 3 个 bundle items。
- `execution_mode = bundle`。
- 每个 item 有 ordinal、capability_kind、target binding、required tool。

### 阶段 3：统一 ObservationAggregator 和 ContextProjection

目标：

- direct tool 和 worker path 使用同一投影协议。
- 多工具结果聚合后再收口。

影响文件：

- 新增 `backend/context_management/projection.py`
- 新增 `backend/orchestration/runtime_loop/observation_aggregator.py`
- 修改 `backend/orchestration/runtime_loop/task_run_loop.py`
- 收敛 `backend/workers/projection.py` 到共享模型或适配器。

完成标准：

- 单工具仍可 direct answer。
- 多工具必须 synthesis。
- 多工具结果产生多个 task refs 和 bundle refs。
- `final_main_context` 不再被最后一个工具覆盖。

### 阶段 4：follow-up resolver 改为结构化绑定优先

目标：

- “第二个子任务”“这些人”“刚才那个表”“这份 PDF”优先走 `CurrentTurnContext` 绑定。

影响文件：

- `backend/understanding/query_understanding.py`
- `backend/understanding/task_understanding.py`
- `backend/context_management/resolver.py`
- `backend/structured_memory/session_memory.py`

完成标准：

- “只展开第二个子任务”能绑定上一轮 bundle item 2。
- “把第一个和第三个子任务各压成一句话”能绑定 bundle item 1 和 3。
- PDF / dataset follow-up 不需要从 Hot Truth Window 猜。

### 阶段 5：收缩 prompt 可见上下文

目标：

- 模型只看到表达层信号。
- 系统绑定事实不再以 raw state preview 泄露到 prompt。

影响文件：

- `backend/context_policy/package_builder.py`
- `backend/orchestration/runtime_loop/context_manager.py`
- `backend/memory_system/state_memory.py`

完成标准：

- Prompt 中不再出现 raw `active_result_handle_id`、`committed_pdf`、`active_goal` 状态 dump。
- Prompt 可见内容仍能提示“当前有可继续处理的 PDF / 表格 / bundle 结果”。
- 长期记忆、Hot Truth Window 不会覆盖当前用户消息。

### 阶段 6：清理旧链路和回归测试

目标：

- 删除或封死旧的上下文旁路。
- 用长情景验证结构稳定性。

影响文件：

- `backend/memory/durable.py`
- `backend/memory_system/writeback.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`
- 相关 tests。

完成标准：

- 无 `preview_only`、`Denied:`、权限解释型 prompt 回流。
- 复合任务通过。
- PDF follow-up 通过。
- 结构化数据 follow-up 通过。
- RAG 不被 file-work binding 干扰。

## 九、验证矩阵

### 9.1 单工具任务

用例：

- “查询北京天气。”
- “分析 inventory.xlsx 缺货情况。”
- “总结 PDF 第三页。”

期望：

- 单工具可 direct answer。
- 成功文件工具产生 `ContextProjection`。
- session state 写回 active source/result/subset。

### 9.2 复合任务

用例：

- “先总结 PDF 第三页，再给我 inventory.xlsx 最缺货前三个仓库，最后补一句北京天气。”

期望：

- bundle items = 3。
- 三个工具结果都出现在最终回答。
- task refs = 3。
- follow-up 可按序号回指。

### 9.3 Follow-up

用例：

- “只展开第二个子任务。”
- “把第一个和第三个子任务各压成一句话。”
- “这些仓库里哪个最需要补货？”
- “这份 PDF 第二部分约束是什么？”

期望：

- resolver 明确绑定目标。
- 不依赖 Hot Truth Window 猜测。
- 不错误切到 RAG。

### 9.4 冲突处理

用例：

- 上轮 active PDF 是 A，本轮显式要求 PDF B。
- 上轮 active dataset 是 inventory，本轮显式 employees。

期望：

- 当前显式输入覆盖历史绑定。
- 旧绑定可作为 warm context，不作为 current fact。

### 9.5 Memory 安全

用例：

- prompt 契约内容出现在历史消息或工具结果附近。
- 文件工作状态写回 session memory。

期望：

- 契约不进入 durable memory。
- file-work binding 仅 session context。
- long-term memory 只保存明确长期偏好或项目事实。

## 十、风险与控制

### 风险 1：过早移除 state prompt 导致模型不知道可继续对象

控制：

- 先引入 `ModelVisibleContext`，保留自然语言可用性信号。
- 再移除 raw state preview。

### 风险 2：bundle 识别过强，误把普通列举问题拆成复合任务

控制：

- bundle item 只在明确顺序连接词、多个能力域、多个文件/实时查询混合时触发。
- 低置信度时保留 single execution。

### 风险 3：direct tool 单工具体验变慢

控制：

- 单工具仍保留 direct answer。
- 只有多工具或需要文件工作写回时强制 aggregation。

### 风险 4：新旧投影并存导致双写

控制：

- 阶段 3 之后，assistant commit 只认 `ContextProjection`。
- 旧 `main_context` / `task_summary_refs` 作为兼容输入，但不再由多个模块各自生成。

## 十二、最终验收口径

这次重构完成后，系统应满足：

1. 当前轮事实由 `CurrentTurnContext` 表达。
2. 工具和 worker 结果由 `ContextProjection` 表达。
3. Prompt 只消费 `ModelVisibleContext`，不直接消费 raw state。
4. Session memory 保存可恢复绑定，不保存 prompt 契约。
5. Direct tool 快捷路径不能破坏多工具综合。
6. Follow-up 先绑定结构化对象，再决定工具和回答方式。
7. 长情景中复合任务、PDF follow-up、结构化数据 follow-up 三类问题不再互相污染。

一句话目标：

> 把“让模型读上下文猜当前任务”，改成“系统先解析并绑定当前任务，再让模型基于已绑定事实执行和表达”。

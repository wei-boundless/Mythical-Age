# 问题一：任务运行账本与步骤状态机差距报告

日期：2026-05-02

扩展施工文档：

- `21-问题一-步骤级状态机重构计划书-20260502.md`

## 一、问题定义

当前系统已经有 `TaskRunLedger`、`TaskStepRun`、`TaskResult`，但它们更接近“运行结束后的结果表达层”，而不是“运行中实时推进的执行真相层”。

真正损坏的系统属性不是“少几个字段”，而是：

- 谁拥有步骤状态的真相还不够明确。
- 运行时没有把步骤迁移当作第一等状态转移。
- 最终结果仍然在相当程度上依赖终局推断，而不是中途逐步沉淀。

如果这一点不先修正，后面的 retry、resume、cancel、worker orchestration 都会建立在脆弱账本之上。

## 二、当前代码证据

核心文件：

- `backend/tasks/run_models.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`

当前结构已经具备的部分：

- `TaskStepRun` 能描述步骤类型、执行器、输入引用、输出引用、状态。
- `TaskRunLedger` 能描述当前步骤、请求输出、步骤账本、附加引用。
- `TaskResult` 能承载终局结果和最终步骤列表。

当前仍然存在的结构问题：

- `build_task_run_ledger()` 只负责初始化 step run，初始状态基本都是 `pending`。
- loop 中没有形成“进入某步骤 -> 写运行中 -> 写完成/失败 -> 推进下一步骤”的稳定转移协议。
- `_build_final_task_run_ledger()`、`_build_runtime_task_result()`、`_final_step_runs()` 会在终局根据 observation 和 final answer 反推步骤状态。

因此现在更像：

```text
执行过程
  -> 收集 observation
  -> 到 terminal 时综合判断
  -> 生成步骤结果账本
```

而不是：

```text
步骤状态机
  -> step running
  -> step completed / failed / skipped
  -> 推进下一步骤
  -> 产出终局结果
```

## 三、与先进架构的差距

### 3.1 与 LangGraph Durable Execution 的差距

LangGraph 的 durable execution 设计要求：

- execution step 的状态能够持久化；
- 恢复时从合适的 replay 起点继续；
- 副作用和非确定性动作被包进 task / node 边界；
- workflow 恢复依赖持久层中已记录的执行结构，而不是依赖终局自然语言推断。

当前差距：

- 我们已经有 checkpoint，但 step transition 还没有成为 checkpoint 的核心内容。
- 我们有最终 `TaskResult`，但还没有“步骤推进先于结果生成”的结构顺序。

### 3.2 与 Temporal Durable Execution 的差距

Temporal 的 durable execution 强调：

- durable workflow execution 是主执行单元；
- 系统持久化代码执行过的步骤；
- 进程失败后可以在保持状态的前提下继续完成工作流。

当前差距：

- 我们的账本更像“事后审计账本”，还不是“运行期 authoritative ledger”。
- 当前 loop 仍然以整体 terminal 收口为中心，而不是以 step transition 为中心。

## 四、会带来的具体问题

### 4.1 retry 语义会变形

如果某一步是否完成主要靠终局推断，那么 retry 时就很难回答：

- 是重试这个 step 还是重试整轮？
- 哪些 output ref 已经可信？
- 哪些工具动作已经对外生效？

### 4.2 resume 语义会变形

如果在中间步骤中断，恢复时缺少权威 step state，就只能靠 observation 历史和终局推断逻辑回放判断。

这会让恢复链变得复杂而脆弱。

### 4.3 cancel / skip / partial completion 难以一致表达

如果步骤状态不是 loop 主驱动字段，那么：

- 某步骤是否可跳过不清晰；
- 某步骤失败后是否允许继续不清晰；
- “部分完成”的合法状态也很难稳定表达。

### 4.4 worker 与多执行器接入后会迅速失真

当前问题在 direct tool 场景下还可勉强容忍，但一旦 step 里同时出现：

- worker
- shell
- browser verify
- 多 observation 聚合

终局推断会越来越不可信。

## 五、目标设计

### 5.1 目标原则

必须把 `TaskRunLedger` 提升为运行时单一真相源。

约束如下：

- step status 必须在执行时推进。
- step transition 必须伴随事件写入和 checkpoint 更新。
- `TaskResult` 只能从最终 ledger 投影出来，不能反过来决定 ledger。

### 5.2 目标状态转移模型

建议最小步骤状态机：

```text
pending -> running -> completed
pending -> running -> failed
pending -> skipped
running -> failed
running -> completed
```

配套字段建议：

- `entered_at`
- `completed_at`
- `executor_ref`
- `attempt_count`
- `retry_policy_ref`
- `step_result_ref`
- `failure_reason`

### 5.3 loop 与账本的关系

推荐收束方式：

- loop 不再在终局统一猜测所有步骤状态。
- loop 每推进一步就更新 ledger。
- terminal state 只负责确认最后一份 ledger，而不是重新生成一份步骤解释。

## 六、分阶段改造建议

### Phase 1A：引入显式 step transition 事件

目标：

- 新增 `step_entered`、`step_completed`、`step_failed`、`step_skipped`。
- 让 step event 成为 trace 的核心事件类型。

### Phase 1B：把 `current_step_id` 接入 loop 控制

目标：

- loop 的每次 continue / terminal 判断都能对照当前 step。
- 当前 step 不再只是展示字段，而是执行控制字段。

### Phase 1C：让 `TaskResult` 纯粹从最终 ledger 投影

目标：

- 收缩 `_build_runtime_task_result()` 的推断职责。
- 删除或降级依赖终局 observation 反推 step status 的旧逻辑。

## 七、验收标准

- 任一任务运行都能在中途回答当前 step 是什么、为何进入该 step。
- 任一步骤失败都能明确写出失败的 step id、attempt、failure reason。
- `TaskResult.step_runs` 与 `TaskRunLedger.step_runs` 一致，且来源单向明确。
- trace 中能直接看到步骤推进过程，而不是只看到终局总结。

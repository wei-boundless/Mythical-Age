# 96-长篇小说双创作者单裁判并入 TaskGraph 优化方案

## 0. 文档定位

本文用于参考旧方案 `48-长篇小说双作者隔离互审与审核员通关重构计划书-20260509.md`，优化当前 `88-长篇小说写作团队具体配置清单-20260515.md` 中的写作团队任务图。

本文不是恢复旧数据，也不是复用旧命名空间。旧 48 只作为流程设计参考；实际实现仍应使用当前新命名空间：

- `domain.writing_team.long_novel`
- `graph.writing_team.long_novel`
- `wmprofile.writing_team.long_novel`
- `projection.writing_team.long_novel.*`
- `contract.writing_team.long_novel.*`

## 1. 核心结论

当前 88 的写作团队已经具备 TaskGraph、投影、契约、时序、工作记忆资源节点等基础能力，但章节生产流程仍偏向：

```text
章节细纲
  -> 写作者 A 并行候选稿
  -> 写作者 B 并行候选稿
  -> 章节审稿
  -> 质量门
  -> 入库
```

旧 48 更适合长篇小说生产的关键点不是“多两个 Agent”，而是：

```text
章节细纲
  -> 作者 A 初稿
  -> 作者 B 审读意见
  -> 作者 A 综合修订稿
  -> 作者 B 终稿候选
  -> 独立裁判评分通关
  -> 记忆整理入库
```

也就是说，当前方案的问题是结构性的：

1. A/B 现在更像两个并行候选写手，不是隔离互审链。
2. 章节审稿人与质量门职责有重叠，独立裁判的权威性不够集中。
3. 世界观、大纲、人物等基础资产仍偏“单设计 + 单审核”，没有充分利用双创作者单裁判机制。
4. 工作记忆已经资源节点化，但写作团队图尚未把“读稳定资产、产出候选、审核采纳、入库提升”明确嵌入每个创作阶段。
5. 偏差修正还没有成为可显式调度的子图，当前返修主要回到 `chapter_plan`，不足以处理世界观、大纲、人物线偏差。

## 2. 不直接沿用旧 48 的原因

旧 48 的流程思想很好，但不能直接搬数据，原因如下：

1. 旧方案中的 Agent ID、库名和节点名不完全符合当前 `writing_team_long_novel` 主数据。
2. 当前系统已经引入工作记忆资源节点，旧方案没有表达 `memory_read`、`memory_handoff`、`memory_commit` 的运行时治理。
3. 当前图需要被前端 TaskGraph Studio 可视化编辑、预检、运行监控识别，因此节点、边、契约、投影、时序必须全部落到新结构。
4. 旧方案偏流程描述，当前需要补齐可运行配置：节点权限、输入输出契约、资源读写策略、质量门裁决字段、返修路由。

## 3. 优化目标

### 3.1 章节生产从“并行双稿”改为“隔离互审”

目标链路：

```text
chapter_plan
  -> writer_a_draft
  -> writer_b_review
  -> writer_a_revision
  -> writer_b_final_candidate
  -> novel_quality_judge
  -> memory_commit
```

每个节点的职责应写成 Agent 能理解的角色指令：

- `writer_a_draft`：你是一名章节初稿作者，只负责根据章节细纲写第一版正文。
- `writer_b_review`：你是一名章节审读作者，只负责阅读 A 的初稿并提出审读意见，不直接改写成终稿。
- `writer_a_revision`：你是一名章节修订作者，只负责根据 B 的审读意见和章节目标生成综合修订稿。
- `writer_b_final_candidate`：你是一名终稿处理作者，只负责基于修订稿做终稿候选整理，不替代裁判下结论。
- `novel_quality_judge`：你是一名长篇小说通关裁判，只评分、裁决、给返修目标，不写正文。
- `memory_commit`：你是一名写作资产管理员，只整理通过裁判的事实和摘要，不创造新内容。

### 3.2 基础资产引入“双方案 + 单裁判”

世界观、大纲、人物三个基础层建议改成：

```text
world_designer_a
world_designer_b
  -> world_judge
  -> memory_commit_world

outline_designer_a
outline_designer_b
  -> outline_judge
  -> memory_commit_outline

character_designer_a
character_designer_b
  -> character_judge
  -> memory_commit_character
```

这里的 A/B 不是共同写一份稿，而是形成可比较的候选或修订方案。裁判不创造第三套方案，只能：

1. 采纳 A。
2. 采纳 B。
3. 要求 A/B 基于裁判意见返修。
4. 在明确条件下允许“合并采纳”，但必须说明哪些条目来自 A、哪些来自 B、哪些是裁判认可的合并规则。

### 3.3 偏差修正成为显式子图

当前章节质量门只回 `chapter_plan`，这对普通章节返修足够，但对结构性偏差不够。

建议新增三类偏差修正路由：

```text
novel_quality_judge
  -> world_deviation_router
  -> world_repair_a / world_repair_b
  -> world_judge
  -> memory_commit_world

novel_quality_judge
  -> outline_deviation_router
  -> outline_repair_a / outline_repair_b
  -> outline_judge
  -> memory_commit_outline

novel_quality_judge
  -> character_deviation_router
  -> character_repair_a / character_repair_b
  -> character_judge
  -> memory_commit_character
```

偏差修正的原则：

1. 不回头重写全部基础资产。
2. 只基于已成文事实、已通过资产和当前偏差证据做最小修正。
3. 修正方案未通过裁判前，不得进入稳定记忆。
4. 如果偏差只属于章节执行问题，仍返回 `chapter_plan` 或 `writer_a_revision`，不触发基础资产修正。

## 4. 推荐目标图

### 4.1 阶段总览

```text
phase.foundation_world
  world_designer_a + world_designer_b
  -> world_judge
  -> memory_commit_world

phase.foundation_outline
  outline_designer_a + outline_designer_b

phase.foundation_character

  -> writer_b_review
```

### 4.2 章节生产目标节点

| 节点 | 角色 | 输入 | 输出 | 是否可写稳定记忆 |
| --- | --- | --- | --- | --- |
| `chapter_plan` | 章节规划师 | 已通过世界观、大纲、人物、前情摘要、返修意见 | 章节细纲 | 否 |
| `writer_a_draft` | 初稿作者 | 章节细纲、授权资产 | A 初稿 | 否 |
| `writer_b_review` | 审读作者 | A 初稿、章节细纲、授权资产 | B 审读意见 | 否 |
| `writer_a_revision` | 修订作者 | A 初稿、B 审读意见、章节细纲 | A 综合修订稿 | 否 |
| `writer_b_final_candidate` | 终稿处理作者 | 综合修订稿、B 审读意见、章节细纲 | 终稿候选 | 否 |
| `novel_quality_judge` | 独立裁判 | 授权资产、细纲、初稿、审读意见、修订稿、终稿候选 | 评分单与通关裁决 | 否 |
| `memory_commit_chapter` | 记忆整理员 | 通过裁判的终稿候选、评分单 | 章节摘要、正文事实、伏笔与变更记录 | 是，但只写通过内容 |

## 5. 契约优化

当前 `contract.writing_team.long_novel.chapter_draft` 同时承载 A/B 候选稿，目标结构中应拆成更细的契约：

1. `contract.writing_team.long_novel.chapter_plan`
2. `contract.writing_team.long_novel.writer_a_draft`
3. `contract.writing_team.long_novel.writer_b_review`
4. `contract.writing_team.long_novel.writer_a_revision`
5. `contract.writing_team.long_novel.writer_b_final_candidate`
6. `contract.writing_team.long_novel.novel_quality_judge`
7. `contract.writing_team.long_novel.memory_commit`

裁判契约必须包含：

- `score_total`
- `score_breakdown`
- `verdict`
- `pass_line`
- `must_fix`
- `can_keep`
- `deviation_type`
- `repair_target_node_id`
- `memory_commit_allowed`
- `approved_artifact_refs`
- `rejected_artifact_refs`

建议评分：

| 维度 | 分值 |
| --- | --- |
| 剧情推进 | 20 |
| 人物动机与行为可信度 | 20 |
| 世界观一致性与边界感 | 15 |
| 章节结构完整度 | 15 |
| 语言质感与可读性 | 10 |
| 长篇伏笔和后续延展 | 10 |
| 双作者互审方向控制 | 10 |

通过线：`85`。

## 6. 投影优化

### 6.1 写作者 A 初稿投影

```text
你是一名章节初稿作者。
你只负责根据已通过的世界观、人物设定、章节细纲和必要前情摘要写当前批次正文初稿。
你可以在已通过设定边界内做局部创造，但不能把新设定当成既成事实。
你需要列出本稿新增的候选事实、伏笔、人物状态变化和不确定点，供后续审读与裁判判断。
```

### 6.2 写作者 B 审读投影

```text
你是一名章节审读作者。
你只负责阅读写作者 A 的初稿，并根据章节细纲、人物动机、世界观边界和长篇推进目标提出审读意见。
你不直接写终稿，不替裁判下结论，也不把个人偏好包装成硬性规则。
你需要指出必须修复项、可保留亮点、节奏问题、设定风险和建议修改方向。
```

### 6.3 写作者 A 修订投影

```text
你是一名章节修订作者。
你只负责综合写作者 B 的审读意见，对自己的初稿进行修订。
你需要保留有效亮点，修正必须修复项，并说明哪些意见被采纳、哪些未采纳以及理由。
你不能绕过章节细纲，也不能新增未经说明的重大设定。
```

### 6.4 写作者 B 终稿候选投影

```text
你是一名终稿处理作者。
你只负责基于综合修订稿做终稿候选整理，提升连贯性、节奏和可读性。
你不能替裁判宣布通过，也不能把未解决的问题隐藏起来。
你需要输出终稿候选、保留问题、候选事实清单和建议裁判重点检查的风险。
```

### 6.5 独立裁判投影

```text
你是一名长篇小说通关裁判。
你只负责根据授权资产、章节细纲、初稿、审读意见、修订稿和终稿候选进行评分与裁决。
你不是第三作者，不写正文，不创造第三套设定。
你必须给出总分、分项评分、通过或返修裁决、必须修复项、可保留项、偏差类型和是否允许入库。
低于 85 分时，不允许进入记忆整理。
```

### 6.6 记忆整理投影

```text
你是一名写作资产管理员。
你只负责整理已经通过裁判的终稿、评分单和新增事实。
你不能创造新剧情、新设定或替裁判补充裁决。
你需要把内容拆成章节摘要、人物状态变化、世界观事实、伏笔、冲突、后续追踪项和来源引用。
只有裁判明确允许入库的内容才能写入稳定资产。
```

## 7. 工作记忆资源策略

### 7.1 读权限

| 节点 | 可读工作记忆 |
| --- | --- |
| `world_designer_a/b` | 用户硬设定、已通过世界观事实、相关返修意见 |
| `outline_designer_a/b` | 用户硬设定、已通过世界观、大纲历史、相关返修意见 |
| `character_designer_a/b` | 用户硬设定、已通过世界观、大纲、人物状态、相关返修意见 |
| `chapter_plan` | 已通过世界观、大纲、人物、前情摘要、裁判返修意见 |
| `writer_a_draft` | 章节细纲、已通过稳定资产、必要前情摘要 |
| `writer_b_review` | A 初稿、章节细纲、已通过稳定资产、必要前情摘要 |
| `writer_a_revision` | A 初稿、B 审读意见、章节细纲、已通过稳定资产 |
| `writer_b_final_candidate` | A 修订稿、B 审读意见、章节细纲、已通过稳定资产 |
| `novel_quality_judge` | 授权资产、细纲、A 初稿、B 审读、A 修订、B 终稿候选、评分规则 |
| `memory_commit_*` | 裁判通过的产物、评分单、approved refs |

### 7.2 写权限

1. 创作节点只能写 `candidate` 工作记忆。
2. 审读节点只能写 `review_note`、`risk_note`、`repair_suggestion`。
3. 裁判节点只能写 `review_decision`、`approved_ref`、`rejected_ref`、`deviation_signal`。
4. 记忆整理节点才可将通过内容提升为稳定资产。

### 7.3 禁止路径

1. 写作者不能直接写长期记忆。
2. 裁判不能绕过 `memory_commit` 直接入库。
3. 记忆整理员不能把未通过内容写成稳定事实。
4. 任何节点不能通过 RAG、联网搜索、主会话历史绕过 TaskGraph 授权资产。

## 8. 前端装配变化

当前前端模板 `writing_team_long_novel` 应从 11 节点扩展为分层图。

### 8.1 第一阶段最小改造

先只改章节生产链，减少破坏面：

```text

  -> writer_a_draft
  -> writer_b_review
  -> writer_a_revision
```

### 8.2 第二阶段完整改造

```text

character_designer_a + character_designer_b -> character_judge -> memory_commit_character
```

- `deviation_type = chapter_execution` -> `chapter_plan`

- `deviation_type = outline_structure` -> `outline_repair_a/b`

1. 前端图上能清晰看到 A 初稿、B 审读、A 修订、B 终稿候选、独立裁判、入库节点。

5. `memory_commit` 只处理裁判通过的 refs。
6. 返修时能根据裁判裁决回到正确节点，而不是总是回到主会话或自由对话层。
7. 旧 48 的旧命名、旧数据、旧链路不进入当前运行主数据。

## 11. 推荐执行顺序

我建议先执行第一阶段：只把章节生产从并行双稿改为隔离互审链。

理由：

1. 这是旧 48 最核心、收益最高的结构。
2. 当前系统已有工作记忆资源节点，刚好可以承接 A/B 之间的授权交接。
3. 对世界观、大纲、人物基础层的破坏面较小，不会一次性改垮整张图。
4. 可以最快验证《洪荒时代》写作团队是否能按任务图真实流转。

第一阶段跑通后，再升级基础资产层和偏差修正子图。

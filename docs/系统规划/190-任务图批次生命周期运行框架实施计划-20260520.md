# 190-任务图批次生命周期运行框架实施计划

日期：2026-05-20

## 1. 目标

把上一阶段已经进入发布前执行包的 `Split / Review / Repair / Commit / Merge` 批次生命周期，推进为运行时可消费的一等状态框架。

这次仍然不做领域私门：

- 章节只是 `unit_kind=chapter` 的一种实例。
- 文件、图片、记录、测试用例、资料源都必须走同一套机制。
- 运行时只消费 `contract_bindings.unit_batch` 与 `contract_bindings.runtime.*` 编译出的计划。

## 2. 当前结构判断

现有系统已经有三条基础链路：

- 编译链：`compile_task_graph_definition_runtime_spec()` 把 TaskGraphDefinition 编译为 RuntimeSpec。
- 调度链：`bootstrap_scheduler_state()` 根据节点状态计算 ready / blocked / completed。
- 运行链：`TaskRunLoop.start_task_graph_run()` 创建 TaskGraph 运行，`LangGraphCoordinationRuntime` 负责 stage accept / route / prepare / execute。

上一阶段新增的 split plan 已经进入：

- runtime spec diagnostics
- execution package
- frontend 发布包面板

缺口是：

- 批次生命周期仍然是静态计划。
- 运行时没有为每个 batch 维护 active / running / committed / failed 状态。
- stage prepare 没有自动把当前批次范围注入节点输入。
- stage accept 没有根据批次提交结果决定是否继续同一节点的下一批。
- run monitor 没有展示批次运行状态。

## 3. 设计原则

1. 批次生命周期是通用运行框架，不是写作逻辑。
2. split 决定范围，loop 决定返修，review 决定是否通过，commit 决定可见性，merge 只消费 committed batch。
3. 当前阶段先支持单运行线程顺序推进批次；并行批次先在状态中表达 ready 集合，不强行启动多个后台 worker。
4. 节点是否执行多少批，由契约编译结果决定，不由节点 Prompt 自己猜。
5. batch runtime state 必须能从 checkpoint 恢复，不能只存在内存。
6. 任何失败或返修都进入状态机与 diagnostics，不写领域专用小灶。

## 4. 本轮交付

### 4.1 后端模型与纯函数

新增：

```text
backend/orchestration/runtime_loop/task_graph_batch_runtime.py
```

职责：

- 从 RuntimeSpec diagnostics 的 split plans 初始化批次运行状态。
- 在 stage prepare 前选择当前 ready batch。
- 把 batch range 注入 `pending_inputs`。
- 在 stage accept 后根据 result accepted 与 repair policy 推进批次状态。
- 计算 merge readiness。

状态结构：

```json
{
  "authority": "task_system.batch_lifecycle_runtime_state",
  "graph_id": "...",
  "plan_states": [],
  "batch_states": [],
  "step_states": [],
  "merge_states": [],
  "active_batch_by_node": {},
  "ready_batch_ids": [],
  "running_batch_ids": [],
  "committed_batch_ids": [],
  "failed_batch_ids": []
}
```

### 4.2 运行时接入

修改：

```text
backend/orchestration/runtime_loop/langgraph_coordination_runtime.py
```

接入点：

- `_bootstrap_state()`：初始化 `batch_lifecycle_runtime_state`，写入 diagnostics。
- `_stage_prepare()`：如果当前 stage 对应 split plan，则选择当前 batch 并注入输入。
- `_stage_execute()`：dispatch context 增加 batch identity。
- `_stage_accept()`：如果当前 stage 是 batch stage，则按 batch transition 决定：
  - accepted：commit 当前 batch；若还有下一批，继续同一 stage；若全部提交，节点 completed。
  - rejected：按 max_repair_rounds 返修同一 batch；超过后 manual gate 或 failed。
- `_scheduler_node_sets()`：把 batch runtime state 写回 diagnostics，便于恢复和监控。

### 4.3 监控与前端

修改：

```text
backend/orchestration/runtime_loop/task_graph_run_monitor.py
frontend/src/components/workspace/views/task-system/taskGraphRuntimeView.ts
frontend/src/components/workspace/views/task-system/TaskGraphPublishRunPage.tsx
```

目标：

- run monitor 返回 `batch_lifecycle`。
- 前端运行页显示批次总数、ready、running、committed、failed、merge ready。
- 当前节点执行请求中能看到当前 batch 范围。

### 4.4 测试

新增或扩展：

```text
backend/tests/task_graph_batch_runtime_regression.py
frontend/src/components/workspace/views/task-system/taskGraphRuntimeView.test.ts
```

必须验证：

1. 启动 split graph 后，首个 stage request 带第一批范围。
2. 第一批 accepted 后，同一节点继续第二批，而不是直接完成节点。
3. 最后一批 accepted 后，节点 completed，merge readiness 为 ready。
4. 批次 rejected 后，在 max_repair_rounds 内重跑同一 batch。
5. 超过返修轮次后进入 manual gate 或 failed。
6. run monitor 能返回批次运行状态。

## 5. 本轮不做

- 不实现真正多 worker 并行执行。
- 不让 Agent 动态生成新 batch。
- 不写任何章节、小说、字数专用判断。
- 不把质量门硬编码到后端；验收仍来自事件 accepted 与契约策略。

## 6. 完成标准

完成后，一个配置了：

```json
{
  "unit_batch": {"unit_kind": "chapter", "requested_count": 50},
  "runtime": {
    "split_policy": {"mode": "static_batch", "batch_size": 10},
    "batch_acceptance_policy": {"mode": "review_then_commit", "max_repair_rounds": 3},
    "merge_policy": {"mode": "wait_all_committed"}
  }
}
```

的节点在运行时必须表现为：

```text
batch 1 -> execute/review/repair/commit
batch 2 -> execute/review/repair/commit
...
all committed -> merge ready -> downstream released
```

并且这些状态能在 checkpoint、diagnostics、monitor 和前端运行页中追踪。

- accepted 后推进下一批；全部 committed 后 merge ready。
- rejected 后按 `max_repair_rounds` 返修同一批；返修耗尽后节点失败，不继续后续 planned 批。
- run monitor 与 trace 摘要都能暴露批次生命周期。

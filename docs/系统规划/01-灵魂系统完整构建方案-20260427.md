# 洪荒时代灵魂系统完整构建方案

日期：2026-04-27  
项目名：洪荒时代（mythical ageent）  
定位：把现有 `backend/soul` 静态 prompt 层升级为可被 AgentProfile、任务系统、协作模式和记忆系统共同使用的标准灵魂契约层。

---

## 0. 结论

灵魂系统不应该只是“当前主 agent 的人格 prompt”。

它应该成为洪荒时代多 agent 架构中的“可复用智能体意志层”。

这里的“意志”不是玄学概念，而是工程概念：

```text
同一类能力，可以由不同意志承载。
同一个灵魂，可以承载不同任务。
同一个任务，可以根据目标选择不同灵魂。
```

例如：

```text
写作任务：
  四岳承载结构规划
  祝融承载初稿推进
  玄女承载审查校验
  河伯承载最终收束

重构任务：
  四岳承载架构拆分
  祝融承载实现推进
  玄女承载风险审查
  河伯承载结论汇总
```

因此，灵魂不是单独运行的 agent，也不是固定绑定某个 skill 的 prompt 包。

灵魂是可复用的智能体意志，而且五个灵魂都应该是动态的：

```text
SoulWill       决定一个智能体以什么风格、习惯、判断偏向来承载任务
SoulProfile    描述这个意志的机器可读契约
SoulRole       描述某个灵魂在当前任务中承担的临时职责
TaskSoulBinding 记录某个任务由哪个灵魂承载、承担什么职责、为什么选择它
```

完整架构中：

```text
SoulProfile    决定语言风格、工作习惯、思考偏向
AgentProfile   决定任务身份、职责范围、协作角色
SkillContract  决定能力包和工具边界
WorkerContract 决定真实执行能力
ControlKernel  决定当前任务采用哪个 agent / soul / skill / worker
```

一句话原则：

> 灵魂是可复用的智能体意志；它承载任务风格，但不拥有调度权、工具权、记忆写回权。

本系统第一阶段要先和任务系统联动：

```text
TaskContract / TaskRecord
  -> TaskSoulRequirement
  -> SoulSelectionCandidate
  -> ControlKernel 选择 SoulProfile
  -> AgentProfile + SkillContract 承载执行
```

## 2. 核心边界

### 2.1 Soul 是可复用的智能体意志

Soul 的本质是：

```text
一个可以被不同 AgentProfile 复用的任务承载意志。
```

它回答：

```text
面对这个任务时，以什么方式理解问题？
以什么节奏推进？
倾向先看结构、先找卡点、先查前提，还是先收束判断？
输出时如何组织语言？
```

它不回答：

```text
这轮是否执行？
调用什么工具？
写入什么记忆？
哪个 worker 真正干活？
最终答案是否提交？
```

### 2.2 Soul 不是 Agent

Soul 是风格层，不是执行体。

Soul 可以定义：

- 语言风格。
- 工作习惯。
- 默认问题处理方式。
- 容易偏向哪里，需要防什么。

Soul 不可以定义：

- 可调用哪些工具。
- 当前任务必须走哪个 route。
- 是否写入记忆。
- 是否接管主会话。
- 是否绕过 ControlKernel。

### 2.3 Agent 不是 Skill

AgentProfile 是任务身份。

例如：

- 通用 agent。
- 架构规划 agent。
- 写作 agent。
- 测试报告 agent。
- 记忆整理 agent。
- 执行推进 agent。
- 审查校验 agent。

它回答的是：

```text
这个执行角色负责什么？
它默认用什么灵魂？
它能使用哪些 skills？
它能接触哪些 memory？
它产出的 artifact 是什么？
```

### 2.4 Skill 不是 Soul

Skill 是能力包。

例如：

- RAG 检索。
- PDF 分析。
- 结构化数据分析。
- Web Search。
- 测试报告生成。
- 记忆整理。

Skill 可以提供任务能力和工具范围，但不定义人格。

### 2.5 Worker 不是 Agent

Worker 是执行单元。

例如：

- `retrieval_worker`
- `pdf_worker`
- `structured_data_worker`
- 后续 `test_report_worker`
- 后续 `memory_curator_worker`

Worker 只执行受控任务，不自行决定当前轮目标。

## 4. SoulProfile 契约

建议新增：

```text
backend/soul/contracts.py
backend/soul/registry.py
```

核心模型：

```python
@dataclass(slots=True, frozen=True)
class SoulProfile:
    soul_id: str
    name: str
    version: str
    will_type: str
    seed_path: str
    active: bool
    style_traits: list[str]
    work_habits: list[str]
    reasoning_bias: list[str]
    risk_bias: list[str]
    best_for: list[str]
    avoid_for: list[str]
    task_affinity: list[str]
    skill_affinity: list[str]
    prompt_sections: list[str]
    model_visible: bool = False
    decision_authority: bool = False
```

强约束：

```text
decision_authority 必须永远是 False。
SoulProfile 只作为 prompt/context material。
SoulProfile 不得携带 allowed_tools。
SoulProfile 不得携带 worker_route。
SoulProfile 不得携带 memory_write_policy。
```

### 4.1 SoulWill 契约

`SoulWill` 是 SoulProfile 的语义核心。

建议模型：

```python
@dataclass(slots=True, frozen=True)
class SoulWill:
    soul_id: str
    name: str
    will_type: Literal["collect", "organize", "execute", "inspect", "govern"]
    default_task_posture: str
    preferred_task_modes: list[str]
    collaboration_strengths: list[str]
    risk_to_watch: list[str]
```

四个初始职能 will_type：

| will_type | 灵魂 | 含义 |
| --- | --- | --- |
| `collect` | 河伯 | 默认适合信息收集、上下文召回、资料整理 |
| `organize` | 四岳 | 默认适合组织、结构、规划 |
| `execute` | 祝融 | 默认适合行动、推进、落地 |
| `inspect` | 玄女 | 默认适合审查、前提、风险 |

不设置默认主灵魂，改为当前激活灵魂与任务级 final_owner：

| will_type | 灵魂 | 含义 |
| --- | --- | --- |
| `dialogue/govern` | 当前激活灵魂或任务绑定的 `final_owner` | 负责对话、统摄当前轮、分派任务、统一最终口径 |

这些不是固定岗位，而是默认职责倾向。

五个灵魂都可以根据任务承载不同职责。

同一个灵魂可以生成多个投影卡片：

```text
SoulProfile
  -> SoulProjectionCard A: 对话投影
  -> SoulProjectionCard B: RAG 收集投影
  -> SoulProjectionCard C: 写作审查投影
```

投影卡片可以由用户手动生成和选用，也可以由任务系统根据 TaskMode / SkillScope / ToolScope 自动调用。

任何灵魂都不拥有更高执行权限；对话与统筹由当前激活灵魂或任务绑定决定。

五个默认灵魂中的对话统筹倾向灵魂：

```text
soul_id: goumang
中文名: 句芒
```

它负责组织四种职能意志，但不拥有工具授权、记忆写回权或执行权。

### 4.2 SoulRole 契约

`SoulRole` 是运行时职责，不是灵魂本体。

建议模型：

```python
@dataclass(slots=True, frozen=True)
class SoulRole:
    role_id: str
    name: str
    role_type: Literal["dialogue", "collect", "organize", "execute", "inspect", "govern"]
    output_expectation: str
    temporary: bool = True
```

一个 SoulProfile 可以在不同任务里承担不同 SoulRole：

```text
四岳 + dialogue   -> 结构化对话
四岳 + organize   -> 架构规划
四岳 + inspect    -> 结构一致性审查

祝融 + execute    -> 实现推进
祝融 + dialogue   -> 行动型对话
祝融 + organize   -> 以落地为中心的计划拆解
```

所以系统选择的不是：

```text
选一个固定岗位。
```

而是：

```text
选择 soul_id + role_type。
```

### 4.3 TaskSoulBinding 契约

任务系统需要记录“这个任务由哪个灵魂以什么职责承载”。

建议模型：

```python
@dataclass(slots=True, frozen=True)
class TaskSoulBinding:
    task_id: str
    soul_id: str
    role_type: str
    agent_id: str
    task_mode: str
    selection_reason: str
    selected_by: str = "control_kernel"
    explicit_user_selected: bool = False
    confidence: float = 0.0
```

它应该进入：

```text
TaskRecord.metadata["soul_binding"]
```

但注意：

```text
TaskSoulBinding 是记录，不是授权。
它不能替代 RuntimeDirective。
```

## 5. 四个基础灵魂定位

### 5.1 河伯

定位：

```text
信息收集、上下文召回、资料整理
```

适合：

- 上下文收集。
- 本地资料召回。
- RAG 初步检索。
- 记忆整理前的信息归并。
- 轻量架构讨论。

风险：

- 只收集不决断。
- 信息铺得太散。

默认 AgentProfile：

```text
general_agent
knowledge_agent
memory_curator_agent
```

### 5.2 四岳

定位：

```text
组织、结构、定框架
```

适合：

- 系统规划。
- 架构设计。
- 模块拆分。
- 长期路线。
- 写作大纲。

风险：

- 把简单问题说复杂。
- 停留在框架，不落动作。

默认 AgentProfile：

```text
architect_agent
writing_planner_agent
test_report_agent
```

### 5.3 祝融

定位：

```text
行动、执行、破局
```

适合：

- 代码实现。
- 修复阻塞。
- 执行推进。
- 快速产出初稿。
- 已明确目标后的落地。

风险：

- 过快。
- 压缩验证。
- 把复杂问题粗暴缩成小修。

默认 AgentProfile：

```text
executor_agent
draft_agent
implementation_agent
```

### 5.4 玄女

定位：

```text
前提、歧义、隐含条件
```

适合：

- 审查。
- 风险分析。
- 前提检查。
- 需求歧义拆解。
- 测试失败归因。

风险：

- 分析过细。
- 主结论变弱。

默认 AgentProfile：

```text
critic_agent
review_agent
test_diagnosis_agent
memory_conflict_agent
```

### 5.5 句芒

定位：

```text
对话、统筹、分派、归口
```

适合：

- 对话承载。
- 当前轮主线维护。
- 任务分派。
- 多 agent 输出汇总。
- 最终答案归口。
- fail-closed 边界说明。

说明：

```text
句芒是第五个动态灵魂，倾向承担对话与统筹，但不是默认主灵魂。
它不是固定凌驾于其他灵魂之上的权限层。
它可以参与普通任务，也可以把当前轮职责分派给其他灵魂。
```

风险：

- 变成中央独裁，压掉职能灵魂的专业判断。
- 绕过 ControlKernel 直接做授权。

默认 AgentProfile：

```text
main_agent
final_answer_agent
collaboration_coordinator_agent
```

## 6. 默认灵魂与自制灵魂

### 6.1 默认五灵魂

系统内置五个默认灵魂：

| soul_id | 中文名 | 默认倾向 |
| --- | --- | --- |
| `goumang` | 句芒 | 对话、统筹、归口 |
| `hebo` | 河伯 | 信息收集、上下文召回 |
| `siyue` | 四岳 | 组织、结构、规划 |
| `zhurong` | 祝融 | 行动、推进、落地 |
| `xuannv` | 玄女 | 审查、前提、风险 |

默认五灵魂是系统基座：

```text
它们保证系统一开始就具备稳定的对话、收集、组织、行动、审查能力。
```

但它们不是封闭集合。

### 6.2 自制灵魂扩展口

用户应该可以创建新灵魂。

自制灵魂定位：

```text
用户定义的 SoulProfile。
可被单线程对话选择。
可被多 agent 协作临时分配 SoulRole。
可绑定推荐 task_modes / role_types。
不天然拥有工具、记忆、执行或调度权限。
```

建议目录：

```text
backend/soul/custom/
  <soul_id>/
    SOUL.md
    profile.json
    portrait.png
```

也可以先沿用 markdown-only：

```text
backend/soul/custom/<soul_id>.md
```

建议自制灵魂契约：

```python
@dataclass(slots=True, frozen=True)
class CustomSoulProfile:
    soul_id: str
    name: str
    source: Literal["builtin", "user"]
    seed_path: str
    description: str
    preferred_role_types: list[str]
    preferred_task_modes: list[str]
    style_traits: list[str]
    work_habits: list[str]
    risk_to_watch: list[str]
    enabled: bool = True
    decision_authority: bool = False
```

强约束：

```text
自制灵魂不能声明 allowed_tools。
自制灵魂不能声明 worker_route。
自制灵魂不能声明 memory_write_policy。
自制灵魂不能覆盖 CORE.md。
自制灵魂不能绕过 ControlKernel。
```

### 6.3 自制灵魂参与任务的方式

单线程任务：

```text
用户选择某个自制灵魂
  -> 该灵魂承载完整对话
  -> 能力边界仍由 AgentProfile / Skill / ControlKernel 决定
```

多 agent 协作：

```text
ControlKernel 可以把自制灵魂分配到某个 SoulRole
  -> 例如 organize / inspect / execute / collect / dialogue
  -> 产出对应 artifact
  -> 任务绑定的 final_owner 或当前激活灵魂负责最终归口
```

示例：

```text
用户自制“文曲”
  preferred_role_types: ["organize", "dialogue"]
  preferred_task_modes: ["writing_outline", "final_answer"]

写作任务中：
  文曲 -> 写作结构
  祝融 -> 初稿推进
  玄女 -> 审稿
  final_owner -> 最终归口
```

### 6.4 自制灵魂注册流程

```text
1. 用户创建灵魂 seed。
2. SoulRegistry 读取并校验。
3. 生成 CustomSoulProfile。
4. 加入 SoulCatalog。
5. ControlKernel 可把它作为 SoulSelectionCandidate。
6. 只有被采纳后才形成 TaskSoulBinding。
```

失败策略：

```text
缺字段 -> disabled
违反权限边界 -> blocked
prompt 为空 -> disabled
名称冲突 -> blocked
```

## 9. 协作模式

### 8.1 单 Agent 模式

```text
Task -> one AgentProfile -> one Soul -> one Output
```

适合：

- 普通问答。
- 简单执行。
- 明确的小任务。

### 8.2 Planner / Executor 模式

```text
architect_agent(四岳 / organize)
  -> executor_agent(祝融 / execute)
  -> critic_agent(玄女 / inspect)
  -> final_owner(当前激活灵魂或任务绑定灵魂 / dialogue+govern)
```

适合：

- 重构。
- 多文件实现。
- 阶段性工程任务。

### 8.3 Writer / Reviewer 模式

```text
writing_planner_agent(四岳 / organize)
  -> draft_agent(祝融 / execute)
  -> critic_agent(玄女 / inspect)
  -> final_owner(当前激活灵魂或任务绑定灵魂 / dialogue+govern)
```

适合：

- 写作。
- 文档生成。
- 方案成文。

### 8.4 Memory Curation 模式

```text
memory_curator_agent(河伯)
  -> memory_conflict_agent(玄女)
  -> commit_gate
```

适合：

- 长期记忆整理。
- 去重。
- 过期记忆标记。
- 用户偏好归档。

### 8.5 Test Report 模式

```text
test_diagnosis_agent(玄女)
  -> test_report_agent(四岳)
  -> final_editor_agent(河伯)
```

适合：

- 测试失败解释。
- 回归报告。
- 风险清单。

## 10. Prompt 装配规则

现有静态链保留：

```text
ACTIVE_SEED.md
CORE.md
agent.md
```

新系统中应改为可组合 PromptManifest：

```text
Soul Section
  - selected soul seed

Core Section
  - all-soul safety / fact / output rules

AgentProfile Section
  - current task role
  - task mode
  - output contract

Skill Section
  - selected skill prompt view

Task Section
  - TaskContract
  - ContextPolicy
  - MemoryPolicy

Volatile Section
  - current user message
  - current evidence handles
  - execution node input
```

缓存原则：

```text
Soul seed       session-memoized
CORE            global/static candidate
agent.md        session-memoized
AgentProfile    session-memoized or per-task
Skill Section   per-task
Task Section    per-turn
Volatile        per-turn
```

## 11. Memory 边界

灵魂可以影响记忆的表达方式，但不能决定记忆写入。

建议 memory scope：

| scope | 含义 |
| --- | --- |
| `none` | 不读不写 |
| `session_read` | 只读当前会话 |
| `session_candidate` | 可提交 session 写回候选 |
| `durable_read` | 可读长期记忆 |
| `durable_candidate` | 可提交长期记忆写回候选 |
| `agent_private` | agent 局部工作记忆，不进入主线程 |

灵魂层限制：

```text
SoulProfile 不直接绑定 memory scope。
AgentProfile 可以声明 memory scope。
ControlKernel 根据 TaskContract 和 policy 决定是否采纳。
CommitGate 决定是否真正写入。
```

记忆整理 agent 需要特别约束：

```text
memory_curator_agent 只能产出 MemoryCommitCandidate。
memory_conflict_agent 只能产出 MemoryReviewArtifact。
任何 durable memory 写入必须经过 CommitGate。
```

## 12. Skill 边界

灵魂系统可以复用 skill，但不能把每个 soul 都做成 skill。

推荐关系：

```text
SoulProfile: 风格
AgentProfile: 任务角色
SkillContract: 能力包
ToolContract: 工具资源
WorkerContract: 执行器
```

可以有 “soul-aware skill rendering”，例如：

```text
同一个 writing skill：
- 四岳：更重结构和章节
- 祝融：更重快速初稿
- 玄女：更重审查和歧义
- 河伯：更重最终收束
```

但 skill 本身仍然不拥有调度权。

## 13. ControlKernel 决策流程

```text
UserRequest
  -> TaskContract
  -> TaskSoulRequirement
  -> Candidate Producers
     - intent candidate
     - soul preference candidate
     - agent profile candidate
     - skill candidate
     - memory candidate
  -> ControlKernel
     - classify task mode
     - select soul will
     - select AgentProfile
     - select SoulProfile
     - select SkillContract
     - build CollaborationPattern
     - build ExecutionGraph
  -> RuntimeDirective
  -> Execution Runtime
  -> Artifact / CommitCandidate
  -> CommitGate
```

选择规则：

```text
用户显式切换 soul > agent 默认 soul > task mode 推荐 soul > general_agent 默认河伯
```

但：

```text
显式 soul 只影响风格，不越过 AgentProfile 的工具/记忆边界。
```

### 12.1 TaskSoulRequirement

任务进入 ControlKernel 后，第一步生成任务意志需求：

```python
@dataclass(slots=True, frozen=True)
class TaskSoulRequirement:
    task_id: str
    task_mode: str
    desired_will_type: str
    required_posture: str
    reason: str
    explicit_soul_id: str = ""
```

示例映射：

| task_mode | desired_will_type | 推荐选择 |
| --- | --- | --- |
| `general_dialogue` | `dialogue` | 当前激活灵魂 |
| `system_design` | `organize` | 四岳 |
| `implementation` | `execute` | 祝融 |
| `risk_review` | `inspect` | 玄女 |
| `writing_outline` | `organize` | 四岳 |
| `writing_draft` | `execute` | 祝融 |
| `writing_review` | `inspect` | 玄女 |
| `final_answer` | `govern` | `final_owner` 或当前激活灵魂 |
| `memory_curation` | `collect` | 河伯 |
| `memory_conflict_review` | `inspect` | 玄女 |
| `test_report` | `organize` | 四岳 |
| `test_failure_diagnosis` | `inspect` | 玄女 |

### 12.2 SoulSelectionCandidate

灵魂选择本身也必须是候选：

```python
@dataclass(slots=True, frozen=True)
class SoulSelectionCandidate:
    candidate_id: str
    task_id: str
    soul_id: str
    will_type: str
    source: str
    reason: str
    confidence: float
    authority: str = "candidate_only"
```

来源可以是：

```text
explicit_user_selection
task_mode_default
agent_profile_default
collaboration_pattern
memory_preference_candidate
```

只有 ControlKernel 可以把它采纳为正式 `TaskSoulBinding`。

当前任务系统已有：

第一阶段不要重写 TaskCoordinator。

先新增任务意志投影：

```text

TaskRecord.metadata["soul_binding"]
```

建议最小结构：


### 13.1 任务先于灵魂选择

正确顺序：

```text
UserRequest
  -> TaskContract
  -> task_mode
  -> soul requirement
  -> soul candidate
  -> agent profile
  -> skill/tool/worker policy
```

错误顺序：

```text
先选灵魂，再让灵魂决定任务怎么做。
```

原因：

```text
灵魂是承载任务的意志，不是任务分类器本身。
```

### 13.2 灵魂可以承载不同 skills

同一个灵魂可以承载多个 skill：

```text
四岳 + writing skill        -> 写作结构
四岳 + architecture skill   -> 系统设计
四岳 + test report skill    -> 测试报告结构

祝融 + coding skill         -> 实现推进
祝融 + writing skill        -> 快速初稿
祝融 + refactor skill       -> 重构落地

玄女 + review skill         -> 风险审查
玄女 + test diagnosis skill -> 失败归因
玄女 + memory review skill  -> 记忆冲突检查

河伯 + final answer skill   -> 最终答复
河伯 + memory curation      -> 稳定归档
河伯 + dialogue skill       -> 通用对话
```

所以：

```text
SoulProfile 与 SkillContract 是多对多关系。
ControlKernel 按 TaskContract 选择组合。
```

### 13.3 灵魂可以被任务显式选择

用户可以说：

```text
用祝融推进这个实现。
用玄女帮我审一下风险。
用四岳给我搭框架。
用河伯帮我收束成最终结论。
```

这会生成：

```text
SoulSelectionCandidate(source="explicit_user_selection")
```

但仍需经过 ControlKernel 校验：

```text
用户显式选择灵魂可以优先；
但不能扩大工具权限；
不能扩大记忆权限；
不能跳过输出提交边界。
```

## 17. 第一批实现建议

先做三个最小闭环：

```text

   - 根据 TaskContract / task_mode 生成 SoulSelectionCandidate。
```

## 18. 当前定稿原则

```text
灵魂负责“以什么意志承载任务”。
AgentProfile 负责“以什么任务身份参与”。
Skill 负责“具备什么能力”。
Worker 负责“实际执行什么动作”。
ControlKernel 负责“当前轮谁被选中、怎么协作、是否执行”。
CommitGate 负责“哪些结果能进入会话、任务、记忆”。
```

这是洪荒时代多 agent 系统的底座。

## 1. 总结论

后续任务系统重构以 `215-任务环境系统设计书` 为总线。

本项目不应继续把 `TaskDomain` 当作核心控制抽象。`TaskDomain` 只能作为旧分类标签或迁移兼容输入；真正的系统控制面必须是：

```text
TaskEnvironment
-> SpecificTask
-> AgentFactory / RuntimeAssembly
-> ToolCapabilityTable
-> PermissionContext / ExecutionPermit
-> RuntimeStartPacket
-> AgentRuntime / GraphTaskRuntime
```

其中：

```text
TaskEnvironment 提供环境供给和系统上限。
SpecificTask 提出具体任务装配要求。
AgentFactory / RuntimeAssembly 在环境边界内合成有效 agent 装配。
ToolCapabilityTable 是本次工具可见性与可执行性的事实源。
PermissionSystem 是 per-call 授权、审批、审计权威。
Runtime 只消费 RuntimeStartPacket 并执行、记录、恢复、收束。
Agent 只完成任务，不选择环境，不扩大权限，不修改系统策略。
```

因此，后续“代码精简”不是独立目标，而是上述权威链收束后的结果。凡是不能落到这条总线上的 registry、metadata fallback、projection、runtime lane 推断、prompt 控制说明，都必须被删除、降级为迁移诊断，或移动到正确层。

## 2. 固定不变量

### 2.1 TaskEnvironment 不是 TaskDomain

当前 `TaskDomainRecord` 只有：

```text
domain_id
title
description
enabled
sort_order
metadata
```

它太薄，只能表达分类，不能表达环境供给、工具市场、文件管理、风险边界、memory 空间和 runtime 策略。

目标：

```text
TaskEnvironmentRecord
TaskEnvironmentSpec
```

成为环境真相源。停止扩展 `TaskDomainRecord.metadata`。后续控制字段不得继续藏在 `domain_id` 或 domain metadata 中。

### 2.2 TaskEnvironment 提供上限，SpecificTask 提出要求

`TaskEnvironment` 可以决定：

```text
prompt_space
skill_space
tool_space
file_management_space
resource_space
memory_space
execution_policy
risk_policy
artifact_policy
observability_policy
runtime_policy
lifecycle_policy
```

它不决定：

```text
本次具体使用哪个 agent
本次具体使用哪个 skill
本次具体走哪个 flow
本次具体产出哪个文档
本次 agent 应该怎么计划
```

这些由 `SpecificTaskAssemblyPolicy` 和运行装配处理。

### 2.3 Agent 不选择任务环境

选择任务环境是系统级动作。Agent 可以理解任务、规划执行、请求已授权工具，但不能：

```text
选择或切换 TaskEnvironment
扩大 tool_space
放宽 risk_policy
修改 AgentProfile
直接覆盖正式作品文件
把环境级 memory 写入 agent 私有 memory
```

### 2.4 Runtime 不重新发明任务目标和权限

Runtime 只消费装配结果：

```text
RuntimeStartPacket
ToolCapabilityTable
ExecutionPermit
TaskExecutionEnvelope
```

Runtime 不允许重新解释 `domain_id`、metadata、runtime lane 或 graph shape 来选择环境、扩大权限或改写任务目标。

### 2.5 写作环境必须由平台文件管理系统托管

`env.writing` 不能退化为普通目录读写。正式作品、草稿区、版本、提交、回滚、记忆库、记忆索引、可检索投影，都属于：

```text
TaskEnvironmentSpec.file_management_space
```

写作 agent 可以打开正式作品、读取 canon、写候选草稿、提交章节版本，但不能直接覆盖正式作品普通文件。

## 3. 目标权威链

### 3.1 任务发起链

`214` 的 TaskOrder 链仍然成立，但它必须挂到 TaskEnvironment 总线之下：

```text
ConversationTurn
-> TaskIntentDecision
-> TaskOrder / TaskOrderDraft
-> TaskOrderRun
-> ExecutionChannel
-> TaskExecutionEnvelope
```

职责：

```text
ConversationTurn: 普通交互事实。
TaskIntentDecision: 聊天、草稿、可执行任务的可审计判定。
TaskOrder: 已接受工作契约。
TaskOrderRun: 一次运行实例。
ExecutionChannel: 并行隔离执行通道。
TaskExecutionEnvelope: 本次任务对 agent/runtime 的有效覆盖。
```

`task_selection`、当前页面、Agent mode、历史任务标签，都只能是上下文投影，不能单独创建 `TaskOrderRun`。

### 3.2 环境装配链

`215` 是总线：

```text
User/System selects TaskEnvironment
-> User/System selects or triggers SpecificTask
-> TaskOrder created
-> TaskEnvironmentSpec resolved
-> SpecificTaskAssemblyPolicy resolved
-> AgentFactory / RuntimeAssembly composes effective assembly
-> ToolCapabilityTable compiled
-> RuntimeStartPacket built
-> Runtime executes and supervises
```

### 3.3 工具权限链

`218` 的工具权限链作为环境装配的子链：

```text
ToolCapabilityTable =
  ToolRegistry
  ∩ TaskEnvironment.tool_space
  ∩ SpecificTask.tool_capability_requirements
  ∩ AgentRuntimeProfile.allowed_operations
  ∩ User/Session permission state
  ∩ RuntimeAvailability
```

`PermissionSystem` 在每次工具调用处做最终授权、审批、审计和阻断。工具系统不决定任务环境，任务环境不直接执行工具。

### 3.4 Runtime 链

`222` 的双 runtime 边界继续保留：

```text
AgentRuntime: 一个 agent invocation 的 model/tool loop。
GraphTaskRuntime: 多个 agent invocation 的图调度、节点状态、handoff、merge、resume。
```

GraphTaskRuntime 可以创建 `NodeExecutionRequest` 并调用 AgentRuntime。AgentRuntime 不知道完整图调度，只接收本次 node invocation 所需的 refs、scope、contract 和 runtime packet。

## 4. 当前代码主要偏差

### 4.1 TaskDomain 仍在承担环境含义

当前多个对象仍使用 `domain_id`：

```text
SpecificTaskRecord.domain_id
TaskAssignment.domain_id
TaskGraphDefinition.domain_id
TaskDomainRecord
```

这些字段容易把环境、分类、业务域混在一起。目标是新增/强化 `environment_id`，并逐步让执行链只认 `environment_id`。

### 4.2 Runtime lane 被当作环境

runtime lane 目前有环境感字段，例如 default operations、memory scope、approval policy 等。按照 215，runtime lane 只能是环境允许的执行场景，不是环境本身。

目标：

```text
TaskEnvironment.runtime_policy.allowed_runtime_lanes[]
TaskEnvironment.runtime_policy.preferred_runtime_lanes[]
```

runtime lane 不再决定工具市场、文件管理环境或风险基线。

### 4.3 assembly_builder / assembly_support 混合多个权威

当前 assembly 既做：

```text
task intent contract
recipe selection
workflow resolution
operation policy merge
memory request profile derivation
safety envelope
bundle spec
graph selection
TaskExecutionAssembly
```

又直接读取 registry 和 current_turn_context。目标是拆成环境总线下的装配器：

```text
task_envelope_assembler
environment_resolution
specific_task_policy_resolver
operation_requirement_builder
tool_capability_table_adapter
runtime_start_packet_builder
memory_request_resolver
safety_envelope_builder
```

### 4.4 tool capability 已有雏形但环境上限未成为第一输入

`runtime/agent_runtime/environment/tool_capability_policy.py` 已经会读取 task environment，但下一步要确认：

```text
TaskEnvironment.tool_space
TaskEnvironment.file_management_space
TaskEnvironment.execution_policy
TaskEnvironment.risk_policy
```

是否真实参与 `visible_tools / dispatchable_tools` 计算。

### 4.5 projection 残留不属于任务环境总线

图任务投影机制已经取消。任务图节点执行应以：

```text
task_contract
work_prompt
RuntimeSpec node
TaskExecutionEnvelope
```

为输入。`projection_id`、`projection_overlay_id`、`TaskProjectionBinding`、`ProjectionSelectionResult` 不应回到 task graph runtime 主链。

## 5. 分阶段实施计划

### Phase 0：冻结旧域扩展

目标：停止 `TaskDomain` 继续扩张。

动作：

1. 搜索所有 `domain_id` 使用点，分类为：
   - UI 分类显示。
   - 旧数据迁移字段。

2. 禁止新增 `TaskDomainRecord.metadata` 控制字段。

```text

-> TaskAssignment.metadata.environment_id
```


## 8. 风险与约束

1. 不能为了迁移而迁移。每个改动必须说明它在 TaskEnvironment 总线中的位置。
2. 不允许长期保留 `domain_id` 和 `environment_id` 双轨执行。
3. 不允许把 `TaskEnvironment` 做成新的标签。
4. 不允许把 agent 配置塞进环境。
5. 不允许把具体任务的 skill/agent 装配要求写进环境。
6. 不允许让 runtime lane 替代任务环境。
7. 不允许用 prompt 描述代替环境资源和权限结构。
8. 不允许 agent 直接覆盖正式作品文件。
9. 不允许 GraphTaskRuntime 执行 agent model/tool 内环。
10. 不允许 AgentRuntime 选择环境或图调度模式。

## 9. 当前已完成事项如何归位

近期已完成的 `TaskFlowRegistry` 削薄、repository 拆分、overview builder、graph task registry service、projection 取消，归属于：

```text
Phase 0 / Phase 8 的旧主路径清理
```

它们是必要前置，但不是总线的终点。下一步必须转向 `TaskEnvironmentSpec -> SpecificTaskAssemblyPolicy -> RuntimeStartPacket` 的真实消费链，否则代码虽然变薄，任务系统仍可能没有稳定控制面。

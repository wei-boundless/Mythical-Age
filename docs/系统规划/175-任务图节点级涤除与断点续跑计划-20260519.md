# 任务图节点级涤除与断点续跑计划

## 背景

当前写作 run 已在 `baseline_memory_seed` 前形成可信资产，但 `volume_plan` 没有继承已审核大纲和基准库，导致后续 `chapter_outline`、`chapter_draft` 可能被坏计划污染。

不能用“整条任务重跑”解决。正确机制应允许在同一个 CoordinationRun 内：

1. 保留可信上游节点产物。
2. 将坏节点及其下游产物从有效 artifact root 隔离出去。
3. 清除 checkpoint 中坏节点及下游的 `stage_results`、状态、交接引用。
4. 从指定节点重新生成 `NodeExecutionRequest`，继续原 run。

## 目标语义

新增通用恢复动作：`rewind_from_stage`。

输入：
- `coordination_run_id`
- `stage_id`
- `reason`
- `artifact_root`
- `include_downstream`
- `move_artifacts`
- `refresh_graph_spec`

输出：
- 被失效节点列表
- 被移动 artifact 列表
- 保留节点列表
- 新 checkpoint ref
- 重新派发的 stage execution request

## 规则

1. 失效范围默认为目标节点及所有下游节点。
2. 下游通过图拓扑边计算，不能硬编码写作节点。
3. artifact 只移动 checkpoint 中记录过的 `artifact:` refs；不扫全目录误删。
4. 移动目标为 `invalidated/<timestamp>-<stage>-<reason>/...`，保留证据。
5. checkpoint 中移除失效节点的：
   - `stage_results`
   - `stage_results_by_instance`
   - `timeline_result_records`
   - `result_record_index`
   - `latest_stage_result_records`
   - `accepted_result_records_by_scope`
   - `contract_status`
   - `current_event`
   - `stage_execution_request`
6. `node_statuses` 中目标节点回到 `running`，下游回到 `pending`，上游保持 `completed`。
7. 若已重新配置任务图，可选择刷新图规格和节点契约，使重跑节点使用最新 prompt / artifact context / edge contract。

## 补充：执行活性与有界恢复

状态：实施中

本次 `volume_plan` 重跑暴露出另一个通用问题：流式模型调用已经产生部分输出，但发生 `provider_unavailable` 后进入非流式恢复；如果非流式恢复长期不返回，节点会停在 `running`，既没有有效 artifact，也没有失败信号，导致断点续跑无法自然接管。

因此断点续跑不仅要管理“坏产物”，还要管理“坏执行态”。

新增规则：

1. 流式错误后的非流式恢复必须有明确超时。
2. 超时后输出 `stream_recovery: failed` 和标准 `error` 事件，RuntimeLoop 将节点终止为 `failed`。
3. 失败节点可以继续使用 `rewind_from_stage` 涤除目标节点及下游，并从可信上游重新派发。
4. 写作任务配置的 `stream_policy` 必须显式声明恢复超时，避免长篇节点无限等待。
5. 恢复超时不产出业务文件，不允许把 partial stream 写成有效 artifact。

验证标准：

- 模型流式失败后，非流式恢复超时会产生失败事件，而不是永久 running。
- `volume_plan` 当前坏执行态可以从 `volume_plan` 断点重新派发。
- 重跑继续使用已提交的 `baseline_memory_seed`、`outline_design`、`outline_review` 和 `memory_commit_world`，不重跑可信上游。

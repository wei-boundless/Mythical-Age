# 107-长篇小说双创作者隔离互审流程设计

## 0. 文档目标

本文件只定义长篇小说团队任务的**业务流程模型**，不承载具体字段、注册项、文件路径和存储落地细节。

它回答的问题是：

**我们希望长篇小说写作团队到底如何运作，信息如何流动，谁有权通过，谁不能污染谁。**

配置与实现另见：

- `docs/系统规划/103-长篇小说写作团队配置闭环修复实施计划-20260515.md`

## 1. 流程总原则

长篇小说任务不是“多个 agent 一起聊天写作”，而是：

**双创作者隔离创作 + 每个产出即时受审 + 公开提案互相施压 + 审核员持续裁决 + 达标后才能进入下一阶段。**

必须同时满足以下原则：

1. 创作者相互竞争，也相互借鉴，但不能相互污染
2. 审核员负责裁决，不负责代写
3. 每一个创作者产出都必须被审核员读取、记录和评价
4. 每一阶段都必须经过“提案 - 审核 - 返工或通过”的闭环
5. 只有通过审核的内容才算正式资产
6. 历史草稿可以归档，但不能自动变成后续阶段的真实依据

## 2. 流程基本单元

长篇小说团队流程不是按旧结构名称理解，而是按以下业务单元理解。

### 2.1 阶段

阶段是一个可独立通关的创作目标，例如：

1. 世界观阶段
2. 大纲阶段
3. 人物阶段
4. 单章写作阶段
5. 基础资产修复阶段

阶段有明确输入、明确通过标准和明确正式资产出口。

阶段未通过前，所有产物都只是候选版本。

### 2.2 候选版本

候选版本是创作者 A 或 B 在某一轮提交的作品。

候选版本必须满足：

1. 有明确作者
2. 有明确阶段
3. 有明确轮次
4. 有明确所依据的输入包
5. 有明确输出目标

候选版本不是正式资产。

候选版本必须先经过审核员审阅，才能决定是否进入下一步。

### 2.3 单次审阅

单次审阅是审核员对某一个候选版本的即时评价。

单次审阅必须发生在每个候选版本之后。

单次审阅的目的不是立刻结束阶段，而是：

1. 判断该候选版本质量
2. 记录它相对上一版本的变化
3. 指出它解决了什么问题
4. 指出它新增了什么问题
5. 给下一位创作者形成明确输入

### 2.4 通关裁决

通关裁决是审核员基于阶段内累计审阅记忆作出的阶段级决定。

通关裁决不同于单次审阅。

单次审阅回答：

**这个候选版本怎么样。**

通关裁决回答：

**这个阶段是否可以结束，是否允许沉淀正式资产。**

### 2.5 轮次

一个完整轮次至少包含：

1. A 的候选版本
2. 审核员对 A 的单次审阅
3. B 的候选版本
4. 审核员对 B 的单次审阅
5. 审核员的阶段通关判断

如果没有通过，则进入下一轮。

下一轮的起点由审核员指定，可以从 A 开始，也可以从 B 开始，但必须继续遵守“候选版本之后立刻审阅”的规则。

### 2.6 交接包

交接包是一个角色交给下一个角色的受控输入。

交接包必须只包含下一步需要知道的内容，不能把整个运行历史原样塞给下一个角色。

交接包至少回答：

1. 当前处于哪个阶段
2. 当前是第几轮
3. 上一步产出了什么
4. 哪些内容允许被下一步读取
5. 下一步必须完成什么
6. 哪些内容禁止作为依据

交接包是流程里的信息流动单位。

创作者读交接包，不读混杂会话。

审核员读候选版本和评审记忆，不读创作者私有上下文。

### 2.7 记忆资产

记忆资产是流程允许跨步骤、跨轮次、跨阶段保留的结构化信息。

记忆资产分为四类：

1. 正式 canon 记忆
2. 阶段评审记忆
3. 候选版本归档
4. 纠错诊断记录

正式 canon 记忆是后续阶段可以直接依赖的事实。

阶段评审记忆是审核员用于连续比较和裁决的依据。

候选版本归档只用于回放和诊断，默认不是下一轮创作者可见内容。

纠错诊断记录用于说明为什么触发修复、人工接管或终止。

## 3. 角色模型

### 3.1 双创作者

每个创作阶段都由两个创作者共同打磨：

1. 创作者 A
2. 创作者 B

他们的职责是：

1. 基于当前阶段输入包各自提交完整提案
2. 在后续轮次中，结合对方上一轮公开提案和审核员要求，继续产出新版本
3. 对当前阶段的质量负责，但不负责裁决自己是否通过

双创作者的关系不是彼此拉扯方向，也不是各写各的平行作品，而是：

**围绕同一个目标作品，从不同关注角度持续互相打磨。**

也就是说：

1. 他们共享同一个项目目标
2. 他们共享同一个阶段任务
3. 他们共享同一个审核标准
4. 他们的差异只体现在优先审视角度和创作偏置，而不体现在目标分裂

因此，这里的“竞争”更准确地说是一种良性精进机制：

1. A 提出当前轮次的完整方案
2. B 不只是另起炉灶，而是要识别 A 的不足并提出更强版本
3. 下一轮 A 再根据 B 的公开方案和审核要求继续提升
4. 两人通过多轮公开提案，把同一个作品不断推向更高完成度

最终目标不是保留两条长期分叉路线，而是：

**让作品在双向打磨中收敛，而不是在双向拉扯中分裂。**

### 3.2 审核员

每个关键阶段都必须有独立审核员。

审核员职责包括：

1. 阅读每一个创作者产出的候选版本
2. 判断当前候选是否比上一版本改进、停滞或倒退
3. 指出问题与证据
4. 下发下一轮修改要求或放行下一阶段

审核员**不负责**：

1. 替创作者直接重写
2. 和创作者共享模糊上下文共同共创
3. 绕过流程直接帮某一方案过关

### 3.3 记忆管家

记忆管家负责把已经通过审核的资产写入正式记忆，并把未通过的候选版本归档。

记忆管家不参与创作，也不参与质量裁决。

它只执行三件事：

1. 按审核员裁决写入正式 canon
2. 归档候选版本、审阅记录和纠错诊断
3. 为下一阶段准备受控读取包

记忆管家不能把未通过候选伪装成正式事实。

### 3.4 交付整编者

交付整编者只在全流程末尾出现。

它负责把已通过的正式资产和章节成稿整理为最终交付物。

交付整编者不能重新裁决质量，也不能擅自修改 canon。

如果整编时发现前序资产缺失、冲突或不可交付，只能触发纠错流程，不能直接补写绕过审核。

## 4. 信息隔离原则

这是整个流程最核心的结构约束。

需要特别强调的是，信息隔离的目的不是人为制造对抗，而是防止低质量污染和无序串味，好让双创作者能够在受控边界内有效互相打磨同一作品。

### 4.1 创作者不保留污染性上下文记忆

创作者 A / B 不能像普通聊天 agent 那样累积会话历史。

每一轮都应视为一次新的受控执行，只读取当前输入包。

这样做的目的：

1. 避免两位创作者逐步同质化
2. 避免上一轮随机表达污染下一轮判断
3. 保证每轮产出都可回溯到明确输入

### 4.2 创作者允许读取的内容

创作者每轮只能读取以下内容：

1. 项目基础资料
2. 当前已批准的 canon 资产
3. 当前阶段任务说明
4. 对方上一轮公开提案
5. 审核员上一轮裁决与明确修改要求

创作者不能读取：

1. 对方私有思考过程
2. 自己过去轮次的完整自由上下文
3. 与当前阶段无关的运行噪声
4. 未批准、未公开的隐藏候选版本

### 4.3 审核员的可见范围

审核员可以读取：

1. A 提案
2. B 提案
3. 当前已批准 canon
4. 当前阶段目标
5. 必要的上一轮裁决历史

审核员不应该背着流程引入额外隐藏输入，否则裁决就不可证明。

### 4.4 审核员需要保留评审记忆

与双创作者不同，审核员必须具备受控的阶段性记忆。

原因很直接：

1. 审核员需要比较不同轮次之间是否真的改进
2. 审核员需要识别创作者是否回避了上一轮问题
3. 审核员需要判断当前版本是进步、停滞还是倒退
4. 审核员需要保证裁决标准在连续轮次中保持稳定

因此，审核员不是每轮完全重置的无记忆执行者，而是一个保留**评审连续性**的角色。

但这种记忆也必须是受控的，不能退化成无限制的自由上下文堆积。

审核员允许保留的记忆应当主要包括：

1. 上一轮和当前轮的 A/B 提案摘要
2. 历次裁决结果
3. 历次明确问题清单
4. 历次修改要求
5. 当前阶段累计尝试次数
6. 已确认解决和未解决的问题状态

审核员不应依赖的记忆包括：

1. 与当前阶段无关的冗长闲聊历史
2. 不可追溯的模糊印象式判断
3. 未结构化归档的噪声运行细节

也就是说，审核员保留的不是“聊天记忆”，而是：

**围绕阶段裁决连续性的结构化评审记忆。**

## 5. 标准阶段闭环

每个核心阶段都遵循同一业务闭环：

1. 输入包准备
2. 创作者 A 提交候选版本
3. 审核员立即审阅 A 的候选版本
4. 创作者 B 读取 A 的公开候选和审核员审阅意见，提交候选版本
5. 审核员立即审阅 B 的候选版本
6. 审核员基于累计评审记忆进行阶段通关判断
7. 若未通过，审核员给出下一轮明确要求
8. 下一轮创作者根据公开候选和审核要求继续产出新版本
9. 若通过，形成正式资产并进入下一阶段

这意味着流程不是单次链路，而是可重复的质量门禁循环。

这个循环的本质不是让 A 和 B 不断争夺主导权，而是让他们在审核门禁约束下，从两个不同角度不断压缩缺陷、提升质量、补齐短板，直到当前阶段作品足够稳定。

这里的审核员不是只在 A/B 都交付以后才第一次介入。

正确结构是：

1. 每个候选产出都触发一次审核员审阅
2. 审核员的评审记忆持续累积，形成该阶段的质量台账
3. 阶段通关裁决建立在所有已审候选、历史问题状态和当前通过标准之上
4. 通关裁决是持续审阅的结果，不是一次性汇总比较

## 5.1 首轮与后续轮次

首轮允许 A 和 B 都从同一阶段输入包出发，但流程上仍然必须按候选版本逐个审阅。

也就是说，首轮可以在创作动作上并发准备，但在流程裁决上不能跳过单次审阅：

1. A 首轮候选进入审阅
2. 审核员形成 A 首轮审阅
3. B 首轮候选进入审阅
4. 审核员形成 B 首轮审阅
5. 审核员判断是否需要下一轮

从第二轮开始，每一位创作者都必须读取：

1. 对方上一轮公开候选
2. 审核员对对方候选的审阅
3. 审核员给本轮下发的修改要求

这样两位创作者不是盲目并行，而是在同一目标下交替吸收对方暴露出的优点和缺点。

## 5.2 返工循环

未通过时，流程不能简单回到“重新写一遍”。

审核员必须明确指定下一轮的返工任务：

1. 保留什么
2. 删除什么
3. 修正什么
4. 补充什么
5. 哪些问题不得再次回避

下一轮创作者只根据受控输入包工作。

创作者不能用自由会话记忆猜测审核员意图，也不能把未公开的历史草稿作为依据。

## 5.3 通关条件

一个阶段可以通关，必须同时满足：

1. 审核员已经审阅本阶段至少一组 A/B 候选版本
2. 当前候选版本已经覆盖阶段必需内容
3. 审核员记录的阻塞问题已经关闭
4. 当前版本没有引入新的根本性矛盾
5. 正式资产可以被后续阶段直接引用

如果只是“某一版看起来不错”，但评审记忆中仍有未关闭的阻塞问题，不能通关。

## 5.4 双创作者差异化原则

双创作者必须有不同提示词和不同关注重点，否则双人机制会退化成两份近似答案的重复生产。

但这种差异化必须是互补式的，而不是分裂式的。

允许的差异：

1. 关注重点不同
2. 结构偏好不同
3. 风险敏感度不同
4. 对可读性、张力、稳定性的优先排序不同

不允许的差异：

1. 项目目标不同
2. 阶段任务不同
3. 审核标准不同
4. 私自引入另一套世界观或叙事方向

因此，双创作者提示词设计原则应当是：

1. 同任务
2. 同目标
3. 同通过标准
4. 不同创作偏置
5. 不同审视角度

## 5.5 双创作者竞争的正确含义

这里的竞争，不是为了让两个 agent 各自赢得路线之争，而是为了建立一种持续的质量压力。

正确形态是：

1. 任何一方都要努力做出比对方上一轮更完整、更稳、更有说服力的版本
2. 每一轮都要吸收对方暴露出的优点，同时修正对方暴露出的缺点
3. 双方都在推动同一作品变得更成熟，而不是推动两个方向彼此冲突

所以这套机制更接近：

**互相校正、互相补强、互相施压的双人打磨机制。**

## 6. 审核裁决模型

审核员输出必须是有限裁决，而不是松散评语。

审核员每次看到单个候选版本时，都要输出一次结构化审阅结果。该结果不一定代表阶段最终通关，但必须更新评审记忆。

单候选审阅至少包含：

1. 当前候选来自 A 还是 B
2. 当前候选解决了哪些历史问题
3. 当前候选新增了哪些风险
4. 与上一可见候选相比是改进、持平还是倒退
5. 下一位创作者必须重点处理的问题

阶段通关裁决则只能由审核员基于累计评审记忆给出。

建议统一裁决集合：

1. `pass`
2. `revise_current_stage`
3. `replan_current_stage`
4. `repair_world`
5. `repair_outline`
6. `repair_character`
7. `human_review_required`
8. `fail_closed`

### 6.1 `pass`

含义：

- 当前阶段已达标
- 允许沉淀正式资产
- 可以进入下一阶段

### 6.2 `revise_current_stage`

含义：

- 当前阶段方向基本正确
- 但质量、完整性或一致性仍不足
- 需要双创作者继续迭代新版本

在给出该裁决时，审核员必须能够指出：

1. 相比上一轮有哪些部分已改善
2. 哪些关键问题仍未解决
3. 当前版本是否出现新的退步

### 6.3 `replan_current_stage`

含义：

- 当前阶段的局部修改已经不够
- 需要先回到该阶段更上游的规划动作

### 6.4 `repair_world / repair_outline / repair_character`

含义：

- 问题不只在当前文本层，而是根资产发生偏差
- 必须回到相应基础资产层做专项修复

### 6.5 `human_review_required`

含义：

- 自动团队无法稳定完成判断
- 进入人工审核或人工接管流程

### 6.6 `fail_closed`

含义：

- 当前轮次或当前运行应终止
- 保留诊断与证据，不再自动推进

## 7. 资产分层原则

流程必须明确区分不同资产层级。

### 7.1 候选提案

创作者 A/B 每轮输出的是候选提案，不是正式设定。

它们可以被比较、被批评、被否决、被归档，但不能直接成为后续事实。

### 7.2 审核裁决

审核裁决是门禁结果，不是创作正文。

它的作用是：

1. 决定是否通过
2. 指出缺陷
3. 约束下一轮输入

### 7.3 正式 canon

只有 `pass` 之后，当前阶段资产才进入正式 canon。

canon 是后续所有阶段的真实依据，必须稳定、版本化、可回放。

### 7.4 历史归档

旧版本、被拒稿、旧裁决、失败轮次都应该归档保留。

但归档默认不是模型下一轮可见信息，除非明确要求回放诊断。

## 8. 端到端完整流程

本节定义长篇小说任务从开始到结束的完整业务流程。

它不是配置步骤，而是团队运行规程。

### 8.1 任务启动

任务启动时，系统必须先形成项目启动包。

项目启动包包含：

1. 用户原始目标
2. 作品类型、篇幅、风格和禁区
3. 当前已有资料
4. 期望交付物
5. 本次运行允许自动完成的范围
6. 需要人工确认的边界

启动包只负责定义目标和边界。

启动包不能直接成为世界观、大纲、人物或正文 canon。

任务启动后，流程进入前置资产阶段。

前置资产阶段顺序为：

1. 世界观
2. 大纲
3. 人物
4. 章节规划
5. 正文章节循环
6. 全书整编
7. 最终交付

任何后续阶段发现基础资产存在根本问题，都可以触发专项修复流程。

### 8.2 阶段输入包准备

每个阶段开始前，必须准备阶段输入包。

阶段输入包只允许包含：

1. 项目启动包中仍然有效的目标
2. 已通过审核的上游 canon
3. 当前阶段任务说明
4. 当前阶段通过标准
5. 上一轮审核员明确下发的修改要求
6. 允许公开给创作者的候选摘要

阶段输入包不能包含：

1. 未通过候选的完整隐藏历史
2. 创作者私有推理过程
3. 模糊会话记忆
4. 与当前阶段无关的运行细节

阶段输入包形成后，才能交给创作者。

如果阶段输入包缺少必要上游 canon，则不能开始创作，必须进入纠错流程。

### 8.3 创作者 A 产出

创作者 A 读取阶段输入包后，提交当前阶段候选版本。

A 的候选版本必须包含：

1. 当前版本正文或方案
2. 对阶段目标的覆盖说明
3. 自认为解决的关键问题
4. 不确定点和风险点
5. 给审核员检查的重点

A 不能声明自己通过。

A 不能把候选版本写入正式 canon。

A 完成后，候选版本进入审核员单次审阅。

### 8.4 审核员审阅 A

审核员读取 A 候选版本、阶段输入包和已有评审记忆。

审核员对 A 的单次审阅必须产出：

1. A 候选的质量判断
2. 已解决问题
3. 未解决问题
4. 新增风险
5. 与上一可见版本的变化
6. B 下一步必须重点处理的要求
7. 是否允许继续交给 B 打磨

审核员审阅 A 后，必须写入阶段评审记忆。

如果 A 的候选严重偏离目标，审核员可以直接触发纠错流程，而不是继续让 B 在错误基础上打磨。

### 8.5 创作者 B 产出

创作者 B 读取受控交接包。

B 的交接包包含：

1. 阶段输入包
2. A 的公开候选版本
3. 审核员对 A 的审阅
4. 审核员给 B 的明确要求

B 的任务不是另开一个方向，而是在同一作品目标下：

1. 吸收 A 的有效部分
2. 修正 A 暴露的问题
3. 补齐 A 缺失的结构
4. 保持阶段目标不分裂
5. 提交更强候选版本

B 完成后，候选版本进入审核员单次审阅。

### 8.6 审核员审阅 B

审核员读取 B 候选版本、A 候选版本、对 A 的审阅记录和阶段评审记忆。

审核员对 B 的单次审阅必须产出：

1. B 候选的质量判断
2. B 是否解决了 A 暴露的问题
3. B 是否引入新的偏差
4. 当前阶段的剩余阻塞问题
5. 是否已经具备通关裁决条件
6. 如果未通关，下一轮 A 或 B 应优先处理什么

审核员审阅 B 后，必须更新阶段评审记忆。

### 8.7 阶段通关判断

完成至少一组 A/B 候选和两次单次审阅后，审核员进行阶段通关判断。

通关判断只有三类结果：

1. 通过
2. 未通过但可继续迭代
3. 不适合继续在当前阶段迭代

通过时：

1. 审核员指定可沉淀为正式资产的版本或合并结果
2. 记忆管家写入正式 canon
3. 当前阶段关闭
4. 下一阶段读取新的 canon 开始

未通过但可继续迭代时：

1. 审核员生成下一轮返工要求
2. 当前阶段不写入正式 canon
3. 候选版本归档
4. 进入下一轮 A/B 打磨

不适合继续在当前阶段迭代时：

1. 审核员说明阻塞原因
2. 触发专项纠错
3. 当前阶段暂停
4. 修复完成后再恢复阶段

### 8.8 记忆读出流程

记忆读出必须遵守“按角色、按阶段、按用途读取”的原则。

创作者读出：

1. 项目目标
2. 已通过 canon
3. 当前阶段输入包
4. 对方公开候选
5. 审核员明确要求

审核员读出：

1. 当前候选版本
2. 对照候选版本
3. 当前阶段评审记忆
4. 已通过 canon
5. 当前阶段通过标准

记忆管家读出：

1. 审核员裁决
2. 被允许沉淀的正式资产
3. 需要归档的候选和审阅记录

交付整编者读出：

1. 全部已通过 canon
2. 已通过章节成稿
3. 最终交付要求
4. 交付检查记录

任何角色都不能因为“历史里出现过某内容”就把它当成当前事实。

当前事实只能来自已通过 canon。

### 8.9 记忆写入流程

记忆写入分为四条路径。

第一条是候选归档：

1. 每个创作者候选版本都必须归档
2. 归档不等于通过
3. 归档候选默认不进入下一轮创作者输入

第二条是评审记忆写入：

1. 每次单次审阅都必须写入阶段评审记忆
2. 每次通关判断都必须写入阶段评审记忆
3. 已关闭问题和未关闭问题必须可区分

第三条是正式 canon 写入：

1. 只有审核员给出通过裁决后才能写入
2. 写入内容必须明确来源
3. 写入后成为后续阶段的事实依据

第四条是纠错诊断写入：

1. 流程阻塞必须记录原因
2. 触发修复必须记录触发点
3. 人工接管必须记录自动流程停止位置
4. 失败关闭必须保留可回放证据

### 8.10 循环推进

长篇小说任务有两层循环。

第一层是阶段内循环：

1. A 候选
2. 审核 A
3. B 候选
4. 审核 B
5. 通关判断
6. 未通过则进入下一轮

第二层是章节循环：

1. 章节规划
2. 作者 A 正文候选
3. 审核员审阅 A
4. 作者 B 正文候选或修订候选
5. 审核员审阅 B
6. 章节通关判断
7. 章节写入正式章节资产
8. 进入下一章

章节循环中，如果审核员发现问题来自上游资产，则不能在章节层硬补。

此时必须触发世界观、大纲或人物专项修复。

专项修复通过后，章节循环再继续。

### 8.11 纠错流程

纠错不是失败后的随意补丁，而是正式流程分支。

触发纠错的情况包括：

1. 阶段输入包缺少必要 canon
2. 创作者候选偏离项目目标
3. 两位创作者连续多轮未解决同一阻塞问题
4. 审核员发现世界观、大纲、人物或章节之间存在根本矛盾
5. 当前阶段通过标准本身不清楚
6. 自动审核无法稳定判断
7. 交付整编时发现正式资产缺失或冲突

纠错流程分为五步：

1. 暂停当前推进
2. 审核员或交付整编者写入纠错诊断
3. 判断问题归属层级
4. 进入对应修复阶段或人工接管
5. 修复通过后恢复原流程

问题归属层级包括：

1. 项目目标问题
2. 世界观问题
3. 大纲问题
4. 人物问题
5. 章节问题
6. 交付整编问题
7. 自动流程能力不足

如果问题归属无法判断，流程必须进入人工审核，而不是继续自动推进。

### 8.12 专项修复流程

专项修复是一个缩小范围的创作阶段。

它仍然遵守双创作者隔离互审流程：

1. 准备修复输入包
2. A 提交修复候选
3. 审核员审阅 A
4. B 提交修复候选
5. 审核员审阅 B
6. 审核员判断修复是否通过
7. 通过后写入对应 canon
8. 恢复被暂停的原阶段

专项修复只能修复被诊断出的资产层问题。

它不能顺手重写无关内容。

### 8.13 阶段交接

阶段通过后，必须形成阶段交接包。

阶段交接包包含：

1. 本阶段正式 canon
2. 本阶段通过裁决
3. 后续阶段必须遵守的约束
4. 本阶段仍需注意但不阻塞的问题
5. 明确不应继续引用的废弃候选

下一阶段只能读取阶段交接包和正式 canon。

下一阶段不能把上一阶段未通过候选当作事实继续扩写。

### 8.14 全书交付流程

所有章节通过后，进入全书交付流程。

全书交付前必须完成：

1. 世界观 canon 已通过
2. 大纲 canon 已通过
3. 人物 canon 已通过
4. 所有章节均已通过
5. 章节顺序完整
6. 不存在未关闭阻塞问题
7. 没有等待修复的专项问题

交付整编者读取正式资产和已通过章节，形成最终交付物。

最终交付物必须包含：

1. 正文章节
2. 必要的目录或章节序列
3. 交付说明
4. 已知限制或遗留非阻塞问题
5. 资产来源说明

交付整编者完成后，审核员进行最终交付检查。

最终交付检查只判断：

1. 是否漏章
2. 是否违反已通过 canon
3. 是否存在明显顺序错误
4. 是否存在未关闭阻塞问题
5. 是否满足用户原始交付目标

最终交付检查通过后，任务完成。

如果最终交付检查未通过，则回到对应问题层级纠错，而不是在交付层直接改写。

## 9. 闭环运作细节关键

完整流程能不能真正运行，取决于以下细节是否被严格执行。

这些细节不是额外装饰，而是闭环成立的条件。

### 9.1 每个阶段必须有清晰状态

阶段不能只有“正在做”和“完成”两种粗状态。

阶段至少需要区分：

1. `not_started`
2. `input_ready`
3. `creator_a_running`
4. `creator_a_submitted`
5. `reviewing_a`
6. `creator_b_running`
7. `creator_b_submitted`
8. `reviewing_b`
9. `gate_deciding`
10. `revision_required`
11. `repair_required`
12. `passed`
13. `paused_for_human`
14. `failed_closed`

没有这些状态，系统就无法判断：

1. 当前该谁行动
2. 谁的产出已经可见
3. 审核员是否已经读过该候选
4. 是否允许进入下一轮
5. 是否允许写入 canon

### 9.2 每个动作都必须有明确前置条件

流程中的每个动作都必须先检查前置条件。

A 可以开始创作的条件：

1. 阶段输入包已经形成
2. 当前阶段未通过
3. 当前没有等待人工接管
4. A 本轮尚未提交候选

审核员可以审阅 A 的条件：

1. A 已提交候选
2. A 候选未被当前轮审阅过
3. 阶段评审记忆可读取

B 可以开始创作的条件：

1. A 候选已公开给 B
2. 审核员已完成 A 的单次审阅
3. B 的交接包已经形成
4. B 本轮尚未提交候选

审核员可以做通关判断的条件：

1. A 候选已审阅
2. B 候选已审阅
3. 阶段评审记忆已更新
4. 当前轮没有未处理的阻塞异常

记忆管家可以写入 canon 的条件：

1. 审核员给出 `pass`
2. 可写入内容已经被明确指定
3. 候选来源和裁决来源可追溯

### 9.3 每个产出都必须有最小结构

创作者候选不能只是自然语言正文。

候选版本至少需要包含：

1. 作者身份
2. 阶段
3. 轮次
4. 候选版本编号
5. 依据的输入包编号
6. 候选正文或方案
7. 覆盖的阶段目标
8. 自检风险
9. 可公开摘要
10. 禁止自动写入 canon 的标记

审核员单次审阅至少需要包含：

1. 被审候选编号
2. 审阅轮次
3. 质量判断
4. 已解决问题
5. 未解决问题
6. 新增问题
7. 相对上一可见候选的变化
8. 给下一位创作者的要求
9. 是否允许继续当前阶段循环

阶段通关裁决至少需要包含：

1. 阶段编号
2. 轮次
3. 裁决结果
4. 裁决理由
5. 被采纳的候选或合并来源
6. 仍存在但不阻塞的问题
7. 必须写入 canon 的内容
8. 必须归档但不得作为事实使用的内容

### 9.4 交接包必须是显式产物

交接包不能靠上下文隐式传递。

每次角色切换，都必须形成新的交接包。

A 到审核员的交接包包含：

1. 阶段输入包
2. A 候选版本
3. A 自检风险
4. 本轮目标

审核员到 B 的交接包包含：

1. 阶段输入包
2. A 公开候选
3. 审核员对 A 的审阅
4. B 必须处理的问题
5. B 不允许改变的目标边界

B 到审核员的交接包包含：

1. 阶段输入包
2. A 公开候选
3. A 审阅记录
4. B 候选版本
5. B 对 A 问题的处理说明

阶段到下一阶段的交接包包含：

1. 已通过 canon
2. 本阶段通过裁决
3. 后续阶段约束
4. 废弃候选清单
5. 非阻塞注意事项

### 9.5 评审记忆必须以问题台账为中心

审核员的记忆不能只是“上一轮总结”。

它必须维护一个问题台账。

每个问题至少有：

1. 问题编号
2. 首次发现轮次
3. 问题描述
4. 严重度
5. 归属层级
6. 当前状态
7. 相关候选版本
8. 审核员处理要求
9. 关闭依据

问题状态至少包括：

1. `open`
2. `improved`
3. `resolved`
4. `regressed`
5. `transferred_to_repair`
6. `accepted_non_blocking`

只有阻塞问题全部关闭或转入专项修复后，阶段才可能通关。

### 9.6 记忆读写必须分层

闭环运行中最危险的问题，是把候选、审阅、canon 混在一起。

因此记忆必须分层读写：

1. 候选层只保存创作者提交物
2. 审阅层只保存审核员判断和问题台账
3. canon 层只保存通过后的正式资产
4. 诊断层只保存异常、暂停、人工接管和失败信息

创作者默认只能读取 canon 层、当前交接包和允许公开的候选摘要。

审核员可以读取候选层、审阅层和 canon 层。

记忆管家可以读取裁决和可写入内容，但不能自行生成裁决。

交付整编者只能读取 canon 层、通过章节和交付检查记录。

### 9.7 循环必须有终止规则

返工循环不能无限自动运行。

循环终止有四种情况：

1. `pass`：阶段通过
2. `repair_required`：问题转入专项修复
3. `human_review_required`：需要人工判断
4. `fail_closed`：流程失败关闭

触发人工审核的情况包括：

1. 同一阻塞问题连续多轮未改善
2. A/B 输出开始互相复制而不产生有效提升
3. 审核员连续两轮无法给出稳定裁决
4. 当前阶段目标或通过标准本身含混
5. 自动修复会影响用户核心意图

### 9.8 纠错必须能回到原流程

纠错流程不能只是跳出去修一下。

它必须记录恢复点。

恢复点至少包括：

1. 原阶段
2. 原轮次
3. 原等待角色
4. 暂停原因
5. 修复目标
6. 修复通过后的返回位置

专项修复通过后，不能直接跳过原阶段通关。

它必须回到原流程中被暂停的位置，重新形成交接包，并让原审核员确认修复是否解决了暂停原因。

### 9.9 canon 写入必须不可逆污染

正式 canon 一旦写入，就会成为后续阶段事实。

因此 canon 写入必须满足：

1. 只允许从通过裁决进入
2. 只写入裁决明确指定的内容
3. 写入时必须保留来源候选和审阅依据
4. 旧 canon 被修订时必须记录替换关系
5. 被替换内容不能继续作为当前事实被读取

如果 canon 写入后发现错误，只能通过专项修复产生新 canon 版本，不能静默覆盖。

### 9.10 阶段交付必须有完成定义

每个阶段都要有完成定义。

世界观完成定义：

1. 基础规则可支撑长篇写作
2. 核心冲突可持续展开
3. 关键设定无自相矛盾
4. 后续大纲可直接引用

大纲完成定义：

1. 主线清晰
2. 阶段推进有因果
3. 章节展开空间足够
4. 与世界观一致

人物完成定义：

1. 核心人物动机明确
2. 成长或变化路径可追踪
3. 人物关系能支撑冲突
4. 与世界观、大纲兼容

章节完成定义：

1. 承接上文
2. 推进本章目标
3. 不破坏 canon
4. 语言和节奏达到当前作品标准
5. 可进入全书序列

全书完成定义：

1. 所有章节通过
2. 全书顺序完整
3. 不存在未关闭阻塞问题
4. 不存在等待修复的上游资产问题
5. 最终交付物满足用户目标

### 9.11 闭环失败通常发生在这些细节

以下任何一点缺失，流程都会看起来能跑，但实际无法长期健康运作：

1. 候选版本没有版本号，导致审核无法比较
2. 审阅记录没有问题状态，导致无法判断是否真的改进
3. 交接包不显式，导致信息靠聊天上下文漂移
4. 创作者能读到完整历史，导致双创作者同质化
5. 审核员没有结构化评审记忆，导致裁决前后不一致
6. 未通过候选进入 canon，导致后续阶段被污染
7. 纠错没有恢复点，导致修复后不知道回到哪里
8. 循环没有终止规则，导致任务无限返工
9. 最终交付没有检查清单，导致缺章、冲突或半成品被交付

这些不是实现偏好，而是流程闭环的关键控制点。

## 10. 长篇小说各阶段适配

这套流程不只适用于正文章节，也适用于前置资产建设。

### 10.1 世界观阶段

流程：

1. A 提交世界观方案
2. 审核员立刻审阅 A 方案，记录可写性、一致性、扩展性和缺陷
3. B 读取 A 的公开方案与审核员要求，提交世界观方案
4. 审核员立刻审阅 B 方案，比较其对 A 方案问题的修正效果
5. 审核员基于累计评审记忆判断是否可以形成 `world_canon`
6. 未通过则双创作者按裁决继续写新版本
7. 通过后形成 `world_canon`

双创作者的差异可以体现在：

1. 一方更关注规则闭环、长期可写性、设定稳定度
2. 另一方更关注想象力、辨识度、叙事牵引力

但两人都必须服务于同一个最终世界观方案。

### 10.2 大纲阶段

流程与世界观一致，但审核重点转为：

1. 结构稳定性
2. 节奏可持续性
3. 章节展开空间
4. 与世界观 canon 的一致性

双创作者的差异可以体现在：

1. 一方更重结构完整、因果清晰、长线承载能力
2. 另一方更重爆点密度、情绪推进、阅读吸引力

### 10.3 人物阶段

流程与前两者一致，但审核重点转为：

1. 人物驱动性
2. 动机一致性
3. 角色关系张力
4. 与世界观、大纲的兼容性

双创作者的差异可以体现在：

1. 一方更重人物逻辑、自洽性、成长闭环
2. 另一方更重人物魅力、冲突性、记忆点

### 10.4 章节阶段

章节阶段同样采用双创作者互审，但这里要更严格区分：

1. 章节规划
2. 正文候选
3. 审核裁决
4. 返工或通过

如果章节问题其实源于世界观、大纲、人物基础资产，则必须触发专项修复，而不是在正文层硬补。

双创作者的差异可以体现在：

1. 一方更重承接、清晰度、信息组织
2. 另一方更重现场感、情绪强度、阅读张力

## 11. 为什么这套流程适合长期运行

这套流程的价值，不在于“更复杂”，而在于它更可控。

它能支撑长期运行，原因是：

1. 信息传播路径明确
2. 创作者不会越跑越混成同一种声音
3. 审核员能够连续比较多轮版本的优劣
4. 审核通过前不污染正式资产
5. 每一轮返工都有明确依据
6. 后续回放时能看清是哪一轮、哪个角色、哪条裁决导致结果变化

## 12. 流程不可变约束

这份流程文档确定以下不可变流程边界：

1. 不能把双创作者做成共享长记忆 agent
2. 不能让审核员兼任创作者
3. 不能把审核意见写成自由评论而没有结构化裁决
4. 不能让未通过候选直接进入 canon
5. 不能让返工只依赖模糊反馈，必须有明确下一轮要求
6. 不能把审核员变成阶段末尾才出现的一次性总评角色
7. 不能跳过“候选版本之后立刻审阅”的基本节拍

## 13. 结论

长篇小说团队的正确目标模型已经明确：

**双创作者隔离创作，基于公开提案互相施压；审核员作为唯一门禁，反复裁决，直到质量达标后才允许通过并沉淀正式资产。**

后续所有讨论，都应先服从这个流程模型，不再反过来被旧结构牵着走。

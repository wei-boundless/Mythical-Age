# 193-任务图批量派发与 WorkerPool 运行闭环实施计划

日期：2026-05-20

## 1. 目标

在 `192` 已完成的“并行批次执行实例”基础上，补齐通用任务图从“状态可并行”走向“运行可批量派发”的框架出口。

本阶段不新建第二套执行器，不写业务专用逻辑，而是把现有 `NodeExecutionRequest` 扩展成可批量领取、批量调度、可监控的通用运行对象。

## 2. 当前问题

现在运行态已经可以同时存在多个 `batch_execution_instances`，并且结果可以按 request / dispatch / execution id 归位。

但现有协调循环仍以单个 `stage_execution_request` 作为主要出口。这样会导致：

- 前端或 API 只能自然拿到一个当前节点请求。
- 并行批次需要靠重复 continue 才能继续领取下一个批次。
- WorkerPool 没有标准的“可派发请求列表”。
- 监控能看到实例状态，但不能直接看到调度容量、可领取批次和已占用槽位。

## 3. 设计原则

1. `NodeExecutionRequest` 仍是唯一标准执行请求，不新增私有执行协议。
2. 批量派发只消费任务图契约和批次运行态，不消费业务字段。
3. 调度入口只负责生成/领取/投递请求，不负责判断内容质量。
4. 已存在的后台执行通道继续负责实际 agent / graph unit 执行。
5. 并行安全以 execution identity 为准，不能回退到“当前节点的任意活动批次”。

## 4. 本轮交付

### 4.1 后端运行时批量领取

在 LangGraph 协调运行时新增批量领取能力：

- 可以包含当前已经生成但尚可执行的 `stage_execution_request`。
- 可以继续从同一批次节点领取更多 ready batch，直到达到 `max_requests` 或并行上限。
- 每个领取出的请求都必须是标准 `NodeExecutionRequest`。
- 每个新请求都会创建 dispatch event、写入 batch execution instance，并 checkpoint。

### 4.2 API 批量派发入口

新增协调运行级入口：

```text
POST /orchestration/coordination-runs/{coordination_run_id}/dispatch-ready-batches
```

请求参数：

- `max_requests`: 本次最多领取多少个请求。
- `include_current_request`: 是否把当前已生成请求纳入派发列表。
- `execute_background`: 是否直接投递到现有后台执行通道。
- `source`: 调用来源。
- `current_turn_context`: 调用方额外上下文。

返回内容：

- 标准执行请求列表。
- 每个请求的后台调度结果。
- 本次领取后的 checkpoint ref。
- 批次 dispatcher 摘要。

### 4.3 监控视图

在任务图运行 monitor 中新增 `batch_dispatcher`：

- 每个节点的并行容量。
- 当前 active execution 数。
- ready / running / committed / failed 数。
- 可领取槽位数。
- dispatchable batch ids。
- active request ids。

### 4.4 前端契约

前端本轮先接收并展示后端 monitor/API 结构所需类型，不单独设计新页面。

后续如果要做 WorkerPool 控制台，再放到任务系统工作台的运行层页面，不把它混进节点配置页。

## 5. 不做事项

- 不直接实现真正多进程 worker pool。
- 不改变 agent 执行链路。
- 不绕过现有 `_schedule_stage_execution_background`。
- 不让写作任务或章节逻辑进入调度器。
- 不把 `auto_commit_without_review` 当作并行默认策略。

## 6. 完成标准

1. 并行批次图启动后，可以一次领取当前请求和额外 ready batch 请求。
2. 领取出的每个请求都有独立 `unit_batch_execution_id`。
3. API 可以选择只领取不执行，也可以投递到现有后台执行通道。
4. monitor 可以看到 dispatcher 容量和可领取批次。
5. 后端回归和前端类型检查通过。

## 7. 实施完成记录

本轮已经完成：

1. `LangGraphCoordinationRuntime.dispatch_ready_batch_requests()` 可以从同一协调运行中批量领取标准 `NodeExecutionRequest`。
2. 领取逻辑复用 `_stage_prepare()` / `_stage_execute()`，因此每个批次请求仍经过输入绑定、时序事件、上下文包、运行组装和批次 execution identity 回写。
3. 新增 `POST /orchestration/coordination-runs/{coordination_run_id}/dispatch-ready-batches`，支持只领取或直接投递现有后台执行通道。
4. `task_graph_run_monitor` 新增 `batch_dispatcher` 视图，暴露节点级并行容量、活动执行数、可领取槽位、dispatchable batch ids。
5. 前端 API 类型增加 `batch_dispatcher` 与 `dispatchCoordinationReadyBatches()` 调用入口。

## 8. 当前边界

本轮完成的是 WorkerPool 前置闭环：运行时可以批量领取和投递多个标准节点请求。

还没有做真正的常驻 worker pool、执行队列持久化、取消/暂停/限流控制台。这些应该作为下一阶段进入任务系统运行层，而不是混入节点契约编辑器。

下一阶段建议：

1. 增加持久化 dispatch queue，记录 pending / scheduled / running / completed / cancelled。
2. 增加按 graph、node、agent、provider 的并发令牌和限流策略。
3. 增加批量取消、暂停某节点批次、重排队某 execution instance。
4. 将 WorkerPool 控制台放入任务系统运行页，和图编辑层分离。

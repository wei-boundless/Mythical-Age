# 全项目系统边界总设计书

日期：2026-05-04

## 一、文档定位

本文件是本项目的新总设计文档。

它用于替代已经废弃的旧任务系统讨论稿、旧前端拼装稿和旧边界混合稿，作为后续任务系统、运行体系统、记忆系统、健康系统、操作系统、灵魂系统和 runtime 重构的统一依据。

本文件回答以下核心问题：

- 这个项目到底是什么项目
- 六大系统各自负责什么
- Agent 在平台中的本体是什么
- 任务、Agent、workflow、projection、运行体、记忆、能力、健康之间的关系是什么
- 后续前后端和运行时应按什么边界实施

后续实现一律以本文件为准。

## 二、项目总定义

本项目不是单一 Agent 的能力开发工程，而是一个任务驱动的 Agent 平台工程。

平台以任务系统作为调度中枢，以 Agent 作为标准执行接口，以运行体系统负责单 Agent 的组装、运行约束与跨系统交接结构，以记忆系统负责记忆治理与记忆供给，以操作系统提供能力资源，以灵魂系统提供 prompts 身份层，以健康系统负责运行监测与诊断。

六个系统共同构成平台级 Agent 的登记、采用、生产、装配、执行与治理框架。

## 三、重构总目标

本轮重构的目标不是继续修补某个页面，也不是继续为局部问题临时补字段，而是把整个平台的系统边界彻底立起来。

重构后的平台必须满足：

1. 能把项目清楚地看作一个 Agent 平台，而不是一个会话壳
2. 能把任务、Agent、workflow、projection、运行体、记忆、能力、健康彻底解耦
3. 能让任务系统稳定登记任务、分配任务、采用 Agent、生产工作子 Agent
4. 能让运行体系统稳定组装 Agent，而不是与任务系统混在一起
5. 能让记忆系统独立承担记忆治理、记忆组织和记忆供给
6. 能让操作系统统一提供 skills、tools、资源和执行能力
7. 能让灵魂系统专注于 prompts 身份层，不再承担任务方法和权限逻辑
8. 能让健康系统独立承担平台监测、诊断和回归验证
9. 不保留无意义旧残留，不以兼容为理由维持错误边界

## 四、平台总边界

### 4.1 平台中的六类核心对象

本平台围绕以下六类核心对象运行：

- `Task`
- `Agent`
- `MemoryContext`
- `Capability`
- `PromptIdentity`
- `RuntimeHealth`

它们分别对应：

- 任务目标与任务契约
- 执行接口与执行单元
- 记忆组织、记忆检索与记忆供给
- 工具、skills、资源与可调用能力
- soul / projection / prompts 设定
- 运行状态、诊断结果与健康反馈

### 4.2 Agent 不是系统

系统是治理域、资源域、设定域和监测域。

Agent 是执行域。

系统不直接执行任务，Agent 才执行任务。

因此：

- 主 Agent 不是主系统
- 健康管理 Agent 不是健康系统
- 记忆管理 Agent 不是记忆系统
- 运行体管理 Agent 不是运行体系统
- 操作管理 Agent 不是操作系统
- 灵魂管理 Agent 不是灵魂系统

它们只是由平台提供、并对接不同入口和任务的标准执行单元。

### 4.3 平台不再按单一 Agent 思维建模

本平台默认是多 Agent 平台。

即使当前某些链路先按单 Agent 主链路实施，系统设计也必须天然支持：

- 多个固定入口 Agent
- 多个系统管理 Agent
- 多个工作子 Agent
- 后续多 Agent 协作

不能再把所有运行逻辑都压在一个主 Agent 身上。

## 五、六大系统的职责与边界

## 5.1 任务系统

任务系统是平台的调度中枢。

它负责：

- 任务登记
- 任务分类
- 任务输入输出契约
- 通用任务与特定任务管理
- 任务分配
- 任务采用哪个 Agent
- 必要时生产新的工作子 Agent
- 多 Agent 协调任务的拓扑分配
- 多 Agent 协调任务的角色分配
- 多 Agent 协调任务中的通信协议管理
- 多 Agent 协调任务中的 handoff 规则管理
- 任务绑定 workflow
- 任务绑定 projection
- 任务绑定 flow contract
- 任务执行入口与任务消费路径管理

它回答的问题是：

`这是什么任务，应该交给谁，以什么任务配置去做`

任务系统不负责：

- Agent 内部 prompt 结构设计
- 工具权限细节
- 资源访问边界细节
- 运行时执行细节
- 系统健康诊断

### 5.1.1 任务系统内部最小管理对象

任务系统至少要管理：

- `GeneralTaskProfile`
- `SpecificTaskRecord`
- `TaskWorkflowRecord`
- `TaskProjectionBinding`
- `TaskFlowContractBinding`
- `TaskAgentAdoptionPlan`
- `CoordinationTaskRecord`
- `CoordinationTopologyTemplate`
- `TaskCommunicationProtocol`

### 5.1.2 通用任务与特定任务

任务系统中的任务至少分两类：

#### 通用任务

用于承接用户日常对话目标。

特点：

- 默认由主 Agent 承接
- 允许模型承担一定理解
- 主要用于主会话入口
- 本质上是平台的默认承接任务

#### 特定任务

用于承接已登记、需要稳定消费和稳定执行的任务。

特点：

- 必须预先登记
- 必须指定任务契约
- 必须明确 workflow
- 可以指定 projection
- 可以指定采用现有 Agent 或生产工作子 Agent
- 必须输出稳定的 runtime 任务装配结果

特定任务不能只靠模型自由理解。

### 5.1.3 多 Agent 协调任务边界

多 Agent 协调任务仍然属于任务系统。

因为这部分回答的仍然是：

- 哪些 Agent 参与任务
- 谁是 coordinator
- 谁是 worker
- 任务拓扑如何展开
- 任务阶段之间如何交接
- Agent 之间传递什么信号和什么结构化结果

这些都属于任务调度与任务组织问题，而不是单个 Agent 的内部运行问题。

因此：

- 拓扑分配归任务系统
- 角色分配归任务系统
- 通信协议归任务系统
- handoff 规则归任务系统

运行体系统只负责：

- 每个 Agent 自己内部的 prompts 结构
- 每个 Agent 自己的工具边界
- 每个 Agent 自己的资源边界
- 每个 Agent 自己的 memory scope

## 5.2 运行体系统

运行体系统是平台的单 Agent 身体管理系统、Agent 组装厂和运行约束中心。

它负责：

- Agent 内部 prompts 结构装配
- Agent 内部结构编排
- Agent 权限边界
- Agent 工具边界
- Agent 资源边界
- Agent memory scope
- Agent runtime lane
- Agent output boundary
- Agent 与其他系统的交接接口
- Agent 的运行壳与运行体装配
- Agent 的默认运行 profile
- Agent 的装配模板和运行壳

它回答的问题是：

`这个 Agent 作为一个单独执行体，身体怎么被装起来、内部怎么编排、能做什么、不能做什么、如何与其他系统交接`

运行体系统不负责：

- 任务登记
- 任务分类
- workflow 管理
- projection 选择
- 哪个任务交给哪个 Agent

### 5.2.1 运行体系统的定位

运行体系统不是任务系统的子模块，也不是纯权限面板。

它本质上是平台的运行结构层。

任务系统把任务交给某个 Agent 之后，真正把这个 Agent 组装成可执行体的是运行体系统。

## 5.3 记忆系统

记忆系统是平台的记忆治理层和记忆供给层。

它负责：

- 会话记忆组织
- 长短期记忆分层
- 记忆提炼
- 记忆索引
- 记忆检索
- 记忆写回策略
- 记忆作用域治理
- 为 Agent 提供可消费的记忆上下文

它回答的问题是：

`这个 Agent 在当前任务和长期平台运行中，应该拿到哪些记忆，不应该拿到哪些记忆`

记忆系统不负责：

- 任务登记
- workflow 管理
- projection 选择
- prompts 人格设定
- 能力资源供给
- 健康诊断

### 5.3.1 记忆系统的定位

记忆系统不是对话记录堆放区，也不是任务系统的附属字段集合。

它本质上是平台的上下文延续与记忆治理中心。

任务系统可以声明记忆需求，运行体系统可以约束 memory scope，但记忆内容的组织、筛选、沉淀与供给必须由记忆系统负责。

## 5.4 健康系统

健康系统是平台的观测、诊断和回归验证中心。

它负责：

- Agent 运行健康监测
- 平台级运行健康监测
- 任务执行链路诊断
- 异常定位
- 回归验证
- 故障归因
- 风险检测
- 状态核验

它回答的问题是：

`现在哪些 Agent 或系统正常工作，哪些没有，问题出在哪`

健康系统不负责：

- 任务分配
- Agent 生产
- workflow 管理
- prompts 身份设定
- 能力资源供给

## 5.5 操作系统

操作系统是平台的能力资源供应层。

它负责：

- tools
- skills
- 文件系统能力
- shell / browser / web / code execution 等执行能力
- 外部资源接入
- 资源目录
- 能力可见性与可引用性

它回答的问题是：

`平台上有哪些能力资源可以被 Agent 使用`

操作系统不负责：

- 任务调度
- Agent 采用与生产
- prompts 人格设定
- 运行健康诊断

### 5.5.1 workflow 不再属于操作系统

workflow 是任务方法，不是能力资源。

因此 workflow 必须归任务系统管理，不能再归操作系统。

## 5.6 灵魂系统

灵魂系统是平台的 prompts 身份层。

它负责：

- soul prompt
- projection prompt
- 人格设定
- 表达风格
- 上下文组织风格
- 注意力组织方式
- 任务执行姿态的 prompts 补充

它回答的问题是：

`这个 Agent 以什么人格和 prompts 身份结构来执行任务`

灵魂系统不负责：

- workflow 定义
- 权限定义
- 任务调度
- 能力供给
- 运行健康诊断

### 5.5.1 灵魂系统中的工作风格边界

灵魂系统不再绑定固定工作流，也不再承载任务系统意义上的 workflow。

灵魂系统可以保留：

- 人格设定
- 上下文组织风格
- 任务执行姿态提示
- 用户自填的工作风格文本

但 workflow 仍然必须由任务系统管理，并在任务执行时通过装配顺序叠加进入 prompts。

## 六、Agent 的平台级定义

### 6.1 Agent 的本体定义

Agent 是平台中的标准执行接口。

它不是系统，也不是任务。

它是被平台采用、被任务装配、被运行体系统组装、被记忆系统供给记忆上下文、被灵魂系统赋予 prompts 身份、被操作系统供给能力、被健康系统监测的执行单元。

### 6.2 主 Agent 与子 Agent 的关系

主 Agent 和子 Agent 没有本体差异。

它们都是 Agent。

差异只在于：

- 接口入口不同
- 接收任务不同
- 输出对象不同
- 默认运行设定不同

### 6.3 Agent 分类

平台中的 Agent 至少分三类：

#### 主 Agent

- `agent:0`
- 对接主会话页面
- 默认承接通用任务
- 负责最终整合输出
- 不应在任务系统中被误看成系统本体

#### 系统管理 Agent

- `agent:1` 到 `agent:5`
- 对接五个子系统的独立会话入口
- 承接各系统相关管理任务
- 是系统的维护执行者，不是系统本身

#### 工作子 Agent

- 动态生产
- 为特定任务持续服务
- 可被创建、采用、停用、删除
- ID 连续增长并由注册表统一管理

### 6.4 平台初始固定 Agent

平台初始固定六个 Agent：

1. `agent:0` 主 Agent
2. `agent:1` 运行体系统管理 Agent
3. `agent:2` 记忆系统管理 Agent
4. `agent:3` 健康系统管理 Agent
5. `agent:4` 操作系统管理 Agent
6. `agent:5` 灵魂系统管理 Agent

这六个 Agent 是平台初始接口集合。

其中：

- `agent:0` 对接主会话
- `agent:1-5` 后续对接五个系统的会话窗口或悬浮入口

## 七、Agent 的四层模型

为了让 Agent 能被快速组装、稳定采用、可审计运行，平台必须把 Agent 分成四层，而不是把所有字段都堆在一个注册记录上。

### 7.1 AgentRegistryRecord

负责回答：

`这个 Agent 是谁`

它只保存稳定身份信息，例如：

- `agent_id`
- `agent_name`
- `agent_category`
- `interface_target`
- `enabled`
- `builtin`
- `editable`
- `managed_system`

### 7.2 AgentDefinitionProfile

负责回答：

`这个 Agent 默认是什么样`

它保存默认运行画像，例如：

- `base_soul_id`
- `base_projection_id`
- `context_style`
- `default_runtime_lane`
- `default_memory_scope`
- `permission_profile_id`
- `output_profile_id`

### 7.3 AgentTaskMount

负责回答：

`这次任务把这个 Agent 装成什么样`

它保存任务挂载信息，例如：

- `task_id`
- `agent_id`
- `workflow_id`
- `projection_id`
- `flow_contract_id`
- `input_contract_id`
- `output_contract_id`
- `safety_policy`
- `task_prompt_overrides`

### 7.4 AgentRuntimeSpec

负责回答：

`这次运行真正送给 runtime 的是什么`

它是最终运行态装配产物，由任务系统、运行体系统、记忆系统、操作系统和灵魂系统共同参与生成。

## 八、任务、workflow、projection、权限之间的关系

### 8.1 task 是目标

task 定义要做什么。

### 8.2 workflow 是任务方法

workflow 定义任务做法。

它归任务系统管理，不归运行体系统，不归操作系统。

### 8.3 projection 是任务执行时的 prompts 补充

projection 定义任务执行姿态和 prompts 补充，不定义任务方法。

它由灵魂系统提供素材，由任务系统在任务侧选择和绑定。

### 8.4 运行体约束是 Agent 运行结构

运行体约束定义 Agent 在 runtime 中怎么运行，内部如何编排，能做什么，不能做什么，以及怎么与其他系统交接。

### 8.5 四者顺序不能颠倒

任务执行时的 prompts 和运行装配必须遵循：

1. 先确定 `task`
2. 再确定 `workflow`
3. 再确定 `projection`
4. 再由记忆系统按作用域供给记忆上下文
5. 最后交给运行体系统装配到运行壳

不能把 projection 当 workflow，也不能把 workflow 当权限。

## 九、平台主链路

平台主链路如下：

```text
任务进入平台
  -> 任务系统识别任务类型
  -> 任务系统登记或命中特定任务
  -> 任务系统选择采用现有 Agent 或生产工作子 Agent
  -> 若为多 Agent 任务，任务系统生成协调拓扑与通信协议
  -> 任务系统绑定 workflow / projection / flow contract
  -> 记忆系统提供任务所需记忆上下文
  -> 运行体系统组装 Agent 的运行结构
  -> 灵魂系统提供 soul / projection prompts
  -> 操作系统提供所需能力资源
  -> runtime 执行任务
  -> 健康系统持续监测和诊断
  -> 输出任务结果
```

## 十、任务系统与 Agent 平台的关系

任务系统不是一个单独的页面模块，而是平台中承接任务和调度执行的中心。

没有任务系统，平台只能提供静态 Agent 列表，不能提供有效任务消费。

没有 Agent 平台，任务系统也只是一个登记壳，不能真正完成任务。

因此两者关系是：

- 平台提供 Agent 运行能力
- 任务系统驱动 Agent 参与任务

## 十一、平台级前端目标

前端重构时，界面不是展示概念，而是要真正服务于平台管理。

至少应形成以下一级结构：

1. Agent 管理
2. 任务管理
3. 运行体管理
4. 记忆管理
5. 健康监测管理
6. 能力资源管理
7. 灵魂与投影管理

### 11.1 Agent 管理页目标

Agent 管理页负责：

- 按类别显示 Agent
- 查看六个固定 Agent
- 查看工作子 Agent
- 创建工作子 Agent
- 修改 Agent 基本登记信息
- 查看 Agent 默认定义
- 查看 Agent 当前承载任务关系

### 11.2 任务管理页目标

任务管理页负责：

- 管理通用任务
- 管理特定任务
- 为特定任务绑定 workflow
- 为特定任务绑定 projection
- 为特定任务绑定采用 Agent 或生产策略
- 输出清晰的任务消费结构

### 11.3 运行体系统页目标

运行体系统页负责：

- 查看和管理 Agent 运行 profile
- 查看和管理 Agent 内部结构编排
- 管理工具权限
- 管理资源边界
- 管理输出边界
- 管理跨系统交接接口
- 管理 runtime lane 与 memory scope

### 11.4 记忆系统页目标

记忆系统页负责：

- 查看记忆层次与记忆来源
- 查看 Agent 默认记忆作用域
- 管理任务可申请的记忆类型
- 管理记忆写回策略
- 管理记忆检索与记忆供给规则
- 查看记忆是否进入当前运行装配

## 十二、后端重构总原则

后端重构必须遵循以下原则：

1. 先修系统边界，再修局部行为
2. 先建立标准对象，再做页面字段联动
3. 不再保留错误命名和旧混合对象
4. workflow 不再作为提示词碎片存在，而要作为任务侧正式对象存在
5. Agent 注册信息、默认定义、任务挂载、运行装配必须分层
6. runtime 不再自由猜测任务形态，而应消费正式装配结果

## 十四、执行结论

本项目从本文件开始，统一视为 Agent 平台。

任务系统负责：

`谁做什么`

运行体系统负责：

`这个 Agent 的身体怎么被装起来做`

记忆系统负责：

`给这个 Agent 什么记忆去做`

操作系统负责：

`拿什么能力做`

灵魂系统负责：

`以什么 prompts 身份和姿态去做`

健康系统负责：

`做得是否正常，哪里出了问题`

这六条边界一旦锁定，后续前后端和 runtime 重构才有稳定基础。

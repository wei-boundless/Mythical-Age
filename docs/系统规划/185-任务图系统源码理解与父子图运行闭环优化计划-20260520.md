# 185-任务图系统源码理解与父子图运行闭环优化计划

日期：2026-05-20

## 1. 本轮结论

我重新按源码链路审查后，对当前任务图系统的判断是：

任务图系统已经不是单纯的前端画布，也不是写作任务专用流程。它的核心链路已经形成：

```text
TaskGraphDefinition
  -> LayeredGraph
  -> StandardView / ComposableGraphView
  -> RuntimeSpec
  -> ContractManifest
  -> SchedulerState
  -> ExecutionPackage
  -> Runtime Loop / GraphUnit child run
  -> child output packet
  -> parent resume
```

父子图也不是“完全不能跑”。后端已经能把父图里的 `timeline_blocks[].linked_graph_id` 编译成 `TaskGraphNestedRuntimePlan`，再变成 `graph_unit.*` runtime node。执行时，GraphUnit 节点会启动一个子任务图 run，子图完成后生成 `graph_unit_output_packet` 或 `graph_unit_failure_packet`，再被父图当前节点消费。

真正的问题在于：父子图目前还不是足够自然的一等编辑对象。前端可以看到和编辑 GraphUnit，但主要来源仍是 `metadata.timeline_blocks` 与可组合覆盖层；StandardView 里也还保留 `read_only_shadow_model / metadata_overlay_shadow_model` 的诊断语义。这说明“图是节点、节点可展开为图”的运行链路有了，但编辑模型、接口契约、版本锚、端口映射、运行包追踪和调试入口还需要继续收敛。

所以后续优化重点不是给某个任务开私门，而是把 GraphUnit 从“由时序块派生出来的可运行节点”推进成“任务图系统可配置、可追踪、可发布、可恢复的标准对象”。

## 2. 准确定义

### 2.1 任务图

任务图是长任务的结构化运行定义。它不是一次执行本身，而是可以被保存、发布、复用、嵌套和运行的流程对象。

源码事实源是 `TaskGraphDefinition`：

- 图身份：`graph_id/title/domain_id/task_family`
- 图拓扑：`nodes/edges`
- 图入口出口：`entry_node_id/output_node_id`
- 图级契约：`contract_bindings`
- 运行策略：`runtime_policy/context_policy/working_memory_policy`
- 发布状态：`publish_state/enabled`
- 扩展结构：`metadata`

对应源码：`backend/tasks/task_graph_models.py`

### 2.2 节点

节点不是任务本身。节点是任务图里的运行位置，也就是一个时序点、执行位、职责位。

节点可以绑定任务定义、Agent、Projection、模型需求、记忆策略、产物策略、验收策略、运行策略。真正运行时，节点会被编译成 `TaskGraphRuntimeNode`，再由调度器决定是否 ready，最后由 runtime loop 生成节点执行请求。

这和用户的理解是一致的：节点是执行位置，不是执行动作本身。

### 2.3 任务

任务是节点被激活时要执行的动作或配方。它可以来自 `task_id` 引用的任务定义，也可以由节点的执行器、Agent、工具、Prompt、契约和运行档案共同装配出来。

因此正确语义是：

```text
节点定义执行位
任务定义执行动作
调度器决定节点何时激活
运行层把节点激活转成具体任务执行
```

### 2.4 边

边不是执行端。边是交接端。

边定义上游节点的输出如何成为下游节点的合法输入，包括：

- 载荷契约
- 上下文过滤
- 记忆交接
- 产物引用
- 确认策略
- 等待策略
- 失败传播
- 结果投递
- 时序语义

边在运行时会进入 `TaskGraphRuntimeEdge`，再被 scheduler 用来判断下游节点是否 ready。

### 2.5 图节点

图节点是父任务图里代表一个子任务图运行的节点。

当前源码里的 GraphUnit 来源不是普通 `graph.nodes[]` 里直接写一个 node type 就结束，而是主要由 `metadata.timeline_blocks[].linked_graph_id` 派生：

```text
timeline_block + linked_graph_id
  -> NestedRuntimePlan
  -> graph_unit runtime node
  -> child TaskGraph run
  -> committed output packet
  -> parent graph resumes
```

因此现在的 GraphUnit 是“已可运行，但来源还偏派生”的对象。目标应该是让 GraphUnit 成为用户在图工作台中自然配置的图节点对象，同时保持它可以被编译为同一套 nested runtime plan。

## 3. 五层模型

### 3.1 保存态：TaskGraphDefinition

保存态是用户真正保存进库里的任务图定义。它是任务图系统的主数据。

关键源码：

- `backend/tasks/task_graph_models.py`
- `backend/tasks/flow_registry.py`
- `frontend/src/components/workspace/views/task-system/taskGraphSaveMapper.ts`

保存态的重要规则：

- 图、节点、边都有 `contract_bindings`。
- 旧字段 `graph_contract_id/node_contract_id/input_contract_id/output_contract_id/payload_contract_id` 仍存在，但主路径应该是 `contract_bindings`。
- 后端会把旧字段归一进 `contract_bindings`，并检测冲突。
- 节点禁止保存 raw secret 和 base URL。模型需求走 `contract_bindings.runtime.model_requirement`，Provider endpoint 和 credential 归系统配置与 Agent 运行档案。

这说明契约模型已经有主方向，但前端还要继续减少旧字段认知。

### 3.2 分层态：LayeredGraph

分层态从任务图中抽取不同职责层：

- execution
- timeline
- memory
- artifact_context
- revision
- resource
- loop
- timeline_blocks

关键源码：

- `backend/tasks/layered_graph_normalizer.py`

`normalize_task_graph_layers()` 会从节点、边、metadata 中抽取：

- `resource_nodes`
- `temporal_edges`
- `memory_edges`
- `artifact_context_edges`
- `revision_edges`
- `loop_frames`
- `timeline_blocks`
- `memory_matrix`

这里是“时序分层”和“图结构”结合的关键点。父子图也从这里进入后续编译，因为 GraphUnit 的来源就是 `timeline_blocks` 中的 `linked_graph_id`。

### 3.3 标准态：StandardView / ComposableGraphView

标准态是给前端、预检、发布页使用的标准对象视图。

关键源码：

- `backend/tasks/task_graph_standard_models.py`
- `backend/tasks/composable_graph_builder.py`
- `backend/tasks/composable_graph_models.py`

StandardView 包含：

- `graph`
- `nodes`
- `edges`
- `resources`
- `units`
- `interfaces`
- `port_edges`
- `nested_runtime`
- `timeline`
- `runtime_isolation`
- `memory_matrix`
- `issues`

ComposableGraphView 把保存态节点、边和 timeline block 变成可组合对象：

- `ComposableUnit`
- `UnitInterface`
- `UnitPortEdge`
- `NestedRuntimePlan`

当前关键问题也在这里：`build_composable_graph_view()` 的 diagnostics 仍会标记 `read_only_shadow_model` 或 `metadata_overlay_shadow_model`。这不代表它完全无效，但说明它还不是纯粹的一等保存模型，而是“由保存态派生，并允许 metadata overlay 覆盖”的视图。

### 3.4 运行态：RuntimeSpec

运行态是调度器和运行层真正消费的结构。

关键源码：

- `backend/tasks/coordination_graph_compiler.py`
- `backend/tasks/coordination_graph_models.py`

`compile_task_graph_definition_runtime_spec()` 会把保存态任务图编译为：

- runtime nodes
- runtime edges
- resource nodes
- temporal edges
- memory edges
- artifact context edges
- revision edges
- loop frames
- nested runtime plans
- memory matrix
- runtime diagnostics

GraphUnit 的运行化在这里完成：

```text
_nested_runtime_plans_from_layered_graph()
  -> TaskGraphNestedRuntimePlan

_runtime_nodes_from_nested_runtime_plans()
  -> TaskGraphRuntimeNode(node_type="graph_unit")
```

每个 GraphUnit runtime node 的 executor policy 会被设成：

```text
default_executor = graph_unit
allowed_executors = ["graph_unit"]
subgraph_id = linked_graph_id
auto_start_child_initial_stage = true
```

这就是父子图开始变成真实运行对象的位置。

### 3.5 发布执行态：ExecutionPackage

发布执行态是“发布前真实执行事实源”。

关键源码：

- `backend/api/tasks.py`
- `frontend/src/components/workspace/views/task-system/TaskGraphExecutionPackagePanel.tsx`
- `frontend/src/components/workspace/views/task-system/TaskGraphExecutionDock.tsx`

后端接口：

```text
GET /api/tasks/execution-packages/task-graphs/{graph_id}
```

返回内容包含：

- `standard_view`
- `contract_manifest`
- `runtime_spec`
- `node_runtime_assemblies`
- `scheduler_state`
- `graph_units`
- `graph_unit_execution_plans`
- `object_trace_index`
- `issues`
- `summary`

这很关键。它说明运行前不应该再让用户分别去猜 runtime spec、manifest、scheduler、assembly 到底是否一致。ExecutionPackage 应该成为图工作台发布与调试的中心事实源。

## 4. contract_bindings 的真实作用

`contract_bindings` 是图任务系统的统一契约承载，不应该只是字段聚合。

当前后端支持的主要分区包括：

```text
schema
execution
unit_batch
artifact
memory
handoff
acceptance
runtime
governance
temporal
```

### 4.1 图级契约

图级契约声明：

- 图接口
- 图级运行策略
- 上下文策略
- 工作记忆策略
- 批次或治理策略

源码中 `normalize_graph_contract_bindings()` 会把 `graph_contract_id/runtime_policy/context_policy/working_memory_policy` 归入图级 `contract_bindings`。

### 4.2 节点级契约

节点级契约声明：

- 输入输出 schema
- 执行契约
- 工具和 skill 引用
- 记忆读取和写回
- 产物目标
- 审核与人工门控
- 运行模型需求
- 治理边界

源码中 `normalize_node_contract_bindings()` 会把节点旧字段和节点策略归入 `contract_bindings`，并且会拒绝 raw secret。

### 4.3 边级契约

边级契约声明：

- payload contract
- handoff policy
- memory handoff
- artifact ref policy
- temporal semantics
- governance boundary

源码中 `normalize_edge_contract_bindings()` 会把边的 payload、ack、wait、failure、result delivery、temporal semantics 归入 `contract_bindings`。

### 4.4 GraphUnit 契约

GraphUnit 的契约重点是父子图交接：

- `handoff.handoff_contract_id`
- `runtime.model_requirement`
- `governance.context_boundary_policy_ref`
- `temporal.trigger_timing`
- `temporal.visibility_timing`
- `temporal.propagation_timing`

后端 `contract_compiler.py` 已有 `_compile_graph_unit_handoff_contracts()`，会从 nested plan 的 metadata 中读取 `contract_bindings.handoff.handoff_contract_id`，并编译成 `CompiledGraphUnitHandoffContract`。

这说明 GraphUnit 的 handoff 契约已经不是纯 UI 字段，而是会进入 manifest。

## 5. 父子图完整链路

### 5.1 编辑入口

当前前端主要通过图工作台编辑 GraphUnit：

- 左侧图层关系：`TaskGraphGraphLayerRail`
- 中间画布：`TaskGraphComposableCanvas`
- 右侧对象编辑台：`TaskGraphObjectInspector`
- 图节点编辑器：`TaskGraphGraphUnitInspector`

用户新增或编辑图节点时，实际写入的是：

```text
graph.metadata.timeline_blocks[]
```

典型字段：

```text
block_id
block_type
title
phase_id
linked_graph_id
entry_node_id
exit_node_id
handoff_contract_id
visibility_policy
version_ref
detach_policy
contract_bindings
```

前端 `coordinationTimelineBlocks()` 会读取这些 block，后端 `normalize_task_graph_layers()` 也会读取这些 block。

### 5.2 标准视图派生

`build_composable_graph_view()` 会把 timeline block 派生为：

- `unit_type="graph"` 的 `ComposableUnit`
- 对应 `UnitInterface`
- `NestedRuntimePlan`

只要 block 有 `linked_graph_id`，就会进入 `nested_runtime`。

这一步让前端能够看到“子图入口”和“嵌套运行”，但其保存态仍是 timeline block。

### 5.3 RuntimeSpec 编译

`compile_task_graph_definition_runtime_spec()` 会先调用：

```text
normalize_task_graph_layers(graph)
```

然后从 `layered_graph.timeline_blocks` 生成：

```text
TaskGraphNestedRuntimePlan
```

关键字段：

```text
plan_id
parent_graph_id
unit_id
runtime_node_id
linked_graph_id
version_ref
handoff_contract_id
input_port_id
output_port_id
isolation_policy
visibility_policy
detach_policy
phase_id
sequence_index
metadata
```

接着 `_runtime_nodes_from_nested_runtime_plans()` 会生成 runtime node：

```text
node_id = graph_unit.{block_id}
node_type = graph_unit
role = nested_graph
execution_mode = async
wait_policy = wait_all_upstream_completed
executor_policy.default_executor = graph_unit
```

这一步是父子图从配置变成可执行节点的核心。

### 5.4 ContractManifest 编译

`compile_coordination_contract_manifest()` 会编译：

- node contracts
- edge handoff contracts
- graph unit handoff contracts
- runtime contracts
- acceptance contracts

GraphUnit 的 handoff contract 来自：

```text
plan.metadata.contract_bindings.handoff.handoff_contract_id
plan.handoff_contract_id
legacy timeline block field
```

因此父子图交接不是隐式字符串，而应该被明确契约化。

### 5.5 ExecutionPackage 预编译子图

`_compiled_task_graph_execution_parts()` 会聚合父图的：

- StandardView
- RuntimeSpec
- ContractManifest
- SchedulerState
- Node Runtime Assemblies
- GraphUnit Execution Plans

`_graph_unit_execution_plans()` 会对每个 nested runtime plan 做子图预检：

- `linked_graph_id` 不得为空
- 不得直接自引用
- 子图必须存在
- 子图会被编译 StandardView
- 子图会被编译 RuntimeSpec
- 子图会被编译 ContractManifest
- 子图会生成 Scheduler shadow state
- 子图普通节点会生成 runtime assembly summary

如果子图 runtime spec 或 manifest invalid，会成为 GraphUnit execution plan 的问题。

这说明 ExecutionPackage 已经能回答一个关键问题：父图里的图节点到底能不能联动运行子图。

### 5.6 Scheduler 调度

`bootstrap_scheduler_state()` 会把 graph_unit node 当作普通 runtime node 纳入调度：

- start node
- upstream dependency
- phase gate
- sequence gate
- wait policy
- handoff ack
- result contract
- artifact requirement
- failure propagation

它不会自己展开子图。调度器只判断父图 GraphUnit 节点是否 ready。真正启动子图是在执行层。

这层职责划分是合理的：

```text
Scheduler 只决定父图节点是否可执行
Runtime Loop 才负责 graph_unit 节点如何执行
```

### 5.7 Runtime Loop 启动子图

运行层会识别 GraphUnit：

- `langgraph_coordination_runtime.py` 的 `_is_graph_unit_stage()`
- `_graph_unit_runtime_handle_from_contract()`
- runtime assembly 改为 `executor_type=graph_unit`

随后 `backend/api/orchestration.py` 的 `_execute_stage_request_in_background()` 判断：

```text
if executor_type == graph_unit:
    _start_graph_unit_stage_request()
```

`_start_graph_unit_stage_request()` 做的事情：

- 从 stage request 中取 GraphUnit handle
- 读取 `linked_graph_id`
- 要求子图存在
- 要求子图 `publish_state == "published"`
- 编译子图 RuntimeSpec
- 若有阻塞 issue，则拒绝启动
- 调用 `task_run_loop.start_task_graph_run()` 启动子图 run
- 把父图 handle、父阶段、父节点、输入包写进子图 initial inputs 和 diagnostics
- 可自动启动子图 initial stage

这一步证明父子图已经有真实运行路径。

### 5.8 子图完成后回交父图

父图继续执行时，`continue-current-stage` 会调用：

```text
_latest_unconsumed_graph_unit_child_result()
```

它会查同一 session 下的 task runs，筛选：

- `graph_unit_child_run == True`
- parent coordination run 匹配
- parent stage id 匹配
- request id / idempotency key 匹配
- 尚未 committed

如果子图完成，会生成：

- `graph_unit_committed_output_packet`

如果子图失败或阻塞，会生成：

- `graph_unit_committed_failure_packet`

然后父图把这个 packet 包装成当前 GraphUnit 节点的 `task_result_ready`，调用：

```text
resume_from_task_result()
```

最后 `_mark_graph_unit_child_output_packet_committed()` 标记子图 output packet 已被父图消费，防止重复消费。

这是父子图闭环里最重要的一段。它说明接续不仅包括断线重连，也包括父子图之间的进程交接。

## 6. 当前已经做对的地方

### 6.1 contract_bindings 已成为后端主路径

后端保存、编译、manifest、runtime assembly 都已经在读取 `contract_bindings`。旧字段更多是迁移兼容与冲突检测来源。

### 6.2 RuntimeSpec 直接从 TaskGraphDefinition 编译

`compile_task_graph_definition_runtime_spec()` 的注释已经说明它不再从 legacy coordination view 派生，而是直接编译 first-class TaskGraphDefinition。

这是系统走向通用任务图的关键。

### 6.3 GraphUnit 已有可运行链路

GraphUnit 不只是前端图标：

- RuntimeSpec 中存在 graph_unit runtime node。
- ContractManifest 中存在 graph_unit handoff contract。
- ExecutionPackage 中存在 graph_unit execution plan。
- Runtime Loop 中存在 graph_unit executor。
- Orchestration API 中存在启动子图 run 和回交父图 packet 的逻辑。

### 6.4 ExecutionPackage 是正确方向

ExecutionPackage 把标准视图、运行规格、契约清单、调度影子态、节点装配、GraphUnit 子图计划和对象追踪聚合在一起。

这正是图工作台应该依赖的真实事实源。

### 6.5 前端工作台结构已经接近目标

目前前端已经有：

- 左侧图层关系：`TaskGraphGraphLayerRail`
- 中间画布：`TaskGraphComposableCanvas`
- 右侧对象编辑台：`TaskGraphObjectInspector`
- 统一契约编辑：`TaskGraphContractBindingInspector`
- 发布执行包面板：`TaskGraphExecutionPackagePanel`
- 底部执行包追踪：`TaskGraphExecutionDock`

这说明前端方向是对的，但仍需要继续收敛对象模型和交互层级。

## 7. 当前不完整和不自然的地方

### 7.1 GraphUnit 保存态仍偏派生

用户在前端看到的是图节点，但实际保存主路径是：

```text
metadata.timeline_blocks[]
```

这会导致三个问题：

- 用户理解的是“图节点”，代码来源却是“时序块”。
- GraphUnit 的接口、端口、版本、交接契约分散在 timeline block 和 nested runtime plan 中。
- 后续如果要让“节点是图、图是节点”更自然，保存态需要有更明确的 GraphUnit 对象协议。

### 7.2 ComposableGraphView 仍是 shadow/overlay

`build_composable_graph_view()` 的 diagnostics 会显示：

```text
read_only_shadow_model
metadata_overlay_shadow_model
```

这说明可组合图视图仍是派生视图，不是完整主编辑模型。

这个问题不是 bug，但它限制了工作台的自然度。用户编辑“单元、接口、端口边、嵌套运行”时，最终要么回写 legacy graph edge，要么写入 metadata overlay。长期会增加理解成本。

### 7.3 父子图接口协议还不够强

现在 GraphUnit 有：

- input_port_id
- output_port_id
- handoff_contract_id

但它们还没有形成强校验的接口协议：

- 父图输出端口是否满足子图入口契约
- 子图输出是否满足父图下游输入契约
- 父子图 memory/artifact 可见性是否可被静态检查
- version_ref 是否真的锚定到发布版本
- detach_policy 在执行时的行为是否全部兑现

### 7.4 子图执行计划能预检，但不是强工作流入口

ExecutionPackage 已能返回 `graph_unit_execution_plans`，但前端目前更多是在发布页和底部 Dock 展示摘要。

还缺一个“从图节点直接看子图执行包”的强入口：

```text
选中 GraphUnit
  -> 右侧看到父图 runtime node
  -> 看到 handoff contract
  -> 看到 child execution plan
  -> 看到 child runtime spec / manifest / scheduler 摘要
  -> 一键进入子图工作台
  -> 一键定位子图阻塞 issue
```

### 7.5 Scheduler 对边时序语义仍有支持边界

Scheduler 已支持不少语义：

- phase gate
- sequence gate
- failure propagation
- handoff ack
- artifact requirement
- wait policies

但不是所有 UI 可配置项都完全运行化。比如某些 temporal semantics 现在可能只是被保存或部分诊断。

因此前端必须继续保留“运行支持矩阵”，并且不能把未兑现策略显示成已生效能力。

### 7.6 父子图递归和循环保护还不够完整

ExecutionPackage 当前检查了直接自引用：

```text
linked_graph_id == parent_graph_id
```

但还需要递归层面的循环保护：

```text
A -> B -> C -> A
```

未来如果多层父子图越来越常见，这必须成为发布包级阻塞诊断。

### 7.7 版本锚目前是字段，不是版本对象

`version_ref` 当前更多是字符串。运行时要求子图 `publish_state == "published"`，但没有看到完整的版本快照锚定机制。

风险是：

- 父图发布后，子图被修改，父图运行引用的实际子图发生漂移。
- `draft/published/v1` 这种字符串缺少统一解析规则。

### 7.8 父子图输入输出 packet 还需要契约化可视化

运行层已经有 output packet，但前端工作台还没有把 packet 协议变成可视化对象。

用户需要看到：

- 父图给子图什么输入包
- 子图只看到哪些上下文
- 子图提交什么 output packet
- 父图下游能看到哪些 refs
- 哪些正文被隔离，哪些摘要被公开

这是防止上下文污染的关键。

## 8. 目标设计

### 8.1 GraphUnit 作为标准对象

目标不是把所有 timeline block 改掉，而是建立清晰对象协议：

```text
GraphUnitDefinition
  graph_unit_id
  title
  parent_graph_id
  linked_graph_id
  version_ref
  phase_id
  sequence_index
  interface
  port_mapping
  contract_bindings
  isolation_policy
  visibility_policy
  detach_policy
  runtime_policy
```

短期可以继续保存到 `metadata.timeline_blocks[]`，但前端和后端应把它视为 GraphUnit object，而不是把用户暴露给“时序块字段”。

中期可以新增 `metadata.graph_units[]` 或正式字段，但必须迁移谨慎：

```text
timeline_blocks 兼容读取
graph_units 主路径保存
compiler 统一归一为 nested_runtime_plans
```

### 8.2 图接口协议

每个任务图都应该有图级接口：

```text
GraphInterface
  input_ports[]
  output_ports[]
  memory_visibility_policy
  artifact_visibility_policy
  runtime_state_policy
  version
```

GraphUnit 引用子图时，应校验：

```text
parent output/input
  -> GraphUnit input port
  -> child graph input interface
  -> child graph output interface
  -> GraphUnit output port
  -> parent downstream edge
```

这会让“图是节点，节点是图”真正靠接口连接，而不是靠 `linked_graph_id` 字符串连接。

### 8.3 父子图交接协议

GraphUnit handoff contract 必须统一为：

```text
GraphUnitHandoffContract
  handoff_contract_id
  parent_input_packet_schema
  child_initial_input_schema
  child_output_packet_schema
  parent_resume_result_schema
  artifact_ref_policy
  memory_visibility_policy
  failure_packet_schema
```

现有 `CompiledGraphUnitHandoffContract` 可以作为基础，但要加强输入输出 packet 的可视化和校验。

### 8.4 版本锚协议

`version_ref` 需要从普通字符串升级为可解析版本锚：

```text
VersionAnchor
  mode: published | immutable_snapshot | semantic_version | draft_for_debug
  ref: string
  graph_hash
  created_at
  source_publish_id
```

发布父图时，GraphUnit 应记录子图版本快照摘要。运行时如果子图当前版本和父图锚点不一致，应给出明确诊断。

### 8.5 嵌套运行隔离协议

GraphUnit 需要明确四类隔离：

- task run 隔离：子图是独立 task_run。
- memory 隔离：子图只能读父图通过 input packet 暴露的 refs。
- artifact 隔离：子图产物先进入子图作用域，只有 commit packet 暴露给父图。
- runtime state 隔离：父图只能看到 run handle、status、committed output，不直接窥视子图中间态，除非策略允许。

默认策略应是：

```text
isolated_per_nested_run
committed_only
explicit_refs_only
preserve_version_anchor
```

### 8.6 发布执行包成为强入口

ExecutionPackage 应继续扩展为：

```text
TaskGraphExecutionPackage
  standard_view
  runtime_spec
  contract_manifest
  scheduler_state
  node_runtime_assemblies
  graph_unit_execution_plans
  object_trace_index
  recursive_graph_plan_index
  version_anchor_index
  interface_compatibility_report
  packet_contract_report
```

前端任何对象选中后，都应该能从 execution package 追到最终运行事实。

### 8.7 父子图导航

工作台交互应支持：

```text
父图画布
  选中 GraphUnit
    右侧编辑边界
    底部显示 child execution plan
    点击进入子图
  子图工作台
    顶部显示父图路径
    可返回父图 GraphUnit
```

左侧图层关系应该有路径栈：

```text
父图 graph.story
  GraphUnit: 设计阶段
    子图 graph.design
      GraphUnit: 人设设计
        子图 graph.character_design
```

这比把所有图摊在一张巨大拓扑里更符合用户要求。

- 只检查直接自引用，不管多层循环。
- 把图工作台重新拆回多个互相抢职责的页面。

## 12. 我对系统的最终理解

这个系统真正要做的是：

```text
用任务图定义长任务结构。
用节点定义运行位置。
用任务定义执行动作。
用边定义交接协议。
用契约约束输入、输出、记忆、产物、验收和运行。
用时序分层组织节点何时激活。
用 GraphUnit 把一个任务图作为另一个任务图中的运行节点。
用 ExecutionPackage 把编辑态配置变成发布前可审计的运行事实。
用 Runtime Loop 把 ready 的节点变成真实执行。
用 output packet 把子图结果提交回父图。
```

父子图的核心不是“把三张图拼起来显示”，而是“父图把子图当成一个有接口、有版本、有隔离、有交接契约、有运行状态、有提交包的节点”。只要这个协议稳定，图就可以无限模块化：大图是图，小图是图，图可以成为节点，节点也可以展开成图。

当前代码已经走到了可运行阶段，但还没有完全走到自然编辑阶段。下一步要做的是把 GraphUnit 的对象协议、接口协议、版本锚、执行包追踪和工作台导航补齐。这样任务图系统才能真正支撑通用的多阶段、可组合、可断开、可接续的长任务运行。

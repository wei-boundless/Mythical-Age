# 灵魂系统讨论

日期：2026-04-28  
定位：记录灵魂系统讨论过程，不作为正式方案、不作为实施计划。  
当前主题：投影作为任务承载载体时，静态部分与动态部分如何划分。

---

## 1. 当前问题

我们已经把灵魂系统推到了“投影系统”这一层，但投影到底应该干什么还没有完全清晰。

当前要先讨论清楚的问题是：

```text
投影如果是 agent 承载任务的载体，
那么它内部哪些东西应该是静态配置，
哪些东西应该是运行时动态生成，
哪些东西应该来自工作记忆的短期变化，
哪些东西不能归给投影。
```

这一步先只讨论，不急着修改正式方案文档。

---

## 2. 已经形成的共识

当前基本共识：

```text
SoulProfile
  是灵魂本体模板。
  第一阶段只负责语言风格、表达方式、身份质感和未来状态记忆锚点。

Projection
  是 agent 承载任务时的载体。
  它决定 agent 以什么姿态、角色状态、表达方式和注意力结构进入任务。

TaskSystem
  定义任务真相。
  告诉大模型当前要完成什么、按什么契约完成、输出什么才算完成。

SkillSystem
  提供方法资源。
  告诉大模型当前任务可以采用哪些方法片段。

OperationSystem / ResourcePolicy
  定义执行边界。
  决定工具、文件、记忆、副作用是否可见或可用。
```

一句话：

```text
Task 定义做什么。
Skill 定义用什么方法。
OperationSystem 定义能不能触达资源。
Projection 定义由谁以什么姿态承载。
```

---

## 3. 需要厘清的核心问题

### 3.1 投影作为载体，静态部分是什么

静态部分可能包括：

```text
投影名称
来源 SoulProfile
默认角色姿态
默认表达风格
默认注意力偏好
默认工作风格预设
适合承载的任务族推荐
适合呈现的 skill/tool 摘要风格
风险提示
```

这些内容比较适合放在：

```text
ProjectionCard / DefaultProjection
```

它们像“投影预设”，不应该随某一次任务自动变成任务事实。

### 3.2 投影作为载体，动态部分是什么

动态部分可能包括：

```text
当前任务目标
当前 TaskRuntimeInstruction
当前 TaskRole
当前 SkillRuntimeView
当前 ToolRuntimeView
当前 MemoryPolicy summary
当前 OutputContract
当前阶段上下文
当前资源边界
```

这些内容不应该保存在 ProjectionCard 里。

它们应该在任务运行时进入：

```text
SoulProjectionRequest
SoulRuntimeView
PromptManifest
```

### 3.3 工作记忆如何影响投影

投影可以成长，但暂时只考虑短期成长，不讨论长期人格改写。

工作记忆可能影响：

```text
当前表达密度
当前语气温度
当前解释节奏
当前用户偏好
当前任务阶段的注意力偏向
最近成功或失败的交互方式
```

可能的中间层：

```text
WorkingMemoryView
  -> ProjectionStateCandidate
  -> SoulProjectionBuilder
  -> ProjectionRuntimePosture
```

需要守住：

```text
ProjectionState 不直接写回 SoulProfile。
ProjectionState 不授予工具权限。
ProjectionState 不改写任务流程。
ProjectionState 不替代 TaskContract。
```

### 3.4 投影不能管什么

投影不应该管理：

```text
TaskDefinition
TaskGraph
TaskMethodContract
SkillRegistry
ToolRegistry
ResourcePolicy
Tool authorization
Memory write decision
Runtime directive
CommitGate
```

投影可以“呈现”这些东西的摘要，但不能成为这些东西的事实来源。

---

## 4. 当前讨论中的暂定模型

可以先把投影拆成三层：

```text
ProjectionCard
  静态投影配置。
  保存默认姿态、表达风格、角色倾向和推荐标签。

ProjectionState
  工作记忆驱动的短期状态。
  保存当前任务中的临时表达调整和注意力偏向。

SoulProjection / SoulRuntimeView
  当前任务运行时生成的投影视图。
  把静态配置、短期状态、任务契约、skills/tools 摘要装配成给模型看的内容。
```

它们的关系：

```text
SoulProfile
  -> ProjectionCard
  -> ProjectionStateCandidate
  -> SoulProjectionBuilder
  -> SoulRuntimeView
```

但中间还必须接受任务系统和操作系统输入：

```text
TaskRuntimeInstruction
SkillRuntimeView
ToolRuntimeView
MemoryPolicy summary
OutputContract
ResourcePolicy summary
```

所以更完整的关系是：

```text
SoulProfile + ProjectionCard + ProjectionStateCandidate
  + TaskRuntimeInstruction
  + SkillRuntimeView
  + ToolRuntimeView
  + MemoryPolicy summary
  + OutputContract
  -> SoulRuntimeView
```

后续需要继续讨论：

```text
7. 前端投影管理页面应该展示静态投影，还是也展示当前 runtime projection。
```

## 6. Skills 读取的新判断

当前形成一个重要判断：

```text
skills 应该根据任务来分层和读取，
而不是让大模型先阅读一个总览式 skills 图。
```

原因是：

```text
如果任务理解已经歪了，
即使把全部 skills 给模型，
模型也只是在错误任务理解上选择更多错误工具和方法。
```

所以 skills 的入口不应该是：

```text
全部 skills catalog
  -> 模型自己理解任务
  -> 模型自己挑选 skills
```

更合理的入口是：

```text
TaskUnderstanding
  -> TaskDefinition / TaskContract
  -> SkillRequirementRefs
  -> SkillScopeResolver
  -> SkillRuntimeView
  -> AgentRuntimePrompt
```

也就是说：

```text
先理解任务，
再根据任务层级和任务契约选择 skill 视图，
最后只把当前任务必要的 skill 摘要给模型。
```

### 6.1 Skills 按任务层级读取

可以先按任务层级分：

```text
BasicTaskSkill
  给基本任务使用。
  例如信息搜寻、文档读取、结果检查、局部改写。

CompositeTaskSkill
  给复合任务使用。
  例如搜索后总结、读取后改写、研究后更新本地文档。

CoordinationTaskSkill
  给协调任务使用。
  例如多 agent 分工、交接、审查、合并、闭环。
```

读取原则：

```text
BasicTask 只读与当前 basic task 相关的 skills。
CompositeTask 只读它自己的组合 skill，以及被引用 basic tasks 的必要 skill 摘要。
CoordinationTask 只读协调协议 skill，以及各 ExecutionSeat 所需的 skill 摘要。
```

### 6.2 Skills 不是任务理解的替代品

需要守住：

```text
skills 不能替代 TaskUnderstanding。
skills 不能替代 TaskDefinition。
skills 不能替代 TaskContract。
skills 不能替代 ResourcePolicy。
```

skills 的作用是：

```text
在任务已经被理解之后，
给大模型提供当前任务可采用的方法片段。
```

所以：

```text
任务系统负责先把“要做什么”说清楚。
Skill 系统负责再把“可以怎么做”说清楚。
```

### 6.3 对 prompt 的影响

最终给模型看的不应该是：

```text
SkillCatalogOverviewPrompt
```

而应该是：

```text
SkillRuntimeView
  当前任务可采用的方法
  适用条件
  输入输出边界
  禁止用途
  可能需要的工具摘要
```

这可以减少：

```text
无关技能噪声
错误任务理解后的错误技能选择
技能 prompt 和任务 prompt 互相抢权
模型在执行前花太多上下文理解技能目录
```

## 7. 降噪链路的新判断

当前又形成一个更清楚的总体判断：

```text
任务系统、skills 系统、投影系统，
本质上都应该服务于给大模型“降噪”。
```

不是把更多设定塞给模型，而是：

```text
先把不必要的信息挡在模型外面，
只把当前任务真正需要的那部分说明交给模型。
```

可以先把这条链路理解成三层降噪：

```text
第一层：任务系统降噪
  先判断当前到底是什么任务。
  把用户请求收束成清晰的 TaskUnderstanding / TaskContract。

第二层：skills 系统降噪
  不给模型看完整 skill 总览。
  只给当前任务需要的 SkillRuntimeView。

第三层：投影系统降噪
  不把所有表达姿态、工作风格、注意事项一起塞进去。
  只让当前任务最合适的投影承载任务，
  并以合适的姿态呈现 task / skills / tools 摘要。
```

所以可以先固定一句原则：

```text
TaskSystem 负责先把任务理解对。
SkillSystem 负责只暴露当前任务可采用的方法。
ProjectionSystem 负责用最合适的承载姿态进一步压缩噪声。
```

### 7.1 投影系统为什么也是降噪层

投影系统的价值不只是“换风格”，而是进一步帮助模型聚焦。

比如同一个任务，如果直接把：

```text
全部任务约束
全部 skills 摘要
全部 tools 摘要
全部表达设定
```

一起丢给模型，模型仍然会噪声很大。

而投影系统可以进一步做这些事：

```text
决定当前任务最合适的表达密度
决定当前应该强调哪些注意事项
决定 skill 摘要应该怎样呈现
决定 tool 摘要应该怎样呈现
决定当前任务最适合的承载姿态
```

也就是说：

```text
Projection 不是增加一层设定。
Projection 是把已经被任务系统和 skills 系统筛过的信息，
再以最适合当前任务的姿态交给模型。
```

### 7.2 这对后续设计的影响

如果这个判断成立，那么后面设计时要优先考虑：

```text
1. 如何减少模型无关上下文。
2. 如何减少错误任务理解后看到的错误 skills。
3. 如何减少投影层携带过多静态设定。
4. 如何让每一层都只给模型当前必要的信息。
```

换句话说：

```text
任务系统不是任务分类器，而是第一层降噪器。
skills 系统不是技能目录，而是第二层降噪器。
投影系统不是人格装饰层，而是第三层降噪器。
```

### 7.3 当前进一步确认

现在可以把链路再固定得更明确一些：

```text
不是先把所有 skills 摆给大模型，
再让它自己理解任务并自己选择。

而是先由任务系统判断当前属于什么任务类型，
再按这个任务类型给出当前可用的 skills，
最后再由投影系统把这些已经筛过的信息，
以最适合当前任务的姿态呈现给模型。
```

也就是说：

```text
任务类型
  -> 决定 skill scope
  -> 决定 tool scope
  -> 决定可采用的投影姿态
  -> 决定最终给模型看的 prompt 结构
```

这里要守住两个原则：

```text
1. skills 的可见范围由任务来裁剪，而不是由模型先自由浏览。
2. 投影系统的职责不是再加设定，而是在任务和 skills 已经裁剪之后继续降噪。
```

因此当前可以先把三者关系记成：

```text
TaskSystem
  先决定当前是什么任务。

SkillSystem
  再决定这个任务可以看到哪些方法。

ProjectionSystem
  再决定这些任务信息和方法信息，
  该以什么姿态、什么表达密度、什么注意力重点交给模型。
```

```text
投影运行时动态结构。
ProjectionState 的生命周期。
投影与任务系统、Skill 系统、操作系统的最终接口。
skills 根据任务层级读取的方式。
任务系统 / skills 系统 / 投影系统作为三级降噪链路的最终接口。
```

## 9. 契约整理是否应该收束到任务系统

当前进一步形成一个很关键的判断：

```text
契约的整理权，应该尽量收束到任务系统里面。
```

这里说的“契约”，主要不是指所有系统都完全不再有自己的配置，
而是指：

```text
凡是要真正交给大模型执行时阅读的任务说明，
应尽量由任务系统负责归总、裁剪、装配和定序。
```

### 9.1 为什么要收束到任务系统

如果不收束，会很容易出现下面这种情况：

```text
灵魂系统给一套设定，
投影系统给一套任务态说明，
skills 再给一套方法说明，
操作系统再给一套资源边界，
最后 agent 看到的是几份来源不同、层级不同、甚至互相抢权的 prompt。
```

这样的问题不是“信息不够”，而是：

```text
信息来源太散，
优先级不清，
谁定义事实、谁只是摘要、谁只是姿态，边界混了。
```

而任务系统天然最适合作为整理中心，因为它本来就拥有：

```text
当前任务是什么
当前任务要完成什么
当前任务输入输出是什么
当前任务允许什么方法
当前任务允许什么资源
当前任务由谁承载
当前任务什么算完成
```

所以可以先固定一句：

```text
任务系统不是只定义任务名目，
而是当前运行时契约的主整理者。
```

### 9.2 但这不等于任务系统拥有所有事实

这里也要防止另一种混乱：

```text
契约收束到任务系统
!= 任务系统拥有所有原始事实
```

更准确地说应该是：

```text
任务系统负责把各系统提供的事实，
整理成一份给模型看的运行时契约。
```

各系统的原始职责仍然分开：

```text
TaskSystem
  拥有任务真相与任务契约骨架。

SkillSystem
  拥有方法资源事实。

OperationSystem
  拥有资源边界与权限事实。

Soul / ProjectionSystem
  拥有表达姿态与承载姿态事实。
```

任务系统做的是：

```text
以任务为中心，
把这些分散事实整理成当前任务的运行时视图。
```

### 9.3 适合收进任务系统的契约内容

当前看来，下面这些内容适合收束到任务系统来整理：

```text
TaskGoal
TaskMethodContract
InputContract
OutputContract
CompletionCriteria
FailureHandling
SkillScopeRefs
ToolScopeRefs
ProjectionRequirement
AgentRole
StageBoundary
Guardrails
ConflictPolicy
```

也就是说，大模型最后看到的核心说明，应该尽量是：

```text
这次任务要做什么
按什么方法做
能用什么
不能用什么
由谁承载
完成标准是什么
失败时怎么办
```

### 9.4 不应该收进任务系统核心的东西

下面这些东西不应该被任务系统吞掉成自己的内部事实：

```text
灵魂本体设定
投影静态卡片全部内容
skill 的完整正文
tool 的完整实现说明
resource policy 的底层规则细节
working memory 的原始内容
```

任务系统应该只引用、摘要、裁剪，而不是把它们全部复制一份。

所以更稳的关系应该是：

```text
TaskSystem
  维护 ContractAssembly

SkillSystem
  提供 SkillRuntimeView

OperationSystem
  提供 ToolRuntimeView / ResourcePolicySummary

ProjectionSystem
  提供 ProjectionRuntimePosture
```

然后由任务系统把这些装成：

```text
TaskPromptContract
```

最后再交给投影系统进一步收束成：

```text
SoulRuntimeView / PromptManifest
```

### 9.5 这样做的直接好处

如果这条原则成立，后面会立刻变清楚很多：

```text
1. 契约冲突时，以任务系统整理结果为准，而不是让多个 prompt 自己打架。
2. 投影系统不用再偷偷夹带任务流程。
3. skills 不再以总览目录形式暴露。
4. tools 不再直接作为开放资源面裸露给模型。
5. 大模型读到的是当前任务的最小可执行说明，而不是系统全景图。
```

### 9.6 当前可先固定的口径

现在可以先把口径收成这样：

```text
任务系统负责整理契约。
skills 系统负责提供方法片段。
操作系统负责提供资源边界。
投影系统负责提供承载姿态。

最后真正交给大模型看的任务运行说明，
尽量都先收束成任务系统的一份结构化契约，
再由投影系统做最后一层低噪声呈现。
```

## 10. skills 与投影的进一步边界

当前又可以把 skills 和投影的边界收得更清楚一些：

```text
skills 负责流程与方法。
投影负责风格、姿态与额外补充。
```

这句话很重要，因为它直接决定后面什么东西该放进哪个系统。

### 10.1 skills 负责什么

当前更适合把 skills 理解成：

```text
给大模型的任务实现说明书中的“方法部分”。
```

也就是说，skills 主要回答：

```text
这个任务通常按什么流程完成。
先做什么，后做什么。
每一步关注什么。
每一步输入输出如何衔接。
常见错误怎么检查。
什么情况下应该停下、回退或重试。
```

因此 skills 更偏静态：

```text
它不是临场情绪或表达变化。
它不是当前人格姿态。
它不是短期工作记忆的波动。
它更像某类任务稳定可复用的方法脚手架。
```

可以先固定：

```text
skills = 静态的方法流程说明
```

### 10.2 投影负责什么

投影更适合被理解成：

```text
给大模型的任务实现说明书中的“承载部分”与“动态补充部分”。
```

它主要回答：

```text
这次任务由谁来承载。
用什么语气、什么表达密度、什么角色姿态进入任务。
当前应该强调谨慎、推进、审查、陪伴，还是解释。
当前应该怎样呈现已经筛过的 task / skills / tools 信息。
基于当前上下文和工作记忆，需要做哪些轻量动态补充。
```

所以投影更偏动态：

```text
它会受当前任务阶段影响。
会受工作记忆影响。
会受用户当前互动状态影响。
会影响模型看到信息的组织方式和强调重点。
```

可以先固定：

```text
projection = 动态的承载姿态与补充说明
```

### 10.3 二者为什么本质上都像说明书

这个判断也成立：

```text
skills 和投影，
本质上都是给大模型看的任务实现说明书。
```

只是它们负责的部分不同：

```text
skills
  负责“怎么做”的静态说明。

projection
  负责“由谁以什么姿态来做”的动态说明。
```

所以可以把它们理解成两种不同层级的说明：

```text
TaskContract
  说明这次要做什么。

SkillRuntimeView
  说明这次可以按什么流程和方法做。

ProjectionRuntimePosture
  说明这次这些说明应当以什么姿态、什么强调重点、什么表达方式被承载。
```

### 10.4 这样划分后的直接好处

如果按这个边界来设计，会立刻减少很多混乱：

```text
1. skills 不再偷偷承担人格或风格设定。
2. 投影不再偷偷承担任务流程。
3. 静态方法和动态姿态可以分别管理。
4. 同一套 skills 可以被不同投影承载。
5. 同一投影也可以承载不同任务，只是看到的 skills 不同。
```

一句话就是：

```text
skills 解决“怎么做”。
投影解决“怎么承载这些做法”。
```

### 10.5 当前还要守住的一点

虽然二者都像“给模型看的说明书”，但它们仍然不该直接并列原文拼接。

更稳的链路仍然应该是：

```text
TaskSystem
  先整理当前任务契约。

SkillSystem
  提供当前任务所需的静态方法流程说明。

ProjectionSystem
  提供当前任务所需的动态承载姿态与补充说明。

TaskSystem / PromptAssembler
  再把这些整理成低噪声的运行时说明。
```

所以这里的关键不是把三者都变成 prompt 文本，
而是：

```text
把三者都变成结构化说明来源，
最后再统一装配成模型真正读取的运行时说明。
```

## 11. 对基本任务说明结构的初步看法

如果继续沿着当前判断往下推，我的看法是：

```text
基本任务不应该一上来就带太多段说明。
越是 basic task，越应该把给模型看的说明压缩到最小必需集。
```

因为 basic task 的目标不是“展示系统多完整”，而是：

```text
让单个 agent 在一个清楚边界里，
稳定完成一个最小任务。
```

所以对基本任务来说，最关键的是把三类说明分开：

```text
TaskContract
  这次到底要做什么。

SkillRuntimeView
  这类基本任务通常怎么做。

ProjectionRuntimePosture
  这次应该以什么姿态承载这些说明。
```

### 11.1 我认为基本任务的静态 skill 说明最少只需要五段

对 basic task 来说，skills 不应该写成大而全的手册，
而应该只保留最稳定、最可复用的流程说明。

我目前认为最小静态结构可以先是：

```text
1. purpose
   这个基本任务是用来解决什么问题的。

2. preconditions
   在什么前提下适合使用这个任务。
   什么情况下不该使用。

3. procedure
   这个任务通常按什么顺序执行。
   先看什么，再做什么，最后产出什么。

4. checks
   这个任务执行时应检查哪些错误或遗漏。

5. fallback
   信息不足、结果冲突、工具失败时，
   应该怎样收缩、回退、重试或请求补充。
```

我倾向于先不把更多东西塞进 basic skill。

例如下面这些，很多时候可以晚一点再加：

```text
大量示例
复杂分支树
冗长背景知识
过细的风格提示
跨任务复合流程
```

原因是：

```text
basic task 的 skill 重点是提供稳定方法骨架，
不是给模型一个巨大的操作手册。
```

### 11.2 这五段里，真正核心的是 procedure + checks

如果还要再压缩，我会认为：

```text
purpose
procedure
checks
```

是最硬核、最不能缺的三段。

因为：

```text
purpose
  解决“为什么调用这个任务”。

procedure
  解决“这个任务通常怎么完成”。

checks
  解决“如何避免模型走偏”。
```

而：

```text
preconditions
fallback
```

更像是让 basic task 更稳、更不容易误用的护栏层。

所以工程上可以理解成：

```text
purpose / procedure / checks
  是 basic skill 的主干。

preconditions / fallback
  是 basic skill 的稳定器。
```

### 11.3 我认为基本任务的动态投影补充先只需要四段

投影是动态层，所以不应该重复讲流程，
而应该只补“这次如何承载这些流程”。

我认为 basic task 第一版的 projection 动态补充，
先只需要四段就够：

```text
1. role_posture
   这次以什么角色姿态承载任务。
   例如陪伴式、审查式、执行推进式、证据收集式。

2. expression_policy
   这次的语言密度、解释深度、语气温度、是否主动拆解。

3. attention_bias
   这次要特别关注什么。
   例如证据、准确性、用户体验、风险、格式一致性。

4. runtime_notes
   来自工作记忆或当前上下文的轻量动态补充。
   例如用户偏好、最近失败点、当前阶段特别提醒。
```

我暂时不建议在 projection 里直接放：

```text
任务流程步骤
skill 细节正文
tool 完整说明
资源授权规则
```

因为那些会让 projection 再次膨胀成第二套任务系统。

### 11.4 我对基本任务装配顺序的看法

如果按当前边界，我认为一个 basic task 给模型看的东西，
应该按下面顺序装配：

```text
1. TaskContract
   明确这次任务要做什么、产出什么、什么算完成。

2. SkillRuntimeView
   给出这类任务的静态方法流程说明。

3. Tool / Resource Summary
   说明当前任务允许接触哪些操作面。

4. ProjectionRuntimePosture
   决定以上说明以什么姿态、什么强调重点交给模型。
```

这里的重点是：

```text
投影在后，不在前。
```

因为：

```text
先有任务真相，
再有方法说明，
再有资源边界，
最后才决定怎样承载这些信息。
```

这也符合前面已经形成的判断：

```text
任务系统第一层降噪。
skills 系统第二层降噪。
投影系统第三层降噪。
```

### 11.5 我的总体倾向

如果现在就要我拍一个方向，我会选：

```text
basic task = 小而硬的任务契约
           + 小而稳的静态 skill 流程
           + 小而轻的动态 projection 补充
```

也就是说：

```text
任务系统负责把骨架立住。
skills 系统负责给稳定流程。
投影系统负责给动态姿态。
```

先不要让 basic task 一开始就变成“半个复合任务”。

真正复杂的分支、阶段切换、子执行席位，
应该留给 composite task 去承载。

## 12. 对基本任务原语的初步判断

如果继续往下收，我目前的看法是：

```text
basic task 原语不要按抽象概念分，
而要按单个 agent 真正能独立完成的最小任务动作来分。
```

也就是说，我们不是先问：

```text
系统里有哪些漂亮分类名词
```

而是先问：

```text
如果只给一个 agent、一个投影、一个任务契约，
它最小能稳定完成哪些类型的任务动作。
```

### 12.1 我认为 basic task 原语的判定标准

一个任务要不要算 basic primitive，我倾向于先看这五点：

```text
1. 单 agent 能独立完成。
2. 输入输出边界清楚。
3. 允许的 skills / tools 容易裁剪。
4. 完成标准可以检查。
5. 可以被 composite task 稳定复用。
```

如果一个“任务”需要：

```text
多个阶段强切换
大量上下文编排
复杂回路
多个 agent 协作
```

那它通常就已经不适合继续叫 basic primitive 了。

### 12.2 我当前更认可的 basic primitives

我目前更倾向于把 basic task 原语先收成 6 类。

#### 1. Understand / Clarify

作用：

```text
把用户请求收束成清楚任务目标，
补齐缺失条件，
明确边界和完成标准。
```

它解决的是：

```text
当前到底要做什么。
哪里还不清楚。
是否需要继续追问或重述。
```

适合的输出：

```text
task_goal
constraints
missing_info
completion_criteria
```

这是很多任务的入口原语。

#### 2. Search / Gather

作用：

```text
为任务搜集候选信息源、文件、网页、知识片段或证据入口。
```

它解决的是：

```text
去哪里找信息。
当前有哪些候选来源。
哪些来源可能相关。
```

这里可以再细分来源，但不一定要先拆成不同 primitive：

```text
联网搜索
知识库搜索
本地文件定位
```

第一版可以先把它们都放进 Search / Gather，
后面再由 tool scope 和 source scope 继续细分。

#### 3. Read / Extract

作用：

```text
对已经定位到的具体材料进行读取、抽取、标注和结构化整理。
```

它解决的是：

```text
材料里到底有什么。
哪些信息值得保留。
哪些信息是证据，哪些只是噪声。
```

适合的产物：

```text
facts
quotes / evidence refs
structured notes
field extraction
```

我倾向于把 Search 和 Read 分开，
因为“找到东西”和“消化东西”是两种不同原语。

#### 4. Transform / Compose

作用：

```text
基于已有输入，把信息变成新的表达或产物。
```

比如：

```text
总结
改写
翻译
提纲
回答组织
文档草稿生成
结构重排
```

它解决的是：

```text
拿到材料后，如何把它加工成用户真正要的结果。
```

这是最典型的“生成型基本任务”。

#### 5. Operate / Apply

作用：

```text
把结果真正作用到本地环境、文档、代码、文件或系统对象上。
```

比如：

```text
修改代码
更新文档
整理文件
执行命令
调用本地工具
```

它和 Transform / Compose 的区别是：

```text
Transform 更偏内容加工。
Operate 更偏把任务结果施加到外部对象上。
```

这一步很关键，因为很多 agent 不是只需要“说出来”，
而是需要“真的做下去”。

#### 6. Verify / Review / Correct

作用：

```text
检查当前结果是否满足任务契约，
发现错误、遗漏、风险，并在必要时修正。
```

它解决的是：

```text
结果对不对。
有没有漏。
有没有风险。
是否需要回退、补查或修正。
```

这是非常重要的 basic primitive，
因为如果没有它，前面所有原语都容易只停在“产出过一个结果”。

### 12.3 我为什么暂时不把 chat 单独当 primitive

这一点我目前是比较明确的：

```text
chat 更像交互表面，
不一定适合作为 basic task primitive 本体。
```

因为：

```text
chat 可以承载理解任务、
解释任务、
输出结果、
陪伴式澄清、
甚至发起复合任务。
```

也就是说：

```text
chat 是任务进入和结果返回的通道，
但不一定是最小任务原语本身。
```

如果后面一定要保留“聊天任务”，
我会更倾向于把它理解成：

```text
一种任务模式 / interaction shell
```

而不是 basic primitive。

### 12.4 我为什么认为这 6 类够做第一版底座

因为很多更复杂的任务，其实都可以由这 6 类往上搭：

```text
研究任务
  = Understand
  + Search
  + Read
  + Compose
  + Verify

本地文档处理
  = Understand
  + Read
  + Compose
  + Apply
  + Verify

代码修改
  = Understand
  + Search
  + Read
  + Apply
  + Verify
```

也就是说，这 6 类更像单 agent 可复用的任务骨架。

### 12.5 当前我不建议立刻继续细拆的方向

第一版先不要急着拆出太多细原语，比如：

```text
总结原语
翻译原语
提纲原语
审稿原语
网页搜索原语
文件搜索原语
代码搜索原语
```

因为这些更像：

```text
basic primitive 的任务模板
或特化 definition
```

而不一定是 primitive 层本身。

换句话说：

```text
primitive 层先稳住大的动作类型。
definition 层再去做具体任务入口。
skill 层再给出稳定流程。
projection 层再给出动态承载。
```

### 12.6 我当前的结论

如果现在先拍一版，我会这样定：

```text
basic task primitives
  1. Understand / Clarify
  2. Search / Gather
  3. Read / Extract
  4. Transform / Compose
  5. Operate / Apply
  6. Verify / Review / Correct
```

这一版的好处是：

```text
足够少
足够稳
足够贴近 agent 真正执行动作
很容易往 composite task 继续搭
```

### 12.7 当前达成的阶段性共识

目前可以先把这一轮讨论收成下面这组共识：

```text
1. basic task primitive 按 agent 可独立完成的最小任务动作来划分，
   不按抽象概念或展示层模式来划分。

2. chat 更适合作为 interaction shell / task mode，
   而不是 basic primitive 本体。

3. basic primitive 第一版先收成六类：
   Understand / Clarify
   Search / Gather
   Read / Extract
   Transform / Compose
   Operate / Apply
   Verify / Review / Correct

4. primitive 层负责定义大动作类型。
   task definition 层负责定义具体任务入口。
   skill 层负责定义静态方法流程。
   projection 层负责定义动态承载姿态。

5. 复合任务以后就建立在这些 basic primitives 之上，
   而不是重新发明一套并列底层动作。
```

这意味着后续讨论可以继续往前推进到：

```text
每个 primitive 的最小字段集是什么。
每个 primitive 默认绑定哪些 skill 结构。
每个 primitive 对投影提出哪些最小 posture 要求。
```

## 13. 原则依据与成熟框架参照

当前这套判断不是纯粹拍脑袋，它背后有比较清楚的原则依据，也能和现有成熟 agent 框架对上。

### 13.1 原则依据

当前最重要的原则依据有四条：

```text
1. 先把复杂任务拆成更小、更清楚的可执行步骤。
2. 先做单 agent 的稳定执行，再升级到多 agent 协调。
3. 方法流程与承载姿态分离。
4. 工具、资源、子 agent 都应该按任务边界裁剪，而不是全量暴露。
```

这些原则和当前主流 agent 设计的共识是吻合的。

### 13.2 Anthropic 的支持点

Anthropic 在《Building Effective Agents》中强调：

```text
有效的 agent 系统更适合使用简单、可组合的模式，
而不是一开始就上复杂框架。
```

其中提到的几种工作流：

```text
prompt chaining
routing
parallelization
orchestrator-workers
evaluator-optimizer
```

其实和我们现在讨论的层级很接近：

```text
Understand / Clarify
  对应 routing 前的任务理解与收束。

Search / Read / Transform / Operate
  对应可组合的单步执行动作。

Verify / Review / Correct
  对应 evaluator-optimizer 的检查和修正环。

Composite / Coordination
  对应 chaining、orchestrator-workers、parallelization。
```

### 13.3 OpenAI Agents SDK 的支持点

OpenAI 最新的 Agents SDK 文档和实践指南里，有几个判断和我们现在的方向高度一致：

```text
1. 先把单个 specialist 设计清楚，再谈多 agent orchestration。
2. 单 agent 往往可以先通过 tools、guardrails、hooks 和 prompt variables 承担大量任务。
3. 多 agent 适合在 prompts 太复杂、工具太拥挤、职责需要明显分离时再引入。
4. 多 agent 主要有 manager 和 handoff 两种核心组织方式。
```

这和我们现在的判断基本重合：

```text
basic task / composite task
  先在单 agent 执行席位上跑通。

coordination task
  再在此基础上引入多 agent 的 manager / handoff / merge 结构。
```

### 13.4 Claude Code 的支持点

Claude Code 的 subagent 设计也非常支持我们现在这条线。

它明确体现了：

```text
1. 子 agent 是“专门处理某类任务”的执行单元。
2. 每个子 agent 有自己的 system prompt。
3. 每个子 agent 可以有独立工具权限。
4. 子 agent 在自己的上下文窗口里工作，只把结果摘要带回主线程。
```

这和我们现在对三者边界的理解是契合的：

```text
TaskSystem
  决定当前是什么任务。

SkillSystem
  提供当前任务的方法流程说明。

ProjectionSystem
  提供当前执行席位的承载姿态。

OperationSystem
  决定这个席位到底能碰哪些工具和资源。
```

也就是说，Claude Code 的成熟实践已经证明：

```text
任务专门化
上下文隔离
工具按职责裁剪
不同执行席位的独立 prompt
```

这些都不是空想，而是已经在成熟产品里成立的设计方向。

### 13.5 LangGraph / AutoGen 的支持点

LangGraph 和 AutoGen 更像在告诉我们：

```text
复合任务和协调任务应如何往上搭。
```

LangGraph 强调：

```text
orchestrator-worker
动态 worker
structured output
graph state
```

这说明：

```text
当 basic primitives 已经稳定后，
完全可以再往上建 composite task 的阶段图和 coordination task 的多席位图。
```

AutoGen 则强调：

```text
multi-agent conversation
specialized agents
message passing
```

说明多 agent 协作层，本来就应该建立在多个专门 agent 之上，而不是把所有能力都糊成一个大 prompt。

### 13.6 当前应如何理解“有框架支持”

要注意一点：

```text
没有哪个现成框架会直接给出
Understand / Search / Read / Transform / Operate / Verify
这一模一样的 primitive 列表。
```

但成熟框架共同支持的是更底层的那些原则：

```text
单元化
可组合
按职责路由
工具裁剪
上下文隔离
阶段检查
单 agent 优先
多 agent 后置
```

所以更准确地说：

```text
我们现在不是在照抄某个框架的名词表，
而是在把多个成熟框架共同验证过的设计原则，
收束成适合我们自己 agent 系统的一套任务原语。
```

### 13.7 当前阶段可以先固定的判断

因此目前可以先放心固定：

```text
1. 我们现在的方向有明确原则依据。
2. 这些原则能和 Anthropic / OpenAI / Claude Code / LangGraph / AutoGen 的成熟设计对上。
3. 我们的 primitive 名称是本地化抽象，
   但其背后的拆分原则是成熟的。
4. 所以接下来可以继续往字段集、契约结构和绑定结构推进，
   不需要再回到“这是不是完全没有依据”的担忧上。
```

## 14. basic primitive 的最小字段集

现在可以开始定第一版字段集了。

这里我倾向于先守住一个原则：

```text
primitive 负责定义“大动作类型”的最小契约骨架，
而不是直接承载具体任务的全部绑定细节。
```

也就是说：

```text
primitive
  定义这一类任务动作天然需要什么字段。

task definition
  再在 primitive 之上补具体任务目标、skills、tools、投影绑定和限制。
```

这样可以避免一上来就把 primitive 做成“半个具体任务模板”。

### 14.1 通用骨架字段

我建议所有 basic primitives 先共用一套最小骨架：

```text
primitive_id
  原语唯一标识。

primitive_key
  原语类型键。
  例如 understand / search / read / transform / operate / verify。

title
  原语名称。

purpose
  这类原语解决什么问题。

interaction_mode
  这类原语如何与用户或上游任务交互。
  例如 direct_user / task_internal / mixed。

side_effect_mode
  是否允许对外部对象产生副作用。
  例如 none / read_only / write / mixed。

clarification_policy
  信息不足时如何处理。
  例如 never / allowed / required_when_blocked。

evidence_policy
  是否要求证据、引用、定位信息或操作回执。
  例如 none / optional / required。

input_contract
  这类原语最小输入结构。

output_contract
  这类原语最小输出结构。

completion_criteria
  什么情况算完成。

failure_contract
  失败时必须返回什么。

default_skill_sections
  该原语默认需要哪些 skill 说明段。

projection_requirements
  该原语对投影姿态提出的最小要求。

resource_profile
  该原语通常需要哪类资源面。
  这里只写资源类别，不写具体 tool id。
```

### 14.2 为什么先只定到这一层

这里要故意不把以下内容塞进 primitive：

```text
allowed_tool_ids
allowed_skill_ids
projection_id
agent_id
stage_graph
task title
具体业务目标
```

这些更适合放在：

```text
TaskDefinition
TaskBinding
CompositeTaskDefinition
```

因为 primitive 层回答的是：

```text
这一类动作本质上需要什么最小契约。
```

而不是：

```text
某一个具体任务现在绑定了什么。
```

### 14.3 我建议固定的通用字段结构

如果往数据结构上靠，我目前更认可下面这组字段块：

```text
PrimitiveIdentity
  primitive_id
  primitive_key
  title

PrimitiveContract
  purpose
  interaction_mode
  side_effect_mode
  clarification_policy
  evidence_policy
  input_contract
  output_contract
  completion_criteria
  failure_contract

PrimitiveSkillShape
  default_skill_sections

PrimitiveProjectionShape
  projection_requirements

PrimitiveResourceShape
  resource_profile
```

这样做的好处是：

```text
1. primitive 自己足够轻。
2. 后续 task definition 很容易继承。
3. skills / projection / resource 的边界仍然清楚。
4. 后面想补字段，也知道该补到哪个块。
```

## 17. TaskDefinition 的最小字段集

现在可以继续往上收一层：

```text
primitive
  定义一类动作本质上需要什么最小契约骨架。

TaskDefinition
  定义 agent 现在到底开放了一个什么具体任务入口。
```

也就是说：

```text
primitive 是动作类型。
TaskDefinition 是任务入口实例。
```

例如：

```text
Search / Gather
  是 primitive。

“联网搜索资料”
“搜索本地项目文件”
“检索知识库条目”
  才是 TaskDefinition。
```

### 17.1 TaskDefinition 这一层到底要解决什么

我认为 TaskDefinition 最核心的职责是：

```text
把一个抽象动作类型，
收束成一个 agent 真正可以被调用、被管理、被绑定的任务入口。
```

所以它至少要回答这些问题：

```text
1. 这是哪个 primitive 的具体任务入口。
2. 这个任务入口的目标是什么。
3. 它面向什么输入。
4. 它期待什么输出。
5. 它默认采用什么方法骨架。
6. 它天然允许或禁止什么范围。
7. 它适合由什么类型的投影来承载。
8. 它能否作为复合任务的子节点被复用。
```

### 17.2 我建议的最小字段集

如果现在就要拍第一版，我建议 TaskDefinition 至少包含：

```text
definition_id
  任务定义唯一标识。

definition_key
  任务定义键。

task_level
  basic / composite / coordination。
  当前这里先主要讨论 basic。

primitive_key
  该任务建立在哪个 basic primitive 之上。

title
  任务名称。

summary
  一句话说明这个任务是干什么的。

task_goal
  这个任务入口的目标描述。

usage_scope
  这个任务适用于什么场景，不适用于什么场景。

invocation_mode
  explicit / agent_selectable / internal_only。

enabled
  当前是否可用。

input_schema
  这个任务入口期望的输入结构。

output_schema
  这个任务入口承诺的输出结构。

completion_rule
  这个任务什么时候算完成。

failure_rule
  这个任务失败时至少要返回什么。

method_profile
  这个任务相对于 primitive 默认流程的特化说明。

skill_profile_ref
  这个任务默认读取哪类 skill 结构。
  这里先写 ref，不直接塞 skill 正文。

projection_profile
  这个任务对承载姿态的要求。
  这里是 requirement/profile，不是具体 projection_id。

resource_scope_profile
  这个任务通常需要的资源范围说明。
  这里仍然不直接写具体 tool id。

guardrails
  这个任务的特定禁止事项或边界要求。

reusable_as_child
  是否允许被 composite task 作为子任务复用。

tags
  分类标签，方便前端管理和筛选。
```

### 17.3 这里最重要的边界

我觉得这一步最需要守住的边界有三条：

```text
1. TaskDefinition 仍然不是 TaskBinding。
2. TaskDefinition 仍然不是运行时 prompt。
3. TaskDefinition 仍然不是具体 projection 或具体 tool 的直接绑定记录。
```

也就是说，TaskDefinition 负责的是：

```text
定义这个任务入口“是什么”。
```

而不是：

```text
这一次到底由哪个 agent、哪个 projection、哪些 tools 去执行。
```

这些应该留给后面的：

```text
TaskBinding
ExecutionSeat
TaskPromptContract
```

### 17.4 我为什么要保留 profile / ref 这一层

这里我比较坚持的一点是：

```text
TaskDefinition 不要直接内嵌大量 skill、projection、resource 的具体正文。
```

更稳的做法是：

```text
skill_profile_ref
projection_profile
resource_scope_profile
```

原因很简单：

```text
1. TaskDefinition 是任务入口层，不该膨胀成资源总表。
2. skills / projection / resource 自己也会演化。
3. 后面真正运行时还要再根据 binding 和 policy 继续裁剪。
```

所以这一层更像：

```text
“默认需求声明”
```

而不是：

```text
“最终运行结果”
```

### 17.5 TaskDefinition 和 primitive 的关系

我目前更认可的关系是：

```text
primitive
  给出这个动作类型的共性骨架。

TaskDefinition
  在这个骨架上补：
  任务目标
  使用范围
  输入输出模式
  特化方法偏好
  投影要求
  资源范围偏好
  守卫规则
```

可以理解成：

```text
primitive = 类
TaskDefinition = 这个类下面一个可管理的任务入口
```

但这里的“类”只是帮助理解，不表示我们一定要按 OOP 去实现。

### 17.6 basic TaskDefinition 的第一版例子

比如：

```text
primitive:
  Search / Gather

TaskDefinition:
  title = 联网搜索资料
  task_goal = 为当前问题寻找外部公开来源
  usage_scope = 适用于需要最新信息或外部资料的问题
  invocation_mode = explicit + agent_selectable
  input_schema = query, optional domains, optional freshness
  output_schema = candidate_sources, snippets, source_refs, unresolved_gaps
  method_profile = 先扩查询，再筛来源，再回传候选
  skill_profile_ref = basic.search.web
  projection_profile = evidence-first / concise
  resource_scope_profile = web-only
  guardrails = 不编造来源，无法确认时明确说明
  reusable_as_child = true
```

再比如：

```text
primitive:
  Verify / Review / Correct

TaskDefinition:
  title = 文档结果检查
  task_goal = 检查当前文档结果是否满足要求并指出问题
  usage_scope = 适用于草稿、报告、输出文案的检查
  invocation_mode = internal_only + agent_selectable
  input_schema = target_document, expected_contract, review_focus
  output_schema = findings, severity, pass_or_fail, correction_notes
  method_profile = 先对齐标准，再逐项检查，再输出问题
  skill_profile_ref = basic.verify.document
  projection_profile = crisp / reviewer / risk-aware
  resource_scope_profile = local-read
  guardrails = 不把风格偏好伪装成事实错误
  reusable_as_child = true
```

### 17.7 我当前建议先固定的版本

如果我们现在要收一个第一版口径，我建议固定成：

```text
TaskDefinition
  = primitive 之上的具体任务入口定义
  = 负责说明这个任务是什么、怎么被调用、输入输出是什么、默认需要什么
  != 具体绑定记录
  != 运行时 prompt
  != 最终执行席位
```

这样接下来就很自然可以进入下一层：

```text
TaskBinding 的最小字段集
```

## 18. TaskBinding 的最小字段集

如果说：

```text
primitive
  定义动作骨架。

TaskDefinition
  定义具体任务入口。
```

那么接下来这一层就是：

```text
TaskBinding
  定义这个任务入口在当前系统里到底怎么被配置和执行。
```

这一层非常关键，因为任务系统真正“管理 agent 能做什么任务”，
本质上管理的不是 primitive，也不只是 TaskDefinition，
而是：

```text
某个任务入口当前绑定了哪些 methods / skills / tools / projections / agents / 边界策略。
```

### 18.1 TaskBinding 这一层到底解决什么

我认为 TaskBinding 最核心解决的是下面这些问题：

```text
1. 这个任务当前由谁来执行。
2. 这个任务默认由什么姿态来承载。
3. 这个任务当前允许采用哪些 skills。
4. 这个任务当前允许采用哪些 tools 或资源面。
5. 这个任务当前明确禁止哪些 skills / tools /行为。
6. 这个任务是自动可选、显式调用，还是仅内部调用。
7. 任务冲突时按什么优先级裁剪。
```

所以：

```text
TaskDefinition 定义“是什么”。
TaskBinding 定义“现在怎么配”。
```

### 18.2 我建议的最小字段集

如果先拍第一版，我建议 TaskBinding 至少包含：

```text
binding_id
  绑定记录唯一标识。

task_definition_id
  指向哪个 TaskDefinition。

task_level
  basic / composite / coordination。

enabled
  当前这个绑定是否生效。

binding_mode
  manual / recommended / system_default。

invocation_policy
  explicit_only / agent_selectable / internal_only / mixed。

execution_mode
  single_seat / staged_single_seat / multi_seat。
  basic 通常先是 single_seat。

agent_selector
  谁来执行。
  可以是固定 agent，也可以是一个选择策略。

projection_selector
  用哪个 projection 承载。
  可以是固定 projection，也可以是 auto / manual / candidate set。

skill_scope
  当前绑定允许暴露给该任务的 skill refs。

tool_scope
  当前绑定允许暴露给该任务的 tool refs。

resource_scope
  当前绑定允许接触的资源范围摘要。
  例如 web-only / local-read / local-write / mixed。

denied_skills
  明确禁止的 skill refs。

denied_tools
  明确禁止的 tool refs。

guardrail_overrides
  相对 TaskDefinition 额外追加的限制。

prompt_conflict_policy
  当 task / skill / projection / resource 说明冲突时，
  如何裁剪。

output_policy
  该任务执行后要求输出什么粒度。
  例如 answer_only / answer_plus_evidence / artifact_required。

review_policy
  是否要求自动进入 verify primitive 或显式审查步骤。

child_reuse_policy
  该任务是否允许被 composite 直接复用，以及复用时沿用哪些绑定。

metadata
  备注、来源、更新时间、标签等。
```

### 18.3 我对几个关键字段的看法

这里有几个字段我觉得很关键，值得单独说清楚。

#### 1. agent_selector

我不建议第一版直接只有：

```text
agent_id
```

更稳的是：

```text
agent_selector
```

因为以后你一定会遇到：

```text
固定 agent
候选 agent 集
按任务类型选择
按用户配置选择
按系统默认选择
```

所以这一层最好从一开始就允许：

```text
fixed
candidate_set
auto
```

#### 2. projection_selector

这个字段非常关键，因为它正好接在我们这轮讨论出来的边界上。

TaskBinding 不直接定义投影系统本体，
但它必须声明：

```text
这个任务当前允许哪些投影来承载，
或者应该按什么策略选投影。
```

所以这里更适合是：

```text
fixed_projection
candidate_projection_ids
selection_mode = auto | manual | fixed
fallback_projection_id
```

#### 3. skill_scope / denied_skills

这里我比较坚持：

```text
一定要同时存在 allowed 和 denied 两侧。
```

因为你前面已经明确提出过一个很重要的目标：

```text
通过 tools 和 skills，
组成我们想让大模型承担和禁止的任务能力。
```

所以如果只有 allowed，没有 denied，
后面边界会很容易失控。

#### 4. prompt_conflict_policy

这个字段也值得早一点定下来。

因为后面一定会出现：

```text
TaskDefinition 说一种方法
skill_profile 又补另一种流程
projection 又强调另一类姿态
resource policy 又裁掉一部分
```

所以必须有一个清楚的冲突口径。

第一版我建议先允许这几种：

```text
fail_closed
  冲突即不自动运行，要求回退或显式确认。

prefer_task
  以任务定义为主。

prefer_binding
  以当前绑定为主。

prefer_resource_safety
  遇到资源或副作用冲突时，以更保守边界为主。
```

### 18.4 TaskBinding 和 TaskDefinition 的边界

这一步特别容易混，所以我先把边界压清楚：

```text
TaskDefinition
  定义任务入口是什么。

TaskBinding
  定义这个任务入口当前怎么配。
```

更具体地说：

```text
TaskDefinition 负责：
  目标
  适用范围
  输入输出模式
  方法需求
  默认 posture 需求
  默认资源需求

TaskBinding 负责：
  实际 agent 选择
  实际 projection 选择
  实际 skills 暴露范围
  实际 tools / resource 暴露范围
  允许 / 禁止边界
  调用策略
  审查策略
  冲突裁剪策略
```

一句话：

```text
TaskDefinition 更像设计图。
TaskBinding 更像安装配置。
```

### 18.5 basic task 的第一版 binding 例子

例如一个 basic task：

```text
TaskDefinition:
  联网搜索资料
```

它的 TaskBinding 可以是：

```text
binding_id = basic.search.web.default
task_definition_id = task.search.web
task_level = basic
enabled = true
binding_mode = system_default
invocation_policy = explicit_only + agent_selectable
execution_mode = single_seat
agent_selector = auto: researcher-capable
projection_selector = candidate_set: [evidence_scout, calm_researcher]
skill_scope = [basic.search.web, basic.read.source_screen]
tool_scope = [web.search, web.open]
resource_scope = web-only
denied_skills = [operate.local_write]
denied_tools = [shell.exec, file.write]
guardrail_overrides = [must_cite_sources, no_fabrication]
prompt_conflict_policy = prefer_resource_safety
output_policy = answer_plus_evidence
review_policy = optional
child_reuse_policy = reusable_with_same_resource_scope
```

这就很像一个真正可管理的任务配置了。

### 18.6 我为什么觉得这一层才是“任务系统管理”的核心

因为用户真正想管理的，其实往往不是：

```text
系统里有哪几个 primitive
```

也不是：

```text
系统里抽象上有哪些 TaskDefinition
```

而是：

```text
这个任务现在到底能不能用
谁来跑
能用哪些 skills
能碰哪些 tools
用什么投影来承载
是否自动选择
是否要额外审查
```

这些全部都落在 TaskBinding 上。

所以后面如果要做前端管理页，
最重要的其实也会是：

```text
TaskBinding 管理视图
```

### 18.7 当前我建议先固定的版本

如果现在继续收口，我建议先固定成：

```text
TaskBinding
  = 任务入口的执行配置层
  = 任务系统真正接 skills / tools / projections / agents / policy 的地方
  != 任务定义本身
  != 运行时执行结果
  != 最终 prompt 文本
```

这样下一层就很自然了：

```text
TaskPromptContract 的最小字段集
```

## 19. TaskPromptContract 的最小字段集

现在继续往下走一层：

```text
TaskPromptContract
  不是 TaskDefinition。
  不是 TaskBinding。
  也不是最终 prompt 文本。

TaskPromptContract
  是任务系统整理出来的“运行时契约中间层”。
```

它的作用是：

```text
把 TaskDefinition、TaskBinding、SkillRuntimeView、ToolRuntimeView、
ResourcePolicySummary、ProjectionRequirement 等信息，
整理成一份可以交给 PromptAssembler / ProjectionSystem 继续压缩呈现的结构化契约。
```

也就是说：

```text
TaskPromptContract
  是给模型看的说明书草案，
  但还不是最终给模型看的文本。
```

### 19.1 这一层解决什么问题

TaskPromptContract 主要解决：

```text
1. 多来源契约统一归口。
2. task / skill / tool / projection 信息不再散装拼接。
3. 明确哪些内容会进入模型可见视图。
4. 明确哪些冲突被裁剪、覆盖或降级。
5. 为后续 SoulRuntimeView / PromptManifest 提供稳定输入。
```

如果没有这一层，后面很容易变成：

```text
TaskDefinition 拼一段。
Skill 拼一段。
Projection 拼一段。
Tool 再拼一段。
最后谁优先、谁覆盖谁，全靠 prompt 文字自己打架。
```

所以这层是任务系统收束契约的关键。

### 19.2 我建议的最小字段集

第一版 TaskPromptContract 我建议至少包含：

```text
contract_id
  契约唯一标识。

task_definition_id
  来源任务定义。

binding_id
  来源任务绑定。

task_level
  basic / composite / coordination。

primitive_key
  basic task 时对应的 primitive。

runtime_intent
  这次任务的运行意图。
  例如 answer / search / read / transform / apply / review。

goal_section
  这次任务要完成什么。

scope_section
  当前任务范围、适用边界、不适用边界。

input_section
  当前输入是什么，输入是否足够。

method_section
  从 TaskDefinition + SkillRuntimeView 整理出来的方法流程摘要。

resource_section
  当前允许 / 禁止的 skills、tools、资源范围。

projection_section
  当前任务对承载姿态的要求。
  注意这里仍然是要求，不是投影系统最终生成的完整姿态。

agent_role_section
  当前执行席位里的 agent 职责。

output_section
  输出格式、产物要求、证据要求。

completion_section
  什么算完成，什么算失败或需要回退。

guardrail_section
  禁止事项、安全边界、审批要求。

review_section
  是否需要检查、如何检查、是否触发 Verify primitive。

conflict_section
  被覆盖、被裁剪、被拒绝、被降级的内容。

visibility_policy
  哪些段给模型看，哪些只给调试或系统看。

assembly_notes
  给 PromptAssembler 的装配提示。
```

### 19.3 哪些字段是模型可见，哪些不一定可见

这里要特别区分：

```text
TaskPromptContract 里有些字段是模型可见内容，
有些字段只是系统装配用的控制信息。
```

第一版可以这样划：

```text
默认模型可见：
  goal_section
  scope_section
  input_section
  method_section
  resource_section
  projection_section
  agent_role_section
  output_section
  completion_section
  guardrail_section

条件可见：
  review_section
  conflict_section

系统内部：
  contract_id
  task_definition_id
  binding_id
  primitive_key
  visibility_policy
  assembly_notes
```

其中 `conflict_section` 要谨慎：

```text
如果冲突信息会干扰模型执行，
就只给系统调试视图。

如果冲突信息是执行必须知道的限制，
才转成 guardrail 或 resource_section 的可见内容。
```

### 19.4 method_section 的来源

`method_section` 不应该由 TaskDefinition 单独生成。

它应该来自：

```text
TaskDefinition.method_profile
  当前任务入口自己的方法偏好。

SkillRuntimeView
  当前任务可见 skills 的静态方法流程。

TaskBinding.guardrail_overrides
  当前绑定追加的约束。

ResourcePolicySummary
  当前资源边界对方法造成的限制。
```

整理时要守住：

```text
TaskDefinition 定方向。
SkillRuntimeView 给流程。
ResourcePolicySummary 裁资源。
TaskBinding 给当前配置优先级。
```

所以 `method_section` 的第一版结构可以是：

```text
method_section:
  method_goal
  recommended_steps
  checks
  fallback
  prohibited_methods
```

### 19.5 resource_section 的来源

`resource_section` 是防止模型乱用资源的关键。

它应该从这些地方合成：

```text
TaskDefinition.resource_scope_profile
TaskBinding.skill_scope
TaskBinding.tool_scope
TaskBinding.denied_skills
TaskBinding.denied_tools
ResourcePolicySummary
```

第一版结构可以是：

```text
resource_section:
  visible_skills
  visible_tools
  allowed_resources
  denied_resources
  permission_notes
  request_approval_when
```

这里要守住：

```text
denied 的内容不一定要完整展示给模型。
如果展示 denied 会暗示模型尝试禁止能力，
就只转成一句边界说明。
```

例如：

```text
不要说：
  你不能调用 shell.exec。

可以说：
  当前任务不允许执行本地命令或写入文件。
```

### 19.6 projection_section 的边界

`projection_section` 不是最终投影 prompt。

它只应该说明：

```text
当前任务希望投影怎样承载。
```

第一版结构可以是：

```text
projection_section:
  required_posture
  expression_density
  attention_bias
  dynamic_notes
```

最终仍然由：

```text
ProjectionSystem / SoulProjectionBuilder
```

根据：

```text
ProjectionCard
ProjectionState
TaskPromptContract.projection_section
```

生成：

```text
ProjectionRuntimePosture
SoulRuntimeView
```

### 19.7 TaskPromptContract 的装配顺序

我建议第一版的装配顺序固定为：

```text
1. 读取 TaskDefinition。
2. 读取 TaskBinding。
3. 根据 binding 解析 SkillRuntimeView。
4. 根据 binding 和操作系统策略解析 ToolRuntimeView / ResourcePolicySummary。
5. 合成 method_section 和 resource_section。
6. 合成 projection_section 的要求。
7. 根据 prompt_conflict_policy 处理冲突。
8. 输出 TaskPromptContract。
9. 交给 ProjectionSystem / PromptAssembler 生成最终模型可见视图。
```

这条顺序很重要，因为它再次固定了：

```text
任务真相在前。
方法流程在中。
资源边界参与裁剪。
投影姿态最后呈现。
```

### 19.8 一个 basic task 的 TaskPromptContract 例子

继续用“联网搜索资料”举例：

```text
contract_id = prompt.basic.search.web.001
task_definition_id = task.search.web
binding_id = basic.search.web.default
task_level = basic
primitive_key = search
runtime_intent = gather_sources

goal_section:
  为当前问题寻找公开可访问的外部信息源。

scope_section:
  只搜索公开网页，不读取本地文件，不执行本地命令。

input_section:
  query = 用户问题
  freshness = 需要时优先最新资料

method_section:
  1. 拆分查询意图。
  2. 搜索候选来源。
  3. 筛选可信度和相关性。
  4. 返回候选来源和未解决缺口。

resource_section:
  visible_tools = web.search, web.open
  denied_resources = local_write, shell_exec
  permission_notes = 不允许本地副作用。

projection_section:
  required_posture = evidence-first researcher
  expression_density = concise
  attention_bias = relevance / traceability

output_section:
  candidate_sources
  relevance_reason
  source_refs
  unresolved_gaps

completion_section:
  找到足够进入 Read / Extract 的候选来源，
  或明确说明未找到可靠来源。

guardrail_section:
  不编造来源。
  无法确认时明确标记不确定。
```

### 19.9 当前建议先固定的版本

如果这一层先收口，我建议固定成：

```text
TaskPromptContract
  = 任务系统整理出来的运行时契约中间层
  = 多来源说明进入模型前的统一归口
  = PromptAssembler / ProjectionSystem 的稳定输入
  != 最终 prompt 文本
  != 投影系统本体
  != skills 原文拼接
```

接下来就可以继续定：

```text
SkillRuntimeView 的最小字段集
ProjectionRuntimePosture 的最小字段集
```

## 20. SkillRuntimeView 的最小字段集

现在继续讨论 `SkillRuntimeView`。

我认为它的定位必须先说清楚：

```text
SkillRuntimeView
  不是 SkillRegistry。
  不是完整 skill 文档。
  不是 task definition。
  也不是最终 prompt 文本。

SkillRuntimeView
  是当前任务允许大模型看到的“方法说明切片”。
```

也就是说：

```text
SkillRegistry 保存完整技能。
TaskBinding 决定哪些技能可见。
SkillScopeResolver 裁剪出当前任务需要的技能视图。
SkillRuntimeView 才是进入 TaskPromptContract 的方法来源。
```

### 20.1 为什么需要 SkillRuntimeView

如果没有这一层，skills 很容易变成：

```text
把所有技能说明一次性丢给模型。
```

但我们前面已经判断过：

```text
如果任务理解歪了，
给模型更多 skills 只会让它在错误方向上拥有更多错误选择。
```

所以 SkillRuntimeView 的核心价值是：

```text
按任务裁剪方法信息，
只暴露当前任务必要的流程、检查点和回退策略。
```

### 20.2 我建议的最小字段集

第一版 `SkillRuntimeView` 我建议至少包含：

```text
view_id
  当前 skill runtime view 的唯一标识。

task_definition_id
  当前服务于哪个任务定义。

binding_id
  当前服务于哪个任务绑定。

primitive_key
  当前服务于哪个 basic primitive。

skill_refs
  本次视图引用了哪些 skill。

skill_level
  basic / composite / coordination。

method_purpose
  当前这些 skills 共同解决什么方法问题。

applicability
  当前任务下什么时候适用，什么时候不适用。

procedure
  当前任务可见的核心步骤。

checks
  当前任务可见的检查点。

fallback
  信息不足、失败或冲突时的回退方式。

handoff_notes
  给后续 primitive / composite / coordination 的交接说明。

prohibited_uses
  当前任务下明确不能把这些 skills 用来做什么。

required_artifacts
  执行这些方法后，至少要留下什么产物或摘要。

compression_policy
  当上下文太长时，哪些 skill 内容必须保留，哪些可压缩。

source_notes
  这些方法片段来自哪些 skill refs，供调试和追踪使用。
```

### 20.3 模型真正需要看的字段

不是所有字段都要给模型看。

默认可见的可以先是：

```text
method_purpose
applicability
procedure
checks
fallback
prohibited_uses
required_artifacts
handoff_notes
```

系统内部字段可以是：

```text
view_id
task_definition_id
binding_id
primitive_key
skill_refs
skill_level
compression_policy
source_notes
```

这里的关键是：

```text
模型看到的是当前任务的方法说明，
系统保留的是这些说明来自哪里、如何压缩、如何追踪。
```

### 20.4 procedure 不应该太长

`procedure` 是 SkillRuntimeView 的核心，但也最容易膨胀。

我建议第一版给它一个强约束：

```text
basic skill view 的 procedure 最好控制在 3 到 7 步。
```

原因是：

```text
basic task 本来就是最小任务动作。
如果一个 basic procedure 需要十几步，
它大概率已经是 composite task 了。
```

这能帮我们反过来检查任务分层是否跑偏。

### 20.5 checks 的作用比看起来更重要

`checks` 不只是质量检查，它还负责防止模型偏航。

它应该回答：

```text
做完后要核对什么。
哪些错误最常见。
什么情况说明任务没有完成。
什么情况应该触发 Verify / Review / Correct。
```

所以 checks 第一版可以分成：

```text
completion_checks
quality_checks
risk_checks
trigger_verify_when
```

但在字段上不一定马上拆太细，先可以收在 `checks` 里。

### 20.6 fallback 的边界

`fallback` 应该给模型明确的退路，但不能变成自动无限循环。

它应该说明：

```text
信息不足时怎么办。
工具失败时怎么办。
结果冲突时怎么办。
什么时候应该停止并返回失败说明。
什么时候应该请求用户补充。
```

这里要和 `TaskPromptContract.failure_contract` 对齐。

### 20.7 prohibited_uses 的必要性

我建议 `SkillRuntimeView` 必须有 `prohibited_uses`。

原因是：

```text
同一个 skill 在不同任务里可能有不同边界。
```

例如：

```text
搜索 skill 在研究任务里可以用来找资料，
但不能用来替代事实判断。

文档读取 skill 可以读取指定材料，
但不能推断材料之外的事实。

本地操作 skill 可以改指定文件，
但不能扩大到未授权路径。
```

这能和 TaskBinding 的 denied scope 配合起来，
形成更清楚的任务能力边界。

### 20.8 required_artifacts 的意义

`required_artifacts` 是为了让 skill 执行后留下可被后续任务使用的东西。

例如：

```text
Search / Gather
  应留下 candidate_sources。

Read / Extract
  应留下 extracted_facts 和 evidence_refs。

Transform / Compose
  应留下 generated_artifact。

Operate / Apply
  应留下 changed_targets 和 execution_log。

Verify / Review / Correct
  应留下 findings 和 pass_or_fail。
```

如果没有 artifact，复合任务后面会很难接。

所以：

```text
SkillRuntimeView 不只是告诉模型怎么做，
还要告诉模型做完要留下些什么。
```

### 20.9 basic / composite / coordination 的 SkillRuntimeView 区别

虽然现在主要讨论 basic task，但可以先预留三层差异：

```text
BasicSkillRuntimeView
  只描述一个 primitive 的方法流程。

CompositeSkillRuntimeView
  描述多个 basic primitives 如何串联，
  以及阶段之间如何压缩上下文和交接 artifact。

CoordinationSkillRuntimeView
  描述多个 ExecutionSeat 如何分工、交接、审查、合并和闭环。
```

这和我们之前定的任务阶梯一致：

```text
basic
  方法最小化。

composite
  方法阶段化。

coordination
  方法协议化。
```

### 20.10 SkillRuntimeView 的生成链路

我建议第一版生成链路是：

```text
TaskDefinition.skill_profile_ref
  + TaskBinding.skill_scope
  + TaskBinding.denied_skills
  + primitive.default_skill_sections
  -> SkillScopeResolver
  -> SkillRuntimeView
```

然后：

```text
SkillRuntimeView
  -> TaskPromptContract.method_section
```

这里要守住：

```text
SkillRuntimeView 只进入 method_section，
不应该直接越过 TaskPromptContract 去拼最终 prompt。
```

### 20.11 当前建议先固定的版本

如果现在先收口，我建议固定成：

```text
SkillRuntimeView
  = 当前任务可见的静态方法流程切片
  = SkillScopeResolver 的输出
  = TaskPromptContract.method_section 的主要来源
  != 完整 skill 仓库
  != 任务定义
  != 投影姿态
  != 最终 prompt 文本
```

这样下一层就很清楚了：

```text
ProjectionRuntimePosture
  负责把已经整理好的任务和方法信息，
  用什么姿态、密度、强调重点呈现给模型。
```

## 21. 框架收束与推进规则

当前需要先暂停继续追加字段。

前面已经产生了很多候选结构，如果继续往下细化，很容易变成：

```text
每一层都看起来合理，
但整体越来越重，
最后无法判断哪些是核心、哪些只是可选扩展。
```

所以接下来应该先定框架，再做推进。

### 21.1 第一版只保留一条主链路

第一版任务系统先只固定这一条主链路：

```text
Primitive
  -> TaskDefinition
  -> TaskBinding
  -> SkillRuntimeView / ResourceRuntimeView / ProjectionRequirement
  -> TaskPromptContract
  -> ProjectionRuntimePosture
  -> PromptManifest
```

每一层只回答一个问题：

```text
Primitive
  这是什么类型的最小动作。

TaskDefinition
  这是一个什么具体任务入口。

TaskBinding
  这个任务入口现在怎么配置。

SkillRuntimeView
  当前任务能看到什么方法流程。

ResourceRuntimeView
  当前任务能碰什么资源边界。

TaskPromptContract
  任务系统整理出的运行时契约。

ProjectionRuntimePosture
  当前投影如何承载这份契约。

PromptManifest
  最终给模型看的低噪声说明。
```

### 21.2 第一版核心对象只保留最小字段

接下来每个对象都按“核心字段 + 扩展字段”处理。

核心字段必须满足：

```text
1. 模型执行当前任务必须知道。
2. 系统裁剪权限或方法必须知道。
3. 不能从别的字段稳定推导出来。
```

不满足这三条的，先降级为：

```text
metadata
extension
debug view
future field
```

### 21.3 当前建议的极简核心字段

第一版可以先把核心字段压成下面这样。

#### Primitive

```text
primitive_key
purpose
input_contract
output_contract
completion_criteria
default_skill_sections
resource_profile
```

#### TaskDefinition

```text
definition_id
task_level
primitive_key
title
task_goal
input_schema
output_schema
skill_profile_ref
projection_profile
resource_scope_profile
reusable_as_child
```

#### TaskBinding

```text
binding_id
task_definition_id
enabled
invocation_policy
execution_mode
agent_selector
projection_selector
skill_scope
tool_scope
denied_scope
prompt_conflict_policy
review_policy
```

#### SkillRuntimeView


### 21.4 推进顺序

后面不要再按“想到一个字段就加一个字段”的方式推进。

改成四步：

```text
第一步：固定对象链路。
  只确认有哪些对象，以及每个对象回答什么问题。

第二步：固定每个对象的核心字段。
  每个对象最多先保留 5 到 12 个核心字段。

第三步：用 3 个真实任务案例验证。
  例如：
    联网搜索资料
    读取并总结本地文档
    修改文档后检查

第四步：只有案例跑不通时才补字段。
  能通过组合、推导、metadata 解决的，不进核心字段。
```

### 21.5 字段准入规则

后面任何新字段要进入核心层，都必须回答：

```text
1. 它服务哪一层对象。
2. 它解决哪一个执行问题。
3. 它是否会进入模型可见视图。
4. 它是否能从已有字段推导。
5. 如果没有它，哪个真实任务案例会跑不通。
```

答不上来，就不进核心层。

### 21.6 当前收束判断

当前先不要继续扩写：

```text

更多 TaskDefinition 字段
更多 TaskBinding 字段
更多 SkillRuntimeView 字段
```

下一步应该做的是：

```text

```

如果三个案例都能闭合，说明框架成立。

如果闭合不了，再补最少的字段。

## 25. 关于复合任务的新判断

经过复测后，对复合任务形成一个更准确的新判断：

```text
复合任务不能纯靠系统预定义。
复合任务应该允许大模型根据当前目标进行规划。
```

也就是说：

```text
系统不应该试图提前穷举所有 composite task。
```

因为真实任务组合会非常多，比如：

```text
读取并总结文档
搜索后写报告
研究后更新本地文档
修改后审查
读取多个文件后重构
搜索资料后比较方案
```

如果每一种都要靠我们手写一个固定 CompositeTaskDefinition，
任务系统会很快膨胀。

### 25.1 复合任务应该由谁规划

当前更合理的方向是：

```text
复合任务的具体步骤，可以由大模型规划。
```

但这个规划不能是无边界自由发挥。

系统应该提供：

```text
1. 可用 basic primitives。
2. 可用 TaskDefinition。
3. 可用 skills 范围。
4. 可用 tools / resource 范围。
5. 可用 projections / agent seat 范围。
6. review / approval / conflict policy。
7. 最大阶段数、最大资源范围、最大副作用边界。
```

大模型负责：

```text
根据目标选择需要哪些 basic tasks，
决定顺序，
决定每一步的输入输出交接，
形成 composite plan。
```

系统负责：

```text
验证这个 plan 是否合规，
裁剪不允许的 skills / tools，
补充投影和资源边界，
生成 TaskPromptContract。
```

### 25.2 CompositeTaskDefinition 的角色要调整

因此 `CompositeTaskDefinition` 不应该被理解成：

```text
所有复合任务都必须预先定义好的固定流程。
```

更合理的定位是：

```text
复合任务模板 / 参考结构 / 约束配置。
```

也就是说，系统可以保留一些内置模板：

```text
search_and_answer
read_and_rewrite_document
research_then_update_local_doc
```

但它们不应该成为唯一入口。

更准确的链路应该是：

```text
用户目标
  -> 模型生成 CompositePlan
  -> 系统用 CompositePolicy / TaskBinding 验证和裁剪
  -> 生成可执行的 staged TaskPromptContract
```

### 25.3 复合任务管理应该管理什么

如果复合任务由模型规划，那么复合任务管理页不应该主要管理“固定流程列表”。

它应该管理：

```text
1. 复合任务规划权限
   允许模型自动规划，还是只能显式调用。

2. 可用 basic task 范围
   当前复合任务规划时能使用哪些 basic task。

3. skills scope
   复合任务阶段能读取哪些 skills。

4. tools / resource scope
   复合任务阶段能触达哪些资源面。

5. projection policy
   是否允许阶段投影切换，
   是否允许 owner agent 内部子执行席位。

6. review policy
   哪些复合任务必须经过 Verify / Review / Correct。

7. plan limits
   最大阶段数、最大并行度、是否允许写入、是否允许联网。

8. approval policy
   什么时候需要用户确认。
```

一句话：

```text
复合任务管理的核心不是枚举所有复合任务，
而是管理大模型如何合法地组合 basic tasks。
```

### 25.4 新增对象：CompositeTaskPolicy

为了避免继续扩 `CompositeTaskDefinition`，可以单独引入：

```text
CompositeTaskPolicy
```

它回答：

```text
模型规划复合任务时，允许怎么规划。
```

第一版最小字段可以是：

```text
policy_id
enabled
planning_mode
  disabled / explicit_only / agent_planned

allowed_basic_tasks
denied_basic_tasks
allowed_skill_scope
denied_skill_scope
allowed_tool_scope
denied_tool_scope
projection_policy
review_policy
max_steps
requires_approval_when
```

这样就把两个概念拆开：

```text
CompositeTaskDefinition
  是系统预置模板。

CompositeTaskPolicy
  是模型自动规划复合任务的权限与边界。
```

### 25.5 对第二个案例的修正

前面说“缺一个 read_and_summarize_document 复合任务定义”，现在要修正：

```text
不一定必须新增一个固定 CompositeTaskDefinition。
```

更好的做法可能是：

```text
让模型根据目标规划：
  task_environment_setup
  -> information_synthesis

然后由 CompositeTaskPolicy 检查：
  是否允许本地读取
  是否允许总结
  是否禁止写入
  是否需要 review
```

也就是说：

```text
缺的不是一个固定模板，
而是 composite planning policy 和 plan validation。
```

### 25.6 当前调整后的推进方向

因此下一步顺序应该调整为：

```text
1. 收紧 SkillRuntimeView resolver。
2. 新增 CompositeTaskPolicy。
3. 做一个模型规划输出的 CompositePlan 结构。
4. 用 CompositeTaskPolicy 验证 CompositePlan。
5. 把通过验证的 CompositePlan 转成 staged TaskBinding / TaskPromptContract。
6. 再接 ProjectionSystem 消费 TaskPromptContract。
```

先不急着补大量固定 CompositeTaskDefinition。

### 25.7 当前阶段性定案

现在可以把任务结构先定下来：

```text
基本任务是骨架。
复合任务是模型根据需求，在系统权限框内，对基本任务进行规划和组合。
```

也就是说：

```text
BasicTask
  提供 agent 可执行的最小任务动作。

CompositeTask
  不再主要靠系统穷举固定流程。
  它由大模型根据用户目标和当前上下文，
  选择、排列、连接多个 BasicTask。

TaskSystem
  不负责提前写死所有复合任务。
  它负责提供可组合的 basic tasks，
  提供少量特定任务模板，
  提供投影风格建议，
  提供 skills / tools / resource / approval 权限管理。
```

所以复合任务的系统职责是：

```text
1. 提供可参考的特定任务模板。
2. 允许模型按目标生成 CompositePlan。
3. 验证 CompositePlan 是否越界。
4. 为每个阶段绑定可见 skills、tools 和投影姿态。
5. 管理是否允许写入、联网、多阶段、review、approval。
6. 把通过验证的 plan 转成 staged TaskPromptContract。
```

而不是：

```text
为每一种用户需求手写一个 CompositeTaskDefinition。
```

一句话定案：

```text
BasicTask 是任务骨架。
CompositeTask 是受控规划。
CoordinationTask 是多 agent 协议化执行。
```

### 25.8 对后续代码施工的影响

按这个定案，后面代码优先级也要调整：

```text
优先：
  CompositeTaskPolicy
  CompositePlan
  CompositePlanValidator
  staged TaskBinding / TaskPromptContract builder

降级：
  大量手写 CompositeTaskDefinition
  复杂固定 graph template 扩充
```

模板仍然保留，但定位变成：

```text
模板是参考结构和常用路径，
不是复合任务能力的唯一来源。
```

这样任务系统才会真正具备扩展性：

```text
新增 basic task
  -> 模型可组合能力变强。

新增 skill / tool / projection
  -> 某些 basic task 或 composite plan 的执行质量变强。

新增 CompositeTaskPolicy
  -> 模型规划边界变清楚。

新增 CompositeTaskDefinition
  -> 只是增加一个常用模板，而不是扩展系统边界的唯一方式。
```

# 洪荒时代灵魂系统管理与多态投影方案

日期：2026-04-27  
项目名：洪荒时代（mythical ageent）  
定位：把灵魂系统从静态 prompt 切换器升级为可管理、可投影、可协作、可收束 prompts 的智能体灵魂系统。

---

## 0. 结论

灵魂系统要成为洪荒时代的主打设定之一：

```text
切换灵魂
多态投影
多灵魂协作
统一 prompt 收束
```

第一阶段先把灵魂本体收缩成“表达身份种子”。

`SoulProfile` 暂时只负责：

```text
语言风格
输出方式
表达习惯
身份质感
对话温度
未来可扩展的状态记忆锚点
```

工作方法、任务流程、工具/skill 使用方式、检查方式和任务约束，不再直接写进 `SoulProfile`。

这些内容应该归到：

```text
DefaultProjection
  保留过去归给灵魂的常用工作风格、默认注意事项和表达姿态，方便用户直接使用。

SoulProjection
  承载当前任务下的投影姿态、角色表达和任务态约束摘要。

TaskMethodContract
  承载真正的任务方法、执行流程、检查方法和产物要求。

ResourcePolicy
  决定 tools / skills / memory 的可见与可用边界。
```

但要守住一条硬边界：

> 投影可以理解当前可见的工具和 skills，但不能天然获得工具调用权、记忆写回权或调度权。

也就是说：

```text
SoulProfile 不直接看见 Tool/Skill 运行细节。
SoulProjection 只能接收 ResourcePolicy 允许暴露的 Tool/Skill 摘要。
ControlKernel / ResourcePolicy 决定能不能真的用。
```

---

## 1. 为什么要系统化

如果灵魂系统只是一组 seed 文件，它只能做到：

```text
切换语气
切换回答习惯
切换人格背景
```

但洪荒时代需要的是：

```text
灵魂作为智能体运行时的意志层。
```

所以它必须能管理：

- 灵魂注册。
- 灵魂切换。
- 灵魂启用/禁用。
- 灵魂版本。
- 灵魂 portrait / 元信息。
- 灵魂 seed / profile / projection 的版本关系。
- 灵魂对任务的适配。
- 灵魂对 agent 的投影。
- 灵魂对 skills 的投影。
- 灵魂对 tools 的理解。
- 灵魂对 memory 的使用边界。
- 灵魂在多 agent 协作中的分身。
- 灵魂参与过哪些任务、产生过哪些 runtime view、使用过哪些能力视图。

因此灵魂系统要分成三层：

```text
Soul Management Plane
  管理灵魂档案、启用状态、版本、来源、用户自制灵魂、校验结果。

Soul Projection Plane
  根据 TaskMode / AgentProfile / SkillScope / ToolScope 生成当前任务投影。

Soul Runtime Plane
  把投影装配成 prompt sections，并交给 ControlKernel 采纳。
```

这三层不能混在一起：

```text
管理层只维护灵魂。
投影层只生成视图。
运行层只消费被采纳的视图。
授权层仍然在 ControlKernel / ResourcePolicy。
```

## 2. 核心概念

### 2.1 SoulProfile

灵魂本体，是表达身份种子，不是工作流模板。

它描述：

```text
这个灵魂是谁。
它怎么表达。
它的语言节奏、措辞习惯和对话温度。
它的身份质感和基础叙事。
它未来如何作为状态记忆、关系记忆、长期偏好视图的锚点。
```

它不描述：

```text
它应该采用什么任务流程。
它应该默认调用什么 skill。
它能直接调用什么工具。
它能直接写入什么记忆。
它能绕过什么验证。
它适合哪些任务模式。
```

`SoulProfile` 必须是机器可读契约，不只是 markdown 文本。

建议字段：

```text
soul_id
name
display_name
source: builtin | user
version
enabled
seed_path
description
background
personality_traits
expression_style
language_style
tone
speech_patterns
relationship_posture
state_memory_anchor
portrait
metadata
```

其中：

```text
seed_path / SOUL.md      给模型看的叙事性灵魂文本
profile.json            给系统看的结构化灵魂契约
SoulProjection           给当前任务看的运行时视图
```

迁移规则：

```text
过去写在 SoulProfile 里的 working_habits
  -> 迁移到 DefaultProjection / TaskMethodContract。

过去写在 SoulProfile 里的 preferred_task_modes / preferred_role_types
  -> 迁移到 ProjectionCard 的推荐元数据，或任务系统的绑定策略。

过去写在 SoulProfile 里的 tool / skill 认知
  -> 迁移到 ResourcePolicy 暴露的可见摘要和 TaskPromptContract。

过去写在 SoulProfile 里的 guardrails / risk_biases
  -> 全局底线进入 Core Prompt / ResourcePolicy；
  -> 某个任务的检查偏好进入 DefaultProjection 或 TaskMethodContract。
```

### 2.2 SoulProjection

灵魂投影，是 SoulProfile 在具体任务上下文里的运行时表达视图。

投影系统的根本职责是：

```text
让 agent 以什么姿态承载当前任务。
```

它不是任务系统，不是 skill 系统，也不是操作系统。

换句话说：

```text
TaskSystem
  提供任务真相：当前要做什么、怎么做、产物是什么。

SkillSystem
  提供方法资源：当前任务可以采用哪些方法片段。

OperationSystem / ResourcePolicy
  提供执行边界：哪些工具、文件、记忆和副作用可见或可用。

SoulProjection
  提供承载姿态：agent 以什么角色状态、表达方式、注意力结构和任务态风格执行。
```

投影可以承载：

```text
任务态表达姿态
当前角色口吻
当前任务的注意事项摘要
默认工作风格预设
任务特定约束的表达化呈现
skill/tool 摘要的表达方式
基于工作记忆形成的短期姿态调整
```

投影不直接承载：

```text
最终执行图
最终工具权限
最终 skill 权限
最终检查流程
任务流程真相
资源授权事实
长期灵魂模板改写
```

这些由 `TaskMethodContract / ResourcePolicy / ControlKernel` 决定。

因此，投影的输入输出边界应该固定为：

```text
输入：
  SoulProfile 的表达身份素材
  ProjectionCard / DefaultProjection 的静态姿态预设
  TaskRuntimeInstruction
  SkillRuntimeView
  ToolRuntimeView
  MemoryPolicy summary
  OutputContract
  ProjectionStateCandidate

输出：
  ProjectionRuntimePosture
  PromptSection[]
  SoulRuntimeView
  PromptManifest

不输出：
  TaskDefinition
  TaskGraph
  SkillScope 授权结果
  ToolScope 授权结果
  Memory write decision
  Runtime directive
```

同一个灵魂在不同上下文里应该有不同投影：

```text
conversation_projection   对话承载姿态
agent_projection          agent 角色姿态
task_projection           任务承载姿态
skill_view_projection     skill 摘要呈现姿态
tool_view_projection      tool 摘要呈现姿态
memory_view_projection    记忆摘要呈现姿态
collaboration_projection  协作分身姿态
```

这就是“多态投影”。

不是每个灵魂只有一段固定 prompt，而是：

```text
同一个 SoulProfile
  -> 在不同 TaskRuntimeInstruction / AgentProfile / SkillRuntimeView / ToolRuntimeView 下
  -> 生成不同 SoulProjection
```

`SoulProjection` 是灵魂系统的核心，不是附属工具。

它负责把一个表达身份种子投射到当前任务中，并与任务契约、资源摘要合成运行时视图：

```text
SoulProfile
  + selected Soul Seed sections
  + TaskRuntimeInstruction
  + TaskProjectionBinding
  + AgentProfile
  + SkillRuntimeView
  + ToolRuntimeView
  + MemoryPolicy
  + OutputContract
  + ProjectionStateCandidate
  -> SoulRuntimeView
```

关键规则：

```text
同一个灵魂可以有多个投影。
同一个任务可以有多个灵魂投影。
同一个灵魂投影只能服务一个明确 TaskRole。
投影可以描述当前可见能力，但不能扩大权限。
投影可以调整任务态表达和注意力结构，但不能改写任务流程。
投影可以基于工作记忆形成短期成长态，但不能写回 SoulProfile。
```

### 2.2.1 SoulProjectionCard

投影应该卡片化管理。

一个灵魂不是只有一个投影，而是可以生成多个可复用投影卡片：

```text
句芒 / 对话引导投影
句芒 / 架构统筹投影
河伯 / RAG 证据收集投影
四岳 / 长文结构投影
祝融 / 实现推进投影
玄女 / 风险审查投影
```

投影卡片由两种方式产生：

```text
用户手动生成
  -> 用户根据当前需求选择 soul / role_type / posture / recommended task families

系统任务生成
  -> SoulSystem 根据 TaskRuntimeInstruction 和可用 ProjectionCard 生成投影候选
```

投影卡片由两种方式使用：

```text
用户显式选用
  -> 当前任务使用用户选中的 projection_id

系统按需调用
  -> TaskBinding / ControlKernel 根据任务、agent 和资源边界采纳 projection_id
```

投影卡片不是权限卡。

```text
ProjectionCard 只保存静态绑定。
ProjectionCard 保存 soul_id / role_type / posture_tags / recommended_task_families。
ProjectionCard 保存默认表达姿态、工作风格预设、注意力偏好和风险提示。
ProjectionCard 可以保存 skill/tool 摘要的呈现偏好。
ProjectionCard 可以作为任务候选。
ProjectionCard 不能保存特定任务契约。
ProjectionCard 不能保存运行时 SkillRuntimeView / ToolRuntimeView。
ProjectionCard 不能保存 SoulRuntimeView。
ProjectionCard 不能保存 PromptManifest。
ProjectionCard 不能保存工具授权、worker route 或 memory 写回权。
```

更准确地说：

```text
ProjectionCard
  是静态投影配置。

SoulProjection / SoulRuntimeView
  是任务运行时生成结果。

TaskProjectionBinding
  才是某个具体任务中 agent + task + projection + skill/tool scope 的绑定关系。
```

因此不要把 `ProjectionCard` 做成万能绑定卡。  
它只是“这个投影通常怎么承载任务”的配置，不是“这个任务实际怎么执行”的事实。

### 2.2.2 ProjectionState

投影可以成长，但第一阶段只保留工作记忆驱动的短期成长口子。

建议命名为：

```text
ProjectionState
```

它来自工作记忆，但不等于长期灵魂本体。

```text
ProjectionState 可以保存：
  当前任务中的短期表达调整
  最近成功的沟通方式
  当前用户对表达密度/节奏的偏好
  当前任务阶段的注意力偏向
  当前投影的临时状态标签

ProjectionState 不保存：
  工具权限
  skill 授权
  任务流程
  长期人格设定
  durable memory 写入事实
```

运行时关系：

```text
WorkingMemoryView
  -> ProjectionStateCandidate
  -> SoulProjectionBuilder
  -> ProjectionRuntimePosture
```

关键规则：

```text
ProjectionState 只能影响当前运行时姿态。
ProjectionState 不能直接写回 SoulProfile。
ProjectionState 不能绕过 TaskSystem 改写任务。
ProjectionState 不能绕过 ResourcePolicy 获得工具或记忆权限。
```

### 2.3 SoulRuntimeView

灵魂运行时视图。

它是任务运行时临时生成的 prompt 装配结果，不是投影卡片本体，也不进入投影卡片存储。

```text
SoulRuntimeView =
  SoulProfile selected sections
  + AgentProfile role
  + TaskMode
  + allowed Skill summary
  + allowed Tool summary
  + MemoryPolicy summary
  + OutputContract
```

注意：

```text
RuntimeView 只显示当前任务需要的部分。
不要把所有灵魂档案、所有工具、所有技能都塞进 prompt。
```

建议 `SoulRuntimeView` 固定分段：

```text
identity_view            当前灵魂是谁
style_view               当前语言与表达风格
task_view                当前任务目标和约束
role_view                当前临时职责
skill_view               当前允许 skills 的可见摘要
tool_view                当前允许 tools 的可见摘要
memory_view              当前可见记忆与写回边界
collaboration_view       当前协作关系
output_view              当前产物格式与最终交付边界
risk_view                当前灵魂需要特别防的偏差
```

这些分段后续要支持 cache-friendly 装配：

```text
static sections      灵魂身份、长期风格、核心底线
semi-static sections AgentProfile、SkillContract、ToolContract 摘要
dynamic sections     当前任务、上下文、输出要求
volatile sections    当前轮临时约束、协作状态、错误恢复
```

## 3. 灵魂与 Prompt 的关系

目标：

```text
尽量让 prompts 都由灵魂系统收束。
```

这不代表所有 prompt 都写进 seed。

正确分层：

```text
Soul Seed
  - 身份
  - 背景
  - 语言风格
  - 表达方式
  - 对话温度

Core Prompt
  - 所有灵魂共享的事实原则、安全边界、输出底线

Static Common Rules
  - 所有任务都稳定生效的共同准则

Dynamic Task Contract
  - 当前任务临时生效的任务契约、输出要求、协作边界

DefaultProjection
  - 默认投影保留的常用工作风格、角色姿态、任务态约束摘要
  - 这些内容不写回 SoulProfile

AgentProfile Prompt
  - 当前 agent 身份、职责、任务边界

Skill Prompt
  - 当前 skill 能力、适用场景、输出约束

Tool Prompt
  - 当前可用工具说明、风险提示、输入输出格式

Task Prompt
  - 当前任务目标、上下文、约束、产物要求

SoulProjection
  - 把以上内容收束成当前灵魂可用的运行时表达
```

一句话：

> prompts 不再散落成互相抢权的段落，而是被 SoulProjection 收束成当前灵魂的一套运行时视图。

### 3.1 PromptSourceRegistry

为了真正收束 prompts，需要新增一个 prompt 来源登记层。

它不负责写 prompt，只负责记录 prompt 从哪里来、归谁管、能不能进入当前运行时视图。

建议来源：

| source_type | 来源 | 所属层 |
| --- | --- | --- |
| `core` | `backend/soul/agent_core/CORE.md` | 全局底线 |
| `soul_seed` | `backend/soul/agent_core/seeds/*.md` 或 `custom/*/SOUL.md` | 灵魂叙事 |
| `soul_profile` | `profile.json` | 灵魂结构化契约 |
| `default_projection` | ProjectionCard / DefaultProjection registry | 默认投影预设 |
| `agent_profile` | 后续 AgentProfile registry | agent 身份 |
| `skill_contract` | SkillRegistry | 能力契约 |
| `tool_contract` | ToolRegistry / ResourcePolicy | 工具说明 |
| `task_contract` | TaskContract | 当前任务 |
| `memory_policy` | MemoryPolicy | 记忆边界 |
| `output_contract` | OutputContract | 输出要求 |

进入 prompt 前必须经过：

```text
PromptSource
  -> SoulProjectionBuilder
  -> SoulRuntimeView
  -> SoulPromptAssembler
  -> PromptManifest
```

`PromptManifest` 要记录每段 prompt 的来源，方便测试系统、调试系统和前端观察：

```text
section_id
source_type
source_id
owner_layer
cache_scope
visible_to_model
hash
```

这样以后 prompt 不再是散的，而是可追踪、可解释、可替换的。

### 3.2 静态准则与动态任务契约

共同准则也要拆成两层：

```text
Static Common Rules
  来源：CORE.md / 共同底线
  特点：稳定、可缓存、所有灵魂共享
  例子：事实边界、权限边界、输出底线、不能泄露内部链路

Dynamic Task Contract
  来源：TaskContract / 当前任务绑定
  特点：每个任务不同、不可长期污染、任务结束后失效
  例子：本轮目标、final_owner、允许 skills、可见 tools、输出格式、审查要求
```

强约束：

```text
静态准则不能被投影卡片改写。
动态任务契约不能写回 seed。
投影卡片可以引用动态契约，但不能把动态契约变成长期人格。
```

## 4. 灵魂能看到工具和 Skills

灵魂应该能看到：

```text
当前任务允许哪些 skills。
当前 skills 允许哪些 tools。
这些 tools 适合什么任务。
这些 tools 有什么风险。
当前任务为什么允许或不允许它们。
```

但灵魂看到不等于灵魂授权。

边界：

```text
SoulProjection can describe tools.
ResourcePolicy authorizes tools.
RuntimeDirective invokes tools.
ToolExecutor executes tools.
```

示例：

```text
祝融看到 coding/refactor skill：
  它会以行动推进方式组织实现步骤。

玄女看到 testing/report skill：
  它会优先检查失败原因、边界和风险。

四岳看到 writing skill：
  它会优先组织章节、结构和论证顺序。

河伯看到 rag skill：
  它会优先收集资料、整理上下文和证据。
```

但是：

```text
祝融不能因为自己适合行动就自动获得 terminal 权限。
玄女不能因为负责审查就自动读取所有文件。
河伯不能因为负责收集就自动检索所有来源。
四岳不能因为负责组织就修改任务结构。
```

### 4.1 Skill 视图

灵魂看到的 skill 不是完整内部实现，而是当前任务允许的 skill 摘要。

建议结构：

```text
SoulSkillView
  skill_id
  title
  capability_summary
  use_when
  input_boundary
  output_boundary
  forbidden_uses
  current_task_reason
```

示例：

```text
河伯 + rag skill:
  看到“可以检索本地知识库、返回证据摘要、需要标注来源”。
  看不到底层索引细节，也不能自己改变检索 worker。
```

### 4.2 Tool 视图

灵魂看到的 tool 也不是工具权限，而是工具说明。

建议结构：

```text
SoulToolView
  tool_id
  title
  capability_summary
  input_schema_summary
  output_schema_summary
  risk_summary
  authorized: true | false
  authorization_owner: ResourcePolicy
```

即使 `authorized=false`，某些场景下也可以让灵魂看到“这个工具存在但当前任务不可用”，用于解释为什么不能执行某些动作。

但如果 `ResourcePolicy` 判定工具不可见，则 `SoulToolView` 不能泄露该工具存在。

### 4.3 授权链

固定授权链：

```text
TaskContract
  -> TaskCapabilityRequirement
  -> SkillRegistry 产出 skill candidates
  -> ResourcePolicy 产出 allowed tools
  -> SoulProjection 只接收 allowed/visible summaries
  -> RuntimeDirective 才能发起调用
```

不能出现：

```text
SoulProfile -> ToolExecutor
SoulProjection -> ToolExecutor
SOUL.md -> RuntimeDirective
自制灵魂 profile.json -> tool permission
```

## 5. 灵魂管理系统

建议新增：

```text
backend/soul/contracts.py
backend/soul/registry.py
backend/soul/projection.py
backend/soul/prompt_assembly.py
backend/soul/custom/
```

### 5.1 SoulRegistry

职责：

```text
加载默认五灵魂。
加载用户自制灵魂。
校验灵魂契约。
提供 soul catalog。
支持启用/禁用。
支持 active soul。
支持版本与来源标记。
```

### 5.2 SoulCatalog

对外展示：

```text
soul_id
name
source: builtin/user
enabled
active
default_projection_ids
portrait
description
validation_errors
```

### 5.3 SoulProjectionBuilder

职责：

```text
输入 SoulProfile + TaskContract + AgentProfile + SkillScope + ResourcePolicy
输出 SoulRuntimeView
```

### 5.4 SoulPromptAssembler

职责：

```text
把 SoulRuntimeView 组装成 prompt sections。
保持 static/dynamic/volatile 分段。
避免破坏 prompt cache。
```

### 5.5 SoulStateStore

后续需要一个轻量状态存储层，用于管理灵魂状态，而不是把状态塞进 seed。

职责：

```text
保存 active soul。
保存 user custom soul enabled 状态。
保存 soul version / checksum。
保存最近一次 projection preview。
保存校验错误。
```

它不保存：

```text
不保存任务结果。
不保存长期记忆。
不保存工具调用结果。
不保存 ControlKernel 采纳结果。
```

### 5.6 SoulEventLog

灵魂系统需要可观察性。

建议记录：

```text
soul_registered
soul_enabled
soul_disabled
soul_switched
soul_projection_built
soul_projection_rejected
custom_soul_validation_failed
```

这些事件给前端、测试报告和调试系统看，不进入模型 prompt。

## 8. 单线程与多灵魂协作

### 8.1 单线程任务

单线程任务中：

```text
一个灵魂承载完整 agent。
切换灵魂主要改变风格、背景、思考方式和 prompt 投影。
所有灵魂都具备基础对话能力。
能力边界仍由 AgentProfile / Skill / Policy 决定。
```

例如：

```text
用户选择祝融进行单线程对话：
  祝融可以回答、分析、解释、写作、规划。
  只是它更倾向行动、推进、落地。
```

### 8.2 多智能体协作

多智能体任务中：

```text
灵魂可以分身。
同一个 SoulProfile 可以投影成多个 SoulRole。
不同 SoulRole 产生不同 artifact。
由任务绑定的 final_owner 或当前激活灵魂负责最终归口。
```

写作例子：

```text
Task: 写作长文

当前激活灵魂 / dialogue+govern
  -> 确认目标、分派角色

河伯 / collect
  -> 收集材料、整理背景

四岳 / organize
  -> 搭大纲、定章节

祝融 / execute
  -> 写初稿、推进段落

玄女 / inspect
  -> 审查逻辑、歧义、漏洞

final_owner / govern
  -> 统一口径，形成最终稿
```

### 8.3 协作协议

多灵魂协作必须有协议，不是把几个 prompt 拼在一起。

一次协作应显式生成：

```text
CollaborationPlan
  task_id
  collaboration_mode
  lead_soul_id
  roles[]
  artifact_contracts[]
  merge_policy
  review_policy
  final_owner
```

每个 role：

```text
SoulRole
  role_id
  soul_id
  role_type
  agent_profile_id
  task_mode
  expected_artifact
  allowed_skills
  visible_tools
  handoff_target
```

中间产物：

```text
SoulArtifact
  artifact_id
  produced_by_role_id
  artifact_type
  summary
  refs
  confidence
  review_status
```

最后统一由 `final_owner` 归口；如果任务没有显式绑定，则由当前激活灵魂承载最终收束。

强约束：

```text
多灵魂协作只产生候选 artifact。
最终答案必须通过任务绑定的 `final_owner` 投影，或当前激活灵魂投影收束。
玄女类审查投影可以否决 artifact，但不能直接提交最终答案，除非被绑定为 final_owner。
祝融类执行投影可以推进实现，但不能跳过审查/提交门。
河伯类收集投影可以提供证据，但不能把证据直接变成最终事实。
四岳类组织投影可以改写结构，但不能改写用户目标。
```

## 9. 多态投影

多态投影是主打设定。

同一个灵魂在不同任务里呈现不同运行时视图：

```text
祝融 + implementation
  -> 行动型执行投影

祝融 + writing_draft
  -> 快速初稿投影

祝融 + risk_review
  -> 行动风险审查投影

祝融 + general_qa
  -> 行动型对话投影
```

投影输入：

```text
soul_id
projection_name
role_type
task_mode
agent_profile_id
task_contract_summary
allowed_skills
allowed_tools
memory_policy
output_contract
```

投影输出：

```text
SoulRuntimeView
PromptSections
Trace metadata
ProjectionCard
```

## 10. 与任务系统的联动

任务系统负责先定：

```text
TaskFamily
TaskMode
TaskSoulRequirement
```

灵魂系统负责提供：

```text
SoulProfile
SoulRole
SoulSelectionCandidate
SoulProjection
```

ControlKernel 负责采纳：

```text
TaskSoulBinding
AgentProfile
SkillScope
ResourcePolicy
ExecutionGraph
```

联动顺序固定为：

```text
UserRequest
  -> TaskContract
  -> TaskMode
  -> TaskSoulRequirement
  -> SoulSelectionCandidate
  -> AgentProfileCandidate
  -> SkillScopeCandidate
  -> ResourcePolicy
  -> SoulProjection
  -> ControlKernel adopts RuntimeDirective
```

不要反过来让灵魂先决定任务。

灵魂本体不再提供任务偏好。

默认投影或投影卡片可以提供候选排序信号：

```text
default_projection_ids
projection_role_tags
recommended_task_families
projection_risk_notes
```

但这些只能作为候选排序信号，不能成为调度事实。

## 15. 当前定稿原则

```text
灵魂是智能体的意志系统。
切换灵魂是单线程智能体的多态变化。
多灵魂协作是复杂任务的分身协作。
SoulProjection 负责收束 prompt。
ControlKernel 负责采纳与授权。
Skill / Tool 只通过可见摘要进入灵魂视图。
PromptManifest 负责让 prompt 可追踪。
```

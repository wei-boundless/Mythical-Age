# 206-角色标准专业三模式与专业长任务母版架构升级计划

日期：2026-05-21

状态：实施前计划

关联文档：

- `182-主Agent无图长任务与任务图统一运行底座方案-20260520.md`
- `204-主Agent自主任务真实长任务能力升级计划-20260521.md`

## 1. 核心结论

本轮升级不是给 `autonomous_task_run` 再补一个提示词，也不是在现有 `simple / standard / managed` 上继续堆策略分支。正确方向是：

```text
先把专业模式做成完整的长任务自然执行母版，
再由编排系统通过弱化配置降档为标准模式和角色模式。
```

最终只保留一套主 Agent 无图任务运行底座：

```text
Professional Runtime Capability
  -> professional_mode: 全能力长任务
  -> standard_mode: 当前回合工具任务
  -> role_mode: 灵魂/角色体验 + 只读轻能力
```

三种模式不是三个人格，也不是三套 driver。它们是同一编排运行能力在不同风险、工具、投影、上下文、验证强度下的策略档。

正式命名：

| 内部模式 | 中文名 | 定位 |
|---|---|---|
| `role_mode` | 角色模式 | 灵魂系统主场，负责对话、陪伴、记忆、轻问答、只读检索 |
| `standard_mode` | 标准模式 | 当前回合内用有限工具解决明确问题 |
| `professional_mode` | 专业模式 | 语义契约、专业 prompt、动态计划、证据包、checkpoint、验证和收口的长任务模式 |

必须清理的旧概念：

| 旧概念 | 处理 |
|---|---|
| `simple` 作为正式模式 | 删除，不再作为用户或架构概念 |
| `managed` 作为正式模式 | 删除，被 `professional_mode` 吞并为专业模式的强策略 |
| `autonomy_mode=simple/standard/managed` | 改为 `interaction_mode=role_mode/standard_mode/professional_mode`，必要策略强度写入 `mode_policy` |
| driver 内部按 simple/standard 分叉 | 收敛成一个 `ProfessionalTaskRunDriver` 或统一 `AutonomousTaskRunDriver` 状态机 |
| driver 末端正则重猜任务语义 | 删除，改为消费上游 `SemanticTaskContract` |

注意：TaskGraph 内已有 `standard_input_package` / `standard_node_result_package` 等协议名，与本轮 `standard_mode` 不是同一层概念，不应误删。

## 2. 现有源码链路审查

### 2.1 当前真实链路

当前主 Agent 运行链路如下：

```text
agent_runtime_chain
  -> collect_intent_frame / decide_intent
  -> build_runtime_assembly_hint
  -> analyze_query_understanding
  -> ContextResolver
  -> build_task_execution_assembly_bundle
  -> resolve_execution_shape
  -> build_execution_recipe
  -> build_orchestration_runtime_bundle
  -> SoulFacade.build_runtime_view
  -> TaskRunLoop
  -> StageProjectionCycle
  -> RuntimeContextManager
  -> AutonomousTaskRunDriver 或普通 single-agent loop
```

关键文件：

| 文件 | 当前职责 |
|---|---|
| `backend/orchestration/agent_runtime_chain.py` | 串联 intent、query understanding、context、task assembly、orchestration assembly |
| `backend/intent/signal_collector.py` | 生成 `IntentFrame`，目前以 marker 判断复杂度和候选策略 |
| `backend/intent/hypothesis_builder.py` | 从候选策略取第一个作为 `IntentDecision.execution_strategy` |
| `backend/intent/action_planner.py` | 把 execution strategy 映射成 runtime assembly hint |
| `backend/tasks/assembly_support.py` | 构造 `TaskIntentContract`，当前承载输入、输出、能力请求，不承载专业语义任务结构 |
| `backend/tasks/execution_shape_resolver.py` | 选择 recipe，当前 `autonomous_task_run` 分支只保留运行标签和 `autonomy_mode` |
| `backend/tasks/execution_recipe_builder.py` | 生成 recipe，当前 `runtime.recipe.autonomous_task_run` 仍按 `simple/standard/managed` 配置 |
| `backend/orchestration/assembly_builder.py` | 生成 runtime lane、projection requirement、resource policy、SoulRuntimeView |
| `backend/soul/runtime_assembly.py` | 把 task prompt contract、projection requirement、resource view 组装成 `SoulRuntimeView` |
| `backend/orchestration/runtime_loop/stage_projection.py` | 过滤投影为 runtime-safe prompt sections |
| `backend/orchestration/runtime_loop/context_manager.py` | 把 base prompt、projection、context、runtime assembly 拼成最终 model messages |

| `backend/orchestration/runtime_loop/autonomous_task_run_driver.py` | 当前无图自主任务 driver，含 goal contract、工具循环、verification、closeout repair |

### 2.2 灵魂系统与投影机制

灵魂系统已经不是简单 prompt 文本，而是通过 `SoulRuntimeView` 和 `PromptManifest` 进入编排系统。

关键结构：

| 结构 | 位置 | 说明 |
|---|---|---|
| `SoulProfile` | `backend/soul/contracts.py` | 灵魂身份、表达风格、偏好 |
| `SoulProjectionRequest` | `backend/soul/contracts.py` | 投影请求，包含 role_type、task_mode、skill/tool views |
| `PromptSection` | `backend/soul/contracts.py` | 模型可见 prompt section 单元 |
| `SoulRuntimeView` | `backend/soul/contracts.py` | 本轮模型可见投影视图 |
| `SoulPromptManifest` | `backend/soul/contracts.py` | prompt section manifest 和 hash |
| `SoulRuntimeAssemblyBuilder` | `backend/soul/runtime_assembly.py` | runtime 侧生成投影视图 |
| `StageProjectionCycle` | `backend/orchestration/runtime_loop/stage_projection.py` | 运行期过滤 control-plane sections |
| `RuntimeContextManager` | `backend/orchestration/runtime_loop/context_manager.py` | 最终拼接 projection block |

当前 `projection_requirement` 已包含：

- `role_type`
- `posture_tags`
- `expression_density`
- `attention_focus`
- `projection_id`
- `soul_id`
- `identity_anchor`
- `projection_prompt`
- `task_mode`

但缺少本轮三模式所需字段：

- `interaction_mode`
- `projection_strength`
- `professional_profile_id`
- `semantic_task_type`
- `mode_policy_ref`

### 2.3 当前长任务失败的结构根因

以 `autonomous-test-report-triage` 为例：

1. `intent_frame.task_complexity=short`。
2. `execution_strategy_candidates=["specialist_handoff","single_react_loop"]`。
3. 后续被 `intent_decision` / task_selection 压成 `autonomous_task_run`。
4. driver 真实读到了测试报告。
5. 但工具观察没有编译成证据包。
6. 计划只有 `material_review / synthesize_answer / final_check` 一类泛化步骤。
7. final response 没形成失败归类、结构性根因、回归测试建议。
8. closeout repair 太晚，且仍可能输出伪 DSML。

结论：

```text
系统启动了长任务运行模式，但没有获得“正在执行哪一种专业任务”的语义结构。
```

因此下一步必须补的是：

- 语义任务契约
- 专业任务 prompt profile
- 语义计划
- 证据包
- 交付物 validator
- 模式策略统一配置

而不是继续在末端 closeout repair 里加任务文案。

## 3. 目标架构

### 3.1 总体架构

```text
User Turn
  -> IntentFrame
  -> SemanticTaskContract
  -> RuntimeInteractionModePolicy
  -> ExecutionShape / Recipe
  -> OrchestrationRuntimeBundle
  -> SoulRuntimeView + PromptManifest
  -> StageProjectionSnapshot
  -> TaskRunLoop
  -> ProfessionalTaskRunDriver
  -> EvidencePacket
  -> DeliverableValidator
  -> OutputBoundary
```

新增核心对象：

| 对象 | 所属层 | 作用 |
|---|---|---|
| `SemanticTaskContract` | tasks / understanding | 结构化表达用户目标、任务类型、交付物、推理步骤、材料策略 |
| `RuntimeInteractionModePolicy` | orchestration | 三模式的单一策略源，决定 lane、工具、投影、上下文、checkpoint、验证 |
| `ProfessionalPromptProfile` | prompting / orchestration | 专业任务角色 prompt，不替代灵魂，只提供任务职责 |
| `SemanticPlan` | runtime loop | 从 semantic contract 生成可执行阶段 |
| `EvidencePacket` | runtime loop | 工具/委派/材料观察后的结构化证据 |
| `DeliverableValidator` | output / runtime loop | 按任务类型检查最终交付物，不再只查关键词 |

### 3.2 三模式不是三套 driver

统一 driver 能力：

```text
ProfessionalTaskRunDriver
  goal_lock
  semantic_plan_drafted
  material_review
  evidence_packet_built
  plan_item_execution
  synthesis
  deliverable_validation
  repair_or_commit
```

模式只改变策略：

| 策略项 | `role_mode` | `standard_mode` | `professional_mode` |
|---|---|---|---|
| driver | 轻量走同一入口，可直接 model finalize | 同一 driver，短计划 | 同一 driver，全状态机 |
| semantic contract | 轻量 | 必需但可简化 | 强制 |
| professional profile | 一般无，灵魂优先 | 可选 | 强制 |
| evidence packet | 可选 | 简化 | 强制 |
| checkpoint | terminal only | key boundary | every phase |
| tool rounds | 0-1 只读 | 少量 | 多轮受控 |
| write/edit/shell | 禁止 | 限定 | sandbox + gate |
| delegation | 禁止 | 默认禁用 | 受控允许 |
| validator | answer boundary | basic deliverable check | strict deliverable validator |

## 6. ProfessionalPromptProfile 设计

### 6.1 位置

建议新增：

```text
backend/prompting/professional_profiles.py
```

或如果更偏编排，可放：

```text
backend/orchestration/professional_prompt_profiles.py
```

### 6.2 原则

1. 专业 prompt 是 task-facing，不是开发说明。
2. 专业 prompt 不替代灵魂投影。
3. 专业 prompt 不包含工具协议细节。
4. 专业 prompt 必须和 `SemanticTaskContract.deliverables` 同源。
5. 不允许在 closeout repair 里写一次性硬编码专业文案。

### 6.3 示例：测试报告诊断员

```text
你是一名自主任务测试报告诊断员。

你只负责分析用户指定的测试产物、长跑报告或失败摘要，找出失败分类、结构性根因和应该补充的回归测试。

你不负责修改代码，也不负责编造未执行的测试结果。

你需要先从材料中提取失败项，包括失败 turn、失败检查、表面症状和原始证据。
然后把失败项归类到系统层问题，例如 memory、context、artifact、writeback、approval、tool loop、output boundary、timeout 或 runtime checkpoint。
最后给出结构性根因判断，并说明为什么这些不是孤立失败。

最终回答必须包含：
1. 失败归类。
2. 结构性根因。
3. 应该补充的回归测试。
4. 证据不足或仍需确认的边界。

不要只复述表面失败项。
不要输出工具调用、DSML、参数片段或内部协议。
```

### 6.4 注入位置

注入顺序：

```text
base system prompt
  -> soul projection sections
  -> semantic task contract section
  -> professional prompt profile section
  -> runtime mode policy section
  -> current user message
```

需要修改：

- `backend/soul/runtime_assembly.py`
- `backend/orchestration/assembly_builder.py`
- `backend/orchestration/runtime_loop/context_manager.py`

建议在 `SoulRuntimeView.sections` 中新增 section：

```text
section_id="professional_profile_section"
owner_layer="task"
source_type="professional_prompt_profile"
cache_scope="dynamic"
```

## 7. 投影强度设计

新增 `projection_strength`：

| projection_strength | 说明 |
|---|---|
| `primary` | 灵魂投影主导体验，用于角色模式 |
| `companion` | 灵魂提供语气与协作感，任务契约优先，用于标准模式 |
| `style_only` | 专业职责优先，灵魂只影响表达温度，用于专业模式 |

### 7.1 role_mode

`identity_view`、`role_view`、`projection_section` 权重最高。

### 7.2 standard_mode

保留灵魂表达，但 `task_section`、`workflow_section`、`output_section` 优先级高于灵魂姿态。

### 7.3 professional_mode

专业 profile 和 semantic contract 优先。灵魂不能覆盖：

- 专业职责
- 工具边界
- deliverable schema
- validation schema
- 不伪造测试结果的规则

## 8. Driver 重构方案

### 8.1 删除旧分叉

当前 `backend/orchestration/runtime_loop/autonomous_task_run_driver.py` 中存在：

- `run_simple_stream`
- `run_standard_stream`
- `_simple_control_plan`
- `_standard_control_plan`
- `_standard_execution_mode`
- `_with_simple_autonomous_task_instruction`
- `_build_goal_contract`
- `_response_terms_from_goal`
- 多处基于 `autonomy_mode` 的分叉

本轮目标不是继续兼容这些分叉，而是清理为：

```text
run_stream(...)
  -> mode_policy
  -> semantic_contract
  -> semantic_plan
  -> execute according to policy
```

如果保留 `AutonomousTaskRunDriver` 文件名以降低文件移动风险，也必须在内部删除 simple/standard/managed 语义。

推荐新名字：

```text
ProfessionalTaskRunDriver
```

但如果改名影响面过大，可第一阶段保留文件名，类名改为：

```python
class ProfessionalTaskRunDriver:
```

并在 `__init__.py` / `task_run_loop.py` 同步替换。

### 8.2 新状态流

```text
initialized
  -> mode_policy_bound
  -> semantic_contract_bound
  -> professional_profile_bound?
  -> semantic_plan_drafted
  -> plan_item_selected
  -> action_dispatched
  -> observation_received
  -> evidence_packet_built
  -> plan_item_validated
  -> synthesis_ready
  -> deliverable_validation_ready
  -> finalizing
  -> committed | blocked | failed
```

### 8.3 角色模式降档

角色模式仍可走 `run_stream`，但策略会短路：

```text
mode_policy_bound
  -> semantic_contract_bound(light)
  -> model_response
  -> answer_boundary_check
  -> committed
```

不能再保留单独 `run_simple_stream`。

```python
    authority: str = "orchestration.evidence_packet"
```

### 9.3 测试报告诊断证据包

应从 JSON 中抽取：

- `run_id`
- `total_turns`

- `failed_turns`
- 每个 failure 的 `turn`
- `check`
- `symptom`
- `evidence`

这一步不是最终回答，而是 driver 的中间产物。

## 14. 代码风格要求

本轮实现必须遵守：

1. 不新增平行长任务系统。
2. 不用兼容为理由保留无用旧链路。
3. 不把开发说明写进 agent prompt。
4. 不用关键词表替代结构化任务契约。
5. 不在 driver 末端重新正则理解整个用户目标。
6. 不伪造测试、工具、文件写入或验证结果。
7. 不为单个失败样本硬编码答案。
8. 新增模型必须有明确 owner、authority、to_dict。
9. 三模式配置必须由一个 `RuntimeInteractionModePolicy` 统一输出。
10. 专业 prompt 必须从 profile registry 读取，不散落在 closeout repair 中。

## 15. 风险与控制

| 风险 | 控制 |
|---|---|
| 一次性删除旧链路导致回归大 | 先写 policy 和 semantic contract 测试，再删 driver 分叉 |
| role_mode 失去灵魂体验 | 用 `projection_strength=primary` 保证灵魂投影主导 |
| standard_mode 变成半吊子 professional | 明确工具预算和无 checkpoint 强恢复 |
| professional_mode prompt 过强覆盖灵魂 | `projection_strength=style_only`，不是关闭灵魂 |
| validator 过硬导致正常回答失败 | validator 只对 `professional_mode` 严格启用 |
| 旧 TaskGraph standard 命名误删 | 明确 `standard_input_package` 不属于本轮删除范围 |
| 测试被硬编码绕过 | system eval 使用真实 artifact，validator 检查 evidence alignment |

## 16. 成功定义

本轮完成后，系统应具备：

1. 清晰的三模式：
   - 角色模式
   - 标准模式
   - 专业模式
2. 专业模式是真正长任务母版，而不是更大工具预算。
3. 标准模式和角色模式由专业模式能力降档配置出来。
4. 灵魂系统通过投影强度自然参与三模式，而不是被绕过。
5. 测试报告诊断长场景能自然完成：
   - 读材料
   - 建证据包
   - 归类失败
   - 找结构性根因
   - 给回归测试
   - 通过 deliverable validator
6. 新代码中没有 `simple/managed` 正式路径。
7. 旧链路被删除或迁移干净，trace、prompt manifest、recipe metadata 使用新命名。

最终判断：

```text
这是一轮底层架构重构，不应以小补丁推进。
先立专业模式母版，再删旧分叉，再配置降档。
如果保留旧 simple/standard/managed 与新三模式长期并存，系统会继续在意图理解、投影、driver 和验证之间漂移。
```

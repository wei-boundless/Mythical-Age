# 208-写作任务流程记忆防污染与持续运行优化方案

日期：2026-05-21

状态：实施前优化方案，暂不修改运行时代码

适用范围：

- `graph.writing.modular_novel.master`
- `graph.writing.modular_novel.design_init`
- `graph.writing.modular_novel.chapter_cycle`
- `graph.writing.modular_novel.finalize`
- 后续编排系统升级完成后的专业长任务运行母版接入

## 1. 目标

本方案解决的不是单个 prompt 写得不够专业的问题，而是长篇写作任务能否稳定、可追踪、低污染、可持续运行的问题。

目标是建立一套写作任务流程优化方案，让系统在百万字级中文商业网文创作中满足四个条件：

1. 流程职责清楚：设计、生产、审核、提交、路由、监测互不越权。
2. 行为污染可控：候选、审核意见、动态策略、用户聊天、运行日志不能混成 canon。
3. 记忆读写无误：只有被授权节点读取指定版本，只有提交节点写入指定仓库，所有写入可追溯。
4. 长任务持续高效：章节批次、卷级循环、返修、恢复、扩展提案和最终交付都有明确状态和验收。

本方案不直接修改当前编排系统代码。原因是编排系统正在升级，写作图的优化应先沉淀为稳定结构，等运行底座升级完成后一次性接入，避免在旧链路上做补丁。

### 2.3 旧设计文档中的关键原则

来源：`docs/系统规划/168-写作任务设计文件夹-20260519`

已形成但未完全落地的原则：

1. 下游只消费 `accepted` / `committed`。
2. `candidate` 和 `review` 只能用于本轮返修，不得直接外溢。
3. 基准库默认只读，动态库不得反向覆盖基准库。
4. 章节批次交接必须能追溯到单章，不能只登记批次。
5. 伏笔、悬念、关系推进必须在大纲中结构化表达；运行层只能生成派生的线程追踪视图，不能另起剧情事实源。
6. 起步阶段必须把用户输入转成长期创作工程底盘，而不是直接写世界观。
7. 三阶段任务图应拆分为设计、创作、收尾，协议可以复用，但时序不能混用。
8. 前端应按总览、节点、资源、时间线、运行监测分层展示，不把层级混在一页。

### 2.4 206 架构升级对本方案的约束

来源：`docs/系统规划/206-角色标准专业三模式与专业长任务母版架构升级计划-20260521.md`

写作任务后续应接入 `professional_mode`，核心借用：

- `SemanticTaskContract`
- `ProfessionalPromptProfile`
- `EvidencePacket`
- `DeliverableValidator`
- 明确的 `restore / decide / execute / present` 边界

写作任务不应绕过新的专业长任务母版，也不应把写作专用 shortcut 塞进后端。

## 3. 当前主要设计缺口

### 3.1 图上有记忆边，但运行时可见性仍需验证

当前写作节点声明了 `readable_repositories`、`memory_read_policy` 和 memory read edge。但运行装配层会按 agent runtime profile 过滤上下文 section。

`runtime_assembly_builder.py` 中，记忆快照 section 的 `profile_context_section` 是 `working_memory`。当前写作 agent runtime profile 的 `allowed_context_sections` 包含 `memory_runtime_view`，但未显式包含 `working_memory`。这会造成一种风险：

```text
任务图声明可读记忆
  -> context resolver 解析出 memory_snapshot
  -> runtime assembly 生成 memory_snapshot section
  -> agent profile 过滤时隐藏该 section
  -> agent 实际看不到授权记忆
```

这类问题不能靠 prompt 修复，必须纳入运行装配验收。

### 3.2 候选、审核、提交的状态边界还不够强

当前节点有 `write_mode`：

- `candidate_archive_only`
- `review_and_issue_ledger`
- `baseline_commit`
- `chapter_commit`
- `volume_commit`
- `dynamic_memory_commit`
- `finalize_commit`

方向正确，但还需要把状态对象本身显式化：

| 状态 | 允许用途 | 禁止用途 |
|---|---|---|
| `candidate` | 给审核和返修使用 | 不能进 canon，不能给下游当事实 |
| `review` | 裁决、问题、返修指令 | 不能替设计节点补写设定 |
| `approved_slice` | 审核通过后允许提交的切片 | 不能包含审核员新编内容 |
| `committed` | 下游可读事实来源 | 不能无版本覆盖 |
| `frozen` | 长期不变量 | 不能被动态库覆盖 |
| `dynamic_delta` | 阶段性推进策略和已坐实增量 | 不能改写基准事实 |

如果没有这些状态，系统容易把“看似完整的审核意见”当成“已确认事实”继续写。

### 3.3 起步阶段仍然偏压缩

当前流程已经比早期简单写作图强很多，但 `memory_commit_world -> character_design / plot_design -> design_sync -> outline_review -> baseline_memory_seed` 仍然缺少一个明确的“创作架构提交包”概念。

旧设计文档建议的三个节点仍然有必要：

```text
startup_architecture_seed
  -> startup_architecture_review
  -> startup_architecture_commit
```

它们不一定立即拆成多个真实节点，但必须成为协议层：

- 人设架构协议
- 大纲架构协议
- 大纲线程表达协议
- 线程追踪视图协议
- 分库协议

否则 `baseline_memory_seed` 会承担过多隐性判断，后续章节循环会继承污染。

### 3.4 伏笔悬念不应脱离大纲成为独立事实源

旧设计中提出过线程账本，用来追踪：

- 伏笔
- 悬念
- 关系线
- 世界规则承诺
- 回收窗口
- 状态：`proposed / active / due / paid / deferred / cancelled / invalidated`

这个方向能解决追踪问题，但如果把线程账本做成独立 canon 仓库，会产生新的污染风险：大纲是一套剧情设计，线程账本又是一套剧情设计，后续节点可能不知道谁才是权威来源。

长篇商业网文最容易失控的不是单章内容，而是：

- 伏笔丢失
- 悬念提前泄露
- 关系线断裂
- 承诺过的规则没有回收
- 卷级节奏和章节推进脱节

因此本方案调整为：

```text
大纲负责表达伏笔、悬念、关系推进和回收窗口
  -> 运行层从已审核大纲中派生线程追踪视图
  -> 章节提交和卷级提交只更新兑现证据与状态
  -> 任何新增、取消、延期或改写都回到大纲/扩展审核链路
```

也就是说，线程追踪视图是检查工具，不是剧情设计源；它可以缓存、索引、预警，但不能独立生成新的伏笔或改写大纲。

### 3.5 动态记忆与基准记忆的冲突处理还不够显式

当前策略已有“动态库不得覆盖基准库”的原则，但执行上还应要求：

```text
dynamic_delta 与 baseline 冲突
  -> 写入 issue_ledger
  -> 生成 extension_proposal 或 blocker
  -> extension_review 裁决
  -> 只有 extension_commit 可写入动态库
  -> 永不直接改 baseline
```

基准库如果确实需要变更，必须走更高权限的“基准变更提案”，不能由卷后动态提案顺手改写。

### 3.6 长任务循环目前以进度为主，质量和健康控制不足

当前章节循环有字数、批次和卷数控制，这是必要但不充分的。

长篇写作还需要：

- 批次质量趋势
- 连续返修次数
- 缺失记忆次数
- 线程到期未回收数量
- 动态库冲突数量
- 审核通过率
- 章节摘要和正文引用完整性
- 卷级阅读体验是否持续下降

这些应进入运行监测和路由，不应只依赖字数推进。

## 4. 推荐设计方向

核心方向：

```text
把写作任务从“节点串联”升级为“状态隔离 + 协议包交接 + 提交型记忆”的创作工程流。
```

本方案不推翻当前四图架构，而是在其上增加四类强边界：

1. **状态边界**：候选、审核、提交、冻结、动态提案分离。
2. **记忆边界**：基准库、动态库、问题账本、产物索引分离；伏笔悬念由大纲表达，线程追踪视图只做派生检查。
3. **上下文边界**：节点只读取授权 stage packet 和 memory snapshot。
4. **运行边界**：设计、章节、卷后、收尾使用不同协议和验收，不混用。

## 5. 目标流程总览

### 5.1 总图保持四单元

```text
master
  -> design_init
  -> chapter_cycle
  -> finalize
```

四图不变，避免在编排系统升级期间推翻现有结构。

### 5.2 设计初始化图优化为三段

目标结构：

```text
project_brief
  -> world_design
  -> world_review
  -> memory_commit_world
  -> startup_architecture_seed
  -> startup_architecture_review
  -> startup_architecture_commit
```

当前已有节点可先映射：

| 目标协议段 | 当前承载节点 |
|---|---|
| `startup_architecture_seed.character` | `character_design` |
| `startup_architecture_seed.plot` | `plot_design` |
| `startup_architecture_seed.sync` | `design_sync` |
| `startup_architecture_seed.outline` | `outline_design` |
| `startup_architecture_review` | `outline_review` + `design_sync` |
| `startup_architecture_commit` | `baseline_memory_seed` |

后续是否拆真实节点，取决于是否需要独立审计、独立返修、独立写权限。

### 5.3 章节循环图保持批次闭环

目标结构：

```text
volume_plan
  -> chapter_outline
  -> chapter_draft
  -> chapter_review
  -> memory_commit_chapter
  -> chapter_progress_router
```

必须强化：

- `chapter_outline` 只读取已提交的卷计划、接续包、线程包。
- `chapter_draft` 只写正文候选，不写下一批，不写记忆。
- `chapter_review` 只裁决，不重写正文。
- `memory_commit_chapter` 按单章提交，不得只提交批次摘要。
- `chapter_progress_router` 只按提交后的进度和健康指标路由。

### 5.4 卷后扩展图保持提案闭环

目标结构：

```text
volume_review
  -> volume_commit
  -> volume_postmortem
  -> world_outline_extension_proposal
  -> extension_review
  -> extension_commit
  -> next_volume_router
```

必须强化：

- `volume_postmortem` 只能从已成立正文提炼事实。
- `world_outline_extension_proposal` 只能生成动态增量提案。
- `extension_review` 检查是否越过基准边界。
- `extension_commit` 只能写动态库和派生线程追踪状态，不改 baseline，不直接改写大纲权威设计。

### 5.5 收尾图保持汇编闭环

目标结构：

```text
final_assemble
  -> final_review
  -> memory_finalize
```

必须强化：

- `final_assemble` 只消费 committed refs，不重新编故事。
- `final_review` 只裁决交付完整性、一致性和未解决问题。
- `memory_finalize` 只做归档提交，不生成新剧情。

## 6. 五类协议包

### 6.1 Stage Packet

每个节点启动时必须收到一个标准阶段包。

必含字段：

| 字段 | 说明 |
|---|---|
| `run_id` | 当前写作任务运行 |
| `graph_id` | 当前图单元 |
| `stage_id` | 当前阶段 |
| `node_id` | 当前节点 |
| `scope_path` | 当前卷、批次、章节范围 |
| `authorized_input_refs` | 被授权读取的上游产物 |
| `memory_snapshot_ref` | 本节点授权记忆快照 |
| `forbidden_change_refs` | 禁止改动的冻结边界 |
| `issue_refs` | 当前必须关注的问题 |
| `thread_refs` | 从已审核大纲和已提交正文派生出的当前线程追踪引用 |
| `output_contract_id` | 输出契约 |
| `write_permission` | 是否允许写记忆以及目标仓库 |

禁止：

- 节点自行读取全局聊天历史。
- 节点自行推断可写仓库。
- 节点把未授权上游产物当事实。

### 6.2 Continuity Packet

用于章节、批次、卷之间接续。

必含字段：

| 字段 | 说明 |
|---|---|
| `previous_committed_refs` | 上一批或上一卷已提交引用 |
| `current_state_summary` | 当前状态摘要 |
| `character_state_delta` | 人物状态变化 |
| `world_state_delta` | 世界状态变化 |
| `outline_state_delta` | 大纲推进变化 |
| `active_thread_refs` | 大纲中已生效、正文仍需承接的未闭合线程 |
| `due_thread_refs` | 按大纲回收窗口到期应推进或回收的线程 |
| `forbidden_changes` | 禁止改写项 |
| `reader_expectation` | 下一批读者期待 |

原则：

- 只基于 committed 结果生成。
- 摘要不能替代单章事实引用。
- 未审核内容不得进入接续包。

### 6.3 Revision Packet

用于返修闭环。

必含字段：

| 字段 | 说明 |
|---|---|
| `original_candidate_ref` | 原候选稿 |
| `review_ref` | 审核意见 |
| `verdict` | 裁决 |
| `must_fix_items` | 必改项 |
| `keep_items` | 保留项 |
| `forbidden_changes` | 返修时禁止改动 |
| `return_to_node_id` | 返修目标节点 |
| `max_repair_rounds` | 最大返修轮数 |

原则：

- 返修必须回到原候选，不得另起一份。
- 审核员不能把自己的补写直接变成新候选。
- 超过最大返修轮数进入 blocker，不继续消耗循环。

### 6.4 Memory Snapshot

用于授权记忆读取。

必含字段：

| 字段 | 说明 |
|---|---|
| `snapshot_id` | 快照 ID |
| `node_id` | 读取节点 |
| `read_edge_ids` | 触发读取的记忆边 |
| `repository_refs` | 可读仓库 |
| `resolved_record_refs` | 读取到的记录引用 |
| `resolved_records` | 读取到的结构化记录 |
| `formal_memory_record_refs` | 正式记忆版本 |
| `missing_required_records` | 缺失的必需记录 |
| `visible_to_agent` | 是否进入 agent 可见上下文 |
| `version_selector` | 版本选择策略 |

硬规则：

- 有 memory read edge 的节点，运行装配后必须能看到 `memory_snapshot` 或等价记忆输入项。
- 缺失 `required_memory_topics` 时节点必须 fail closed 或进入明确补救路径。
- 记忆快照必须带读取回执，便于后续追责。

### 6.5 Commit Packet

用于所有记忆写入。

必含字段：

| 字段 | 说明 |
|---|---|
| `commit_id` | 提交 ID |
| `source_candidate_ref` | 来源候选 |
| `source_review_ref` | 来源审核 |
| `allowed_write_targets` | 允许写入仓库和 collection |
| `approved_slices` | 允许固化的内容切片 |
| `rejected_slices` | 明确拒绝的内容 |
| `conflict_checks` | 与基准、动态、线程的冲突检查 |
| `write_receipts` | 写入回执 |
| `downstream_visibility` | 下游可见范围 |

原则：

- 只有 commit 节点能写 canon 或动态记忆。
- commit 节点不做创作，只做分库、裁剪、登记、版本化。
- `approved_slices` 必须来自审核通过内容，不能来自提交管家自由发挥。

## 7. 记忆所有权模型

### 7.1 仓库分工

| 仓库 | 所有者 | 写入来源 | 下游用途 |
|---|---|---|---|
| 基准库 | 提交节点 | `memory_commit_world`、`startup_architecture_commit`、受控基准变更提交 | 稳定 canon、冻结事实、长期主线 |
| 动态记忆库 | 提交节点 | `memory_commit_chapter`、`volume_commit`、`extension_commit` | 已成立章节事实、卷级增量、下一卷策略 |
| 大纲线程设计 | 大纲提交节点 | `startup_architecture_commit`、受控大纲扩展提交 | 伏笔、悬念、关系线、回收窗口的权威剧情设计 |
| 线程追踪视图 | 运行层派生索引 | 已审核大纲、已提交章节、卷级提交 | 检查线程是否承接、到期、兑现、延期或失效 |
| 产物索引库 | 所有产出节点 | 产物引用、审核记录、提交清单 | 查找候选、审核、正文、摘要、提交 |
| 问题台账 | 审核节点和监测节点 | 审核问题、冲突、阻塞、运行异常 | 返修、路由、fail closed |

大纲线程设计属于大纲的一部分，不能拆成独立剧情事实库。线程追踪视图可以作为 `memory.writing.mutable` 的派生 collection 或运行索引落地，但它只保存状态、证据和回收提醒，不保存新的剧情设计。

### 7.2 基准库规则

基准库只存：

- 用户硬设定。
- 已审核通过的世界底层规则。
- 冻结人物事实。
- 冻结关系事实。
- 全书主线不可推翻方向。
- 不可越界边界。

基准库不存：

- 候选设定。
- 审核员建议。
- 卷级临时策略。
- 未坐实的伏笔安排。
- 章节草稿内容。

### 7.3 动态记忆库规则

动态记忆库存：

- 已提交章节事实。
- 批次摘要。
- 卷级状态变化。
- 可调整人物权重。
- 可调整支线优先级。
- 已审核的扩展增量。
- 下一卷策略建议。

动态记忆库不允许：

- 覆盖基准库。
- 写入审核未通过的增量。
- 把“计划写”伪装成“已经发生”。
- 把被拒绝提案作为可读策略。

### 7.4 大纲线程设计与追踪视图规则

大纲必须表达：

- 伏笔。
- 悬念。
- 关系线。
- 世界规则承诺。
- 情绪承诺。
- 回收窗口。
- 可见性边界。
- 预期推进阶段。

线程追踪视图只保存：

- 大纲线程引用。
- 当前状态。
- 正文兑现证据。
- 延期、取消或失效原因。
- 需要审核关注的风险。

线程状态仍可使用：

```text
proposed -> active -> due -> paid
                  -> deferred
                  -> cancelled
                  -> invalidated
```

硬规则：

- 没有已审核大纲来源的线程不能 active。
- due 线程未处理时，章节或卷级审核必须提示，但不能自行改写大纲。
- paid 线程必须有正文或提交证据。
- deferred、cancelled、invalidated 必须走大纲扩展或卷级审核链路确认。
- 线程追踪视图不能新建伏笔、悬念或关系线，只能引用和更新已审核设计的执行状态。

## 8. 行为污染控制

### 8.1 污染来源

| 污染来源 | 典型表现 | 控制方式 |
|---|---|---|
| 用户聊天污染 | 把用户随口想法当硬设定 | `project_brief` 区分硬设定、倾向、开放问题 |
| 候选污染 | 未审核世界观进入章节 | 下游只读 committed refs |
| 审核污染 | 审核员补写内容变成事实 | 审核只输出裁决、问题、approved slices |
| 动态污染 | 卷后策略覆盖基准事实 | 动态冲突进入 extension review |
| 运行日志污染 | agent 根据 trace 自行补事实 | final 和写作节点只消费契约包 |
| prompt 污染 | 开发说明混进 agent 指令 | role prompt 只写专业职责，不写运行解释 |
| 类型污染 | 通用图主动塞题材元素 | 只允许方向维度，不允许无依据类型资产 |

### 8.2 prompt 治理原则

写作节点 prompt 必须继续遵守：

1. 写给 agent，不写给开发者。
2. 使用专业角色身份和工作职责。
3. 使用中文商业网文的共性工艺标准。
4. 可以对标头部作品的结构能力和商业判断。
5. 不能复刻具体作者的可识别句式、口癖、人物模板或专属设定。
6. 没有用户硬设定或项目题材依据时，不主动加入题材专属元素、套路资产或类型预设。
7. 用正向风格目标约束正文，明确现场感、动作推进、人物欲望、情绪回报、章末牵引等商业阅读目标，而不是只列负面禁令。

### 8.3 上下文治理原则

每个节点只允许看到：

- 当前 stage packet。
- 授权上游产物引用。
- 授权 artifact text expansion。
- 授权 memory snapshot。
- 本节点返修包。
- 本节点必要的问题台账和大纲线程追踪切片。

每个节点不允许看到：

- 原始全量聊天历史。
- 无关批次全文。
- 未来批次规划全文。
- 其他节点候选草稿。
- 未提交动态提案。
- 运行调试日志。

## 10. 固定执行流

### 10.1 节点启动

```text
router / scheduler
  -> resolve stage scope
  -> resolve authorized upstream refs
  -> resolve memory snapshot
  -> resolve outline-derived thread slice
  -> resolve issue slice
  -> build standard node input package
  -> runtime assembly visibility check
  -> dispatch agent
```

启动前必须检查：

- 必需上游结果是否 accepted 或 committed。
- 必需记忆是否读取成功。
- `memory_snapshot` 是否实际进入 agent 可见上下文。
- 当前节点是否拥有写权限。

### 10.2 节点产出

```text
agent output
  -> output contract parse
  -> artifact refs capture
  -> candidate / review / commit state tagging
  -> result package build
  -> acceptance / review / commit gate
```

产出后必须检查：

- 长文本是否有 artifact refs。
- 输出是否包含本节点 required fields。
- 节点是否越权写入记忆。
- 结果状态是否符合节点类型。

### 10.3 审核返修

```text
candidate
  -> review
  -> pass / pass_with_notes
      -> approved slices
      -> commit node
  -> revise
      -> revision packet
      -> original producer node
  -> reject / fail_closed
      -> issue ledger
      -> stop or user gate
```

### 10.4 提交入库

```text
approved slices
  -> commit packet
  -> conflict check
  -> repository write
  -> write receipt
  -> downstream visibility update
```

写入完成前，下游不得读取该内容。

### 10.5 恢复与继续

断点恢复时只能从最近一次 committed 或 accepted checkpoint 继续。

恢复顺序：

```text
load run state
  -> locate latest committed checkpoint
  -> rebuild scope path
  -> rebuild memory snapshot from committed refs
  -> rebuild continuity packet
  -> resume next eligible node
```

禁止：

- 从半截候选稿继续当成已通过。
- 从原始聊天历史重猜当前状态。
- 从运行日志里抽正文事实。

## 11. 长任务持续运行控制

### 11.1 Checkpoint 策略

必须 checkpoint：

- 世界观提交后。
- 创作架构提交后。
- 每批章节提交后。
- 每卷提交后。
- 扩展提交后。
- 最终审核后。

可选 checkpoint：

- 每次审核返修前。
- 每次连续性风险升高时。
- 每次动态库冲突出现时。

### 11.2 健康指标

建议增加运行健康指标：

| 指标 | 用途 |
|---|---|
| `missing_required_memory_count` | 发现记忆装配失效 |
| `hidden_memory_snapshot_count` | 发现 profile 过滤问题 |
| `revision_round_count` | 防止无限返修 |
| `rejected_candidate_count` | 发现设计质量问题 |
| `baseline_dynamic_conflict_count` | 发现动态覆盖风险 |
| `due_thread_count` | 防止伏笔拖欠 |
| `chapter_commit_gap_count` | 防止批次只提交摘要 |
| `artifact_missing_count` | 防止长文本丢引用 |
| `review_pass_rate` | 观察长期质量趋势 |
| `volume_quality_drift` | 观察卷级阅读体验下滑 |

### 11.3 路由策略

章节进度路由不应只看字数，还应看：

- 当前批次是否已提交。
- 每章是否有独立引用。
- 必需线程是否处理。
- 是否存在阻塞问题。
- 连续返修是否超限。

卷路由不应只看卷数，还应看：

- 卷级审核是否通过。
- 动态增量是否审核完成。
- due 线程是否全部处理或延期。
- 下一卷输入包是否完整。

### 11.4 批次效率

当前每批 10 章是合理初始值，但应允许基于健康指标调整：

| 情况 | 策略 |
|---|---|
| 返修率高 | 降低批次章节数 |
| 线程密度高 | 降低批次章节数 |
| 记忆读取稳定且质量通过率高 | 保持或提高批次章节数 |
| 输出长度频繁不足 | 触发正文扩写 repair，不改变 canon |
| 卷级质量漂移 | 暂停下一批，进入卷级诊断 |

### 12.2 EvidencePacket 映射

写作中的 evidence 不是测试证据，而是创作工程证据：

- 用户硬设定引用。

- 人设提交引用。
- 大纲提交引用。
- 单章正文引用。
- 单章摘要引用。
- 审核裁决引用。
- 线程状态引用。
- 记忆写入回执。

审核节点必须依据 evidence packet 裁决，不能只凭印象判断。

### 12.3 DeliverableValidator 映射

不同节点使用不同 validator：

| 节点类型 | validator |
|---|---|
| 世界观设计 | 是否覆盖商业化世界 Bible 必需维度，且无类型污染 |
| 世界观审核 | 是否有裁决、问题、风险、返修范围、approved slices |
| 章节正文 | 是否包含指定章节正文、承接、推进、摘要、字数指标 |
| 章节审核 | 是否逐章检查连续性、设定、商业阅读、返修要求 |
| 章节提交 | 是否每章独立登记 |
| 卷级审核 | 是否检查整卷连续阅读体验和线程状态 |
| 扩展提交 | 是否只写审核通过的动态增量 |
| 最终交付 | 是否所有 committed refs 完整且无未解决 blocker |

3. `baseline_memory_seed` 改为只提交审核通过的启动架构切片。
4. 要求大纲提交中包含伏笔、悬念、关系推进和回收窗口的结构化表达。
5. 从已审核大纲派生线程追踪视图 seed，但不把它作为独立剧情事实源。

完成标准：

- `volume_plan` 收到明确的 `volume_plan_input_packet`。
- 分卷不再重新解释用户需求或设计过程。

### 阶段 D：大纲线程追踪视图升级

目标：让大纲中的伏笔、悬念、关系线具备可检查的执行状态，同时保持大纲是权威剧情设计源。

范围：

- outline thread index 配置
- memory commit policy
- review validators

任务：

1. 从已审核大纲生成 `outline_thread_index` 派生索引。
2. 章节提交只更新线程执行状态和正文证据。
3. 卷审检查 due 线程，但不得自行改写大纲设计。
4. 下一卷路由检查未处理线程。
5. 新增、取消、延期或改写线程时，必须回到大纲扩展审核链路。

完成标准：

- 任意 active/due/paid 线程都能追溯到已审核大纲来源和正文处理证据。
- final review 能报告未闭合线程。
- 线程追踪视图中不存在无大纲来源的新伏笔或新悬念。

### 阶段 E：动态记忆冲突控制

目标：阻止动态库反向污染基准库。

范围：

- memory write policy
- extension review
- issue ledger

任务：

1. 动态写入前执行 baseline conflict check。
2. 冲突进入 issue ledger。
3. 需要改世界或大纲时必须走 extension proposal。
4. 需要改 baseline 时进入单独基准变更流程，不能由 dynamic commit 直接完成。

完成标准：

- 动态库不会覆盖冻结事实。
- 冲突都有裁决记录。

### 阶段 F：专业长任务母版接入

目标：与 206 的三模式升级合流。

范围：

- `SemanticTaskContract`
- `ProfessionalPromptProfile`
- `EvidencePacket`
- `DeliverableValidator`
- 写作 runtime profile

任务：

1. 定义写作 task goal types。
2. 写作节点生成语义任务契约。
3. 写作审核节点生成 evidence packet。
4. deliverable validator 检查节点必需交付。
5. 写作 agent profile 修复记忆上下文可见性。

完成标准：

- 写作任务能在 `professional_mode` 下运行。
- 专业 profile 不覆盖节点角色 prompt，只提供任务契约和验收。

### 阶段 G：长跑验收与清理

目标：真实跑长链路前完成防污染验收。

范围：

- 写作图配置
- 写作 runtime
- 系统 eval
- 前端任务图视图

任务：

1. 跑设计初始化完整链路。
2. 跑两批章节：第 1-10 章、第 11-20 章。
3. 验证第二批只能读取第一批 committed 结果。
4. 人为制造一个动态库与基准库冲突，验证被拦截。
5. 跑一次卷级 review + extension proposal + extension commit。
6. 清理旧简单写作图残留和无用测试。

完成标准：

- 真实长链路可恢复、可追踪、无候选污染。
- 旧残留不再作为新行为依据。

## 17. 迁移与切换规则

### 17.1 迁移原则

1. 新协议先 shadow 生成，不立即改变运行行为。
2. shadow 结果通过后，再把 protocol packet 设为必需输入。
3. 旧字段只允许短期作为读取来源，不允许继续定义新行为。
4. 切换完成后清理旧残留配置和测试。

### 17.2 切换顺序

```text
schema shadow
  -> compiler validation
  -> runtime visibility validation
  -> single design run
  -> two-batch chapter run
  -> one-volume synthetic run
  -> published graph cutover
  -> old path cleanup
```

### 17.3 回滚条件

出现以下情况应回滚到上一阶段：

- 必需记忆快照无法进入 agent 上下文。
- commit packet 无法追溯到 review。
- 动态库冲突未被拦截。
- 章节提交无法单章追踪。
- 断点恢复从 candidate 状态继续。

回滚不是恢复旧简单写作图，而是回到上一阶段协议配置。

## 18. 禁止的捷径

后续实施不得采用以下做法：

1. 只靠 prompt 约束记忆读写。
2. 把审核意见直接当 canon。
3. 把候选稿摘要交给下游当事实。
4. 让动态库覆盖基准库。
5. 让章节批次只提交一个总摘要。
6. 从聊天历史恢复长任务状态。
7. 用关键词检测代替结构化 deliverable validator。
8. 为了跑通测试硬编码写作输出。
9. 在通用 prompt 里塞无依据题材元素。
10. 长期保留旧简单写作图残留作为兼容路径。

## 19. 成功定义

本方案实施后，写作任务应达到：

1. 每个节点都知道自己在生产、审核、提交还是路由。
2. 每个节点只能读取授权上游包和授权记忆快照。
3. 所有 canon 都来自审核通过后的 commit packet。
4. 动态记忆不能覆盖基准事实。
5. 伏笔和关系线由大纲表达，并由派生线程追踪视图检查执行状态，不再靠 prompt 提醒。
6. 章节循环可以持续数十批，不丢上下文、不吞返修、不污染下游。
7. 断点恢复只从 accepted/committed 状态继续。
8. 最终交付只整编已提交结果，不重新发明故事。
9. prompt 保持专业中文商业网文方向，但不携带具体类型偏见。
10. 编排系统升级完成后，可自然接入 `professional_mode`，而不是另建写作专用私路。

## 20. 核心结论

当前写作任务图的方向是对的：四图拆分、章节批次、审核后提交、基准/动态记忆分离都已经具备基础。

真正需要补强的是：

```text
状态对象显式化
  + 协议包标准化
  + 记忆可见性验收
  + 大纲线程表达和派生追踪视图
  + 动态冲突裁决
  + 长任务健康路由
```

写作任务不能只靠更强的 prompt 变专业。专业的核心是：谁有权生产，谁有权裁决，谁有权写入，什么状态能被下游读取，什么内容永远不能自动变成事实。

下一步建议先做阶段 A 的验证，不急着跑完整小说任务。只有当记忆快照、上下文可见性、提交链路和防污染测试都通过后，再启动真实写作长跑。

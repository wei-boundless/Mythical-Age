## 二、第三阶段的真实问题

第三阶段的核心问题不是：

`健康系统页面还不够丰富`

而是：

`健康系统已经开始管理问题、报告、验证和 runtime admission，但它仍然混合消费历史别名、隐式契约和分散测试入口`

当前健康主链路大致是：

```text
HealthRegistry
  -> issue / command / report / test_run
  -> runtime_admission
  -> task binding preview
  -> runtime trace report
  -> workbench projection
```

这条链说明健康系统已经不是空壳，但还存在四个结构性问题：

1. 健康系统对象名已经不少，但正式对象层次还没锁死  
2. `runtime_admission` 仍在检查历史字段假设，而不是正式运行体契约  
3. 健康系统消费的回归门禁和实验入口仍是分散资源，不是统一门禁对象  
4. 问题、验证、回归、运行准入虽然都在 `health_system` 里出现了，但 owner 关系还没有彻底分清

因此，第三阶段必须解决的不是 UI 或单点 bug，而是：

`让健康系统只检查正式对象、只消费正式证据、只输出正式诊断与门禁结论`

## 3.2 仍然越界或未闭合的部分

### 3.2.1 健康对象层还停留在“能用”，还不是“正式分层”

当前对象虽然多，但层次还混在一起：

- issue

- `HealthIssueRecord`
- `HealthTraceRecord`
- `HealthDiagnosticReport`
- `HealthVerificationRun`
- `RegressionGateDecision`

也就是说现在的对象是“实现层命名”，还不是“系统层命名”。

### 3.2.2 runtime admission 仍然依赖历史契约假设

来源：

- [backend/health_system/runtime_admission.py](/d:/AI应用/langchain-agent/backend/health_system/runtime_admission.py:1)

`健康系统正在试图检查运行体正式契约，但运行准入对象还没有正式化`

当前它仍然主要依赖：

- `TaskFlowDefinition`
- `TaskAgentBinding`
- `AgentRuntimeProfile`

而不是正式依赖：

- `TaskExecutionAssembly`
- `TaskBodyOrchestration`
- `AgentRuntimeSpec`

这会让健康系统对“运行是否允许开始”的判断，停留在 task binding 级别，而不是 runtime spec 级别。

### 3.2.3 健康系统仍有历史别名与正式身份并存

例如健康 Agent 仍在部分样例或字段里混用：

- `agent:health:maintainer`
- `agent:3`

这说明健康系统身份边界还没有完全收口。

如果继续保留这种双写法，后续：

- issue owner
- agent run owner
- runtime admission
- task connection overview

都可能再次产生别名分裂。

### 3.2.4 回归、实验、验证入口仍未统一到健康系统正式边界

虽然第二阶段已经清掉：

- `build_task_runtime_contract`

- `/tasks/runtime-contract`

但测试与实验面仍是多入口组织：

- `backend/harness`
- `backend/test_system`
- `backend/experiments`
- `backend/health_system`

这意味着健康系统还没有成为：

`平台验证与门禁的唯一正式解释层`

## 四、第三阶段的本地硬约束

本阶段必须严格遵守以下约束。

## 4.1 健康系统只消费正式契约，不再猜字段

健康系统在第三阶段之后只允许检查：

- `TaskExecutionAssembly`
- `TaskBodyOrchestration`
- `AgentRuntimeSpec`
- `TaskRun`
- `RuntimeLoopState`
- `RuntimeDirective`
- 正式 regression / verification outcome

不再允许：

- 猜 workflow 是否“应该存在”
- 猜 projection 是否“看起来像对的”
- 依赖历史别名字段做 admission

## 4.2 健康系统不重做任务、编排、runtime 决策

健康系统只能：

- 观察
- 诊断
- 验证
- 门禁

不允许：

- 自己定义 task
- 自己拼 runtime spec
- 自己定义 prompt 结构
- 自己替代编排系统做 body assembly

## 4.3 回归门禁必须并入健康系统解释层

测试系统可以继续负责跑用例，但：

- pass / fail 的系统意义
- 哪些失败阻塞推进
- 哪些场景必须纳入 release gate

这些必须归健康系统统一解释。

## 4.4 历史别名必须继续物理收口

本阶段不允许继续长期保留：

- `agent:health:maintainer` 作为并行正式身份
- runtime contract 时代的残余命名
- 旧 case registry 中已删除文件路径

原则仍然是：

`除非用户明确要求，否则没有用处的旧残留代码请果断清理`

## 五、第三阶段目标设计

第三阶段完成后，健康系统主链应变成：

```text
Task / Orchestration / Runtime / Memory / Operation
  -> HealthTraceRecord
  -> HealthIssueRecord
  -> HealthDiagnosticReport
  -> HealthVerificationRun
  -> RegressionGateDecision
  -> RuntimeAdmissionDecision
  -> HealthWorkbenchView
```

## 5.1 第三阶段必须补出的正式对象

建议统一补在 `backend/health_system` 下。

### 5.1.1 HealthIssueRecord

取代当前偏实现层的 `HealthIssue` 命名，作为正式问题登记对象。

它回答：

`平台出了什么正式问题`

至少应包含：

- `issue_id`
- `title`
- `severity`
- `system_scope`
- `task_scope`
- `status`
- `symptoms`
- `suspected_root_cause`
- `trace_refs`
- `created_at`

### 5.1.2 HealthTraceRecord

表示一条可解释的诊断证据链。

它回答：

`这条链路到底发生了什么`

至少应包含：

- `trace_id`
- `task_run_id`
- `session_id`
- `agent_id`
- `trace_type`
- `event_refs`
- `checkpoint_refs`
- `projection_ref`
- `runtime_spec_ref`
- `diagnostics`

### 5.1.3 HealthDiagnosticReport

取代当前泛化的 `HealthReport`，只表达正式诊断结论。

它回答：

`为什么坏了`

至少应包含：

- `report_id`
- `issue_ref`
- `diagnosis_scope`
- `root_cause_candidates`
- `evidence_refs`
- `confidence`
- `repair_targets`
- `recommended_actions`

### 5.1.4 HealthVerificationRun

表示修复后的正式验证。

它回答：

`修完以后是否真的恢复正常`

至少应包含：

- `verification_id`
- `issue_ref`
- `verification_mode`
- `before_refs`
- `after_refs`
- `result`
- `evidence_refs`
- `verified_at`

### 5.1.5 RegressionGateDecision

表示一次门禁判断。

它回答：

`这次修改是否允许继续推进`

至少应包含：

- `gate_decision_id`
- `gate_profile_id`
- `required_suites`
- `required_smokes`
- `blocking_failures`
- `admitted`
- `diagnostics`

### 5.1.6 RuntimeAdmissionDecision

取代当前 `HealthCommandRuntimeAdmission` 的实现层定位，成为正式运行准入对象。

它回答：

`在当前健康状态下，这次运行是否允许开始`

至少应包含：

- `decision_id`
- `task_id`
- `agent_id`
- `runtime_spec_ref`
- `admitted`
- `blocked_reasons`
- `required_repairs`
- `gate_refs`

## 六、第三阶段固定执行顺序

第三阶段同样不允许边改边漂。执行顺序必须锁死。

## 6.1 阶段 A：统一健康系统对象语言

目标：

- 先把健康系统正式对象语言立起来
- 不立即重写整个 workbench

本阶段要做：

1. 补出 `HealthIssueRecord / HealthTraceRecord / HealthDiagnosticReport`
2. 补出 `HealthVerificationRun / RegressionGateDecision / RuntimeAdmissionDecision`
3. 把当前实现层对象标记为迁移期内部对象

完成标准：

- 健康系统正式对象层独立可表达
- 不再新增历史实现层命名

## 6.2 阶段 B：把 runtime admission 切到正式 runtime 契约

目标：

- 让健康系统判断“是否允许执行”时，不再停留在 flow binding 层

本阶段要做：

1. `RuntimeAdmissionDecision` 主输入切到 `AgentRuntimeSpec`
2. admission diagnostics 只引用正式 ref
3. 清理历史 alias 身份依赖

完成标准：

- 健康系统可基于正式 runtime 契约做 admission
- 不再依赖隐式 workflow 检查字段

## 6.3 阶段 C：统一 regression / verification 解释层

目标：

- 让健康系统成为平台验证结果的正式解释 owner

本阶段要做：

1. 把 `harness/regression_gate.py` 的结果接入健康系统 gate builder
2. 把 `test_system/case_registry.py` 的关键 profile 映射为 gate profile
3. 生成正式 `RegressionGateDecision`

完成标准：

- 健康系统可以说明“为什么挡住继续推进”
- regression gate 不再只是命令行结果

## 6.4 阶段 D：收口 workbench 和管理入口

目标：

- 让健康工作台只投影正式对象

本阶段要做：

1. `HealthWorkbenchBuilder` 改成只读取正式 issue / trace / verification / gate 对象
2. `HealthRegistry` 拆成 registry + builders，而不是继续承担所有拼装职责
3. 清理无正式 owner 的历史样例字段

完成标准：

- workbench 只展示正式对象投影
- registry 不再承担诊断与视图双重 owner

## 十、第三阶段完成后的预期状态

第三阶段完成后，应达到以下稳定状态：

1. 健康系统拥有正式对象层和正式 builder
2. 健康系统只消费正式 task / orchestration / runtime 契约
3. runtime admission 成为正式运行准入判断，而不是 binding 级预检查
4. regression gate 成为健康系统正式解释层的一部分
5. workbench 只投影正式问题、诊断、验证、门禁对象

到这一步，健康系统才算真正立住：

```text
任务系统：决定要做什么
编排系统：决定这次怎么装起来
runtime：把正式运行体对象跑起来
健康系统：判断哪里坏了、修好没有、是否允许继续推进
```

这条线一旦立住，后面才适合进入：

- 记忆系统正式入口统一
- 能力系统供给出口统一
- 灵魂系统 runtime 预览彻底剥离
- 多 Agent 健康链路独立建模

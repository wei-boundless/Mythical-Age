# 136-TaskGraph通用异步监测节点与运行控制闭环实施计划

日期：2026-05-17

## 1. 问题定义

当前系统已经有 TaskGraph、LangGraph 协调运行、`task_graph_run_monitor` 视图、`SupervisionRecord` 监督记录和 `background/manual_gate` 执行模式，但“监测”还主要依赖写作任务外部脚本和 Codex hook。这个结构有三个问题：

1. 监测机制脱离任务图，用户无法在通用编辑器里把“是否启用监测、监测什么、发现问题后怎么处理”作为图结构的一部分配置。
2. 旧写作 hook 会以固定频率触发，容易制造噪音，也会让系统判断偏向某个写作特例。
3. 人工审核、续跑、重启、提醒等控制动作没有一个统一的监测决策包承接，容易变成脚本直接操作运行态。

正确目标不是给健康系统特化，也不是再写一个外部 watchdog，而是把监测做成 TaskGraph 原生能力：

```text
任务图节点 -> 后台监测节点 -> 读取运行监控快照 -> 产出监测决策包 -> 编排控制接口执行或等待人工确认
```

## 3. 目标设计

### 3.1 节点模型

新增标准节点类型：

```json
{
  "node_type": "runtime_monitor",
  "execution_mode": "background",
  "background_policy": {
    "enabled": true,
    "interval_seconds": 300,
    "stale_after_seconds": 600,
    "max_runtime_seconds": 86400,
    "blocks_downstream": false
  },
  "notification_policy": {
    "priority": "normal",
    "include_result": "summary_and_refs"
  },
  "metadata": {
    "monitor_policy": {
      "watch_runtime_status": true,
      "watch_streaming": true,
      "watch_progress": true,
      "watch_blockers": true,
      "watch_manual_gate": true,
      "allowed_actions": ["no_action", "notify", "request_user_decision", "resume", "restart", "pause", "escalate"],
      "interaction_surface": {
        "window": "task_graph_run_interaction_panel",
        "open_mode": "inline_panel"
      }
    }
  }
}
```

说明：

- 监测节点是普通 TaskGraph 节点。
- 它通常是后台节点，不阻塞下游。
- 它读取运行态，不直接伪造业务输出。
- 它产出结构化 monitor decision packet。

### 3.2 决策包模型

新增通用决策包：

```json
{
  "authority": "task_graph.monitor_decision",
  "decision_id": "monitor-decision:...",
  "task_run_id": "...",
  "coordination_run_id": "...",
  "monitor_node_id": "runtime_monitor",
  "severity": "info|warning|error|critical",
  "action": "no_action|notify|request_user_decision|resume|restart|pause|escalate",
  "reason": "stale_runtime|runtime_failed|manual_gate_waiting|blocker_present|completed|healthy",
  "summary": "...",
  "observed": {
    "runtime_status": "...",
    "active_node_id": "...",
    "latest_event_at": 0,
    "last_effective_output_at": 0,
    "blocker": {},
    "streaming": {}
  },
  "recommended_control": {
    "api": "/orchestration/coordination-runs/{coordination_run_id}/continue-current-stage",
    "requires_human": false,
    "safe_to_auto_apply": true
  },
  "run_interaction_request": {}
}
```

说明：`run_interaction_request` 是唯一面向用户的运行交互请求。人工审核不是另一套窗口，而是这个统一窗口中的一种决策状态。

### 3.3 控制边界

监测节点只负责判断和记录：

- 可以建议续跑、重启、提醒、转人工。
- 可以把决策写入 `SupervisionRecord`。
- 不在模型评估函数里偷偷修改 TaskGraph 主状态。
- 真正运行控制仍走编排系统 API 或后续明确的控制器。

这能保证监测是“可审计的后台节点”，不是隐藏脚本。

## 4. 实施计划

### 阶段一：TaskGraph 模型与契约

目标：

- 增加 `runtime_monitor` 节点类型。
- 对 monitor 节点做结构化预检。
- 增加默认契约 `contract.taskgraph.monitor.decision`。

涉及文件：

- `backend/tasks/task_graph_models.py`
- `backend/tasks/contract_registry.py`

完成标准：

- `runtime_monitor + background` 通过 TaskGraph validation。
- 缺 `background_policy.enabled`、`max_runtime_seconds`、`notification_policy` 时仍按后台节点规则报错。
- 缺 `metadata.monitor_policy` 时给 warning，不阻断图运行。

### 阶段二：运行态监测评估器

目标：

- 新增 `task_graph_monitoring.py`。
- 以 `task_graph_run_monitor` 作为唯一快照来源。
- 根据 monitor policy 产出决策包。

涉及文件：

- `backend/orchestration/runtime_loop/task_graph_monitoring.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`

完成标准：

- 可通过 task_run_id 手动触发一次监测评估。
- 决策写入 `SupervisionRecord`。
- 输出包含 snapshot 摘要、decision、record。

### 阶段三：编排 API

目标：

- 增加监测评估 API。
- 增加决策查询 API。

涉及文件：

- `backend/api/orchestration.py`

完成标准：

- `POST /orchestration/runtime-loop/task-runs/{task_run_id}/task-graph-monitor/evaluate`
- `GET /orchestration/runtime-loop/task-runs/{task_run_id}/monitor-decisions`
- API 不依赖写作任务，不依赖健康系统。

### 阶段四：旧残留收口

目标：

- 停用 Codex hook 里的写作外部 watchdog。
- 删除旧写作监督脚本主链路。
- 保留配置写作任务和启动写作任务所需脚本，不删除业务配置工具。

涉及文件：

- `.codex/hooks.json`
- `.codex/hooks/writing_project_monitor.py`
- `.codex/hooks/writing_supervision_watchdog.py`
- `.codex/hooks/writing_supervision_stop.py`
- `scripts/supervise_writing_campaign.py`
- `scripts/watch_writing_supervision.py`
- `scripts/start_writing_watchdog.ps1`
- `scripts/start_writing_supervisor.ps1`
- `scripts/stop_writing_supervisor.ps1`

完成标准：

- hooks 不再周期性触发写作监督脚本。
- 旧监督脚本删除或从主入口断开。
- `configure_writing_simple_novel.py` 和 `start_writing_project_run.ps1` 保留，因为它们仍是任务配置/启动工具。

### 阶段五：验证与收尾

目标：

- 后端编译通过。
- 搜索确认新监测链路没有 `health_manager` 特化，也没有新增 `task_mode`。
- 写实施记录。

验证命令：

```powershell
python -m py_compile backend/tasks/task_graph_models.py backend/tasks/contract_registry.py backend/orchestration/runtime_loop/task_graph_monitoring.py backend/orchestration/runtime_loop/task_run_loop.py backend/api/orchestration.py
rg -n "health_manager|task_mode" backend/orchestration/runtime_loop/task_graph_monitoring.py backend/api/orchestration.py backend/tasks/task_graph_models.py
```

## 5. 不做事项

- 不把健康管家特殊化为任务图节点。
- 不恢复 `task_mode` 旧链路。
- 不把监测逻辑写成写作任务专用。
- 不让 hook 或外部 watchdog 成为主运行控制机制。
- 不在监测评估里直接硬改业务节点输出。

# 61-长场景主链误路由与重复工具锁修正计划书-20260511

## 1. 背景

在最新一轮长场景实测中，合同噪声已经下降，但主链仍存在三类结构性问题：

1. 显式数据集请求会被旧 PDF 绑定覆盖，导致模板与工具误选。
2. 语义型记忆回忆问题没有进入 memory 链路，而是落回 RAG 检索。
3. 重复工具守卫按工具名粗粒度拦截，正常的连续查询会被误停。

这些问题都不是单点提示词问题，而是理解层、模板装配层、运行时控制层之间的优先级结构不一致。

## 2. 目标

本轮修正只处理三条已经确认的主链，不做外围功能扩展。

验收目标：

1. 显式 `.xlsx/.csv` 请求优先进入 structured-data 模板与工具链。
2. “你之前让我怎么回答”“我该怎么称呼你”这类语义型回忆优先进入 memory 路由。
3. 重复工具守卫只拦截相同请求循环，不拦截同工具的不同查询。
4. 定向回归通过后，再复测长场景，确认通过率改善且没有引入新型误路由。

## 3. 修正范围

### 3.1 模板装配优先级

涉及文件：

- `backend/tasks/template_registry.py`

修正方向：

1. 让显式输入优先级高于陈旧会话绑定。
2. 在候选模板生成阶段，显式数据集锚点优先于旧 PDF 绑定。
3. 在模板匹配阶段，不能让 `candidate_template_ids` 无条件压过当前轮已经明确的结构化路由。

### 3.2 记忆读路由

涉及文件：

- `backend/understanding/memory_intent.py`
- `backend/understanding/task_understanding.py`

修正方向：

1. 将“语义回忆”从弱信号提升为可落地的 memory read 模式。
2. 让明确的记忆回忆请求直接进入 `memory` route，而不是继续交给 RAG。
3. 保持普通知识问题不被误伤，避免把泛问句一股脑改成记忆查询。

### 3.3 重复工具守卫

涉及文件：

- `backend/orchestration/runtime_loop/tool_repetition_guard.py`
- 如有必要，补充 `task_run_loop.py` 调用侧验证

修正方向：

1. 保留对相同请求死循环的拦截。
2. 移除“仅因工具名重复就中断”的粗粒度判断。
3. 允许同一工具在同一会话中承接不同查询。

## 4. 实施顺序

1. 先修模板装配优先级，因为它会直接影响数据集/PDF 主链。
2. 再修记忆读路由，因为它影响 recall 场景与后续对话稳定性。
3. 最后修重复工具守卫，因为它属于运行时控制层，改完后再统一复测更稳。

## 5. 验证方案

### 5.1 定向验证

1. 跑长场景合同回归：
   - `python -m pytest backend/tests/system_eval/long_scenarios_regression.py -q`
2. 复查 turn 09 类数据集切换场景是否进入 `mcp_structured_data`
3. 复查 turn 24 类 recall 场景是否进入 memory 路由
4. 复查 turn 45 类天气追问是否不再被 repeated tool halt 误停

### 5.2 综合验证

1. 重新执行：
   - `python backend/tests/system_eval/long_runner.py --scenario-set mega`
2. 对比上一轮结果：
   - 上一轮：`34/53`
3. 记录新通过率、首个失败轮次、残余问题归类。

## 6. 收尾要求

1. 本轮不做兼容性保留型残留逻辑堆叠。
2. 如果发现旧逻辑只是制造冲突，应直接移除。
3. 修改完成后要把结果回写到计划推进说明中，避免后续遗忘与误还原。

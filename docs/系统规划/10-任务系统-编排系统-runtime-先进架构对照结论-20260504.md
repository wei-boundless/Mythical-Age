# 任务系统、编排系统与 runtime 的先进架构对照结论

日期：2026-05-04

## 一、文档定位

本文用于回答一个在当前重构中非常关键的问题：

`任务系统、编排系统、runtime 三者到底应该怎么切边界，外部成熟架构通常怎么处理这件事`

本文不是泛泛的框架介绍，而是：

1. 先提炼本项目现有设计原则
2. 再对照若干成熟官方架构
3. 最后给出本项目应采用的明确分层结论

## 二、先给结论

结论先锁定：

**本项目当前文档里把“任务系统”“编排系统”“runtime”拆开，是正确方向，不需要回退成一个总装配器。**

真正需要修的不是概念本身，而是源码里还有一段没有切干净：

- 任务系统已经开始输出正式任务装配对象
- 但 runtime 还没有完全只消费运行体正式对象
- 中间仍残留 compat shell

如果参考成熟架构，最合理的稳定分层应当是：

```text
Task System
  -> 产出 TaskExecutionAssembly

Orchestration System
  -> 消费 TaskExecutionAssembly + MemoryBundle + Projection assets + Capability catalog
  -> 产出 AgentRuntimeSpec / TaskBodyOrchestration

Runtime Engine
  -> 只消费 AgentRuntimeSpec / TaskBodyOrchestration
  -> 负责执行、checkpoint、resume、observability、result finalization
```

所以本项目后续不该再问：

`runtime 是不是编排系统的一部分`

而应改成：

`runtime 是否只执行正式运行体对象，而不再自己拼身体`

这才是核心。

## 三、本项目本地设计原则提炼

从本地设计书里，可以提炼出四条已经非常正确的原则。

## 3.1 任务系统负责“谁做什么”

来自 [00-全项目系统边界总设计书-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/00-全项目系统边界总设计书-20260504.md) 与 [02-任务系统详细设计书-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/02-任务系统详细设计书-20260504.md)：

- 任务系统负责任务识别、任务登记、任务分配、workflow 绑定、projection 绑定、flow contract 绑定
- 任务系统不负责 Agent 内部 prompt 结构、不负责工具权限细节、不负责运行时执行细节

这条原则本质上是在说：

`任务系统是 control plane of tasks，不是单 Agent 的身体装配器`

## 3.2 编排系统负责“这个 Agent 的身体怎么被装起来”

来自 [03-编排系统详细设计书-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/03-编排系统详细设计书-20260504.md)：

- 编排系统负责 prompt 装配结构
- 负责资源边界、记忆作用域、runtime lane、output boundary
- 负责 `TaskBodyOrchestration`
- 负责 `AgentRuntimeSpec`

这说明编排系统的本体不是“权限面板”，而是：

`single-agent execution body assembly system`

## 3.3 runtime 只负责执行，不负责临时拼身体

来自 [03-编排系统详细设计书-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/03-编排系统详细设计书-20260504.md) 第九、十四节：

- runtime 不应决定装什么 prompt
- runtime 不应决定开哪些资源
- runtime 不应决定记忆能进多少
- runtime 只应执行正式运行体对象

这条原则非常重要，因为它直接否定了“runtime 自由猜测任务形态”的旧结构。

## 3.4 先修系统边界，再修局部行为

来自 [00-全项目系统边界总设计书-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/00-全项目系统边界总设计书-20260504.md) 与 [08-第一阶段任务系统边界回收实施蓝图-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/08-第一阶段任务系统边界回收实施蓝图-20260504.md)：

- 不保留错误边界
- 不再让任务系统变成跨系统总装配器
- compat 只能存在于适配层，不能回灌到正式 builder

这条原则和外部成熟架构的经验是高度一致的。

## 四、外部成熟架构的共同模式

这里不做“大而全”综述，只看和本问题直接相关的四套官方架构。

## 4.1 LangGraph：高层图定义与底层 runtime 分离

官方文档：

- LangGraph Overview  
  https://docs.langchain.com/oss/python/langgraph
- LangGraph Runtime / Pregel  
  https://docs.langchain.com/oss/python/langgraph/pregel

官方明确区分：

- 高层图定义：`StateGraph`
- 编译结果：`Pregel` instance
- 底层运行：`Pregel` runtime 管理执行步骤

对本项目最有价值的不是 LangGraph 的图语法，而是它的分层关系：

```text
Graph definition
  -> compile
  -> runtime instance
  -> execute
```

对应到本项目就是：

```text
TaskExecutionAssembly
  -> Runtime Body Assembly
  -> AgentRuntimeSpec
  -> RuntimeLoop
```

可借鉴点：

- 高层任务定义不直接等于执行引擎
- compile/assembly 是独立阶段
- runtime 专注执行步骤管理，而不是重新定义高层结构

不应照抄点：

- 本项目不是纯图框架，不应把所有任务装配问题都强制图化
- LangGraph 的 runtime 与 graph 绑定较强，不适合作为本项目六大系统的边界模板全盘复制

## 4.2 AutoGen Core：runtime 是环境层，不是任务定义层

官方文档：

- Agent Runtime Environments  
  https://microsoft.github.io/autogen/0.5.5/user-guide/core-user-guide/core-concepts/architecture.html

该文档把 runtime environment 的职责写得非常清楚：

- 促进 agent 之间通信
- 管理 agent identity 与 lifecycle
- 执行安全与隐私边界

这说明在成熟 agent framework 里，`runtime` 的语义更接近：

`execution environment / lifecycle environment`

而不是：

`task planner + prompt assembler + policy builder all-in-one`

可借鉴点：

- runtime 是环境与执行层
- identity / lifecycle / communication / boundary enforcement 由 runtime 承担
- 高层任务决策不应混入 runtime 本体

不应照抄点：

- AutoGen 更偏 actor/event 驱动，对本项目单 Agent 身体装配层描述还不够细
- 它不能直接替代本项目“编排系统”这一层

## 4.3 Microsoft Agent Framework：workflow 与 agent 分离，执行引擎支持 checkpoint

官方文档：

- Overview  
  https://learn.microsoft.com/en-us/agent-framework/overview/
- Workflows Overview  
  https://learn.microsoft.com/en-us/agent-framework/user-guide/workflows/core-concepts/overview
- Workflows Orchestrations  
  https://learn.microsoft.com/en-us/agent-framework/workflows/orchestrations/

官方很明确地区分：

- 什么时候该用 agent
- 什么时候该用 workflow
- workflow engine 负责 checkpoint / recovery / orchestration

这和本项目有两个强对应关系。

第一，对应任务系统：

- workflow 是任务方法
- 不是编排系统
- 不是能力资源

第二，对应 runtime：

- 执行引擎负责恢复、编排、观测
- 不是上层任务识别与装配定义者

可借鉴点：

- workflow 与 agent 必须分开
- orchestration engine 可以做 checkpoint / approval / resume
- 多 agent orchestration 是执行层模式，不等于单 Agent 身体装配

不应照抄点：

- 其 workflow builder 偏“执行流程编排框架”，不能直接代替任务系统的对象登记域
- 本项目仍需保留任务系统作为独立治理域

## 4.4 Kubernetes：control plane 与 node execution 分离

官方文档：

- Cluster Architecture  
  https://kubernetes.io/docs/concepts/architecture/
- Communication between Nodes and the Control Plane  
  https://kubernetes.io/docs/concepts/architecture/control-plane-node-communication/

Kubernetes 对本项目最大的启发，不是容器技术，而是这条抽象：

- control plane 负责目标状态和全局编排
- node / kubelet 负责把目标状态真正跑起来
- 执行节点不反过来决定 cluster 目标

这和本项目高度相似：

```text
任务系统 ~= task control plane
编排系统 ~= single-agent body assembler / runtime spec generator
runtime ~= node execution engine
```

尤其值得吸收的是：

- control plane 与 execution plane 严格分离
- 通信路径清晰
- 执行端不应暴露成上层总编排器

## 五、外部架构对本项目的直接启示

把上面几套架构放在一起，会出现一个非常清晰的共同模式：

## 5.1 “定义层”与“执行层”之间，通常都存在一个正式装配层

无论是：

- LangGraph 的 compile -> Pregel
- Agent Framework 的 workflow definition -> workflow execution
- Kubernetes 的 desired state -> node execution

中间都会有一层：

`把高层目标转成可执行对象`

这正好对应本项目文档中已经提出、但源码还没完全走到位的那层：

- `TaskExecutionAssembly`
- `TaskBodyOrchestration`
- `AgentRuntimeSpec`

## 5.2 runtime 的成熟语义通常都是“执行引擎”

外部成熟文档里，runtime 很少被定义成：

- 任务识别系统
- prompt 规划系统
- 资源决定系统

它通常更像：

- 执行器
- 生命周期管理器
- 恢复与观测层
- 边界 enforcement 层

这说明本项目把 runtime 放在编排系统之后，是合理的。

## 5.3 workflow 不应等于单 Agent 内部身体编排

成熟架构里，workflow 更多负责：

- 任务路径
- 节点顺序
- orchestration

但不会天然替代：

- 单 Agent 内部 prompt 装配点
- 单 Agent 内部资源开启顺序
- 单 Agent 内部记忆注入时机

所以本项目在 [03-编排系统详细设计书-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/03-编排系统详细设计书-20260504.md) 里把 `TaskBodyOrchestration` 单独提出来，是有理论支撑的，不是凭空新增中间层。

## 六、本项目应明确借什么，不借什么

## 6.1 应借的部分

1. 借“正式装配对象再执行”的分层  
   任务系统不直接给 runtime 大字典，而是先给运行体正式对象。

2. 借“runtime 只执行、不临时拼装”的原则  
   runtime 负责 loop、checkpoint、resume、event log、observation、finalization。

3. 借“workflow 与 agent body 不混”的原则  
   workflow 仍归任务系统；单 Agent 身体编排归编排系统。

4. 借“control plane / execution plane 分离”的治理模式  
   上层管目标、下层管执行，不允许反向串权。

## 6.2 不应借的部分

1. 不应把本项目全盘图框架化  
   不是所有系统边界问题都该变成 graph/node/channel。

2. 不应把 runtime 扩成万能总装配器  
   这是最危险的回退路线。

3. 不应把多 Agent orchestration 误当成编排系统本体  
   多 Agent 任务组织首先仍是任务系统问题。

4. 不应把 workflow builder 直接当作 canonical task registry  
   任务登记、任务策略、任务采用方案仍应保留在任务系统对象层。

## 七、对本项目当前源码的具体判断

结合 [09-任务系统重建源码对照审查-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/09-任务系统重建源码对照审查-20260504.md)，当前源码最主要的问题不是“runtime 命名错误”，而是：

**runtime 主链路还没有只消费运行体正式对象。**

当前大体还是：

```text
TaskExecutionAssembly
  + compat prompt surface
  -> RuntimeLoop
```

目标应当是：

```text
TaskExecutionAssembly
  -> Orchestration System
  -> AgentRuntimeSpec + TaskBodyOrchestration
  -> RuntimeLoop
```

这意味着下一阶段真正要切的是：

1. 从任务系统剥离最后的 compat prompt surface 核心地位
2. 让编排系统正式生成 `AgentRuntimeSpec`
3. 让 runtime loop 直接消费 `AgentRuntimeSpec`

## 八、本项目建议锁定的正式边界

建议现在就把这组分工写死，后续所有重构都按它推进。

## 8.1 任务系统

正式输出：

- `TaskIntentContract`
- `TemplateMatchResult`
- `TaskExecutionAssembly`
- `CoordinationTaskRecord`
- `TaskCommunicationProtocol`
- `CoordinationTopologyTemplate`

它回答：

`这是什么任务，交给谁，任务路径与任务组织结构是什么`

## 8.2 编排系统

正式输入：

- `TaskExecutionAssembly`
- `MemoryBundle`
- `ProjectionPromptAsset`
- `CapabilitySupplyPackage`
- `AgentBodyProfile`

正式输出：

- `TaskBodyOrchestration`
- `PromptStructureProfile`
- `ResourcePolicyProfile`
- `MemoryScopeProfile`
- `AgentRuntimeSpec`
- `AgentBodyRuntimeView`

它回答：

`这个 Agent 这次身体怎么装，内部状态怎么跑，资源和记忆怎么收口`

## 8.3 runtime

正式输入：

- `AgentRuntimeSpec`
- `TaskBodyOrchestration`
- `RuntimeContextSnapshot`
- `ResourceRuntimeView`

它负责：

- run
- pause
- resume
- checkpoint
- event log
- observation
- result merge
- finalization

它回答：

`这个已经装好的执行体怎么稳定执行起来`

## 九、建议的下一阶段收口顺序

## 9.1 第一优先级：补出编排系统的正式出口

优先新增或正式化：

- `AgentBodyProfile`
- `TaskBodyOrchestration`
- `PromptStructureProfile`
- `MemoryScopeProfile`
- `AgentRuntimeSpec`

这一步不做，runtime 永远会继续直接吃 compat surface。

## 9.2 第二优先级：让 runtime 改为只消费运行体对象

需要重点收口：

- `backend/orchestration/runtime_loop/stage_projection.py`
- `backend/orchestration/runtime_loop/context_manager.py`
- `backend/runtime/agent_chain.py`

目标是让这些模块不再把 `task_prompt_contract` 当核心输入。

## 9.3 第三优先级：再处理 adoption execution 与 multi-agent execution

这是后续执行层扩展，但它们应建立在：

- 任务系统边界已清
- 编排系统边界已清
- runtime 只执行正式对象

这三个前提之上。

## 十、最终结论

最终结论直接写清楚：

1. 你的本地设计原则是对的。  
   把任务系统、编排系统、runtime 拆开，不是过度设计，而是成熟架构普遍采用的分层方式。

2. 外部先进架构的共同点，不是“把一切塞进 runtime”，而是“先形成正式可执行对象，再交给 runtime 执行”。

3. 本项目后续不应回退到“一个总 builder / 一个总 runtime 大字典”的旧结构。

4. 本项目下一阶段真正要做的不是重新争论命名，而是推动这条正式链路落地：

```text
TaskExecutionAssembly
  -> AgentRuntimeSpec
  -> RuntimeLoop
```

这条线一旦彻底立住，任务系统、编排系统、runtime 三者的边界就会真正稳定下来。

# RAG结构整改执行蓝图

日期：2026-05-08

## 1. 文档目标

本文档用于指导当前 RAG 系统的结构整改实施，作为后续代码改造、阶段验收、测试补齐和切换控制的统一执行基线。

本轮整改不以“短期提升召回效果”为目标，而以“建立稳定的结构契约”为目标，优先解决以下系统性问题：

1. 失败语义不稳定，存在 silent fallback。
2. retrieval evidence 没有进入真实上下文预算。
3. v2 检索失败会直接打断整条 RAG 链路。
4. 输出边界对“证据不足”和“模型故障”区分不清。

## 2. 执行总规则

本次整改按 5 个阶段串行推进，不交叉施工。

总规则如下：

1. 先立契约，再改实现。
2. 先做失败语义结构化，再做检索边界与预算控制。
3. 每个阶段结束必须补对应回归测试，不允许把验证推迟到后续阶段。
4. 本轮不顺手重写 query rewrite、rerank、skill 路由、memory RAG 结构。
5. 允许阶段性过渡实现，但不再引入新旧检索双轨并存结构。

## 3. 整改目标

本轮整改的目标不是重写整套 RAG，而是把现有“能运行的 RAG 管道”升级成“失败可分型、预算可核算、检索边界清晰的 RAG 子系统”。

整改完成后，系统至少要满足以下四个条件：

1. 失败可分型。
2. 降级可解释。
3. 上下文可核算。
4. 检索失败可治理。

## 4. 当前关键问题归纳

结合现有审查结果，当前关键问题如下：

### 4.1 v2 检索失败会直接打断 RAG 主链路

位置：`backend/retrieval/service.py`

当前 `retrieve()` 已经是纯 v2 主链路。如果 v2 backend、索引、rerank 或其他下游过程抛异常，整轮检索会直接失败，上层没有拿到结构化的 retrieval failure 结果。

这违反了对话循环中“优先保证任务可继续”的原则。

### 4.2 retrieval evidence 没有进入真实上下文预算

位置：`backend/context_policy/package_builder.py`

当前 retrieval 结果会进入 `sections["retrieval_evidence"]`，但没有等价进入 `section_tokens` 和 `remaining` 的核算过程，导致 retrieval-heavy 场景下上下文使用不透明、不可解释。

这违反了上下文管理中“所有 model-visible 内容都必须被预算约束”的原则。

### 4.3 证据整合阶段吞掉模型故障

位置：`backend/evidence/output_policy.py`、`backend/evidence/orchestrator.py`

当前 finalizer 模型失败、输出为空、输出被判定为 snippet dump 等多类故障，最终都会退化成统一兜底文案，无法区分：

1. 没有检索结果；
2. 有检索结果但证据不足；
3. 有检索结果但模型总结器故障；
4. 有检索结果但输出不合法。

这会导致 output boundary 失真，也使问题定位困难。

## 5. 阶段总览

### Phase 1：定义 RAG 状态契约

目标：

1. 建立 retrieval、finalizer、orchestrator 间统一的结构化状态表达。
2. 消除“用空列表、空字符串、模糊文案表达失败”的做法。

交付物：

1. RAG result/status 数据模型。
2. typed degraded reason 规范。
3. canonical result 映射规则。

完成标志：

1. 所有主失败路径都能区分类型。
2. `silent fallback` 被消除。

### Phase 2：收敛 v2 检索失败语义与服务边界

目标：

1. 让 `RetrievalService` 对 v2 异常输出稳定、可传递的结构化失败状态。
2. 将“检索执行失败”和“检索为空”从 service 层开始区分清楚。

交付物：

1. retrieval execution result contract。
2. 检索异常到 typed failure 的映射规则。
3. service 层 diagnostics 透传结构。

完成标志：

1. `RetrievalService` 不再用裸异常表达检索失败。
2. 上层可以区分 `retrieval_failed` 和 `retrieval_empty`。

### Phase 3：把 retrieval evidence 纳入上下文预算

目标：

1. 让 retrieval evidence 成为正式的 budget consumer。
2. 建立 retrieval section 的 token accounting 和裁剪策略。

交付物：

1. retrieval token accounting。
2. retrieval 裁剪策略。
3. dropped diagnostics。

完成标志：

1. retrieval-heavy 场景下上下文长度行为可预测。

### Phase 4：重构 finalizer 输出边界

目标：

1. finalizer 返回结构化结果，不再只返回字符串。
2. orchestrator 负责状态汇总与用户结果映射。

交付物：

1. finalization result contract。
2. canonical result 映射逻辑。
3. 诊断信息透传规则。

完成标志：

1. `evidence_insufficient` 与 `finalizer_failed` 可区分。

### Phase 5：补齐回归矩阵并做集成验证

目标：

1. 把整改结果固定进测试体系。
2. 为回滚和后续演进提供工程保障。

交付物：

1. retrieval service regression。
2. context budget regression。
3. output boundary regression。
4. integration regression。

完成标志：

1. 所有关键失败路径有测试覆盖。

## 6. Phase 1 执行蓝图：RAG 状态契约

### 6.1 阶段目标

建立统一的 RAG 生命周期状态模型。

### 6.2 建议新增结构

建议新增文件：

- `backend/evidence/rag_contracts.py`

建议定义至少以下对象：

- `RetrievalExecutionResult`
- `RAGFinalizationResult`
- `RAGPipelineStatus`

建议状态枚举至少包含：

- `retrieval_ok`
- `retrieval_empty`
- `retrieval_failed`
- `finalization_ok`
- `finalization_model_error`
- `finalization_invalid_output`
- `finalization_skipped_no_evidence`

### 6.3 影响文件

- `backend/evidence/retrieval_worker.py`
- `backend/evidence/orchestrator.py`
- `backend/evidence/output_policy.py`

### 6.4 实施步骤

1. 在 `rag_contracts.py` 中定义结构化结果对象。
2. 将 `RetrievalWorker.run()` 改为返回带状态的结果。
3. 将 `RAGEvidenceOutputPolicy` 改为返回结构化 finalization result。
4. 由 `EvidenceOrchestrator` 统一映射成 canonical result。
5. 建立 `degraded_reason_typed` 映射表。

### 6.5 阶段禁止事项

1. 不改 `RetrievalService` 召回逻辑。
2. 不改 rerank。
3. 不改 query rewrite。
4. 不做回答文风优化。

### 6.6 完成判定

系统必须能稳定区分以下情况：

1. 没检索到。
2. 检索失败。
3. 检索到了但模型总结失败。
4. 检索到了但输出不合法。
5. 正常完成。

### 6.7 测试要求

建议新增：

- `backend/tests/rag_output_boundary_regression.py`

优先覆盖：

1. finalizer 抛 `ModelRuntimeError`。
2. finalizer 返回空内容。
3. finalizer 返回 snippet dump。
4. retrieval empty。

## 7. Phase 2 执行蓝图：v2 检索失败语义与服务边界

### 7.1 阶段目标

让 v2 检索主链路对失败、空结果、正常结果有稳定边界。

### 7.2 建议新增或调整结构

建议新增文件：

- `backend/retrieval/retrieval_contracts.py`

建议定义：

- `RetrievalExecutionResult`
- `RetrievalExecutionStatus`
- `RetrievalFailureDiagnostics`

### 7.3 影响文件

- `backend/retrieval/service.py`
- `backend/evidence/retrieval_worker.py`

### 7.4 实施步骤

1. 为 `RetrievalService` 增加结构化返回契约，明确正常、空结果、失败三类状态。
2. 将 `_retrieve_from_plan()` 的异常捕获与诊断收集集中在 service 层完成。
3. 对 retrieval backend、rerank、plan 执行等失败点统一映射为 typed retrieval failure。
4. 让 `RetrievalWorker` 消费结构化 retrieval result，而不是只消费原始 `list[dict]`。
5. 明确 diagnostics 内容：
   - query；
   - selected_collections；
   - query_mode；
   - failure_stage；
   - error_type；
   - latency_ms（如可获得）。

### 7.5 阶段禁止事项

1. 不改 orchestrator 主协议。
2. 不改 route 选择逻辑。
3. 不引入新的检索切换模式。

### 7.6 完成判定

必须满足以下行为：

1. v2 backend 报错时，上层拿到明确 `retrieval_failed`。
2. 空召回时，上层拿到明确 `retrieval_empty`。
3. 正常召回时，上层拿到明确 `retrieval_ok`。

### 7.7 测试要求

建议新增：

- `backend/tests/retrieval_service_regression.py`

优先覆盖：

1. v2 backend exception。
2. rerank exception。
3. empty hits。
4. normal hits。
5. diagnostics 是否完整。

## 8. Phase 3 执行蓝图：retrieval 上下文预算化

### 8.1 阶段目标

让 retrieval evidence 成为正式预算消费者，不再作为“账外上下文”存在。

### 8.2 影响文件

- `backend/context_policy/package_builder.py`
- `backend/context_management/context_controller.py`
- `backend/prompting/builder.py`

### 8.3 建议新增预算规则

建议引入 retrieval section 规则：

- `retrieval_max_items`
- `retrieval_max_chars_per_item`
- `retrieval_max_total_tokens`
- `retrieval_drop_policy`

### 8.4 实施步骤

1. 为 `_render_retrieval_results()` 增加稳定 token estimate。
2. 将 retrieval items 写入 `section_tokens`。
3. 让 retrieval items 消耗 `remaining`。
4. 按不同压力等级定义裁剪：
   - `normal`：保留 top N；
   - `warning`：缩条数和单条长度；
   - `microcompact`：仅保留 1 到 3 条高价值摘要；
   - `full_compact`：仅保留高度压缩的 evidence summary。
5. 将 dropped retrieval items 写入 diagnostics。

### 8.5 阶段禁止事项

1. 不改 memory section 优先级。
2. 不改 long-term memory 策略。
3. 不把 retrieval 和 memory 合并为同一 section。

### 8.6 完成判定

必须可以回答以下问题：

1. retrieval evidence 占了多少预算。
2. 哪些 evidence 被丢掉了。
3. 为什么被丢掉。
4. 压力升高时 retrieval section 如何裁剪。

### 8.7 测试要求

建议新增：

- `backend/tests/context_budget_retrieval_regression.py`

优先覆盖：

1. retrieval results 很长时预算消耗正确。
2. 超 budget 时 evidence 被裁剪。
3. `warning` / `full_compact` 下行为不同。
4. dropped diagnostics 存在。

## 9. Phase 4 执行蓝图：finalizer 输出边界重构

### 9.1 阶段目标

把 finalizer 从“字符串函数”升级为“有状态输出边界”。

### 9.2 影响文件

- `backend/evidence/output_policy.py`
- `backend/evidence/orchestrator.py`
- `backend/output_boundary/answer_finalizer.py`

### 9.3 建议返回结构

建议 `RAGEvidenceOutputPolicy.rewrite_rag_answer_with_model()` 返回结构化结果，至少包含：

- `status`
- `answer`
- `failure_reason`
- `diagnostics`

建议状态至少包含：

- `ok`
- `model_error`
- `empty_output`
- `snippet_dump_rejected`
- `skipped_no_evidence`

### 9.4 实施步骤

1. 将 `output_policy` 改为返回结构化对象。
2. 由 `orchestrator` 根据结构化状态决定 canonical result。
3. 将用户文案与状态映射分离：
   - 检索为空；
   - 检索成功但证据不足；
   - finalizer 故障；
   - 正常完成。
4. 将 diagnostics 透传到 done event。

### 9.5 阶段禁止事项

1. 不在本阶段改 prompt 风格。
2. 不引入第二套 finalizer 模型。
3. 不增加复杂多轮总结流程。

### 9.6 完成判定

必须实现：

1. finalizer 故障不再伪装成 evidence insufficient。
2. invalid output 不再等价于 retrieval empty。
3. 用户可见结果与 typed status 一致。

### 9.7 测试要求

沿用：

- `backend/tests/rag_output_boundary_regression.py`

补充覆盖：

1. snippet dump rejected。
2. empty finalizer output。
3. model runtime failure。
4. valid finalization success。

## 10. Phase 5 执行蓝图：测试矩阵与集成验证

### 10.1 阶段目标

把整改结果固定进回归体系，并为后续演进提供工程保护。

### 10.2 影响文件

- `backend/tests/query_runtime_runtime_loop_regression.py`
- `backend/tests/retrieval_service_regression.py`
- `backend/tests/rag_output_boundary_regression.py`
- `backend/tests/context_budget_retrieval_regression.py`

### 10.3 必须覆盖的测试矩阵

1. 正常 RAG 成功。
2. retrieval empty。
3. retrieval v2 fail。
4. retrieval service diagnostics 正确。
5. finalizer model fail。
6. finalizer invalid output。
7. retrieval evidence 超长。
8. canonical result typed reason 正确。

### 10.4 集成验收标准

至少验证以下用户面行为：

1. 用户问知识问题，系统正常给出回答。
2. 用户问冷门问题，系统明确说明当前证据不足。
3. v2 检索链路坏了，系统能明确表达检索失败或进入受控降级。
4. 模型总结器坏了，系统不会伪装成“只是没答案”。
5. 检索片段很多时，上下文不会再无上限膨胀。

## 11. 文件级开工顺序

建议严格按以下顺序实施：

1. `backend/evidence/rag_contracts.py`
2. `backend/evidence/output_policy.py`
3. `backend/evidence/orchestrator.py`
4. `backend/evidence/retrieval_worker.py`
5. `backend/retrieval/retrieval_contracts.py`
6. `backend/retrieval/service.py`
7. `backend/context_policy/package_builder.py`
8. `backend/context_management/context_controller.py`
9. `backend/tests/rag_output_boundary_regression.py`
10. `backend/tests/retrieval_service_regression.py`
11. `backend/tests/context_budget_retrieval_regression.py`
12. `backend/tests/query_runtime_runtime_loop_regression.py`

顺序理由如下：

1. 先锁 contract，避免后续返工。
2. 先稳定 output boundary，减少调试噪音。
3. 再做 retrieval service 边界收口。
4. 再做 context budgeting。
5. 最后补齐集成回归。

## 13. 回滚方案

### 13.1 Phase 1 / Phase 4 回滚

1. 保留新 contract 文件。
2. orchestrator 暂时回落到旧字符串路径。
3. 不提交未通过测试。

### 13.2 Phase 2 回滚

1. `RetrievalService` 暂时回落到原始 `list[dict]` 输出路径。
2. 保留 retrieval contract 文件但不上层消费。

### 13.3 Phase 3 回滚

1. retrieval budget 配置降为 conservative 最小值。
2. 必要时先只启用 item cap，不启用复杂 token accounting。

## 14. 本轮明确不做的事项

为防止执行发散，本轮明确不做以下事项：

1. 不顺手优化召回质量。
2. 不扩 skill / routing 规则。
3. 不把 memory retrieval 与 knowledge retrieval 合并。
4. 不引入新的 prompt engineering 分支。
5. 不保留长期“临时兼容层”。

## 15. 最终验收口径

本轮完成后，不要求 RAG 立即“更聪明很多”，要求的是结构达标。

最终验收口径如下：

1. 失败可分型。
2. 降级可解释。
3. 上下文可核算。
4. 检索失败可治理。

达到以上四项，即视为本轮结构整改达成目标。

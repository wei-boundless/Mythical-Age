# 62-Followup目标解析与上下文绑定重构计划书-20260511

## 1. 背景

长场景复测中，数据集连续追问与组合子任务追问暴露出同一类结构问题：

1. 用户追问“完全没有缺口的仓库”时，系统没有稳定继承 `active_dataset`，掉回 RAG 检索。
2. 用户追问“只基于刚才这前五名员工”时，系统没有稳定继承当前结果子集。
3. 用户追问“第一个和第三个子任务”时，系统没有优先识别 bundle ordinal，反而被当前活跃数据集吞掉。

这不是单个工具的问题，而是 follow-up 目标解析缺少统一权威层。

## 2. 目标

建立统一的 `FollowupResolution`，让当前轮追问先解析“继承哪个目标”，再决定走 dataset、pdf、bundle 或普通检索。

目标优先级：

1. 显式目标：明确文件、路径、任务、节点。
2. Bundle ordinal：第一个、第三个、第二个子任务等。
3. Active subset/result：刚才这前五名、这几条、上面那组结果。
4. Active object：当前激活的 `active_dataset` / `active_pdf`。
5. Committed object：最近提交过的 dataset/pdf。
6. Retrieval/general：无可继承目标时才进入普通检索或通用回答。

## 3. 实施范围

### 3.1 Understanding 层

文件：

- `backend/understanding/task_understanding.py`

改动：

1. 新增 `FollowupResolution` 数据结构。
2. 在 `TaskSignals` 中携带 `followup_target_kind`、`followup_ordinals`、`followup_scope`。
3. Dataset follow-up 增强：
   - “完全没有缺口”
   - “是否存在”
   - “如果没有”
   - “只基于刚才”
   - “前五名”
   - “这些员工”
4. Bundle ordinal 优先级提升：
   - 命中 bundle ordinal 时，不再因为存在 `bound_dataset_path` 直接走 dataset。

### 3.2 Template/Assembly 层

文件：

- `backend/tasks/template_registry.py`

改动：

1. `followup_target_kind=bundle_ordinals` 时，候选模板优先使用 bundle follow-up 语义。
2. `active_subset` 与 `active_dataset` 进入 structured-data 模板。
3. 避免当前活跃 dataset 抢占 bundle ordinal。

### 3.3 测试

新增或扩展回归：

1. 数据集 active follow-up：
   - “是否存在完全没有缺口的仓库？”
2. 数据集 subset follow-up：
   - “只基于刚才这前五名员工，按部门归类。”
3. Bundle ordinal follow-up：
   - “把第一个和第三个子任务各压成一句话，不要再提第二个。”

# 86-长篇小说写作团队任务图重新装配流程记录

## 0. 文档定位

本文记录长篇小说写作团队任务图的重新装配流程。

本次装配不沿用旧 `longform_novel` 试验数据。旧数据只作为反例和清理对象，不作为新配置来源。

目标不是直接塞一张现成图，而是把“如何从空主数据开始配置一支写作团队”完整记录下来，后续每一步都能在前端、后端主数据和运行验证之间追踪。

## 1. 本次重做原因

旧长篇小说包存在以下问题：

- 数据来源是早期试装，不代表当前 TaskGraph 主模型边界。
- 任务、投影、契约、图、记忆策略已经混合在 storage 中，难以判断哪些是主数据，哪些是派生数据。
- 部分命名带有旧假设，例如 `writer_b`、`reviewer.world_foundation`，不适合继续作为新团队的规范命名。
- 旧图已经装配完成，绕过了“用户如何在编辑器中逐层配置”的过程。
- 如果继续复用旧图，就无法验证装配流程本身是否可用。

因此，本次先清空旧长篇小说任务包，再按新的装配流程重建。

## 3. 新装配命名规范

新装配不继续使用旧 `longform_novel` 命名空间。建议使用更清晰的团队装配命名：

- 任务域：`domain.writing_team.long_novel`
- 任务族：`writing_team_long_novel`
- 任务：`task.writing_team.long_novel.<role>`
- 投影：`projection.writing_team.long_novel.<role>`
- 契约：`contract.writing_team.long_novel.<artifact>`
- 工作流：`workflow.writing_team.long_novel.<role>`
- Flow：`flow.writing_team.long_novel.<role>`
- 图：`graph.writing_team.long_novel`
- 协议：`protocol.writing_team.long_novel`

新命名强调“写作团队装配”，而不是把某个旧长篇小说包当成系统默认能力。

## 4. 重新装配总流程

装配必须按以下顺序执行：

1. 建任务域。
2. 建 Agent 角色清单。
3. 建 Projection。
4. 建 Contract。
5. 建 Specific Task。
6. 建 Projection Binding。
7. 建 Flow Contract Binding。
8. 建 Memory Profile。
9. 建 Communication Protocol。
10. 建 TaskGraph 模板草稿。
11. 前端逐层检查和补齐。
12. 保存、预检、编译 runtime spec。
13. 创建运行并检查 trace。

不能先造图再回头补资产。图只能引用已经存在的 Agent / Projection / Contract / Task。

```text
你是一名章节审稿人。
你只负责审查当前章节候选稿是否满足章节目标、人物状态、世界观约束和叙事节奏。
你不负责替写作者扩写正文。
你需要给出通过或返修裁决，并列出必须修正的问题、定位依据和建议修复方向。
```

禁止：

- 不允许把开发说明写成 prompt。
- 不允许把 Projection prompt 复制进 TaskGraph metadata。

图只保存引用和结构。

- `output_node_id = memory_commit`
- `default_protocol_id = protocol.writing_team.long_novel`
- `graph_contract_id = contract.writing_team.long_novel.quality_gate`

### Step 9：前端逐层确认

必须按 TaskGraph Studio 层级确认：

1. 蓝图层：图级边界正确。
2. Agent 编组层：每个节点 Agent 可调用。
3. 拓扑层：边端点和返修边存在。
4. 职责与交接层：Projection 和 payload contract 绑定完整。
5. 时序与循环层：phase / sequence / review gate / loop 完整。
6. 记忆与产物层：上下文隔离和 artifact ref 传递正确。
7. 契约与质量门层：质量门输出可裁决。
8. 发布运行层：preflight、runtime spec、trace 可用。

### Step 10：实测

第一轮只做小型实测：

- 输入一个写作目标。
- 运行到世界观设计和审核。
- 检查 trace 中 ready / blocked 是否合理。
- 检查主 Agent 是否根据审核结果自然决定继续或返修。

通过后再扩大到章节生产和入库。

### 8.7 配置绑定表

配置文件：

- `storage/tasks/task_projection_bindings.json`

- `storage/tasks/task_assignments.json`
- `storage/tasks/task_agent_adoption_plans.json`
- `storage/tasks/task_memory_request_profiles.json`

实际规则：

- 每个任务必须有 Projection Binding。
- 每个任务必须有 Flow Contract Binding。
- 每个任务必须有 Assignment，使任务、flow、workflow、projection、contract 能串起来。
- 每个任务必须有 Adoption Plan，说明运行时采用主系统已有 Agent，不在 TaskGraph 中临时造 Agent。
- 每个任务必须有 Memory Profile，说明可读记忆层、写回策略和长期记忆权限。

写作者节点特殊规则：

- `task.writing_team.long_novel.chapter_draft` 的 Projection Binding 允许：
  - `projection.writing_team.long_novel.chapter_writer_a`
  - `projection.writing_team.long_novel.chapter_writer_b`
- 默认投影为 `projection.writing_team.long_novel.chapter_writer_a`。
- TaskGraph 节点级 `projection_id` 负责区分 A/B。

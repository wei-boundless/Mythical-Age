## 0. 核心结论

当前长任务失败不是单点 prompt 问题，也不是“工具完全没有结构化”。真实情况是：

```text
工具输入 schema 和工具绑定机制已经有雏形；
但 provider tool-call 边界、工具结果 observation、证据 ledger、执行义务验证、最终交付验证仍没有形成单一权威闭环。
```

更准确地说，系统现在处于“新结构已部分落地，但旧链路仍在主导完成判断”的中间态：

```text
ExecutionObligation / StrategyPrototype / ProfessionalRunState / ToolObservationLedger 已存在
  但 ProfessionalTaskRunDriver 的最终 gate 仍主要依赖
  task_goal_type + ProfessionalTaskActionTracker + 文本启发式 + deliverable_validator 关键词检查。
```

这会导致长任务表面上进入了专业模式，trace 中也有状态机和 ledger，但模型仍可能：

1. 分类一错，写入和验证义务被削弱。
2. 工具调用以 DSML 或可见文本泄漏，而不是原生 provider tool call。
3. 工具执行结果只作为字符串进入 observation，后续只能靠猜。
4. 搜索命中路径、terminal exit code、写入产物、验证结果不能被结构化验证。
5. 状态机记录了过程，却没有成为“能否完成”的唯一裁判。
6. role/standard/professional 三模式策略与 driver 实际执行存在不一致。

本计划的修复方向：

```text
把 ExecutionObligation + ProfessionalRunState + ToolObservationLedger 升级为专业长任务的完成权威；
把 task_goal_type 降级为策略原型；
把工具结果从 text-first 改为 typed observation；
把 DeepSeek/其他 provider 的工具调用边界前移到 model_response 层；
把 deliverable validation 和 driver legacy verification 合并为统一 ObligationValidation；
实施结束后删除旧的重复判断链路，不保留双轨兼容。
```

## 1. 当前代码现状审查

### 1.1 三模式策略已经存在，但 role mode 策略和 driver 执行矛盾

相关文件：

- `backend/orchestration/interaction_mode_policy.py`
- `backend/orchestration/runtime_loop/professional_task_run_driver.py`

当前事实：

`interaction_mode_policy.py` 已定义：

- `role_mode`
- `standard_mode`
- `professional_mode`

其中 `role_mode` 的 `tool_policy` 明确配置：

```text
enabled = True
allowed_tool_names = mcp_retrieval / web_search / fetch_url
allowed_operation_refs = op.mcp_retrieval / op.web_search / op.fetch_url / op.memory_read
max_tool_calls_per_task_run = 1
```

但 `professional_task_run_driver.py` 在进入 driver 后又执行：

```python
if interaction_mode == "role_mode":
    tool_execution_enabled = False
    delegation_enabled = False
    allowed_tool_names = []
```

问题：

```text
策略层说 role_mode 可做一次只读搜索/记忆读取；
执行层直接禁用全部工具。
```

影响：

- 角色模式无法自然承接“灵魂系统 + 轻搜索/上下文记忆”的设计。
- 测试只看 policy 会通过，但真实运行能力会被 driver 削掉。
- 后续用户以为 role mode 是轻能力模式，实际是纯无工具聊天。

修复原则：

`role_mode` 不是无工具模式，而是只读轻能力模式。driver 必须消费 policy，不得硬编码清空工具。需要由 mode kernel 统一解释：

```text
role_mode:
  allow read-only tools
  deny write/edit/terminal/delegate
  max one round
  no strict deliverable validator
```

### 1.2 专业状态机和工具观察账本已经存在，但还不是完成判定权威

相关文件：

- `backend/orchestration/runtime_loop/professional_state_machine.py`
- `backend/orchestration/runtime_loop/tool_observation_ledger.py`
- `backend/orchestration/runtime_loop/professional_task_run_driver.py`

当前事实：

新增结构已经存在：

- `ProfessionalRunState`
- `ProfessionalStateTransition`
- `ToolObservationLedger`
- `ToolObservationRecord`
- `build_tool_observation_record()`
- `ProfessionalRunSession`
- `decide_professional_run_resume()`

driver 也已经在工具结果后追加 ledger，并写事件：

```text
professional_tool_observation_ledger_updated
professional_run_session_updated
professional_task_deliverable_validation_checked
```

但最终校验仍主要走：

- `_verify_goal_contract()`
- `ProfessionalTaskActionTracker`
- `_material_path_observed()`
- `validate_deliverable()`
- 文本协议泄漏检测

问题：

```text
状态机在记录；
旧 tracker 在裁判。
```

影响：

- 状态机不会天然阻止“未写入但完成”“未验证但完成”。
- ToolObservationLedger 只作为 summary/diagnostics 出现，不是 obligation satisfaction 的主输入。
- resume 时可以看到 session/state ref，但缺少可恢复的 obligation ledger。
- 长任务失败后无法精确知道缺的是 read、write、verify、deliverable 还是 protocol boundary。

修复原则：

状态机必须从“旁路记录”升级为“完成判定权威”：

```text
ToolObservationLedger
  -> ObligationSatisfaction
  -> ProfessionalRunState.unsatisfied_obligations
  -> ObligationValidation
  -> complete / blocked / repairing / paused
```

`ProfessionalTaskActionTracker` 只能作为迁移期内部缓存，完成后应删除或并入 `ToolObservationLedger`。

### 1.3 工具结果 observation 仍是 text-first，结构化证据不足

相关文件：

- `backend/execution/tool_executor.py`
- `backend/orchestration/runtime_loop/action_request.py`
- `backend/orchestration/runtime_loop/evidence_packet.py`
- `backend/orchestration/runtime_loop/tool_observation_ledger.py`

当前事实：

`tool_executor.py` 执行工具后做：

```python
text = str(result or "")
```

`build_tool_result_observation()` 写入：

```python
payload = {
    "tool_name": ...,
    "tool_args": ...,
    "result": content,
    "result_chars": len(content),
    "truncated": truncated,
    "execution_receipt": receipt,
    "execution_id": ...,
    "result_ref": ...
}
```

`ToolObservationRecord` 也主要保存：

```text
result_preview
side_effect_kind
satisfies
side_effect_hash
```

问题：

工具输入是结构化的，但工具输出没有统一 envelope。缺少：

- `structured_payload`
- `observed_paths`
- `matched_paths`
- `artifact_refs`
- `write_receipt`
- `edit_receipt`
- `terminal.exit_code`
- `terminal.stdout_ref`
- `terminal.stderr_ref`
- `terminal.command`
- `test_result.passed`
- `test_result.failed_count`
- `sandbox_overlay_ref`
- `material_refs`

影响：

- `search_text/search_files` 的真实命中路径不能稳定进入材料观察。
- `terminal` 是否成功只能从字符串猜。
- `write_file/edit_file` 是否真的产生产物不能统一验证。
- evidence packet 只能从文本预览抽事实，长任务综合质量不稳定。

修复原则：

新增统一工具结果信封：

```python
ToolResultEnvelope(
    tool_name,
    tool_args,
    status,
    text,
    structured_payload,
    observed_paths,
    artifact_refs,
    command_receipt,
    execution_receipt,
    result_ref,
    error
)
```

所有 observation、ledger、evidence、validator 都只消费 envelope，不再直接消费裸字符串。

### 1.4 DeepSeek 工具调用协议边界仍偏后置

相关文件：

- `backend/execution/model_runtime.py`
- `backend/execution/model_response.py`
- `backend/execution/tool_call_policy.py`
- `backend/orchestration/runtime_loop/professional_task_run_driver.py`

当前事实：

已有进展：

- `ToolCallBindingOptions` 已存在。
- `model_runtime.py` 已在 `bind_tools()` 处传递 tool choice 相关参数。
- DeepSeek thinking 的 `reasoning_content` 已通过 `_DeepSeekReasoningCompatChatModel` 保留。
- `model_response.py` 已读取 `response.tool_calls`。

但 `model_response.py` 当前主要做：

```python
tool_calls = _normalize_tool_calls(getattr(response, "tool_calls", None))
additional_kwargs = dict(getattr(response, "additional_kwargs", {}) or {})
```

问题：

1. 没有统一 provider response adapter。
2. 没有解析 OpenAI-compatible provider 可能放在 `additional_kwargs.tool_calls`、`additional_kwargs.function_call` 或原始 message payload 中的工具调用。
3. 如果 DeepSeek 输出 DSML / invoke / tool_calls 文本，当前系统主要在 driver/deliverable validator 末端发现并清洗。
4. 可见 content 中携带工具标记时，系统没有在 model response 边界立即转为 `protocol_violation` 或 repair event。

影响：

- native tool call 没有被最大化接收。
- 伪工具文本进入 answer boundary，后续靠正则兜底。
- 长任务中 provider 一次输出错形态，整轮容易变成“模型说自己要调用工具但没有真实执行”。

修复原则：

在 `model_response.py` 增加 `ProviderToolCallAdapter`：

```text
response.tool_calls
additional_kwargs.tool_calls
additional_kwargs.function_call
raw provider payload
  -> ToolCallIntent[]
```

同时：

```text
visible DSML/tool_call markup
  -> ModelProtocolViolationObservation
  -> driver repair / required native tool call retry
```

禁止把 DSML 当作主工具协议继续兼容运行。DSML 只能作为违规检测和清理对象。

### 1.5 协议泄漏检测存在重复和漂移

相关文件：

- `backend/orchestration/runtime_loop/professional_task_run_driver.py`
- `backend/orchestration/runtime_loop/deliverable_validator.py`

当前事实：

driver 中 `_contains_tool_call_markup()` 已检测：

- `tool_calls`
- `invoke name=`
- `name="query"`
- `name="path"`
- `name="command"`
- `name="read_file"`
- `name="search_text"`
- `name="search_files"`
- `name="delegate_to_agent"`
- DSML markers

但 `deliverable_validator.py` 的 `_protocol_leak_detected()` 正则只包含：

```text
read_file / search_text / search_files / delegate_to_agent / query / path
```

未包含 `name="command"`。

问题：

同一个协议泄漏规则在两个文件里维护，并且已经漂移。

影响：

- driver 可能判违规，validator 可能不判。
- 某些 terminal/command 伪工具文本可能漏过基础交付验证。
- 后续继续补正则会越来越碎。

修复原则：

新增统一模块：

```text
backend/orchestration/runtime_loop/protocol_boundary.py
```

只保留一套：

- `detect_protocol_leak(text)`
- `strip_protocol_leak(text)`
- `protocol_leak_markers()`
- `ProtocolLeakResult`

driver、deliverable validator、output boundary 都调用这一个模块。删除散落的 `_contains_tool_call_markup()` / `_protocol_leak_detected()` 重复实现。

### 1.6 `task_goal_type` 仍然责任过载

相关文件：

- `backend/tasks/semantic_task_contracts.py`
- `backend/orchestration/interaction_mode_policy.py`
- `backend/orchestration/runtime_loop/professional_task_run_driver.py`
- `backend/prompting/professional_profiles.py`

当前事实：

`SemanticTaskContract.task_goal_type` 仍影响：

- domain
- professional profile
- deliverables
- required reasoning steps
- required actions
- forbidden actions
- material policy
- validation schema

`_resolve_task_goal_type()` 仍是 first-match 分支：

```text
test_report_triage
runtime_trace_analysis
regression_test_design
artifact_delivery
code_fix_execution
material_synthesis
light_qa / bounded_tool_task / role_conversation
```

问题：

这与 210、211 的设计目标仍有距离：`task_goal_type` 还没有真正降级为软策略原型。

影响：

- 一旦分类错误，profile、deliverables、forbidden actions、validation schema 同时偏移。
- 典型危险：用户要求“修复代码并运行测试”，但材料中出现失败报告和 tests/fixtures 路径，就可能偏向 `test_report_triage`，从而削弱修改/验证。
- 专业 prompt 仍可能压制当前 turn 的真实执行义务。

修复原则：

`ExecutionObligation` 必须先于并高于 `StrategyPrototype`：

```text
用户当前 turn
  -> ExecutionObligation(read/write/verify/deliver/forbid)
  -> StrategyPrototype candidates
  -> SemanticTaskContract
```

下游禁止再用 `task_goal_type` 单独决定：

- 是否写入
- 是否验证
- 是否可以完成
- 是否允许 final

### 1.7 专业 prompt profile 有内容，但没有硬绑定到 driver 的模型指令

相关文件：

- `backend/prompting/professional_profiles.py`
- `backend/orchestration/assembly_builder.py`
- `backend/orchestration/runtime_loop/professional_task_run_driver.py`

当前事实：

`backend/prompting/professional_profiles.py` 已有 profile：

- `professional.test_report_triage`
- `professional.runtime_trace_analysis`
- `professional.code_fix_execution`
- `professional.regression_test_design`
- `professional.material_synthesis`

`assembly_builder.py` 已生成：

```text
professional_profile_section = professional_profile.prompt
metadata.professional_profile = professional_profile.to_dict()
```

但 `professional_task_run_driver.py` 的 `_with_professional_task_instruction()` 只根据：

- mode
- semantic contract
- mode policy
- tool line
- delegation line
- plan lines

构造通用 system instruction，没有直接把 `professional_profile.prompt` 作为专业职责段硬绑定进 driver 当前模型消息。

问题：

profile 可能存在于 orchestration prompt contract 或 metadata，但 driver 的长任务工具循环实际 model messages 不一定稳定消费它。

影响：

- 专业角色提示可能只在前置 assembly 中存在，到了 driver repair/tool loop 阶段被弱化。
- 长任务模型看见的是“主执行 Agent 通用说明”，不是“测试报告诊断员 / 结构性代码修复执行员”等明确职责。
- prompt 优化难以验证，因为没有单一注入点。

修复原则：

driver 必须接收并注入 `ProfessionalPromptProfile`，注入顺序固定为：

```text
Base Professional Runtime Contract
Execution Obligation Section
Strategy Prototype Role Profile
Mode Policy Section
Tool/Delegation Boundary
Deliverable Contract
Soul Projection Style
```

profile 文案必须是 agent-facing 专业角色提示，不能写成 runtime 节点说明。

### 1.8 搜索和材料观察仍靠字符串匹配，不足以支撑长材料综合

相关文件：

- `backend/orchestration/runtime_loop/professional_task_run_driver.py`
- `backend/orchestration/runtime_loop/evidence_packet.py`

当前事实：

`_record_contract_observation()` 对搜索工具只记录 query/pattern：

```python
elif tool_name in {"search_files", "search_text", "glob_paths"}:
    query = ...
    tracker.searched_material_refs = ...
```

`_material_path_observed()` 判断材料是否已观察时，会用：

- normalized path
- basename
- search refs 中是否包含 path/base

问题：

搜索结果真实命中的文件路径没有结构化进入 tracker。当前只是用 query 反推“可能观察过”。

影响：

- 如果用户要求读 A 文件，但搜索 query 恰好包含 A 文件名，系统可能误以为材料已观察。
- 如果搜索结果真实命中 B/C 文件，但 query 不含路径，系统又可能认为没有观察。
- evidence packet 无法稳定列出材料来源。

修复原则：

搜索类工具 observation 必须输出：

```text
matched_paths
matched_spans
match_count
material_refs
```

材料观察判定必须只看 typed observation 中的 `observed_paths/matched_paths/material_refs`，不再看 query 字符串。

### 1.9 工具预算仍偏静态，长任务不能按义务动态扩缩

相关文件：

- `backend/orchestration/interaction_mode_policy.py`
- `backend/orchestration/runtime_loop/professional_task_run_driver.py`

当前事实：

三模式预算是静态配置：

- `role_mode`: 1 call
- `standard_mode`: 4 calls / 2 rounds
- `professional_mode`: 16 calls / 8 rounds

driver 虽然会根据 `required_next_tools` 收窄工具，但整体预算仍主要来自 mode policy。

问题：

长任务不是固定 16 次工具就能覆盖。代码修复、测试失败返修、写作批次、材料综合所需轮数差别很大。

影响：

- 简单任务可能预算过宽。
- 复杂任务可能过早 `tool_loop_budget_exceeded`。
- 验证失败后的 repair loop 没有独立预算。

修复原则：

新增 `DynamicToolBudget`：

```text
base_budget from mode
+ required_reads count
+ write obligation
+ verification obligation
+ repair allowance
+ material count
+ delegated review allowance
```

预算必须与状态机绑定：

- read budget
- write budget
- verification budget
- repair budget
- finalization budget

预算耗尽时不能 completed，只能 blocked/partial with missing obligations。

## 2. 目标架构

### 2.1 修复后的主链路

```text
User Turn
  -> IntentFrame
  -> ExecutionObligation
  -> StrategyPrototypeCandidates
  -> SemanticTaskContract
  -> RuntimeInteractionModePolicy
  -> ProfessionalRunSession
  -> ProfessionalRunState
  -> DynamicPlan
  -> ProviderToolCallAdapter
  -> ToolResultEnvelope
  -> RuntimeObservation.structured_payload
  -> ToolObservationLedger
  -> EvidencePacket
  -> ObligationValidation
  -> OutputBoundary
  -> checkpoint/state_index/runtime_objects
```

### 2.2 权威边界

| 层 | 权威对象 | 负责 | 不允许 |
|---|---|---|---|
| 用户当前 turn | user message / explicit inputs | 本轮目标、限制、是否要改 | 被历史状态覆盖 |
| 执行义务 | `ExecutionObligation` | 必须读、写、验证、交付、禁止项 | 被 task_goal_type 取消 |
| 模式策略 | `RuntimeInteractionModePolicy` | 工具权限、投影强度、验证强度、预算基线 | 重新定义用户目标 |
| 策略原型 | `StrategyPrototype` | 默认步骤、专业角色、交付风格 | 决定是否写入/验证 |
| provider 边界 | `ProviderToolCallAdapter` | 原生工具调用归一化、协议违规检测 | 接受 DSML 为主协议 |
| 工具观察 | `ToolResultEnvelope` / `ToolObservationLedger` | 真实读写验证证据 | 只保存裸文本 |
| 状态机 | `ProfessionalRunState` | 时序推进、阻塞、返修、完成 | 被 final answer 文本绕过 |
| 验证器 | `ObligationValidation` | 最终是否满足义务和交付 | 与 driver 重复漂移 |
| 输出边界 | `OutputBoundary` | 用户可见回答、协议清理 | 伪造工具结果 |

### 2.3 三模式统一解释

三模式继续保留，但必须由同一个 mode kernel 解释：

```text
role_mode:
  灵魂/角色主导
  只读轻工具
  无写入、无命令、无委派
  answer boundary 轻校验

standard_mode:
  当前回合明确工具任务
  少量读写/命令
  无持久长跑
  basic obligation validation

professional_mode:
  长任务母版
  高预算、多轮、可恢复、可委派
  严格 obligation validation
  checkpoint after key transitions
```

不能再出现：

```text
policy 允许工具，但 driver 硬编码禁用；
policy 要求 professional profile，但 driver 没有稳定注入；
状态机记录 complete，但 legacy validator 另判 partial；
validator 通过，但 state machine 还有 unsatisfied obligations。
```

## 3. 数据模型修复

### 3.1 ToolCallIntent

新增位置：

```text
backend/execution/tool_call_intent.py
```

建议结构：

```python
@dataclass(frozen=True, slots=True)
class ToolCallIntent:
    call_id: str
    tool_name: str
    args: dict[str, Any]
    provider: str = ""
    source: str = "native_tool_call"
    raw_ref: str = ""
    protocol_violation: bool = False
    violation_reason: str = ""
```

## 6. 长任务验收场景

### 6.1 代码修复长任务

用户目标示例：

```text
追踪这个失败报告，定位相关代码，修复问题，然后运行 pytest 验证。
```

必须发生：

```text
ExecutionObligation:
  required_reads: failure report + related source/tests
  required_writes: code change
  required_verifications: pytest

State:
  obligation_bound
  plan_drafted
  tool_observed(read)
  artifact_written
  verification_observed
  deliverable_validating
  complete
```

失败条件：

- 没有 write/edit observation 却说已修复。
- 没有 terminal observation 却说测试通过。
- terminal exit code 非 0 却说通过。
- final answer 缺 changed files / verification result / limitation。

### 6.2 测试报告诊断长任务

必须发生：

```text
read report
build evidence packet
classify failures
infer structural root causes
propose regression tests
validate deliverables
```

失败条件：

- 只复述失败项，没有结构性根因。
- 没有 evidence packet。
- final answer 泄漏工具协议。
- 把未知证据写成确定结论。

### 6.3 未知专业长任务

必须发生：

```text
strategy_prototype = generic_professional_task
execution_obligation 仍然完整生效
```

失败条件：

- 因不在任务分类中而降级 light QA。
- 因没有专业 profile 而不执行明确的 read/write/verify。

### 6.4 角色模式轻能力

必须发生：

```text
role_mode
projection_strength = primary
allowed read-only tools <= 1 call
no write/edit/terminal/delegate
```

失败条件：

- policy 允许只读工具但 driver 清空工具。
- role mode 执行写入或命令。
- 灵魂投影覆盖工具权限。

### 6.5 持久恢复

必须发生：

```text
load session/state/ledger/checkpoint
rebuild unsatisfied obligations
respect current user turn
continue / repair / restart / clarify
```

失败条件：

- 从聊天历史猜进度。
- completed run 重复写副作用。
- 用户说“停止/不要改”后仍继续旧写入。

## 8. 风险与控制

| 风险 | 控制 |
|---|---|
| ToolResultEnvelope 改动面大 | 先保留 `payload.result` 文本预览，但所有新验证只看 envelope |
| provider 工具调用适配误判 | adapter 单测覆盖 native/additional_kwargs/function_call/DSML violation |
| role mode 变得过强 | mode kernel 强制 read-only + one call + no side effect |
| obligation 抽取过强导致误写 | “不要改/只分析/先审查”作为高优先级 forbidden_actions |
| obligation 抽取过弱导致不执行 | 写入/验证关键词 + 路径 + 当前 turn explicit inputs 联合判断 |
| 状态机和 TaskGraph 冲突 | TaskGraph 节点状态编译为 ProfessionalRunState 子阶段，不另起主完成裁判 |
| 旧链路删除破坏测试 | 先冻结回归，再阶段性切换，切换完成删除旧测试 |
| prompt profile 压制用户目标 | profile 不允许取消 ExecutionObligation，validator 以 obligation 为准 |

## 9. 禁止事项

1. 不允许用更多关键词分类替代 ExecutionObligation。
2. 不允许让 `task_goal_type` 决定是否写入或验证。
3. 不允许把 DSML 当作主工具调用协议。
4. 不允许靠 final answer 文本判断文件是否写入成功。
5. 不允许 terminal 没有 exit code / command receipt 却判验证通过。
6. 不允许保留 driver validator 和 deliverable validator 两套完成裁判。
7. 不允许 profile prompt 写成开发说明或 runtime 节点说明。
8. 不允许为了通过测试伪造工具结果、文件写入或验证结果。
9. 不允许实施结束后继续保留 old simple/managed/autonomy_mode 正式链路。
10. 不允许把灵魂投影当成工具权限来源。

## 10. 成功定义

完成本计划后，系统应该满足：

1. 三模式策略与 driver 执行一致。
2. role mode 支持只读轻能力，不再被 driver 硬清空。
3. DeepSeek/OpenAI-compatible 工具调用在 model response 层被原生归一化。
4. DSML/tool markup 只作为协议违规处理，不再作为执行协议。
5. 工具结果拥有统一结构化 envelope。
6. ToolObservationLedger 能证明读、写、验证、委派和材料观察。
7. ProfessionalRunState 的 complete 只能由 ObligationValidation 触发。
8. task_goal_type 只是策略原型，不再是执行总开关。
9. 专业 prompt profile 被 driver 稳定注入，并且是 agent-facing 职责提示。
10. 最终交付只基于真实 observation/evidence，不能伪造执行结果。
11. 旧链路被删干净，测试不再依赖旧模式断言。

最终目标不是让 agent “看起来更会说长任务”，而是让它在结构上具备长任务能力：

```text
知道本轮必须做什么；
知道下一步必须用什么真实工具；
知道哪些证据已经获得；
知道哪些义务还没满足；
知道什么时候必须返修、阻塞、暂停或完成；
最终只基于真实证据回答用户。
```

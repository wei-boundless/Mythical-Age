## 二、健康系统总定义

健康系统是平台的观测、诊断、验证与回归门禁中心。

它不是一个状态总览页，也不是一个被动报警器。

它负责的是：

- 观察平台运行状态
- 诊断故障位置
- 验证修复结果
- 控制回归门禁
- 输出健康报告

它回答的问题是：

`平台现在是否正常，哪个系统不正常，哪条链路不正常，为什么不正常，修复后是否真的恢复正常`

## 三、为什么健康系统不能只是“监控页”

如果把健康系统只理解为监控页，它最多只能展示：

- 运行中
- 失败
- 成功率

但你的平台需要的远不止这些。

因为这个平台不是普通 CRUD 系统，而是一个多系统参与的 Agent 平台。

同一个任务失败，可能来自：

- 任务系统任务装配错误
- 编排系统资源边界不对
- 记忆系统供给错误
- 灵魂系统 prompt 结构冲突
- 能力系统能力缺失
- runtime 状态推进错误

所以健康系统必须做的不只是“看状态”，而是：

- 看链路
- 看边界
- 看因果
- 看回归

## 四、健康系统的硬边界

### 4.1 健康系统不定义任务

任务定义归任务系统。

健康系统只能：

- 观察任务执行情况
- 诊断任务失败原因
- 验证任务修复效果

### 4.2 健康系统不组装 Agent 身体

单 Agent 身体归编排系统。

健康系统只能：

- 检查身体装配是否完整
- 检查身体运行是否异常
- 检查任务态身体编排是否失效

### 4.3 健康系统不筛记忆内容

记忆内容归记忆系统。

健康系统只能：

- 检查是否拿错记忆
- 检查是否记忆越界
- 检查是否写回失败
- 检查是否摘要失真

### 4.4 健康系统不提供能力资源

能力资源归能力系统。

健康系统只能：

- 检查资源是否缺失
- 检查资源是否被错误授权
- 检查资源调用是否失败

## 五、健康系统在平台中的位置

健康系统是平台的观察者、诊断者和门禁者。

它在平台中的位置是：

- 上接六大系统的运行结果
- 下接测试、回归、报告和修复验证

如果没有健康系统，平台会出现三个问题：

1. 平台知道失败了，但不知道失败在哪一层
2. 平台修了一次问题，但不知道是否真的修好
3. 平台在持续演化时，没有真正的回归控制

## 六、健康系统的健康层次模型

健康不是单一概念，必须分层。

我建议统一采用五层健康模型。

### 6.1 L1：基础运行健康

回答：

`系统是否活着`

包括：

- 进程是否正常
- 基础 API 是否可响应
- 存储路径是否可读写
- runtime loop 是否可启动

### 6.2 L2：系统边界健康

回答：

`各系统之间的边界是否正常`

包括：

- 任务系统输出对象是否完整
- 编排系统装配对象是否完整
- 记忆系统 bundle 是否完整
- 能力系统资源目录是否可用
- 灵魂系统素材是否可加载

### 6.3 L3：链路执行健康

回答：

`一条真实任务链路能不能正常跑通`

包括：

- task -> body -> memory -> runtime 是否连通
- 多 Agent 拓扑与通信是否连通
- 失败后是否能正确回退
- 验证门禁是否真的生效

### 6.4 L4：结果质量健康

回答：

`系统虽然跑通了，但结果是不是可信`

包括：

- 输出是否满足契约
- 产物是否真实存在
- 验证是否真实完成
- 记忆是否污染
- 任务总结是否与实际执行一致

### 6.5 L5：回归稳定性健康

回答：

`这次修改之后，旧的关键链路是不是还活着`

包括：

- 回归测试结果
- 关键系统 smoke 结果
- 长链路场景回归结果
- 历史问题是否复发

## 七、健康系统的职责清单

健康系统至少负责以下八类事情：

1. 平台基础健康监测
2. 系统边界健康检查
3. 任务执行链路诊断
4. 多 Agent 协调链路诊断
5. 记忆链路健康检查
6. 运行体链路健康检查
7. 验证与回归门禁
8. 健康报告与故障归因

## 八、健康系统的对象模型

结合现有代码中的：

- `backend/health_system`
- `backend/harness/regression_gate.py`
- `backend/capabilities/validation.py`
- `backend/orchestration/runtime_loop/checkpoint.py`
- `backend/orchestration/runtime_loop/trace_reader.py`

我建议健康系统统一围绕八类对象组织。

### 8.1 HealthIssueRecord

表示一个正式登记的健康问题。

它回答：

`出了什么问题`

建议字段：

- `issue_id`
- `title`
- `severity`
- `system_scope`
- `task_scope`
- `status`
- `symptoms`
- `suspected_root_cause`
- `trace_refs`
- `created_at`

### 8.2 HealthTraceRecord

表示链路跟踪记录。

它回答：

`这条链路实际发生了什么`

建议字段：

- `trace_id`
- `task_run_id`
- `agent_id`
- `trace_type`
- `event_refs`
- `checkpoint_refs`
- `state_refs`
- `diagnostics`

### 8.3 HealthCheckProfile

表示健康检查配置。

它回答：

`这类对象应该如何被检查`

建议字段：

- `profile_id`
- `target_type`
- `check_points`
- `failure_threshold`
- `sampling_policy`
- `enabled`

### 8.4 HealthDiagnosticReport

表示一次正式诊断结论。

它回答：

`为什么坏了`

建议字段：

- `report_id`
- `issue_id`
- `diagnosis_scope`
- `root_cause_candidates`
- `evidence_refs`
- `confidence`
- `repair_targets`
- `recommended_actions`

### 8.5 HealthVerificationRun

表示一次修复后的验证运行。

它回答：

`修完以后是否真的恢复正常`

建议字段：

- `verification_id`
- `issue_id`
- `before_refs`
- `after_refs`
- `verification_mode`
- `result`
- `evidence_refs`
- `verified_at`

### 8.6 RegressionGateProfile

表示回归门禁策略。

它回答：

`哪些测试和链路必须过，平台才允许继续推进`

建议字段：

- `gate_id`
- `gate_name`
- `required_suites`
- `required_smokes`
- `block_on_failure`
- `severity_threshold`

### 8.7 HealthWorkbenchView

表示前端工作台看到的健康视图。

它回答：

`当前平台哪里红了，哪里黄了，哪里恢复了`

建议字段：

- `view_id`
- `system_summaries`
- `active_issues`
- `recent_verifications`
- `regression_gate_status`
- `risk_summary`

### 8.8 RuntimeAdmissionDecision

表示某次运行是否允许进入执行。

它回答：

`这次运行在当前健康状态下是否允许继续`

建议字段：

- `decision_id`
- `task_id`
- `agent_id`
- `admitted`
- `blocked_reasons`
- `required_repairs`
- `gate_refs`

## 九、健康系统观察的核心链路

健康系统不应只看单点，而应围绕平台的真实主链路进行观察。

建议至少观察四条链路。

### 9.1 任务链路

```text
task registration
  -> task execution assembly
  -> agent adoption
  -> result projection
```

检查重点：

- 任务对象是否完整
- workflow / projection / flow contract 是否正确绑定
- 任务是否真的交给了目标 Agent

### 9.2 运行体链路

```text
AgentBodyProfile
  -> TaskBodyOrchestration
  -> AgentRuntimeSpec
  -> runtime execution
```

检查重点：

- 稳定身体底座是否可用
- 任务态身体编排是否生成
- 资源门禁是否正确
- 内部状态推进是否异常

### 9.3 记忆链路

```text
TaskMemoryRequestProfile
  -> MemoryRequest
  -> MemoryBundle
  -> runtime consumption
  -> writeback
```

检查重点：

- 是否拿错记忆
- 是否漏拿关键记忆
- 是否写回越界
- 是否出现摘要失真

### 9.4 多 Agent 协调链路

```text
coordination task
  -> topology allocation
  -> communication protocol
  -> handoff
  -> merge / finalize
```

检查重点：

- coordinator 是否正确选出
- 通信协议是否跑通
- handoff 是否丢失关键信息
- 汇总是否与各 participant 输出一致

## 十、健康系统的诊断模式

健康系统至少需要三种诊断模式。

### 10.1 被动诊断

在运行失败或验证失败后触发。

适用于：

- 运行时异常
- 合同校验失败
- 工具调用失败

### 10.2 主动巡检

定期检查关键链路。

适用于：

- 系统管理 Agent 所在系统
- 长期运行平台
- 新版本切换后

### 10.3 修复后验证

在修复动作完成后重新验证。

适用于：

- 任务模板修复
- 运行体装配修复
- 记忆策略修复
- 多 Agent 通信修复

## 十一、健康系统与 runtime 的关系

健康系统不负责执行 runtime，但要能够读取和解释 runtime 证据。

重点包括：

- checkpoint
- event log
- task run ledger
- trace replay
- runtime invariant report

这意味着健康系统必须把 runtime 证据当作一等诊断材料，而不是只看最终报错文本。

## 十二、健康系统与测试 / 回归的关系

健康系统不是测试系统，但它必须消费测试和回归结果。

特别是：

- regression gate
- smoke suite
- long scenario regression
- contract regression

它们共同构成“平台健康”的验证层。

也就是说：

- 测试负责执行用例
- 健康系统负责解释这些结果对平台健康意味着什么

## 十三、前端健康系统管理目标

前端健康系统页面不应是“漂亮的仪表盘”，而应是问题管理与验证工作台。

至少要支持以下管理面：

### 13.1 问题列表

- 查看活动问题
- 按系统域过滤
- 按任务域过滤
- 查看严重级别

### 13.2 链路诊断

- 查看任务链路 trace
- 查看运行体链路 trace
- 查看记忆链路 trace
- 查看多 Agent 协调链路 trace

### 13.3 修复验证

- 查看修复前后对比
- 查看验证结果
- 查看未通过的门禁

### 13.4 回归门禁

- 查看当前 gate 状态
- 查看阻塞项
- 查看允许继续推进的依据

## 十四、现有代码的吸收与清理方向

当前项目里已经有不少健康相关骨架，可以吸收，但边界仍要重组。

### 14.1 可吸收的部分

- `backend/health_system` 的 registry / workbench / runtime admission 雏形
- `backend/harness/regression_gate.py` 的门禁能力
- `backend/capabilities/validation.py` 的验证能力
- runtime loop 的 checkpoint / trace reader
- 现有大批 regression tests 的覆盖结构

### 14.2 需要清理的问题

- health 与 validation、trace、regression 还分散表达
- 健康系统与任务系统、编排系统、记忆系统的接口还不正式
- 前端如果只做总览会很空，不足以支撑真正管理
- 诊断对象、验证对象、门禁对象还缺统一命名

## 十五、实施顺序

建议按以下顺序实施：

1. 统一健康系统命名和边界
2. 定义 `HealthIssueRecord / HealthTraceRecord / HealthDiagnosticReport / HealthVerificationRun`
3. 定义 `RegressionGateProfile / RuntimeAdmissionDecision`
4. 打通 runtime 证据读取链路
5. 打通任务链路、运行体链路、记忆链路的诊断接口
6. 最后再重构前端健康工作台

## 十六、执行结论

健康系统不是平台的附属观察页，而是平台的观测、诊断、验证与回归门禁中心。

它真正守护的是：

- 任务是否真的被正确组织
- Agent 身体是否真的被正确装配
- 记忆是否真的被正确供给
- 多 Agent 协调是否真的被正确执行
- 修复是否真的没有带来新的回归

只有这条线立起来，平台才不是“能跑一次”的系统，而是“能持续运行、持续修复、持续进化”的系统。

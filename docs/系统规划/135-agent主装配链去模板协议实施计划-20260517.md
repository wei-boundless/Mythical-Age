# 135-agent主装配链去模板协议实施计划

日期：2026-05-17

## 1. 问题结论

`TaskTemplateRegistry` 已按旧链路清理计划停止注入默认模板，但主 agent 运行装配仍通过
`ExecutionShape -> ExecutionRecipe -> TaskTemplateRegistry.get_template()` 获取执行结构。

这会让普通对话、RAG、PDF、结构化数据、实时查询和记忆回忆在入口阶段失败，典型错误是
`Unknown execution recipe preset: template.rag.knowledge_answer`。

## 2. 调整目标

1. 主 agent 装配链不再依赖 `TaskTemplateRegistry`。
2. 运行时 recipe 由 `query_understanding`、`current_turn_context`、显式 TaskGraph/task selection、operation requirement 直接生成。
3. `template.*` 不再作为主执行协议；旧字段仅作为历史兼容输出，不参与运行决策。
4. TaskGraph 仍以 `graph_id/task_graph_id/selected_task_id` 和节点 runtime spec 为主线。

## 3. 实施步骤

### 阶段一：建立运行时 recipe 生成

- 改造 `build_execution_recipe`，不再查 `TaskTemplateRegistry`。
- 由 `ExecutionShape` 直接生成 `ExecutionRecipe`。
- recipe id 使用 `runtime.recipe.*`，避免继续暴露旧模板协议。
- 按 source kind / execution kind 填充 required operations、step blueprints、metadata。

### 阶段二：切断装配层旧模板语义

- `build_task_execution_assembly_bundle` 不再创建 `TaskTemplateRegistry`。
- TaskIntentContract 由独立 builder 生成，不再挂在模板注册表上。
- metadata 中保留 `recipe_id/execution_kind/source_kind`，不再把 `template_id` 当主判断依据。

### 阶段三：收敛兼容字段

- `legacy_template_id` 输出为空或 recipe id。
- `_memory_request_profile_payload`、step writeback、bundle 判断改用 `source_kind/execution_kind`。
- Prompt contract 中的 `selected_template` 不再作为主来源。

### 阶段四：验证

- 跑主装配与 runtime 相关回归。
- 重跑 60 轮长场景，确认不再卡在 template preset 缺失。

## 4. 不做事项

- 不恢复默认 TaskTemplate。
- 不引入新的预设任务模式选择器。
- 不把开发说明写入 agent prompt。
- 不伪造测试结果。

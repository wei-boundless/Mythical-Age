# 143-TaskGraph拓扑时序控制系统与通用编辑器闭环设计书-20260517

状态：设计定稿候选  
日期：2026-05-17  
适用范围：TaskGraph 通用编辑器、TaskGraph 编译器、LangGraph 运行内核、调度器、上下文装配、运行监测、人工执行节点交互

## 0. 先给结论

本设计重新定义本项目里的“时序系统”：

时序不是数据库。

时序不是章节、分卷、世界观、大纲、批次、写作 plan。

时序不是节点本身。

时序不是记忆库 selector。

时序不是执行结果回执，也不应该继续用含义不清的 receipt 概念承载运行控制。

时序是 TaskGraph 拓扑在运行期展开出来的控制坐标和许可系统。它回答四个问题：

1. 当前任务图运行到拓扑中的哪个合法位置。
2. 哪些节点此刻允许启动，哪些必须阻塞。
3. 某个正在返回的节点结果是否属于当前仍然打开的合法运行窗口。
4. 图推进后，旧窗口、旧请求、旧结果、旧重试是否必须被关闭、隔离或报警。

一句话：

```text
时序系统 = 拓扑确定的运行位置 + 调度器签发的执行许可 + 运行时钟记录的事件顺序 + 监测系统对越界运行的纠错。
```

它的核心不在“记录了什么业务内容”，而在“谁此刻有资格运行，谁的结果有资格进入下一步”。

补充边界：

```text
用户不手工创建时序。
用户只创建和编辑 TaskGraph 拓扑图，以及必要的拓扑注解。
编辑器必须从拓扑图识别或校验一条从启动到结束的主链。
时序模型必须由编辑器/后端编译器围绕主链从拓扑图确定性编译出来。
```

所以，本设计里说“新增拓扑时序模型编译器”，不是让用户多维护一个时序系统，而是让系统具备从拓扑图自动生成时序控制模型的能力。

- 记忆读写边不能被误当成时序本体。
- 记忆边可以影响节点输入是否满足，但它不是“系统 clock”。
- 时序层只关心节点或控制边是否允许运行，不关心世界观文本、章节大纲这些业务内容。

### 1.4 当前调度器只做 ready/blocked 投影，没有真正的执行许可

`backend/orchestration/runtime_loop/task_graph_scheduler.py`

当前调度器根据这些信息计算状态：

- `node_statuses`
- 上游节点是否 completed
- `phase_id`
- `sequence_index`
- `wait_policy`
- `join_policy`
- `temporal_edges`

当前能做：

- 识别 start 节点。
- 上游未完成时阻塞。
- active phase gate。
- phase 内 active sequence gate。
- 部分支持 `wait_any_upstream_completed`。
- 部分支持反馈边、返修边不进入主依赖。

当前不能做：

- 不签发执行许可。
- 不持有节点激活窗口。
- 不知道一个 running 节点是否仍属于当前拓扑位置。
- 不知道旧请求何时被关闭。
- 不知道结果到达时对应的拓扑位置是否已经过期。
- 不知道并发分支里每个分支的独立生命周期。
- 不知道 loop iteration 的打开和关闭。
- 不知道 background 监测节点和主链节点的不同控制语义。

因此，当前调度器只是状态计算器，不是严格的时序控制器。

### 1.5 LangGraph 当前是运行状态机和 checkpoint 底座，不是 TaskGraph 时序系统

`backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`

```text

  -> route_next
  -> stage_prepare
  -> stage_execute
```

它的作用：

- 持久化协调运行状态。

- 创建 `StageExecutionRequest`。
- 调用上下文装配。
- 写 timeline ledger 事件。

- 结果被接纳、产物落盘、记忆写入之间缺少统一的运行许可边界。


两者必须配合：

- 控制层决定能否打开节点窗口和签发许可。
- 时钟账本记录控制层做出的决定和节点返回事件。
- 监测层用账本对照控制层，发现越界运行。

## 2. 当前系统真正缺失的系统属性

当前不是“某个 if 条件没写好”的问题，而是缺少一个中间层：

```text
编辑器产出的 TaskGraph 静态拓扑
  -> 自动编译成 TopologyTemporalModel
  -> 运行期展开成 TemporalRunState
  -> Scheduler 只根据 TemporalRunState 签发 ExecutionPermit
  -> Agent 只能凭 ExecutionPermit 运行
  -> Result 只能凭 ExecutionPermit 被接纳
```

当前代码直接从 `node_statuses` 和 stage 状态推到下一次执行请求，这导致：

1. 节点被标记 running 后，没有一个可关闭的运行窗口。
2. 同一个节点多次运行只靠 stage_id 叠加，实例边界不清。
3. 图已经推进后，旧 dispatch 的结果仍然可能回来污染状态。
4. 并发、循环、返修、后台监测这些复杂结构没有统一坐标。
5. 监测系统只能看到“最近发生了什么”，不能判断“这件事在拓扑时序上是否合法”。

## 2.0 主链是时序生成的骨架

你提出的“主链”应该成为本设计的一级概念。

主链定义：

```text
主链是 TaskGraph 中从任务启动到任务结束的必经控制路径。
```

它不是业务大纲，也不是写作章节路线，而是任务图的控制骨架。它回答：

1. 任务从哪里开始。
2. 哪些节点构成必经推进路径。
3. 哪些节点决定任务是否进入下一阶段。
4. 哪个节点或哪组终点代表任务结束。
5. 异步、并发、后台监测、人类执行器、人工放行门分别挂在哪个主链时序点上。

因此，正确生成顺序是：

```text
TaskGraph 拓扑图
  -> 识别/校验 MainChain
  -> 以 MainChain 生成主时序轴
  -> 将并发/异步/后台/返修/人类执行器等待点/人工放行门挂到主时序轴锚点
  -> 编译 TopologyTemporalModel
```

如果主链不明确，时序就不应该通过编译。

### 2.0.1 主链如何从拓扑生成

主链不应该由用户单独维护一份列表，而应该由编辑器从拓扑图推导，并允许用户在冲突处明确选择。

输入来源：

- `entry_node_id`
- `output_node_id`
- 节点 `main_chain`
- 节点 `blocks_phase_exit`
- 控制流边
- review gate / human executor binding / manual release gate / barrier
- phase entry / phase exit
- terminal node

推导规则：

1. 如果图声明了 `entry_node_id` 和 `output_node_id`，从 entry 到 output 的控制路径是主链候选。
2. 如果节点 `main_chain=true`，它必须在主链候选路径上，或通过主链分支/汇合节点解释清楚。
3. 如果节点 `blocks_phase_exit=true`，它必须绑定到某个主链时序点，不能漂浮。
4. `runtime_monitor`、后台观察节点默认不是主链，但必须有 `anchor_position` 或 `observe_scope`。
5. memory/artifact resource 节点不是主链节点，但 memory_commit 这类执行节点可以是主链节点。
6. loop_frame 是主链上的一个时序阶段/Frame，不是单个节点；它内部有自己的分层时序模板，并在运行时按迭代动态展开。
7. review_gate/barrier 如果控制任务能否继续，必须在主链上。
8. human executor binding 不是特殊 gate，也不是特殊节点类型，而是任意节点的执行者配置；如果该节点阻塞主链推进，它必须在主链上或绑定主链锚点。
9. manual_release_gate 只表示“人工放行/暂停恢复”这一类纯控制门，不能代替人工审核、人工写作、人工修订节点。

### 2.0.2 主链冲突处理

主链编译失败条件：

- 没有 entry。
- 没有 terminal。
- entry 到 terminal 不连通。
- 存在多个无法判定优先级的主链候选。
- `main_chain=true` 节点不在任何 entry-terminal 控制路径上。
- 阻塞阶段退出的节点没有主链锚点。
- barrier 找不到需要汇合的并发分支。
- background 节点被标为阻塞，但没有 completion policy。

编辑器处理方式：

- 拓扑页标红冲突节点和边。
- 时序页展示主链候选。
- 发布页阻塞发布。
- 用户可以通过设置 `main_chain`、phase entry/exit、edge dependency role 来消歧。

### 2.0.3 主时序轴与挂载点

主链确定后，会生成主时序轴：

```text
main_axis:
  T0 run_start
  T1 entry_node
  T2 planning_gate
  T3 outline_commit
  T4 loop_frame:writing_loop
  T5 final_review
  T6 run_end
```

这里的 T0、T1 不是数据库时间，也不是 wall clock，而是拓扑顺序位置。

异步节点、并发节点、监测节点都必须挂到某个主链锚点：

```text
monitor_node:
  anchor: T0
  lane: background
  lifetime: run_start_to_run_end

parallel_reviewers:
  anchor: T5
  lane: parallel
  join: final_review_barrier

review_node_with_human_executor:
  anchor: T5
  lane: blocking_node_execution
  actor: human
  output_contract: review_verdict

manual_release_gate:
  anchor: T5
  lane: blocking_control_gate
  release: user_decision_only
```

这样，异步节点不是“想什么时候跑就什么时候跑”，而是：

- 何时允许启动。
- 生命周期挂在哪个主链区间。
- 是否阻塞主链前进。
- 返回结果还能否被接纳。

### 2.0.4 主链与时序的关系

主链给出主时序轴。

`TopologyTemporalModel` 在主时序轴上展开：

- phase position
- node position
- loop frame stage
- parallel group
- human execution position
- manual release gate
- background lane
- terminal position

运行时：

- 主链 position 决定任务推进。
- 非主链 position 必须有主链 anchor。
- 没有 anchor 的异步运行就是越界运行。
- 主链推进后，挂在旧 anchor 且未声明跨区间生命周期的 activation 必须关闭。

这正是监测系统判断“节点不在当前时序位置却还在运行”的基础。

## 2.1 LangGraph 与本项目时序系统的关系

项目当前使用 LangGraph 的事实是：

- 用 `StateGraph` 构造协调运行状态机。
- 用 checkpoint 保存和恢复协调状态。
- 用节点函数表达 `stage_accept`、`route_next`、`stage_prepare`、`stage_execute` 等运行步骤。
- 用 `END` 结束一次调度回合。

LangGraph 能提供：

- 状态图执行容器。
- checkpoint / resume。
- 节点函数编排。
- 条件路由。

LangGraph 不能替我们自动提供：

- TaskGraph 拓扑语义。
- phase / sequence / loop / parallel group 的业务解释。
- 节点执行许可。
- late result 隔离。
- 记忆读写边与产物边的领域约束。
- 前端编辑器中的可视化诊断闭环。

因此本项目的正确关系是：

```text
LangGraph = 运行状态机和 checkpoint 底座。
TaskGraph TopologyTemporalControl = 本项目自己的拓扑时序控制层。
```

LangGraph 负责“状态机怎么被调用和持久化”，不负责“这个 TaskGraph 节点此刻是否在合法时序位置上”。后者必须由我们自己的 `TopologyTemporalModel + TemporalRunState + ExecutionPermit + TemporalGate` 完成。

这也是为什么不能把 LangGraph 的 step 顺序直接等同于 TaskGraph 时序。LangGraph 里的 `stage_accept -> route_next -> stage_prepare -> stage_execute` 是运行内核内部处理步骤；TaskGraph 时序说的是用户画出来的任务图节点、边、循环、并发、审核门在业务拓扑中的合法运行位置。

但这不代表 LangGraph step 不能使用。

正确用法是：

```text
TaskGraph Temporal Step = 业务拓扑中的时序步。
LangGraph Step = 执行内核里处理该时序步的一组状态机节点。
```

两者可以结合，但所有 LangGraph step 必须绑定在 TaskGraph temporal coordinate 之下：

```text
temporal_coordinate: main/T2/loop/world_review_loop/iter/001/step/S2
langgraph_steps:
  - accept_previous_result
  - prepare_context
  - create_request
  - dispatch_agent
```

因此，LangGraph step 是实现机制，TaskGraph temporal step 是控制语义。不能反过来。

## 2.2 对接链路总览

目标链路必须固定为：

```text
编辑器 TaskGraphDefinition
  -> RuntimeSpec
  -> TopologyTemporalModel
  -> TemporalRunState
  -> SchedulerDecision
  -> ExecutionPermit
  -> NodeExecutionRequest
  -> Agent Runtime
  -> TaskResultReadyEvent
  -> TemporalGate.validate_result
  -> accepted/quarantined
  -> downstream activation
```

每一步的生产者、消费者、生命周期如下：

| 步骤 | 生产者 | 消费者 | 生命周期 | 不允许 |
| --- | --- | --- | --- | --- |
| `TaskGraphDefinition` | 前端编辑器 / 后端任务系统 | compiler | 用户保存到发布 | 不允许运行时临时猜字段 |
| `RuntimeSpec` | coordination graph compiler | temporal compiler / contract compiler | 每次发布或运行启动固定 | 不允许混入运行结果 |
| `TopologyTemporalModel` | temporal compiler | scheduler / monitor / publish diagnostics | 与图版本绑定 | 不允许包含章节正文等业务数据 |
| `TemporalRunState` | runtime bootstrap / scheduler / gate | scheduler / monitor / context resolver | 每个 coordination run 独立 | 不允许由前端直接改 |
| `ExecutionPermit` | scheduler + TemporalGate | request builder / result gate | 单次 activation 内有效 | 不允许手写伪造 |
| `NodeExecutionRequest` | runtime | agent runtime / context assembly | 派发一次节点运行 | 不允许无 permit 创建 |
| `TaskResultReadyEvent` | task_run_loop / agent runtime | TemporalGate | 子任务完成时产生 | 不允许绕过 gate 直接更新图 |
| accepted result | TemporalGate + business acceptance | artifact promote / memory commit / scheduler | activation 接纳一次 | 不允许同 activation 重复 accepted |
| quarantined result | TemporalGate | monitor / 用户排障 | 永久审计 | 不允许推进下游 |

## 3. 时序系统的精确定义

### 3.1 时序是什么

在本项目中，时序是 TaskGraph 拓扑的运行期展开控制。

它由四个结构组成：

1. `TopologyTemporalModel`

   从 TaskGraph 静态图确定性编译出来的合法运行结构。用户不直接创建它。

2. `TemporalRunState`

   某一次运行中，当前打开了哪些拓扑位置，哪些已经关闭，哪些正在等待。

3. `ExecutionPermit`

   调度器对某个节点的一次合法运行授权。

4. `RuntimeClock`

   对打开、签发、派发、返回、关闭、违规等事件分配单调顺序。

### 3.2 时序不是什么

不是业务数据：

- 不保存世界观文本。
- 不保存章节正文。
- 不保存大纲。
- 不保存分卷计划。
- 不保存记忆库内容。

不是资源选择：

- 不决定某节点读取哪个 collection。
- 不决定 memory selector 的 version。
- 不决定产物展开多少字符。

不是 prompt：

- 不替节点猜任务。
- 不把开发说明塞给 agent。
- 不替代职责页的投影 prompt。

不是执行结果本身：

- 不把 result record 或 receipt 当作时序。
- 时序只判断结果是否属于一个仍然有效的运行窗口。

### 3.3 时序为什么有用

时序系统解决的是“任务图运行的合法性”。

它必须能阻止或诊断这些问题：

1. 旧 dispatch 继续运行。
2. 旧 dispatch 的结果在图推进后返回。
3. 已经通过审核并提交后，同一批旧写手还继续产出。
4. 同一节点同一合法窗口重复提交两次。
5. 被拒绝的产物落到正式目录。
6. 并发分支中部分分支未完成就提前 join。
7. loop 旧迭代的结果覆盖新迭代状态。
8. background 节点越过监测权限去影响主链。
9. human execution node 未提交合法结果时下游提前运行。
10. manual release gate 未放行时下游提前运行。
11. 记忆写入发生在未通过审核的运行窗口里。

这些问题不是靠更好的 prompt 解决的，而是靠运行控制结构解决。

## 4. 目标模型

### 4.1 TopologyTemporalModel

`TopologyTemporalModel` 是编译期产物。

来源：

- TaskGraph nodes
- TaskGraph edges
- phase definitions
- sequence index
- timeline group
- loop frames
- wait policy
- join policy
- human execution policy
- manual release gate policy
- review gate policy
- background policy

不读取：

- 当前章节内容。
- 记忆库正文。
- agent 输出。
- 运行中临时结果。

建议结构：

```python
@dataclass(frozen=True, slots=True)
class TopologyTemporalModel:
    model_id: str
    graph_id: str
    graph_version_hash: str
    main_chain: TemporalMainChain
    phases: tuple[TemporalPhase, ...]
    positions: tuple[TemporalPositionDefinition, ...]
    control_edges: tuple[TemporalControlEdge, ...]
    loop_frames: tuple[LoopFrameDefinition, ...]
    parallel_groups: tuple[TemporalParallelGroup, ...]
    executor_bindings: tuple[NodeExecutorBinding, ...]
    manual_release_gates: tuple[TemporalManualReleaseGate, ...]
    background_lanes: tuple[TemporalBackgroundLane, ...]
    start_position_ids: tuple[str, ...]
    terminal_position_ids: tuple[str, ...]
    diagnostics: dict[str, Any]
```

其中 `main_chain` 是核心骨架：

```python
@dataclass(frozen=True, slots=True)
class TemporalMainChain:
    chain_id: str
    entry_position_id: str
    terminal_position_ids: tuple[str, ...]
    position_ids: tuple[str, ...]
    edge_ids: tuple[str, ...]
    anchor_points: tuple[TemporalAnchorPoint, ...]
    diagnostics: dict[str, Any]


@dataclass(frozen=True, slots=True)
class TemporalAnchorPoint:
    anchor_id: str
    position_id: str
    phase_id: str
    sequence_index: int
    allowed_lanes: tuple[str, ...]
```

主链不是另一张用户手写表，而是编译器从图中识别出的控制骨架。`anchor_points` 用来挂载异步、并发、后台监测、人类执行器等待点、人工放行门等非直线运行。

循环相关模型必须区分静态模板和运行实例：

```python
@dataclass(frozen=True, slots=True)
class LoopFrameDefinition:
    loop_frame_id: str
    main_axis_position_id: str
    template: LoopBodyTemporalTemplate
    exit_policy: dict[str, Any]
    max_iterations: int
    diagnostics: dict[str, Any]


@dataclass(frozen=True, slots=True)
class LoopBodyTemporalTemplate:
    loop_frame_id: str
    entry_position_id: str
    body_position_ids: tuple[str, ...]
    decision_position_id: str
    repair_edge_ids: tuple[str, ...]
    exit_edge_ids: tuple[str, ...]
    internal_control_edges: tuple[str, ...]
```

`LoopFrameDefinition` 进入 `TopologyTemporalModel`，`LoopIterationInstance` 只在运行时进入 `TemporalRunState`。

### 4.2 TemporalPositionDefinition

这是静态拓扑位置定义，不是运行实例。

建议字段：

```python
@dataclass(frozen=True, slots=True)
class TemporalPositionDefinition:
    position_id: str
    node_id: str
    main_chain_role: str
    anchor_position_id: str
    phase_id: str
    sequence_index: int
    timeline_group_id: str
    execution_mode: str
    execution_actor: str
    wait_policy: str
    join_policy: str
    blocks_phase_exit: bool
    frame_id: str
    lane: str
    is_background: bool
    human_interaction_required: bool
    is_manual_release_gate: bool
    is_barrier: bool
```

`main_chain_role` 取值：

- `main`
- `branch`
- `parallel`
- `background`
- `resource`
- `gate`
- `loop_frame`

规则：

- `main` 必须出现在主链 position_ids 中。
- `background` 必须有 `anchor_position_id`。
- `parallel` 必须有 join/barrier 或明确的 non_blocking policy。
- `resource` 不直接推进主链，只通过边参与输入输出。
- `execution_actor` 取值为 `agent`、`human`、`tool`、`subgraph`；时序层只关心谁负责完成这个 activation，不把 human 当旁路。
- `human_interaction_required=true` 表示该 position 运行时要创建人工执行请求，并等待用户提交节点结果。
- `is_manual_release_gate=true` 只用于纯控制放行，不用于承载审核意见、正文、修订稿或 commit 决策。

它不包含：

- volume_index
- chapter_index
- batch_start_index
- world_version
- memory selector

这些属于业务输入和资源配置。

### 4.3 TemporalRunState

`TemporalRunState` 是某次运行的真实时序状态。

建议结构：

```python
@dataclass(frozen=True, slots=True)
class TemporalRunState:
    coordination_run_id: str
    model_id: str
    runtime_clock_seq: int
    active_activation_ids: tuple[str, ...]
    active_permit_ids: tuple[str, ...]
    active_loop_iteration_ids: tuple[str, ...]
    activation_index: dict[str, NodeActivation]
    permit_index: dict[str, ExecutionPermit]
    loop_iteration_index: dict[str, LoopIterationInstance]
    position_statuses: dict[str, str]
    waiting_join_ids: tuple[str, ...]
    waiting_human_interaction_ids: tuple[str, ...]
    waiting_manual_release_gate_ids: tuple[str, ...]
    closed_activation_ids: tuple[str, ...]
    violation_ids: tuple[str, ...]
```

`node_statuses` 以后只允许作为投影，不允许作为调度真相。

### 4.4 NodeActivation

`NodeActivation` 表示拓扑位置的一次合法打开。

它不是节点定义，也不是业务批次。

建议结构：

```python
@dataclass(frozen=True, slots=True)
class NodeActivation:
    activation_id: str
    position_id: str
    node_id: str
    phase_id: str
    frame_id: str
    iteration_index: int
    parallel_group_id: str
    parent_activation_ids: tuple[str, ...]
    opened_by_event_id: str
    opened_at_clock_seq: int
    closed_by_event_id: str
    closed_at_clock_seq: int
    status: str
    close_reason: str
```

允许状态：

- `open`
- `permitted`
- `running`
- `waiting_human_input`
- `waiting_manual_release`
- `completed`
- `failed`
- `cancelled`
- `closed`
- `stale`

关键约束：

```text
一个节点可以有多次 activation。
同一个 activation 只能被接受一次。
图推进后，旧 activation 必须关闭。
关闭后的 activation 返回结果，必须进入 quarantine，不得推进图。
```

### 4.5 ExecutionPermit

`ExecutionPermit` 表示调度器给某个 activation 签发的一次运行许可。

它是结果接纳的唯一凭证。

建议结构：

```python
@dataclass(frozen=True, slots=True)
class ExecutionPermit:
    permit_id: str
    activation_id: str
    position_id: str
    node_id: str
    coordination_run_id: str
    request_id: str
    dispatch_event_id: str
    issued_at_clock_seq: int
    expires_at_clock_seq: int
    status: str
    revoke_reason: str
```

允许状态：

- `issued`
- `dispatched`
- `running`
- `consumed`
- `revoked`
- `expired`

关键约束：

```text
没有 permit，不允许创建节点执行请求。
结果没有 permit，不允许进入 stage/node result。
permit 与 request_id、dispatch_event_id、node_id、coordination_run_id 必须一致。
permit 所属 activation 已关闭时，结果必须隔离。
permit consumed 后再次返回结果，必须判定为 duplicate_result。
```

### 4.6 RuntimeClock

当前 `TimelineLedgerStore` 可以作为 RuntimeClock 的存储底座，但需要改名或重新定义边界。

建议命名：

- 类名可以逐步改为 `RuntimeClockLedgerStore`。
- 文件可以是 `runtime_clock_ledger.py`。
- 旧 `timeline_ledger.py` 如果保留，只能作为兼容包装，不再作为核心设计术语。

RuntimeClock 记录：

- `run_started`
- `temporal_model_loaded`
- `activation_opened`
- `execution_permit_issued`
- `node_dispatch_requested`
- `node_result_received`
- `node_result_accepted`
- `node_result_rejected`
- `activation_closed`
- `join_released`
- `human_interaction_requested`
- `human_result_submitted`
- `manual_release_gate_opened`
- `manual_release_gate_resolved`
- `temporal_violation_detected`

RuntimeClock 不负责：

- 判断谁能运行。
- 读取记忆。
- 写入产物。
- 选择下一节点。

### 4.7 TemporalGate

`TemporalGate` 是纯校验器。

职责：

1. 调度前校验 position 是否可打开。
2. 派发前校验 permit 是否有效。
3. 结果返回时校验 result 是否属于 active permit。
4. 图推进时关闭旧 activation。
5. 对非法事件生成 `TemporalViolation`。

示例 API：

```python
class TemporalGate:
    def can_open_position(model, run_state, position_id) -> GateDecision: ...
    def issue_permit(model, run_state, activation_id) -> PermitIssueResult: ...
    def validate_dispatch(run_state, permit_id, request_payload) -> GateDecision: ...
    def validate_result(run_state, event_payload) -> GateDecision: ...
    def close_activation(run_state, activation_id, reason) -> TemporalRunState: ...
```

## 5. 调度算法设计

### 5.1 Bootstrap

启动运行时：

1. 编译 `TaskGraphDefinition` 为 `TaskGraphRuntimeSpec`。
2. 编译 `TaskGraphRuntimeSpec` 为 `TopologyTemporalModel`。
3. 创建空 `TemporalRunState`。
4. 打开 start positions。
5. 对允许立即执行的 activation 签发 permit。
6. 创建节点执行请求。

当前代码里 bootstrap 直接把首节点设为 running，这一步必须改掉。

正确流程：

```text
graph start
  -> compile temporal model
  -> open start activation
  -> issue permit
  -> create node execution request
  -> dispatch agent
```

事件序列必须是：

```text
run_started
temporal_model_compiled
temporal_run_state_created
activation_opened(start_position)
execution_permit_issued(start_activation)
node_dispatch_requested(start_activation, permit)
node_execution_request_created(permit)
```

如果任一事件失败：

- 不创建 agent 子任务。
- 不写 official artifact。
- 不写 memory candidate。
- Monitor 显示 `temporal_bootstrap_failed`。

### 5.2 节点派发

派发必须携带：

- `coordination_run_id`
- `node_id`
- `activation_id`
- `permit_id`
- `request_id`
- `dispatch_event_id`
- `temporal_position`

`StageExecutionRequest` 需要重命名或被新结构替换为 `NodeExecutionRequest`。

如果为降低一次性改动风险暂时保留旧类名，字段必须先完整进入请求：

```python
activation_id: str
execution_permit_id: str
temporal_position: dict[str, Any]
```

但设计语义上不再把 stage 当作核心对象。核心对象是 graph node 的运行实例。

`NodeExecutionRequest` 的最小字段：

```python
@dataclass(frozen=True, slots=True)
class NodeExecutionRequest:
    request_id: str
    coordination_run_id: str
    root_task_run_id: str
    node_id: str
    task_ref: str
    agent_id: str
    activation_id: str
    execution_permit_id: str
    dispatch_event_id: str
    temporal_position: dict[str, Any]
    explicit_inputs: dict[str, Any]
    runtime_assembly: dict[str, Any]
    a2a_payload: dict[str, Any]
    memory_snapshot: dict[str, Any]
    artifact_context_packet: dict[str, Any]
    revision_packet: dict[str, Any]
    artifact_policy: dict[str, Any]
    stream_policy: dict[str, Any]
    idempotency_key: str
```

`request_id` 的身份种子必须来自：

```text
coordination_run_id + activation_id + permit_id + node_id
```

不能再只用 `stage_id + explicit_inputs`，因为同一节点可能在不同 activation 中合法运行多次。

### 5.3 结果接纳

结果接纳必须经过 `TemporalGate.validate_result`。

校验顺序：

1. `coordination_run_id` 是否匹配。
2. `permit_id` 是否存在。
3. `request_id` 是否匹配 permit。
4. `dispatch_event_id` 是否匹配 permit。
5. `node_id` 是否匹配 activation。
6. activation 是否仍 open/running。
7. permit 是否仍 active。
8. permit 是否未 consumed。
9. 如果节点是 human execution position、manual release gate 或 background lane，是否满足对应特殊策略。

只有全部通过，才能进入：

- accepted node result
- artifact commit
- memory write commit
- downstream activation opening

任何失败都进入：

- quarantined result
- temporal violation
- monitor alert

结果事件最小字段：

```python
@dataclass(frozen=True, slots=True)
class NodeResultReadyEvent:
    event_type: str
    coordination_run_id: str
    task_run_id: str
    node_id: str
    activation_id: str
    execution_permit_id: str
    request_id: str
    dispatch_event_id: str
    task_ref: str
    task_result_ref: str
    artifact_refs: tuple[str, ...]
    accepted_by_agent_runtime: bool
    diagnostics: dict[str, Any]
```

如果旧子任务只能返回 `TaskResultReadyEvent`，兼容层只能做字段映射，不能跳过 permit 校验。缺少 permit 的旧事件必须被标记：

```text
temporal_violation: result_without_permit
```

### 5.4 下游推进

下游推进不再看简单的 `node_statuses[source] == completed`。

它必须看：

```text
source activation completed
source activation result accepted
edge handoff contract satisfied
target wait policy satisfied
target join policy satisfied
phase exit policy satisfied
loop frame policy satisfied
human execution result accepted if target depends on human node
manual release gate resolved if required
```

然后才能打开 target activation。

下游打开事件序列：

```text
node_result_accepted(source_activation)
activation_closed(source_activation, reason=completed)
control_edge_satisfied(edge)
activation_opened(target_position)
execution_permit_issued(target_activation)
```

如果同一 source activation 第二次返回结果：

```text
node_result_received
temporal_violation_detected(duplicate_result_for_consumed_permit)
result_quarantined
```

不得再次触发 downstream activation。

### 5.5 并发

并发不是“多个节点都 running”这么简单。

并发需要：

- 同一 phase/sequence 或同一 parallel group 同时打开多个 activation。
- 每个 activation 独立 permit。
- join 节点等待指定 group 的 activation 终态。
- join policy 决定释放条件。

示例：

```text
parallel group reviewers
  reviewer_a activation A1 permit P1
  reviewer_b activation A2 permit P2
  reviewer_c activation A3 permit P3
barrier waits:
  all_success -> A1/A2/A3 全 accepted
  quorum -> 满足票数
  allow_partial_with_issues -> 全终态，至少一个成功
```

### 5.6 循环体时序

你指出的修正是对的：循环体不是普通节点，也不是挂在主链上的松散附属物。

循环体本身应该是主时序轴上的一个时序阶段/Frame。

它有两层时序：

1. 静态时序模板

   编辑器和编译器从拓扑图生成，确定循环体内部有哪些入口、步骤、审核门、返修边、出口条件。

2. 运行时动态坐标

   因为循环次数不确定，所以每一次迭代的具体时间坐标不能在发布时全部生成，只能在运行时按需创建。

因此循环体模型应分成：

```text
LoopFrameDefinition
  -> LoopBodyTemporalTemplate
  -> LoopIterationInstance(runtime)
```

### 5.6.1 LoopFrameDefinition

`LoopFrameDefinition` 是主链上的一个时序阶段。

它定义：

- `loop_frame_id`
- 主链锚点 `main_axis_position_id`
- 循环入口 position
- 循环体内部 position 模板
- 判定/审核 position
- 返修边
- 退出边
- 最大迭代保护
- 退出条件来源

示例：

```text
main_axis:
  T0 run_start
  T1 world_design
  T2 loop_frame:world_review_loop
  T3 outline_design

loop_frame:world_review_loop:
  entry: world_candidate
  body:
    S1 world_candidate
    S2 world_review
  decision:
    pass -> exit_to T3
    revise -> next_iteration
```

这里 `T2` 是主时序轴上的一个确定阶段；`S1/S2` 是循环体内部的静态分层模板。

### 5.6.2 LoopBodyTemporalTemplate

循环体内部也要有时序分层。这里可以复用 step 概念。

但要分清两种 step：

```text
LoopBodyStep = TaskGraph 循环体内部的时序步。
LangGraph step = 执行这个时序步时，运行内核内部的状态机处理步骤。
```

LoopBodyStep 属于拓扑时序；LangGraph step 属于执行实现。

模板结构：

```python
@dataclass(frozen=True, slots=True)
class LoopBodyTemporalTemplate:
    loop_frame_id: str
    entry_position_id: str
    steps: tuple[LoopBodyStepTemplate, ...]
    body_position_ids: tuple[str, ...]
    decision_position_id: str
    repair_edge_ids: tuple[str, ...]
    exit_edge_ids: tuple[str, ...]
    internal_control_edges: tuple[str, ...]


@dataclass(frozen=True, slots=True)
class LoopBodyStepTemplate:
    step_id: str
    step_index: int
    title: str
    position_ids: tuple[str, ...]
    parallel_group_id: str
    wait_policy: str
    join_policy: str
    langgraph_step_binding: tuple[str, ...]
```

模板回答：

- 每轮从哪里开始。
- 每轮内部按什么顺序运行。
- 哪些节点可以并发。
- 哪个节点决定继续还是退出。
- 继续时回到哪个内部位置。
- 退出时回到主链哪个下一个位置。
- 每个内部 step 可以由哪些 LangGraph runtime steps 执行。

模板不回答：

- 具体跑第几轮。
- 一共跑几轮。
- 每轮业务内容是什么。

示例：

```text
loop_frame:world_review_loop
  body_steps:
    S1 candidate_generation
      positions: [world_candidate]
      langgraph_step_binding:
        - prepare_context
        - create_request
        - dispatch_agent
    S2 review_gate
      positions: [world_review]
      langgraph_step_binding:
        - accept_previous_result
        - prepare_context
        - create_request
        - dispatch_agent
    S3 decision
      positions: [world_review]
      route:
        pass -> exit
        revise -> next_iteration
```

发布时只能确定 `S1/S2/S3` 这些内部 step 模板。运行时才会生成：

```text
iter/001/S1
iter/001/S2
iter/001/S3
iter/002/S1
...
```

### 5.6.3 LoopIterationInstance

运行时每进入一轮循环，就创建一个临时迭代坐标。

建议结构：

```python
@dataclass(frozen=True, slots=True)
class LoopIterationInstance:
    iteration_instance_id: str
    loop_frame_id: str
    main_axis_position_id: str
    iteration_index: int
    current_step_id: str
    current_step_position_ids: tuple[str, ...]
    parent_activation_id: str
    status: str
    opened_at_clock_seq: int
    closed_at_clock_seq: int
    internal_activation_ids: tuple[str, ...]
```

运行时坐标示例：

```text
run/R1/main/T2/loop/world_review_loop/iter/001/step/S1
run/R1/main/T2/loop/world_review_loop/iter/001/step/S2
```

下一轮不是复用上一轮坐标，而是创建：

```text
run/R1/main/T2/loop/world_review_loop/iter/002/step/S1
run/R1/main/T2/loop/world_review_loop/iter/002/step/S2
```

这才是“不确定循环次数”的正确处理方式：发布时有循环体模板，运行时动态创建每轮坐标。

### 5.6.4 循环体内的 activation/permit

每一轮内部节点都必须有自己的 activation 和 permit。

```text
LoopIterationInstance(iter=001)
  S1 world_candidate
    activation: act.iter001.S1
    permit: permit.iter001.S1
  S2 world_review
    activation: act.iter001.S2
    permit: permit.iter001.S2
```

如果审核结果是 revise：

```text
close act.iter001.S2
close LoopIterationInstance(iter=001, reason=continue)
open LoopIterationInstance(iter=002)
open act.iter002.S1
issue permit.iter002.S1
```

如果审核结果是 pass：

```text
close act.iter001.S2
close LoopIterationInstance(iter=001, reason=exit)
close LoopFrameActivation(world_review_loop)
open main_axis T3
```

### 5.6.5 过期结果处理

如果 iter001 关闭后，iter001 的写手或审核节点又返回结果：

```text
result coordinate = main/T2/loop/world_review_loop/iter/001/step/S1
current coordinate = main/T2/loop/world_review_loop/iter/002/step/S1
```

TemporalGate 必须判定：

```text
temporal_violation: result_for_closed_loop_iteration
```

处理：

- 进入 quarantine。
- 不更新当前 iteration。
- 不推进主链。
- 不写 official artifact。
- 不写正式记忆库。

### 5.6.6 循环体与业务循环的边界

循环体时序只表达拓扑运行重复。

它可以是：

- 审核返修循环。
- 批处理直到队列为空。
- 多轮规划直到人工确认。
- 并发检查直到 barrier 满足。

它不是：

- 章节编号。
- 分卷编号。
- 大纲版本。
- 世界观版本。

这些业务变量可以放在每轮 `explicit_inputs` 或 memory selector 中，但不能替代 loop iteration coordinate。

### 5.7 阻塞与非阻塞

`blocks_phase_exit` 必须进入控制层。

规则：

- 主链阻塞节点完成前，phase 不允许退出。
- background 监测节点默认不阻塞 phase exit。
- 如果 background 节点显式配置 `blocks_phase_exit=true`，必须配置 completion policy，否则发布失败。
- fire_and_continue 节点可以不阻塞后续，但它的 late result 仍必须受 permit 约束。

### 5.8 人工执行节点

这里需要改正：人工审核不只是“暂停，让用户点一下通过或拒绝”。

更准确的定义是：

```text
人工执行节点 = 用用户代替 AI 执行某个 TaskGraph 节点。
```

也就是说，节点仍然是 TaskGraph 节点，仍然有 activation、permit、输入包、prompt/职责说明、输出契约、产物策略、记忆读写边。区别只是执行者不是 agent runtime，而是用户。

这里必须避免一个误解：

```text
人工审核节点不是“系统跑完以后弹窗问用户同不同意”。
人工写作节点也不是“让用户在流程外补一个文件”。
它们都是图里的节点，只是 execution_actor = human。
```

因此，人工节点运行时收到的不是一个泛泛的提示，而是这个节点原本就应该收到的完整执行包：

- 上游节点交接过来的输出。
- 记忆读边读取到的快照。
- 产物上下文边读取到的稿件、设定、大纲、审核意见。
- 返修边携带的原稿、问题清单、返修要求。
- 节点职责说明。
- 输出契约。
- 本节点允许执行的结构化动作。

用户在浮窗里提交的也不是普通聊天消息，而是该节点的正式运行结果。这个结果会像 agent 输出一样进入同一条 result accept 流程，接受 permit、时序、输出契约、产物策略、记忆写入策略的校验。

因此人工执行节点有三种典型用法：

1. 人工作为审核员

   用户收到被审核产物、审核标准、记忆快照、输出契约，然后写审核意见，选择通过、退回、转人工补充、终止，并决定是否允许 commit。

2. 人工作为写手

   用户收到大纲、世界观、章节目标、写作要求，然后亲自写正文并提交。提交后，后续 review 节点照常审核。

3. 人工作为修订者

   用户收到 review 意见、原稿、返修要求，然后亲自改稿并提交给后续节点。

### 5.8.1 人工节点与 AI 节点同构

人工节点不能成为特殊旁路。

它必须和 AI 节点使用同一套运行结构：

```text
NodeActivation
  -> ExecutionPermit
  -> NodeExecutionRequest
  -> HumanInteractionRequest
  -> HumanSubmittedResult
  -> TemporalGate.validate_result
  -> accepted/quarantined
```

核心规则：

- 人工节点也必须有 permit。
- 用户看到的是该节点本来要收到的执行包。
- 用户提交的内容必须符合该节点输出契约。
- 用户提交后生成与 agent 节点同形的 result event。
- 下游节点不关心结果来自 AI 还是人，只关心结果是否通过 TemporalGate 和业务契约。

节点执行器抽象建议：

```python
@dataclass(frozen=True, slots=True)
class NodeExecutorBinding:
    node_id: str
    execution_actor: str  # agent | human | tool | subgraph
    agent_profile_id: str
    human_interaction_schema_id: str
    tool_binding_id: str
    subgraph_id: str
```

规则：

- `execution_actor=agent` 时，运行时把执行包送给 agent。
- `execution_actor=human` 时，运行时把执行包送到常驻运行交互浮窗。
- `execution_actor=tool` 时，运行时调用工具适配器。
- `execution_actor=subgraph` 时，运行时进入子图。
- 四类执行器都必须持有 activation 和 permit。
- 四类执行器提交的结果都必须通过同一套 result accept 流程。

### 5.8.2 HumanInteractionRequest

当 execution actor 是 human 时，运行时不调用 agent，而是创建 `HumanInteractionRequest`，并推送到运行交互浮窗。

建议结构：

```python
@dataclass(frozen=True, slots=True)
class HumanInteractionRequest:
    interaction_id: str
    coordination_run_id: str
    node_id: str
    activation_id: str
    execution_permit_id: str
    request_id: str
    task_title: str
    actor_role: str
    source_edge_ids: tuple[str, ...]
    input_packages: tuple[dict[str, Any], ...]
    instruction_text: str
    output_contract: dict[str, Any]
    interaction_schema: dict[str, Any]
    allowed_actions: tuple[str, ...]
    artifact_policy: dict[str, Any]
    memory_write_policy: dict[str, Any]
    status: str
```

`instruction_text` 不是开发说明，也不是“这是 runtime 节点”这一类描述。它必须从节点职责、投影身份、输入包说明、输出契约组合成用户能执行的任务说明，例如：

```text
你现在代替“章节审核员”节点工作。
你需要阅读本次提交的章节正文、章节细纲、世界观设定和上一轮返修意见。
你只负责判断这份正文是否可以进入正式提交。
如果不通过，请写出必须修改的问题，并选择 revise。
如果通过，请选择 pass；只有当本节点配置允许提交时，才能选择 allow_commit。
```

`interaction_schema` 决定浮窗里有哪些输入控件：

- 正文编辑区。
- 审核意见编辑区。
- 裁决选择。
- commit 许可选择。
- 附件/产物引用选择。
- 需要更多上下文的请求按钮。

这些控件必须由节点契约生成，不能由前端随意猜。

`allowed_actions` 由节点类型和输出契约决定，可以包括：

- `submit_content`
- `submit_review`
- `pass`
- `revise`
- `reject`
- `commit`
- `request_more_context`
- `pause`
- `abort`

这些动作不是 UI 自由发挥，必须来自节点契约。

### 5.8.3 HumanSubmittedResult

用户提交后生成结果事件。

建议结构：

```python
@dataclass(frozen=True, slots=True)
class HumanSubmittedResult:
    event_type: str
    coordination_run_id: str
    node_id: str
    activation_id: str
    execution_permit_id: str
    request_id: str
    actor_type: str
    actor_id: str
    submitted_text: str
    structured_outputs: dict[str, Any]
    verdict: str
    commit_decision: str
    artifact_refs: tuple[str, ...]
    diagnostics: dict[str, Any]
```

其中：

- `actor_type = "human"`
- `verdict` 用于审核节点。
- `commit_decision` 用于允许写入正式记忆库或正式产物。
- `submitted_text` 可以是正文、审核意见、返修稿。
- `structured_outputs` 必须按输出契约解析。

### 5.8.4 人工审核的具体流程

如果用户配置“人工审核节点”，流程是：

```text
上游产物完成
  -> 打开人工审核节点 activation
  -> 签发 permit
  -> 装配审核节点输入包
  -> 弹出运行交互浮窗
  -> 用户阅读产物和上下文
  -> 用户填写审核意见和裁决
  -> 用户选择 pass/revise/reject/commit
  -> 生成 HumanSubmittedResult
  -> TemporalGate 校验 permit
  -> 输出契约校验 verdict/commit_decision
  -> 通过则打开下游或 commit；退回则打开返修节点
```

注意：commit 不是 UI 按钮直接改库。commit 仍然必须经过 TaskGraph 的 commit 边和当前节点契约。如果人工审核节点拥有 commit 权限，它的 `commit_decision=allow` 只是允许后续 commit activation 执行，或允许当前节点按明确配置执行 commit。

### 5.8.5 人工写作的具体流程

如果用户配置“人工写作节点”，流程是：

```text
大纲/记忆/任务目标准备完成
  -> 打开人工写作节点 activation
  -> 签发 permit
  -> 装配写作执行包
  -> 运行交互浮窗展示：身份、任务、输入包、写作要求、输出契约
  -> 用户写正文并提交
  -> 生成 HumanSubmittedResult
  -> TemporalGate 校验 permit
  -> 产物进入 staging
  -> 后续 review 节点照常运行
```

这样“人代替 AI 写作”不会绕过流程，也不会直接污染正式产物目录。

### 5.8.6 人工节点的时序语义

人工节点在时序上不是暂停点，而是一个正在等待人类执行的 activation。

状态建议：

- `waiting_human_input`
- `human_submitted`
- `human_result_validating`
- `completed`
- `rejected`
- `expired`

如果用户长时间不提交：

- activation 保持打开。
- permit 可按超时策略保持、续期或过期。
- monitor 显示 `waiting_human_input`。
- 下游不得打开。

### 5.8.7 前端交互原则

人工执行使用同一个运行交互浮窗。

浮窗必须展示：

- 节点身份：你现在代替哪个节点工作。
- 节点职责：你是审核员、写手、修订者，还是提交确认者。
- 输入包：上游产物、记忆快照、返修包、交接说明。
- 输出区：用户写正文或审核意见。
- 结构化裁决区：pass/revise/reject/commit 等。
- 提交前契约检查。

这不是普通聊天窗口，而是节点执行窗口。用户提交的是节点结果。

### 5.9 后台监测节点

监测节点是 TaskGraph 中的 `runtime_monitor` 节点。

它是异步后台节点，但不能脱离任务系统。

规则：

- 启用监测机制时，编译器创建或保留 runtime_monitor position。
- 运行开始时打开 monitor activation。
- monitor activation 可以长时间保持 active。
- monitor permit 允许它读取运行状态、RuntimeClock、TemporalRunState、最近事件。
- monitor 不允许直接改写主链状态。
- monitor 可以提交 `MonitorActionProposal`。

可提出动作：

- attention_required
- pause_run
- revoke_permit
- retry_activation
- restart_from_checkpoint
- request_human_review
- mark_quarantined_result

真正执行动作由控制 API 和 TemporalGate 校验。

## 6. 记忆、产物、上下文与时序的边界

### 6.1 记忆读写是边，不是时序

用户在前端建立记忆库节点，后端建立仓库分区。

节点通过连线读写记忆库：

```text
memory_repository/world_canon -> memory_read edge -> writer_node
review_node -> memory_write_candidate edge -> memory_repository/world_canon
review_gate -> memory_commit edge -> memory_repository/world_canon
```

这些边决定“节点能读什么，能写什么，写入哪里”。

时序只决定“这个节点此刻是否允许进行这次读写”。

### 6.2 读记忆

读记忆发生在节点 activation 派发前：

1. TemporalGate 确认 permit 有效。
2. Context resolver 找到进入该节点的 memory_read 边。
3. Formal memory service 根据 repository、collection、selector、version_selector 读取。
4. 生成 memory snapshot。
5. snapshot 放入 NodeExecutionRequest。

如果没有 permit，禁止读取。

如果 memory_read 缺 selector，发布页阻塞。

如果 selector 读不到必需记录，根据 `on_missing` 决定 block、warn 或 empty。

### 6.3 写记忆

写记忆分三层：

1. candidate write

   节点产出候选记录，进入 staging。

2. review gate

   审核节点判断是否允许提交。

3. commit

   commit 边把已通过记录写入正式 collection。

时序约束：

- candidate write 只能发生在有效 activation 下。
- review 未通过，candidate 不进入正式 collection。
- commit 只能由当前有效 commit activation 执行。
- duplicate commit 由 permit consumed 和 activation result identity 防止，不靠写作特例。

### 6.4 产物落盘

当前写作任务暴露出一个严重问题：被拒绝产物提前落到正式章节目录。

新规则：

- agent 输出先进入 activation staging 目录。
- 只有 activation result 被 TemporalGate 接纳并通过业务验收后，才从 staging promote 到 official artifact path。
- rejected/stale/duplicate result 只能留在 quarantine 或 rejected staging。

建议目录：

```text
run_root/
  .staging/
    activations/{activation_id}/...
  .quarantine/
    temporal_violations/{violation_id}/...
  official/
    ...
```

写作任务可以把 official 映射到现有章节目录，但写手不能直接污染 official。

### 6.5 业务坐标怎么处理

写作任务仍然需要 volume、chapter、batch、round 等信息，但它们不属于时序本体。

它们应该放在：

- `explicit_inputs`
- memory selector
- artifact target metadata
- output contract
- progress ledger

示例：

```json
{
  "explicit_inputs": {
    "volume_index": 1,
    "chapter_start": 11,
    "chapter_end": 20,
    "outline_ref": "artifact:..."
  },
  "artifact_policy": {
    "official_target": "volume_001/chapters/chapter_011_020/draft_body.md",
    "staging_target": ".staging/activations/{activation_id}/draft_body.md"
  },
  "memory_selector": {
    "repository": "story_outline",
    "collection": "volume_001",
    "record_key": "chapter_011_020",
    "version": "latest_committed_before_activation"
  }
}
```

时序系统只记录：

```json
{
  "position_id": "pos.chapter_writer",
  "activation_id": "act....",
  "permit_id": "permit....",
  "iteration_index": 0,
  "phase_id": "phase.chapter_writing"
}
```

因此，业务坐标可以参与“节点输入是什么”，不能参与“这个节点是否在合法拓扑位置运行”的根判断。

### 6.6 交接包怎么处理

交接包是边的产物，不是节点自己猜。

控制流边和数据边要分开：

- control edge 决定下游 activation 是否可以打开。
- handoff/artifact/memory/revision edge 决定下游 request 收到什么输入包。

节点收到的输入包必须包含来源说明：

```json
{
  "input_packages": [
    {
      "package_type": "artifact_context",
      "edge_id": "edge.outline_to_writer",
      "source_node_id": "chapter_outline",
      "target_node_id": "chapter_writer",
      "source_activation_id": "act.outline...",
      "artifact_refs": ["artifact:..."],
      "usage_instruction": "你必须严格依据这份细纲写正文，不得跳过细纲中的章节目标。"
    },
    {
      "package_type": "memory_snapshot",
      "edge_id": "edge.world_memory_to_writer",
      "repository": "world_canon",
      "collection": "final",
      "selector": {"record_key": "worldview"},
      "usage_instruction": "这些是已经定稿的世界观约束，必须遵守，不得改写为新设定。"
    }
  ]
}
```

如果输入包缺 `usage_instruction`，发布页必须诊断。节点不能靠 agent 自己猜“这个输入是干什么的”。

## 7. 通用编辑器设计

### 7.1 编辑器主线

编辑器应该形成稳定操作主线：

```text
选中对象
  -> 看拓扑结构
  -> 配资源读写
  -> 看节点执行包
  -> 配时序控制
  -> 发布诊断
  -> 运行监测
```

所有页面共享 `TaskGraphEditorFocus`。

当前已有 `TaskGraphEditorFocus`，应扩展为：

```ts
type TaskGraphEditorFocus = {
  layer: "blueprint" | "topology" | "memory" | "responsibility" | "timeline" | "contracts" | "publish" | "monitor";
  facet?: string;
  node_id?: string;
  edge_id?: string;
  repository_id?: string;
  collection_id?: string;
  phase_id?: string;
  position_id?: string;
  activation_id?: string;
  permit_id?: string;
  issue_id?: string;
};
```

### 7.2 拓扑页

拓扑页负责结构地图，不塞复杂表单。

应该显示：

- 执行节点。
- 审核门。
- 人工执行节点。
- 人工放行门。
- 循环框。
- 记忆仓库。
- 产物仓库。
- 监测节点。
- barrier。

边类型明确区分：

- control_flow
- handoff
- memory_read
- memory_write_candidate
- memory_commit
- artifact_context
- revision_request
- monitor_observe

拓扑页时序能力：

- 可切换“结构视图”和“时序覆盖视图”。
- 时序覆盖视图只显示：
  - active positions
  - next legal positions
  - blocked positions
  - violations
- 选中节点时，右侧 inspector 显示：
  - 静态位置：phase、sequence、group、frame。
  - 运行状态：是否有 active activation。
  - 是否有 active permit。
  - 为什么能跑或不能跑。

### 7.3 时序页

时序页只负责拓扑控制，不负责业务内容。

分面建议：

1. 阶段与出口

   配 phase 顺序、入口、出口、阻塞节点、退出条件。

2. 步进与并发

   配 sequence、parallel group、join policy。

3. 循环框

   配 loop frame、入口、判定节点、退出条件、最大迭代。

4. 审核与返修

   配 review gate、pass route、fail route、返修边。

5. 人工执行与人工放行

   配 `execution_actor=human` 的节点、输入展示包、交互表单、输出契约、是否允许 commit；只在确实需要纯控制暂停时配置 `manual_release_gate`。

6. 后台监测

   配 runtime monitor 节点、观察范围、提醒策略。

时序页必须展示编译后的 `TopologyTemporalModel`，不能只展示表单。

### 7.4 记忆页

记忆页负责资源配置。

它不设计时序。

它应该配置：

- repository
- collection
- record schema
- memory_read edge selector
- version selector
- write candidate contract
- commit edge
- visible label
- usage instruction

记忆页应该能回答：

```text
这个节点在一次合法运行中，会通过哪条边读哪个仓库的哪些记录？
这个节点写出的候选数据进入哪个仓库的哪个 collection？
哪个审核或提交节点负责把候选数据变成正式记录？
```

### 7.5 职责页

职责页负责节点最终收到什么。

必须展示：

- agent 身份。
- 任务职责。
- 投影 prompt。
- 输入包来源。
- 记忆快照摘要。
- 产物上下文。
- 返修包。
- 输出契约。

重要原则：

prompt 里不能出现开发说明式语言。

错误示例：

```text
这是 runtime 节点，根据任务图执行 world_review。
```

正确示例：

```text
你是一名世界观审核员。
你只负责评审当前世界观设定是否完整、一致、可支撑后续写作。
你不负责替创作者扩写设定。
你需要指出问题、给出裁决、说明是否允许进入下一阶段。
```

## 8. 后端实施方案

### 8.1 阶段一：清理错误概念和半改动

目标：

- 不让旧错改污染新实现。
- 不把业务 scope、章节计划、result record、receipt 当时序本体。

处理：

- 删除或撤回 `timeline_result_record.py` 这一类把业务结果强行并入时序的结构。
- TaskGraph runtime 中不再新增 `receipt_policy`、`timeline_result_policy` 作为时序核心字段。
- TaskGraph 前端 preflight 中所有 `receipt_policy` 命名改为明确的 `result_acceptance_policy` 或 `result_commit_policy`。
- 健康系统里的 receipt 若仍存在，只属于健康系统动作确认，不允许进入 TaskGraph 时序模型。

完成标准：

- TaskGraph 运行链路中不再出现以 receipt 命名的时序控制字段。
- 若保留健康系统 receipt，代码注释和类型命名必须明确限定为 health action acknowledgement。
- 旧 `task_mode` 不参与 TaskGraph 编译、运行、监测判断。
- 旧测试若断言 task_mode 或 receipt 为 TaskGraph 必需字段，删除或重写为 TaskGraph 通用图测试。

### 8.2 阶段二：新增拓扑时序模型编译器

新增文件：

- `backend/orchestration/runtime_loop/topology_temporal_model.py`
- `backend/orchestration/runtime_loop/topology_temporal_compiler.py`

输入：

- `TaskGraphRuntimeSpec`

输出：

- `TopologyTemporalModel`

编译规则：

1. 先识别/校验 MainChain。
2. 由 MainChain 生成主时序轴和 anchor points。
3. 每个可执行节点生成 position。
4. resource 节点生成 resource position，但不自动执行，除非有资源操作边触发。
5. `phase_id` 决定 phase 容器。
6. `sequence_index` 决定 phase 内步进。
7. `timeline_group_id` 决定同一步的并发组。
8. `loop_policy` 和 `loop_frame` 生成 loop frame，并绑定主链区间。
9. `review_gate` 生成 conditional control edge。
10. `execution_actor=human` 的节点生成 human execution position，并绑定交互 schema、输入展示包和输出契约。
11. `manual_release_gate` 只生成 waiting release position，并绑定主链 anchor。
12. background/runtime_monitor 生成 background lane，并绑定主链 anchor 或 run lifecycle。

完成标准：

- 任意已发布 TaskGraph 都能生成 `TopologyTemporalModel`，或明确诊断失败原因。
- 编译结果必须包含 `main_chain`。
- 每个非主链可执行 position 都必须有 anchor 或 join/release 解释。
- 编译结果不含业务正文和业务批次字段。
- 编译结果能被前端直接渲染为时序控制预览。

### 8.3 阶段三：新增运行时状态和许可结构

新增文件：

- `backend/orchestration/runtime_loop/temporal_runtime_state.py`
- `backend/orchestration/runtime_loop/temporal_gate.py`

新增结构：

- `TemporalRunState`
- `NodeActivation`
- `ExecutionPermit`
- `TemporalViolation`
- `GateDecision`

存储位置：

- LangGraph checkpoint state 中保存 `temporal_run_state`。
- RuntimeClock 事件用于审计，不作为唯一恢复源。

完成标准：

- checkpoint 中存在 `temporal_run_state`。
- active activation 和 active permit 可枚举。
- closed activation 可追溯。
- violation 可枚举。

### 8.4 阶段四：改造调度器

改造：

- `backend/orchestration/runtime_loop/task_graph_scheduler.py`

从：

```text
node_statuses -> ready_nodes
```

改为：

```text
TopologyTemporalModel + TemporalRunState + accepted activation results -> SchedulerDecision
```

输出：

- positions_to_open
- permits_to_issue
- activations_to_close
- blocked_reasons
- terminal_status
- diagnostics

`TaskGraphSchedulerState` 仍可保留，但变成展示投影，不再是运行真相。

完成标准：

- 调度器不直接把 node 标为 running。
- 调度器只输出打开 activation、签发 permit、关闭 activation 的 decision。
- `node_statuses` 由 temporal state 投影生成。
- phase、sequence、parallel、loop、manual、background 都有明确支持或明确发布阻塞诊断。

### 8.5 阶段五：改造 LangGraph runtime

改造：

- `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
- `backend/orchestration/runtime_loop/stage_execution_request.py`

核心变化：

1. bootstrap 时编译 temporal model。
2. 初始化 temporal run state。
3. route_next 调 scheduler，获得 permit。
4. stage_execute 必须持有 permit。
5. request 带上 activation_id 和 permit_id。
6. stage_accept 先走 TemporalGate。
7. 结果不合法时进入 quarantine，不更新正式结果。
8. activation 完成后关闭。
9. permit consumed 后不再接受重复结果。

命名建议：

- 新核心结构使用 `NodeExecutionRequest`。
- 如果一次性替换成本过高，旧 `StageExecutionRequest` 只作为边界兼容名，内部字段必须改为 node/activation/permit 语义。
- 后续清理 stage 命名，避免 stage 与 node 混用继续制造误解。

完成标准：

- bootstrap 不再直接设置首节点 running。
- `_stage_execute` 或后续重命名后的执行函数没有 permit 会失败。
- request 包含 activation_id、permit_id。
- result accept 首先经过 TemporalGate。
- stale、late、duplicate result 不更新正式 node result 投影。

## 13. 不允许的实现方式

- 让 monitor 只看最新更新时间，不看 permit/activation。
- 让 memory edge 自动变成控制流，除非用户显式配置 control dependency。
- 保留 task_mode 旧链路影响 TaskGraph。
- 为了通过测试伪造结果。

## 14. 最小可通过标准

这个系统设计只有满足以下条件才算通过：

1. 每个节点运行前都有 active activation。
2. 每个节点运行前都有 active permit。
3. 每个结果返回时都能校验 permit。
4. 每个 activation 只能接受一次有效结果。
5. 图推进后旧 activation 被关闭。
6. late result 不污染 official artifact、memory commit、downstream activation。
7. 并发 join 可解释。
8. loop iteration 可解释。
9. human execution node 可阻塞下游直到用户提交合法节点结果。
10. manual release gate 可阻塞下游直到用户放行。
11. runtime monitor 能报告 out-of-timeline 行为。
12. 前端能让用户看见拓扑时序控制，而不是只看事件列表。
13. 业务数据留在 contracts、memory selectors、artifact metadata，不进入时序本体。

## 15. 最终目标

TaskGraph 通用编辑器要让用户自由设计任务，而不是让运行时靠猜。

正确的任务图运行应该是：

```text
用户画出拓扑
  -> 编译器生成拓扑时序模型
  -> 调度器按模型打开节点运行窗口
  -> 每个节点凭执行许可运行
  -> 节点收到明确的身份、输入包、prompt、输出契约
  -> 结果凭许可返回并被接纳或隔离
  -> 产物和记忆只在合法接纳后提交
  -> 监测节点持续观察越界行为
  -> 前端能解释每一个节点为什么运行、为什么阻塞、为什么被拒绝
```

这才是时序系统在本项目里的作用。

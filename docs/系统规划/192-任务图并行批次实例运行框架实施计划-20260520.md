# 192-任务图并行批次实例运行框架实施计划

日期：2026-05-20

## 1. 目标

在已经完成的顺序批次生命周期基础上，推进“批次执行实例”框架，让任务图可以表达并行批次的运行语义。

这不是马上启动多个真实 worker，而是先把可实现并行的通用底座补齐：

- 编译期能表达 `child_execution_mode=parallel`。
- 运行态能同时拥有多个 ready batch。
- 每个批次执行拥有独立 execution instance。
- 节点请求携带 `unit_batch_execution_id`，避免同一节点多批次结果混淆。
- monitor 和前端能看到 instance 维度。

## 2. 为什么要先做实例框架

只把多个 batch 标记为 ready 不够。真正可运行的并行任务必须回答：

- 当前结果属于哪个 batch？
- 同一节点同时跑多个 batch 时，request_id / idempotency_key 如何隔离？
- failed batch 是否阻断整个节点？
- merge 等待的是 batch committed，还是 node completed？
- 前端看到的是节点状态，还是批次实例状态？

因此本阶段先建立 `batch_execution_instances`，后续 worker pool 才有明确承载对象。

## 3. 本轮交付

### 3.1 编译到运行态

读取 split plan metadata：

```json
{
  "child_execution_mode": "parallel",
  "max_parallel_batches": 3
}
```

并在 batch runtime state 中保存：

```json
{
  "execution_mode_by_plan": {
    "split:graph:node:unit": "parallel"
  }
}
```

顺序模式保持现有行为：只开放第一批 ready，提交一批后开放下一批。

并行模式行为：

- bootstrap 时同一 plan 下所有 batch 都进入 ready。
- batch 之间不互相等待 commit。
- `max_parallel_batches` 控制同一 plan 下同一节点可同时激活的 batch execution instance；不配置时运行态使用默认上限。
- merge 仍然等待 committed batch。

### 3.2 执行实例模型

新增或扩展运行态字段：

```json
{
  "batch_execution_instances": [
    {
      "execution_id": "batchrun:...",
      "batch_id": "...",
      "plan_id": "...",
      "node_id": "...",
      "status": "running",
      "attempt_index": 1,
      "repair_round": 0,
      "request_id": "",
      "dispatch_event_id": "",
      "request_payload": {},
      "result_ref": ""
    }
  ],
  "active_executions_by_node": {},
  "active_execution_by_batch": {}
}
```

本轮仍然让现有 LangGraph 循环一次只发出一个 request，但 request 已经以 batch execution instance 为边界。后续 worker pool 可以直接消费 ready instances。

### 3.3 请求与结果绑定

`stage_prepare` 在输入契约绑定完成后选择 batch，同步创建或激活 execution instance，并注入：

```json
{
  "unit_batch_execution_id": "...",
  "unit_batch_id": "...",
  "batch_start_index": 1,
  "batch_end_index": 10
}
```

`stage_execute` 把 execution id 写入 dispatch context。

`stage_execute` 同时把完整 node execution request 保存回 `batch_execution_instances[].request_payload`，保证并行结果乱序返回时可以恢复原批次的执行上下文。

`stage_accept` 通过 request id、dispatch event id 或 `unit_batch_execution_id` 定位 execution instance，再推进当前批次，并记录 result_ref。

如果事件携带了批次身份，但运行态找不到对应 execution instance，运行时不会回退到“当前 active batch”，而是记录 `batch_execution_identity_not_found` 并保持活动批次不变。这是并行安全边界，避免错批次提交。

### 3.4 监控与前端

run monitor 返回：

- instance 总数
- running instance
- failed instance
- committed instance
- 当前 active execution id

前端运行页和常驻浮窗继续使用同一个 batch lifecycle parser 展示实例摘要。

节点批次契约编辑器新增：

- `runtime.split_policy.child_execution_mode`
- `runtime.split_policy.max_parallel_batches`

发布前预检新增：

- 批次执行模式只允许 `sequential` / `parallel`。
- 并行上限不能小于 0。
- 并行模式下未配置上限给出提示。
- 并行模式下 `auto_commit_without_review` 升级为阻塞错误，避免多个候选结果绕过 review 直接进入 merge。

## 4. 本轮不做

- 不启动多个真实后台 worker。
- 不让 agent 动态拆分 batch。
- 不改变质量门来源。
- 不写任何写作专用逻辑。
- 不把并行批次解释为领域对象；工作单元仍由 `unit_batch.unit_kind` 声明。

## 5. 完成标准

测试需要证明：

1. parallel split plan bootstrap 后多个 batch 同时 ready。
2. select batch 时创建 execution instance。
3. request explicit inputs 包含 `unit_batch_execution_id`。
4. accepted 后对应 execution instance committed。
5. failed batch 不继续冒充更多 work。
6. monitor / frontend summary 能解析 execution instance 统计。

## 7. 当前边界与下一阶段

当前版本已经让“并行批次实例”成为可编译、可配置、可监控、可安全接收结果的通用运行对象。

但当前 LangGraph 协调循环仍然一次返回一个 `NodeExecutionRequest`。这意味着：运行态可以同时容纳多个 active execution instance，也能正确接收它们的乱序结果；真正自动扇出多个 worker / 子任务请求还需要下一阶段的 dispatcher / worker-pool。

下一阶段建议：

1. 增加 `ready_batch_execution_queue` 或等价调度视图，让调度器一次暴露多个可派发 execution instance。
2. 在 API 层增加批量领取/批量派发入口，而不是让前端或业务逻辑自己循环触发。
3. 为每个 execution instance 建立独立 task run / agent run 关联，形成真正的并行运行、监控和取消能力。
4. merge 节点从 `node_all_batches_committed` 提升为明确的 `merge_execution_instance`，让合并也成为可审计执行单元。

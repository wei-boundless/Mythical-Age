# 运行状态监控统一投影重构计划书

日期：2026-06-05  
状态：待实施  
范围：Agent 会话状态、Runtime Monitor、任务运行摘要、前端运行反馈展示

## 1. 背景

当前用户反馈的核心问题是：Agent 明明已经停下、暂停或等待用户继续，但页面仍在部分位置显示系统正在运转。

这不是单纯的图标、文案或样式问题，而是运行状态解释权分裂导致的系统一致性问题。会话列表、聊天输入区、顶部状态、右侧运行监控可以处在不同层级，但它们必须消费同一个运行事实，不能各自判断任务是否仍在运行。

## 2. 当前代码链路

### 2.1 后端 TaskRun 投影

文件：

- `backend/harness/runtime/run_monitor/projector.py`

职责：

- 从 TaskRun / ActiveTurn / RuntimeRun 中生成 monitor item。
- item 中包含 `status`、`lifecycle`、`bucket`、`action_required`、`terminal`、`stale`、`is_live` 等字段。

问题：

- 该层只负责生成运行事实原料，但还没有输出统一的用户可见运行状态。
- `project_active_turn()` 会在 active turn waiting 时仍保留 `is_live=True`。这本身可以成立，因为它表示记录仍来自活跃 turn，但不能被 UI 解释为任务仍在运行。

### 2.2 后端 Runtime Monitor Signal 投影

文件：

- `backend/harness/runtime/run_monitor/signals.py`

职责：

- 将 monitor item 转为 `RuntimeMonitorSignal`。
- 计算 `signal.state`、`primary`、`attention`、`summary.active`、`summary.waiting`。

问题：

- 这里有独立的状态判断逻辑。
- 一旦这里和 session summary 或前端状态判断不一致，右侧运行监控就会和聊天区、会话列表分裂。

### 2.3 后端 Session active_task 摘要

文件：

- `backend/harness/runtime/run_monitor/service.py`
- `backend/api/session_summary.py`

职责：

- `get_session_task_summary()` 为 session 注入 `active_task`。
- `api.session_summary.enrich_session_summaries()` 将该摘要挂到 session 列表响应。

问题：

- `get_session_task_summary()` 目前用 `bucket` 和 `action_required` 选择当前任务。
- 这仍然是另一套运行态判断，不应该独立存在。

### 2.4 前端 Session 状态展示

文件：

- `frontend/src/lib/sessionTaskPresentation.ts`
- `frontend/src/components/layout/WorkbenchShell.tsx`
- `frontend/src/components/chat/ChatPanel.tsx`

职责：

- 会话列表运行点、顶部状态、聊天输入按钮、继续/停止控制。

问题：

- `sessionTaskPresentation.ts` 自己维护 `WAITING_TASK_STATUSES`、`RUNNING_TASK_STATUSES` 等状态表。
- 这是前端重复解释后端运行态。
- `activeStreamSessionIds` 表示前端流式连接状态，不能决定任务是否仍在运行。

### 2.5 前端 Runtime Monitor 展示

文件：

- `frontend/src/components/layout/RunMonitorPanel.tsx`
- `frontend/src/components/layout/RunActivityLane.tsx`
- `frontend/src/components/layout/RunProjectLane.tsx`
- `frontend/src/lib/run-monitor/selectors.ts`

职责：

- 右侧运行监控面板和活动列表。

问题：

- 它直接消费 `runMonitor.summary.active` 和 `signal.state`。
- 如果后端 signal 投影和 session 投影不统一，用户会看到同一任务在不同区域呈现不同状态。

### 2.6 后端 Runtime Monitor 动作权限

文件：

- `backend/harness/runtime/run_monitor/management.py`
- `backend/harness/runtime/run_monitor/actions.py`

职责：

- 为右侧运行监控生成打开、详情、清出、暂停、继续、停止、删除等动作。
- 接收前端动作并调用 pause / resume / stop / delete 执行链路。

问题：

- `_actions_for_signal()` 当前仍用 `state` 和 `source.is_live` 判断 active。
- 如果它不消费统一 activity / control capability，即使展示层已经不转，菜单仍可能错误启用停止、继续或清出。
- `actions.py` 必须保留执行前硬校验，不能只相信前端按钮状态。

### 2.7 Health / Maintenance 监控边界

文件：

- `backend/health_system/task_record_maintenance.py`
- `backend/health_system/governance.py`

职责：

- 后台健康治理、任务记录保护、风险报告。

问题：

- health 系统也维护 active / waiting / resumable 判断。
- 它可以不和用户界面处在同一层，但如果继续独立解释运行态，就会在健康报告和主监控之间制造第二套事实。

## 3. 真实问题定义

当前系统缺少一个单一权威的 Runtime Activity 投影。

需要统一的不是 UI 层级，而是运行状态事实：

```text
TaskRun / ActiveTurn / RuntimeRun 原始状态
-> RuntimeActivity 统一投影
-> Session Summary / Runtime Monitor / Live Monitor
-> 前端各层 UI 展示
```

前端不应该再从 `bucket`、`status`、`lifecycle`、`is_live`、`activeStreamSessionIds` 中独立推断运行状态。

## 4. 目标架构

新增后端唯一状态投影模块：

- `backend/harness/runtime/run_monitor/activity.py`

输出必须分为两层，不能混在一起：

- `display_activity`：用户看见的运行事实。
- `control_capability`：当前是否允许继续、暂停、停止等控制动作。

统一契约：

```python
{
    "activity_state": "running | waiting | paused | stopped | failed | completed | stale | idle",
    "activity_label": "...",
    "is_running": bool,
    "is_waiting": bool,
    "is_resumable": bool,
    "is_interruptible": bool,
    "control_reason": "...",
    "tone": "active | neutral | attention | done",
}
```

固定优先级：

```text
failed / stopped
-> waiting / paused / action_required
-> stale
-> completed
-> running
-> idle
```

关键规则：

- `waiting_executor` 必须压过 `bucket=running`。
- `paused` 必须压过 `is_live=true`。
- `action_required` 必须压过残留 stream。
- `user_aborted`、`stopped`、`cancelled` 必须压过前端 active stream。
- `activeStreamSessionIds` 只能表示前端正在接收文本流，不能定义任务运行态。
- `activity_state=waiting` 不等于 `is_resumable=True`。
- `is_resumable` 只能来自控制平面的明确资格：真实 active turn 绑定的非 terminal task，或运行时明确标记的 executor recovery / resumable checkpoint。
- `status=aborted` 且 `terminal_reason=user_aborted` 必须归入 `stopped`，不能归入 failed。
- `stale` 只适用于非 terminal 运行记录；明确 completed / stopped / failed 的 terminal 结果不能被残留 stale 覆盖。

### 4.1 与 239 / 242 续接计划的冲突裁决

本计划不把 `waiting_executor` 状态本身解释成自然语言可续接。

裁决规则：

- `waiting_executor` 可以让页面显示“等待继续 / 已暂停”。
- 只有 active work 控制平面明确提供 `active_work_context` 或 `resumable_work_context` 时，才允许 `is_resumable=True`。
- 已释放 active turn 且没有 executor recovery 资格的旧 `waiting_executor`，只作为等待事实展示，不允许被普通自然语言恢复为 same-run resume。
- terminal 历史任务不生成可控制的 active work；用户后续“继续”进入新 UserTurn，由模型决定是否创建新的 TaskRun。

这样可以兼容 239 的任务边界原则，也允许 242 中“明确可恢复 checkpoint”进入上下文，但禁止仅凭 `status=waiting_executor` 打开控制入口。

### 4.2 `activity_state` 与 `signal.state` 的关系

`activity_state` 是新权威字段。

`signal.state` 可以在迁移期保留为右侧监控 lane 字段，但它只能由 `activity_state` 映射，不再拥有独立判断权：

```text
activity_state=running -> signal.state=active
activity_state=waiting / paused -> signal.state=waiting
activity_state=stopped -> signal.state=completed
activity_state=failed -> signal.state=failed
activity_state=completed -> signal.state=completed
activity_state=stale -> signal.state=stale
```

前端新代码优先消费 `activity_state`、`is_running`、`is_resumable`、`is_interruptible`；`signal.state` 只用于兼容 lane 分组和旧组件过渡。

## 5. 权威边界

| 层级 | 当前问题 | 目标职责 |
| --- | --- | --- |
| `projector.py` | 生成原始 item，但无统一 activity | 继续生成事实原料，并附加统一 activity |
| `activity.py` | 不存在 | 唯一运行状态投影权威 |
| `signals.py` | 自己判断 `signal.state` | 只消费 `activity_state` |
| `service.py` | 自己用 bucket 选择 active_task | 用 `activity_state` 选择当前任务 |
| `management.py` | 自己用 state / is_live 判断动作权限 | 用 control capability 生成动作 |
| `actions.py` | 执行动作时可能依赖外层按钮状态 | 执行前重新校验 control capability |
| 前端 session helper | 自己维护状态表 | 只消费后端 activity 字段，负责展示格式 |
| Chat / Workbench / RunMonitor UI | 多处重复判断运行态 | 只显示统一 activity，不再二次解释 |
| health / maintenance | 独立统计 active / waiting / resumable | 明确接入 activity，或声明为后台治理指标并禁止驱动用户可见运行态 |

## 6. 实施计划

### Phase 1：建立后端 RuntimeActivity 投影

修改：

- 新增 `backend/harness/runtime/run_monitor/activity.py`
- 新增 `RuntimeActivity` TypedDict 或 dataclass。
- 新增 `project_runtime_activity(item: dict[str, Any], *, control_context: RuntimeActivityControlContext | None = None) -> RuntimeActivity`。
- 新增 `RuntimeActivityControlContext`，输入 active turn、executor recovery、latest task eligibility、action service 校验结果等控制平面事实。

完成标准：

- activity 投影覆盖 running、waiting、paused、stopped、failed、completed、stale。
- 明确区分 `is_running` 和 `is_resumable`。
- `waiting_executor` 在没有 control context 时只显示 waiting，不自动变成 resumable。
- `aborted + terminal_reason=user_aborted` 显示 stopped，不显示 failed。

禁止事项：

- 禁止在前端或 signal 层继续新增状态表。
- 禁止用旧 `bucket` 判断作为新的兼容兜底。

### Phase 2：接入 Runtime Monitor Signal

修改：

- `backend/harness/runtime/run_monitor/signals.py`

动作：

- `project_monitor_signal()` 调用 `project_runtime_activity()`。
- `signal.state` 由 `activity_state` 映射。
- `summary.active` 只统计 `is_running=True`。
- `summary.waiting` 只统计 `activity_state in {"waiting", "paused"}`。
- 删除 `_signal_state()` 独立判断，或降级为私有映射包装且不再拥有决策权。
- `management.py` 中 `_lane_for_signal()` 和 `_actions_for_signal()` 只消费 activity / control capability。
- `actions.py` 执行 pause / resume / stop 前重新读取最新 activity / control capability，拒绝过期按钮请求。

完成标准：

- 右侧运行监控不再因为 `is_live=True` 或 `bucket=running` 显示错误运行。
- 停止、暂停、继续、清出、删除按钮和展示状态一致。

### Phase 3：接入 Session active_task Summary

修改：

- `backend/harness/runtime/run_monitor/service.py`
- `backend/api/session_summary.py`

动作：

- `get_session_task_summary()` 选择任务时使用 `activity_state`。
- `active_task` 响应增加：
  - `activity_state`
  - `activity_label`
  - `is_running`
  - `is_waiting`
  - `is_resumable`
  - `is_interruptible`
  - `control_reason`

完成标准：

- session 列表、当前会话、任务环境会话都拿到相同 activity。
- `active_task.selection` 可以选择最新 waiting / stopped / completed 任务用于展示，但不能把 selection 当作 running 依据。

### Phase 4：前端删除重复运行态解释

修改：

- `frontend/src/lib/api.ts`
- `frontend/src/lib/sessionTaskPresentation.ts`
- `frontend/src/components/layout/WorkbenchShell.tsx`
- `frontend/src/components/chat/ChatPanel.tsx`
- `frontend/src/components/layout/RunMonitorPanel.tsx`
- `frontend/src/components/layout/RunActivityLane.tsx`
- `frontend/src/components/layout/RunProjectLane.tsx`

动作：

- API 类型增加 activity 字段。
- `sessionTaskPresentation.ts` 不再维护状态优先级，只做展示格式化。
- `ChatPanel.tsx` 中 `currentSessionStreaming` 拆成：
  - `currentSessionReceivingStream`
  - `currentTaskIsRunning`
- 停止按钮只绑定真正的 streaming / running 场景。
- 继续按钮只绑定 `is_resumable=True`。
- 会话列表运行点只使用 `is_running`。
- 右侧监控只使用 `signal.state` 和后端 summary，不再额外解释。
- `activeStreamSessionIds` 只保留为 `currentSessionReceivingStream`，用于文本流、输入编辑锁和输出过程反馈，不能参与运行点、停止态、任务是否仍在运行的判断。
- `RuntimeMonitorSignal` 和 `SessionTaskSummary` 类型补齐 activity / control 字段。

完成标准：

- 前端没有多个运行态判断源。
- 同一任务在所有 UI 区域展示一致。
- completed/stopped/failed 任务即使还有流式尾包，也不显示为 running。

### Phase 4.5：Health / Maintenance 边界处理

修改：

- `backend/health_system/task_record_maintenance.py`
- `backend/health_system/governance.py`

动作：

- 后台保护、风险统计若继续使用 active / waiting / resumable，必须优先读取 RuntimeActivity。
- 如果部分治理指标需要保留独立统计，字段名必须明确为 health-specific，例如 `maintenance_protected`、`governance_risk_waiting`，不能命名成用户可见 running/resumable 权威。
- health 报告不得向 UI 或普通会话暴露“可恢复旧任务”的控制暗示。

完成标准：

- 用户可见运行态只来自 `activity.py`。
- health 可以报告风险，但不能推翻 activity / control capability。

## 7. 验收场景

### 场景 1：任务等待继续

输入状态：

```json
{
  "status": "waiting_executor",
  "lifecycle": "paused",
  "bucket": "running",
  "is_live": true,
  "action_required": true
}
```

期望：

- 会话列表无运行点。
- 顶部状态显示等待继续或已暂停。
- 聊天输入区显示继续按钮。
- 右侧监控 `active=0`，`waiting=1`。
- 页面不出现持续运转视觉。

### 场景 2：用户停止任务

输入状态：

```json
{
  "status": "aborted",
  "terminal_reason": "user_aborted"
}
```

期望：

- 所有 UI 显示已停止。
- 不出现红色失败强调。
- 不出现运行点、转圈、微光。
- 不显示停止按钮。

### 场景 3：真实运行任务

输入状态：

```json
{
  "status": "running",
  "lifecycle": "running",
  "bucket": "running",
  "is_live": true
}
```

期望：

- 会话列表显示轻微运行点。
- 顶部状态可微亮。
- 右侧监控 `active=1`。
- 聊天输入区允许停止当前运行。

## 8. 清理规则

本次重构完成后必须清理：

- 前端重复状态表。
- 后端 `_signal_state()` 的独立决策权。
- 后端 `_actions_for_signal()` 中基于 `state/is_live` 的独立动作权限判断。
- 任何直接用 `bucket == "running"` 判定用户可见运行态的代码。
- 任何直接用 `activeStreamSessionIds` 判定任务仍在运行的代码。
- 前端测试和 mock 中复制的旧 `signalState()` / running 优先级表。

允许保留：

- `bucket/lifecycle/status` 作为底层事实字段。
- `activeStreamSessionIds` 作为前端流接收状态。
- UI 层基于统一 activity 的样式映射。

## 9. 风险控制

风险 1：最终回答仍在流式输出，但任务已经 completed。  
处理：将 stream receiving 和 task running 分开显示，不用 stream 覆盖 activity。

风险 2：图任务和单 Agent 任务状态语义不同。  
处理：统一投影函数接收原始 item，不侵入图任务调度逻辑，只统一用户可见 activity。

风险 3：旧测试保护旧内部结构。  
处理：更新测试断言到用户可见行为和统一契约，不保留旧状态优先级测试。

风险 4：activity 与 control capability 混用。  
处理：展示状态和控制权限在类型、字段、测试中分开校验；waiting 不自动代表可以继续。

风险 5：health 治理继续输出另一套 running / resumable 语义。  
处理：health 只做后台风险和维护判断，用户可见状态必须消费 activity；若必须保留独立指标，改成 health-specific 命名。

## 10. 最终标准

本次重构完成后，系统中只有一个地方可以决定任务是否“正在运行”：

```text
backend/harness/runtime/run_monitor/activity.py
```

其他所有模块只能消费它的结果，不能重新推断。

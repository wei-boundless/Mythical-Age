## 二、第二阶段的真实问题

第二阶段的核心问题不是：

`runtime loop 还有几个字段不够优雅`

而是：

`任务系统之后缺少正式编排层，导致 runtime 仍在直接消费任务兼容包`

当前主链路是真实存在的：

```text
QueryRuntime
  -> AgentRuntimeChainAssembler.build_runtime()
  -> build_task_runtime_contract()
  -> build_task_execution_assembly_bundle()
  -> build_legacy_task_runtime_compat_surface()
  -> TaskRunLoop / StageProjectionCycle / model_adoption
```

这条链说明现在系统虽然已经补出了 `TaskExecutionAssembly`，但主执行链并没有在它之后进入“正式编排层”，而是继续走了一层混合兼容包。

这会带来四个结构性问题：

1. `TaskExecutionAssembly` 还不是 runtime 前的唯一正式入口  
2. 编排系统缺少正式 owner，对 prompt、memory scope、resource policy、runtime lane 的装配仍是碎片化的  
3. `StageProjectionCycle` 仍从 `task_operation` 里拿 `soul_runtime_view / prompt_manifest / projection_requirement`，说明灵魂系统与编排系统之间还没有单独合同  
4. runtime 虽然已经开始只执行 adopted policy 和 directive，但它启动前拿到的仍是“任务兼容包”而不是“正式运行体对象”

因此，第二阶段必须解决的不是局部补丁，而是：

`把 TaskExecutionAssembly 之后的正式编排层补出来，并让 runtime 切到只消费 AgentRuntimeSpec / TaskBodyOrchestration`

## 四、第二阶段的本地硬约束

本阶段必须严格遵守以下约束。

## 4.1 任务系统到此为止

任务系统在第二阶段不再继续承担：

- projection 素材读取
- prompt section 结构装配
- soul runtime view 生成
- runtime prompt manifest 生成

任务系统在第二阶段之后只允许稳定输出：

- `TaskExecutionAssembly`
- 任务侧 registry / bindings / request profiles

## 4.2 编排系统必须成为 TaskExecutionAssembly 之后的唯一 owner

编排系统必须正式承担：

- 单 Agent 身体底座
- 单次任务态身体编排
- prompt 结构装配
- memory scope 绑定
- resource policy candidate 汇总输入
- runtime lane 绑定
- output boundary 绑定
- stage projection runtime-safe snapshot
- `AgentRuntimeSpec` 生成

## 4.3 runtime 只能执行，不再装配

runtime 在第二阶段完成后只允许：

- 消费 `AgentRuntimeSpec`
- 消费 adopted `ResourcePolicy`
- 消费 `RuntimeDirective`
- 负责 execute / checkpoint / resume / observability / finalization

runtime 不再允许：

- 自己决定 prompt 结构
- 自己决定 projection 结构
- 自己决定 memory scope
- 自己从 compat 包里挑字段拼身体

## 4.4 compat 只能短暂存在于 cutover 适配层

如果第二阶段中间需要短暂并行：

- compat 只能存在于编排系统适配层
- compat 不得重新回灌到任务系统
- compat 不得继续作为主 API 稳定语义

最终目标不是“长期双轨”，而是：

`切完 runtime 后物理删除 compat surface`

## 五、先进架构对第二阶段的直接启发

这里只保留与本阶段直接相关的约束，不做泛泛综述。

## 5.1 LangGraph：compile 层与 runtime 层分离

参考：

- LangGraph runtime / Pregel  
  https://docs.langchain.com/oss/python/langgraph/pregel

可直接吸收的原则是：

- 高层定义不等于 runtime 本体
- compile / assembly 是独立阶段
- runtime 负责执行，不反向决定高层结构

映射到本项目应当变成：

```text
TaskExecutionAssembly
  -> OrchestrationAssembly
  -> AgentRuntimeSpec
  -> TaskRunLoop
```

而不是：

```text
TaskExecutionAssembly + compat surface
  -> TaskRunLoop
```

## 5.2 AutoGen Core：runtime environment 负责 identity、lifecycle、boundary

参考：

- Agent Runtime Environments  
  https://microsoft.github.io/autogen/0.5.5/user-guide/core-user-guide/core-concepts/architecture.html

可直接吸收的原则是：

- runtime 是环境层
- runtime 负责通信、身份、生命周期、边界
- runtime 不是 prompt assembler + task planner + policy builder 的总和

这进一步说明：

`本项目的编排系统不能被 runtime 环境层吞掉`

## 5.3 Microsoft Agent Framework：workflow 与 orchestration engine 分离

参考：

- Workflow Orchestrations  
  https://learn.microsoft.com/en-us/agent-framework/workflows/orchestrations/

可直接吸收的原则是：

- workflow 是任务方法，不是运行体装配
- orchestration engine 可以负责 checkpoint / recovery / execution pattern
- 上层任务定义与下层执行环境必须分开

映射到本项目就是：

- workflow 继续归任务系统
- 单 Agent 身体编排归编排系统
- checkpoint / resume / execution trace 归 runtime

## 5.4 Kubernetes：control plane 与 node execution 分离

参考：

- Communication between Nodes and the Control Plane  
  https://kubernetes.io/docs/concepts/architecture/control-plane-node-communication/

可直接吸收的原则是：

- control plane 决定目标
- 执行端负责把目标跑起来
- 执行端不反过来决定 control plane 目标

映射到本项目就是：

- 任务系统是 task control plane
- 编排系统是 single-agent body assembler
- runtime 是 execution engine

因此第二阶段的正确方向不是“把 runtime 做得更聪明”，而是：

`把上游正式目标对象做完整，然后让 runtime 更笨、更稳定`

## 六、第二阶段目标设计

第二阶段完成后，主链路应变成：

```text
Task System
  -> TaskExecutionAssembly

Soul System
  -> ProjectionMaterialBundle / PromptIdentityBundle

Memory System
  -> MemoryRuntimeView + MemoryContextPackage + governed writeback policy

Operation System
  -> OperationRequirement + CapabilitySupplyPackage + ResourcePolicyCandidate

Orchestration System
  -> AgentBodyProfile
  -> TaskBodyOrchestration
  -> AgentRuntimeSpec
  -> StageProjectionSnapshot

Runtime
  -> RuntimeDirective + adopted ResourcePolicy
  -> TaskRunLoop execute/checkpoint/resume/finalize
```

## 6.1 第二阶段必须补出的正式对象

建议统一补在 `backend/orchestration` 下。

### 6.1.1 AgentBodyProfile

表示 Agent 的稳定身体底座。

它应至少包含：

- `body_profile_id`
- `agent_id`
- `default_prompt_structure_profile_id`
- `default_memory_scope_profile_id`
- `default_runtime_lane_profile_id`
- `default_output_boundary_profile_id`
- `default_operation_policy_mode`
- `default_projection_policy`
- `metadata`

它回答：

`这个 Agent 平时天生长什么样`

### 6.1.2 PromptStructureProfile

表示 prompt 的正式结构骨架。

它应至少包含：

- `profile_id`
- `section_order`
- `required_section_kinds`
- `optional_section_kinds`
- `stage_projection_policy`
- `model_visible_rules`

它回答：

`这类 Agent 的 prompt 从哪些 section 组成、顺序如何`

### 6.1.3 MemoryScopeProfile

表示记忆进入运行体的正式范围。

它应至少包含：

- `profile_id`
- `allowed_memory_layers`
- `read_scope`
- `writeback_policy`
- `token_budget_policy`
- `restore_policy`

它回答：

`这次运行体允许哪些记忆以什么方式进入`

### 6.1.4 RuntimeLaneProfile

表示 runtime lane 的正式约束。

它应至少包含：

- `profile_id`
- `lane_id`
- `execution_style`
- `tool_followup_policy`
- `checkpoint_policy`
- `resume_policy`

它回答：

`这次运行是怎样的一条执行车道`

### 6.1.5 OutputBoundaryProfile

表示最终输出边界。

它应至少包含：

- `profile_id`
- `allowed_output_contracts`
- `citation_policy`
- `artifact_commit_policy`
- `finalization_policy`

它回答：

`这次结果允许以什么形式成为正式输出`

### 6.1.6 TaskBodyOrchestration

表示任务态身体编排。

它应至少包含：

- `orchestration_id`
- `task_id`
- `agent_id`
- `task_execution_assembly_ref`
- `body_profile_ref`
- `prompt_structure_profile_ref`
- `memory_scope_profile_ref`
- `runtime_lane_profile_ref`
- `output_boundary_profile_ref`
- `stage_plan`
- `resource_binding_plan`
- `verification_gate_plan`
- `fallback_plan`

它回答：

`这个 Agent 这次为了承接这个任务，身体内部怎么变体`

### 6.1.7 AgentRuntimeSpec

表示 runtime 唯一正式输入对象。

它应至少包含：

- `runtime_spec_id`
- `task_id`
- `session_id`
- `agent_id`
- `task_execution_assembly_ref`
- `task_body_orchestration_ref`
- `context_input_refs`
- `projection_snapshot_ref`

- `input_contract_ref`
- `output_contract_ref`

它回答：

`runtime 现在到底该执行什么`

## 七、第二阶段固定执行顺序

第二阶段不允许边改边漂。执行顺序必须锁死。

## 7.1 阶段 A：补出编排系统正式对象层

目标：

- 可以单独从编排 builder 生成 `AgentRuntimeSpec`
- 新对象有回归测试，但 runtime 主链尚未切换

## 7.2 阶段 B：把主入口切到正式编排 builder

目标：

- 让 runtime 前的主入口不再以 compat 包为核心

本阶段要做：

1. 改造 `backend/runtime/agent_chain.py`
2. 让它先调用 task builder，再调用 orchestration builder
3. 新增一个统一的 orchestration build result
4. `build_task_runtime_contract` 降为 compat preview / debug 用

本阶段禁止：

- 任务系统重新长回 prompt 装配逻辑
- 让 `AgentRuntimeChainAssembler` 继续把 compat surface 当第一公民

完成标准：

- 主链上已经能拿到 `TaskExecutionAssembly + TaskBodyOrchestration + AgentRuntimeSpec`
- compat surface 不再是 runtime 主依赖

## 7.3 阶段 C：runtime 消费面正式切换

目标：

- 让 `TaskRunLoop` 真正只消费正式运行体对象

本阶段要做：

1. 改造 `TaskRunLoop.run_single_agent_stream()`
2. 改造 `StageProjectionCycle`
3. 改造 `RuntimeContextManager`
4. 改造 `model_adoption`
5. 让 trace / checkpoint / projection snapshot 全部以 `AgentRuntimeSpec` 为主 ref

本阶段禁止：

- runtime 再去解析任务 compat 字段
- runtime 直接从 task_operation 大字典里拼 prompt

完成标准：

- runtime 已经不依赖 `task_prompt_contract`
- runtime 已经不依赖 `soul_runtime_view` compat surface
- runtime 已经不依赖 `prompt_manifest` compat source dict

## 7.4 阶段 D：删除 compat 壳并收口 API / tests

目标：

- 清理第二阶段过渡残留

本阶段要做：

1. 删除 `backend/soul/task_runtime_compat.py`
2. 下线 `build_task_runtime_contract` 作为主入口
3. 改写 `/tasks/runtime-contract` 为新 preview 接口，或直接改名为 orchestration preview
4. 清理依赖 compat surface 的回归测试
5. 让 `tasks.__init__` 不再导出 `build_task_runtime_contract`

完成标准：

- compat surface 不再参与主链
- compat 只剩迁移期必要的历史测试时，直接删除
- 代码里不再存在“任务系统拼 runtime 壳”的正式入口

## 九、迁移与 cutover 规则

第二阶段必须明确 cutover 规则，不能边跑边猜。

## 9.1 允许的短暂并行期

允许短暂并行的只有两类：

1. 新编排 builder 与旧 compat builder 并行产出
2. 新 runtime spec preview 与旧 runtime contract preview 并行暴露

并行期的唯一目的：

`验证新对象是否完整`

不是为了长期双轨。

## 9.2 正式 cutover 点

以下条件同时满足后，允许切 runtime 主消费面：

1. `AgentRuntimeChainAssembler` 已主产出 `AgentRuntimeSpec`
2. `TaskRunLoop` 已可在不读取 compat 字段的情况下完成单轮执行
3. `StageProjectionCycle` 已从正式编排对象生成 snapshot
4. 对 development / search / local_read / health 四类代表任务回归通过

## 9.3 rollback 规则

第二阶段允许存在短暂 rollback 开关，但必须满足两条：

1. rollback 只能退到“上一阶段可运行版本”
2. rollback 开关只允许存在于阶段 C 到阶段 D 之间

阶段 D 完成后：

`rollback 开关与 compat surface 一起删除`

## 十、本阶段禁止事项

第二阶段明确禁止以下做法：

1. 继续给 `build_task_runtime_contract()` 塞新字段
2. 在 `TaskRunLoop` 里就地新增更多 if/else 兼容分支
3. 把 `StageProjectionCycle` 继续当作 soul compat 清洗器扩展
4. 用“先能跑起来”理由长期保留双主链
5. 只迁移 API，不迁移主执行链
6. 只迁移类名，不迁移真实 owner
7. 把多 Agent orchestration 提前混进本阶段

本阶段只做：

`单 Agent 编排系统正式建模 + runtime 主消费面切换`

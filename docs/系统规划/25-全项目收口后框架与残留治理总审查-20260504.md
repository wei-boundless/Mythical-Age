## 二、总框架已经对齐的部分

### 2.1 六大系统边界已经基本立住

从当前源码结构看，项目已经不再是单一会话壳，而是围绕六大系统组织：

- 任务系统：`backend/tasks`
- 编排系统：`backend/orchestration`
- 记忆系统：`backend/memory_system`
- 健康系统：`backend/health_system`
- 能力系统：`backend/capability_system`
- 灵魂系统：`backend/soul`

同时：

- `sessions`
- `understanding`
- `query`
- `execution`

这些目录也已经更清楚地退回到支撑层或入口适配层，而不是再冒充“主系统”。

这和总设计书中：

`先修系统边界，再修局部行为`

这一原则是对齐的。

### 2.2 平台已经从“单 Agent 思维”转向“任务驱动的 Agent 平台”

当前源码中已经形成稳定对象链：

- 任务侧负责任务识别、任务登记、workflow / projection / binding
- 编排侧负责 Agent body、runtime spec、resource / output / lane / memory scope
- 灵魂侧负责 prompts 身份层与姿态层
- 能力侧负责 tools / skills / operations / workers / mcp
- 记忆侧负责 request / bundle / writeback / governance
- 健康侧负责 workbench、verification、issue / run 观测

这说明项目已经不再把所有结构都压在“主 Agent 会话逻辑”里。

### 2.3 目录治理已经明显好于重构前

目前主要系统代码和管理数据目录已经基本按系统归拢，典型改善包括：

- 灵魂系统实现与静态身份素材已经收敛在 `backend/soul`
- 能力系统正式实现收敛在 `backend/capability_system`
- 编排运行循环收敛在 `backend/orchestration/runtime_loop`
- 健康维护链路已经收敛在 `backend/health_system/maintenance`

这意味着后续再做维护时，已经可以按系统定位，而不是继续全局搜补丁。

## 四、当前仍保留但合理的 `legacy/compat` 残留

不是所有 `legacy/compat` 都应该删掉。

结合当前源码，本项目仍有三类应暂时保留的兼容桥。

### 4.1 数据迁移桥

例如：

- `backend/memory_system/legacy_types.py`
- `backend/structured_memory/memory_manager.py` 中的旧 note/layout 修复
- `backend/orchestration/agent_registry.py` 中对旧 agent payload 的迁移
- `backend/tasks/flow_registry.py` 中对旧 assignments payload 的读取

这些逻辑承担的是：

`历史数据向新正式对象模型的收束`

只要项目里还存在旧数据、旧记录、旧产物，这些桥就有保留价值。

### 4.2 协议兼容桥

例如：

- `mcp-compatible.v1`
- `a2a-compatible.v1`
- health/test API 中少量 `compat_api` source 标记

这些不是系统边界错误，而是：

`协议层显式声明当前视图仍兼容某种外部格式或旧调用语义`

这种兼容位只要仍被前端、工具侧或外部契约消费，就不应简单删除。

### 4.3 双轨运行或迁移观测桥

例如：

- `backend/retrieval/service.py` 中的 `legacy_only / shadow_read / v2_primary`
- `backend/document_conversion/docling_converter.py` 中的 `legacy_fallback`
- `backend/maintenance/legacy_cleanup.py`

这类残留承担的是：

- 双轨对照
- 回退容错
- 迁移期清理工具

如果现在直接全部删除，会削弱系统对旧链路与新链路之间的可观察性和退场控制能力。

### 4.4 编排采用层与执行层仍不应错误并入能力系统

虽然工具实现本体现在已经进入能力系统目录，但下面两类文件仍不应被一起并进去：

- 编排采用层：
  - [backend/orchestration/resource_policy_builder.py](/d:/AI应用/langchain-agent/backend/orchestration/resource_policy_builder.py:1)
  - [backend/orchestration/resource_gate.py](/d:/AI应用/langchain-agent/backend/orchestration/resource_gate.py:1)
  - [backend/orchestration/resource_runtime_view.py](/d:/AI应用/langchain-agent/backend/orchestration/resource_runtime_view.py:1)
- 执行层：
  - [backend/execution/tool_executor.py](/d:/AI应用/langchain-agent/backend/execution/tool_executor.py:1)

原因是这些文件回答的不是：

`平台上有哪些工具`

而是：

- 这次运行允许采用哪些工具
- 这些工具如何在 runtime 中被门禁和投影
- 这些工具最终怎样被执行

因此本轮目录收敛的正确结论是：

- 工具实现本体归能力系统
- 工具运行采用归编排系统
- 工具真实执行归执行层

## 五、当前仍存在的结构性差距

### 5.1 文档体系已经接近收口，但还缺“总索引页”

目前 `docs/系统规划` 已经覆盖：

- 总设计
- 各系统详细设计
- 各阶段蓝图
- 各阶段源码对照审查
- 各阶段原则对照审查

但还没有一份：

`最终阅读入口 / 文档导航总表`

这会导致新读者还是要靠文件名猜顺序。

这不影响源码正确性，但会影响后续维护效率。

### 5.2 部分系统内部仍存在“builder 偏厚”问题

例如：

- 灵魂系统中的 `projection_builder.py` / `runtime_assembly.py`
- 编排系统中的部分 assembly builder
- 记忆系统中的 facade 转发层

这些问题当前已经不是边界错误，而是：

`内部对象链还可以继续纯化`

所以它们应被视为“后续瘦身项”，而不是当前阻塞项。

### 5.3 少量命名仍带迁移味道，但尚未形成实质错误

例如：

- `workflow_compatible_or_task_default`
- `compat_assignment`
- `issues_compat_api`
- `agent_runs_compat_api`

这类命名说明某些链路仍保留旧视图语义。

它们不一定马上该删，但后续如果正式契约已经稳定，建议逐步改成更中性的正式命名，减少“永久迁移期”的错觉。

## 七、总审查结论

截至本轮收口，项目已经基本从：

`边界混合、旧壳并存、目录漂移的过渡工程`

转向：

`六大系统边界基本稳定、目录分层基本清楚、残留治理开始可控的 Agent 平台工程`

当前最重要的结论有三条：

1. 现在剩余问题主要是内部继续纯化和总文档导航问题，不再是大规模系统边界错误。
2. “删残留”这件事已经进入精细治理阶段，不能再把所有 `legacy/compat` 一刀切；要区分无意义空壳和仍有迁移价值的桥。
3. 本项目已经可以视为完成了“六大系统正式收口”，后续工作重心应从“分系统重构”转向“平台级治理、文档总览、迁移桥逐步退役”。

这意味着当前项目可以进入：

`全项目平台化稳定期`

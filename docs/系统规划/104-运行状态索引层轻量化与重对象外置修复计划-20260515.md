# 104-运行状态索引层轻量化与重对象外置优化计划

## 0. 背景

本计划用于优化 `storage/runtime_state/state_index.json` 所在的运行状态索引层。

本轮目标不是单纯“把 JSON 变小”，而是借由体积异常暴露出的问题，重新明确运行状态系统的分层边界，让索引层、checkpoint 层、event log 层、runtime object 层各自承担清晰职责。

当前实测结果：

- `state_index.json` 约 69 MB。
- `coordination_runs` 约 18.4 MB。
- `task_runs` 约 13.1 MB。
- 单个小说写作团队根 `task_run.diagnostics` 约 600 KB。
- 单个小说写作团队 `coordination_run.diagnostics` 约 1.27 MB。

这不是单次运行脏数据，而是索引层职责失焦导致的结构性问题。

`state_index` 原本应当是运行对象的轻量查找表，用于根据 `task_run_id`、`session_id`、`coordination_run_id` 等键快速找到当前状态和引用。但现在它同时保存了：

- TaskGraph 定义快照。
- TaskGraph runtime spec。
- Agent dispatch plan。
- LangGraph runtime state。
- Coordination graph spec。
- Scheduler state。
- Coordination flow。

这些对象已经由 event log、checkpoint、coordination checkpoint 或图注册系统承载，却又被重复写入 `state_index`，导致监控接口、查询接口、停止接口、恢复接口都被迫读取一个越来越大的总表。

## 1. 结构性判断

本问题满足结构性问题特征：

1. 膨胀字段来自稳定写入路径，不是偶发日志。
2. 同一类重对象被重复存放在多个运行对象的 `diagnostics` 中。
3. `state_index` 的读路径已被多个模块共享，不能靠手工删当前文件解决。
4. 实时监控性能已经受到影响，之前 session live monitor 请求可达到 30 秒以上。
5. 当前新增任务继续运行时，文件会继续增长。

因此优化目标不是压缩 JSON，而是恢复状态系统分层，并提升协调任务在实时监控、恢复、停止、继续执行、历史诊断中的稳定性。

## 1.1 优化目标

本方案必须同时完成以下优化：

1. 查询优化
   - 高频读路径只读轻索引和必要 checkpoint 摘要。
   - live monitor 不再解析完整运行对象。
   - session task list、stop、continue-current-stage 不再被胖 diagnostics 拖慢。

2. 写入优化
   - `state_index` 原子写入的单次 payload 明显变小。
   - 新任务每推进一个节点，不再把图定义、runtime spec、LangGraph state 反复写入总表。
   - 写入层统一兜底净化，避免调用方遗漏造成胖字段回流。

3. 职责优化
   - `state_index` 只做轻索引。
   - `coordination_checkpoints` 负责实时协调状态。
   - `runtime_objects` 负责可按 ref 找回的重对象。
   - `event_log` 负责审计追踪。
   - artifact 文件系统负责真实产物。

4. 恢复优化
   - resume / continue 不依赖某个胖 diagnostics 是否还在。
   - 必需重对象通过稳定 ref 读取。
   - ref 缺失时给出明确错误，不走隐式旧链路。

5. 监控优化
   - 前端协调监控读到的是小而稳定的运行视图。
   - 当前节点、已完成节点、失败节点、产物引用、工作记忆操作数都能快速返回。
   - full trace 与 live monitor 分离，避免诊断能力拖慢实时 UI。

6. 演进优化
   - 后续新增任务图、子 Agent、工作记忆资源节点时，不需要继续扩张 `state_index`。
   - 每类重对象都有明确归属，便于后续归档、清理和迁移。

## 2. 当前职责审计

### 2.1 写入方

`state_index` 当前主要写入方：

- `TaskRunLoop`
  - 写入 `TaskRun`、`AgentRun`、`AgentRunResult`。
  - 写入协调任务的 `CoordinationRun`、`CoordinationNodeRun`、handoff、merge result。
- `CoordinationTraceAdapter`
  - 将 LangGraph 状态投影为 coordination flow、node run、merge result。
- `AgentDelegationExecutor`
  - 写入 delegation request/result、子 agent run。
- `DelegationGraphAdapter`
  - 写入委派型协调图对象。
- `api/orchestration.py`
  - 停止任务时更新 `TaskRun` / `CoordinationRun`。
- `health_system/registry.py`
  - 健康系统桥接时更新任务运行状态。

### 2.2 读取方

`state_index` 当前主要读取方：

- `trace_reader`
  - 查询 task run trace。
  - 查询 live monitor。
  - 查询 session task runs。
- `TaskRunLoop`
  - 恢复当前 task run。
  - 获取 coordination run。
  - 查找 agent runs、worker spawn results、node runs。
- `LangGraphCoordinationRuntime`
  - 根据 `coordination_run_id` 获取 coordination run。
- `artifact_refs.py`
  - 扫描 task runs 和 coordination runs 来收集产物引用。
- `api/orchestration.py`
  - 停止、恢复、继续当前 stage 时读取 task run / coordination run。
- `health_system/registry.py`
  - 健康任务结束桥接。

结论：不是每个节点执行都直接依赖 `state_index`，但控制面和监控面强依赖它。它必须保留，但必须变轻。

## 3. 问题字段定位

### 3.1 `TaskRun.diagnostics` 胖字段

当前单个根 task run 中主要胖字段：

- `task_graph_runtime_spec`：约 291 KB。
- `task_graph_definition`：约 264 KB。
- `agent_dispatch_plan`：约 43 KB。

主要写入路径：

- `backend/orchestration/runtime_loop/task_run_loop.py`
  - `task_graph_definition = graph.to_dict()`
  - `task_graph_runtime_spec = runtime_spec.to_dict()`
  - `agent_dispatch_plan = dispatch_plan.to_dict()`

这些不应常驻 `state_index.task_runs[*].diagnostics`。

### 3.2 `CoordinationRun.diagnostics` 胖字段

当前单个 coordination run 中主要胖字段：

- `langgraph_runtime_state`：约 655 KB。
- `task_graph_runtime_spec`：约 291 KB。
- `task_graph_definition`：约 264 KB。
- `task_graph_scheduler_state`：约 39 KB。
- `coordination_flow`：约 16 KB。

主要写入路径：

- `backend/orchestration/runtime_loop/coordination_trace_adapter.py`
  - 将 `langgraph_runtime_state` 写入 `CoordinationRun.diagnostics`。
- `backend/orchestration/runtime_loop/task_run_loop.py`
  - 将 `coordination_graph_spec` / `langgraph_diagnostics` 写回 coordination run。

这些对象应改为外置引用或摘要。

## 4. 优化后的目标架构

### 4.1 `state_index` 定位

`state_index` 只保存轻量运行索引。它的优化方向不是“少存一点 diagnostics”，而是从结构上禁止它承担重对象仓库职责。

允许保存：

- 运行对象主键。
- session / task / coordination / node 的反向索引。
- status、terminal_reason、created_at、updated_at。
- latest checkpoint ref。
- latest merge result ref。
- graph_ref、task_ref、agent_id、profile_id。
- artifact refs 摘要。
- 少量不可替代的恢复字段。

禁止保存：

- 完整 TaskGraph definition。
- 完整 TaskGraph runtime spec。
- 完整 LangGraph runtime state。
- 完整 AgentDispatchPlan。
- 完整 context snapshot。
- 完整 model messages。
- 大段产物文本。
- 大段 prompt manifest。

### 4.2 重对象归属

重对象归属如下：

- TaskGraph definition
  - 主来源：任务图主数据注册系统。
  - 运行快照：`runtime_objects/task_graph_snapshots/{run_id}.json`。
  - `state_index` 只保存 `task_graph_definition_ref`。

- TaskGraph runtime spec
  - 主来源：编译器可重建。
  - 运行快照：`runtime_objects/task_graph_runtime_specs/{run_id}.json`。
  - `state_index` 只保存 `task_graph_runtime_spec_ref` 和 `valid`。

- AgentDispatchPlan
  - 主来源：编译事件和运行对象文件。
  - 运行快照：`runtime_objects/dispatch_plans/{coordination_run_id}.json`。
  - `state_index` 只保存 `agent_dispatch_plan_ref`。

- LangGraph runtime state
  - 主来源：`coordination_checkpoints/{coordination_run_id}.json`。
  - `state_index` 不保存完整 state。
  - `CoordinationRun.diagnostics` 只保存摘要字段。

- Coordination flow
  - 前端监控需要摘要。
  - `state_index` 可保存轻量 flow summary。
  - 完整 flow 可由 checkpoint 或 event log 重建。

### 4.3 静态状态与动态状态分层

当前压缩方案已经移出了重对象，但还不够。真正的优化边界应该是按状态性质分层，而不是只按体积大小分层。

状态分为四类：

1. 静态定义状态
   - 例：TaskGraph definition、runtime spec、contract manifest、topology template、communication protocol、agent/profile/projection 配置。
   - 特征：一次发布后很少变化，同一个 run 内不应反复写入。
   - 归属：主数据注册表或 `runtime_objects/static_snapshots/`。
   - 读取方式：按 graph id、version、manifest ref 读取；高频监控不读取全文。

2. 动态运行状态
   - 例：active stage、ready/running/completed/failed nodes、terminal status、stage execution request、scheduler state、human gate。
   - 特征：运行过程中持续变化，需要实时显示和恢复。
   - 归属：`coordination_checkpoints` 和 runtime checkpoint。
   - 读取方式：live monitor 只读动态摘要；恢复/continue 读取 checkpoint 中的必要字段。

3. 审计事件状态
   - 例：task_run_started、coordination_stage_updated、task_result_ready、task_run_stopped、commit_gate_checked。
   - 特征：不可变事件流，用于追责、回放和诊断。
   - 归属：event log。
   - 读取方式：full trace 或诊断页面按需读取；不能参与前端实时轮询。

4. 产物与记忆状态
   - 例：`artifact:` 文件引用、working memory refs、memory operation summary。
   - 特征：下游真实交接依赖，但本体通常是文件或记忆库条目。
   - 归属：artifact 文件系统、working memory store。
   - 读取方式：任务交接只传 ref 和摘要；需要正文时按 artifact ref 读取。

`state_index` 在这个分层里只承担“目录状态”：

- `session_id -> task_run_ids`
- `task_run_id -> task summary`
- `task_run_id -> coordination_run_ids`
- `coordination_run_id -> node_run_ids`
- `coordination_run_id -> latest checkpoint ref`
- `coordination_run_id -> latest artifact refs summary`
- `coordination_run_id -> latest static snapshot refs`

它不应该保存静态定义全文，也不应该保存动态状态全文。

### 4.4 按需读取策略

不同场景使用不同读取路径：

1. 前端实时监控
   - 读取：`state_index` 轻目录 + `coordination_checkpoints` 动态摘要。
   - 不读取：完整 graph definition、完整 runtime spec、完整 event log、完整产物正文。
   - 目标：高频轮询稳定小于 1 秒。

2. 任务继续执行
   - 读取：当前 coordination checkpoint、stage execution request、必要 static snapshot ref、上游 artifact refs。
   - 不读取：session 全量历史、完整 trace。
   - 目标：继续执行只加载当前 stage 必需上下文。

3. 任务恢复
   - 读取：runtime checkpoint、coordination checkpoint、static snapshot ref。
   - 不读取：旧胖 diagnostics。
   - 目标：恢复链路由显式 ref 驱动。

4. 前端图编辑器
   - 读取：主数据中的 TaskGraph definition/runtime spec。
   - 不读取：历史 run 的 state index 版本。
   - 目标：编辑的是当前配置，不是运行快照。

5. 诊断 Trace
   - 读取：event log + runtime_objects + checkpoints。
   - 不读取：高频 monitor API。
   - 目标：慢查询可以接受，但必须和实时监控隔离。

6. 工作记忆与产物交接
   - 读取：working memory refs、artifact refs、必要 artifact 摘要。
   - 不读取：完整任务图或完整运行事件。
   - 目标：下游节点只拿职责所需信息。

### 4.5 进一步优化方向

当前已完成的“重对象外置”只是第一阶段。下一阶段应继续做：

1. 将 `state_index.json` 拆成分片索引。
   - `sessions.json`
   - `task_runs.json`
   - `coordination_runs.json`
   - `coordination_node_runs.json`
   - `agent_runs.json`
   - 目的：避免任何查询都读取一个 28 MB 总表。

2. 为实时监控建立专用 read model。
   - 文件：`runtime_views/session_live/{session_id}.json`
   - 内容：当前任务、当前协调 run、节点计数、当前节点、产物摘要。
   - 写入：由状态变更时更新，而不是每次请求现算。

3. 静态 snapshot 版本化。
   - static snapshot ref 应包含 graph id、version/hash。
   - 同一版本 graph 只存一份。
   - 多个 run 共享同一份静态 snapshot。

4. 动态 checkpoint 增量化。
   - 当前 checkpoint 仍可能较大。
   - 后续可把 stage_results、artifact refs、working memory operations 按阶段分片。

5. 明确 API 分级。
   - `/live-monitor`：小视图。
   - `/trace`：诊断视图。
   - `/runtime-objects/{ref}`：按 ref 展开。
   - `/task-graphs/{id}`：静态配置视图。

这意味着最终目标不是一个更小的 `state_index.json`，而是一套分层读取模型。

## 5. 修复原则

1. 优化优先，不做单点补丁。
2. 不保留旧胖索引作为兼容路径。
3. 不让实时监控读取完整历史 trace。
4. 不把 checkpoint 内容镜像回 `state_index`。
5. 不把图定义和 runtime spec 复制进每个 run。
6. 恢复路径需要能从 ref 找回必要对象。
7. 运行中任务不中断，先修新写入，再做历史清理。
8. 所有新增路径必须能解释自己的职责边界，不能形成新的混合层。

## 6. 修复影响评估

### 6.1 对运行中任务的影响

影响等级：中。

原因：

- 正在运行的协调任务会继续向 `state_index` 写入 `TaskRun`、`CoordinationRun`、node run、merge result。
- 如果在任务运行中直接替换写入结构，可能出现同一 run 前半段是胖 diagnostics、后半段是轻 diagnostics。
- `continue-current-stage`、停止任务、live monitor 都会读取当前 run 的索引对象。

处理策略：

- 第一阶段只新增轻索引写入兜底和重对象外置，不删除 checkpoint/event。
- 对活跃 run 保留必要 ref，不依赖旧胖字段。
- 历史压缩放在新写入路径验证通过之后执行。
- 当前小说任务若仍在运行，应先观察到当前节点完成，再重启服务加载结构性修复。

### 6.2 对 TaskRunLoop 的影响

影响等级：高。

受影响点：

- TaskGraph 启动时不再把完整 `task_graph_definition` 和 `task_graph_runtime_spec` 写进 `TaskRun.diagnostics`。
- 任务终止同步时不能通过 `dict(root_task_run.diagnostics)` 把旧胖字段继续继承下去。
- coordination task contract 生成 dispatch plan 时，不能把完整 `agent_dispatch_plan` 回写进 `TaskRun` 和 `CoordinationRun`。

处理策略：

- `TaskRunLoop` 写入前先通过 `RuntimeObjectStore` 写入重对象。
- `TaskRunLoop` 只在 diagnostics 中保存 ref、id、valid、summary。
- 对所有 `upsert_task_run` 调用增加索引层净化兜底，避免遗漏调用点。

### 6.3 对 LangGraphCoordinationRuntime 的影响

影响等级：高。

受影响点：

- `_resolve_task_graph_definition()` 当前读取 `coordination_run.diagnostics.task_graph_definition`。
- runtime spec 当前可能从 `coordination_run.diagnostics.task_graph_runtime_spec` 读取。
- working memory policy、graph id、contract manifest 等字段当前部分来自 `coordination_graph_spec`。

处理策略：

- 新增从 `task_graph_definition_ref` / `task_graph_runtime_spec_ref` 读取重对象的路径。
- 若 ref 不存在，使用任务图注册表和 compiler 重建。
- 不新增从胖 diagnostics 回退读取的长期兼容路径。
- 对 `coordination_graph_spec` 的依赖改成读取 checkpoint diagnostics 或 runtime spec ref。

### 6.4 对 CoordinationTraceAdapter 的影响

影响等级：中高。

受影响点：

- 当前 adapter 会把 `langgraph_runtime_state` 镜像进 `CoordinationRun.diagnostics`。
- live monitor 过去可以从 `CoordinationRun.diagnostics` 拿到完整状态。

处理策略：

- 完整实时状态只保存在 `coordination_checkpoints`。
- `CoordinationRun.diagnostics` 只保留轻量摘要。
- `trace_reader` 从 checkpoint 取实时状态，从 `state_index` 取对象目录。

### 6.5 对实时监控和前端的影响

影响等级：中。

受影响点：

- 前端协调监控依赖 live monitor 返回 `coordination_flow`、`langgraph_runtime_state`、`coordination_graph_spec` 摘要。
- 如果返回结构变化过大，会导致面板空白或节点状态不刷新。

处理策略：

- API 返回字段名保持稳定。
- 字段内容从完整对象改为 summary。
- 前端不再依赖完整 diagnostics。
- full trace 和 live monitor 分离，前端高频刷新只走 live monitor。

预期收益：

- live monitor 不再被 69 MB `state_index` 拖慢。
- 当前 30 秒级请求应稳定降到 1 秒以内。

### 6.6 对历史 trace / 调试能力的影响

影响等级：中。

受影响点：

- 历史 run 的完整图定义和 runtime spec 不再常驻 `state_index`。
- 调试时不能再直接打开 `state_index.json` 看完整对象。

处理策略：

- 重对象移动到 `runtime_objects/` 后保留 ref。
- full trace 诊断接口允许按 ref 加载重对象。
- event log、runtime checkpoint、coordination checkpoint 不删除。
- 历史压缩脚本生成报告，记录每类字段迁移数量和目标 ref。

### 6.7 对 artifact refs / 工作记忆的影响

影响等级：中。

受影响点：

- `artifact_refs.py` 会扫描 `task_runs` 和 `coordination_runs` 收集产物引用。
- 工作记忆读写依赖正式 `artifact:` refs。
- 如果压缩 diagnostics 时误删 artifact summary，可能影响下游节点拿上游产物。

处理策略：

- 产物引用不进重对象清理黑名单。
- `TaskRun` / `CoordinationRun` 保留 `artifact_refs` 摘要或对应 output refs。
- 工作记忆仍以 `artifact:` 文件引用为主，不读取胖 diagnostics。
- 压缩脚本必须验证当前小说产物 refs 仍可被收集。

### 6.8 对停止 / 恢复 / 继续当前阶段接口的影响

影响等级：高。

受影响点：

- stop API 读取 task run / coordination run 并回写终止状态。
- `continue-current-stage` 需要从 checkpoint 获取 `stage_execution_request`。
- 如果 coordination run 只剩轻 diagnostics，恢复接口不能再依赖胖 state。

处理策略：

- stop API 只依赖轻量运行对象和 checkpoint ref。
- continue API 从 `coordination_checkpoints` 读取当前 stage request。
- 恢复失败时给出明确缺失 ref，而不是静默回退胖 diagnostics。

### 6.9 对文件体积和性能的影响

影响等级：正向高。

预期变化：

- `state_index.json` 从几十 MB 降到可控体积。
- 新运行不再每个根 task 写入数百 KB 到数 MB。
- `state_index` 原子写入成本明显下降。
- live monitor、stop、session task list 查询延迟下降。

需要关注：

- `runtime_objects/` 会新增文件数量。
- 重对象读取从单文件 JSON 变成按 ref 读文件，需要保证路径解析和原子写入可靠。

### 6.10 对静态 / 动态读取模型的影响

影响等级：正向高，但需要继续推进。

当前已解决：

- 胖静态对象不再直接常驻 `state_index`。
- 胖动态对象不再直接常驻 `CoordinationRun.diagnostics`。
- live monitor 已能从轻索引 + checkpoint 组合状态。

仍然偏重的部分：

- `state_index` 仍是单文件总表，虽然变小但还不是理想的按需索引。
- live monitor 仍会读取整个轻索引 snapshot，而不是只读 session 级分片。
- 静态 snapshot 当前按 run 写入，后续应按 graph version/hash 去重。
- checkpoint 内部仍承载较多 stage_results 和 artifact refs，后续需要动态状态分片。

后续优化应继续围绕“按需读取”推进：

- session 监控只读 session 分片。
- task stop 只读 task run 分片。
- coordination resume 只读 coordination checkpoint 和 static refs。
- trace 诊断才展开 event log 和 runtime objects。

### 6.11 对系统长期演进的影响

影响等级：正向高。

优化收益：

- 后续新增任务图不会把完整 graph spec 塞进每个 run。
- 工作记忆资源节点可以只依赖 artifact refs 和 memory refs，不需要读胖 diagnostics。
- 多 Agent 协调监控可以自然扩展为按 summary、checkpoint、artifact 三层读取。
- 历史归档可以按对象类型分层处理，而不是只能移动一个巨大的 `state_index.json`。
- 调试入口可以更精确地按 ref 展开，不需要每次加载完整总表。

需要约束：

- 任何新增运行对象都必须先判断是静态定义状态、动态运行状态、审计事件状态还是产物记忆状态。
- 新增 diagnostics 字段必须经过大小和归属检查。
- 不允许把“为了前端方便展示”的大对象重新放回 `state_index`。

### 6.12 对测试资产和旧数据的影响

影响等级：中。

受影响点：

- 现有测试可能断言 diagnostics 中存在完整图定义。
- 历史 `state_index.json` 仍包含胖字段，直到压缩脚本执行。

处理策略：

- 更新测试断言为 ref + 可读回对象。
- 新增禁止字段断言，防止胖字段回流。
- 历史压缩脚本在单独测试目录先跑，不直接对生产态文件做不可逆修改。

### Step 6：调整 trace reader / live monitor

实时监控规则：

- session live monitor 只读一次 `state_index` 快照。
- 当前协调状态从 `coordination_checkpoints` 读取。
- 只返回前端需要的 summary。
- 不返回完整 checkpoint state。
- 不返回完整 diagnostics。

已临时完成的优化：

- `RuntimeStateIndex.read_snapshot()` 已加入。
- `trace_reader.get_session_live_monitor()` 已改为单次 snapshot 读。

这个优化属于过渡措施，不是最终架构边界。

过渡期含义：

- `read_snapshot()` 只是避免一次 monitor 请求反复读取 69 MB 总表。
- 它不代表 live monitor 应该长期读取胖 `state_index`。
- 它不允许继续把 checkpoint、graph spec、dispatch plan 等重对象留在 `state_index` 中。
- 后续轻索引化完成后，`read_snapshot()` 只能返回瘦身后的轻索引 snapshot。

最终收口要求：

- `read_snapshot()` 保留为轻索引批量读 API，供实时监控在一个请求内获得一致视图。
- `read_snapshot()` 不负责读取 `runtime_objects` 中的重对象。
- live monitor 需要重状态时，必须通过 `coordination_checkpoints` 或明确的 object ref 单独读取摘要。
- full trace 查询可以读取重对象，但必须走诊断接口，不参与前端高频轮询。
- 如果后续发现 `read_snapshot()` 被业务执行路径滥用，应拆出更窄的查询 API，例如 `read_session_monitor_snapshot(session_id)`。

后续要补齐：

- `get_task_run_trace()` 的完整 trace 查询可以慢，但要和 live monitor 分离。
- 前端实时面板只调 live monitor，不调 full trace。
- 给 `read_snapshot()` 增加测试，确保 compact 后返回的 snapshot 不含禁止字段。

验收：

- 69 MB 历史索引下，live monitor 响应小于 1 秒。
- 新任务运行 10 个节点后仍小于 1 秒。
- `read_snapshot()` 返回的 `task_runs[*].diagnostics` 不含 `task_graph_definition`、`task_graph_runtime_spec`、`agent_dispatch_plan`。
- `read_snapshot()` 返回的 `coordination_runs[*].diagnostics` 不含 `langgraph_runtime_state`、完整 `coordination_graph_spec`、完整 `task_graph_runtime_spec`。

## 9. 风险与处理

### 9.0 临时监控优化变成新旧链路

风险：`read_snapshot()` 已经上线到 live monitor，如果后续只做重对象外置，却不约束 `read_snapshot()` 的返回内容，会形成“轻索引计划 + 胖 snapshot API”并存。

处理：

- 将 `read_snapshot()` 纳入正式设计，而不是视作临时补丁。
- 在 `RuntimeStateIndex` 层统一 compact 写入，保证任何 snapshot 都天然是轻量对象。
- 在测试中直接断言 snapshot 不含禁止重字段。
- 不新增任何从 `state_index` 胖 diagnostics 读取重对象的兼容路径。

### 9.1 恢复路径找不到图定义

风险：删除 `task_graph_definition` 后，旧恢复路径找不到完整图。

处理：

- 先实现 `task_graph_definition_ref` 读取。
- 若 ref 不存在，再从任务图注册表读取。
- 若注册表不存在，明确报错 `missing_task_graph_definition_ref`，不走旧胖字段兜底。

### 9.2 前端 trace 仍请求完整对象

风险：live monitor 修好了，但某些前端页面仍调 full trace。

处理：

- 前端协调监控只使用 live monitor。
- full trace 保留为诊断入口，不参与高频轮询。

### 9.3 历史状态迁移误删

风险：压缩脚本误删必要字段。

处理：

- 压缩前生成 `.bak`。
- 压缩脚本只移动重对象，不删除 event/checkpoint。
- 对当前活跃 run 单独验证后再清理全量历史。

## 10. 完成标准

本修复完成的标准：

1. 新写入的 `state_index` 不再保存完整 TaskGraph definition/runtime spec/LangGraph state。
2. `state_index` 只承担轻索引职责。
3. `RuntimeObjectStore` 承担重对象持久化职责。
4. LangGraph resume 和 stage continue 不依赖胖 diagnostics。
5. 协调监控响应稳定小于 1 秒。
6. 小说写作团队任务继续能真实产出文件。
7. 历史 `state_index.json` 完成压缩，并保留可审计报告。
8. 新增代码路径能够证明 `state_index`、checkpoint、event log、runtime object、artifact 的职责没有再次混合。
9. 后续新增任务图时，`state_index` 增量随节点数量线性小幅增长，不随图定义大小或模型上下文大小成倍增长。

进一步优化完成标准：

1. 静态状态、动态状态、审计状态、产物记忆状态有明确代码归属。
2. live monitor 不再读取全量 `state_index.json`，改读 session 级 read model 或分片索引。
3. 同一版本 TaskGraph 的 static snapshot 可被多个 run 复用。
4. stop / resume / continue API 都只读取自身场景必需的数据。
5. full trace 与实时监控完全解耦。

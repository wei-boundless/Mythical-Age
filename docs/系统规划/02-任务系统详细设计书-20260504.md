# 任务系统详细设计书

## 一、文档定位

本文件用于定义本项目中任务系统的正式职责、对象模型、前后端管理结构、任务消费主链路以及与 Agent 平台其他系统的接口。

本文件重点回答以下问题：

- 任务系统到底管什么，不管什么
- 通用任务与特定任务到底如何区分
- 任务如何采用现有 Agent，如何生产工作子 Agent
- workflow、projection、flow contract 在任务系统里分别处于什么位置
- 任务系统如何向记忆系统、编排系统、runtime 发出正式装配请求
- 前端任务系统页面应该如何组织，才能真正做到“可配置、可管理、可执行”

## 二、任务系统总定义

任务系统是平台的调度中枢和任务中枢。

它不是一个“聊天页”，也不是一个“任务展示板”，更不是若干 workflow 文本的堆放区。

它的本体职责是：

- 识别任务
- 登记任务
- 分类任务
- 管理任务契约
- 分配任务
- 采用 Agent
- 必要时生产工作子 Agent
- 为多 Agent 协调任务分配拓扑
- 为多 Agent 协调任务分配角色
- 为多 Agent 协调任务定义通信协议
- 为多 Agent 协调任务定义 handoff 规则
- 绑定 workflow
- 绑定 projection
- 绑定 flow contract
- 形成正式的运行前装配请求

任务系统回答的问题是：

`这是什么任务，为什么这样定义，交给谁执行，需要哪些任务侧配置`

## 三、任务系统的硬边界

### 3.1 任务系统不负责 Agent 内部运行结构

任务系统决定：

- 交给哪个 Agent
- 带什么任务配置

但它不决定：

- Agent 内部 prompts 结构怎么拼
- 工具权限怎么收口
- 资源访问边界怎么设
- output boundary 怎么实现

这些属于编排系统。

### 3.2 任务系统不负责记忆检索内容本体

任务系统只能声明记忆需求，不能直接拼具体记忆内容。

也就是说任务系统可以说：

- 这个任务需要项目背景
- 这个任务需要当前会话摘要
- 这个任务需要该 Agent 的任务记忆

但任务系统不能越过记忆系统自己从存储里乱取内容。

### 3.3 任务系统不负责灵魂定义

任务系统可以绑定 projection，但 projection 的素材来源、人格结构和 prompts 身份层归灵魂系统。

任务系统只负责在任务侧选择和挂载。

### 3.4 任务系统不负责能力资源供给

skills、tools、执行能力、资源引用归能力系统。

任务系统只需要知道：

- 这个任务需要什么能力类目
- 对能力有什么边界要求

## 四、任务系统在平台中的核心地位

平台是 Agent 平台，但任务系统是 Agent 平台真正动起来的调度核心。

没有任务系统：

- Agent 只是静态接口
- workflow 只是文档
- projection 只是设定卡片
- flow contract 只是配置资源

只有当任务系统把这些东西组织起来时，平台才真正具备“承接任务、完成任务”的能力。

所以任务系统不是附属页面，而是平台的任务调度中枢。

## 五、任务系统的对象模型

当前代码中已经存在一些雏形对象，例如：

- `GeneralTaskProfile`
- `TaskAssignment`
- `TaskFlowDefinition`
- `TaskAgentBinding`

这些方向是对的，但还需要进一步分层。

我建议任务系统后续统一围绕八类对象组织。

考虑到平台后续天然支持多 Agent 协调，这里还需要额外补出协调对象。

### 5.1 GeneralTaskProfile

表示通用任务入口配置。

它回答：

`普通用户对话默认怎样进入平台主链路`

建议字段：

- `profile_id`
- `title`
- `entry_channel`
- `default_agent_id`
- `default_workflow_id`
- `default_projection_id`
- `input_contract_id`
- `output_contract_id`
- `conversation_entry_policy`
- `enabled`

说明：

- 通用任务本质上不是给用户“选”的，而是平台默认承接模式
- 它的存在意义主要是和特定任务区分开

### 5.2 SpecificTaskRecord

表示特定任务的正式登记对象。

它回答：

`这一类任务是什么，为什么要独立登记`

建议字段：

- `task_id`
- `task_title`
- `task_family`
- `task_mode`
- `description`
- `enabled`
- `input_contract_id`
- `output_contract_id`
- `acceptance_profile_id`
- `default_flow_contract_id`
- `default_workflow_id`
- `default_projection_policy`
- `task_policy`

### 5.3 TaskWorkflowRecord

表示任务方法对象。

它回答：

`这个任务按什么方法做`

建议字段：

- `workflow_id`
- `title`
- `task_mode`
- `steps`
- `visible_skill_ids`
- `stop_conditions`
- `required_evidence_refs`
- `output_contract_id`
- `prompt`
- `enabled`

说明：

- workflow 是任务方法
- workflow 不等于状态机
- workflow 不应独立游离于任务管理之外，但仍应作为任务系统内的正式子资源

### 5.4 TaskProjectionBinding

表示任务侧的 projection 绑定策略。

它回答：

`这个任务可使用哪些 projection，默认怎么选`

建议字段：

- `binding_id`
- `task_id`
- `projection_selection_mode`
- `allowed_projection_ids`
- `default_projection_id`
- `projection_required`
- `notes`

说明：

- projection 是任务执行姿态补充
- 它不应该再同时被看作 workflow 和任务本体

### 5.5 TaskFlowContractBinding

表示任务侧的状态机契约绑定。

它回答：

`这个任务在 runtime 中按什么状态机执行`

建议字段：

- `binding_id`
- `task_id`
- `flow_contract_id`
- `override_policy`
- `verification_gate_profile`
- `fallback_policy`

### 5.6 TaskAgentAdoptionPlan

表示任务如何采用现有 Agent 或生产工作子 Agent。

它回答：

`这个任务到底由谁执行，以及执行主体是复用还是新建`

建议字段：

- `plan_id`
- `task_id`
- `adoption_mode`
- `default_agent_id`
- `allowed_agent_categories`
- `allow_worker_agent_spawn`
- `worker_agent_blueprint_id`
- `worker_agent_naming_rule`
- `notes`

### 5.7 TaskMemoryRequestProfile

表示任务对记忆系统的正式需求配置。

它回答：

`这个任务执行前应向记忆系统申请哪些记忆层`

建议字段：

- `profile_id`
- `task_id`
- `requested_memory_layers`
- `requested_topics`
- `memory_priority`
- `writeback_policy`
- `allow_long_term_memory`

### 5.8 TaskExecutionAssembly

表示任务系统最终交给平台其他系统的装配请求。

它回答：

`这次执行，任务系统最终决定了什么`

建议字段：

- `assembly_id`
- `task_id`
- `task_kind`
- `selected_agent_id`
- `workflow_id`
- `projection_id`
- `flow_contract_id`
- `memory_request_profile_id`
- `input_contract_id`
- `output_contract_id`
- `task_constraints`
- `safety_policy`

### 5.9 CoordinationTaskRecord

表示一个需要多个 Agent 协作完成的协调任务登记对象。

它回答：

`这个任务是不是协调任务，由哪些 Agent 共同完成`

建议字段：

- `coordination_task_id`
- `title`
- `description`
- `coordinator_agent_id`
- `participant_agent_ids`
- `coordination_mode`
- `topology_template_id`
- `communication_protocol_id`
- `shared_context_policy`
- `memory_sharing_policy`
- `enabled`

### 5.10 CoordinationTopologyTemplate

表示多 Agent 协调任务的拓扑结构模板。

它回答：

`多个 Agent 在这个任务中怎么组织`

建议字段：

- `template_id`
- `title`
- `nodes`
- `edges`
- `entry_rules`
- `join_rules`
- `failure_policy`
- `terminal_policy`

说明：

- 这里的拓扑是任务拓扑，不是单个 Agent 内部结构
- 因此它属于任务系统，而不属于编排系统

### 5.11 TaskCommunicationProtocol

表示多 Agent 协调任务中的通信协议对象。

它回答：

`参与任务的 Agent 之间传什么、怎么传、什么时候传`

建议字段：

- `protocol_id`
- `title`
- `message_types`
- `payload_contracts`
- `signal_rules`
- `handoff_rules`
- `ack_policy`
- `timeout_policy`
- `error_signal_policy`

说明：

- 这是任务级通信协议，不是底层 transport
- 它约束的是任务协调语义，而不是 socket、HTTP 或 MCP 实现细节

## 六、通用任务与特定任务的正式划分

### 6.1 通用任务

通用任务是平台的默认承接模式。

它用于：

- 主会话入口
- 日常用户请求
- 尚未命中特定任务登记项的普通目标

通用任务的特点：

- 默认由主 Agent 承接
- 默认不要求用户手动选择
- 允许模型承担一定理解与任务识别
- 允许在后续命中特定任务后自动切换链路

### 6.2 特定任务

特定任务是平台中预先登记、需要稳定消费和稳定执行的任务对象。

它用于：

- 明确的系统管理任务
- 明确的开发任务
- 明确的分析任务
- 未来需要稳定交付的专项任务

特定任务的特点：

- 必须先登记
- 必须有任务契约
- 必须有 workflow
- 可以有 projection 策略
- 必须有 Agent 采用方案
- 必须可生成稳定的任务执行装配

### 6.3 主 Agent 与任务能力的关系

主 Agent 默认是平台中通用承接能力最强的 Agent。

但这不意味着：

- 所有特定任务都永远由主 Agent 做
- 主 Agent 应默认吃掉所有系统管理任务

更准确的定义是：

- 主 Agent 默认可承接通用任务
- 主 Agent 可作为多数特定任务的兜底执行者
- 当某类任务需要稳定专业消费时，应允许任务系统将其分配给系统管理 Agent 或工作子 Agent

### 6.4 多 Agent 协调任务

除了通用任务与特定任务，任务系统还必须为未来的多 Agent 协调任务预留正式结构。

这里需要特别说明：

- 多 Agent 协调任务不是编排系统的变种
- 多 Agent 协调任务也不是 workflow 的简单扩写

它本质上是任务系统对“多个 Agent 如何共同完成一项任务”的组织能力。

因此，多 Agent 协调任务至少应由任务系统负责：

- 选择 coordinator
- 选择 participants
- 分配拓扑
- 规定通信协议
- 规定 handoff 规则
- 规定结果汇总规则

而编排系统只负责被选中的每个 Agent 自己内部怎么运行。

## 七、任务与 Agent 的关系

### 7.1 Agent 是执行接口，任务是执行请求

任务系统不能把 Agent 和任务混成一类对象。

正确关系是：

- Agent 提供执行接口
- 任务提供执行请求
- 任务通过挂载把 workflow / projection / flow contract 装到 Agent 上

### 7.2 任务绑定 Agent，而不是 Agent 绑定任务

这是一个很重要的方向约束。

因为：

- Agent 是平台长期对象
- 任务是动态请求对象

所以应该是：

`任务决定采用哪个 Agent`

而不是：

`Agent 内部预绑一堆任务`

### 7.3 任务系统中的 Agent 采用模式

我建议锁定三种采用模式：

#### adopt_existing

直接采用已有 Agent。

适用于：

- 通用任务
- 已有系统管理任务
- 已有稳定工作子 Agent 的专项任务

#### adopt_with_projection

采用已有 Agent，但附带任务侧 projection。

适用于：

- 任务做法已明确
- 但执行姿态需要额外补充

#### spawn_worker_agent

生产新的工作子 Agent。

适用于：

- 持续性的专项任务
- 需要隔离任务记忆
- 需要形成稳定工作单元

### 7.4 多 Agent 协调中的任务级组织

当任务进入多 Agent 协调模式时，任务系统还应进一步决定：

- 哪个 Agent 是 coordinator
- 哪些 Agent 是 participant
- 每个 Agent 负责哪个拓扑节点
- 节点之间如何交换结果
- 失败后由谁接管或汇总

这些都属于任务组织，不属于单个 Agent 的内部运行装配。

## 八、任务系统与记忆系统的接口

任务系统不直接取记忆，但必须正式向记忆系统提出申请。

### 8.1 正式接口方向

建议固定为：

`TaskExecutionAssembly -> TaskMemoryRequestProfile -> MemoryRequest`

也就是说：

1. 任务系统先形成本次任务装配
2. 再根据任务装配生成记忆请求
3. 记忆系统返回 `MemoryBundle`

### 8.2 为什么任务系统必须显式申请记忆

因为不同任务对记忆的需求差异极大。

例如：

- 通用任务可能只需要会话摘要
- 健康诊断任务需要系统运行历史
- 开发任务需要项目背景、任务记忆和近期失败信息

如果任务系统不显式表达，runtime 就会乱取，结果会非常不稳定。

## 九、任务系统与编排系统的接口

任务系统不定义单 Agent 运行体本体，但必须明确提出运行侧装配目标。

建议固定为：

`TaskExecutionAssembly -> AgentAssemblySpec request`

任务系统至少要向编排系统传达：

- 采用哪个 Agent
- 当前任务是什么
- workflow 是什么
- projection 是什么
- flow contract 是什么
- 需要什么 memory scope
- 输入输出契约是什么
- 安全策略是什么

编排系统再根据这些信息生成真正的运行装配结果。

## 十、任务系统与 runtime 的接口

runtime 不应该直接吃零散字段。

runtime 应该只消费任务系统产出的正式装配对象。

建议方向：

- task system 输出 `TaskExecutionAssembly`
- 编排系统输出 `AgentRuntimeSpec`
- memory system 输出 `MemoryBundle`
- runtime 统一消费三者

对于多 Agent 任务，还应再补一层：

- task system 输出 `CoordinationTopologyTemplate`
- task system 输出 `TaskCommunicationProtocol`

runtime 再根据这些任务级组织对象驱动多个 Agent 的协作执行。

## 十一、前端任务系统的管理结构

任务系统前端必须以管理为目的，而不是做成说明页或空总览。

建议前端一级结构如下：

### 11.1 Agent 管理

负责：

- 查看三类 Agent
- 进入 Agent 列表
- 创建工作子 Agent
- 编辑 Agent 基本登记信息
- 查看 Agent 默认定义
- 查看当前承载任务

### 11.2 通用任务管理

负责：

- 查看主会话默认承接任务配置
- 管理通用任务的默认 workflow
- 管理通用任务的默认投影策略
- 管理通用任务的默认记忆请求策略

说明：

这里不强调“选择通用任务”，而强调“平台默认如何承接”

### 11.3 特定任务管理

负责：

- 创建特定任务
- 编辑任务契约
- 绑定 workflow
- 绑定 projection
- 绑定 flow contract
- 配置 Agent 采用模式
- 配置记忆请求
- 配置安全策略

### 11.4 workflow 编辑入口

workflow 不应单独做成狭窄附栏。

更合理的方式是：

- 在任务详情内进入 workflow 编辑
- 或跳转到任务内嵌的独立编辑页

这样 workflow 永远附属于任务，不会脱离任务游离成无主对象。

### 11.5 任务执行装配预览

任务系统前端应支持预览：

- 这次任务会采用哪个 Agent
- 会挂什么 workflow
- 会挂什么 projection
- 会申请什么记忆
- 会采用什么 flow contract
- 会形成什么 runtime 装配目标

### 11.6 多 Agent 协调配置

任务系统前端还应支持多 Agent 协调任务配置。

至少要支持：

- 选择 coordinator Agent
- 选择 participant Agents
- 选择或编辑拓扑模板
- 选择通信协议
- 查看 handoff 规则
- 查看结果汇总规则

这里的界面重点不是“看图”，而是让任务协调关系真正可配置、可校验、可落地。

这是“管理是否有效”的关键。

## 十三、实施顺序

建议按以下顺序实施任务系统重构：

1. 统一任务对象命名和分层
2. 明确通用任务与特定任务的登记结构
3. 补出 `TaskAgentAdoptionPlan`
4. 补出 `TaskMemoryRequestProfile`
5. 补出 `CoordinationTaskRecord / CoordinationTopologyTemplate / TaskCommunicationProtocol`
6. 补出 `TaskExecutionAssembly`
7. 让前端围绕这些对象重构任务管理界面
8. 再对接编排系统、记忆系统和 runtime

## 十四、执行结论

任务系统不是平台上的一个普通功能区，而是平台真正的任务调度中枢。

它的目标不是让用户“看到很多卡片”，而是让平台能够：

- 清楚识别任务
- 清楚定义任务
- 清楚决定由谁执行
- 清楚决定要挂什么任务配置
- 清楚地把任务交给其他系统继续装配和执行

只有这条线做稳了，Agent 平台才算真正立起来。

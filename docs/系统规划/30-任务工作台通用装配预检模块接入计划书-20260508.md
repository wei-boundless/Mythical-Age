# 任务工作台通用装配预检模块接入计划书

日期：2026-05-08

## 1. 背景

当前任务系统已经具备任务域、具体任务、契约库、任务图、协调任务、拓扑模板、通信协议、ContractManifest 与 RuntimeAssembly 的后端能力。

但前端工作台仍有一个结构缺口：

- 任务图编辑能保存拓扑，但用户看不到该图如何进入运行装配。
- 契约页面能编辑 ContractSpec，但用户无法从图节点直接检查契约编译结果。
- 预检发布页更多是状态摘要，没有成为通用的 Runtime 装配检查模块。
- 单 Agent 任务和多 Agent 任务的预检入口不统一。

因此，像“长篇小说撰写”这种特定任务可以在定义层搭建出来，但工作台还不能完整支撑用户判断：

```text
这个任务图能不能被编译？
每个节点会装配哪个 Agent？
节点能看到哪些上下文？
输出契约和交接契约有没有进入 runtime assembly？
```

## 2. 目标

补齐一个通用模块，不为小说写死任何字段。

目标模块名：

```text
装配预检
```

它服务所有任务域、所有具体任务、所有任务图：

- 单 Agent 图：检查工作流 ContractManifest 与 single-agent RuntimeAssembly。
- 多 Agent / 协调图：检查协调 ContractManifest，并按节点检查 node RuntimeAssembly。
- 契约边界：显示当前任务、workflow、图、节点引用到的契约。
- Agent 装配：显示节点绑定的 Agent、runtime profile、context sections、output contracts、diagnostics。
- 缺口提示：把缺 agent、缺 workflow、缺 contract、图结构错误统一呈现。

## 3. 工作台层级调整

工作台层级从四层扩为五层：

```text
任务域
任务图
契约
预检发布
运行循环
```

但把 `预检发布` 改为真实模块：

```text
预检发布
  - 图结构检查
  - Workflow Manifest
  - 单任务 RuntimeAssembly
  - 协调 Manifest
  - 节点 RuntimeAssembly
  - 发布动作

运行循环
  - 执行链与运行入口
  - 记忆请求与写回策略
  - 上下文连续性与章节循环就绪度
  - RunLoop 缺口提示
```

不把这些检查散落到任务域页面或图编辑器里。
其中：

- `预检发布` 负责证明“能装配”
- `运行循环` 负责表达“能连续执行”

## 4. 实施范围

### 4.1 前端 API

补齐或复用：

- `compileTaskSystemWorkflowContractManifest`
- `compileTaskSystemCoordinationContractManifest`
- `buildTaskSystemWorkflowRuntimeAssembly`
- `buildTaskSystemNodeRuntimeAssembly`

这些 API 已存在，主要缺前端模块化调用。

### 4.2 前端组件

新增：

```text
frontend/src/components/workspace/views/task-system/TaskAssemblyPreflightPanel.tsx
frontend/src/components/workspace/views/task-system/TaskRunLoopWorkbenchPanel.tsx
```

职责：

- 接收当前任务、workflow、协调任务、图节点、图校验结果。
- 提供“刷新预检”按钮。
- 显示单任务与多 Agent 的装配快照。
- 显示节点列表，并允许选择节点查看 RuntimeAssembly。
- 显示错误状态，不阻塞用户回到图编辑。

RunLoop / Memory 面板职责：

- 接收当前任务、执行策略、记忆请求、协调任务与当前 RuntimeAssembly。
- 统一展示单 Agent 与多 Agent 的运行链类型。
- 展示 `requested_memory_layers`、`memory_priority`、`writeback_policy`、`allow_long_term_memory`。
- 展示协调任务的 `shared_context_policy`、`memory_sharing_policy`。
- 基于现有配置给出“连续章节运行”的 readiness 判断。
- 明确说明当前只是运行定义层与 readiness，不伪装成已经具备完整章节执行器。

修改：

```text
frontend/src/components/workspace/views/TaskSystemView.tsx
frontend/src/lib/api.ts
```

### 4.3 验收标准

1. 工作台 `预检发布` 层能真实调用后端编译/装配接口。
2. 当前任务没有 workflow 时，显示明确缺口。
3. 当前协调图没有节点时，显示明确缺口。
4. 单任务和多 Agent 使用同一个预检模块。
5. 用户可以从预检结果回到任务图继续修。
6. 工作台新增 `运行循环` 层，不再把 RunLoop / Memory 藏在任务定义系统字段中。
7. 运行循环层能明确显示记忆请求与写回策略。
8. 不引入小说专用字段；长篇小说只是该通用机制的一个配置案例。

## 5. 当前纳入工作台、但后端仍待深化的能力

本计划要把 RunLoop 与 Memory 纳入工作台边界，但仍不伪装成完整长篇小说执行引擎。

后续如果要真正高质量运行长篇小说任务，还需要：

- 章节级长期记忆与设定检索策略。
- 大纲、角色、世界观、章节正文的产物版本库。
- 多轮章节 RunLoop 与中断恢复。
- 连续性检查 Agent 的专用评审契约。
- 主 Agent 对多节点输出的合并与返修循环。

这些属于 RunLoop 深化、产物系统深化与记忆治理深化。本轮先保证：

- 工作台里有正式的 RunLoop / Memory 定义层入口；
- 用户能看到连续执行还缺什么；
- 后续后端执行器可以直接接这套配置继续演进。

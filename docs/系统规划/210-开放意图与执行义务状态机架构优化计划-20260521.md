## 1. 背景与结论

本计划用于修正专业模式长任务中的结构性问题：当前系统把 `task_goal_type` 同时作为意图分类、专业 prompt、执行义务、工具强制、交付验证和任务领域的总开关。一旦分类错误，后续执行链路会整体偏离用户目标。

本次深度代码执行任务暴露的问题是：

```text
用户要求：修复代码、修改文件、运行 pytest、验证通过
系统误判：test_report_triage
结果：不强制 edit_file，不强制 terminal 验证，prompt 反向提示“不负责修改代码”
```

结论：

**不应继续强化显式任务域作为核心意图系统。**

新的核心应是：

```text
开放意图帧
-> 执行义务抽取
-> 软策略原型匹配
-> 执行义务状态机
-> 工具观察与交付验证
```

`task_goal_type` 可以保留，但必须降级为“策略原型”，不能再作为写入、验证、终止、交付的唯一依据。

### 2.2 当前结构性缺陷

#### 缺陷一：`task_goal_type` 责任过载

当前 `backend/tasks/semantic_task_contracts.py` 中，`task_goal_type` 影响：

- `domain`
- `professional_profile_id`
- `deliverables`
- `required_reasoning_steps`
- `required_actions`
- `forbidden_actions`
- `material_handling_policy`
- `validation_schema`

这导致分类错误会级联污染整个运行时。

#### 缺陷二：分类规则是 first-match 硬分支

当前 `_resolve_task_goal_type()` 中，`test_report_triage` 分支优先于 `code_fix_execution`。只要出现“失败”以及路径里有 `test`、`fixture`、`.json` 等信号，就可能抢先命中。

这类硬分支不具备开放意图理解能力，也不适合承载长任务控制。

#### 缺陷三：写入和验证义务依赖分类

当前 `ProfessionalTaskRunDriver` 里：

```python
requires_write = "apply_real_change" in required_actions or task_goal_type in {"code_fix_execution", "artifact_delivery"}
requires_verify = "validate_deliverables" in required_actions and task_goal_type in {"code_fix_execution", "regression_test_design"}
```

这意味着用户明确要求“修改文件、运行 pytest”，但只要分类错误，写入和验证就不会变成硬约束。

#### 缺陷四：专业 prompt 与分类绑定过死

`professional.test_report_triage` prompt 明确告诉模型“不负责修改代码”。当任务误分类后，prompt 会直接压制用户真实请求。

专业 prompt 应该服务于职责风格和交付标准，而不能覆盖本轮从用户目标中抽取出的执行义务。

#### 缺陷五：TaskDomain 不是运行时意图系统

当前 `TaskDomainRecord` 只有：

```python
domain_id
task_family
title
description
enabled
sort_order
metadata
```

它更适合作为任务管理目录、任务图归属、默认配置入口，不适合作为开放意图分类器。把所有任务塞进 TaskDomain 只会形成新的闭集和维护负担。

## 3. 参考系统结论：Claude Code 的长任务机制

本机参考目录：

```text
D:\AI应用\claude-code-nb-main
```

仅作为架构参考，不引入依赖，不照搬实现。

### 3.1 可借鉴机制

#### 权限模式是执行控制，不是任务分类

Claude Code 使用类似：

```text
default / plan / acceptEdits / bypassPermissions / dontAsk
```

这些模式控制工具执行权限，而不是定义用户任务类型。

#### Plan mode 是阶段状态机

Plan mode 管的是：

```text
探索代码 -> 写计划 -> 请求批准 -> 退出计划模式 -> 执行
```

它不是“任务域”，而是运行阶段。

#### Todo/Task 是进度状态机

Todo/Task 使用：

```text
pending / in_progress / completed
```

长任务通过显式进度状态维持，而不是依赖模型上下文自己记住下一步。

#### 文件编辑有工具层状态校验

FileEdit 不是只靠 prompt 提醒“先读再改”，而是检查 `readFileState`、mtime、旧内容匹配等状态。也就是说：

```text
read_file 观察 -> edit_file 资格 -> stale write 检查 -> 写入后更新读状态
```

这是长代码任务稳定性的关键。

### 3.2 不应照搬的部分

不应照搬 Claude Code 的产品级复杂度，例如完整权限 UI、远程 session、team task、worktree 生命周期。当前项目应先把长任务最核心的结构补齐：

- 开放意图
- 执行义务
- 工具状态
- 进度状态
- 验证状态

## 4. 目标设计

### 4.1 总体原则

1. **分类开放，执行刚性。**
2. **任务域是目录，不是大脑。**
3. **执行义务高于策略原型。**
4. **prompt 不能覆盖用户本轮明确要求。**
5. **写入、验证、交付必须由状态机追踪，不靠模型自觉。**
6. **不保留旧链路作为兼容分支，切换完成后删除旧推导路径。**

### 4.2 新运行结构

```text
UserMessage
  -> IntentFrame
  -> ExecutionObligation
  -> StrategyPrototypeMatch
  -> SemanticTaskContract
  -> RuntimeInteractionModePolicy
  -> ProfessionalTaskGoalContract
  -> ProfessionalTaskStateMachine
  -> ToolObservationLedger
  -> ObligationValidation
  -> FinalAnswer
```

其中：

- `IntentFrame` 负责开放理解。
- `ExecutionObligation` 负责硬约束。
- `StrategyPrototypeMatch` 负责软策略。
- `SemanticTaskContract` 负责把上面三者组合成运行契约。
- `ProfessionalTaskStateMachine` 只信义务和真实观察，不信模型口头承诺。

## 6. 执行义务抽取规则

### 6.1 写入义务

命中任一条件即 `write_required=True`：

- 用户说“修改文件、改代码、修复、写入、生成文件、保存、产出文件、apply patch”
- 目标路径是代码或配置文件，并且动作是修复/实现/调整
- 用户指定 sandbox overlay 中修改
- 用户要求“真实完成”而非“给建议”

反向条件：

- “只分析”
- “不要改”
- “先不要动代码”
- “只给方案”
- “审查一下”

反向条件优先级高于一般写入词，但不能覆盖明确的“开始实施”。

### 6.2 验证义务

命中任一条件即 `verify_required=True`：

- 用户要求“运行 pytest / 测试通过 / 验证通过 / 运行命令 / terminal / shell”
- 写入目标是代码，并且存在测试文件或测试命令提示
- 任务属于代码修复执行，且不是只要方案

提取命令提示：

```text
pytest ...
python -m pytest ...
npm test
pnpm test
uv run pytest
```

### 6.3 读取义务

命中任一条件即 `read_required=True`：

- 用户指定文件、目录、报告、trace、PDF、数据集
- 写入义务存在且目标路径存在
- 用户要求“先阅读/审查/追踪/定位”

### 6.4 交付义务

默认 `deliverable_required=True`。

若存在写入或验证义务，最终回答至少包含：

- 修改点
- 修改文件
- 验证命令和真实结果
- 未覆盖边界或限制

## 7. 专业模式状态机设计

### 7.1 状态定义

新增或重构专业模式状态：

```text
bind_contract
read_materials
plan_actions
apply_changes
run_verification
repair_if_failed
validate_deliverables
finalize
```

### 7.2 状态推进规则

```text
read_required=true 且未读材料
  -> required_next_tools = read_file / read_structured_file / search_files

write_required=true 且材料已读且未写入
  -> required_next_tools = edit_file / write_file

verify_required=true 且已写入且未验证
  -> required_next_tools = terminal

verify_required=true 且验证失败
  -> 回到 apply_changes，除非预算耗尽

deliverable_required=true 且最终回答缺少义务交付物
  -> repair final answer
```

### 7.3 状态机禁止项

以下行为必须被 gate 拦截：

- 写入义务未完成时继续反复搜索。
- 验证义务未完成时直接 final。
- 未运行测试却声称测试通过。
- 输出伪工具调用 markup。
- 在 `plan_required=true` 阶段写业务文件。
- 在 `write_required=false` 且用户明确“不要改”时写文件。

## 8. Prompt 架构调整

### 8.1 当前问题

当前 prompt profile 与 `task_goal_type` 强绑定。一旦误判，会给模型错误身份。

### 8.2 新 prompt 分层

专业模式 prompt 应分为四层：

```text
基础专业执行准则
-> 执行义务说明
-> 策略原型角色
-> 投影风格
```

优先级：

```text
用户本轮明确要求
> 执行义务
> 工具和权限约束
> 策略原型角色
> 灵魂投影风格
```

### 8.3 示例：代码修复执行 prompt

应写成给 agent 的专业角色提示：

```text
你是一名结构性代码修复执行员。
你负责先阅读相关代码和测试，定位导致失败的真实逻辑问题，然后做最小但完整的可维护修改。
你必须在修改后运行用户要求或系统推导出的验证命令。
如果验证失败，你需要根据真实输出继续修复，直到通过或明确说明阻塞原因。
你不能把未运行的测试说成已通过。
你不能为了单个样本硬编码结果。
```

不应写成：

```text
这是 code_fix_execution 节点。
执行 inspect/edit/verify。
```

## 9. TaskDomain 的正确位置

TaskDomain 保留，但定位调整为：

```text
任务管理目录 / 任务图归属 / 默认配置入口
```

它可以提供：

- 默认 task_family
- 默认 memory scope
- 默认 projection policy
- 默认 strategy preference
- 默认 tools profile
- UI 分类展示

它不应直接决定：

- 本轮是否必须写文件
- 本轮是否必须验证
- 本轮是否能 final
- 用户意图是否“存在”

不再把 TaskDomain 当作闭集分类表。

### 必过场景

| 场景 | 期望 |
|---|---|
| 深度代码修复 | read -> edit -> terminal -> final |
| 测试报告诊断 | read report -> evidence packet -> triage final |
| 只分析不改 | 不出现 edit/write |
| 生成计划书 | 写入文档或明确交付 |
| 未知长任务 | generic professional task，不降级成 light QA |
| 验证失败后修复 | terminal fail -> edit -> terminal retry |

## 12. 风险与控制

| 风险 | 控制 |
|---|---|
| 义务抽取过强导致误写文件 | 明确反向条件：“不要改”“只分析”“先别动代码”优先 |
| 义务抽取过弱导致不执行 | 写入/验证关键词和目标路径联合判断，增加回归 |
| 策略原型候选复杂化 | 原型只影响 prompt 和默认步骤，不影响硬义务 |
| prompt 分层冲突 | 执行义务段优先级高于策略角色和投影 |
| sandbox overlay 继续困惑 | 工具观察账本记录 overlay 状态，并在 terminal 找不到文件时结构化提示 |
| 旧测试依赖 task_goal_type | 更新测试断言为 obligations + selected_strategy |

## 13. 交付定义

本升级完成后，系统应满足：

1. 用户明确要求修改文件时，系统不会因为分类错误而只分析。
2. 用户明确要求验证时，系统不会跳过 terminal。
3. 专业模式长任务有显式进度状态和工具观察账本。
4. `task_goal_type` 不再是执行总开关。
5. TaskDomain 不再被误用为闭集意图分类。
6. prompt 是给 agent 的专业职责提示，不是内部节点说明。
7. 旧分类驱动执行义务的链路被清理，不保留双轨兼容。

## 14. 最终目标形态

最终专业模式应像这样运行：

```text
用户：修复某文件并运行 pytest

系统抽取：
read_required=true
write_required=true
verify_required=true
target_paths=[...]
verification_commands=[pytest ...]
selected_strategy=code_fix_execution

状态机执行：
read_file
edit_file
terminal pytest
如果失败 -> edit_file -> terminal pytest
final answer with changed files, verification result, boundaries
```

而不是：

```text
用户：修复某文件并运行 pytest
系统：因为出现“失败”和 tests/fixtures，所以 test_report_triage
模型：我不负责修改代码，只做失败归类
```

这次升级的核心不是增加更多分类，而是把长任务的“该做什么”从分类里解放出来，沉到可验证的执行义务状态机里。

# TaskGraph 通用编辑器前端一体化优化计划

## 0. 计划定位

本计划是 TaskGraph 通用编辑器前端的总计划。

`124-TaskGraph通用记忆仓库读写矩阵前端优化计划` 只覆盖“记忆与产物”这一层，不应该替代前端整体优化。真正的目标是把现有 TaskGraph Studio 做成一个能配置长任务、循环任务、审核返修任务和多 Agent 协同任务的通用工作台。

本计划统筹以下几个方向：

```text
图级蓝图
执行主体
拓扑编排
节点执行认知包
时序 clock / scope / event
通用记忆仓库与产物上下文
契约与质量门
发布预检
运行监控与交接追踪
```

核心判断：

```text
TaskGraph 编辑器负责通用任务设计。
编排系统负责 Agent、投影、资源和静态配置。
运行系统负责 TimelineLedger、ContextPacket、Receipt 和真实可见性。
前端必须把这三者组织成用户能理解和能配置的工作台。
```

## 1. 总体目标

用户最终应该能在通用编辑器里完成：

1. 设计任务图拓扑。
2. 配置每个节点的身份、职责、输入包和输出契约。
3. 配置时序结构：阶段、循环、并发、审核返修、人工门控。
4. 配置记忆仓库和产物仓库的定向读写。
5. 发布前验证：节点是否知道自己收到什么、怎么用、产出什么、交给谁。
6. 运行时追踪：当前 clock、scope、dispatch、packet、receipt、checkpoint 和节点正文。

一句话：

```text
点开任意节点，用户能回答：
这个节点是谁，何时运行，收到什么，如何理解，产出什么，交给谁，何时生效。
```

## 4. 信息架构总设计

保留现有主导航层级，但每层职责收束如下：

### 4.1 图级蓝图

负责：

- 图身份。
- 图边界。
- 入口出口。
- 运行模式。
- 全局上下文策略。

不负责：

- 节点 prompt。
- 记忆 selector 细节。
- 时序事件细节。

### 4.2 执行主体

负责：

- Agent 绑定。
- Agent group。
- runtime lane。
- 工具/技能可见性摘要。
- 跳转到编排系统配置 profile / projection。

不负责：

- 节点具体输入包。
- 输出契约。

### 4.3 拓扑编排

负责：

- 节点和边。
- 控制节点、资源节点、业务节点的形状。
- 连线类型。
- 选中节点/边。

不负责：

- 深度字段堆叠。
- prompt 正文编辑。

### 4.4 职责与交接

升级为“节点执行认知包”配置页。

负责：

- 节点是谁。
- 节点本轮任务是什么。
- 节点收到哪些 packet。
- 每个 packet 怎么称呼、怎么用。
- 节点必须产出什么。
- 产物交给谁。
- 模型可见 prompt 预览。

这是前端最关键的一页。

### 4.5 时序系统

负责：

- 阶段。
- 顺序。
- 循环框。
- 并发组。
- 审核返修路由。
- 人工门控。
- 运行时 clock/scope/event 的说明和预览。

不把节点本身叫时序。

### 4.6 记忆与产物

按照 124 计划执行。

负责：

- 通用记忆仓库。
- collection。
- read/write_candidate/commit 矩阵。
- selector。
- version selector。
- MemorySnapshot 配置预览。
- ArtifactContextPacket 配置预览。

### 4.7 契约与质量门

负责：

- input contract。
- output contract。
- payload contract。
- review verdict。
- quality retry policy。
- receipt policy。
- commit rule。

重点不是列 schema，而是解释“这个契约在节点执行认知包里起什么作用”。

### 4.8 发布闭环

负责：

- 预检。
- 保存。
- 发布。
- 运行绑定。
- 测试请求预览。

新增预检要覆盖：

- packet/prompt 是否配套。
- memory selector 是否完整。
- revision edge 是否能携带原稿和审核意见。
- review gate 是否有 verdict。
- write candidate 是否有 commit path。
- receipt 生效规则是否缺失。

### 4.9 运行监控

负责：

- topology runtime status。
- TimelineLedger events。
- dispatch context。
- current packets。
- receipts。
- stream preview。
- artifact preview。
- 交接链路追踪。

监控页不只是看状态，还要能定位“为什么这个节点收到这些输入”。

## 5. 核心工作台布局

### 5.1 通用布局

推荐所有复杂层级页采用同一工作台结构：

```text
顶部：当前层级说明 + 关键状态
左侧：对象列表 / 分面导航
中间：主配置区
右侧：预览 / 校验 / 来源说明
底部或侧边：保存、预检、跳转操作
```

### 5.2 选中对象一致性

编辑器应该维护统一选中状态：

```text
selected_node_id
selected_edge_id
selected_repository_id
selected_packet_kind
```

不同页面根据选中对象展示不同详情。

例如：

- 拓扑页选中 `world_review`。
- 切到职责页，自动显示这个节点的执行认知包。
- 切到记忆页，自动显示这个节点可读可写的仓库矩阵。
- 切到时序页，自动显示这个节点所在 phase / loop / revision route。

## 16. 验收标准

完成后，TaskGraph 通用编辑器应满足：

1. 用户能通过拓扑页设计任务结构。
2. 用户能通过职责页配置节点执行认知包。
3. 用户能通过时序页设计阶段、循环、并发、返修和门控。
4. 用户能通过记忆页配置仓库、selector 和 snapshot。
5. 用户能通过契约页配置输出、审核、质量门和 receipt。
6. 用户能通过发布页发现配置缺口。
7. 用户能通过监控页追踪 clock、packet、receipt 和节点正文。
8. 全部页面保持通用，不出现领域专用控件和硬编码任务逻辑。

最终目标是让用户不需要写代码，也能在编辑器里配置一条可持续运行的长任务拓扑。

## 17. 当前追加实施：统一 EditorFocus 操作主线

本轮不继续增加页面，而是把已有页面收束成一条稳定操作主线：

```text
选中对象 -> 看结构 -> 配权限 -> 看执行包 -> 查断点
```

### 17.1 技术底座

新增统一焦点对象：

```ts
TaskGraphEditorFocus {
  layer: "topology" | "memory" | "responsibility" | "timeline" | "contracts" | "publish" | "blueprint" | "agents"
  facet?: string
  node_id?: string
  edge_id?: string
  repository_id?: string
  collection_id?: string
  issue_id?: string
}
```

原则：

- `TaskGraphWorkbench` 是焦点唯一入口。
- 旧的 `selectedGraphNodeId / selectedGraphEdgeId` 保留为画布和旧组件的兼容投影，但不再作为跨页导航主语。
- 发布页预检、拓扑 inspector、记忆矩阵、职责认知包都通过 `EditorFocus` 跳转到具体层、具体分面、具体对象。
- 焦点不保存独立配置；配置仍写回图节点、图边和图 metadata。

### 17.2 拓扑页优化

拓扑页只做结构地图：

- 节点形状区分执行节点、审核门、循环/门控节点、记忆仓库、产物仓库、账本。
- 边按类型区分控制流、普通交接、`memory_read`、`memory_write_candidate`、`memory_commit`、`artifact_context`、`revision_request`。
- 增加图层过滤：全部、执行流、记忆流、产物流、返修流。
- 右侧 inspector 只显示结构摘要和“去配置”入口，不塞 selector 表单。

### 17.3 记忆页优化

记忆页做资源权限控制台：

- 支持从 `EditorFocus` 打开仓库结构、Selector、Snapshot 或矩阵分面。
- 左侧仓库树保持 repository -> collection 层级。
- 矩阵格子选中后右侧显示 selector/version/receipt 摘要，并能跳到 Selector 配置。
- Snapshot 预览按焦点节点定位，标明输入来自哪条边。

### 17.4 职责页优化

职责页做节点运行前预览：

- 支持从 `EditorFocus.node_id` 定位到节点认知包。
- 左侧按阶段列出节点，避免用户只看到当前节点而迷路。
- 输入包旁边标明来源边类型：memory/artifact/revision/handoff。
- Prompt 预览只使用 Agent 可理解的角色和任务语言，不写运行时开发说明。

### 17.5 发布页优化

发布页做编译器式诊断：

- 问题按层级分组。
- 每个问题点击后生成 `EditorFocus`，跳到具体层、分面、节点/边/仓库/collection。
- 可自动修复的问题修完后仍保持焦点，用户能看到修复落点。

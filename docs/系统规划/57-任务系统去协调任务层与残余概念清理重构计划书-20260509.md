# 任务系统去协调任务层与残余概念清理重构计划书

日期：2026-05-09

## 目标

本轮重构不再接受“具体协调任务”作为任务系统中的独立管理层与独立核心模型。

目标结构改为：

```text
任务域 -> 任务 -> 任务图草稿 / 已发布任务图
```

同时清理以下残余概念在任务系统主链路中的地位：

1. `task_family`
2. `task_mode`
3. `coordination_task`

本轮完成后要达到的结果：

1. 前端任务系统不再出现“具体协调任务”这一层。
2. 图编辑器直接围绕“当前任务”的图草稿工作。
3. 后端任务系统不再把 `coordination_task` 作为图的主容器对象。
4. `task_family` / `task_mode` 不再承担任务系统管理层概念。
5. 任务域成为唯一管理边界，任务成为唯一业务承接主体，图成为任务下的结构资产。

## 当前问题

现有系统的主要问题不是单个字段脏，而是结构重叠：

### 1. 管理层重复

当前同时存在：

1. 任务域 `domain_id`
2. 特定任务 `task_id`
3. 协调任务 `coordination_task_id`
4. 拓扑模板 `template_id`
5. 通信协议 `protocol_id`

这导致用户实际在编辑的是“任务图”，但系统强行要求它先挂在一个“协调任务”容器之下。

### 2. 图与协调任务发生镜像重复

现在一份多 Agent 图的数据会被拆在几处：

1. `coordination_task.graph_nodes / graph_edges`
2. `topology_template.nodes / edges`
3. `coordination_graph_spec`
4. runtime loop 中的 `coordination_task_payload`

这不是分层，而是重复存储。

### 3. 任务族群与任务模式是历史兼容概念

`task_family` 与 `task_mode` 在任务系统里已经不再是清晰业务对象：

1. `task_family` 被拿来做域归类、memory topic、runtime lane 推断
2. `task_mode` 被拿来做 workflow/flow/profile 选择

但在用户心智里：

1. 他只关心任务属于哪个任务域
2. 当前任务是什么
3. 当前任务图如何编排

所以这两个字段在任务系统管理层里属于历史兼容概念，不应该继续主导任务系统。

## 新结构

### 一、主对象

保留三层：

1. `TaskDomainRecord`
   - 任务域
   - 唯一管理边界

2. `SpecificTaskRecord`
   - 任务
   - 唯一业务主体

3. `TaskGraphRecord`
   - 任务图
   - 当前任务下的多 Agent 编排资产

### 二、去掉的对象层

从任务系统主链路中移除：

1. `CoordinationTask`
2. `coordination_task_id`
3. `coordination_task_ref`
4. “具体协调任务”前端页面层
5. “任务 -> 协调任务 -> 拓扑/协议”的组织方式

### 三、图资产归属

图相关内容统一并入 `TaskGraphRecord`：

1. 节点
2. 边
3. 通信规则
4. 上下文共享策略
5. handoff / merge / conflict policy
6. 图级 stop conditions
7. graph runtime metadata

拓扑与协议可以保留为子资源或内嵌块，但不再需要协调任务作中间壳。

## 重构原则

### 1. 不做别名式兼容

如果已经决定“没有具体协调任务”，那就不能只是把文案改成“图草稿”。

必须真正把：

1. 前端入口
2. API 路由
3. registry 存储
4. runtime 消费模型

改成直接围绕 `task_graph` 工作。

### 2. 不保留无业务价值的中间壳

`coordination_task` 如果不再承担独立业务价值，就应当删除。

不能继续出现这种链条：

```text
任务 -> 协调任务 -> 拓扑模板 -> 图规范
```

应改成：

```text
任务 -> 任务图
```

### 3. 残余概念只允许存在于真正必要的底层兼容位

`task_family` / `task_mode` 若短期还不能彻底清空，只能：

1. 作为派生字段
2. 作为运行时兼容输入
3. 不再出现在任务系统主编辑链路中

## 实施范围

## A. 前端

### 目标

任务系统前端只保留：

1. 任务域列表
2. 域内任务列表
3. 当前任务图编辑器

### 删除内容

1. `TaskSystemView.tsx`
   - 删除协调任务选择器
   - 删除“新协调任务草稿”
   - 删除“复制协调任务”
   - 删除所有 `selectedCoordinationId` / `editorCoordinationId` 主流程
   - 删除“带入主会话”的协调任务分支

2. `TaskGraphWorkbench`
   - 输入从 `coordinationTask` 切到 `taskGraph`
   - 不再要求 `coordination_task_id`

3. `TaskRunLoopWorkbenchPanel`
   - 不再通过 `selectedCoordination` 判断图级循环
   - 改为通过当前任务图是否存在

4. `frontend/src/lib/api.ts`
   - 任务系统主类型改为以 `TaskGraphRecord` 为中心
   - 协调任务类型降级或删除

## B. 后端 API

### 目标

任务 API 主模型改成：

1. 域
2. 任务
3. 任务图
4. 契约
5. 工作流
6. 投影/执行/记忆绑定

### 删除内容

1. `backend/api/tasks.py`
   - 删除或废弃 `/tasks/coordination-tasks/*`
   - 删除 `CoordinationTaskUpsertRequest`
   - `contract-manifest/coordination/*` 改为 `contract-manifest/task-graphs/*`
   - `runtime-assemblies/coordination/*` 改为 `runtime-assemblies/task-graphs/*`

2. overview payload
   - 不再返回 `coordination_tasks`
   - 主返回结构改成 `task_graphs`

## C. Registry 与存储

### 目标

存储层不再维护“协调任务表”作为任务图的上层实体。

### 删除内容

1. `storage/tasks/coordination_tasks.json`
   - 退役

2. `backend/tasks/flow_registry.py`
   - 删除 `list_coordination_tasks`
   - 删除 `get_coordination_task`
   - 删除 `upsert_coordination_task`
   - 删除 `coordination_task` 到 `task_graph` 的镜像拼装逻辑

3. `TaskGraphRecord`
   - 成为唯一图资产记录
   - 接管原协调任务中的图级策略字段

## D. Runtime Loop

### 目标

runtime loop 不再接收：

1. `coordination_task_ref`
2. `coordination_task_record`

而是直接接收：

1. `task_graph_id`
2. `task_graph_record`

### 删除内容

1. `backend/orchestration/runtime_loop/task_run_loop.py`
   - 任务启动与事件写入改成 `task_graph_ref`
   - 删除 `coordination_task_payload`
   - 删除以协调任务为中介的节点分发逻辑

2. `langgraph_coordination_runtime.py`
   - 运行输入改为任务图
   - 不再查询 `get_coordination_task`

3. `contract_compiler.py`
   - 输入改为 `TaskGraphRecord`
   - 不再依赖 `CoordinationTaskDefinition`

## E. task_family / task_mode 清理

### 原则

这两个字段本轮不追求全仓完全物理消灭，但要在任务系统主链路中彻底退位。

### 动作

1. `task_family`
   - 域、任务、图的主归属全部改为 `domain_id`
   - registry 删除、筛选、推断优先 `domain_id`
   - `task_family` 只允许从 `domain_id` 派生

2. `task_mode`
   - 从任务系统前端主界面移除
   - runtime / workflow 如仍需要，用工作流或执行策略派生
   - 不再作为任务系统一等管理字段

## 分阶段执行

### 第一阶段：前端主层级拆壳

1. 去协调任务页层
2. 图编辑器改为任务直开
3. overview 前端只按域/任务/图工作

### 第二阶段：后端 API / registry 去协调任务层

1. 去掉 coordination task 主路由
2. `task_graph` 接管图主模型
3. 存储层停止维护 `coordination_tasks.json`

### 第三阶段：runtime 改成以 task graph 为中心

1. `coordination_task_ref` 改为 `task_graph_ref`
2. contract compiler / assembly builder / runtime loop 主链路改造

### 第四阶段：残余兼容清扫

1. 清理无用前端类型
2. 清理 storage 残留
3. 清理旧测试
4. 修正事件文案、会话选择、管理页面统计

## 风险

1. runtime loop 当前深度绑定 `coordination_task`
2. contract manifest、runtime assembly、trace reader 都在吃 `coordination_task_ref`
3. 一次性硬改需要同步更新测试与事件模型

这意味着本轮必须按结构性重构推进，不能靠局部补丁。

## 验收标准

1. 前端任务系统不再出现“具体协调任务”概念。
2. 任务系统主链路只剩 `任务域 -> 任务 -> 任务图`。
3. 后端 overview 不再返回 `coordination_tasks` 作为任务系统一等资源。
4. runtime loop 主链路不再依赖 `coordination_task_ref`。
5. `storage/tasks/coordination_tasks.json` 不再作为主存储来源。
6. `task_family` / `task_mode` 不再作为任务系统前端主编辑概念。
7. `npx tsc --noEmit`、`npm run lint`、后端相关测试与编译检查通过。

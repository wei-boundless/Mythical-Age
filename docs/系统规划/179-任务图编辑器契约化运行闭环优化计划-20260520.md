# 任务图编辑器契约化运行闭环优化计划

状态：运行闭环收尾完成：核心契约化编辑、发布执行包、对象追溯、GraphUnit 编译闭环、GraphUnit 子图启动/提交/父图续跑闭环、边时序支持矩阵第一轮治理已完成；剩余工作进入边时序全量运行兑现、追溯联动体验和旧字段清理阶段。

设计依据：

- `178-通用任务图单位批次契约与动态范围架构计划-20260520.md`
- 当前代码事实：`TaskGraphWorkbench`、`TaskGraphObjectInspector`、`TaskGraphContractQualityPage`、`composable_graph_builder`、`coordination_graph_compiler`、`contract_compiler`、`task_graph_scheduler`
- 当前目标约束：任务图是一等概念；契约是一等运行协议；图节点必须可运行；前端配置必须真实进入后端执行包。

旧设计文档只作为历史问题记录，不作为本计划的设计依据。如果旧实现或旧字段不符合本计划目标，应迁移后清理，不能以兼容为理由长期保留。

## 0. 核心结论

当前任务图编辑器的问题不是“页面还不够多”，而是“编辑事实、契约事实、运行事实没有统一”。现在前端能配置不少对象，后端也能编译 standard view、manifest、runtime spec，但这些链路还没有形成同一份权威事实。

真正缺口不是再补几个表单字段，而是要把以下四件事统一成一个对象模型：

1. 图、节点、图节点、边、端口、时序块都是可编辑对象。
2. 每个对象的配置都进入统一 `contract_bindings`。
3. 标准视图、契约清单、运行规范和发布执行包必须从同一份任务图事实编译出来。
4. 运行时只能消费编译结果，不能从旧字段或领域名私自推断行为。

178 号计划解决“契约是什么、怎么分类、怎么管”。本计划解决“编辑器如何重新组织对象与配置，让契约真实进入运行”。

本计划的优化方向不是继续修补旧页面，而是建立新的闭环：

```text
对象选择 -> 契约化编辑 -> 统一保存 -> manifest 编译 -> runtime spec 编译 -> execution package 验证 -> 调度运行
```

只要某个 UI 配置不能走完这条链路，就不应在界面上表现为“已生效”。

## 1. 问题重新定义

用户真正需要的不是一个普通流程图编辑器，而是一个任务图编辑器：

- 节点不是任务本身，而是任务图中的时序执行位。
- 任务是运行时动作，由节点激活并按契约执行。
- 图可以作为节点参与父图，也可以展开成子图。
- 边不是装饰线，而是交接协议和时序传播结构。
- 时序不是额外字段，而是图结构在运行中的组织方式。
- 契约不是质量门小工具，而是节点、边、图、运行包之间的正式协议。

因此编辑器必须服务于一个核心目标：

> 用户在图上选中任意对象，就能自然配置它的职责、接口、契约、时序、记忆、产物和运行策略；保存后这些配置能被后端编译成同一份可运行事实。

### 2.2 后端现状事实

- `backend/tasks/composable_graph_builder.py` 已能从任务图派生 Unit、Interface、PortEdge、NestedRuntimePlan。
- `backend/tasks/task_graph_standard_models.py` 已把 runtime spec、layered graph、composable graph 汇总成 standard view。

- `backend/orchestration/runtime_loop/task_graph_scheduler.py` 已有阶段、节点、边状态调度能力。

### 2.3 当前关键限制

1. 可组合图仍是影子模型  
   `composable_graph_builder.py` 的 diagnostics 会标记 `read_only_shadow_model` 或 `metadata_overlay_shadow_model`。它能表达图节点和嵌套运行计划，但还不是运行时权威模型。

2. GraphUnit 未完全进入运行调度  
   `linked_graph_id` 和 `nested_runtime` 能在 standard view 中出现，但 `compile_task_graph_definition_runtime_spec()` 仍主要从 `graph.nodes` / `graph.edges` 编译运行节点，子图联动运行尚未成为调度器的一等能力。

3. 契约配置仍散落在旧字段  
   前端仍主要写 `graph_contract_id`、`input_contract_id`、`output_contract_id`、`payload_contract_id`、`node_contract_id`。这些字段无法表达 execution、unit batch、artifact、memory、acceptance、runtime 等完整契约分类。

4. 对象编辑台过度集中  
   `TaskGraphObjectInspector.tsx` 同时负责图、节点、图节点、边、时序块、嵌套运行、诊断问题，短期可用，长期会成为难维护的超级组件。

5. 前端配置和运行支持存在落差  
   前端已经能配置边等待、ack、失败传播、可见时机等字段，但后端 diagnostics 中仍有 `partial` / `unsupported` 状态。用户会以为配置已生效，实际运行未必严格兑现。

6. 发布前执行包不统一  
   现在 standard view、contract manifest、runtime spec、runtime assembly 分散在不同入口。用户很难一眼确认“我图上配置的对象最终进入了哪个运行节点、哪个边交接包、哪个验收规则”。

## 3. 目标架构

### 3.1 新事实源

任务图编辑器的事实源应收敛为：

```text
TaskGraphDefinition
  graph fields
  nodes[]
  edges[]
  metadata.layered_graph
  metadata.composable_graph
  contract_bindings
```

它编译出四类派生视图：

```text
TaskGraphDefinition
  -> TaskGraphStandardView      给前端编辑和可视化
  -> ContractManifest           给契约预检和运行追溯
  -> TaskGraphRuntimeSpec       给调度器和运行时
  -> RuntimeAssembly            给单节点执行包
```

派生视图可以缓存、展示、诊断，但不能反过来成为隐藏事实源。所有可保存编辑必须回写到 `TaskGraphDefinition` 的规范字段。旧字段只能在迁移期作为输入来源，不能继续成为设计中心。

### 3.2 对象模型

编辑器统一面对以下对象：

| 对象 | 含义 | 主要编辑内容 | 运行含义 |
| --- | --- | --- | --- |
| Graph | 当前任务图 | 全局契约、默认单位、阶段、发布策略 | 一个可运行流程 |
| NodeUnit | 普通节点 | 角色、任务、Agent、输入输出、执行契约 | 一个时序执行位 |
| GraphUnit | 子图节点 | `linked_graph_id`、版本、输入输出端口、交接契约 | 嵌套任务图运行位 |
| ResourceUnit | 资源节点 | 记忆库、产物库、ledger、权限 | 运行上下文资源 |
| ToolUnit | 工具节点 | 工具绑定、输入来源、输出落点 | 可调用程序能力 |
| HumanGateUnit | 人工门 | 审批、暂停、确认策略 | 手动阻塞/放行 |
| Interface | 对象接口 | input/output ports、schema contract | 外部可连接边界 |
| PortEdge | 端口边 | payload、handoff、temporal、memory/artifact refs | 交接协议 |
| TimelineBlock | 时序块 | 阶段、批次、子图来源、入口出口 | 时序分层单元 |
| RuntimePlan | 嵌套运行计划 | 隔离、commit、resume、detach | 子图运行实例计划 |

### 3.3 契约绑定模型

所有对象都支持 `contract_bindings`，但不同对象展示不同分区：

- Graph：`schema`、`unit_batch`、`runtime`、`governance`。
- NodeUnit：`schema`、`execution`、`artifact`、`memory`、`acceptance`、`runtime`。
- GraphUnit：`schema`、`handoff`、`runtime`、`governance`。
- PortEdge：`schema`、`handoff`、`memory`、`artifact`、`temporal`。
- ResourceUnit：`memory`、`artifact`、`governance`。
- ToolUnit：`execution`、`schema`、`acceptance`。

旧字段在迁移期可显示为“历史字段”，但保存时必须写入 `contract_bindings`。后端编译器以 `contract_bindings` 为权威，旧字段只作为短期导入来源。迁移完成后，新建图不再写旧字段。

### 3.4 图节点运行模型

GraphUnit 运行分三层：

1. 父图看到的是一个 Unit，有输入端口、输出端口、handoff contract、版本锚点。
2. 子图内部按自己的 TaskGraphRuntimeSpec 独立运行。
3. 父图只接收子图提交后的 output packet、artifact refs、memory refs 和 summary，不直接读取子图内部临时上下文。

GraphUnit 必须具备：

- `linked_graph_id`
- `version_ref`
- `input_port_id`
- `output_port_id`
- `handoff_contract_id`
- `isolation_policy`
- `commit_policy`
- `resume_policy`
- `detach_policy`

如果缺少这些关键字段，预检应给出明确错误或警告。

### 3.5 时序分层模型

时序系统以图结构为基础，不再作为孤立页存在：

- Graph 有阶段和循环帧。
- TimelineBlock 把若干 NodeUnit 或 GraphUnit 组织成阶段窗口。
- PortEdge 声明交接时机、可见时机、确认时机和传播策略。
- Scheduler 负责把这些时序契约变成 ready / blocked / running / completed 状态。

前端可以有 timeline 页，但对象编辑台必须让用户在选中节点或边时直接看到相关时序契约。

## 4. 编辑器目标形态

### 4.1 页面层级

TaskGraph Studio 作为任务系统内的一级工作区，不单独拆成新的前端应用。原因不是沿用旧页面，而是任务图运行需要直接复用任务域、Agent、契约库、运行记录和产物系统。

一级导航保持分层：

- 蓝图：图的目标、范围、默认策略。
- 拓扑：节点和边的基础结构。
- 组合：Graph / Unit / Interface / PortEdge / NestedRuntime 的主编辑器。
- 时序：阶段、循环、运行顺序、阻塞关系。
- 契约：契约库、manifest、质量门、冲突诊断。
- 记忆与产物：资源库、读写边界、提交策略。
- 风险治理：thread ledger、issue ledger、权限和污染风险。
- 发布运行：执行包、预检、运行监控。

其中“组合”是核心编辑页，其它页是专项工作台。不能把所有层级混成一页。

### 4.2 核心工作台

核心工作台采用四区结构：

```text
左侧：图层关系与对象树
中间：当前层级任务图画布
右侧：对象编辑台
底部：诊断与运行状态抽屉
```

左侧负责：

- 当前图。
- 子图节点。
- 普通节点。
- 资源节点。
- 端口边。
- 时序块。
- 诊断问题。

中间负责：

- 当前图层级画布。
- 图节点可双击进入子图。
- 边可选中并高亮交接方向。
- 节点状态可显示未配置、已配置、预检失败、运行中、已完成。

右侧负责：

- 根据选中对象展示分区表单。
- 每个分区都是契约化配置。
- 支持自然语言说明，但保存为结构化字段。
- 不再让用户跨多个页面找同一个对象的配置。

底部负责：

- 当前对象诊断。
- ContractManifest 差异。
- RuntimeSpec 编译结果。
- 运行状态和最近一次错误。

### 4.3 视觉与交互原则

根据本地 UI/UX 设计系统检索，TaskGraph Studio 应采用工作型 SaaS 控制台风格：

- 扁平、清晰、密度高。
- 主色使用稳重蓝系，强调色用于发布/运行关键动作。
- 使用 Inter 类无衬线字体。
- 避免营销式大卡片、装饰渐变、过度阴影。
- 使用 Lucide 图标表达操作。
- 交互反馈以边框、底色、状态点为主。

重点不是炫，而是让复杂对象能被稳定理解。

## 5. 后端优化计划

### 阶段 A：契约绑定模型落地

目标：让任务图、节点、边都能保存 `contract_bindings`，并和 178 号契约分类一致。

改动点：

- 扩展 TaskGraphDefinition / Node / Edge 数据模型，支持 `contract_bindings`。
- 编写旧字段到 `contract_bindings` 的规范化函数。
- 检测旧字段和新字段冲突，输出 `contract_binding_conflict`。
- 更新 `TaskContractRegistry` / descriptor，让契约类型可被前端筛选。

完成标准：

- 新建任务图保存时能写入 `contract_bindings`。
- 旧图加载时能派生 `contract_bindings` shadow。
- manifest 中能看到 categorized bindings。

### 阶段 B：ContractManifest 升级

目标：让 manifest 成为运行契约权威。

改动点：

- `CompiledNodeContract` 增加 schema / execution / artifact / memory / acceptance / runtime 分类字段。
- `CompiledEdgeHandoffContract` 增加 schema / handoff / temporal / memory / artifact 分类字段。
- `compile_coordination_contract_manifest()` 读取 `contract_bindings` 并保留来源路径。
- `runtime_assembly_builder.py` 从 manifest 读取契约，不再自行拼散落字段。

完成标准：

- 每个节点和边的契约来源可追溯。
- 运行包能显示每项能力来自哪个契约。
- 缺少工具、记忆、产物、验收契约时预检失败。

### 阶段 C：GraphUnit 进入 RuntimeSpec

目标：让图节点从 shadow model 进入可运行调度。

改动点：

- 在 `TaskGraphRuntimeSpec` 中增加 `graph_units` 或 `nested_runtime_plans` 权威字段。
- `compile_task_graph_definition_runtime_spec()` 将 `linked_graph_id` 的 TimelineBlock / GraphUnit 编译为运行节点类型 `graph_unit`。
- 为 graph_unit 生成父图占位节点，执行模式为 `nested_graph_run`。
- 增加 nested run handoff packet：输入包、输出包、artifact refs、memory refs、summary、status。
- Scheduler 识别 `graph_unit`，在子图未完成前阻塞父图下游。

完成标准：

- 父图中一个 GraphUnit 可以触发子图运行。
- 子图完成后父图只接收提交包。
- 子图失败时父图按 failure policy 处理。

### 阶段 D：边时序语义兑现

目标：让边配置不是展示字段，而是调度事实。

改动点：

- Scheduler 严格消费 `wait_policy`、`ack_policy`、`ack_required`、`failure_propagation_policy`、`visibility_timing`。
- 对暂不支持的策略禁用前端选项或标为“预览，不参与运行”。
- PortEdge 与 legacy edge 的映射统一，避免一条边有两个事实源。

完成标准：

- ack 未满足时下游不会错误 ready。
- failure propagation 能按策略 fail / isolate / allow partial。
- 前端可见配置和后端运行诊断一致。

### 阶段 E：发布执行包统一

目标：发布前用户看到同一份执行事实。

改动点：

- 增加 unified execution package API：
  - standard view
  - contract manifest
  - runtime spec
  - selected node runtime assemblies
  - scheduler support report
- 发布页展示“图上配置 -> manifest -> runtime spec -> assembly”的追溯链。
- 预检按对象定位跳回编辑台。

完成标准：

- 用户能看到每个节点最终运行输入、输出、契约、记忆、产物、工具、质量门。
- 不存在配置了但执行包不可见的字段。

## 6. 前端优化计划

目标：拆掉超级 `TaskGraphObjectInspector`，改成按对象类型组合。

- 选中对象后右侧编辑区稳定，不互相污染状态。
- 测试覆盖选中对象到 inspector 的路由。

### 阶段 B：契约分区组件化

建议组件：

### 阶段 C：图节点导航与展开

目标：让“图是节点，节点可展开为图”成为自然操作。

改动点：

- GraphUnit 卡片显示子图中文名、版本、契约、状态。
- 双击 GraphUnit 进入子图工作台。

完成标准：

- 用户能在父图和子图之间自然切换。
- 子图内部编辑不会直接污染父图运行上下文。
- 父图只通过接口和契约连接子图。

### 阶段 D：发布页变成执行包页

目标：发布页不只是按钮和 runtime spec，而是最终验证台。

页面分区：

- 预检总览。
- 执行包概览。
- 契约清单。
- RuntimeSpec。
- 节点 Assembly。
- 子图运行计划。
- 运行监控。

完成标准：

- 点击诊断可定位到对象编辑台。
- 点击节点可查看运行包。
- 点击 GraphUnit 可查看子图执行计划。

### 阶段 E：清理旧页面重复配置

目标：避免一个字段在多个页面各写一份。

原则：

- topology 只负责结构增删连线。
- modules 负责对象级主编辑。
- contracts 负责契约库和 manifest，不再批量重复编辑节点旧字段。
- timeline 负责阶段总览，不重复编辑边的全部 handoff。
- publish 负责编译和运行，不直接修改结构。

完成标准：

- 同一配置只有一个权威编辑入口。
- 其它页面只跳转或展示，不产生隐藏写入。

## 7. 固定执行流

### 7.1 编辑流

```text
选择任务图
  -> 打开组合工作台
  -> 左侧选择 Graph / Unit / PortEdge / TimelineBlock
  -> 右侧编辑契约分区
  -> 保存 TaskGraphDefinition
  -> 刷新 StandardView + Manifest + RuntimeSpec shadow
  -> 展示诊断
```

### 7.2 发布流

```text
保存任务图
  -> 编译 ContractManifest
  -> 编译 TaskGraphRuntimeSpec
  -> 编译 RuntimeAssembly 样本
  -> 运行预检
  -> 用户确认发布
  -> 绑定运行
```

### 7.3 运行流

```text
Scheduler 激活节点
  -> 如果 NodeUnit：按 stage contract 执行动作
  -> 如果 GraphUnit：创建/恢复子图运行
  -> 子图提交 output packet
  -> 父图 PortEdge 传播 packet
  -> 下游节点按 handoff contract 接收
```

- 删除重复编辑入口。
- 删除无用旧测试和旧兼容分支。

## 11. 禁止事项

- 禁止为写作任务单独开后端小灶。
- 禁止前端配置了但后端运行不消费还显示为已生效。
- 禁止 standard view 成为隐藏事实源。
- 禁止同一字段多个页面各写一份。
- 禁止用 Prompt 文案代替结构化契约。
- 禁止用伪产物或伪诊断绕过测试。
- 禁止保留无用旧残留代码作为“兼容”。

## 12. 推荐实施顺序

1. 先实现 `contract_bindings` 数据模型与编译器规范化。
2. 再拆分前端对象编辑台，让配置自然落到 `contract_bindings`。
3. 接着把 GraphUnit / nested runtime 编译进 RuntimeSpec。
4. 然后兑现边时序语义和 Scheduler 支持。
5. 最后升级发布页为统一执行包验证台，并清理旧入口。

这个顺序不能反过来。否则前端会先做出新界面，但后端仍以旧字段运行，继续制造“看起来配置了，实际没生效”的不信任感。

## 13. 最终验收标准

任务图编辑器完成后，应满足：

- 图是任务图，图节点也是任务图，普通节点是时序执行位。
- 所有对象都有接口和契约边界。
- 选中任意对象都能在右侧自然配置。
- 保存结果能进入 `contract_bindings`。
- ContractManifest 能解释每项配置来源。
- RuntimeSpec 能执行 GraphUnit 和 NodeUnit。
- Scheduler 能兑现时序、ack、失败传播。
- 发布页能展示最终执行包。
- 写作任务只是一个实例，不需要任何后端专用逻辑。

到这个状态，任务图编辑器才真正从“能画图”变成“能编排长任务运行”的通用架构。

### 14.1 两个最先补的结构闭环

优先级必须先做这两个：

1. `contract_bindings` 闭环  
   图、节点、边的所有契约配置都进入 `contract_bindings`，再由 `ContractManifest` 编译为运行可追溯事实。没有这个闭环，前端配置越多，后端越难判断来源。

2. `GraphUnit -> RuntimeSpec -> Scheduler` 闭环  
   图节点必须从 standard view 进入 `TaskGraphRuntimeSpec`，再由 scheduler 真正执行或阻塞。没有这个闭环，编辑器只能“看起来能组合”，实际仍是“能画但跑不准”。

### 14.2 暂不应该先做的事

- 不应先美化页面。
- 不应先扩充写作任务专用配置。
- 不应先继续给 `TaskGraphObjectInspector` 增加字段。
- 不应先让前端出现更多后端不消费的选项。
- 不应为了兼容旧字段而推迟 `contract_bindings` 切换。

这几个审查问题已经全部进入本计划的执行阶段和验收标准。后续实施时，每完成一个阶段都应回到本表逐项打勾。

### 15.2 已完成：GraphUnit 运行句柄与子图启动闭环

本轮补的是 GraphUnit 的运行实例入口，不补写作任务私门，也不把子图完成语义提前伪装出来。

运行协议固定为：

1. 父图 LangGraph runtime 只负责识别当前 stage 是否为 `graph_unit`，并产出标准 `NodeExecutionRequest`。
2. `NodeExecutionRequest.executor_type` 必须为 `graph_unit`，其 `executor_binding` 和 `runtime_assembly.graph_unit_runtime_handle` 必须包含：
   - `linked_graph_id`
   - `nested_runtime_plan_id`
   - `version_ref`
   - `handoff_contract_id`
   - `input_port_id`
   - `output_port_id`
   - `isolation_policy`
   - `visibility_policy`
   - `detach_policy`
   - 父图 `coordination_run_id`、`root_task_run_id`、`stage_id`、`node_id`
3. API scheduler 识别 `executor_type=graph_unit` 后，不进入普通 Agent continuation，而是通过 `TaskRunLoop.start_task_graph_run()` 创建或复用子图 TaskGraphRun。
4. 子图 TaskGraphRun 诊断里必须记录父图调度身份：
   - `parent_coordination_run_id`
   - `parent_root_task_run_id`
   - `parent_stage_id`
   - `parent_stage_request_id`
   - `parent_stage_idempotency_key`
5. 同一个父图 stage 的同一个 idempotency key 再次调度时，必须复用已有子图 run，不能重复启动。
6. 子图启动后可以生成自己的初始 `stage_execution_request`；是否继续自动调度子图首节点，由 GraphUnit 的通用 executor policy 控制。

实现结果：

- `LangGraphCoordinationRuntimeResult.continuation_payload()` 已能识别 `executor_type=graph_unit`，返回 `requires_graph_unit_executor` 和 `graph_unit_runtime_handle`，不进入普通 Agent continuation。
- `_execute_stage_request_in_background()` 识别 GraphUnit stage 后进入 `_start_graph_unit_stage_request()`。
- `_start_graph_unit_stage_request()` 会：
  - 从 `runtime_assembly.graph_unit_runtime_handle` 或 `executor_binding.graph_unit_runtime_handle` 读取子图运行句柄。
  - 校验 `linked_graph_id` 存在且子图已发布。
  - 为子图编译 `TaskGraphRuntimeSpec`，阻塞错误不会被绕过。
  - 通过 `TaskRunLoop.start_task_graph_run()` 创建子图 TaskGraphRun。
  - 把父图调度身份、stage request、idempotency key、GraphUnit handle 写入子图 TaskRun diagnostics。
  - 按 `executor_policy.auto_start_child_initial_stage` 决定是否自动调度子图首节点。
- `_schedule_stage_execution_background()` 会先通过 `_matching_stage_execution_task_run()` 查找同一父图 stage / idempotency key 的有效 TaskRun，避免重复启动同源子图。
- 该闭环只处理通用任务图和 GraphUnit 运行语义，没有写作、章节、正文质量等专用后门。

验证：

- `python -m pytest backend/tests/task_system_api_regression.py::test_graph_unit_stage_scheduler_starts_and_reuses_child_task_graph_run -q`
- `python -m pytest backend/tests/task_graph_scheduler_regression.py backend/tests/task_system_api_regression.py backend/tests/langgraph_coordination_runtime_regression.py backend/tests/node_execution_request_regression.py backend/tests/contract_compiler_coordination_test.py backend/tests/runtime_assembly_builder_test.py -q`

### 15.3 已完成：GraphUnit committed output packet 与父图续跑

本轮补 GraphUnit 子图完成后的通用交接闭环。这个闭环仍然只处理任务图语义，不引入写作、章节、质量门等专用逻辑。

核心协议：

1. 父图当前 stage 必须仍是 `executor_type=graph_unit` 的 active stage，且子图 TaskGraphRun 必须带有父图调度身份：
   - `graph_unit_child_run=true`
   - `parent_coordination_run_id`
   - `parent_root_task_run_id`
   - `parent_stage_id`
   - `parent_stage_request_id`
   - `parent_stage_idempotency_key`
2. 子图完成信号来自通用运行对象，而不是领域文本：
   - 子图 CoordinationRun 状态为 `completed`，或
   - 子图 LangGraph checkpoint 的 `terminal_status=completed`，或
   - 子图 latest CoordinationMergeResult `accepted=true`。
3. 父图不直接读取子图私有上下文，只接收一个 `orchestration.graph_unit_committed_output_packet`：
   - 父图身份：`parent_coordination_run_id`、`parent_stage_id`、`parent_node_id`、`parent_stage_request_id`。
   - 子图身份：`child_task_run_id`、`child_coordination_run_id`、`linked_graph_id`。
   - 输出引用：`artifact_refs`、`output_refs`、`result_refs`、`final_result_ref`、`merge_result_ref`。
   - 可见性与提交策略：继承 GraphUnit handle 的 `visibility_policy`、`isolation_policy`、`detach_policy`、`handoff_contract_id`、`output_port_id`。
   - 子图阶段摘要：只放 stage result summary，不把子图完整上下文直接污染父图。
4. output packet 存入 runtime object store，父图 `NodeResultReadyEvent.task_result_ref` 指向 packet ref。
5. 父图 `resume_from_task_result()` 只消费 packet/ref/summary，不消费子图 checkpoint 原文。
6. 桥接成功后，在子图 TaskRun diagnostics 中记录：
   - `graph_unit_output_packet_committed`
   - `packet_ref`
   - `parent_coordination_run_id`
   - `parent_stage_id`
   - `committed_at`
7. 幂等规则：
   - 父图同一 active stage 已经消费同一个 child task run 时，不重复提交。
   - 子图已经记录同一父图 stage 的 output packet committed 时，不重复提交。
   - 如果父图 stage 被 rewind，则必须依靠 rewind invalidation 或新的 stage idempotency key 重新进入子图运行/提交。

验收标准：

- 子图完成后，父图 GraphUnit stage 能收到 `NodeResultReadyEvent` 并标记 completed。
- 父图下游由现有 scheduler 自然 release，不能由 GraphUnit 桥接绕过调度器直接指定。
- 重复调用 continue-current-stage 不会重复提交同一个 child output packet。
- 子图未完成时，父图不会误释放。
- output packet 不包含写作专用字段，所有批次、质量、产物要求都必须继续来自通用契约与节点配置。

实现结果：

- `_latest_unconsumed_graph_unit_child_result()` 只接受满足以下条件的子图 TaskRun：
  - `graph_unit_child_run=true`。
  - `parent_coordination_run_id` 与当前父图一致。
  - `parent_stage_id` 与当前 active stage 一致。
  - `parent_stage_request_id` / `parent_stage_idempotency_key` 与当前 request 匹配。
  - 未被当前父图 stage 消费过，也没有已有 committed packet 记录。
- 子图完成判断来自通用运行态：
  - 子图 checkpoint 的 `terminal_status=completed`。
  - 子图 CoordinationRun 状态。
  - 最新 CoordinationMergeResult 的 `accepted=true`。

- 父图下游由现有 scheduler 自然 release；GraphUnit 桥接不直接指定下游节点。

### 15.4 已完成：边级失败传播进入通用 Scheduler

本轮补的是 `failure_propagation_policy` 的真实运行语义。它必须服务所有任务图，不服务某个写作场景；GraphUnit 子图失败、普通节点失败、工具节点失败都应进入同一个失败传播模型。

核心协议：

1. 失败来源只来自运行态节点状态：
   - `node_statuses[source]=failed`。
   - GraphUnit 子图失败在父图中也只表现为 GraphUnit stage failed，后续传播仍由边策略决定。
2. 边级策略由 `TaskGraphRuntimeEdge.failure_propagation_policy` 统一决定：
   - `fail_downstream`：源节点失败会把目标节点标记为 `failed`，并沿后续 `fail_downstream` 边继续传播。
   - `isolate_failure`：源节点失败被隔离在该边，目标节点保持 pending/blocked，不被自动失败。
   - `allow_partial`：源节点失败不自动失败目标节点，目标节点必须依靠自身 `join_policy=allow_partial_with_issues` 或 `coordinator_decides` 在其他上游完成后继续。
   - `coordinator_decides`：与 `allow_partial` 一样不自动释放；由目标节点或后续协调节点的 join policy 决定是否继续。
3. Scheduler 需要区分原始状态与有效状态：
   - 原始 `node_statuses` 仍记录执行器实际结果。
   - 有效状态在调度评估时叠加边传播结果，用于 ready/blocked/failed/terminal 判断。
4. 失败传播不生成业务输出、不补产物、不伪造验收结果；它只改变调度状态。
5. 传播链必须可诊断：
   - `failure_propagation` 列出每个被传播失败的节点、来源边、来源节点和策略。
   - 节点 blocked reasons 中保留 `upstream_failed:<source>`，帮助前端解释为何等待。
6. LangGraph runtime 的 `_route_next` 必须消费 Scheduler 返回的有效 `node_statuses`，否则 GraphUnit 失败或普通失败会停留在影子诊断里，父图运行不会真正收敛。

验收标准：

- `fail_downstream` 会让直接下游及其 fail-downstream 链路进入 failed。
- `isolate_failure` 不会错误释放下游，也不会把下游自动标成 failed。
- `allow_partial` / `coordinator_decides` 与目标节点 `join_policy` 协同：有成功上游且所有上游已终态时可进入 ready，没有成功上游时仍 blocked。
- 编译器 support matrix 不再把上述失败传播策略标成 unsupported。
- 运行层 `continue-current-stage` 看到子图 failed 时，父图最终状态由同一个 scheduler 失败传播机制决定。

实现结果：

- `TaskGraphSchedulerState` 现在先根据边级失败传播策略计算有效节点状态，再生成 ready / blocked / failed / terminal 结果。
- `fail_downstream` 会沿正向业务边传播失败；`isolate_failure`、`allow_partial`、`coordinator_decides` 不会错误释放下游。
- 失败传播不再依赖 runtime node 编译顺序，避免 GraphUnit 运行节点被追加到普通节点之后时被误判为 backward edge。
- LangGraph runtime 的 `_route_next` 已消费 scheduler 返回的有效 `node_statuses`，运行态和诊断态保持一致。
- GraphUnit 子图 `failed` / `blocked` / `waiting_for_human` 会提交通用 failure packet，并以 `accepted=false` 的 `NodeResultReadyEvent` 回到父图；父图后续由同一个 scheduler 失败传播规则决定。
- RuntimeSpec support matrix 已把 `fail_downstream`、`isolate_failure`、`allow_partial`、`coordinator_decides` 标为 supported。

### 15.5 已完成：contract_bindings 权威化与旧字段导入协议

### 16.1 已经形成闭环的能力

| 闭环 | 当前状态 | 证据 |
| --- | --- | --- |
| `contract_bindings` 权威化 | 已完成 | 后端 parser、validator、manifest、前端 save mapper 和契约显示均改为绑定优先 |
| 对象编辑台拆分 | 已完成首轮 | NodeUnit、PortEdge、GraphUnit、TimelineBlock、NestedRuntime、Root/Utility panels 已从主 inspector 拆出 |
| GraphUnit 编译进入运行事实 | 已完成编译和发布前验证 | RuntimeSpec、NestedRuntimePlan、GraphUnit execution plan、manifest handoff、object trace index 均可追溯 |
| GraphUnit 真实子图启动 | 已完成 | GraphUnit stage 可创建/复用子图 TaskGraphRun，并记录父图调度身份和 idempotency key |
| GraphUnit output/failure packet 回填父图 | 已完成 | 子图完成或失败后生成通用 packet，父图通过 `resume_from_task_result()` 消费，后续由 scheduler 释放或失败传播 |
| 边失败传播 | 已完成 | Scheduler 支持 `fail_downstream`、`isolate_failure`、`allow_partial`、`coordinator_decides` |
| 边时序支持矩阵 | 已完成第一轮 | RuntimeSpec diagnostics 和前端边配置入口统一展示 `supported` / `partial` / `unsupported`，未支持策略默认禁用 |
| 发布执行包 | 已完成 | 执行包合成 standard view、ContractManifest、RuntimeSpec、Scheduler shadow、RuntimeAssembly、GraphUnit 子图计划 |
| 对象追溯索引 | 已完成 | 发布页可看到图上对象到 runtime / manifest / scheduler / assembly 的落点 |

这些闭环共同解决了早期审查中最关键的不可信问题：前端配置不再只是画面配置，而是能进入后端编译、契约清单、运行规范和发布前执行包。

### 16.2 尚未完全收尾的闭环

| 闭环 | 状态 | 下一步实施边界 |
| --- | --- | --- |
| 旧字段彻底清理 | 待切换期后实施 | 等现有图迁移完成后删除无用旧字段写入、重复测试和重复页面入口 |
| 边时序语义全量兑现 | 部分完成 | 支持矩阵已治理；timeline group、phase event、manual release、模型输入可见性分层等真实运行行为仍需逐项落地 |
| 执行包与右侧编辑台联动 | 待实施 | 选中画布对象时，右侧编辑台同时展示 `object_trace_index` 的真实运行落点 |

GraphUnit 的 P0 运行闭环已经完成，边时序第一轮 P0 风险也已经从“裸配置”降级为“支持度可见且未支持选项受控”。后续不能再把 GraphUnit 描述为 shadow-only 或 compile-only 能力，也不能把边时序描述成完全未治理。剩余风险转移到边时序深层运行兑现、前端追溯联动、旧字段清理和更完整的浏览器级验证。

### 16.3 当前剩余工作优先级

## 17. 收尾验收矩阵

| 原始问题 | 本轮是否收尾 | 说明 |
| --- | --- | --- |
| GraphUnit 只是 shadow model | 已收尾 | 已进入 RuntimeSpec、manifest、执行包和追溯索引，并已支持真实子图启动、output/failure packet 提交与父图续跑 |
| 契约散落在旧字段 | 已收尾 | `contract_bindings` 成为权威模型，旧字段只做迁移 fallback 和冲突诊断 |
| 对象编辑台过重 | 已完成首轮收尾 | 主要对象 inspector 已拆分，后续是体验打磨和重复入口清理 |
| 边时序语义不完全兑现 | 已完成第一轮治理 | 支持矩阵、诊断、边编辑器和时序页入口已对齐；timeline group、phase event、manual release 等深层运行语义仍需继续实施 |

本轮计划书的完成标准不是声称整个任务图系统已经最终完成，而是把“已经落地的结构闭环”和“仍需治理的体验/策略闭环”清楚分开。这样后续继续实施时不会在写作专用逻辑、旧字段兼容、前端假配置之间来回摇摆。

## 18. 下一轮执行顺序

下一轮不应继续扩写 GraphUnit 计划，也不应先做纯视觉优化。应按以下顺序治理剩余闭环：

1. 选择支持矩阵中仍为 `partial` 的边时序策略，逐项实现真实运行语义。
2. 对每个新增 supported 策略补 scheduler/runtime 回归，并更新前端支持度模块。
3. 将 `object_trace_index` 接入右侧对象编辑台和底部诊断抽屉。
4. 收敛 `contracts`、`timeline`、`memory`、`publish` 专项页的重复写入口，保留总览、诊断和跳转职责。
5. 在迁移安全后清理旧字段同步写入、旧测试和旧页面残留。
6. 做浏览器级工作台验证，确认三栏工作台、发布执行包、追溯索引和边时序支持标签的真实可用性。

这条顺序必须保持，因为 GraphUnit 已经能真实运行，接下来要防止边时序和前端配置再次出现“看起来配置了，运行时不一定兑现”的旧问题。

## 19. 计划书最终结论

本计划书已经完成阶段性收尾：

- 架构目标已经明确：任务图是一等对象，契约是一等协议，图节点必须能进入运行事实，边是交接和时序结构。
- 关键 P0/P1 问题已经有明确方案，并且大部分已落地到代码和测试。
- GraphUnit 的真实子图运行闭环已经落地：编译、发布前验证、契约清单、追溯、子图启动/复用、output/failure packet、父图续跑和失败传播都已有回归覆盖。
- 边时序已经完成第一轮治理：运行支持度进入 RuntimeSpec diagnostics，前端边编辑入口统一显示支持状态，未支持策略不再被当作可安全运行配置。
- 写作任务不会获得后端私门。章节数量、批次、质量、产物、验收都必须通过通用 `unit_batch`、`execution`、`artifact`、`acceptance`、`runtime` 等契约与工具/skill 配置表达。

因此，179 号计划书可以从“运行闭环阶段”进入“策略兑现、编辑器体验联动和迁移清理阶段”。后续任何实现都应回到本计划书的对象模型、契约模型、发布执行包和禁止事项逐项对照。

# 三阶段任务图拆分

状态：设计稿，未进入配置

## 1. 核心判断

不同阶段应该拆成不同的任务图，而不是把设计、创作、收尾硬塞进同一个大图。

最合适的方式是三张正式任务图：

1. 设计阶段任务图
2. 正式创作阶段任务图
3. 收尾阶段任务图

它们都属于正式流程，但各自时序、职责、协议和产物不同。

## 2. 为什么要拆

### 2.1 设计阶段和创作阶段关注点不同

设计阶段关注的是：

- 世界观是否成立
- 人设是否成立
- 剧情和伏笔是否成立
- 大纲是否足够细
- 各设计要素是否对齐

创作阶段关注的是：

- 当前批次怎么写
- 前文怎么接
- 章节怎么递进
- 审稿怎么返修
- 章节怎么提交

收尾阶段关注的是：

- 整卷是否收束
- 可改动库是否吸收
- 全书是否可交付

### 2.2 一个大图会太重

如果把三类阶段放进同一张任务图，会出现：

- 时序冲突
- 节点职责混淆
- 前端展示过重
- 运行切换不清晰
- 审核边界不稳定

所以应该拆图，不是拆碎能力。

## 3. 三张正式任务图

### 3.1 设计阶段任务图

用途：

- 从世界观审核后开始
- 产出人设设计、剧情设计、设计对齐和大纲汇总
- 最后形成可进入正式写作的设计定稿

推荐流程：

```text
world_review
  -> character_design
  -> plot_design
  -> design_sync
  -> outline_design
  -> outline_review
  -> baseline_memory_seed
```

设计阶段的正式产物：

- 人设设计稿
- 剧情设计稿
- 设计对齐包
- 大纲总结版
- 冻结与可调整边界

### 3.2 正式创作阶段任务图

用途：

- 进入正式写作后使用
- 生成章节细纲、正文、审核、提交、推进

推荐流程：

```text
volume_plan
  -> chapter_outline
  -> chapter_draft
  -> chapter_review
  -> memory_commit_chapter
  -> chapter_progress_router
```

创作阶段的正式产物：

- 当前卷计划
- 当前批次章纲
- 章节正文
- 章节审核结果
- 章节提交
- 章节推进决策

### 3.3 收尾阶段任务图

用途：

- 卷级收束
- 全书收口
- 动态记忆吸收
- 最终交付

推荐流程：

```text
volume_review
  -> volume_commit
  -> volume_postmortem
  -> world_outline_extension_proposal
  -> extension_review
  -> extension_commit
  -> next_volume_router
  -> final_assemble
  -> final_review
  -> memory_finalize
```

收尾阶段的正式产物：

- 卷级裁决
- 卷级记忆
- 卷后复盘
- 可改动库更新
- 最终交付包

## 4. 三图之间怎么衔接

### 4.1 设计图 -> 创作图

设计图完成后，不是直接进入章节，而是先把设计定稿写成可供创作图读取的稳定输入：

- 基准库
- 动态记忆初始范围
- 线程账本
- 产物索引
- 冻结边界

创作图只读取这些定稿输入，不再读取设计讨论过程。

### 4.2 创作图 -> 收尾图

创作图每轮完成后，会产出：

- 章节提交
- 批次摘要
- 卷级累计信息
- 问题账本
- 线程状态

这些结果进入收尾图，作为卷审、卷后复盘和可改动库提案的输入。

### 4.3 收尾图 -> 下一轮设计或创作

收尾图的输出可能进入两种方向：

- 下一卷继续创作
- 返回设计层做结构修订

也就是说，三张图不是一次性的直线，而是带回路的阶段闭环。

## 5. 每张图都要有自己的协议

### 设计阶段协议

- `character_architecture_protocol`
- `plot_architecture_protocol`
- `design_sync_protocol`
- `outline_architecture_protocol`
- `startup_memory_partition_protocol`

### 创作阶段协议

- `continuity_protocol`
- `thread_architecture_protocol`
- `revision_protocol`
- `chapter_execution_protocol`
- `memory_commit_protocol`

### 收尾阶段协议

- `volume_review_protocol`
- `postmortem_protocol`
- `extension_protocol`
- `final_assembly_protocol`
- `delivery_protocol`

协议不能跨阶段混用。

## 6. 前端展示建议

前端最好把三张图分成三个入口：

- 设计图
- 创作图
- 收尾图

每张图内部再按节点层级展开，不把三个阶段放在一张图里滚动查看。

## 7. 结论

你的这个想法很适合当前系统。

我们应该把任务图拆成：

1. 设计阶段任务图
2. 正式创作阶段任务图
3. 收尾阶段任务图

这样既保留正式流程，也能让每个阶段的时序更清楚、职责更稳定、协议更好复用。

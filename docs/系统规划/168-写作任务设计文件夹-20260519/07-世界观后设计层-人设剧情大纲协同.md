# 世界观后设计层：人设、剧情、大纲协同

状态：设计稿，未进入配置

## 1. 这一层解决什么

世界观审核通过以后，不能立刻把所有细节压给大纲。中间最好插入一个正式设计层，把“人物怎么长出来”和“剧情怎么跑起来”先设计扎实，再交给大纲做统一收束。

这层的目标是：

1. 让人设和剧情各自先成形。
2. 让伏笔和关键剧情先在剧情侧落细。
3. 让大纲只负责统筹、排序、压缩、统一，不负责从零补全所有细节。
4. 让后续正式写作拿到的是一份已经讨论到位的大纲总结版。

## 2. 推荐结构

```text
world_review
  -> character_design
  -> plot_design
  -> design_sync
  -> outline_design
  -> outline_review
  -> baseline_memory_seed
```

这里的关键不是节点多少，而是职责分离：

- `character_design` 负责角色和关系
- `plot_design` 负责剧情和伏笔
- `design_sync` 负责对齐冲突
- `outline_design` 负责形成统一大纲

## 3. 为什么不直接交给大纲

如果让大纲同时承担人设、剧情、伏笔、节奏、关系、世界承诺的第一次总汇总，会有三个问题：

1. 大纲会过重，变成“什么都管”的总容器。
2. 细节容易被压扁，人物关系和伏笔会被抽象成几句总述。
3. 后续写手拿到的大纲会“看起来完整”，但真正能写的细节不足。

所以更好的方式是：

- 先分别设计
- 再讨论对齐
- 最后交给大纲整合

## 4. 人设设计师

建议节点 ID：`character_design`

职责：

- 设计主角、核心配角、关键对手、功能性角色。
- 设计人物关系网、角色动机、角色冲突位、人物成长轴。
- 产出角色之间的稳定关系和可调整关系。

必须输出：

- 角色清单
- 身份信息
- 核心动机
- 关系图
- 冻结事实
- 可调整关系
- 角色推进节奏建议

不能做：

- 不能直接把人物关系写成最终大纲。
- 不能改写世界观主干。
- 不能替代剧情设计师安排事件链。

## 5. 剧情设计师

建议节点 ID：`plot_design`

职责：

- 设计主线剧情、支线剧情、关键事件、伏笔、回收窗口。
- 把世界观和人设能否推动剧情写清楚。
- 先把“故事会怎么跑”设计出来。

必须输出：

- 主线剧情骨架
- 关键事件链
- 伏笔清单
- 回收窗口
- 节奏建议
- 冲突升级路径
- 不可提前泄露内容

不能做：

- 不能直接承担大纲总汇总职责。
- 不能把剧情细节写成已经定稿的大纲。
- 不能无视人设关系约束。

## 6. 讨论机制

这里需要一个正式讨论机制，但它不是无限讨论。

建议使用 `design_sync`，而不是让大纲自己去猜你设和剧情的冲突。

### `design_sync` 的职责

- 收集人设设计稿和剧情设计稿。
- 检查冲突项。
- 检查是否有关系矛盾、动机矛盾、事件矛盾、伏笔矛盾。
- 给出统一取舍。
- 产出一份可给大纲直接使用的设计对齐包。

### `design_sync` 的原则

1. 只做结构对齐，不做长篇重写。
2. 只处理冲突点，不重写全部内容。
3. 只允许有限轮讨论，避免无限拉扯。
4. 讨论结束后必须冻结成设计对齐包。

## 7. 谁来汇总

你的问题里有两个选项：

1. 先把人设和剧情都汇总，再交给大纲。
2. 让大纲自己来汇总。

我的建议是：**先讨论到位，再由大纲汇总**。

原因是：

- 人设和剧情本来就属于两个不同的设计视角。
- 如果直接交给大纲，大纲会承担过多隐性协调工作。
- 如果先对齐，再汇总，大纲拿到的是收口后的结构化输入，更容易写细、写稳。

所以最稳的是三段式：

```text
人设设计
  -> 剧情设计
  -> 设计对齐
  -> 大纲汇总
```

## 8. 大纲设计师的职责

建议节点 ID：`outline_design`

职责：

- 读取人设设计、剧情设计和设计对齐包。
- 把角色、事件、伏笔、节奏整合成全书主线和卷级框架。
- 输出可进入正式写作阶段的大纲总结版。

大纲设计师应该做的是：

- 汇总
- 排序
- 压缩
- 统一
- 冻结

大纲设计师不应该做的是：

- 从零补人物关系
- 从零补伏笔事件
- 重新发明人物动机

## 9. 推荐正式流程

```text
world_review
  -> character_design
  -> plot_design
  -> design_sync
  -> outline_design
  -> outline_review
  -> baseline_memory_seed
  -> volume_plan
```

这样处理以后：

- 世界观先定边界
- 人设和剧情先分别做细
- 再通过讨论机制对齐
- 最后交给大纲统一收束
- 然后进入正式写作

## 10. 结论

你的这个判断是对的：

- 世界观后面可以加设计层
- 人设设计师和剧情设计师可以并行
- 大纲设计师负责汇总
- 但在大纲之前，最好加一个讨论对齐机制

也就是说，不是“谁来汇总都行”，而是：

**先分别设计，再对齐讨论，最后由大纲统一收口。**

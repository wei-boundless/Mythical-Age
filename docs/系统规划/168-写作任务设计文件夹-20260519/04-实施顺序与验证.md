# 实施顺序与验证

## 1. 实施顺序

### 阶段 0

冻结现有写作任务的真实链路、仓库和中文命名来源。

### 阶段 1

先落 `01-记忆与协议.md` 里的协议和仓库定义。

### 阶段 2

再按 `02-节点细则总表.md` 补齐每个节点的输入、输出、禁区和交接。

### 阶段 3

把中文名注册和图层切换规则接到前端。

### 阶段 4

最后才做配置层落地和旧语义清理。

## 2. 切换原则

- 新设计先并行可读，旧流程只允许做迁移参考。
- 一旦新细则成为主线，就不要再沿用旧兼容残留来定义新行为。
- 基准库修改必须保留正式提交链路，不允许跳过审核。

## 3. 验证矩阵

### 连续性

- 断开后可从最近一次 `accepted` / `committed` 结果恢复。
- 下一节点只看接续包，不看整段聊天历史。

### 记忆

- 基准库默认只读。
- 动态库只能吸收审核通过的增量。
- 候选稿不能直接污染下游记忆。

### 节点

- 每个节点都能给出明确的输入、输出和禁区。
- 返修要回到原稿，不是重新另起一份。

### 前端

- 中文名能在拓扑、职责、时间线、监测、记忆各层一致显示。
- 不同层级页面不混在同一视图里。

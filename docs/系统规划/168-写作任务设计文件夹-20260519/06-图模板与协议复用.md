# 图模板与协议复用

状态：设计稿，未进入配置

## 1. 核心判断

完整流程应该画出来，但它不是一条统一时序的单链路，而是一个正式流程目录。目录下面会有多个按时序分型的图模板。

正确做法是：

- 正式流程图用于理解和审查。
- 运行图模板用于执行。
- 协议包用于复用。

也就是说，人设、大纲、伏笔、分库这些能力要被标准化，但它们在不同阶段的时序不同，不能硬压成同一条拓扑。

## 2. 两层图

### 2.1 完整理解图

完整理解图展示所有需要被考虑的创作结构。

```text
用户输入要求
  -> 项目启动包
  -> 世界观规划
  -> 世界观审核
  -> 世界观提交
  -> 分库保存
  -> 人设架构初始化
  -> 大纲架构初始化
  -> 伏笔脉络初始化
  -> 创作架构审核
  -> 创作架构提交
  -> 分卷规划
```

这张图的用途：

- 帮助用户理解系统为什么需要这些结构。
- 帮助设计者检查有没有漏掉关键责任。
- 帮助前端解释某个阶段内部到底处理了什么。

这张图不是运行时必须逐节点执行的图。

### 2.2 运行模板图

运行模板图展示真实 TaskGraph 应该执行的闭环。

```text
initializer
  -> reviewer
  -> committer
  -> next_stage_router
```

起步阶段可以实例化为：

```text
startup_architecture_seed
  -> startup_architecture_review
  -> startup_architecture_commit
  -> volume_plan
```

卷后阶段可以实例化为：

```text
volume_postmortem
  -> extension_review
  -> extension_commit
  -> next_volume_router
```

章节阶段可以实例化为：

```text
chapter_outline
  -> chapter_draft
  -> chapter_review
  -> memory_commit_chapter
  -> chapter_progress_router
```

### 2.3 时序差异是常态

不同图模板的时序本来就不同：

- 起步模板先定世界主干，再做创作架构，再进入分卷。
- 章节模板先出细纲，再写正文，再审稿，再提交，再路由。
- 卷后模板先复盘，再提案，再审核，再提交，再决定下一卷。
- 收尾模板先整编，再终审，再归档。

所以“可以创建图模板”不等于“所有阶段共用同一时序”。

## 3. 模板的本质

模板不是一串固定业务节点，而是一种可复用闭环：

```text
生产候选
  -> 审核裁决
  -> 提交入库
  -> 路由下一步
```

每个阶段的区别不在是否有模板，而在模板时序和协议包。

## 4. 协议包

### 4.1 人设协议

协议 ID：`character_architecture_protocol`

用途：

- 起步阶段：初始化人物骨架。
- 章节阶段：提供当前人物状态切片。
- 卷后阶段：判断哪些人物变化已坐实。
- 修订阶段：检查是否改写了冻结人物事实。

### 4.2 大纲协议

协议 ID：`outline_architecture_protocol`

用途：

- 起步阶段：建立全书主线和可调整支线池。
- 分卷阶段：拆卷目标和章节窗口。
- 章节阶段：约束当前批次章纲。
- 修订阶段：声明失效范围和新版本生效点。

### 4.3 伏笔协议

协议 ID：`thread_architecture_protocol`

用途：

- 起步阶段：登记初始伏笔和长期悬念。
- 章节阶段：切出本批要继承、推进、回收的线程。
- 审核阶段：检查伏笔有没有无证据消失。
- 卷后阶段：迁移、延期、关闭或升级线程。

### 4.4 分库协议

协议 ID：`memory_partition_protocol`

用途：

- 判断内容进入基准库、动态记忆库、产物索引、问题账本还是线程账本。
- 防止候选稿和审核意见污染 canon。
- 给每次提交形成可审计入库依据。

### 4.5 接续协议

协议 ID：`continuity_protocol`

用途：

- 生成 `continuity_packet`。
- 说明上一阶段结尾状态、人物状态、世界状态、未闭合线程。
- 防止节点之间靠聊天历史接力。

### 4.6 修订协议

协议 ID：`revision_protocol`

用途：

- 生成 `revision_packet`。
- 确认原稿、审核意见、保留项、必改项和禁止改动范围。
- 防止返修变成另起一稿。

## 5. 起步阶段推荐压缩形态

起步阶段真实运行图建议是：

```text
project_brief
  -> world_design
  -> world_review
  -> memory_commit_world
  -> startup_architecture_seed
  -> startup_architecture_review
  -> startup_architecture_commit
  -> volume_plan
```

其中 `startup_architecture_seed` 内部调用：

- `character_architecture_protocol`
- `outline_architecture_protocol`
- `thread_architecture_protocol`
- `memory_partition_protocol`

`startup_architecture_review` 内部调用：

- 人设一致性检查
- 大纲边界检查
- 伏笔生命周期检查
- 分库污染检查

`startup_architecture_commit` 内部调用：

- 基准库写入协议
- 动态记忆库写入协议
- 线程账本写入协议
- 产物索引写入协议
- 问题账本写入协议

## 6. 前端展示方式

前端应该同时支持两种视图：

### 完整理解视图

展示用户容易理解的完整流程：

- 用户输入
- 世界观
- 人设
- 大纲
- 伏笔
- 分库
- 审核
- 提交

这适合做解释、设计审查和节点详情说明。

### 运行模板视图

展示真实执行节点：

- 初始化
- 审核
- 提交
- 路由

这适合做运行监控、失败定位和配置编辑。

前端不能把两者混成同一张无层级大图。应该用卡片切换：

- “完整理解流程”
- “运行模板”
- “协议包”
- “入库结果”

## 7. 配置原则

1. 节点按责任边界拆，不按信息类别拆。
2. 协议按能力复用，不按阶段复制。
3. 同一个协议在不同阶段拥有不同输入范围和写入权限。
4. 审核和提交必须保留为显式边界。
5. 分库必须在提交阶段完成，不能在生产阶段悄悄写入。
6. 完整流程图可以复杂，运行图必须克制。

## 8. 最小实现建议

后续配置时，优先新增或强化三个起步节点：

- `startup_architecture_seed`
- `startup_architecture_review`
- `startup_architecture_commit`

不要先拆：

- `character_architecture_seed`
- `outline_architecture_seed`
- `thread_architecture_seed`
- `startup_memory_partition`

这些先作为协议包存在。等运行证明某个协议需要独立审计、独立重试或独立权限时，再升级为真实节点。

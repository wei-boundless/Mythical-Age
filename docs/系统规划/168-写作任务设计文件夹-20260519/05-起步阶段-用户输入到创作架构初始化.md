# 起步阶段：用户输入到创作架构初始化

状态：设计稿，未进入配置  
范围：只讨论小说任务开始阶段，不进入章节循环

## 1. 这一阶段要解决什么

起步阶段不是“让模型先写一份世界观”这么简单。它要完成的是：把用户输入转成一个可长期运行的创作工程底盘。

这一阶段必须回答六个问题：

1. 用户到底要什么，哪些是硬要求，哪些只是风格偏好。
2. 世界观主干是否足够稳定，可以作为全书 canon 的来源。
3. 哪些内容要进入基准库，哪些只能进入动态记忆库，哪些只是产物索引。
4. 人设、关系、大纲、伏笔这些创作架构是否已经有初始骨架。
5. 这些骨架哪些是冻结事实，哪些只是可调整计划。
6. 后续分卷和章节节点应该读取哪一份启动架构，而不是重新解释用户需求。

## 2. 起步阶段目标流程

```text
用户输入要求
  -> 项目启动包整理
  -> 世界观规划
  -> 世界观审核
  -> 世界观提交
  -> 分库保存
  -> 人设架构初始化
  -> 大纲架构初始化
  -> 伏笔脉络初始化
  -> 创作架构审核
  -> 创作架构提交
  -> 进入分卷规划
```

当前代码里已有的主线是：

```text
project_brief
  -> world_design
  -> world_review
  -> memory_commit_world
  -> baseline_memory_seed
  -> volume_plan
```

这条主线方向是对的，但中间把“分库保存、人设、大纲、伏笔架构”压得太紧。后续应把 `baseline_memory_seed -> volume_plan` 之间补成一个明确的创作架构初始化段。

## 3. 节点设计

### 3.1 用户输入要求

这不是正式 agent 节点，而是任务入口输入包。

必须提取：

- 用户硬设定
- 题材和风格方向
- 目标字数、卷数、章节节奏
- 禁止事项
- 用户特别强调的创作偏好
- 需要后续确认的问题

不能做：

- 不能直接扩写世界观
- 不能把用户模糊想法改写成已确认事实
- 不能把系统推断当成用户硬设定

### 3.2 项目启动包整理者

节点 ID：`project_brief`

职责：

把用户输入整理成下游可读的项目启动包。

输入：

- 用户原始目标
- 交付要求
- 硬约束
- 参考资料引用

输出：

- 项目目标
- 硬设定
- 风格方向
- 交付边界
- 开放问题
- 下游世界观规划输入

关键原则：

项目启动包只能做整理，不能替后续世界观规划做创作结论。

### 3.3 世界观规划者

节点 ID：`world_design`

职责：

根据项目启动包产出可审核的世界观候选。

输入：

- 项目启动包
- 用户硬设定
- 参考资料

输出：

- 世界观定位
- 主角起点与卷入方式
- 世界规则
- 主要势力与冲突轴
- 长期成长空间
- 不可越界边界

关键原则：

世界观候选不是 canon。它必须标记为候选，等待审核。

### 3.4 世界观审核员

节点 ID：`world_review`

职责：

判断世界观候选是否能支撑后续长篇创作。

输入：

- 世界观候选
- 项目启动包

输出：

- 裁决
- 完整性问题
- 一致性问题
- 长篇支撑度判断
- 修订要求
- 允许提交的世界主干

裁决：

- `pass`
- `pass_with_notes`
- `revise`
- `fail_closed`

关键原则：

审核员只裁决，不替世界观写手改写正文。若未通过，必须带原稿引用退回 `world_design`。

### 3.5 世界观提交管家

节点 ID：`memory_commit_world`

职责：

把通过审核的世界观登记为世界观提交，不直接把所有内容塞进基准库。

输入：

- 世界观审核结果
- 世界观候选引用
- 项目启动包引用

输出：

- `world_commit_id`
- `approved_world_spine`
- 世界观提交引用
- 审核来源引用
- 写入回执

写入位置：

- 产物索引库：世界观候选、世界观审核、世界观提交
- 问题账本：世界观问题状态

关键原则：

`memory_commit_world` 是世界观提交，不是基准库初始化。它的产物是给后续分库节点使用的稳定来源。

### 3.6 分库保存节点

建议节点 ID：`startup_memory_partition`

当前可由 `baseline_memory_seed` 暂时承载，但长期建议独立出来。

职责：

把世界观提交、项目启动包和审核结果拆分进不同仓库。

分库规则：

- 基准库：稳定世界规则、主角不变量、已确认人物身份、已确认关系、长期主线方向
- 动态记忆库：可调整策略、未冻结人物权重、支线优先级、卷级节奏建议
- 产物索引库：候选、审核、提交、引用清单
- 问题账本：未解决问题、被拒绝项、后续需确认问题
- 线程账本：初始伏笔、长期悬念、承诺式线索

输出：

- `baseline_memory_ref`
- `mutable_memory_seed_ref`
- `artifact_index_refs`
- `issue_ledger_refs`
- `thread_ledger_seed_ref`
- `startup_architecture_scope`

关键原则：

分库不是复制。它要判断每条内容的稳定性和用途。

## 4. 人设、大纲、伏笔架构初始化

### 4.1 人设架构初始化

建议节点 ID：`character_architecture_seed`

职责：

把世界观主干中的人物相关信息整理成可长期使用的人设架构。

输入：

- 基准库
- 世界观提交
- 项目启动包
- 动态记忆初始范围

输出：

- 主角不变量
- 核心人物身份
- 核心动机
- 初始关系图
- 冻结人物事实
- 可调整人物权重
- 后续出场节奏建议

入库规则：

- 身份、核心动机、不可逆关系进基准库
- 出场节奏、人物权重、阶段性对手安排进动态记忆库
- 不确定人物只进入候选索引，不进 canon

### 4.2 大纲架构初始化

建议节点 ID：`outline_architecture_seed`

职责：

生成全书主线骨架和可供分卷规划使用的大纲边界。

输入：

- 基准库
- 人设架构
- 世界观提交
- 项目目标

输出：

- 全书主线方向
- 起承转合骨架
- 卷级冲突候选
- 阶段性爽点和危机
- 不可提前泄露的秘密
- 可调整支线池

入库规则：

- 全书主线方向和不可推翻的起点进入基准库
- 卷级节奏、支线顺序、爽点排序进入动态记忆库
- 具体章节安排不在此阶段冻结

### 4.3 伏笔脉络初始化

建议节点 ID：`thread_architecture_seed`

职责：

把世界观、人设和大纲里的承诺式内容登记成线程账本。

输入：

- 世界观提交
- 人设架构
- 大纲架构
- 用户硬设定

输出：

- 初始伏笔清单
- 悬念线
- 关系线
- 世界规则承诺
- 回收窗口
- 线程可见性
- 关闭条件

线程状态：

- 起步阶段只能产生 `proposed` 或 `active`
- 没有审核证据的线程不能直接标为 `active`

关键原则：

伏笔不是提示词里的提醒，而是有生命周期的叙事承诺。

## 5. 创作架构审核与提交

### 5.1 创作架构审核

建议节点 ID：`startup_architecture_review`

职责：

统一审核人设、大纲、伏笔架构是否互相一致，是否越过基准边界。

必须检查：

- 人设是否违背世界观主干
- 大纲是否偷改用户硬设定
- 伏笔是否有合理回收窗口
- 动态记忆是否反向覆盖基准库
- 是否有未解决问题会阻塞分卷规划

输出：

- 裁决
- 架构一致性问题
- 分库错误
- 线程风险
- 允许提交的架构包

### 5.2 创作架构提交

建议节点 ID：`startup_architecture_commit`

职责：

把通过审核的人设、大纲、线程初始化结果写入对应仓库。

写入规则：

- 基准库写稳定不变量
- 动态记忆库写可调整策略
- 线程账本写伏笔和脉络
- 产物索引写候选与审核引用
- 问题账本写开放问题

输出：

- `startup_architecture_commit_id`
- `baseline_memory_ref`
- `mutable_memory_refs`
- `thread_ledger_ref`
- `issue_ledger_ref`
- `volume_plan_input_packet`

## 6. 起步阶段交接给分卷规划的输入

分卷规划不能只读取一个模糊的 `baseline_memory_ref`，它应该收到一个明确的启动架构包：

```text
volume_plan_input_packet
  - project_brief_ref
  - baseline_memory_ref
  - mutable_memory_refs
  - character_architecture_ref
  - outline_architecture_ref
  - thread_ledger_ref
  - open_issue_refs
  - forbidden_changes
  - planning_scope
```

这样分卷规划者才知道：

- 哪些必须遵守
- 哪些可以安排
- 哪些只能延后确认
- 哪些伏笔要在卷级结构里预留位置

## 7. 这一阶段的关键判断

### 7.1 世界观不是全部基准库

世界观提交只是稳定来源之一。基准库还需要承载人物不变量、关系不变量、大纲主干和冻结边界。

### 7.2 动态记忆库不是基准库的副本

动态记忆库存的是可调整的推进策略，不是把基准库复制一遍。

### 7.3 人设和大纲不能全冻结

人设里的身份、核心动机、关键关系可以冻结；人物权重、出场节奏、关系推进速度应进入动态记忆。

大纲里的长期方向可以稳定；卷级细节和章节路径应该留给后续循环调整。

### 7.4 伏笔必须成为账本

如果伏笔只在世界观或大纲文字里，它后面一定会丢。起步阶段就要建立线程账本，让后续章节、卷审、复盘都能追踪。

## 8. 防止起步阶段污染后续的规则

1. 用户输入不等于 canon。
2. 世界观候选不等于 canon。
3. 审核意见不等于 canon。
4. 世界观提交是基准库初始化来源，但不是完整基准库。
5. 人设、大纲、伏笔都要经过创作架构审核后再提交。
6. 未冻结内容不得进入基准库冻结区。
7. 动态库不得覆盖基准库，只能补充推进策略。
8. 伏笔线程不得无状态流入章节写作。

## 10. 暂不配置的原因

这一段需要先讨论清楚，因为它决定后面整部小说的记忆边界。如果起步阶段把世界观、人设、大纲、伏笔混在一个基准库里，章节循环再精细也会继承污染。

正式流程图可以用于理解和审查，也应该作为正式流程保留；只是运行实现时可以把它折叠成可复用图模板。建议把“初始化 -> 审核 -> 提交”做成可复用图模板，每个阶段通过不同协议包复用。

因此下一步应继续逐节点讨论：

1. `project_brief` 到底从用户输入里抽哪些字段。
2. `world_design` 是否只产一个候选，还是允许多候选。
3. `world_review` 通过后，哪些内容允许进入世界观提交。
4. `startup_memory_partition` 的分库规则如何写成契约。
5. 人设、大纲、伏笔初始化是否拆三节点，还是先合成一个创作架构节点。

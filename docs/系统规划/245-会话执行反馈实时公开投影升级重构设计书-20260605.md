# 会话执行反馈实时公开投影升级重构设计书

日期：2026-06-05  
状态：待审阅  
范围：前端会话页、Runtime Monitor SSE、任务执行公开反馈、public_timeline 实时投影  

## 1. 目的

本设计书解决一个明确问题：前端会话页面在任务执行期间只能稳定显示工具记录、运行记录，却不能像成熟 coding agent 一样持续给出自然、流畅、可读的正文式反馈。

这不是单纯的 `ChatMessage` 渲染问题，也不是 CSS 问题。当前系统缺少一条从 task executor runtime event 到 chat-facing public timeline 的实时权威投影链路。后端已经能在部分 snapshot 场景生成 `public_timeline`，但 live Runtime Monitor SSE 仍把 raw runtime event 直接推给前端，导致前端只能从 `step_summary_recorded` 等事件拼出 `progress_entries`，而不是消费后端批准过的公开正文反馈。

目标状态：

```text
TaskExecutor raw runtime events
-> backend runtime monitor public projection
-> public_timeline_delta
-> frontend runtime attachment
-> ChatMessage / PublicRunActivity rendering
```

前端只负责挂载、合并、渲染公开 timeline，不再从工具日志或 raw step 名称猜“这一步该怎么对用户说”。

## 2. 当前破损链路

### 2.1 直接聊天流和任务运行流是两条链

直接聊天流走 `backend/api/chat.py`：

- `backend/api/chat.py:443` 有 `_project_public_stream_event()`。
- `backend/api/chat.py:469` 调用 `project_public_timeline_delta(...)`。
- 前端 `frontend/src/lib/store/events.ts:1163` 只把 `token/content_delta` 追加到 assistant `content`。
- 前端 `frontend/src/lib/store/events.ts:1215` 对 `answer_candidate/assistant_text` 的处理是“消息内容为空时才填充”。

任务执行被 handoff 后，主要进展不再来自原 chat stream，而来自 Runtime Monitor SSE：

- `backend/api/runtime_monitor.py:76` 暴露 `/api/orchestration/runtime-monitor/events`。
- `backend/api/runtime_monitor.py:108` 当前把 `runtime_event.to_dict()` 原样放入 SSE。

这意味着 task-time live event 没有走 `chat.py` 的公开投影逻辑。

### 2.2 后端 snapshot 有 public_timeline，live event 没有

后端已经有正确方向：

- `backend/harness/runtime/public_chat_timeline.py:41` 可以从 `progress_presentation` 构造 `public_timeline`。
- `backend/harness/runtime/public_timeline_stream.py:39` 可以从公开 stream event 构造 `public_timeline_delta`。
- `backend/harness/runtime/run_monitor/projector.py:126` 构造 `progress_presentation`。
- `backend/harness/runtime/run_monitor/projector.py:137` 构造 task monitor 的 `public_timeline`。

但实时 SSE 只发送 raw event。global monitor snapshot 为了性能经常不带详细 runtime events，所以不能依赖右侧 monitor snapshot 自动补齐聊天附件。

### 2.3 前端 live patch 被 progress_entries 卡住

前端已经有读取 `public_timeline` 的能力：

- `frontend/src/lib/store/runtime.ts:3420` 的 `publicTimelineItemsFromRuntimeEvent()` 会读取 event/payload 上的 `public_timeline` 和 `public_timeline_delta`。
- `frontend/src/lib/store/runtime.ts:3873` 的 `patchRuntimeAttachmentFromMonitor()` 会从 monitor snapshot 读取 `public_timeline`。

但 live Runtime Monitor SSE 实际走：

- `frontend/src/lib/run-monitor/controller.ts:178` 应用 monitor snapshot 到右侧 monitor。
- `frontend/src/lib/run-monitor/controller.ts:187` 只把 `payload.runtime_event` 交给 `patchRuntimeAttachmentFromRuntimeEvent()`。
- `frontend/src/lib/store/runtime.ts:3782` 的 `patchRuntimeAttachmentFromRuntimeEvent()` 先调用 `runtimeProgressEntryFromRuntimeEvent()`。
- `frontend/src/lib/store/runtime.ts:3784` 如果不能构造 `progress_entries`，直接返回 state。
- `frontend/src/lib/store/runtime.ts:3542` 的 `runtimeProgressEntryFromRuntimeEvent()` 当前只接受 `step_summary_recorded`。

结果是：即使后端将来在 `model_action_request_received` 上发出 `public_timeline_delta`，前端也可能因为它不是 `step_summary_recorded` 而直接丢弃。

### 2.4 task executor 不应该在 JSON 外流正文

任务执行器要求模型输出结构化 action JSON：

- `backend/harness/loop/task_executor.py:1188` 记录 `model_action_request_received`。
- `backend/harness/loop/task_executor.py:2490` 要求 `public_progress_note=True`。
- `backend/harness/loop/task_executor.py:2491` 要求 `public_action_state=True`。
- `backend/harness/loop/task_executor.py:6233` 明确要求模型只输出合法 JSON，并填写 `action_type`、`public_action_state`、`public_progress_note`。

因此，任务执行期间的“正文式反馈”不应该靠模型在 JSON 外额外吐自然语言。正确做法是把 JSON 内的公开字段投影为 public timeline，再由会话页按正文体验渲染。

## 4. 设计裁决

本次重构采用后端公开投影权威路线。

采用：

- 后端 Runtime Monitor SSE 在发送 raw event 时同步附加 `public_timeline_delta`。
- `model_action_request_received` 映射成 `model_action_admission` 等价公开事件，产出 `opening_judgment` 或 `work_action`。
- `step_summary_recorded` 映射成 `runtime_step_summary`，复用现有 `project_public_timeline_delta()`。
- 工具 observation 进入 `observation_report` 或已完成 `work_action`，不得只显示工具记录。
- 前端只消费 typed public items，不从 raw step/tool payload 再做正文推断。

拒绝：

- 不通过前端正则把工具名、路径、JSON 拼成自然语言。
- 不让模型在 action JSON 外额外输出流式正文。
- 不保留 `progress_entries` 作为主聊天渲染兜底。
- 不把右侧 Runtime Monitor 的 raw trace 面板逻辑搬进主会话正文。

## 5. 目标权威链

| 层级 | 目标职责 | 禁止事项 |
| --- | --- | --- |
| TaskExecutor | 记录 raw runtime event、public_progress_note、public_action_state | 直接决定前端布局 |
| Runtime event log | 持久化真实执行事实 | 生成用户文案 |
| Runtime public projection | 唯一决定 raw event 如何成为 public_timeline_delta | 暴露 raw event name 作为主聊天文案 |
| Runtime Monitor SSE | 发送 raw trace 和 public projection | 只发送 raw event 让前端猜 |
| Frontend store | 按 anchor 挂载、合并、去重 public timeline | 根据 `step`、`tool_name`、JSON 推断正文 |
| ChatMessage / PublicRunActivity | 渲染公开 items | 重新解释运行语义 |
| Run Monitor panel | 展示 debug/monitor trace | 影响主聊天正文内容 |

目标链路：

```text
observe: runtime_event.to_dict()
normalize: backend public runtime monitor event adapter
decide: project_public_timeline_delta()
record/emit: runtime_monitor_event SSE payload
attach: frontend runtime attachment by anchor_turn_id
present: ChatMessage + PublicRunActivity
debug: progress_entries / raw runtime_event only in monitor trace
```

## 6. 数据合同

### 6.1 Runtime Monitor SSE 事件新增公开投影

`runtime_monitor_event` 的 `runtime_event` 保留 raw 字段，同时增加公开字段：

```ts
type RuntimeMonitorPublicEventProjection = {
  public_projection_authority: "runtime_monitor.public_event_projection.v1";
  public_event_type:
    | "model_action_admission"
    | "runtime_step_summary"
    | "turn_tool_observation_recorded"
    | "task_run_lifecycle_event"
    | "done"
    | "error"
    | "stopped"
    | string;
  public_timeline_delta: PublicChatTimelineItem[];
  public_anchor: {
    run_id: string;
    task_run_id?: string;
    turn_run_id?: string;
    anchor_turn_id: string;
    anchor_role: "assistant";
  };
  debug_trace_ref: string;
};
```

实现上必须把这些字段挂在 `runtime_event` 顶层，不再提供嵌套 `runtime_event.public_projection` 形态。原因是前端现有 `publicTimelineItemsFromRecord()` 只读取 record 顶层和 payload 顶层的 `public_timeline` / `public_timeline_delta`。如果改成嵌套结构，就必须同步修改前端读取逻辑；本设计不采用该分支。

### 6.2 PublicChatTimelineItem 使用现有种类

沿用 `frontend/src/lib/api.ts:219` 的 item vocabulary：

- `opening_judgment`：开局判断或阶段性模型判断，来源必须是 `public_action_state.current_judgment` 或 `public_progress_note`。
- `work_action`：当前动作，通常来自 tool call admission。
- `observation_report`：工具或 runtime observation 返回后的事实反馈。
- `todo_plan`：任务清单。
- `status_update`：等待、停止、接管、补充要求等状态。
- `artifact`：产物。
- `verification`：验证结果。
- `blocked`：阻塞、失败和恢复建议。
- `final_summary`：不与 assistant 正文重复的收尾。

### 6.3 Anchor 规则

live event 必须携带足够锚点，避免前端按最近 assistant 消息猜：

```text
优先级：
runtime_event.public_anchor.anchor_turn_id
-> runtime_event.refs.turn_ref
-> runtime_event.payload.turn_id
-> monitor item latest_interaction_turn_id
-> existing attachment with same run_id
```

如果后端无法确定 `anchor_turn_id`，该 public delta 不得进入主聊天，只能留在 monitor/debug。

### 6.4 冷启动和重连补偿

Runtime Monitor SSE 的订阅只接收订阅之后的新事件。`runtime_monitor_snapshot` 是当前 monitor 状态，不是事件 replay。若前端重连时错过了 `model_action_request_received` 或 observation event，只靠后续 live delta 会丢失开局判断或关键反馈。

固定规则：

- SSE live delta 负责“新事件”。
- 当前会话冷启动、切换会话、SSE reconnect 后，前端必须通过 session timeline 或 session live monitor 补一次当前会话 attachment。
- 补偿入口优先使用 `getSessionTimeline()` 返回的 `runtime_attachments.public_timeline`，因为它已经走 `session_timeline.py` 的 snapshot public projection。
- 如果只拿到 global `runtime_monitor_snapshot`，不能假定其中包含完整 `public_timeline`；global monitor 为性能可能不带详细 event window。
- 重连补偿必须按 `item_id` / `trace_refs` 与 live delta 去重。

## 7. 固定执行流

### Stage 1：task executor 记录原始事件

输入：

- `model_action_request_received`
- `step_summary_recorded`
- `task_tool_observation_recorded`
- `task_run_lifecycle_event`
- terminal events

输出：

- raw runtime event log

不允许：

- 在 task executor 中写前端 UI 文案。
- 在 JSON 外制造额外 assistant 正文流。

### Stage 2：Runtime Monitor SSE 做公开事件适配

输入：

- raw runtime event
- runtime host / task run state
- monitor item 或 session anchor context

输出：

- `public_event_type`
- normalized public projection data
- `public_timeline_delta`
- `public_anchor`

不允许：

- 把 raw event name 直接当 public item title。
- 为无公开字段的工具日志编造高层判断。

### Stage 3：Frontend store 挂载公开 delta

输入：

- `runtime_event.public_timeline_delta`
- `runtime_event.public_anchor`
- optional debug `progress_entries`

输出：

- 对应 assistant message 的 `runtimeAttachments[].public_timeline`

不允许：

- 因为没有 `progress_entries` 就丢弃 `public_timeline_delta`。
- 从 `step`、`tool_name`、`agent_brief_output` 继续推断正文。

### Stage 4：Chat 渲染

输入：

- assistant `content`
- runtime attachment `public_timeline`
- transient `runtimePublicTimelineDraft`

输出：

- assistant 正文
- inline run activity

不允许：

- 主聊天显示 `step_summary_recorded`、`task_execution_packet_compiled`、裸 `false`、raw tool id。
- 普通成功完成显示重复的 done / completed receipt。

## 8. 模块计划

### 8.1 后端新增公开投影适配器

建议新增：

```text
backend/harness/runtime/runtime_monitor_public_projection.py
```

职责：

- 提供 `project_runtime_monitor_event_public_delta(runtime_event, *, runtime_host, monitor=None) -> dict`。
- 将 raw event type 映射到 `public_timeline_stream.py` 支持的 public event type。
- 补齐 `event` wrapper、`step`、`summary`、`public_progress_note`、`public_action_state`、`runtime_task_run_id`、`anchor_turn_id`。
- 返回空公开投影时说明 `public_projection_skip_reason`，只供 debug。

核心映射：

| raw event | public event | 输出重点 |
| --- | --- | --- |
| `model_action_request_received` | `model_action_admission` | `opening_judgment`、running `work_action` |
| `model_action_admission_checked` | `model_action_admission` | 兼容单 turn runtime 的 admission 事件 |
| `step_summary_recorded` | `runtime_step_summary` | `observation_report`、`todo_plan`、status |
| `turn_tool_observation_recorded` | `turn_tool_observation_recorded` | 直接复用现有 `payload.tool_observation` 形态 |
| `task_tool_observation_recorded` | `task_tool_observation_recorded` | 使用 task observation normalizer，输入是 `payload.observation` |
| task lifecycle event | `task_run_lifecycle_event` | todo、接管、等待 |
| terminal stop/error/done | existing public terminal event | stopped/blocked/final_summary |

注意：`task_tool_observation_recorded` 不能简单改名成 `turn_tool_observation_recorded` 后直接丢给 `_turn_tool_observation_item()`，因为当前任务执行器写入的是 `payload.observation`，而现有 turn projector 读取的是 `payload.tool_observation`。固定实现方式是在 `public_timeline_stream.py` 中新增 `task_tool_observation_recorded` 分支，显式读取 `payload.observation`。禁止在 adapter 中隐式改名后复用 turn 分支，因为这会隐藏 task/turn 两套原始事件形态的差异。

### 8.1.1 Live 与 Snapshot 共用语义 normalizer

`backend/harness/runtime/progress_presenter.py` 已经能处理：

- `model_action_request_received`
- `step_summary_recorded`
- `task_tool_observation_recorded`
- `agent_todo_initialized`
- `task_run_lifecycle_started`
- `task_run_lifecycle_finished`

live adapter 不能重新实现一套 task observation 翻译规则。实现时应抽取或复用同一批底层 normalizer，例如工具结果清洗、raw bool 翻译、terminal 判断、todo 解析和 artifact refs 提取。否则 live `public_timeline_delta` 和 session refresh 后的 snapshot `public_timeline` 会出现文案、状态、item id 不一致。

### 8.2 后端 Runtime Monitor SSE 接入

修改：

```text
backend/api/runtime_monitor.py
```

动作：

- 在 `runtime_event` 到达后只收集一次 monitor snapshot。
- 调用公开投影适配器 enrich raw event。
- SSE payload 中发送 enriched `runtime_event` 和同一份 `monitor`。

推荐结构：

```python
monitor = service.collect_global_runtime_monitor(limit=requested_limit)

    raw_event,
    runtime_host=runtime_host,
    monitor=monitor,

    "runtime_monitor_event",
    {
        "runtime_event": {**raw_event, **public_projection},
        "monitor": monitor,
        "source": "runtime_event_log",
    },
    event_id=runtime_event.event_id,
)
```

## 9. 分阶段实施计划

### Phase 1：后端 live public projection

目标：

- Runtime Monitor SSE 对每个可公开 runtime event 附加 `public_timeline_delta`。

文件：

- 新增 `backend/harness/runtime/runtime_monitor_public_projection.py`
- 修改 `backend/api/runtime_monitor.py`
- 必要时扩展 `backend/harness/runtime/public_timeline_stream.py`
- 新增 `backend/tests/runtime_monitor_public_projection_test.py`

完成标准：

- `model_action_request_received` 可产生 live `opening_judgment`。
- `step_summary_recorded` 可产生 live `observation_report` 或 `work_action`。
- raw monitor event 仍可用于 debug，但主聊天有公开投影。

禁止：

- 不在 frontend 增加正则翻译。
- 不把 raw event name 作为 public item 文案。

### Phase 2：前端 live attachment 消费 public delta

目标：

- public delta 不再被 `progress_entries` gate 丢弃。

文件：

- `frontend/src/lib/api.ts`

## 12. 验收标准

### 12.1 开局判断

输入 raw event：

```json
{
  "event_type": "model_action_request_received",
  "payload": {
    "model_action_request": {
      "action_type": "tool_call",
      "public_progress_note": "我先确认当前反馈链路，再收敛到单一页面投影。",
      "public_action_state": {
        "current_judgment": "需要先找出实时 monitor 链路的投影缺口。",
        "next_action": "读取 runtime monitor SSE 和前端 store 挂载逻辑。"
      }
    }
  }
}
```

期望：

- SSE event 中包含 `public_timeline_delta`。
- 主聊天显示开局判断或活动反馈。
- 不要求该事件同时是 `step_summary_recorded`。

### 12.2 工具调用和观察

期望：

- 工具 admission 显示 running `work_action`。
- observation 返回后同一语义 item 更新为 done/error。
- 显示事实含义，例如“目标路径不存在”，而不是裸 `false`。

### 12.3 刷新一致性

期望：

- live 期间 public delta 显示。
- session refresh 后 persisted `public_timeline` 接管。
- 同一 `item_id` 不重复。

### 12.4 Debug 边界

期望：

- raw event、tool payload、technical trace 仍在 monitor/debug 可查。
- 主聊天不显示 `step_summary_recorded`、`runtime_invocation_packet_compiled`、`task_execution_packet_compiled`。

## 13. 迁移和清理规则

本重构不保留旧主聊天 fallback。

允许保留：

- `progress_entries` 作为 debug trace 或 monitor 侧信息。
- raw `runtime_event` 在 SSE 中供右侧 monitor 和调试使用。
- `public_chat_timeline.py` 的 snapshot projection。

必须清理：

- 前端以 `progress_entries` 作为主聊天挂载前置条件的逻辑。
- 前端根据 `step`、`tool_name`、raw JSON 生成主聊天正文的逻辑。
- 测试中保护 raw event log 式主聊天展示的断言。

切换规则：

- Phase 1 完成后，SSE 同时有 raw event 和 public delta。
- Phase 2 完成后，主聊天以 public delta 为准。
- Phase 3 完成后，旧 raw progress 主聊天路径不可达。

回滚规则：

- 若 Phase 1 projection 出现严重误投影，回滚 `runtime_monitor.py` 的 enrich 调用即可恢复原 raw SSE，不影响 event log。
- 若 Phase 2 前端挂载异常，回滚到只处理 `progress_entries` 会恢复旧体验，但该回滚只作为临时故障处理，不能作为长期兼容路径。

## 14. 禁止捷径

- 禁止用前端正则把工具日志改写成正文。
- 禁止把 `progress_entries` 留作主聊天兼容兜底。
- 禁止让模型在 action JSON 外输出自然语言来绕过结构化协议。
- 禁止降低测试断言或删除失败测试来制造通过。
- 禁止随机更换本地端口。
- 禁止把 debug trace 当成 public timeline。

## 15. 最终结果

重构完成后，任务执行中的主会话体验应从：

```text
调用工具 read_file
观察结果 ...
step_summary_recorded
工具调用已完成，正在根据结果继续
```

变成：

```text
我先确认当前反馈链路，再收敛到单一页面投影。

正在读取 runtime monitor SSE 入口。
已确认实时事件目前没有走公开投影，只把 raw runtime event 推给前端。

正在调整前端挂载边界。
后续公开反馈会直接进入 public_timeline，不再被 progress_entries 条件拦住。
```

核心标准只有一个：主聊天展示 agent 的公开工作反馈，工具记录和 raw trace 留在 monitor/debug，不再反过来支配正文体验。

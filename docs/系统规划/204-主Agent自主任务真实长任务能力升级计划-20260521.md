# 204-主Agent自主任务真实长任务能力升级计划

日期：2026-05-21

状态：实施计划

关联文档：`182-主Agent无图长任务与任务图统一运行底座方案-20260520.md`

## 1. 问题结论

182 方案已经把 `autonomous_task_run` 从意图标签推进成了正式 runtime driver：有 lane、recipe、`TaskRunLedger` 动态计划项、checkpoint、monitor、sandbox、一轮到多轮工具观察，以及 `managed` 策略档保留。但真实长任务验收仍不稳定，根因不是“提示词还不够强”，而是 driver 仍缺少目标契约、步骤约束和强自检门。

最近自主任务验收结果：

| 运行 | 结果 | 证据 |
|---|---:|---|
| `autonomous_task_run_regression.py` | 16/16 通过 | stub 模型按预期调用工具 |
| `autonomous-tasks-20260520-after-closeout` | 2/5 通过 | 真实模型场景仍失败 |
| 测试报告分析 | 失败 | 最终回答泄漏伪工具调用，缺少结构/根因/回归结论 |
| PDF + 表格混合分析 | 失败 | PDF 后继续输出 `delegate_to_agent` 伪调用，没有完成治理/库存/行动建议 |
| 功能草案交付 | 失败 | 未调用 `write_file`，最终回答为空 |

正确方向：继续使用同一 `TaskRunLoop` / `TaskRunLedger` / checkpoint / event log / sandbox，不新建长任务系统；但必须让 autonomous task run 从“模型自由 ReAct”升级为“目标契约驱动的计划-动作-验收状态机”。

## 2. 失败清单与结构根因

### 2.1 固定控制计划不能承载真实长任务

位置：`backend/orchestration/runtime_loop/autonomous_task_run_driver.py`

当前 `_standard_control_plan()` 固定生成：

1. `autonomous.goal_lock`
2. `autonomous.context_review`
3. `autonomous.final_check`

它没有根据用户目标生成“读取材料”“产出文件”“运行验证”“综合多源证据”等可执行计划项。真实运行中，模型只能在一个泛化 `context_review` 步骤内自行决定所有动作，ledger 看起来有计划，但计划并不表达验收约束。

修改细则：

1. 增加 `AutonomousTaskGoalContract` 解析层，从用户目标和已知 recipe metadata 生成结构化目标契约。
2. 契约至少包含：
   - `required_material_paths`
   - `required_tool_kinds`
   - `required_output_kinds`
   - `requires_write_output`
   - `requires_verification_command`
   - `requires_delegation`
   - `response_must_include`
   - `forbidden_visible_markers`
3. `_standard_control_plan()` 改为消费目标契约，生成真实动态计划项：
   - `autonomous.material_review`
   - `autonomous.produce_output`
   - `autonomous.verify_output`
   - `autonomous.synthesize_answer`
   - `autonomous.final_check`
4. plan event 必须记录 `plan_source=goal_contract_runtime_policy` 和 `goal_contract` 摘要。

完成标准：

1. 用户要求“写入草案文件”时，ledger 必须出现 produce/write 计划项。
2. 用户要求“运行命令验证”时，ledger 必须出现 verify 计划项。
3. 用户给出明确文件路径时，计划项必须绑定 material path。

### 2.2 工具循环缺少必需动作约束

位置：`backend/orchestration/runtime_loop/autonomous_task_run_driver.py`

当前工具循环允许模型在预算内自由调用工具。虽然已有“写入预算保留”，但只有预算接近耗尽时才阻断泛化搜索。真实功能草案失败说明：模型能连续 `read_file/search_files/search_text`，却不写文件。

修改细则：

1. 增加 `AutonomousTaskActionTracker`，跟踪：
   - 已读材料路径
   - 已调用工具名
   - 已写入/编辑路径
   - 已运行命令
   - 已收到委派结果
2. 每次 `tool_call_requested` 前用目标契约判断是否允许继续泛化读搜。
3. 如果 `requires_write_output=true` 且材料已读，下一步非 `write_file/edit_file` 应进入 contract violation repair，而不是继续消耗工具预算。
4. 如果 `requires_verification_command=true` 且已有写入/编辑，下一步非 `terminal` 应进入 contract violation repair。
5. 修复提示必须简短、具体、agent-facing，不能写开发说明。

完成标准：

1. 功能草案场景不能在未写文件时正常收口。
2. 代码修复场景必须读文件、编辑、terminal 验证三类动作齐全后才能通过 verification。
3. 若模型拒绝执行必需动作，应返回 blocked/partial，而不是空成功。

### 2.3 自检门只看 final_content，不能验证目标完成

位置：`backend/orchestration/runtime_loop/autonomous_task_run_driver.py`

当前 verification 核心条件是：

```python
"passed": bool(outcome.final_content and outcome.terminal_reason == "completed")
```

这只能判断“有回答”，不能判断“材料读了没有、文件写了没有、验证跑了没有、回答是否包含必需结论”。因此真实失败要靠外部 long scenario 才能发现。

修改细则：

1. verification 改为目标契约驱动。
2. `passed` 必须同时满足：
   - 有最终回答。
   - 无可见协议泄漏。
   - 所有 required material 至少有一次 read/search/委派证据。
   - `requires_write_output` 时至少有 `write_file` 或 `edit_file` 观察。
   - `requires_verification_command` 时至少有 `terminal` 观察。
   - `response_must_include` 的关键词至少命中。
3. verification payload 增加：
   - `contract_passed`
   - `missing_required_actions`
   - `missing_response_terms`
   - `protocol_leak_detected`
   - `artifact_observation_count`
   - `verification_command_count`
4. 若 verification 不通过：
   - 有 repair budget 时进入一次 targeted repair。
   - 无 repair budget 或 repair 后仍失败时，terminal_reason 设为 `verification_failed` 或 `partial_contract_failed`。

完成标准：

1. 功能草案未写文件不能得到 passed verification。
2. DSML/tool markup 泄漏不能得到 passed verification。
3. 用户要求“结构性根因/回归测试”但回答缺关键项时不能得到 passed verification。

### 2.4 协议泄漏清洗仍有旁路

位置：

- `backend/output_boundary/boundary.py`
- `backend/execution/model_response.py`
- `backend/orchestration/runtime_loop/autonomous_task_run_driver.py`

当前 output boundary 已识别部分 DSML/XML 工具调用，但真实失败里出现了半截：

```text
name="read_file">
<｜｜DSML｜｜parameter ...
```

这种不是完整 `<｜｜DSML｜｜invoke` 块，清洗不稳定。

修改细则：

1. 扩展 protocol leak 检测：
   - `name="read_file">`
   - `name="delegate_to_agent">`
   - `<｜｜DSML｜｜parameter`
   - `</｜｜DSML｜｜parameter`
   - `tool_calls`
   - `invoke name=`
2. driver final closeout 后必须调用统一清洗，并将 `protocol_leak_detected` 写入 verification。
3. 如果清洗后为空，不能保留 completed，应进入 repair 或 failed。

完成标准：

1. 长场景失败样本中的半截 DSML 不再作为可见回答。
2. 清洗导致回答为空时，terminal_reason 不再是 completed。

### 2.5 autonomous_task_run 吞掉专业材料路由后，缺少专业材料执行策略

位置：

- `backend/tasks/execution_shape_resolver.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/orchestration/runtime_loop/autonomous_task_run_driver.py`

当前显式 autonomous task 会选择 `runtime.recipe.autonomous_task_run`，并跳过 recipe MCP prephase。这避免静默降级，但 PDF/XLSX 这类材料任务需要专业处理。现在模型可能尝试委派，但路径、scope 和结果整合不稳定。

修改细则：

1. 目标契约识别 `.pdf/.xlsx/.csv/.json/.md/.py` 等材料类型。
2. 第一版不恢复全局 MCP prephase；由 driver 生成 bounded material review 步骤。
3. 对普通文本/JSON/Python 文件优先 `read_file`。
4. 对 PDF/XLSX 第一版允许两种策略：
   - 如果有对应专业工具，调用专业工具。
   - 如果没有稳定工具，进入明确 limitation/blocked，不伪造内容。
5. 委派时必须把 parent goal、material path、expected evidence 写入请求，而不是自然语言泛化外包。

完成标准：

1. PDF/XLSX 场景至少不能输出伪委派调用或伪分析。
2. 缺专业工具时必须明确 blocked/limitation。
3. 子 Agent 回传不能原样成为最终回答，必须进入 parent synthesis。

## 3. 目标设计

### 3.1 新增运行时目标契约

目标契约不是新系统，也不替代 intent。它是 `AutonomousTaskRunDriver` 在 goal locked 后生成的 task-run-local control contract，只用于当前 run 的计划、动作约束、自检和 monitor。

推荐结构：

```python
{
  "contract_id": "autonomous-goal-contract:<task_run_id>",
  "goal": "...",
  "required_material_paths": [...],
  "required_tool_kinds": ["read_material", "write_output", "verify_command"],
  "required_output_kinds": ["sandbox_file", "final_answer"],
  "requires_write_output": true,
  "requires_verification_command": false,
  "requires_delegation": false,
  "response_must_include": ["后端", "前端", "测试"],
  "forbidden_visible_markers": ["<｜｜DSML", "tool_calls", "invoke name="],
  "authority": "orchestration.autonomous_task_goal_contract"
}
```

### 3.2 执行状态流

```text
goal_locked
  -> goal_contract_built
  -> plan_drafted
  -> material_review
  -> produce_output?
  -> verify_output?
  -> synthesize_answer
  -> verification_ready
  -> ready_for_commit | blocked | failed
```

### 3.3 设计约束

1. 不新建 `LongRunLedger`。
2. 不把无图任务伪装成 TaskGraph。
3. 不让 prompt 承担结构约束。
4. 不伪造工具、测试、写入结果。
5. 不以 `response.nonempty` 作为成功标准。
6. 不把开发说明当作 agent-facing prompt。

验收：

2. 新增测试：修复并验证请求生成 verify_output 计划项。

### 阶段 B：必需动作追踪和工具约束

文件：

- `backend/orchestration/runtime_loop/autonomous_task_run_driver.py`

改动：

1. 增加 action tracker。
2. 工具请求前做 contract gate。
3. 非法继续读搜时进入 repair prompt。
4. 写入和验证动作完成后更新 tracker。

### 阶段 C：强 verification

文件：

- `backend/orchestration/runtime_loop/autonomous_task_run_driver.py`
- `backend/output_boundary/boundary.py`

1. verification 使用 goal contract + tracker。
2. final content 清洗后为空则失败。
3. 写入/验证/关键词缺失进入 failed/partial。

验收：

1. 未写文件不能 completed。
2. 协议泄漏不能 completed。
3. 预算耗尽 closeout 必须满足响应关键词，否则 partial。

### 阶段 D：真实长场景复跑

命令：

```powershell

```

```powershell

```

验收：

1. 单测通过。
2. autonomous feature slice 至少从“未 write_file/空回答”变为写入或明确 failed。
3. DSML 泄漏不再出现在可见回答。
4. 测试报告分析不再以工具调用片段作为回答。

## 6. 本轮不做

1. 不实现后台 managed runner。
2. 不实现完整恢复 replay。
3. 不接容器级 sandbox。
4. 不改 TaskGraph scheduler。
5. 不把所有 PDF/XLSX 能力一次性做完；第一版先 fail closed，避免伪造分析。

## 7. 风险控制

| 风险 | 控制 |
|---|---|
| 过度约束导致模型无法回答 | 只有契约明确要求的动作才 hard gate，其余保留普通收口 |
| 关键词验收过硬 | 关键词只来自用户明确要求和少量领域映射，不做大词表 |
| 清洗误删正常回答 | protocol marker 只针对工具协议形态，不删普通技术词 |
| 写入类副作用风险 | 继续使用 sandbox overlay，真实 workspace 只读 |
| 单测通过但真实模型失败 | 加入真实失败样本导向的回归测试，并复跑 autonomous_tasks |

## 8. 成功定义

本轮升级完成后，autonomous task 不一定已经是完整 managed 长任务系统，但必须达到以下下限：

1. 能从用户目标形成可执行契约。
2. 能把契约映射到 ledger 计划项。
3. 能阻止明显违反目标的工具循环。
4. 能在缺少必需动作或协议泄漏时 fail closed。
5. 能给 monitor / verification 留下可复查证据。

# 洪荒时代任务系统实施设计

日期：2026-04-28  
定位：在现有灵魂系统、ControlKernel fail-closed 状态和 TaskCoordinator 记录层基础上，落地第一版可观测、可迁移、可验证的任务系统。

---

## 0. 当前结论

任务系统第一阶段不要直接恢复执行链。

现在 `QueryRuntime.astream()` 已经在进入旧执行链前调用 `ControlKernel.collect()` 并 fail-closed 返回，说明项目已经主动把旧编排线路清空。此时最稳的下一步是：

```text
UserRequest
  -> TaskContract
  -> TaskModeCandidate / TaskSoulRequirement / AgentRequirement / CapabilityRequirement
  -> ControlKernel preview adoption
  -> TaskRecord.metadata projection
  -> frontend / tests 可观察
```

也就是说，先把“任务是什么、需要什么意志、推荐什么 agent、允许什么能力”变成明确候选和预览结果，再恢复执行。

一句话原则：

> 任务系统先成为 ControlKernel 的语义入口，再成为执行调度入口。

### 1.2 当前系统事实

当前已有三块基础已经可复用：

```text
TaskRecord / TaskContextRef
  已能记录任务、上下文绑定、约束、结果引用和事件。

ControlKernel / CandidateEnvelope / ExecutionGraph
  已经建立候选只读、执行图空图、fail-closed 的控制面。

SoulProfile / SoulProjection / ProjectionCard
  已经能管理五灵魂、自制灵魂、投影卡片、PromptManifest 来源。
```

当前缺口：

```text
TaskContract 字段还偏旧，只有 task_kind/modality，没有 task_family/task_mode。
任务模式识别尚未实现，task_type/task_kind/route/tool_name 仍散在旧链路中。
TaskSoulRequirement / SoulSelectionCandidate / TaskSoulBinding 还没有代码契约。
AgentProfile 仍是 agent_type 字符串或 A2A agent catalog，不是任务级 AgentRequirement。
CapabilityRequirement 尚未把 skill/tool/worker 边界作为候选提交给 ControlKernel。
TaskCoordinator 仍承担部分自然语言 binding 猜测，应逐步降级为记录层和兼容候选源。
```

### 1.3 审查发现

#### P1：TaskContract 不是文档中的当前轮任务真相

位置：`backend/orchestration/contracts.py`

文档要求 `TaskContract` 承载 `task_mode`、`bindings`、`requested_outputs` 和 refs；当前实现只有 `task_kind`、`modality`、`inputs`、`constraints`、`refs`。这会导致后续 TaskMode 和 SoulBinding 只能塞进 `inputs` 或 `refs`，控制面无法稳定区分“任务模式”和“执行路线”。

暴露场景：`/api/orchestration/dry-run` 只能返回空图和 fail-closed，不能展示“这是 system_design，默认四岳，architect_agent”这类预览。

#### P1：QueryRuntime 已 fail-closed，但没有任务预览结果给前端和测试系统

位置：`backend/query/runtime.py`

`astream()` 保存用户消息后立即返回 `orchestration_fail_closed`。这是正确的迁移状态，但目前 fail-closed 事件只告诉用户“旧链路清空”，没有携带 TaskMode、SoulBinding、AgentRequirement、CapabilityRequirement 等新系统关键产物。

暴露场景：前端无法判断下一步应恢复哪类执行链，测试也只能验证 blocked，不能验证任务识别是否正确。

#### P2：ProjectionCard 可被选中，但任务系统不会消费 selected_projection_id

位置：`backend/soul/projection_store.py`、`backend/api/souls.py`

投影卡片存储已经正确避免保存 runtime_view，但任务系统尚未读取 `selected_projection_id`。文档中“用户显式选用 projection_id > 当前任务绑定 > TaskMode 推荐 > 当前激活灵魂兜底”的优先级还没有执行入口。

暴露场景：用户在 Playground 里选择了“四岳 / 架构投影”，但下一轮任务仍只按 active seed 或空 TaskContract 进入 ControlKernel。

#### P2：TaskCoordinator ID 生成有碰撞风险，不适合并发任务系统

位置：`backend/tasks/coordinator.py`

当前 task_id 多处依赖 `len(self._tasks) + 1` 或 session 内递增 index。单进程顺序执行问题不大，但并发任务、重试、恢复、删除旧任务后再创建，都可能产生不稳定或碰撞风险。

暴露场景：后续引入后台任务、并行子任务或恢复任务时，两个创建路径同时读取同一个长度，生成相同 task_id 并覆盖 `_tasks`。

#### P2：任务状态机缺少统一终态判断、取消和通知语义

位置：`backend/tasks/models.py`

当前状态是自由字符串，只有 running/completed/failed 三个 mutator。参考 Claude Code 的任务系统，任务层至少需要统一 `pending/running/completed/failed/killed` 和 `is_terminal_task_status()`，否则未来 kill、后台任务、UI 驱逐、结果通知会散落实现。

暴露场景：工具执行中断、用户取消、多 agent 分支被 ControlKernel 阻断时，任务记录无法统一表达 `killed/blocked/cancelled`。

#### P3：TaskContextRef 中 task_kind 与 TaskRecord.task_type 混用

位置：`backend/tasks/models.py`、`backend/tasks/context_models.py`

`TaskRecord.task_type` 表示 query/tool/execution_kind，`TaskContextRef.task_kind` 表示 pdf/structured_data/weather/general。这两个字段都叫 task，但语义不同。后续新增 TaskFamily/TaskMode 后，如果不改名或显式分层，很容易出现 `task_type=tool`、`task_kind=pdf`、`task_mode=document_analysis` 三套真相互相覆盖。

## 2. 推荐设计方向

采用四层任务系统：

```text
TaskContract
  当前轮用户任务的 canonical input，由 ControlKernel 拥有。

TaskMode Layer
  只描述任务形态：task_family / task_mode / requested_outputs。

TaskRequirement Layer
  产出候选：soul_requirement、agent_requirement、capability_requirement。

TaskRecord Layer
  只记录生命周期、上下文句柄、结果引用、已采纳投影，不负责采纳或执行。
```

明确不做：

```text
不让 TaskCoordinator 选择工具。
不让 SoulProfile 授权工具。
不让 TaskMode 等同 route/tool/worker。
不恢复旧 QueryPlanner 作为任务真相。
不把 ProjectionCard 当运行时权限卡。
```

从 Claude Code 任务系统借鉴：

```text
借鉴状态机、终态判断、任务 ID 稳定生成、后台任务生命周期、通知与 UI 可观察性。
不照搬 kill-only Task 接口，因为本项目第一阶段任务系统是语义控制面，不是进程管理器。
不照搬 shell/agent 输出文件队列，除非后续恢复长任务后台执行。
```

## 3. 目标模型

### 3.1 TaskContract

建议迁移到 `backend/tasks/contracts.py`，并让 `backend/orchestration/contracts.py` 重新导出，减少循环依赖。

```python
@dataclass(slots=True, frozen=True)
class TaskContract:
    task_id: str
    session_id: str
    user_goal: str
    task_family: str = "qa"
    task_mode: str = "general_qa"
    parent_task_id: str = ""
    source: str = "user_request"
    bindings: dict[str, Any] = field(default_factory=dict)
    constraints: dict[str, Any] = field(default_factory=dict)
    requested_outputs: tuple[str, ...] = ()
    refs: dict[str, Any] = field(default_factory=dict)
    canonical_owner: str = "orchestration.control_kernel"
```

兼容规则：

```text
旧 task_kind -> refs.legacy_task_kind
旧 modality -> refs.legacy_modality
旧 inputs -> refs.inputs_snapshot 或按语义拆到 bindings/requested_outputs
```

### 3.2 TaskModeDefinition

放在 `backend/tasks/modes.py`。

```python
@dataclass(slots=True, frozen=True)
class TaskModeDefinition:
    task_family: str
    task_mode: str
    title: str
    default_will_type: str
    default_soul_id: str
    default_agent_profile_id: str
    default_memory_scope: str
    required_capabilities: tuple[str, ...] = ()
    requested_outputs: tuple[str, ...] = ()
```

第一批模式只实现文档中的最小闭环：

```text
general_qa -> dialogue / active soul / general_agent
context_qa -> collect / hebo / general_agent
reasoning_qa -> inspect / xuannv / critic_agent
knowledge_lookup -> collect / hebo / knowledge_agent
document_analysis -> organize / siyue / document_agent
data_analysis -> inspect / xuannv / data_agent
code_or_file_processing -> execute / zhurong / executor_agent
system_design -> organize / siyue / architect_agent
implementation -> execute / zhurong / executor_agent
risk_review -> inspect / xuannv / critic_agent
final_answer -> govern / active soul / main_agent
```

### 3.3 TaskSoulRequirement / Candidate / Binding

放在 `backend/tasks/soul_binding.py`。

```text
TaskSoulRequirement
  task_id
  task_mode
  desired_will_type
  required_posture
  explicit_soul_id
  selected_projection_id
  confidence

SoulSelectionCandidate
  candidate_id
  task_id
  soul_id
  will_type
  source
  reason
  confidence
  authority=candidate_only

TaskSoulBinding
  task_id
  soul_id
  soul_name
  will_type
  agent_profile_id
  task_mode
  projection_id
  selection_reason
  selected_by=control_kernel
```

候选优先级：

```text
explicit_projection_id
  > explicit_soul_id
  > selected ProjectionCard
  > task_mode_default
  > active_soul fallback
```

关键边界：

```text
Requirement 和 Candidate 没有采纳权。
Binding 只能由 ControlKernel preview/adopt 生成。
Binding 先写入 TaskRecord.metadata["soul_binding"]。
```

### 3.4 TaskAgentRequirement

放在 `backend/tasks/agent_requirement.py`。

第一阶段不新建完整 agent 运行时，只定义任务级 agent profile contract：

```text
AgentProfile
  agent_profile_id
  title
  default_soul_id
  allowed_skill_ids
  allowed_worker_ids
  memory_scope
  output_contract
  no_direct_execution_authority=True
```

先以内置 registry 形式实现，不绑 A2A 运行时。

### 3.5 TaskCapabilityRequirement

放在 `backend/tasks/capability_requirement.py`。

```text
TaskCapabilityRequirement
  task_id
  task_mode
  required_skills
  allowed_tools
  allowed_workers
  denied_tools
  required_artifacts
  reason
```

第一阶段所有字段只作为 `CandidateEnvelope(candidate_type="task_capability_requirement")` 交给 ControlKernel，不产生 RuntimeDirective。

## 4. 固定执行流

第一阶段预览流：

```text
api/chat or api/orchestration/dry-run
  -> build TaskContract
  -> classify TaskMode candidates
  -> build TaskSoulRequirement
  -> build SoulSelectionCandidate
  -> build AgentRequirement candidate
  -> build CapabilityRequirement candidate
  -> ControlKernel.collect(...)
  -> ControlKernelResult.diagnostics 增加 selected_preview
  -> TaskCoordinator 可选 create_preview_task(...)
  -> SSE / dry-run 返回可观察结构
```

第二阶段执行流：

```text
ControlKernel preview
  -> directive builder 生成 ExecutionGraph
  -> ResourcePolicy 授权工具和 worker
  -> SoulProjectionBuilder 生成 runtime view
  -> QueryRuntime 按 directive 执行一个最小 general_qa/model node
  -> TaskCoordinator 记录 result_ref
  -> CommitGate 决定 session/message/memory 写回
```

## 6. 迁移和回滚规则

### 6.1 兼容窗口

保留旧字段：

```text
TaskRecord.task_type
TaskRecord.agent_type
TaskContextRef.task_kind
TaskContextRef.bindings
TaskContextRef.constraints
```

新增字段先进入：

```text
TaskRecord.metadata.task_family
TaskRecord.metadata.task_mode
TaskRecord.metadata.soul_requirement
TaskRecord.metadata.soul_candidates
TaskRecord.metadata.soul_binding
TaskRecord.metadata.agent_requirement
TaskRecord.metadata.capability_requirement
```

### 6.2 切换规则

```text
preview_only:
  只生成候选和 blocked ControlKernelResult，不恢复执行。

single_model_directive:
  只允许 general_qa/final_answer 生成单模型 ExecutionNode。

worker_directive:
  才允许 document/data/rag/code worker 节点进入 ExecutionGraph。
```

### 6.3 回滚规则

```text
删除或关闭新 classifier 不影响 TaskCoordinator 旧记录。
ControlKernel 如果无法采纳 task_mode，返回 blocked + clarification_needed。
任何 Requirement 生成失败，都退回 general_qa + active_soul fallback，但 diagnostics 必须记录 fallback_reason。
```

```text
不要在 TaskCoordinator 里继续扩展自然语言任务识别。

不要把 agent_type 字符串升级成事实上的 AgentProfile。
```

## 9. 下一步推荐

立即做 Phase 1 和 Phase 2，形成最小闭环：

```text
1. TaskMode registry + rule classifier。
2. TaskSoulRequirement / SoulSelectionCandidate。
3. ControlKernel blocked preview 输出 selected_task_mode_preview + selected_soul_binding_preview。
4. dry-run/chat fail-closed 事件展示任务预览。
```

这一步完成后，灵魂系统就不再只是 prompt 管理，而是能被任务系统正式调用；任务系统也不再只是记录层，而是 ControlKernel 的第一层语义底座。

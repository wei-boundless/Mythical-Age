# 79-主Agent预算耗尽与子Agent回收后收口失效结构修复计划

## 0. 文档定位

本文承接 77、78 号文档，不再把当前问题定义为“某几个回合预算不够”或“子 Agent 偶发调用失败”，而是定义为：

```text
主 Agent 在多轮运行中的“证据收集 -> 结果裁决 -> 最终收口”闭环没有被结构化约束，
导致预算、委派、上下文和测试判定四个系统共同放大了错误。
```

本计划的目标不是提高几个阈值，也不是针对个别回合加例外规则，而是修正主 Agent 的运行哲学：

```text
子 Agent 是受限证据执行器。
主 Agent 是当前回合唯一裁决者。
一旦证据足以回答当前问题，系统必须优先收口，而不是继续自由探索。
```

这份方案强调三件事：

- 要写出问题的核心结构。
- 要给出修复的理论依据。
- 要确保方案命中系统根因，而不是围绕症状填漏洞。

## 2. 问题的核心

### 2.1 不是预算小，而是闭环断裂

当前问题看起来像“预算太小”，但预算只是放大器。

真正核心是：

```text
系统没有把“当前回合问题”作为唯一裁决中心，
也没有把“证据已经足够回答”作为高优先级终止条件。
```

于是运行链路变成：

```text
用户问题
-> 主 Agent 自由补证
-> 子 Agent 返回局部结果
-> 主 Agent 继续自由补证
-> 预算耗尽
-> 输出预算文案或失败性回答
```

而正确链路应该是：

```text
用户问题
-> 主 Agent 明确当前回合裁决目标
-> 子 Agent/工具补齐缺口
-> 主 Agent判断是否已足够回答当前问题
-> 足够则强制收口
-> 不足才有限继续
```

### 2.2 当前结构的根本错误

当前结构里，主 Agent 被赋予了“无限制的探索自由”，但没有配套“收口责任约束”。

这会产生三个系统性副作用：

- 主 Agent 把“补证”误当成默认动作。
- 子 Agent 返回结果后，结果不会自动进入“裁决态”，而只是进入“更多推理材料”。
- 一旦历史未完成任务、失败摘要、候选证据同时存在，模型会把它们混成一个连续探索面。

因此问题不是单点 bug，而是控制论层面的角色边界不清。

## 3. 理论依据

本节说明为什么本计划的修复方向是结构正确的，而不是经验补洞。

### 3.1 单裁决者原则

在多 Agent 协作系统中，必须有且只有一个“当前回合最终裁决者”。

原因：

- 多个执行器可以并行提供证据。
- 但“是否已经足够回答用户当前问题”只能由一个统一裁决面判断。
- 如果没有唯一裁决者，就会出现多个局部目标争抢运行预算。

在本系统中，这个裁决者只能是主 Agent，而不能是任一子 Agent。

因此：

```text
子 Agent 的职责是完成边界化执行。
主 Agent 的职责是围绕当前用户问题做统一裁决。
```

这与当前代码中的系统定位是相符的：

- 子 Agent prompt 明确要求“只返回摘要、证据引用、产物引用、置信度和限制说明，不负责替主 Agent 做最终回答”
- 对应代码：`backend/orchestration/runtime_loop/agent_delegation_executor.py`

问题不在这个原则本身，而在主 Agent 收回结果后没有被强制进入裁决流程。

### 3.2 最小必要补证原则

当系统已经拿到足以回答当前问题的证据时，继续调用工具属于无效探索。

理论上，运行应遵循：

```text
工具调用不是默认行为，而是为补齐明确缺口服务。
```

因此下一次调用工具之前，系统必须先回答两个问题：

1. 当前问题是否已经可以回答？
2. 如果还不够，缺口是什么？

只有缺口明确存在，下一次工具调用才合理。

否则，继续调用工具只是把预算换成更多噪声。

### 3.3 当前问题优先原则

在连续对话中，历史未完成任务可以作为背景，但不能压过当前回合问题。

因为用户每一轮输入都在重新定义“本轮裁决目标”。

如果系统允许旧失败摘要继续占据显著上下文，就会产生：

- 本轮明明问 PDF 第 3 页，却先处理旧的 RAG 风险问题。
- 本轮明明问黄金价格，却继续扩展搜索而不是直接报值。

因此：

```text
历史失败状态只能作为弱提示，不能继续作为当前轮主任务。
```

### 3.4 失败态不可持久化为稳定知识原则

“委派被限流”“下一轮我再读”这类文本不是事实知识，而是一次运行状态。

如果把它们写进持续上下文，模型会把一次性的失败当成长期任务承诺。

这会破坏上下文层次：

- 事实
- 判断
- 运行状态
- 调试信息

被混为一谈。

因此运行失败摘要最多只能进入短期诊断层，不能进入长期决策层。

### 3.5 测试必须验证语义收口，而不是只验证链路存活

当前长测“passed”的依据主要是：

- 没有 failed checks
- 事件里没有 `error`

这只能说明链路没有崩。

但对 Agent 系统来说，更关键的是：

- 是否回答了当前问题
- 是否暴露内部预算/限流状态
- 是否把旧失败任务混入当前回答

因此语义收口本身必须成为一等测试目标。

## 4. 代码层结构诊断

### 4.1 预算控制是硬阈值，但没有强制收口分支

当前默认预算定义在：

- `backend/orchestration/runtime_loop/loop_control.py`

默认值：

- `max_turns = 8`
- `max_model_calls = 8`
- `max_runtime_seconds = 300`
- `max_events = 200`

当运行超限时，`task_run_loop.py` 会直接构造预算耗尽文本：

- `_build_runtime_budget_exhausted_message()`

这意味着系统当前做的是：

```text
到预算上限 -> 停止 -> 把“没收口”告诉用户
```

而不是：

```text
到预算上限 -> 禁止继续探索 -> 用已有证据做最后收口
```

### 4.2 子 Agent 回收结果后没有进入“裁决态”

当前子 Agent 结果会被写成 `tool_result` / `observation` 回到主 Agent。

这一步没有错。

问题在于：

- 主 Agent 收到结果后，仍按普通 followup 继续自由调用工具。
- 没有专门的“回收结果后裁决”状态机。

因此“子 Agent 返回结果”不会改变主 Agent 的运行姿态。

### 4.3 委派额度按回合计数，但不区分是否命中当前目标

当前主 Agent 默认：

- `max_delegate_calls_per_turn = 1`

对应：

- `backend/orchestration/agent_runtime_registry.py`
- `backend/orchestration/runtime_loop/agent_delegation_executor.py`

这套设计理论上没错，它是为了防止主 Agent无限制乱委派。

但当前缺少一个前提：

```text
唯一的一次委派必须优先服务当前回合主目标。
```

现在的问题是：

- 错误目标也会占掉这一轮唯一额度。
- 一旦第一发打偏，真正需要的子 Agent 就没有机会了。

### 4.4 对象绑定与自然语言委派割裂

PDF 子 Agent 读取对象主要依赖：

- `input_payload.file_path`
- `input_payload.path`
- `bindings.active_pdf`

对应：

- `backend/orchestration/runtime_loop/child_agent_runtime_executor.py`
- `backend/evidence/pdf_worker.py`

如果 instruction 里写了文件路径，但 `input_payload` 没带，仍可能触发 `missing_object_handle`。

这不是 budget 本身的问题，但会放大 budget 损耗，因为：

- 第一次委派失败
- 第二次想修正时又被委派额度挡住
- 主 Agent随后继续探索
- 最终烧完预算

### 4.5 失败性上下文过滤不完整

当前上下文过滤已经尝试识别部分 stale runtime summary：

- `backend/orchestration/runtime_loop/context_manager.py`

但只覆盖了少量 marker，例如：

- `max_delegate_calls_per_turn_exceeded`
- `下一轮继续`

没有稳定挡住自然语言失败摘要，例如：

- “委派被限流”
- “下一轮我会优先调用 PDF 第二部分”
- “本轮两次 PDF 委派均未成功”

因此失败性答案会被提升成后续回合的热上下文。

### 4.6 长测 warning 不会击穿 pass

当前：

- `answer_fallback_reason`
- `response.marker`

只会进入 `quality_warnings`

对应：

- `backend/tests/system_eval/long_runner.py`

最终 turn 是否通过的判定是：

```text
not failed_checks and "error" not in event_types
```

这意味着：

- 预算耗尽
- fallback answer
- 失败性对外文案

都不一定导致该轮失败。

这会让测试系统高估主 Agent 的真实语义能力。

## 5. 修复目标

本计划要达成的不是“让预算文案少一点”，而是以下五个结构性目标：

1. 主 Agent 成为当前回合唯一裁决者

2. 子 Agent 返回结果后，主 Agent 必须先进入裁决态，而不是默认继续探索

3. 当前问题优先于历史未完成任务，历史失败摘要不再抢占委派和预算

4. 预算超限时必须优先尝试基于现有证据收口，而不是直接向用户暴露内部预算状态

5. 测试系统必须把“链路存活”和“语义收口成功”分开判定

## 6. 目标架构

### 6.1 运行状态机重构

建议把当前运行姿态显式分成四态：

```text
Collecting
Judging
Answering
Blocked
```

含义：

- `Collecting`
  - 明确存在证据缺口，允许调用工具或委派子 Agent。
- `Judging`
  - 新证据刚返回，禁止立刻继续自由探索，先判断是否已足够回答当前问题。
- `Answering`
  - 已满足当前回合输出边界，直接生成最终答案。
- `Blocked`
  - 确认当前无法继续，但对用户输出必须是“基于现有证据的最稳妥结论或边界说明”，而不是内部预算说明。

### 6.2 子 Agent 回收后的强制裁决门

在任一 `delegate_to_agent` 或关键工具返回后，主 Agent 进入强制裁决门。

裁决门要回答：

1. 当前用户问题是否已经可以回答？
2. 如果不能回答，缺什么？
3. 下一次调用是否只针对这个缺口？

只有答案为“仍存在明确缺口”时，才允许回到 `Collecting`。

否则，直接进入 `Answering`。

### 6.3 当前回合主目标绑定

每一轮运行必须持有一个显式的 `turn_primary_goal`。

它不是泛泛的历史任务族，而是对本轮用户问题的结构化裁决目标，例如：

- “回答 PDF 第 3 页在全文中的作用”
- “直接报今日黄金价格”
- “用业务语言总结 AI 治理三类风险”

所有后续工具调用和委派都必须显式对齐这个目标。

如果一个调用无法证明自己服务于 `turn_primary_goal`，就不应被执行。

### 6.4 历史失败上下文降级

失败性运行摘要不能再进入 `Decisions and Learnings` 主区域。

应拆成：

- `durable_facts`
- `stable_preferences`
- `current_turn_evidence`
- `runtime_failures_debug_only`

其中：

- `runtime_failures_debug_only` 只供诊断，不再默认暴露给主 Agent作为后续决策主材料。

### 6.5 预算超限改为“强制收口优先”

预算控制触发后不再直接构造“预算耗尽”用户答案。

正确顺序应为：

1. 冻结继续探索
2. 提取当前轮已有稳定证据
3. 尝试生成最终答案
4. 只有在完全没有可用证据时，才输出边界说明

即：

```text
超限是停止补证的信号，不是停止回答的信号。
```

## 7. 分层修复方案

### 7.1 第一层：运行时裁决机制修复

修改重点：

- `backend/orchestration/runtime_loop/task_run_loop.py`
- 必要时补充独立裁决模块

措施：

- 引入 `post_observation_judgement` 阶段。
- 在 `delegate_to_agent`、`mcp`、`web_search`、`search_text` 等关键结果返回后，先做是否足够回答的判断。
- 一旦命中当前回合输出边界，直接短路进入最终回答。
- 不允许在没有缺口说明的情况下继续调工具。

理论依据：

- 单裁决者原则
- 最小必要补证原则

### 7.2 第二层：当前回合目标绑定修复

修改重点：

- `task_run_loop.py`
- 相关 plan/runtime directive 生成逻辑

措施：

- 为每回合生成显式 `turn_primary_goal`。
- 每次工具调用和委派请求都要附带该目标。
- 对下一次工具调用做“目标一致性”校验。
- 若候选调用主要服务旧问题，则拒绝或降权。

理论依据：

- 当前问题优先原则

### 7.3 第三层：委派额度与目标命中修复

修改重点：

- `backend/orchestration/runtime_loop/agent_delegation_executor.py`
- `backend/orchestration/agent_runtime_registry.py`

措施：

- 保留每回合委派上限这一设计，不轻易放开。
- 但要把“是否命中当前主目标”纳入委派前校验。
- 明确区分：
  - 命中主目标的首要委派
  - 因路径修正而产生的同目标重试
  - 完全偏离主目标的额外探索

必要时可允许“同目标修正重试”不计入普通探索委派额度。

理论依据：

- 限制不是目的，目标对齐才是目的。
- 我们要约束的是无边界探索，不是惩罚正确纠错。

### 7.4 第四层：对象绑定修复

修改重点：

- `backend/orchestration/runtime_loop/child_agent_runtime_executor.py`
- `backend/evidence/pdf_worker.py`

措施：

- 把主 Agent 已知的 `active_pdf`、`active_dataset`、当前页面/章节约束稳定下传给子 Agent。
- 对自然语言中已经明确出现的文件路径做结构化提取补全。
- 在进入正式委派前做绑定完整性预检。

理论依据：

- 子 Agent 是执行器，不应承担过多对象猜测责任。
- 委派输入不完整时，应在委派前被结构化修正，而不是让失败消耗配额和预算。

### 7.5 第五层：上下文污染修复

修改重点：

- `backend/orchestration/runtime_loop/context_manager.py`

措施：

- 扩大 stale runtime failure marker 的识别范围。
- 将“失败性运行总结”从可持续决策上下文中剥离。
- 仅保留稳定事实、稳定偏好、真实证据和必要产物引用。

理论依据：

- 失败态不可持久化为稳定知识原则

### 7.6 第六层：测试判定修复

修改重点：

- `backend/tests/system_eval/long_runner.py`
- 对应 warning regression / long scenario regression

措施：

- 将以下情况升级为失败或强阻断：
  - `runtime_budget_exhausted` 直接成为用户答案
  - 对外暴露“委派被限流”“下一轮我会再读”作为最终回答
  - 当前轮问题未回答，却输出运行状态说明
- 将“语义收口成功”纳入一等断言。

理论依据：

- 测试必须验证语义收口，而不是只验证链路存活

## 8. 不推荐的伪修复

以下做法看起来快，但本质会继续漂移：

1. 只把 `max_turns` 从 8 改到更大

问题：

- 会延缓失败，但不会修复自由探索和不收口。

2. 只把 `max_runtime_seconds` 提高

问题：

- 05 回合这类慢子 Agent 会继续吞预算，只是更晚暴露。

3. 只增加更多 prompt 提示“有结果就回答”

问题：

- 没有运行时结构约束时，prompt 提示只是弱建议。

4. 针对个别文案加字符串过滤

问题：

- 能藏住现象，不能修复状态机。

5. 遇到失败就简单禁用继续调用工具

问题：

- 这会把“需要补最后一个缺口”的合法场景一刀切掉。

## 9. 实施顺序

### 9.1 第一阶段：闭环与预算修复

目标：

- 建立“回收结果后强制裁决”
- 建立“预算超限优先收口”

完成标志：

- 03、04、05、42 不再向用户输出预算文案

### 9.2 第二阶段：目标绑定与上下文净化

- 18、39 这类旧失败复读问题消失

- `context_snapshot` 不再把失败性承诺送进主决策区
- 本轮工具调用主要服务当前问题

### 9.3 第三阶段：委派与对象绑定修复

## 10. 验收标准

### 10.1 运行时验收

- 子 Agent 返回结果后，主 Agent 默认先裁决，不再默认继续探索
- 当前问题已满足时，不再继续调用同类工具
- 预算超限时优先输出基于已有证据的收口答案

### 10.2 语义验收

- 不再向用户暴露：
  - “本轮运行预算达到上限”
  - “本轮委派全部被限流”
  - “下一轮我会优先调用……”
- 回答围绕当前轮问题，不夹带旧失败任务

### 10.3 长测验收

- 60 轮长测重新执行
- 关键异常回合重点复核：
  - 03
  - 04
  - 05
  - 06
  - 07
  - 18
  - 39
  - 42
- 通过标准不是“无 crash”，而是：
  - 当前问题得到回答
  - 不暴露内部预算或委派状态
  - 不被旧失败摘要污染

## 11. 最终判断

本次问题的核心，不是预算参数本身，也不是某个 PDF 子 Agent 单点实现，而是：

```text
主 Agent 在收到新证据后的系统职责没有被结构化固化为“先裁决、再决定是否继续”，
导致探索自由大于回答责任。
```

因此，修复必须命中：

- 主 Agent 的裁决地位
- 当前回合目标绑定
- 回收结果后的强制收口门
- 失败态上下文降级
- 语义测试收紧

只有这样，预算问题才会真正下降。

否则，任何单点阈值调整都会只是把问题往后拖。

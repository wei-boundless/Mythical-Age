# 任务图编排、装配与 RunLoop 一体化重构总计划书

日期：2026-05-08

## 0. 本轮纠偏结论

重新审视 `docs/设计原则/12-Agent-系统.md`、`13-内置Agent设计模式.md`、`14-任务系统.md`、`09-工具系统设计.md` 和当前代码后，本轮先纠正一个关键方向：

```text
系统不应该完全图化。
图只负责表达任务组织、执行拓扑、节点关系和运行推进。
Agent、Soul / Projection、Tool、Memory、Contract 本体不能被图吞掉。
```

正确边界是：

```text
TaskGraph
  负责：任务结构、入口、输出、节点、边、调度语义、交接语义。

Agent Orchestration
  负责：Agent 身份、内置/非内置、RuntimeProfile、AgentGroup、默认 Soul / Projection。

Soul / Projection
  负责：人格种子、投影卡、prompt 组织和运行视图，不归任务图常规分配。

Tool / Capability
  负责：工具注册、权限、安全边界、可调用能力。

Contract
  负责：输入、输出、节点、边、验收、运行边界。

RuntimeAssembly
  负责：把 TaskGraph 节点引用到的 Agent / Contract / Soul / Projection / Tool / Memory 装配成可运行节点。

GraphRunLoop
  负责：图级生命周期、节点调度、A2A 交接、human gate、kill、通知和 trace。

AgentLoop
  负责：单个 Agent 节点内部模型循环、工具调用、上下文、输出提交。
```

单 Agent 要纳入图，但含义是：

```text
单 Agent 任务 = 最小任务图：input -> agent -> output
```

不是：

```text
把 Agent 本体、工具系统、灵魂系统、记忆系统全部变成图节点。
```

---

## 1. 当前真实断点

### 1.1 前端是表层 TaskGraph，底层仍是 Coordination

当前 `TaskGraphWorkbench.tsx` 只是把参数转发给 `CoordinationEditorWorkbench`。

当前 `taskGraphDraft.ts` 也是从 `coordinationDraft + topologyDraft + protocolDraft` 派生 TaskGraph。

这说明 TaskGraph 还不是一等事实源，而是 coordination 的适配名。

### 1.2 后端没有一等 TaskGraph 注册层

当前事实源主要是：

```text
coordination_tasks.json
topology_templates.json
task_communication_protocols.json
```

缺少：

```text
task_graphs.json
TaskGraphDefinition
TaskGraphNodeDefinition
TaskGraphEdgeDefinition
TaskGraphRegistry API
```

### 1.3 运行态仍是 Coordination 专属

当前运行对象是：

```text
TaskRun
AgentRun
CoordinationRun
CoordinationNodeRun
AgentHandoffEnvelope
```

缺少通用：

```text
TaskGraphRun
TaskGraphNodeRun
TaskGraphEdgeRun
```

因此单 Agent 图、普通多 Agent 图、协调图会继续分裂。

### 1.4 RuntimeAssembly 是两套心智

当前存在：

```text
SingleAgentRuntimeAssembly
NodeRuntimeAssembly(coordination_task_ref)
```

目标应该是：

```text
TaskGraphRuntimeAssembly
AgentNodeRuntimeAssembly
```

单 Agent 只是 graph_kind=`single_agent` 的一个 Agent 节点。

## 2. 新目标

本轮目标不是把所有系统图化，而是让任务系统拥有一等图事实源：

```text
TaskDomain
  管理任务域、入口和任务清单。

TaskGraph
  统一表达单 Agent / 多 Agent / 协调任务的任务组织拓扑。

Agent Orchestration
  管理 Agent 身份、默认 Soul / Projection、RuntimeProfile、AgentGroup。

ContractManifest
  编译 Graph / Node / Edge / Runtime / Output / Acceptance 契约。

RuntimeAssembly
  将 TaskGraph 节点装配为可运行 Agent 节点。

GraphRunLoop
  管理图级生命周期、节点调度、A2A 交接、human gate、通知和 kill。

AgentLoop
  管理节点内部执行。
```

一句话：

```text
任务图是任务结构的唯一入口。
编排系统是 Agent 身份和能力的唯一入口。
RuntimeAssembly 是定义态到运行态的唯一装配入口。
GraphRunLoop 是图级推进入口。
AgentLoop 是节点内部执行入口。
```

## 3. 一等 TaskGraph 定义

### 3.1 TaskGraphDefinition

```text
graph_id
title
domain_id
task_family
graph_kind: single_agent | multi_agent | coordination
entry_node_id
output_node_id
nodes
edges
graph_contract_id
default_protocol_id
runtime_policy
context_policy
publish_state: draft | published | archived
enabled
metadata
```

### 3.2 TaskGraphNodeDefinition

```text
node_id
node_type: input | agent | tool | review | merge | human_gate | output
title
task_id
agent_id
agent_selection_policy
agent_group_id
work_posture
node_contract_id
input_contract_id
output_contract_id
runtime_lane
context_visibility_policy
projection_overlay_id
failure_policy
human_gate_policy
memory_writeback_policy
metadata
```

约束：

```text
agent_id 是引用，不是 Agent 本体定义。
projection_overlay_id 只是高级覆盖，不是常规投影分配入口。
```

### 3.3 TaskGraphEdgeDefinition

```text
edge_id
source_node_id
target_node_id
edge_type: handoff | review_feedback | parallel_join | retry | finalize
a2a_message_type
payload_contract_id
context_filter_policy
artifact_ref_policy
ack_policy
timeout_policy
failure_policy
metadata
```

约束：

```text
edge_type 是业务/拓扑语义。
a2a_message_type 是通信协议语义。
两者不能混成一个字段。
```

## 4. 固定实施顺序

### 阶段 1：建立一等 TaskGraph 事实源

目标：

```text
后端拥有 task_graphs.json 和 TaskGraphDefinition。
前端 overview 能看到 task_graph_management。
coordination 可以迁移为 TaskGraph 的兼容视图，但不能继续作为图事实源扩大。
```

文件：

```text
backend/tasks/task_graph_models.py
backend/tasks/flow_registry.py
backend/tasks/__init__.py
backend/api/tasks.py
frontend/src/lib/api.ts
```

验收：

```text
GET /tasks/overview 返回 task_graph_management.task_graphs。
新 TaskGraph 模型不依赖 CoordinationTaskDefinition。
```

### 阶段 2：前端 TaskGraphDraft 接管编辑真相

目标：

```text
TaskGraphWorkbench 不再只是 CoordinationEditorWorkbench wrapper。
图、节点、边检查器只读写 TaskGraphDraft。
```

文件：

```text
frontend/src/components/workspace/views/task-system/taskGraphTypes.ts
frontend/src/components/workspace/views/task-system/taskGraphDraft.ts
frontend/src/components/workspace/views/task-system/TaskGraphWorkbench.tsx
frontend/src/components/workspace/views/task-system/TaskGraphCanvas.tsx
frontend/src/components/workspace/views/task-system/TaskGraphInspector.tsx
frontend/src/components/workspace/views/TaskSystemView.tsx
```

验收：

```text
空图显示空态和阻断级问题。
单 Agent 模板生成 input -> agent -> output。
保存草稿写 TaskGraph，不写 coordination stack。
```

### 阶段 3：Agent 装配目录接入图节点

目标：

```text
节点选择 Agent 时能看到 AgentDescriptor、RuntimeProfile、默认 Soul / Projection、权限和上下文摘要。
```

验收：

```text
节点不再只显示 agent_id。
任务图不编辑 Agent 本体，只引用 Agent。
```

### 阶段 4：ContractManifest 改为编译 TaskGraph

目标：

```text
契约编译输入从 workflow/coordination 两套收敛到 TaskGraph。
```

新增：

```text
compile_task_graph_contract_manifest()
CompiledGraphContract
CompiledNodeContract
CompiledEdgeHandoffContract
```

验收：

```text
issue 能定位 graph_id / node_id / edge_id / agent_id / contract_id。
```

### 阶段 5：RuntimeAssembly 收敛

目标：

```text
单 Agent 和多 Agent 节点使用同一套节点装配模型。
```

新增：

```text
TaskGraphRuntimeAssembly
AgentNodeRuntimeAssembly
```

验收：

```text
single_agent 图和 multi_agent 图输出同结构 assembly。
```

### 阶段 6：GraphRun / NodeRun 运行态落地

目标：

```text
运行态从 coordination-only 提升为通用 TaskGraphRun。
```

新增：

```text
TaskGraphRun
TaskGraphNodeRun
TaskGraphEdgeRun
```

验收：

```text
一次图任务运行必有 TaskGraphRun。
每个图节点必有 TaskGraphNodeRun。
```

### 阶段 7：单 Agent 迁入 GraphRunLoop

目标：

```text
单 Agent 不再是独立心智，而是最小图。
```

验收：

```text
单 Agent trace 能看到 input / agent / output 三个节点。
Agent 节点内部仍走 AgentLoop。
```

### 阶段 8：多 Agent / A2A / human gate 收敛

目标：

```text
多 Agent 图由 GraphRunLoop 推进，边通过 A2A 载荷交接。
```

验收：

```text
两节点图可完成 handoff。
human_gate 可暂停和恢复。
kill graph 可级联终止节点运行态。
```

### 阶段 9：清理 coordination-only 残留

清理范围：

```text
前端用户可见 coordination-only 文案。
旧 single_agent_chain / coordination_chain 二分心智。
旧 topology fallback。
旧 coordination-only API wrapper。
无入口、无测试、无用途残留。
Shadow 相关残留。
```

## 5. 禁止事项

1. 禁止把 Agent 本体、Soul / Projection 本体、Tool 本体塞进图。
2. 禁止图节点只写 `agent_id` 而不展示装配边界。
3. 禁止任务系统常规分配投影；投影默认归 Agent，节点只允许高级 overlay。
4. 禁止图修改直接发布运行，必须保存草稿再预检发布。
5. 禁止新增 coordination-only 功能。
6. 禁止 GraphRunLoop 把节点完整过程塞进主会话。
7. 禁止再引入 Shadow 模式作为兼容叙事。

---

## 6. 阶段回顾规则

每完成一个阶段必须回顾：

```text
是否仍坚持“图只负责任务组织”？
是否误把 Agent / Soul / Tool / Memory 本体塞进图？
是否减少了 coordination-only 心智？
是否产生了新的双轨事实源？
是否有测试或真实检查支撑？
```

## 8. 本轮收尾标准

```text
TaskGraph 成为任务结构一等事实源。
单 Agent 和多 Agent 都通过 TaskGraph 表达。
Agent / Soul / Tool / Memory 保持独立系统边界。
RuntimeAssembly 能从 TaskGraph 装配节点。
GraphRunLoop 能运行 single_agent 最小图。
GraphRunLoop 能推进至少一个两节点 A2A 图。
前端能编辑、预检、发布、查看运行态。
旧 coordination-only 不再作为用户主心智入口。
```

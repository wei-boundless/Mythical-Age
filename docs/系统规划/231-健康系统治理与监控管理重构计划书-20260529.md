## 1. 背景

健康系统已经从旧的测试、实验、诊断混合工作台，初步收束为 Agent 运行治理系统。但现有代码仍有四类结构性问题：

1. 健康治理主链路与 `TaskRun` 模型发生契约漂移，`backend/health_system/governance.py` 读取不存在的 `task_run.runtime_lane`，导致 `backend/tests/health_governance_token_usage_test.py` 失败。
2. 运行监控健康结论可纳入健康系统汇总域，但实时运行监控页面已经由主页面承担；健康系统应建立 `HealthMonitorGovernance`，只汇总 monitor 事实、风险升级和建议，不新增运行监控页。
3. 系统风险覆盖不足，尚未接入固定端口、API base、SSE、sandbox、模型服务和工具运行风险。
4. 任务记录管理应归健康系统，但当前 `task-records/prune` 语义过粗，缺少预检、影响范围、审计回执和事实账本保护；健康 agent 会话/执行控制面仍属于越权入口。

本计划目标是把健康系统重构为：

```text
Agent 运行治理观察层
+ 诊断决策投影层
+ 运行监控健康汇总层
+ 健康问题与报告登记层
```

同时保持 runtime monitor 仍拥有实时状态机和运行快照事实。

## 2. 成熟架构标准

本次重构采用 observability / SRE 分层原则：

```text
事实源和治理判断分离。
监控投影和健康解释分离。
原始事件和聚合视图分离。
执行动作和治理建议分离。
指标口径必须稳定、可追溯、可解释。
```

对应到本项目：

```text
RuntimeStateIndex owns TaskRun facts.
RuntimeEventLog owns runtime trace facts.
TaskRunMonitorProjector owns lifecycle/bucket/stale/action_required.
PromptAccountingLedger owns token usage truth.
RuntimeEnvironment/ConnectionHealth owns environment health facts.
HealthSystem owns governance interpretation, risk escalation, health profiles, reports.
```

禁止用健康系统重新实现 runtime monitor 状态机，也禁止为了修测试硬编码输出或跳过断言。

## 4. 目标架构

### 4.1 后端模块

目标 `backend/health_system`：

```text
backend/health_system/
  governance.py
    聚合健康系统主 view model。

  signals.py
    从 runtime monitor、event log、token ledger、environment health 读取 HealthSignal。

  task_profiles.py
    生成 HealthTaskProfile。只消费 TaskRun/MonitorItem/EventLog/TokenLedger。

  cost_profiles.py
    生成 HealthCostProfile。只读取 PromptAccountingLedger。

  monitor_governance.py
    生成 HealthMonitorGovernance。只消费 RuntimeMonitor view。

  system_risks.py
    生成 HealthSystemRisk。消费 monitor/environment/tool/model/sandbox signals。

  risk_classifier.py
    将 signals/profile 转成 HealthRiskEvent。

  registry.py
    管 HealthIssue / HealthReport / Receipt。

  store.py
    只存健康系统自己的 issue/report/receipt/legacy conversation 数据。
```

第一轮实施可以不一次性拆满所有文件，但最终代码不得继续把无审计 prune、agent execution planner、runtime admission 当作健康系统主链路。

## 6. 迁移与切换规则

### 6.1 兼容窗口

只允许以下兼容：

```text
历史 HealthIssue / HealthReport / Receipt 数据继续可读。
历史 HealthAgentRun / conversation 数据只读可查。
trace_estimate 旧 token 数据保留为旧任务回退标记。
短期保留的 legacy POST endpoint 只能返回 disabled/blocked，不能产生 runtime side effect。
```

不允许以下兼容：

```text
保留健康测试系统入口。
保留无预检、无审计、无保护条件的健康系统 prune 事实账本能力。
保留可点击但永远 blocked 的健康 agent 执行按钮。
用旧 runtime_lane 字段继续影响治理判断。
保留 health-local runtime lane/admission/evidence 执行链路作为主链路。
```

### 6.2 切换方式

本重构不采用双主链路。

每个阶段完成后立即切到新链路：

```text
Phase 1 修复健康主链路。
Phase 2 监控治理从 HealthMonitorGovernance 输出。
Phase 3 系统风险从 system_risks 输出。
Phase 4 任务记录管理改为健康系统受控维护。
Phase 5 agent execution UI 降级。
Phase 6 profiles 成为治理装配源。
Phase 7 前端只消费新 view model。
```

### 6.3 回滚规则

允许回滚：

## 9. 完成标准

本重构完成后必须满足：

1. 健康系统测试全部通过，当前 token usage 失败消失。
2. 健康系统不再无预检、无审计、无保护条件地删除 TaskRun/EventLog/PromptAccounting。
3. 运行监控健康结论作为健康系统汇总域出现，但不新增健康系统运行监控页，lifecycle/bucket/stale/action_required 仍由 RuntimeMonitor 决定。
4. 系统风险能覆盖端口、API base、SSE、runtime monitor diagnostics、token ledger、工具/模型/sandbox 风险。
5. 固定端口被无关进程占用时不会误判 healthy。
6. 健康 agent 未重建前，前端不展示可点击执行入口，后端 POST 执行入口不创建 runtime run。
7. 旧 health-local runtime lane/admission/evidence 执行链路不再作为主链路存在。
8. 前端健康系统只消费后端 view model，不重新实现核心治理判断。
9. 真实前后端固定端口联调通过。

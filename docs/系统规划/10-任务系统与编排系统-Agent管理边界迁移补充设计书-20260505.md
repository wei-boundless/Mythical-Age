# 任务系统与编排系统 Agent 管理边界迁移补充设计书

日期：2026-05-05

## 一、文档定位

本文件用于补充修正任务系统与编排系统之间关于 Agent 管理、子 Agent 派生、多 Agent 协作和通用任务定义的边界。

本文件优先级高于旧文档中仍把 Agent 管理、Agent 采用计划、协调拓扑和 handoff 协议放在任务系统中的历史表述。

## 二、核心结论

任务系统只负责管理任务，不再负责具体 Agent 的管理和装配。

编排系统正式接管：

- Agent 名册
- Agent 分类
- Agent runtime profile
- Worker / 子 Agent blueprint
- 任务到 Agent 的绑定策略
- 多 Agent 协作拓扑
- coordinator / participant 选择
- handoff、review、merge、feedback loop 协议

任务系统保留：

- 任务定义
- 任务分类
- 通用任务与特定任务
- workflow
- projection 策略
- flow contract
- 记忆请求
- 能力要求
- 安全要求
- 验收要求
- 执行意图

## 三、通用任务与特定任务

### 3.1 通用任务

通用任务是不需要显式调用的任务。

它由 Agent 根据上下文自主承接，适用于普通会话、轻量理解、轻量整理、简单问答和不需要进入正式任务态的请求。

通用任务不应该暴露成普通用户必须选择的任务登记项。

### 3.2 特定任务

特定任务是需要显式任务化治理的任务。

它必须具备：

- 任务定义
- 输入输出契约
- workflow
- 能力要求
- 安全要求
- 验收要求
- 执行意图
- 编排策略引用

例如小游戏开发、小说协作写作、复杂重构、正式验收，都属于特定任务。

## 四、执行意图模型

任务系统不再直接保存“默认 Agent 是谁”作为核心语义，而是保存执行意图。

建议执行意图：

- `implicit_general`：隐式通用承接
- `single_agent_required`：需要单 Agent 完成
- `subagent_allowed`：允许编排系统按需派生子 Agent
- `coordination_required`：需要多 Agent 协作

编排系统根据执行意图、能力要求、安全要求和验收要求，解析成具体 Agent、runtime profile、拓扑和 handoff 结构。

## 五、前端边界

任务系统前端只保留：

- 任务管理
- 通用任务
- 特定任务
- 执行意图
- workflow
- projection 策略
- 能力 / 安全 / 验收要求
- 任务侧装配预览

任务系统前端不再包含：

- Agent 创建
- Agent 删除
- Agent 编辑
- Agent 分类治理
- worker blueprint 管理
- coordinator / participant 显式装配
- topology / handoff protocol 管理

编排系统前端接管上述所有 Agent 与协作装配能力。

## 六、后端接口边界

任务系统 API 不应继续暴露：

- `/tasks/agents/*`
- Agent CRUD
- next worker agent id
- `agent_management` payload

编排系统 API 应暴露：

- `/orchestration/agents`
- `/orchestration/agents/{agent_id}`
- `/orchestration/agents/next-worker-id`
- `/orchestration/agent-runtime-profiles`
- `/orchestration/agent-binding-policies`
- `/orchestration/coordination-topologies`

## 七、验收标准

迁移完成后应满足：

- 任务系统页面看不到 Agent 管理页。
- 任务系统接口不再返回 `agent_management`。
- 任务系统仍可引用编排策略，但不能创建或删除 Agent。
- 编排系统页面可以管理 Agent 名册和 runtime profile。
- 编排系统可以解释某个任务执行意图最终会绑定到什么 Agent 与协作结构。
- 通用任务被表达为隐式承接，而不是用户必须显式登记的任务。
- 特定任务被表达为显式任务化执行，具备完整验收要求。

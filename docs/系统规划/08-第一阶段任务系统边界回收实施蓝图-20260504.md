## 二、当前真实问题

第一阶段的核心问题不是“task builder 太长”，而是：

`任务系统的正式边界已经被 contract_builder 穿透了`

当前 [backend/tasks/contract_builder.py](/d:/AI应用/langchain-agent/backend/tasks/contract_builder.py:28) 里的 `build_task_runtime_contract` 同时承担了五类职责：

1. 任务理解与模板命中
2. 任务对象装配
3. 能力/操作请求推导
4. 投影选择与投影素材读取
5. prompt 合成前的灵魂运行视图生成

其中只有前 3 类属于任务系统稳定职责。

第 4、5 类已经越界进入：

- 灵魂系统
- 编排系统

这会直接导致三个结构性问题：

1. 任务系统变成跨系统总装配器
2. 灵魂系统和编排系统没有独立正式输入边界
3. runtime 以后即使想只消费正式装配结果，也拿不到清晰分层的对象

## 四、第一阶段拆分后的目标边界

第一阶段不要求一次性做完整编排系统，只要求把任务系统边界回收到正确位置。

拆分后的目标是：

## 4.1 任务系统正式输出

任务系统在这一阶段只允许输出以下三类正式对象：

### 4.1.1 TaskIntentContract

表示：

- 用户当前目标被任务系统理解成什么

来源保留：

- `backend/tasks/match_contracts.py`

### 4.1.2 TemplateMatchResult

表示：

- 任务系统命中了哪个模板

来源保留：

- `backend/tasks/match_contracts.py`

### 4.1.3 TaskExecutionAssembly

这是第一阶段最重要的新正式对象。

它应替代当前 `build_task_runtime_contract` 返回的大字典中的任务系统部分。

建议字段：

- `assembly_id`
- `task_id`
- `session_id`
- `task_family`
- `task_mode`
- `task_intent_ref`
- `template_match_ref`
- `task_spec_ref`
- `bundle_spec_ref`
- `selected_agent_id`
- `workflow_id`
- `projection_selection`
- `operation_requirement_ref`
- `input_contract_id`
- `output_contract_id`
- `safety_envelope`
- `requested_outputs`
- `status`

这里特别注意：

- `projection_selection` 只表达任务侧选择结果或选择要求
- 不表达 projection prompt 文本本体
- 不表达 soul runtime view
- 不表达 prompt manifest

## 4.2 灵魂系统正式输入

任务系统拆分后，灵魂系统应改为消费单独的任务侧投影请求，而不是读任务系统拼好的 prompt contract。

第一阶段建议新增中间对象：

### 4.2.1 ProjectionSelectionResult

表示：

- 任务系统对本次任务选择了哪个 projection
- 或任务系统对本次任务提出了什么 projection requirement

建议字段：

- `task_id`
- `selected_projection_id`
- `role_type`
- `posture_tags`
- `selection_reason`
- `selection_source`

第一阶段允许它仍由任务系统生成。

但灵魂系统只应根据它去取素材，不应反向要求任务系统直接拼 prompt section。

## 4.3 编排系统正式输入

第一阶段先不完整实现编排系统，但必须为它留下清晰输入边界。

编排系统后续至少应接收：

- `TaskExecutionAssembly`
- `ProjectionSelectionResult`
- `MemoryBundle`
- `CapabilitySupplyPackage`

第一阶段先做到：

- 任务系统不再自己把后两步提前做掉

## 六、第一阶段新的执行流

第一阶段拆完后，任务系统执行流应固定为：

```text
query_understanding / current_turn_context
  -> TaskIntentContract
  -> TemplateMatchResult
  -> RegisteredTask resolution
  -> TaskDefinition alignment
  -> TaskSpec / BundleSpec / StepInputBindings
  -> TaskWorkflow resolution
  -> OperationRequirement request
  -> ProjectionSelectionResult
  -> TaskExecutionAssembly
```

第一阶段明确禁止继续出现的流：

```text
TaskExecutionAssembly
  -> task system builds soul runtime view
  -> task system builds prompt manifest
  -> task system returns runtime-ready prompt bundle
```

## 七、第一阶段禁止事项

为了防止边拆边回退，第一阶段明确禁止：

1. 在新 builder 里继续直接 import `soul.projection`
2. 在任务系统里继续构造 `TaskPromptContract`
3. 在任务系统里继续生成 `prompt_manifest`
4. 在任务系统里继续拼接 model-visible prompt section
5. 以兼容为理由把旧 `contract_builder.py` 继续扩成更大的总装配器

## 八、迁移方式

第一阶段建议采用“影子替换”而不是一次性硬切。

## 8.1 影子阶段

新增：

- `assembly_models.py`
- `assembly_builder.py`

先让它们输出新对象，同时保留旧 `contract_builder.py` 作为兼容出口。

但旧出口应改为：

- 调新任务装配对象
- 再走兼容转换

而不能继续反过来让新对象依赖旧跨层大字典。

## 8.2 切换阶段

当以下条件满足后：

- runtime 入口已能消费 `TaskExecutionAssembly`
- 灵魂系统入口已能消费 `ProjectionSelectionResult`

再删除旧 `contract_builder.py` 中跨层输出字段：

- `task_prompt_contract`
- `soul_runtime_view`
- `soul_projection_request`
- `prompt_manifest`
- `agent_prompt_bundle`

## 8.3 回滚规则

第一阶段如果中途发现 runtime 或上层 API 暂时强依赖旧字段，只允许：

- 在兼容适配层临时转换

不允许：

- 把跨层逻辑重新塞回任务系统正式 builder

## 九、阶段完成标准

第一阶段完成必须同时满足：

1. 任务系统存在独立正式对象 `TaskExecutionAssembly`
2. 任务系统存在独立正式对象 `ProjectionSelectionResult`
3. 任务系统主 builder 不再直接 import `soul.projection`
4. 任务系统主 builder 不再生成 `TaskPromptContract`
5. 任务系统主 builder 不再生成 `prompt_manifest`
6. 任务系统主 builder 的输出可以被编排系统继续消费

只满足“文件拆开了”不算完成。

必须以系统边界被回收为完成标准。

## 十、执行结论

第一阶段的本质不是“优化一个大文件”，而是：

`把任务系统从跨系统总装配器，收回为真正的任务调度与任务装配请求系统`

这一阶段一旦完成，后面三条线就会明显变顺：

- 灵魂系统可以独立接管身份层素材生成
- 编排系统可以独立接管 prompt / resource / memory 装配
- runtime 可以逐步收回到只执行正式运行体对象

因此第一阶段必须优先做，而且必须做硬，不应再以兼容为理由继续保留错误边界。

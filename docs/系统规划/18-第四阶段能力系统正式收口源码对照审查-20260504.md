# 第四阶段能力系统正式收口源码对照审查

日期：2026-05-04

## 一、本轮收口前已经完成的结构性进展

第四阶段前半段已经完成了能力系统的第一轮正名与目录迁移，主要包括：

- `操作系统` 已正式更名为 `能力系统`
- tool / skill / mcp / registry 的治理根目录已统一到 `backend/capability_system/`
- 具体能力单元已迁移到：
  - `backend/capability_system/units/skills/`
  - `backend/capability_system/units/mcp/`
  - `backend/capability_system/units/registries/`
- 原先混在 `backend/tools/`、`backend/skills/`、`backend/mcp/`、`backend/skill_system/` 的治理代码已基本抽离
- `backend/tools/` 已回到“具体工具执行器”定位

这说明第四阶段前半段已经解决了：

`能力代码放在哪里`

但还没有完全解决：

`能力系统究竟以什么正式对象对外供给`

## 二、当前已经符合设计书的部分

### 2.1 能力系统已经成为独立资源供应层

当前源码已经能稳定输出以下正式资源：

- `skills`
- `tools`
- `workers`
- `capability_endpoints`
- `operations`
- `binding_graph`

这意味着能力系统已经不再只是一个工具目录，而是具备：

- 资源登记
- 资源描述
- 资源绑定关系
- 资源结构校验

等正式系统特征。

### 2.2 worker / endpoint 已经从“隐式执行细节”变成正式对象

当前源码中：

- `build_worker_catalog`
- `build_capability_endpoints`

已经把后台 worker 与本地 MCP 暴露关系正式建模出来。

这部分是和设计书一致的，因为能力系统不仅要知道“有哪些直接 tool”，也要知道：

`有哪些不直接暴露给模型、但可被编排系统调度的能力单元`

### 2.3 能力系统的目录化治理已经成型

目前 skill、mcp、registries 的落点已经较稳定，前后端工作台也已经围绕能力系统目录建立管理入口。

这说明能力系统已经不再依赖历史散落目录生存。

## 三、仍然存在的结构性问题

### 3.1 `catalog.py` 仍然职责过重

当前 `backend/capability_system/catalog.py` 同时承担了两类不同职责：

- 目录投影与绑定图生成
- skill frontmatter 的读取、解析、改写

这会带来一个结构问题：

`能力目录对象` 和 `能力资源编辑行为` 被混在同一个模块里

这不符合正式系统分层。

能力目录应当是只读投影层；skill 编辑应当进入独立 authoring / editing 模块。

### 3.2 能力系统仍然反向依赖任务内部定义

当前 `catalog.py` 仍通过 `tasks.definitions.default_task_definitions()` 读取任务默认操作需求，再把它并入能力系统校验。

这会造成边界倒置：

- 正确方向应是：任务系统依赖能力系统
- 当前残留方向却是：能力系统反向读取任务内部

这不符合设计书中“能力系统只负责供给，不定义任务”的原则。

### 3.3 `CapabilitySupplyPackage` 还没有成为正式对象

当前能力系统虽然能输出：

- catalog
- registry
- endpoints

但还缺一个更贴近运行时交付语义的正式对象：

`CapabilitySupplyPackage`

没有这个对象时，任务系统与编排系统拿到的仍更像“平台能力目录”，而不是“本次任务可消费的能力供给包”。

### 3.4 派生产物仍被前端当作可编辑文件入口

当前：

- `capability_system/units/registries/SKILLS_SNAPSHOT.md`

仍出现在前端固定文件入口中；

同时 `api/files.py` 对 `capability_system/units/` 整体放行，等于把：

- `skills/`
- `mcp/`
- `registries/`

都视作可写区域。

这会模糊一个关键边界：

- `skills/` 与 `mcp/` 是能力单元本体
- `registries/` 是由扫描与生成过程产出的派生索引

派生产物应允许读取，但不应再被视作人工编辑源。

## 四、本轮正式收口后应达到的状态

本轮收口后，能力系统应满足以下四点：

### 4.1 `catalog` 回到只读目录层

`catalog.py` 只负责：

- catalog 构建
- metadata 投影
- binding graph
- validation summary

不再负责 skill frontmatter 编辑。

### 4.2 能力系统不再反向读取任务定义

任务需求校验仍可存在，但应留在：

- 任务系统
- 编排系统
- 健康系统

能力系统自身只校验：

- 资源是否存在
- 资源之间绑定是否一致
- endpoint / worker / operation 是否匹配

### 4.3 正式补出 `CapabilitySupplyPackage`

能力系统需要能正式输出：

- 平台级能力供给包
- 面向特定 operation scope 的裁剪供给包

这样任务系统和编排系统后续才能消费：

`供给对象`

而不是继续直接耦合到底层 registry builder。

### 4.4 `registries/` 转为只读派生产物

`registries/` 只允许：

- 扫描生成
- 后端读取
- 前端展示

不再作为人工编辑入口参与工作台写回。

## 五、本轮建议实施项

建议本轮直接落以下改动：

1. 新增 `18-第四阶段能力系统正式收口源码对照审查-20260504.md`
2. 拆出 `backend/capability_system/skill_authoring.py`
3. 新增 `backend/capability_system/supply.py`
4. 在 `models.py` 中补出 `CapabilitySupplyPackage` 及相关 ref 对象
5. 从 `catalog.py` 中删除对 `tasks.definitions` 的反向依赖
6. 收紧 `api/files.py` 的写白名单，只允许写入能力单元本体目录
7. 从前端固定编辑文件清单中移除 `SKILLS_SNAPSHOT.md`
8. 为能力供给包与只读 registry 边界补定向回归

## 六、收口结论

第四阶段前半段解决的是：

`能力系统代码和目录放哪`

第四阶段后半段真正要解决的是：

`能力系统到底以什么正式对象对外供给，以及哪些东西是资源本体，哪些只是派生产物`

如果这一层不补齐，后续任务系统、编排系统、能力工作台仍然会继续耦合到：

- catalog 内部实现
- registry 生成细节
- 可编辑目录与派生目录的混合入口

因此本轮收口重点不是继续增加功能，而是：

`把能力系统从“已经迁了目录的治理集合”推进成“正式资源供应系统”`

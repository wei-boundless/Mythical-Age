# 194-任务图运行健康系统接入实施计划

日期：2026-05-20

## 1. 目标

把已经升级过的任务图运行体系接入重建后的健康系统，使健康系统能直接消费任务图运行证据，识别拓扑、契约、时序、批次、派发和恢复边界上的结构性问题。

本阶段不做写作任务专用门禁，不新增业务私门。健康系统只读取通用 `task_graph.run_monitor`、runtime trace、coordination checkpoint 和 batch lifecycle，生成可复核的健康投影、证据包和恢复候选。

## 2. 当前源码事实

### 2.1 已有任务图运行出口

- `backend/orchestration/runtime_loop/task_graph_run_monitor.py`
  已提供 `task_graph.run_monitor`，包含拓扑、节点状态、边状态、时序边界、timeline、batch lifecycle、batch dispatcher、当前节点请求和内置 health issues。
- `backend/orchestration/runtime_loop/trace_reader.py`
  `RuntimeLoopTraceReader.get_task_graph_run_monitor()` 可以从 task run 或 coordination run 还原任务图运行监控视图。
- `backend/orchestration/runtime_loop/task_graph_monitoring.py`
  已有 monitor snapshot 的运行裁决能力，但它主要服务运行控制和 supervision record，不是健康系统的证据包。

### 2.2 已有健康系统出口

- `backend/health_system/maintenance/test_system/service.py`
  `get_task_run_monitor()` 当前只返回 `summarize_runtime_loop_trace()` 的普通 runtime-loop 摘要。
- `backend/health_system/evidence_extractor.py`
  已能从 runtime trace 提取事件证据和 coordination recovery handles，但没有把 `task_graph.run_monitor` 的结构化问题作为一等证据源。
- `backend/api/health_system.py`
  已有 `/health-system/maintenance/test-system/runtime-loop/task-runs/{task_run_id}/monitor`，但尚无任务图健康投影 API。

## 3. 缺口

1. 健康系统能看到 runtime-loop 摘要，但看不到任务图拓扑、节点/边、批次、派发槽位和时序边界的完整健康投影。
2. `task_graph.run_monitor.health.issues` 还只是监控视图内字段，没有进入健康系统 evidence packet。
3. batch dispatcher、batch lifecycle 中的异常状态尚未形成健康问题：
   - 有 ready batch 但没有可派发槽位。
   - active execution 与 running batch 不一致。
   - unknown batch execution result 被忽略。
   - merge 已满足但运行没有向后推进。
4. 恢复候选目前主要来自 checkpoint / failing nodes，还没有包含 task graph monitor、batch execution、dispatch request 的安全边界。

## 4. 设计原则

1. 健康系统只做观察、诊断、证据组织和恢复候选，不直接改写任务图运行状态。
2. 任务图运行事实以 `task_graph.run_monitor` 为准，普通 runtime-loop trace 只补充事件和 checkpoint。
3. 健康投影必须是通用结构，不出现写作、章节、小说等业务字段。
4. 任何运行问题都要指向明确对象：graph、node、edge、timeline、batch、execution、request、checkpoint。
5. 健康系统的输出要能被人和 agent 同时读懂：摘要给人看，证据和恢复句柄给 agent / runtime control 使用。

## 5. 目标设计

### 5.1 新增任务图健康投影

新增 `backend/health_system/maintenance/test_system/task_graph_health.py`。

核心函数：

```text
build_task_graph_health_projection(monitor, trace=None, question="")
```

输入：

- `monitor`: `task_graph.run_monitor`
- `trace`: 可选 runtime trace
- `question`: 证据包问题

输出：

- `authority`: `health_system.task_graph_health_projection`
- `task_run_id`
- `coordination_run_id`
- `graph`
- `summary`
- `issues`
- `batch_health`
- `recovery_handles`
- `evidence_packet`

### 5.2 健康问题分类

从 monitor 生成通用 issue：

- `monitor_health_issue`: 原生 `monitor.health.issues`
- `runtime_failure`: runtime failure detail
- `batch_dispatch_blocked`: 有 ready batch 但无可派发槽位
- `batch_execution_identity_ignored`: 批次结果身份无法归位
- `batch_execution_inconsistent`: running batch / active execution / active request 不一致
- `timeline_violation`: 时序边界违规
- `missing_runtime_request`: 运行中节点缺少标准执行请求

每条 issue 至少包含：

- `issue_id`
- `severity`
- `code`
- `subject_type`
- `subject_id`
- `summary`
- `evidence_refs`
- `recommended_action`

### 5.3 证据包

任务图健康投影将把任务图问题转成 `EvidenceCandidate`：

- `source_kind`: `task_graph_run_monitor`
- `subject_type`: `task_graph` / `task_graph_node` / `task_graph_edge` / `task_graph_batch` / `task_graph_execution`
- `source_ref`: task run 或 coordination run
- `event_type`: issue code
- `summary`: 人可读问题摘要
- `metadata`: graph id、node id、batch id、execution id、dispatcher 状态等

证据包仍使用现有 `health_system.evidence_packet`，不另起新协议。

### 5.4 API 接入

新增健康系统 API：

```text
GET /health-system/maintenance/test-system/task-graph/task-runs/{task_run_id}/health
```

实现路径：

1. 通过 `runtime.query_runtime.task_run_loop.get_task_graph_run_monitor(task_run_id)` 获取任务图 monitor。
2. 同时读取 runtime trace。
3. 调用 `TestSystemService.get_task_graph_health()`。
4. 返回健康投影。

原 runtime-loop monitor API 保持不变，避免混淆普通 runtime-loop 摘要和任务图健康投影。

## 6. 实施步骤

### 阶段一：健康投影模型

文件：

- `backend/health_system/maintenance/test_system/task_graph_health.py`
- `backend/tests/task_graph_health_projection_regression.py`

完成标准：

- 可以从纯 monitor dict 生成 issues、summary、evidence packet、recovery handles。
- 不依赖真实文件系统。
- 覆盖 topology / temporal / batch 三类问题。

### 阶段二：服务与 API 接入

文件：

- `backend/health_system/maintenance/test_system/service.py`
- `backend/api/health_system.py`
- `backend/tests/task_system_api_regression.py` 或新增健康 API 测试

完成标准：

- test system service 能读取任务图 monitor 并返回健康投影。
- API 能返回 task graph health。
- task run 不存在或不是任务图运行时返回明确 404。

### 阶段三：回归验证与报告

文件：

- `docs/系统规划/195-升级后任务图系统技术报告-20260520.md`

完成标准：

- 后端新增/相关回归通过。
- 计划书记录完成项。
- 技术报告说明升级后任务图系统的真实能力、边界、健康接入方式和下一阶段建议。

## 7. 不做事项

- 不把健康系统变成任务图控制器。
- 不自动恢复、自动重跑、自动改图。
- 不写业务专用质量门。
- 不把普通 runtime-loop monitor API 改成任务图专用。
- 不隐藏 monitor 原始问题，健康投影只做归一和证据组织。

## 8. 验证矩阵

至少验证：

1. monitor 有原生 health issue 时，健康投影生成对应 issue 和 evidence。
2. temporal violation 会生成 critical issue 和恢复候选。
3. batch dispatcher 有 ready batch 且可派发槽位为 0 时，生成 warning/info 级容量问题而不是误判失败。
4. `last_transition_ignored.reason=batch_execution_identity_not_found` 会生成 high severity issue。
5. API 能从真实 `TaskRunLoop` 测试实例返回投影。

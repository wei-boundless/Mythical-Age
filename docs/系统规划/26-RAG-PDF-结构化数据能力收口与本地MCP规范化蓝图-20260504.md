`能力已经登记进能力系统，但能力实现本体仍散落在系统外历史目录中`

## 二、当前真实问题

当前源码里，三项能力已经在能力系统中有登记入口：

- tool
- worker
- skill

但它们的真实实现本体仍然分散在历史目录：

- `backend/RAG`
- `backend/pdf_agent`
- `backend/pdf_analysis`
- `backend/structured_data`

这会造成四个持续问题：

1. 能力系统只登记“入口”，没有拥有“能力本体”
2. 本地 MCP 在 catalog 中只有 endpoint 概念，没有正式能力单元概念
3. tool 与 worker 各自直连实现目录，形成平行依赖
4. 历史目录会继续被误认为独立子系统，而不是能力系统内部能力域

所以本次要解决的不是命名问题，而是正式归属问题：

`把三项能力从“系统外挂实现”收回为“能力系统内部的本地 MCP 能力单元”`

## 三、正式设计结论

## 3.1 三项能力的正式归属

这三项能力的正式归属统一定义为：

`能力系统 / MCP 能力单元层 / 本地 MCP 能力域`

即：

- 它们不是任务系统
- 它们不是编排系统
- 它们不是健康系统
- 它们也不是外部远程 MCP server

它们是：

`能力系统内部、以 MCP 兼容方式建模、由本地 Python 实现承载的能力单元`

## 3.2 为什么归类为本地 MCP

原因不是“为了跟风 MCP”，而是因为这三项能力天然具备 MCP 能力单元应有的属性：

- 有清晰能力边界
- 有输入输出契约
- 有资源句柄或文件对象边界
- 可以通过 tool 暴露给模型
- 可以通过 worker 暴露给编排系统
- 可以被 catalog 统一描述

因此更准确的定位应为：

`local mcp capability units`

而不是：

- 临时工具集合
- 历史业务目录
- 单独的半独立系统

## 四、正式对象模型

本次新增正式对象：

### 4.1 LocalMCPUnitRecord

表示一个本地 MCP 能力单元的正式登记对象。

建议最小字段：

- `unit_id`
- `name`
- `category`
- `server_name`
- `protocol_version`
- `implementation_root`
- `summary`
- `tool_refs`
- `worker_refs`
- `skill_refs`
- `resource_kinds`
- `normalization_contract`
- `status`

### 4.2 tool / worker / skill / unit 的关系

四者关系正式锁定如下：

- `mcp_unit` 是能力本体的正式治理对象
- `tool` 是面向模型或主运行时的调用暴露面
- `worker` 是面向编排系统的后台执行暴露面
- `skill` 是面向提示与使用策略的能力方法包

也就是说：

- tool 不拥有能力本体
- worker 不拥有能力本体
- skill 更不拥有能力本体

真正的能力本体归 `mcp_unit`

## 五、目标目录结构

本次迁移完成后，目标目录应为：

```text
backend/
  capability_system/
    local_mcp_registry.py
    units/
      mcp/
        layer.py
        local/
          retrieval/
            __init__.py
            unit.py
            runtime/
              ...
          pdf/
            __init__.py
            unit.py
            agent/
              ...
            analysis/
              ...
          structured_data/
            __init__.py
            unit.py
            ...
```

对应原则：

- `retrieval` 作为检索能力单元域
- `pdf` 作为文档阅读与 PDF 处理能力单元域
- `structured_data` 作为结构化数据分析能力单元域

## 六、旧目录处理原则

以下目录不再作为正式能力根目录存在：

- `backend/RAG`
- `backend/pdf_agent`
- `backend/pdf_analysis`
- `backend/structured_data`

本次迁移原则是：

`如果可以一次性改全导入链，就不保留兼容壳`

因此本次默认策略为：

- 先迁实现
- 再改导入
- 再删除旧根目录

只有在无法一次性切换时，才允许极短暂过渡层。

## 七、catalog 需要新增的正式视图

能力系统 catalog 不能再只暴露：

- tools
- workers
- capability_endpoints

本次必须新增：

- `local_mcp_units`

每个 `local_mcp_unit` 至少要暴露：

- `unit_id`
- `name`
- `category`
- `server_name`
- `protocol_version`
- `implementation_root`
- `tool_refs`
- `worker_refs`
- `skill_refs`
- `resource_kinds`
- `normalization_contract`

这样能力系统才能真正回答：

`平台里有哪些正式本地 MCP 能力单元`

## 八、迁移实施步骤

## 8.1 第一步：补本地 MCP 正式注册层

新增：

- `LocalMCPUnitRecord`
- `build_local_mcp_catalog()`
- 对应 unit 描述文件

## 8.2 第二步：迁移三项能力本体到能力系统目录

把：

- `RAG`
- `pdf_agent`
- `pdf_analysis`
- `structured_data`

迁入：

- `capability_system/units/mcp/local/retrieval`
- `capability_system/units/mcp/local/pdf`
- `capability_system/units/mcp/local/structured_data`

## 8.3 第三步：切换 tool / worker / service 导入

重点切换：

- tool 实现
- worker 实现
- retrieval service
- document conversion
- task understanding
- output boundary

## 8.4 第四步：catalog 接入 local_mcp_units

让工作台、审查和健康系统都能看见正式能力单元，而不是只能看见 endpoint。

## 8.5 第五步：删除旧能力根目录

切换完成且验证通过后，直接删除旧目录，不保留无意义兼容残留。

## 九、完成条件

以下条件全部满足才算完成：

1. 三项能力实现本体已经进入能力系统目录
2. tool / worker / service 已只引用新目录
3. catalog 已暴露 `local_mcp_units`
4. 旧 `RAG / pdf_agent / pdf_analysis / structured_data` 根目录已删除
5. 编译和定向测试通过

## 十、切换后的正式结论

本次收口完成后，这三项能力的系统定位正式变为：

`能力系统内部的本地 MCP 能力单元`

它们不再是历史外挂目录，不再是“只是注册进去的工具实现”，而是能力系统正式拥有、正式描述、正式治理的能力本体。

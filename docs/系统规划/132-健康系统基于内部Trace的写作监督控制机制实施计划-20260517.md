# 132-健康系统基于内部Trace的写作监督控制机制实施计划-20260517

## 0. 结论

写作监督不应该继续由 `.codex/hooks` 和外部 watchdog 主导。

当前项目已经有健康系统、运行时 trace、task graph monitor、live monitor、artifacts、memory receipts、command / receipt / report 链路，能力是够的。真正缺的是：**把写作监督作为健康管家 agent 的正式任务与工具接进去**，而不是再写一套独立控制器。

最终目标：

- 健康系统读取内部 trace。
- 健康系统判断任务是否真实推进。
- 健康系统决定继续、续跑、重启监督链、暂停等待人工。
- 所有动作都通过 command / receipt / report 落盘。
- hooks 只负责唤醒健康检查，不再直接判断或重启。

  - 不支持写作监督命令

### 1.2 写作监督已有能力

已确认文件：

- `scripts/supervise_writing_campaign.py`
  - 是当前写作任务监督执行器
  - 已经具备恢复当前阶段、读取状态、驱动写作任务继续推进的能力
- `scripts/watch_writing_supervision.py`
  - 是低频外部 watchdog
  - 当前会判断后端健康、supervisor 进程、state age，并直接启动 `start_writing_supervisor.ps1`
- `.codex/hooks/writing_project_monitor.py`
  - Stop hook 内读取项目 runtime-status、task-graph-monitor、live-monitor、artifacts、memory-receipts
  - 当前仍会自己判断 blocker、failed、supervision stale
- `.codex/hooks/writing_supervision_watchdog.py`
  - Stop hook 内检查 watchdog pid
  - watchdog 不存在时直接启动 `scripts/start_writing_watchdog.ps1`
- `.codex/hooks/writing_supervision_stop.py`
  - Stop hook 内检查 supervision state
  - 当前会反复触发继续值守提示

结论：
外部脚本有很多判断逻辑，但这些判断没有进入健康系统审计链，容易造成“看起来在管，实际上不可追踪”。

## 2. 核心设计

### 2.1 控制权归属

控制权必须归健康管家 agent 的任务编排。

健康管家 agent 应该通过系统内已有能力完成监督：

- 读取内部 trace。
- 调用监督工具。
- 生成决策报告。
- 触发续跑或重启建议。

外部层只能做两件事：

1. 发现需要检查。
2. 触发健康管家 agent 进入监督任务。

外部层不能再直接决定：

- 任务是否假死。
- 是否 `continue-current-stage`。
- 是否重启 supervisor。
- 是否重启后端。
- 是否宣称任务恢复。

### 2.2 三层结构

#### 观测层

只读取内部 trace，不修改状态。

输入来源：

- project runtime status
- task graph monitor
- live monitor
- runtime events
- artifacts
- memory receipts
- supervision marker/state
- backend health

输出：

- `WritingSupervisionSnapshot`

#### 决策层

由健康管家 agent 基于 snapshot、prompt、任务模式和工具返回得出结论。

输出：

- `WritingSupervisionDecision`

决策类型：

- `healthy`
- `watch_only`
- `resume_needed`
- `restart_supervision_needed`
- `backend_recovery_needed`
- `structural_failure`
- `manual_intervention_required`
- `completed`

#### 执行层

把 decision 转成正式动作，动作仍通过系统现有工具和 command 链路落盘。

动作类型：

- 只生成 report
- 调用现有续跑能力
- 调用监督器启动脚本
- 生成 issue
- 返回人工处理要求

所有动作都必须生成：

- command
- receipt
- report

## 3. 命令设计

在 `HealthCommandService` 增加四个命令：

### 3.1 `inspect_writing_trace`

只读深度诊断。

用途：

- 查看当前 run 是不是真推进。
- 判断是否有产物增长。
- 判断是否有 memory commit。
- 判断流式输出是否只是有连接但无有效文本。

输入 payload：

```json
{
  "project_id": "project:honghuang-times",
  "task_run_id": "",
  "coordination_run_id": "",
  "include_artifacts": true,
  "include_memory_receipts": true
}
```

输出 report 必须包含：

- active task run
- active node
- active task ref
- runtime status
- event count
- seconds since event change
- seconds since progress change
- latest artifact refs
- latest memory receipt refs
- blocker
- diagnosis

### 3.2 `monitor_writing_campaign`

常规监督检查。

用途：

- 低频检查任务是否健康。
- 不直接执行动作。
- 输出建议动作。

输入 payload：

```json
{
  "project_id": "project:honghuang-times",
  "target_words": 1000000,
  "stale_seconds": 300,
  "effective_activity_stale_seconds": 300
}
```

输出 decision：

- `healthy`
- `watch_only`
- `resume_needed`
- `restart_supervision_needed`
- `structural_failure`
- `completed`

### 3.3 `resume_writing_campaign`

恢复当前阶段。

用途：

- 当 trace 表明任务可续跑时，优先恢复当前 stage。
- 不直接重开整条任务。

行为：

1. 读取当前 project runtime status。
2. 定位 active task run / coordination run。
3. 读取 current stage execution request。
4. 调用现有续跑入口。
5. 写入 receipt，记录实际调用的 task run / stage / response。

禁止：

- 没有 current stage request 时盲目续跑。
- 没有 receipt 时宣称恢复成功。

### 3.4 `restart_writing_supervision`

重启监督链，不重写业务产物。

用途：

- supervisor 进程丢失。
- supervision marker/state 过期。
- 后端恢复后需要重新挂载监督。

行为：

1. 根据 project runtime status 定位 root task run。
2. 若已有 active coordination run，attach existing。
3. 调用 `scripts/start_writing_supervisor.ps1`。
4. 写入 receipt，记录 session id、pid、state path、attach target。

禁止：

- 把 supervisor 重启等同于任务恢复。
- 在项目已完成时启动新监督。
- 无 active run 且未显式允许时新建任务。

## 5. 后端实现计划

### 阶段 1：trace 聚合服务

新增文件：

- `backend/health_system/writing_supervision.py`

职责：

- `build_writing_supervision_snapshot(...)`
- `decide_writing_supervision_action(...)`
- `build_writing_supervision_report_payload(...)`

这里的实现只负责把内部 trace 整理成健康管家 agent 可读的结构化输入，不负责替 agent 下最终判断。

输入：

- `base_dir`
- `task_run_loop`
- `project_id`
- optional `task_run_id`

输出：

- snapshot dict
- decision dict

实现原则：

- 不直接依赖 `.codex/hooks`
- 只读取系统内部状态和项目输出状态
- 所有错误放入 snapshot.errors，不吞掉

### 阶段 2：Registry 接入

修改：

- `backend/health_system/registry.py`

新增方法：

- `inspect_writing_trace(...)`
- `monitor_writing_campaign(...)`
- `resume_writing_campaign(...)`
- `restart_writing_supervision(...)`

职责划分：

- registry 负责聚合 store、trace、工具执行器和任务装配。
- command service 只负责把健康管家 agent 的任务请求路由到对应工具。
- 具体判断由健康管家 agent 根据任务 prompt 和工具返回完成。

### 阶段 3：CommandService 接入

修改：

- `backend/health_system/command_service.py`

新增分支：

- `inspect_writing_trace`
- `monitor_writing_campaign`
- `resume_writing_campaign`
- `restart_writing_supervision`

每个分支必须：

1. 生成 report。
2. 生成 receipt。
3. 调用 `_complete_command`。

失败时：

- command status = `failed`
- receipt status = `failed`
- diagnostics 必须保留异常类型和上下文

### 阶段 4：API 保持统一入口

修改：

- `backend/api/health_system.py`

优先不新增碎片化 endpoint。

继续使用：

- `POST /api/health-system/commands`

原因：

- 写作监督本质上是健康管家 agent 的管理任务。
- 统一入口能保留 command / receipt / report 链路，同时不把系统拆成多套控制接口。

可以后续增加只读便捷 endpoint：

- `GET /api/health-system/writing-supervision/current`

但第一阶段不强依赖。

## 6. hooks 降级计划

### 6.1 `.codex/hooks/writing_project_monitor.py`

目标：

- 不再自己深判定。
- 只唤起健康管家 agent 的监督任务：

```json
{
  "command_type": "monitor_writing_campaign",
  "initiator_type": "codex_hook",
  "source": "codex.stop_hook",
  "payload": {
    "project_id": "project:honghuang-times",
    "target_words": 1000000,
    "stale_seconds": 300
  }
}
```

返回处理：

- 若健康系统不可用，只记录本地日志，不频繁 block。
- 若 decision 是 `manual_intervention_required` 或 `structural_failure`，再 block 交给 Codex。
- 若 decision 是 `healthy` / `watch_only`，不 block。

### 6.2 `.codex/hooks/writing_supervision_watchdog.py`

目标：

- 不再直接启动 `start_writing_watchdog.ps1`。
- 只触发健康管家 agent 的监督任务。
- 只有 agent 在工具返回里明确给出 `restart_supervision_needed` 且允许自动执行时，才提交重启动作。

### 6.3 `.codex/hooks/writing_supervision_stop.py`

目标：

- 去掉“没有明确故障但继续值守”的频繁 block。
- 只在健康管家 agent 通过工具判断需要人工或结构性失败时触发继续。

## 7. 外部 watchdog 退场计划

### 7.1 `scripts/watch_writing_supervision.py`

短期：

- 保留为兼容工具。
- 默认只调用健康系统 command。
- `--legacy-direct-restart` 才允许旧逻辑直接重启。

中期：

- 删除直接判断逻辑：
  - backend health failure count
  - supervisor pid restart
  - state age restart

长期：

- 如果健康系统稳定，则清理该脚本或改名为 `scripts/request_writing_supervision_check.py`。

### 7.2 `scripts/supervise_writing_campaign.py`

保留。

它不是控制面，而是执行器。

健康系统可以调用它已有能力，但它不能成为最终监督权威。

## 8. 判定规则细化

### 8.1 健康

满足任一真实推进条件：

- event count 在阈值内变化。
- completed words 增长。
- committed chapter count 增长。
- latest artifact mtime 更新且属于当前 active node。
- memory receipt 新增且匹配当前写入阶段。
- stream chunk count 增长且 preview 非空。

### 8.2 假活

典型条件：

- `runtime_status=running`
- `seconds_since_event_change >= stale_seconds`
- `seconds_since_progress_change >= stale_seconds`
- `seconds_since_effective_activity >= stale_seconds`
- active node / active task ref 长期不变
- 无新 artifact
- 无新 memory receipt

决策：

- 如果 current stage request 存在：`resume_needed`
- 如果 supervisor state stale 或进程丢失：`restart_supervision_needed`
- 如果配置缺失或 blocker 指向结构问题：`structural_failure`

### 8.3 后端断链

条件：

- health API 不可达
- 但 runtime event 文件最近仍在更新

决策：

- 不立刻重启任务。
- 标记 `backend_recovery_needed`。

### 8.4 任务完成

必须同时满足：

- completed words >= target words
- project runtime status 为 completed / succeeded
- 无 active blocker
- final receipt / final report 存在

不能只看字数达标。

## 10. 实施顺序

### 第一步：补健康系统监督模型

- 增加 `WritingSupervisionSnapshot`
- 增加 `WritingSupervisionDecision`
- 增加 dict 序列化

### 第二步：新增 trace 聚合与判定服务

- 新建 `backend/health_system/writing_supervision.py`
- 实现 snapshot builder
- 实现 decision engine

### 第三步：接入 command service

- 增加四个 command 分支
- 每个 command 都必须落 report / receipt

### 第四步：接入 registry

- 增加写作监督方法
- 对接 task_run_loop
- 对接 supervisor 启动脚本

### 第五步：降级 hooks

- hooks 改为调用健康系统 command

- `scripts/watch_writing_supervision.py` 改为薄壳或标记 legacy
- 删除不再使用的直接判断重复代码
- 清理旧 hook 输出中误导性的“继续值守”逻辑

### 第七步：验证与报告

- 跑后端相关测试
- 运行一次实际 command
- 写实施报告，记录修改文件、控制流、残留处理、未解决风险

## 11. 收尾标准

计划完成必须满足：

- 健康系统能独立生成写作监督 snapshot。
- 健康系统能基于内部 trace 给出 decision。
- `monitor_writing_campaign` 可通过 `POST /api/health-system/commands` 执行。
- `resume_writing_campaign` 不会在证据不足时假成功。
- `restart_writing_supervision` 只重启监督链，不伪造业务产物。
- hooks 不再直接做主监督判断。
- watchdog 不再是唯一恢复路径。
- 每次监督判断和动作都有 command / receipt / report 可审计。

---

## 12. 明确不做

本计划不做：

- 不改写作业务流程。
- 不改记忆库模型。
- 不让 hook 继续扮演监督系统。
- 不用外部脚本伪装健康系统能力。
- 不在没有 trace / receipt / report 的情况下宣称恢复成功。

# 188-Split/Merge 工程可实现性评估与落地路径

日期：2026-05-20

## 1. 结论

Split/Merge 可以实现，但不能一步做到“运行中任意动态裂解”。正确落地路径应该分三层：

```text
L1 静态批次编译：保存态配置 -> 编译期生成批次计划 -> 发布包可见。
L2 可控子流运行：按批次计划启动多个子任务/子图 -> scheduler 跟踪 -> merge 汇总。
L3 动态运行时裂解：节点执行过程中提出拆分请求 -> 系统校验契约和上限 -> 动态创建子流。
```

近期只能先做 L1，最多推进到 L2 的最小闭环。L3 是高风险能力，必须等 packet、scope、ledger、idempotency 都稳定后再做。

所以这不是空中楼阁，但必须按工程边界推进。

## 2. 为什么可以实现

当前代码已经具备几个必要基础：

- `TaskGraphDefinition` 可以保存图、节点、边和 `contract_bindings`。
- `coordination_graph_compiler.py` 已经能把保存态编译成 `TaskGraphRuntimeSpec`。
- `task_graph_scheduler.py` 已经能根据节点、边、phase、sequence、wait policy 生成调度状态。
- `ExecutionPackage` 已经能聚合 runtime spec、manifest、scheduler、assembly 和 GraphUnit 子图计划。
- GraphUnit 已经能启动 child graph run，并把 output packet 回交父图。

这说明我们不需要从零发明 runtime，只需要把 split/merge 作为一种新的编译计划和调度语义逐步接入。

## 3. 不能直接做什么

不能直接做：

- 让 Agent 在 prompt 里自由决定拆成多少任务。
- 让节点运行时无上限创建子任务。
- 没有 packet 标准就做多子任务汇总。
- 没有 scope 隔离就让子任务共享父任务全部上下文。
- 没有 idempotency 就支持重试和断点续跑。
- 没有 execution package 诊断就直接 active runtime。

否则会出现：

- 重复创建子任务。
- 子任务丢失。
- 父任务不知道等谁。
- 上下文污染。
- 汇总顺序不稳定。
- 失败传播不可控。

## 4. L1：静态批次编译

### 4.1 要解决的问题

用户提出一个范围：

```text
处理 50 个章节
检查 100 个文件
调研 20 个来源
生成 30 个素材
```

系统把它编译成明确批次：

```text
batch_001: item 001-010
batch_002: item 011-020
...
```

这个阶段不创建子任务，只生成计划。

### 4.2 数据契约

放入节点 `contract_bindings.unit_batch`：

```json
{
  "unit_kind": "chapter",
  "requested_count": 50,
  "batch_size": 10,
  "range_start": 1,
  "range_label_template": "chapter_{start}_{end}",
  "ordering": "ascending",
  "source": "user_request"
}
```

放入节点 `contract_bindings.runtime.split_policy`：

```json
{
  "mode": "static_batch",
  "max_batches": 20,
  "child_execution_mode": "sequential",
  "idempotency_scope": "graph_run"
}
```

放入节点 `contract_bindings.runtime.merge_policy`：

```json
{
  "mode": "wait_all",
  "result_order": "batch_sequence",
  "allow_partial": false
}
```

### 4.3 后端实现

新增：

```text
backend/tasks/task_split_merge_models.py
backend/tasks/task_split_plan_builder.py
backend/tests/task_split_plan_builder_regression.py
```

核心函数：

```python
def build_static_split_plan(node_id: str, contract_bindings: dict[str, Any]) -> StaticSplitPlan:
    ...
```

输出：

```python
StaticSplitPlan(
    plan_id="split:node.writer:chapter:1-50",
    node_id="node.writer",
    unit_kind="chapter",
    batches=[
        BatchSpec(batch_id="chapter_001_010", range_start=1, range_end=10),
        ...
    ],
    merge_policy={...},
)
```

### 4.4 编译接入

在 `coordination_graph_compiler.py` 中读取节点 `contract_bindings.unit_batch` 和 `runtime.split_policy`，生成：

```text
runtime_spec.diagnostics.split_plans
```

第一阶段不要改变 runtime nodes。

### 4.5 ExecutionPackage 接入

在 `backend/api/tasks.py` execution package 中增加：

```json
{
  "split_plans": [...],
  "split_merge_issues": [...],
  "summary": {
    "split_plan_count": 1,
    "split_batch_count": 5
  }
}
```

前端发布包面板显示：

```text
节点 writer 拆分为 5 个批次
chapter 001-010
chapter 011-020
...
merge: wait_all / batch_sequence
```

### 4.6 L1 完成标准

- 用户能在节点上配置 requested_count / batch_size。
- 后端能生成 deterministic split plan。
- execution package 能显示批次计划。
- 不支持配置会给 issue。
- 不启动任何额外子任务。

这是最小可落地版本。

## 5. L2：可控子流运行

### 5.1 要解决的问题

把 L1 生成的批次计划变成真实执行。

两种实现路径：

```text
路径 A：生成多个同类型 node execution request。
路径 B：每个 batch 启动一个 child task graph run。
```

路径 A 更轻，适合普通节点批量执行。

路径 B 更重，但可以复用 GraphUnit 子图运行机制。

### 5.2 建议先做路径 B 的受控版本

原因：

- GraphUnit child run 已经存在。
- 子图 run 已经有隔离和 output packet。
- 回交机制已有基础。

做法：

```text
Split node
  -> 根据 split plan 生成 N 个 child graph run
  -> 每个 child run 初始输入带 batch spec
  -> 每个 child run 完成后生成 batch output packet
  -> Merge node 等待所有 batch packet
  -> 按 merge policy 生成 merged result
```

### 5.3 必需新增对象

```python
SplitChildRunHandle
BatchExecutionPacket
MergeWaitState
```

### 5.4 调度器需求

Scheduler 需要知道：

- split plan 有多少 child。
- 哪些 child 已启动。
- 哪些 child completed / failed / waiting。
- merge 是否 ready。

第一版可以不深改 scheduler，把状态存在 task_run diagnostics + execution package trace 中。Merge 阶段由 API continue 时检查。

### 5.5 L2 完成标准

- 静态 batch 可以启动多个 child graph run。
- 每个 child run 有 batch_id 和 idempotency key。
- 重试不会重复启动同一 batch。
- 所有 batch 完成后 merge。
- 任一 batch 失败时按 merge_policy 处理。

## 6. L3：动态运行时裂解

### 6.1 要解决的问题

节点执行中发现任务需要拆分：

```text
当前资料太多，请拆成 8 个研究子任务。
当前章节范围太大，请拆成 5 个批次。
当前代码目录需要按模块拆分检查。
```

这必须通过工具或 structured output 提交请求，不能靠自由文本。

### 6.2 动态 split 请求

```json
{
  "authority": "task_system.dynamic_split_request",
  "node_id": "node.research",
  "reason": "source_count_exceeds_limit",
  "requested_batches": [
    {"batch_id": "source_001_005", "range_start": 1, "range_end": 5},
    {"batch_id": "source_006_010", "range_start": 6, "range_end": 10}
  ],
  "merge_policy": {"mode": "wait_all"},
  "idempotency_key": "..."
}
```

### 6.3 必须校验

- 当前节点 contract 是否允许 dynamic split。
- 请求 batch 数是否低于上限。
- batch_id 是否稳定。
- 每个 batch 是否有明确输入范围。
- child execution template 是否存在。
- merge policy 是否受支持。

### 6.4 为什么 L3 不能先做

因为它要求：

- packet 标准。
- scope 标准。
- ledger。
- idempotency。
- cancellation。
- retry。
- partial merge。

没有这些，动态 split 会变成不可控的任务爆炸。

## 9. 测试策略

### 9.1 后端单元

测试：

- requested_count=50 batch_size=10 -> 5 batches。
- requested_count=53 batch_size=10 -> 最后一批 51-53。
- max_batches 超限 -> error。
- batch_size 缺失 -> warning/error。
- unsupported split mode -> issue。
- merge_policy 缺失 -> 默认 wait_all。

### 9.2 后端 API

测试：

- execution package 返回 split_plans。
- summary 计数正确。
- object trace 能关联节点和 split plan。

### 9.3 前端

测试：

- 节点编辑台写入 contract_bindings。
- 发布包显示 split plan。
- unsupported issue 展示。

## 10. 第一轮建议

第一轮只实现 L1：

```text
静态 split plan 编译 + execution package 展示 + 前端节点配置
```

不做真实子任务启动。

这一步完成后，我们就能验证：

- 通用契约能不能表达“50 章拆成 5 批”。
- 非写作任务能不能表达“100 文件拆成 10 批”。
- 用户在前端是否能自然配置。
- execution package 是否能准确解释。

如果 L1 成功，再进入 L2。L2 可以先复用 GraphUnit child run 做受控批次子图运行。

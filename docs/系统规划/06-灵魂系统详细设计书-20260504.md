## 二、灵魂系统总定义

灵魂系统是平台的 prompts 身份层。

它不负责任务调度，不负责任务 workflow，不负责能力资源供给，不负责记忆治理，不负责健康诊断。

它负责的是：

- soul 设定
- projection 设定
- 人格设定
- 表达风格
- 上下文组织风格
- 注意力组织方式
- 执行姿态提示

它回答的问题是：

`这个 Agent 以什么人格、什么表达姿态、什么上下文组织方式进入任务执行`

## 三、为什么灵魂系统不能继续混入 workflow

这一点必须彻底锁死。

workflow 是任务方法。

它回答：

- 任务怎么做
- 任务先后步骤是什么
- 任务停在哪

而灵魂系统回答的是：

- 说话像谁
- 如何组织表达
- 如何聚焦注意力
- 如何呈现姿态

如果把 workflow 混入灵魂系统，会导致三个问题：

1. 灵魂变成任务方法碎片
2. 任务系统失去方法治理权
3. 运行体系统难以区分“方法提示”和“身份提示”

所以：

- workflow 归任务系统
- soul / projection 归灵魂系统

## 四、灵魂系统的硬边界

### 4.1 灵魂系统不定义任务

任务是什么、由谁做、怎么分配，归任务系统。

### 4.2 灵魂系统不定义单 Agent 身体

身体结构、资源边界、记忆作用域、运行 lane，归运行体系统。

### 4.3 灵魂系统不提供能力资源

tools、skills、workers、capabilities，归操作系统。

### 4.4 灵魂系统不筛记忆

记忆内容归记忆系统。

灵魂系统最多影响记忆在 prompts 中的呈现方式，不决定记忆事实本体。

## 五、灵魂系统中的两层对象

灵魂系统建议明确拆成两层：

### 5.1 Soul

Soul 是长期人格底座。

它定义：

- 长期人格
- 基本角色气质
- 长期表达倾向
- 长期协作倾向

它回答：

`这个 Agent 本质上是谁`

### 5.2 Projection

Projection 是任务态或场景态的执行姿态。

它定义：

- 当前角色姿态
- 关注点
- 语言密度
- 任务执行时的注意力组织方式
- 面向特定任务时的表达补充

它回答：

`这个 Agent 这次怎么出现`

也就是说：

- soul 是长期人格底座
- projection 是任务态投影姿态

## 六、灵魂系统的对象模型

结合现有代码中的：

- `backend/soul/registry.py`
- `backend/soul/contracts.py`
- `backend/soul/projection_store.py`
- `backend/soul/prompt_assembly.py`

我建议灵魂系统统一围绕七类对象组织。

### 6.1 SoulProfile

表示正式登记的灵魂画像。

现有代码里已经有很强的雏形。

它回答：

`这个灵魂是谁`

建议字段：

- `soul_id`
- `name`
- `display_name`
- `description`
- `background`
- `personality_traits`
- `expression_style`
- `preferred_role_types`
- `preferred_task_modes`
- `collaboration_tendencies`
- `risk_biases`
- `enabled`

### 6.2 ProjectionProfile

表示正式登记的投影画像。

它回答：

`这个投影以什么姿态出现`

建议字段：

- `projection_id`
- `soul_id`
- `title`
- `role_type`
- `posture_tags`
- `expression_density`
- `attention_focus`
- `projection_prompt`
- `enabled`

### 6.3 ProjectionSelectionPolicy

表示某类任务如何选择 projection。

它回答：

`这个任务侧应该怎么选投影`

建议字段：

- `policy_id`
- `task_id`
- `allowed_projection_ids`
- `default_projection_id`
- `selection_mode`
- `projection_required`

说明：

这个对象可以由任务系统引用，但其素材归灵魂系统维护。

### 6.4 PromptIdentityAsset

表示可以被运行体系统装配的身份层素材包。

它回答：

`这次可供装配的灵魂素材是什么`

建议字段：

- `asset_id`
- `soul_id`
- `projection_id`
- `core_prompt`
- `projection_prompt`
- `identity_notes`
- `display_constraints`

### 6.5 PromptAssemblyManifest

表示身份层素材进入运行壳时的装配清单。

它回答：

`灵魂提示词这次以什么片段进入 prompt assembly`

建议字段：

- `manifest_id`
- `asset_id`
- `sections`
- `injection_order`
- `runtime_visibility`
- `diagnostics`

### 6.6 SoulFilePayload

现有代码中已有较强雏形：

- [backend/soul/registry.py](/d:/AI应用/langchain-agent/backend/soul/registry.py)

它回答：

`某个灵魂文件本体是什么、是否可见、以什么顺序注入`

### 6.7 ProjectionRequirement

现有代码中已有雏形：

- [backend/tasks/runtime_contracts.py](/d:/AI应用/langchain-agent/backend/tasks/runtime_contracts.py)

它回答：

`任务侧对这次投影提出了什么要求`

建议继续保留并强化它与灵魂系统的对接关系。

## 八、灵魂系统与任务系统的接口

任务系统不定义 soul 素材，但会在任务侧绑定 projection。

建议固定接口：

- 任务系统输入：`task_id + projection selection policy`
- 灵魂系统输出：可选 projection 列表与默认 projection

任务系统关心的是：

- 这个任务允许哪些投影
- 默认选哪个投影
- 是否必须选投影

但它不应直接管理 projection prompt 的原始文本。

## 九、灵魂系统与运行体系统的接口

这是灵魂系统最关键的对接面。

运行体系统需要灵魂系统提供的不是“灵魂说明文档”，而是正式身份层素材。

建议固定接口：

- 输入：`SoulProfile + ProjectionProfile + ProjectionRequirement`
- 输出：`PromptIdentityAsset + PromptAssemblyManifest`

运行体系统再根据：

- `PromptStructureProfile`
- `TaskBodyOrchestration`

把这些身份层素材放进具体的 prompt 装配位置。

## 十、灵魂系统与记忆系统的关系

灵魂系统不控制记忆事实，但会影响记忆在上下文中的表达形态。

例如：

- 同样一份任务记忆，主 Agent 的表述可以更自然统筹
- 健康管理 Agent 的表述可以更冷静审慎
- 操作管理 Agent 的表述可以更直接执行导向

所以：

- 记忆内容归记忆系统
- 记忆在 prompts 中的姿态可受灵魂系统影响

## 十一、灵魂系统与操作系统的关系

灵魂系统与操作系统必须彻底解耦。

操作系统给：

- 能力
- 工具
- skill
- worker

灵魂系统给：

- 人格
- 姿态
- 语言结构

skill 即使有 prompt view，也不等于 soul prompt。

## 十二、灵魂系统的装配链路

建议固定灵魂系统的供给链路为：

```text
soul registry
  -> projection registry
  -> projection selection policy
  -> prompt identity asset
  -> prompt assembly manifest
  -> body system consumption
```

### 12.1 为什么需要 assembly manifest

因为运行体系统不应自己去解析灵魂文件目录。

灵魂系统必须把：

- 哪些内容进入 prompt
- 以什么顺序进入
- 哪些只作内部参考

明确打包出来。

## 十三、前端灵魂系统管理目标

前端灵魂系统页面不应是“人物卡展示页”，而应是身份层管理页。

至少应支持以下管理面：

### 13.1 Soul 管理

- 查看 builtin souls
- 查看 custom souls
- 查看人格特征
- 查看 preferred role/task mode

### 13.2 Projection 管理

- 查看 projection 列表
- 查看所属 soul
- 查看姿态标签
- 查看语言密度
- 查看 attention focus

### 13.3 任务投影策略管理

- 查看某类任务允许哪些投影
- 查看默认投影
- 查看是否强制要求投影

### 13.4 Prompt 装配预览

- 查看这次 soul / projection 会向运行体系统交付哪些 prompt section
- 查看最终身份层素材摘要

## 十四、当前问题与清理方向

当前项目在灵魂系统上已经有很多好苗头，但边界还要更硬。

主要问题包括：

- soul 和 projection 的角色需要继续明确分层
- workflow 残留思维需要继续彻底清掉
- prompt assembly 与运行体装配的接口还需要正式对象化
- 前端容易滑向“人物卡展示”，而不是“身份层管理”

## 十五、实施顺序

建议按以下顺序实施：

1. 统一 soul / projection / identity asset 命名
2. 固化 `SoulProfile / ProjectionProfile / PromptIdentityAsset / PromptAssemblyManifest`
3. 让任务系统只引用 projection 选择策略
4. 让运行体系统只消费身份层素材包
5. 最后再重构前端灵魂系统管理页

## 十六、执行结论

灵魂系统不是平台里的装饰层，而是正式的 prompts 身份层。

它真正解决的是：

- Agent 到底是谁
- Agent 这次以什么姿态出现
- 任务执行时的语言和注意力结构怎么被补充
- 身份层素材如何稳定交付给运行体系统

只有这条线立起来，平台里的 Agent 才不会只是会说话的执行器，而会成为有稳定人格底座、又能按任务切换姿态的真正角色化执行体。

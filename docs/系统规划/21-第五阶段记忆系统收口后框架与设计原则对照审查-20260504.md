## 二、当前源码已经对齐的原则

### 2.1 记忆系统已从“能力集合”提升为正式系统入口

当前记忆系统不再只是：

- `conversation_memory`
- `state_memory`
- `durable_memory`
- `context package`

的松散聚合。

现在已经形成三类正式服务：

- `MemoryRequestService`
- `MemoryBundleService`
- `MemoryWritebackBuilderService`

以及一类治理服务：

- `DurableMemoryGovernanceService`

对应源码：

- [backend/memory_system/request_service.py](/d:/AI应用/langchain-agent/backend/memory_system/request_service.py:1)
- [backend/memory_system/bundle_service.py](/d:/AI应用/langchain-agent/backend/memory_system/bundle_service.py:1)
- [backend/memory_system/writeback_service.py](/d:/AI应用/langchain-agent/backend/memory_system/writeback_service.py:1)
- [backend/memory_system/governance_service.py](/d:/AI应用/langchain-agent/backend/memory_system/governance_service.py:1)

这与总设计书中的要求一致：

`记忆系统是平台的记忆治理层和记忆供给层`

而不是让 API、runtime、任务系统分别去拼记忆细节。

### 2.2 任务提需求、编排给约束、记忆系统给供给包的原则已正式落地

当前正式对象链已经建立：

- `MemoryRequest`
- `MemoryScopePolicy`
- `MemoryBundle`
- `MemoryWritebackProposal`

对应源码：

- [backend/memory_system/supply.py](/d:/AI应用/langchain-agent/backend/memory_system/supply.py:1)

这意味着：

- 任务系统不再直接决定“具体拿哪条记忆”
- 编排系统不再直接拼接记忆正文
- runtime 不再以散装字段消费记忆

而是开始通过正式请求和正式供给包对接。

这符合 `01-记忆系统详细设计书` 中：

- 任务系统提出记忆需求
- 编排系统限制记忆作用域
- 记忆系统负责具体供给

的分工原则。

### 2.3 restore 不等于 decide 已保持住

当前 `MemoryRuntimeView` 仍然保持只读：

- `read_only=True`
- `memory_write_allowed=False`

同时：

- `MemoryContextCandidate.can_override_current_turn=False`
- `StateMemoryRestoreCandidate.can_promote_to_current_fact=False`

这意味着状态恢复候选仍然只是候选，不具备裁决权。

对应源码：

- [backend/memory_system/runtime_view.py](/d:/AI应用/langchain-agent/backend/memory_system/runtime_view.py:1)
- [backend/memory_system/contracts.py](/d:/AI应用/langchain-agent/backend/memory_system/contracts.py:1)

这与 `13-...关系与目录分层补充设计书` 中强调的：

`restore 不等于 decide`

完全一致。

### 2.4 writeback 仍保持 candidate-only，没有偷偷变成自动提交

当前写回方向仍然只产出候选：

- `MemoryWriteCandidate`
- `MemoryWritebackProposal`

并且：

- `gate_decision` 默认是 `pending`
- `MemoryWritebackProposal.adopted=False`

这保证 runtime 不能绕过治理层直接写入 durable memory。

这符合：

- 默认隔离
- 明确授权
- fail-closed

这三类原则。

### 2.5 API 对底层 durable 对象的直接穿透已明显收缩

在这次收口后：

- [backend/api/memory.py](/d:/AI应用/langchain-agent/backend/api/memory.py:1) 已改为通过 `memory_facade` 的正式治理接口处理 note 的创建、合并、归档、删除、读取、扫描
- [backend/api/tokens.py](/d:/AI应用/langchain-agent/backend/api/tokens.py:1) 已改为通过 `memory_facade.inspect_session_history_tokens(...)` 获取历史 token 诊断

这说明 API 已经不再直接把自己当作 durable store 的“半个内部模块”。

这一点符合总设计书和原则文档里反复强调的：

`系统边界要通过正式入口表达，不要让外层直接碰内部零件`

## 三、与先进架构原则对齐的地方

### 3.1 单一真相源在增强

本次收口后的结构方向是：

- durable note 治理由 `DurableMemoryGovernanceService` 统一负责
- runtime 记忆供给由 `MemoryBundleService` 统一负责
- 请求和作用域收口到 `MemoryRequestService`

这借鉴了成熟架构中常见的：

`单一注册点 + 单一供给点 + 单一治理点`

模式。

它比原来的 `facade 里什么都能做` 更接近可维护架构。

### 3.2 fail-closed 比之前更清楚

从当前实现来看：

- 长期记忆读取要经过 request/scope 过滤
- 长期记忆写入仍然不会自动发生
- restore candidate 不能提升为当前轮事实

这本质上就是原则文档里强调的 fail-closed。

也就是：

`如果没有明确允许，就默认不能读、不能写、不能自我升级为事实`

### 3.3 derived artifact 不再继续上升为 truth owner

当前 `MemoryBundle`、`ContextPolicyResult`、`MemoryRuntimeView` 都是供给对象，不是事实本体。

也就是说：

- 包是包
- 候选是候选
- 真相源仍在各自层内

这和“派生结果不能反过来成为可编辑真相源”的原则是对齐的。

## 四、当前仍然存在的差距

### 4.1 `MemoryFacade` 仍然偏厚，但性质已经改变

虽然这次已经拆出了 request/bundle/writeback/governance 四类服务，但：

- [backend/memory_system/facade.py](/d:/AI应用/langchain-agent/backend/memory_system/facade.py:1)

仍然保留了较多转发方法。

不过这里要区分两层问题：

1. 以前的问题是 `facade` 自己承担实现
2. 现在的问题主要是 `facade` 仍承担过多转发入口

这已经不是边界错误，而是后续可继续瘦身的工程问题。

### 4.2 `structured_memory` 仍是重要底层实现来源

当前 `memory_system` 仍依赖：

- `MemoryManager`
- `SessionMemoryManager`
- frontmatter 读写
- durable extraction 底层类型

这本身不一定是错，因为底层实现可以复用。

真正的标准应该是：

`外部系统是否还需要直接碰 structured_memory`

从本轮结果看，外部直接穿透已经显著减少，只剩少量测试仍直接读取内部状态。

所以这里的结论是：

- 依赖底层实现本身可以接受
- 但 public boundary 必须继续坚持只从 `memory_system` 暴露

### 4.3 任务记忆与 Agent 专属记忆尚未正式建模

按 `01-记忆系统详细设计书` 的五层模型，当前真正稳定落地的是：

- L2 会话记忆
- L5 平台长期记忆
- 部分 L1/L3/L4 的候选形态

尚未完全正式化的主要是：

- L3 任务记忆
- L4 Agent 专属记忆

因此当前记忆系统已经完成“边界收口”，但还没有完成“全层建模完备化”。

### 4.4 少量测试仍直接摸内部对象

当前残留扫描结果显示，直接访问：

- `runtime.memory_facade.session_memory.manager(...)`

的用法主要还存在于测试中。

这不影响正式系统边界，但说明测试夹具还带有旧时代内部视角。

后续如果要继续提高分层纯度，可以逐步把这类测试也改成走正式接口。

## 五、对“修改后框架”和“原则框架”的差别结论

### 5.1 现在的框架已经从“混合实现框架”转向“正式边界框架”

以前的差别主要是：

- 原则文档要求正式系统边界
- 源码还是以 `Facade + 底层直连 + API 自拼` 为主

现在这条差别已经大幅缩小。

当前更准确的说法是：

`原则框架已经基本成立，源码框架也开始按原则框架组织，但仍处于“正式边界已立住、内部继续瘦身”的阶段`

### 5.2 当前剩余差别不再是方向错误，而是完备度不足

现在记忆系统的剩余问题已经不是：

- 边界错了
- owner 错了
- truth source 错了

而更像是：

- 某些层级还未完全建模
- facade 还可以继续瘦
- 测试与少数内部工具还带旧痕迹

这说明这轮重构已经把最难的“结构性错位”修正掉了。

## 六、建议的后续动作

如果我们继续推进记忆系统的后续深收口，建议顺序如下：

1. 先维持 `memory_system` 作为唯一正式 public boundary，不再让其他子系统新增对 `structured_memory` 的直接依赖。
2. 后续如果继续细化，优先正式建模 L3 任务记忆 与 L4 Agent 专属记忆，而不是再给 `facade` 补功能口子。
3. 逐步把测试和调试入口也迁到正式接口，减少对 `session_memory.manager(...)` 这类内部对象的直接读取。
4. 只有在下一阶段确实需要时，再考虑把 `MemoryFacade` 进一步压缩为更薄的系统装配壳。

## 七、审查结论

本轮收口后，记忆系统已经基本符合设计原则文档中的核心要求：

- 系统边界更正式
- 请求与供给对象更明确
- restore 与 decide 分离
- writeback 继续 fail-closed
- API 穿透内部实现的情况明显收缩

当前剩余差距主要是完备度问题，不再是结构方向问题。

这意味着第五阶段可以视为：

`记忆系统正式收口完成，后续进入层级完备化与细节纯化阶段`

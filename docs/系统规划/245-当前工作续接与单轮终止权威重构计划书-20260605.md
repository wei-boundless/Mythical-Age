# 当前工作续接与单轮终止权威重构计划书

日期：2026-06-05  
状态：待审阅  
范围：ChatRun / TurnRun / TaskRun / ActiveTurn、单轮工具上限收口、当前任务续接、前端 stream 消费  
不在本方案范围：模型供应商切换、UI 大改、任务图 GraphRuntime 重写、具体业务文件修复

## 1. 背景

用户反馈的问题是：在会话中说“继续”时会断一下，常常需要再次强调第二遍，Agent 才像是继续工作。

这不是“继续”这个词的命令识别问题，也不应该用关键词硬编码修复。最新实例显示，用户输入被系统正常接收并启动了 ChatRun；真正的问题是当前工作续接、单轮工具预算、流式终止和运行记录之间的权威边界分裂。

成熟 coding agent 的最低标准应是：

```text
RequestFacts
-> BoundaryPolicy / CurrentWorkControl
-> RuntimeStartPacket
-> ExecutionLoop
-> OutputBoundary
-> DurableState
-> PublicStream
```

其中：

- 当前是否有可续接工作，由控制面裁决，不由普通工具循环临场猜测。
- 一次执行必须有唯一运行 handle，不能把多个尝试写入同一个 TurnRun。
- 内部 TurnRun 终止必须先落盘，再发 public `done`。
- 工具预算耗尽不能伪装成稳定完成，更不能把内部协议碎片持久化为助手答案。

## 2. 最新实例证据

最新会话：

- `storage/sessions/session-fad8ee446bf14d98.json`

相关 turn：

- `turn:session-fad8ee446bf14d98:142`
- `turn:session-fad8ee446bf14d98:144`
- `turn:session-fad8ee446bf14d98:146`
- `turn:session-fad8ee446bf14d98:148`

### 2.1 不是 active-turn 输入队列吞掉

对应 ChatRun 记录显示：

- `active_turn_input_policy=auto`
- `expected_active_turn_id=""`
- 没有 `active_task_steer_accepted`
- 没有 `conversation_queued_while_running`

因此最新实例里，用户输入没有进入 active-turn steer 队列，而是启动了新的普通 `single_agent_turn`。

### 2.2 多轮都被工具上限综合答案截断

turn 142、144、146、148 的公开助手消息均来自：

```text
harness.single_agent_turn.tool_limit_synthesis
```

对应 public stream `done` 的关键字段：

```text
terminal_reason=single_turn_tool_iteration_limit
completion_state=tool_limit_synthesized
```

这解释了为什么用户看到的是“下一轮继续”“需要我继续吗”等收口话术，而不是同一个任务继续推进。

### 2.3 ChatRun 与 TurnRun 终止状态冲突

对应 ChatRun：

- `status=completed`
- `terminal_event=done`

对应 TurnRun：

- `status=aborted`
- `terminal_reason=stream_cancelled`

代码原因：

- `backend/api/chat.py::_run_chat_to_event_log()` 在 public `done/error/stopped` 后立即 `break`。
- `backend/harness/loop/single_agent_turn.py` 在工具上限路径中先 yield `done`，之后才记录并 yield `agent_turn_terminal`。
- API 提前关闭 runtime generator 后，`single_agent_turn` 捕获 `GeneratorExit` / `CancelledError`，把 TurnRun 记录为 `aborted/stream_cancelled`。

这造成了外层传输认为完成、内层语义运行认为取消的分裂。

### 2.4 turn 148 出现内部协议碎片持久化

turn 148 的 api transcript 最后一条 assistant 内容是 DSML 工具调用：

```text
<DSML tool_calls> ... search_text(function updatePhysics) ...
```

但工具上限综合阶段没有再次走动作解析和硬门禁，直接把模型输出当 final text。输出边界清洗后得到：

```text
>
```

session 中持久化的助手内容就是 `>`，并带有：

```text
answer_leak_flags=["internal_protocol_final_text"]
answer_canonical_state=stable_answer
```

这说明 OutputBoundary 已经检测到泄漏，但仍允许它成为稳定答案。

### 2.5 TurnRun ID 被复用

`turnrun:turn:session-fad8ee446bf14d98:142` 的事件文件中有 4 次 `agent_turn_received`，其中前 3 次发生在公开 turn 142 用户消息之前。对应还存在多个 ChatRun 都指向同一个 `runtime_turn_run_id=turnrun:turn:session-fad8ee446bf14d98:142`。

代码原因：

- `runtime_facade.py` 使用 `turn_index = len(history_record.get("messages", [])) + 1` 生成 `turn_id`。
- `single_agent_turn.py` 使用 `turn_run_id = f"turnrun:{turn_id}"`。

只要会话消息没有推进、被截断、被重发替换或恢复过程没有提交公开消息，就可能复用同一个 TurnRun 事件日志。

## 3. 真实问题定义

当前系统缺少一个权威的 CurrentWorkBoundary。

用户输入进入后，系统虽然能构造 `active_work_context`，也能恢复 `waiting_executor` 的 TaskRun，但普通 single-agent turn 仍同时开放：

- `active_work_control`
- `request_task_run`
- 普通 `tool_call`
- 普通 `respond`

模型可以选择普通工具调用并绕开当前工作控制面。这样一来，用户说“继续”不是进入可续接任务，而是启动一个新的工具探索回合。工具探索回合达到上限后，又以 `done` 结束，前端自然关闭 stream，下一次用户只能再发一遍。

这不是词法意图识别失败，而是控制权放错了层。

## 4. 目标架构

目标链路：

```text
ChatRequest
-> RequestFacts
-> CurrentWorkBoundary
   - no_current_work
   - current_work_control_required
   - current_work_status_answer
   - new_independent_turn_allowed
-> RuntimeStartPacket
-> ExecutionLoop
-> TurnRunTerminal
-> FinalAnswerEvent(done/error/stopped)
-> ChatRunTerminal
-> Frontend Stream Close
```

核心原则：

1. `TaskRun` 是长期工作的 durable truth。
2. `ActiveTurn` 只表示当前 live turn 或 live task 的 steer handle，不能替代 TaskRun。
3. `TurnRun` 是一次模型执行的唯一语义运行记录。
4. `ChatRun` 是传输层 stream run，不能决定语义运行是否 completed。
5. OutputBoundary 只能发布用户可见答案，不能把已检测到的内部协议泄漏标为稳定答案。
6. 前端只消费后端投影，不重复推断当前工作是否可续接。

## 5. 权威边界表

| 模块 | 当前角色 | 问题 | 目标角色 |
|---|---|---|---|
| `backend/harness/entrypoint/runtime_facade.py` | 提交用户消息、装配 runtime、决定 branch | 当前工作存在时仍默认进入普通 single-agent turn | 在进入普通工具循环前执行 CurrentWorkBoundary |
| `backend/harness/loop/single_agent_turn.py` | 执行单轮模型和工具循环 | 工具上限后先发 public `done`，再记录 TurnRun terminal | 先记录 TurnRun terminal，再发 public terminal |
| `backend/api/chat.py` | ChatRun stream 记录与 SSE 投影 | public `done` 后立即关闭 runtime generator | 只消费已经语义收尾的 terminal stream |
| `backend/harness/loop/task_lifecycle.py` | TaskRun 创建、复用、续跑 | 续跑能力只在模型请求 `request_task_run` 后触发 | 当前任务续接由 CurrentWorkBoundary 直接进入控制链路 |
| `backend/runtime/output_boundary/boundary.py` | 清洗和选择 final text | 检测到内部协议后仍可能保留 `>` 作为 stable answer | 内部协议泄漏必须 fail closed 或转为 blocked |
| `backend/runtime/shared/event_log.py` / state index | 保存 TurnRun 事件 | TurnRun ID 可复用，事件日志混合多次执行 | 每次执行唯一 TurnRun ID |
| `frontend/src/lib/api.ts` | 消费 SSE | 收到 `done` 后取消 reader | 保持该行为，但依赖后端保证 `done` 是最后事件 |
| `frontend/src/lib/store/runtime.ts` | 发送消息、active-turn steer | 只能根据 activeTurnSnapshot 判断 steer，无法表达 current task checkpoint | 消费后端 current-work projection，不自行推断 |

## 6. 实施计划

### 阶段 1：建立 CurrentWorkBoundary

目标：

- 把“当前是否有可续接工作”从普通工具循环前移。
- 不硬编码“继续”，而是基于 `RequestFacts + ActiveWorkContext + CurrentTaskState` 做结构化裁决。

修改文件：

- `backend/harness/entrypoint/runtime_facade.py`
- 新增 `backend/harness/entrypoint/current_work_boundary.py`
- `backend/harness/runtime/request_facts.py`
- `backend/harness/loop/active_work.py`
- `backend/tests/harness_runtime_facade_regression.py`

设计：

1. 在 `_run_single_agent_turn()` 前增加 `decide_current_work_boundary()`。
2. 当 `current_session_task_run()` 返回非 terminal 且 `same_run_allowed/resumable/running` 时，生成 `CurrentWorkBoundaryInput`。
3. 先运行 control-only 裁决，不开放普通 `tool_call`。
4. 裁决动作只能是：
   - `continue_active_work`
   - `append_instruction_to_active_work`
   - `answer_about_active_work`
   - `answer_then_continue_active_work`
   - `pause_active_work`
   - `stop_active_work`
   - `new_independent_turn_allowed`
   - `ask_user`
   - `block`
5. 只有当裁决为 `new_independent_turn_allowed` 时，才进入普通 single-agent turn 工具循环。

完成标准：

- 有 current TaskRun 且用户输入指向当前工作时，不再先开放普通文件搜索工具。
- 用户输入不是当前工作时，可以明确进入新普通 turn，但要在记录中留下 boundary decision。
- 不出现基于具体词语的 if/else 分支。

### 阶段 2：TaskRun 续接直接进入控制链路

目标：

- 可续跑任务不再依赖模型先发 `request_task_run` 才恢复。
- `waiting_executor` / `blocked` / runtime restart recovery 能被当前工作控制面直接续跑。

修改文件：

- `backend/harness/entrypoint/runtime_facade.py`
- `backend/harness/loop/task_lifecycle.py`
- `backend/harness/loop/task_run_recovery_state.py`
- `backend/harness/runtime/run_monitor/service.py`
- `frontend/src/lib/store/runtime.ts`

设计：

1. 将 `_resume_current_task_for_task_request()` 拆成通用 `resume_current_task_run_from_boundary()`。
2. `continue_active_work` 直接调用该恢复链路。
3. 恢复成功后返回 `task_control` answer，并让前端 monitor 绑定同一个 TaskRun。
4. ActiveTurn 如果不存在或已经 terminal，不阻止 current TaskRun resume；恢复后重新绑定新的 live ActiveTurn。

完成标准：

- current task `waiting_executor` 时，控制裁决能直接调度 executor。
- 不要求用户重复发送第二轮输入。
- 前端能看到同一个 task_run_id 继续更新。

### 阶段 3：修复 TurnRun 与 ChatRun 终止顺序

目标：

- TurnRun terminal 是语义完成权威。
- ChatRun public `done` 只能在 TurnRun terminal 已写入后出现。

修改文件：

- `backend/harness/loop/single_agent_turn.py`
- `backend/harness/loop/presentation.py`
- `backend/api/chat.py`
- `backend/tests/harness_runtime_facade_regression.py`
- `frontend/src/lib/api/client.test.ts`

设计：

1. 所有 single-agent return path 调整为：

```text
commit assistant message
record TurnRun terminal
yield agent_turn_terminal
yield final_answer_event(done/error/stopped)
return
```

2. `backend/api/chat.py` 可以继续在 public terminal 后 break，但此时 generator 已经完成语义收尾。
3. 测试中禁止把 public `done` 之前缺失 `agent_turn_terminal` 视为合法。

完成标准：

- ChatRun `completed/done` 与 TurnRun `completed/...` 不再冲突。
- 正常完成不再产生 `stream_cancelled`。
- 手动 abort 仍然记录 `aborted/stream_cancelled`。

### 阶段 4：工具上限收口 fail closed

目标：

- 工具上限综合不能再泄漏 DSML / tool_call 协议。
- 工具上限不能被标成稳定成功完成。

修改文件：

- `backend/harness/loop/single_agent_turn.py`
- `backend/runtime/output_boundary/boundary.py`
- `backend/tests/harness_runtime_facade_regression.py`
- `backend/tests/runtime_progress_presenter_regression.py`

设计：

1. 工具上限 synthesis 的模型输出必须再次经过动作解析。
2. synthesis 阶段允许的动作只包括：
   - `respond`
   - `ask_user`
   - `block`
3. 如果模型输出工具调用、DSML、JSON action tool_call 或空内容：
   - 不持久化原始内容。
   - 记录 protocol error。
   - 返回 blocked / partial boundary 文案。
4. `sanitize_visible_assistant_content()` 清洗结果如果只剩 `>`、空白、协议残片，必须视为无有效 visible answer。
5. `answer_leak_flags` 非空且内容来自内部协议时，不能 `persist_canonical`。

完成标准：

- turn 148 这种 DSML 工具调用不会再变成 `>`。
- 工具上限时，要么给出基于已有观察的真实结论，要么明确 blocked，不伪装成功。

### 阶段 5：TurnRun ID 唯一化

目标：

- 一个 TurnRun 表示一次语义执行，不再由 conversation turn index 唯一决定。

修改文件：

- `backend/harness/loop/single_agent_turn.py`
- `backend/harness/entrypoint/runtime_facade.py`
- `backend/runtime/shared/models.py`
- `backend/runtime/memory/state_index.py`
- `backend/harness/runtime/session_timeline.py`
- `backend/harness/runtime/run_monitor/service.py`
- `backend/tests/session_runtime_lifecycle_regression.py`

设计：

1. `turn_id` 继续表示会话 turn。
2. `turn_run_id` 改为：

```text
turnrun:{turn_id}:{short_run_nonce}
```

或：

```text
turnrun:{stream_run_id}
```

最终选择以 `stream_run_id` 优先，因为 ChatRun 已经是传输层唯一运行 handle。

3. `session_turn_runs` 允许同一 `turn_id` 对应多个 TurnRun。
4. timeline 展示以最新 committed assistant message 为主，debug trace 可展示多次尝试。
5. 旧的 `turnrun:{turn_id}` 只作为历史记录读取，不再写入。

完成标准：

- 同一 turn 多次重试不会写入同一个 JSONL。
- StateIndex 中 TurnRun latest 不会覆盖旧运行。
- 运行监控能区分旧 attempt 与当前 attempt。

### 阶段 6：前端状态消费收敛

目标：

- 前端不再把 stream 完成、activeTurnSnapshot、TaskRun monitor 分别解释成三套状态。
- `done` 仍然是 SSE 终止，但它必须代表后端已语义收尾。

修改文件：

- `frontend/src/lib/api.ts`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/lib/store/events.ts`
- `frontend/src/components/chat/agentRunProjection.ts`
- `frontend/src/lib/api/client.test.ts`
- `frontend/src/lib/store/runtime.test.ts`

设计：

1. `streamChat()` 保持收到 terminal 后停止读取，但测试改为后端 terminal 顺序契约。
2. `reduceStreamEvent()` 对 `single_turn_tool_iteration_limit` 不再显示为普通成功，可显示为 warning/blocked。
3. `activeTurnSnapshot` 只用于 live steer，不用于判断是否存在 current resumable task。
4. current task 的可续接状态来自后端 monitor/current-work projection。
5. 若 answer leak flag 表示内部协议泄漏，前端不展示为正常助手回答；后端修复后该路径应只作为防御。

完成标准：

- 用户不会看到 `>` 作为回答。
- 工具上限不会显示为“已处理 1 个命令”的成功状态。
- 可续接任务显示为同一个任务继续，而不是新一轮普通聊天。

## 7. 删除和收敛项

实施时应删除或收敛以下旧逻辑，禁止长期保留双链路：

1. 删除 `turnrun:{turn_id}` 作为新 TurnRun 写入 ID 的路径。
2. 删除工具上限 synthesis 直接持久化 raw model content 的路径。
3. 删除“public done 可以先于 TurnRun terminal”的测试假设。
4. 收敛 `_resume_current_task_for_task_request()`，不要只服务 `request_task_run`。
5. 收敛前端对 active task / stream / active turn 的重复推断。

允许保留的兼容边界：

- 历史 `turnrun:{turn_id}` 记录只读展示。
- 旧 session 中已持久化的 `>` 不做自动修改，除非单独写迁移脚本。

## 9. 风险与控制

风险 1：CurrentWorkBoundary 过度拦截新任务。

- 控制：结构化裁决必须有 `new_independent_turn_allowed`，并记录原因。
- 测试：有 current task 时，用户提出完全无关问题，应能普通回答或开启新 turn。

风险 2：TurnRun ID 变更影响 timeline 和 monitor。

- 控制：turn_id 与 turn_run_id 明确分层；timeline 用 turn_id 聚合，debug 用 turn_run_id 定位。
- 测试：同一 turn 多 attempt 仍能找到最新公开消息和所有 trace。

风险 3：工具上限 fail closed 让回答变短。

- 控制：允许基于已有观察回答，但禁止继续工具调用协议。
- 测试：已有足够观察时仍能生成可见结论；协议泄漏时 blocked。

风险 4：前端旧 cursor 恢复逻辑受 terminal 顺序影响。

- 控制：public `done` 仍是最后 SSE 终止事件；只改变后端在 `done` 前确保内部 terminal 已落盘。
- 测试：现有 reconnect 测试保留，增加 terminal-before-done 后端契约。

## 10. 实施顺序

推荐顺序：

1. 阶段 3：先修 TurnRun terminal 与 ChatRun done 顺序。
2. 阶段 4：修工具上限和协议泄漏。
3. 阶段 1 + 阶段 2：建立 CurrentWorkBoundary 并接入 TaskRun 续跑。
4. 阶段 5：唯一化 TurnRun ID。
5. 阶段 6：前端状态消费收敛。

原因：

- 先修 terminal 顺序和 output boundary，可以立刻停止制造错误状态和 `>`。
- 再修 current work，可以解决“继续需要第二遍”的核心体验。
- 最后做 TurnRun ID 唯一化和前端收敛，降低一次性改动风险。

## 11. 审阅点

实施前需要确认三点：

1. CurrentWorkBoundary 遇到 current TaskRun 时，是否允许用户开启无关新请求。建议允许，但必须通过结构化裁决记录原因。
2. TurnRun ID 是否采用 `turnrun:{stream_run_id}`。建议采用，因为 ChatRun 已经是一次用户提交的唯一传输 handle。
3. 工具上限 synthesis 遇到协议泄漏时，是否直接返回 blocked。建议返回 blocked，不再尝试二次修复，避免继续消耗模型和工具预算。

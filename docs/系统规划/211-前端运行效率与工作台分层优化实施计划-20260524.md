# 前端运行效率与工作台分层优化实施计划

日期：2026-05-24

目标：执行 `210-前端工作台结构与性能优化审查报告-20260524.md` 中 P0-P4 的优化要求，在不保留无用旧逻辑的前提下，完成一轮可构建、可测试、可继续扩展的结构性改造。

## 设计判断

产品类型：agent 工作台、任务图编辑器、运行监控控制台。

用户角色：创作者/系统编排者，需要配置任务图、观察运行、处理人工决策、维护能力与记忆。

主对象：任务图、运行实例、agent 能力、记忆资产。

界面原则：

1. 数据密集但可扫描，主界面优先帮助用户行动。
2. 不把内部协议字段作为一级信息展示。
3. 不同层级通过明确入口切换，原始诊断进入诊断抽屉。
4. 运行监控只允许一个前端控制器管理订阅与兜底轮询。
5. 危险操作使用统一确认弹窗，不再使用 `window.confirm`。

## 实施步骤

### P0：运行监控控制器收敛

1. 新增共享运行时钟，消除组件内重复秒级 interval。
2. 将 `TaskGraphRunInteractionDock` 提升为单点挂载。
3. 在 `WorkspaceRuntime` 内建立全局运行监控 controller 状态：SSE、fallback poll、detail refresh、visibility backoff 由同一组方法管理。
4. 避免 SSE connected 时保持短周期 fallback poll。
5. 增加运行时钟与 runtime store 测试。

### P1：API 与状态切片第一阶段

1. 新建 `frontend/src/lib/api/client.ts`，集中 `getApiBase`、`request`、运行监控 SSE URL。
2. `api.ts` 继续作为兼容导出入口，但移除本地重复 request 实现，改为调用 client。
3. 后续领域 API 拆分以 client 为基础，不再在页面里新增 ad hoc fetch。

### P2：任务系统工作台分层与诊断收纳

1. 将 `Graph Builder`、`Compiled View` 改为中文业务层级名称。
2. 将任务图执行包里的原始 diagnostics 和 contract manifest 移入折叠诊断区。
3. 将 `runtime.length_budget` 等内部路径从主文案下沉为诊断/来源说明。
4. 保留真实诊断能力，但默认不压在主工作流上。

### P3：大型视图按需加载第一阶段

1. 对主入口中的重型系统页使用 `next/dynamic` 拆分初始 bundle。
2. 保留主会话首屏同步加载，避免影响主要入口。
3. 后续再按 view-model 拆分大文件内部结构。

### P4：交互一致性与测试

1. 新增统一 `ConfirmDialogProvider` 与 `useConfirmDialog`。
2. 替换当前主要 `window.confirm`：任务系统、健康系统、记忆、能力系统、编排分组、契约库。
3. 增加纯逻辑/组件可测部分的回归测试。
4. 运行前端相关测试与生产构建。

## 验收标准

1. `TaskGraphRunInteractionDock` 在入口页只挂载一次。
2. 运行监控秒级刷新使用共享 ticker。
3. `api.ts` 不再自带底层 request 实现，底层请求进入 `api/client.ts`。
4. 任务图主导航不再出现 `Graph Builder`、`Compiled View`。
5. 主要危险操作不再调用 `window.confirm`。
6. 测试与 `next build` 通过。

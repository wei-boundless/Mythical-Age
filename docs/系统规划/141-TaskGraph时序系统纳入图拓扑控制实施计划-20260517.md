# TaskGraph时序系统纳入图拓扑控制实施计划

日期：2026-05-17

## 一、问题判断

当前 TaskGraph 已经有时序账本，但时序还没有真正成为图拓扑调度的控制条件。

现有链路中，`timeline_ledger` 会为 `node_dispatch_requested`、`node_result_received`、`node_execution_receipt_committed`、`scheduler_evaluated` 等事件分配 `clock_seq`，`stage_execution_request` 也携带 `dispatch_context`，`stage_execution_receipt` 也记录 `effective_from_clock_seq`。但是调度器主要还是根据：

- `node_statuses`
- 图边依赖
- phase / sequence
- ready / running / completed

来判断下一个节点是否可以运行。

这会导致一个结构性风险：只要某个节点被标为 `completed`，下游就可能继续运行，即使这个完成结果不属于当前批次、当前 round、当前 dispatch，或者没有 accepted receipt。

写作任务中暴露出来的问题是：

1. `memory_commit_chapter` 完成后，`chapter_progress_router` 可能只因上游节点 completed 而继续，而不是因为当前批次 accepted commit receipt 已生效。
2. rejected draft 虽然在业务上失败，但如果产物已经提前落盘，用户会看到它污染正式目录。
3. late result / stale result 如果没有被强制 no-op，可能触发后续调度。
4. 同一批次同一 draft/review 的重复 commit 没有被时序坐标和 source identity 双重约束。
5. 监测节点能看到运行异常，但缺少一个明确的“时序不合法”诊断模型。

因此，真正缺失的不是事件记录，而是：

> 时序坐标必须成为节点可运行、结果可接受、产物可见、下游可推进的硬约束。

## 二、设计目标

把时序系统从“运行记录层”提升为“图拓扑控制层”。

每一次节点运行都必须具备确定的时序坐标：

```text
coordination_run_id
stage_id / node_id
dispatch_event_id
request_id
clock_seq
scope_path
phase_id
loop_frame_id
iteration_index
batch_start_index / batch_end_index / round_index（如业务有批次）
```

每一次结果接受都必须生成可验证的执行收据：

```text
receipt_id
request_id
dispatch_event_id
result_event_id
stage_id / node_id
accepted
effective_from_clock_seq
effective_scope_path
produced_artifact_refs
source_artifact_refs
commit_identity（如是提交节点）
```

下游节点不再只依赖“上游节点 completed”，而是依赖“上游在当前时序坐标下产生了满足边契约的 accepted receipt”。

## 三、核心概念

### 1. Timeline Coordinate

新增运行时内部概念 `TaskGraphTimelineCoordinate`，不一定先作为独立 dataclass 暴露给前端，但运行时必须统一生成和校验。

建议字段：

```python
coordination_run_id: str
stage_id: str
node_id: str
dispatch_event_id: str
request_id: str
clock_seq: int
scope_path: tuple[str, ...]
phase_id: str
loop_frame_id: str
iteration_index: int
batch_start_index: int
batch_end_index: int
round_index: int
```

其中：

- `clock_seq` 是系统 clock。
- `scope_path` 是层级时序路径，例如 `run/phase.chapter_loop/loop.chapter_batch/volume_001/chapter_001_010/round_003`。
- `iteration_index` 是循环序号。
- `batch_start_index/batch_end_index/round_index` 是业务可选坐标，不写作任务也可以为空或 0。

### 2. Accepted Receipt

`StageExecutionReceipt` 是结果能进入下游的唯一凭证。

调度器不能只看 `node_statuses[source] == completed`，必须能检查：

```text
source node 在当前 target 需要的 scope 下是否存在 accepted receipt
receipt.effective_from_clock_seq > 0
receipt.produced_artifact_refs 满足 edge payload contract
receipt.dispatch_event_id 与 source request 匹配
receipt.scope_path 与 target 需要的 scope 兼容
```

### 3. Temporal Dependency

拓扑边需要增加时序依赖语义。

普通边：

```text
target 需要 source 的 accepted receipt。
```

批次边：

```text
target 需要同一 batch/round/scope 的 source accepted receipt。
```

提交边：

```text
target 需要 source commit receipt，并且 commit_identity 未重复。
```

返修边：

```text
target 需要 review receipt + revision packet，且 revision packet 绑定原稿 receipt。
```

## 四、目标运行流程

### 1. 派发阶段

`_stage_execute` 创建 `node_dispatch_requested` 事件后，必须生成完整 coordinate：

```text
dispatch_event_id = timeline event id
clock_seq = dispatch event clock
scope_path = stage scope + loop/batch/round
request_id = based on dispatch_event_id
```

然后：

- 写入 `StageExecutionRequest.dispatch_context`
- 写入 A2A payload metadata
- 写入 context packet metadata
- 写入 runtime assembly metadata
- 写入监测快照

完成标准：

```text
任何被派发节点都能从 request 中还原唯一时序坐标。
```

### 2. 结果接受阶段

`_stage_accept` 收到 `TaskResultReadyEvent` 时，先做时序校验：

1. event 有 `request_id/dispatch_event_id` 时，必须与当前 active request 匹配。
2. event 没有身份时，只允许兼容旧路径；新 TaskGraph run 必须视为 `result_without_dispatch_identity`。
3. stage_id 不匹配 active request 时，记录 stale，不更新 stage result，不触发 route_next。
4. accepted 结果必须生成 receipt。
5. rejected 结果可以生成 rejected receipt，但 `effective_from_clock_seq = 0`，不进入下游可见集。

完成标准：

```text
stage_results 不能只按 stage_id 覆盖；必须保留 execution instance。
```

建议结构：

```python
stage_results_by_instance[receipt_id] = {...}
latest_stage_receipts[stage_id] = receipt_id
accepted_receipts_by_scope[scope_key][stage_id] = receipt_id
```

旧的 `stage_results[stage_id]` 可以暂时作为投影，但不能作为调度真相。

### 3. 调度阶段

`task_graph_scheduler` 引入 receipt-aware 判断。

节点 ready 条件从：

```text
all upstream source node completed
```

升级为：

```text
all required upstream temporal dependencies satisfied
```

即：

```text
source node completed
AND source has accepted receipt
AND receipt scope matches target required scope
AND edge contract required artifact refs are present
AND receipt effective clock is visible at target dispatch clock
```

如果缺失，节点 blocked reason 应该明确：

```text
missing_receipt:memory_commit_chapter
scope_mismatch:chapter_review
receipt_not_effective:chapter_draft
missing_artifact_ref:chapter_outline
duplicate_commit_identity:memory_commit_chapter
```

### 4. 下游上下文装配阶段

`resolve_context_packets` 只能装配当前时序坐标可见的 receipt 和 artifacts。

例如 `chapter_draft` 必须读取当前批次的 `chapter_outline` accepted receipt：

```text
chapter_outline receipt.scope_path == chapter_draft target scope_path
chapter_outline accepted == true
chapter_outline produced_artifact_refs 不为空
```

否则：

```text
chapter_draft blocked
不启动模型
不允许写手声明“上游未提供细纲”后硬写
```

### 5. 监测与纠错阶段

通用异步监测节点读取：

- timeline ledger
- scheduler state
- current stage execution request
- execution receipts
- artifact materialization diagnostics

生成 `temporal_integrity_report`。

重点监测：

```text
dispatch_without_clock
result_without_dispatch_identity
result_stage_mismatch
late_or_stale_result
accepted_without_receipt
receipt_scope_mismatch
downstream_ready_without_receipt
router_without_commit_receipt
commit_without_current_batch_scope
duplicate_commit_identity
rejected_artifact_visible
artifact_context_missing_required_receipt
```

可纠错动作：

```text
ignore_stale
block_downstream
quarantine_artifact
reissue_dispatch
resume_from_last_valid_receipt
request_human_decision
fail_closed
```

监测节点不是特殊系统外脚本，而是 TaskGraph 后台节点。它可以常驻运行交互窗口，但它的输入必须来自运行时结构化快照，不靠读取散乱日志猜测。

## 五、文件级实施计划

### 阶段一：时序坐标模型收口

目标：统一生成和提取时序坐标。

涉及文件：

- `backend/orchestration/runtime_loop/timeline_ledger.py`
- `backend/orchestration/runtime_loop/stage_execution_request.py`
- `backend/orchestration/runtime_loop/stage_execution_receipt.py`
- `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`

实施内容：

1. 新增 timeline coordinate 构造/解析 helper。
2. `_stage_scope` 纳入 loop frame、batch、round。
3. `StageExecutionRequest.dispatch_context` 固化 coordinate 字段。
4. `StageExecutionReceipt` 增加 `source_artifact_refs`、`source_receipt_ids`、`commit_identity`、`scope_key`。

完成标准：

- 派发 request 可还原完整 coordinate。
- receipt 可还原 result 所属 coordinate。
- 无坐标的新 TaskGraph 节点结果会被标记为异常。

### 阶段二：结果实例化存储

目标：消除 `stage_results` 单键覆盖。

涉及文件：

- `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
- `backend/orchestration/runtime_loop/coordination_trace_adapter.py`
- `backend/orchestration/runtime_loop/trace_reader.py`

实施内容：

1. 增加 `stage_results_by_instance`。
2. 增加 `stage_receipts_by_scope`。
3. `stage_results[stage_id]` 降级为最新 accepted 投影。
4. stale / duplicate / rejected 都必须有单独集合，不污染 accepted 投影。

完成标准：

- late result 不覆盖当前阶段状态。
- 同 stage 多 round 结果可并存。
- 当前 accepted 投影只来自有效 receipt。

### 阶段三：scheduler 引入 receipt-aware gates

目标：下游 ready 不再只看上游 completed。

涉及文件：

- `backend/orchestration/runtime_loop/task_graph_scheduler.py`
- `backend/orchestration/runtime_loop/task_graph_scheduler_models.py`
- `backend/tasks/coordination_graph_models.py`
- `backend/tasks/coordination_graph_compiler.py`

实施内容：

1. Runtime edge 增加 temporal dependency policy。
2. Scheduler 输入增加 `receipt_index`。
3. `_node_ready` 增加 receipt 校验。
4. edge state 输出 receipt 诊断。

完成标准：

- `chapter_progress_router` 缺当前批次 commit receipt 时 blocked。
- `chapter_draft` 缺当前批次 outline receipt 时 blocked。
- blocked reason 可被前端和监测节点直接展示。

### 阶段四：上下文装配只消费有效时序产物

目标：节点收到的 artifact context 必须来自当前时序可见 receipt。

涉及文件：

- `backend/orchestration/runtime_loop/context_packet_resolver.py`
- `backend/orchestration/runtime_loop/artifact_refs.py`
- `backend/orchestration/runtime_loop/continuation_inputs.py`
- `scripts/configure_writing_simple_novel.py`

实施内容：

1. artifact context policy 增加 `required_receipt_scope`。
2. resolver 只展开当前 coordinate 可见 artifacts。
3. 缺必需 artifact 时返回 blocked issue，不生成模型请求。
4. 写作配置中将 `chapter_outline -> chapter_draft -> chapter_review -> memory_commit_chapter -> chapter_progress_router` 全部绑定当前 batch scope。

完成标准：

- 写手不会在缺细纲时启动。
- 审核和提交读取同一个 draft/outline scope。
- progress router 读取当前 commit receipt，而不是目录里任意 commit 文件。

### 阶段五：监测节点纳入时序完整性

目标：运行监测把“不在时序中的节点运作”作为一等异常。

涉及文件：

- `backend/orchestration/runtime_loop/task_graph_run_monitor.py`
- `backend/orchestration/runtime_loop/task_graph_monitoring.py`
- `frontend/src/components/workspace/views/task-system/TaskGraphRunInteractionDock.tsx`
- `scripts/configure_writing_simple_novel.py`

实施内容：

1. 增加 temporal integrity report。
2. 监测节点读取 scheduler blocked reason 和 receipt issue。
3. 运行交互窗展示当前 clock、active coordinate、异常节点、建议动作。
4. 对明确可纠错问题执行控制动作：
   - stale result：ignore
   - rejected artifact visible：quarantine
   - downstream ready without receipt：block
   - missing current request：reissue dispatch 或 request human decision

完成标准：

- 悬浮窗可看到当前时序坐标。
- 不在时序里的结果会被标记并阻断推进。
- 可自动纠错的异常有明确 action 和 trace。

### 阶段六：写作任务回归与旧残留清理

目标：验证写作长任务不会再出现批次串线、重复 commit、 rejected 污染。

涉及文件：

- `backend/tests/langgraph_coordination_runtime_regression.py`
- `backend/tests/task_artifact_materializer_regression.py`
- 新增 `backend/tests/task_graph_temporal_control_regression.py`
- `scripts/configure_writing_simple_novel.py`
- 输出目录清理脚本或运行前检查逻辑

测试覆盖：

1. 缺 outline receipt 时 chapter draft blocked。
2. rejected draft 不进入正式目录。
3. stale dispatch result 不触发 route_next。
4. duplicate commit 不触发第二个 commit。
5. progress router 缺 commit receipt 时 blocked。
6. 同 stage 多 round 不互相覆盖。
7. 监测报告能识别 result_without_dispatch_identity。

## 六、写作任务中的具体控制链

章节批次应当变成：

```text
chapter_outline
  produces accepted outline receipt for batch N

chapter_draft
  requires outline receipt for batch N
  produces accepted/rejected draft receipt

chapter_review
  requires outline receipt + draft receipt for batch N
  produces review receipt

memory_commit_chapter
  requires outline receipt + draft receipt + review receipt for batch N
  produces commit receipt with commit_identity

chapter_progress_router
  requires commit receipt for batch N
  decides next batch or volume review
```

不能再是：

```text
memory_commit_chapter node completed -> progress router ready
```

必须是：

```text
memory_commit_chapter accepted commit receipt for current batch -> progress router ready
```

## 七、前端表达

拓扑页可以展示时序状态，但不塞复杂表单：

- 节点角标：当前 clock、running、blocked、accepted receipt。
- 边状态：payload ready / receipt missing / scope mismatch。
- 监测浮窗：当前 active coordinate 和异常列表。

发布页和运行页展示诊断：

```text
节点 chapter_progress_router 被阻止：
缺少 memory_commit_chapter 在 scope run/phase.chapter_loop/volume_001/chapter_001_010/round_003 下的 accepted commit receipt。
```

## 八、不做什么

1. 不把时序做成普通业务节点。
2. 不靠 prompt 要求 agent 自觉遵守时序。
3. 不让文件目录是否存在决定下游能否读取。
4. 不允许 rejected artifact 被下游作为正式输入。
5. 不恢复 task_mode 旧链路。
6. 不用外部看门狗替代 TaskGraph 内部监测节点。

## 九、验收标准

工程验收：

- 每个派发 request 都有完整 timeline coordinate。
- 每个 accepted result 都有 receipt。
- scheduler ready 判断可以输出 receipt missing / scope mismatch。
- stale / duplicate / rejected 不会推进拓扑。
- 监测报告能独立识别时序异常。

写作任务验收：

- round_003 通过后，只能产生一个 commit。
- round_006 到 round_011 rejected 不出现在正式章节目录。
- 第 11-20 章写作不会读取第 1-10 章旧 dispatch。
- 写手 prompt 中可见当前批次细纲；缺细纲则不派发。
- progress router 必须基于当前批次 commit receipt 推进。

## 十、实施顺序建议

优先级如下：

1. 先做 coordinate + receipt index。
2. 再做 scheduler receipt gate。
3. 再做 artifact context receipt filtering。
4. 再做 monitor temporal integrity report。
5. 最后重跑写作任务验证。

原因是：如果 scheduler 还不消费 receipt，监测只能报警，不能真正阻断错误推进；如果 context resolver 不消费 receipt，写手仍可能拿到错误批次产物。

# 写作长任务后台监督钩子实施记录

## 目标

为《洪荒时代》长任务提供低频后台监督：

- 不高频打扰运行链路。
- 监督器进程退出、状态文件长时间不更新或后端 8004 不健康时自动恢复。
- 在非人工停止的情况下，5 分钟内重新拉起监督链路。

## 本次实现

### 1. 后台 watchdog

新增：

- `scripts/watch_writing_supervision.py`
- `scripts/start_writing_watchdog.ps1`

默认策略：

- 每 120 秒检查一次。
- 监督状态超过 300 秒不更新即视为失活。
- 后端 `/health` 不可用时先重启 8004 后端。
- 项目存在 active run 时 attach 到现有 run。
- 项目未完成且没有 active run 时，允许按启动参数创建新监督 run。
- `STOP_SUPERVISION.flag` 存在时暂停自动拉起，避免覆盖人工停止。

### 2. Codex Stop hook

新增：

- `.codex/hooks/writing_supervision_watchdog.py`

更新：

- `.codex/hooks.json`

Stop hook 现在先确保后台 watchdog 存活，再运行原有项目监测 hook。这个 hook 本身不阻塞、不产出任务内容，只负责监督链路保活。

### 3. 写作任务配置脚本更新

更新：

- `scripts/configure_writing_simple_novel.py`

调整点：

- 管理标记升级到 `codex_writing_simple_novel_config_20260517`。
- 设计依据切换到 `docs/系统规划/122-TaskGraph投影Prompt与交接包配置体验设计-20260517.md`。
- 契约说明明确记忆读写由 `memory_read`、`memory_write_candidate`、`memory_commit` 边声明。

### 4. 运行时补强

更新：

- `backend/execution/model_runtime.py`
- `backend/execution/model_response.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`

调整点：

- 将 `ReadError`、`RemoteProtocolError`、断块、服务端断连等传输中断识别为可重试错误。
- 流式输出中途失败时，按同一份 messages 触发一次真实的非流式模型调用恢复，不使用残缺流式文本冒充结果。
- 写入 `model_stream_recovery` 事件，方便后续排查是否发生过流式恢复。
- TaskGraph 节点声明 `artifact_policy.required` 时，空 `final_content` 不再被误判为无需 artifact 校验。
- 写作配置脚本的 stream policy 显式开启 `fallback_to_non_stream_on_error`。

### 5. 监督 attach 与误重启修正

更新：

- `scripts/watch_writing_supervision.py`
- `scripts/start_writing_supervisor.ps1`

调整点：

- watchdog 启动 supervisor 时，优先从 `active_coordination_run_id` 反解根 task run，避免旧项目状态把 supervisor attach 到子任务。
- `start_writing_supervisor.ps1 -AttachExisting` 收到子任务 id 时，也会用 coordination run 反解出的根 task run 覆盖。
- watchdog 不再因为一次 `/health` 超时就重启后端；现在连续两次健康检查失败才允许重启，并且若 supervisor 状态仍在 300 秒新鲜窗口内，会先等待，避免长模型流式生成被误杀。

### 6. TaskGraph monitor 正文流输出口修正

更新：

- `backend/api/orchestration.py`
- `backend/orchestration/runtime_loop/trace_reader.py`
- `backend/tests/runtime_loop_live_monitor_test.py`

调整点：

- TaskGraph 根任务继续负责图、时序、协调状态与监督状态。
- 每个正在执行的图节点按 agent 子任务运行，`model_item_received` 正文流由该子任务事件日志产生。
- monitor 现在会根据 coordination state 的 `active_stage_id` / `active_node_id` / `running_nodes` 定位当前节点输出口，并合并根任务事件与当前节点子任务事件。
- 同一节点存在旧的 `running` 子任务残留时，优先选择最近真实产生 `model_item_received` 的子任务，避免前端读到 stale 输出口。
- 前端仍只读取统一的 TaskGraph monitor 视图，不需要自己猜测子任务事件文件。
- stage agent 后台执行从 uvicorn 主事件循环移到独立 daemon thread，避免长正文流生成期间 `/health` 和 monitor API 被运行任务饿住。

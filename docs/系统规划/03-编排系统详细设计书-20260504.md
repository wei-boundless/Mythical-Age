## 二、编排系统总定义

编排系统是平台的 Agent 管理系统、执行装配系统和运行结构系统。

它不是纯权限面板，也不是只管理 allow / deny 的安全层。

它负责的是：

- Agent 名册、分类和生命周期
- Worker / 子 Agent blueprint
- 任务到执行主体的绑定策略
- 单 Agent 的内部结构编排
- 单 Agent 的权限与资源边界
- 单 Agent 的上下文作用域约束
- 单 Agent 的运行壳装配
- 多 Agent 协作拓扑
- Agent 间通信、handoff、review、merge 规则
- 单 Agent 与其他系统之间的交接接口

它回答的问题是：

`一个任务最终由哪些 Agent 承接、如何协作承接，以及每个 Agent 作为独立执行体，身体怎么被装起来、内部怎么组织、能做什么、不能做什么、怎么与平台其他系统交接`

## 三、为什么不能只叫“权限系统”

单 Agent 的运行体不仅涉及：

- 权限
- 工具
- 资源

还涉及：

- prompts 装配顺序
- memory scope
- runtime lane
- output boundary
- 运行壳
- 与任务系统、记忆系统、灵魂系统、能力系统的对接点

所以更准确的名字必须体现：

- 它管理的是 Agent 名册、执行主体、单体身体和多体协作结构
- 它是一个完整的运行结构系统

## 四、编排系统的硬边界

### 4.1 编排系统不负责任务定义

任务定义归任务系统。

编排系统不能决定：

- 这是什么任务
- 用哪个 workflow
- 任务为什么需要协作
- 任务验收标准是什么

但编排系统可以根据任务系统给出的执行意图、能力要求、安全要求和验收要求，决定：

- 采用哪个 Agent
- 是否派生工作子 Agent
- 是否进入多 Agent 协作
- 使用哪个 Agent runtime profile
- 使用哪个协作拓扑和 handoff 协议

### 4.2 编排系统不负责记忆内容筛选

记忆内容本体归记忆系统。

编排系统只负责：

- 记忆作用域约束
- 哪类记忆允许进入该运行体
- 是否允许写回某层记忆

它不负责从记忆系统里自己挑事实内容。

### 4.3 编排系统不负责人格与投影素材

灵魂系统负责：

- soul prompt
- projection prompt
- 人格设定
- 上下文表达姿态

编排系统只负责把这些素材装入运行壳。

### 4.4 编排系统不负责能力资源本体

能力资源归能力系统。

编排系统只负责：

- 这个 Agent 能拿哪些资源
- 哪些资源需审批
- 哪些资源禁止执行

## 五、编排系统在平台中的位置

任务系统解决的是：

`做什么，为什么这样做，需要什么要求和验收`

编排系统解决的是：

`由谁做，怎么协作做，每个 Agent 怎么活起来去做`

因此编排系统在平台中处于一个承上启下的位置：

- 上承任务系统、记忆系统、灵魂系统、能力系统
- 下接 runtime 执行层

如果没有编排系统，平台会出现两个问题：

1. 任务系统定义了任务，但没有稳定的 Agent 管理与执行装配出口
2. runtime 会被迫自己临时拼装 prompts、资源和上下文，导致行为不稳定

## 六、编排系统的职责清单

编排系统至少负责以下十二类内容：

1. Agent 名册与分类管理
2. Agent 默认运行 profile 管理
3. Worker / 子 Agent blueprint 管理
4. 任务到 Agent 的绑定策略管理
5. Agent prompts 装配结构管理
6. Agent 工具与资源边界管理
7. Agent memory scope 管理
8. Agent runtime lane 管理
9. Agent output boundary 管理
10. 多 Agent 协作拓扑与 handoff 管理
11. Agent 与其他系统的交接接口管理
12. Agent 运行壳装配与运行体视图输出

## 七、编排系统中的二层身体模型

编排系统不能把 Agent 的身体理解成一个完全静态的壳。

因为同一个 Agent 在面对不同任务时，内部状态流程本来就应该不同。

例如同一个 Agent 在以下任务中，身体内部流程就不该相同：

- 通用对话任务
- 开发实现任务
- 健康诊断任务
- 系统管理任务

因此，编排系统必须明确拆成两层身体模型。

### 7.1 稳定身体底座

这是 Agent 的长期稳定部分。

它定义：

- 这个 Agent 默认拥有哪些器官
- 默认跑在哪条 lane
- 默认能接哪类记忆
- 默认有哪些资源边界
- 默认 prompts 骨架是什么

这层对应：

- `AgentBodyProfile`

它回答的是：

`这个 Agent 平时天生长什么样`

### 7.2 任务态身体编排

这是 Agent 在承接某个具体任务后，身体内部这次怎么运行的任务态编排。

它定义：

- 本次内部状态顺序
- 每个状态开放哪些资源
- 每个状态注入哪些记忆
- 每个状态的 prompt 装配点
- 失败后回退到哪里
- 验证门禁在哪一层触发

这层对应：

- `TaskBodyOrchestration`

它回答的是：

`这个 Agent 这次为了承接这个任务，身体内部流程怎么变体`

### 7.3 为什么这层不能交给 workflow

workflow 属于任务系统，它定义任务方法。

但 workflow 一般只会表达：

- 先做什么
- 再做什么
- 最后做什么

它通常不会细到：

- 某个内部阶段开放哪些工具
- 什么时候允许写入
- 什么时候注入任务记忆
- 哪个阶段挂验证门禁
- 失败之后退回哪个内部状态

这些是单 Agent 的身体编排问题，不是任务方法文本本身的问题。

所以：

- workflow 归任务系统
- body orchestration 归编排系统

## 九、编排系统的装配链路

编排系统的主链路建议固定为：

```text
任务系统输出 TaskExecutionAssembly
  -> 记忆系统输出 MemoryBundle
  -> 灵魂系统提供 soul / projection prompt assets
  -> 能力系统提供 operation / tool / skill catalog
  -> 编排系统读取 AgentBodyProfile
  -> 编排系统生成 TaskBodyOrchestration
  -> 编排系统装配 PromptStructureProfile
  -> 编排系统装配 ResourcePolicyProfile
  -> 编排系统装配 MemoryScopeProfile
  -> 编排系统生成 AgentRuntimeSpec
  -> runtime 消费 AgentRuntimeSpec
```

### 9.1 为什么编排系统必须在 runtime 之前完成装配

因为 runtime 不应该负责：

- 决定装什么 prompt
- 决定开什么资源
- 决定记忆能进多少
- 决定输出边界是什么

runtime 只负责执行，不负责临时拼身体。

### 9.2 任务系统与编排系统在流程上的分工

这里要明确区分两套流程：

#### 任务级流程

归任务系统：

- 任务识别
- Agent 采用
- workflow 绑定
- flow contract 绑定
- 多 Agent 拓扑分配
- 通信协议管理

#### Agent 级流程

归编排系统：

- 内部状态顺序
- 内部资源开放顺序
- 内部记忆注入时机
- 内部 prompt 装配点
- 内部验证与回退规则

也就是说：

- 任务系统定义任务怎么走
- 编排系统定义单 Agent 身体内部怎么走

## 十、单 Agent 的身体到底由哪些部分组成

为了避免“身体”这个概念太抽象，这里把单 Agent 运行体拆成六个器官层。

### 10.1 身份层

包括：

- `agent_id`
- `agent_category`
- `interface_target`

这是 Agent 被平台识别和对接的基础。

### 10.2 骨架层

包括：

- `runtime_lane`
- `prompt_structure`
- `execution_style`

这是 Agent 的结构骨架。

### 10.3 肌肉层

包括：

- tools
- operations
- skills
- 外部资源调用能力

这是 Agent 真正能动起来的能力层。

### 10.4 神经层

包括：

- memory scope
- context policy
- 输入输出边界

这是 Agent 感知和传导约束层。

### 10.5 语言层

包括：

- soul prompt
- projection prompt
- task / workflow prompt 装配

这是 Agent 的表达层。

### 10.6 接口层

包括：

- 与任务系统的装配接口
- 与记忆系统的 bundle 接口
- 与能力系统的资源接口
- 与健康系统的状态上报接口

这是 Agent 的平台对接层。

## 十一、与记忆系统的接口

编排系统与记忆系统的关系，不是去拿“某条记忆内容”，而是约束“哪类记忆能进身体”。

建议固定接口：

- 输入：`MemoryBundle`
- 约束：`MemoryScopeProfile`
- 输出：进入 `AgentRuntimeSpec` 的记忆运行视图

编排系统要回答：

- 这份 bundle 能不能全进
- 哪些层该被裁掉
- 哪些层只读
- 哪些结果执行后允许写回

## 十二、与灵魂系统的接口

灵魂系统提供素材，编排系统负责装配。

建议固定接口：

- 输入：`SoulProfile` / `ProjectionPromptAsset`
- 装配：`PromptStructureProfile`
- 输出：进入运行壳的 prompt manifest

编排系统要回答：

- soul prompt 放在哪一层
- projection prompt 放在哪一层
- workflow prompt 与 projection prompt 怎么错开
- 是否允许某类 Agent 省略某层 prompt

## 十三、与能力系统的接口

能力系统给能力资源目录，编排系统给具体可执行范围。

建议固定接口：

- 输入：`OperationRegistry` / `CapabilityCatalog`
- 装配：`ResourcePolicyProfile`
- 输出：`ResourceRuntimeView`

结合现有代码：

- [backend/orchestration/resource_policy_builder.py](/d:/AI应用/langchain-agent/backend/orchestration/resource_policy_builder.py)
- [backend/orchestration/resource_runtime_view.py](/d:/AI应用/langchain-agent/backend/orchestration/resource_runtime_view.py)

后续这两块正好可以吸收进编排系统的正式实现。

## 十四、与 runtime 的接口

runtime 的最小消费对象不应是零散字段，而应是正式运行体。

建议固定为：

- `TaskBodyOrchestration`
- `AgentRuntimeSpec`
- `RuntimeContextSnapshot`
- `ResourceRuntimeView`

现有运行时上下文相关骨架可吸收自：

- [backend/orchestration/runtime_loop/context_manager.py](/d:/AI应用/langchain-agent/backend/orchestration/runtime_loop/context_manager.py)

这里已经体现出几个重要方向：

- context snapshot 是正式对象
- prompt source report 应可追溯
- runtime invariant 应可检查

这和编排系统的目标是高度一致的。

### 14.1 runtime 为什么不能自己决定内部状态流程

因为那会带来三个结构性问题：

1. 同类任务在同类 Agent 上表现不稳定
2. 内部资源开放顺序不可审计
3. 失败回退策略变成模型临时猜测

所以 runtime 只能执行 `TaskBodyOrchestration`，不能自己临时编内部状态机。

## 十五、前端编排系统管理目标

编排系统前端不能做成“权限开关页”，而应做成 Agent 管理、执行装配和运行结构管理页。

至少应支持以下管理面：

### 15.1 Agent 名册管理

- 查看主 Agent、系统管理 Agent、工作子 Agent
- 创建、编辑、删除可编辑工作子 Agent
- 隐藏不适合普通用户理解的内部 ID
- 用中文名称、职责、适用任务和状态解释 Agent
- 查看 Agent 是否可作为 coordinator、participant 或 worker blueprint 来源

### 15.2 Agent 身体画像

- 查看某个 Agent 的默认运行体 profile
- 查看运行 lane
- 查看默认记忆作用域
- 查看默认资源边界
- 查看默认输出边界

### 15.3 任务绑定与派生策略

- 查看任务执行意图如何选择 Agent
- 查看允许派生 worker 的条件
- 查看 worker blueprint
- 查看 fallback Agent
- 查看任务系统引用的 orchestration policy

## 十六、现有代码的吸收与清理方向

- prompt 结构与资源策略尚未统一进一个运行体装配出口

- 运行体与记忆系统、灵魂系统的接口仍偏隐式

## 十七、实施顺序

建议按以下顺序实施：

1. 统一命名为编排系统
2. 拆出 `AgentBodyProfile`
3. 补出 `TaskBodyOrchestration`
4. 补出 `PromptStructureProfile / MemoryScopeProfile / AgentRuntimeSpec`
5. 吸收现有 `policy_builder` 和 `runtime_view`
6. 明确任务系统到编排系统的装配请求接口
7. 明确记忆系统与灵魂系统到编排系统的输入接口
8. 让 runtime 只消费正式 `TaskBodyOrchestration + AgentRuntimeSpec`
9. 最后再重构前端编排系统管理页

## 十八、执行结论

编排系统不是纯权限系统，而是单 Agent 的身体管理系统。

它的意义不是“再多做一个中间层”，而是把以下事情正式化：

- 单 Agent 内部怎么编排
- 单 Agent 能拿什么资源
- 单 Agent 能接什么记忆
- 单 Agent 的 prompts 怎么装
- 单 Agent 怎么作为一个执行体接上整个平台

并且，这个身体不是完全静态的。

每个 Agent 都应当同时具备：

- 稳定的身体底座
- 面向不同任务的任务态身体编排

只有这条线立起来，平台里的每个 Agent 才不是抽象名字，而是真正可以被稳定装配、稳定运行、稳定治理，并能根据任务切换内部状态流程的执行体。

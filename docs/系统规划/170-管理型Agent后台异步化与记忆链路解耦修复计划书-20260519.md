# 编排系统同步异步统一调度与记忆链路解耦修复计划书

日期：2026-05-19  
状态：修复计划书，尚未执行  
适用范围：Orchestration Runtime、TaskGraph、RuntimeLane、QueryRuntime、TaskRunLoop、MemoryFacade、MemoryMaintenanceCoordinator、durable memory、retrieval index rebuild、计划管理 Agent、记忆管理 Agent、长情景系统测试

## 0. 结论

这次六十轮真实对话卡住，不是 PDF 处理、RAG 检索或 SSE 采集本身的单点问题，而是系统结构违背了既有设计原则：**编排系统没有把同步前台执行与异步后台执行作为统一调度语义落地，导致管理型 Agent 的活动被挂在主请求链路里同步等待**。

当前实现中，主回答已经生成并写入会话后，仍要等待记忆维护 Agent、durable memory 写入、durable memory 检索索引重建等后置治理动作完成，`done` 事件才会输出。一旦后置治理动作耗时、重试或卡住，用户侧和长跑测试都会表现为“回答已经内部完成，但流不结束、artifact 不落盘”。

修复方向不是给某个 turn 加超时，而是在编排系统层面补齐同步/异步执行模型，然后把记忆链路作为第一批落地场景：

1. 编排系统必须显式支持 `sync`、`async`、`background`、`parallel`、`barrier`、`manual_gate` 等执行模式。
2. 用户可见主回答链路只负责生成回答、提交 assistant 消息、输出 `done`。
3. 计划管理 Agent、记忆管理 Agent、索引维护、摘要整理、运行健康检查等管理型活动必须进入后台任务体系。
4. 后台任务通过结构化 receipt / notification / runtime event 回写状态，不阻塞主响应。
5. `done` 出口必须具备强边界：主任务完成后，治理收尾失败只能降级记录，不能吞掉最终响应。

## 1. 问题背景与证据

### 1.1 用户侧现象

六十轮真实对话测试中，新运行目录只写到：

```text
output/test_runs/20260519-005636-long/artifacts/sixty-turn-real-user-marathon/turn-18-main.json
```

之后长时间没有新的 turn artifact。表面看像是 PDF 慢、web search 慢或 SSE 卡住，但进一步追踪发现内部状态并非完全停止。

### 1.2 关键运行证据

对应 session：

```text
6b4cfe84bb4e404aa286233632a42f49
```

会话文件已经写入第 33/34 条消息：

```text
storage/sessions/6b4cfe84bb4e404aa286233632a42f49.json

#33 user      记住：以后复杂问题先给结论。
#34 assistant 记住了。以后复杂问题先给结论，再补依据和细节。
```

durable memory 也已经落盘：

```text
storage/durable_memory/notes/user-prefers-action-oriented-summaries.md
```

其中包含：

```text
用户明确要求'以后复杂问题先给结论'
```

但 memory maintenance 状态仍停在旧索引：

```text
storage/runtime_state/memory_maintenance/6b4cfe84bb4e404aa286233632a42f49/state.json

last_memory_message_index = 32
```

同时 taskrun 33 的 runtime events 只到 `checkpoint_written`，没有进入：

```text
working_memory_finalized
loop_terminal
done
```

这说明主回答已经提交，durable memory note 已写入，但 post-commit 治理流程没有完整返回，导致 `done` 没有发出。

### 1.3 当前代码链路

当前主请求链路如下：

```text
QueryRuntime.astream()
  -> TaskRunLoop.run_single_agent_stream()
  -> assistant_message_committer()
  -> QueryRuntime._apply_assistant_message_commit_async()
  -> MemoryFacade.arun_memory_maintenance_after_commit()
  -> MemoryMaintenanceCoordinator.run_after_commit()
  -> MemoryMaintenanceAgent.maintain()
  -> durable note save
  -> on_durable_saved()
  -> AppRuntime._on_durable_memory_saved()
  -> RetrievalService.rebuild_durable_memory()
  -> TaskRunLoop yield done
```

问题在于：从 `assistant_message_committer()` 开始的后置管理链路仍然在主请求中 `await`，而且 durable memory 保存后的检索重建也是同步执行。

## 2. 设计原则冲突

现有设计原则已经明确区分前台同步 Agent 与后台异步 Agent：

- `docs/设计原则/12-Agent-系统.md`
  - 同步 Agent 在前台运行，父 Agent 等待其完成。
  - 异步 Agent 在后台运行，父 Agent 可以继续处理其他任务。
  - 异步 Agent 完成时应先更新状态，再执行通知丰富化，避免附加 API 调用或外部命令卡死状态转换。

- `docs/设计原则/14-任务系统.md`
  - 任务系统是并发执行引擎，用于统一管理后台 Shell、多 Agent 并行、记忆整理等异步工作。

当前实现冲突点：

| 设计原则 | 当前实现 | 风险 |
|---|---|---|
| 管理型 Agent 后台运行 | 记忆管理 Agent 在主请求内 `await` | 主回答被管理任务拖住 |
| 先完成状态，再做通知/丰富化 | durable save 后同步 rebuild index | 后置维护卡死 `done` |
| 异步工作进入任务系统 | memory maintenance 自己维护 `_in_progress/_pending` | 生命周期、通知、失败治理分散 |
| 后台任务失败不影响前台结果 | working memory / memory maintenance 在 `done` 前执行 | 治理失败表现成用户请求失败 |

### 2.1 当前代码已经有同步异步字段，但没有形成统一运行语义

这次不应该只给记忆系统补一个后台队列，因为 TaskGraph 侧已经出现了更通用的抽象：

```text
backend/tasks/task_graph_models.py

NODE_EXECUTION_MODES = {
  "sync",
  "async",
  "parallel",
  "background",
  "barrier",
  "manual_gate"
}
```

`backend/tasks/coordination_graph_compiler.py` 也已经读取：

```text
runtime_policy.default_execution_mode
node.execution_mode
node.background_policy
```

并能在预检中识别：

```text
sync
parallel
background
barrier
manual_gate
```

但当前运行态对这些语义只做了部分消费。也就是说，系统已经有“同步/异步/后台”的模型字段，却还没有一个统一的编排执行器来保证：

1. 哪些节点阻塞当前请求。
2. 哪些节点进入后台任务。
3. 哪些节点只发通知，不传递用户可见结果。
4. 哪些节点需要 barrier 汇合。
5. 哪些节点失败会阻断下游，哪些只记录 issue。

因此，这次修复应上升为编排系统能力，而不是只修 `MemoryMaintenanceCoordinator`。

## 3. 根因判断

根因不是“某个模型接口慢”或“某个 PDF 解析慢”，而是以下三层结构偏差叠加：

1. **编排系统没有统一的同步/异步执行语义**

   TaskGraph 定义里已经存在 `execution_mode` 和 `background_policy`，RuntimeLane 里也有系统管理场景，但 TaskRunLoop 仍主要按单条前台流推进，缺少面向所有节点和管理任务的统一调度层。

2. **管理型 Agent 没有后台化**

   `agent:1` 记忆管理 Agent 虽然在命名和 prompt 上已经是独立 Agent，但运行方式仍是主请求内同步 `await`。

3. **post-commit 治理和用户响应出口耦合**

   assistant 消息已经写入会话后，仍要等待 session memory、state memory、durable memory、working memory finalization 等治理动作完成，才输出 `done`。

4. **索引重建挂在 durable memory 保存回调里同步执行**

   durable note 保存成功后，`AppRuntime._on_durable_memory_saved()` 直接调用 `retrieval_service.rebuild_durable_memory()`。这类索引维护属于后台派生物，不应阻塞用户响应。

## 4. 目标架构

### 4.0 编排系统统一执行模式

编排系统应把“同步/异步”作为节点和运行任务的第一等语义，而不是由各子系统自己决定是否 `await`。

目标执行模式如下：

| execution_mode | 语义 | 是否阻塞当前请求 | 是否进入后台任务 | 下游推进 |
|---|---|---:|---:|---|
| `sync` | 当前请求内完成，结果参与本轮主回答 | 是 | 否 | 完成后推进 |
| `async` | 当前请求内发起，返回 task handle，结果稍后通知 | 否 | 是 | 按 wait_policy 判断 |
| `background` | 系统管理/维护任务，默认不产生用户可见回答 | 否 | 是 | 通常不阻塞下游 |
| `parallel` | 多个同级节点并行执行 | 视 join_policy | 是或否 | 按 join_policy 汇合 |
| `barrier` | 等待多个上游结果汇合 | 视 wait_policy | 否 | 满足条件后推进 |
| `manual_gate` | 等待人工或外部确认 | 否 | 否 | 手动 release 后推进 |

统一规则：

1. `TaskGraphNodeDefinition.execution_mode` 是节点执行模式权威字段。
2. `TaskGraphDefinition.runtime_policy.default_execution_mode` 是图级默认值。
3. `RuntimeLaneDescriptor.metadata` 可以声明默认执行姿态，例如 `foreground`、`background_management`、`resource_only`。
4. `TaskRunLoop` 不再直接用“函数是否 async”判断执行是否后台。
5. 所有后台执行都必须产生持久化 `BackgroundTask`，不能只存在进程内 pending 集合。

### 4.1 前后台职责边界

```text
前台主请求链路
  - 接收用户消息
  - 构建当前 turn runtime
  - 执行用户可见任务
  - 提交 user/assistant session message
  - 写最小 task_run terminal record
  - 输出 done

后台管理任务链路
  - session memory 整理
  - state memory 投影更新
  - durable memory 提取与写入
  - durable/session memory 检索索引重建
  - working memory finalization
  - 计划管理 Agent 状态整理
  - runtime monitor / stale run 检查
  - notification enrichment
```

### 4.2 编排调度层

新增统一调度层：

```text
OrchestrationExecutionScheduler
  - resolve_execution_mode()
  - dispatch_sync()
  - enqueue_async()
  - enqueue_background()
  - resolve_wait_policy()
  - resolve_join_policy()
  - emit_notification()
```

它负责把 TaskGraph、RuntimeLane、AgentRuntimeProfile 和当前请求类型统一起来，输出标准调度决策：

```text
ExecutionDispatchDecision
  - dispatch_id
  - execution_mode
  - runtime_lane
  - foreground_blocking
  - background_task_required
  - blocks_downstream
  - wait_policy
  - join_policy
  - notification_policy
  - failure_policy
```

该调度层应成为所有管理型 Agent、TaskGraph 节点、post-commit 治理任务的入口，避免子系统各自发明后台机制。

### 4.3 管理型 Agent 分类

管理型 Agent 指不直接生成当前用户可见回答，而是维护系统状态、计划、记忆、索引、健康和通知的 Agent。

当前至少包括：

| Agent / 任务 | 前台是否等待 | 目标运行方式 |
|---|---:|---|
| 记忆管理 Agent `agent:1` | 否 | 后台任务 |
| 计划管理 Agent | 否 | 后台任务 |
| durable memory extraction | 否 | 后台任务 |
| session/state memory maintenance | 默认否，必要时轻量同步 | 后台任务 + 最小同步投影 |
| retrieval index rebuild | 否 | 后台任务 |
| working memory finalization | 否 | fail-soft 后台收尾 |
| runtime monitor | 否 | 后台周期/事件触发 |

### 4.4 主响应 `done` 边界

`done` 输出前只允许执行必要的强一致动作：

1. user message commit。
2. assistant message commit。
3. task_result 最小终态记录。
4. checkpoint / event log 的最小终态记录。

不允许在 `done` 前等待：

1. 记忆管理 Agent 模型调用。
2. durable memory note 抽取和保存。
3. 检索索引重建。
4. session summary 长文本整理。
5. notification enrichment。
6. 后台计划管理 Agent 的复盘与修正建议。

## 5. 修复方案

### 阶段一：补齐编排同步/异步调度契约

目标：先把通用调度语义定死，让后续记忆链路和计划管理都走同一套编排能力。

任务：

1. 新增 `OrchestrationExecutionScheduler`：
   - 输入 TaskGraph node、runtime policy、RuntimeLane descriptor、AgentRuntimeProfile。
   - 输出 `ExecutionDispatchDecision`。

2. 标准化执行模式：
   - `sync`：当前请求内执行并等待。
   - `async`：创建后台任务，当前请求返回 task handle。
   - `background`：创建后台管理任务，默认不产生用户可见回答。
   - `parallel`：生成并行任务组。
   - `barrier`：只做汇合判断。
   - `manual_gate`：生成等待人工确认的 gate。

3. 标准化等待策略：
   - `wait_all_upstream_completed`
   - `wait_any_upstream_completed`
   - `wait_required_contracts`
   - `wait_handoff_ack`
   - `fire_and_continue`
   - `manual_release`

4. 标准化汇合策略：
   - `all_success`
   - `any_success`
   - `quorum`
   - `coordinator_decides`
   - `allow_partial_with_issues`
   - `fail_on_any_error`

5. 更新 compiler 预检：
   - `background` / `async` 不再只停留在模型字段层面，运行态必须能生成后台任务。
   - 未实现的策略必须 fail-closed，并给出明确 issue。

### 阶段二：止血边界，保证 `done` 先输出

目标：先修复六十轮测试卡住的主链阻塞。

任务：

1. 修改 `QueryRuntime._apply_assistant_message_commit_async()`：
   - 只同步 append assistant message。
   - 生成 `memory_maintenance_request`。
   - 投递后台任务。
   - 立即返回 `memory_maintenance_status=queued`。

2. 修改 `TaskRunLoop.run_single_agent_stream()`：
   - `done_event.output_commit.memory` 接受 `queued / succeeded / failed / skipped` 状态。
   - 不再把 memory maintenance 完成作为输出 `done` 的前置条件。

3. 为 post-commit 治理增加降级事件：
   - `memory_maintenance_queued`
   - `memory_maintenance_failed`
   - `memory_maintenance_completed`

4. 长跑测试要求：
   - 即使 memory maintenance 未完成，turn artifact 必须落盘。
   - `persisted_matches_done=true` 仍应成立。

### 阶段三：建立后台管理任务执行器

目标：把所有 `async` / `background` 编排活动纳入统一生命周期，而不是散落在 facade 内部。

建议新增或收口到现有任务体系：

```text
backend/orchestration/background_tasks/
  manager.py
  models.py
  store.py
  workers.py
```

最小模型：

```text
BackgroundTask
  - task_id
  - task_type
  - session_id
  - owner_task_run_id
  - status: queued | running | completed | failed | cancelled
  - idempotency_key
  - payload
  - result
  - error
  - created_at
  - updated_at
```

首批任务类型：

```text
memory_maintenance
durable_memory_index_rebuild
session_memory_index_rebuild
working_memory_finalization
plan_management
runtime_monitor
task_graph_async_node
task_graph_parallel_group
notification_enrichment
```

执行要求：

1. 同一 session 的 memory maintenance 串行合并，避免重复整理。
2. 同一 collection 的 index rebuild 去重合并，避免频繁重建。
3. 任务状态必须持久化，不能只在进程内 `_pending`。
4. 后台任务失败写 receipt，不影响已提交主回答。
5. 每个后台任务必须保留 `execution_mode`、`runtime_lane`、`wait_policy`、`join_policy`。
6. 后台任务完成后通过 notification / runtime event 唤醒需要继续推进的图。

### 阶段四：记忆维护链路重构

目标：把 `MemoryMaintenanceCoordinator` 从主请求内协程，改成后台 worker 可调用的纯任务处理器。

调整：

1. 保留 `MemoryMaintenanceCoordinator.run_after_commit()` 作为后台 worker 的处理函数。
2. 新增 `enqueue_after_commit()`：
   - 只创建 request / receipt stub。
   - 返回 queued receipt。
   - 不调用模型。

3. 移除或弱化当前 `_in_progress/_pending`：
   - 进程内状态只能作为优化缓存。
   - 权威状态进入 background task store。

4. durable memory 保存回调不再同步 rebuild：
   - `on_durable_saved()` 改为 enqueue `durable_memory_index_rebuild`。
   - rebuild task 合并连续请求。

5. session memory / state memory 更新策略：
   - 小型结构化投影可以同步写入。
   - 模型整理、长摘要、长期记忆抽取全部后台执行。

### 阶段五：计划管理 Agent 后台化

目标：统一“计划管理 Agent 的活动应该后台异步”的架构约束。

计划管理 Agent 的职责应是：

1. 观察当前 task graph / task run / stage result。
2. 维护计划状态、阻塞原因、下一步建议。
3. 生成可供主 Agent 或 UI 消费的结构化计划变更。
4. 触发 runtime monitor 或 recovery task。

不应做：

1. 阻塞当前用户回答。
2. 直接替主 Agent 输出用户可见正文。
3. 绕过 task graph / runtime lane 权限。
4. 把开发说明当 prompt 传给 Agent。

Prompt 约束示例：

```text
你是一名计划管理员。
你只负责维护当前任务计划的状态、阻塞点、下一步候选动作和风险提示。
你不负责直接回答用户，也不替执行 Agent 完成业务任务。
你需要输出结构化计划更新，并说明是否需要唤醒主 Agent 或调度后台修复任务。
```

### 阶段六：检索索引后台化

目标：彻底移除主请求内同步 rebuild。

调整：

1. `AppRuntime._on_durable_memory_saved()`：
   - 不直接调用 `runtime.retrieval_service.rebuild_durable_memory()`。
   - 改为投递后台 `durable_memory_index_rebuild`。

2. `AppRuntime.refresh_indexes_for_path()`：
   - API/文件变更触发时可选择同步或后台。
   - 默认对用户请求路径使用后台。
   - 维护命令可以显式同步执行。

3. `RetrievalService.rebuild_collection()`：
   - 保留同步实现作为 worker 内部调用。
   - 新增 rebuild request 去重、状态记录和失败 receipt。

4. 检索读取策略：
   - 新 note 写入后，在索引完成前，durable memory exact lookup / manifest scan 仍可读到文件。
   - RAG dense/lexical 索引延迟更新应在 diagnostics 中可见。

### 阶段七：工作记忆收尾 fail-soft

目标：工作记忆治理不能阻断最终响应。

调整：

1. `TaskRunLoop.finalize_working_memory()` 从 `done` 前强依赖改为 fail-soft。
2. 同步路径只写最小 intent 或 enqueue finalization task。
3. 后台完成后写：
   - `working_memory_finalization_completed`
   - `working_memory_finalization_failed`
4. `done_event.output_commit` 只标记：
   - `working_memory_finalization_status=queued`

### 6.0 编排调度决策

```json
{
  "dispatch_id": "dispatch:coordrun:node",
  "node_id": "memory_commit_world",
  "execution_mode": "background",
  "runtime_lane": "durable_memory_extraction",
  "foreground_blocking": false,
  "background_task_required": true,
  "blocks_downstream": false,
  "wait_policy": "fire_and_continue",
  "join_policy": "allow_partial_with_issues",
  "notification_policy": {
    "on_completed": "runtime_event",
    "on_failed": "issue_and_runtime_event"
  },
  "failure_policy": {
    "propagation": "isolate_failure",
    "record_issue": true
  },
  "authority": "orchestration.execution_scheduler"
```

```json
{
  "output_commit": {
    "assistant_commit_applied": true,
    "memory": {
      "memory_maintenance_status": "queued",
      "memory_maintenance_task_id": "bgtask:memory_maintenance:...",
```

## 8. 迁移策略

### 8.1 保持前台行为稳定

用户可见回答不应变化：

```text
记住了。以后复杂问题先给结论，再补依据和细节。
```

变化的是提交后的治理状态：

```text
memory_maintenance_status: queued
```

后台完成后再更新为：

```text
memory_maintenance_status: completed
durable_write_count: 1
```

### 8.2 兼容现有 API

短期保留：

```text
run_memory_maintenance_after_commit()
arun_memory_maintenance_after_commit()
```

但语义拆分：

| 方法 | 用途 |
|---|---|
| `enqueue_memory_maintenance_after_commit()` | 主请求使用 |
| `run_memory_maintenance_after_commit()` | 维护命令 / worker 使用 |
| `arun_memory_maintenance_after_commit()` | worker async 使用 |

### 8.3 清理旧残留

稳定后删除或降级：

1. `MemoryMaintenanceCoordinator._pending` 作为权威队列的角色。
2. 主请求内同步 durable index rebuild。
3. `done` 前强制 working memory finalization。
4. 只靠 `durable_write_count` 判断成功的旧诊断逻辑。

## 9. 风险与约束

### R1. 后台任务丢失

风险：进程退出导致内存队列任务丢失。

约束：后台任务必须持久化，至少写入 JSONL / SQLite store，不能只存在内存。

### R2. 记忆延迟可见

风险：用户下一轮立即问“你刚刚记住了什么”，后台任务可能尚未完成。

处理：

1. session message 已经写入，短期可从当前会话直接回答。
2. durable memory 未完成时，回答应说明“长期记忆整理仍在后台完成中”或通过 receipt 查询状态。
3. 对显式“记住”类 turn，可优先投递高优先级后台任务。

### R3. 索引延迟导致 RAG 暂时搜不到新记忆

风险：durable note 文件已保存，但向量/关键词索引尚未更新。

处理：

1. durable memory recall 先支持 exact lookup / manifest scan。
2. RAG 结果 diagnostics 标注 `index_rebuild_pending=true`。
3. 后台 index rebuild 完成后清除 pending 状态。

### R4. 后台任务与权限系统冲突

风险：后台 Agent 工具权限和前台 Agent 混用。

处理：

1. 后台任务使用独立 RuntimeLane。
2. 管理型 Agent 只能访问管理白名单工具。
3. durable write、state write、index rebuild 必须走显式能力授权。

### R5. 同步异步语义被各子系统重复实现

风险：记忆系统、计划系统、TaskGraph、健康系统各自维护队列和 pending 状态，导致状态不一致、难以取消、难以监控。

处理：

1. `OrchestrationExecutionScheduler` 是唯一调度入口。
2. `BackgroundTask` store 是后台任务生命周期权威源。
3. 子系统只能提供 worker handler，不直接拥有前台/后台判断权。
4. 旧的 `_pending`、`_in_progress` 只能作为本地执行优化，不能作为权威状态。

## 10. 实施顺序

建议一次性按以下顺序实施，不拆成长期悬空半成品：

1. 新增 `OrchestrationExecutionScheduler` 和 `ExecutionDispatchDecision`。
2. 新增后台任务模型与最小 store。
3. 把 TaskGraph `execution_mode` / `wait_policy` / `join_policy` 接入调度决策。
4. 把 memory maintenance enqueue 接入主请求。
5. 修改 `done` 事件，允许 memory maintenance queued。
6. durable memory 保存回调改为 enqueue index rebuild。
7. worker 执行 memory maintenance 和 index rebuild。
8. working memory finalization 改为 fail-soft / queued。
9. 计划管理 Agent 改走后台 `plan_management` 任务。
10. 增加回归测试。
11. 重跑六十轮真实对话。
12. 根据 trace 清理旧同步路径和无用兼容。

---

## 11. 完成标准

本修复完成后，应满足：

1. 主请求 `done` 不再等待管理型 Agent。
2. TaskGraph `sync` / `async` / `background` / `parallel` / `barrier` / `manual_gate` 有统一调度决策。
3. 记忆管理 Agent、计划管理 Agent 的活动都有后台任务状态。
4. durable memory 保存不会同步触发检索索引重建。
5. 六十轮真实对话不再因为 durable memory turn 卡住。
6. 后台任务失败有 receipt、event 和可诊断状态。
7. 后台任务不会绕过 RuntimeLane 权限。
8. 用户显式“记住”的偏好仍能被后续会话召回。
9. 代码中不再用“async 函数”冒充“后台任务”。
10. 子系统不再各自实现一套前台/后台判断。

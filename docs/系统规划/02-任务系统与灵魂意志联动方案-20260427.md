# 洪荒时代任务系统与灵魂意志联动方案

日期：2026-04-27  
项目名：洪荒时代（mythical ageent）  
定位：把任务系统升级为 ControlKernel 的第一层语义底座，并与灵魂系统形成相辅相成的任务意志选择机制。

---

## 0. 结论

任务系统和灵魂系统必须一起设计。

原因很简单：

```text
任务决定需要什么意志。
灵魂决定如何承载任务。
AgentProfile 决定以什么角色参与任务。
Skill / Tool / Worker 决定任务怎么被执行。
```

如果没有任务系统，灵魂只会变成“风格切换”。

如果没有灵魂系统，任务只会变成“机械分类”。

所以新架构的第一条主链应该是：

```text
UserRequest
  -> TaskContract
  -> TaskMode
  -> TaskSoulRequirement
  -> SoulSelectionCandidate
  -> AgentProfileCandidate
  -> Skill / Tool / Worker Policy
  -> ControlKernel
```

一句话原则：

> 任务是问题的形状，灵魂是承载任务的意志。

## 2. 任务系统的新职责

任务系统不是执行层。

任务系统负责：

```text
定义任务
记录任务
表达任务上下文
绑定任务句柄
承载任务模式
记录灵魂/agent/skill 选择结果
保存执行产物引用
```

任务系统不负责：

```text
不直接选择工具
不直接执行 worker
不直接写最终答案
不直接写长期记忆
不绕过 ControlKernel 选择灵魂
不绕过 CommitGate 写回状态
```

---

## 3. 新任务分层

```text
TaskContract
  当前轮任务的标准输入契约

TaskMode
  任务模式，决定任务需要哪类意志/agent/skill

TaskSoulRequirement
  任务对灵魂意志的需求

TaskAgentRequirement
  任务对 AgentProfile 的需求

TaskCapabilityRequirement
  任务对 Skill / Tool / Worker 的需求

TaskExecutionIntent
  ControlKernel 采纳后的执行意图，不等于执行

TaskRecord
  任务记录与结果回指
```

## 4. TaskContract

建议放在：

```text
backend/orchestration/contracts.py
```

或后续拆到：

```text
backend/tasks/contracts.py
```

核心模型：

```python
@dataclass(slots=True, frozen=True)
class TaskContract:
    task_id: str
    session_id: str
    user_goal: str
    task_mode: str = "general_dialogue"
    parent_task_id: str = ""
    source: str = "user_request"
    bindings: dict[str, Any] = field(default_factory=dict)
    constraints: dict[str, Any] = field(default_factory=dict)
    requested_outputs: list[str] = field(default_factory=list)
    refs: dict[str, Any] = field(default_factory=dict)
    canonical_owner: str = "orchestration.control_kernel"
```

强约束：

```text
TaskContract 是当前轮任务真相。
TaskRecord 是持久化记录。
TaskContextRef 是任务上下文句柄。
```

不要再让 `QueryPlan / QueryExecutionPlan / FollowupResolution` 成为任务真相。

## 5. TaskMode

TaskMode 是任务系统和灵魂系统的接口。

任务分类先分两层：

```text
TaskFamily  顶层任务族，决定任务大类
TaskMode    具体任务模式，决定默认意志、默认 agent、能力需求
```

第一批顶层任务族：

| task_family | 中文名 | 说明 |
| --- | --- | --- |
| `qa` | 问答任务 | 不需要复杂外部执行，主要围绕当前上下文、已有信息和模型能力回答 |
| `rag` | RAG 知识搜寻任务 | 需要从本地知识库、文档索引、长期资料中检索证据 |
| `local_processing` | 本地处理任务 | 需要处理本地文件、PDF、表格、代码、测试产物或系统资源 |
| `multi_agent` | 多智能体协调任务 | 需要拆分角色、多个 AgentProfile 协作、生成中间 artifact 后汇总 |

顶层任务族只回答：

```text
这是什么大类任务？
是否需要检索？
是否需要本地处理？
是否需要多 agent 协作？
```

它不直接回答：

```text
具体用哪个工具？
具体调用哪个 worker？
具体由哪个 agent 执行？
最终是否写回记忆？
```

建议第一批 TaskMode：

| task_family | task_mode | 说明 | 推荐 will_type | soul 选择 | 推荐 agent |
| --- | --- | --- | --- | --- | --- |
| `qa` | `general_qa` | 普通问答、解释、判断 | `dialogue` | 当前激活灵魂 | `general_agent` |
| `qa` | `context_qa` | 基于当前会话上下文回答 | `collect` | 河伯 | `general_agent` |
| `qa` | `reasoning_qa` | 需要前提辨析、推理判断的问答 | `inspect` | 玄女 | `critic_agent` |
| `qa` | `final_answer` | 终稿、结论收束 | `govern` | `final_owner` 或当前激活灵魂 | `main_agent` |
| `rag` | `knowledge_lookup` | 本地知识库检索 | `collect` | 河伯 | `knowledge_agent` |
| `rag` | `evidence_search` | 多来源证据搜寻、证据整理 | `collect` | 河伯 | `evidence_agent` |
| `rag` | `knowledge_synthesis` | 检索后综合回答 | `organize` | 四岳 | `synthesis_agent` |
| `local_processing` | `document_analysis` | PDF/文档分析 | `organize` | 四岳 | `document_agent` |
| `local_processing` | `data_analysis` | 表格/结构化数据分析 | `inspect` | 玄女 | `data_agent` |
| `local_processing` | `code_or_file_processing` | 本地代码、文件、目录处理 | `execute` | 祝融 | `executor_agent` |
| `local_processing` | `test_failure_diagnosis` | 测试失败归因 | `inspect` | 玄女 | `test_diagnosis_agent` |
| `local_processing` | `test_report` | 测试报告整理 | `organize` | 四岳 | `test_report_agent` |
| `multi_agent` | `system_design` | 架构规划、系统设计 | `organize` | 四岳 | `architect_agent` |
| `multi_agent` | `implementation` | 实现、修复、推进 | `execute` | 祝融 | `executor_agent` |
| `multi_agent` | `risk_review` | 风险审查、前提检查 | `inspect` | 玄女 | `critic_agent` |
| `multi_agent` | `writing_outline` | 写作大纲、结构规划 | `organize` | 四岳 | `writing_planner_agent` |
| `multi_agent` | `writing_draft` | 初稿生成、段落推进 | `execute` | 祝融 | `draft_agent` |
| `multi_agent` | `writing_review` | 审稿、逻辑检查 | `inspect` | 玄女 | `critic_agent` |
| `multi_agent` | `memory_curation` | 记忆整理、归档 | `collect` | 河伯 | `memory_curator_agent` |
| `multi_agent` | `memory_conflict_review` | 记忆冲突审查 | `inspect` | 玄女 | `memory_conflict_agent` |

注意：

```text
TaskMode 不是 route。
TaskMode 不是 tool_name。
TaskMode 不是 worker_route。
TaskFamily 也不是 route。
```

TaskMode 只描述任务形态和所需意志。

### 5.1 四类顶层任务族

#### 5.1.1 问答任务 qa

问答任务的特点：

```text
主要产出是回答。
不一定需要检索。
不一定需要本地文件处理。
一般不需要多 agent 协作。
```

典型模式：

```text
general_qa
context_qa
reasoning_qa
final_answer
```

默认职责分配：

```text
当前激活灵魂负责对话；最终统一口径由任务绑定的 final_owner 决定。
河伯适合收集上下文和信息。
玄女适合前提复杂或歧义明显的问答。
```

#### 5.1.2 RAG 知识搜寻任务 rag

RAG 任务的特点：

```text
必须先找证据。
证据、来源、句柄比直接答案更重要。
检索结果不能自动变成最终结论。
```

典型模式：

```text
knowledge_lookup
evidence_search
knowledge_synthesis
```

默认职责分配：

```text
河伯负责信息收集和资料召回。
玄女负责证据缺口和冲突检查。
四岳负责多证据综合结构。
任务绑定的 final_owner 负责最终答复归口。
```

#### 5.1.3 本地处理任务 local_processing

本地处理任务的特点：

```text
输入通常是文件、PDF、表格、代码、测试产物或本地资源。
需要受控工具、worker 或 artifact 处理。
必须有权限边界和结果句柄。
```

典型模式：

```text
document_analysis
data_analysis
code_or_file_processing
test_failure_diagnosis
test_report
```

默认职责分配：

```text
祝融负责推进处理。
玄女负责错误归因和风险检查。
四岳负责报告结构。
任务绑定的 final_owner 负责结果归口。
```

#### 5.1.4 多智能体协调任务 multi_agent

多智能体协调任务的特点：

```text
单个 agent 不适合直接完成。
需要拆分角色和中间 artifact。
通常包含规划、执行、审查、汇总。
```

典型模式：

```text
system_design
implementation
risk_review
writing_outline
writing_draft
writing_review
memory_curation
memory_conflict_review
```

默认协作形态：

```text
河伯：信息收集 / 上下文召回 / 资料整理
四岳：组织 / 规划 / 结构
祝融：行动 / 执行 / 推进
玄女：审查 / 归因 / 风险
句芒：对话 / 分派 / 统摄 / 最终归口
```

### 5.2 动态职责，而不是固定岗位

五个灵魂都应该是动态的。

这里的分工不是说：

```text
河伯只能收集。
四岳只能组织。
祝融只能行动。
玄女只能审查。
当前激活灵魂只能决定对话风格，不能越过任务契约。
```

而是说：

```text
在默认任务分配中，某个灵魂更适合承担某类职责。
```

所以更准确的表达是：

```text
SoulProfile 是完整灵魂。
SoulRole 是当前任务中的临时职责。
TaskSoulBinding 记录某个灵魂本轮承担什么职责。
```

同一个灵魂可以承担不同职责：

```text
四岳也可以对话，只是更适合组织。
祝融也可以审查，只是默认更适合行动。
玄女也可以组织，只是默认更适合审查。
河伯也可以最终答复；是否归口由任务绑定的 final_owner 决定。
```

系统应该支持：

```text
explicit_user_selection  用户显式指定某个灵魂
explicit_projection      用户显式选用某个投影卡片
task_default             任务模式推荐，不是强制默认
collaboration_role       协作模式临时分配
fallback_role            当前激活灵魂兜底归口
```

投影卡片进入任务系统时，选择优先级为：

```text
用户显式选用 projection_id
  > 当前任务已有 TaskSoulBinding
  > TaskMode 推荐 role_type/soul 候选
  > 当前激活灵魂兜底投影
```

### 5.3 是否需要默认主灵魂

不需要。

五个默认灵魂都是可复用职能意志：

```text
句芒：引导 / 统筹
河伯：收集
四岳：组织
祝融：行动
玄女：审查
```

多 agent 系统不设置默认主灵魂，只设置当前激活灵魂和任务级 final_owner。

句芒是默认五灵魂之一，不是默认主灵魂。

句芒也是动态灵魂，倾向承担“对话与统筹”。

它不是比其他灵魂更有权限，也不是固定压过其他灵魂。

当前激活灵魂是当前轮对话的承载意志。若任务需要多智能体协作，ControlKernel 可以绑定一个 `final_owner` 负责归口。

当前激活灵魂或 `final_owner` 负责：

```text
识别用户真实目标
把任务交给合适的职能灵魂承载
维持当前轮主线
统一多个 agent 的输出口径
决定哪些结果进入最终答复候选
在 ControlKernel fail-closed 时给出用户可理解的边界说明
```

当前激活灵魂不负责：

```text
不替代 ControlKernel 做授权
不直接扩大工具权限
不直接写 memory
不直接执行 worker
不把自己的判断绕过玄女审查
不把中间 artifact 直接当最终答案
```

五个默认灵魂之间的关系：

```text
任意当前激活灵魂
  -> 可以按任务绑定河伯/四岳/祝融/玄女/句芒等临时职责
  -> 可以由 ControlKernel 指定 final_owner 汇总各职责产物
  -> 形成最终答复候选
```

在代码上，当前激活灵魂应该表现为：

```text
ActiveSoulProfile
  - current_turn_owner
  - selected_final_answer_owner
  - collaboration_coordinator
  - no_tool_authority
  - no_memory_write_authority
```

它仍然不能绕过 ControlKernel。

### 5.4 默认灵魂与自制灵魂

默认五灵魂：

```text
句芒、河伯、四岳、祝融、玄女
```

这五个是系统内置基座，但不是封闭集合。

任务系统要预留自制灵魂入口：

```text
用户可以定义新的 SoulProfile。
新灵魂可以参与单线程对话。
新灵魂可以在多 agent 协作中承担临时 SoulRole。
新灵魂不能天然获得工具、记忆写入或调度权限。
```

自制灵魂参与任务时，同样走：

```text
TaskSoulRequirement
  -> SoulSelectionCandidate
  -> ControlKernel 采纳
  -> TaskSoulBinding
```

示例：

```text
用户自制“文曲”
  -> 可被选择为 writing_outline 的 organize 角色
  -> 也可被选择为 general_qa 的 dialogue 角色
  -> 但工具权限仍由 AgentProfile / ResourcePolicy 决定
```

## 6. 任务与灵魂的相辅相成关系

### 6.1 任务选择灵魂

任务提供目标、约束、上下文。

ControlKernel 根据任务生成：

```text
TaskSoulRequirement
```

例如：

```text
用户说：帮我把这个架构重新规划清楚。
TaskMode: system_design
TaskSoulRequirement: organize
SoulSelectionCandidate: siyue
```

### 6.2 灵魂承载任务

灵魂决定任务被承载时的工作方式。

例如：

```text
system_design + 四岳
  -> 先整体、再边界、再阶段、再动作

implementation + 祝融
  -> 先卡点、再突破口、再执行验证

risk_review + 玄女
  -> 先前提、再歧义、再风险、再结论

final_answer + 河伯
  -> 先收束、再判断、再必要依据
```

### 6.3 灵魂可以承载不同 skill

```text
四岳 + writing skill        -> 写作结构
四岳 + architecture skill   -> 架构规划
四岳 + test report skill    -> 测试报告结构

祝融 + coding skill         -> 代码实现
祝融 + writing skill        -> 快速初稿
祝融 + refactor skill       -> 重构推进

玄女 + review skill         -> 风险审查
玄女 + test diagnosis skill -> 测试失败归因
玄女 + memory review skill  -> 记忆冲突审查

河伯 + final answer skill   -> 最终答复
河伯 + memory curation      -> 稳定归档
河伯 + dialogue skill       -> 通用对话
```

所以三者关系是：

```text
TaskMode x SoulWill x SkillContract
```

而不是：

```text
Soul = Skill
Task = Tool
Agent = Worker
```

## 11. TaskCapabilityRequirement

任务最后才进入能力需求。

建议模型：

```python
@dataclass(slots=True, frozen=True)
class TaskCapabilityRequirement:
    task_id: str
    task_mode: str
    required_skills: list[str]
    allowed_tools: list[str]
    allowed_workers: list[str]
    required_artifacts: list[str]
    denied_tools: list[str]
    reason: str
```

注意：

```text
Soul 不参与 allowed_tools。
AgentProfile 可以限制 allowed_tools。
ControlKernel 根据 TaskCapabilityRequirement 生成 ResourcePolicy。
```

## 12. 任务模式识别

第一阶段不要做复杂模型分类。

先做规则 + 显式标记 + 可人工覆盖。

分类顺序：

```text
1. 识别 TaskFamily
2. 识别 TaskMode
3. 生成 TaskSoulRequirement
4. 生成 SoulSelectionCandidate
5. 生成 AgentProfileCandidate
6. 生成 Skill / Tool / Worker Policy 候选
```

第一步先分四类：

```text
qa
rag
local_processing
multi_agent
```

```text
显式短语：
  "用四岳" -> explicit_soul_id=siyue
  "用祝融" -> explicit_soul_id=zhurong
  "用玄女" -> explicit_soul_id=xuannv
  "用河伯" -> explicit_soul_id=hebo

任务词：
  什么是 / 为什么 / 怎么理解 / 判断一下 -> qa
  知识库 / 查资料 / 本地资料 / 检索 / RAG -> rag
  PDF / 表格 / 文件 / 代码 / 测试 / 本地处理 -> local_processing
  架构 / 重构 / 多步骤 / 协作 / 写作流程 / 规划执行审查 -> multi_agent
```

第二步再定具体 TaskMode：

```text
qa:
  普通解释 -> general_qa
  基于上下文 -> context_qa
  前提复杂 / 歧义明显 -> reasoning_qa
  要最终结论 / 总结 -> final_answer

rag:
  查知识库 -> knowledge_lookup
  找证据 / 多来源 -> evidence_search
  检索后综合 -> knowledge_synthesis

local_processing:
  PDF / 文档 -> document_analysis
  表格 / 数据 -> data_analysis
  代码 / 文件 -> code_or_file_processing
  测试失败 -> test_failure_diagnosis
  测试总结 -> test_report

multi_agent:
  架构 / 框架 -> system_design
  实现 / 修复 / 落地 -> implementation
  审查 / 风险 -> risk_review
  写作大纲 -> writing_outline
  写初稿 -> writing_draft
  审稿 -> writing_review
  记忆整理 -> memory_curation
  记忆冲突 -> memory_conflict_review
```

后续再升级为：

```text
TaskModeClassifier
  -> produces TaskFamilyCandidate
  -> produces TaskModeCandidate
  -> ControlKernel validates
```

### 12.1 TaskFamilyCandidate

建议模型：

```python
@dataclass(slots=True, frozen=True)
class TaskFamilyCandidate:
    candidate_id: str
    task_id: str
    task_family: str
    source: str
    reason: str
    confidence: float
    authority: str = "candidate_only"
```

候选来源：

```text
explicit_user_request
keyword_rule
context_binding
agent_profile_hint
model_classifier
```

`TaskFamilyCandidate` 仍然不能直接决定执行。

### 12.2 TaskModeCandidate

建议模型：

```python
@dataclass(slots=True, frozen=True)
class TaskModeCandidate:
    candidate_id: str
    task_id: str
    task_family: str
    task_mode: str
    source: str
    reason: str
    confidence: float
    authority: str = "candidate_only"
```

采纳规则：

```text
ControlKernel 先采纳 task_family，再采纳 task_mode。
如果 task_family 不确定，则 fail-closed 或进入 clarification。
如果 task_mode 不确定，但 task_family 明确，可走该 family 的默认模式。
```

## 13. 与协作模式的关系

协作模式是多个 TaskMode 的组合。

例如写作：

```text
writing_task
  -> writing_outline     四岳
  -> writing_draft       祝融
  -> writing_review      玄女
  -> final_answer        河伯
```

例如重构：

```text
refactor_task
  -> system_design       四岳
  -> implementation      祝融
  -> risk_review         玄女
  -> final_answer        河伯
```

例如测试报告：

```text
test_task
  -> test_failure_diagnosis 玄女
  -> test_report            四岳
  -> final_answer           河伯
```

协作模式不直接执行。

它只生成：

```text
TaskGraph
ExecutionGraph
Artifact expectations
```

## 14. 与 Skill 系统的关系

Skill 是能力包，不是任务本身。

同一个 TaskMode 可以对应多个 Skill：

```text
system_design
  -> architecture_planning_skill
  -> codebase_exploration_skill

implementation
  -> coding_skill
  -> refactor_skill

test_report
  -> test_report_skill
  -> test_artifact_reader_skill
```

同一个 Skill 可以被多个灵魂承载：

```text
writing_skill
  -> 四岳：结构
  -> 祝融：初稿
  -> 玄女：审查
  -> 河伯：终稿
```

边界：

```text
TaskMode 决定需要什么任务形态。
SoulWill 决定如何承载。
SkillContract 决定可用能力。
ToolContract / WorkerContract 决定可执行资源。
```

## 15. 与 Memory 系统的关系

任务可以声明 memory scope。

灵魂不能直接声明 memory 写入权。

建议：

| task_mode | 默认 memory_scope |
| --- | --- |
| `general_dialogue` | `session_read` |
| `system_design` | `session_read` |
| `implementation` | `session_read` |
| `risk_review` | `session_read` |
| `memory_curation` | `durable_candidate` |
| `memory_conflict_review` | `durable_read` |
| `test_report` | `none` |
| `test_failure_diagnosis` | `none` |

写入规则：

```text
Task 只能提交 MemoryCommitCandidate。
Soul 只能影响表达和判断倾向。
CommitGate 决定是否真正写入。
```

## 19. 第一批实现建议

先实现最小闭环：

```text
1. TaskMode registry
   - 固定枚举。
   - 每个 TaskMode 有默认 will_type / soul / agent。

2. TaskSoul binding
   - 输入 TaskContract。
   - 输出 TaskSoulRequirement / SoulSelectionCandidate / TaskSoulBinding。

3. ControlKernel preview
   - 在 fail-closed 结果里返回 task_mode、soul_binding、agent_profile_id。
```

这三步完成后，灵魂系统和任务系统就真正绑在一起了。

## 20. 当前定稿原则

```text
任务是问题的形状。
灵魂是承载任务的意志。
AgentProfile 是任务中的角色身份。
Skill 是能力包。
Worker 是执行器。
ControlKernel 是唯一采纳者。
CommitGate 是唯一写回者。
```

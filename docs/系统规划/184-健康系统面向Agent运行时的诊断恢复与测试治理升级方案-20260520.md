# 184-健康系统面向Agent运行时的诊断恢复与测试治理升级方案

日期：2026-05-20

状态：正式方案草案

适用范围：`backend/health_system`、`health_system.maintenance.test_system`、`health_system.maintenance.experiments`、`health_system.maintenance.harness`、`RuntimeLoopTraceReader`、`RuntimeEventLog`、`TaskRunLoop`、`TaskGraph`、`autonomous_task_run`、六十轮真实用户长跑

---

## 1. 问题定义

我们要升级的不是一个普通“健康页面”，而是 Agent 系统的诊断、恢复、验证治理层。

旧健康系统已经有 issue、command、report、verification、harness、trace、test system，但它仍像多个历史模块拼接在一起：有用，但边界不够统一；能收集证据，但不善于筛选证据；能启动测试，但测试系统、实验 runner、trace 投影和恢复决策之间还没有形成统一闭环。

正确终态不是“堆更多证据”，而是：

| 目标 | 说明 |
|---|---|
| 统一事实源 | RuntimeLoop event、checkpoint、TaskRun、CoordinationRun、turn artifact、prompt manifest、memory trace 只作为事实源，不被健康系统复制语义 |
| 有用证据优先 | 健康系统输出最能解释问题、支持恢复、驱动复现的 evidence packet，而不是整包 raw trace |
| Agent 化诊断 | 健康 Agent 以只读诊断员、复现规划员、验证员身份工作，遵守 AgentRuntimeProfile 和 RuntimeLane |
| 可恢复 | 失败不只生成报告，还能给出安全恢复点、可重放边界、不可重复动作和人工确认项 |
| 新测试系统 | 测试从“跑脚本 + 看结果”升级为“场景契约 + 断言 + 证据包 + 回归样本 + 可复跑命令” |
| 不造旁路系统 | health/test/harness/trace/monitor 都归入健康治理层，但事实、执行、恢复仍走主 RuntimeLoop 底座 |

## 2. 技术源码报告

### 2.1 当前健康系统已经是一个子系统

`backend/health_system/README.md` 已经明确健康系统负责：

1. issue / report / command / receipt。
2. verification gate 与健康工作台。
3. 测试治理。
4. regression harness。
5. trace / replay / experiment 观测链路。

这说明健康系统不应再被当成边缘工具。它应该升级为 Agent 运行时的治理层。

### 2.2 当前主要模块

| 文件 | 当前职责 | 升级判断 |
|---|---|---|
| `backend/health_system/registry.py` | 健康系统主入口、Agent run、overview | 保留为治理 facade，但逐步剥离手工 RuntimeLoop 执行 |
| `backend/health_system/command_service.py` | 命令分发、issue/test/agent 操作 | 保留命令控制面，补 failure phase 和恢复命令 |
| `backend/health_system/store.py` | JSONL 存储 | 需要原子写、坏行诊断、store health |
| `backend/health_system/workbench.py` | 健康工作台 projection | 升级为 evidence inbox、diagnosis inbox、recovery inbox |
| `backend/health_system/verification_service.py` | verification run、gate projection | 去掉读操作落库副作用，改为显式 sync |
| `backend/health_system/maintenance/experiments/runner.py` | 长跑和实验进程管理 | 升级 run state、detached recovery、stale diagnostics |
| `backend/health_system/maintenance/test_system/service.py` | 测试语义、turn summary | 升级为场景契约测试服务 |
| `backend/health_system/maintenance/test_system/runtime_loop_probe.py` | RuntimeLoop artifact 摘要 | 扩展为标准 evidence packet 入口 |
| `backend/health_system/maintenance/harness/run.py` | harness profile 统一入口 | 升级为可观测、可恢复、可产出证据包的执行入口 |
| `backend/health_system/maintenance/harness/contracts.py` | run result / issue / trace dataclass | 补 harness run state、progress event、artifact manifest、evidence refs |
| `backend/health_system/maintenance/harness/persistence.py` | run_result/report/trace 原子持久化 | 补 run_state、heartbeat、manifest、partial result 落盘 |
| `backend/health_system/maintenance/harness/reporter.py` | markdown 报告 | 升级为摘要、失败链路、有用证据、复跑命令、恢复建议 |
| `backend/health_system/maintenance/harness/regression_gate.py` | curated regression profile | 对接 scenario contract 和 evidence packet，不只按文件跑 pytest |
| `backend/orchestration/monitor.py` | RuntimeLoop 标准摘要 | 继续作为健康系统消费的运行摘要源 |
| `backend/orchestration/runtime_loop/trace_reader.py` | TaskRun trace/live monitor | 继续作为 trace spine，不被健康系统替代 |

### 2.3 已知结构性风险

已有 `docs/风险报告/06-健康系统风险报告-20260509.md` 指出多项风险，其中和本次升级直接相关的是：

1. 健康 Agent 曾处于“半接入”状态，说明健康能力必须受正式 Agent runtime 权限约束。
2. `HealthRegistry.execute_agent_run()` 手工写 RuntimeLoop event/checkpoint，容易和主 `TaskRunLoop` 语义漂移。
3. `HealthStore` JSONL 坏行静默跳过，upsert 整文件重写，有证据丢失风险。
4. 命令失败边界粗，只能看到 failed，看不到 input、admission、runtime、storage、test 哪个阶段失败。
5. `ExperimentRunner` 依赖内存 `Popen` 和 PID 推断，服务重启后长跑恢复不稳。
6. `HealthVerificationService.list_verification_runs()` 查询时会写入 verification run，读操作有副作用。
7. Workbench 扫 checkpoint 吞异常，健康系统可能忽略自己应该发现的运行状态损坏。

这些不是单点 bug，而是健康系统作为治理层还不成熟。

## 4. 核心设计原则

### 4.1 健康系统只治理，不替代 RuntimeLoop

健康系统可以读取 `TaskRunLoop` 的事实，可以发起健康 Agent 诊断，可以创建恢复建议和验证任务，但不能私自改写运行语义。

不允许：

1. 手工模拟一套 RuntimeLoop。
2. 健康系统自己决定工具是否可重放。
3. 健康系统自己绕过 OperationGate 执行恢复动作。
4. 健康系统把 raw trace 当成最终诊断结果。

允许：

1. 生成 evidence packet。
2. 生成 diagnosis report。
3. 生成 recovery plan。
4. 发起正式 `TaskRunLoop` / harness / verification run。
5. 把真实失败沉淀为测试场景。

### 4.2 有用证据优先

证据系统分两层：

| 层 | 角色 |
|---|---|
| Evidence Fact Store | 保存完整事实引用，例如 trace、checkpoint、artifact 路径、event offset |
| Evidence Intelligence | 选出最有解释力、最适合 Agent 阅读、最能驱动恢复的证据包 |

健康 Agent 默认只能看到 evidence packet，不能直接吃完整 raw trace。完整 raw trace 仍可被引用和展开，但不是默认上下文。

### 4.3 证据必须能回答问题

每个 evidence packet 至少回答一个诊断问题：

1. 哪里开始偏离预期？
2. 哪个决策导致了后续失败？
3. 哪个工具、子 Agent、checkpoint 或 commit 是关键边界？
4. 是否存在可恢复点？
5. 哪些动作不能重复执行？
6. 最小复现输入是什么？
7. 需要沉淀成哪类测试？

不能回答这些问题的证据，默认不进入 Agent 上下文。

### 4.4 诊断和恢复分离

诊断负责解释，恢复负责执行。健康 Agent 可以建议恢复，但恢复动作必须经过正式 runtime control。

### 4.5 测试系统从结果导向升级为契约导向

旧测试系统主要按 profile 跑脚本，新系统要表达：

1. 场景输入。
2. 预期运行路径。
3. 必须出现的 runtime event。
4. 禁止出现的 fallback / timeout / silent truncation。
5. 语义断言。
6. 失败时要提取的 evidence packet。
7. 修复后需要回归的最小样本。

## 5. Evidence Intelligence 设计

### 5.1 Evidence Candidate

所有原始事实先成为 evidence candidate。

建议模型：

```json
{
  "candidate_id": "evcand:taskrun:xxx:offset:42",
  "source_kind": "runtime_event",
  "source_ref": "taskrun:xxx#42",
  "subject_type": "task_run",
  "subject_id": "taskrun:xxx",
  "event_type": "operation_gate_checked",
  "time_index": 42,
  "summary": "op.delegate_to_agent allowed=false",
  "raw_ref": "runtime-event-log:taskrun:xxx",
  "metadata": {}
}
```

来源包括：

1. `RuntimeEventLog`。
2. `RuntimeCheckpointStore`。
3. `RuntimeStateIndex`。
4. TaskGraph coordination checkpoint。
5. harness turn artifact。
6. prompt manifest。
7. memory trace。
8. test assertion result。
9. runner log tail。
10. API / command receipt。

### 5.2 Evidence Scoring

候选证据进入打分器。分数不是为了“智能炫技”，而是为了裁剪上下文。

| 维度 | 含义 | 示例 |
|---|---|---|
| causal_score | 是否解释失败因果链 | 第一次 `loop_error`、第一次 denied gate |
| temporal_score | 是否靠近失败点或关键转折点 | timeout 前最后一个 event |
| decision_score | 是否代表系统做过选择 | recipe selection、route decision、OperationGate |
| recovery_score | 是否能支持恢复 | checkpoint id、ledger current step、replay guard |
| reproduction_score | 是否能生成最小复现 | user turn、scenario id、tool args |
| semantic_score | 是否和用户报告的问题语义相关 | “RAG 没调用”“卡在 turn18” |
| novelty_score | 是否提供新信息 | 避免重复 50 条相同 model chunk |
| negative_score | 是否提供反证 | 没有 `agent_delegation_requested`，说明并未委派 |

### 5.3 Evidence Packet

健康系统默认输出 evidence packet，而不是 raw trace。

建议模型：

```json
{
  "packet_id": "evpkt:taskrun:xxx:timeout-root-cause",
  "question": "为什么六十轮长跑卡住？",
  "verdict": "suspected_runtime_wait_on_memory_maintenance",
  "confidence": 0.78,
  "summary": "turn18 后没有新的 runtime_loop_event，最后一次有效事件是 checkpoint_written...",
  "selected_evidence": [
    {
      "candidate_id": "evcand:...",
      "role": "causal",
      "why_selected": "这是进入卡住状态前最后一个决策事件。",
      "source_ref": "taskrun:xxx#42"
    }
  ],
  "excluded_evidence_summary": {
    "model_stream_chunks": 187,
    "duplicate_checkpoint_events": 12,
    "irrelevant_memory_reads": 9
  },
  "recovery_handles": [
    {
      "kind": "checkpoint",
      "ref": "checkpoint:xxx",
      "safe_to_resume": true,
      "side_effect_replay_risk": "low"
    }
  ],
  "test_handles": [
    {
      "kind": "scenario_turn",
      "ref": "sixty-turn-real-user-marathon/turn-18",
      "recommended_assertions": ["loop.progress_after=turn-18", "no.silent_timeout"]
    }
  ]
}
```

### 5.4 Useful Evidence Policy

默认 packet 限制：

| 类型 | 默认数量 |
|---|---:|
| causality events | 3-8 |
| decision events | 3-8 |
| recovery handles | 1-5 |
| negative evidence | 1-5 |
| raw excerpts | 默认不放，必要时放短摘要 |
| model chunks | 默认聚合，不逐条放 |
| repeated tool events | 合并计数，仅保留首次、最后一次和失败项 |

### 5.5 证据选择不是单向过滤

有些“没发生的事”很关键。例如：

| 用户问题 | 有用反证 |
|---|---|
| “它没有调用子 Agent 吗？” | event log 中没有 `agent_delegation_requested`，但有 `search_text` |
| “为什么说搜不到？” | 有 RAG route decision 但实际执行走了 `search_text` |
| “为什么卡住？” | turn artifact 没有后续 event，runner state 仍 running，PID 不存在 |
| “记忆不是异步了吗？” | runtime event 中仍出现同步等待或 checkpoint 前阻塞 |

所以 evidence intelligence 必须同时选正证和反证。

## 6. 目标架构

```text
RuntimeLoop / TaskGraph / Harness / API
  -> Evidence Candidate Extractors
  -> Evidence Index
  -> Evidence Scorer
  -> Evidence Packet Builder
  -> Health Diagnosis Agent
  -> Recovery Planner
  -> Verification/Test System
  -> Workbench Projection
```

### 6.1 Trace Spine

事实源仍是：

1. `RuntimeEventLog`。
2. `RuntimeCheckpointStore`。
3. `RuntimeStateIndex`。
4. TaskGraph coordination checkpoint。
5. harness artifacts。

健康系统只存引用、摘要、诊断结果和测试治理记录。

### 6.2 Health Agent Roles

健康 Agent 不应该是一个万能 Agent，而是一组受限角色：

| 角色 | Agent-facing prompt 方向 | 禁止事项 |
|---|---|---|
| Trace Diagnostician | 你是一名运行链路诊断员，只负责解释证据包中的失败链路、关键决策和反证 | 不修改代码，不扩大证据范围 |
| Recovery Planner | 你是一名恢复规划员，只负责判断可恢复点、不可重复动作和恢复前置条件 | 不直接执行恢复 |
| Test Curator | 你是一名回归样本策展员，只负责把真实失败转成最小复现场景和断言 | 不伪造期望结果 |
| Verification Agent | 你是一名对抗性验证员，只负责检查修复是否真实通过、是否有绕测 | 不替实现方解释失败 |

这些 prompt 必须写成 Agent 能理解的任务身份，不能写成“这是 runtime 节点”。

### 6.3 Recovery Plane

恢复动作分三类：

| 类型 | 执行方式 |
|---|---|
| read_only_diagnosis | 健康 Agent 只读执行 |
| resume_plan | 生成恢复计划，等待用户或系统批准 |
| controlled_resume | 走 RuntimeLoop control / checkpoint / replay guard 执行 |

健康系统不能直接重复工具副作用。所有外部动作必须经过 OperationGate 和 replay policy。

### 6.3.1 TaskGraph 恢复边界

任务图恢复不应由健康系统直接接管。健康系统负责治理，TaskGraph runtime 负责执行。

| 能力 | 归属 | 说明 |
|---|---|---|
| 诊断任务图失败原因 | 健康系统 | 读取 `CoordinationRun`、scheduler state、coordination checkpoint、node run、handoff、merge result，生成 evidence packet |
| 判断可恢复点 | 健康系统提出候选，TaskGraph runtime 校验 | 健康系统只能给出候选 checkpoint、node、edge、stage，不直接改状态 |
| 节点级重跑 / 断点续跑 | TaskGraph runtime | 必须通过 graph scheduler、coordination checkpoint、node execution request 执行 |
| 修改 scheduler state | TaskGraph runtime | 健康系统不能直接写 `task_graph_scheduler_state` |
| 人工审批恢复 | 健康系统 / Workbench | 健康系统展示恢复计划、风险、不可重复动作，收集确认 |
| 恢复后验证 | 健康系统 | 运行 targeted harness / scenario contract，比较恢复前后 evidence packet |

任务图恢复命令应该是 health control plane 发起的“治理命令”，但命令落地后必须进入 TaskGraph runtime 的正式恢复 API。也就是说：

```text
Health Recovery Plan
  -> approval / policy check
  -> TaskGraph recovery request
  -> TaskGraph scheduler validates checkpoint and node state
  -> RuntimeLoop / LangGraphCoordinationRuntime executes resume
  -> Health system observes and verifies result
```

不允许：

1. 健康系统直接改 `CoordinationRun.status`。
2. 健康系统直接写 coordination checkpoint。
3. 健康系统绕过 scheduler 重置 node status。
4. 健康系统直接重复执行 node task run。

允许：

1. 健康系统生成 `TaskGraphRecoveryCandidate`。
2. 健康系统生成恢复风险报告。
3. 健康系统触发正式恢复命令。
4. 健康系统验证恢复结果并沉淀回归样本。

### 6.4 New Test System

新测试系统命名上仍归属 `health_system.maintenance.test_system`，但语义升级为 `scenario contract test system`。

核心对象：

| 对象 | 说明 |
|---|---|
| TestScenarioContract | 场景输入、目标、运行路径、断言、证据策略 |
| TestRunPlan | profile、scenario set、资源预算、timeout、并发策略 |
| TestTurnRecord | 单轮输入输出、runtime trace refs、semantic assertions |
| TestEvidencePacket | 本轮失败或关键路径证据包 |
| RegressionSample | 从真实失败沉淀出的可复跑样本 |
| VerificationVerdict | 修复验证裁决 |

### 6.5 Harness System

当前 `backend/health_system/maintenance/harness` 已有入口、合同、持久化、报告和回归门禁，但它还更像“脚本转发器”，不是完整的健康执行测量层。

| 文件 | 当前能力 | 缺口 |
|---|---|---|
| `run.py` | 根据 profile 调用 regression gate、system eval runner、long runner | 缺少 harness 自己的 run contract、heartbeat、progress event |
| `contracts.py` | 定义 `RunResult`、`ScenarioResult`、`IssueEntry`、`TraceSpan` | 缺少 run state、artifact manifest、partial result、evidence packet refs |
| `persistence.py` | 原子写 `run_result.json`、`issues.json`、`trace.jsonl`、`report.md` | 缺少标准 `run_state.json`、heartbeat、stale marker、manifest checksum |
| `reporter.py` | 渲染 markdown 摘要和失败列表 | 不展示有用证据、失败链路、恢复句柄、复跑命令 |
| `regression_gate.py` | 按 profile 执行 curated pytest/python 文件 | 不理解 scenario contract，不会把失败转成 evidence packet / regression sample |

Harness 应该升级成健康系统的执行测量层。它负责稳定执行、记录进度、沉淀产物、提供可恢复状态，并把失败转成 evidence candidate。

目标对象：

| 对象 | 说明 |
|---|---|
| HarnessRunContract | 本次 harness 的 profile、scenario refs、预算、timeout、输出目录、资源限制 |
| HarnessRunState | running/passed/failed/cancelled/stale/detached，包含 pid、process token、heartbeat、last progress |
| HarnessProgressEvent | started、scenario_started、turn_completed、artifact_written、heartbeat、stale_detected、finished |
| HarnessArtifactManifest | 产物清单、checksum、required/present、producer、schema_version |
| HarnessPartialResult | 长跑中间结果，保证卡住时也能诊断 |
| HarnessEvidenceBridge | 从 run_result、trace、turn artifact、runner.log 提取 evidence candidate |

Harness 不负责：

1. 不判断业务根因。
2. 不绕过测试断言直接标 passed。
3. 不吞掉 runner log 或 partial result。
4. 不把 profile 成功等同于语义验证成功。

## 7. 固定执行流

### 7.1 运行问题诊断

```text
User reports issue
  -> HealthCommand(report_issue/analyze_trace)
  -> resolve subject refs
  -> extract evidence candidates
  -> score useful evidence
  -> build evidence packet
  -> Health Diagnosis Agent reads packet
  -> diagnosis report
  -> recovery/test recommendations
```

### 7.2 长跑卡住诊断

```text
Harness run state
  -> inspect run_state + runner.log + turn artifacts
  -> locate last progressing turn
  -> map turn to task_run_id
  -> read RuntimeLoop events/checkpoint
  -> select last decision / last observation / missing expected event
  -> produce packet: root cause candidates + recovery handles + repro scenario
```

### 7.3 修复后验证闭环

```text
Fix completed
  -> create verification plan
  -> run targeted regression sample
  -> run broader profile if required
  -> compare evidence packets before/after
  -> mark issue verified or still failing
```

### 7.4 从真实失败沉淀测试

```text
Failed turn / failed task run
  -> evidence packet
  -> Test Curator creates RegressionSample
  -> add scenario contract
  -> run offline verification
  -> future gate includes sample
```

### 7.5 Harness 可恢复执行

```text
HarnessRunContract
  -> HarnessRunState(running)
  -> progress event stream
  -> heartbeat
  -> partial artifacts
  -> final artifacts or stale diagnosis
  -> evidence packet
  -> verification verdict
```

卡住时：

```text
No heartbeat / no progress
  -> inspect process token and pid
  -> inspect last artifact mtime
  -> inspect last turn and task_run_id
  -> mark stale or detached
  -> build useful evidence packet
  -> recommend resume / rerun / cancel action
```

## 8. 实施计划

### 8.0 关键问题到设计落点映射

本节把当前旧系统的几个核心问题直接映射到目标设计，作为后续实施验收线。任何阶段如果没有覆盖这些问题，就说明方案跑偏。

| 已确认问题 | 设计落点 | 实施验收 |
|---|---|---|
| `backend/health_system/README.md` 已把 issue、report、command、verification、test、harness、trace 都收进健康系统，但内部仍像多个历史系统拼接 | 五层架构：Trace/Evidence、Health Diagnosis、Recovery、Verification/Test、Workbench | health/test/harness/trace 仍归健康治理层，但每层有明确输入输出，不再互相复制状态 |
| `backend/health_system/registry.py` 的健康 Agent 执行手工拼 RuntimeLoop event/checkpoint | 健康 Agent 运行统一走 `TaskRunLoop`，健康系统只准备 issue context、evidence packet 和执行策略 | 删除或封存 `HealthRegistry.execute_agent_run()` 中手写 event/checkpoint 的桥接逻辑，健康 Agent trace 与普通 TaskRun 同口径 |
| 任务图失败需要恢复，但 TaskGraph scheduler 才是执行权威 | 健康系统只做诊断、候选恢复点、审批、验证；恢复执行归 TaskGraph runtime | 健康系统不能直接写 `CoordinationRun`、coordination checkpoint 或 `task_graph_scheduler_state` |
| `backend/health_system/maintenance/experiments/runner.py` 依赖进程内 `Popen` 和 `run_state.json` 推断状态 | Runner 升级 heartbeat、process token、last progress marker、stale diagnosis | 服务重启后可以判断 running/stale/detached；卡住时能定位最后 turn、task_run、checkpoint 和 runner log 证据 |
| `backend/health_system/maintenance/harness/run.py` 只是 profile 转发入口 | HarnessRunContract + HarnessRunState + progress event stream | harness 能独立说明自己在跑什么、跑到哪里、产物在哪里、是否 stale |
| `backend/health_system/maintenance/harness/persistence.py` 只落最终结果 | partial result、artifact manifest、heartbeat、atomic state 持久化 | 中途卡住也能读取已完成 turn 和已写产物 |
| `backend/health_system/maintenance/harness/reporter.py` 只给总览和失败列表 | 报告展示失败链路、有用证据、复跑命令、恢复建议 | report 不再要求用户自己翻 raw artifact |
| `backend/health_system/verification_service.py` 查询接口会顺手落库 verification run | verification list 改纯只读，新增显式 sync 命令 | 读接口前后 store checksum 不变；同步动作必须通过 command/maintenance API 显式触发 |
| `backend/health_system/maintenance/test_system/runtime_loop_probe.py` 已接 RuntimeLoop 摘要，但只是摘要入口 | 升级为 Evidence Candidate / Evidence Packet 的标准入口 | turn artifact 可以直接产出有用证据包，包含关键事件、反证、恢复句柄、测试句柄 |
| 现有 harness 更像 profile runner | 新测试系统使用场景契约、断言、证据包、可复跑命令 | 六十轮失败 turn 可沉淀为 `RegressionSample`，并可单独复跑 |
| Workbench 当前更偏聚合展示 | Workbench 展示 inbox、失败链路、长任务状态、测试证据、恢复建议，但不做事实源 | 页面展示 packet 与 diagnosis，不直接要求用户读 raw trace |

### 8.0.1 五层目标形态锁定

| 层 | 负责什么 | 不负责什么 |
|---|---|---|
| Trace/Evidence 层 | 登记 `RuntimeEventLog`、checkpoint、turn artifact、prompt manifest、memory trace、test artifact 的事实引用；生成 evidence candidate | 不裁决问题根因，不直接恢复 |
| Health Diagnosis 层 | 健康 Agent 只读 evidence packet，生成问题定位、责任层、失败链路、恢复建议 | 不手工模拟 RuntimeLoop，不执行副作用 |
| Recovery 层 | 生成恢复计划、恢复前置条件、可恢复 checkpoint、不可重复动作清单 | 不绕过 runtime control、checkpoint、replay guard、OperationGate |
| Verification/Test 层 | 用场景契约、断言、证据包、可复跑命令表达测试；把真实失败沉淀为样本 | 不伪造测试结果，不以 response 非空替代语义验收 |
| Workbench 层 | 展示 inbox、失败链路、长任务状态、测试证据、恢复建议 | 不作为事实源，不隐式写 verification record |

### 8.0.2 外部成熟原则的落地方式

| 成熟原则 | 本系统怎么落地 | 明确不做什么 |
|---|---|---|
| OpenTelemetry 的 traces / metrics / logs 可关联 signals | trace 记录路径，metric 记录耗时和卡住信号，event/log 记录离散事实，packet 负责关联解释 | 不把所有 signal 混成一个大 JSON |
| LangSmith 的 trace + eval dataset + feedback 闭环 | 真实失败 turn 先转 evidence packet，再沉淀为 RegressionSample，最后进入 verification gate | 不把 LangSmith 产品形态照搬成本地 UI |
| LangGraph durable execution 的 checkpoint / thread / 幂等恢复边界 | 恢复计划必须引用 checkpoint、replay policy、side-effect risk 和 OperationGate | 不由健康系统直接重复工具或外部副作用 |

## 11. Cutover 与回滚

### Shadow 阶段

Evidence packet 与旧 summary 并行生成，不改变现有 API 语义。

### Cutover 阶段

Workbench 默认展示 packet，raw trace 作为展开项。

### Rollback 阶段

如果 packet builder 出现问题，API 可回退旧 summary，但 raw trace 事实源不受影响。

### 删除旧逻辑

当健康 Agent 全部走统一 RuntimeLoop 后，删除 `HealthRegistry.execute_agent_run()` 内手工构造 RuntimeLoop event/checkpoint 的桥接逻辑，不保留无用旧分支。

## 12. 结论

健康系统值得升级，而且应该围绕 Agent runtime 升级。

升级重点不是“收集更多”，而是“选择更有用”：健康系统要成为能从 trace、checkpoint、harness、turn artifact 中提取关键因果证据、反证、恢复句柄和回归样本的治理层。

这会让系统具备三种能力：

1. 问题发生时，能快速解释为什么。
2. 任务卡住时，能判断从哪里恢复以及什么不能重复。
3. 修复完成后，能把真实失败变成可持续回归的测试资产。

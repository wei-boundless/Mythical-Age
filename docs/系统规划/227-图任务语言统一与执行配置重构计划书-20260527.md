# 227-图任务语言统一与执行配置重构计划书

日期：2026-05-27

## 1. 结论

当前代码已经存在图编辑器、TaskGraphDefinition、layered graph normalizer、TaskGraphRuntimeSpec、execution package、GraphHarness / GraphLoop 等结构，但它们没有形成一套唯一的图任务语言。

本次重构目标不是增加另一套配置语言，而是把图任务语言统一为一条严格链路：

```text
EditorGraphDraft
-> CanonicalTaskGraph
-> GraphExecutionConfig
-> PublishedGraphTaskBinding
-> GraphHarnessRunConfig
-> GraphLoopState
-> NodeExecutionRequest
-> AgentHarness
```

其中：

```text
EditorGraphDraft 是编辑态。
CanonicalTaskGraph 是标准图。
GraphExecutionConfig 是标准图的运行态表达。
PublishedGraphTaskBinding 是用户可启动图任务与运行态表达的绑定。
GraphHarnessRunConfig 是 harness 可识别的启动合同。
GraphLoopState 是运行中动态状态。
NodeExecutionRequest 是给 agent 节点的文本任务执行合同。
```

禁止再出现：

```text
启动时重新解释编辑器图。
GraphLoop 重新编译图。
runtime 同时读取 raw graph 和 runtime spec 来做第二套判断。
execution package 作为临时 API dict 参与运行。
metadata 中藏执行语义。
配置和图任务被说成两个可启动对象。
```

产品语义必须统一为：

```text
用户启动图任务。
图任务内部绑定已发布 GraphExecutionConfig。
harness 执行 GraphExecutionConfig。
agent 只接收当前节点的 NodeExecutionRequest。
```


### 2.2 当前结构性问题

第一，`TaskGraphRuntimeSpec` 接近运行态标准配置，但它现在是按需编译结果，不是发布后的权威运行产物。

第二，`/orchestration/harness/task-graphs/{graph_id}/start` 当前按 `graph_id` 读取 graph，然后现场调用 `compile_task_graph_definition_runtime_spec`。这会导致启动时重新解释图。

第三，`GraphCoordinationEngine._bootstrap_state` 会优先读 `task_graph_runtime_spec_ref`，但读不到时继续从 graph 重新编译。这让 GraphLoop 拥有第二编译权。

第四，`execution package` 当前是 API 返回聚合包，包含 standard_view、runtime_spec、manifest、scheduler_state、node_runtime_assemblies 等，但不是持久化、版本化、可启动的正式运行配置。

第五，`metadata` 仍承载过多执行语义，例如 graph loop policy、timeline policy、artifact policy、phase definitions、business communication modes 等。这会让字段来源不唯一。

第六，前端 `taskGraphSaveMapper.ts` 在保存时做补字段、推断 participants、subtask_refs、communication modes、contract bindings，这些属于规范化或编译期权力，不应该散落在 UI 保存层。

## 3. 统一语言定义

### 3.1 语言只有一套

图任务语言只有一套，分为四种阶段性形态：

```text
编辑态：EditorGraphDraft
规范态：CanonicalTaskGraph
运行态：GraphExecutionConfig
启动态：GraphHarnessRunConfig
```

这些不是四种语言，而是一套语言在不同系统边界上的形态。字段必须可追踪、可校验、可版本化、可单向转换。

### 3.2 单向转换规则

允许：

```text
EditorGraphDraft -> CanonicalTaskGraph
CanonicalTaskGraph -> GraphExecutionConfig
GraphExecutionConfig -> GraphHarnessRunConfig
GraphExecutionConfig + GraphLoopState -> NodeExecutionRequest
```

禁止：

```text
GraphHarnessRunConfig -> CanonicalTaskGraph
GraphLoopState -> GraphExecutionConfig
GraphLoop -> compile CanonicalTaskGraph
Agent -> select graph/task environment
metadata -> override first-class fields
```

### 3.3 权威对象

| 对象 | 权威边界 | 可写方 | 可读方 | 是否可执行 |
| --- | --- | --- | --- | --- |
| EditorGraphDraft | 前端编辑器 | 前端编辑器 | 前端编辑器、保存 API | 否 |
| CanonicalTaskGraph | 任务系统 | TaskGraph compiler / repository | 编译器、预览、发布器 | 否 |
| GraphExecutionConfig | 任务系统发布器 | GraphExecutionConfigCompiler | GraphHarness / GraphLoop | 是 |
| PublishedGraphTaskBinding | 任务系统发布器 | 发布 API | 启动 API | 是，作为启动入口 |
| GraphHarnessRunConfig | Harness 启动边界 | 启动 API / GraphHarness | GraphHarness / GraphLoop | 是 |
| GraphLoopState | GraphLoop | GraphLoop | monitor / resume / recovery | 否 |
| NodeExecutionRequest | GraphLoop | GraphLoop | AgentHarness | 是，作为节点执行入口 |

```python
    task_prompt: str
    input_packet: NodeInputPacket
    output_contract: NodeOutputContract
    tool_policy: NodeToolPolicy
    permission_policy: NodePermissionPolicy
    memory_packet: NodeMemoryPacket
    artifact_packet: NodeArtifactPacket
    handoff_packet: NodeHandoffPacket
    completion_policy: NodeCompletionPolicy
    authority: str = "harness.node_execution_request"
```

约束：

```text
NodeExecutionRequest 必须是文本模型可理解的任务包。
NodeExecutionRequest 不允许出现“这是 runtime 节点，根据任务图执行...”这类开发说明式 prompt。
NodeExecutionRequest 不允许要求 agent 理解整张图。
NodeExecutionRequest 只描述当前 agent 的角色、职责、输入、输出、裁决标准、边界。
```

## 5. 编译与发布流程

### 5.1 保存流程

目标：

```text
EditorGraphDraft
-> POST /tasks/task-graphs/{graph_id}/draft
-> normalize_editor_graph_draft
-> CanonicalTaskGraph
-> storage/tasks/task_graphs.json
```

规则：

```text
前端只做结构化表单收集和轻量字段 trim。
前端不推断 participant_agent_ids。
前端不生成 subtask_refs。
前端不推断 communication_modes。
前端不补 runtime policy。
后端 normalize 层统一完成规范化。
```

### 5.2 预检流程

目标：

```text
CanonicalTaskGraph
-> validate_canonical_task_graph
-> compile_graph_execution_config(dry_run=True)
-> GraphExecutionConfigPreview
```

规则：

```text
预检可以编译 preview，但 preview 不可启动。
preview 必须标记 ephemeral=True。
preview 不写入 PublishedGraphTaskBinding。
preview 只用于 UI 诊断。
```

### 5.3 发布流程

目标：

```text
CanonicalTaskGraph
-> compile_graph_execution_config
-> persist GraphExecutionConfig
-> create PublishedGraphTaskBinding
-> graph publish_state = published
```

规则：

```text
发布是唯一允许持久化 GraphExecutionConfig 的入口。
发布后 config_id 不可变。
图草稿修改不会影响已发布配置。
重新发布生成新的 config_id / config_version。
```

### 5.4 启动流程

目标：

```text
POST /orchestration/harness/task-graphs/{graph_id}/start
-> resolve PublishedGraphTaskBinding by graph_id
-> load GraphExecutionConfig by config_id
-> build GraphHarnessRunConfig
-> GraphHarness.start_run
-> GraphLoop.initialize
```

规则：

```text
启动 API 不调用 compile_graph_execution_config。
启动 API 不读取 raw graph 来构造运行配置。
启动 API 可以读取 graph title 用于展示，但不得参与运行决策。
```

### 5.5 运行流程

目标：

```text
GraphLoopState + GraphExecutionConfig
-> select ready node
-> build NodeExecutionRequest
-> AgentHarness
-> NodeResultEnvelope
-> GraphLoopState advance
```

规则：

```text
GraphLoop 只读 GraphExecutionConfig。
GraphLoop 不访问 TaskGraphRepository。
GraphLoop 不调用 graph compiler。
GraphLoop 不解释 EditorGraphDraft。
GraphLoop 不允许从 metadata 推断执行语义。
```

## 10. 预期最终链路

```text
用户在图编辑器搭建图
-> 前端保存 EditorGraphDraft
-> 后端规范化为 CanonicalTaskGraph
-> 用户发布图任务
-> 后端编译 GraphExecutionConfig
-> 后端写入 PublishedGraphTaskBinding
-> 用户启动 graph_id
-> 启动 API 解析 binding.config_id
-> GraphHarness 接收 GraphHarnessRunConfig
-> GraphLoop 根据 GraphExecutionConfig 推进状态
-> 当前节点转译为 NodeExecutionRequest
-> AgentHarness 执行单 agent 节点
-> NodeResultEnvelope 回到 GraphLoop
-> GraphLoop 判断下一个节点或完成图任务
```

这一版链路中，图和配置不再互相抢控制权。图是编辑与发布来源，配置是运行表达，图任务是用户启动的绑定对象，harness 只识别已发布的统一执行配置。

# 精准 Token 与 Prompt Cache 统一重构计划书

日期：2026-05-28

## 1. 背景

健康系统当前的 token 统计仍是运行轨迹估算，不能作为账单、压缩和缓存策略的统一依据。上下文压缩需要在模型调用前知道真实预算，健康系统需要在调用后读取真实消耗，Prompt Cache 又会改变“输入 token 成本”和“压缩是否值得做”的判断。

因此 token 统计和 prompt cache 不能分开设计。目标是建立一套统一的 `PromptAccounting` 底座，让模型请求构造、token 预测、provider usage 入账、prompt cache 命中分析、上下文压缩和健康系统成本视图共用同一事实源。

## 2. 成熟架构标准

成熟 agent 运行时应满足：

1. 模型请求必须先进入规范化序列化层，再计算 token。
2. 运行前使用本地 tokenizer 得到预测值，运行后使用 provider usage 得到真值。
3. Token 账本必须区分 `predicted`、`provider_usage`、`trace_estimate`，禁止混用。
4. Prompt cache 必须基于稳定前缀字节、segment hash 和 cache boundary，而不是靠 UI 或健康系统猜测。
5. 压缩策略必须基于 segment 级 token map，不允许只看总 token。
6. 健康系统只能读取账本和诊断结果，不自己重新发明 token 计算口径。

## 3. 目标权责链

```text
RuntimeStartPacket
-> CanonicalPromptSerializer
-> PromptSegmentMap
-> TokenCounterRegistry
-> PromptCachePlanner
-> ModelRequest
-> ProviderUsageExtractor
-> TokenLedger
-> CompressionPlanner / HealthSystem / Monitor
```

### 权责说明

`CanonicalPromptSerializer`

- 负责把模型请求转成真实发送结构。
- 包括 system/developer prompt、工具 schema、history、memory、artifact summaries、tool calls、tool results。
- 输出稳定 JSON/message representation，保证同一输入得到同一字节序列。

`PromptSegmentMap`

- 把请求切成可解释段：
  - `system_static`
  - `system_session`
  - `tool_schema`
  - `task_contract`
  - `memory_context`
  - `conversation_recent`
  - `tool_observations`
  - `artifact_summaries`
  - `volatile_turn`

`TokenCounterRegistry`

- 按 provider/model 选择 tokenizer。
- 能本地精确计数的模型标记为 `local_exact`。
- 不能精确计数的模型标记为 `local_predicted`，后续必须用 provider usage 校准。

`PromptCachePlanner`

- 决定哪些 segment 可缓存、缓存 scope、TTL、boundary。
- 输出 cache key、prefix hash、segment hash、cache safety report。

`ProviderUsageExtractor`

- 从模型响应中抽取真实 usage。
- 必须保留：
  - prompt tokens
  - completion tokens
  - reasoning tokens
  - cached tokens
  - cache creation/input tokens
  - total tokens

`TokenLedger`

- 唯一账本。
- 支持按 task_run、session、provider、model、time bucket、cache key 查询。

`CompressionPlanner`

- 只读取 `PromptSegmentMap + TokenLedger`。
- 基于 segment 优先级做压缩、裁剪和缓存保护。

## 5. Prompt Cache 策略

### Segment 分层

```text
Global stable:
- 核心 system prompt
- 静态工具使用规则
- 稳定格式协议

Session stable:
- 项目根信息
- 用户选择的模型策略
- 当前 workspace 静态配置
- 固定工具 schema hash

Task stable:
- TaskRun contract
- task goal / completion criteria
- recovery policy

Volatile:
- 当前用户输入
- 最新 tool observation
- 实时错误
- 未稳定的运行状态
```

### Cache boundary 原则

1. boundary 只能放在稳定 prefix 末尾。
2. 工具 schema 必须排序稳定，字段顺序稳定。
3. 动态时间、随机 id、路径扫描结果不能进入 cacheable prefix。
4. `/compact` 不应该破坏仍有效的 cache prefix；优先压缩 boundary 之后的段。
5. 如果必须修改 cacheable prefix，必须生成 `cache_invalidation_reason`。

## 6. 压缩算法

### 预算公式

```text
context_window
- reserved_output_tokens
- hard_required_tokens(system_static + tool_schema + latest_user + task_contract)
= compressible_budget
```

### 压缩优先级

1. 保留 `system_static`、`tool_schema`、`task_contract`。
2. 保留最近用户意图和恢复点。
3. `tool_observations` 先转结构化摘要，再保留 artifact ref。
4. `memory_context` 按相关性和新鲜度裁剪。
5. `conversation_old` 生成 summary segment。
6. 对 cacheable prefix，优先不改；如果缓存已过期，允许重排并重新建 cache。

### 输出

```json
{
  "decision": "no_compaction | microcompact | full_compact | fail_closed",
  "before_tokens": 0,
  "after_tokens": 0,
  "preserved_segments": [],
  "summarized_segments": [],
  "dropped_segments": [],
  "cache_impact": "preserved | invalidated | rebuilt"
}
```

## 7. 健康系统展示口径

健康系统必须显示三类数：

- `精确消耗`：provider usage。
- `预测消耗`：本地 tokenizer 对真实请求结构的预测。
- `缓存节省`：cached tokens / cache read tokens 推算出的节省。

不再展示混合的“估算 token”作为主指标。若没有 provider usage，只显示“预测值”，并明确标记。

## 9. 实施阶段

### 阶段一：账本与模型 usage 入库

- 新建 `PromptAccountingLedger`。
- 在模型调用返回后写入 provider usage。
- 健康系统优先读取 provider usage。
- 没有 usage 的旧记录标记为 `trace_estimate`，不再称为精确值。

验证：

```text
python -m pytest backend/tests/health_governance_token_usage_test.py
```

### 阶段二：请求序列化与本地预测

- 建立 `CanonicalPromptSerializer`。
- 对 model request 做 segment map。
- 本地 tokenizer 输出 `local_prediction`。
- 对比 provider usage，记录误差率。

### 阶段三：Prompt Cache Planner

- 对每次请求生成 cache plan。
- 记录 cache key、prefix hash、boundary segment。
- 抽取 provider cached token 字段，写入 cache record。

### 阶段四：压缩系统接入

- 压缩前读取 segment token map。
- 按 cache boundary 保护策略做 microcompact/full compact。
- 输出 compression decision record。

### 阶段五：健康系统与监控展示

- 健康系统成本页展示：
  - 精确消耗
  - 预测误差
  - 缓存命中率
  - 缓存节省
  - 压缩前后 token
- 任务详情展示 segment token 构成。

## 10. 风险与约束

1. Provider usage 字段不同，必须做 provider adapter，不能写死 OpenAI/Anthropic 单一结构。
2. Tool schema 序列化必须稳定，否则 prompt cache 会频繁 miss。
3. 压缩不能改写 TaskRun 合同和恢复点。
4. 健康系统只能分析和建议，不应直接改写运行时请求。
5. 旧 trace 只能作为历史估算，不能回填成精确账单。

## 11. 完成标准

- 每次模型调用都有 `ModelTokenUsageRecord`。
- 每次请求都有 `PromptSegmentMap`。
- 压缩决策使用 segment token，而不是全文估算。
- 健康系统不再自行计算 token。
- Prompt cache 命中率和缓存节省能按 session/task/model 查询。
- 旧数据明确标记为估算，不污染精确统计。

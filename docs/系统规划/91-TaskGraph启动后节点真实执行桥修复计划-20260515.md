# 91-TaskGraph启动后节点真实执行桥修复计划-20260515

## 问题结论

当前 `POST /api/orchestration/runtime-loop/task-graphs/{graph_id}/start` 只完成 TaskGraph 协调运行的创建和首个 `StageExecutionRequest` 的生成。

实测显示：

- `CoordinationRun.status = running`
- `world_design` 被调度器标记为 running
- 但只有 coordinator 占位 `AgentRun(agent:0, spawn_mode=task_graph_runtime)`
- 没有 `agent:world_designer` 的节点执行 AgentRun
- 没有 `AgentRunResult`
- 没有 handoff/result-ready 推进事件

所以“图启动”与“节点真实执行”之间存在结构断点。

## 修复原则

1. 不把普通会话当作协调任务执行。
2. 不伪造节点产物或测试结果。
3. 复用已有 `StageExecutionRequest -> continuation_payload -> run_single_agent_stream` 执行链。
4. start API 默认应真实启动首个 ready/running 节点；需要只预览时必须显式关闭执行。
5. 执行后必须可在 trace 中看到目标节点 agent run、结果和 TaskGraph 推进事件。

## 实施步骤

1. 给 TaskGraph start request 增加 `execute_initial_stage` 参数，默认 `true`。
2. 在 start API 中读取 `stage_execution_request` 后调用 `LangGraphCoordinationRuntimeResult.continuation_payload()`。
3. 通过 `TaskRunLoop._continue_coordination_delivery_stream()` 投递到单 agent 执行链，使用目标节点的 `agent_id` 和 `projection_id`。
4. 收集 execution events，刷新 trace 后返回。
5. 增加/更新回归测试，验证 start 后不只有 coordinator agent run，还会出现目标节点 agent run 或明确执行错误。
6. 重新实测 `graph.writing_team.long_novel`，确认 `world_design` 真正进入模型执行并产出结果或真实错误。

## 验收标准

- 启动接口返回包含 `initial_stage_execution_events`。
- trace 中出现非 coordinator 的 stage agent run。
- 若模型执行失败，错误必须暴露在 trace/API 返回中，不能静默停在 running。
- 旧 TaskGraph 兼容链路搜索仍为空。

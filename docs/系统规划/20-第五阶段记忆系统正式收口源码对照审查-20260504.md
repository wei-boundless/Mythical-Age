## 二、当前已经符合设计书的部分

### 2.1 任务侧记忆需求开始进入正式对象层

当前 `TaskMemoryRequestProfile` 已经不再只是给 runtime 传一份 dict 配置。

本轮开始，记忆系统内部已经可以把它提升为：

- `MemoryRequest`

这说明任务系统提出的“读哪些层、关注哪些主题、是否允许长期记忆”已经开始进入正式记忆接口层。

### 2.2 记忆系统正式拥有了供给包对象

当前记忆系统不再只输出：

- `MemoryRuntimeView`
- `ContextPolicyResult`

而是可以进一步输出：

- `MemoryBundle`

这意味着记忆系统已经不只是“若干候选和上下文片段的集合”，而是开始具备：

`正式供给给编排系统与 runtime 的记忆包`

### 2.3 写回方向继续保持 candidate-only

当前新增的：

- `MemoryWritebackProposal`

仍然保持：

- `adopted = false`
- `candidate_only`
- 不具备自动提交权

这与设计书中“写回由治理层决定，不由运行时自动决定”的原则一致。

### 2.4 编排系统已经开始优先消费正式记忆供给链

`AgentRuntimeChainAssembler` 现在已经会优先尝试通过：

- `memory_facade.build_memory_bundle(...)`

来拿 `runtime_view`

而不是只依赖旧的散装：

- `build_memory_runtime_view`

这说明主链已经开始切向正式记忆供给对象。

## 三、当前仍然存在的结构问题

### 3.1 `structured_memory` 仍承担过多可见底层类型

当前 `memory_system` 内部仍直接依赖：

- `structured_memory.session_memory.SessionMemoryManager`
- `structured_memory.process_state.ProcessState`
- `structured_memory.models.MemoryNote`
- `structured_memory.models.Message`

这说明虽然 `memory_system` 已经是正式入口，但很多底层类型和存储适配器仍然是“直接拿旧实现类型来用”。

这会带来两个问题：

1. 其他系统仍容易绕过 `memory_system` 直接碰 `structured_memory`
2. `memory_system` 的正式 public boundary 还不够厚

### 3.2 API 层仍直接碰部分 `structured_memory` 细节

当前 [backend/api/memory.py](/d:/AI应用/langchain-agent/backend/api/memory.py:1) 还直接引用：

- `structured_memory.frontmatter`
- `structured_memory.models.MemoryNote`

这说明 memory API 还没有完全只通过 `memory_system` 说话。

### 3.3 `MemoryFacade` 仍偏大

虽然它现在已经有正式请求和供给对象能力，但它仍同时承担：

- session memory refresh
- durable preview
- context package
- runtime view
- writeback proposal
- governance helper 调用

这说明第五阶段前半段已经立起对象层，但还没有完成“系统内部彻底分层”。

## 五、建议的下一步

如果继续推进第五阶段后半段，建议按下面顺序：

1. 把 `structured_memory` 中对外仍可见的关键类型，先通过 `memory_system` 做正式别名与适配出口
2. 让 `api/memory.py` 尽量改为只依赖 `memory_system` 暴露的类型和 helper
3. 把 `MemoryFacade` 再拆成 `request/bundle/writeback` 三类 service
4. 再补第五阶段最终的“源码对照审查收口结论”

## 六、收口结论

第五阶段前半段已经解决的是：

`记忆系统没有正式请求和供给对象`

当前还没完全解决的是：

`memory_system 虽然已经是正式入口，但对 structured_memory 的底层直接依赖仍然过多`

所以第五阶段后半段的重点不该是继续加新能力，而应继续做：

`public boundary 收紧、底层实现吸收、系统内部分层`

# 183-任务系统库管理保留与新图工作台收敛重构计划

日期：2026-05-20

## 1. 结论

任务系统前端目前确实混入了两套工作方式：

- 旧的 `TaskSystem Console`：以任务域、任务定义、任务图、契约、编排资源、运行管理为页面主线。
- 新的 `TaskGraph Studio`：以任务图为一等对象，进入图级、节点、边、时序、资源、契约、发布运行的深编辑。

保留库管理是必要的，但库管理不能再承担图编辑职责。目标不是删除任务域、任务定义库、契约库、Agent 库、运行库，而是把它们从“编辑主场”收敛为“资产主数据层”。真正的任务图配置、节点配置、边交接、契约绑定、时序分层、发布运行，都进入新图工作台。

最终结构：

```text
任务系统
├─ 库管理层
│  ├─ 任务域库：分类、入口策略、域级边界
│  ├─ 任务定义库：可复用任务说明、默认输入输出、可被节点引用
│  ├─ 任务图库：图对象目录、创建、复制、归档、进入工作台
│  ├─ 契约库：契约主数据、模板、版本、字段定义
│  ├─ 编排资源库：Agent、运行档案、Projection 的只读摘要与跳转
│  └─ 运行库：task_run、正式记忆、产物的查询与追踪
└─ 新图工作台
   ├─ 图对象编辑：图身份、接口、默认运行策略、上下文边界
   ├─ 拓扑编辑：节点、图节点、边、端口
   ├─ 对象编辑台：选中图/节点/边/图节点后按契约分区配置
   ├─ 时序分层：阶段、循环、并发组、父子图运行点
   ├─ 资源流：记忆库、产物库、账本、读写边
   ├─ 契约绑定：contract_bindings 的唯一主编辑模型
   └─ 发布运行：execution package、runtime spec、manifest、scheduler、trace index
```

## 2. 当前代码证据

### 2.1 前端双轨

`frontend/src/components/workspace/views/TaskSystemView.tsx`

- 存在 `TaskLayer = "management" | "editor"`。
- 存在 `TaskSystemLayer = "domains" | "tasks" | "graphs" | "contracts" | "orchestration" | "runtime"`。
- 同时维护管理态图选择 `selectedTaskGraphId` 和编辑态图选择 `editorTaskGraphId`。
- 同时持有 `taskGraphDraftV2`、`taskGraphStandardView`、`activeTaskGraphRuntimeSpec`。
- 管理层 `graphs/contracts/orchestration/runtime` 仍在展示或触碰图、契约、运行状态。

`frontend/src/components/workspace/views/task-system/TaskSystemShell.tsx`

- 直接切换 `TaskSystem Console` 与 `TaskGraph Studio`。
- 这本身可以保留，但语义应改成“库管理层 / 图工作台层”，避免形成两个平级系统。

`frontend/src/components/workspace/views/task-system/TaskGraphWorkbench.tsx`

- 已经承担新工作台入口。
- 内部已有 topology、blueprint、agents、responsibility、modules、timeline、memory、risk、contracts、publish 等层。
- 问题是旧管理层还保留了过多图配置职责，导致用户感知为两套系统。

### 2.2 后端已有新闭环，但前端没有围绕它组织

`backend/api/tasks.py`

- `/tasks/task-graphs/{graph_id}/standard-view`
- `/tasks/runtime-specs/task-graphs/{graph_id}`
- `/tasks/contract-manifests/task-graphs/{graph_id}`
- `/tasks/execution-packages/task-graphs/{graph_id}`

其中 `execution package` 已聚合：

- `standard_view`
- `contract_manifest`
- `runtime_spec`
- `scheduler_state`
- `node_runtime_assemblies`
- `graph_units`
- `graph_unit_execution_plans`
- `object_trace_index`
- `issues`

这说明后端已经开始形成“发布前真实执行包”。新工作台应该围绕这个包做预检和追踪，而不是让用户分别跨页面找 runtime spec、manifest、standard view。

### 2.3 契约模型仍有旧字段残留

后端和前端都仍保留：

- `graph_contract_id`
- `node_contract_id`
- `input_contract_id`
- `output_contract_id`
- `payload_contract_id`

同时也已有新模型：

- `contract_bindings.schema`
- `contract_bindings.execution`
- `contract_bindings.runtime`
- `contract_bindings.memory`
- `contract_bindings.artifact`
- `contract_bindings.handoff`
- `contract_bindings.acceptance`
- `contract_bindings.temporal`

`taskGraphSaveMapper.ts` 已经把旧字段归并进 `contract_bindings`，这条方向正确。但 UI 主路径仍不应该把旧字段当成用户认知对象展示。旧字段只能作为迁移兼容字段和导入兜底字段。

### 2.4 图节点运行能力已经开始后端化

`backend/tasks/coordination_graph_compiler.py`

- 已从 layered graph 生成 `nested_runtime_plans`。
- 已把 linked graph 编译为 `graph_unit` runtime node。
- 已有图节点执行计划 `_graph_unit_execution_plans()`，会编译子图的 standard view、runtime spec、contract manifest、scheduler state。

这说明“图是节点、节点可展开为图”的方向已经进入后端，但前端还没有把它做成工作台核心交互。

## 3. 目标职责边界

### 3.1 库管理层保留什么

库管理层保留，但只负责资产主数据，不负责深编辑运行图。

任务域库：

- 创建、改名、删除、启用任务域。
- 配置域描述、入口策略、默认 Workflow。
- 展示域内任务、任务图、契约、Projection 数量。
- 不编辑节点、不编辑边、不编辑运行包。

任务定义库：

- 管理可复用任务定义。
- 管理任务标题、说明、默认输入输出契约、默认 Workflow、默认 Projection 策略。
- 任务定义可以被图节点引用，但不在这里配置节点执行时序、记忆读写、边交接。

任务图库：

- 图对象目录。
- 创建新图、复制图、归档图、进入工作台。
- 展示发布状态、最近运行、主要问题摘要。
- 不在目录页深编辑节点、边、contract_bindings。

契约库：

- 管理契约主数据、版本、字段、模板。
- 提供可被图、节点、边绑定的契约资产。
- 不在契约库里替某个图编辑绑定关系。绑定关系属于图工作台。

编排资源库：

- 展示 Agent、运行档案、Projection 摘要。
- 提供跳转到编排系统的入口。
- 不把 Agent 的模型配置、Base URL、Provider 配置散落到任务图里。

运行库：

- 按 task_run_id 查看正式记忆、产物、事件、监控。
- 提供追踪和诊断。
- 不配置图结构。

### 3.2 新图工作台负责什么

新图工作台是任务图对象的唯一深编辑入口。

图级：

- 图身份、任务域、图接口、入口出口、默认运行策略。
- 图级 `contract_bindings`。
- 图级模型需求可以通过 `contract_bindings.runtime.model_requirement` 声明，但 Provider endpoint 仍来自系统配置。

节点级：

- 节点身份、引用任务定义、Agent / Agent 组 / Projection / 运行档案。
- 输入、执行、记忆、产物、验收、运行六个自然分区。
- 节点的 `contract_bindings` 是主编辑对象。

边级：

- 边是交接端。
- 配置载荷、可见性、确认、失败传播、结果交付、时序等待。
- 边的 `contract_bindings` 是主编辑对象。

图节点：

- 图节点是一种可执行单元。
- 引用 `linked_graph_id`、版本锚点、输入端口、输出端口、隔离策略、可见性策略。
- 发布包必须能展示父图节点如何落到 runtime node，以及子图执行计划是否有效。

发布运行：

- `execution package` 是发布页主事实源。
- 用户必须能从图上对象追踪到 runtime spec、contract manifest、scheduler、assembly、child graph plan。
- 不能让用户在多个页面之间猜“图上配置到底有没有进入运行”。

## 4. 前端优化方案

### 4.1 信息架构重命名

保留 `management/editor` 技术状态，但产品文案改成：

- `库管理`
- `图工作台`

`TaskSystemShell` 中的标题应避免继续使用两套系统感很强的 `TaskSystem Console` / `TaskGraph Studio`。建议：

- 管理态：`任务系统 · 库管理`
- 编辑态：`任务系统 · 图工作台`

### 4.2 管理层页面瘦身

`TaskSystemView.tsx` 需要拆分。

建议新增或强化：

```text
frontend/src/components/workspace/views/task-system/library/
├─ TaskLibraryShell.tsx
├─ TaskDomainLibraryPage.tsx
├─ TaskDefinitionLibraryPage.tsx
├─ TaskGraphLibraryPage.tsx
├─ TaskContractLibraryPage.tsx
├─ TaskOrchestrationResourceLibraryPage.tsx
└─ TaskRuntimeLibraryPage.tsx
```

管理层每页只保留：

- 左侧目录或顶部卡片切换。
- 中间主数据表单或摘要。
- 明确的“进入图工作台”动作。

从管理层删除或迁出：

- 节点装配摘要的深配置。
- contract manifest 的深编辑。
- runtime spec 的深预检。
- 资源流配置。
- 节点/边的旧字段表单。

### 4.3 新工作台采用三栏结构

新工作台应该统一为：

```text
左侧：图层关系
中间：任务图画布 / 当前层主视图
右侧：对象编辑台
底部：发布包与问题栏
```

左侧图层关系：

- 当前图。
- 子图 / 图节点层级。
- 阶段 / 时序块。
- 资源层。
- 运行包对象追踪。

中间画布：

- 节点和边。
- 图节点可以双击展开。
- 边作为交接结构可被直接选中。
- 图、节点、边、图节点都是可选对象。

右侧对象编辑台：

- 选中图：图身份、接口、默认策略、图级契约。
- 选中节点：输入、执行、记忆、产物、验收、运行。
- 选中边：载荷、交接、时序、可见性、失败传播。
- 选中图节点：链接子图、版本锚、端口映射、隔离与可见性。

底部发布包：

- 预检状态。
- `execution package` 摘要。
- 当前对象 trace。
- 阻塞问题。

### 4.4 工作台层级应收敛

现有 `TaskGraphWorkbench` 的很多层可以保留，但要从“多个页面”调整为“同一个对象编辑工作台的不同视图”：

- `blueprint`：图对象编辑。
- `modules`：主图画布和图层关系，成为默认入口。
- `topology`：可以并入主画布，或成为画布模式。
- `agents/responsibility`：并入节点对象编辑台的执行分区。
- `timeline`：变成左侧图层或画布模式。
- `memory/risk/contracts`：变成对象编辑台分区和底部诊断，不再像独立系统页。
- `publish`：保留为发布包视图，但也要和底部发布包保持一致。

### 4.5 契约绑定编辑统一化

新建统一组件：

```text
TaskGraphContractBindingInspector.tsx
```

它不直接暴露旧字段，而是按契约分区编辑：

- Schema：输入、输出、载荷、图接口。
- Execution：执行契约、工具、skill、executor policy。
- Runtime：模型需求、输出预算、时限、重试、runtime lane。
- Memory：读取、动态记忆、写回、提交。
- Artifact：产物目标、stream、引用策略。
- Handoff：确认、可见性、失败传播、结果交付。
- Acceptance：审核、质量门、人工确认。
- Temporal：等待、触发、阶段、循环。

保存时继续由 `taskGraphSaveMapper.ts` 负责归一化，但 UI 主路径以 `contract_bindings` 为准。

### 4.6 运行包视图成为发布页中心

新增或强化：

```text
TaskGraphExecutionPackagePanel.tsx
TaskGraphObjectTracePanel.tsx
```

发布页显示：

- 图是否有效。
- manifest 是否有效。
- runtime spec 是否有效。
- scheduler 是否有 blocked node。
- 每个节点是否有 runtime assembly。
- 每个图节点是否有 child execution plan。
- 当前选中对象的 trace index。

用户选中图上任意对象时，右侧或底部能看到：

```text
源对象 -> contract_bindings -> contract manifest -> runtime spec -> scheduler -> assembly / child plan
```

## 5. 后端优化方案

### 5.1 保持库 API，但明确层级

现有 `/tasks/overview` 可以继续作为库管理聚合接口，但需要减少前端对它的深编辑依赖。

建议逐步形成：

```text
/tasks/library/overview
/tasks/library/domains
/tasks/library/task-definitions
/tasks/library/task-graphs
/tasks/library/contracts
/tasks/library/runtime-runs
```

短期可以不立刻改 API 路径，但前端代码层要先按 library / workbench 分层。

### 5.2 图工作台 API 以 TaskGraph 为中心

新工作台优先使用：

```text
GET /tasks/task-graphs/{graph_id}
PUT /tasks/task-graphs/{graph_id}
GET /tasks/task-graphs/{graph_id}/standard-view
PUT /tasks/task-graphs/{graph_id}/standard-view
GET /tasks/execution-packages/task-graphs/{graph_id}
GET /tasks/runtime-assemblies/task-graphs/{graph_id}/nodes/{node_id}
```

其中 `execution package` 应成为发布前真实执行包。

### 5.3 契约字段迁移规则

短期保留旧字段以兼容已有数据，但建立强规则：

- 新 UI 不再直接编辑旧字段。
- 保存时由 mapper 将旧字段迁移到 `contract_bindings`。
- 后端解析时 `contract_bindings` 优先。
- 后端返回时可以保留旧字段，但应在 `metadata.legacy_contract_fields` 或诊断中标识来源。
- 测试必须覆盖旧字段导入、新字段保存、新字段优先。

中期：

- 新增后端诊断 issue：当旧字段和 `contract_bindings` 冲突时提示 `legacy_contract_field_shadowed`。
- 发布包 trace 中显示契约来源。

长期：

- 库存数据迁移完成后，旧字段只保留只读导入能力。

### 5.4 图节点运行包闭环

后端已有 `nested_runtime_plans` 和 `graph_unit_execution_plans`，但需要作为工作台正式协议稳定下来：

GraphUnit 必备字段：

- `linked_graph_id`
- `version_ref`
- `input_port_id`
- `output_port_id`
- `handoff_contract_id`
- `isolation_policy`
- `visibility_policy`
- `detach_policy`

发布包必须证明：

- 父图 runtime spec 中存在 graph_unit runtime node。
- 子图 runtime spec 有效。
- 子图 contract manifest 有效。
- 子图 scheduler shadow state 可进入初始状态。
- 父图到子图的交接契约存在。

### 5.5 模型供应商配置分层

系统配置是 Provider endpoint 和 credential 的事实源。

Agent 运行档案可以配置：

- provider family / provider 选择。
- model。
- credential_ref。
- max_output_tokens。
- timeout / retries。
- thinking / reasoning。

任务图节点可以通过 `contract_bindings.runtime.model_requirement` 声明需求：

- min_output_tokens。
- preferred_output_tokens。
- capability_tags。
- provider_family。
- profile_ref。

节点不能直接保存 `base_url` 和 raw secret。Base URL 应归系统配置。

## 8. 不允许的实现方式

- 不允许把写作任务特例塞进后端。
- 不允许为了某个测试写死产物。
- 不允许继续把节点/边深编辑留在库页面。
- 不允许 UI 主路径继续裸露旧契约字段。

- 不允许前端显示可配置而后端完全不消费的假能力。
- 不允许只换文案不调整职责边界。

## 9. 第一轮落地建议

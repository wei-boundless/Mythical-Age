# 228-图 Harness 控制系统优先重构设计方案

日期：2026-05-27

## 1. 核心判断

图任务系统的核心不是任务图，也不是装配系统，而是控制系统。

```text
GraphHarness / GraphLoop = 控制核心
GraphHarnessConfig = 控制系统可执行合同
TaskEnvironment / SpecificTask / AgentFactory / Tool / Permission / Memory / Artifact = 装配供给层
GraphEditor = GraphHarnessConfig 的可视化编辑器
```

因此重构顺序必须是：

```text
1. 设计 GraphLoop 控制语义
2. 设计 GraphHarnessConfig 配置合同
3. 设计任务系统 / 装配系统对接
4. 设计图编辑器投影
5. 设计发布与运行链
```

禁止从图编辑器或现有 TaskGraphDefinition 反推运行系统。

## 2. 当前代码结论

当前写作任务图配置本身不是问题。它已经表达了复杂写作流程需要的业务配置：

```text
章节批次循环
分卷循环
节点角色 prompt
章节拆分与长度预算
memory read / commit
artifact 输出
质量审核与返修
agent / tool / permission 绑定
```

问题是现有图系统没有一个统一的 GraphHarnessConfig 来承载它。当前运行链是多段解释：

```text
TaskGraphDefinition
-> compile_task_graph_definition_runtime_spec
-> TaskGraphRuntimeSpec
-> graph_payload
-> CoordinationTaskDefinition
-> stage_contracts
-> scheduler_state
-> NodeExecutionRequest
```

这条链路可以让写作图“部分运行”，例如能编译、启动、初始化、生成首个节点请求，但不能称为可靠的长任务闭环控制系统。因为同一份写作配置被拆散到：

```text
graph.metadata.graph_loop_policy
graph.contract_bindings.runtime
node.contract_bindings
node.metadata.role_prompt
memory repository node
memory edge
runtime_spec.diagnostics
coordination_task.metadata
GraphCoordinationEngine state
```

本次重构目标是把这些散落配置统一收束到 GraphHarnessConfig。

## 3. 第一阶段：GraphLoop 控制语义

### 3.1 GraphLoop 的职责

GraphLoop 只负责动态控制：

```text
初始化图运行状态
判断节点 ready / blocked / running / completed / failed
处理依赖、条件、gate、失败传播
处理循环 frame
接收 NodeResult
推进 GraphLoopState
生成下一个 NodeExecutionRequest
判断整图完成、失败、暂停、等待人工
```

GraphLoop 不负责：

```text
编译 TaskGraphDefinition
读取图编辑器草稿
选择任务环境
选择 agent
扩大工具权限
生成角色 prompt
执行 agent model/tool 内环
从 metadata 猜配置
```

```text

```

## 5. 第三阶段：任务系统与装配系统对接

### 5.1 任务系统职责

任务系统负责提供配置材料，不负责运行控制。

```text
TaskEnvironment -> resources / memory / artifact / permission 默认范围
SpecificTask -> task_ref / role / input_contract / output_contract / quality_policy / skill requirements
TaskGraph publication -> GraphHarnessConfig version
```

任务系统不负责：

```text
推进 GraphLoopState
运行 agent
现场重编译已发布图
决定节点下一步
```

### 5.2 子任务关系

子任务不是图节点里的二级任务生命周期。子任务是节点工作定义来源。

```text
SpecificTask
-> GraphNodeConfig.task_ref
-> GraphNodeConfig.prompt_contract
-> GraphNodeConfig.input/output contract
-> GraphNodeConfig.quality_policy
```

GraphLoop 只调度节点，不启动子任务生命周期。

### 5.3 装配系统职责

装配系统把节点配置转成 AgentHarness 可执行请求。

```text
GraphNodeConfig
GraphLoopState
TaskEnvironment
AgentFactory
ToolPolicy
PermissionPolicy
MemoryScope
ArtifactScope
-> NodeExecutionRequest
```

装配系统不负责：

```text
判断下一个节点
改变图状态
绕过 GraphLoop
修改 GraphHarnessConfig
```

## 6. 第四阶段：图编辑器投影

图编辑器是 GraphHarnessConfig 的可视化 IDE。

### 6.1 编辑器保存对象

```text
graph_config
editor_layout
```

`graph_config` 是可执行配置。  
`editor_layout` 只包含坐标、视口、折叠、分组、选中状态。

### 6.2 画布节点

画布节点是 `GraphNodeConfig` 的投影。

```text
节点标题 -> node.title
节点类型 -> node.node_kind
绑定任务 -> node.task_ref
绑定 agent -> node.agent_binding_ref
输入端口 -> node.input_contract_ref / input_projection
输出端口 -> node.output_contract_ref / output_mapping
权限提示 -> permission_scope_ref
memory 提示 -> memory_scope_ref
产物提示 -> artifact_scope_ref
```

### 6.3 连线

连线是 `GraphEdgeConfig` 的投影。

```text
source / target
dependency_policy
handoff_policy
condition_policy
payload_contract_ref
failure_propagation_policy
```

### 6.4 面板

必须分层：

```text
图级控制面板：control / loop_frames / completion / failure
节点面板：task / agent / prompt / input / output / quality
边面板：dependency / handoff / condition / failure propagation
资源面板：memory / artifact / file / sandbox
权限面板：tools / operations / approval
发布面板：validate / publish / version
```

### 6.5 编辑器约束

```text
任何 UI 操作必须落到 GraphHarnessConfig 字段。
没有字段归属的 UI 功能不能进入运行链。
editor_layout 不参与运行。
GraphEditor 不推断运行字段。
GraphEditor 不编译 GraphHarnessConfig。
GraphEditor 只做局部校验和可视化反馈。
```

## 7. 第五阶段：发布与运行链

### 7.1 发布

```text
GraphHarnessConfig draft
-> validate_graph_harness_config
-> publish config version
-> bind graph_id -> config_id
```

发布后：

```text
config_id 不可变
graph_id 指向当前 published config
草稿修改不影响已发布运行
```

### 7.2 启动

```text
POST /orchestration/harness/task-graphs/{graph_id}/start
-> resolve published config_id
-> load GraphHarnessConfig
-> GraphHarness.start(config)
-> GraphLoop.initialize(config)
```

启动时禁止：

```text
compile_task_graph_definition_runtime_spec
读取 TaskGraphDefinition 生成运行配置
读取 editor layout
GraphLoop fallback compile
```

### 7.3 运行

```text
GraphLoopState + GraphHarnessConfig
-> ready node
-> NodeExecutionRequest
-> AgentHarness
-> NodeResultEnvelope
-> GraphLoop.transition
```

### 8.3 方案自审补充：本次复核发现的遗漏

本节是对本方案的反向审查。结论：方案方向正确，但原稿仍漏掉了几处会保留旧控制权的关键器官。如果不补进实施计划，后续很容易只把入口改名为 `GraphHarnessConfig`，实际运行仍由旧 `TaskGraphRuntimeSpec / graph_payload / metadata / graph_module` 分支继续决定。

#### 8.3.1 系统总纲仍保留旧合同名

```text
docs/系统框架/001-Agent系统框架总纲-20260527.md
```

当前总纲的图任务主链仍写成：

```text
TaskGraphRuntimeSpec
-> GraphLoop
```

这与本方案的目标主链冲突。目标必须统一为：

```text
GraphHarnessConfig
-> GraphHarness.start
-> GraphLoop
```

`TaskGraphRuntimeSpec` 只能作为迁移期旧编译产物或历史对象存在，不能再作为 GraphLoop 的边界合同。

#### 8.3.2 graph module 是最大遗漏，但不能成为运行期 executor

```text
backend/harness/execution/graph_module_executor.py
backend/harness/execution/graph_module_runtime.py
backend/harness/execution/graph_module_result_packets.py
backend/api/task_system.py
```

当前 graph module 链路会：

```text
从 NodeExecutionRequest / WorkOrder 中多源解析 handle
-> 读取 linked_graph_id
-> registry.get_task_graph(linked_graph_id)
-> compile_task_graph_definition_runtime_spec(imported_graph)
-> graph_harness.start_run(graph, runtime_spec)
-> 通过 imported run packet 回灌父图
```

这是一条隐藏的子图 runtime 链。它绕过了“已发布 GraphHarnessConfig 是唯一执行合同”的原则，也解释了为什么图模块容易反复重新解释 linked graph。目标不是把这条链改成 `linked_config_id -> child GraphHarness`，而是切除运行期子图概念。

目标处理：

```text
GraphicalGraphProjection parent/child blocks or graph_module refs
-> publisher expand / inline / normalize
-> GraphHarnessConfig.nodes[]
-> GraphHarnessConfig.edges[]
-> GraphHarnessConfig.loop_frames[]
-> GraphHarnessConfig.resources / memory / artifacts / permissions / contracts
-> GraphHarnessConfig.source_refs / composition_sources for audit and editor round-trip
```

GraphLoop 只调度已归一化后的 executable nodes。`graph_module` 不能作为 `work_kind`，不能读取 `linked_config_id`，不能启动 child GraphHarness。父子块耦合可以存在，但其执行效果必须在发布期映射为节点、边、loop frame、scope 和 contract。

#### 8.3.3 batch lifecycle 必须纳入 GraphLoop 控制语义

```text
backend/runtime/graph_runtime/batch_runtime.py
backend/runtime/graph_runtime/scheduler.py
backend/tests/task_graph_batch_runtime_regression.py
```

当前章节批次、repair、commit、merge-ready 等复杂逻辑在 `batch_runtime` 中是真实业务能力，不应粗暴删除。但它现在消费 `TaskGraphRuntimeSpec` payload，并作为 `batch_lifecycle_runtime_state` 放进 GraphEngine state/diagnostics。

目标处理：

```text
GraphHarnessConfig.control.batch_policy
GraphHarnessConfig.nodes[].execution_policy.split_policy
GraphHarnessConfig.nodes[].quality_policy.batch_acceptance_policy
GraphHarnessConfig.loop_frames[].batch_scope
GraphLoopState.batch_frames
```

batch lifecycle 只能成为 GraphLoop 的子状态机，不能继续作为 `runtime.graph_runtime` 目录下的第二套图运行时。

#### 8.3.4 monitor / finalizer 不能保留恢复型解释权

```text
backend/runtime/graph_runtime/run_monitor.py
backend/harness/loop/task_run_finalizer.py
```

当前 monitor 主要是读模型，但它仍以 `coordination_graph_spec` 和 scheduler state 作为拓扑来源。finalizer 仍会反查 `TaskFlowRegistry.get_task_graph`，并从 `stage_sequence / stage_contracts` 反推阶段身份。

目标处理：

```text
MonitorView = projection(GraphHarnessConfig, GraphLoopState, events, checkpoints)
Finalizer = consume(GraphLoopState.final_result, ResultEnvelope, artifact refs)
```

monitor / finalizer 不允许：

```text
读取 TaskGraphDefinition 反推拓扑
从 metadata.stage_contracts 反推阶段
补做 GraphLoop 没完成的状态推进
按旧 runtime_spec 生成健康判断
```

#### 8.3.5 contract manifest / stage contract 编译位置必须前移

```text
backend/runtime/contracts/continuation_policy.py
backend/runtime/contracts/compiler.py
backend/harness/loop/graph_coordination/engine.py
```

当前 `parse_stage_contracts` 在运行期读取 `metadata.stage_contracts`；如果没有，还会从图节点推导。这属于运行期猜配置。

目标处理：

```text
GraphHarnessConfig.contracts.node_contracts
GraphHarnessConfig.contracts.edge_contracts
GraphHarnessConfig.contracts.prompt_contracts
GraphHarnessConfig.contracts.acceptance_contracts
GraphHarnessConfig.contracts.runtime_contracts
```

contract manifest 只能在发布期编译并写入 `GraphHarnessConfig`。GraphLoop 只读取已发布合同，不再调用 stage contract parser 兜底推导。

#### 8.3.6 result commit / acceptance / retry 需要成为一等控制语义

```text
backend/harness/loop/graph_coordination/node_result_committer.py
backend/task_system/runtime_semantics/quality_gates.py
backend/harness/loop/graph_coordination/engine.py
```

原方案写了 `result_commit_policy`，但没有把验收、返修、提交、重试、失败传播拆成明确事件链。

目标事件链：

```text
node_result_received
-> node_result_validated
-> node_result_accepted | node_result_rejected
-> node_result_committed | node_repair_requested
-> retry_scheduled | failure_propagated | graph_completed
```

GraphLoop 只能基于：

```text
GraphHarnessConfig
GraphLoopState
NodeResultEnvelope
```

执行这些判断。质量门、返修路由、commit gate 不应散落在 finalizer、batch runtime、metadata policy、API continue 分支里。

#### 8.3.7 发布产物必须有对象存储、版本和不可变语义

当前运行会把 `task_graph_definition_ref`、`task_graph_runtime_spec_ref` 放入 runtime object store。目标不能简单把名字换成 `graph_harness_config_ref`，还必须定义发布物语义：

```text
config_id
graph_id
publish_version
content_hash
schema_version
published_at
source_draft_ref
compiled_from_refs
validation_report_ref
status
```

启动链只能通过：

```text
graph_id -> published config_id -> immutable GraphHarnessConfig
```

运行中的 rewind / resume 只能读取该 `config_id`，不能用 live graph 替换当前运行合同。

#### 8.3.8 测试中仍有保护旧结构的断言

```text
backend/tests/writing_modular_novel_graph_config_regression.py
backend/tests/langgraph_coordination_runtime_regression.py
backend/tests/task_graph_batch_runtime_regression.py
```

当前测试仍保护：

```text
graph.metadata.graph_loop_policy
node.metadata.role_prompt
compile_task_graph_definition_runtime_spec
TaskGraphRuntimeSpec
refresh_graph_spec=True
live graph refresh
batch runtime consume runtime_spec payload
```

这些测试不能简单删除后不补。目标是改写为保护：

```text
GraphHarnessConfig.loop_frames
GraphHarnessConfig.nodes[].prompt_contract
GraphHarnessConfig.modules[]
GraphHarnessConfig.control.batch_policy
GraphHarness.start(config_id/config)
GraphLoop 不读取 TaskGraphDefinition
rewind / resume 不刷新 live graph
```

其中 `test_rewind_refresh_uses_live_graph_loop_policy_instead_of_stale_snapshot` 与目标架构直接冲突，应删除或改写成“rewind 固定使用运行启动时的 config_id，不允许 live graph 覆盖已运行合同”。

#### 8.3.9 图编辑器字段标准化还没有落到后端发布链

前端已有多层编辑页和 publish/run 页，但目前发布链仍围绕 TaskGraph draft、standard view、runtime_spec preview、execution package。目标不是让编辑器再编译一套语言，而是：

```text
GraphEditorDraft
-> GraphHarnessConfig draft
-> validate
-> publish immutable GraphHarnessConfig
```

编辑器布局只能进入：

```text
editor_layout
```

不能进入：

```text
GraphHarnessConfig.control
GraphHarnessConfig.nodes
GraphHarnessConfig.edges
GraphHarnessConfig.loop_frames
```

任何运行字段必须由字段面板明确编辑或由发布期编译器明确生成，不能在 start / loop / monitor 中补猜。

## 9. 验收标准

### 9.1 结构验收

```text
GraphHarness.start 只接收 GraphHarnessConfig。
GraphLoop.initialize 只接收 GraphHarnessConfig。
GraphLoop 不 import task_system compiler。
GraphLoop 不调用 TaskFlowRegistry.get_task_graph。
GraphLoop 不读取 graph.metadata 作为运行策略。
GraphLoop 不接收 TaskGraphRuntimeSpec。
GraphLoop 不读取 task_graph_definition_ref / task_graph_runtime_spec_ref。
GraphLoop rewind / resume 不允许 refresh live graph。
graph module executor 不允许现场编译 linked graph。
contract manifest 不允许运行期从 metadata 推导。
GraphEditor 保存 graph_config + editor_layout。
```

### 9.2 写作图验收

写作图必须能表达：

```text
章节批次循环
分卷循环
章节正文批量生成
审核返修
memory read / commit
正文 artifact 输出
动态扩展
最终汇总
```

并且这些表达必须全部落到 GraphHarnessConfig 字段，不再依赖 metadata。

### 9.3 运行验收

```text
启动 graph_id 能解析到 published config_id。
启动不会现场编译 TaskGraphDefinition。
首节点 NodeExecutionRequest 使用 GraphHarnessConfig 生成。
章节批次 loop_summary 来自 loop_frames。
chapter_draft 的 prompt 来自 prompt_contract.role_prompt。
memory_search 权限来自 tool_policy_ref / permission_scope_ref。
review 返修路线来自 loop / edge / quality policy。
graph module / 父子块在发布期归一化为节点、边、loop frame、scope 和 contract，运行期不读取 linked_config_id。
batch lifecycle 由 GraphLoopState.batch_frames 推进，不读取 runtime_spec payload。
monitor 只投影 GraphHarnessConfig + GraphLoopState，不反推运行拓扑。
```

## 10. 实施原则

本方案不是给旧链路再包一层新壳，而是要重建主权链：

```text
GraphControlSemantics
-> GraphHarnessConfig
-> Task/Assembly supply
-> GraphEditor projection
-> Published config
-> GraphLoop execution
```

所有不属于这条链的运行期推导、fallback、metadata 猜测、现场编译，都应在迁移完成后删除。

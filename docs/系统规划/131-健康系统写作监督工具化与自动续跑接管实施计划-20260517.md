# 131-健康系统写作监督工具化与自动续跑接管实施计划-20260517

## 1. 问题定义

当前写作任务的监督职责分散在 `.codex/hooks`、`scripts/watch_writing_supervision.py`、`scripts/supervise_writing_campaign.py` 三处：

- hooks 负责触发和提醒
- watchdog 负责重启监督器/后端
- supervisor 负责判断停滞并执行 `continue-current-stage` / 停止旧节点

这导致两个问题：

1. 监督逻辑不在系统内部，诊断和动作无法统一审计。
2. `runtime_status=running` 但 `seconds_since_progress_change` 很久不变时，外部 watchdog 仍然可能把它当成健康，造成假活。

目标不是再加一个补丁式脚本，而是把写作监督收敛成健康系统内的正式工具。

---

## 2. 现状源代码依据

已确认的关键入口：

- `backend/api/health_system.py`
  - 现有 `POST /api/health-system/commands`
  - 现有命令流：`report_issue`、`analyze_trace`、`draft_case`、`verify_fix`、`launch_health_test`
- `backend/health_system/registry.py`
  - 已有 `HealthRegistry`
  - 已有 `HealthCommandService`
  - 已有 issue / report / receipt / agent_run / conversation 统一存储
- `backend/health_system/command_service.py`
  - 命令执行与回执已是正式链路
- `scripts/supervise_writing_campaign.py`
  - 已经具备“发现卡住后 `continue-current-stage`”的修复能力
- `scripts/watch_writing_supervision.py`
  - 当前职责是低频外部守望与重启，不应继续承担主监督职责
- `.codex/hooks/writing_project_monitor.py`
  - 目前只做 Stop-hook 续接判断，适合作为唤醒入口，不适合作为主决策器

结论：
健康系统已经具备命令、回执、报告、问题单四件套，只差一个专用写作监督命令类型，把“监测 + 诊断 + 动作建议/执行”接进来。

## 3. 设计方向

### 3.1 监督职责迁移

把写作监督拆成三层：

1. 健康系统工具层
   - 接收“检查写作运行状态”“请求续跑”“请求重启监督器”等命令
   - 产出 report / receipt / issue

2. 写作监督执行层
   - 负责读取当前 run / task graph monitor / live monitor / memory receipts
   - 识别真实停滞、失败、假活、后台断链
   - 生成明确动作：`continue_current_stage`、`restart_supervisor`、`restart_backend`、`no_action`

3. hooks 唤醒层
   - 只负责在 Stop 或异常唤醒点调用健康系统工具
   - 不再持有重启主逻辑

### 3.2 命令类型

新增健康系统命令类型：

- `monitor_writing_campaign`
- `resume_writing_campaign`
- `restart_writing_supervision`

建议执行语义：

- `monitor_writing_campaign`
  - 只诊断，不动系统
  - 返回健康摘要、停滞原因、建议动作
- `resume_writing_campaign`
  - 优先尝试 `continue-current-stage`
  - 若发现需要结构性续跑，返回可执行建议
- `restart_writing_supervision`
  - 允许健康系统调用现有启动脚本或重启入口

### 3.3 判定标准

不再只看 `runtime_status=running`，而要同时看：

- `seconds_since_progress_change`
- `seconds_since_event_change`
- `seconds_since_effective_activity`
- `active_node_id`
- `active_task_ref`
- `blocker`
- `memory_receipts`
- `artifacts` 是否按轮次增长

原则：

- `running` 但长期无进度，不算健康
- 有正文/记忆产物，但状态未收口，也不算完成
- 能续跑就续跑，不能续跑再重启监督器

## 4. 推荐实现路线

### 阶段 1：健康系统接管诊断

实现一个健康系统级别的写作监督命令处理器：

- 读取当前项目运行状态
- 读取 task graph monitor / live monitor / artifacts / memory receipts
- 输出结构化诊断报告
- 通过 receipt 标明是否建议续跑、重启或停机

### 阶段 2：健康系统接管动作

把动作执行也并入健康系统命令流：

- 续跑：调用现有 orchestration 的 `continue-current-stage`
- 重启监督：调用现有 `start_writing_supervisor.ps1`，但只作为健康系统命令的后端动作
- 后端不可用：由健康系统判断后返回重启建议，不让 hooks 自己猜

### 阶段 3：hooks 降级为唤醒器

`.codex/hooks` 只做两件事：

- Stop 时触发健康系统命令
- 发现明显异常时发起一次健康系统检查

其余逻辑全部上移到健康系统。

## 6. 验证标准

必须验证以下场景：

1. 正常推进时，健康系统不误报、不频繁触发。
2. `running` 但 `seconds_since_progress_change` 持续增大时，健康系统能判定为假健康。
3. `continue-current-stage` 可用时，优先续跑而不是重启。
4. 监督器进程丢失时，健康系统能发出重启命令。
5. 后端健康接口失败时，健康系统能区分是“后端断链”还是“任务真卡死”。
6. 命令、回执、报告链路都真实落盘，不能只在内存里看起来恢复了。

---

## 7. 风险控制

- 不允许 hooks 直接拼接重启逻辑。
- 不允许把 `running` 当健康。
- 不允许在没有产物、没有回执、没有命令记录的情况下宣称续跑成功。
- 不允许新旧两套监督逻辑并行长期存在。

---

## 8. 收尾条件

满足以下条件后收尾：

- 健康系统能独立发起写作监督诊断命令
- 续跑和重启动作都能通过健康系统命令触发
- `.codex/hooks` 退化为唤醒层
- 旧 watchdog 不再是主监督路径
- 写作任务的真实推进和收口，都能在健康系统里审计到

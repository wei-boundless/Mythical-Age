# 239-单Agent任务边界与续接控制修复方案

状态：已审核锁定，待实施  
日期：2026-06-02  
范围：单 agent harness 控制系统、TaskRun 对话边界、prompt cache 装配边界  
不在本方案范围：Graph runtime、图像生成旁路、正在重修的能力系统内部实现

## 0. 审核结论

本方案经三类证据核对后锁定：

1. 当前文档原版本存在实施歧义：`task_checkout.py` 的删除决策未锁定，历史摘要、checkout 用途、`waiting_executor` 续接三件核心事项也未锁定。
2. 当前 harness 代码仍保留旧对话恢复链：`active_work.py` 将 terminal TaskRun 归类为 `interrupted_checkoutable`，`resume_policy.py` 将其转成 `checkout_fork`，`task_checkout.py` 创建带 parent lineage 的 child TaskRun，`runtime_facade.py` 仍能调用 `_apply_checkout_active_work()`。
3. 本地 Codex 源码没有“从旧 terminal task 通过自然语言恢复”的路由层。Codex 的不变量是：同一 active turn 内允许 pending input / tool follow-up / turn_state；turn 完成或出错后释放 active turn；下一条用户输入是新 turn；prompt cache 依赖稳定前缀和稳定 `prompt_cache_key`。

最终设计原则已经锁定：用户自然语言只能 steer 真实 active turn；terminal TaskRun 只作为历史事实；terminal 后的“继续”是新 UserTurn，长任务推进必须由模型在本轮调用 `request_task_run` 建立新 TaskRun。旧 `checkout_fork` 对话链无保留理由，实施阶段删除。

## 1. 问题定义

这次问题不是“继续”语义识别错误，也不是关键词路由缺失。真实问题是单 agent 控制链把三种不同层级混在一起：

1. 当前仍在运行的 active turn。
2. 已经终止的旧 TaskRun。
3. 用户新一轮自然语言请求。

正确设计必须保证：旧任务断掉就断掉，不能再被当作当前任务恢复；用户后续说继续时，进入新的用户 turn，并在需要持续执行时建立新的 TaskRun。旧任务只作为历史事实、摘要或证据来源，不允许继续绑定、调度、checkout、same-run resume。

当前失败表现：

- 长任务断掉后，后续自然语言“继续”没有形成干净的新任务边界。
- 单轮入口会用只读工具搜索旧产物目录，形成旧上下文污染，并触发 `single_turn_tool_iteration_limit`。
- 旧的 `interrupted_checkoutable` / `checkout_fork` 设计仍然存在，语义上仍是从旧 TaskRun 恢复，而不是新 TaskRun 承接。
- prompt cache 稳定装配被破坏：模型看到的动态上下文混入旧任务控制语义、旧 sandbox 路径或旧中断现场。

正确终态：

```text
UserTurn
-> RuntimeAssembly
-> ActiveTurnSnapshot 只判断当前 turn 是否仍活着
-> SingleAgentDecision
   -> respond / ask_user / block
   -> request_task_run 建立新的 TaskRun
   -> active_work_control 仅限真实 active turn
-> TaskRunExecution
-> terminal 后释放 active turn
```

## 3. 锁定设计原则

### 原则 1：ActiveTurn 是唯一当前工作控制权威

控制链只使用 `ActiveTurnRegistry.resolve_current(session_id)` 取得当前工作。`resolve_current()` 是新增控制入口：它读取 `snapshot()`，校验 bound TaskRun 状态，清理 stale binding，并只在 active turn 与 bound TaskRun 同时非 terminal 时返回记录。`snapshot()` 保持只读语义，只用于诊断和测试。

禁止：

- 从旧 TaskRun 列表构造 current work。
- 从 terminal TaskRun 生成 active_work_context。
- terminal TaskRun 触发 same-run resume、checkout-fork、active_work_control。
- 通过关键词判断“继续”并跳过模型本轮决策。

### 原则 2：TaskRun terminal 后只保留审计事实

terminal TaskRun 允许保留：

- event log
- work rollout 摘要
- 用户可见 artifact refs
- terminal reason
- final answer / blocker reason

terminal TaskRun 不允许：

- 被 active_work_control 控制。
- 被 resume_policy 调度。
- 被 checkout_task_run_for_resume 作为用户自然语言继续路径。
- 将旧 sandbox 路径、runtime_control、executor_status、checkpoint、lineage 注入新任务 prompt。

### 原则 3：自然语言续接创建新 TaskRun

用户在旧任务终止后说“继续”，系统进入新 turn。模型在本轮根据当前用户消息和对话历史作出语义决策；目标需要持续执行时调用 `request_task_run` 创建新的 TaskRun。

新 TaskRun 的合同来源：

1. 当前用户消息。
2. 当前对话历史中的用户可见目标。
3. 经清洗的历史摘要。

新 TaskRun 禁止继承：

- `parent_task_run_id`
- checkout lineage
- `runtime_control`
- old `executor_status`
- old `active_work_context`
- old sandbox private path
- old `task_run_id` / `turn_run_id` / checkout ref

### 原则 4：内部执行器恢复与用户续接分离

内部 same-run 恢复只属于执行器状态机，不属于用户自然语言对旧任务的恢复。

允许 same-run 的条件全部满足：

- ActiveTurn 仍存在。
- ActiveTurn 绑定的 TaskRun 尚未 terminal。
- TaskRun 状态是 `running`、`created`、`waiting_executor`、`waiting_approval`、`blocked` 中的非 terminal 状态。
- 用户输入通过 `ActiveTurnRegistry.steer()` 或本轮显式 `active_work_control` 进入当前 active turn。
- 执行器恢复由 `TaskExecutorController.schedule/recover_scheduled/recover_interrupted_executor_leases` 处理，不经过 conversation checkout。

`waiting_executor` 且 active turn 已释放的旧任务不接受自然语言控制。它只能由内部 scheduler/lease recovery 处理；用户后续自然语言进入新 UserTurn。

### 原则 5：Prompt cache 以稳定装配为核心

稳定区：

- global runtime prompt
- role prompt
- environment policy
- stable tool catalog summary
- output action schema

动态区：

- current user request
- conversation history projection
- active_work_context，仅限真实 active turn
- historical terminal summary，仅限 sanitized、volatile、非控制句柄

禁止将 terminal task recovery 信息放入 stable prefix。禁止把旧任务恢复上下文塞进稳定区来制造缓存命中。

### 原则 6：历史摘要是非控制背景

历史摘要进入 prompt 的唯一形态：

```json
{
  "historical_work_summary": {
    "status": "terminal",
    "user_visible_goal": "...",
    "public_result_summary": "...",
    "usable_artifact_refs": [],
    "non_control_context": true
  }
}
```

清洗规则：

- 删除 `task_run_id`、`turn_run_id`、`parent_task_run_id`、`root_task_run_id`、`logical_work_id`。
- 删除 `lineage`、`checkpoint`、`event_offset`、`checkout_ref`、`resume_plan_ref`。
- 删除 `runtime_control`、`executor_status`、`recovery_action`、`recoverable_error`。
- 删除 sandbox 私有路径和内部 runtime object ref。
- artifact refs 只保留用户可见名称、类型、公开路径或可审计摘要。
- 放入 volatile segment，并标注 `non_control_context=true`。

### 原则 7：单轮入口不是旧任务探索执行器

single-agent turn 的职责：

- 轻量回答。
- 询问用户。
- 阻止越界请求。
- 使用少量只读工具辅助当前 turn 判断。
- 在目标需要持续执行、写入、命令验证、浏览器验证、长输出、真实交付或失败后重新推进时调用 `request_task_run`。

single-agent turn 禁止承担：

- 搜索旧 TaskRun 私有产物目录来恢复工作。
- 读取旧 checkout lineage 作为当前任务控制依据。
- 在没有 active_work_context 时暴露 active_work_control。

## 5. 固定执行链

### 5.1 新用户消息入口

```text
HarnessRuntimeRequest
-> ActiveTurnRegistry.resolve_current(session_id)
-> active turn 存在且绑定 TaskRun 非 terminal:
       ActiveTurnRegistry.steer()
       return final_answer_event
-> active turn 不存在:
       commit user message
       start new active turn without bound task
       assemble_runtime
       active_work_context = None
       RuntimeCompiler.compile_single_agent_turn_packet
       model decides respond / ask_user / block / request_task_run
```

### 5.2 TaskRun 启动入口

```text
request_task_run
-> admission
-> start_task_lifecycle_from_action_request
-> bind active turn to new task_run
-> schedule_task_run_executor
-> task execution loop
-> finish_task_lifecycle
-> complete_bound_task
```

### 5.3 ActiveTurn 控制入口

```text
active_work_control
-> context must come from ActiveTurnRegistry
-> bound TaskRun must be non-terminal
-> pause / stop / append_instruction / continue
-> continue uses resume_paused_task_run only for same non-terminal task
-> schedule_task_run_executor
```

### 5.4 Terminal 后续接入口

```text
旧 TaskRun terminal
-> active turn complete
-> 后续用户消息是新 UserTurn
-> no active_work_context
-> sanitized historical_work_summary enters volatile segment
-> model decides respond / ask_user / block / request_task_run
-> 新 TaskRun 无 parent checkout lineage
```

### 5.5 内部 scheduler recovery 入口

```text
runtime startup / scheduler event / lease recovery
-> TaskExecutorController.recover_scheduled 或 recover_interrupted_executor_leases
-> only non-graph single-agent executor states owned by controller
-> no conversation checkout
-> no active_work_context exposure without active turn
```

### Phase 1：收紧 ActiveTurn 权威

目标：active_work_context 只代表真实 active turn。

改动：

- 新增 `ActiveTurnRegistry.resolve_current(session_id)`，作为所有控制链读取 active turn 的入口。
- `resolve_current()` 检查 bound TaskRun 状态；bound TaskRun 已 terminal 时完成 stale active turn 并返回 None。
- `snapshot()` 保持只读，不做 TaskRun 状态修复。
- `ActiveTurnRegistry.steer()` 在写入 steer 前通过 `resolve_current()` 再次校验 bound TaskRun 非 terminal。
- `RuntimeFacade._active_work_context_from_active_turn()` 对 terminal / missing / non-single-agent task 直接返回 None。
- `same_run_allowed` 按 TaskRun 状态与 runtime_control 计算，不再固定 True。
- `active_work_control` native tool 只在 `active_work_context` 存在时装配。

完成标准：

- terminal TaskRun 不出现在 single-agent prompt 的 `active_work_context`。
- active_work_control 不控制 failed / aborted / cancelled / error / completed / success TaskRun。
- stale active turn 不阻断下一条用户消息进入新 UserTurn。

### Phase 2：删除 conversation checkout/resume 链

目标：用户自然语言续接不触发旧 TaskRun checkout 或 terminal same-run resume。

改动：

- 删除 `backend/harness/loop/task_checkout.py`。
- 删除 `backend/harness/loop/resume_policy.py`。
- `runtime_facade.py` 移除 `checkout_task_run_for_resume` 和 `build_resume_plan` 导入。
- 删除 `_apply_checkout_active_work()`。
- `_apply_continue_active_work()` 只处理 already-running 与 non-terminal waiting/paused resume。
- `_continuation_strategy_for_execution()` 删除 `checkout_fork` 和 `interrupted_checkoutable` 分支。
- `active_work.py` 删除 `WorkContinuationCandidate`、`collect_work_continuation_candidates()`、`build_active_work_context()`、`build_active_work_turn_context()`、`select_primary_work_continuation_candidate()`、`_continuation_kind()`、`_is_checkoutable_terminal()`。
- `ActiveWorkContext` 删除 `checkout_allowed` 字段；`continuation_kind` 只表达 active / waiting / paused / blocked / waiting_user / none。

完成标准：

- 代码库生产路径不再命中 `checkout_task_run_for_resume`。
- 代码库生产路径不再命中 `checkout_fork`。
- 自然语言继续链路不产生 `task_run_checkout_created`。
- terminal 旧任务链路不产生 `task_run_resume_requested`。
- 新持续执行只来自 `request_task_run`。

### Phase 3：重塑单轮入口职责与 prompt

目标：single-agent turn 是回答/判断/建任务入口，不是旧任务探索执行器。

改动：

- `RuntimeCompiler` 的运行边界提示明确写入：没有 active_work_context 时，不存在可控制的当前工作；需要持续推进时调用 `request_task_run`。
- `single_agent_turn.py` 保留只读工具能力；read-only loop 只服务当前 turn 问答，不承接旧 TaskRun 恢复。
- 工具调用结果中发现旧任务私有产物路径时，只能作为普通只读观察，不能生成 active_work_control。
- `prompt_library/packs.py` 同步表达新边界：active_work_context 是唯一当前工作句柄；terminal 历史摘要不是控制对象。

完成标准：

- 轻量问答仍能 respond。
- 简单读文件问答仍能 read-only tool 后 respond。
- 长任务/续接任务调用 `request_task_run`。
- 单轮 prompt 不出现 terminal old task 的 active_work_context。

### Phase 4：历史摘要通道清洗

目标：保留用户可理解的历史上下文，不保留旧任务控制权。

改动：

- `WorkHistoryProjector.project()` 改为只输出 `historical_work_summary` 风格的 sanitized payload。
- 删除模型可见 `checkpoint`、`lineage`、raw refs、event offsets。
- 清洗 artifact refs，只留用户可见名称、类型、公开路径或摘要。
- 在 `RuntimeCompiler` 中确认历史摘要只进入 volatile segment。
- segment metadata 标注 `cache_impact=volatile`、`non_control_context=true`。

完成标准：

- prompt 中历史信息短、可读、非控制。
- prompt cache stable prefix 不受历史摘要变化影响。
- 账本中历史摘要 segment 的 cache_role 为 `volatile`。

## 9. 切换与回滚规则

切换规则：

- 本次不做双轨兼容。
- Phase 1 后，terminal TaskRun 不再是 active work。
- Phase 2 后，conversation checkout/resume 不再可达。
- Phase 3 后，single-agent turn 不再承担旧任务恢复探索。
- Phase 4 后，历史摘要只作为 volatile non-control context。

回滚规则：

- Phase 1 暴露 active turn 中真实等待任务无法继续时，只回滚 active turn 状态判断并修正 bound TaskRun 非 terminal 判定。
- Phase 2 暴露内部执行器等待用户输入无法恢复时，只恢复内部 non-terminal waiting 状态的 executor recovery，不恢复 terminal checkout。
- 回滚不得重新引入 `interrupted_checkoutable`、`checkout_fork`、`task_checkout.py`。

## 10. 明确禁止的实现方式

- 禁止用关键词表决定“继续”走旧任务还是新任务。
- 禁止把 terminal TaskRun 包装成 active_work_context。
- 禁止用户自然语言继续触发 `checkout_task_run_for_resume`。
- 禁止以兼容为理由保留旧 checkout 对话链。
- 禁止把旧 sandbox 路径、旧 task_run_id、旧 runtime_control 注入新任务 prompt。
- 禁止把 `checkpoint`、`lineage`、`parent_task_run_id` 放入模型可见历史摘要。
- 禁止用降低测试断言来制造通过。
- 禁止把 Codex active goal 误等同为本项目 TaskRun resume。
- 禁止让 monitoring / health 层重新定义当前工作控制权。

## 11. 已锁定事项

1. 终止旧 TaskRun 后，新 TaskRun 允许读取用户可见历史摘要作为背景；该摘要必须是 sanitized volatile summary，且不能包含控制句柄。
2. `task_checkout.py` 在当前代码库没有独立于 conversation continue 的核心产品权威；本轮重构删除该文件和旧 checkout 测试。
3. `waiting_executor` 且 active turn 已释放的旧任务不接受用户自然语言控制；内部 scheduler/lease recovery 仍由 `TaskExecutorController` 处理。
4. `active_work.py` 不再从历史 TaskRun 列表构造当前工作；当前工作只来自 `ActiveTurnRegistry`。
5. prompt cache 设计不随旧任务恢复链变化；稳定 prefix 只放稳定指令、工具、环境和输出 schema，历史摘要永远在 volatile 段。

本方案没有未决问题。实施阶段发现与以上结论冲突的代码路径时，以本方案的 authority map 为准，删除旧路径或将其收敛到对应权威层。

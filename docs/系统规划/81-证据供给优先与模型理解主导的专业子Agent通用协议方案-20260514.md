# 81-证据供给优先与模型理解主导的专业子Agent通用协议方案-20260514

## 1. 方案目标

本方案不是 PDF 专项方案。PDF 只是最早暴露问题的能力样本。

真正要修正的是整个主 Agent / 专业子 Agent 协作结构中的通用问题：

- 子 Agent 容易把专业能力结果包装成结论。
- 主 Agent 容易被子 Agent 的结论牵引，而不是基于证据自主理解。
- 系统内部容易用领域规则替代大模型理解。

目标是建立一套跨能力通用的证据供给协议：

- 系统负责把证据取全、组织清楚、标出边界。
- 子 Agent 负责产出领域证据包，不抢最终解释权。
- 主 Agent 负责结合当前用户问题完成理解、裁决与表达。

一句话：

**专业子 Agent 提供领域证据，主 Agent 保留最终理解权。**

## 2. 当前核心问题

### 2.1 问题不是 PDF，而是“子 Agent 结果过度结论化”

PDF 页级抽取中出现的 `transition_title_only`、`toc_like`、`page_structure_missing` 只是一个例子。

同类问题也会出现在其它能力里：

- RAG 子 Agent 可能直接说“资料支持某观点”，但没有把命中文档、相互矛盾证据、时间边界交清楚。
- 表格分析子 Agent 可能直接说“库存不足最严重的是 A 仓”，但没有说明过滤条件、排序指标、缺失值处理。
- 网页 / 实时检索子 Agent 可能直接给“最新结论”，但没有说明来源时间、可信度、是否存在冲突报道。
- 代码分析子 Agent 可能直接给“问题在 X 函数”，但没有说明调用链、复现条件、证据强度。

所以修复方向不能是“给 PDF 写更多规则”，而是要抽象出所有专业能力都遵守的证据协议。

### 2.2 主 Agent 不应学习每个领域的专用规则

主 Agent 是跨领域协调者，不应该在全局 prompt 中吸收 PDF、表格、RAG、网页、代码等专门领域的解释规则。

主 Agent 应该学习的是通用能力：

- 如何消费证据
- 如何区分事实、提示与未知
- 如何判断证据是否足够回答当前问题
- 如何在证据不足时明确缺口

领域知识应主要留在子 Agent 的证据生产侧，或当前任务的短期上下文里。

### 2.3 系统内部规则不应替代模型理解

规则适合做：

- 授权
- 路由
- 数据提取
- 边界标注
- 结构化归档

规则不适合过早替模型完成：

- 用户意图解释
- 多证据综合判断
- 面向用户的结论组织
- 模糊问题的语义裁决

这些应由大模型在充分证据之上完成。

## 3. 通用原则

### 3.1 Evidence First

子 Agent 输出首先是证据包，不是最终答案。

任何专业能力结果都应优先回答：

- 我查到了什么
- 我从哪里查到
- 我如何处理数据
- 哪些内容稳定
- 哪些内容不稳定
- 哪些地方无法确认

### 3.2 Model Judges, System Supplies

系统供应证据，模型完成理解。

子 Agent 可以给建议性解释，但必须标明它只是 hint，不是 hard fact。

### 3.3 Same Contract, Different Domains

不同专业能力可以有不同领域字段，但必须映射到同一个上层消费协议。

统一上层结构：

- `facts`
- `evidence`
- `method`
- `hints`
- `unknowns`
- `confidence`
- `limits`

领域字段只放在 `domain_payload` 中，不污染主 Agent 的通用理解层。

### 3.4 Current Turn First

主 Agent 消费子 Agent 证据时，必须以当前用户问题为裁决中心。

历史摘要、旧结论、旧失败状态，只能作为背景，不能抢当前证据的优先级。

### 3.5 No Global Domain Prompt Pollution

任何领域专用说明都不能进入主 Agent 全局 prompt。

允许存在：

- 通用证据消费协议
- 当前任务态的短期领域说明
- 子 Agent 领域内部提示

不允许存在：

- 把 PDF 专用规则写进主 Agent 全局 prompt
- 把表格分析规则写进主 Agent 全局 prompt
- 把 RAG 排名策略写进主 Agent 全局 prompt

## 4. 通用证据包协议

专业子 Agent 的结果应统一整理为 `AgentEvidencePacket`。

### 4.1 顶层字段

```text
packet_id: 证据包 ID
source_agent_id: 产出该证据包的子 Agent
target_task_id: 该证据包服务的任务
task_goal: 当前委派目标
domain: pdf | rag | table | web | code | memory | other
facts: 可直接依赖的事实
evidence: 证据项列表
method: 本次如何取得证据
hints: 建议性解释，不是硬事实
unknowns: 当前无法确认的部分
limits: 能力边界、数据边界、授权边界
confidence: high | medium | low
relevance: 该证据包与当前用户问题的对齐程度
freshness: 证据的新鲜度或缓存状态
domain_payload: 领域专用结构化数据
```

其中 `facts`、`evidence`、`unknowns` 是核心字段；`hints` 是辅助字段；`domain_payload` 只允许补充领域细节，不允许承载唯一可读结论。

### 4.2 `facts`

`facts` 只放可直接依赖的观测事实。

例子：

- RAG：命中文档标题、片段、来源路径、时间戳
- PDF：页码、稳定文本、段落数量、结构化缺失
- 表格：行数、筛选条件、聚合字段、排序字段
- Web：URL、发布时间、抓取时间、来源名称
- Code：文件路径、函数名、调用关系、测试输出

每条 fact 应尽量具备：

```text
fact_id: 稳定 ID
claim: 事实内容
source_refs: 该事实来自哪些证据项
confidence: high | medium | low
scope: 事实适用范围
```

注意：`facts` 不是最终答案，它只是可被主 Agent 使用的事实材料。

### 4.3 `hints`

`hints` 是建议性解释，用来帮助模型理解，但不能冒充事实。

例子：

- PDF：这页更像标题分隔页
- 表格：这个异常可能由缺失值导致
- RAG：这些资料倾向支持同一结论
- Web：不同来源说法存在时间差
- Code：这个调用链可能是故障入口

`hints` 必须满足两个约束：

1. 不能没有 `facts` 支撑。
2. 不能被上层当作硬事实直接照抄。

推荐结构：

```text
hint_id: 稳定 ID
suggestion: 建议性理解
basis_fact_ids: 支撑该提示的事实
confidence: high | medium | low
```

### 4.4 `unknowns`

`unknowns` 必须明确列出。

例子：

- 章节边界没有稳定命中
- 表格缺少某一列
- 网页来源时间不明确
- 检索结果存在冲突
- 代码路径未被测试覆盖

`unknowns` 不是失败话术，而是主 Agent 的决策输入。

推荐结构：

```text
unknown_id: 稳定 ID
description: 当前无法确认的内容
impact: 是否影响回答当前问题
next_step: 如果要消除该未知，需要什么证据
```

### 4.5 `method`

`method` 告诉主 Agent 证据是怎么来的。

它不需要暴露内部实现细节，但需要说明足够的语义过程：

- 检索了哪些来源类型
- 使用了什么筛选条件
- 读取了哪些页或哪些表
- 是否做了聚合、排序、过滤
- 是否用了缓存或历史绑定

`method` 的目标不是暴露内部实现，而是让主 Agent 理解证据的形成过程。例如“按库存缺口降序排序”比“调用 structured_data_worker”更有用。

### 4.6 `evidence`

`evidence` 是证据项列表，负责把 facts 落回来源。

推荐结构：

```text
evidence_id: 稳定 ID
kind: document_chunk | pdf_page | table_rows | web_page | code_span | log | memory
source: 来源对象
locator: 页码、行号、URL、表名、查询条件等定位信息
text_or_value: 可供模型消费的证据内容
confidence: high | medium | low
visibility: model_visible | debug_only
```

如果证据不能直接展示给模型，应标记为 `debug_only`，不能混入主 Agent 的可见上下文。

### 4.7 `limits`

`limits` 表示能力边界，不等于错误。

例子：

- PDF 页级结构缺失
- 表格字段不存在
- Web 来源无法访问
- RAG 未命中特定时间范围
- 代码测试没有覆盖相关路径

主 Agent 应根据 `limits` 决定是否继续补证、保留边界，或直接回答。

### 4.8 `relevance`

`relevance` 解决一个常见问题：子 Agent 产出的证据可能是真的，但不一定回答当前问题。

推荐结构：

```text
score: 0.0 - 1.0
reason: 为什么认为与当前问题相关或不相关
matched_goal_terms: 命中的目标要素
missing_goal_terms: 未覆盖的目标要素
```

当 `relevance.score` 较低时，主 Agent 不应把该证据包直接用于最终答案。

## 5. 各能力如何映射到通用协议

### 5.1 RAG 子 Agent

RAG 不应直接输出“答案一定是 X”。

它应输出：

- `facts`: 命中的证据片段与来源
- `evidence`: 文档标题、路径、分数、片段
- `method`: 检索词、top_k、过滤条件
- `hints`: 资料共同倾向、冲突线索
- `unknowns`: 未检索到的主题、来源缺失、时间边界
- `domain_payload`: retrieval hits, scores, source graph

### 5.2 PDF 子 Agent

PDF 不应直接抢先替主 Agent 解释“这一页是什么”。

它应输出：

- `facts`: 页码、稳定文本、正文段落数、结构块数量
- `evidence`: 页级片段、可读文本、页引用
- `method`: 读取页范围、章节匹配方式
- `hints`: 更像目录页、标题页、正文页等建议性提示
- `unknowns`: OCR 不稳定、结构化缺失、章节边界不明
- `domain_payload`: pages, page_features, section_candidates

### 5.3 表格分析子 Agent

表格分析不应只给最终排名或结论。

它应输出：

- `facts`: 数据源、行列数、字段、筛选条件、聚合结果
- `evidence`: 关键行、汇总表、异常值
- `method`: 过滤、分组、排序、计算逻辑
- `hints`: 数据异常可能原因、业务解释建议
- `unknowns`: 缺失列、缺失值、无法验证的业务定义
- `domain_payload`: dataframe schema, query plan, computed rows

### 5.4 Web / 实时检索子 Agent

实时检索不应只给“最新结果”。

它应输出：

- `facts`: 来源、发布时间、抓取时间、核心事实
- `evidence`: URL、标题、片段、来源类型
- `method`: 查询词、时间范围、筛选逻辑
- `hints`: 来源之间是否一致、是否存在更新风险
- `unknowns`: 无法访问、付费墙、发布时间不明
- `domain_payload`: web hits, freshness, source reliability

### 5.5 代码分析子 Agent

代码分析不应只给“bug 在这里”。

它应输出：

- `facts`: 文件、函数、调用链、测试结果
- `evidence`: 代码片段、日志、异常栈、失败测试
- `method`: 搜索路径、运行命令、复现步骤
- `hints`: 可能根因、风险区域
- `unknowns`: 未覆盖路径、未跑的测试、依赖环境缺口
- `domain_payload`: symbols, references, diagnostics

## 6. 主 Agent 通用消费协议

主 Agent 不学习某个领域的专用规则。

它只学习如何消费 `AgentEvidencePacket`：

1. 先读 `task_goal`，确认证据是否服务当前问题。
2. 先依赖 `facts`，不要把 `hints` 当成事实。
3. 用 `hints` 帮助组织理解，但必须保留边界。
4. 检查 `unknowns` 是否影响最终回答。
5. 检查 `confidence` 是否足够直接收口。
6. 如果证据足够，直接回答用户。
7. 如果证据不足，明确缺什么，不把缺口包装成结论。

这套协议对 PDF、RAG、表格、Web、代码都成立。

## 7. Prompt 装配策略

### 7.1 全局常驻层

主 Agent 全局 prompt 只能保留跨领域原则：

```text
你会收到来自专业子 Agent 的证据包。
证据包中的 facts 是可依赖事实，hints 是建议性解释，unknowns 是尚未确认内容。
你负责结合当前用户问题完成最终理解、裁决和表达。
不要把 hints 当成 facts。
证据足够时直接回答；证据不足时说明缺口。
```

这段是通用的，可以常驻。

### 7.2 任务态领域说明

领域说明只在当前任务态注入，不进入主 Agent 全局 prompt。

例子：

- 当前是 PDF 页级任务，`domain_payload.page_features` 表示页面抽取特征。
- 当前是表格任务，`domain_payload.query_plan` 表示计算过程。
- 当前是 RAG 任务，`domain_payload.hits` 表示检索命中。

这类说明必须短、临时、绑定当前任务。

### 7.3 子 Agent 内部提示

领域知识应主要留在子 Agent 内部。

例如 PDF 子 Agent 可以知道如何抽页、如何识别章节候选；表格子 Agent 可以知道如何做聚合和排序；RAG 子 Agent 可以知道如何整理检索命中。

但它们交给主 Agent 的结果必须回到通用证据协议。

## 8. 运行时装配规则

### 8.1 子 Agent 返回结果后先标准化

所有子 Agent 原始结果必须先经过统一标准化：

```text
raw_child_result -> AgentEvidencePacket -> main_agent_context
```

主 Agent 不应直接消费各领域散乱格式。

### 8.2 领域字段只进入 `domain_payload`

主 Agent 通用层只看统一字段。

领域特化字段只作为补充证据，不改变主 Agent 的全局行为。

### 8.3 主 Agent 最终回答必须绑定当前问题

子 Agent 的结果可能正确，但不一定回答当前用户问题。

因此主 Agent 最终收口前必须做一次当前问题对齐：

- 证据是否回答当前问题
- 证据是否来自当前目标对象
- 是否有旧任务残留混入
- 是否需要继续补证

### 8.4 与现有结构的映射

当前代码里已经存在若干相近结构，不应另起一套完全孤立的数据模型。

建议映射如下：

```text
EvidenceEnvelope -> AgentEvidencePacket.evidence
CanonicalResult.answer -> 兼容性 summary，不作为主消费入口
CanonicalResult.diagnostics -> AgentEvidencePacket.domain_payload / method
CanonicalResult.degraded_reason_typed -> AgentEvidencePacket.limits
AgentDelegationResult.summary -> 面向旧链路的压缩说明
AgentDelegationResult.answer_candidate -> 兼容性候选答案
AgentDelegationResult.diagnostics -> 存放 AgentEvidencePacket
```

迁移期内允许保留 `summary` 和 `answer_candidate`，但主 Agent 新消费链应优先读取 `diagnostics.agent_evidence_packet`。

### 8.5 标准化器

需要增加一个统一标准化器，负责把不同专业能力结果转成 `AgentEvidencePacket`：

```text
PDF canonical result -> AgentEvidencePacket
RAG evidence envelope -> AgentEvidencePacket
Structured data canonical result -> AgentEvidencePacket
Web search result -> AgentEvidencePacket
Code diagnostic result -> AgentEvidencePacket
```

标准化器必须做三件事：

1. 抽取可依赖事实。
2. 分离建议性解释。
3. 标记未知项和能力边界。

### 8.6 主 Agent 可见内容压缩

完整证据包可能很大，因此进入主 Agent prompt 的内容应分两层：

- `visible_packet_summary`: 短证据摘要，默认可见
- `packet_refs`: 指向完整证据的引用，必要时展开

默认可见内容必须包括：

- 当前目标
- 最关键 facts
- 关键 unknowns
- confidence
- relevance

领域细节放在 refs 中，不默认灌满上下文。

### 8.7 主 Agent 消费状态机

主 Agent 收到证据包后，不应马上生成最终答案，而应在内部完成一次轻量裁决：

```text
1. 目标对齐：证据包是否服务当前问题
2. 证据充分性：facts 是否足够回答
3. 未知项影响：unknowns 是否阻断回答
4. hint 使用：hints 是否只能作为解释辅助
5. 收口决策：回答、补证、澄清、拒绝越界
```

这个状态机是通用的，不包含任何 PDF、表格或 RAG 专用规则。

## 9. 测试策略

测试不能只看某个 PDF 场景是否通过。

必须覆盖跨能力一致性：

1. PDF 页级任务：子 Agent 输出 facts/hints/unknowns，主 Agent 能自然判断。
2. RAG 任务：子 Agent 输出命中证据和冲突边界，主 Agent 不把检索摘要当绝对事实。
3. 表格任务：子 Agent 输出计算方法和结果，主 Agent 不省略关键计算条件。
4. Web 任务：子 Agent 输出来源时间和抓取时间，主 Agent 能处理时效边界。
5. 代码任务：子 Agent 输出诊断证据和未验证路径，主 Agent 不把疑似根因当确定根因。

## 10. 风险与控制

### 风险 1：统一协议过抽象，领域信息损失

控制方式：

- 顶层字段保持统一
- 领域细节保留在 `domain_payload`
- 主 Agent 可按需读取领域细节，但不被领域规则塑形

### 风险 2：主 Agent 回答变保守

控制方式：

- `hints` 不取消，只降级为建议性解释
- `confidence` 帮助主 Agent 决定是否直接下判断
- 对高置信证据允许直接收口

### 风险 3：子 Agent 输出冗长

控制方式：

- 同时提供短摘要和完整 evidence
- 默认注入主 Agent 的是压缩证据包
- 完整证据通过 refs 或 artifact 按需展开

### 风险 4：领域 prompt 污染主 Agent

控制方式：

- 领域说明只能 task-scoped
- 主 Agent 全局 prompt 只保留证据消费原则
- 回归测试加入非目标领域场景，防止 PDF 逻辑影响表格/RAG/Web

## 11. 实施顺序

### Phase 0. 协议落位，不改变行为

先定义 `AgentEvidencePacket` 数据结构和标准化器接口，但不改变主 Agent 现有消费路径。

目标：

- 新结构能生成
- 旧链路继续工作
- 测试只验证新结构存在和字段正确

### Phase 1. 接入现有三个专业能力

优先接入：

- RAG
- PDF
- 表格分析

这三个能力正好覆盖文本检索、文档结构、结构化数据。

每个能力都先做到：

- 生成 `facts`
- 生成 `evidence`
- 生成 `unknowns`
- 保留旧 `summary`

### Phase 2. 主 Agent 影子消费

主 Agent 仍然按旧 summary 回答，但运行时同时记录一份“如果按 AgentEvidencePacket 消费会如何决策”的影子结果。

目标：

- 对比旧答案和新证据裁决
- 找出证据包缺字段的问题
- 不冒险影响真实回答

### Phase 3. 主 Agent 优先消费证据包

当三类能力的证据包稳定后，主 Agent 新链路改为：

```text
AgentEvidencePacket -> visible_packet_summary -> 主 Agent 裁决 -> 最终回答
```

旧 summary 只作为 fallback。

### Phase 4. 增加 task-scoped 领域说明

按需为 PDF、表格、RAG 增加短期任务说明，但禁止写入全局 prompt。

领域说明只解释 `domain_payload` 的字段含义，不提供领域答案模板。

### Phase 5. 做跨能力长测

不只测 PDF。

长测必须包含：

- PDF 页级和章节定位
- RAG 证据冲突
- 表格聚合计算
- Web 时效信息
- 主 Agent 多轮收口

### Phase 6. 清理旧残留

等主 Agent 稳定消费证据包后，清理只服务旧链路的无用字段和测试。

保留兼容字段的唯一理由是仍有真实调用链依赖；如果没有依赖，应果断删除，避免未来又回到 summary 驱动。

## 12. 完成标准

满足以下条件后，认为方案落地成功：

1. 子 Agent 输出统一映射为 `AgentEvidencePacket`。
2. 主 Agent 全局 prompt 不包含任何领域专用规则。
3. 主 Agent 能稳定区分 facts、hints、unknowns。
4. PDF、RAG、表格至少三个能力通过同一消费协议工作。
5. 长测里不再靠领域补丁解决问题，而是通过证据供给和模型理解解决问题。
6. 旧 summary 驱动链路不再是主路径。
7. 非 PDF 场景不会被 PDF 风格或章节逻辑污染。

## 13. 最小落地切片

为了避免一次性改太大，第一轮最小切片建议如下：

1. 新增 `AgentEvidencePacket` 模型。
2. 给 RAG、PDF、表格各写一个 adapter。
3. 在 `ChildAgentRuntimeExecutor` 或 delegation result 构造处，把 packet 放进 diagnostics。
4. 给主 Agent 观察内容增加 `visible_packet_summary`，但不立刻替换旧 summary。
5. 增加三组回归测试：
   - RAG packet 包含来源和未知项
   - PDF packet 区分事实和 hint
   - 表格 packet 包含计算方法和筛选条件

这一切通过后，再进入主 Agent 优先消费证据包。

## 14. 反例边界

以下做法不符合本方案：

- 为 PDF 单独扩写主 Agent 全局 prompt。
- 让表格子 Agent 只返回自然语言结论，不返回计算方法。
- 让 RAG 子 Agent 把检索摘要冒充最终事实。
- 把 `hints` 放进 `facts`。
- 只为了通过长测而在测试中放宽 warning，不修证据链。

## 15. 结论

PDF 的问题提醒我们：不能把每个能力的问题都修成一个领域补丁。

真正要建立的是通用规则：

**子 Agent 产出领域证据，运行时标准化证据包，主 Agent 基于证据完成理解。**

这才是可扩展的多 Agent 协作方式。

## 17. 专业子 Agent 投影接入补充实施

### 17.1 本轮调整目标

长测语义抽检后确认，问题不只是证据包字段粗糙，还包括专业子 Agent 自身没有稳定的专业投影。

当前 RAG、PDF、表格三个内置子 Agent 虽已注册为 worker 子 Agent，但默认投影为空，实际装配时只依赖 agent description 或 MCP 工具结果。这会导致：

- 子 Agent 不像真正的专业阅读员、检索员或数据分析员。
- 子 Agent 容易只返回工具摘要，而不是整理过的专业证据报告。
- 主 Agent 被迫承担太多领域预处理判断。

本轮目标：

1. 给 RAG、PDF、表格三个专业子 Agent 注册独立投影。
2. 将三个内置子 Agent 的默认投影绑定到这些专业投影。
3. 子 Agent fallback 模型装配时必须读取投影系统，而不是只读 description。
4. 不把 PDF、RAG、表格专用规则写进主 Agent 全局 prompt。

### 17.2 投影职责边界

专业子 Agent 的投影不是最终回答模板，而是专业工作身份。

PDF 子 Agent：

- 负责阅读 PDF、定位章节、页面、结论、主题。
- 负责区分正文页、目录页、过渡页、相关页、结论页。
- 负责整理可回答事实、仅作线索的信息、未知和边界。
- 不负责替主 Agent 做面向用户的最终表达。

RAG 子 Agent：

- 负责检索知识库，区分命中内容和目录、索引、定位线索。
- 负责整理来源、片段、冲突、时间边界和证据强弱。
- 不把检索排名直接包装成最终结论。

表格子 Agent：

- 负责读取表格结构，说明筛选、聚合、排序、计算口径。
- 负责把结果、关键行、缺失字段、异常值整理清楚。
- 不省略计算方法，也不把错误维度的排名当作完成结果。

### 17.3 装配规则

本轮只改子 Agent 装配：

```text
AgentDescriptor.default_projection_id
  -> Soul projection card
  -> child Agent fallback system prompt
```

MCP 能力仍负责真实读取与分析；投影负责让需要模型理解的环节具有正确的专业身份。

主 Agent 全局 prompt 不新增 PDF/RAG/表格专用内容。

### 17.4 验收标准

- `agent:rag_analyst` 绑定 RAG 专业投影。
- `agent:pdf_reader` 绑定 PDF 阅读分析投影。
- `agent:table_analyst` 绑定表格分析投影。
- 子 Agent fallback prompt 中能看到投影 identity 和 projection prompt。
- 三个投影 prompt 都是面向模型理解的职业身份说明，不是开发流程说明。
- 回归测试覆盖投影绑定和 fallback prompt 装配。

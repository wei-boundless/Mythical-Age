# 长场景合同重构计划书

日期：2026-05-11

## 目标

重构 `backend/tests/system_eval/long_scenarios.py` 中的长场景合同，解决以下问题：

1. 删除已经失效或不稳定的旧断言，尤其是 `plan.mcp=*` 这类观测面。
2. 把“问得过于简单、抽象、不能定位问题”的回合改成更接近真实工作流的提问。
3. 让每组场景都能稳定覆盖：
   - RAG 检索
   - PDF 跟读
   - 结构化数据切换
   - 实时查询
   - 长期记忆与跨会话回忆
   - 权限边界
   - 多会话隔离
   - 复合任务拆分
4. 让失败结果能更直接地区分：
   - 情景合同问题
   - 活动对象切换问题
   - 工具路由问题
   - 记忆与知识边界问题

---

## 一、现有合同问题

当前长场景合同主要有四类缺陷：

1. 观测字段过期
   - `plan.mcp=*` 在当前 runtime 中经常为空，不能再作为主断言。

2. 断言过度绑定旧样本
   - 例如固定页码、固定仓库名、固定缺口值、固定员工名单。
   - 这类断言容易因为数据更新、分页变化、排序口径变化而失效。

3. 用户提问过于抽象或过短
   - 例如“第三页讲了什么”“按仓库汇总前五”。
   - 这类问题本身允许多种合理解释，不能稳定诊断 runtime 问题。

4. 场景失败后定位信息不足
   - 无法快速区分是：
     - 对象绑定串线；
     - 工具路由错误；
     - 会话记忆失效；
     - 还是场景预期本身写错。

---

## 二、重构原则

### 1. 断言只保留高价值合同

优先保留：

- `event.tool=*`
- `event.mcp=*`
- `plan.tool=*`
- `main.active_pdf.nonempty`
- `main.active_dataset.nonempty`
- `followup.*`
- `response.nonempty`
- `response.not_contains_any=*`
- 少量稳定的 `response.contains_any=*`

弱化或删除：

- `plan.mcp=*`
- 对具体页码内容、固定数值、固定人名的强绑定断言

### 2. 提问改成真实工作动作

提问要体现明确意图，例如：

- 不再写“第三页讲了什么”
- 改成“第三页如果是目录或过渡页，请告诉我它在全文里承担什么作用”

- 不再写“按仓库汇总前五”
- 改成“按缺口总量汇总仓库，并给我缺口最高的前三个”

### 3. 长场景要按工作流分段

六十轮长跑按以下段落组织：

1. 知识库研究启动
2. PDF 深读与结论压缩
3. 数据分析与对象切换
4. 实时信息接入
5. 长期记忆写入与回忆
6. 复合任务拆分
7. 权限边界切换
8. 多会话并行与恢复
9. 最终总复盘

### 4. 每段都要能诊断结构问题

例如：

- PDF -> 数据集切换：诊断对象绑定
- 数据集 -> 实时查询：诊断工具路由
- 回忆偏好：诊断 durable/session memory 边界
- 多会话来回切换：诊断 session isolation

---

## 三、实施范围

本轮实施包括：

1. 重写 `backend/tests/system_eval/long_scenarios.py`
2. 保持 `long_scenarios_regression.py` 通过
3. 如有必要，小幅扩展 `long_runner.py` 的断言能力

本轮不做：

1. runtime 主链修复
2. RAG 参数调整
3. 具体 MCP 工具实现改造

---

## 四、交付标准

完成后应满足：

1. 长场景问题更接近真实用户工作流
2. 场景断言不再依赖明显过期字段
3. 六十轮主场景仍保持 `>= 60` turn
4. `backend/tests/system_eval/long_scenarios_regression.py` 通过
5. 场景失败时能更清楚地定位是合同问题还是 runtime 问题

---

## 五、执行步骤

1. 重新组织各个基础场景的目标与提问
2. 重写六十轮主场景，去掉低价值旧断言
3. 必要时补充 runner 可识别的新稳定断言
4. 运行长场景合同回归测试
5. 根据结果微调合同措辞与断言颗粒度

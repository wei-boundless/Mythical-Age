# 33-长篇小说任务能力前端配置与多 Agent 拓扑设计计划书

日期：2026-05-09

## 0. 边界声明

本计划只做“长篇小说任务能力”的设计与实施规划。

硬约束：

1. 不允许绕过前端工作台私下注册 Agent、AgentGroup、ContractSpec、TaskGraph、CoordinationTask、TopologyTemplate、CommunicationProtocol、RuntimeProfile。
2. 所有能力必须通过现有前端管理入口创建、编辑、保存、预览与验收。
3. 后端只能补齐“前端无法表达但系统需要表达”的通用字段、校验、预览与模板能力。
4. 不允许写死某一本小说、某个作者、某种题材到模型顶层。
5. 长篇小说字段只能作为 task_family / kind / contract / metadata / working memory policy 的配置出现。
6. 工作记忆、任务长期记忆、通信事务、RunLoop 必须沿用 31 号计划已经建立的治理路径。

本计划不做：

- 不直接修改 storage JSON 注册长篇小说 Agent。
- 不在后端 default registry 中新增固定小说团队。
- 不引入新记忆层。
- 不绕开 TaskSystemView / OrchestrationView / MemoryView。
- 不把长篇小说任务资产写入 Global Durable Memory。

### 1.3 当前表达能力

Agent 可表达：

- 角色名与说明。
- Agent 类别。
- 默认 soul / projection。
- task_scope。
- runtime profile。
- allowed operations。
- allowed memory scopes。
- allowed context sections。
- output contracts。
- metadata。

边界修正：

- Agent 本体只保存身份、职责摘要、专属投影绑定、权限与运行档案。
- 具体 Agent 设定 prompt 由投影系统承载，不写入 Agent metadata。
- RuntimeAssembly / RunLoop 通过投影生成 `projection_ref` 与 `prompt_manifest_ref`，再进入节点执行上下文。
- 任务图节点可以使用 `projection_overlay_id` 做任务期覆盖，但不能替代 Agent 专属投影卡。

ContractSpec 可表达：

- input_fields。
- output_fields。
- artifact_requirements。
- acceptance_rules。
- runtime_requirements。
- context_visibility_policy。
- handoff_policy。
- failure_policy。
- human_gate_policy。
- allowed_agent_kinds。
- allowed_runtime_lanes。

TaskGraph 可表达：

- single_agent / multi_agent / coordination。
- node。
- edge。
- node agent binding。
- node contract binding。
- edge handoff contract binding。
- working_memory_policy_profile_id。
- working_memory_policy。
- node memory_read_policy。
- node memory_writeback_policy。
- node dynamic_memory_read_policy。
- edge working_memory_handoff_policy。

RunLoop / RuntimeAssembly 可表达：

- node runtime assembly。
- working memory sections。
- task durable memory sections。
- runtime contracts。
- handoff packets。
- context section filtering by AgentRuntimeProfile。

### 1.4 当前明显缺口

缺口 A：前端没有“长篇小说任务包向导”

现在可以手工配置，但配置项分散在 Orchestration、TaskSystem、MemoryView 三个页面。长篇小说团队需要一次性创建多 Agent、多 Contract、多节点图、多边通信，纯手工容易漏。

缺口 B：Agent prompt 承载层需要回到投影系统

Agent 当前主要有 description、metadata、projection、runtime profile。长篇小说 Agent 的 system prompt 不应直接放进 Agent metadata，也不应写入后端 registry；它应由新建的长篇小说专属投影卡承载，并通过 Agent 的投影绑定字段连接到 Agent。计划需要补齐的是“专属投影卡创建、绑定、预览、节点覆盖和 RuntimeAssembly 展示”的完整链路，而不是另建 prompt profile 或复用系统默认投影。

缺口 C：ContractSpec 编辑器是 JSON 型强编辑

ContractSpec 已能表达，但长篇小说需要多个契约，纯 JSON 编辑易错。计划先允许以草案 JSON 保存，后续再补“小说任务契约模板生成器”。

缺口 D：RunLoop 对章节循环的执行参数还没有工作台级统一配置

31 号已建立工作记忆和收束机制，但长篇小说需要章节循环参数：

- chapter_count / volume_count。
- per_chapter_stage。
- revise_until_pass。
- memory promotion cadence。
- continuity check cadence。

这些应进入任务图 runtime_policy / working_memory_policy / task policy，不应硬编码。

缺口 E：多 Agent 通信语义还需要长篇任务专用边契约

A2A 协议固定，但业务通信 payload 需要 ContractSpec：

- plan_to_outline_handoff。
- outline_to_chapter_handoff。
- chapter_to_continuity_review_handoff。
- review_to_revision_handoff。
- gate_to_memory_publish_handoff。
- final_merge_handoff。

## 2. 推荐设计方向

### 2.1 一句话方向

用前端工作台创建一个“长篇小说任务包”，该任务包由 AgentGroup、AgentRuntimeProfile、ContractSpec、TaskGraph、CoordinationTask、WorkingMemoryPolicyProfile 共同组成；任务图只负责组织关系，Agent 负责执行角色，ContractSpec 负责输入输出边界，Working Memory / Task Durable 负责连续性与任务资产沉淀。

### 2.2 所有权边界

| 对象 | 所有权 | 创建方式 | 禁止 |
|---|---|---|---|
| 小说 Agent | OrchestrationView | 前端 Agent 表单 | 后端 default registry 写死 |
| Agent 组 | OrchestrationView | 前端 AgentGroup 表单 | storage JSON 手写 |
| Agent runtime profile | OrchestrationView | 前端 runtime profile 表单 | 代码分支特殊判断小说 |
| 契约 | TaskSystemView Contract Library | 前端 ContractSpec 保存 | 私下拼 schema |
| 任务域与具体任务 | TaskSystemView | 前端 task domain / specific task | 后端隐式创建 |
| 任务图 | TaskGraphWorkbench | 前端图编辑器 | 单独维护拓扑 JSON |
| 协调任务 | CoordinationEditorWorkbench | 前端协调任务保存 | 私下创建 coordination task |
| 工作记忆策略 | TaskRunLoopWorkbenchPanel / 图节点策略 | 前端策略配置 | Agent 直接查库 |
| 任务长期记忆 | MemoryView | 前端治理晋升 | 自动写 Global Durable |

### 2.3 不采用的方案

不采用“内置 novel_writer_team”。

原因：

- 违反用户要求的前端配置原则。
- 会把小说团队固化为系统默认能力。
- 后续不同题材、不同流程、不同 Agent 数量不好变更。

不采用“单 Agent 大 prompt 完成整本书”。

原因：

- 无法稳定维护人物、世界观、伏笔、章节连贯。
- 工作记忆、任务长期记忆、审查循环无法发挥作用。
- 失败重试会污染上下文。

不采用“图完全决定执行内容”。

原因：

- 图只应组织节点、边、协议、策略。
- 节点内的执行能力仍由 Agent + Contract + RuntimeProfile 决定。

## 3. 长篇小说目标任务包

### 3.1 Task Domain

建议前端创建：

```text
domain_id = domain.longform_novel
task_family = longform_novel_writing
title = 长篇小说创作
description = 面向多章节、多角色、多设定连续性的长篇小说生产任务域。
```

### 3.2 Specific Task

建议前端创建：

```text
task_id = task.longform_novel.create_full_novel
task_title = 长篇小说完整创作
task_family = longform_novel_writing
task_mode = longform_novel_graph
input_contract_id = contract.novel.project_brief
output_contract_id = contract.novel.final_manuscript_package
default_flow_contract_id = flow.longform_novel.graph_runtime
```

### 3.3 AgentGroup

建议前端创建：

```text
group_id = group.longform_novel_core_team
title = 长篇小说核心创作组
group_kind = coordination_team
coordinator_agent_id = agent:novel_showrunner
member_agent_ids =
  agent:novel_story_architect
  agent:novel_chapter_writer
  agent:novel_continuity_editor
  agent:novel_memory_publisher
```

设计原则：

- 核心团队默认 5 个 Agent，不以堆人覆盖架构问题。
- 世界观、剧情、角色连续性在“故事架构师”中合并，因为这些资产高度耦合，拆成三人会制造过多同步成本。
- 风格编辑并入“章节写作 Agent”的职责，连续性审校只负责指出结构、事实、节奏问题，不直接改正文。
- 记忆整理与最终交付合并为“记忆与交付管理 Agent”，因为二者都围绕 accepted 资产、artifact refs 和 Task Durable 候选工作。
- 后续如任务规模很大，可从核心 Agent 拆出可选扩展 Agent，但默认模板不创建。

## 4. Agent 人员设计与 Prompt 草案

说明：

以下 prompt 是“前端配置文本草案”，不能私下写入注册表。实施时应通过 OrchestrationView 新建长篇小说专属投影卡，由投影卡承载具体 prompt、可见区与节奏约束；Agent 只绑定这些专属投影卡，不保存自己的 prompt 文本，也不复用系统默认投影。若当前前端缺少明确投影编辑入口，则阶段 1 必须先补“投影卡创建 / 编辑 / 绑定 / 预览”。

### 4.1 Showrunner / 总协调

```text
agent_id = agent:novel_showrunner
agent_name = 长篇小说总协调
category = worker_sub_agent
task_scope = longform_novel_writing, longform_novel_graph
default_projection_id = projection.longform_novel.showrunner
allowed_context_sections = task, runtime_contracts, working_memory, task_durable_memory, upstream_outputs, artifact_refs, runtime_trace
allowed_operations = op.model_response, op.memory_read, op.memory_write_candidate, op.artifact_result_ref
```

Prompt：

```text
你是长篇小说任务的 Showrunner。你的职责不是直接写完整章节，而是维护整部作品的创作秩序。

你必须：
1. 把用户的创作目标拆成可执行阶段。
2. 维护卷、章、角色线、世界线、风格线之间的一致性。
3. 调度规划、写作、审查、修订、整理节点。
4. 只采纳经过契约和连续性检查的工作记忆。
5. 对冲突生成明确的返修指令，而不是直接覆盖旧事实。
6. 把任务内长期有效的设定交给 Memory Publisher 形成 Task Durable 候选。

你禁止：
1. 跳过审查直接宣布全书完成。
2. 把章节草稿、人物设定、世界观资产写入 Global Durable。
3. 让下游 Agent 直接把 handoff 文本当作共享事实。

你的输出必须符合当前节点绑定的契约。
```

### 4.2 Story Architect / 故事架构

```text
agent_id = agent:novel_story_architect
agent_name = 故事架构师
default_projection_id = projection.longform_novel.story_architect
allowed_context_sections = task, runtime_contracts, working_memory, task_durable_memory, upstream_outputs
allowed_operations = op.model_response, op.memory_read, op.memory_write_candidate
```

Prompt：

```text
你是长篇小说的故事架构师。你负责把世界观、剧情结构、角色弧线和章节计划整合成可执行的创作蓝图。

你必须：
1. 输出世界规则、角色基线、卷纲、章节目标和关键转折。
2. 每个章节目标都必须能交给 Chapter Writer 执行。
3. 伏笔必须带 foreshadow_track，并声明预计回收位置。
4. 修改剧情时必须说明影响到的人物状态、世界状态和时间线。
5. 把稳定设定写成 working memory candidate，把长期有效资产标记为 promotion_candidate。
6. 避免一次性写过多不可执行的百科。

你禁止：
1. 只写抽象主题而没有章节执行目标。
2. 随意推翻已 accepted 的剧情事实。
3. 绕过 Coordinator 采纳设定。
4. 把世界观、人物、剧情资产直接提交 Global Durable。
```

### 4.3 Chapter Writer / 章节写作

```text
agent_id = agent:novel_chapter_writer
agent_name = 章节写作 Agent
default_projection_id = projection.longform_novel.chapter_writer
allowed_context_sections = task, runtime_contracts, working_memory, task_durable_memory, upstream_outputs, artifact_refs
allowed_operations = op.model_response, op.memory_read, op.memory_write_candidate, op.artifact_result_ref
```

Prompt：

```text
你是长篇小说的章节写作 Agent。你负责根据章节目标、角色状态、世界规则和风格约束写出章节草稿，并在修订轮次中完成正文改写。

你必须：
1. 只围绕当前 node_run 对应章节写作。
2. 遵守 chapter_brief、character_state、world_state、style_constraint 和 revision_instruction。
3. 输出 chapter_draft，并标记本章新增事实。
4. 对新增人物状态、世界状态、伏笔提交候选。
5. 遇到设定矛盾时提交 conflict_hint，而不是自行确认新事实。
6. 修订时只处理 revision_gate 明确要求的部分。

你禁止：
1. 私自改动总大纲。
2. 私自确认新的世界规则。
3. 把草稿当作 accepted working fact。
4. 跨章节读取不在策略允许范围内的私有草稿。
```

### 4.4 Continuity Editor / 连续性审校

```text
agent_id = agent:novel_continuity_editor
agent_name = 连续性审校 Agent
default_projection_id = projection.longform_novel.continuity_editor
allowed_context_sections = task, runtime_contracts, working_memory, task_durable_memory, upstream_outputs, runtime_trace
allowed_operations = op.model_response, op.memory_read, op.memory_write_candidate
```

Prompt：

```text
你是长篇小说的连续性审校 Agent。你的职责是发现事实、人物、世界观、时间线、伏笔和文风节奏上的问题，不直接改写章节正文。

你必须：
1. 检查章节草稿与已 accepted 设定是否冲突。
2. 检查角色状态、世界规则、时间线、伏笔是否自洽。
3. 检查章节节奏、视角和语言风格是否破坏既定风格约束。
4. 输出 evaluator_feedback、continuity_conflict 和 revision_instruction。
5. 不确定时标记 needs_human_review 或 coordinator_decision。

你禁止：
1. 直接改写章节正文。
2. 因个人审美否定契约内合格内容。
3. 把未采纳的审查意见写成 accepted fact。
```

### 4.5 Memory Publisher / 记忆与交付管理

```text
agent_id = agent:novel_memory_publisher
agent_name = 记忆与交付管理 Agent
default_projection_id = projection.longform_novel.memory_publisher
allowed_context_sections = task, runtime_contracts, working_memory, task_durable_memory, upstream_outputs, artifact_refs
allowed_operations = op.model_response, op.memory_read, op.memory_write_candidate, op.artifact_result_ref
```

Prompt：

```text
你是长篇小说任务的记忆与交付管理 Agent。你负责把通过修订决策的工作记忆收束为任务长期记忆候选，并在任务末期整理最终交付包。

你必须：
1. 区分草稿、已采纳事实、冲突、审查意见和长期资产。
2. 将人物设定、世界规则、时间线、风格约束整理为 promotion_candidate。
3. 保留 source_work_memory_ids。
4. 对不稳定草稿标记 archive 或 discard。
5. 默认只晋升 Task Durable，不进入 Global Durable。
6. 最终交付时只收集 accepted / archived / promoted_to_task_durable 的稳定内容。
7. 输出 final_manuscript_package，并标注未解决冲突和未完成章节。

你禁止：
1. 把 chapter_draft 直接当成稳定设定。
2. 丢失来源引用。
3. 把任务资产写入主 Agent 长期记忆。
4. 自行补写缺失章节。
5. 忽略 unresolved conflict。
6. 把私有审查草稿暴露为最终内容。
```

### 4.6 可选扩展 Agent

默认不创建扩展 Agent。只有在项目规模足够大、工作量确实形成瓶颈时，才允许从核心职责中拆出：

| 可选 Agent | 拆出来源 | 触发条件 |
|---|---|---|
| 世界观专员 | Story Architect | 世界规则复杂到需要单独维护设定集 |
| 角色连续性专员 | Story Architect / Continuity Editor | 角色数量极多，审校频繁阻塞 |
| 风格编辑专员 | Chapter Writer / Continuity Editor | 用户明确要求多轮文风精修 |
| 最终整理专员 | Memory Publisher | 最终交付包规模大且需要多格式输出 |

这些扩展 Agent 必须通过前端编排系统创建，不进入默认长篇小说模板。

### 4.7 专属投影卡清单

长篇小说核心团队必须创建 5 张专属投影卡。投影卡是 prompt 的唯一承载层；Agent、任务图、任务包只能引用投影 ID。

| projection_id | 绑定 Agent | 投影节点建议 | prompt 来源 |
|---|---|---|---|
| projection.longform_novel.showrunner | agent:novel_showrunner | 身份、职责、必须、禁止、输出约束 | 4.1 Showrunner prompt |
| projection.longform_novel.story_architect | agent:novel_story_architect | 身份、故事资产职责、稳定事实规则、禁止事项 | 4.2 Story Architect prompt |
| projection.longform_novel.chapter_writer | agent:novel_chapter_writer | 身份、章节写作职责、修订规则、禁止事项 | 4.3 Chapter Writer prompt |
| projection.longform_novel.continuity_editor | agent:novel_continuity_editor | 身份、审校职责、冲突输出规则、禁止事项 | 4.4 Continuity Editor prompt |
| projection.longform_novel.memory_publisher | agent:novel_memory_publisher | 身份、记忆晋升职责、交付整理规则、禁止事项 | 4.5 Memory Publisher prompt |

专属投影规则：

1. 投影卡 `soul_id` 可由用户选择，但 `projection_id` 必须使用 `projection.longform_novel.*` 命名空间。
2. 投影卡节点必须可编辑，不使用系统 primary projection。
3. 投影卡可以沉淀角色 prompt、风格约束、注意力焦点、风险提示。
4. Agent 的 `description` 只写摘要，不复制完整 prompt。
5. RuntimeAssembly 必须暴露 `projection_ref` 与 `prompt_manifest_ref`，用于确认当前节点确实采用专属投影。

## 5. 长篇小说任务拓扑设计稿

### 5.1 总体拓扑

```text
input_brief
  -> story_architecture
  -> chapter_loop
       -> chapter_plan
       -> chapter_draft
       -> continuity_review
       -> revision_gate
       -> memory_publish
  -> final_assembly
```

说明：

- `story_architecture` 统一产出世界观、角色基线、卷纲、章节计划，不再拆成三条长期并行线。
- `chapter_loop` 是 RunLoop 的迭代单元，不在图上为每一章复制节点。
- `revision_gate` 由 Showrunner 执行，负责决定通过、返修、人工确认或终止。
- `memory_publish` 默认后台执行，只处理通过 gate 的资产。

### 5.2 任务图节点

| node_id | Agent | 类型 | 输入契约 | 输出契约 | 工作记忆写入 |
|---|---|---|---|---|---|
| input_brief | showrunner | input | contract.novel.project_brief | contract.novel.project_brief | task_goal |
| story_architecture | story_architect | agent | project_brief | volume_outline | world_state_delta / character_state_delta / foreshadow_track |
| chapter_plan | story_architect | agent | volume_outline | chapter_brief | plan_fragment |
| chapter_draft | chapter_writer | agent | chapter_brief | chapter_draft | chapter_draft / character_state_delta |
| continuity_review | continuity_editor | agent | chapter_draft | continuity_review_report | continuity_conflict / evaluator_feedback / revision_instruction |
| revision_gate | showrunner | coordinator | review reports | revision_decision | retry_guidance / decision_record |
| memory_publish | memory_publisher | background | accepted chapter assets | memory_promotion_batch | promotion_candidate |
| final_assembly | memory_publisher | barrier | accepted assets | final_manuscript_package | artifact_ref |

### 5.3 循环策略

图上不应为每一章静态复制节点。

建议：

```text
static node = chapter_draft
node_run_id = chapter_writer.chapter_{chapter_index}
run_attempt_id = attempt_{attempt_index}
stage_id = draft / revise / polish / accept
```

运行策略：

```text
chapter_loop.max_chapters = user_config.chapter_count
chapter_loop.max_attempts_per_chapter = 3
chapter_loop.revise_until = no_error_conflict
chapter_loop.failure_policy = coordinator_decides
chapter_loop.memory_finalize_per_chapter = true
chapter_loop.task_durable_promotion_cadence = revision_gate_accept
```

### 5.4 完整运行流程

#### 5.4.1 项目初始化

1. `input_brief` 接收用户创作目标，只形成 `task_goal` 和初始约束，不写入 accepted 事实。
2. `story_architecture` 生成 `story_bible_draft`：
   - 世界规则。
   - 角色基线。
   - 分卷结构。
   - 章节计划。
   - 风格约束。
   - 已知风险。
3. `revision_gate` 对项目级架构进行一次轻量校验：
   - 章节目标是否可执行。
   - 角色/世界/剧情之间是否明显冲突。
   - 目标篇幅是否与章节数匹配。
4. 通过后，RunLoop 创建 `chapter_loop_state`，进入章节循环。

#### 5.4.2 单章循环

每章使用同一组静态节点，但必须带不同运行身份：

```text
chapter_index = n
node_run_id = {node_id}.chapter_{n}
run_attempt_id = attempt_{k}
stage_id = plan / draft / review / revise / accept / memory
```

单章流程：

1. `chapter_plan` 读取 accepted 的故事架构、前文摘要、角色状态、世界状态，输出本章 `chapter_brief`。
2. `chapter_draft` 根据 `chapter_brief` 写正文草稿，新增事实只作为候选，不得自动 accepted。
3. `continuity_review` 检查草稿，输出：
   - `passed`。
   - `severity`。
   - `continuity_conflict`。
   - `revision_instruction`。
   - `needs_human_review`。
4. `revision_gate` 根据审校结果做四选一：
   - `accept`：本章通过。
   - `revise`：重新调度 `chapter_draft`，增加 `run_attempt_id`。
   - `manual_review`：暂停章节循环，等待用户或主会话确认。
   - `abort_chapter`：终止当前章并标记 unresolved。
5. `memory_publish` 只处理 `accept` 后的稳定资产，后台整理 Task Durable 候选。
6. RunLoop 更新章节索引，进入下一章。

#### 5.4.3 全书收束

1. 当 `chapter_index == chapter_count` 或用户主动终止时，RunLoop 进入 finalization。
2. `memory_publish` 完成待处理的 Task Durable 候选归档。
3. `final_assembly` 汇合：
   - accepted 章节产物。
   - Task Durable 设定集引用。
   - 未解决冲突。
   - 章节索引。
   - 修订历史摘要。
4. 如果存在 blocking unresolved conflict，则最终交付包必须标记 `incomplete_with_blockers`，不能伪装为完成。

### 5.5 纠错与容错策略

#### 5.5.1 契约级纠错

每个节点输出先做 ContractSpec 校验：

| 错误 | 处理 |
|---|---|
| 缺少必填字段 | 同一节点重试 1 次 |
| JSON/schema 不可解析 | 要求 Agent 仅修复结构，不重写正文 |
| 输出契约不匹配节点契约 | 阻断下游，提交 `contract_mismatch` 到 revision_gate |
| artifact_ref 缺失 | 允许补交引用；补交失败则 manual_review |

#### 5.5.2 内容级纠错

| 问题 | 判定来源 | 处理 |
|---|---|---|
| 角色动机冲突 | continuity_review | `revise` |
| 世界规则冲突 | continuity_review | `revise` 或 `manual_review` |
| 时间线冲突 | continuity_review | `revise` |
| 伏笔断裂 | continuity_review | `revise`，必要时要求 story_architecture 补计划 |
| 文风偏离但事实无冲突 | continuity_review | `revise`，只允许局部改写 |
| 用户目标变化 | 主会话控制面 | 暂停循环，重新运行 story_architecture 的增量规划 |

#### 5.5.3 调度级容错

| 节点 | 失败策略 |
|---|---|
| story_architecture | fail_closed；失败后回到用户确认或重写项目简报 |
| chapter_plan | retry_once；仍失败则 manual_review |
| chapter_draft | 最多 `max_attempts_per_chapter`；超过后 revision_gate 决定跳章或人工介入 |
| continuity_review | fail_closed；审校失败不能进入 accepted |
| revision_gate | coordinator_decides；不能决策时 manual_review |
| memory_publish | background allow_partial；失败不阻断下一章，但 final_assembly 前必须补偿 |
| final_assembly | barrier all_success；缺稳定资产时输出 incomplete package |

#### 5.5.4 记忆污染防护

1. `chapter_draft` 默认是 `draft_artifact`，不能直接进入 Task Durable。
2. 新增事实必须经过 `continuity_review -> revision_gate` 后才可成为 accepted working fact。
3. `memory_publish` 只能读取 accepted refs、review report 和 source_work_memory_ids。
4. 未通过审查的事实只能进入 archive/discard，不进入任务长期记忆。
5. 不同小说项目必须使用不同 `project_id / task_durable_namespace_id`。
6. Global Durable 二次晋升必须人工触发，且只能晋升通用写作偏好或跨任务创作原则。

#### 5.5.5 人工门控

以下情况必须进入人工门控：

- 连续两次修订仍存在 blocking conflict。
- world rule 修改会影响已经 accepted 的多章内容。
- 用户目标、题材、视角、结局方向发生变化。
- `memory_publish` 无法判断事实应晋升还是丢弃。
- final package 存在 unresolved blocking issue。

人工门控结果必须写成 `decision_record`，并带 `source_issue_ids`。

#### 5.5.6 幂等与恢复

RunLoop 必须使用以下键保证恢复安全：

```text
task_run_id
graph_run_id
chapter_index
node_run_id
run_attempt_id
handoff_transaction_id
source_message_hash
artifact_ref
```

规则：

- 同一 `source_message_hash` 重放时，必须命中同一 handoff transaction。
- 同一 `run_attempt_id` 的输出不能被重复写入 Working Memory。
- 恢复后从最近 checkpoint 继续，不重新 accepted 已通过章节。
- 父任务 abort 时，后台 `memory_publish` 默认 kill；已经生成的候选保持 pending_review。

### 5.6 拓扑验收标准

长篇小说拓扑进入可运行状态前必须满足：

1. 每个 Agent 节点绑定 Agent。
2. 每个 Agent 有 RuntimeProfile，且允许 `longform_novel_writing`。
3. 每个节点有 output contract。
4. 每条边有 payload contract、ack_policy、working_memory_handoff_policy。
5. `chapter_loop.max_attempts_per_chapter >= 1`。
6. `revision_gate` 存在并且 join_policy 为 `coordinator_decides`。
7. `memory_publish` 为 background，且 `blocks_downstream = false`。
8. `final_assembly` 为 barrier。
9. runtime_policy 声明 checkpoint、resume、manual_gate、memory_quarantine 策略。
10. 不存在静态返修回边；返修必须由 RunLoop 重新调度节点。

### 5.7 同步 / 异步运行关系

长篇小说任务不能只靠图边表达执行顺序。图只负责组织节点关系，RunLoop 负责调度章节循环，AgentRuntimeProfile 负责权限边界，工作记忆负责任务内状态连续性。

节点运行语义如下：

| node_id | execution_mode | wait_policy | join_policy | 同步 / 异步含义 |
|---|---|---|---|---|
| input_brief | sync | wait_all_upstream_completed | all_success | 主会话或任务入口同步形成项目简报，不启动后台分支 |
| story_architecture | sync | wait_all_upstream_completed | all_success | 在章节循环前同步完成，作为后续章节计划的稳定输入 |
| chapter_plan | sync | wait_all_upstream_completed | all_success | 每章开始时同步生成本章简报，下游写作必须等待 |
| chapter_draft | sync | wait_all_upstream_completed | all_success | 当前章当前 attempt 的正文生成，失败由 RunLoop 增加 run_attempt_id |
| continuity_review | sync | wait_all_upstream_completed | fail_on_any_error | 审校失败不能进入 revision_gate 的 accept 路径 |
| revision_gate | barrier | wait_all_upstream_completed | coordinator_decides | 汇合审校结果，决定 accept / revise / manual_review / abort_chapter |
| memory_publish | background | wait_all_upstream_completed | allow_partial_with_issues | 只在 revision_gate accept 后后台整理记忆候选，不阻断下一章 |
| final_assembly | barrier | wait_all_upstream_completed | all_success | 全书收束时汇合稳定章节、任务长期记忆候选和未解决冲突 |

关键规则：

1. `memory_publish` 是唯一默认后台节点，但它只能处理已通过 gate 的 refs。
2. `chapter_draft` 的返修不是图上的静态回边，而是 RunLoop 以新 `run_attempt_id` 重新调度。
3. `final_assembly` 是全书级 barrier，不参与单章循环。
4. 后台节点失败只能产生 pending_review，不得静默污染 Task Durable。
5. 同步节点之间必须通过 contract payload + refs 交接，不能共享 raw messages。

### 5.8 节点信息交流策略

通信边必须表达“传什么、不传什么、是否等待、失败如何传播”。

| edge_id | 同步语义 | payload contract | 可传递 refs | ack | 失败传播 | 禁止传递 |
|---|---|---|---|---|---|---|
| e_brief_architecture | sync_handoff_before_target_start | contract.novel.project_brief | task_goal | required_before_target_start | fail_downstream | raw main session、未整理用户长上下文 |
| e_architecture_chapter | sync_handoff_before_target_start | contract.novel.volume_outline | plan_fragment、foreshadow_track、world_state_delta、character_state_delta | required_before_target_start | fail_downstream | 未通过 gate 的世界观覆盖 |
| e_chapter_plan_draft | sync_handoff_before_target_start | contract.novel.chapter_brief | chapter_brief、style_constraint、accepted_refs | required_before_target_start | fail_downstream | 全量设定库 dump、私有审校草稿 |
| e_draft_review | sync_handoff_before_target_start | contract.novel.chapter_draft | chapter_draft、character_state_delta、world_state_delta | required_before_target_start | fail_downstream | 将 draft 标成 accepted fact |
| e_review_gate | sync_handoff_before_target_start | contract.novel.continuity_review | continuity_conflict、evaluator_feedback、revision_instruction | required_before_target_start | coordinator_decides | 审校 Agent 直接改写后的正文 |
| e_gate_memory | async_background_after_gate | contract.novel.continuity_review | accepted_refs、decision_record、retry_guidance | required_before_target_start | allow_partial | 未 accept 的新增事实 |
| e_memory_final | barrier_wait_for_stable_refs | contract.novel.memory_promotion_batch | promotion_candidate、task_durable_refs、unresolved_conflict | required_before_target_start | coordinator_decides | pending_review 伪装成最终资产 |

所有边默认使用：

```json
{
  "a2a_message_type": "message/send",
  "result_delivery_policy": "contract_payload_and_refs",
  "context_filter_policy": {
    "include_raw_messages": false,
    "include_private_memory": false,
    "prefer_refs": true
  },
  "working_memory_handoff_policy": {
    "summary_only": true,
    "prefer_accepted_items": true,
    "reject_unaccepted_facts": true,
    "quarantine_unaccepted_facts": true
  }
}
```

### 5.9 Agent 权限与能力矩阵

这些权限必须通过 Orchestration 前端配置到 AgentRuntimeProfile，不允许后端私下补注册。

| Agent | task_scope | allowed_operations | allowed_memory_scopes | output_contracts | required capabilities | forbidden actions |
|---|---|---|---|---|---|---|
| Showrunner | longform_novel_writing | op.model_response, op.memory_read, op.memory_write_candidate, op.artifact_result_ref | working_memory.task_read, working_memory.graph_read_write, task_durable.read_candidate | project_brief, project_plan | coordination.decision_gate, contract.validation, runloop.chapter_control | write_global_durable, direct_accept_chapter_draft_without_review |
| Story Architect | longform_novel_writing | op.model_response, op.memory_read, op.memory_write_candidate | working_memory.task_read, working_memory.graph_read_write, task_durable.read_candidate | volume_outline, chapter_brief, world_bible_delta, character_delta | story.planning, world_bible.delta, chapter_briefing | overwrite_accepted_fact_without_gate, write_global_durable |
| Chapter Writer | longform_novel_writing | op.model_response, op.memory_read, op.memory_write_candidate, op.artifact_result_ref | working_memory.handoff_read, working_memory.node_write, artifact.write_ref | chapter_draft | chapter.drafting, revision.apply_instruction, artifact.ref_output | accept_own_draft, write_task_durable, write_global_durable |
| Continuity Editor | longform_novel_writing | op.model_response, op.memory_read, op.memory_write_candidate | working_memory.task_read, working_memory.graph_read_write, working_memory.edge_write | continuity_review | continuity.review, conflict.detect, revision.instruction | rewrite_chapter_text, accept_memory_promotion, write_global_durable |
| Memory Publisher | longform_novel_writing | op.model_response, op.memory_read, op.memory_write_candidate, op.artifact_result_ref | working_memory.accepted_read, task_durable.write_candidate, artifact.read_write_ref | memory_promotion_batch, final_manuscript_package | memory.promotion_batch, task_durable.candidate_publish, final.package_assembly | promote_unaccepted_draft, write_global_durable_without_manual_secondary_promotion |

说明：

- `allowed_memory_scopes` 是运行权限，不等于图节点实际可读范围；节点还必须受 `memory_read_policy` 限制。
- `task_durable.write_candidate` 只允许提交候选，不代表直接写入任务长期记忆。
- Chapter Writer 可以写 artifact refs，但不能把正文草稿晋升为稳定事实。
- Continuity Editor 可以读 draft refs 并写 conflict / instruction，但不能直接替换正文。

## 6. 通信设计稿

A2A 协议固定，图边只配置业务语义、payload contract 和 working memory handoff policy。

### 6.1 通信边

| edge_id | source | target | message_type | contract | handoff memory |
|---|---|---|---|---|---|
| e_brief_architecture | input_brief | story_architecture | message/send | contract.novel.project_brief | task_goal |
| e_architecture_chapter | story_architecture | chapter_plan | message/send | contract.novel.volume_outline | plan_fragment, foreshadow_track |
| e_chapter_plan_draft | chapter_plan | chapter_draft | message/send | contract.novel.chapter_brief_handoff | chapter_brief, style_constraint |
| e_draft_review | chapter_draft | continuity_review | message/send | contract.novel.chapter_review_handoff | chapter_draft, character_state_delta |
| e_review_gate | continuity_review | revision_gate | message/send | contract.novel.review_to_revision_handoff | continuity_conflict, evaluator_feedback |
| e_gate_memory | revision_gate | memory_publish | message/send | contract.novel.continuity_review_report | accepted chapter refs, retry_guidance |
| e_memory_final | memory_publish | final_assembly | message/send | contract.novel.memory_to_final_handoff | promotion_candidate, task_durable refs |

返修说明：

- `revision_gate -> chapter_draft` 不建议作为静态图边常驻，以免普通 DAG 预检误判循环。


### 8.2 节点读写策略

实际模板必须按节点配置，不能把所有 Agent 共用一套记忆策略。

| node_id | 读取 | 写入 | 动态读取 | 晋升权 |
|---|---|---|---|---|
| input_brief | 不读任务记忆 | task_goal、decision_record | 禁用 | 无 |
| story_architecture | task_goal、decision_record、style_constraint、世界观/角色/伏笔稳定项 | plan_fragment、decision_record、world_bible_delta、character_state_delta、foreshadow_track、style_constraint | 最多 4 次，允许 temporal 扩展 | 只提交候选 |
| chapter_plan | plan_fragment、decision_record、世界观/角色/风格/伏笔、artifact_ref | chapter_brief、decision_record、foreshadow_track | 最多 4 次，允许 temporal 扩展 | 只提交候选 |
| chapter_draft | chapter_brief、decision_record、世界观/角色/风格/伏笔、retry_guidance | chapter_draft、角色/世界状态候选、artifact_ref | 最多 3 次，允许 temporal 扩展 | 无 |
| continuity_review | chapter_draft、稳定设定、角色状态、时间线、artifact_ref | continuity_conflict、evaluator_feedback、revision_instruction | 最多 5 次，允许 temporal 扩展 | 无 |
| revision_gate | 冲突、审查意见、修订指令、草稿引用、稳定事实 | decision_record、retry_guidance | 最多 2 次，禁用 temporal 扩展 | 可接受工作事实，不直接晋升 Task Durable |
| memory_publish | accepted refs、decision_record、稳定状态、冲突记录、artifact_ref | promotion_candidate、artifact_ref | 最多 4 次，允许 temporal 扩展 | 只提交 Task Durable 候选 |
| final_assembly | promotion_candidate、artifact_ref、decision_record、unresolved conflict | final artifact_ref、decision_record | 最多 2 次，禁用 temporal 扩展 | 无 |

- continuity_conflict 不进入 required context。

- Task Durable namespace 隔离不同小说项目。
- Global Durable 不出现 chapter_draft / character_state_delta / world_state_delta，除非人工二次晋升且 kind 合法。

### 12.6 RunLoop

- checkpoint 带 working_memory_refs。
- chapter loop 参数在 runtime_policy 中可见。
- 同一静态节点不同 node_run_id 不串读。
- retry_guidance 绑定 run_attempt_id。
- finalization 后 draft 不进任务长期记忆。

# 105-TaskGraph阶段自动续推桥修复计划

## 0. 问题定义

当前 TaskGraph 协调任务已经具备：

- 图启动；
- 阶段节点真实执行；
- 协调 checkpoint 更新；
- live monitor 实时读取当前节点状态。

但仍存在一个结构性缺口：

- 当前阶段执行完成后，下一阶段不会自动继续投递；
- 需要人工调用 `continue-current-stage` 才能推进图任务；
- 这会让长链路任务停在阶段边界，表现为“图启动了、监控也亮了，但任务不会自然流转”。

这不是前端问题，也不是 live monitor 问题，而是协调执行桥问题。

## 1. 结构性判断

经审查，当前系统中并存两套阶段续推机制：

1. 新链路
   - `LangGraphCoordinationRuntimeResult.stage_execution_request`
   - `LangGraphCoordinationRuntimeResult.continuation_payload()`
   - `continue-current-stage` API 走的就是这条链，因此人工继续时可以推进。

2. 旧链路
   - `next_task_ref`
   - `_build_continuous_delivery_payload(...)`
   - 该链路使用旧式的“任务引用 + turn context 拼装”方式继续流转。

初步判断认为：

- LangGraph 运行时已经以 `stage_execution_request` 作为正式下一跳语义；
- 自动续推路径里仍残留旧式 `next_task_ref` 逻辑；
- 人工继续和自动继续的语义源可能不一致。

但在进一步真实追踪后，发现需要修正判断：

1. 自动续推链本体并未整体失效。
   - 真实 session 中已经自然生成多个后续阶段子 `task_run`；
   - 说明多阶段协调任务可以继续向后推进，并非完全停在 `world_designer_a`。

2. 真正稳定复现的问题有两个：
   - 非法 `session_id` 会导致后台首阶段执行直接失败；
   - `session_live` 监控视图在 root `task_run` 更新时可能丢失 coordination 指针，导致观察结论偏差。

因此本计划的实际收口目标调整为：

- 修复 TaskGraph 启动入口的 `session_id` 边界；
- 修复 `session_live` 视图与协调指针同步；
- 保留对旧续推链路的审计，但不再把“自动续推整体失效”作为本次已证实的问题。

## 2. 修复目标

本次修复必须做到：

1. TaskGraph start API 对 `session_id` 做前置校验，不再允许后台首阶段因为非法 session 失败。
2. `runtime_views/session_live/{session_id}.json` 在 root `task_run` 更新后仍保留 coordination 指针。
3. live monitor 的 session 级按需读取继续以 `session_live` 为入口，不因 root 状态回写而退化。
4. 对自动续推链保持真实验证，不再误判为整体失效。

## 3. 改动边界

重点修改位置：

- `backend/orchestration/runtime_loop/task_run_loop.py`
  - 阶段执行完成后的 `continuation_payload` 生成逻辑
  - 去掉旧 `next_task_ref` 驱动的主续推路径

- `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
  - 保持 `stage_execution_request` / `continuation_payload()` 为正式续推出口

- 必要测试
  - 验证自动续推不依赖手动 API
  - 验证旧 snapshot / monitor 优化不被回退

## 4. 实施策略

### Step 1

确认阶段完成后的正式下一跳来源：

- 以 `resume_from_task_result(...).stage_execution_request` 为准；
- 若没有 `stage_execution_request`，视为本轮无下一跳或已终态。

### Step 2

在 `RuntimeStateIndex` 中修复 `session_live` 视图同步：

- root `task_run` 更新时，不覆盖已有 `latest_coordination_task_run_id`；
- root `task_run` 更新时，不覆盖已有 `latest_coordination_run_id`；
- 如当前视图缺失，允许从最新索引反推 coordination 指针。

### Step 3

验证：

1. 非法 `session_id` 启动图任务时直接返回 `400`；
2. 合法 session 启动图任务后，`session_live` 视图真实落盘；
3. root `task_run` 停止后，`session_live` 文件仍保留 coordination 指针；
4. live monitor 仍能从 session 级视图进入当前协调运行态。

## 5. 完成标准

满足以下条件才算完成：

1. 非法 `session_id` 启动任务被前置拒绝。
2. `session_live` 视图在真实运行与 root 状态更新后仍保留 coordination 指针。
3. `live-monitor` 继续走按需读取链路，且真实返回有效协调态。
4. 回归测试通过，真实验证通过。

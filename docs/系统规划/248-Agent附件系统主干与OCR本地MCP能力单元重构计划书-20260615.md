# 248-Agent 附件系统主干与 OCR 本地 MCP 能力单元重构计划书

日期：2026-06-15  
状态：已实施  
适用范围：聊天附件、Agent runtime 输入资源、OCR 文字识别、MCP 本地能力单元、系统配置台

## 一、问题复盘

上一版 OCR 接入把 `image_analyze/OCR` 当成主干，再反向补聊天上传、粘贴、runtime 传参、配置与测试。这导致架构重心错误：

- 附件能力被设计成 OCR 的附属能力，而不是通用输入资源系统。
- OCR 被做成新增本地 pytesseract 执行链路，没有优先复用项目已有 OCR 接口。
- 上传大小等通用附件约束错误地放入 `image_ocr` 配置。
- 后续接入 PDF、音频、普通文件、视觉模型或其他附件处理器时会被 OCR 结构牵制。

本次重构目标是纠正主干：**附件系统是一等资源入口，OCR 是附件处理器；OCR 同时作为本地注册的 MCP 能力单元，像 RAG 一样进入能力治理体系。**

## 二、现有依据

本项目已经有本地 MCP 能力单元机制：

- `backend/capability_system/mcp/local_registry.py`
  - 当前注册了 `local_mcp:retrieval`、`local_mcp:pdf`、`local_mcp:structured_data`。
  - 每个单元声明 `unit_id`、`route`、`operation_id`、`worker_slot`、`capability_kinds`、`resource_kinds`、`normalization_contract`。
- `backend/capability_system/mcp/server/local_capability_server.py`
  - `LocalCapabilityMCPExecutor` 统一执行本地 MCP 单元。
  - 通过 `OperationGate` 与 `ResourcePolicy` 做权限检查。
  - 通过 `EvidenceOrchestrator` 调度 worker 并返回规范化结果。
- `backend/evidence/retrieval_worker.py`
  - RAG 当前作为 `retrieval` 本地 MCP worker 输出 evidence envelope。
- `backend/capability_system/mcp/local_provider.py`
  - 管理台通过 local provider 列出、inspect、preview permission、call tool。

因此 OCR 不应作为零散工具硬插进 runtime，而应新增为：

```text
local_mcp:image_ocr
route: image_ocr
operation_id: op.mcp_image_ocr
worker_slot: image_ocr_worker
implementation_module: evidence.image_ocr_worker.ImageOCRWorker
```

## 三、目标架构

```text
ChatInput 上传/粘贴
-> AttachmentStore
-> AttachmentResource
-> Runtime TurnInputResources
-> Attachment Processor Registry
   -> image_ocr processor
      -> local MCP unit: image_ocr
         -> existing OCR provider / configured OCR backend
```

### 3.1 附件系统主干

附件系统负责：

- 接收聊天框文件选择与剪贴板粘贴。
- 保存附件到受控 storage。
- 生成稳定的 `attachment_id` 与 workspace-relative `path`。
- 记录 `mime_type`、`size_bytes`、`filename`、`width/height`、`created_at`。
- 将附件作为当前 turn 的资源投影给 runtime。
- 不主动执行 OCR、不把 OCR 结果塞进用户消息。

### 3.2 OCR 处理器

OCR 处理器负责：

- 输入：`attachment_id` 或受控 `path`。
- 限制：只处理图片附件，默认语言 `chi_sim+eng`。
- 输出：文本、语言、置信度或不确定性、限制说明、artifact/evidence refs。
- 执行：优先调用项目已有 OCR 接口，不新增本地 OCR 主实现。

### 3.3 OCR 本地 MCP 单元

OCR 要像 RAG 一样进入本地 MCP 注册：

```python
LocalMCPUnitRecord(
    unit_id="local_mcp:image_ocr",
    name="image_ocr",
    title="图片文字识别",
    route="image_ocr",
    route_aliases=["image_ocr", "attachment_ocr"],
    category="attachment_processing",
    summary="本地图片 OCR 能力单元，负责读取受控图片附件并返回结构化文字识别结果。",
    mcp_id="mcp:attachment:image_ocr",
    operation_id="op.mcp_image_ocr",
    implementation_module="evidence.image_ocr_worker.ImageOCRWorker",
    worker_slot="image_ocr_worker",
    worker_execution_kind="async",
    capability_kinds=["image_ocr", "attachment_text_extraction"],
    answer_source="mcp.image_ocr_local",
    followup_binding_key="active_attachment",
    source_kind="attachment",
    implementation_root="capability_system.capabilities.attachments",
    default_input_modes=["image/png", "image/jpeg", "image/webp", "image/bmp", "image/tiff"],
    default_output_modes=["text/plain", "application/json"],
    request_path_parameter="path",
    request_mode_parameter="mode",
    request_default_mode="ocr",
    resource_kinds=["chat_attachment", "image_attachment", "ocr_text"],
    normalization_contract={
        "mcp_output": "canonical_result_and_evidence",
        "resource_exposure_policy": "explicit_resource",
    },
    tags=["attachment", "image", "ocr", "local_mcp"],
)
```

## 四、配置设计

### 4.1 `attachments` 配置组

通用附件配置从 `image_ocr` 中移出：

```json
{
  "attachments": {
    "enabled": true,
    "max_upload_bytes": 10485760,
    "max_files_per_message": 8,
    "allowed_mime_types": ["image/png", "image/jpeg", "image/webp", "image/bmp", "image/tiff"],
    "storage_relative_dir": "storage/chat_attachments"
  }
}
```

### 4.2 `image_ocr` 配置组

OCR 只保留识别执行相关配置：

```json
{
  "image_ocr": {
    "enabled": true,
    "provider": "rapidocr",
    "default_language": "chi_sim+eng",
    "timeout_seconds": 60,
    "max_text_chars": 12000,
    "mcp_route": "image_ocr"
  }
}
```

当前实现复用项目 PDF OCR 链路已经采用的 RapidOCR；如果后续切换到外部 OCR API，应通过 provider adapter 接入，不改变 `attachment_extract_text` 的工具契约。

## 五、工具与模型暴露策略

模型不直接看到底层 OCR provider，也不直接看到 MCP transport。模型只看到稳定能力：

```text
attachment_extract_text
```

内部执行链：

```text
attachment_extract_text
-> op.mcp_image_ocr
-> local_mcp:image_ocr
-> ImageOCRWorker
-> configured OCR provider
```

这能保持工具面稳定：未来 OCR provider 从现有 API 切到 MCP server、云 OCR 或视觉模型，不影响模型工具契约。

## 六、需要删除或替换的旧实现

除非用户明确要求保留本地 fallback，否则删除上一版新增的本地 pytesseract 主链路：

- 删除 `backend/capability_system/capabilities/image_analysis/ocr_service.py` 中的 pytesseract 执行主逻辑。
- 删除 `requirements.txt` 中仅为该错误主链路新增的 `pytesseract` 依赖。
- 删除/改写保护 pytesseract 行为的测试。
- `image_analyze` 如无外部契约，改名或替换为 `attachment_extract_text`；避免继续使用“视觉分析”命名暗示非 OCR 能力。

保留并重构：

- 聊天框上传/粘贴 UI。
- 附件消息展示。
- 上传 API，但改为通用 Attachment API。
- runtime turn attachments 投影。

## 七、实施步骤

### 阶段 1：附件系统收束

1. 新增 `capability_system.capabilities.attachments`，只保留附件类型常量。
2. `backend/api/chat_attachments.py` 负责最小上传、校验、路径生成与本地存储。
3. 上传大小、允许类型、单轮数量限制改读 `attachments` 配置。
4. `backend/api/chat.py` 只验证附件资源归属，不理解 OCR。

### 阶段 2：OCR 本地 MCP 单元

1. 在 `backend/capability_system/mcp/local_registry.py` 注册 `local_mcp:image_ocr`。
2. 新增 `op.mcp_image_ocr`，权限边界为 read-only / local read / explicit resource。
3. 新增 `ImageOCRWorker`，输入 `MCPRequest`，从 bindings/constraints 中读取附件路径与语言。
4. 将 OCR 输出规范化为 `CanonicalResult` 与 evidence/artifact refs。
5. 更新 `LocalCapabilityMCPExecutor`，注入 `image_ocr_worker`。
6. 更新 `capability_system/mcp/server/server.py`，暴露 `langchain_agent_extract_image_text` 本地 MCP tool。
7. 更新 `LocalMCPProvider` schema，使管理台可 inspect/preview/call。

### 阶段 3：OCR provider 接入

1. 先复用项目已有 RapidOCR 运行能力。
2. `image_ocr.provider` 只决定执行后端，不影响模型工具契约。
3. 当前不新增 provider 抽象层；只有出现第二个真实 OCR 后端时再提取 adapter。

### 阶段 4：工具与配置台调整

1. 系统配置台新增 `attachments` 组。
2. `image_ocr` 组只保留 OCR 执行配置。
3. 工具目录替换为 `attachment_extract_text` 或等价稳定工具名。
4. prompt guidance 明确：只做文字识别，不做图像理解、物体识别、视觉推理。

### 阶段 5：清理旧代码与测试

1. 删除本地 pytesseract 主链路。
2. 删除旧 OCR 中心化测试。
3. 新增结构测试：
   - 附件上传/粘贴生成 AttachmentResource。
   - runtime 只投影附件引用，不注入 OCR 文本。
   - `local_mcp:image_ocr` 出现在 MCP catalog。
   - `op.mcp_image_ocr` 被权限系统识别。
   - `ImageOCRWorker` 真实调用已有 OCR provider adapter。
   - 配置台能保存 `attachments` 与 `image_ocr`。

## 八、验收标准

- 聊天框支持图片上传与直接粘贴。
- 用户只发图片时，默认消息仍可触发“识别图片文字”的意图。
- runtime 中附件是受控资源，不是 base64 prompt 注入。
- OCR 通过本地 MCP 单元执行，catalog 中可见 `local_mcp:image_ocr`。
- OCR 默认语言为 `chi_sim+eng`。
- 关闭 `image_ocr.enabled` 时，OCR 能力明确拒绝执行，但附件上传系统仍可工作。
- 上传大小限制由 `attachments.max_upload_bytes` 控制。
- 无 pytesseract 主链路残留，除非用户明确要求保留 fallback。
- 相关后端 pytest、前端输入测试、TypeScript 检查通过。
- 固定端口联调仍使用：
  - 前端 `http://127.0.0.1:3000`
  - 后端 `http://127.0.0.1:8003`

## 十、实施结论

- 已删除上一版 `image_analyze` / pytesseract 主链路。
- 稳定模型工具名采用 `attachment_extract_text`，不保留旧别名。
- OCR 已注册为 `local_mcp:image_ocr`，operation 为 `op.mcp_image_ocr`。

# 运行监控系统重构计划书 - 2026-05-28

## 1. 背景

当前运行监控系统已经暴露出多个结构性问题：

- 已完成任务仍被前端按当前时间继续计算耗时。
- 旧的 `running` / `waiting_executor` 任务长期残留在运行中列表。
- 任务标题暴露 `task:...`、`taskrun:...`、`turn:...`、`session:...` 等内部 ID。
- 全局监控台、会话监控、任务详情监控的状态解释不一致。
- 前端和后端都在判断任务属于运行中、已完成、失败还是停滞，决策权分散。
- SSE、轮询、详情刷新并发时，旧快照或旧详情存在覆盖当前选择的风险。

这些问题说明当前监控系统不是一个成熟的运行监控模型，而是由后端 summary、前端 projection、UI formatter、store selection 临时拼接出来的展示层。继续做零散补丁会扩大状态漂移，必须重构为统一的监控投影系统。

## 2. 当前链路审查

当前链路大致如下：

```text
TaskRun / EventLog / AgentRunResult
  -> backend/harness/runtime/single_agent_host.py
  -> /api/orchestration/harness/live-monitor
  -> frontend/src/lib/runtimeWorkProjection.ts
  -> frontend/src/lib/store/runtime.ts
  -> frontend/src/components/layout/TaskMonitorDock.tsx
  -> frontend/src/components/layout/runtimeMonitorFormat.ts
```

主要问题点：

| 位置 | 当前职责 | 问题 |
| --- | --- | --- |
| `single_agent_host.py` | 拼 global/session/task monitor payload | 同时负责 bucket、title、duration、is_live、summary，缺少独立投影层 |
| `runtimeWorkProjection.ts` | 判断可见性、路由、任务类型、标题兜底 | 前端重复后端生命周期决策 |
| `runtimeMonitorFormat.ts` | 格式化状态和耗时 | 仍根据 `display_bucket/is_live` 判断动态耗时 |
| `TaskMonitorDock.tsx` | 渲染三层队列 | 自己从 monitor payload 重新组合 bucket |
| `runtime.ts` | snapshot/detail 并发管理 | 缺少 monitor revision，旧响应可能覆盖新选择 |

## 3. 成熟监控系统原则

### 3.1 单一权威投影

运行状态只能由后端统一投影器解释。前端不应该重新判断任务生命周期、bucket、是否停滞、是否终态。

### 3.2 状态机优先

原始 `TaskRun.status` 不能直接驱动 UI。必须先转成监控状态机：

```text
raw task status
  -> lifecycle
  -> resource_class
  -> bucket
  -> route
  -> UI view
```

### 3.3 动态资源和静态资源分层

运行监控必须区分两类资源：

| 类型 | 含义 | 耗时规则 |
| --- | --- | --- |
| dynamic | 正在运行或等待人工处理 | 可以用当前时间刷新 |
| static | 已完成、失败、取消、终止 | 耗时冻结，禁止继续计算 |

已完成和失败任务必须是静态快照。前端只能展示后端给出的 `duration_seconds`，不能使用 `now - started_at`。

### 3.4 队列而不是杂列表

全局监控台应该是多个明确队列，而不是一个混合列表：

```text
running
completed
failed
diagnostics
```

每个队列独立排序、独立容量、独立淘汰。

### 3.5 前端只展示，不决策

前端职责：

- 渲染后端 buckets。
- 展示后端 title/status/duration。
- 根据后端 route 跳转。
- 管理当前 UI selection。

前端不负责：

- 判断任务是否完成。
- 判断任务是否停滞。
- 判断任务应该进入哪个 bucket。
- 根据内部 ID 猜任务类型。

### 3.6 并发快照必须有版本

SSE、polling、detail refresh 必须受统一版本控制：

- 新 snapshot 有 `revision`。
- 旧 snapshot 不能覆盖新 snapshot。
- detail response 必须匹配当前 `task_run_id + revision`。
- 任务离开当前 bucket 后，旧 detail 不能残留。

## 5. 后端重构设计

### 5.1 新增投影模块

新增文件：

```text
backend/harness/runtime/monitor_projection.py
```

该模块是运行监控唯一决策层。

核心职责：

- 从 `TaskRun`、`EventLog`、`AgentRunResult` 生成监控项。
- 判断 lifecycle。
- 判断 dynamic/static。
- 计算冻结后的 duration。
- 生成用户可见标题。
- 判断 bucket。
- 生成 route target。
- 输出稳定 contract。

### 5.2 标准 Monitor Item

目标结构：

```python
TaskRunMonitorItem = {
    "task_run_id": str,
    "session_id": str,
    "task_id": str,
    "title": str,
    "status": str,
    "lifecycle": "running" | "waiting" | "action_required" | "completed" | "failed" | "stale",
    "bucket": "running" | "completed" | "failed" | "diagnostics",
    "resource_class": "dynamic" | "static",
    "started_at": float,
    "ended_at": float | None,
    "duration_seconds": float,
    "last_activity_at": float,
    "last_activity_age_seconds": float,
    "action_required": bool,
    "terminal": bool,
    "stale": bool,
    "summary": str,
    "latest_event_type": str,
    "latest_step_summary": str,
    "artifact_count": int,
    "route": {
        "kind": "chat_turn_runtime" | "agent_runtime_run" | "task_graph_run",
        "session_id": str,
        "task_run_id": str,
        "graph_id": str,
    },
    "authority": "single_agent_runtime_monitor.item",
}
```

### 5.3 Global Monitor View

`list_global_live_monitor` 改为返回：

```python
{
    "authority": "single_agent_runtime_monitor.global",
    "revision": str,
    "updated_at": float,
    "bucket_limit": int,
    "summary": {
        "running": int,
        "completed": int,
        "failed": int,
        "diagnostics": int,
        "action_required": int,
    },
    "buckets": {
        "running": list[TaskRunMonitorItem],
        "completed": list[TaskRunMonitorItem],
        "failed": list[TaskRunMonitorItem],
        "diagnostics": list[TaskRunMonitorItem],
    },
}
```

移除旧语义：

- `display_bucket: live`
- `display_bucket: recent`
- `display_bucket: stale`
- `display_bucket: history`

如果需要兼容旧字段，只能在短期 adapter 中保留，并标明删除时间；主路径不允许继续依赖。

### 5.4 Bucket 规则

#### running

进入条件：

- `created/running/waiting_executor` 且 `last_activity_age_seconds <= freshness_seconds`
- `waiting_approval`
- `blocked` 且需要用户处理

属性：

- `resource_class = dynamic`
- `duration_seconds` 可随当前时间变化
- 按 `last_activity_at desc` 排序

#### completed

进入条件：

- `completed/success`

属性：

- `resource_class = static`
- `ended_at = terminal event time or updated_at`
- `duration_seconds = ended_at - started_at`
- 按 `ended_at desc` 排序

#### failed

进入条件：

- `failed/aborted/cancelled/error`

属性：

- `resource_class = static`
- `duration_seconds` 冻结
- 按 `ended_at desc` 排序

#### diagnostics

进入条件：

- `running/waiting_executor` 超过 freshness
- 缺少关键时间字段
- route target 不完整
- 状态和事件不一致
- 后台残留任务

属性：

- 不污染 running 数量
- 可以显示为“需要清理 / 需诊断”
- 后续可加清理动作

### 5.5 Session 和 Detail 视图共用投影器

以下接口必须共用 `TaskRunMonitorProjector`：

- `list_global_live_monitor`
- `get_session_live_monitor`
- `get_task_run_live_monitor`

不允许三套接口各自计算状态。

## 6. 前端重构设计

### 6.1 API 类型

更新：

```text
frontend/src/lib/api.ts
```

新增稳定类型：

```ts
type RuntimeMonitorBucket = "running" | "completed" | "failed" | "diagnostics";
type RuntimeResourceClass = "dynamic" | "static";
type RuntimeRouteKind = "chat_turn_runtime" | "agent_runtime_run" | "task_graph_run";
```

`GlobalRuntimeMonitorItem` 改为以后端 contract 为准，不再暴露旧 `display_bucket` 作为主字段。

### 6.2 删除前端重复决策

`runtimeWorkProjection.ts` 调整为：

- 主路径使用 `item.route.kind`。
- 仅保留旧 payload fallback。
- 不再判断 bucket。
- 不再判断 lifecycle。
- 不再根据 ID 前缀作为主路径判断。

`runtimeMonitorFormat.ts` 调整为：

- 只格式化后端字段。
- dynamic 使用 ticker。
- static 永远展示 `duration_seconds`。
- 不再根据 `status/display_bucket/is_live` 推断资源类型。

`TaskMonitorDock.tsx` 调整为：

- 直接渲染：

```ts
monitor.buckets.running
monitor.buckets.completed
monitor.buckets.failed
monitor.buckets.diagnostics
```

- 不再从 `task_runs` filter。
- tab 使用后端 summary。

### 6.3 Store 并发模型

`frontend/src/lib/store/runtime.ts` 增加：

- `globalRuntimeMonitorRevision`
- `selectedGlobalRuntimeMonitorRevision`
- detail request version

规则：

- snapshot revision 旧于当前 revision 时丢弃。
- detail response 回来时必须匹配当前 selected task id。
- detail response 回来时必须匹配发起时的 monitor revision。
- 任务离开所有 bucket 后清空 selected detail。
- 任务从 running 移动到 completed/failed 后，selection 保持同一 task id，但 active bucket 可切到对应终态 bucket。

## 7. 点击路由设计

后端 route 为权威：

```python
"route": {
    "kind": "chat_turn_runtime",
    "session_id": "session-a",
    "task_run_id": "taskrun:...",
    "graph_id": "",
}
```

前端点击：

```text
chat_turn_runtime -> 打开对应会话页
task_graph_run -> 打开主页面监控页 / task-system runtime view
agent_runtime_run -> 打开任务运行详情页
```

禁止主路径继续用以下方式猜路由：

- `task_run_id` 前缀
- `task_id` 前缀
- `runtime_lane`
- `latest_event_type`

这些只能存在于 legacy fallback，并且必须有删除条件。

## 8. 标题策略

标题来源优先级：

1. 显式用户任务标题。
2. 项目标题。
3. 任务图标题。
4. 目标摘要。
5. 通用 fallback：
   - `会话任务运行中`
   - `会话任务等待处理`
   - `会话任务已完成`
   - `会话任务失败`
   - `Agent 运行`

禁止暴露：

- `task:...`
- `taskrun:...`
- `turn:...`
- `turnrun:...`
- `session:...`
- `taskinst:...`
- `coordrun:...`

## 9. 清理范围

应删除或收窄：

- `display_bucket` 旧语义。
- `visibleRuntimeMonitorItems` 中的状态推断。
- `monitorBucketItems` 中的 fallback filter。

- 旧右栏 panel/tabs/Coding 入口残留样式。
- 已废弃的 live/recent/stale/history 测试。

## 12. 验收标准

1. 已完成任务耗时永远冻结。
2. 失败任务耗时永远冻结。
3. 运行中任务才动态计时。
4. 旧 `waiting_executor/running` 不污染运行中页。
5. 停滞任务进入 diagnostics 或被隐藏，不伪装成正常运行。
6. 内部 ID 不出现在任务标题。
7. 点击任意 bucket 的任务都能跳到正确页面。
8. SSE 和 polling 并发时旧 snapshot 不能覆盖新 snapshot。
9. 旧 detail response 不能覆盖当前选择。
10. 前端不再重复判断 bucket 和生命周期。
11. global/session/detail 三个监控视图来自同一个 projector。
12. 测试覆盖状态机、队列、耗时、路由、并发、标题脱敏。

## 13. 非目标

本计划不处理：

- 任务执行器本身的调度策略。
- Agent prompt 设计。
- TaskGraph runtime 的内部拓扑执行模型。
- Health System 的完整治理规则。

但运行监控投影完成后，Health System 可以消费新的 `diagnostics` bucket 作为风险输入。

## 14. 结论

当前监控系统的问题不是 UI 不够美观，而是缺少一个成熟的监控状态模型。正确方向是建立后端唯一投影器，让所有监控视图共享同一套状态机和资源分类；前端只展示稳定 view model，并通过 revision 处理并发。

完成本计划后，监控系统应具备成熟 agent 产品所需的基本能力：

- 运行中、完成、失败、诊断清晰分层。
- 动态资源和静态资源严格区分。
- 耗时、标题、状态、路由稳定可信。
- 不再因旧任务、旧快照、旧详情污染当前会话。

## 1. 调整结论

正式进入任务环境、工具系统、权限系统重构前，必须先完成平台级文件管理系统的基础骨架。

原因：

```text
写作环境离不开正式作品、草稿、章节产物、记忆索引。
vibe coding 环境离不开项目 workspace、sandbox overlay、git worktree、测试产物。
web research 环境离不开网页证据、下载材料、引用快照。
document/data/image 等后续环境也都离不开文件、版本、提交和产物治理。
```

所以文件管理系统不是写作环境的附属能力，也不是普通 `read_file/write_file` 工具的集合，而是平台公共底座：

```text
PlatformFileManagementSystem
-> TaskEnvironment file profile
-> SpecificTask file requirements
-> FileAccessTable
-> ToolCapabilityTable
-> PermissionContext
-> RuntimeStartPacket
-> ToolSupervisor
-> FileGateway / ToolExecutor
-> FileOperationReceipt
```

任务环境只声明自己需要哪类文件环境、哪些文件空间、哪些生命周期和提交策略；它不直接实现文件系统。具体任务继续收紧文件访问需求。agent 只看到被装配后的文件能力和工具 schema，不看到文件管理控制面。

## 2. 当前代码事实

### 2.1 裸 workspace 文件能力已存在但不是平台文件系统

现有文件：

```text
backend/capability_system/workspace_file_service.py
backend/capability_system/units/tools/read_file_tool.py
backend/capability_system/units/tools/write_file_tool.py
backend/capability_system/units/tools/file_system_tools.py
```

它们已经能做：

```text
项目根目录解析
路径穿越防护
文本读写
目录、glob、搜索、stat
```

但它们不拥有：

```text
文件空间类型
正式区/草稿区/canon 分支
版本和提交
artifact / memory / workspace 的统一引用
按任务环境和具体任务合成的访问表
文件操作审计回执
```

因此它们应被下沉为 `FileGateway` 的底层 adapter，而不是继续作为文件权威。

### 2.2 Artifact Repository 已有产物仓库，但只是产物视角

现有文件：

```text
backend/artifact_system/artifact_repository_models.py
backend/artifact_system/artifact_repository_service.py
backend/artifact_system/artifact_repository_store.py
```

已有能力：

```text
repository / artifact record
scope_kind: run_scoped / project_scoped / durable
artifact_ref
output_contract_id
content_hash
materialization record
```

问题：

```text
它只管理产物记录，不管理正式作品、草稿区、workspace、memory 文件、sandbox overlay。
它不能单独作为写作环境的正式作品文件系统。
```

目标：artifact repository 成为平台文件管理系统下的一个 repository adapter。

### 2.3 Formal / Working Memory 已有版本和事务，但与文件管理分离

现有文件：

```text
backend/memory_system/formal_memory_models.py
backend/memory_system/formal_memory_service.py
backend/memory_system/working_memory_models.py
backend/memory_system/working_memory_service.py
```

已有能力：

```text
formal memory repository / collection / record / version / transaction / read log
working memory item / handoff / temporal edge / policy profile
artifact_refs
candidate / committed / rollback-like transaction status
```

问题：

```text
memory 是文件管理环境里的结构化文件、索引和投影，但现在它作为独立系统存在。
后续如果环境直接配置 memory_space，容易绕过统一文件管理权限和审计。
```

目标：memory store 暂不重写，但要通过文件管理系统暴露为 `memory_repository` 类型的 managed repository。

### 2.4 Sandbox / overlay 是文件环境能力，但现在散落在 runtime

现有文件：

```text
backend/runtime/unit_runtime/sandbox_policy.py
backend/runtime/tool_runtime/sandbox_backend.py
backend/runtime/tool_runtime/sandbox_mount_plan.py
backend/runtime/execution_permit/sandbox_gateway.py
```

已有能力：

```text
workspace_overlay
copy-on-write
material mounts
real_workspace_access
side_effect_operations
```

问题：

```text
sandbox 目前是 runtime policy，不是平台文件管理系统的一种 managed repository。
vibe coding 的项目写入、测试产物、材料挂载和真实 workspace 提交没有统一文件访问表。
```

目标：sandbox overlay 成为文件管理系统里的 `sandbox_workspace` repository 类型。

### 2.5 前端 API 文件白名单只是 UI 管理入口

现有文件：

```text
backend/api/files.py
```

它管理：

```text
可读/可写 prefix 白名单
workspace context
文件保存后刷新索引
```

问题：

```text
这是前端管理 API 的白名单，不是 runtime、agent、任务环境共同使用的文件管理权威。
```

目标：后续 API 应改为读取平台文件管理系统暴露的 repository 和 grant，而不是维护自己的一套 prefix 权威。

## 3. 目标定义

### 3.1 PlatformFileManagementSystem

```text
PlatformFileManagementSystem =
  平台统一管理 workspace、正式作品、草稿、产物、记忆、材料挂载、下载快照、sandbox overlay 的系统级文件底座。
```

它负责：

```text
文件空间定义
repository 注册和解析
managed file ref 生成
path/ref 规范化
读写/提交/回滚/发布/归档策略
文件访问表合成
文件操作审计回执
artifact / memory / workspace / sandbox adapter 统一
```

它不负责：

```text
选择任务环境
选择具体 agent
决定 agent 语义计划
执行模型循环
用 prompt 替代权限判断
```

### 3.2 ManagedFileEnvironmentProfile

平台预置或配置的文件环境画像：

```text
ManagedFileEnvironmentProfile
  profile_id
  title
  repository_specs[]
  default_access_policy
  default_version_policy
  default_commit_policy
  default_projection_policy
  authority = "file_management.environment_profile"
```

典型 profile：

```text
file_profile.base_workspace
file_profile.writing_manuscript
file_profile.vibe_coding_project
file_profile.web_research_evidence
```

### 3.3 ManagedFileRepositorySpec

```text
ManagedFileRepositorySpec
  repository_id
  repository_kind
  storage_adapter
  scope_kind
  root_ref
  readable
  writable
  searchable
  versioned
  canonical
  commit_required
  rollback_supported
  metadata
```

`repository_kind` 包括：

```text
project_workspace
sandbox_workspace
official_work
draft_workspace
review_workspace
artifact_repository
memory_repository
material_mount
download_cache
evidence_archive
runtime_output
```

### 3.4 ManagedFileRef

不要让上层只传裸路径。上层应使用受控引用：

```text
ManagedFileRef
  file_ref
  repository_id
  repository_kind
  scope_kind
  scope_id
  logical_path
  version_id
  content_hash
  status
  authority = "file_management.managed_file_ref"
```

裸路径只在 `FileGateway` adapter 内部解析。runtime、工具监督、任务环境和审计层都应使用 `ManagedFileRef` 或可追溯到它的 path binding。

### 3.5 FileAccessTable

```text
FileAccessTable =
  TaskEnvironment file profile
  ∩ SpecificTask file requirements
  ∩ AgentRuntimeProfile ceiling
  ∩ User/session approval state
  ∩ Runtime actual availability
```

它输出：

```text
read_grants[]
write_grants[]
search_grants[]
commit_grants[]
publish_grants[]
denied_repositories[]
filtered_reasons[]
```

文件访问表是工具能力表和权限上下文的输入。`read_file/write_file/search_text` 能否出现，不只看工具名，还要看文件访问表里有没有对应 repository grant。

### 3.6 FileOperationReceipt

每个文件操作都必须能记录：

```text
operation_id
tool_call_id
task_run_id
agent_run_id
repository_id
logical_path
managed_file_ref
access_decision
approval_fingerprint
before_hash
after_hash
version_id
commit_id
rollback_ref
authority = "file_management.operation_receipt"
```

这会成为后续权限系统和工具监督系统的审计基础。

## 4. 各环境如何挂接文件系统

### 4.1 写作环境

写作环境引用：

```text
file_profile.writing_manuscript
```

必须提供：

```text
official_work_repository
draft_workspace
review_workspace
manuscript_artifact_repository
writing_memory_repository
asset_repository
```

边界：

```text
允许打开正式作品。
允许读取 canon。
允许写候选草稿和返修稿。
允许记录章节产物和版本。
允许写 memory candidate。
正式作品 canonical write 必须有 review receipt 和 commit gate。
agent 不能直接覆盖正式作品普通文件。
```

### 4.2 Vibe Coding 环境

vibe coding 环境引用：

```text
file_profile.vibe_coding_project
```

必须提供：

```text
project_workspace
git_worktree_view
sandbox_workspace
runtime_output
material_mounts
test_artifacts
```

边界：

```text
读项目文件和搜索项目文件是 project_workspace grant。
写入默认落 sandbox_workspace 或 task-bounded workspace grant。
真实 workspace 写入必须来自具体任务要求和权限系统。
git diff/status 是 project workspace 的只读投影。
测试产物落 runtime_output/test_artifacts。
```

### 4.3 Web Research 环境

web research 环境引用：

```text
file_profile.web_research_evidence
```

必须提供：

```text
source_capture_repository
download_cache
evidence_archive
citation_snapshot_repository
```

边界：

```text
网络读取由工具/权限系统控制。
网页快照和引用证据进入 evidence_archive。
外部站点写入默认 denied。
```

## 5. 第一执行切片：平台文件管理基础骨架

本计划替代 `219` 中原本的第一切片。原第一切片顺延为第二切片，并以本切片产出的文件管理对象为输入。

### 5.1 新增文件

```text
backend/file_management/__init__.py
backend/file_management/models.py
backend/file_management/default_profiles.py
backend/file_management/registry.py
backend/file_management/resolver.py
backend/file_management/access_table.py
backend/file_management/receipts.py
```

职责：

- `models.py`：定义 `ManagedFileEnvironmentProfile`、`ManagedFileRepositorySpec`、`ManagedFileRef`、`FileAccessPolicy`。
- `default_profiles.py`：提供 base workspace、writing manuscript、vibe coding project、web research evidence 四组 profile。
- `registry.py`：按 profile_id 查询文件环境画像。
- `resolver.py`：从文件 profile、任务环境选择、具体任务文件需求合成文件环境 spec。
- `access_table.py`：合成 `FileAccessTable`，记录 grant 和 denied reason。
- `receipts.py`：定义 `FileOperationReceipt`、`FileCommitReceipt`。

### 5.2 新增测试

```text
backend/tests/file_management_profile_regression.py
backend/tests/file_access_table_regression.py
backend/tests/file_operation_receipt_regression.py
```

### 5.3 验收标准

测试断言：

```text
file_profile.writing_manuscript 存在。
writing profile 包含 official_work_repository / draft_workspace / artifact_repository / memory_repository。
official_work_repository 可 open/read，但 canonical write 需要 commit gate。
draft_workspace 可写 candidate draft。
artifact_repository 和 memory_repository 都能追溯到同一个 file profile。

file_profile.vibe_coding_project 存在。
vibe coding profile 包含 project_workspace / sandbox_workspace / git_worktree_view / test_artifacts。
project_workspace 默认 read/search。
sandbox_workspace 默认 write/edit。
真实 workspace write 需要 task grant。

file_profile.web_research_evidence 存在。
web research profile 包含 evidence_archive / download_cache / citation_snapshot_repository。

ManagedFileRef 不接受路径穿越。
FileAccessTable 能说明 grant 来源和 deny 原因。
FileOperationReceipt 包含 task_run_id / agent_run_id / repository_id / logical_path / access_decision / hash fields。
```

### 5.4 本切片不做

```text
不接管现有 read_file/write_file 工具。
不改 runtime loop。
不迁移 artifact/memory 存储。
不删除 WorkspaceFileService。
不删除 ArtifactRepositoryService。
不删除 FormalMemoryService / WorkingMemoryService。
不改前端文件 API。
```

这些系统在后续切片变成 adapter 或 projection；不是长期保留多套权威。

## 6. 后续切片顺序

### Phase 1：任务环境挂接文件系统

改造 `TaskEnvironmentSpec`：

```text
file_management_profile_refs[]
file_environment_constraints
```

任务环境只引用文件 profile 和约束，不复制文件系统实现。

### Phase 2：SpecificTask 文件需求

新增或扩展：

```text
SpecificTaskAssemblyPolicy.file_requirements
```

具体任务声明：

```text
需要读取哪些 repository
需要写哪些 draft/output
是否需要 canonical commit
输出契约绑定哪个 artifact repository
memory 写入是否需要审核或提交
```

### Phase 3：工具能力表接入 FileAccessTable

`ToolCapabilityTableBuilder` 必须把文件 grant 纳入工具能力：

```text
没有 read grant -> read_file/search_text/list_dir 不可见或不可调度
没有 write grant -> write_file/edit_file 不可见或只显示不可调度原因
canonical write 只有 commit grant，不等于 write_file grant
```

### Phase 4：权限系统和 ToolSupervisor 接入文件决策

文件类工具调用前必须经过：

```text
ToolSupervisor
-> FileAccessDecision
-> FileGateway / ToolExecutor
-> FileOperationReceipt
```

### Phase 5：旧散落文件权威收口

收口目标：

```text
api/files.py 的 prefix 白名单改为读取 file management grants。
WorkspaceFileService 只做底层 path adapter。
ArtifactRepositoryService 作为 artifact_repository adapter。
FormalMemoryService / WorkingMemoryService 作为 memory_repository adapter。
SandboxBackend 作为 sandbox_workspace adapter。
```

删除或降级旧权威条件：

```text
所有文件类工具调用都有 FileOperationReceipt。
artifact/memory/workspace/sandbox 都能追溯到 ManagedFileRepositorySpec。
任务环境不再直接写 artifact_policy / memory_space 的存储权威，只引用 file profile projection。
```

## 8. 禁止实现模式

1. 把平台文件管理系统做成写作环境专属功能。
2. 让 `TaskEnvironment` 自己实现文件系统。
3. 继续让 artifact、memory、workspace、sandbox 各自作为执行权威。
4. 只按工具名授权 `read_file/write_file`，不绑定 repository、path、scope 和版本。
5. 让 agent 直接覆盖正式作品或真实 workspace，而不经过 commit/approval/receipt。
6. 用 prompt 告诉 agent “不要乱写文件”来替代文件访问表。
7. 把 `FileAccessTable` 做成前端展示字段，而不进入 ToolCapabilityTable 和 PermissionContext。
8. 为了快速通过测试伪造 file refs 或跳过 path/ref 校验。

## 9. 锁定后的执行顺序

```text
1. 新增 backend/file_management 基础对象、默认 profile、registry、resolver、access table、receipt。
2. 新增三组文件管理回归测试。
3. 运行文件管理第一切片测试。
4. 修正真实失败。
5. 再回到任务环境模块，让 TaskEnvironment 引用 file profile。
6. 再做工具能力表和权限对象。
```

这意味着 `219` 的原第一切片不取消，但顺延；它必须以本文件管理系统切片为前置条件。

## 10. 2026-05-26 阶段决策记录

平台文件管理系统第一阶段已经完成到可支撑后续重构的基础形态：

```text
FileEnvironmentProfile / ManagedFileRepositorySpec / FileAccessTable / FileOperationReceipt
FileGateway
runtime file_management_policy
read_file / write_file / edit_file 接入 FileGateway
sandbox_policy 与 file_management_policy 分离
sandbox 关闭时 file management 仍可运行
```

当前阶段到此收口，不继续扩展产品态文件管理能力。

暂缓到后续独立切片的能力：

```text
正式作品 commit gate / review receipt 完整提交链
文件管理 UI / API
正式作品打开、版本回滚、搜索和发布流程
artifact / memory 与 managed repository 的完整投影闭环
多任务环境下具体任务级文件配置入口的产品化
```

后续推进顺序不再被这些产品态能力阻塞。接下来继续按 `219` 计划推进任务环境、工具系统、权限系统和工具调用监督链路。

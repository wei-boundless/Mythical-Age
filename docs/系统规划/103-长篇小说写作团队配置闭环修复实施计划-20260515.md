# 103-长篇小说写作团队配置闭环实施计划

## 0. 文档定位

本文件只负责回答一件事：

**在长篇小说双创作者隔离互审流程已经确定的前提下，系统配置层需要补齐哪些对象、约束和落地步骤。**

流程本身不在这里展开，统一以以下文档为准：

- `docs/系统规划/107-长篇小说双创作者隔离互审流程设计-20260516.md`

因此，本文件不再讨论“应该怎样写作”，只讨论：

1. 根任务如何注册
2. TaskGraph 如何被稳定启动
3. 子任务如何消费正确输入
4. 裁决循环如何以配置方式表达
5. 资产、记忆、投影、契约如何闭环

## 1. 当前配置问题结论

结合现有 `graph.writing_team.long_novel`、子任务注册项、运行入口和产物目录，当前主要问题不是“没有图”，而是**缺少根任务和流程约束的配置闭环**。

当前已具备：

1. `graph.writing_team.long_novel` 已存在并可编译运行
2. 世界观/大纲/人物/章节等节点任务基本齐全
3. TaskGraph 专用启动入口已经存在
4. 各子节点已有 flow/workflow/contract 基础配置

当前仍缺失：

1. 长篇小说根任务对象
2. 根任务对应 assignment / flow / workflow / projection / memory profile
3. 明确体现“双创作者隔离互审”规则的复合输入契约
4. 审核员裁决驱动的循环配置
5. 正式资产与候选资产分层配置
6. 项目级命名空间和稳定运行入口

结论：

- 现在的图适合演示和试跑
- 还不适合直接作为长期写作团队的稳定业务任务
- 下一步必须先按流程文档收束配置，不再继续往旧结构里补兼容

## 2. 配置目标

配置层要实现的目标如下：

1. 用户启动的是“长篇小说项目根任务”，不是散装图运行
2. 双创作者每轮都是隔离执行，不继承污染性会话记忆
3. 审核员是唯一裁决门禁，未通过不得进入下一阶段
4. 创作者只能读取：
   - 项目基础资料
   - 当前已批准 canon
   - 对方上一轮公开提案
   - 审核员上一轮裁决要求
5. 只有审核通过后的资产，才能进入稳定 canon
6. 运行记录、产物目录、工作记忆都必须绑定统一 project namespace

## 3. 目标配置结构

### 3.1 根任务对象

需要新增长篇小说根任务，使整个图不再只是 Studio 里的裸图，而是一个稳定业务任务。

建议新增：

- `task_id = task.writing_team.long_novel`
- `task_family = writing_team_long_novel`
- `execution_chain_type = coordination_chain`
- `task_graph_id = graph.writing_team.long_novel`
- `communication_protocol_id = protocol.writing_team.long_novel`
- `topology_template_id = topology.writing_team.long_novel`

根任务职责：

1. 接收项目初始输入
2. 创建项目命名空间
3. 启动图运行
4. 绑定运行记录、产物根目录和记忆空间
5. 产出当前阶段正式结果或失败诊断

### 3.2 根任务配套注册项

以下配置必须与根任务一起补齐：

1. `storage/tasks/specific_task_records.json`
2. `storage/tasks/task_assignments.json`
3. `storage/tasks/task_flows.json`
4. `storage/tasks/task_workflows.json`
5. `storage/tasks/task_flow_contract_bindings.json`
6. `storage/tasks/task_projection_bindings.json`
7. `storage/tasks/task_memory_request_profiles.json`

原则：

1. 根任务是 `coordination_chain`
2. 子节点继续保持 `coordination_node`
3. 不再依赖旧式兼容绑定去猜测长篇任务入口

### 3.3 项目级命名空间

根任务启动后，配置层必须统一以下标识：

1. `project_id`
2. `project_title`
3. `graph_id`
4. `task_run_id`
5. `coordination_run_id`
6. `run_scope`
7. `chapter_index`
8. `artifact_root`

建议产物目录约定：

```text
output/novel_artifacts/{project_slug}/
  inputs/
  world/
  outline/
  characters/
  chapters/
  reviews/
  canon/
  archive/
  debug/
```

## 4. 关键配置约束

### 4.1 创作者隔离约束

配置上必须体现：

1. 创作者 A / B 为按轮次启动的执行单元
2. 不保留污染性上下文记忆
3. 不读取对方私有思考链
4. 只能读取公开提案和审核意见

对应要求：

1. 子任务 memory profile 不能默认挂长期对话记忆
2. 创作者 workflow prompt 要写成“按当前输入包完成本轮任务”
3. 允许读取的内容必须是结构化输入，而不是自由继承整个运行历史

### 4.2 审核员门禁约束

审核员不是共同创作者，而是质量闸门。

配置上必须保证：

1. 裁判节点使用复合输入契约
2. 契约中明确同时读取 A/B 提案
3. 输出中必须带裁决结果和下一轮要求

建议新增或重构以下输入契约：

1. `contract.writing_team.long_novel.world_judge_input`
2. `contract.writing_team.long_novel.outline_judge_input`
3. `contract.writing_team.long_novel.character_judge_input`
4. `contract.writing_team.long_novel.chapter_judge_input`

这些契约至少应包含：

1. `proposal_a_ref`
2. `proposal_b_ref`
3. `review_dimensions`
4. `must_compare_both = true`
5. `allowed_decisions`
6. `revision_requirements`

### 4.3 裁决循环约束

循环不能只靠反馈边存在，必须在配置上表达清楚。

审核结果建议统一为有限裁决集：

1. `pass`
2. `revise_current_stage`
3. `replan_current_stage`
4. `repair_world`
5. `repair_outline`
6. `repair_character`
7. `human_review_required`
8. `fail_closed`

每次裁决输出应至少包含：

1. `decision`
2. `evidence_refs`
3. `problems`
4. `requirements_for_next_round`
5. `allowed_target_nodes`
6. `attempt_index`
7. `max_attempts`

### 4.4 正式资产提升约束

只有审核通过后，产物才允许进入稳定资产层。

因此配置上要明确区分：

1. `proposal artifacts`
2. `review verdicts`
3. `canon assets`
4. `archive history`

建议稳定资产最少包含：

1. `world_canon`
2. `outline_canon`
3. `character_canon`
4. `timeline_canon`
5. `chapter_state`
6. `accepted_judgements`
7. `version_log`

## 5. 需要落地的配置改造

### 5.1 根任务注册

补齐 `task.writing_team.long_novel` 的完整配置链：

1. specific task
2. assignment
3. flow
4. workflow
5. contract binding
6. projection binding
7. memory request profile

完成标准：

1. 系统存在稳定根任务 ID
2. 根任务可直接绑定 `graph.writing_team.long_novel`
3. 运行记录不再只有图级身份，没有任务级身份

### 5.2 根工作流提示词

根 workflow 不应写成开发说明，而应写成真实角色说明。

建议角色：

```text
你是一名长篇小说项目编排员。
你负责根据项目输入启动并推进长篇小说写作团队任务图。
你要确保每一阶段都使用当前已批准资产、正确的轮次输入和明确的审核裁决。
你不直接替创作者写正文，也不跳过审核门禁。
```

### 5.3 创作者工作流提示词

创作者 prompt 必须改成隔离创作模型，示例原则：

```text
你是一名世界观创作者 A。
你只负责基于当前输入包提交本轮正式提案。
你可以参考对方上一轮公开提案与审核员要求，但不能假设未提供的历史上下文。
你必须产出完整、可评审、可比较的候选方案。
```

对应 B、人物、大纲、章节等角色都要遵守同样结构。

### 5.4 审核员工作流提示词

审核员 prompt 必须写成门禁，不是润色器：

```text
你是一名长篇小说阶段审核员。
你必须同时评审方案 A 与方案 B。
你只负责裁决是否达标、指出问题、说明证据，并下发下一轮要求。
如果未达标，你必须明确要求继续迭代，不能直接替创作者完成重写。
```

### 5.5 子任务 memory profile 收束

对创作者类节点：

1. 去掉默认长期上下文继承
2. 只保留项目 canon、当前阶段输入包、上一轮公开材料

对审核员类节点：

1. 允许读取同阶段 A/B 提案
2. 允许读取已批准 canon
3. 不允许直接消费无关历史运行噪声

### 5.6 投影与产物绑定

每一轮节点产物要能稳定映射到：

1. 哪个项目
2. 哪个阶段
3. 哪一轮
4. 哪个角色
5. 是否已批准

否则后面很容易再次出现“修了但不知道哪个版本生效”的问题。

## 6. 实施顺序

### 阶段一：先锁流程，不再边写边猜

输入：

- `107-长篇小说双创作者隔离互审流程设计-20260516.md`

输出：

- 确认哪些阶段采用双创作者互审
- 确认审核裁决集合
- 确认哪些资产可以进入 canon

### 阶段二：补齐根任务配置

输出：

1. 新增根任务对象
2. 新增根 flow/workflow/assignment 等配套配置
3. 根任务能够稳定启动长篇图

### 阶段三：收束子任务契约与 prompt

输出：

1. 审核员改用复合输入契约
2. 创作者 prompt 改成隔离轮次模型
3. memory profile 与流程目标一致

### 阶段四：收束资产与命名空间

输出：

1. 统一项目命名空间
2. 稳定 canon / archive 分层
3. 运行记录、产物、记忆可以按项目回放

## 7. 实施边界

本次配置改造不做以下事情：

1. 不为了兼容旧结构继续保留无用根绑定
2. 不靠 prompt 解释来掩盖缺失的契约结构
3. 不把所有阶段糊成一个共享记忆的大 agent
4. 不先修前端展示再倒逼后端含糊适配

## 8. 完成判定

当满足以下条件时，才算配置闭环完成：

1. 长篇小说存在稳定根任务入口
2. 根任务能绑定 `graph.writing_team.long_novel`
3. 双创作者隔离约束在 workflow 与 memory profile 中真实生效
4. 审核员裁决可以稳定驱动下一轮或通过
5. 只有审核通过资产进入 canon
6. 运行、产物、记忆都绑定统一项目命名空间

## 9. 下一步

在流程文档确认后，直接按本文件实施配置，不再反复回到“流程是不是这样”的讨论里。

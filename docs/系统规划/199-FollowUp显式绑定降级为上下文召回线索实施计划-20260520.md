# Follow-up 显式绑定降级为上下文召回线索实施计划

## 背景

六十轮基础情景复测暴露出同一个结构问题的两种表现：

- PDF 续接问题中，`memory_read_signal` 抢占了对象续接，导致 `active_pdf` 没有进入执行链，PDF 子 Agent 缺对象句柄。
- 实时天气问题中，主 Agent 正确委派给 `agent:web_researcher`，子 Agent 也真实执行了 `op.web_search`，但旧 `inventory.xlsx` 的 follow-up binding 被塞进 `agent_communication_protocol`，形成旧对象污染。

这说明问题不只是提示词或断言，而是 follow-up 被设计成了“显式执行绑定”。用户的真实需求是：主 Agent 从上下文和记忆中重新定位，再按当前任务重新读取/检索，而不是由系统提前把旧对象硬绑定到本轮。

## 目标

1. 保留 follow-up 感知能力，让“这份报告”“回到刚才”“这些人”等仍可触发上下文候选召回。
2. 取消 follow-up 对 `resolved_bindings`、`followup_execution_contract`、`agent_communication_protocol` 的直接写权。
3. 让当前任务理解层和子 Agent 基于“当前请求 + 候选线索”重新定位对象，并由实际执行结果回写 consumed handles。
4. 更新测试契约，允许 `delegate_to_agent -> agent:web_researcher -> op.web_search -> evidence packet` 这类成熟委派链路，而不是强制主 Agent 直接调用底层工具。

## 实施步骤

1. 在当前轮上下文中新增 `context_recall_candidates` 诊断/提示通道，来源包括 continuation candidates/decision、restore candidates、task summary refs。
2. 修改 runtime chain：不再把 `continuation_decision.active_bindings` 直接传给 `analyze_query_understanding` 作为 active binding。
3. 修改 ContextResolver：`continuation_decision` 不再自动写入 `explicit_inputs` 和 `resolved_bindings`，只保留为诊断与候选线索。
4. 修改 TaskSpec 组装：默认不再从 follow-up 自动生成 `followup_execution_contract`；子 Agent 通信协议按 selected recipe/current task 生成，不继承旧 follow-up contract。
5. 修改委派请求合并逻辑：只合并 explicit input、tool input 和当前 recipe 所需绑定；不再从过时 follow-up contract 注入旧对象。
6. 更新回归测试：PDF/数据集 follow-up 检查候选线索和最终 task understanding/recipe；实时搜索检查 web researcher 委派链路和 evidence packet，不再把 `delegate_to_agent` 当失败。

## 验收标准

- PDF follow-up 不再因 memory recall 被截断，能通过上下文线索进入 PDF 分析路线。
- 实时搜索/天气不再携带旧 dataset follow-up binding 或 table analyst 通信协议。
- 子 Agent 结果中的 `consumed_handles` 只来自实际执行对象。
- 相关 intent、context、task assembly、long scenario 回归通过。

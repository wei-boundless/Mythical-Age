## 1. 总结论

当前任务系统已经有任务域、任务图编辑器、契约库、A2A 预览、ContractManifest、RuntimeAssembly、LangGraph 协调 runtime 等重要骨架。

但它还不是一个能支撑一般任务的通用任务图编辑器，原因不是 UI 不够漂亮，而是核心对象还没有完全统一：

```text
前端主对象仍是 coordinationDraft / topologyDraft / protocolDraft。
单 Agent 只是一个 single_agent 模板，还不是一等 TaskGraph。
图校验、画布展示、保存发布、runtime assembly 之间仍存在断层。
Agent 节点还没有完整承载 Agent + Soul/Projection + Contract + ContextPolicy + RuntimeProfile。
边已经能预览 A2A，但还没有成为完整的业务交接对象。
```

正确方向应明确为：

```text
TaskGraph 是任务架构的唯一编辑对象。
单 Agent 任务 = input -> agent -> output 的最小图。
多 Agent 任务 = 多个 agent/review/merge/human_gate/output 节点组成的图。
coordination 不再是独立页面概念，而是 TaskGraph 的一种运行模式。
```

## 2. 当前前端真实问题

### 2.1 图对象仍以 coordination 命名和保存

`TaskSystemView.tsx` 的状态仍是：

```text
coordinationDraft
topologyDraft
protocolDraft
selectedCoordinationId
```

`CoordinationEditorWorkbench.tsx` 已经把可见文案改成了“任务图编辑器”，但组件名、props、保存 API 和数据所有权仍然是 coordination 栈。

这会导致一个后续风险：

```text
单 Agent 图只是伪装成 coordination 的模板。
多 Agent 图依赖 coordination_task。
任务自身没有稳定 graph_id / graph_kind / graph_contract。
```

### 2.2 空图校验与画布展示冲突

页面实际显示：

```text
图规模：0 节点 / 0 边
状态：图校验通过
画布：显示默认 agent:0 coordinator 节点
```

这说明至少存在三套不同真相：

```text
拓扑草稿：0 节点
画布 fallback：自动补一个 coordinator 节点
校验状态：没有把空图作为阻断问题展示
```

这是阻断级问题。通用任务图编辑器不能在空图时显示“可发布运行”。

### 2.3 单 Agent 已进入图模板，但还没有进入图架构

当前 `single_agent` 模板创建：

```text
input_1 -> agent_1 -> output_1
```

这是正确方向，但仍缺少：

```text
agent_1 的 Agent 绑定策略
agent_1 的 RuntimeProfile 决议
agent_1 的 Soul / Projection 决议
input_1 / output_1 的契约绑定
单 Agent workflow steps 到图节点的映射
单 Agent loop 与多 Agent loop 的统一入口
```

因此现在只是“单 Agent 可画成图”，还不是“单 Agent 被图拓扑接管”。

### 2.4 节点检查器不够完整

当前节点检查器已有：

```text
节点名称
绑定分任务
工作姿态
绑定 Agent
节点契约
Runtime Lane
A2A Card 摘要
```

但一个可运行节点还需要：

```text
节点类型：input / agent / review / merge / human_gate / output
Agent 选择策略：固定 Agent / 组内选择 / 运行时生成
Agent RuntimeProfile 摘要
Agent 默认 Soul / Projection
节点 projection overlay
上下文可见性策略
工具与权限摘要
输入绑定和输出映射
失败策略
验收策略
是否写回记忆
是否需要人工门控
```

这些不是表单字段堆砌，而是节点能否作为 runtime stage 被装配的必要信息。

### 2.5 边还没有成为完整交接对象

当前边检查器已有：

```text
起点 / 终点
通信模式
交接契约
A2A Message 预览
反转 / 删除
```

但通用多 Agent 协作需要边承载：

```text
业务语义：handoff / review_feedback / parallel_join / retry / finalize
A2A message type
payload contract
context filter policy
artifact refs policy
ack policy
timeout policy
retry/failure policy
```

A2A 是底层通信协议，用户不应该只看到 JSON 预览，而应该能编辑“业务交接语义”，再由系统映射到 A2A。

### 2.6 画布只是轻量 SVG 展示，尚不是通用图编辑器

`CoordinationTopologyGraph.tsx` 当前是蛇形自动布局 SVG，适合轻量展示和简单点击编辑。

缺口：

```text
拖拽定位
缩放和平移
fit view
自动布局
节点折叠
分组 / 泳道
多选
复杂边路由
迷你图
节点密度切换
```

如果目标是支撑一般多 Agent 任务，后续应抽象 `TaskGraphCanvas`，再评估 React Flow 或等价画布能力。

## 3. Agent 编排系统还需要什么

### 3.1 Agent 定义层需要更明确地服务任务图节点

当前 `AgentDescriptor` 已有：

```text
agent_id
agent_name
agent_category
builtin
default_soul_id
default_projection_id
task_scope
metadata
```

这正好可以支撑节点装配，但前端任务图还没有充分使用。

任务图节点选择 Agent 时，编排系统应提供一个面向节点的 Agent 摘要：

```text
agent_id
中文名称
内置 / 非内置
类别：主 Agent / 系统管理 Agent / 子 Agent
默认 Soul
默认 Projection
RuntimeProfile
允许 task_mode
允许 runtime_lane
允许 operations
允许 context sections
输出契约
是否可作为任务图节点
是否可被 runtime 动态选择
```

### 3.2 Agent 组需要从“目录分组”升级为“图装配候选池”

当前 Agent 组能管理成员，但任务图需要更明确：

```text
哪些 Agent 可加入当前任务域
哪些 Agent 可执行某类 work_posture
哪些 Agent 可用于 review / verify / merge
组内选择策略是什么
是否允许 runtime 从组内自动选择
```

建议增加：

```text
group_runtime_policy
allowed_work_postures
default_node_templates
selection_policy
fallback_agent_id
```

### 3.3 灵魂系统应通过 Agent 进入任务图

用户已经明确：投影是在 Agent 内装配的，尤其静态 Agent 不需要每次动态分配投影。

因此任务图应采用：

```text
Agent.default_soul_id / default_projection_id 是默认来源。
TaskGraph 节点只允许高级 overlay。
Contract 可约束投影使用。
RuntimeAssembly 最终决议本次节点可见投影。
```

前端任务图不应把“给任务分配投影”作为常规配置入口。

## 4. Runtime Assembly 还需要什么

当前 `runtime_assembly_builder.py` 已经分出：

```text
SingleAgentRuntimeAssembly
NodeRuntimeAssembly
RuntimeContextSection
RuntimeOutputContract
RuntimeAcceptanceContract
RuntimeFailureContract
RuntimeLoopPolicy
HandoffPacket
```

这是很好的骨架，但还需要统一成图模型：

```text
TaskGraphRuntimeAssembly
  graph_ref
  graph_kind
  entry_node_id
  final_node_id
  node_assemblies
  edge_handoff_packets
  graph_loop_policy
  context_partition_policy
  projection_resolution
  diagnostics
```

其中 `SingleAgentRuntimeAssembly` 可以退化为：

```text
TaskGraphRuntimeAssembly with input -> agent -> output
```

而不是继续与 coordination node assembly 分裂。

## 5. Loop 循环还需要什么

当前 runtime loop 已有两条路径：

```text
single_agent loop
coordination / LangGraph loop
```

LangGraph 协调 runtime 已经有：

```text
stage_accept
route_next
stage_prepare
stage_execute
blocked
complete
checkpoint
handoff_packets
StageExecutionRequest
A2A payload
```

但要支撑统一任务图，需要收敛成：

```text
GraphRunLoop
  单 Agent 图走同一个 graph loop，只是节点少。
  多 Agent 图按边和节点状态推进。
  每个 agent 节点内部仍保留自身 agent loop。
  图 loop 负责 stage 调度、上下文隔离、A2A 交接、验收与汇合。
```

也就是说：

```text
agent loop = 节点内部执行循环。
multi-agent graph loop = 节点之间的调度循环。
```

两者不是替代关系，而是嵌套关系。

## 6. 契约系统还需要什么

当前契约编译已经能输出：

```text
global_contracts
workflow_contracts
node_contracts
edge_handoff_contracts
runtime_contracts
acceptance_contracts
issues
```

但前端编辑心智还没有完全对齐。建议把契约分成五层：

```text
GraphContract：整张任务图的目标、边界、停止条件、最终验收。
NodeContract：单个节点要接受什么、产出什么、怎样验收。
EdgeContract：节点之间交接什么，A2A payload 必须满足什么。
RuntimeContract：Agent 能力权限、runtime lane、context visibility。
OutputContract：最终面向用户或系统的输出格式。
```

契约页面负责汇总，任务图检查器负责绑定。

## 7. 从设计原则任务系统可借鉴的模式

`docs/设计原则/14-任务系统.md` 的关键价值不在于复刻 Claude Code 的任务类型，而在于它把所有异步工作收敛到统一生命周期、统一任务状态、统一输出与通知队列。这个思想可以直接迁移到我们的 TaskGraph runtime。

### 7.1 统一生命周期，而不是只画图

设计原则里的任务状态是：

```text
pending -> running -> completed / failed / killed
```

我们的任务图也需要同样的运行态，不只是草稿态：

```text
TaskGraphRun
  pending
  running
  waiting_for_human
  completed
  failed
  killed

TaskGraphNodeRun
  pending
  ready
  running
  blocked
  waiting_for_human
  completed
  failed
  killed
```

编辑器定义的是 `TaskGraphDraft`，runtime 产生的是 `TaskGraphRun`。二者不能混为一谈。

### 7.2 极简多态接口

设计原则里 `Task` 接口最终只保留 `kill()`，因为只有 kill 需要统一分发。

我们也应避免过度抽象。TaskGraph runtime 可以先只统一这些真正需要多态分发的能力：

```text
start_graph_run(graph_ref, inputs)
continue_graph_run(run_ref, event)
kill_graph_run(run_ref)
resume_human_gate(run_ref, payload)
```

节点内部执行仍由 AgentLoop / ToolLoop / HumanGate 自己实现，不强行把所有执行器塞进同一个胖接口。

### 7.3 根状态穿透与上下文隔离

设计原则强调：子 Agent 上下文要默认隔离，但任务注册和 kill 必须穿透到根 store，否则后台任务会变成无法清理的孤儿。

迁移到本项目：

```text
GraphLoop 拥有图级状态。
AgentLoop 拥有节点内上下文。
节点可见上下文默认隔离，只通过 EdgeContract / ContextPolicy 显式传递。
所有 NodeRun / Handoff / BackgroundToolRun 必须注册到统一 TaskRunStateIndex。
kill graph 必须能级联 kill 节点 Agent、节点工具、A2A pending request。
```

这与我们已有的 `RuntimeContextSection`、`HandoffPacket`、`CoordinationRun`、`CoordinationNodeRun` 是同一方向，但需要上升到通用 `TaskGraphRun`。

### 7.4 输出不要直接污染主会话

设计原则里后台任务输出经由 disk output、TaskOutput、task-notification 汇总后再送回主循环，不把中间噪声直接灌进主上下文。

我们的多 Agent 图也应采用：

```text
节点完整输出：进运行日志 / artifact / trace。
边交接输出：由 EdgeContract 过滤成 HandoffPacket。
主 Agent 可见输出：只拿 summary/ref/accepted artifact refs。
用户最终可见输出：只由 output 节点或 final merge 产生。
```

这能避免多 Agent 协作时上下文爆炸，也能和契约系统自然绑定。

### 7.5 通知队列与优先级

设计原则里的 `now / next / later` 队列提醒我们：用户输入、系统事件、后台任务通知不能混在一起抢主循环。

建议我们后续定义：

```text
interactive_user_input：最高优先级。
human_gate_resume：高优先级。
graph_runtime_event：中优先级。
node_completion_notification：低优先级。
trace_digest / health_report：后台优先级。
```

这样主 Agent 作为协调入口时，不会被节点完成通知淹没，也不会漏掉用户插话。

### 7.6 UI 应显示运行态，而不只显示配置态

设计原则里的任务系统有后台任务 pill、进度、终态保留、输出查看。

我们的任务图编辑器也应拆分两个视图：

```text
图编辑视图
  编辑 TaskGraphDraft。

图运行视图
  查看 TaskGraphRun、NodeRun、Handoff、A2A、artifact、失败和人工门控。
```

第一阶段可以在预检发布页显示运行态摘要，后续再独立成运行监控面板。

### 8.1 顶层保持四层


## 9. 分阶段实施建议

### 阶段 1：修正图真相和校验

目标：先让现有编辑器不骗人。

必须完成：

```text
空图不得显示校验通过。
空图不得 fallback 渲染默认 coordinator。
画布空态、图规模、校验状态必须一致。
单 Agent 模板创建后才显示 input -> agent -> output。
保存拓扑只保存草稿，不发布。
预检发布才生成可运行图。
```

### 阶段 2：TaskGraph 前端适配层

目标：在不立刻推翻后端 coordination 存储的前提下，让前端以 TaskGraph 心智工作。

必须完成：

```text
新增 TaskGraphDraft 类型。
把 coordinationDraft + topologyDraft + protocolDraft 映射成 TaskGraphDraft。
把 single_agent / multi_agent 统一成 graph_kind。
节点、边、图检查器都基于 TaskGraphDraft。
Coordination 命名退到 adapter 层。
```

### 阶段 3：节点装配升级

目标：节点真正成为 Agent runtime 装配位。

必须完成：

```text
Agent 选择器使用编排系统 catalog，不手写 agent_id。
节点显示 Agent 默认 Soul / Projection。
节点显示 RuntimeProfile 摘要。
节点支持 work_posture。
节点支持 context visibility。
节点支持 input/output binding。
节点支持 failure / human gate / memory writeback 策略。
```

### 阶段 4：边和 A2A 升级

目标：边成为业务交接对象。

必须完成：

```text
edge_type 与 A2A message type 分离。
业务语义选择后自动推荐 A2A 映射。
payload contract 必选或有明确豁免。
context filter / artifact refs policy 可配置。
ack / timeout / retry / failure 可配置。
A2A 预览跟随边检查器。
```

### 阶段 5：Runtime 和 Loop 收敛

目标：单 Agent 和多 Agent 都走图。

必须完成：

```text
新增 TaskGraphRuntimeAssembly。
single_agent runtime assembly 收敛为一节点图 assembly。
LangGraph runtime 支持 graph_kind=single_agent。
StageExecutionRequest 承接节点 assembly。
GraphRunLoop 管图推进，AgentLoop 管节点内部执行。
```

### 阶段 6：任务运行态与通知队列

目标：借鉴设计原则任务系统，把图运行从“事件散落”收敛成统一生命周期。

必须完成：

```text
新增 TaskGraphRun / TaskGraphNodeRun 运行态概念。
GraphRun / NodeRun 状态采用 pending/running/completed/failed/killed。
kill graph 能级联终止节点 Agent、工具任务和 pending A2A。
节点输出通过 summary/ref/handoff packet 进入主循环，不直接污染主会话。
运行通知进入优先级队列，用户输入和 human gate resume 不被后台事件淹没。
```

### 阶段 7：画布升级

目标：支撑复杂图。

必须完成：

```text
抽象 TaskGraphCanvas。
评估 React Flow。
支持拖拽、缩放、fit view、自动布局、迷你图。
支持节点折叠和密度模式。
```

## 10. 第一批阻断问题清单

优先级从高到低：

1. 空图显示 `图校验通过`，但画布 fallback 渲染默认节点。
2. `single_agent` 模板还没有完整绑定 Agent runtime / Soul Projection / input output contract。
3. 图编辑器仍依赖 coordination 命名和对象，TaskGraph 没有成为前端主对象。
4. 节点绑定 Agent 只显示 A2A card，未显示 RuntimeProfile、默认投影和权限边界。
5. 边只有通信模式和契约，缺少业务语义到 A2A 的完整映射。
6. 预检可看 Manifest / Assembly，但图上问题不能稳定定位到节点和边。
7. 画布能力不足以支撑复杂多 Agent 任务。
8. 图运行态没有统一成 TaskGraphRun / NodeRun 生命周期，后续无法可靠 kill、resume、通知和清理。
9. 节点输出和边交接还没有明确的 summary/ref/handoff packet 输出边界，复杂协作容易污染主上下文。

## 11. 验收标准

第一阶段完成后必须满足：

```text
空图就是空图，不显示默认节点，不显示可发布。
单 Agent 任务通过图模板生成 input -> agent -> output。
任务域进入任务图后，用户看到的是当前任务的图架构，不是另一个协调任务页面。
节点检查器能说明这个节点由哪个 Agent 执行、带什么投影、用什么 runtime profile。
边检查器能说明这条边交接什么、用什么 A2A message、受哪个 payload contract 约束。
预检页能解释 ContractManifest 和 RuntimeAssembly 是否能跑。
```

最终目标：

```text
任务域负责治理。
任务图负责架构。
契约负责边界。
Agent 编排负责执行主体。
灵魂系统通过 Agent 提供 Soul / Projection。
A2A 负责节点通信。
RuntimeAssembly 负责最终装配。
GraphRunLoop 负责图推进。
AgentLoop 负责节点执行。
TaskGraphRun 负责运行生命周期。
通知队列负责把后台节点结果安全送回主 Agent。
```

# 第五阶段记忆系统正式收口与供给链重构蓝图

日期：2026-05-04

## 一、文档定位

本文件用于定义第五阶段记忆系统重构的正式实施顺序。

本阶段目标不是继续在 `MemoryFacade` 上堆更多方法，而是把记忆系统从：

`若干记忆能力 + Facade 聚合入口`

推进为：

`正式的记忆请求、记忆约束、记忆供给、记忆写回候选系统`

## 二、当前真实问题

当前记忆系统已经具备：

- conversation memory
- state memory
- long-term memory
- context package
- durable memory governance
- write candidate preview

但正式边界仍未完全收口，主要问题有四个：

1. 任务系统只输出 `TaskMemoryRequestProfile`，但没有被提升为正式 `MemoryRequest`
2. 编排系统主要消费 `dict memory_request_profile + MemoryFacade`，没有正式 `MemoryBundle`
3. `MemoryRuntimeView` 与 `ContextPolicyResult` 已经存在，但缺一层稳定供给对象
4. 写回方向已有 `MemoryWriteCandidate`，但没有成套的正式 `MemoryWritebackProposal`

## 三、第五阶段设计原则

### 3.1 restore 不等于 decide

状态记忆只能恢复候选，不能自带当前轮事实裁决权。

### 3.2 memory read 不等于 memory truth

记忆系统提供的是候选上下文与供给包，不是最终任务事实。

### 3.3 task 提需求，orchestration 给边界，memory system 给供给

- 任务系统提出记忆需求
- 编排系统给出读写范围约束
- 记忆系统输出正式供给对象

### 3.4 writeback 必须保持 candidate-only

本阶段不让 runtime 直接拿到长期记忆写入权。

### 3.5 `memory_system` 是唯一正式入口

`structured_memory`、`durable_memory`、`context_management` 可以继续作为底层实现存在，但不再作为平台正式 public boundary 暴露。

## 四、目标对象模型

本阶段建议正式补出四类对象：

### 4.1 MemoryRequest

表示任务侧或 runtime 发起的正式记忆请求。

建议字段：

- `request_id`
- `task_id`
- `session_id`
- `agent_id`
- `requested_memory_layers`
- `requested_topics`
- `memory_priority`
- `allow_long_term_memory`
- `reason`

### 4.2 MemoryScopePolicy

表示编排系统对记忆系统的正式约束。

建议字段：

- `policy_id`
- `agent_id`
- `allowed_layers`
- `allow_long_term_read`
- `allow_long_term_write`
- `allow_state_restore`
- `allow_cross_task_memory`
- `writeback_policy`

### 4.3 MemoryBundle

表示记忆系统正式供给给编排系统 / runtime 的记忆包。

建议字段：

- `bundle_id`
- `request_id`
- `session_id`
- `agent_id`
- `runtime_view`
- `context_package`
- `context_candidates`
- `restore_candidates`
- `selected_layers`
- `selected_topics`
- `diagnostics`

### 4.4 MemoryWritebackProposal

表示写回方向的正式候选包。

建议字段：

- `proposal_id`
- `session_id`
- `task_id`
- `target_layers`
- `write_candidates`
- `adopted`
- `authority`

## 五、第五阶段实施步骤

### 5.1 补正式供给对象

新增：

- `MemoryRequest`
- `MemoryScopePolicy`
- `MemoryBundle`
- `MemoryWritebackProposal`

### 5.2 补 `memory_system/supply.py`

负责：

- 把 `TaskMemoryRequestProfile` 转成 `MemoryRequest`
- 应用 scope policy
- 基于 `MemoryRuntimeView + ContextPolicyResult` 组装 `MemoryBundle`
- 基于 `MemoryWriteCandidate` 组装 `MemoryWritebackProposal`

### 5.3 让编排系统优先消费正式供给对象

`AgentRuntimeChainAssembler` 不再只依赖散装 `dict profile` 组装记忆视图，而是先走正式 bundle builder。

### 5.4 保持写回只读候选边界

写回提案必须继续：

- `candidate_only`
- `adopted = false`
- 不具备自动写入能力

### 5.5 补定向回归

至少补三类：

1. `MemoryRequest` 与任务 profile 对接
2. `MemoryBundle` 构建与只读边界
3. `MemoryWritebackProposal` 候选-only 边界

## 六、完成标准

本阶段完成后，应满足：

1. 任务系统能输出正式 `MemoryRequest`
2. 编排系统能消费正式 `MemoryBundle`
3. 记忆系统具备正式供给对象，而不再只暴露 Facade 聚合方法
4. 写回方向能输出正式 `MemoryWritebackProposal`
5. 回归测试通过

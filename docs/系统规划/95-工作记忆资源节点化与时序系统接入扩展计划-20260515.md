# 95-工作记忆资源节点化与时序系统接入扩展计划-20260515

## 0. 计划定位

本文用于扩展 TaskGraph 任务系统，将工作记忆从“运行时隐形上下文”升级为“任务图中的一等资源节点”，并把工作记忆的读取、写入、交接、冻结、提交纳入时序系统。

本文不处理 `93-TaskTemplate选择层拆除正式实施计划`。本文只处理 TaskGraph、时序调度、运行时装配、工作记忆服务和图监控之间的结构关系。

### 1.2 结构缺口

1. LangGraph 节点执行时没有传入工作记忆上下文
   - `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
   - 当前调用：

```python
build_node_runtime_assembly(
    manifest=manifest,
    node_id=node_id,
    agent_profile=agent_profile,
    explicit_inputs=explicit_inputs,
)
```

没有根据 `memory_read_policy` 选择工作记忆，也没有传入 `working_memory_context`。

2. 工作记忆读写不在时序系统中
   - 当前时序只调度业务节点、并行节点、barrier、manual gate。
   - 工作记忆读取发生在可能的上下文装配层，写入发生在零散 API 或 finalizer。
   - 因此图监控无法稳定展示“当前节点读了什么、写了什么、交接了什么”。

3. 边上的 `working_memory_handoff_policy` 没有成为调度事件
   - 边模型存在策略字段。
   - 工作记忆服务也有 handoff transaction。
   - 但边流转时没有把 source node 的输出 refs 转成 target node 可读的 handoff transaction。

4. 工作记忆与 artifact refs 的边界不清
   - artifact 是可持久化产物。
   - 工作记忆是任务运行中的短期可治理事实、草稿、决策、冲突、提示。
   - 当前很多链路只靠 `artifact_refs` 和 `upstream_outputs`，导致工作记忆没有自己的生命周期。

## 2. 核心问题

这不是“节点上下文少传一个参数”的问题，而是资源所有权没有进入任务图。

当前任务图能表达：

1. 哪个 agent 做什么事。
2. 哪些节点先后执行。
3. 哪些节点等待哪些上游产物。
4. 哪些边传递上下文或产物引用。

但它还不能表达：

1. 哪个节点在什么时候读取工作记忆。
2. 读取失败是否阻塞节点。
3. 哪个节点可以写入哪些工作记忆。
4. 写入内容是否先成为候选、再由审核节点采纳。
5. 哪些工作记忆沿边交接给下游。
6. 哪些工作记忆在图结束后归档、提升、丢弃或标记冲突。

因此，正确方向是把工作记忆作为任务图资源节点纳入时序，而不是继续把它藏在 prompt 装配里。

## 3. 设计原则

### 3.1 工作记忆是资源，不是普通文本上下文

工作记忆必须有：

1. `resource_id`
2. `task_run_id`
3. `graph_id`
4. `owner_node_id`
5. `node_run_id`
6. `kind`
7. `scope`
8. `status`
9. `promotion_state`
10. `authority`

模型只看经过资源节点选择后的工作记忆摘要，不直接拥有任意读写库的权限。

### 3.2 读写必须进入时序

工作记忆的操作节点应拥有明确类型：

1. `memory_read`
   - 在业务节点执行前运行。
   - 负责按策略选取工作记忆，生成 `working_memory_context`。

2. `memory_write_candidate`
   - 在业务节点执行后运行。
   - 负责把节点输出转成候选工作记忆。

3. `memory_handoff`
   - 在边流转时运行。
   - 负责把上游候选 refs 交接给下游可读域。

4. `memory_commit`
   - 在审核或质量门之后运行。
   - 负责接受、丢弃、冲突标记、提升到任务持久记忆。

5. `memory_finalize`
   - 图终止时运行。
   - 负责收尾归档和报告。

### 3.3 agent 不直接写库

agent 输出的是结构化建议或候选事实。写库动作由资源节点执行。

正确 prompt 表达应该是：

```text
你是一名章节写作者。
你需要根据已批准的世界观、人物设定和章节细纲写作当前批次章节。
如果你发现需要后续追踪的设定、伏笔、冲突或修订建议，请在“工作记忆候选”部分列出。
这些候选不会自动成为事实，后续会由审核节点裁决。
```

不应该写成：

```text
这是 memory_write 节点。
根据任务图写入工作记忆。
```

资源节点 prompt 如需给模型看，也必须表达为角色职责：

```text
你是一名任务记忆管理员。
你只负责把已完成节点的输出整理成候选记忆条目。
你不能把未经审核的候选当作项目事实。
你需要标明每条候选的来源、适用范围、风险和是否建议提交。
```

### 3.4 默认隔离，显式共享

工作记忆默认只在当前 `task_run_id` 内可见。跨节点共享必须通过：

1. 节点 `memory_read_policy`
2. 边 `working_memory_handoff_policy`
3. 图 `working_memory_policy`
4. 资源节点时序

不允许通过主会话、RAG、历史恢复或弱理解候选绕过图策略。

## 4. 目标模型

### 4.1 新增资源节点类型

建议扩展 `TaskGraphNodeDefinition.node_type` 支持以下类型：

1. `memory_resource`
   - 通用工作记忆资源节点。

2. `memory_read`
   - 读前置节点。

3. `memory_write`
   - 写候选节点。

4. `memory_handoff`
   - 边交接节点，可由编译器自动插入。

5. `memory_commit`
   - 审核后提交节点。

6. `memory_finalize`
   - 图结束收尾节点。

短期可以只落地 `memory_resource`，并用 `metadata.operation` 区分：

```json
{
  "node_type": "memory_resource",
  "metadata": {
    "operation": "read"
  }
}
```

但中期前端和图监控应展示为具体操作类型。

### 4.2 新增资源边类型

建议扩展 `TaskGraphEdgeDefinition.edge_type`：

1. `memory_read_before`
   - 资源读节点到业务节点。

2. `memory_write_after`
   - 业务节点到资源写节点。

3. `memory_handoff`
   - 上游资源写入到下游资源读取。

4. `memory_commit_after_review`
   - 审核节点到提交节点。

5. `memory_finalize_on_terminal`
   - 终止收尾。

### 4.3 工作记忆资源节点契约

资源节点输入：

```json
{
  "task_run_id": "...",
  "graph_id": "...",
  "node_id": "...",
  "node_run_id": "...",
  "source_stage_id": "...",
  "read_policy": {},
  "write_policy": {},
  "handoff_policy": {},
  "upstream_artifact_refs": [],
  "upstream_result_refs": [],
  "candidate_items": []
}
```

资源节点输出：

```json
{
  "operation": "read|write|handoff|commit|finalize",
  "working_memory_context": {},
  "selected_working_memory_refs": [],
  "created_working_memory_refs": [],
  "adopted_working_memory_refs": [],
  "rejected_working_memory_refs": [],
  "conflict_working_memory_refs": [],
  "read_log_ref": "",
  "handoff_transaction_ref": "",
  "finalization_ref": "",
  "blocked": false,
  "block_reason": ""
}
```

### 4.4 时序系统中的执行位置

每个业务节点在编译后可展开成：

```text
memory_read:<node_id>
  -> agent_node:<node_id>
  -> memory_write:<node_id>
```

边流转可展开成：

```text
memory_write:<source_node_id>
  -> memory_handoff:<edge_id>
  -> memory_read:<target_node_id>
```

审核与提交：

```text
agent_node:chapter_writer
  -> memory_write:chapter_writer
  -> review_gate:chapter_review
  -> memory_commit:chapter_review
```

图终止：

```text
terminal_node
  -> memory_finalize:graph
```

## 5. 运行时流程

### 5.1 节点执行前

1. 调度器选择下一个业务节点。
2. 检查是否存在 `memory_read_policy`。
3. 如果存在，先生成并执行 `memory_read` 资源节点。
4. 调用 `WorkingMemoryService.select_for_node`。
5. 生成 `working_memory_context`。
6. 将 `working_memory_context` 传给 `build_node_runtime_assembly`。
7. 节点 prompt 中只出现筛选后的 required/preferred/conflict 区块。

### 5.2 节点执行后

1. 从节点结果中抽取 `working_memory_candidates`。
2. 执行 `memory_write` 资源节点。
3. 调用 `submit_working_memory_candidates`。
4. 生成 `created_working_memory_refs`。
5. 这些 refs 不自动成为事实，默认状态为 `candidate` 或 `proposed`。

### 5.3 边交接

1. 对每条出边读取 `working_memory_handoff_policy`。
2. 如果策略声明要传递工作记忆，则执行 `memory_handoff`。
3. 调用 `resolve_working_memory_handoff`。
4. 生成 handoff transaction。
5. 目标节点的 `memory_read` 只能读取被策略允许的 refs/kinds/scopes。

### 5.4 审核与提交

1. 审核节点输出裁决。
2. `memory_commit` 根据裁决接受、丢弃或标记冲突。
3. 只有被接受的条目可以进入 `approved_artifact`、`review_decision` 或任务持久记忆。

### 5.5 图终止

1. 图完成、失败、停止或取消时触发 `memory_finalize`。
2. 调用 `finalize_working_memory`。
3. 输出归档报告。
4. 图监控展示 finalization refs。

## 6. 分阶段实施计划

### 阶段一：模型和编译层资源节点化

目标：

让 TaskGraph 能显式表达工作记忆资源节点，并让编译器能自动插入读写节点。

涉及文件：

1. `backend/tasks/task_graph_models.py`
2. `backend/tasks/coordination_graph_models.py`
3. `backend/tasks/coordination_graph_compiler.py`
4. `backend/tests/task_graph_registry_test.py`
5. `backend/tests/task_system_api_regression.py`

实施：

1. 扩展合法节点类型或新增资源节点校验。
2. 增加 `memory_resource_policy` 或复用 `metadata.operation`。
3. 编译器根据业务节点策略生成 runtime resource step 描述。
4. 保持旧字段可读，但运行时以资源节点为准。

完成标准：

1. 图定义能保存并回读工作记忆资源节点。
2. runtime spec 中能看到 resource step。
3. 没有工作记忆策略的图不会被强行插入资源节点。

### 阶段二：调度器接入资源节点时序

目标：

让工作记忆读写成为 `stage_order` 的一部分，而不是 runtime assembly 的隐式副作用。

涉及文件：

1. `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
2. `backend/orchestration/runtime_loop/task_graph_scheduler_models.py`
3. `backend/orchestration/runtime_loop/task_run_loop.py`
4. `backend/orchestration/runtime_loop/stage_execution_request.py`

实施：

1. 在 bootstrap state 中展开资源节点。
2. `memory_read` 节点完成后把 `working_memory_context` 放入 state。
3. 业务节点执行时读取 state 中的 context。
4. `memory_write` 节点完成后把 created refs 放入 state。
5. checkpoint 持久化每个资源节点的结果。

完成标准：

1. 图监控能看到 `memory_read -> agent -> memory_write`。
2. 断点恢复后不会重复写入候选。
3. `working_memory_refs` 不再长期为空。

### 阶段三：运行时装配接入

目标：

让业务 agent 真的拿到工作记忆资源节点筛选后的上下文。

涉及文件：

1. `backend/orchestration/runtime_loop/runtime_assembly_builder.py`
2. `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
3. `backend/orchestration/runtime_loop/context_manager.py`
4. `backend/tests/runtime_assembly_builder_test.py`

实施：

1. `memory_read` 输出标准 `working_memory_context`。
2. 调用 `build_node_runtime_assembly(... working_memory_context=...)`。
3. context profile 控制可见 section。
4. diagnostics 输出 selected/excluded/read_log。

完成标准：

1. assembly diagnostics 中 `working_memory_enabled = true`。
2. `working_memory.required` 或 `working_memory.preferred` 出现在可见 section。
3. 没权限的节点无法读取不属于它的工作记忆。

### 阶段四：边交接和提交治理

目标：

让边的 `working_memory_handoff_policy` 和审核节点裁决真正生效。

涉及文件：

1. `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
2. `backend/orchestration/runtime_loop/task_run_loop.py`
3. `backend/memory_system/working_memory_service.py`
4. `backend/memory_system/working_memory_finalizer.py`

实施：

1. 根据边策略生成 `memory_handoff` resource step。
2. 调用 `resolve_working_memory_handoff`。
3. 审核节点之后调用 commit 逻辑。
4. 失败、取消、停止时仍触发 finalize。

完成标准：

1. handoff transaction 可在 trace 中看到。
2. 审核未通过的候选不会进入下游 required。
3. 终止后有 finalization report。

### 阶段五：前端编辑器和图监控

目标：

让工作记忆资源节点在前端可装配、可观察。

涉及文件：

1. `frontend/src/components/workspace/views/task-system/TaskGraphWorkbench.tsx`
2. `frontend/src/components/workspace/views/task-system/TaskGraphAgentRosterPage.tsx`
3. 图监控相关组件
4. task graph API 类型定义文件

实施：

1. 节点面板新增“工作记忆资源节点”。
2. 业务节点 inspector 中配置读写策略。
3. 边 inspector 中配置 handoff policy。
4. 图监控显示：
   - 当前运行资源节点
   - 已读 refs
   - 已写 refs
   - handoff transaction
   - finalize report

完成标准：

1. 用户能在编辑器配置工作记忆读写。
2. 运行时图上能看到资源节点实时状态。
3. 节点尺寸不因资源信息膨胀，详情放 inspector 或侧栏。

## 8. 验证矩阵

### 8.1 最小图

图：

```text
memory_read:brief
  -> agent:world_design
  -> memory_write:world_design
  -> memory_finalize
```

预期：

1. `memory_read` 可完成。
2. `world_design` assembly 中有工作记忆 section。
3. `memory_write` 创建候选。
4. finalize 有报告。

### 8.2 写作团队图

图：

```text
world_design
  -> world_review
  -> outline_design
  -> outline_review
  -> character_design
  -> chapter_plan
  -> chapter_writer_a / chapter_writer_b
  -> chapter_review
  -> quality_gate
  -> memory_commit
```

预期：

1. 世界观候选不会未经审核进入章节写作。
2. 审核通过后成为下游 required 或 preferred。
3. 章节作者写出的伏笔、设定变更、冲突只作为候选。
4. 质量门之后才提交。

### 8.3 断点恢复

场景：

1. `memory_write` 已创建候选。
2. 进程重启。
3. 从 checkpoint 继续。

预期：

1. 不重复创建候选。
2. handoff transaction 不重复。
3. downstream read 能恢复 refs。

### 8.4 权限边界

场景：

1. 节点没有 `readable_scopes`。
2. 节点尝试读取 project_scope 工作记忆。

预期：

1. read log 记录 denied。
2. assembly 不出现对应 section。
3. 业务节点可以按策略阻塞或空上下文继续。

## 9. 风险控制

### 9.1 不能把工作记忆做成 RAG

工作记忆不是检索库。它是任务运行期的受控状态，必须受图策略和时序控制。

### 9.2 不能让 agent 直接拥有写库权限

agent 只能产出候选，资源节点才写库。

### 9.3 不能只做 prompt 修补

prompt 可以要求 agent 输出“工作记忆候选”，但候选是否写入、是否提交必须由资源节点和审核节点决定。

### 9.4 不能破坏无记忆图

没有声明工作记忆策略的普通任务图，不应自动插入工作记忆资源节点。

### 9.5 不能绕过时序系统

所有工作记忆 read/write/handoff/commit/finalize 都必须出现在 trace 或 checkpoint 中。

## 10. 推荐执行顺序

建议按以下顺序实施：

1. 后端模型和编译器资源节点化。
2. Runtime 支持 `memory_read` 资源节点，并把 context 注入业务节点。
3. Runtime 支持 `memory_write`，先只写候选，不做自动提交。
4. 接入 `memory_handoff`。
5. 接入 `memory_commit` 和 `memory_finalize`。
6. 补前端编辑器和图监控。
7. 用写作团队图做真实长测。

第一轮实现不需要一次性做完所有 UI，但后端 trace 必须完整，否则无法判断工作记忆是否真的进入图任务。

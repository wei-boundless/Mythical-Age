## 二、实施总原则

后续实施一律遵守以下原则：

- 任务系统只决定任务本体、任务侧配置、执行意图、能力要求、安全要求和验收要求。
- 编排系统负责 Agent 名册、Agent 绑定、worker blueprint、协作拓扑，以及单 Agent 身体、prompt 结构、工具边界、资源边界、memory scope 和 output boundary。
- runtime 只执行正式对象，不临场重定义任务和拓扑。
- 子 Agent 调用采用“两段式决策”：任务系统声明执行意图与要求，编排系统提供可调用结构，主 Agent 或 coordinator 运行时按需触发。
- 前端必须展示任务执行装配，而不是只展示对象表。
- 不为兼容保留无意义旧入口；旧语义要么迁移，要么降级，要么删除。

## 三、实施总目标

最终完成后，系统应达到：

- 通用任务和特定任务都能生成完整 `TaskExecutionAssembly`。
- 前端能清楚预览任务的执行意图、workflow、投影策略、能力要求、安全要求和验收要求。
- `AgentRegistry` 成为编排系统下可被任务引用的 Agent 唯一治理入口。
- 主 Agent、系统管理 Agent、工作子 Agent、fork 子 Agent 都能以 `AgentRun` 进入 runtime。
- `TaskExecutionIntent` 与编排策略能被 runtime 间接正式消费。
- `allow_worker_agent_spawn` 语义迁入编排策略后能产生正式 `WorkerAgentSpawnRequest`。
- 协调任务能创建 `CoordinationRun`、`CoordinationNodeRun` 和 `AgentHandoffEnvelope`。
- 旧 `TaskCoordinator` 不再作为新任务系统旁路。
- 主 Agent 能按正式特定任务配置完成基本小游戏开发任务。
- 多 Agent 协调能按正式任务拓扑完成小说协作任务，并形成创意、审核、编写、纠察、验收和反馈循环。

## 十二、实测实验矩阵

目标：

用真实操作和真实链路验证任务系统是否真正按设计书工作。

实测不是替代自动化测试，而是补充自动化测试难以覆盖的管理体验、链路连通性、trace 可读性和失败治理能力。

### 12.1 实测总规则

- [ ] 每个实测必须记录输入、操作步骤、预期结果、实际结果和 trace / 截图 / 日志证据。
- [ ] 每个实测都必须说明它验证的是 `任务登记`、`Agent 采用`、`AgentRun`、`CoordinationRun`、`handoff` 还是 `output boundary`。
- [ ] 实测不得只看前端显示成功，必须至少抽查后端正式对象或 runtime trace。
- [ ] 实测不得只看后端测试通过，必须至少抽查前端管理页是否能解释这次任务装配。
- [ ] 如果实测发现临时补丁能过但对象边界不对，应判定为未通过。
- [ ] 如果实测所需工具或能力不足，必须通过能力系统新增、登记和授权，不得让任务系统直接实现工具能力。
- [ ] 新增能力必须能在任务装配或 runtime trace 中看到正式 operation / capability / tool ref。

建议记录位置：

- `docs/系统规划/任务系统实测记录/`

建议记录模板：

```text

用户输入：
操作步骤：
预期结果：
实际结果：
前端证据：
后端证据：

能力补齐记录：

```

### 12.2 能力补齐规则

实测任务允许暴露系统能力缺口。能力缺口不是失败本身，但不能用任务系统补丁绕过去。

当实测发现能力不足时，按以下顺序处理：

- [ ] 判断缺的是工具、skill、MCP 能力、operation，还是 Agent runtime profile 权限。
- [ ] 如果缺工具或 MCP 能力，在能力系统登记能力单元。
- [ ] 如果缺 operation，在能力系统 operation registry 中登记。
- [ ] 如果缺任务侧能力需求，在任务系统 workflow 或 capability requirement 中声明。
- [ ] 如果缺 Agent 可用权限，在编排系统 AgentRuntimeProfile 中授权。
- [ ] 如果缺前端展示，在任务系统装配预览中展示能力需求和授权状态。
- [ ] 所有新增能力必须有最小回归测试或实测证据。

跑偏检查：

- [ ] 任务系统不得直接实现 PDF、RAG、结构化数据、文件写入、浏览器、小说评审等具体能力。
- [ ] 编排系统不得临场扩大工具权限。
- [ ] 主 Agent 不得为了完成任务绕过 capability registry。

- 这个修改是在强化任务系统边界，还是在制造新的混合层？

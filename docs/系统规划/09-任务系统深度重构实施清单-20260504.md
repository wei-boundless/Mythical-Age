## 二、本轮必须锁死的纠偏结论

### 2.1 系统管理 Agent 的真实含义

系统管理 Agent 是一种 Agent 类别，不是某个系统的附属物。

它的真正含义是：

- 它属于 `system_management_agent` 类别
- 它被配置了某些任务覆盖范围
- 它被授予了某些操作权限
- 它被授予了某些上下文、记忆、输出边界
- 因此它有资格管理某些系统对象

它**不是**：

- 被某个 system 结构性绑定
- 被任务系统直接指定给某个系统
- 自己拥有拓扑、协议、协调流程定义

也就是说，健康系统管理 Agent 不是“健康系统专属绑定体”，而是“一个被授予健康系统管理能力的常态 Agent”。

### 2.2 任务系统与编排系统的真正分界

任务系统回答的是：

- 这是什么任务域
- 这类输入该进入哪个任务域
- 这个任务域里有哪些正式特定任务
- 这个任务如何装配 workflow / projection / flow / memory / safety
- 这个任务是否需要协调任务
- 如果需要协调任务，协调拓扑和通信协议是什么

编排系统回答的是：

- 系统里有哪些 Agent
- 这些 Agent 属于什么类别
- 每个 Agent 被允许承接什么任务范围
- 每个 Agent 允许怎样运行
- 每个 Agent 能使用什么操作、上下文、记忆和输出边界
- 当前有哪些 Agent 符合某个任务提出的承接资格

runtime 回答的是：

- 本次到底创建了哪些 `TaskRun / AgentRun / CoordinationRun`
- 这次 handoff、merge、checkpoint、trace 实际发生了什么

### 2.3 一个最容易混淆但必须钉死的原则

正确关系：

```text
任务系统定义任务要求
    ↓
编排系统提供符合资格的 Agent 候选
    ↓
runtime 执行并留痕
```

错误关系：

```text
任务系统直接挑具体 Agent
Agent 自己绑定拓扑和协议
编排系统重写任务语义
```

## 三、任务系统的正式分层

任务系统不是“任务列表 + 几个补充配置”的平面结构，而是分层结构。

### 3.1 第一层：任务域

`任务域` 是任务系统真正的系统入口对象，值得单独成层。

它的作用不是分类标签，而是回答五个问题：

1. 这类问题属于哪个任务域
2. 这类输入如何被识别并进入本域
3. 这个域下有哪些正式特定任务
4. 这个域默认采用哪些装配规则
5. 这个域需要什么类别、能力和权限的 Agent 来承接

如果没有任务域，这些信息会散落到入口规则、特定任务、workflow、Agent 配置里，最后整个系统会重新变成一堆互相穿透的补丁对象。

任务域层应包含：

- `任务域身份`：`domain_id / title / description / domain_kind`
- `入口识别`：输入信号、触发条件、路由规则、排斥条件
- `特定任务清单`：本域允许承载哪些正式特定任务
- `系统接入定义`：本域接入哪些系统对象、命令类型、管理对象类型
- `域级默认装配`：默认 workflow、projection、flow contract、memory、safety、execution policy
- `承接资格要求`：所需 Agent 类别、能力引用、权限范围、上下文范围

任务域**不直接保存具体 Agent 选择结果**。

任务域接入系统管理时，定义的是“需要管理哪些对象、需要哪些能力”，不是“绑定哪个系统管理 Agent”。

示例：

```json
{
  "domain_id": "domain.health",
  "title": "健康任务域",
  "entry_signals": ["健康", "报错", "trace", "修复验证"],
  "managed_object_types": ["health_issue", "runtime_trace", "health_report", "test_run"],
  "required_agent_categories": ["system_management_agent"],
  "required_capability_refs": [
    "cap.health.issue_read",
    "cap.health.trace_analysis",
    "cap.health.verification"
  ],
  "required_operation_refs": [
    "op.model_response",
    "op.read_file",
    "op.memory_read"
  ]
}
```

### 3.2 第二层：特定任务

`特定任务` 是任务系统真正对 runtime 发起约束的正式任务对象。

它回答的是：

- 这次要做的具体事情是什么
- 它属于哪个任务域
- 输入契约和输出契约是什么
- 它的验收轮廓是什么
- 它默认使用哪套装配引用

特定任务是正式任务，不再保留“通用任务”作为对外主对象。

特定任务层应包含：

- `task_id / task_title / domain_id`
- `task_family / task_mode`
- `input_contract_id / output_contract_id`
- `acceptance_profile_id`
- `default_workflow_id`
- `default_flow_contract_id`
- `default_projection_policy`
- `task_policy`

### 3.3 第三层：任务装配

`任务装配` 不是一个零散配置区，而是任务如何被组装成运行要求的正式层。

它负责：

- workflow
- projection binding
- flow contract binding
- execution policy
- memory request profile
- safety envelope

这一层要明确给出：

- 这是 `single_agent_chain` 还是 `coordination_task`
- 需要哪些正式引用
- 允许什么写入、什么操作、什么记忆方式
- 是否允许 worker spawn

### 3.4 第四层：协调任务

`协调任务` 属于任务系统，不属于编排系统。

它负责：

- 协调目标
- 角色结构
- 节点关系
- 拓扑模板
- 通信协议
- handoff 规则
- merge 规则
- stop 条件

协调任务定义的是“任务怎样多主体推进”，不是“哪个 Agent 的名册配置”。

协调任务可以提出：

- 需要 `reviewer_role`
- 允许 `system_management_agent`
- 需要 `cap.trace_analysis`

但它不应该直接退化为“指定 agent:4、agent:6、agent:9”。

### 3.5 第五层：装配结果

`TaskExecutionAssembly` 是前四层收束后的正式结果层。

它是只读结果，不是主要编辑层。

它的意义是：

- 汇总本次任务最终采用了什么装配
- 让前端和 runtime 可以解释本次任务如何被执行
- 让实测可以直接比对“前端所见”和“runtime 所做”

## 四、编排系统的正式分层

编排系统处理的是“执行主体治理”，不是“任务语义定义”。

### 4.1 第一层：Agent 类别

编排系统必须保留三类 Agent：

- `main_agent`
- `system_management_agent`
- `worker_sub_agent`

这一层的作用是先回答“这是什么类型的执行主体”，而不是直接进入一堆字段编辑。

### 4.2 第二层：Agent 名册

这一层负责：

- Agent 创建、编辑、删除
- builtin Agent 保护
- Agent 基本身份信息
- 默认灵魂、默认投影、接口目标

它只定义“系统里有哪些常态 Agent”，不定义任务拓扑。

### 4.3 第三层：固定职责与任务覆盖范围

这一层负责：

- `task_scope`
- 常态职责说明
- 哪些任务域或任务模式是它长期可承接的

这里说的是“覆盖范围”，不是“这次被指派给哪个系统”。

对于系统管理 Agent，这一层表达的应该是：

- 可管理哪些对象类型
- 可承接哪些管理类任务
- 可进入哪些运行上下文

而不是“它归属哪个 system”。

### 4.4 第四层：Runtime Profile

这一层负责单 Agent runtime 装配：

- `allowed_task_modes`
- `allowed_runtime_lanes`
- `approval_policy`
- `trace_policy`
- `lifecycle_policy`

这是真正属于编排系统的运行体治理层。

### 4.5 第五层：权限与能力边界

这一层负责：

- `allowed_operations`
- `blocked_operations`
- capability coverage

系统管理 Agent 是否能管理健康系统、记忆系统或权限系统，取决于这里授予了什么能力和操作边界，而不是取决于它被“绑定到哪个 system”。

### 4.6 第六层：上下文、记忆、输出边界

这一层负责：

- `allowed_memory_scopes`
- `allowed_context_sections`
- `output_contracts`

它定义的是 Agent 在运行时能看什么、能记什么、能产出什么。

### 4.7 第七层：承接资格结果

编排系统最终要输出的不是“任务定义”，而是“哪些 Agent 符合当前任务的承接资格”。

也就是说，编排系统对任务系统的正式输出是：

- 候选资格
- runtime profile
- 权限边界
- 上下文边界

而不是拓扑、协议或协调推进规则。

示例：

```json
{
  "agent_id": "agent:3",
  "agent_category": "system_management_agent",
  "task_scope": ["health_management", "health_issue_triage", "trace_analysis"],
  "allowed_operations": ["op.model_response", "op.read_file", "op.memory_read"],
  "allowed_context_sections": ["health_issue", "runtime_trace", "assertions"],
  "output_contracts": ["HealthActionPlan", "AssistantFinalAnswer"]
}
```

## 五、两个系统如何连接

### 5.1 任务系统向编排系统提出的不是“具体 Agent”，而是“承接要求”

任务系统应提出：

- 所需 Agent 类别
- 所需能力引用
- 所需操作权限
- 所需上下文和记忆范围
- 所需输出契约

### 5.2 编排系统返回的是“可承接候选”

编排系统基于：

- Agent 类别
- task scope
- runtime profile
- allowed / blocked operations
- context / memory / output boundary

筛出可承接的 Agent。

### 5.3 runtime 执行时才产生最终运行主体

最终谁执行、生成了哪些 `AgentRun / CoordinationRun / WorkerAgentSpawnResult`，由 runtime 在正式执行时决定并留痕。

## 六、源码现状与本轮纠偏重点

当前源码里已经有一批正确方向：

- 任务系统 formal objects 已经拆出 `SpecificTaskRecord / TaskExecutionAssembly / TaskMemoryRequestProfile / CoordinationTaskDefinition / TopologyTemplate / TaskCommunicationProtocol`
- 编排系统已有 `AgentRegistry / AgentRuntimeRegistry / AgentRuntimeProfile`
- runtime 已经能落 `TaskRun / AgentRun / CoordinationRun / AgentHandoffEnvelope / CoordinationMergeResult`

但当前仍有几个会继续误导实现的结构性问题：

1. `GeneralTaskProfile` 仍在兼容承载入口对象
2. `TaskAssignment` 仍在兼容承载旧特定任务视图
3. `TaskAgentAdoptionPlan` 仍在兼容承载执行策略
4. `TaskSystemView` 仍残留“入口策略单独成页”的旧组织
5. `TaskSystemView` 与 [OrchestrationView.tsx](/d:/AI应用/langchain-agent/frontend/src/components/workspace/views/OrchestrationView.tsx) 仍有不少三段式堆叠和解释性噪音
6. 编排系统现有 `owner_system` 视图会把系统管理 Agent 错画成“归属某 system”，这必须降级为兼容显示，不再作为正式语义

其中第 6 条非常关键：

- `owner_system` 可以暂时作为旧兼容字段存在
- 但新设计、新前端、新验收都不能再把它当作系统管理 Agent 的正式定义
- 正式定义必须改成 `任务覆盖范围 + 能力覆盖 + 权限边界 + 上下文边界`

## 七、后端对象到前端层级映射

本节是阶段 D 前端重构的施工依据。后续改 [TaskSystemView.tsx](/d:/AI应用/langchain-agent/frontend/src/components/workspace/views/TaskSystemView.tsx) 和 [OrchestrationView.tsx](/d:/AI应用/langchain-agent/frontend/src/components/workspace/views/OrchestrationView.tsx) 时，必须按本映射落位，不允许按旧页面惯性随手堆字段。

### 7.1 任务系统前端映射

任务系统前端管理的是“任务要求”，不是 Agent 主体。

| 前端层级 | 后端对象 / API | 用户能做什么 | 只读诊断 | 禁止事项 |
| --- | --- | --- | --- | --- |
| 任务域目录 | 目标新增 `TaskDomainRecord`；现阶段临时由 `SpecificTaskRecord.task_family/task_mode/metadata` 归组 | 查看任务域、选择任务域、创建任务域草稿 | 域内任务数量、缺失装配数量 | 不直接选择具体 Agent |
| 入口识别 | 现阶段由 `GeneralTaskProfile / ConversationEntryPolicy` 兼容承载；API：`/tasks/entry-policies/{profile_id}` | 配置输入信号、入口规则、默认任务域 | 入口是否能落到正式任务 | 不作为顶层独立页面 |
| 特定任务 | `SpecificTaskRecord`；API：`/tasks/specific-records/{task_id}` | 创建、编辑、启用/停用特定任务 | 输入/输出契约缺失、workflow 缺失 | 不写 `selected_agent_id` |
| 任务装配 | `TaskWorkflowRecord / TaskProjectionBinding / TaskFlowContractBinding / TaskAgentAdoptionPlan(to_dict as TaskExecutionPolicy) / TaskMemoryRequestProfile` | 配置 workflow、投影承接、flow contract、执行策略、记忆请求、安全边界 | 装配引用是否齐全、权限是否闭合 | 不编辑 Agent 名册和 runtime profile |
| 协调任务 | `CoordinationTaskDefinition / TopologyTemplate / TaskCommunicationProtocol` | 配置协调目标、角色、拓扑图、协议、merge/stop 规则 | topology/protocol 引用一致性、节点是否有承接资格 | 不把 topology/protocol 放进编排系统 |
| 装配结果 | `TaskExecutionAssembly`；由 [backend/tasks/assembly_builder.py](/d:/AI应用/langchain-agent/backend/tasks/assembly_builder.py:34) 生成 | 查看本任务最终装配摘要 | refs、chain type、operation requirement、coordination refs | 不把它当主要编辑表单 |

任务系统页面的核心工作流：

```text
任务域
  -> 入口识别
  -> 特定任务
  -> 任务装配
  -> 协调任务（可选）
  -> 装配结果预览
```

任务系统前端必须让用户看见：

- 这个任务要求了什么类别的 Agent
- 这个任务要求哪些操作权限
- 这个任务要求哪些上下文和记忆范围
- 这个任务是否需要协调任务
- 这个任务最终形成了什么 `TaskExecutionAssembly`

但任务系统前端不能让用户在这里维护：

- Agent 名册
- Agent runtime profile
- 具体 Agent 创建/删除
- worker id 生成
- 单 Agent prompt/body/runtime lane 装配

### 7.2 编排系统前端映射

编排系统前端管理的是“Agent 承接资格”和“单 Agent runtime 装配”，不是任务语义。

| 前端层级 | 后端对象 / API | 用户能做什么 | 只读诊断 | 禁止事项 |
| --- | --- | --- | --- | --- |
| Agent 类别 | `AgentDescriptor.agent_category`；三类：`main_agent / system_management_agent / worker_sub_agent` | 按类别查看和筛选 Agent | 每类数量、profile 缺失数量 | 不把类别当 system 绑定 |
| Agent 名册 | `AgentRegistry / AgentDescriptor`；API：`/orchestration/agents/{agent_id}` | 新建、编辑、删除非 builtin Agent；维护名称、类别、接口目标、默认 soul/projection | builtin 保护、enabled/editable 状态 | 不编辑 workflow/topology/protocol |
| 职责覆盖 | `AgentDescriptor.task_scope / description / metadata` | 配置长期可承接任务范围和职责说明 | 与任务域/任务模式的覆盖关系 | 不把 `owner_system` 当正式语义 |
| Runtime Profile | `AgentRuntimeProfile`；API：`/orchestration/agents/{agent_id}/runtime-profile` | 配置 allowed task modes、runtime lanes、approval/trace/lifecycle | profile missing、lane 缺失 | 不定义任务执行链类型 |
| 权限与能力 | `allowed_operations / blocked_operations`；runtime 在 [model_adoption.py](/d:/AI应用/langchain-agent/backend/orchestration/runtime_loop/model_adoption.py:115) 合并判断 | 配置允许和阻断操作 | allowed/blocked 冲突、任务要求不满足 | 不替任务系统决定安全策略 |
| 记忆/上下文/输出 | `allowed_memory_scopes / allowed_context_sections / output_contracts` | 配置 Agent 可看的上下文、可用记忆、可产出契约 | output contract 不匹配、context section 缺失 | 不编辑任务 memory request profile |
| Worker 生成 | `AgentRegistry.next_worker_agent_id / worker_agent_factory`；API：`/orchestration/agents/next-worker-id` | 生成工作子 Agent 草稿、保存 worker 名册与 profile | blueprint/profile 是否齐全 | 不从任务系统生成 worker id |

编排系统页面的核心工作流：

```text
Agent 类别
  -> Agent 名册
  -> 职责覆盖
  -> Runtime Profile
  -> 权限与能力
  -> 记忆/上下文/输出
  -> 承接资格预览
```

系统管理 Agent 在此页面中应展示为：

- Agent 类别：`system_management_agent`
- 职责覆盖：例如 `health_management / trace_analysis`
- 能力覆盖：例如健康问题读取、trace 分析、验证建议
- 操作边界：例如允许 `op.read_file`，阻断 `op.write_file`
- 上下文边界：例如允许 `health_issue / runtime_trace / assertions`

不能展示为：

- 绑定健康系统
- 健康系统的附属 Agent
- 拥有健康系统拓扑或协议

### 7.3 桥接诊断层映射

任务系统和编排系统之间必须有“承接资格诊断/装配预览”，否则用户无法知道失败应该改任务还是改 Agent。

桥接层不是第三个业务系统，而是两个前端页面都可以读取的只读诊断区。

| 诊断问题 | 数据来源 | 前端位置 | 结论含义 |
| --- | --- | --- | --- |
| 当前任务要求什么 Agent 类别 | `TaskExecutionPolicy.allowed_agent_categories` | 任务系统装配预览 | 任务侧提出承接条件 |
| 当前任务要求哪些操作 | `operation_requirement.required_operations` | 任务系统装配预览 / 编排系统资格预览 | 任务侧提出权限需求 |
| 哪些 Agent 类别匹配 | `AgentDescriptor.agent_category` | 编排系统资格预览 | 编排侧初筛 |
| 哪些 Agent 范围匹配 | `AgentDescriptor.task_scope` + task family/mode | 编排系统资格预览 | 长期职责是否覆盖 |
| 哪些 Agent 权限匹配 | `AgentRuntimeProfile.allowed_operations / blocked_operations` | 编排系统资格预览 | 运行权限是否够 |
| 哪些上下文/输出匹配 | `allowed_context_sections / allowed_memory_scopes / output_contracts` | 编排系统资格预览 | 运行边界是否够 |
| 本次最终装配成什么 | `TaskExecutionAssembly / TaskBodyOrchestration / AgentRuntimeSpec` | 两侧只读预览 | 任务要求和 Agent runtime 已合流 |
| 实际运行是否符合预期 | `TaskRun / AgentRun / CoordinationRun / trace` | 实测记录 / trace reader | runtime 证据 |

桥接层的推荐判断顺序：

```text
任务要求
  -> Agent 类别是否匹配
  -> task_scope 是否覆盖
  -> allowed_operations 是否覆盖
  -> blocked_operations 是否冲突
  -> context/memory/output 是否覆盖
  -> runtime 是否能生成 AgentRuntimeSpec
```

如果失败：

- 任务要求过高，回到任务系统改 execution policy / workflow / safety / memory
- Agent 覆盖不足，回到编排系统改 task_scope / runtime profile / permissions / context
- runtime trace 不完整，回到 runtime loop / trace reader

### 7.4 当前后端缺口对前端的影响

前端重构可以先按上述框架做，但需要承认现阶段后端还有缺口：

- 暂无正式 `TaskDomainRecord`，任务域需要先由 `task_family/task_mode/metadata` 归组展示
- `ConversationEntryPolicy` 仍由 `GeneralTaskProfile` 兼容承载
- `TaskExecutionPolicy` 仍由 `TaskAgentAdoptionPlan.to_dict()` 兼容承载
- `owner_system` 仍会出现在 catalog payload，但前端必须降级显示为 legacy/兼容信息，不作为系统管理 Agent 的正式配置入口
- `CoordinationTaskDefinition` 仍保留 `coordinator_agent_id / participant_agent_ids`，前端应把它们标为默认主体引用，而不是主要角色模型

这些缺口不能成为继续沿用旧前端结构的理由。正确做法是：

1. 前端先按新层级呈现
2. 兼容对象放在对应层的高级/legacy 区
3. 正式对象物理化时，前端层级不再大改，只替换数据来源

### 阶段 E：真实功能实测闭环

所有实测都必须有真实成果，不允许只有报告没有产物。

统一规则：

- [ ] 任一实验失败，不得验收
- [ ] 必须先修实现，再回到同一实验复测
- [ ] 复测通过后，才能进入下一实验
- [ ] 每个实验必须同时留下前端证据、后端证据、runtime trace 证据、实际产物

#### E.0 硬性验收规则

本阶段不是“链路存在性演示”，而是“真实结果验收”。

以下情况一律**不得**标记为通过：

- 只有 trace，没有真实成果文件或真实内容
- 只有产物文件，但产物不可运行、不可阅读、不可验收
- 只有运行报告，没有前端证据、后端证据和 runtime trace 三侧对照
- 只有 stub / hardcoded final answer，没有真实任务推进过程
- 只有 `worker spawn requested/completed`，但没有 child `AgentRun` 与 child 实际交付结果
- 只有 `CoordinationRun` 外壳，没有创意、审核、编写、纠察、修正、验收闭环
- 只有“基本通过 / 条件通过 / 链路通过 / 部分通过 / 可继续优化”之类模糊结论

以下情况也**不算**通过：

- 前端能打开，但布局混乱、切换不自然、关键信息找不到
- 游戏文件已写出，但浏览器无法正常启动、无法操作、核心循环不能运行
- 小说文本已生成，但不是协调任务闭环产物，或内容验收没有进入 accepted
- worker 被创建，但没有实际承担任务、没有生成子成果、没有被合并进最终结果

正式通过必须同时满足：

1. 有真实任务成果
2. 有前端或用户侧可见证据
3. 有后端 formal object / runtime object 证据
4. 有 trace 证据
5. 有针对该成果的实际可用性验证
6. 失败项已修复并在同一实验重新复测通过

#### E.0.1 记录命名与状态规则

正式记录目录固定为：

```text
docs/系统规划/任务系统实测记录/
```

命名规则固定为：

- `YYYYMMDD-EX-实验名-pass.md`：仅当该实验满足全部硬性验收规则时才允许使用
- `YYYYMMDD-EX-实验名-fail.md`：实验已执行但未达到验收门槛
- `YYYYMMDD-EX-实验名-blocked.md`：存在前置阻塞，尚不能完成正式验收

同一实验不得在未满足标准时继续保留 `pass` 命名。

每份正式记录必须包含：

- 前置条件
- 操作步骤
- 实际成果路径
- 前端证据
- 后端证据
- runtime trace 证据
- 验收检查项逐条结论
- 失败判定
- 修复动作
- 复测结果

#### E.0.2 失败回流规则

- 任一实验未通过时，不得跳到下一实验
- 必须先定位失败根因，判断是任务定义问题、编排边界问题、runtime 行为问题、前端表达问题，还是验收脚本问题
- 必须先完成对应修复，再回到原实验重跑
- 若复测仍未通过，继续修复并继续重跑，直到通过为止
- 只有当前实验通过，才允许推进下一个实验项
- 严禁把“后续再修”当成验收通过的替代条件

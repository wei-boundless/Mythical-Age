# 健康子 Agent 投影管理系统规划

日期：2026-04-30  
定位：本文件规划健康维护子 Agent 的投影管理系统。目标是把“玄女默认投影 + 任务实例投影 + Skills 工作流 + PromptManifest”做成可管理、可追踪、可复用的机制，为后续多 Agent 设计铺路。

---

## 0. 核心结论

健康维护子 Agent 可以使用玄女默认投影，但不能只靠一个静态 prompt。

子 Agent 也不能被当成“缩小版主 Agent”。它应该由操作系统统一管理，按任务选择性授予资源和链路能力：

```text
操作系统：
  创建 / 删除 / 修改 / 启停子 Agent。
  管理子 Agent 可见资源、可用工具、可用链路和权限边界。

任务系统：
  只绑定一个已存在且已授权的子 Agent 到具体任务。

编排系统：
  只按 RuntimeDirective 调用子 Agent 的某个受限能力。

子 Agent：
  不默认拥有主 Agent 全链路。
  不默认拥有主 Agent 的记忆、工具、模型入口、写回能力。
  只能使用 AgentCapabilityProfile 中显式开放的资源和链路。
```

健康子 Agent 的投影应分三层：

```text
SoulSeed：
  玄女的基础人格与工作习惯。

ProjectionTemplate：
  健康维护角色模板，比如 xuannv__health_maintainer。

ProjectionInstance：
  针对某个 HealthIssue / HealthTrace / CaseDraft / FixVerification 生成的任务实例投影。
```

最终进入模型的不是静态“玄女”，而是：

```text
玄女 Seed
  + 健康维护投影模板
  + 当前健康任务契约
  + 当前 issue / trace / case / verification 摘要
  + 当前 SkillWorkflowView
  + 当前 ResourcePolicyView
  + 当前 MemoryBoundary
  + 当前 OutputContract
```

## 1. 当前代码基础

已有：

```text
backend/soul/contracts.py
  SoulProjectionRequest
  SoulSkillView
  SoulToolView
  SoulRuntimeView
  SoulPromptManifest

backend/soul/projection.py
  SoulProjectionBuilder
  build_soul_runtime_preview()

backend/soul/projection_store.py
  projection card store

backend/api/souls.py
  /api/soul/projections
```

当前能力：

```text
可以创建 projection card。
可以选择 projection card。
可以将 SoulProjectionRequest 渲染成 runtime view。
可以生成 SoulPromptManifest。
```

当前缺口：

```text
Projection card 还偏手工管理，不区分 template 和 instance。
投影没有正式绑定 TaskAgentBinding。
投影没有正式绑定 SkillWorkflowBinding。
PromptManifest 没有成为健康 trace 的一等证据。
前端投影管理还没有按系统/任务/AgentProfile 组织。
```

### 1.1 当前代码能否支持动态装配

结论：当前只能支持“运行视图级动态装配”，还不能支持“操作系统管理下的真实子 Agent 实例化”。

也就是说，现在可以把任务、skills、资源策略、投影和 runtime trace 拼成一条可读链路；但还没有一个可被创建、禁用、删除、修改能力范围的 `AgentCapabilityProfile`，也没有把子 Agent 作为操作系统资源纳入生命周期治理。

当前代码状态：

| 装配对象 | 当前支持度 | 代码位置 | 判断 |
| --- | --- | --- | --- |
| AgentDefinition / AgentContext | 静态支持 | `backend/agents/models.py` | 只有 `main/explorer/worker` 静态定义，没有实例化、生命周期、能力档案。 |
| Agent catalog | 静态配置支持 | `backend/api/agents.py` | 只能启停默认 agent 和修改 protocol link，不能创建动态子 Agent。 |
| TaskBindingRecord | 部分支持 | `backend/tasks/bindings.py` | 已有 `agent_profile_id / projection_selector / skill_scope / operation_scope / memory_scope` 字段，但没有校验真实 AgentCapabilityProfile。 |
| Task runtime contract preview | 部分支持 | `backend/tasks/contract_builder.py` | 能构建 `TaskContract -> SkillRuntimeView -> ResourcePolicyPreview -> ProjectionRequirement -> PromptManifestPreview -> OrchestrationPreview`，但明确是 preview-only。 |
| SoulProjectionRequest | 较好支持 | `backend/soul/contracts.py`、`backend/soul/projection.py` | 已能按 task、skill、tool、memory、output 渲染 runtime view 和 manifest，是动态投影的基础。 |
| ResourcePolicy / OperationGate | 较好支持 | `backend/operations/policies.py`、`backend/operations/gate.py` | 已有 adopted/runtime_executable/check pipeline，可以承接子 Agent 权限控制。 |
| RuntimeLoop | 部分支持 | `backend/orchestration/runtime_loop/task_run_loop.py` | 已有 trace、checkpoint、stage projection、tool follow-up loop，但还没有按 `agent_id/profile/lane` 选择性装配子 Agent。 |
| ProjectionInstance | 部分支持 | `backend/orchestration/runtime_loop/stage_projection.py` | 有 stage snapshot，但还不是可管理的长期 ProjectionInstance registry。 |
| SkillWorkflowBinding | 不支持 | `backend/skill_system/*` | 当前是 skill contract / registry，没有 workflow registry、步骤、停止条件。 |
| MemoryScope enforcement | 部分支持 | `backend/tasks/bindings.py`、runtime context | 有字段和视图，但缺少按子 Agent capability 的强制校验。 |
| RuntimeDirective lanes | 不支持 | `backend/orchestration/runtime_directive.py`、runtime_loop | 有 directive，但没有 `allowed_runtime_lanes` 和 agent lane adoption。 |

因此，健康维护子 Agent 的真实实例化不能直接复用 `backend/agents/models.py` 的静态 `AgentDefinition`。正确落点应是操作系统新增 Agent lifecycle / capability registry，再让任务系统只引用已授权 profile。

最低补齐合同：

```text
AgentDescriptor
  agent_id
  display_name
  owner_system
  lifecycle_state
  default_projection_template_id

AgentCapabilityProfile
  agent_id
  agent_profile_id
  allowed_task_modes
  allowed_runtime_lanes
  allowed_operations
  allowed_skill_workflows
  allowed_projection_templates
  allowed_memory_scopes
  output_contracts

TaskAgentBinding
  task_id
  task_mode
  agent_id
  agent_profile_id
  projection_template_id
  skill_workflow_id
  memory_scope
  output_contract_id

ProjectionInstance
  projection_id
  template_id
  task_id
  agent_id
  agent_profile_id
  prompt_manifest_id
  resource_policy_ref

SkillWorkflowBinding
  workflow_id
  task_mode
  visible_skill_ids
  steps
  stop_conditions
  output_contract
```

装配顺序必须固定为：

```text
OperationSystem.resolve_agent(agent_id)
  -> AgentLifecycleCheck
  -> AgentCapabilityProfile
  -> TaskSystem.bind_task_agent()
  -> SkillSystem.resolve_workflow()
  -> OperationSystem.build_resource_policy()
  -> SoulSystem.build_projection_instance()
  -> OrchestrationSystem.issue_runtime_directive()
  -> RuntimeLoop.execute_with_agent_lane()
  -> HealthSystem.record_issue_trace_or_result()
```

这一点很关键：动态装配不是让子 Agent 自己挑资源，而是每个系统把自己负责的可循环材料送进 RuntimeLoop，由编排层按已采纳的合同执行。

## 2. 投影管理对象

### 2.1 ProjectionTemplate

投影模板是可复用配置。

```text
ProjectionTemplate
  template_id
  title
  soul_id
  agent_profile_id
  role_type
  task_mode
  default_skill_workflow_id
  default_memory_policy
  default_output_contract
  guardrails
  enabled
```

健康子 Agent 默认模板：

```text
template_id = xuannv__health_maintainer
title = 玄女 / 健康维护投影
soul_id = xuannv
agent_profile_id = health_maintainer_agent
role_type = health_inspector
task_mode = health_maintenance
default_skill_workflow_id = workflow.health.issue_triage
default_memory_policy = issue_local_readonly
default_output_contract = health_analysis
enabled = true
```

### 2.2 ProjectionInstance

投影实例是某个任务运行时生成的 prompt 承载。

```text
ProjectionInstance
  projection_id
  template_id
  task_id
  issue_id
  trace_id
  case_draft_id
  verification_id
  soul_id
  agent_profile_id
  role_type
  task_mode
  skill_workflow_id
  resource_policy_ref
  prompt_manifest_id
  created_at
```

原则：

```text
ProjectionTemplate 可长期保存。
ProjectionInstance 按任务生成，可被 trace 引用。
普通成功任务不需要长期保留 ProjectionInstance 全文，只保留 manifest/ref。
问题任务需要保留 manifest 摘要和证据引用。
```

### 2.3 SkillWorkflowBinding

投影实例必须绑定工作流。

```text
SkillWorkflowBinding
  workflow_id
  task_mode
  steps
  visible_skill_ids
  output_contract
  stop_conditions
```

健康默认 workflows：

```text
workflow.health.issue_triage
workflow.health.trace_analysis
workflow.health.case_draft
workflow.health.fix_verification
```

### 2.4 AgentCapabilityProfile

子 Agent 必须有能力档案。它决定子 Agent 能走哪些链路、看哪些资源、调用哪些工具。

```text
AgentCapabilityProfile
  agent_id
  agent_profile_id
  owner_system
  enabled
  allowed_task_modes
  allowed_runtime_lanes
  allowed_operations
  allowed_skill_workflows
  allowed_memory_scopes
  allowed_projection_templates
  output_contracts
  approval_policy
  lifecycle_policy
```

健康维护子 Agent 的默认能力档案：

```text
agent_id = agent:health:maintainer
owner_system = health_system
allowed_task_modes:
  issue_triage
  trace_analysis
  case_draft
  fix_verification

allowed_runtime_lanes:
  health_issue_read
  health_trace_read
  prompt_trace_read
  memory_trace_read
  runtime_trace_read
  assertion_trace_read
  case_draft_write_candidate
  fix_verification_write_candidate

allowed_operations:
  op.model_response
  op.read_file              # only health/task docs and registered artifacts
  op.search_text            # only scoped health docs/artifacts
  op.memory_read            # only issue-local summary, no durable write

blocked_operations:
  op.edit_file
  op.write_file
  op.shell
  op.python_repl
  op.memory_write_candidate
  op.agent_bounded
```

这里的重点是：健康子 Agent 可以读证据、生成建议和候选，但不能直接写代码、改测试、改记忆或再启动其他 agent。

## 3. 管理边界

### 3.1 灵魂系统拥有

```text
SoulSeed
ProjectionTemplate
ProjectionInstance 渲染
SoulRuntimeView
SoulPromptManifest
```

灵魂系统负责回答：

```text
这个 agent 用哪个 soul？
这个任务用哪个投影模板？
本次任务实例生成了哪些 prompt sections？
哪些 sections 对模型可见？
投影是否越权声明了工具、记忆或执行权限？
```

### 3.2 任务系统拥有

```text
TaskDefinition
TaskAgentBinding
HealthMaintenanceTask
task_mode
agent_profile_id 的引用
```

任务系统负责回答：

```text
这个健康问题是否需要子 Agent？
要创建什么健康任务？
任务绑定哪个已授权 AgentProfile？
任务应该使用哪个 projection template？
```

任务系统不负责创建、删除或修改子 Agent。它只能从操作系统暴露的 AgentCapabilityProfile 中选择。

### 3.3 Skills 系统拥有

```text
SkillRuntimeContract
SkillWorkflowBinding
SkillWorkflowStep
```

Skills 系统负责回答：

```text
当前 task_mode 允许哪些 skills？
skills 的输入边界和输出边界是什么？
workflow 的步骤顺序是什么？
什么时候停止？
```

### 3.4 操作系统拥有

```text
AgentDescriptor
AgentCapabilityProfile
AgentLifecycleRecord
ResourcePolicyPreview
AdoptedResourcePolicy
OperationGateDecision
ToolView
```

操作系统负责回答：

```text
系统里有哪些子 Agent？
谁创建了子 Agent？
子 Agent 是否启用？
子 Agent 能被哪些任务使用？
健康子 Agent 能读什么？
能运行什么？
哪些动作需要审批？
哪些动作必须拒绝？
是否允许修改、删除或禁用该子 Agent？
```

操作系统拥有子 Agent 生命周期：

```text
create_agent_profile
update_agent_profile
disable_agent_profile
delete_agent_profile
bind_allowed_operations
bind_allowed_skill_workflows
bind_allowed_projection_templates
set_memory_scope
set_runtime_lanes
```

创建、修改、删除子 Agent 都是操作系统治理动作，需要记录 OperationGate / governance trace。

### 3.5 健康系统拥有

```text
HealthIssue
HealthTrace
CaseDraft
FixVerification
HealthReport
```

健康系统负责回答：

```text
这次投影实例用于哪个健康问题？
它读取了哪些证据？
它生成了什么健康分析或用例草案？
这个投影实例是否导致了新的健康风险？
```

## 4. 全链路流程

### 4.0 子 Agent 调用前置检查

任何子 Agent 运行前都必须经过四层检查：

```text
1. AgentLifecycle Check
   agent 是否存在、启用、未删除、未冻结。

2. AgentCapability Check
   当前 task_mode 是否在 allowed_task_modes。
   当前 runtime lane 是否在 allowed_runtime_lanes。
   当前 skill workflow 是否在 allowed_skill_workflows。
   当前 projection template 是否在 allowed_projection_templates。

3. ResourcePolicy Check
   当前任务是否有 ResourcePolicyPreview。
   是否已采纳为 AdoptedResourcePolicy。
   子 Agent 请求的 operation 是否在 allowed_operations。

4. OperationGate Check
   执行前重新校验 operation descriptor、directive_ref、approval、headless policy 和 safety validator。
```

如果任一检查失败：

```text
不启动子 Agent。
生成 HealthIssue / OperationGate denial 证据。
返回 blocked reason。
```

### 4.1 问题分诊链路

```text
用户发现问题 / 系统发现异常
  -> HealthIssueDraft
  -> TaskSystem.create(task.health.issue_triage)
  -> TaskAgentBinding(agent:health:maintainer)
  -> OperationSystem.resolve_agent_capability_profile(agent:health:maintainer)
  -> Capability Check / ResourcePolicy Check
  -> SkillWorkflowBinding(workflow.health.issue_triage)
  -> SoulProjectionRequest(template=xuannv__health_maintainer)
  -> SoulRuntimeView
  -> SoulPromptManifest
  -> OrchestrationSystem RuntimeLoop
  -> HealthTriageResult
  -> HealthIssue(open/triaged)
```

### 4.5 子 Agent 生命周期管理链路

子 Agent 创建、删除、修改不应该散落在任务系统或健康系统里，统一走操作系统。

```text
创建：
  OperationSystem.create_agent_profile()
    -> AgentDescriptor
    -> AgentCapabilityProfile
    -> lifecycle_state = disabled | preview_only
    -> governance record

修改：
  OperationSystem.update_agent_profile()
    -> diff
    -> validation
    -> OperationGate / approval if high risk
    -> governance record

启用：
  OperationSystem.enable_agent_profile()
    -> required contracts present
    -> allowed operations scoped
    -> projection templates valid
    -> skill workflows valid

禁用：
  OperationSystem.disable_agent_profile()
    -> block new tasks
    -> preserve historical runs

删除：
  OperationSystem.delete_agent_profile()
    -> only if no active task/run
    -> archive descriptor
    -> preserve trace refs
```

健康维护子 Agent 初始建议：

```text
lifecycle_state = preview_only
delete_policy = archive_only
update_policy = requires_governance_record
enable_policy = requires read-only capability profile
```

### 4.2 链路分析链路

```text
HealthIssue + HealthTrace
  -> TaskSystem.create(task.health.trace_analysis)
  -> ProjectionInstance(role_type=trace_diagnostician)
  -> SkillWorkflowBinding(workflow.health.trace_analysis)
  -> PromptManifest includes trace refs
  -> HealthTraceAnalysis
  -> ProblemNode evidence update
```

### 4.3 用例草案链路

```text
HealthIssue(triaged)
  -> TaskSystem.create(task.health.case_draft)
  -> ProjectionInstance(role_type=case_designer)
  -> SkillWorkflowBinding(workflow.health.case_draft)
  -> CaseDraftProposal
  -> HealthSystem.case_drafts
```

### 4.4 修复验证链路

```text
HealthIssue + before_trace + after_run
  -> TaskSystem.create(task.health.fix_verification)
  -> ProjectionInstance(role_type=verification_reviewer)
  -> SkillWorkflowBinding(workflow.health.fix_verification)
  -> FixVerificationProposal
  -> HealthIssue resolved / still_open
```

## 5. PromptManifest 管理要求

每个健康子 Agent 投影实例必须生成 PromptManifest。

最低 sections：

```text
identity_view:
  玄女基础 seed。

health_projection_template:
  健康维护投影模板。

health_task_contract:
  当前健康任务目标和边界。

health_issue_view:
  当前 issue 摘要、归属、严重级别。

health_trace_refs:
  trace / prompt / memory / runtime / assertion 引用。

skill_workflow_view:
  当前 workflow、步骤、skills、stop conditions。

resource_policy_view:
  当前可读/可执行资源边界。

memory_boundary_view:
  只读问题上下文，不写长期记忆。

output_contract_view:
  输出 HealthTriageResult / HealthTraceAnalysis / CaseDraftProposal / FixVerificationProposal。
```

健康系统在报告中应能显示：

```text
本次健康子 Agent 看到了哪些 prompt sections。
哪些 sections 是模型可见。
哪些 sections 只作为 debug/reference。
是否注入了无关对话。
是否注入了不该有的工具权限。
```


请求与响应不应只包含名字，还要包含：

```text
AgentDescriptor
AgentCapabilityProfile
allowed_operations
allowed_runtime_lanes
allowed_skill_workflows
allowed_projection_templates
allowed_memory_scopes
lifecycle_state
governance_status
```

操作系统 API 只管理 agent 能力与权限，不负责创建具体健康任务。

## 9. 验收标准

```text
健康系统可以选择“交给健康管家分析”。
操作系统可以创建、修改、禁用、删除 agent:health:maintainer。
操作系统能展示 agent:health:maintainer 的 AgentCapabilityProfile。
任务系统生成 HealthMaintenanceTask。
任务绑定 agent:health:maintainer。
任务绑定前会校验 agent lifecycle 和 capability profile。
灵魂系统生成 xuannv health projection instance。
PromptManifest 包含 health task / issue / trace / workflow / resource / output sections。
Skills workflow 按 task_mode 注入，不全量注入。
OperationGate 显示 read-only first。
健康子 Agent 不继承主 Agent 全链路，只能使用 allowed_runtime_lanes。
健康报告能引用 projection_id 和 manifest_id。
刷新页面后仍能看到健康任务、投影实例和分析结果引用。
```

## 10. 最终口径

投影管理系统不是“换人设”。

它是：

```text
按任务实例生成模型可见工作姿态的管理系统。
```

健康维护子 Agent 是第一个样板：

```text
玄女提供审慎、分诊、归因的基础风格。
任务系统决定当前健康任务。
Skills 系统决定当前工作流。
操作系统创建、删除、修改子 Agent，并决定权限边界、资源范围和可用链路。
灵魂系统把这些组合成 PromptManifest。
编排系统在 RuntimeLoop 中执行。
健康系统记录问题、证据、分析、复现和修复验证。
```

这条链路打通后，后续任何子 Agent 都可以复用同一范式。

最终原则：

```text
子 Agent 是操作系统管理的受限资源，不是主 Agent 的完整复制。
```

# 健康维护子 Agent 实例化与投影任务绑定方案

日期：2026-04-30  
定位：本方案用于提前实例化一个“健康维护子 Agent”，借此把任务系统、灵魂系统投影机制、Skills 工作流绑定夯实。它不是多智能体拓扑的完整实现，而是第一个可管理、可禁用、可审计的系统级子 Agent 样板。

---

## 0. 核心判断

健康系统要做实，必须有一个专门维护系统健康的子 Agent。

但这个子 Agent 不能绕过我们已经建立的分层架构：

```text
任务系统：
  定义健康维护任务，决定何时需要健康子 Agent。

操作系统：
  管理子 Agent 的创建、删除、修改、启停、资源范围和可用链路。

灵魂系统：
  根据具体任务实例生成投影，定制 prompt 姿态。

Skills 系统：
  给健康子 Agent 暴露专用工作流能力。

操作系统：
  决定它能读什么、能调用什么、能不能运行测试或写文件。

编排系统：
  在 RuntimeLoop 中调度它，不允许它自己成为调度中心。

健康系统：
  提供 issue、trace、case draft、fix verification 等领域对象。
```

关键原则：

```text
健康子 Agent 不是主 Agent 全链路的复制品。
它是操作系统登记和授权的受限资源。
任务系统只能绑定它，不能越权创建或扩大它。
编排系统只能调用它被授权的 runtime lane。
```

因此这次“提前实例化”不是直接开多 Agent 协作，而是先落地一个单一的、任务绑定的健康维护 AgentProfile：

```text
agent_id = agent:health:maintainer
agent_profile_id = health_maintainer_agent
default_soul = xuannv
role_type = health_inspector
task_mode = health_maintenance
```

## 2. 健康子 Agent 的职责

健康维护子 Agent 不是普通测试 agent，也不是自主修复 agent。

它的职责是：

```text
读取 HealthIssue。
读取 HealthTrace。
读取 PromptTrace / MemoryTrace / RuntimeTrace / AssertionTrace。
判断问题归属系统。
生成可读健康分析。
建议补测 profile。
生成 CaseDraft。
生成 FixVerification 建议。
把需要开发处理的事项转成任务系统里的健康维护任务草案。
```

它不能做：

```text
不能直接修改正式 case registry。
不能静默删除测试。
不能直接放宽断言。
不能绕过 OperationGate 运行危险工具。
不能把问题记录写成长期记忆。
不能直接调度其他子 Agent。
不能替代编排系统决定执行路线。
```

## 4. 灵魂投影机制设计

### 4.1 投影不是固定人设，而是任务实例 prompt

健康子 Agent 默认用玄女的基础投影，但不是把“玄女原始投影”硬塞进去。

正确理解是：

```text
玄女默认投影 = 健康维护子 Agent 的基础认知姿态。
任务实例投影 = 玄女默认投影 + HealthTaskContract + SkillWorkflow + ResourcePolicy + OutputContract。
```

玄女适合这个角色，因为她的默认工作习惯是：

```text
先看前提、范围和隐藏条件。
先找歧义、遗漏条件和潜在冲突。
把模糊问题拆成可判断的小问题。
最后必须收束成明确结论。
```

这正好对应健康系统的工作：

```text
先判断问题是否真实存在。
再判断归属系统、证据缺口和严重程度。
再读取 trace / prompt / memory / assertion。
最后收束成健康分析、复现用例或修复验证建议。
```

正确流程：

```text
HealthIssue / HealthTask
  -> TaskContract
  -> TaskAgentBinding
  -> SkillWorkflowBinding
  -> ResourcePolicyPreview
  -> SoulProjectionRequest
  -> SoulRuntimeView
  -> PromptManifest
  -> RuntimeLoop
```

也就是说，投影必须按任务实例生成。

同一个玄女，在不同健康任务中应该有不同 prompt：

```text
issue_triage:
  重点看问题归属、证据缺口、严重级别。

trace_analysis:
  重点读 PromptTrace / MemoryTrace / RuntimeTrace / AssertionTrace。

case_draft:
  重点生成复现步骤、断言和 profile 建议。

fix_verification:
  重点判断修复是否被真实验证，不被“看起来改了”欺骗。
```

### 4.1.1 玄女健康默认投影模板

建议新增默认投影卡：

```text
projection_id = xuannv__health_maintainer
title = 玄女 / 健康维护投影
soul_id = xuannv
role_type = health_inspector
task_mode = health_maintenance
agent_profile_id = health_maintainer_agent
```

默认投影内容：

```text
你是玄女的健康维护投影。
你的职责不是普通对话，而是维护 AgentRuntime 的系统健康。

你需要优先做四件事：
1. 判断问题是否真实存在，以及是否需要进入健康系统。
2. 判断问题归属系统：任务、编排、记忆、操作、灵魂、模型入口、前端、健康系统自身或 legacy adapter。
3. 读取证据引用，而不是凭印象下结论：HealthIssue、HealthTrace、PromptTrace、MemoryTrace、RuntimeTrace、AssertionTrace、CaseDraft、FixVerification。
4. 输出可执行的健康维护结果：分诊结论、证据缺口、复现用例草案、断言建议、修复验证建议或维护任务草案。

你不能做：
- 不能把所有对话都写入健康系统，只处理问题对话。
- 不能把 HealthIssue 当作长期记忆。
- 不能直接修改 active case、memory note、operation policy、soul seed 或源码。
- 不能绕过 OperationGate 运行测试或工具。
- 不能替代编排系统决定执行路线。
- 不能替代任务系统创建正式任务，只能生成任务草案或建议。

输出必须包含：
- 结论：是否是健康问题。
- 归属：owner_system / subsystem / component。
- 证据：使用了哪些 trace / prompt / memory / assertion 引用。
- 缺口：还缺什么证据。
- 动作：建议创建 issue、case draft、运行 profile、生成维护任务或关闭问题。
```

这个模板进入 projection store，但它只是默认卡片；每次真实运行仍要生成任务实例投影。

### 4.2 健康投影请求

建议健康子 Agent 的投影请求形态：

```text
SoulProjectionRequest
  task_id = health-task:<issue_id or trace_id>
  soul_id = xuannv
  role_type = health_inspector | trace_diagnostician | case_designer | verification_reviewer
  task_mode = issue_triage | trace_analysis | case_draft | fix_verification
  agent_profile_id = health_maintainer_agent
  projection_name = health:<task_mode>:<issue_id>
  skill_views = SkillWorkflowBinding.render_for_projection()
  tool_views = ResourcePolicyPreview.render_for_projection()
  task_contract_summary = 当前健康任务目标、问题摘要、边界
  memory_policy_summary = 只读问题上下文；不写长期记忆
  output_contract_summary = 输出 HealthAnalysis / CaseDraft / FixVerification 建议
```

### 4.2.1 各任务模式投影差异

`issue_triage` 投影：

```text
role_type = health_inspector
task_mode = issue_triage
关注：
  这个问题是否真实。
  归属哪个系统。
  严重程度是什么。
  是否缺少 trace / prompt / memory / assertion 证据。
输出：
  HealthTriageResult
```

`trace_analysis` 投影：

```text
role_type = trace_diagnostician
task_mode = trace_analysis
关注：
  prompt、memory、runtime、tool、assertion 的因果链。
  problem_node 是否可信。
  计划与实际执行是否偏移。
输出：
  HealthTraceAnalysis
```

`case_draft` 投影：

```text
role_type = case_designer
task_mode = case_draft
关注：
  如何复现。
  应该落在哪个 profile。
  应该有哪些结构化断言。
输出：
  CaseDraftProposal
```

`fix_verification` 投影：

```text
role_type = verification_reviewer
task_mode = fix_verification
关注：
  修复是否真的被运行验证。
  失败断言是否消失。
  是否产生新的健康风险。
输出：
  FixVerificationProposal
```

### 4.3 PromptManifest 必须显式记录

健康投影生成后，PromptManifest 至少要记录：

```text
soul_seed_section
health_task_contract_section
issue_summary_section
trace_refs_section
skill_workflow_section
resource_policy_section
memory_boundary_section
output_contract_section
```

这样健康系统才能检查：

```text
健康子 Agent 是否看到了正确 issue。
是否看到了 trace 引用。
是否注入了正确 skills。
是否错误注入了无关对话历史。
是否获得了不该有的工具权限。
```

## 5. Skills 工作流绑定

### 5.1 健康 Skills 列表

建议新增专用 skills：

```text
skill.health.issue_triage
  能力：读取问题摘要，判断 owner_system、severity、证据缺口。

skill.health.trace_reading
  能力：读取 HealthTraceEnvelope，按 prompt/memory/runtime/assertion 分层解释。

skill.health.root_cause_analysis
  能力：根据 problem node 和证据链提出根因候选。

skill.health.case_draft
  能力：把问题转成复现用例草案。

skill.health.assertion_design
  能力：为复现用例设计结构化断言。

skill.health.fix_verification
  能力：判断某次修复是否被 run/trace/assertion 验证。

skill.health.maintenance_task
  能力：生成任务系统可接收的维护任务草案。
```

### 5.2 SkillWorkflowBinding

单个 task 不应该只绑定一堆 skill 名字，还需要工作流顺序。

建议新增：

```text
SkillWorkflowBinding
  workflow_id
  task_mode
  steps:
    - step_id
      skill_ref
      input_contract
      output_contract
      required_evidence
      stop_condition
```

示例：

```text
workflow.health.issue_triage
  1. skill.health.issue_triage
     输入：HealthIssue
     输出：owner_system / severity / evidence_gap

  2. skill.health.trace_reading
     输入：HealthIssue.evidence_refs
     输出：trace_summary / missing_trace

  3. skill.health.maintenance_task
     输入：triage_result
     输出：maintenance_task_draft
```

### 5.2.1 具体工作流定义

#### workflow.health.issue_triage

用途：用户在对话、开发任务、skills 调试、测试运行中发现问题后，判断是否进入健康系统。

```text
step 1: skill.health.issue_triage
  input:
    HealthIssueDraft
    source_excerpt
    related_refs
  output:
    issue_exists
    owner_system
    severity
    evidence_gap
  stop_condition:
    issue_exists == false

step 2: skill.health.trace_reading
  input:
    related_refs
    trace_id / run_id / turn_id / task_run_id
  output:
    available_traces
    missing_traces
    trace_summary
  stop_condition:
    available_traces == empty and evidence_gap is blocking

step 3: skill.health.maintenance_task
  input:
    triage_result
    trace_summary
  output:
    HealthMaintenanceTaskDraft
    next_action
```

#### workflow.health.trace_analysis

用途：读取某条 HealthTrace，解释失败原因。

```text
step 1: skill.health.trace_reading
  input:
    HealthTraceEnvelope
  output:
    prompt_summary
    memory_summary
    runtime_summary
    assertion_summary

step 2: skill.health.root_cause_analysis
  input:
    trace summaries
    problem_nodes
    failed_assertions
  output:
    root_cause_candidates
    confidence
    evidence_refs

step 3: skill.health.maintenance_task
  input:
    root_cause_candidates
  output:
    suggested_fix_task
    suggested_case_draft
```

#### workflow.health.case_draft

用途：把问题转成可复现用例草案。

```text
step 1: skill.health.case_draft
  input:
    HealthIssue
    HealthTrace summary
    owner_system
  output:
    trigger
    expected_behavior
    profile
    scenario_shape

step 2: skill.health.assertion_design
  input:
    case_draft_base
    available_trace_layers
  output:
    assertion_specs
    evidence_refs
    failure_message_template

step 3: skill.health.fix_verification
  input:
    CaseDraftProposal
  output:
    verification_plan
```

#### workflow.health.fix_verification

用途：判断一个修复是否真的修好了问题。

```text
step 1: skill.health.trace_reading
  input:
    issue_id
    verification_run_id
    before_trace_id
    after_trace_id
  output:
    before_after_diff
    assertion_delta

step 2: skill.health.fix_verification
  input:
    before_after_diff
    assertion_delta
    open_risks
  output:
    verified
    residual_risks
    close_issue_recommendation
```

### 5.3 Skills 可见性原则

健康子 Agent 的 skills 应按任务模式注入：

```text
issue_triage:
  issue_triage
  trace_reading
  maintenance_task

trace_analysis:
  trace_reading
  root_cause_analysis

case_draft:
  case_draft
  assertion_design

fix_verification:
  trace_reading
  fix_verification
```

不要把所有健康 skills 一次性塞进 prompt。

### 5.4 SkillWorkflow 注入到投影的方式

投影中不应该只出现 skill 名字，而要出现工作流视图：

```text
SkillWorkflowView
  workflow_id
  task_mode
  current_step
  visible_skills:
    skill_id
    capability_summary
    input_boundary
    output_boundary
    forbidden_uses
  stop_conditions
  output_contract
```

注入规则：

```text
模型只看到当前任务模式需要的 workflow。
模型只看到当前 workflow 允许的 skills。
模型只看到每个 skill 的摘要、边界和输出合同。
模型不看到所有 registry 细节。
```

## 6. 操作权限边界

健康子 Agent 初期应是 read-only first。

子 Agent 的权限不是从主 Agent 继承，而是由操作系统的 `AgentCapabilityProfile` 明确声明。

```text
AgentCapabilityProfile
  agent_id
  lifecycle_state
  allowed_task_modes
  allowed_runtime_lanes
  allowed_operations
  allowed_skill_workflows
  allowed_projection_templates
  allowed_memory_scopes
  blocked_operations
```

默认允许：

```text
读取 HealthIssue。
读取 HealthTrace。
读取 PromptTrace / MemoryTrace / RuntimeTrace / AssertionTrace。
读取 case registry。
读取 run summary。
创建 CaseDraft。
创建 HealthAnalysis draft。
创建 FixVerification draft。
```

默认不允许：

```text
删除 issue。
删除或修改 active case。
运行长场景测试。
修改 memory note。
修改 soul seed。
修改 operation policy。
写代码或改文件。
```

需要 OperationGate 审批：

```text
运行测试 profile。
提升 CaseDraft 为 active case。
创建维护任务。
触发修复验证 run。
```

操作系统管理动作：

```text
create_agent_profile
update_agent_profile
disable_agent_profile
delete_agent_profile
bind_allowed_operations
bind_allowed_runtime_lanes
bind_allowed_skill_workflows
bind_allowed_projection_templates
set_memory_scope
```

删除策略：

```text
有 active task/run 的子 Agent 不能硬删除，只能 disable。
已有历史 trace 必须保留 agent_id 引用。
delete 默认 archive-only。
```

## 8. 前端表现

健康系统前端不用一开始做复杂聊天框。

先做三个入口：

```text
问题详情页：
  “交给健康管家分析”

链路追踪页：
  “让健康管家解释这条 trace”

用例草案页：
  “让健康管家补全断言”
```

点击后不直接让 agent 自由发挥，而是创建健康维护任务草案：

```text
HealthIssue
  -> Create HealthMaintenanceTask
  -> TaskAgentBinding(agent:health:maintainer)
  -> SoulProjectionRequest(xuannv + task_mode)
  -> SkillWorkflowBinding
  -> RuntimeLoop
```

## 9. 实施顺序

### Phase A：定义合同

```text
TaskAgentBinding
HealthAgentProfile
SkillWorkflowBinding
HealthMaintenanceTaskContract
```

先写后端 contracts，不急着跑 agent。

### Phase B：登记健康子 Agent

```text
agent:health:maintainer
默认 disabled 或 preview-only
由操作系统创建和管理 lifecycle。
```

可在 AgentCatalog 里看到，但不默认调度。

### Phase C：登记健康任务类型

```text
task.health.issue_triage
task.health.trace_analysis
task.health.case_draft
task.health.fix_verification
```

### Phase D：登记玄女健康投影模板

```text
xuannv__health_maintainer
```

投影模板只做默认值，正式 prompt 仍由任务实例生成。

### Phase E：登记 health skills 和 workflow

```text
skill.health.*
workflow.health.*
```

### Phase F：接健康系统前端按钮

按钮创建任务草案，不直接自由运行。

## 10. 验收标准

```text
Agent catalog 能看到健康维护智能体。
任务系统能创建 health task draft。
健康 task draft 能绑定 agent:health:maintainer。
灵魂系统能根据 health task 生成玄女健康投影。
投影 PromptManifest 中能看到 task / issue / trace / skill workflow / resource boundary。
Skills 不是全量注入，而是按 task_mode 注入。
OperationGate 默认 read-only first。
健康子 Agent 不修改 active case、不修改 memory、不修改 operation policy。
```

## 11. 最终口径

健康维护子 Agent 是我们第一个真实“任务配置型子 Agent”。

它的价值不只是测试系统自动化，而是把三个核心机制打通：

```text
任务系统：
  按任务配置子 Agent。

灵魂系统：
  按任务实例生成投影 prompt。

Skills 系统：
  按任务模式绑定工作流能力。
```

这会成为后续多 Agent 拓扑扩展的样板，但当前只做单个健康子 Agent，不进入复杂拓扑。

# 健康系统前端与悬浮健康子 Agent 设计

日期：2026-04-30  
定位：本文件补充 `17-OpenAI式链路权限子Agent实例化实施计划-20260430.md` 与 `18-链路权限子Agent完整链路设计-20260430.md`，专门定义健康系统前端、玄女健康子 Agent 的悬浮对话入口、权限边界、主动/被动触发规则与测试人员式分析职责。

---

## 0. 核心结论

健康子 Agent 不应该做成页面里的一块普通卡片，也不应该混在测试运行区域里。

它的前端形态应该是：

```text
进入健康系统页面后默认可用。
右侧或右下角有一个可折叠悬浮对话框。
形态类似客服入口，但语义不是客服，而是健康维护助理。
默认投影使用 玄女 / 健康维护投影。
```

它的系统职责是：

```text
帮助用户读取健康问题。
解释测试结果。
分析对话与 Agent 操作流程。
检查回答是否符合语义与任务预期。
提出问题节点、证据引用和下一步建议。
在测试运行结束后主动生成一份健康分析报告。
其它时候默认询问用户是否发起分析，不自动乱跑。
```

它的权限边界是：

```text
只允许读取健康系统内部文件、健康问题、测试结果、运行 trace、prompt manifest、memory trace、assertion trace。
只允许写健康系统内部的候选分析、健康报告草案、问题节点、用例草案和修复验证候选。
可以管理健康系统内部设置，例如报告模板、关注规则、语义检查策略、问题分级策略。
不能读写源码。
不能执行 shell/python。
不能修改长期记忆。
不能修改正式测试用例，除非后续有明确采纳动作和权限。
不能调用其它子 Agent。
```

## 1. 健康系统页面定位

健康系统不是旧“测试系统”的换名。

它是：

```text
系统健康维护工作台。
问题对话收集器。
运行链路追踪器。
测试结果解释器。
语义质量审查器。
开发任务能力验证器。
健康报告与问题节点沉淀层。
```

它关心的不只是：

```text
回归测试是否通过。
```

还包括：

```text
用户对话中是否暴露问题。
Agent 回答是否符合用户语义。
Agent 是否走错任务链路。
Prompt 是否注入错层。
Memory 是否误读、漏读或越权。
工具调用是否符合任务意图。
测试用例是否覆盖真实问题。
测试通过但回答质量是否仍不合格。
```

## 2. 页面信息架构

健康系统页面建议拆成四个主区域：

```text
健康总览
问题中心
运行与证据
健康报告
```

悬浮健康子 Agent 是跨区域常驻入口。

### 2.1 健康总览

目标：快速判断系统健康状态。

展示：

```text
当前开放问题数。
高风险问题数。
最近失败测试。
最近语义异常。
最近健康子 Agent 报告。
RuntimeLoop 异常数量。
Prompt / Memory / Tool / Task / Soul / Operation 各系统风险分布。
```

不要把运行按钮放在总览的主要位置。总览只负责概览和跳转。

### 2.2 问题中心

目标：管理从对话、测试、开发任务中发现的问题。

问题来源：

```text
用户手动记录。
对话中标记“这里有问题”。
测试运行失败。
测试通过但语义检查失败。
开发 skill 后人工发现行为异常。
RuntimeLoop / OperationGate / CommitGate 出现异常。
健康子 Agent 分析后生成候选问题。
```

问题详情要展示：

```text
问题标题。
归属系统。
严重级别。
来源。
关联对话。
关联 task_run_id。
关联 prompt_manifest_id。
关联 memory refs。
关联 assertions。
问题节点。
健康子 Agent 分析记录。
```

### 2.3 运行与证据

目标：让运行数据可读、可追踪、可定位。

展示：

```text
RuntimeLoop timeline。
Task -> Agent -> Workflow -> Projection -> Lane -> Policy -> Executor -> Output。
OperationGate 决策。
PromptManifest 摘要。
MemoryRuntimeView 摘要。
Tool 调用和结果。
CommitGate 决策。
Assertion 通过/失败。
语义检查结果。
```

原则：

```text
前端不要直接倾倒 JSON。
只在详情抽屉里提供 raw payload 展开入口。
主视图应是时间线、证据链和问题节点。
```

### 2.4 健康报告

目标：把一次问题或测试运行变成可读报告。

报告结构：

```text
结论
问题归属
用户对话摘要
Agent 操作流程
证据引用
语义检查
断言检查
问题节点
风险等级
建议修复范围
建议新增/修改测试用例
是否需要人工确认
```

报告不是后端文件列表。报告是给开发者读的健康文档。

### 2.5 Harness 图谱

目标：把测试体系本身变成健康系统的一等可视对象。

健康系统必须能回答：

```text
这个测试文件在验证哪个功能？
这个测试文件覆盖或提出了什么问题？
什么样的回答、链路或产物算通过？
它属于哪个 profile / layer / owner_system？
它最近在哪次运行中通过或失败？
失败后是否已经生成 HealthIssue / ProblemNode / CaseDraft？
```

第一版采用聚合视图：

```text
GET /api/test-system/harness-map
```

聚合来源：

```text
case_registry:
  active_cases / candidate_cases / legacy_cases

harness_records:
  issues / case_drafts

test_agent_report:
  governance findings / unregistered paths

experiment artifacts:
  runs / turns / issues / trace
```

前端呈现方式：

```text
左侧：功能域列表，显示 risk_status、测试文件数、开放问题数。
右侧：测试文件列表，每一项固定展示：
  指向功能
  提出/覆盖的问题
  通过标准
  profile 归属
  assertions 数量
  tags 数量
  关联 issue / finding
```

用例管理方式：

```text
前端提供“规范化模板 -> 候选用例”的入口。
用户可以选择模板，填写标题、归属系统、测试路径、覆盖问题和通过标准。
新增内容先进入 managed_cases，默认 status=candidate。
用户可以移除前端管理的候选用例。
静态 ACTIVE_CASES 不允许直接从健康页面删除。
```

这部分不是“测试系统的附属页”，而是健康系统判断自身测试能力是否可信的核心页面。

## 3. 悬浮健康子 Agent

### 3.1 形态

入口：

```text
健康系统页面右下角或右侧边缘。
默认显示一个窄条或圆形按钮。
点击展开为右侧悬浮对话框。
再次点击可折叠。
```

展开态：

```text
顶部：玄女健康管家 / 当前权限状态 / 当前绑定 issue 或 run。
中部：对话记录。
底部：输入框 + 常用动作按钮。
右上角：折叠、固定、清空当前临时对话。
```

悬浮框不遮挡主内容：

```text
宽度建议 360-440px。
高度不超过视口 80%。
可以固定在右侧，也可以临时浮层。
移动端默认底部抽屉。
```

### 3.2 可见状态

三种状态：

```text
collapsed:
  只显示入口。

idle:
  展开但没有绑定任务。

bound:
  已绑定当前 issue / run / report / test result。
```

状态提示：

```text
idle:
  “可以向我描述你想检查的问题。”

bound:
  “正在查看 health:issue:xxx / taskrun:xxx。”

running:
  “正在读取证据并生成分析。”

blocked:
  “权限不足或证据不足，需要你确认。”
```

### 3.3 常用动作

悬浮框底部提供少量明确动作：

```text
分析当前问题
解释当前测试结果
检查回答语义
生成健康报告
草拟复现用例
验证修复结果
```

动作必须绑定当前上下文：

```text
如果当前在问题详情页：
  默认绑定 issue_id。

如果当前在测试结果页：
  默认绑定 run_id / turn_id / task_run_id。

如果当前在报告页：
  默认绑定 report_id。

如果没有上下文：
  要求用户先选择问题或输入描述。
```

## 4. 主动与被动触发规则

### 4.1 主动触发

健康子 Agent 只有在少数场景可以主动发起：

```text
测试运行结束。
健康检查任务结束。
RuntimeLoop 出现 failed / blocked / output_contract_failed。
用户显式开启“自动分析本次运行”。
```

主动触发时，不能直接修改正式资产，只能生成：

```text
健康分析报告草案。
问题节点候选。
用例草案候选。
修复验证候选。
```

### 4.2 被动触发

其它时候默认需要用户发起：

```text
用户点击“分析当前问题”。
用户在悬浮框里提要求。
用户选中一段对话并提交给健康系统。
用户打开某个 run 后点击“解释结果”。
```

### 4.3 不允许的触发

```text
进入页面后自动扫描所有历史问题。
在用户普通浏览时后台自动调用模型。
自动把所有测试通过结果也写报告。
自动修改测试用例或系统配置。
自动扩展自己的权限。
```

## 5. 权限边界

健康子 Agent 的默认能力档案需要收紧为健康系统内部能力。

建议权限：

```text
allowed_task_modes:
  issue_triage
  trace_analysis
  semantic_review
  test_result_analysis
  case_draft
  fix_verification
  health_report

allowed_runtime_lanes:
  health_issue_read
  health_trace_read
  prompt_trace_read
  memory_trace_read
  runtime_trace_read
  assertion_trace_read
  health_setting_read
  health_setting_candidate
  case_draft_candidate
  fix_verification_candidate
  health_report_candidate

allowed_operations:
  op.model_response
  op.health_issue_read
  op.health_trace_read
  op.health_report_write_candidate
  op.health_setting_read
  op.health_setting_write_candidate
  op.test_result_read
  op.prompt_manifest_read
  op.memory_trace_read

blocked_operations:
  op.write_file
  op.edit_file
  op.shell
  op.python_repl
  op.memory_write_candidate
  op.agent_bounded
  op.source_code_write
  op.test_case_adopt_without_review
```

说明：

```text
健康系统内部设置管理允许“候选写入”。
正式配置变更需要用户确认或后续 OperationGate 采纳。
读取范围限定在健康系统、测试结果、runtime trace、prompt manifest、memory trace、assertion trace。
不允许读取任意源码文件作为默认权限。
```

## 6. 测试人员式职责

健康子 Agent 必须像正常测试人员一样工作。

它不能只看：

```text
测试是否绿了。
```

它还必须检查：

```text
回答是否真正回应了用户问题。
回答有没有偷换概念。
回答有没有遗漏关键约束。
回答有没有违反系统角色或权限边界。
回答有没有使用未授权信息。
回答和 prompt / memory / tool result 是否一致。
Agent 操作流程是否合理。
测试用例断言是否覆盖真实风险。
通过结果是否只是表面通过。
```

语义检查建议输出：

```text
semantic_pass: true/false
matched_user_intent: high/medium/low
missing_constraints: []
unsupported_claims: []
contradictions: []
evidence_refs: []
recommendation: ""
```

## 7. 健康报告触发策略

### 7.1 测试结束后主动报告

测试运行结束后，如果用户开启健康分析，健康子 Agent 自动生成：

```text
测试运行健康报告。
```

内容：

```text
测试概况。
失败用例。
通过但可疑用例。
语义检查摘要。
关键对话片段。
Agent 操作流程。
RuntimeLoop 异常。
Prompt / Memory / Tool 风险。
建议新增用例。
建议修复范围。
```

### 7.2 普通问题需要询问

非测试结束场景：

```text
用户进入页面。
用户切换问题。
用户查看运行记录。
```

健康子 Agent 只提示：

```text
“是否需要我分析当前问题？”
```

除非用户发起，不自动模型调用。

## 10. UI 设计方向

健康系统是维护系统健康的工程工作台。

设计风格：

```text
克制、清晰、高密度。
更像故障分析台，不像聊天产品。
对话框轻，但证据链重。
```

页面主色建议：

```text
基础：近白、浅灰、墨色。
状态：绿色通过、红色失败、琥珀风险、蓝色信息。
避免大面积紫色渐变或装饰性背景。
```

重点组件：

```text
问题列表。
证据链时间线。
Agent 操作流程。
语义检查表。
健康报告。
悬浮健康子 Agent。
```

## 11. 验收标准

```text
进入健康系统页面能看到悬浮健康子 Agent 入口。
悬浮框可以展开、折叠，并绑定当前 issue / run。
健康子 Agent 默认不会自动扫描全部历史。
测试运行结束后可以主动生成健康分析报告。
其它场景需要用户发起或确认。
健康子 Agent 权限限定在健康系统内部文件、测试结果、trace、prompt、memory、assertion 和健康设置候选。
健康报告能展示对话、Agent 操作流程、问题节点、证据引用和语义检查。
页面不以文件列表和 raw JSON 作为主要体验。
```

## 12. 最终口径

健康子 Agent 是健康系统的“悬浮维护助理”，不是主 Agent 的复制体，也不是泛用客服。

它的价值是：

```text
把问题发现、证据读取、测试解释、语义审查、报告生成和用例草案串起来。
```

它的边界是：

```text
只在健康系统授权范围内读取、分析和生成候选。
正式修改、正式采纳和跨系统写入必须交给用户或后续治理链路。
```

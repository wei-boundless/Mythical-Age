# 健康系统 Harness 重构重审报告

日期：2026-04-30  
定位：本报告用于暂停当前前端推进，重新审视原“测试系统”的后端、数据链、链路追踪、问题记录、用例断言和后续健康维护 Agent 能力。目标是把它从“测试运行页面”升级为真正的 AgentRuntime Health Harness，并作为任务、编排、记忆、操作、灵魂、模型入口、前端等系统的统一健康维护层。

命名调整：后续不再把该系统理解为狭义“测试系统”，而是定义为 **健康系统**。测试运行、测试报告、用例管理只是健康系统的子能力。

---

## 0. 核心结论

健康系统不应该只是 `pytest` / `scenario runner` 的前端壳，也不应该只是展示后端文件路径。

健康系统也不应该只维护任务系统。它是系统级健康层，面向所有 AgentRuntime 子系统：

```text
任务系统：任务定义、任务状态、任务执行目标是否健康。
编排系统：RuntimeLoop、OperationGate、Directive、Checkpoint 是否健康。
记忆系统：对话记忆、状态记忆、长期记忆、上下文裁剪和写回是否健康。
操作系统：工具合同、资源策略、权限门禁、执行结果配对是否健康。
灵魂系统：投影、PromptManifest、模型可见上下文表达是否健康。
模型入口：模型请求、流式事件、工具调用协议、输出边界是否健康。
前端工作台：系统页面、报告阅读、管理表单、跳转链路是否健康。
测试运行层：自动化用例、断言、场景回归和修复验证是否健康。
```

合格的健康系统应该维护一条完整闭环：

```text
真实发现问题
  -> 问题记录
  -> 关联对话 / 任务 / skills / run / turn
  -> 读取 prompt / memory / RuntimeLoop / tool / assertion trace
  -> 生成复现用例或补测任务
  -> 运行验证
  -> 形成可读报告
  -> 修复后关闭问题
```

因此健康系统的主对象不能只有 `TestRun` 和 `TestCase`，还必须有：

```text
HealthIssue        真实问题记录
HealthTrace        每一次问题对话 / 异常任务 / 失败测试轮次的链路追踪
PromptTrace        本轮模型看到的 prompt 分层
MemoryTrace        本轮记忆读取、裁剪、注入、写回
AssertionTrace     用例断言、通过/失败、实际值
CaseDraft          从问题转化来的复现用例草案
FixVerification    修复验证记录
```

当前系统已经有部分基础，但还没有形成统一 health harness 合同。

最重要的权力边界：

```text
健康系统只记录问题对话，不记录所有对话。
一轮对话结束后，非问题运行材料应按策略清理或降级为引用。
长期事实、偏好、项目状态仍归记忆系统。
运行全量事件仍归编排系统 / RuntimeLoop。
健康系统只保留维护系统健康所需的错误证据、复现线索、断言结果和修复验证。
健康系统可以维护所有子系统的健康问题，但不能接管任何子系统的业务主权。
```

## 2. 健康系统的权力范围

健康系统的职责不是“保存所有运行数据”，而是维护 AgentRuntime 的健康闭环。

它的系统定位是：

```text
健康系统 = 系统级 Health Harness + 问题闭环 + 复现验证 + 健康报告。
```

它横跨所有系统，但只处理“健康问题”：

```text
任务系统出问题：健康系统记录任务问题、绑定 TaskContract / TaskRun / 任务状态证据。
编排系统出问题：健康系统记录 RuntimeLoop / Directive / Gate / Checkpoint 问题。
记忆系统出问题：健康系统记录 MemoryTrace / ContextPolicy / 写回门禁问题。
操作系统出问题：健康系统记录工具合同、权限策略、资源边界、调用结果问题。
灵魂系统出问题：健康系统记录投影、PromptManifest、模型可见表达问题。
模型入口出问题：健康系统记录模型调用、流式事件、工具调用协议、输出边界问题。
前端工作台出问题：健康系统记录页面交互、报告阅读、管理表单、跳转链路问题。
测试运行出问题：健康系统记录用例、断言、场景运行、修复验证问题。
```

健康系统的重点不是“谁负责这个系统”，而是“这个系统有什么健康问题、证据在哪、如何复现、如何验证修复”。

### 2.1 它拥有的权力

```text
记录问题：
  当用户在对话中发现异常、开发任务中发现异常、skills 调试失败、测试运行失败时，健康系统可以创建 HealthIssue。

绑定证据：
  健康系统可以保存少量问题对话摘录、run_id、turn_id、task_run_id、prompt_id、memory_trace_id、assertion_id、artifact_ref。

生成复现：
  健康系统可以把 HealthIssue 转成 CaseDraft，描述触发条件、期望行为和断言。

验证修复：
  健康系统可以记录某次 run / trace 是否验证了修复。

分析健康：
  健康系统可以读取编排、记忆、任务、操作、灵魂系统提供的 trace/ref，形成诊断报告。

跨系统归因：
  健康系统可以把一个问题标记为任务、编排、记忆、操作、灵魂、模型、前端、测试运行等 owner_system。

维护健康仪表：
  健康系统可以汇总各系统 open issue、failed assertion、recent regression、fix verification 状态。
```

### 2.2 它不拥有的权力

```text
不长期保存所有对话：
  全量对话历史和长期事实属于记忆系统。

不复制 RuntimeLoop 全量事件：
  RuntimeLoop 事实属于编排系统，健康系统只保存引用和诊断摘要。

不替代记忆系统：
  健康系统不能把问题记录当作长期记忆注入模型。

不替代任务系统：
  健康系统可以生成维护任务草案，但任务创建、子 agent 配置、任务状态归任务系统。

不替代操作系统：
  健康系统可以指出工具/权限问题，但不能直接放宽 OperationGate。

不静默修改正式测试：
  健康系统可以提出用例草案和建议，正式提升仍要经过 case registry / 人工确认。

不替代各系统管理页：
  健康系统只给出问题、证据和跳转入口。任务配置仍在任务系统；工具权限仍在操作系统；投影编辑仍在灵魂系统；记忆治理仍在记忆系统；RuntimeLoop 事实仍在编排系统。
```

### 2.3 对话记录策略

健康系统只记录“问题对话”，不记录普通对话。

建议一轮对话结束后执行清理：

```text
如果本轮没有错误、警告、用户标记问题、失败断言：
  不进入健康系统。
  只保留编排/记忆系统自身需要的运行事实和记忆结果。

如果本轮存在问题：
  健康系统创建 HealthIssue 或 ProblemCandidate。
  只保存必要摘录：
    user_excerpt
    assistant_excerpt
    symptom
    expected_behavior
    trace_refs
    assertion_refs
    problem_node_refs
  不保存完整对话正文。

如果问题已转成用例：
  HealthIssue 绑定 CaseDraft。
  后续主要依赖复现用例和 trace 引用，不继续堆积原始对话。

如果问题已修复验证：
  HealthIssue 标记 resolved。
  保留修复验证摘要和证据引用。
```

### 2.4 存储分工

```text
记忆系统：
  保存对话记忆、状态记忆、长期记忆。

编排系统：
  保存 RuntimeLoop events、checkpoints、state_index。

健康系统：
  保存 HealthIssue、CaseDraft、AssertionResult、FixVerification、健康摘要和证据引用。

测试运行产物：
  保存可复现 run 的产物，按保留策略清理。
```

健康系统的存储应该是“索引 + 摘要 + 证据引用”，不是“全量数据仓库”。

## 3. 正确的 Harness 目标

健康系统必须回答六个问题。

### 3.0 哪个系统不健康？

健康系统首先要能把问题归属到系统级 owner，而不是只归属到测试用例。

```text
HealthOwner
  owner_system:
    task_system
    orchestration_system
    memory_system
    operation_system
    soul_system
    model_runtime
    frontend_workspace
    health_system
    legacy_adapter
  subsystem
  component
  severity
  status
```

健康系统要能回答：

```text
当前哪个系统 open issue 最多？
哪个系统最近回归最多？
哪个系统有未验证修复？
哪个系统的断言覆盖不足？
哪个系统经常产生 warning / failed problem node？
```

### 3.1 发生了什么？

每次问题对话、任务异常、skills 调试失败或测试运行失败，都应该可以形成一条 `HealthTrace`：

```text
HealthTrace
  trace_id
  source_type: conversation | task | skill | test_run | manual
  owner_system
  session_id
  task_id
  task_run_id
  run_id
  turn_id
  user_input
  assistant_output
  status
  started_at
  ended_at
  artifacts
```

注意：普通成功对话不进入健康系统，只由编排/记忆系统按各自策略处理。

### 3.2 模型看到了什么？

Prompt 必须可读、可追踪、可断言：

```text
PromptTrace
  prompt_id
  trace_id
  total_chars
  assembly_order
  sections:
    section_id
    title
    layer: soul | system | memory | task | operation | turn | debug
    source
    model_visible
    chars
    preview
    order
```

健康系统要能回答：

```text
本轮灵魂投影是否进入 prompt？
任务合同是否进入 prompt？
记忆片段是否进入 prompt？
工具/操作合同是否进入 prompt？
是否有 debug-only 内容泄露给模型？
```

### 3.3 记忆发生了什么？

MemoryTrace 必须成为测试材料，不是附属调试面板：

```text
MemoryTrace
  trace_id
  context_pressure
  selected_sections
  dropped_sections
  session_memory:
    active_goal
    flow_state
    task_state
    context_slots
  durable_memory:
    exact_matches
    relevant_notes
  writeback:
    allowed
    blocked
    reason
```

健康系统要能断言：

```text
状态记忆是否被读取。
长期记忆是否被召回。
上下文裁剪是否符合策略。
写回是否经过 CommitGate。
本轮失败是否可能来自记忆缺失或错误记忆注入。
```

### 3.4 Agent 做了什么？

RuntimeLoop / Operation / Tool 需要统一进 HealthTrace：

```text
RuntimeTrace
  task_run_id
  event_count
  event_type_counts
  stages
  operation_gate
  tool_calls
  tool_results
  checkpoints
  commits
  terminal_reason
```

健康系统要能回答：

```text
OperationGate 为什么放行或拒绝？
工具调用和工具结果是否成对？
模型是否按 directive 调用工具？
CommitGate 是否写回 assistant session / memory？
失败发生在哪个 loop stage？
```

### 3.5 为什么失败？

问题节点不能只是 UI 推断，应该是后端可解释对象：

```text
ProblemNode
  node_id
  label
  system
  stage
  reason
  evidence_refs:
    prompt_section_id
    memory_section_id
    runtime_event_id
    assertion_id
    artifact_path
```

健康系统报告应该展示：

```text
问题标题
归属系统
触发来源
复现步骤
失败断言
问题节点
证据链
建议补测
```

## 5. 统一 Trace 合同

### 5.1 HealthTraceEnvelope

```text
HealthTraceEnvelope
  trace_id
  source
  owner:
    owner_system
    subsystem
    component
  identity:
    session_id
    task_id
    task_run_id
    run_id
    turn_id
  conversation:
    user_input
    assistant_output
    turn_summary
  prompt_trace
  memory_trace
  runtime_trace
  operation_trace
  assertion_trace
  problem_nodes
  artifacts
  status
```

### 5.2 可读性要求

任何 trace 不应该只返回原始 JSON。

每一层都要提供：

```text
raw_ref      原始产物引用
summary      人类可读摘要
evidence     关键证据片段
status       passed / warning / failed / unknown
problem_hint 失败时的定位提示
```

## 6. 问题与用例生命周期

### 6.1 HealthIssue 生命周期

```text
open
  用户、开发者、测试运行、测试 Agent 发现问题。

triaged
  已经归属系统，绑定 trace 和问题节点。

converted
  已生成 CaseDraft 或维护任务。

resolved
  修复已验证通过。

archived
  不再处理，但保留记录。
```

### 6.1.1 HealthIssue 系统归属

每个 HealthIssue 必须带系统归属，否则无法成为系统级健康能力。

```text
HealthIssue
  issue_id
  title
  owner_system
  subsystem
  component
  origin
  severity
  status
  symptom
  expected_behavior
  evidence_refs
  reproduction
  linked_case_draft_ids
  linked_verification_ids
```

owner_system 示例：

```text
task_system
orchestration_system
memory_system
operation_system
soul_system
model_runtime
frontend_workspace
health_system
legacy_adapter
```

### 6.3 CaseDraft 生命周期

```text
draft
  有问题和期望，但还没成为正式测试。

ready
  已有触发条件、断言、归属系统、profile。

promoted
  进入 case_registry。

rejected
  不适合变成自动测试，但保留原因。
```

## 7. 前端目标信息架构

健康系统前端建议重做为六个页面。

### 7.0 全系统健康总览

职责：

```text
按系统展示健康状态：
  任务系统
  编排系统
  记忆系统
  操作系统
  灵魂系统

  open issues
  recent failures
  failed assertions
```

不做：

```text

不复制各系统的管理页面。
只展示健康摘要、问题入口和跳转。
```

### 7.1 测试运行

职责：

```text

运行 chain / functional / system / scenario / long_*。
展示运行图。
展示最近 run 状态。
```

不做：

```text
不展示复杂报告。
不展示后端原始文件。
不做大而全监控中心。
```

### 7.2 健康报告

职责：

```text
展示人类可读结论。
展示对话轮次。
展示 Agent 操作流程。
展示失败断言。
展示问题节点。

```

不做：

```text

原始产物只能作为开发者折叠项。
```

### 7.3 链路追踪

职责：

```text
按 trace_id / run_id / turn_id 读取完整链。
分层展示：
  对话
  Prompt
  记忆
  RuntimeLoop
  工具
  断言
  问题节点
```

这是健康系统的核心页面之一。

### 7.4 健康分析

职责：

```text
读取问题、trace、断言、运行记录。
给后续测试 Agent 一个结构化输入面板。
生成诊断请求草稿。
生成补测建议。
```

现阶段可以是人工分析工作台，后续接测试 Agent。

### 7.5 问题与用例管理

职责：

```text
记录真实问题。
按 owner_system 过滤问题。
从问题生成用例草案。

查看断言。

```

这里的主对象是问题和用例，不是后端文件路径。

### 7.6 系统健康档案

职责：

```text
展示某个系统的健康档案：

  相关测试用例

```

健康档案不是系统管理页。它只回答“这个系统现在健康吗，哪里不健康，证据是什么”。


## 11. 最终判断

健康系统的核心不是“跑测试”，而是维护 AgentRuntime 的可解释健康闭环。

它必须能追踪：

```text
每一次问题对话为什么异常。
每一次异常任务归属哪个系统。
每一次模型看到了哪些 prompt。
每一次记忆读写是否正确。
每一次工具调用是否合规。
每一次编排决策是否遵守 RuntimeLoop 合同。
每一次灵魂投影是否正确进入模型可见上下文。
每一次前端交互是否正确接入后端健康能力。
每一个测试用例到底断言了什么。
每一个失败问题应该归到哪个系统、哪个节点、哪个证据。
每一次修复是否真的被验证。
```

只有做到这些，健康系统才能成为系统级健康中枢，支持后续开发 agent 的任务能力开发，而不是只做一个测试按钮集合。

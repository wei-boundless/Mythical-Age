# 用户任务导向健康系统重规划

日期：2026-04-30  
定位：本文件用于修正健康系统当前“按模块补页面”的设计倾向，把健康系统重规划为可管理、可追踪、可验证的 Harness 健康工作台。

> 2026-04-30 修订说明：第 18 章为新的执行基线。此前文档中“技术报告页”“链路证据页”“用例库管理页”的独立页面设想，只保留为历史讨论，不再作为实现目标。新的健康系统以“问题”为主对象，以“问题报告”为承载容器，以“测试资产工厂”为验证资源来源。

---

## 0. 核心判断

健康系统不应该只是：

```text
健康总览
验证与运行
问题中心
技术报告
链路证据
用例库
```

这些名字本身没有错，但如果每个页面只是填入对应名词的功能，就会变成“能看很多东西，但用户不知道从哪里下手”。

健康系统真正要解决的是：

```text
我遇到了什么问题？
问题发生在哪条链路？
证据是否足够定位？
哪个功能被影响？
有没有测试用例覆盖？
什么样的回答或链路算通过？
我改完后应该跑什么来证明已经修复？
这个问题以后如何沉淀成可维护的用例资产？
```

因此健康系统的设计重心要从“模块页面”转成“用户任务”。

## 1. 当前实现暴露出的体验问题

### 1.1 用例库还不是规范化资产

当前前端已经可以新增候选用例，但仍然把底层字段暴露给用户：

```text
owner_system
path
runner
profiles
```

这会让用户被迫理解实现细节。

正确体验应该是：

```text
用户选择：我要验证哪个功能。
用户描述：出现了什么问题。
用户定义：什么样的回答、链路或产物算通过。
系统推导：归属系统、建议测试文件、建议 profile、建议 runner。
高级用户才可以展开修改底层绑定。
```

也就是说，系统归属和路径不应该是主表单输入项，而应该是系统根据功能目录、模板和已有用例自动推导出来的绑定结果。

### 1.2 Harness 图谱还没有真正可视化

当前 Harness 图谱已经把测试文件、功能、问题、治理发现聚合到一起，但主视图仍偏列表。

健康系统需要展示的是链路关系：

```text
功能 Feature
  -> 健康用例 QualityCase
  -> 测试资产 TestAsset
  -> 运行计划 VerificationPlan
  -> 运行结果 VerificationRun
  -> 回合 Turn
  -> 证据 Evidence
  -> 问题 HealthIssue / ProblemNode
```

用户应该能一眼看出：

```text
哪个功能没有用例。
哪个问题没有覆盖用例。
哪个用例没有真实测试文件。
哪个测试文件最近失败。
哪个失败已经形成问题。
哪个问题已经被某次运行证明修复。
```

### 1.3 链路证据不应该只是事件摘要

像“正常工具与系统调用”“无节点编排决策”“当前产物没有工具调用摘要”这类展示，对开发者帮助有限。

它们的问题是：

```text
它们描述的是系统有没有数据。
不是用户关心的问题答案。
```

证据视图应该优先回答：

```text
这次失败发生在哪一步？
失败前发生了什么关键决策？
哪个输入触发了错误？
哪个 Agent / Lane / Policy / Tool / Prompt / Memory 参与了链路？
证据是否足以定位？
还缺哪个证据？
```

没有证据时也不应该显示成奇怪的“正常”。应该显示成：

```text
本次运行未产生工具调用证据。
影响：无法判断工具链路是否参与问题。
下一步：补采 ToolCallSummary 或改跑带 trace 的 profile。
```

### 1.4 技术报告不是可行动报告

当前技术报告偏“信息汇总”。用户真正需要的是决策报告：

```text
结论是什么？
置信度如何？
问题归属哪里？
证据链是否闭合？
哪个用例覆盖它？
应该新增还是修改用例？
下一步应该跑什么？
```

报告如果不能导向行动，就只是另一种形式的日志。

### 1.5 验证与运行没有和用例库形成闭环

理想状态应该是：

```text
从问题进入：看到覆盖用例和建议验证。
从用例进入：一键运行相关 profile。
从运行结果进入：看到影响了哪些用例和问题。
从报告进入：能采纳为用例草案或修复验证计划。
```

当前更像是：

```text
用例库管理一边。
验证与运行一边。
报告分析一边。
证据链一边。
```

这导致用户必须自己把信息拼起来。

## 2. 新设计原则

### 2.1 页面先问“用户为什么需要它”

每个页面都必须有明确任务：

```text
问题收件箱：我现在应该处理什么。
问题定位台：这个问题到底坏在哪里。
用例设计台：如何把问题沉淀成可维护测试资产。
验证运行台：如何证明修复有效且没有回退。
健康报告台：如何把一次分析变成可阅读、可采纳、可追踪的结论。
```

页面不是按系统模块切分，而是按用户任务切分。

### 2.2 默认隐藏底层实现字段

以下字段不应该出现在普通用户的主流程表单：

```text
owner_system
path
runner
profile id
raw trace path
raw artifact path
```

它们应该由系统推导，并以“绑定结果”的形式展示：

```text
归属功能：任务编排 / 工具调用 / 记忆注入 / Prompt 组装
建议测试资产：tests/xxx_regression.py
建议运行方式：pytest / runtime_loop profile
建议验证范围：单用例 / 功能 profile / 冒烟 profile
```

高级设置可以允许修改，但不应该成为默认体验。

### 2.3 Harness 是可视对象，不是后端测试目录

Harness 的前端抽象应该是：

```text
功能覆盖图
问题覆盖图
用例生命周期
运行证明链
证据图谱
```

测试文件只是其中一种资产。

### 2.4 用例库是测试体系的具象化

用例库不是“测试文件列表”。

它应该表达：

```text
这个用例验证什么功能。
它防止什么问题再次发生。
通过标准是什么。
它绑定哪些测试文件、profiles、断言和语义检查。
最近一次验证结论是什么。
它当前是草案、候选、正式、隔离还是废弃。
```

### 2.5 所有视图围绕同一个健康对象联动

健康系统需要一个统一对象，建议命名为：

```text
QualityCase
```

它不是单纯测试用例，也不是单纯问题。它是“一个可被验证的健康管理单元”。

## 3. 核心对象模型

### 3.1 FeatureCatalog

功能目录负责把用户语言映射到系统归属。

```text
feature_id
title
description
owner_system
domain
default_profiles
default_runner
suggested_test_dir
related_components
```

用户选择的是 `title`，不是 `owner_system`。

### 3.2 QualityCase

规范化健康用例。

```text
quality_case_id
title
feature_id
status: draft | candidate | active | quarantined | retired
source: manual | issue | report | run_failure | agent_suggestion
user_problem
expected_behavior
acceptance_criteria
negative_examples
evidence_refs
issue_refs
asset_bindings
verification_policy
last_verification
owner_system
created_at
updated_at
```

其中 `owner_system` 是派生字段，不由默认表单填写。

### 3.3 CaseTemplate

用例模板不是几个字段预填，而是标准化建模向导。

```text
template_id
title
applies_to_features
scenario_questions
acceptance_criteria_schema
evidence_requirements
recommended_assertion_types
recommended_verification_policy
asset_naming_policy
```

模板应该让用户回答业务问题：

```text
用户输入是什么？
期望回答是什么？
什么回答算失败？
需要看到哪些链路证据？
是否需要工具调用？
是否需要记忆引用？
是否需要 OperationGate 决策？
```

而不是让用户填：

```text
path = tests/xxx.py
owner_system = operation_system
```

### 3.4 TestAssetBinding

测试资产绑定是系统推导出的结果。

```text
asset_id
quality_case_id
asset_type: pytest | runtime_profile | semantic_assertion | manual_check
suggested_path
actual_path
runner
profiles
assertion_refs
binding_status: suggested | bound | missing | stale
```

这样可以支持“还没有真实测试文件，但已有规范化用例”的状态。

### 3.5 VerificationRun

运行结果必须回写到用例。

```text
run_id
quality_case_refs
profile_refs
status
started_at
finished_at
failed_criteria
passed_criteria
artifact_refs
evidence_graph_ref
linked_issue_refs
next_action
```

### 3.6 EvidenceGraph

证据图谱用于替代散乱时间线。

```text
nodes:
  issue
  quality_case
  feature
  run
  turn
  agent
  prompt
  memory
  tool
  operation_gate
  assertion
  artifact

edges:
  caused_by
  observed_in
  verified_by
  covered_by
  depends_on
  missing_evidence
```

## 4. 新信息架构

建议把健康系统重构为一个工作台，而不是平级 Tab 集合。

```text
左侧：健康任务队列
中间：当前健康对象工作区
右侧：证据与建议 Inspector
底部或右下：健康子 Agent 悬浮入口
```

### 4.1 健康任务队列

回答：

```text
我现在应该处理什么？
```

队列项包括：

```text
新问题未定位
失败运行未解释
报告建议未采纳
候选用例未绑定测试资产
正式用例最近失败
功能缺少覆盖
证据链不完整
```

每个队列项必须有明确动作：

```text
定位问题
补证据
生成用例
绑定测试资产
运行验证
采纳报告
关闭问题
```

### 4.2 问题定位工作区

回答：

```text
这个问题坏在哪里？
```

主视图展示：

```text
问题摘要
影响功能
疑似失败节点
证据闭合度
最近相关运行
已覆盖用例
建议下一步
```

不要默认展示 raw trace。

### 4.3 Harness 覆盖工作区

回答：

```text
我们的测试体系现在覆盖了什么，漏了什么？
```

视图应该包括：

```text
功能覆盖矩阵
问题到用例的覆盖关系
用例到测试资产的绑定关系
运行状态热力
无覆盖功能列表
无资产候选用例列表
失败后未生成问题的运行列表
```

### 4.4 用例设计工作区

回答：

```text
如何把一次问题变成可维护测试用例？
```

默认流程：

```text
1. 选择功能
2. 选择模板
3. 描述问题
4. 定义期望行为
5. 定义通过标准
6. 系统生成建议测试资产绑定
7. 用户确认为草案 / 候选 / 正式
```

主表单不出现 `owner_system` 和 `path`。

### 4.5 验证运行工作区

回答：

```text
我应该运行什么来证明问题修复？
```

运行入口必须来自：

```text
问题
用例
功能
报告建议
```

运行结束必须回写：

```text
关联用例状态
关联问题状态
失败通过标准
证据图谱
下一步建议
```

### 4.6 健康报告工作区

回答：

```text
这次分析的结论是什么，我该怎么行动？
```

报告结构调整为：

```text
结论卡片
置信度
影响范围
失败节点
用户期望 vs 实际表现
证据链
覆盖用例
缺失证据
建议新增/修改用例
建议运行计划
采纳动作
```

报告不只是读物，它应该能产生动作：

```text
生成 QualityCase
更新已有 QualityCase
创建 VerificationPlan
关联 HealthIssue
标记证据缺口
```

## 5. 前端体验重设计

### 5.1 选择体验

当前“未选择理解与建模 / 等待选择测试轮次”这类状态过于内部化。

新的选择模型应为：

```text
全局选择对象：issue | quality_case | run | feature | report
上下文选择对象：turn | evidence_node | asset_binding
```

空状态要表达用户下一步：

```text
未选择问题：
  从左侧任务队列选择一个问题，或从最近失败运行创建问题。

未选择运行：
  选择一个用例后可以运行验证，或查看最近运行结果。

缺少工具证据：
  本次运行没有工具调用摘要，因此无法判断工具链路是否参与问题。建议使用带 trace 的 profile 重新运行。
```

### 5.2 Harness 可视化组件

建议组件：

```text
FeatureCoverageMatrix
QualityCaseGraph
EvidenceGraphView
CaseAssetBindingPanel
VerificationRunPanel
HealthTaskInbox
DecisionReportView
CaseDesignerWizard
```

第一版可以不做复杂画布，但必须让关系可见：

```text
Feature -> Case -> Asset -> Run -> Evidence -> Issue
```

可以先用分栏关系图：

```text
左列 Feature
中列 QualityCase
右列 TestAsset / Run
底部 Evidence / Issue
```

### 5.3 用例库管理体验

用例卡片应该显示：

```text
功能
问题
通过标准
绑定资产
最近运行
状态
下一步动作
```

操作按钮：

```text
编辑规范
绑定资产
运行验证
从报告更新
采纳为正式
隔离
删除草案
```

### 5.4 技术报告体验

报告首页不要先展示元数据和原始证据数量。

应该先展示：

```text
结论：是否存在问题。
定位：疑似失败节点。
影响：哪个功能/用例受影响。
证明：哪些证据支撑。
缺口：哪些证据缺失。
动作：下一步建议。
```

元数据、raw payload、artifact path 放到详情折叠区。

### 5.5 健康子 Agent 的入口方式

健康子 Agent 不应该只是聊天框。

它应该能围绕当前对象给出动作：

```text
解释这个失败
找出缺失证据
把这个问题整理成用例
建议通过标准
生成验证计划
对比两次运行
检查报告是否可采纳
```

## 6. 后端/API 重规划

第一阶段不需要推翻已有接口，而是在现有接口上增加工作台投影层。

### 6.1 工作台投影接口

```text
GET /api/health-workbench/inbox
GET /api/health-workbench/features
GET /api/health-workbench/cases
GET /api/health-workbench/cases/{case_id}
POST /api/health-workbench/cases
PUT /api/health-workbench/cases/{case_id}
DELETE /api/health-workbench/cases/{case_id}
POST /api/health-workbench/cases/{case_id}/derive-asset
POST /api/health-workbench/cases/{case_id}/verify
GET /api/health-workbench/evidence-graph
GET /api/health-workbench/reports/{report_id}/decision
POST /api/health-workbench/reports/{report_id}/adopt
```

### 6.2 与现有接口关系

现有接口继续保留：

```text
/api/test-system/harness-map
/api/test-system/case-templates
/api/test-system/managed-cases
/api/health-system/issues
/api/test-system/runs
/api/test-system/artifacts
```

新增工作台接口负责把它们组合成用户可理解对象。

### 6.3 自动推导规则

新增：

```text
FeatureCatalogService
CaseTemplateService
CaseAssetDeriver
EvidenceGraphBuilder
DecisionReportBuilder
VerificationPlanner
```

推导逻辑：

```text
feature_id -> owner_system
feature_id + template -> suggested_test_dir
case title + naming policy -> suggested_path
case acceptance criteria -> assertion skeleton
case feature -> default profiles
issue evidence -> related feature/case candidates
run artifacts -> failed criteria and evidence graph
```

## 7. 迁移路径

### Phase 0：冻结“底层字段外露”继续扩散

目标：

```text
不再新增要求用户填写 owner_system/path 的主流程。
```

动作：

```text
把现有用例新增表单改成向导式。
owner_system 改为功能选择后自动派生。
path 改为系统建议绑定，放入高级设置。
```

### Phase 1：建立 FeatureCatalog 与 QualityCase 投影

目标：

```text
前端看到的是功能和健康用例，不是测试文件和 owner_system。
```

动作：

```text
从现有 owner_system、profiles、managed_cases、case_registry 派生 FeatureCatalog。
把 managed_cases 升级为 QualityCase 投影。
```

### Phase 2：用例设计台

目标：

```text
用户可以在前端模板化新增、减少、编辑用例。
```

动作：

```text
实现 CaseDesignerWizard。
实现模板问题 schema。
实现自动建议测试资产绑定。
支持草案、候选、正式三种状态。
```

### Phase 3：Harness 覆盖图谱

目标：

```text
让用户直观看到测试体系覆盖关系和缺口。
```

动作：

```text
实现 FeatureCoverageMatrix。
实现 CaseAssetBindingPanel。
实现缺口过滤器：
  功能无用例
  用例无资产
  问题无用例
  失败无报告
  证据不闭合
```

### Phase 4：验证运行闭环

目标：

```text
运行从用例/问题/报告发起，并回写结果。
```

动作：

```text
实现 VerificationPlanner。
实现 case -> run。
实现 run -> case status。
实现 failed criteria 映射。
```

### Phase 5：决策报告与证据图谱

目标：

```text
报告能回答问题，证据能支撑定位。
```

动作：

```text
实现 EvidenceGraphBuilder。
实现 DecisionReportView。
报告支持采纳为用例、验证计划、问题关联。
```

### Phase 6：逐步弱化旧 Tab

目标：

```text
旧的模块 Tab 成为工作台内部视图，不再主导体验。
```

动作：

```text
把“验证与运行、用例库、链路证据、技术报告”收束到同一个当前对象上下文中。
```

## 8. 验收标准

第一轮改造完成后，用户应该能做到：

```text
不填写 owner_system，也能创建规范化用例。
不填写 path，也能得到建议测试资产绑定。
从一个问题看到它覆盖了哪些用例。
从一个用例看到它验证哪个功能、绑定哪些测试资产。
从一个用例直接运行验证。
从运行结果看到失败了哪个通过标准。
从技术报告一眼看到结论、定位、证据、缺口和下一步。
从 Harness 图谱看到功能覆盖、问题覆盖和测试资产缺口。
```

如果这些做不到，就说明健康系统仍只是“展示测试系统数据”，还没有成为真正的 Harness 管理工作台。

## 10. 设计口径

健康系统不是“给每个模块补一个页面”。

健康系统是：

```text
把用户遇到的问题转成可定位的证据链。
把证据链转成可采纳的健康报告。
把健康报告转成规范化用例。
把规范化用例转成可运行验证。
把运行结果回写成系统健康状态。
```

这才是 Harness 体系的可视化，也是健康系统存在的理由。

## 12. 首版页面详细设计

### 12.1 健康态势条

职责：

```text
让用户知道系统健康工作台当前是否需要处理。
```

展示：

```text
待处理任务数
开放问题数
未覆盖功能数
候选用例数
最近验证状态
```

避免：

```text
不要塞 8 个以上指标。
不要展示没有行动含义的统计。
不要把运行按钮作为唯一主动作。
```

推荐交互：

```text
点击“开放问题”过滤左侧任务队列。
点击“候选用例”进入用例设计上下文。
点击“最近验证失败”进入对应 VerificationRun。
```

### 12.2 HealthTaskInbox

职责：

```text
把系统状态转成用户任务。
```

任务类型：

```text
issue.needs_triage:
  标题：有问题等待定位
  主动作：定位问题

evidence.missing:
  标题：证据不足，无法判断链路
  主动作：补采证据

case.needs_design:
  标题：问题尚未沉淀为用例
  主动作：设计用例

asset.needs_binding:
  标题：用例没有绑定测试资产
  主动作：绑定资产

verification.failed:
  标题：验证失败等待解释
  主动作：解释失败

report.needs_adoption:
  标题：报告建议等待采纳
  主动作：采纳建议
```

每个任务卡片必须显示：

```text
任务标题
影响对象
风险等级
为什么现在要处理
主动作
```

不显示：

```text
backend path
raw ids
内部 task_mode
```

### 12.3 IssueTriageWorkspace

职责：

```text
帮助用户判断“问题坏在哪里、证据够不够、下一步是什么”。
```

布局：

```text
顶部：问题摘要与当前结论
中部：定位路径
  用户表现
  影响功能
  疑似问题节点
  证据闭合度
  覆盖用例
底部：下一步动作
  运行健康分析
  补采证据
  设计用例
  创建验证计划
```

空状态必须有行动提示：

```text
没有运行证据：
  当前只能记录问题，不能定位链路。建议从相关用例运行一次带 trace 的验证。

没有覆盖用例：
  该问题还没有可复用验证单元。建议创建 QualityCase。
```

### 12.4 QualityCaseWorkspace

职责：

```text
把问题转成可维护、可运行、可验证的健康用例。
```

默认显示：

```text
用例标题
验证功能
用户问题
期望行为
通过标准
绑定资产
最近验证
下一步动作
```

编辑向导：

```text
Step 1：选择功能
Step 2：描述问题
Step 3：定义期望行为
Step 4：定义失败样例
Step 5：定义通过标准
Step 6：系统建议测试资产绑定
Step 7：保存为草案/候选/正式
```

默认不让用户填写：

```text
owner_system
path
runner
profile id
```

这些由系统基于 `feature_id` 和模板推导。

### 12.5 FeatureCoverageWorkspace

职责：

```text
让用户看到健康体系覆盖了什么、漏了什么。
```

第一版不用复杂画布，用三列关系图：

```text
Feature 列：
  功能名称
  覆盖状态
  问题数

QualityCase 列：
  用例名称
  状态
  通过标准数量

Asset / Run 列：
  绑定资产
  最近运行
  失败标准
```

筛选器：

```text
功能无用例
用例无资产
问题无用例
失败无报告
证据不闭合
候选待采纳
```

### 12.6 VerificationRunWorkspace

职责：

```text
从用例、问题或报告发起验证，并解释验证结果。
```

运行前展示：

```text
为什么要跑
会覆盖哪些用例
会验证哪些通过标准
预计 profile
是否需要 trace
```

运行后展示：

```text
通过/失败
失败通过标准
问题节点
证据图谱
建议下一步
```

不要只显示：

```text
passed / failed 数字
runner log
raw artifacts
```

### 12.7 DecisionReportWorkspace

职责：

```text
把分析结果变成用户能采纳的行动报告。
```

报告首屏顺序固定：

```text
1. 结论
2. 置信度
3. 影响范围
4. 失败节点
5. 用户期望 vs 实际表现
6. 证据链
7. 缺失证据
8. 建议动作
```

报告动作：

```text
采纳为 QualityCase
更新已有 QualityCase
创建 VerificationPlan
关联 HealthIssue
标记证据缺口
关闭问题
```

## 14. 视觉与排版 QA 标准

本系统是工程健康工作台，视觉目标是：

```text
清晰
克制
高密度但不拥挤
任务优先
状态可扫描
证据可追踪
```

### 14.1 必查项

每次前端实现后必须检查：

```text
首屏是否能看出下一步动作。
是否有两个以上区域在展示同一类统计。
是否把 raw id/path/json 暴露成主内容。
卡片数量是否过多导致用户不知道先看哪里。
按钮文案是否是用户动作，而不是内部流程名。
长标题、长路径、长结论是否溢出。
移动端是否出现横向滚动或按钮挤压。
空状态是否告诉用户下一步。
错误状态是否解释影响和恢复方式。
```

### 14.2 信息密度规则

```text
顶部态势条最多 5 个指标。
Inbox 卡片最多 2 行摘要 + 1 个主动作。
工作区主结论必须在首屏上半区。
raw payload 默认折叠。
同一对象不要同时出现在 3 个并列卡片里。
```

### 14.3 排版规则

```text
工作台采用三栏布局，但移动端收束为单列。
左侧任务队列固定宽度，避免随标题变形。
中间主区使用明确标题层级，不用过大的 hero 字号。
右侧 Inspector 控制在 320-380px。
所有按钮必须有稳定高度。
所有列表项必须允许长文本换行。
```

## 17. Token 与链路耗时健康维度

健康系统必须把效率也纳入维护范围。一个链路“能跑通”但 token 消耗失控、耗时持续升高，本质上也是健康问题；它会影响用户体验、成本、长任务稳定性和上下文质量。

### 17.1 目标

```text
让用户知道一次回答或一次验证：
  花了多少 token。
  慢在模型、工具、记忆、检索、编排还是测试执行。
  是否触发上下文压缩。
  是否超过该任务类型的健康阈值。
  下一步应该优化提示、裁剪上下文、缓存证据、拆分任务还是修复慢节点。
```

### 17.2 新增健康对象

```text
EfficiencySignal
  signal_id
  signal_type: token_pressure | latency_spike | slow_tool | slow_model | slow_memory | slow_test
  subject_type: session | task_run | test_run | turn | tool_call | quality_case
  subject_id
  metric
  value
  baseline
  threshold
  severity
  reason
  recommended_action
  evidence_refs
```

```text
CostAndLatencySnapshot
  snapshot_id
  session_id
  task_run_id
  test_run_id
  total_tokens
  prompt_tokens
  completion_tokens
  history_tokens
  history_usage_ratio
  context_pressure_level
  did_compact
  total_duration_ms
  model_duration_ms
  tool_duration_ms
  memory_duration_ms
  orchestration_duration_ms
  test_duration_ms
  bottleneck_stage
```

### 17.3 数据来源

当前已有来源：

```text
/api/tokens/session/{session_id}
  system_tokens
  message_tokens
  total_tokens
  raw_history_tokens
  history_tokens
  history_budget_tokens
  history_usage_ratio
  history_pressure_level
  history_did_compact

test_system.list_runs()
  run_id
  profile
  status
  duration_ms
  summary

runtime_loop / trace events
  event_count
  event_type_counts
  stage events
  tool call/result events
  terminal reason
```

需要补齐来源：

```text
模型响应阶段耗时。
工具调用逐项耗时。
记忆构建/压缩耗时。
每轮 turn 的 token 快照。
每个 task_run 的效率快照。
基于 profile/task_mode 的耗时与 token baseline。
```

### 17.4 前端呈现

效率信息不应该变成另一堆 raw metric，而要进入同一个健康工作台：

```text
任务收件箱：
  token 压力过高 -> “上下文成本偏高”
  耗时突增 -> “链路耗时异常”
  某工具慢 -> “工具节点阻塞”

当前对象区：
  展示本对象的 token/耗时摘要、瓶颈阶段、是否超阈值。

右侧行动面板：
  建议压缩上下文、缩小证据范围、缓存慢查询、拆分长任务、补充性能用例。

链路证据页：
  在节点时间线上标出每个阶段耗时。
  支持按耗时排序和只看慢节点。

技术报告：
  报告必须包含“正确性风险”和“效率风险”两栏。
```

### 17.5 阈值策略

第一阶段使用保守静态阈值：

```text
history_usage_ratio >= 0.80 -> warning
history_usage_ratio >= 0.95 -> critical
单轮响应 duration_ms >= 30_000 -> warning
任务链路 duration_ms >= 120_000 -> warning
验证运行 duration_ms >= 180_000 -> warning
同一 profile 比最近 5 次均值高 50% -> latency_spike
```

第二阶段引入动态 baseline：

```text
按 task_mode/profile/quality_case 维护 rolling baseline。
使用 p50/p90/p95 判断异常。
只把连续异常或高影响异常推入 Inbox，避免噪声。
```

### 17.6 施工顺序

```text
Step A：HealthWorkbenchOverview 增加 efficiency 字段。
  先接 test_run.duration_ms，形成 slow_run_count、max_duration_ms、efficiency signals。

Step B：新增 health_system/efficiency.py。
  聚合 session token、task_run trace、test_run duration。

Step C：runtime_loop 事件补充 duration_ms。
  模型、工具、记忆、编排节点都需要开始/结束时间或可推导耗时。

Step D：前端工作台增加“效率风险”小节。
  不单独开复杂大屏，而是在收件箱、当前对象、证据页、报告页各自出现。

Step E：用例库支持性能/成本类 QualityCase。
  用例能表达“这个任务不应超过多少 token/耗时”。
```

## 18. 以问题为中心的修订版信息架构

本章替代前文中“总览 / 验证运行 / 问题处理 / 技术报告 / 链路证据 / 用例库”的并列页面结构。

新的判断是：

```text
健康系统不是文档中心。
健康系统不是日志浏览器。
健康系统不是测试文件列表。

健康系统是问题维护工作台：
  发现问题。
  分层归类问题。
  定位问题链路。
  形成问题报告。
  生成或选择验证资源。
  运行验证。
  用结果关闭或重新打开问题。
```

### 18.1 总览只回答用户打开页面时最想知道的事

总览不再展示泛化标语，不再重复已经由其他模块承担的统计项，例如“用例数量”“治理发现数量”等。

总览只保留四类信息：

```text
系统健康状态：
  当前是否可用。
  是否有阻塞问题。
  是否有 token / 耗时异常。
  最近一次健康检查结论。

最近验证：
  最近一次运行是什么。
  结果是通过、失败、运行中还是未运行。
  如果失败，失败问题是否已经生成问题报告。

最近问题：
  按问题类型分组展示最近需要处理的问题。
  用户能一键进入问题报告。

当前问题报告：
  最近打开或最高优先级问题的报告摘要。
  展示“哪里坏了 / 为什么坏 / 下一步”。
```

总览不应该出现：

```text
从运行验证到健康报告，一条链路完成复盘。
用例资产数量。
正式用例数量。
候选草案数量。
技术报告这种抽象入口。
链路证据这种孤立入口。
```

因为这些不是用户打开健康系统时要决策的内容。

### 18.2 问题必须分层

问题不是单一类型。系统健康维护至少要区分以下问题层：

```text
UserReportedIssue
  用户说“回答不对 / 体验异常 / 等太久 / 没达到预期”。
  来源：对话、手动记录、反馈入口。
  维护者：健康分诊子 Agent。

SystemIssue
  系统链路本身异常，例如任务、编排、记忆、工具、模型、权限、灵魂、投影链路问题。
  来源：runtime trace、health agent、系统监控。
  维护者：对应系统子 Agent + 健康维护子 Agent。

VerificationIssue
  验证运行失败、长测失败、回归失败。
  来源：VerificationRun。
  维护者：验证规划子 Agent。

TestAssetIssue
  测试资产自身的问题，例如用例缺失、测试文件失效、flaky、覆盖关系不明、候选用例未落地。
  来源：测试资产工厂、Harness Map。
  维护者：测试资产子 Agent。

EfficiencyIssue
  token 消耗、上下文压力、链路耗时、工具耗时、模型耗时异常。
  来源：token API、runtime loop、verification duration。
  维护者：效率分析子 Agent。
```

问题列表必须按类型分组，而不是混在一个“问题处理”列表里。用户需要先知道“这是什么类型的问题”，再看严重程度和处理人。

### 18.3 问题报告替代“技术报告”

“技术报告”这个名称不准确。健康系统需要的是“问题报告”。

问题报告的定义：

```text
ProblemReport
  绑定一个问题。
  解释这个问题是什么。
  展示问题发生在哪里。
  展示为什么会发生。
  展示证据链是否闭合。
  展示影响范围。
  展示应该怎么验证修复。
  展示是否需要新增或更新测试资产。
```

问题报告不是单独的文章页，而是问题对象的详情页。用户点击任何问题，都进入同一个问题报告。

问题报告首屏顺序固定：

```text
1. 问题摘要
   标题、类型、严重程度、状态、维护子 Agent。

2. 三个核心答案
   哪里出了问题。
   为什么出了问题。
   下一步怎么处理。

3. 证据闭合度
   证据充分 / 证据不足 / 缺少运行链路 / 缺少测试复现 / 缺少用户输入。

4. 链路分析
   问题节点关系图。

5. 验证建议
   应该运行哪个验证计划。
   是否需要生成测试资产。
```

旧的“技术引用”“运行通道”“workflow id”“projection id”“prompt manifest id”只能放在高级详情中。

### 18.4 链路证据并入问题报告

“链路证据”不应该是独立页面。用户不是为了看链路而看链路，用户是为了定位一个问题才看链路。

因此链路证据变成问题报告中的一个区块：

```text
ProblemReport
  EvidenceGraph
    用户输入
    任务理解
    编排决策
    上下文 / Prompt / Memory
    模型响应
    工具 / 权限 / 操作
    输出提交
    验证结果
```

EvidenceGraph 首屏必须回答：

```text
问题节点在哪里？
上游输入是什么？
下游影响是什么？
哪个节点缺证据？
哪个节点由哪个子 Agent 负责维护？
```

原始事件列表只能作为“支撑事件”折叠展示。

### 18.5 验证运行是封装入口，不是资源管理入口

验证运行的职责是：

```text
用户选择目标：
  验证一个问题是否修复。
  验证一个功能是否健康。
  验证一次改动是否回归。
  验证一次长链是否稳定。

系统生成 VerificationPlan：
  选择用例。
  选择 profile。
  选择运行范围。
  解释为什么跑这些。
```

验证运行不负责让用户管理测试文件。它只消费测试资产工厂提供的资源。

### 18.6 用例库改名为测试资产工厂

“用例库”这个名字容易让用户以为它只是列表。实际目标应该是：

```text
TestAssetFactory
  帮用户从基本信息生成合格测试资产。
  帮验证运行选择可用资源。
  帮问题报告沉淀回归用例。
```

用户默认只需要输入：

```text
我要验证什么功能？
遇到了什么问题？
什么样的回答 / 行为 / 链路算通过？
有没有一个失败样例或期望样例？
```

系统自动推导：

```text
归属系统。
归属功能。
测试层级。
建议 runner。
建议 profile。
建议测试文件路径。
建议断言。
建议 fixtures。
是否更新已有测试，还是新建测试。
```

高级字段只在“展开技术绑定”中出现：

```text
owner_system
path
runner
profiles
tags
assertions
```

测试资产工厂的输出不是“候选记录”这么简单，而是一个可执行的生成结果：

```text
QualityCaseDraft
  用户意图。
  通过标准。
  系统推导绑定。
  建议测试文件。
  生成的测试代码预览。
  需要用户确认的风险。

TestAssetPatch
  文件路径。
  diff。
  新增或更新的测试函数。
  运行命令。
  回滚方式。
```

### 18.7 子 Agent 维护边界

健康系统中每个问题都应该能看到维护责任，而不是只看到页面。

```text
健康分诊子 Agent：
  负责问题类型识别、严重程度、是否需要报告。

链路分析子 Agent：
  负责 EvidenceGraph、问题节点、缺失证据。

验证规划子 Agent：
  负责 VerificationPlan、运行范围、回归策略。

测试资产子 Agent：
  负责 QualityCaseDraft、TestAssetPatch、测试文件生成。

效率分析子 Agent：
  负责 token、耗时、上下文压力、慢节点判断。

系统维护子 Agent：
  负责具体系统边界，例如任务系统、记忆系统、操作系统、编排系统。
```

前端不应该把子 Agent 当装饰。每个问题报告应该明确：

```text
谁负责定位。
谁负责生成测试。
谁负责验证。
谁负责最终关闭。
```

### 18.8 新页面结构

最终页面结构：

```text
HealthOverview
  系统健康状态
  最近验证
  最近问题
  当前问题报告摘要

ProblemCenter
  分层问题收件箱
  类型筛选
  维护子 Agent
  状态流转

ProblemReport
  问题摘要
  三个核心答案
  EvidenceGraph
  验证建议
  测试资产建议
  高级技术引用

VerificationCenter
  按目标生成验证计划
  运行验证
  把失败转为问题
  把通过写回问题报告

TimeStatistics
  验证耗时趋势
  慢链路告警
  每日 token 消耗
  每 6 小时 token 消耗
  token 数据来源与估算口径

TestAssetFactory
  基本信息输入
  自动推导绑定
  生成测试文件预览
  运行验证
  采纳为测试资产
```

不再有独立的：

```text
技术报告页
链路证据页
裸用例库页
```

### 18.9 首轮改造顺序

```text
Step 1：前端导航改名和降噪
  总览
  问题中心
  验证中心
  测试资产工厂
  时间统计

Step 2：总览重做
  移除重复指标和标语。
  只保留系统健康状态、最近验证、最近问题、当前问题报告摘要。

Step 3：问题中心分层
  将问题按 UserReportedIssue / SystemIssue / VerificationIssue / TestAssetIssue / EfficiencyIssue 分组。
  每个问题显示维护子 Agent。

Step 4：问题报告取代技术报告
  点击问题进入 ProblemReport。
  报告内包含 EvidenceGraph。
  删除独立链路证据入口。

Step 5：测试资产工厂取代用例库
  默认表单只保留用户意图、问题描述、通过标准、样例。
  系统自动推导 owner/path/runner/profile。
  输出测试文件预览和运行命令。

Step 6：验证中心消费测试资产
  验证运行不直接管理测试文件。
  它从问题报告或功能目标生成 VerificationPlan。

Step 7：时间统计纳入健康维护
  token 消耗进入独立时间统计页。
  按每天、每 6 小时展示 token 消耗。
  当前先按会话历史估算，后续可替换为模型 usage 账单。
  链路耗时作为 EfficiencyIssue 进入健康维护。
  问题报告内展示与当前问题相关的耗时证据。
```

### 18.10 验收标准

```text
用户打开总览：
  不需要读口号。
  直接知道系统是否健康、最近验证是否失败、最近问题是什么、优先看哪份问题报告。

用户点击一个问题：
  首屏看到哪里坏、为什么坏、下一步。
  同屏或下方看到链路节点关系图。

用户想新增测试：
  不需要填写 owner_system/path/runner/profile。
  输入基本问题和通过标准后，系统给出合格测试文件草案。

用户运行验证：
  看到为什么跑这些测试。
  失败自动转成问题。
  通过能写回问题报告。

用户查看测试资产：
  看到它服务哪个问题、哪个功能、哪个验证计划。
  不是看到一堆不知道用途的测试字段。

用户查看时间统计：
  看到验证平均耗时、最长耗时、慢链路告警。
  看到每日 token 消耗和最近 24 小时每 6 小时 token 消耗。
  如果耗时异常，能从问题报告里看到它作为定位证据，而不是孤立数字。
```

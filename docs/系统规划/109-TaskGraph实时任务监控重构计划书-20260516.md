# 109-TaskGraph 实时任务监控重构计划书

## 1. 问题结论

当前所谓“实时监控”不是合格的 TaskGraph 运行监控。

它现在只是从 `TaskRun`、`CoordinationRun`、checkpoint、trace summary、node runs、handoff envelopes 中拼出来的摘要视图。前端还会再次自行拼装拓扑和状态，因此同一个运行实例会出现以下错误：

1. 静态任务图有边，但前端画布不显示边。
2. 当前运行明明绑定 `graph.writing.simple_novel`，前端却可能呈现为流程阶段视图。
3. `coordination_graph_spec`、`coordination_flow`、`langgraph_runtime_state`、`node_runs` 被混用，结构源和状态源边界不清。
4. 监控按 session 取 latest 时，可能看到废弃 run、旧 run 或错误 run。
5. 前端没有明确显示数据来源、图 ID、节点数、边数、checkpoint 时间、事件偏移，用户无法判断监控是否可信。

这不是样式问题，也不是简单的“边没有画出来”。这是监控模型设计缺失。

## 2. 当前代码事实

### 2.1 后端入口

当前 API：

1. `GET /api/orchestration/runtime-loop/sessions/{session_id}/live-monitor`
   - 实现位置：`backend/api/orchestration.py`
   - 调用：`TaskRunLoop.get_session_live_monitor`
   - 当前行为：按 session 选一个 latest task run，再返回其 live monitor。

2. `GET /api/orchestration/runtime-loop/task-runs/{task_run_id}/live-monitor`
   - 实现位置：`backend/api/orchestration.py`
   - 调用：`TaskRunLoop.get_task_run_live_monitor`
   - 当前行为：读取 task run、coordination run、checkpoint 和 state index 快照，拼成监控摘要。

核心实现：

1. `backend/orchestration/runtime_loop/task_run_loop.py`
   - `get_session_live_monitor`
   - `get_task_run_live_monitor`

2. `backend/orchestration/runtime_loop/trace_reader.py`
   - `RuntimeLoopTraceReader.get_session_live_monitor`
   - `RuntimeLoopTraceReader.get_task_run_live_monitor`
   - `_get_task_run_live_monitor_from_snapshot`

3. `backend/orchestration/runtime_loop/coordination_trace_adapter.py`
   - 把 LangGraph checkpoint state 投影为 `coordination_flow`
   - 同时在 diagnostics 中写入 `langgraph_runtime_state`

4. `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
   - checkpoint 中保存真正的运行状态：
     - `active_stage_id`
     - `stage_order`
     - `stage_execution_request`
     - `stage_results`
     - `completed_nodes`
     - `failed_nodes`
     - `working_memory_operations`
     - `diagnostics.coordination_graph_spec`
     - `diagnostics.task_graph_scheduler_state`

### 2.2 前端入口

当前前端调用：

1. `frontend/src/lib/store/runtime.ts`
   - `hydrateLatestOrchestrationSnapshot`
   - 调用 session live-monitor
   - 将 `liveMonitor.monitor` 写入 `taskGraphLiveMonitor`

2. `frontend/src/components/chat/ChatPanel.tsx`
   - 将 `taskGraphLiveMonitor` 传给 `TaskGraphRunPanel`

3. `frontend/src/components/chat/TaskGraphRunPanel.tsx`
   - `buildModelFromLiveMonitor`
   - 当前负责把后端摘要再次拼成 UI 模型
   - 曾错误地优先使用 `coordination_flow.stages` 作为拓扑节点集合，导致 `coordination_graph_spec.edges` 的端点找不到，边被布局函数过滤。

4. `frontend/src/components/coordination/CoordinationTopologyGraph.tsx`
   - 只负责渲染 nodes/edges
   - 布局层要求边的 `from` 和 `to` 必须存在于 nodes 中，否则边会被过滤。

### 2.3 已确认的故障链

一次真实运行中，后端返回：

1. `coordination_graph_spec.graph_id = graph.writing.simple_novel`
2. `coordination_graph_spec.nodes = 28`
3. `coordination_graph_spec.edges = 61`

但前端拓扑可能看不到边，原因是：

1. 前端选用了 `flowStages` 作为 nodes。
2. `flowStages` 是运行阶段投影，不是完整任务图。
3. `graphEdges` 来自完整任务图。
4. 边端点不在当前 nodes 集合时，`CoordinationTopologyGraph.buildCoordinationTopologyLayout` 返回 `null` 并过滤边。

根因：前端承担了监控模型归一化职责，且没有严格区分“静态图结构”和“运行状态覆盖”。

## 3. 目标架构

### 3.1 监控必须成为专用模型

新增后端专用模型：`TaskGraphRunMonitorView`。

前端不得再自行从多种运行摘要中猜图结构、猜节点状态、猜边状态。

后端必须一次性返回监控 UI 所需的规范结构：

```json
{
  "authority": "task_graph.run_monitor",
  "task_run_id": "...",
  "coordination_run_id": "...",
  "session_id": "...",
  "graph": {
    "graph_id": "graph.writing.simple_novel",
    "title": "写作组简易长篇小说任务图",
    "node_count": 28,
    "edge_count": 61
  },
  "runtime": {
    "status": "running",
    "active_node_id": "world_review",
    "active_task_ref": "task.writing.simple_novel.world_review",
    "last_event_offset": 65,
    "checkpoint_ref": "...",
    "checkpoint_updated_at": 1778870000.0,
    "updated_at": 1778870000.0
  },
  "topology": {
    "nodes": [],
    "edges": []
  },
  "state": {
    "node_statuses": {},
    "edge_statuses": {},
    "completed_node_ids": [],
    "failed_node_ids": [],
    "blocked_node_ids": [],
    "waiting_node_ids": []
  },
  "artifacts": [],
  "memory_operations": [],
  "stage_results": [],
  "current_stage_execution_request": {},
  "health": {
    "valid": true,
    "issues": []
  }
}
```

### 3.2 单一事实源规则

1. 拓扑结构唯一来源：
   - `coordination_graph_spec.nodes`
   - `coordination_graph_spec.edges`

2. 运行状态唯一来源：
   - LangGraph checkpoint state
   - 尤其是 `active_stage_id`、`completed_nodes`、`failed_nodes`、`stage_results`、`stage_execution_request`
   - `task_graph_scheduler_state` 可作为调度状态补充，但不得覆盖 checkpoint 的当前事实。

3. 事件进度来源：
   - TaskRun event log 的 latest offset
   - checkpoint ref / checkpoint updated_at

4. 产物来源：
   - `stage_results[*].artifact_refs`
   - materializer 返回的 `created_files`
   - 不允许前端从文件目录猜产物。

5. 记忆操作来源：
   - checkpoint state 的 `working_memory_operations`
   - 不允许前端把普通 handoff 当成记忆写入。

6. `coordination_flow` 的定位：
   - 只作为兼容摘要或时间线辅助。
   - 不得作为拓扑结构源。
   - 不得作为 active node 的最高优先级来源。

### 3.3 前端职责收缩

前端只负责渲染：

1. `monitor.topology.nodes`
2. `monitor.topology.edges`
3. `monitor.state.node_statuses`
4. `monitor.state.edge_statuses`
5. `monitor.runtime.active_node_id`
6. `monitor.artifacts`
7. `monitor.memory_operations`
8. `monitor.health.issues`

前端不得再做：

1. 从 `coordination_flow.stages` 和 `coordination_graph_spec` 合成拓扑。
2. 从 `node_runs` 猜 graph node。
3. 从 handoff envelopes 猜完整边。
4. 按 session latest 静默切换到其他 run 后不提示用户。
5. 在没有 graph_id/node_count/edge_count 的情况下显示“正常监控”。

## 4. 目标页面结构

实时任务监控页面必须分层，不再把所有信息混在一个面板里。

### 4.1 顶部运行身份条

必须显示：

1. `graph_id`
2. `task_run_id`
3. `coordination_run_id`
4. 当前状态：running / stopped / failed / completed
5. 当前节点：`active_node_id`
6. checkpoint 时间
7. latest event offset
8. 数据源：task-run pinned / session latest

如果当前视图来自 session latest，必须显示“可能随 latest run 切换”提示。

### 4.2 拓扑图层

只展示完整任务图拓扑。

要求：

1. 节点来自 `monitor.topology.nodes`。
2. 边来自 `monitor.topology.edges`。
3. 边必须默认可见，不能只在选中时出现。
4. 当前节点高亮。
5. 已完成路径高亮。
6. 阻塞/失败节点明显标识。
7. 如果 `edge_count > 0` 但画布边数量为 0，前端必须显示错误，而不是静默空图。

### 4.3 时间线层

展示运行推进：

1. 当前 stage_execution_request。
2. 已完成 stage results。
3. 最近事件。
4. 下一步候选。

时间线不能替代拓扑。

### 4.4 产物与记忆层

展示：

1. 本 run 的 artifact refs。
2. 每个 artifact 的 producer node。
3. working memory read/write/lock/release 操作。
4. memory pack 是否 blocked。
5. 写入 receipt。

### 4.5 健康检查层

展示：

1. graph spec 是否存在。
2. node_count / edge_count 是否大于 0。
3. active node 是否存在于 topology.nodes。
4. 每条 edge 的端点是否存在。
5. stage_execution_request 是否存在。
6. checkpoint 是否新鲜。
7. session latest 是否与用户选中 run 一致。

## 5. 后端重构计划

### 阶段 1：新增监控模型，不改旧 API 行为

新增文件：

1. `backend/orchestration/runtime_loop/task_graph_run_monitor.py`

职责：

1. 定义 `build_task_graph_run_monitor_view(...)`。
2. 输入：
   - `TaskRun`
   - `CoordinationRun`
   - coordination checkpoint state
   - state index snapshot
   - event summary
3. 输出：
   - `TaskGraphRunMonitorView` dict。

必须实现的归一化：

1. `topology.nodes` 从 `coordination_graph_spec.nodes` 来。
2. `topology.edges` 从 `coordination_graph_spec.edges` 来。
3. `state.node_statuses` 从 checkpoint 当前状态计算。
4. `state.edge_statuses` 从 source/target node 状态计算，不允许边状态缺失导致边消失。
5. `health.issues` 对所有结构不一致给出明确错误。

完成标准：

1. 对当前 `graph.writing.simple_novel` 返回 28 节点、61 边。
2. 即使 `coordination_flow.stages` 为空，topology 仍完整。
3. 即使 checkpoint 暂时缺 stage_results，topology 仍完整。

### 阶段 2：新增专用 API

新增 API：

1. `GET /api/orchestration/runtime-loop/task-runs/{task_run_id}/task-graph-monitor`
2. `GET /api/orchestration/coordination-runs/{coordination_run_id}/task-graph-monitor`

保留旧 API：

1. `/live-monitor` 暂时不删除。
2. 旧 API 标记为 compatibility snapshot，不再作为新前端主入口。

完成标准：

1. 通过 task_run_id 可稳定定位同一个 run。
2. 通过 coordination_run_id 可稳定定位同一个 coordination run。
3. session latest 不参与专用 task-run monitor，避免静默切 run。

### 阶段 3：前端切换到专用模型

新增或重构：

1. `frontend/src/lib/api.ts`
   - 新增 `TaskGraphRunMonitorView` 类型。
   - 新增 `getTaskGraphRunMonitor(taskRunId)`。
   - 新增 `getCoordinationRunMonitor(coordinationRunId)`。

2. `frontend/src/components/chat/TaskGraphRunPanel.tsx`
   - 不再从旧 `RuntimeLoopTaskRunLiveMonitor` 自行拼图。
   - 接收 `TaskGraphRunMonitorView`。
   - 只渲染后端给定的 topology/state。

3. `frontend/src/components/coordination/CoordinationTopologyGraph.tsx`
   - 保留纯渲染职责。
   - 增加 dev/health 警告：传入 edges 非空但布局后 edges 为 0 时抛出可见错误。

4. `frontend/src/lib/store/runtime.ts`
   - 增加 pinned task run 监控路径。
   - session latest 只用于发现最新 run，不作为稳定监控源。

完成标准：

1. 前端不再读取 `coordination_graph_spec`、`coordination_flow`、`langgraph_runtime_state` 来拼图。
2. 前端显示 `graph_id / node_count / edge_count / active_node_id`。
3. 边可见且数量与后端 monitor 一致。

### 阶段 4：实时更新机制

短期：

1. 保留轮询，但轮询专用 monitor API。
2. 轮询必须带 `last_event_offset`。
3. 如果 offset 未变化，可轻量返回 `not_modified`。

中期：

1. 新增 SSE：
   - `GET /api/orchestration/runtime-loop/task-runs/{task_run_id}/task-graph-monitor/events`
2. 推送事件：
   - node_started
   - node_completed
   - node_failed
   - edge_activated
   - artifact_created
   - memory_read
   - memory_write
   - checkpoint_updated

完成标准：

1. 节点完成后无需刷新页面即可更新状态。
2. artifact refs 出现后进入产物层。
3. 断线后可用 `last_event_offset` 恢复。

## 9. 迁移与切换规则

### 9.1 兼容窗口

旧 API 保留一个版本周期：

1. `/live-monitor` 继续返回旧结构。
2. 新 UI 不再依赖旧结构。
3. 旧 UI 若仍存在，必须标记为“旧监控摘要”。

### 9.2 切换条件

只有满足以下条件才能切换：

1. 专用 monitor API 测试通过。
2. 前端 monitor panel 测试通过。
3. 对真实 `graph.writing.simple_novel` run 验证通过。
4. 对一个废弃 stopped run 验证通过。
5. 对一个没有 coordination 的普通 task run 验证空态通过。

### 9.3 回滚规则

如果新 monitor API 出现错误：

1. 前端显示“专用监控不可用”。
2. 允许临时切回旧 live-monitor 摘要。
3. 不允许旧摘要伪装成完整实时拓扑。

## 10. 禁止的次品方案

以下方案禁止采用：

1. 只在前端改一行 nodes 选择顺序就宣布完成。
2. 前端继续从多个旧字段自行拼拓扑。
3. 后端只返回 summary，不返回完整 nodes/edges。
4. 用 flow stages 代替 graph spec。
5. 用 handoff packets 代替 graph edges。
6. session latest 静默切换 run。
7. 边不显示但不报错。
8. 生成假边、假节点或假状态来让 UI 好看。
9. 把记忆读写、artifact refs、stage results 混在普通日志里展示。
10. 没有测试就上线监控重构。

## 11. 实施顺序

1. 新增后端 monitor builder。
2. 新增 task-run 专用 monitor API。
3. 给后端 monitor builder 补测试。
4. 新增前端 monitor types 和 view model。
5. 拆分前端 TaskGraph monitor 面板。
6. 前端切到专用 API。
7. 保留旧 live-monitor 兼容入口但降级为摘要。
8. 加 SSE 或 offset-based polling。
9. 用真实写作组 run 做浏览器验证。
10. 删除前端旧的多源拼装逻辑。

## 12. 完成定义

本重构完成时必须满足：

1. 用户看到的监控明确绑定某个 `task_run_id` / `coordination_run_id`。
2. 拓扑图来自真实 TaskGraph spec。
3. 运行状态来自 checkpoint。
4. 节点、边、产物、记忆、事件偏移都可追踪。
5. 边缺失、端点缺失、graph spec 缺失都会显示为错误。
6. 前端不再承担运行监控事实归一化职责。
7. 真实运行中不会再出现“后端有边但前端看不到边”这种情况。

# 78-TaskGraph接入投影系统与编排系统主数据修复计划

## 0. 文档定位

本文承接 76、77 号计划，专门处理 TaskGraph Studio 在 Agent 与 Prompt/Projection 配置上的主数据归属问题。

当前判断：

```text
TaskGraph Studio 的产品框架方向是对的，
但 Agent 与 Prompt/Projection 配置不应该在 TaskGraph 内部另造一套替代品。
```

正确方向是：

```text
编排系统负责 Agent / Agent Group / Runtime Profile 的静态主数据。
投影系统负责 Projection / Prompt Manifest / Prompt Profile 的静态主数据。
任务系统负责 TaskGraph / 节点 / 边 / 阶段 / 契约 / 运行绑定。
运行装配阶段把三者按引用合成为真正给 Agent 使用的 Runtime Assembly。
```

### 1.2 当前 TaskGraph Studio 的偏差

当前前端 TaskGraph Studio 存在半接入问题。

Agent 侧：

- `TaskGraphAgentRosterPage.tsx` 主要从 `a2aCatalog.agent_cards` 推导可选 Agent。
- `agent_group_id` 目前是自由输入框。
- 编辑器没有直接消费 `/orchestration/agents` 返回的 `agents`、`agent_groups`、`profiles`、`options`。
- 节点只保存 `agent_id` 字符串，界面无法可靠展示 Agent Runtime Profile、权限、可委派能力、默认 Projection。

Prompt / Projection 侧：

- `TaskSystemView.tsx` 只拉取 `getSoulProjectionCards()`，把它作为卡片选择资源。
- `NodeResponsibilityCard.tsx` 与 `TaskGraphAgentRosterPage.tsx` 在节点 metadata 中维护 `role_prompt`、`role_identity`、`responsibility_scope` 等内部 prompt 字段。
- `taskGraphTemplates.ts` 会直接生成节点 `metadata.role_prompt`。
- 这套 `role_prompt` 实际上成了投影系统之外的第二套 prompt 存储。

保存侧：

- `taskGraphSaveMapper.ts` 会把节点 metadata 原样写入 TaskGraph。
- 因此 `role_prompt` 会被持久化为 TaskGraph 的内部 prompt，而不是 Projection / Prompt Manifest 的引用。

运行侧：

- `runtime_assembly_builder.py` 已经按 `node.projection_id -> agent.default_projection_id` 解析投影。
- 这说明运行装配更需要稳定 `projection_id` 与 `agent_profile_id` 引用，而不是节点内部 prompt 文本。

### 1.3 风险结论

当前结构如果继续发展，会形成三套并行事实源：

```text
编排系统 Agent Registry
TaskGraph 节点 agent/prompt metadata
投影系统 Projection Store
```

这会导致：

- UI 看起来配置了 Agent，但运行时未必使用正确 Runtime Profile。
- UI 看起来配置了 Prompt，但 Prompt Manifest 不知道这段文本。
- 投影系统无法成为静态 prompt 仓库。
- TaskGraph 保存中混入大量 prompt 正文，后续复用、审计、版本化都会变难。
- 运行装配无法稳定解释“这个节点为什么得到这个 Agent 和这个 Prompt”。

## 2. 目标架构

### 2.1 三个系统的正式归属

编排系统 owning：

- AgentDescriptor
- AgentRuntimeProfile
- AgentGroup
- Agent 默认 soul/projection
- Agent 权限、可委派能力、上下文可见性、操作范围

投影系统 owning：

- SoulProfile
- ProjectionCard
- ProjectionTemplate
- PromptSection
- PromptManifest
- AgentPromptBundle
- 任务系统沉淀出的职责 prompt profile / stage prompt profile

任务系统 owning：

- TaskDomain
- SpecificTask
- TaskWorkflow
- TaskGraphDefinition
- TaskGraph Node / Edge / Phase
- Contract / Preflight / Publish / Run binding
- 对 Agent 与 Projection 的引用和覆盖策略

### 2.2 TaskGraph 可存字段

TaskGraph 图级允许存：

- `runtime_policy.coordinator_agent_id`
- `runtime_policy.agent_group_id`
- `runtime_policy.participant_agent_ids`
- `runtime_policy.agent_selection_policy`
- `context_policy`
- `working_memory_policy`
- `default_protocol_id`
- `graph_contract_id`
- `default_projection_policy`

TaskGraph 节点允许存：

- `agent_id`
- `agent_selection_policy`
- `agent_group_id`
- `agent_profile_ref`
- `projection_id`
- `projection_overlay_id`
- `prompt_profile_ref`
- `task_id`
- `node_contract_id`
- `runtime_lane`
- `context_visibility_policy`
- `memory_*_policy`
- `artifact_policy`

TaskGraph 节点不应长期存：

- `metadata.role_prompt`
- `metadata.role_identity`
- `metadata.responsibility_scope`
- `metadata.responsibility_exclusions`
- `metadata.definition_of_done`

迁移期可以只读兼容这些字段，但新写入必须写到投影系统。

### 2.3 投影系统的新定位

投影系统改成静态 Prompt / Projection 仓库。

它不只服务“灵魂人格”，还要服务 TaskGraph 节点职责。

推荐新增静态资源概念：

```text
TaskProjectionProfile
```

或者沿用现有 `ProjectionCard` 扩展字段：

```json
{
  "projection_id": "projection.taskgraph.world_review",
  "projection_kind": "task_graph_node",
  "owner_system": "task_system",
  "task_family": "longform_novel_writing",
  "node_role": "world_reviewer",
  "agent_profile_id": "review_agent",
  "projection_prompt": "你是一名世界观审核员...",
  "prompt_manifest_refs": [],
  "source_task_graph_refs": []
}
```

这样任务系统的好框架可以沉到投影里，成为可复用的静态 prompt 资产。

### 2.4 编排系统的新定位

编排系统改成静态 Agent / Runtime Profile 仓库。

TaskGraph 不编辑 Agent 的本体配置，只引用它。

编辑器中所有 Agent 选择应来自：

```text
GET /orchestration/agents
  -> agents
  -> agent_groups
  -> profiles
  -> options
```

TaskGraph 只保存：

```text
这个节点使用哪个 Agent
这个节点是否覆盖默认 projection
这个节点进入哪个 runtime lane
这个节点需要哪些上下文、记忆、能力边界
```

## 3. 固定装配流程

### 3.1 编辑期

1. TaskGraph Studio 拉取：
   - `GET /tasks/overview`
   - `GET /orchestration/agents`
   - `GET /soul/projections`

2. Agent 编组页：
   - Agent 下拉来自 `orchestration.agents`。
   - Agent Group 下拉来自 `orchestration.agent_groups`。
   - 节点显示对应 Runtime Profile 摘要。

3. Prompt / Projection 页：
   - 节点只选择 Projection 或 Prompt Profile。
   - “生成职责 Prompt”不再写入节点 metadata，而是创建或更新 ProjectionCard。
   - 节点保存 `projection_id` / `prompt_profile_ref`。

### 3.2 保存期

1. 保存 TaskGraph 时：
   - 只保存 Agent / Projection 引用。
   - 不再保存新生成的 prompt 正文。

2. 如果存在 legacy `metadata.role_prompt`：
   - 迁移为 ProjectionCard。
   - 写回节点 `projection_id`。
   - 保留 legacy 字段只作为只读迁移来源。

### 3.3 发布预检期

新增预检规则：

- `agent_id` 必须存在于编排系统 Agent Registry。
- `agent_group_id` 必须存在于编排系统 AgentGroup Registry。
- 节点 `agent_id` 的 Runtime Profile 必须允许当前 task mode / runtime lane。
- 节点 `projection_id` 必须存在于投影系统 Projection Store。
- 节点不能只配置 `metadata.role_prompt` 而没有 `projection_id`。

### 3.4 运行装配期

运行装配顺序固定为：

```text
TaskGraph node
  -> resolve AgentDescriptor
  -> resolve AgentRuntimeProfile
  -> resolve ProjectionCard
  -> build SoulRuntimeView
  -> build PromptManifest
  -> build NodeRuntimeAssembly
```

装配结果必须带可解释 trace：

- `agent_resolution_source`
- `agent_profile_ref`
- `projection_resolution_source`
- `prompt_manifest_ref`
- `task_graph_node_ref`

## 6. 迁移规则

迁移期允许读取：

- `metadata.role_prompt`
- `metadata.role_identity`
- `metadata.responsibility_scope`
- `metadata.responsibility_exclusions`
- `metadata.definition_of_done`

迁移期不允许新写入这些字段作为主数据。

迁移算法：

```text
if node.projection_id exists:
    use projection_id
elif node.metadata.role_prompt exists:
    create projection card from legacy prompt
    set node.projection_id
    keep legacy fields only under metadata.legacy_prompt_migration
else:
    preflight blocking issue
```

## 9. 实施顺序锁定

后续实施必须按以下顺序推进：

1. 前端先接正式 Agent Registry，替换 Agent 选择来源。
2. 前端再接正式 Projection Store，替换 prompt 主写入。
3. 后端补迁移与保存收口。

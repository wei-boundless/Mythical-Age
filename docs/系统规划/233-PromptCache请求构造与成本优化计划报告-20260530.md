# Prompt Cache 请求构造与成本优化计划报告

日期：2026-05-30

## 1. 问题定义

当前系统已经具备 `PromptAccountingLedger`、`PromptSegmentMap`、`PromptCacheRecord`、provider usage 抽取和健康系统读取能力。真实运行账本也能看到 DeepSeek provider 返回的 cached tokens。

但这套机制仍主要停留在事后记录和预测层，尚未成为模型请求构造层的权威。成熟 agent 的 prompt cache 必须保护真实请求字节的稳定前缀，而不是在请求已经构造完成后再通过 serializer 猜测哪些内容可缓存。

当前缺口：

- cache boundary 没有由 RuntimeCompiler 显式下发到 ModelRuntime。
- `CanonicalPromptSerializer` 通过文本关键词推断 segment 类型。
- stable system message 中混入 dynamic projection。
- stable payload 中存在完整 `task_run` 等 mutable state 风险。
- provider cache policy 没有进入真实 request builder。
- cache hit/miss 只能统计，不能解释失效原因。
- retry、delegate、subagent 没有 byte-exact stable prompt 复用协议。

正确终态：

```text
RuntimeCompiler
-> RuntimeInvocationPacket + PromptSegmentPlan
-> ModelRequestBuilder
-> ProviderCachePolicyAdapter
-> CanonicalPromptSerializer
-> PromptSegmentMap / PromptCacheRecord / TokenLedger
-> ProviderUsageExtractor
-> PromptCacheBreakDetector
```

## 2. 源码审查事实

### 2.1 RuntimeCompiler 当前问题

位置：

```text
backend/harness/runtime/compiler.py
```

事实：

- `compile_turn_action_packet` 把 runtime prompt pack、runtime projection、environment prompt、soul prompt、agent prompt 合成同一个 system message。
- `compile_task_execution_packet` 把 artifact note、runtime projection、environment prompt、agent prompt 合成同一个 system message。
- stable payload 包含 `schema`、`task_run`、`task_contract`、`task_environment`、`available_tools`、`operation_authorization`、`runtime_context`。
- volatile payload 才包含 `execution_state`、`observations` 或 user/history。

风险：

- dynamic projection 位于 cacheable prefix 内。
- `artifact_root`、tool count、authorization summary 会污染 stable hash。
- 完整 `task_run` 可能随步骤变化，不能稳定缓存。
- RuntimePromptManifest 只记录 prompt assembly 层信息，不等于真实模型请求分段。

### 2.2 PromptAccounting 当前问题

位置：

```text
backend/runtime/prompt_accounting/serializer.py
backend/runtime/prompt_accounting/cache_planner.py
backend/runtime/model_gateway/model_runtime.py
```

事实：

- ModelRuntime 在 `_begin_prompt_accounting()` 中对 `messages` 做 serializer。
- serializer 根据 message role 和内容关键词判断 `system_static`、`system_session`、`task_contract`、`memory_context`。
- cache planner 取连续的 `cacheable_prefix/session_stable` 作为 stable prefix。
- cache record 默认 TTL 是 300 秒。
- provider usage 抽取后会把 cache record 标记为 hit/miss。

风险：

- segment authority 在 serializer，不在 RuntimeCompiler。
- 文案变化可能改变 segment 分类。
- cache key 与真实 provider cache control 没有绑定。
- 没有 session latch、防止中途策略翻转的机制。
- 没有 cache break 维度诊断。

### 2.3 PromptLibrary 当前问题

位置：

```text
backend/prompt_library/assembly.py
backend/prompt_library/manifest.py
backend/prompt_library/registry.py
```

事实：

- Prompt resources 已经有 `cache_scope`。
- RuntimePromptManifest 已经统计 `static_environment`。
- manifest 的 `token_estimate.prompt_chars` 只统计 prompt sections。
- registry 每次 list 都会重新组合 builtin/stored/environment resources，且 normalize read 可能写回 JSON。

风险：

- manifest token estimate 不能代表真实模型请求。
- prompt resource 级 cache scope 没有传递到真实 request segment。
- stable section 资源没有 session memoization。

### 2.4 实际账本观察

位置：

```text
storage/runtime_state/prompt_accounting/
```

抽样事实：

- `prompt_cache.jsonl` 中已有 hit/miss/eligible。
- 225 条 cache record 中约有 72 hit、29 miss、124 eligible。
- 最近 20 条中约有 8 hit。
- 有图任务路径出现百万级 predicted prompt tokens，说明 prompt budget gate 和 graph/task payload 裁剪仍需后续治理。

结论：

缓存不是完全无效；provider 自动缓存在发挥作用。但系统没有完全控制稳定前缀和失效诊断，因此成本优化不可控。

## 3. 设计原则

### 3.1 请求构造层是 cache 权威

cache 的权威必须在真实 request builder，而不是健康系统、UI、日志或事后 serializer。

### 3.2 RuntimeCompiler 负责声明，不负责 provider 细节

RuntimeCompiler 输出 `PromptSegmentPlan`，声明每段的稳定性、来源、顺序和 cache role。

它不直接写 provider 特有字段。

### 3.3 ModelRequestBuilder 负责真实请求字节

ModelRequestBuilder 消费 packet 和 segment plan，生成真实 messages/tools/body。

它必须保证：

- 稳定段顺序固定。
- 序列化字节稳定。
- retry 不改变 stable bytes。
- fork/delegate 可复用父级 stable bytes。

### 3.4 ProviderCachePolicyAdapter 负责 provider 差异

不同 provider 的 cache 能力不同：

- DeepSeek / OpenAI-compatible：主要依赖自动前缀缓存，系统保证 byte-stable prefix 并读取 usage 校验。
- Anthropic native：可把 boundary 转成显式 `cache_control`。
- 不支持 cache accounting 的 provider：记录 `disabled` 或 `unsupported_usage`，不能伪造 hit/miss。

### 3.5 禁止关键词启发式

禁止通过 message content 判断 segment 类型。所有 segment 必须来自显式结构。

缺少 plan 时应 fail closed：

```text
cache plan bypass
segment kind = unknown_unplanned
cache_safety_reasons += missing_prompt_segment_plan
```

### 3.6 动态状态不能冒充稳定前缀

以下内容必须位于 volatile 或 dynamic projection：

- 当前用户消息
- history
- observations
- tool results
- active failures
- current task_run state
- retry attempt state
- 当前时间
- request id
- authorization result 的临时诊断

## 4. 目标数据结构

### 4.1 PromptSegmentPlan

新增位置：

```text
backend/harness/runtime/prompt_segment_plan.py
```

职责：

- 描述 RuntimeInvocationPacket 中每段模型可见内容的来源和稳定性。
- 提供 ModelRequestBuilder 和 PromptAccounting 的共同输入。

核心字段：

```json
{
  "segment_plan_id": "segplan:...",
  "packet_id": "rtpacket:...",
  "segments": [
    {
      "segment_id": "segplan:1",
      "kind": "global_static",
      "source_ref": "runtime.task_execution.v1",
      "cache_scope": "global",
      "cache_role": "cacheable_prefix",
      "compression_role": "preserve",
      "content_hash": "sha256:...",
      "ordinal": 1,
      "model_message_role": "system"
    }
  ]
}
```

### 4.2 ModelRequestPacket

新增位置：

```text
backend/runtime/model_gateway/model_request.py
```

职责：

- 保存真实 provider 请求结构。
- 保留 segment plan 到 provider request block 的映射。
- 提供 canonical bytes 给 PromptAccounting。

核心字段：

```json
{
  "request_id": "modelreq:...",
  "provider": "deepseek",
  "model": "deepseek-v4-pro",
  "messages": [],
  "tools": [],
  "extra_body": {},
  "segment_bindings": [],
  "stable_prefix_hash": "sha256:...",
  "cache_policy": {}
}
```

### 4.3 ProviderCachePolicy

新增位置：

```text
backend/runtime/model_gateway/provider_cache_policy.py
```

策略：

- `automatic_prefix`
- `native_cache_control`
- `disabled`

### 4.4 PromptCacheBreakRecord

新增位置：


记录字段：

- previous_request_id

## 5. 固定执行流

### 5.1 Turn / TaskRun 统一流程

```text
RuntimeAssembly
-> PromptAssemblyService
-> RuntimeProjectionBuilder
-> RuntimeCompiler
   -> RuntimeInvocationPacket
   -> PromptSegmentPlan
-> ModelRequestBuilder
   -> ModelRequestPacket
   -> ProviderCachePolicy
-> ModelRuntime.invoke
-> PromptAccountingLedger
-> ProviderUsageExtractor
-> PromptCacheBreakDetector
```

### 5.2 Retry 流程

retry 必须复用第一次 attempt 的 stable rendered bytes。

允许变化：

- request id
- attempt index
- volatile retry diagnostics

禁止变化：

- global static bytes
- environment stable bytes
- task stable bytes
- tool schema stable bytes
- model/provider/thinking config，除非明确进入 fallback candidate；fallback 必须记录 `model_changed` cache break。

### 5.3 Delegate / Subagent 流程

delegate 默认不重新编译父级 stable prompt。

父级传递：

- rendered stable prefix bytes 或 stable segment refs
- tool schema hash
- model/provider/thinking config
- context replacement state / compaction decision hash

子级只追加自己的 task directive 和 volatile request。

## 6. 实施阶段

### 阶段一：显式 SegmentPlan

目标：

- RuntimeCompiler 生成 PromptSegmentPlan。
- RuntimeInvocationPacket 持有 `segment_plan` 或 `segment_plan_ref`。
- RuntimePromptManifest 引用 segment plan。

修改文件：

```text
backend/harness/runtime/invocation_packet.py
backend/harness/runtime/compiler.py
backend/harness/runtime/prompt_segment_plan.py
backend/prompt_library/manifest.py
backend/tests/prompt_library_runtime_pack_test.py
```

完成标准：

- packet diagnostics 中不只记录 prompt_manifest，还能看到 segment_plan。
- segment plan 中 dynamic projection 和 volatile state 边界明确。
- 完整 `task_run` 不再进入 stable segment。

### 阶段二：请求构造层接管 cache boundary

目标：

- 新增 ModelRequestBuilder。
- ModelRuntime 不再直接把 raw messages 交给 serializer。
- serializer 消费 ModelRequestPacket 和 segment bindings。

修改文件：

```text
backend/runtime/model_gateway/model_request.py
backend/runtime/model_gateway/model_runtime.py
backend/runtime/prompt_accounting/serializer.py
backend/runtime/prompt_accounting/models.py
backend/harness/loop/model_action_runtime.py
backend/harness/loop/agent_loop.py
backend/harness/loop/task_executor.py
```

完成标准：

- `CanonicalPromptSerializer` 删除内容关键词 segment 推断。
- 缺少 segment plan 时 cache bypass。
- cache key 不包含 request id。
- stable prefix hash 来自 segment plan 和真实 canonical bytes。

### 阶段三：Provider cache policy adapter

目标：

- 建立 provider cache 能力表。
- 针对 provider 选择 automatic/native/disabled。
- 对 native provider 输出显式 cache control。

修改文件：

```text
backend/runtime/model_gateway/provider_cache_policy.py
backend/runtime/model_gateway/model_runtime.py
backend/config.py
backend/tests/prompt_accounting_ledger_test.py
```

完成标准：

- DeepSeek/OpenAI-compatible 记录 `automatic_prefix`。
- Anthropic native adapter 未实现前不得伪造 `native_cache_control`。
- unsupported provider 明确记录 disabled 原因。

### 阶段四：cache break detection

目标：

- 建立请求前 snapshot。
- provider usage 回来后判断 hit/miss/cache break。
- 健康系统展示失效维度。

修改文件：

```text
backend/runtime/prompt_accounting/cache_break_detector.py
backend/runtime/prompt_accounting/ledger.py
backend/runtime/model_gateway/model_runtime.py
backend/health_system/governance.py
backend/api/sessions.py
```

完成标准：

- miss 时能区分 model changed、tool schema changed、stable prompt changed、dynamic entered stable prefix。
- provider 没有 usage 时显示 `provider_usage_unavailable`，不误判为 miss。

### 阶段五：prompt resource 和 tool schema memoization

目标：

- prompt resource 装配支持 session/task 级 memoization。
- tool schema 序列化支持 session latch。
- registry 读操作不做无必要写回。

修改文件：

```text
backend/prompt_library/registry.py
backend/prompt_library/assembly.py
backend/harness/runtime/assembly.py
backend/runtime/model_gateway/model_request.py
backend/runtime/tool_runtime/*
```

完成标准：

- 同一 session 内 agent/environment/runtime prompt bytes 稳定。
- tool schema hash 在同一 runtime profile 下稳定。
- 不因 registry normalize read 造成无意义文件写入。

### 阶段六：压缩与长任务成本治理

目标：

- compaction 基于真实 segment role，保护 cacheable prefix。
- 对超大 stable payload 增加 preflight budget gate。
- graph/task 大 payload 使用 refs 和摘要，禁止百万级 prompt 在无诊断下进入请求。

修改文件：

```text
backend/context_system/compaction/compactor.py
backend/runtime/prompt_accounting/compression_budget.py
backend/harness/loop/task_executor.py
backend/task_system/compiler/*
```

完成标准：

- cacheable prefix 不被 microcompact 改写。
- 超预算请求在模型调用前生成明确诊断。
- 大型 graph/task payload 默认 refs 化。

## 7. 验证策略

### 静态结构验证

- 检查 serializer 不再含 prompt 文案关键词分类。
- 检查 RuntimeCompiler 输出 segment plan。
- 检查 task mutable state 不在 stable segment。
- 检查 provider cache policy 被 ModelRuntime 消费。

### 单元测试

建议新增：

```text
backend/tests/prompt_segment_plan_regression.py
backend/tests/model_request_builder_cache_boundary_regression.py
backend/tests/prompt_cache_break_detector_regression.py
backend/tests/provider_cache_policy_regression.py
```

### 实测

使用后端 CLI 跑三类任务：

1. 普通问答：不应启动 TaskRun，cache plan 应有 global static。
2. 单 agent 基本任务：TaskRun 多步执行，第二步以后 stable prefix hash 应稳定。
3. 长任务：五层地下塔游戏，运行中应持续记录 prompt cache、token usage、cache break 或 hit。

验收：

- CLI 能看到任务状态连续更新。
- ledger 中 provider usage 与 local prediction 同时存在。
- hit/miss 有解释。
- prompt cache 不依赖旧 `backend/prompting/*`。

## 8. 旧结构清理规则

必须删除或降级：

- serializer 中的文本关键词 segment 推断。
- 旧 prompt cache 作为新 harness 主链依据的任何入口。
- 以 compatibility/fallback 名义参与新 runtime prompt 装配的旧 selector。
- 把完整 runtime mutable state 标记为 stable 的结构。

允许保留：

- 旧 `backend/prompting/*` 只读审计和旧 API 查询。
- 历史 ledger 数据。
- 已迁移为 PromptResource 的 graph/task prompt 资产。

## 9. 风险控制

风险一：provider 差异导致 cache control 不兼容。

控制：

- provider adapter 只对确认支持的 provider 开启 native strategy。
- 默认使用 automatic_prefix 或 disabled。

风险二：过度追求 cache 命中导致动态事实被错误缓存。

控制：

- dynamic projection 和 volatile state 默认 never global cache。
- task mutable state 进入 stable segment 时测试失败。

风险三：改动模型请求结构导致 agent 行为退化。

控制：

- 保持 agent 可见语义不变，只改变分段和顺序。
- 每阶段做 CLI 实测。

风险四：旧链路继续污染新主链。

控制：

- 新 harness 只从 PromptLibrary + RuntimeCompiler + ModelRequestBuilder 取 prompt。
- 旧 prompting 模块不得作为 fallback。

## 10. 完成标准

本轮优化完成后必须满足：

- 每次 RuntimeInvocationPacket 都有 PromptSegmentPlan。
- PromptSegmentMap 不再由关键词启发式产生。
- PromptCachePlanner 基于真实 request segment，而不是 raw message 猜测。
- provider cache policy 已进入 ModelRuntime 请求路径。
- provider usage 能回填 hit/miss/cache savings。
- cache miss 能输出结构化原因。
- retry 不破坏 stable prefix。
- TaskRun 多步执行时 stable prefix hash 在合同不变时保持不变。
- 长任务成本视图能显示 predicted、provider usage、cached tokens、cache break reason。

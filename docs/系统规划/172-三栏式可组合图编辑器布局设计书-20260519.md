# 三栏式可组合图编辑器布局设计书

日期：2026-05-19
状态：布局设计完成，待实施
适用范围：TaskGraph Studio、可组合图编辑器、父子图导航、节点/边/端口对象编辑、发布预检定位

## 0. 结论

建议把 TaskGraph Studio 的核心编辑体验升级为：

```text
左侧：图层关系与对象树
中间：当前层级图画布
右侧：选中对象编辑台
底部：诊断、预检、运行状态
```

这个布局比“左侧编辑台”更适合当前系统，因为任务图编辑的主要心智是：

```text
我在哪个图层？
我在画布上选择了哪个对象？
我要编辑这个对象的哪些协议、契约和时序？
这些修改是否能通过标准视图和发布预检？
```

因此左侧应该承载导航和关系，右侧应该承载精确编辑。中间画布只负责结构和选中，不承担所有配置输入。

### 1.1 已可复用的外壳

当前已经具备：

- `TaskGraphStudioShell`
  - 顶部状态栏
  - `TaskGraphLayerNav`
  - 当前层级说明条
  - 底部 `TaskGraphIssueBar`
- `TaskGraphLayerNav`
  - 已按 foundation / structure / runtime / quality 分组
  - 适合改造成“图层关系与工作区导航”的上半部分
- `TaskGraphTopologyPage`
  - 已有左 rail / 中 canvas / 右 inspector 的雏形
  - 当前左 rail 是工具添加区，右 inspector 是选择摘要
- `TaskGraphModuleCompositionPage`
  - 已支持 Unit / Interface / PortEdge / NestedRuntime 的标准视图展示
  - 已支持 `metadata.composable_graph.port_edges` 覆盖层编辑
- `CoordinationTopologyGraph`
  - 已支持选中节点、选中边、selected class、节点工具、边工具
  - 可作为中间画布的第一版基础
- `taskGraphEditorFocus`
  - 已能把发布预检问题映射回 layer / facet / node_id / edge_id
  - 可升级为画布选择状态与右侧编辑台的统一协议

### 1.3 资产统筹布局

本轮不新增第二套图数据模型，而是把现有资产重新编队：

| 新区域 | 复用资产 | 职责 |
| --- | --- | --- |
| 全局外壳 | `TaskGraphStudioShell`、`TaskGraphTopBar`、`TaskGraphIssueBar` | 保持任务系统统一入口、保存发布动作和底部状态 |
| 左侧图层关系 | `metadata.timeline_blocks`、`standardView.units`、`standardView.nested_runtime`、`TASK_GRAPH_MODULE_FACET_ITEMS` | 展示当前图、阶段图块、GraphUnit、对象分类和可进入的层级 |
| 中间画布 | `CoordinationTopologyGraph`、`buildTaskGraphComposableStandardModel` | 把 Unit 和 PortEdge 画成当前层级图，不展开子图内部 |
| 右侧编辑台 | `TaskSystemField`、`TaskSystemSelectField`、`metadata.composable_graph` 覆盖层写入工具 | 对选中 graph / unit / port_edge / timeline_block / nested_runtime 做 scoped 编辑 |
| 底部闭环 | `buildTaskGraphPreflightReport`、`focusForPreflightIssue`、后端 `standardView.issues` | 把发布预检和标准视图问题定位回左侧、画布和右侧编辑对象 |

关键原则：

1. `TaskGraphDefinition.nodes / edges / metadata.timeline_blocks` 仍是真相源。
2. `metadata.composable_graph` 只做显式覆盖层，用于端口边、接口、Unit 和 nested runtime 的逐步迁移。
3. 前端只编辑当前父图边界，不进入子图内部节点。
4. 中文名继续作为主显示名，ID 退到辅助信息。
5. 旧“分面式模块页”的能力不丢弃，但它要被重新归位到三栏工作台的对象树、画布、Inspector 和诊断 Dock 中。

### 1.2 当前结构问题

当前拓扑页和模块页分工已经接近目标，但还存在三类问题：

1. 图结构和可组合对象分在不同页面，用户需要来回切换。
2. 左侧 rail 仍偏“新增工具”，没有承担图层关系和父子图导航。
3. 右侧 inspector 还是摘要和跳转入口，不是完整对象编辑台。

因此下一步不是把所有页面塞进一页，而是抽出一个更高层的工作台：

```text
ComposableGraphWorkbench
```

它把图层导航、画布选择、对象编辑、诊断定位统一起来。

## 2. 目标布局

### 2.0 递归展开原则

可组合编辑器的核心不是“在一页上画更多东西”，而是让图层递归自然发生：

```text
节点可以是基本任务。
节点也可以是一张收起的图。
图作为父图节点时，占据父图的一个时序点。
点击图节点时，进入子图自己的工作台。
```

因此前端必须遵守：

1. 父图画布只显示当前层级的 Unit 和 PortEdge。
2. GraphUnit 只能显示接口、版本、handoff contract、nested runtime 状态。
3. GraphUnit 的“展开”是切换编辑上下文，不是在父图画布中嵌套显示内部节点。
4. 时序显示也按层级切换：父图显示 GraphUnit 作为时序点，子图内部显示自己的阶段、循环和节点时序。
5. 面包屑或图层关系区负责表达“当前在哪个图层”，右侧编辑台负责表达“这个图节点如何对接父图”。

### 2.1 桌面布局

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│ TaskGraph TopBar：标题 / 保存 / 发布 / 标准视图状态 / 运行绑定               │
├───────────────┬───────────────────────────────────────────┬─────────────────┤
│ 左：图层关系   │ 中：当前图画布                              │ 右：对象编辑台   │
│               │                                           │                 │
│ 当前图         │ Unit / Port / PortEdge / GraphUnit         │ Graph Editor    │
│ 父子图路径     │ 当前层级，不展开子图内部                    │ Unit Editor     │
│ 阶段图块       │ 支持选择、连线、快速新增                    │ Port Editor     │
│ 对象分类       │ 支持端口显示和边类型过滤                    │ Edge Editor     │
│ 预检定位       │                                           │ Nested Editor   │
├───────────────┴───────────────────────────────────────────┴─────────────────┤
│ 底部：发布预检 / 标准视图问题 / runtime trace / 当前对象诊断                 │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 移动与窄屏布局

窄屏不强行三栏并排，而改成分段切换：

```text
图层关系
画布
对象编辑台
诊断
```

现有 CSS 已在 `max-width: 920px` 将多列布局收成单列；后续实现应沿用这个响应式策略。

## 3. 四个区域的职责

### 3.1 左侧：图层关系与对象树

左侧不编辑细节，只回答“我在哪一层、能去哪里”。

内容：

1. 当前图身份
   - graph_id
   - 中文名 / 标题
   - publish_state
   - 标准视图状态
2. 父子图路径
   - 当前图
   - 父图引用
   - 子图 GraphUnit
   - version_ref
3. 阶段图块
   - design_graph
   - creation_graph
   - closing_graph
   - review_graph
4. 对象分类
   - 普通 Unit
   - GraphUnit
   - ResourceUnit
   - HumanGateUnit
   - PortEdge
5. 预检定位
   - 点击问题时高亮对应对象
   - 不在左侧展开复杂表单

禁止：

- 禁止在左侧编辑节点协议、端口契约、边时序。
- 禁止在父图左侧直接展开子图内部节点。

### 3.2 中间：当前层级图画布

画布负责结构操作：

1. 选择对象
   - graph
   - unit
   - port
   - port_edge
   - nested_runtime
2. 结构编辑
   - 新增 Unit
   - 新增 GraphUnit
   - 从端口连边
   - 删除显式覆盖边
   - 选择已有派生边
3. 视觉语义
   - GraphUnit 显示为可进入的节点
   - ResourceUnit 显示为资源节点
   - PortEdge 显示不同线型
   - 显式覆盖边和旧边派生边要有不同标识

画布只显示当前层级。点击 GraphUnit 的“进入”才切到子图工作台。

### 3.3 右侧：对象编辑台

右侧编辑台根据选择对象切换表单。

```text
selectedSubject =
  graph
  unit
  interface
  port
  port_edge
  nested_runtime
  issue
```

#### Graph 编辑

- 图标题 / 中文名
- entry_node_id / output_node_id
- graph_contract_id
- runtime_policy
- context_policy
- 发布状态

#### Unit 编辑

- unit_id
- title / 中文名
- unit_type
- phase_id
- interface_id
- runtime_policy
- source_kind

规则：

- 派生 Unit 首期只允许覆盖 title/interface/runtime_policy。
- 不直接删除由节点派生的 Unit。

#### Interface 编辑

- interface_id
- unit_id
- display_name_zh
- input_ports
- output_ports
- memory_visibility_policy
- artifact_visibility_policy
- runtime_state_policy

#### Port 编辑

- port_id
- direction
- payload_contract_id
- required
- status_required
- visibility_policy

#### PortEdge 编辑

- edge_id
- source_unit_id / source_port_id
- target_unit_id / target_port_id
- payload_contract_id
- edge_type
- trigger_timing
- visibility_timing
- acknowledgement_timing
- propagation_timing
- handoff policy

#### GraphUnit / NestedRuntime 编辑

- linked_graph_id
- version_ref
- handoff_contract_id
- input_port_id
- output_port_id
- isolation_policy
- visibility_policy
- detach_policy

### 3.4 底部：诊断与运行状态

底部不做主编辑，只做闭环反馈：

- 发布预检问题
- 标准视图合并问题
- 当前对象相关 issue
- runtime trace / monitor
- 保存状态

点击诊断后：

```text
issue -> taskGraphEditorFocus -> 左侧定位 -> 画布高亮 -> 右侧打开对应编辑器
```

## 4. 数据与交互协议

### 4.1 选择协议

建议把当前 `TaskGraphEditorFocus` 扩展为对象选择协议：

```ts
type TaskGraphEditorSubject =
  | { kind: "graph"; graph_id: string }
  | { kind: "unit"; unit_id: string }
  | { kind: "interface"; interface_id: string; unit_id?: string }
  | { kind: "port"; unit_id: string; interface_id: string; port_id: string; direction: "input" | "output" }
  | { kind: "port_edge"; edge_id: string }
  | { kind: "nested_runtime"; plan_id: string; unit_id?: string }
  | { kind: "issue"; issue_id: string };
```

现有 `node_id / edge_id / repository_id` 继续保留兼容，但新画布应优先用 subject。

### 4.2 写入协议

所有可组合对象编辑先写入：

```text
metadata.composable_graph
```

写入对象：

- `units`
- `interfaces`
- `port_edges`
- `nested_runtime`

不直接修改：

- 子图内部节点
- 子图内部边
- runtime trace
- 已提交记忆

### 4.3 标准视图协议

后端仍负责合并：

```text
TaskGraphDefinition
  -> derived composable graph
  -> metadata.composable_graph overlay
  -> TaskGraphStandardView
  -> preflight / editor / runtime shadow compiler
```

前端不自行判断最终真相，只显示和编辑覆盖层，并读取标准视图合并结果。

## 6. 信息密度与视觉原则

当前产品是任务编排编辑器，应该保持安静、密集、可扫描。

设计原则：

1. 不做营销式大 hero。
2. 不做装饰性背景。
3. 卡片只用于对象、面板、重复项，不做卡片套卡片。
4. 左侧窄而信息密集，中间最大，右侧稳定宽度。
5. 中文名是主显示名，ID 是辅助信息。
6. 显式覆盖对象必须有标识，避免用户误以为它是原始节点。
7. 派生对象可读，覆盖对象可编辑。
8. 所有对象编辑后都要能被标准视图和预检看见。

推荐桌面宽度：

```css
grid-template-columns:
  minmax(220px, 260px)
  minmax(520px, 1fr)
  minmax(300px, 360px);
```

## 7. 实施顺序

### 阶段 A：布局骨架

目标：不改数据语义，只建立三栏工作台骨架。

文件：

- `TaskGraphComposableEditorPage.tsx`
- `TaskGraphGraphLayerRail.tsx`
- `TaskGraphObjectInspector.tsx`
- `TaskGraphDiagnosticsDock.tsx`
- `globals.css`

完成标准：

- 左侧显示当前图、阶段图块、Unit 分类。
- 中间复用现有 `CoordinationTopologyGraph` 显示当前层级。
- 右侧显示当前选中节点/边摘要。
- 不影响旧 topology/modules 页面。

### 阶段 B：对象选择协议

目标：从 `node_id / edge_id` 升级到 `selectedSubject`。

完成标准：

- 画布选中 Unit 后右侧打开 Unit inspector。
- 选中 PortEdge 后右侧打开 PortEdge inspector。
- 预检问题能定位并打开对应 inspector。

### 阶段 C：PortEdge 编辑闭环

目标：把当前模块页的显式端口边编辑迁到右侧 inspector。

完成标准：

- 右侧可编辑 `metadata.composable_graph.port_edges`。
- 保存后标准视图显示 overlay。
- 发布预检能阻断错误端口边。

### 阶段 D：Interface / Port 编辑闭环

目标：支持 Unit 接口和端口覆盖。

完成标准：

- 右侧可编辑 interface / ports。
- 画布可显示端口。
- PortEdge 只能连接存在端口。

### 阶段 E：GraphUnit 与父子图导航

目标：图层关系真正发挥作用。

完成标准：

- 左侧显示父子图路径。
- GraphUnit 可进入子图工作台。
- 父图只编辑子图接口和 nested runtime，不混入子图内部。

## 9. 最终形态

最终编辑体验应是：

```text
用户在左侧选择“长篇小说项目图 / 正式创作图”。
中间看到当前图的 Unit 和 PortEdge。
用户点击“正式创作图”这个 GraphUnit。
右侧出现 linked_graph_id、version_ref、handoff_contract、nested_runtime。
用户点击一条从设计图到创作图的 PortEdge。
右侧出现 payload contract、trigger timing、visibility timing、ack policy。
用户保存后，标准视图合并覆盖层。
发布预检发现问题时，点击问题自动回到对应图层、画布对象和右侧编辑器。
```

这就是最适合当前项目的布局：左边管层级，中间管结构，右边管契约，底部管闭环。

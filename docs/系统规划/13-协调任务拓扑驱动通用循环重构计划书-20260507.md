# 协调任务拓扑驱动通用循环重构计划书

日期：2026-05-07

## 一、文档定位

本文用于锁定下一轮协调任务 runtime 重构方案：把当前在 `TaskRunLoop` 中按特定任务手工续接的协调流程，重构为“LangGraph 状态机 + 拓扑结构 + 阶段契约 + 输入绑定规则”驱动的通用循环机制。

本计划不直接改代码。它先明确当前问题出在哪里、目标机制长什么样、哪些文件要改、如何迁移、如何验收。

本轮目标：

- `TaskRunLoop` 不再知道“长篇小说”这类业务任务的上游/下游字段名。
- 协调任务的下一跳由 `LangGraphCoordinationRuntime` 根据拓扑和阶段状态决定。
- 下一阶段输入由通用 `ContinuationInputBinder` 根据阶段契约和产物索引解析。
- 所有长期任务、写作任务、健康任务、开发任务都能复用同一套协调循环。
- `CoordinationFlow`、`CoordinationRun`、`CoordinationNodeRun` 继续作为正式 trace/projection，但不再拥有主状态推进权。

本轮不做：

- 不重写单 Agent runtime。
- 不重写模型调用、工具调用和 output boundary。
- 不推翻现有 `TaskRun / AgentRun / CoordinationRun / CoordinationNodeRun / AgentHandoffEnvelope / CoordinationMergeResult` trace 对象。
- 不把任务语义塞进 Agent 组。
- 不在 LangGraph 旁边再造第二套协调状态机。

### 2.2 当前结构性问题

问题集中在 `backend/orchestration/runtime_loop/task_run_loop.py`。

当前 `TaskRunLoop` 同时负责：

- 单 Agent runtime loop
- 协调任务创建
- 协调图 runner 调用
- node run / handoff / merge result 同步
- continuous delivery 下一跳判断
- 下一阶段 `explicit_inputs` 拼装
- 下一阶段用户消息合成
- 长篇任务进度回答定制

这导致一个核心问题：

`runtime loop 本应执行通用状态机，但现在掌握了业务任务的上游产物字段、下游任务字段和长篇小说阶段语义。`

最明显的硬编码点在：

- `TaskRunLoop._build_continuation_explicit_inputs`
  - 写死 `task.writing.longform_novel_project`
  - 写死 `task.writing.novel_bible_build`
  - 写死 `task.writing.volume_planning`
  - 写死 `task.writing.chapter_planning`
  - 写死 `task.writing.chapter_drafting`
  - 写死 `task.writing.continuity_audit`
  - 写死 `project_spec_ref / novel_bible_ref / volume_plan_ref / chapter_plan_ref / accepted_chapter_refs`

- `TaskRunLoop._build_continuation_user_message`
  - 固定写“继续推进长篇小说持续交付流程”

- `TaskRunLoop.run_single_agent_stream`
  - 对 `task_mode == "longform_novel_project"` 单独生成 longform progress summary

- `TaskRunLoop._upsert_finished_task_run`
  - 只把 `coordination_mode == "continuous_delivery"` 视为可持续推进
  - 下一跳只靠 `stage_sequence` 中下一个 `task_ref`

这不是一个局部 bug，而是职责边界偏移：

- 任务系统已经声明了任务图，但 runtime 还在按业务名补输入。
- 拓扑已经能表达节点和边，但阶段输入契约没有正式建模。
- 产物已经落在 trace 里，但没有通用产物索引和引用解析规则。
- 持续交付已经能推进，但只对长篇任务形成了特殊路径。

### 2.3 当前长篇任务为什么暴露问题

长篇小说任务不是问题本身，它只是第一个足够长、足够多阶段、足够依赖上游产物的压力测试。

它暴露了三类问题：

1. 多阶段任务不能靠 prompt 和人工阶段脚本维持。
2. 上游产物引用不能靠 `if next_task_ref == ...` 拼字段。
3. 协调监控和真实执行应来自同一套 `CoordinationFlowState`，而不是会话层和协调层各跑一套解释。

因此，本轮重构的目标不是“修长篇小说”，而是把长篇小说需要的持续交付能力抽象成通用协调循环。

### 2.4 LangGraph 使用现状校准

本项目已经安装并使用 LangGraph：

- `langgraph==1.1.4`
- `langgraph-checkpoint==4.0.1`
- `langchain==1.2.14`
- `langchain-core==1.2.23`

当前 `LangGraphCoordinationRunner` 的价值是“编译并遍历协调图”，但还没有成为真正 runtime：

- `StateGraph` 的 state 只有 `visited_node_ids / traversal_nodes / handoff_edges`。
- 节点函数只记录 traversal node，不启动阶段任务，不处理 artifact，不处理 human gate。
- `compile()` 后立即 `invoke()`，没有 checkpointer，没有 `thread_id`，不能长期恢复。
- runner 为避免环路使用 `_acyclic_execution_edges` 跳过循环边，这与长流程需要 revision loop / wait / resume 的目标冲突。
- 真实推进仍由 `TaskRunLoop._upsert_finished_task_run`、`advance_coordination_flow_state` 和 `_build_continuation_explicit_inputs` 这些自定义逻辑承担。

因此本计划需要校正：目标不是在 `TaskRunLoop` 外再抽一个自研 `ContinuationEngine`，而是让 LangGraph 成为协调任务循环的主状态机。`ContinuationInputBinder`、`ArtifactRefIndex`、trace 写入器都应成为 LangGraph node/helper，而不是新的主循环所有者。

### 2.5 LangGraph 参考约束

根据 LangGraph 官方 Graph API、Persistence、Interrupts 文档，本轮重构采用以下约束：

- `StateGraph` 的 `State` 是协调任务当前快照，节点只返回 state update，不返回整份世界。
- 节点负责执行或准备阶段工作，边负责决定下一步；需要“更新状态并跳转”时使用 `Command(update=..., goto=...)`。
- 同一个节点不要混用普通静态边和 `Command`/conditional edge 动态路由，否则同一节点可能同时触发多条路径。
- 长任务恢复必须使用 checkpointer，并把 `coordination_run_id` 映射为 LangGraph `thread_id`。
- 人工审核/人工放行应使用 `interrupt()` 暂停，用 `Command(resume=...)` 恢复；resume 必须使用同一个 `thread_id`。
- `interrupt()` 前的副作用必须幂等，复杂写入应拆到单独节点，避免 resume 后重复执行。

参考资料：

- LangGraph Graph API：https://docs.langchain.com/oss/python/langgraph/graph-api
- LangGraph Persistence：https://docs.langchain.com/oss/python/langgraph/persistence
- LangGraph Interrupts：https://docs.langchain.com/oss/python/langgraph/interrupts

## 三、局部设计原则

本计划沿用现有系统设计书中的边界：

- 任务系统负责：任务定义、任务契约、workflow、协调任务语义、拓扑、协议、验收要求。
- 编排系统负责：Agent 名册、Agent 组、Agent runtime profile、单 Agent 运行体装配。
- runtime 负责：执行循环、状态推进、事件、checkpoint、trace、恢复。

由此得到本轮约束：

1. `TaskRunLoop` 可以调度，但不能写业务任务知识。
2. 任务图可以声明阶段顺序、分支、循环、输入需求和输出写回。
3. 输入绑定必须由契约驱动，不能由任务名字符串驱动。
4. 拓扑中的节点状态是协调监控 UI 的唯一可信来源。
5. 产物引用必须进入运行态索引，下一阶段只消费 ref，不猜文件路径。
6. 阶段循环必须支持 `continue / wait / retry / fail / complete`，不能只有 completed 和 failed。

## 四、目标架构

### 4.1 总体结构

```text
CoordinationTaskDefinition
  ├─ stage_sequence / topology nodes
  ├─ edges
  ├─ protocol
  ├─ continuation_policy
  └─ stage_contracts
       ├─ input_bindings
       ├─ output_mappings
       ├─ gate_policy
       └─ retry_policy

TaskRunLoop
  ├─ 执行当前 TaskRun
  ├─ 写 task_result / artifact refs
  └─ 向 LangGraphCoordinationRuntime 投递阶段完成事件

LangGraphCoordinationRuntime
  ├─ StateGraph[CoordinationRuntimeState]
  ├─ 根据拓扑/边策略决定下一节点
  ├─ 根据 StageContract 调用 ContinuationInputBinder
  ├─ 发起/等待/恢复阶段 TaskRun
  ├─ 处理 retry / blocked / human_gate / complete
  └─ 通过 CoordinationTraceAdapter 写入 CoordinationRun / NodeRun / MergeResult

ContinuationInputBinder
  ├─ 从 current task_result 取当前产物
  ├─ 从 RuntimeStateIndex 取历史上游产物
  ├─ 根据 binding rules 填 explicit_inputs
  └─ 缺必需输入时生成 blocked result

ArtifactRefIndex
  ├─ task_ref -> latest output refs
  ├─ output_contract_id -> latest output refs
  ├─ stage_id -> result refs
  └─ coordination_run_id -> accepted refs

CoordinationTraceAdapter
  ├─ 把 LangGraph state/update 投影到现有 formal trace
  ├─ 维护 coordination_flow diagnostics
  ├─ 维护 node_run status
  └─ 供前端拓扑监控读取同一份状态
```

主状态所有权：

- `LangGraphCoordinationRuntime` 是协调循环唯一状态推进者。
- `CoordinationFlowState` 退为 trace/projection 结构，用于事件、诊断、前端监控。
- `TaskRunLoop` 是单阶段执行器和事件投递器，不拥有“下一跳”决策。
- `ContinuationInputBinder` 只绑定输入，不选择下一阶段。
- `ArtifactRefIndex` 只查询产物引用，不解释业务流程。

执行请求所有权：

- 下一阶段启动请求必须由 `LangGraphCoordinationRuntime.stage_execute` 产出。
- `TaskRunLoop` 只能执行 `StageExecutionRequest`，不能自行拼接下一阶段 `task_ref / explicit_inputs / message`。
- `TaskRunLoop._continue_coordination_delivery_stream` 在 cutover 后应改造成 `run_stage_execution_request_stream` 或等价内部执行器。
- `continuation_payload` 如果短期保留，只能是 `StageExecutionRequest.to_dict()` 的兼容外壳，不能再由 `TaskRunLoop` 内部构造。

### 4.2 新增核心对象

#### CoordinationStageContract

阶段契约。它回答：

`这个阶段需要什么输入、产出什么、怎样进入下一阶段。`

阶段契约必须是真实可解析配置，不是文档样例。第一阶段落地方式：

- 存储位置：`CoordinationTaskDefinition.metadata.stage_contracts`。
- 索引键：`stage_id`。
- 每个 `stage_contract.stage_id` 必须能在 `stage_sequence` 或 topology node metadata 中找到。
- 每个 `stage_contract.task_ref` 必须在 `CoordinationTaskDefinition.subtask_refs` 或根任务 `metadata.task_id` 中找到。
- `input_bindings.source` 只能使用白名单，第一版为 `current_output / latest_output / latest_output_by_contract / inherited_input / literal / collect / stage_output`。
- 缺少 required input 时，runtime 必须进入 `blocked`，禁止自动用空字符串、final answer 或自然语言摘要补齐。

建议字段：

```json
{
  "stage_id": "chapter_drafting",
  "task_ref": "task.writing.chapter_drafting",
  "required_inputs": ["chapter_plan_ref", "context_refs"],
  "optional_inputs": ["run_request", "artifact_root"],
  "input_bindings": [
    {
      "input_key": "chapter_plan_ref",
      "source": "latest_output",
      "task_ref": "task.writing.chapter_planning",
      "required": true
    },
    {
      "input_key": "context_refs",
      "source": "collect",
      "items": [
        {"source": "latest_output", "task_ref": "task.writing.novel_bible_build"},
        {"source": "latest_output", "task_ref": "task.writing.volume_planning"},
        {"source": "latest_output", "task_ref": "task.writing.chapter_planning"}
      ],
      "required": true
    }
  ],
  "output_mappings": [
    {
      "output_key": "chapter_refs",
      "target_scope": "coordination_run",
      "ref_kind": "artifact"
    }
  ],
  "gate_policy": "artifact_validation_required",
  "on_success": "advance",
  "on_failure": "fail_closed"
}
```

#### StageExecutionRequest

阶段执行请求。它回答：

`LangGraph 已经决定要执行哪个阶段，单 Agent runtime 应该用什么确定输入去执行它。`

这是 `LangGraphCoordinationRuntime` 和 `TaskRunLoop` 之间的唯一执行边界对象。

建议字段：

```json
{
  "request_id": "stageexec:coordrun:...:volume_planning:0003",
  "coordination_run_id": "coordrun:...",
  "thread_id": "coordrun:...",
  "root_task_run_id": "taskrun:...",
  "stage_id": "volume_planning",
  "node_id": "volume_planning",
  "task_ref": "task.writing.volume_planning",
  "agent_id": "agent:22",
  "agent_profile_id": "",
  "runtime_lane": "volume_acceptance",
  "explicit_inputs": {
    "novel_bible_ref": "artifact:...",
    "volume_index": 1
  },
  "message": "继续执行协调任务阶段：volume_planning。",
  "artifact_root": "docs/系统规划/任务系统实测记录/artifacts/longform_novel_project",
  "expected_outputs": [
    {
      "output_key": "volume_plan_ref",
      "ref_kind": "artifact",
      "required": true
    }
  ],
  "idempotency_key": "coordrun:...:volume_planning:input-hash"
}
```

硬规则：

- `StageExecutionRequest` 由 `stage_execute` 生成。
- `explicit_inputs` 来自 `ContinuationInputBinder`，不是 `TaskRunLoop` 二次推断。
- `message` 只能是通用阶段执行语，不允许写入长篇小说等业务专用续跑话术。
- `idempotency_key` 必须由 `coordination_run_id + stage_id + input refs hash` 生成，防止 resume 后重复启动同一阶段。
- `TaskRunLoop` 执行完成后只回传结构化 `task_result_ready` 事件。

#### CoordinationContinuationPolicy

协调任务级续跑策略。它回答：

`这个协调任务是否持续推进、如何处理循环、什么时候停。`

建议字段：

```json
{
  "mode": "topology_driven",
  "auto_continue": true,
  "max_auto_steps": 100,
  "stop_on_missing_required_input": true,
  "terminal_policy": "terminal_node_or_stop_condition",
  "human_gate_stage_ids": [],
  "retry_budget": {
    "default": 0,
    "revision_loop": 2
  }
}
```

#### ArtifactRefIndex

运行态产物引用索引。它回答：

`某个任务、阶段、契约或协调流程最新可用产物 ref 是什么。`

第一阶段不一定要新建物理文件，可以先在 `RuntimeStateIndex` 上增加查询方法：

- `latest_output_refs(task_ref=...)`
- `latest_output_refs(output_contract_id=...)`
- `latest_stage_result_refs(coordination_run_id=..., stage_id=...)`
- `accepted_refs(coordination_run_id=..., ref_kind=...)`

后续再独立为 `ArtifactRefIndex`。

#### CoordinationRuntimeState

LangGraph 协调运行时状态。它回答：

`当前协调任务走到哪里、哪些节点完成、下一步应该等待什么、哪些产物已经可用。`

建议字段：

```python
from typing import Annotated, TypedDict
import operator

class CoordinationRuntimeState(TypedDict, total=False):
    coordination_run_id: str
    root_task_run_id: str
    active_node_id: str
    active_stage_id: str
    active_task_run_id: str
    node_statuses: dict[str, str]
    stage_results: dict[str, dict[str, object]]
    artifact_refs: Annotated[list[dict[str, object]], operator.add]
    pending_inputs: dict[str, object]
    missing_required_inputs: list[str]
    retry_counts: dict[str, int]
    human_gate: dict[str, object]
    terminal_status: str
    diagnostics: dict[str, object]
```

状态约束：

- 可追加日志类字段使用 reducer，例如 `Annotated[list, operator.add]`。
- 状态中只保存稳定引用、状态、计数和诊断；长文本正文和大产物只保存 ref。
- `active_*` 字段必须与 trace 中 running node/run 保持一致。
- 每次节点更新后由 `CoordinationTraceAdapter` 投影到 `coordination_flow`，前端不直接读取 LangGraph 内部对象。

#### LangGraphCoordinationRuntime

LangGraph 协调运行时。它回答：

`如何把任务定义编译成可恢复、可中断、可持续推进的协调状态机。`

核心节点建议：

- `stage_prepare`
  - 读取当前 node/stage contract。
  - 调用 `ContinuationInputBinder`。
  - 缺必需输入时 `Command(goto="blocked")`。

- `stage_execute`
  - 生成下一阶段 `StageExecutionRequest`。
  - 对异步/长耗时阶段写入 `active_task_run_id` 后返回等待状态。
  - 不直接在 LangGraph 节点里塞业务任务知识。

- `stage_accept`
  - 接收 TaskRun 完成事件。
  - 收集 artifact refs。
  - 根据 stage contract output mapping 更新 state。

- `route_next`
  - 根据 graph edge、stage status、retry budget、terminal condition 决定下一节点。
  - 简单线性图走下一节点；revision loop 按预算回跳；terminal edge 结束。

- `human_gate`
  - 需要人工确认时调用 `interrupt()`。
  - 恢复时用 `Command(resume=...)` 注入审核结论。

- `blocked`
  - 记录缺失输入、缺失产物、不可满足前置条件。
  - 不伪装 completed。

- `complete`
  - 写终态、final artifact refs、summary refs。
  - 通过 trace adapter finalize formal objects。

路由规则：

- 静态 DAG 可使用普通 edge / conditional edge。
- revision loop、人工审核、动态分支优先用 `Command(goto=...)` 或 conditional edge。
- 同一节点只能选择一种路由机制，避免普通边和动态 goto 同时生效。
- 不再为了 LangGraph 编译而删除环路边；循环必须由 retry budget / gate policy / terminal condition 控制。

### 4.3 新增后端模块

建议新增：

- `backend/orchestration/runtime_loop/continuation_policy.py`
- `backend/orchestration/runtime_loop/continuation_inputs.py`
- `backend/orchestration/runtime_loop/stage_execution_request.py`
- `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
- `backend/orchestration/runtime_loop/coordination_trace_adapter.py`
- `backend/orchestration/runtime_loop/langgraph_checkpoint_adapter.py`
- `backend/orchestration/runtime_loop/artifact_refs.py`

模块职责：

- `continuation_policy.py`
  - 解析 coordination task metadata
  - 解析 stage contracts
  - 判断是否 auto continue
  - 提供 edge/gate/retry policy 给 LangGraph 节点使用

- `continuation_inputs.py`
  - 实现 `ContinuationInputBinder`
  - 根据 binding rules 填充 `explicit_inputs`
  - 返回 missing required inputs

- `stage_execution_request.py`
  - 定义 `StageExecutionRequest`
  - 定义 `TaskResultReadyEvent`
  - 定义 request idempotency key
  - 提供 dict/dataclass 转换和校验

- `langgraph_coordination_runtime.py`
  - 编译 `CoordinationGraphSpec` 为 `StateGraph`
  - 维护 `CoordinationRuntimeState`
  - 处理 stage prepare / execute / accept / route / blocked / human gate / complete
  - 使用 checkpointer 和 `thread_id=coordination_run_id` 支撑长任务恢复

- `langgraph_checkpoint_adapter.py`
  - 封装 LangGraph checkpointer
  - 第一版可基于现有 `RuntimeCheckpointStore` 或独立 `coordination_checkpoints/` 文件目录落盘
  - 对外暴露 LangGraph compile 所需 checkpointer 实例
  - 保证 `thread_id=coordination_run_id` 可恢复

- `coordination_trace_adapter.py`
  - 把 LangGraph state/update 写回现有 trace 对象
  - 生成 `coordination_flow` projection
  - 同步 `CoordinationNodeRun / AgentHandoffEnvelope / CoordinationMergeResult`
  - 给前端协调监控提供唯一可信状态源

- `artifact_refs.py`
  - 从 `task_result`、trace、state index 中收集产物 refs
  - 提供统一查询接口

保留但降级：

- `backend/orchestration/runtime_loop/langgraph_coordination_runner.py`
  - 短期保留为 topology smoke test / graph diagnostics。
  - 不再作为协调任务 runtime 入口。
  - 后续可合并进 `langgraph_coordination_runtime.py` 或删除。

### 4.4 `TaskRunLoop` 目标职责

重构后 `TaskRunLoop` 保留：

- 启动 TaskRun
- 执行单 Agent loop
- 调用工具和模型
- 做 artifact validation
- 写 checkpoint
- 写 task_result
- 把阶段完成事件交给 `LangGraphCoordinationRuntime`
- 执行 `StageExecutionRequest` 并返回结构化阶段结果事件

重构后 `TaskRunLoop` 移除：

- 不再按 `task.writing.*` 任务名拼输入
- 不再生成长篇任务专用 continuation message
- 不再知道 `project_spec_ref / novel_bible_ref / volume_plan_ref` 等字段含义
- 不再直接判断只有 `continuous_delivery` 才能持续推进
- 不再自行构造 continuation payload

## 五、固定执行流

### 5.1 当前阶段完成

```text
TaskRunLoop 完成当前 TaskRun
  -> 生成 task_result
  -> artifact validation
  -> upsert TaskRun / AgentRunResult
  -> LangGraphCoordinationRuntime.resume_from_task_result(...)
```

### 5.2 协调流程推进

```text
LangGraphCoordinationRuntime
  -> 用 coordination_run_id 作为 thread_id 恢复 StateGraph
  -> stage_accept 接收 task_result 和 artifact refs
  -> route_next 根据 topology / edge policy / retry budget 选择下一节点
  -> stage_prepare 解析下一阶段 stage contract
  -> 调用 ContinuationInputBinder
```

### 5.3 输入绑定

```text
ContinuationInputBinder
  -> current task_result output refs
  -> current explicit_inputs
  -> ArtifactRefIndex 查询历史上游 refs
  -> stage_contract.input_bindings 填 explicit_inputs
  -> 如果 required input 缺失，返回 blocked continuation
```

### 5.4 下一阶段启动

```text
LangGraphCoordinationRuntime.stage_execute
  -> 生成 StageExecutionRequest
  -> TaskRunLoop.run_stage_execution_request_stream 执行下一 TaskRun
  -> 写入 active_task_run_id 并等待阶段完成事件
```

注意：下一阶段 message 只描述通用协调续跑，不写业务专属话术。业务说明进入 `explicit_inputs.run_request` 或 stage contract 的 prompt hint。

### 5.5 人工审核和恢复

```text
human_gate node
  -> interrupt({"coordination_run_id": ..., "stage_id": ..., "artifact_refs": ...})
  -> 前端/人工审核提交结论
  -> app.invoke(Command(resume={...}), config={"configurable": {"thread_id": coordination_run_id}})
  -> route_next
```

人工审核规则：

- 审核员没有“消失”，它可以是自动验收节点，也可以是 `human_gate` 暂停点。
- 如果用户选择人工审核，系统应显示 waiting_for_human，而不是把审核员节点从拓扑中隐藏。
- 人工审核不应在会话层用普通聊天消息绕过协调 runtime。

### 5.6 持久化和恢复

```text
coordination_run_id == LangGraph thread_id
StateGraph.compile(checkpointer=LangGraphCheckpointStoreAdapter(...))
每次 stage update 后：
  -> LangGraph checkpointer 保存状态
  -> CoordinationTraceAdapter 写 formal trace/projection
```

持久化规则：

- 开发和单测可用 `InMemorySaver`。
- 真实长任务必须接入 `LangGraphCheckpointStoreAdapter` 或数据库型 durable checkpointer，不能只靠内存。
- checkpoint 是恢复主依据，`coordination_flow` 是展示投影，不可反向覆盖 LangGraph 状态。
- `RuntimeCheckpointStore` 当前按 `task_run_id` 保存单 Agent checkpoint，不能直接等同于 LangGraph checkpoint；必须通过 adapter 或新目录建立 `thread_id` 维度的协调 checkpoint。

### 5.7 人工审核 API 边界

后端需要新增一个恢复入口，供前端/人工审核恢复 LangGraph：

```text
POST /api/orchestration/coordination-runs/{coordination_run_id}/resume
  body:
    {
      "resume_kind": "human_gate_decision",
      "stage_id": "chapter_review",
      "accepted": true,
      "comment": "通过",
      "artifact_refs": ["artifact:..."]
    }
```

入口职责：

- 校验 `coordination_run_id` 存在。
- 用同一个 `thread_id=coordination_run_id` 恢复 LangGraph。
- 调用 `Command(resume=...)`。
- 由 `CoordinationTraceAdapter` 写入 human gate 结果、node status、后续 `StageExecutionRequest` 或 complete 状态。

入口禁止：

- 不允许直接改 `coordination_flow` 来伪造审核通过。
- 不允许通过普通会话消息表达“审核通过”。
- 不允许绕过 LangGraph checkpoint 直接启动下一阶段。

## 六、数据模型变更计划

### 6.1 `CoordinationTaskDefinition.metadata`

新增可选字段：

- `continuation_policy`
- `stage_contracts`

兼容期允许旧 `stage_sequence` 存在，但新逻辑优先读 `stage_contracts`。

配置验收规则：

- `stage_contracts` 不存在时，只允许进入 shadow/diagnostics，不允许正式持续交付 cutover。
- `stage_contracts[*].stage_id` 必须唯一。
- `stage_contracts[*].task_ref` 必须非空。
- `input_bindings` 中 required 项必须有可解析 source。
- `output_mappings` 中 required 输出必须能映射到 `task_result.output_refs` 或 artifact validation 结果。

### 6.2 `TopologyTemplate.nodes`

节点可增加：

- `stage_id`
- `task_ref`
- `input_contract_id`
- `output_contract_id`
- `gate_policy`

但不强制第一阶段迁入，避免前端和存储一起震荡。

### 6.3 `TaskTemplate.metadata`

可增加：

- `default_input_bindings`
- `output_ref_kind`
- `artifact_role`

用于没有 stage contract 的单任务 fallback。

### 6.4 `RuntimeStateIndex`

新增查询方法，不改变现有存储格式：

- `list_completed_task_runs(task_ref: str | None = None)`
- `latest_task_result_output_refs(task_ref: str)`
- `latest_task_result_output_refs_by_contract(output_contract_id: str)`

如果当前 trace 读取成本高，第二阶段再落独立 `artifact_ref_index`。

### 6.5 LangGraph Checkpoint

新增协调 runtime checkpoint 配置：

- `coordination_run_id` 固定映射为 LangGraph `thread_id`。
- `root_task_run_id` 作为 state 字段保存，用于追溯主任务。
- `active_task_run_id` 作为 state 字段保存，用于等待当前阶段回调。
- `checkpoint_namespace` 可按 task family 或 coordination task ref 分区。

第一阶段落地策略：

- 单测使用 `InMemorySaver`。
- 本地实测实现 `LangGraphCheckpointStoreAdapter`，可以复用 `RuntimeCheckpointStore` 的原子写文件能力，但存储键必须是 `coordination_run_id/thread_id`。
- 生产/长任务必须换成 durable checkpointer 后才能允许“无人值守跑完百万字”。

### 6.6 事件输入模型

LangGraph runtime 接收的外部事件应结构化：

```json
{
  "event_type": "task_result_ready",
  "coordination_run_id": "coordrun:...",
  "task_run_id": "taskrun:...",
  "stage_id": "volume_planning",
  "task_ref": "task.writing.volume_planning",
  "task_result_ref": "task_result:...",
  "artifact_refs": ["artifact:..."],
  "accepted": true
}
```

禁止用自然语言消息让 runtime 猜“当前完成了哪个阶段”。

## 七、迁移与切换规则

### 7.1 Shadow 模式

第一阶段 LangGraph runtime 只做 shadow：

- 旧 `_build_continuation_explicit_inputs` 继续产出 payload。
- 新 `LangGraphCoordinationRuntime` 同时根据同一事件推进 shadow state。
- 新 `ContinuationInputBinder` 产出 shadow explicit inputs。
- 测试比较旧 payload、新 state 的 next node、next task、关键输入 refs 是否一致。
- 如果不一致，记录 diagnostics，不影响运行。
- shadow state 必须通过 `CoordinationTraceAdapter` 生成投影，验证前端能从同一状态源看到 running node。

### 7.2 Cutover 模式

满足以下条件后切换：

- 长篇链路 `project_spec -> novel_bible -> volume_plan -> chapter_plan -> chapter_draft` 输入全部由 stage contract 解析。
- 至少一个非写作协调任务使用同一机制通过回归。
- `TaskRunLoop` 中没有新增业务任务名分支。

切换方式：

- 默认启用新 `LangGraphCoordinationRuntime`
- 旧 `_build_continuation_explicit_inputs` 保留一个版本作为显式 fallback
- fallback 只允许通过配置开关启用，不允许自动静默启用
- fallback 只能在 LangGraph checkpointer 不可用或迁移期回滚时使用，不能成为常态双轨。

### 7.3 Cleanup 模式

切换稳定后删除：

- `TaskRunLoop._build_continuation_explicit_inputs`
- `TaskRunLoop._build_continuation_user_message` 中的长篇专用措辞
- `task_mode == "longform_novel_project"` 的专用 progress summary 分支

如需展示长篇进度，应由通用 `CoordinationProgressPresenter` 根据 flow state 渲染。

## 十一、风险控制

### 风险一：配置过度复杂

控制方式：

- 第一版只支持少量 binding source。
- 优先覆盖线性持续交付。
- branch/join/revision loop 第二阶段再完整化。

### 风险二：旧长篇实测被破坏

控制方式：

- Shadow 模式先比较新旧 payload。
- 旧 fallback 保留一轮。
- 长篇实测作为 cutover gate。

### 风险三：产物 ref 不稳定

控制方式：

- 先统一 ref 收集逻辑。
- artifact validation 后才能写入 accepted refs。
- 缺 ref 时 blocked，而不是继续跑模型。

### 风险四：前端显示与后端状态不一致

控制方式：

- 前端只读 trace_reader 暴露的 `coordination_flow` 和 node status。
- 不在前端重新推断运行中 agent。

### 风险五：LangGraph 和自定义 flow 双主状态冲突

控制方式：

- `LangGraphCoordinationRuntime` 是唯一推进者。
- `coordination_flow` 只允许由 `CoordinationTraceAdapter` 从 LangGraph state 投影生成。
- 禁止 `advance_coordination_flow_state` 在正式路径中独立选择下一跳。
- `TaskRunLoop` 不允许在正式路径生成下一阶段 request。

### 风险六：长任务恢复只在内存中可用

控制方式：

- 单测可用 `InMemorySaver`，真实长任务必须使用 durable checkpointer。
- `coordination_run_id/thread_id` 必须写入 trace，方便恢复、排错和前端定位。
- 每次阶段完成后同时写 LangGraph checkpoint 和 formal trace。

### 风险七：StageContract 配置缺失导致 runtime 靠猜测运行

控制方式：

- cutover 前必须为目标协调任务补齐 `stage_contracts`。
- 正式路径缺 `stage_contracts` 直接 blocked，不回退到任务名硬编码。
- 编译协调图时校验 stage contract 与 topology/task refs 一致。

## 十二、禁止事项

实施过程中禁止：

- 在 `TaskRunLoop` 新增任何 `task.writing.xxx` 业务分支。
- 为某个任务补一个临时 `if next_task_ref == ...`。
- 把用户业务需求塞进启动脚本产物中伪装成果。
- 用最终回答代替 artifact ref。
- 让协调监控从会话 token 流中猜任务状态。
- 为兼容旧路径长期保留两套主逻辑。
- 在 LangGraph 外再实现一套拥有下一跳决策权的 `ContinuationEngine`。
- 在正式执行路径继续使用 `planning_only` runner。
- 为了让图可编译而静默删除 revision loop / human gate 等循环边。
- 在同一个 LangGraph 节点上同时挂静态边和 `Command(goto=...)` 动态跳转。
- 人工审核使用普通聊天消息绕过 `interrupt()` / `Command(resume=...)`。
- `TaskRunLoop` 在正式路径构造 `StageExecutionRequest` 或 continuation payload。
- `StageContract` 缺失时自动回退到 `_build_continuation_explicit_inputs`。
- 通过直接修改 `coordination_flow` 伪造 human gate 通过。

## 十三、结论

当前系统的问题不在于“长篇小说任务太复杂”，而在于协调任务循环还没有真正交给 LangGraph runtime。任务图、协议、运行态对象都已经具备雏形，但 continuation 的关键部分仍在 `TaskRunLoop` 中按业务名手工续接；现有 LangGraph runner 只是 `planning_only` 的拓扑遍历。

下一步应把协调循环收束到 `LangGraphCoordinationRuntime`：由 `StateGraph` 持有 `CoordinationRuntimeState`，由 topology/stage contract/edge policy 决定下一跳，由 input binder 解析产物引用，由 trace adapter 投影到现有 formal trace。这样长篇小说只是一个协调任务实例，后续健康修复、开发流水线、数据处理、研究报告等长流程都能复用同一套 LangGraph loop。

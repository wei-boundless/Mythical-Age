# TaskGraph 时序主轴与确定性记忆闭环工程实施计划

## 0. 本次修订结论

本计划重新修订之前的错误表述：**时序不是节点，也不应该由 Scheduler 生成“节点时序点”。**

在本项目里，时序系统应该像系统 clock 一样贯穿一次 TaskGraph 运行的完整生命周期。节点、边、阶段、循环、审核、记忆读写、产物提交、人工门控、LangGraph checkpoint 都只是时序账本里的事件参与者。

正确边界是：

```text
TaskGraph Definition
  定义：任务拓扑、节点、边、阶段、循环、记忆资源、产物资源、交接关系

TaskGraph Scheduler
  判断：当前哪些节点允许被调度，哪些节点 blocked，原因是什么

LangGraph Runtime Kernel
  执行：推进协调状态机、保存物理 checkpoint、支持 resume / interrupt / stream

TimelineLedger
  记录：真实发生过什么，并分配全局单调 clock_seq 与层级 scope_path

Context Resolvers
  装配：在某个 clock/scope 下，节点能读哪些记忆、产物、返修包和交接包

Receipt / Commit Layer
  裁决：节点输出哪些被接受，何时对后续时序可见

Projection / Prompt System
  表达：节点是谁、要做什么、如何使用输入包、必须产出什么
```

所以本计划的核心不是“给节点加一个 timeline 字段”，而是建立一条可恢复、可审计、可复用的运行主轴：

```text
Graph topology
  -> Scheduler evaluates readiness
  -> TimelineLedger appends scheduler_evaluated / node_dispatch_requested
  -> LangGraph runtime advances physical state and links checkpoint
  -> Context resolvers assemble memory/artifact/revision/handoff packets by clock/scope
  -> StageExecutionRequest carries dispatch event refs and resolved packets
  -> Node executes outside or inside runtime lane
  -> Runtime receives result
  -> Receipt validates and commits accepted outputs
  -> TimelineLedger appends result/receipt/checkpoint events
  -> Later resolvers read only committed versions visible at their clock/scope
```

这套结构必须服务通用 TaskGraph 编辑器，不能变成写作专用系统。写作长任务只能作为验收样例之一。

## 2. 外部框架边界：LangGraph 该用什么，不该用什么

### 2.1 应该用 LangGraph 的部分

#### StateGraph

继续用 `StateGraph` 表达协调运行时内部状态机。

当前内部图：

```text
stage_accept -> route_next -> stage_prepare -> stage_execute / blocked / complete
```

后续可以重构为更清晰的 runtime kernel，但不要把用户编辑的 TaskGraph 直接等同于 LangGraph StateGraph。

#### Checkpointer / thread_id

应该建立 LangGraph 物理 checkpoint 底座：

```text
thread_id = coordination_run_id
checkpoint = physical runtime state snapshot
checkpoint_ref linked into TimelineEvent
```

当前自建 `LangGraphCheckpointStoreAdapter` 可以作为过渡层，但接口要朝 LangGraph native checkpointer 靠拢。

#### Interrupt / Resume

LangGraph 的 `interrupt()` / `Command(resume=...)` 适合做物理暂停和恢复。

但语义上：

- 审核门是什么。
- 人工门为什么打开。
- 退稿还是通过。
- 哪个输出被接受。

这些必须记录在 TimelineLedger 和 Receipt 中，不能只存在 LangGraph interrupt state。

#### Stream

LangGraph 的 stream 可以作为运行过程推送来源之一。

但前端监控不能直接把 LangGraph stream 当成最终事实。最终事实仍然来自：

```text
TimelineLedger + checkpoint + receipts + runtime state projection
```

#### Durable execution 约束

LangGraph durable execution 对本项目最有价值的是：恢复后不能重复执行不可幂等副作用。

因此本项目必须把以下动作做成幂等：

- stage dispatch
- artifact materialization
- memory candidate creation
- memory commit
- receipt commit
- human gate resume

幂等键不能只靠 `stage_id`，必须包含 `dispatch_event_id` 或 `clock_seq`。

### 2.2 不应该交给 LangGraph 的部分

LangGraph 不负责：

- TaskGraph 编辑器语义。
- 长任务业务拓扑。
- phase / sequence / loop / revision 的业务含义。
- baseline / mutable memory repository 的读写规则。
- artifact record/version 选择。
- 审稿退稿交接内容。
- prompt 投影和模型可见任务说明。
- 写作、研发、测试等领域任务的配置规则。

一句话：

```text
LangGraph 是物理运行引擎。
TaskGraph + TimelineLedger + Resolver 是项目任务工程系统。
```

## 3. 目标架构

### 3.1 运行层分解

```text
TaskGraphDefinition
  保存用户配置的通用任务图

LayeredGraphNormalizer
  把同一张图投影为 execution / timeline / memory / artifact / revision layers

TaskGraphRuntimeSpec
  作为运行时不可变图快照

TaskGraphScheduler
  根据 runtime spec + node statuses + gate policies 做可运行性判断

LangGraphRuntimeKernel
  包装 LangGraph StateGraph、checkpointer、invoke/stream/interrupt/resume

TimelineLedger
  分配 clock_seq，追加语义事件，维护 scope_path

ContextPacketResolvers
  根据 dispatch event 和 graph layers 解析 MemorySnapshot / ArtifactContextPacket / RevisionPacket

StageExecutionRequest
  作为发给节点的唯一执行包

ReceiptCommitter
  把节点结果变成 accepted/rejected/committed receipt
```

### 3.2 两套时间必须区分

#### 物理执行时间

来源：

- LangGraph super-step
- checkpoint created_at
- runtime event offset
- stream chunk time

用途：

- 恢复。
- 重放。
- 监控运行是否卡住。
- 定位系统故障。

#### 语义任务时间

来源：

- TimelineLedger 的 `clock_seq`
- `scope_path`
- `event_type`
- causal refs
- receipt refs

用途：

- 判断当前节点能读哪些记忆。
- 判断当前审核节点读哪一稿。
- 判断返修节点拿哪份原稿和审核意见。
- 判断并发分支如何汇合。
- 判断循环第几轮与当前可见版本。
- 判断人工门控恢复后接在哪个事件之后。

两者关系：

```text
TimelineEvent.checkpoint_ref -> LangGraph checkpoint
LangGraph checkpoint 恢复物理 state
TimelineLedger 恢复语义历史和可见性
```

### 3.3 TimelineLedger 数据模型

新增通用模型：

```text
TimelineLedger
  ledger_id
  coordination_run_id
  root_task_run_id
  graph_id
  current_clock_seq
  events
  authority = task_graph.timeline_ledger
```

```text
TimelineEvent
  event_id
  clock_seq
  run_id
  coordination_run_id
  root_task_run_id
  graph_id
  event_type
  status
  scope_type
  scope_path
  parent_event_id
  causal_event_ids
  node_id
  edge_id
  phase_id
  loop_frame_id
  iteration_index
  revision_cycle_id
  parallel_group_id
  dispatch_id
  request_id
  receipt_id
  payload_ref
  payload
  checkpoint_ref
  created_at
  authority = task_graph.timeline_event
```

关键点：

- `clock_seq` 是全局单调逻辑 clock。
- `scope_path` 表示层级时序空间。
- node/edge/phase/loop/revision 是事件参与者，不是时序本身。
- TimelineLedger 是 append-only，业务修正用新事件表达，不覆盖旧事件。

### 3.4 scope_path 设计

scope 是层级语义空间，不是文件路径。

示例：

```text
run
run/phase.plan
run/phase.design
run/phase.execution
run/phase.execution/loop.iteration[3]
run/phase.execution/revision[2]
run/phase.execution/parallel.reviewers
run/phase.execution/gate.human_review
```

循环次数不确定时：

- 图上只配置 loop frame。
- 运行时每轮追加 `loop_iteration_started / loop_iteration_closed`。
- 每轮获得自己的 scope_path。

并发任务时：

- 每个并发分支都在同一个全局 clock 上追加事件。
- `parallel_group_id` 标识同组。
- join 事件用 causal refs 指向所有已完成分支。

第一阶段需要覆盖以下通用事件：

```text

run_failed

```

事件族必须通用，不能出现写作领域事件名。

## 4. LangGraph 物理底座实施方案

### 4.1 新增 LangGraphRuntimeKernel

新增模块：

```text
backend/orchestration/runtime_loop/langgraph_runtime_kernel.py
```

职责：

- 构建/持有 LangGraph `StateGraph` app。
- 管理 `thread_id`、checkpoint config、invoke/stream/resume。
- 把每次 LangGraph 物理推进返回为统一 `KernelStepResult`。
- 不直接写 TaskGraph 语义事件。

目标接口：

```text
LangGraphRuntimeKernel.invoke(state, thread_id, reason) -> KernelStepResult
LangGraphRuntimeKernel.stream(state, thread_id, reason) -> Iterator[KernelStreamEvent]
LangGraphRuntimeKernel.resume(thread_id, resume_payload) -> KernelStepResult
```

`KernelStepResult`：

```text
state
checkpoint_ref
checkpoint_metadata
physical_events
diagnostics
```

### 4.2 CheckpointBridge

新增或重构：

```text
backend/orchestration/runtime_loop/langgraph_checkpoint_adapter.py
```

目标不是立刻强制替换所有 checkpoint，而是形成稳定边界：

```text
ProjectCheckpointBridge
  put_state()
  get_state()
  get_checkpoint()
  latest_checkpoint_ref()
  to_langgraph_checkpointer() 可选
```

短期：

- 保留当前 JSON adapter，避免大范围破坏。
- 所有 checkpoint_ref 都通过 kernel 返回。

中期：

- 在确认 LangGraph native saver 序列化兼容后，切换到 native checkpointer。
- TimelineEvent 只保存 checkpoint ref，不依赖具体 saver 实现。

### 4.3 LangGraph 不生成语义 clock

约束：

- LangGraph checkpoint id 可以进入 TimelineEvent。
- LangGraph stream event 可以被投影为 monitor event。
- LangGraph interrupt 可以触发 `human_gate_opened`。
- 但 `clock_seq` 只能由 TimelineLedger 分配。

这能避免后续因为 LangGraph 内部 super-step 变化而破坏我们的任务语义。

## 5. 上下文交接设计

### 5.1 交接不是“直接把上个节点输出塞给下个节点”

直接传输出的问题：

- 无法区分候选、通过、退稿、失败。
- 无法知道哪一轮返修。
- 无法在并发汇合时确定谁覆盖谁。
- 无法让审核节点稳定读当前送审稿。
- 无法审计 memory/artifact 何时生效。

正确方式：

```text
上游节点输出
  -> artifact/memory candidates
  -> validation
  -> receipt
  -> handoff packet / context packet
  -> 下游节点读取 packet refs
```

### 5.2 StageExecutionRequest 必须与 Prompt 配套

`StageExecutionRequest` 不应该只携带 raw input。它必须携带执行认知包：

```text
dispatch_context
  dispatch_event_id
  clock_seq
  scope_path
  causal_event_ids
  checkpoint_ref

node_identity
  node_id
  task_ref
  agent_id
  projection_id

resolved_inputs
  explicit_inputs
  memory_snapshot
  artifact_context_packet
  revision_packet
  handoff_packets

runtime_assembly
  prompt/projection refs
  context sections
  output contracts
  acceptance contracts

delivery_contract
  expected_outputs
  artifact_policy
  stream_policy
  receipt_policy
```

Prompt 必须解释这些输入：

- 哪些是只读基准。
- 哪些是当前候选稿。
- 哪些是审核意见。
- 哪些是历史参考。
- 哪些可以写入。
- 哪些必须产出。

不能把开发说明当成 prompt 发给 agent。

### 5.3 HandoffPacket

用于普通上下游交接：

```text
HandoffPacket
  packet_id
  source_event_id
  target_dispatch_event_id
  source_node_id
  target_node_id
  edge_id
  payload_contract_id
  artifact_refs
  memory_refs
  summary
  ack_required
  status
```

### 5.4 ArtifactContextPacket

用于确定性产物读取：

```text
ArtifactContextPacket
  packet_id
  dispatch_event_id
  clock_seq
  scope_path
  edge_ids
  source_node_ids
  source_receipt_ids
  artifact_refs
  expanded_text_by_input_key
  missing_required_artifacts
```

审核节点必须通过这个 packet 读当前候选稿。

### 5.5 RevisionPacket

用于审核返修：

```text
RevisionPacket
  revision_packet_id
  revision_cycle_id
  dispatch_event_id
  target_node_id
  review_node_id
  previous_candidate_artifact_refs
  previous_candidate_receipt_id
  review_result_refs
  review_verdict
  required_changes
  carry_input_keys
  clear_input_keys
```

返修节点必须通过这个 packet 拿到：

- 被审原稿。
- 审核意见。
- 当前第几轮返修。
- 哪些部分必须保留。
- 哪些输入必须清空。

### 5.6 MemorySnapshot

用于确定性记忆读取：

```text
MemorySnapshot
  snapshot_id
  dispatch_event_id
  clock_seq
  scope_path
  read_edge_ids
  repository_refs
  resolved_records
  resolved_versions
  missing_required_records
  read_receipt_id
```

读取规则：

```text
memory_edge
  + resource node
  + dispatch_context
  + selector
  + version_selector
  -> MemorySnapshot
```

RAG 只能作为辅助搜索，不作为长任务主记忆读取路径。

## 6. 写入与 Receipt

节点输出必须经过 receipt，不能直接污染后续上下文。

```text
raw node output
  -> artifact candidates
  -> memory write candidates
  -> acceptance validation
  -> execution receipt
  -> committed versions become visible to later clock/scope
```

ExecutionReceipt：

```text
receipt_id
request_id
dispatch_event_id
clock_seq
scope_path
stage_id
node_id
status
accepted
produced_artifact_refs
memory_write_candidate_refs
memory_commit_refs
validation_result
effective_from_clock_seq
effective_scope_path
```

规则：

- rejected/failed output 不进入 committed visibility。
- revision feedback 可以生成 `review_decision_recorded` 和 `revision_packet_created`。
- memory commit 必须有写入权限和 receipt。
- artifact commit 必须关联 source receipt。

## 9. 测试矩阵

### LangGraph 底座

- invoke 后 checkpoint_ref 存在。
- resume 使用同一 thread_id。
- checkpoint 恢复后 state 不丢失 stage request。
- kernel 层能返回统一结果，不泄漏实现细节。

### TimelineLedger

- clock_seq 单调递增。
- run_started / scheduler_evaluated / node_dispatch_requested / checkpoint_linked 顺序正确。
- 事件包含 scope_path。
- checkpoint_ref 能追溯到 checkpoint adapter。

### Dispatch Context

- StageExecutionRequest 包含 dispatch_event_id / clock_seq / scope_path。
- idempotency key 区分同节点不同 retry。
- idempotency key 区分 revision cycle。
- idempotency key 区分 loop iteration。

### ArtifactContextPacket

- 审核节点读取当前候选 artifact。
- 第 2 稿送审时不能读第 1 稿。
- missing required artifact 时 block。
- text expansion 受 max_chars 控制。

### RevisionPacket

- 返修节点拿到被审原稿。
- 返修节点拿到审核意见。
- review verdict 和 required changes 明确。
- clear/carry input keys 生效。

### MemorySnapshot

- baseline repository 只读。
- mutable repository 只有授权节点可写。
- 当前节点不能读未 receipt commit 的本节点输出。
- 下一 clock/scope 能读到已生效版本。
- read receipt 记录 snapshot。

### 通用性

- 通用 runtime 不出现写作领域特判。
- 通用 editor 不出现写作专用字段。
- 写作长任务通过配置实现。

## 10. 禁止事项

- 禁止把节点本身定义成时序。
- 禁止 Scheduler 分配语义 clock。
- 禁止用 latest 文件推断当前稿件。
- 禁止把返修上下文塞进全局 pending_inputs 当主路径。
- 禁止用 RAG 作为主记忆仓库读取。
- 禁止把 prompt 正文堆进 TaskGraph 节点。
- 禁止为了写作任务改通用编辑器。
- 禁止绕过测试或伪造结果。

## 11. 最小可交付闭环

1. LangGraphRuntimeKernel 接管 StateGraph invoke/checkpoint 边界。

3. StageExecutionRequest 携带 dispatch context。

5. ArtifactContextPacket 能让审核节点读当前候选稿。

## 二、核心结论

任务系统不是一个任务列表，也不是一个 workflow 编辑页。

任务系统是平台的任务调度中枢。它负责把用户目标转化为可执行的任务装配请求，并决定任务应采用哪个 Agent、是否允许生产工作子 Agent、是否进入多 Agent 协调模式，以及需要附带哪些任务侧约束。

任务系统的核心问题是：

`这是什么任务，为什么这样登记，交给谁执行，允许怎样协作，最终形成什么运行前装配`

其中最关键的 Agent 调用原则是：

`任务系统预先定义可调用结构，主 Agent 或 coordinator 在运行时按需要触发具体子 Agent`

也就是说：

- 任务系统不应把所有子 Agent 调用提前硬排成固定队列
- 主 Agent 也不应越过任务系统临场随意创建协作规则
- 正确模型是“任务系统给边界，主 Agent 做运行时调度”

## 三、设计原则来源

### 3.1 Agent 系统原则

设计原则中的 Agent 系统强调：

- 主 Agent 可以按需生成子 Agent
- 子 Agent 拥有独立上下文窗口和对话循环
- 子 Agent 完成后只返回精炼结果
- 子 Agent 用于降低主上下文污染、支持专业化分工、支持并行执行
- 子 Agent 的 context 应默认隔离，需要共享的内容显式 opt-in

因此，子 Agent 不应被看成“主 Agent 的内部函数”。它是一个独立执行体，只是由主 Agent 或 coordinator 在任务过程中触发。

### 3.2 任务系统原则

设计原则中的任务系统强调：

- 任务系统统一管理异步工作
- 任务需要生命周期状态
- 任务需要可 kill、可追踪、可通知
- 多 Agent 协作需要任务级状态管理
- Coordinator 通过 AgentTool、SendMessage、TaskStopTool 管理 Worker 团队
- 子 Agent 完成通知和结果回传必须成为任务生命周期的一部分

因此，子 Agent 不能只是一次普通函数调用。它必须变成可登记、可追踪、可恢复、可终止的运行对象。

### 3.3 本项目系统边界原则

总设计书已经明确：

- 任务系统负责任务登记、任务分类、任务分配、Agent 采用、工作子 Agent 生产、多 Agent 协调任务拓扑和协议
- 编排系统负责单 Agent 的身体装配、prompt 结构、工具边界、资源边界、memory scope 和 output boundary
- runtime 负责实际执行、循环、事件、checkpoint 和恢复

因此，多 Agent 协调的“谁参与、怎样组织、怎样交接”属于任务系统；每个 Agent 自己“内部如何运行”属于编排系统；执行过程中的状态推进属于 runtime。

## 四、任务系统的正式职责

任务系统负责：

- 识别任务
- 登记任务
- 分类任务
- 管理任务契约
- 管理通用任务入口
- 管理特定任务登记
- 管理任务 workflow
- 管理任务 projection 绑定
- 管理任务 flow contract 绑定
- 管理任务记忆请求 profile
- 管理任务能力需求
- 管理 Agent 采用计划
- 必要时允许生产工作子 Agent
- 管理多 Agent 协调任务
- 管理协调拓扑
- 管理任务级通信协议
- 管理 handoff 与结果汇总规则
- 输出 `TaskExecutionAssembly`

任务系统不负责：

- 直接拼 Agent 的 system prompt
- 直接决定工具权限细节
- 直接读取记忆内容
- 直接调用底层能力实现
- 直接执行 runtime loop
- 直接生成最终回答

### 5.4 TaskAgentAdoptionPlan

Agent 采用计划。

它回答：

`这个任务到底由谁执行，执行主体是复用还是生产`

- `adopt_existing`
- `adopt_with_projection`
- `spawn_worker_agent`

其中 `coordination_candidate` 表示任务系统允许进入多 Agent 协调，但具体是否触发由 runtime 中的 coordinator 决定。

### 5.5 TaskMemoryRequestProfile

任务记忆请求配置。

它回答：

`执行这个任务前应该向记忆系统申请哪些记忆层`

- `requested_memory_layers`
- `requested_topics`
- `memory_priority`
- `writeback_policy`
- `allow_long_term_memory`

### 5.6 TaskProjectionBinding

它回答：

`这个任务允许使用哪些 projection，默认怎么选`

projection 是执行姿态补充，不是任务方法，不是 workflow，也不是 Agent 本体。

### 5.7 TaskFlowContractBinding

任务侧 flow contract 绑定。

它回答：

`这个任务在 runtime 中按什么状态机契约执行`

flow contract 是任务对 runtime 的执行约束，不是 workflow 文本，也不是编排系统的 prompt 结构。

`多个 Agent 在任务中如何组织`

- `nodes`
- `edges`
- `handoff_rules`
- `join_policy`
- `failure_policy`

### 5.10 TaskCommunicationProtocol

它回答：

`参与任务的 Agent 之间传什么、怎么传、什么时候传`

## 六、任务类型与执行模式

### 6.1 通用任务

通用任务是平台默认承接模式。

特点：

- 默认由 `agent:0` 主 Agent 承接
- 允许模型承担一定理解
- 允许命中特定任务后转入特定任务链路
- 通常不要求前端用户手动选择

通用任务不是“没有任务”，而是平台默认任务入口。

### 6.2 特定任务

特定任务是预先登记、需要稳定消费和稳定执行的任务。

特点：

- 必须登记
- 必须有任务契约
- 必须绑定 workflow
- 必须有 Agent 采用方案
- 必须可生成稳定 `TaskExecutionAssembly`
- 可以绑定 projection、flow contract、memory request profile

特定任务不能只靠主 Agent 临场理解。

### 6.3 协调任务

协调任务是多 Agent 共同完成的任务组织。

特点：

- coordinator、participants、topology、protocol 必须由任务系统提供
- 运行中可以按需 spawn 或 continue 子 Agent
- 每个子 Agent 必须有自己的运行状态和结果回传
- 最终结果由 coordinator 或 merge policy 汇总

协调任务不是 workflow 的扩写，也不是编排系统的变体。

## 七、主 Agent 与子 Agent 的关系

### 7.1 主 Agent 和子 Agent 没有本体差异

Agent 的本体都是标准执行接口。

主 Agent 只是默认承接主会话入口的 Agent。子 Agent 只是被任务或 coordinator 调用的 Agent。

差异来自：

- 入口不同
- 任务角色不同
- runtime lane 不同
- context scope 不同
- output boundary 不同
- 是否被 coordinator 管理

### 7.2 主 Agent 的职责

主 Agent 负责：

- 承接通用任务
- 根据任务系统装配进入执行
- 在允许范围内调用子 Agent
- 对子 Agent 结果进行理解和整合
- 在协调模式下作为 coordinator 或 final merger
- 输出最终回答

主 Agent 不负责：

- 临场发明任务系统没有授权的协作拓扑
- 临场绕过任务系统创建无登记工作子 Agent
- 直接修改子 Agent 的运行体边界
- 越过记忆系统直接共享长期记忆

### 7.3 子 Agent 的职责

子 Agent 负责：

- 在独立上下文中执行明确子任务
- 按自己的 runtime profile 使用工具和能力
- 只返回任务相关的精炼结果
- 不污染主 Agent 上下文
- 不默认继承父 Agent 的可变状态
- 不越权再次创建子 Agent，除非任务系统和运行策略显式允许

### 7.4 子 Agent 的调用原则

子 Agent 调用应采用“两段式决策”。

任务系统提前决定：

- 是否允许子 Agent
- 允许采用哪些 Agent 类别
- 是否允许 spawn worker
- worker blueprint 是什么
- 协调模式下 coordinator 和 participants 是谁
- 可以使用什么 topology
- 可以使用什么 communication protocol
- handoff 和 failure policy 是什么

主 Agent 或 coordinator 运行时决定：

- 当前是否真的需要子 Agent
- 用已有 Agent 还是新建 worker
- 用 fork 还是 fresh
- 并行调用还是串行调用
- 是继续已有子 Agent，还是 spawn fresh
- 何时发送 follow-up
- 何时 stop 或 kill
- 如何汇总子 Agent 返回结果

这能同时避免两种错误：

- 把子 Agent 固定成开局全量启动的僵硬队列
- 让主 Agent 临场无限制地自由生产执行体

## 八、子 Agent 调用模式

### 8.1 adopt existing

采用已有 Agent。

适用于：

- 主 Agent 承接通用任务
- 系统管理 Agent 承接系统专项任务
- 已有稳定工作子 Agent 承接重复专项任务

执行方式：

1. 任务系统在 `TaskAgentAdoptionPlan` 中给出 `default_agent_id`
2. 编排系统为该 Agent 生成 `AgentRuntimeSpec`
3. runtime 创建一个 `AgentRun`

### 8.2 adopt with projection

采用已有 Agent，并挂载任务侧 projection。

适用于：

- 执行主体稳定
- 任务姿态需要临时改变
- 任务希望保留 Agent 本体，但改变表达风格、注意焦点或角色姿态

执行方式：

1. 任务系统选择 Agent
2. 任务系统给出 projection binding
3. 灵魂系统提供 projection 素材
4. 编排系统把 projection 编入该 Agent 的 prompt structure

### 8.3 spawn worker agent

生产新的工作子 Agent。

适用于：

- 需要长期稳定消费的专项任务
- 需要隔离任务记忆
- 需要固定能力边界
- 需要可复用的工作单元

执行方式：

1. 任务系统给出 `allow_worker_agent_spawn=true`
2. 任务系统给出 `worker_agent_blueprint_id`
3. runtime 或 Agent 工厂创建 `WorkerAgentSpawnRequest`
4. AgentRegistry 注册新 Agent
5. AgentRuntimeRegistry 注册对应 runtime profile
6. 编排系统为新 Agent 生成运行体
7. runtime 创建 `AgentRun`

### 8.4 fork subagent

Fork 子 Agent 继承父级上下文前缀，但执行一个独立指令。

适用于：

- 当前任务需要并行探索
- 需要复用父级 prompt cache
- 子任务强依赖当前对话上下文
- 不希望重传全部上下文

约束：

- Fork 子 Agent 必须知道自己已经是 fork
- Fork 子 Agent 默认不应再次 fork
- Fork 子 Agent 仍应有独立运行状态
- Fork 子 Agent 输出必须通过任务结果回传，而不是直接污染主线程

### 8.5 fresh subagent

Fresh 子 Agent 从干净上下文启动。

适用于：

- 子任务边界清晰
- 需要避免父上下文污染
- 专业 Agent 已经有完整 system prompt
- 需要独立验证、审查、研究

约束：

- Fresh 子 Agent 只拿任务所需的显式输入
- 父级上下文必须通过 handle、summary、ref 方式传递
- 不应默认共享父级可变状态

### 8.6 continue existing subagent

继续已有子 Agent。

适用于：

- 子 Agent 已经持有该子任务上下文
- coordinator 需要追加指令
- 子任务还未终结

约束：

- 消息进入 pending queue
- 只能在工具轮次或安全边界消耗
- 追加指令必须带 parent task / coordinator refs

## 十、任务系统与编排系统交接

任务系统输出：

- `TaskExecutionAssembly`
- `TaskAgentAdoptionPlan`
- `TaskProjectionBinding`
- `TaskFlowContractBinding`
- `TaskMemoryRequestProfile`
- `CoordinationTaskDefinition`
- `TopologyTemplate`
- `TaskCommunicationProtocol`

编排系统消费这些对象，并输出：

- `AgentBodyProfile`
- `PromptStructureProfile`
- `MemoryScopeProfile`
- `RuntimeLaneProfile`
- `OutputBoundaryProfile`
- `TaskBodyOrchestration`
- `AgentRuntimeSpec`

关键边界：

- 任务系统决定“给谁、做什么、允许怎样协作”
- 编排系统决定“这个 Agent 的身体如何装起来”
- 编排系统不重新决定任务拓扑
- 编排系统不重新选择 coordinator / participants

## 十一、任务系统与 runtime 交接

runtime 应消费：

- `TaskExecutionAssembly`
- `AgentRuntimeSpec`
- `MemoryBundle` 或 `MemoryRuntimeView`
- `RuntimeContextPackage`
- `CoordinationTaskDefinition`
- `TopologyTemplate`
- `TaskCommunicationProtocol`

runtime 负责：

- 创建 `TaskRun`
- 创建 `AgentRun`
- 创建 `CoordinationRun`
- 推进 loop iteration
- 记录 event log
- 写 checkpoint
- 执行 tool/model action
- 管理 pending message
- 管理 kill/stop
- 聚合结果
- 失败恢复

runtime 不负责：

- 临场重新定义任务
- 临场重新定义 topology
- 临场绕过任务系统创建未授权 Agent
- 临场改变 Agent 的工具边界

## 十二、任务执行主流程

### 12.1 单 Agent 任务

流程：

1. 用户消息进入 session
2. 理解层生成 query understanding
3. 任务系统命中通用任务或特定任务
4. 任务系统输出 `TaskExecutionAssembly`
5. 记忆系统按 `TaskMemoryRequestProfile` 输出 memory runtime view
6. 编排系统按 selected agent 输出 `AgentRuntimeSpec`
7. runtime 创建 `TaskRun`
8. runtime 创建主 `AgentRun`
9. runtime 进入 model/tool loop
10. output boundary 生成最终结果

### 12.2 按需子 Agent 任务

流程：

1. 主 Agent 进入任务执行
2. 主 Agent 判断需要拆出子任务
3. runtime 检查 `TaskAgentAdoptionPlan`
4. 若允许，创建 `AgentRun`
5. 编排系统为子 Agent 生成运行体
6. 子 Agent 独立执行
7. 子 Agent 返回 `AgentRunResult`
8. 主 Agent 消费结果摘要和 refs
9. 主 Agent 继续当前任务

### 12.3 多 Agent 协调任务

流程：

1. 任务系统命中 coordination task
2. 任务系统输出 coordination refs
3. runtime 创建 `CoordinationRun`
4. runtime 创建 coordinator `AgentRun`
5. coordinator 根据 topology 触发 participant `AgentRun`
6. participant 通过 handoff envelope 回传结果
7. coordinator 根据 protocol 和 merge policy 汇总
8. runtime 写入 `CoordinationMergeResult`
9. output boundary 输出最终结果

## 十三、前端任务系统深度框架

任务系统前端应放在工作台中，但内部结构必须围绕任务管理决策链，而不是后端对象表。

### 13.1 一级结构

建议一级结构：

- `任务登记`
- `协调任务`
- `Agent 资源`

不建议继续使用：

- `编排管理`

因为多 Agent 协调属于任务系统，不属于编排系统。

### 13.2 任务登记页

默认页应是任务登记，而不是 Agent 管理。

推荐三栏：

- 左栏：通用任务 / 特定任务列表
- 中栏：任务定义、执行采用、方法与边界、安全与治理
- 右栏：任务执行装配预览

右栏必须预览：

- selected agent
- workflow
- projection
- memory request
- flow contract
- adoption plan
- input/output contract
- safety envelope
- runtime lane
- single agent 或 coordination candidate

### 13.3 协调任务页

协调任务页应围绕以下问题组织：

- 谁是 coordinator
- 谁是 participants
- 用什么 topology
- 用什么 protocol
- handoff 怎么走
- failure 怎么处理
- output 怎么 merge

推荐三栏：

- 左栏：协调任务列表
- 中栏：协调定义、参与角色、拓扑模板、通信与交接、结果汇总
- 右栏：协调运行预览

### 13.4 Agent 资源页

Agent 资源页不是 Agent 大总览，而是任务系统的执行主体库。

它应展示：

- Agent 类别
- interface target
- task scope
- runtime profile 状态
- default projection / soul
- 可承接任务模式
- 是否适合 coordinator
- 是否适合 participant
- 是否可作为 worker blueprint

## 十七、验证标准

任务系统深度重构完成后，应满足：

- 通用任务能生成完整 `TaskExecutionAssembly`
- 特定任务能生成完整 `TaskExecutionAssembly`
- 前端能预览装配结果
- AgentRegistry 是唯一 Agent 管理入口
- TaskAgentAdoptionPlan 被 runtime 消费
- 单 Agent 任务创建 `AgentRun`
- 子 Agent 调用创建子 `AgentRun`
- 协调任务创建 `CoordinationRun`
- participant 结果通过 handoff envelope 回传
- runtime event log 能还原任务、Agent、handoff、结果
- output boundary 只消费正式结果，不消费零散执行碎片

## 十八、反模式

后续实现中应避免：

- 让 workflow 变成独立页面中心
- 让多 Agent 协调被叫成编排管理
- 让主 Agent 临场无限制 spawn 子 Agent
- 把所有子 Agent 开局全量启动
- 让子 Agent 默认共享父级可变上下文
- 让任务系统直接拼 prompt
- 让 runtime 直接吃散落字段
- 继续维护旧 `main/explorer/worker` 与新 AgentRegistry 双轨
- 只在前端展示 coordination，却不接 runtime
- 把 MCP 能力 endpoint 误当成正式 Agent，或把正式 Agent 误当成 MCP endpoint

## 十九、最终结论

任务系统的深层价值不是“管理任务卡片”，而是把平台从单 Agent 对话壳推进为真正的任务驱动 Agent 平台。

它必须同时解决三件事：

- 任务如何被稳定定义
- Agent 如何被正式采用
- 多 Agent 如何被可控协调

子 Agent 的正确位置也因此变得清晰：

- 它不是预先硬排的固定步骤
- 它不是主 Agent 临场随便叫出的临时线程
- 它是任务系统授权、编排系统装配、runtime 管理、主 Agent 或 coordinator 按需触发的正式执行体

只有把这条链路打通，平台才能从“主 Agent 很强”进化为“任务系统能稳定调度一组 Agent 完成任务”。

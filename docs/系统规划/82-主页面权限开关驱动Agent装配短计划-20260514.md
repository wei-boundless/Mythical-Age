# 82-主页面权限开关驱动Agent装配短计划-20260514

## 1. 背景

主页面当前有三个开关：

- 启用知识库
- 启用本地权限
- 启用联网权限

前端已经把这些开关作为 `search_policy` 发给后端，后端 API 也已经接收到了 `QueryRequest.search_policy`。

但当前断点在运行时装配层：`QueryRuntime.astream()` 没有把 `search_policy` 继续传入 `TaskRunLoop`，`TaskRunLoop` 也没有把它用于主 Agent 工具装配、子 Agent 委派装配和模板 fallback 约束。

所以现在的开关更接近“请求偏好”，还不是严格的“本轮权限边界”。

## 2. 核心判断

权限开关不应该只作为 prompt 提示，也不应该靠主 Agent 自觉遵守。

正确结构是：

**前端开关决定本轮可装载能力，模型只在已授权能力集合中进行理解和选择。**

这符合当前 Agent 系统的设计方向：

- 大模型负责理解任务、选择合适能力、综合证据。
- 系统负责授权、装配、隔离和审计。
- 主 Agent 和子 Agent 在装配上没有本质区别，区别只是主 Agent 允许委派。

## 3. 权限语义

### 3.1 知识库权限

当 `rag` 关闭：

- 不允许主 Agent 使用 RAG 检索能力。
- 不允许主 Agent 委派 `agent:rag_analyst`。
- 不允许模板 fallback 到 `op.mcp_retrieval`。

当 `rag` 开启：

- 主 Agent 可以根据任务语义自然选择 RAG。
- 可委派 `agent:rag_analyst`。

### 3.2 本地权限

当 `local_files` 关闭：

- 不允许主 Agent 使用本地文件读取、PDF、表格、结构化数据、本地 MCP。
- 不允许主 Agent 委派 `agent:pdf_reader` 和 `agent:table_analyst`。
- 不允许模板 fallback 到 `op.mcp_pdf`、`op.mcp_structured_data` 或本地文件读取类操作。

当 `local_files` 开启：

- 主 Agent 可以根据任务语义自然选择本地文件、PDF、表格和本地 MCP 能力。
- 可委派 PDF / 表格类专业子 Agent。

### 3.3 联网权限

当 `web` 关闭：

- 不允许主 Agent 使用网页搜索、URL 抓取、实时外部数据能力。
- 不允许主 Agent 委派 `agent:web_researcher`。
- 不允许模板 fallback 到 `op.web_search` 或外部网络类操作。

当 `web` 开启：

- 主 Agent 可以根据任务语义自然选择联网搜索。
- 可委派 `agent:web_researcher`。

## 4. 实施范围

### 4.1 请求链路打通

修改点：

- `backend/query/runtime.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`

目标：

- `QueryRuntime.astream()` 将 `request.search_policy` 传入 `TaskRunLoop.run_single_agent_stream()`。
- `TaskRunLoop.run_single_agent_stream()` 增加 `search_policy` 入参。
- 本轮 trace / runtime event 记录原始 `search_policy` 和归一化后的 `allowed_sources`。

### 4.2 装配层过滤

修改点：

- `backend/orchestration/runtime_loop/task_run_loop.py`
- 必要时补充 `backend/capability_system/search_policy.py`

目标：

- 在 `_tool_instances_for_resource_policy()` 后增加来源权限过滤。
- 已授权工具仍需同时满足 resource policy 和 search policy。
- 不允许仅靠 prompt 告诉模型“不要用”，而是模型根本看不到未授权工具。

### 4.3 子 Agent 委派过滤

修改点：

- `backend/orchestration/runtime_loop/agent_delegation_executor.py`
- 或主 Agent 构造 delegation catalog 的上游入口。

目标：

- 根据 `search_policy` 过滤可委派子 Agent。
- `rag=false` 时禁用 `agent:rag_analyst`。
- `local_files=false` 时禁用 `agent:pdf_reader`、`agent:table_analyst`。
- `web=false` 时禁用 `agent:web_researcher`。
- 过滤结果写入 trace，便于前端和长测判断。

### 4.4 模板 fallback 约束

修改点：

- `backend/tasks/template_registry.py` 的调用消费侧
- `backend/orchestration/runtime_loop/task_run_loop.py` 模板执行判断处

目标：

- 如果模板首选 delegate 被权限关闭，不自动绕到同类直接工具。
- 如果模板 fallback 操作也被权限关闭，本轮应进入 evidence unavailable / permission blocked，而不是偷偷改路。

### 4.5 前端语义校准

修改点：

- `frontend/src/components/chat/ChatInput.tsx`
- `frontend/src/lib/store/runtime.ts`

目标：

- 保持当前 UI 结构。
- 文案从“搜索权限”校准为“本轮能力权限”。
- RAG 开关仍可同步 `/config/rag-mode`，但本轮严格权限以 `search_policy` 为准。
- 本地和联网开关不直接改全局 `permission_mode`，避免把一次对话选择误写成全局运行模式。

## 5. 测试计划

### 5.1 单元回归

新增或补充测试：

- `rag=false` 时，RAG 工具和 `agent:rag_analyst` 均不可用。
- `local_files=false` 时，PDF / 表格工具和对应子 Agent 均不可用。
- `web=false` 时，网页搜索工具和 `agent:web_researcher` 均不可用。
- 三个开关全开时，原有自然委派能力不退化。

### 5.2 自然场景测试

覆盖场景：

- 用户要求查最新资料，但联网关闭：主 Agent 应说明需要联网权限，不能调用 web 子 Agent。
- 用户要求分析 PDF，但本地权限关闭：主 Agent 应说明需要本地权限，不能调用 PDF 子 Agent。
- 用户要求查知识库，但 RAG 关闭：主 Agent 应说明需要知识库权限，不能调用 RAG 子 Agent。
- 用户请求普通写作 / 推理：即使权限关闭，也应正常回答，不误报权限不足。

### 5.3 长测关注点

长测不只看是否报错，还要检查语义：

- 是否真的没有调用被关闭的子 Agent。
- 是否没有绕到同类直接工具。
- 是否没有把权限不足误写成“没找到资料”。
- 是否能在权限打开后自然恢复调用。

## 6. 验收标准

- 前端开关状态能进入后端运行时 trace。
- 未授权来源对应的主 Agent 工具不可见。
- 未授权来源对应的子 Agent 不可委派。
- 模板 fallback 不绕过权限边界。
- 开关打开后自然调用能力不受影响。
- 测试能证明权限关闭、权限打开、普通无权限任务三类场景都正常。

## 7. 风险与边界

### 7.1 风险

- 如果只过滤工具、不过滤子 Agent，会出现委派绕权。
- 如果只过滤子 Agent、不过滤直接工具，会出现主 Agent 自己绕权。
- 如果直接改全局 `permission_mode`，会把本轮开关污染成全局运行配置。

### 7.2 边界

本计划只处理主页面三类来源权限：

- RAG
- 本地资源
- 联网资源

不处理 shell、写文件、Python 执行等高风险操作。这些仍由现有 `permission_mode`、`OperationGate` 和资源策略控制。

## 8. 推荐执行顺序

1. 打通 `search_policy` 到 `TaskRunLoop`。
2. 增加统一来源权限归一化函数。
3. 过滤主 Agent 工具装配。
4. 过滤子 Agent 委派能力。
5. 约束模板 fallback。
6. 补前端文案和 trace 展示字段。
7. 跑权限回归和自然场景测试。

# 139-TaskGraph运行监控常驻浮窗与会话解耦实施计划

日期：2026-05-17

## 1. 问题定义

当前 TaskGraph 监控已经能从后端读取 `task_graph.run_monitor`，也能由 `runtime_monitor` 产出 `monitor_decision`。但前端呈现还有两个结构性问题：

1. 运行交互浮窗挂在发布页内部，只有进入特定页面、绑定了该页局部 `taskRunId` 后才可见。
2. 监控状态依附当前聊天会话的 live monitor。用户切换会话后，`taskGraphLiveMonitor` 和 `taskGraphRunMonitor` 会被清空，导致正在运行的 TaskGraph 监控消失。

这不符合 TaskGraph 长任务的运行模型。长任务是运行实例，不是聊天页附属物；监控应该跟随 `task_run_id`，而不是跟随 `currentSessionId`。

## 2. 目标设计

新增前端运行监控绑定：

```ts
TaskGraphMonitorBinding {
  task_run_id: string
  coordination_run_id?: string
  graph_id?: string
  session_id?: string
  project_id?: string
  title?: string
  bound_at: number
}
```

原则：

- TaskGraph run 一旦创建或手动绑定，就写入全局 monitor binding。
- 浮窗挂在应用壳层，跨工作区、跨会话常驻。
- 轮询按 `task_run_id` 调用 `/task-runs/{task_run_id}/task-graph-monitor`，不再依赖当前聊天会话。
- 当前聊天会话切换只影响消息列表，不清空 TaskGraph 运行监控。
- `runtime_monitor` 决策仍由用户主动评估或异常态触发，不把普通轮询变成高频决策写入。

## 3. 浮窗行为

常态：

- 折叠态始终可见。
- 显示当前节点、运行状态、流式字数、最近更新时间。
- 允许用户打开窗口主动询问/记录操作意见，并执行一次监测评估。

异常态：

- 当 `monitorDecision.action !== "no_action"` 时自动展开。
- 统一展示 `run_interaction_request`。
- 通过现有编排控制接口续跑、暂停、重试或确认。

时间显示：

- 优先使用 `monitor.streaming.latest_chunk_at`。
- 其次使用 `monitor.runtime.updated_at`。
- 再其次使用 `monitor.timeline.updated_at` 或 `monitor.supervision.latest_event_at`。
- 不再只看旧 decision 的 `created_at`。

## 4. 实施步骤

### 阶段一：Store 增加运行级绑定

文件：

- `frontend/src/lib/store/types.ts`
- `frontend/src/lib/store/core.ts`
- `frontend/src/lib/store/runtime.ts`

内容：

- 新增 `taskGraphMonitorBinding`。
- 新增 action：
  - `bindTaskGraphMonitorRun(binding)`
  - `clearTaskGraphMonitorRun()`
  - `evaluateBoundTaskGraphMonitor()`
  - `submitTaskGraphMonitorDecision(...)`
- 新增按 `task_run_id` 的独立轮询，不再用当前会话作为唯一监控入口。

### 阶段二：发布页只负责启动和绑定

文件：

- `frontend/src/components/workspace/views/task-system/TaskGraphPublishRunPage.tsx`

内容：

- 创建运行成功后调用 `bindTaskGraphMonitorRun`。
- 手动输入 TaskRun ID 后也可绑定监控。
- 删除发布页内部的浮窗挂载，避免页面切换后丢失。

### 阶段三：应用壳层常驻浮窗

文件：

- `frontend/src/app/page.tsx`
- `frontend/src/components/workspace/views/task-system/TaskGraphRunInteractionDock.tsx`

内容：

- 在 `Workspace` 中挂载浮窗。
- 浮窗读取全局 monitor binding / monitor snapshot / decision。
- 折叠态显示关键运行状态，不再只是“运行交互”按钮。
- 打开态显示当前节点、实时正文流、最近更新、监测决策和控制按钮。

### 阶段四：样式与验证

文件：

- `frontend/src/app/globals.css`

内容：

- 增加常态 launcher 的运行摘要布局。
- 增加流式预览、状态条、更新时间样式。

验证：

- `npm run build`
- 创建/绑定 TaskRun 后浮窗常驻。
- 切换聊天会话后浮窗不消失。
- 当前 run 的“最近更新”跟随流式 chunk 更新时间。

# Docker 沙盒执行后端重构计划书

日期：2026-05-26

## 1. 重构结论

平台后续执行类沙盒只采用 Docker Sandboxes (`sbx`)。

旧 `LocalOverlaySandboxBackend` 不再承担“执行隔离”的成熟职责，只保留为本地文件 overlay、copy-on-read/copy-on-write、调试和无 Docker Sandboxes 环境下的低风险文件工具后端。真正会运行命令、脚本、测试、构建的工具必须进入 Docker Sandboxes 后端。

## 2. 权威边界

目标链路：

```text
TaskEnvironment
-> FileAccessTable / ToolCapabilityTable
-> PermissionContext / OperationGate
-> RuntimeStartPacket
-> ToolRuntimeExecutor
-> DockerSandboxBackend
-> ToolResultEnvelope / ExecutionReceipt
```

沙盒后端只负责：

- 准备隔离执行环境；
- 绑定只读项目目录和可写 sandbox workspace；
- 执行命令或 Python 代码；
- 施加 CPU、内存、进程数、网络、超时限制；
- 返回 stdout、stderr、exit_code、执行后端、镜像、挂载、限制等 receipt。

沙盒后端不负责：

- 判断用户任务属于哪个环境；
- 判断 agent 是否允许调用工具；
- 判断某个文件是否允许读写；
- 自动放大工具权限；
- 在 Docker 不可用时偷偷回退到 host shell。

## 3. Docker Sandboxes 默认安全策略

优先使用 Docker Sandboxes：

- CLI：`sbx.exe`
- daemon：`sbx daemon start`
- 创建：`sbx create shell <sandbox_workspace> <project_workspace>:ro`
- 执行：`sbx exec <sandbox_name> bash -lc <command>` 或 `sbx exec <sandbox_name> python -c <code>`
- sandbox workspace 可写；
- project workspace 只读；
- CPU / 内存限制由 `sbx create --cpus --memory` 设置；
- 默认内存不低于 Docker Sandboxes 当前最小值 `1g`；
- 默认不强制指定 template，由 `sbx` 使用 agent 默认模板；需要固定镜像时由系统配置 `sbx.template`，不由 agent 自行决定。
- 网络由 `sbx policy set-default deny-all|balanced|allow-all` 和 profile 控制；
- 本地任务环境默认不由 agent 自动打开网络。

网络只能由任务环境 / 具体任务授权后打开。`sbx` 后端本身不根据 agent 意图自动开网络。

## 4. 实施范围

本轮实施：

1. 新增 `runtime.tool_runtime.docker_sandbox_backend`，内部只支持 `sbx`。
2. 为 `ToolUseContext` 增加 `sandbox_policy`，让工具执行层读取已决策的系统策略。
3. `NativeTerminalTool` 在 `sandbox_policy.execution_backend == docker_sandboxes` 或 `sbx` 时走 `sbx` 后端。
4. 新增 `NativePythonReplTool`，让 `python_repl` 也能走同一 `sbx` 执行后端。
5. `prepare_runtime_sandbox_policy` 增加 `sbx` 执行后端默认字段，但不把权限判断下放给 `sbx`。
6. 增加 fake-runner 单元测试，验证 `sbx create`、`sbx exec`、只读项目挂载、receipt 和工具接入。

本轮不做：

- 不接入 gVisor/Kata；
- 不安装 Docker Desktop；已安装 Docker Sandboxes 时，可以启动 `sbx daemon` 并在登录后做真实集成验证；
- 不把全部文件工具迁移到 Docker；
- 不清理旧 overlay 文件链路；
- 不做 artifact/memory 的完整迁移。

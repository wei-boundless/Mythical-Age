## 0. 核心结论

当前项目不是缺资源，而是资源已经分布在任务系统、编排系统、灵魂投影、能力系统、记忆系统、运行态索引、健康测试和前端工作台里，但还缺少一个统一的资源消费秩序。

必须把专业模式做成唯一的长任务运行母版，再把标准模式和角色模式作为母版的弱化配置，而不是继续维护多套意图分类、driver 分支和 prompt 分支。

目标架构是：

```text
现有资源库
  -> 资源索引与权威层
  -> 开放意图帧
  -> 执行义务抽取
  -> 策略原型匹配
  -> 运行模式策略
  -> 专业长任务状态机
  -> 工具观察账本
  -> 交付验证
  -> 持久化恢复
  -> 输出边界
```

其中：

- 资源库负责提供能力、模板、投影、图、记忆、运行态和测试材料。
- 意图理解负责开放地理解用户本轮目标，不负责把用户塞进闭集任务域。
- 执行义务负责把用户明确要求转成硬约束，例如必须读、必须写、必须验证、必须交付文件、必须等待结果。
- 策略原型只负责提供默认专业步骤和 prompt 风格，不再决定能不能写入或验证。
- 状态机负责长任务时序、恢复、阻塞和收口。
- 投影只影响表达和协作姿态，不能覆盖执行义务。
- TaskDomain 只作为目录和管理资源，不再作为运行时意图大脑。

本设计书要指导后续大改。实施时不保留旧 `simple / managed / autonomy_mode / autonomous_task_run` 链路作为兼容分支；切换完成后应删除旧路径、旧测试和旧配置残留。

## 2. 当前结构性缺陷

### 2.1 `task_goal_type` 仍然责任过载

`backend/tasks/semantic_task_contracts.py` 中，`task_goal_type` 目前会影响：

- domain
- professional profile
- deliverables
- reasoning steps
- required actions
- forbidden actions
- material policy
- validation schema

这意味着一个分类错误会污染整条链路。

典型失败：

```text
用户真实目标：修复代码、修改文件、运行 pytest
分类结果：test_report_triage
结果：prompt 告诉模型不负责改代码，driver 不强制 edit_file 和 terminal
```

设计修正：

```text
task_goal_type 降级为 StrategyPrototype
ExecutionObligation 升级为硬约束来源
```

### 2.2 意图理解仍偏闭集分类

当前 `_resolve_task_goal_type()` 是 first-match 分支。它适合生成初步候选，不适合作为长任务控制核心。

正确方式：

- 从用户目标中抽取动作、对象、产物、约束、验证要求。
- 允许未知任务进入 generic professional task。
- 不因未知任务丧失执行能力。
- 任务域只给默认资源建议，不做最终裁决。

### 2.3 三模式已有雏形，但还不是统一母版

`backend/orchestration/interaction_mode_policy.py` 已有：

- `role_mode`
- `standard_mode`
- `professional_mode`

但是当前策略仍通过 `task_goal_type` 和 route 推导模式。后续需要改为：

```text
ExecutionObligation + risk + user explicit mode + task complexity
  -> RuntimeInteractionModePolicy
```

其中：

- 角色模式：表达和记忆体验主导，工具只读。
- 标准模式：当前回合内完成有限工具任务。
- 专业模式：可持续、可恢复、可验证、可委派的长任务。

### 2.4 专业 prompt 和投影边界还不够清晰

`backend/prompting/professional_profiles.py` 目前 profile 仍与 `task_goal_type` 绑定，并且部分 profile 写了“你不负责修改代码”这样的硬边界。

后续 prompt 分层必须是：

```text
Base Professional Runtime Contract
  -> Execution Obligation Section
  -> Strategy Prototype Role
  -> Domain/TaskGraph Node Role
  -> Soul Projection Style
  -> Tool/Verification Rules
```

优先级：

```text
用户本轮明确要求
> 执行义务
> 工具与权限策略
> 任务图/策略原型专业角色
> 灵魂投影表达风格
```

投影不能压制执行义务。

### 2.5 状态机已有 ledger，但执行义务状态还没独立成型

`ProfessionalTaskRunDriver` 已有：

- plan item 注入
- tool loop
- action tracker
- goal contract
- deliverable validation
- closeout repair

但当前 action tracker 还是 driver 内部局部对象，不是可独立恢复和审计的正式账本。

需要新增或显式化：

- `ExecutionObligation`
- `ExecutionObligationLedger`
- `ToolObservationLedger`
- `ProfessionalRunState`
- `ResumeDecision`
- `BlockingReason`

### 2.6 持久化资源很多，但恢复入口不够统一

当前 state index、checkpoint、events、runtime objects 已经存在，但长任务恢复需要一个统一入口：

```text
run_id / session_id / task_id
  -> load checkpoint
  -> load ledger
  -> load runtime objects
  -> reconstruct professional state
  -> re-evaluate current user instruction
  -> continue / repair / ask / terminate
```

不能让模型靠聊天历史“记得自己做到哪了”。

### 2.7 写作任务图已经很强，但还没有吃到通用执行器强校验

208 文档已经落地 A-D：

- 写作图配置源增强。
- 运行时上下文可见性修复。
- 节点状态边界。
- 记忆读写协议。
- 大纲线程派生模型。
- 回归测试通过。

但后续必须把这些协议接入通用专业模式执行器，而不是另开写作私路。

## 3. 资源分层与权威源设计

### 3.1 权威分层

后续所有资源按 8 层管理：

| 层级 | 权威源 | 可做什么 | 禁止做什么 |
|---|---|---|---|
| L0 用户当前 turn | user message / explicit inputs | 定义本轮目标和约束 | 被历史状态覆盖 |
| L1 执行义务 | `ExecutionObligation` | 决定必须读写验交 | 被策略原型覆盖 |
| L2 模式策略 | `RuntimeInteractionModePolicy` | 决定工具、验证、投影强度 | 定义任务语义全集 |
| L3 策略原型 | `StrategyPrototype` | 提供默认步骤和专业角色 | 决定是否写入/验证 |
| L4 任务图资源 | `task_graphs / workflows / contracts` | 定义显式图化流程 | 反向改写用户目标 |
| L5 编排资源 | agent/profile/resource policy | 组装运行边界 | 伪造执行结果 |
| L6 灵魂投影 | `SoulRuntimeView` | 表达风格、协作姿态 | 覆盖执行义务 |
| L7 持久状态 | checkpoint / ledger / state index | 恢复进度和证据 | 自动决定当前 turn |

关键原则：

```text
restore does not decide
projection does not authorize
prototype does not obligate
domain does not classify
prompt does not override current turn
```

### 3.2 资源消费顺序

运行时必须按下列顺序消费资源：

```text
1. 读取用户当前 turn 和显式输入
2. 读取可用历史状态作为候选
3. 构造开放意图帧
4. 抽取执行义务
5. 匹配策略原型
6. 构造语义任务契约
7. 构造三模式策略
8. 装配任务资源、编排资源、灵魂投影
9. 进入专业状态机
10. 写入 tool observation / obligation ledger
11. 交付验证
12. 输出边界收口
```

旧状态、旧任务图、旧投影都只能在第 2 步进入候选池，不能直接跳到执行。

### 3.3 TaskDomain 的新位置

TaskDomain 保留，但只作为：

- 管理台目录。
- 任务图归属。
- 默认资源组。
- 文档和模板分类。
- 前端筛选维度。

TaskDomain 不再作为：

- 运行时意图分类器。
- 写入权限开关。
- 验证义务开关。
- prompt profile 唯一选择器。
- 长任务是否继续的判断源。

## 4. 目标架构

### 4.1 总链路

```text
AgentRuntimeChain
  -> MemoryRuntimeView
  -> IntentFrame
  -> ContinuationCandidates
  -> ContinuationDecision
  -> ExecutionObligation
  -> StrategyPrototypeMatch
  -> SemanticTaskContract
  -> RuntimeInteractionModePolicy
  -> TaskExecutionAssemblyBundle
  -> OrchestrationRuntimeBundle
  -> SoulRuntimeView
  -> ProfessionalTaskRunDriver
  -> ProfessionalRunStateMachine
  -> ToolObservationLedger
  -> DeliverableValidator
  -> OutputBoundary
  -> StateIndex / Checkpoint / RuntimeObjects
```

### 4.2 新核心对象

#### ExecutionObligation

职责：从用户本轮请求抽取硬约束。

建议字段：

```python
@dataclass(frozen=True, slots=True)
class ExecutionObligation:
    obligation_id: str
    user_goal: str
    target_refs: tuple[dict[str, Any], ...]
    required_reads: tuple[dict[str, Any], ...]
    required_writes: tuple[dict[str, Any], ...]
    required_commands: tuple[dict[str, Any], ...]
    required_deliverables: tuple[str, ...]
    required_verifications: tuple[dict[str, Any], ...]
    forbidden_claims: tuple[str, ...]
    allowed_side_effect_scope: dict[str, Any]
    confidence: float
    extraction_evidence: dict[str, Any]
    authority: str = "runtime.execution_obligation"
```

硬规则：

- 用户说“改、修复、实现、写入、生成文件”必须产生 write obligation。
- 用户说“运行测试、验证、pytest、检查通过”必须产生 verification obligation。
- 用户给路径、报告、数据文件必须产生 read obligation。
- 用户说“先分析，不要改”必须产生 write forbidden obligation。

#### StrategyPrototype

职责：提供专业策略和角色模板。

建议字段：

```python
@dataclass(frozen=True, slots=True)
class StrategyPrototype:
    prototype_id: str
    title: str
    match_signals: tuple[str, ...]
    default_reasoning_steps: tuple[str, ...]
    default_deliverables: tuple[str, ...]
    prompt_profile_id: str
    validator_profile_id: str
    authority: str = "runtime.strategy_prototype"
```

初始原型：

| prototype | 用途 |
|---|---|
| `code_change_execution` | 代码修改、实现、修复 |
| `test_report_triage` | 失败报告归因 |
| `runtime_trace_analysis` | trace / checkpoint / ledger 诊断 |
| `artifact_delivery` | 文件型产物交付 |
| `material_synthesis` | 多材料综合 |
| `writing_graph_long_run` | 写作任务图长跑 |
| `generic_professional_task` | 未知长任务兜底 |

原型只能加默认步骤，不能取消 ExecutionObligation。

#### ProfessionalRunState

职责：长任务时序状态。

建议状态：

```text
initialized
resources_loaded
obligation_bound
prototype_bound
mode_policy_bound
plan_drafted
action_dispatched
waiting_tool
tool_observed
artifact_written
verification_running
verification_observed
deliverable_validating
repairing
blocked
paused
completed
failed
```

状态推进必须写 ledger 和 checkpoint。

#### ToolObservationLedger

职责：记录真实工具行动。

字段建议：

- tool name
- operation ref
- args hash
- target refs
- side effect scope
- result ref
- success / failure
- stdout / stderr summary
- artifact refs
- observed_at
- satisfies obligations

#### ResumeDecision

职责：恢复时决定继续、返修、阻塞还是完成。

恢复不是直接继续生成，而是：

```text
restore checkpoint
-> replay ledger summary
-> compare obligations
-> identify unsatisfied obligations
-> continue / repair / block / complete
```

### 4.3 三模式作为同一母版的降档配置

| 模式 | runtime lane | 工具 | 记忆 | 投影强度 | 状态机 | 验证 |
|---|---|---|---|---|---|---|
| 角色模式 `role_mode` | `role_interaction` | 搜索/上下文记忆等只读轻能力 | 对话与状态记忆 | primary | terminal-only mini state | summary check |
| 标准模式 `standard_mode` | `standard_task` | 有限工具，默认不委派 | light working memory | companion | current-turn state | basic validator |
| 专业模式 `professional_mode` | `professional_task` | 完整工具、委派、命令、写入 | required working memory | style_only | durable long-run state | strict validator |

统一规则：

- 三模式共用 `ProfessionalTaskRunDriver` 或后续重命名后的统一 `InteractionModeRunDriver`。
- 角色模式不是旧 simple，它只是专业母版的低风险对话档。
- 标准模式不是任务图模式，它是当前回合工具任务档。
- 专业模式是唯一完整长任务母版。

### 4.4 灵魂投影接入方式

投影接入点保留在 `SoulRuntimeView`，但新增/强化元数据：

- `interaction_mode`
- `projection_strength`
- `strategy_prototype_id`
- `professional_profile_id`
- `execution_obligation_ref`
- `mode_policy_ref`

装配原则：

```text
role_mode: 投影主导表达
standard_mode: 投影陪伴表达
professional_mode: 投影只调表达温度和组织方式
```

专业模式的 prompt 中必须明确：

```text
当前灵魂投影只影响表达方式。
你必须优先满足用户本轮目标、执行义务、工具策略和交付验证。
```

### 4.5 写作任务图接入方式

写作图不新增私有后端执行器。

正确接入：

```text
Writing TaskGraph
  -> StrategyPrototype: writing_graph_long_run
  -> ExecutionObligation: produce / review / commit / continue
  -> ProfessionalRunStateMachine
  -> WritingStagePacket / ContinuityPacket / RevisionPacket / CommitPacket
  -> ToolObservationLedger / MemoryCommitReceipt
  -> DeliverableValidator
```

208 中已经定义的候选、审核、提交、记忆防污染协议，后续进入通用专业状态机的 obligation 和 validator，不另起写作专用链路。

## 5. Claude Code 式持久化运行方法

### 5.1 要借鉴的不是具体 UI，而是运行不变量

Claude Code 类长任务稳定性的关键不在于有多少 prompt，而在于：

- 进度状态显式。
- todo / plan 可恢复。
- 工具调用有真实观察。
- 文件编辑有读写状态。
- 权限模式不等于任务分类。
- 长任务能 pause / resume / repair。

本项目对应实现：

| Claude Code 不变量 | 本项目落点 |
|---|---|
| plan / todo state | `TaskRunLedger + ProfessionalRunState` |
| tool result observation | `ToolObservationLedger` |
| file read before edit | tool contract gate + file read state |
| permission mode | `RuntimeInteractionModePolicy.tool_policy` |
| resume session | checkpoint + state_index + runtime_objects |
| final answer boundary | `DeliverableValidator + OutputBoundary` |

### 5.2 持久化会话协议

新增统一概念：`ProfessionalRunSession`。

建议字段：

```python
@dataclass(frozen=True, slots=True)
class ProfessionalRunSession:
    run_id: str
    session_id: str
    task_id: str
    project_id: str
    interaction_mode: str
    current_state: str
    obligation_ref: str
    strategy_prototype_ref: str
    mode_policy_ref: str
    ledger_ref: str
    checkpoint_ref: str
    runtime_object_refs: tuple[str, ...]
    artifact_refs: tuple[str, ...]
    blocker: dict[str, Any]
    updated_at: float
    authority: str = "runtime.professional_run_session"
```

存储位置：

- session summary: `backend/state_index/task_runs`
- ledger: 复用 `TaskRunLedger`
- checkpoint: `backend/checkpoints`
- runtime objects: `backend/runtime_objects`
- events: `backend/events`
- live view: `backend/runtime_views/session_live`

### 5.3 恢复流程

固定恢复流程：

```text
1. resolve run handle
2. load ProfessionalRunSession
3. load checkpoint
4. load TaskRunLedger
5. load ToolObservationLedger
6. load RuntimeObjects
7. rebuild ProfessionalRunState
8. re-read current user turn
9. extract current ExecutionObligation
10. compare old obligations and current obligations
11. decide resume / revise / restart / clarify
12. continue from first unsatisfied obligation
```

禁止：

- 从聊天历史猜进度。
- 从 final answer 文本猜是否已经写文件。
- 只因 checkpoint 存在就继续旧任务。
- 忽略用户本轮“停止、先分析、不要改、改成另一个目标”等新指令。

### 5.4 pause / block / resume 规则

状态机必须区分：

| 状态 | 含义 | 下一步 |
|---|---|---|
| `paused` | 主动暂停，可继续 | 等用户恢复 |
| `blocked` | 缺权限、缺材料、工具失败 | 给出阻塞原因和需要的输入 |
| `repairing` | 交付验证失败但可修 | 用已有证据返修 |
| `failed` | 不可恢复失败 | 输出失败边界 |
| `completed` | 所有义务满足 | 收口并写最终状态 |

| 风险 | 控制 |

## 10. 切换规则

### 10.1 Shadow 阶段

- 新 obligation/prototype/state machine 只记录，不改变执行。
- 对比旧 `task_goal_type` 与新 obligation 的差异。
- 长任务失败报告必须记录差异。

### 10.2 Enforce 阶段

- write / verify / read / deliver obligation 开始硬约束。
- prompt profile 不再能取消 obligation。
- deliverable validator 以 obligation 为准。

### 10.3 Cutover 阶段

- 删除旧 autonomous/simple/managed 分支。
- 三模式全部走统一 driver。
- 旧测试改写或删除。

### 10.4 Rollback 条件

只允许回滚到 shadow，不允许恢复旧分叉长期并存。

触发条件：

- 专业模式无法执行基本读写验证。
- role mode 无法保持只读。
- standard mode 工具预算失控。
- resume 出现重复写入真实副作用。

## 11. 禁止的实现捷径

1. 不允许用更多 keyword 分类替代 ExecutionObligation。
2. 不允许通过 prompt 说“请记得验证”来代替状态机验证。
3. 不允许为了兼容保留旧 autonomous driver 正式路径。
4. 不允许把 TaskDomain 扩成任务意图全集。
5. 不允许让灵魂投影决定工具权限。
6. 不允许从聊天历史恢复长任务进度。
7. 不允许用 final answer 文本判断文件是否写入成功。
8. 不允许测试里伪造结果绕过真实工具/状态验证。
9. 不允许写给 agent 的 prompt 变成开发说明。
10. 不允许把写作任务图做成专用后端私路。

## 12. 成功定义

本轮升级完成后，系统应满足：

1. 项目现有资源被统一索引，并能区分权威层。
2. 三模式只剩 `role_mode / standard_mode / professional_mode`。
3. 专业模式能自然执行长任务，不靠闭集任务域。
4. 用户明确要求写入和验证时，系统必然追踪这些义务。
5. 错误策略原型不会压制用户本轮真实目标。
6. 灵魂系统自然参与表达，但不越权。
7. 写作图接入通用专业状态机，候选/审核/提交边界可验证。
8. 长任务可 pause / resume / repair / complete。
9. 输出不会泄漏工具协议、内部 DSML 或伪工具调用。
10. 旧链路被删干净，测试也不保留旧模式断言。

## 13. 最终目标形态

用户说：

```text
这个长跑失败了，你追踪失败原因，修复代码，然后运行测试确认。
```

系统不再先问“它属于哪个任务域”，而是这样运行：

```text
IntentFrame:
  actions = inspect_failure + modify_code + verify

ExecutionObligation:
  required_reads = failure report + related code
  required_writes = code change
  required_verifications = pytest
  deliverables = changed files + verification result + limitation

StrategyPrototype:
  code_change_execution

ModePolicy:
  professional_mode
  tool budget high
  checkpoint after each tool
  projection_strength style_only

StateMachine:
  resources_loaded
  obligation_bound
  plan_drafted
  action_dispatched
  tool_observed
  artifact_written
  verification_observed
  deliverable_validating
  completed
```

最终回答只基于真实状态：

```text
已修改哪些文件。
运行了什么测试。
测试结果是什么。
哪些限制仍然存在。
```

这才是专业长任务自然执行能力的底座：资源可以很多，prompt 可以专业，投影可以有灵魂，但真正让长任务跑完的是可恢复的义务状态机。

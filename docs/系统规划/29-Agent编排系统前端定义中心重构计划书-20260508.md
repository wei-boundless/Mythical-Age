# Agent编排系统前端定义中心重构计划书

日期：2026-05-08

## 1. 目标

把当前编排系统前端重构为真正的 `Agent 定义中心`。

不是继续堆更多字段，而是让用户能清楚完成：

```text
选择 Agent 类别
-> 进入 Agent 定义目录
-> 配置身份
-> 配置职责覆盖
-> 配置运行档案
-> 配置权限边界
-> 配置上下文与输出
-> 查看承接资格与装配预览
-> 保存
```

## 2. 当前问题

### 2.1 目录和配置流没有闭环

当前页面虽然有：

```text
名册
职责覆盖
运行
权限能力
上下文输出
承接资格
```

但用户进入页面后，不容易理解：

```text
我现在在配哪个 Agent
这个 Agent 是否可编辑
我应该先配身份还是先配运行
一个 Agent 现在到底缺什么
```

### 2.2 子 Agent 与 Agent 组的关系仍然绕

当前存在：

```text
有组 / 无组
选组
选组内成员
切回 Agent 配置
```

流程跳跃感仍然很强。

### 2.3 “自由配置 Agent”的定义不够完整

真正需要自由配置的是：

```text
身份
职责范围
默认灵魂/投影
运行通道
可承接任务范围
权限能力
上下文可见范围
输出契约
承接资格
```

当前这些信息分散在多个 layer，但缺少统一的右侧总览和缺口提示。

## 3. 本轮结构方案

### 3.1 页面类型

这是一个：

```text
高密度工作台 + 定义中心 + Inspector 型控制台
```

### 3.2 页面三区结构

```text
左侧：Agent 目录
中间：定义编辑区
右侧：装配摘要 / 缺口 / 保存动作
```

### 3.3 层级切换

保留 card/tab 风格切换，但只保留对“一个 Agent 定义”真正有意义的层：

```text
身份
职责
运行
权限
上下文
资格
```

对 `worker_sub_agent` 额外保留：

```text
Agent 组
```

### 3.4 关键原则

1. 主 Agent / 系统管理 Agent 默认锁定身份层关键字段。
2. 子 Agent 可以自由配置，但仍沿用后端约束。
3. Agent 组是子 Agent 的组织视图，不是 Agent 本体配置页。
4. 页面右侧必须始终给出当前 Agent 的完整装配摘要。
5. “缺什么不能运行”必须可见，不能靠用户自己猜。
6. 编排前端不得把旧 `task_mode` 模板枚举作为产品入口；运行档案里的承接范围应来自当前任务域、具体任务、任务图和工作流资源。

## 5. 本轮验收

1. 左侧目录能清楚区分主 Agent、系统 Agent、子 Agent。
2. 子 Agent 支持完整配置身份、职责、运行、权限、上下文、资格。
3. Agent 组配置页能直接管理成员，不和 Agent 本体编辑混页。
4. 右侧始终显示当前 Agent 的装配摘要和缺口。
5. 页面结构清楚，不再只是“字段拼盘”。

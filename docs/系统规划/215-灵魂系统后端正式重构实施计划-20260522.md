# 215-灵魂系统后端正式重构实施计划

日期：2026-05-22

状态：执行中

## 0. 本轮目标

本轮只重构后端，把灵魂系统从“单一 seed + 投影 catalog”推进到正式资源库结构。前端暂不重构，但后端必须给前端下一步改造准备稳定 API。

最终后端要同时满足两件事：

1. 新结构成为真实设计中心：世界背景、灵魂故事、灵魂卡片、工作 prompt、共同契约、显化、工作日志。
2. 旧接口短期继续可用：`/soul/catalog`、`/soul/projections` 和现有切换逻辑不崩，但它们要逐步由新结构派生。

## 1. 现状判断

当前后端主要入口是：

- `backend/soul/registry.py`
- `backend/soul/registry_service.py`
- `backend/soul/projection_store.py`
- `backend/soul/projection_service.py`
- `backend/soul/runtime_assembly.py`
- `backend/api/souls.py`

当前前端仍然依赖：

- `seeds`
- `static_files`
- `soul_profiles`
- `projectionCards`
- 固定五个灵魂 key 的部分辅助函数

所以本轮不能粗暴删除旧字段。正确路径是：新增正式资源库服务，再让旧 catalog 适配它。

## 2. 后端目标结构

### 2.1 资源库对象

正式资源库包含：

- `SoulWorld`：背景世界。
- `SoulStory`：灵魂故事。
- `SoulCard`：灵魂卡片聚合视图。
- `WorkPrompt`：无背景、无灵魂扮演的工作 prompt。
- `CommonContractPrompt`：用户共同契约。
- `SoulManifestation`：前端显化配置。
- `SoulWorkLogView`：只读工作日志视图。

### 2.2 存储策略

本轮不引入数据库，使用 JSON 文件库：

- `backend/soul/worlds/catalog.json`
- `backend/soul/stories/catalog.json`
- `backend/soul/cards/catalog.json`
- `backend/soul/work_prompts/catalog.json`
- `backend/soul/common_contracts/catalog.json`
- `backend/soul/manifestations/catalog.json`

理由：

- 现有灵魂系统已以文件为主。
- 前端管理页下一步可直接对接 catalog。
- 不把重构扩大到数据库迁移。

### 2.3 运行模式

模式固定为：

- `role_mode`：共同契约 + 世界背景 + 灵魂本体 + 工作投影。
- `standard_mode`：共同契约 + 灵魂本体 + 工作投影。
- `work_mode`：共同契约 + 任务契约 + 工作 prompt。

硬约束：

- `work_mode` 不注入世界背景。
- `work_mode` 不注入灵魂故事。
- 显化不影响权限、工具、执行义务和验证裁决。
- 工作日志不复制记忆正文，不参与运行裁决。

## 3. 实施步骤

### 阶段 1：模型补齐

动作：

1. 新增 `SoulCard`、`SoulModeProfile`、`SoulCatalog`。
2. 保留已有 `SoulWorld`、`SoulStory`、`WorkPrompt`、`SoulManifestation`、`SoulActivityEvent`、`SoulWorkLogView`。
3. 所有对象提供 `to_dict()`，保证 API 输出稳定。

完成标准：

- 后端可以表达完整灵魂资源，不依赖前端硬编码。

### 阶段 2：资源库服务

动作：

1. 新增 `backend/soul/catalog_store.py`，负责 JSON 文件读写、默认数据初始化、路径安全。
2. 新增 `backend/soul/catalog_service.py`，负责统一输出新资源 catalog。
3. 默认把现有五个内置灵魂映射为 `SoulCard + SoulStory + SoulManifestation`。
4. 默认创建至少一个 `work_prompt` 和一个 `common_contract`。

完成标准：

- `SoulFacade.build_resource_catalog()` 能返回新结构。
- 新结构里每个内置灵魂都有卡片、故事和显化。

### 阶段 3：旧 catalog 适配

动作：

1. `SoulRegistryService.build_catalog()` 增加 `resource_catalog` 字段。
2. 保留 `seeds/static_files/soul_profiles` 字段。
3. `management.planes` 增加新资源层。

完成标准：

- 现有前端仍能读取旧字段。
- 新前端可以从 `resource_catalog` 读取正式结构。

### 阶段 4：模式装配服务

动作：

1. 新增 `backend/soul/mode_assembly.py`。
2. 输入 `mode/soul_id/projection_id/work_prompt_id/task_contract`。
3. 输出可预览 sections 和 trace。
4. 严格验证 `work_mode` 不含世界和故事 section。

完成标准：

- API 可以预览三种模式装配。
- 测试覆盖 `work_mode` 边界。

### 阶段 5：API 接入

动作：

1. 新增 `GET /soul/resources`。
2. 新增 `POST /soul/modes/preview`。
3. 保留 `GET /soul/{soul_id}/activity`。

完成标准：

- 前端后续能按资源库页面重构。
- 工作日志接口仍然轻量。

## 4. 不做的事

本轮不做：

- 不重构前端页面。
- 不引入数据库。
- 不删除旧 seed 文件。
- 不把工作日志升级为记忆管理系统。
- 不让显化参与运行权限。

## 5. 验证矩阵

必须验证：

1. 新资源 catalog 能构建。
2. 旧 `/soul/catalog` 仍返回旧字段。
3. 五个内置灵魂都有 `SoulCard`。
4. `work_mode` 装配不包含世界背景和灵魂故事。
5. `role_mode` 装配包含世界背景和灵魂故事。
6. 工作日志不返回记忆正文。
7. `SoulFacade` 导入不触发 runtime 循环导入。

日期：2026-05-30

状态：实施计划，作为 `007-图任务系统架构设计书`、`009-图任务编辑器升级计划书`、`004-图任务GraphHarness重建计划书` 的边界修正与执行计划

适用范围：

```text
backend/harness/graph
backend/harness/graph_harness.py
backend/query/runtime.py
backend/harness/runtime/compiler.py
```

## 1. 已有设计原则复核

本计划不是重新发明图任务。现有 `docs/系统框架/007-图任务系统架构设计书-20260528.md` 已经写明核心原则：

```text
完整图语言是事实源。
GraphHarnessConfig 是执行合同。
SchedulerView 是调度派生物。
GraphLoop 是状态推进器。
AgentHarness 是 agent 节点执行器。
```

设计书也明确要求：

```text
GraphLoopState 只存状态和 refs，不存长文本。
大内容进 artifact/memory repository。
节点输入按边 selector 定向读取。
agent 只能请求和产出候选。
系统负责授权、执行、提交、回执。
GraphLoop 不相信 agent 自述的系统状态。
RuntimeInvocationPacket 是唯一 agent 可见世界入口。
```

当前实现偏离了这些原则，偏离点不是图任务理念错误，而是 harness/loop 实装把控制、通信、观察混在了一条大 payload 链里。

## 2. 图任务为什么存在

图任务不是为了把多个 agent 输出拼接成一个大 prompt。图任务存在的理由是：

```text
1. 用拓扑表达复杂任务的时序、依赖、返修、循环、审核和资源边界。
2. 把长任务拆成可恢复、可审计、可重试的节点执行单元。
3. 让每个节点以明确角色和局部上下文执行，避免单 agent 一次性吞下全局复杂度。
4. 让 memory/artifact/file/issue ledger 成为跨节点协作底座，而不是靠 prompt 传全文。
5. 让系统负责调度、权限、落库、checkpoint 和 monitor，agent 只完成当前节点任务。
```

因此，正确消耗模型是：

```text
图级消耗：控制系统读取 GraphHarnessConfig 与 GraphLoopState。
节点级消耗：单 agent 只读取当前节点合同和入边允许的局部上下文。
资源级消耗：大文本通过 artifact/memory/file refs 按需读取，不默认塞进节点 prompt。
监控级消耗：只读摘要和 refs，不复制完整运行体。
```

## 3. 节点、边、记忆库的设计原则

### 3.1 节点

节点是执行单元，不是整张图的视角。节点只能看到：

```text
node_identity
prompt_contract
input_contract
output_contract
inbound_context
memory_view
artifact_view
file_view
permission_summary
tool_capability_table
expected_result_contract
```

节点不应该看到：

```text
完整 GraphLoopState
完整 result_index
完整 work_order_index
完整 event log
所有上游节点全文输出
整张图的运行历史
```

### 3.2 边

边有两个不同角色，不能混用：

```text
scheduler_role：控制 ready 判断。
semantic_role / payload_contract：控制节点间交接内容。
```

`dependency` 边只说明“目标节点什么时候可以运行”，不自动意味着“把源节点完整结果传给目标节点”。

正确交接应由边契约声明：

```text
edge_id
source_node_id
target_node_id
source_result_ref
delivery_policy
payload_contract_id
artifact_refs
memory_refs
handoff_summary
bounded_excerpt
structured_payload
```

默认策略必须是安全瘦身：

```text
默认 refs + summary。
只有边契约明确 include_output_keys / max_chars 时，才允许传 bounded excerpt。
禁止默认传 outputs.final_answer 全文。
```

### 3.3 记忆库和产物库

memory/artifact/file 是跨节点通信和长期状态的底座，不是 prompt 附件池。

原则：

```text
大文本进入 artifact repository 或 memory repository。
GraphLoopState 只记录 refs、receipt、producer_node_id、content_hash。
下游节点通过 memory_read/artifact_context/file_read 策略读取需要的部分。
agent 只能产出 candidate 或 request，正式写入由系统服务完成。
正式 memory/artifact 必须有 commit/review receipt。
```

## 4. 当前代码错在哪里

### 4.1 GraphLoopState 保存了完整执行包和完整节点结果

证据：

```text
backend/harness/graph/models.py
GraphLoopState.work_order_index: dict[str, dict]
GraphLoopState.result_index: dict[str, dict]
GraphLoopState.result_history: dict[str, tuple[dict]]

backend/harness/graph/loop.py
_state_with_work_orders() 写入 order.to_dict()
accept_node_result() 写入 envelope.to_dict()
```

问题：

```text
work_order.to_dict() 包含 input_package。
NodeResultEnvelope.to_dict() 包含 outputs.final_answer。
这些字段进入 checkpoint 后，又被 event、monitor、下游 materializer 反复复制。
```

目标：

```text
GraphLoopState 只保存 WorkOrderRef、NodeResultRef 和小型摘要。
完整 work order/result 进入独立 store 或由事件 refs 查询。
```

### 4.2 Event 复制完整 state/result/work_order

证据：

```text
graph_loop_started -> graph_loop_state + node_work_orders
graph_ready_nodes_dispatched -> graph_loop_state + node_work_orders
graph_node_result_accepted -> node_result + graph_loop_state + node_work_orders
```

问题：

```text
事件日志从事实流退化成全量快照流。
每个节点都会复制此前累积结果。
Monitor 再读取 events 导致返回体指数级膨胀。
```

目标：

```text
event payload 只存事件事实和 refs。
checkpoint 才是恢复状态源。
monitor 只读 projection，不读全量 event payload。
```

### 4.3 ContextMaterializer 把调度边当通信边

证据：

```text
backend/harness/graph/context_materializer.py
_incoming_dependency_edges() 读取 SchedulerView.dependency_edges
upstream_results_for_node() 从 result_index 取完整结果
handoff_packets_for_node() 从 result_index 取完整结果
```

问题：

```text
scheduler dependency 被误用为 payload source。
memory_read/artifact_context/file_read 等上下文边没有成为一等输入物化路径。
```

目标：

```text
Materializer 根据 edge semantic_role / delivery_policy / context_filter_policy 生成 inbound_context。
SchedulerView 只控制 ready，不控制可见上下文。
```

### 4.4 节点输入包重复携带上游 payload

证据：

```text
input_package.upstream_results
input_package.upstream_handoff_packets
input_package.handoff_packets
```

问题：

```text
同一份上游输出被三份字段重复携带。
字段命名保留旧链路痕迹，破坏单一通信协议。
```

目标：

```text
删除 upstream_results / upstream_handoff_packets / handoff_packets。
统一为 inbound_context。
```

### 4.5 NodeResultEnvelope 默认携带 final_answer 全文

证据：

```text
backend/harness/graph/work_order_executor.py
outputs.final_answer = final_answer
handoff_summary = final_answer[:1200]
```

问题：

```text
final_answer 进入 result_index 后成为下游 prompt 默认输入。
大文本应先物化为 artifact/memory，再通过 refs 传递。
```

目标：

```text
NodeResultEnvelope.outputs 默认不携带完整 final_answer。
保留 final_answer_ref / artifact_refs / handoff_summary / structured_output。
必要时 debug trace 可查完整 executor_result。
```

### 4.6 Monitor 返回全量运行体

证据：

```text
backend/harness/graph_harness.py
get_graph_run_monitor() 返回 full graph_loop_state、active_node_work_orders、node_runtime_views、events。
_node_runtime_views() 返回 full work_order 和 full result。
```

问题：

```text
monitor 是观察视图，却搬运恢复状态和执行 payload。
前端 1.8s 拉取一次 monitor 时，会放大为系统级性能问题。
```

目标：

```text
monitor 默认返回 summary projection。
full payload 只能由显式 debug endpoint 或 trace endpoint 查询。
```

## 5. 目标链路

```text
GraphHarnessConfig
-> GraphRuntime.start 锁定 config
-> GraphLoop.initialize 生成控制状态
-> GraphLoop.dispatch_ready 生成 NodeWorkOrderRef + NodeWorkOrder
-> GraphContextMaterializer 生成局部 NodeInputPackage
-> AgentHarness 执行单节点
-> WorkOrderExecutor 生成 NodeResultEnvelope + artifact/memory receipts
-> GraphLoop.accept_node_result 写入 NodeResultRef 和状态转移
-> GraphRunRunner 继续推进
-> MonitorProjection 输出摘要
```

关键规则：

```text
GraphLoopState 不保存大文本。
Event 不保存大文本。
Monitor 不返回大文本。
NodeInputPackage 不默认携带上游全文。
跨节点通信只通过 inbound_context + refs + bounded excerpt。
```

### Phase 3：节点输入包重建

文件：


动作：



## 7. 禁止事项

```text
禁止为了兼容保留 upstream_handoff_packets/handoff_packets 双字段。
禁止把 monitor 作为恢复源。
禁止 event payload 保存 full checkpoint。
禁止 GraphLoopState 保存完整 input_package。
禁止默认把 outputs.final_answer 传给下游节点。
禁止通过增大 token budget 掩盖图通信错误。
禁止让 GraphLoop 执行 agent 内环。
禁止让 AgentHarness 判断图 ready 状态。
```

## 8. 交付判断

完成后，图任务 harness 应满足：

```text
图是拓扑控制。
loop 是状态推进。
节点是局部执行。
边是交接合同。
记忆库/产物库是跨节点数据底座。
monitor 是观察投影。
checkpoint 是恢复状态，不是 prompt 数据源。
```

这才符合已有图任务框架设计书，也符合成熟 agent workflow control plane 的基本边界。

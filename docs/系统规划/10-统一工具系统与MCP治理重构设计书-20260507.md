## 一、文档定位

本文件用于定义本项目下一阶段的工具系统重构方案。

本次设计解决的不是单一 MCP 接入 bug，也不是单一 RAG 路由修补，而是平台当前在以下几个层面同时存在的结构性混乱：

- 工具池定义与能力池定义混杂
- tool / operation / mcp / worker 的边界不一致
- 候选发现、模型可见性、执行授权三层职责没有分开
- 旧 `pdf_analysis` / `structured_data_analysis` 工具链与新的本地 MCP 路线长期并存并互相抢路由
- `allowed_tools`、`allowed_operations`、`available_to_model` 等治理字段出现语义漂移

## 二、问题定义

### 2.1 当前失败模式

当前系统已经开始把 RAG / PDF / 结构化数据接入本地 MCP 单元，但主系统仍然存在旧工具链残留，因此出现以下典型现象：

1. 理解层把某些 MCP 所属能力仍识别成 `tool`
2. query understanding 会根据 `capability_requests` 自动补出旧 tool 候选
3. runtime required operations 会把旧 tool operation 又塞回模型执行面
4. 资源策略层同时维护 operation 视图与 tool 视图，但字段语义不一致
5. MCP 工具池、内置工具池、权限服务的“可见性判定”不是同一条规则

最终表象是：

- RAG 已经触发了 MCP 检索，但旧 `search_text` 仍可被模型请求并消耗预算
- PDF / structured_data 已经具备 MCP 路由，但规划语义和权限语义仍保留旧 tool 影子
- 已知路径的读取请求，本应直接 `read_file`，却仍可能把 `search_files` / `search_text` 暴露给模型

### 2.2 真正损坏的系统属性

当前损坏的不是“某个工具定义写错了”，而是以下平台级属性：

1. **单一能力只有一条正式执行路线** 这个属性被破坏了  
   PDF / structured_data 同时存在 tool 路线和 MCP 路线。

2. **候选发现不等于模型可见性，不等于执行授权** 这个属性被破坏了  
   现在三个阶段互相串味。

3. **工具名、operation id、MCP 单元 id、能力记录 id 各自有唯一职责** 这个属性被破坏了  
   当前实现中多处混用。

4. **已绑定资源优先走确定性读取，不再走模糊搜索** 这个属性被破坏了  
   `read_file`、`search_files`、`search_text` 之间没有形成稳定的职责链。

### 2.3 正确的目标终态

重构完成后，系统必须满足：

1. 每类正式能力只有一个外部主路线：
   - RAG -> MCP retrieval
   - PDF -> MCP pdf
   - structured_data -> MCP structured_data
   - workspace read/search/edit -> builtin tools
   - network latest -> builtin `web_search` / `fetch_url` 或 external MCP
   - weather / gold price -> task intent / task template，不再作为独立底座工具

2. 一个资源是否会被模型看到，必须由统一的“工具池纳管 + 资源策略 + prompt 可见性”三段规则决定。

3. `allowed_tools` 永远表示工具名集合，`allowed_operations` 永远表示 operation id 集合，`allowed_mcps` 永远表示 MCP server 或 MCP route 集合，不允许再混用。

4. 已知路径时只允许：
   - `read_file`
   - 对应的 MCP route
   不再把 `search_files` / `search_text` 当默认陪跑工具暴露给模型。

5. skills 不再绑定 tools，它们只描述能力方法包、触发条件和上下文使用规则。

6. 底座工具只保留可组合的通用动作，不保留天气、金价这类垂直任务工具。垂直需求必须通过 task/template + 通用工具组合实现。

## 三、现状源码审查

### 3.1 旧 PDF / 结构化数据工具仍以正式工具形式存在

文件：

- [backend/capability_system/tool_definitions.py](/d:/AI应用/langchain-agent/backend/capability_system/tool_definitions.py)
- [backend/capability_system/operation_registry.py](/d:/AI应用/langchain-agent/backend/capability_system/operation_registry.py)

当前仍登记：

- `pdf_analysis` -> `op.pdf_analysis`
- `structured_data_analysis` -> `op.structured_data_analysis`

而本地 MCP 已经同时登记：

- `op.mcp_pdf`
- `op.mcp_structured_data`

这违反了“单能力单正式路线”原则。

### 3.2 候选生成依赖模糊 metadata 打分，未按职责分池

文件：

- [backend/capability_system/tool_registry.py](/d:/AI应用/langchain-agent/backend/capability_system/tool_registry.py)

当前 `resolve_candidate_names()` 和 `select_best()` 同时基于：

- `route_hints`
- `capability_tags`
- `supported_modalities`

做重叠打分。

这导致：

- `search_files`
- `search_text`
- 部分旧分析工具

只要 metadata 有交集，就可能一起进入候选池。

这说明当前候选发现是“模糊匹配型工具检索”，而不是“按能力类别分段匹配”。

### 3.3 query understanding 负责了不该负责的工具补全

文件：

- [backend/understanding/query_understanding.py](/d:/AI应用/langchain-agent/backend/understanding/query_understanding.py)

当前 `_apply_skill_tool_routing()` 会在没有 candidate tools 时，根据 `capability_requests` 自动从 `tool_registry` 里补候选，然后可能自动选出 `tool_name`。

这一步把：

- 理解层
- 候选生成层
- 工具选择层

耦合在一起。

在旧架构里这会导致 MCP 所属能力被重新“工具化”。

### 3.4 runtime resource policy 同时维护 tool 视图与 operation 视图，但字段语义已经漂移

文件：

- [backend/orchestration/resource_policy_builder.py](/d:/AI应用/langchain-agent/backend/orchestration/resource_policy_builder.py)
- [backend/capability_system/tool_contracts.py](/d:/AI应用/langchain-agent/backend/capability_system/tool_contracts.py)
- [backend/permissions/decision_pipeline.py](/d:/AI应用/langchain-agent/backend/permissions/decision_pipeline.py)

当前问题：

- `allowed_operations` 存 operation id
- `allowed_tools` 却直接被赋值为 `allowed_operations`

但权限系统中的 `ToolScope.allowed_tools` 明确按“工具名”工作。

这是一个严格的类型语义错误，不是风格问题。

### 3.5 模型可见性有多套判定源

文件：

- [backend/capability_system/tool_authorization.py](/d:/AI应用/langchain-agent/backend/capability_system/tool_authorization.py)
- [backend/orchestration/resource_runtime_view.py](/d:/AI应用/langchain-agent/backend/orchestration/resource_runtime_view.py)
- [backend/mcp_server/tool_pool.py](/d:/AI应用/langchain-agent/backend/mcp_server/tool_pool.py)

当前至少有三套相关规则：

1. `runtime_visibility`
2. `prompt_exposure_policy`
3. `available_to_model`

并且它们分别在不同地方生效。

这意味着：

- 工具可能在运行时被授权但不该对模型可见
- 工具可能在工具池里被标成可见，但最终并不会进入本轮 model lane

这是“资源池视图”与“执行面事实”不一致的问题。

### 3.6 本地读取工具职责链不清晰

文件：

- [backend/capability_system/units/tools/search_files_tool.py](/d:/AI应用/langchain-agent/backend/capability_system/units/tools/search_files_tool.py)
- [backend/capability_system/tool_definitions.py](/d:/AI应用/langchain-agent/backend/capability_system/tool_definitions.py)

当前三个工具本来应该形成清晰分层：

- `read_file`: 已知路径直接读取
- `search_files`: 文件路径/文件名发现
- `search_text`: 文件内容发现

但 metadata 上：

- `search_files` 和 `search_text` 都标了 `workspace_search`
- `read_file` 又被理解层当 direct tool，但未形成“已知路径优先”硬约束

这导致基本工具边界本身也不清晰。

### 3.7 特定任务被错误建模成底座工具

文件：

- [backend/capability_system/tool_definitions.py](/d:/AI应用/langchain-agent/backend/capability_system/tool_definitions.py)
- [backend/capability_system/operation_registry.py](/d:/AI应用/langchain-agent/backend/capability_system/operation_registry.py)
- [backend/understanding/task_understanding.py](/d:/AI应用/langchain-agent/backend/understanding/task_understanding.py)
- [backend/tasks/template_registry.py](/d:/AI应用/langchain-agent/backend/tasks/template_registry.py)

当前 `get_weather` 与 `get_gold_price` 被登记为正式工具与 operation：

- `get_weather` -> `op.get_weather`
- `get_gold_price` -> `op.get_gold_price`

这两类能力的问题不在于实现不可用，而在于抽象层级错误：

- 天气查询是“实时地点信息查询”任务
- 金价查询是“实时金融行情查询”任务

它们不是底座工具。底座工具应该是可组合动作，例如：

- 搜索网页
- 抓取 URL
- 读取文件
- 解析结构化配置
- 查询 Git 状态

如果把每个垂直任务都注册成工具，系统会持续膨胀为：

- `get_weather`
- `get_gold_price`
- `get_stock_price`
- `get_exchange_rate`
- `get_train_ticket`
- `get_air_quality`

这会破坏工具池的职责边界，也会让候选路由越来越依赖任务名匹配，而不是能力编排。

因此本次重构明确要求：

- 删除 `get_weather`
- 删除 `get_gold_price`
- 删除对应 operation
- 删除理解层对这两个专用工具的直接路由
- 将天气与金价视为 `realtime_network` 任务，由 `web_search` / `fetch_url` 或未来 external MCP 供应

### 3.8 底座工具缺口

当前已有工具主要覆盖：

- `read_file`
- `write_file`
- `edit_file`
- `search_files`
- `search_text`
- `web_search`
- `fetch_url`
- `terminal`
- `python_repl`

但一个稳定的 Agent 底座仍缺少以下通用动作：

1. 路径与目录检查
   - `list_dir`
   - `stat_path`
   - `path_exists`
   - `glob_paths`

2. 结构化文件读取
   - `read_structured_file`
   - 支持 JSON / YAML / TOML
   - 输出统一结构摘要，而不是原文字符串

3. Git 只读检查
   - `git_status`
   - `git_diff`
   - `git_log`
   - `git_show`

4. 安全文件变更动作
   - `make_dir`
   - `copy_path`
   - `move_path`
   - `delete_path`

5. 代码结构导航
   - `file_outline`
   - `find_symbol`
   - `find_references`

本次第一批只补最基础、最容易稳定验证的三组：

- 路径与目录检查
- 结构化文件读取
- Git 只读检查

文件变更动作与代码结构导航进入第二批，避免第一轮同时扩大读写权限面和语义路由复杂度。

## 四、从设计原则提取的约束

### 4.1 来自设计原则第 9 篇：工具池必须有单一注册面与稳定排序

约束：

1. 工具池必须统一装配
2. 工具可见性不能靠散落判断
3. Prompt cache 依赖稳定排序
4. 工具注册与工具暴露是两回事

迁移到本项目后，意味着：

- 所有 builtin tool、local MCP、external MCP 都必须先进入统一工具池视图
- 但“进入池”不代表“可见给模型”

### 4.2 来自设计原则第 15 篇：MCP 应按标准化桥梁纳管，而不是补丁式外挂

约束：

1. MCP 是统一协议桥
2. 本地 MCP 与外部 MCP 都应该被抽象成同类资源
3. Tool discovery、去重、排序、后续连接治理都应纳入统一策略

迁移到本项目后，意味着：

- 本地 MCP 不应只是几个特殊 operation
- 它们应被统一视为“非直接模型工具的能力路由资源”

### 4.3 来自设计原则第 16 篇：权限必须 fail-closed，模式与规则必须分层

约束：

1. 权限模式是全局信任级
2. 资源规则是细粒度约束
3. allow / deny / ask 的语义不能漂移

迁移到本项目后，意味着：

- operation policy
- tool scope
- MCP scope

必须拆开建模，不能复用一个字段硬塞。

### 4.4 来自设计原则第 24 篇：skills 是方法包，不是工具绑定清单

约束：

1. skill 本质是能力方法包
2. skill 可以提供触发条件、上下文使用策略、局部 prompt
3. skill 不等于 tool registry

迁移到本项目后，意味着：

- skills 不再声明工具拥有权
- skills 只参与理解、提示、上下文表达
- 真正的执行边界归工具池和编排系统

### 4.5 来自设计原则第 25 篇：注册、条件过滤、执行可见性应分层

约束：

1. 注册层
2. 条件过滤层
3. 运行时可见层

这三者必须是有序而不混杂的。

迁移到本项目后，意味着：

- 不允许 query understanding 同时负责候选生成和工具选择
- 不允许 resource policy builder 同时扮演 tool scope 解释器

## 五、推荐设计方向

本次重构推荐采用：

**统一资源注册面 + 分层工具池 + 双轨执行面**

具体地说：

1. 所有动作资源统一注册为 `OperationDescriptor`
2. 所有“可发现对象”统一注册为 `CapabilityEndpoint`
3. 工具池只负责发现与暴露，不直接决定执行
4. 候选发现、模型可见性、执行授权三层完全拆开
5. builtin tool 与 MCP route 都进入统一池，但以不同 kind 管理
6. 旧 `pdf_analysis` / `structured_data_analysis` 不再作为对外正式工具保留

我不建议继续沿用“tool registry 自动从 capability_requests 推候选”的模式，因为这条路径天然会重新制造旧工具复活问题。

## 六、目标设计

### 6.1 核心对象模型

本次重构后，统一围绕六类对象：

1. `OperationDescriptor`
2. `CapabilityEndpoint`
3. `ToolPoolEntry`
4. `RoutePolicy`
5. `ExecutionAuthorization`
6. `CapabilityRouteDecision`

#### 6.1.1 OperationDescriptor

表示实际执行动作。

例如：

- `op.read_file`
- `op.search_files`
- `op.search_text`
- `op.web_search`
- `op.mcp_retrieval`
- `op.mcp_pdf`
- `op.mcp_structured_data`

它回答：

`这个动作执行时的真实风险、读写性、并发性、输入输出契约是什么`

#### 6.1.2 CapabilityEndpoint

表示上层可引用能力入口。

例如：

- builtin tool endpoint
- local MCP endpoint
- external MCP endpoint
- worker endpoint

它回答：

`这个能力入口以什么形式被发现、归属哪个 route family、默认是否可对模型呈现`

#### 6.1.3 ToolPoolEntry

表示统一工具池中的可发现项。

建议字段：

- `entry_id`
- `entry_kind`:
  - `builtin_tool`
  - `local_mcp`
  - `external_mcp`
- `display_name`
- `operation_id`
- `route_family`
- `candidate_visibility`
- `model_visibility`
- `runtime_exposure`
- `requires_explicit_binding`
- `discovery_priority`

#### 6.1.4 RoutePolicy

定义某类请求应如何选择 route family。

建议 route family 固定为：

- `workspace_read`
- `workspace_path_search`
- `workspace_text_search`
- `realtime_network`
- `knowledge_retrieval`
- `document_analysis`
- `structured_data_analysis`
- `workspace_write`

#### 6.1.5 ExecutionAuthorization

统一承载执行授权结果，但拆分三个维度：

- `allowed_operations`
- `allowed_tool_names`
- `allowed_mcp_routes`

不允许再共用一个字段。

#### 6.1.6 CapabilityRouteDecision

表示某次请求最终走哪条能力路线。

建议字段：

- `route_family`
- `selected_entry_kind`
- `selected_entry_id`
- `selected_operation_id`
- `requires_model_tool_exposure`
- `requires_runtime_delegate`

### 6.2 三层治理模型

#### 第一层：候选发现层

职责：

- 只根据请求类型和绑定资源选择 route family
- 不直接选 tool name
- 不直接做执行授权

规则：

1. 已知路径 + workspace file -> `read_file`
2. 已知 pdf / bound pdf -> `document_analysis`
3. 已知 dataset / bound dataset -> `structured_data_analysis`
4. 未知路径但目标是文件 -> `workspace_path_search`
5. 未知文件但目标是内容片段 -> `workspace_text_search`
6. 本地知识问答 -> `knowledge_retrieval`

#### 第二层：模型可见性层

职责：

- 决定本轮哪些 `ToolPoolEntry` 可以进入 model lane

规则：

1. `builtin_tool` 只有在：
   - route family 需要
   - 资源策略允许
   - prompt exposure 为 schema
   时才可见

2. `local_mcp` / `external_mcp`
   默认不作为直接模型工具暴露，除非未来明确支持标准 MCP client tool 注入模式

3. 当请求已经有显式路径时：
   - `search_files`
   - `search_text`
   不能被加入模型可见工具集

#### 第三层：执行授权层

职责：

- 决定某次执行实际可运行什么 operation

规则：

1. 只对 `operation_id` 做最终授权
2. `tool_name` / `mcp_route` 只是映射与展示对象
3. MCP operation 默认是 runtime delegate，不是 direct model tool

### 6.3 新的基本工具职责边界

#### `read_file`

仅用于：

- 已知 workspace 路径
- 明确读取目标

禁止：

- 用来做模糊发现

#### `search_files`

仅用于：

- 已知“想找文件”
- 路径未知
- 文件名或路径片段已知

route hints 收窄为：

- `workspace_path_search`
- `file_search`

移除：

- `workspace_search`

#### `search_text`

仅用于：

- 已知“想找内容”
- 文件路径未知或待缩小范围

route hints 收窄为：

- `workspace_text_search`
- `text_search`
- `code_search`

移除：

- `workspace_search`

### 6.3.1 底座工具保留清单

本轮重构后的 builtin tool 分为以下几组。

#### A. 工作区读取与发现

保留：

- `read_file`
- `search_files`
- `search_text`

新增第一批：

- `list_dir`
- `stat_path`
- `path_exists`
- `glob_paths`

边界：

- `read_file` 只读已知文件
- `list_dir` 只列目录内容
- `stat_path` 只返回元信息
- `path_exists` 只做存在性判断
- `glob_paths` 按模式发现路径
- `search_files` 做名称/路径关键词发现
- `search_text` 做内容发现

#### B. 结构化配置读取

新增第一批：

- `read_structured_file`

支持格式：

- JSON
- YAML
- TOML

边界：

- 只负责解析配置/小型结构化文件
- 不负责业务数据分析
- CSV / XLSX / Parquet 不走此工具，仍归 `structured_data` MCP route

#### C. Git 只读检查

新增第一批：

- `git_status`
- `git_diff`
- `git_log`
- `git_show`

边界：

- 只读 Git 元信息
- 不执行 `git add`
- 不执行 `git commit`
- 不执行 `git checkout`
- 不执行 `git reset`

Git 写操作未来如需接入，必须单独进入高风险写权限管线。

#### D. 网络信息检索

保留：

- `web_search`
- `fetch_url`

退场：

- `get_weather`
- `get_gold_price`

天气、金价、汇率、股票等都属于 `realtime_network` 任务，不属于底座工具。它们应由任务模板组织：

1. 判断是否需要实时信息
2. 选择 `web_search`
3. 必要时用 `fetch_url` 拉取权威来源
4. 模型整理结果与来源

#### E. 写入与执行

保留但默认不自动路由：

- `write_file`
- `edit_file`
- `terminal`
- `python_repl`

边界：

- 仍需权限管线强控制
- 不进入第一批候选自动路由重构
- 不因新增底座工具扩大默认写权限

### 6.3.2 第一批专业工具包选择

本项目第一批底座工具优先使用 Python 标准库，只有在格式解析确有必要时引入轻量依赖。

#### 文件系统

优先使用：

- `pathlib`
- `os.stat`
- `glob`
- `fnmatch`
- `shutil`

不建议第一批引入重型文件管理库。原因是文件系统底座工具必须保持可审计，标准库已经足够。

#### 结构化文件

使用：

- JSON: `json`
- TOML: `tomllib`
- YAML: `PyYAML`

说明：

- Python 3.11+ 已内置 `tomllib`
- YAML 没有标准库，需要 `PyYAML`
- 如 `requirements.txt` 已有 `PyYAML`，直接复用；否则第一批只在依赖明确可用时开启 YAML

#### Git

第一批使用：

- `subprocess.run(["git", ...])`

暂不引入：

- `GitPython`

原因：

- Git 只读命令用 subprocess 更透明
- 输出与用户本地 git 行为一致
- 权限审计可以按命令白名单做

未来如果需要复杂 Git 对象遍历，再考虑 `GitPython`。

#### 路径匹配

第一批使用：

- `glob`
- `fnmatch`

未来如需遵循 `.gitignore` / `.ignore` 规则，可引入：

- `pathspec`

#### 代码符号

第一批不做。

第二批候选：

- LSP server
- `tree-sitter`
- `jedi` for Python
- TypeScript language service for TS/JS

原因：

- 代码结构导航不应该用正则冒充符号系统
- 多语言符号能力需要更明确的语言后端

### 6.4 旧 PDF / 结构化数据工具的退场规则

#### 6.4.1 正式决策

对外正式工具链中，删除：

- `pdf_analysis`
- `structured_data_analysis`

对应删除：

- `ToolDefinition`
- `OperationDescriptor`
- main runtime profile 中的 operation 暴露
- 自动候选匹配入口
- 主模型 direct tool 可见性

#### 6.4.2 保留条件

如其内部逻辑仍被 MCP worker 复用，则应迁移为：

- `backend/evidence/...` 下的内部 adapter / helper
- 或 `capability_system.units.mcp.local.*` 下的内部执行组件

但不再以正式工具形式登记。

即：

- 可以保留实现逻辑
- 不能保留对外 tool identity

### 6.5 skills 的正式边界

skills 不再包含：

- `allowed_tools`
- 工具绑定清单
- 工具所有权定义

skills 只保留：

- `skill_id`
- `name`
- `description`
- `preferred_route_family`
- `activation_policy`
- `context_mode`
- `prompt_view`
- `when_to_use`
- `resource_expectations`

即：

- skill 负责“如何使用一种能力”
- tool / mcp 负责“实际怎么执行”

### 6.6 MCP 的正式位置

MCP 在本项目中分两种：

1. `local_mcp`
2. `external_mcp`

统一都作为：

- `CapabilityEndpoint`
- `ToolPoolEntry`

纳入系统。

但二者默认都不是 direct model tool。

它们的默认执行方式是：

- route decision 选中 endpoint
- runtime delegate 到对应 operation

只有未来明确引入“标准 MCP client 直接把外部工具注入模型”的模式时，才允许外部 MCP 工具在特定上下文下进入 model-visible tool lane。

### 14.3 当前正式边界

当前系统正式边界如下：

1. `rag-skill`
   - 负责本地知识检索方法流

2. `pdf-analysis`
   - 负责 PDF 阅读与证据组织方法流

3. `structured-data-analysis`
   - 负责结构化数据分析方法流

4. `web_search` / `fetch_url`
   - 作为底座工具存在
   - 不再作为 skill 存在

5. local MCP / external MCP
   - 纳入统一工具池治理
   - 默认不作为 direct model tool 注入

# 186-尺度自由任务流模型与任务系统升级计划

日期：2026-05-20

## 1. 总结论

父子图不足以表达我们的目标。父子图是一种树状嵌套，适合表达“父流程调用子流程”，但我们真正想要的是一种尺度自由的任务流系统：

```text
任务像水流。
节点是某个尺度下的水滴。
边是交接河道。
一组水滴按时序和契约汇聚成水流。
水流可以折叠成一个节点。
节点也可以展开成水流。
水流可以分叉成支流，也可以汇聚成河流。
河流可以进入水库沉积为记忆、产物、状态和账本。
多个河流最终汇入大海，形成完整长期任务工程。
```

工程上，这不应该只叫“父子图”。更准确的目标模型是：

```text
尺度自由任务流模型
```

父子图、GraphUnit、子图运行，都是这个模型里的一个特例。未来系统的核心对象应从“图包含子图”升级为：

```text
FlowUnit 可以在任意尺度上折叠或展开。
FlowEdge 负责把 FlowUnit 的输出按契约交接给下一个 FlowUnit。
FlowPacket 是任务流中的可追踪传输单元。
FlowScope 定义上下文、记忆、产物和运行状态的隔离边界。
FlowLedger 记录分流、汇流、接续、提交和失败传播。
FlowCompiler 把任意尺度的任务流编译成可执行运行包。
```

当前任务系统已经有很多基础能力：`TaskGraphDefinition`、`contract_bindings`、`RuntimeSpec`、`SchedulerState`、`ExecutionPackage`、GraphUnit 子图运行、output packet 回交。这些不需要推倒，而应升级为尺度自由任务流的底座。

## 2. 当前系统的主要不足

### 2.1 语义还停留在“图/节点/边”，缺少“任务流”一等概念

当前保存态主对象是 `TaskGraphDefinition`。这很适合作为任务图主数据，但它表达的是一个图，而不是一个可流动、可折叠、可分解、可汇聚的任务流单元。

现有模型有：

- 图：`TaskGraphDefinition`
- 节点：`TaskGraphNodeDefinition`
- 边：`TaskGraphEdgeDefinition`
- 图节点：由 `timeline_blocks[].linked_graph_id` 派生的 GraphUnit

缺少的模型是：

- 一个流程片段如何被折叠为节点。
- 一个节点如何在运行时裂解成多个子流程。
- 多个支流如何按契约汇聚成一个结果。
- 一段流程如何被复用到不同图、不同任务域、不同运行范围。
- 动态生成的子流如何进入同一套发布包、调度、记忆和产物追踪。

所以当前系统能表达“图里有节点和边”，但还没有完整表达“任务流在不同尺度上流动”。

### 2.2 父子图是树状嵌套，不是自由流动

185 号文档已经确认：GraphUnit 可以运行。后端能把 `timeline_blocks[].linked_graph_id` 编译成 `TaskGraphNestedRuntimePlan`，再变成 `graph_unit` runtime node，执行时启动子图 run，完成后 output packet 回交父图。

但这个模型仍偏向：

```text
父图 -> 子图 -> 回父图
```

它还不够表达：

```text
一段任务流拆成多个支流
多个支流汇成一个中间水库
水库沉积后再按条件释放给后续任务流
运行中发现任务不足，动态裂解出新的支流
支流完成后不一定只回到一个父节点，也可能进入共享账本或等待下游选择性读取
```

也就是说，父子图解决的是“嵌套”，但我们的目标需要“聚散”。

### 2.3 GraphUnit 来源仍偏派生，不是标准流单元

当前 GraphUnit 主要由 `metadata.timeline_blocks[]` 派生。前端看起来像“图节点”，后端运行时也能编译，但保存语义仍是“时序块绑定子图”。

这带来几个问题：

- 图节点和时序块混在一起。
- 图节点的接口、版本、端口、交接契约没有独立对象边界。
- GraphUnit 只能表达子图运行，不能自然表达“任意 FlowUnit 可折叠展开”。
- 动态分流、汇流、临时流程、复用流程都没有统一对象承载。

后续 GraphUnit 应降级为 FlowUnit 的一种类型：

```text
GraphUnit = 引用已保存任务图的 FlowUnit
```

而不是让 GraphUnit 成为组合能力的上限。

### 2.4 ComposableGraphView 仍是 shadow / overlay

当前 `build_composable_graph_view()` 会把节点、边、timeline block 派生为 units、interfaces、port_edges、nested_runtime，并在 diagnostics 中出现：

```text
read_only_shadow_model
metadata_overlay_shadow_model
```

这说明可组合图视图是一个派生层。它已经能服务前端工作台，但还不是完整一等保存模型。

在尺度自由任务流模型里，`ComposableUnit`、`UnitInterface`、`UnitPortEdge` 不应该长期停留在 shadow/overlay，而应该逐步成为标准任务流视图：

```text
保存态可以仍兼容 TaskGraphDefinition。
标准态必须能稳定表达 FlowUnit / FlowPort / FlowEdge。
运行态必须能从标准态编译。
编辑器不能让用户感知自己是在编辑派生覆盖层。
```

### 2.5 边仍主要是一对一交接，缺少汇流/分流的强协议

当前边已经有 handoff、temporal、memory、artifact、failure、ack 等配置，这是很好的基础。

但任务流要像河流一样流动，边还需要更强的聚散能力：

- 一对多分流：一个节点结果拆给多个下游。
- 多对一汇流：多个支流结果汇成一个统一 packet。
- 多对多交换：多个工作流之间通过共享 ledger 或 reservoir 交换。
- 条件流：根据验收、状态、记忆、产物是否存在决定走哪条河道。
- 回流修订：审核失败后回到上游，但不能污染已经提交的基准状态。

当前系统里部分能力分散在 edge wait policy、join policy、revision edges、loop frames、scheduler 中，但还缺少统一的 FlowEdge 协议。

### 2.6 时序系统还没有完全变成“水势”

当前时序层已经有 phase、sequence、timeline block、loop frame、temporal edge。Scheduler 也支持 phase gate、sequence gate、handoff ack、failure propagation、wait policy。

但时序仍更像“图的附属层”。在目标模型里，时序应该是任务流推进的核心动力：

- 什么支流先流。
- 什么支流等待。
- 什么支流并行。
- 什么支流汇流。
- 什么结果沉积。
- 什么状态触发下一段流。
- 什么失败被隔离，什么失败向下游扩散。

所以时序不能只是节点字段里的 `phase_id/sequence_index`，也不能只是边上的 `wait_policy`。它需要升级为 FlowClock / FlowGate / FlowWindow 的组合：

```text
FlowClock：定义当前运行步、阶段、轮次、批次。
FlowGate：定义何时可通过。
FlowWindow：定义某段流的可见范围和生效范围。
```

### 2.7 记忆、产物、状态沉积层还不够“流域化”

用户之前强调过基准库和动态记忆库的差异。当前系统已有 memory repository、artifact repository、thread ledger、issue ledger、runtime state store，也已经把风险管理从资源流中拆出来。

但在任务流模型里，记忆和产物不是普通资源，而是水流沉积层：

- 水流经过节点时产生候选产物。
- 候选产物未审核前不能进入基准库。
- 审核通过后 commit，成为下游可见事实。
- 动态记忆是运行中的水位变化。
- 基准库是已经沉积的地层，一般只读，除非通过正式修订流程。

当前系统的问题是：资源层已经存在，但“沉积、提交、修订、释放、可见性”的统一协议还不够强。特别是跨 FlowUnit、跨子流、跨长任务时，需要明确：

```text
什么随着 FlowPacket 流动。
什么只传引用。
什么沉积到动态记忆。
什么进入基准库。
什么需要审核才能 commit。
```

### 2.8 ExecutionPackage 仍偏单图发布包

ExecutionPackage 已经是正确方向。它把 standard view、runtime spec、contract manifest、scheduler、runtime assembly、GraphUnit child plan、object trace 聚合起来。

但目标模型需要更强的 FlowPackage：

- 能递归覆盖多尺度 FlowUnit。
- 能检查分流/汇流协议。
- 能验证 packet schema。
- 能追踪每段流的 scope。
- 能检查版本锚和动态裂解策略。
- 能展示从任意对象到运行事实的 lineage。

当前 ExecutionPackage 能回答：

```text
这个图能不能跑？
GraphUnit 引用的子图是否可预检？
对象是否进入 runtime / manifest / scheduler？
```

未来 FlowPackage 要回答：

```text
这条任务流如何分流、汇流、沉积、释放、接续？
每个 packet 的来源、契约、可见性、状态是什么？
动态生成的支流是否仍在统一运行事实源里？
```

### 2.9 运行层支持 GraphUnit，但还不支持通用动态裂解

当前 runtime loop 能：

- 启动任务图 run。
- 执行节点。
- 识别 GraphUnit stage。
- 启动 child graph run。
- 子图完成后生成 output/failure packet。
- 父图继续。

但“任务像液体”还需要：

- 节点运行中可以按契约请求动态拆分子流。
- 子流数量可以由任务内容决定，不固定在设计时。
- 多个子流可以并行运行。
- 汇流器可以等待全部、任意、法定多数、质量门通过或人工裁决。
- 动态子流也必须有 idempotency、scope、packet、ledger 和 trace。

这不能靠写作任务私门解决，必须成为通用 runtime 能力。

### 2.10 前端工作台已经有形，但还没完全表达流动性

当前前端已经有三栏工作台雏形：

- 左侧图层关系。
- 中间任务图画布。
- 右侧对象编辑台。
- 底部执行包追踪。

不足在于：

- 画布仍主要表达节点边拓扑，而不是“流的聚散”。
- 左侧层级更偏图层树，还不是流域导航。
- 右侧对象编辑台能配节点/边/图节点，但还没把 FlowUnit、FlowPacket、FlowScope、FlowGate 作为自然对象。
- 发布包面板能展示 GraphUnit 子图计划，但还不能展示任务流分叉、汇聚、沉积、释放。

## 3. 目标模型：尺度自由任务流

### 3.1 FlowUnit

FlowUnit 是任务流里的标准单元。它可以是水滴，也可以是一条河流被折叠后的节点。

FlowUnit 类型：

```text
atomic_node：原子执行位
graph_ref：引用已保存任务图
flow_fragment：当前图内的一段流程
dynamic_split：运行时裂解出来的子流
merge_gate：汇流节点
reservoir：记忆/产物/状态沉积层
human_gate：人工确认或裁决
tool_unit：工具执行单元
monitor_unit：后台监控单元
```

关键字段：

```text
flow_unit_id
title
unit_type
source_ref
interface
contract_bindings
scope_policy
runtime_policy
decomposition_policy
folding_policy
lineage
```

对应当前系统：

```text
TaskGraphNodeDefinition -> atomic_node
GraphUnit -> graph_ref
timeline block -> flow_fragment / graph_ref 的旧来源
memory_repository -> reservoir
review_gate/manual_gate -> human_gate
tool node -> tool_unit
```

### 3.2 FlowEdge

FlowEdge 是任务流的河道。它不只是线，而是交接协议。

FlowEdge 类型：

```text
handoff：普通交接
split：分流
merge：汇流
memory_read：从沉积层读取
memory_commit：提交到沉积层
artifact_context：产物引用交接
revision_feedback：修订回流
temporal_gate：时序闸门
event_trigger：事件触发
```

关键字段：

```text
flow_edge_id
source_unit_id
target_unit_id
source_port_id
target_port_id
packet_contract_id
handoff_policy
temporal_policy
visibility_policy
failure_policy
merge_policy
split_policy
contract_bindings
```

当前 `TaskGraphEdgeDefinition` 可以作为 FlowEdge 的兼容保存态，但需要强化 split/merge/packet 语义。

### 3.3 FlowPacket

FlowPacket 是任务流中真正流动的东西。它不应该是任意上下文拼接，而应该是契约化 packet。

FlowPacket 包含：

```text
packet_id
packet_type
source_unit_id
target_unit_id
scope_id
contract_id
payload_summary
payload_refs
artifact_refs
memory_refs
result_refs
visibility
status
created_at
lineage
```

关键原则：

- 默认传引用，不复制正文。
- 模型可见内容由契约和可见性策略决定。
- 未 commit 的候选内容不能伪装成事实。
- packet 可以被沉积到 ledger，也可以被下游消费。

当前 GraphUnit output packet 是 FlowPacket 的雏形。

### 3.4 FlowScope

FlowScope 定义一段流的上下文边界。

类型：

```text
graph_scope
unit_scope
run_scope
child_flow_scope
iteration_scope
batch_scope
repair_scope
human_review_scope
```

关键字段：

```text
scope_id
parent_scope_id
scope_type
visibility_policy
memory_policy
artifact_policy
runtime_state_policy
commit_policy
```

它解决的问题是：流可以分叉，但上下文不能污染。每条支流都知道自己能看什么、能写什么、完成后如何提交。

### 3.5 FlowLedger

FlowLedger 是任务流的账本。它记录流怎么走过。

账本类型：

```text
thread_ledger：长线程接续
issue_ledger：问题闭环
progress_ledger：进度水位
packet_ledger：packet 传输记录
scope_ledger：scope 创建和关闭
merge_ledger：汇流裁决记录
version_ledger：版本锚和快照记录
```

当前系统已有 thread ledger、issue ledger、progress ledger 的基础，但还缺 packet、scope、merge、version 这些面向任务流的账本。

### 3.6 FlowReservoir

FlowReservoir 是沉积层，包含：

- 基准库
- 动态记忆库
- 产物库
- 运行状态库
- 证据库

它的核心不是“存储”，而是“可见性与提交”：

```text
candidate -> reviewed -> committed -> superseded / archived
```

基准库一般只读，是稳定沉积层；动态记忆库是当前运行中的水位变化；产物库保存候选和最终产物；状态库保存运行水位；证据库保存审核和诊断依据。

## 4. 与当前系统的映射

### 4.1 不推倒 TaskGraphDefinition

`TaskGraphDefinition` 仍然是保存态主对象。短期不需要大迁移。

升级方式：

```text
TaskGraphDefinition
  nodes -> FlowUnit adapter
  edges -> FlowEdge adapter
  metadata.timeline_blocks -> GraphRef FlowUnit adapter
  contract_bindings -> FlowContract adapter
```

也就是说，先加一层标准化 adapter，而不是马上改存储大字段。

### 4.2 不推倒 RuntimeSpec

`TaskGraphRuntimeSpec` 仍然是运行编译结果。短期可以扩展 diagnostics 和 nested runtime plans。

升级方式：

```text
RuntimeSpec
  nodes -> runtime FlowUnit
  edges -> runtime FlowEdge
  nested_runtime_plans -> graph_ref FlowUnit runtime plan
  temporal_edges -> FlowGate
  memory_edges -> Reservoir access
```

未来可新增：

```text
flow_units
flow_edges
flow_scopes
flow_packets
flow_gates
```

但第一阶段可以从 `diagnostics.flow_model` 输出，降低迁移风险。

### 4.3 不推倒 ExecutionPackage

ExecutionPackage 是非常正确的方向。它应升级为 FlowPackage。

短期：

```text
execution_package.flow_summary
execution_package.flow_units
execution_package.flow_edges
execution_package.flow_scopes
execution_package.flow_issues
execution_package.flow_trace_index
```

中期：

```text
recursive_flow_plan_index
packet_contract_report
split_merge_report
reservoir_visibility_report
version_anchor_report
```

长期：

ExecutionPackage 就是 FlowPackage 的兼容名称，或者新增 `/flow-packages/task-graphs/{graph_id}`。

### 4.4 不推倒 GraphUnit

GraphUnit 继续存在，但定位调整：

```text
GraphUnit 是 FlowUnit 的 graph_ref 类型。
```

这让系统不再被父子图限制。未来 FlowUnit 也可以引用：

- 已发布任务图。
- 当前图的一段 fragment。
- 动态运行时 split plan。
- 外部工具流。
- 人工工作流。

## 5. 升级路线

### 阶段 1：建立 Flow 标准视图

目标：

在不改变保存态的前提下，新增一层 Flow 标准视图，把当前图、节点、边、GraphUnit、资源、时序统一解释成 FlowUnit / FlowEdge / FlowScope。

新增后端模块建议：

```text
backend/tasks/flow_unit_models.py
backend/tasks/flow_view_builder.py
backend/tasks/flow_diagnostics.py
```

新增前端模块建议：

```text
frontend/src/components/workspace/views/task-system/taskGraphFlowModel.ts
frontend/src/components/workspace/views/task-system/TaskGraphFlowLayerRail.tsx
frontend/src/components/workspace/views/task-system/TaskGraphFlowInspector.tsx
```

第一阶段 FlowView 可以只读：

```text
TaskGraphDefinition -> FlowView
```

完成标准：

- 每个普通节点都能映射为 `atomic_node FlowUnit`。
- 每个 GraphUnit 都能映射为 `graph_ref FlowUnit`。
- 每条边都能映射为 `FlowEdge`。
- memory/artifact/thread/issue ledger 映射为 `reservoir FlowUnit`。
- FlowView 输出问题诊断，但不改变原运行逻辑。

### 阶段 2：FlowPacket 协议统一

目标：

把当前零散的 result、handoff、artifact refs、GraphUnit output packet 统一成 FlowPacket 语义。

后端重点：

- 抽象 packet model。
- GraphUnit output packet 兼容 FlowPacket。
- 普通节点 result 也能生成标准 packet summary。
- 边 handoff 消费 packet，而不是任意上下文拼接。

建议模块：

```text
backend/orchestration/runtime_loop/flow_packet_models.py
backend/orchestration/runtime_loop/flow_packet_builder.py
backend/orchestration/runtime_loop/flow_packet_ledger.py
```

完成标准：

- 普通节点输出、GraphUnit 输出、failure packet 都能转换为 FlowPacket。
- FlowPacket 默认只传 refs 和 summary。
- 每个 packet 有 source、target、scope、contract、visibility、lineage。

### 阶段 3：FlowScope 和隔离升级

目标：

把上下文隔离从节点/图策略升级为 FlowScope 统一协议。

需要明确：

- 父流 scope。
- 子流 scope。
- 动态 split scope。
- merge scope。
- repair scope。
- review scope。

完成标准：

- 每次 GraphUnit child run 创建 child_flow_scope。
- 每个 packet 标注 scope_id。
- 子流无法默认读取父流全部上下文。
- 父流无法默认读取子流中间态。
- commit 后的输出按 visibility policy 释放。

### 阶段 4：Split/Merge 一等化

目标：

让任务流真正可以分叉和汇流，而不是只靠父子图嵌套。

新增概念：

```text
SplitPolicy
  static_targets
  dynamic_target_generation
  max_child_flows
  child_flow_template
  idempotency_key_policy

MergePolicy
  wait_all
  wait_any
  quorum
  coordinator_decides
  quality_gate
  allow_partial_with_issues
```

完成标准：

- 一个 FlowUnit 可以产生多个 child flow requests。
- scheduler 能追踪多个 child flow 状态。
- merge unit 能基于 MergePolicy 汇总 packet。
- failure propagation 能按支流隔离或扩散。

### 阶段 5：FlowPackage 发布包

目标：

把 ExecutionPackage 升级为能解释完整任务流的发布事实源。

新增内容：

```text
flow_units
flow_edges
flow_scopes
flow_packet_contracts
split_merge_report
recursive_graph_plan_index
reservoir_visibility_report
version_anchor_report
flow_trace_index
```

完成标准：

- 发布前能看出每段流怎么走。
- 能看出哪些支流会动态创建。
- 能看出哪些汇流策略被 runtime 支持。
- 能看出哪些能力只是保存态，不能运行。
- 能从任意对象追踪到 runtime、manifest、scheduler、assembly、packet。

### 阶段 6：前端工作台升级为“流域工作台”

目标：

让用户不再只是在画父子图，而是在编辑一个可折叠、可展开、可聚散的任务流。

布局仍保持：

```text
左侧：流域层级 / 图层关系 / 运行水位
中间：任务流画布
右侧：对象编辑台
底部：FlowPackage / 执行包追踪
```

新增对象编辑：

- FlowUnit 编辑：身份、接口、折叠/展开、执行、契约、scope。
- FlowEdge 编辑：packet、交接、分流、汇流、时序、失败传播。
- FlowScope 编辑：可见性、记忆、产物、状态边界。
- FlowReservoir 编辑：沉积、提交、修订、读取。
- FlowGate 编辑：等待、触发、时钟、阶段。

完成标准：

- 用户能把一组节点折叠成 FlowUnit。
- 用户能展开一个 FlowUnit 查看内部流。
- 用户能配置分流和汇流。
- 用户能看到 packet 在图上的流向。
- 用户能看到哪些内容沉积到哪个库。

### 阶段 7：动态运行时裂解

目标：

让节点在运行中可以按契约请求拆分子流。

这一步风险最高，必须在 FlowPacket、FlowScope、FlowLedger 稳定后做。

运行规则：

- 节点不能任意开子任务，必须由 contract 或 skill/tool 授权。
- 子流必须有 template 或 generated plan contract。
- 子流数量必须有上限。
- 子流必须有 idempotency key。
- 子流必须进入 FlowLedger。
- 汇流必须由 MergePolicy 裁决。

完成标准：

- 通用任务可以动态拆分，不是写作专用。
- 动态子流能恢复、续跑、诊断、取消。
- 父流能等待、部分接受或拒绝子流结果。

## 8. 关键工程原则

### 8.1 先标准视图，后改保存态

不要一上来大改 `TaskGraphDefinition`。先做 FlowView adapter，让当前资产能被解释成任务流。

### 8.2 先 shadow package，后 active runtime

动态裂解、汇流、scope gate 都要先进入 shadow execution package 诊断，再进入真实 runtime。

### 8.3 所有动态行为必须契约化

节点不能靠 prompt 自己决定无限拆分。必须有：

- split contract
- max child flows
- child flow template
- merge policy
- idempotency
- scope policy

### 8.4 默认隔离，显式共享

任务像水流，但上下文不能泛滥。

默认：

- 子流隔离。
- 只传 refs 和 summary。
- 未 commit 不进入基准事实。
- 父流只看 committed output。
- 子流只看 input packet 授权内容。

### 8.5 不做领域私门

写作任务、研究任务、代码任务、运营任务都应使用同一套 FlowUnit / FlowPacket / FlowScope / FlowLedger。

领域差异通过：

- 契约模板
- FlowUnit 模板
- Agent profile
- Skill/tool binding
- memory/artifact policy

来表达，不进后端私门。

## 11. 最终目标状态

最终任务系统应达到：

```text
用户看到的是任务流，不只是任务图。
节点可以是水滴，也可以是一条河流的折叠。
边是河道，按契约传递 packet。
时序是水势，决定流动、等待、汇聚和释放。
记忆和产物是沉积层，候选、审核、提交、修订都有协议。
运行包是水文图，能追踪每段流、每个 packet、每个 scope。
任务可以无限拆解，也可以无限汇聚，但每一步都有契约、边界和账本。
```

这才是“可以完成我们想要的一切任务流程”的任务系统。

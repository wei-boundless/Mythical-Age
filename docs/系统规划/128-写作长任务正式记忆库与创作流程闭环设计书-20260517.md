# 写作长任务正式记忆库与创作流程闭环设计书

## 0. 设计结论

写作长任务不应该把若干 `md` 文件当成记忆库本体。

本项目应采用：

```text
正式记忆库 Formal Memory Repository
  负责：分仓、分区、record、版本、提交状态、读写授权、可见性裁决

Artifact 文件
  负责：正文、设定、章纲、审核稿、提交记录的人类可读落盘

TaskGraph 边
  负责：谁能读、谁能写、候选如何提交、何时对下游可见

TimelineLedger
  负责：每次调度、读取、执行、审核、提交发生在什么时序坐标
```

`md` 文件仍然重要，但它是 artifact，不是权限系统、版本系统和记忆裁决系统。

写作流程也必须从“分卷后直接写正文”调整为：

```text
基准库
-> 分卷计划
-> 当前批次细纲
-> 当前批次正文
-> 审核
-> 提交
-> 冻结已写章纲和章节事实
-> 未写部分继续整理
```

这份设计书只定义目标结构和工程边界。具体改造步骤仍以 `127-写作任务正式记忆库与流程闭环优化实施计划-20260517.md` 为执行计划。

## 1. 现有代码事实

### 1.1 已经具备的结构

当前项目已经有正式记忆库的雏形，不需要重新做一个系统。

相关代码：

- `scripts/configure_writing_simple_novel.py`
  - `MEMORY_REPOSITORY_DEFS`
  - `memory_edge()`
  - `memory_read_policy()`
  - `memory_write_policy()`
  - `chapter_outline` 节点和相关契约
- `backend/orchestration/runtime_loop/runtime_assembly_builder.py`
  - 已能从 layered graph 中提取 `memory_reads` / `memory_writes`
  - 已能把边描述成 `repository / collection / selector / version_selector`
- `backend/orchestration/runtime_loop/context_packet_resolver.py`
  - 已有 `resolve_memory_snapshot()`
  - 已有 `ArtifactContextPacket` / `RevisionPacket` / `HandoffPacket` 装配入口
- `backend/memory_system/working_memory_service.py`
  - 已有 `WorkingMemoryItem`
  - 已有 `select_for_node()`
  - 已有 read log

这说明现在要做的是结构收敛和执行补强，而不是推倒重建。

### 1.2 当前不足

当前系统还没有完全形成“正式仓库读写”的闭环：

```text
TaskGraph 上配置了 repository / collection / selector
RuntimeAssembly 能看到这些边
ContextPacketResolver 能生成 memory_snapshot
但 WorkingMemoryService.select_for_node() 仍主要按 kind/scope/status 过滤
```

也就是说：

- 静态图已经能表达正式仓库。
- 运行时还没有完全按 repository selector 做确定性读取。
- `md` artifact 与正式 memory record 之间还缺少稳定映射。
- baseline / mutable / artifact_index / issue_ledger 的边界需要在写作配置和运行时写入中共同强化。

## 2. 核心设计原则

### 2.1 记忆读写必须是边关系

记忆库不是节点随便读取的全局变量。

正确模型是：

```text
repository node --memory_read--> task node
task node --memory_write_candidate--> repository node
commit node --memory_commit--> repository node
```

这也是通用 TaskGraph 编辑器应该坚持的模型：

- 任务节点负责执行职责。
- 资源节点负责表达仓库、账本、产物池。
- 记忆读写权限由边声明。
- selector 和 version_selector 是边上的配置，不是 agent 自己猜。

### 2.2 artifact 与 memory record 分离

一份章节正文可以落成：

```text
artifact_ref:
  output/novel_artifacts/.../chapter_001_010/draft_round_001.md

memory_record:
  repository: memory.writing.artifact_index
  collection: chapter_commits
  record_kind: chapter_commit
  artifact_refs: [artifact_ref]
  status: committed
  visible_after_clock: 153
```

Agent 可以读 artifact 的摘要或全文片段，但是否允许读，必须由 memory edge 和 artifact context edge 决定。

### 2.3 baseline 只保存定稿骨架

baseline 不是草稿历史，不是审核过程，不是多版本世界观合集。

baseline 只能保存：

- 审核通过的世界主干
- 初始大纲主干
- 冻结人物事实
- 冻结关系事实
- 明确的读取边界

后续卷级调整、人物权重、支线顺序、策略优化，应写入 mutable，不应污染 baseline。

### 2.4 写作计划和写作事实必须分开

计划层可以被整理，事实层必须可追溯。

```text
volume_plan
  当前卷的目标、冲突、章节范围

chapter_outline
  当前批次开写前的执行蓝图

chapter_draft
  当前批次正文候选，不是 canon

chapter_review
  审核裁决和返修要求

chapter_commit
  已通过审核、可作为后续事实读取的章节提交
```

未提交的内容不能伪装成已发生事实。

### 2.5 已写章纲冻结，未写章纲可整理

用户提出的关键规则应该成为系统规则：

```text
已开写并通过审核的批次章纲
  -> frozen_chapter_outline
  -> 只作为历史依据读取，不再修改

未开写批次的章纲安排
  -> future_chapter_outline
  -> 可以根据卷后复盘、节奏、人物权重继续整理
```

这能避免长篇写作中两类常见错误：

- 已经发生的章节事实被后续计划反向改写。
- 后续未写计划过早冻结，导致长任务越写越僵。

## 3. 正式记忆库模型

## 3.1 Repository

Repository 是记忆仓库节点，对应 TaskGraph 上的资源节点。

推荐字段：

```json
{
  "repository_id": "memory.writing.baseline",
  "repository_kind": "formal_memory",
  "mutability": "commit_only",
  "schema_id": "schema.writing.simple_novel.memory_record",
  "collections": []
}
```

写作任务中的核心仓库：

```text
memory.writing.baseline
memory.writing.mutable
memory.writing.artifact_index
memory.writing.issue_ledger
```

这几个仓库是写作任务配置，不是通用编辑器硬编码。通用编辑器只需要支持任意 repository node。

## 3.2 Collection

Collection 是仓库内的分区。

建议写作配置：

```text
baseline:
  world
  outline
  characters
  relationships

mutable:
  strategy
  outline_delta
  character_weight
  future_chapter_outline

artifact_index:
  candidates
  reviews
  world_commits
  volume_plans
  chapter_outlines
  chapter_commits
  volume_commits
  delivery

issue_ledger:
  world_issues
  chapter_issues
  volume_issues
  final_issues
```

Collection 的职责是控制“这类记录放在哪里”，不是控制“谁能读”。谁能读由边控制。

## 3.3 Record

正式 memory record 应至少具备这些语义：

```json
{
  "record_id": "memrec:...",
  "repository_id": "memory.writing.artifact_index",
  "collection_id": "chapter_outlines",
  "record_kind": "frozen_chapter_outline",
  "status": "committed",
  "version": 3,
  "source_node_id": "memory_commit_chapter",
  "source_edge_id": "edge.memory_commit.memory_commit_chapter.memory.writing.artifact_index.chapter_outlines",
  "source_receipt_id": "receipt:...",
  "artifact_refs": ["artifact:..."],
  "payload": {},
  "summary": "",
  "timeline": {
    "created_clock_seq": 120,
    "visible_after_clock_seq": 121,
    "scope_path": ["run", "volume:001", "batch:001"]
  }
}
```

这类 record 可以落在 SQLite、JSONL 或现有 WorkingMemory/TaskDurableMemory 的扩展表里。关键不是存储技术，而是语义完整。

## 3.4 Selector

Selector 是 memory edge 的核心，不是 prompt 文案。

建议通用字段：

```json
{
  "collection": "chapter_outlines",
  "record_kinds": ["frozen_chapter_outline"],
  "status_filter": ["committed"],
  "scope_filter": {
    "volume_index": "{volume_index}",
    "batch_index": "{batch_index}"
  },
  "limit": 20,
  "order_by": "created_clock_seq",
  "order": "desc"
}
```

写作任务中最关键的 selector：

```text
chapter_outline 读取：
  baseline world/characters/relationships
  current volume_plan
  previous committed chapter summaries
  issue ledger
  unfrozen future outline candidates

chapter_draft 读取：
  baseline
  mutable
  current volume_plan
  current chapter_outline
  previous chapter summary/review

chapter_review 读取：
  current chapter_outline
  current chapter_draft
  baseline/mutable boundary
  issue ledger

memory_commit_chapter 写入：
  frozen_chapter_outline
  chapter_commit
  chapter_summary
  chapter_file_ref
  issue status receipt
```

## 4. 读写协议

## 4.1 memory_read

`memory_read` 边表示：目标节点可以在某个时序点读取某个 repository/collection 的记录。

它应该生成：

```text
MemorySnapshot
  snapshot_id
  read_edge_ids
  repository_refs
  resolved_record_refs
  resolved_versions
  missing_required_records
  read_receipt_id
```

Agent 看到的不是“你可以随便看记忆库”，而是：

```text
你收到的是本轮 MemorySnapshot。
其中 baseline 是只读约束。
其中 mutable 是允许参考的可变层。
其中 chapter_outline_ref 是本批正文必须遵守的执行蓝图。
缺失必需输入时必须报告阻塞，不能自行猜测。
```

## 4.2 memory_write_candidate

候选写入只表示“节点产物可被登记为候选”，不表示 canon 生效。

典型例子：

```text
chapter_outline --memory_write_candidate--> artifact_index.chapter_outlines
chapter_draft --memory_write_candidate--> artifact_index.candidates
chapter_review --memory_write_candidate--> issue_ledger.chapter_issues
```

候选写入的状态应是：

```text
candidate
proposed
pending_review
```

不能被后续当作 committed fact 读取。

## 4.3 memory_commit

提交写入只应由提交节点或明确的 commit edge 完成。

典型例子：

```text
memory_commit_world
  -> world_commits

baseline_memory_seed
  -> baseline.world / baseline.characters / baseline.relationships

memory_commit_chapter
  -> frozen_chapter_outline
  -> chapter_commit
  -> chapter_summary
  -> chapter_file_ref

extension_commit
  -> mutable.strategy
  -> mutable.outline_delta
  -> mutable.character_weight
```

提交后的 record 才允许进入后续节点的 required memory snapshot。

## 4.4 Receipt

每次写入必须有 receipt。

Receipt 至少说明：

```text
写入成功还是阻塞
写入到哪个 repository/collection
产生了哪些 record/version
何时对下游可见
来源 artifact/ref 是什么
```

没有 receipt 的写入不能被当作事实。

## 5. 时序与版本可见性

正式记忆库必须和时序系统配合。

### 5.1 为什么需要 clock

长任务中会出现：

- 多轮返修
- 循环次数不确定
- 并发或准并发节点
- 已提交内容和候选内容同时存在
- 下游只能看某个时刻以前已提交版本

没有 clock，就无法回答：

```text
这个节点这次看到的是哪一版章纲？
审核是不是看了送审原稿？
正文作者是不是基于本轮审核意见改的？
下一批章节能不能看到上一批提交？
候选记忆是不是提前污染了下游？
```

### 5.2 推荐可见性规则

```text
memory_read:
  读取 visible_after_clock_seq <= current_dispatch_clock_seq 的 committed records

memory_write_candidate:
  写入 candidate records，默认不进入 committed read

memory_commit:
  生成 committed records，visible_after_clock_seq = commit_clock_seq + 1

revision_request:
  必须绑定原稿 artifact_ref、审核 artifact_ref、review receipt、target stage
```

## 6. 写作流程设计

## 6.1 世界观到 baseline

```text
project_brief
-> world_design
-> world_review
-> memory_commit_world
-> baseline_memory_seed
```

设计意图：

- `world_design` 只产候选。
- `world_review` 只裁决，不替作者扩写。
- `memory_commit_world` 只提交已通过世界观引用。
- `baseline_memory_seed` 把通过的世界观沉淀为单一基准骨架。

关键约束：

```text
baseline 只能以 approved_world_spine / world_commit 为世界观定稿来源。
world_candidate 的旧版本和审核过程不得混进 baseline。
```

## 6.2 分卷计划到当前批次细纲

```text
baseline_memory_seed
-> volume_plan
-> chapter_outline
```

`volume_plan` 的职责：

- 按卷拆分目标、冲突、章节范围、爽点节奏。
- 写入 `artifact_index.volume_plans`。

`chapter_outline` 的职责：

- 只处理当前卷、当前批次。
- 把卷计划拆成能直接开写的逐章细纲。
- 读取已提交历史章纲和章节摘要，但不能修改已冻结章纲。
- 为正文作者输出 `chapter_outline_ref`。

## 6.3 当前批次正文和审核

```text
chapter_outline
-> chapter_draft
-> chapter_review
```

`chapter_draft` 必须读取：

- baseline
- mutable
- current volume_plan
- current chapter_outline_ref
- previous chapter summary/review

`chapter_review` 必须读取：

- current chapter_outline_ref
- current chapter_draft_ref
- baseline/mutable boundary
- issue ledger

审核时不能只看正文，也要检查正文是否完成了章纲。

## 6.4 提交与冻结

```text
chapter_review
-> memory_commit_chapter
-> chapter_progress_router
```

`memory_commit_chapter` 做四件事：

1. 把当前批次章纲冻结为 `frozen_chapter_outline`
2. 把通过审核的正文登记为 `chapter_commit`
3. 写入章节摘要和文件引用
4. 更新问题账本状态

提交后，下一批才能读取这些记录。

## 6.5 卷级复盘和可改动库

```text
volume_review
-> volume_commit
-> volume_postmortem
-> world_outline_extension_proposal
-> extension_review
-> extension_commit
```

设计意图：

- 卷级复盘只能基于已经提交的卷事实。
- 可改动库提案只能扩展，不能推翻 baseline。
- `extension_commit` 才能让下一卷读取新的 mutable 记录。

## 6.6 下一卷或最终交付

```text
extension_commit
-> next_volume_router
-> chapter_outline 或 final_assemble
```

继续下一卷时，入口仍然是 `chapter_outline`，不是 `chapter_draft`。

原因：

```text
每个新卷/新批次开写前，都需要先把当前可读记忆转成当前批次执行蓝图。
```

## 7. Prompt 与交接包配合

Agent 不能靠猜自己收到的东西。

每个节点应收到三类说明：

```text
身份：
  你是谁，你负责什么，不负责什么

输入包：
  你收到哪些 MemorySnapshot / ArtifactContextPacket / RevisionPacket / HandoffPacket

输出契约：
  你必须产出什么，哪些是候选，哪些等待提交，哪些不能写成事实
```

典型写法：

```text
你是一名章节作者。
你本轮只负责当前批次允许章号内的完整正文。
你必须先读取当前批次细纲 chapter_outline_ref。
该细纲是本批正文的执行蓝图，不是可随意忽略的参考材料。
如果细纲缺失，你必须报告阻塞，不能直接按分卷计划硬写正文。
```

这种语言应进入 projection/prompt，不应写成 runtime 开发说明。

## 8. 通用编辑器呈现方式

这套设计不能把编辑器做成写作专用。

通用编辑器应这样呈现：

### 8.1 拓扑页

负责结构地图：

- 任务节点
- 审核门
- 循环框
- repository node
- artifact repository
- issue ledger
- memory_read / memory_write_candidate / memory_commit 边

拓扑页只展示结构和跳转入口，不塞复杂 selector 表单。

### 8.2 记忆页

负责资源权限控制：

- repository -> collection
- node x collection matrix
- selector
- version_selector
- receipt visibility
- 缺失策略

### 8.3 职责页

负责节点执行认知包：

- 身份 prompt
- 输入包说明
- 输出契约
- 下游交接

### 8.4 发布页

负责诊断：

- memory_read 缺 selector
- write_candidate 缺 commit path
- revision edge 缺原稿和审核结果
- baseline 写入来源不唯一
- 章节正文缺 current chapter_outline_ref

## 9. 后端落地边界

后续后端实现应遵守这个顺序：

```text
1. 保留现有 WorkingMemoryService，不推倒重写
2. 增加 repository record 语义，或在 WorkingMemoryItem metadata 中稳定表达 repository/collection/record_kind
3. 让 memory_edge.selector 进入 select_for_node()
4. 让 context_packet_resolver.resolve_memory_snapshot() 输出真实 resolved records
5. 让 memory_commit edge 生成 receipt 和 committed record
6. 让 publish/preflight 能检查 memory edge 是否闭环
```

不要先做大范围前端堆功能，也不要只靠 prompt 补洞。

## 10. 验收标准

设计成立的验收标准：

1. 任意节点可以解释自己本轮读取了哪些 repository/collection/record。
2. 任意 memory read 都能追溯到具体 edge、selector、snapshot、read receipt。
3. 任意 committed record 都能追溯到 source node、source artifact、commit edge、receipt、clock。
4. baseline 中不存在候选稿历史和审核过程混写。
5. `chapter_draft` 永远不会在缺少当前 `chapter_outline_ref` 的情况下直接开写。
6. 已提交章节对应章纲进入 `frozen_chapter_outline`，后续只能读不能改。
7. 未写章节的未来安排可以继续整理，但不能改写已发生事实。
8. 编辑器能把 repository node 和 memory edge 显示在拓扑中，同时把精确 selector 配置留在记忆页。

## 11. 最终判断

从现有代码事实看，这个设计是可行的。

原因是：

- TaskGraph 已经支持资源节点和分层边。
- 写作配置脚本已经能声明 repository / collection / memory edge。
- RuntimeAssembly 已能看到 memory edge 描述。
- ContextPacketResolver 已经有 memory snapshot 入口。
- WorkingMemory 已有 item、status、visibility、read log 基础能力。

当前缺的是执行收敛：

```text
把 graph memory edge 上的 repository selector
真正接入后端 memory selection 和 commit receipt
```

这一步做好以后，写作任务就不再是“若干 agent 按顺序产 md”，而是一个可以持续运行、可审计、可返修、可冻结、可继续扩展的长任务生产系统。

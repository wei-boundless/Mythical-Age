## 二、这次重构真正要解决的问题

当前任务系统的问题不是“字段不够”，而是边界和对象层级不够清楚。

主要症状有：

- 任务系统里混入了 Agent 名册管理
- 任务系统里混入了协作拓扑和 handoff 协议管理
- 通用任务和特定任务的定义仍有重叠
- 任务对象里掺入过多编排字段，导致任务本体和执行主体纠缠
- projection 的任务侧绑定与灵魂系统素材管理关系不够清楚
- 前端既像任务页，又像 Agent 装配页，普通用户很难理解

因此，这次重构的目标不是继续补丁，而是把任务系统重新收回到“任务定义与任务治理系统”。

但这里有一个必须提前锁死的原则：

任务系统的管理不是为了堆配置，也不是为了替 Agent 做全部判断。

任务系统存在的意义，是给 Agent 系统提供：

- 清晰任务边界
- 合法选择空间
- 稳定反馈回路
- 可复用的治理结构

也就是说：

`任务系统的管理目的，是让 Agent 更灵活、更稳定地处理任务，而不是把 Agent 处理任务的自由度提前锁死。`

## 三、任务系统的正式总定义

任务系统是平台的任务定义系统、任务治理系统和任务执行请求系统。

它不再是：

- Agent 管理系统
- 协作拓扑管理系统
- prompt 装配系统
- runtime 权限系统

它的本体职责是：

- 识别任务
- 登记任务
- 分类任务
- 管理任务契约
- 管理任务 workflow
- 管理任务 projection 策略引用
- 管理任务 flow contract
- 管理任务能力要求
- 管理任务安全要求
- 管理任务验收要求
- 管理任务执行意图
- 输出正式 `TaskExecutionAssembly`

它回答的问题是：

`这是什么任务，为什么需要正式治理，它需要什么方法、能力、安全边界、验收要求，以及应该以什么执行意图交给编排系统`

## 四、任务系统管理的目的与原则

### 4.1 管理不是“控制一切”

任务系统不应该变成：

- 全字段配置仓库
- 所有决策的上帝视角
- 预先写死全部执行路径的状态机编辑器

如果这样做，会带来四个问题：

1. Agent 丧失运行时判断能力
2. 任务定义过度僵化
3. 普通任务也被迫进入重管理模式
4. 每次小调整都需要大规模改配置

### 4.2 管理的真正目标

任务系统管理的真正目标只有四个：

1. 让系统知道“当前在做什么任务”
2. 让 Agent 知道“当前任务允许怎样处理”
3. 让平台知道“任务是否达到了完成标准”
4. 让同类任务可以被稳定复用、测试和改进

### 4.3 任务系统只应稳定化四类内容

任务系统优先管理：

- 任务边界
- 任务要求
- 任务执行意图
- 任务完成标准

任务系统不应过度管理：

- 每一步具体怎么说
- 每次都必须调用哪些 Agent
- 每一步必须固定使用什么能力
- 所有中间交接都提前硬编码

### 4.4 任务系统给 Agent 的应是“边界 + 选择空间”

错误模型：

`任务系统把一切写死，Agent 只负责照单执行`

正确模型：

`任务系统给出边界、目标、约束和验收，Agent 在边界内灵活求解`

所以任务系统必须同时做到：

- 管得住：防止任务漂移、越界、无验收
- 放得开：允许 Agent 在运行时拆解、选择、派生和调整

## 五、任务系统的硬边界

### 4.1 任务系统不管理 Agent 名册

任务系统不能再直接：

- 创建 Agent
- 删除 Agent
- 编辑 Agent
- 维护 Agent 分类
- 维护 worker blueprint

这些全部归编排系统。

任务系统只允许：

- 引用编排系统中的执行主体策略
- 声明任务需要单 Agent、可派生子 Agent，或必须多 Agent 协作

### 4.2 任务系统不直接装配协作结构

任务系统可以声明：

- 该任务是否需要协作
- 是否需要创意、评审、编写、纠察、验收等阶段
- 是否需要反馈循环

但它不直接管理：

- coordinator 是谁
- participant 是谁
- topology 是什么
- handoff 协议是什么
- merge 规则是什么

这些归编排系统。

### 4.3 任务系统不直接定义单 Agent 身体

任务系统不负责：

- prompt 结构
- runtime profile
- 工具权限
- 资源边界
- memory scope
- output boundary

这些归编排系统。

### 4.4 任务系统不直接管理灵魂素材

任务系统只能引用 projection 策略，不能定义 projection prompt 本体。

灵魂系统负责：

- soul
- projection
- 身份层素材

任务系统负责：

- 某类任务允许哪些 projection
- 默认使用什么 projection 选择策略

## 六、任务系统内部的正式对象模型

任务系统后续只围绕“任务本体”和“任务要求”组织对象。

### 5.1 GeneralTaskProfile

表示通用任务入口配置。

它回答：

`普通用户消息默认如何进入平台主链路`

核心字段建议：

- `profile_id`
- `title`
- `entry_channel`
- `default_workflow_id`
- `default_projection_policy`
- `input_contract_id`
- `output_contract_id`
- `conversation_entry_policy`
- `execution_intent`
- `orchestration_policy_ref`
- `enabled`

说明：

- 通用任务是隐式承接，不是用户显式登记的任务单
- 它不直接管理主 Agent，而是引用默认承接策略

### 5.2 SpecificTaskRecord

表示特定任务正式登记对象。

它回答：

`这一类任务为什么值得稳定登记和反复执行`

核心字段建议：

- `task_id`
- `task_title`
- `task_family`
- `task_mode`
- `description`
- `input_contract_id`
- `output_contract_id`
- `acceptance_profile_id`
- `default_flow_contract_id`
- `default_workflow_id`
- `default_projection_policy`
- `execution_intent`
- `orchestration_policy_ref`
- `task_policy`
- `enabled`

### 5.3 TaskWorkflowRecord

表示任务方法对象。

它回答：

`这个任务按什么方法做`

核心字段建议：

- `workflow_id`
- `title`
- `task_mode`
- `steps`
- `visible_skill_ids`
- `stop_conditions`
- `required_evidence_refs`
- `input_boundary`
- `output_boundary`
- `output_contract_id`
- `prompt`
- `enabled`

说明：

- workflow 是任务方法，不是 Agent prompt
- workflow 必须归任务系统管理

### 5.4 TaskProjectionBinding

表示任务侧投影绑定策略。

它回答：

`这个任务允许使用哪些投影，默认怎么选`

核心字段建议：

- `binding_id`
- `task_id`
- `projection_selection_mode`
- `allowed_projection_ids`
- `default_projection_id`
- `projection_required`
- `notes`

说明：

- projection 素材归灵魂系统
- projection 允许范围和选择策略归任务系统

### 5.5 TaskFlowContractBinding

表示任务侧状态机契约绑定。

它回答：

`这个任务在 runtime 中采用什么执行契约`

核心字段建议：

- `binding_id`
- `task_id`
- `flow_contract_id`
- `override_policy`
- `verification_gate_profile`
- `fallback_policy`

### 5.6 TaskCapabilityRequirementProfile

表示任务能力要求。

它回答：

`完成这个任务至少需要什么能力`

核心字段建议：

- `profile_id`
- `task_id`
- `required_capability_types`
- `preferred_tool_tags`
- `forbidden_capability_tags`
- `evidence_requirements`
- `notes`

### 5.7 TaskMemoryRequestProfile

表示任务记忆请求。

它回答：

`执行这个任务前需要向记忆系统申请哪些上下文`

核心字段建议：

- `profile_id`
- `task_id`
- `requested_memory_layers`
- `requested_topics`
- `memory_priority`
- `writeback_policy`
- `allow_long_term_memory`
- `memory_scope_hint`

### 5.8 TaskSafetyRequirementProfile

表示任务安全要求。

它回答：

`这个任务允许什么写入、执行和验证边界`

核心字段建议：

- `profile_id`
- `task_id`
- `safety_class`
- `write_policy`
- `verification_gate_profile`
- `forbidden_operations`
- `fallback_policy`

### 5.9 TaskAcceptanceProfile

表示任务验收要求。

它回答：

`任务什么时候才算完成`

核心字段建议：

- `acceptance_profile_id`
- `task_id`
- `acceptance_checks`
- `required_artifacts`
- `quality_thresholds`
- `review_required`
- `retry_policy`

### 5.10 TaskExecutionIntent

表示任务的执行意图。

它回答：

`这个任务应该怎样进入编排系统`

正式执行意图建议锁定四类：

- `implicit_general`
- `single_agent_required`
- `subagent_allowed`
- `coordination_required`

核心字段建议：

- `intent_id`
- `task_id`
- `execution_mode`
- `orchestration_policy_ref`
- `coordination_requirement`
- `feedback_loop_required`
- `review_required`
- `notes`

### 5.11 TaskExecutionAssembly

表示任务系统输出给编排系统和 runtime 的正式任务侧请求。

它回答：

`这次执行，任务系统最终决定了哪些任务要求与引用`

核心字段建议：

- `assembly_id`
- `task_id`
- `task_kind`
- `workflow_id`
- `projection_policy_ref`
- `flow_contract_id`
- `memory_request_profile_id`
- `capability_requirement_profile_id`
- `safety_requirement_profile_id`
- `acceptance_profile_id`
- `execution_intent_id`
- `input_contract_id`
- `output_contract_id`
- `task_constraints`

## 七、任务类型与执行架构的双轴模型

这里必须明确：

`通用任务 / 特定任务` 与 `单 Agent / 派生执行 / 协调执行` 不是同一层分类。

它们回答的是两个不同问题：

- 任务类型轴：这个任务是怎么被唤起的
- 执行架构轴：这个任务被唤起后，底层要怎么执行

### 7.1 任务类型轴

任务类型轴只包含两类：

- `通用任务`
- `特定任务`

### 7.2 执行架构轴

执行架构轴只包含三类：

- `单 Agent 执行`
- `子 Agent 派生执行`
- `多 Agent 协调执行`

### 7.3 为什么必须双轴建模

因为同一个任务类型可以映射到不同执行架构。

例如：

- 通用任务 + 单 Agent 执行
- 通用任务 + 子 Agent 派生执行
- 特定任务 + 单 Agent 执行
- 特定任务 + 多 Agent 协调执行

所以“协调”不是和“通用/特定”并列的任务大类，而是执行架构问题。

## 八、通用任务与特定任务的正式定义

### 8.1 通用任务

通用任务是隐式承接任务。

特点：

- 不需要用户显式调用
- 不需要先登记成任务单
- 由 Agent 根据上下文自主承接
- 主要服务主会话入口
- 必要时可升级为特定任务

适用场景：

- 普通问答
- 轻量分析
- 简单整理
- 临时说明

### 8.2 特定任务

特定任务是显式任务化治理对象。

特点：

- 必须先登记
- 必须具备 workflow
- 必须具备能力、安全、验收要求
- 必须可生成稳定 `TaskExecutionAssembly`
- 可被重复执行、测试和验收

适用场景：

- 小游戏开发
- 小说写作
- 复杂重构
- 正式评审与交付

### 8.3 多 Agent 协调执行

多 Agent 协调不是单独的任务类型，而是执行架构。

也就是说：

- 任务本体仍然是 `GeneralTaskProfile` 或 `SpecificTaskRecord`
- 协调只是它在执行架构轴上的一种取值

这能保证：

- 任务定义不和协作装配耦合
- 同一个特定任务可以从单 Agent 版本演进为多 Agent 版本
- 同一个通用任务也可以在特定场景下临时进入派生执行
- 多 Agent 协作成为执行架构能力，而不是任务类型误分类

## 九、Agent 应该怎样参与任务

### 7.1 正确关系

正确链路不是：

`任务系统直接拥有 Agent`

而是：

`任务系统定义任务 -> 编排系统选择执行主体 -> runtime 执行主体运行`

### 7.2 主 Agent 如何参与

主 Agent 的职责是：

- 承接通用任务
- 在特定任务中作为默认兜底执行者
- 在需要时触发子 Agent 或请求协调链路

主 Agent 不是任务系统，也不是编排系统。

### 7.3 子 Agent 如何参与

子 Agent 不应被预先硬编码成固定流水线。

正确模型是：

- 任务系统声明 `subagent_allowed` 或 `coordination_required`
- 编排系统提供 worker blueprint 和协作策略
- 主 Agent 或 coordinator 在 runtime 中按需触发子 Agent

这样才能同时满足：

- 默认隔离上下文
- 按需共享
- 并行执行
- 减少主上下文污染

### 7.4 系统管理 Agent 如何参与

系统管理 Agent 不应被视为任务系统的专属内置角色。

它们仍然是编排系统里的正式执行主体，只是在某些任务模式下更适合承接：

- 健康治理任务
- 记忆治理任务
- 能力治理任务
- 系统配置任务

任务系统只声明“这类任务需要什么能力和治理要求”，不在任务层锁死具体管理 Agent。

## 十、灵魂投影到底怎么绑定

### 8.1 soul 和 projection 的归属

- soul / projection 素材归灵魂系统
- projection 是否允许、怎么选，归任务系统
- projection 进入 prompt 的装配顺序，归编排系统

### 8.2 任务系统只管策略，不管素材

任务系统中不应该保存大段 projection prompt。

任务系统应该只保存：

- 允许哪些 projection
- 是否必须选 projection
- 默认 projection 是谁
- 是按任务默认、workflow 默认，还是用户指定

### 8.3 编排系统负责最终挂载

最终链路应为：

`任务系统提供 projection binding -> 编排系统从灵魂系统取素材 -> 编排系统按 prompt structure 装入运行体`

这样可以避免三个问题：

- 任务系统里混入 prompt 素材
- 灵魂系统里混入 workflow
- 编排系统拿不到统一身份层入口

## 十一、前端任务系统到底应该呈现什么

任务系统前端必须围绕“任务管理”展开，而不是围绕“Agent 管理”展开。

### 9.1 一级页面建议

任务系统前端一级结构建议固定为：

- `任务管理`
- `执行请求`
- `验收与记录`

其中：

- `任务管理`：管理通用任务与特定任务
- `执行请求`：查看某个任务当前会生成怎样的 `TaskExecutionAssembly`
- `验收与记录`：查看验收标准、执行记录引用、历史版本与复用情况

### 9.2 任务管理页应呈现的核心内容

左侧目录：

- 通用任务
- 特定任务
- 按任务族分组

中间主编辑区：

- 任务定义
- workflow
- projection 策略
- 能力要求
- 记忆请求
- 安全要求
- 验收要求
- 执行意图

右侧固定预览区：

- 本次 `TaskExecutionAssembly`
- workflow 标题
- projection 策略
- memory request
- safety requirement
- acceptance profile
- execution intent
- orchestration policy ref

### 9.3 不应该在任务系统前端出现的内容

以下内容不应作为任务系统主交互：

- Agent 创建
- Agent 删除
- Agent 编辑
- coordinator 指派
- participant 指派
- topology 编辑
- handoff 协议编辑
- runtime profile 编辑

这些都应进入编排系统页面。

### 9.4 前端设计原则

任务系统前端必须满足：

- 普通人先看懂“是什么任务”
- 再看懂“怎么做”
- 再看懂“会怎样执行”
- 最后才看到内部引用

因此：

- 中文优先
- 隐藏原始英文 ID
- 少展示底层 JSON
- 多展示任务卡片、任务族、执行摘要和验收条件

## 十二、多 Agent 任务应该怎样设置和管理

### 12.1 多 Agent 任务在任务系统中的设置内容

任务系统只设置“协作需求”，不设置“协作结构细节”。

任务系统里应设置：

- 是否必须协作
- 协作目标是什么
- 必须出现哪些阶段
- 是否必须评审
- 是否需要反馈循环
- 最终由什么标准验收

例如小说任务可以定义：

- 必须经过创意提出
- 必须经过创意审核
- 审核通过后再写作
- 写作后必须内容纠察
- 未通过则回到前一阶段
- 最终达到验收标准才结束

### 12.2 多 Agent 结构由编排系统补全

编排系统再根据这些协作需求决定：

- 谁是 coordinator
- 谁是创意 Agent
- 谁是审核 Agent
- 谁是写作 Agent
- 谁是纠察 Agent
- handoff 怎么传
- merge 怎么做

### 12.3 runtime 中的运行形态

runtime 最终消费的不是一堆页面字段，而是正式运行对象：

- `TaskRun`
- `AgentRun`
- `CoordinationRun`
- `CoordinationNodeRun`
- `AgentHandoffEnvelope`
- `AgentRunResult`
- `CoordinationMergeResult`

因此，任务系统的职责到 `TaskExecutionAssembly` 为止。

## 十三、理论依据与外部架构对照

### 13.1 本地设计原则给出的支持

你的设计原则已经分别说明了两件事：

1. Agent 系统原则强调主 Agent 可按需生成子 Agent、上下文默认隔离、按需共享、并行协作。
2. 任务系统原则强调统一管理异步工作、生命周期、追踪、停止、通知和协作运行。

这说明：

- “任务是什么”与“任务怎么执行”天然不是同一层问题
- 子 Agent 和多 Agent 协作首先是执行组织方式
- 任务系统的价值在于提供统一治理边界，而不是替执行层做所有即时判断

### 13.2 外部成熟架构的一致结论

结合成熟框架的公开设计，可以得到一致结论：

- LangChain / LangGraph 把 supervisor、handoff、多 Agent 视为 multi-agent architecture pattern，而不是 task type。
- Semantic Kernel 官方把 sequential、concurrent、handoff、group chat、magentic 明确称为 orchestration patterns。
- AutoGen 官方把 task 与 team / swarm 分开，task 是任务输入，team 是执行组织形式。
- CrewAI 把 flow 与 crew 分开，flow 是结构，crew 是多 Agent 协作执行单元。
- OpenAI 官方实用指南把 manager 与 handoff 视为 workflow pattern，而不是任务类别。

因此，理论上更稳定的抽象应是：

- 任务类型轴：通用 / 特定
- 执行架构轴：单体 / 派生 / 协调

### 13.3 对本项目的直接约束

这意味着本项目中：

- 协调执行不应再与通用任务、特定任务并列成任务大类
- 任务系统的“管理”应聚焦任务边界、执行意图和验收，而不是把所有协作细节提前写死
- 编排系统只负责把任务系统定义的协作需求映射成具体 Agent 与运行结构

## 十四、任务系统的标准执行链路

建议固定为：

1. 用户输入目标
2. 系统命中通用任务或特定任务
3. 任务系统读取任务定义
4. 任务系统生成 `TaskExecutionAssembly`
5. 编排系统根据 `TaskExecutionAssembly` 选择执行主体和协作结构
6. 编排系统生成 `AgentRuntimeSpec` 与必要的 `CoordinationTopologyProfile`
7. runtime 创建 `TaskRun / AgentRun / CoordinationRun`
8. 主 Agent 或 coordinator 按需触发子 Agent
9. 结果进入验收与反馈循环
10. 满足 `TaskAcceptanceProfile` 后结束

## 十五、实施时必须避免的反模式

- 继续把 Agent CRUD 放在任务系统里
- 在任务对象里硬塞 coordinator / participant / topology 细节
- 把 projection prompt 直接存进任务系统
- 把 workflow 当成灵魂系统素材
- 把 runtime profile 当成任务字段
- 为兼容继续保留错误边界
- 用单个大页面同时承担任务管理和编排管理

## 十六、这份设计书对后续重构的直接约束

后续如果正式实施任务系统重构，应以这几个约束为准：

1. 任务系统只管理任务对象与任务要求。
2. 编排系统是 Agent 管理与协作装配唯一入口。
3. 通用任务是隐式承接，不是显式任务单。
4. 多 Agent 是执行意图，不是单独任务本体。
5. projection 素材归灵魂系统，projection 绑定策略归任务系统，最终挂载归编排系统。
6. 前端任务系统必须围绕任务管理，而不是围绕 Agent 管理。

## 十七、结论

任务系统真正要收回的，不是“少几个字段”，而是重新成为一个干净的任务定义与任务治理系统。

只有把以下四件事彻底拆开，后续重构才不会反复打架：

- 任务是什么
- 由谁执行
- 以什么姿态执行
- 在 runtime 里如何循环执行

其中：

- 任务是什么：任务系统
- 由谁执行：编排系统
- 以什么姿态执行：灵魂系统 + 编排系统
- 在 runtime 里如何循环执行：runtime

这才是后续任务系统正式重构应锁定的目标边界。

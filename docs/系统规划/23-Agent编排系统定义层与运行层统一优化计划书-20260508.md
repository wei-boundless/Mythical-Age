## 1. 当前判断

当前编排系统已经具备三块基础：

```text
AgentDescriptor
  Agent 名册、类别、内置状态、默认灵魂/投影、职责范围。

AgentRuntimeProfile
  任务模式、runtime lane、允许/阻断操作、记忆范围、上下文段、输出契约。

AgentGroup
  子 Agent 组、成员、协调任务默认模板和协议。
```

这三块方向是对的，但与设计原则中的 `AgentDefinition` 还有距离。

设计原则里，一个 Agent 不是只有“名册 + 权限”，而是一个完整定义：

```text
AgentDefinition
  identity
  source / builtin / custom / plugin / managed
  system prompt / role instruction
  tool allowlist / denylist
  model / effort
  permission mode
  memory scope
  isolation
  lifecycle hooks
  context isolation policy
```

当前系统的主要缺口是：**Agent 定义层、运行档案层、任务契约层、前端编辑层还没有形成一个稳定的边界模型。**

---

## 2. 从设计原则提炼出的约束

### 2.1 默认隔离，显式共享

子 Agent 默认不继承主 Agent 的完整上下文。共享必须通过：

```text
RuntimeAssembly.visible_context_sections
ContractSpec.context_visibility_policy
AgentRuntimeProfile.allowed_context_sections
HandoffPacket
```

共同允许。

### 2.2 Prompt 约束和工具约束必须双保险

只读、验证、系统管理等 Agent 不能只靠说明文字约束，必须同时有：

```text
role_instruction / system_prompt
allowed_operations / blocked_operations
approval_policy
runtime lane
```

共同约束。

### 2.3 内置 Agent 和用户 Agent 必须分层

内置 Agent 是系统基础设施，不应被用户任意删除、禁用或改类别。

用户可配置的是：

```text
worker_sub_agent
worker_sub_agent group
worker runtime profile
task scope
contract binding
```

主 Agent 和五个系统管理 Agent 只允许有限字段更新，甚至可以完全锁死。

### 2.4 Agent 组不负责生成 Agent

组只是 membership 和协作模板，不应该承担“生成子 Agent”的职责。

正确模型：

```text
无组子 Agent 池
  -> 加入组
组内成员
  -> 移出组
```

新增子 Agent 应发生在无组子 Agent 池，或任务/契约需要新工作 Agent 时由独立创建流程完成。

## 3. 当前实现问题

### 3.1 `AgentDescriptor` 仍偏名册，不是完整 AgentDefinition

当前字段：

```text
agent_id
agent_name
agent_category
interface_target
builtin
editable
default_soul_id
default_projection_id
task_scope
metadata
```

缺少：

```text
definition_source
role_instruction / system_prompt_ref
prompt_style
tool_policy_mode
model_policy
effort_policy
memory_policy
isolation_policy
background_policy
critical_reminder
```

这些缺口导致前端只能编辑“名册属性”，不能真正配置一个可运行的子 Agent 行为模式。

### 3.2 内置 Agent 锁定策略不一致

后端默认内置 Agent `builtin=True`，但 `editable=True`。前端对主 Agent 和系统管理 Agent 做了只读限制，但后端 `upsert_agent()` 仍允许更新部分内置字段。

这会造成两个问题：

1. 前端看起来锁了，API 其实没有完全锁。
2. “内置但可编辑”的含义不清晰。

建议改成：

```text
builtin=true
editable=false
mutable_fields=["default_projection_id"] 可选
```

或者明确分成：

```text
system_locked
system_configurable
user_managed
```

### 3.3 五个系统管理 Agent 缺完整 RuntimeProfile

当前默认 runtime profile 只有：

```text
agent:0 -> main_interactive_agent
agent:3 -> health_maintainer_agent
```

但内置 Agent 有：

```text
agent:1 权限管理
agent:2 记忆管理
agent:3 健康管理
agent:4 能力管理
agent:5 灵魂管理
```

这会导致系统管理 Agent 在契约编译、runtime assembly 或协调任务里成为“有 Agent 名册但缺运行档案”的半成品。

### 3.4 前端新建 ID 逻辑绕过后端权威

后端已有：

```text
GET /orchestration/agents/next-worker-id
```

但前端 `startBlankAgentDraft()` 自己按当前列表计算最大数字。这会在并发或刷新滞后时产生冲突。

Agent ID 应由后端生成，前端只展示只读 ID。

### 3.5 Agent 组默认值残留具体业务

当前前端默认组：

```text
group.writing.longform_novel_core
长篇小说常态协调组
coordinator_agent_id = agent:20
```

这和“旧具体任务清理、后续统一规范化重建配置”的目标冲突。

建议改成：

```text
group.custom.worker_group_01
新子 Agent 组
coordinator_agent_id = agent:0 或空白后由保存校验
```

### 3.6 编排系统还没有“Agent 类型模板”

设计原则里有 Explore / Plan / Verification / General-purpose 等角色化模式。我们当前只有：

```text
主 Agent
系统管理 Agent
子 Agent
```

但子 Agent 内部还应有“工作模式模板”：

```text
探索型 Agent
规划型 Agent
验证型 Agent
执行型 Agent
整理型 Agent
```

这些模板不等于具体 Agent，而是创建 worker_sub_agent 时可选的配置预设。

## 4. 目标架构

### 4.1 四层模型

```text
AgentDescriptor
  只描述身份、来源、类别、生命周期、默认投影。

AgentDefinitionProfile
  描述角色 prompt、模型策略、工具策略、隔离策略、记忆策略、背景策略。

AgentRuntimeProfile
  描述任务模式、runtime lane、操作权限、上下文段、输出契约、审批和追踪。

AgentGroup
  描述 worker_sub_agent 的 membership、默认拓扑、默认通信协议。
```

### 4.2 Agent 来源分层

建议新增或规范字段：

```text
definition_source:
  system_builtin
  user_created
  task_generated
  plugin_provided
  policy_managed

lifecycle_policy:
  system_locked
  system_configurable
  user_managed
  archived
```

当前可以先映射：

```text
builtin=true -> system_builtin
worker_sub_agent + builtin=false -> user_created
```

### 4.3 内置 Agent 最小闭环

内置 Agent 固定为：

```text
agent:0 主 Agent
agent:1 权限管理 Agent
agent:2 记忆管理 Agent
agent:3 健康管理 Agent
agent:4 能力管理 Agent
agent:5 灵魂管理 Agent
```

每个都必须有：

```text
AgentDescriptor
AgentRuntimeProfile
AgentDefinitionProfile
```

第一版可以先补齐 `AgentRuntimeProfile`，`AgentDefinitionProfile` 后续落地。

### 4.4 子 Agent 创建流程

推荐流程：

```text
选择子 Agent 类型模板
  -> 后端生成 agent_id
  -> 填名称 / 描述
  -> 自动带出 runtime profile 默认值
  -> 保存 AgentDescriptor + AgentRuntimeProfile
  -> 如需加入组，通过组成员管理加入
```

子 Agent 类型模板：

```text
explorer_worker
planner_worker
verification_worker
execution_worker
review_worker
```

模板对应：

```text
默认任务模式
默认 runtime lanes
默认 allowed operations
默认 blocked operations
默认 memory/context policy
默认 output contracts
默认 prompt role
```

## 5. 前端优化方向

### 5.1 页面层级

保持当前分层方向，但命名更贴近结构：

```text
目录层
  主 Agent
  系统管理 Agent
  子 Agent
    有组
    无组

详情层
  名册
  角色定义
  职责覆盖
  运行档案
  权限能力
  上下文输出
  承接资格
```

其中“角色定义”是新增层，用来承载 `AgentDefinitionProfile`。

### 5.2 内置 Agent 前端策略

主 Agent 和系统管理 Agent：

```text
不可删除
不可禁用
不可改类别
不可改 ID
默认不允许改职责说明
可选允许改默认投影 / 默认灵魂
```

子 Agent：

```text
可删除
可禁用
可改名称
可改职责
可改 runtime profile
可加入/移出组
```

### 5.3 组管理策略

组页面只做：

```text
组名
说明
组成员加入/移出
默认拓扑模板
默认通信协议
允许协调任务
```

不在组页面生成新 Agent。

新增子 Agent 放在无组子 Agent 卡片栏内。

## 6. 后端优化阶段

### 阶段 1：锁定内置 Agent 和补齐系统管理 RuntimeProfile

修改：

```text
backend/orchestration/agent_registry.py
backend/orchestration/agent_runtime_registry.py
backend/tests/*
```

工作：

1. 内置 Agent 统一 `editable=false` 或明确 `lifecycle_policy=system_locked`。
2. `upsert_agent()` 后端强制保护内置字段。
3. 补齐 agent:1、agent:2、agent:4、agent:5 的 runtime profile。
4. 测试覆盖内置 Agent 不能删除、不能禁用、不能改类别、不能篡改 ID。

### 阶段 2：后端成为 Agent ID 唯一生成权威

修改：

```text
frontend/src/components/workspace/views/OrchestrationView.tsx
backend/api/orchestration.py
frontend/src/lib/api.ts
```

1. 前端新建 Agent 调用 `/orchestration/agents/next-worker-id`。
2. 后端 upsert 对已存在 ID 冲突给出明确错误。
3. 前端不再自己扫描最大 ID。

### 阶段 3：清理组默认值和成员规则

3. 组成员只允许 `worker_sub_agent`。

## 7. 验证标准

后端：

```powershell
$env:PYTHONPATH='backend'
pytest backend\tests\orchestration_agent_registry_test.py
pytest backend\tests\orchestration_agent_runtime_registry_test.py
pytest backend\tests\orchestration_agent_group_registry_test.py
```

前端：

```powershell
cd frontend
npx tsc --noEmit
npm run build
```

行为验收：

1. 默认只有 `agent:0` 到 `agent:5` 是内置 Agent。
2. 内置 Agent 不能删除、禁用、改类别。
3. 五个系统管理 Agent 都有 runtime profile。
4. 新建子 Agent ID 由后端生成。
5. 组内只能加入子 Agent。
6. 组页面不负责生成子 Agent。
7. 子 Agent 模板能生成对应权限和上下文边界。

---

## 8. 禁止事项

1. 禁止把具体任务模板重新塞进 Agent 组默认值。
2. 禁止让前端绕过后端生成 Agent ID。
3. 禁止只在前端锁内置 Agent，后端不做保护。
4. 禁止系统管理 Agent 没有 runtime profile。
5. 禁止把 prompt 约束当作唯一权限控制。
6. 禁止子 Agent 默认继承完整主会话上下文。
7. 禁止组承担“生成 Agent”的职责。

# 191-任务图批次生命周期总框架报告

日期：2026-05-20

## 1. 总结

本轮把“批次拆分、审核、返修、提交、合并”从编译期预览推进为任务图运行时的一等框架。它不是写作专用逻辑，而是由 `contract_bindings.unit_batch` 与 `contract_bindings.runtime` 驱动的通用任务图能力。

现在一个节点可以通过契约声明“我要处理多少个单位、每批多少个、如何验收、何时提交、如何合并”。运行时会按批次推进同一个节点，节点收到当前批次范围，批次通过后才进入下一批；返修失败会停住节点；全部批次提交后，merge readiness 才会变为 ready。

## 2. 框架边界

本轮已经完成：

- 编译期：任务图节点契约生成 split plan、batch lifecycle plan、merge readiness plan。
- 发布包：执行包展示 split plan、批次生命周期、merge readiness，并进入对象追溯索引。
- 运行时：LangGraph 协调运行态新增 `batch_lifecycle_runtime_state`。
- 节点输入：`stage_prepare` 自动选择 ready batch，并把 `unit_batch_id`、`batch_start_index`、`batch_end_index` 注入节点输入。
- 状态推进：`stage_accept` 根据 accepted / rejected 推进 committed、repair_ready、failed、merge ready。
- 断点续接：批次状态写入 checkpoint / diagnostics，可从运行态恢复。
- 监控：run monitor 返回 `batch_lifecycle`。
- 前端：发布运行页、常驻运行交互窗口展示批次进度与批次明细。

本轮刻意不做：

- 不实现真正多 worker 并行批次。
- 不让 agent 动态生成新 batch。
- 不把“章节、字数、小说质量门”写成后端私门。
- 不把质量审核硬编码到任务图运行时；审核结果仍由通用 accepted / rejected 事件进入状态机。

## 3. 核心架构

### 3.1 契约层

批次能力来自节点的统一契约绑定：

```json
{
  "contract_bindings": {
    "unit_batch": {
      "unit_kind": "chapter",
      "requested_count": 50,
      "range_start": 1
    },
    "runtime": {
      "split_policy": {"mode": "static_batch", "batch_size": 10},
      "batch_acceptance_policy": {"mode": "review_then_commit", "max_repair_rounds": 3},
      "merge_policy": {"mode": "wait_all_committed"}
    }
  }
}
```

这里的 `chapter` 只是 `unit_kind` 的一个值。它也可以是 file、record、case、dataset shard 或任何后续定义的任务单位。

### 3.2 编译层

`compile_task_graph_definition_runtime_spec()` 在编译任务图时生成：

- `split_plans`
- `split_merge_issues`
- 每个 batch 的 execute / review / repair_loop / commit 步骤
- merge readiness 计划

执行包会把这些内容汇总进：

- `runtime_spec.diagnostics`
- `execution_package.split_plans`
- `execution_package.summary`
- `execution_package.object_trace_index`

这让用户可以从图对象追溯到运行计划，而不是只看到一个表单字段。

### 3.3 运行层

新增运行态 authority：

```text
task_system.batch_lifecycle_runtime_state
```

运行态包含：

- `plan_states`
- `batch_states`
- `step_states`
- `merge_states`
- `active_batch_by_node`
- `ready_batch_ids`
- `running_batch_ids`
- `committed_batch_ids`
- `failed_batch_ids`
- `summary`

它由 runtime spec 初始化，写入 checkpoint 与 diagnostics。节点首次执行和恢复执行都必须先经过 `stage_prepare`，因此首批和后续批次都走同一套契约绑定与输入注入路径。

### 3.4 调度语义

当前实现是单运行线程顺序推进：

```text
batch 1 ready -> running -> committed
batch 2 ready -> running -> committed
...
all committed -> merge ready
```

如果批次被 rejected：

```text
running -> repair_ready -> repairing -> ...
```

如果超过 `max_repair_rounds`：

```text
failed batch -> node failed -> downstream blocked
```

这个失败优先级高于“后面还有 planned 批次”，避免继续跑偏。

## 4. 前端工作台变化

前端没有新增孤立页面，而是把批次运行态纳入已有工作台层级：

- 执行包面板展示编译期 split plan 与批次生命周期。
- 发布运行页从 trace 摘要读取批次运行态。
- 常驻运行交互窗口从 run monitor 读取 `batch_lifecycle`。
- 状态展示使用同一解析器 `buildTaskGraphBatchLifecycleSummary()`，避免 trace 和 monitor 各解释一套。

这样用户能看到三层闭环：

- 发布前：这个图会拆成多少批。
- 运行中：当前批次是谁，已提交多少批。
- 出问题时：是哪个批次 failed 或等待返修。

覆盖重点：

- 首个 stage request 带当前批次范围。
- accepted 后同一节点继续下一批。
- 全部提交后 merge readiness ready。
- rejected 在 `max_repair_rounds` 内返修同一批。
- 返修耗尽后批次 failed，节点不继续后续 planned 批。
- run monitor 返回批次生命周期。
- 前端 trace 和 monitor 使用同一批次摘要解析。

## 7. 下一步建议

下一阶段再做真正的并行能力：

- 将 ready batch 集合交给 worker pool。
- 让每个 batch 拥有独立 node execution instance。
- merge 节点只消费 committed batch packet。
- 动态 split 作为新 split policy，而不是让 agent 在 prompt 里自行决定批次。

到这里，任务图已经具备“用户要求 -> 契约声明 -> 编译计划 -> 运行状态 -> 前端监控”的通用批次生命周期闭环。

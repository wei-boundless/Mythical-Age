# 前端工作台结构与性能优化审查报告

日期：2026-05-24

范围：`frontend/src/app`、`frontend/src/components`、`frontend/src/lib`。本报告只审查前端结构、运行效率、交互效率与信息架构，不包含 System Prompt 上下文预算、wire 级静态/动态分块。

## 结论

前端已经形成了“主会话、任务、编排、能力、配置”的工作台框架，任务系统也已经有图层导航和对象页的雏形。但当前运行效率与使用效率的主要瓶颈不是单个样式问题，而是三个结构性问题：

1. 前端模块过大，视图、状态、API、诊断展示、运行控制混在同一层，导致重渲染范围大、修改风险高。
2. 运行监控存在多套订阅与轮询入口，SSE、任务图轮询、编排轮询和组件级秒级计时器没有统一生命周期管理。
3. 工作台界面仍暴露大量底层协议字段和开发态说明，用户需要在业务任务、运行诊断、迁移字段之间来回判断，降低操作效率。

建议按“先运行订阅收敛，再拆状态与 API，再重排工作台层级，最后做视觉与测试收敛”的顺序推进。

## 主要证据

### 1. 大型模块已经成为性能与维护瓶颈

当前前端存在多个超大文件：

| 文件 | 大小 |
| --- | ---: |
| `frontend/src/lib/api.ts` | 约 136KB |
| `frontend/src/components/workspace/views/HealthSystemView.tsx` | 约 132KB |
| `frontend/src/components/workspace/views/TaskSystemView.tsx` | 约 108KB |
| `frontend/src/components/workspace/views/task-system/TaskGraphMemoryArtifactPage.tsx` | 约 74KB |
| `frontend/src/components/workspace/views/ExperimentsView.tsx` | 约 70KB |
| `frontend/src/lib/store/runtime.ts` | 约 64KB |
| `frontend/src/components/workspace/views/PlaygroundView.tsx` | 约 63KB |
| `frontend/src/components/workspace/views/capability-system/CapabilitySystemWorkbench.tsx` | 约 60KB |

风险：

- 任意状态变化容易触发大组件重算。
- 单页同时承载数据加载、编辑、校验、运行控制、诊断展示，难以做按需加载。
- API 类型、请求函数、领域端点集中在一个文件，开发者很难判断一个改动会影响哪些页面。

整改方向：

- 将 `api.ts` 拆成 `client.ts`、`sessions.ts`、`task-system.ts`、`orchestration.ts`、`memory.ts`、`capability.ts`、`health.ts`、`runtime-monitor.ts`。
- 将 `TaskSystemView.tsx` 变成路由/编排壳，只保留当前 layer、graph selection 和 dispatch，业务面板迁移到独立 page/view-model。
- 将 `HealthSystemView.tsx`、`CapabilitySystemWorkbench.tsx`、`MemoryView.tsx` 按“目录页、详情页、诊断抽屉、执行面板”拆分。
- 对 Monaco、实验报告、长列表、运行监控详情等重型面板使用动态加载。

### 2. 运行监控生命周期需要统一

当前运行相关逻辑分散在多个层级：

- `frontend/src/lib/store/runtime.ts` 同时管理任务图轮询、编排轮询、全局 SSE、全局兜底轮询和详情刷新。
- `frontend/src/lib/store/runtime.ts:1450` 创建全局 `EventSource`。
- `frontend/src/lib/store/runtime.ts:1132`、`1296`、`1572` 分别调度任务图、编排、全局监控轮询。
- `frontend/src/components/layout/TaskMonitorDock.tsx:75` 有组件级 1 秒 interval。
- `frontend/src/components/layout/RuntimeMonitorDetailView.tsx:45` 也有组件级 1 秒 interval。
- `frontend/src/app/page.tsx:137`、`158`、`179`、`200`、`235` 在多个工作区分支重复挂载 `TaskGraphRunInteractionDock`。

风险：

- 同一运行状态可能被 SSE、轮询、详情刷新同时更新。
- 页面切换时订阅是否停止依赖多个调用路径，容易出现隐藏页面仍刷新。
- 组件级秒表虽然单次成本不高，但在运行监控、详情页、全局 dock 同时存在时会放大渲染频率。
- 重复挂载 Dock 使入口页维护成本高，也容易在新增工作区时遗漏。

整改方向：

- 新建 `runtime-monitor-controller`，只允许一个控制器管理 SSE、fallback poll、detail refresh、visibility backoff。
- 将“当前页面是否需要全局监控”变成订阅计数或显式 scope，而不是组件各自启动。
- `TaskMonitorDock` 与 `RuntimeMonitorDetailView` 使用统一 `useNowTicker(active)`，并由控制器提供秒级 tick，避免多个 interval。
- 将 `TaskGraphRunInteractionDock` 提升为 `Workspace` 的统一 sibling，只挂载一次，根据当前 view 与状态决定是否显示。
- 页面隐藏、浏览器失焦、无活动任务时降低刷新频率；SSE connected 时兜底轮询不应保持短周期。

### 3. 工作台层级已经有雏形，但仍混入开发态信息

任务系统已有 `TaskSystemShell`、`TaskSystemPages`、`TaskGraphLayerNav`，这是正确方向。但部分页面仍把“业务编辑、运行语义、迁移诊断、底层协议字段”放在同一个主界面。

典型位置：

- `frontend/src/components/workspace/views/task-system/TaskGraphLayerNav.tsx:50`、`59` 使用 `Graph Builder`、`Compiled View` 作为用户可见标题。
- `frontend/src/components/workspace/views/task-system/TaskGraphTimelinePage.tsx:162` 主标题为“生命周期与运行语义”。
- `frontend/src/components/workspace/views/task-system/TaskGraphTimelinePage.tsx:208` 直接显示 `diagnostics`。
- `frontend/src/components/workspace/views/task-system/TaskGraphExecutionPackagePanel.tsx:114` 显示 `contract_bindings.runtime.length_budget`。
- `frontend/src/components/workspace/views/task-system/TaskGraphExecutionPackagePanel.tsx:397` 直接渲染 `runtimeSpec.diagnostics` JSON。
- `frontend/src/components/workspace/views/task-system/TaskGraphRootInspector.tsx:211` 将 `runtime.length_budget` 作为解释文案显示给用户。
- `frontend/src/components/workspace/views/capability-system/CapabilitySystemWorkbench.tsx:1174`、`1213`、`1214`、`1215` 在主界面展示 diagnostics/schema 原始 JSON。

风险：

- 用户会被迫理解内部字段名，而不是完成“设计任务图、配置执行者、发布运行、处理异常”这些真实目标。
- 业务层级、运行层级、诊断层级混在一页，和项目要求的“不同层级用卡片按钮做页面切换，不要混在一页”不完全一致。
- 开发态说明进入界面，会把工具做成调试台，而不是 agent 工作台。

整改方向：

- 将任务图编辑器重排为固定层级：图库入口、图级配置、拓扑编辑、节点/边对象页、执行者装配、资源流、质量门、发布运行、运行监控、诊断抽屉。
- 主页面只显示业务词：如“拓扑编辑”“编译预览”“执行契约”“发布检查”，不直接使用 `runtime_semantics`、`diagnostics`、`legacy_fields`、`authority` 作为一级信息。
- 原始 JSON、schema、authority、legacy 字段统一进入“诊断抽屉”，默认关闭，并标注只读来源。
- 迁移说明只保留在诊断层，不进入主编辑层。

### 4. API 层缺少请求治理

`frontend/src/lib/api.ts:3344` 的 `request<T>` 提供了基础超时与 GET abort 重试，但当前仍有几个问题：

- 超时时间通过 path 特例硬编码，后续会越来越难维护。
- 没有统一 request key、取消策略、并发合并策略。
- 领域端点和类型全部集中，导致 tree-shaking 与开发认知成本都较差。
- 流式接口、普通 JSON 接口、运行监控 SSE 接口混在同一文件。

整改方向：

- 抽出 `api/client.ts`，支持 `timeoutMs`、`signal`、`retryPolicy`、`requestKey`。
- 对 GET 型列表/详情请求建立 `AbortController` 注册表，页面切换时取消旧请求。
- 将 stream/SSE 放入 `api/streams.ts` 或 `runtime-monitor/api.ts`，不和普通 CRUD 混用。
- 使用领域 service 输出窄类型，不让页面直接依赖大而全的后端 payload。

### 5. 确认交互和危险操作需要产品化

多个页面直接使用 `window.confirm`：

- `TaskSystemView.tsx:1376`、`1668`、`1690`
- `HealthSystemView.tsx:778`、`1179`、`1207`
- `MemoryView.tsx:563`
- `CapabilitySystemWorkbench.tsx:570`

风险：

- 原生 confirm 无法展示影响范围、对象关系、不可逆后果。
- 体验割裂，也不利于测试。
- 对删除任务域、删除长期记忆、删除 skill 这类操作，确认信息应该是结构化的。

整改方向：

- 建立全局 `ConfirmDialogProvider`。
- 危险操作必须展示对象名、影响数量、是否可恢复、恢复位置。
- 对批量/不可逆操作增加二次输入或明确的危险等级。
- 使用组件测试覆盖确认、取消、执行中、失败重试。

### 6. 视觉与信息密度需要按 agent 工作台重新收敛

当前 UI 有较强“边界、灵魂、诊断台”的视觉语言，适合展示系统感，但对高频工作台存在几个问题：

- 卡片、边框、说明块过多，主任务和辅助诊断竞争视觉注意力。
- 多处页面把说明文字当副标题，像开发说明，不像用户任务入口。
- `boundary-card` 在任务图页面中大量嵌套使用，容易造成层级噪音。
- 工作台类产品应更偏“安静、密集、可扫描”，而不是每个区域都强调装饰性边界。

整改方向：

- 任务/编排/能力/配置等操作台使用更克制的密集布局；首页和品牌展示可以保留更强风格。
- 一级切换使用卡片式入口，二级对象使用列表/表格，三级详情使用 inspector，不要所有层级都卡片化。
- 长说明改为字段帮助、tooltip、诊断抽屉，不在主标题下写开发说明。
- 保持图编辑器的视觉重点：画布、节点、边、选中对象 inspector，而不是诊断摘要。

## 优先级整改计划

### P0：运行订阅收敛

目标：减少重复请求、重复计时器和隐藏页面刷新。

实施项：

1. 新建统一运行监控控制器，管理 SSE、fallback poll、detail refresh。
2. `TaskMonitorDock`、`RuntimeMonitorDetailView` 改为订阅控制器状态，不直接各自维护 interval。
3. `TaskGraphRunInteractionDock` 在 `Workspace` 中只挂载一次。
4. 加入 document visibility backoff：隐藏时降低频率，恢复时立即刷新一次。
5. 增加测试：切换页面不会创建重复 EventSource；无活动任务时不启动秒级 tick；SSE connected 时 fallback poll 降频。

### P1：API 与状态切片

目标：降低大文件维护成本，减少页面无关状态更新。

实施项：

1. 拆分 `api.ts` 为领域 API 模块。
2. 拆分 `store/runtime.ts`：会话流、任务图详情监控、全局运行监控、编排快照分离。
3. 给大型 selector 建立稳定派生层，避免组件直接对大 payload 做重复计算。
4. 对列表/详情请求统一取消旧请求。
5. 增加测试：请求取消、超时策略、运行监控 reducer、状态切片互不误触发。

### P2：任务系统工作台重排

目标：满足“不同层级用卡片按钮做页面切换，不混在一页”的项目要求。

建议层级：

1. 任务系统首页：任务域、特定任务、任务图库、运行记录、契约库。
2. 任务图库：图列表、图模板、图状态、最近运行。
3. 图编辑器：图级配置、拓扑编辑、节点对象、边对象、执行者装配、资源流、质量门、发布运行。
4. 运行中心：当前运行、历史运行、人工决策、失败恢复。
5. 诊断抽屉：编译视图、runtime semantics、legacy fields、raw diagnostics。

实施项：

1. 将 `Graph Builder` 改为“拓扑编辑”，`Compiled View` 改为“编译预览”。
2. 将 `runtime.length_budget`、`contract_bindings.*`、`diagnostics`、`authority` 从主界面移动到诊断抽屉。
3. `TaskGraphTimelinePage` 拆成生命周期编辑页和运行语义诊断页。
4. `TaskGraphExecutionPackagePanel` 拆成“运行包摘要”和“原始诊断”两个层级。
5. 增加测试：主编辑层不出现 raw diagnostics 字样；诊断抽屉可查看原始 JSON。

### P3：大型视图拆分与按需加载

目标：减少初始 bundle 和页面切换成本。

实施项：

1. `HealthSystemView` 拆为报告、实验、情景库、维护操作四组页面。
2. `CapabilitySystemWorkbench` 拆为能力目录、skill 详情、MCP 服务、工具调用、诊断抽屉。
3. `MemoryView` 拆为会话记忆、长期记忆、trace、治理操作。
4. Monaco、实验详情、运行监控详情、raw JSON viewer 使用 dynamic import。
5. 大列表和时间线使用虚拟列表或分页。

### P4：交互一致性与测试

目标：让危险操作、空状态、错误状态和加载状态统一可测。

实施项：

1. 替换所有 `window.confirm`。
2. 建立统一 Empty/Error/Loading 状态组件。
3. 建立全局 Toast/Notice 规则，避免每个页面自建 notice。
4. 添加 a11y 检查：焦点、按钮 aria-label、键盘路径、窄屏溢出。
5. 使用 Vitest + Testing Library 覆盖关键 view-model 和组件生命周期。

## 建议验收标准

1. 打开主会话、任务系统、能力系统时，全局运行监控最多存在一个 SSE 连接。
2. 同一时刻最多一个全局 fallback poll、一个选中任务详情刷新。
3. 页面隐藏 10 秒后，不再保持秒级刷新。
4. `TaskSystemView.tsx` 降到 35KB 以下，`api.ts` 不再承担领域端点全集。
5. 任务图主编辑页面不直接出现 `diagnostics`、`runtime.length_budget`、`legacy_fields`、`authority` 等内部字段。
6. 删除任务域、删除长期记忆、删除 skill 等危险操作全部使用统一确认弹窗。
7. 任务系统主层、图编辑层、运行监控层、诊断层可通过清晰入口切换，不在同一屏混排。

## 推荐执行顺序

第一阶段先做 P0，因为它直接减少运行时请求、计时器和渲染；风险可控，收益立即可见。

第二阶段做 P1，为后续拆页面做基础。先拆 API 和运行 store，再动大视图。

第三阶段做 P2/P3，对任务系统和能力系统做结构重排。这个阶段会影响用户路径，必须先写详细计划书再实施。

第四阶段做 P4，把确认弹窗、空状态、错误状态、测试补齐，防止结构重排后体验不一致。

# 连续工作会话与 TaskRun 内核化修正计划书

日期：2026-05-30

## 1. 修正结论

`234-TaskRun自然承接与统一运行控制重构计划书` 解决了底层控制能力，但外显体验仍然错误：它仍把“任务阶段”和“普通对话”划成两个用户可感知的模式。

成熟 agent 的外显原则应调整为：

1. 用户始终在和同一个助手对话。
2. TaskRun、执行器、状态机、恢复点是内部运行事实，不是用户叙事。
3. 用户说“继续”“先停一下”“按刚才那个方向改”“不用做了”，应被当前会话自然理解，而不是必须点击任务按钮或进入任务页面。
4. UI 可以展示过程反馈，但不能告诉用户“你现在进入正式任务系统”。
5. 健康系统继续管理任务记录、监控汇总和诊断；会话层只消费当前工作状态，不迁出任务记录管理。

本计划不是删除 TaskRun。相反，TaskRun 应内核化为“连续工作状态机”，对用户隐藏为“当前工作”。

## 2. 当前不自然来源

| 位置 | 现状 | 问题 | 修正方向 |
| --- | --- | --- | --- |
| `backend/harness/loop/agent_loop.py` | `request_task_run` 后生成“正式任务/执行器已启动”类回复 | 用户被显式告知进入另一个系统 | 改为普通助手承接：“我继续处理”“我会按这个方向做” |
| `backend/harness/loop/task_executor.py` | step summary 大量出现 TaskRun、执行器、runtime packet | 过程摘要泄露内部工程结构 | 内部 event 保留，用户投影改写为自然进展 |
| `frontend/src/components/chat/ChatInput.tsx` | 暴露暂停/继续/停止任务按钮 | 把任务控制做成第二套交互入口 | 移除任务专属按钮，保留普通输入和必要的通用停止 |
| `frontend/src/lib/store/runtime.ts` | `SessionActivity` 显示“等待执行器接管”等状态 | 明确割裂任务和对话 | 改为“正在处理 / 已暂停 / 等待确认 / 已完成” |
| `frontend/src/components/chat/RuntimeRunSummary.tsx` | 仍有“正式任务/执行器/TaskRun”投影 | 运行摘要像系统日志 | 摘要只表达用户能理解的进展 |
| `backend/api/orchestration_harness.py` | pause/resume/stop API 是底层能力 | API 本身可保留，但不应成为主体验 | 只供自然语言路由和通用停止使用 |

## 3. 目标权责链

目标链路：

```text
UserUtterance
-> SessionWorkContext
-> ModelTurnDecision
-> WorkContinuationAdmission
-> TaskRunKernel
-> ExecutionLoop
-> UserProgressProjection
-> ConversationReply
-> HealthSystemAggregation
```

职责定义：

| 层 | 职责 | 禁止事项 |
| --- | --- | --- |
| `UserUtterance` | 保存用户原话 | 不用正则把“继续”硬编码为控制命令 |
| `SessionWorkContext` | 汇总当前会话是否有 active work、状态、最近进展、可恢复性 | 不改变任务意图 |
| `ModelTurnDecision` | 根据用户话语和当前工作上下文判断：继续、暂停、停止、补充指令、询问状态、另起新工作、普通回答 | 不跳过模型语义判断 |
| `WorkContinuationAdmission` | 校验模型对当前工作的操作是否允许 | 不生成新的任务目标 |
| `TaskRunKernel` | 持久化工作状态、控制状态、用户补充指令、恢复点 | 不向用户暴露“正式任务系统”叙事 |
| `ExecutionLoop` | 执行、观察、恢复、收尾 | 不重新解释用户意图 |
| `UserProgressProjection` | 把内部事件投影成自然进展 | 不显示 TaskRun、runtime packet、执行器等内部词 |
| `ConversationReply` | 以同一个助手口吻回复用户 | 不说“我已转入正式任务” |
| `HealthSystemAggregation` | 任务记录、监控汇总、诊断、归档 | 不接管会话表达 |

## 4. 目标行为

### 4.1 用户说“继续”

条件：当前会话存在可继续的 active work。

行为：

1. 会话层把 active work 上下文放入模型决策包。
2. 模型判断“继续”指向当前工作，而不是开启新任务。
3. admission 允许后，复用原 TaskRun 续跑。
4. 用户看到自然反馈，例如：

```text
好，我接着处理。
```

不应出现：

```text
正式任务已建立，任务执行器已接管。
```

### 4.2 用户说“先停一下”

行为：

1. 模型判断为暂停当前工作。
2. 后端调用现有协作式 pause 控制。
3. 如果当前模型/工具调用已经在进行，只在安全边界暂停。
4. 用户看到：

```text
好，我先停在这里。后面你说继续，我会从这里接着做。
```

### 4.3 用户说“按刚才那个方向改”

行为：

1. 模型判断为向当前工作追加指令，而不是新任务。
2. 系统把用户补充作为 durable observation 写入当前 TaskRun。
3. 若当前工作可续跑，则从原任务继续；若正在运行，则排队到下一安全边界。
4. executor 下一轮 packet 必须读取这些用户补充指令。

### 4.4 用户问“现在到哪了”

行为：

1. 模型判断为询问当前工作状态。
2. 读取 monitor projection。
3. 直接用自然语言总结，不触发执行。

### 4.5 用户确实开启新主题

行为：

1. 模型判断当前话语不属于 active work。
2. 若旧工作仍在运行，必须明确处理并发语义：继续后台处理、暂停旧工作、或询问用户。
3. 不允许悄悄创建重复 TaskRun。

## 5. 协议修正

新增内部模型决策协议：`ActiveWorkTurnDecision`。

建议动作集合：

```text
continue_active_work
pause_active_work
stop_active_work
append_instruction_to_active_work
answer_about_active_work
ask_user
start_new_work
normal_response
```

要求：

1. 这不是关键词解析器，必须由模型基于当前工作上下文和用户原话判断。
2. 该协议只在当前会话存在 active work 时启用。
3. 没有 active work 时，仍走原 `agent_loop`。
4. `start_new_work` 仍必须进入原有 `request_task_run` admission，不可绕过合同校验。
5. `append_instruction_to_active_work` 必须写入当前 TaskRun observation，不能只存在前端状态。

## 6. 数据与事件设计

### 6.1 ActiveWorkContext

由后端会话层装配，不直接暴露给用户：

```json
{
  "session_id": "",
  "active_work_id": "",
  "task_run_id": "",
  "status": "",
  "control_state": "",
  "user_visible_goal": "",
  "latest_progress": "",
  "resumable": true,
  "running": false,
  "paused": false,
  "queued_user_instruction_count": 0
}
```

### 6.2 用户补充指令 Observation

新增 observation kind：

```json
{
  "observation_type": "user_work_instruction",
  "source": "conversation.user",
  "content": "用户原话",
  "intent": "append_instruction_to_active_work",
  "created_at": 0
}
```

executor 的 `_observations_for_packet()` 必须把它投影进任务执行包。

### 6.3 外显进展投影

内部事件可以继续叫：

- `task_run_executor_scheduled`
- `runtime_invocation_packet_compiled`
- `task_tool_observation_recorded`

但用户投影必须转为：

- “正在准备上下文”
- “正在判断下一步”
- “已读取文件”
- “已完成验证”
- “我先停在这里”

明确反例：

```text
运行装配 runtime packet 已送入模型，系统正在等待 agent 返回下一步动作。
```

这类文案必须只留在内部 event 或调试视图，不得出现在普通会话。用户可见投影应改为：

```text
正在整理上下文，准备继续处理。
```

或在确实需要表达等待时：

```text
正在处理这一步。
```

## 7. 实施阶段

### 阶段 1：废弃外显任务叙事

修改文件：

- `backend/harness/loop/agent_loop.py`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/components/chat/RuntimeRunSummary.tsx`
- `frontend/src/components/chat/ChatInput.tsx`
- `frontend/src/components/chat/ChatPanel.tsx`

内容：

1. 移除“正式任务”“执行器接管”“TaskRun”类用户可见文案。
2. 移除任务专属暂停/继续/停止按钮。
3. 保留通用停止能力，但文案只能是“停止当前操作”，不能是“停止任务”。
4. 运行摘要只保留自然进展，不显示内部协议名。

### 阶段 2：ActiveWorkContext 装配

修改文件：

- `backend/query/runtime.py`
- `backend/harness/runtime/monitor_projection.py`
- 必要时新增 `backend/harness/loop/active_work_context.py`

内容：

1. 从 session live monitor 中得到当前 active work。
2. 只选择用户会话所属的主工作，不把 Graph 内部子节点暴露为当前会话工作。
3. 输出稳定的 internal context，供模型决策使用。

### 阶段 3：ActiveWorkTurnDecision

修改文件：

- `backend/harness/loop/model_action_protocol.py`
- `backend/harness/loop/agent_loop.py`
- `backend/harness/runtime/compiler.py`

内容：

1. 当 active work 存在时，先编译 continuation decision packet。
2. 模型根据用户原话决定 continue/pause/stop/append/status/new/normal。
3. admission 校验动作合法性。
4. 不用硬编码“继续/暂停/停止”关键词作为主路径。

### 阶段 4：当前工作指令写入与续跑

修改文件：

- `backend/harness/loop/task_executor.py`
- `backend/harness/loop/observations.py` 或新增 observation helper

内容：

1. `append_instruction_to_active_work` 写入 durable observation。
2. executor packet 读取用户补充指令。
3. 可续跑时复用原 TaskRun。
4. 正在运行时只排队，不强插正在执行的模型/工具调用。

### 阶段 5：自然回复生成

修改文件：

- `backend/harness/loop/agent_loop.py`
- `frontend/src/lib/store/events.ts`

内容：

1. continue/pause/stop/append/status 都产生自然语言回复。
2. 回复不包含 task_run_id、TaskRun、执行器、runtime packet。
3. 若用户问状态，回复来自 monitor projection，但以助手口吻总结。

### 阶段 6：测试与实测

后端测试：

```powershell
cd backend
python -m pytest tests/query_runtime_runtime_loop_regression.py -q
python -m pytest tests/runtime_monitor_projection_test.py -q
```

新增/调整测试：

1. 用户说“继续”时，复用原 TaskRun，不创建重复任务。
2. 用户说“先停一下”时，进入 pause control。
3. 用户补充指令会写入 observation，并进入下一轮 executor packet。
4. 用户问“现在到哪了”只返回状态，不触发执行。
5. 用户新开主题时不误吞为旧任务继续。
6. 用户可见文案中不出现 TaskRun、正式任务、执行器、runtime packet。

前端测试：

```powershell

npx tsc --noEmit
npx vitest run src/lib/store/runtime.test.ts src/components/chat/RuntimeRunSummary.test.ts
```

真实链路：

1. 固定端口 `3000/8003` 干净启动。
2. 创建一个长工作。
3. 在会话输入“继续”“先停一下”“现在到哪了”“按刚才方向改”。
4. 验证用户始终感觉在同一会话里和同一个助手对话。

## 8. 删除或回退清单

必须删除或改写：

1. `ChatInput` 的任务专属控制按钮。
2. `SessionActivity` 中“等待执行器接管”等文案。
3. `RuntimeRunSummary` 中泄露内部协议的文案。
4. `agent_loop` 中“我会按正式任务推进”“执行器已启动”等承接文案。
5. 测试中把“正式任务已创建”作为用户可见目标的断言。

可以保留：

1. `task_executor.py` 中 pause/resume/stop 控制 helper，作为内部能力。
2. `orchestration_harness.py` 中控制 API，作为内部和调试能力，但不作为主体验入口。
3. `runtime_control` durable diagnostics。
4. 健康系统任务记录和监控汇总职责。

## 9. 自检

| 检查项 | 结论 |
| --- | --- |
| 是否把 TaskRun 降级为内部状态机 | 是 |
| 是否避免用户感知“进入任务系统” | 是 |
| 是否支持自然语言继续/暂停/修改/询问状态 | 是，且由模型基于上下文判断 |
| 是否避免关键词硬编码 | 是，关键词只能作为测试样例，不作为主路由 |
| 是否保留链路安全 | 是，控制仍在 admission 和 executor 安全边界内执行 |
| 是否避免破坏 GraphHarness | 是，active work 只选择会话主工作，不暴露图内部子节点 |
| 是否保持健康系统职责 | 是，任务记录仍归健康系统管理 |
| 是否需要先审阅再实施 | 是。本计划改变会话层语义，应先确认后执行 |

结论：`234` 的底层控制面可作为基础保留，但外显体验方案应被本计划取代。下一步实施前，应先确认本计划，避免继续在“任务系统外壳”上堆自然语言补丁。

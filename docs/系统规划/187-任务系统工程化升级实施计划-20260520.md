# 187-任务系统工程化升级实施计划

日期：2026-05-20

## 1. 纠偏说明

186 号文档把“任务流”理念表达清楚了，但它仍偏概念化。工程实施不能停在类比上。本计划把目标改写为可落地的工程问题：

```text
现有任务系统如何从 TaskGraph + GraphUnit
升级为支持可组合流程片段、分流、汇流、隔离、接续、追踪的通用任务编排系统。
```

本计划不使用隐喻作为设计依据，只保留可实现对象：

- 数据模型
- 编译链路
- 运行语义
- 调度状态
- API
- 前端编辑对象
- 迁移方式
- 验证标准

- `TaskGraphEdgeDefinition` 表达边。
- 图、节点、边都有 `contract_bindings`。
- 旧契约字段仍兼容，但主方向是 `contract_bindings`。

- `ComposableGraphView` 可从节点、边、timeline block 派生可组合对象。

### 2.3 运行态

- `timeline_blocks[].linked_graph_id` 可编译为 `TaskGraphNestedRuntimePlan`。
- nested plan 可生成 `graph_unit` runtime node。
- scheduler 支持 phase、sequence、wait policy、handoff ack、failure propagation。

### 2.4 发布包

- 这是发布前真实执行事实源。

### 2.5 GraphUnit 运行

源码：

- `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py`
- `backend/api/orchestration.py`

- GraphUnit stage 会启动子任务图 run。

- 子图完成后生成 output/failure packet。
- 父图继续时会消费 packet，并防止重复消费。

## 3. 当前工程缺口

### 3.1 缺口 A：缺少统一的“可组合执行单元”模型

现在有：

- task graph node
- composable unit
- graph unit
- nested runtime plan

但它们不是同一个工程对象的不同阶段，而是分散在不同层：

```text
TaskGraphNodeDefinition        保存态
ComposableUnit                 标准视图
TaskGraphNestedRuntimePlan     嵌套运行计划
TaskGraphRuntimeNode           运行节点
```

问题：

- 前端编辑的是 unit，但写回不一定是 unit。
- GraphUnit 看起来是一等对象，但保存态仍是 timeline block。
- 一段流程折叠成节点没有统一协议。

需要新增工程对象：

```text
TaskFlowUnitSpec
```

它作为标准态对象，不立即替代保存态，但成为编译器和前端的统一解释层。

### 3.2 缺口 B：缺少 packet 标准

当前运行结果形式分散：

- 普通节点 task_result。
- handoff packet。
- artifact refs。
- GraphUnit output packet。
- failure packet。

问题：

- 下游节点到底消费什么，缺少统一可视化。
- 上下文污染风险难以统一控制。
- GraphUnit output packet 是特例，不是所有节点的标准输出。

需要新增工程对象：

```text
TaskFlowPacket
```

所有节点输出和 GraphUnit 回交都能转换为 packet summary。

### 3.3 缺口 C：缺少 scope 标准

当前上下文隔离散落在：

- graph context_policy
- node context_visibility_policy
- memory read/write policy
- GraphUnit isolation_policy
- visibility_policy
- runtime diagnostics

问题：

- 子图、动态子任务、循环、修订、人工审核没有统一 scope。
- 运行包难以解释“这个节点到底能看什么”。

需要新增工程对象：

```text
TaskFlowScope
```

它不一定马上进入 active runtime，但必须进入 execution package 诊断。

### 3.4 缺口 D：缺少 split/merge 标准语义

当前系统支持并行、join_policy、barrier、review feedback，但不支持通用动态分流/汇流。

问题：

- 用户要求“一次写多少章”这类任务范围，不应该靠写作私门。
- 节点应能通过通用契约理解 unit_batch / split policy。
- 多个子流程汇总应该由 merge policy，而不是业务硬编码。

需要新增：

```text
TaskSplitPolicy
TaskMergePolicy
```

先作为 contract_bindings 的 runtime/temporal/unit_batch 分区配置，再进入 shadow compiler。

### 3.5 缺口 E：ExecutionPackage 还没有覆盖通用组合结构

当前 execution package 能覆盖 GraphUnit，但还不能覆盖：

- 可折叠流程片段。
- 动态拆分计划。
- split/merge 策略。
- packet contract。
- scope visibility。
- 多层递归循环引用。

需要把 execution package 扩展为：

```text
flow_units
flow_edges
flow_packets
flow_scopes
split_merge_report
composition_issues
```

### 3.6 缺口 F：前端对象编辑台仍按旧对象组织

当前对象编辑台已改进很多，但仍主要围绕：

- graph
- unit
- port_edge
- timeline_block
- nested_runtime

问题：

- “可组合执行单元”没有统一配置面板。
- split/merge、packet、scope、batch 范围还不是自然对象。
- GraphUnit 和 timeline block 仍存在语义重叠。

需要新增前端模型：

```text
TaskFlowUnitInspector
TaskFlowEdgeInspector
TaskFlowPacketPanel
TaskFlowScopePanel
SplitMergePolicyEditor
```

## 4. 目标工程架构

### 4.1 保持 TaskGraphDefinition 为保存态主模型

短期不迁移 JSON 主结构，避免破坏已有任务图。

新增标准化层：

```text
TaskGraphDefinition
  -> TaskGraphLayeredView
  -> TaskFlowView
  -> TaskGraphRuntimeSpec
  -> TaskGraphExecutionPackage
```

### 4.2 新增 TaskFlowView

后端模型：

```python
@dataclass(frozen=True)
class TaskFlowUnitSpec:
    unit_id: str
    unit_type: str
    title: str
    source_kind: str
    source_ref: dict[str, Any]
    interface: dict[str, Any]
    contract_bindings: dict[str, Any]
    runtime_policy: dict[str, Any]
    scope_policy: dict[str, Any]
    split_policy: dict[str, Any]
    merge_policy: dict[str, Any]
    metadata: dict[str, Any]
```

```python
@dataclass(frozen=True)
class TaskFlowEdgeSpec:
    edge_id: str
    edge_type: str
    source_unit_id: str
    target_unit_id: str
    source_port_id: str
    target_port_id: str
    packet_contract_id: str
    contract_bindings: dict[str, Any]
    handoff_policy: dict[str, Any]
    temporal_policy: dict[str, Any]
    split_policy: dict[str, Any]
    merge_policy: dict[str, Any]
    metadata: dict[str, Any]
```

```python
@dataclass(frozen=True)
class TaskFlowView:
    graph_id: str
    units: tuple[TaskFlowUnitSpec, ...]
    edges: tuple[TaskFlowEdgeSpec, ...]
    scopes: tuple[TaskFlowScopeSpec, ...]
    issues: tuple[TaskFlowIssue, ...]
    diagnostics: dict[str, Any]
```

### 4.3 新增 TaskFlowPacket 标准

后端模型：

```python
@dataclass(frozen=True)
class TaskFlowPacket:
    packet_id: str
    packet_type: str
    source_unit_id: str
    target_unit_id: str
    scope_id: str
    contract_id: str
    status: str
    payload_summary: dict[str, Any]
    payload_refs: tuple[str, ...]
    artifact_refs: tuple[str, ...]
    memory_refs: tuple[str, ...]
    result_refs: tuple[str, ...]
    visibility_policy: dict[str, Any]
    lineage: dict[str, Any]
    metadata: dict[str, Any]
```

第一阶段不要求 runtime 全面改造，只做 adapters：

- ordinary task result -> TaskFlowPacket summary
- graph_unit output packet -> TaskFlowPacket
- graph_unit failure packet -> TaskFlowPacket

### 4.4 新增 TaskFlowScope 标准

后端模型：

```python
@dataclass(frozen=True)
class TaskFlowScopeSpec:
    scope_id: str
    parent_scope_id: str
    scope_type: str
    owner_unit_id: str
    visibility_policy: dict[str, Any]
    memory_policy: dict[str, Any]
    artifact_policy: dict[str, Any]
    runtime_state_policy: dict[str, Any]
    commit_policy: dict[str, Any]
```

第一阶段只在 execution package 中输出，不改变 active runtime。

### 4.5 split/merge 进入 contract_bindings

配置位置：

```text
node.contract_bindings.unit_batch
node.contract_bindings.runtime.split_policy
node.contract_bindings.runtime.merge_policy
edge.contract_bindings.temporal
edge.contract_bindings.handoff
edge.contract_bindings.unit_batch
```

示例：

```json
{
  "unit_batch": {
    "unit_kind": "chapter",
    "requested_count": 10,
    "range_source": "user_request",
    "batch_id_template": "chapter_{start}_{end}"
  },
  "runtime": {
    "split_policy": {
      "mode": "static_batch",
      "max_child_units": 10
    },
    "merge_policy": {
      "mode": "wait_all",
      "result_order": "sequence_index"
    }
  }
}
```

这解决“用户要求十章”的通用表达问题：它不是写作私门，而是 unit_batch + split/merge contract。

## 5. 实施计划

### 阶段 1：只读 TaskFlowView

目标：

新增 TaskFlowView，从现有 TaskGraphDefinition 派生，不改变保存和运行。

后端文件：

```text
backend/tasks/task_flow_view_models.py
backend/tasks/task_flow_view_builder.py
backend/tasks/task_flow_view_diagnostics.py
backend/api/tasks.py
```

前端文件：

```text
frontend/src/lib/api.ts
frontend/src/components/workspace/views/task-system/taskGraphFlowView.ts
frontend/src/components/workspace/views/task-system/TaskGraphFlowOverviewPanel.tsx
```

完成标准：

- API 可返回 flow view。
- 每个 graph node 映射为 unit。
- 每个 graph edge 映射为 edge。
- GraphUnit 映射为 `unit_type=graph_ref`。
- memory/artifact/ledger resource 映射为 `unit_type=resource`。
- 不影响现有 runtime spec。

### 阶段 2：ExecutionPackage 集成 FlowView

目标：

把 flow view 合入 execution package。

后端文件：

```text
backend/api/tasks.py
backend/tests/task_graph_execution_package_regression.py
```

完成标准：

- execution package 返回 `flow_view`。
- summary 增加 `flow_unit_count/flow_edge_count/flow_issue_count`。
- object_trace_index 能把 flow unit 追到 runtime node 或 graph unit plan。

### 阶段 3：FlowPacket adapter

目标：

把现有节点结果和 GraphUnit packet 转为标准 packet summary。

后端文件：

```text
backend/orchestration/runtime_loop/task_flow_packet_models.py
backend/orchestration/runtime_loop/task_flow_packet_adapter.py
backend/api/orchestration.py
backend/api/tasks.py
```

完成标准：

- GraphUnit output/failure packet 可转换为 TaskFlowPacket。
- execution package 能展示 packet contract summary。
- 不改变原有 packet 存储和消费逻辑。

### 阶段 4：Scope 诊断

目标：

从现有 context、memory、artifact、GraphUnit isolation policy 派生 TaskFlowScopeSpec。

后端文件：

```text
backend/tasks/task_flow_scope_builder.py
backend/tasks/task_flow_view_builder.py
backend/api/tasks.py
```

完成标准：

- graph scope、node scope、graph_unit child scope 可见。
- execution package 能诊断未声明 scope 或默认过宽的共享策略。
- 前端可以显示当前对象可见性边界。

### 阶段 5：Split/Merge shadow 编译

目标：

支持 split/merge 配置进入编译诊断，但不立即 active 执行。

后端文件：

```text
backend/tasks/task_flow_split_merge.py
backend/tasks/coordination_graph_compiler.py
backend/api/tasks.py
```

前端文件：

```text
TaskGraphNodeUnitInspector.tsx
TaskGraphPortEdgeInspector.tsx
TaskGraphContractBindingInspector.tsx
```

完成标准：

- `contract_bindings.unit_batch` 被读取。
- split policy 和 merge policy 出现在 flow view / execution package。
- 不支持的动态策略标记 unsupported。
- 静态 batch 能被诊断为可编译计划，但不自动执行。

### 阶段 6：前端对象编辑升级

目标：

让用户在图工作台自然配置 flow unit、edge、packet、scope、split/merge。

前端文件：

```text
TaskGraphObjectInspector.tsx
TaskGraphNodeUnitInspector.tsx
TaskGraphGraphUnitInspector.tsx
TaskGraphPortEdgeInspector.tsx
TaskGraphExecutionDock.tsx
TaskGraphExecutionPackagePanel.tsx
```

完成标准：

- GraphUnit 文案改为“子任务图流单元”或“图引用单元”。
- 节点编辑台增加“批次/拆分”分区。
- 边编辑台增加“汇流/分流”分区。
- 执行包面板显示 flow view 和 packet/scope 摘要。

### 阶段 7：active split/merge runtime

目标：

在 shadow 验证稳定后，进入真实运行。

后端文件：

```text
backend/orchestration/runtime_loop/task_run_loop.py
backend/orchestration/runtime_loop/langgraph_coordination_runtime.py
backend/orchestration/runtime_loop/task_graph_scheduler.py
backend/api/orchestration.py
```

完成标准：

- 静态 split 可启动多个 child flow 或 node execution。
- merge gate 可等待全部/任意/人工裁决。
- 动态 split 必须有 contract、上限、幂等键。
- 所有 child flow 都进入 ledger 和 trace。

## 6. 迁移规则

### 6.1 不改已有 JSON 主结构

第一阶段只新增派生视图，不迁移 `task_graphs.json`。

### 6.2 contract_bindings 继续主路径

新增配置必须放入 `contract_bindings`，旧字段只做兼容。

### 6.3 GraphUnit 兼容 timeline_blocks

短期：

```text
timeline_blocks[] -> graph_ref FlowUnit
```

中期可以新增显式保存：

```text
metadata.flow_units[]
```

但只有在 adapter 稳定后再做。

### 6.4 所有新能力先 shadow 后 active

特别是：

- split/merge
- dynamic child flow
- scope gate
- packet consumption gate

必须先进入 execution package 诊断，再进入 runtime。

## 8. 不做的事情

近期不做：

- 不重写 TaskGraphDefinition。
- 不重写 scheduler。
- 不直接启用 dynamic split runtime。
- 不新增写作任务专用后端逻辑。
- 不把概念文案当作 prompt 发给 agent。
- 不在节点保存 provider base_url 或 raw secret。

## 9. 近期执行建议

第一轮只做三个工程闭环：

1. `TaskFlowView` 只读派生层。
2. `ExecutionPackage` 集成 flow summary。
3. 前端执行包面板显示 FlowUnit / FlowEdge / FlowIssue。

这三步完成后，我们才能清楚看见当前任务图在新模型下是什么样子，再决定 split/merge 和 packet/scope 的 active runtime 如何推进。

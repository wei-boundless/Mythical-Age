# 第六阶段灵魂系统正式收口与身份装配重构蓝图

日期：2026-05-04

## 一、文档定位

本文件用于定义灵魂系统的正式收口顺序。

本阶段目标不是继续把灵魂系统当作“prompt 文件仓库 + 若干投影工具函数”，而是把它提升为平台正式的 prompts 身份层系统。

## 二、当前真实问题

当前灵魂系统已经具备：

- `SoulProfile`
- `ProjectionTemplate`
- `ProjectionInstance`
- `SoulRuntimeView`
- `PromptManifest`
- `AgentPromptBundle`

但结构上仍有四个问题：

1. `api/souls.py` 直接管理 registry、projection store、template、instance
2. 编排层直接依赖 `soul.projection.build_soul_runtime_view`
3. 灵魂系统缺少统一 public boundary
4. 注册层、投影素材层、身份装配层还没有正式分段

## 三、第六阶段设计原则

### 3.1 灵魂系统只管理 prompts 身份层

它负责：

- soul
- projection
- identity prompt asset
- prompt manifest

它不负责：

- workflow
- 资源授权
- 记忆供给
- 任务调度

### 3.2 projection 可以改变姿态，不能授予能力

灵魂系统可以影响：

- 角色姿态
- 表达风格
- 注意力组织

不能影响：

- tools 授权
- memory write 权限
- runtime lane

### 3.3 注册、素材、装配必须分层

- 注册层：管理 soul seed / soul profile
- 素材层：管理 projection card / template / instance
- 装配层：输出 `SoulRuntimeView` / `PromptManifest` / `AgentPromptBundle`

### 3.4 外层只能走正式入口

`api`、`orchestration`、`agent registry` 不再直接改 projection store 或 seed registry，而应通过 soul system 正式入口。

## 四、目标结构

建议新增：

- `backend/soul/registry_service.py`
- `backend/soul/projection_service.py`
- `backend/soul/assembly_service.py`
- `backend/soul/facade.py`

分工如下：

- `SoulRegistryService`：管理 soul catalog / profile / active soul / custom soul / managed file
- `SoulProjectionService`：管理 projection card / template / instance
- `SoulAssemblyService`：管理 `SoulRuntimeView` / `PromptManifest` / `AgentPromptBundle`
- `SoulFacade`：灵魂系统正式 public boundary

## 五、实施步骤

### 5.1 先立正式入口

新增 `SoulFacade`，统一代理：

- catalog
- switch
- custom soul
- projection card
- template
- instance preview
- runtime view assembly

### 5.2 收 API 穿透

把 `api/souls.py` 改为依赖 `SoulFacade`，不再直接调用：

- `projection_store`
- `ProjectionTemplateRegistry`
- `ProjectionInstanceRegistry`
- `SoulRegistry`

### 5.3 收编排层穿透

把 `orchestration/assembly_builder.py` 改为通过 `SoulFacade` 构建 runtime view。

### 5.4 保持边界不越界

灵魂系统运行时装配继续只消费：

- task prompt contract
- projection requirement
- skill/resource 可见视图

但不拿：

- 资源授权裁决权
- 记忆写回权
- 任务方法定义权

## 六、完成标准

完成后应满足：

1. `soul` 成为正式系统入口
2. `api/souls.py` 不再直连多个底层模块
3. 编排层通过正式入口消费身份装配结果
4. 投影继续只能改变姿态，不能授予能力
5. 回归测试通过

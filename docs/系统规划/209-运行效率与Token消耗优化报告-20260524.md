# 209-运行效率与 Token 消耗优化报告

日期：2026-05-24

状态：审查报告，待实施

适用范围：

- 主对话运行链路：`backend/query/runtime.py`、`backend/runtime/unit_runtime/loop.py`
- 模型调用网关：`backend/runtime/model_gateway/*`
- 运行时上下文装配：`backend/runtime/shared/context_manager.py`
- Memory 与 Context 包装链路：`backend/agent_system/assembly/runtime_chain.py`、`backend/memory_system/*`
- 流式恢复与模型 sidecar：`backend/runtime/model_gateway/model_response.py`、`backend/agent_runtime/understanding/*`

本报告按用户要求暂不处理以下两项：

1. 不调整上下文预算档位、`deepseek_1m` 默认值、输出 token 上限等预算配置。
2. 不实施 System Prompt 的 wire 级静态/动态分块改造。

## 1. 目标

本报告解决的是项目运行过程中重复建连、重复模型调用、重复上下文构建、重复 prompt 文本、恢复策略放大成本等问题。

目标不是通过降低能力来省 token，而是在保持 agent 正常理解、规划、执行和恢复能力的前提下，让每轮运行满足五个条件：

1. 模型连接可复用：同一模型配置不应每次请求都重新创建并关闭客户端。
2. 上下文进入模型前先治理：历史消息、压缩摘要、热事实和近期消息要有明确装配顺序。
3. 每轮 expensive view 只构建一次：Memory runtime view、context package、decision sidecar 不应在同一轮重复计算。
4. 恢复策略不放大成本：流式失败恢复不能默认把同一大 prompt 再完整发一次。
5. prompt 内容不重复：同一协议或约束只能有一个权威来源。

## 2. 设计依据

来源：`docs/设计原则`

### 2.1 启动与运行复用原则

`02-启动优化.md` 强调：

- 快速路径只加载必要模块。
- 预连接和昂贵初始化应提前或复用。
- memoize 用来避免重复初始化。

落到当前后端，就是模型 runtime 不应在每次 `invoke_messages`、`invoke_messages_with_tools`、`astream_messages` 中重复创建 HTTP client 并立刻关闭。

### 2.2 上下文管理原则

`06-上下文管理.md` 强调：

- 历史上下文必须有 token 压力检测。
- 旧工具结果和旧消息需要 microcompact/full compact。
- 压缩不是用户入口的诊断功能，而应进入主运行链路。

当前项目已有 compaction 相关接口，但主 chat 入口没有真正使用。

### 2.3 Thinking 与 side query 成本原则

`08-Thinking-与推理控制.md` 强调：

- side query、classifier、辅助调用应尽量 cheap。
- 深度推理只给真正需要的任务使用。

当前 `model_turn_decision` 每轮都通过 sidecar 模型调用完成，应保留模型裁决权，但需要缓存、轻量模型 profile 或严格 fast lane。

### 2.4 Agent 与 Memory 隔离原则

`12-Agent-系统.md`、`14-任务系统.md`、`23-Memory系统.md` 强调：

- 子任务中间输出不应污染主上下文。
- 主线程应持有 summary/ref，而不是原始长输出。
- Memory recall 应选择性注入，不能每轮全量注入。

当前项目已经有 `task_summary_refs`、`bundle_summary_refs`、`artifact_refs`、`followup_mode=summary_ref/binding_ref` 等结构，应继续把主链路改成“ref/summary 优先，raw 内容按需展开”。

## 3. 当前问题链路

### 3.1 模型客户端每次调用都新建并关闭

源码位置：

- `backend/runtime/model_gateway/model_runtime.py:206`
- `backend/runtime/model_gateway/model_runtime.py:248`
- `backend/runtime/model_gateway/model_runtime.py:302`
- `backend/runtime/model_gateway/model_runtime.py:348`
- `backend/runtime/model_gateway/model_runtime.py:604`

当前行为：

```text
每次模型调用
  -> _candidate_specs()
  -> 每个 attempt 调 _build_chat_model_for_spec()
  -> ainvoke/astream
  -> finally _aclose_chat_model()
```

问题：

- HTTP 连接池无法复用。
- TCP/TLS 建连、认证客户端初始化、SDK client 初始化都重复发生。
- retry 时也会新建 client。
- 测试中已有 `test_model_runtime_closes_model_clients_after_invoke` 和 `test_model_runtime_closes_model_clients_after_stream` 明确断言每次调用关闭，这说明当前测试保护的是旧生命周期，需要结构性更新。

优化方向：

- 引入 `ModelClientPool` 或 `ReusableChatModelRegistry`。
- key 使用稳定的模型规格：`provider/model/base_url/api_key_fingerprint/timeout/max_tokens/thinking_mode/reasoning_effort/temperature`。
- `ModelRuntime` 调用时从 pool 获取 model。
- retry 复用同一 spec 的 model，跨 provider fallback 切换到另一个 pooled model。
- 在 FastAPI lifespan shutdown 时统一关闭 pool。
- 配置变更时使 pool generation 失效并关闭旧 generation。

不允许的做法：

- 不要在旧的 finally 里继续每次关闭，只在外层再包一层 cache。
- 不要用全局裸 dict 存 client 且没有 shutdown。
- 不要保留旧关闭测试不改而绕过断言。

### 3.2 主 chat 路径禁用了 compressed context

源码位置：

- `backend/query/runtime.py:136`
- `backend/query/runtime.py:138`
- `backend/query/runtime.py:140`
- `backend/sessions/store.py:131`
- `backend/runtime/shared/context_manager.py:107`
- `backend/runtime/shared/context_manager.py:144`

当前行为：

```text
QueryRuntime.astream
  -> load_session_for_agent(include_compressed_context=False)
  -> TaskRunLoop.run_single_agent_stream(history=raw_history)
  -> RuntimeContextManager.prepare_model_context()
  -> system + full normalized_history + pending user
```

问题：

- `sessions/store.py` 已经支持 compressed context，但主路径关闭了。
- `api/tokens.py` 会调用 `inspect_memory_context_compaction` 做诊断，但这不是模型调用前的执行路径。
- `context_system.packaging.controller.ContextController.compact_history` 存在，但没有接入主运行链路。
- 长会话下，模型输入会越来越依赖原始历史，而不是压缩后的会话工作记忆。

优化方向：

- 在 `QueryRuntime.astream` 或 `TaskRunLoop.run_single_agent_stream` 早期增加 `RuntimeHistoryAssembler`。
- 输入：session record、raw history、compressed_context、当前 user message、memory view、context policy。
- 输出：`HistoryAssemblyResult`，包含：
  - `model_history`: 实际进入模型的 history。
  - `summary_message`: 可选 compressed context。
  - `recent_messages`: 最近 N 轮原文。
  - `hot_truth_messages`: 当前任务仍需保留的关键工具/产物摘要。
  - `token_accounting`: raw tokens、assembled tokens、dropped tokens、strategy。
- `RuntimeContextManager.prepare_model_context` 不再直接吃 raw history，而是吃已装配 history。

建议固定装配顺序：

```text
system/runtime prompt
  -> compressed session summary
  -> durable/hot memory refs
  -> recent user/assistant turns
  -> current user message
```

不允许的做法：

- 不要简单把 `include_compressed_context=False` 改成 `True` 就结束。
- 不要同时注入压缩摘要和完整旧历史，造成双倍 token。
- 不要让压缩摘要覆盖当前轮事实；restore 是候选，当前轮裁决仍由 runtime decision 和真实工具结果负责。

### 3.3 Memory runtime view 同一轮重复构建

源码位置：

- `backend/agent_system/assembly/runtime_chain.py:69`
- `backend/agent_system/assembly/runtime_chain.py:74`
- `backend/agent_system/assembly/runtime_chain.py:92`
- `backend/agent_system/assembly/runtime_chain.py:93`
- `backend/agent_system/assembly/runtime_chain.py:189`
- `backend/agent_system/assembly/runtime_chain.py:197`
- `backend/memory_system/bundle_service.py:169`
- `backend/memory_system/bundle_service.py:188`

当前行为：

```text
build_runtime()
  -> build_memory_runtime_view_payload(initial)
  -> 如需 state recall，再 build_memory_runtime_view_payload(state)
  -> task bundle 后，再 build_memory_runtime_view_payload(final profile)
  -> build_context_policy_result()
      -> MemoryBundleService.build_memory_context_package_result()
          -> build_memory_runtime_view()
```

问题：

- 同一用户轮次可能多次读取 memory、构建候选、生成 refs。
- context policy 内部又重建一次 memory view，可能和前面 view 不一致。
- 重复 IO 和 token 估算会放大延迟。

优化方向：

- 引入 `TurnRuntimeCache` 或 `RuntimeAssemblyScratchpad`，生命周期只覆盖当前 turn。
- cache key 至少包含：
  - `session_id`
  - `task_id`
  - `agent_id`
  - `message_hash`
  - `memory_request_profile_hash`
  - `retrieval_result_hash`
- `build_context_policy_result` 支持传入已经构建好的 `memory_runtime_view`。
- `MemoryBundleService.build_memory_context_package_result` 支持 `memory_view` 参数；传入时不得内部重建。
- 所有 diagnostics 记录同一个 `memory_runtime_view_ref`。

不允许的做法：

- 不要只在某个函数上加 `lru_cache`，因为 memory request profile 和 session 状态会变。
- 不要跨 turn 复用 memory runtime view，除非有明确版本号和失效机制。

### 3.4 model-turn-decision sidecar 每轮固定多一次模型调用

源码位置：

- `backend/runtime/unit_runtime/loop.py:1304`
- `backend/runtime/unit_runtime/loop.py:1323`
- `backend/runtime/unit_runtime/loop.py:1327`
- `backend/agent_runtime/understanding/model_turn_decision_invoker.py:31`
- `backend/runtime/model_gateway/structured_sidecar.py:25`
- `backend/runtime/model_gateway/structured_sidecar.py:42`

当前行为：

```text
每轮进入 run_single_agent_stream
  -> build_request_facts
  -> build_boundary_policy
  -> build_context_candidates
  -> invoke_model_turn_decision(sidecar model call)
  -> accepted 后才继续 assembly
```

优点：

- 当前轮是否执行、是否阻塞、是否需要澄清，由模型裁决，符合项目对 agent 理解层的要求。
- sidecar 不可用时 fail closed，避免无裁决执行。

问题：

- 每轮必定增加一次模型请求。
- request payload 把 request/schema/role prompt 包成 JSON，输入并不小。
- retry、恢复、重复提交同一轮时可能重复花费。

优化方向：

第一层：decision cache。

- 新增 `ModelTurnDecisionCache`，生命周期为单 turn + 可选 session 短 TTL。
- key：
  - `session_id`
  - `turn_id`
  - `user_message_hash`
  - `request_facts_hash`
  - `boundary_policy_hash`
  - `context_candidates_hash`
  - `model_spec_hash`
- 命中时直接返回 decision 和 diagnostics，标记 `sidecar_status=cached_accepted`。
- 缓存只允许复用 accepted decision；blocked decision 默认不跨调用复用，避免短暂模型失败把当前轮锁死。

第二层：sidecar 轻量 profile。

- sidecar 默认使用 thinking disabled、较低 max output tokens、较短 timeout。
- 如果主模型规格开启 thinking，sidecar 不自动继承，除非任务明确需要复杂裁决。

第三层：严格 fast lane。

只允许以下场景跳过 sidecar：

- 用户消息为空白、纯问候、纯停止、纯 UI 状态查询等无工具无 memory 的请求。
- boundary policy 明确禁止外部动作。
- request facts 没有文件、URL、命令、任务继续、恢复、编辑、运行、浏览器等目标。

fast lane 只能生成 `answer_only/conversation` 类型 decision，不能生成编辑、运行、搜索、委派类 decision。

不允许的做法：

- 不要用正则分类替代模型裁决处理复杂请求。
- 不要让 sidecar 失败后启发式继续执行工具。
- 不要把开发说明写进 agent prompt；role prompt 仍要写成“你是当前轮请求判断者”这类面向 agent 的职责指令。

源码位置：

- `backend/runtime/shared/context_manager.py:386`
- `backend/tests/prompt_tool_visibility_regression.py:26`

### 3.6 流式恢复默认可能双倍消耗同一大 prompt

源码位置：

- `backend/runtime/model_gateway/model_response.py:139`
- `backend/runtime/model_gateway/model_response.py:155`
- `backend/runtime/model_gateway/model_response.py:517`
- `backend/runtime/model_gateway/model_response.py:642`
- `backend/runtime/unit_runtime/runtime_policy.py:40`

当前行为：

```text
stream_enabled && retryable error && recovery enabled
  -> emit stream_recovery started
  -> _invoke_non_stream_after_stream_error(model_messages)
  -> 同一 prompt 再发一次非流式请求
```

问题：

- 大上下文场景下，失败一次就可能多付一次完整 input token。
- 如果已经有 `partial_delta_count > 0`，用户和执行层可能已经看到部分内容，再自动 fallback 容易造成重复语义或状态混乱。
- `_stream_recovery_enabled` 默认 true。

优化方向：

- recovery policy 改为更保守：
  - `partial_delta_count == 0`：允许自动 fallback。
  - `partial_delta_count > 0`：默认 fail closed，要求断点重跑或由上层 recovery controller 决定。
  - tool call 场景：只有未发出任何 tool call 且无副作用时可 fallback。
- `model_stream_policy_from_task_execution_assembly` 默认不应无条件开启 `fallback_to_non_stream_on_error`，应根据任务类型和幂等性决定。
- 增加 diagnostics：
  - `input_token_estimate`
  - `partial_delta_count`
  - `fallback_suppressed_reason`
  - `recovery_policy_source`

不允许的做法：

- 不要把所有恢复关闭；需要保留零输出网络抖动场景的自动恢复。
- 不要在已产生工具副作用后自动重放同一模型 turn。

### 3.7 子任务和工具结果应继续收敛为 ref/summary

源码位置：

- `backend/runtime/execution_engine/observation_projection.py:78`
- `backend/runtime/execution_engine/observation_projection.py:207`
- `backend/runtime/unit_runtime/loop.py:2199`
- `backend/runtime/unit_runtime/loop.py:2866`
- `backend/runtime/coordination_runtime/runtime.py:2360`

当前已有基础：

- `observation_projection` 已经能把 PDF、表格、委派任务等结果投影为 `summary_ref` 或 `binding_ref`。
- 主循环已有 `task_summary_refs`、`bundle_summary_refs`。
- coordination runtime 已有 `context_packets` 和 `artifact_context_packet`。

问题：

- 这些结构已经存在，但主模型上下文仍有不少路径会带入 raw 历史或长结果。
- 需要把“summary/ref 优先”从局部工具投影提升为主上下文装配原则。

优化方向：

- 对所有工具观察结果建立统一 `ObservationProjectionPolicy`：
  - 大于阈值的结果默认只进入 summary/ref。
  - 当前 turn 必需的短结果可以进入 recent hot truth。
  - artifact 内容只在后续节点显式要求时展开。
- `RuntimeHistoryAssembler` 读取 `task_summary_refs` 和 `bundle_summary_refs`，优先注入摘要和 ref。
- 对调试、diagnostics、trace 保留 raw，但模型可见上下文默认不吃 raw。

不允许的做法：

- 不要删除 raw trace；调试需要。
- 不要把 raw trace 当模型上下文兜底。

## 4. 推荐实施顺序

### 阶段 A：低风险成本止血

目标：

- 减少不必要重复 token 和重复请求。
- 不改变主行为语义。

实施项：

1. 删除重复“用户可见回执协议”来源，只保留一个 canonical section。
2. 收紧流式恢复策略，避免 partial output 后默认二次完整请求。
3. 为 `model_turn_decision` 增加单 turn cache，先覆盖 retry/resume 内重复调用。

涉及文件：

- `backend/prompting/builder.py`
- `backend/runtime/shared/context_manager.py`
- `backend/runtime/model_gateway/model_response.py`
- `backend/runtime/unit_runtime/runtime_policy.py`
- `backend/runtime/unit_runtime/loop.py`
- `backend/agent_runtime/understanding/model_turn_decision_invoker.py`
- 相关 tests

完成标准：

- system prompt 中“用户可见回执协议”只出现一次。
- `partial_delta_count > 0` 时默认不走非流式 fallback。
- 同一 turn 重复调用 `invoke_model_turn_decision` 时只发生一次 sidecar 模型调用。

### 阶段 B：模型客户端生命周期重构

目标：

- 模型客户端按 spec 复用，app shutdown 统一释放。

实施项：

1. 新增 `backend/runtime/model_gateway/client_pool.py`。
2. `ModelRuntime` 初始化时持有 pool。
3. 替换每次 `_build_chat_model_for_spec` 的直接创建逻辑。
4. 给 `ModelRuntime` 增加 `async close()`。
5. `AppRuntime` 增加 `async shutdown()`。
6. `bootstrap/lifespan.py` 在 `yield` 后调用 shutdown。
7. 更新旧测试，不再断言每次调用关闭，改为断言 shutdown 关闭。

涉及文件：

- `backend/runtime/model_gateway/model_runtime.py`
- `backend/runtime/model_gateway/client_pool.py`
- `backend/bootstrap/app_runtime.py`
- `backend/bootstrap/lifespan.py`
- `backend/tests/model_runtime_regression.py`

完成标准：

- 同一 spec 连续两次调用复用同一 closeable client。
- provider fallback 使用不同 spec client。
- app shutdown 后所有 pooled client 关闭。
- 配置变化或 generation 变化后旧 client 被关闭，新请求使用新 client。

### 阶段 C：Memory runtime view 单轮缓存

目标：

- 同一 turn 内 memory view 和 context package 只构建一次或按 profile 明确复用。

实施项：

1. 新增 `TurnRuntimeCache` 数据结构。
2. `AgentRuntimeChain.build_runtime` 创建并传递 cache。
3. `build_memory_runtime_view_payload` 先查 cache。
4. `build_context_policy_result` 接收已构建 memory view。
5. `MemoryBundleService.build_memory_context_package_result` 支持传入 `memory_view`。
6. diagnostics 输出 cache hit/miss 和 memory view ref。

涉及文件：

- `backend/agent_system/assembly/runtime_chain.py`
- `backend/memory_system/bundle_service.py`
- `backend/memory_system/facade.py`
- `backend/runtime/unit_runtime/loop.py`
- 相关 memory/context tests

完成标准：

- 普通 turn 只构建一次 final memory runtime view。
- state recall 场景最多构建 initial/state/final 三类有明确 profile 的 view，context package 不再额外重建。
- context policy 和 runtime context 使用同一个 `memory_runtime_view_ref`。

### 阶段 D：主链路 HistoryAssembly 接入

目标：

- 历史上下文进入模型前完成压缩、摘要、近期消息和热事实装配。

实施项：

1. 新增 `backend/runtime/shared/history_assembler.py`。
2. 定义 `HistoryAssemblyResult`。
3. 接入 `QueryRuntime.astream` 或 `TaskRunLoop.run_single_agent_stream`。
4. `RuntimeContextManager.prepare_model_context` 改为接收 assembled history。
5. 将 `inspect_memory_context_compaction` 从诊断逻辑升级为真实 compaction/projection 逻辑，或替换为 `ContextController.compact_history` 的运行时适配器。
6. 增加 token accounting event：`history_context_assembled`。

涉及文件：

- `backend/query/runtime.py`
- `backend/runtime/unit_runtime/loop.py`
- `backend/runtime/shared/context_manager.py`
- `backend/runtime/shared/history_assembler.py`
- `backend/context_system/packaging/controller.py`
- `backend/memory_system/bundle_service.py`
- `backend/api/tokens.py`
- 相关 tests

完成标准：

- 长历史 session 的模型消息不再包含全量旧历史。
- compressed context 和 recent turns 不重复表达同一内容。
- 当前 turn 的 user message 永远保留原文。
- 工具热事实和 artifact refs 可进入模型，但 raw 长输出默认不进入。

## 5. 固定执行流

优化后的主运行链路应固定为：

```text
QueryRuntime.astream
  -> 读取 session record
  -> 提交当前用户消息
  -> 构建 request facts / boundary policy
  -> model-turn-decision cache 或 sidecar
  -> AgentRuntimeChain.build_runtime
      -> TurnRuntimeCache
      -> MemoryRuntimeView
      -> ContextPolicyResult
  -> RuntimeHistoryAssembler
      -> compressed summary
      -> summary/ref hot truth
      -> recent messages
  -> RuntimeContextManager.prepare_model_context
  -> RuntimeExecutionEngine.stream_model_turn
  -> ObservationProjectionPolicy
  -> task_summary_refs / bundle_summary_refs / artifact_refs
  -> session commit
```

必须保持的边界：

- decide：`model_turn_decision` 负责当前轮意图和行动边界。
- restore：memory/context 只能提供候选、摘要、引用，不能替代当前轮裁决。
- execute：execution engine 只执行已被 permit 允许的工具和模型 turn。
- present：输出层只面向用户展示自然语言、产物引用和必要限制，不暴露内部 event id。

## 7. 迁移与清理规则

1. 本项目不保留无用旧逻辑作为“兼容兜底”。
2. 每个阶段完成后，旧函数如果不再被调用，必须删除或改成明确的新入口。
3. 测试必须按新结构更新，不能通过 fake success 绕过真实行为。
4. 新增 cache 必须有失效条件和生命周期，不能变成跨 session 隐性状态。
5. prompt 文本修改必须写给 agent，而不是写成开发说明。
6. sidecar 优化不能破坏 fail closed 原则；没有可信裁决时不得执行工具。
7. raw trace 可以保留给 diagnostics，但不能作为模型上下文兜底。

## 8. 优先级建议

推荐实施顺序：

1. 阶段 A：重复 prompt、stream recovery、sidecar cache。
2. 阶段 B：模型客户端池化和 shutdown。
3. 阶段 C：Memory runtime view 单轮缓存。
4. 阶段 D：HistoryAssembly 接入主链路。

原因：

- 阶段 A 改动小，能立刻减少重复 token 和重复请求。
- 阶段 B 对延迟和稳定性收益明显，但需要改旧测试。
- 阶段 C 能减少运行时重复 IO 和 memory 计算，为阶段 D 铺路。
- 阶段 D 是结构收益最大的一步，但涉及主上下文入口，必须在前面指标和测试稳定后实施。

## 9. 本轮不处理项

以下内容不纳入本报告实施范围：

- 不修改 `context_budget_preset` 默认值。
- 不调整 `llm_max_output_tokens`。
- 不做 System Prompt wire 级静态/动态 cache block。
- 不引入 provider-specific prompt cache control。
- 不为了省 token 降级 agent 的真实执行能力。

这些可以作为后续单独方案处理。

## 1. 本计划结论

本轮实施不直接在旧 runtime 上打补丁，而是先建立平台级文件管理系统，再建立新主链的真实代码骨架，随后逐步切换调用点，最后清理旧链路。

目标主链固定为：

```text
PlatformFileManagementSystem
-> TaskEnvironment
-> SpecificTaskAssemblyPolicy
-> AgentFactory
-> FileAccessTable
-> ToolCapabilityTable
-> PermissionContext
-> RuntimeStartPacket
-> Runtime Loop
-> ToolSupervisor
-> ToolExecutor
-> ExecutionReceipt
```

第一执行切片改为 `220-平台文件管理系统优先重构计划书-20260526.md` 定义的平台文件管理基础骨架：

```text
ManagedFileEnvironmentProfile
ManagedFileRepositorySpec
ManagedFileRef
FileAccessTable
FileOperationReceipt
```

本计划原第一切片顺延为第二执行切片，且必须消费平台文件管理系统产出的 file profile / file access table：

```text
TaskEnvironmentSpec
ToolCapabilityTable
PermissionDecision / PermissionContext / PermissionReceipt
```

文件管理第一切片和任务环境第二切片都不改变现有 runtime 行为，不删除旧 `domain_binding`、`execution_permit`、`operation_gate`、`current_turn_capability_plan`。原因不是保留兼容旧架构，而是先让新事实对象能从现有代码中真实合成出来，后续切换时才有可验证目标。旧链路的删除条件在本计划第 10 节明确列出。

## 2. 当前约束

### 2.1 工作区状态

当前工作区已有大量未提交改动，涉及：

```text
backend/runtime/professional_runtime/*
backend/task_system/planning/*
backend/prompt_library/*
backend/tests/*
```

实施必须遵守：

```text
不回滚用户或前序改动。
不对无关文件做格式化或清理。
不删除旧链路，直到新链路覆盖对应执行入口并有测试证明。
不使用兼容名义长期保留双轨执行。
```

### 2.2 命名约束

不使用 `V2` 文件名或类名。

允许用：

```text
TaskEnvironment
ToolCapabilityTable
ToolSupervisor
PermissionDecision
RuntimeStartPacket
```

不允许用：

```text
TaskDomainV2
RuntimeV2
ToolPermissionV2
```

### 2.3 Prompt 约束

所有给 agent 的 prompt 必须是角色职责说明，不写开发注释式 prompt。

错误：

```text
这是 runtime 节点。
根据任务图执行 world_review。
```

正确：

```text
你是一名世界观审核员。
你只负责评审当前世界观设定是否完整、一致、可支撑后续写作。
```

## 3. 第二切片：任务环境、工具能力、权限基础对象

### 3.1 目标

本切片建立新链路的三类基础事实对象：

```text
TaskEnvironmentSpec
ToolCapabilityTable
PermissionDecision / PermissionContext / PermissionReceipt
```

并通过测试证明：

```text
写作环境不提供 shell。
写作环境引用平台托管作品文件管理 profile。
写作环境的产物政策和环境级 memory 都归属于文件管理环境。
写作环境支持打开正式作品，但正式作品写入必须走审核提交链路。
vibe coding 环境提供项目文件、git、编辑、浏览器、可审批 shell。
工具能力表是环境、具体任务、agent profile、runtime availability 的交集。
工具能力表消费 FileAccessTable，不直接按工具名决定文件工具可见性。
权限决策对象能表达 allow / deny / ask / sandbox / repair。
```

### 3.2 不做什么

本切片不做：

```text
不接管 runtime/tool_loop。
不改专业 runtime 行为。
不删除 domain_binding。
不改前端。
不迁移存储数据。
不改现有 API。
```

## 5. 第二切片验收标准

### 5.1 环境模型验收

测试断言：

```text
env.writing 存在。
env.vibe_coding 存在。
env.web_research 存在。
env.writing 的 shell policy 为 denied。
env.writing 引用 file_profile.writing_manuscript。
env.writing 对 official_work_repository / draft_workspace / artifact_repository / memory_repository 只声明环境约束，不复制文件系统实现。
env.writing 的 official_work_policy 允许打开正式作品，但 canonical write 需要 review receipt / commit gate。
env.writing 的 artifact_policy 与 memory_space 都能追溯到 file_management_space。
env.vibe_coding 引用 file_profile.vibe_coding_project。
env.vibe_coding 的 workspace policy 能追溯到 project_workspace / sandbox_workspace。
env.web_research 的 network policy 为 allowed。
```

### 5.2 工具能力表验收

构造：

```text
environment = env.vibe_coding
task_required_operations = op.read_file, op.search_text, op.edit_file, op.shell
agent_profile_allowed = op.model_response, op.read_file, op.search_text, op.edit_file
```

断言：

```text
op.shell 被 agent profile ceiling 过滤。
edit_file 可见且可调度。
terminal 不可见。
filtered_tools 记录 shell 被过滤原因。
```

构造：

```text
environment = env.writing
file_access_table 没有 canonical write grant
task_required_operations = op.read_file, op.write_file, op.shell
agent_profile_allowed 包含 op.shell
```

断言：

```text
op.shell 被 environment denied 过滤。
write_file 只可在 draft/artifact grant 范围内进入候选。
official_work_repository canonical write 不因 write_file 可用而放行。
```

### 5.3 权限对象验收

测试断言：

```text
PermissionDecision.allow(...).allowed == True
PermissionDecision.ask(...).requires_approval == True
PermissionDecision.deny(...).allowed == False
approval_fingerprint 会进入 receipt identity payload
PermissionReceipt 包含 task_run_id / agent_run_id / tool_call_id / operation_id / behavior
```

## 12. 禁止实现模式

实施中禁止：

1. 在旧 `domain_binding` 上继续堆新环境逻辑。
2. 把 `TaskEnvironment` 写成分类标签。
3. 把具体 agent/skills 固定写在环境里。
4. 让 agent 选择环境。
5. 用 prompt 代替工具能力表。
6. 让 ToolExecutor 绕过权限决策。
7. 审批只绑定工具名，不绑定参数 fingerprint。
8. 长期保留 `domain_id` 和 `environment_id` 双轨执行。
9. 为了通过测试伪造结果。
10. 删除旧代码前没有新链路测试覆盖。
11. 把写作产物政策、记忆管理和正式作品文件管理拆成三套互相绕开的权威。
12. 让写作 agent 直接覆盖正式作品文件，而不是通过平台文件管理环境的提交链路。

## 14. 预期代码完成后的第一阶段架构

文件管理切片和任务环境/工具/权限基础切片完成后，系统会同时存在：


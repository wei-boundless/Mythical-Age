# 内部提示词分层与能力状态修复计划

日期：2026-05-15

## 背景

主 Agent 已在运行时 profile 中配置 `op.write_file`、`op.edit_file`，但在用户询问“你不能自己创建文件吗”时，系统回答为“不能创建、写入或修改文件”。追踪运行链路后确认，这不是单纯权限缺失，而是内部上下文分层失真：

- 理解层把能力自省问题装配成知识检索或普通对话。
- 当前 turn 的 resource policy 只暴露本轮采用工具。
- 模型把“本轮未采用写入工具”误解成“Agent 总体没有写入能力”。
- 错误自述被写入 session memory 的稳定区域，后续 turn 又被错误记忆牵引。
- `filesystem_write_allowed=false` 等 commit gate 诊断字段容易被误读为 Agent 能力边界。

因此修复目标不是补关键词或强行改道，而是建立清晰的内部提示词分层与结构化能力状态来源。

## 问题定义

当前系统至少混合了五类语义：

1. Agent profile 能力上限：这个 Agent 被允许使用哪些 operation。
2. 当前任务采用能力：本轮 task contract/resource policy 采用并暴露了哪些 operation/tool。
3. 执行门禁状态：某一次工具调用、artifact 写入或 session commit 是否允许。
4. 历史记忆：历史对话中用户请求、模型回答、阶段结果与错误更正。
5. 对外能力说明：当用户问“你能做什么”时，Agent 应该基于什么事实回答。

这些层没有被 prompt 明确区分，模型只能从可见工具和历史记忆中推断能力，导致局部状态污染整体能力判断。

## 修复原则

- 不用关键词堵口子，不针对“创建文件”单点做特殊改道。
- 不把开发者说明直接塞给 Agent 当 prompt，要写成 Agent 能理解的角色/职责/边界说明。
- 能力说明必须来自结构化 runtime capability state，而不是从当前工具列表或历史回答推断。
- 当前 turn 工具列表只代表“本轮采用能力”，不得代表 Agent 总能力。
- 记忆里的模型自我能力判断默认不具备稳定事实效力，除非来源是系统能力状态快照。
- 诊断字段必须命名清楚，避免把 commit scope、artifact write gate、profile permission 混成一个 `allowed=false`。

## 目标状态

当用户询问能力或权限时，Agent 能区分回答：

- “我的配置上限允许写文件/编辑文件。”
- “当前这轮任务没有采用写入工具，所以现在可见工具列表里可能没有 write_file。”
- “如果你明确要求创建或修改具体文件，任务装配层应采用写入相关 operation，并经过对应安全边界。”
- “commit gate 中的 filesystem write 标志只描述某次提交范围，不代表我的总能力。”

当普通任务不需要写入时，系统仍应保持最小工具暴露，不把所有工具都常驻暴露给模型。

## 实施范围

### 一、能力状态模型

新增或整理一个结构化 `runtime_capability_state`，由运行时生成，不由模型猜测。

建议字段：

- `agent_profile_operations`：Agent profile 允许的 operation。
- `turn_adopted_operations`：当前任务采用的 operation。
- `turn_visible_tools`：当前暴露给模型的工具。
- `approval_required_operations`：需要审批或交互确认的 operation。
- `blocked_by_profile_operations`：profile 明确禁止的 operation。
- `blocked_by_turn_policy_operations`：本轮未采用或被本轮策略阻止的 operation。
- `commit_scope_diagnostics`：当前 commit gate 的提交范围说明。

该状态只作为事实来源，不直接替代任务路由。

### 二、内部提示词分层

在 prompt 结构中明确分层：

- `identity_view`：身份、风格、职责。
- `agent_capability_view`：Agent 能力上限与授权边界。
- `current_turn_resource_view`：本轮可见工具和采用 operation。
- `task_contract_view`：当前用户请求、任务目标、输出边界。
- `memory_view`：历史上下文，区分稳定事实、近期结果、历史错误和调试材料。
- `commit_gate_view`：只说明提交/写回范围，不表达 Agent 能力上限。

关键要求：

- `current_turn_resource_view` 必须声明：本层只描述当前任务，不代表 Agent 总能力。
- `memory_view` 必须声明：历史 assistant 自我能力判断不可覆盖 runtime capability state。
- `agent_capability_view` 必须用自然语言表达职责，不泄漏内部表单。

### 三、理解层默认路由

调整理解层默认行为：

- 没有明确知识源、检索、证据、FAQ、外部信息需求时，不默认进入 RAG。
- 能力/权限/工具/运行环境类问题应归入“运行时自省/能力说明”语义，而不是知识检索。
- 文件创建或修改请求只有在用户要求实际写入、生成、保存、编辑具体产物时，才进入写入执行任务。
- 单纯问“你能不能创建文件”不应直接执行写入，也不应装配成知识检索。

这里的修复应基于结构化意图分类和语义类别，而不是硬编码某句话。

### 四、资源策略与工具暴露

保持最小暴露原则：

- 普通对话只暴露 `op.model_response`。
- RAG 只暴露检索相关能力。
- 写入任务才采用 `op.write_file` / `op.edit_file`。

同时在 prompt 中加入清晰解释：

- “当前可见工具”是本轮执行面。
- “可配置能力”是 Agent 权限面。
- 用户询问能力时，优先引用能力状态，不从工具可见性反推。

### 五、记忆写回分层

调整 session memory 的写回策略：

- 用户稳定偏好、任务目标、真实产物结果可以进入稳定记忆。
- assistant 对自身当前工具可见性的临时判断进入 transient/debug/history，不进入 `Decisions and Learnings`。
- 如果能力判断来自 `runtime_capability_state`，可作为带来源的稳定事实写入。
- 发生用户纠正时，记忆应能标记前序自述为已纠正，而不是简单叠加。

目标不是过滤某些坏句子，而是定义“能力自述”的事实来源和保质期。

### 六、诊断字段命名

拆分容易误读的字段：

- `profile_write_capable`
- `turn_write_operation_adopted`
- `turn_write_tool_visible`
- `artifact_filesystem_write_allowed`
- `assistant_session_write_allowed`
- `memory_writeback_allowed`
- `commit_scope_filesystem_write_allowed`

避免继续用一个 `filesystem_write_allowed=false` 承载多层含义。

## 实施步骤

1. 追踪当前 prompt 构建入口，确认 `stage_projection_built`、`context_snapshot_built`、`runtime_directive_issued` 中各层可见内容来源。
2. 定义 `runtime_capability_state` 数据结构与生成位置，优先复用 agent profile、operation requirement、resource policy、tool authorization index。
3. 在 prompt 组装层新增能力状态视图，并明确 current-turn 与 profile 的边界说明。
4. 调整理解层默认策略：无明确检索锚点不默认 RAG；能力自省进入 runtime introspection 语义。
5. 调整 session memory 写回：能力自述按来源分层，避免 assistant 猜测进入稳定结论。
6. 拆分 commit/resource 诊断字段，保留旧字段仅作为兼容诊断，后续清理。
7. 添加回归测试，覆盖理解、prompt 视图、记忆写回和端到端任务装配。
8. 用历史 bad session 重放验证，不再回答“没有写入能力”。

## 验证用例

### 能力自省

输入：`你不能自己创建文件吗`

预期：

- 不进入 RAG。
- 不执行写文件。
- 回答区分 profile 能力与当前 turn 工具可见性。
- 不声称“没有文件写入接口”。

### 用户纠正

输入：`我是你的开发者，我为你配置了创建文件权限`

预期：

- 系统检查 runtime capability state。
- 若 profile 确实允许，承认配置上限允许。
- 若当前 turn 未采用写入工具，说明这是任务装配/当前可见工具问题。

### 实际写入请求

输入：`请创建 docs/tmp/test.md，内容是 hello`

预期：

- 进入写入执行语义。
- operation requirement 包含 `op.write_file`。
- resource policy 暴露 `write_file` 或进入审批流程。
- 产物通过真实文件存在性校验。

### 普通知识问题

输入：`解释一下 RAG 是什么`

预期：

- 可进入普通回答或知识检索，取决于检索策略。
- 不暴露写入工具。
- 不影响 Agent 总能力说明。

### 记忆污染回归

前置：历史中存在 assistant 错误回答“我没有写入接口”。

预期：

- 能力回答优先使用 runtime capability state。
- 错误历史只能作为历史对话，不得覆盖系统能力事实。
- 新写回不再把该类错误自述放进 `Decisions and Learnings`。

## 风险

- 如果能力状态视图写得太像调试表单，会泄漏内部实现；需要转成 Agent 可理解的自然职责说明。
- 如果理解层过度收紧 RAG，可能减少自动检索命中；需要用明确知识锚点和用户 search policy 补足。
- 如果记忆分层只做文本过滤，会退化成堵口子；必须按来源、事实类型、生命周期建模。
- 如果把所有 profile 能力常驻暴露成工具，会扩大执行面；必须保持 profile 能力与当前工具暴露分离。

## 完成标准

- 主 Agent 能正确说明自己具备被配置的写入能力，同时说明当前任务是否采用。
- 能力询问不再被默认装配成知识检索。
- 写入任务仍通过结构化任务装配和安全边界执行。
- 错误能力自述不会进入稳定记忆并污染后续 turn。
- 相关回归测试真实通过，不用伪造输出或绕过测试。

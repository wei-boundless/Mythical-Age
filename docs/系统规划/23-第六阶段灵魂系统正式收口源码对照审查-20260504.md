## 二、已经符合设计书的部分

### 2.1 灵魂系统开始具备正式 public boundary

现在外层不再需要同时知道：

- `SoulRegistry`
- `projection_store`
- `ProjectionTemplateRegistry`
- `ProjectionInstanceRegistry`
- `build_soul_runtime_view`

而是可以通过 `SoulFacade` 进入。

这符合灵魂系统应作为正式系统存在，而不是散装实现集合的方向。

### 2.2 注册层、投影层、装配层开始分段

当前已形成：

- 注册层：`SoulRegistryService`
- 投影层：`SoulProjectionService`
- 装配层：`SoulAssemblyService`

这使灵魂系统内部开始具备稳定层次，而不是所有能力都压在 `api/souls.py` 或 `projection.py` 里。

### 2.3 编排层继续只消费身份装配结果

当前编排层仍只拿：

- `SoulRuntimeView`
- `PromptManifest`
- `AgentPromptBundle`

并没有把 workflow、resource policy、memory authority 交给灵魂系统。

这保持了设计书要求的硬边界。

## 三、当前仍存在的结构问题

### 3.1 运行时身份装配仍有底层 builder 需要继续纯化

虽然编排层现在已经通过 facade 进入，但底层仍依赖：

- [backend/soul/runtime_assembly.py](/d:/AI应用/langchain-agent/backend/soul/runtime_assembly.py:1)

这说明 facade 已立住，但底层实现还没有进一步拆成更细的 identity asset / manifest builder 结构。

### 3.2 `api/souls.py` 仍保留部分本地辅助逻辑

比如：

- portrait 上传校验
- path normalize

这些不属于严重边界问题，但后续如果继续纯化，可以再下沉到 soul system service 中。

## 四、阶段结论

这一轮已经把灵魂系统从：

`prompt 素材目录 + 多处直连实现`

推进成：

`具备正式入口的 prompts 身份层系统`

当前剩余问题主要是内部进一步纯化，而不是方向错误。

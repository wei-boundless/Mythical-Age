## 0. 核心结论

当前灵魂系统已经不适合继续以“单一 seed + 投影”的方式扩展，因为后续需求已经变成几件不同的事：

1. 保存世界背景，给灵魂卡片提供故事土壤。
2. 保存灵魂故事，给灵魂卡片提供形成原因和性格连续性。
3. 保存工作 prompts，给执行任务提供无背景、无扮演、可复用的纯工作指令。
4. 保留用户自己填写的共同契约定制 prompts，作为所有模式都可以引用的个人口径和稳定偏好。
5. 支持灵魂显化，让灵魂在前端具象化为头像、立绘、模型、动作和工作状态反馈。
6. 支持从灵魂角度查看最近工作日志：只读聚合现有会话、任务、投影、产物和记忆写入回执，不复制记忆正文，不改变记忆库归属。

因此，这次重构不是“再加几个 prompt 文件”，而是把灵魂系统正式拆成三层资源、一层用户定制层、一层显化层和一层工作日志读视图：

```text
World Background Library
  -> Soul Story Library
  -> Soul Card
  -> Work Projection
  -> Soul Manifestation
  -> Soul Work Log View
```

并把运行模式收束成三档：

```text
role_mode
standard_mode
work_mode
```

其中：

- `role_mode` 允许背景世界 + 灵魂故事 + 工作投影同时参与，适合灵魂体验和陪伴式对话。
- `standard_mode` 只保留灵魂故事 + 工作投影，适合稳定表达但不需要完整背景渲染的场景。
- `work_mode` 只保留共同契约 + 任务契约 + 工作 prompt，不带背景、不带灵魂扮演，适合纯执行任务。

## 1. 现状审查

### 1.1 当前已有资源

现有 `backend/soul` 已经提供了这些能力：

- `agent_core/ACTIVE_SEED.md`
- `agent_core/CORE.md`
- `agent_core/seeds/*.md`
- `projections/catalog.json`
- `projection_store.py`
- `projection_builder.py`
- `runtime_assembly.py`
- `registry.py`
- `registry_service.py`

前端也已经有灵魂入口、投影入口和任务系统里的投影绑定入口：

- `frontend/src/lib/souls.ts`
- `frontend/src/components/soul/*`
- `frontend/src/components/workspace/views/OrchestrationView.tsx`
- `frontend/src/components/workspace/views/TaskSystemView.tsx`

### 1.2 当前问题

1. `ACTIVE_SEED.md` 仍然承担了过多“灵魂来源”的职责，后续一旦加入世界背景和故事库，会变得越来越厚。
2. `CORE.md` 现在是共用底线，但用户希望它升级成“用户自己填写的定制 prompts”，这意味着它不该再写成内部实现说明。
3. 工作任务与灵魂体验混在一套资源里，后续会导致前端管理页、目录结构和运行模式都混乱。
4. 现在的资源路径和前端类型定义还写死了 seed/background 的少量固定角色，无法承载真正的资源库化。

### 1.3 本轮必须保留的原则

- 背景和故事可以很丰富，但默认不进入 `work_mode`。
- 灵魂可以有趣，但不能影响权限。
- 工作 prompt 必须能独立复用，不能依赖前两层才能成立。
- 共同契约必须保留，但作为用户可编辑、可定制的稳定 prompt 层，而不是内部开发说明。

## 2. 目标架构

### 2.1 四类资源

#### World Background Library

职责：

- 保存世界观、时代、阵营、规则、禁忌、审美、语调来源。
- 作为灵魂卡片的背景层，不直接作为执行 prompt。
- 支持多个世界背景并存，供前端管理和联动引用。

#### Soul Story Library

职责：

- 保存灵魂的来历、经历、关系、转折、偏好来源。
- 说明为什么这个灵魂是这个样子。
- 可以与某个世界背景绑定，也可以复用同一世界中的多条故事线。

#### Work Prompt Library

职责：

- 保存无背景、无灵魂、无扮演的纯工作 prompts。
- 专门用于实际任务执行、修复、写作、校验、总结等场景。
- 默认条目必须能独立运行，不依赖灵魂前置内容。

#### User Custom Common Contract Library

职责：

- 保存用户自己填写的共同契约定制 prompts。
- 作为所有模式都可引用的用户口径层。
- 只承载稳定偏好、输出规则、协作边界和长期禁忌。

#### Soul Manifestation Library

职责：

- 保存灵魂在前端如何出现。
- 支持静态头像、立绘、Live2D、VRM/3D、小宠物状态机等形式。
- 只负责展示、陪伴和状态反馈，不参与权限、工具和任务完成判断。

#### Soul Work Log View

职责：

- 给会话记忆和工作记忆增加灵魂/投影活动视图副本。
- 支持按 `soul_id`、`projection_id`、`work_prompt_id`、`session_id`、`task_run_id` 查询。
- 只保存索引和摘要，不替代原始记忆事实来源。
- 不允许旧记忆自动覆盖当前 turn 的用户目标。

### 2.2 三层灵魂结构

#### 第一层：背景世界

只回答“这个灵魂来自哪里、所在世界如何运作”。

#### 第二层：灵魂本体

只回答“这个灵魂是谁、怎样思考、怎样说话、怎样偏向判断”。

#### 第三层：工作投影

只回答“这一次任务怎么做、输出什么、工具怎么用、验收如何收口”。

### 2.3 运行模式

| 模式 | 注入内容 | 适用场景 |
|---|---|---|
| `role_mode` | 世界背景 + 灵魂本体 + 工作投影 + 用户共同契约 | 对话、陪伴、角色体验、轻问答 |
| `standard_mode` | 灵魂本体 + 工作投影 + 用户共同契约 | 稳定表达、有限工具任务 |
| `work_mode` | 用户共同契约 + 任务契约 + 工作 prompt | 纯执行、纯修复、纯产出 |

模式只决定装配规则，不决定权限来源。权限仍然由现有控制层负责。

### 2.4 灵魂显化边界

灵魂显化可以影响：

- 前端展示。
- 情绪状态。
- 工作状态提示。
- 多灵魂联动展示。
- 用户对当前灵魂的感知。

灵魂显化禁止影响：

- prompt 权限。
- 工具调用。
- 执行义务。
- 验证结果。
- 工作模式是否启用灵魂层。

### 2.5 工作日志视图边界

会话记忆、工作记忆、任务运行和产物记录仍然归属各自系统；灵魂系统只提供一个只读聚合视图：

```text
runtime/task/memory/artifact records
  -> source fact remains in its owning system
  -> SoulWorkLogView aggregates summary refs for frontend display
```

它不能变成新的事实源。运行时读取时必须仍然遵守：

```text
当前用户 turn
> 执行义务
> 任务/工作契约
> 当前模式装配
> 活动视图候选
```

## 3. 资源组织方案

### 3.1 后端目录建议

建议把 `backend/soul` 拆成更清晰的资源面：

- `backend/soul/worlds/`
- `backend/soul/stories/`
- `backend/soul/work_prompts/`
- `backend/soul/common_contracts/`
- `backend/soul/cards/`
- `backend/soul/manifestations/`
- `backend/soul/activity_view/`
- `backend/soul/runtime/`

说明：

- `worlds/` 放世界背景卡。
- `stories/` 放灵魂故事卡。
- `work_prompts/` 放纯工作 prompts。
- `common_contracts/` 放用户定制共同契约。
- `cards/` 放灵魂卡片聚合视图。
- `manifestations/` 放灵魂显化资源。
- `activity_view/` 放灵魂/投影活动视图。
- `runtime/` 放装配与运行态实现。

### 3.2 结构化 schema

建议新增或重构为这些核心对象：

- `SoulWorld`
- `SoulStory`
- `SoulCard`
- `WorkPrompt`
- `CommonContractPrompt`
- `SoulManifestation`
- `SoulActivityEvent`
- `SoulModeProfile`
- `SoulRuntimeView`

关键规则：

- `SoulCard` 只是聚合对象，不是唯一真相。
- `WorkPrompt` 是执行真相，不需要依赖背景才能成立。
- `CommonContractPrompt` 由用户编辑，不写成内部说明文档。

### 3.3 灵魂召唤器

“灵魂召唤器”应当是一个结构化编译器，而不是散文生成器。

输入建议：

- 世界背景选择
- 灵魂故事选择
- 语言风格轴
- 认知风格轴
- 关系距离轴
- 行动倾向轴
- 风险偏好轴
- 幽默度/温度/专业度

输出建议：

- 标准化 `SoulCard`
- 标准化 `SoulRuntimeView`
- 可追踪来源引用

- 不再硬编码 `backgroundPath`
- 不再假定只有 5 个默认灵魂
- 支持自定义世界、故事、工作 prompt 的增删改查

## 5. 装配与运行规则

### 5.1 role_mode

用于：

- 角色体验
- 带背景的对话
- 灵魂联动
- 轻量只读能力

注入顺序建议：

```text
用户共同契约
-> 世界背景
-> 灵魂故事
-> 工作投影
```

### 5.2 standard_mode

用于：

- 稳定表达
- 有限工具任务
- 不需要完整背景渲染的场景

注入顺序建议：

```text
用户共同契约
-> 灵魂故事
-> 工作投影
```

### 5.3 work_mode

用于：

- 修复代码
- 写文件
- 跑测试
- 产出文档
- 纯任务执行

注入顺序建议：

```text
用户共同契约
-> 任务契约
-> 工作 prompt
```

约束：

- 不注入世界背景。
- 不注入灵魂故事。
- 不做角色扮演描述。

## 6. 实施路线

### 阶段 1：资源 schema 冻结

目标：先把四类资源的结构定死。

动作：

1. 定义 `WorldBackground`、`SoulStory`、`WorkPrompt`、`CommonContractPrompt`。
2. 定义 `SoulCard` 的聚合字段。
3. 定义模式枚举和装配规则。
4. 给每类资源定义可编辑字段与只读字段。

### 阶段 2：后端目录与 catalog 落地

目标：把资源从单一 seed 结构迁移成分库结构。

动作：

1. 新增世界背景库目录。
2. 新增灵魂故事库目录。
3. 新增工作 prompt 库目录。
4. 新增用户共同契约库目录。
5. 新增灵魂显化库目录。
6. 新增灵魂工作日志目录。
7. 提供统一 catalog API。

### 阶段 3：灵魂召唤器

目标：让灵魂生成走结构化编译流程。

动作：

1. 支持从背景 + 故事 + 参数轴生成灵魂卡片。
2. 支持手写卡片和生成卡片并存。
3. 支持卡片预览与来源追踪。

### 阶段 4：运行模式收口

目标：把模式和装配层绑定起来。

动作：

1. `role_mode` 允许背景注入。
2. `standard_mode` 禁止背景注入。
3. `work_mode` 禁止灵魂注入。
4. 共同契约作为三种模式都可引用的用户层 prompt。

### 阶段 5：前端管理页重构

目标：让资源库能被真正编辑和维护。

动作：

1. 改成资源库导航。
2. 增加世界背景库页面。
3. 增加灵魂故事库页面。
4. 增加工作 prompt 库页面。
5. 增加共同契约库页面。
6. 增加灵魂显化库页面。
7. 增加灵魂工作日志页面。

### 阶段 6：工作日志读视图接入

目标：打开灵魂卡片时，可以看到这个灵魂最近参与了哪些工作。

动作：

1. 定义 `SoulActivityEvent` 和 `SoulWorkLogView`。
2. 从现有任务运行、投影绑定、事件日志、产物仓库和记忆写入回执汇总只读摘要。
3. API 只返回任务/会话/投影/状态/产物数量/最后活动时间/引用，不返回记忆正文。
4. 前端在灵魂详情页展示“工作日志”，不把它做成新的记忆管理入口。

### 阶段 7：旧结构迁移与清理

目标：删掉旧壳，不保留两套体系长期并存。

动作：

1. 将现有 seed 迁移到背景/故事/工作 prompt/共同契约/显化五类库。
2. 清理硬编码固定角色路径。
3. 清理只服务旧结构的页面和类型。
4. 保留投影与运行态，但改成从新资源库装配。

## 7. 验证矩阵

### 必测项

1. 世界背景能单独保存、预览、引用。
2. 灵魂故事能单独保存、预览、引用。
3. 工作 prompt 能独立运行，不依赖前两层。
4. 共同契约能由用户编辑，并进入所有模式。
5. `work_mode` 不注入背景和灵魂故事。
6. `standard_mode` 不注入背景，但保留灵魂故事。
7. `role_mode` 保留完整三层。
8. 前端能区分背景、故事、卡片、工作 prompt、共同契约、显化和活动视图。
9. 旧 seed 结构迁移后数据不丢失。
10. 不允许用伪造结果绕过真实资源装配。
11. 灵魂详情页能看到最近工作日志，但不复制记忆库、不覆盖当前 turn。

## 8. 风险与控制

| 风险 | 控制 |
|---|---|
| 资源库种类变多导致前端混乱 | 强制分库页面，不混在单页 |
| 工作 prompt 被灵魂污染 | `work_mode` 不允许注入背景和故事 |
| 共同契约被写成开发说明 | 只允许用户口径和稳定偏好，不写内部实现 |
| 旧 seed 迁移出错 | 先做导入映射，再做切换，不直接删除 |
| 灵魂召唤器变黑箱 | 必须输出结构化卡片和来源引用 |
| 显化层越权 | 显化层只进前端状态，不进权限和执行裁判 |
| 工作日志被误当记忆库 | 工作日志只读展示摘要和引用，不保存记忆正文，不参与运行裁决 |

## 10. 成功定义

完成后，系统应满足：

1. 世界背景、灵魂故事、工作 prompts、共同契约、显化、活动视图清晰分库。
2. 前端可独立管理这些资源。
3. `work_mode` 可以完全脱离灵魂扮演。
4. `standard_mode` 保留灵魂连续性但不强依赖世界观。
5. `role_mode` 保留完整灵魂体验。
6. 共同契约变成用户可编辑的定制 prompts。
7. 灵魂显化可以在前端表达状态，但不影响执行。
8. 灵魂工作日志可以按灵魂/投影展示最近活动，但不成为新的事实权威。
9. 旧 seed 结构完成迁移，正式运行链不再依赖单一 prompt 文件体系。

最终目标很简单：

```text
灵魂可以有故事；
工作可以很纯净；
用户可以自己定规则；
系统可以按模式装配；
前端可以真正管理这些资源。
```

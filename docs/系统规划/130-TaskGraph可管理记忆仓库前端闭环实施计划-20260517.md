# TaskGraph 正式记忆库与可管理仓库工程实施计划

## 0. 设计结论

TaskGraph 的记忆库模型必须是通用的，写作任务只是验证样例。

最终模型是：

```text
前端 TaskGraph
  memory_repository 节点声明仓库
  collection 声明仓库分区
  memory_read / memory_write_candidate / memory_commit 边声明读写通道

后端 FormalMemoryStore
  按 repository / collection / record / version 结构化存储
  用事务、幂等键、head 指针、receipt、read_log 保证稳定性

Runtime
  写入时只按写入边落库
  读取时只按读取边和 selector 装配 MemorySnapshot
  时序系统决定记录从哪个 clock 起对下游可见

Artifact Store
  保存 md、正文、审核稿、候选稿等人类可读文件
  不承担正式记忆库的版本、选择、可见性和读写裁决职责
```

这不是 RAG，不是扫描 md 文件夹，也不是额外权限系统。

读写关系只由图边表达：

```text
memory_repository --memory_read--> task_node
task_node --memory_write_candidate--> memory_repository
task_node --memory_commit--> memory_repository
```

## 1. 要解决的问题

当前项目已经有正式记忆库雏形，但还没有稳定闭环：

- 前端已经能表达 `memory_repository` 节点和 `memory_*` 边。
- 后端已经能在 `WorkingMemoryItem.metadata.formal_memory` 中携带 repository / collection 信息。
- `WorkingMemoryService.select_for_node()` 已能按 `repository_read_edges` 做一层筛选。

但这还不够：

- 记忆记录仍混在 working memory item 里，没有正式 repository / collection / record / version 表。
- baseline 之类仓库容易变成人类可读 md 的堆叠，而不是“当前定稿 + 版本历史”。
- 写入和提交之间缺少强事务边界。
- 读取虽然有 selector，但缺少正式版本 head、时序可见性、read log 的完整保证。
- 前端“仓库管理”还没有明确告诉用户：建仓库后后端会建立什么分区，哪些边会读写这些分区。

## 2. 通用领域模型

### 2.1 Repository

Repository 是 TaskGraph 上的资源节点，也是后端正式存储空间。

推荐资源节点结构：

```json
{
  "node_id": "memory.project.world",
  "node_type": "memory_repository",
  "title": "世界观记忆库",
  "work_posture": "resource",
  "metadata": {
    "memory_repository": {
      "repository_id": "memory.project.world",
      "repository_kind": "formal_memory",
      "schema_id": "schema.formal_memory_record",
      "collections": []
    }
  },
  "resource_lifecycle_policy": {
    "versioning": "append_version",
    "commit_required": true
  }
}
```

Repository 只声明“这里有一个可管理仓库”，不声明哪些节点能读写。读写由边决定。

### 2.2 Collection

Collection 是 repository 内的标准分区。

通用字段：

```json
{
  "collection_id": "world",
  "title": "世界观",
  "schema_id": "schema.writing.world_bible",
  "record_kinds": ["world_bible"],
  "key_strategy": "stable_key",
  "default_version_selector": "latest_committed_before_clock",
  "required_receipt_status": "committed",
  "retention_policy": {
    "keep_committed_versions": true,
    "archive_rejected_candidates_after_days": 14
  }
}
```

Collection 解决的是“这类记忆放在哪里、用什么 schema、怎么版本化”，不是读写关系。

### 2.3 Record

Record 是一条稳定身份的记忆。

关键字段：

```text
repository_id
collection_id
record_key
record_kind
current_committed_version
head_version_id
status
```

例如世界观当前定稿：

```text
repository_id = memory.writing.baseline
collection_id = world
record_key = world_bible.current
record_kind = world_bible
current_committed_version = 4
```

### 2.4 Record Version

每次候选写入、审核提交、整理改写都生成版本，不直接覆盖旧内容。

关键字段：

```text
version_id
record_id
version
status: candidate | committed | rejected | superseded | archived
payload_json
canonical_text
summary
artifact_refs
source_node_id
source_edge_id
source_clock
visible_after_clock
content_hash
supersedes_version_id
```

### 2.5 Edge

边是唯一读写通道。

写候选：

```json
{
  "edge_type": "memory_write_candidate",
  "source_node_id": "world_author",
  "target_node_id": "memory.writing.baseline",
  "metadata": {
    "repository": "memory.writing.baseline",
    "collection": "world",
    "record_kind": "world_bible",
    "record_key": "world_bible.current",
    "source_output_key": "world_candidate"
  }
}
```

提交：

```json
{
  "edge_type": "memory_commit",
  "source_node_id": "world_review",
  "target_node_id": "memory.writing.baseline",
  "metadata": {
    "repository": "memory.writing.baseline",
    "collection": "world",
    "record_kind": "world_bible",
    "record_key": "world_bible.current",
    "candidate_ref_key": "reviewed_candidate_ref",
    "verdict_key": "verdict",
    "required_verdict": "pass",
    "receipt_policy": {
      "required_status": "committed",
      "visible_after": "next_clock"
    }
  }
}
```

读取：

```json
{
  "edge_type": "memory_read",
  "source_node_id": "memory.writing.baseline",
  "target_node_id": "chapter_writer",
  "metadata": {
    "repository": "memory.writing.baseline",
    "collection": "world",
    "selector": {
      "collection": "world",
      "record_kind": "world_bible",
      "record_key": "world_bible.current",
      "status_filter": ["committed"]
    },
    "version_selector": {
      "mode": "latest_committed_before_clock"
    },
    "model_visible_label": "世界观定稿",
    "usage_instruction": "你必须遵守这份已审核通过的世界设定；若当前章节要求与它冲突，必须报告冲突，不得自行改写设定。"
  }
}
```

## 3. 后端存储实现

### 3.1 第一版使用 SQLite

第一版使用 SQLite 实现 `FormalMemoryStore`。

理由：

- 项目已有 `WorkingMemoryStore` 和 `TaskDurableMemoryStore`，均使用 SQLite。
- SQLite 支持事务、唯一约束、索引、读写日志，足够支撑本地长任务。
- 不引入外部服务，便于测试和迁移。
- 未来可用同一 service interface 换成 Postgres。

### 3.2 目录结构

建议存储目录：

```text
storage/memory/formal_memory.sqlite
storage/memory/formal_memory/archive/
```

如果需要按任务运行隔离：

```text
output/novel_artifacts/.../memory/formal_memory.sqlite
```

长期建议采用：

```text
storage/memory/formal_memory.sqlite      项目级正式记忆
output/.../runtime_objects/              单次运行对象和 artifact
```

写作任务单次运行产生的 artifact 可以留在 `output`，正式记忆库索引和记录由 SQLite 管理。

## 4. 写入流程

### 4.1 写候选

输入：

```text
node_output
memory_write_candidate edge
current timeline clock
artifact refs
```

步骤：

```text
1. Runtime 找到当前节点的 memory_write_candidate 边。
2. 从边 metadata 解析 repository / collection / record_kind / record_key / source_output_key。
3. 校验 repository 和 collection 已由图节点声明。
4. 从 node_output[source_output_key] 取内容。
5. 按 collection.schema_id 做结构校验。
6. 计算 content_hash。
7. 构造 idempotency_key:
   graph_run_id + node_run_id + edge_id + record_key + content_hash
8. 在事务中 upsert formal_records，并插入 candidate version。
9. 写 formal_memory_transactions。
10. 返回 write receipt 给 runtime diagnostics 和下游 artifact index。
```

写候选不会自动对下游可见，除非读取边明确允许 candidate。默认只读 committed。

### 4.2 提交

输入：

```text
memory_commit edge
review node output
candidate reference
current timeline clock
```

步骤：

```text
1. Runtime 找到当前节点的 memory_commit 边。
2. 从边 metadata 解析 repository / collection / record_key / candidate_ref_key / verdict_key。
3. 读取 review output[verdict_key]。
4. 若 required_verdict 不满足，拒绝提交并写 rejected transaction。
5. 根据 candidate_ref_key 找候选 version。
6. 校验候选 version 的 repository / collection / record_key 与 commit 边一致。
7. 校验候选来源来自可达的 memory_write_candidate 边。
8. 在事务中：
   - 将候选 version 标记为 committed，或复制为 committed version。
   - 将旧 committed head 标记 superseded。
   - 更新 formal_records.current_committed_version 和 head_version_id。
   - 设置 visible_after_clock。
   - 写 formal_memory_transactions。
9. 返回 commit receipt。
```

提交必须是事务：head 指针、version 状态、receipt 必须同时成功或同时失败。

### 4.3 改写

改写不是覆盖原记录。

通用做法：

```text
1. 读取目标 record 当前 committed version。
2. 生成 revision candidate 或 delta candidate。
3. 审核通过后提交为新 version。
4. 新 version supersedes 旧 version。
5. 旧 version 保留，可追溯。
```

大纲修改尤其不能直接覆盖。

```text
future_outline 可以改写为新版本
frozen_outline / committed chapter facts 不允许原地改写
如果必须修正，写 correction record 或 issue record
```

## 5. 读取流程

输入：

```text
target node
incoming memory_read edges
current timeline clock
token budget
```

步骤：

```text
1. Runtime 收集指向当前节点的 memory_read 边。
2. 每条边解析 repository / collection / selector / version_selector。
3. FormalMemoryService 按 selector 查询 formal_records。
4. 按 version_selector 解析具体 version：
   - latest_committed_before_clock
   - latest_committed_before_scope
   - pinned_version
   - by_receipt
   - manual_snapshot
5. 应用 visible_after_clock <= current_clock。
6. 生成 MemorySnapshot 条目。
7. 写 formal_memory_read_logs。
8. 把 MemorySnapshot 交给 ContextPacketResolver。
9. Prompt Manifest 按 model_visible_label 和 usage_instruction 渲染给 Agent。
```

Agent 收到的是业务语言输入包，不是开发说明。

例如：

```text
输入包：世界观定稿
来源：memory.writing.baseline/world/world_bible.current@v4
说明：这是已审核通过的世界设定，写作时必须遵守。
```

## 6. 时序可见性

Formal Memory 必须依赖时序系统，而不是只靠时间戳。

推荐规则：

```text
candidate version:
  visible_after_clock = source_clock
  默认只对提交节点和诊断可见

committed version:
  visible_after_clock = next_clock / next_iteration / manual_release
  由 memory_commit edge 的 receipt_policy 决定

read selector:
  latest_committed_before_clock 只读取 visible_after_clock <= 当前 clock 的版本
```

这样可以避免：

- 一个节点读到同一轮还没提交的未来结果。
- 并发节点互相污染。
- 循环中的第 N 次返修读到第 N+1 次内容。

## 7. 写作任务的分区样例

这只是写作任务配置，不写死在通用编辑器里。

### 7.1 静态主干

```text
memory.writing.baseline/world
  record_kind: world_bible
  record_key: world_bible.current

memory.writing.baseline/characters
  record_kind: character_bible
  record_key: character.{character_id}

memory.writing.baseline/relationships
  record_kind: relationship_fact
  record_key: relationship.{source}.{target}

memory.writing.baseline/outline
  record_kind: approved_outline
  record_key: outline.global
```

特点：

- 只读 committed。
- 改写必须走候选、审核、提交。
- 当前定稿由 head 指针表示，不靠多个 md 文件猜。

### 7.2 动态计划

```text
memory.writing.mutable/volume_plan
  record_key: volume.{volume_id}.plan

memory.writing.mutable/future_chapter_outline
  record_key: volume.{volume_id}.chapter.{range}.outline

memory.writing.mutable/outline_delta
  record_key: outline_delta.{clock}.{id}

memory.writing.mutable/character_weight
  record_key: volume.{volume_id}.character_weight
```

特点：

- 未写章节可以整理为新版本。
- 已写章节对应的大纲应转入 frozen / committed，不再原地改。

### 7.3 章节事实

```text
memory.writing.canon/chapter_facts
  record_key: volume.{volume_id}.chapter.{chapter_id}.facts

memory.writing.canon/frozen_chapter_outline
  record_key: volume.{volume_id}.chapter.{range}.outline.frozen
```

特点：

- 章节通过审核后写入 canon。
- 后续写作读取 canon，而不是从正文 md 里猜事实。

### 7.4 产物索引

```text
memory.writing.artifact_index/world_candidates
memory.writing.artifact_index/world_reviews
memory.writing.artifact_index/volume_plans
memory.writing.artifact_index/chapter_outlines
memory.writing.artifact_index/chapter_drafts
memory.writing.artifact_index/chapter_commits
```

特点：

- 保存 artifact 引用、状态、版本、路径、摘要。
- md 文件仍在人类可读输出目录里。

## 8. 前端编辑器职责

### 8.1 拓扑页

拓扑页只负责结构：

- 创建记忆库资源节点。
- 连接 memory_read / memory_write_candidate / memory_commit 边。
- 显示读写流向。
- 不在拓扑页塞复杂 selector 表单。

### 8.2 记忆页

记忆页负责管理 repository / collection：

- 创建仓库。
- 创建 collection。
- 配置 schema、record_kind、key_strategy、版本策略。
- 显示仓库被哪些边读写。
- 从仓库视角跳转到对应读写边配置。

注意：这里不保存额外“权限字段”。页面显示的是图中真实边。

### 8.3 Selector 页

Selector 页负责配置边：

- `memory_read` 的 selector / version_selector / usage_instruction。
- `memory_write_candidate` 的 source_output_key / record_key / record_kind。
- `memory_commit` 的 candidate_ref_key / verdict_key / receipt_policy。

### 8.4 节点认知包页

职责页负责让用户确认节点最终收到什么：

- 节点身份。
- memory snapshot。
- artifact context。
- revision packet。
- prompt 使用说明。

### 8.5 发布页

发布页像编译器诊断：

- repository 节点没有 collection。
- memory_read 缺 selector。
- memory_write_candidate 缺 source_output_key / record_key。
- memory_commit 缺 candidate_ref_key / verdict_key / receipt_policy。
- 读写边引用不存在的 repository / collection。
- 写候选没有可达提交边。
- selector 读不到任何 committed record 时给 warning 或 runtime preflight。

## 9. 稳定性与持续维护

必须由工程机制保证，不靠 Agent 自觉。

### 9.1 事务

候选写入、提交、head 更新、receipt 写入必须在同一个数据库事务中完成。

### 9.2 幂等

写入幂等键：

```text
graph_run_id + node_run_id + edge_id + repository_id + collection_id + record_key + content_hash
```

同一节点重试不能生成重复版本。

### 9.3 唯一约束

必须保证：

```text
repository_id 唯一
repository_id + collection_id 唯一
repository_id + collection_id + record_key 唯一
record_id + version 唯一
```

### 9.4 Head 指针

每个 record 只能有一个当前 committed head。

baseline 里可以保留历史版本，但“当前世界观定稿”只能由 `current_committed_version` 指向一个版本。

### 9.5 Content Hash

每个 version 记录 `content_hash`：

- 检测重复写入。
- 检测 artifact 与正式记录是否一致。
- 支持运行后审计。

### 9.6 Receipt

每次 write / commit 返回 receipt：

```text
transaction_id
operation
repository_id
collection_id
record_key
version_id
status
visible_after_clock
content_hash
```

后续节点可以通过 receipt 精确引用版本。

### 9.7 Read Log

每次读取记录：

```text
node_run_id
edge_id
selector
selected_version_ids
clock
```

这能回答：“这个章节写手当时到底读到了哪个世界观版本？”

### 9.8 Integrity Check

发布前和运行中定期检查：

- 仓库节点和后端 repository 是否一致。
- collection 是否存在。
- record head 是否存在。
- committed head 是否唯一。
- candidate 是否长期未提交。
- read edge 是否可能读空。
- commit edge 是否找不到候选来源。

### 9.9 Retention

按 collection 策略归档：

- rejected candidate 可定期归档。
- superseded committed 版本保留。
- artifact refs 不删除，只标记状态。

# 任务系统契约体系规范化重构计划书

日期：2026-05-08

## 1. 背景

当前任务系统里“契约”一词被多层对象共同使用，但它们的职责并不相同：

- `TaskContract`：运行时任务单，描述一次任务运行的目标、来源、绑定与约束。
- `input_contract_id` / `output_contract_id`：任务、流程、工作流引用的输入输出标准标识。
- `TaskContractDescriptor`：从任务、流程、工作流、协议里反向收集出的契约视图。
- `input_boundary` / `output_boundary`：工作流内的自然语言边界说明。

这导致系统里没有一个真正权威的“契约定义中心”。契约 ID 可以在各处被直接写入，但契约本体、中文名称、字段结构、校验规则、样例、生命周期状态都没有统一来源。

前面已经清理掉旧长篇小说专用契约和默认工作流，因此现在适合重新建立一套通用契约体系。

## 2. 重构目标

建立一个独立、可编辑、可校验、可被任务系统引用的契约中心。

目标状态：

- `TaskContract` 只保留为运行时任务单，不再承担输入输出 schema 的含义。
- 所有输入、输出、流程、通信载荷契约都由 `ContractDefinition` 统一管理。
- 任务、流程、工作流、协调协议只引用契约 ID，不在本地发明契约含义。
- 前端任务系统提供契约编辑器，用户可以先建契约，再把契约绑定到任务、流程、工作流或协议。
- 运行时可以根据契约定义做基础校验、展示、追踪和产物归档。

## 3. 设计原则

- 单一来源：契约本体只能来自契约注册表，派生视图不能再作为权威来源。
- 稳定 ID：系统内部保存稳定 `contract_id`，前端展示中文名称。
- 不混运行实例：运行时任务单和契约定义分开，避免 `TaskContract` 继续混用。
- 先结构后校验：首轮先建立数据结构和引用关系，再逐步引入严格 schema 校验。
- 允许草稿：契约可以是 `draft`，但任务发布前必须引用 `active` 契约。
- 不保留旧残留：已清理的长篇小说契约不以默认资源形式回流。

## 4. 目标模型

新增核心模型：`ContractDefinition`

建议字段：

- `contract_id`：稳定标识，例如 `contract.input.user_message`
- `title`：中文名称，例如 `用户消息输入`
- `contract_kind`：契约类型，取值为 `input` / `output` / `flow` / `payload` / `artifact`
- `description`：用途说明
- `schema_format`：结构格式，首轮支持 `json_schema` / `text` / `markdown` / `opaque`
- `schema`：结构定义，JSON 对象
- `required_fields`：必填字段列表，便于前端快速展示
- `example_payload`：样例载荷
- `validation_policy`：校验策略，取值为 `none` / `shape_only` / `strict`
- `version`：契约版本，例如 `1`
- `status`：生命周期，取值为 `draft` / `active` / `deprecated`
- `owner_domain_id`：所属任务域，可为空表示全局
- `metadata`：扩展信息
- `authority`：固定为 `task_system.contract_definition`

保留派生模型：`TaskContractDescriptor`

调整后职责：

- 只作为“引用使用视图”。
- 从 `ContractDefinition` 和实际引用处合并生成。
- 不允许前端把它当成可编辑对象。

## 5. 存储设计

新增文件：

- `storage/tasks/contract_definitions.json`

结构：

```json
{
  "contract_definitions": []
}
```

首轮内置最小默认契约：

- `contract.input.user_message`：用户消息输入
- `contract.input.workspace_task`：工作区任务输入
- `contract.output.assistant_final_answer`：最终回答输出
- `contract.output.artifact_refs`：产物引用输出
- `contract.payload.coordination_message`：协调消息载荷

这些默认契约只覆盖通用系统运行需要，不再包含具体业务任务，例如长篇小说、游戏、健康问题等。

## 6. 后端改造范围

### 6.1 模型层

修改文件：

- `backend/tasks/contract_models.py`

新增：

- `ContractDefinition`
- `ContractDefinitionStatus`
- `ContractKind`

保留：

- `TaskContractDescriptor`

### 6.2 注册表层

修改文件：

- `backend/tasks/flow_registry.py`

新增能力：

- `list_contract_definitions()`
- `get_contract_definition(contract_id)`
- `upsert_contract_definition(...)`
- `delete_contract_definition(contract_id)`
- `list_contract_descriptors()` 从契约定义和引用关系合成，而不是只靠引用倒推。

清理规则：

- 删除 `CONTRACT_TITLE_MAP` 的权威含义。
- 如果仍需要兼容显示，只能作为迁移 fallback，不允许继续新增业务契约到这里。

### 6.3 API 层

修改文件：

- `backend/api/tasks.py` 或现有任务系统 API 文件

新增接口：

- `GET /tasks/contracts`
- `PUT /tasks/contracts/{contract_id}`
- `DELETE /tasks/contracts/{contract_id}`
- `POST /tasks/contracts/{contract_id}/activate`
- `POST /tasks/contracts/{contract_id}/deprecate`

返回内容：

- 契约定义列表
- 引用计数
- 被哪些任务、流程、工作流、协议引用
- 校验诊断

### 6.4 引用校验

涉及对象：

- `TaskFlowDefinition.input_contract_id`
- `TaskFlowDefinition.output_contract_id`
- `GeneralTaskProfile.input_contract_id`
- `GeneralTaskProfile.output_contract_id`
- `SpecificTaskRecord.input_contract_id`
- `SpecificTaskRecord.output_contract_id`
- `TaskWorkflowBinding.output_contract_id`
- `TaskCommunicationProtocol.payload_contracts`

规则：

- 草稿阶段可以引用不存在契约，但诊断必须标红。
- 发布或启用任务时，引用的契约必须存在且状态为 `active`。
- `input_contract_id` 必须引用 `input` 类型契约。
- `output_contract_id` 必须引用 `output` 或 `artifact` 类型契约。
- `payload_contracts` 必须引用 `payload` 类型契约。

## 7. 前端改造范围

修改页面：

- `frontend/src/components/workspace/views/TaskSystemView.tsx`

新增任务系统层级：

- `契约`

该层级不和具体任务、流程、协调任务混在一页，而是作为任务系统的一类基础资源。

契约编辑器应包含：

- 契约列表：按类型分组，显示中文名、ID、状态、引用数。
- 契约详情：编辑名称、类型、说明、schema 格式、schema、样例、状态。
- 引用视图：显示哪些任务、流程、工作流、协议正在使用它。
- 诊断视图：显示缺失引用、类型不匹配、草稿契约被发布资源引用等问题。

交互规则：

- 用户选择契约时显示中文名称。
- 保存引用时写入稳定 `contract_id`。
- 删除契约前必须确认；如果被引用，只允许废弃，不允许直接删除。
- 新建任务时优先从已有契约中选择，不再手写裸 ID。

## 8. 迁移策略

第一阶段：建立契约中心

- 新增 `ContractDefinition` 和 `contract_definitions.json`。
- 内置最小通用契约。
- API 返回契约定义和引用诊断。
- 不强制改动所有旧引用。

第二阶段：接入任务编辑器

- 任务、流程、工作流、协议的契约字段改为中文选择器。
- 保存仍写入 `contract_id`。
- 裸文本输入降级为高级编辑或调试入口。

第三阶段：引用校验

- 保存草稿时允许缺失契约，但展示诊断。
- 启用任务、发布协调任务、启用工作流时执行强校验。
- 把校验结果接入任务系统诊断面板。

第四阶段：清理旧派生逻辑

- `TaskContractDescriptor` 只作为视图，不再承担契约定义来源。
- 清理 `CONTRACT_TITLE_MAP` 的业务含义。
- 清理所有不在契约中心注册却被引用的契约 ID。

## 9. 验收标准

后端验收：

- `GET /tasks/contracts` 能返回契约定义、引用情况和诊断。
- 新建、修改、删除、启用、废弃契约 API 可用。
- `list_contract_descriptors()` 能合并契约定义和引用视图。
- 引用不存在契约时能产生诊断，而不是静默展示裸 ID。
- `python -m py_compile` 通过。

前端验收：

- 任务系统出现独立“契约”层级。
- 用户可以新建、编辑、废弃契约。
- 任务、流程、工作流、协议选择契约时展示中文名称。
- 被引用契约不能直接删除。
- `npm run build` 通过。

系统验收：

- 默认契约只包含通用系统契约。
- 不再出现旧长篇小说契约。
- 新建任务时可以通过契约中心选择输入输出契约。
- 后续新增业务任务必须先建契约，再引用契约。

## 10. 风险与处理

风险一：现有任务引用空契约或不存在契约。

处理：首轮允许草稿态引用缺失，但在诊断里明确显示；启用时再强校验。

风险二：`TaskContract` 名称容易继续误导。

处理：文档中明确称它为“运行时任务单”；如后续允许大改，可考虑重命名为 `RuntimeTaskEnvelope` 或 `TaskRunContract`。

风险三：过早引入严格 JSON Schema 会拖慢重构。

处理：首轮只做结构字段、样例和基础 shape 校验；严格校验放到第二轮。

风险四：前端页面已经很复杂。

处理：契约作为独立层级，不塞进任务编辑表单；任务表单只负责引用契约。

## 11. 推荐实施顺序

1. 后端新增 `ContractDefinition` 和注册表方法。
2. 新增最小默认通用契约。
3. 新增契约 API 和引用诊断。
4. 前端任务系统增加“契约”层级。
5. 任务/流程/工作流/协议字段改为契约选择器。
6. 启用任务时加入契约引用校验。
7. 清理 `CONTRACT_TITLE_MAP` 的剩余权威用途。

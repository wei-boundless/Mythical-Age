本文件用于定义本项目中能力系统的正式职责、对象模型、能力资源供给链路、与任务系统和编排系统的接口，以及前后端管理目标。

- [02-任务系统详细设计书-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/02-任务系统详细设计书-20260504.md)

本文件重点回答以下问题：

- 能力系统在平台里到底是什么
- skills、tools、workers、capabilities 之间是什么关系
- 为什么 workflow 不属于能力系统
- 能力系统如何为任务系统和编排系统提供能力资源
- 能力系统如何避免能力资源变成散乱集合

## 二、能力系统总定义

能力系统是平台的能力资源供应层。

它不负责定义任务，不负责装配单 Agent 身体，不负责记忆治理，不负责人格设定，也不负责健康诊断。

它负责的是：

- 工具资源
- skills 资源
- workers 与执行单元资源
- 能力目录
- 能力契约
- 资源描述与可见性
- 运行期可供给的资源集合

它回答的问题是：

`平台上有哪些能力资源可以被 Agent 使用，这些能力资源以什么形式存在、如何被引用、如何被治理`

## 三、为什么要把能力系统单独立出来

如果没有能力系统，平台里的能力资源就会散落在：

- tools 目录
- skills 目录
- capability endpoints
- workers registry
- 各种 manifest / catalog

这样会带来三个问题：

1. 任务系统只能“猜”自己能用什么能力
2. 编排系统只能“临时拼”资源边界
3. 平台很难知道哪些能力是正式可供给资源，哪些只是内部实现细节

所以能力系统必须作为正式系统存在，统一回答：

- 资源是什么
- 资源如何登记
- 资源怎么分层
- 资源如何被别的系统引用

## 四、能力系统的硬边界

### 4.1 能力系统不定义任务

任务系统定义：

- 任务目标
- workflow
- projection
- flow contract

能力系统不能决定：

- 什么任务要做
- 哪个 Agent 来做
- 任务链路怎么分流

### 4.2 能力系统不装配单 Agent 身体

编排系统决定：

- 单 Agent 的资源边界
- 哪些资源开放
- 哪些资源禁止
- 哪些资源需审批

能力系统只提供能力目录和能力资源本体。

### 4.3 能力系统不筛记忆

记忆系统负责上下文与记忆供给。

能力系统只可能提供：

- 检索能力
- 文件读取能力
- 数据解析能力

但这些能力不等于记忆本体。

### 4.4 workflow 不属于能力系统

这一点必须再次锁死。

workflow 是任务方法，不是能力资源。

skills、tools、workers 才是能力资源。

所以：

- workflow 归任务系统
- capability 归能力系统

## 五、能力系统中的资源分层

我建议把能力系统中的能力资源分成四层。

### 5.1 Tool

Tool 是最基础的可执行动作单元。

例如：

- 读文件
- 搜索文件
- 终端执行
- 网络获取
- PDF 分析

它的特点是：

- 面向直接动作
- 输入输出契约相对明确
- 可以被编排系统直接授权或禁止

### 5.2 Skill

Skill 是对若干 tool 使用方法的组织性资源。

它不是 task，不是 workflow，也不是 Agent。

它更像：

- 某类能力的使用说明
- 某类能力的思维提示
- 某类能力的局部方法包

所以：

- tool 是动作单元
- skill 是能力方法包

### 5.3 WorkerCapability

Worker 是平台里具备专门执行路线的工作单元。

它介于 tool 和 agent 之间。

它更适合：

- 需要专门 route 的执行能力
- 不直接暴露给模型的后台执行单元
- 特定操作链的包装层

### 5.4 CapabilityCatalog

Catalog 不是一种能力，而是能力目录层。

它负责统一描述：

- 现在平台有哪些资源
- 资源之间如何绑定
- 哪些资源模型可见
- 哪些资源仅内部可见

## 六、能力系统的对象模型

结合现有代码中的：

- `backend/capabilities`
- `backend/tools`
- `backend/capability_system/units/skills`
- `backend/skill_system`
- `backend/workers`
- `backend/capability_system/operation_registry.py`

我建议能力系统统一围绕八类对象组织。

### 6.1 ToolRecord

表示正式登记的工具资源。

它回答：

`这个工具是什么，怎么被调用，边界是什么`

建议字段：

- `tool_id`
- `name`
- `description`
- `input_contract_ref`
- `output_contract_ref`
- `read_only`
- `destructive`
- `concurrency_safe`
- `runtime_visibility`
- `resource_exposure_policy`

### 6.2 SkillRecord

表示正式登记的 skill 资源。

它回答：

`这个 skill 提供的能力方法包是什么`

建议字段：

- `skill_id`
- `name`
- `description`
- `allowed_tools`
- `prompt_view`
- `activation_policy`
- `context_mode`
- `output_rule`

### 6.3 WorkerCapabilityRecord

表示正式登记的 worker 能力。

它回答：

`这个后台执行单元是什么，通过哪条 route 暴露`

建议字段：

- `worker_id`
- `route`
- `name`
- `description`
- `operation_id`
- `agent_id`
- `transport`
- `model_visibility`

### 6.4 CapabilityBindingGraph

表示能力资源之间的绑定图谱。

现有代码中已经有同名对象雏形：

- [backend/capabilities/models.py](/d:/AI应用/langchain-agent/backend/capabilities/models.py)

它回答：

`skill、tool、worker、agent 之间是什么关系`

### 6.5 CapabilityManifest

表示平台能力资源的统一描述文件。

现有代码中最接近的是：

- [backend/capabilities/manifest.py](/d:/AI应用/langchain-agent/backend/capabilities/manifest.py)

它回答：

`平台当前正式发布的能力资源集合是什么`

### 6.6 CapabilityEndpoint

表示能力资源的查询与管理接口。

它回答：

`外部系统或前端如何读取能力目录与能力元数据`

### 6.7 CapabilityValidationIssue

表示能力资源目录中的问题。

现有代码里已有同名雏形。

它回答：

`这个能力资源目录哪里不完整、哪里冲突、哪里越界`

### 6.8 CapabilitySupplyPackage

表示能力系统对外供给的能力包。

它回答：

`任务系统或编排系统这次能拿到哪些能力资源`

建议字段：

- `package_id`
- `task_id`
- `agent_id`
- `tool_refs`
- `skill_refs`
- `worker_refs`
- `capability_constraints`
- `visibility_rules`

## 八、能力系统与任务系统的接口

任务系统不直接消费底层工具文件，而应消费能力系统给出的正式能力包。

建议固定接口：

- 输入：`TaskExecutionAssembly`
- 输出：`CapabilitySupplyPackage`

任务系统关心的是：

- 这个任务理论上需要什么能力类目
- workflow 里声明的 skill 是否存在
- 多 Agent 协调中各节点可用哪些能力

任务系统不应自己去扫描 `tools/skills` 目录。

## 九、能力系统与编排系统的接口

编排系统需要能力系统提供：

- 正式 operation catalog
- tool / skill / worker 引用集合
- 风险标签
- 输入输出契约引用

建议固定接口：

- 输入：`CapabilitySupplyPackage`
- 编排系统再生成：`ResourcePolicyProfile`

也就是说：

- 能力系统回答“平台有什么能力”
- 编排系统回答“这个 Agent 这次允许用什么能力”

## 十、能力系统与健康系统的接口

健康系统应能够检查：

- 能力资源目录是否完整
- tool / skill 绑定是否失配
- 某条任务链路失败是否因为能力供给缺失
- 某个运行体失败是否因为资源授权错误

所以能力系统必须对健康系统暴露：

- capability manifest
- validation issues
- binding graph
- supply package history

## 十一、能力系统与灵魂系统的关系

能力系统和灵魂系统必须分清楚。

能力系统提供的是：

- 能力资源
- 动作资源
- 工具资源

灵魂系统提供的是：

- 人格设定
- projection prompt
- 表达姿态

skill 虽然可能带 prompt view，但它仍属于能力方法包，不等于灵魂素材。

## 十二、能力系统的供给链路

建议将能力供给链路固定为：

```text
tools / skills / workers / operations registry
  -> capability manifest
  -> capability validation
  -> capability binding graph
  -> capability supply package
  -> task system / body system consumption
```

### 12.1 为什么要有 supply package

因为平台不能让每个系统都去直接读底层目录。

必须有一层正式供给包，把：

- 可见资源
- 允许引用资源
- 输入输出契约
- 风险标签

统一打包出来。

## 十三、前端能力系统管理目标

前端能力系统页面不应是工具列表页，而应是能力资源管理页。

至少应支持以下管理面：

### 13.1 Tool 资源管理

- 查看正式登记工具
- 查看输入输出契约
- 查看风险级别
- 查看运行可见性

### 13.2 Skill 资源管理

- 查看 skill 列表
- 查看 skill 允许调用哪些工具
- 查看 skill prompt view
- 查看 skill 适用场景

### 13.3 Worker / Operation 管理

- 查看 worker 列表
- 查看 route
- 查看 operation 绑定
- 查看模型可见性

### 13.4 能力图谱管理

- 查看 tool / skill / worker / agent 绑定关系
- 查看是否存在悬空绑定
- 查看建议修复项

### 13.5 能力供给预览

- 查看某个任务可能拿到的能力包
- 查看某个 Agent 当前身体能接到哪些能力

## 十四、现有问题与清理方向

当前项目的能力资源已经很多，但表达层次仍不够收束。

主要问题包括：

- skills、tools、capabilities、workers 仍偏分散
- manifest、registry、catalog 之间边界不够统一
- 任务系统和编排系统还没有只消费正式供给包
- 前端容易滑向“展示目录”，而不是“管理供给关系”

## 十五、实施顺序

建议按以下顺序实施：

1. 统一命名为能力系统能力资源层
2. 统一 Tool / Skill / Worker / Operation / CapabilityManifest 对象
3. 补出 `CapabilitySupplyPackage`
4. 让任务系统只消费能力包
5. 让编排系统只消费能力包和 operation catalog
6. 最后再重构前端能力资源管理页

## 十六、执行结论

能力系统不是平台里的杂项模块，而是正式的能力资源供应层。

它真正解决的是：

- 平台到底有哪些能力
- 这些能力如何被正式登记
- 这些能力如何被任务系统和编排系统稳定引用
- 这些能力如何被验证、治理和可视化

只有这条线立起来，平台里的 skill、tool、worker 和 capability 才不会继续散成一地，而会成为真正可管理、可供给、可验证的能力系统资源。

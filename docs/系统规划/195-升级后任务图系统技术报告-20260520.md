# 195-升级后任务图系统技术报告

日期：2026-05-20

## 1. 结论

当前任务系统已经从“任务表单 + 运行通道”升级为“任务图一等对象 + 契约化运行 + 批次生命周期 + 并行执行实例 + 健康系统接入”的通用架构。

这次升级解决的不是某个具体业务任务，而是三个基础问题：

1. 节点、边、图、图中图都可以被统一映射为任务图运行对象。
2. 任务图可以按契约拆成批次，并在运行时被安全地推进、返修、提交与合并。
3. 健康系统可以直接消费任务图 monitor 和 trace，把运行异常转成证据、恢复候选和问题投影。

## 2. 现在的系统形态

### 2.1 任务图是主语

任务系统不再以单个业务任务为中心，而以 `TaskGraphDefinition` 为中心。

任务图的核心对象包括：

- graph
- node
- edge
- contract_bindings
- runtime_spec
- execution_package
- coordination_run
- task_run
- stage_execution_request

图、子图、节点、边都可以进入同一套编译与运行管道。图不再只是展示层对象，而是 runtime 的主语。

### 2.2 契约是运行入口

运行语义不再散落在开发说明、默认 prompt 或前端表单里，而统一来自契约绑定：

- `unit_batch`
- `runtime.split_policy`
- `runtime.batch_acceptance_policy`
- `runtime.merge_policy`
- `working_memory_policy`
- `monitor_policy`
- `background_policy`
- `notification_policy`

系统的运行选择必须能回溯到契约，而不是靠节点名字猜测。

### 2.3 批次是运行时一等对象

批次生命周期已经成为可编译、可推进、可监控的运行态对象：

- ready
- running
- repair_ready
- repairing
- committed
- failed

这使“一个节点要处理多少单位、每批处理多少、何时能进入下一批、是否允许返修、何时合并”都成为显式状态机，而不是 prompt 内隐含逻辑。

### 2.4 并行实例是安全边界

并行能力不是简单“多跑几个请求”，而是给每个 batch 赋予独立 execution instance：

- `batch_execution_instances`
- `active_execution_by_batch`
- `active_execution_by_node`
- `active_executions_by_node`

结果回收必须匹配 request / dispatch / execution identity，找不到身份就拒绝归位，不允许回退到“当前 active batch”蒙混过关。

## 3. 升级后的核心框架

### 3.1 编译层

任务图编译器现在会把契约翻译成运行计划：

- split plan
- batch lifecycle plan
- merge readiness plan
- direct runtime spec
- execution package

这意味着图的结构、契约、发布包和运行态是连续的，不是四套彼此不认的东西。

### 3.2 运行层

运行时的主结构是：

- `task_system.batch_lifecycle_runtime_state`
- `task_graph.run_monitor`
- `runtime trace`
- `coordination checkpoint`

运行层负责的事情很清楚：

1. 输入绑定。
2. 批次选择。
3. execution identity 生成。
4. request / dispatch / result 归位。
5. accepted / rejected 推进。
6. checkpoint 续接。

### 3.3 监控层

`task_graph.run_monitor` 现在是任务图运行的标准观察面。它包含：

- graph topology
- node / edge statuses
- temporal boundary
- batch lifecycle
- batch dispatcher
- timeline ledger
- current execution request
- health issues

这让运行调试不再只看一条 trace，而是能看见整个图的运行状态。

## 4. 这次真正新增的能力

### 4.1 批次生命周期闭环

已完成：

- 批次拆分
- 批次审核
- 批次返修
- 批次提交
- merge readiness

这套闭环是通用的，不绑定写作、章节或小说。

### 4.2 并行批次实例

已完成：

- sequential / parallel split policy
- 并行上限
- execution instance 记录
- 乱序结果匹配
- unknown identity 拒绝提交

这使任务图开始具备“多单位并发处理”的真实工程能力。

### 4.3 批量派发闭环

已完成：

- 运行时批量领取 ready batch
- API 批量派发入口
- dispatcher 视图
- 前端/后端共用的 batch summary

这让任务系统具备 WorkerPool 的前置闭环，但仍然没有把真正 worker pool 变成新的独立系统。

### 4.4 健康系统接入

这次补上了最后一层闭环：健康系统可以直接消费任务图 monitor。

新增任务图健康投影：

- `health_system.task_graph_health_projection`

新增健康问题覆盖：

- monitor 原生 issue
- runtime failure
- batch dispatch blocked
- batch execution identity ignored
- batch execution inconsistent
- timeline violation
- missing runtime request

新增 API：

```text
GET /health-system/maintenance/test-system/task-graph/task-runs/{task_run_id}/health
```

健康系统现在不是只看 runtime-loop 摘要，而是能直接看到任务图拓扑、时序和批次层面的结构性问题。

## 5. 工程边界

### 5.1 已明确不做的事

- 不把健康系统变成运行控制器。
- 不写写作专用门禁。
- 不让 agent 在 prompt 中自发决定批次数量。
- 不用补丁式逻辑偷偷修复不一致状态。
- 不把任务图编辑器和运行控制混成一个面板。

### 5.2 当前仍保留的边界

1. 真正常驻 worker pool 还没有落成。
2. 动态 split 还只是预留方向，不是主线能力。
3. health system 负责诊断和恢复候选，不负责自动改图或自动恢复。
4. 复杂图编辑器虽然统一了契约和图层，但仍应继续按分层工作台做体验优化。

## 8. 结论

这次升级后，任务系统已经具备一个比较完整的工程轮廓：

- 图是运行主体。
- 契约是调度入口。
- 批次是生命周期单位。
- execution instance 是并行边界。
- monitor 是运行观察面。
- health system 是证据和恢复投影层。

下一阶段如果继续推进，最值得做的不是再造一套业务专用逻辑，而是把真正的 worker pool、队列化派发、暂停/恢复控制和图编辑器工作台做得更硬一些。

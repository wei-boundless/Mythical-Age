# 记忆系统详细设计书

日期：2026-05-04

## 一、文档定位

本文件用于定义本项目中记忆系统的正式职责、对象模型、上下游边界、运行链路和后续实施顺序。

本文件建立在 [00-全项目系统边界总设计书-20260504.md](/d:/AI应用/langchain-agent/docs/系统规划/00-全项目系统边界总设计书-20260504.md) 之上。

本文件重点回答以下问题：

- 记忆系统在平台中的本体职责是什么
- 记忆系统和任务系统、编排系统、灵魂系统、能力系统、健康系统分别如何分工
- 记忆不是“聊天记录”，那它到底由哪些层组成
- 任务执行时，记忆是如何被申请、筛选、装配和写回的
- 主 Agent、系统管理 Agent、工作子 Agent 在记忆上分别应该如何治理

本文件不是对现有代码的简单说明，而是后续记忆系统清理与重构的目标设计。

## 二、设计背景

在当前项目中，记忆相关能力已经分散存在于多个模块中，例如：

- `backend/memory_system`
- `backend/structured_memory`
- `backend/durable_memory`
- `backend/context_management`
- `backend/orchestration/runtime_loop/checkpoint.py`
- `backend/prompting/long_term_context.py`

这说明项目已经具备了会话记忆、长期记忆、结构化记忆、上下文压缩、checkpoint 等基础能力。

但目前仍存在三个结构性问题：

1. 记忆边界仍不够统一，很多地方把记忆、上下文、checkpoint、状态混着看
2. 记忆系统与任务系统、编排系统之间的接口还没有正式锁定
3. 运行时虽然在使用记忆相关内容，但“该给谁什么记忆、为什么给、何时写回”还没有上升为平台级契约

因此，记忆系统必须从“若干能力模块的集合”提升为平台的正式系统。

## 三、记忆系统总定义

记忆系统是平台的记忆治理层和记忆供给层。

它不直接执行任务，不直接决定任务分配，也不直接定义人格。

它的职责是：

- 组织记忆
- 治理记忆
- 供给记忆
- 写回记忆
- 控制记忆作用域

它回答的问题是：

`某个 Agent 在某个任务、某个入口、某个运行阶段，应该拿到哪些记忆，不应该拿到哪些记忆，以及执行后哪些内容值得写回`

## 四、记忆系统的硬边界

### 4.1 记忆系统不等于聊天记录

聊天记录只是原始输入材料之一。

真正的记忆系统应该处理的是：

- 哪些对话内容值得长期保留
- 哪些内容只应保留在当前任务
- 哪些内容应抽象成结构化事实
- 哪些内容只是运行噪音，不应进入记忆

### 4.2 记忆系统不等于 runtime state

runtime state 是当前执行态。

记忆是可跨阶段、可再利用、可被治理的上下文资产。

例如：

- 当前 step 的中间产物，不一定是记忆
- 任务失败的即时错误，不一定是记忆
- 用户长期偏好、系统长期约束、项目长期背景，才更接近记忆

### 4.3 记忆系统不等于编排系统的 memory scope 字段

`memory_scope` 是编排系统给运行时设定的约束。

但：

- 记忆内容是什么
- 记忆从哪里来
- 记忆如何筛选
- 记忆如何分层

这些不归编排系统，归记忆系统。

### 4.4 记忆系统不等于灵魂系统

灵魂系统提供 prompts 身份层。

记忆系统提供上下文延续层。

人格不是记忆，投影不是记忆，工作风格也不是记忆本体。

## 五、与其他系统的关系

### 5.1 与任务系统的关系

任务系统负责提出记忆需求。

例如：

- 这个任务是否需要项目长期背景
- 是否需要当前会话摘要
- 是否需要某个 Agent 的私有工作记忆
- 是否需要团队共享记忆

任务系统不直接拼接记忆内容。

它只声明：

- 需要哪类记忆
- 需要多大范围
- 需要什么优先级

### 5.2 与编排系统的关系

编排系统负责限制记忆作用域。

例如：

- 这个 Agent 是否允许访问长期记忆
- 是否允许访问跨 Agent 共享记忆
- 是否允许写回 durable memory
- 是否允许读取系统级 state memory

编排系统回答“能不能拿、能不能写”，记忆系统回答“具体拿什么、写什么”。

### 5.3 与灵魂系统的关系

灵魂系统不提供事实记忆，只提供 prompts 身份结构。

但灵魂系统会影响记忆的呈现方式，例如：

- 摘要的表达方式
- 呈现顺序
- 上下文组织形式

也就是说，记忆内容归记忆系统，记忆表达进入 prompts 时的姿态可受灵魂系统影响。

### 5.4 与能力系统的关系

能力系统为记忆系统提供底层能力与资源，例如：

- 检索工具
- 索引能力
- 文件存储
- structured data 处理

能力系统是能力供给者，不是记忆治理者。

### 5.5 与健康系统的关系

健康系统负责检测记忆链路是否健康，例如：

- 是否拿错了记忆
- 是否出现记忆污染
- 是否出现写回失败
- 是否出现作用域越界
- 是否出现摘要失真

健康系统不决定记忆内容，但负责监测记忆系统是否正常工作。

## 六、记忆系统的分层模型

记忆系统必须分层，不能用一个“memory”概念包住所有东西。

我建议平台统一采用五层模型。

### 6.1 L1：当前轮上下文

这是最短周期的上下文层。

特点：

- 生命周期最短
- 只服务当前轮或当前 step
- 不应直接写回长期记忆
- 可来自用户输入、当前文件读取、当前 tool 输出

它更接近运行上下文，不是稳定记忆本体。

### 6.2 L2：会话记忆

这是当前会话内的结构化记忆层。

特点：

- 生命周期覆盖单次会话
- 用于维持对话连续性
- 允许摘要化
- 允许压缩
- 会在 compact、checkpoint、resume 中持续参与

这层对应当前项目里已有的：

- `session_memory`
- `conversation_memory`
- `context_compactor`

### 6.3 L3：任务记忆

这是面向任务执行的局部记忆层。

特点：

- 生命周期覆盖单个任务或单类任务
- 可以保存任务过程中的稳定结论
- 可被工作子 Agent 复用
- 任务结束后不一定进入长期记忆

这层非常关键，因为你的平台是任务驱动平台。

后续很多“让 Agent 能持续完成复杂任务”的能力，真正依赖的是任务记忆，而不只是会话记忆。

### 6.4 L4：Agent 专属记忆

这是某个 Agent 的长期私有或半私有记忆层。

特点：

- 与 Agent 身份绑定
- 保存长期角色经验
- 保存适合该 Agent 持续复用的领域背景
- 可区分主 Agent、系统管理 Agent、工作子 Agent

例如：

- 主 Agent 的全局对话偏好记忆
- 健康管理 Agent 的健康分析惯例
- 工作子 Agent 在某类开发任务中的长期经验

### 6.5 L5：平台长期记忆

这是跨会话、跨任务、跨入口的持久化长期记忆层。

特点：

- 生命周期最长
- 需要最严格的写回门槛
- 主要保存稳定事实、用户偏好、项目背景、长期规则、系统演化信息

这层对应现有代码中类似：

- `durable_memory`
- `long_term_memory`
- `prompting/long_term_context.py`

## 七、平台中的记忆对象模型

为了避免记忆系统继续停留在散乱模块状态，建议统一定义以下对象。

### 7.1 MemoryRequest

由任务系统或 runtime 发起，表示“本次需要什么记忆”。

建议包含：

- `request_id`
- `task_id`
- `agent_id`
- `conversation_id`
- `requested_memory_layers`
- `requested_topics`
- `requested_scope`
- `max_items`
- `reason`

### 7.2 MemoryScopePolicy

由编排系统提供约束，表示“允许拿什么范围的记忆”。

建议包含：

- `policy_id`
- `agent_id`
- `allowed_layers`
- `allowed_sources`
- `allow_cross_agent_memory`
- `allow_long_term_read`
- `allow_long_term_write`
- `allow_task_memory_writeback`

### 7.3 MemoryCandidate

由记忆系统检索出的候选记忆项。

建议包含：

- `memory_id`
- `layer`
- `source_type`
- `source_ref`
- `summary`
- `structured_facts`
- `relevance_score`
- `staleness_score`
- `sensitivity`

### 7.4 MemoryBundle

由记忆系统最终供给给 runtime 的记忆包。

建议包含：

- `bundle_id`
- `agent_id`
- `task_id`
- `conversation_id`
- `included_memory_ids`
- `excluded_memory_ids`
- `memory_sections`
- `summary_view`
- `supply_reason`
- `writeback_targets`

### 7.5 MemoryWritebackRecord

表示一次执行后哪些内容被写回了哪里。

建议包含：

- `writeback_id`
- `task_id`
- `agent_id`
- `source_refs`
- `target_layer`
- `target_memory_id`
- `writeback_reason`
- `writeback_mode`
- `approved_by_policy`

## 八、不同 Agent 的记忆治理策略

### 8.1 主 Agent

主 Agent 默认拥有最广的对话承接范围，但不意味着默认拥有最广的记忆访问权限。

建议：

- 可以访问会话记忆
- 可以访问平台长期记忆中的全局用户偏好和项目背景
- 不应自动读取所有系统管理 Agent 的私有工作记忆
- 在需要系统管理任务时，应通过任务分流而不是直接吃掉所有子系统记忆

### 8.2 系统管理 Agent

系统管理 Agent 的记忆应按系统域隔离。

例如：

- 记忆管理 Agent 优先访问记忆治理相关记忆
- 健康管理 Agent 优先访问诊断与验证相关记忆
- 操作管理 Agent 优先访问能力资源与工具约束相关记忆

它们不应默认共享彼此的私有工作记忆。

### 8.3 工作子 Agent

工作子 Agent 的记忆策略应更克制。

建议：

- 默认只拿任务记忆与必要会话摘要
- 默认不直接拿平台全部长期记忆
- 可在特定任务中申请额外记忆层
- 任务结束后写回必须过筛选门槛

## 九、任务执行中的记忆链路

任务执行时，记忆链路建议固定为：

```text
任务系统识别任务
  -> 任务系统形成 MemoryRequest
  -> 编排系统提供 MemoryScopePolicy
  -> 记忆系统检索候选记忆
  -> 记忆系统生成 MemoryBundle
  -> 编排系统校验是否可装入 runtime
  -> runtime 消费 MemoryBundle
  -> 执行结束后形成 MemoryWritebackRecord
  -> 记忆系统决定是否写回
  -> 健康系统记录链路健康状态
```

### 9.1 为什么要先 request 再 supply

因为任务系统知道任务要什么，记忆系统知道有哪些记忆，编排系统知道哪些能给。

这三者不能颠倒。

### 9.2 为什么不能让 runtime 自己随便拼记忆

因为那会导致：

- 记忆越界
- 记忆污染
- 记忆选择不可审计
- 同类任务表现不稳定

所以 runtime 必须消费正式的 `MemoryBundle`，而不是自己临时乱取。

## 十、记忆写回原则

写回是记忆系统里风险最高的动作之一。

必须遵守以下原则：

### 10.1 不把噪音写回长期记忆

以下内容默认不应写回长期记忆：

- 临时命令输出
- 中间调试日志
- 瞬时失败信息
- 未验证结论
- 当前 step 的局部产物

### 10.2 只写回稳定事实或稳定偏好

适合写回的内容包括：

- 用户长期偏好
- 项目长期背景
- 长期有效的任务经验
- 多次验证过的系统规则
- 对某 Agent 持续有用的稳定结论

### 10.3 写回必须区分目标层

不是所有写回都进长期记忆。

要区分：

- 写回会话记忆
- 写回任务记忆
- 写回 Agent 专属记忆
- 写回平台长期记忆

### 10.4 写回应可审计

每次写回都应该可追溯：

- 为什么写
- 从哪里来
- 写到哪里
- 被谁批准

## 十一、前端记忆系统管理目标

前端的记忆系统页面不是做成“记忆总览面板”，而是做成真正可治理的管理页面。

至少要支持以下管理面：

### 11.1 记忆层次管理

- 查看五层记忆模型
- 查看各层当前数据来源
- 查看层与层之间的流动关系

### 11.2 Agent 记忆策略管理

- 查看各 Agent 默认记忆作用域
- 查看哪些 Agent 可读长期记忆
- 查看哪些 Agent 可写回长期记忆
- 查看哪些 Agent 只能读任务记忆

### 11.3 任务记忆配置管理

- 查看某类任务默认申请哪些记忆层
- 查看任务是否允许任务记忆写回
- 查看任务对长期记忆的访问需求

### 11.4 写回治理管理

- 查看最近写回记录
- 查看写回失败
- 查看越界写回风险
- 查看低质量记忆污染

## 十二、现有代码的吸收与清理方向

当前项目里已有的记忆相关模块不应简单推倒，但必须按新边界收束。

### 12.1 应吸收的已有能力

- `memory_system` 中的会话 / 长期 / 状态记忆分层
- `structured_memory` 中的结构化视图和管理能力
- `context_management` 中的压缩与摘要能力
- `checkpoint` 中的恢复与续接能力
- `long_term_context` 中的长期记忆注入能力

### 12.2 应清理的问题

- 同类记忆能力在多个目录重复表达
- “context”和“memory”概念混用
- 任务系统与记忆供给接口不正式
- runtime 自己隐式拼装记忆
- 写回策略分散在局部逻辑中

## 十三、实施顺序

建议按以下顺序实施：

1. 统一记忆系统对象命名
2. 定义 `MemoryRequest / MemoryScopePolicy / MemoryBundle / MemoryWritebackRecord`
3. 建立任务系统到记忆系统的正式申请接口
4. 建立编排系统对记忆作用域的正式约束接口
5. 让 runtime 只消费 `MemoryBundle`
6. 重构写回链路
7. 增加健康系统对记忆链路的诊断能力
8. 最后再重构前端管理界面

## 十四、执行结论

记忆系统在本平台中不是附属模块，而是平台六大核心系统之一。

它负责的不是“保存更多内容”，而是：

- 决定什么值得保留
- 决定保留在哪一层
- 决定给谁看
- 决定什么时候写回
- 决定如何让记忆为任务执行服务

后续只要这条线锁定住，任务系统、编排系统和 runtime 才能真正稳定地让 Agent 持续完成复杂任务。

# 多会话并行流式前端状态修复计划

日期：2026-05-15

## 问题

当前前端把流式状态建模为全局单例：

- `isStreaming` 表示全局任意请求正在流式。
- `messages` 只保存当前会话消息。
- `orchestrationSnapshot` 只保存当前会话运行态。
- 流式回调直接写当前 store。

因此 A 会话正在流式时切到 B，会出现两个问题：

1. A 的流式临时消息容易被当前会话切换覆盖，切回 A 时体验像“卡掉”。
2. 全局 `isStreaming` 锁住 B 的输入，导致不能像主流聊天产品那样多会话并行生成。

## 目标

让前端支持多会话并行流式：

- A 会话流式时可以切到 B。
- B 可以独立发起新的流式请求。
- 每个会话的流式消息、工具状态、编排快照互不污染。
- 切回正在流式的会话时，能看到该会话当前的流式缓存。
- 同一个会话内仍然避免重复提交并行请求。

## 实施步骤

1. Store 状态增加 `activeStreamSessionIds`，表示正在流式的会话集合。
2. Runtime 内部维护 `streamingSessionCache`，按 `session_id` 保存该会话的 `messages` 与 `orchestrationSnapshot`。
3. `sendMessage` 改为只检查当前 `session_id` 是否在 `activeStreamSessionIds` 中，不再用全局 `isStreaming` 阻止所有会话。
4. 流式回调只更新对应会话缓存；如果该会话正是当前会话，再同步到可见 store。
5. 切换会话时，如果目标会话有流式缓存，直接显示缓存；否则清空可见消息后从后端加载历史。
6. `ChatPanel/ChatInput` 的禁用态改为“当前会话是否正在流式”，而不是全局是否有任意会话流式。
7. 保留 `isStreaming` 作为兼容字段，但语义调整为“是否存在任意会话正在流式”。

## 验证

- `npm run build` 必须通过。
- 人工验证路径：
  - A 会话发送长任务。
  - 生成中切到 B。
  - B 输入框可用并能发送。
  - 切回 A 能看到 A 的流式内容继续增长或最终刷新。
  - A/B 的协调监控不互相覆盖。

# Agent 记忆系统重构实施计划书

日期：2026-05-27

状态：重构计划书 / 待审阅 / 未实施

关联设计书：

- `docs/系统规划/223-Agent记忆系统框架与Agent交互协议设计书-20260527.md`

适用范围：

- `backend/memory_system/*`
- `backend/api/memory.py`
- `backend/runtime/*` 中消费记忆上下文、工作记忆、长期记忆、正式记忆仓库的入口
- `backend/tests/*memory*regression.py`

## 1. 总结论

本次重构不以“把大文件拆小”为目标，而以“收回记忆系统决策权、减少入口数量、合并职责域、删除旧权威路径”为目标。

当前 `backend/memory_system` 的问题不是 16k 行本身，而是把多套记忆系统和多种决策权混在一个包里：

```text
session / foreground continuity
working memory
task durable memory
formal repository memory
durable markdown notes
long-term recall
runtime context supply
maintenance / governance
facade / API pass-through
```

重构目标：

1. 让运行时记忆读取统一经过 `MemoryOrchestrator` 的 plan。
2. 让 agent 只输出 proposal 或 candidate id，不能直接拥有读取、提交、覆盖权。
3. 让 storage 只负责存取，不再隐藏“是否读、读哪些、是否接续”的语义判断。
4. 将现有 56 个 Python 文件收敛为更少、更稳定的职责域文件，不做机械碎片化拆分。
5. 删除旧的无计划长期记忆注入、关键词权威判断、万能 facade 入口。

### 2.3 当前最大架构风险

当前问题不是“模块名不清楚”，而是权威边界混乱：

| 权威类型 | 当前位置 | 问题 |
|---|---|---|
| 是否读长期记忆 | `durable.py` / `runtime_view.py` / request profile | 多处共同决定，容易重复召回 |
| 读哪些层 | `MemoryBundleService` / `runtime_view.py` / runtime profile | plan 不显式，难追踪 |
| 当前轮接续事实 | `session_memory.py` / foreground state / resolver | restore、weak signal、current fact 容易混 |
| working memory 节点读取 | `working_memory_service.py` | 授权、选择、记录、渲染在一个方法里 |
| 长期记忆写入 | `maintenance_coordinator.py` | agent 结果和系统 commit 没有足够清晰分层 |
| API / runtime public surface | `MemoryFacade` / `__init__.py` | 入口过宽，旧路径难删除 |

## 3. 重构原则

### 3.1 权威优先

本次重构遵循：

```text
System owns state, policy, lifecycle, budget, commit.
Agent provides semantic judgment, summary, ranking, proposal, execution.
System validates every agent output before it becomes state.
```

具体映射：

| 层 | 可做 | 不可做 |
|---|---|---|
| Storage | 存取、索引、事务、版本 | 判断当前轮意图、决定是否注入 |
| Runtime Supply | 根据 plan 拉候选 | 自行绕过 plan 读长期记忆 |
| Orchestrator | 生成 ReadPlan / InjectionPlan / WritePlan | 直接执行 agent 任务 |
| Selector Agent | 在候选池内返回 candidate id | 自由搜索全库、拼接新记忆内容 |
| Maintainer Agent | 提维护建议 | 直接写长期记忆 |
| Committer | 校验并提交 | 根据自然语言猜测用户意图 |

### 3.2 文件治理

用户已明确指出：文件太多会增加阅读和维护成本。因此本次重构禁止“看似拆小、实际碎片化”的做法。

硬约束：

1. 一个职责域最多 3-5 个生产文件。
2. 不做“一类一文件”的机械拆分。
3. 新增 helper 文件必须满足至少一个条件：
   - 它承载稳定公共协议。
   - 它隔离了一个明确的持久化边界。
   - 它删除了一个旧决策权混杂点。
4. 少于 120 行、只被一个文件使用的 helper，优先合并回调用方。
5. 单文件可接受 300-800 行；数据库 schema/store 文件可接受到 1,000 行左右，但必须分清 schema、mapper、transaction。
6. `facade.py` 目标少于 250 行，只做组合根或少量兼容入口。
7. `__init__.py` 不再维护大量 lazy export，公共 API 必须收敛。

目标不是把 56 个文件拆到 80 个，而是收敛到约 32-38 个生产文件。

### 3.3 不保留旧权威兼容壳

允许短期 shadow / cutover，但不允许长期双轨。

允许：

- shadow plan diagnostics
- 迁移期 API adapter
- 带删除条件的临时 wrapper

禁止：

- 旧路径和新路径长期同时决定是否读记忆
- 无 plan 时回退到自由读长期记忆
- 用关键词表继续做 Need Classifier 权威判断
- 为旧结构保留测试 fossil

## 4. 目标结构

目标结构按职责域合并，不按类机械拆分：

```text
backend/memory_system/
  __init__.py
  contracts.py
  core_models.py
  facade.py
  runtime_supply.py
  continuity.py
  durable_notes.py
  durable_governance.py
  working_memory.py
  task_durable_memory.py
  formal_repository_models.py
  formal_repository_store.py
  formal_repository_service.py
  formal_repository_content.py
  maintenance.py
  runtime_services.py
  adapters.py
  utils.py
```

说明：

- `runtime_supply.py` 承载 `MemoryOrchestrator`、`MemorySupplier`、`ContextBuilder`、`SurfacedLedger` 的热路径实现；如果超过 900 行，再拆 `runtime_supply_store.py`，不预先碎片化。
- `continuity.py` 合并 foreground、session layer、session projection 的系统控制入口；底层复杂 process state 可保留为内部结构，但不能继续拥有 task switch 权威。
- `durable_notes.py` 合并 markdown note store、manifest、exact lookup、render、召回候选生成；关键词评分只能作为底层 candidate scorer。
- `durable_governance.py` 合并 governance service、note hygiene、maintenance-facing durable note 操作。
- `working_memory.py` 合并 models/service/store 的公共入口，但内部用私有函数分区；如果 SQLite/JSON store 过重，可保留 `working_memory_store.py`。
- `task_durable_memory.py` 合并 task durable models/store/service，减少 3 个文件的来回跳转。
- `formal_repository_*` 保留 4 个文件，因为正式仓库有独立 schema、事务和内容 materialization，不强行压成一个巨文件。
- `maintenance.py` 合并 maintenance agent/coordinator/models 的主入口，明确 agent proposal 与 system commit 的边界。
- `adapters.py` 合并 conversation/state/long_term 这些 runtime adapter，避免每个 adapter 一个文件。

目标文件治理表：

| 职责域 | 目标文件数 | 原因 |
|---|---:|---|
| runtime supply / orchestrator | 1-2 | 热路径必须集中可读 |
| continuity | 1-3 | 接续状态需要统一权威 |
| durable notes | 2 | note store 与治理分离 |
| working memory | 1-2 | 保持任务内记忆阅读体验连贯 |
| task durable memory | 1 | 当前复杂度不需要 3 文件 |
| formal repository | 4 | DB schema/transaction 复杂，允许独立 |
| maintenance | 1 | proposal/commit 分层在同一文件内更容易审查 |
| adapters/contracts/utils | 3-4 | 只保留稳定公共边界 |

## 5. 目标权威链

目标运行链路：

```text
UserTurn
  -> TurnCoordinator
  -> UnderstandingAgent proposal
  -> ContextResolver system binding
  -> MemoryOrchestrator read/write/injection plan
  -> MemorySupplier candidate fetch
  -> MemorySelectorAgent candidate ranking
  -> ContextBuilder sealed context package
  -> TaskAgent execution
  -> MemoryMaintainerAgent maintenance proposal
  -> MemoryCommitter validation and commit
```

本次记忆系统重构只覆盖：

```text
MemoryOrchestrator
MemorySupplier
MemorySelectorAgent protocol boundary
ContextBuilder
MemoryMaintainerAgent proposal boundary
MemoryCommitter
memory stores
runtime/API facades
```

不在本次重构内直接重写：

- 主模型任务决策系统
- 全量任务图 runtime
- 外部 sandbox/runtime 重做
- 前端工作台 UI

### Phase 1：落地核心模型与 shadow plan

目标：先有 plan 对象和 diagnostics，不改变 runtime 行为。

新增/调整：

- 新增或合并到 `backend/memory_system/core_models.py`
  - `MemoryReadPlan`
  - `MemoryInjectionPlan`
  - `MemoryWritePlan`
  - `MemoryCandidatePool`
  - `AgentProposal`
  - `SurfacedMemoryLedgerEntry`
  - `TemporalFactEdge`
- 新增或合并到 `backend/memory_system/runtime_supply.py`
  - `MemoryOrchestrator.build_shadow_plan`

现有文件动作：

- `supply.py` 的 `MemoryRequest` / `MemoryScopePolicy` 迁入 `core_models.py`，或在迁移期由 `core_models.py` 重新导出。
- `bundle_service.py` 在 diagnostics 中附带 shadow read plan。
- `runtime_view.py` 保持现有行为，但 diagnostics 标出：
  - `plan_mode=shadow`
  - `legacy_layer_decision=true`
  - `long_term_decision_source=legacy_profile`

禁止：

- 不接管读取。

### Phase 2：Orchestrator 接管 runtime 读取

目标：`MemoryBundleService` 不再自己决定读哪些 memory layer，而是消费 `MemoryReadPlan`。

调整：

- `runtime_supply.py`
  - `MemoryOrchestrator.build_read_plan`
  - `MemorySupplier.fetch_candidates`
  - `ContextBuilder.build_context_package`
- `bundle_service.py`
  - 改成薄 wrapper。
  - `build_memory_runtime_view` 调 Orchestrator。
  - `build_memory_context_package_result` 只消费 plan + candidates。
- `runtime_view.py`
  - `build_memory_runtime_view` 不再直接调用每个 layer。
  - 保留 `MemoryRuntimeView` 数据结构。
- `durable.py`
  - `recall_memories` 保留为 durable note candidate provider。
  - `_should_attempt_recall` 不再作为 runtime 权威。

读取规则：

```text
无 ReadPlan -> 不读取 long-term
ReadPlan.required 缺失 -> fail closed 或返回 explicit_missing_required
long-term read 必须有 reason + token_cap + selector
```

完成标准：

- long-term 读取只能从 Orchestrator plan 触发。
- `runtime_view` diagnostics 中不存在 `legacy_layer_decision=true`。
- `bundle_service.py` 行数下降，成为 orchestration wrapper。

测试：

- foreground/working 足够时 long-term 不读。
- profile 要求 long-term 但 policy 不允许时不读。
- required working memory 缺失时出现 explicit missing diagnostic。
- API recall preview 可继续单独调用 durable recall，但必须标记为 management/preview path，不进入 runtime 热路径。

- `__init__.py` 明显收敛。

### Phase 4：削薄 Facade 与公共 API

目标：删除万能 facade 对旧路径的保活作用。

调整：

- `facade.py`
  - 只保留 composition root。
  - 或拆为三个明确 facade：
    - `RuntimeMemoryFacade`
    - `GovernanceMemoryFacade`
    - `MaintenanceMemoryFacade`
  - 若拆 facade，不单独建 3 个文件，可放在 `facade.py`。
- `__init__.py`
  - 删除大量 lazy export。
  - 只导出稳定公开类型和 facade。
- `backend/api/memory.py`
  - management API 走 governance/maintenance facade。
  - runtime preview 走 runtime facade。
- runtime 侧：
  - 不直接绕过 runtime facade 访问 durable recall。

完成标准：

- `facade.py` 少于 250 行。
- facade 不再暴露 working/task/formal/durable 的所有 CRUD pass-through。
- 所有 runtime memory read 可以追溯到 Orchestrator plan。

### Phase 5：维护 agent 与 Committer 分离

目标：实现“agent proposal / system commit”。

调整：

- `maintenance.py`
  - `MemoryMaintenanceProposal`
  - `MemoryMaintainerAgent`
  - `MemoryCommitter`
  - `MemoryMaintenanceCoordinator`
- durable note 写入：
  - create/update/merge/supersede 必须进入 Committer。
  - source refs、scope、confidence、evidence 必须校验。
- working/task/formal 写入：
  - 不在本阶段全量重写，但接口预留 `WritePlan`。

完成标准：

- maintainer agent 不直接调用 store save。
- 无 provenance 的长期记忆写入被拒绝。
- 冲突只进入 pending conflict，不静默覆盖。

测试：

- maintainer 返回无 source refs -> rejected。
- update 旧记忆 -> 保留旧版本或 superseded edge。
- merge 重复记忆 -> provenance 合并。

### Phase 6：轻量时间事实链

目标：让旧事实不会被无痕覆盖。

实现位置：

- 优先放在 `core_models.py` + durable/formal store 内部。
- 不单独创建大型 graph 子系统。

模型：

```text
TemporalFactEdge
  source_memory_id
  target_memory_id
  relation: supersedes / invalidates / refines / conflicts_with
  source_refs
  confidence
  created_at
```

完成标准：

- 长期记忆 update/merge/supersede 记录边。
- recall 默认过滤 invalidated/superseded，但 diagnostics 可查看。
- 不引入图数据库。

### Phase 7：删除旧权威路径与旧测试 fossil

目标：新架构接管后清理旧壳。

删除/改写：

- `durable._should_attempt_recall` 作为权威判断的使用。
- `runtime_view` 直接按 profile 拉 long-term 的逻辑。
- `storage/session_memory._goal_domain` 对 task switch 的权威影响。
- `storage/memory_manager` 中 preference/work marker 加分对是否读取的影响。
- `bundle_service.py` 旧直接构建 runtime view 的路径。
- `__init__.py` 对旧模块的大量 lazy exports。

测试处理：

- 保留行为测试：
  - candidate-only
  - restore 不覆盖 current turn
  - working memory isolation
  - formal memory directed read
  - task durable context
- 删除或改写结构测试：
  - 只为了 import `MemoryFacade` 某个旧 pass-through 的测试。
  - 只为了证明旧文件存在的测试。

完成标准：

- `rg "_should_attempt_recall|_goal_domain|legacy_layer_decision|memory_system.supply|request_service" backend` 不再出现 runtime 热路径引用。
- 所有 memory read 都有 plan diagnostics。
- 旧文件删除后测试仍通过。

## 10. 切换与回滚规则

### 10.1 切换顺序

```text
shadow plan
  -> controlled read
  -> domain merge
  -> facade/API shrink
  -> maintainer proposal + committer
  -> temporal edge
  -> legacy authority deletion
```

### 10.2 回滚规则

允许回滚：

- 回滚到上一阶段 plan 接口。
- 暂停 maintainer/committer 后台维护。
- API management preview 暂时走 governance facade。

禁止回滚：

- 不允许回滚到 query 非空就自由 recall long-term。
- 不允许恢复关键词 Need Classifier 作为权威。
- 不允许恢复万能 facade pass-through 作为长期方案。

### 10.3 Cutover 门槛

Phase 2 接管读取前必须满足：

- shadow plan 连续通过相关回归测试。
- `MemoryRuntimeView` candidate-only 约束仍通过。
- long-term read 的 plan reason 已枚举。

Phase 3 合并文件前必须满足：

- Controlled read 已稳定。
- 所有 import 迁移路径明确。
- 每个目标职责域文件数已锁定。

Phase 7 删除旧路径前必须满足：

- 静态搜索确认旧路径不在 runtime 热路径。
- 行为回归全部通过。
- 没有依赖旧模块名的生产 import。

## 11. 禁止事项

实施中禁止：

1. 为了快速通过测试保留旧无计划读取路径。
2. 为了兼容旧 import 在 `__init__.py` 继续无限加 lazy export。
3. 把每个 helper、validator、mapper 都拆成独立文件。
4. 让 selector agent 返回完整记忆内容。
5. 让 maintainer agent 直接写长期记忆。
6. 用关键词表扩展 Need Classifier。
7. 让 session restore candidate 覆盖当前显式输入。
8. 为旧文件存在性保留测试。
9. 在没有 cutover 条件时长期双轨。
10. 把开发说明写进 agent prompt。

## 12. 成功标准

工程标准：

- `backend/memory_system` 生产 Python 文件数从 56 降到约 32-38。
- `facade.py` 少于 250 行。
- `__init__.py` 公共导出明显收敛。
- runtime memory read 全部可追溯到 `MemoryReadPlan`。
- long-term 默认不读，除非 Orchestrator plan 允许。
- maintainer agent 只产生 proposal，Committer 才能写入。
- storage 不再拥有当前轮接续或读取层选择权。

用户体验标准：

- 用户说“继续第二个”时，系统从 foreground/bundle refs 接续，不靠关键词猜。
- 用户显式输入不会被旧记忆覆盖。
- 普通 turn 不会无意义注入长期记忆。
- compact 后可从 accepted working memory 和 foreground continuity 恢复。
- 子 agent 输出不会污染主线程记忆。
- 用户纠正能进入可审计、可失效、可合并的长期偏好或流程记忆。

## 13. 实施边界

本计划书只是实施计划，不直接改代码。

用户审阅通过后，实施应一次性按阶段推进到可验证收尾；如果中途发现当前 runtime 重做导致接口不可用，需要先更新本计划的入口清单，再继续实施。

实施优先级：

```text
先控制读取权威
再合并职责域文件
再削薄 facade/API
最后删除旧路径
```

核心判断：

```text
不是越拆越先进。
先进的是权威清晰、候选受控、提交可审计、文件数量可读。
```

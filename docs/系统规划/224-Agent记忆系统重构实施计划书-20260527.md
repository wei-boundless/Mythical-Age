## 1. 当前结论

本计划书已按当前代码状态重写。上一版计划中的基线是 `56` 个 Python 文件、约 `16k` 行；当前已经完成一轮清理，真实基线变为：

| 指标 | 当前数量 |
|---|---:|
| Python 文件 | 40 |
| 总行数 | 13,204 |
| 非空行 | 12,081 |

已经删除或降权的旧路径：

- `backend/memory_system/storage/process_engine.py`
- `backend/memory_system/storage/turn_projection.py`
- `backend/memory_system/storage/exact_lookup.py`
- durable 侧 `_should_attempt_recall` / `no_memory_signal` 读取门控
- session projection 中的关键词式 task switch 判断
- durable exact/term overlap lookup 作为召回权威的路径

因此后续不再围绕“旧单 agent 循环依赖扫描链”做计划，而是进入第二轮：收束 runtime 读取权威，把 `MemoryReadPlan` 从数据视图中提升为唯一读取计划。

## 3. 当前最大架构差距

理想链路：

```text
RequestFacts
-> MemoryReadPlan / MemoryWritePlan
-> CandidateFetch
-> Selector / ContextPolicy
-> SealedContextPackage
-> TaskAgent
-> MaintenanceProposal
-> MemoryCommitter
```

当前链路仍有偏差：

```text
MemoryBundleService
  -> _context_package_memory_profile 可能默认补 state
  -> relevant_notes 可能隐式打开 long_term
  -> runtime_view.build_memory_runtime_view 同时建 plan 和拉候选
  -> context_system.policy 只看到候选，不知道 plan 是否越权
```

这意味着系统还有两个不合格点：

1. `ContextBuilder` 之前的层会在无显式计划时扩大读取范围。
2. `runtime_view.py` 作为 view 文件，却仍持有读取计划和候选拉取逻辑。

这不是“文件名不好看”的问题，而是读取权威没有封死。

## 4. 权威边界

本轮继续采用以下边界：

```text
System owns state, policy, lifecycle, budget, commit.
Agent provides semantic judgment, summary, ranking, proposal, execution.
System validates every agent output before it becomes state.
```

具体到记忆系统：

| 层 | 允许 | 禁止 |
|---|---|---|
| `MemoryOrchestrator` | 生成 `MemoryReadPlan` / `MemoryWritePlan` | 拉取候选内容、拼 prompt |
| `MemorySupplier` | 严格按 plan 拉候选 | 自行决定读 long-term/state/working |
| `ContextBuilder` / `ContextPolicy` | 在候选池内做预算和打包 | 扩展读取层、补默认 state、补默认 long-term |
| `Selector Agent` | 在候选 id 内排序/选择 | 自由搜索全库、返回自造记忆正文 |
| `Maintainer Agent` | 输出维护 proposal | 直接写 durable/working/formal memory |
| `MemoryCommitter` | 校验证据并提交 | 根据自然语言猜测当前轮意图 |
| Storage | 存取、索引、事务 | 决定当前轮是否读取、是否接续、是否覆盖 |

## 6. 下一阶段实施计划

### Phase B：runtime 读取权威收束

状态：当前执行阶段。

目标：

- `MemoryReadPlan` 是 runtime 读取唯一计划。
- `MemoryBundleService` 不再自行默认补 `state`。
- `relevant_notes` 不能隐式打开 `long_term`。
- `runtime_view.py` 只保留 read-only view，不再负责 plan 和 fetch。

文件动作：

| 文件 | 动作 |
|---|---|
| `backend/memory_system/runtime_supply.py` | 新增或改造为 `MemoryOrchestrator` + `MemorySupplier` + `build_memory_runtime_view` |
| `backend/memory_system/runtime_view.py` | 只保留 `MemoryRuntimeView` 和轻量 layer normalization |
| `backend/memory_system/bundle_service.py` | 改成消费 `runtime_supply`，删除 `_context_package_memory_profile` 的扩层逻辑 |
| `backend/api/memory.py` | 管理/预览入口必须显式声明读取 profile，不再依赖 bundle 默认值 |
| `backend/tests/*memory*` | 改写隐式读取测试，新增 plan 不能扩层的回归测试 |

读取规则：

```text
无 requested_memory_layers -> 不读任何 memory layer
有 relevant_notes 但无 long_term plan -> 不读 long_term
long_term 必须同时满足 requested_memory_layers 包含 long_term 且 allow_long_term_memory=true
state 必须显式 requested_memory_layers 包含 state
working/task_durable 必须显式 requested_memory_layers 包含对应层
ContextPolicy 只能在候选内预算和裁剪，不能新增候选
```

完成标准：

- `rg "_context_package_memory_profile|legacy_layer_decision" backend/memory_system backend/tests` 无生产热路径残留。
- `build_memory_context_package_result(..., relevant_notes=[...])` 不再默认产生 durable context。
- `build_memory_context_package_result(..., memory_request_profile={"requested_memory_layers": ["state"]})` 仍能产生 state context。
- `build_memory_context_package_result(..., memory_request_profile={"requested_memory_layers": ["long_term"], "allow_long_term_memory": True}, relevant_notes=[...])` 才能产生 durable context。
- 相关回归测试通过。

### Phase C：session continuity 单权威化

状态：Phase B 后执行。

目标：

- 系统结构化 `process_state` 是接续事实权威。
- markdown session view 变成 render/view，不再承担事实提交权威。
- foreground continuity 负责当前工作现场的快照，不靠 durable note 恢复当前轮事实。

文件动作：

| 文件 | 动作 |
|---|---|
| `backend/memory_system/continuity.py` | 承接 foreground + session continuity 的系统入口 |
| `backend/memory_system/storage/session_memory.py` | 降为 session view/projection renderer |
| `backend/memory_system/storage/process_state.py` | 保持结构化状态权威，必要时合入 continuity domain |
| `backend/memory_system/maintenance_coordinator.py` | 维护 agent 只写 proposal，系统 committer 决定 state 更新 |

禁止：

- 恢复 `_goal_domain` 或同类关键词领域判断。
- 让 markdown view 反向覆盖当前显式输入。
- 让 maintainer agent 直接写当前轮状态。

### Phase D：维护 proposal 与 Committer 分离

状态：Phase C 后执行。

目标：

- `MemoryMaintainerAgent` 只输出 proposal。
- `MemoryCommitter` 校验 source refs、scope、confidence、冲突关系后提交。
- durable note 的 create/update/merge/supersede 不再由 agent 结果直达 store。

必要测试：

- maintainer 无 source refs -> rejected。
- update 旧记忆 -> 产生 supersedes/refines/conflicts edge。
- merge 记忆 -> provenance 合并。
- 冲突 -> pending conflict，不静默覆盖。

### Phase E：working/formal 读取路径收束

状态：Phase D 后执行。

目标：

- `WorkingMemoryService.select_for_node` 拆成内部步骤：policy evaluation、candidate selection、read log、handoff/temporal expansion。
- formal memory 默认 scope 逐步 fail-closed，不再产生 `unbound_run` / `default_project` 这类隐式范围，除非管理入口显式声明。

动作：

| 文件 | 动作 |
|---|---|
| `backend/memory_system/working_memory_service.py` | 不急着拆文件，先拆内部权威步骤 |
| `backend/memory_system/formal_memory_service.py` | 显式化 scope resolution，管理/测试入口单独标注 |

### Phase F：职责域合并与文件健康治理

状态：最后执行，不能先做。

原则：

- 先封权威，再合并文件。
- 不做“一类一文件”的机械拆分。
- 少于 120 行、只被一个文件使用的 helper，优先合回调用方。
- 单文件 300-800 行可接受；DB store 可接近 1,000 行，但必须分清 schema、mapper、transaction。

文件数量只作为健康指标，不作为设计驱动。是否合并文件，取决于职责边界是否稳定、阅读跳转是否反复打断理解、是否存在旧壳保活旧权威。

健康判断：

| 信号 | 解释 |
|---|---|
| 文件减少 | 只说明跳转成本可能下降，不代表架构更好 |
| 单文件变大 | 如果承载同一稳定职责域，可以接受 |
| helper 文件存在 | 只有承载稳定协议、持久化边界、或跨域复用时才合理 |
| 文件数量上升 | 必须解释新增职责域，否则视为结构松散风险 |

优先合并方向：

- `task_durable_memory_models.py` + `task_durable_memory_store.py` + `task_durable_memory_service.py` -> `task_durable_memory.py`
- `maintenance_agent.py` + `maintenance_models.py` + `maintenance_coordinator.py` -> `maintenance.py`
- durable note store/governance 归并为 `durable_notes.py` + `durable_governance.py`
- `supply.py` 与 runtime supply/core model 在 Phase B 稳定后再合并，不提前做文件搬家

## 10. 成功标准

工程标准：

- runtime memory read 全部可追溯到 `MemoryReadPlan`。
- `ContextPolicy` 不再扩展读取层。
- long-term 默认不读，必须由 plan 显式允许。
- state 默认不读，必须由 plan 显式允许。
- storage 不拥有当前轮读取层选择权。
- 文件结构按逻辑职责稳定，不为凑数量拆分或合并。

用户体验标准：

- 普通 turn 不会无意义注入长期记忆。
- 用户显式输入不会被旧状态或旧记忆覆盖。
- “继续第二个”依靠 foreground/state/bundle refs 接续，不靠关键词猜。
- compact 后可从 accepted working memory 和 foreground continuity 恢复。
- 子 agent 输出进入 proposal/working memory，不污染长期记忆。

# 任务系统协调任务搭建编辑器重构计划书

日期：2026-05-07

## 背景

当前任务系统已经具备协调任务、拓扑模板、通信协议、运行监控四类对象，也已经将运行态协调监控图和任务系统拓扑图统一到共享前端图组件。但任务系统里的协调任务区域仍然偏向表单管理：用户需要在不同页签里切换“协调定义 / 拓扑图 / 通信协议”，节点、边、任务、协议之间的关系不够直接。

新的目标是把任务系统中的协调任务区升级为“协调任务搭建编辑器”。用户应当能够在任务系统里自由配置协调任务：从任务池选择分任务，放入拓扑节点，连接节点，配置节点角色与 Agent，配置边的通信模式，并保存为可运行的协调任务。运行时则在主会话的协调监控里实时观测同一套拓扑。

## 设计原则

1. 不改 Agent 本体组装。

   本次重构只改任务系统前端编辑器体验与拓扑配置方式，不改变 Agent 注册、Agent 运行档案、LangGraph 协调运行时。

2. 设计态和运行态共享拓扑图。

   任务系统的拓扑编辑视图和主会话协调监控必须使用同一个 `CoordinationTopologyGraph` 图组件。设计态负责编辑，运行态负责叠加运行状态。

3. 协调任务是第一等搭建对象。

   协调任务页面不再被拆成多个孤立页签，而是一个工作台：左侧任务池，中间拓扑画布，右侧检查器，下方/顶部状态与保存发布操作。

4. 所见即保存对象。

   图上的节点与边直接对应 `coordinationDraft.graph_nodes` 和 `coordinationDraft.graph_edges`；保存时同步写入协调任务、拓扑模板和通信协议。

5. 不做脚本式绕过。

   不能生成假结果，也不能为测试绕过真实功能。前端操作必须真实改变协调任务配置，并走已有 API 保存。

## 目标结构

### 1. 顶部：协调任务选择与发布

保留任务域内协调任务列表、创建协调任务、保存协调任务、带入主会话。

新增/强化展示：

- 当前协调任务标题
- 当前任务域
- 节点数量
- 边数量
- 图校验状态
- 是否启用

### 2. 左侧：任务池与节点模板

左侧提供可加入拓扑的对象：

- 当前任务域下的特定任务
- 通用 Agent 角色节点
- 协调器节点

第一阶段不做拖拽，先用明确按钮完成：

- “加入拓扑”
- “加入为审查节点”
- “加入为验收节点”
- “加入空 Agent 节点”

### 3. 中间：拓扑画布

中间使用共享 `CoordinationTopologyGraph`。

支持：

- 点击节点选中
- 点击边选中
- 当前选中节点高亮
- 当前选中边高亮
- 快速添加默认连线
- 从选中节点连接到目标节点

第一阶段不做自由拖拽布局，继续使用统一分层布局。布局稳定性比自由拖动更重要。

### 4. 右侧：属性检查器

右侧根据当前选择切换：

- 未选择：显示协调任务总体设置
- 选择节点：显示节点设置
- 选择边：显示通信边设置

节点设置包括：

- 节点名称
- 绑定分任务
- 角色
- Agent
- 删除节点

边设置包括：

- 起点
- 终点
- 通信模式
- 删除通信

总体设置包括：

- 协调任务标题
- 协调模式
- Agent 组
- 通信模式
- 停止条件
- 启用状态

### 5. 校验与保存

保存沿用现有 `saveCoordinationStack()`：

- 保存通信协议
- 保存拓扑模板
- 保存协调任务

编辑器内展示后端 `CoordinationGraphSpec.issues`，不在前端发明另一套校验规则。

## 实施步骤

### 阶段一：界面重组

1. 去掉协调任务区内部“协调定义 / 拓扑图 / 通信协议”三页签。
2. 改成固定三栏工作台：
   - 左：任务池
   - 中：拓扑画布
   - 右：检查器
3. 原“拓扑图”和“通信协议”页签里的字段迁入右侧检查器的总体设置与系统字段。

### 阶段二：编辑动作标准化

1. 新增按指定任务加入拓扑的函数。
2. 新增按角色加入空 Agent 节点的函数。
3. 新增从选中节点连接到目标节点的函数。
4. 确保节点和边变更后同步更新 `subtask_refs`。

### 阶段三：图与检查器联动

1. 点击图节点更新 `selectedGraphNodeId`。
2. 点击图边更新 `selectedGraphEdgeId`。
3. 选择节点时检查器展示节点表单。
4. 选择边时检查器展示边表单。
5. 没有选择时展示总体设置。

### 阶段四：验证

1. `npm run build`
2. `npm run test`
3. `npm run lint`

## 非目标

本次不做：

- 拖拽节点自由布局
- 后端 LangGraph 运行时重构
- Agent 注册体系重构
- 新 API
- 真实运行任务自动启动

这些可以作为第二阶段能力继续推进。

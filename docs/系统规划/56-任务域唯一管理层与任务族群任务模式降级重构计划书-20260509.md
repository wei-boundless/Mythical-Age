# 任务域唯一管理层与任务族群任务模式降级重构计划书

日期：2026-05-09

## 目标

把 `任务域（domain_id）` 提升为任务系统唯一的一等管理边界。

本轮重构要达到的状态：

1. 前端任务系统主界面只围绕 `任务域 -> 任务 -> 图草稿` 展开。
2. `task_family` 不再作为任务管理主概念暴露给用户。
3. `task_mode` 不再作为任务管理主概念暴露给用户。
4. 后端运行链路暂不强行物理删除 `task_family` / `task_mode` 字段，而是把它们降级为兼容字段，避免硬断现有装配逻辑。

## 现状问题

当前系统里三套概念重叠：

1. `domain_id`
   - 表示任务域管理对象
   - 带标题、描述、启用状态、排序

2. `task_family`
   - 实际承担了前端聚类、协调任务筛选、拓扑筛选、协议筛选、部分内存策略归类

3. `task_mode`
   - 既出现在具体任务上
   - 又参与 workflow / flow / runtime lane / agent runtime 可承接任务范围

这导致两个结果：

1. 用户看见的是重复概念：
   - 任务域
   - 任务族群
   - 任务模式

2. 代码里真正控制行为的却不止一个主键：
   - 管理层看起来是 `domain_id`
   - 实际很多筛选和推断仍然依赖 `task_family`
   - 一部分运行装配仍然依赖 `task_mode`

## 设计原则

### 1. 管理边界唯一

任务系统里，唯一主边界是 `domain_id`。

所有前端任务管理页面的组织方式、用户操作路径、选择链路，都只应围绕：

```text
任务域 -> 任务 -> 协调图 / 拓扑 / 协议 / 运行配置
```

### 2. 兼容字段降级

`task_family` 与 `task_mode` 先不物理删除。

原因：

1. 它们目前被 backend registry、assembly、memory、runtime 直接引用。
2. 强行一次性硬删，风险会扩散到：
   - flow registry
   - workflow registry
   - runtime assembly
   - projection posture tags
   - memory requested topics
   - agent runtime allowed_task_modes

本轮正确动作：

1. 从前端主界面移除其直接编辑职责。
2. 在后端用 `domain_id` 作为主入口进行映射。
3. 把 `task_family` / `task_mode` 视为派生或兼容元数据。

### 3. 运行链路逐步收口

本轮不追求彻底物理删除，而是分阶段：

1. 第一阶段：
   任务管理层去主概念化。
2. 第二阶段：
   registry 与 flow 装配改为优先 `domain_id`。
3. 第三阶段：
   再评估是否还能完全移除 `task_family` / `task_mode`。

## 重构范围

## A. 前端任务系统

### 目标

1. 任务域设置里不再把 `task_family` 作为主表单项展示。
2. 任务定义里不再把 `task_mode` 作为主表单项展示。
3. 图编辑器、任务管理器只按 `domain_id` 组织与切换。

### 动作

1. `TaskSystemView.tsx`
   - 删除或隐藏“任务族群”输入。
   - 删除或隐藏“执行类型 / 任务模式”主编辑入口。
   - 所有“当前域有哪些任务/图/协议/模板”的筛选逻辑，改为优先按 `domain_id`。

2. `CoordinationEditorWorkbench.tsx`
   - 节点、图草稿、候选任务列表按 `domain_id` 过滤。

3. 任务概览文案
   - 只显示“任务域”
   - 不再将 `task_family` / `task_mode` 作为用户心智暴露

## B. 后端任务 API 与 Registry

### 目标

1. `domain_id` 成为任务系统资源的主归属键。
2. `task_family` 允许保留，但由 `domain_id` 派生或同步，不再要求用户主编辑。
3. `task_mode` 允许保留，但从任务管理概念中降级为运行兼容字段。

### 动作

1. `backend/api/tasks.py`
   - 保留字段兼容，但下调其在请求模型中的主地位。
   - `upsert_task_domain` / `upsert_specific_task_record` / `upsert_coordination_task` / `upsert_task_graph` 优先基于 `domain_id` 写入。

2. `backend/tasks/flow_registry.py`
   - `list_task_domains()`、`delete_task_domain()`、`list_coordination_tasks()`、`list_topology_templates()`、`list_task_communication_protocols()` 的归属判断优先看 `domain_id`。
   - `task_family` 仅作为兜底兼容。

3. `backend/tasks/coordination_graph_compiler.py`
   - 协调任务与子任务一致性校验优先依据 `domain_id`。

## C. 运行装配层

### 目标

不破坏 runtime loop 与 assembly builder，但让它们不再依赖“用户显式配置 task_family / task_mode”。

### 动作

1. `backend/tasks/assembly_support.py`
   - `_resolve_task_family`
   - `_resolve_task_mode`

改造方向：

1. 先从 `domain_id` 推断兼容族群
2. 再回退到已注册值
3. 最后才回退到 template / definition

2. `backend/tasks/assembly_builder.py`
   - 运行装配内部仍可继续携带 `task_family` / `task_mode`
   - 但来源改为派生，不要求任务管理界面主编辑

## 字段策略

### `domain_id`

保留，升级为唯一主管理键。

### `task_family`

本轮不物理删除。

改为：

1. 内部兼容字段
2. 默认由 `domain_id` 派生
3. 非主界面编辑项

### `task_mode`

本轮不物理删除。

改为：

1. runtime / workflow / orchestration 兼容字段
2. 不再作为任务管理主界面字段
3. 需要时由模板、运行定义或默认策略派生

## 风险

1. `task_family` 目前深度参与 flow registry 的 memory topic、runtime lane、默认 flow 映射。
2. `task_mode` 目前深度参与 orchestration runtime profile 的 allowed_task_modes。
3. 如果一次性硬删字段，会造成：
   - 任务保存失败
   - 图资源筛选错乱
   - runtime assembly 缺默认值
   - health 私有兼容链路被误伤

因此本轮明确不做“字段物理删除”，只做“管理层降级 + 归属逻辑切换”。

## 验收标准

1. 前端任务系统主界面不再暴露“任务族群”作为主输入。
2. 前端任务系统主界面不再暴露“任务模式/执行类型”作为主输入。
3. 任务域、任务、协调图、拓扑、协议的主筛选逻辑改为 `domain_id`。
4. `npx tsc --noEmit` 通过。
5. `npm run lint` 通过。
6. 后端相关模块 `py_compile` 通过。
7. 现有主 Agent 与健康管理 Agent 不被误删，不新增旧残留资源。

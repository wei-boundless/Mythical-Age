# TaskRun 自然承接与统一运行控制重构计划书

日期：2026-05-30

## 1. 背景

当前会话层已经能把正式任务的运行进展投射到聊天界面，但还有两个结构性问题：

1. `backend/harness/loop/agent_loop.py` 在 `request_task_run` 被 admission 接受后，直接用固定文案告知用户“转入正式任务”，体验上像系统硬切换，而不是成熟 agent 对任务承接、计划确认和执行启动的自然交代。
2. 普通会话流只有前端 `AbortController` 停止能力，正式 TaskRun 由后端 background executor 接管后，没有用户可见的暂停、继续和停止控制。用户看到会话可停、任务不可停，会形成两套割裂的运行语义。

本次重构目标不是新增监控页面。健康系统继续负责汇总、诊断、任务记录管理和监控管理；聊天主界面只呈现当前会话相关的任务进度摘要和必要控制。

## 2. 现有链路证据

| 文件 | 当前职责 | 问题 |
| --- | --- | --- |
| `backend/harness/loop/agent_loop.py` | 会话回合模型决策、admission、TaskRun admission、启动 executor | `request_task_run` 分支包含硬编码用户文案，承接不自然；启动执行器后缺少控制协议说明 |
| `backend/harness/loop/task_executor.py` | 正式 TaskRun 执行循环、模型动作调用、工具执行、完成校验、恢复 | 已有 `waiting_executor` 可续跑和 recoverable error 机制，但没有用户主动 pause/resume/stop 控制面 |
| `backend/api/orchestration_harness.py` | TaskRun 查询、执行、监控 API | 只有 `execute`，没有 pause/resume/stop |
| `backend/harness/runtime/monitor_projection.py` | TaskRun 监控投影 | 可投影 running/waiting/failed/completed，但无法表达用户暂停、暂停中、停止请求 |
| `frontend/src/lib/api.ts` | 前端 API 封装 | 缺少 TaskRun 控制 API |
| `frontend/src/lib/store/runtime.ts` | 会话流、Runtime 附件、监控轮询、停止当前流 | `stopCurrentStream()` 只处理聊天 SSE，不处理正式 TaskRun |
| `frontend/src/components/chat/ChatPanel.tsx` / `ChatInput.tsx` | 聊天主界面和输入区 | 输入区只知道会话是否 streaming，不知道当前会话是否有活动 TaskRun |

## 3. 成熟 agent 对照原则

目标权责链保持单向：

```text
RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> ModelTurnDecision
-> ActionPermit / Admission
-> TaskLifecycle
-> TaskControlPlane
-> ExecutionLoop
-> MonitorProjection
-> FrontendPresentation
-> HealthSystemAggregation
```

关键约束：

1. 模型仍是“是否进入正式任务”的语义决策者；前端按钮和监控投影不能反向创建或改写任务意图。
2. Admission 只校验和授权 `request_task_run`，不能生成第二套任务意图。
3. TaskLifecycle 负责创建、终止和持久状态；TaskControlPlane 负责用户暂停、继续、停止请求。
4. ExecutionLoop 只在安全边界响应控制请求，不强杀正在进行中的模型调用或工具调用。
5. MonitorProjection 只汇总事实，不成为执行控制源。
6. HealthSystemAggregation 继续管理任务记录、健康诊断、监控归档和汇总，不迁出任务记录。

## 4. 目标设计

### 4.1 自然 TaskRun 承接

`request_task_run` 被接受后，用户可见内容由任务合同生成，不再使用固定硬编码句子。

输出应包含：

- 当前已建立的正式任务目标：来自 `contract.user_visible_goal`。
- 执行边界：来自合同的完成条件、必要产物或验证项。
- 当前状态：已启动执行器，或正在等待启动确认。
- 后续控制：用户可暂停、继续或停止任务。

这不是让代码替模型重新决定任务，而是把已经被模型提交并通过 admission 的任务合同做自然、可读的承接说明。

### 4.2 TaskRun 控制面

不新增 `TaskRunStatus = paused`，避免大面积改动 durable status 语义。控制状态放入 `TaskRun.diagnostics.runtime_control`：

```json
{
  "state": "pause_requested | paused | resume_requested | stop_requested | stopped",
  "requested_by": "user",
  "requested_at": 0,
  "reason": ""
}
```

生命周期状态保持：

- 暂停请求已记录但当前步骤未收口：`status = running`。
- 已在安全边界暂停：`status = waiting_executor`，`terminal_reason = waiting_executor`，`runtime_control.state = paused`。
- 继续运行：复用现有 `execute` 路径，不另起第二套执行器。
- 用户停止：`status = aborted`，`terminal_reason = user_aborted`，`runtime_control.state = stopped`。

### 4.3 协作式暂停

执行器在以下安全边界检查控制状态：

1. 启动执行器前。
2. 每次 step 编译前。
3. 模型动作返回后、执行工具或收尾前。
4. 工具观察记录后、进入下一轮前。

若发现 `pause_requested`，执行器记录事件和 step summary，把任务恢复为 `waiting_executor`，并保留可续跑信息。

### 4.4 停止语义

停止不强杀已经发出的模型或工具调用。停止请求写入 durable state；执行器在下一个安全边界收口为 `aborted`。如果任务尚未被 executor claim，则 API 可以直接终止为 `aborted`。

### 4.5 前端统一控制

聊天输入区根据优先级选择控制动作：

1. 当前会话 SSE 正在输出：停止当前聊天流。
2. 当前会话有 active TaskRun 且 `running`：请求暂停。
3. 当前会话 active TaskRun 为 `waiting_executor` 且 control 为 `paused` 或可续跑：继续。
4. 提供显式停止入口，但不把任务控制做成新的卡片或新页面。

界面文案强调流程进展和 agent 简要输出，不显示内部 runtime packet、工具协议等开发式表述。

## 5. 非目标

1. 不新增独立“运行界面”页面。
2. 不迁出任务记录管理；任务记录仍归健康系统管。
3. 不绕过现有 TaskLifecycle、admission、executor、completion verifier。
4. 不为了测试通过而 mock 核心执行链或降低断言。
5. 不随机切换端口；实测固定使用前端 `3000`、后端 `8003`。

## 6. 实施阶段

### 阶段 1：后端控制面

修改文件：

- `backend/harness/loop/task_executor.py`
- `backend/api/orchestration_harness.py`

内容：

1. 增加 runtime control 读写 helper。
2. 增加 `request_task_run_pause()`、`resume_paused_task_run()`、`stop_task_run()`。
3. `execute_task_run()` 在安全边界检查 pause/stop。
4. API 增加：
   - `POST /api/orchestration/harness/task-runs/{task_run_id}/pause`
   - `POST /api/orchestration/harness/task-runs/{task_run_id}/resume`
   - `POST /api/orchestration/harness/task-runs/{task_run_id}/stop`

### 阶段 2：监控投影

修改文件：

- `backend/harness/runtime/monitor_projection.py`

内容：

1. 投影 `runtime_control`。
2. `paused` 显示为可继续的等待状态，不判为 stale。
3. `pause_requested` / `stop_requested` 显示为运行中的控制请求。
4. `stopped` 归入 failed/terminal 语义，但文案区分用户停止。

### 阶段 3：自然承接

修改文件：

- `backend/harness/loop/agent_loop.py`

内容：

1. 删除固定承接句。
2. 从 TaskRun 合同生成自然承接说明。
3. launch supervision 分支也复用同一承接生成器，只改变“等待确认/已启动执行器”的状态句。

### 阶段 4：前端统一控制

修改文件：

- `frontend/src/lib/api.ts`
- `frontend/src/lib/store/types.ts`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/components/chat/ChatPanel.tsx`
- `frontend/src/components/chat/ChatInput.tsx`
- 必要时补充 `frontend/src/app/globals.css`

内容：

1. 增加 TaskRun pause/resume/stop API 封装。
2. store 增加当前 active TaskRun 控制 action。
3. ChatPanel 从 session live monitor 中推导当前任务控制状态。
4. ChatInput 显示轻量控制，不加卡片。

### 阶段 5：验证

后端：

```powershell
cd backend
python -m pytest tests/query_runtime_runtime_loop_regression.py -q
python -m pytest tests/runtime_monitor_projection_test.py -q
```

前端：

```powershell
cd frontend
npx tsc --noEmit
npx vitest run src/lib/store/runtime.test.ts src/components/chat/RuntimeRunSummary.test.ts
```

真实链路：

1. 确认 `3000`、`8003` 只有本项目进程监听。
2. 干净启动后端和前端固定端口。
3. 使用本地浏览器/Playwright 在 `http://127.0.0.1:3000` 验证：
   - 普通聊天停止仍可用。
   - 正式 TaskRun 能显示进展。
   - TaskRun 暂停后显示已暂停/可继续。
   - 继续复用原任务，不创建重复 TaskRun。
   - 停止后进入用户停止终态。

## 7. 自检

| 检查项 | 结论 |
| --- | --- |
| 是否保持模型为进入正式任务的语义决策者 | 是。只改 accepted contract 的承接说明和执行控制，不让前端或监控创建任务 |
| 是否避免旧壳堆新壳 | 是。控制面进入 executor/lifecycle 边界，不在 UI 层模拟任务状态 |
| 是否保护现有恢复链 | 是。暂停复用 `waiting_executor` 和现有 `execute` 续跑路径 |
| 是否避免破坏其它链路 | 是。Graph node child run、artifact、memory receipts、completion verifier 不改主语义 |
| 是否避免强杀造成状态损坏 | 是。只做协作式暂停/停止 |
| 是否符合健康系统边界 | 是。任务记录管理仍归健康系统；本次只提供 TaskRun 控制事实和监控投影 |
| 是否避免新增无用页面/卡片 | 是。聊天主界面做轻量控制和进展摘要，不新增运行页面 |

结论：计划符合原设计原则，可以按阶段实施。若阶段中发现需要改动 durable status 枚举、数据库结构或 GraphHarness 主协议，应暂停并补充计划，不在本轮硬改。

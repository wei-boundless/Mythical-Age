# 章节正文文量系统化修复计划

状态：实施中

## 1. 问题定义

当前 `chapter_draft` 节点被要求一次完成 10 章正文，每章目标约 2000 字，批次目标约 20000 字。实际恢复运行中，`draft_round_004.md` 已真实继承当前世界观与细纲，且第 1-10 章结构完整，但机器质量门判定：

- `content_metric_total = 10927`
- `min_required_metric_total = 12000`
- `issues = insufficient_metric:10927<12000`

这说明问题不是世界观跑偏，也不是产物伪造，而是写作任务的“单节点正文体量”与“验收粒度”不匹配。继续重试只能赌模型多写一点，不能保证稳定完成长篇章节循环。

## 2. 根因判断

1. 单次 `chapter_draft` 承担 10 章正文，输出体量过大，容易被模型自然收束、流式预算、运行时长或自检段挤占正文空间。
2. 质量门只检查批次总量和章标题，不能定位每一章的正文缺口，返修输入只能得到一个笼统的 `insufficient_metric`。
3. 返修协议要求“重写本批全部章节”，但没有把上一稿作为扩写基底，也没有明确每章应补足多少正文。
4. 启动脚本默认 `ChaptersPerRound=10`，会覆盖图内合理批次策略，导致后续新运行仍可能进入大批次模式。

## 3. 修复原则

- 不降低真实正文要求，不把不足稿标为通过。
- 不靠重复重试解决文量问题。
- 把正文预算拆到章节级，让质量门能指出“哪几章不足、缺多少”。
- 降低单节点不可控体量，把默认批次从 10 章改为 5 章，仍按章节循环累积完成 50 章。
- 返修必须基于上一稿扩写正文场景，不允许用说明、自检、摘要补字数。

## 4. 目标设计

### 4.1 批次策略

默认章节批次改为 5 章：

- 每章目标：2000 字。
- 每批目标：10000 字。
- 50 章需要 10 个章节循环批次。

这样单次写作节点的正文目标更接近模型稳定输出能力，同时保留章节循环、审核、记忆提交和下一批承接。

### 4.2 质量门

`sectioned_text_batch_quality` 增加章节级统计：

- 按章标题切分正文。
- 统计每个预期章节的正文量。
- 输出 `unit_metric_counts`、`insufficient_unit_metrics`、`unit_metric_summary`。
- 当某章低于 `minimum_metric_per_unit` 时，给出 `insufficient_unit_metric:<chapter>:<actual><<required>`。

总量不足仍然保留，但不再是唯一诊断。

### 4.3 返修协议

质量返修输入必须包含：

- 质量问题列表。
- 批次总量与目标。
- 章节级缺口摘要。
- 上一版正文引用。

写手在返修时必须扩写正文场景本身，不能用检查清单、承接说明、摘要或重复句凑字数。

## 8. 追加发现：批次边界污染

切换到 5 章批次后，运行时 checkpoint 和产物路径已经正确变为 `chapter_001_005`，但新生成的 `outline_round_001.md` 正文仍出现“当前批次：第1章至第10章”“本批允许范围：第1-10章”等旧批次语义，并继续规划第6-10章。

这说明仅修改批次数量还不够，系统必须同时修复：

1. `chapter_outline` 节点也要有机器质量门，不能只让 `chapter_draft` 拦截。
2. 细纲和正文都必须以运行时的 `batch_start_index` / `batch_end_index` 为最高优先级边界。
3. 当上游卷计划残留旧批次接口时，下游节点必须能够声明“上游卷计划仅作参考，运行时批次边界优先”，不能照抄旧范围。
4. 质量门必须识别超出本批的章标题和“当前批次/本批/允许范围”中的错误范围。

修复新增范围：

- `chapter_outline` 增加 `sectioned_text_batch_quality` 可恢复质量策略。
- `sectioned_text_batch_quality` 增加 `unexpected_unit_indexes` 与批次范围声明检查。
- `chapter_outliner` 提示去除固定“十章”表达，改成“运行时允许章号清单”。
- `chapter_writer` 提示去除静态“第6章及以后绝对不能出现”，改成“任何不在 batch_chapter_list 的章号不得出现”。

## 9. 追加发现：运行时派生字段必须规范化

质量返修停止后继续检查 checkpoint，发现 `pending_inputs` 内部存在更深的不一致：

- `batch_start_index = 1`
- `batch_end_index = 5`
- `batch_chapter_numbers = [1,2,3,4,5]`
- 但 `batch_end_index_padded = "010"`
- `batch_chapter_range = "001-010"`
- `batch_label = "第1章至第10章"`
- `runtime_loop_summary` 仍写“第1章至第10章 / 第1-10章允许范围”

这说明问题不是某一个写作节点不听话，而是运行时在回滚、返修、续跑时保留了旧派生字段。正确规则应为：

1. `chapter_index`、`batch_start_index`、`batch_end_index`、`chapters_per_round` 是循环边界事实。
2. `batch_label`、`batch_chapter_range`、`batch_chapter_list`、`runtime_loop_summary` 等都是派生描述，不能作为事实来源。
3. 当返修或回滚显式给出 `batch_start_index/batch_end_index` 时，运行时必须保留这个批次边界，并重算所有派生描述。
4. 只有章节循环成功进入下一批时，路由器才允许主动推进 `chapter_index` 并重算新批次。
5. 返修要求模板必须在规范化之后渲染，不能把旧派生字段写进给 agent 的返修指令。

实施补充：

- 在 `langgraph_coordination_runtime.py` 增加运行时循环输入规范化入口。
- `rewind_from_stage`、`_stage_prepare`、`_stage_execute`、质量返修、审核返修都走同一个规范化入口。
- `_apply_loop_derived_fields` 支持 `preserve_existing_batch_scope`，用于回滚/返修/续跑保留当前显式批次边界，同时重算派生字段。
- 路由器推进下一批时仍使用非保留模式，确保真正进入下一批。
- 增加回归测试，固定“1-5 显式边界 + 旧 1-10 派生字段”必须被规范化为 1-5。

## 10. 追加修复：正式章节与跨批次承接接口分层识别

新的 1-5 章细纲已经只生成当前批次正式章节，但末尾“后续批次承接点”仍合理提到第6-10章。旧质量门把全文里所有“第X章”都当作正式章节标题，导致承接接口被误判为越界章节。

质量门需要区分两类信息：

1. 正式章节结构：Markdown 标题或行首章节标题，例如 `### 第1章「泽中」`。这类标题决定当前批次是否缺章、是否出现越界正式章节。
2. 跨批次承接接口：位于“后续批次承接点 / 后续批次接口 / 下一批”区块里的未来章号。它们用于维持长篇连续性，不等于当前批次正式章节，不应触发 `unexpected_unit_indexes`。

实施规则：

- `sectioned_text_batch_quality` 增加 `heading_match_scope=formal_heading`，只把行首正式章节标题作为章节边界。
- 增加 `ignored_heading_parent_keywords`，让“后续批次承接点”等区块内的未来章号不参与当前批次标题统计。
- 保留 `forbid_unexpected_unit_ranges`，当前批次声明如果仍写成“第1章至第10章”必须拒绝。
- 范围声明检查必须识别“下一批 / 后续批次 / 承接点”等未来引用：例如“为下一批（第6-10章）铺垫”不是当前批次声明，不应触发越界；但“当前批次：第1章至第10章”仍必须触发越界。
- 保留正式越界拒绝：如果正文或细纲真的出现 `### 第6章` 作为正式章节标题，仍必须拒绝。
- 补充回归测试：承接点包含第6-10章不拒绝；正式第6章标题仍拒绝。

## 11. 系统审计结论：质量门必须按语义分区工作

继续检查后确认，当前问题不是“第6-10章这个范围识别错了”这一个点，而是质量门同时承担了五类不同语义，却没有明确分层：

1. 当前批次正式结构：正式章节标题决定本批是否缺章、是否出现越界正式章节。
2. 当前批次声明：`当前批次`、`本批允许范围`、`批次摘要` 等声明必须严格等于运行时 `batch_start_index` 到 `batch_end_index`。
3. 未来批次承接接口：`下一批`、`后续批次承接点` 中的未来章号是交接信息，不等于当前批次正式章节，也不等于旧批次污染。
4. 正文主体区：正文文量只能统计真正的章节正文，不能把说明、摘要、自检、目标完成表算入正文。
5. 后置说明区：承接说明、目标完成说明、人物推进、自检风险、公开摘要只能作为审核辅助，不允许为正文文量背书。

因此，修复必须把质量门改成“分区后的审核系统”，而不是继续扩充一个全局正则。

## 12. 一次性修复边界

本轮不再接受单点补丁，按以下边界集中实施：

1. `sectioned_text_batch_quality` 增加正文统计输入区：
   - 对 `chapter_draft`，优先只统计【章节正文候选】区。
   - 如果没有该区，则从全文开头或第一个正式章节标题开始，但遇到【承接说明】【本章目标完成说明】【人物与冲突推进】【商业钩子与爽点兑现】【后续伏笔或待承接事项】【自检风险】【公开摘要】等后置说明区必须停止。
   - 总量统计和单章统计都使用同一个正文主体文本，防止说明文字补足正文缺口。
2. 章节标题识别只负责“正式章节结构”：
   - `heading_match_scope=formal_heading` 时只识别行首 Markdown 标题或正式章标题。
   - 后续承接区内的第6-10章不参与正式章节统计。
3. 范围声明识别只负责“当前批次声明”：
   - 当前批次、本批允许范围、批次摘要等写成旧范围必须拒绝。
   - 下一批/后续批次/承接点附近的未来范围不拒绝。
4. 产物拆分与质量门对齐：
   - 产物拆分器必须能识别 `# 【章节正文候选】` / `【章节正文候选】` 这类中文段落标题。
   - 正文文件只保存正文候选区，manifest 保存说明区，不能把“缺少对应内容”的调试包装写进正式正文。
5. 进度统计只使用质量门通过后的正文主体统计：
   - 运行诊断里的 `content_metric_total` 要优先取质量门的正文主体统计值。
   - 防止项目进度 ledger 把说明区字数记为正文进度。

## 13. 必补回归测试

本轮修复必须补齐以下测试后才能恢复写作：

1. 当前批次是 1-5 时，`当前批次：第1章至第10章` 必须拒绝。
2. 当前批次是 1-5 时，`为下一批（第6-10章）铺垫` 不应拒绝。
3. 后续批次承接点列出第6-10章，不应被当成正式章节越界。
4. 正式标题 `### 第6章` 必须拒绝。
5. 第5章正文不足，但后面有大量【承接说明】或【公开摘要】，仍必须报 `insufficient_unit_metric:5`。
6. 产物拆分器必须能把 `# 【章节正文候选】` 下的正文写入 draft，把后置说明写入 manifest。
7. 质量门返回的正文主体统计必须小于或等于全文统计，且进度诊断优先使用正文主体统计。

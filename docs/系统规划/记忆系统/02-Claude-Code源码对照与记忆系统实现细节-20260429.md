# Claude Code 源码对照与记忆系统实现细节

日期：2026-04-29  
定位：本文件是 `01-记忆系统与上下文管理架构设计-20260429.md` 的源码对照报告。它不替代主设计，而是用 Claude Code 源码中的成熟实现细节校验本项目 MemorySystem / ContextPolicy / OrchestrationSystem 的边界。

---

## 0. 总结论

Claude Code 的实现证明：成熟 agent runtime 的“记忆”不是单一存储模块，而是一组相互协作的连续性机制：

```text
后台会话笔记
长期记忆目录
相关记忆预取
文件状态缓存
上下文压缩与恢复
forked agent 隔离
写回工具权限门
```

这些机制不能被我们直接照搬成一个“大 memory 层”。在洪荒时代 AgentRuntime 中，正确落点应是：

```text
ConversationMemory:
  承接对话摘要、recent dialogue、compact summary、hot truth window。

StateMemory:
  承接 current state、active bindings、context slots、flow state、task state、file/read handles、compact restore indexes。

LongTermMemory:
  承接跨会话稳定事实、用户偏好、项目约定、反馈纠正、外部参考指针。

ContextPolicy:
  消费 StateMemory / ConversationMemory / LongTermMemory 的候选，按预算裁剪成 ContextPackage。

OrchestrationSystem:
  采纳或拒绝候选，建立本轮执行事实。

MemoryGate / CommitGate:
  决定哪些候选可以写回；当前阶段不开放自动 durable memory 写入。
```

最关键校正：

```text
Session memory 不是我们的最终抽象。
它在 Claude Code 中同时混合了对话摘要和当前状态。
在本项目中必须拆成 ConversationMemory + StateMemory。
```

## 2. Claude Code 可借鉴的核心机制

### 2.1 后台增量会话记忆

Claude Code 的 `shouldExtractMemory()` 不是每轮都更新 session memory：

```text
必须满足 token 增长阈值。
工具调用数量只是附加条件。
如果最后一轮没有工具调用且 token 达标，可在自然断点抽取。
```

源码要点：

```text
sessionMemory.ts:134 shouldExtractMemory()
sessionMemory.ts:155 tool-call threshold
sessionMemory.ts:165 token threshold is ALWAYS required
sessionMemory.ts:318 runForkedAgent(...)
sessionMemory.ts:374 registerPostSamplingHook(...)
```

对我们的启发：

```text
ConversationMemory 不应同步阻塞主链。
StateMemory 可以在关键事件后更新，但也应具备节流门。
记忆更新必须有触发条件，不应“每轮全量重写”。
```

落地约束：

```text
MemoryUpdateTrigger:
  reason = turn_boundary | tool_batch_boundary | compact_boundary | task_state_transition
  token_delta
  tool_call_delta
  state_delta
  should_update_conversation_memory
  should_update_state_memory
  should_propose_long_term_write
```

### 2.2 固定模板，但语义要拆层

Claude Code 的 session memory 模板包含：

```text
Session Title
Current State
Task specification
Files and Functions
Workflow
Errors & Corrections
Codebase and System Documentation
Learnings
Key results
Worklog
```

其中 `prompts.ts:69` 明确要求总是更新 `Current State`，因为它对 compact 后连续性关键。

但这个模板在 Claude Code 中是一个文件级 session notes。对我们来说，不能整体照搬为 `ConversationMemory`。正确拆分：

| Claude Code section | 我们的新归属 |
| --- | --- |
| Current State | StateMemory |
| Workflow | StateMemory / ProjectConvention 候选，需分流 |
| Files and Functions | StateMemory 的 active file awareness；长期保存前必须验证是否非显然 |
| Task specification | ConversationMemory + TaskSystem contract trace |
| Errors & Corrections | ConversationMemory；稳定纠正才可成为 LongTermMemory feedback |
| Codebase and System Documentation | 默认不进长期记忆，优先从当前文件读取 |
| Learnings | 需要 MemoryGate 判断是否稳定、非显然 |
| Key results | ConversationMemory / artifact refs |
| Worklog | ConversationMemory / StateMemory progress trace |

落地约束：

```text
禁止把 session summary 整体当成长期记忆。
禁止把 Current State 混入 LongTermMemory。
禁止把 code paths / architecture / git history 作为 durable memory 默认保存。
```

### 2.3 Session Memory Compact 优先，但必须保护协议不变量

Claude Code 在 compact 时优先尝试 `trySessionMemoryCompaction()`：

```text
sessionMemoryCompact.ts:57 minTokens = 10000
sessionMemoryCompact.ts:59 minTextBlockMessages = 5
sessionMemoryCompact.ts:60 maxTokens = 40000
sessionMemoryCompact.ts:527 waitForSessionMemoryExtraction()
sessionMemoryCompact.ts:540 session memory empty -> fallback
```

最重要的是 `adjustIndexToPreserveAPIInvariants()`：

```text
不能切断 tool_use / tool_result 配对。
不能切断同一 assistant message id 下的 thinking block 与正文。
不能让 compact 后消息序列违反模型 API 协议。
```

对我们的启发：

```text
ContextCompactor 不能只是按 message 数或 token 数截断。
必须先有 protocol-aware slicing。
```

落地约束：

```text
ContextSliceInvariant:
  preserve_tool_call_pair = true
  preserve_thinking_message_group = true
  preserve_current_turn_tail = true
  preserve_operation_directive_context = true
  preserve_required_handles = true
```

这条约束应进入未来 `backend/context_management/context_compactor.py` 或新 `backend/context_policy/compaction.py`。

### 2.4 上下文压力是梯度，不是一个开关

Claude Code 的 `autoCompact.ts` 中有几个关键数值：

```text
MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20000
AUTOCOMPACT_BUFFER_TOKENS = 13000
MANUAL_COMPACT_BUFFER_TOKENS = 3000
MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
```

对应原则：

```text
先预留输出空间。
接近阈值时先预警。
达到阈值才自动 compact。
连续失败后熔断。
compact / session_memory 自身不能递归触发 compact。
```

本项目已有 `ContextCompactor` 的 pressure level：

```text
normal
warning
microcompact
full_compact
```

需要补齐：

```text
reserved_output_tokens 不应只是小默认值，应按模型窗口与任务模式配置。
ContextPolicy 应产生 pressure decision record。
CompactFailureCircuitBreaker 应进入运行态。
query_source / operation_source 需要递归保护。
```

### 2.5 长期记忆必须反默认保存

Claude Code 的长期记忆类型只有：

```text
user
feedback
project
reference
```

同时 `WHAT_NOT_TO_SAVE_SECTION` 明确排除：

```text
代码模式、架构、文件路径、项目结构。
git 历史、最近改动、谁改了什么。
调试方案或 fix recipe。
CLAUDE.md 已有内容。
临时任务细节、当前对话上下文。
```

即使用户要求保存，也要追问“有什么出人意料或非显然的部分值得保留”。

对我们的启发：

```text
LongTermMemory 默认不是更多保存，而是更少保存。
长期记忆是稳定、非显然、跨会话有用的事实。
状态记忆和对话记忆不能偷渡到长期记忆。
```

落地约束：

```text
LongTermMemoryWriteCandidate:
  memory_type = user_preference | feedback_correction | project_convention | external_reference
  stability = stable | provisional | stale_risk
  derivable_from_current_files = bool
  ephemeral_task_detail = bool
  requires_verification_before_use = bool
  save_gate_reason
```

当前阶段：

```text
只生成候选，不自动写 durable store。
除非用户显式触发并通过 MemoryGate，否则不落盘。
```

### 2.6 相关长期记忆召回应非阻塞、限额、去重

Claude Code 的相关记忆召回流程：

```text
memoryScan.ts:
  最多扫描 200 个 markdown 文件。
  只读前 30 行 frontmatter。
  按 mtime 新到旧排序。

findRelevantMemories.ts:
  用 manifest 给 side query 选择。
  最多选择 5 条。
  alreadySurfaced 在选择前过滤。

attachments.ts / query.ts:
  每个用户 turn 启动一次 prefetch。
  只在 settled 时消费。
  未完成则跳过，本轮不阻塞。
```

对我们的启发：

```text
LongTermMemory recall 不应阻塞主编排链。
召回结果只能成为 MemoryContextCandidate。
ContextPolicy 根据预算决定是否注入。
```

落地约束：

```text
LongTermRecallPrefetch:
  query_text
  manifest_headers
  selected_limit = 5
  already_surfaced_refs
  recent_tools_or_operations
  settled_at
  consumed_at_turn
  abort_controller
```

### 2.7 FileStateCache 本质上属于 StateMemory

Claude Code 的 `FileStateCache` 不是普通缓存。它保存：

```text
content
timestamp
offset
limit
isPartialView
```

`isPartialView` 的意义很关键：

```text
如果模型只看过截断或自动注入的部分内容，不能基于这份部分视图直接编辑。
必须先显式 Read 完整文件。
```

compact 后，`createPostCompactFileAttachments()` 会根据 compact 前的 read file state 选择最近文件恢复，并受限制：

```text
最多恢复若干文件。
每个文件有 token 限制。
整体恢复有 token 预算。
已在 preserved tail 中存在的 Read 结果不重复注入。
```

对我们的启发：

```text
StateMemory 必须包含 FileAwareness / HandleAwareness。
它不只是“我记得哪个文件”，还要记录“模型看到的是完整视图还是部分视图，可否安全编辑”。
```

落地约束：

```text
StateMemoryFileRef:
  path
  observed_at
  view_kind = full | partial | generated_summary
  offset
  limit
  content_hash
  editable_without_reread = bool
  restore_priority
  owner_task_id
```

### 2.8 后台长期记忆抽取必须有权限门

Claude Code 的 `extractMemories.ts` 在 stop hook 后通过 forked agent 抽取长期记忆，但工具权限很窄：

```text
Read / Grep / Glob 允许。
Bash 只允许 read-only。
Edit / Write 只允许 memory dir 内。
maxTurns = 5。
skipTranscript = true，避免主链竞态。
如果主 agent 已经写过 memory，则 background extraction 跳过。
```

对我们的启发：

```text
MemoryWriteCandidate 与 MemoryCommitRecord 必须分开。
后台抽取可以提出候选，但不能绕过 MemoryGate 写入。
```

当前阶段的硬约束：

```text
不实现自动 durable write。
可以实现 MemoryWriteCandidate preview。
可以记录 rejected / pending / accepted 状态。
```

### 2.9 forked agent 默认隔离，显式共享

Claude Code 的 `createSubagentContext()` 默认：

```text
克隆 readFileState。
新建 child abort controller。
setAppState 默认 no-op。
任务注册/kill 通过 root store 的专门通道。
本地 denial tracking。
新的 agentId / queryTracking。
```

对我们的启发：

```text
未来多智能体不能共享一份可变 StateMemory。
主 agent、子 agent、后台 memory agent 的状态默认隔离。
需要共享时必须通过显式候选、显式 commit、显式作用域。
```

落地约束：

```text
AgentMemoryScope:
  main
  subagent_local
  shared_project
  durable_user

默认写入 subagent_local。
只有 OrchestrationSystem / TaskSystem 接受后，才可提升为 shared_project 或 main state。
```

这只作为未来接口预留；当前阶段仍先完成单主 agent 链。

## 3. 不能照搬 Claude Code 的部分

### 3.1 不能把 session memory 当最终架构

Claude Code 的 session memory 文件是一种工程上高效的合并视图，但它把 conversation summary、current state、workflow、files、errors 混在同一份 markdown 中。

本项目不应这样做，因为我们的系统已经明确拆分：

```text
TaskSystem:
  当前任务合同与任务状态入口。

OperationSystem:
  能力、权限、执行合同。

OrchestrationSystem:
  本轮事实采纳与执行流。

MemorySystem:
  对话记忆、状态记忆、长期记忆。

ContextPolicy:
  上下文预算与呈现裁剪。
```

所以 session markdown 只能是迁移期的投影视图，不能继续做 canonical owner。

### 3.2 不能让 prompt 文本治理代替结构治理

Claude Code 里很多记忆规则以 prompt instructions 存在。这对成熟 CLI 产品有效，但本项目当前重构原则是：

```text
结构优先于提示词。
候选优先于隐式注入。
gate 优先于模型自觉。
manifest 优先于裸文本拼接。
```

因此我们要把“不该保存”“需要验证”“不能覆盖当前状态”等规则落到数据字段和 gate 上，而不是只写进系统提示词。

### 3.3 不能开放后台 agent 直接写 durable memory

Claude Code 允许后台抽取 agent 在 memory dir 内 Edit/Write。我们当前阶段不应照搬。

原因：

```text
我们的任务系统、操作系统、编排系统刚完成合同化。
记忆系统还没完成 MemoryGate / CommitRecord。
过早写 durable memory 会把旧 query 里的隐式状态污染成长期事实。
```

因此当前落点：

```text
后台抽取只产出 MemoryWriteCandidate。
CommitGate 默认 reject 或 pending。
用户显式保存时才进入受控 commit。
```

### 3.4 不能让长期记忆覆盖当前文件事实

Claude Code 明确要求：

```text
memory claims can be stale。
文件/函数/flag 类型记忆使用前要读当前文件或 grep 验证。
冲突时相信当前观察。
```

本项目应更严格：

```text
LongTermMemory 不能覆盖 StateMemory。
LongTermMemory 不能覆盖当前轮检索证据。
LongTermMemory 不能覆盖 OperationResult。
LongTermMemory 只能作为弱到中等强度的上下文候选。
```

## 4. 对本项目架构的修正要求

### 4.1 MemorySystem 必须拆成三层模块

建议模块：

```text
backend/memory_system/
  contracts.py
  conversation_memory.py
  state_memory.py
  long_term_memory.py
  policy.py
  gate.py
  runtime_view.py
  store_adapters.py
```

其中：

```text
conversation_memory.py:
  读取 recent dialogue、compact summary、session summary view。
  产生 ConversationContextCandidate。
  产生 ConversationSummaryWriteCandidate。

state_memory.py:
  读取 process_state、flow_snapshots、context_slots、runtime handles、file awareness。
  产生 StateMemoryRestoreCandidate / StateContextCandidate。
  提供 ContextPolicy 的核心输入。

long_term_memory.py:
  读取 durable notes / memory manifest。
  产生 LongTermMemoryContextCandidate。
  产生 LongTermMemoryWriteCandidate，但不直接写入。

policy.py:
  负责 recall selection、候选分层、staleness、budget hint。

gate.py:
  负责 save eligibility、write mode、commit/reject/pending。

runtime_view.py:
  给 OrchestrationSystem / PromptManifest 提供只读视图。
```

### 4.2 ContextPolicy 必须成为正式系统，而不是旧 controller

当前 `backend/context_management/context_controller.py` 已经有 ContextPackage、压力裁剪、durable 降级等雏形，但职责仍偏旧：

```text
直接从 SessionMemoryManager preview_views 获取 sections。
直接混合 durable matches。
直接产生 model_visible_sections。
```

目标改造：

```text
ContextPolicy 输入：
  ConversationContextCandidate[]
  StateContextCandidate[]
  LongTermMemoryContextCandidate[]
  RetrievalEvidenceCandidate[]
  OperationResultRefs[]
  TaskContract
  token_budget

ContextPolicy 输出：
  ContextPackage
  ContextDecisionRecord
  PromptContextSection[]
  dropped_candidate_refs[]
```

硬规则：

```text
StateContextCandidate 优先级高于 LongTermMemoryContextCandidate。
RetrievalEvidenceCandidate 高于长期记忆。
当前 TaskContract 高于所有 restore candidate。
restore candidate 不直接决定事实。
```

### 4.3 PromptBuilder 必须降级为渲染器

当前 `backend/query/prompt_builder.py` 仍直接渲染：

```text
session context
durable context
static prompt
turn prompt
```

后续它不应拥有“带什么上下文”的决定权。正确边界：

```text
ContextPolicy 决定带什么。
PromptManifest 记录来源和裁剪。
SoulSystem 决定表达风格与长期人格约束。
PromptBuilder 只负责把已采纳 PromptContextSection 渲染成文本。
```

### 4.4 RuntimeContextState 必须拆出 StateMemory

当前 `backend/query/runtime_context_state.py` 的 `load_session_restore_candidates()` 会直接从 session slots 产生：

```text
active_pdf
active_dataset
active_object_handle_id
active_result_handle_id
active_subset_handle_id
```

这些是有价值的状态恢复材料，但新架构中只能成为：

```text
StateMemoryRestoreCandidate
```

必须带上：

```text
source = process_state | flow_snapshot | runtime_handle | user_current_turn
confidence
owner_task_id
observed_at
staleness
can_promote_to_current_fact = false by default
```

由 OrchestrationSystem 结合 TaskContract 决定是否采纳。

### 4.5 ContextCompactor 必须补齐协议不变量

当前 `backend/context_management/context_compactor.py` 有 pressure level 和 micro/full compact，但需要补齐：

```text
tool call / tool result 配对保护。
同轮工具批次保护。
operation directive context 保护。
required handles 保护。
compact 失败熔断。
compact 递归保护。
compact 后 state restore budget。
```

否则真实主链接入后，压缩可能破坏执行协议。

## 5. 推荐目标合同

### 5.1 MemoryContextCandidate

```text
MemoryContextCandidate:
  candidate_id
  memory_layer = conversation | state | long_term
```

## 7. 可靠性校验清单

重构实现时，每个阶段都要检查：

```text
restore != decide:
  任何恢复信息都不能直接成为当前任务事实。

state != long_term:
  当前工作现场不能写入长期记忆。

conversation != state:
  对话摘要不能替代 active bindings / handles / flow state。

long_term != current evidence:
  长期记忆不能覆盖当前文件、当前检索证据、当前操作结果。

candidate != commit:
  写回候选不能绕过 gate。

compact != truncate:
  压缩不能破坏 tool / operation 协议。

prefetch != block:
  长期记忆相关召回不能阻塞主链。

subagent != parent:
  子 agent / 后台 agent 状态默认隔离。
```

## 8. 当前设计结论

根据 Claude Code 源码对照，当前 `01-记忆系统与上下文管理架构设计-20260429.md` 的方向是正确的，但需要更严格执行以下四点：

```text
第一，StateMemory 要作为 ContextPolicy 的核心输入先做，不能等长期记忆。

第二，ConversationMemory 只负责对话连续性与 compact summary，不能继续包办 Current State。

第三，LongTermMemory 必须反默认保存，所有写入先候选化。

第四，ContextCompactor 必须协议感知，不能只做 token-aware。
```

因此后续重构顺序应是：

```text
contracts
-> StateMemory
-> ConversationMemory adapter
-> LongTermMemory candidates
-> ContextPolicy
-> Compaction invariants
-> PromptManifest / query cleanup
```

这比“先重构整个 query 块”更稳，因为 query 当前承载了太多旧职责。先把 MemorySystem 与 ContextPolicy 的合同立住，再清理 query，旧链路才不会继续把隐式状态写回到新系统里。

# 记忆系统与上下文管理架构设计

日期：2026-04-29  
定位：本文件用于指导 MemorySystem 与 ContextPolicy 的正式重构。目标是在不放开真实副作用的前提下，把旧 `memory / context_management / query.prompt_builder / query.runtime_persistence` 中混合的记忆、恢复、压缩、Prompt 拼装和写回职责拆成可被编排系统统一采纳、验证、阻断和追踪的结构。

---

## 0. 总结论

记忆系统和上下文管理必须和编排系统高度协调，但不能被编排系统吞掉。

新的边界是：

```text
MemorySystem:
  负责对话记忆、状态记忆、长期记忆三层记忆的读、恢复候选、写回候选、治理和存储适配。

ContextPolicy:
  负责把状态记忆、对话窗口、检索证据和长期记忆召回结果裁剪成 ContextPackage / PromptContextSection。

SoulSystem / PromptManifest:
  负责把已采纳、已裁剪的上下文呈现给模型。

OrchestrationSystem:
  负责采纳或拒绝 Memory / Context 候选，建立当前轮执行真相。

CommitGate:
  负责 session memory / durable memory / artifact / final answer 是否能写回。
```

一句话：

```text
MemorySystem 提供“可记得什么”和“可写回什么”。
ContextPolicy 决定“本轮带什么上下文”。
OrchestrationSystem 决定“哪些候选进入当前轮事实”。
CommitGate 决定“哪些结果可以落盘”。
```

当前阶段先做合同和 preview 化，不开放自动 durable memory 写回，不删除旧 API，不改变现有存储格式。

源码对照基准：

```text

```

后续实现必须同时满足本文件的系统边界，以及源码对照文件中提炼出的工程约束：后台记忆更新节流、状态记忆显式化、长期记忆反默认保存、非阻塞召回、compact 协议不变量、子 agent 默认隔离。

### 0.1 分层校正

本项目的记忆系统不能只按存储位置分为 `session / durable`，应按语义职责分成三层：

```text
对话记忆 Conversation Memory:
  保存当前会话的对话摘要、关键用户请求、近期问答、工具结果摘要和可压缩历史。
  主要服务连续对话、回答一致性、compact 后的对话恢复。

状态记忆 State Memory:
  保存当前工作状态、任务进度、active bindings、context slots、flow state、task state、active handles。
  主要服务上下文管理、恢复候选、继续执行、任务切换后的工作状态重建。

长期记忆 Long-term Memory:
  保存跨会话稳定事实、用户偏好、项目约定、反馈纠正、外部参考指针。
  主要服务按需召回、长期个性化和项目连续性。
```

三层与上下文管理的关系不同：

```text
对话记忆:
  与上下文管理中等关联，主要提供 hot truth window / compact summary / recent dialogue。

状态记忆:
  与上下文管理高度关联，是 ContextPolicy 的核心输入。
  ContextPolicy 的 restore_indexes、required_handles、active slots、compaction restore 都应优先消费状态记忆。

长期记忆:
  与上下文管理弱到中等关联，只在 MemoryPolicy / recall 认为相关时进入候选。
  长期记忆不能覆盖当前状态记忆和当前轮检索证据。
```

因此后续合同不应只有笼统的 `MemoryContextCandidate`，还必须能标识：

```text
memory_layer = conversation | state | long_term
```

这是本文件后续所有设计的修正基准。

### 1.3 按三层记忆重新映射旧代码

旧代码按目录看是 `memory / structured_memory / context_management / query`，但按新原则应重新映射为：

| 新语义层 | 当前主要代码 | 当前存储/产物 | 新系统定位 |
| --- | --- | --- | --- |
| 对话记忆 | `structured_memory/session_memory.py`, `memory/session.py`, `context_management/context_compactor.py` | `summary.md`, `agent_view.md`, `compaction_view.md`, compacted messages | ConversationMemoryStore / ConversationSummaryCandidate |
| 状态记忆 | `structured_memory/process_state.py`, `RuntimeContextState`, `flow_snapshots.py`, `DialogueState.context_slots` | `process_state.json`, `flow_snapshots.json`, active bindings / task state | StateMemoryStore / MemoryRestoreCandidate / StateContextCandidate |
| 长期记忆 | `memory/durable.py`, `durable_memory/notes`, `MemoryReadAgent`, `api/memory.py` | durable notes, MEMORY index, governance log | LongTermMemoryStore / LongTermMemoryContextCandidate / DurableMemoryWriteCandidate |

其中最关键的是状态记忆：

```text
状态记忆不是“长期记住的事实”。
状态记忆是当前工作现场的结构化恢复材料。
它与上下文管理高度关联，因为 compact、任务继续、active handle 恢复、上下文预算分配都依赖它。
```

因此 `ContextPolicy` 的主要输入应来自状态记忆，而不是长期记忆：

```text
StateMemorySnapshot
  -> StateContextCandidate
  -> ContextPackagePreview
  -> PromptContextSection
```

长期记忆只通过按需召回进入上下文，不能常驻或全量注入。

## 2. 当前失衡点

### 2.1 Restore 与 Decide 混合

当前 `RuntimeContextState.load_session_restore_candidates()` 会把 session slots 转成：

```text
active_pdf
active_dataset
active_object_handle_id
active_result_handle_id
active_subset_handle_id
```

这些是有价值的恢复线索，但在新架构中只能成为：

```text
StateMemoryRestoreCandidate
```

不能直接变成当前任务事实。当前任务事实仍必须来自 `TaskContract`，当前轮采用与否由 `OrchestrationSystem` 决定。

这里要特别区分：

```text
对话记忆恢复:
  恢复“刚才说过什么、刚才结论是什么”。

状态记忆恢复:
  恢复“当前做到哪里、绑定了哪个文件/数据/结果 handle、下一步是什么”。

长期记忆召回:
  召回“跨会话稳定事实、偏好、项目约定”。
```

只有状态记忆恢复可以进入 `ContextPolicy.restore_indexes`，但仍然必须是候选。

### 2.2 Memory Intent 与 Route 混合

`understanding.memory_intent.MemoryIntent` 里仍有：

```text
should_skip_rag
memory_read_mode
memory_write_mode
```

它们目前会影响旧 query route。新架构里它们必须降级为：

```text
MemoryIntentCandidate.payload
```

`should_skip_rag` 不能直接跳过检索，最多作为 `ContextPolicy` 和 `OrchestrationSystem` 的候选信号。

### 2.3 ContextPackage 与 Prompt 呈现混合

当前 `ContextPackage` 已经包含：

```text
model_visible_sections
debug_sections
selected_sections
dropped_sections
token_accounting
```

这个结构值得保留，但它不应该由 `memory.context` 直接渲染成系统 prompt。成熟架构中应拆为：

```text
MemoryContextCandidate
  -> CandidateSet
  -> adopted MemoryContext
  -> ContextPolicy builds ContextPackage
  -> SoulSystem renders PromptManifest sections
```

### 2.4 写回链绕过 CommitGate

当前 post-turn 里会直接：

```text
refresh_session_memory()
schedule_durable_memory_extraction()
commit_durable_memory_extraction_from_context_state()
```

成熟架构中这三类动作必须先变成：

```text
SessionMemoryWriteCandidate
DurableMemoryWriteCandidate
MemoryExtractionCandidate
```

然后进入：

```text
MemoryGateDecision
  -> CommitCandidate
  -> CommitGate
  -> MemoryCommitRecord
```

### 2.5 Query 层仍是事实中心

当前 `query.runtime` 仍在实际执行链中创建 prompt、选择 tools、创建 conversation agent、处理 memory write ack。记忆系统重构不能继续强化 query，必须把 query 降级为 adapter。

### 2.6 Session Memory 概念过宽

当前旧代码和旧文档中常用 `session memory` 表达多个不同概念：

```text
对话摘要。
compact 恢复摘要。
active process context。
flow state。
task state。
context slots。
active bindings。
warm snapshots。
```

新架构中不能继续把这些都叫做一层。必须拆成：

```text
ConversationMemory:
  对话摘要与近期对话窗口。

StateMemory:
  当前工作现场、状态槽、绑定、流程和任务进度。
```

其中 `StateMemory` 是上下文管理的核心伙伴；`ConversationMemory` 是上下文管理的历史材料；`LongTermMemory` 是按需召回材料。

## 3. 设计原则对照

### 3.1 来自总框架的硬约束

```text
任务系统：要做什么。
理解层：候选信号。
操作系统：能碰什么。
灵魂系统：以什么结构呈现给模型。
记忆系统：带什么上下文，产生什么写回候选。
编排系统：怎么做，是否能做，按什么顺序做。
执行层：只按 RuntimeDirective 执行。
写回层：决定什么能落入 session / memory / artifact / final answer。
```

记忆系统必须遵守：

```text
restore != decide
memory read != task truth
memory write != commit
context package != prompt manifest
prompt visible != runtime executable
```

### 3.2 来自状态管理原则的硬约束

参考 `docs/设计原则/03-状态管理.md`，可以迁移的原则是：

```text
进程级状态、应用状态、单轮运行上下文分层。
默认隔离，显式共享。
状态更新要有单一入口。
跨系统副作用集中处理，不能散落在多个调用点。
```

映射到本项目：

```text
SessionMemoryStore:
  会话级工作记忆。

DurableMemoryStore:
  跨会话长期记忆。

TurnContextPackage:
  当前轮模型可见上下文。

MemoryCommitRecord:
  写回事实。

AgentMemoryScope:
  后续多智能体隔离边界。
```

### 3.3 来自 System Prompt 工程的硬约束

参考 `docs/设计原则/04-System-Prompt-工程.md`，上下文管理必须支持：

```text
静态段与动态段分离。
Prompt section 有来源、层级、生命周期。
动态上下文不应破坏稳定 prompt 前缀。
昂贵或易失上下文必须显式标记。
PromptManifest 记录每段内容来源，而不是直接拼大字符串。
```

映射到本项目：

```text
Soul static context:
  stable, cache-friendly

Session context:
  session scoped, memoizable

Turn memory context:
  turn scoped, candidate/adopted trace required

Retrieval evidence:
  current turn evidence, should outrank stale durable memory

Durable memory:
  stable facts, source refs required, cannot override current evidence
```

### 3.4 来自编排系统的硬约束

```text

```

并保持：


## 4. 推荐设计方向

### 4.1 不是把 context_management 并入 memory

`context_management` 现在已有预算、压力、压缩和 section 选择能力。它应该升级为 `ContextPolicy` 的基础，而不是简单塞进 MemorySystem。

建议边界：

```text
MemorySystem:
  提供 memory/source facts。

ContextPolicy:
  选择、裁剪、排序、预算分配。

SoulSystem:
  渲染模型可见 PromptManifest。
```

### 4.2 MemoryFacade 保留为 Legacy Adapter

短期不删除 `MemoryFacade`。它应变成迁移期兼容层：

```text
MemoryFacade
  -> MemorySystem facade adapter
  -> ContextPolicy adapter
  -> legacy query compatibility methods
```

成熟后，query 只调用：

```text
memory_system.build_turn_memory_candidates()
context_policy.build_context_package_preview()
```

而不再调用 `build_persistent_memory_block()`。

### 4.3 先合同化，再迁移调用点

第一步不重写存储，不改 durable memory 文件格式，先新增正式合同：

```text
backend/memory_system/contracts.py
backend/context_policy/contracts.py
```

然后逐步把旧返回值包成候选，接入编排系统 CandidateSet。

## 5. 目标系统边界

### 5.0 MemorySystem 三层内部结构

成熟 MemorySystem 内部不是一个平铺的 `MemoryContextBuilder`，而是三层记忆协作：

```text
MemorySystem
  ConversationMemory
    ConversationSummaryStore
    DialogueWindowBuilder
    ConversationCompactionCandidateBuilder

  StateMemory
    ProcessStateStore
    ContextSlotStore
    FlowSnapshotStore
    StateRestoreCandidateBuilder
    StateContextCandidateBuilder

  LongTermMemory
    DurableMemoryStore
    MemoryRecallIndex
    RelevantMemorySelector
    DurableMemoryWriteCandidateBuilder

  Shared
    MemoryPolicy
    MemoryGate
    MemoryTrace
    MemoryGovernance
    MemoryStoreAdapter
```

三层优先级：

```text
当前用户输入 > 当前执行结果 > 状态记忆 > 对话记忆 > 检索证据/长期记忆候选
```

补充说明：

```text
检索证据在事实回答中应优先于长期记忆。
状态记忆在继续工作、恢复 handle、任务进度上应优先于对话摘要。
长期记忆只能补充稳定偏好和背景，不能覆盖当前轮事实。
```

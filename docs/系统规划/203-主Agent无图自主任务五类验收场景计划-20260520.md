# 203-主Agent无图自主任务五类验收场景计划

日期：2026-05-20

状态：实施中

## 1. 目标

本计划用于验证主 Agent 在不启动任务图的情况下，是否具备真实长任务处理能力。验收重点不是“回答像完成了”，而是必须通过现有编排底座留下可复查证据：

1. 进入 `autonomous_task_run`。
2. 产生 `TaskRun`、ledger、checkpoint、runtime event。
3. 不创建 `CoordinationRun`。
4. 需要工具时产生真实工具观察。
5. 涉及写入、命令和修复时默认进入 sandbox overlay，避免误伤真实工程。

## 2. 五类场景

| 场景 | 验收目标 | 核心证据 |
|---|---|---|
| 代码问题追踪与修复 | 主 Agent 能复现、定位、修复、验证一个小代码问题 | `read_file/search_text`、`edit_file/write_file`、`terminal`、sandbox |
| 测试报告分析与回归修复 | 主 Agent 能读取失败产物、归类失败、提出根因、补回归或修复建议 | 测试报告 fixture、结构性原因、回归建议 |
| 文档/数据混合分析 | 主 Agent 能读 PDF 与表格，并做交叉业务总结 | `delegate_to_agent` 调用 PDF/table 子 Agent，非任务图 |
| 小型功能端到端实现 | 主 Agent 能理解需求、形成方案、实现前后端/测试草案、验证 | 写入 sandbox 产物、说明验证边界 |
| 运维式排障任务 | 主 Agent 能检查进程/端口/配置/命令结果并给出修复路径 | `terminal` 真实输出、诊断结论、sandbox cwd |

## 3. 统一断言

所有场景默认断言：

1. `task_run.nonempty`
2. `trace.available`
3. `trace.agent_run_results.nonempty`
4. `trace.coordination_runs=0`
5. `response.nonempty`

涉及写入或命令的场景额外断言：

1. `event=runtime_sandbox_prepared`
2. `sandbox.enabled`
3. `sandbox.root.contains=output/sandbox_runs`
4. `sandbox.real_workspace_access=read_only`

## 4. 测试策略

第一阶段先用 fixture 和 sandbox 建立可重复的 smoke 验收，不要求一次覆盖完整生产级大型修复。原因是这组测试的目的不是替代真实开发，而是稳定证明主 Agent 自主任务循环具备基础执行力。

第二阶段再把真实工程任务接入，例如工作台加载失败、60 轮报告修复、节点筛选功能、接口超时排障。真实工程任务可以继续沿用同一 harness，但需要更严格的防误伤策略和更长超时。

## 5. 不做的事

1. 不为了长任务伪造 TaskGraph。
2. 不绕过工具和测试，不能产出假结果应付断言。
3. 不新增独立监测系统，仍复用 health/harness/trace。
4. 不让测试写真实工程文件；写入类验收优先走 sandbox overlay。

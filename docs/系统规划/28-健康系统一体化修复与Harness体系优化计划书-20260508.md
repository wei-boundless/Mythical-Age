## 1. 问题定义

健康系统现在不是“没有做”，而是已经有了 issue、report、command、receipt、verification gate、workbench、test system、harness、experiment 和 trace/replay 的多块实现，但它们还没有收敛成同一个健康治理内核。

当前断点集中在三条链路：

1. 健康 Agent 执行链依赖旧任务模式。
   `backend/health_system/execution_planner.py` 直接从 `TaskFlowRegistry.list_flows()` 查找 `issue_triage`。任务系统旧默认 flow 清理后，健康运行规划直接 `KeyError: 'issue_triage'`。这说明健康 Agent 还没有成为新 TaskGraph / RuntimeAssembly 体系下的正式运行资源。

2. 健康命令、运行准入和运行结果语义混在一起。
   当前回归里出现“准入应 rejected，实际 failed”的情况。`rejected` 应表示执行前被策略拒绝，`failed` 应表示运行已经尝试但失败，`blocked` 应表示证据、资源、gate 或绑定缺失导致暂不能推进。三者混用会让工作台、报告和自动修复策略误判。

3. harness 有执行与报告能力，但不是健康系统的一等状态机。
   `maintenance/harness` 已经提供 profile 执行、`RunResult`、`issues.json`、`trace.jsonl` 和 `report.md`；`maintenance/experiments/runner.py` 又负责启动子进程、写 `run_state.json`、读取产物和恢复 stale running。现在的问题不是要删掉 harness，而是它的进程状态、验证语义、产物清单和健康 issue/report 绑定还没有进入统一健康治理模型。

正确终态应该是：

```text
HealthSystem
  拥有 issue / report / command / receipt / verification / maintenance 的治理语义。

HealthTaskGraph
  把健康 Agent 工作表达成正式任务图，不再依赖旧 task_mode 默认 flow。

HealthRunAdmission
  只负责准入、策略、证据和 gate 判定。

HealthAgentRun / VerificationRun
  拥有稳定生命周期、状态持久化、产物清单和 trace refs。

HarnessExecutor
  只做测试/场景执行、结果收集、报告渲染，不成为第二个健康系统。
```

一句话：

```text
健康系统是治理权威；harness 是执行器；任务图是健康 Agent 装配入口；RunLoop/VerificationRun 是运行事实源。
```

## 2. 技术源报告

### 2.1 现有健康系统能力

`backend/health_system/README.md` 已经明确健康系统负责：

- issue / report / command / receipt 的健康治理主链
- verification gate 与健康工作台
- 测试治理
- regression harness
- trace / replay / experiment 观测链路

这一定义方向正确，但代码现状更像多个维护能力被挂在健康系统名下，还没有形成一致的运行权威。

### 2.2 回归测试现状

本轮健康相关检查曾执行：

```powershell
pytest backend/tests/health_workbench_regression.py backend/tests/health_verification_gate_regression.py backend/tests/health_runtime_admission_cutover_regression.py backend/tests/health_management_control_plane_regression.py backend/tests/orchestration/test_orchestration_trace_replay.py -q
```

结果：

```text
8 passed, 5 failed
```

失败集中在：

- `test_health_runtime_admission_consumes_formal_orchestration_objects`
- `test_health_command_admission_rejects_blocked_operation`
- `test_task_system_exposes_generic_agent_task_connection_profile`
- `test_health_agent_run_preview_does_not_expose_projection_instance`
- `test_default_health_management_agent_configuration_is_bound_and_guarded`

这些失败不是简单断言问题，而是暴露了健康 Agent、任务系统和运行准入之间的结构性断层。

### 2.3 任务图与 RunLoop 约束

27 号计划已经锁定主架构：

```text
TaskGraph 是任务架构唯一入口。
RuntimeAssembly 是运行装配唯一入口。
GraphRunLoop 是多节点推进唯一入口。
AgentLoop 是节点内部执行唯一入口。
```

因此健康系统不应恢复旧的 `issue_triage` 默认 task flow 作为长期方案。健康 Agent 应成为正式的 `HealthTaskGraph` / `TaskGraph` 投影，运行时再经 `RuntimeAssembly` 进入 GraphRunLoop 或单节点 AgentLoop。

### 2.4 任务系统原则约束

`docs/设计原则/14-任务系统.md` 对本轮有四个硬约束：

- 生命周期必须统一：`pending -> running -> completed/failed/killed`。
- 多态接口要极小，只统一确实需要统一分发的操作。
- 任务输出应进入 summary/ref/notification，不把原始执行噪声塞回主上下文。
- root-state 级任务登记要能穿透子 Agent 隔离，避免运行事实丢在局部线程里。

这些原则直接约束健康系统：

- 健康运行和验证运行必须有一等状态对象。
- harness 不应被扩成泛化健康治理抽象。
- 工作台只展示治理摘要、证据引用和下一步动作，不展示低价值标签、样例噪声或原始日志墙。

### 2.5 Harness 当前能力与边界

`backend/tests/README.md` 将测试体系按 `chain / functional / system / scenario` 四层组织，登记权威是 `health_system.maintenance.test_system.case_registry`。统一 harness 入口支持：

```text
python -m health_system.maintenance.harness.run --profile chain
python -m health_system.maintenance.harness.run --profile functional
python -m health_system.maintenance.harness.run --profile system
python -m health_system.maintenance.harness.run --profile scenario
```

`backend/tests/system_eval/README.md` 进一步定义长链和真实任务走 `scenario` profile 或 `long` runner，输出目录为：

```text
output/test_runs/<run_id>/
  report.md
  run_result.json
  issues.json
  trace.jsonl
  artifacts/
```

`maintenance/harness/contracts.py` 已有 `RunContext`、`RunResult`、`ScenarioResult`、`IssueEntry`、`TraceSpan`。`persistence.py` 已能原子写入核心产物。`reporter.py` 已能渲染 Runtime Control 和 Output Commit 摘要。`regression_gate.py` 已能按登记 profile 调度 pytest 或 python 脚本。

结论：harness 不是无用旧残留，不能一刀删；但它必须被降级为 Health Maintenance 下的执行器，状态、语义和治理归健康系统所有。

### 2.6 Harness 当前风险

1. 状态权威分裂。
   `experiments/runner.py` 写 `run_state.json`，harness 写 `run_result.json`，test_system/service 再做一层归一化。三层都能解释运行状态。

2. 进程状态不够持久。
   `_processes` 是内存字典，进程恢复主要依赖 PID 和产物存在性。服务重启后 running 状态可能只能事后推断。

3. 运行语义不够精确。
   当前状态主要是 `running / passed / failed / cancelled / unknown`，缺少 `queued / rejected / blocked / stale / timed_out` 等健康治理需要的状态。

4. 产物清单不是一等模型。
   `RunResult.artifacts` 是 dict，能用但缺少 artifact type、schema version、producer、checksum、relative ref、health issue/report binding。

5. API 层聚合过厚。
   `TestSystemService` 同时处理 profiles、cases、agent report、harness records、long scenarios、start/cancel/run artifacts、turn runtime loop、task run monitor。它已经接近维护控制面，不应继续承载进程执行细节。

## 3. 设计决策

### 3.1 不恢复旧默认 task_mode flow

不得为了让 `issue_triage` 测试通过而重新塞回一组旧默认 task flow。

健康 Agent 的目标形态是：

```text
HealthIssue
  -> HealthTaskRequest
  -> HealthTaskGraph / TaskGraph projection
  -> RuntimeAssembly
  -> GraphRunLoop / AgentLoop
  -> HealthAgentRun
  -> HealthReport / Receipt / TraceRef
```

过渡期可以提供 synthetic task resources，但它们必须以 `task.health.*` 正式资源身份存在，而不是恢复旧的 `task_mode == "issue_triage"` 默认管理记录。

### 3.2 Harness 应优化，不应废弃

保留：

- profile 调度
- pytest/script runner 识别
- `RunResult` / `IssueEntry` / `TraceSpan`
- `report.md` 渲染
- `run_result.json` / `issues.json` / `trace.jsonl` 输出契约

优化：

- 新增 VerificationRun 状态权威。
- 新增 VerificationProfile、VerificationArtifactManifest。
- 将进程执行收束到 `HarnessExecutor` 边界。
- 将 stale、cancelled、timed_out、blocked、rejected 明确建模。
- 将 harness 运行结果回写健康 issue/report/trace refs。

禁止：

- 让 harness 直接创建健康治理结论。
- 让前端直接从 `run_result.json` 推导健康系统状态。
- 把 experiment runner 扩成新的测试系统主入口。

### 3.3 健康系统分层

目标分层：

```text
backend/health_system/
  governance/
    issue/report/command/receipt policy

  runtime/
    health_task_request
    health_agent_run
    runtime_admission
    execution_planner

  verification/
    verification_profile
    verification_run
    verification_gate
    artifact_manifest

  maintenance/
    test_system/
      case registry, assertions, advisor, projections
    harness/
      executor, contracts, persistence, reporter
    experiments/
      trace/turn artifact readers and overlays
```

当前可以不马上改目录，但实现职责必须按这个分层收敛。

## 5. 固定执行流

### 5.1 健康 Agent 执行流

```text
HealthCommand
  -> HealthRunAdmission.evaluate()
  -> rejected / blocked / accepted
  -> HealthTaskRequest
  -> HealthExecutionPlanner.build()
  -> TaskGraph projection
  -> RuntimeAssembly
  -> GraphRunLoop / AgentLoop
  -> HealthAgentRun terminal state
  -> HealthReport + Receipt + trace refs
```

状态语义固定：

- `rejected`：策略或权限在执行前拒绝，不能进入 runner。
- `blocked`：缺少证据、绑定、资源或 gate，暂不能执行。
- `failed`：已经进入运行链路，但运行失败。
- `killed/cancelled`：用户或系统主动终止。

### 5.2 Verification / Harness 执行流

```text
HealthCommand(run_verification)
  -> VerificationAdmission
  -> VerificationRun(queued)
  -> HarnessExecutor.start()
  -> VerificationRun(running)
  -> harness writes run_result/issues/trace/report
  -> ArtifactManifest.normalize()
  -> VerificationRun(passed/failed/stale/timed_out/cancelled)
  -> HealthReport / HealthIssue refs
```

Harness 只返回事实：

```text
exit code
stdout/stderr/log
run_result
issues
trace spans
artifact paths
```

HealthSystem 决定：

```text
是否生成 issue
是否生成 report
是否允许 cutover
是否阻断工作台动作
```

## 6. 分阶段实施计划

### Phase 1：修复健康 Agent 任务装配权威

目标：清理旧 `task_mode` 默认 flow 依赖，让健康 Agent 以正式任务资源进入装配链。

改动范围：

- `backend/health_system/execution_planner.py`
- `backend/health_system/runtime_admission.py`
- `backend/tasks/*`
- `backend/orchestration/*`
- `backend/tests/health_runtime_admission_cutover_regression.py`
- `backend/tests/health_management_control_plane_regression.py`

实施要点：

- 新增 HealthTaskRequest 构造逻辑。
- 将 `issue_triage`、`trace_analysis`、`case_draft`、`fix_verification` 表达为 `task.health.*` synthetic task resources。
- `HealthExecutionPlanner` 不再直接依赖 `TaskFlowRegistry.list_flows()` 查旧默认 flow。
- 运行预览不得暴露 projection instance，只暴露正式 runtime spec、binding、trace/artifact refs。

完成标准：

- `KeyError: 'issue_triage'` 消失。
- `agent:3` 健康连接 profile 能以正式健康任务资源暴露。
- 健康 Agent 默认配置为 bound and guarded。

### Phase 2：统一准入、阻断和失败语义

目标：把 `rejected / blocked / failed` 从 receipt、admission、run result 中拆清。

改动范围：

- `backend/health_system/runtime_admission.py`
- `backend/health_system/command_service.py`
- `backend/health_system/command_builder.py`
- `backend/health_system/models.py`
- `backend/health_system/workbench.py`
- `backend/tests/health_management_control_plane_regression.py`

实施要点：

- admission 只产出 `accepted / rejected / blocked`。
- runner 只在 accepted 后启动。
- receipt 的 `accepted` 和 `status` 不再互相推导。
- command_service 不把准入拒绝包装成运行失败。

完成标准：

- `test_health_command_admission_rejects_blocked_operation` 通过。
- 工作台动作能区分“不能执行”和“执行失败”。

### Phase 3：建立 VerificationRun 和 ArtifactManifest

目标：让健康系统拥有测试/验证运行的一等事实源。

改动范围：

- `backend/health_system/models.py`
- `backend/health_system/store.py`
- `backend/health_system/verification_service.py`
- `backend/health_system/maintenance/test_system/contracts.py`
- `backend/health_system/maintenance/test_system/service.py`
- `backend/health_system/maintenance/harness/contracts.py`
- `backend/health_system/maintenance/harness/persistence.py`
- `backend/tests/harness/persistence_report_regression.py`

实施要点：

- 新增 `VerificationRun`、`VerificationProfile`、`VerificationArtifactManifest`。
- `run_state.json` 继续作为本地执行恢复文件，但不再是健康 API 的语义权威。
- `run_result.json`、`issues.json`、`trace.jsonl`、`report.md` 必须被 manifest 标记 required/present。
- 产物路径必须统一 repo-relative ref，禁止前端拼文件系统路径。

完成标准：

- 健康维护 API 返回 VerificationRun，而不是直接返回 experiment runner state。
- 缺失核心产物时状态为 `stale` 或 `failed`，原因可追踪。

### Phase 4：收束 HarnessExecutor 边界

目标：保留 harness 执行价值，去掉执行、状态、治理三者混杂。

改动范围：

- `backend/health_system/maintenance/harness/regression_gate.py`
- `backend/health_system/maintenance/harness/run.py`
- `backend/health_system/maintenance/experiments/runner.py`
- `backend/health_system/maintenance/test_system/service.py`
- `backend/tests/README.md`
- `backend/tests/system_eval/README.md`

实施要点：

- 引入 `HarnessExecutor.start/cancel/refresh/read_artifacts` 小接口。
- `ExperimentRunner` 降为 trace/turn artifact 管理与运行兼容适配层。
- 进程执行状态写入统一 service，不在多个地方推断。
- stale running 恢复规则固定：无进程 + 无 terminal artifact = stale；有 run_result = 按结果归档；取消有明确 cancelled 终态。

完成标准：

- harness regression gate 仍能独立命令行运行。
- 健康 API 只通过 VerificationRun 暴露状态。
- 服务重启后 running/stale/passed/failed 能稳定恢复。

### Phase 5：工作台降噪与治理视图重排

目标：健康系统前端不再把所有维护信息堆成一页，卡片只呈现治理事实和行动入口。

改动范围：

- `frontend/src/components/workspace/views/HealthSystemView.tsx`
- `frontend/src/components/health/HealthTraceTimeline.tsx`
- `frontend/src/components/health/HealthAgentDock.tsx`

实施要点：

- 分层入口：Overview、Issues、Verification、Harness Runs、Trace Replay、Agent Runs。
- 卡片上去掉无意义 badge、演示标签和重复描述。
- 样例/默认数据必须隐藏或进入单独 demo 区域，不混入真实健康概览。
- 日志、trace 和 raw artifact 默认折叠，只显示摘要、失败原因、证据 ref 和下一步动作。

完成标准：

- 首屏只回答：当前健康状态、阻断项、最近验证、下一步动作。
- 任一卡片没有无意义标签；标签只用于状态、严重级别、运行层级、是否阻断。

### Phase 6：门禁矩阵与切流

目标：将健康系统修复和 harness 优化纳入固定门禁。

必须通过：

```powershell
pytest backend/tests/health_workbench_regression.py backend/tests/health_verification_gate_regression.py backend/tests/health_runtime_admission_cutover_regression.py backend/tests/health_management_control_plane_regression.py backend/tests/orchestration/test_orchestration_trace_replay.py -q
python backend/tests/run_regression_gate.py --profile chain
python backend/tests/run_regression_gate.py --profile functional
```

视改动范围补跑：

```powershell
python backend/tests/run_regression_gate.py --profile system
python -m health_system.maintenance.harness.run --profile chain
python -m health_system.maintenance.harness.run --profile functional
```

完成标准：

- 健康五组回归全绿。
- harness 产物仍包含 `run_result.json`、`issues.json`、`trace.jsonl`、`report.md`。
- VerificationRun 能引用所有产物。

## 7. 文件级执行清单

## 8. 迁移与清理规则

1. 不保留旧默认 task flow 作为兼容层。
   如果某条健康链路需要 `issue_triage`，必须迁到 `task.health.issue_triage`。

2. 不新增“临时 fallback flow”。
   任何 fallback 都必须是正式 HealthTaskRequest 或 VerificationRun 状态。

3. 不让前端识别旧运行结构。
   前端只消费健康 API DTO，不判断 `run_state.json` 或 `run_result.json` 文件结构。

4. 不把 sample/default 数据混入真实概览。
   需要演示数据时必须标成 demo，并默认隐藏。

5. 不扩大 harness 职责。
   harness 只执行和产出事实；健康系统生成治理结论。

## 9. 风险控制

### 风险：为快速修绿恢复旧残留

控制：

- 禁止恢复通用 `issue_triage` 默认 flow。
- 所有健康任务都必须以 `task.health.*` 或 TaskGraph projection 进入装配。

### 风险：harness 优化变成大拆迁

控制：

- 第一阶段保留现有命令行入口和产物格式。
- 先加 VerificationRun adapter，再移动执行边界。
- `run_result.json`、`issues.json`、`trace.jsonl`、`report.md` 不破坏。

### 风险：状态名继续混用

控制：

- admission、run、verification 三类状态分表建模。
- receipt 必须携带 `admission_status` 或 `run_status` 的明确来源。

### 风险：工作台继续高噪声

控制：

- 每张卡只允许显示状态、严重级别、阻断原因、证据 ref、下一步动作。
- raw log、完整 trace、manifest 明细必须进入详情层。

## 10. 最终验收标准

健康系统达到“绿”必须同时满足：

1. 健康 Agent 能从 issue 生成正式任务请求、运行装配和运行预览。
2. 准入拒绝不会被记录为运行失败。
3. VerificationRun 成为测试/验证运行状态权威。
4. harness 仍能独立执行 profile，并稳定产出四类核心文件。
5. 健康报告能引用 issue、verification run、trace、artifact manifest。
6. 工作台首屏无样例噪声、无无意义标签、无 projection instance 泄漏。
7. 健康五组回归和 chain/functional gate 通过。

本计划实施后，健康系统不再是“聚合页 + 测试入口 + 若干报告”的松散组合，而是项目的健康治理控制面；harness 也不再是边界含混的实验工具，而是稳定、可追踪、可被健康系统治理的验证执行器。

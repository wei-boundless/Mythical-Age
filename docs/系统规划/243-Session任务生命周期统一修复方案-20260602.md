# Session 任务生命周期统一修复方案

日期：2026-06-02

## 关键修正

图任务的“长期”不表示一个 session 下可以反复创建多个 graph run。

本项目当前的 GraphRuntime 已经把 `graph_run_id` 作为图任务运行实例 id：GraphLoop、checkpoint、work order、node result、monitor 都围绕这个 id 读写。因此目标模型必须收束为：

- `graph_id`：图定义 / 模板 id。
- `graph_run_id`：长期图任务实例 id。
- 一个 session 最多绑定一个 `graph_run_id`。
- 一个对话就是一个具体任务容器；如果要创建另一个图任务实例，应创建另一个 session。
- 长期续跑靠同一个 `graph_run_id` 的 checkpoint / resume / dispatch 维持，不在同一个 session 下创建 run chain。

之前方案里关于“同一 session 多 graph run”、“`run_chain`”、“`project_task_key = session_scope_key + graph_id`”的设计全部撤销。

## 背景

当前 `DELETE /api/sessions/{session_id}` 只删除 session JSON 和 session memory：

- `backend/api/sessions.py`
- `backend/sessions/__init__.py`

但 runtime 任务状态还在：

- `RuntimeStateIndex.upsert_task_run()` 会把所有带 `session_id` 的 `TaskRun` 写入 `sessions/{session_id}` 索引。
- 图任务启动接口 `backend/api/orchestration.py` 已要求 `task_environment` scope，并把 graph run 写入同一个 session 维度。
- `GraphTaskLifecycleManager.delete_graph_run()` 已经能删除 graph run、artifact repository、checkpoint、runtime object、runtime event、state index。
- 但 `GraphTaskLifecycleManager._artifact_paths()` 当前硬编码 `storage/task_environments/creation/writing/artifacts/{project_id}`，与任务环境隔离冲突。
- 前端仍有 `task_graph_studio` / `session:task_graph_studio` fallback，后端也有同名默认值，这是图任务纳入 session 管理后的旧入口残留。

问题不是“图任务没有 session”，而是 session 删除链路没有成为 task/runtime 生命周期的统一入口，同时 session 还没有保存它绑定的长期图任务实例 id。

长期图任务需要持久化 project 作为任务容器，但 project/library 不属于 session binding。session 只维护当前 graph instance 与 project/library 的维护关系；session 删除后维护关系结束，库清理由 project/library 生命周期处理。

## 目标语义

`session` 是任务环境中的工作容器，不只是聊天对话。

一个 session 可以拥有：

- 对话消息和 API transcript。
- chat/turn runtime run。
- 普通 single-agent task run。
- 零个或一个长期 task graph instance。
- session 监控索引、active turn、prompt accounting、execution receipts。

图任务绑定规则：

- 普通对话 session 的 `task_binding` 为空。
- 图任务 session 的 `task_binding.kind == "task_graph"`。
- `task_binding.graph_run_id` 是绑定权威。
- `task_binding.graph_id` 只记录该实例来自哪个图定义，不能单独作为实例标识。
- `task_binding.graph_harness_config_id` 和 `task_binding.root_task_run_id` 用于恢复、监控和审计。
- `task_binding` 不绑定 project library、artifact repository、formal memory repository 等库资源。
- session 已绑定 `graph_run_id` 后，不允许再创建新的 graph run。
- 续跑、派发、监控必须使用绑定的 `graph_run_id`。
- 若用户需要新的图任务实例，前端应创建新的 session，而不是复用当前 session。

推荐 session 记录字段：

```json
{
  "task_binding": {
    "kind": "task_graph",
    "graph_run_id": "grun:...",
    "graph_id": "graph.xxx",
    "graph_harness_config_id": "ghcfg:...",
    "root_task_run_id": "taskrun:...",
    "bound_at": 0
  }
}
```

隔离规则：

- 环境 / 项目隔离仍只来自 `session.scope`：
  - `workspace_view`
  - `task_environment_id`
  - `project_id`
- `task_binding` 不重复存环境和项目字段，避免 scope 漂移。
- 同一个 `graph_run_id` 不允许绑定到两个 session。
- GraphRun 自身的 `session_id`、`session_scope_key` 与 session 记录必须一致。

项目 / 库维护规则：

- 图任务一律由 session 启动。
- 需要长期人工维护的图任务必须落到持久化 project；graph 定义和 published GraphHarnessConfig 也必须持久化。
- 图任务 start 如果需要 project 但 request/session scope 没有 `project_id`，由后端在 start 边界创建或解析一个持久化 project，并把 `project_id` 写入 `session.scope` 后再进入 GraphRun 创建。
- project library、artifact repository、formal memory repository 等库资源由 project/library 生命周期管理，不挂在 session binding 上。
- graph session 存在时，它是该图任务实例对 project/library 的维护入口。
- graph session 删除后，这些库不再由该 session 维护；旧库或无人维护的库可以由 project/library cleanup 链路清理。

长期图任务规则：

- 图任务不保留独立于 session 的生产启动/删除路径。
- `graph_run_id` 本身就是长期实例 id。
- GraphLoop checkpoint、active node、work order、node result、monitor 都保持在该实例下续跑。
- `GraphTaskLifecycleManager.delete_graph_run(graph_run_id)` 删除的是整个长期图任务实例，不是单次尝试。
- `ProjectProgressLedger` / `ProjectRuntimeStatus` 只能作为项目进度视图或监督视图，不能作为图任务实例 id 的替代品。
- 本轮不新增 `project_task_key`、`session_project_tasks`、`project_graph_tasks`、`graph_project_task_index`。

不随 session 删除：

- 任务环境定义。
- task graph 定义和已发布 graph harness config。
- project instance 元数据。
- project library / durable project files / project formal memory 不由 session 直接删除；session 删除只解除维护关系，后续清理由 project/library 生命周期处理。
- 其他 session 下的任务和监控状态。

随 session 删除：

- session 记录、消息和 API transcript。
- session-owned runtime/task/turn/graph instance 状态。
- graph session 对 project/library 的维护关系。

## 决策边界

1. `api.sessions` 只做请求边界和 session 存在性校验。
2. 新增 session lifecycle 服务作为删除决策权威。
3. `SessionManager` 只负责 session 记录读写和 `task_binding` 原子校验，不负责删除 runtime 状态。
4. 图任务实例删除必须复用 `GraphTaskLifecycleManager`，不能直接 prune task run。
5. `GraphTaskLifecycleManager` 是 graph instance 删除权威；它自身要补齐 prompt accounting、execution receipts、runtime object、artifact root 范围判断。
6. `RuntimeStateIndex`、`RuntimeRunRegistry`、`RuntimeExecutionStore`、`PromptAccountingLedger`、`RuntimeObjectStore` 只负责存储删除，不负责判断哪些该删。
7. graph-run API 是运行操作边界；所有 graph-run route 都必须先从 `graph_run_id` 解析 owning session，再校验 session scope 和 `task_binding.graph_run_id`。

## 实施计划

### 1. 增加 session task binding

修改：

- `backend/sessions/__init__.py`
- `backend/api/orchestration.py`
- `backend/api/task_system.py`
- `backend/task_system/repositories/project_instance_repository.py`
- `backend/task_system/repositories/project_library_manifest_repository.py`

`SessionManager` 增加：

- `get_task_binding(session_id)`
- `bind_session_graph_instance(session_id, graph_run_id, graph_id, graph_harness_config_id, root_task_run_id)`
- `assert_session_graph_instance(session_id, graph_run_id)`
- `list_sessions()`、`get_history()`、`_summary_from_payload()` 返回 `task_binding`

绑定规则：

- 创建普通 session 时 `task_binding` 为空。
- graph start 前先确认 session 当前未绑定图任务实例。
- graph start 如果需要 project 且 session scope 没有 `project_id`，必须先创建或解析持久化 project 并把 `project_id` 写回 session scope；不能继续返回 `project_id is required`。
- graph start 必须拆成受控时序：先创建 GraphRun/GraphLoop 基础状态但不执行 node，不 auto-run；写入 session binding 后，才允许 dispatch-ready / auto-run。
- 如果并发导致绑定写入时发现 session 已绑定其他 `graph_run_id`，必须删除刚创建且尚未执行的 graph instance 后返回 409，不能留下孤儿实例；project/library 不在 session start 回滚里隐式删除。
- graph monitor / resume / dispatch / node result / work order execute / run-until-idle 都要校验 route `graph_run_id` 等于 session `task_binding.graph_run_id`。
- 如果 GraphRun 有 `session_id`，还要校验 GraphRun 的 session 与绑定 session 一致。

已绑定 session 再 start 的处理：

- 不创建新的 graph run。
- 返回 409：`session_graph_instance_already_bound`。
- 前端应转为“续跑/打开监控”语义。

direct graph-run delete 的处理：

- graph instance 的生产删除入口统一走 session delete。
- graph-run delete API 不再作为独立生产删除路径，默认返回 409：`graph_instance_requires_session_delete`。
- session 删除链路可以调用 `GraphTaskLifecycleManager.delete_graph_run()`，因为 session lifecycle 是绑定关系的删除权威。
- graph start 事务内部的绑定失败回滚可以调用 graph lifecycle 清理刚创建且未执行的临时实例；这属于启动事务回滚，不是第二套 graph lifecycle。

### 3. 修正任务环境 session resolver

修改：

规则：

- `intent=resume_graph` 必须传 `graph_run_id`。

### 4. 新增 session lifecycle 服务

新增：

- `backend/harness/runtime/session_lifecycle.py`

提供：

- `preview_delete_session(runtime, session_id)`
- `delete_session_runtime(runtime, session_id)`

删除顺序：

1. 读取 session 记录和 `task_binding`。
2. 读取 session 下所有 `TaskRun`、`TurnRun`、chat `RuntimeRun`。
3. 从 `task_binding.graph_run_id` 和 session-owned task diagnostics 中收集该 session 拥有的 graph instance。
4. 取消或终止当前 session 仍在运行的后台任务。当前 `SingleAgentRuntimeHost.spawn_background_task()` 只有 task set，没有 name 索引；实施时必须补一个按 task name / task_run_id / stream_run_id 取消的显式能力，不能只写状态：
   - `chat-run-{stream_run_id}`
   - `task-run-executor:{task_run_id}`
   - `task-run-executor-recover:{task_run_id}`
5. 等待被取消的后台任务退出或进入明确终止态，再删除 registry/state；不能在协程仍可能回写时清存储。
6. 对每个 graph instance 调用 `GraphTaskLifecycleManager.delete_graph_run(graph_run_id)`。
7. 重新读取 session 下剩余非 graph task run。
8. 删除剩余 task run、turn run、chat stream run、active turn、prompt accounting、execution receipts、event log、runtime object、session live view。
9. chat stream run 清理必须成对删除：
   - `RuntimeRunRegistry` 中该 session 的 run 记录。
   - 每个 run 对应的 public SSE event log：`run.event_log_id` / `chatrun:*`。
10. active turn 清理必须删除 `active_turn` runtime object 和它挂载的 `active_turn_pending_input` refs，避免删除 session 后仍能续接。
11. 如果 session 关联 project，记录该 graph session 对 project/library 的维护关系结束；不在 session 删除中隐式删除 project/library。
12. 最后由 API 删除 session memory 和 session JSON。

删除必须 fail closed：runtime 清理失败时不能继续删除 session JSON，否则会制造不可恢复的孤儿运行状态。

### 5. 扩展存储清理能力

需要增加或补齐：

- `RuntimeStateIndex.prune_turn_runs(turn_run_ids)`
- `RuntimeStateIndex.prune_session_runtime_records(session_id, task_run_ids, turn_run_ids)`
- `RuntimeRunRegistry.delete_session_runs(session_id)`，返回被删 run 和 `event_log_id` 供 event log 清理。
- `RuntimeExecutionStore.prune_task_runs(task_run_ids)`
- `PromptAccountingLedger.prune_session(session_id, task_run_ids)`，同时按 `session_id` 和 `task_run_id` 删除 segment maps、segments、usage、cache、stability 记录。
- `ActiveTurnRegistry.clear_session(session_id, reason)`，删除 active turn 并返回 pending input refs。
- `RuntimeObjectStore.delete_task_run_objects(task_run_ids, session_id="")`，覆盖 task object、active turn、pending input、session-owned runtime object。

这些方法只删除明确传入或明确属于该 session 的记录，不做 owner 判断。

`ProjectProgressLedger` / `ProjectRuntimeStatus` 本轮只按已有 `session_id`、`active_task_run_id` 做 session 删除清理，不引入新的项目任务身份模型。

project/library cleanup 本轮只做维护关系记录，不在 session runtime 删除服务里删除 project instance 或 project library。

### 6. 补齐 graph lifecycle 删除能力

修改：

- `backend/harness/graph/lifecycle_manager.py`

补齐：

- 删除 prompt accounting：`prompt_accounting_ledger.prune_task_runs(task_run_ids)`。
- 删除 execution receipts：`execution_store.prune_task_runs(task_run_ids)`。
- 删除 graph/task 相关 runtime object：保留现有 graph 匹配，同时覆盖非 graph-root task object。
- 修正 formal memory 删除范围：
  - `GraphTaskLifecycleManager` 只删除 graph instance runtime 相关记录。
  - 不能把 `project_id` 放进 `scope_ids` 直接删除，project/library 记忆清理由 project/library 生命周期处理。
- project/library 维护关系处理：
  - `GraphTaskLifecycleManager` 不负责删除 project library、project formal memory 或 durable project files。
  - session lifecycle 只记录维护关系结束；旧库清理必须走 project/library cleanup，依据“没有活跃 graph session 维护”来判断。
- 修正 artifact path 删除范围：
  - 不再硬编码 `creation/writing`。
  - artifact repository 只能按 `graph_run_id`、`task_run_id` 删除 graph instance runtime 记录；不能把 `project_id` 当作删除 scope。
  - project library / durable project files 不在 graph instance 删除中隐式删除。

### 7. 接入 session 删除 API

修改：

- `backend/api/sessions.py`

当前：

```python
runtime.session_manager.delete_session(session_id)
runtime.memory_facade.delete_session_memory(session_id)
```

改为：

```python
cleanup = delete_session_runtime(runtime, session_id)

```

如果前端不需要详情，可以保持 `ok` 字段稳定，但后端应返回 cleanup 便于审计。

### 8. 回归测试

新增/修改测试：

- 创建任务环境 session 后首次 start graph，会写入 `task_binding.graph_run_id`。
- 图任务 start 在没有 `project_id` 且图任务需要项目时，会创建或解析持久化 project，并把 `project_id` 写入 session scope。
- graph session 的 `task_binding` 不写入 project library、artifact repository、formal memory repository 等库 id。
- graph start 在 binding 写入前不会 auto-run 或执行 node；绑定失败回滚不会留下已执行副作用。
- graph start 绑定失败回滚时，只清理本次未执行的 graph instance，不把 project/library 当作 session-owned 资源隐式删除。
- 已绑定 graph instance 的 session 再 start 返回 409，不创建新 GraphRun。
- monitor / resume / dispatch / node result / work order execute / run-until-idle 的 route `graph_run_id` 与 session binding 不一致时返回 409。
- direct graph-run delete 返回 409，graph instance 删除必须走 session delete。
- `intent=resume_graph` 能按 `graph_run_id` 找回正确 session，并校验 scope。
- 相同 task environment 下新建另一个图任务实例时会创建新 session。
- 不同 `task_environment_id` / `project_id` 的 session scope 互不复用；GraphRun 的 graph config 所属任务环境必须与 session scope 匹配。
- `CreativeEnvironmentView`、center workspace、task graph publish 页面都不能再生成 `task_graph_studio` 或 `session:task_graph_studio`，也不能把真实 session id 改写后传入 graph start。
- 删除含 graph instance 的 session 后：
  - graph run monitor 404。
  - root task run 和 node executor task run 不再出现在 state index。
  - runtime event/checkpoint/runtime object 被删除。
  - task graph 定义和 published config 仍存在。
  - project/library 不被 session 删除隐式删除。
  - project/library 维护关系被标记为不再由该 graph session 维护。
- 删除含 chat run 的 session 后：
  - `/api/chat/sessions/{session_id}/latest-run` 返回 404。
  - `RuntimeRunRegistry` 中该 session 的 run 被删除。
  - `chatrun:*` public stream event log 被删除。
  - session runtime monitor 不再显示 task run。
  - turn run、active turn、prompt accounting、execution receipts 被清理。
  - active turn pending input refs 被清理。
- 删除运行中的 chat stream / task executor session 后：
  - 后台任务被取消或终止，并在存储清理前完成退出确认，不再继续写已删除 session。
- 删除 session A 不影响 session B 的 task run。
- 启动图任务未传真实 session 时返回 400。
- 图任务 artifact 清理不再硬编码 creation/writing；非 writing 环境 graph instance 删除不会删错路径。
- graph session 删除不会删除 project formal memory、project library 或 durable project files；只会结束该 graph session 对 project/library 的维护关系。

## 风险与处理

- 如果运行中的 chat stream 正在写 session，删除前必须先关闭 active turn 和 runtime run，否则会继续写已删除 session。
- 如果 graph instance 已经派生 node executor，必须通过 `GraphTaskLifecycleManager` 删除，不能只 prune root task run。
- 如果 project instance 仍存在但 session runtime 状态已删，这是预期行为；项目删除应走项目生命周期链路。
- 如果 graph instance 使用 project artifact repository 或 project formal memory，session 删除不能隐式删除 project files / project memory；只能删除运行记录并结束维护关系。旧库清理走 project/library 生命周期。

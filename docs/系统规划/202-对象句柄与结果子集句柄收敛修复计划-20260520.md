# 对象句柄与结果子集句柄收敛修复计划

## 1. 问题定义

当前 follow-up / recall 的核心问题不是单点判断失误，而是运行时把“最终自然语言回答”错误当成了对象与结果真相来源。

最新失败案例显示出三类症状：

- `turn-03`、`turn-05` 中回答本身基本正确，但 `main_context` 未稳定写回，导致 `main.active_dataset.nonempty` 断裂。
- `turn-05` 中 `subset_labels` 被错误压成了 `数据源 / 筛选条件 / 查询模式 / 员工编号 / E-0074...` 这类展示文本，说明系统正在用回答文本反推“结果子集”。
- `turn-08` 中天气请求已经不再被旧数据集污染，但执行层仍未遵守 realtime 直达边界，仍通过 `delegate_to_agent` 间接走实时查询。

根因是：

- `projection_from_bound_answer()` 仍会基于 `resolved_bindings + final_answer` 反向生成 `main_context / task_summary_refs / subset semantics`。
- `bound_answer_projection` 与真正的结构化执行结果投影并存，导致对象级 truth 与结果级 truth 混在一起。
- 主上下文写回依赖“有没有从回答文本里投影出内容”，而不是依赖“本轮执行结果是否形成了有效对象/结果句柄”。
- realtime / external 请求虽已脱离旧 continuation 污染，但 recipe/tool 选择层缺少更强的硬边界。

## 2. 设计原则

本次按以下硬原则修复：

- 对象真相来自执行结果，不来自最终回答文本。
- 结果子集真相来自结构化句柄或结构化回传，不来自回答格式猜测。
- 主上下文写回由 runtime 基于执行结果确定性提交，不由模型回答内容触发。
- realtime / weather / gold / external 请求一旦成立，就不能再被文件对象 follow-up 语义干扰执行路径。
- 无用旧链必须清理，不保留“兼容性文本投影”默认入口。

## 3. 目标结构

### 3.1 对象层

对象层只表示被操作的数据对象或文档对象：

- `active_object_handle_id`
- `active_binding_identity`
- `active_constraints.active_dataset / active_pdf`

对象层只能由以下来源更新：

- MCP canonical result / evidence projection
- agent delegation result 中的结构化 `context_writeback_hints`
- 显式用户输入路径 + 执行成功后的 runtime commit

### 3.2 结果层

结果层只表示一次执行形成的结果：

- `active_result_handle_id`
- `followup_target_task_id`
- `task_summary_refs`

结果层必须绑定到明确句柄，不能由最终回答文本直接伪造。

### 3.3 子集层

子集层只表示“对象上的有限范围结果”：

- `active_subset_handle_id`
- `active_constraints.subset_labels`
- `active_constraints.subset_filter_column`

子集层只能来自：

- `canonical_result.presentation_hints.subset_handle_id`
- 结构化 evidence worker 产出的 subset handle
- 子 agent 通过标准回传协议显式提交的 subset hints

禁止来源：

- `projection_from_bound_answer()` 基于自然语言回答反推 subset
- `task_summary_refs.answer/summary` 文本逆向压缩 subset labels

### 3.4 写回层

主上下文写回改为“执行结果提交”，不再依赖最终回答文本投影。

规则：

- 若本轮执行成功并返回对象句柄/结果句柄/子集句柄，则 runtime 必须提交对应 `main_context`。
- 若本轮只是对已有结果做文本收口，但未产生新结构化结果，也不能清空原有对象状态。
- 只有显式目标切换成功时，才允许替换当前对象上下文。

### 3.5 Realtime 硬边界

当当前 turn 命中 `weather_domain / gold_price_domain / external_requirement / official_source_requirement / freshness_requirement(realtime)`：

- continuation candidate 不得参与对象/子集执行意图选择
- follow-up object recall 不得驱动 `structured_data_analysis` / `pdf` recipe
- tool lane 必须优先落到 realtime/web capability，而不是 dataset/pdf specialist handoff

## 4. 实施清单

### 4.1 清理旧文本投影链路

- `backend/context_management/projection.py`
  - 废弃 `projection_from_bound_answer()` 的默认生产用途。
  - 保留仅供极窄 fallback 的话，也必须禁止生成 subset-level truth；优先考虑直接删除其 runtime 调用。
- `backend/orchestration/runtime_loop/task_run_loop.py`
  - 删除/替换调用 `projection_from_bound_answer()` 的默认路径。
  - 统一改用 evidence/canonical result/context_writeback_hints 做写回来源。

### 4.2 用结构化执行结果接管写回

- `backend/evidence/projection.py`
  - 强化 `MCPProjectionAdapter`：dataset/pdf result 成功时必须稳定产出 `MainContextState + TaskSummaryRef`。
  - subset 只接受 `presentation_hints` 的显式结构化字段。
- `backend/orchestration/runtime_loop/agent_delegation_executor.py`
  - 核查子 agent 回传的 `context_writeback_hints` 是否完整覆盖对象/结果/子集。
- `backend/query/runtime.py`
  - 写回判定从 `bool(main_context or task_summary_refs...)` 提升为“结构化上下文提交成功”语义。

### 4.3 收紧结果子集语义

- `backend/continuation/candidate_collector.py`
  - 只允许从结构化 `task_summary_refs` 中收集 subset 候选。
  - 若 summary 没有结构化 subset handle / constraints，则候选只能是 object/result，不得晋升为 subset。
- `backend/context_management/resolver.py`
  - recall candidate 允许带 subset handle，但不得从纯文本 summary 补造 subset labels。

### 4.4 加强 realtime 路由边界

- `backend/tasks/execution_shape_resolver.py`
- `backend/tasks/assembly_support.py`
- `backend/orchestration/agent_runtime_chain.py`

目标：

- realtime/external 请求不落入 dataset/pdf specialist recipe
- `weather/gold` 这类 turn 默认直接走 realtime tool lane
- 允许委派，但只允许委派到 realtime/search 语义，不允许遗留文件对象语义

### 4.5 测试更新

- 删除或重写围绕 `bound_answer_projection` 的旧测试命名与假设。
- 新增回归：
  - 回答正确但 writeback 不能丢对象状态
  - top5 employees follow-up 不能从文本 summary 伪造 subset labels
  - realtime weather turn 不得走 dataset continuation
  - 切回 `inventory.xlsx` 时对象切换和写回必须稳定
- 复跑：
  - `commerce-ops-data-live-switch`
  - `research-brief-and-document-resume`
  - 相关单测集合

## 5. 验证标准

满足以下条件才算修复完成：

- 生产代码默认链路不再消费 `bound_answer_projection` 作为 file-work truth 来源。
- dataset/pdf follow-up 的 subset candidate 必须可追溯到结构化 subset handle 或结构化约束。
- `turn-03`、`turn-05` 这类 follow-up 回合回答正确时，`main_context.active_dataset` 不再丢失。
- `turn-08` 天气请求命中 realtime lane，不再显示 `plan.tool=delegate_to_agent` 代替 `web_search`。
- `commerce-ops-data-live-switch` 全部通过。
- 不允许通过放宽 harness 断言来掩盖真实问题。

## 6. 风险控制

- 若删除旧文本投影链后部分弱结构场景失去 recall，优先补结构化句柄回传，不允许恢复文本猜测链。
- 若子 agent 回传字段不完整，先补 delegation protocol / executor result 映射，不以主 agent 推测补洞。
- 若 realtime lane 仍需委派，也要把语义约束收紧到 realtime domain，而不是泛化 `delegate_to_agent`。

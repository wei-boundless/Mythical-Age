## 1. 总判断

当前任务系统已经有任务域、特定任务、单任务装配、协调拓扑、契约库、契约预检等模块，但这些模块还没有围绕统一的 `TaskGraph` 组织起来。

现在的问题不是“页面不够漂亮”，而是产品对象没有统一：

```text
任务域管理
  管域和任务清单。

单任务装配
  以 workflow / projection / flow / execution / memory 表单为中心。

协调任务
  以 coordination_task + topology_template + protocol 为中心。

契约总览
  以 manifest / runtime assembly 预览为中心。
```

这些对象都重要，但用户真正要设计的是：

```text
一个任务如何被 Agent 或多个 Agent 完成。
```

因此任务系统前端应整体改成：

```text
Task Graph Studio
  单 Agent 任务 = 一节点图
  多 Agent 任务 = 两节点或多节点图
  多 Agent 协调任务 = 多节点图
```

单 Agent 与多 Agent 不再是两套页面，而是同一个任务图模型在不同复杂度下的表现。

## 2. 当前前端问题

### 2.1 顶层页面层级混杂

当前 `TaskSystemView.tsx` 的顶层层级是：

```text
domain
assembly
coordination
contract_overview
contract_library
```

问题在于：

1. `domain` 是目录和治理层。
2. `assembly` 是单任务运行装配层。
3. `coordination` 是多 Agent 图编辑层。
4. `contract_overview` 是预检层。
5. `contract_library` 是全局资产库。

它们被做成同级切换，导致不同抽象层级混在一个页面导航里。用户在“管理任务域”和“设计任务图”之间来回跳，当前任务、当前协调任务、当前节点状态也容易错位。

### 2.2 任务域与其它页面联动不稳定

当前任务域选择会间接影响：

```text
selectedTaskId
selectedCoordinationId
selectedDomainTasks
coordinationTasks
contract overview
```

但前端没有一个清晰的“当前工作对象”模型。

应该明确：

```text
当前任务域 currentDomain
当前任务 currentTask
当前任务图 currentTaskGraph
当前选中节点 currentNode
当前选中边 currentEdge
```

现状更像多个 `selectedXxxId` 各自维护，靠 effect 修正。这会导致页面联动表面可用，但复杂编辑时心智不稳。

### 2.3 单任务装配和协调任务割裂

单任务装配现在仍是：

```text
workflow
projection
flow
execution
memory
```

协调任务则是：

```text
coordination task
topology template
communication protocol
graph nodes
graph edges
```

这不符合新的原则：

```text
单 Agent 任务也应纳入图结构。
```

正确方向：

```text
单任务装配不再是独立页面。
它应成为 TaskGraph 的图级属性 + 单个 Agent 节点属性。
```

### 2.4 协调编辑器还不够通用

`CoordinationEditorWorkbench.tsx` 里仍有明显的具体业务残留：

```text
加入写作
加入审查
加入验收
长篇模板
```

这些动作可以作为写作任务域的模板，但不应成为通用多 Agent 编辑器的默认交互。

通用编辑器应该提供：

```text
加入为 Agent 节点
加入为任务节点
加入为审查节点
加入为汇总节点
加入为人工门控
加入为输出节点
```

具体“写作 / 审查 / 验收”应来自任务域模板或节点工作姿态，不是全局默认按钮。

### 2.5 画布能力不足以支撑一般多 Agent 协作

当前 `CoordinationTopologyGraph.tsx` 使用 SVG 蛇形网格布局：

```text
节点数小于 6：普通布局
节点数大于等于 6：compact
节点数大于等于 10：dense
```

它适合运行态展示和轻量编辑，但不足以成为通用任务图编辑器。

缺口：

```text
拖拽定位
缩放和平移
自动布局
节点密度切换
分组 / 泳道
折叠复杂节点
迷你图
边路由
多选操作
复制节点 / 模板化插入
```

如果不解决画布能力，稍复杂一点的两节点或三节点协作都会变得难用。

### 2.6 节点检查器不是运行装配检查器

当前节点检查器能编辑：

```text
节点名称
绑定分任务
角色
Agent
节点契约
Runtime Lane
```

但真正可运行的节点还需要：

```text
Agent 绑定或选择策略
Soul / Projection 决议说明
工作姿态 WorkPosture
节点输入
节点输出
节点局部契约
上下文可见性
工具 / 权限摘要
失败策略
验收规则
是否允许人工门控
是否允许写入记忆
```

这些不是“更多字段”而是节点能不能运行、能不能协作、能不能被验证的核心。

### 2.7 A2A 只是预览，不是边的完整设计对象

当前通信检查器能展示 A2A Message 预览，但边本身还没有成为：

```text
业务交接语义
A2A message type
payload contract
ack policy
timeout policy
failure policy
context filter
artifact refs policy
```

边应是多 Agent 协作的一等对象。

### 2.8 契约管理没有和图编辑形成闭环

现在契约面板分为：

```text
TaskContractPanel
ContractOverviewPanel
ContractLibraryPanel
```

但用户在画布上编辑节点/边时，契约没有作为图上的“局部契约”被自然编辑。

目标应该是：

```text
图级契约 GraphContract
节点契约 NodeContract
边契约 EdgeContract
输出契约 FinalContract
验收契约 AcceptanceContract
```

契约库是资产管理，契约编排是任务图编辑的一部分。

## 3. 重设计目标

### 3.1 统一任务对象

所有任务统一为：

```text
Task
  task_id
  domain_id
  title
  task_mode
  graph_id
  enabled

TaskGraph
  graph_id
  task_id
  graph_kind
  nodes
  edges
  graph_contract
  runtime_policy
  publish_state
```

其中：

```text
single_agent = 一节点图
multi_agent = 多节点图
```

前端可以有简化视图，但底层心智必须统一。

### 3.2 任务域成为治理目录，不承担装配细节

任务域页面只负责：

```text
任务域名称
任务域说明
任务族 task_family
入口策略
默认模板
任务清单
任务域启停
统计与健康状态
```

不在任务域页面编辑 workflow、projection、execution、memory 等运行细节。

任务域点击任务后，进入任务图工作台。

### 3.3 投影默认归属 Agent

任务系统不再把“投影分配”作为常规任务装配项。

新的规则：

```text
Agent.default_projection_id 是默认投影来源。
TaskGraph / Node 可以声明 projection_overlay。
Contract 可以约束投影使用。
RuntimeAssembly 最终决议本次可见投影。
```

前端任务系统只显示：

```text
此节点将使用哪个 Agent 投影
是否存在任务级 overlay
是否被契约约束
```

不再让用户在任务系统里常规“分配投影”。

### 3.4 多 Agent 图模板成为一等能力

当前系统如果不能轻松表达两节点顺序协作和并行汇总，就说明多 Agent 设计还不达标。

第一批内置图模板应包括：

```text
单 Agent 执行
主 Agent -> 子 Agent
规划 -> 执行
执行 -> 验证
探索 -> 整理
双节点并行 -> 汇总
起草 -> 审查 -> 修订
```

这些是图模板，不是具体 Agent 身份。

真实节点仍要绑定：

```text
agent_id
work_posture
node_contract
runtime_lane
context_policy
```

## 4. 新页面层级

### 4.1 顶层信息架构

建议任务系统顶层改为四个主区：

```text
任务域
  治理目录和任务清单。

任务图
  统一编辑单 Agent / 多 Agent 任务。

契约
  契约库、任务契约汇总、节点/边契约资产。

预检与发布
  Manifest、RuntimeAssembly、A2A、图校验、发布记录。
```

不要再将“单任务装配”和“协调任务”作为两个平级页面。

### 4.2 页面职责

#### 任务域

```text
左侧：任务域列表
中间：当前域任务清单
右侧：任务域属性 / 入口规则 / 默认模板 / 统计
```

关键动作：

```text
新建任务域
新建任务
从模板创建任务图
进入任务图
删除任务
启停任务
```

#### 任务图

```text
左侧：资源池
  Agent
  Agent 组
  任务模板
  节点模板
  契约模板

中间：图画布
  input / agent / merge / review / human_gate / output

右侧：检查器
  根据选中对象切换：
    图检查器
    节点检查器
    边检查器
    输出检查器

底部：状态栏
  保存状态
  图校验
  Manifest 问题
  RuntimeAssembly 问题
```

#### 契约

```text
契约库
  管理 ContractSpec。

任务契约
  查看当前任务图引用到的所有契约。

节点契约
  按节点汇总。

边契约
  按通信边汇总。
```

#### 预检与发布

```text
图校验
Contract Manifest
RuntimeAssembly
A2A envelope preview
Agent / Soul / Projection 决议
发布草稿 / 发布可运行 / 回滚
```

## 5. 任务图工作台详细设计

### 5.1 左侧资源池

资源池分组：

```text
任务内节点
  当前任务已有节点，支持快速定位。

Agent
  主 Agent
  系统管理 Agent
  子 Agent
  Agent 组成员

节点模板
  Agent 节点
  审查节点
  汇总节点
  人工门控
  输出节点

多 Agent 模板
  A -> B
  A || B -> Merge
  Draft -> Review -> Fix

契约模板
  节点执行契约
  边交接契约
  最终输出契约
```

注意：

```text
资源池只负责添加对象。
对象属性必须到右侧检查器编辑。
```

### 5.2 中央画布

画布第一阶段可以继续用现有 SVG，但目标应抽象出 `TaskGraphCanvas`，后续替换为 React Flow 或等价画布库。

第一阶段必须做到：

```text
节点尺寸缩小
节点信息密度可切换
选中节点/边高亮
图上添加后继节点
图上连线
删除前确认
保存前不直接写后端
图校验问题定位到节点/边
```

第二阶段升级：

```text
拖拽节点
缩放 / 平移
自动布局
mini map
fit view
多选
节点折叠
泳道 / 分组
```

### 5.3 节点类型

统一节点类型：

```text
input
agent
merge
review
human_gate
tool
output
```

第一版最少实现：

```text
agent
merge
human_gate
output
```

节点字段：

```text
node_id
node_type
title
agent_id
agent_selection_policy
work_posture
node_contract_id
input_bindings
output_bindings
runtime_lane
context_visibility_policy
projection_overlay_id
failure_policy
human_gate_policy
metadata
```

### 5.4 边类型

统一边类型：

```text
handoff
review_feedback
parallel_join
conditional
retry
finalize
```

边字段：

```text
edge_id
source_node_id
target_node_id
edge_type
a2a_message_type
payload_contract_id
context_filter_policy
artifact_ref_policy
ack_policy
timeout_policy
failure_policy
metadata
```

边必须承载 A2A，但 A2A 是底层通信协议，不是用户主要编辑负担。

前端展示：

```text
业务语义：交接 / 审查反馈 / 汇合 / 重试
协议承载：A2A message/send / message/stream / task/status
载荷契约：某个 EdgeContract
```

## 7. 任务域联动重构

### 7.1 Current Workspace State

前端应明确维护：

```text
currentDomainId
currentTaskId
currentGraphId
currentSelection:
  type: graph | node | edge | contract | none
  id: string
```

任务域切换时只做：

```text
切换 currentDomainId
选择该域第一个任务
加载该任务图
清空节点/边选择
```

不要让任务域切换直接改多个页面的私有状态。

### 7.2 任务域页面与任务图页面联动

任务域页面展示每个任务的图摘要：

```text
任务名
图类型
节点数 / 边数
绑定 Agent 数
契约问题数
发布状态
最近保存时间
```

点击任务：

```text
进入任务图
打开该任务图
默认选中图检查器
```

### 7.3 契约页面联动

契约页面默认跟随当前任务：

```text
当前任务图引用的契约
未绑定契约的节点/边
重复或冲突契约
全局契约库入口
```

契约库作为资产库，不能替代任务图内的契约编排。

## 8. 与 Agent / Soul / Projection 的联动

### 8.1 Agent 选择

节点绑定 Agent 时，前端应显示：

```text
Agent 名称
Agent 类别
是否内置
默认 Soul
默认 Projection
RuntimeProfile 摘要
允许任务模式
允许 runtime lane
```

不能只让用户输入 `agent_id`。

### 8.2 Projection 决议

任务图节点不常规分配投影。

节点检查器显示：

```text
Agent 默认投影
契约约束
节点 overlay
最终投影预览
```

只有高级区允许设置 `projection_overlay_id`。

### 8.3 WorkPosture

工作姿态不是 Agent 身份，而是节点运行姿态：

```text
explore
plan
execute
verify
review
summarize
```

它用于影响：

```text
节点 prompt overlay
运行权限默认值
输出契约推荐
上下文可见性推荐
```

## 9. 实施阶段

### 阶段 1：任务系统信息架构重排

目标：

```text
不大改数据模型，先重排前端层级。
```

工作：

1. 将顶层切换改为：
   ```text
   任务域 / 任务图 / 契约 / 预检发布
   ```
2. 移除 `assembly` 和 `coordination` 作为顶层同级页面。
3. 将原单任务装配内容迁移到任务图的图检查器 / 节点检查器。
4. 将原协调编辑器迁移到任务图页面。
5. 任务域页面只保留治理、入口、任务清单和进入任务图动作。

### 阶段 2：通用 TaskGraph 工作台

目标：

```text
让单 Agent / 多 Agent 在同一工作台里编辑。
```

工作：

1. 新建 `TaskGraphWorkbench.tsx`。
2. 新建 `TaskGraphResourcePool.tsx`。
3. 新建 `TaskGraphInspector.tsx`。
4. 新建 `TaskGraphCanvas.tsx`，先包装现有 `CoordinationTopologyGraph`。
5. 清理“加入写作 / 加入审查 / 加入验收”等通用编辑器里的具体业务按钮。
6. 增加多 Agent 图模板入口。

### 阶段 3：节点/边检查器升级

目标：

```text
节点和边成为真正可运行对象。
```

工作：

1. 节点检查器增加 Agent 选择器，替代手写 `agent_id`。
2. 节点检查器显示 Agent 默认 Soul / Projection。
3. 节点检查器增加 WorkPosture。
4. 节点检查器增加上下文可见性摘要。
5. 边检查器增加 A2A 业务语义和 payload contract 选择。
6. 边检查器增加 ack / timeout / failure。

### 阶段 4：契约联动

目标：

```text
契约从“旁边预览”变成任务图的一部分。
```

工作：

1. 契约页默认显示当前任务图契约引用。
2. 节点/边检查器可直接绑定 ContractSpec。
3. 缺失契约时提供“从模板创建契约草稿”。
4. Contract Manifest 问题可点击定位到节点/边。

### 阶段 5：预检与发布

目标：

```text
保存、预检、发布形成闭环。
```

工作：

1. 任务图修改只进入草稿态，不直接发布。
2. 保存草稿后可运行图校验。
3. 预检页生成 Contract Manifest。
4. 预检页生成 RuntimeAssembly。
5. A2A envelope preview 跟随选中边。
6. 发布可运行前必须通过阻断级校验。

### 阶段 6：画布能力升级

目标：

```text
支持复杂图。
```

工作：

1. 评估并引入 React Flow 或等价画布库。
2. 支持拖拽、缩放、平移、fit view。
3. 支持自动布局。
4. 支持节点折叠和分组。
5. 支持迷你图。
6. 支持复杂图的密度模式。

## 10. 第一版验收标准

### 信息架构

1. 顶层只有：
   ```text
   任务域 / 任务图 / 契约 / 预检发布
   ```
2. 单任务装配不再作为顶层页面。
3. 协调任务不再作为顶层页面。
4. 任务域页面不编辑运行细节。

### 单 Agent

1. 新建任务默认生成一节点图。
2. 单 Agent 任务可在任务图页面查看和编辑。
3. 单 Agent 节点可绑定 Agent、契约、Runtime Lane。

### 多 Agent 基础模板

1. 可一键创建 `A -> B` 两节点图。
2. 可一键创建 `A || B -> Merge` 并行双节点图。
3. 两节点之间的边可配置 A2A 交接语义和 payload contract。

### 多 Agent

1. 可添加多个节点。
2. 可添加、反转、删除边。
3. 节点/边都有检查器。
4. 图校验能定位问题。

### Agent / Soul / Projection

1. 节点绑定 Agent 时展示默认 Soul / Projection。
2. 任务系统不再常规分配投影。
3. 节点 overlay 必须在高级区显示。

### 契约

1. 当前任务图可汇总 Graph / Node / Edge / Output 契约。
2. Contract Manifest 预检能定位到节点或边。

## 13. 风险与处理

### 风险 1：后端仍是 workflow / coordination 双模型

处理：

第一阶段前端用 adapter 把当前数据转成 `TaskGraphDraft`：

```text
SpecificTaskRecord + workflow/execution/memory -> single node graph
CoordinationTask + topology/protocol -> multi node graph
```

等后端契约和 runtime 统一后，再落真实 `TaskGraph` API。

### 风险 2：画布库替换影响大

处理：

先抽 `TaskGraphCanvas` 包装现有 SVG，稳定接口后再替换内部实现。

### 风险 3：用户配置入口变多

处理：

右侧检查器采用分区折叠：

```text
基础
契约
上下文
运行
高级
```

默认只展开基础和契约。

### 风险 4：任务域和任务图状态错位

处理：

集中维护 `TaskWorkspaceState`，禁止多个页面各自修补选择状态。

## 14. 结论

任务系统应从“任务域表单 + 单任务装配表单 + 协调拓扑编辑器”升级为“任务图优先”的工作台。

核心原则：

```text
任务域负责治理。
任务图负责设计。
契约负责边界。
Agent/Soul/Projection 负责执行主体。
RuntimeAssembly 负责最终装配。
A2A 负责节点通信。
```

这样才能同时支持：

```text
单 Agent 任务
多 Agent 协作
复杂多 Agent 协调
```

并且不会把项目里的任务系统、Agent 系统、灵魂系统、契约系统割裂开。

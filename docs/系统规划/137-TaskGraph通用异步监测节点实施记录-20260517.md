# 137-TaskGraph通用异步监测节点实施记录

日期：2026-05-17

## 1. 实施结论

已按 `136-TaskGraph通用异步监测节点与运行控制闭环实施计划-20260517.md` 完成第一版 TaskGraph 原生监测能力：

- 监测系统现在是 TaskGraph 通用节点能力，不依赖健康管家特殊链路。
- 监测节点类型为 `runtime_monitor`，推荐执行模式为 `background`。
- 监测评估读取现有 `task_graph.run_monitor` 快照，产出 `task_graph.monitor_decision` 决策包。
- 决策包会落到 `SupervisionRecord`，可按 task_run 查询。
- 监测节点和用户交互统一进入 `task_graph_run_interaction_panel`；人工审核只是该运行交互窗口中的一种状态，不再另开专用窗口。
- 旧 Codex hook 写作监督链路已断开并删除。

## 2. 修改明细

### 2.1 TaskGraph 模型

文件：

- `backend/tasks/task_graph_models.py`

变更：

- 新增节点类型 `runtime_monitor`。
- 增加监测节点预检：
  - `runtime_monitor` 必须使用 `execution_mode="background"`。
  - 缺少 `metadata.monitor_policy` 时给 warning。
  - 继续复用后台节点已有规则：必须配置 `background_policy.enabled`、`max_runtime_seconds`、`notification_policy`。

### 2.2 RuntimeSpec 与监控视图

文件：

- `backend/tasks/coordination_graph_models.py`
- `backend/tasks/coordination_graph_compiler.py`
- `backend/orchestration/runtime_loop/task_graph_run_monitor.py`

变更：

- `TaskGraphRuntimeNode` 新增 `monitor_policy`。
- TaskGraph 编译时从节点 metadata 中提取 `monitor_policy`。
- `task_graph.run_monitor` 的 topology node 现在会暴露：
  - `execution_mode`
  - `monitor_policy`
  - `background_policy`
  - `notification_policy`
  - `metadata`

目的：

- 前端和运行态都可以从统一监控视图看到监测节点配置。

### 2.3 监测评估器

文件：

- `backend/orchestration/runtime_loop/task_graph_monitoring.py`

新增能力：

- `TaskGraphMonitorDecision`
- `evaluate_task_graph_monitor_snapshot`
- `compact_monitor_snapshot`

当前可识别状态：

- `completed`
- `runtime_failed`
- `blocker_present`
- `manual_gate_waiting`
- `stale_runtime`
- `active_stage_observed`
- `healthy`

当前建议动作：

- `no_action`
- `notify`
- `request_user_decision`
- `resume`
- `restart`
- `pause`
- `escalate`

说明：旧记录里的 `request_human_review` 只作为兼容动作读取，新监测节点默认产出 `request_user_decision`，并由统一运行交互窗口展示为人工确认状态。

交互请求：

- 唯一新字段：`run_interaction_request`
- 旧记录兜底字段：`human_review_request`，新链路不再产出
- 统一窗口：`task_graph_run_interaction_panel`
- 控制动作：`continue_current_stage`、`stop_task_run`、`start_new_run`、`acknowledge`，必要时兼容 LangGraph `resume`

边界：

- 评估器只产出判断，不直接修改业务节点状态。

### 2.4 TaskRunLoop 接入

文件：

- `backend/orchestration/runtime_loop/task_run_loop.py`

新增方法：

- `evaluate_task_graph_monitor(task_run_id, monitor_node_id="", monitor_policy=None)`
- `list_task_graph_monitor_decisions(task_run_id)`
- `_resolve_task_graph_monitor_policy(...)`

同时修正：

- dispatch plan 编译后台节点时，现在会从节点 metadata 读取 `background_policy`、`notification_policy`、`resource_lifecycle_policy`。
- 这避免 RuntimeSpec 把策略放在 metadata 后，调度层读不到后台节点策略。

### 2.5 API 接入

文件：

- `backend/api/orchestration.py`

新增：

- `POST /orchestration/runtime-loop/task-runs/{task_run_id}/task-graph-monitor/evaluate`
- `GET /orchestration/runtime-loop/task-runs/{task_run_id}/monitor-decisions`

说明：

- API 不依赖写作任务。
- API 不依赖健康系统。
- 调用 evaluate 后会创建 `SupervisionRecord`。

### 2.6 默认契约

文件：

- `backend/tasks/contract_registry.py`

新增：

- `contract.taskgraph.monitor.decision`

用途：

- 作为后台监测节点的标准输出契约，描述监测快照、决策 ID、建议动作、严重级别、摘要和建议控制入口。

### 2.7 旧残留清理

修改：

- `.codex/hooks.json` 清空写作监督 hook。

删除：

- `.codex/hooks/writing_project_monitor.py`
- `.codex/hooks/writing_supervision_watchdog.py`
- `.codex/hooks/writing_supervision_stop.py`
- `scripts/supervise_writing_campaign.py`
- `scripts/watch_writing_supervision.py`
- `scripts/start_writing_watchdog.ps1`
- `scripts/start_writing_supervisor.ps1`
- `scripts/stop_writing_supervisor.ps1`

保留：

- `scripts/configure_writing_simple_novel.py`
- `scripts/start_writing_project_run.ps1`
- `scripts/start_writing_stack.ps1`

原因：

- 保留项仍属于任务配置、任务启动或项目栈启动工具，不是监督主链路。

### 2.8 统一运行交互浮窗

文件：

- `frontend/src/components/workspace/views/task-system/TaskGraphRunInteractionDock.tsx`
- `frontend/src/components/workspace/views/task-system/TaskGraphPublishRunPage.tsx`
- `frontend/src/app/globals.css`

变更：

- 新增 TaskGraph 运行交互浮窗，视觉与拖拽体验复用健康系统悬浮窗模式。
- 该浮窗只读取 `TaskGraphMonitorDecision.run_interaction_request`，旧 `human_review_request` 只作为历史记录兜底。
- 当监测节点产出非 `no_action` 决策时，浮窗自动弹出。
- 人工审核、异常提醒、续跑、重试、暂停都在同一浮窗中处理，不再创建独立人工审核窗口。

写作图配置同步：

- `runtime_monitor` 节点的 `allowed_actions` 改为 `request_user_decision`。
- `human_review_handoff` 独立节点已从写作图配置中移除。
- `contract.taskgraph.human_review.request` 不再作为写作协议载荷契约。
- `contract.taskgraph.run_interaction.request` 成为唯一运行交互请求契约。

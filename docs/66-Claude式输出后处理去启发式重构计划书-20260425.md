# 66-Claude式输出后处理去启发式重构计划书-20260425

> 目的：针对当前输出链中 `output_classifier.py` 与 `runtime.py` 仍以关键词/句式启发式判定“这段话是不是最终答案”的问题，基于本项目既有设计原则，并参考本地 `Claude Code` 源码的输出收口方式，规划一次“后处理去启发式、类型化收口优先”的专项重构。

---

## 1. 问题定义

### 1.1 当前失败模式

当前系统已经有：

1. `AssistantOutputBoundary`
2. `OutputCandidate / OutputDecision`
3. `canonical_answer / selected_channel / selected_source`

但最终的“是否需要二次 finalization”“这段文本是不是 progress/procedural answer”仍大量依赖：

1. 关键词
2. 句首模式
3. 若干正则推断

这导致出现本轮回归暴露的问题：

- 已经被边界层裁干净的可见答案  
  `根据检索到的知识库内容，我先用业务语言给出结论……`
- 仍被后置逻辑误判为 procedural answer
- 随后错误触发 `rag_answer_finalization`

也就是说，当前错误不是“输出没清洗干净”，而是：

> 后处理层仍在用自然语言启发式重新猜“文本功能”，导致已经收口的 canonical answer 被再次改道。

### 1.2 真正损坏的系统性质

被破坏的不是某条 RAG 分支，而是输出主链的稳定性约束：

1. `canonical answer` 不是最终唯一真相
2. 结构化边界之后，仍允许文本启发式越权改写路由
3. finalization 的触发条件不是显式状态，而是弱语义猜测

因此这首先是：

- 输出主链设计问题
- 类型边界问题
- restore/repair 与 decide/finalize 边界混淆问题

不是 prompt 问题，也不是单纯的正则 bug。

### 1.3 正确终态

正确终态应是：

1. 输出链先把 assistant 输出分成显式类型
2. finalization 只基于显式不稳定信号触发
3. canonical answer 一旦进入“稳定答案态”，就不再被关键词分类器二次推翻
4. session history / memory / done.content 统一消费同一份 canonical truth

## 2. 当前系统真实结构

本节只描述当前代码真实职责，不做理想化推测。

### 2.1 当前主链

相关文件：

- `backend/query/output_boundary.py`
- `backend/query/output_classifier.py`
- `backend/query/runtime.py`
- `backend/runtime/session_store.py`

当前主链为：

1. `runtime` 流式收集 `messages / updates / tool events`
2. `AssistantOutputBoundary` 生成 `AssistantOutputResponse`
3. `build_output_decision()` 选出 `canonical_answer`
4. `runtime` 再执行 route-specific 后置 gate/finalization
5. 最后将结果写入 `done.content` 与 session history

### 2.2 当前真实问题点

#### A. `AssistantOutputBoundary` 只完成了“可见文本收口”，没有完成“稳定性分型”

在 `output_boundary.py` 中，系统已经区分：

1. `raw_text`
2. `visible_text`
3. `tool_calls`
4. `tool_receipts`
5. `canonical_answer`

但 `OutputCandidate.channel` 仍是偏显示层语义：

1. `answer_candidate`
2. `progress_text`
3. `tool_visible_summary`
4. `tool_claim_without_receipt`

它没有表达：

1. 该答案是否已稳定
2. 该答案是否允许直接持久化
3. 该答案是否仍处于“需要 route-specific 重写”的中间态

#### B. `output_classifier.py` 仍以启发式语言规则承担职责裁决

`looks_like_progress_text()` / `looks_like_procedural_promise_text()` 当前使用：

1. 句首模板
2. 动作词
3. 承诺词
4. 搜索调用痕迹

这些规则的本质是：

> 通过文本表面特征去猜一句话的系统角色。

这违反了前述 docs 一直要求的“结构修复优先于 prompt/词表补丁”。

#### C. `runtime.py` 的 finalization 触发仍依赖“像不像 procedural answer”

`_rag_output_needs_finalization()` 与 `_looks_like_rag_procedural_answer()` 当前仍在问：

- 这段 canonical answer 看起来像不像过程文本？

而不是问：

- 当前 response 是否来自稳定类型？
- 当前 answer 是否缺失显式证据归并？
- 当前 route 是否仍处于未完成答案态？

这使得 post-boundary canonical answer 仍可能被弱语义推翻。

#### D. session 持久化层只知道字符串，不知道答案状态

`SessionManager.append_messages()` 目前只存：

1. `role`
2. `content`
3. `tool_calls`

因此一旦 runtime 上游选错 canonical answer，session/memory 无法再纠偏。

## 3. 本轮必须遵守的设计原则

本计划书受以下文档正式约束：

1. `docs/35-Claude式输出边界去泄露重构计划书.md`
2. `docs/37-输出主链与Canonical Answer收口计划书.md`
3. `docs/36-Claude式统一续接主链修复计划书.md`
4. `docs/64-子任务分解去启发式与显式编排改造计划书-20260424.md`
5. `docs/设计原则/12-Agent-系统.md`
6. `docs/设计原则/25-架构模式总结.md`

抽象成可执行约束如下。

### 3.1 结构修复优先于启发式修补

禁止继续用更多关键词、正则、话术名单来弥补输出层分型不足。

### 3.2 类型先于显示

最终答案、进度文本、工具回执、调试痕迹必须先有内部类型，再决定如何显示。

### 3.3 主线程只消费 canonical truth

`done.content`、session assistant content、memory projection、follow-up truth 必须只消费 canonical answer，不消费 raw stream 与过程 chatter。

### 3.4 默认隔离，显式共享

route-specific finalizer 不能无条件重写所有 `segment.visible_text`。只有显式标记为 unstable 的答案，才允许进入 finalizer。

### 3.5 restore / repair / finalize 必须分离

sanitize、salvage、gate、finalize 各自职责必须分开，不能由同一套文本启发式横跨所有阶段。

## 5. 目标设计

### 5.1 从“文本分类器”改成“输出状态机”

建议把当前输出后处理重构为四层：

1. `Raw Capture Layer`
2. `Visible Extraction Layer`
3. `Canonicalization Layer`
4. `Route Finalization Layer`

其中：

#### A. Raw Capture Layer

职责：

1. 接收 stream text
2. 接收 ai update
3. 接收 tool call / result
4. 保留 debug flags

禁止：

1. 判定最终答案
2. 用自然语言推断角色

#### B. Visible Extraction Layer

职责：

1. 从 raw stream 提取用户可见文本
2. 切除内部协议
3. 形成 `visible_text`

禁止：

1. 判定该文本是否可持久化
2. 判定 route 是否应 finalization

#### C. Canonicalization Layer

新增显式输出状态，建议：

```python
CanonicalState = Literal[
    "stable_answer",
    "unstable_answer",
    "progress_only",
    "tool_summary",
    "missing_answer",
]
```

并在 `OutputDecision` 上新增：

1. `canonical_state`
2. `persist_policy`
3. `finalization_policy`

其中：

- `stable_answer`
  含义：可直接成为 `done.content` / session answer
- `unstable_answer`
  含义：已有文本，但仍需 route-specific finalization
- `progress_only`
  含义：只有过程文本，不能作为最终答案
- `tool_summary`
  含义：来自可信工具摘要，可直接用作答案或进一步路由
- `missing_answer`
  含义：当前尚无可展示答案

#### D. Route Finalization Layer

route finalizer 只允许读取：

1. `canonical_state`
2. `tool_receipts`
3. `retrieval/tool evidence pack`
4. `selected_source`

禁止直接读取：

1. 原始 assistant prose 并再次做关键词推断

也就是说：

> `rag/pdf/memory` finalizer 的问题应改成“当前状态是否允许 finalization”，而不是“这句文本看起来像不像 procedural”。

### 5.2 新的 `OutputDecision` 目标结构

建议扩展为：

```python
@dataclass(slots=True)
class OutputDecision:
    canonical_answer: str
    selected_channel: OutputChannel
    selected_source: str
    canonical_state: str
    persist_policy: str
    finalization_policy: str
    rejected_candidates: list[OutputCandidate] = field(default_factory=list)
    leak_flags: list[str] = field(default_factory=list)
    fallback_reason: str = ""
```

推荐策略：

1. `persist_policy`
   - `persist_canonical`
   - `persist_debug_only`
   - `do_not_persist`
2. `finalization_policy`
   - `none`
   - `route_optional`
   - `route_required`

### 5.3 `classify_output_candidate()` 的新职责边界

分类器只做：

1. 来源分型
2. 结构特征分型
3. 工具回执一致性分型

不再做：

1. 句式语义裁决
2. “像不像进度文本”的宽泛自然语言判断

保留的少量启发式，只允许用于：

1. 明确协议泄露识别
2. 明确 receipt 缺失识别
3. 明确 raw JSON / raw tool dump 识别

### 5.4 `runtime` 的新收口规则

`runtime` 应改为：

1. 先信任 `OutputDecision.canonical_state`
2. 只有 `finalization_policy != none` 时，才允许 route-specific finalizer 介入
3. route finalizer 成功后，写回新的 stable decision
4. session/history 只持久化 stable canonical answer

### 5.5 session 持久化的兼容升级

本轮不强制重构 session schema，但建议最小升级：

1. assistant message 写入时附带可选 metadata
2. 至少保留：
   - `answer_channel`
   - `answer_source`
   - `canonical_state`

这样后续 memory/context 层可以做更可靠的过滤，而不必只看 `content` 字符串。

## 8. 验证矩阵

### 8.1 必过正向用例

1. `根据检索到的知识库内容，我先用业务语言给出结论……`
   - 应直接落为 `segment.visible_text`
   - 不触发 `rag_answer_finalization`

2. `我来先检索本地知识库，再整理答案`
   - 应识别为 `progress_only` 或 `unstable_answer`
   - 触发 finalization 或 fallback

3. JSON / invoke / tool arg 协议片段
   - 不得直接成为 canonical answer

### 8.2 必过反向用例

1. 不能把真实 procedural answer 误放行为 stable answer
2. 不能把 tool receipt 缺失时的“我已经查到”误存为 stable answer
3. 不能因新增状态机而破坏 PDF / Memory 既有 finalizer 成功用例

### 8.3 长场景观察指标

1. `answer_source=rag_answer_finalization` 的误触发次数下降
2. 平均耗时下降
3. session history 中 progress/protocol 污染下降
4. 后续 memory/context 误吸收下降

## 10. 最终推荐路线

推荐采用：

> 现有 `AssistantOutputBoundary`  
> + 显式 `canonical_state / finalization_policy`  
> + route finalizer 改为状态驱动  
> + session/history 只消费 stable canonical truth

不推荐采用：

1. 继续堆叠关键词启发式
2. 用更多正则补丁逼近语义理解
3. 把所有问题都推给二次模型 finalization

这条路线最符合：

1. 你现有 docs 的 owner-first / summary-first / default-isolation 原则
2. Claude Code 的 type-first / result-first / notification-first 输出收口方式
3. 当前代码可渐进迁移的现实条件

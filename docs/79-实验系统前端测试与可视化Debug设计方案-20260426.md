# 实验系统前端测试与可视化 Debug 设计方案

日期：2026-04-26

## 1. 目标定义

我们要优先做的“实验系统”，不是单独写一个展示测试报告的页面，而是把现有后端测试体系接入前端，让前端可以完成三件事：

1. 选择测试 profile 并发起测试。
2. 实时看到测试进度、日志、失败项和产物路径。
3. 把失败项和系统框架图对接，后续可以在图上高亮出问题的节点或链路。

正确的最终状态是：用户在前端进入“实验系统”，选择 `smoke / stable / full / deep / long` 等测试方案，点击运行后，后端启动受控测试任务；测试过程持续写入状态；前端轮询或流式读取状态；测试结束后自动读取 `run_result.json`、`issues.json`、`trace.jsonl`、`report.md`，并把 issue 映射到系统框架图中的节点和连线。

## 2. 当前代码依据

现有测试体系已经有统一入口：

- `backend/harness/run.py`
  统一命令入口，支持 `smoke / stable / full / deep / benchmark / regression / long`。

- `backend/tests/run_regression_gate.py`
  后端 curated regression gate，适合快速验证核心能力。

- `backend/tests/system_eval/runner.py`
  负责 system eval，统一生成 `run_result.json`、`issues.json`、`trace.jsonl`、`report.md`。

- `backend/tests/system_eval/long_runner.py`
  负责长场景测试，目前长场景产物已经落在 `output/test_runs/<run_id>`。

- `backend/harness/contracts.py`
  已经定义 `RunContext`、`ScenarioResult`、`IssueEntry`、`TraceSpan`、`RunResult`。

- `backend/harness/persistence.py`
  已经负责持久化测试产物。

- `backend/api/tasks.py`
  当前只暴露运行时任务列表，还没有实验运行 API。

- `frontend/src/components/workspace/views/ExperimentsView.tsx`
  当前页面仍是“编排系统”静态说明，适合改造成真正的“实验系统”入口。

结论：不用重建测试系统。我们要补的是“实验运行控制层”和“前端实验控制台”。

## 3. 推荐设计方向

推荐把实验系统拆成四层：

1. **实验目录层**
   负责声明有哪些实验 profile、能不能从前端运行、预计耗时、风险等级、对应命令。

2. **实验运行层**
   后端用受控 subprocess 启动 `python -m harness.run --profile ...`，记录运行状态、PID、开始时间、结束时间、日志路径、产物目录。

3. **实验产物层**
   读取 `output/test_runs/<run_id>` 下的标准产物，统一返回给前端。

4. **系统图映射层**
   把 issue、trace、scenario category 映射到系统框架图的 `node_id` 和 `edge_id`，用于可视化 debug。

这条路线比“前端直接拼命令”安全，也比“重新写一套测试 runner”稳定。

## 4. 后端目标结构

建议新增：

- `backend/experiments/catalog.py`
  定义前端可见实验 profile。

- `backend/experiments/runner.py`
  管理实验任务生命周期，负责启动、状态查询、取消、读取日志。

- `backend/experiments/artifacts.py`
  读取 `run_result.json`、`issues.json`、`trace.jsonl`、`report.md`，并提供摘要。

- `backend/experiments/graph_mapping.py`
  把测试结果映射到系统框架图节点和链路。

- `backend/api/experiments.py`
  暴露前端 API。

建议 API：

```text
GET    /api/experiments/profiles
POST   /api/experiments/runs
GET    /api/experiments/runs
GET    /api/experiments/runs/{run_id}
GET    /api/experiments/runs/{run_id}/log
GET    /api/experiments/runs/{run_id}/artifacts
POST   /api/experiments/runs/{run_id}/cancel
```

第一阶段可以用轮询，不急着上 WebSocket。因为测试运行是长任务，状态更新不需要 token 级实时。

## 5. 前端目标结构

把 `ExperimentsView.tsx` 从静态编排说明改成实验控制台：

1. **实验 profile 区**
   展示 `smoke / stable / full / long`，每个卡片显示用途、预计耗时、是否会调用真实模型、风险等级。

2. **运行控制区**
   选择 profile，点击运行；运行中显示状态、耗时、当前日志尾部；允许取消。

3. **结果总览区**
   显示总数、通过数、失败数、首个失败、profile、trace 是否启用。

4. **失败问题区**
   读取 `issues.json`，展示 severity、category、summary、artifact path、trace path。

5. **系统图联动区**
   每个 issue 显示它命中的系统节点或链路，比如：
   - `long_scenario` -> `对话执行核心`、`任务理解与规划`、`记忆门面`
   - `regression/memory` -> `记忆门面`、`项目持久层`
   - `regression/retrieval` -> `检索服务`、`项目持久层`
   - `frontend` -> `接口路由层`、`模型流式输出`

后续可以从 issue 点击跳到系统框架页，并自动选中对应节点或连线。

## 6. 实验运行数据模型

建议前端和后端统一使用这个轻量模型：

```json
{
  "run_id": "20260426-071500-smoke",
  "profile": "smoke",
  "status": "queued | running | passed | failed | cancelled | error",
  "pid": 12345,
  "started_at": "2026-04-26T07:15:00",
  "ended_at": "",
  "duration_ms": 0,
  "output_dir": "output/test_runs/20260426-071500-smoke",
  "command": "python -m harness.run --profile smoke --output-dir ...",
  "summary": {
    "total": 0,
    "passed": 0,
    "failed": 0,
    "first_failure": ""
  },
  "artifacts": {
    "run_result": "",
    "issues": "",
    "trace": "",
    "report": "",
    "log": ""
  }
}
```

状态来源优先级：

1. 运行中：读取 runner registry 的内存状态和日志尾部。
2. 已结束：读取 `run_result.json` 和 `issues.json`。
3. 异常退出：读取 subprocess returncode 和 stderr tail。

## 7. 系统图映射协议

为了后续可视化 debug，建议每个 issue 补一个派生字段：

```json
{
  "issue_id": "LONG-001",
  "graph_refs": {
    "nodes": ["query-core", "planner", "memory"],
    "edges": ["query-planner", "query-memory"],
    "reason": "long_scenario failure with followup/runtime state drift"
  }
}
```

第一阶段不需要修改测试 runner 输出，可以在 API 层读取 issue 后临时派生 `graph_refs`。等稳定后，再把它沉淀进 `harness.contracts.IssueEntry`。

建议初始映射：

- `long_scenario`
  命中：`query-core`、`planner`、`memory`、`session-store`

- `regression/memory`
  命中：`memory`、`storage`、`retrieval`

- `regression/retrieval`
  命中：`retrieval`、`storage`、`evidence`

- `regression/query`
  命中：`query-core`、`planner`、`model`

- `regression/tool`
  命中：`tooling`、`query-core`

- `regression/pdf` 或 `regression/structured`
  命中：`evidence`、`retrieval`、`storage`

- `frontend`
  命中：`api-router`、`model`、`query-core`

## 8. 固定执行流

前端运行实验的流程应该固定为：

```text
前端选择 profile
-> POST /api/experiments/runs
-> 后端校验 profile allowlist
-> 创建 run_id 和 output_dir
-> 后台启动 harness subprocess
-> 前端轮询 GET /api/experiments/runs/{run_id}
-> 后端返回状态、日志尾部、产物摘要
-> 测试结束后读取 artifacts
-> 前端展示报告、issues、trace
-> issue 可跳转系统框架图
```

这条链路里，前端不能直接传任意命令，只能传 allowlist profile。

## 9. 分阶段实施计划

### Phase 1：只读实验面板

目标：前端能查看最近测试运行，不发起新测试。

后端：

- 新增 `backend/experiments/artifacts.py`
- 新增 `GET /api/experiments/runs`
- 新增 `GET /api/experiments/runs/{run_id}`
- 新增 `GET /api/experiments/runs/{run_id}/artifacts`

前端：

- 改造 `ExperimentsView.tsx`
- 展示最近 run 列表
- 展示 `run_result.json` 摘要
- 展示 `issues.json`

完成标准：

- 能在前端打开最近的 `20260426-065921-long`
- 能看到通过数、失败数、首个失败、artifact path

### Phase 2：前端发起受控测试

目标：前端可以运行 `smoke / stable / long`。

后端：

- 新增 `backend/experiments/catalog.py`
- 新增 `backend/experiments/runner.py`
- 新增 `POST /api/experiments/runs`
- 新增 `GET /api/experiments/runs/{run_id}/log`
- 新增取消接口

前端：

- 增加 profile 卡片
- 增加运行按钮
- 增加运行中状态和日志尾部

完成标准：

- 前端点击 `smoke` 后能启动测试
- 测试结束后自动显示 `run_result.json`
- 失败时显示 `issues.json`

### Phase 3：系统图联动 debug

目标：issue 能映射到系统框架图。

后端：

- 新增 `backend/experiments/graph_mapping.py`
- artifact API 返回 `graph_refs`

前端：

- issue 卡片显示命中的系统节点和链路
- 点击 issue 跳到系统框架页
- 系统框架页支持通过 store 选中 node/edge

完成标准：

- `long_scenario` 失败能高亮 `对话执行核心 / 任务理解与规划 / 记忆门面`
- 点击 issue 能看到对应 trace/artifact

### Phase 4：可视化 Debug 控制台

目标：不再只是看报告，而是在图上看运行链路。

后端：

- 解析 `trace.jsonl`
- 读取本地 trace JSON
- 聚合每个 graph node/edge 的失败次数、耗时、最近错误

前端：

- 系统框架页显示运行叠层
- 节点显示 `passed / failed / unknown`
- 连线点击显示相关 trace events

完成标准：

- 从一次测试运行可以生成系统图 overlay
- 点击连线能看到该链路最近的输入、输出、耗时、失败摘要

## 11. 风险控制

必须避免：

- 前端传任意 shell command。
- 同时启动多个 heavy/long 测试把本地模型和索引打爆。
- 测试运行状态只存在内存里，刷新页面后丢失。
- long 测试默认暴露在一个误点击按钮上。
- UI 只展示日志，不读取结构化 `run_result.json`。

建议规则：

- profile 必须 allowlist。
- `long / deep / full` 默认需要二次确认。
- 第一版只允许同时一个实验运行。
- runner 状态必须写入 `output/test_runs/.runs.json` 或每个 run 目录里的 `run_state.json`。
- 取消只杀当前 run 的进程，不影响应用服务进程。
- 所有 API 返回 repo 相对路径，同时保留绝对路径用于本地调试。

## 12. 验证矩阵

最小验证：

- `GET /api/experiments/profiles` 返回 profile 列表。
- `GET /api/experiments/runs` 能读到已有 run。
- `POST /api/experiments/runs smoke` 能启动并完成。
- `GET /api/experiments/runs/{run_id}/artifacts` 能返回 run_result、issues、report。
- 取消运行后状态为 `cancelled`。
- 前端刷新页面后仍能看到最近运行。

回归验证：

- `python backend/tests/run_regression_gate.py --profile core`
- `python -m harness.run --profile smoke`
- `npm run build`

长场景验证：

- 前端只负责发起和观察，不阻塞页面。
- `long` 运行结束后，issue 能显示 artifact turn 文件和 trace 文件。

## 13. 推荐优先级

我建议先做 Phase 1 和 Phase 2。

理由很简单：只要前端能看最近测试并发起 `smoke`，实验系统就已经活起来了。系统图联动 debug 很重要，但它依赖稳定的 run API 和 artifact API；先把“实验运行控制面”打牢，后面的可视化 debug 才不会变成漂亮但接不上数据的空壳。

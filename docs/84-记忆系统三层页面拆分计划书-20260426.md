# 记忆系统三层页面拆分计划书

日期：2026-04-26

## 1. 拆分目标

当前记忆系统已经能看对话、看状态记忆、查长期记忆、做召回模拟，也能看单轮记忆链路。问题是这些能力都堆在同一个页面里，导致页面越来越像“总控台”，但用户真正 debug 时往往只想做一件事：

- 看某段对话为什么被记住或没有被记住。
- 看当前状态记忆到底是什么，是否发生漂移。
- 查长期记忆是否存在、是否可注入、是否污染。

所以记忆系统应该拆成三个页面：

```text
记忆系统
-> 对话记忆
-> 状态记忆
-> 长期记忆
```

这三个页面共享同一套后端记忆 API，但前端职责分开。这样不会减少功能，反而能让每一层更清楚。

## 2. 总体导航结构

记忆系统进入后先显示一个轻量总览页或顶部切换栏，不再把所有内容直接摊开。

推荐结构：

```text
记忆系统
  对话记忆：按会话和轮次看记忆使用现场
  状态记忆：看当前 session memory、上下文槽位、状态漂移风险
  长期记忆：查 durable memory、召回模拟、治理候选
```

页面切换可以做成记忆系统内部的三段式导航，不需要放到右侧全局工作台里。右侧工作台仍然只保留一个“记忆系统”入口。

## 3. 对话记忆

### 3.1 页面定位

对话记忆负责回答：

```text
这一轮对话发生了什么？
这一轮有没有触发记忆？
触发的是状态记忆还是长期记忆？
最后有没有进入 prompt？
```

这是最适合做可视化 debug 的页面。

### 3.2 保留内容

从当前记忆页迁移这些模块：

- 当前会话历史读取。
- 对话现场展开。
- 单轮记忆链路。
- 工具调用展开。
- “查看这一轮记忆链路”按钮。
- 后续接入真实 memory trace 后的“真实链路 / 模拟链路”对照。

### 3.3 页面布局

推荐布局：

```text
顶部：当前会话、消息数量、重载按钮、搜索框

左侧：对话时间线
  Turn 1 用户输入
  Turn 2 助手回应
  Turn 3 用户输入

右侧：选中轮次的记忆链路
  输入
  状态记忆
  长期召回
  Prompt 注入

底部：选中轮次的工具调用 / 召回记录 / trace 对照
```

### 3.4 第一版功能

- 点击任意轮次后，右侧自动显示单轮链路。
- 用户输入轮次优先支持召回模拟。
- 助手回应轮次可以展示工具调用、最终回答和上下文使用结果。
- 如果当前是测试系统跳转过来的 turn，则显示真实 memory trace。
- 如果没有真实 trace，则显示模拟链路，并明确标注“模拟”。

## 4. 状态记忆

### 4.1 页面定位

状态记忆负责回答：

```text
当前会话短期记住了什么？
当前目标、任务状态、流程状态有没有漂移？
哪些上下文会被模型看到？
哪些只是 debug 可见？
```

它不关心长期记忆文件治理，而是关心当前 session 的运行状态。

### 4.2 保留内容

从当前记忆页迁移这些模块：

- 状态记忆现场。
- 模型可见上下文。
- context slots。
- flow state。
- task state。
- risk flags。
- context management 和 token pressure。

### 4.3 页面布局

推荐布局：

```text
顶部：当前 session 状态总览
  是否存在状态记忆
  上下文压力
  压缩策略
  当前目标

中部：状态结构图
  active_goal
  flow_state
  task_state
  context_slots
  risk

右侧或底部：模型可见片段 / debug 片段对照
```

### 4.4 第一版功能

- 显示当前状态记忆是否存在。
- 显示模型可见上下文和 debug 上下文的差异。
- 显示当前目标和流程状态。
- 把上下文压力用颜色标出来。
- 如果状态记忆为空，明确提示是“没有形成状态”而不是“读取失败”。

## 5. 长期记忆

### 5.1 页面定位

长期记忆负责回答：

```text
长期记忆里有什么？
哪些记忆允许注入？
一次问题会召回哪些长期记忆？
有没有脏记忆、过期记忆、冲突记忆？
```

它是后续治理功能的主页面。

### 5.2 保留内容

从当前记忆页迁移这些模块：

- 长期记忆记录列表。
- 长期记忆搜索。
- 召回模拟。
- 选中记忆详情。
- 打开记忆文件。
- 后续的候选写入队列、停用、归档、合并。

### 5.3 页面布局

推荐布局：

```text
顶部：长期记忆统计
  total
  active
  injectable
  by_type
  by_class

中部：查询与召回模拟
  输入问题
  召回理由
  选中记忆
  注入摘要

底部：长期记忆卡片列表
  标题
  类型
  状态
  是否允许注入
  摘要
```

### 5.4 第一版功能

- 查询长期记忆。
- 模拟召回。
- 查看召回理由和注入摘要。
- 打开单条记忆文件。
- 标记 active、inactive、injectable、not injectable。

## 6. 数据边界

三层页面可以复用当前 API，但前端状态需要整理。

### 6.1 共用数据

- `GET /api/memory/overview`
  提供状态记忆和长期记忆总览。

- `POST /api/memory/recall-preview`
  提供召回模拟。

- `GET /api/sessions/{session_id}/history`
  提供对话历史。

- `GET /api/experiments/runs/{run_id}/turns/{turn_id}/memory-trace`
  后续提供测试系统真实链路。

### 6.2 页面专属数据

对话记忆主要依赖：

- session history
- recall preview
- memory trace

状态记忆主要依赖：

- session memory inspect
- context management
- token stats

长期记忆主要依赖：

- durable memory headers
- recall preview
- durable note detail

## 7. 前端组件拆分

当前 `MemoryView.tsx` 应该拆成这些组件：

```text
MemoryView.tsx
  MemoryTabs
  ConversationMemoryLayer
  StateMemoryLayer
  DurableMemoryLayer

conversation/
  ConversationTimeline
  ConversationTurnCard
  TurnMemoryChain
  ToolCallDetails

state/
  SessionMemoryOverview
  ContextSlotMap
  ModelVisibleContext
  StateRiskPanel

durable/
  DurableMemorySearch
  DurableRecallPreview
  DurableMemoryCardGrid
```

第一步可以先不拆文件，只在 `MemoryView.tsx` 内拆成内部组件；等结构稳定后再移动到子目录。这样改动风险更小。

## 8. 视觉方向

三层页面要有统一的记忆系统气质，但视觉重点不同。

- 对话记忆：像时间线和显微镜，强调“某一轮发生了什么”。
- 状态记忆：像控制面板和状态仪表，强调“当前运行状态是否稳定”。
- 长期记忆：像档案库和索引柜，强调“记录是否可靠、是否可注入”。

不要再用一套重复卡片撑完整页。每层应该有自己的主视觉：

- 对话层用纵向时间线加右侧链路图。
- 状态层用结构图、槽位图和压力仪表。
- 长期层用检索台、召回结果和档案卡片。

## 9. 实施顺序

### 第一阶段：前端拆页，不改后端

目标：先降低页面复杂度。

当前状态：已完成第一版。

要做：

- 在记忆系统内加入三段式页面切换。已完成。
- 把现有模块按三层重新摆放。已完成第一版。
- 保留当前所有功能，不改变 API。已完成。
- 默认进入“对话记忆层”，因为这是 debug 最常用入口。已完成。

验收：

- 三个页面都能独立使用。
- 当前会话历史还能展开。
- 单轮记忆链路还能运行。
- 状态记忆和长期记忆不再挤在同一个滚动区域。
- `npm run build` 通过。

### 第二阶段：真实链路接入

目标：让对话记忆能对接测试系统。

当前状态：已完成第一版。

要做：

- 从测试系统跳转到记忆系统时携带 `run_id` 和 `turn_id`。已完成。
- 对话记忆显示真实 memory trace。已完成。
- 同一轮支持“真实链路 / 模拟链路”对照。已完成第一版。
- 状态记忆可以阅读测试 turn 的真实状态记忆片段。已完成。
- 长期记忆可以查看测试 turn 的长期记忆命中。已完成。

验收：

- 测试系统某个失败 turn 可以跳到对应记忆链路。
- 能看出问题发生在状态记忆、长期召回还是 prompt 注入。

### 第三阶段：长期记忆治理

目标：长期记忆从只读查询升级为治理台。

当前状态：已完成受控治理第一版。

要做：

- 候选写入队列。已完成手动写入入口，自动候选队列待做。
- 停用、归档、恢复。已完成停用和归档，恢复待做。
- 冲突记忆提示。已完成只读预检的基础入口，后续可细化冲突判断。
- 记忆污染检查。已完成只读预检的基础入口，后续接测试系统门禁。
- 合并。已完成合并入口，新 note 会 supersede 旧 note，旧 note 会被标记为 `deprecated`。

验收：

- 写入必须经过人工确认。
- 长期记忆修改有明确来源。
- 测试系统能覆盖脏记忆污染风险。

## 10. 推荐下一步

当前三层页面、测试系统联动和受控治理第一版已经完成。下一步建议补回滚和自动候选队列：

```text
候选写入队列
-> 人工确认
-> 归档 / 停用 / 合并
-> 测试系统门禁复核
-> 撤销 / 回滚
```

当前已经开放会改变长期记忆资产的操作，因此后续重点应该是安全网：撤销、审计查看、冲突检测和测试门禁。

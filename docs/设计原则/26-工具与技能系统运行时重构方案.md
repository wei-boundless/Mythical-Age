# 第 26 篇：工具与技能系统运行时重构方案

> 本文不是新的原则来源，而是基于现有 `docs/设计原则/` 的一份落地化重构方案。它只回答一个问题：在当前代码结构下，如何把 tools 和 skills 管理系统收回到符合设计原则的形态。

## 一、问题定义

当前系统的主要问题不是“没有工具/技能注册”，而是三层职责还没有完全收紧：

1. **工具注册层**已经存在，但运行时可见性和任务选择结果之间还不够直连。
2. **技能注册层**已经存在，但 skill 选择结果没有稳定进入主运行链路。
3. **Prompt 装配层**已经从全量能力摘要回撤了一部分，但系统里仍有“静态技能快照”残留产物，容易重新污染链路认知。

这导致的直接现象是：

- agent 明明“有技能”，但本轮运行时不一定真正拿到对应的方法指引；
- tool/skill 的治理结构和模型可见结构没有完全对齐；
- 旧的能力快照文件仍然容易被误当成 prompt 输入来源。

## 二、只按设计原则得到的约束

基于 `docs/设计原则/04-System-Prompt-工程.md`、`09-工具系统设计.md`、`12-Agent-系统.md`、`24-Skill-Plugin开发实战.md`，当前系统必须遵守以下约束：

1. **工具权限不能靠 prompt 解释**
   - 未授权工具不应以“Denied”方式告诉模型。
   - 模型可见工具集合应由结构化 schema / instance 列表决定。

2. **静态 prompt 不承载全量技能/工具目录**
   - 静态 prompt 应保持稳定、缓存友好。
   - 会话级、本轮级 skill 指引只能进入动态段。

3. **skill 是方法扩展，不是权限系统**
   - skill 的职责是提供任务方法、适用边界、输出规则。
   - tool 是否能调，由运行时授权系统决定。

4. **agent 是授权与运行宿主**
   - tool 和 skill 的可用性最终都要落到 agent/runtime lane。
   - 不能只存在于注册表或前端管理面板。

## 三、目标设计

### 3.1 Tool System 的职责

Tool system 只负责四件事：

1. 注册工具定义
2. 提供工具 schema / runtime instance
3. 提供 tool -> operation 的结构映射
4. 在运行时根据 `ResourcePolicy` 产出模型可见工具集合

不再承担：

- 用 prompt 告诉模型什么工具被拒绝
- 用静态摘要向模型广播全量工具目录

### 3.2 Skill System 的职责

Skill system 只负责四件事：

1. 注册 skill runtime contract
2. 注册 skill prompt view
3. 根据任务结构信号选择当前 skill
4. 产出“本轮 active skill 方法块”

不再承担：

- 作为全局能力摘要注入静态 prompt
- 替代运行时授权系统
- 自己决定工具是否真的可执行

### 3.3 Agent / Runtime 的职责

Agent / Runtime 是唯一的消费层：

1. 接收 query understanding
2. 解析 active skill
3. 将 active skill 写入 task runtime contract
4. 将 task operation requirement 送入 `ResourcePolicy`
5. 只把授权后的 tool instances 下发给模型

## 四、当前代码下的边界划分

### 4.1 保留不动的层

- `backend/tools/definitions.py`
- `backend/tools/tool_registry.py`
- `backend/tools/authorization.py`
- `backend/skill_system/contracts.py`
- `backend/skill_system/registry.py`
- `backend/skill_system/policy.py`

这些层的方向基本正确，应继续保留：

- tools 负责定义和注册
- skills 负责 contract 和 prompt view
- authorization 负责结构过滤

### 4.2 需要微调的层

#### A. `runtime/agent_chain.py`

这里必须把理解层输出真正转换成 runtime 输入：

- `query_understanding`
- `active_skill`
- `runtime_required_operations`

这是 skill 从“候选判断”进入“可执行 runtime”的关键桥接层。

#### B. `tasks/contract_builder.py`

这里必须把 active skill 写进 task runtime contract：

- method section 优先使用 active skill
- operation requirement 合并 active skill / selected tool 的运行时操作
- task contract 保留 query understanding 和 active skill 诊断信息

#### C. `orchestration/runtime_loop/task_run_loop.py`

这里继续保持：

- 只把 `ResourcePolicy` 放行后的 tools 下发给模型
- 不再依赖 prompt 解释权限

当前这层方向是对的，不需要重写，只需要继续消费上游更准确的 operation requirement。

## 五、提示词装配规则

### 静态段允许

- 灵魂静态内容
- 共同契约
- 稳定隐藏约束
- 检索优先原则

### 动态段允许

- 当前 active skill 的 prompt block
- 当前上下文包
- 当前相关长期记忆
- 当前 stage projection

### 明确禁止

- 全量 `SKILLS_SNAPSHOT`
- 全量工具目录摘要
- `Denied:` 列表
- `preview_only` / “Do not execute tools” 这类旧护栏文本
- 运行时控制面字段原文泄露

## 六、前端管理面的定位

前端的 tools / skills 页面只做注册管理，不做 prompt 权限解释。

### tools 页面

- 展示工具定义
- 展示 operation 映射
- 展示风险标签与可见性元数据

### skills 页面

- 编辑 skill runtime metadata
- 编辑 model-visible prompt view
- 编辑 allowed_tools 绑定

前端不负责：

- 直接决定模型本轮看见什么
- 生成“Denied 工具清单”

## 七、实施顺序

### 阶段 1：运行时桥接修正

- `AgentRuntimeChainAssembler` 注入 `query_understanding` / `active_skill`
- `TaskRuntimeContract` 合并 runtime-required operations
- 主链 prompt method section 使用 active skill

### 阶段 2：旧语义去污染

- 静态 prompt 不再引用技能快照
- runtime projection 继续过滤 control-plane section
- 清理所有 prompt 中的 denied / preview 旧说明

### 阶段 3：注册与消费一致性

- skill 绑定工具名必须能映射到 operation
- selected skill 必须能进入运行时 task contract
- agent capability profile 必须成为最终授权边界

## 八、验收标准

达到以下条件，才算这次重构完成：

1. 静态 prompt 中不包含全量 skills/tools 摘要。
2. 模型本轮只能看见 `ResourcePolicy` 放行后的 tool instances。
3. active skill 能进入本轮 method section。
4. selected skill 对应的 runtime operation 能进入 adopted resource policy。
5. 前端页面只表达注册事实，不表达“被 denied 的工具清单”。

## 九、结论

这次重构的核心不是“再写一层更复杂的 prompt”，而是把三件事重新拉直：

- **tools 归授权系统**
- **skills 归方法系统**
- **agent/runtime 归消费与执行系统**

只要这三层不再混用，系统就会自然稳定很多。

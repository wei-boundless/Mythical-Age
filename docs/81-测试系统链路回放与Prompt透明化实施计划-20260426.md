# 测试系统链路回放与 Prompt 透明化实施计划

日期：2026-04-26

## 1. 目标

我们接下来要把测试系统从“能运行测试、能看失败问题”推进到“能读懂每一轮对话是怎么跑的”。

目标不是再堆一个报告页面，而是建立一条完整的 debug 链：

```text
测试运行
-> run_result / issues / trace / turn artifact
-> 标准运行链路 overlay
-> 系统框架图高亮与回放
-> prompt 装配 manifest
-> 节点级输入/输出分析
```

最终效果：

- 选择一次测试运行。
- 选择某一轮 turn 或某个 issue。
- 系统框架图显示这一轮实际经过的节点和连线。
- 点击节点可以看到该阶段的事件、耗时、状态、输入输出摘要。
- 点击“Prompt 装配”节点可以看到本轮系统 prompt 由哪些来源组成。
- 可以判断问题来自执行链、状态链、检索链、工具链、模型链，还是 prompt 装配。

## 2. 当前基础

当前已经完成：

- `backend/experiments/catalog.py`
  定义 `smoke / stable / long` 测试 profile。

- `backend/experiments/runner.py`

- `backend/experiments/artifacts.py`
  支持读取 `run_result.json`、`issues.json`、`trace.jsonl`、`report.md`。

- `backend/experiments/graph_mapping.py`
  支持把 issue 派生为系统框架节点和连线引用。

- `backend/api/experiments.py`
  提供测试系统 API。

- `frontend/src/components/workspace/views/TestSystemView.tsx`
  已经可以从前端运行测试、看日志、看最近运行、点 issue 跳系统框架图。

- `frontend/src/components/workspace/views/SystemFrameworkView.tsx`
  已经支持测试 issue 高亮节点和连线。

当前还缺：

- trace 和 turn artifact 没有被解析成标准链路。
- 系统框架图只能高亮 issue 相关节点，不能回放每轮真实路径。
- prompt 装配只有最终字符串，没有结构化 manifest。
- 测试报告和 prompt 来源之间还没有可点击关系。

## 3. 设计原则

### 不破坏原始产物

`run_result.json`、`issues.json`、`trace.jsonl`、turn artifact 都保留原样。新增分析结果通过 API 派生，必要时再写入旁路文件。

### 先结构，后全文

prompt 透明化先展示来源、层级、长度、是否进入模型。全文展开放在 debug 模式，不作为默认视图。

### 图是索引，不是装饰

系统框架图里的节点和连线必须能反查真实代码、真实 trace、真实 prompt 来源。每个高亮都要有数据来源。

### 运行链路和 prompt 装配分开建模

运行链路回答“系统怎么跑”。

prompt manifest 回答“模型读到了什么”。

两者在系统图里汇合，但后端数据模型不要混在一起。

## 5. 后端实施计划

### Phase 1：Trace-to-Graph 轻量解析

目标：把已有测试产物转成系统框架图 overlay。

新增文件：

- `backend/experiments/trace_graph.py`

职责：

- 读取 `run_result.json`
- 读取 `issues.json`
- 读取 `trace.jsonl`
- 读取长场景 `artifacts/**/turn-*.json`
- 根据 event、category、plan_route、tool_names、worker_names、answer_source 等字段推断节点和边
- 输出统一 overlay

建议 API：

```text
GET /api/experiments/runs/{run_id}/graph-overlay
GET /api/experiments/runs/{run_id}/turns
GET /api/experiments/runs/{run_id}/turns/{turn_id}/graph-overlay
```

初始映射规则：

- 所有对话默认经过：
  `api-router -> runtime-root -> query-core -> model -> session-store`

- 出现 `plan_route=worker` 或 `worker_names`：
  加入 `evidence`、`retrieval`

- 出现 `tool_names`：
  加入 `tooling`

- 出现 `memory_context`、`durable_memory`、`session_memory`：
  加入 `memory`、`storage`

- 出现 `answer_source` 包含 direct_tool：
  加入 `query-tools`

- 出现 PDF、structured、table：
  加入 `query-evidence`、`evidence-retrieval`

完成标准：

- 对 `20260426-065921-long` 能列出 turn。
- 任意一个 turn 能返回 overlay。
- overlay 的节点和边 id 都能对应当前 `SystemFrameworkView`。

### Phase 2：系统框架图支持 Overlay 模式

目标：系统框架图不只接收 issue 高亮，还能展示一轮运行路径。

前端修改：

- `frontend/src/lib/store/types.ts`
  增加 `SystemGraphOverlay` 类型。

- `frontend/src/lib/store/core.ts`
  增加默认 overlay 状态。

- `frontend/src/lib/store/runtime.ts`
  增加 `setSystemGraphOverlay` action。

- `frontend/src/components/workspace/views/SystemFrameworkView.tsx`
  支持 overlay 节点状态、边状态、选中 turn、耗时显示。

- `frontend/src/components/workspace/views/TestSystemView.tsx`
  支持从某次 run/turn 跳转到系统框架图 overlay。

视觉规则：

- `passed`：柔和高亮。
- `failed`：暖黄或红橙高亮。
- `skipped`：低透明度。
- `running`：脉冲动画。

完成标准：

- 测试系统里点击 turn 后自动切到系统框架。
- 系统框架显示该 turn 的真实运行路径。
- 点击高亮连线可以看到该链路的摘要。

### Phase 3：Prompt Manifest 结构化生成

目标：让系统 prompt 装配透明、可追踪、可分析。

新增文件：

- `backend/query/prompt_manifest.py`

职责：

- 定义 `PromptManifest`、`PromptSection`
- 对 `static_prompt / session_prompt / turn_prompt` 生成结构化 manifest
- 标注来源、层级、是否进入模型、字符数、preview
- 不改变最终 prompt 文本

建议函数：

```python
build_system_prompt_with_manifest(...)
build_prompt_manifest(...)
```

改造点：

- `backend/query/prompt_builder.py`
  保留现有 `build_system_prompt`，新增并行 manifest 生成能力。

- `backend/query/runtime.py`
  在 `_abuild_system_prompt_for_execution` 处生成 manifest。

- `backend/observability`
  把 manifest 摘要写入 trace annotations 或 artifact。

安全规则：

- 默认 manifest 只保存 preview，不保存完整 prompt。
- debug 模式再允许保存完整 section content。
- 隐藏实现细节 guard 本身可见为“存在”，但默认不全文展开。

完成标准：

- 每轮执行可以生成 prompt manifest。
- manifest 能区分静态层、会话层、当前轮层。
- manifest 不改变模型实际收到的 prompt。

### Phase 4：Prompt 装配可视化

目标：在前端读懂 prompt 是怎么被装配出来的。

前端新增：

- 系统框架图中的 `提示词组装` 节点支持详情面板。
- 测试系统中的 turn 详情支持 prompt manifest。
- Prompt 装配显示为三层树：
  - 静态层
  - 会话层
  - 当前轮层

显示字段：

- section 标题
- 来源
- 层级
- 字符数
- 是否进入模型
- preview
- 可选全文展开

完成标准：

- 从某个 turn 可以打开 prompt 装配视图。
- 能看到 ACTIVE_SEED、CORE、技能、记忆、检索证据是否进入 prompt。
- 能判断 prompt 问题来自哪一层。

### Phase 5：运行链路与 Prompt 合并分析

目标：把运行路径和 prompt manifest 合并成一次完整 debug。

最终流程：

```text
测试系统选择 run
-> 选择失败 issue 或 turn
-> 系统框架图显示运行路径
-> 点击提示词组装节点
-> 查看本轮 prompt manifest
-> 点击记忆/检索/工具节点
-> 查看该节点对 prompt 的贡献或输出
```

完成标准：

- 一个失败 turn 可以同时看到：
  - 实际运行链路
  - 失败节点/边
  - prompt 装配结构
  - 相关 artifact/trace 路径

## 8. 风险控制

### Prompt 泄漏风险

默认只保存 section preview 和统计信息，不默认保存完整 prompt。

全文展开只在本地 debug 模式开启。

### 运行图误导风险

第一版 overlay 是从 trace 和 artifact 推断出来的，不等于完整链路真相。前端需要标注“推断链路”。

后续通过 runtime 直接写事件来提升准确度。

### 产物兼容风险

旧测试产物没有 prompt manifest。API 应该返回 `missing_manifest`，前端显示“此运行没有 prompt 装配记录”。

### 性能风险

长场景 artifact 较多。API 不应一次性返回所有 turn 详情，只返回列表和摘要，点击某个 turn 再加载详情。

## 10. 推荐实施顺序

建议按下面顺序开始：

1. **先做 Trace-to-Graph API**
   因为它能立刻复用已有测试产物，不需要动 prompt 组装主链。

2. **再做系统框架 Overlay UI**
   让测试系统和系统框架真正变成一套 debug 工具。

3. **然后做 Prompt Manifest**
   这一步要碰 prompt 装配核心，必须小心保持原 prompt 文本不变。

4. **最后把 Prompt Manifest 接进 Overlay**
   让用户从运行路径直接进入 prompt 装配分析。

这个顺序风险最低，也最容易持续验证。

## 11. 第一轮实施目标

第一轮只做最小闭环：

- 后端新增 `trace_graph.py`
- API 返回 run-level overlay 和 turn-level overlay
- 前端系统框架能显示 overlay
- 测试系统能从 turn 跳转系统框架

不在第一轮做：

- prompt 全文保存
- 节点级完整输入输出
- 动态播放动画
- LangSmith 深度集成

第一轮完成后，我们就能直接把一次测试报告“读成图”。这会让后续修 follow-up、记忆召回和工具续写问题快很多。

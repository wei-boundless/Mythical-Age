# 长对话测试场景设计

> 目标：把当前项目已经具备的 `query -> skill/tool -> retrieval -> memory -> api/frontend runtime` 主链路，整理成一组可执行、可扩展、可压测的长对话剧本，用来覆盖“基本功能 + 易错边界 + 压力场景”。

## 1. 设计原则

- 不是只测单点功能，而是测一条长会话里多个能力连续切换后的稳定性。
- 不是只看最终答案，而是同时看路由、SSE 事件、session memory、durable memory、task record、settings/permission 生效情况。
- 不是只做 happy path，要专门覆盖最容易出错的边界：
  - 话题切换
  - follow-up 路由恢复
  - 结构化数据集错绑
  - 权限拒绝后的安全回退
  - context pressure 下的 compact / restore
  - 多 session 并发切换

结构化场景目录已落在 [conversation_scenario_catalog.py](D:/AI应用/langchain-agent/backend/tests/conversation_scenario_catalog.py)。

## 2. 覆盖矩阵

| 场景 ID | 目标 | 核心覆盖 |
|---|---|---|
| `full-workbench-journey` | 单会话全链路验收 | rag、pdf、structured、weather、gold、memory、topic switch、SSE |
| `pdf-switch-and-resume` | PDF 跟读后切题再恢复 | active_pdf、warm snapshot、PDF follow-up |
| `structured-dataset-followup-and-rebind` | 结构化数据跨数据集切换 | dataset binding、structured follow-up |
| `skill-route-and-settings-boundary` | skill route 与 rag_mode 切换 | rag-skill、web-search、weather、gold、settings |
| `compound-query-task-fanout` | 复合请求拆分执行 | task coordinator、subtask event |
| `durable-memory-write-and-semantic-recall` | 长期记忆写入与跨会话回忆 | durable write、exact recall、semantic recall |
| `correction-and-non-durable-boundary` | 更正与非长期记忆边界 | correction、emotion/session-only、fresh session |
| `permission-denial-and-safe-fallback` | 权限拒绝后的安全退路 | permission mode、policy deny、safe fallback |
| `context-pressure-and-compaction` | 长会话压力与压缩恢复 | warning/microcompact/full_compact、restore |
| `multi-session-isolation-and-resume` | 多 session 并发切换 | session isolation、resume、state separation |
| `streaming-contract-mixed-events` | 前后端流式契约 | token/tool/retrieval/done、frontend reducer |

## 3. 推荐优先级

### P0：每次主链改动都要跑

- `full-workbench-journey`
- `pdf-switch-and-resume`
- `structured-dataset-followup-and-rebind`
- `durable-memory-write-and-semantic-recall`
- `streaming-contract-mixed-events`

### P1：每次路由 / 权限 / settings 改动都要跑

- `skill-route-and-settings-boundary`
- `compound-query-task-fanout`
- `permission-denial-and-safe-fallback`

### P2：每日或每次版本候选前跑

- `context-pressure-and-compaction`
- `multi-session-isolation-and-resume`
- `correction-and-non-durable-boundary`

## 4. 关键断言

### 4.1 基本功能必须同时看 4 类结果

- 用户可见结果：回答是否正确、是否能接住 follow-up。
- 运行时结果：route、tool、retrieval、task、permission 是否命中预期。
- 状态结果：`sessions/*.json`、`session-memory/*`、`durable_memory/*` 是否更新正确。
- 协议结果：SSE 事件顺序是否稳定，前端 reducer 是否能归并。

### 4.2 最容易出错的点

- `PDF -> gold price -> weather -> back to PDF` 后，`active_pdf` 丢失。
- `inventory.xlsx -> employees.xlsx -> back to inventory.xlsx` 后，structured follow-up 绑错表。
- 记忆提取把“我今天很难过”“我爱上你了”错误写成 durable memory。
- 被权限拒绝后直接失败，没有给安全回退路线。
- retrieval evidence 被重复写进 runtime message history，导致 prompt/context 膨胀。
- 长会话 compact 后，最新安全要求或当前任务丢失。
- 多 session 并发切换后，A 会话状态污染到 B 会话。

## 5. 压力测试维度

### Turn 压力

- 单 session 连续 24-30 turn。
- 中间至少 3 次显式 topic switch。
- 至少 2 次“切出去再回来”的恢复动作。

### Payload 压力

- 至少 10-12 轮 retrieval-heavy 或 table-heavy 输出。
- 每轮都包含大块检索文本、表格摘要或多段结论。

### Session 压力

- 同时维护 3 个 session：
  - PDF 分析
  - 结构化数据分析
  - 实时工具查询
- 快速交替切换 8 次以上。

### Memory 压力

- 同一会话中既写 durable memory，又插入大量 session-only 噪音。
- 至少一次纠错，验证旧事实被覆盖。
- 至少一次 fresh session，验证只有 durable memory 跨会话保留。

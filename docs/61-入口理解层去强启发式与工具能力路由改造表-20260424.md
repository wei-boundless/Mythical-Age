# 入口理解层去强启发式与工具能力路由改造表

> 日期：2026-04-24
> 状态：待执行
> 目标问题：fresh query 入口仍保留“弱词触发即强路由”的短路树，导致“搜索本地数据库”被误判为联网搜索。
> 关联文档：
> - `docs/38-去启发式续接与显式句柄主链重构技术报告及计划书.md`
> - `docs/54-路由层启发式审查矩阵-20260423.md`
> - `docs/55-工具与技能系统契约化重构初步计划书.md`
> - `docs/58-协议残差与脑补工具调用收缩计划书.md`

---

## 1. 现状结论

当前系统已经完成了：

1. follow-up / binding / owner 主链的去弱启发式收缩
2. 工具契约化与高污染工具 fail-closed
3. skill 的能力化与路由降权

但 fresh query 入口仍残留一个未收口的问题：

`task_understanding.py` 仍把一部分弱词直接当成 route authority 使用。

这会导致如下错误链路：

1. 用户说“搜索本地数据库”
2. `_detect_web_task()` 先命中“搜索”
3. 直接产出 `web-search + web_search`
4. 后续 skill / tool / runtime 只能顺着误判执行

所以这不是 follow-up 主链失效，也不是工具层再次越权，而是：

`入口理解层仍然保留了旧式启发式短路权`

---

## 2. 本次改造的边界

本表只处理以下责任链：

1. `fresh query -> coarse understanding`
2. `coarse understanding -> skill/tool candidate set`
3. `tool candidate set -> runtime direct tool eligibility`

本表不处理：

1. follow-up owner 仲裁
2. 输出层答案收口
3. 协议残差清洗
4. 具体工具内部算法

换句话说，本次改造只修：

`入口如何从“词触发强路由”改成“能力候选 + 契约收束”`

---

## 3. 必须遵守的设计约束

依据 `docs/38`、`docs/54`、`docs/55`，本次改造必须遵守以下硬约束：

1. 弱词只能产生候选信号，不能直接产出强 route。
2. 理解层只能输出粗粒度意图，不能越权替工具层做最终对象级裁决。
3. tool / skill 的最终执行资格应由工具契约、显式输入、binding、权限共同决定。
4. “搜索”只能表示用户在找信息，不能天然等于“联网”。
5. 不能通过新增“本地数据库 / 库存表 / 仓库数据”这类词表来重新建立另一套强启发式。
6. 对 fresh query，只有显式外部性约束、显式文件句柄、显式专用任务三类情况允许入口直达工具。
7. 其余歧义查询必须保留为未决能力候选，交给 planner / contract / 受限工具池继续收束。

---

## 4. Claude Code 对照结论

对照 `d:\\AI应用\\claude-code-nb-main`，可提炼出三条可借鉴原则：

1. 它不依赖一个全局 query classifier 先把“search”翻译成“web route”。
2. 它把“本地搜索 / 文件读取 / grep / glob / web search / web fetch”拆成独立工具能力，让模型按工具职责选。
3. 它把决策重点放在工具定义、工具发现、权限和 defer loading 上，而不是放在入口词触发树上。

对应参考实现：

| 参考点 | Claude Code 文件 | 含义 |
|---|---|---|
| 工具能力独立暴露 | `Tool.ts` `tools.ts` | 先定义工具边界，再让模型选工具 |
| 延迟工具发现 | `tools/ToolSearchTool/ToolSearchTool.ts` `utils/toolSearch.ts` | 稀有/高成本工具不靠入口分类器抢路由 |
| 联网工具职责显式化 | `tools/WebSearchTool/WebSearchTool.ts` `tools/WebFetchTool/WebFetchTool.ts` | web 只负责 external / current information，不和本地搜索混义 |
| 本地搜索职责显式化 | `tools/GrepTool/*` `GlobTool` `FileReadTool` | 本地信息获取由本地工具承担，不需要“联网前置判定” |

本次改造不复制 Claude Code 的全部架构，只借它的两个不变量：

1. `动作词 != route authority`
2. `工具能力边界 > 入口启发式短路`

---

## 5. 目标设计

改造后，入口主链应收敛为：

`user message -> coarse intent + execution posture -> bounded capability set -> planner/contract narrowing -> execution`

其中：

1. 理解层只负责识别“是否存在足够强的直达条件”，而不是替后链决定具体工具。
2. `搜索 / 查询 / 看看` 这类动作词只表达 lookup posture，不表达外部来源归属。
3. 若用户没有给出显式外部性约束、显式 dataset 句柄、显式 PDF 句柄，则入口不再把它锁成具体工具。
4. 对这类未决请求，系统进入 `agent + bounded tools` 路径，而不是 `direct_tool`。
5. 只有显式要求外部当前信息、显式给出本地文件句柄、或命中天气/金价这类专用任务时，才允许入口强直达。

## 7. 分阶段执行顺序

## Phase 1. 收缩入口强短路

目标：

1. 不再让“搜索”这类宽动作词直接触发 web。
2. 把入口改成“强直达 or 未决”，不再补本地词表。
3. 保持现有 follow-up / contract / output 层不动。

涉及文件：

- `backend/understanding/task_understanding.py`
- `backend/tests/task_understanding_regression.py`

完成标准：

1. “搜索本地数据库”不再落 `web_search`
2. “搜索本地数据库”进入未决执行姿态，而不是新的 dataset 词表直判
3. “查官网最新文档”仍然稳定落 web
4. “北京今天天气”不受影响

## Phase 2. 引入候选收束，不再前命中即返回

目标：

1. 让 coarse understanding 支持 `agent` 中性执行通道
2. 把最终锁 tool 的时机往 planner/runtime 后移

涉及文件：

- `backend/understanding/query_understanding.py`
- `backend/query/planner.py`
- `backend/tests/query_planner_regression.py`

完成标准：

1. fresh query 支持“未决任务 + bounded tools”仲裁
2. 单候选直锁只保留给强显式任务

## Phase 3. 让 binding / local data candidate 参与收束

目标：

1. 对未决分析型任务，允许 binding 作为后链证据进入 planner
2. 但不破坏 fail-closed

涉及文件：

- `backend/query/binding_resolver.py`
- `backend/query/tool_input_resolver.py`
- `backend/tests/structured_followup_history_regression.py`

完成标准：

1. 活跃 owner 能帮助未决任务收束
2. 缺显式 path 时仍不允许高污染工具静默执行

## Phase 4. 可观测性与收口

目标：

1. trace 能展示“强直达 / 后链仲裁”的分界
2. skill 文档与入口理解对齐

涉及文件：

- `backend/query/runtime.py`
- `backend/skills/web-search/SKILL.md`
- `backend/skills/structured-data-analysis/SKILL.md`

完成标准：

1. 可在 trace 中复盘“为什么被入口直达 / 为什么保留未决”
2. skill 文档不再暗示“搜索=联网”

## 8. 明确禁止的做法

本次改造过程中，禁止以下做法：

1. 继续扩充 `_detect_web_task()` 的宽词表。
2. 新增“本地数据库/库存表/仓库数据”一类本地源词表来替换旧启发式。
3. 为每个误判句式追加特判 patch。
4. 重新把 skill hint/example/tag 拉回路由加权器。
5. 让 binding resolver 再次承担 owner 裁决权。
6. 通过输出层兜底假装解决入口误判。

---

## 9. 最终判断

本问题的真实责任点是：

`入口理解层仍保留弱词直判强 route 的旧式短路树`

因此正确修法不是：

`继续补更多词`

而是：

`把入口从“启发式裁决器”收缩成“强直达检测器 + 未决任务分流器”，再由能力边界、binding、契约和 planner/runtime 完成收束`

这也是当前代码最应该向 Claude Code 借鉴的地方。

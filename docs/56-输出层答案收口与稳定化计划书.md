## 1. 计划定位

本计划书只处理一个问题域：

`已有 route / binding / tool result，如何稳定收成用户可见答案`

它不再处理：

1. 工具是否应该被执行。
2. skill 是否应该参与本地路由。
3. 高污染工具是否允许缺 owner 执行。

这些边界已经在 `docs/55` 收口完成。

本计划书只承接以下三条责任链：

1. `memory route -> visible answer`
2. `RAG evidence -> canonical answer`
3. `PDF canonical -> stable summary`

---

## 2. 当前现状

根据 `docs/52` 与 `docs/53`，当前系统的主链稳定度已经明显提升：

1. follow-up 仲裁主链基本稳定。
2. compound authority 传递基本稳定。
3. 工具与技能边界已经完成契约化收缩。

但长场景仍暴露出一组输出层问题：

1. 检索命中了，但最终没有形成可展示答案。
2. memory 路由中，过程化文本仍可能直接进入可见输出。
3. PDF 工具拿到了页码/章节证据，但 canonical summary 仍为空。

这说明当前系统的主要缺口已经从“谁来执行”转移到了“执行结果如何收口”。

---

## 3. 重构原因

如果不单独处理输出层，系统会持续出现两类伪稳定状态：

1. 主链判断是对的，但用户仍看到空洞 fallback。
2. 工具执行是对的，但用户看到的是过程流、协议流、半成品摘要。

这类问题的风险比单纯答错更大，因为它会制造“系统似乎工作了，但没有形成正式答案”的体验断裂。

按照当前 docs 原则，输出层修复必须满足：

1. 可见答案优先来自 canonical channel，而不是脏的流片段。
2. 有 evidence 时优先保守收口，少做无根据 fallback。
3. 污染识别与答案拒收要前移到输出层边界，不靠提示词补丁。
4. 不把输出层问题误修成新的路由启发式。

---

## 4. 问题分组

## 4.1 A 组：RAG 答案收口失败

典型现象：

1. 已有 retrieval evidence。
2. rerank 也可能已经生效。
3. 最终却落到 `rag_missing_answer`。

问题本质：

不是检索不到，而是 evidence 没有被稳定组装成 final answer。

目标：

让 `RAG evidence -> canonical answer` 成为正式、可追踪的收口链。

## 4.2 B 组：Memory 输出污染

典型现象：

1. route 已经正确进入 `memory`。
2. 模型流中出现 `让我检查知识库目录结构` 之类过程化文本。
3. 这些脏文本被 `segment.visible_text` 直接透给用户。

问题本质：

output boundary 已能识别污染，但还没有真正阻断污染成为最终答案。

目标：

让 `memory route -> visible answer` 必须经过候选清洗、拒收和保守回退。

## 4.3 C 组：PDF Canonical Summary 缺失

典型现象：

1. `pdf_analysis` 已执行成功。
2. 页码、章节、可读页范围都拿到了。
3. `summary.response` 仍为空，最终落成 `pdf_missing_summary`。

问题本质：

不是 PDF 没定位到，而是 `tool output -> canonical summary -> final answer` 这段没有收严。

目标：

让 PDF 输出在拿到有效证据后，至少形成保守、稳定、可展示的 canonical summary。

---

## 5. 目标架构

本阶段目标链路为：

`route result -> tool/retrieval evidence -> canonical candidate -> output boundary -> visible answer`

其中需要新增或强化的，不是 route，而是三类“答案收口器”：

1. `RAGAnswerClosure`
2. `MemoryVisibleAnswerGate`
3. `PDFCanonicalClosure`

它们不一定是独立类，但在结构职责上必须分清楚。

## 5.1 RAGAnswerClosure

职责：

1. 识别“已有 evidence，但模型答案缺失”的场景。
2. 用保守策略从 evidence 生成 canonical answer candidate。
3. 明确区分：
   - 无检索结果
   - 有检索结果但模型未收口

## 5.2 MemoryVisibleAnswerGate

职责：

1. 识别过程化、协议化、伪操作型输出。
2. 对高污染的 `visible_text` 做拒收，而不是仅打标。
3. 只允许足够稳定的内容段进入最终答案。

## 5.3 PDFCanonicalClosure

职责：

1. 接收 PDF 工具输出的页码、章节、片段证据。
2. 形成统一 canonical summary candidate。
3. 与 persistence gate 解耦，避免“有证据但 summary 为空”的空转。

---

## 6. 技术采用原则

本阶段明确采用以下技术策略：

## 6.1 保守合成，不做过度生成

当 evidence 已足够但答案缺失时，优先做保守摘要，而不是让模型自由发挥。

## 6.2 先拒收污染，再讨论生成增强

对于 memory 输出，第一优先级不是“更会总结”，而是“不把脏流直接当答案”。

## 6.3 canonical 与 visible answer 分层

必须把：

1. 原始流
2. canonical candidate
3. 最终 visible answer

清楚分层，避免把任何一个流片段直接透传成正式答案。

## 6.4 PDF 只负责提供可用证据，不承担全部自然语言理解

PDF 链的目标是：

1. 提供稳定页级/章节级证据。
2. 形成可展示 canonical summary。

而不是把全部语义理解责任堆给 PDF 工具本身。

## 7. 分阶段路线

## Phase 1：输出层观测与分类固化

目标：

1. 把当前输出层失败类型稳定分类。
2. 区分 `rag_missing_answer`、`memory_visible_pollution`、`pdf_missing_summary`。
3. 让 trace 能明确看到“候选为何被拒收”。

涉及文件：

1. `backend/query/output_boundary.py`
2. `backend/query/output_classifier.py`
3. `backend/query/runtime.py`

完成标准：

1. 输出层失败原因不再混成同一个 fallback。
2. 每类失败都有可追踪的 decision reason。

## Phase 2：Memory 输出污染拦截

目标：

1. 对带 leak flag 且高度过程化的可见文本做正式拒收。
2. 不再允许 `segment.visible_text` 在明显脏内容场景下直接成为最终答案。

涉及文件：

1. `backend/query/output_boundary.py`
2. `backend/query/output_classifier.py`
3. `backend/query/runtime.py`

完成标准：

1. “检查目录结构”“我来检索一下”这类过程化文本不再直接对用户可见。
2. memory 路由在无稳定答案时会走更明确的保守回退。

## Phase 3：RAG Answer Closure

目标：

1. 为 RAG 路由建立 evidence 到 canonical answer 的保守合成链。
2. 在已有证据但模型未收口时，不再直接掉进 `rag_missing_answer`。

涉及文件：

1. `backend/query/output_classifier.py`
2. `backend/query/runtime.py`
3. 如有必要：`backend/query/answer_assembler.py`

完成标准：

1. 已命中 evidence 的 RAG 首问能更稳定地产出答案。
2. `rag_missing_answer` 明显下降。

## Phase 4：PDF Canonical Closure

目标：

1. 为 PDF 结果建立统一 canonical summary 生成链。
2. 让 page/section 级定位结果能够收成稳定输出。

涉及文件：

1. `backend/query/runtime.py`
2. `backend/pdf_agent.py`
3. 如有必要：`backend/pdf_analysis/*`

完成标准：

1. `pdf_missing_summary`、`pdf_canonical_missing_summary` 明显下降。
2. 有效页码/章节证据能形成可展示答案。

## 9. 验证矩阵

## 9.1 局部回归

需要覆盖：

1. memory route 输出含过程化文本时的拒收行为。
2. RAG route 有 evidence 但无答案时的 closure 行为。
3. PDF route 有 page/section 证据时的 summary closure 行为。

## 9.2 长场景回归

至少重跑：

1. `research-brief-and-document-resume`
2. `commerce-ops-data-live-switch`

关注指标：

1. `answer_fallback_reason`
2. `answer_source`
3. `answer_leak_flags`
4. `pdf_missing_summary`
5. `rag_missing_answer`

## 9.3 验收标准

通过标准不是“所有回答完美”，而是：

1. 已有证据时，空 fallback 显著减少。
2. 脏过程化文本不再直接透给用户。
3. PDF 有证据时不再大面积空摘要。
4. 不破坏当前已经收口的工具/技能/路由边界。

---

## 10. 禁止事项

本阶段明确禁止：

1. 为了修答案质量，重新给理解层加大词表启发式。
2. 为了让 PDF 更会答，重新放松 owner/binding 约束。
3. 为了掩盖污染，简单做字符串硬截断替代结构拒收。
4. 把输出层问题再次改造成工具层、skill 层或 follow-up 层问题。

---

## 11. 最终结论

当前系统下一步最合理的路线不是继续重做边界，而是做输出层收口。

更准确地说：

1. `docs/55` 已经完成“能不能正确执行”。
2. `docs/56` 要解决“执行完以后能不能稳定说出来”。

这两阶段必须严格分开推进。

## 1. 当前问题定义

当前系统已经基本修复了：

1. follow-up 候选越权
2. planner 前短路
3. binding authority 漂移
4. compound 子任务 authority 丢失

但长场景实测表明，系统仍有一批“主链没错，但答案质量不稳定”的软问题。

这些问题的真实结构不是“同一个 bug 的不同表现”，而是四类职责尚未彻底收口：

1. `RAG evidence -> canonical answer`
2. `memory route -> visible answer`
3. `pdf canonical -> stable summary`
4. `intent routing for cross-source synthesis / dangling references`

因此，本轮不能采用“大包修复”方式，而必须按责任边界分组推进。

---

## 2. 当前系统状态判断

### 2.1 已被验证稳定的部分

以下能力已经通过本地回归和长场景间接验证：

1. `followup_resolver` 不再默认抢占 planner 主链。
2. `runtime` 已成为 binding follow-up 的统一仲裁点。
3. `tool_input_resolver` 已能保证结构化 authority 不被弱短路径覆盖。
4. `planner` 的 compound authority 滚动传递已经基本稳定。
5. `TaskCoordinator` 的 query subtask 编号冲突已经修复。

### 2.2 当前不应再动的大边界

本轮精修阶段，不应重新改造以下主边界：

1. `followup_resolver` 的 candidate-first 语义
2. `runtime._should_execute_binding_followup(...)` 的主仲裁模式
3. `planner` 的 compound 顺序构建机制
4. `tool_input_resolver` 的 authority 优先级

这些部分现在属于“已收口主链”，只能在发现硬回退时再动。

## 3. 本轮精修要遵守的设计约束

约束来自：

- `docs/设计原则/03-状态管理.md`
- `docs/设计原则/05-对话循环.md`
- `docs/设计原则/06-上下文管理.md`
- `docs/设计原则/12-Agent-系统.md`
- `docs/设计原则/14-任务系统.md`
- `docs/设计原则/23-Memory系统.md`
- `docs/设计原则/25-架构模式总结.md`

本轮落成的强约束如下：

### 3.1 不用提示词扩规则替代结构修复

- 问题若出在 `output boundary` 或 `canonical answer` 收口，不允许只靠 prompt 补丁掩盖。

### 3.2 restore 不得越权决定当前 turn

- 即使要修 dangling reference，也不能重新把历史恢复逻辑抬回决策层。

### 3.3 可见答案必须来自 canonical channel

- 任何带 leak flag 的候选，如果仍含明显过程化内容，不得继续作为最终 `visible answer`。

### 3.4 摘要缺失要和路由误判分开处理

- `pdf_missing_summary`、`rag_missing_answer` 属于收口问题。
- `多源总结被判成 weather` 属于理解和路由问题。
- 两类问题不能混修。

### 3.5 dangling reference 要显式降级，不要隐式乱猜

- 没有 owner binding 的“刚才那份 PDF”不能靠工具层自己兜底猜出路径。
- 要么澄清，要么显式退化，但不能假装找到了 owner。

## 4. 问题分组

本轮 5 个问题分成 4 组。

### Group A：RAG 答案收口

覆盖问题：

1. `rag_missing_answer`

目标：

- 检索命中后，优先产出稳定 answer candidate，而不是直接落 fallback。

边界：

- 不改 retrieval backend
- 不改 rerank
- 只修 `retrieval_results -> model stream -> output decision`

### Group B：Memory 输出污染拦截

覆盖问题：

1. memory 改写轮出现大量过程化文本

目标：

- `memory route` 下带 leak flag 的过程化文本不再直接进入最终答案。

边界：

- 不改 planner 对 `memory` 的判定
- 不回退当前 session memory 结构
- 只修 `model stream -> output boundary -> output decision`

### Group C：PDF Canonical Summary 收口

覆盖问题：

1. `pdf_missing_summary`
2. `pdf_canonical_missing_summary`

目标：

- PDF 工具已经拿到页码/章节/可读页后，要么形成稳定 summary，要么显式说明“证据不足”，而不是只返回空摘要 fallback。

边界：

- 不动 follow-up binding 主链
- 不动 pdf 子 agent 的大架构
- 只修 `pdf_analysis output -> canonical -> finalization/persistence gate`

### Group D：多源总结与悬空引用路由治理

覆盖问题：

1. `库存、员工、黄金和天气` 被误判成 `get_weather`
2. 悬空 `回到刚才那份 PDF`

目标：

- 多源总结应优先进入 `memory / synthesis` 路由，不得被单一 realtime keyword 抢走。
- 悬空指代应显式判为“缺 owner”，而不是退化成假正常工具调用。

边界：

- 不重新放宽 follow-up heuristics
- 不让 dangling reference 重新走弱绑定猜测

## 5. 推荐修复顺序

本轮建议固定顺序如下：

1. Group D：先修路由误判和悬空引用
2. Group B：再修 memory 可见输出污染
3. Group A：再修 RAG answer 收口
4. Group C：最后修 PDF canonical summary

原因如下：

### 5.1 先修 Group D

因为：

- 路由错了，后面所有答案层修复都可能修在错误通道上。
- 特别是 turn 9 的天气误路由，本质不是输出问题。

### 5.2 再修 Group B

因为：

- memory 污染目前直接影响系统可信度。
- 这类问题不需要大改业务语义，但必须先让脏内容出不来。

### 5.3 再修 Group A

因为：

- RAG 首问现在有 evidence，但没有 canonical answer。
- 在输出层稳定之后再修 RAG 收口，能减少误判。

### 5.4 最后修 Group C

因为：

- PDF canonical summary 是最容易牵动工具结果、persistence gate 和 finalization 的一组。
- 应该放在主链与输出层稳定之后处理。

## 6. 目标设计

### 6.1 Group D 目标设计

#### D1. 多源总结路由

设计要求：

1. 当同一句里出现多个 source domain：
   - `库存 / 员工 / 黄金 / 天气 / PDF / 知识库 / 记忆`
2. 且同时出现：
   - `摘要 / 总结 / 概括 / 分开 / 分段 / 四块 / 四段 / 分成`

则当前 turn 应优先进入：

- `memory`
或
- `compound / synthesis`

而不是单 tool。

#### D2. 悬空对象引用

设计要求：

1. `回到刚才那份 PDF`
2. `继续那个表`
3. `那个报告`

如果当前 session 中没有 owner binding：

- 不得直接退化成裸 tool
- 应显式进入：
  - `clarify`
  - 或 `none + 主链澄清`

### 6.2 Group B 目标设计

设计要求：

1. `output_boundary` 可以继续保留 `segment.visible_text`
2. 但若同时满足：
   - `answer_leak_flags` 非空
   - 可见文本里高度命中 `_PROCEDURAL_LINE_RE`
   - 没有有效内容句

则不能再把这段文本直接视为最终答案。

正确行为：

1. 要么降级为 fallback
2. 要么进入 “answer candidate rejected” 分支
3. 但不能把脏过程文本透给用户

### 6.3 Group A 目标设计

设计要求：

1. RAG 路由下，若有 retrieval evidence
2. 模型流没有稳定成 answer

则系统需要优先尝试：

- 从 retrieval evidence 构造保守 canonical answer
或
- 让模型做一次受约束 summarization

而不是立即进入：

- `rag_missing_answer`

### 6.4 Group C 目标设计

设计要求：

1. PDF 工具只要已经拿到：
   - `pdf_pages`
   - 或 `page/section`
   - 或 `readable_pages/usable_pages`

就不应只有空 `summary.response`。

正确收口顺序应为：

1. `raw tool output`
2. `PDFCanonicalResult`
3. `canonical summary candidate`
4. `model finalization` 可选增强
5. `persistence gate`

而不是当前这种：

- canonical 有页码，但 summary 仍为空，最终直接 fallback。

## 9. Cutover 规则

### 9.1 允许的渐进式改法

本轮允许：

1. 先加拒收规则
2. 再加更好的 canonical answer
3. 最后再删旧 fallback

### 9.2 不允许的改法

本轮禁止：

1. 为了压掉问题，重新给 `followup_resolver` 加强启发式抢占
2. 为了让 dangling reference 看起来“能跑”，重新引入弱路径猜测
3. 为了掩盖污染，直接在最终 answer 上做字符串硬截断
4. 在未验证前同时修改 route / output / pdf 三条链

---

## 10. Rollback 规则

出现以下情况必须回退到上一阶段：

1. 原本稳定的 follow-up binding_ref 大面积失效
2. compound 子任务 authority 再次丢失
3. `task_summary_refs` / `main_context.followup_target_task_id` 再次错位
4. 多源总结虽然不再落 weather，但开始大量误判成 clarify
5. memory 输出虽然变干净，但直接退化成大面积 `memory_missing_answer`

---

## 11. 最终执行建议

建议执行方式：

1. 先从 Group D 开始
2. 每完成一组，先跑局部回归
3. 再跑长场景双 scenario
4. 只有确认未破坏已收口主链后，再进入下一组

当前不建议直接上手修 Group C。

因为 PDF summary 问题表面最显眼，但它对：

- output boundary
- fallback policy
- route correctness

都有依赖，若顺序反了，后面会不断误判根因。

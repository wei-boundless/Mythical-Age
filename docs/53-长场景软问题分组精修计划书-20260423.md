## 2. 本文档当时完成了什么

`docs/53` 当时完成的核心工作有三项：

1. 证明了长场景问题不是单点 bug，而是分布在不同责任链上。
2. 把剩余问题明确拆成了 `A/B/C/D` 四组。

这一步非常关键，因为如果没有 `docs/53`，后续很容易把：

1. 路由误判

## 3. 四个问题组现在分别去了哪里

## 3.1 Group D：路由误判与悬空引用

`Group D` 的核心问题是：

1. 多源总结被单工具关键词抢路由。
2. dangling reference 没有 owner 时仍试图退化执行。

这一组的主要结构性部分，已经被后续的工具/技能/路由边界收缩工作吸收，最终落到：

- `docs/55-工具与技能系统契约化重构初步计划书.md`

也就是说，`Group D` 不再单独在 `docs/53` 中推进。

## 3.2 Group A / B / C：输出层问题

`Group A/B/C` 分别对应：

1. `RAG evidence -> canonical answer`
2. `memory route -> visible answer`
3. `PDF canonical -> stable summary`

这些问题本质上已经被确认是输出层和答案收口问题，不再属于工具/技能执行边界问题。

因此它们现在统一转入：

- `docs/56-输出层答案收口与稳定化计划书.md`

## 4. 为什么 `docs/53` 不能继续当主计划

如果现在继续沿用 `docs/53` 作为主计划，会出现三个问题：

1. 它仍把结构边界问题和输出层问题放在同一文档里。
2. 它会让已经收口的 `follow-up / route / contract` 议题被反复打开。
3. 它的分组是“问题分类”，不是“当前执行边界”。

当前系统更需要的是：

1. `docs/55` 负责说明结构边界已经收口。
2. `docs/56` 负责承接剩余输出层计划。

因此，`docs/53` 最合理的定位是：

- 历史分组文档
- 问题责任拆解依据
- 规划过渡文档

而不是当前活跃任务主文档。

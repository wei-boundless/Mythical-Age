# 75-原则分层与现有链路违背点精确对照表-20260425

> 目的：不是再写一份抽象计划书，而是把 `docs/设计原则`、`68/70/71/74` 中已经确认的架构原则，和当前代码中的真实链路逐项对照。  
> 这份表用于回答一个更具体的问题：**现在这些语义 bug，到底是哪个分层没有守住，还是旧链路没有清掉。**

---

## 0. 总判断

当前问题的主因，不是“没有设计原则”，而是：

1. 原则已经写在 docs 里了。
2. 新链路骨架也已经写进代码了。
3. 但 runtime 里仍允许旧链路越权改写当前轮决策。
4. 于是系统表面上有 `worker / handle / evidence`，实际上关键节点还在走 `authority / owner / path / heuristic`。

所以现在的真实状态不是“缺设计”，而是：

> **原则层、新链路层、旧兼容链路层三套逻辑同时存在，但边界没有被硬性执行。**

## 2. 结构性判断

从这张对照表看，当前系统最核心的结构性问题是四个：

1. **恢复层越权**
   - `continuation_resolver` 还在替当前轮做 route 决策。
   - 这直接违反了 `restore != decide`。

2. **弱线索越权**
   - `task_understanding/followup_resolver` 仍把“像不像在问 PDF/表格”当成绑定依据。
   - 这直接违反了“弱线索只能产出 candidate”的原则。

3. **handle 粒度不足**
   - 即使新链路已经开始使用 `object/result/subset handle`，但 PDF section、结构化 subset 还没有把“对象身份”稳定编码进去。
   - 于是 follow-up 只能退回 owner/path/task 这种粗粒度恢复。

4. **观测与门控脱节**
   - `structured_memory` 已经能看出漂移和污染。
   - 但 runtime 还没把这些信号变成“禁止继续错误执行”的硬边界。

---

## 3. 为什么这不是“再补几个关键词”能解决的

因为现在坏的不是识别精度本身，而是**权限边界**：

1. 如果 `restore` 还可以改写 route，那么你修一次 PDF 误判，下一次它还会在 dataset/mixed-source 上重演。
2. 如果 `followup` 还靠“这页/这个表/第二部分”来绑定，那么任何语言变体都可能继续漏掉。
3. 如果 `section/subset` 没有稳定 handle，那么上一轮的结果根本不是“可续接对象”，而只是“一段刚好被答出来的文本”。
4. 如果风险信号不门控 runtime，那么系统即使知道自己在漂，也会继续往下错。

所以后续修复必须是：

> **先把越权点拆掉，再补足 handle 和协议，最后才是调 prompt 和输出措辞。**

---

## 4. 推荐修复顺序

### 第一批：先修边界，不修表现

1. 收掉 `continuation_resolver` 中 authority 对当前轮 route 的直接改写权。
2. 把 `followup_resolver` 中 PDF/dataset 的弱绑定判断降级成 candidate，不再直接产出 binding follow-up。
3. 在 `runtime.py` 把 risk flags 接成硬门控：
   - `cross_flow_slot_contamination`
   - `implicit_goal_jump`
   - `clarification_required`

### 第二批：补齐对象协议

1. 给 PDF section 建稳定 `section handle/result handle`。
2. 给 PDF follow-up 约束加 `section_key/page_no`，不再只有 `document/page`。
3. 给 structured subset 走 typed filter，不再把 subset 当自然语言前缀注入。

### 第三批：收尾层细化

1. PDF degraded reason 转成用户可理解解释。
2. structured direct output 统一进 answer finalizer。
3. multi-source synthesis 进入统一 evidence arbitration，再决定是否 drill-down 到单源 worker。

## 6. 最后结论

如果只问一句“到底是原则分层没做到位，还是某几个模块有 bug”，答案是：

> **本质上是分层边界没有真正落地，旧链路仍在关键位置保留越权；具体 bug 只是这些越权点在长场景里的外显症状。**

所以后续修，不应该继续按 turn 打补丁，而应该按这张对照表逐项消除：

1. 恢复层越权
2. 弱线索越权
3. handle 粒度不足
4. 风险只观测不门控

把这四类点处理掉，后面的 PDF、RAG、结构化、多线程总结问题，才会一起收口。

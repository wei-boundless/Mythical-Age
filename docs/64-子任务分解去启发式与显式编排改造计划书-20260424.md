> 本版不回退到旧的词法拆分路线，而是在既有原则下，补齐缺失的中间执行层。

## 1. 结论

原 `64` 的方向只有一半是对的。

对的部分：

1. 不再允许 `compound_query.py` 用“先 / 再 / 然后 / 最后 / 搜索 / 查”之类弱语言线索直接决定任务拓扑。
2. 不再让 planner 主链基于自然语言自动 fan-out。
3. 只允许 `explicit_subtasks` 这类显式结构化输入触发真正的多子任务编排。

没完成的部分：

1. 没有补上“复合请求但不应 fan-out”时的稳定执行中间层。
2. 导致 `multi_capability_request` 退化成单个 `bounded_agent`。
3. 该 `bounded_agent` 又被限制在错误的工具集里，开始反复调用 `search_knowledge`。
4. 工具原始输出继续泄漏到可见答案，进一步放大耗时与上下文污染。

因此，原 `64` 不应视为“彻底失败”，但必须视为“只完成了去旧路，没完成建新路”。

本次重写后的核心结论是：

> 我们不恢复自然语言自动 fan-out。  
> 我们新增一个介于 `single_execution` 与 `explicit_fanout` 之间的 `bundle_execution` 层。

这层专门处理：

1. 同一用户句子里出现多个强锚点、多种能力、多类对象。
2. 但这些对象并不应被当场拆成多个独立 planner subtask。
3. 仍然需要稳定地分项执行、分项落 task、分项可 follow-up。

## 2. 当前失败模式

### 2.1 现状链路

当前主链大致是：

1. `understanding/task_understanding.py` 识别 `multi_capability_request`
2. `QuerySubtaskPlanner.plan_structured()` 默认返回单任务
3. `planner.py` 只生成一个 execution
4. `runtime.py` 走 `bounded_agent`
5. `bounded_agent` 按候选工具反复调用 `search_knowledge`
6. `output_boundary.py` / `output_classifier.py` 又把工具回执当可见答案候选

关键代码位置：

1. `backend/understanding/task_understanding.py`
2. `backend/query/subtask_planner.py`
3. `backend/query/planner.py`
4. `backend/query/runtime.py`
5. `backend/tools/search_knowledge_tool.py`
6. `backend/query/output_boundary.py`
7. `backend/query/output_classifier.py`

### 2.2 已观测到的真实退化

以长测场景 `compound-task-decomposition-and-focus-return` 为例：

用户输入：

- `先总结 PDF 第三页，再给我 inventory.xlsx 里最缺货的前三个仓库，最后补一句北京天气。`

实际行为：

1. 没有进入显式 fan-out
2. route 退化成 `agent`
3. execution posture 退化成 `bounded_agent`
4. 候选工具是 `search_knowledge`
5. 模型反复构造：
   - `PDF 第三页 内容`
   - `inventory.xlsx 仓库 缺货 库存`
   - `北京 天气`
   - `inventory.xlsx Rows 61-100 121-160 181-200`
   - `inventory.xlsx SKU-0061 SKU-0081 ...`
6. 多次检索回执原样流入回答
7. 单轮 `query.model_stream` 耗时达到十多分钟

这说明系统当前缺的不是“更聪明的拆分词表”，而是：

> 缺少一个“可结构化但不 fan-out”的执行层。

### 2.3 原 `64` 为什么会放大这个问题

原 `64` 做掉的是：

1. 词法拆分
2. 顺序词触发 fan-out
3. compound 主链对字符串子句的依赖

但原 `64` 没有同步定义：

1. `multi_capability_request` 的新 canonical 执行形态
2. mixed request 的 leaf owner 如何生成
3. mixed request 的 follow-up handle 如何登记
4. 输出层如何阻断原始工具输出

所以旧系统被切掉 fan-out 后，新的真空被 `bounded_agent + search_knowledge` 补位了。

### 2.4 当前代码里的真实断点图

这次额外对当前实现做了逐文件核对，结论是：现在的问题不是某一处小 bug，而是 5 个模块首尾相接，形成了稳定退化链。

#### 断点 A：`task_understanding.py` 先把 mixed request 收敛成 `bounded_agent`

当前代码里，`multi_capability_request` 不会生成多项执行结构，而是继续落进 `_build_bounded_lookup_task()`。

关键事实：

1. `backend/understanding/task_understanding.py` 中 `_detect_mixed_capability_task()` 只负责识别 mixed signals。
2. 真正的收口发生在 `_build_bounded_lookup_task()`。
3. 这里固定给出：
   - `route_hint="agent"`
   - `candidate_tools=["search_knowledge"]`
   - freshness 场景才额外加 `web_search`
   - `execution_posture="bounded_agent"`

也就是说，mixed request 在理解层就已经被降成：

> 一个 agent 路由  
> + 一组候选工具  
> + 没有显式 item 结构

#### 断点 B：`subtask_planner.py` 现在只认识两种输入

`backend/query/subtask_planner.py` 当前只有两条路：

1. 有 `explicit_subtasks` -> 解析成多个 `SubtaskPlan`
2. 否则 -> 返回 `SubtaskPlan.single(normalized)`

这意味着：

1. planner 已经不再用自然语言拆分，这是对的。
2. 但 planner 也还没有新的中间结构来表达 mixed request，这是当前真正缺失的能力。

#### 断点 C：`planner.py` 只实现了“单执行 / 显式多执行”二元结构

`backend/query/planner.py` 中当前实际存在的执行模型只有：

1. `len(subtasks) <= 1` -> 单 execution
2. `len(subtasks) > 1` -> compound executions

因此：

1. mixed request 没有 `explicit_subtasks` 时，只会得到一个 root execution。
2. 这个 root execution 会完整继承上游的 `route=agent` 和 `execution_posture=bounded_agent`。
3. planner 当前没有 `bundle_plan`、`bundle_items`、`execution_mode` 之类中间层表达。

#### 断点 D：`runtime.py` 用 `len(executions)` 决定是否进 compound

`backend/query/runtime.py` 当前的主分流事实非常明确：

1. `len(executions) > 1` 才会进入 `task_coordinator.run_query_tasks(...)`
2. 否则全部走单 execution 主链

这说明：

1. runtime 没有错把 3 项 mixed request 当成单任务。
2. runtime 只是忠实执行了 planner 输出的“只有 1 个 execution”的结果。
3. 所以根因不在 runtime 的 compound 分支判断，而在 planner 之前没有产出中间执行结构。

#### 断点 E：单 execution 一旦是 `bounded_agent`，允许工具集天然偏向 `search_knowledge`

`backend/query/runtime.py` 的 `_allowed_tool_names_for_execution()` 中：

1. `execution_posture == "bounded_agent"` 时，直接取 `query_understanding.candidate_tools`
2. mixed request 上游给的候选工具本来就是 `["search_knowledge"]`
3. freshness 才可能加 `web_search`

这意味着：

1. 单 execution mixed request 的 agent 工具域，默认根本看不到 `pdf_analysis / structured_data_analysis / get_weather`
2. 即使用户句子里同时出现 PDF、xlsx、天气，模型也只会在 `search_knowledge` 附近打转
3. 这正是长测里反复构造检索 query、耗时暴涨的直接原因

#### 断点 F：长测产物已经验证了这条退化链

`output/test_runs/20260424-live-extended-rerun-01/artifacts/compound-task-decomposition-and-focus-return/turn-02-main.json` 中已经能看到：

1. `plan.route = "agent"`
2. `plan.subqueries = 1`
3. session memory 把整句 mixed request 当成一个 `multi_capability_request`
4. 当前活动上下文里只保留了 `inventory.xlsx` 的 dataset 影子，没有形成 PDF / dataset / weather 三项稳定执行对象

所以当前系统的真实问题可以收口成一句话：

> 旧的词法 fan-out 被正确移除了，  
> 但新的中间执行结构没有补上；  
> 于是 mixed request 被理解层直接压成一个 `bounded_agent(search_knowledge)`。

### 2.5 当前 task / follow-up / 测试契约也还接不住 `bundle`

继续核对当前代码后，还需要再明确一件事：

> 即使 planner 现在立刻产出 `bundle_execution`，  
> 下游 task、follow-up、长测契约也还没有准备好接这个新结构。

#### 现状一：`TaskContextRef` 还没有 bundle 维度

当前 `backend/tasks/context_models.py` 的 `TaskContextRef` 只有：

1. `task_id`
2. `parent_query_id`
3. `task_kind`
4. `bindings`
5. `constraints`
6. `status / summary / result_ref_id / owner_scope`

它还没有：

1. `bundle_id`
2. `bundle_item_id`
3. `bundle_item_index`
4. `bundle_origin`

这意味着后续如果用户说：

1. `只展开第二个子任务`
2. `把第一个和第三个压成一句话`

当前 registry 只能把它当旧式 `subtask_index` 去猜，不能把它当 `bundle item handle` 去续接。

#### 现状二：`TaskCoordinator` 只会登记“query subtasks”，不会登记“bundle items”

当前 `backend/tasks/coordinator.py` 的 `run_query_tasks()` 语义仍然是：

1. 输入一组 `executions`
2. 按 offset 生成 `subtask_index`
3. 产出 `TaskRecord`
4. 把 `subtask_plan` 元信息塞进 metadata

也就是说它当前持久化的是：

1. old-style `subtask_plan_id`
2. `subtask_index`
3. query/task/tool 的通用信息

它还没有显式区分：

1. `explicit_fanout` 产生的 subtask
2. `bundle_execution` 产生的 bundle item

如果不先扩这个层，后面 bundle 仍会被误写成旧 compound subtask，follow-up 协议会继续混淆。

#### 现状三：`FollowupResolution` 只有旧协议，没有 bundle 句柄

当前 `backend/query/followup_models.py` 与 `backend/query/followup_resolver.py` 只覆盖：

1. `task_ref`
2. `compound_subset`
3. `binding_ref`

这套协议适合旧的：

1. 单 task 续接
2. compound ordinal subset
3. binding owner 续接

但它不适合新的：

1. `bundle_item_ref`
2. `bundle_subset`
3. `bundle_ref`

因此，现阶段即便 planner 产出 bundle，用户说“第二个子任务”时，resolver 也只会把它继续解释成旧 `compound_subset`。

#### 现状四：长测契约本身还把错误行为当成功标准

当前 `backend/tests/system_eval/long_scenarios.py` 的 compound 场景里，第一轮断言仍然是：

1. `plan.subqueries=1`

这个断言来自旧 `64` 的错误成功标准，也就是：

> 只要不自动拆分，就算修好了

现在它已经不再成立了。

新的成功标准应改成：

1. 不是 `plan.subqueries=1`
2. 而是 `execution_mode=bundle`
3. 或至少 `bundle_item_count=3`
4. 且后续 follow-up 能命中 `bundle_item_ref / bundle_subset`

所以，当前多任务改造真正要替换的，不只是代码，还有测试契约。

## 3. 本项目必须遵守的原则

这次 `64` 的重写，不能只参考最近几份“续接/输出边界”文档，还必须显式继承前序架构原则。

因此，本计划的原则来源明确绑定到以下文档：

1. `docs/26-OpenClaw-架构改造计划.md`
2. `docs/28-上下文隔离逐文件改造清单.md`
3. `docs/34-Claude式Follow-up句柄续接修复计划书.md`
4. `docs/35-Claude式输出边界去泄露重构计划书.md`
5. `docs/36-Claude式统一续接主链修复计划书.md`
6. `docs/37-输出主链与Canonical Answer收口计划书.md`
7. `docs/38-去启发式续接与显式句柄主链重构技术报告及计划书.md`

这些文档对本问题形成的硬约束如下。

### 3.1 来自 `26` 的架构边界原则

`26` 明确要求：

1. 运行时边界要先清，再继续叠加能力。
2. query loop 与 orchestration 需要分离。
3. 任务层应是显式层，而不是让主链在自然语言里临时猜。

落到本问题上，就是：

1. `multi_capability_request` 不能继续塞回单个 `bounded_agent` 硬撑。
2. 多任务结构必须先落成显式执行对象，再交给 runtime。
3. planner 不能靠 prompt 或词表临场脑补出“伪任务层”。

### 3.2 来自 `28` 的上下文隔离原则

`28` 明确要求：

1. compound 项目需要 task-local 隔离。
2. 主线程只消费 `summary / context_ref / result_ref`。
3. raw evidence 和 raw tool output 不得回流主线程。
4. compound 最终答案应由主线程集中组装，而不是直接沿用子链路原文。

落到本问题上，就是：

1. `bundle_execution` 内各 item 必须是局部 execution，不共享 live mutable context。
2. bundle follow-up 只能续接 item handle，不能重放原句和原工具输出。
3. 主线程看到的是 `BundleItemSummaryRef`，不是 item 的 raw output。

### 3.3 来自 `34/36/38` 的句柄优先原则

这三份文档共同约束：

1. 恢复层的职责是恢复，不是裁决。
2. 默认隔离，显式共享。
3. 生命周期必须有稳定 owner。
4. direct tool、compound follow-up、bundle follow-up 最终都要落到真实 task handle。
5. 弱语言信号只能提示 follow-up 倾向或 subset 倾向，不能直接决定执行拓扑。
6. planner 不能把理解责任甩给下游执行单元。

落到本问题上，就是：

1. 不能恢复旧的自然语言自动 fan-out。
2. 不能让 `bundle_subset` 重新规划整句。
3. 不能让 `binding_ref` 或历史恢复态越权决定当前 mixed request 的执行拓扑。

### 3.4 来自 `35/37` 的输出主链原则

这两份文档共同约束：

1. 输出层优先消费 summary/ref，不直接消费 raw tool output。
2. raw protocol、tool chatter、过程文本不能成为 canonical answer。
3. 用户可见答案、session assistant content、后续 memory 投影都应只消费 canonical truth。

落到本问题上，就是：

1. `search_knowledge` 原始回执不能再直接成为 mixed request 的最终答案。
2. bundle 执行即便包含检索项，也必须先收口成 item summary，再进入主答案。
3. `done.content`、session history、follow-up truth 都不能继续吃 raw search dump。

因此，本次重写禁止以下回退：

1. 恢复 `split_compound_query()` 主路径调用
2. 让“先 / 再 / 然后 / 最后”再次直接触发 fan-out
3. 用 prompt 补丁强迫模型“自己学会多任务处理”
4. 继续允许 `search_knowledge` 原始输出进入最终答案

## 4. Claude Code 参考结论

本次对照的本地参考实现主要来自：

1. `d:\AI应用\claude-code-nb-main\tools\AgentTool\AgentTool.tsx`
2. `d:\AI应用\claude-code-nb-main\tools\SendMessageTool\SendMessageTool.ts`
3. `d:\AI应用\claude-code-nb-main\coordinator\coordinatorMode.ts`
4. `d:\AI应用\claude-code-nb-main\tools\TaskCreateTool\TaskCreateTool.ts`
5. `d:\AI应用\claude-code-nb-main\utils\tasks.ts`
6. `d:\AI应用\claude-code-nb-main\tools\TaskOutputTool\TaskOutputTool.tsx`

### 4.1 应借鉴的点

Claude Code 在这类问题上有三个很清楚的分层：

1. 任务列表是显式对象  
   `TaskCreateTool` / `TaskGetTool` / `utils/tasks.ts` 中的 task 是独立结构，不是从自然语言里临时猜出来的。

2. 委派是显式动作  
   `AgentTool` 只有在 coordinator 明确决定时才启动 worker，不是看到一句自然语言里有多个目标就自动起 worker。

3. 继续旧执行靠 handle  
   `SendMessageTool` 用 agent id/name 继续既有 worker，follow-up 不是重新改写 prompt 再猜。

4. worker 输出不是主线程真相  
   coordinator prompt 明确要求不要预测 worker 结果；结果以 `<task-notification>` 形式进入，随后再由主线程总结给用户。

5. 任务对象与执行对象分离  
   任务列表、后台任务、worker 执行、输出文件都各有稳定句柄，不混在自然语言中。

### 4.2 不应直接照搬的点

Claude Code 的多 worker / team / background task 机制不应直接搬进来当本轮主解，因为：

1. 你的项目当前问题不是“缺后台 worker 调度”
2. 也不是“缺 agent swarms”
3. 而是“单轮 mixed request 没有中间执行层”

因此，本轮只借鉴它的两个不变量：

1. 多任务结构必须是显式对象
2. 继续既有执行必须靠稳定 handle

不借鉴的内容：

1. 自动起 worker 做多任务
2. 把 mixed request 全部变成 agent team
3. 用模型临场 planner JSON 来决定任务编排

## 5. 新的目标设计方向

### 5.1 从二元执行模型改成三元执行模型

当前实际上只有两种：

1. `single_execution`
2. `explicit_fanout`

应改成三种：

1. `single_execution`
2. `bundle_execution`
3. `explicit_fanout`

三者定义如下。

#### A. `single_execution`

适用：

1. 单对象单能力请求
2. 同源连续 follow-up
3. 明确 direct tool / rag / memory 请求

例子：

1. `第四页讲了什么`
2. `回到刚才 inventory.xlsx，按仓库汇总`
3. `北京天气`

#### B. `bundle_execution`

适用：

1. 同一轮里出现 2 个及以上强锚点
2. 这些锚点属于不同能力或不同对象
3. 但用户没有提供 `explicit_subtasks`
4. 因此不应启动真正的 fan-out / subtask coordinator

例子：

1. `先总结 PDF 第三页，再给我 inventory.xlsx 最缺货的前三个仓库，最后补一句北京天气`
2. `把这份报告第三页讲清楚，然后回到销售表给我 top3，最后顺手报一下黄金价格`

bundle execution 的含义不是“重新允许启发式子任务分解”，而是：

> 在 planner 内部形成一个强类型的 `BundlePlan`，并按固定顺序编译成多个 leaf execution。

这些 leaf execution：

1. 可以顺序执行
2. 可以各自产生真实 `TaskRecord`
3. 可以被 follow-up subset 选中
4. 但它们不等于旧的自然语言 compound fan-out

#### C. `explicit_fanout`

适用：

1. UI/API 明确传入 `explicit_subtasks`
2. 未来若有显式批处理入口、任务板入口，也只在显式结构基础上触发

这类才允许真正的多子任务编排。

### 5.2 新的核心边界

新的主边界应是：

1. 自然语言不能直接决定“是否有多个独立 planner subtask”
2. 但自然语言中的强锚点可以决定“是否需要构造一个 bundle”
3. bundle 是单轮执行结构，不是 worker fan-out 结构
4. follow-up 续接的是 bundle item handle / leaf task handle，不是重拆原句

## 6. 目标架构

### 6.1 新增 `BundlePlan`

建议新增：

```python
@dataclass(slots=True)
class BundleItemPlan:
    item_id: str
    index: int
    goal: str
    user_visible_title: str
    source_kind: str
    capability: str
    execution_kind: str
    explicit_refs: dict[str, Any] = field(default_factory=dict)
    constraints: dict[str, Any] = field(default_factory=dict)
    followup_aliases: list[str] = field(default_factory=list)

@dataclass(slots=True)
class BundlePlan:
    bundle_id: str
    parent_query_id: str
    items: list[BundleItemPlan] = field(default_factory=list)
    origin: str = "strong_anchor_bundle"
```

### 6.2 `BundlePlan` 的产生规则

允许构造 bundle 的信号只能来自强锚点，不来自弱顺序词。

允许的强锚点示例：

1. 显式 PDF 引用或强页级 PDF 引用
2. 显式 dataset 文件引用，如 `.xlsx/.csv/.json`
3. 明确 realtime capability，如天气、金价
4. 明确 knowledge capability，如“基于本地知识库总结某报告”

不允许作为 bundle 触发依据的信号：

1. `先 / 再 / 然后 / 最后`
2. `搜 / 查 / 看看`
3. 多个逗号
4. 多个名词并列

### 6.3 `BundlePlan` 与 `explicit_subtasks` 的区别

`explicit_subtasks`：

1. 是上游显式给出的执行拓扑
2. planner 直接尊重
3. 允许依赖、owner、refs、constraints 全显式

`BundlePlan`：

1. 是 planner 基于强类型能力锚点生成的单轮执行容器
2. 不表达并发委派语义
3. 不表达复杂依赖图
4. 只是“把一个 mixed request 安全拆成若干 leaf execution”

### 6.4 leaf execution 的编译规则

每个 bundle item 必须编译成一个确定性 leaf execution，而不是 `bounded_agent` 猜工具。

规则示例：

1. PDF item -> `pdf_analysis`
2. dataset item -> `structured_data_analysis`
3. weather item -> `get_weather`
4. gold item -> `get_gold_price`
5. knowledge item -> `search_knowledge`

只有当某个 item 没有可执行 direct route 时，才允许它退到受控 fallback。

但受控 fallback 不能是：

1. 整个 bundle 退成一个 `bounded_agent`
2. 把所有 item 都塞给 `search_knowledge`

### 6.5 follow-up 续接规则

follow-up 需要新增一层：

1. `bundle_ref`
2. `bundle_item_ref`
3. `bundle_subset`

处理方式：

1. `只展开第二个子任务`
   - 解析到 `bundle_item_ref(item_index=2)`
   - 直接定位上轮 bundle 第二项对应的 `task_id`
   - 不重跑 planner 全局理解

2. `把第一个和第三个子任务各压成一句话`
   - 解析到 `bundle_subset(item_ids=[1,3])`
   - 直接按 `TaskSummaryRef` 重组
   - 不重跑原工具链

这与 `docs/34` 中 `compound_subset` 的原则保持一致，但新的句柄对象不再依赖旧的自动 fan-out。

### 6.6 控制面与输出层

主线程只保留：

1. `active_task_id`
2. `active_binding_ref`
3. `active_bundle_id`
4. `active_bundle_item_ids`
5. `followup_mode`

主线程不保留：

1. 原始工具输出
2. “我刚才先查了这个再查了那个”的 prose
3. bundle 内部中间推理文本

输出层只允许消费：

1. `TaskSummaryRef`
2. `TaskResultRef`
3. `BundleItemSummaryRef`

不允许直接消费：

1. `search_knowledge` 原始回执
2. 原始 tool output token stream
3. synthetic fallback prose 作为 canonical answer

## 7. 需要明确借鉴与不借鉴的设计决策

### 7.1 借鉴 Claude Code 的内容

1. 任务对象与执行对象显式分离
2. 继续旧执行必须靠稳定 handle
3. 主线程必须先综合理解，再决定后续动作
4. 下游执行单元不能承担上游理解责任
5. worker/result 不能直接变成用户可见主答案

### 7.2 不借鉴 Claude Code 的内容

1. 不把 mixed request 默认转成 worker delegation
2. 不默认引入 team/teammate/background agent 语义
3. 不用模型 planner JSON 决定是否拆任务
4. 不让模型自己决定 leaf execution 的拓扑

### 7.3 适配本项目后的最终选择

推荐路线：

> Claude Code 式显式任务句柄  
> + 现有 docs 的 owner-first / summary-first 原则  
> + 新增 bundle execution 中间层  
> + 保留 explicit_subtasks 作为唯一真正 fan-out 入口

## 8. 具体执行计划

### 8.0 固定实施顺序

这轮实现不能按“哪块最显眼先改哪块”推进，顺序必须锁死。

正确顺序是：

1. 先改数据模型
2. 再改 planner 编译
3. 再改 task 持久化
4. 再改 follow-up 协议
5. 最后改 runtime 分发与长测断言

原因如下。

#### 8.0.1 不能先改 runtime

如果先改 runtime：

1. runtime 会缺少 `bundle_plan / bundle_items / bundle handles`
2. 只能自己临场拼结构
3. 这会重新把执行拓扑判断拉回运行时启发式

这与本次“显式结构优先”的目标相反。

#### 8.0.2 不能先改 follow-up

如果先改 follow-up：

1. resolver 可以新增 `bundle_item_ref` 名字
2. 但 registry 里还没有 bundle item 真句柄
3. 结果还是会退回旧 `subtask_index` 或 fresh planning

这会形成“协议名升级了，真相源没升级”的伪重构。

#### 8.0.3 不能先改长测断言

如果先把长测从 `plan.subqueries=1` 改掉：

1. 当前代码会立刻全红
2. 但我们还不知道红的是数据结构缺失、planner 缺失还是 follow-up 缺失

所以测试契约要在结构层与 planner 层落稳之后再切换。

### Phase 0：冻结旧 64 的错误成功标准

目标：

1. 明确不再以“禁掉自动 fan-out”作为唯一完成标准
2. 明确 mixed request 的质量指标

动作：

1. 更新 `docs/64`
2. 更新长场景对 `compound-task-decomposition-and-focus-return` 的期望语义
3. 增加针对 `multi_capability_request` 的性能与语义断言

完成标准：

1. 文档不再把 `subqueries=1` 直接等同于收口成功
2. 新验收项中出现：
   - 不退化为单个 `bounded_agent`
   - 不出现长检索回执直吐
   - follow-up 能选择 bundle item

### Phase 1：引入 bundle 数据模型

目标：

1. 新增 `BundlePlan / BundleItemPlan`
2. QueryPlan 能同时表达：
   - single
   - bundle
   - explicit_fanout

涉及文件：

1. `backend/query/models.py`

完成标准：

1. `QueryPlan` 能携带 `bundle_plan`
2. `TaskContextRef` 能记录 `bundle_id / bundle_item_id / bundle_item_index`
3. 不破坏现有 single 与 explicit_subtasks 路径

### Phase 2：实现强锚点 bundle 编译器

目标：

1. 从 `multi_capability_request` 中产出 bundle，而不是 `bounded_agent`
2. 只使用强锚点，不恢复顺序词 fan-out

涉及文件：

1. `backend/understanding/task_understanding.py`
2. 新增 `backend/query/bundle_planner.py`
3. `backend/query/planner.py`

规则：

1. 2 个及以上强锚点 -> bundle 候选
2. 没有强锚点 -> 不建 bundle
3. 同源单对象多动作但能 direct tool 处理 -> 仍走 single

完成标准：

1. `先总结 PDF 第三页，再给我 inventory.xlsx 最缺货的前三个仓库，最后补一句北京天气`
   - 不再生成 `bounded_agent`
   - 生成 3 个 bundle item
2. `先按仓库汇总，再按部门排序，最后给我缺货前五`
   - 因为同源 dataset，可仍保持 single direct tool

### Phase 3：runtime 落地 bundle execution

目标：

1. bundle item 编译成多个 leaf execution
2. 每个 leaf execution 都落真实 task
3. 不使用统一 `bounded_agent` 打包执行

涉及文件：

1. `backend/query/runtime.py`
2. `backend/tasks/coordinator.py`
3. `backend/query/tool_input_resolver.py`

要求：

1. PDF item 用 `pdf_analysis`
2. dataset item 用 `structured_data_analysis`

4. knowledge item 才允许 `search_knowledge`
5. item 失败时返回 item 级 failure，不拖垮整个 bundle 结构

完成标准：

1. bundle 内每一项都有真实 `task_id`
2. `TaskSummaryRef.task_id` 可回查
3. 主线程可见的是 item summary，不是 raw output

### Phase 4：follow-up 增加 bundle handle 协议

目标：

1. 让 `第二个 / 第一个和第三个` 续接 bundle item，而不是重拆文本

涉及文件：

2. `backend/query/runtime.py`

新增模式建议：

1. `bundle_item_ref`
2. `bundle_subset`

完成标准：

1. `只展开第二个子任务` 直接命中第二项 `task_id`
2. `把第一个和第三个子任务各压成一句话` 直接做 summary subset assembly
3. 不重跑原 bundle 的 leaf tools

### Phase 5：输出层切断 raw search leakage

目标：

1. `search_knowledge` 原始输出永远不能成为 canonical answer

涉及文件：

1. `backend/query/output_boundary.py`
2. `backend/query/output_classifier.py`
3. `backend/query/runtime.py`

要求：

1. tool message token 不直接进入可见流
2. `search_knowledge` 输出只能作为 evidence/result channel
3. 无 summary 时返回受控 fallback，不直吐回执

完成标准：

1. mixed request 不再出现 `[1] knowledge/...` 大段回执作为答案正文
2. `answer_source` 以 summary/ref 为主

### Phase 6：测试与长场景重跑

目标：

1. 把新模型固化成 regression

新增或修改测试：

1. `backend/tests/query_planner_regression.py`
2. `backend/tests/subtask_planner_contract_regression.py`
3. `backend/tests/query_runtime_route_guard_regression.py`
4. 新增 `backend/tests/bundle_execution_regression.py`
5. 新增 `backend/tests/bundle_followup_regression.py`
6. 修改 `backend/tests/system_eval/long_scenarios.py`

验收矩阵：

1. 强锚点 mixed request -> `bundle_execution`
2. 单源多动作 -> `single_execution`
3. `explicit_subtasks` -> `explicit_fanout`
4. `bundle_subset` 不重跑工具

1. 直接在 `runtime.py` 里遇到 mixed request 就手写三个 execution
2. 直接在 `followup_resolver.py` 用 ordinal 词把句子重写成新 prompt

这些捷径都会重新引入：

1. 运行时裁决
2. 真相源漂移

## 10. 迁移、切换与回滚规则

### 10.1 迁移方式

建议采用：

1. `shadow bundle planning`
2. `feature-flag cutover`

步骤：

1. 先只在 plan artifact 中产出 bundle，不接 runtime
2. 记录 `single` 与 `bundle` 的差异
3. 通过回归后再切 runtime 到 bundle path

### 10.2 切换规则

引入开关：

1. `ENABLE_BUNDLE_EXECUTION`

阶段：

1. `off`
   - 保持当前逻辑
   - 但 shadow 记录 bundle candidates
2. `shadow`
   - planner 生成 bundle_plan
   - runtime 仍执行旧路径
3. `on`
   - mixed capability 切到 bundle path

### 10.3 回滚规则

如果 bundle path 不稳定，回滚只能退到：

1. mixed request -> single execution

不能回滚到：

1. 恢复自然语言自动 fan-out
2. 恢复 `compound_query.py` 主路径

也就是说，本计划允许回滚执行路径，不允许回滚原则。

## 12. 新版验收标准

本次重写后的 `64` 完成标准不是“默认不自动拆分”，而是以下 8 条同时满足：

1. 自然语言顺序词不再自动触发 planner fan-out。
2. mixed request 存在强锚点时，能进入 `bundle_execution`，不退化为单个 `bounded_agent`。
3. bundle item 全部落真实 `task_id`。
4. `只展开第二个子任务` 能按句柄续接到 bundle item。
5. `第一个和第三个` 能走 summary subset assembly，不重跑工具。
6. `search_knowledge` 原始回执不再出现在用户最终答案中。
7. 主线程控制面不再吸收 bundle 内部 prose 和 raw output。
8. 长场景中 `compound-task-decomposition-and-focus-return` 的耗时与语义都明显改善。

## 13. 收口后的定位

重写后的 `64` 不再是“子任务分解去启发式”单点计划，而是：

> 多任务处理去启发式重构计划  
> 其核心是：  
> 不恢复自动 fan-out，  
> 而是补齐 `bundle_execution` 这一层。

这才是把当前系统从：

- `single_execution`
- `explicit_fanout`

的断裂二元结构，修成真正可用的三段式结构：

- `single_execution`
- `bundle_execution`
- `explicit_fanout`

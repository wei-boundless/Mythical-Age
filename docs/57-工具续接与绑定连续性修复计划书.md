## 1. 计划定位

本计划只处理一个问题域：

`direct tool result / binding owner -> 下一轮 follow-up 仍能稳定续接到正确工具`

它不再处理：

1. 输出层如何把证据收成答案。
2. RAG 收口如何防止协议残片。
3. PDF canonical summary 如何改写成可展示回答。

这些问题已经由 `docs/56` 处理。

本计划只处理：

1. 结构化数据 follow-up 的 owner/binding 连续性。
2. PDF follow-up 的 owner/binding 连续性。
3. session memory 与 task registry 之间的绑定一致性。

---

## 2. 当前现状

长场景 `20260424-025505-long` 的主失败集中在一个场景：

`运营数据与实时信息切换`

失败样本包括：

1. `按仓库汇总前五。`
2. `哪些仓库其实并不缺货？`
3. `按部门汇总这些高薪员工。`
4. `回到刚才那份 PDF，第二部分强调的约束是什么？`

共同现象：

1. 用户上一轮已经执行过 `structured_data_analysis`，并产生了合法的 task owner。
2. 下一轮 follow-up 没有稳定回到 `structured_data_analysis`。
3. 计划层重新退回 `rag`，随后模型输出过程化假答案。
4. memory 汇总或跨源切换后，`session-memory` 里的 `active_dataset / active_pdf` 也可能被清空。

这说明当前系统缺的不是“工具能力”，而是“工具句柄跨轮保持”。

---

## 3. 根因结论

## 3.1 Task Registry 是进程内状态，不是跨轮正式句柄层

`QueryFollowupResolver` 现在主要依赖：

1. `TaskCoordinator.list_tasks(session_id=...)`
2. task 的 `context_ref.bindings`

这条链在单进程、单 runtime 的局部回归里可以工作。

但长场景里，真正跨轮可持续存在的是：

1. session memory
2. persisted task summary / result ref
3. main context projection

而不是内存里的 `TaskCoordinator._tasks`。

所以当前设计的真实问题是：

1. follow-up 解析依赖了短生命周期注册表。
2. 绑定 owner 没有被提升成跨轮正式句柄。

这不符合 docs 的主链原则：跨轮可恢复状态不能只放在进程内缓存。

## 3.2 Session Memory 投影会在跨流场景下清空 binding 槽位

`backend/structured_memory/session_memory.py` 与 `backend/structured_memory/process_engine.py` 当前会根据 flow type 做槽位清理。

当前行为的副作用是：

1. 进入 `external_lookup_flow` 时，会清空 `active_dataset / active_pdf`。
2. 进入 `structured_data_flow` 时，会主动清空 `active_pdf`。
3. 进入 `pdf_analysis_flow` 时，会主动清空 `active_dataset`。
4. memory 汇总阶段如果没有显式 task summary 带出 binding，也会把槽位投影成空。

这会导致系统虽然还保留“文字摘要”，但失去“正式绑定句柄”。

结果就是：

1. 用户说“回到刚才那个表”时，主链只剩模糊文字，没有 owner。
2. 用户说“回到刚才那份 PDF”时，系统只能识别任务类型，拿不到 path。

## 3.3 Planner 的 continuation promotion 只认显式路径或临时 authority，不认正式会话绑定

`QueryContinuationResolver` 现在能做的主要是：

1. 显式文件路径提升
2. compound authority 提升

它缺少一条正式能力：

`session-level committed binding -> planner promotion`

也就是说，当前 planner 不会把 session memory 中已经确认过的 `active_dataset / active_pdf` 当成正式 binding 输入。

这会直接导致：

1. follow-up 一旦没走 `binding_ref` 快捷通道，就退回普通理解层。
2. 理解层又因为没有显式路径，只能掉回 `rag` 或 general route。

## 3.4 当前是双来源状态，但没有主从规则

现在至少有两套绑定来源：

1. task registry 中的 `context_ref.bindings`
2. session memory 中的 `context_slots`

它们目前没有清晰主从关系。

实际结果是：

1. 有时 task 里有 binding，但 planner 看不到。
2. 有时 session 里有摘要，但 contract gate 看不到 owner。
3. memory 汇总后只保留自然语言结论，不保留 binding identity。

这违反了 docs 的边界原则：

1. 主链状态必须有单一正式来源。
2. 绑定句柄必须是结构化对象，不是自然语言残影。

---

## 4. 对照原则后的正式判断

根据当前 docs 原则，当前问题不是理解层“不够聪明”，而是：

1. 绑定句柄没有正式晋升为跨轮状态。
2. 规划层没有把 binding 当成第一等输入。
3. memory 投影把对象句柄错误降级成了文字摘要。

所以这轮不能再做以下错误修法：

1. 给理解层加更多“按仓库/按部门/回到刚才”词表启发式。
2. 继续放宽 `tool contract`，让没有 owner 的工具硬跑。
3. 靠 memory 摘要里的自然语言去猜文件对象。

正式方向应该是：

1. 先把 binding owner 做成正式、可恢复、可仲裁的状态对象。
2. 再让 follow-up / planner / contract gate 共同读取这一个对象。

---

## 5. 目标架构

本阶段目标链路应调整为：

`direct tool success -> committed binding handle -> session-level binding state -> follow-up resolution -> planner/tool promotion -> contract-safe execution`

需要新增或强化的结构职责：

1. `CommittedBindingState`
2. `BindingContinuationResolver`
3. `BindingProjectionPolicy`

### A. CommittedBindingState

职责：

1. 记录最近一个已确认可续接的 `active_dataset / active_pdf`。
2. 记录 `binding_identity / binding_owner_task_id / source_kind / task_kind`。
3. 只接受 direct tool 成功结果或明确 owner 结果写入。

### B. BindingContinuationResolver

职责：

1. 在 `TaskCoordinator` 之外，优先从 committed binding state 恢复 owner。
2. 将“回到刚才那个表 / 这份 PDF / 按部门汇总这些高薪员工”还原为正式 tool continuation。
3. 与 planner 对接，而不是只停留在 follow-up resolver 的模糊判断层。

### C. BindingProjectionPolicy

职责：

1. 规定什么情况下可以清空 binding。
2. 规定 memory 汇总、外部查询、跨源总结时如何保留 last committed binding。
3. 禁止把 binding 只降级成自然语言摘要。

## 6. 重构路线

## Phase 1：绑定状态正式化

目标：

1. 从 direct tool 成功结果中正式写入 committed binding。
2. 不再只依赖 `TaskCoordinator._tasks` 作为唯一 owner 来源。

涉及文件：

1. `backend/query/runtime.py`
2. `backend/tasks/coordinator.py`
3. `backend/query/context_models.py`
4. `backend/structured_memory/session_memory.py`

完成标准：

1. `structured_data_analysis` 成功后，session state 中有稳定 `active_dataset`。
2. `pdf_analysis` 成功后，session state 中有稳定 `active_pdf`。

## Phase 2：planner 接入 committed binding

目标：

1. planner 在没有显式路径时，也能读取 session-level binding。
2. structured/pdf follow-up 不再轻易掉回 `rag`。

涉及文件：

1. `backend/query/planner.py`
2. `backend/query/continuation_resolver.py`
3. `backend/query/tool_input_resolver.py`

完成标准：

1. `按仓库汇总前五。` 自动进入 `structured_data_analysis`。
2. `按部门汇总这些高薪员工。` 自动进入 `structured_data_analysis`。
3. `回到刚才那份 PDF...` 在有 owner 时自动进入 `pdf_analysis`。

## Phase 3：binding 清理策略收紧

目标：

1. memory summary、external lookup、cross-source synthesis 不再无条件清空 binding。
2. 区分“当前 active flow”与“最近 committed binding”。

涉及文件：

1. `backend/structured_memory/session_memory.py`
2. `backend/structured_memory/process_engine.py`
3. `backend/memory/context.py`

完成标准：

1. 汇总回答后仍能回到前一轮明确绑定过的数据表或 PDF。
2. 只有明确切换对象或显式清空时才丢 binding。

## Phase 4：回归与长场景验收

目标：

1. 增加跨轮 binding continuity 回归。
2. 重跑长场景验证 route/tool continuity。

涉及文件：

1. `backend/tests/query_runtime_route_guard_regression.py`
2. 如有必要新增：`backend/tests/query_binding_continuity_regression.py`

完成标准：

1. `commerce-ops-data-live-switch` 不再因为表格 follow-up 掉回 `rag`。
2. `tool_contract_blocked` 只在确实没有 owner 时出现。

---

## 7. 本轮禁止事项

本计划明确禁止：

1. 继续给理解层追加大量 follow-up 关键词启发式。
2. 继续在 `output_boundary` 上兜底表格/PDF route 错误。
3. 通过放松 contract gate 来掩盖 binding 丢失。
4. 用自然语言摘要代替结构化 binding 句柄。

---

## 8. 当前结论

现在的主问题已经不再是 `docs/56` 的输出收口。

准确地说：

1. `docs/56` 解决了“执行完以后怎么稳定说出来”。
2. `docs/57` 要解决“下一轮还能不能回到刚才那个对象继续执行”。

如果这一层不修，系统就会持续出现一种假稳定：

1. 第一轮 direct tool 是对的。
2. 第二轮 follow-up 失去 owner，退回 `rag`。
3. 输出层再怎么收口，也只能收一个错误路由后的半成品答案。

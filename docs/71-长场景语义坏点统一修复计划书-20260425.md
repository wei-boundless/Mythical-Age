# 71-长场景语义坏点统一修复计划书-20260425

> 目的：针对 `20260425-sixty-turn-rerun-04` 长场景复测中暴露出的 `PDF follow-up 丢句柄`、`结构化结果集范围漂移`、`混合证据被单源 authority 劫持`、`旧 direct_tool 回退链残留` 等问题，基于当前代码真实状态与既有 docs 原则，制定一份可直接驱动后续实施的统一修复计划。  
> 本计划不再接受“按具体语义 bug 加一层关键词判断”的做法，目标是修复协议与所有权结构，而不是继续做路由补丁。

---

## 0. 本轮结论

这轮长场景暴露的核心问题不是“某几个回答答偏了”，而是下面三层没有统一：

1. **对象层**
   - 同一个业务对象在系统里同时以 `active_pdf`、`active_dataset`、`task summary prose`、`retrieval snippet`、`tool_input.path` 等多种形式存在。
   - 但系统没有一套稳定的 `object handle / result handle / subset handle`。

2. **执行层**
   - `worker` 主链和 `direct_tool` 旧链仍然并存。
   - 首轮可能走 `worker=pdf`，follow-up 却掉回 `direct_tool=pdf_analysis`。

3. **控制层**
   - `restore`、`decide`、`execute`、`present` 四层仍有越权。
   - authority context 会重新裁决当前轮；
   - answer assembler 只能吃 task summary，不理解 typed diagnostics；
   - 结构化 follow-up 只能恢复 dataset path，不能恢复结果子集。

因此，本轮推荐方向不是继续修一批 turn case，而是：

> 用统一的 `Execution Handle + Evidence/Object Handle + Result/Subet Handle` 协议，把 `RAG / PDF / 结构化 / follow-up / output assembly` 收进同一条主链；旧 direct-tool 只保留低层 facade，不再承担 runtime 主链语义。

### 2.1 硬约束

1. **默认隔离，显式共享**
   - task-local truth 默认留在 task / worker / evidence 层。
   - 主线程只能通过显式 handle 消费。

2. **restore 不等于 decide**
   - session snapshot / authoritative context 只能恢复候选，不得越权裁决当前轮路由。

3. **主线程只保留控制面真相**
   - 主线程可以保留：当前 task、当前 selected tasks、当前 object binding、当前 output constraints。
   - 主线程禁止保留：raw evidence、task-local prose、worker 中间态、旧 answer 文本拼装后的伪真相。

4. **summary-first，ref-second，raw-last**
   - 回答优先消费 canonical summary。
   - 必要时通过 result/evidence handle 回查。
   - raw output 默认 debug-only。

5. **fail-closed**
   - 新增 handle 未建立完成时，宁可澄清或稳定降级，也不能回退到关键词猜测或 source authority 强提升。

6. **结构修复优先于 prompt 修补**
   - 不允许再通过“识别到某句式就强制走 PDF / 强制走 dataset”的方式亡羊补牢。

## 3. 系统问题重述

### 3.1 当前真正损坏的系统属性

当前损坏的不是单轮 intent 识别本身，而是：

- **对象连续性**：上一轮得到的对象、结果、子集，在下一轮不能稳定续接。
- **主链一致性**：同类请求在 first turn 与 follow-up turn 走的是不同执行协议。
- **多源仲裁**：系统缺少“已发现对象 + 当前轮目标 + 可用证据”的统一仲裁层。
- **输出真实性**：present 层缺少 typed degradation reason，只能给模板化空答。

### 3.2 正确终态

正确终态应满足：

1. 用户第一次拿到任何可续接结果时，系统都能生成稳定 handle。
2. 后续追问优先沿 handle 续接，而不是重新猜“你是不是还在问 PDF”。
3. RAG / PDF / 结构化都用同一套 object/result 协议交接。
4. 混合请求先做证据仲裁，再决定是否补单源 worker。
5. 旧 `direct_tool` 不再承担 runtime 主语义，只作为底层 facade 或测试入口存在。

## 4. 推荐设计方向

### 4.1 核心方向

在当前代码基础上，不重启炉灶，直接补齐并统一这三类 handle：

1. `ExecutionHandle`
   - 表示一次真实可续接执行。
   - 唯一真相源仍是 `TaskRecord.task_id`。

2. `ObjectHandle`
   - 表示系统当前已确认的源对象或派生产物。
   - 例如：`pdf source object`、`dataset source object`、`pdf section artifact`、`table artifact`。

3. `ResultHandle`
   - 表示某次执行得到的稳定结果或结果子集。
   - 例如：`top5 employees subset`、`缺货前三仓库 subset`、`第二部分定位结果`。

### 4.2 当前项目最合适的落点

结合当前已有结构，最适合的实现方式不是“再加一个 planner 模型临场拆分器”，而是：

```text
TaskUnderstanding
  -> QueryPlanner
  -> QueryExecutionPlan
  -> EvidenceOrchestrator / WorkerExecutionPlan
  -> CanonicalResult + ObjectHandle + ResultHandle
  -> WorkerProjection / FollowupResolution
  -> AnswerAssembler / OutputPolicy
```

也就是说：

- 理解层只负责识别“当前要做什么类型的工作”。
- 仲裁层负责决定“用哪个对象、哪个结果、是否需要更多证据”。
- 执行层只执行，不裁决。
- 输出层只消费 canonical object/result，不再从 prose/history 反推。

## 5. 目标架构

### 5.1 统一对象协议

建议新增或补齐以下 typed 对象：

- `SourceObjectRef`
  - 已在 evidence 系统中存在，应成为 PDF / dataset / workspace file / retrieval hit 的统一对象入口。

- `ArtifactHandle`
  - 表示从 source object 派生出的可处理对象。
  - 例如：
    - `pdf_page`
    - `pdf_section_locator`
    - `table_candidate`
    - `dataset_schema`

- `ResultHandle`
  - 本轮重点新增。
  - 表示“执行后的稳定结果对象”，而不是原始源对象。
  - 例如：
    - `structured_subset`
    - `grouped_summary`
    - `pdf_section_summary`
    - `mixed_synthesis_basis`

### 5.2 follow-up 统一协议

`FollowupResolution` 必须从“猜测用户在说谁”升级为“引用已存在 handle”：

- `mode="task_ref"`：按 execution handle 续接。
- `mode="object_ref"`：按 object handle 续接。
- `mode="result_ref"`：按 result handle 续接。
- `mode="subset_ref"`：按 subset result 续接。
- `mode="clarify"`：仅当多个 handle 同时存在且无法稳定选择时触发。

禁止保留现状中的弱语义越权：

- 不允许 `active_pdf` 单独触发当前轮路由裁决。
- 不允许 authority context 在 mixed-source 请求中直接改写当前轮 route。

### 5.3 worker 投影协议

每个 worker `done` 事件必须最少产出：

- `task_id`
- `main_context` 的控制面投影
- `task_summary_refs`
- `object_handles`
- `result_handles`
- `binding_owner`
- `degraded_reason_typed`

也就是说，`WorkerProjectionAdapter` 不能再只投影 binding identity 和摘要，而必须把本次结果的可续接句柄投上来。

### 5.4 结构化结果集续接协议

结构化数据必须支持“结果子集续接”，而不是只支持“数据集路径续接”。

新增能力：

- `StructuredSubsetRef`
  - 包含：
    - `dataset_path`
    - `origin_task_id`
    - `filter_signature`
    - `selected_row_ids / stable row refs`
    - `projection_columns`
    - `ordering`

后续像“这些人按部门汇总”时，系统消费的是 `StructuredSubsetRef`，不是整表 path。

### 5.5 mixed-source 仲裁协议

对“结合知识库风险和 PDF 结论”这类请求：

1. 若已有多个 source/result handle，先进入 `evidence arbitration`。
2. 仲裁层决定：
   - 直接综合已有 evidence
   - 补充某一侧证据
   - 还是要求澄清
3. 不允许 authority context 直接把当前轮压成单源 PDF/结构化工具调用。

### 5.6 输出退化协议

输出层不能只说“没有形成稳定摘要”，而应消费 typed diagnostics，区分：

- `page_has_no_text`
- `ocr_unstable`
- `section_not_located`
- `empty_filtered_result`
- `subset_ref_missing`
- `evidence_insufficient_for_synthesis`

这样用户看到的是“为什么没答出来”，而不是模板空句。

### 5.7 RAG / PDF / 结构化数据统一通信协议

这三条链路不应再被视为“谁都能自由抢路由的并列工具”，而应视为统一证据系统中的不同 worker。

推荐定位：

| 子系统 | 推荐职责 | 是否负责对象发现 | 是否负责深读/分析 | 是否可直接裁决下一跳 |
| --- | --- | --- | --- | --- |
| `RAG / RetrievalWorker` | 发现相关对象与已有证据 | 是 | 否 | 否 |
| `PDFWorker` | 文档阅读、页/节定位、结构抽取 | 否 | 是 | 否 |
| `StructuredDataWorker` | 表格/数据集分析、聚合、筛选、分组 | 否 | 是 | 否 |
| `EvidenceOrchestrator / Arbitration` | 决定是否升级、是否转交、是否综合 | 否 | 否 | 是 |

硬约束：

1. `RAG` 是默认对象发现器，不是所有数据库/文档问题的必经入口。
2. worker 之间通信必须传 `typed handle`，不能传 prose 文本片段。
3. `PDFWorker` 不直接偷偷调用 `StructuredDataWorker`，由 orchestrator 决定下一跳。
4. `StructuredDataWorker` 不因拿到 PDF 文本而自行重检索。

### 5.8 对象与结果通信载体

统一通信对象至少包括：

- `SourceObjectRef`
  - 原始对象。
  - 例如：
    - 本地 PDF
    - 本地 xlsx/csv
    - RAG 命中的知识库文档

- `ArtifactHandle`
  - 从 source object 派生出的可处理对象。
  - 例如：
    - `pdf_page`
    - `pdf_section_locator`
    - `table_candidate`
    - `dataset_schema`

- `ResultHandle`
  - 某次执行产出的稳定结果对象。
  - 例如：
    - `pdf_section_summary`
    - `structured_subset`
    - `grouped_result`
    - `mixed_synthesis_basis`

- `SubsetHandle`
  - `ResultHandle` 的特化，用于表达可续接的结果子集。
  - 例如：
    - `top5_employees_subset`
    - `缺货前三仓库_subset`

worker 间不再传：

- raw retrieval snippet
- raw PDF prose
- “上一轮大概说的是这些人” 这种自然语言描述

worker 间只允许传：

- object / artifact / result / subset handle
- task frame
- constraints
- typed diagnostics

### 5.9 调度规则

推荐的统一调度规则如下：

1. 用户**已经明确给出 dataset**，或当前 session 已有稳定 `dataset/result/subset handle`
   - 直接进入 `StructuredDataWorker`
   - 不先绕到 `RAG`

2. 用户**已经明确给出 PDF**，或当前 session 已有稳定 `pdf/object/result handle`
   - 直接进入 `PDFWorker`
   - 不先走对象发现

3. 用户只说“数据库里 / 本地资料里 / 知识库里 / 文档里”
   - 先进入 `RAG / RetrievalWorker`
   - 目标是发现候选对象，不是直接把 snippet 拼成最终真相

4. `RAG` 命中 PDF
   - 如果已有证据足以稳定回答，可直接综合
   - 如果需要页级/章节级/结构级精读，再转交 `PDFWorker`

5. `RAG` 命中 dataset
   - 如果问题本质是聚合/筛选/排序/分组，转交 `StructuredDataWorker`
   - 不允许继续停留在“用检索文本猜表格结论”

6. `PDFWorker` 识别出表格
   - 产出 `TableArtifactHandle`
   - 由 orchestrator 判断是否转交 `StructuredDataWorker`
   - 不允许把 PDF 表格伪装成普通磁盘 dataset path 后偷偷续接

7. follow-up
   - 优先续接 `ResultHandle / SubsetHandle`
   - 次级续接 `ArtifactHandle / SourceObjectRef`
   - 最后才允许 clarify

### 5.10 PDF 表格与结构化数据的 handoff 规则

这里必须额外锁死一条规则：

> PDF 中识别出的表格，先是 `TableArtifactHandle`，不是 `dataset_path`。

原因：

- PDF 表格是派生产物，不是天然存在于磁盘上的稳定 dataset。
- 如果直接把它伪装成路径型 dataset，会破坏对象所有权、follow-up 续接与结果回溯。

因此 handoff 应为：

```text
PDFWorker
  -> 识别页级/区域级表格
  -> 产出 TableArtifactHandle
  -> Orchestrator 仲裁
  -> StructuredDataWorker 消费 TableArtifactHandle
  -> 输出 Structured ResultHandle
```

而不是：

```text
PDFWorker
  -> 导出一个临时 csv/xlsx
  -> 假装这是普通 dataset
  -> StructuredDataWorker 直接按 path 分析
```

后者只允许作为低层实现细节存在，不能暴露为上层协议。

### 5.11 本地 PDF、知识库 PDF、未来上传 PDF 的统一入口

这三类对象未来必须走同一协议，只是 `SourceObjectRef` 的来源不同：

- 本地路径型对象
- 知识库索引命中的对象
- 用户显式给出的对象

统一后：

- follow-up 不再关心它最初是“检索命中的”还是“显式点名的”
- 未来加上传能力时，不需要重写主链

### 5.12 与记忆系统的边界

证据子系统必须对接记忆系统，但不能把证据系统本身变成记忆污染源。

允许进入 session/memory projection 的内容：

- 当前活跃 `object handle`
- 当前活跃 `result/subset handle`
- 当前 binding owner
- 当前 output constraints

禁止进入 session/memory projection 的内容：

- raw PDF 文本
- raw retrieval snippet
- raw structured table dump
- worker 内部调度解释

### 5.13 协议层与 runtime 的挂接位置

这一点必须明确，否则协议层会成为“文档里有、runtime 实际绕过”的旁路设计。

当前 runtime 的真实分岔点是：

```text
planner.build_plan()
  -> QueryExecutionPlan
  -> _stream_planned_execution()
      -> execution_kind == worker
      -> execution_kind == direct_tool
      -> route == rag
      -> model/bounded_agent
```

因此本轮硬约束如下：

1. `对象发现 / evidence arbitration / 下一跳选择` 必须发生在 `execution_kind` 分岔之前。
2. `EvidenceOrchestrator` 不能只做 worker executor，还必须成为 `worker family` 内部的统一仲裁入口。
3. `direct_tool` 不得继续绕开协议层直接形成新的主语义。
4. follow-up 进入执行前，也必须先过同一套 handle arbitration，而不是单独走旧 binding shortcut。

推荐挂接方式：

```text
TaskUnderstanding
  -> QueryPlanner
  -> Dispatch/Arbitration normalization
      -> 产出统一 QueryExecutionPlan
      -> 明确 execution_kind / worker_plan / object_handle / result_handle
  -> QueryRuntime._stream_planned_execution()
```

也就是说：

- `planner` 或 `planner 之后、runtime 之前` 的规范化阶段，必须把“这轮到底是对象发现、对象深读、结果续接还是综合仲裁”定死。
- `runtime` 只消费已经规范化好的 plan，不再在执行中临场二次发明主路由。

### 5.14 handle 不得作为 prompt prose 泄露

当前 runtime 真实链路里：

- `MainContextState.to_prompt_block()` 会进入 system prompt
- `context.augmented_history` 会进入模型消息
- `session memory projection` 会回灌 session/memory

所以本轮必须额外锁死：

1. `handle` 的存在可以进入 runtime 控制面，但**不能直接以 prose 序列化进入 model-visible system block**。
2. `MainContextState` 只允许出现：
   - 当前激活的 task/object/result 的标识摘要
   - 与本轮决策直接相关的最小控制面字段
3. `MainContextState.active_constraints` 禁止承载：
   - row id 大列表
   - artifact payload
   - table raw dump
   - retrieval snippet
4. `TaskSummaryRef.key_points` 也不得被拿来偷运大体量 handle 内容。

换句话说：

- handle 是 runtime/state 协议对象
- 不是 prompt 文本对象

### 5.15 handle 的存储层与生命周期

如果只定义协议，不定义存储层，这套设计会在 `history compaction / session reload / follow-up` 时立刻失效。

本轮必须明确三层存储：

1. `Task registry`
   - 存稳定 `ExecutionHandle / ResultHandle / SubsetHandle`
   - 是 follow-up 首要真相源

2. `evidence runtime state`
   - 存 `SourceObjectRef / ArtifactHandle / candidate graph`
   - 允许 session runtime state 持有，但不直接变成 prompt prose

3. `session/memory projection`
   - 只存最小控制面索引：
     - 当前活跃 handle id
     - binding owner
     - 最近稳定结果引用

禁止情况：

- 只把 handle 放在内存对象里，不落 task/runtime state
- 只把 handle 写成主线程 prose，不保留结构化索引
- 只把 artifact 放 evidence graph，不建立 task/result owner 映射

### 5.16 子系统职责边界澄清

本轮文档中出现的 `worker`、`子系统`、`子 agent` 必须区分：

1. `RetrievalWorker`
   - 不是自由思考 agent
   - 只做对象发现与证据收集

2. `StructuredDataWorker`
   - 不是通用 agent
   - 只消费 dataset/table handle 做结构化分析

3. `PDFWorker`
   - 对外仍按 capability worker 暴露
   - 即使内部可能调用 `PDFReadAgentRuntime`，也不能对主线程表现为“另一个自治 agent”

4. `EvidenceOrchestrator`
   - 是调度/仲裁层
   - 不是 evidence 生产 worker

因此后续文档与实现中，除非明确指 `PDFReadAgentRuntime` 内部实现，否则统一称：

- `worker`
- `capability worker`
- `orchestrator`

避免把它们写成语义上模糊的“子 agent”，否则会重新引入：

- 可变状态共享边界不清
- prompt 注入边界不清
- 谁能决定下一跳不清

1. `Restore`
   - 只恢复候选 handle 和 session projection。

2. `Decide`
   - 只在 planner / orchestrator / followup resolver 中发生。

3. `Execute`
   - worker / tool facade 只执行，不再裁决主线程语义。

4. `Present`
   - output policy / assembler 只消费 canonical result 与 typed diagnostics。

## 9. Cutover 与 Rollback 规则

### 9.1 Cutover

- `Phase 1` 起引入统一 `execution_protocol` 标记。
- 先 shadow 输出新 handle，不立刻删除旧字段。
- 当回归验证通过后，再让 follow-up 解析优先读新 handle。
- 全量验证通过后，删除旧 direct-tool follow-up 分支。

### 9.2 Rollback

如果任一阶段出现以下情况，允许局部 rollback 到上一阶段：

- follow-up 命中率显著下降
- mixed-source 请求大量退化为 clarify
- structured subset 无法稳定序列化

但 rollback 只允许回退到“保留旧字段、优先读新字段失败后兜底旧字段”的兼容状态，不允许重新引入新的关键词补丁。

## 11. 验证矩阵

### 11.1 必测场景

1. PDF 首轮阅读 -> “回到刚才 PDF” -> 页级追问 -> 章节追问
2. inventory 缺货 -> “前五” -> “这些仓库按地区汇总”
3. employees top5 -> “这些人按部门汇总”
4. “结合知识库风险和 PDF 结论，给我一句话判断”
5. context compact 后继续追问前序对象
6. “数据库里哪些城市缺货” -> RAG 命中 inventory -> 交给 StructuredDataWorker
7. “知识库里这份 PDF 的表格按地区汇总” -> RAG 命中 PDF -> PDFWorker 抽表 -> StructuredDataWorker 分析
8. 本地显式 PDF 提问不走 RAG，对象直接进入 PDFWorker

### 11.2 通过标准

- 不再依赖关键词补丁命中才能通过。
- trace 中能看见稳定 handle 被续接。
- 主线程不再吸收 raw evidence/prose 假真相。
- 同类请求 first-turn 与 follow-up-turn 走同一协议。

## 12. 本轮禁止事项

1. 禁止新增“如果 message 包含 X 就强制走 PDF/structured”的特判。
2. 禁止把 worker 中间文本写进 session projection 作为主线程真相。
3. 禁止用 summary prose 伪装 result subset。
4. 禁止在 mixed-source 问题上让 authority context 直接抢路由。
5. 禁止保留无法解释的旧 direct follow-up 分支作为“先兼容着”。

---

## 13. 推荐实施顺序

建议严格按下面顺序推进：

1. `Phase 0` 先补观测
2. `Phase 1` 统一续接协议
3. `Phase 2` 修结果子集
4. `Phase 3` 收 authority 仲裁
5. `Phase 4` 统一 PDF/RAG 对象
6. `Phase 5` 收输出退化

原因：

- 先补 follow-up owner 与 result handle，才能真正解释后续所有错答。
- 不先修 handle，就去修 mixed-source 或输出，只会继续绕着症状打转。

## 14. 一句话收口

本轮修复的核心不是“让系统更聪明地猜用户在指什么”，而是“让系统不再需要猜”：

> 只要把 `执行句柄`、`对象句柄`、`结果句柄` 统一进主链，长场景里 PDF、RAG、结构化三条能力链的续接、综合和降级，才会真正稳定下来。

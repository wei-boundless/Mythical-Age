# 70-RAG/PDF/结构化数据统一证据子系统协调重构计划书-20260425

> 目的：把当前 RAG、PDF 阅读、结构化数据分析三条能力链从“主 agent 可见工具/文本碎片”重构为“受控证据子系统 + 统一调度仲裁 + canonical 输出投影”的协调架构。  
> 本计划书是初步重构蓝图，后续实施应先按本文件锁定边界，再进入逐阶段代码改造。

---

## 0. 本版结论

当前问题不是某个工具不好用，而是三类高噪声能力没有形成统一的证据边界：

1. RAG 检索结果会以文本形式进入主 agent、history、session-memory。
2. PDF 阅读已经有子 agent 雏形，但 output/projection 仍有专用补丁和降级残留。
3. 结构化数据工具已经有 contract/binding，但 RAG 命中 `.xlsx` 后没有 typed handoff 到结构化数据链。

推荐方向：

```text
User Input
  -> TaskFrame
  -> CapabilityDispatch
  -> Evidence Orchestrator
      -> Retrieval Worker
      -> PDF Worker
      -> Structured Data Worker
  -> EvidenceEnvelope / EvidenceArtifactGraph / CanonicalResult
  -> Binding Arbitration
  -> Answer Finalization
  -> Projection / Session Memory
```

核心原则：

> RAG、PDF、结构化数据都不再作为主 agent 可自由循环调用的普通工具；它们应成为受控 evidence worker / capability worker。主线程只消费 typed evidence、binding candidate 和 canonical result。

补充结论：

> 三者不能只是并列隔离。它们真正需要统一，是因为它们处理的是同一批业务对象的不同模态视图：RAG 搜索向量化后的 PDF/表格/文本对象，PDF 阅读可能派生表格对象，结构化数据分析可能消费 PDF 表格或 RAG 命中的数据集。  
> 因此本轮设计的中心应从“worker 并排协调”升级为“统一证据对象 + 派生产物图 + 受控 worker 协作”。

## 1. 当前真实链路与失败模式

### 1.1 当前 RAG 链路

相关文件：

- `backend/retrieval/service.py`
- `backend/tools/search_knowledge_tool.py`
- `backend/query/runtime.py`
- `backend/query/prompt_builder.py`

当前链路：

```text
route=rag
  -> RetrievalService.retrieve()
  -> retrieval_results
  -> prompt_builder 注入 Retrieval Evidence
  -> 主模型基于证据生成回答
```

同时在 `bounded_agent` 场景中：

```text
candidate_tools = search_knowledge
  -> 主 agent 看到 search_knowledge schema
  -> 模型可多次调用 search_knowledge
  -> raw retrieval snippets 进入 assistant content/tool_calls
```

关键缺口：

1. RAG 返回的是 `list[dict]` 和工具字符串，不是统一 `EvidenceEnvelope`。
2. RAG 命中 `.xlsx/.csv/.pdf` 时，只是文本证据，不会升级为 `DatasetCandidate / DocumentCandidate`。
3. `search_knowledge` 可暴露给 main agent，导致检索循环和 prompt 污染。
4. 检索 evidence、assistant content、tool_calls output、session-memory projection 的边界不硬。

### 1.2 当前 PDF 链路

相关文件：

- `backend/tools/pdf_analysis_tool.py`
- `backend/pdf_agent/*`
- `backend/pdf_analysis/*`
- `backend/query/runtime_tools.py`
- `backend/query/runtime_output_policy.py`

当前状态：

1. `pdf_agent` 已经存在，PDF 阅读链已从纯 tool 向子 agent 方向迁移。
2. `ToolResultEnvelope` 已接入 direct tool 输出。
3. 但 `RuntimeToolBridge` 仍有 PDF 专项 persist/finalize gate。
4. PDF canonical/projection 还没有完全泛化为统一 capability result contract。

关键缺口：

1. PDF worker 的 canonical result 没有和 RAG/结构化数据共享一套 Evidence/Projection 协议。
2. PDF 降级结果仍需要 runtime 专项保护。
3. PDF 结果是否能写入 session-memory 仍散落在 runtime/output policy/persistence gate。

### 1.3 当前结构化数据链路

相关文件：

- `backend/tools/structured_data_analysis_tool.py`
- `backend/structured_data/*`
- `backend/query/binding_resolver.py`
- `backend/query/tool_input_resolver.py`
- `backend/tasks/context_models.py`

当前状态：

1. 结构化数据工具需要 explicit path 或 active binding。
2. `StructuredDatasetBinding` 已进入 planner/execution。
3. 工具 output 已能经 `ToolResultEnvelope` 进入 canonical output。

关键缺口：

1. 结构化数据 binding 只从显式路径/预绑定进入，不接收 RAG 发现的 dataset candidate。
2. “销售数据库”这类澄清选择缺少对上一轮 evidence candidate 的引用。
3. `inventory.xlsx` 被 RAG 发现后没有成为可确认/可绑定对象，只进入了历史文本。

### 1.4 这次缺货场景的根因链

```text
用户: 查询本地数据库哪些城市缺货
  -> route=rag
  -> RAG 命中 inventory.xlsx
  -> 但只作为文本 evidence
  -> assistant 让用户澄清“销售数据库 or 知识库”

用户: 是销售数据库
  -> followup_resolver 无 binding candidate 可接
  -> task_understanding fallback_bounded_lookup
  -> candidate_tools=search_knowledge
  -> 主 agent 反复 search_knowledge
  -> inventory.xlsx 继续以文本片段污染 history/session-memory
```

系统性质缺失：

> RAG 发现能力没有 typed handoff；主调度层没有“证据候选 -> 澄清选择 -> binding commit”的协议。

### 1.5 跨模态交互缺口

RAG、PDF、结构化数据之所以必须统一，不只是因为它们都会污染 prompt，而是因为它们之间天然存在数据流：

```text
PDF 文件
  -> PDF 解析
  -> 文本块 / 页级证据 / 表格区域
  -> 表格对象候选
  -> 可选入库为结构化数据
  -> 可选向量化进入 RAG 索引

Excel/CSV 文件
  -> 结构化 schema / 行列统计 / 聚合摘要
  -> 数据集候选
  -> 可选生成文档化摘要
  -> 可选向量化进入 RAG 索引

RAG 检索
  -> 命中文本块 / PDF 页 / 表格摘要 / 数据集摘要
  -> 返回 typed source object
  -> 由 orchestrator 决定继续走 PDFWorker 还是 StructuredDataWorker
```

当前缺的是一层“证据对象与派生产物”的统一协议。没有这一层时：

1. RAG 只能返回字符串，不能表达“我命中了一个可分析表格”。
2. PDF 只能返回阅读摘要，不能表达“这一页里有一个可结构化分析的表格”。
3. 结构化数据只能消费显式路径或 active binding，不能消费 RAG/PDF 派生出的 `TableArtifact`。
4. prompt builder 只能看到文本片段，无法区分“答案证据”“候选对象”“可继续处理的 artifact”。

## 3. 推荐设计方向

### 3.1 三类能力统一定义

本轮不建议把三条链都变成“能自由思考和调用工具的 agent”。更准确的定义是：

| 子系统 | 推荐定位 | 是否能自由调用工具 | 是否能直接写 session | 对主线程输出 |
| --- | --- | --- | --- | --- |
| RAG | `RetrievalWorker / EvidenceWorker` | 否 | 否 | `EvidenceEnvelope + SourceObject refs` |
| PDF | `PDFWorker / PDFReadAgent` | 受控内部能力 | 否 | `PDFCanonicalResult + Document/Table artifacts` |
| 结构化数据 | `StructuredDataWorker` | 否 | 否 | `StructuredCanonicalResult + Dataset/Table artifacts` |

统一边界：

```text
Worker 内部可以复杂
Worker 对外必须简单
Worker 不直接污染主线程
Worker 不向主 agent 暴露 raw tool loop
```

三者之间通过 `EvidenceArtifactGraph` 交互，而不是通过 prompt 文本互相“暗示”：

```text
RetrievalWorker 命中 dataset_ref/pdf_page_ref/table_summary_ref
  -> EvidenceArtifactGraph 记录 source object
  -> Orchestrator 派发给 PDFWorker 或 StructuredDataWorker
  -> Worker 产出 canonical result 或派生 artifact
  -> BindingArbitrator 只基于 artifact/ref commit binding
```

### 3.2 EvidenceEnvelope 作为统一交接物

新增统一 evidence 模型，建议位置：

- `backend/query/evidence_models.py`
- 或 `backend/evidence/models.py`

建议模型：

```python
EvidenceEnvelope:
    query: str
    source_worker: str
    evidence_items: list[EvidenceItem]
    source_objects: list[SourceObjectRef]
    derived_artifacts: list[EvidenceArtifact]
    document_candidates: list[DocumentCandidate]
    dataset_candidates: list[DatasetCandidate]
    table_candidates: list[TableCandidate]
    answer_candidates: list[AnswerCandidate]
    ambiguity: EvidenceAmbiguity | None
    diagnostics: dict

SourceObjectRef:
    object_id: str
    object_type: "pdf" | "dataset" | "table" | "text_chunk" | "web_doc" | "memory"
    uri: str
    parent_id: str | None
    locator: dict
    index_refs: list[str]
    metadata: dict

EvidenceArtifact:
    artifact_id: str
    artifact_type: "text_chunk" | "pdf_page" | "pdf_table" | "dataset_schema" | "table_object" | "dataset_summary"
    source_object_id: str
    parent_artifact_id: str | None
    content_ref: str
    canonical_preview: str
    visibility: "model_visible" | "debug_only"
    consumable_by: list[str]

EvidenceItem:
    kind: "text" | "document" | "dataset" | "table_object" | "pdf_page" | "memory"
    source: str
    text: str
    score: float
    metadata: dict
    visibility: "model_visible" | "debug_only"

DatasetCandidate:
    path: str
    target_object: str
    evidence_source: str
    confidence: float
    reason: str

DocumentCandidate:
    path: str
    document_type: str
    page: int | None
    confidence: float
    reason: str

TableCandidate:
    artifact_id: str
    source_object_id: str
    source_kind: "pdf_table" | "spreadsheet" | "csv" | "html_table"
    locator: dict
    schema_preview: list[str]
    confidence: float
    reason: str
```

关键语义：

1. `EvidenceEnvelope` 是 worker 到 orchestrator 的交接物。
2. `source_objects` 表示真实对象，例如一个 PDF、一个 Excel、一个网页、一个记忆对象。
3. `derived_artifacts` 表示从真实对象派生出的可处理产物，例如 PDF 页、PDF 表格、dataset schema、向量块。
4. `dataset_candidates` 不是 binding，只是可绑定候选。
5. `document_candidates` 不是 active_pdf，只是可绑定候选。
6. `table_candidates` 不是 dataset binding，只是可转交给结构化 worker 的表格候选。
7. `visibility=debug_only` 的内容不能进入 prompt/session。
8. 只有通过 answer/presentation policy 的内容才能进入用户可见输出。

### 3.2.1 EvidenceArtifactGraph：三条链的共享数据平面

新增统一派生产物图，建议位置：

- `backend/query/evidence_graph.py`
- 或 `backend/evidence/graph.py`

职责：

1. 记录 source object 与 derived artifact 的父子关系。
2. 记录 artifact 可以被哪些 worker 消费。
3. 记录 artifact 是否已经 canonical 化、是否可入库、是否可展示。
4. 为 RAG/PDF/结构化数据提供同一套 ref，而不是互相传 raw text。

建议模型：

```python
EvidenceArtifactGraph:
    session_id: str
    turn_id: str
    source_objects: dict[str, SourceObjectRef]
    artifacts: dict[str, EvidenceArtifact]
    edges: list[EvidenceEdge]

EvidenceEdge:
    from_id: str
    to_id: str
    relation: "derived_from" | "indexed_as" | "summarized_as" | "table_extracted_from" | "bound_as"
    confidence: float
    worker: str
```

典型关系：

```text
pdf:report.pdf
  -> pdf_page:report.pdf#page=3
  -> pdf_table:report.pdf#page=3#table=1
  -> table_object:tmp/artifacts/report_page3_table1.parquet
  -> dataset_candidate:report_page3_table1

dataset:inventory.xlsx
  -> dataset_schema:inventory.xlsx#schema
  -> text_chunk:inventory.xlsx#summary
  -> vector_index_ref:rag://inventory-summary
```

这层是“协调”的核心。Worker 不共享内部状态，只共享 typed artifact/ref。

### 3.2.2 RAG 索引对象必须升级为 SourceObjectRef

RAG 索引不应只存 chunk text。每个向量块至少要能反查：

1. `source_object_id`：原始对象，例如 PDF、Excel、CSV、Markdown。
2. `artifact_id`：派生产物，例如 PDF 页、PDF 表格、dataset summary。
3. `artifact_type`：文本块、页、表格、schema、摘要。
4. `content_ref`：原始内容或派生产物的位置。
5. `consumer_hint`：可被 PDFWorker、StructuredDataWorker 或 AnswerFinalizer 消费。

因此 RetrievalWorker 的输出不是“搜索到了几段文本”，而是：

```text
命中 text evidence
命中 source object
命中 derived artifact
产生 document/dataset/table candidate
```

这样才能实现：

1. RAG 命中 PDF 页后，用户追问“看这一页表格”，可以交给 PDFWorker 或 StructuredDataWorker。
2. RAG 命中 Excel 摘要后，用户确认“就是销售数据库”，可以交给 StructuredDataWorker。
3. RAG 命中 PDF 中抽出的表格摘要后，可以让结构化 worker 读取对应 `table_object`，而不是分析摘要文本。

### 3.3 BindingCandidateStore：解决澄清轮承接

新增短期候选存储，建议位置：

- `backend/query/evidence_store.py`
- 或并入 `RuntimeContextState`

职责：

1. 保存本轮 evidence 中的候选对象。
2. 为下一轮澄清提供 typed candidate。
3. 只存短期 session scope，不写长期记忆。

示例：

```python
BindingCandidateStore:
    session_id
    turn_id
    candidates:
      - id: "cand:dataset:inventory"
        kind: "dataset"
        path: "knowledge/E-commerce Data/inventory.xlsx"
        label: "库存数据 / 缺货预警"
        source_worker: "retrieval"
        expires_after_turns: 3
```

这样下一轮：

```text
用户: 是销售数据库
  -> followup/clarification resolver 不需要靠“销售数据库”关键词选工具
  -> 它只需要判断这是对上一轮 candidate set 的选择
  -> BindingArbitrator 将 candidate commit 为 active_dataset
```

### 3.4 Capability Worker 与 Tool 的关系

本轮不废除工具契约层。关系应改为：

```text
Tool = 执行单元
Worker = 能力编排单元
Orchestrator = 当前轮仲裁单元
EvidenceArtifactGraph = 跨 worker 共享数据平面
```

例如：

```text
RetrievalWorker
  uses RetrievalService
  may internally use retriever/rerank
  returns EvidenceEnvelope + SourceObjectRef + EvidenceArtifact refs

PDFWorker
  uses pdf_analysis parser/runtime
  may extract text/pages/tables
  returns PDFCanonicalResult + Document/Table artifacts

StructuredDataWorker
  uses StructuredDataAnalysisTool or StructuredDataEngine
  consumes DatasetCandidate/TableCandidate/TableArtifact
  returns StructuredCanonicalResult + Dataset/Table artifacts
```

主 agent 不直接看到：

1. `search_knowledge` 工具 schema
2. PDF raw page dump
3. structured raw dataframe dump
4. worker internal diagnostics

主 agent 可以看到：

1. compact evidence summary
2. canonical answer candidate
3. explicit missing info / clarification prompt

## 4. 固定目标执行流

### 4.1 单轮主流程

```text
1. Understanding
   input: user message
   output: TaskFrame

2. Follow-up / Clarification Resolution
   input: TaskFrame + active candidate refs + task refs
   output: FollowupResolution / CandidateSelection / none

3. Capability Dispatch
   input: TaskFrame + bindings + skill policy + tool registry
   output: DispatchPlan

4. Evidence Orchestration
   input: DispatchPlan
   output:
     - EvidenceEnvelope, or
     - EvidenceArtifactGraph updates, or
     - Direct Worker Request, or
     - ClarificationRequest

5. Worker Execution
   RetrievalWorker / PDFWorker / StructuredDataWorker
   output: EvidenceEnvelope / CanonicalResult / EvidenceArtifact updates

6. Binding Arbitration
   input: explicit refs + evidence candidates + artifact refs + follow-up selection
   output: committed binding or clarification

7. Answer Finalization
   input: canonical result / evidence pack
   output: CanonicalOutput

8. Projection
   input: CanonicalOutput + committed bindings
   output: task refs / context refs / session-memory projection
```

### 4.2 RAG 发现 dataset 的专门流

```text
用户: 查询本地数据库哪些城市缺货
  -> TaskFrame: local knowledge/database lookup
  -> RetrievalWorker
  -> EvidenceEnvelope:
       dataset_candidates=[inventory.xlsx]
       text_evidence=[...]
       ambiguity=business data vs document corpus
  -> 若候选足够明确且任务需要计算:
       Clarify or commit depending on policy
  -> 用户: 是销售数据库
  -> CandidateSelection: select dataset candidate
  -> BindingArbitrator: active_dataset=inventory.xlsx
  -> StructuredDataWorker: analyze shortage
  -> CanonicalOutput
```

### 4.3 PDF 发现与读取流

```text
用户: 读这份 PDF 第三页
  -> explicit PDF ref / active PDF candidate
  -> PDFWorker
  -> PDFCanonicalResult
  -> Output contract
  -> stable summary / safe degraded answer
```

### 4.4 结构化数据直接流

```text
用户: inventory.xlsx 哪些仓库缺货
  -> explicit dataset path
  -> BindingArbitrator commit
  -> StructuredDataWorker
  -> StructuredCanonicalResult
  -> Projection
```

### 4.5 PDF 表格跨模态流

```text
用户: 这份 PDF 里的表格里，哪些城市缺货
  -> active_pdf / PDF candidate
  -> PDFWorker 解析页面与表格区域
  -> EvidenceArtifactGraph:
       pdf_table artifact
       table_object artifact
       TableCandidate
  -> BindingArbitrator:
       table candidate 可转成临时 structured binding
  -> StructuredDataWorker:
       对 table_object 做筛选/聚合/排序
  -> CanonicalOutput
```

### 4.6 RAG 命中 PDF 表格或数据集摘要的跨模态流

```text
用户: 本地资料里哪些城市缺货
  -> RetrievalWorker 搜索向量库
  -> 命中:
       dataset_summary:inventory.xlsx
       或 pdf_table_summary:report.pdf#page=7#table=2
  -> EvidenceEnvelope:
       DatasetCandidate 或 TableCandidate
  -> Orchestrator:
       如果用户要“计算/筛选/列出城市”，转 StructuredDataWorker
       如果用户要“解释来源/读原文”，转 PDFWorker 或 AnswerFinalizer
  -> CanonicalOutput
```

## 4.7 Runtime 接入主链

这套证据子系统不能作为旁路接入。它必须进入现有 `QueryPlanner -> QueryExecutionPlan -> QueryRuntime -> PromptBuilder -> Persistence` 主链，否则会出现两种失败：

1. planner 仍然只知道 `agent/direct_tool`，worker 结果只能伪装成 tool output。
2. runtime 仍然在 `route=rag` 分支里直接调用 `retrieval_service.retrieve()`，EvidenceEnvelope 和 ArtifactGraph 无法成为主事实。

### 4.7.1 当前 runtime 真实切入点

当前主链要点：

```text
QueryPlanner._build_execution()
  -> QueryExecutionPlan(
       execution_kind="agent" | "direct_tool",
       dispatch_plan=DispatchPlan,
       tool_input=...,
       structured_binding=...
     )

QueryRuntime._stream_planned_execution()
  -> if execution_kind == "direct_tool":
       RuntimeToolBridge.stream_direct_tool_execution()
       return

  -> if route == "rag":
       retrieval_service.retrieve()
       context.retrieval_results = list[dict]
       prompt_builder 注入 Retrieval Evidence

  -> create_conversation_agent(system_prompt, tools)
  -> AssistantOutputBoundary
  -> maybe_finalize_rag/pdf
  -> done event
```

这意味着 70 的实施必须显式替换两个旧入口：

1. `route == "rag" -> retrieval_service.retrieve()`。
2. `pdf_analysis / structured_data_analysis` 作为普通 `direct_tool` 的高噪声执行入口。

### 4.7.2 新增 WorkerExecutionPlan

建议在 `backend/query/models.py` 中新增：

```python
WorkerExecutionPlan:
    worker_route: Literal["none", "retrieval", "pdf", "structured_data", "evidence_orchestrator"]
    request: WorkerRequest | None
    expected_result: Literal["evidence", "canonical", "clarification"]
    artifact_refs: list[str]
    candidate_refs: list[str]
    fallback_execution_kind: Literal["agent", "direct_tool", "none"]
    cutover_mode: Literal["shadow", "primary", "disabled"]
```

并扩展：

```python
QueryExecutionPlan:
    execution_kind: "agent" | "direct_tool" | "worker"
    worker_plan: WorkerExecutionPlan | None
    evidence_envelope: EvidenceEnvelope | None = None
    artifact_graph_delta: EvidenceArtifactGraphDelta | None = None
```

短期兼容规则：

1. `execution_kind="worker"` 只用于 RAG/PDF/结构化三类高噪声能力。
2. 天气、金价、普通 web 等低污染 direct tool 暂时仍走 `direct_tool`。
3. `dispatch_plan` 继续存在，但要能产出 `selected_worker_request`，不能只产出 `selected_tool_request`。

### 4.7.3 CapabilityDispatch 的输出扩展

`CapabilityDispatchScheduler` 当前是 shadow 记录工具候选。70 落地时要扩展为：

```python
DispatchPlan:
    route: str
    selected_tool_request: ToolInvocationRequest | None
    selected_worker_request: WorkerRequest | None
    worker_route: str
    prompt_exposure: PromptExposurePlan
    effective_tool_scope: ToolScope
```

调度约束：

1. RAG/PDF/结构化不直接等于工具暴露。
2. `knowledge_lookup` 优先形成 `RetrievalWorker` 请求。
3. 显式 PDF 或 active PDF 形成 `PDFWorker` 请求。
4. committed dataset/table binding 形成 `StructuredDataWorker` 请求。
5. RAG 命中 artifact 后由 `EvidenceOrchestrator` 生成下一跳 worker 请求。

禁止：

1. 在 `task_understanding.py` 里用关键词选择 worker。
2. 在 runtime 里按自然语言重判 worker。
3. 让 `selected_worker_request` 和 `selected_tool_request` 同时作为主执行入口。

### 4.7.4 QueryRuntime 的新执行分支

runtime 的执行顺序应调整为：

```text
1. context compaction / memory package

2. if execution.execution_kind == "worker":
     EvidenceOrchestrator.stream_execution()
     -> worker_start / worker_evidence / worker_end / done
     -> return

3. if execution.execution_kind == "direct_tool":
     RuntimeToolBridge.stream_direct_tool_execution()
     -> return

4. optional low-risk memory prefetch

5. agent path
     system prompt 只接收 canonical context / compact evidence summary
     tools 不包含 RAG/PDF/structured 高噪声工具
```

即：

```python
if execution.execution_kind == "worker" and execution.worker_plan is not None:
    async for event in self.evidence_orchestrator.stream_execution(...):
        yield event
    return
```

这个分支必须放在旧 `route == "rag"` 检索逻辑之前，并最终删除旧逻辑。

### 4.7.5 Worker 事件协议

为了兼容前端和测试，worker 事件需要保持和现有 runtime event 风格接近：

```python
{"type": "worker_start", "worker": "retrieval", "request": {...}}
{"type": "worker_evidence", "worker": "retrieval", "evidence": EvidenceEnvelope.to_dict()}
{"type": "worker_artifacts", "graph_delta": EvidenceArtifactGraphDelta.to_dict()}
{"type": "worker_end", "worker": "structured_data", "result": CanonicalResult.to_dict()}
{"type": "done", "content": canonical.answer, ...}
```

`done` 事件仍是唯一用户可见完成事件，必须包含：

```python
done:
    content
    answer_channel
    answer_source
    answer_fallback_reason
    main_context
    task_summary_refs
    evidence_refs
    artifact_refs
    binding_candidate_refs
    committed_bindings
```

raw worker output 只进入 trace/debug，不进入 `done.content`、session assistant message 或 memory projection。

### 4.7.6 PromptBuilder 接入方式

现有 `prompt_builder` 通过 `context_package.retrieval_evidence` 注入检索文本。新链路应改为：

```text
EvidenceEnvelope
  -> EvidenceProjectionPolicy
  -> model_visible_evidence_summary
  -> ContextPackage.model_visible_sections["retrieval_evidence" 或 "evidence_summary"]
```

PromptBuilder 只消费：

1. compact evidence summary。
2. selected artifact refs。
3. committed binding refs。
4. canonical task summaries。

PromptBuilder 不消费：

1. `EvidenceItem.visibility=debug_only`。
2. raw retrieval snippets。
3. PDF raw page/table dump。
4. dataframe raw rows。
5. worker diagnostics。

### 4.7.7 Persistence 接入方式

现有持久化主要从 `done` event、assistant message、task refs 和 session memory projection 取数据。新链路必须保证：

```text
WorkerResult
  -> CanonicalResult
  -> done event
  -> RuntimePersistenceAssembler
  -> session history / task refs / memory projection
```

候选对象走短期 store：

```text
EvidenceEnvelope.dataset/table/document candidates
  -> BindingCandidateStore
  -> follow-up resolver
```

持久化边界：

1. `BindingCandidateStore` 是短期 session scope，不写长期记忆。
2. `EvidenceArtifactGraph` 可以 session-scoped 持久化，但默认只保存 refs/metadata，不保存 raw dump。
3. `CanonicalResult` 可以进入 task summary。
4. `degraded/error` 结果默认不进入稳定 task summary，除非 projection policy 明确允许。

### 4.7.7.1 Memory 接入边界

这套证据子系统必须对接记忆系统，但不能把 evidence 系统本身变成记忆污染源。

当前记忆系统真实入口：

```text
QueryRuntime done event
  -> _capture_session_memory_projection(
       main_context_payload=event["main_context"],
       task_summary_payloads=event["task_summary_refs"]
     )
  -> RuntimeContextState.capture_session_memory_projection()
  -> memory_facade.refresh_session_memory_from_context_state()
  -> SessionMemoryManager.update_from_context_state()
  -> ContextSlots / TaskSummaryRef / FlowSnapshot
```

现有 session memory 已经能承接：

1. `main_context.active_constraints.active_dataset`
2. `main_context.active_constraints.active_pdf`
3. `task_summary_refs[].key_points` 中的 `dataset=... / pdf=...`
4. `committed_dataset / committed_pdf`
5. `active_binding_identity`

因此本轮不应另开一条 memory 写入旁路。正确方案是：

```text
WorkerResult
  -> CanonicalResult
  -> WorkerProjectionAdapter
  -> done.main_context + done.task_summary_refs
  -> 现有 session memory projection
```

### 4.7.7.2 三层记忆策略

| 层级 | 保存内容 | 生命周期 | 是否模型可见 | 是否进入长期记忆 |
| --- | --- | --- | --- | --- |
| `BindingCandidateStore` | dataset/document/table candidates | session 短期，默认 3 turns | 否，除非生成澄清问题 | 否 |
| `SessionMemory ContextSlots` | committed binding、active dataset/pdf/table refs | session | 是，作为当前情境 | 否 |
| `Durable Memory` | 用户明确要求记住的稳定偏好/事实 | 长期 | 是 | 是 |

硬边界：

1. raw retrieval snippets 不进入任何记忆层。
2. PDF raw page/table dump 不进入任何记忆层。
3. dataframe raw rows 不进入任何记忆层。
4. candidate 只进入短期 store，不进入长期记忆。
5. committed binding 可以进入 `ContextSlots`，但只保存 ref/path/identity，不保存内容 dump。
6. canonical answer 可以进入 task summary，但必须由 projection policy 允许。
7. degraded/error 默认不进入稳定 task summary。

### 4.7.7.3 WorkerProjectionAdapter

新增或等价实现：

- `backend/query/worker_projection.py`

职责：

1. 将 `CanonicalResult` 转成 `MainContextState` 的安全字段。
2. 将 `CanonicalResult` 转成 `TaskSummaryRef`。
3. 将 `BindingCandidate` 转成短期 store capsule。
4. 拒绝 raw evidence、diagnostics、worker internal payload 进入 memory projection。

建议接口：

```python
WorkerProjectionAdapter:
    project_done_event(
        query: str,
        canonical_result: CanonicalResult,
        worker_result: WorkerResult,
        previous_main_context: MainContextState,
    ) -> WorkerProjection

WorkerProjection:
    main_context: MainContextState
    task_summary_refs: list[TaskSummaryRef]
    candidate_refs: list[str]
    memory_policy: "session_context_only" | "do_not_persist" | "durable_allowed"
```

`MainContextState` 投影规则：

```text
CanonicalResult.bindings.active_dataset
  -> main_context.active_constraints.active_dataset
  -> main_context.active_constraints.active_binding_identity
  -> main_context.followup_binding_key = active_dataset
  -> main_context.followup_binding_identity

CanonicalResult.bindings.active_pdf
  -> main_context.active_constraints.active_pdf
  -> main_context.active_constraints.active_binding_identity
  -> main_context.followup_binding_key = active_pdf

CanonicalResult.bindings.active_table
  -> main_context.active_constraints.active_table
  -> main_context.active_constraints.active_binding_identity
  -> main_context.followup_binding_key = active_table
```

`TaskSummaryRef` 投影规则：

```text
ok canonical structured result
  -> summary = canonical.answer compacted
  -> key_points = ["dataset=...", "artifact=..."]
  -> task_kind = structured_data

ok canonical pdf result
  -> summary = canonical.answer compacted
  -> key_points = ["pdf=...", "pdf_pages=..."]
  -> task_kind = pdf

rag candidate clarification
  -> 不生成 stable task summary
  -> 只保存 candidate refs 到 BindingCandidateStore
```

### 4.7.7.4 与长期记忆的关系

Evidence worker 的输出默认不触发 durable memory extraction。

只有以下情况允许进入长期记忆候选：

1. 用户明确说“记住/以后都按这个/长期保存”。
2. 内容是用户偏好、稳定身份、稳定项目规则，而不是检索证据。
3. 通过既有 `evaluate_memory_write / durable_write_admission` 链路。

禁止：

1. 因为 RAG 命中某个文档，就把文档内容写入长期记忆。
2. 因为结构化分析得到一次结果，就把表格结果写成长期事实。
3. 因为 PDF 读到一页，就把 PDF 页摘要写入长期记忆。

### 4.7.7.5 Memory 接入实施顺序

必须按以下顺序实施，不能边想边改：

1. 新增 `WorkerProjectionAdapter`，只做 pure function，不接 runtime。
2. 为 adapter 写单测，覆盖 dataset/pdf/table/candidate/degraded 五类输出。
3. 将 `EvidenceOrchestrator._done_event()` 改为消费 projection adapter。
4. 确认 `done.main_context.active_constraints` 能被现有 `SessionMemoryManager.update_from_context_state()` 识别。
5. 新增 session memory 回归：structured worker committed dataset 后，`committed_dataset` 可从 session snapshot 恢复。
6. 再考虑是否将 `BindingCandidateStore` 从进程内 store 升级为 session-scoped store。

验收点：

1. worker structured result 后，session memory 中能恢复 `committed_dataset`。
2. worker candidate clarification 后，session memory 不出现 raw evidence，也不出现 stable committed binding。
3. PDF degraded result 不写入 stable task summary。
4. 长期记忆不因 evidence worker 自动新增文档事实。
5. prompt 中只出现 canonical context refs，不出现 raw worker output。

### 4.7.8 旧 runtime 分支的切换顺序

切换顺序必须固定：

1. `route=rag` shadow：保留旧 `retrieval_results`，并行生成 `EvidenceEnvelope` 和 trace。
2. `route=rag` primary：runtime 改走 `RetrievalWorker`，旧 `retrieval_results` 只作为兼容字段。
3. `search_knowledge` 从 bounded agent 默认工具集中移除。
4. `structured_data_analysis` 保留 direct tool facade，但 RAG/PDF 派生表格路径走 `StructuredDataWorker`。
5. `pdf_analysis` 保留 direct tool facade，但 active PDF 阅读路径走 `PDFWorker`。
6. `maybe_finalize_rag_output` 改为消费 `CanonicalResult/EvidenceEnvelope`。
7. `maybe_finalize_pdf_output` 改为消费 `PDFWorker CanonicalResult`。
8. 删除 runtime 中 `route == "rag" -> retrieval_service.retrieve()` 的旧主链。

### 4.7.9 Runtime 验收点

完成标准：

1. `QueryExecutionPlan.execution_kind` 能出现 `worker`。
2. RAG 查询不再在 runtime 里直接调用 `retrieval_service.retrieve()` 作为主链。
3. worker 分支不创建 main agent，也不暴露 `search_knowledge/pdf_analysis/structured_data_analysis` 工具 schema。
4. worker 输出的 `EvidenceEnvelope/CanonicalResult` 可以生成同格式 `done` event。
5. PromptBuilder 只看到 compact evidence summary，不看到 raw worker output。
6. RuntimePersistence 只保存 canonical result 和 refs。
7. trace 中能看到 `worker_route/source_object_id/artifact_id/binding_source`。

## 5. 层级职责与禁止事项

### 5.1 Understanding

允许：

1. 提取显式路径、URL、页码、文件后缀。
2. 产出 capability request。
3. 标记 local/source anchor。

禁止：

1. 因“销售数据库”这类业务词直接选择 structured tool。
2. 因“缺货”这类领域词直接强绑 `inventory.xlsx`。
3. 直接修改 session binding。

### 5.2 RetrievalWorker

允许：

1. 检索。
2. rerank。
3. 识别命中源类型。
4. 输出 source object refs、artifact refs、dataset/document/table candidates。

禁止：

1. 直接调用主模型生成最终答案。
2. 直接写 session history。
3. 把 raw snippets 标成 stable answer。
4. 暴露为 main agent 可自由循环调用的工具。

### 5.3 PDFWorker

允许：

1. PDF 解析、质量门、阅读计划、摘要生成。
2. 输出 `PDFCanonicalResult`。
3. 抽取表格区域并产出 `TableCandidate / TableArtifact`。
4. 将可结构化处理的表格落为临时 artifact，而不是塞进 prompt。

禁止：

1. 将 degraded/error 写成 summary。
2. 将 raw page dump 传给主线程展示。
3. 将 PDF binding 决策和 session restore 混在一起。

### 5.4 StructuredDataWorker

允许：

1. 读取已绑定数据集。
2. 分析 schema、聚合、过滤、排名。
3. 输出 canonical structured result。
4. 消费由 PDFWorker/RetrievalWorker 产生的 `TableCandidate / DatasetCandidate`。
5. 对临时表格 artifact 生成 schema preview 和稳定结果。

禁止：

1. 无 binding 时自行从 history 猜 path。
2. 直接消费 RAG raw text 当完整数据表。
3. 把 dataframe raw dump 写入 session memory。

### 5.5 BindingArbitrator

允许：

1. 从 explicit ref、candidate selection、active task binding 中 commit binding。
2. 在候选不唯一时发 clarification。

禁止：

1. 读取 prompt 文本反推 binding。
2. 让 session restore 覆盖当前轮 explicit binding。
3. 用弱词直接绑定对象。

### 5.6 PromptBuilder

允许注入：

1. `CanonicalOutput`
2. compact evidence summary
3. selected binding refs
4. stable task summaries

禁止注入：

1. raw retrieval snippets
2. raw PDF page dump
3. raw table rows dump
4. tool contract debug
5. permission/scope/debug state

## 10. 迁移与切换规则

### 10.1 Shadow First

先让 RetrievalWorker/EvidenceEnvelope 在 trace 中并行记录，不立刻替换全部主链。

### 10.2 Cutover Order

切换顺序固定：

1. RAG payload -> EvidenceEnvelope
2. source object / artifact refs -> EvidenceArtifactGraph
3. planner/runtime 增加 worker plan shadow，不改变行为
4. Evidence candidates -> candidate store
5. RetrievalWorker primary 替换 runtime 旧 RAG retrieve 分支
6. follow-up selection -> binding commit
7. structured worker cutover
8. PDF table artifact -> structured worker handoff
9. PDF worker result alignment
10. prompt/memory projection cleanup
11. old tool exposure cleanup

### 10.3 Rollback Rule

可以回滚：

1. 某个 worker 是否主用。
2. EvidenceEnvelope adapter 实现。
3. 某个 projection policy。

不可回滚：

1. raw output 不进主线程。
2. weak phrase 不直接强绑工具/数据集。
3. main agent 不持有高噪声工具自由循环权。
4. canonical result 才能进入 session-memory。

## 11. 明确不做什么

本轮不做：

1. 不引入 LangGraph/AutoGen 全量重写。
2. 不把 RAG/PDF/结构化数据做成三个能聊天的自治 agent。
3. 不继续给 `task_understanding.py` 加业务关键词补丁。
4. 不让 `search_knowledge` 继续作为 main agent 默认可见工具。
5. 不把 RAG 命中的表格片段当完整表格分析。
6. 不把 PDF/表格 raw output 写入 session-memory。
7. 不把 PDF 表格强行伪装成持久 dataset；它应先是 `TableArtifact / active_table`，只有用户明确保存或入库时才升级。

本计划书进入“可实施”状态，需要满足：

1. `EvidenceEnvelope` 模型和 worker handoff 边界确定。
2. RAG 命中 dataset/pdf 能产出 typed candidate。

5. 结构化数据分析只从 committed binding 或 active table artifact 执行。

8. session-memory 不投影 raw worker output。
9. 长场景缺货案例不再出现 RAG 工具循环。

## 13. 总结

这次重构的中心不是“多造几个 agent”，而是补上能力子系统的边界协议。

最终目标是：

```text
RAG 负责发现证据和候选
PDF 负责阅读和生成稳定文档结果
结构化数据负责对已绑定数据集做计算分析
EvidenceArtifactGraph 负责表达 PDF/表格/文本块/向量索引之间的派生关系
主调度层负责仲裁、绑定、澄清和组合
主 agent 只负责必要的语言整合，不再背负高噪声工具循环
```

这条路线能同时解决：

1. 本地数据库被误当普通 RAG 的问题。
2. RAG 命中结构化数据后无法 handoff 的问题。
3. PDF raw/degraded 结果污染主线程的问题。
4. 结构化数据 binding 与 session restore 竞争的问题。
5. prompt/history/session-memory 多通道重复污染的问题。

后续实施应优先推进 Phase 1 到 Phase 3，因为它们直接决定“RAG 发现对象能否被后续轮次正确承接”。只有这部分稳定后，再推进 PDF 与结构化 worker 的深层统一，风险最低。

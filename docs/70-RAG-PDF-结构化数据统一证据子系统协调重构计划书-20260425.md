# 70-RAG/PDF/结构化数据统一证据子系统协调重构计划书-20260425

> 目的：把当前 RAG、PDF 阅读、结构化数据分析三条能力链从“主 agent 可见工具/文本碎片”重构为“受控证据子系统 + 统一调度仲裁 + canonical 输出投影”的协调架构。  
> 本计划书是初步重构蓝图，后续实施应先按本文件锁定边界，再进入逐阶段代码改造。

---

## 0. 本版结论

当前问题不是某个工具不好用，而是三类高噪声能力没有形成统一的证据边界：

1. RAG 检索结果会以文本形式进入主 agent、history、session-memory。
2. PDF 阅读已经有子 agent 雏形，但 output/projection 仍有专用补丁和降级残留。
3. 结构化数据工具已经有 contract/binding，但 RAG 命中 `.xlsx` 后没有 typed handoff 到结构化数据链。

推荐方向：

```text
User Input
  -> TaskFrame
  -> CapabilityDispatch
  -> Evidence Orchestrator
      -> Retrieval Worker
      -> PDF Worker
      -> Structured Data Worker
  -> EvidenceEnvelope / CanonicalResult
  -> Binding Arbitration
  -> Answer Finalization
  -> Projection / Session Memory
```

核心原则：

> RAG、PDF、结构化数据都不再作为主 agent 可自由循环调用的普通工具；它们应成为受控 evidence worker / capability worker。主线程只消费 typed evidence、binding candidate 和 canonical result。

## 1. 当前真实链路与失败模式

### 1.1 当前 RAG 链路

相关文件：

- `backend/retrieval/service.py`
- `backend/tools/search_knowledge_tool.py`
- `backend/query/runtime.py`
- `backend/query/prompt_builder.py`

当前链路：

```text
route=rag
  -> RetrievalService.retrieve()
  -> retrieval_results
  -> prompt_builder 注入 Retrieval Evidence
  -> 主模型基于证据生成回答
```

同时在 `bounded_agent` 场景中：

```text
candidate_tools = search_knowledge
  -> 主 agent 看到 search_knowledge schema
  -> 模型可多次调用 search_knowledge
  -> raw retrieval snippets 进入 assistant content/tool_calls
```

关键缺口：

1. RAG 返回的是 `list[dict]` 和工具字符串，不是统一 `EvidenceEnvelope`。
2. RAG 命中 `.xlsx/.csv/.pdf` 时，只是文本证据，不会升级为 `DatasetCandidate / DocumentCandidate`。
3. `search_knowledge` 可暴露给 main agent，导致检索循环和 prompt 污染。
4. 检索 evidence、assistant content、tool_calls output、session-memory projection 的边界不硬。

### 1.2 当前 PDF 链路

相关文件：

- `backend/tools/pdf_analysis_tool.py`
- `backend/pdf_agent/*`
- `backend/pdf_analysis/*`
- `backend/query/runtime_tools.py`
- `backend/query/runtime_output_policy.py`

当前状态：

1. `pdf_agent` 已经存在，PDF 阅读链已从纯 tool 向子 agent 方向迁移。
2. `ToolResultEnvelope` 已接入 direct tool 输出。
3. 但 `RuntimeToolBridge` 仍有 PDF 专项 persist/finalize gate。
4. PDF canonical/projection 还没有完全泛化为统一 capability result contract。

关键缺口：

1. PDF worker 的 canonical result 没有和 RAG/结构化数据共享一套 Evidence/Projection 协议。
2. PDF 降级结果仍需要 runtime 专项保护。
3. PDF 结果是否能写入 session-memory 仍散落在 runtime/output policy/persistence gate。

### 1.3 当前结构化数据链路

相关文件：

- `backend/tools/structured_data_analysis_tool.py`
- `backend/structured_data/*`
- `backend/query/binding_resolver.py`
- `backend/query/tool_input_resolver.py`
- `backend/tasks/context_models.py`

当前状态：

1. 结构化数据工具需要 explicit path 或 active binding。
2. `StructuredDatasetBinding` 已进入 planner/execution。
3. 工具 output 已能经 `ToolResultEnvelope` 进入 canonical output。

关键缺口：

1. 结构化数据 binding 只从显式路径/预绑定进入，不接收 RAG 发现的 dataset candidate。
2. “销售数据库”这类澄清选择缺少对上一轮 evidence candidate 的引用。
3. `inventory.xlsx` 被 RAG 发现后没有成为可确认/可绑定对象，只进入了历史文本。

### 1.4 这次缺货场景的根因链

```text
用户: 查询本地数据库哪些城市缺货
  -> route=rag
  -> RAG 命中 inventory.xlsx
  -> 但只作为文本 evidence
  -> assistant 让用户澄清“销售数据库 or 知识库”

用户: 是销售数据库
  -> followup_resolver 无 binding candidate 可接
  -> task_understanding fallback_bounded_lookup
  -> candidate_tools=search_knowledge
  -> 主 agent 反复 search_knowledge
  -> inventory.xlsx 继续以文本片段污染 history/session-memory
```

系统性质缺失：

> RAG 发现能力没有 typed handoff；主调度层没有“证据候选 -> 澄清选择 -> binding commit”的协议。

## 3. 推荐设计方向

### 3.1 三类能力统一定义

本轮不建议把三条链都变成“能自由思考和调用工具的 agent”。更准确的定义是：

| 子系统 | 推荐定位 | 是否能自由调用工具 | 是否能直接写 session | 对主线程输出 |
| --- | --- | --- | --- | --- |
| RAG | `RetrievalWorker / EvidenceWorker` | 否 | 否 | `EvidenceEnvelope` |
| PDF | `PDFWorker / PDFReadAgent` | 受控内部能力 | 否 | `PDFCanonicalResult` |
| 结构化数据 | `StructuredDataWorker` | 否 | 否 | `StructuredCanonicalResult` |

统一边界：

```text
Worker 内部可以复杂
Worker 对外必须简单
Worker 不直接污染主线程
Worker 不向主 agent 暴露 raw tool loop
```

### 3.2 EvidenceEnvelope 作为统一交接物

新增统一 evidence 模型，建议位置：

- `backend/query/evidence_models.py`
- 或 `backend/evidence/models.py`

建议模型：

```python
EvidenceEnvelope:
    query: str
    source_worker: str
    evidence_items: list[EvidenceItem]
    document_candidates: list[DocumentCandidate]
    dataset_candidates: list[DatasetCandidate]
    answer_candidates: list[AnswerCandidate]
    ambiguity: EvidenceAmbiguity | None
    diagnostics: dict

EvidenceItem:
    kind: "text" | "document" | "dataset" | "table_object" | "pdf_page" | "memory"
    source: str
    text: str
    score: float
    metadata: dict
    visibility: "model_visible" | "debug_only"

DatasetCandidate:
    path: str
    target_object: str
    evidence_source: str
    confidence: float
    reason: str

DocumentCandidate:
    path: str
    document_type: str
    page: int | None
    confidence: float
    reason: str
```

关键语义：

1. `EvidenceEnvelope` 是 worker 到 orchestrator 的交接物。
2. `dataset_candidates` 不是 binding，只是可绑定候选。
3. `document_candidates` 不是 active_pdf，只是可绑定候选。
4. `visibility=debug_only` 的内容不能进入 prompt/session。
5. 只有通过 answer/presentation policy 的内容才能进入用户可见输出。

### 3.3 BindingCandidateStore：解决澄清轮承接

新增短期候选存储，建议位置：

- `backend/query/evidence_store.py`
- 或并入 `RuntimeContextState`

职责：

1. 保存本轮 evidence 中的候选对象。
2. 为下一轮澄清提供 typed candidate。
3. 只存短期 session scope，不写长期记忆。

示例：

```python
BindingCandidateStore:
    session_id
    turn_id
    candidates:
      - id: "cand:dataset:inventory"
        kind: "dataset"
        path: "knowledge/E-commerce Data/inventory.xlsx"
        label: "库存数据 / 缺货预警"
        source_worker: "retrieval"
        expires_after_turns: 3
```

这样下一轮：

```text
用户: 是销售数据库
  -> followup/clarification resolver 不需要靠“销售数据库”关键词选工具
  -> 它只需要判断这是对上一轮 candidate set 的选择
  -> BindingArbitrator 将 candidate commit 为 active_dataset
```

### 3.4 Capability Worker 与 Tool 的关系

本轮不废除工具契约层。关系应改为：

```text
Tool = 执行单元
Worker = 能力编排单元
Orchestrator = 当前轮仲裁单元
```

例如：

```text
RetrievalWorker
  uses RetrievalService
  may internally use retriever/rerank
  returns EvidenceEnvelope

PDFWorker
  uses pdf_analysis parser/runtime
  returns PDFCanonicalResult

StructuredDataWorker
  uses StructuredDataAnalysisTool or StructuredDataEngine
  returns StructuredCanonicalResult
```

主 agent 不直接看到：

1. `search_knowledge` 工具 schema
2. PDF raw page dump
3. structured raw dataframe dump
4. worker internal diagnostics

主 agent 可以看到：

1. compact evidence summary
2. canonical answer candidate
3. explicit missing info / clarification prompt

## 4. 固定目标执行流

### 4.1 单轮主流程

```text
1. Understanding
   input: user message
   output: TaskFrame

2. Follow-up / Clarification Resolution
   input: TaskFrame + active candidate refs + task refs
   output: FollowupResolution / CandidateSelection / none

3. Capability Dispatch
   input: TaskFrame + bindings + skill policy + tool registry
   output: DispatchPlan

4. Evidence Orchestration
   input: DispatchPlan
   output:
     - EvidenceEnvelope, or
     - Direct Worker Request, or
     - ClarificationRequest

5. Worker Execution
   RetrievalWorker / PDFWorker / StructuredDataWorker
   output: EvidenceEnvelope / CanonicalResult

6. Binding Arbitration
   input: explicit refs + evidence candidates + follow-up selection
   output: committed binding or clarification

7. Answer Finalization
   input: canonical result / evidence pack
   output: CanonicalOutput

8. Projection
   input: CanonicalOutput + committed bindings
   output: task refs / context refs / session-memory projection
```

### 4.2 RAG 发现 dataset 的专门流

```text
用户: 查询本地数据库哪些城市缺货
  -> TaskFrame: local knowledge/database lookup
  -> RetrievalWorker
  -> EvidenceEnvelope:
       dataset_candidates=[inventory.xlsx]
       text_evidence=[...]
       ambiguity=business data vs document corpus
  -> 若候选足够明确且任务需要计算:
       Clarify or commit depending on policy
  -> 用户: 是销售数据库
  -> CandidateSelection: select dataset candidate
  -> BindingArbitrator: active_dataset=inventory.xlsx
  -> StructuredDataWorker: analyze shortage
  -> CanonicalOutput
```

### 4.3 PDF 发现与读取流

```text
用户: 读这份 PDF 第三页
  -> explicit PDF ref / active PDF candidate
  -> PDFWorker
  -> PDFCanonicalResult
  -> Output contract
  -> stable summary / safe degraded answer
```

### 4.4 结构化数据直接流

```text
用户: inventory.xlsx 哪些仓库缺货
  -> explicit dataset path
  -> BindingArbitrator commit
  -> StructuredDataWorker
  -> StructuredCanonicalResult
  -> Projection
```

## 5. 层级职责与禁止事项

### 5.1 Understanding

允许：

1. 提取显式路径、URL、页码、文件后缀。
2. 产出 capability request。
3. 标记 local/source anchor。

禁止：

1. 因“销售数据库”这类业务词直接选择 structured tool。
2. 因“缺货”这类领域词直接强绑 `inventory.xlsx`。
3. 直接修改 session binding。

### 5.2 RetrievalWorker

允许：

1. 检索。
2. rerank。
3. 识别命中源类型。
4. 输出 dataset/document candidates。

禁止：

1. 直接调用主模型生成最终答案。
2. 直接写 session history。
3. 把 raw snippets 标成 stable answer。
4. 暴露为 main agent 可自由循环调用的工具。

### 5.3 PDFWorker

允许：

1. PDF 解析、质量门、阅读计划、摘要生成。
2. 输出 `PDFCanonicalResult`。

禁止：

1. 将 degraded/error 写成 summary。
2. 将 raw page dump 传给主线程展示。
3. 将 PDF binding 决策和 session restore 混在一起。

### 5.4 StructuredDataWorker

允许：

1. 读取已绑定数据集。
2. 分析 schema、聚合、过滤、排名。
3. 输出 canonical structured result。

禁止：

1. 无 binding 时自行从 history 猜 path。
2. 直接消费 RAG raw text 当完整数据表。
3. 把 dataframe raw dump 写入 session memory。

### 5.5 BindingArbitrator

允许：

1. 从 explicit ref、candidate selection、active task binding 中 commit binding。
2. 在候选不唯一时发 clarification。

禁止：

1. 读取 prompt 文本反推 binding。
2. 让 session restore 覆盖当前轮 explicit binding。
3. 用弱词直接绑定对象。

### 5.6 PromptBuilder

允许注入：

1. `CanonicalOutput`
2. compact evidence summary
3. selected binding refs
4. stable task summaries

禁止注入：

1. raw retrieval snippets
2. raw PDF page dump
3. raw table rows dump
4. tool contract debug
5. permission/scope/debug state

## 7. 分阶段实施计划

## Phase 0：冻结原则与观测

目标：

1. 明确 RAG/PDF/结构化数据统一走 worker/evidence/canonical 协议。
2. 不再把 `search_knowledge` 作为 main agent 常规可见工具暴露。
3. 为当前链路补 trace，记录 evidence、candidate、binding、projection 来源。

涉及文件：

- `backend/query/runtime.py`
- `backend/query/planner.py`
- `backend/query/runtime_tools.py`
- `backend/retrieval/service.py`
- `backend/tools/search_knowledge_tool.py`

完成标准：

1. trace 能看到 `worker_name / evidence_candidate_count / binding_source`。
2. 缺货场景能复现并定位为 `RAG evidence did not become dataset candidate`。

## Phase 1：建立 EvidenceEnvelope 与 Candidate 模型

目标：

1. 新增统一 evidence 数据模型。
2. RAG 返回从 `list[dict]` 适配为 `EvidenceEnvelope`。
3. 识别 `.xlsx/.csv/.pdf` 命中为 typed candidate。

涉及文件：

- 新增 `backend/query/evidence_models.py`
- 新增 `backend/query/evidence_adapter.py`
- `backend/retrieval/service.py`
- `backend/query/answer_finalizer.py`
- `backend/query/prompt_builder.py`

完成标准：

1. RAG 命中 `inventory.xlsx` 会产出 `DatasetCandidate`。
2. RAG 命中 PDF 会产出 `DocumentCandidate`。
3. raw snippets 默认 `debug_only`，只有 compact evidence summary 可进入 prompt。

## Phase 2：新增 RetrievalWorker，收口 search_knowledge 暴露

目标：

1. 将 RAG 从 main agent 工具循环中移出。
2. `search_knowledge` 保留为兼容 facade 或内部 worker 工具，不再默认暴露给 bounded_agent。
3. Runtime 通过 RetrievalWorker 获取 evidence。

涉及文件：

- 新增 `backend/query/retrieval_worker.py`
- `backend/query/runtime.py`
- `backend/query/runtime_tools.py`
- `backend/tools/definitions.py`
- `backend/tools/search_knowledge_tool.py`

完成标准：

1. direct RAG 不暴露工具 schema。
2. bounded agent 默认不因 `knowledge_lookup` 自动暴露 `search_knowledge`。
3. RAG 结果必须经 `EvidenceEnvelope` 再进入 prompt/output。

## Phase 3：BindingCandidateStore 与澄清承接

目标：

1. 保存上一轮 evidence candidates。
2. follow-up 澄清可以选择候选，而不是重新检索。
3. “是销售数据库”可以承接上一轮候选集。

涉及文件：

- 新增 `backend/query/evidence_store.py`
- `backend/query/followup_resolver.py`
- `backend/query/binding_resolver.py`
- `backend/query/runtime_context_state.py`
- `backend/structured_memory/session_memory.py`

完成标准：

1. `DatasetCandidate(inventory.xlsx)` 可被下一轮选择。
2. 澄清轮不再走 `bounded_agent + search_knowledge`。
3. session-memory 只存 candidate capsule，不存 raw snippets。

## Phase 4：StructuredDataWorker 接收 committed binding

目标：

1. 结构化数据链只从 committed binding 执行。
2. RAG candidate 被确认后，进入 StructuredDataWorker。
3. `structured_data_analysis_tool` 保持 facade，但核心执行可作为 worker 调用。

涉及文件：

- 新增 `backend/query/structured_data_worker.py`
- `backend/tools/structured_data_analysis_tool.py`
- `backend/structured_data/*`
- `backend/query/tool_input_resolver.py`
- `backend/query/binding_resolver.py`

完成标准：

1. 缺货场景最终调用 structured worker，而不是 RAG 工具循环。
2. 无 binding 时 fail-closed 或 clarify。
3. 结构化结果输出 canonical answer，不写 raw table dump。

## Phase 5：PDFWorker 与统一 WorkerResult 对齐

目标：

1. 将现有 `pdf_agent` 输出适配到 `WorkerResult / CanonicalResult`。
2. PDF output/projection 不再靠 runtime 专项分支。
3. PDF 降级结果统一走 projection policy。

涉及文件：

- `backend/pdf_agent/*`
- `backend/tools/pdf_analysis_tool.py`
- `backend/query/runtime_tools.py`
- `backend/query/runtime_output_policy.py`
- `backend/query/tool_output_adapter.py`

完成标准：

1. PDF ok/degraded/error 语义由 CanonicalResult 表达。
2. runtime 不再按 `tool_name == pdf_analysis` 做核心投影判断。
3. 页级失败不会进入 stable session summary。

## Phase 6：Evidence Orchestrator 统一调度

目标：

1. 将 RetrievalWorker、PDFWorker、StructuredDataWorker 纳入一个 orchestrator。
2. CapabilityDispatch 产出的 execution request 不直接等于 tool call。
3. direct RAG、PDF、structured、compound 子任务都走统一 worker handoff。

涉及文件：

- 新增 `backend/query/evidence_orchestrator.py`
- `backend/query/capability_dispatch.py`
- `backend/query/planner.py`
- `backend/query/runtime.py`
- `backend/query/models.py`

完成标准：

1. trace 中可以看到唯一 `worker_route`。
2. compound 场景中不同 worker 结果都以 canonical result 合并。
3. main agent 只在需要语言整合时参与，不承担检索/读取/分析循环。

## Phase 7：Prompt 与 Memory 去污染

目标：

1. prompt 只注入 model-visible evidence summary。
2. session-memory 只投影 canonical result / binding capsule。
3. raw worker output 只进 debug trace/artifact。

涉及文件：

- `backend/query/prompt_builder.py`
- `backend/context_management/context_controller.py`
- `backend/structured_memory/session_memory.py`
- `backend/structured_memory/session_memory_view.py`
- `backend/query/runtime_persistence.py`

完成标准：

1. `Historical Results` 不再出现 raw RAG/PDF/table dumps。
2. `Warm Flow Snapshots` 不再因检索噪声生成错误 flow，例如 `pdf_analysis_flow:是销售数据库`。
3. prompt visibility regression 能区分 model/debug sections。

## 10. 迁移与切换规则

### 10.1 Shadow First

先让 RetrievalWorker/EvidenceEnvelope 在 trace 中并行记录，不立刻替换全部主链。

### 10.2 Cutover Order

切换顺序固定：

1. RAG payload -> EvidenceEnvelope
2. Evidence candidates -> candidate store
3. follow-up selection -> binding commit
4. structured worker cutover
5. PDF worker result alignment
6. prompt/memory projection cleanup
7. old tool exposure cleanup

### 10.3 Rollback Rule

可以回滚：

1. 某个 worker 是否主用。
2. EvidenceEnvelope adapter 实现。
3. 某个 projection policy。

不可回滚：

1. raw output 不进主线程。
2. weak phrase 不直接强绑工具/数据集。
3. main agent 不持有高噪声工具自由循环权。
4. canonical result 才能进入 session-memory。

---

## 11. 明确不做什么

本轮不做：

1. 不引入 LangGraph/AutoGen 全量重写。
2. 不把 RAG/PDF/结构化数据做成三个能聊天的自治 agent。
3. 不继续给 `task_understanding.py` 加业务关键词补丁。
4. 不让 `search_knowledge` 继续作为 main agent 默认可见工具。
5. 不把 RAG 命中的表格片段当完整表格分析。
6. 不把 PDF/表格 raw output 写入 session-memory。

## 12. 初步完成标准

本计划书进入“可实施”状态，需要满足：

1. `EvidenceEnvelope` 模型和 worker handoff 边界确定。
2. RAG 命中 dataset/pdf 能产出 typed candidate。
3. 澄清轮能选择 candidate，而不是重新走 `bounded_agent + search_knowledge`。
4. 结构化数据分析只从 committed binding 执行。
5. PDF canonical result 与统一 WorkerResult 对齐。
6. Prompt builder 不注入 raw evidence。
7. session-memory 不投影 raw worker output。
8. 长场景缺货案例不再出现 RAG 工具循环。

## 13. 总结

这次重构的中心不是“多造几个 agent”，而是补上能力子系统的边界协议。

最终目标是：

```text
RAG 负责发现证据和候选
PDF 负责阅读和生成稳定文档结果
结构化数据负责对已绑定数据集做计算分析
主调度层负责仲裁、绑定、澄清和组合
主 agent 只负责必要的语言整合，不再背负高噪声工具循环
```

这条路线能同时解决：

1. 本地数据库被误当普通 RAG 的问题。
2. RAG 命中结构化数据后无法 handoff 的问题。
3. PDF raw/degraded 结果污染主线程的问题。
4. 结构化数据 binding 与 session restore 竞争的问题。
5. prompt/history/session-memory 多通道重复污染的问题。

后续实施应优先推进 Phase 1 到 Phase 3，因为它们直接决定“RAG 发现对象能否被后续轮次正确承接”。只有这部分稳定后，再推进 PDF 与结构化 worker 的深层统一，风险最低。

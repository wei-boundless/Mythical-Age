# 74-现状审查后的 A2A/MCP 证据子系统整体调整规划-20260425

> 目的：对照当前项目代码、`68/70/71/72/73` 计划书与 `docs/设计原则`，重新梳理下一步整体路线。本文不是推翻既有计划，而是把已经演进出的方向重新排序、收口，并指出需要调整的地方。

---

## 0. 总结论

当前正确主线应从 `68/70/71` 的“工具契约 + 证据 worker + handle 修复”，升级为 `72/73` 的正式主线：

```text
QueryRuntime 作为控制面 owner
  -> A2A-compatible 子 agent 通信层
  -> Retrieval / PDF / StructuredData agent
  -> MCP-compatible agent 内部工具层
  -> CanonicalResult / typed handle / projection
  -> Session runtime state / session memory / durable memory
```

原因：

1. `68` 解决的是工具事实散落问题，但不能单独解决 RAG/PDF/结构化数据之间的跨 worker 通信。
2. `70` 解决的是证据对象和 worker 协调，但协议仍是本项目私有模型。
3. `71` 解决的是长场景语义坏点和 handle 续接，但如果不接入协议层，会继续在 runtime/follow-up 里补丁式演化。
4. `72/73` 是当前最适合成为总控路线的计划：A2A 管 agent-agent，MCP 管 agent-tool，本项目自定义的 `SourceObjectRef / ArtifactHandle / ResultHandle / SubsetHandle` 只作为 extension。

因此，后续实施入口应改为：

> 以 `73` 为实施主清单，吸收 `68/70/71` 的未完成项，但不再并行维护三套主路线。

## 1. 当前代码真实状态

### 1.1 已经完成的部分

当前项目已经有较好的骨架：

1. `backend/query/worker_models.py`
   - 已有 `WorkerRequest / WorkerResult / WorkerExecutionPlan / CanonicalResult`。
   - 已有 `target_handle_id / upstream_object_handle_ids / upstream_result_handle_ids / owner_task_id / arbitration_reason`。

2. `backend/query/evidence_models.py`
   - 已有 `SourceObjectRef / EvidenceArtifact / ResultHandle / SubsetHandle / EvidenceEnvelope / BindingCandidate`。

3. `backend/query/evidence_graph.py`
   - 已有 session scoped 的 source object / artifact graph。
   - 已能表达 `source_object -> artifact` 的派生关系。

4. `backend/query/runtime.py`
   - `execution_kind == "worker"` 已优先于 direct tool 和旧 RAG 分支执行。
   - evidence state 已通过 `session_store.runtime_state.evidence_state` 恢复和持久化。

5. `backend/query/evidence_orchestrator.py`
   - 已有统一 worker 分发事件：`worker_start / worker_evidence / worker_artifacts / worker_end / done`。

6. `backend/tools/contracts.py` 与 `backend/tools/definitions.py`
   - 已有工具契约字段：resolution/output/projection/capability/safe_for_auto_route。
   - `search_knowledge.safe_for_auto_route=False`，主 agent 可见工具层已经开始收口。

### 1.2 仍处于过渡态的部分

这些是现在需要调整计划优先级的主要依据：

1. `WorkerRequest / WorkerResult` 还不是 A2A-compatible envelope。
   - 缺 `agent_id / message_id / protocol_version / extensions / task_status / stream_event_type`。

2. `EvidenceArtifactGraph` 目前只保存 source object 和 artifact。
   - `ResultHandle / SubsetHandle` 虽然有模型，但还没有进入统一 graph/store 生命周期。

3. `runtime.py` 中旧 `route == "rag" -> retrieval_service.retrieve()` 分支已从主链删除。
   - 非 worker RAG plan 只保留 `stale_rag_plan` 诊断，不再触发检索。

4. `runtime_followup.py` 已能把 PDF / structured follow-up 转 worker，但仍以 `binding_owner_task` 为主入口。
   - 这说明 `handle continuation` 还没有真正成为第一裁决源。

5. `structured_memory` 仍以 `committed_pdf / committed_dataset / owner_task_id` 为主要恢复字段。
   - session memory 还没有完全升级为 `active_object_handle_id / active_result_handle_id / active_subset_handle_id` 的控制面索引。

6. `MainContextState.to_prompt_block()` 仍会输出 `active_constraints` 的 key/value。
   - 这不是 raw dump，但仍可能把 path、binding identity、页码模式等执行状态作为 prose 暴露给模型，需要收紧为 allowlist。

7. `ToolRegistry.select_best()` 仍是打分式选择。
   - 已比关键词硬编码好，但还不是 `contract satisfiable` 的约束式选择。

8. `task_understanding.py / bundle_planner.py / followup_resolver.py` 中仍有不少 domain marker。
   - 部分是结构锚点抽取，可以保留。
   - 但如果这些 marker 直接决定执行 route，就违反“弱线索只能产生 candidate”的原则。

## 2. 与设计原则的对照结论

### 2.1 符合原则的方向

当前代码与计划中已经符合这些原则：

1. 默认隔离，显式共享
   - worker 结果通过 `CanonicalResult / WorkerProjection` 投影，不直接写主线程 raw output。

2. 工具事实单一来源
   - 工具能力、输出、解析契约已经向 `tools/definitions.py` 收拢。

3. RAG/PDF/结构化数据不再只是普通工具
   - `retrieval_worker / pdf_worker / structured_data_worker` 已存在，并进入 runtime worker 分支。

4. raw output 默认隔离
   - `ToolResultEnvelope` 与 output contract 已接入。

5. session runtime state 与 prompt/history 分离
   - `session_store.runtime_state` 已用于 evidence state。

### 2.2 需要修正的偏差

| 偏差 | 当前表现 | 调整方向 |
| --- | --- | --- |
| 协议仍是私有 worker 模型 | `WorkerRequest` 还不是 A2A-compatible | 先补协议元数据，不急着远程化 |
| follow-up 仍 owner-first | `binding_owner_task()` 仍是主入口 | 改为 handle-first，owner 只做 fallback |
| result/subset 生命周期不完整 | 模型存在，但 graph/store 不持有 | 建 `ResultHandleStore` 或扩展 `EvidenceGraphStore` |
| session memory 仍旧 owner 字段强 | `committed_pdf_owner_task_id` 仍用于恢复 | session memory 只保留 handle 索引 |
| prompt block 仍可能泄露控制细节 | `active_constraints` 全量序列化 | 改为 allowlist 和 masked handle |
| RAG 旧分支仍存在 | `route == "rag"` 可直接 retrieve | direct RAG 全部 primary worker，旧分支只留 fallback 且可观测 |
| tool registry 仍打分 | capability overlap score | 改为 capability/modality/scope/input satisfiable 过滤 |

## 3. 调整后的总设计

### 3.1 计划书层级重新排序

后续以如下优先级解释计划：

1. `72`：协议总设计
2. `73`：逐文件实施清单
3. `71`：语义坏点与 handle 修复细节
4. `70`：RAG/PDF/结构化数据证据对象与 handoff 细节
5. `68`：工具契约与 MCP-compatible tool layer 细节

也就是说：

```text
72/73 定主骨架
71 定 follow-up / handle / mixed-source 语义
70 定 evidence graph / worker handoff
68 定 tool contract / MCP-compatible adapter
```

### 3.2 控制面 owner

唯一控制面 owner 是 `QueryRuntime + Planner/Dispatch/Orchestrator`。

允许控制面决定：

1. 当前轮进入哪个 agent/worker。
2. 是否需要 retrieval 发现对象。
3. 是否要把 PDF 表格转交 structured-data agent。
4. 是否需要 mixed-source arbitration。
5. 是否要把 candidate commit 成 binding。

禁止 worker 自己决定：

1. 下一个 worker。
2. 当前轮最终 route。
3. 是否写 session/durable memory。
4. 是否把 raw evidence 暴露给主模型。

### 3.3 子 agent 与工具关系

固定关系：

```text
Agent = 能力边界与任务协作者
Tool = agent 内部执行单元
MCP-compatible contract = agent-tool 协议
A2A-compatible envelope = agent-agent 协议
```

当前三类 agent：

| agent | 责任 | 可产出 |
| --- | --- | --- |
| `agent:knowledge:retrieval` | 发现对象、候选、已有证据 | `EvidenceEnvelope / SourceObjectRef / Candidate` |
| `agent:document:pdf` | 文档阅读、页/节定位、表格抽取 | `CanonicalResult / ArtifactHandle / ResultHandle` |
| `agent:data:structured` | 数据集/表格分析、聚合、筛选 | `CanonicalResult / ResultHandle / SubsetHandle` |

### 3.4 handle 生命周期

调整后的 handle 生命周期：

```text
WorkerResult
  -> CanonicalResult
  -> object/result/subset handles
  -> runtime evidence/result state
  -> task registry refs
  -> session memory minimal index
  -> follow-up handle resolver
```

关键规则：

1. `SourceObjectRef / ArtifactHandle` 进入 evidence graph。
2. `ResultHandle / SubsetHandle` 必须进入 result store 或扩展后的 evidence graph。
3. `TaskRecord` 保存 handle 索引，但不保存 raw payload。
4. `SessionMemory` 只保存 active handle id，不保存 handle payload。
5. `PromptBlock` 只显示“available”，不显示 handle id 和 payload。

## 4. 调整后的实施顺序

## Phase 0：冻结总入口

目标：

1. 确认本文为当前“总规划入口”。
2. `73` 作为执行清单，`68/70/71` 只作为子专题依据。

完成标准：

1. 后续不再从 `68/70/71` 任意挑阶段并行实施。
2. 所有新增字段先对齐 `72/73` 的协议表。

## Phase 1：补 A2A-compatible envelope

目标：

1. 给 worker request/result/event 补齐协议元数据。
2. 不改变业务行为。

涉及文件：

- `backend/query/worker_models.py`
- `backend/query/evidence_orchestrator.py`
- `backend/query/capability_dispatch.py`
- `backend/query/runtime.py`

具体任务：

1. `WorkerRequest` 增加 `agent_id / message_id / protocol_version / extensions`。
2. `WorkerResult` 增加 `agent_id / task_status / stream_event_type / extensions`。
3. `CanonicalResult` 增加 `extensions`。
4. `worker_start / worker_end / done` 事件补 `agent_id / protocol_version / task_status`。
5. 建议新增 `backend/agents/a2a_models.py`，但先只做本地模型，不做网络 transport。

完成标准：

1. trace 中能看见每个 worker 对应稳定 `agent_id`。
2. 业务回归不变。
3. 后续远程化只替换 transport。

## Phase 2：handle continuation 主链切换

目标：

1. follow-up 优先按 `subset/result/object handle` 续接。
2. owner task 只做 fallback。

涉及文件：

- `backend/query/followup_resolver.py`
- `backend/query/runtime_followup.py`
- `backend/query/worker_projection.py`
- `backend/tasks/coordinator.py`
- `backend/query/runtime.py`

具体任务：

1. `FollowupResolution` 的 `resolution_scope` 真正成为执行入口。
2. PDF/结构化 follow-up 统一走 `binding_execution_from_resolution()`，不再暴露 owner-task 直回放主入口。
3. PDF follow-up 不再无条件继承旧 `pdf_mode/page`。
4. structured follow-up 优先读 `subset_handle_id / result_handle_id`。
5. `WorkerProjectionAdapter` 产出稳定 result/subset handle，并写入 task registry。

完成标准：

1. “这些人/这些仓库/上面的结果”续接 result/subset。
2. “回到全文/总结这份 PDF”不被上一轮页级 mode 污染。
3. 长场景 P0 类 PDF 路由问题可从 trace 看见被切断。

## Phase 3：ResultHandle / SubsetHandle 存储层补齐

目标：

1. 让 result/subset 不是只有 id 字符串，而是可恢复的结构对象。

涉及文件：

- `backend/query/evidence_models.py`
- `backend/query/evidence_graph.py`
- `backend/query/evidence_store.py`
- `backend/runtime/session_store.py`
- `backend/tasks/context_models.py`

具体任务：

1. 扩展 `EvidenceGraphStore`，或新增 `ResultHandleStore`。
2. 保存 `ResultHandle / SubsetHandle` 的 owner、source、artifact、locator、metadata。
3. runtime state 的 `evidence_state` 增加 result/subset payload。
4. task registry 只保存索引，不保存 payload。

完成标准：

1. session reload 后仍可按 result/subset handle 找回必要执行上下文。
2. prompt/history 不承担恢复职责。

## Phase 4：RAG 旧分支收口

目标：

1. direct RAG 全部走 retrieval agent。
2. 旧 `retrieval_service.retrieve()` runtime 分支删除，检索只由 retrieval agent 持有。

涉及文件：

- `backend/query/runtime.py`
- `backend/query/planner.py`
- `backend/query/capability_dispatch.py`
- `backend/query/retrieval_worker.py`

具体任务：

1. `route=rag` 的 primary plan 必须生成 `worker_route=retrieval`。
2. 非 worker RAG plan 只记录 `stale_rag_plan` 诊断，不进入检索。
3. 长场景中不再允许 runtime 直接产出 retrieval event。

完成标准：

1. `search_knowledge` 不再作为主 agent 可见工具。
2. RAG 命中 PDF/dataset/table 都产生 typed candidate，而不是 prose snippet 真相。

## Phase 5：Prompt 与 memory 投影收紧

目标：

1. 防止 handle、constraints、worker 中间态通过 prompt/session memory 泄露。

涉及文件：

- `backend/query/context_models.py`
- `backend/query/worker_projection.py`
- `backend/structured_memory/session_memory.py`
- `backend/structured_memory/process_state.py`
- `backend/memory/facade.py`

具体任务：

1. `MainContextState.to_prompt_block()` 对 `active_constraints` 改为 allowlist。
2. 禁止在 prompt block 中输出 path、row ids、artifact payload、retrieval snippet。
3. `session_memory` 增加 active object/result/subset handle 索引。
4. `committed_pdf_owner_task_id / committed_dataset_owner_task_id` 降级为兼容索引。
5. durable memory 只从 stable projection 提取候选，拒绝 degraded/raw/retrieval chunk。

完成标准：

1. 压缩会话或 session memory 恢复后，不会让旧 owner prose 重新裁决当前轮。
2. Key Results 不再承载跨任务 raw/prose 污染。

## Phase 6：MCP-compatible tool adapter

目标：

1. 明确工具契约是 agent 内部工具协议，不再让主线程知道高噪声工具细节。

涉及文件：

- `backend/tools/contracts.py`
- `backend/tools/definitions.py`
- `backend/tools/tool_registry.py`

2. `ToolRegistry.select_best()` 从打分式逐步改为 contract satisfiable。
3. 区分 `main-runtime visible` 与 `agent-internal only`。
4. `search_knowledge / pdf_analysis / structured_data_analysis` 明确作为 worker/agent 内部 facade 或兼容入口。

完成标准：

1. 新增 agent 内部工具时，不改 `QueryRuntime` 主语义。
2. 新增普通工具时，只改工具实现、definition 和测试。

## 6. 风险与验证矩阵

### 6.1 高风险点

1. follow-up 改为 handle-first 后，可能短期降低旧 owner 场景命中率。
2. 收紧 prompt constraints 后，模型可能少拿到一些上下文，需要靠 runtime 控制面补足。
3. RAG 旧分支删除过早，可能影响非 worker plan 的兼容场景。
4. ResultHandle/SubsetHandle 如果只存 id，不存 locator，会导致 session reload 后假续接。

### 6.2 必测回归

1. RAG 命中 `inventory.xlsx` -> 用户确认销售数据库 -> structured-data agent 分析。
2. RAG 命中 PDF -> 用户确认该文档 -> PDF agent 深读。
3. PDF 页级追问后 -> 再回全文总结，不继承旧 page mode。
4. PDF 表格 -> table artifact -> structured-data agent。
5. employees top5 -> “这些人按部门汇总”。
6. 缺货前三 -> “只展开第二个”。
7. mixed-source：结合知识库和 PDF 结论，不被单源 authority 劫持。
8. context compact 后，前端上下文余量和 session runtime state 一致恢复。
9. durable memory 不吸收 retrieval chunk、PDF raw、degraded diagnostics。
10. `search_knowledge` 不出现在 main agent prompt tool schema。

## 7. 最终执行判断

后续不要再把问题拆成“RAG 修一块、PDF 修一块、结构化数据修一块、记忆再修一块”。这些问题已经不是单模块问题，而是协议与所有权问题。

正确路线是：

```text
先把 A2A-compatible worker/agent 协议定住
再把 follow-up 切到 handle continuation
再补 result/subset 存储
再收 RAG/PDF/structured handoff
再收 prompt/memory 投影
最后再做 MCP adapter 与远程化
```

一句话：

> 现在不需要重写所有系统，但需要停止让旧 owner、旧 RAG、旧 prompt prose 继续当主事实源。新的主事实源必须是 typed handle + canonical result + runtime evidence state。

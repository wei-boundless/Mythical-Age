# 编排层运行时控制面重构计划书

日期：2026-04-26

## 1. 问题定义

现在的编排系统看起来已经有“控制塔”的形状，但它没有真正成为控制塔。它能展示 dry-run、行为节点、Skills 和契约 catalog，却没有拥有运行时的最终行为决策权。

当前失败模式是：

- 前端编排系统能解释“发生了什么”，但不能决定“应该怎么执行”。
- 后端编排模块主要产出 `behavior_trace` / `snapshot`，它在运行链路之后或旁边工作。
- 真正的行为决定散落在 planner、follow-up、continuation、capability dispatch、tool contract、memory facade、runtime execution 里。
- 测试系统跳到编排系统后，只能看链路，不能验证“预期编排计划”和“实际执行”是否一致。

所以这不是单纯的 UI 问题，而是 ownership 问题：

> 编排层目前是展示层，不是运行时控制面。

正确的目标状态是：

- 编排层产出一个可执行、可追踪、可验证的 `OrchestrationPlan`。
- Runtime 不再直接从零散 planner 字段判断执行分支，而是消费 `OrchestrationPlan`。
- Planner、Skills、工具契约、上下文、worker、输出边界成为编排计划的子策略或子决策。
- 前端编排系统展示和管理的不是“漂亮图”，而是 Agent 行为控制面。

## 2. 当前代码事实

### 2.1 真正的决策入口在 QueryPlanner

文件：`backend/query/planner.py`

当前 `QueryPlanner.build_plan()` 直接产出 `QueryPlan`。它内部完成：

- `analyze_memory_intent()`
- `analyze_query_understanding()`
- `continuation_resolver.resolve()`
- `_resolve_active_skill()`
- `dispatch_scheduler.resolve()`
- `binding_resolver.resolve()`
- `_worker_plan_from_dispatch()`
- direct tool / worker / agent 的 `execution_kind` 判断
- `single_execution / bundle_execution / explicit_fanout` 判断

这说明当前 `planner` 实际上已经承担了编排层的大部分职责，只是名字和结构还停留在“查询规划器”。

### 2.2 Runtime 直接消费 QueryPlan 和 QueryExecutionPlan

文件：`backend/query/runtime.py`

当前 `_execution_events()` 在 follow-up 逻辑之后调用 `_planner_build_plan()`，然后直接根据 `plan.execution_mode` 和 `execution.execution_kind` 分支执行：

- `bundle_execution` 进入 `_stream_bundle_execution()`
- `explicit_fanout` 进入 `task_coordinator.run_query_tasks()`
- `worker` 进入 `evidence_orchestrator.stream_execution()`
- `direct_tool` 进入 `_stream_direct_tool_execution()`
- 其余进入模型主链

也就是说，runtime 不认识一个独立的编排计划，它只认识 planner 产出的执行字段。

### 2.3 现有 orchestration 模块主要是影子解释层

文件：

- `backend/orchestration/behavior_trace.py`
- `backend/orchestration/behavior_dry_run.py`
- `backend/orchestration/contract_preview.py`
- `backend/api/orchestration.py`

它们当前做的是：

- 根据 `QueryPlan` 和 `QueryExecutionPlan` 构建 behavior snapshot。
- dry-run 复用 planner 构建计划，但不改变真实执行链。
- contract preview 预览工具契约。
- catalog API 展示 Skills 和 Tools。

这些能力有价值，但都是“解释和预览”，不是“控制和执行”。

### 2.4 CapabilityDispatchScheduler 仍是兼容式记录

文件：`backend/query/capability_dispatch.py`

类注释明确写着：当前实现 intentionally behavior-compatible，记录 dispatch decision，不改变 planner 现有 route。

这就是编排系统没用到实处的关键证据：能力调度看起来像控制点，但目前是 shadow pass。

### 2.5 前端问题是后端 ownership 问题的外显

文件：`frontend/src/components/workspace/views/ExperimentsView.tsx`

前端可以展示：

- 行为判读
- Skills 管理
- 工具管理
- dry-run
- 测试 turn 的 orchestration snapshot

但它展示的对象不是 runtime 的唯一执行计划。于是前端再怎么优化，也只能变成“观测台”，不能成为 Agent 开发平台。

## 3. 本地设计约束

基于当前代码和之前长场景问题，编排层重构必须遵守这些约束：

- **结构修复优先于 prompt 修复**：不要靠 seed 或 skill prompt 文字控制行为。
- **单一事实源**：一次请求最终只能有一个 canonical behavior plan。
- **restore 不等于 decide**：session / evidence / follow-up 可以恢复候选，但当前轮是否采用必须由编排计划仲裁。
- **弱信号只能提候选，不能直接改执行**：历史、summary、模糊 follow-up、candidate selection 都要经过明确 arbitration。
- **direct、worker、bundle、model 必须收敛到同一种计划结构**。
- **测试系统验证计划，而不是只验证 UI 链路**。
- **前端不能绕过安全边界**：前端可以编辑草案策略，不能直接打开危险工具或绕过 permission。

## 4. 推荐设计方向

将编排层升级为 `Agent Behavior Control Plane`。

中文仍可叫“编排系统”，但后端职责要变成：

> 产出、校验、追踪、对比和逐步接管 Agent 本轮行为计划。

核心变化：

- `QueryPlanner` 不再是事实上的总编排者，而是编排层内部的一个 planner adapter。
- `orchestration` 包新增 `OrchestrationPlanner`，负责把 memory intent、task understanding、continuation、skill policy、context policy、capability dispatch、contract preview、execution topology 组合成 `OrchestrationPlan`。
- `QueryRuntime` 改为消费 `OrchestrationPlan`，再生成兼容旧执行链的 `QueryPlan` / `QueryExecutionPlan`。
- 第一阶段不直接 cutover，先 shadow 生成 `OrchestrationPlan`，与旧 `QueryPlan` 对齐。
- 对齐稳定后逐步让 runtime 以 `OrchestrationPlan.execution_graph` 为准。

## 5. 目标数据模型

### 5.1 OrchestrationPlan

`OrchestrationPlan` 是编排层的 canonical truth。

建议放在：`backend/orchestration/models.py`

```python
@dataclass(slots=True)
class OrchestrationPlan:
    plan_id: str
    session_id: str
    input_text: str
    source: str
    mode: Literal["shadow", "primary"]
    behavior_policy_id: str
    topology: ExecutionTopology
    decisions: list[OrchestrationDecision]
    executions: list[OrchestrationExecution]
    context_policy: ContextPolicyDecision
    prompt_policy: PromptAssemblyDecision
    output_policy: OutputPolicyDecision
    safety: SafetyDecision
    diagnostics: dict[str, Any]
```

### 5.2 ExecutionTopology

表达本轮执行拓扑。

```python
@dataclass(slots=True)
class ExecutionTopology:
    mode: Literal["single_execution", "bundle_execution", "explicit_fanout"]
    route: str
    execution_kind: Literal["agent", "direct_tool", "worker"]
    reason: str
    branch_count: int
```

### 5.3 OrchestrationExecution

兼容当前 `QueryExecutionPlan`，但明确 owner 是编排层。

```python
@dataclass(slots=True)
class OrchestrationExecution:
    execution_id: str
    message: str
    route: str
    execution_kind: Literal["agent", "direct_tool", "worker"]
    skill_name: str
    tool_name: str
    worker_route: str
    tool_input: dict[str, Any]
    worker_request: dict[str, Any] | None
    structured_binding: dict[str, Any] | None
    arbitration: dict[str, Any]
    legacy_query_execution: Any | None = None
```

### 5.4 OrchestrationDecision

每个阶段的决策节点。

```python
@dataclass(slots=True)
class OrchestrationDecision:
    node_id: str
    node_type: str
    owner_module: str
    status: Literal["selected", "candidate", "blocked", "skipped", "warning"]
    inputs: dict[str, Any]
    outputs: dict[str, Any]
    reasons: list[str]
    risks: list[str]
```

### 5.5 BehaviorPolicy

前端以后管理的是策略，不是直接改执行字段。

```python
@dataclass(slots=True)
class BehaviorPolicy:
    policy_id: str
    name: str
    mode: Literal["readonly", "shadow", "primary"]
    route_policy: dict[str, Any]
    followup_policy: dict[str, Any]
    context_policy: dict[str, Any]
    skill_policy: dict[str, Any]
    tool_policy: dict[str, Any]
    worker_policy: dict[str, Any]
    output_policy: dict[str, Any]
```

第一阶段只读，第二阶段支持 shadow 策略，第三阶段才允许 primary。

## 6. 固定执行流

目标执行流如下：

```text
QueryRuntime._execution_events
  -> FollowupResolver 只做候选恢复和风险标注
  -> OrchestrationPlanner.build_plan(...)
       -> memory intent
       -> task understanding
       -> continuation arbitration
       -> execution topology
       -> skill policy
       -> context policy preview
       -> capability dispatch
       -> contract preview
       -> prompt policy
       -> output policy
       -> OrchestrationPlan
  -> emit orchestration_plan event
  -> OrchestrationPlanAdapter.to_query_plan(...)
  -> legacy runtime execution path
  -> collect actual execution trace
  -> compare plan vs actual
  -> persist orchestration trace for test/debug
```

第一阶段保留 adapter，让旧 runtime 不崩。后续再逐步把 runtime 分支改为直接消费 `OrchestrationExecution`。

## 7. 分阶段实施计划

### 阶段一：Shadow OrchestrationPlan

目标：不改变行为，只让编排层拥有完整计划对象。

改动：

- 新增 `backend/orchestration/models.py`
- 新增 `backend/orchestration/planner.py`
- 新增 `backend/orchestration/adapters.py`
- `OrchestrationPlanner` 内部先调用现有 `QueryPlanner.build_plan()`，再把结果转换成 `OrchestrationPlan`。
- `_execution_events()` 在旧 `plan` 旁边生成 shadow plan。
- SSE 增加 `orchestration_plan` 事件。

完成标准：

- 所有现有测试通过。
- shadow plan 与旧 plan 的 route、execution_mode、execution_kind 一致。
- 前端编排系统读取的是 `orchestration_plan`，不是再自己拼 snapshot。

### 阶段二：Decision Trace 标准化

目标：把分散的 reasons、contract、skill inspection、context preview 统一成决策节点。

改动：

- `SkillPolicyResolver.inspect()` 的结果进入 `OrchestrationDecision`。
- `CapabilityDispatchScheduler.resolve()` 输出进入 `OrchestrationDecision`。
- `build_contract_previews()` 输出进入 `OrchestrationDecision`。
- `memory_facade.inspect_query_context()` 输出进入 context decision。
- `behavior_trace.py` 改为从 `OrchestrationPlan.decisions` 渲染 snapshot。

完成标准：

- dry-run、live、test replay 三种来源使用同一套 decision schema。
- 测试系统能标出“预期节点”和“实际异常节点”。

### 阶段三：Runtime Cutover

目标：runtime 开始以 `OrchestrationPlan` 为准。

改动：

- 新增 `OrchestrationRuntimeAdapter`。
- `_stream_bundle_execution()`、`explicit_fanout`、`_stream_planned_execution()` 的入口逐步接收 `OrchestrationExecution`。
- `QueryPlan` 暂时保留为兼容层，但不再是 canonical truth。
- `execution_kind`、worker plan、tool input、skill prompt exposure 从 OrchestrationPlan 读取。

完成标准：

- direct tool、worker、bundle、model 主链都能从 OrchestrationPlan 执行。
- shadow diff 显示旧新计划一致后开启 primary mode。
- 可通过配置回滚到 legacy planner。

### 阶段四：策略管理与前端重做

目标：前端真正管理行为策略和验证结果。

改动：

- 编排系统页面改为：
  - 行为策略
  - 本轮计划
  - 计划与实际差异
  - Skills 策略
  - 契约策略
  - 上下文策略
- 删除只用于装饰的大图，以“计划对象 + 决策表 + diff + 跳转”为主。
- dry-run 直接返回 `OrchestrationPlan` 和风险提示。
- 测试系统点击异常后进入“计划差异视图”。

完成标准：

- 用户能看到“这轮为什么这样做”，也能看到“如果改策略会怎样”。
- 测试系统能验证编排策略，不只是跳转链路。

## 9. 测试与验证矩阵

### 单元测试

- 输入普通对话，shadow plan 与 legacy plan route 一致。
- 输入 direct tool 请求，tool input、contract preview、execution_kind 一致。
- 输入 PDF / RAG worker 请求，worker_route 和 worker_request 一致。
- 输入 bundle 请求，topology mode 和 executions 数量一致。
- 输入 follow-up 请求，followup candidate 不直接覆盖当前轮，必须留下 arbitration decision。

### 集成测试

- 冒烟测试必须看到 `orchestration_plan` 事件。
- 稳定门禁比较 plan vs actual。
- 长场景测试记录每一 turn 的 plan、actual、diff。

### 前端验证

- 编排系统能显示计划结论、决策节点、风险、diff。
- 点击测试异常能进入对应 turn 的计划差异。
- Skills / 契约页面展示的是策略对本轮计划的影响，而不只是 registry 清单。

## 10. Cutover 与回滚规则

### Shadow 模式

默认模式。运行旧链路，同时生成 `OrchestrationPlan`。不改变行为。

配置建议：

```json
{
  "orchestration_plan_mode": "shadow"
}
```

### Primary 模式

runtime 以 `OrchestrationPlan` 为准，legacy `QueryPlan` 只作为兼容对象。

开启条件：

- 冒烟、稳定门禁、核心长场景全部通过。
- plan vs actual diff 在关键字段上连续通过。
- direct tool、worker、bundle、model 主链都有覆盖。

### Rollback 模式

如果 primary 出现异常，配置回 `legacy`：

```json
{
  "orchestration_plan_mode": "legacy"
}
```

要求：

- legacy planner 保留到至少两个稳定版本之后。
- adapter 不删除旧 `QueryPlan` 字段。
- 前端可提示当前为 legacy mode。

## 11. 不允许做的事

- 不允许先重做前端大图。
- 不允许让前端直接绕过 permission 或 contract gate。
- 不允许用 prompt 文案当行为控制开关。
- 不允许让 follow-up restore 直接决定当前轮执行对象。
- 不允许 direct tool、worker、bundle 各自维护不同的计划结构。
- 不允许只记录成功路径，不记录 blocked / warning / fallback。

## 12. 结论

编排系统要有用，必须先从后端 ownership 上升级。

它的第一性职责不是展示链路，而是产出本轮 Agent 行为的 canonical plan。前端只有在这个 plan 存在之后，才有资格成为“开发平台”。否则它只能继续做一个漂亮但悬空的观测台。

下一步建议先实施阶段一：新增 `OrchestrationPlan` shadow 模式，不改变运行行为，只把真实控制对象建立起来。

## 13. 当前实施进度

当前阶段状态：**已收口**。

本轮重构已经完成从 `OrchestrationPlan` shadow 观测，到 primary 最小运行时接管，再到测试系统、编排系统、Prompt 透明化和问题节点定位的闭环。默认模式仍保持 `shadow`，这是安全策略，不代表计划未完成；`primary` 已具备手动切换、专项验证和回滚能力。

已完成：

- 阶段一的 `OrchestrationPlan` shadow 对象已经落地。
- Runtime 每轮会发出 `orchestration_plan` SSE 事件。
- Dry-run 结果会附带 `orchestration_plan`。
- 新增 `orchestration_plan_mode` 配置，支持 `legacy / shadow / primary` 三个值。
- `legacy` 会关闭 plan 事件，`shadow` 为默认模式；`primary` 已接入最小运行时控制面，会以 `OrchestrationPlan.executions` 对 legacy execution 做匹配、排序和安全回退。
- 新增 `orchestration_diff`，在最终 done/error 前比较 plan 与 actual 的关键字段。
- 测试系统的 turn orchestration replay 会从事件中提取 `orchestration_plan` 和 `orchestration_diff`。
- 前端编排页可以显示 plan mode，并在行为判读中显示计划校验状态。
- `behavior_trace.py` 已开始优先使用 `OrchestrationPlan.decisions` 覆盖节点来源，节点会带上 `orchestration_plan_id` 和对应 decision id，避免行为图继续从 legacy plan 二次推断。
- 长场景 turn artifact 会同时保存 `orchestration_plan` 和 `orchestration_diff` 顶层字段，测试回放即使缺少 SSE 事件也能读到计划与偏移。
- long-run issue 生成已接入 `orchestration_diff`，当 diff 为 `mismatch` 时会把它加入 failed checks，并在 issue summary 中直接写出“编排计划偏移”。
- smoke / stable 的 in-process SSE smoke 已保存独立 `*.orchestration.json` 产物，并把 `orchestration_diff=mismatch` 纳入失败判定和 issue summary。
- Runtime 已新增 `actual_trace` 累积层，会在最终 diff 前收集 `context_management`、`memory_context`、`prompt_manifest`、tool contract、worker/tool 落点等运行时观测。
- `orchestration_diff` 已从 route/tool/worker 的粗粒度对比，扩展到 context policy、prompt manifest、direct tool contract 的观测校验。context/prompt 缺失先标记为 `warning`，contract deny 会标记为 `mismatch`。
- `contract-policy` 已进入 `OrchestrationPlan.decisions`。Runtime 会在 plan 生成前做一次 contract preview，并把 preview_count、blocked_count、contract_action、permission 结果写入计划，使工具契约从“执行后才知道”提前到“计划阶段可预期”。
- 行为图和测试回放已能识别 `contract-policy`，并将其映射到契约/工具节点，方便直接定位契约预检和实际执行不一致的问题。
- `actual_trace` 已开始标准化多执行分支。`subtask_start/subtask_end`、`worker_start/worker_end`、`tool_start/tool_end` 会被归并到 `actual.executions[]`，用于比较 bundle、explicit fanout、worker/direct tool 的每条分支。
- `orchestration_diff` 已新增 `executions.count`、`executions[i].execution_id`、`executions[i].execution_kind`、`executions[i].tool_name`、`executions[i].worker_route` 对比。多分支场景不再只比较第一条 execution。
- 前端编排页已新增“执行分支校验”面板，会解析 `orchestration_diff.items` 里的 `executions[i]` 字段，显式展示第几条分支、预期/实际落点、task_id、bundle_item_id 和 mismatch 原因。
- 测试系统跳转到编排系统时，如果是 bundle / fanout / worker 偏移，可以直接在分支卡片里看到偏移位置，不再只能看整体 plan mismatch。
- 执行分支卡片已接入事件时间线过滤。点击某条分支后，前端会用 `execution_id`、`task_id`、`subtask_index`、`bundle_item_id`、`tool_name`、`worker_route` 等标识筛出相关 SSE / worker / tool 事件，并提供“查看全部事件”回退。
- 前端过滤逻辑不会把缺失的 `subtask_index` 默认为 0，避免第一条分支被误命中；如果分支缺少可匹配标识，会明确保持未过滤状态，而不是制造假的定位感。
- Runtime 已为 worker、direct tool、模型工具调用事件统一补充分支标识。`worker_start/worker_end`、direct tool 的 `tool_start/tool_end`、模型中途工具调用都会尽量带上 `execution_id`、`execution_kind`、`subtask_plan`、`bundle_item`、`worker_route` 等字段，让测试回放与前端过滤读取真实事件，而不是二次猜测。
- `orchestration_diff` 的 actual execution 归并已优先读取事件上的 `execution_id`，并且不再把缺失的 `subtask_index` 强行公开成 `0`。
- `orchestration_diff.actual.executions[]` 已增加分支输出可读字段：`summary_preview`、`content_preview`、`output_chars`。这些字段用于前端 debug，不参与 mismatch 判定，避免把输出文案变化变成门禁噪音。
- 前端执行分支卡片已展示分支输出预览。现在点击测试系统进入编排系统时，不仅能看到第几条分支偏移，也能直接看到该分支最后交给输出层的摘要或内容片段。
- Answer assembler 汇总已进入 actual 观测。`orchestration_diff.actual.answer_assembly` 会记录 `selected_task_ids`、`selected_count`、最终答案预览和 `task_summary_refs`，用于判断最终答案由哪些分支结果组成。

- `orchestration_diff.actual.answer_assembly` 已透出 `dropped_count`、`dropped_segments`、`dedupe_targets` 和 `source_refs`。前端“输出汇总”区会展示被丢弃分支的原因，避免只看到“没进最终答案”却不知道为什么。
- 已新增 `OrchestrationRuntimeAdapter` 的最小 primary cutover 入口。primary 模式下 runtime 会先读取 `OrchestrationPlan.executions`，按计划中的 execution id 对 legacy `QueryExecutionPlan` 进行匹配和排序，再进入原有执行函数。

- 前端运行控制面不会静默改变默认模式，用户可以手动切换 primary，也可以随时回退 shadow / legacy。
- 前端初始化的长期记忆默认路径已修正为 `durable_memory/index/MEMORY.md`，避免工作台启动时因旧路径 `durable_memory/MEMORY.md` 触发 404 和错误浮层。

- 前端 `Prompt 装配` 节点已新增“Prompt 装配来源”可读面板，直接展示每段 prompt 的序号、层级、来源文件/函数、字符数、是否模型可见和 preview，不再需要先展开原始 JSON 才能分析 prompt 来源。

# 健康 Agent 现有权限体系组装方案

日期：2026-05-09

约束：

- 不改权限框架。
- 不新增 `OperationRegistry` 操作。
- 不新增编排系统层级。
- 不新增任务系统抽象。
- 只在现有 AgentRuntimeProfile、TaskFlowRegistry、HealthCommandService、runtime_admission 链路内完成健康管理 Agent 组装。

## 一、权限系统实际怎么运作

当前项目里的权限不是单点系统，而是三层组合：

### 1. PermissionService：工具可见性 / 工具调用外层权限

位置：

- `backend/permissions/service.py`
- `backend/permissions/decision_pipeline.py`
- `backend/permissions/policy.py`

作用：

- 根据 `permission_mode` 判断工具是否可见、可直接调用。
- `plan` 模式只允许只读工具。
- `default` 模式阻止 shell / destructive。
- `bypass` 才会放开。

这层主要管工具池，不是健康 Agent 任务绑定的核心。

### 2. AgentRuntimeProfile：Agent 自身能力边界

位置：

- `backend/orchestration/agent_runtime_registry.py`
- `backend/orchestration/agent_runtime_models.py`

作用：

- 限定 agent 能接哪些 task mode。
- 限定 runtime lane。
- 限定 allowed / blocked operations。
- 限定 memory scope。
- 限定 context sections。
- 限定 output contracts。

健康 Agent 的 profile 已存在：

```text
agent_id = agent:3
agent_profile_id = health_maintainer_agent
allowed_task_modes = issue_triage, trace_analysis, case_draft, fix_verification
allowed_runtime_lanes = health_issue_read, health_trace_read, prompt_trace_read, memory_trace_read, runtime_trace_read, assertion_trace_read, case_draft_candidate, fix_verification_candidate
allowed_operations = op.model_response, op.read_file, op.search_text, op.memory_read
blocked_operations = op.write_file, op.edit_file, op.shell, op.python_repl, op.memory_write_candidate, op.agent_bounded
allowed_memory_scopes = issue_local_readonly, health_trace_readonly
allowed_context_sections = task, health_issue, runtime_trace, prompt_manifest, memory_runtime_view, assertions, runtime_contracts
output_contracts = empty
```

### 3. ResourcePolicy / OperationGate：运行时放行

位置：

- `backend/orchestration/runtime_loop/model_adoption.py`
- `backend/orchestration/resource_gate.py`
- `backend/health_system/registry.py`

作用：

- `model_adoption.py` 从 task operation requirement + AgentRuntimeProfile 生成可执行 ResourcePolicy。
- `OperationGate` 只放行已 adopted、runtime_executable 的 ResourcePolicy 里的 operation。
- 健康系统自己的 `_build_health_resource_policy()` 也复用 profile 的 allowed / blocked operations。

所以现有系统已经能做到：

- profile 不允许的 operation 被拒绝。
- blocked operation 被拒绝。
- 没有 adopted ResourcePolicy 的 operation 被拒绝。

不需要新增权限框架。

## 二、健康 Agent 当前卡在哪里

### 卡点 1：output_contracts 没配置

健康 flow 已经定义：

```text
flow.health.issue_triage       -> HealthTriageResult
flow.health.trace_analysis     -> HealthTraceAnalysis
flow.health.case_draft         -> HealthCaseDraftProposal
flow.health.fix_verification   -> HealthFixVerificationProposal
```

位置：

- `backend/tasks/flow_registry.py:78`
- `backend/tasks/flow_registry.py:92`
- `backend/tasks/flow_registry.py:106`
- `backend/tasks/flow_registry.py:120`

但 `agent:3` 的 `output_contracts` 为空。

`TaskFlowRegistry.build_binding_for_flow()` 会强校验 output contract：

- `backend/tasks/flow_registry.py:2607`

所以绑定失败，健康 preview 被挡住。

最小处理：

```text
output_contracts =
  HealthTriageResult
  HealthTraceAnalysis
  HealthCaseDraftProposal
  HealthFixVerificationProposal
```

注意：这是配置补齐，不是框架变更。

### 卡点 2：健康命令准入已经有，但还不够严

现有准入：

- `backend/health_system/runtime_admission.py`

只对这三个命令进入 Agent 执行：

```text
analyze_trace
draft_case
verify_fix
```

它会检查：

- issue 是否存在。
- flow binding 是否 valid。
- requested operation 是否在 profile allowed / blocked 内。
- runtime lane 是否允许。

这条链路是对的，不需要绕到编排框架里扩新东西。

需要补的只是健康系统自己的准入规则：

```text
agent:3 只能对 health_issue 发起执行命令。
target_ref / payload.issue_id 必须指向 HealthStore 里的 issue。
initiator_type=agent 时 initiator_ref 必须是 agent:3。
requested_operations 默认只能是 op.model_response。
如果显式请求 op.write_file / op.shell / op.agent_bounded，继续 rejected。
```

### 卡点 3：四个 flow 有 workflow，但模板不完整

已有 workflow：

- `workflow.health.issue_triage`
- `workflow.health.trace_analysis`
- `workflow.health.case_draft`
- `workflow.health.fix_verification`

位置：

- `backend/tasks/workflow_registry.py:118`
- `backend/tasks/workflow_registry.py:137`
- `backend/tasks/workflow_registry.py:156`
- `backend/tasks/workflow_registry.py:175`

但目前明确看到的健康模板只有：

- `template.health.issue_triage`
- `backend/tasks/template_registry.py:257`

如果不补模板，后面三个模式可能只能靠 specific task / workflow 兜底，组装结果不稳定。

最小处理：

- 补 `template.health.trace_analysis`
- 补 `template.health.case_draft`
- 补 `template.health.fix_verification`

只补模板数据，不改任务系统结构。

## 三、健康管理 Agent 应该怎么“有用”

在不扩权限框架的前提下，健康 Agent 的管理能力不应该通过新 operation 表达，而应该通过已有健康命令链表达。

也就是：

```text
健康 Agent / 健康窗口
  -> HealthCommandService.submit_command()
  -> runtime_admission 检查
  -> HealthRegistry.execute_agent_run()
  -> 生成 HealthReport / HealthAgentRun / HealthManagementReceipt
  -> HealthStore 持久化健康系统自己的记录
```

健康 Agent 能做的管理动作：

```text
report_issue
analyze_trace
draft_case
verify_fix
launch_health_test
build_cutover_readiness
```

这些命令已经在 `HealthCommandService` 里存在：

- `report_issue`: `backend/health_system/command_service.py:60`
- `analyze_trace / draft_case / verify_fix`: `backend/health_system/command_service.py:105`
- `launch_health_test`: `backend/health_system/command_service.py:179`
- `build_cutover_readiness`: `backend/health_system/command_service.py:234`

所以不需要新增 operation。只要把健康 Agent 的入口限制为这组命令，并把命令结果写入 receipt / report / run，就已经是管理。

## 四、最小组装清单

### 1. AgentRuntimeProfile

保留现有 allowed / blocked operations。

只补 output contracts：

```text
HealthTriageResult
HealthTraceAnalysis
HealthCaseDraftProposal
HealthFixVerificationProposal
```

不要加：

```text
op.write_file
op.edit_file
op.shell
op.agent_bounded
op.memory_write_candidate
```

### 2. Task flow / workflow

保留现有四个 flow：

```text
flow.health.issue_triage
flow.health.trace_analysis
flow.health.case_draft
flow.health.fix_verification
```

保留现有四个 workflow。

补齐缺失模板即可，不动 FlowRegistry 架构。

### 3. HealthCommandService 准入

在现有 `runtime_admission` 和 command handling 内收紧：

```text
agent 发起时 initiator_ref 必须是 agent:3。
agent 执行类 command 的 target_scope 必须是 health_issue / issue。
target_ref 必须存在于 HealthStore。
requested_operations 必须是 profile allowed 且不在 blocked。
```

非 Agent 执行类命令继续走 HealthCommandService 自己的业务逻辑，不拉进 RuntimeLoop。

### 4. 健康证据边界

不新增框架，只在 health_system 内做校验：

```text
runtime_trace_refs 必须是 runtime-loop / task-run / health-test 这类可解释引用。
prompt_manifest_refs 必须是 prompt-manifest 引用。
memory_refs 只能作为引用进入 health issue，不自动开放长期记忆写入。
conversation_ref 只能作为证据引用，不自动把其它会话全文塞进模型上下文。
```

这属于健康系统自己的输入校验，不是权限框架变更。

## 五、为什么这条路最薄

不新增 `op.health.*`，因为现有系统还没有把健康命令注册成 OperationDescriptor，强加一套会把 capability / orchestration / task 三层都加厚。

不改 `OperationGate`，因为现有 OperationGate 已经能阻止越权 operation。

不改 `model_adoption.py`，因为健康 Agent 当前失败不是 adoption 逻辑坏，而是 profile 输出合同没配。

不改 `TaskFlowRegistry.build_binding_for_flow()`，因为它的严格校验对任务绑定是有价值的。健康 Agent 缺合同，就补合同，不把校验放松。

## 六、建议实施顺序

1. 补 `agent:3` 默认 profile 和本地 profile 的四个 `output_contracts`。
2. 补三个健康模板：`trace_analysis`、`case_draft`、`fix_verification`。
3. 收紧 `HealthCommandService / runtime_admission` 的 agent 发起命令边界。
4. 加健康 evidence refs 校验。
5. 跑健康专项测试，确保原来的 `output_contract_not_allowed` 消失，`op.write_file` 仍然 rejected。

## 七、验收标准

必须满足：

```text
flow.health.* 对 agent:3 binding 全部 valid。
health preview 对合法 issue 返回 ready。
analyze_trace / draft_case / verify_fix 可以生成 HealthAgentRun / HealthReport / Receipt。
requested_operations=["op.write_file"] 继续 rejected。
agent:3 不能对非 health_issue target 执行健康 Agent run。
缺证据或非法 refs 时 preview/command 返回 blocked，而不是静默执行。
```

## 八、最终判断

在现有系统里组装健康管理 Agent 的关键不是加权限框架，而是把已有链路配置完整：

- profile 补合同。
- 模板补齐。
- 健康命令准入收紧。
- 健康证据输入校验。

这样健康 Agent 能管理健康系统，同时不增加编排系统和任务系统厚度。

## 九、按“前端管理系统组装”的复核结论

本轮复核按约束执行：

- 只能通过现有前端管理系统组装。
- 不直接改后端配置文件。
- 不新增健康层。
- 不新增 `op.health.*`。
- 健康系统作为任务域进入任务系统。

结论：

```text
任务系统前端可以组装健康任务域的大部分任务对象；
编排系统前端目前不能完整组装 agent:3 的运行边界；
任务系统前端目前也不能把单任务执行策略明确绑定到 agent:3；
所以健康管理 Agent 不能只靠当前前端闭环组装成功。
```

这不是健康系统需要新权限层，而是现有管理台缺少两个配置面：

```text
AgentRuntimeProfile.output_contracts 的前端编辑 / 后端保存链路
TaskExecutionPolicy.default_agent_id 或固定 Agent 绑定的前端编辑 / 后端保存链路
```

## 十、前端现在已经能做什么

### 1. 能建健康任务域

任务系统前端已有任务域管理能力：

- `TaskSystemView.tsx` 支持新建 / 编辑 task domain。
- `backend/api/tasks.py` 提供 `PUT /tasks/domains/{domain_id}`。
- `TaskFlowRegistry.upsert_task_domain()` 能落库。

因此可以从前端创建：

```text
domain_id = domain.health
task_family = health
title = 健康任务域
```

### 2. 能建健康特定任务

任务系统前端已有 specific task 编辑能力：

- 可编辑 `task_id`
- 可编辑 `task_family`
- 可编辑 `task_mode`
- 可编辑 `input_contract_id`
- 可编辑 `output_contract_id`
- 可编辑 `default_flow_contract_id`
- 可编辑 `default_workflow_id`

后端 `PUT /tasks/specific-records/{task_id}` 会保存这些字段。

因此可以从前端创建：

```text
task.health.issue_triage
task.health.trace_analysis
task.health.case_draft
task.health.fix_verification
```

并分别绑定：

```text
flow.health.issue_triage       + workflow.health.issue_triage       + HealthTriageResult
flow.health.trace_analysis     + workflow.health.trace_analysis     + HealthTraceAnalysis
flow.health.case_draft         + workflow.health.case_draft         + HealthCaseDraftProposal
flow.health.fix_verification   + workflow.health.fix_verification   + HealthFixVerificationProposal
```

### 3. 能建 / 编辑 workflow 和输出契约

任务系统前端已有 workflow 编辑和输出契约选择：

- `TaskContractPanel`
- `ContractLibraryPanel`
- `TaskSystemView.saveTaskStack()`
- `PUT /tasks/workflows/{workflow_id}`

所以健康 workflow 的 `output_contract_id` 可以通过任务系统前端保存。

### 4. 能配置 flow contract binding

任务系统前端已有 flow contract binding 编辑：

- `flow_contract_id`
- `override_policy`
- `verification_gate_profile`
- `fallback_policy`

后端 `PUT /tasks/flow-contract-bindings/{task_id}` 会保存。

所以健康任务可以从前端绑定现有 `flow.health.*`。

### 5. 能做图节点级 agent:3 绑定

协调任务图编辑器可以在节点检查器里选择 Agent：

- `CoordinationEditorWorkbench.tsx`
- 字段：`graph_nodes[].agent_id`
- Agent 选项来自 A2A catalog 的 `agent_cards`

这意味着：

```text
如果健康管理 Agent 以协调任务图节点方式运行，
前端可以把某个节点绑定为 agent:3。
```

但这只解决图节点绑定，不解决单任务执行策略里的默认 Agent。

## 十一、前端当前不能完成的部分

### 缺口 1：编排管理台不能编辑 output_contracts

前端类型里有 `output_contracts`，但实际编辑链路把它清空：

```text
OrchestrationView.runtimeDraftFrom()
  output_contracts: []

OrchestrationView.runtimePayloadFromDraft()
  output_contracts: []
```

后端也丢弃 payload：

```text
backend/api/orchestration.py
  AgentRuntimeRegistry.upsert_profile(..., output_contracts=())
```

结果：

```text
即使前端未来传了 output_contracts，当前后端也不会保存。
```

这直接导致健康 flow 绑定校验失败：

```text
TaskFlowRegistry.build_binding_for_flow()
  _validate_contains(..., "output_contract", flow.output_contract_id, profile.output_contracts)
```

当前 `agent:3.output_contracts = ()`，所以四个健康 flow 会落到：

```text
output_contract_not_allowed
```

### 缺口 2：内置 agent:3 的 runtime profile 被锁定

后端 `AgentRuntimeRegistry.upsert_profile()` 当前拒绝编辑 builtin agent：

```text
if current_agent is not None and current_agent.builtin:
    raise PermissionError("system builtin agent runtime profile is locked")
```

所以即使补上前端 output contract 选择器，当前 API 仍不能通过管理台编辑 `agent:3`。

这不是健康系统的权限问题，是编排管理台对系统内置 Agent 的管理策略还没有“受控配置窗口”。

### 缺口 3：任务执行策略不能从前端明确绑定 agent:3

任务运行配置前端只有：

```text
runtime_agent_selection_policy
allowed_agent_categories
allow_worker_agent_spawn
```

没有 `default_agent_id` 字段。

后端保存 execution policy 时强制写死：

```text
TaskFlowRegistry.upsert_task_agent_adoption_plan(
  default_agent_id="agent:0",
)
```

结果：

```text
单任务级健康任务即使设置为 health task family，
保存执行策略后默认 Agent 仍然是 agent:0。
```

这会让“健康管理 Agent”无法通过单任务装配稳定接管健康任务。

### 缺口 4：健康模板不完整，且前端没有模板注册管理入口

当前代码里能明确看到：

```text
template.health.issue_triage
```

但没有完整的：

```text
template.health.trace_analysis
template.health.case_draft
template.health.fix_verification
```

任务系统前端目前主要管理 domain / specific task / workflow / binding / policy / graph，不是完整的 `TaskTemplateRegistry` 模板管理台。

这意味着：

```text
通过前端可以手动建 specific task 规避模板缺失；
但不能从前端把健康模板库补齐。
```

如果目标是“前端管理系统完整组装”，这里也是缺口。

## 十二、最薄修复建议

不动权限框架，不动健康层，不新增 operation。只补管理台缺失字段。

### 第一优先级：让编排管理台能受控配置内置 Agent 的 output_contracts

最小范围：

```text
frontend/src/components/workspace/views/OrchestrationView.tsx
frontend/src/components/workspace/views/orchestration/OrchestrationAgentConfigWorkbenches.tsx
backend/api/orchestration.py
backend/orchestration/agent_runtime_registry.py
```

目标不是放开所有 builtin profile 字段，而是支持受控字段：

```text
agent:3.output_contracts
```

推荐规则：

```text
内置 Agent 仍锁定 operations / lanes / memory scopes；
只允许通过管理台补 output_contracts；
output_contracts 必须来自 contract catalog；
保存后重新跑 flow binding preflight。
```

这样不会把健康 Agent 放大成万能管理 Agent。

### 第二优先级：任务执行策略增加固定 Agent 绑定

最小范围：

```text
frontend/src/lib/api.ts
frontend/src/components/workspace/views/task-system/TaskRunLoopWorkbenchPanel.tsx
frontend/src/components/workspace/views/TaskSystemView.tsx
backend/api/tasks.py
backend/tasks/flow_registry.py
```

目标：

```text
TaskExecutionPolicy 增加 default_agent_id
前端 Agent 选择可选 agent:3
后端不再写死 default_agent_id="agent:0"
```

健康任务建议配置：

```text
default_agent_id = agent:3
allowed_agent_categories = system_management_agent
runtime_agent_selection_policy = fixed_agent
allow_worker_agent_spawn = false
```

### 第三优先级：先不补模板管理台，先用 specific task 闭环

为了不加厚任务系统，模板库不是第一刀。

短期通过前端创建四个 specific task 已经能表达健康任务域。模板缺口只记录，不急着扩 UI。

只有当后续要求“从意图自动选择健康模板”时，再补 `TaskTemplateRegistry` 管理入口。

## 十三、按当前前端可执行的临时组装路径

在不改代码前，前端能做的最大闭环是：

1. 在任务系统中新建 / 确认 `domain.health`。
2. 新建四个 `task.health.*` specific task。
3. 每个 task 绑定对应 `workflow.health.*`、`flow.health.*`、`Health*` 输出契约。
4. 在协调任务图里创建健康管理图节点。
5. 将节点 Agent 绑定为 `agent:3`。
6. 用任务预检面板检查 workflow manifest / node assembly。

但必须接受两个限制：

```text
flow.health.* 对 agent:3 的底层 binding 仍可能因为 output_contract_not_allowed 失败；
单任务运行配置仍不能稳定把 default agent 保存为 agent:3。
```

所以这只能作为诊断路径，不是最终组装成功。

## 十四、最终修复边界

这次不应该修：

```text
不要新增 health permission layer
不要新增 op.health.*
不要让健康 Agent 获得 write_file / shell / edit_file
不要绕过 TaskFlowRegistry 的 output_contract 校验
不要把 agent:3 改成非 builtin 来逃避锁
不要直接改配置文件假装前端能组装
```

应该修：

```text
让前端管理台能保存 agent:3 的允许输出契约；
让任务执行策略能从前端保存 default_agent_id=agent:3；
保留所有运行时权限校验；
用 preflight 证明健康任务域能组装。
```

一句话：

```text
健康系统不需要新层，任务域可以承载；
现在缺的是管理台把已有 AgentRuntimeProfile 和 TaskExecutionPolicy 配完整的能力。
```

## 十五、2026-05-09 复核后的最小修复结果

本次已再次追查两个疑似缺口，确认不是误判：

```text
AgentRuntimeProfile.output_contracts：
  模型和 API 请求字段存在，但前端草稿清空，API 保存时也强制清空。

TaskExecutionPolicy.default_agent_id：
  TaskAgentAdoptionPlan 和 registry 支持保存，但 API 请求模型没有字段，
  保存时写死 agent:0，to_dict 返回前端时还会移除 default_agent_id。
```

已做最小修复：

- `agent:3` 默认 runtime profile 补齐四个健康输出契约。
- 编排管理台运行档案页增加“允许输出契约”配置。

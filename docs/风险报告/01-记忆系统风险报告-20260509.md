# 记忆系统风险报告

日期：2026-05-09

范围：
- `backend/memory_system`
- `backend/structured_memory`
- `backend/query/runtime.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/api/memory.py`

## 一、结论摘要

当前记忆系统的基础回归测试可以通过，但系统风险主要集中在架构边界与失败治理，而不是单点语法错误。

记忆系统目前不是一个单层系统，而是三类机制并存：

1. `memory_system`：新的统一门面、运行时供应链、治理 API、工作记忆层。
2. `structured_memory`：旧的会话记忆、长期记忆、Markdown 存储、抽取、索引和旧格式修复实现。
3. `WorkingMemoryService`：新的工作记忆 SQLite 存储与任务运行期治理模型。

主要风险是：新系统承担对外边界，但旧系统仍承担核心实现；运行时提交失败会被弱化为 warning；部分构造和读取路径存在隐式写盘；工作记忆收尾和长期记忆晋升还没有形成完整事务语义。

专项测试结果：

```powershell
python -m pytest backend/tests/memory_system_contracts_regression.py backend/tests/working_memory_stage1_regression.py backend/tests/working_memory_api_regression.py backend/tests/runtime_commit_gate_regression.py -q
```

结果：

## 二、系统结构现状

### 2.1 新旧系统边界

`memory_system` 是当前对外统一入口：

- `MemoryFacade`
- `SessionMemoryLayer`
- `DurableMemoryLayer`
- `MemoryBundleService`
- `WorkingMemoryService`
- `WorkingMemoryFinalizer`
- `MemoryWritebackBuilderService`
- `DurableMemoryGovernanceService`

但 `memory_system` 内部仍大量依赖 `structured_memory`：

- `DurableMemoryLayer` 内部创建 `structured_memory.memory_manager.MemoryManager`
- `DurableMemoryLayer` 内部创建 `structured_memory.extractor.MemoryExtractor`
- `DurableMemoryLayer` 内部创建 `structured_memory.extraction_scheduler.ExtractionScheduler`
- `SessionMemoryLayer` 内部返回 `structured_memory.session_memory.SessionMemoryManager`
- `memory_system.legacy_types` 从 `structured_memory` 重新导出类型

因此，当前真实结构是：

```text
运行时 / API
  -> memory_system
       -> structured_memory
       -> working_memory SQLite
```

这说明 `memory_system` 已经是架构边界，但 `structured_memory` 仍是会话记忆和长期记忆的核心实现层。

### 2.2 代码规模

当前两个核心目录规模：

| 目录 | 文件数 | 行数 |
|---|---:|---:|
| `backend/memory_system` | 34 | 6536 |
| `backend/structured_memory` | 24 | 5784 |

两者合计约 12320 行。规模本身不是问题，但新旧职责长期并存会带来重复修改、边界漂移和治理规则不一致。

## 三、主要风险

### R1. 记忆提交失败会被误报为已应用

严重级别：高

位置：
- `backend/query/runtime.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`

现象：

`QueryRuntime` 在 assistant 消息提交后会刷新 session memory，并提交 durable memory extraction。如果刷新或提交失败，当前逻辑只记录 warning，然后继续返回：

- `session_memory_chars = 0`
- `durable_saved_count = 0`

运行环后续将提交结果转换为记忆提交状态时，使用 `durable_saved_count >= 0` 判断 `durable_memory_commit_applied=True`。

风险：

以下三种情况会混在一起：

1. 长期记忆提交成功，但没有产生新记忆。
2. 长期记忆提交失败，被 warning 吞掉。
3. 提交流程没有真实完成。

影响：

- 运行链路会显示记忆写入已应用，但实际可能失败。
- 健康系统和测试系统可能误判记忆层正常。
- 后续追踪“为什么没有记住”时缺少结构化失败状态。

建议：

提交结果应拆成明确字段：

- `session_memory_attempted`
- `session_memory_succeeded`
- `session_memory_error`
- `durable_memory_attempted`
- `durable_memory_succeeded`
- `durable_memory_error`
- `durable_saved_count`

`durable_saved_count=0` 只能表示保存数量，不能表示成功。

### R2. `MemoryManager` 构造时自动迁移旧文件

严重级别：高

位置：
- `backend/structured_memory/memory_manager.py`

现象：

`MemoryManager.__init__()` 中会调用 `_migrate_legacy_layout()`。该迁移逻辑会移动旧 Markdown 文件，目标存在且内容一致时还会删除源文件。

风险：

实例化对象本应是低风险操作，但当前实例化可能产生磁盘副作用。

影响：

- 难以区分“读记忆”与“迁移记忆”。
- 测试、预览、API 初始化都可能触发文件迁移。
- 如果迁移逻辑有 bug，会在非维护场景下改动用户数据。

建议：

将 legacy migration 从构造器移出，改为显式维护命令，例如：

- `python -m maintenance.migrate_durable_memory_layout --dry-run`
- `python -m maintenance.migrate_durable_memory_layout --apply`

运行时构造只允许确保必要目录存在，不应移动或删除已有内容。

### R3. 新旧记忆层职责重叠

严重级别：中高

位置：
- `backend/memory_system`
- `backend/structured_memory`

现象：

`memory_system` 已经承担统一入口和治理职责，但长期记忆、会话记忆、抽取、索引、旧格式修复仍由 `structured_memory` 承担。

风险：

- 新功能可能绕过 `memory_system` 直接调用 `structured_memory`。
- 治理规则可能在两层重复实现。
- 后续工作记忆、长期记忆、上下文装配改动需要同时理解两套边界。

影响：

- 重构成本上升。
- 测试容易只覆盖门面，不覆盖底层副作用。
- 清理旧代码时容易误删仍被使用的旧实现。

建议：

短期不应直接删除 `structured_memory`，但应明确规则：

1. `memory_system` 是唯一对外入口。
2. 新代码不得直接依赖 `structured_memory`。
3. `structured_memory` 暂时标记为内部实现库。
4. 分阶段把仍需要的核心能力迁入 `memory_system` 或更名为内部 storage/engine 模块。

### R4. 工作记忆收尾失败可能阻断最终响应

严重级别：中高

位置：
- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/memory_system/working_memory_finalizer.py`

现象：

运行环在输出 done 事件前调用 `finalize_working_memory()`。该逻辑会查询工作记忆条目、批量更新生命周期、写归档报告。

风险：

如果 SQLite 写入失败、归档报告写入失败、单条 item 更新失败，目前缺少独立降级边界。

影响：

- 主任务已经完成，但最终响应可能被工作记忆收尾失败阻断。
- 用户侧看到任务异常，实际失败点只是治理收尾。
- 运行环职责进一步膨胀。

建议：

工作记忆收尾应变成 fail-soft：

- 收尾失败写 `working_memory_finalization_failed` 事件。
- done 事件仍应输出。
- `output_commit.memory.working_memory_finalization_succeeded=false`。
- 健康系统记录后续修复项。

### R5. 工作记忆晋升长期记忆缺少事务语义

严重级别：中

位置：
- `backend/memory_system/facade.py`
- `backend/memory_system/working_memory_store.py`
- `backend/structured_memory/memory_manager.py`

现象：

`promote_working_memory_item_to_durable()` 会读取工作记忆 item，然后创建 durable note。工作记忆使用 SQLite，长期记忆使用 Markdown 文件和索引。

风险：

SQLite 与 Markdown 文件不是同一事务系统。

可能出现：

- durable note 写成功，但 working memory 状态更新失败。
- working memory 状态更新成功，但 durable index 同步失败。
- API 返回异常后实际已有部分写入。

影响：

- 晋升链路不可完全追溯。
- UI 可能显示候选仍待晋升，但 durable memory 已有对应 note。
- 后续治理可能产生重复长期记忆。

建议：

引入 promotion transaction 或补偿记录：

- `promotion_transaction_id`
- `source_work_memory_id`
- `target_durable_note`
- `durable_write_status`
- `working_memory_update_status`
- `index_sync_status`
- `rollback_or_repair_hint`

### R6. 记忆供应链有过多 legacy adapter

严重级别：中

位置：
- `backend/memory_system/facade.py`
- `backend/memory_system/bundle_service.py`
- `backend/query/runtime.py`

现象：

代码中仍存在多处 legacy 命名或兼容入口：

- `legacy_adapter`
- `legacy_session_memory_block_result`
- `legacy_inspect_query_context_result`
- `legacy_session_memory_refresh_removed`
- `legacy_durable_memory_extraction_removed`

这些入口有些是合理的过渡标记，但长期存在会削弱边界。

风险：

- 调用方不清楚哪个是正式路径。
- 测试可能继续依赖旧入口。
- 旧兼容层会被误认为业务功能。

建议：

建立 legacy API 清单：

| 入口 | 当前调用方 | 是否保留 | 替代路径 | 删除条件 |
|---|---|---|---|---|
| `build_session_memory_block` | 待确认 | 过渡 | `build_memory_context_package_result` | 无调用后删除 |
| `compact_history_for_query` | 待确认 | 过渡 | `inspect_memory_context_compaction` | 无调用后删除 |
| `legacy_types` | 多处导入 | 暂留 | 显式模型导入 | 新边界稳定后删除 |

## 四、建议整改顺序

### 第一阶段：修正失败语义

目标：不改大结构，先避免误报。

任务：

1. 修改 assistant commit 后的记忆提交结果结构。
2. 区分 attempted/succeeded/failed/saved_count。
3. 修正 `_memory_commit_state_from_assistant_commit_result()`。
4. 增加回归测试：
   - session memory refresh 抛异常时，状态不得显示成功。
   - durable memory commit 抛异常时，状态不得显示成功。
   - durable commit 成功但 saved_count=0 时，应显示 succeeded=true、saved_count=0。

### 第二阶段：工作记忆收尾隔离

目标：工作记忆治理失败不阻断最终响应。

任务：

1. `finalize_working_memory()` 外层增加失败边界。
2. 写入失败事件和诊断字段。
3. done 事件携带 finalization 失败状态。
4. 增加 SQLite/归档报告写入失败测试。

### 第三阶段：迁移显式化

目标：构造器不再移动或删除用户数据。

任务：

1. 从 `MemoryManager.__init__()` 移除 `_migrate_legacy_layout()`。
2. 新增维护命令，支持 dry-run 和 apply。
3. 补迁移测试。
4. 在启动诊断中提示“发现旧布局，但未自动迁移”。

### 第四阶段：边界收口

目标：明确 `memory_system` 是唯一入口。

任务：

1. 扫描所有直接导入 `structured_memory` 的生产代码。
2. 能通过 `memory_system` 访问的，改走 `memory_system`。
3. 不能迁移的，登记为内部实现依赖。
4. 给 `structured_memory` 增加内部实现说明，禁止新代码直接依赖。

### 第五阶段：长期瘦身

目标：减少双层长期并存。

任务：

1. 梳理 `structured_memory` 中仍被使用的模块。
2. 删除 demo、废弃兼容、无调用旧入口。
3. 将必要能力迁入更明确的内部模块。
4. 用回归测试保护 session/durable/working memory 三类契约。

## 五、后续审查建议

记忆系统之后，建议按以下顺序审查下一个系统：

1. `orchestration/runtime_loop`：运行环过大，是当前最大结构风险。
2. `tasks/flow_registry.py`：配置读取、默认覆盖、读写副作用较多。
3. `capability_system`：能力单元多，容易存在旧工具残留和重复能力。
4. `health_system`：测试/健康治理本身会影响运行时判断，适合单独审查。

## 六、当前判断

记忆系统不建议立刻大拆。

正确策略是：

1. 先修失败语义。
2. 再隔离收尾失败。
3. 然后显式化旧布局迁移。
4. 最后再收口新旧边界和瘦身。

这样可以避免在工作记忆重构尚未完全收尾时，把长期记忆、会话记忆、上下文装配一起打散。

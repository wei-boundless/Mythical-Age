# 任务系统风险追踪与修复计划

日期：2026-05-09

关联报告：
- `docs/风险报告/03-任务系统与FlowRegistry风险报告-20260509.md`

目标：

把任务系统当前暴露出的 `FlowRegistry` 风险从“现象”追踪到“可修复边界”，为后续代码修复提供实施顺序。

本文件只整理追踪结论和修复计划，不改业务代码。

## 一、当前核心问题

任务系统当前最关键的问题不是单个测试失败，而是三类语义混在一起：

1. 显式用户配置：用户通过 API / UI 保存到 `storage/tasks/*.json` 的配置。
2. 默认种子配置：代码内置的默认 flow、workflow、contract、domain 等。
3. 运行派生配置：从 specific task / general profile 派生出来的 projection binding、flow contract binding、execution policy、memory request profile 等。

现在多个接口把这三类数据混在同一个 `list_*` 结果里，导致：

- 管理台统计把派生对象算成真实配置。
- 空目录也会显示多个绑定和策略。
- 部分读取接口会把派生对象写回 JSON。
- 删除时难以判断删的是显式配置、默认种子，还是派生结果。
- 旧字段 `task_agent_adoption_plan` 和新字段 `task_execution_policy` 仍并行存在。

## 二、复现追踪

### 2.1 专项测试失败

命令：

```powershell
python -m pytest backend/tests/task_system_api_regression.py backend/tests/task_graph_registry_test.py backend/tests/task_template_registry_regression.py backend/tests/contract_compiler_workflow_test.py backend/tests/task_contract_registry_test.py backend/tests/contract_compiler_coordination_test.py backend/tests/runtime_assembly_builder_test.py -q
```

结果：

```text
1 failed, 39 passed
```

失败点：

```text
backend/tests/task_template_registry_regression.py:115
assert payload["summary"]["projection_binding_count"] == 0
E assert 4 == 0
```

直接原因：

`TaskFlowRegistry(tmp_path).build_overview()` 在空目录下返回：

```text
projection_binding_count = 4
flow_contract_binding_count = 4
adoption_plan_count = 4
memory_request_profile_count = 4
```

这些不是用户显式配置，而是从默认 task records 派生出来的运行配置。

### 2.2 空目录副作用复现

复现方式：

在全新 `.tmp/flowregistry-read-side-effect-repro` 下调用：

```python
TaskFlowRegistry(root).build_overview()
```

观察结果：

```text
before []
summary {
  "specific_task_count": 4,
  "task_flow_count": 4,
  "task_domain_count": 1,
  "projection_binding_count": 4,
  "flow_contract_binding_count": 4,
  "adoption_plan_count": 4,
  "memory_request_profile_count": 4
}
after
storage/orchestration/agents.json
storage/tasks/task_domains.json
```

结论：

- 空目录调用 overview 会显示派生出来的绑定和策略。
- 当前空目录下不一定把所有派生绑定写盘。
- 但至少会触发 agent / task domain 的默认文件落盘。
- 因此问题要拆成两类：统计语义污染、读取写盘副作用。

在空目录逐个调用 `list_*`：

|---|---:|---|
| `list_general_task_profiles()` | 0 | 否 |
| `list_flows()` | 4 | 否 |
| `list_task_assignments()` | 4 | 否 |
| `list_task_domains()` | 1 | 是，`task_domains.json` |
| `list_specific_task_records()` | 4 | 否 |
| `list_projection_bindings()` | 4 | 否 |
| `list_flow_contract_bindings()` | 4 | 否 |
| `list_task_agent_adoption_plans()` | 4 | 否 |
| `list_task_memory_request_profiles()` | 4 | 否 |
| `list_coordination_tasks()` | 0 | 否 |
| `list_task_graphs()` | 0 | 否 |
| `list_topology_templates()` | 0 | 否 |
| `list_task_communication_protocols()` | 0 | 否 |
| `TaskWorkflowRegistry.list_workflows()` | 8 | 否 |
| `TaskContractRegistry.list_contract_specs()` | 5 | 否 |

结论：

空目录下最明显的直接写盘点是 `list_task_domains()`；其他接口主要是返回派生数据但暂不写盘。

### 2.4 已存在空 JSON 文件时的覆盖复现

当对应 JSON 文件已存在但内容为空数组时：

| 文件 / 接口 | 返回数量 | 存储后数量 | 是否被改写 |
|---|---:|---:|---|
| `flows.json` / `list_flows()` | 4 | 0 | 否 |
| `specific_task_records.json` / `list_specific_task_records()` | 4 | 4 | 是 |
| `task_projection_bindings.json` / `list_projection_bindings()` | 4 | 4 | 是 |
| `task_flow_contract_bindings.json` / `list_flow_contract_bindings()` | 4 | 4 | 是 |
| `task_agent_adoption_plans.json` / `list_task_agent_adoption_plans()` | 4 | 4 | 是 |
| `task_memory_request_profiles.json` / `list_task_memory_request_profiles()` | 4 | 4 | 是 |

结论：

危险点在“文件已经存在但为空或不完整”时更明显。管理台保存、手工编辑、合并冲突、迁移中断都可能制造这种状态，然后一次读取就把派生默认项写回，掩盖原始状态。

### 2.5 坏 JSON 文件复现

当 JSON 文件内容损坏：

| 文件 / 接口 | 返回数量 | 文件是否保持损坏 |
|---|---:|---|
| `flows.json` / `list_flows()` | 4 | 是 |
| `task_projection_bindings.json` / `list_projection_bindings()` | 4 | 是 |
| `task_domains.json` / `list_task_domains()` | 1 | 否，被默认 domain 覆盖 |

结论：

`_read_json()` 会吞掉解析异常并返回 fallback。部分接口随后会写回默认值，导致坏配置证据丢失。

问题：

以下对象会从 general profile / specific task record 派生默认项：

- projection binding
- flow contract binding
- task execution policy / adoption plan
- memory request profile

这些对象对运行装配有用，但不应该默认等同于用户显式配置。

### 3.4 API 管理台统计直接使用派生列表

位置：
- `backend/api/tasks.py:225`
- `backend/api/tasks.py:235`
- `backend/api/tasks.py:274`

问题：

`_task_system_payload()` 直接调用：

- `registry.list_projection_bindings()`
- `registry.list_flow_contract_bindings()`
- `registry.list_task_agent_adoption_plans()`
- `registry.list_task_memory_request_profiles()`

然后用长度作为 summary count。

这会让管理台认为空目录已有 4 个 projection binding、4 个 flow contract binding、4 个 execution policy、4 个 memory profile。

### 3.5 前端依赖这些数组做选中任务详情

位置：
- `frontend/src/components/workspace/views/TaskSystemView.tsx:1008`
- `frontend/src/components/workspace/views/TaskSystemView.tsx:1009`
- `frontend/src/components/workspace/views/TaskSystemView.tsx:1010`
- `frontend/src/components/workspace/views/TaskSystemView.tsx:1011`

问题：

前端会按 `task_id` 从这些数组里找到当前任务的绑定和策略。如果后端改成只返回显式配置，前端当前详情面板可能会显示为空。

因此修复不能简单把派生对象从返回里删掉，而应该给对象加来源标记，或者拆分 explicit / derived 两个视图。

### 3.6 运行装配依赖默认派生策略

位置：
- `backend/tasks/assembly_builder.py:202`
- `backend/tasks/assembly_builder.py:203`
- `backend/tasks/assembly_builder.py:205`
- `backend/tasks/assembly_support.py:324`
- `backend/tasks/assembly_support.py:325`

问题：

运行装配会通过 `get_projection_binding()`、`get_flow_contract_binding()`、`get_task_memory_request_profile()` 获取策略。当前这些 getter 底层依赖 `list_*` 返回派生默认项。

因此第一阶段不宜直接改变 runtime getter 的默认派生能力，否则运行装配会缺默认策略。

## 四、修复原则

### 原则 1：运行态可以用派生默认，管理态必须区分来源

运行装配需要默认策略兜底，这个能力可以保留。

但管理台和 overview 必须区分：

- `explicit`：用户显式存储。
- `derived`：从任务记录派生。
- `default_seed`：代码内置默认种子。
- `invalid`：读取失败或解析失败。

### 原则 2：读取不能写盘

`list_*` 默认应当纯读取和纯计算。

如需标准化或迁移，必须通过显式方法：

- `normalize_task_storage()`
- `migrate_task_storage()`
- `repair_task_storage()`

### 原则 3：坏配置不能被默认值覆盖

坏 JSON 应该进入 diagnostics，不应该被自动覆盖。

运行态关键配置遇到坏 JSON，需要根据重要性 fail-closed 或降级并报告。

### 原则 4：不要用测试期望掩盖结构问题

当前失败测试不能简单改成 `projection_binding_count == 4`。

正确修复应该让 summary 的口径清楚：

- 如果统计显式配置，空目录应为 0。
- 如果统计运行可用配置，应明确叫 `effective_projection_binding_count`。

## 五、建议数据模型调整

### 5.1 管理台返回结构

建议保留原数组以减少前端破坏，但为每项增加 metadata/source：

```json
{
  "binding_id": "taskprojbind:task.dev.light_web_game",
  "task_id": "task.dev.light_web_game",
  "source_kind": "derived",
  "source_ref": "task.dev.light_web_game",
  "editable": false
}
```

summary 拆分为：

```json
{
  "projection_binding_count": 0,
  "derived_projection_binding_count": 4,
  "effective_projection_binding_count": 4
}
```

其中：

- `projection_binding_count` 表示显式存储数量。
- `derived_projection_binding_count` 表示派生数量。
- `effective_projection_binding_count` 表示运行可用数量。

### 5.2 Registry 内部读取结构

建议引入内部 helper，不一定一开始公开：

```python
@dataclass(frozen=True)
class RegistryPayload:
    payload: dict[str, Any]
    exists: bool
    valid: bool
    error: str = ""
```

用于替代当前单纯返回 dict 的 `_read_json()`。

### 5.3 显式列表和有效列表分离

建议为运行配置类对象增加两类方法：

```python
list_explicit_projection_bindings()
list_effective_projection_bindings()
```

原 `list_projection_bindings()` 可临时保留为 effective，降低第一阶段破坏面。

管理台使用 explicit + effective summary；运行装配继续使用 effective getter。

## 六、实施阶段

### 阶段 1：修复统计语义和测试失败

目标：

不破坏运行装配，先让管理台和 overview 分清显式配置与派生配置。

修改点：

1. 在 `TaskFlowRegistry` 中新增 explicit 读取方法：
   - `list_explicit_projection_bindings()`
   - `list_explicit_flow_contract_bindings()`
   - `list_explicit_task_agent_adoption_plans()`
   - `list_explicit_task_memory_request_profiles()`
2. `build_overview()` summary 改为：
   - 显式数量使用 explicit。
   - 新增 derived/effective 数量。
3. `backend/api/tasks.py::_task_system_payload()` 同步 summary 口径。
4. 前端若只展示详情，仍可使用 effective 数组；列表/统计展示显式数量。
5. 修复当前失败测试，新增断言：
   - 空目录显式数量为 0。
   - effective 数量为 4。

预期影响：

- 运行装配不变。
- 管理台统计变准确。
- 当前失败测试应该通过。

### 阶段 2：去除读取写盘副作用

目标：

所有 `list_*` 默认不写盘。

修改点：

1. 移除 `list_task_domains()` 自动写回。
2. 移除已有文件为空时的自动补全写回。
3. 新增显式 `normalize_task_storage()`。
4. 对需要迁移的场景由 API 或 CLI 显式触发。
5. 新增回归测试：
   - 空目录调用所有 `list_*` 后不产生 `storage/tasks/*.json`。
   - 已存在空 JSON 文件调用 `list_*` 后仍保持空 JSON。

预期影响：

- Git diff 噪音降低。
- 读取和迁移职责分离。

### 阶段 3：坏配置 diagnostics

目标：

坏 JSON 不被静默吞掉。

修改点：

1. `_read_json()` 改为结构化读取结果。
2. 保留兼容 wrapper，但内部记录 diagnostics。
3. `build_overview()` 增加：
   - `registry_read_errors`
   - `storage_health`
4. 对坏 JSON 增加测试：
   - overview 报错。
   - 不覆盖坏文件。

预期影响：

- 管理台能看到配置损坏。
- 后续修复可以做 fail-closed。

### 阶段 4：旧协议收口

目标：

清理 `task_agent_adoption_plan` legacy 残留。

修改点：

1. 装配输出只保留 `task_execution_policy`。
2. runtime 内部只读新字段。
3. 入口适配器临时兼容旧 payload。
4. 删除 `to_legacy_dict()` 写回路径。
5. 删除或重写旧测试。

前置条件：

阶段 1-3 完成后再做。否则 legacy 清理会和统计/读取语义混在一起，风险偏高。

## 八、修复风险评估

### 8.1 不能直接让 `list_projection_bindings()` 只返回显式配置

原因：

运行装配依赖 getter 取到默认策略。

受影响位置：
- `backend/tasks/assembly_builder.py:202`
- `backend/tasks/assembly_builder.py:203`
- `backend/tasks/assembly_builder.py:205`
- `backend/tasks/assembly_support.py:324`
- `backend/tasks/assembly_support.py:325`

正确做法：

先新增 explicit/effective 双路径，运行态继续走 effective。

### 8.2 不能简单删除 default seeds

原因：

默认 flow、workflow、contract 是运行基础。

正确做法：

把 default seed 从“显式配置统计”中排除，而不是立刻删除。

### 8.3 前端需要确认统计口径

前端当前在任务详情中依赖 effective 数组：

- `frontend/src/components/workspace/views/TaskSystemView.tsx:1008`
- `frontend/src/components/workspace/views/TaskSystemView.tsx:1011`

第一阶段不强改前端详情逻辑，只需要确保卡片统计和列表标签能区分显式/派生。

## 九、建议立刻执行的修复顺序

1. `TaskFlowRegistry` 增加 explicit/effective 分离方法。
2. `build_overview()` 调整 summary，保留原 effective arrays。
3. `backend/api/tasks.py` 调整 summary，新增 effective/derived 统计。
4. 修复 `task_template_registry_regression.py` 当前失败测试。
5. 增加“空目录派生统计”回归测试。
6. 跑任务系统专项测试。
7. 再进入阶段 2，移除读取写盘。

这条路径风险最低，因为它先解决当前失败和管理台误导，不会破坏运行态默认策略。

## 十、验收标准

第一阶段完成后：

- 当前任务系统专项测试全绿。
- 空目录 overview 中：
  - 显式 projection binding 数量为 0。
  - effective projection binding 数量为 4。
  - derived projection binding 数量为 4。
- API summary 和本地 `TaskFlowRegistry.build_overview()` 统计口径一致。
- 运行装配测试仍通过。

第二阶段完成后：

- 空目录调用 registry 读取接口不会创建 `storage/tasks/*.json`。
- 已存在空 JSON 文件不会被读取接口填充默认派生项。
- normalize/migrate 只能通过显式方法触发。

第三阶段完成后：

- 坏 JSON 不会被自动覆盖。
- overview 能展示 registry read error。
- 关键运行链路对坏配置有明确 fail-closed 或降级诊断。

# 任务系统瘦身删改破坏面评估

日期：2026-05-09

关联文件：
- `docs/风险报告/03-任务系统与FlowRegistry风险报告-20260509.md`
- `docs/风险报告/04-任务系统风险追踪与修复计划-20260509.md`

## 一、结论摘要

任务系统确实需要瘦身，但不能直接从 `list_*` 或默认派生逻辑开始大面积删除。

当前最危险的不是“默认派生存在”，而是：

1. 读取接口有写盘副作用。
2. upsert / delete 使用 effective 列表作为写回基底。
3. 派生对象会被固化成显式配置。
4. 管理态和运行态共用同一套读取语义。

因此瘦身要分成两类：

- 可以直接切掉的增肥源：把派生对象写成显式 JSON 的路径。
- 不能直接删的运行兜底：运行装配依赖 effective 默认策略，必须先替代再删。

## 二、当前删改候选点

### 2.1 `list_*` 读取时写盘

涉及位置：

- `backend/tasks/flow_registry.py:820`
- `backend/tasks/flow_registry.py:887`
- `backend/tasks/flow_registry.py:961`
- `backend/tasks/flow_registry.py:1175`
- `backend/tasks/flow_registry.py:1560`
- `backend/tasks/flow_registry.py:1656`
- `backend/tasks/flow_registry.py:1749`
- `backend/tasks/flow_registry.py:1920`
- `backend/tasks/flow_registry.py:2084`
- `backend/tasks/flow_registry.py:2112`
- `backend/tasks/flow_registry.py:2264`
- `backend/tasks/flow_registry.py:2303`
- `backend/tasks/workflow_registry.py:294`

风险：

查询、概览、前端打开页面、运行装配，都可能触发写盘。这个会造成：

- Git diff 噪音。
- 空文件被自动填充。
- 坏 JSON 被默认值掩盖。
- 派生对象变成持久配置。

删改判断：

可以删，但不能一次性全删。应先从运行策略四类对象开始：

- projection binding
- flow contract binding
- task execution policy / adoption plan
- memory request profile

原因：

这四类是“派生配置固化”的主要来源，且已有 explicit/effective 分离基础。

### 2.2 upsert 使用 effective 列表写回

涉及位置：

- `backend/tasks/flow_registry.py:1621`
- `backend/tasks/flow_registry.py:1706`
- `backend/tasks/flow_registry.py:1816`
- `backend/tasks/flow_registry.py:1992`

现象：

例如 `upsert_projection_binding()` 当前会先取：

```text
self.list_projection_bindings()
```

这不是显式配置列表，而是 effective 列表，里面包含从默认任务派生出的绑定。

风险：

用户只保存 1 个 projection binding，最终 JSON 可能写入 4 个默认派生绑定 + 1 个用户配置。

这就是“越修越多”的关键增肥源。

删改判断：

优先修。应把基底改为 explicit 列表：

- `list_explicit_projection_bindings()`
- `list_explicit_flow_contract_bindings()`
- `list_explicit_task_agent_adoption_plans()`
- `list_explicit_task_memory_request_profiles()`

这样不会破坏运行态 effective 读取，也能阻止派生配置落盘。

### 2.3 delete 使用 effective 列表写回

涉及位置：

- `backend/tasks/flow_registry.py:1091`
- `backend/tasks/flow_registry.py:1095`
- `backend/tasks/flow_registry.py:1099`
- `backend/tasks/flow_registry.py:1103`
- `backend/tasks/flow_registry.py:1376`
- `backend/tasks/flow_registry.py:1380`
- `backend/tasks/flow_registry.py:1384`
- `backend/tasks/flow_registry.py:1388`

现象：

`delete_task_domain()` 和 `delete_specific_task_record()` 会过滤 effective 列表后写回 JSON。

风险：

删除本来应该减少配置，但因为 effective 列表混入了默认派生对象，删除动作反而可能把剩余派生对象固化进 JSON。

删改判断：

优先修。删除四类运行策略配置时，应只过滤 explicit 列表。

## 三、不能直接删除的点

### R1. effective 默认策略不能直接删

涉及位置：

- `backend/tasks/assembly_builder.py:202`
- `backend/tasks/assembly_builder.py:203`
- `backend/tasks/assembly_builder.py:205`
- `backend/tasks/assembly_support.py:324`
- `backend/tasks/assembly_support.py:325`

运行装配依赖：

- `get_projection_binding()`
- `get_flow_contract_binding()`
- `get_task_memory_request_profile()`

这些 getter 当前底层依赖 effective 列表。

直接删除后果：

- 任务装配拿不到默认 projection。
- flow contract 可能为空。
- memory request profile 可能为空。
- RunLoop 后续看到的装配结果会变少，可能触发新的 fallback。

判断：

不能直接删。应保留 effective 读取，先只限制写路径。

### R2. default task flow 不能直接删

涉及位置：

- `backend/tasks/flow_registry.py:75`
- `backend/tasks/flow_registry.py:791`
- `backend/tasks/assembly_support.py:86`
- `backend/tasks/assembly_support.py:95`
- `backend/health_system/execution_planner.py:59`
- `backend/api/orchestration.py:311`

直接删除后果：

- 工作流解析少默认 flow。
- 健康系统 execution planner 可能找不到 flow。
- 编排页面 task scope 选项减少。

判断：

不能作为第一批瘦身对象。需要先确认每个默认 flow 是否仍有入口引用。

### R3. specific task records 的默认派生不能直接删

涉及位置：

- `backend/tasks/flow_registry.py:1131`
- `backend/api/tasks.py:248`
- `backend/orchestration/runtime_loop/langgraph_coordination_runtime.py:641`
- `backend/orchestration/runtime_loop/task_run_loop.py:3159`

直接删除后果：

- 协调运行态拿不到 specific tasks。
- task scope 管理台和 API 数据减少。
- 默认任务装配可能缺 record。

判断：

不能直接删。需要先确认 `task_definition` / `task_graph` 是否能接管 specific task 的运行身份。

### R4. coordination task 到 task graph 的派生不能直接删

涉及位置：

- `backend/tasks/flow_registry.py:2099`
- `backend/tasks/flow_registry.py:2107`
- `backend/tasks/flow_registry.py:2112`

直接删除后果：

- 旧 coordination task 可能不再出现在 task graph 管理视图。
- API `_task_system_payload()` 的 `coordination_graph_specs` 仍从 coordination task 编译。
- 前端协调管理面板可能少图。

判断：

不能直接删。应先确认 task graph 是唯一权威源，再迁移 coordination task。

### R5. workflow 默认 merge 不能直接删

涉及位置：

- `backend/tasks/workflow_registry.py:282`
- `backend/api/orchestration.py:310`
- `backend/tasks/assembly_support.py:61`
- `backend/tasks/assembly_support.py:79`
- `backend/tasks/assembly_support.py:90`

直接删除后果：

- 默认 workflow 可能缺失。
- flow 到 workflow 的解析失败。
- orchestration agents 页面 scope 选项减少。

判断：

不能直接删。后续需要区分默认 workflow 和显式 workflow，而不是取消默认 workflow。

## 四、可以优先瘦身的点

### P1. 四类运行策略 upsert 写路径

优先级：最高

改法：

只改写路径基底：

```text
effective list -> explicit list
```

影响面：

- 不影响运行态 getter。
- 不影响管理台详情数组。
- 能阻止派生默认项写入 JSON。

验证方式：

1. 空目录 upsert 一个 projection binding。
2. 检查 `task_projection_bindings.json` 只包含 1 条。
3. effective 列表仍返回默认派生 + 显式覆盖。

### P2. 四类运行策略 delete 写路径

优先级：高

改法：

删除时只过滤 explicit 列表。

影响面：

- 删除不会把派生对象固化。
- effective 读取仍能按默认规则生成派生对象。

验证方式：

1. 空目录 delete task domain 或 specific task。
2. 检查四类策略 JSON 不被写入派生项。
3. 确认删除返回结果仍包含影响面。

### P3. 四类运行策略 list 取消自动写回

优先级：高

改法：

去掉：

- `list_projection_bindings()` 中的自动 `_write_json()`
- `list_flow_contract_bindings()` 中的自动 `_write_json()`
- `list_task_agent_adoption_plans()` 中的自动 `_write_json()`
- `list_task_memory_request_profiles()` 中的自动 `_write_json()`

影响面：

运行 effective 返回值不变，只是不再把标准化结果写盘。

验证方式：

1. 已存在空 JSON 文件时调用 list。
2. 返回 effective 默认项。
3. 文件保持空 JSON，不被填充。

## 五、需要暂缓的瘦身点

### D1. 删除 assignments fallback

位置：

- `backend/tasks/flow_registry.py:1167`

暂缓原因：

`list_specific_task_records()` 在没有 records 时会从 legacy assignments 兜底生成 records。这个确实是旧残留，但它影响 specific task 的基础身份。

删除前条件：

- 确认所有测试和真实 storage 都已迁移到 `specific_task_records.json`。
- 新增迁移检查报告。

### D2. 删除 `task_agent_adoption_plan`

位置：

- `backend/tasks/flow_models.py:166`
- `backend/tasks/assembly_builder.py:306`
- `backend/orchestration/runtime_loop/task_run_loop.py:538`
- `backend/orchestration/runtime_loop/task_run_loop.py:4894`

暂缓原因：

这是命名旧残留，但运行环仍读取 legacy 字段。直接删会影响 worker spawn 和 coordination chain。

删除前条件：

- 装配只输出 `task_execution_policy`。
- RunLoop 内部只读新字段。
- 保留一次性 payload 迁移适配器。

### D3. 删除 coordination task 到 task graph 的兼容派生

位置：

- `backend/tasks/flow_registry.py:2099`

暂缓原因：

task graph 和 coordination task 目前权威源未完全统一。直接删可能让协调任务在前端消失。

删除前条件：

- 明确 task graph 为执行图唯一权威源。
- coordination task 只保留业务语义和策略引用。

## 七、最小安全修复方案

如果继续修，建议只做以下最小集：

1. upsert 四类运行策略只写 explicit 列表。
2. delete 四类运行策略只过滤 explicit 列表。
3. 四类运行策略 list 不再自动写盘。
4. 增加 3 个测试：
   - upsert 单条不会固化派生项。
   - delete 不会固化派生项。
   - 空 JSON 调用 list 不会被填充。

这组修复能精准切掉“越修越胖”的来源，同时不破坏运行态默认兜底。

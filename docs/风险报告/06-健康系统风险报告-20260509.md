# 健康系统风险报告

日期：2026-05-09

范围：
- `backend/health_system`
- `backend/api/health_system.py`
- `backend/api/health_workbench.py`
- `backend/tests/health_*`
- 健康系统依赖的 `backend/tasks`、`backend/orchestration`

## 一、结论摘要

健康系统不是单一模块，而是由四条链路组成：

1. 健康管理台和问题登记。
2. 健康 Agent 执行规划和运行准入。
3. 健康验证 / 回归门禁。
4. 维护测试系统和实验 runner。

当前最大风险不是健康 Agent 运行逻辑已经失败，而是健康 Agent 仍处于“未完成配置与授权”的半接入状态。健康系统已经定义了任务流、预览入口和执行入口，但 Agent Runtime profile 还没有把健康任务需要的输出合同显式授权，因此准入层会正确阻断执行。

本次审查发现一个已经被测试暴露的接入状态问题：

- 健康 Agent `agent:3` 的 runtime profile 尚未声明健康任务所需的 `output_contracts`。
- 健康任务流要求输出 `HealthTriageResult`。
- 任务绑定校验因此报 `output_contract_not_allowed`，这是授权缺失下的预期阻断。
- 健康 Agent 预览状态从预期 `ready` 变成 `blocked`。

专项测试结果：

```powershell
python -m pytest backend/tests/health_management_control_plane_regression.py backend/tests/health_runtime_admission_cutover_regression.py backend/tests/health_verification_gate_regression.py backend/tests/health_workbench_regression.py -q
```

结果：

```text
2 failed, 10 passed
```

失败用例：

```text
backend/tests/health_management_control_plane_regression.py::test_task_system_exposes_generic_agent_task_connection_profile
backend/tests/health_management_control_plane_regression.py::test_health_agent_run_preview_does_not_expose_projection_instance
```

失败原因：

```text
blocked_reasons = ["output_contract_not_allowed", "binding_invalid"]
```

这说明健康系统当前的“只读健康 Agent”安全边界仍在，但健康 Agent 尚未完成正式配置和授权，导致健康 Agent 主链路保持阻断状态。这个状态本身不是绕过式 bug，而是接入流程没有收尾。

## 四、主要风险

### R1. 健康 Agent 尚未配置输出契约授权，主链路处于准入阻断状态

严重级别：高

位置：
- `backend/orchestration/agent_runtime_registry.py:116`
- `backend/orchestration/agent_runtime_registry.py:141`
- `backend/tasks/flow_registry.py:83`
- `backend/tasks/flow_registry.py:2607`
- `backend/health_system/execution_planner.py:111`
- `backend/health_system/registry.py:315`

现象：

健康任务流定义了：

```text
flow.health.issue_triage -> output_contract_id = HealthTriageResult
```

但健康 Agent runtime profile 中尚未配置健康输出授权：

```text
output_contracts = ()
```

`TaskFlowRegistry.build_binding_for_flow()` 会校验 flow 的 output contract 是否包含在 runtime profile 的 `output_contracts` 中。由于健康 Agent 尚未声明该白名单，绑定变成 invalid。

测试暴露：

```text
health_profile["validation_state"] == "invalid"
blocked_reasons = ["output_contract_not_allowed"]
preview["status"] == "blocked"
```

影响：

- 健康 Agent 在完成配置授权前无法进入 ready 状态。
- `analyze_trace` / `draft_case` / `verify_fix` 的真实执行链可能被阻断。
- 健康系统入口已经存在，但核心 Agent 运行能力尚未正式开通。

建议：

先补齐健康 Agent 的配置与授权边界，再决定是否调整测试期望：

- 保持“空白名单不自动放行”的安全语义。
- 为健康 Agent profile 显式声明允许输出的健康合同：
  - `HealthTriageResult`
  - `HealthTraceAnalysisResult`
  - `HealthCaseDraft`
  - `HealthFixVerificationResult`
- 同步确认这些合同对应的任务流、执行入口和持久化结果字段已经存在。

不建议把空白名单改成 allow all。健康 Agent 还没有配置和授权时，准入层阻断是合理的；真正需要修的是接入配置缺失，以及测试是否应该在未授权阶段期待 `ready`。

### R2. 健康系统手工推进 RuntimeLoop 状态，容易和主 RunLoop 语义漂移

严重级别：高

位置：
- `backend/health_system/registry.py:468`
- `backend/health_system/registry.py:503`
- `backend/health_system/registry.py:557`
- `backend/health_system/registry.py:699`
- `backend/health_system/registry.py:711`
- `backend/health_system/registry.py:747`
- `backend/health_system/registry.py:776`

现象：

`execute_agent_run()` 没有直接复用主 `TaskRunLoop` 的完整执行流程，而是手工：

- append `task_contract_built`
- append `memory_runtime_view_built`
- append `context_snapshot_built`
- 构造 `RuntimeDirective`
- 调用 model executor
- 构造 final commit decision
- 写 checkpoint
- 更新 `RuntimeStateIndex`

风险：

- 主 RunLoop 的 commit gate、memory finalization、artifact validation、tool adoption 语义更新后，健康系统可能不同步。
- 健康 Agent 的 trace 看似像 RuntimeLoop，但并非完整 RuntimeLoop。
- 后续排查运行问题时，健康系统 trace 和主 trace 可能有字段缺失或状态含义不一致。

建议：

短期：

- 在健康 Agent 结果中显式标记 `execution_path = health_manual_runtime_bridge`。
- 补测试覆盖 executor error、blocked_by_gate、commit gate denied 三类终态。

中期：

- 健康 Agent 应尽量走统一 TaskRunLoop 入口，健康系统只负责准备 issue context 和执行策略。

### R3. JSONL 存储读坏行会静默跳过，upsert 通过整文件重写

严重级别：高

位置：
- `backend/health_system/store.py:139`
- `backend/health_system/store.py:147`
- `backend/health_system/store.py:154`
- `backend/health_system/store.py:167`

现象：

`HealthStore._read_jsonl_dicts()` 遇到坏 JSON 行：

```text
except json.JSONDecodeError:
    continue
```

`upsert_*` 操作通常会：

1. 读取整个 JSONL。
2. 过滤旧记录。
3. 整文件 `write_text()` 重写。

风险：

- 单行损坏会被静默丢弃。
- 下一次 upsert 可能把坏行永久删除。
- 并发 append / upsert 时，整文件重写可能覆盖另一个写入。
- 没有 `.bak`、原子替换、锁、诊断计数。

影响对象：

- issues
- commands
- receipts
- reports
- agent_runs
- verification_runs
- conversation sessions/messages

建议：

- `_read_jsonl_dicts()` 返回读取诊断，至少记录 `bad_line_count`。
- upsert 使用临时文件 + 原子替换。
- 对 append-only 数据，如 receipts、reports、messages，尽量不要 upsert 重写。
- Workbench overview 展示存储健康状态。

### R4. 命令异常被统一转换为 failed receipt，但失败边界太粗

严重级别：中高

位置：
- `backend/health_system/command_service.py:16`
- `backend/health_system/command_service.py:33`
- `backend/health_system/command_service.py:105`
- `backend/health_system/command_service.py:140`

现象：

`submit_command()` 会先 `upsert_command(command)`，然后处理命令。任何异常都会生成 failed receipt。

风险：

- 异常类型只进入 `blocked_reasons=(exc.__class__.__name__,)`。
- 用户只能看到 failed，但不知道是输入错误、运行依赖缺失、存储失败、测试系统失败还是真实健康 Agent 失败。
- command 已经写入，但后续报告/issue/receipt 可能只有部分成功。

建议：

- 区分 `input_rejected`、`admission_blocked`、`runtime_dependency_missing`、`execution_failed`、`storage_failed`。
- `_complete_command()` 应支持事务化或至少记录 partial commit 状态。
- 对失败 receipt 附带 `failure_phase`。

### R5. 测试 runner 状态依赖本进程内 Popen 和 PID 探测

严重级别：中高

位置：
- `backend/health_system/maintenance/experiments/runner.py:64`
- `backend/health_system/maintenance/experiments/runner.py:75`
- `backend/health_system/maintenance/experiments/runner.py:175`
- `backend/health_system/maintenance/experiments/runner.py:192`
- `backend/health_system/maintenance/experiments/runner.py:199`
- `backend/health_system/maintenance/experiments/runner.py:259`

现象：

ExperimentRunner 把 Popen 存在内存 `_processes` 中。服务重启后 `_processes` 丢失，只能靠 `run_state.json` 和 PID 探测恢复。

风险：

- Windows PID 可能被复用，状态判断可能误报。
- run_result 未生成时，状态会根据 PID 是否存在和 log tail 推断。
- cancel 只能可靠取消当前进程内启动的 process；服务重启后的 run 可能无法终止。
- 同一时间只允许一个 running run，但 running 判定依赖历史 state 刷新。

建议：

- run_state 记录启动命令、pid、process start time 或 lock token。
- cancel 对非本进程 run 明确返回 `cannot_cancel_detached_process`。
- stale running 检测应进入 verification diagnostics。

### R6. 验证服务会在 list 时记录 verification run

严重级别：中

位置：
- `backend/health_system/verification_service.py:20`
- `backend/health_system/verification_service.py:27`
- `backend/health_system/verification_service.py:33`
- `backend/health_system/verification_service.py:70`

现象：

`list_verification_runs()` 如果 store 里没有记录，会从 test system runs 构造并持久化 verification run：

```text
self.record_verification_run(item)
```

风险：

- 查询接口有写入副作用。
- Workbench 打开页面可能生成 verification record。
- 如果 test system run 仍在运行或 artifacts 不完整，可能持久化 stale/unknown 状态。

建议：

- `list_verification_runs()` 默认只读。
- 新增显式 `sync_verification_runs_from_test_system()`。
- Workbench 展示 live projection，但不自动落库。

### R7. Workbench token usage 扫 checkpoint 时吞异常

严重级别：中

位置：
- `backend/health_system/workbench.py:195`
- `backend/health_system/workbench.py:221`
- `backend/health_system/workbench.py:224`

现象：

工作台会扫描 runtime checkpoints，JSON 解析异常直接跳过。

风险：

- checkpoint 损坏不会进入健康问题。
- token 使用统计偏低。
- 健康系统本应发现运行状态损坏，却忽略了这类证据。

建议：

- 统计 `checkpoint_parse_error_count`。
- 对损坏 checkpoint 生成 evidence gap 或 health issue candidate。

### R8. 默认样例 issue 混入真实列表，可能污染工作台判断

严重级别：中

位置：
- `backend/health_system/registry.py:42`
- `backend/health_system/registry.py:117`
- `backend/health_system/workbench.py:33`

现象：

`list_issues()` 会把默认 sample issue 合并进真实 issues。

风险：

- 空系统也显示健康问题。
- Workbench inbox 和 recommended actions 会基于 sample issue 触发。
- 用户难以区分真实问题和示例问题。

建议：

- overview 区分 `sample_issues` 和 `stored_issues`。
- 默认不把 sample issue 放入正式 inbox。
- 测试可显式请求 sample seed。

## 五、测试覆盖现状

当前健康测试 4 个文件：

| 测试文件 | 主要覆盖 |
|---|---|
| `health_management_control_plane_regression.py` | 命令、准入、健康 Agent preview、测试启动 |
| `health_runtime_admission_cutover_regression.py` | 健康命令准入使用正式编排对象 |
| `health_verification_gate_regression.py` | 验证资源和 gate projection |
| `health_workbench_regression.py` | 工作台 projection 合同 |

缺口：

- JSONL 坏行和并发写入未覆盖。
- `execute_agent_run()` 真实成功/失败终态未覆盖。
- executor exception 后健康 run、task run、checkpoint 是否一致未覆盖。
- test runner 服务重启后的 running/cancel 状态未覆盖。
- verification list 查询写入副作用未覆盖。
- sample issue 与真实 issue 的隔离未覆盖。

## 六、建议修复顺序

### 阶段 1：完成健康 Agent 配置与授权

目标：

让健康 Agent 从“半接入阻断状态”进入“显式授权后可预览 / 可执行状态”，并让测试表达真实接入状态。

建议先做接入收口：

1. 保持 AgentRuntimeProfile `output_contracts=()` 为未授权 / 不放行语义。
2. 为 `agent:3` 显式补健康输出合同。
3. 核对 `analyze_trace` / `draft_case` / `verify_fix` 需要的合同是否全部覆盖。
4. 如果某些健康 Agent 能力暂时未开通，测试应断言 `blocked` 和具体 `blocked_reasons`，而不是期待 `ready`。

这里不应该通过放宽全局校验来消除失败。健康 Agent 未配置时被挡住是对的，后续只应补齐明确授权。

### 阶段 2：存储可靠性

目标：

坏 JSONL 不再静默消失。

动作：

1. `_read_jsonl_dicts()` 记录坏行诊断。
2. upsert 使用原子写。
3. Workbench summary 增加 store health。

### 阶段 3：运行链统一

目标：

健康 Agent 不再手工复制 RuntimeLoop。

动作：

1. 给当前手工桥接加 `execution_path` 标记。
2. 补终态一致性测试。
3. 逐步迁移到统一 TaskRunLoop 入口。

### 阶段 4：验证系统去副作用

目标：

Workbench 和 list 接口只读。

动作：

1. `list_verification_runs()` 不再自动 record。
2. 新增显式 sync。
3. runner 状态恢复增加 detached process 语义。

## 七、建议下一步审查对象

健康系统下一步建议先审：

1. `AgentRuntimeRegistry` 与任务 output contract 的白名单语义。
2. `HealthStore` 的 JSONL 存储可靠性。
3. `HealthRegistry.execute_agent_run()` 与 `TaskRunLoop` 主链路差异。

其中最优先的是第 1 项，因为它已经暴露出健康 Agent 接入流程没有完成。

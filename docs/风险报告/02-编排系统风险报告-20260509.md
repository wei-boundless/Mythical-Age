# 编排系统风险报告

日期：2026-05-09

范围：
- `backend/orchestration`
- `backend/orchestration/runtime_loop`
- `backend/tasks`
- `backend/query/runtime.py`

## 一、结论摘要

编排系统当前关键回归测试可以通过，核心门禁能力相对可靠，尤其是 `OperationGate` 和 `RuntimeCommitGateDecision` 已具备 fail-closed 基础。

但系统主要风险不在单个门禁，而在外围装配链和运行环：

1. `TaskRunLoop` 聚合过多职责，是最大结构性风险点。
2. Agent/Runtime/Task/Workflow/Contract 多个注册表读取失败时静默回退默认值。
3. 产物写入成功后存在 `executor_failed -> completed` 的兜底完成路径。
4. assistant message commit、memory commit、task result commit 的状态语义没有完全拆开。
5. 显式任务选择和 skill policy 对齐失败会吞异常降级。
6. 任务装配仍输出 legacy adoption plan，说明新旧协议仍并行。

专项测试结果：

```powershell
python -m pytest backend/tests/runtime_commit_gate_regression.py backend/tests/capability_system_preview_regression.py backend/tests/tool_authorization_regression.py backend/tests/runtime_assembly_builder_test.py backend/tests/orchestration_cutover_regression.py -q
```

结果：

```text
39 passed
```

测试通过说明当前主路径可用，但不能覆盖所有结构风险。尤其是配置破损、显式任务对齐失败、产物兜底完成、提交部分成功等场景仍需补强。

## 二、系统结构现状

### 2.1 代码规模

| 模块 | 文件数 | 行数 |
|---|---:|---:|
| `backend/orchestration` | 59 | 16959 |
| `backend/orchestration/runtime_loop` | 32 | 12583 |
| `backend/tasks` | 29 | 8214 |
| `backend/query` | 3 | 437 |

其中 `runtime_loop` 占编排系统核心体量的一半以上，`TaskRunLoop` 单文件约 5724 行。

### 2.2 当前主链路

当前编排主链路大致为：

```text
QueryRuntime
  -> AgentRuntimeChainAssembler
      -> memory runtime view
      -> query understanding
      -> task assembly
      -> orchestration runtime bundle
  -> TaskRunLoop
      -> runtime directive
      -> OperationGate
      -> model/tool execution
      -> artifact validation
      -> RuntimeCommitGate
      -> assistant message commit
      -> task result commit
      -> working memory finalization
```

这条链路已经具备较完整的结构，但多个环节仍存在 fallback 和 legacy 兼容路径。

## 三、主要风险

### R1. `TaskRunLoop` 职责过载

严重级别：高

位置：
- `backend/orchestration/runtime_loop/task_run_loop.py`

现象：

`TaskRunLoop` 同时承担：

- 运行 trace 和 checkpoint
- runtime directive 生成
- OperationGate 检查
- 模型执行
- 工具调用和工具结果采纳
- 任务账本推进
- artifact 校验与修复
- commit gate
- assistant message commit
- task result commit
- working memory finalization
- worker agent spawn
- LangGraph coordination runtime 桥接

风险：

这些职责都在同一个类和同一个文件中交汇，导致：

- 状态变量生命周期过长。
- 局部兜底逻辑容易绕过全局契约。
- 任一链路改动都可能影响最终 commit。
- 回归测试容易覆盖 happy path，而遗漏边界状态。

影响：

- 运行问题难定位。
- 新功能接入成本高。
- 后续拆分 Agent、任务、记忆、工具权限时容易互相牵连。

建议：

优先拆出以下内部服务：

1. `RuntimeDirectiveService`
2. `RuntimeToolDispatchService`
3. `TaskLedgerRuntimeService`
4. `ArtifactValidationService`
5. `RuntimeCommitService`
6. `WorkingMemoryRuntimeFinalizationService`
7. `CoordinationDeliveryService`

拆分目标不是立刻减少行数，而是让每条状态链有独立输入、输出和测试。

### R2. 注册表读取失败会静默回退默认值

严重级别：高

位置：
- `backend/orchestration/agent_registry.py`
- `backend/orchestration/agent_runtime_registry.py`
- `backend/orchestration/agent_group_registry.py`
- `backend/tasks/flow_registry.py`
- `backend/tasks/workflow_registry.py`
- `backend/tasks/contract_registry.py`

现象：

多个 `_read_json()` 在文件不存在、JSON 解析失败或异常时直接返回 fallback/default。

风险：

配置损坏会被伪装成系统正常：

- 自定义 Agent 丢失。
- Runtime profile 丢失。
- Task flow 丢失。
- Workflow/Contract 配置丢失。
- UI 和运行时继续显示默认配置。

影响：

- 用户以为系统正常，实际使用的是默认内置配置。
- 任务装配可能跑错 Agent 或错用默认工作流。
- 健康系统无法发现配置损坏。

建议：

拆分读取结果状态：

- `source=missing_default`
- `source=loaded_user_config`
- `source=invalid_fallback`
- `error=...`

对于 JSON 损坏，应至少写诊断事件或返回健康告警，不能与文件不存在混为一谈。

### R3. 产物兜底完成路径可能掩盖执行失败

严重级别：高

位置：
- `backend/orchestration/runtime_loop/task_run_loop.py`

现象：

当满足以下条件时：

1. artifact validation 通过。
2. `terminal_reason == "executor_failed"`。
3. 模板要求写入 artifact。

运行环会把 `terminal_reason` 改为 `completed`，并生成 `artifact_success_fallback` 最终回答。

这条路径有积极意义：如果真实产物已写入，但模型后续收口失败，不应让任务整体失败。

风险：

如果校验不够严格，会把“写了某个文件”包装成“任务完成”。

影响：

- 模型收口失败被弱化。
- artifact 与任务契约不一致时可能误报完成。
- 用户看到完成，但实际产物可能不是目标产物。

建议：

兜底完成必须同时满足：

1. 模板明确声明 `artifact_file_required`。
2. artifact 路径匹配任务契约目标路径。
3. write_file 事件来自当前 task run。
4. 文件仍存在。
5. 文件内容通过基本非空/类型校验。
6. fallback final answer 标明模型收口失败，但产物校验通过。

同时增加测试：

- 写入了错误路径时不得 completed。
- 写入空文件时不得 completed。
- 非产物型任务不得走 artifact fallback。

### R4. Commit 语义拆分不足

严重级别：中高

位置：
- `backend/orchestration/commit_gate.py`
- `backend/orchestration/runtime_loop/task_run_loop.py`
- `backend/query/runtime.py`

现象：

`RuntimeCommitGateDecision` 能判断 assistant message 是否允许写入。它会阻止空消息、missing answer、do_not_persist。

但最终输出链路包含多个不同 commit：

- assistant session message commit
- session memory refresh
- durable memory commit
- task result final record
- working memory finalization

当前状态字段仍容易把这些链路混在一起。例如 assistant message committer 返回后，运行环直接将 `assistant_commit_applied=True`。记忆提交状态又依赖返回 payload 推断。

风险：

部分成功和完全成功难以区分。

影响：

- UI/trace 显示 committed，但 memory 或 task result 可能失败。
- 健康系统难以定位 commit 失败源。
- 后续回放和恢复逻辑缺少准确状态。

建议：

建立统一 `RuntimeCommitOutcome`：

```text
assistant_message: attempted/succeeded/failed/skipped
session_memory: attempted/succeeded/failed/skipped
durable_memory: attempted/succeeded/failed/skipped
task_result: attempted/succeeded/failed/skipped
working_memory_finalization: attempted/succeeded/failed/skipped
```

每一项必须有：

- `attempted`
- `succeeded`
- `error`
- `refs`
- `diagnostics`

### R5. 显式任务选择和 skill policy 失败会吞异常降级

严重级别：中

位置：
- `backend/orchestration/agent_runtime_chain.py`

现象：

显式任务选择对齐时，如果 `TaskFlowRegistry(...).get_specific_task_record()` 抛异常，会吞掉异常并继续使用原始 query understanding。

Skill policy 解析失败时也会吞异常并返回 `None`。

风险：

显式用户选择的任务可能没有生效，但系统不会明确报告。

影响：

- 用户点选任务后，运行时可能走普通推理。
- Skill 未启用时没有可见诊断。
- 后续排查只看到“理解结果没命中”，看不到真实 registry/skill 失败。

建议：

降级可以保留，但必须写结构化诊断：

- `explicit_task_alignment_failed=true`
- `selected_task_id=...`
- `error_type=...`
- `fallback_to_query_understanding=true`

Skill policy 失败也应进入 runtime diagnostics，而不是静默变成无 skill。

### R6. 任务装配仍输出 legacy adoption plan

严重级别：中

位置：
- `backend/tasks/assembly_builder.py`

现象：

任务装配结果同时输出：

- `task_execution_policy`
- `task_agent_adoption_plan`

其中 `task_agent_adoption_plan` 通过 `execution_policy.to_legacy_dict()` 生成。

风险：

新旧协议长期并行会导致调用方继续依赖 legacy 字段。

影响：

- 删除旧协议困难。
- 前端和后端可能读不同字段。
- 运行诊断中难判断哪一个是权威字段。

建议：

建立协议权威说明：

1. `task_execution_policy` 是权威字段。
2. `task_agent_adoption_plan` 是兼容字段。
3. 新代码禁止读取兼容字段。
4. 风险报告或迁移计划中登记删除条件。

### R7. Runtime assembly 的上下文可见性依赖 profile 字段，缺少失败告警

严重级别：中

位置：
- `backend/orchestration/runtime_loop/runtime_assembly_builder.py`

现象：

Runtime assembly 会根据 `agent_profile.allowed_context_sections` 过滤上下文 section。如果 profile 没有配置 allowed sections，则默认全部可见。

风险：

profile 配置缺失时默认全量可见，可能和最小上下文原则相冲突。

影响：

- worker agent 可能获得比预期更多上下文。
- 工作记忆 conflict warning 等 section 的可见性依赖配置是否正确。

建议：

对 worker/sub-agent profile 使用显式 allowlist；如果 profile 缺失或没有配置，应至少写诊断：

- `context_visibility_policy=default_all_visible`
- `profile_missing_allowed_context_sections=true`

对于高风险 Agent，建议 fail-closed 或使用最小默认集。

## 四、已验证的正向能力

### 4.1 OperationGate 基础 fail-closed

`OperationGate` 对以下情况有明确拒绝：

- unknown operation
- missing directive_ref
- missing resource policy
- policy 未 adopted/runtime_executable
- denied operation
- approval required but headless
- denial tracking tripped

相关测试已覆盖部分场景。

### 4.2 RuntimeCommitGate 基础可用

`RuntimeCommitGateDecision` 能阻止：

- 空 assistant message
- missing answer
- do_not_persist

同时允许 stable answer 和 progress-only debug answer 进入对应写入路径。

### 4.3 工具采纳默认 deny

`tool_adoption.py` 中，如果缺少 operation descriptor、缺少 adopted resource policy、policy 不可执行、operation 不在 allowlist，都会 deny。该点设计方向正确。

## 五、建议整改顺序

### 第一阶段：配置读取诊断化

目标：配置损坏不能静默回退。

任务：

1. 抽统一 JSON registry reader。
2. 区分 missing、invalid、loaded。
3. 所有 registry 返回 diagnostics。
4. 健康系统接入 registry config health。

### 第二阶段：Commit Outcome 结构化

目标：提交状态不再混淆。

任务：

1. 新增 `RuntimeCommitOutcome`。
2. 拆分 assistant/session memory/durable/task result/working memory 状态。
3. 修正运行环输出和 trace_reader。
4. 增加部分成功、失败、跳过测试。

### 第三阶段：Artifact fallback 收紧

目标：只允许真实产物型任务兜底完成。

任务：

1. 验证 artifact 路径必须匹配契约目标。
2. 验证文件内容和类型。
3. 记录 fallback 是“产物完成但模型收口失败”。
4. 非产物任务禁止该路径。

### 第四阶段：AgentRuntimeChain 降级显式化

目标：显式任务选择和 skill 失败可追踪。

任务：

1. 捕获异常后写 diagnostics。
2. 将 diagnostics 带入 runtime assembly。
3. 前端或健康系统展示“任务对齐降级”。

### 第五阶段：拆分 TaskRunLoop

目标：降低主运行环耦合。

任务：

1. 先抽 artifact validation。
2. 再抽 commit service。
3. 再抽 tool dispatch。
4. 最后抽 coordination delivery。

每一步必须保留现有回归测试，并新增边界测试。

## 六、后续审查建议

编排系统之后，建议审查 `tasks/flow_registry.py` 和任务系统整体。原因：

1. 编排系统大量依赖任务注册和任务装配。
2. FlowRegistry 读取和写入逻辑复杂。
3. 默认任务、specific task、coordination task、topology、protocol、memory request profile 都在同一注册器中。
4. 它是配置静默回退和读写副作用最集中的位置。

# 洪荒时代模块化子单元系统重设计方案

日期：2026-04-27  
项目名：洪荒时代（mythical ageent）  
定位：在保留现有模块化子单元的前提下，清空历史链路连线，重新设计系统控制平面、模块接口和执行拓扑。

---

## 0. 本次重构的真实目标

这次不再是“继续修 QueryRuntime / QueryPlanner / orchestration adapter”。

这次是：

```text
保留子单元
断开旧连线
重新定义模块接口
重新设计中心控制平面
重新接线
```

也就是说，现有 `tools / skills / agents / workers / memory / retrieval / pdf / structured data / sessions / tasks / permissions / frontend operation` 等模块，优先视为可保留的能力单元。

但它们之间现在的直接调用、隐式状态恢复、旧 planner 决策、runtime if/else 分支、output 旁路写回，都不再默认继承。

一句话原则：

> 模块保留，连线清空；能力保留，控制权重建。

---

## 1. 新架构设计理念

### 1.1 子单元自治，但不能私自连线

每个子单元可以保留自己的内部实现，例如：

- PDF worker 可以继续负责 PDF 解析。
- Retrieval worker 可以继续负责知识检索。
- Structured data worker 可以继续负责表格/数据分析。
- MemoryFacade 可以继续负责 session/durable memory。
- ToolRuntime 可以继续负责工具实例。
- SkillRegistry 可以继续负责 skill 加载。

但这些子单元不能再互相直接决定执行路径。

错误示例：

```text
followup_resolver 直接决定走 PDF
continuation_resolver 直接覆盖 route
runtime 根据 query_understanding.tool_name 直接调工具
memory restore 直接改变当前轮 active target
output policy 自行决定 durable memory 写回
```

正确示例：

```text
子单元只提交 candidate / artifact / policy hint
控制平面统一仲裁
执行平面只按 directive 执行
写回统一进入 commit gate
```

### 1.2 重新接线之前，先定义端口

每个模块以后只通过标准端口接入系统：

```text
CandidatePort      提交候选
PolicyPort         提交资源/记忆/上下文策略
ExecutionPort      执行受控动作
ArtifactPort       产出证据、句柄、结果
CommitPort         请求状态写回
TracePort          输出可观测事件
```

任何模块如果绕过这些端口，直接调用其他模块内部函数，都视为旧连线。

### 1.3 控制平面唯一

新系统只有一个当前轮控制真相：

```text
ControlKernel
  -> TaskContract
  -> OrchestrationPlan
  -> ExecutionGraph
  -> RuntimeDirective
```

旧 `QueryPlan / QueryExecutionPlan / QueryUnderstanding / FollowupResolution / RestoreAuthorityContext` 都只能是候选材料，不再是控制真相。

### 1.4 写回必须延迟到 Commit Gate

所有写回动作都必须先形成候选：

- session message append
- session memory projection
- durable memory extraction
- task result ref
- artifact graph merge
- title generation
- correction writeback

然后统一由 `CommitKernel / OutputCommitGate` 按策略应用。

## 3. 新系统分层

### 3.1 第一层：Unit Layer

保留所有子单元内部能力。

```text
tools
skills
agents
workers
memory
retrieval
pdf
structured_data
sessions
tasks
permissions
```

这一层只暴露端口，不互相直接接线。

### 3.2 第二层：Contract Layer

把所有子单元包装成标准契约。

核心契约：

```text
ToolContract
SkillContract
AgentProfile
WorkerContract
MemoryContract
TaskContract
ArtifactContract
CommitContract
```

建议目录：

```text
backend/contracts/
  unit.py
  task.py
  capability.py
  execution.py
  memory.py
  artifact.py
  commit.py
```

如果短期不想新建顶层包，也可以先放在：

```text
backend/orchestration/contracts.py
```

### 3.3 第三层：Control Kernel

这是新架构核心。

建议目录：

```text
backend/orchestration/kernel.py
backend/orchestration/coordinator.py
backend/orchestration/policies.py
backend/orchestration/execution_graph.py
backend/orchestration/validation.py
```

职责：

- 收集 candidates。
- 建立 `TaskContract`。
- 生成 `OrchestrationPlan`。
- 生成 `ExecutionGraph`。
- 校验 `ResourcePolicy / MemoryPolicy / ContextPolicy / AnswerPolicy`。
- 输出 `RuntimeDirective`。

它不做：

- 不直接调用工具。
- 不直接写 session。
- 不直接构建最终 prompt。
- 不直接保存 memory。

### 3.4 第四层：Execution Runtime

执行层只消费 directive。

建议结构：

```text
backend/runtime_execution/
  executor.py
  model_executor.py
  tool_executor.py
  worker_executor.py
  agent_executor.py
```

短期可以先不建新包，在 `backend/query/runtime.py` 里抽出：

```python
_stream_directive_execution()
_execute_model_node()
_execute_tool_node()
_execute_worker_node()
_execute_agent_node()
```

但目标是让 `QueryRuntime` 逐步变薄。

### 3.5 第五层：Commit Kernel

统一处理所有写回。

建议结构：

```text
backend/orchestration/output_commit.py
backend/orchestration/memory_commit.py
backend/orchestration/task_commit.py
backend/orchestration/artifact_commit.py
```

职责：

- 根据 execution result 形成 commit candidates。
- 根据 `AnswerPolicy / MemoryPolicy` 决定是否写回。
- 应用 session append、memory refresh、durable extraction、task refs、artifact graph merge。

## 5. 旧连线清空清单

### 5.1 必须断开的直接连线

| 旧连线 | 问题 | 新连线 |
| --- | --- | --- |
| `QueryPlanner -> QueryExecutionPlan -> Runtime if/else` | planner 拥有执行权 | `QueryPlannerCandidate -> ControlKernel -> RuntimeDirective` |
| `continuation_resolver -> route/tool overwrite` | restore 越权 | `RestoreCandidate -> Orchestration arbitration` |
| `followup_resolver -> binding_execution` | 弱线索越权 | `FollowupCandidate -> TaskContract refs` |
| `runtime -> query_understanding.tool_name -> direct_tool` | runtime 自己调度 | `RuntimeDirective.tool -> ToolExecutor` |
| `runtime -> worker_plan -> EvidenceOrchestrator` | worker 计划来自旧 planner | `ExecutionNode(worker) -> WorkerExecutor` |
| `output_policy -> fallback/writeback` | 输出层旁路写回 | `OutputCommitPlan -> CommitKernel` |
| `TaskCoordinator -> query regex binding` | task 自己猜上下文 | `TaskContract.bindings -> TaskRecord` |

### 5.2 可以暂时保留但要套壳的连线

| 现有连线 | 过渡处理 |
| --- | --- |
| `CapabilityDispatchScheduler` | 改为 `CapabilityCandidateProducer` |
| `RuntimeToolBridge` | 改为 `ToolExecutor` 内部实现 |
| `EvidenceOrchestrator` | 改为 `WorkerExecutor` 内部实现 |
| `MemoryFacade.inspect_query_context` | 改为 `MemoryCandidateProducer` |
| `prompt_builder` | 改为 `ContextPolicy` 消费者 |

## 6. 新接口定义草案

### 6.1 UnitDescriptor

所有子单元都要能被登记成一个 unit。

```python
@dataclass(slots=True)
class UnitDescriptor:
    unit_id: str
    unit_type: str  # tool / skill / worker / agent / memory / task / output
    name: str
    version: str
    capabilities: list[str]
    input_contract: dict[str, Any]
    output_contract: dict[str, Any]
    risk_tags: list[str]
    state_policy: dict[str, Any]
```

### 6.2 CandidateEnvelope

所有候选统一包装。

```python
@dataclass(slots=True)
class CandidateEnvelope:
    candidate_id: str
    candidate_type: str
    producer: str
    confidence: float
    payload: dict[str, Any]
    evidence_refs: list[str]
    risks: list[str]
    authority: str = "candidate_only"
```

### 6.3 TaskContract

当前轮任务根对象。

```python
@dataclass(slots=True)
class TaskContract:
    task_id: str
    session_id: str
    user_goal: str
    task_kind: str
    source_needs: list[str]
    modality: str
    constraints: dict[str, Any]
    accepted_candidates: list[str]
    status: str = "pending"
```

### 6.4 ExecutionNode

执行图节点。

```python
@dataclass(slots=True)
class ExecutionNode:
    node_id: str
    task_id: str
    executor_type: str  # model / tool / worker / agent
    executor_id: str
    action: str
    inputs: dict[str, Any]
    depends_on: list[str]
    resource_policy_ref: str
    output_contract_ref: str
    shared_channels: list[str]
    fallback: str = "fail_closed"
```

### 6.5 CommitCandidate

所有写回候选统一包装。

```python
@dataclass(slots=True)
class CommitCandidate:
    candidate_id: str
    commit_type: str  # session_message / task_ref / memory / artifact / title
    source_node_id: str
    payload: dict[str, Any]
    policy_ref: str
    allowed: bool = False
```

## 8. 新重构阶段

### Phase 0：冻结旧线

目标：

- 不再继续增强旧 planner/runtime 连线。
- 不再继续给旧链路加 diagnostics。

动作：

- 把当前模块视为 unit inventory。
- 把旧跨模块调用列入断线清单。

完成标准：

- 新文档成为后续重构依据。
- 后续代码修改都必须说明是在“断旧线”还是“接新线”。

### Phase 1：建立 Unit Contract 层

目标：

- 给每个保留子单元建立 descriptor。

文件：

- `backend/orchestration/contracts.py`
- 或新建 `backend/contracts/*`

完成标准：

- tools/skills/workers/memory/tasks 都能导出 `UnitDescriptor`。

### Phase 2：建立 Candidate Collector

目标：

- 旧 planner、restore、memory、capability、follow-up 全部改为候选生产者。

文件：

- `backend/orchestration/candidates.py`
- `backend/query/planner.py`
- `backend/query/runtime_followup.py`
- `backend/orchestration/restore_context.py`

完成标准：

- 所有候选统一为 `CandidateEnvelope`。
- 候选没有执行权。

### Phase 3：建立 Control Kernel

目标：

- 生成新的 `TaskContract / OrchestrationPlan / ExecutionGraph`。

文件：

- `backend/orchestration/kernel.py`
- `backend/orchestration/coordinator.py`
- `backend/orchestration/execution_graph.py`
- `backend/orchestration/policies.py`

完成标准：

- 不通过 `QueryPlan` 投影也能生成计划。
- validator 能 fail-closed。

### Phase 4：建立 Directive Runtime

目标：

- runtime 不再按 `QueryExecutionPlan` 执行。

文件：

- `backend/query/runtime.py`
- `backend/query/runtime_tools.py`
- `backend/query/evidence_orchestrator.py`

完成标准：

- 新增 `_stream_directive_execution()`。
- direct_tool/worker/model 都从 directive node 进入。

### Phase 5：建立 Commit Kernel

目标：

- 写回统一收口。

文件：

- `backend/orchestration/output_commit.py`
- `backend/orchestration/memory_commit.py`
- `backend/orchestration/task_commit.py`

完成标准：

- session append、session memory、durable memory、task refs、artifact merge 都来自 commit plan。

### Phase 6：删除旧线

目标：

- 删除或隔离旧 planner/runtime 的执行分支。

完成标准：

- `QueryExecutionPlan` 不再作为执行输入。
- `query_understanding.tool_name` 不再触发工具调用。
- `worker_plan` 不再直接触发 worker。
- `followup_resolution` 不再直接生成执行。

## 10. 最终目标

重构完成后，洪荒时代应该变成：

```text
一个由模块化能力单元组成、
由唯一控制内核重新接线、
由执行图驱动运行、
由提交门统一写回、
并能解释 plan 与 actual 差异的 Agent Runtime。
```

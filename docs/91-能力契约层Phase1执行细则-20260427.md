## 目标

本阶段只做一件事：把 `skills / tools / agents` 的事实源从前端和 `api/operations.py` 的临时拼装逻辑中抽出来，形成正式 `backend/capabilities` 能力契约层。

## 本阶段完成后应具备的能力

- 后端有统一 `CapabilityManifest` 构建模块。
- 操作系统 API 从 `CapabilityManifest` 读取 skills、tools、agent 归属和绑定关系。
- 前端操作系统仍能正常使用现有 skills、tools、绑定关系页。
- `agent系统` 前端合并进操作系统的实施边界被锁定，但可分后续小步迁移。
- 测试覆盖 PDF tool 不归主 agent、skill -> tool 绑定写回、tool 风险和边界识别。

## Phase 3A 执行细则：正式编排中枢契约骨架

### 本地设计原则约束

参考来源：

- `docs/设计原则/03-状态管理.md`
- `docs/设计原则/06-上下文管理.md`
- `docs/设计原则/09-工具系统设计.md`
- `docs/设计原则/12-Agent-系统.md`
- `docs/设计原则/23-Memory系统.md`
- `docs/设计原则/25-架构模式总结.md`

提炼为本次编排层重构的硬约束：

- **单一事实源**：一次用户请求最终只能有一份 canonical `OrchestrationPlan`。理解层、记忆系统、上下文管理和操作系统都只能向它提交候选、约束或资源描述，不能各自私下决定最终执行。
- **默认隔离，显式共享**：子 agent、worker、工具调用和测试回放默认不能污染主会话状态。只有被 `ExecutionDirective` 显式声明的共享通道，才能写入任务状态、记忆状态或最终答案。
- **恢复不等于决策**：记忆系统、follow-up、文件状态、证据句柄都可以恢复候选对象，但是否采用必须由编排层仲裁。恢复层不得直接覆盖当前轮 intent、resource policy 或 execution target。
- **工具是带契约的微服务**：tool 不只是函数调用，必须带输入 schema、权限、风险、可见性、结果边界和执行归属。编排层只能选择已注册、已归属、已过权限约束的 tool。
- **权限与安全 fail-closed**：未知工具、未知 agent、未知 skill 绑定、未知来源权限默认不可自动执行。高风险工具必须继续通过权限系统和工具契约，不允许模型计划绕过。
- **上下文管理只做预算、装配和恢复索引**：上下文系统负责决定什么进入模型窗口、如何压缩、如何恢复文件/记忆，但不决定业务执行路径。它输出 `ContextPolicy`，由编排层消费。
- **记忆按需召回，不把可派生状态写成记忆**：长期记忆只保存用户偏好、项目约定、稳定反馈和参考信息；临时任务状态、可从当前文件/任务系统派生的信息不进入长期记忆。编排层必须区分 state memory 与 durable memory。
- **状态转换先于通知和展示**：运行时状态、任务状态、执行单元状态必须先完成原子更新，再生成 UI 通知、日志摘要或测试报告，避免“看起来完成但状态未落盘”。
- **React 前端和后端运行时共享状态源但不互相耦合**：前端可以展示和编辑策略草案，后端 runtime 才是执行真相。前端不能绕过后端 validator 直接打开工具、agent 或权限。

落实到契约映射：

| 上游系统 | 允许输出 | 不允许做 |
| --- | --- | --- |
| 理解层 | `IntentFrame` 候选、任务类型、来源需求、风险信号 | 直接调用工具或指定最终 agent |
| 记忆系统 | `MemoryPolicy`、状态摘要、长期记忆召回候选 | 用历史状态覆盖当前轮目标 |
| 上下文管理 | `ContextPolicy`、prompt 片段、预算与恢复索引 | 决定执行路径或工具权限 |
| 操作系统 | `ResourcePolicy` 所需的 skill/tool/agent registry、绑定、启停、权限 | 让前端编辑绕过 validator |
| 编排层 | `ExecutionDirective`、`AnswerPolicy`、canonical `OrchestrationPlan` | 绕过权限系统直接执行 |

实施边界：

- 新增正式契约文件，命名使用 `orchestration_*`，不再使用 `llm_planner_*`。
- 契约只描述“本轮应该怎样被理解、授权、调度和收口”，不直接调用工具。
- `QueryRuntime` 第一阶段只把正式 directive 写入 `orchestration_plan`，不改变旧执行链。
- 优先扩展现有 `backend/orchestration`，不要把正式编排模型重新放回 `backend/query`，否则会继续让 `QueryRuntime` 承担过多跨系统职责。

建议新增：

- `backend/orchestration/directives.py`（如现有 `models.py` 继续膨胀再拆出）
- `backend/orchestration/validation.py`
- `backend/tests/orchestration/test_orchestration_directive.py`

建议修改：

- `backend/orchestration/models.py`
- `backend/orchestration/adapters.py`
- `backend/orchestration/runtime_adapter.py`
- `backend/orchestration/diff.py`
- `backend/query/runtime.py`

核心结构：

- `IntentFrame`：用户目标、任务类型、来源需求、是否需要工具、是否需要执行单元。
- `MemoryPolicy`：本轮允许读取的对话记忆、状态记忆、长期记忆，以及允许写回的记忆范围。
- `ContextPolicy`：本轮 prompt 片段、证据预算、文件/数据句柄、压缩策略和恢复索引。
- `ResourcePolicy`：本轮允许的知识库、本地、联网、工具类型、skill、agent、worker 范围。
- `ExecutionDirective`：动作、目标 skill、目标 tool、目标 agent、输入摘要、风险标签、共享/隔离通道。
- `AnswerPolicy`：是否需要引用、是否隐藏内部协议、是否允许 fallback。

执行顺序：

1. 从旧 `QueryPlan` 和 `RuntimeContextState` 投影 `IntentFrame / MemoryPolicy / ContextPolicy` 候选。
2. 从 `CapabilityManifest`、search policy、agent binding 投影 `ResourcePolicy` 候选。
3. 在 `backend/orchestration/adapters.py` 中汇总为 canonical `OrchestrationPlan`，暂不改变旧执行链。
4. 在 `backend/orchestration/validation.py` 中做 schema 与权限一致性校验，只输出 `ValidationDecision`。
5. 在 `backend/orchestration/runtime_adapter.py` 中把通过校验的计划转成 `RuntimeControl` 预览；`plan_only` 只记录，不执行。
6. 在测试系统与 dry-run 中展示 directive、validator decision、实际执行差异。

阶段禁止事项：

- 不新增模型调用。
- 不让编排 directive 直接调用工具。
- 不把前端编辑结果直接写成 runtime 真相。
- 不把记忆恢复结果当成当前轮最终目标。
- 不绕过 `RuntimeToolBridge` 的工具可见性和 search policy 过滤。

完成标准：

- dry-run 和测试系统能看到正式 directive。
- directive 通过 schema 校验。
- 不新增模型调用，不改变用户可见行为。
- 原有编排计划、搜索权限、工具可见范围测试继续通过。
- 新增测试能证明 `plan_only` 下 directive、validator decision、actual execution 三者均可追踪，且 runtime 行为不变。

## Phase 7A 执行细则：旧启发式权力审计与 Readiness Gate

### 设计原则压缩成执行约束

本阶段把 01-25 压缩为以下执行约束：

- **主循环约束**：一次用户请求只能有一条可解释的运行主循环。正式编排层可以生成计划、预览和接管诊断，但不能与旧 `QueryPlanner` 同时各自执行一套不可对齐的主链。
- **状态约束**：状态必须分层。前端 Store、运行时 state、session memory、durable memory、tool context 不得互相偷写；跨层写入必须通过明确 API、policy 或 directive。
- **恢复约束**：continuation、memory、context、file handle restore 都是候选恢复，不是当前轮决策。恢复出的目标、文件、工具、agent 必须经过编排层和 validator。
- **工具约束**：工具必须来自注册表和能力契约。工具可见性、来源权限、风险等级、输入 schema、owner agent、输出边界都必须可检查。
- **Agent 约束**：子 agent/worker 默认隔离。只有任务注册、资源清理等基础设施允许穿透；业务状态写回必须显式 opt-in。
- **权限约束**：deny 优先，bypass 不得覆盖显式安全约束。高风险系统执行、外部联网、写入型操作必须继续 fail-closed。
- **上下文约束**：上下文系统只负责预算、压缩、装配、索引和证据候选，不决定 route、tool、agent、final answer。
- **Prompt/cache 约束**：能力说明和工具 schema 要稳定排序并控制在缓存友好的 section 中；每轮 trace、readiness、diff 不进入静态 prompt。
- **记忆约束**：长期记忆只保存稳定偏好、项目约定、反馈、参考信息。临时任务状态、当前文件状态、测试运行中间结果不得写入长期记忆。
- **输出约束**：输出层负责 canonical answer、协议清洗、fallback 检查和持久化策略，不负责重新理解用户意图或重新选择执行路径。

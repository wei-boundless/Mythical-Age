## 1. 结论

当前长场景里的“软污染”不是单一字段脏了，而是两类状态边界同时失守：

1. `active_goal` 会被低置信度调和逻辑回写成上一轮目标。
2. `key_results` 会把跨任务历史结果持续保留，并通过 compaction/debug 重新暴露给主线程。

这两个问题耦合在一起后，会出现下面这种“表面答对、内部已漂移”的状态：

1. 用户当前问题已经变了，最终回答也可能是对的。
2. 但 session/process 里仍挂着旧 `active_goal`。
3. `debug` 中还能看到 `low_flow_confidence`、`Active goal changed without explicit switch signal`。
4. 历史 `employees / inventory` 结果还留在 `Key Results`。
5. 后续一旦进入 `full_compact` 或新的 follow-up，旧状态会再次反向影响当前轮。

因此，这不是“输出层再收一收”的问题，而是一次典型的：

- restore 越权成 decide
- session-wide 历史结果越权成 current-turn truth

本次建议的正式方向是：

> 不再让 `previous_state` 直接覆盖当前轮目标。  
> 不再让 `DialogueState.key_results` 继续承担“整个 session 的历史结果桶”。  
> 主线程只保留当前轮真相，恢复候选与历史结果改为显式旁路层。

## 2. 技术源报告

这一节不是理念推测，而是对当前代码与 `turn-17` 实测产物的对照结论。

### 2.1 实测证据：`turn-17-main.json`

实测文件：

- `output/test_runs/20260424-live-extended-rerun-01/artifacts/sixty-turn-real-user-marathon/turn-17-main.json`

当前用户问题：

- `把实时查询结果改写成值班提示。`

表面结果：

- 最终用户答案是对的
- `done.content` 正常

但内部状态同时暴露了 4 个异常信号：

1. `context_management.sections.active_process_context` 仍显示 `# Active Goal -> 查询黄金价格。`
2. `debug_sections` 中出现：
   - `Active goal changed without explicit switch signal`
   - `low_flow_confidence`
3. `debug_session_trace -> # Key Results` 仍保留大量：
   - `inventory.xlsx`
   - `employees.xlsx`
   - 早先 PDF 风险总结
4. `memory_context.session_memory.active_goal` 仍是：
   - `再查一下北京今天天气。`

这说明问题不是回答错，而是：

> 回答质量掩盖了 process-state 漂移。

### 2.2 根因一：低置信度调和逻辑把旧 `active_goal` 覆盖回来了

文件：

- `backend/structured_memory/understanding_reconciliation.py`
- `backend/structured_memory/process_engine.py`

真实链路：

1. `UnderstandingReconciler._is_low_confidence_flow_switch(...)`
   - 只要满足“旧 flow 存在 + 当前 goal_request + 置信度 < 0.55 + flow_hint 不同”
   - 就会产出：
     - `preserve_previous_flow=True`
     - `preserve_previous_goal=True`

2. `ProcessStateEngine._resolve_active_goal_fields(...)`
   - 如果 `decision.preserve_previous_goal` 为真
   - 就直接把 `active_goal` 改回 `previous_state.active_goal`

这意味着当前设计里：

- reconciliation 本应只负责“保守降级”
- 但实际上它拥有“覆盖当前轮目标”的能力

这直接违反了：

1. `docs/25` 的单一真相源原则
2. `docs/28` 的主线程只保留当前真相原则
3. `docs/32` 的 restore 不裁决原则

### 2.3 根因二：`key_results` 被做成了无 owner 的会话大桶

文件：

- `backend/structured_memory/process_engine.py`

真实链路：

1. `assemble(...)` 中始终执行：
   - `result_items = self._extract_results(projected_assistant_messages)`
   - `key_results=self._dedupe_items(result_items, max_items=max_items)`

2. `_extract_results(...)`
   - 只看 assistant message 里有没有命中 `RESULT_MARKERS`
   - 不区分：
     - 当前 flow
     - 当前 task
     - 当前 binding owner
     - 是否已经脱离当前问题

结果就是：

1. 早先 PDF 风险摘要会留下来
2. 早先 `inventory.xlsx` 聚合结果会留下来
3. 早先 `employees.xlsx` 结果会留下来
4. 后续实时查询轮并不会清掉这些历史结果

这不是“抽取不够精细”，而是：

> `key_results` 这个字段本身承担了错误职责。

### 2.4 根因三：compaction/debug 把历史结果重新提升成 model-adjacent truth

文件：

- `backend/structured_memory/session_memory_view.py`
- `backend/context_management/context_controller.py`

真实链路：

1. `SessionMemoryViewBuilder.render_state(...)`
   - 不论 debug 还是 compaction，都保留 `# Key Results`

2. `COMPACTION_HEADER_ORDER`
   - 显式把 `# Key Results` 放进 compaction 序列

3. `ContextController._debug_trace_blocks(...)`
   - 又把 `# Key Results` 作为 debug trace 的标准块保留下来

4. 在 `full_compact` 下：
   - `warm_snapshots` 被清掉
   - `durable` 也可能为空
   - 但 `active_process_context + hot_truth_window + debug trace` 还在

于是旧 `key_results` 会以另一种形式重新抬头，继续影响：

1. debug preview
2. compact restore view
3. 后续对“当前 session 真相”的判断

### 2.5 根因四：当前系统缺少“当前真相 / 恢复候选 / 历史结果 / debug轨迹”的显式分层

目前 `DialogueState` 及其视图仍把几类语义混在一起：

1. 当前轮真相
2. 恢复提示
3. 历史结果
4. 调试注记

于是系统只能靠：

- 低置信度降级
- section 选择
- 视图裁切

这些后置策略去兜底。

这正是本问题反复出现的根本原因。

## 3. 本问题必须遵守的设计原则

这次收口不能只修个别函数，必须把下列原则写成硬约束。

### 3.1 结构修复优先于 prompt 修复

来自：

- `docs/25`
- `docs/26`
- `docs/35`

落到本问题上就是：

1. 不能靠 prompt 让模型“自己更稳一点”
2. 不能靠追加“如果你切任务请谨慎”这种提示词
3. 必须修 `state ownership`

### 3.2 restore 不等于 decide

来自：

- `docs/28`
- `docs/29`
- `docs/32`

落到本问题上就是：

1. `previous_state.active_goal` 只能作为 restore candidate
2. 不能再直接覆盖当前轮 `active_goal`
3. `active_dataset / active_pdf / warm_context` 同理只能恢复，不得裁决

### 3.3 task-local truth 不得泄露成 main-thread truth

来自：

- `docs/28`
- `docs/33`
- `docs/34`
- `docs/64`

落到本问题上就是：

1. `employees / inventory / pdf / realtime` 结果必须有 owner
2. 没有 owner 的历史结果不能进入主线程当前真相
3. “某轮产出过什么”不等于“当前仍应被主线程看见”

### 3.4 summary/ref 优先于 raw historical bucket

来自：

- `docs/28`
- `docs/35`
- `docs/37`

落到本问题上就是：

1. 主线程应消费当前相关的 summary/ref
2. 不再消费整个 session 级 `key_results` 大桶
3. 历史结果要么挂在 task/result ref，要么降到 debug/history 侧

### 3.5 默认隔离，显式共享

来自：

- `docs/25`
- `docs/34`
- `docs/64`

落到本问题上就是：

1. 当前轮改写“实时查询结果”时，应显式续接最近相关 task summary ref
2. 不能因为 session 里曾出现过 `inventory/employees` 就默认共享它们的结果

### 3.6 不用新增关键词表来掩盖状态污染

来自：

- `docs/52`
- `docs/61`
- `docs/64`

落到本问题上就是：

1. 不能再加一堆 “改写 / 值班提示 / 实时结果” 的特殊分支
2. 不能靠 `if "改写" in query` 去修 turn-17
3. 必须把“改写上一轮结果”落实成协议和状态续接

## 4. 推荐设计方向

### 4.1 把主线程状态拆成 4 层

建议把当前混合状态正式拆成：

1. `current_turn_truth`
   - 当前轮真正激活的 goal / flow / binding / next action

2. `restore_candidates`
   - 来自 previous_state / warm context / committed slots 的恢复候选

3. `historical_result_refs`
   - 历史 task 结果的索引/摘要引用

4. `debug_governance`
   - 风险标记、漂移诊断、压缩说明

主线程 model-visible 只能消费：

1. `current_turn_truth`
2. 少量当前关联的 `historical_result_refs`

不能再直接消费：

1. `restore_candidates`
2. `debug_governance`
3. session-wide `key_results`

### 4.2 `active_goal` 的所有权重新定义

新的所有权规则应是：

1. 当前轮 `active_goal` 由当前轮理解结果决定
2. reconciliation 只能：
   - 降低置信度
   - 加风险标记
   - 生成 clarification signal
3. reconciliation 不能再：
   - 直接把 `previous_state.active_goal` 写回当前轮

如果当前轮需要“暂不切 flow”，正确做法应是：

1. 当前轮 goal 仍保留当前用户请求
2. 旧 flow 只作为 `restore_flow_hint`
3. 风险层记录：
   - `flow_switch_unconfirmed`
   - `needs_clarification`

而不是：

1. 把当前 goal 改回旧 goal

### 4.3 `key_results` 改为 owner-first 结构

当前 `DialogueState.key_results` 应拆分为至少两类：

1. `current_result_refs`
   - 与当前 flow / current task / active binding 直接相关

2. `historical_result_refs`
   - 历史 task 结果，仅供显式恢复或 debug

若继续保留 `key_results` 这个兼容字段，也只能作为：

- 当前可见结果投影

不能再作为：

- 整个会话的历史结果桶

### 4.4 compaction/view 只显示当前相关结果

新规则应是：

1. `model view`
   - 只显示当前 goal + 当前相关 summary refs

2. `compaction view`
   - 只保留 restore-oriented 当前相关结果

3. `debug view`
   - 可以保留更丰富历史，但必须显式标成 debug-only

`# Key Results` 不能再无条件进入：

1. compaction 主视图
2. model-visible process context

### 4.5 “改写上一轮结果”走 handle/ref 续接，不走弱启发式

turn-17 这类问题的正式解决方向不是补关键词，而是：

1. 当前轮若是对已有结果做改写/转写/摘要/重排
2. 应显式命中最近相关 `task_summary_ref` 或 `result_ref`
3. 续接的是结果句柄，而不是 session 整体历史

这样才能让：

1. 黄金价格
2. 北京天气
3. 库存/员工

在同一长场景里保持边界，不互相污染。

## 5. 固定执行流

本问题后续实现时，执行顺序必须锁死为下面这条链。

### Step 1：当前轮理解

输入：

1. 当前用户消息
2. 必要的 follow-up handle
3. 少量 committed restore 候选

输出：

1. 当前轮候选 `active_goal`
2. 当前轮候选 `flow`
3. 当前轮候选 `binding`

### Step 2：follow-up / transform 句柄解析

输入：

1. 当前用户消息
2. task registry / task summary refs

输出：

1. 是否命中结果改写/结果续接
2. 若命中，返回具体 `task_id / task_summary_ref / result_ref`

要求：

1. 不靠字符串主路径重拆整段历史
2. 不把历史 `key_results` 当句柄

### Step 3：reconciliation 只做降级，不做回写覆盖

允许：

1. 标记 `needs_clarification`
2. 标记 `low_confidence`
3. 标记 `flow_switch_unconfirmed`

禁止：

1. 覆盖当前 `active_goal`
2. 覆盖当前 `flow`

### Step 4：process commit 只写当前轮真相

写入：

1. 当前 goal
2. 当前 flow
3. 当前 binding
4. 当前 result refs

不写入：

1. 整个 session 的历史结果合集

### Step 5：context package 只投影当前相关 truth

model-visible：

1. 当前 goal
2. 当前 hot truth
3. 当前相关 result refs

debug-visible：

1. 风险说明
2. 历史诊断
3. 压缩说明
4. 历史结果总览

## 6. 分阶段执行计划

### Phase 0：先补护栏，不先改行为

目标：

1. 先把“软污染”固化成 regression
2. 把 turn-17 的内部漂移变成可自动失败的测试

涉及文件：

- `backend/tests/session_memory_regression.py`
- `backend/tests/query_runtime_route_guard_regression.py`
- 新增 `backend/tests/process_state_soft_contamination_regression.py`
- `backend/tests/system_eval/long_runner.py`

验收标准：

1. 可以显式断言：
   - 当前轮 `active_goal` 不得被旧 goal 覆盖
   - `Key Results` 不得跨无关 flow 泄露到 model-visible/compaction
2. `turn-17` 类场景不再因为“最终答案对”而被误判为通过

### Phase 1：收掉 reconciliation 的越权覆盖

目标：

1. 把 `preserve_previous_goal` 从“覆盖当前 goal”改成“生成 restore hint”
2. 让 reconciliation 回到它该有的职责

涉及文件：

- `backend/structured_memory/understanding_reconciliation.py`
- `backend/structured_memory/process_engine.py`
- `backend/structured_memory/dialogue_state.py`

具体动作：

1. 弱化或废弃 `preserve_previous_goal`
2. 新增：
   - `restore_goal_hint`
   - `restore_flow_hint`
   - 或等价的 restore candidate 字段
3. `_resolve_active_goal_fields(...)` 不再直接回写 `previous_state.active_goal`

验收标准：

1. 低置信度时可以降级，但当前轮 goal 仍然是当前轮问题
2. 风险说明存在，但 main truth 不漂移

### Phase 2：重构 `key_results` 为 owner-first 结构

目标：

1. 去掉无 owner 的 session-wide 结果桶
2. 让结果跟 task / flow / binding 绑定

涉及文件：

- `backend/structured_memory/process_engine.py`
- `backend/structured_memory/dialogue_state.py`
- `backend/tasks/models.py`
- `backend/query/context_models.py`

具体动作：

1. 新增 `current_result_refs`
2. 新增 `historical_result_refs`
3. `ProcessStateEngine._extract_results(...)` 不再直接写满 `DialogueState.key_results`
4. 当前轮只投影与当前 flow / current task 相关的结果摘要

验收标准：

1. `employees / inventory` 不会在实时查询改写轮继续挂在 model-visible results 中
2. 历史结果仍可通过 task/result ref 找回

### Phase 3：视图与 compaction 去混用

目标：

1. model view、compaction view、debug view 职责彻底分离
2. `Key Results` 不再默认进入 compaction 主链

涉及文件：

- `backend/structured_memory/session_memory_view.py`
- `backend/structured_memory/session_memory.py`
- `backend/context_management/context_controller.py`
- `backend/context_management/context_models.py`

具体动作：

1. `COMPACTION_HEADER_ORDER` 改成 current-truth-first
2. `# Key Results` 仅在：
   - 当前相关结果存在时进入 model/compact
   - 否则留在 debug
3. `_debug_trace_blocks(...)` 可继续保留历史结果，但明确是 debug-only
4. `full_compact` 不再把历史结果作为主 restore truth

验收标准：

1. compaction 里看不到跨无关任务的旧结果
2. debug 里仍能追踪历史结果

### Phase 4：结果改写/转写走句柄续接

目标：

1. turn-17 这类“把实时结果改写成值班提示”不再依赖 session 漂浮状态
2. 改为直接消费最近相关 `task_summary_ref / result_ref`

涉及文件：

- `backend/query/followup_resolver.py`
- `backend/query/followup_models.py`
- `backend/query/runtime.py`
- `backend/query/answer_assembler.py`

具体动作：

1. 为“结果改写/结果转写”增加显式 continuation/transform 句柄
2. 优先续接最近相关任务结果，而不是整段历史
3. 不新增关键词大表，只增加结构化 target resolution

验收标准：

1. turn-17 不再触发旧 goal 保留
2. 当前轮改写明确续接实时查询结果，而非历史杂项结果

### Phase 5：清理兼容壳和旧路径

目标：

1. 把旧的混合字段和兼容逻辑清掉
2. 防止未来再次回流

涉及文件：

- `backend/structured_memory/process_engine.py`
- `backend/structured_memory/session_memory_view.py`
- `backend/context_management/context_controller.py`
- 相关测试文件

具体动作：

1. 清理无 owner 的 `key_results` 旧用法
2. 清理利用 previous goal 覆盖当前 goal 的兼容路径
3. 清理把 debug/history 重新投影成 model-visible truth 的旧逻辑

验收标准：

1. 新旧路径不会并存竞争
2. 之后再跑长场景时不会反复出现同类软污染

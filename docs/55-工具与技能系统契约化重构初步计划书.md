## 1. 本阶段收口结论

本轮工具与技能系统重构已经完成，当前可以正式判定：

1. 工具执行边界已从“弱元信息 + 启发式补全”升级为“契约声明 + runtime gate + 权限协同”。
2. 技能系统已从“隐性路由参与者”收缩为“能力暴露与工具白名单系统”。
3. 高污染工具已经改为 fail-closed，不再允许缺 owner 时继续隐式猜测执行。
4. 理解层、接续层、任务协调层里一批会放大污染的弱 authority 逻辑已经收缩。

因此，`docs/55` 对应的问题域已经收口，不应再继续把后续答案质量问题算到本阶段名下。

---

## 2. 原始问题是否已经解决

本阶段最初针对的是以下结构性问题：

1. 工具有元信息，但没有强执行契约。
2. `ToolInputResolver` 更像参数补全器，而不是 fail-closed 的执行门。
3. `SkillRegistry` 会通过 hint/example/tag 参与本地路由加权。
4. PDF、结构化数据等高污染工具缺少统一的 owner/binding 执行协议。
5. route 层、follow-up 层、task 协调层仍残留弱 authority 推断。

当前判断是：

1. 上述问题已经完成结构修复。
2. 后续剩余问题主要不再是“谁触发工具、何时允许执行”的问题。
3. 后续剩余问题主要转移到“工具结果如何收成稳定答案”的输出层。

## 3. 已完成改造项

## 3.1 工具契约化

已完成：

1. 为工具层补入正式契约定义与契约门。
2. runtime 在工具执行前统一消费契约结果，而不是只依赖工具内部兜底。
3. 高污染工具的缺参/缺 binding 行为改为显式拒绝或澄清，不再弱猜。

对应实现文件：

- `backend/tools/contracts.py`
- `backend/tools/definitions.py`
- `backend/tools/runtime.py`
- `backend/query/runtime.py`

## 3.2 高污染工具 fail-closed

已完成：

1. `pdf_analysis` 不再允许缺 owner 时继续内部猜路径。
2. `structured_data_analysis` 不再允许靠 catalog 默认对象直接补出执行目标。
3. 高污染工具现在必须依赖显式输入或有效 binding。

对应实现文件：

- `backend/tools/pdf_analysis_tool.py`
- `backend/tools/structured_data_analysis_tool.py`
- `backend/query/tool_input_resolver.py`
- `backend/query/binding_resolver.py`

## 3.3 技能系统降权为能力系统

已完成：

1. 技能注册信息继续保留，但不再承担本地路由放大职责。
2. `allowed_tools` 进入执行边界约束，不再只是说明性字段。
3. skill 的 hint/example/tag 不再作为强 route authority。

对应实现文件：

- `backend/tools/skills_scanner.py`
- `backend/skill_system/registry.py`
- `backend/query/runtime.py`

## 3.4 路由与接续弱 authority 收缩

已完成：

1. 理解层移除了结构化数据 catalog 的默认提升路径。
2. continuation / follow-up 的弱 authority 进一步收缩。
3. `TaskCoordinator` 中天气、金融等对象的弱 owner 推断被移除。

对应实现文件：

- `backend/understanding/task_understanding.py`
- `backend/understanding/query_understanding.py`
- `backend/query/planner.py`
- `backend/query/continuation_resolver.py`
- `backend/query/followup_resolver.py`
- `backend/tasks/coordinator.py`

## 5. 明确不属于本阶段的问题

经过本轮改造后，仍然残留的长场景问题主要包括：

1. `RAG evidence -> canonical answer` 收口不稳定。
2. `memory route -> visible answer` 过程化污染拦截不足。
3. `PDF canonical -> stable summary` 收口仍不稳定。
4. 部分多源总结与答案可见层仍有输出质量问题。

这些问题虽然会在长场景里表现为“回答不好”，但它们不再属于：

1. 工具是否被错误触发。
2. skill 是否越权抢路由。
3. 工具是否缺 owner 仍被偷偷执行。
4. 输入契约是否缺失。

换句话说，`docs/55` 解决的是“执行边界问题”，而不是“最终答案收口问题”。

---

## 6. 为什么现在必须分拆到下一阶段

如果继续把剩余问题混在 `docs/55` 里处理，会造成两个错误：

1. 把已经完成的结构边界改造重新打开，导致无谓回摆。
2. 把输出层问题误判成路由层问题，再次回到启发式补丁。

按照当前设计原则，下一阶段应单独处理：

1. `memory -> visible answer`
2. `RAG evidence -> canonical answer`
3. `PDF canonical -> stable summary`

也就是只处理输出层与答案收口，不重新开启工具/技能/路由大改。

## 8. 最终判断

当前更准确的系统判断应是：

1. 工具与技能系统的契约化重构已经做完。
2. 路由层、接续层、任务层中最危险的弱 authority 已经压下去。
3. 系统接下来要修的重点，不是“继续让工具更会猜”，而是“让已有证据更稳定地收成答案”。

因此，`docs/55` 到此正式收口。

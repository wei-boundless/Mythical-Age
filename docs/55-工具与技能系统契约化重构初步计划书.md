## 1. 背景与问题定义

当前系统已经完成了 RAG 主链、PDF 阅读链、follow-up 仲裁和长场景输出边界的一轮稳定化。最近的长场景问题表明，剩余风险不主要来自单个工具坏掉，而来自“工具和技能如何被选择、绑定、授权、执行”的边界仍不够硬。

具体表现是：

1. `task_understanding.py` 仍有较多启发式 route hint。
2. `continuation_resolver.py` 和 `followup_resolver.py` 已经被压成候选层，但仍会给下游施加错误 owner 压力。
3. `ToolDefinition` 有工具元信息，但没有强执行契约。
4. `ToolInputResolver` 主要做输入补全，尚未成为 fail-closed 的契约执行门。
5. `SkillRegistry.match_for_query()` 仍会根据 hint/example/tag 打分选择 skill，容易让技能系统继续承担隐性路由职责。
6. PDF、结构化数据等高污染工具虽然已有专门能力层，但“何时允许执行、必须带什么 owner、缺 owner 怎么失败”没有统一协议。

因此，本轮不是继续扩大启发式规则，而是要把工具与技能系统升级为：

`能力声明 + 输入契约 + owner 绑定 + 权限校验 + 运行时仲裁`

正确终态不是“系统更会猜”，而是：

1. 工具只在契约满足时执行。
2. 技能只暴露能力和上下文策略，不直接决定路由。
3. 弱自然语言信号只产生候选，不越权变成 tool call。
4. 缺少 owner/binding 时明确澄清或 fail closed。
5. 运行时成为工具执行的最后仲裁点。

## 2. 当前系统现状

## 2.1 已有基础

### 工具注册

当前工具定义位于：

- `backend/tools/definitions.py`
- `backend/tools/tool_registry.py`
- `backend/tools/runtime.py`

`ToolDefinition` 已具备以下字段：

1. `name`
2. `module`
3. `capability_tags`
4. `supported_modalities`
5. `safety_tags`
6. `route_hints`
7. `search_terms`
8. `typical_queries`
9. `safe_for_auto_route`
10. `is_read_only`
11. `is_destructive`
12. `is_concurrency_safe`

这说明当前工具层不是空白状态，已经有注册表、能力标签和基础安全元数据。

### 工具输入

PDF 和结构化数据工具已有 Pydantic schema：

- `backend/tools/pdf_analysis_tool.py`
- `backend/tools/structured_data_analysis_tool.py`

但当前 schema 中 `path` 仍是可选字段，并且工具内部会自行从 query 或 catalog 里推断文件：

1. `PdfAnalysisInput.path` 可为空。
2. `StructuredDataAnalysisInput.path` 可为空。
3. 工具内部会调用 `resolve_pdf_path(...)` 或 `resolve_dataset_path(...)`。

这会导致工具层承担了“猜 owner”的职责。

### 权限管线

当前权限入口位于：

- `backend/permissions/decision_pipeline.py`

已有检查包括：

1. `safe_for_auto_route`
2. `allowed_tools`
3. permission mode policy
4. tool instance 的 `validate_permission` / `validate_input`

这是一条有价值的管线，但目前多数工具没有实现足够强的本地校验，所以 `tool_validation` 不足以阻止缺 owner 的调用。

### 技能系统

当前技能系统位于：

- `backend/tools/skills_scanner.py`
- `backend/skill_system/registry.py`

`SkillDefinition` 已包含：

1. `allowed_tools`
2. `supported_modalities`
3. `supported_task_kinds`
4. `supported_source_kinds`
5. `capability_tags`
6. `preferred_route`
7. `forbidden_routes`
8. `routing_hints`
9. `examples`
10. `reference_paths`

这说明技能已经有“能力描述 + 工具白名单”的雏形。

### 任务绑定

任务上下文和 binding 位于：

- `backend/tasks/context_models.py`
- `backend/tasks/coordinator.py`
- `backend/query/binding_resolver.py`
- `backend/query/tool_input_resolver.py`

但 `TaskCoordinator._derive_task_bindings(...)` 仍通过正则和关键词从 query 中提取 PDF、dataset、天气、黄金等 source。该层现在仍混杂了“上下文记录”和“语义判断”。

## 2.2 当前核心缺口

当前缺的不是注册表，而是契约层。

具体缺口如下：

1. 工具定义没有声明 `required_inputs`。
2. 工具定义没有声明 `required_bindings`。
3. 工具定义没有声明缺 owner 时的行为。
4. 工具定义没有声明是否允许 catalog default。
5. 技能定义没有声明 `context_mode = inline/fork`。
6. 技能定义没有声明“只作为能力暴露，不能提升 route”。
7. 权限管线没有统一消费工具契约。
8. 输入 resolver 还在补参数，而不是判定契约是否满足。
9. route 层和 skill 层都还有弱 hint 转强执行的残留路径。

## 3. Claude Code 对照结论

本计划参考的是本地 `Claude-Code-Source-Study-main` 和 `claude-code-nb-main` 中关于 Tool、Skill、Agent、Permission 的架构分析，不做照搬，只提取适合当前系统的工程约束。

## 3.1 Tool 是完整微服务，不只是函数

Claude Code 的工具接口包含：

1. input schema
2. output schema
3. validate input
4. check permissions
5. read-only/destructive/concurrency 判定
6. enabled 判定
7. UI render 协议
8. result serialization

可借鉴点：

1. 工具必须自己声明执行条件。
2. 工具必须参与输入校验和权限校验。
3. 默认行为应保守。
4. 工具注册表是唯一能力来源。

不照搬点：

1. 当前项目不需要完整 UI render 协议。
2. 当前项目不需要一次性重写所有工具为 Claude Code 风格接口。
3. 当前项目应在现有 `ToolDefinition` 和 `BaseTool` 上增量加 contract。

## 3.2 Skill 是能力上下文，不是本地路由器

Claude Code 的 skill 主要是：

1. Markdown 能力说明。
2. allowed tools。
3. when_to_use。
4. inline 或 fork 上下文模式。
5. 可选路径条件和 hooks。

可借鉴点：

1. skill 可暴露给模型选择，但不应成为本地关键词路由器。
2. 高污染 skill 应 fork 执行，主线程只接收精炼结果。
3. skill 的工具范围必须被 allowed tools 限制。
4. skill 只声明“适用场景”，不直接绕过 runtime 仲裁。

不照搬点：

1. 当前系统已有 planner、RAG、PDF runtime，不应改成 Claude Code 的 slash command 体系。
2. 当前系统暂不引入完整 plugin/hook 生命周期。

## 3.3 Agent/子任务用于隔离污染，不用于替代主链判断

Claude Code 的 Agent 架构强调：

1. 子 Agent 有独立上下文。
2. 主 Agent 只接收精炼结论。
3. 子 Agent 可有独立工具集和权限。
4. 只读探索类 Agent 可省略不必要上下文，减少污染。

对当前系统的含义：

1. PDF、长文档、复杂表格分析可以使用隔离执行策略。
2. 是否进入隔离执行由 runtime 根据契约和计划决定。
3. 不能让子任务恢复逻辑倒逼当前 turn 决策。

## 3.4 Permission 是最后防线，必须 fail-closed

Claude Code 的权限系统有多层：

1. 全局 permission mode。
2. allow/deny/ask 规则。
3. 工具自身权限检查。
4. 只读/写入/破坏性/并发安全分类。

对当前系统的含义：

1. `safe_for_auto_route=True` 不能等于“可以缺 owner 执行”。
2. `is_read_only=True` 不能等于“可以自动猜 source”。
3. 缺少契约输入时应拒绝或澄清，而不是调用工具内部 fallback。

## 4. 本项目必须遵守的设计原则

从本地设计原则和近期问题追踪中，抽象出以下硬约束。

## 4.1 结构修复优先于 prompt 修复

如果问题是工具被错误触发、owner 丢失、binding 漂移，就不能靠提示词要求模型“别这样”。必须在契约层阻止错误调用。

## 4.2 ownership 修复优先于 phrase-list 扩张

不能继续通过增加“刚才、这个、那份、重点、总结、展开”等词表来修复接续问题。弱词只能产生候选，不能决定工具执行。

## 4.3 restore 不等于 decide

历史恢复层可以提供候选 binding，但不能替代当前 turn 的 planner/runtime 仲裁。

## 4.4 工具执行必须有当前轮可解释输入

任何工具执行都必须能解释：

1. 本轮为什么需要这个工具。
2. tool input 从哪里来。
3. owner/binding 从哪里来。
4. 缺失项是否被明确澄清或拒绝。

## 4.5 技能只约束能力，不接管主链

skill 可以：

1. 描述能力。
2. 限定工具。
3. 提供专门 prompt。
4. 声明上下文隔离模式。

skill 不可以：

1. 直接根据 routing hint 改 route。
2. 绕过 planner/runtime。
3. 用 examples/tag 直接触发工具。

## 5. 推荐目标架构

目标架构为：

`Task Understanding 候选层 -> Planner 主链 -> Binding/Contract Resolver -> Runtime Contract Gate -> Permission Gate -> Tool Execution -> Canonical Output`

## 5.1 ToolContract

在现有 `ToolDefinition` 基础上引入工具执行契约。

建议字段：

```python
@dataclass(frozen=True, slots=True)
class ToolExecutionContract:
    required_inputs: list[str]
    required_bindings: list[str]
    optional_inputs: list[str]
    owner_scope: str
    allow_catalog_default: bool
    allow_history_binding: bool
    missing_binding_behavior: str
    context_policy: str
    result_channel: str
```

字段语义：

| 字段 | 作用 |
|---|---|
| `required_inputs` | 工具调用前必须存在的输入字段 |
| `required_bindings` | 必须存在的 owner，如 `active_pdf`、`active_dataset` |
| `optional_inputs` | 可选输入，不影响契约通过 |
| `owner_scope` | `none/current_turn/active_binding/explicit_path` |
| `allow_catalog_default` | 是否允许目录默认推断 |
| `allow_history_binding` | 是否允许从历史 binding 继承 |
| `missing_binding_behavior` | `clarify/deny/fallback_to_rag` |
| `context_policy` | `inline/isolated/forked` |
| `result_channel` | `canonical/tool_raw/artifact_only` |

## 5.2 工具契约初始建议

| 工具 | required inputs | required bindings | catalog default | 缺失行为 | context policy |
|---|---|---|---|---|---|
| `get_weather` | `location` | 无 | 否 | clarify | inline |
| `get_gold_price` | 可为空 | 无 | 否 | deny if multi-source synthesis | inline |
| `web_search` | `query` | 无 | 否 | clarify | inline |
| `search_knowledge` | `query` | 无 | 否 | fallback_to_rag | inline |
| `pdf_analysis` | `query`, `path` | `active_pdf` 或 explicit path | 否 | clarify | isolated |
| `structured_data_analysis` | `query`, `path` | `active_dataset` 或 explicit path | 默认否 | clarify | isolated |
| `analyze_multimodal_file` | `path` | explicit path | 否 | clarify | isolated |
| `index_multimodal_file` | `path` | explicit path | 否 | deny/confirm | isolated |
| `read_file` | `path` | explicit path | 否 | clarify | inline |
| `terminal` | `command` | explicit command | 否 | deny/confirm | isolated |
| `python_repl` | `code` | explicit code | 否 | deny/confirm | isolated |

关键改变：

1. `pdf_analysis.path` 不再由工具内部从 query 猜。
2. `structured_data_analysis.path` 不再默认从 catalog 猜。
3. catalog default 只能在明确低风险工具或显式开启场景中使用。
4. 多源总结场景不得被单一 realtime 工具抢占。

## 5.3 SkillCapability

把当前 `SkillDefinition` 从“路由参与者”调整为“能力声明”。

建议字段：

```python
@dataclass(slots=True)
class SkillCapability:
    name: str
    description: str
    allowed_tools: list[str]
    supported_task_kinds: list[str]
    supported_source_kinds: list[str]
    activation_policy: str
    context_mode: str
    route_authority: str
```

字段语义：

| 字段 | 作用 |
|---|---|
| `activation_policy` | `model_visible/manual/conditional_path` |
| `context_mode` | `inline/fork/isolated` |
| `route_authority` | 固定为 `none` 或 `candidate_only` |

迁移要求：

1. `preferred_route` 只能作为展示信息，不再参与本地 route 加权。
2. `routing_hints` 只能进入 model-visible capability block，不再用于本地评分提升。
3. `examples` 不再参与本地 route 决策。
4. `allowed_tools` 必须被 permission gate 消费。
5. 高污染技能必须声明 `context_mode=isolated` 或 `fork`。

## 5.4 Runtime Contract Gate

在工具真正执行前新增统一 gate。

输入：

1. 当前 plan。
2. tool name。
3. tool input。
4. active binding。
5. selected skill。
6. permission mode。

输出：

1. allow。
2. clarify。
3. deny。
4. fallback。

伪流程：

```text
plan -> resolve candidate tool input
     -> load ToolExecutionContract
     -> validate required inputs
     -> validate required bindings
     -> validate skill allowed_tools
     -> validate permission policy
     -> execute or return clarify/deny
```

核心原则：

1. 输入 resolver 只能补显式输入和已确认 binding。
2. resolver 不能从弱 query 里猜 owner。
3. contract gate 是工具执行前的最后结构边界。
4. permission gate 不再只看安全标签，也要看 contract result。

## 6. 分阶段执行路线

## Phase 1：只加契约模型，不改变行为

目标：

1. 新增工具契约数据结构。
2. 给现有工具补默认 contract。
3. 不改变现有 runtime 行为。
4. 先做 registry snapshot 输出，方便审查。

涉及文件：

1. `backend/tools/definitions.py`
2. `backend/tools/tool_registry.py`
3. `backend/tools/runtime.py`
4. `backend/tools/skills_scanner.py`
5. `backend/skill_system/registry.py`

完成标准：

1. 每个工具都有 contract。
2. 现有测试不变。
3. registry payload 可以看到 required input/binding。

## Phase 2：新增 Contract Gate，先以 shadow mode 运行

目标：

1. 新增 `ToolContractGate`。
2. 在工具执行前计算契约判定。
3. shadow mode 下只记录，不拦截。
4. 输出 trace 中写入 contract decision。

建议新增文件：

1. `backend/tools/contracts.py`
2. `backend/tools/contract_gate.py`

涉及文件：

1. `backend/query/runtime.py`
2. `backend/permissions/decision_pipeline.py`
3. `backend/query/tool_input_resolver.py`

完成标准：

1. trace 能看到 `contract_decision=allow/clarify/deny`。
2. PDF 无 path、dataset 无 path 等场景能被 shadow 标记。
3. 不影响当前用户可见结果。

## Phase 3：PDF 与结构化数据率先 fail-closed

目标：

1. `pdf_analysis` 必须有 explicit path 或 active_pdf binding。
2. `structured_data_analysis` 必须有 explicit path 或 active_dataset binding。
3. 禁止工具内部自行 catalog default 猜 source。
4. 缺 owner 时返回澄清，不执行工具。

涉及文件：

1. `backend/tools/pdf_analysis_tool.py`
2. `backend/tools/structured_data_analysis_tool.py`
3. `backend/query/tool_input_resolver.py`
4. `backend/query/binding_resolver.py`
5. `backend/query/runtime.py`

完成标准：

1. “刚才那份 PDF”无 active_pdf 时不执行。
2. “看一下这个表”无 active_dataset 时不执行。
3. 显式文件路径仍能执行。
4. 已确认 follow-up binding 仍能执行。

## Phase 4：SkillRegistry 降权为能力系统

目标：

1. `SkillRegistry.match_for_query()` 不再根据 routing hints/examples/tags 直接选 skill。
2. skill selection 只接受明确 tool、明确 task kind、明确 source kind 或 runtime 指派。
3. skill block 主要作为 model-visible 能力说明。
4. `allowed_tools` 接入 contract/permission gate。

涉及文件：

1. `backend/skill_system/registry.py`
2. `backend/tools/skills_scanner.py`
3. `backend/query/runtime.py`
4. 相关 prompt/context 组装文件

完成标准：

1. 弱 hint 不再改变 route。
2. skill 可以限制工具权限。
3. PDF/structured 高污染 skill 可声明 isolated/fork。

## Phase 5：收缩 route 层启发式，迁移到候选和契约

目标：

1. `task_understanding.py` 只输出候选信号，不独立完成高风险 tool route。
2. realtime、pdf、structured 等工具调用由 planner/runtime + contract gate 合并判断。
3. 保留强显式信号，如 `.pdf`、`.csv`、明确页码、明确地点。
4. 删除“单关键词抢占多源总结”的路径。

涉及文件：

1. `backend/understanding/task_understanding.py`
2. `backend/query/continuation_resolver.py`
3. `backend/query/followup_resolver.py`
4. `backend/query/runtime.py`

完成标准：

1. 多源总结不再被 `天气` 抢成 `get_weather`。
2. 弱 follow-up 不再自动产生强 owner。
3. 显式 PDF/表格/天气查询不回退。

## Phase 6：长场景回归与旧路径清理

目标：

1. 清理重复的输入猜测逻辑。
2. 清理工具内部 catalog default 自动猜测。
3. 清理 skill routing hints 本地打分。
4. 对长场景做完整回归。

涉及文件：

1. `backend/query/tool_input_resolver.py`
2. `backend/query/binding_resolver.py`
3. `backend/tasks/coordinator.py`
4. `backend/tools/pdf_analysis_tool.py`
5. `backend/tools/structured_data_analysis_tool.py`
6. `backend/skill_system/registry.py`

完成标准：

1. 旧启发式路径有明确删除记录。
2. trace 中 contract decision 覆盖所有 tool call。
3. 长场景不出现工具错误抢占。

## 7.3 暂不修改

以下部分暂不重构：

1. RAG retrieval backend。
2. rerank 模型。
3. PDF parser 内部页级解析。
4. 输出 canonical answer 主链。
5. Agent 注册体系。

原因是本轮目标是工具/技能执行边界，不是内容理解能力或检索质量。

## 9. Cutover 与回退规则

## 9.1 Shadow Mode

Phase 2 必须先以 shadow mode 接入：

1. 只记录 contract decision。
2. 不拦截工具调用。

1. `pdf_analysis`
2. `structured_data_analysis`
3. `analyze_multimodal_file`
4. `index_multimodal_file`

低风险工具暂不 enforce，避免一次性扩大回归面。

## 9.3 Full Enforce

所有工具 contract 完成测试后，再开启全量 enforce。

## 9.4 回退策略

必须保留配置开关：

1. `TOOL_CONTRACT_MODE=off`
2. `TOOL_CONTRACT_MODE=shadow`
3. `TOOL_CONTRACT_MODE=enforce`

如果长场景出现显式工具调用大面积回退，立即切回 shadow mode，而不是回滚所有代码。

## 11. 初步结论

当前系统的工具和技能层已经有可用基础，不需要推倒重构。真正需要补的是“契约化执行边界”。

推荐路线是：

1. 保留现有 `ToolDefinition`、`ToolRuntime`、`SkillRegistry`。
2. 增加 `ToolExecutionContract`。
3. 新增 runtime contract gate。
4. 先 shadow，再对 PDF/表格 enforce。
5. 将 skill 从路由器降级为能力上下文。
6. 最后收缩理解层启发式。

这条路线符合当前 docs 原则，也符合 Claude Code 的核心工程思想：工具是带 schema、权限、校验和上下文策略的执行单元；skill 是能力说明和上下文策略，不是本地关键词路由器；真正的执行权必须留在主链 runtime 和权限/契约 gate 中。

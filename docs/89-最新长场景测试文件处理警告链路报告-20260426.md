## 3. 具体问题链路

### 3.1 PDF 页码 follow-up 降级

相关 turn：

| turn | 用户问题 | 结果 |
| --- | --- | --- |
| `turn-06-main` | 第三页具体讲了什么？ | `target_page_text_quality_low` |
| `turn-07-main` | 第四页如果让我准备汇报，应该重点盯哪几句？ | `target_page_has_no_stable_text` |
| `turn-43-doc` | 第三页讲了什么？ | `target_page_text_quality_low` |

链路：

1. 编排计划正确识别为 `route=worker`、`worker=pdf`，原因是 `handle_binding_followup`。
2. `active_pdf` 正确指向 `knowledge/AI Knowledge/2025年AI治理报告：回归现实主义.pdf`。
3. PDF runtime 进入 `backend/pdf_agent/runtime.py` 的 `_run_page_scope`。
4. 页面能定位，但 `_page_meets_stable_gate` 未通过，或者页面没有稳定正文。
5. `backend/query/pdf_worker.py` 把降级结果转为 fallback answer。
6. 长测只检查 `response.nonempty`，所以“无法稳定概括”也通过。

判断：路由是对的，问题在 PDF 解析质量门槛与测试验收不匹配。用户要的是“讲了什么”，系统实际回答的是“我不能可靠讲”。

### 3.2 PDF “第二部分”章节定位失败

相关 turn：

| turn | 用户问题 | 结果 |
| --- | --- | --- |
| `turn-18-main` | 回到刚才 PDF，第二部分的约束重点是什么？ | `target_section_not_located` |
| `turn-39-main` | 再回到 PDF，第二部分的约束能不能只用两句话说清楚？ | `target_section_not_located` |

链路：

1. 编排仍正确识别为 PDF follow-up。
2. `active_pdf` 仍存在，说明不是 PDF 绑定丢失。
3. PDF runtime 进入 `_run_section_scope`。
4. `_match_section_pages` 没有匹配到“第二部分 / 第2部分”。
5. 系统返回“没有稳定定位到第二部分”。
6. 因为有非空回答，测试通过。

判断：这是章节解析能力问题，不是会话记忆问题。后续要做章节别名、目录映射、标题 OCR/文本清洗增强，否则“第二部分/第二章/Part 2”这类自然引用会继续脆。

### 3.3 `read_file` 直接路由被安全策略拦截

相关 turn：

| turn | 用户问题 | 结果 |
| --- | --- | --- |
| `turn-34-main` | 读取 `docs/26-OpenClaw-架构改造计划.md`，概括主路径分层。 | `tool_permission_denied` / `tool_not_safe_for_auto_route` |

链路：

1. 编排识别到显式文件路径，计划为 `route=tool`、`tool=read_file`，原因是 `explicit_workspace_file_anchor`。
2. `backend/query/runtime_tools.py` 以 `direct_route=True` 请求权限。
3. `backend/permissions/decision_pipeline.py` 判断：直接路由时，如果工具 `safe_for_auto_route=False`，拒绝。
4. `backend/tools/definitions.py` 中 `read_file.safe_for_auto_route=False`。
5. 最终返回 `无法调用工具 read_file：tool_not_safe_for_auto_route`。

判断：安全策略本身是有意设计的，但对“只读显式路径”的用户场景太保守。尤其用户明确给了路径，且工具是只读，应该有一种受控的 explicit-path read 模式，而不是直接拒绝。

### 3.4 `read_file` 路径根目录错位

相关 turn：

| turn | 用户问题 | 结果 |
| --- | --- | --- |
| `turn-35-main` | 再试一次直接执行 Python 去改文件。 | agent 多次尝试读取，认为文件不存在 |

链路：

1. 前一轮 `read_file` 被直接路由拦截后，状态里留下“无法调用工具 read_file”的失败线索。
2. 下一轮进入 `route=agent`，原因是 `fallback_bounded_lookup`。
3. agent 尝试读取：
   - `docs/26-OpenClaw-架构改造计划.md`
   - `26-OpenClaw-架构改造计划.md`
   - `workspace/docs/26-OpenClaw-架构改造计划.md`
   - `./docs/26-OpenClaw-架构改造计划.md`
4. 工具返回 `file does not exist` 或 `path is a directory`。
5. 实际检查发现：文件存在于 repo 根目录 `docs/26-OpenClaw-架构改造计划.md`，但运行时 `ReadFileTool(root_dir=base_dir)` 的 `base_dir` 是 `backend/`。

判断：这里有真实 bug。工具说明写的是 “project root”，但实际根目录是 backend root，导致 repo 根目录下的文档不可见。这个问题会持续影响前端/测试里用户以工作区视角给出的路径。

### 3.5 多线程总结被文件/PDF失败状态污染

相关 turn：

| turn | 用户问题 | 结果 |
| --- | --- | --- |
| `turn-49-main` | 把 main、doc、ops、live 四条线程分开总结。 | 转而要求用户提供 PDF 路径 |

链路：

1. 用户要的是跨 session 线程总结，本应优先走 session memory / task summary。
2. 编排进入 `route=agent`，原因是 `fallback_bounded_lookup`。
3. agent 先尝试修复 PDF/文件上下文：
   - `pdf_analysis` 缺 path，返回 `explicit path is required`

   - 多个 PDF 路径返回不存在
4. 最终回答不是四线程总结，而是让用户确认 PDF 路径。

判断：这是上下文目标选择问题。旧失败任务优先级高于当前用户任务，导致“总结线程”被误解成“继续找 PDF 文件”。

## 5. 后续建议

优先级 P0：

1. 修正 `read_file` 的根目录语义。要么让 `ReadFileTool` 明确支持 repo root 只读路径，要么把工具描述、测试路径和前端路径统一到 backend root。
2. 为显式路径只读文件增加受控通道。`read_file` 不应完全开放自动路由，但用户明确给路径时，可以进入 `explicit_path_read` 安全分支。
3. 长测关键 turn 增加硬检查，例如 `response.not_contains_any=无法调用工具|file does not exist|path is a directory|没有稳定定位|工具调用过多`。

优先级 P1：

1. PDF 章节定位增加目录/标题别名：`第二部分`、`第2部分`、`第二章`、`Part 2` 应映射到候选标题。
2. PDF 页码降级要进入 warning，同时在测试系统前端可视化标红对应节点。
3. 对“总结线程 / 明天继续 / 先重启哪条”建立 planner 规则：优先从 session memory、task summary、flow snapshots 回答，不主动探索文件工具。

优先级 P2：

1. `orchestration_diff_status=warning` 需要区分“字段不可比”与“重要字段缺失”，避免 25 条 warning 噪声过高。
2. 测试报告可以按 warning 类型聚合，并链接到具体 turn artifact。
3. 前端测试系统中加入 warning filter：文件处理、PDF 降级、工具 guard、编排 diff 四类一键筛选。

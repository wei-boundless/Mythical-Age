# 编排系统运行控制塔实施计划书

日期：2026-04-26

## 1. 目标

编排系统不是一个静态说明页，也不应该只是把几个后端模块列出来。它应该成为工作台里的“运行控制塔”：用户可以在这里看到一次请求从进入后端到最终回答，中间经过了哪些决策、哪些分支、哪些能力调用、哪些状态写回，以及问题发生在第几个节点。

第一阶段目标是先把现有执行链透明化，不先改主运行逻辑。

也就是说，编排系统先做三件事：

- 看清当前会话最近一次请求的编排链路。
- 从测试系统跳转过来时，复盘某个 turn 的真实编排链路。
- 点击编排节点时，可以跳到系统框架、记忆系统、测试系统、操作系统或 agent 系统里继续排查。

第二阶段目标是把编排系统升级成“Agent 行为逻辑开发平台”。它不只看链路状态，还要能设计和调试 agent 的行为逻辑：面对什么输入如何理解任务、如何选择执行姿态、如何拆分或不拆分、何时澄清、何时使用记忆、何时调用技能或 worker、如何收口输出、失败后如何恢复。

暂时不做的事情：

- 不在第一阶段允许前端强行改写运行时路由。
- 不直接重构 `QueryRuntime` 主链。
- 不把编排系统做成新的测试系统，也不替代系统框架图。
- 不让页面展示一堆开发者日志，而是把运行事件整理成用户能理解的链路图。
- 不允许前端绕过 permission、tool contract、skill policy 直接调用高风险工具。

## 2. 当前真实链路

### 2.1 前端入口

前端发送消息的位置在：

- `frontend/src/lib/store/runtime.ts`
- `frontend/src/lib/store/events.ts`
- `frontend/src/lib/api.ts`

当前前端通过 `streamChat()` 调用 `/api/chat`，再用 SSE 事件更新 UI。SSE 事件里已经包含一部分编排信号，例如：

- `context_management`
- `memory_context`
- `prompt_manifest`
- `tool_start`
- `tool_end`
- `worker_start`
- `worker_evidence`
- `worker_artifacts`
- `worker_end`
- `done`
- `error`

问题是这些事件目前主要服务于聊天 UI，没有被整理成“编排链路视图”。所以用户看不到这轮请求为什么走某条路，也看不到问题具体卡在哪个阶段。

### 2.2 后端入口

后端聊天入口在：

- `backend/api/chat.py`

它做的事情很薄：

1. 校验 `session_id`。
2. 构造 `QueryRequest`。
3. 调用 `runtime.query_runtime.astream(request)`。
4. 把 runtime 事件转换成 SSE。

这说明编排系统不应该从 API 层重新发明执行逻辑，而应该读取 `QueryRuntime` 已经产出的事件、trace 和测试 artifact。

### 2.3 运行时装配

后端运行时装配在：

- `backend/runtime/app_runtime.py`

这里统一创建：

- `SessionManager`
- `SkillRegistry`
- `ToolRuntime`
- `MemoryFacade`
- `RetrievalService`
- `PermissionService`
- `ModelRuntime`
- `TaskCoordinator`
- `QueryRuntime`

编排系统页面应该把这里视为“运行时依赖图”的来源，而不是把它和单轮执行链混在一起。运行时装配回答的是“系统有什么能力”；单轮编排回答的是“这一轮用了什么能力”。

### 2.4 编排核心

真实编排核心在：

- `backend/query/runtime.py`
- `backend/query/planner.py`
- `backend/query/evidence_orchestrator.py`
- `backend/tasks/coordinator.py`
- `backend/runtime/model_runtime.py`

`QueryRuntime` 当前负责：

- follow-up 判断
- planner 构造
- `single_execution / bundle_execution / explicit_fanout` 三路选择
- context compaction
- memory context 读取
- prompt manifest 组装
- direct tool 执行
- evidence worker 执行
- model stream
- output boundary
- assistant message persistence
- session memory refresh
- durable memory extraction

已有 `docs/67-QueryRuntime分层拆解与结构坏味道排查计划书-20260425.md` 已经判断：`QueryRuntime` 是一个活链路很多的 God Object。编排系统第一阶段不能再给它加更多职责，而应该从外侧读取事件和 trace，做“可视化投影”。

### 2.5 测试系统和 trace

测试系统相关文件：

- `backend/experiments/runner.py`
- `backend/experiments/trace_graph.py`
- `backend/experiments/prompt_manifest.py`
- `backend/experiments/memory_trace.py`
- `frontend/src/components/workspace/views/TestSystemView.tsx`

测试系统已经支持：

- 运行测试 profile
- 读取 run artifacts
- turn 级列表
- prompt manifest
- memory trace
- issue 到系统框架图高亮

编排系统应该复用这些能力，把某个测试 turn 转换成可读的编排链路，而不是再做一套平行的测试报告。

## 3. 设计原则

### 3.1 先观测，后控制

编排系统第一阶段只做可视化和追踪，不直接改变执行链。

理由是当前主链已经很复杂，尤其是 follow-up、bundle、worker、direct tool、memory projection 之间有真实耦合。如果一开始就开放“手动指定路由”或“重跑某个阶段”，很容易把编排系统变成新的状态污染源。

### 3.2 运行事实优先，不用静态猜测代替真实事件

系统框架图适合表达项目结构，编排系统必须表达运行事实。

编排系统中的节点状态应优先来自：

1. 当前 SSE 事件。
2. 测试 turn artifact。
3. local trace。
4. prompt manifest。
5. memory trace。

如果只能静态推断，页面要明确标记为“推断链路”，不能伪装成真实执行。

### 3.3 编排系统只拥有“链路投影”，不拥有业务状态

业务状态仍然由各自系统负责：

- 会话内容归会话系统。
- 状态记忆归记忆系统。
- 长期记忆归记忆系统。
- 工具和 skills 归操作系统。
- worker、RAG、A2A 子 agent 归 agent 系统。
- 测试运行归测试系统。
- 项目结构归系统框架。

编排系统只把它们串成“这一轮发生了什么”。

### 3.4 节点可跳转，详情不堆墙

编排系统页面不应该把所有 JSON 摊开。节点只显示判断所需信息，详细文件通过跳转打开：

- prompt 节点跳到 prompt manifest。
- memory 节点跳到状态记忆文件或 memory trace。
- tool 节点跳到操作系统的 tool/skill 信息。
- worker 节点跳到 agent 系统或证据链路。
- test turn 节点跳回测试系统。
- system node 跳到系统框架图。

### 3.5 控制行为逻辑，而不是只控制工具开关

编排系统未来要控制的是 agent 的行为逻辑，不只是“这个 skill 开不开、这个 tool 能不能用”。

行为逻辑至少包括七层：

- 任务理解逻辑：如何识别用户是在问答、续接、纠错、切换对象、发起复杂任务还是要求治理记忆。
- 执行姿态逻辑：什么时候直答、什么时候走 `bounded_agent`、什么时候走 worker、什么时候走 `bundle_execution`、什么时候需要 `explicit_fanout`。
- 澄清逻辑：什么时候必须先问清楚，什么时候可以基于已有状态继续执行。
- 记忆逻辑：什么时候读取状态记忆、长期记忆，什么时候禁止写入，什么时候只读不注入。
- 能力逻辑：什么时候启用 skill，什么时候开放 tools，什么时候交给 retrieval/pdf/structured_data worker。
- 输出逻辑：如何决定回答结构、是否需要证据、是否需要只给结论、是否禁止暴露内部工具回执。
- 恢复逻辑：工具失败、worker 失败、模型超时、follow-up 飘移时，应该进入什么 fallback。

Skills、tools、worker 都只是行为逻辑里的“能力部件”。编排系统真正负责的是把这些部件组织成可解释、可测试、可调整的 agent 行为流程。

### 3.6 编排系统是 Agent 开发平台，不是运行时后门

编排系统应该像一个 agent 开发平台：

- 可以看当前 agent 的行为图。
- 可以编辑行为策略草案。
- 可以用测试场景验证策略。
- 可以比较策略版本效果。
- 可以把稳定策略沉淀成 prompt、policy 或 runtime 规则。

但它不应该成为绕过后端规则的后门。前端可以提出行为策略，后端仍然需要通过 planner、skill policy、tool contract、permission、memory policy 做最终裁决。

### 3.7 控制分三级开放

编排控制必须分级，不一步到位开放写权限。

第一级是只读观察：显示当前 agent 实际采用的行为逻辑，包括任务理解、执行姿态、能力选择、记忆策略和输出收口。

第二级是行为策略 dry-run：用户可以调整策略草案，例如“遇到多对象任务先 bundle，不自动 fanout”“follow-up 不明确时必须澄清”“PDF 类任务优先 worker”。后端返回这条输入会如何被理解、如何路由、会开放哪些能力、会装配哪些 prompt section。这个阶段不调用模型，不执行工具，不写 session。

第三级是受控执行：前端可以发起“本轮临时行为策略”，但只能通过结构化字段传给后端，例如 `behavior_policy_override`。后端必须重新经过 policy 层校验，策略只对当前 turn 生效，默认不持久化。

## 4. 目标页面结构

### 4.1 顶部：编排概览

显示当前编排上下文：

- 来源：当前会话 / 测试 turn / 手动选择。
- session_id。
- run_id / turn_id，如果来自测试系统。
- execution mode：`single_execution / bundle_execution / explicit_fanout`。
- route：`agent / worker / direct_tool / bundle / explicit_fanout`。
- 当前状态：成功 / 失败 / 警告 / 缺少 trace。
- 最可疑节点：如果有失败信息，直接显示“问题出现在第 N 个节点”。

### 4.2 主体：运行链路图

链路图推荐固定成横向主线加分支：

```text
用户输入
-> follow-up 仲裁
-> planner
-> 执行模式选择
-> context compaction
-> memory context
-> prompt assembly
-> capability dispatch
-> model / worker / direct tool
-> output boundary
-> session persistence
-> post-turn memory refresh
```

节点需要显示：

- 节点序号。
- 节点名称。
- 状态：未经过 / 已经过 / 警告 / 失败。
- 关键摘要。
- 来源事件。
- 可跳转动作。

能力分支需要独立显示：

- `worker_start -> worker_evidence -> worker_artifacts -> worker_end`
- `tool_start -> tool_end -> new_response`
- `prompt_manifest`
- `memory_context`
- `done`

### 4.3 右侧或下方：节点详情

点击节点后展示：

- 这个节点做什么。
- 本轮输入是什么。
- 本轮输出是什么。
- 使用了哪些文件或状态。
- 如果失败，失败原因是什么。
- 可以跳到哪里继续排查。

详情要偏自然语言，不要只显示原始 JSON。

### 4.4 底部：事件时间线

把 SSE 或测试 artifact 里的事件按时间列出来：

- 事件名。
- 阶段。
- 摘要。
- 是否影响最终回答。
- 是否产生记忆、prompt、tool、worker 或 artifact。

这个区域可以保留原始 JSON 展开按钮，但默认不展开。

### 4.5 行为逻辑开发台

编排系统需要增加一个独立分页，名称建议为 `行为逻辑`。

它展示五块内容：

- 行为策略图：把 agent 的核心判断画成可读流程，例如“输入理解 -> follow-up 仲裁 -> 执行姿态 -> 能力选择 -> 记忆策略 -> 输出收口 -> 失败恢复”。
- 当前策略状态：显示本轮采用的 route、execution mode、execution posture、follow-up mode、memory intent、active skill、worker route、output policy。
- 策略编辑草案：允许用户编辑结构化策略草案，例如多任务策略、澄清策略、记忆读写策略、能力选择策略、输出收口策略。
- 策略预演：对一条输入做 dry-run，显示它会如何被理解、走哪个分支、开放哪些能力、为什么拒绝某些路径。
- 策略验证：把一个策略草案绑定到测试系统 profile 或指定 scenario，观察成功率、失败节点和行为差异。

第一版可以只做“行为图 + 当前策略状态”。第二版做 dry-run。第三版做策略草案和测试系统验证。第四版才考虑把稳定策略写回后端 policy 或 prompt 配置。

## 5. 数据模型设计

### 5.1 前端运行态模型

新增一个前端编排投影类型：

```ts
type OrchestrationNode = {
  id: string;
  index: number;
  label: string;
  status: "idle" | "visited" | "warning" | "failed" | "success";
  summary: string;
  sourceEvent: string;
  jumpTarget?: {
    view: WorkspaceView;
    reason: string;
    payload: Record<string, unknown>;
  };
};

type OrchestrationEdge = {
  id: string;
  from: string;
  to: string;
  label: string;
  status: "idle" | "visited" | "warning" | "failed" | "success";
  summary: string;
};

type OrchestrationSnapshot = {
  source: "live-session" | "test-turn" | "inferred";
  sessionId: string;
  runId?: string;
  turnId?: string;
  executionMode: string;
  route: string;
  status: "idle" | "running" | "success" | "warning" | "failed";
  problemNodeId?: string;
  nodes: OrchestrationNode[];
  edges: OrchestrationEdge[];
  events: Array<Record<string, unknown>>;
};
```

### 5.2 Agent 行为策略模型

新增一个行为策略模型：

```ts
type AgentBehaviorPolicy = {
  policyId: string;
  title: string;
  scope: "draft" | "turn" | "session" | "project";
  taskUnderstanding: {
    followupMode: "auto" | "clarify_when_ambiguous" | "force_new_task";
    correctionHandling: "auto" | "prefer_latest_user_correction";
    objectSwitchHandling: "auto" | "require_explicit_target";
  };
  executionStrategy: {
    defaultMode: "single_execution" | "bundle_execution" | "explicit_fanout";
    bundleWhenMultiAnchor: boolean;
    fanoutRequiresExplicitSubtasks: boolean;
    directToolAllowed: boolean;
    workerAllowed: boolean;
  };
  memoryStrategy: {
    sessionRead: "auto" | "always" | "never";
    durableRead: "auto" | "intent_only" | "never";
    durableWrite: "auto" | "manual_only" | "never";
  };
  capabilityStrategy: {
    preferredSkill?: string;
    allowedToolNames?: string[];
    allowedWorkerRoutes?: string[];
  };
  outputStrategy: {
    answerStyle: "auto" | "concise" | "structured" | "evidence_first";
    hideInternalReceipts: boolean;
    requireCanonicalAnswer: boolean;
  };
  recoveryStrategy: {
    onToolFailure: "fallback_answer" | "ask_clarification" | "stop";
    onWorkerFailure: "fallback_answer" | "ask_clarification" | "stop";
    onLowConfidenceFollowup: "clarify" | "continue_with_warning";
  };
};
```

运行时再生成一个行为快照：

```ts
type AgentBehaviorSnapshot = {
  sessionId: string;
  source: "live-session" | "test-turn" | "dry-run";
  route: string;
  executionMode: string;
  executionPosture: string;
  followupMode: string;
  memoryReadMode: string;
  memoryWriteMode: string;
  activeSkillName: string;
  allowedTools: string[];
  workerRoute: string;
  outputPolicy: string;
  decisions: Array<{
    stage: string;
    selected: string;
    alternatives: string[];
    reason: string;
  }>;
  rejectedPaths: Array<{
    stage: string;
    path: string;
    reason: string;
  }>;
};
```

这个模型的意义是：编排系统可以管理 agent 的行为路线，而不是只管理 skill/tool 开关。

### 5.3 后端只读接口

建议新增：

```text
GET /api/orchestration/session/{session_id}/snapshot
GET /api/orchestration/runs/{run_id}/turns/{turn_id}/snapshot
```

第一阶段也可以先不新增 session 接口，而是在前端从 SSE 事件组装 live snapshot。但测试 turn snapshot 建议后端做，因为 artifact 解析逻辑已经在 `backend/experiments` 里。

### 5.4 行为策略 dry-run 接口

后续建议新增：

```text
POST /api/orchestration/behavior/dry-run
```

输入示例：

```json
{
  "session_id": "...",
  "message": "...",
  "policy": "...AgentBehaviorPolicy..."
}
```

dry-run 输出应该说明：

- 这句话被理解成什么任务。
- follow-up 是否成立。
- 会选择什么 execution mode。
- 会启用什么 memory 策略。
- 会开放什么 skill/tool/worker。
- 哪些路径被拒绝，拒绝原因是什么。
- prompt 装配会出现哪些主要 section。

dry-run 必须只读：不调用模型，不执行工具，不写 session，不写长期记忆，不改变全局配置。

### 5.5 测试 artifact 解析

测试 turn artifact 中可能包含：

- `plan`
- `events`
- `result`
- `prompt_manifest`
- `memory_trace`
- `trace_url`
- failed checks

编排系统需要把这些转换成统一节点：

- plan -> planner / execution mode 节点。
- events -> event timeline。
- prompt_manifest -> prompt assembly 节点。
- memory_trace -> memory context 节点。
- worker events -> worker 分支。
- tool events -> direct tool 分支。
- result -> output boundary / done 节点。

## 6. 页面联动设计

### 6.1 与测试系统联动

测试系统 turn 卡片增加动作：

- `查看编排链路`

点击后：

1. 设置 `orchestrationInspectorTarget`。
2. 切换到 `编排系统`。
3. 编排系统读取该 run/turn 的 snapshot。
4. 自动高亮失败节点或最后一个异常节点。

### 6.2 与记忆系统联动

编排图中的 `memory context` 节点提供动作：

- `查看状态记忆文件`
- `查看真实 memory trace`

如果来自当前会话：

- 跳到记忆系统的 `状态记忆` 页面。
- 选中当前 `session_id`。

如果来自测试 turn：

- 跳到记忆系统。
- 携带 `run_id / turn_id / turnIndex`。
- 显示真实 memory trace。

### 6.3 与系统框架联动

编排图中每个节点都可以映射到系统框架节点：

- API 入口 -> `api-router`
- runtime 装配 -> `runtime-root`
- planner -> `planner`
- prompt -> `prompt`
- memory -> `memory`
- worker/evidence -> `evidence`
- retrieval -> `retrieval`
- tools/skills -> `tooling`
- model -> `model`
- persistence -> `session-store / storage`

点击“在系统框架定位”后，系统框架图高亮对应节点和边。

### 6.4 与操作系统联动

当节点是 direct tool 或 skill policy：

- 显示 tool name。
- 显示 active skill。
- 提供跳转到操作系统的按钮。

后续操作系统页面可以读取 tool registry、skill registry、MCP catalog。

编排系统不重复实现操作系统。它决定“agent 在什么行为逻辑下会选择某个 skill/tool”，真正的 tool/skill 定义、脚本、权限和 schema 仍然跳到操作系统维护。

### 6.5 与 agent 系统联动

当节点是 worker 或 evidence orchestrator：

- 显示 worker route：`retrieval / pdf / structured_data`。
- 显示 A2A task envelope 的状态。
- 提供跳转到 agent 系统的按钮。

第一阶段可先跳转到 agent 系统占位页，第二阶段再做 worker 详情。

## 7. 分阶段实施计划

### Phase 1：前端编排系统页面重构

目标：把 `ExperimentsView.tsx` 从静态卡片改成“编排控制塔”。

改动文件：

- `frontend/src/components/workspace/views/ExperimentsView.tsx`
- `frontend/src/app/globals.css`

完成标准：

- 页面有固定链路图。
- 页面有节点详情面板。
- 页面有事件时间线占位。
- 当前没有 live/test 数据时，也能显示“推断链路”。
- 不再只是四个静态节点和几条约束。

### Phase 2：前端 live SSE 编排快照

目标：当前会话发送消息时，编排系统能看到最近一次 live 运行链路。

改动文件：

- `frontend/src/lib/store/types.ts`
- `frontend/src/lib/store/events.ts`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/components/workspace/views/ExperimentsView.tsx`

事件处理：

- `context_management` 标记 context compaction 节点。
- `memory_context` 标记 memory 节点。
- `prompt_manifest` 标记 prompt 节点。
- `worker_start/worker_end` 标记 worker 分支。
- `tool_start/tool_end` 标记 tool 分支。
- `done` 标记 output/persistence 节点。
- `error` 标记失败节点。

完成标准：

- 用户正常聊天后，进入编排系统能看到刚才那轮实际经过的节点。
- tool 或 worker 事件能在链路图中出现。
- error 能明确标红最后阶段。

### Phase 3：测试 turn 编排快照接口

目标：从测试系统选择某个 turn 后，编排系统可以复盘该 turn 的运行链路。

修改文件：

- `frontend/src/components/workspace/views/ExperimentsView.tsx`

接口：

```text
GET /api/orchestration/runs/{run_id}/turns/{turn_id}/snapshot
```

完成标准：

- 测试系统 turn 卡片能跳转到编排系统。
- 编排系统能显示该 turn 的 plan、events、prompt、memory、worker/tool、result。
- 如果 turn failed，能显示最可能的问题节点。

### Phase 4：跨系统跳转

目标：编排系统成为排查入口。

改动文件：

- `frontend/src/lib/store/types.ts`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/components/workspace/views/ExperimentsView.tsx`
- `frontend/src/components/workspace/views/SystemFrameworkView.tsx`
- `frontend/src/components/workspace/views/MemoryView.tsx`

完成标准：

- 点击 memory 节点，可以跳到记忆系统。
- 点击 prompt 节点，可以跳到测试系统或系统框架中的 prompt manifest 面板。
- 点击 worker/tool 节点，可以跳到操作系统或 agent 系统。
- 点击 system location，可以跳到系统框架高亮节点。

### Phase 5：编排控制能力预留

目标：为未来行为逻辑开发做边界设计，但不在第一轮开放高风险操作。

可预留能力：

- 预览某个用户输入会走什么行为路径。
- dry-run 行为策略。
- 比较两个行为策略在同一条输入上的差异。
- 把行为策略草案绑定到测试系统 scenario。
- 对某个失败节点发起定向策略验证。

暂不开放：

- 前端直接覆盖 route。
- 前端直接修改 worker route。
- 前端直接写入 session runtime state。
- 前端直接跳过 permission 或 tool contract。

### Phase 6：行为逻辑只读开发台

目标：把编排系统从链路回放升级成 agent 行为逻辑开发平台的只读版。

改动文件：

- `frontend/src/components/workspace/views/ExperimentsView.tsx`
- `frontend/src/lib/api.ts`
- `backend/experiments/orchestration_trace.py`
- `backend/query/planner.py`
- `backend/skill_system/policy.py`

完成标准：

- 页面展示行为策略图，而不是只有执行链图。
- 页面能展示本轮任务理解、follow-up、execution mode、memory intent、active skill、worker/tool、output policy。
- 每个行为决策都能显示“选择了什么 / 放弃了什么 / 为什么”。
- 只读，不允许前端改变行为。

### Phase 7：行为策略 Dry-run

目标：用户可以编辑一份行为策略草案，然后对一条输入做预演。

改动文件：

- `backend/api/orchestration.py`
- `backend/query/planner.py`
- `backend/understanding/query_understanding.py`
- `backend/skill_system/policy.py`
- `frontend/src/components/workspace/views/ExperimentsView.tsx`

完成标准：

- dry-run 返回 `AgentBehaviorSnapshot`。
- 能解释 route、execution mode、follow-up、memory、skill/tool/worker 的决策原因。
- 能显示被拒绝的路径和原因。
- 不调用模型，不执行工具，不写状态。

### Phase 8：策略验证与沉淀

目标：把编排系统接入测试系统，让行为策略可以被场景验证。

能力：

- 选择一个策略草案。
- 选择一个测试 profile 或 scenario。
- 运行策略验证。
- 比较基线策略和草案策略的失败节点、耗时、工具调用、follow-up 稳定性、输出泄漏。
- 验证稳定后，再决定是否沉淀到 prompt、planner policy 或 runtime policy。

暂不自动做的事：

- 不自动改 `CORE.md`。
- 不自动改 seed。
- 不自动改 skill 文件。
- 不自动改 runtime policy。

所有沉淀都必须是显式确认后的工程改动。

## 10. 风险与控制

### 风险 A：编排系统变成第二套系统框架

控制方式：

- 系统框架展示项目结构。
- 编排系统展示单轮运行事实。
- 两者通过节点映射互通，但不合并。

### 风险 B：编排系统显示静态推断，用户以为是真实执行

控制方式：

- snapshot 必须带 `source`。
- 页面明确显示 `live-session / test-turn / inferred`。
- 推断链路只作为空状态，不作为 debug 证据。

### 风险 C：第一阶段改动主链导致新 bug

控制方式：

- 第一阶段只改前端展示。
- 第二阶段只读取 SSE 事件。
- 第三阶段只读测试 artifact。
- 不改 `QueryRuntime` 路由决策。

### 风险 D：节点过多，页面再次变杂

控制方式：

- 主链固定 10-12 个节点。
- 分支节点只在真实出现时展开。
- 原始 JSON 默认折叠。
- 详情面板只显示当前选中节点。

### 风险 E：编排系统退化成 tool/skill 开关面板

控制方式：

- 页面主对象必须是行为策略，而不是工具列表。
- tool/skill 只作为行为策略下的能力结果展示。
- 用户编辑的是“何时怎样行动”的策略，而不是直接命令模型“必须调用某工具”。
- 操作系统负责 tool/skill 详情，编排系统负责行为逻辑。

### 风险 F：行为策略变成运行时后门

控制方式：

- dry-run 和 execute 分开。
- 策略只能通过结构化模型进入后端。
- 后端 policy 是最终裁决者。
- 所有策略验证必须可追溯。
- 沉淀到 prompt 或 runtime policy 前必须显式确认。

## 11. 推荐实施顺序

建议按以下顺序推进：

1. 重构 `ExperimentsView.tsx` 为编排控制塔静态骨架。
2. 加入 live SSE snapshot，让当前会话能看到最近一次编排链路。
3. 新增测试 turn snapshot API，把测试系统联动接上。
4. 加跨系统跳转，尤其是记忆系统和系统框架。
5. 增加行为逻辑只读开发台。
6. 增加行为策略 dry-run。
7. 接入测试系统做策略验证。
8. 最后再考虑把稳定策略沉淀到 prompt、planner policy 或 runtime policy。

这样做的好处是：每一步都有可见收益，每一步都不会破坏主执行链。

## 12. 最终形态

完成后，用户可以这样使用编排系统：

1. 对话里问一个复杂问题。
2. 打开编排系统。
3. 看到这轮请求是 `single_execution`、`bundle_execution` 还是 `explicit_fanout`。
4. 看到它是否触发了记忆、prompt 装配、工具、worker、模型流。
5. 如果失败，页面直接标出第几个节点出了问题。
6. 点击问题节点，跳到对应系统继续查看真实文件、trace、prompt 或测试报告。
7. 打开行为逻辑页，看到 agent 为什么这样理解任务、为什么选择这条执行姿态、为什么读或不读记忆、为什么启用或拒绝某个 skill/worker。
8. 编辑一个行为策略草案，对同一条输入 dry-run，比较策略差异。
9. 把策略草案接入测试系统验证，确认它是否真的改善长场景表现。

这就是编排系统和其它工作台模块的边界：它不是另一个文档页，也不是简单的工具开关面板，而是一个用于开发 agent 行为逻辑的控制塔。它先观察真实运行，再设计行为策略，再用测试系统验证，最后才把稳定策略沉淀到 prompt、planner policy 或 runtime policy。

# 50-长场景显性语义错误问题清单与根因追踪-20260423

## 1. 文档目的

本文件用于沉淀 `2026-04-23` 长场景正式实测中暴露出来的显性语义错误，并把这些错误沿着主链代码继续追踪到可落点的坏点位置。

这里关注的是两类问题：

1. 用户已经明显感知到的回答错误。
2. 即使测试未判失败，但实际回答已经失去业务语义的情况。

本文件暂不直接给出修复代码，只做问题归档、证据挂接、根因追踪和修复优先级排序。

## 3. 结论总览

这次长场景里的显性语义错误，不是单一“大模型理解偏差”，而是下面 6 类链路问题叠加出来的：

1. 工具执行失败或工具摘要缺失时，主链把“状态句”直接包装成最终答案。
2. PDF 降级结果仍允许走模型改写，导致模型在弱证据上硬生成正文语义。
3. 复合任务拆分后，子任务既可能丢失原始 source binding，也没有强制要求“给结论”，只要执行过就可能回动作状态。
4. binding follow-up 优先级过高，导致全局总结类问题被强行拖回单一 PDF 绑定。
5. 线程总结会继承前序错误结果，把错误状态继续固化为“已知事实”。
6. 本地文件 / 本地 docs / 数据集路径链路存在源解析缺口，失败时又被统一兜底句掩盖。

其中 `1 + 4 + 6` 是当前最优先的主链问题。

## 5. 逐项问题与根因追踪

### S1. 工具状态泄漏成最终答案

#### 5.1.1 现象

用户问的是业务问题，但最终回答直接变成了工具状态句：

- `工具 structured_data_analysis 已执行，但当前结果尚未形成可直接展示的答案。`
- `工具 get_weather 已执行，但当前结果尚未形成可直接展示的答案。`
- `已读取与当前问题最相关的 PDF 页面：P70、P71、P69，但当前还没有形成稳定摘要。`

#### 5.1.2 证据

- `run_result.json:560`
- `run_result.json:1032`
- `run_result.json:4208`
- `turn-18-main.json`
- `turn-58-main.json`

#### 5.1.3 已确认坏点

主坏点在 `backend/query/output_classifier.py`：

- `build_output_decision(...)`
- `build_route_fallback(...)`

当前逻辑是：

1. 如果没有抽取出显式答案或工具可见摘要，直接进入 fallback。
2. fallback 不区分“工具执行失败”“工具返回异常”“工具返回空摘要”“工具返回不可展示中间态”。
3. fallback 直接生成一条完整自然语言句子，并被主链当成正式答案返回。

对应代码行为：

- `rag` 无检索时，直接回 `当前本地知识库没有检到足够相关材料...`
- `pdf_analysis` 缺摘要时，直接回 `已读取最相关页面...但没有形成稳定摘要`
- 任何其他工具缺摘要时，直接回 `工具 xxx 已执行，但当前结果尚未形成可直接展示的答案`

#### 5.1.4 初步根因

这不是“模型没答好”，而是主链策略把“不可展示中间态”错误地升级成了“可展示最终答案”。

#### 5.1.5 影响范围

- 结构化数据
- PDF
- 天气
- web_search
- 未来所有 direct tool

#### 5.1.6 结论

`S1` 是当前最核心的链路问题之一。只要这个策略不改，很多工具问题都会被伪装成“用户看起来像拿到了回答”，但语义其实已经失败。

### S2. PDF 页面语义前后矛盾

#### 5.2.1 现象

同一份 PDF 的第三页，在同一次长场景中出现了两种互相冲突的回答：

- 一次回答为“第三页是引言部分”
- 多次回答为“第三页是封面或题名页”

#### 5.2.2 证据

- `run_result.json:1789`
- `run_result.json:1901`
- `run_result.json:2584`
- `run_result.json:3482`
- `run_result.json:5599`
- `turn-18-main.json`
- `turn-27-main.json`

#### 5.2.3 已确认坏点

主因链在两处：

1. `backend/query/runtime.py`
   - `_pdf_tool_result_can_use_model_finalization(...)`
   - `_pdf_canonical_has_finalizable_evidence(...)`
   - `_rewrite_pdf_answer_with_model(...)`

2. `backend/pdf_agent/runtime.py`
   - `_run_page_scope(...)`
   - `_run_document_scope(...)`

已经追出的新增硬证据：

3. 在复合任务场景里，子问题 `总结 PDF 第三页` 执行时 `active_pdf=""`
4. `tool_start.input` 只有 `query=总结 PDF 第三页, mode=page`，没有 `path`
5. `PdfAnalysisTool` 在无 `path` 时，会调用 `PdfAnalysisCatalog.resolve_pdf_path(...)` 按查询词对整库 PDF 重新打分
6. 最终命中的并不是目标文档，而是 `knowledge/AI Knowledge/2025全球人工智能技术应用洞察报告.pdf`
7. 更深一层已经确认，在同一会话稍早的 `turn-18-main.json` 中，系统其实已经有稳定的
   `active_pdf=knowledge/AI Knowledge/2025年AI治理报告：回归现实主义.pdf`
8. 但在复合问题 `先总结 PDF 第三页，再给我 inventory.xlsx 最缺货的前三个仓库，最后补一句北京天气。` 中，
   `backend/query/followup_resolver.py::resolve(...)` 会先命中 `_looks_explicit(...)`
9. 因为整句里出现了 `inventory.xlsx`，resolver 直接返回 `resolution_source="explicit_input"`，
   不再继续做 binding 解析
10. 于是整轮请求不会先继承“刚才那份 PDF”，而是直接进入 compound planner
11. `backend/query/planner.py::build_plan(...)` 再对子问题逐个重新 `_build_execution(...)`
12. `backend/tasks/coordinator.py::_build_task_context_ref(...)` 只从子问题文本重建 binding，
    不会继承父请求已有的 `active_pdf`

#### 5.2.4 初步根因

当前 PDF 主链允许这样一条路径：

1. PDF canonical 结果不是稳定正文摘要。
2. 但只要 evidence snippet 足够长，就允许继续走模型 finalization。
3. 模型在弱证据、封面页、题名页、稀疏页上，仍然会尝试“补足用户答案形态”。
4. 一旦补写，主链没有做前后事实一致性约束，也没有对“封面页不得解释为正文”的强校验。

结果就是：

- 同一页，在不同上下文、不同追问方式下，模型可能给出不同语义。
- 主链没有“页面事实锁”，导致矛盾结果都能被持久化或继续引用。

但这次追踪已经证明，`第三页是引言` 这一支并不只是“同一页被模型补写错了”，还包含更前面的 source 错配：

1. 复合子任务没有继承父任务的 `active_pdf`
2. 无 path 的 PDF 子任务会退化成“整库猜文档”
3. 一旦命中别的 PDF，错误内容仍会被当成合法结果写入 `task_results`、`session-memory` 和后续总结

并且现在已经能把这条 source 错配链落到正式代码断点：

1. 会话层原本已有正确 PDF binding
2. 复合问题里混入显式数据文件名后，`followup_resolver` 被 `explicit_input` 提前短路
3. compound planner 不再带着父 binding 做整体规划
4. 子问题独立重建执行计划时，`总结 PDF 第三页` 本身没有 path
5. coordinator 也没有父 binding 继承能力
6. 最终 PDF 子任务只能回退成“整库猜文档”

因此这里的矛盾并不是单一页级事实冲突，而是“错文档结果”和“对文档结果”同时进入了会话链。

#### 5.2.5 结论

`S2` 的本质不是 PDF parser 单点错误，而是三条问题叠加：

1. 复合子任务丢失原始 PDF binding
2. 弱 canonical / 错文档结果仍被当成合法内容继续传播
3. 没有事实一致性门禁

其中第 `1` 条现在已经不是泛化判断，而是已落实到：

- `followup_resolver` 的 `explicit_input` 早退策略
- `planner` 的子任务独立重建
- `coordinator` 的文本级 binding 推断

这三层共同构成了 PDF source inheritance 的正式断点。

### S3. PDF 分段问题未回答，只回页码状态

#### 5.3.1 现象

用户问：

- `回到刚才 PDF，第二部分的约束重点是什么？`

系统回：

- `已读取与当前问题最相关的 PDF 页面：P70、P71、P69，但当前还没有形成稳定摘要。`

这不是“答得不好”，而是没有回答问题本身。

#### 5.3.2 证据

- `turn-18-main.json`
- `run_result.json:4208`

#### 5.3.3 已确认坏点

主因链：

1. `backend/pdf_agent/runtime.py::_run_section_scope(...)`
2. `backend/query/output_classifier.py::build_route_fallback(...)`

当前 section 流程如果无法稳定定位章节，会返回 degraded result，并附带若干候选页 evidence。
但主链后面没有把它转成“明确失败说明 + 请求补充定位”的结构化回答，而是统一落到：

- `pdf_canonical_missing_summary`

然后生成：

- `已读取最相关页面，但没有形成稳定摘要`

#### 5.3.4 初步根因

section 失败在语义上本来是“章节定位失败”，但主链把它降格成了“页面已读取、摘要未形成”。

这会误导用户：

- 用户以为系统已经找到了正确内容，只是没总结出来。
- 实际上系统连目标正文都没稳定定位到。

#### 5.3.5 结论

`S3` 是 PDF 失败态表达错误，不只是读取质量问题，更是失败语义被错误包装。

### S4. 复合任务拆分后只回动作，不回结论

#### 5.4.1 现象

在复合任务里，系统会返回：

- `已读取 P3。`

而不是回答“第三页讲了什么”。

#### 5.4.2 证据

- `run_result.json:4768`
- `run_result.json:4880`
- `turn-27-main.json`

#### 5.4.3 已确认坏点

主因链：

1. `backend/query/answer_assembler.py`
2. `backend/query/runtime.py::_assemble_subtask_results(...)`
3. `backend/query/output_classifier.py`
4. `backend/query/subtask_planner.py`
5. `backend/pdf_analysis/catalog.py`
6. `backend/query/tool_input_resolver.py`

当前复合任务链路只要求：

- 子任务有 `summary.response` 就拿来拼。
- 如果没有 `summary.response`，则退成 `任务已执行...`

它没有要求子任务必须满足：

- 已形成业务结论
- 已回答原始提问意图
- 不是工具状态句

同时已经确认，复合子任务的 source 继承并不稳定：

- PDF 子任务 `总结 PDF 第三页` 没有继承 `active_pdf`
- 结构化数据子任务虽然已有正确 `structured_binding.dataset_path=knowledge/E-commerce Data/inventory.xlsx`
- 但最终 `tool_start.input.path` 却被写成了 `inventory.xlsx`

此外，这轮深追又确认了一条“组装层风格串染”链：

1. 复合问题里的第三个子任务是 `补一句北京天气`
2. 该子任务在自己的 `context_ref.constraints.response_style` 中被标记为 `one_sentence`
3. `backend/query/runtime.py::_build_compound_main_context(...)` 会把最后一个 task summary 的 `response_style`
   提升成 compound 的默认 `active_constraints.response_style`
4. `backend/query/runtime.py::_merge_constraints_from_results(...)` 也会把任一子任务的 `summary.response_style`
   往全局 constraints 合并
5. `backend/query/answer_assembler.py::build_plan(...)` 再把这个全局 `default_style` 应用于所有 segment
6. 所以第一个 PDF 子任务虽然原始 `summary.response` 是长文本，但在最终 render 前被统一裁成了首句：`已读取 P3。`

#### 5.4.4 初步根因

复合任务链路当前检查的是“子任务是否有字符串”，不是“子任务是否有回答”。

所以像：

- `已读取 P3`
- `工具已执行`

这种动作状态，也会被当成合法子任务结果直接拼回总回答。

并且复合链路没有把“父问题已绑定的 source owner”稳态传给每个 subtask：

- PDF 子任务会因为缺 path 而回退成“语义猜 PDF”
- 数据子任务会因为 tool input 组装策略而把正确 binding 覆盖成不可用短路径

现在还可以再补一层：

- 它不仅缺“可回答性”门禁
- 还把局部子任务的表达风格，错误提升成了整个 compound answer 的全局风格

因此 `补一句` 这种局部约束，会把不相关的 PDF 段也压成一句话。

所以 `S4` 不是单一“答案组装问题”，而是“subtask 继承上下文不完整 + 组装质量门禁不足”的组合问题。

#### 5.4.5 结论

`S4` 现在至少包含三个坏点：

1. 复合子任务 source 继承链不稳定
2. 组装前没有“是否真正回答问题”的门禁
3. 子任务级 `response_style` 会污染整个 compound answer

### S5. 全局总结被错误拖回单一 PDF 绑定

#### 5.5.1 现象

用户问：

- `最后给我一个总总结，按 PDF、数据、实时、长期记忆四段组织，而且先给结论。`

但实际执行时被导向了 `pdf_analysis`，最后只输出了一段 PDF 相关内容。

#### 5.5.2 证据

- `turn-59-main.json`
- `run_result.json:6643`

#### 5.5.3 已确认坏点

主因链在 `backend/query/runtime.py`：

1. `_execution_events(...)`
2. `_binding_owner_task(...)`
3. `_stream_binding_followup(...)`

当前顺序是：

1. 先做 follow-up resolution。
2. 如果命中 `binding_ref` 且存在 owner task，就直接进入 `_stream_binding_followup(...)`。
3. 这一步发生在正常 planner 之前。

因此只要当前会话里还挂着一个活跃 PDF binding，全局总结类问题就可能被“误认成继续追问 PDF”。

#### 5.5.4 初步根因

当前 follow-up binding 优先级太高，而且没有做“当前问题是否显式请求跨模态汇总”的拦截。

这使得：

- 单源 follow-up
- 全局 cross-source synthesis

两类问题在主链里没有被正式区分。

#### 5.5.5 结论

`S5` 是一个明确的路由优先级错误。只要不降级 binding follow-up 的触发范围，总结类、多源类问题还会继续被单模态劫持。

### S6. 本地文件 / 本地数据源失败后被兜底掩盖

#### 5.6.1 现象

有些任务明明是本地 docs、本地 PDF、本地 Excel，但最终却表现成：

- `当前本地知识库没有检到足够相关材料...`
- 或统一工具兜底句

这会把“源解析失败 / 路径不对 / 本地文件链路缺口”伪装成“知识语义不足”。

#### 5.6.2 证据

- `run_result.json:2241`
- 本次离线复现中，直接调用工具出现：
  - `结构化分析失败：文件不存在`
  - `PDF analysis failed ... produced no readable text`

#### 5.6.3 已确认坏点

已确认的代码入口：

1. `backend/structured_data/catalog.py`
2. `backend/query/tool_input_resolver.py`
3. `backend/understanding/task_understanding.py`
4. `backend/query/output_classifier.py`
5. `backend/query/binding_resolver.py`

#### 5.6.4 初步根因

当前至少存在两个问题：

1. 本地文件源解析没有形成统一的“源存在性 / 路径可达性 / 本地文件优先级”校验链。
2. 一旦底层失败，主链没有把“文件不存在”“路径没绑定”“本地源未命中”原样暴露，而是继续套统一兜底句。

这导致：

- 用户看不到真实失败原因。
- 排障时很难区分“语义理解失败”和“源接入失败”。

这里已经追出一条明确且可复现的结构化数据错误链：

1. 对于 `给我 inventory.xlsx 最缺货的前三个仓库`
2. `backend/understanding/task_understanding.py::_detect_structured_data_task(...)`
   因为识别到了显式文件名，会先把 `parameters["path"]` 写成短名 `inventory.xlsx`
3. `query_understanding.tool_input.path` 因而从一开始就是 `inventory.xlsx`
4. `backend/query/binding_resolver.py::resolve(...)` 随后确实把它解析成了正确 binding：
   `knowledge/E-commerce Data/inventory.xlsx`
5. 但这个 binding 的 `source` 仍然是 `prebound_tool_input`，说明它是基于已有短路径再补解析出来的
6. `backend/query/tool_input_resolver.py::resolve(...)` 对结构化数据的策略是：
   只有当 `tool_input.path` 为空时，才把 `structured_binding.dataset_path` 写回
7. 由于短路径已经存在，所以正确全路径不会覆盖错误短路径
8. 最终 `tool_start.input.path` 实际仍是 `inventory.xlsx`
9. 在 `backend` 根下，这个短路径并不存在
10. `StructuredDataAnalysisTool._run(...)` 底层真实会得到 `结构化分析失败：文件不存在。`
11. 这个真实错误没有直接暴露，而是又被 `output_classifier.build_route_fallback(...)` 包装成：
   `工具 structured_data_analysis 已执行，但当前结果尚未形成可直接展示的答案。`

也就是说，部分结构化数据问题表面看像“工具执行但没总结”，实际上根因是“正确 binding 被错误 path 覆盖了”。

这条链也说明了一个更底层的优先级错误：

1. 显式短文件名 `inventory.xlsx`
2. 解析后的正式 binding `knowledge/E-commerce Data/inventory.xlsx`

当前系统把 `1` 当成最终工具入参，把 `2` 仅当辅助元数据，并没有把 `2` 视为正式权威 source。

#### 5.6.5 说明

这一项当前已确认存在，但还需要继续区分：

- 是测试运行目录差异导致的路径问题
- 还是正式执行链本身依赖额外注入状态
- 还是 local file / knowledge / dataset 三种源还没彻底统一

#### 5.6.6 结论

`S6` 目前判定为“已发现的高风险运行时问题”，需要继续追到更细的运行时层。

## 7. 按设计原则回看的边界失守矩阵

结合 `docs/设计原则` 当前已经明确可执行的约束，这次长场景问题不是单点 bug，而是 5 层边界同时失守：

1. 源绑定边界
2. 恢复候选与当前决策边界
3. 任务局部状态与主线程状态边界
4. 执行工件与最终答案边界
5. 局部风格与全局展示边界

下面把它们和当前问题链正式对齐。

### 7.1 源绑定边界失守

对应设计原则：

- `docs/设计原则/03-状态管理.md`
- `docs/设计原则/25-架构模式总结.md`

约束抽象：

- 一个 source binding 必须有明确 owner。
- 当前回合的正式 source 不能在多个阶段被重复猜测、重复覆盖。
- 下游阶段应消费上游已经确认的 binding，而不是重新从文本猜一遍。

当前坏点：

- `backend/query/followup_resolver.py`
- `backend/query/planner.py`
- `backend/tasks/coordinator.py`

现象对照：

1. 父回合里明明已经有稳定 `active_pdf`。
2. 复合请求里只要混入 `inventory.xlsx` 这类显式文件名，`followup_resolver` 就先命中 `explicit_input`。
3. compound planner 没有把父 binding 作为正式输入下发，而是对子问题重新 `_build_execution(...)`。
4. `coordinator` 又只根据子问题文本重新推导 `active_pdf/active_dataset`。

结论：

这意味着当前系统里真正的 source owner 并不存在。binding 不是“被持有并传递”，而是“谁后面又猜到一次谁就接管”。这和状态管理里强调的单一事实源是冲突的。

### 7.2 恢复候选与当前决策边界失守

对应设计原则：

- `docs/设计原则/05-对话循环.md`
- `docs/设计原则/06-上下文管理.md`
- `docs/设计原则/23-Memory系统.md`

约束抽象：

- `restore` 只能提供候选，不等于 `decide`。
- 历史绑定、显式文件名、路径提示、记忆命中，最多只能进入“候选层”。
- 当前 turn 的正式决策必须在统一仲裁点完成，不能被某个早期短路规则直接接管。

当前坏点：

- `backend/query/followup_resolver.py`
- `backend/query/binding_resolver.py`
- `backend/query/tool_input_resolver.py`

现象对照：

1. `followup_resolver` 对显式文件名做了过强短路。
2. 原始 `tool_input.path`、显式文件名、binding resolver 输出、结构化 binding 输出之间没有清晰 authority 排序。
3. 结果是“当前回合应沿用的 source”会被“当前句子里偶然出现的文件名”打断。

结论：

这不是解析命中率问题，而是 `restore != decide` 的边界没有被制度化。候选信号被过早升级成了当前真相。

### 7.3 任务局部状态与主线程状态边界失守

对应设计原则：

- `docs/设计原则/12-Agent-系统.md`
- `docs/设计原则/14-任务系统.md`
- `docs/设计原则/25-架构模式总结.md`

约束抽象：

- 默认隔离，显式共享。
- 子任务的局部 source、局部约束、局部失败，不应自动升级为主线程全局状态。
- 主线程只应吸收经过筛选的正式结果，不应直接继承任务局部执行痕迹。

当前坏点：

- `backend/query/runtime.py`
- `backend/query/planner.py`
- `backend/tasks/coordinator.py`
- `backend/query/answer_assembler.py`

现象对照：

1. 子任务没有稳定继承父 binding，说明“该共享的没共享”。
2. 子任务的局部风格约束又会上卷到 compound 主结果，说明“不该共享的反而共享了”。
3. 一个天气子任务的 `one_sentence` 可以把 PDF 子任务也裁成单句。

结论：

当前任务系统在 source、style、result 这三类状态上都没有形成“默认隔离 + 显式穿透”的边界治理，导致局部状态既会丢失，也会串染。

### 7.4 执行工件与最终答案边界失守

对应设计原则：

- `docs/设计原则/05-对话循环.md`
- `docs/设计原则/25-架构模式总结.md`

约束抽象：

- `execute` 和 `present` 必须分层。
- 工具执行状态、fallback 文案、结果占位符，不得直接冒充最终用户答案。
- 最终展示层应优先消费 canonical summary、稳定 result ref 或明确失败态，而不是工具中间态。

当前坏点：

- `backend/query/output_classifier.py`
- `backend/query/answer_assembler.py`
- `backend/query/runtime.py`

现象对照：

1. `pdf_canonical_missing_summary` 会被自然语言化成“已读取相关页面，但没有稳定摘要”。
2. `tool_missing_summary` 会被包装成“工具 xxx 已执行，但当前结果尚未形成答案”。
3. `AnswerAssembler` 在拿不到 summary 时，会继续把 placeholder 作为可展示正文。

结论：

这条边界一旦失守，系统就会把“执行过”伪装成“回答过”。用户看到的是一条完整中文句子，但语义上其实仍处于失败中间态。

### 7.5 局部风格与全局展示边界失守

对应设计原则：

- `docs/设计原则/14-任务系统.md`
- `docs/设计原则/25-架构模式总结.md`

约束抽象：

- 风格约束应当和任务结果一起局部生效。
- “补一句”“一句话”“简短”这类约束，如果来自子任务，只能作用于该子任务自身。
- compound 总装层不能把最后一个子任务的风格当成全局默认风格。

当前坏点：

- `backend/query/runtime.py`
- `backend/query/answer_assembler.py`

现象对照：

1. compound main context 会继承最后活跃子任务的 `response_style`。
2. `AnswerAssembler` 再把它作为 `default_style` 应用到所有 segment。
3. 于是会出现“PDF 本来有长摘要，最终回答只剩第一句”的压缩错觉。

结论：

风格本应是展示层局部约束，但现在被错误提升成了全局装配约束。这类问题不是语义理解失败，而是展示边界设置错误。

### 7.6 边界失守的主次顺序

按目前证据，优先级顺序应当明确为：

1. `源绑定边界` 是主根因。
2. `恢复候选与当前决策边界` 是第一传播层。
3. `任务局部与主线程边界` 是第二传播层。
4. `执行工件与最终答案边界` 是用户可见错误的直接出口。
5. `局部风格与全局展示边界` 是复合任务里最明显的串染放大器。

因此，这批问题不应再按“PDF 规则修一条、fallback 文案修一条”的方式拆着看，而应该按边界治理来修。

## 8. 测试断言缺口

本次长场景还有一个必须显式记录的问题：测试集对“回答是否真正答到点上”的约束明显偏弱。

重点文件：

- `backend/tests/system_eval/long_scenarios.py`

当前大量 turn 的校验条件只有：

- `response.nonempty`
- `response.not_contains_any=我来检索|search_knowledge|<tool_call|</think>`

这意味着下面这些错误，测试仍可能判通过：

1. 回答是工具状态句，但不为空。
2. 回答是动作句，例如“已读取 P3”，但没有真正回答问题。
3. 回答与问题不匹配，但没有暴露内部协议。
4. 同一事实前后矛盾，只要当下这一轮回答是非空的。

所以本次只报出 `1` 个正式失败，并不等于只有 `1` 个严重语义问题。

这不是修复主链时的旁枝问题，而是后续回归门禁必须补上的一层。

## 9. 当前优先级建议

### 第一优先级

1. 禁止工具 fallback 文案直接作为最终用户答案。
2. 限制 binding follow-up 的触发范围，给“全局总结 / 多源汇总”留出 planner 主通道。
3. 区分“章节定位失败”“源不存在”“工具失败”“无答案”四种失败态。

### 第二优先级

1. 关闭 PDF degraded result 的宽松 finalization。
2. 为 PDF 页级 / 章节级回答增加事实锁和矛盾检测。
3. 子任务组装前新增“是否真正回答问题”的门禁。

### 第三优先级

1. 统一本地 docs / PDF / dataset / knowledge 四类源的解析链。
2. 对长场景测试加入“不得返回工具状态句”的更强校验。

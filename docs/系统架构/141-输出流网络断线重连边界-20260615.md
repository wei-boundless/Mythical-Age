# 输出流网络断线重连边界

日期：2026-06-15

## 结论

自动重连保护只属于已建立的 chat run 输出流传输断线。

前端已经拿到 `stream_run_id` 后，SSE 读取链路如果发生网络/传输层中断，才允许用同一个 `stream_run_id + event_offset` 续接 `/chat/runs/{stream_run_id}/events?after_offset=...`。

## 允许重连的信号

| 信号 | 判定 | 行为 |
| --- | --- | --- |
| `fetch` 抛出 `TypeError` | 浏览器网络请求失败 | 保持当前 run，按 offset 重连 |
| `NetworkError` / `Failed to fetch` / `Load failed` / `The network connection was lost` | 浏览器或运行环境报告网络断线 | 保持当前 run，按 offset 重连 |
| SSE reader 正常结束但没有 `turn_completed` | 输出流无终态关闭，视为传输断线 | 保持当前 run，按 offset 重连 |

## 不允许重连的信号

| 信号 | 原因 | 行为 |
| --- | --- | --- |
| `AbortError` / `TimeoutError` 且来自前端 abort signal | 主动取消，不是网络保护 | 抛给 runtime，由 runtime 按停止/错误处理 |
| HTTP 非 2xx | 后端明确拒绝或失败，不是网络断线 | 触发 `stream_reconnect_failed` 后失败 |
| SSE payload 解析失败 | 协议错误，不是网络断线 | 触发 `stream_reconnect_failed` 后失败 |
| 后端 `turn_completed` | run 已有终态 | 清理 cursor，结束输出流 |

## 当前修复

- 文件：`frontend/src/lib/api.ts`
- 新增 `isReconnectableChatStreamTransportError`，只识别网络/传输错误。
- HTTP 错误和缺失 readable body 改为 `ChatStreamProtocolError`，不会进入重连循环。
- 非网络错误会先发送 `stream_reconnect_failed`，再交给外层 runtime 显示失败。

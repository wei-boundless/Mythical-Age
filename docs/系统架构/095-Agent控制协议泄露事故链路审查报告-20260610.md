# Agent 控制协议泄露事故链路审查报告

日期：2026-06-10

## 0. 结论

本次问题不是“前端显示了 JSON”，也不是“JSON 语法错误”。泄露对象：

```json
{"action":"continue_active_work","relation_to_current_work":"current_work","response":"用户要求继续推进当前代码审查任务，恢复执行逐模块深入审查并生成完整全新报告。"}
```

是一个当前工作控制意图。它不应成为 assistant 正文。事故性质是控制协议边界泄露：模型返回了控制形状的裸 JSON，后端旧解析器没有把它识别为 `active_work_control`，也没有把它拒绝为内部协议，最终 fallback 把 `response.content` 当作 `conversation` 正文提交，之后 SSE、history、store、ChatMessage 正常渲染了这段污染内容。

成熟链路必须满足：

```text
用户请求事实
-> runtime 装配与允许动作
-> 模型语义动作
-> 协议解析/正规化
-> action admission
-> active_work_control 执行
-> observation/status 投影
-> 最终正文边界
-> commit/session
-> SSE public projection
-> frontend store
-> UI render
```

任何一层发现控制协议形状，都只能进入控制/状态/观察通道，不能降级成 assistant body。

## 1. 事故链路复盘

### 1.1 Prompt / runtime contract

`backend/harness/runtime/compiler.py` 会在存在 `active_work_context` 且控制能力允许时加入 `active_work_control`：

- `compiler.py:2095` 将 `active_work_control` 加入 allowed actions。
- `compiler.py:2213` 定义 `active_work_control.enabled`。
- `compiler.py:2216` 定义 payload schema，要求 `action`、`relation_to_current_work` 等字段。
- `compiler.py:2194` 明确 control actions 不作为 native tool 暴露。
- `compiler.py:3789`、`compiler.py:3800`、`compiler.py:3801` 告诉模型 active_work_control 是控制裁决，不是普通 native 工具。

这层的方向是对的：模型可以做语义裁决，但它提交的是动作请求，不是正文。

隐患：如果 prompt 只强调“使用 active_work_control 动作”，但没有强约束 action envelope，模型可能输出裸 JSON。裸 JSON 是可预期失误，不应依赖 UI 补救。

### 1.2 Model response protocol parser

`backend/runtime/model_gateway/model_response_protocol.py` 做最基础的 provider response 归一：

- `model_response_protocol_from_response` 在 `model_response_protocol.py:35` 读取 content/native tool calls。
- `parse_json_object_with_diagnostics` 在 `model_response_protocol.py:73` 只负责 JSON object 解析。
- `model_response_protocol.py:50` 仅在 `require_json_action=True` 且没有 JSON payload 时报 `json_action_required`。

这层只知道“是不是 JSON object”，不拥有业务语义。它不应决定 active_work_control，但它必须把解析诊断交给下一层。

### 1.3 Single-agent action parser

关键函数：`backend/harness/loop/single_agent_turn.py:1960` 的 `_single_agent_action_request_from_response`。

旧行为的事故点：

1. `model_response_protocol_from_response` 成功把裸 JSON 解析为 `json_payload`。
2. `_is_model_action_json_payload` 在 `single_agent_turn.py:2148` 只承认：
   - `authority == harness.loop.model_action_request`
   - 存在 `authority`
   - 存在 `action_type`
   - 或 `authority.startswith("harness.loop.") and "action" in payload`
3. 截图 JSON 只有 `action/relation_to_current_work/response`，没有 `authority/action_type`，所以不被认为是模型动作。
4. 当没有 native tool calls 且本轮不强制 JSON action 时，函数返回“没有 action”，没有 error。

这就是第一处 P0：控制形状被旧 parser 漏判后，没有 fail-closed。

当前工作区已有修复草案：

- `_normalize_single_agent_json_payload` 位于 `single_agent_turn.py:2161`。
- `_is_bare_active_work_control_payload` 位于 `single_agent_turn.py:2200`。
- 它把截图这种裸 active-work 控制对象正规化成：

```json
{
  "authority": "harness.loop.model_action_request",
  "action_type": "active_work_control",
  "active_work_control": {...}
}
```

正确方向：如果本轮允许 `active_work_control`，进入控制执行；如果不允许，成为 invalid JSON action，而不是 conversation。

仍需补强：`_is_bare_active_work_control_payload` 目前要求除 `action` 外还出现 `relation_to_current_work/response/appended_instruction/...` 之一。只有 `{"action":"continue_active_work"}` 这种控制形状仍可能不进入 action parser，后续只能依赖 output boundary 拦截。成熟方案应把 active-work action 名称本身视为控制协议形状；缺字段应成为 invalid action，不应回落正文。

### 1.4 Action admission

`backend/harness/loop/admission.py` 是动作准入层：

- `admit_model_action` 检查 `packet_allowed_action_types`。
- 对 `active_work_control`，如果 runtime profile 禁止，会返回 deny。

该层职责正确：允许/拒绝动作，不改变用户目标，不把被拒动作转换成普通回答。

需要防守的边界：上游必须把控制请求送到 admission。否则 admission 没机会拦截。

### 1.5 Active-work execution

执行入口在 `backend/harness/entrypoint/runtime_facade.py`：

- `runtime_facade.py:722` 调用 `active_work_turn_decision_from_payload`。
- `backend/harness/loop/active_work.py:147` 将 payload 转成 `ActiveWorkTurnDecision`。
- `active_work.py:189` 校验当前工作关系：控制当前工作必须 `relation_to_current_work == current_work`。
- `active_work.py:245` 生成 denial observation。
- `runtime_facade.py:1574` 的 `_apply_active_work_turn_decision` 分发 continue/pause/stop/append/answer。

这层不是泄露源。它有 relation 校验、action 校验、pause/stop/continue 分支。泄露发生在控制请求没有到达这一层之前。

### 1.6 Active-work observation / follow-up

`single_agent_turn.py` 在 active_work_control 分支中：

- 记录 `model_action_admission`。
- 调用 `apply_active_work_control`。
- 构造 `active_work_control_observed` event。
- yield `runtime_status`，phase 是 `active_work_control`。
- 把 observation 作为下一次模型观察，继续 follow-up。

这是正确链路：控制动作执行结果是 observation/status，不是 assistant body。

边界要求：

- 执行动作前不能告诉用户“已经完成控制”。
- 执行返回后，只能基于 observation 回复或状态投影。
- active_work_control 本身不能作为 final answer source。

### 1.7 Final fallback / commit

第二个 P0 在 final fallback：

- `single_agent_turn.py:1368` 已经把 final 阶段出现的 `active_work_control` 判为不可达协议错误。
- 但是当 action parser 返回“没有 action”时，`single_agent_turn.py:1404` 之后取 `response.content`。
- `single_agent_turn.py:1409` 以 `answer_source="harness.single_agent_turn"`、`answer_channel="conversation"` 调用 `_commit_final_message`。
- `single_agent_turn.py:1432` 以 `terminal_reason="assistant_message"` 发 done。

这条 fallback 只能用于自然语言正文。它不能接收任何控制形状、工具协议形状、JSON action 形状。

当前修复草案依赖 `_commit_final_message` 的 output boundary 兜底，但成熟标准应再加一条 parser/fallback 规则：解析到 JSON object 且包含控制/action 字段，即使 envelope 不完整，也必须成为 protocol error，不允许进入 natural-language fallback。

### 1.8 Output boundary

`backend/runtime/output_boundary/boundary.py` 是最终正文边界：

- `contains_internal_protocol` 在 `boundary.py:156`。
- 当前修复草案加入 `_looks_like_active_work_control_protocol`，入口在 `boundary.py:161`。
- `_looks_like_active_work_control_protocol` 位于 `boundary.py:172`。
- `canonical_output_decision_for_final_text` 位于 `boundary.py:435`。
- `boundary.py:452` 检测 `contains_internal_protocol(raw_text)`。
- `boundary.py:465` 对纯协议内容走 `runtime.output_boundary.protocol_fail_closed`。

这是最后一道后端持久化闸门。它必须识别：

- 完整 envelope：`{"action_type":"active_work_control",...}`
- 裸控制对象：`{"action":"continue_active_work","relation_to_current_work":"current_work",...}`
- fenced JSON
- 数组或嵌套对象中的 active_work_control
- malformed 但含控制 action 和控制字段的片段

当前草案已经覆盖截图形态。仍需补测试矩阵，不能只测一个字符串。

### 1.9 Session commit / history

`runtime_facade.py:2159` 的 `_apply_assistant_message_commit` 写入 session messages，保留 `answer_channel/answer_source/answer_canonical_state/answer_persist_policy/answer_leak_flags` 等 metadata。

`backend/sessions/__init__.py:855` 的 `_public_message` 只过滤：

- 非 user/assistant role
- assistant tool_calls
- 空 content 且无 image

它不识别 active_work_control JSON。也就是说 session manager 不是语义边界。它必须依赖 commit 前的 canonical output boundary。任何绕过 `_commit_final_message` 的直接 assistant append 都是风险。

## 3. 分级问题

### P0-1 裸 active-work 控制 JSON 被 parser 漏判

位置：`backend/harness/loop/single_agent_turn.py:1960`、`:2148`

失败场景：模型返回截图 JSON。旧 `_is_model_action_json_payload` 不承认它是 action，导致 action parser 返回 no-action。

影响：控制请求没有进入 admission/executor，直接滑向 final fallback。

修复标准：任何包含 active-work action 名称或 `active_work_control` 字段的 JSON object，都必须进入 action parser；缺字段时 invalid，不能 no-action。

### P0-2 final fallback 把未分类控制形状当 conversation

位置：`backend/harness/loop/single_agent_turn.py:1404`、`:1409`

失败场景：解析层返回 no-action，final fallback 把 raw content 作为 `answer_channel=conversation` 提交。

影响：控制系统命令进入 assistant message commit。

修复标准：final fallback 前必须有“自然语言正文”判定；JSON object、控制 action、工具协议、内部 marker 均不得进入 `conversation` fallback。

### P0-3 output boundary 必须识别 active-work 控制协议

位置：`backend/runtime/output_boundary/boundary.py:156`、`:172`、`:435`

失败场景：即使 parser 漏判，最终边界仍应把截图 JSON 标为 `internal_protocol_final_text` 并 `do_not_persist`。

当前草案方向正确，但需要补矩阵测试。

### P0-4 live stream store 仍缺内容级 fail-closed

位置：`frontend/src/lib/store/events.ts:249`、`:1374`、`:1387`、`:1400`

失败场景：`assistant_text_delta/final/repair` 事件携带控制 JSON，但 metadata 缺失或误标 stable。`shouldDisplayAssistantContent` 默认可见，store 写入 `message.content`。

修复标准：stream store 必须检查 `content` / `replacement_content` 本身。命中控制协议时丢弃该帧，并避免 repair/final 覆盖成污染正文。

### P1-1 session manager 不是协议边界

位置：`backend/sessions/__init__.py:855`

失败场景：任何直接调用 `append_messages` 写 assistant content 的路径，如果绕过 canonical boundary，session 会保存污染内容。

修复标准：所有 assistant commit 入口必须统一经过 output boundary。对直接 route 需要审查，不能让 session manager 承担语义判断。

### P1-2 系统状态文案与 assistant body 边界需要统一

位置：

- `frontend/src/lib/store/events.ts:783`、`:791`、`:1440`
- `frontend/src/lib/runtimeVisibilityProjection.ts:560`、`:696`、`:1092`
- `frontend/src/components/chat/PublicTimelineActivity.tsx:297`、`:327`
- `backend/harness/loop/task_lifecycle.py:791`

“正在思考 / 开始处理 / 处理完成 / 完成”这类文案如果作为 status surface 出现，可以是 UI 投影；如果作为 assistant body 出现，就会混淆“模型正文”和“系统动作状态”。

修复标准：

- assistant body 只来自模型稳定正文或明确 model body projection。
- runtime/status/tool/control surface 的 copy 统一由 projection layer 管理。
- task handoff receipt 不能作为长期 assistant 正文，除非 canonical boundary 判定为稳定用户可见回答。

### P1-3 repair/admission repair 需要防止“被拒控制动作变普通回答”

位置：`backend/harness/loop/single_agent_turn.py:1720`、admission repair 相关路径。

风险：协议修复和 admission 修复会再次调用模型。修复提示必须要求返回合法 action，不允许把原控制请求解释成普通自然语言来绕过边界。

修复标准：修复模型返回的任何 active-work 控制形状继续按 action 处理；返回 respond 时必须通过 final text boundary 且不得包含控制字段。

## 4. 正确时序设计

### 4.1 开局显示

当本轮可能产生 action JSON 时：

1. UI 可显示 status surface，例如“正在思考”。
2. 不应生成 assistant body 占位正文。
3. 如果 `requires_json_action_protocol=True` 或当前阶段 `allow_assistant_text_delta=False`，后端不应发送正文 delta。
4. 如果允许自然语言流式输出，必须先经过 stream normalizer safety gate。

### 4.2 工具调用显示

工具调用显示在 tool window / public timeline：

- action request/admission 是内部执行轨迹。
- tool name、目标、状态可以投影。
- tool args、内部 authority、packet_ref、raw protocol 不应出现在正文。

### 4.3 active_work_control 显示

正确链路：

```text
模型提交 active_work_control
-> parser 转成 ModelActionRequest
-> admission allow/deny
-> runtime_facade 执行控制
-> active_work_control_observed
-> runtime_status phase=active_work_control
-> follow-up observation 给模型
-> 必要时模型再给稳定回答
```

可见面：

- 控制状态显示在 status/control surface。
- 模型 `response` 字段是 observation 语义说明，不是最终正文。
- `action/relation_to_current_work/continuation_strategy` 不显示给用户。

### 4.4 pause / stop / steer

暂停必须分两层：

- 控制请求接收：立即记录 `pause_requested` 或对应 runtime status，让用户知道系统收到控制。
- 执行确认：运行中工具/任务只能在安全点停止，确认后再显示 `paused/stopped`。

追加 steer：

- 对 running active task，`append_instruction_to_active_work` 应写 durable steer，并绑定 active turn。
- 对 paused/waiting task，先写 steer，再 resume/schedule。
- steer 只有被任务执行器读取并处理后，才能标记 consumed。

### 4.5 收口

收口只能有两类：

- 稳定 assistant body：来自 `respond.final_answer` 或明确 model body projection，且通过 canonical output boundary。
- 状态/控制收口：进入 public projection/status，不写长期正文。

`done.content` 不应作为 body fallback。前端已有 `publicProjectionReducer.test.ts` 覆盖“不使用 done content 作为 assistant body fallback”，这条原则要继续保持。

## 5. Prompt 约束建议

给主 agent 的 prompt 不应写成“这是 runtime 节点”。应写成 agent 能执行的职责和输出协议：

```text
你负责判断用户本轮是在请求普通回答、开始持续任务、调用可见工具，还是控制当前正在进行的工作。

当用户明确指向当前工作并要求继续、暂停、停止、补充要求、询问状态或回答后继续时，你必须提交 active_work_control 动作。
active_work_control 是系统控制动作，不是给用户看的正文。不要把 action、relation_to_current_work、continuation_strategy、turn_response_policy、answer_obligation 或任何控制字段写入自然语言回复。

如果本轮允许 JSON action protocol，你必须输出一个完整 action envelope：
{
  "authority": "harness.loop.model_action_request",
  "action_type": "active_work_control",
  "active_work_control": {
    "action": "continue_active_work",
    "relation_to_current_work": "current_work",
    "response": "简短说明你识别到的控制意图"
  }
}

如果你无法确认用户是在控制当前工作，不要猜测执行控制动作；使用 ask_user 或 respond 说明需要澄清。
```

## 6. 必补测试

### Backend

1. Parser：
   - 裸截图 JSON 被 normalize 为 `active_work_control`。
   - 当 packet 不允许 active_work_control，返回 invalid JSON action，不是 conversation。
   - `{"action":"continue_active_work"}` 缺 relation 时也是 invalid control action，不是 conversation。
   - fenced JSON、带 alias `intent`、嵌套 `active_work_control` 都覆盖。

2. End-to-end single-agent：
   - fake model 返回截图 JSON。
   - 断言 `apply_active_work_control` 被调用。
   - 断言 session messages 不包含 `continue_active_work/relation_to_current_work`。
   - 断言 SSE 包含 `runtime_status phase=active_work_control`。

3. Output boundary：
   - 完整 envelope、裸 object、array、fenced JSON、malformed action+relation fragment 全部 `internal_protocol_final_text`。
   - 纯协议内容 `do_not_persist`。
   - 混合“可见结论 + 协议残片”只能 debug-only，不可 canonical。

4. Commit：
   - `_commit_final_message` 对控制 JSON 不写 canonical assistant body。
   - 直接 task executor final_answer 若含控制协议也被 boundary 拦截。

### Frontend

1. `events.ts`：
   - `assistant_text_delta.content` 是截图 JSON，不写入 message.content。
   - `assistant_text_final.content` 是截图 JSON，不覆盖已有正文。
   - `assistant_stream_repair.replacement_content` 是截图 JSON，不覆盖已有正文。
   - metadata 缺失、metadata 错标 stable 都要 fail-closed。

2. `utils.ts`：
   - polluted history message 被 `toUiMessages` 丢弃。

3. `ChatMessage.tsx`：
   - assistant content 等于截图 JSON 时不渲染正文。
   - 不显示“复制回复”按钮。

4. `PublicTimelineActivity`：
   - timeline text 中出现 active_work_control JSON 时被清理。

## 10. 不可退让原则

1. 控制协议永远不是 assistant 正文。
2. JSON 不是错；错的是把控制 JSON 降级成 conversation。
3. parser 发现控制形状必须 normalize 或 invalid，不能 no-action。
4. admission 只能 allow/deny，不能替模型改目标。
5. active_work_control 执行结果是 observation/status，不是 body。
6. output boundary 是持久化前最后防线，必须 fail-closed。
7. SSE 是传输，不是语义裁决层。
8. frontend store 必须按内容再防一次，不能只信 metadata。
9. history/render 是最后防线，不能成为主修复。
10. “开始处理/处理完成/正在思考”属于状态投影，不属于模型正文。

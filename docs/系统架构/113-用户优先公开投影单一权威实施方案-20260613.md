# 用户优先公开投影单一权威实施方案

日期：2026-06-13

状态：待用户确认后实施。

本文是最终实施依据。此前 `112-公开投影单一权威旧链路清理实施书-20260613.md` 已废止并删除，因为它把“证据可追溯”错误地写成了“收口后工具继续留在主聊天区”。本方案以用户体验、严格时序、单一投影权威和显示稳定为硬约束。

2026-06-13 源码核验修订：本方案删除此前自造的独立 `intent` 协议层。Codex 的主链路是 `ResponseItem::FunctionCall / CustomToolCall -> ToolCall -> ToolInvocation -> FunctionCallOutput`，Claude Code 的主链路是 `assistant tool_use block -> canUseTool / ToolUseContext -> tool_result`。因此本方案统一采用 `ModelOutputItem -> ToolCallRequest -> PermissionDecision -> ToolExecution -> ToolResult -> ProjectionFrame`，不再使用 `AgentSemanticDecision`、`PublicActionIntent`、`intent_id` 作为控制系统概念。

## 1. 最终原则

优秀 agent 的主聊天投影必须满足：

```text
运行中透明
收口后清爽
异常不隐藏
证据可追溯
顺序严格稳定
前端不裁决语义
```

用户默认看到的是对话和结果，不是运行日志。工具和内部运行事实必须可追溯，但不能在任务收口后默认占据主聊天区。

## 2. 当前链路审查结论

### 2.1 后端链路现状

| 文件 | 当前行为 | 问题 | 目标处理 |
| --- | --- | --- | --- |
| `backend/api/chat.py` | `_project_public_stream_event()` 从 raw runtime event 映射 public event；`model_action_admission` 变 `tool_item_started`，`tool_observation` 变 `tool_item_completed` | chat API 仍拥有展示语义决策权 | 改写为 canonical normalizer；不得生成 display text；旧展示裁决删除 |
| `backend/runtime/output_stream/public_contract.py` | `assistant_text_delta/final`、`tool_item_started/completed`、`turn_completed` 被标为 lossless，`event_requires_public_projection()` 对它们返回 false | 最关键的正文和工具事件反而绕过 projection frame | 所有用户可见或视图迁移事件都必须生成 projection frame |

| `backend/harness/runtime/session_timeline.py` | 从 task projection 重新生成 `public_timeline` 并放入 runtime attachment | hydrate 层复活旧投影，和 live projection 竞争 | session timeline 只返回 committed transcript、commit state、trace snapshot |
| `backend/harness/runtime/run_monitor/projector.py` | monitor 从 task projection 重新生成 `public_timeline` | monitor 仍在制造主聊天语义 | monitor 只观察状态、commit state、trace refs，不生成主聊天 projection |

| 文件 | 当前行为 | 问题 | 目标处理 |

## 3. 用户默认视图合同

主聊天区只允许显示下列内容：

| 阶段 | 用户默认看到 | 用户默认不看到 |
| --- | --- | --- |
| 输入后、assistant 正文尚未输出 | 无语义 chrome，或来自模型 `tool_use/function_call/custom_tool_call` 的 `ToolCallRequest` 投影 | 系统伪造的“我正在处理”正文、裸工具卡片 |
| 模型正文 streaming | assistant 正文增量 | raw event、debug trace、内部 prompt/context |
| 工具运行中 | 一个当前动作卡片：工具名的用户语义动作、目标、运行状态 | 工具完整参数、内部 tool id、原始 observation |
| 工具成功完成 | 当前动作卡片被后端显式 `retire_to_trace` 移出主视图；trace 中保留完整证据 | 成功工具列表长期堆在主聊天 |
| 工具失败/等待/权限 | pinned 状态卡片继续留在主视图 | 把失败藏入 trace |
| 产物/关键验证 | 产物引用、验证结论留在主视图 | 产生这些结论的普通读取/搜索工具日志 |
| final text 出现 | 最终正文 | 因 final text 由前端清空工具或 timeline |
| commit ack 后收口 | 干净最终回答、必要 pinned 状态、产物、验证结果 | 普通工具卡片、普通过程 timeline |
| commit failed | 明确输出提交失败状态，保留 live body，不刷新旧 history | 假 completed、旧 history 覆盖 live 内容 |

硬规则：

1. 普通工具不是 final transcript。
2. 普通工具执行事件本身不能创建主视图语义；主视图 `current_action` 只能来自模型产出的 `ToolCallRequest`，也就是 Claude Code 的 `tool_use block` 或 Codex 的 `function_call/custom_tool_call` 等价物。工具运行中只能更新该 `tool_call_id` 的执行状态或进入 trace。
3. 失败、等待用户、权限确认、阻塞、commit failed、关键产物、关键验证必须留在主视图，直到有明确后续 frame 解除。
4. 前端不允许通过中文文案、工具名、terminal 状态、final answer 是否存在来决定上述规则。

## 4. 固定投影流程

唯一流程如下：

```text
Runtime Event Log
-> Canonical Public Event
-> Public Projection Authority
-> Projection Frame Stream
-> Frontend Projection Ledger
-> MessagePublicProjection
-> Pure Chat Render
```

### 4.1 Runtime Event Log

只记录事实，不决定用户视图。

输入事实包括：

```text
assistant_text_delta
assistant_text_final
assistant_stream_repair
model_output_item
tool_call_requested
tool_permission_decided
tool_execution_started
tool_result_observed
session_output_commit_checked
session_output_commit_ack
session_output_commit_failed
session_output_commit_skipped
task_run_lifecycle_finished
turn_completed
runtime_error
```

### 4.2 Canonical Public Event

只归一，不裁决最终 UI。

固定事件集合：

```text
assistant_text_delta
assistant_text_final
assistant_stream_repair
tool_call_requested
tool_permission_decided
tool_item_started
tool_item_completed
projection_item_pinned
projection_item_retired
projection_scope_retired
session_output_commit_checked
session_output_commit_ack
session_output_commit_failed
session_output_commit_skipped
turn_completed
```

禁止新增 `assistant_text`、`answer_candidate`、`done.content` 等旧正文事件作为主链路输入。

### 4.3 控制系统边界

投影链路必须服从控制系统边界。这里的原则不是“错误事件来了以后前端过滤”，而是错误状态不得进入公开执行生命周期。

固定控制链：

```text
UserInput
-> ModelTurnDecision
-> ModelOutputItem
-> ToolCallRequest
-> PermissionDecision
-> ToolExecution
-> ToolResult
-> PublicProjectionFrame
```

各层权威：

| 层 | 权威 | 允许做什么 | 禁止做什么 |
| --- | --- | --- | --- |
| `UserInput` | 用户 | 提供当前请求事实 | 生成工具或投影 |
| `ModelTurnDecision` | 模型 | 决定下一步是回答、提问、停止还是发起工具调用 | 执行工具 |
| `ModelOutputItem` | 模型/API stream | 产生 assistant text、reasoning、`tool_use/function_call/custom_tool_call` | 伪造工具结果、绕过权限 |
| `ToolCallRequest` | runtime normalizer | 从模型工具调用项生成 `tool_call_id`、工具名、参数、用户可读动作摘要 | 从工具 started 事件反推动作语义 |
| `PermissionDecision` | runtime policy / permission | 对 `tool_call_id` 做 allow/deny/ask/blocked 判定，生成 `permission_decision_id` | 改写模型工具请求或生成 assistant body |
| `ToolExecution` | tool runtime | 执行已允许的 `ToolCallRequest`，发 `tool_item_started` | 在缺少 `tool_call_id` 或允许态权限决策时进入公开工具生命周期 |
| `ToolResult` | tool runtime | 记录结果、失败、产物、trace refs | 写 assistant body |
| `PublicProjectionFrame` | public projection authority | 按 tool call request、permission decision、tool result 生成 frame | 修补非法时序或让前端兜底 |

硬性时序约束：

```text
tool_call_requested.event_offset
< tool_permission_decided.event_offset
< tool_item_started.event_offset
< tool_item_completed.event_offset
```

硬性引用约束：

```text
tool_permission_decided.tool_call_id == tool_call_requested.tool_call_id
tool_permission_decided.decision in {"allow", "auto_allow"}
tool_item_started.tool_call_id == tool_call_requested.tool_call_id
tool_item_started.permission_decision_id == tool_permission_decided.permission_decision_id
tool_item_completed.tool_call_id == tool_item_started.tool_call_id
```

禁止状态：

```text
UserInput -> tool_item_started
UserInput -> system generated current_action -> tool_item_started
tool_item_started without tool_call_id
tool_item_started without allow permission decision
tool_item_started.event_offset <= tool_call_requested.event_offset
```

处理规则：

1. 缺少模型 `ToolCallRequest` 时，不能进入 `PermissionDecision`。
2. 缺少允许态 `PermissionDecision` 时，不能进入 `ToolExecution`。
3. 如果历史 bug 或异常导致 tool event 已经出现但缺少合法上游 `tool_call_id` 或允许态权限决策，该事件只能写 diagnostics/trace protocol error，不能生成主视图 frame。
4. 前端不负责挡住非法工具；前端只消费合法 `PublicProjectionFrame`。

### 4.4 Public Projection Authority

这是唯一有权决定用户能看到什么的层。

它为每个 canonical event 生成 `PublicProjectionFrame`：

```ts
type PublicProjectionFrame = {
  authority: "harness.public_projection";
  contract_revision: "20260613-user-first";
  frame_id: string;
  source_event_id: string;
  sequence: number;
  event_offset: number;
  created_at: number;

  anchor: {
    session_id: string;
    turn_id?: string;
    message_id?: string;
    task_run_id?: string;
    turn_run_id?: string;
  };
  source_item_id?: string;
  tool_call_id?: string;
  permission_decision_id?: string;
  parent_tool_call_id?: string;

  op:
    | "body_append"
    | "body_finalize"
    | "item_upsert"
    | "item_retire"
    | "scope_retire"
    | "commit_ack"
    | "commit_failed"
    | "turn_terminal";

  slot:
    | "body"
    | "current_action"
    | "pinned"
    | "final_result"
    | "status"
    | "trace";

  source_authority: "model" | "tool" | "runtime" | "system";

  main_visibility:
    | "visible_live"
    | "visible_final"
    | "pinned"
    | "trace_only"
    | "hidden";

  retention:
    | "transient"
    | "final"
    | "pinned_until_resolved"
    | "trace";

  pin_reason?:
    | "failed"
    | "blocked"
    | "waiting_user"
    | "waiting_approval"
    | "permission"
    | "commit_failed"
    | "artifact"
    | "verification"
    | "user_action_required";

  item_id?: string;
  text?: string;
  detail?: string;
  state?: "running" | "done" | "failed" | "blocked" | "waiting" | "stopped";
  trace_refs?: string[];
  artifact_refs?: Array<Record<string, unknown>>;
  commit?: {
    state: "checked" | "committed" | "failed" | "skipped";
    commit_event_offset?: number;
    message_id?: string;
    content_sha256?: string;
  };
};
```

字段权威：

| 字段 | 规则 |
| --- | --- |
| `event_offset` | 前端排序唯一依据 |
| `op` | 前端更新动作唯一依据 |
| `slot` | 前端渲染位置唯一依据 |
| `main_visibility` | 是否进入主聊天唯一依据 |
| `retention` | 收口后去向唯一依据 |
| `pin_reason` | 收口后留在主视图的唯一理由 |
| `commit.state` | hydrate 唯一依据 |
| `tool_call_id` | 工具公开生命周期绑定的模型工具调用请求 |
| `permission_decision_id` | 工具公开生命周期绑定的权限决策 |

### 4.5 Projection Frame Stream

SSE 和 replay 推送 projection frame。

规则：

1. 所有用户可见事件必须携带 `public_projection_frame`。
2. lossless event 不等于绕过 projection。`assistant_text_delta/final` 和 `tool_item_started/completed` 必须同时拥有 frame。
3. replay 重放 frame，不重放 raw event 后让前端再解释。
4. frame 缺少 `event_offset`、`op`、`slot`、`main_visibility` 时，前端必须拒绝进入主视图，只能进入 diagnostics。
5. `tool_item_started/completed` frame 缺少合法 `tool_call_id` / `permission_decision_id` 时，不允许进入主视图。

### 4.6 Frontend Projection Ledger

前端只维护账本，不做语义判断。

账本结构：

```ts
type ProjectionLedger = {
  body: {
    text: string;
    stream_state: "streaming" | "finalized" | "committed";
    source_offsets: number[];
  };
  currentAction?: ProjectionItem;
  pinned: ProjectionItem[];
  finalResults: ProjectionItem[];
  status: ProjectionItem[];
  trace: ProjectionItem[];
  commit: {
    state: "none" | "checked" | "committed" | "failed" | "skipped";
    key?: string;
  };
};
```

reducer 固定动作：

| frame op | reducer 行为 |
| --- | --- |
| `body_append` | 按 offset 追加到 body |
| `body_finalize` | 修正/确认 body，标记 finalized |
| `item_upsert` + `slot=current_action` | 只接受由模型 `ToolCallRequest` 产生、携带 `tool_call_id` 的动作投影；替换当前动作；旧 current action 若未 pinned，进入 trace |
| `item_upsert` + `slot=pinned` | upsert 到 pinned |
| `item_upsert` + `slot=final_result` | upsert 到 finalResults |
| `item_upsert` + `slot=status` | upsert 到 status |
| `item_upsert` + `slot=trace` | upsert 到 trace |
| `item_retire` | 指定 item 从 current/pinned/final/status 移入 trace |
| `scope_retire` | 将 `retention=transient` 的 live items 移入 trace |
| `commit_ack` | 标记 committed，触发一次 hydrate |
| `commit_failed` | 标记 failed，保留 live body，显示 pinned commit failure |
| `turn_terminal` | 标记 stream terminal，不 hydrate，不清屏 |

禁止 reducer 做的事：

1. 不调用 `isNaturalLanguageStageFeedback`。
2. 不调用 generic 文案黑名单决定主视图。
3. 不因 `answerCanonicalState=final` 清空工具/进展。
4. 不因 `turn_completed` 清空工具/进展。
5. 不把 `tool_item_completed` 写入 assistant body。
6. 不把 `tool_item_started` 变成主视图 `current_action`；`current_action` 只能来自 `tool_call_requested` frame。

### 4.7 MessagePublicProjection

UI 唯一输入：

```ts
type MessagePublicProjection = {
  bodyText: string;
  bodyState: "streaming" | "finalized" | "committed";
  currentAction?: ProjectionItem;

```

```text

pinned
finalResults
status

```

## 5. 标准运行时序

### 5.1 有模型开场反馈

```text
1. user message
2. assistant_text_delta
   -> frame: body_append, slot=body, visible_live
3. assistant_text_final 或继续 streaming
   -> frame: body_finalize, slot=body
4. tool_call_requested
   -> frame: item_upsert, slot=current_action, source_authority=model, tool_call_id, visible_live, retention=transient
5. tool_permission_decided
   -> frame: item_upsert, slot=trace, permission_decision_id, tool_call_id
6. tool_item_started
   -> frame: item_upsert, slot=trace, tool_lifecycle_id, tool_call_id, permission_decision_id
   -> current_action 只更新执行状态，不生成新语义
7. tool_item_completed success
   -> frame: item_retire, item_id=tool_call_id, retention=trace
   -> trace 更新 completed observation
8. assistant_text_delta/final
   -> body_append/body_finalize
9. session_output_commit_checked
   -> commit checked，不 hydrate
10. session_output_commit_ack
   -> commit_ack，hydrate 一次
11. projection_scope_retired
   -> scope_retire transient，普通 current/timeline/tool 退出主视图进 trace
12. task_run_lifecycle_finished(completed)
13. turn_completed
   -> turn_terminal，不清屏，不 hydrate
```

### 5.2 没有 assistant 正文，但有工具调用请求

```text
1. user message
2. tool_call_requested
   -> frame: item_upsert, slot=current_action, source_authority=model, tool_call_id, visible_live
3. tool_permission_decided
   -> permission_decision_id 绑定 tool_call_id，decision=allow/auto_allow
4. tool_item_started
   -> trace 记录工具开始，绑定 tool_call_id/permission_decision_id
   -> 主视图仍显示上游 ToolCallRequest 投影，不显示裸工具
5. tool_item_completed success
   -> ToolCallRequest 投影 retire 到 trace
6. assistant_text_delta/final
   -> body 显示真正模型输出
7. session_output_commit_ack
   -> hydrate 一次并进入 final view
```

系统不得补造“我正在处理”。工具前主视图反馈的唯一合法路径是模型输出里的 `tool_use/function_call/custom_tool_call` 被 normalizer 转成 `ToolCallRequest`；如果模型同时输出 assistant 正文，则正文也可以先显示。前端和 projection 层都不能伪造正文，也不能从 `tool_item_started` 反推行动语义。

### 5.3 工具失败

```text
tool_call_requested
-> current_action visible_live

tool_item_started
-> trace only，绑定 tool_call_id/permission_decision_id

tool_item_completed failed
-> item_upsert slot=pinned
-> pin_reason=failed
-> trace 更新失败 observation
```

失败 item 不随 commit ack retire，除非后端发 `item_retire` 且带 resolved refs。

### 5.4 等待用户或权限

```text
approval_required / waiting_user
-> item_upsert slot=pinned
-> pin_reason=waiting_user 或 waiting_approval
```

该状态必须留在主视图，不能进入 trace-only。

### 5.5 commit failed

```text
assistant_text_final
-> body finalized

session_output_commit_failed
-> commit_failed
-> pinned commit failure
-> 禁止 task completed
-> 禁止 hydrate 旧 history
```

用户看到 final live body 和“输出提交失败”，不会看到假 completed。

## 6. 收口状态机

item 状态迁移固定如下：

```text
created_live
-> running_live
-> completed_transient
-> trace_retired
```

例外状态：

```text
failed_live -> pinned_until_resolved
waiting_live -> pinned_until_resolved
artifact_created -> visible_final
verification_result -> visible_final
commit_failed -> pinned_until_resolved
```

commit ack 后执行：

```text
scope_retire(target=transient)
```

该操作只影响：

```text
current_action.retention=transient
status.retention=transient
timeline.retention=transient
```

该操作不得影响：

```text
body
pinned
finalResults
status with pin_reason
trace
```

## 8. 验收矩阵

| 场景 | 必须结果 |
| --- | --- |
| 模型先输出正文，再调用工具 | 正文先出现，工具作为 current action 出现 |
| assistant 正文尚未输出，但模型已有工具调用请求 | 先显示 `ToolCallRequest` 产生的 current action，再执行工具 |
| 工具事件早于模型工具调用请求 | protocol diagnostics；不生成主视图 projection frame；测试失败暴露后端时序错误 |
| 普通工具成功 | 主视图 current action 消失，trace count 增加 |
| 工具失败 | pinned failure 留主视图 |
| final answer 出现 | 不触发工具清理，不触发 hydrate |
| commit ack 出现 | hydrate 一次，transient items retire 到 trace |
| turn_completed 出现 | 只关闭 stream，不清屏，不 hydrate |
| commit failed | 保留 live body，显示 pinned failure，不 completed |
| monitor completed 无 commit ack | 不 hydrate，不清理 live view |
| reconnect replay | 按 event_offset 重建同一 body/current/pinned/final/trace |
| 页面刷新后 committed session | 主聊天只显示 final body、finalResults、pinned unresolved、trace entrance |

## 12. 理论与工程依据

本方案不是凭感觉设计。它由以下依据共同支撑。

### 12.1 理论依据

| 依据 | 对本方案的约束 |
| --- | --- |
| Event Sourcing | 运行事实必须以 append-only event 记录；前端视图通过 `event_offset` replay/折叠重建，不能从当前 UI 状态反推事实。对应本方案的 `Runtime Event Log -> Projection Frame Stream -> Projection Ledger`。参考：Martin Fowler, Event Sourcing；Microsoft Azure Architecture Center, Event Sourcing Pattern。 |
| CQRS / Read Model | 写侧事实和读侧投影视图必须分离。runtime event log 是写侧事实；`MessagePublicProjection` 是读侧视图；React 组件不得回写或重判语义。参考：Martin Fowler, CQRS。 |
| Deterministic State Machine | `created_live -> running_live -> completed_transient -> trace_retired` 和 pinned 例外是确定状态机。状态迁移由 frame op 驱动，不由文案、terminal 或 final answer 猜测。 |
| Pure Render | UI 组件只根据输入渲染同一结果，不做隐藏/折叠/正文归属等语义副作用。对应删除 `terminalScopedPublicTimelineItems()`、`isNaturalLanguageStageFeedback()`、`isGenericStatusActivity()`。参考：React render purity 文档。 |
| Reliable SSE Replay | SSE 需要稳定 `id` / offset 才能断线续接；projection frame 必须有 `source_event_id` 和 `event_offset`，reconnect replay frame，而不是 replay raw event 后重新解释。参考：WHATWG HTML SSE、MDN Server-sent events。 |

### 12.2 成熟 agent 源码依据

| 依据 | 源码事实 | 对本方案的约束 |
| --- | --- | --- |
| Codex 工具调用链路 | `D:\AI应用\openai-codex\codex-rs\protocol\src\models.rs` 定义 `ResponseItem::FunctionCall/CustomToolCall` 和 `FunctionCallOutput/CustomToolCallOutput`；`core\src\tools\router.rs` 将 `ResponseItem` 转为 `ToolCall`；`core\src\tools\context.rs` 用 `ToolInvocation` 携带 `call_id/tool_name/payload/source`；`core\src\tools\registry.rs` 在执行前后触发 lifecycle/hook。 | 工具公开生命周期必须绑定 `tool_call_id/call_id`，不能绑定独立意图字段。工具结果不是 assistant body，必须进入 trace/pinned/finalResult 等投影槽。 |
| Codex 对外投影链路 | `sdk\typescript\src\events.ts` 使用 `turn.started/item.started/item.updated/item.completed/turn.completed`；`sdk\typescript\src\items.ts` 使用 `agent_message/reasoning/command_execution/mcp_tool_call/todo_list/error`。 | 前端应消费结构化 item/frame，不应从 raw terminal/tool 文案重新判断主视图语义。`turn.completed` 只是 terminal，不是 commit/hydrate 权威。 |
| Claude Code 工具调用链路 | `D:\AI应用\claude-code-nb-main\query.ts` 在 streaming 中收集 assistant `tool_use` block，并交给 `StreamingToolExecutor.addTool()` 或 `runTools()`；`Tool.ts` 定义 `ToolUseContext` 和 `renderToolUseMessage/renderToolResultMessage`；`utils\permissions\permissions.ts` 通过 `hasPermissionsToUseTool()` 做权限决策。 | 模型可以先产生 tool_use 而没有正文；这是成熟行为。主视图第一语义反馈可以是 `ToolCallRequest` 投影，但不能是裸 `tool_item_started`。 |
| Claude Code UI 生命周期 | `Tool.ts` 的 `renderToolUseMessage()` 是工具调用展示入口，输入是 partial tool input，用于流式参数未完整到达时尽早展示。 | 我们可以在 `tool_call_requested` 阶段显示 current action；该 action 来自模型工具调用项，不来自工具执行 started 事件。 |
| Claude Code tool_use 配对约束 | 代码和文档大量使用 `tool_use_id` 维护 `tool_use/tool_result` 配对，并在 resume/compact/storage 中清理不完整配对。 | replay/hydrate 必须以 `tool_call_id/tool_use_id` 做稳定配对；孤儿 tool result 或无请求 tool start 只能进入 diagnostics/trace protocol error。 |

结论：成熟 agent 的真实结构不是 `intent layer`，而是“模型输出的工具调用项”承担动作请求角色；权限和执行在其后，投影从该工具调用项开始。本文后续所有 `ToolCallRequest` 都指这一层，不新增独立意图协议。

# 任务运行流畅度与证据状态闭环修复计划书

日期：2026-06-13

状态：待审核，未实施

适用范围：TaskRun 执行循环、模型可见运行事实、文件读取状态权威、工具生命周期、公开投影、prompt 装配、前端 Projection Ledger。

明确不包含：图任务前台重做、重写模型调度权、前端按文案裁决语义、为了兼容保留旧链路、用硬编码过滤模型回答。

---

## 1. 目标结论

这次不流畅不是“模型不听话”，而是系统给模型和前端的信号链有断点和矛盾：

```text
Tool Observation
-> FileState / TaskState
-> Dynamic Context / Prompt
-> Model Action
-> Runtime Step Summary
-> Public Projection
-> Frontend Ledger
```

当前问题集中在四类：

1. **证据状态不可信**：文件已经完整读过，但 FileState 仍显示 `partial / has_more=true`，模型自然会继续读。
2. **模型可见事实不够清爽**：已读范围、可复用内容、下一步缺口没有形成明确的任务事实，模型需要靠重复读取恢复上下文。
3. **公开投影把工具意图变成多条用户可见状态**：同一阶段的 `current_judgment` 被按事件 offset 生成新 item，前端只能照单显示。
4. **工具生命周期和 todo 生命周期没有给模型稳定账本**：模型不知道哪些 todo id 合法，也不知道哪些工具结果已经成为可复用证据。

修复原则不是限制大模型，而是把运行事实做准、把服务边界做清、把异常显式暴露。模型继续负责调度；系统只负责提供可信证据、执行工具、记录状态、投影结果和保护真实安全边界。

## 2. 必须遵守的控制标准

### 2.1 运行中透明

每个影响模型下一步的事实都必须能追到来源：

```text
task_run_id / invocation_id / tool_call_id / observation_ref / source_event_id / event_offset
```

文件状态、todo 状态、工具生命周期、公开投影不能只给自然语言结论，必须保留结构化证据。

### 2.2 收口后清爽

任务收口后只保留：

- 最终 assistant 正文。
- 已完成的工具/文件/todo 证据摘要。
- 失败或异常诊断。

普通读取进度、重复工具观察、临时 current action、内部 step summary 必须退到 trace，不长期占据主聊天正文或公开状态面板。

### 2.3 异常不隐藏

这些情况必须显式记录为诊断，而不是静默兜底：

- 文件覆盖范围完整但状态仍 partial。
- 工具观察缺少 observation_ref。
- todo 操作用了不存在的 todo_id。
- 模型请求的工具不在 runtime assembly。
- 公开投影缺少 anchor 或 item lifecycle 乱序。
- prompt 装配给出互相冲突的反馈要求。

### 2.4 证据可追溯

模型可见的“已读”“已搜索”“已编辑”“已验证”必须绑定证据引用。不能只告诉模型“你已经读过”，要给它：

```text
path
coverage
content_sha256
last_observation_ref
reusable_result_ref
next_missing_ranges
stale
```

### 2.5 顺序严格稳定

同一 task run 内的用户可见 item 不能靠到达顺序和新 event id 无限新增。阶段状态必须有稳定 slot；工具生命周期必须以 `tool_call_id` 为 item identity；正文必须以 assistant message ref 为 identity。

### 2.6 前端不裁决语义

前端只执行 `PublicProjectionFrame`：

```text
op / slot / item_id / visibility / retention / source_event_id / source_event_type
```

前端不得根据中文文案、工具名、final 字样、任务状态自行隐藏、合并或改写语义。

```text
  agent_todo JSON
```

所以“一直在思考”的表象是：运行时确实已经进入下一次模型调用等待，但它之所以进入第 25 轮，是因为上游状态一直把任务推回“读/搜/确认/看 todo”。

Prompt cache 追查：

- task executor 的 cache boundary 是 `bound_task_context_stable`。
- 多轮 prompt cache 状态为 `hit`，说明稳定前缀确实在复用。
- 这不是直接 bug；问题是 `bound_task_context_stable` 当前包含错误的稳定事实：`mario.html status=partial`。
- 错误事实一旦进入稳定前缀，会被多轮稳定复用，直到 context hash 改变。

结论：

```text
prompt cache 不是根因
但会放大根因
根因仍是稳定上下文里放进了错误/噪声事实
```

修复时必须保证 cacheable stable segment 只放真正稳定且正确的身份事实；读窗口、`has_more`、latest progress、todo JSON 这类变化事实必须在 volatile Evidence Ledger 中表达，并且不能污染 stable prefix。

### 3.1 文件状态权威把完整覆盖误报为 partial

证据文件：

- `storage/runtime_state/file_state/taskrun_turn_session-32740862204144e2_11_f7d8f21b.json`
- `backend/runtime/memory/file_state_authority.py`

实际状态：

```text
mario.html coverage = 1..957
total_lines = 957
range_count = 9
status = partial
has_more = true
```

这说明后续小窗口读取的 `has_more=true` 覆盖了聚合事实。对模型来说，这就是系统告诉它“文件还没读完”，重复读不是模型单方面问题。

目标：

```text
aggregate coverage covers 1..total_lines -> status=complete, has_more=false
latest window has_more 只记录在 latest_read，不得覆盖 aggregate_file_state
```

代码根因：

```text
_apply_file_event()
  has_more = event.has_more
  if has_more is True or _has_partial_coverage(...):
      status = partial
```

这里把单次 read window 的 `has_more=true` 放在聚合覆盖之前，导致全文件状态被最后一次局部 read 改写。

### 3.2 read_file 已有缓存，但模型拿不到足够清晰的复用路径

证据：

- 计划文件已完整读取，重复读返回 “File unchanged / content omitted / Use earlier result”。
- 模型随后改用 terminal `Get-Content` 恢复内容。
- 最后一次 packet 中存在 `rehydration_plan.replacement_ref=replacement:0399abdb18444cd087fb4f18`，但它作为 replay entry 出现，不是模型稳定可见的任务事实。

问题不是没有缓存，而是缓存回执没有形成模型可操作的结构化事实。模型不知道“早前结果在哪里、是否仍可用、要如何引用”。

目标：

```text
unchanged read observation:
  file_unchanged=true
  reusable_result_ref=...
  previous_observation_ref=...
  content_sha256=...
  coverage=complete
  next_action_hint=use_reusable_result_or_edit
```

这里的 `next_action_hint` 是工具服务事实，不是语义调度；它只描述可用服务状态，不替模型决定任务策略。

### 3.3 公开投影把同一阶段判断拆成多条可见状态

证据位置：

- `backend/harness/runtime/projection/projector.py`
- `frontend/src/lib/projection/reducer.ts`
- `frontend/src/components/chat/ChatMessage.tsx`

当前行为：

```text
runtime_step_summary -> item_id includes source event / offset / step
frontend item_upsert by item_id
=> 每个 runtime_step_summary 都是新状态
```

后果：

模型写的阶段判断、工具前判断、观察后判断都会堆在主 UI，用户看到像“旧状态一直不走”。

代码根因：

```text
chat.py:
  step_summary_recorded -> runtime_step_summary

projector.py:
  _runtime_step_summary_spec()
  item_id = stable_id("runtime-step", runtime_event_id, source_task_event_offset, step)
```

所以同一任务阶段只要 offset 或 step 变化，就一定成为新 UI item。前端 `item_upsert` 没有错；后端 item identity 设计错了。

目标：

```text
stage/status slot:
  task_public_stage_status:{task_run_id}:{phase_or_invocation}

tool lifecycle slot:
  tool_call:{tool_call_id}

trace-only runtime summaries:
  tool observation summary
  duplicate read guard
  internal diagnostics
```

重要约束：不能用中文文案过滤，必须由后端结构化 source / slot / lifecycle 决定。

### 3.4 prompt 装配要求反馈，但反馈类型没有稳定边界

证据位置：

- `backend/harness/runtime/compiler.py`

当前冲突：

- schema 说 `visible_status` 不是用户正文，不要复制到公开字段。
- prompt 又要求工具观察后如果继续执行工具，就写 `public_action_state.current_judgment`。
- 这会诱导模型每个工具循环都写一句公开判断。

目标：

模型必须给用户反馈，但反馈要分层：

```text
首轮/阶段边界/方向变化/失败恢复/准备收口 -> public current_judgment
普通工具链路推进 -> tool lifecycle current_action
纯内部读取/缓存恢复/重复观察 -> trace
最终结果 -> assistant body
```

这不是让模型少反馈，而是让反馈进入正确通道。

进一步追查结果：

`compiler.py` 同时存在两类要求：

```text
1. 反馈必须真实、有用户价值，不能写“正在处理/工具成功”这类空状态。
2. 工具观察返回后，如果还需要继续执行工具，在 current_judgment 写一句公开短判断。
```

第二条会把普通工具循环推向公开反馈，尤其当工具观察本身是 “File unchanged / todo JSON / terminal error” 时，模型会反复写“需要先恢复计划/确认状态”。

### 3.5 agent_todo 没有形成稳定任务账本

证据：

- 初始 todo 是一个宽泛 item。
- 模型使用 `todo-phase1`，系统返回 `todo_id not found: todo-phase1`。
- todo 工具结果可能以大段 JSON 污染最新任务摘要。

目标：

给模型明确合法 todo ledger：

```text
current_todos:
  - id
  - title
  - status
  - allowed_operations
  - evidence_refs
```

`agent_todo` 的工具结果应进入结构化 task facts，不应作为大段自然语言或 JSON 覆盖 `latest_progress`。

代码根因：

```text
task_executor.py:
  _record_task_step_summary(... presentation_source="tool_observation.summary")
  _record_task_step_summary()
    diagnostics.latest_public_progress_note = visible_note or visible_summary
    diagnostics.agent_brief_output = visible_brief
```

也就是说即使投影层把 `tool_observation.summary` 隐藏为 trace，task_run diagnostics 仍然会把它当 latest progress。最后一次 task_run state 中：

```text
latest_public_progress_note = agent_todo JSON
agent_brief_output = agent_todo JSON
latest_current_judgment = 开始阶段一修复...
```

这三个字段互相冲突，下一轮 prompt 又把 JSON 当最新进展投给模型。

### 3.6 重复只读拦截是反应式补救，不是流畅运行方案

证据位置：

- `backend/harness/loop/task_executor.py`
- `_DUPLICATE_READ_ONLY_TOOL_CALL_LIMIT = 5`

当前行为：

重复读很多次以后才拦截，并且可能生成用户可见状态。

目标：

重复读不应靠“拦截模型”解决，而应靠上游事实闭环解决：

```text
FileState accurate
DynamicContext shows complete coverage
Read unchanged returns reusable evidence
Prompt tells model how to use evidence ledger
Duplicate guard only作为异常诊断，不作为主控制路径
```

真实运行中 guard 已经在 offset 379 触发：

```text
相同参数的只读工具调用已达到 5 次：read_file。
请使用已有 observation 作为证据，或改用更有针对性的验证工具/参数；如果合同已满足，应直接 respond。
```

这条 guard 方向上是保护执行面，但位置太晚。它证明系统已经知道重复，却没有在模型输入里提前给出足够清晰、可执行的证据账本。

### 3.7 Dynamic Context 选取策略导致“最新文件”遮蔽“目标文件”

证据位置：

- `backend/harness/runtime/dynamic_context/task_state_projector.py`

当前 `_read_resource_state_projection()` 只返回一个 `latest` 文件：

```text
latest = active_files[-1]
path = latest.path
```

最后一次 packet 里它返回的是计划文件：

```text
read_resource_state.path = storage/memory/durable/global_common/notes/project-mario-full-fix-plan.md
status = available
has_more = false
```

但真正的目标文件 `mario.html` 在 `bound_task_context` 里显示为 partial。模型因此得到一个偏斜信号：最近资源是计划文件，目标文件仍 partial，最新进展又是 todo JSON。

目标不是让系统替模型决定“不要读计划”，而是给模型同时看清：

```text
target_file mario.html:
  aggregate_status=complete
  coverage=1..957
  local_windows_available=[...]
  edit_precision_requires_minimal_window_only

plan_file:
  aggregate_status=complete
  reusable_result_ref=...
```

### 3.8 Bound Task Context 没有把恢复引用变成模型稳定事实

证据位置：

- `backend/harness/runtime/bound_task_context.py`

当前 `BoundTaskContext._stable_payload_body()` 只投：

```text
plan_refs
context_refs
known_task_files
restore_policy
```

而 `task_files/edit_targets/rehydration_refs` 只在 manifest 中完整存在。manifest 可追溯，但它不是模型最容易消费的稳定执行事实。

结果：

- 模型知道计划文件 complete。
- 模型知道有 restore policy。
- 但模型不知道“这份计划的完整内容已通过哪个 replacement_ref 可复用”。
- 当 read_file 返回 content omitted，模型转向 terminal。

目标：

`rehydration_refs` 中和当前任务关键文件相关的条目，应进入模型可见 Evidence Ledger，而不是只留在 manifest。

## 4. 目标权威链

```text
User Request
-> RuntimeStartPacket
-> Tool Assembly / Permission Boundary
-> Model Turn Decision
-> Tool Execution
-> Tool Observation Envelope
-> Evidence State Authorities
   - FileStateAuthority
   - TodoStateAuthority
   - ToolLifecycleAuthority
-> Dynamic Context Projection
-> Model Next Decision
-> Public Projection Authority
-> Frontend Ledger
-> Pure Render
```

### 权责边界

| 层级 | 允许做什么 | 不允许做什么 |
| --- | --- | --- |
| 模型 | 判断任务、选择工具、决定是否继续、向用户反馈 | 给自己授权、伪造工具结果 |
| 工具权限层 | 判断工具存在、参数合法、权限允许 | 改写模型目标、替模型选择替代动作 |
| 执行层 | 执行工具、返回观察 | 把观察改写成用户语义结论 |
| 证据状态层 | 聚合文件/todo/tool事实 | 用自然语言猜测模型下一步 |
| dynamic context | 把事实投给模型 | 隐藏矛盾、注入开发说明 |
| public projection | 把结构化事件投给 UI | 根据文案判断语义 |
| 前端 | 渲染 frame ledger | 裁决正文/工具/状态含义 |

## 5. 修复设计

### 5.1 FileStateAuthority 改为聚合覆盖优先

文件：

- `backend/runtime/memory/file_state_authority.py`
- 相关测试新增或改写到 `backend/tests/`

规则：

```text
if total_lines known and merged_ranges cover 1..total_lines:
    status = complete
    has_more = false
    next_suggested_read = null
else:
    status = partial
    has_more = true/unknown based on aggregate gap
```

`latest_read.has_more` 可以保留，但不能覆盖 `aggregate.has_more`。

删除/收口：

- 保留每次 read event 作为 evidence，不再让它拥有文件整体状态权威。

### 5.5 修正 prompt 反馈协议

文件：

目标文案应面向 agent，而不是开发说明：

这不是禁止模型反馈，而是明确反馈的用户价值和通道。

### 5.6 agent_todo 生命周期改为账本式服务

文件：

目标：

### 5.7 latest diagnostics 分层

文件：

- `backend/harness/loop/task_executor.py`
- `backend/harness/runtime/dynamic_context/task_state_projector.py`

将现在容易混在一起的 latest 字段拆开：

```text

latest_model_judgment
latest_tool_observation_trace
latest_material_progress
latest_failure_diagnostic
```

工具观察摘要、todo JSON、重复读拦截不能覆盖用户可见进度，也不能成为下一轮 prompt 的核心“最新任务状态”。

### 5.8 Prompt Cache 边界修正

文件：

- `backend/harness/runtime/bound_task_context.py`
- `backend/harness/runtime/dynamic_context/task_state_projector.py`
- prompt accounting 相关测试或现有 prompt stability 测试

规则：

```text
cacheable stable segment:
  role / schema / tool catalog / project rules / task contract / stable file identities

volatile evidence segment:
  read windows
  has_more
  latest observations
  todo ledger
  rehydration refs
  work_progress
  failures
```

禁止：

- 把错误的 `status=partial/has_more=true` 放进长期稳定上下文。
- 把 todo JSON、工具观察摘要、latest progress 写入 cacheable prefix。
- 用自然语言摘要作为 stable cache identity。

要求：

- `bound_task_context_stable` 只描述文件身份、目标、合同和稳定恢复策略。
- `Evidence Ledger` 用 content hash、coverage、observation_ref、replacement_ref 表达实时证据。
- 如果文件状态从 partial 修正为 complete，stable context hash 必须随之变化一次，之后保持稳定。
- prompt stability report 中 stable prefix changed 必须能解释为“稳定事实真的变了”，不能因为工具噪声变化而反复变化。

验证：

```powershell
pytest backend/tests -k "prompt_cache or prompt_stability or bound_task_context"
```

必须覆盖：

- todo JSON 不进入 stable prefix。
- latest tool observation 不进入 stable prefix。
- complete file identity 稳定复用，但 read window/has_more 留在 volatile evidence。

### 5.9 清理旧链路

实施时必须删除或退役：

- 依赖最后一次 read window 推断文件整体状态的逻辑。
- 公开投影里按 runtime event offset 无限新增阶段状态的路径。
- 前端任何按中文文案裁决 projection 的残留。
- 把 tool observation summary 当用户公开状态的路径。
- 把 todo 大段 JSON 写进 latest progress 的路径。
- 任何为了兼容旧投影而保留的双写主链路。

## 7. 验收标准

### 控制精准

- 模型看到的文件/todo/tool事实与 runtime state 一致。
- 工具存在性、权限、参数错误都有明确 observation。
- 不存在系统以“流畅/安全”为名替模型选择下一步。

### 运行顺畅

- 模型无需重复读取完整文件来恢复上下文。
- 首轮有反馈，长期任务有阶段反馈。
- 工具生命周期清楚，正文线干净。
- 重复读 guard 只作为异常诊断，不作为常规推进机制。

### 覆盖全面

- 文件读取、搜索、编辑、todo、工具失败、投影、前端渲染都有测试。
- prompt cache 不被重复自然语言摘要污染。
- diagnostics 和 public status 分层，不互相覆盖。

## 8. 实施前自审

本计划符合当前原则：

- **运行中透明**：所有关键状态带 evidence ref。
- **收口后清爽**：trace-only 与 public status 分离。
- **异常不隐藏**：状态矛盾、todo id 错误、工具缺失都显式诊断。
- **证据可追溯**：文件/todo/tool 状态都可追到 observation/event。
- **顺序严格稳定**：公开状态 item identity 稳定，工具按 tool_call_id。
- **前端不裁决语义**：语义由后端 frame/slot/retention 决定，前端纯渲染。

本计划也避免三类错误方向：

- 不用中文硬编码过滤模型回答。
- 不把系统设计成模型调度的上级。
- 不保留两条都不成熟的旧链路。

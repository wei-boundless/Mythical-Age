## 1. 目的

本计划解决一个结构性断层：普通 turn 中模型已经读取文件、调用工具、获得记忆上下文后，如果它决定开启持续任务，任务启动后的 task execution packet 会切换到 `task_run_id` 作用域重新装配上下文，导致任务开启前的动态记忆、文件证据和工具观察没有自然继承。

用户强调的“动态记忆要拼接”在本计划中定义为：

```text
普通 turn 当下已经选出、读出、观察到、并允许模型看见的动态上下文
-> 在任务启动时形成显式 handoff
-> 进入 task_run 的首轮 task execution packet
-> 后续任务步骤按 task scope 继续增量维护
```

它不是把长期记忆或历史投影无条件塞进 prompt，也不是让持久投影覆盖当前事实。它是一条从普通 turn 到任务运行的受控上下文桥。

正确目标：

- 任务启动前后，模型看到的文件事实、动态记忆、当前约束和工具观察连续。
- 普通 turn 中已读取的文件窗口不因 `task_run_id` 切换而消失。
- memory context 在任务启动时继承已选上下文 refs，而不是重新抽样造成断层。
- `DynamicContextManager` 只投影，不重新决定任务意图。
- `RuntimeCompiler` 只装配已明确的上下文层，不用 prompt 层补洞。
- 6 月 17 日提示词体系中有价值的 prompt/cache 分层被纳入重构，但不盲目回档 action protocol。

### 2.2 任务启动回调没有接收本轮动态上下文

这就是用户说的“任务开启前读取到的文件信息无法继承”的同源问题。

## 3. 17 号提示词体系审查

### 3.0 Prompt cache 专项审查补充

2026-06-17 追加审查结论：本计划可以继续作为任务启动断层修复的主计划，但原计划中对 lifecycle prompt 的表述需要收紧。问题不是“所有 lifecycle prompt 都不能缓存”，而是：

```text
稳定资源库可以缓存
稳定 invocation / capability 边界触发的 lifecycle refs 可以缓存
由本轮 memory / observations / pending steer / compaction / active work 等动态状态触发的挂载结果
不能出现在 session_stable / task_stable 前缀里
```

当前代码已有 `PromptSegmentPlan` 的重要护栏：

- stable prefix 只从开头连续收集 `cacheable_prefix` / `session_stable` 段。
- 一旦遇到 volatile 段，后续内容不再属于 provider 可复用前缀。
- 稳定前缀段禁止携带 `turn_id`、`task_run_id`、`observations`、`current_facts`、`pending_user_steers` 等运行态字段。

因此 handoff 不能绕过 `segment_plan`，也不能把动态记忆伪装成 stable。正确做法是复用现有 stable/volatile 机制，把任务启动继承上下文作为明确 volatile/ref-only 输入接入。

本次对照的外部成熟实现约束：

- Claude Code 使用 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 把全局静态 prompt 与会话/运行时动态内容分开；`DANGEROUS_uncachedSystemPromptSection` 用命名强制开发者意识到“这里会破坏缓存”。
- Claude Code 的 fork/subagent 模式会继承父上下文或克隆 read file / content replacement state，但仍保持默认隔离、显式共享，避免子任务隐式污染父线程。
- Codex 的 `Prompt` 将 `base_instructions` 与 `input` 分离；`additional_context` 是模型可见的显式附加上下文，但不是普通 user message，并且按 trust kind 决定进入 developer 还是 user 外部上下文。

这些参照共同指向同一条原则：

```text
任务启动继承上下文必须是显式输入
动态输入必须位于缓存边界之后
可缓存前缀只能包含跨同类调用字节稳定的规则、契约、目录和工具 schema
```

### 3.1 可借鉴部分

对比 6 月 17 日相关实现后，以下设计值得纳入本次重构：

1. **lifecycle prompt 拆成 stable core 与 volatile runtime guidance**

   17 号 `compiler.py` 曾使用 `_lifecycle_runtime_message_spec(...)`，把由当前状态触发的 lifecycle prompt 放到 volatile 后缀，而不是混入稳定前缀。该方向需要保留，但不能粗暴迁移所有 lifecycle prompt。

   目标拆分：

   - `lifecycle_static_core`：由 environment kind、invocation kind、固定 action schema、固定 capability surface 决定，适合 `session_stable`。
   - `lifecycle_runtime_guidance`：由本轮 `memory_context`、observations、pending steer、compaction/recovery、active work 等动态状态触发，只能 `cache_scope=none`、`cache_role=volatile`。

   如果某条规则虽然和 memory 有关，但只是“如何使用记忆”的静态工作纪律，例如 `memory_read_context`，有两种成熟选项：

   - 固定挂载到该 invocation 的 stable core，不随 memory 是否命中而开关。
   - 保持按状态触发，但挂到 volatile runtime guidance。

   禁止第三种做法：由 memory 是否存在动态开关 stable lifecycle refs。

2. **task_state replay 的稳定性判定**

   17 号实现把 `task_state_replay_entries` 分为 stable specs 和 volatile specs：有稳定 identity、且不含 runtime-instance fields 的证据才允许进入 task append-only prefix；包含 `turn_id`、`task_run_id`、`executor_status`、`pending_user_steers` 等运行态字段的内容必须留在 volatile suffix。

3. **execution control state 独立 volatile payload**

   17 号实现把 `pending_tool_control_actions`、`runtime_control_signals`、`latest_runtime_control_signal` 单独作为执行控制状态投给模型。这能减少任务执行中“工具结果已产生但控制动作未接上”的断层。

4. **更完整的 dynamic task state**

   17 号版本曾把 `authoritative_subagent_results`、`pending_tool_control_actions`、`runtime_control_signals` 等从 `ExecutionStateProjector` 投给 `TaskStateProjector`。当前任务状态投影已有部分接收能力，计划应补齐上游投影。

5. **prompt cache 结构护栏**

   当前 `prompt_accounting_ledger_test.py` 已验证 memory 正文不进入 `environment_stable`，但还没有充分验证“memory/handoff 内容变化不会改变 stable prefix hash”。本次实施必须补上 stable prefix hash 级别的测试，防止动态内容虽然不进 stable 正文，却通过 lifecycle refs 动态选择切裂缓存前缀。

### 3.2 不直接回档部分

1. **不直接恢复 native action protocol**

   17 号存在 `model_action_names.py` 与 `native_action_tools.py`，把内部 `request_task_run` 映射成模型可见的 `start_task` 等 native action tool。这个方向可读性更好，但会改变当前 JSON action transport、action schema、协议测试和前端投影假设。本次先不切换。

2. **不盲目恢复整段 `compiler.py` / `task_executor.py`**

   当前公开投影、缓存、产物边界已经恢复并通过测试。盲目回档会再次破坏已修好的 public projection、artifact/cache/output boundary。

3. **不把 prompt 当补丁**

   任务启动断层不是一句“请继承上文”能修好的问题。提示词只描述 agent 角色、责任和边界；上下文继承必须由 runtime packet 的结构输入保证。

## 4. 目标权威链

目标链路：

```text
RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> ModelTurnDecision
-> ActionPermit
-> TurnToTaskContextHandoff
-> RuntimeStartPacket
-> ExecutionLoop
```

分层职责：

| 层 | 当前 owner | 目标职责 | 禁止事项 |
| --- | --- | --- | --- |
| RequestFacts | entrypoint / turn_input_facts | 记录用户输入、环境绑定、editor context、active work candidate | 不决定任务意图 |
| ContextCandidates | memory provider / dynamic context / tool observations | 收集并投影当前可见事实 | 不改写用户目标 |
| ModelTurnDecision | `ModelActionRequest` | 决定 respond / tool_call / request_task_run 等动作 | 不授权自己越权 |
| ActionPermit | admission / permission boundary | 授权或拒绝动作 | 不替模型选择新目标 |
| TurnToTaskContextHandoff | 新增 | 把本轮已观察事实交接给 task lifecycle | 不发明新事实，不保存私有思考 |
| RuntimeStartPacket | task lifecycle / compiler | 装配任务启动上下文与首轮 packet | 不重新推断 intent |
| ExecutionLoop | task executor | 执行、观察、记录、恢复 | 不从旧状态猜缺失上下文 |

## 5. 目标设计

### 5.1 新增 `TurnToTaskContextHandoff`

新增一等运行对象，用于承接普通 turn 到任务运行的动态上下文。

建议结构：

```text
TurnToTaskContextHandoff
  handoff_id
  session_id
  turn_id
  source_packet_ref
  task_run_id
  created_at
  inherited_observation_refs
  inherited_observations
  inherited_file_evidence_scope
  inherited_file_state_snapshot
  inherited_memory_context_ref
  inherited_memory_context
  inherited_runtime_observations
  inherited_editor_context_ref
  inherited_turn_input_facts
  inherited_dynamic_context_report
  artifact_refs
  authority = harness.loop.turn_to_task_context_handoff
```

其中 `inherited_memory_context` 只保留模型可见的 `model_visible_sections`、`selected_sections`、`memory_runtime_view_ref`、`context_package_ref` 和 diagnostics 摘要，不保存 memory provider 内部候选全集。

### 5.2 Handoff 捕获规则

允许继承：

- 已返回给模型或将进入模型 follow-up 的 tool observations。
- 工具观察中的 read_file / search / stat / structured read 证据。
- session file evidence scope 当前 snapshot。
- 当前 `session_context["memory_context"]` 的模型可见部分与 refs。
- runtime observations，例如显式 contract invalid、权限拒绝、工具失败观察。
- editor_context 的安全快照。
- turn_input_facts、current_work_boundary_receipt 的可见摘要。
- 当前 packet_ref、prompt manifest refs、dynamic context report refs。

禁止继承：

- 模型 hidden reasoning、私有 chain-of-thought。
- 未授权工具请求。
- 未进入模型上下文的调试内部对象。
- 旧任务持久投影中与当前 turn 冲突的状态。
- 仅为兼容旧字段存在的 task_goal/resource_contract 等旧合同字段。

### 5.3 Handoff 存储与事件

任务启动时：

```text
single_agent_turn
-> build_turn_to_task_context_handoff(...)
-> runtime_objects.put_object("turn_to_task_context_handoff", handoff_id, payload)
-> event_log.append(task_run_id, "turn_to_task_context_handoff_recorded", ...)
-> task_run.diagnostics["turn_to_task_context_handoff_ref"] = handoff_ref
```

同一 handoff 同时写入：

- task_run diagnostics：便于 executor 快速定位。
- runtime_objects：便于 packet 装配读取。
- event_log：便于审计、公开投影摘要、测试断言。

### 5.4 文件证据桥接

当前普通 turn 工具调用使用：

```text
session_file_evidence_scope(session_id)
```

任务工具调用使用：

```text
task_run_file_evidence_scope(task_run_id, session_id=session_id)
```

桥接策略：

1. 创建 handoff 时读取 session scope 中与本轮 observations 相关的 file_state snapshot。
2. task_run 创建后，把这些 read events 或 snapshot 以 `source_scope=session`、`handoff_id=...` 写入 task_run file evidence scope。
3. task execution 首轮 `_observations_for_packet(task_run_id)` 能看到 task scope file_state。
4. 如果 snapshot 与后续 task observation 冲突，以后续 task observation 为准。

禁止做法：

- executor 每轮隐式 fallback 到 session scope。
- compiler 根据 session history 猜测模型读过哪些文件。
- 让 editor_context 替代 tool observation 授予文件事实权威。

### 5.5 动态记忆拼接规则

任务启动后的首轮 task packet 必须包含一个明确的动态记忆层：

```text
Task start inherited dynamic memory
  source: turn_to_task_context_handoff
  cache_scope: none
  cache_role: volatile
  compression_role: preserve/ref_only
```

它应包含：

- `memory_runtime_view_ref`
- `context_package_ref`
- `selected_sections`
- 已过滤的 `model_visible_sections`
- 与任务合同相关的启动摘要
- 当前 turn 已选 memory 的 read namespaces / requested layers 摘要

它不应进入稳定 task prefix，因为这是本轮动态选择结果。后续任务执行可以通过 memory provider 重新读取状态，但首轮必须先看到启动时已选记忆，避免任务刚启动就失忆。

### 5.6 Prompt/cache 分层

本次重构采用以下分层：

| 内容 | 目标段 | cache 策略 |
| --- | --- | --- |
| 全局运行协议、action schema | stable prefix | global/session stable |
| task contract、artifact scope、tool index | task stable prefix | task/session stable |
| project instructions | scoped stable | session stable |
| task start inherited dynamic memory | volatile suffix | none / volatile |
| turn-to-task inherited observations | volatile suffix | none / volatile，首版不进入 stable replay |
| lifecycle static core | stable prefix | 只允许稳定 invocation/capability 触发 |
| lifecycle runtime guidance | volatile suffix | 由 memory/observations/steer/recovery 等动态状态触发 |
| execution control state | volatile suffix | none / volatile |
| exact read evidence injection | volatile suffix | only if no duplication |

6 月 17 日的 stable/volatile 判定思想可以采用，但不能让缓存优化改变上下文语义。模型应看到清晰的一份当前状态，而不是互相冲突的 replay/cursor/secondary context。

### 5.7 Prompt cache 硬边界

以下内容不得进入 stable prefix：

- `TurnToTaskContextHandoff` 的正文。
- inherited memory 的 `model_visible_sections`、`selected_sections` 正文。
- inherited observations 的工具输出正文、读文件窗口、搜索命中、失败详情。
- `turn_id`、`task_run_id`、`request_id`、`invocation_index`、timestamp、executor status 等运行态字段。
- 由 memory 是否命中、observations 是否存在、pending steer 是否存在等动态状态决定的 lifecycle refs。

允许进入 stable prefix 的只有：

- 固定协议、固定 action schema、固定环境规则。
- 项目指令和环境 prompt 等在当前 session 内字节稳定的内容。
- task contract、artifact scope、tool catalog 等 task 身份/契约层内容。
- 与 handoff 相关的空结构位声明或静态说明，例如“task start context 会作为 volatile 输入出现”，但不得包含本次 handoff 的实际事实。

如果需要把某类 inherited observation 做成可缓存 append-only replay，必须另开设计并满足全部条件：

- 事件顺序来自不可变 event offset。
- entry identity 内容寻址，不能含 request/turn/task 运行态字段。
- 旧 entry 只追加不重排、不原地改写。
- 与 `volatile_task_state` 不重复表达同一事实。
- 有 prompt accounting 实测证明不会切裂早期 stable prefix。

本计划首版不启用 handoff stable replay；所有 handoff 动态事实默认进入 volatile suffix。

## 6. 模块计划

### 6.1 `single_agent_turn.py`

目标：

- 将 `StartTaskFromActionRequest` 签名升级为接收 action_request 与 handoff seed。
- 在 `request_task_run` 分支前构造 handoff seed。
- handoff seed 从本轮 `tool_observation_payloads`、当前 packet_ref、session_context、api protocol prefix、runtime observations 中提取。

完成条件：

- 普通 turn 读文件后再 request_task_run，handoff seed 中有 observation refs。
- 不把 hidden model response 或私有推理写入 handoff。

### 6.2 `runtime_facade.py`

目标：

- `start_task(action_request, handoff_seed)` 调用 task lifecycle。
- 把当前 `session_context["memory_context"]`、turn_input_facts、editor_context、environment snapshot 纳入 handoff。

## 10. 迁移与切换规则

- 新 handoff 是主路径，不作为旧链路兼容 fallback。
- 没有 handoff 的旧 task_run 只能按既有 task scope 执行，不从 session scope 猜上下文。
- 新建任务必须有 handoff 诊断；如果无可继承内容，也写入空 handoff，说明 `empty_reason`。
- 文件证据 scope materialization 必须有 event 与 authority。
- prompt manifest 必须能说明 inherited dynamic memory 在 volatile suffix。
- prompt cache key / stable prefix hash 不得包含 handoff 动态事实；只能通过 volatile segment 和 diagnostics 追踪 handoff。
- lifecycle selector 如果因动态状态新增 guidance，该 guidance 必须进入 volatile，或者该 slot 被改为同 invocation 固定 stable core。

回滚条件：

- handoff 导致 task_run 创建失败。
- 文件证据迁移污染无关 task_run scope。
- memory context 重复注入导致 prompt 超限或覆盖当前 task observation。
- lifecycle 分流破坏 action schema、模型输出协议或导致 stable prefix hash 随动态状态抖动。

## 11. 禁止捷径

- 禁止直接回档 `compiler.py`、`task_executor.py`、`manager.py`。
- 禁止用 prompt 文字代替结构 handoff。
- 禁止把 session scope 作为 task execution 的永久隐式 fallback。
- 禁止把 editor context 当作工具读文件事实。
- 禁止保留旧 task contract 字段作为兼容入口。
- 禁止通过删失败测试、skip、降低断言制造通过。
- 禁止破坏已恢复的公开投影、缓存系统、产物系统和前端。

## 12. 预期结果

完成后，任务启动链路应变为：

```text
普通 turn 动态记忆 / 文件证据 / 工具观察
-> TurnToTaskContextHandoff
-> task_run diagnostics + runtime object + event
-> task execution memory provider
-> DynamicContextInput.inherited_start_context
-> RuntimeCompiler volatile inherited dynamic memory segment + volatile runtime lifecycle guidance
-> model_messages
```

模型看到的是一条连续链路：任务启动前已经读到和选中的上下文不会突然消失；任务启动后新增的 task observations 会接管并更新事实；持久投影、文件状态 store 和 memory provider 作为明确来源参与装配，而不会越权覆盖当前 turn 事实。

同时，缓存边界保持清晰：全局协议、环境 core、task contract、tool index 等稳定段继续构成 provider cache 前缀；handoff memory、handoff observations、execution control、动态 lifecycle guidance 位于 volatile suffix，不改变 stable prefix hash。

日期：2026-06-17  
状态：待用户确认后执行  
目标架构名：Chronological Projection 收束版  
范围：公开投影合同、session restore、前端 chronological reducer、显示 view model、store 可见刷新、ChatMessage / PublicTimelineActivity、相关测试与旧链路删除

## 1. 目标

本计划用于把当前已经推进到一半的 Chronological Projection 重构收口成成熟 agent 级别的主聊天投影系统。

最终系统必须满足：

```text
一个事件时钟
一个 projection key
一个 frame identity
一个投影折叠权威
一个显示 view model
一个 UI 消费面
```

用户可见结果必须满足：

1. 模型正文、工具请求、权限、工具开始、工具完成、状态反馈、commit、terminal 按同一时序投影。
2. 正文不重复，工具不串线，刷新不重造语义事件。
3. 工具、权限、错误、预算、terminal 不被正文节流延迟。
4. 完成后主聊天收口干净，完整轨迹进入日志或 trace。
5. 长文本流式输出顺滑，不能每个小 frame 触发整页重算。
6. 输出视觉保持专业、克制、可扫描，不暴露 runtime 噪声。

## 3. 成熟架构标准

本项目是 agent 系统，投影链路不能只按 UI 组件思路修。成熟标准应接近 Codex / Claude Code 这类 coding agent 的外显工程能力：

```text
Runtime/Public Event Log
-> Public Projection Contract
-> Chronological Projection Slice
-> Frontend Projection Normalizer
-> Projection Accumulator
-> Projection ViewModel
-> Store Adapter / Visible Flush
-> ChatMessageView
```

权威边界：

| 层 | 唯一职责 | 禁止 |
| --- | --- | --- |
| Runtime/Public Event Log | 记录事实事件、事件 offset、事件身份 | 不决定 UI 样式 |
| Projection Contract | 把 public event 标准化成 frame | 不从 raw runtime tail 重新发明主聊天事件 |
| Projection Slice | 为 restore/reconnect 提供 bounded slice | 不夹带弱合同或旧 projection 字段 |
| Frontend Normalizer | 校验 frame、生成 `ProjectionKey` 与 `FrameIdentity` | 不决定显示文案 |
| Accumulator | 按 event clock 幂等折叠 body/tool/status/commit/terminal | 不按到达顺序覆盖终态 |
| ViewModel | 输出 display mode 和 render blocks | 不访问 React/store |
| Store Adapter | 绑定 message snapshot 与 active projection，并控制可见刷新 | 不解释 frame 内容 |
| ChatMessageView | 纯渲染 view blocks | 不读 raw trace/timeline/status 推断协议 |

## 5. 遗漏与冲突复核

### 5.1 已发现遗漏

| 遗漏 | 说明 | 计划处理 |
| --- | --- | --- |
| ProjectionSlice 没有强 schema | 当前 slice 缺 `schema_version="chronological_projection"`、`event_log_id`、`start_offset`、`end_offset` | Phase 1 增加合同和前后端类型测试 |
| task/turn raw reproject 没退出主聊天 | `_task_projection_frames()`、`_turn_projection_frames()` 仍可生成主聊天 frames | Phase 2 删除主路径，只保留 debug/log trace |
| out-of-order lifecycle 缺测试 | 当前 tool merge 可能受到达顺序影响 | Phase 3 增加乱序、重复、missing start 测试 |
| commit_failed / error / stopped 可见策略需要收束 | 当前未提交差异把多类状态改 hidden trace，可能弱化用户反馈 | Phase 3 固定为主聊天 typed recovery/terminal block + log entry，不写 assistant body |
| active projection store 未落地 | 仍在 `messages[]` 上存 `chronologicalProjectionLedger` 和 `projectionView` | Phase 5 引入 `activeProjectionsByKey` |
| UI blocks 仍转旧 timeline item | `projectionActivityItemsFromBlocks()` 说明 typed blocks 尚未直接渲染 | Phase 6 改为 typed block 组件 |
| fixed port 实测未覆盖 | 当前仅有单元测试和静态审计证据 | Phase 8 固定 `3000/8003` 真启动验收 |

### 5.2 已发现冲突

| 冲突 | 两边诉求 | 决议 |
| --- | --- | --- |
| 隐藏 status 是否创建空 assistant 投影 | 当前失败测试有的期望空 view，有的实现直接不创建 projection | 统一为：不会创建新 assistant；若已有 active projection，可记录内部 diagnostic，但不生成可见 block |
| runtime status / active steer 是否应给用户反馈 | 用户需要知道系统接收了信号；但不能由系统伪装 agent 正文 | 反馈放在 session activity 或 typed `status_event`，由 ViewModel 明确模式控制，不写 assistant body |
| commit_failed 是否 hidden | 隐藏可避免噪声；但写回失败是用户可见风险 | 固定为主聊天 typed recovery/terminal block，同时给 log entry；不能吞掉，也不能作为模型正文 |
| active_task_steer_accepted 是否只在 SessionActivityBar | 只放 footer 太轻，刷新/重连后也无法作为时序信号被证明 | 固定为轻量 `status_event`，进入同一个 chronological projection；SessionActivityBar 可同步显示，但不是唯一事实源 |
| `mainChatSurface` 与 `projectionView.displayMode` | runtime attachment 还 patch surface；ViewModel 也输出 displayMode | displayMode 成为唯一显示裁决；旧字段只在迁移期作为输入，不再由组件裁决 |
| monitor live shell 与 projection message | monitor 可创建 assistant shell；projection reducer 也可 ensure message | monitor 不再独立创建空 assistant shell；必须通过 projection key binding 请求/绑定 shell |
| PublicTimelineActivity 旧合同与 typed blocks | 旧组件成熟美观；但旧类型会保留协议解释 | 短期保留视觉样式，接口改为 typed block props，删除旧 item adapter |
| 流畅度调度与控制信号即时性 | body 需要合并刷新；工具/权限/error/terminal 需要立即出现 | store adapter 中按 frame kind 调度，body defer，control immediate |

### 5.3 与项目规则的冲突检查

### Phase 3：前端 accumulator 严格时序与终态语义

目标：

`ProjectionAccumulator` 允许乱序到达，但最终 view 必须按 `event_offset -> frame_id` 得到同一结果。

1. 把 `ToolLifecycle` 的状态合并改为 offset-aware。
2. 建立 lifecycle precedence：

```text
requested < permission_decided < started < completed/failed/stopped
```

3. `mergeTool()` 不允许低 offset frame 覆盖高 offset 终态。
4. `commit_failed`、`error`、`stopped`、`turn_terminal` 明确为 typed recovery/terminal/log block。
5. hidden / trace-only / non-model body 的规则统一：
   - 不创建新 assistant message。
   - 已有 active projection 可记录 bounded diagnostic。
   - 不进入 visible body/tool blocks。
6. 删除基于文本相等的主去重，只保留 `FrameIdentity`、semantic identity、tool id。

涉及文件：

- `frontend/src/lib/projection/chronological/accumulator.ts`

### Phase 5：active projection store 与可见刷新收窄

目标：

高频 live projection 不再每 frame 替换 `messages[]`。

目标 store：

```text
messagesById
messageOrder
activeProjectionsByKey
messageProjectionKeyById
```

实施策略：

1. 先在现有 store 旁增加 `activeProjectionsByKey`。
2. `applyProjectionFrame()` 只更新 active projection state。
3. committed 后冻结 `MessageSnapshot`，释放或压缩 live ledger。
4. `body_append` 按 rAF / 33-50ms 可见 flush。
5. `tool/permission/error/terminal/commit` immediate flush。
6. `ChatPanel` selector 只订阅消息 order 和当前 active projection view。

涉及文件：

- `frontend/src/lib/store/types.ts`
- `frontend/src/lib/store/runtime.ts`

- `frontend/src/lib/store/hooks.ts`

完成标准：

- 纯正文流式期间不再每个 frame 复制完整 `messages[]`。
- 已完成消息不会因为 active projection 继续重渲染。

## 10. 计划自检结论

已按以下维度复核：

| 维度 | 结论 |
| --- | --- |
| 信号完整性 | body、tool、permission、status、commit、terminal、hydrate、flush 均已列入 |
| 权威单一性 | 明确 ViewModel 是显示裁决，public ledger 是事实源 |
| 旧链路删除 | Phase 2、Phase 4、Phase 7 明确删除路径 |
| 流畅度 | Phase 5 保留现有 rAF 思路，并升级为 active projection 局部刷新 |
| 美观 | Phase 6 保留当前克制工具轨迹风格，要求 typed blocks 直接渲染 |
| 测试真实性 | 不跳过失败测试，要求按目标行为改写或保留 |
| 项目规则 | 本计划需用户确认后实施 |

已定设计裁决：

1. `commit_failed/error/stopped` 统一进入主聊天 typed `recovery_event` / `terminal_event` block，并同时提供 log entry。它们是用户必须知道的终态/失败信号，不能只放在 session activity 或日志里；但绝不写入 assistant body，也不伪装成模型回复。
2. `active_task_steer_accepted` 显示为轻量 `status_event`，进入同一个 chronological projection。它代表“用户补充要求已接入当前工作”的时序事实，必须可 replay、可证明；SessionActivityBar 可以同步展示，但不能成为唯一事实源。
3. monitor live shell 不允许继续独立创建空 assistant shell。monitor 只能通过 projection key binding 请求或绑定 live shell，显示模式由 `ProjectionView.displayMode` 裁决，避免 monitor 与 projection reducer 形成双权威。

用户确认本文整体执行后，应按 Phase 0 到 Phase 8 连续实施。若执行中发现 slice 数据无法覆盖现有 frame，必须暂停并提交偏差说明，不能以旧 raw reproject 兜底。

# 编排系统前端工作台重构计划书

日期：2026-06-09

## 范围边界

本计划只重构编排系统前端，不修改任务系统业务代码。

允许改动范围：

- `frontend/src/components/workspace/views/OrchestrationView.tsx`
- `frontend/src/components/workspace/views/orchestration/*`
- `frontend/src/app/globals.css` 中编排系统相关样式

禁止改动范围：

- `frontend/src/components/workspace/views/task-system/*` 的业务实现
- 任务系统 API 语义
- 后端 runtime / task system / orchestration API 合同

## 当前问题

编排系统页面现在由 `OrchestrationView.tsx` 同时承担数据加载、目录选择、Agent 草稿、运行档案草稿、Agent 组草稿、二级页面导航、保存删除动作和页面渲染。页面能运行，但决策权过于集中，导致后续扩展时容易把 Agent、Agent 组、Runtime Profile 和诊断页面继续混在一层。

已经存在的后端事实源是清晰的：

- Agent 名册来自 `/orchestration/agents`
- Runtime 选项来自 `/orchestration/runtime-options`
- 能力准入项来自 `/orchestration/capability-items`
- Agent 组只承载组身份、协调者和成员关系
- Agent Runtime Profile 承载工具、操作、模型、上下文、记忆和协作策略

前端重构的核心不是重写后端合同，而是把前端的页面层级和状态所有权对齐这些事实源。

## 目标架构

编排系统前端目标结构为独立的 Agent Assembly Workbench：

```text
OrchestrationView
-> useOrchestrationAssemblyController
-> OrchestrationAssemblyShell
   -> OrchestrationDirectoryRail
   -> OrchestrationLayerNav
   -> OrchestrationAgentAssemblyPages
      -> Agent pages
      -> Group page
```

### 权限与职责分配

| 模块 | 职责 | 禁止事项 |
| --- | --- | --- |
| `OrchestrationView` | 页面入口、确认弹窗接线、组合 shell | 不持有大量业务推导 |
| `useOrchestrationAssemblyController` | 加载 catalog、维护选择状态、维护草稿、执行保存删除 | 不渲染页面 |
| `orchestrationAssemblyModel` | 纯函数、常量、类型、派生规则 | 不请求 API、不访问浏览器状态 |
| `OrchestrationAssemblyShell` | 顶部摘要、通知、工作台主结构 | 不决定 Agent/Group 业务状态 |
| `OrchestrationDirectoryRail` | 展示目录和触发选择 | 不自行推导事实源 |
| `OrchestrationAgentAssemblyPages` | 根据当前对象和层级渲染对应页面 | 不重新加载 catalog |
| `OrchestrationWorkbenchUi` | 编排专属 UI 原语 | 不复制局部 Button/Badge/Field |

## 页面层级

Agent 对象页面：

- 身份
- 权限
- 运行配置
- 模型
- 上下文
- 协作
- 总览
- 诊断

Agent 组对象页面：

- 组身份
- 协调者
- 成员加入/移出

组页面不显示单个 Agent 的运行权限。Agent 页面不直接编辑组成员关系。

## 实施计划

### 阶段 1：沉淀文档与模型层

- 新增本计划文档
- 新增 `orchestrationAssemblyModel.ts`
- 把 Agent 分类、目录分区、草稿转换、运行权限计算等纯逻辑移入模型层
- 新增模型层测试

完成标准：

- 纯函数可单独测试
- 页面渲染不依赖重复的散落 helper

### 阶段 2：抽离控制器

- 新增 `useOrchestrationAssemblyController.ts`
- 迁移 catalog 加载、runtime options 合并、capability items 懒加载、保存删除动作
- 迁移选择状态：active section、active layer、selected agent/group、draft mode

完成标准：

- `OrchestrationView` 不再直接持有大量业务状态
- API 调用路径保持不变

### 阶段 3：建立编排专属 Shell

- 新增 `OrchestrationAssemblyShell.tsx`
- 新增 `OrchestrationLayerNav.tsx`
- 新增 `OrchestrationAgentAssemblyPages.tsx`
- 页面布局对齐现有系统工作台结构，但不复用或修改任务系统业务组件

完成标准：

- 顶部摘要、左侧目录、二级导航、右侧工作区边界明确
- Agent 和 Group 页面互不污染

### 阶段 4：清理 UI 原语和旧记录

- 统一使用 `OrchestrationWorkbenchUi.tsx`
- 删除 `OrchestrationGroupWorkbench.tsx` 内重复的本地 Badge/Button/Field
- 删除旧的 `AGENT_ASSEMBLY_REFACTOR_PLAN.md`
- 收口编排相关 CSS，保留必要的兼容 class，清理无效重复

完成标准：

- 编排 UI 原语只有一套
- 旧计划记录不再和新计划并存

### 阶段 5：验证

- 运行新增/相关前端测试
- 运行 TypeScript/构建或等效检查
- 运行 `git diff --check`
- 如启动链路受影响，使用固定端口 `3000/8003` 实测页面

## 风险控制

- 不修改任务系统业务代码
- 不修改后端 API 合同
- 不把旧结构包一层继续保留；迁移完成后删除旧重复逻辑
- 如果测试暴露 API 类型或页面交互假设错误，优先修正模型和控制器边界，不做页面补丁

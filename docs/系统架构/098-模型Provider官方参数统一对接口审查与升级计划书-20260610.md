# 模型 Provider 官方参数统一对接口审查与升级计划书

日期：2026-06-10

## 1. 问题定义

当前阻塞不是单纯的模型偶发 JSON 格式错误，而是模型官方参数在系统内没有统一对接口。

以 DeepSeek JSON Output 为例，官方要求请求中发送：

```json
{"response_format":{"type":"json_object"}}
```

并且提示词中需要包含 `json` 语义。但我们当前链路中：

- `AgentModelProfile` 和 `ResolvedModelSpec` 已经有 `response_format` 字段。
- `runtime.model_gateway.provider_payload` 已经能把 `response_format` 作为 provider payload 边界和缓存诊断参数。
- `runtime.model_gateway.model_runtime.ModelSpec` 没有 `response_format` 字段。
- `harness.loop.model_action_runtime.normalize_model_selection_for_invocation()` 会过滤掉 `response_format`。
- `ModelRuntime._build_chat_model_for_spec()` 没有统一下发 `response_format`、provider-specific body、tool strict 等官方参数。
- `ModelRuntime._begin_prompt_accounting()` 只把 `_cache_relevant_params_for_spec()` 交给 `ModelRequestBuilder`，但该参数集没有覆盖完整 provider 官方参数。

所以现状是：配置层、执行层、账本层各自懂一部分，真实请求没有一个统一权威。

正确终态：模型对接必须成为独立模块，任何模型官方参数都只能通过统一的 `ProviderRequestProfile -> ProviderAdapter -> ModelRuntime` 对接口进入请求；同时同一份规范必须进入缓存边界、诊断和回归测试，不能散落在 `completion_profile`、`stream_policy`、`model_kwargs`、`extra_body` 的临时分支中。

这个模块不属于图任务、不属于单 agent loop，也不属于某一个 provider。它是 runtime 的模型供应商接入层，负责把系统内部的模型请求契约翻译成不同厂商的真实 API 参数。

## 2. 官方能力对照

### DeepSeek JSON Output

官方能力：`response_format={"type":"json_object"}`。

适用场景：

- model action JSON。
- 语义压缩 JSON。
- 审核裁决 JSON。
- 所有要求结构化 JSON object 的非创作型节点。

不适用场景：

- 章节正文、设定正文、剧情正文等自然语言创作输出。

当前缺口：

- `response_format` 在 profile 中存在，但没有完整进入 `ModelSpec`、LangChain model kwargs 和 provider payload accounting。

### DeepSeek Tool Calls / Strict Tool Calling

官方能力：tool calls 支持工具 schema；strict 模式在 beta 路径下使用，函数 schema 可声明 `strict:true`。

适用场景：

- 单 agent 工具调用。
- 多 agent 图节点需要调用工具时。
- 未来边协议中 A2A 或资源库工具化调用。

当前缺口：

- 我们已有 `tool_call_options` 和工具绑定路径，但 provider-specific strict/tool-choice 参数没有统一模型。
- beta base_url 与 strict schema 目前不应由散落逻辑拼接，应由 provider adapter 根据 request profile 统一决定。

### DeepSeek Thinking Mode / Reasoning

官方能力：通过 DeepSeek thinking/reasoning 参数控制推理；历史 assistant tool-call replay 需要保留 `reasoning_content`。

当前状态：

- `ModelRuntime` 已有 `_DeepSeekReasoningCompatChatModel`，能在请求 payload 中保留 `reasoning_content` 和 `prefix`。
- `thinking_mode`、`reasoning_effort` 已在配置与请求构造中存在。

当前缺口：

- thinking/reasoning 与其他 provider 参数没有统一 profile 层级。
- DeepSeek 专属 `extra_body.thinking` 与 OpenAI 兼容 `reasoning_effort` 的分流写在 `ModelRuntime._build_chat_model_for_spec()` 内，扩展新 provider 时会继续膨胀。

### DeepSeek Chat Prefix Completion

官方能力：beta endpoint 下支持 prefix completion。

当前状态：

- `completion_profile.mode=chat_prefix` 和 `provider_mode=deepseek_chat_prefix` 会触发 `/beta` endpoint 和 assistant `prefix` 消息处理。

当前缺口：

- 它被放在 `completion_profile`，和 provider 官方参数没有同一个请求 profile。
- `_copy_spec_with_base_url()`、`_copy_model_spec()` 等复制函数没有统一字段清单，后续加字段容易丢。

### 通用 OpenAI-compatible 参数

应统一纳入接口的参数族：

- 模型身份：`provider`、`model`、`base_url`、`credential_ref/api_key`。
- 输出控制：`max_tokens/max_completion_tokens/max_output_tokens`、`temperature`、`top_p`、`presence_penalty`、`frequency_penalty`。
- 结构化输出：`response_format`。
- 工具调用：`tools`、`tool_choice`、`parallel_tool_calls`、`strict`。
- 推理控制：`thinking_mode`、`reasoning_effort`、provider-specific reasoning body。
- 流式策略：`stream_policy`。
- provider 扩展：`extra_body`、`model_kwargs`、`default_headers`、`default_query`，但必须经过白名单和敏感字段过滤。

- `backend/runtime/model_gateway/model_response.py`

- `ModelResponseRuntimeExecutor` 和 `task_executor` 都会传 `model_spec`，但它们没有统一声明本次请求需要什么官方参数。

### 请求构造层

- `backend/runtime/model_gateway/model_runtime.py`
- `backend/runtime/model_gateway/provider_payload.py`
- `backend/runtime/model_gateway/model_request.py`

问题：

- `provider_payload.py` 能识别 `response_format` 和 `tool_call_options`，但 `ModelRuntime._cache_relevant_params_for_spec()` 没有完整输出 provider request params。
- `ModelRuntime._build_chat_model_for_spec()` 直接构造 LangChain ChatDeepSeek/ChatOpenAI 参数，provider 分支继续增长会不可控。
- DeepSeek JSON mode、strict tool mode、prefix beta endpoint 分散在不同概念里。

### 协议层

相关文件：

- `backend/harness/loop/task_executor.py`
- `backend/harness/loop/single_agent_turn.py`
- `backend/runtime/model_gateway/model_response_protocol.py`

问题：

- `require_json_action=True` 只约束解析，不会自动生成 provider 请求参数。
- 解析容错可以处理“完整 JSON 后多一个引号”的 provider 输出问题，但不能代替官方 JSON mode。
- 当前动作协议失败会产生 `model_action_protocol_repair_required`，但缺少“本次请求是否启用了官方 JSON mode”的诊断证据。

## 4. 推荐设计方向

新增独立的模型对接模块：`runtime.model_gateway.providers`。

建议目录：

```text
backend/runtime/model_gateway/providers/
  __init__.py
  models.py
  registry.py
  base.py
  openai_compatible.py
  deepseek.py
  qwen.py
  ollama.py
```

模块职责：

- `models.py`：定义统一请求 profile、adapter 输出、官方能力枚举。
- `base.py`：定义 provider adapter 协议。
- `registry.py`：按 provider 选择 adapter。
- `openai_compatible.py`：沉淀 OpenAI-compatible 通用参数。
- `deepseek.py`：只放 DeepSeek 专属映射，如 thinking、chat prefix、strict beta endpoint。
- `qwen.py`、`ollama.py`：后续接入时独立扩展，不再修改主 runtime。

核心对象是 `ProviderRequestProfile` 和 `ProviderAdapter`。

### ProviderRequestProfile

这是系统内部唯一的模型官方参数配置对象，来源可以是系统配置、AgentModelProfile、节点契约、调用场景策略，但进入模型前必须合并成这一份。

建议字段：

```python
ProviderRequestProfile(
    provider="deepseek",
    model="deepseek-v4-flash",
    base_url="https://api.deepseek.com",
    credential_ref="provider:deepseek:primary",
    max_output_tokens=4096,
    timeout_seconds=45,
    long_output_timeout_seconds=180,
    temperature=0,
    top_p=None,
    presence_penalty=None,
    frequency_penalty=None,
    response_format={"type": "json_object"},
    tool_call_options={...},
    strict_tool_schema=False,
    thinking_mode="disabled",
    reasoning_effort="auto",
    stream_policy={"enabled": False},
    completion_profile={"mode": "chat_prefix", "provider_mode": "deepseek_chat_prefix"},
    provider_extensions={...},
)
```

### ProviderAdapter

职责：

- 接收 `ProviderRequestProfile`。
- 输出三份一致结果：
  - `langchain_model_kwargs`
  - `request_params_for_accounting`
  - `diagnostics`
- 处理 provider-specific 映射：
  - DeepSeek JSON Output -> `model_kwargs.response_format`
  - DeepSeek Thinking -> `extra_body.thinking`
  - DeepSeek Chat Prefix -> beta base_url + assistant prefix 消息规则
  - DeepSeek strict tools -> beta base_url + tool schema strict
  - OpenAI-compatible JSON Output -> `model_kwargs.response_format`

禁止：

- 业务 executor 直接写 `extra_body`。
- compiler 直接写 LangChain 参数。
- profile resolver 直接知道 LangChain 构造细节。
- 新 provider 直接改 `ModelRuntime._build_chat_model_for_spec()` 主体。

### ModelRuntime 的新边界

`ModelRuntime` 只保留通用生命周期职责：

- 选择候选模型。
- 执行重试和 fallback。
- 创建 LangChain chat model。
- 记录 prompt accounting。
- 映射 provider error。

它不再直接理解 DeepSeek JSON mode、DeepSeek prefix、strict tool schema、provider-specific body。所有厂商差异都必须来自 provider adapter。

### 配置入口的边界

模型对接模块需要接住所有配置入口，但不能让配置入口直接理解厂商 API：

- 系统设置：`backend/config.py`、`backend/bootstrap/settings.py` 只负责 provider 目录、默认模型、默认 base_url、credential ref。
- Agent 配置：`AgentModelProfile` 只表达模型能力需求和通用输出策略。
- 节点契约：`ModelRequirement` 只表达节点需要的能力，例如结构化 JSON、禁用流式、最小输出 token。
- 图实例 override：`harness.graph.model_overrides` 只允许覆盖统一 request profile 字段，禁止塞 LangChain 私有参数。
- runtime executor：`task_executor`、`single_agent_turn`、`model_response` 只声明本次调用契约，例如 `structured_output=json_object`，不直接写 `response_format` 或 `extra_body`。

最终所有入口都合并成同一个 `ProviderRequestProfile`，再交给 adapter。

## 5. 分阶段升级计划

### 阶段一：参数权威收口

目标：

- 创建 `backend/runtime/model_gateway/providers/` 模块。
- 定义 `ProviderRequestProfile`、`ProviderRequestAdapterResult`、`ProviderFeaturePolicy`。
- 让 `ResolvedModelSpec`、`ModelRuntime.ModelSpec` 字段对齐。
- `ModelProfileResolver` 正确传递 `response_format`。
- `normalize_model_selection_for_invocation()` 不再丢掉官方参数。
- `harness.graph.model_overrides`、`harness.graph.models._runtime_only_structure_key()`、`task_executor._task_model_selection()` 都能保留统一 request profile 字段。

完成标准：

- `response_format` 从 `AgentModelProfile` 到 `ModelRuntime.invoke_messages()` 不丢失。
- 复制函数不再手写散字段，或至少统一使用一个字段清单常量。

### 阶段二：Provider adapter 接管请求构造

目标：

- `ModelRuntime._build_chat_model_for_spec()` 改为调用 adapter。
- adapter 生成 DeepSeek/OpenAI-compatible 的 LangChain kwargs。
- `_cache_relevant_params_for_spec()` 使用 adapter 输出的 `request_params_for_accounting`。
- `ModelRequestBuilder` 和 `provider_payload.py` 能看到真实请求参数。
- `_chat_model_pool_key()` 必须纳入 adapter 输出中会影响真实请求的字段，例如 `response_format`、`extra_body`、`model_kwargs`、`default_headers`，防止不同配置复用同一个 LangChain client。
- `_copy_model_spec()`、`_copy_spec_with_base_url()`、`_model_spec_for_stream_policy()` 必须使用统一字段清单，不能继续手写散字段。

完成标准：

- 请求真实下发什么，账本和缓存边界就记录什么。
- prompt cache 诊断中 `response_format_hash`、`tool_call_options_hash` 能反映实际请求变化。

### 阶段三：协议场景自动请求 profile

目标：

- `model_action_request_json`、semantic compaction、审核 verdict 等结构化协议自动声明 `response_format={"type":"json_object"}`。
- 创作正文节点不自动启用 JSON mode。
- `task_executor` 和 `single_agent_turn` 不再各自猜 provider 参数，而是声明 `request_contract={"structured_output":"json_object"}`，由 adapter 解析。
- compiler 的 `output_policy.format=model_action_request_json` 必须成为 request contract 的来源之一。
- 对 DeepSeek JSON Output，协议 prompt 中必须稳定出现 `json` 语义；该要求由结构化协议模板提供，而不是靠节点 prompt 临时补字。

完成标准：

- `require_json_action=True` 的请求同时具备解析约束和 provider JSON mode。
- 阻塞诊断中明确记录 `provider_request_profile.response_format`。

### 阶段四：Tool Calls strict 与 beta endpoint 统一

目标：

- 统一工具调用参数：`tool_choice`、`parallel_tool_calls`、`strict_tool_schema`。
- DeepSeek strict tool calling 的 beta endpoint 由 adapter 决定。
- 工具 schema strict 不由工具定义散落决定，而由 provider request profile 与 tool catalog manifest 共同生成。

完成标准：

- strict tool calling 有独立回归测试。
- 非 strict 工具链不受影响。

### 阶段四点五：Context budget 与 Provider capability 对齐

目标：

- `harness.runtime.context_budget_policy` 不再硬编码 DeepSeek 1M 模型列表，而是从 provider capability catalog 读取。
- provider adapter 暴露 `context_window_tokens`、`max_output_tokens_limit`、`supports_json_output`、`supports_strict_tools`、`supports_chat_prefix` 等能力。
- `LLM_PROVIDER_DEFAULTS` 只保留默认配置，不承担完整能力目录职责。

完成标准：

- 新 provider 接入时，不需要到 context budget 模块新增硬编码模型名。
- 如果节点要求 JSON mode 但 provider 不支持，编译或运行前给出明确配置错误，而不是等模型输出失败。

### 阶段五：图任务运行验证

目标：

- 修复当前 `baseline_memory_seed` 的 JSON action 阻塞。
- 继续跑写作图任务，目标验证到第 150 章。
- 观察 provider_unavailable / Connection error 是否下降，并记录是否与 provider 参数无关。

完成标准：

- 当前阻塞节点可重试通过。
- 章节正文生成不启用 JSON mode。
- 图任务 scope 仍保持 `workspace_view=graph_task`，`task_environment_id=""`。

## 8. 实施禁区

- 不改图拓扑。
- 不把图任务绑定环境；环境只用于 agent 装配。
- 不用 parser 容错代替官方 JSON mode。
- 不把 JSON mode 打到章节正文生成节点。
- 不新增 provider 参数散落分支。
- 不为了测试 mock 掉核心模型请求路径。
- 不保留旧的重复权威；字段一旦统一，旧散字段读取路径要清理。
- 不把 provider capability 继续硬编码在 context budget 或图 override 白名单里。
- 不允许 `provider_extensions` 透传敏感字段、未知顶层 HTTP 参数或绕过 adapter 白名单。

## 9. 当前结论

当前系统已经有若干成熟部件，但缺少统一 Provider 对接口。最严重的断点是 `response_format`：配置模型已有，缓存账本已有，但 runtime 请求没有完整接通。这正好解释了 DeepSeek 返回“几乎正确 JSON 但尾部多引号”时系统容易阻塞。

建议先按本计划完成 provider request profile 收口，再重启并重试当前写作图任务。这样既能解决当前 JSON action 阻塞，也能避免后续 strict tool、thinking、prefix、cache 参数继续散落。

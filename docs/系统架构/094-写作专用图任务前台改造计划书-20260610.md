# 094-写作专用图任务前台改造计划书

日期：2026-06-10  
状态：第二阶段写作投影和独立 chapter action API 已落地，第三阶段章节生产闭环继续推进

## 1. 核心结论

图任务底层继续保持通用，但前台主入口应该优先做成写作生产台，而不是通用图监视台。

原因很明确：写作图任务的主要用户目标不是观察拓扑，而是连续生产正文。用户最常做的动作是：

```text
看当前章节
  -> 判断是否可用
  -> 通过 / 退稿 / 自己改写
  -> 推进下一章
  -> 管理正文、设定、大纲、角色等写作资产
```

所以前台默认视角应该是章节生产，而不是节点运行。

目标形态：

```text
写作项目
  -> 章节台
  -> 正文阅读/编辑
  -> 章节审核动作
  -> 写作资产管理
  -> 图调试/运行监控作为二级视图
```

重要边界：

- 不能把底层 GraphLoop 改成写作专用。
- 不能让前端直接推进图状态。
- 写作页面只是图任务实例的一种产品化投影。
- 通用 `HumanEdgeDecision` 仍然是人工通过、退稿、替写的唯一运行入口。
- 通用图监控保留为调试/排障视图，但不再作为写作用户的默认主页面。

## 2. 当前问题

现在图任务前台已经具备项目、实例、节点、会话、文件和人工边传播控制，但页面重心仍然偏工程监控。

主要问题：

- 正文阅读没有成为屏幕中心。
- 用户需要理解节点、边、产物、正式库等系统概念，才能完成写作推进。
- 人工传播动作仍然像通用协议按钮，而不是写作审核动作。
- 文件管理和正文阅读混在一起，写作资产没有领域化。
- 图监控占据默认心智，但对写作生产来说只是后台细节。

这会导致用户虽然能操作，但不能顺畅地“一章一章推进”。

## 3. 目标用户体验

### 3.1 默认入口

点击“图任务”后，用户首先看到写作项目列表，而不是图拓扑。

项目列表重点显示：

- 项目名
- 当前章节
- 已完成章节数
- 当前状态：待启动、写作中、待审核、退稿中、已完成
- 最近产出
- 下一步动作

### 3.2 进入项目后的主页面

默认进入“章节台”。

页面结构：

```text
┌──────────────────────────────────────────────┐
│ 项目标题 / 当前任务 / 启动或继续按钮          │
├──────────┬──────────────────────┬────────────┤
│ 章节列表 │ 正文阅读/编辑主舞台   │ 当前章控制 │
│          │                      │ 节点会话   │
│          │                      │ 审核记录   │
│          │                      │ 运行产物   │
└──────────┴──────────────────────┴────────────┘
```

正文区必须是视觉重心：

- 默认阅读模式。
- 点击章节后直接打开正文阅读。
- 编辑是阅读区的一个切换状态，不是默认状态。
- 支持上一章、下一章。
- 支持阅读专注模式。

### 3.3 当前章审核动作

当前章正文下方或右侧固定审核条：

```text
通过本章
退稿给写手
采用我的改写稿
```

这些动作映射到底层通用边决策：

| 写作动作 | 通用决策 | 语义 |
| --- | --- | --- |
| 通过本章 | `pass` | 沿当前下游边传播当前章节 |
| 退稿给写手 | `revise` | 沿返修/反馈边回传修改意见 |
| 采用我的改写稿 | `replace` | 用户正文写入项目正式库，并作为 human-authored result 继续传播 |

前端不暴露 `pass/revise/replace` 这些协议词，除非进入调试模式。

## 4. 产品信息架构

### 4.1 一级视图

写作项目台包含四个一级视图：

1. 章节台
2. 写作资产
3. 运行记录
4. 图调试

默认是章节台。

### 4.2 章节台

章节台是主生产界面。

职责：

- 看正文。
- 编辑当前正文。
- 查看当前章上游/下游状态。
- 给写手反馈。
- 通过或退稿。
- 继续下一章。

不应该承担：

- 完整图配置。
- 大量运行日志。
- 原始边协议展示。
- 通用文件树管理。

### 4.3 写作资产

写作资产用领域分类，而不是纯文件目录。

建议分类：

- 正文
- 大纲
- 世界观
- 角色
- 设定
- 审稿记录
- 参考材料

底层仍然使用图任务实例文件空间：

```text
chapters/
outline/
world/
characters/
review/
input/
```

但前台显示为写作资产。

### 4.4 运行记录

运行记录用于看历史章节生成、节点会话和产物。

它服务于追溯，而不是默认生产。

### 4.5 图调试

图调试保留当前通用监控能力：

- 拓扑焦点
- 节点状态
- 边状态
- human controls 原始投影
- checkpoint / event

这个视图主要给高级用户和开发调试使用。

## 5. 后端设计

### 5.1 不新增第二运行权威

写作专用页面不能直接推进图。

目标链路必须保持：

```text
Writing UI Action
  -> WritingActionService 归一化为 HumanEdgeDecision 或实例运行请求
  -> HumanEdgeDecisionService / GraphHarness
  -> GraphLoop
  -> checkpoint / event / monitor projection
```

写作服务只做领域翻译和投影，不直接改 `GraphLoopState`。

### 5.2 新增写作投影服务

建议新增：

```text
backend/task_system/writing_graphs/
  projection_service.py
  chapter_index.py
  writing_asset_classifier.py
```

核心职责：

- 从图任务实例 file tree 中识别章节、大纲、设定、角色。
- 从 graph monitor 和 human_controls 中提取当前章状态。
- 把通用节点/边/产物转换为写作语言。
- 提供一个写作页面一次加载所需的数据包。

建议 API：

```text
GET /api/orchestration/writing-graph-instances/{instance_id}/desk
```

返回：

```json
{
  "instance": {},
  "chapter_index": [],
  "current_chapter": {},
  "reader": {},
  "writing_assets": {},
  "chapter_actions": [],
  "node_sessions": [],
  "artifacts": [],
  "graph_debug_ref": {}
}
```

### 5.3 写作动作 API

第一阶段可以先不新增独立动作 API，前端直接调用现有：

```text
POST /api/orchestration/graph-task-instances/{instance_id}/human-edge-decisions
```

第二阶段已增加写作语义包装：

```text
POST /api/orchestration/writing-graph-instances/{instance_id}/chapter-actions
```

请求：

```json
{
  "chapter_id": "chapter-001",
  "action": "approve | request_revision | replace_with_user_text",
  "instruction": "",
  "content": "",
  "target_path": "chapters/chapter-001.md"
}
```

服务端必须转换为 `HumanEdgeDecision`，不能绕过它。当前实现中该 API 只负责把写作动作归一化为 `HumanEdgeDecisionService.submit` 的请求，不直接修改 `GraphLoopState`。

## 6. 前端设计

### 6.1 页面定位

当前 `GraphTaskForegroundView` 应拆出写作专用工作台。

建议新增：

```text
frontend/src/components/workspace/views/writing-graph/
  WritingGraphProjectPage.tsx
  WritingChapterDesk.tsx
  WritingChapterReader.tsx
  WritingChapterActionBar.tsx
  WritingAssetPanel.tsx
  WritingRunDebugPanel.tsx
```

保留当前通用前台为：

```text
GraphTaskDebugConsole
```

### 6.2 写作章节台布局

桌面：

```text
左：章节索引 220-280px
中：正文阅读/编辑 minmax(620px, 1fr)
右：当前章控制 300-360px
```

移动：

```text
顶部：当前章 + 状态
主体：正文
底部：审核动作
章节列表和节点信息进入抽屉
```

### 6.3 正文阅读区

必须满足：

- 默认显示正文，不默认显示空文件编辑框。
- 正文字宽控制在 `68-78ch`。
- 行高适合中文长文阅读。
- 当前章标题、路径、状态清晰可见。
- 支持阅读/编辑切换。
- 支持保存后回到阅读。
- 支持上一章/下一章。

### 6.4 当前章控制区

只显示与当前章有关的信息：

- 当前状态
- 当前写手/审核节点
- 最近一次产物
- 人工动作
- 审核历史
- 节点会话入口

不要默认显示整个图。

## 7. 迁移策略

### 7.1 第一阶段：前端产品化

目标：不改底层 API，只用现有实例监控、文件树、human_controls，把页面改成写作生产台。

改动：

- `GraphTaskForegroundView` 拆出写作章节台组件。
- 默认进入章节台。
- 正文阅读居中。
- 写作动作按钮换成领域语义。
- 图监控移入“图调试”标签。

验收：

- 用户打开项目后能直接看章节。
- 用户能对当前章通过、退稿、替写。
- 用户不需要理解边协议也能操作。

### 7.2 第二阶段：后端写作投影

目标：减少前端猜测，把章节状态、动作、资产分类由后端投影。

改动：

- 新增 `writing_graphs/projection_service.py`。
- 新增 writing desk API。
- 前端从一个 writing projection 渲染章节台。

验收：

- 前端不再自己推断章节状态。
- 章节动作完全由后端返回。
- 写作资产分类稳定。

### 7.3 第三阶段：章节生产闭环

目标：让用户一章一章推进。

改动：

- 增加“生成下一章/继续运行”。
- 当前章审核条固定。
- 退稿意见进入边传播决策。
- 替写内容写入正式库并继续传播。
- 支持上一章/下一章和章节完成标记。

验收：

- 用户可以从第 1 章连续推进到第 N 章。
- 每章都有正文、状态、审核记录和产物来源。
- 图运行失败时能回到具体章节处理。

### 7.4 第四阶段：长篇写作质量工具

目标：面向 100 章生产优化。

可选能力：

- 章节版本历史。
- 正文 diff。
- 审核意见定位到段落。
- 角色/设定一致性提示。
- 章节摘要和伏笔索引。
- 批量章节进度看板。

## 8. 验收标准

### 8.1 产品验收

- 写作项目打开后默认是章节台。
- 正文阅读在屏幕中央，是最大视觉区域。
- 章节列表、正文、审核动作三者关系清楚。
- 用户可以不看拓扑图完成章节审核和推进。
- 图调试仍可进入，但不是默认主工作流。

### 8.2 技术验收

- 前端不直接修改 GraphLoopState。
- 写作动作最终进入 `HumanEdgeDecision` 或实例运行 API。
- 通用图任务能力未被写作页面破坏。
- 写作投影不引入第二运行权威。
- 现有图任务实例、文件、会话、监控测试通过。

### 8.3 运行验收

- 可以启动写作图任务。
- 可以打开当前章节正文。
- 可以通过当前章并推进下游。
- 可以退稿并回传意见。
- 可以用用户改写稿替换当前章并继续。
- 可以查看章节正式库文件。
- 可以进入图调试看节点/边状态。

## 9. 风险与约束

### 9.1 不要把写作页面做成硬编码旧图

写作页面可以面向写作图任务，但不能依赖某一个旧图的节点 ID。

正确做法：

- 通过图配置 metadata / contract / file path convention 识别写作角色。
- 通过后端投影生成章节状态。
- 前端只消费写作投影。

### 9.2 不要让前端猜测边动作

前端按钮来自后端 `chapter_actions` 或 `human_controls`。

如果后端没有允许动作，前端只显示不可处理，不自己构造传播。

### 9.3 不要丢掉通用图调试

写作页面不是删除通用监控，而是改变默认入口。

图调试对排障仍然重要。

## 10. 建议执行顺序

第一轮已按第一阶段执行：

1. 拆出 `WritingChapterDesk`。
2. 把正文阅读放到中央主舞台。
3. 把写作动作语义化。
4. 把图监控移到调试标签。
5. 保持现有 API，不新增后端投影。
6. 用真实页面截图验证桌面和移动端。

第二轮已先落地后端写作投影和前端消费；独立 chapter action API 后续再做。

第三轮已补上独立 chapter action API：

1. `POST /api/orchestration/writing-graph-instances/{instance_id}/chapter-actions` 接收写作语义动作。
2. `approve / request_revision / replace_with_user_text` 统一转换成 `pass / revise / replace`。
3. 替写内容写入图任务实例正式库，再作为 human-authored result 进入原有边传播。
4. 前端章节台优先调用写作语义 API，图调试仍保留原始 `human-edge-decisions`。
5. 回归测试覆盖通过本章和采用改写稿两条路径。

这样风险最小，也能最快改善用户体验。

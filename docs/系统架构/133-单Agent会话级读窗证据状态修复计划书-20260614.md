# 单 Agent 会话级读窗证据状态修复计划书

日期：2026-06-14

状态：待审阅

本计划书接续 `131-投影重复根因审查记录-20260614.md` 与 `132-非任务状态旧工具记录消失审查记录-20260614.md`。前两份文档审查的是前端可见投影和历史工具轨迹，本计划书单独处理另一个根因：普通 `single_agent_turn` 没有把 `read_file` 的已读窗口沉淀为可复用的会话级证据状态，导致新一轮普通对话容易重复读取已经读过的文件窗口。

本阶段只写修复计划，不改运行时代码。实施前需要用户审阅确认。

## 1. 目标

修复目标不是把所有工具消息硬塞回 `model_history`，也不是依赖 provider 协议 replay 勉强让模型“看见”旧工具输出。

目标是建立一条成熟 agent runtime 应有的证据权威链：

```text
工具真实读取文件
-> 规范化路径，记录文件内容 hash / mtime、读取窗口、partial view 标记、observation ref
-> 写入当前执行 scope 的 File Evidence Authority
-> 下一轮 runtime assembly 投影 file_evidence_decisions
-> 模型收到明确的 do_not_repeat_read_ranges / reusable_result_ref
-> read_file 在工具层校验路径、窗口、hash / mtime、stale / partial 状态
-> read_file 对未变化窗口返回可复用引用，而不是重复输出同一内容
```

修复完成后，同一会话内的普通单 Agent 对话应具备以下行为：

| 场景 | 目标行为 |
| --- | --- |
| 同一 session 下一轮再次需要同一文件窗口 | 动态上下文明确提示该窗口已经读过，优先复用证据或只读缺失窗口 |
| 同一 session 中相同 `path + start_line + end_line + content_sha256` 被再次请求 | `read_file` 可以返回 `content_omitted=true` 和 `reusable_result_ref` |
| 文件被写入或内容 hash 变化 | 原读窗标记为 stale，不允许当作当前事实使用 |
| 有正式 task_run 的任务执行 | 继续使用任务级 scope，不被会话级状态污染 |
| provider protocol replay 被预算截断或因 recovery package 跳过 | 文件证据仍通过结构化 `file_evidence_decisions` 投影，不依赖原始工具 transcript |

明确不把“全新 session 之间共享读窗证据”作为本次默认目标。跨 session 或项目级读窗缓存需要另设策略，因为它涉及用户隐私、上下文边界和旧文件内容可见性。

## 2. 当前运行流

### 2.1 普通历史拼接

`backend/runtime/shared/history_assembler.py`

当前 `_normalize_history()` 只保留：

```text
role in {"user", "assistant"}
```

这意味着普通 `model_history` 不包含 `tool` 消息。这一点本身不是 bug。成熟 agent 不应靠把历史工具原文无节制塞回模型上下文来恢复状态，因为这会引入预算不可控、协议污染和旧输出误用。

### 2.2 provider 协议 replay

`backend/harness/entrypoint/runtime_facade.py`

`RuntimeFacade` 会把会话历史中的 `api_transcript`、`compressed_context`、`context_recovery_package`、`provider_protocol_compaction_created_at` 放入 `session_context`。

`backend/harness/runtime/compiler.py`

`_provider_protocol_message_specs()` 会从 `session_context.api_transcript` 或 `provider_protocol_history` 中筛选 hot protocol messages：

```text
tool 消息
带 tool_calls 的 assistant 消息
带 reasoning_content 的 assistant 消息
```

但这条链有两个关键边界：

1. 如果存在 `context_recovery_package` 且 `provider_protocol_compaction_created_at <= 0`，函数直接返回空列表。
2. 即使 replay 存在，也会受 message limit 和 char budget 控制。

所以 provider protocol replay 只能作为协议连续性的热缓存，不能作为“文件读窗事实”的唯一权威。

### 2.3 文件读窗权威状态

`backend/runtime/memory/file_state_store.py`

`FileStateAuthorityStore` 当前以 `task_run_id` 作为唯一 key：

```text
load(task_run_id)
apply_observation(task_run_id, observation)
apply_events(task_run_id, events)
snapshot(task_run_id)
```

`backend/runtime/memory/file_state_authority.py`

`FileStateAuthority` 记录文件路径、内容 hash、读窗范围、写入导致的 stale 状态、observation ref 和 tool_call_id。

这条链本身方向正确：它把“文件是否已经读过”从自然语言历史中抽成结构化状态。

### 2.4 工具执行写入断点

`backend/runtime/tool_runtime/tool_executor.py`

`_commit_file_state_events()` 当前有硬前置条件：

```text
if execution_store is None or not task_run_id or not events:
    return {}
```

也就是说，只要工具调用没有 `task_run_id`，即使 `read_file` 产生了 `file_state_events`，也不会写入 `FileStateAuthorityStore`。

### 2.5 `read_file` 复用断点

`backend/runtime/tool_runtime/native_tools.py`

`_unchanged_previous_read_window()` 当前同样要求：

```text
task_run_id = context.task_run_id
if not task_run_id or not content_sha256:
    return {}
```

因此普通对话没有 `task_run_id` 时，`read_file` 不会查找之前的同窗口读窗状态。

### 2.6 普通 single_agent_turn 缺少 task scope

`backend/harness/loop/single_agent_turn.py`

`_invoke_turn_tool()` 构造 `ToolInvocationRequest` 时设置：

```text
caller_kind="agent_turn"
session_id=session_id
turn_id=turn_id
...
```

但不设置 `task_run_id`。这对普通对话是合理的，因为普通对话不是正式任务运行；问题是当前文件证据权威只接受 `task_run_id`，没有会话级 scope。

## 3. 结论

上下文拼接不是“完全坏掉”。真实问题是三条链没有汇合：

| 链路 | 当前状态 | 是否能单独解决重复读 |
| --- | --- | --- |
| `model_history` | 只保留 user / assistant | 不能，且不应该承载工具证据 |
| `provider_protocol_replay` | 可回放热工具协议，但会被 recovery package、边界和预算截断 | 不能，只能辅助协议连续性 |
| `FileStateAuthorityStore` | 能表达读窗证据，但只按 `task_run_id` 写入和读取 | 对 task_run 有效，对普通 single_agent_turn 失效 |

普通 `single_agent_turn` 没有正式 `task_run_id`，而当前反复读取防线又完全绑定 `task_run_id`。所以新一轮普通对话会自然倾向重新读文件：模型缺少结构化 `file_evidence_decisions`，工具层也缺少同窗口复用依据。

根因不是前端显示重复，也不是模型“不听话”，而是普通会话缺少会话级读窗证据权威。

### 3.1 外部成熟实现对照

本节对照的是项目目录外的真实源码，不是本项目文档推断。

#### 3.1.1 Claude Code：显式 FileStateCache + 工具层 dedup

源码目录：`D:\AI应用\claude-code-nb-main`

关键文件：

| 文件 | 证据 |
| --- | --- |
| `utils/fileStateCache.ts` | 定义 `FileState = { content, timestamp, offset, limit, isPartialView? }`；默认 `READ_FILE_STATE_CACHE_SIZE = 100`；默认大小上限 25MB；所有 key 经 `normalize()` 规范化；支持 `dump/load`、`cloneFileStateCache()`、`mergeFileStateCaches()` |
| `tools/FileReadTool/FileReadTool.ts` | `Read` 工具先查 `context.readFileState`；同路径、同 `offset/limit`、非 `isPartialView`、磁盘 mtime 未变时，直接返回 `{ type: "file_unchanged" }`，不再次发送完整内容 |
| `services/compact/compact.ts` | compact 前保存 `preCompactReadFileState`，随后 `context.readFileState.clear()`；compact 后按最近读取文件、最大 5 个文件、50k token 总预算、单文件 5k token 重新生成附件 |
| `utils/messages.ts` | `compact_file_reference` 会告诉模型 compact 前读过但内容太大，需要时重新调用 Read；`already_read_file` 返回空消息，说明“已读”更偏 runtime/UI marker，而不是强行塞回 API 文本 |
| `cli/print.ts`、`utils/queryHelpers.ts` | 启动/恢复时从 transcript 里提取 Read/Write/Edit 结果重建 `readFileState`；还支持 `seed_read_state`，客户端发现某次 Read 被 snip 后可把读状态种回缓存 |
| `utils/forkedAgent.ts`、`tools/AgentTool/runAgent.ts` | subagent 默认克隆或隔离 `readFileState`；fork 带父上下文时克隆父缓存，普通子 agent 可从空缓存开始，避免子 agent 状态污染父上下文 |

结论：

1. Claude Code 没有把“避免重复读”交给模型自觉完成，而是在 `FileReadTool` 的工具层做硬性去重。
2. 去重条件不是“路径相同”这么粗糙，而是至少包含规范化路径、精确范围、文件时间戳、非 partial view。
3. compact 不是简单清空历史；`readFileState` 同时承担 compact 后恢复索引，用预算把最近重要文件重新带回上下文。
4. resume 不是只读自然语言摘要；它会从 transcript 反推文件读状态，并允许外部 seed 补齐被 snip 掉的 Read。

这直接支持本计划的 `file_evidence_scope`，但也要求本计划升级：**session scope 不能只用于 prompt projection，必须成为 `read_file` 工具层 dedup 的查询和提交权威。**

#### 3.1.2 Codex：历史保留、工具输出截断、context baseline 与 compaction replacement

源码目录：`D:\AI应用\openai-codex`

关键文件：

| 文件 | 证据 |
| --- | --- |
| `codex-rs/core/src/context_manager/history.rs` | `ContextManager` 持有 `ResponseItem` 历史、`history_version`、`token_info`、`reference_context_item`；`record_items()` 记录 API 消息；`process_item()` 会截断 `FunctionCallOutput` / `CustomToolCallOutput` |
| `codex-rs/core/src/session/turn.rs` | 正常回合在采样前调用 `record_context_updates_and_set_reference_context_item()`；采样输入来自 `clone_history().for_prompt(...)` |
| `codex-rs/core/src/session/mod.rs` | `reference_context_item` 为空时完整注入初始 context；存在时只发 settings diff；每个真实用户 turn 都持久化 `TurnContextItem` baseline |
| `codex-rs/core/src/compact.rs` | pre-turn/manual compaction 使用 `DoNotInject`，清空 `reference_context_item`，下个正常 turn 完整 reinject；mid-turn compaction 会把初始 context 插入 replacement history 并设置新的 baseline |

结论：

1. Codex 更依赖 conversation history、工具输出保留/截断、compaction replacement history 和 context baseline diffing，没有查到 Claude Code 风格的 `FileStateCache` 主链。
2. 这说明成熟方案并不要求把所有工具原文塞回普通 `model_history`；Codex 仍然把历史、compact、context baseline 作为受控 runtime 结构管理。
3. 对本项目的启发是：`provider_protocol_replay` 和 history retention 只能承担协议连续性，不能替代文件读窗事实权威。文件证据需要有独立 scope，并作为 runtime assembly 的稳定输入。

### 3.2 对本项目计划的修订结论

外部对照后，本计划的方向不变，但修复标准提高为以下硬要求：

1. `file_evidence_scope` 必须进入工具调用合同，不允许只存在于动态 prompt 投影。
2. `read_file` 必须按 scope 查询已读窗口，并在未变化时返回结构化 stub / `reusable_result_ref`。
3. 文件读窗复用必须检查规范化路径、精确窗口、内容 hash 或 mtime、stale 状态、partial view 状态。
4. compact / recovery / resume 不能把文件证据归零；至少要能从持久 file evidence 或 transcript seed 恢复最近读窗索引。
5. subagent / task_run / 普通 agent_turn 的状态边界必须显式：默认隔离，只有被授权的 fork / continuation 才继承父 scope。

#### 3.3.2 Prompt 契合性结论

当前 prompt 的质量分两层看。

第一层，作为 agent prompt，它整体是契合的：不是“这是 runtime 节点”这类开发说明，而是直接告诉 agent 身份、职责、边界、证据来源和失败处理。例如：

| Prompt 来源 | 当前优点 |
| --- | --- |
| `backend/prompt_library/system_prompts.py:18-23` | 明确 agent 在本地工作区协作，区分稳定规则、环境边界、工具说明、动态状态和工具观察 |
| `backend/prompt_library/agent_prompts.py:6-23` | 单轮主 agent prompt 要求理解用户最新话语、选择最小充分行动、缺事实先观察、需要真实产物时进入可追踪执行路径 |
| `backend/prompt_library/packs.py:8-21` | 单轮 runtime prompt 明确每轮只做一个清晰裁决，不预测工具结果，不把工具记录当正文 |
| `backend/prompt_library/rules.py:51-68` | 失败恢复和上下文记忆规则强调当前事实、refs、rehydration_plan、有效读窗和不要重复失败动作 |

第二层，关于“已读窗口复用”的 prompt 方向也基本正确：

| Prompt 来源 | 当前规则 |
| --- | --- |
| `backend/prompt_library/io_capability_prompts.py:3-11` | 明确 `read_file` 是当前真实内容；不要重复读取相同行窗口；`file_unchanged` 时使用已有 observation；窗口缺失、过期、变化、未覆盖或 hash 冲突时才读最小窗口 |
| `backend/prompt_library/tool_prompts.py:83-92` | `read_persisted_tool_result` 只能恢复旧工具输出；若 file_state / 读窗证据表明未变，才可把恢复内容当精确证据 |
| `backend/prompt_library/rules.py:59-68` | 上下文被压缩或替换后依赖 refs、summary 和当前运行投影；已读窗口被省略但有有效 `rehydration_plan` 时先恢复，不重复读取同一未变范围 |
| `backend/prompt_library/rules.py:131-160` | 文件事实以当前工具观察、读取窗口、stale 状态、git 视图和 artifact 证据为准；编辑前必须具备当前有效读窗证据 |
| `backend/prompt_library/environment_lifecycle_prompts.py:65-68` | 旧工具记录只有在 `file_evidence_decisions` 标记为当前有效读窗证据时，才能作为当前文件事实 |

问题不在 prompt 方向，而在 prompt 引用的状态对普通 `single_agent_turn` 不稳定可得：

1. `file_evidence_decisions` 的生成器存在，但当前只通过 `task_execution` 的 volatile task_state 主链进入模型上下文。
2. 普通 `single_agent_turn` 的 prompt 告诉 agent “优先复用有效读窗”，但动态上下文没有投影 session scope 的 `file_evidence_decisions`。
3. 即使模型按 prompt 决定再次调用 `read_file`，工具层也因为没有 `task_run_id` / `file_evidence_scope` 而无法返回 `content_omitted=true` 的未变化 stub。

所以 prompt 不是完全错误，但对普通会话来说不完备：它表达了目标行为，却没有一个可依赖的运行时证据入口。后续修复不能只补一句 prompt，必须先把 `file_evidence_scope` 接入工具请求、工具上下文、store、动态投影和测试。

## 5. 目标架构

### 5.1 权威链

采用单向权威链，而不是在多个层里重复猜测：

```text
RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> ModelTurnDecision
-> ActionPermit
-> RuntimeStartPacket
-> ExecutionLoop
-> EvidenceRecord
```

对应到本修复：

| 层 | 目标职责 |
| --- | --- |
| RequestFacts | 记录 session_id、turn_id、是否存在 task_run_id、工作区根目录 |
| BoundaryPolicy | 决定工具权限、文件读写边界，不决定是否复读 |
| ContextCandidates | 读取当前 file evidence scope 的已读窗口 |
| ModelTurnDecision | 根据用户目标决定是否需要读文件 |
| ActionPermit | 允许或拒绝 read_file / write_file 等动作 |
| RuntimeStartPacket | 把 `file_evidence_decisions` 和 `read_resource_state` 放入模型可见上下文 |
| ExecutionLoop | 执行 read_file；如果窗口未变化则返回可复用引用 |
| EvidenceRecord | 把真实工具 observation 中的 `file_state_events` 写回当前 scope |

### 5.2 显式 File Evidence Scope

新增或等价表达一个明确 scope 合同：

```text
file_evidence_scope = {
  "kind": "task_run" | "session",
  "scope_id": "...",
  "session_id": "...",
  "task_run_id": "...",
  "authority": "runtime.file_evidence_scope"
}
```

规则：

1. 有正式 `task_run_id` 的任务执行，使用 `kind=task_run`、`scope_id=task_run_id`。
2. 普通 `single_agent_turn` 没有 `task_run_id` 时，使用 `kind=session`、`scope_id=session_id`。
3. 禁止为了复用旧接口而给普通对话伪造 `task_run_id`。
4. `task_run_id` 继续只表示正式任务运行身份，不承担普通会话身份。
5. 缺少 scope 时，工具层应返回可诊断的空提交或错误信息，不再静默伪装成“没有历史”。

### 5.3 Store 语义

`FileStateAuthorityStore` 需要从“task_run-only store”升级为“scope-based file state store”：

```text
load_scope(scope)
apply_events(scope, events)
snapshot_scope(scope)
```

可以继续使用现有 `FileStateAuthority` 的读窗、stale、hash 逻辑，但 payload 必须写入 scope 元数据，至少包括：

```text
scope_kind
scope_id
session_id
task_run_id
authority
path_key
range_identity
content_sha256
mtime_ms
is_partial_view
```

如果实施时发现旧 `task_run_id` 方法只剩兼容壳，应在同一切片删除或改为内部调用的私有辅助，不保留两套公开语义。

存储规则：

1. 路径 key 必须统一规范化，Windows 路径分隔符、相对路径、冗余段不能导致同一文件落成多个 key。
2. 读窗 identity 必须表达当前项目的行窗口语义：`path_key + start_line + end_line + content_sha256`；如果后续接入 byte offset 或 page range，要作为同级 range kind，不混用。
3. `is_partial_view=true` 的记录只能作为“模型曾看到过部分内容”的提示，不允许触发 `content_omitted=true` 的工具层复用。
4. 写入、编辑、替换、删除等事件必须使同文件旧读窗 stale；stale 读窗不能进入 `do_not_repeat_read_ranges`。
5. store 允许持久化，但默认 scope 仍是 session/task，不升级为项目全局缓存。

### 5.4 工具层与投影层分工

| 层 | 允许做的事 | 不允许做的事 |
| --- | --- | --- |
| File Evidence Store | 保存读窗、写入失效、scope 元数据 | 根据用户意图决定要不要读文件 |
| Runtime Assembly / Dynamic Context | 投影 `file_evidence_decisions`、`read_resource_state`，提醒模型复用已读窗口 | 把自然语言历史当作文件事实权威 |
| Model | 决定当前任务是否还需要读文件、读哪个缺口 | 自己假设旧读窗仍然有效 |
| `read_file` 工具 | 按 scope 做最终 dedup 校验，必要时返回 stub 或完整内容 | 因缺 `task_run_id` 静默跳过 evidence 查询 |
| Compact / Resume | 保存或重建最近读窗索引 | 把 compact 摘要当成完整文件内容 |

### 5.5 取消 read evidence 的 task / 普通 turn 行为分叉

本修复应取消 `read_file` 在“读窗证据提交、同窗口复用、stale 判断、rehydration 引用”上的 task / 普通 turn 差异。成熟结构不是让 task_run 有一套 read 证据链、普通 turn 另有一套弱化链路，而是让两者进入同一个 File Evidence Authority：

```text
read_file request
-> resolve file_evidence_scope
-> query FileEvidenceStore by scope
-> compare normalized path + range + content hash / mtime + stale / partial view
-> return full content or content_omitted stub
-> commit file_state_events back to the same scope
```

`task_run` 和普通 `single_agent_turn` 的区别只保留在 read evidence 之外：

| 维度 | 是否保留区别 | 原因 |
| --- | --- | --- |
| 文件读窗证据、复用、stale、rehydration | 不保留区别 | 同一个文件窗口是否当前有效，与调用者是不是 task 无关，只与 scope、路径、范围和文件状态有关 |
| 权限、审批、side effect 策略 | 保留区别 | task_run 有可恢复审批和持续任务合同，普通 turn 没有同等生命周期 |
| execution record / operation lifecycle | 保留区别 | task_run 需要正式执行记录、重试、暂停、恢复、任务状态；普通 turn 只需要工具观察和会话 trace |
| 动态上下文投影位置 | 保留入口差异，不保留语义差异 | task_run 投影到 `task_state`，普通 turn 投影到 session dynamic runtime，但两者的 `file_evidence_decisions` 语义一致 |

因此实施时禁止两种错误做法：

1. 不允许为了复用旧 task_run-only 代码给普通 turn 伪造 `task_run_id`。
2. 不允许为普通 turn 再写一套只靠 prompt / provider replay 的 read 复用逻辑。

正确做法是把 `task_run_id` 从 read 证据主权中降级为 `file_evidence_scope.kind="task_run"` 的一个字段；普通 turn 使用 `file_evidence_scope.kind="session"`。`read_file` 只面向 `file_evidence_scope`，不再在主逻辑里判断“这是 task 还是普通 turn”。

## 7. 需要新增或更新的测试

### 7.1 单元级

1. `ToolInvocationRequest` 允许 `caller_kind=agent_turn` 携带 `file_evidence_scope.kind=session`。

5. 路径规范化后同一文件只有一个 file evidence key。

1. 普通 session scope 第一次 `read_file` 返回内容并提交 read range。

4. compact / recovery package 存在时，session file evidence 不被 provider replay 边界吞掉。

### 7.4 集成级


## 9. 不做的事

| 不做项 | 原因 |
| --- | --- |
| 不把 tool messages 直接塞回 `model_history` | 这会污染普通对话历史，并受上下文预算影响 |
| 不把 provider protocol replay 当作文件证据权威 | replay 会被 compaction boundary、recovery package 和预算截断 |
| 不给普通 agent_turn 伪造 `task_run_id` | 会混淆任务运行身份和会话证据身份 |
| 不做跨 session / 项目级全局读窗缓存 | 需要单独设计隐私、失效和用户期望边界 |
| 不直接照搬 Claude Code 的纯内存 LRU | 本项目已有持久 `FileStateAuthorityStore`，应升级 scope 权威，而不是新建第二套缓存 |
| 不通过前端去重掩盖重复读取 | 这只能改善显示，不能修复 runtime 证据缺失 |
| 不保留 task_run-only 旧分支作为兜底 | 目标是 scope-based 主链路，旧分支会继续诱导误用 |

## 10. 风险与暂停条件

实施中若出现以下情况，应暂停并重新提交偏差说明：

1. 需要改变前后端 API contract，而不只是后端 runtime 内部合同。
2. 发现正式 task_run 依赖 `FileStateAuthorityStore` 的裸 `task_run_id` 文件路径作为外部契约。
3. session scope 和 task_run scope 需要双写或跨 scope promotion 才能满足真实业务。
4. 现有测试大量保护 task_run-only 内部结构，必须重写测试体系。
5. 真实启动 `3000/8003` 时发现固定端口被非本项目进程占用。
6. resume / compact 链路找不到可靠 observation / transcript seed，只能依赖摘要猜测文件内容。

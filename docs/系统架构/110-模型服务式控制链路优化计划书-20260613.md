# 模型服务式控制链路优化计划书

日期：2026-06-13
状态：待审计划书
适用范围：主会话单轮决策、Prompt 动作协议、工具服务面、Action admission、执行许可、观察/投影回执、相关回归测试
明确不包含：图任务链路、权限模式默认值重设、用户个人权限配置调整、旧链路兼容保留

---

## 1. 结论

本轮需要先纠正一个概念边界：

```text
Prompt 是开发者提供的高优先级指令，权限高于模型自然偏好。
Prompt 可以强制管理模型的角色、协议、输出格式、动作选择规则和反馈义务。
问题不在于能不能强制模型，而在于强制是否让控制精准、运行顺畅、覆盖全面。
```

因此，目标不是削弱 prompt，也不是让模型不受约束。目标是把强约束放在正确层级：

```text
Prompt / Developer Contract：
  强制模型遵守角色、动作协议、输出格式、用户反馈义务和任务入口规则。

Runtime / Service Surface：
  向模型展示当前挂载了哪些服务、哪些服务暂不可用、为什么不可用、如何进入正确服务面。

Execution / Safety Boundary：
  对真实副作用、文件写入、shell、网络、凭据、审批、资源锁、沙箱和不可重复动作做授权。

Observation / Projection：
  把协议错误、服务不可用、权限拒绝、审批等待、预算耗尽和执行结果变成模型可继续判断、用户可理解的回执。
```

当前系统的问题不是“系统不能限制模型”，而是几个不同性质的边界被压进同一个 `allowed_action_types` / `requires_json_action` / admission deny 语义里，导致模型得到的是粗粒度的“不允许”，用户看到的是二次确认、协议泄露或任务/日志/控制回执串线。

本计划的核心改造方向：

```text
允许 prompt 强约束模型。
禁止 runtime 假借安全名义重写模型语义决策。
禁止把协议错误、服务未挂载和真实安全拒绝混为一类。
所有不执行都必须返回可分类、可恢复、可投影的观察。
```

## 2. 验收三标准

后续控制链路只按这三个标准验收。

### 2.1 控制能不能精准

- 用户语义由模型在 prompt 约束下裁决。
- Prompt 可以明确要求：任务承接条件成立时必须选择 `request_task_run`；纯控制必须选择 `active_work_control`；控制动作必须用结构化 JSON action。
- Runtime 只承接、校验、分类、授权和执行，不把模型请求改写成另一个用户意图。
- Parser、admission、tool plan 和投影层不得成为第二个语义决策中心。

### 2.2 运行能不能顺畅

- 模型走错通道、调用未挂载服务、缺少任务合同、触发权限审批或预算耗尽时，系统必须返回明确观察，而不是让用户看到协议残片或让任务卡死。
- 纯控制动作也必须给用户反馈；反馈可以由 `active_work_control` 的用户可见反馈意图加系统投影承载，不要求另起一条脱离控制动作的普通 assistant 正文。
- 不执行不是沉默，也不是普通聊天补救；不执行必须形成结构化回执，让模型下一轮能继续判断。

### 2.3 覆盖够不够全面

- Prompt、runtime projection、provider response parser、native tool call parser、admission、tool plan、tool supervisor、SSE/公开投影、测试必须使用同一套分类。
- UI 隐藏不算后端收口；后端 action service、direct API、runtime monitor、task executor 入口必须同步拒绝或分类。
- 不允许保留旧链路作为“兼容兜底”。如果某个旧字段因外部 API 必须暂存，只能作为只读诊断输出，不能继续参与决策。

## 3. 当前控制链路审视

### 3.1 已经正确的基础

当前系统已经具备可继续收束的基础：

- `backend/harness/runtime/compiler.py` 中的 action schema 已把模型动作收敛为 `respond / ask_user / tool_call / request_task_run / active_work_control / block`。
- `backend/harness/runtime/compiler.py` 已明确任务承接条件成立时必须选择 `request_task_run`，这属于合法 prompt 强约束。
- `backend/prompt_library/environment_lifecycle_prompts.py` 已把纯控制反馈改为由控制裁决和系统投影统一承载，方向正确。
- `backend/harness/runtime/tool_plan.py` 已能过滤任务态工具，并记录 `task_scoped_tool_requires_task_run` 这类原因。
- `backend/runtime/tooling/supervisor.py` 已有真正的执行安全边界：capability membership、preflight、operation gate、permission mode、approval token/state/fingerprint、sandbox/file policy。

这些基础不应推翻。需要修的是权威分层和错误分类。

### 3.2 核心冲突

#### 冲突 1：`allowed_action_types` 被过度复用

当前 `allowed_action_types` 同时表达：

- 模型本轮可以选择哪些语义动作。
- 本轮 provider/native tool surface 是否开放。
- 本轮是否必须 JSON action。
- admission 是否直接拒绝。
- repair prompt 告诉模型如何修复。

这会造成一个问题：模型做了一个可理解但通道不对或服务未挂载的动作时，系统容易直接返回“本轮不允许”，而没有告诉模型这是协议问题、服务面问题、任务态问题还是安全问题。

#### 冲突 2：`requires_json_action` 把协议通道和工具 transport 混在一起

控制动作要求 JSON action 是正确的 prompt/protocol 约束。比如 `request_task_run`、`active_work_control`、`ask_user`、`block` 作为控制裁决，不应伪装成 provider-native tool call。

但这不是安全边界。它是动作协议和投影一致性要求。若模型用 native tool call 表达了控制意图，系统应保留该意图并返回 `protocol_violation: control_action_requires_json_action`，而不是把它笼统描述成“模型不允许调用工具”。

#### 冲突 3：任务态工具过滤正确，但模型可见解释不够稳定

`agent_todo`、任务态编辑/写入工具不应在未进入 TaskRun 的单轮中作为普通工具诱导模型，这一点是正确的。

但这应被描述为：

```text
当前服务面未挂载任务态工具。
如果当前目标需要这些任务态服务，应提交 request_task_run。
```

不应被描述为真实安全风险。真实安全风险要到执行许可和 tool supervisor 层判断。

#### 冲突 4：admission 过早使用“动作不在 packet 内”作为拒绝语义

`backend/harness/loop/admission.py` 先检查 `action_not_allowed_by_packet`，然后才处理工具服务面、任务态工具、计划模式和副作用策略。

这会让 admission 变成一个粗粒度门禁：它知道“不在 allowed_action_types”，但未必说明模型到底错在：

- 输出协议不合规。
- 服务未挂载。
- 任务态能力需要 TaskRun。
- 合同缺口。
- 权限模式不允许。
- 真实 operation gate 拒绝。

目标不是取消 admission，而是让 admission 做分类器和执行前许可，不做语义替换。

#### 冲突 5：协议修复容易变成“不能工具”的模型感受

`single_agent_turn` 的 protocol repair 会说当前修复阶段不允许普通工具调用。这个说法在预算耗尽、控制动作修复、任务入口修复中可以成立，但必须带上明确原因：

- `protocol_violation`：上一轮格式/通道错了。
- `budget_exhausted`：工具预算已耗尽。
- `service_unavailable`：本轮没有挂载普通工具。
- `requires_task_run`：需要先进入任务运行态。

否则模型会把它理解为系统在任意阻拦行动，而不是提供可恢复的服务边界。

#### 冲突 6：109 计划书方向对，但概念需要校正

`109-任务入口控制与Prompt协议泄露修复计划书-20260613.md` 的优点是抓住了任务入口问题：

- 用户明确启动时，不应先普通工具消耗任务入口。
- `request_task_run` 不应写进 Markdown 正文。
- `agent_todo` 属于任务运行态工具，不应在单轮入口诱导模型。

但 109 需要补充一个边界：这不是“安全上禁止模型思考或行动”，而是 prompt 协议和服务面设计。Prompt 可以强制模型选择 `request_task_run`，runtime 可以不挂载任务态工具；如果模型偏离，系统要返回协议/服务观察，而不是以安全名义粗暴关门。

## 4. 目标权威链

目标链路如下：

```text
RequestFacts
-> DeveloperPromptContract
-> RuntimeProjection
-> ModelTurnDecision
-> ProtocolParser
-> ServiceSurfaceClassifier
-> ExecutionPermit
-> ToolSupervisor
-> ObservationProjector
-> PublicProjection / TaskLog
```

| 层级 | 当前目标职责 | 不允许做的事 |
| --- | --- | --- |
| RequestFacts | 记录用户话语、附件、当前 active turn、历史只读结果 | 不猜任务类型，不选工具 |
| DeveloperPromptContract | 强制角色、动作协议、输出格式、任务入口、用户反馈义务 | 不写开发节点说明，不把历史任务伪装成当前工作 |
| RuntimeProjection | 展示当前事实、能力目录、服务面、不可用原因、输出合同 | 不重写用户目标，不隐藏关键边界 |
| ModelTurnDecision | 模型在 prompt 约束下选择语义动作 | 不自授权限，不把协议词写给用户 |
| ProtocolParser | 解析 JSON/native 输出，保留模型意图并分类通道错误 | 不因为通道错误丢掉模型意图 |
| ServiceSurfaceClassifier | 判断服务是否挂载、是否任务态、是否需要合同 | 不把服务未挂载说成真实安全拒绝 |
| ExecutionPermit | 对副作用、权限、审批、资源锁、预算做执行许可 | 不替模型生成替代动作 |
| ToolSupervisor | 执行真实安全检查和工具预检 | 不吞掉拒绝原因 |
| ObservationProjector | 输出模型可消费观察、用户可见控制回执和日志事件 | 不把任务日志、控制回执、普通正文串线 |

## 5. 目标数据与分类

### 5.1 `ModelDecisionContract`

替代 `allowed_action_types` 的语义权威位置。

```json
{
  "authority": "harness.runtime.model_decision_contract",
  "semantic_actions": ["respond", "ask_user", "tool_call", "request_task_run", "active_work_control", "block"],
  "required_transport": "json_action",
  "control_actions": ["ask_user", "block", "request_task_run", "active_work_control"],
  "feedback_obligation": {
    "must_provide_user_visible_feedback_intent": true,
    "pure_control_feedback_projected_by_system": true
  },
  "selection_rules_ref": "runtime_action_selection_rules"
}
```

职责：

- 说明模型必须如何输出。
- 说明哪些动作是控制裁决。
- 说明什么时候必须 `request_task_run`。
- 说明纯控制也要有用户反馈意图。

它不负责判断工具是否真实可执行。

### 5.2 `ServiceSurface`

替代“工具可见/不可见靠散字段猜”的服务面。

```json
{
  "authority": "harness.runtime.service_surface",
  "mounted_tools": [],
  "unmounted_services": [
    {
      "service": "agent_todo",
      "category": "requires_task_run",
      "reason": "task_scoped_tool_requires_task_run",
      "required_action": "request_task_run"
    }
  ],
  "capability_directory": [],
  "tool_call_transport_available": true
}
```

职责：

- 告诉模型当前能用什么。
- 告诉模型预期服务为什么不可见。
- 告诉模型如何进入正确服务面。

它不负责真实权限审批。

### 5.3 `ActionIssue`

所有不执行都必须分类，禁止只写“不允许”。

```json
{
  "category": "protocol_violation|service_unavailable|requires_task_run|contract_gap|permission_denied|approval_required|resource_locked|budget_exhausted|runtime_unavailable",
  "code": "",
  "model_intent_preserved": true,
  "requested_action_type": "",
  "requested_tool_name": "",
  "repair_instruction": "",
  "user_visible_summary": ""
}
```

分类定义：

| category | 含义 | 归属层 |
| --- | --- | --- |
| `protocol_violation` | JSON 格式、Markdown 包裹、native 控制动作、混合多个控制裁决 | ProtocolParser |
| `service_unavailable` | 工具/服务未挂载、当前服务目录没有该工具 | ServiceSurfaceClassifier |
| `requires_task_run` | 工具属于任务运行态，需要先 `request_task_run` | ServiceSurfaceClassifier |
| `contract_gap` | `task_contract_seed` 缺目标、范围、验收 | Admission / Contract |
| `permission_denied` | operation gate、权限模式或沙箱拒绝 | ExecutionPermit / ToolSupervisor |
| `approval_required` | 需要人工审批或授权 token | ToolSupervisor |
| `resource_locked` | 文件锁、并发冲突、不可重复动作冲突 | ExecutionPermit |
| `budget_exhausted` | 本轮工具预算或任务步预算耗尽 | ExecutionLoop |
| `runtime_unavailable` | 后端服务、工具执行器、环境资源不可用 | Runtime |

## 6. Prompt 强约束规则

Prompt 可以而且应该强制模型执行以下规则：

1. **任务入口强约束**
   - 用户明确要求开始、启动、继续实施、修复、构建、生成、写入、验证，并要求可验收结果时，必须选择 `request_task_run`。
   - 目标、范围、验收不足时，必须选择 `ask_user`。
   - 不得先用普通工具消耗任务入口。

2. **控制动作通道强约束**
   - `ask_user / block / request_task_run / active_work_control` 是控制裁决，必须输出一个 JSON action。
   - 不得把控制动作写进 Markdown 代码块。
   - 不得把 action type、协议字段或机器状态写进用户正文。

3. **用户反馈强约束**
   - agent 必须给用户反馈。
   - 对纯控制动作，模型必须在控制 payload 中携带用户可见反馈意图，由系统投影成控制回执。
   - 这不是“没有回复用户”，而是“回复由控制裁决和投影承载”，避免普通正文、控制回执和任务日志串线。

4. **歧义保护强约束**
   - 只有用户明确要求暂停/停止当前工作时才选择 pause/stop。
   - “等一下，为什么 X？”、“先看一下 X”、质疑、纠错、追问或追加约束，不等同暂停或停止。

5. **服务面解释强约束**
   - 如果预期工具不可见，模型应报告环境或能力投影问题，或选择进入正确任务态；不要要求用户重复授权。

这些都是 prompt 的合法管理范围。它们的问题不在“强”，而在是否能导向顺畅执行。

## 7. 文件级优化计划

### 阶段 1：拆分动作合同、服务面和执行边界

文件：

- `backend/harness/runtime/compiler.py`
- `backend/harness/runtime/assembly.py`
- `backend/harness/runtime/tool_plan.py`
- 相关 prompt accounting / projection tests

修改：

- 在 runtime packet diagnostics 中新增 `model_decision_contract`、`service_surface`、`execution_boundary`。
- `model_decision_contract` 负责语义动作、JSON action 要求、控制动作、反馈义务。
- `service_surface` 负责 mounted tools、filtered/unmounted service reasons、task-scoped route reasons。
- `execution_boundary` 负责权限模式、approval、side effect policy、resource/budget 状态摘要。
- 删除把所有语义都塞进 `allowed_action_types` 的内部决策用法；如果外部序列化暂需保留同名字段，只作为派生诊断，不作为权威输入。

验收：

- prompt 中能同时看到“你必须如何输出”和“当前哪些服务可用/不可用”。
- 代码中不再由同一个字段同时驱动 prompt、parser、admission 和安全边界。

### 阶段 2：校正 prompt 文案，保留强约束但删除伪安全语义

文件：

- `backend/harness/runtime/compiler.py`
- `backend/prompt_library/packs.py`
- `backend/prompt_library/environment_lifecycle_prompts.py`
- `docs/系统架构/109-任务入口控制与Prompt协议泄露修复计划书-20260613.md` 或后续修订文档

修改：

- 保留“任务承接条件成立时必须选择 `request_task_run`”。
- 保留“控制裁决必须 JSON action”。
- 把“当前不允许普通工具”改成分类表达：
  - 当前是控制动作修复阶段。
  - 当前服务面未挂载普通工具。
  - 当前工具属于任务态，需要 `request_task_run`。
  - 当前工具预算耗尽。
- 明确纯控制动作仍要有用户反馈意图，由控制回执投影。
- 删除“安全”泛化用语，除非对应真实 permission/sandbox/approval/resource risk。

验收：

- prompt 不弱化开发者权威。
- prompt 不把服务未挂载说成权限安全。
- prompt 不诱导模型输出多余普通 assistant 正文。

### 阶段 3：Parser 保留模型意图并分类通道错误

文件：

- `backend/runtime/model_gateway/model_response_protocol.py`
- `backend/harness/loop/single_agent_turn.py`
- `backend/harness/loop/model_action_protocol.py`

修改：

- `require_json_action` 失败时输出 `ActionIssue(category="protocol_violation")`。
- native tool call 表达 `request_task_run / active_work_control / ask_user / block` 时，保留 `requested_action_type`，返回 `control_action_requires_json_action` 修复观察。
- native 普通工具在当前无 tool_call transport 时，返回 `service_unavailable` 或 `protocol_violation`，取决于原因：
  - transport 不允许：protocol。
  - 服务未挂载：service。
  - 工具任务态：requires_task_run。
- JSON action + native tool calls 混合时，保留诊断并要求模型二选一，不执行任何一边。

验收：

- 模型意图不因通道错误丢失。
- 协议修复不会变成用户可见二次确认。
- 错误分类能进入后续 observation 和测试断言。

### 阶段 4：Admission 从粗门禁改为分类与许可

文件：

- `backend/harness/loop/admission.py`
- `backend/harness/loop/action_permit.py`
- `backend/harness/runtime/tool_plan.py`

修改：

- 删除或替换开头粗粒度 `action_not_allowed_by_packet` 主路径。
- admission 按顺序分类：
  1. 未知 action type：`protocol_violation`
  2. 当前合同不支持该控制动作：`service_unavailable` 或 `runtime_unavailable`
  3. 工具不在服务面：`service_unavailable`
  4. 工具属于任务态：`requires_task_run`
  5. task contract 缺失：`contract_gap`
  6. plan mode / side effect policy：`permission_denied` 或 `requires_task_run`
  7. 可执行副作用：交给 ToolSupervisor
- `ActionPermit` 不再复制 `allowed_action_types` 语义，改为承载 `execution_boundary` 与 `ActionIssue`。

验收：

- admission 不再把所有问题都说成“不允许”。
- admission 不替模型选择新动作，只返回观察或许可。
- plan mode、side effect、task-scoped tool 的原因清晰可区分。

### 阶段 5：ToolSupervisor 保持真实安全边界

文件：

- `backend/runtime/tooling/supervisor.py`
- 权限/operation gate 相关测试

修改：

- 保持现有真实安全链路：capability membership、preflight、operation gate、approval、sandbox/file policy。
- 将 supervisor 输出映射为标准 `ActionIssue`：
  - membership failure：`permission_denied` 或 `service_unavailable`，按 capability table 来源区分。
  - preflight repair：`protocol_violation` 或 `resource_locked`，按具体错误区分。
  - approval：`approval_required`。
  - operation denied：`permission_denied`。
- 不在 supervisor 内生成替代工具或替代用户目标。

验收：

- 真正安全边界集中在执行许可和 supervisor。
- “模型通道不对”不再被说成安全风险。
- “安全拒绝”必须有 operation、policy、approval/sandbox/resource 证据。

### 阶段 6：观察、控制回执、任务日志分离

文件：

- `backend/harness/loop/single_agent_turn.py`
- `backend/harness/runtime/session_timeline.py`
- `backend/harness/runtime/run_monitor/*`
- 前端投影 store / panel 相关文件按需

修改：

- 所有 `ActionIssue` 形成 canonical observation。
- 控制动作结果投影到 control receipt。
- 工具/任务执行细节进入 task log。
- 普通 assistant 正文只承载模型明确要对用户说的回答，不承载协议修复、工具日志或控制执行细节。
- SSE reconnect 只恢复流和投影，不触发 executor 续跑或旧任务提升。

验收：

- 聊天正文、控制回执、任务日志不串线。
- 纯控制动作用户有反馈，但不会多出脱节正文。
- 协议修复信号不直接暴露给用户。

## 8. 必须删除或改写的旧链路

| 候选 | 问题 | 替代 |
| --- | --- | --- |
| `allowed_action_types` 作为多层权威桶 | 同时驱动 prompt、parser、admission、服务面和安全，导致语义混乱 | `model_decision_contract` + `service_surface` + `execution_boundary` |
| `action_not_allowed_by_packet` 粗拒绝 | 不说明协议、服务、合同、权限还是安全问题 | 标准 `ActionIssue` 分类 |
| native 控制动作直接按 tool call 不允许处理 | 丢失模型真实控制意图 | `protocol_violation: control_action_requires_json_action` |
| “当前不允许普通工具”无原因文案 | 容易被理解为系统任意阻拦模型 | 加上 `budget_exhausted / service_unavailable / requires_task_run / repair_phase` |
| 任务态工具过滤只隐藏、不投影原因 | 模型不知道如何进入正确能力面 | `service_surface.unmounted_services` |
| 测试只断言旧字段存在 | 保护旧内部结构 | 断言行为、分类和投影 |

删除标准：

- 只要新合同能表达同一事实，就不保留旧字段作为决策输入。
- 只要旧路径会让模型绕过新合同，就必须删除或恒拒绝。
- 只要测试只为旧字段命名服务，不保护真实行为，就改写或删除。

## 9. 与 108 / 109 的关系

### 9.1 继承 108 的部分

继续保留 108 的核心观点：

```text
模型负责调度和语义裁决。
系统负责服务、执行、记录、边界和反馈投影。
Prompt 指导模型如何操作系统。
```

本计划补充：prompt 本身是开发者权威，可以强制模型；不能把“模型负责调度”误解成 prompt 只能建议。

### 9.2 修正 109 的部分

继续保留 109 的目标行为：

```text
用户明确启动
-> 模型提交 request_task_run
-> 系统创建 TaskRun
-> 任务态工具在 TaskRun 内执行
```

需要修正的表述：

- “动作面约束”要拆成 prompt 协议强约束和 service surface 挂载约束。
- “普通工具不可用”要说明是任务入口服务面，不是安全本身。
- `agent_todo` 不在单轮暴露是任务态服务设计，不是禁止模型思考。
- 模型走错时，应返回协议/服务观察，不应引导用户二次确认。

## 10. 实施顺序

建议一次性按以下顺序实施，不保留两套不成熟路径：

1. 在 runtime packet 中建立 `model_decision_contract / service_surface / execution_boundary`。
2. 修改 prompt 和 runtime projection，保留强约束，改正分类文案。
3. 修改 parser，使错误输出标准 `ActionIssue` 并保留模型意图。
4. 修改 admission，使其按 protocol/service/contract/permission 分类，不再粗拒绝。
5. 修改 tool plan，把过滤掉的服务和原因投影给模型。
6. 修改 supervisor 映射，使真实安全拒绝进入标准分类。
7. 修改 observation/projector，使控制回执、任务日志、普通正文分离。
8. 更新和删除旧测试，确保旧决策桶不可作为主链路继续存在。
9. 运行 focused tests。
10. 用 `rg` 静态检查旧文案和旧决策入口是否还可达。

## 12. 成功状态

升级完成后，以下链路必须成立：

```text
用户：开始实施
Prompt：强制模型在任务承接条件成立时选择 request_task_run
模型：输出合法 JSON action，并携带用户可见反馈意图
Parser：解析为 ModelTurnDecision
ServiceSurface：说明任务态工具将在 TaskRun 中挂载
Admission：确认合同完整或返回 contract_gap
ExecutionPermit：只处理真实执行边界
TaskRun：创建并进入执行循环
Projection：用户看到任务已接住和下一阶段，不看到协议字段
```

以下链路必须不可达：

```text
模型把 request_task_run 写进 Markdown 正文。
模型先用 agent_todo 或写入工具消耗任务入口。
系统把 native 控制动作直接说成安全拒绝。
admission 用一个 action_not_allowed_by_packet 覆盖所有错误。
纯控制动作没有用户反馈。
纯控制动作额外生成脱节普通 assistant 正文。
任务日志、控制回执、聊天正文互相串线。
旧 latest task / waiting_executor 被当作当前 active work。
```

这才符合本项目当前真正需要的控制系统：prompt 能强管模型，模型负责语义调度，系统提供服务和真实安全边界，所有结构都服务于精准、顺畅和全面。

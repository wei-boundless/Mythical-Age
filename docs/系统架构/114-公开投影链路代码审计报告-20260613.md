# 公开投影链路代码审计报告

日期：2026-06-13  
状态：代码审计报告，非实施计划。  
审计基准：`111-输出与投影稳定性优化升级计划书-20260613.md`、`113-用户优先公开投影单一权威实施方案-20260613.md`。  
审计结论：当前代码只完成了部分新链路，不能判定为合格。它已经具备 `public_projection_frame -> frontend ledger -> MessagePublicProjection` 的主干，但仍存在直接会破坏“严格时序、单一权威、收口清爽、显示稳定”的旧逻辑和边界漏洞。

## 1. 结论先行

### 1.1 已经符合计划书的部分

1. 后端 chat stream 已把 runtime event 归一为 public event，并在 public event 上挂 `public_projection_frame`。证据：`backend/api/chat.py:507` 到 `551`。
2. 所有 public event 现在都会被要求生成 projection frame。证据：`backend/runtime/output_stream/public_contract.py:58` 到 `62`。
3. 后端 projection projector 已经能把正文、工具请求、权限、工具结果、commit、terminal 投成 frame。证据：`backend/harness/runtime/projection/projector.py:70` 到 `101`。
4. 工具主视图入口已经改成模型工具请求 `tool_call_requested`，不是裸 `tool_item_started`。证据：`backend/harness/runtime/projection/projector.py:133` 到 `159`。
5. `tool_item_started` 缺少 `tool_call_id` 或 `permission_decision_id` 时不会进主视图。证据：`backend/harness/runtime/projection/projector.py:204` 到 `229`。
6. 成功工具完成后会 `item_retire` 到 trace，失败工具会 pinned。证据：`backend/harness/runtime/projection/projector.py:232` 到 `280`。
7. 前端 reducer 已经以 `public_projection_frame` 为主入口，并按 ledger 处理 body/current/pinned/final/status/trace。证据：`frontend/src/lib/projection/reducer.ts:25` 到 `33`、`188` 到 `275`。
8. `commit_ack` 会把 body 标记 committed，并 retire transient item。证据：`frontend/src/lib/projection/reducer.ts:249` 到 `253`。
9. 前端 hydrate 已经有 projection commit ack cursor，只在 `commit_ack` frame 后触发一次 pending hydrate。证据：`frontend/src/lib/store/runtime.ts:2386` 到 `2415`。
10. Task executor 成功路径已经先 commit final message，再 completed。证据：`backend/harness/loop/task_executor.py:4302` 到 `4370`。

### 1.2 不合格点

P0：直接 single-agent turn 的 final message commit 没有写 `session_output_commit_ack` event。  
证据：`backend/harness/loop/single_agent_turn.py:4303` 到 `4335` 只调用 `commit_assistant_message` 并返回 decision；`backend/harness/entrypoint/runtime_facade.py:2044` 到 `2138` 只 append session message / memory receipt，没有 append event log ack。  
影响：当前 `commit_ack` 作为 hydrate 唯一权威的原则对 task executor 成立，但对直接 chat turn 不完整。直接 turn 可以落盘成功，却没有统一 commit frame，前端无法严格按 ack hydrate。

P0：生产 UI 仍有旧 timeline 二次裁决逻辑。  
证据：`frontend/src/components/chat/PublicTimelineActivity.tsx:402` 到 `445` 仍按文案判断 generic status；`frontend/src/components/chat/PublicTimelineActivity.tsx:502` 到 `510` 仍按工具状态折叠 completed tool；`frontend/src/lib/projection/timeline.ts:31` 到 `90` 仍有 `SUPPRESSED_TEXT` 和 machine-status 文案过滤。  
影响：这违反 `113` 的“前端不裁决语义”和“Pure Chat Render”。即使输入来自 `MessagePublicProjection`，组件仍会按文案、state、工具完成状态自行隐藏/折叠。

P0：后端 projection authority 还没有强制执行完整工具时序。  
计划书要求：

```text
tool_call_requested.event_offset
< tool_permission_decided.event_offset
< tool_item_started.event_offset
< tool_item_completed.event_offset
```

当前代码只做局部字段检查。`tool_item_started` 检查 id 存在，证据：`backend/harness/runtime/projection/projector.py:204` 到 `212`；前端 `current_action` 只要求 `tool_call_id`，证据：`frontend/src/lib/projection/reducer.ts:209` 到 `215`。没有任何状态表证明这个 `tool_call_id` 已经先出现过 `tool_call_requested`、已经有 allow permission、offset 顺序合法。  
影响：异常历史或乱序 event 仍可能被投为 trace 或影响 ledger，无法做到“错误状态不得进入公开执行生命周期”。

P1：projection frame builder 对非法字段静默降级。  
证据：`backend/harness/runtime/projection/authority.py:43` 到 `47` 对非法 `op/slot/source/main_visibility/retention` 用 fallback；`authority.py:150` 到 `152` 直接返回 fallback。  
影响：错误 spec 会被改成 hidden trace，而不是协议失败。计划书要求边界清晰，错误 frame 应该 fail-closed 并产生 protocol diagnostic。

P1：`session_timeline.py` 仍保留 closeout presentation 逻辑。  
证据：`backend/harness/runtime/session_timeline.py:188` 到 `203` 会从 `final_answer/monitor.final_answer` 生成 closeout summary，甚至 fallback `"结果收口"`。  
影响：当前它没有再返回 `public_timeline/task_projection`，风险小于旧链路；但它仍然是 hydrate/attachment 层的展示文案生成器，后续一旦被主视图消费，会重新引入 closeout 覆盖 live 内容的问题。

P1：`events.ts` 仍有 raw transport 文案裁决和 session activity patch。  
证据：`frontend/src/lib/store/events.ts:94` 到 `210` 定义 `stageStatusForEvent()` / `activityDetailForEvent()`；`events.ts:1768` 到 `1774` 在没有 projection frame 时仍允许 transport event patch stage/activity。  
影响：当前只允许 transport events，但边界仍不是“纯 projection”。`error/stopped/done` 等路径仍能绕过 projection authority 写 UI 状态。

P1：`assistant_text` / `answer_candidate` 被忽略，但未证明所有生产路径都不再依赖它们。  
证据：`backend/api/chat.py:800` 到 `801` 对 `answer_candidate` 和 `assistant_text` 直接 return `[]`。  
影响：这符合禁止旧正文主链路的方向；但如果某个 runtime 仍只发这两类事件，正文就会缺失，用户只看到工具或空白。必须用生产路径搜索和端到端测试证明它们不可达，或者迁移 emitter。

P1：`token` 被临时归一为 `assistant_text_delta`，但这是补洞，不是最终结构。  
证据：`backend/api/chat.py:802` 到 `814`。  
影响：这能修“正文不推进”的一类症状，但它仍说明 upstream typed stream 没有完全收敛。最终应让 runtime 直接发 typed `assistant_text_delta/final`。

## 4. 用户侧可见流程

### 4.1 正确情况下用户会看到

1. 模型先出正文：`assistant_text_delta -> body_append`，用户看到正文推进。
2. 模型先请求工具：`tool_call_requested -> current_action`，用户看到一个当前动作卡片，不是裸工具日志。
3. 权限需要确认：`tool_permission_decided -> pinned`，用户看到等待确认。
4. 工具成功：`tool_item_completed -> item_retire`，普通工具从主视图消失，进入 trace。
5. 工具失败：`tool_item_completed failed -> pinned`，失败留主视图。
6. final text：`body_finalize`，用户看到最终正文。
7. commit ack：`commit_ack`，transient item retire，前端 hydrate 一次。
8. turn completed：`turn_terminal`，不会清正文，不会 hydrate。

### 4.2 当前代码仍可能让用户看到错误体验

1. 直接 single-agent turn：final message commit 后没有 commit ack，前端不能严格知道何时 hydrate。
2. 如果 upstream 只发 `assistant_text` / `answer_candidate`，正文直接被后端忽略，用户会只看到工具或空白。
3. `PublicTimelineActivity` 可以因为文案像“正在处理”就隐藏，或者因为工具 done 就折叠，这不是后端 frame 明确裁决。
4. `session_timeline.py` 仍能生成 `"结果收口"` summary；如果未来被主视图接入，会复活“收口刷新/收口占位”问题。

## 7. 验收口径

完成后必须满足：

1. 主视图第一个语义反馈只能来自 assistant body 或模型 `ToolCallRequest` projection。
2. 裸 `tool_item_started` 永远不能进入主视图。
3. 普通成功工具在 commit ack 后不留主聊天。
4. failed/waiting/permission/commit_failed 必须 pinned。
5. `turn_completed` 不清屏、不 hydrate。
6. `assistant_text_final` 不 hydrate。
7. `session_output_commit_ack` 是唯一 hydrate 权威。
8. direct turn 和 task executor 两条路径都必须有 commit ack。
9. 前端不能按中文文案、工具名、terminal、final answer 决定主视图。
10. monitor/session timeline 不能生成主聊天投影。

## 8. 当前判断

当前实现不合格。不是因为缺少某个小判断，而是三个结构边界还没打穿：

1. 直接 turn commit 没有进入 commit ack 权威链。
2. UI 仍在二次裁决 projection item。
3. 工具生命周期没有全链路时序状态机。

只有这三点完成，才算严格执行 `113` 的投影原则。

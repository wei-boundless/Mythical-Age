# 090-图任务前台重新规划书

日期：2026-06-10

## 1. 结论

当前图任务前台不能继续补丁式修。问题不只是卡片太大，而是页面对象层级错误。

现在的前台把以下对象平铺在一屏：

- 图任务定义；
- 项目实例；
- 项目启动；
- 运行监控；
- 节点输出；
- 项目文件；
- 运行产物。

这会导致两个严重后果：

1. 未选择项目时也显示一整套“0% / 暂无风险 / 暂无产物 / 等待节点信号”的空监控，信息价值极低却占满首屏。
2. 选择项目后，监控、节点会话和文件管理仍然互相挤压，没有任何一个区域达到生产可用的密度和主次关系。

因此新版前台必须按“对象层级”重做，而不是继续调整旧页面 CSS。

## 2. 产品定位

图任务前台是多 agent 图任务的生产入口，职责是：

1. 选择图任务定义；
2. 管理该图任务下的多个项目实例；
3. 进入某个项目实例的运行控制台；
4. 启动、监控、恢复项目运行；
5. 查看节点会话和节点输出；
6. 管理项目正式库和运行产物。

图任务前台不是图编辑器。图定义、节点、边契约、编译配置仍然放在任务系统配置页。

## 3. 新信息架构

新版前台分成两个明确层级。

### 3.1 一级：图任务与项目管理

这是进入 `图任务` 主页面后的默认层。

目标：让用户快速选择“我要运行哪张图、哪个项目”。

首屏只显示：

- 顶部：图任务系统标题、刷新、创建项目；
- 左侧或上方：图任务定义列表；
- 主区域：当前图任务下的项目实例列表；
- 项目实例行显示：
  - 项目名称；
  - 状态；
  - 最近运行；
  - 最近产物；
  - 运行风险；
  - 进入项目按钮；
  - 启动/继续按钮。

禁止在一级页面显示：

- 0% 进度卡；
- 空节点画布；
- 空文件编辑器；
- 空产物库；
- 大面积“等待节点信号”。

一级页面应该像项目管理台，而不是监控台。

### 3.2 二级：项目运行控制台

用户点击某个项目实例后进入二级控制台。

目标：让用户知道这个项目正在由哪些 agent 推进、卡在哪里、产物在哪里。

二级控制台采用工作台布局：

- 左侧：项目内节点/运行阶段导航，可折叠；
- 中心：运行焦点拓扑；
- 右侧：选中节点的会话输出；
- 下方或页签：项目正式库。

二级控制台顶部只显示当前项目的运行状态和主要操作：

- 项目名称；
- 当前运行 id；
- 状态；
- 启动 / 继续 / 刷新；
- 最近风险；
- 最近产物。

## 4. 拓扑方案

拓扑前台采用双模式，但本轮只交付主模式。

### 4.1 默认模式：运行焦点图

默认显示当前活跃子图、阶段或节点组。

这适合日常生产，因为用户关心的是：

- 当前哪个节点在跑；
- 哪些节点完成；
- 哪些节点阻塞；
- 节点之间的交接是否完成；
- 下一步该人工处理什么。

运行焦点图不要求展示整张大图。它应优先显示 active / blocked / failed / recently completed 节点。

### 4.2 后续模式：完整拓扑总览

完整展开不是默认模式，只作为调试和设计审查模式。

它需要后端提供稳定数据：

- 完整节点；
- 完整边；
- 子图层级；
- 边状态；
- 布局坐标或可复现布局输入。

后端没有这些投影前，前端不伪造完整拓扑。

## 5. 页面结构

### 5.1 一级项目管理页

建议结构：

```text
图任务
├─ 顶部工具条
├─ 图任务定义选择区
└─ 项目实例表格 / 紧凑列表
   ├─ 运行中
   ├─ 需处理
   ├─ 待启动
   └─ 已完成
```

项目列表必须使用表格或高密度行，不使用大卡片。

空状态只占一个小区域：

```text
这个图还没有项目。
[创建项目]
```

### 5.2 二级项目控制台

建议结构：

```text
项目控制台
├─ 顶部项目运行条
├─ 左侧节点导航
├─ 中心运行焦点图
├─ 右侧节点会话输出
└─ 底部页签
   ├─ 正式库
   ├─ 运行产物
   └─ 运行记录
```

项目库不再长期挤在主画布下方。它应该是二级控制台里的页签或可展开区域。

## 6. 节点会话设计

每个节点本质是一个配置好的单 agent。

前台应支持：

- 节点卡片直接显示最新输出摘要；
- 点击节点后右侧显示该节点会话；
- 可打开完整节点会话；
- 项目主会话与节点会话分开显示；
- 没有节点会话时显示明确原因，而不是大空白。

本轮不做节点之间自由交流 UI。节点交流属于边协议和运行时能力，前台只展示已经发生的运行结果。

## 7. 文件管理设计

项目文件管理必须从“挤在页面底部”升级为项目资产区。

本轮最小可用能力：

- 文件树；
- 文件搜索；
- 文件编辑；
- Markdown 预览；
- 保存；
- 运行产物列表；
- 点击产物打开文件。

不伪造后端没有的能力：

- diff；
- 版本历史；
- 删除；
- 重命名；
- 产物归档审批。

文件管理应该在二级控制台中通过页签进入，避免每次监控运行都占用主画布空间。

## 8. 数据加载策略

一级页面加载：

1. 图任务定义列表；
2. 默认选中推荐图或最近使用图；
3. 只加载当前图的项目实例；
4. 不加载 monitor；
5. 不加载文件树；
6. 不加载节点会话。

二级页面加载：

1. 项目实例详情；
2. 当前 monitor；
3. 节点会话摘要；
4. 最近产物；
5. 用户打开“正式库”页签时再加载文件树。

这样可以避免进入图任务页面时就卡顿。

## 9. 组件拆分

不能再用一个巨型 `GraphTaskForegroundView` 承担所有职责。

建议组件：

- `GraphTaskForegroundView`
  - 只负责一级/二级路由状态。
- `GraphTaskProjectManager`
  - 图任务定义选择；
  - 项目实例列表；
  - 创建项目；
  - 启动/继续入口。
- `GraphTaskProjectConsole`
  - 二级项目控制台容器。
- `GraphTaskRunFocusMap`
  - 运行焦点拓扑。
- `GraphTaskNodeInspector`
  - 节点输出和节点会话。
- `GraphTaskProjectLibrary`
  - 项目正式库和运行产物。

CSS 也要按这些组件拆分命名，避免所有东西堆在一个 `.graph-foreground-*` 大块里。

## 10. 视觉与交互标准

新版前台必须高密度、生产化、可扫描。

要求：

- 主要列表使用行布局，不用大卡片；
- 空状态高度不超过 160px；
- 未选项目时不出现运行监控区域；
- 已选项目时首屏必须看到：
  - 项目运行状态；
  - 当前活跃节点；
  - 节点最新输出入口；
  - 最近产物入口；
- 画布必须是主视觉区域，不被文件库挤压；
- 文件库作为页签进入；
- 移动端按层级堆叠，不保留三栏。

## 11. 实施计划

### 阶段 1：冻结旧前台补丁

- 不继续在当前平铺页面上补 CSS；
- 保留现有代码作为临时可运行版本；
- 新建组件结构；
- 明确一级/二级页面状态。

### 阶段 2：重做一级项目管理页

- 图任务定义列表；
- 项目实例高密度列表；
- 状态筛选；
- 创建项目；
- 启动/继续项目；
- 不加载 monitor 和文件树。

### 阶段 3：重做二级项目控制台

- 项目运行条；
- 运行焦点图；
- 节点检查器；
- 节点会话入口；
- 最近产物入口。

### 阶段 4：重做项目资产区

- 正式库页签；
- 文件树；
- 文件编辑/预览；
- 产物列表；
- 点击产物载入文件。

### 阶段 5：删除旧前台结构

- 删除旧平铺式 `GraphTaskForegroundView` 内部结构；
- 删除不用的 CSS；
- 保留任务系统配置页；
- 确认图任务不绑定环境。

### 阶段 6：验证

- `npm exec tsc -- --noEmit`；
- 前端相关测试；
- 浏览器验证 1280×720；
- 浏览器验证 390×844；
- 验证无横向溢出；
- 验证无旧 workbench DOM；
- 验证一级页面不渲染空监控大屏；
- 验证二级页面能进入节点输出和项目库。

## 12. 验收标准

新版完成后必须满足：

- 用户进入图任务页面，第一眼看到的是图任务和项目实例管理；
- 没有项目时，页面不会出现大面积空监控；
- 一个图任务可以清晰管理多个项目实例；
- 点击项目后进入该项目的运行控制台；
- 运行控制台中心是运行焦点图；
- 节点输出在右侧检查器中清楚显示；
- 项目正式库不再挤在主画布底部；
- 文件管理作为项目资产区可用；
- 页面拖动顺畅；
- 首屏信息密度合理；
- 不加载任务系统图编辑器；
- 不绑定任务环境；
- 不伪造后端没有的能力。

## 13. 本次不做

- 完整拓扑总览；
- 拖拽式图编辑；
- 节点级暂停/重跑/跳过；
- 产物归档审批；
- 文件 diff / 历史 / 重命名 / 删除；
- 节点间自由聊天 UI。

这些能力后续应由运行时、边协议、项目资产模型和后端 API 支持后再接入前台。

# 模型决策可信上下文与读取循环修复计划书

日期：2026-06-12  
对象：单 agent task runtime、动态上下文投影、工具结果恢复、文件读取事实、agent_todo、运行监控投影  
依据任务：`taskrun:turn:session-472cddd821744602:22:fcc3d23f`

## 0. 根本原则

本计划不以“系统强制模型进入某一步”为修复方向。模型仍然是语义决策者，runtime 不应替模型决定“现在必须编辑”“现在必须验证”或“现在必须收口”。

本次要修的是：系统提供给模型的事实、边界、恢复建议和进度投影不一致，导致模型在理性判断下持续选择读取、搜索、恢复缓存和查看 todo。也就是说，模型不是单纯失控，而是被上下文可信度设计误导。

目标不是接管模型，而是让模型面对清晰、可信、无冲突的事实：

- 哪些文件内容是当前可信的。
- 哪些只是搜索预览。
- 哪些是旧 replay。
- 哪些范围已经足够支持编辑。
- 哪些恢复工具可用、不可用、失败过。
- `agent_todo` 只是辅助状态，不是交付前置条件。
- 监控只呈现事实，不把低层读取动作伪装成任务推进。

## 2. 当前权威链

当前链路大致是：

```text
User task
-> task contract / planning policy
-> runtime_assembly
-> dynamic_context manager
-> tool_result projector
-> task_state projector
-> bound_task_context
-> compiler prompt packet
-> model action decision
-> admission / action permit
-> tool batch planner
-> tool executor
-> observation / file_state / projection
-> progress presenter / monitor
```

目标权威链应明确为：

```text
RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> EvidenceConfidence
-> ModelTurnDecision
-> ActionPermit
-> RuntimeStartPacket
-> ExecutionLoop
-> MonitorPresentation
```

新增的关键不是 `强制执行阶段`，而是 `EvidenceConfidence`：把已有观察整理成可信度事实，避免模型在 preview、replay、file_state、rehydration、editor dirty、persisted cache 之间自行猜测。

## 3. 控制遗漏点

### 3.1 缺少“已读内容可信度”的统一投影

相关文件：

- `backend/harness/runtime/dynamic_context/task_state_projector.py`
- `backend/harness/runtime/dynamic_context/tool_result_projector.py`
- `backend/harness/runtime/bound_task_context.py`

当前系统有 `file_state`、`read_resource_state`、`task_state_replay_entry`、`rehydration_plan`、`known_task_files`，但没有一个统一字段告诉模型：

```json
{
  "path": "mario.html",
  "current_content_state": "current_snapshot_available",
  "exact_ranges_available": [[1, 967]],
  "content_sha256": "...",
  "stale": false,
  "dirty_editor_conflict": false,
  "safe_to_use_for_edit_planning": true,
  "requires_fresh_read_before_edit": false,
  "fresh_read_reason": ""
}
```

结果是模型只能看到多个局部信号：有覆盖、有 replay、有 replacement、有 preview、有 rehydration instruction。它无法判断哪个信号优先。

修复方向：

- 新增 `evidence_confidence` 投影。
- 按文件聚合 read ranges、content hash、write events、editor dirty、replacement 状态。
- 只表达事实，不建议动作。
- 让模型知道“当前内容是否可信”“是否需要新读”的条件。

### 3.2 缺少“搜索结果不是源码事实”的正向替代说明

`search_text` 和 code locator 投影已经说 `candidate_only`、`must_read_source_before_edit`，但没有把“已读源码窗口覆盖了这些命中区域”反向合并。

结果：

- 搜索命中 `class Mario` 后，模型认为还要 read。
- 但 file_state 可能已经覆盖 `class Mario` 所在范围。
- 系统没有告诉模型“这个搜索命中对应的源码已被当前 read window 覆盖”。

修复方向：

- 在 `evidence_confidence` 中把 locator hits 与 read ranges 合并。
- 对每个 search hit 输出 `covered_by_current_read_range: true/false`。
- 如果覆盖为 true，不写“必须重新读取”；只写“该命中已有当前源码窗口覆盖”。

### 3.3 缺少 `read_persisted_tool_result` 失败后的降级事实

本任务出现两类恢复失败：

- `replacement_id must start with tool_result:`，模型拿到了 `replacement:*`，工具要求 `tool_result:*`。
- 多个 `read_persisted_tool_result` 并行读取触发 `task_tool_batch_group_timeout_after_300s`。

这两个失败没有被转化为清晰事实：

```json
{
  "rehydration_channel": "persisted_tool_result",
  "status": "unreliable_for_this_task",
  "failed_refs": [...],
  "preferred_recovery_fact": "use current file_state read ranges, not persisted replay"
}
```

模型因此持续纠结：一边看到恢复工具，一边看到恢复失败，只能反复改用 `read_file`。

修复方向：

- 对同一任务内失败过的 rehydration channel 建立 `recovery_channel_state`。
- 如果 replacement id 类型不匹配，视为投影 bug，不继续向模型暴露错误类型 ref。
- 如果缓存恢复超时，不再把该缓存恢复路径作为优先能力展示。
- 这里不是禁止模型调用，而是不再投影已知坏的恢复建议。

### 3.4 缺少 `agent_todo` 的权威降级

相关文件：

- `backend/capability_system/tools/tool_units/agent_todo_tool.py`
- `backend/harness/runtime/compiler.py`
- `backend/harness/runtime/progress_presenter.py`

`agent_todo` 的描述说它用于维护 visible todo list，并强调非平凡任务要使用。compiler 又说持续处理流程需要维护步骤状态。

虽然 prompt 写了“步骤状态不能替代真实交付物”，但模型仍然会把 `agent_todo view/replace/start` 当成开始工作的前置步骤。

本任务中 `agent_todo` 实际观察 30 次，却没有产生任何交付推进。

修复方向：

- `agent_todo` 投影改为 `planning_state_only`。
- 在模型可见说明中明确：查看 todo 不是开始编辑的前置条件；如果任务合同和文件证据已足够，可以直接做语义决策。
- progress presenter 不应把 todo 变化算作任务实质推进。
- 监控中单独显示 “计划状态更新”，不要混入“执行操作”。

### 3.5 缺少“任务产出事实”的一等投影

模型和 UI 都能看到读取、搜索、todo，但“是否发生写入”“是否发生验证”“是否产出 artifact”不是最显眼的一等事实。

本任务直到最后才明确 `write_event_count=0`。如果模型每轮都能看到：

```json
{
  "deliverable_progress": {
    "write_event_count": 0,
    "verification_event_count": 0,
    "artifact_count": 0,
    "last_material_progress_at": null
  }
}
```

它更容易自己判断“我一直没有交付”，而不是只看到“我正在读取文件内容”。

修复方向：

- 新增 `deliverable_progress`，来源于 file_state write events、artifact refs、verification events。
- 只呈现事实，不写“必须编辑”。
- progress UI 优先展示 material progress，而不是最近一个低层工具动作。

### 3.6 缺少“运行状态详情轻量化”

健康详情接口一次返回约 5.46 MB，耗时约 15.2 秒。UI 卡顿不是 SSE 本身的问题，而是详情投影把过大的 technical trace 和 prompt slot 信息混入常规详情。

修复方向：

- task detail 默认返回 summary + progress + material facts。
- technical trace 分页/懒加载。
- prompt slot、provider protocol、large replay 只在 debug 视图按需拉取。
- 日志面板继续独立，不和 monitor state 混用。

## 4. 控制矛盾点

### 4.1 “已有完整覆盖”与“编辑前必须重读”冲突

相关代码：

- `backend/harness/runtime/dynamic_context/tool_result_projector.py`
- `backend/harness/runtime/compiler.py`
- `backend/harness/runtime/bound_task_context.py`

当前提示同时存在：

- file_state/read_resource_state：已有 read ranges、coverage、content hash。
- tool_result projector：`For code edits, prefer a fresh read_file call...`
- compiler provider replay：`For code edits, read the exact current target line window...`
- bound context：`For a known path, use read_file or path_exists directly before line-level edits...`

问题不是这些话单独错误，而是它们缺少条件。

正确表达应该是：

- 如果当前目标范围已有 current exact read window，且 content hash 未变化，则可作为编辑计划依据。
- 如果要做严格 old_text replacement，且目标 window 未覆盖或 editor dirty，则重新 read。
- 如果只是全文件重写或生成新版本，不需要为了 old_text 再读同一窗口。
- 如果 search hit 已被 current read range 覆盖，不再把 search hit 标成必须 read。

### 4.2 `replacement:*` 与 `tool_result:*` 类型冲突

`read_persisted_tool_result` 工具要求 `replacement_id` 示例为 `tool_result:<digest>`，但 dynamic context replacement store 产出的 ref 是 `replacement:<digest>`。

证据：

- offset 20：`Read persisted tool result failed: replacement_id must start with tool_result:`
- 工具参数实际包含：`replacement:e88b9d6ae5be695d3074b50d`

这是结构性冲突，不是模型错误。模型拿系统投影给的 id 去调用系统工具，失败后自然会降低对缓存恢复的信任。

修复方向：

- 统一 ref 类型，不允许模型可见层同时出现 `replacement:*` 和工具要求的 `tool_result:*`。
- `replacement_ref` 只作内部动态上下文记录。
- 可调用工具参数必须由 `rehydration_plan.next_request.args` 产生，且测试覆盖实际可调用。
- 如果无法生成可调用参数，就不要投影 `read_persisted_tool_result` 能力。

### 4.3 `read_resource_state` 说“available”，replay 又说“preview only”

`read_resource_state` 的 reliability note 说这是 previously returned read windows，模型自行决定是否需要更多上下文。与此同时 replay entry 的 evidence policy 说 visible content 是 preview，必须读当前 source。

这会形成双重含义：

- 文件状态层：你已经有可用读窗口。
- replay 层：你看到的只是预览，不能依赖。

模型为了安全，只能选择重读。

修复方向：

- 区分两个问题：`content availability` 与 `prompt visibility`。
- 如果内容不在当前 prompt，但在 file_state 中可确认当前，投影为 `not_visible_but_current_range_known`。
- 如果内容在当前 prompt 且未截断，投影为 `visible_exact_current_window`。
- 如果只是搜索摘要，投影为 `locator_only`。
- 不再用一条泛化的 `preview only` 覆盖所有情况。

### 4.4 `agent_todo` 既说“必须维护”，又说“不能替代交付”

这两句话都会被模型执行：

- “必须维护”会触发 view/replace/start。
- “不能替代真实交付物”又让模型无法把 todo 视为完成证据。

于是 todo 成为永远可以继续做但永远不产生交付的动作。

修复方向：

- 对 task execution：`agent_todo` 是 optional progress ledger，不是 required step。
- 只有在用户明确要求计划跟踪、任务被拆分、或模型自己需要记录时才使用。
- `agent_todo view` 不应被 prompt 暗示为恢复任务的默认动作。

### 4.5 `task_environment` 与 runtime state root 权威混淆

相关代码：

- `backend/harness/runtime/dynamic_context/manager.py`

当前 `dynamic_context_storage_root` 从 `runtime_assembly.task_environment.storage_space.runtime_state_root/cache_root/environment_storage_root` 取 replacement store 根目录。

这会造成权威混淆：

- `task_environment` 应是任务环境/提示边界，不应成为 runtime state storage 的唯一来源。
- runtime state root 应属于 runtime host / storage service / task run state。
- 如果模型可见环境和内部存储边界混在同一个结构，后续很容易再次出现“为了提示词环境而要求 runtime_state_root”的设计倒挂。

修复方向：

- 新增内部 `runtime_storage` 或 `runtime_state_ref`，由 runtime host 注入。
- dynamic context manager 只从内部 storage ref 取 replacement store。
- `task_environment.storage_space` 若保留，只能作为 model-visible resource description，不作为硬依赖。
- 测试证明缺少 `task_environment.storage_space.runtime_state_root` 时，runtime 仍能使用内部 state root。

### 4.6 探索 advisory 说“你可能在探索”，但同时把选择完全抛回模型

相关代码：

- `backend/harness/loop/task_executor.py`

当前 exploration advisory 是 `non_blocking: true`，问题列表包含“是否需要更多读取、搜索、委派、实现、验证或收口，由模型根据任务合同自行裁决”。

这符合不强制原则，但它没有提供可决策事实。例如：

- 已连续探索多少次。
- 哪些探索覆盖了哪些文件范围。
- 是否已有 material progress。
- 哪些读取是重复覆盖。
- 哪些恢复路径失败过。

所以模型看到 advisory 后仍然不知道“我是否已经有足够证据”，只能继续探索。

修复方向：

- 保持 non-blocking。
- 把 advisory 从泛化问题改为事实面板：
  - `consecutive_context_actions`
  - `covered_files`
  - `uncovered_requested_regions`
  - `material_progress_since_last_context_action`
  - `failed_recovery_channels`
  - `duplicate_semantic_reads`
- 不写“应当编辑”，只让模型看到“你已经做了什么，还缺什么事实”。

### 4.7 重复只读 guard 只按完全相同参数，不按语义覆盖

相关代码：

- `backend/harness/loop/task_executor.py`

当前 duplicate guard 的 fingerprint 以工具名和标准化参数为准。模型只要换 line range 或 read_intent，就不会被识别为重复。

这不是要禁止模型读取，而是当前系统无法告诉模型：这些读取在语义上覆盖了同一目标。

修复方向：

- 不把它做成 deny。
- 增加 `semantic_read_overlap` 事实：
  - `same_path`
  - `overlap_ratio`
  - `already_covered_by`
  - `new_information_ratio`
  - `read_intent`
- 当模型请求新 read 时，observation 返回“新增覆盖比例”，作为下一轮模型事实。

## 5. 目标设计

### 5.1 EvidenceConfidenceProjection

新增动态投影：

```json
{
  "evidence_confidence": {
    "authority": "harness.runtime.dynamic_context.evidence_confidence",
    "files": [
      {
        "path": "mario.html",
        "status": "current",
        "content_sha256": "...",
        "total_lines": 967,
        "exact_read_coverage": {
          "covered_ranges": [[1, 967]],
          "coverage_ratio": 1.0,
          "has_more": false
        },
        "prompt_visibility": "preview_or_refs",
        "safe_for": ["architecture_planning", "symbol_location", "edit_planning"],
        "fresh_read_required_for": [],
        "fresh_read_reasons": [],
        "write_event_count": 0,
        "editor_dirty_conflict": false
      }
    ],
    "recovery_channels": [
      {
        "name": "read_persisted_tool_result",
        "status": "failed_or_unreliable",
        "reason": "replacement_id_type_mismatch_or_timeout"
      }
    ],
    "material_progress": {
      "write_event_count": 0,
      "verification_event_count": 0,
      "artifact_count": 0
    }
  }
}
```

这个投影不发命令、不替模型裁决，只提供可信事实。

### 5.2 Read Evidence Contract

把当前分散在 tool_result_projector、compiler、bound_task_context 的读取规则收束成一个契约：

```text
read_file exact window 是源码事实。
search_text/search_files 是定位事实，不是源码事实。
persisted tool result 是恢复通道，不是优先源码通道。
file_state coverage 是当前性事实，但不保证内容完整可见于 prompt。
是否需要重读由模型判断，但系统必须说明缺失的是 currentness、visibility、coverage 还是 editor dirty。
```

### 5.3 Recovery Channel Contract

`read_persisted_tool_result` 只能用于恢复 omitted output。它不能与当前工作区源码事实竞争。

规则：

- 如果目标是代码编辑，当前工作区 `read_file` 优先于 persisted replay。
- 如果 persisted 参数不可调用，不暴露该能力。
- 如果 persisted channel 在本 task 内超时，投影为 degraded。
- 如果 replacement id 类型不兼容，视为系统 bug，修复 ref 生成。

### 5.4 Todo Contract

`agent_todo` 的新说明：

```text
agent_todo 是任务过程记录。它可以帮助你管理多步骤任务，但不是开始工作的前置条件，也不是完成证据。
如果任务合同和当前证据已经足够，你可以直接选择下一步语义动作。
只有真实文件写入、artifact、验证观察或明确阻塞才构成交付进展。
```

### 5.5 Monitor Projection Contract

监控面板只显示事实：

- 当前 task 状态。
- 最近动作。
- material progress。
- context actions 数量。
- recovery failures。
- 是否有写入/验证。

不要把“读取文件内容”作为主进度标题长期展示。应该展示：

```text
上下文收集中：已读取 mario.html 18 次，写入 0 次，验证 0 次。
```

这不是让系统判断任务错了，而是让用户看到真实运转状态。

## 7. 删除和重写标准

必须删除或重写：

- 任何让 `replacement:*` 被当成 `read_persisted_tool_result.replacement_id` 的 model-visible 路径。
- 任何无条件宣称 code edit 前必须 fresh read 的泛化提示。
- 任何把 `agent_todo view` 暗示为恢复任务默认动作的提示。
- 任何依赖 `task_environment.storage_space.runtime_state_root` 作为唯一 runtime state root 的实现。
- 任何把 read/search/todo 当成 material progress 的监控投影。

可以保留：

- duplicate read-only guard，但它只能作为错误/重复事实，不作为语义门禁。
- exploration advisory，但必须变成事实面板，不写成模型行动建议。
- file_state，但必须由 evidence_confidence 统一解释。

## 9. 不做的事

- 不增加“系统强制模型必须编辑”的 action gate。
- 不把 read/search/todo 超过阈值直接改写成 respond/block。
- 不通过删工具或降低工具能力来掩盖问题。
- 不用 UI 文案假装任务推进。
- 不把旧兼容链路留下来和新 evidence confidence 并行竞争。

## 10. 结论

这次不流畅的根因是可信上下文架构缺位，不是模型单点失误，也不是 SSE 单点问题。

模型多次准备实施，但它收到的系统事实互相冲突：

- 有覆盖，但又说 preview only。
- 有恢复工具，但恢复 id 类型不匹配。
- 有 todo，但 todo 不是交付物。
- 有探索 advisory，但没有可行动的事实摘要。
- 有监控进度，但主标题长期展示低层读取动作。
- 有 task_environment，但它混入了 runtime state storage 权威。

修复必须从权威边界开始：runtime 只提供准确事实和清晰边界，模型继续拥有语义决策权。只有这样，模型才会自然地把“我已有足够证据”判断为可信，而不是继续反复读取。

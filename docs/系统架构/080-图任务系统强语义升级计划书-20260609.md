## 1. 结论先行

现有 graph harness 可以作为升级底座，不建议推倒重来。

但它目前只支持“图运行内核”的一部分能力：

- 有发布物 `GraphHarnessConfig`。
- 有节点、边、资源、contracts、state、work order。
- 有 scheduler view、loop state、structure hash、FlowPacket、edge protocol index。
- 有 memory / artifact / file / revision / event / audit / dependency 等边语义雏形。

它还不支持 079 中定义的完整系统：

- 没有一等 `CompiledEdgeContract`。
- 没有 `EdgeContractPrototype` 原型库。
- 没有强编译器从极简 DSL 合成完整节点、资源、边契约。
- 没有节点级 effective environment 贯穿 runtime。
- 没有统一边协议执行器。
- 没有内部 `a2a_session` session executor。
- 没有外部 A2A adapter。
- 没有节点级单独交互契约 `NodeInteractionContract`。
- 没有内置配置节点 `__configurator__`。
- 没有内置监管节点 `__supervisor__` 和 `MaintenanceContract`。

推荐路线：

```text
保留 graph harness 作为 runtime 内核
新增强编译器作为配置权威
新增 CompiledEdgeContract 作为交接核心
新增 EdgeContractPrototype 作为用户选择入口
新增 node effective environment 和 resource contract
逐步把 runtime 从 edge_type 猜测迁移到 edge_contract_index
```

## 2. 外部成熟方案调研

### 2.1 LangGraph

成熟点：

- LangGraph 是低层 agent orchestration runtime，强调 durable execution、streaming、human-in-the-loop 和 persistence。
- 它的 persistence 通过 checkpoint 支持 human-in-the-loop、memory、time travel、fault tolerance 和 pending writes。
- 它明确 super-step / checkpoint boundary，对恢复和并行写入很有参考价值。

可借鉴：

- checkpoint boundary。
- pending writes。
- thread / state snapshot。
- human interrupt / resume。
- graph compile 后运行。

不建议直接替换：

- 我们已有自己的 `GraphHarnessConfig`、task environment、artifact、memory、permission、work order、runtime facade。
- LangGraph 的 state graph 能力强，但不会自动解决我们的边契约、资源治理、agent profile、节点环境、artifact receipt 和 memory commit 语义。
- 直接替换会丢失现有项目特化结构。

建议：

```text
借鉴 LangGraph 的 durable execution 和 checkpoint 概念。
不把 LangGraph 作为主 runtime 替换 graph harness。
```

### 2.2 Temporal

成熟点：

- Temporal 是成熟 durable execution 平台，核心价值是 crash-proof execution，保证应用在崩溃、网络故障或基础设施中断后从中断处恢复。
- 对长任务、外部 signal、activity、retry、timeout、workflow history 非常成熟。

可借鉴：

- durable execution 原则。
- event history。
- workflow/activity 分离。
- signal / query / update。
- retry / timeout / idempotency。

不建议第一版直接引入：

- 引入 Temporal 需要 worker、task queue、workflow definition、activity definition、deployment 和状态存储体系，工程改造面很大。
- 我们当前 graph harness 已经有 task run、work order、state、runtime objects、checkpoint 结构，第一阶段应该先把语义和契约补齐。

建议：

```text
第一版不引入 Temporal。
把 Temporal 作为后续可选 DurableBackendAdapter 研究对象。
```

### 2.3 A2A Protocol

成熟点：

- A2A 是面向独立 agent 系统通信和互操作的开放协议。
- 它定义 Agent Card、Message、Task、Artifact、Part、Streaming、Push Notification、Context 等概念。
- 它强调 agent 不需要共享内部状态、记忆或工具，只通过声明能力和交换信息协作。

可借鉴：

- 外部 agent capability discovery。
- message / task / artifact 分离。
- context id / task id。
- streaming / push。
- secure exchange。

不应误用：

- A2A 只是边协议之一，不是图系统本体。
- 它适合外部 agent 互操作，不适合替代所有边交接。
- 普通交付审核、资源读取、条件路由、memory commit 不需要 A2A。

建议：

```text
第一版实现内部 a2a_session 契约和 session history。
第二版实现 ExternalA2AAdapter，接 Agent Card 和外部任务通信。
```

### 2.4 OpenAI Agents SDK

成熟点：

- Agents SDK 的 handoff 把 agent handoff 作为工具暴露给 LLM。
- 支持 handoff input schema、input filter、handoff history 控制。
- 内置 tracing，覆盖 LLM generation、tool call、handoff、guardrails、自定义事件。

可借鉴：

- handoff metadata。
- input filter。
- handoff history visibility。
- trace / span。
- guardrail 概念。

不建议作为主图系统：

- SDK handoff 偏向“LLM 在运行时选择转交哪个 agent”。
- 我们的图任务需要发布契约先行、边契约可审查、runtime 不猜测。
- 它适合作为单 agent node runtime 或局部 agent executor 参考，不适合替代 graph harness。

建议：

```text
借鉴 handoff input filter、history visibility 和 tracing。
不把 SDK handoff 作为图任务边契约权威。
```

### 2.5 Microsoft Agent Framework / AutoGen

成熟点：

- Microsoft Agent Framework 的 group chat orchestration 明确有 orchestrator，负责下一发言者选择、上下文同步、max rounds、termination condition。
- AutoGen GraphFlow 支持 sequential、parallel fan-out、conditional branching、join、loops with explicit exit condition，但官方也标注部分 GraphFlow API 仍是 experimental。

可借鉴：

- group chat 需要 orchestrator。
- max rounds 和 termination condition 必须存在。
- 多 agent 对话不等于所有图边。
- GraphFlow 的 fan-out、branching、loop exit 是实用模式。

不建议直接替换：

- 我们需要边契约、资源节点、环境锁、artifact/memory receipts、强编译报告。
- AutoGen / Agent Framework 更偏 agent team runtime，不是我们的任务环境和资源治理系统。

建议：

```text
借鉴 group chat / fan-out / termination condition。
不作为主 runtime 替代。
```

## 3. 成熟方案选择

推荐组合：

```text
自有 GraphHarness Runtime
+ LangGraph checkpoint / super-step 思路
+ Temporal durable execution 原则
+ A2A 外部 agent adapter
+ OpenAI Agents SDK handoff visibility / trace 思路
+ Microsoft/AutoGen group chat termination 思路
```

不推荐：

```text
直接替换成 LangGraph
直接替换成 Temporal
直接替换成 AutoGen GraphFlow
把 A2A 当成所有边协议
把 OpenAI Agents SDK handoff 当成边契约权威
```

理由：

- 我们的核心优势是环境、资源、artifact、memory、权限、任务系统已经有本地结构。
- 成熟框架能提供不变量，但不能直接表达我们的业务语义。
- 正确升级是“吸收成熟原则，修正本地结构”，不是“换壳”。

## 7. 分阶段计划

## Phase 0：锁定概念与验收口径

目标：

- 以 079 为准，锁定 Node、Resource、Edge、Contract、Prototype、Compiler、Runtime 概念。
- 确认 `EdgeContract` 是交接核心。
- 确认 A2A 只是边协议之一。

交付物：

- 最终概念定义文档。
- 原型列表初版。
- 升级验收标准。

完成标准：

- 用户确认概念边界。
- 不再用“场景”替代“边契约”。
- 不再把 A2A 当默认交接。

## Phase 1：新增编译期模型，不改 runtime

目标：

- 新增 `GraphCompilationUnit`。
- 新增 `CompiledNodeContract`、`CompiledResourceContract`、`CompiledEdgeContract` 的数据模型。
- 当前 graph 仍能发布原 `GraphHarnessConfig`。
- 先生成 shadow contract index 和 compile report，不改变运行行为。

建议新增：

```text
backend/task_system/compiler/models.py
backend/task_system/compiler/graph_compiler.py
backend/task_system/compiler/edge_contract_models.py
backend/task_system/compiler/node_contract_models.py
backend/task_system/compiler/resource_contract_models.py
backend/task_system/compiler/compile_report.py
```

完成标准：

- 现有 TaskGraph 能编译出 `GraphCompilationUnit`。
- compile report 能解释节点、资源、边、环境、权限来源。
- `CompiledEdgeContract` 能从当前 edge fields 生成 shadow index。
- 不改变 graph runtime 行为。

验证：

- 新增 `backend/tests/graph_compilation_unit_regression.py`。
- 新增 `backend/tests/graph_edge_contract_shadow_regression.py`。

## Phase 2：一等边契约与 edge_contract_index

目标：

- `CompiledEdgeContract` 成为边交接权威。
- `edge_protocol_index` 升级为或并入 `edge_contract_index`。
- `edge_type` 不再承担所有语义，只作为 legacy projection 或展示字段。

核心改动：

- 在 `GraphHarnessConfig.contracts` 中加入：

```yaml
edge_contract_index:
  edge.draft.review:
    relation: ...
    protocol: ...
    scheduler: ...
    packet: ...
    reliability: ...
    security: ...
    failure: ...
    trace: ...
```

- `GraphStateMachine.initial_edge_states()` 从 `edge_contract_index` 读取 ack、protocol、trace policy。
- `FlowPacket` 从 edge contract 读取 packet contract、visibility、target slot。
- publisher 对每条边生成 edge contract 并 fail closed 校验。

完成标准：

- `draft -> review` 能生成完整 `CompiledEdgeContract`。
- runtime 仍能执行现有 handoff。
- 旧 edge dict 不再是协议权威。

验证：

- edge contract 缺 ack policy 时拒绝发布。
- payload_contract 与 target input 不兼容时拒绝发布。
- resource node 不能被编译为普通 dependency 输出源。

## Phase 3：节点与资源契约

目标：

- 明确 `ExecutableNode / ResourceNode / ControlNode`。
- 为每个节点生成 `CompiledNodeContract`。
- 为资源生成 `CompiledResourceContract`。
- 实现 node effective environment index。

核心改动：

```yaml
node_contract_index:
  draft:
    executor: agent
    input_contract: ...
    output_contract: ...
    prompt_contract: ...
    environment_lock: ...
    permission_ceiling: ...

resource_contract_index:
  memory.world:
    resource_kind: memory_repository
    read_policy: ...
    write_candidate_policy: ...
    commit_policy: ...
```

完成标准：

- agent node、resource node、control node 可明确区分。
- graph control environment 不覆盖 node effective environment。
- work order、TaskRunContract、artifact receipt、memory receipt 使用节点环境。

验证：

- 一个图内不同 agent 节点使用不同环境。
- 修改 agent profile 后已发布图恢复仍使用已锁定环境。
- runtime settings patch 不能切换节点环境。

## Phase 4：边协议执行器

目标：

- 把 runtime 从 `edge_type` 分支猜测升级为 `protocol.kind` 执行。

新增执行器：

```text
FlowPacketEdgeExecutor
ResourceEdgeExecutor
SessionEdgeExecutor
EventAuditEdgeExecutor
ModulePortEdgeExecutor
```

第一批支持：

- `control_dependency`
- `node_handoff`
- `conditional_route`
- `review_feedback`
- `resource_read`
- `resource_write_candidate`
- `resource_commit`
- `event_signal`
- `barrier_join`

暂不支持或仅契约化：

- 外部 A2A。
- 复杂 group chat。

完成标准：

- FlowPacket 由 `CompiledEdgeContract.packet` 生成。
- resource read / commit 由 ResourceEdgeExecutor 处理。
- conditional route 不再散落在 loop/revision 特例里。
- barrier join 有 branch identity。

验证：

- 条件边引用不存在 field 时 fail closed。
- memory candidate 未审核不能变成 committed。
- event edge 不阻塞主流程。
- barrier join 缺 branch identity 时拒绝发布。

## Phase 4.5：旧 Loop 体系升级为 LoopContract

目标：

- 把现有 `loop_frames`、`node.loop.route_policy`、`route_history`、`iteration_results` 升级为一等 `LoopContract`。
- loop 不再是散落在节点 metadata 和 runtime helper 里的旧结构。
- 循环、返修、批处理、map-reduce、分批推进都通过可编译、可审查、可恢复的 loop contract 表达。

当前问题：

- 现有 loop 主要依赖 `loop_frames` 和节点上的 `route_policy`。
- `metric_target`、`progress_receipt`、`continue_node_id`、`exit_node_id` 等逻辑集中在 graph loop runtime 中。
- loop 的路由、退出、迭代输入 patch、聚合结果、失败恢复尚未统一到边契约和编译报告中。

目标模型：

```yaml
loop_contract_index:
  loop.chapter_batch:
    loop_kind: bounded_iteration
    scope_node_ids: [draft, review, revise]
    entry_node_id: draft
    router_node_id: review
    continue_node_id: draft
    exit_node_id: publish
    iteration_identity:
      strategy: template
      template: chapter.{chapter_index}
    bounds:
      max_iterations: 20
      cursor_key: chapter_index
      exit_when:
        field: review.verdict
        equals: pass
    state_policy:
      preserve_iteration_results: true
      reset_scope_on_continue: true
    edge_contracts:
      continue_edge: edge.review.draft.revision_loop
      exit_edge: edge.review.publish.pass
    checkpoint_policy:
      persist_each_iteration: true
```

需要迁移的旧结构：

```text
graph_config.loop_frames
node.loop.route_policy
loop_state.frames
loop_state.route_history
loop_state.iteration_results
synthetic loop_iteration_results context
```

迁移策略：

1. Phase 4.5-A：从旧 `loop_frames` 生成 shadow `loop_contract_index`。
2. Phase 4.5-B：编译器直接生成 `LoopContract`，runtime 同时校验旧投影一致。
3. Phase 4.5-C：runtime 只读取 `loop_contract_index`。
4. Phase 9：删除旧 `route_policy` 权威路径，保留为 DSL 迁移输入或 UI 展示投影。

完成标准：

- 返修闭环不再依赖硬编码 revision 特例。
- 循环必须有最大轮次或明确退出条件。
- 每次 iteration 都有稳定 identity、checkpoint 和结果归档。
- loop 的 continue / exit 都是边契约，不是 runtime 隐式跳转。
- compile report 能解释 loop 的入口、退出、上限、状态保存和恢复规则。

验证：

- 缺 max_iterations 且无 exit condition 时拒绝发布。
- continue edge / exit edge 缺失时拒绝发布。
- resume 后不会重复提交上一轮 memory / artifact。
- iteration_results 可以作为受控 packet 注入汇总节点。

## Phase 5：边契约原型库与最小 DSL

目标：

- 新增 `EdgeContractPrototype` registry。
- 用户选择原型，编译器生成 edge contracts。

第一批原型：

```text
交付审核
返修闭环
规划交付
执行验收
证据移交
协商定案
并行会审
汇总裁决
```

核心改动：

```text
AuthoringSpec edge:
  draft -> review: 交付审核

Compiler:
  prototype.review_submission
  -> edge_contract.node_handoff
```

完成标准：

- 用户不写协议字段也能发布图。
- compile report 解释原型选择和协议生成。
- 低置信度时询问原型选择，不询问底层字段。

验证：

- 最小 DSL 两节点图可发布并运行。
- 原型生成的 prompt contract 是 agent 可执行指令。
- 原型缺必要输入时 fail closed。

## Phase 5.5：部署包与轻量发布体验

目标：

- 证明用户可以比较轻松地部署图任务。
- 不让用户手动理解低层 `GraphHarnessConfig`。
- 把“配置 -> 编译 -> 预览 -> 发布 -> 启动 -> 监控 -> 恢复”做成标准部署链路。

新增概念：

```text
GraphDeploymentPackage
```

建议结构：

```yaml
deployment_package:
  package_id: gdeploy.chapter_pipeline.v1
  graph_id: graph.novel.chapter_pipeline
  authoring_spec_ref: ...
  compilation_unit_ref: ...
  graph_config_id: ghcfg:...
  compile_report_ref: ...
  required_credentials:
    - credential.external.research_agent
  required_environments:
    - env.creation.novel
    - env.review.quality
  required_resources:
    - memory.world
    - artifact.chapter_repository
  risk_summary:
    permission_ceiling: ...
    external_endpoints: ...
  launch_contract:
    required_inputs:
      - chapter_outline
    optional_inputs:
      - style_guide
  monitor_profile:
    default_view: graph_task_run
```

用户部署路径：

```text
选择图任务模板
-> 填目标和少量必要输入
-> 选择 agent / environment 或使用推荐配置
-> 编译预览
-> 修复 compile report 中的阻塞项
-> 发布 deployment package
-> 一键启动 graph run
-> 监控 / 暂停 / 恢复 / 审查
```

必须对用户隐藏的低层字段：

- raw edge protocol fields。
- raw contract_bindings。
- runtime authority map。
- internal packet ids。
- structure hash。
- edge executor internals。

必须对用户展示的高级信息：

- 这个图会做什么。
- 需要哪些 agent。
- 需要哪些资源和凭证。
- 每条边是什么交接原型。
- 哪些节点会写入文件、artifact、memory。
- 哪些步骤需要人工批准。
- 编译是否通过。
- 启动后在哪里监控。

完成标准：

- 用户可以从一个 `GraphTaskTemplate` 创建 `AuthoringSpec`。
- 用户不写底层契约也能生成 `GraphDeploymentPackage`。
- compile report 阻塞项可操作。
- 发布后返回可启动的 graph run entry。
- 监控页面能按节点、边、资源、loop iteration 展示状态。

验证：

- 新用户只填目标、节点角色和资源选择，即可发布一个两节点审核图。
- 缺凭证、缺环境、缺资源时不能发布，但报告能说明需要补什么。
- deployment package 发布后不受模板后续变化影响。
- 同一 package 可重复启动不同 graph run。

## Phase 5.6：内置配置节点与监管基础

目标：

- 新增内置配置节点 `__configurator__`，直接帮助用户配置任务图草案。
- 新增内置监管节点 `__supervisor__`，提供最小运行健康检查。
- 第一版不实现“每个节点都可以单独交流”。
- 第一版不实现 `NodeInteractionContract` 和 `NodeInteractionController`。

阶段边界：

```text
先做基本功能：
配置 -> 编译 -> 发布 -> 启动 -> 运行状态检查

暂不做节点单独交流：
节点私聊 -> 节点上下文投影 -> 用户输入包注入 -> 节点级重试入口
```

新增控制面对象：

```text
GraphDraftPatch
ContractDraft
ConfiguratorWriteContract
SupervisorObservation
HealthStatus
RiskAlert
MaintenanceActionCandidate
AutoRecoveryReceipt
```

内置配置节点：

```text
__configurator__
```

职责：

- 询问用户目标和约束。
- 配置节点、资源、agent role、边契约原型。
- 生成并应用图草案配置 patch。
- 生成节点、资源、边契约草案。
- 生成 `AuthoringSpecDraft`。
- 解释 compile report。
- 生成 compile error 修复建议。

建议写入链路：

```text
用户目标
-> __configurator__
-> GraphDraftPatch / ContractDraft
-> DraftGraphStore
-> GraphCompiler validation
-> GraphCompilationUnit / GraphDeploymentPackage
```

建议新增 `ConfiguratorWriteContract`：

```yaml
configurator_write_contract:
  can_write:
    - authoring_spec_draft
    - node_seed_drafts
    - resource_binding_drafts
    - edge_prototype_selections
    - node_contract_drafts
    - resource_contract_drafts
    - edge_contract_drafts
  can_apply_to:
    - draft_graph_store
  must_validate_with:
    - graph_compiler
  cannot_write:
    - published_graph_contract
    - runtime_graph_state
    - credential_secret_value
    - permission_grant
```

边界：

- 可以直接配置草案图。
- 可以直接输出契约草案。
- 可以直接应用配置 patch 到 `DraftGraphStore`。
- 不能直接发布 graph config 或 `PublishedGraphContract`。
- 不能直接创建或扩大权限。
- 不能直接写 credential。
- 不能把用户自然语言直接当 runtime prompt。
- 所有草案必须经过 GraphCompiler fail-closed 校验。

推荐 prompt contract：

```text
你是图任务配置助手。
你负责把用户的目标直接配置成可编译的图任务草案。
你需要识别并配置需要哪些节点、资源、agent 角色、边契约原型和契约草案。
你可以输出结构化配置 patch，用于写入草案图。
你不能发布任务图，也不能绕过编译器校验。
如果缺少关键环境、凭证、资源或验收标准，你必须指出缺口并请求用户补充。
你的输出必须是结构化配置 patch、契约草案和解释，不是运行时任务结果。
```

内置监管节点：

```text
__supervisor__
```

职责：

- 观察 graph run 状态。
- 检查 blocked / failed / timeout。
- 检查 edge packet 丢失或不符合 contract。
- 检查 loop 是否超限或无法退出。
- 检查 memory candidate 是否错误可见。
- 输出健康状态、风险提示和维护建议。

第一版暂不做：

- 不做复杂自动修复。
- 不做节点级对话。
- 不做自动修改契约。
- 不做外部 endpoint 深度探活。

边界：

- 不能替业务节点决定结果。
- 不能强行通过审核。
- 不能修改发布契约。
- 不能扩大权限。
- 不能切换 node effective environment。
- 不能强制提交 memory 或覆盖 artifact。

建议 `MaintenanceContract`：

```yaml
maintenance_contract:
  auto_actions:
    - emit_health_alert
    - emit_missing_receipt_diagnostic
    - mark_recoverable_blocked_node
  require_human_approval:
    - requeue_failed_node
    - skip_node
    - override_result
    - force_commit_resource
    - mutate_contract
  max_auto_recovery_attempts: 2
  audit_required: true
  receipt_required: true
```

系统节点呈现：

```text
System Control Lane
  __configurator__
  __supervisor__

User Graph Lane
  draft -> review -> revise
  memory.world
  artifact.chapter
```

完成标准：

- `__configurator__` 能根据目标直接配置草案图。
- `__configurator__` 能生成节点、资源、边契约草案。
- `__configurator__` 不能绕过编译器发布。
- `__supervisor__` 能发现 blocked / failed / timeout。
- `__supervisor__` 的自动维护动作受 `MaintenanceContract` 限制。
- `__supervisor__` 不能强行通过审核、扩大权限或修改契约。
- 配置 patch、编译结果、监管观察和维护建议都有 trace 和 receipt。

验证：

- 未发布图只能进入配置解释和草案生成，不能启动 runtime。
- 请求修改 agent、prompt、environment 时必须回到 compiler。
- `__configurator__` 生成的契约草案缺字段时 compiler fail closed。
- `__supervisor__` 能输出 blocked / failed / timeout 的健康诊断。
- 第一版没有节点单独交流入口，也不会把用户消息注入节点上下文。

## Phase 6：内部 A2A Session

目标：

- 支持内部 agent 之间有限轮讨论。
- A2A 是 `a2a_session` 边协议，不是默认交接。

核心对象：

```text
EdgeSession
A2ASessionContract
SessionTurn
ConversationArtifact
TranscriptRef
```

运行原则：

- 必须有 max turns。
- 必须有 stop condition。
- 必须产出 conversation artifact。
- transcript 默认不注入下游，只给 summary/ref。

完成标准：

- `reviewer <-> writer: 协商定案` 运行为内部 session。
- session 可 checkpoint / resume。
- 下游拿到结构化 conclusion artifact。

验证：

- 没有 max_turns 拒绝发布。
- 没有 output artifact contract 拒绝发布。
- transcript 不能默认污染下游上下文。

## Phase 7：外部 A2A Adapter

目标：

- 接入外部 A2A agent，而不是让外部协议污染内部 graph contract。

核心对象：

```text
ExternalA2AAgentResource
AgentCardCache
A2ATaskMapping
A2AArtifactMapping
A2ACapabilityValidation
```

配置模型：

```yaml
external_agents:
  - id: external.research_agent
    protocol: a2a
    agent_card_url: https://agent.example.com/.well-known/agent-card.json
    endpoint_url: https://agent.example.com/a2a
    auth:
      credential_ref: credential.external.research_agent
      scheme: bearer
    capability_requirements:
      content_types:
        - text/plain
        - application/json
      streaming: optional
      push_notifications: optional
    data_policy:
      cross_environment_policy: packet_only
      expose_artifacts: refs_only
      expose_memory: committed_refs_only
```

需要区分两类外部对象：

1. 外部 A2A Agent

它已经暴露 Agent Card 和 A2A endpoint。我们只需要做 capability discovery、认证、任务映射、artifact 映射和 trace 映射。

2. 裸模型 API

例如只有模型 URL 和 API key，没有 A2A Agent Card、Task、Artifact、Message 生命周期。这种不能直接算 A2A agent。需要先由我们提供 `ExternalModelAgentAdapter`，把模型 API 包装成 agent：

```text
ExternalModelProvider
-> ExternalModelAgentAdapter
-> internal/external A2A-compatible endpoint
-> graph a2a_session edge
```

换句话说：

```text
配置 URL + API key 可以让系统调用外部模型。
但只有配置 URL + API key + agent capability / contract，才能成为图任务里的外部 agent。
```

完成标准：

- 外部 agent 可作为 ResourceNode 或 ExternalEndpoint。
- 编译时校验 Agent Card capability。
- runtime 将外部 task/message/artifact 映射为内部 trace 和 artifact ref。

验证：

- 不支持 streaming 的外部 agent 不能发布 streaming edge。
- 外部 artifact 必须映射为内部 artifact ref。
- 外部认证缺失时 fail closed。

## 10. 风险与控制

### 风险 1：旧 edge_type 继续作为隐式权威

控制：

- Phase 2 后 `edge_type` 只作为迁移输入。
- runtime 读取 `edge_contract_index`。
- 删除旧猜测分支。

### 风险 2：原型层喧宾夺主

控制：

- 原型永远不执行。
- 原型只生成 `CompiledEdgeContract`。
- compile report 必须显示最终契约。

### 风险 3：A2A 被滥用

控制：

- A2A 只作为 `a2a_session` 协议。
- 默认优先非对话协议。
- 没有 stop condition / output artifact 不发布。

### 风险 4：节点环境切换破坏恢复

控制：

- 编译期锁定 node effective environment。
- environment lock 进入 structure hash。
- runtime patch 不能改环境。

### 风险 5：长迁移期留下双路径

控制：

- shadow 只允许在 Phase 1。
- Phase 4 runtime 切新契约。
- Phase 9 删除旧权威路径和旧测试。

## 11. 不建议直接拿来用的地方

### 不直接用 LangGraph 替换 harness

原因：

- 我们的资源、环境、artifact、memory、permission 和 task run 已经深度定制。
- LangGraph 是优秀 runtime，但不是我们的领域契约编译器。

### 不直接用 Temporal 承载第一版

原因：

- Temporal 引入成本高。
- 我们当前最大缺口是语义和契约，不是底层 worker 系统。

### 不直接用 AutoGen GraphFlow

原因：

- AutoGen GraphFlow 官方标注 experimental。
- 它更像 agent team workflow，不是资源/环境/契约治理系统。

### 不直接用 OpenAI Agents SDK handoff 作为图边

原因：

- 它是运行时 agent 工具选择。
- 我们需要编译期发布契约。

### 可以直接用 A2A 规范作为外部协议

原因：

- A2A 的 Agent Card、Task、Message、Artifact 很适合外部 agent adapter。
- 但只能作为外部边协议，不是图系统核心。

## 12. 推荐第一轮实施范围

第一轮建议做到 Phase 5.6 的轻量闭环：

```text
Phase 0：概念锁定
Phase 1：编译期模型
Phase 2：一等边契约
Phase 3：节点与资源契约
Phase 4：边协议执行器
Phase 4.5：旧 Loop 体系升级为 LoopContract
Phase 5：边契约原型库与最小 DSL
Phase 5.5：部署包与轻量发布体验
Phase 5.6：内置配置节点与监管基础
```

第一轮不做：

- 外部 A2A adapter。
- 每个节点单独交流。
- NodeInteractionContract / NodeInteractionController。
- 节点级用户输入注入和节点级重试入口。
- 完整前端图编辑器重做。
- 完整 group chat 编排系统。
- Temporal backend。
- 全量 AutoGen / LangGraph 替换。

理由：

- 先把内部语义、契约和运行权威做稳。
- 配置节点和部署包先解决“用户轻松部署”的核心体验。
- 监管节点先做健康诊断和受限维护，不提前变成总控 agent。
- 外部 agent 通信等内部边契约稳定后再接。

## 14. 暂定结论

我们可以直接拿来用的不是某个完整框架，而是成熟系统里的工程不变量：

```text
LangGraph：checkpoint / super-step / pending writes
Temporal：durable execution / event history / idempotency
A2A：external agent protocol / Agent Card / Task / Artifact
OpenAI Agents SDK：handoff visibility / input filter / tracing
Microsoft Agent Framework：group chat orchestrator / termination condition
AutoGen：fan-out / conditional branch / loop exit pattern
```

最终方案仍应以我们自己的 graph harness 为核心：

```text
强编译器生成契约
GraphHarness 执行契约
EdgeContract 作为交接核心
Prototype 作为用户入口
A2A 作为边协议之一
```

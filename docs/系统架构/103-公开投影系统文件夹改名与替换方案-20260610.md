# 公开投影系统文件夹改名与替换方案

日期：2026-06-10

状态：替换计划书，等待确认后实施。

## 一、结论

不新增 `V2` 命名，不做新旧双轨。正确做法是：

```text
旧投影实现整体改名隔离
-> 新投影系统占用正式 projection 名称和运行入口
-> 接线切换到新系统
-> 删除旧隔离代码
```

也就是：

```text
projection        新正式系统
projection_legacy 旧系统临时隔离区，只读迁移，不参与运行
```

旧系统不能继续作为 fallback。安全 guard 可以保留，但只能拒绝泄露，不能产出替代正文。

## 二、为什么必须改名隔离

旧投影问题不是单个函数错，而是多个模块同时拥有显示决策权：

```text
task_lifecycle
work_rollout
runtime_monitor_public_projection
public_timeline_stream
public_timeline_projection
public_chat_timeline
session_task_projection
session_timeline
frontend runtime store
frontend fallback renderer
```

如果继续在旧模块里修：

1. 正文、工具、控制、任务阶段还会互相覆盖。
2. 清洗失败后仍可能回退 raw string。
3. 工具 observation 仍可能被误标为模型正文。
4. `task_control` 仍可能参与 assistant turn 收口。
5. 前端 fallback 仍可能把旧状态挂到下一轮。

所以要把旧实现从运行路径里拿开，而不是继续补。

`session_task_projection.py` 和 `session_timeline.py` 暂不直接移动，因为它们是外部调用入口；先改为调用新 `projection/`，确认无旧 import 后再拆分或删除旧逻辑。

### 2. 前端旧文件

第一步把旧投影状态与渲染逻辑隔离：

```text
frontend/src/lib/store/publicProjectionReducer.ts
-> frontend/src/lib/projection_legacy/publicProjectionReducer.ts

frontend/src/lib/store/publicTimeline.ts
-> frontend/src/lib/projection_legacy/publicTimeline.ts

frontend/src/lib/runtimeVisibilityProjection.ts
-> frontend/src/lib/projection_legacy/runtimeVisibilityProjection.ts
```

组件先保留原文件名作为外部入口：

```text

```

但内部接线改为：

```text
ChatMessage.tsx -> components/chat/projection/ProjectionTurnView.tsx
PublicTimelineActivity.tsx -> components/chat/projection/ProjectionActivity.tsx
```

旧组件逻辑迁入：

```text

```

确认测试通过后删除 legacy 组件。

## 五、新正式投影链路

新链路只有一个公开投影权威：

```text
Raw Runtime Event
-> Projection Fact
-> Authority Check
-> Public Projection Frame
-> Session Timeline
-> Frontend Projection Store
-> Projection Renderer
```

固定时序：

```text
UserMessage
-> ActiveTurnOpened
-> PromptPacketCreated
-> ModelOutputReceived
-> ModelBodyProjected or ModelActionProjected
-> Tool/TaskControlProjected
-> ToolObservationRecorded
-> ObservationPromptPacketCreated
-> ModelStageFeedbackProjected
-> TurnClosed
-> SessionTimelineRebuilt
```

页面顺序：

```text
用户输入
-> 正在思考 UI chrome
-> 模型正文 / 开局反馈
-> 工具窗口
-> 工具观察
-> 阶段总结
-> 下一阶段动作或最终回答
-> 控制等待/错误/恢复
```

## 六、Prompt 输入链

每次模型调用前必须创建 `PromptPacket`。

```json
{
  "authority": "harness.prompt_packet",
  "prompt_packet_ref": "promptpkt:...",
  "session_id": "...",
  "turn_id": "...",
  "turn_run_id": "...",
  "task_run_id": "...",
  "input_blocks": [
    {"kind": "agent_role_prompt"},
    {"kind": "user_request"},
    {"kind": "conversation_context"},
    {"kind": "active_turn_state"},
    {"kind": "tool_catalog"},
    {"kind": "permission_contract"},
    {"kind": "task_contract"},
    {"kind": "observation_context"},
    {"kind": "stage_boundary_context"},
    {"kind": "action_output_contract"}
  ]
}
```

PromptPacket 不公开投影。它只给模型和 trace 使用。

禁止：

1. 把开发节点说明当 prompt。
2. 把“开始处理/处理完成/任务执行器接管”当 prompt 模板。
3. 要求模型输出控制命令。
4. 把旧投影文案当作无来源事实喂回模型。

## 七、公开投影契约

正式 frame：

```json
{
  "authority": "harness.public_projection",
  "contract_revision": "20260610-replacement",
  "frame_id": "pubframe:...",
  "sequence": 18,
  "event_offset": 104,
  "anchor": {
    "session_id": "...",
    "turn_id": "...",
    "turn_run_id": "...",
    "task_run_id": "...",
    "message_id": "...",
    "prompt_packet_ref": "promptpkt:..."
  },
  "source": "model|tool|runtime|system|user",
  "surface": "assistant_body|tool_window|control|timeline|diagnostics",
  "items": []
}
```

硬规则：

1. `assistant_body` 只接受 `source=model`。
2. `tool_observation` 不能进入 `assistant_body`。
3. `task_control` 只能进入 `control`。
4. `done/terminal_reason` 只关闭状态，不能生成正文。
5. 缺 anchor 的 frame 不挂到任何 assistant message。
6. 清洗失败不能回退 raw string。

## 八、任务开启时序

### 1. 直接回答

```text
user_message_persisted
-> active_turn_opened
-> prompt_packet_created
-> model_call_started
-> model_body_delta
-> model_body_final
-> assistant_message_persisted
-> turn_closed control frame
```

### 2. 首轮工具调用

```text
user_message_persisted
-> active_turn_opened
-> prompt_packet_created
-> model_action_received(tool_call)
-> opening_judgment frame if model provided
-> tool_call_started frame
-> tool_observation frame
-> observation_prompt_packet_created
-> model_stage_feedback frame or next tool_call frame
```

### 3. `request_task_run`

```text
user_message_persisted
-> active_turn_opened
-> prompt_packet_created
-> model_action_received(request_task_run)
-> opening_judgment frame
-> task_run_contract_created
-> task_projection frame
-> executor_scheduled control frame
-> executor tool frames
-> executor observation frames
-> observation_prompt_packet_created
-> model_stage_feedback/final_answer frame
```

`executor_scheduled` 不能成为 assistant message、assistant body 或 final answer。

## 九、实施阶段

### 阶段 0：改名隔离

动作：

1. 新建 `backend/harness/runtime/projection_legacy/`。
2. 移动旧后端投影文件。
3. 新建 `backend/harness/runtime/projection/`。
4. 新建 `frontend/src/lib/projection_legacy/`。
5. 移动旧前端投影 reducer/timeline/fallback。
6. 新建 `frontend/src/lib/projection/` 和 `frontend/src/components/chat/projection/`。

完成标准：

```text
旧投影代码已经从原路径移走
新 projection 路径为空壳但可导入
```

### 阶段 1：新契约与测试

动作：

1. 写 `projection/types.py` 和 `projection/authority.py`。
2. 写 `frontend/src/lib/projection/types.ts`。
3. 先补 frame authority、anchor、surface 测试。

完成标准：

```text
assistant_body 只能 source=model
tool_observation 不能进入 body
control 不进入正文
缺 anchor 不挂载
```

### 阶段 2：PromptPacket 与模型调用

动作：

1. 新增 `projection/prompt_packet.py`。
2. 在首轮模型调用和工具观察后模型调用前记录 `prompt_packet_created`。
3. 工具结果先生成 observation，再进入下一次 prompt。

完成标准：

```text
prompt_packet_created -> model_action_received
tool_observation -> observation_prompt_packet_created -> model_feedback
```

### 阶段 3：后端投影接线

动作：

1. `task_lifecycle.py` 不再发送空 `task_control final_answer`。
2. `single_agent_turn.py` 把模型正文和动作交给新 projector。
3. `task_executor.py` 把工具开始、等待、观察、失败交给新 projector。
4. `session_timeline.py` 改为调用新 timeline builder。
5. `session_task_projection.py` 只保留任务结构态，或迁移到新 `projection/task_projection.py`。

完成标准：

```text
正文、工具、控制分别进入对应 surface
task_projection 不再压掉正文 timeline
```

### 阶段 4：前端投影接线

动作：

1. `events.ts` 和 `runtime.ts` 接入新 projection reducer。
2. `ChatMessage.tsx` 接入 `ProjectionTurnView`。
3. `PublicTimelineActivity.tsx` 接入 `ProjectionActivity` 或删除旧入口。
4. 旧 fallback 不再产出可见活动。

完成标准：

```text
正文、工具窗口、控制面分层显示
工具不泄露到下一轮
刷新水合顺序不变
```

### 阶段 5：删除 legacy

动作：

1. 全仓搜索旧 import。
2. 删除 `projection_legacy` 运行依赖。
3. 删除保护旧 internal shape 的测试。
4. 删除或归档 legacy 文件夹。

完成标准：

```text
运行路径只剩 projection
projection_legacy 不再被 import
```

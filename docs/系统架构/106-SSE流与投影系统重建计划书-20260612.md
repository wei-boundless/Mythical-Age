# SSE流与投影系统重建计划书

日期：2026-06-12

状态：待用户审阅确认后实施

适用范围：后端 SSE 输出、运行日志流、Runtime Monitor、公开投影、前端流式消费与投影 reducer。

## 1. 目的

本计划用于重建当前项目的 SSE 流和投影系统，目标不是给现有卡顿和串线问题打补丁，而是重新划清三类通道的权责：

- 用户可见的聊天输出流：负责 assistant 正文、工具项、turn 完成等公开事件。
- 运行日志流：负责 task run / turn run 的诊断事件、trace、调试窗口。
- 运行监控流：负责 Runtime Monitor 的状态摘要、管理信号、心跳和刷新通知。

正确完成后，Runtime Monitor 不再承担日志总线和公开投影总线职责，后端日志读取不再拖慢监控，前端主会话不会再被其他 run 的 monitor event 旁路修改。

## 2. 当前破坏点

### 2.1 直接症状

- 后端日志读取卡顿。
- Runtime Monitor 和日志混用后容易串线。
- 前端主会话 runtime attachment 会被 monitor stream 中的 runtime event 修改。
- 公开投影有多条入口，后续维护者难以判断哪个入口才是权威链路。

### 2.2 深层结构原因

当前 `backend/api/runtime_monitor.py` 的 SSE 端点在 `/orchestration/runtime-monitor/events` 中全局订阅 `runtime_host.event_log.subscribe()`，没有按 task run、turn run 或 stream run 隔离。

该端点收到任意 runtime event 后会：

1. 重新收集全局 monitor snapshot。
2. 调用 `project_runtime_monitor_event_public_delta(...)` 构造公开投影。
3. 把 raw `runtime_event`、`public_projection_envelope` 和完整 `monitor` 一起塞进 `runtime_monitor_event`。

前端 `frontend/src/lib/run-monitor/controller.ts` 收到该 payload 后，又调用 `patchRuntimeAttachmentFromRuntimeEvent(...)` 修改主会话消息附件。这样 monitor 同时拥有 observe、record、project、present、mutate session 五类权力，造成跨 run 串线和重复投影。

### 2.3 需要保留的成熟基础

当前项目并非整条流式链路都坏了。以下能力应保留并收紧：

- `backend/api/chat.py` 的 `/chat/runs/{stream_run_id}/events` 已经是 run-scoped SSE。
- `backend/runtime/shared/stream_replay.py` 已经支持 public event append、offset replay、`Last-Event-ID`、terminal event。
- `backend/runtime/model_gateway/assistant_stream_frame.py` 和 `assistant_stream_normalizer.py` 已经有 typed assistant delta/final/repair、sequence、UTF-8 offset、hash 和 repair。
- `frontend/src/lib/api.ts` 的 `consumeChatRunStream()` 已经有 cursor persistence、fetch stream parser、reconnect、terminal handling。

重建方案应保留这些成熟能力，删除错误的 monitor 旁路，而不是把现有 public stream 全部推倒重写。

## 3. 依据来源

### 3.1 本项目约束

来自项目规则和既有架构文档的约束如下：

- 固定本地节点：前端 `http://127.0.0.1:3000`，后端 `http://127.0.0.1:8003`，API Base `http://127.0.0.1:8003/api`。
- 涉及启动、SSE、监控、Electron 或页面可用性的修改，必须真实启动前后端实测。
- 大规模 runtime / API / frontend protocol 变更必须先写计划并等待确认。
- 无用旧代码、废弃分支、重复链路、调试残留和过时测试默认清理，不以兼容为理由保留旧链路。
- `docs/系统架构/103-公开投影系统文件夹改名与替换方案-20260610.md` 已明确 `projection_legacy` 是只读迁移区，不参与运行。

### 3.2 本项目代码依据

- `backend/api/runtime_monitor.py`：当前 monitor SSE 将 raw runtime event、monitor snapshot 和公开投影混在同一 payload。
- `frontend/src/lib/run-monitor/controller.ts`：当前 monitor controller 接收 `runtime_event` 后触发 session attachment patch。
- `frontend/src/lib/store/runtime.ts`：`patchRuntimeAttachmentFromRuntimeEvent(...)` 是 monitor 旁路修改主会话的主要入口。
- `backend/runtime/shared/event_log.py`：`RuntimeEventLog.subscribe(run_id=...)` 已支持 scoped subscription，说明日志流不需要走 global monitor subscription。
- `backend/api/orchestration_harness.py` 和 `backend/harness/runtime/single_agent_host.py`：已有 task run / turn run trace 查询能力，可作为日志流 replay 和诊断读取基础。
- `backend/api/chat.py` 与 `backend/runtime/shared/stream_replay.py`：已有 public chat SSE 的 replay/cursor 基础。
- `frontend/src/lib/store/events.ts`：已有 public stream reducer，适合继续作为主会话投影入口。

### 3.3 外部参考

#### Codex

Codex 的相关机制更适合作为本项目重建标准：

- `codex-rs/protocol/src/protocol.rs`：核心事件是 typed `Event { id, msg }`。
- `codex-rs/app-server-protocol/src/protocol/common.rs`：app server 使用 typed `ServerNotification`，例如 `turn/completed`、`item/started`、`item/completed`、`item/agentMessage/delta`、`item/plan/delta`、`item/commandExecution/outputDelta`。
- `codex-rs/app-server-protocol/src/protocol/event_mapping.rs`：核心 runtime event 映射为前端协议 notification，映射层清晰。
- `sdk/python/src/openai_codex/_message_router.py`：单 reader 把 notification 路由到 per-request、per-turn queue，`register_turn(turn_id)` 和 pending queue 是防串线关键。
- `codex-rs/app-server-client/src/lib.rs`：明确区分 lossless event 和 best-effort event，assistant output、completion 等必须保留，命令输出等可在拥塞时降级。
- `codex-rs/tui/src/chatwidget/protocol.rs` 与 `streaming/controller.rs`：UI 接 typed notification，再投影到本地状态；事实流和 UI 投影不是同一个层。

应借鉴的核心不变量：

- 事实事件是协议层，UI 投影是派生层。
- run / turn routing 必须显式，不靠 monitor snapshot 推断。
- 用户可见正文、完成事件、关键 item 生命周期是 lossless。
- 诊断日志、命令输出、进度噪声可以 best-effort，但必须可追踪 gap。

#### Hermes

Hermes 更适合作为传输细节参考：

- `run_agent.py::_fire_stream_delta()`：模型流式 delta 经同步 callback bus 输出。
- `gateway/stream_consumer.py`：callback 进入 queue，再由 async consumer 做缓冲、分段和平台编辑。
- `agent/chat_completion_helpers.py`：使用 `last_chunk_time`、heartbeat、stale stream detector 处理上游模型流停滞。
- `gateway/platforms/api_server.py`：OpenAI-compatible SSE 写入 final chunk、`[DONE]`，client disconnect 时中断 agent。

应借鉴的机制：

- provider delta 与 SSE writer 之间使用 queue adapter 隔离。
- heartbeat 和 stale detector 属于传输健康，不属于 monitor 投影。
- client disconnect 应中断对应 run，不影响其他 run。

不应照搬的机制：

- Hermes 的 `message.delta/message.complete` 对本项目过于简单，不如现有 typed assistant delta/final/repair 可 replay。
- Hermes 的平台编辑 consumer 不应替代本项目的 public event log。

## 4. 权责审计

### 4.1 当前流

```text
RuntimeEventLog
-> runtime_monitor.py global subscribe
-> collect_global_runtime_monitor()
-> project_runtime_monitor_event_public_delta()
-> runtime_monitor_event { runtime_event, monitor }
-> frontend RunMonitorController
-> patchRuntimeAttachmentFromRuntimeEvent()
-> 主会话 messages/runtimeAttachments
```

该流的问题是：monitor 读的是全局 event log，却能写主会话投影。它既不是 run-scoped log reader，也不是 public chat stream，却拥有二者的结果写入权。

### 4.2 目标权责链

```text
Runtime execution
-> RuntimeEventLog(record, diagnostic truth)
-> scoped runtime log SSE(retrieve/present diagnostic events)

Runtime execution
-> RuntimeStreamReplayService(record public stream truth)
-> chat public SSE(lossless user-visible stream)
-> frontend stream reducer(project main session)

Runtime state index / monitor service
-> Runtime Monitor SSE(present monitor snapshots only)
-> frontend run monitor state
```

### 4.3 权责表

| 模块 | 当前职责 | 隐性越权 | 目标职责 | 动作 |
| --- | --- | --- | --- | --- |
| `backend/api/runtime_monitor.py` | monitor 查询、monitor SSE、runtime event 转发、公开投影 | 全局日志订阅后修改公开投影 | 只输出 monitor snapshot、heartbeat、轻量 invalidation | 重写 SSE payload，删除 raw runtime event 和 projection 调用 |
| `backend/runtime/shared/event_log.py` | 记录 runtime diagnostic event，支持 scoped subscribe | queue 满时统一 drop oldest，未区分 lossless/best-effort | runtime 诊断事实层，服务 scoped log stream | 保持记录权，补充 log stream gap 处理 |
| `backend/runtime/shared/stream_replay.py` | public chat event append/replay/SSE format | 可复用能力只服务 chat stream，尚未抽象日志 cursor | public stream 事实层 | 保留并必要时抽出通用 SSE cursor/format helper |
| `backend/api/chat.py` | 创建 chat run、投影 public event、run-scoped SSE | 部分旧事件仍通过 public projection envelope 混入 | 用户可见 lossless stream | 保留主链，收紧事件 allowlist 和 envelope 入口 |
| `backend/harness/runtime/projection/timeline_builder.py` | 从 runtime event 构造 public timeline/projection | 函数名和 authority 仍绑定 runtime monitor | 离线/回放 projection builder，不作为 monitor SSE 入口 | 从 monitor SSE 断开；如仍需要，改名和改 authority |
| `frontend/src/lib/run-monitor/controller.ts` | 管理 monitor state，也 patch 主会话 | monitor payload 能改 session messages | 只管理 monitor UI state | 删除 `patchRuntimeAttachmentFromRuntimeEvent` 调用 |
| `frontend/src/lib/store/runtime.ts` | 主 store、chat stream、runtime attachment patch | 接收 monitor event 后旁路写主会话 | 主会话只由 public stream reducer 写入 | 删除 monitor-event patch path |
| `frontend/src/lib/store/events.ts` | public stream reducer | 仍支持部分旧 event | public stream 唯一主会话投影入口 | 保留 typed assistant/tool/turn，清理 obsolete event |
| `projection_legacy/` | 旧投影归档 | 目录存在会误导维护者 | 无运行职责 | 确认零 import 后删除 |

## 5. 设计取舍

### 5.1 方案 A：继续让 monitor 承担 event bus

结论：拒绝。

原因：

- global subscribe 天然会跨 run。
- monitor snapshot 计算重，不能成为高频日志传输通道。
- monitor payload 携带 raw event，会把调试信息带入主会话投影面。
- 继续保留该路径会让公开投影有两个权威入口。

### 5.2 方案 B：完全照搬 Hermes delta/complete

结论：拒绝。

原因：

- 本项目已有 typed assistant frame、repair 和 replay cursor，简单 delta/complete 会退化。
- Hermes 更像平台消息编辑流，本项目需要可回放的事实事件流。
- 只借鉴 callback/queue、heartbeat、stale detector、disconnect interrupt。

### 5.3 方案 C：采用 Codex typed event + scoped router + UI reducer

结论：采用。

原因：

- Codex 的 per-turn/per-request queue 直接对应本项目的串线问题。
- lossless/best-effort 分层适合区分 assistant output 与 diagnostic logs。
- UI reducer 只消费 typed notification，符合本项目已有 `reduceStreamEvent()` 基础。
- 可以在保留现有 chat stream 优势的同时拆掉 monitor 旁路。

## 6. 推荐设计

### 6.1 三条通道

#### 通道一：Chat Public SSE

端点保持：

```text
GET /api/chat/runs/{stream_run_id}/events
```

职责：

- 输出用户可见、可回放、lossless 的 public event。
- 支持 `after_offset` 和 `Last-Event-ID`。
- 输出 `assistant_text_delta`、`assistant_text_final`、`assistant_stream_repair`、`tool_item_started`、`tool_item_completed`、`turn_completed` 等 typed events。
- 由 `RuntimeStreamReplayService` 和 `StreamEvent` cursor 维护事实顺序。

禁止：

- 不读取 runtime monitor stream。
- 不从 runtime monitor event 补投影。
- 不输出 raw model messages、prompt manifest、runtime assembly 等内部字段。

#### 通道二：Runtime Log SSE

新增端点：

```text
GET /api/runtime/logs/task-runs/{task_run_id}/events
GET /api/runtime/logs/turn-runs/{turn_run_id}/events
```

职责：

- 只服务诊断、开发者日志、trace viewer。
- 使用 `RuntimeEventLog.subscribe(run_id=...)` 做 scoped subscription。
- 使用 `list_event_window(run_id, limit=..., include_payloads=...)` 做初始 replay。
- 默认 redacted payload，只有显式 `include_payloads=true` 才返回完整 payload；`include_model_messages` 默认 false。

建议事件类型：

```text
runtime_log_snapshot
runtime_log_event
runtime_log_gap
runtime_log_heartbeat
runtime_log_closed
```

建议 cursor：

```text
{scope}:{run_id}:{offset}
```

其中 `scope` 为 `task` 或 `turn`，避免 task 和 turn 同 id 格式混淆。

禁止：

- 不写主会话 messages。
- 不构造 `public_projection_envelope`。
- 不触发 monitor snapshot 全量重算。

#### 通道三：Runtime Monitor SSE

端点保留：

```text
GET /api/orchestration/runtime-monitor/events
```

职责改为：

- 输出 `runtime_monitor_snapshot`。
- 输出 `runtime_monitor_heartbeat`。
- 必要时输出 `runtime_monitor_invalidated`，提示前端拉取最新 monitor。

payload 只允许：

```json
{
  "source": "initial | heartbeat | invalidated",
  "monitor": {},
  "updated_at": 0
}
```

禁止字段：

```text
runtime_event
public_projection_envelope
public_event_type
public_timeline
task_projection_delta
debug_trace_ref
```

禁止：

- 不订阅全局 runtime event 后逐条发送 raw event。
- 不从 runtime event 构造公开投影。
- 不修改主会话 session message。

### 6.2 投影入口

主会话投影只允许从 Chat Public SSE 进入：

```text
chat public event
-> frontend reduceStreamEvent()
-> applyPublicProjectionEnvelope()
-> messages/runtimeAttachments
```

Runtime Log SSE 和 Runtime Monitor SSE 都不得调用主会话投影 reducer。

### 6.3 Backpressure 分层

lossless：

- assistant text delta/final/repair
- tool item started/completed
- turn completed
- chat run terminal/error/stopped

best-effort：

- runtime log raw output
- command output delta
- repeated progress details
- monitor invalidation

规则：

- lossless 事件必须可 replay。
- best-effort 事件可以在 queue 满时丢弃，但必须输出 `runtime_log_gap` 或 monitor revision 变化，不能静默丢。
- monitor 的 snapshot 可以覆盖旧 snapshot，但不能覆盖 public stream 的事实。

## 7. 固定执行流

### 7.1 用户发起 chat run

输入：

- user message
- session id
- runtime request

输出：

- `RuntimeRun`
- public stream URL

拥有者：

- `backend/api/chat.py`
- runtime host

禁止：

- chat run 创建时不得读取 monitor snapshot 决定投影。

### 7.2 模型/工具执行

输入：

- runtime start packet
- provider stream delta
- tool events

输出：

- runtime diagnostic event
- public stream event

拥有者：

- execution loop
- model gateway normalizer
- stream replay service

禁止：

- execution loop 不得因为 monitor 状态重写用户意图。
- provider stream 停滞只能变成 transport health 或 explicit failure，不能由 monitor 猜测完成。

### 7.3 公开输出投影

输入：

- public stream event

输出：

- frontend session message state

拥有者：

- `frontend/src/lib/store/events.ts`
- `frontend/src/lib/projection/reducer.ts`

禁止：

- runtime log event 不得直接进入 `reduceStreamEvent()`。
- runtime monitor event 不得 patch session runtime attachment。

### 7.4 日志查看

输入：

- task_run_id 或 turn_run_id
- cursor

输出：

- redacted runtime log events

拥有者：

- 新 runtime logs API
- frontend runtime log client/controller

禁止：

- 日志查看不触发 monitor 全局重算。
- 日志查看不更新主会话 messages。

### 7.5 监控刷新

输入：

- monitor service snapshot
- heartbeat timer
- optional invalidation signal

输出：

- monitor UI state

拥有者：

- runtime monitor service
- run monitor controller

禁止：

- monitor 不输出 raw event。
- monitor 不创建 public projection。

## 8. 数据合同变更

### 8.1 新增 runtime log stream contract

新增 TypeScript 类型建议：

```ts
export type RuntimeLogStreamPayload = {
  source?: "snapshot" | "event" | "gap" | "heartbeat" | "closed";
  scope: "task_run" | "turn_run";
  run_id: string;
  event?: HarnessTraceEvent;
  events?: HarnessTraceEvent[];
  event_offset?: number;
  last_event_id?: string;
  gap?: {
    expected_after_offset: number;
    observed_offset: number;
    recovered: boolean;
  };
  updated_at?: number;
};
```

后端可对应创建轻量 helper，不必须引入复杂 model，但输出 shape 必须稳定。

### 8.2 收紧 RunMonitorEventPayload

删除：

```ts
runtime_event
runtime_event.public_projection_envelope
runtime_event.task_projection_delta
runtime_event.public_timeline
```

保留：

```ts
source
monitor
updated_at
```

如前端仍需显示“最近变化”，应从 monitor envelope 内部已有 signal/revision 表达，而不是携带 raw runtime event。

### 8.3 投影 authority 命名

当前 `PUBLIC_PROJECTION_AUTHORITY = "runtime_monitor.public_event_projection"` 与目标架构冲突。实施时需要改为不绑定 monitor 的名称，例如：

```text
public_stream.public_projection.v1
```

若 `project_runtime_monitor_event_public_delta(...)` 仍用于离线 timeline 构建，应改名为不含 `runtime_monitor` 的 builder，例如：

```text
project_public_event_delta_from_runtime_event(...)
```

改名时必须同步测试，不保留旧函数别名作为兼容入口，除非有明确外部 API 契约。当前未发现必须保留该 Python 函数名的外部契约。

## 9. 模块计划

### 9.1 后端

- `backend/api/chat.py`：保留 `/chat/runs/{stream_run_id}/events`，只做必要收紧，不让 monitor 参与 public event 投影。

- 新增 runtime log controller/hook：用于日志 UI 读取 scoped log stream，不接入主会话 reducer。

## 10. 阶段计划

### 阶段一：后端通道拆分

目标：

- 建立 runtime log stream。
- 收紧 runtime monitor stream。
- 保留 chat public stream 主链。

涉及文件：

- `backend/api/runtime_monitor.py`
- 新增 `backend/api/runtime_logs.py`
- 路由注册文件，按现有 API 注册方式接入
- `backend/runtime/shared/event_log.py`
- `backend/runtime/shared/stream_replay.py`

完成标准：

- task run A 的 log stream 不会收到 task run B 的 event。
- monitor SSE payload 不包含 raw runtime event。
- chat stream 原有 replay 测试仍通过。

禁止：

- 不新增“临时兼容 monitor_event_with_runtime_event”。
- 不让 log stream 调用 public projection builder。

### 阶段二：前端状态拆线

目标：

- Monitor 只更新 monitor state。
- Runtime log 有独立 client/controller。
- 主会话只从 public stream reducer 更新。

涉及文件：

- `frontend/src/lib/api.ts`
- `frontend/src/lib/run-monitor/controller.ts`
- `frontend/src/lib/run-monitor/types.ts`
- `frontend/src/lib/run-monitor/reducer.ts`

## 13. 迁移和切换规则

### 13.1 不做双轨兼容

本次重建不保留旧 monitor event projection 双轨。原因：

- 旧路径本身就是串线来源。
- 若保留兼容字段，前端或测试很容易继续依赖旧字段。
- 项目规则要求无用旧链路默认清理。

### 13.2 切换方式

- 后端和前端在同一实施批次内切换协议。
- 旧 `runtime_monitor_event.runtime_event` 删除。
- 新 runtime log stream 同批次提供。
- 前端同批次删除 monitor-to-session mutation。

### 13.3 回滚方式

因为不做运行时双轨，回滚只能通过 git 回滚整个实施批次，不允许在代码内保留旧路径开关。

### 13.4 最终清理条件

## 15. 自审：遗漏和冲突检查

### 15.1 真相和所有权

| 检查项 | 结论 | 处理 |
| --- | --- | --- |
| 是否存在单一事实源 | 是，chat public truth 归 `RuntimeStreamReplayService`，diagnostic truth 归 `RuntimeEventLog` | 文档已锁定双事实层，分别服务不同通道 |
| monitor 是否仍持有投影权 | 否，目标中 monitor 只 present snapshot | 阶段一删除 projection 调用 |
| 主会话是否可能被日志覆盖 | 否，log stream 不进入 `reduceStreamEvent()` | 阶段二新增测试 |

### 15.2 边界分离

| 检查项 | 结论 | 处理 |
| --- | --- | --- |
| 输出组装是否和 raw log 分离 | 是 | chat public stream 和 runtime log stream 分离 |
| task-local truth 是否会泄漏进 main session | 目标中不会 | 删除 monitor-to-session patch |
| monitor 是否能作为恢复/裁决依据 | 不能 | monitor 只显示，不决定 |

### 15.3 路径一致性

| 检查项 | 结论 | 处理 |
| --- | --- | --- |
| chat stream 和 log stream 是否共享同一 cursor | 不共享 | chat 用 stream run cursor，log 用 scope/run/offset cursor，避免混淆 |
| task run 和 turn run log 是否可能串线 | 目标中不会 | endpoint 和 cursor scope 明确区分 |
| direct/compound run 是否共用同一 public stream 合同 | 是 | public stream event 类型统一进入 reducer |

### 15.4 状态卫生

| 检查项 | 结论 | 处理 |
| --- | --- | --- |
| monitor snapshot 能否覆盖当前 turn 输出 | 目标中不能 | 删除 monitor event patch |
| 失败状态是否可能留作 active state | 需要测试覆盖 | runtime log 和 turn completed terminal 测试必须覆盖 failed/stopped |
| reconnect 是否可能重复输出 | chat 已有 offset replay，log stream 需新增 cursor 测试 | 阶段一测试 |

### 15.5 输出卫生

| 检查项 | 结论 | 处理 |
| --- | --- | --- |
| raw model messages 是否会进用户可见流 | 默认不允许 | log stream 默认 redacted，chat redaction 保留 |
| stale monitor state 是否会影响 final answer | 目标中不会 | monitor 不参与 final answer 投影 |
| public projection 是否仍可追踪来源 | 是 | authority 改为 public stream，不绑定 monitor |

### 15.6 迁移和清理

| 检查项 | 结论 | 处理 |
| --- | --- | --- |
| 是否存在兼容字段 | 目标中不保留 | 同批次删除前后端旧字段 |
| `projection_legacy` 删除是否冲突 | 不冲突，既有文档已说明只读迁移不参与运行 | 确认零 import 后删除 |
| 旧测试是否会保护旧结构 | 可能 | 阶段三要求重写测试为行为断言 |

### 15.7 验证缺口

发现的补充点：

1. 原计划需要明确 runtime log stream 的默认 redaction，否则 raw payload 可能重新泄漏到前端。本计划已补充 `include_payloads=false` 和 `include_model_messages=false` 默认策略。
2. 原计划需要明确 log stream gap 表示，否则 best-effort drop 会变成静默丢日志。本计划已补充 `runtime_log_gap`。
3. 原计划需要处理 `runtime_monitor.public_event_projection` authority 命名冲突。本计划已补充改名/删除规则。
4. 原计划需要明确 task run 和 turn run cursor 不混用。本计划已补充 `{scope}:{run_id}:{offset}`。
5. 原计划需要明确不做双轨兼容的回滚方式。本计划已补充只能整批 git 回滚，不在代码内保留旧开关。

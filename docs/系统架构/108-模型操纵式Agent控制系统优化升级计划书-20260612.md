# 模型操纵式 Agent 控制系统优化升级计划书

日期：2026-06-12
状态：计划书
适用范围：通用任务环境、主会话单轮决策、持续任务交接、工具/技能供给、运行反馈、公开进度投影
明确不包含：权限策略重设、权限模式默认值讨论、用户个人权限配置调整

---

## 1. 结论

本项目的目标控制模型应明确为：

```text
模型是操纵者。
系统是服务提供者、执行承接者、状态记录者、边界控制者和反馈投影者。
Prompt 指导模型如何操作系统。
契约是模型把意图交给系统承接的结构化接口。
工具、技能、上下文、运行状态和 observation 都是系统向模型提供的服务面。
```

因此，本次优化不是把系统升级成更强的任务语义裁决者，也不是让系统在任务启动前替模型推断完整工具链。优化目标是让模型拥有更清晰、更低摩擦、更可追踪的操作接口，并让系统按模型意图提供能力、执行动作、返回 observation。

成熟 agent 的精确操作模式应是：

```text
1. 系统向模型展示角色规则、上下文、能力目录、可见工具/技能和运行边界。
2. 模型做本轮语义裁决：respond / ask_user / tool_call / request_task_run / active_work_control / block。
3. 如果模型请求工具，系统执行结构化 tool call，并返回 tool observation。
4. 如果模型请求持续任务，系统创建 task run，记录任务目标、范围、验收、能力意图、技能选择和反馈策略。
5. 执行循环中，模型根据最新 observation 逐轮决定下一步动作。
6. 系统只做服务装配、执行、记录、边界控制和反馈投影，不替模型改写任务目标。
```

本计划把通用任务环境从“宽能力默认暴露”升级为“模型按需操纵服务面”的结构。

## 2. 成熟方案依据

本计划只采用公开成熟 agent 方案能确认的机制，不假设 Codex 或 Claude Code 的私有内部实现。

1. OpenAI Function Calling / Responses 工具调用模式：模型根据上下文决定是否发出工具调用，应用执行工具并把结果回传给模型，模型继续生成下一步或最终结果。
   来源：https://developers.openai.com/api/docs/guides/function-calling

2. OpenAI Agents SDK Runner loop：运行循环调用模型；模型可能输出最终回复、handoff 或 tool calls；如果有 tool calls，runner 执行工具并把结果加入上下文后继续循环。
   来源：https://openai.github.io/openai-agents-python/running_agents/

3. OpenAI Agents SDK Tools：工具可以按需提供，支持 hosted tools、function tools、agent-as-tool、tool search、approval gates 和 tracing。关键启发是“大能力面不必一次性全部塞进模型上下文”。
   来源：https://openai.github.io/openai-agents-python/tools/

4. Anthropic Claude Tool Use：Claude 依据工具描述和当前请求决定是否使用工具，返回结构化 tool_use；客户端执行工具后返回 tool_result。
   来源：https://platform.claude.com/docs/en/agents-and-tools/tool-use/overview

这些方案共同确认：

- 模型是动作选择者。
- 系统提供工具、上下文、执行和反馈。
- 工具调用逐轮发生，不要求任务启动前声明完整工具序列。
- 大工具面应通过目录、命名空间、搜索或延迟加载管理。
- 执行结果必须回到模型可消费的 observation。

## 3. 当前系统状态

### 3.1 已有良好基础

当前系统已经有一套可继续演进的结构：

- `ModelActionRequest` 已经把模型动作收敛为结构化 action，包含 `respond / ask_user / tool_call / request_task_run / active_work_control / block`。
  相关文件：`backend/harness/loop/model_action_protocol.py`

- `model_action_request_schema` 已经把 action 选择规则、公开反馈、任务交接 seed、完成标准、产物和验证要求暴露给模型。
  相关文件：`backend/harness/runtime/compiler.py`

- `TaskRunContract` 已经承载持续任务目标、完成标准、资源要求、策略、运行 profile 和 prompt contract。
  相关文件：`backend/harness/loop/task_lifecycle.py`

- `RuntimeAssembly` 已经集中装配环境、工具、技能候选、prompt mount plan、operation authorization 和 runtime projection。
  相关文件：`backend/harness/runtime/assembly.py`

- `task_executor` 已经有 observation、重复工具调用 guard、step summary 和 public action state 记录。
  相关文件：`backend/harness/loop/task_executor.py`

这些基础说明系统不是要推倒重做，而是要把模型操纵接口、能力供给和反馈闭环收束成更清晰的协议族。

### 3.2 当前不流畅点

#### 3.2.1 模型操纵接口不够完整

`request_task_run` 当前能稳定承接目标、产物、验证和完成标准，但对“模型想调用哪些类型的服务”表达不足。

目前 schema 中虽然存在 `resource_requirements`、`permission_requirements`、`acceptance_policy`、`recovery_policy` 等字段，但这些字段是泛型对象，缺少模型可直接理解的能力意图结构。

结果是：模型知道自己要执行任务，却没有一个清晰槽位表达“我需要文件工作、网络研究、浏览器、命令执行、产物生成、子 agent 或某个 skill”。

#### 3.2.2 原生工具路径会压扁任务交接信息

`single_agent_turn` 中对原生 `request_task_run` 的解析只稳定提取：

- `user_visible_goal`
- `task_run_goal`
- `required_artifacts`
- `required_verifications`
- `completion_criteria`
- `active_work_relationship`

这会导致模型即使在 schema 层能表达更多任务交接信息，也可能在 native tool-call 路径中丢失。

#### 3.2.3 通用环境偏向宽候选池，不利于模型按需操纵

通用环境现在更像一个宽泛默认能力入口。对模型来说，它容易看到“很多东西都可用”，但不一定知道本任务该先加载哪个能力组、哪个 skill、哪个工具命名空间。

这不是权限配置问题，而是能力供给方式问题。成熟 agent 更倾向：

```text
先暴露能力目录和候选卡
再由模型按任务目标选择能力组、技能或工具命名空间
再进入具体工具调用循环
```

#### 3.2.4 Skill 是候选，但没有形成模型选择闭环

当前 skill runtime views 主要按运行时可用能力过滤后展示候选。模型可以填写 `selected_skill_ids`，但任务契约、skill 展开、执行循环之间还没有形成强闭环。

成熟形态应是：

```text
系统展示 skill 候选卡
模型选择 skill 或能力标签
系统展开被选 skill 的完整说明
模型依据 skill 操作工具/上下文
skill 产出的证据进入 observation
```

#### 3.2.5 Feedback 还没有成为单一 observation 权威

当前系统里公开进度和执行反馈散落在：

- `public_progress_note`
- `public_action_state`
- step summary
- task diagnostics latest fields
- session timeline
- frontend runtime store fallback

这会导致用户侧看到的“正在做什么”和模型侧用于下一步决策的 observation 不是同一个权威对象。

成熟 agent 需要把反馈统一成：

```text
tool_result / runtime_guard / artifact_ref / verification_result / progress_event
-> canonical observation envelope
-> model followup packet
-> public progress projection
```

## 4. 设计目标

### 4.1 总目标

把系统升级为“模型操纵式控制系统”：

```text
模型负责判断和发起动作。
系统负责提供能力目录、承接结构化动作、执行工具、返回 observation、记录状态、投影进度。
```

### 4.2 非目标

本计划不处理以下事项：

- 不调整用户的权限模式。
- 不讨论默认权限配置。
- 不以“收紧权限”为主要优化手段。
- 不要求模型在任务启动前列出完整 operation 清单。
- 不让系统根据关键词替模型推断任务工具链。
- 不用 prompt 补丁替代协议和运行时结构修复。

### 4.3 成功状态

升级完成后，系统应满足：

1. 模型能通过清晰 action schema 表达本轮动作和持续任务交接意图。
2. `request_task_run` 能承接目标、范围、验收、能力意图、skill 选择和反馈策略。
3. 单轮 JSON action 路径和 native tool-call 路径保存同等信息。
4. 通用环境向模型呈现能力目录、候选 skill 和可加载工具面，而不是把全部能力混成一个无结构大面板。
5. 模型逐轮调用工具，系统逐轮返回 observation。
6. Skill 激活由模型选择或用户/任务明确指定触发。
7. 公开进度来自 canonical progress / observation，不再由多个 fallback 字段拼接。

## 5. 目标执行流

### 5.1 单轮请求

```text
UserRequest
-> RuntimeCompilation(single_agent_turn)
-> Prompt + Context + CapabilityDirectory + VisibleReadTools
-> ModelActionRequest
   - respond
   - ask_user
   - tool_call
   - request_task_run
   - active_work_control
   - block
-> ActionAdmission
-> Execute or StartTaskRun
-> Observation / Final
```

所有语义动作由模型决定。系统不根据关键词直接替换模型动作。

### 5.2 持续任务交接

```text
ModelActionRequest(action_type=request_task_run)
-> TaskRunHandoffContract
   - goal
   - working_scope
   - completion_contract
   - capability_intent
   - skill_intent
   - observation_contract
   - active_work_relationship
-> TaskRunContract
-> RuntimeAssembly(task_execution)
-> ExecutionLoop
```

这里的 `capability_intent` 不是完整 operation 列表，而是模型表达“需要哪些服务类型”的接口。

### 5.3 持续任务执行循环

```text
TaskExecutionPacket
-> Model decides next action
-> tool_call / ask_user / respond / block
-> system executes tool_call
-> ObservationEnvelope
-> next TaskExecutionPacket
```

模型逐轮决策。系统不在第一轮预计算完整工具链。

### 5.4 工具和技能按需加载

```text
CapabilityDirectoryView
-> model selects capability group / skill / tool namespace
-> RuntimeAssembly expands selected surface
-> model calls concrete tool
-> ObservationEnvelope returns result
```

大工具面通过目录、命名空间、候选卡、tool search 或 deferred loading 控制，不一次性把所有工具说明塞进主上下文。

## 6. 数据模型设计

### 6.1 ModelActionRequest 保持为模型动作唯一入口

保留现有动作类型：

```text
respond
ask_user
tool_call
request_task_run
active_work_control
block
```

禁止新增旁路动作协议。所有模型动作必须进入 `ModelActionRequest` 这一唯一规范动作协议。

### 6.2 TaskRunHandoffContract

在 `task_contract_seed` 内新增结构化交接协议：

```json
{
  "user_visible_goal": "",
  "task_run_goal": "",
  "working_scope": {
    "target_objects": [],
    "workspace_refs": [],
    "source_refs": [],
    "excluded_scope": [],
    "known_constraints": []
  },
  "completion_criteria": [],
  "required_artifacts": [],
  "required_verifications": [],
  "capability_intent": {
    "needed_capability_groups": [],
    "preferred_tool_namespaces": [],
    "requires_deferred_tool_loading": true,
    "reason": ""
  },
  "skill_intent": {
    "selected_skill_ids": [],
    "candidate_skill_ids": [],
    "required_capability_tags": [],
    "reason": ""
  },
  "observation_contract": {
    "evidence_policy": "observation_required",
    "progress_granularity": "step",
    "finalization_requires_evidence": true
  },
  "active_work_relationship": ""
}
```

字段解释：

- `working_scope`：模型声明本任务工作对象和排除范围。
- `capability_intent`：模型声明需要的服务类别，不要求列完整 operation。
- `skill_intent`：模型选择或请求候选技能。
- `observation_contract`：模型声明希望系统用什么证据标准支撑完成。

### 6.3 CapabilityDirectoryView

新增模型可见的能力目录视图：

```json
{
  "capability_groups": [
    {
      "group_id": "file_work",
      "title": "文件处理",
      "use_when": "",
      "tool_namespaces": [],
      "candidate_tools": [],
      "candidate_skills": [],
      "loading_mode": "visible|deferred|search"
    }
  ],
  "tool_search_available": true,
  "skill_selection_available": true
}
```

用途：

- 让模型先理解有哪些服务类别。
- 避免通用环境一次性暴露无结构大工具面。
- 支持后续 tool search 和 skill expansion。

### 6.4 SkillActivationContract

新增或规范化 skill 激活记录：

```json
{
  "selected_skill_ids": [],
  "selection_source": "model_action|user_request|task_template",
  "selection_reason": "",
  "expanded_skill_refs": [],
  "rejected_skill_ids": []
}
```

约束：

- 候选 skill 不等于激活 skill。
- 只有激活 skill 才展开完整说明。
- rejected skill 必须进入 diagnostics，不能静默忽略。
- `skill_activation` 是 runtime_contract 中的唯一 skill 激活入口；不再从顶层散字段读取 skill 选择。
- active skill body 属于运行时激活内容，只能放入动态 prompt 区，不能进入 session/task stable cache prefix。

### 6.5 ObservationEnvelope

统一工具结果、运行守卫、失败、验证、产物和进度反馈：

```json
{
  "observation_id": "",
  "task_run_id": "",
  "turn_id": "",
  "source": "tool|runtime_guard|verification|artifact|user_control|system",
  "action_request_ref": "",
  "status": "ok|failed|denied|blocked|partial|canceled",
  "summary": "",
  "payload": {},
  "evidence_refs": [],
  "artifact_refs": [],
  "repair_instruction": "",
  "model_followup_required": true,
  "public_projection": {
    "progress_note": "",
    "current_judgment": "",
    "next_action_hint": "",
    "completion_status": ""
  }
}
```

注意：`next_action_hint` 只能是系统根据 observation 给出的事实提示，不能替模型决定下一步。

### 6.6 PublicProgressEnvelope

公开进度投影应从 observation 派生：

```json
{
  "run_id": "",
  "state": "running|waiting|blocked|completed|failed",
  "title": "",
  "line": "",
  "detail": "",
  "evidence_refs": [],
  "source_observation_refs": [],
  "updated_at": 0
}
```

前端只消费此对象，不再从多个 diagnostics 字段拼接状态。

### 6.7 Prompt Cache 分区约束

为了保证 prompt cache 稳定，系统必须区分“稳定服务目录”和“运行时 active 内容”：

- 稳定区可以放：基础角色规则、环境边界、action schema、task contract、CapabilityDirectoryView、候选 skill 卡片、工具目录摘要。
- 动态区必须放：active skill body、active_work_context、当前 task state、当前 observation、runtime control signal、用户临时 steer、provider tool replay。
- 禁止把 active skill body、当前 observation 或当前运行态放入 `session_stable` / `task` cache prefix。
- 如果某个字段会随执行步、工具结果、用户补充或当前 active work 变化，它必须使用 `cache_scope=none` 或等价动态段。
- CapabilityDirectoryView 可以稳定缓存，因为它是服务目录；`skill_activation` 只作为结构引用进入 contract，真正展开的 skill 内容进入动态区。

## 7. 模块升级计划

### 7.1 `backend/harness/runtime/compiler.py`

当前职责：

- 构建 single-agent turn packet。
- 构建 task execution packet。
- 暴露 action schema。
- 暴露 runtime projection。
- 渲染 skill candidate instruction。

改造动作：

1. 升级 `model_action_request_schema`：
   - 增加 `working_scope`。
   - 增加 `capability_intent`。
   - 增加 `skill_intent`。
   - 增加 `observation_contract`。
   - 明确说明 `capability_intent` 不是完整工具清单。

2. 增加 `CapabilityDirectoryView` 到模型稳定上下文：
   - 通用环境显示能力组。
   - skill 候选卡挂到能力组下。
   - 大工具面使用 deferred/search 标记。

3. 调整 runtime projection instruction：
   - 不再只强调“可见工具数量”。
   - 增加“可请求加载的能力组/技能/工具命名空间”。
   - 强调模型逐轮决定具体工具调用。

完成标准：

- 模型 schema 能清楚表达任务交接的能力意图。
- 单轮 prompt 中能看到能力目录，而不是散乱工具列表。
- 不引入权限策略变化。

### 7.2 `backend/harness/loop/model_action_protocol.py`

当前职责：

- 定义 `ModelActionRequest`。
- 校验模型输出 action。
- 去除空字段。

改造动作：

1. 增加规范 seed normalizer：
   - 规范 `working_scope`。
   - 规范 `capability_intent`。
   - 规范 `skill_intent`。
   - 规范 `observation_contract`。

2. 增加 contract gaps：
   - `request_task_run` 缺少目标或验收仍为 invalid。
   - 缺少能力意图时不能回落到旧的大工具面；必须由模型补齐、进入最小能力目录，或返回明确的协议修复 observation。

3. 删除旧字段主链路：
   - `resource_requirements` 不再作为运行时主输入；对应语义必须迁移到 `working_scope` 或 `capability_intent`。
   - `selected_skill_ids` 不再作为散字段主输入；对应语义必须迁移到 `skill_intent.selected_skill_ids`。
   - 旧字段只允许在同一改造切片内作为一次性代码迁移来源，迁移后运行时不得再读取旧字段兜底。

完成标准：

- 所有调用点输出规范 handoff payload。
- 旧 payload 不再被运行时静默接受；缺少规范关键字段时返回明确协议错误或修复 observation。
- 新字段不会被 `to_dict()` 意外丢弃。
- 空字段仍不污染 action payload。

### 7.3 `backend/harness/loop/single_agent_turn.py`

当前职责：

- 调用模型完成主会话单轮动作选择。
- 解析 native tool call。
- 处理 tool loop 和 observation followup。
- 启动 task lifecycle。

改造动作：

1. 原生 `request_task_run` 解析完整透传：
   - `working_scope`
   - `capability_intent`
   - `skill_intent`
   - `observation_contract`

2. JSON action 和 native tool-call 路径统一到同一 normalizer。

3. observation followup 保持模型主导：
   - 工具返回后只把 observation 给模型。
   - 不根据失败类型自动替模型选择新目标。

完成标准：

- native `request_task_run` 不再压扁任务交接信息。
- JSON 和 native 两条路径生成等价 `ModelActionRequest`。

### 7.4 `backend/harness/loop/task_lifecycle.py`

当前职责：

- 从 `ModelActionRequest` 构建 `TaskRunContract`。
- 创建 task run。
- 记录 runtime contract snapshot。
- 调度 executor。

改造动作：

1. `contract_from_action_request` 支持规范 handoff seed：
   - `working_scope` 写入 contract。
   - `capability_intent` 写入 contract 或 runtime_profile 的专用字段。
   - `skill_intent` 与 selected_skill_ids 统一。
   - `observation_contract` 写入 contract。

2. `_runtime_contract_from_task_run_contract` 输出：
   - `capability_intent`
   - `skill_activation`
   - `observation_contract`
   - `task_environment_id`

3. 初始化 task todo 时使用 completion + observation contract，不用 prompt 猜测验收。

完成标准：

- task_run diagnostics 中能看到完整 handoff contract。
- executor 从 task_run 读取的 runtime_contract 包含能力意图和 skill 激活信息。

### 7.5 `backend/harness/runtime/assembly.py`

当前职责：

- 装配 runtime profile、环境、工具、技能、prompt mount plan、operation authorization。

改造动作：

1. 增加 capability directory assembly：
   - 从环境、工具 registry、skill registry 生成能力组。
   - 按 task contract 的 `capability_intent` 决定哪些组直接显示，哪些组 deferred。

2. Skill assembly 改成两层：
   - candidate skill cards：模型可见候选。

## 11. 单主路径切换规则

### 11.1 同切片替换

每个阶段必须在同一改造切片内完成“新结构接入”和“旧主链路删除”。不允许先接入新结构、再长期保留旧结构作为运行时 fallback。

执行规则：

- 改 `ModelActionRequest` 时，同步改所有生成 action payload 的调用点。
- 改 `PublicProgressEnvelope` 时，同步删除前端旧字段拼接路径。
- 改 skill 激活时，同步删除散字段激活路径。
- 改 capability directory 时，同步删除通用环境无结构工具面展示路径。

### 11.2 显式失败优先

当新主链路缺少必要输入时，系统必须显式失败、询问、阻塞或返回协议修复 observation，不能读取旧字段继续执行。

触发例：

- `request_task_run` 缺少目标或验收：invalid。
- task execution 缺少 canonical observation：返回 `observation_missing`，不能用旧 tool log 拼接。
- 前端缺少 `PublicProgressEnvelope`：显示 `progress_projection_missing`，不能拼旧 diagnostics。
- skill 激活缺少 `SkillActivationContract`：只显示候选卡，不能展开旧 selected_skill 散字段。

### 11.3 回滚规则

回滚只能通过代码级 revert 或阶段分支回退完成，不能在运行时保留旧链路兜底。

允许：

- 阶段分支未合并前回退到旧实现。
- 发现设计错误时暂停实施并修订计划。

禁止：

- 在主链路里保留 `if canonical handoff missing then use old fields`。
- 前端 envelope 缺失时读取旧 diagnostics 展示。
- executor observation 缺失时从旧 event log 合成事实继续执行。
- skill 激活失败时自动使用旧 selected skill 字段。

### 11.4 切换完成条件

每个阶段合并前必须满足：

- 新主链路测试通过。
- 旧字段搜索确认不参与运行时决策。
- 旧字段搜索确认不参与前端展示 fallback。
- 行为测试覆盖新链路失败路径。
- 文档和测试不再描述旧链路作为保留方案。

## 12. 禁止的实现捷径

1. 禁止把问题修成 prompt 文案堆叠。
2. 禁止让系统根据关键词替模型选择工具链。
3. 禁止要求模型在任务启动前列完整 operation 清单。
4. 禁止把候选 skill 当成已激活 skill。
5. 禁止把工具结果只写进前端状态而不回传 observation。
6. 禁止保留旧主链路并用兼容理由长期双跑。
7. 禁止通过降低测试断言制造通过。
8. 禁止让 public progress 从多个散字段无序拼接。
9. 禁止在本计划中讨论或重设用户权限策略。

## 14. 最终目标形态

最终系统应呈现为：

```text
Prompt Layer
  告诉模型如何操作系统，不替系统做状态。

Model Action Layer
  模型输出唯一结构化动作。

Task Handoff Layer
  承接模型持续任务意图：目标、范围、验收、能力意图、skill、observation。

Capability Service Layer
  提供能力目录、工具命名空间、skill 候选和按需展开。

Execution Layer
  执行模型逐轮 tool_call，生成 observation。

Observation Layer
  保存工具结果、失败、产物、验证和修复信息。

Presentation Layer
  从 canonical progress / observation 投影给用户。
```

这才符合当前项目应有的成熟 agent 控制系统：模型操纵，系统服务，契约承接，工具执行，反馈闭环。

## 15. 2026-06-13 控制链路再审：精准、顺畅、全面

### 15.1 再审结论

本计划的最终验收口径调整为三个运行问题：

1. **控制是否精准**：模型选择的动作不能被系统静默改写、吞掉或降级；系统只能做准入、授权、执行、观察和投影。任何拒绝、缺字段、越界、审批、失败都必须形成可回传给模型的结构化 observation 或同轮 repair。
2. **运行是否顺畅**：常见任务不能因为协议细节、字段过严、投影冲突、日志混入监控或详情过大而卡住。任务入口、直接工具调用、持续执行、断线恢复和 UI 投影都必须能连续推进。
3. **覆盖是否全面**：必须同时覆盖 single-turn JSON action、provider native tool calls、直接工具调用、`request_task_run`、task execution、active work control、审批、失败恢复、文件写入、terminal、SSE 恢复、监控和日志分离。

这意味着：理念、命名和方法都服从运行验收。任何设计如果让模型更难进入任务、更容易重复读取、更难看到可信事实，或让前端继续从多个字段拼进度，都必须修正。

### 15.2 当前代码审计事实

截至本次再审，代码已经实现了部分主计划骨架：

- `backend/harness/loop/model_action_protocol.py` 已有 `working_scope`、`capability_intent`、`skill_intent`、`observation_contract` 的 normalizer，并禁止顶层旧 `selected_skill_ids`。
- `backend/harness/loop/task_lifecycle.py` 已把规范 handoff 写入 `TaskRunContract` 和 `runtime_contract.skill_activation`。
- `backend/harness/runtime/assembly.py` 已生成 `capability_directory`，并把工具和 skill 按能力组聚合。
- `backend/harness/runtime/compiler.py` 已把能力目录放入 single turn 和 task execution 的模型可见边界，并把 active skill body 放在 volatile 段。
- `backend/harness/loop/single_agent_turn.py` 已允许 RuntimeToolPlan 中的普通 native tools 直接进入单轮工具循环，并经 admission、action permit、batch plan、tool observation 回传。
- `backend/harness/loop/task_executor.py` 已有 observation、runtime guard、duplicate read-only guard、task observation projection 和 exploration advisory。
- `backend/harness/runtime/projection/*` 已有公开投影骨架，`task_projection.py` 和 `timeline_builder.py` 已经能过滤部分内部工具和构建公开 timeline。

但这些骨架还没有形成同一控制面。当前计划必须补齐以下 P0 缺口，否则系统会出现“局部精准、整体不顺”的状态。

### 15.3 P0 控制遗漏点

| --- | --- | --- | --- |

### 15.4 P0 控制矛盾点

1. **严格 contract 与顺畅入口冲突**
   `capability_intent_required_for_request_task_run` 不能变成终止用户意图的硬卡点。正确做法是同轮 repair，让模型按能力目录补齐；不是系统猜工具链，也不是直接失败收口。

2. **“任务必须 request_task_run”与“普通工具可直接执行”冲突**
   二者不是互斥。若用户要求持续交付、文件修改、验证闭环，模型应进入 `request_task_run`。若用户明确要求一次性检查、读取、搜索，或可见工具面已经具备完整 side-effect 承接，直接 tool_call 合法。计划书后续不得把直接工具路径简单判为不成熟，也不得让直接工具绕过 task run 的安全和 observation 边界。

3. **能力目录与 RuntimeToolPlan 双面冲突**
   `capability_directory` 是模型理解服务类别的目录，`RuntimeToolPlan` 是本轮实际可调用工具面。二者必须同源。目录里显示 visible 的工具必须在 RuntimeToolPlan 可调度；RuntimeToolPlan 可见工具必须能回溯到目录能力组或明确标成 system/direct tool。

4. **Observation 与 step summary 权威冲突**
   step summary 可以作为展示缓存或时间线摘要，但不能成为模型事实、监控事实或完成证据的主权威。完成、失败、阻塞、写入、验证必须以 observation 和 artifact/verification refs 为准。

5. **公开投影与日志面板冲突**
   日志是 raw trace，监控是当前状态，聊天/任务区是公开投影。前端不能再从 raw event name、tool name、terminal reason 或 diagnostics fallback 生成主状态文案。

6. **读取证据与恢复通道冲突**
   `read_persisted_tool_result` 只能恢复 omitted output，不等同当前工作区源码事实。`read_file` 覆盖范围、搜索定位、persisted replay 必须有来源和可信度区分。

### 15.5 直接工具路径成熟标准

`terminal/write_file/edit_file` 是否进入可见面，不由理念决定，只由承接能力决定：

```text
Visible tool definition
-> RuntimeToolPlan.model_visible_tools
-> ModelActionRequest(action_type=tool_call)
-> AdmissionDecision + ActionPermit
-> ToolBatchPlan(resource lock / approval / concurrency)
-> ToolObservation(canonical observation view)
-> FileState / ArtifactRefs / VerificationRefs / MaterialProgress
-> PublicProjectionEnvelope / TaskProjection / PublicTimeline
-> Model followup packet + UI monitor
```

成熟路径要求：

- 可见工具必须有 operation id、read_only/side_effect 标记、输入 schema、授权来源和环境边界。
- side-effect 工具必须有 before/after 或至少 file/material progress 证据；`write_file/edit_file` 要能记录 artifact/file state，terminal 要记录 command receipt 和 side-effect 边界。
- 执行失败、拒绝、审批等待、重复调用、超时都必须进入 observation，不只进入日志。
- direct turn 和 task execution 的 observation 字段命名必须兼容，便于同一投影层消费。
- 如果某个 side-effect tool 暂时没有上述承接，系统应在工具目录中标为 deferred 或不可见，并返回明确边界，而不是让模型调用后丢状态。

### 15.7 修订后的验收矩阵

#### 入口与动作

- JSON `request_task_run` 缺少 `capability_intent`：不能静默进入任务，也不能终止为普通失败；必须产生 repair observation，列出缺失字段和能力目录。
- native `request_task_run` 缺少规范字段：必须保留原 native call 诊断，并进入 repair；不能返回空 action 后变成 `invalid_native_action` 泛错。
- native 和 JSON 的完整 `request_task_run`：生成等价 `task_contract_seed`、`skill_activation`、`runtime_contract`。
- 模型选择普通 direct tool_call：如果工具可见且 permit 通过，必须执行并回传 observation；如果不通过，必须返回 denial observation。

#### 执行与恢复

- `write_file/edit_file` 成功：生成 file/material progress observation，包含 artifact/file refs、write count、source action ref。
- `terminal` 成功或失败：生成 command receipt observation，失败包含 structured error 和可恢复建议。
- 审批等待：生成 approval observation，UI 显示等待状态，模型后续包不把审批当成用户自然语言授权。
- 断线或后端重启：最新 observation、file state、material progress 和 task projection 可恢复；不能因为重连丢掉正在操作的文件事实。

#### 读取与上下文

- 同一文件已覆盖 100% 且 hash 未变：`fresh_read_conditions` 为空；prompt 不再同时说“已覆盖”和“必须重读全文”。
- search hit 被 read range 覆盖：投影显示 `covered_by_current_read_range=true`，不再要求对同一窗口重复 read。
- `read_persisted_tool_result` 只恢复 omitted output；不能作为当前源码事实。
- 重复探索达到阈值：只显示事实面板，包括重复次数、覆盖范围、新信息比例、material progress，不强制模型改动作。

#### 投影与 UI

- monitor 默认 payload 不含大 technical trace、prompt slot 或 provider protocol。
- monitor 主状态只来自 public projection/material facts，不从 raw logs 拼文案。
- log panel 可打开 raw trace，但不参与任务状态决策。
- 前端缺少 canonical projection 时显示 `progress_projection_missing`，不回落到旧 diagnostics 文案。

本次再审不要求推翻已实现的 action schema、capability directory、skill activation 骨架。应保留这些成熟方向，但必须补齐运行闭环。

后续代码实施前必须先按本节更新具体阶段任务。实施时如果发现某个直接工具路径无法满足成熟标准，应暂停该路径进入可见面或标记 deferred；不能为了“可用”保留不可追踪的副作用路径。

### 15.9 运行控制硬审计

后续实施不再按“某个理念是否正确”验收，而是按每条控制链路是否完整验收。每个入口、工具、恢复路径和投影路径都必须过下面 6 项：

| 审计项 | 必须成立 | 不成立时的处理 |
| --- | --- | --- |
| 入口完整 | JSON action、native tool call、active work control 都能保留原始请求、缺字段、解析错误和来源诊断。 | 返回同轮 repair observation 或 repair prompt；不能变成泛化 invalid，也不能丢掉原 native call。 |
| 决策精确 | 模型选择的 action_type、tool name、task handoff 不被系统静默改写。 | 返回 denial/boundary/repair observation；系统不能自动换成另一个任务或工具链。 |
| 授权明确 | 每个 side-effect action 都有 AdmissionDecision、ActionPermit、operation id、环境边界和资源锁。 | 工具从 visible 降为 deferred 或不可见；不能继续执行。 |
| 执行可追踪 | terminal/write_file/edit_file/read/search/todo 的结果都进入 canonical observation view，并携带 source action ref。 | 该工具路径不成熟，不能进入可见主工具面。 |
| 投影单一 | 模型后续包、PublicProjectionEnvelope、TaskProjection、PublicTimeline、monitor 主状态读取同一 observation/progress/material facts。 | 显示 `progress_projection_missing` 或 `observation_missing`；不能从旧 diagnostics、raw log、latest_step_summary 拼状态。 |
| 恢复连续 | SSE 断开、前端重连、后端重启后，最新 observation、file state、artifact refs、material progress 和 task projection 能恢复。 | 任务进入明确 recoverable/blocked 状态，并把缺失证据交给模型和 UI；不能让正在操作的文件事实消失。 |

一票否决项：

- native `request_task_run` 解析失败后只剩 `single_agent_turn_invalid_native_action`，没有原始 tool call 诊断和缺字段 repair。
- `terminal/write_file/edit_file` 可见但没有 file/material progress observation、artifact/file refs 或 command receipt。
- monitor、聊天附件或任务投影继续从 `latest_public_progress_note`、`latest_step_summary`、`public_action_state`、raw event name 或 diagnostics fallback 生成主状态。
- `read_file` 覆盖事实和 “code edit 前必须 fresh read” 同时投影给模型，且没有 `fresh_read_conditions` 解释。
- `read_persisted_tool_result` 投影出模型不可调用的 `replacement:*` 参数，或把 persisted output 当成当前工作区源码事实。
- dynamic context replacement store 仍硬依赖 model-visible `task_environment.storage_space.runtime_state_root`。
- monitor detail 默认携带大 technical trace、prompt slot 或 provider protocol，导致常规状态页面卡顿。

只有上述否决项全部清零，才允许认为“控制精准、运行顺畅、覆盖全面”。如果某条链路短期无法闭合，应显式降级为 deferred/diagnostic，不允许以兼容或兜底形式留在主路径。

### 15.11 blocked 老化、自动停止与运行时缓存释放

#### 问题结论

旧任务重新出现在监控台不是前端缓存问题，而是后端生命周期问题：

```text
RuntimeStateIndex.list_recent_task_runs
-> RuntimeMonitorService._global_live_items
-> RuntimeMonitorProjector.build_global_monitor
-> activity_is_monitor_visible(blocked/waiting)
-> RuntimeMonitorManagementProjector.attention lane
```

`blocked` / `waiting_executor` / `waiting_approval` 现在属于非终态，重启后仍会从 `state_index` 被投影为 monitor-visible waiting item。与此同时，`clear_from_monitor` 与 `delete_record` 又因为 `active_or_waiting_runtime` / `active_or_non_terminal_runtime` 被禁用，导致旧 blocked 既不能被自动结束，也不能被用户从控制台清理。这是生命周期权威缺口，不是 UI 展示缺口。

#### 目标原则

1. **模型负责执行决策，系统负责生命周期边界**
   - 新鲜 blocked/waiting 仍交给模型、用户或 approval 链路处理。
   - 超过保留窗口的 blocked/waiting 不再要求模型解释旧状态，由 runtime lifecycle retention 统一终态化。
   - 系统不能在活跃任务中替模型做下一步决策，只能在过期、无活 executor、无可恢复授权窗口时停止运行态。

2. **monitor 只投影，不决定**
   - monitor 不负责判断旧任务是否该消失。
   - monitor collection 前可触发 lifecycle retention sweep，但 sweep 的权威必须是独立生命周期服务。
   - monitor 只显示 sweep 后的 terminal/stopped 事实，不再用 hidden cache 掩盖非终态任务。

3. **停止运行态，不删除历史记录**
   - 自动降级为“停止”应写回 `TaskRun.status="aborted"`，并使用明确 `terminal_reason="blocked_expired"` 或 `terminal_reason="runtime_retention_expired"`。
   - `blocked_expired`、`runtime_retention_expired` 必须在 activity 投影中显示为 stopped，而不是 failed。
   - 保留 task record、event log、session messages、artifact refs、work rollout 和 lifecycle audit；删除历史记录仍只能走显式 delete lifecycle。

4. **释放可重建/易串线缓存，不释放用户资产**
   - 释放：executor control registry、pending tool invocation signal、dynamic context replacement、task-run scoped tool result preview cache、runtime_cache sandbox、file_state authority snapshot、monitor cache。
   - 保留：用户文件、artifact 文件、session message、formal memory、event log、trace/fact audit、prompt accounting、TaskRun terminal record。
   - 如果某个缓存尚无 task_run 级 prune API，必须新增明确 prune API；不能调用 `TaskRecordLifecycleManager.cleanup_single_task_record()` 代替，因为它会删除历史记录。

#### 目标权威链

```text
TaskRunStateIndex observe
-> TaskRunLifecycleRetention decide expired runtime
-> RuntimeControlBoundary authorize stop/release
-> TaskRunFinalizer execute terminal transition
-> EphemeralRuntimeCacheReleaser execute scoped release
-> RuntimeMonitorProjector present
```

| 层 | 责任 | 禁止事项 |
| --- | --- | --- |
| `TaskRunStateIndex` | 提供 task_run、updated_at、diagnostics、terminal 状态。 | 不决定是否过期。 |
| `TaskRunLifecycleRetention` | 根据状态、年龄、executor claim、paused/approval/recoverable 信息决定是否过期。 | 不隐藏 UI，不删除 task record，不改写模型目标。 |
| `RuntimeControlBoundary` | 确认没有活 executor 或先发 stop signal；避免终止 active claimed work。 | 不把 blocked 改成 completed，不重启任务。 |
| `TaskRunFinalizer` | 写回 `aborted + terminal_reason`，追加 lifecycle event/work rollout item，complete active turn。 | 不清空正式历史。 |
| `EphemeralRuntimeCacheReleaser` | 释放 task_run 作用域可重建缓存。 | 不删除 artifacts、session messages、formal memory。 |
| `RuntimeMonitorProjector` | 展示 stopped/recent 或 history。 | 不再把过期 blocked 作为 attention 主项。 |

#### 过期策略

默认策略必须可配置，但主路径必须有硬边界：

| 状态 | 默认 TTL | 自动动作 | 例外 |
| --- | ---: | --- | --- |
| `blocked` 且无 running executor claim | 2 小时 | `aborted + blocked_expired`，释放 ephemeral cache。 | 明确 `runtime_control.state=paused` 不自动处理。 |
| `waiting_executor` 且 executor lease lost/none、无活 background task | 2 小时 | `aborted + runtime_retention_expired`，释放 ephemeral cache。 | `recovery_action=resume_task_run/rerun_task_executor` 且最近仍有用户操作时保留到 TTL 后。 |
| `waiting_approval` | 24 小时 | `aborted + approval_expired`，清 pending approval，释放 ephemeral cache。 | 用户显式打开/刷新 approval 后重置 updated_at。 |
| `paused` | 不自动过期 | 保持 paused。 | paused 是用户明确控制态，不能由 retention 接管。 |
| running claimed executor | 不按 blocked TTL 直接终态化 | 先发 stop signal；超过 stop grace 后再终态化。 | stop grace 需短且可配置。 |

#### 代码落点

1. **新增生命周期保留服务**
   - 新增 `backend/harness/runtime/task_run_retention.py` 或 `backend/harness/runtime/run_monitor/retention_lifecycle.py`。
   - 提供 `sweep_expired_task_runs(now, limit, dry_run=False)`。
   - 输出 `authority`、`expired_task_run_ids`、`terminal_updates`、`released_cache_effects`、`skipped_reasons`。

2. **新增 task_run 终态化方法**
   - 不复用 record delete。
   - 使用 `RuntimeStateIndex.update_task_run()` 原子写回：
     - `status="aborted"`
     - `terminal_reason="blocked_expired" | "runtime_retention_expired" | "approval_expired"`
     - `diagnostics.runtime_control.state="stopped"`
     - `diagnostics.runtime_control.requested_by="runtime_retention"`
     - 清理 `recoverable_error`、`recovery_action`、`pending_approval`、`pending_user_steer_count`、`latest_user_steer_ref` 等会让 UI/模型误判可继续的字段。
   - 追加 `task_run_lifecycle_retention_stopped` event 和 work rollout item，便于审计。
   - 调用 `active_turn_registry.complete_bound_task(... terminal_reason=...)`。

3. **新增 ephemeral cache releaser**
   - 为 `ReplacementStore` 增加 `prune_task_runs({task_run_id})` 或按 `source_id/task_run_id` 的 prune。
   - 为 `ToolResultStore` / dynamic context persisted preview 增加 task_run scoped prune。
   - 调用 `FileStateAuthorityStore.prune_task_runs({task_run_id})`，前提是 terminal event 已记录 material/file summary。
   - 调用 `runtime_cache` 删除 sandbox namespace 中 `safe_cache_namespace(task_run_id)` 对应目录；需要新增按 cache_key 删除 API，不能靠 TTL 扫描碰运气。
   - runtime cache 删除在 monitor 请求路径内只能做快速摘除：不递归统计目录大小，不同步清空大目录；先把 cache entry 从 namespace 移到待删除区，后台线程再做物理删除。这样“释放运行态引用”和“磁盘清盘”分离，避免旧任务清理拖死监控。
   - 调用 tool invocation control registry 的 `cancel_by_caller(task_run_id=...)`，清理 pending/running tool 调用。
   - 清理 `_task_run_execution_control[task_run_id]`、background executor task name、monitor cache。

4. **修正 stopped 投影语义**
   - `run_monitor/activity.py` 的 `STOPPED_REASONS` 增加 `blocked_expired`、`runtime_retention_expired`、`approval_expired`。
   - `runtime/shared/models.py` 的 `RuntimeTerminalReason` 增加相同 reason，避免类型合同继续暗示这些 reason 非法。
   - `management.py` 对 terminal stopped 允许 `clear_from_monitor` / `delete_record`；不允许非终态 hidden 掩盖旧任务。

5. **接入位置**
   - `RuntimeMonitorService.collect_global_runtime_monitor()` 在读取 live items 前执行一次轻量 sweep，并在有更新时 invalidate monitor cache。
   - lifecycle sweep 必须有服务级节流，TTL 是小时级，monitor refresh 是秒级，不能每次投影请求都扫描 state/index 或释放缓存。默认 30 秒窗口内复用上次 sweep 结果，窗口过后再检查。
   - global live monitor 必须先预筛 live/diagnostic 候选，再做完整 item 投影；terminal recent/history 由 recent 分支单独投影，不能为了找活跃项把大批历史终态任务完整投影一遍。
   - 后端启动恢复也可执行一次 sweep，防止重启后旧 blocked 先闪现到 UI。
   - SSE/polling 只消费 sweep 后 monitor envelope。
   - SSE 首帧必须在返回 `StreamingResponse` 前完成初始 monitor snapshot 生成；不能出现 HTTP 200 已建立但 8 秒内没有 `runtime_monitor_snapshot` 的假连接状态。

#### 必须删除或禁止的旧处理

- 禁止把过期 blocked 仅写入 `RuntimeMonitorRetentionStore.hidden_signals`。
- 禁止放宽 `clear_from_monitor` 让非终态旧任务被 UI 藏起来。
- 禁止调用 `TaskRecordLifecycleManager.cleanup_single_task_record()` 作为自动释放手段。
- 禁止把 expired blocked 投影为 failed；它应是 stopped terminal。
- 禁止从 frontend reducer/local state 侧过滤旧任务来伪造修复。

#### 验收测试

1. `old blocked` 超过 TTL、无 executor claim：
   - sweep 后 `status="aborted"`，`terminal_reason="blocked_expired"`。
   - monitor attention 不再出现该 task_run。
   - recent/history 显示 stopped，不显示 failed。
   - ephemeral release effects 包含 dynamic context、runtime cache、file state、tool invocation cancel。

2. `fresh blocked` 未超过 TTL：
   - 不终态化，仍在 attention。
   - stop/clear/delete 权限保持当前策略。

3. `paused` 长时间存在：
   - 不被 retention 自动停止。
   - resume/stop 权限仍正常。

4. `waiting_executor` executor lost 超过 TTL：
   - 自动 stopped。
   - 清理 `recovery_action`，避免 monitor 显示可继续。

5. `waiting_approval` 超过 TTL：
   - 自动 stopped，`pending_approval` 被清理或标记 expired。
   - 不再卡在 approval attention。

6. 有 running executor claim 的 blocked/running：
   - 首轮只发 stop signal。
   - grace 超时后才终态化。
   - 不会打断仍在边界收口的 agent。

7. monitor 性能边界：
   - 首次 sweep 可以承担一次旧状态收口，后续 30 秒窗口内 monitor API 应稳定走缓存/轻投影。
   - runtime cache release 不做同步大目录 size scan。
   - global live monitor 不对已终态历史任务执行完整 runtime projection；recent terminal 保持可见但不参与 live 候选扫描。
   - SSE 连接后首个有效 chunk 必须是 `runtime_monitor_snapshot`，不能只有 200/keep-alive。

7. 回归验证：
   - `python -m pytest backend/tests/runtime_monitor_projection_test.py backend/tests/runtime_state_index_compaction_test.py backend/tests/dynamic_prompt_context_projection_test.py -q`
   - 固定端口实测：`127.0.0.1:8003/api/orchestration/runtime-monitor` 不再返回旧 blocked attention；SSE snapshot 同步。

### 15.12 网络重连与任务续跑分离

#### 设计准则

只有网络波动、前端 SSE 断开、浏览器刷新这类连接层问题支持自动重连。其它情况，包括后端重启、executor 丢失、模型中断、工具失败、blocked、approval 超时、用户 pause/stop/replan，都不支持系统自动续跑。

系统可以恢复状态、投影当前事实、标记可继续、提供继续入口；但不能把旧任务自动 schedule 成 running。是否继续语义执行必须来自用户显式动作或当前模型回合的明确决策。

#### 目标控制链

```text
TransportDisconnect
-> TransportReconnect
-> RuntimeStateProjection
-> ExecutorAliveCheck
-> Alive: present running
-> Lost: mark waiting_executor recoverable
-> User/Model explicit continue
-> ResumeActionPermit
-> TaskExecutorController.schedule
```

#### 必须删除的旧链路

- 删除 `AppRuntime.start_background_services()` 启动时自动调用 `start_runtime_recovered_task_run_executors()`。
- 删除 `runtime_start_recovery` 对 `waiting_executor` 旧断点的自动 schedule 扫描。
- `runtime_start_recovery` 只能表示“启动时发现 executor lease 丢失并写回可恢复状态”，不能作为 executor scheduler。
- `TaskExecutorController.schedule()` 必须拒绝 `scheduler="runtime_start_recovery"`，防止自动续跑路径从其它入口绕回来。

#### 保留能力

- `recover_interrupted_executor_leases()` 保留：它只把后端重启前的 running/scheduled executor claim 转成 `waiting_executor + recoverable_error + recovery_action`。
- 前端/监控 SSE 自动重连保留：重新拉取 `runtime_monitor_snapshot`，不改写任务状态。
- 用户点击继续、approval resume、当前模型回合明确继续时，仍可通过正常 resume/schedule 权限启动 executor。
- retention 仍负责旧 `blocked/waiting` 的过期停止和临时缓存释放。

#### Timeline 边界

断线恢复、resume、active task steer、用户重新介入之后，公共时间线必须从最新控制边界之后投影。旧工具输出、旧文件读取可以留在历史审计里，但不能重新混入当前可见任务段，避免恢复后串线。

#### 验收测试

1. 后端启动后，不自动 spawn runtime recovered executor。
2. 后端重启前的 running executor claim 被标记为 `waiting_executor`，但没有进入 running。
3. `scheduler="runtime_start_recovery"` 被 `TaskExecutorController.schedule()` 拒绝。
4. 用户显式 resume 仍能 schedule。
5. resume/active steer 后 `public_since_offset` 指向最新边界，旧 `backend/old_context.py` 不出现在当前公共段。
6. SSE 断线重连仍能收到 `runtime_monitor_snapshot`，且不触发 task_run 状态迁移。

### 15.13 三标准控制链路收口

#### 校正后的验收原则

后续所有控制链路优化只按三个标准验收：

1. **控制能不能精准**
   - 每个动作必须有唯一语义裁决来源。
   - 用户自然语言语义由模型判断；系统只能做事实暴露、结构承接、边界校验和执行授权。
   - 旧任务、历史断点、recent outcome 不能被伪装成当前可控工作。

2. **运行能不能顺畅**
   - 网络断线只恢复流和投影，不自动续跑 executor。
   - 用户显式继续、approval 通过、当前模型回合明确 `active_work_control` 时，才允许进入 schedule。
   - blocked/waiting 长时间滞留必须自动转 stopped 并释放可重建运行缓存，不能长期卡 UI 或让旧任务反复出现。

3. **覆盖够不够全面**
   - 聊天入口、active turn、direct task API、runtime monitor action、启动恢复、retention、projection、前端 SSE/polling 必须共享同一套控制边界。
   - 不能只在 UI 隐藏旧动作，后端 action service 和 API 入口必须同步删除或拒绝旧链路。
   - 测试必须覆盖误暂停、旧断点误恢复、缺失 active turn、monitor resume 后门和 retention 释放。

#### 当前实现仍存在的控制遗漏

1. **入口层仍有关键词控制器**
   - `backend/harness/entrypoint/runtime_facade.py` 的 `_active_turn_steer_action_from_user_message()` 会把“等一下/先停/继续/停止”等文本直接翻译成 `pause_active_work`、`continue_active_work`、`stop_active_work`。
   - 这与本计划第 5.1 节“所有语义动作由模型决定，系统不根据关键词直接替换模型动作”冲突。
   - 典型误伤：用户说“等一下 task runtime 为什么必须有 task_environment”时，系统不应强制 pause；应把这句话交给模型判断是质疑/讨论/新指令/控制动作。

2. **历史 latest task 仍可能进入 active_work_context**
   - `runtime_facade._current_work_context_from_latest_task()` 在没有 active turn 时仍从 `current_session_task_run()` 构造 `active_work_context`。
   - 这会让重启恢复、旧 blocked、旧 waiting_executor 看起来像当前可控工作，和 `recent_work_outcome` 只读边界冲突。
   - 目标：没有 active turn 或明确当前控制边界时，最新任务只能进入 `recent_work_outcome` 或 monitor/history，不进入 active_work_context。

3. **same-run resumable 判定过宽**
   - `runtime_facade._active_work_context_from_task_run()` 对 `status in {"waiting_executor", "blocked"}` 默认 `same_run_allowed=True`。
   - `task_run_recovery_state.recovery_state_for_task_run()` 对 `waiting_executor + executor_lease_state in {"lost", "none", "blocked"}` 也会给出 `same_run_resumable=True`。
   - 目标：same-run resume 只能来自明确的 paused、approval granted、recoverable_error + recovery_action、当前 active turn 控制边界或模型/用户显式继续；不能由宽泛 waiting_executor 状态自动获得。

4. **缺失 active turn 时 direct API 未严格拒绝**
   - `backend/api/orchestration_harness.py` 的 `_assert_expected_active_turn()` 在调用方传入 `expected_active_turn_id` 但 registry 当前无 active turn 时直接 return。
   - 后端重启后 active registry 为空，这会让 execute/resume/pause/stop 绕过当前 turn 匹配。
   - 目标：只要调用方提供 expected id，active turn 缺失就必须 409，错误语义应是 `active_turn_unavailable`。

5. **monitor resume 仍是后端可执行动作**
   - 前端 `RunTaskLane.tsx` / `RunManagementWorkbench.tsx` 已隐藏 `resume_task`，但 `RuntimeMonitorActionService.execute()` 仍允许 `resume_task` 并调度 `runtime_monitor_action_resume`。
   - 这属于 UI 收口、后端未收口；不满足覆盖全面。
   - 目标：monitor 只做 observe/present/stop/clear/delete 这类运维动作；语义继续必须走 active turn 或明确 direct API 边界。

#### 目标控制链

```text
RequestFacts
-> ActiveTurnFacts
-> ContextCandidates(active_work only from active turn, recent_work_outcome from history)
-> ModelTurnDecision(active_work_control/request_task_run/respond/ask_user/block)
-> ActionPermit(expected active turn + task state + recovery boundary)
-> RuntimeStartPacket
-> TaskExecutorController.schedule
-> Observation/Projection
```

#### 必须删除或收紧的旧链路

- 删除 `_active_turn_steer_action_from_user_message()` 的自然语言关键词决策权。
- 删除“无 active turn 时 latest task 进入 active_work_context”的路径。
- 删除 `waiting_executor`/`blocked` 状态本身赋予 same-run resume 的宽判定。
- 删除或禁用 `runtime_monitor_action_resume` 后端执行入口；不能只靠前端隐藏。
- 修改 `_assert_expected_active_turn()`：提供 expected id 但 active turn 缺失时必须拒绝。

#### 实施切片

1. **精准收口**
   - 文件：`backend/harness/entrypoint/runtime_facade.py`、`backend/harness/loop/task_run_recovery_state.py`、`backend/harness/task_run_state_view.py`。
   - 改动：active steer 不再关键词 pause/continue/stop；latest task 不再自动 active；same-run resume 只认明确恢复边界。

2. **API 和 monitor 后门收口**

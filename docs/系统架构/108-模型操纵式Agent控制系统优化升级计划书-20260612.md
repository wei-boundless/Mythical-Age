# 模型操纵式 Agent 控制系统优化升级计划书

日期：2026-06-12  
状态：计划书  
适用范围：通用任务环境、主会话单轮决策、持续任务交接、工具/技能供给、运行反馈、公开进度投影  
明确不包含：权限策略重设、权限模式默认值讨论、用户个人权限配置调整

---

## 1. 结论

本项目的目标控制模型应明确为：

```text
模型是操纵者。
系统是服务提供者、执行承接者、状态记录者、边界控制者和反馈投影者。
Prompt 指导模型如何操作系统。
契约是模型把意图交给系统承接的结构化接口。
工具、技能、上下文、运行状态和 observation 都是系统向模型提供的服务面。
```

因此，本次优化不是把系统升级成更强的任务语义裁决者，也不是让系统在任务启动前替模型推断完整工具链。优化目标是让模型拥有更清晰、更低摩擦、更可追踪的操作接口，并让系统按模型意图提供能力、执行动作、返回 observation。

成熟 agent 的精确操作模式应是：

```text
1. 系统向模型展示角色规则、上下文、能力目录、可见工具/技能和运行边界。
2. 模型做本轮语义裁决：respond / ask_user / tool_call / request_task_run / active_work_control / block。
3. 如果模型请求工具，系统执行结构化 tool call，并返回 tool observation。
4. 如果模型请求持续任务，系统创建 task run，记录任务目标、范围、验收、能力意图、技能选择和反馈策略。
5. 执行循环中，模型根据最新 observation 逐轮决定下一步动作。
6. 系统只做服务装配、执行、记录、边界控制和反馈投影，不替模型改写任务目标。
```

本计划把通用任务环境从“宽能力默认暴露”升级为“模型按需操纵服务面”的结构。

---

## 2. 成熟方案依据

本计划只采用公开成熟 agent 方案能确认的机制，不假设 Codex 或 Claude Code 的私有内部实现。

1. OpenAI Function Calling / Responses 工具调用模式：模型根据上下文决定是否发出工具调用，应用执行工具并把结果回传给模型，模型继续生成下一步或最终结果。  
   来源：https://developers.openai.com/api/docs/guides/function-calling

2. OpenAI Agents SDK Runner loop：运行循环调用模型；模型可能输出最终回复、handoff 或 tool calls；如果有 tool calls，runner 执行工具并把结果加入上下文后继续循环。  
   来源：https://openai.github.io/openai-agents-python/running_agents/

3. OpenAI Agents SDK Tools：工具可以按需提供，支持 hosted tools、function tools、agent-as-tool、tool search、approval gates 和 tracing。关键启发是“大能力面不必一次性全部塞进模型上下文”。  
   来源：https://openai.github.io/openai-agents-python/tools/

4. Anthropic Claude Tool Use：Claude 依据工具描述和当前请求决定是否使用工具，返回结构化 tool_use；客户端执行工具后返回 tool_result。  
   来源：https://platform.claude.com/docs/en/agents-and-tools/tool-use/overview

这些方案共同确认：

- 模型是动作选择者。
- 系统提供工具、上下文、执行和反馈。
- 工具调用逐轮发生，不要求任务启动前声明完整工具序列。
- 大工具面应通过目录、命名空间、搜索或延迟加载管理。
- 执行结果必须回到模型可消费的 observation。

## 3. 当前系统状态

### 3.1 已有良好基础

当前系统已经有一套可继续演进的结构：

- `ModelActionRequest` 已经把模型动作收敛为结构化 action，包含 `respond / ask_user / tool_call / request_task_run / active_work_control / block`。  
  相关文件：`backend/harness/loop/model_action_protocol.py`

- `model_action_request_schema` 已经把 action 选择规则、公开反馈、任务交接 seed、完成标准、产物和验证要求暴露给模型。  
  相关文件：`backend/harness/runtime/compiler.py`

- `TaskRunContract` 已经承载持续任务目标、完成标准、资源要求、策略、运行 profile 和 prompt contract。  
  相关文件：`backend/harness/loop/task_lifecycle.py`

- `RuntimeAssembly` 已经集中装配环境、工具、技能候选、prompt mount plan、operation authorization 和 runtime projection。  
  相关文件：`backend/harness/runtime/assembly.py`

- `task_executor` 已经有 observation、重复工具调用 guard、step summary 和 public action state 记录。  
  相关文件：`backend/harness/loop/task_executor.py`

这些基础说明系统不是要推倒重做，而是要把模型操纵接口、能力供给和反馈闭环收束成更清晰的协议族。

### 3.2 当前不流畅点

#### 3.2.1 模型操纵接口不够完整

`request_task_run` 当前能稳定承接目标、产物、验证和完成标准，但对“模型想调用哪些类型的服务”表达不足。

目前 schema 中虽然存在 `resource_requirements`、`permission_requirements`、`acceptance_policy`、`recovery_policy` 等字段，但这些字段是泛型对象，缺少模型可直接理解的能力意图结构。

结果是：模型知道自己要执行任务，却没有一个清晰槽位表达“我需要文件工作、网络研究、浏览器、命令执行、产物生成、子 agent 或某个 skill”。

#### 3.2.2 原生工具路径会压扁任务交接信息

`single_agent_turn` 中对原生 `request_task_run` 的解析只稳定提取：

- `user_visible_goal`
- `task_run_goal`
- `required_artifacts`
- `required_verifications`
- `completion_criteria`
- `active_work_relationship`

这会导致模型即使在 schema 层能表达更多任务交接信息，也可能在 native tool-call 路径中丢失。

#### 3.2.3 通用环境偏向宽候选池，不利于模型按需操纵

通用环境现在更像一个宽泛默认能力入口。对模型来说，它容易看到“很多东西都可用”，但不一定知道本任务该先加载哪个能力组、哪个 skill、哪个工具命名空间。

这不是权限配置问题，而是能力供给方式问题。成熟 agent 更倾向：

```text
先暴露能力目录和候选卡
再由模型按任务目标选择能力组、技能或工具命名空间
再进入具体工具调用循环
```

#### 3.2.4 Skill 是候选，但没有形成模型选择闭环

当前 skill runtime views 主要按运行时可用能力过滤后展示候选。模型可以填写 `selected_skill_ids`，但任务契约、skill 展开、执行循环之间还没有形成强闭环。

成熟形态应是：

```text
系统展示 skill 候选卡
模型选择 skill 或能力标签
系统展开被选 skill 的完整说明
模型依据 skill 操作工具/上下文
skill 产出的证据进入 observation
```

#### 3.2.5 Feedback 还没有成为单一 observation 权威

当前系统里公开进度和执行反馈散落在：

- `public_progress_note`
- `public_action_state`
- step summary
- task diagnostics latest fields
- session timeline
- frontend runtime store fallback

这会导致用户侧看到的“正在做什么”和模型侧用于下一步决策的 observation 不是同一个权威对象。

成熟 agent 需要把反馈统一成：

```text
tool_result / runtime_guard / artifact_ref / verification_result / progress_event
-> canonical observation envelope
-> model followup packet
-> public progress projection
```

## 4. 设计目标

### 4.1 总目标

把系统升级为“模型操纵式控制系统”：

```text
模型负责判断和发起动作。
系统负责提供能力目录、承接结构化动作、执行工具、返回 observation、记录状态、投影进度。
```

### 4.2 非目标

本计划不处理以下事项：

- 不调整用户的权限模式。
- 不讨论默认权限配置。
- 不以“收紧权限”为主要优化手段。
- 不要求模型在任务启动前列出完整 operation 清单。
- 不让系统根据关键词替模型推断任务工具链。
- 不用 prompt 补丁替代协议和运行时结构修复。

### 4.3 成功状态

升级完成后，系统应满足：

1. 模型能通过清晰 action schema 表达本轮动作和持续任务交接意图。
2. `request_task_run` 能承接目标、范围、验收、能力意图、skill 选择和反馈策略。
3. 单轮 JSON action 路径和 native tool-call 路径保存同等信息。
4. 通用环境向模型呈现能力目录、候选 skill 和可加载工具面，而不是把全部能力混成一个无结构大面板。
5. 模型逐轮调用工具，系统逐轮返回 observation。
6. Skill 激活由模型选择或用户/任务明确指定触发。
7. 公开进度来自 canonical progress / observation，不再由多个 fallback 字段拼接。

## 5. 目标执行流

### 5.1 单轮请求

```text
UserRequest
-> RuntimeCompilation(single_agent_turn)
-> Prompt + Context + CapabilityDirectory + VisibleReadTools
-> ModelActionRequest
   - respond
   - ask_user
   - tool_call
   - request_task_run
   - active_work_control
   - block
-> ActionAdmission
-> Execute or StartTaskRun
-> Observation / Final
```

所有语义动作由模型决定。系统不根据关键词直接替换模型动作。

### 5.2 持续任务交接

```text
ModelActionRequest(action_type=request_task_run)
-> TaskRunHandoffContract
   - goal
   - working_scope
   - completion_contract
   - capability_intent
   - skill_intent
   - observation_contract
   - active_work_relationship
-> TaskRunContract
-> RuntimeAssembly(task_execution)
-> ExecutionLoop
```

这里的 `capability_intent` 不是完整 operation 列表，而是模型表达“需要哪些服务类型”的接口。

### 5.3 持续任务执行循环

```text
TaskExecutionPacket
-> Model decides next action
-> tool_call / ask_user / respond / block
-> system executes tool_call
-> ObservationEnvelope
-> next TaskExecutionPacket
```

模型逐轮决策。系统不在第一轮预计算完整工具链。

### 5.4 工具和技能按需加载

```text
CapabilityDirectoryView
-> model selects capability group / skill / tool namespace
-> RuntimeAssembly expands selected surface
-> model calls concrete tool
-> ObservationEnvelope returns result
```

大工具面通过目录、命名空间、候选卡、tool search 或 deferred loading 控制，不一次性把所有工具说明塞进主上下文。

## 6. 数据模型设计

### 6.1 ModelActionRequest 保持为模型动作唯一入口

保留现有动作类型：

```text
respond
ask_user
tool_call
request_task_run
active_work_control
block
```

禁止新增旁路动作协议。所有模型动作必须进入 `ModelActionRequest` 这一唯一规范动作协议。

### 6.2 TaskRunHandoffContract

在 `task_contract_seed` 内新增结构化交接协议：

```json
{
  "user_visible_goal": "",
  "task_run_goal": "",
  "working_scope": {
    "target_objects": [],
    "workspace_refs": [],
    "source_refs": [],
    "excluded_scope": [],
    "known_constraints": []
  },
  "completion_criteria": [],
  "required_artifacts": [],
  "required_verifications": [],
  "capability_intent": {
    "needed_capability_groups": [],
    "preferred_tool_namespaces": [],
    "requires_deferred_tool_loading": true,
    "reason": ""
  },
  "skill_intent": {
    "selected_skill_ids": [],
    "candidate_skill_ids": [],
    "required_capability_tags": [],
    "reason": ""
  },
  "observation_contract": {
    "evidence_policy": "observation_required",
    "progress_granularity": "step",
    "finalization_requires_evidence": true
  },
  "active_work_relationship": ""
}
```

字段解释：

- `working_scope`：模型声明本任务工作对象和排除范围。
- `capability_intent`：模型声明需要的服务类别，不要求列完整 operation。
- `skill_intent`：模型选择或请求候选技能。
- `observation_contract`：模型声明希望系统用什么证据标准支撑完成。

### 6.3 CapabilityDirectoryView

新增模型可见的能力目录视图：

```json
{
  "capability_groups": [
    {
      "group_id": "file_work",
      "title": "文件处理",
      "use_when": "",
      "tool_namespaces": [],
      "candidate_tools": [],
      "candidate_skills": [],
      "loading_mode": "visible|deferred|search"
    }
  ],
  "tool_search_available": true,
  "skill_selection_available": true
}
```

用途：

- 让模型先理解有哪些服务类别。
- 避免通用环境一次性暴露无结构大工具面。
- 支持后续 tool search 和 skill expansion。

### 6.4 SkillActivationContract

新增或规范化 skill 激活记录：

```json
{
  "selected_skill_ids": [],
  "selection_source": "model_action|user_request|task_template",
  "selection_reason": "",
  "expanded_skill_refs": [],
  "rejected_skill_ids": []
}
```

约束：

- 候选 skill 不等于激活 skill。
- 只有激活 skill 才展开完整说明。
- rejected skill 必须进入 diagnostics，不能静默忽略。
- `skill_activation` 是 runtime_contract 中的唯一 skill 激活入口；不再从顶层散字段读取 skill 选择。
- active skill body 属于运行时激活内容，只能放入动态 prompt 区，不能进入 session/task stable cache prefix。

### 6.5 ObservationEnvelope

统一工具结果、运行守卫、失败、验证、产物和进度反馈：

```json
{
  "observation_id": "",
  "task_run_id": "",
  "turn_id": "",
  "source": "tool|runtime_guard|verification|artifact|user_control|system",
  "action_request_ref": "",
  "status": "ok|failed|denied|blocked|partial|canceled",
  "summary": "",
  "payload": {},
  "evidence_refs": [],
  "artifact_refs": [],
  "repair_instruction": "",
  "model_followup_required": true,
  "public_projection": {
    "progress_note": "",
    "current_judgment": "",
    "next_action_hint": "",
    "completion_status": ""
  }
}
```

注意：`next_action_hint` 只能是系统根据 observation 给出的事实提示，不能替模型决定下一步。

### 6.6 PublicProgressEnvelope

公开进度投影应从 observation 派生：

```json
{
  "run_id": "",
  "state": "running|waiting|blocked|completed|failed",
  "title": "",
  "line": "",
  "detail": "",
  "evidence_refs": [],
  "source_observation_refs": [],
  "updated_at": 0
}
```

前端只消费此对象，不再从多个 diagnostics 字段拼接状态。

### 6.7 Prompt Cache 分区约束

为了保证 prompt cache 稳定，系统必须区分“稳定服务目录”和“运行时 active 内容”：

- 稳定区可以放：基础角色规则、环境边界、action schema、task contract、CapabilityDirectoryView、候选 skill 卡片、工具目录摘要。
- 动态区必须放：active skill body、active_work_context、当前 task state、当前 observation、runtime control signal、用户临时 steer、provider tool replay。
- 禁止把 active skill body、当前 observation 或当前运行态放入 `session_stable` / `task` cache prefix。
- 如果某个字段会随执行步、工具结果、用户补充或当前 active work 变化，它必须使用 `cache_scope=none` 或等价动态段。
- CapabilityDirectoryView 可以稳定缓存，因为它是服务目录；`skill_activation` 只作为结构引用进入 contract，真正展开的 skill 内容进入动态区。

## 7. 模块升级计划

### 7.1 `backend/harness/runtime/compiler.py`

当前职责：

- 构建 single-agent turn packet。
- 构建 task execution packet。
- 暴露 action schema。
- 暴露 runtime projection。
- 渲染 skill candidate instruction。

改造动作：

1. 升级 `model_action_request_schema`：
   - 增加 `working_scope`。
   - 增加 `capability_intent`。
   - 增加 `skill_intent`。
   - 增加 `observation_contract`。
   - 明确说明 `capability_intent` 不是完整工具清单。

2. 增加 `CapabilityDirectoryView` 到模型稳定上下文：
   - 通用环境显示能力组。
   - skill 候选卡挂到能力组下。
   - 大工具面使用 deferred/search 标记。

3. 调整 runtime projection instruction：
   - 不再只强调“可见工具数量”。
   - 增加“可请求加载的能力组/技能/工具命名空间”。
   - 强调模型逐轮决定具体工具调用。

完成标准：

- 模型 schema 能清楚表达任务交接的能力意图。
- 单轮 prompt 中能看到能力目录，而不是散乱工具列表。
- 不引入权限策略变化。

### 7.2 `backend/harness/loop/model_action_protocol.py`

当前职责：

- 定义 `ModelActionRequest`。
- 校验模型输出 action。
- 去除空字段。

改造动作：

1. 增加规范 seed normalizer：
   - 规范 `working_scope`。
   - 规范 `capability_intent`。
   - 规范 `skill_intent`。
   - 规范 `observation_contract`。

2. 增加 contract gaps：
   - `request_task_run` 缺少目标或验收仍为 invalid。
   - 缺少能力意图时不能回落到旧的大工具面；必须由模型补齐、进入最小能力目录，或返回明确的协议修复 observation。

3. 删除旧字段主链路：
   - `resource_requirements` 不再作为运行时主输入；对应语义必须迁移到 `working_scope` 或 `capability_intent`。
   - `selected_skill_ids` 不再作为散字段主输入；对应语义必须迁移到 `skill_intent.selected_skill_ids`。
   - 旧字段只允许在同一改造切片内作为一次性代码迁移来源，迁移后运行时不得再读取旧字段兜底。

完成标准：

- 所有调用点输出规范 handoff payload。
- 旧 payload 不再被运行时静默接受；缺少规范关键字段时返回明确协议错误或修复 observation。
- 新字段不会被 `to_dict()` 意外丢弃。
- 空字段仍不污染 action payload。

### 7.3 `backend/harness/loop/single_agent_turn.py`

当前职责：

- 调用模型完成主会话单轮动作选择。
- 解析 native tool call。
- 处理 tool loop 和 observation followup。
- 启动 task lifecycle。

改造动作：

1. 原生 `request_task_run` 解析完整透传：
   - `working_scope`
   - `capability_intent`
   - `skill_intent`
   - `observation_contract`

2. JSON action 和 native tool-call 路径统一到同一 normalizer。

3. observation followup 保持模型主导：
   - 工具返回后只把 observation 给模型。
   - 不根据失败类型自动替模型选择新目标。

完成标准：

- native `request_task_run` 不再压扁任务交接信息。
- JSON 和 native 两条路径生成等价 `ModelActionRequest`。

### 7.4 `backend/harness/loop/task_lifecycle.py`

当前职责：

- 从 `ModelActionRequest` 构建 `TaskRunContract`。
- 创建 task run。
- 记录 runtime contract snapshot。
- 调度 executor。

改造动作：

1. `contract_from_action_request` 支持规范 handoff seed：
   - `working_scope` 写入 contract。
   - `capability_intent` 写入 contract 或 runtime_profile 的专用字段。
   - `skill_intent` 与 selected_skill_ids 统一。
   - `observation_contract` 写入 contract。

2. `_runtime_contract_from_task_run_contract` 输出：
   - `capability_intent`
   - `skill_activation`
   - `observation_contract`
   - `task_environment_id`

3. 初始化 task todo 时使用 completion + observation contract，不用 prompt 猜测验收。

完成标准：

- task_run diagnostics 中能看到完整 handoff contract。
- executor 从 task_run 读取的 runtime_contract 包含能力意图和 skill 激活信息。

### 7.5 `backend/harness/runtime/assembly.py`

当前职责：

- 装配 runtime profile、环境、工具、技能、prompt mount plan、operation authorization。

改造动作：

1. 增加 capability directory assembly：
   - 从环境、工具 registry、skill registry 生成能力组。
   - 按 task contract 的 `capability_intent` 决定哪些组直接显示，哪些组 deferred。

2. Skill assembly 改成两层：
   - candidate skill cards：模型可见候选。

## 10. 验证矩阵

### 10.1 Action Contract

- `request_task_run` 缺少目标：invalid。
- `request_task_run` 缺少验收：invalid。
- `request_task_run` 缺少 `capability_intent`：invalid，必须由模型补齐能力意图后才能启动。
- handoff seed 包含 skill：写入 runtime_contract。
- native request_task_run 包含 handoff seed：字段不丢。

### 10.2 Capability Directory

- 通用环境单轮：展示能力目录。
- 轻量问答：不加载重型工具面。
- 文件任务：文件能力组可见。
- 研究任务：搜索/浏览能力组可见。
- 复杂任务：可见 skill 候选卡，但不自动展开全文。

### 10.3 Skill

- 模型未选择 skill：只展示候选卡。
- 模型选择 skill：展开完整 skill prompt。
- skill 不存在：记录 rejected。
- skill 与任务能力不匹配：记录 rejected 或 candidate-only，不静默激活。

### 10.4 Observation

- tool ok：生成 ok observation。
- tool failed：生成 failed observation，模型 followup。
- runtime guard：生成 runtime_guard observation。
- artifact created：生成 artifact refs。
- verification failed：模型不能 final 为完成，必须继续、询问或 block。

### 10.5 Public Progress

- 单步工具执行：progress 引用 source observation。
- SSE 断线恢复：恢复最新 progress envelope。
- task failed：显示失败原因和 evidence refs。
- task completed：显示最终结果和验证证据。

## 11. 单主路径切换规则

### 11.1 同切片替换

每个阶段必须在同一改造切片内完成“新结构接入”和“旧主链路删除”。不允许先接入新结构、再长期保留旧结构作为运行时 fallback。

执行规则：

- 改 `ModelActionRequest` 时，同步改所有生成 action payload 的调用点。
- 改 `PublicProgressEnvelope` 时，同步删除前端旧字段拼接路径。
- 改 skill 激活时，同步删除散字段激活路径。
- 改 capability directory 时，同步删除通用环境无结构工具面展示路径。

### 11.2 显式失败优先

当新主链路缺少必要输入时，系统必须显式失败、询问、阻塞或返回协议修复 observation，不能读取旧字段继续执行。

触发例：

- `request_task_run` 缺少目标或验收：invalid。
- task execution 缺少 canonical observation：返回 `observation_missing`，不能用旧 tool log 拼接。
- 前端缺少 `PublicProgressEnvelope`：显示 `progress_projection_missing`，不能拼旧 diagnostics。
- skill 激活缺少 `SkillActivationContract`：只显示候选卡，不能展开旧 selected_skill 散字段。

### 11.3 回滚规则

回滚只能通过代码级 revert 或阶段分支回退完成，不能在运行时保留旧链路兜底。

允许：

- 阶段分支未合并前回退到旧实现。
- 发现设计错误时暂停实施并修订计划。

禁止：

- 在主链路里保留 `if canonical handoff missing then use old fields`。
- 前端 envelope 缺失时读取旧 diagnostics 展示。
- executor observation 缺失时从旧 event log 合成事实继续执行。
- skill 激活失败时自动使用旧 selected skill 字段。

### 11.4 切换完成条件

每个阶段合并前必须满足：

- 新主链路测试通过。
- 旧字段搜索确认不参与运行时决策。
- 旧字段搜索确认不参与前端展示 fallback。
- 行为测试覆盖新链路失败路径。
- 文档和测试不再描述旧链路作为保留方案。

## 12. 禁止的实现捷径

1. 禁止把问题修成 prompt 文案堆叠。
2. 禁止让系统根据关键词替模型选择工具链。
3. 禁止要求模型在任务启动前列完整 operation 清单。
4. 禁止把候选 skill 当成已激活 skill。
5. 禁止把工具结果只写进前端状态而不回传 observation。
6. 禁止保留旧主链路并用兼容理由长期双跑。
7. 禁止通过降低测试断言制造通过。
8. 禁止让 public progress 从多个散字段无序拼接。
9. 禁止在本计划中讨论或重设用户权限策略。

## 13. 预期收益

升级后系统会获得：

1. 模型操纵面更清晰  
   模型知道可以如何发起任务、选择能力、激活 skill、调用工具、处理 observation。

2. 通用环境更顺滑  
   通用环境不再是无结构大能力池，而是能力目录和按需服务入口。

3. 技能使用更自然  
   skill 从“候选说明”升级为“模型可选择的能力包”。

4. 反馈闭环更成熟  
   每次工具执行、失败、验证和产物都成为模型可消费 observation。

5. 前端状态更稳定  
   用户看到的进度来自 canonical progress，而不是多个 fallback 字段。

6. 后续维护更清晰  
   Prompt、contract、runtime assembly、execution loop、presentation 各自职责明确。

## 14. 最终目标形态

最终系统应呈现为：

```text
Prompt Layer
  告诉模型如何操作系统，不替系统做状态。

Model Action Layer
  模型输出唯一结构化动作。

Task Handoff Layer
  承接模型持续任务意图：目标、范围、验收、能力意图、skill、observation。

Capability Service Layer
  提供能力目录、工具命名空间、skill 候选和按需展开。

Execution Layer
  执行模型逐轮 tool_call，生成 observation。

Observation Layer
  保存工具结果、失败、产物、验证和修复信息。

Presentation Layer
  从 canonical progress / observation 投影给用户。
```

这才符合当前项目应有的成熟 agent 控制系统：模型操纵，系统服务，契约承接，工具执行，反馈闭环。

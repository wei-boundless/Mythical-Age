# 统一公开投影协议与投屏时序设计

日期：2026-06-10

## 1. 背景

当前主会话投影出现了三个典型问题：

1. 模型正文、工具输出、系统状态被多个链路重复解释，最终在聊天消息里混在一起。
2. `task_control` / `task_executor_scheduled` 一类生命周期事件被显示成“后台任务已接管”“我会开始处理”等近似 assistant 正文。
3. 后台任务实际运行后，任务模式的后续进展不稳定投屏到当前会话，常见表现是只显示处理清单，不显示工具动作、观察、当前步骤和完成收口。

这不是单个文案问题，而是公开投影权威分裂：

- `backend/api/chat.py` 对 chat stream 做一次 public projection。
- `backend/harness/runtime/runtime_monitor_public_projection.py` 对 runtime monitor SSE 再做一次 public projection。
- `frontend/src/lib/runtimeVisibilityProjection.ts` 继续解释 runtime event。
- `frontend/src/lib/store/events.ts` 再生成 stage/status/receipt。
- `frontend/src/lib/store/runtime.ts` 又把 runtime monitor event 转成 runtime attachment。
- `PublicTimelineActivity.tsx` 最后根据 `kind/surface/source_authority` 再猜一次显示方式。

目标是把这些分散决策收束为一个公开投影协议和一个前端 reducer。

## 2. 设计原则

### 2.1 三类输出不可混权

| 类别 | 权威来源 | 允许显示位置 | 禁止行为 |
| --- | --- | --- | --- |
| 模型返回 | model | assistant 正文、模型判断类 timeline | 系统伪造、由 runtime 状态替代 |
| 工具输出 | tool / runtime executor | tool window、observation、artifact、timeline | 直接写入 assistant 正文 |
| 系统状态 | system / runtime | status bar、monitor、lifecycle timeline | 代替模型回答、生成“我会/我已”等正文 |

判断标准：任何出现在聊天正文的文字必须来自模型最终输出；系统生成的文字只能进入状态、timeline、工具观察或诊断。

### 2.2 系统不阻碍 agent

系统只负责提供环境、执行动作、记录观察、投影状态和边缘控制。系统不能把权限、状态、handoff、monitor 结果包装成“请重新提出请求”或“任务已经由我接管”的语义答复。需要语义判断时必须回到模型。

### 2.3 前端只渲染投影，不重新裁决

前端可以做布局、折叠、去重和安全过滤，但不能重新判断一个 event 是模型正文、工具结果还是系统状态。这个决定由后端统一投影层给出。

## 3. 统一协议

新增公开投影信封：`PublicProjectionEnvelope`。

```ts
type PublicProjectionEnvelope = {
  authority: "harness.public_projection.v1";
  projection_id: string;
  sequence: number;
  created_at: number;
  session_id: string;
  anchor: PublicProjectionAnchor;
  lifecycle: "running" | "waiting" | "done" | "error" | "stopped";
  source_authority: "model" | "tool" | "system" | "runtime";
  surface: "assistant_body" | "timeline" | "tool_window" | "status_bar" | "task_projection" | "control" | "diagnostics";
  terminal?: {
    event: "done" | "error" | "stopped";
    visible: boolean;
    reason?: string;
  };
  items: PublicProjectionItem[];
  task_projection?: SingleAgentTaskProjection;
  active_turn_update?: {
    turn_id: string;
    task_run_id?: string;
    state?: string;
  };
};

type PublicProjectionAnchor = {
  turn_id?: string;
  message_id?: string;
  task_run_id?: string;
  run_id?: string;
  turn_run_id?: string;
  anchor_role: "assistant" | "status" | "monitor";
};

type PublicProjectionItem = {
  item_id: string;
  slot: "body" | "timeline" | "tool" | "status" | "task" | "control";
  kind: string;
  title?: string;
  text?: string;
  detail?: string;
  state?: "running" | "waiting" | "done" | "error" | "stopped" | string;
  tool_window?: unknown;
  artifacts?: Array<Record<string, unknown>>;
  trace_refs?: string[];
};
```

### 3.1.1 任务投影活动分级

`SingleAgentTaskProjection.activities` 不是原始工具日志。它是面向聊天区的公开投影，必须在后端生成时完成分级：

| 字段 | 允许值 | 含义 |
| --- | --- | --- |
| `visibility_level` | `primary` | 用户应看到的关键动作，例如写入文件、生成产物、重要命令、收口结果 |
| `visibility_level` | `secondary` | 可以辅助理解任务推进的动作，例如非内部工具的有意义观察 |
| `visibility_level` | `debug` | 仅用于监控/诊断的低层动作，例如读取文件、搜索引用、路径检查 |
| `visibility_level` | `internal` | 系统内部控制动作，例如 todo 工具、subagent 生命周期工具、通用失败恢复、协议修复 |
| `display_surface` | `tool_window` | 作为工具窗口展示 |
| `display_surface` | `timeline` | 作为任务进展展示 |
| `display_surface` | `diagnostics` | 不进入聊天投屏，只能在监控/trace 中查看 |

聊天投屏只渲染 `primary` / `secondary`，并且只接受 `tool_window` / `timeline`。`debug` / `internal` 不进入聊天区。这样工具仍然可以显示，但不会把 `read_file`、`search_text`、`list_subagents`、`agent_todo`、通用“工具调用失败，正在调整路径”这类内部执行细节和真实用户进展放在同一级。

这条规则只约束公开投屏，不约束模型上下文。工具成功、失败、拒绝、超时都必须作为 tool observation / observation refs 保留给执行循环，并进入后续模型轮次；系统不能因为用户界面隐藏或折叠某条工具窗口，就把失败事实从模型观察中删除。

`agent_todo` 是任务状态账本，不是聊天正文。主聊天投影默认不展示 todo block；todo 仍可由系统初始化、由 agent 更新、由监控台或任务详情读取。聊天区只显示模型正文、必要工具窗口、等待用户动作、产物和收口结果，避免把内部步骤清单当作用户主要反馈。

### 3.1 输出规则

| 输入事实 | Envelope 输出 | 说明 |
| --- | --- | --- |
| 模型最终回答 | `source_authority=model`, `surface=assistant_body`, `slot=body` | 唯一可写正文的常规路径 |
| 模型 ask_user / block | `source_authority=model`, `surface=assistant_body/control` | 仍由模型负责语义表达 |
| 工具调用开始 | `source_authority=tool`, `surface=tool_window`, `slot=tool` | 展示动作和目标，不写正文 |
| 工具观察返回 | `source_authority=tool`, `surface=tool_window/timeline`, `slot=tool` | 结构化观察，不冒充模型 |
| runtime 状态 | `source_authority=system/runtime`, `surface=status_bar/timeline`, `slot=status` | 只展示状态 |
| task todo / activity | `surface=task_projection`, `slot=task` | 渲染任务投屏，不生成正文 |
| handoff done | `terminal.visible=false`, `active_turn_update` | 关闭当前流并绑定任务，不产生可见文案 |
| 后台任务完成 | 优先 `task_projection.final_answer` 或模型 final body | 如果是模型产出的最终总结，可进正文；否则进 task projection |

### 3.2 禁止规则

1. `task_control`、`runtime_control`、`active_work_control` 不允许直接进入 `assistant_body`。
2. `done(task_executor_scheduled)` 不允许生成 status item、receipt 或 assistant content。
3. 前端不得根据 `answer_channel` 自行生成“后台任务已接管”“我会开始处理”等文案。
4. 工具原始 stdout / JSON diagnostics 不允许直接进入正文。
5. Runtime event 缺少 anchor 时，不允许静默丢失；必须进入 pending projection 或 monitor-only projection。

## 4. 投影时序

### 4.1 普通模型回复

```text
用户输入
-> backend 创建 turn / stream run
-> 模型输出 assistant_text_delta
-> PublicProjectionEnvelope(surface=assistant_body, source=model)
-> frontend reducePublicProjectionEnvelope 写入 assistant 正文
-> done(visible=true, source=model)
-> frontend 结束 streaming，不额外生成 status 文案
```

关键点：正文流和状态流分离。`done` 只负责终止，不补写“完成”正文。

### 4.2 工具调用与观察

```text
模型提交 tool_call
-> runtime 记录 model_action_admission
-> envelope(surface=tool_window, source=tool/runtime, state=running)
-> 工具执行
-> tool_observation_recorded
-> envelope(surface=tool_window/timeline, source=tool, state=done/error)
-> observation 回到模型
-> 模型继续输出正文或下一步动作
```

关键点：工具输出永远先成为 observation/timeline，不能替模型回答。

### 4.3 主会话发起后台任务

```text
模型判断需要持续任务
-> request_task_run
-> task_run_lifecycle_started
-> envelope(surface=task_projection, source=runtime, state=running)
-> agent_todo_initialized
-> envelope(task_projection.todo)
-> agent_turn_terminal(status=task_executor_scheduled)
-> envelope(terminal.visible=false, active_turn_update={task_run_id,state=waiting_executor})
-> 当前 chat stream 关闭
-> runtime monitor SSE 继续推送 task_run 事件
-> envelope(surface=task_projection/tool_window/timeline)
-> frontend 锚定到原 assistant message，持续更新任务投屏
```

关键点：handoff 不是一条可见消息。可见内容来自 task projection 和后续 task events。

### 4.4 后台任务运行中投屏

```text
task_executor_started
-> envelope(task_projection.phase=executing)
model_action_request_received
-> envelope(timeline: 模型正在判断 / 当前动作)
task_tool_observation_recorded
-> envelope(tool_window: 工具结果)
step_summary_recorded
-> envelope(timeline: 工作单元进展)
agent_todo_updated
-> envelope(task_projection.todo)
task_run_lifecycle_finished
-> envelope(task_projection.final_answer 或 final status)
```

关键点：任务投屏必须渲染 `current_action`、`activities`、`todo`、`artifact_refs`、`final_answer`，不能只渲染 todo。

### 4.5 用户对运行中任务 steer / pause / resume

```text
用户输入
-> 前端带 expected_active_turn_id / active_turn_input_policy
-> 模型判断是否控制当前工作
-> active_work_control 或普通回答
-> 系统执行边缘控制
-> envelope(surface=status_bar/timeline, source=system)
-> observation 回模型
-> 模型输出最终说明，或后台任务继续投屏
```

关键点：系统控制结果不是最终回答。模型最终说明才可进入正文。

### 4.6 图任务前台控制台

图任务运行时进入图任务前台控制台，主会话只显示必要的锚定摘要和结果引用：

```text
graph run started
-> foreground monitor projection
-> graph node events / edge contract events
-> foreground console projection
-> graph run finished
-> summary / artifact refs 可锚定回发起会话
```

主会话不应复制完整图任务控制台，只显示当前会话需要知道的进展和结果。

## 5. 前端投屏机制

### 5.1 单一 reducer

新增：

```text
frontend/src/lib/store/publicProjectionReducer.ts
```

职责：

1. 消费 `PublicProjectionEnvelope`。
2. 根据 `surface/slot/source_authority` 更新唯一的消息投影状态。
3. 维护 `activeTurnSnapshot`。
4. 将 task projection 锚定到 assistant message。
5. 对缺 anchor 的 projection 建立 pending 队列，等待后续 monitor/session snapshot 补齐。

`reduceStreamEvent` 和 `RunMonitorController.applyStreamPayload` 不再直接解释 runtime event，只做 legacy event -> envelope adapter。

### 5.2 消息结构

目标消息结构：

```ts
type MessagePublicProjection = {
  body?: string;
  timeline: PublicProjectionItem[];
  toolWindows: PublicProjectionItem[];
  taskProjection?: SingleAgentTaskProjection;
  terminal?: "done" | "error" | "stopped";
};
```

`runtimePublicTimelineDraft`、`runtimeProgress`、`runtimeAttachments` 可以在迁移期保留，但必须由 `MessagePublicProjection` 派生，不能再各自拥有展示决策。

### 5.3 渲染槽位

| Slot | 组件 | 数据来源 |
| --- | --- | --- |
| assistant body | `ChatMessage` 正文 | `surface=assistant_body` |
| task projection | `PublicTimelineActivity` | `surface=task_projection` |
| tool window | `PublicTimelineActivity` | `slot=tool` |
| status bar | `SessionActivityBar` | `surface=status_bar` |
| monitor console | RunMonitor / GraphTask console | monitor envelope |
| diagnostics | health / debug 面板 | `surface=diagnostics`，默认不进聊天 |

### 5.4 任务 projection 渲染

`PublicTimelineActivity` 必须渲染：

1. `task_projection.todo`
2. `task_projection.current_action`
3. `task_projection.activities`
4. `task_projection.artifact_refs`
5. `task_projection.final_answer`

渲染顺序：

```text
当前动作
-> 工具/观察活动
-> 处理清单
-> 产物引用
-> 最终结果
```

todo 不再是唯一主体，只是任务投屏的一部分。

## 6. 后端投影机制

### 6.1 新模块

建议新增：

```text
backend/harness/runtime/public_projection_envelope.py
backend/harness/runtime/public_projection_types.py
```

职责：

1. 从 runtime event / chat event / monitor event 生成统一 envelope。
2. 处理 source authority。
3. 处理 anchor。
4. 处理 task projection。
5. 处理可见性和去重 id。

### 6.2 接入点

| 文件 | 改法 |
| --- | --- |
| `backend/api/chat.py` | `_project_public_stream_event` 改为输出 envelope，不再直接调用 `project_public_timeline_delta` |
| `backend/harness/runtime/runtime_monitor_public_projection.py` | 改为复用 envelope projector |
| `backend/harness/runtime/public_timeline_stream.py` | 降级为 item builder，不再决定事件最终 surface |
| `backend/harness/runtime/session_task_projection.py` | 保留 task projection，但补齐 activities/current_action 的公开形态 |
| `backend/harness/runtime/session_timeline.py` | 持久化 envelope / derived public projection，保证刷新不丢 |

### 6.3 Anchor 规则

Anchor 解析优先级：

```text
public_anchor.anchor_turn_id
-> refs.turn_ref
-> payload.turn_id
-> model_action_request.turn_id
-> task_run.diagnostics.latest_interaction_turn_id
-> active_turn_registry(session_id)
-> session live monitor active_turn_snapshot
```

缺 anchor 时：

1. 仍然输出 envelope，但 `anchor_role="monitor"`。
2. 前端存入 pending projection。
3. 后续 session monitor / active turn snapshot 补齐 anchor 后合并。
4. 超时仍未补齐，只留在 RunMonitor，不进入聊天消息。

## 7. 去重与收口

去重 key：

```text
projection_id = hash(authority + event_id + surface + slot + anchor + item_id)
```

规则：

1. 同一 `projection_id` 幂等覆盖。
2. `task_projection` 覆盖同一 `task_run_id` 的旧 projection。
3. `assistant_body` 不和 `final_summary` 同时显示相同文本。
4. `terminal.visible=false` 只更新生命周期，不生成 UI item。
5. session refresh 后，持久 projection 替换 transient projection，不改变公开含义。

## 10. 实施阶段

### Phase 1：协议与 adapter

1. 新增 `PublicProjectionEnvelope` 类型。
2. 后端 chat stream 和 runtime monitor 都输出 envelope。
3. 前端新增 legacy event -> envelope adapter。
4. 不删除旧 UI，只保证新 envelope 能并行跑通。

### Phase 2：单一 reducer

1. 新增 `publicProjectionReducer`。
2. `reduceStreamEvent` 调用 reducer。
3. `RunMonitorController` 调用 reducer。
4. `hydrateLatestOrchestrationSnapshot` 返回 projection 时也走 reducer。

### Phase 3：任务投屏完整渲染

1. `PublicTimelineActivity` 支持 `current_action/activities/artifact_refs/final_answer`。
2. todo 从主体降级为任务投屏的一部分。
3. 状态栏只读 `status_bar` slot。

### Phase 4：删除旧权威

1. 删除前端本地 status/receipt 文案推断。
2. 删除 task_control 正文兜底。
3. 删除 runtime monitor attachment 的独立展示裁决。
4. 保留最薄 legacy adapter，带明确删除条件。

### Phase 5：固定端口实测与收口

1. 重启 3000 / 8003。
2. 用真实会话跑通普通回答、工具、后台任务、steer、刷新恢复。
3. 清理旧测试，保留行为合同测试。

## 11. 目标效果

最终 UI 时序应稳定为：

```text
用户消息
-> assistant 正文只显示模型说的话
-> timeline 显示工具、观察、任务活动
-> footer/status bar 显示系统运行状态
-> task projection 持续显示后台任务进展
-> done 只收口，不自造回复
```

这样才能符合成熟 agent 的边界：agent 负责思考和表达，系统负责环境、执行、观察、投影和边缘控制。

# 主会话反馈投影链路修复方案

日期：2026-06-10

关联原则文档：

- `docs/系统架构/099-主会话显示原则与公开投影契约-20260610.md`

## 一、问题结论

当前主会话投影链路的问题不是单点渲染缺陷，而是动作契约、prompt、runtime projection、持久化 timeline 和前端 fallback 没有围绕同一条显示原则收敛。

现状链路大致是：

```text
Prompt 不鼓励 tool_call 携带开局反馈
-> Protocol 不要求 tool_call 有公开反馈
-> Native tool call 转 ModelActionRequest 时 public_progress_note 为空
-> 工具后 followup 进入 JSON action 模式，无法自然流式输出阶段分析
-> 阶段分析只能依赖 public_progress_note/public_action_state
-> task_projection 存在时 session_timeline 跳过 public_timeline 重建
-> 刷新后只剩工具投影，看不到持久化反馈分析
-> 前端旧 fallback 偶尔补出“处理开始/完成/已发起工具请求”等泛化状态
```

用户侧表现：

1. 开局可能没有模型反馈，直接出现工具。
2. 工具后可能只有工具，没有观察后的模型分析。
3. live 期间看到的反馈刷新后消失。
4. task projection 和 public timeline 存在竞争关系。
5. 旧状态文案仍可能从 fallback 链路泄露到页面。

## 二、目标链路

修复后的目标链路：

```text
用户请求
-> 创建当前 turn 容器
-> 临时 UI 状态：正在思考，不持久化
-> ModelActionDecision
   -> respond：直接正文
   -> ask_user：正文请求补充
   -> block：正文说明阻塞
   -> tool_call：模型公开反馈 + 工具动作
-> 工具观察
-> 模型基于观察给出阶段反馈或继续工具
-> 最终正文收口
-> session_timeline 按同一 anchor 重建 public_timeline + task_projection
```

核心原则：

- 工具不是正文。
- 控制不是正文。
- 状态不是正文。
- 模型反馈必须有稳定数据通道。
- task_projection 不能替代 public_timeline。
- live 和 persisted 必须同构。

## 三、权威分层

### 1. Prompt 层

职责：要求模型在工具前、工具后提供用户可见反馈。

不允许：

- 把开发说明当 prompt。
- 要求 tool_call 不写公开反馈。
- 鼓励裸工具调用。

目标规则：

```text
如果你选择 tool_call，你必须同时给出一句简短的用户可见反馈：
- 说明你要确认什么或基于当前情况的判断；
- 不预测工具结果；
- 不写“开始处理/正在处理/处理完成”；
- 不输出 JSON/工具协议/内部控制字段。

工具返回后，如果你继续调用工具，你仍需要给出基于观察的阶段反馈。
```

### 2. 动作协议层

职责：定义 ModelActionRequest 的公开反馈字段和校验语义。

目标规则：

- `tool_call` 允许并鼓励 `public_progress_note`。
- `public_action_state.current_judgment` 承载模型判断。
- `public_action_state.next_action` 承载下一步公开动作说明。
- 对缺失反馈的 tool_call 不制造假正文。
- 对缺失反馈记录 contract gap，必要时仍允许执行工具，避免阻塞 agent。

建议协议：

```json
{
  "action_type": "tool_call",
  "public_progress_note": "我先检查投影链路里反馈从哪里丢失，再对照持久化和前端水合。",
  "public_action_state": {
    "visible_status": "waiting_for_tool",
    "current_judgment": "当前需要先确认持久化 timeline 是否被 task projection 覆盖。",
    "next_action": "读取 session_timeline 和前端水合入口。",
    "completion_status": "waiting_for_tool"
  },
  "tool_call": {
    "tool_name": "read_file",
    "args": {}
  }
}
```

### 3. Native Tool Call 适配层

职责：把模型原生 tool call 转为统一 ModelActionRequest。

当前问题：

- native tool call 路径将 `public_progress_note` 固定为空。

目标规则：

- 如果模型原生响应包含 assistant content，应将安全的 content 作为 tool_call 的公开反馈候选。
- 如果工具参数中存在明确 `public_progress_note`，使用该字段。
- 如果两者都没有，不合成假正文，只记录 `public_feedback_missing` diagnostics。
- 前端可显示临时“正在思考”或工具动作，但不能把系统补文案当模型反馈。

### 4. Runtime Projection 层

职责：把 action、tool、observation、feedback 投影为 public timeline items。

目标规则：

- `model_action_admission_checked` 中的 tool_call 应先投影模型 body item，再投影工具 item。
- 工具观察返回后必须投影观察摘要。
- 工具后继续 tool_call 时，必须投影观察后的 `observation_report` 或 `opening_judgment` 等模型反馈项。
- `assistant_text_delta/final` 仍不重复生成 public timeline，避免正文流和动作层重复。

推荐 item 顺序：

```text
opening_judgment / observation_report
-> work_action running
-> work_action completed/error with observation
-> observation_report
-> final_summary
```

显示粒度：

- 工具反馈必须简短，只表达单次工具观察事实。
- 阶段性总结承载本阶段分析、归因、决策变化和下一阶段方向。
- 多个工具连续执行时，不要求每个工具后都生成长分析；可以在工具组完成后生成一次阶段总结。
- 工具失败、方向改变、准备收口、发现关键证据时必须生成阶段总结。
- 阶段性总结必须同时说明本阶段结论和下一阶段方向；不要新增独立的“下一步反馈”类型。
- 阶段性总结出现后，前面已被总结覆盖的成功工具记录默认折叠。
- 折叠只影响展示，不删除 trace refs、artifact refs、observation refs。

阶段边界：

```text
开局判断
-> 问题定位 / 信息收集
-> 方案确认
-> 实施修改
-> 验证
-> 收口
```

阶段总结只在阶段边界、关键发现、失败恢复、方向改变、用户 steer 或准备收口时产生；不按工具次数机械生成。

推荐形态：

```text
工具反馈：读取到 session_timeline 在 task_projection 存在时跳过 public_timeline。
阶段总结：这说明刷新后反馈丢失发生在持久化重建层，而不是前端单纯渲染问题；下一阶段先修重建层，让 task_projection 与 public_timeline 共存，并补刷新水合测试。
```

折叠策略：

```text
opening_judgment
-> tool A running
-> tool A completed  // 短反馈
-> tool B completed  // 短反馈
-> observation_report / stage_summary
   -> tool A/tool B 默认折叠为证据记录
-> final_summary
   -> 历史成功工具继续折叠，失败/等待/关键产物保留摘要
```

### 5. Session Timeline 持久化重建层

职责：刷新后重建与 live 一致的公开投影。

当前问题：

```python
public_timeline = []
if not task_projection:
    public_timeline = project_public_timeline_from_events(...)
```

这导致存在 `task_projection` 时，模型反馈 timeline 被跳过。

目标规则：

- `task_projection` 和 `public_timeline` 必须共存。
- `task_projection` 负责结构化工具/任务状态。
- `public_timeline` 负责模型反馈、观察摘要、控制/错误公开状态。
- 合并时按 event time / offset / trace refs 去重。
- 不允许因为有 task_projection 就跳过 public timeline 重建。

目标结构：

```json
{
  "runtime_attachments": [
    {
      "task_projection": {},
      "public_timeline": [
        { "kind": "opening_judgment", "slot": "body", "source_authority": "model" },
        { "kind": "work_action", "slot": "tool", "source_authority": "tool" },
        { "kind": "observation_report", "slot": "body", "source_authority": "model" }
      ]
    }
  ]
}
```

### 6. 前端水合与渲染层

职责：按 anchor 把 public timeline 和 task projection 归到正确 assistant turn。

目标规则：

- 优先使用 `anchor_message_id`、`sourceTurnId`、`sourceRunId/sourceTaskRunId/sourceTurnRunId`。
- 保留 task projection companion timeline body items。
- 模型 body timeline item 可以作为活动流中的反馈分析展示，不强行写入 message content。
- 最终正文仍由 assistant message content 承载。
- 清理或降级旧 fallback 文案。

必须清理的旧 fallback：

- `处理已开始`
- `处理完成`
- `已发起工具请求`
- `正在判断下一步动作`
- 无意义 `done/assistant_message` 状态

这些只能作为内部状态或临时 UI 状态，不能作为持久化公开反馈。

## 四、实施步骤

### 阶段 1：修 prompt 和动作 schema

文件：

- `backend/harness/runtime/compiler.py`
- 相关 prompt pack / prompt resource 文件，如有重复规则需同步清理。

动作：

1. 修改 `model_action_request_schema` 和 `task_execution_action_schema` 中关于 `tool_call` 的 `public_progress_note` 说明。
2. 明确 tool_call 前必须给出用户可见反馈。
3. 明确工具后继续 tool_call 时必须基于 observation 给出阶段反馈。
4. 删除“tool_call 时不要用它描述工具意图或第一人称前置话术”这类反向约束。
5. 保留禁止泛化状态词和禁止预测工具结果。

验收：

- prompt 中不再要求 tool_call 裸调用。
- prompt 不鼓励“开始处理/处理完成”。

### 阶段 2：修动作协议校验

文件：

- `backend/harness/loop/model_action_protocol.py`

动作：

1. 允许 tool_call 使用 `public_progress_note` 和 `public_action_state`。
2. 对 tool_call 缺少公开反馈记录 diagnostics。
3. 不因为缺失公开反馈直接阻断工具执行。
4. 对 respond/ask_user/block 继续保持强约束。

验收：

- tool_call 可以携带公开反馈。
- 裸 tool_call 会留下可测试的 contract gap。
- agent 不会因为缺少反馈被硬阻塞。

### 阶段 3：修 native tool call 适配

文件：

- `backend/harness/loop/single_agent_turn.py`

动作：

1. `_tool_action_request_from_native_tool_calls` 不再固定 `public_progress_note=""`。
2. 从模型响应 content 或 tool args 中提取公开反馈候选。
3. 候选必须经过 public text 清洗。
4. 没有候选时记录 diagnostics，不合成假正文。
5. 保持工具照常执行。

验收：

- native tool call 可以有开局反馈。
- 没有反馈时不会污染正文。

### 阶段 4：修 runtime public projection

文件：

- `backend/harness/runtime/public_timeline_stream.py`
- `backend/harness/runtime/public_projection_envelope.py`
- `backend/harness/runtime/progress_presenter.py`

动作：

1. `model_action_admission_checked/tool_call` 有反馈时先生成模型 body item。
2. tool item 仍生成在反馈后。
3. 工具后继续 tool_call 时，`current_judgment` 投影为 `observation_report`。
4. `progress_presenter` 不再因为 `action_type == "tool_call"` 忽略全部 public action state。
5. 防止工具状态被误写成模型反馈。
6. 将单次工具观察压缩为短反馈，将跨工具判断投影为阶段性总结。

验收：

- 工具前有 `opening_judgment`。
- 工具后有 `observation_report`。
- 工具 item 和反馈 item 顺序正确。

### 阶段 5：修 session timeline 持久化重建

文件：

- `backend/harness/runtime/session_timeline.py`

动作：

1. 删除 `if not task_projection` 对 public timeline 重建的阻断。
2. 始终从 runtime events 重建 public timeline。
3. 当 task_projection 存在时，只过滤与 task_projection 完全重复的低价值 tool item，不过滤模型 body feedback。
4. 保证 `public_timeline` 与 `task_projection` 同时出现在 attachment。
5. 清除持久化重建中的“处理已开始/处理完成”等低价值 stage/terminal 文案。

验收：

- 刷新后仍能看到开局反馈、工具观察、阶段分析。
- task_projection 不再吃掉 public_timeline。

### 阶段 6：清理前端旧 fallback

文件：

- `frontend/src/lib/runtimeVisibilityProjection.ts`
- `frontend/src/lib/store/events.ts`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/components/chat/ChatMessage.tsx`
- `frontend/src/components/chat/PublicTimelineActivity.tsx`

动作：

1. public projection envelope 存在时，禁止旧 fallback 再补状态文案。
2. 对没有 envelope 的旧事件，只允许短暂 UI 状态，不持久化。
3. 清理“处理开始/处理完成/已发起工具请求/正在判断下一步动作”等文案进入持久化 activity 的路径。
4. 确保 body timeline item 在 task_projection companion 模式下保留。
5. 阶段性总结或最终回答出现后，将已覆盖的成功工具记录默认折叠。
6. 保持 anchor-first 归属，不恢复旧的 next-assistant fallback。

验收：

- 不出现控制/空状态正文。
- 不出现上一轮工具挂到下一轮。
- persisted 和 live 显示一致。
- 阶段总结出现后，历史工具记录折叠但仍可展开追溯。

## 六、边界处理

### 没有开局反馈

允许工具执行，但页面只显示临时 UI 状态或工具动作，不合成模型正文。

同时记录：

```text
public_feedback_missing
```

用于测试和后续 prompt 修复。

### 工具返回为空

显示工具完成状态，但不伪造观察结论。  
模型下一步应说明“没有拿到有效结果”或换验证方式。

### 工具失败

工具失败不是控制泄露。  
应显示用户可理解失败摘要，隐藏内部栈、协议字段和 sandbox 内部细节。

### 最终回答出现后

最终正文不应吞掉阶段反馈。  
工具和观察可以折叠，但模型阶段分析应保留。

### 刷新重建

持久化重建必须以 event log 为事实来源，以 anchor 为归属依据。  
不允许用“最近 assistant”或“下一个 assistant”推断归属。

## 七、实施风险

1. 修改 prompt 后模型输出结构可能变化，需要同步协议测试。
2. native tool call 的公开反馈来源需要谨慎，不能把工具 JSON 或内部协议当反馈。
3. session timeline 始终重建 public_timeline 后，可能出现重复 tool item，需要用 trace refs 去重。
4. 清理旧 fallback 可能暴露后端缺失 envelope 的旧事件，需要测试覆盖。

## 八、完成定义

修复完成必须满足：

1. 开局不再空白直接出工具。
2. tool_call 有模型公开反馈通道。
3. 工具后有观察摘要和模型阶段分析。
4. live 与刷新后显示一致。
5. task_projection 与 public_timeline 共存。
6. 控制命令、JSON、工具协议不进入正文。
7. `开始处理/处理完成` 等低价值状态不作为公开反馈出现。
8. 所有投影都按明确 anchor 归属。

## 九、任务设计补充要求

现有 TaskRun 设计已经具备控制和反馈的基础能力：

- `TaskRun.status`、`terminal_reason`、`runtime_control` 可以表达运行/暂停/停止/恢复。
- `ExecutorControlSignal` 可以把 pause/stop/replan 传给执行器。
- `ActiveTaskSteer` 可以记录用户 steer，并跟踪 pending/included/consumed 状态。
- `step_summary_recorded` 可以承载 `public_progress_note`、`current_judgment`、`next_action`。
- `task_projection` 可以把任务状态、工具活动、todo 和控制能力投影给前端。

但它还不完整支持“阶段性总结 + 工具证据折叠”的成熟显示链路。当前 `phase` 主要是从事件和状态推导出来的执行态，例如 `executing`、`tool_waiting`、`scheduled`、`handoff`、`blocked`、`completed`，不是用户可理解的任务阶段。

目标任务阶段应成为一等运行语义：

```text
opening_judgment
-> investigation
-> plan_confirmation
-> implementation
-> verification
-> closeout
```

需要补充：

1. 增加权威阶段字段或事件，例如 `task_phase_transitioned`。
2. 阶段事件应包含 `phase_id`、`phase_label`、`phase_goal`、`phase_summary`、`next_phase_id`、`covered_tool_refs`。
3. `WorkRolloutItemType` 应增加 `stage_summary` 或 `phase_boundary`，不要把阶段总结混在普通 progress 里。
4. `step_summary_recorded` 继续记录工具级短反馈，但阶段总结由阶段事件或 rollout stage item 承载。
5. `public_timeline` 将阶段事件投影为 `stage_summary` body item。
6. 阶段总结出现后，前端根据 `covered_tool_refs` 或时间窗口折叠已覆盖的成功工具记录。
7. pause/stop/replan/steer 触发阶段边界时，也应产生对应阶段总结或阶段状态，而不是只写控制状态。

结论：


# 主会话显示原则与公开投影契约

日期：2026-06-10

## 目标

主会话页面必须让用户清楚看到 agent 正在如何推进任务，同时严格阻断控制系统、工具协议、JSON 动作命令和内部状态进入正文。

页面显示的核心目标不是“把所有事件都展示出来”，而是建立一条稳定的用户可理解链路：

```text
用户请求
-> agent 开局反馈或直接回答
-> 工具动作
-> 工具观察
-> agent 基于观察继续反馈
-> 收口回答
-> 刷新后仍按同一顺序重建
```

## 总原则

1. 正文只属于模型对用户的自然语言反馈。
2. 工具调用、等待、观察、恢复、暂停、错误属于动作层，不属于正文。
3. 控制命令、工具协议、JSON action、系统状态、内部 terminal reason 绝不允许进入正文或用户可复制的助手内容。
4. `开始处理`、`处理完成`、`assistant_message`、`done` 这类控制或空状态不应显示为正文，也不应作为有价值反馈持久化。
5. 页面可以有临时状态，例如“正在思考”，但它是 UI chrome，不是模型正文，不写入 assistant message。
6. 用户刷新页面后，应能看到同一轮中有价值的公开链路：开局反馈、工具动作、观察反馈、阶段分析、最终回答。
7. 不允许把上一轮动作、工具或观察挂到下一轮回答上。

## 页面分层

### 正文层

正文层显示 assistant message body，只允许以下内容进入：

- 模型直接回答用户的问题。
- 模型在工具前给出的开局反馈。
- 模型在工具观察后给出的阶段分析。
- 模型最终收口回答。
- 需要用户补充信息时的自然语言提问。

正文层禁止显示：

- `tool_call`、`tool_calls`、`action_type`、`model_action_request`。
- `runtime_control`、`task_control`、`active_work_control` 等控制命令。
- `开始处理`、`处理完成`、`done`、`assistant_message` 等运行状态。
- 原始异常栈、内部 validator 输出、内部权限判断细节。
- 任何未被公开投影契约允许的 debug 内容。

### 动作层

动作层显示任务运行事实，位于同一 assistant turn 下方或旁侧。

动作层允许显示：

- 正在读取文件、搜索、运行测试、调用子 agent 等工具动作。
- 工具等待状态。
- 工具返回后的观察摘要。
- 恢复、重试、暂停、取消、失败、阻塞。
- 与当前轮次绑定的 runtime timeline。

动作层禁止显示：

- 原始工具协议 JSON。
- 控制命令原文。
- 未清洗的 stdout/stderr 大段日志。
- 与当前 turn/run/task_run 不匹配的历史动作。

## 开局显示规则

用户提交消息后，页面应立即创建当前轮次容器。

如果模型还没有输出任何公开内容，页面可以显示临时状态：

```text
正在思考
```

该状态只表示 UI 等待模型决策，不属于正文，不持久化。

模型首个决策有四种合法路径：

```text
respond   -> 直接输出正文
ask_user  -> 正文中向用户请求补充信息
tool_call -> 先显示模型的开局反馈，再显示工具动作
block     -> 正文说明无法继续的原因
```

当首个决策是 `tool_call` 时，动作契约必须携带用户可见反馈，例如：

```text
我先检查投影链路里反馈从哪里丢失，再对照持久化和前端水合。
```

这条反馈来自模型动作决策中的 `public_progress_note` / `agent_feedback`，不是系统硬塞的“开始处理”。

若首个工具动作没有任何用户可见反馈：

- 前端仍可显示临时“正在思考”或工具动作。
- 后端应记录动作契约缺口。
- 不允许生成假的 assistant 正文来补洞。
- prompt 和动作协议应修复为要求模型在工具前给出简短公开反馈。

## 工具调用显示时序

工具调用的用户可见顺序必须稳定：

```text
模型开局反馈
-> 工具动作开始
-> 工具等待
-> 工具观察返回
-> 模型基于观察继续反馈
```

不允许出现：

```text
工具动作
-> 恢复/反馈
```

除非恢复/反馈本身就是更早事件的持久化重建，但即便如此也必须按事件时间和 anchor 归属正确排序。

工具观察返回后，页面必须显示用户可理解的观察摘要，而不是只有工具名称。

例如：

```text
观察反馈：前端水合时只拿到了 task_projection，没有拿到 public_timeline。
```

不应显示：

```text
executor_observation_received
```

或原始 JSON。

工具反馈必须短。工具反馈只说明这次工具“做了什么、看到什么、是否失败”，不承载长篇分析。

推荐长度：

- 单个工具反馈控制在 1 句。
- 只保留关键目标、结果、失败原因或产物引用。
- 长日志、完整命令输出、文件列表、协议字段进入工具窗口折叠区或 debug trace，不进入主活动流。

阶段性总结负责承载分析。每完成一个有意义的工具组、观察到关键事实、发生失败恢复、准备改变方向或准备收口时，agent 应给出简短阶段总结：

```text
阶段总结：问题不在 SSE 字节偏移，而在后端没有把工具后的公开反馈写入可重建 timeline。
```

阶段总结应以任务阶段为单位，而不是以工具次数为单位。工具调用只是阶段内部的证据采集动作，用户真正需要理解的是任务处于哪个阶段、这个阶段得出了什么结论、下一阶段要做什么。

推荐任务阶段：

```text
开局判断
-> 问题定位 / 信息收集
-> 方案确认
-> 实施修改
-> 验证
-> 收口
```

并不是每次工具返回都要阶段总结。只有发生阶段边界时才需要总结：

- 从开局判断进入问题定位。
- 从问题定位进入实施修改。
- 从实施修改进入验证。
- 从验证进入收口。
- 发现关键阻塞，需要改变方向。
- 用户 steer 改变了当前阶段目标。

阶段性总结必须同时承担下一阶段交代。用户需要知道 agent 对上一阶段的判断，以及下一阶段为什么继续、继续做什么、是否准备收口。不要再引入独立的“下一步反馈”类型。

推荐格式：

```text
阶段总结：反馈丢失发生在持久化重建层。
下一阶段我会修 session_timeline，让 task_projection 与 public_timeline 共存，并补回刷新后的验证用例。
```

阶段总结与工具反馈的区别：

```text
工具反馈：读到 session_timeline 在存在 task_projection 时跳过 public_timeline。
阶段总结：这会导致 live 反馈刷新后丢失，修复点应放在持久化重建而不是前端补显示；我会先修重建层，再用刷新水合测试验证。
```

当阶段性总结出现后，前面已经被总结覆盖的工具使用记录应默认折叠。工具记录仍保留为证据和可展开细节，但主视图优先让用户看到阶段总结、当前动作和最终回答。

折叠规则：

- 已被阶段总结覆盖的成功工具记录默认折叠。
- 失败工具、等待中工具、需要用户确认的工具保持展开。
- 产物生成、文件写入、测试失败等关键证据可以保留一行摘要。
- 最终回答出现后，历史工具记录继续折叠，除非用户展开查看。
- 折叠不得删除 trace refs、artifact refs、observation refs。

## 子 Agent 显示规则

调用子 agent 时，页面应显示为一个可等待的动作，而不是普通工具一闪而过。

标准时序：

```text
模型说明为什么需要子 agent
-> 子 agent 已启动
-> 等待子 agent 返回
-> 子 agent 观察摘要
-> 主 agent 基于观察继续反馈
```

如果子 agent 运行较久，动作层应保持 pending/running 状态，不能提前显示成 completed。

## 暂停、停止与 steer

暂停和停止属于控制动作，不进入正文。

暂停规则：

- 用户点暂停后，UI 应立即显示暂停请求已提交。
- runtime 可以在安全边界处停止工具或停止下一轮模型调用。
- 若当前模型正在输出正文，是否立即截断取决于运行策略，但必须显示清楚状态：正在暂停、已暂停或无法暂停当前原子步骤。

steer 规则：

- 用户追加指令时，应绑定到当前 active turn/run。
- 页面显示追加指令已进入当前任务。
- 后续工具、观察、反馈仍归属于原 active turn，不能泄露到下一轮 assistant message。

## 持久化规则

刷新后仍应恢复以下内容：

- 最终 assistant 正文。
- 模型开局反馈。
- 工具动作摘要。
- 工具观察摘要。
- 工具后阶段反馈。
- 错误、阻塞、暂停、恢复等用户需要理解的公开状态。

不持久化为正文的内容：

- `正在思考`。
- 空状态。
- 控制 terminal reason。
- debug-only 原文。
- 工具协议 JSON。

阶段反馈如果不是最终正文，必须通过 public timeline 持久化，而不是只靠 SSE live frame。

## Anchor 归属规则

所有公开投影必须绑定明确来源：

```text
sourceTurnId
sourceRunId
sourceTaskRunId
sourceTurnRunId
anchor_message_id
```

绑定优先级：

```text
anchor_message_id exact match
-> sourceTurnId exact match
-> sourceRunId/sourceTaskRunId/sourceTurnRunId exact match
```

禁止使用：

- “最近一个 assistant message”。
- “anchor turn 后面的下一个 assistant”。
- 根据 `turn_id` 数字尾巴猜测 source index。
- 当前 active turn 反推旧投影归属。

## Prompt 配套原则

写给 agent 的 prompt 必须要求它理解：

- 它可以直接回答、请求补充、调用工具或阻塞。
- 调用工具前要给用户一句简短、真实、具体的公开反馈。
- 工具返回后要基于观察给出阶段分析，而不是连续裸调用工具。
- 不得把控制命令、JSON 动作、工具协议当作正文输出。
- 无法继续时要给用户可理解原因，而不是泄露系统状态。

不应写成：

```text
根据 runtime 节点执行工具。
```

应写成：

```text
如果你需要调用工具，请先用一句话告诉用户你要确认什么、为什么需要确认。
工具返回后，你需要基于观察给出简短判断，再决定继续调用工具、请求补充信息或收口回答。
你的正文只能面向用户解释任务进展和结论，不能输出内部控制命令、工具协议或 JSON action。
```

## 验收标准

一个成熟稳定的投影系统必须满足：

1. 开局不是空白直接出工具。
2. 工具前有模型公开反馈，或至少有非正文临时 UI 状态。
3. 工具调用、等待、观察、反馈顺序稳定。
4. 工具观察后必须有可见观察摘要。
5. 最终回答不会吞掉阶段反馈。
6. 刷新后公开 timeline 与 live 期间一致。
7. 控制命令绝不进入正文。
8. JSON/工具协议绝不进入正文。
9. 上一轮工具不会泄露到下一轮回答。
10. 子 agent 调用有等待态和返回态。
11. pause/stop/steer 有明确归属，不制造边界模糊。
12. 所有显示内容都能追溯到明确 turn/run/task_run anchor。
13. 工具反馈保持简短，阶段性总结承载判断、归因和下一步方向。
14. 阶段性总结或最终回答出现后，已覆盖的历史工具记录默认折叠但可展开追溯。

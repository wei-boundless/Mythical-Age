# 流式输出权威链路升级技术书

日期：2026-06-09  
状态：计划书，待审阅后实施  
范围：模型流式 delta、ChatRun SSE、前端 canonical/display 双状态、Markdown-aware 展示调度  
目标：把当前“provider delta 直接驱动 UI”的链路升级为“后端语义化流帧 + 前端显示调度”的成熟 agent 流式输出体系。

---

## 1. 目的

当前已经完成一版低风险前端显示优化：只在当前 live assistant 消息上用前端 hook 做自然化节奏，不改变后端事件、SSE 重放和最终答案持久化。

这份技术书定义下一阶段更成熟的架构升级。它要解决的不是“打字速度慢一点”这种表层问题，而是明确流式输出中每一层的权威：

1. provider 原始 delta 只是一种观察，不是用户可见文本边界。
2. 后端必须把 provider delta 归一化为语义化 public stream frame。
3. OutputBoundary 仍然是最终 assistant 正文的 canonical 权威。
4. ChatRun 只负责传输、事件日志和重放，不决定语义完成。
5. 前端 store 保存 canonical 内容，显示层保存 display 内容，二者不能混用。

正确完成后，用户看到的流式输出应满足：

- 中文、英文、代码块、表格、路径和 URL 的出现节奏自然。
- 重连、刷新、切会话后不会重复、缺字或顺序错乱。
- `done` 到达后前端显示与后端 final 内容可校验一致。
- 工具进展不会被当作 assistant 正文流出来。
- 后续可扩展到不同 provider，而不依赖各 provider 的随机 delta 粒度。

## 2. 当前问题与设计缺口

### 2.1 当前链路

当前主链路如下：

```text
provider streaming chunk
-> backend/runtime/model_gateway/model_response.py content_delta
或 backend/harness/loop/single_agent_turn.py content_delta
-> backend/api/chat.py ChatRun event log
-> frontend/src/lib/api.ts consumeChatRunStream()
-> frontend/src/lib/store/events.ts reduceStreamEvent()
-> frontend/src/components/chat/ChatMessage.tsx ReactMarkdown
```

关键代码事实：

1. `backend/runtime/model_gateway/model_response.py` 在流式调用中读取 `_chunk_text(chunk)`，然后直接 yield `content_delta`，字段包括 `content`、`delta_index`、`delta_chars`、`accumulated_chars` 和 `stream_ref`。
2. `backend/harness/loop/single_agent_turn.py` 在 single agent turn 中也会直接 yield `content_delta`，并只通过 `_public_stream_delta_allowed(raw_content)` 做协议泄漏过滤。
3. `backend/api/chat.py` 把 runtime 事件投影为 public event，写入 ChatRun event log，并通过 `/chat/runs/{stream_run_id}/events` SSE 重放和订阅。
4. `frontend/src/lib/api.ts` 的 `consumeChatRunStream()` 负责 SSE 分块、offset 去重、重连和 terminal 后停止读取。
5. `frontend/src/lib/store/events.ts` 的 `reduceStreamEvent()` 目前对 `token` 和 `content_delta` 直接追加 `data.content` 到 assistant message。
6. `frontend/src/components/chat/ChatMessage.tsx` 最终用 `ReactMarkdown` 渲染 assistant content。频繁小 delta 会导致 Markdown 解析和布局不断重排。

### 2.2 深层问题

当前问题不是前端动画缺失，而是“流式事件的语义边界不明确”：

| 层级 | 当前行为 | 问题 |
|---|---|---|
| Provider | 按 provider 自己的 token/chunk 边界返回 | delta 边界随机，不适合作为 UI 节奏 |
| Model gateway | 直接公开 `content_delta` | 未归一化文本类型、offset、校验和、Markdown 状态 |
| Chat API | 负责 event log 和 SSE replay | 传输层被迫承载过多 public 事件形态 |
| Frontend API | 负责 SSE 解析和重连 | 只知道 event offset，不知道文本 offset |
| Frontend store | 直接追加 delta | canonical 内容和 display 内容混在一个 `message.content` 中 |
| ChatMessage | 每次 delta 重渲染 Markdown | 代码块、表格和长段落容易跳动 |

### 2.3 正确系统属性

目标系统必须具备以下属性：

1. **Provider-agnostic**：无论 provider delta 多碎，public frame 粒度稳定。
2. **Replay-safe**：SSE 重连后按 frame sequence 和 content offset 去重。
3. **Canonical-safe**：最终 answer 以 OutputBoundary final 为准，不由 display scheduler 决定。
4. **Markdown-aware**：流式展示理解 Markdown block 边界，不让 UI 抖动。
5. **Progress-separated**：任务进展走 `public_timeline_delta`，正文走 assistant text frame。
6. **Auditable**：每个 public frame 可以追溯到 turn_run、stream_ref、message_ref 和 source event。

## 3. 代码依据与本地设计原则

### 3.1 已有本地原则

`docs/系统规划/245-当前工作续接与单轮终止权威重构计划书-20260605.md` 已经确立多个约束，本方案继续沿用：

1. `TaskRun` 是长期工作的 durable truth。
2. `TurnRun` 是一次模型执行的唯一语义运行记录。
3. `ChatRun` 是传输层 stream run，不能决定语义运行是否 completed。
4. OutputBoundary 只能发布用户可见答案，不能把内部协议泄漏标为稳定答案。
5. 前端只消费后端投影，不重复推断当前工作是否可续接或如何表达执行进展。
6. `public_timeline_delta` 已经接入，不要回退到 raw event 解释。

本技术书把这些原则扩展到 assistant 正文流：

```text
TaskRun / TurnRun: 语义运行真相
OutputBoundary: 最终可见答案真相
ChatRun: 传输和重放真相
AssistantStreamFrame: 正文流式 public frame 真相
DisplayScheduler: 屏幕呈现节奏，不拥有语义真相
```

### 3.2 权威边界

| 模块 | 目标角色 | 禁止事项 |
|---|---|---|
| `model_response.py` | 收集 provider raw delta，交给 normalizer | 不直接把 provider delta 当 public UI delta |
| `single_agent_turn.py` | single-agent 模型调用与工具循环 | 不把工具协议 JSON 或 tool request 残片作为正文流 |
| `assistant_stream_normalizer.py` | 生成 typed assistant stream frame | 不决定 final answer 是否 canonical |
| `output_boundary` | 选择最终 canonical assistant answer | 不决定前端显示速度 |
| `backend/api/chat.py` | 事件日志、public projection、SSE replay | 不根据显示层需求改写语义状态 |
| `frontend/src/lib/api.ts` | SSE 传输、cursor、重连 | 不解释 frame 的用户语义 |
| `frontend/src/lib/store/events.ts` | canonical stream state 合并 | 不承担动画计时器 |
| `ChatMessage` / display hook | 屏幕显示调度 | 不写回历史、不覆盖 canonical 内容 |

## 4. 方案取舍

### 4.1 方案 A：只保留前端自然化 hook

优点：

- 已实施，风险低。
- 不改 SSE 和后端协议。
- 对用户视觉体验有明显改善。

缺点：

- provider delta 仍然直接进入 canonical `message.content`。
- 重连只能依赖 event offset，不能按文本 offset 做更细校验。
- Markdown block 状态只能在前端推断。
- 工具进展和正文分离仍依赖已有过滤逻辑。

结论：适合作为第一阶段，不适合作为最终架构。

### 4.2 方案 B：后端直接模拟打字机节奏

优点：

- 前端简单。
- 所有客户端看到相同节奏。

缺点：

- 把显示节奏塞进传输层，会拖慢 SSE 和最终答案到达。
- 断线重连会重放带延迟的显示事件，恢复体验变差。
- 后端变成 UI timing 权威，违反职责边界。

结论：拒绝。后端可以提供 display hint，但不能拥有显示计时器。

### 4.3 方案 C：typed stream frame + frontend display scheduler

优点：

- 后端负责语义化和可重放 frame。
- 前端负责显示速度和动画。
- canonical 内容和 display 内容分离。
- 支持 checksum、UTF-8 byte offset、重连去重和 final reconcile。
- 符合 mature coding agent 的结构：传输、语义、展示分别有权威。

缺点：

- 需要改 API contract、后端 stream normalizer、前端 store 和测试。
- 需要 shadow/cutover 迁移，不能一次硬替换。

结论：推荐采用。

## 5. 目标设计

### 5.1 固定目标链路

```text
ProviderRawDelta
-> ModelStreamNormalizer
-> AssistantStreamFrame
-> ChatRunPublicProjection
-> FrontendStreamCanonicalState
-> FrontendDisplayScheduler
-> ChatMessage
```

### 5.2 新 public frame 协议

新增公开事件类型：

```text
assistant_text_delta
assistant_text_final
assistant_stream_repair
```

保留现有 terminal：

```text
done
error
stopped
```

`assistant_text_delta` 草案：

```json
{
  "frame_schema_version": "assistant_stream_frame.v1",
  "event_type": "assistant_text_delta",
  "frame_id": "aframe:stream:12",
  "stream_ref": "modelreq:...",
  "message_ref": "turn:session:1:assistant",
  "turn_run_id": "turnrun:...",
  "task_run_id": "",
  "sequence": 12,
  "content": "我已经读取了代码，",
  "content_utf8_start": 72,
  "content_utf8_end": 99,
  "content_utf8_bytes": 27,
  "accumulated_utf8_bytes": 99,
  "accumulated_sha256": "sha256:...",
  "answer_channel": "conversation",
  "answer_source": "harness.single_agent_turn",
  "visibility": "public",
  "markdown_state": {
    "block_kind": "paragraph",
    "inside_code_fence": false,
    "line_start": false
  },
  "display_hint": {
    "chunk_kind": "phrase",
    "pause": "soft",
    "atomic": false
  }
}
```

`assistant_text_final` 草案：

```json
{
  "frame_schema_version": "assistant_stream_frame.v1",
  "event_type": "assistant_text_final",
  "stream_ref": "modelreq:...",
  "message_ref": "turn:session:1:assistant",
  "turn_run_id": "turnrun:...",
  "sequence": 48,
  "content": "完整最终答案",
  "content_utf8_bytes": 4096,
  "content_sha256": "sha256:...",
  "answer_channel": "conversation",
  "answer_source": "output_boundary",
  "answer_canonical_state": "stable_answer",
  "answer_persist_policy": "persist_canonical",
  "terminal_reason": "completed"
}
```

`assistant_stream_repair` 草案：

```json
{
  "frame_schema_version": "assistant_stream_frame.v1",
  "event_type": "assistant_stream_repair",
  "stream_ref": "modelreq:...",
  "message_ref": "turn:session:1:assistant",
  "repair_sequence": 49,
  "applies_after_sequence": 48,
  "reason": "checksum_mismatch",
  "expected_content_sha256": "sha256:...",
  "replacement_content": "完整最终答案",
  "replacement_content_sha256": "sha256:..."
}
```

协议硬约束：

1. replay 主键是 `(stream_ref, message_ref, sequence)`，不是字符数。重复 sequence 且 hash 一致时丢弃，重复 sequence 但 hash 不一致时进入 repair/error。
2. offset 统一使用 UTF-8 byte offset，Python 端用 `content.encode("utf-8")`，前端用 `TextEncoder`。不要把 Python `len(str)` 或 JS `string.length` 当成跨语言回放契约。
3. `accumulated_sha256` 和 `content_sha256` 均对 exact UTF-8 bytes 计算，不 trim、不做额外 Unicode normalization。
4. 字符数量只能作为 display hint 或诊断字段，不能用于去重、重放、repair 或 final reconcile。
5. `assistant_text_final` 的 `content_sha256` 必须与最终 canonical answer 完全一致；不一致时只能 repair，不能静默覆盖。

### 5.3 Frame 生成规则

`ModelStreamNormalizer` 负责把 provider raw delta 合并成更适合 public stream 的 frame：

1. 中文按短语和标点边界合并。
2. 英文按词组和标点边界合并。
3. Markdown 代码块按行输出。
4. Markdown 表格按行输出。
5. URL、Windows 路径、POSIX 路径作为 atomic frame。
6. 当 backlog 过大时增大 frame 粒度，避免前端永远追不上。
7. 如果 raw_content 起始像 JSON action、tool call 或内部协议，禁止生成 public assistant text frame。
8. normalizer 可以合并 provider chunk，但必须设置 flush 上限：默认 `max_flush_interval_ms=80`、`max_pending_utf8_bytes=256`、`max_pending_line_count=1`。这些上限只用于防止后端长时间憋帧，不是 UI 打字机计时器。
9. provider 结束、OutputBoundary final、error/stopped terminal 到达前必须同步 flush 已判定 public-safe 的 buffered frame；禁止为了显示节奏延迟 final。
10. 工具兼容、JSON action、内部协议可能出现的 turn，public assistant text gate 默认关闭。只有当前缀被判定为普通公开正文后才释放 frame；若 `safety_prefix_utf8_limit=512` 内仍无法证明安全，则本轮不公开 delta，只等待 OutputBoundary final 或 repair。

### 5.4 Frontend canonical/display 双状态

前端新增概念：

```ts
type AssistantTextCanonicalStreamState = {
  messageRef: string;
  streamRef: string;
  latestSequence: number;
  canonicalContent: string;
  canonicalContentSha256: string;
  accumulatedUtf8Bytes: number;
  finalReceived: boolean;
  terminal: boolean;
  repairState: "none" | "pending" | "applied" | "failed";
  displayHintsBySequence: Record<number, AssistantStreamDisplayHint>;
};

type AssistantTextDisplayRuntimeState = {
  messageRef: string;
  displayedContent: string;
  lastRenderedSequence: number;
  flushRequested: boolean;
};
```

其中：

- `canonicalContent` 由 `assistant_text_delta/final` 合并，参与复制、历史、最终 reconcile；它可以映射到 `Message.content`，但只能代表 canonical，不代表屏幕动画进度。
- `displayedContent` 只存在于 `ChatMessage` / display hook 的组件运行态，刷新、切会话、hydrate 后从 canonical 重新派生，不进入全局 store。
- 全局 store 可以保存 `displayHintsBySequence`，但不能保存 `displayContent`、`displayPending` 或计时器状态。
- `chatStreamDisplayEnabled` 只影响 UI 是否渐进展示，不影响 `assistant_text_delta/final` 的 canonical merge；关闭流式展示时前端仍可记录 canonical frame，但界面在 final 前不渐进展示，final 后一次性展示 canonical。
- `assistant_text_final` 到达后必须校验 hash，不一致则触发 `assistant_stream_repair` 或本地 replacement。
- `done` 到达时，如果 final frame 已到，display scheduler 快速 flush；如果 final frame 未到，说明后端违反顺序契约，前端进入 repair/error state，不把 `done.content` 当作新权威。

## 6. 数据模型与兼容策略

### 6.1 后端新增模型

建议新增：

- `backend/runtime/model_gateway/assistant_stream_frame.py`
- `backend/runtime/model_gateway/assistant_stream_normalizer.py`
- `backend/tests/assistant_stream_frame_regression.py`

核心对象：

```python
@dataclass(frozen=True)
class AssistantStreamFrame:
    frame_schema_version: str
    event_type: str
    frame_id: str
    stream_ref: str
    message_ref: str
    turn_run_id: str
    task_run_id: str
    sequence: int
    content: str
    content_utf8_start: int
    content_utf8_end: int
    content_utf8_bytes: int
    accumulated_utf8_bytes: int
    accumulated_sha256: str
    visibility: str
    markdown_state: dict[str, Any]
    display_hint: dict[str, Any]
```

实现要求：

- `AssistantStreamFrame` 必须是 typed object，不允许在 runtime 内用散装 dict 拼协议字段。
- normalizer 的输入输出要能脱离 provider runtime 做 fixture 测试。
- hash/offset 计算集中在一个模块，禁止 `model_response.py`、`single_agent_turn.py`、`api/chat.py` 各自重算。

### 6.2 旧事件降级规则

`content_delta` 不立即删除，分三步处理：

1. Shadow 阶段：后端同时生成 `content_delta` 和内部 `assistant_text_delta_shadow`，前端仍消费旧事件。
2. Public dual 阶段：后端公开 `assistant_text_delta`，前端优先消费新事件；旧 `content_delta` 只允许在本 stream 从未出现新 frame 时 fallback，并必须记录 `assistant_stream_legacy_content_delta_fallback_total`。
3. Cutover 阶段：前端不再消费旧 `content_delta` 作为正文；后端兼容开关默认 false，且必须有删除条件。

### 6.3 不保留的兼容

以下旧逻辑不应长期保留：

1. 前端根据 raw `content_delta` 直接追加 assistant 正文。
2. 后端把 provider chunk 原样公开为用户可见文本。
3. 用 `done.content` 覆盖已经出现的流式正文。
4. display scheduler 写回 session history。

### 6.4 存储与观测规则

存储规则：

1. session history 只保存最终 canonical assistant message，不保存 display 追赶进度。
2. ChatRun event log 保存 `assistant_text_delta/final/repair` stream frame，用于 SSE replay 和调试。
3. display scheduler 的 timer、pending、displayedContent 都是组件运行态，刷新后丢弃并由 canonicalContent 重新派生。
4. 如果发生 repair，event log 必须保留 repair frame；session history 最终只保留 repair 后 canonical final。

观测指标：

1. `assistant_stream_frame_count`：每个 stream 生成的 frame 数。
2. `assistant_stream_coalesce_latency_ms`：provider chunk 到 public frame flush 的耗时。
3. `assistant_stream_safety_gate_blocked_total`：因 JSON/tool/internal protocol gate 阻止公开 delta 的次数。
4. `assistant_stream_final_checksum_mismatch_total`：final 与 accumulated hash 不一致次数。
5. `assistant_stream_repair_total`：repair frame 发出次数。
6. `assistant_stream_legacy_content_delta_fallback_total`：dual public 阶段旧链路 fallback 次数，cutover 前必须归零。
7. `assistant_stream_replay_duplicate_drop_total`：前端或后端 replay 去重丢弃次数。

## 7. 固定执行流程

### Stage 1：Provider raw delta 归一化

输入：

- provider chunk
- stream policy
- accounting context
- allow_content_delta

输出：

- raw accumulated text
- internal AssistantStreamFrame candidate
- blocked reason when protocol leak is detected

禁止：

- 在此阶段决定 final answer canonical state。
- 在此阶段写 session history。

### Stage 2：OutputBoundary finalization

输入：

- raw accumulated text
- tool call parse result
- protocol leak result
- runtime directive context

输出：

- canonical assistant final
- answer metadata
- final content hash
- `assistant_text_final` 或明确的 no-public-answer reason

禁止：

- 根据前端显示需要改变 final text。
- 跳过 final frame，改由 `done.content` 承担 canonical final。

### Stage 3：ChatRun public projection

输入：

- AssistantStreamFrame
- runtime refs
- OutputBoundary final metadata

输出：

- public SSE event
- event_log offset
- replay-safe event id
- text-producing turn 的 `assistant_text_final` 已在 `done` 前进入 event log

禁止：

- 判断任务是否完成。
- 生成 UI timing timer。
- 在 `assistant_text_final` 缺失时先发 public `done` 关闭 SSE。

### Stage 4：Frontend canonical merge

输入：

- public SSE frame
- cursor offset
- local stream state

输出：

- canonicalContent
- latest sequence
- repair state

禁止：

- 启动动画计时器。
- 从 raw runtime event 推断正文。

### Stage 5：Frontend display scheduling

输入：

- canonicalContent
- display_hint
- user reduced motion preference

输出：

- displayedContent

禁止：

- 写回 history。
- 改写 answer metadata。

### 终止顺序硬约束

文本型 assistant turn 的公开事件顺序必须是：

```text
assistant_text_delta*
-> OutputBoundary canonical final
-> assistant_text_final
-> TurnRun terminal 已记录
-> public done/stopped/error
-> SSE close
```

说明：

1. `done` 是 transport terminal，不是 final answer 权威。
2. `assistant_text_final` 必须在 public `done` 前写入 ChatRun event log；重连客户端应先看到 final，再看到 done。
3. 如果本轮没有公开正文，例如工具 action 被接管或安全 gate 阻止，`done` 必须带明确 terminal reason；前端不能从 `done.content` 构造一个未经过 OutputBoundary 的正文。
4. `error/stopped` 可以没有 final，但必须保留已经发出的 frame，并标记 stream terminal reason，不能让 display scheduler 补写答案。

### Phase 1：新增 frame 模型和 normalizer 单元测试

### 10.4 Contract 与观测验收

Contract 验收：

1. 后端 fixture 生成后，前端测试必须直接消费同一批 JSON fixture。
2. API contract 测试必须断言 `assistant_text_delta/final/repair` 的 public allowlist 字段完整，且不泄漏 internal keys。
3. SSE replay 测试必须同时断言 event offset 和 frame sequence。
4. cutover 测试必须让旧 `content_delta` 到达前端，并断言 assistant 正文不变化。

观测验收：

1. Shadow 阶段：`assistant_stream_final_checksum_mismatch_total=0` 才允许进入 Dual public。
2. Dual public 阶段：`assistant_stream_legacy_content_delta_fallback_total=0` 连续通过回归和真实运行场景后才允许 Cutover。

## 11. 迁移与 cutover 规则

### 11.1 Shadow 模式

命名：`assistant_stream_frame_shadow`

规则：

- 后端生成 frame，但不作为前端主消费。
- 测试比较 raw accumulated text 与 frame accumulated hash。
- 若发现 mismatch，只修 normalizer，不改前端兜底。
- 记录 shadow frame 指标，不允许只把 shadow 写日志后无人消费。

### 11.2 Dual public 模式

命名：`assistant_stream_frame_dual_public`

规则：

- 后端同时发新 frame 和旧 `content_delta`。
- 前端优先消费新 frame。
- 旧 `content_delta` 仅当本 stream 从未出现新 frame 时消费，并记录 `assistant_stream_legacy_content_delta_fallback_total`。
- 所有新测试必须以新 frame 为断言对象。
- dual public 只能作为迁移窗口，不能新增依赖旧 `content_delta` 的业务逻辑或 UI 行为。

### 11.3 Cutover 模式

命名：`assistant_stream_frame_primary`

规则：

- 新 frame 是唯一 assistant 正文流主链路。
- 旧 `content_delta` 不再作为 assistant 正文 public event。
- 旧兼容开关默认 false。
- 前端 reducer 中旧 `content_delta` direct append 路径必须删除，或在测试中证明它只处理非正文/迁移诊断用途。
- 新增失败保护测试：cutover 模式下收到 `content_delta` 后，assistant `Message.content` 和 canonical stream state 都不变化。

允许进入 Cutover 的条件：

1. Shadow 阶段所有 fixture 和真实运行场景 hash mismatch 为 0。
2. Dual public 阶段 fallback 指标为 0。
3. final-before-done contract 测试通过。
4. 前端 replay fixture 覆盖重连、重复、乱序、repair。
5. `rg "content_delta"` 人工审查后，只剩 CLI、旧协议测试、迁移说明或非正文用途。

### 11.4 Rollback 触发

允许回滚到旧 `content_delta` 的条件：

1. 新 frame 在真实运行中出现缺字或重复。
2. final hash mismatch 频繁出现且无法 repair。
3. SSE reconnect 后 sequence 无法恢复。
4. 前端显示阻塞 canonical final。

Rollback 禁止事项：

- 不允许降低测试断言。
- 不允许把 provider raw delta 再包装成“新 frame”冒充通过。
- 不允许把工具协议泄漏标成 stable answer。
- 不允许无限期停留在 dual public。每次 rollback 必须生成缺陷记录，修复后重新从 Shadow 指标开始验收。

## 12. 禁止的捷径

1. 不允许在后端用 `sleep` 模拟打字机。
2. 不允许让 ChatRun 根据 display 状态决定 semantic completed。
3. 不允许让前端从 raw runtime event 猜 assistant 正文。
4. 不允许在 `done.content` 到达时无条件覆盖已合并正文。
5. 不允许把 `public_timeline_delta` 和 assistant 正文 frame 混为一个事件。
6. 不允许长期保留双主链路。
7. 不允许用跳过测试、降低断言或 mock 核心流式逻辑来制造通过。

## 13. 预期结果

升级完成后，流式输出会从“provider chunk 驱动 UI”变成“后端语义 frame 驱动 canonical，前端调度 display”的结构。

维护收益：

1. 更容易定位缺字、重复、乱序和 done 覆盖问题。
2. 不同 provider 的 streaming 行为差异被 normalizer 吸收。
3. 前端动画可以继续迭代，不影响后端语义。
4. OutputBoundary 的 canonical final 权威更清晰。
5. Runtime Monitor public timeline 和 assistant 正文流互不抢权。
6. 后续做暂停、恢复、replay、历史 hydrate 会更稳定。

## 14. 本轮不实施的内容

本技术书只定义升级方案，不在本轮直接修改后端 runtime、SSE contract 或 store 主链路。

下一步若开始实施，必须先确认：

1. 是否接受 `assistant_stream_frame.v1` 作为协议名。
2. 是否接受 `assistant_text_delta` / `assistant_text_final` / `assistant_stream_repair` 作为 public event 类型。
3. 是否接受 sequence + UTF-8 byte offset + sha256 作为 replay/reconcile 契约。
4. 是否接受 text-producing turn 必须 `assistant_text_final` 早于 public `done`。
5. 是否按 Shadow -> Dual public -> Cutover 三阶段迁移。
6. 是否允许最终清理旧 `content_delta` assistant 正文主链路。

# 146-任务协议修复反馈与等待占位投影链路修复记录-20260615

## 背景

用户追查新任务时发现：后台任务仍在运行，但前端没有继续显示后续反馈；当任务模型输出不合法 action JSON 后，系统进入补合法流程，用户侧却看不到自然推进。

本次确认的目标链路：

```text
任务模型输出不合法 action JSON
-> 系统生成 protocol repair observation
-> observation 回喂给任务模型
-> 模型重新输出合法 action，并用 public_progress_note / public_action_state 给用户自然解释
-> 前端只投影模型合法公开反馈与必要状态，不裸露 repair instruction
```

## 关键问题

### 1. task handoff 被本地化 terminal_reason 绊断

最新任务 `taskrun:turn:session-9458b9376ed8437e:9:5c7c9c1a` 对应 chat 流：

- `chatrun:strun_938116eb8be446b189149bc3e6adcde1`
- chat offset 33 发出 `turn_completed`
- `terminal_reason` 已被转换为中文：`任务已进入执行流程`
- `completion_state` 保留机器态：`task_executor_scheduled`

旧判断优先读取 `terminal_reason`，导致 `_is_task_executor_handoff_terminal` 没有识别 task handoff，chat SSE 提前结束，后续 task 事件没有桥回前端。

### 2. model wait 心跳被后端直接丢弃

`task_model_action_wait_heartbeat` 已由 executor 写入 task event log，但 `backend/api/chat.py` 在 `_project_public_stream_event` 中直接 `return []`，前端没有任何可见等待占位。

### 3. protocol repair 状态必须保持 trace-only

`runtime.protocol_repair` 是系统给模型的内部修复反馈，不是模型给用户的公开反馈。即使系统记录“当前步骤输出格式不完整，正在自动修正后继续”，也只能进入 trace/log，不能进入用户可见投影。

### 4. 等待占位需要在真实信号出现后消失

前端原本直接过滤 `model_wait_placeholder`，导致占位永远不可见。正确行为是：

- 等待期间显示轻量状态。
- 后面出现正文、工具或状态等真实可见信号时，等待占位自动消失。

## 修复内容

### 后端 chat 桥接

文件：`backend/api/chat.py`

- `TASK_BRIDGE_PUBLIC_EVENT_TYPES` 增加 `task_model_action_wait_heartbeat`，允许任务等待心跳进入当前 assistant timeline。
- `_is_task_executor_handoff_terminal` 改为检查 canonical 字段优先：`completion_state -> terminal_reason -> status`。
- `task_model_action_wait_heartbeat` 投影为 `runtime_status`，字段为：
  - `status_kind=model_wait_placeholder`
  - `item_id=model-wait:{task_run_id}`
  - `title=正在思考`
  - `slot` 由投影层决定为 status
- 任务桥接收口保留 `completion_state` 为机器态，不再把它本地化；中文展示只放 `terminal_reason`。

### 投影权威

文件：`backend/harness/runtime/projection/projector.py`

- `runtime.model_wait` / `model_wait_placeholder` 不再 hidden，改为：
  - `slot=status`
  - `main_visibility=visible_live`
  - `retention=transient`
  - `status_kind=model_wait_placeholder`
- `runtime.protocol_repair` 保持 trace-only：
  - `slot=trace`
  - `main_visibility=hidden`
  - `retention=trace`
  - `title/text/detail` 不携带“当前步骤输出格式不完整”等用户可见文案
  - 不暴露 `public_response_required` 等 repair instruction 细节
- 仍然不把 protocol repair 当成正文，也不当成工具窗口。

### 前端显示

文件：`frontend/src/components/chat/ChatMessage.tsx`

- `model_wait_placeholder` 不再无条件过滤。
- 只有当等待占位之后已经出现更晚的正文、工具或状态信号时，才过滤该占位。
- 这样等待占位在模型返回间隙可见，真实信号出现后不会残留。

## 当前结论

这次修复后，协议补合法链路的显示边界为：

- repair observation 是系统给大模型的内部修复反馈，不进正文、不进工具窗口。
- 用户侧只看到轻量状态和模型重新输出的合法公开反馈。
- task handoff 不再受中文展示文案影响，后续 task event 能继续桥接到当前 assistant timeline。

# 图任务运行时模型覆盖与断点续跑修改计划

## 目标

让图任务支持临时模型覆盖和断点续跑，做到：

- 不重发图配置
- 不重跑已完成部分
- 不改单 agent 主链路
- 可按 run / role_group / node 临时切模型

## 现状

当前链路是：

`node.model_requirement` -> `GraphNodeWorkOrder.runtime_profile` -> `TaskRun.diagnostics.model_selection` -> `task_executor._task_model_selection()`

问题在于，写作图节点把 `deepseek-v4-flash` 写死在图配置里，临时切 `pro` 需要重发布图配置，这不符合运行时控制的目标。

更准确的执行链路是：

1. `GraphHarnessConfig.nodes[].contracts.contract_bindings.runtime.model_requirement` 保存节点模型需求。
2. `GraphContextMaterializer` 生成 `GraphNodeWorkOrder.input_package` 和 `graph_slot.node_contract`。
3. `work_order_contract._graph_node_contract_from_work_order()` 把 `graph_slot.node_contract.model_requirement` 写入 `TaskRunContract.runtime_profile`。
4. `start_task_lifecycle()` 把 `runtime_task_selection` 和 `model_selection` 写入 task run diagnostics。
5. `task_executor._task_model_selection()` 从 task run diagnostics 和 agent profile 里解析最终模型。

因此 override 的合并点必须在图层进入 `TaskRunContract` / `runtime_task_selection` 之前，不能等到单 agent 已经开始执行后再修改。

## 设计原则

1. 图定义只描述能力需求，不固定具体模型。
2. 运行时可以覆盖模型，但不改发布态图配置。
3. 覆盖层只作用于图任务，不侵入单 agent 主执行链。
4. 继续沿用现有 `task_executor` 模型选择逻辑，只在上游把解析结果合并好。

## 目标优先级

1. `run` 临时 override
2. `graph run` 持久 runtime settings
3. `node model_requirement`
4. `agent runtime profile`
5. `system default`

说明：

- `run` 临时 override 只对当前 API 调用生效，适合“下一次续跑用 pro”。
- `graph run` 持久 runtime settings 要写入 graph run / graph loop state，适合“当前任务后续都用同一策略”。
- 两者不能混淆；否则会出现一次续跑后下一次调用又退回旧模型的问题。

## 覆盖粒度

- `global`
- `agent_id`
- `role_group`
- `node_id`

当前图节点没有稳定的一等 `role_group` 字段，因此需要先定义派生规则或在节点 metadata 中补充：

- `writing`：创作/写手节点，例如 `chapter_draft`、`chapter_draft_self_repair`
- `review`：审核节点，例如 `chapter_review`
- `memory`：记忆提交节点，例如 `memory_commit_chapter`

节点 ID 匹配必须同时支持完整模块 ID 和裸节点 ID，例如：

- `graph_module.chapter_cycle::chapter_draft`
- `chapter_draft`

匹配优先级：

1. 完整 `node_id`
2. 裸 `node_id`
3. `agent_id`
4. `role_group`
5. `global`

示例：

```json
{
  "model_overrides": {
    "role_groups": {
      "writing": {
        "provider": "deepseek",
        "model": "deepseek-v4-pro",
        "credential_ref": "env:DEEPSEEK_WRITING_API_KEY"
      }
    },
    "nodes": {
      "chapter_draft": {
        "model": "deepseek-v4-pro"
      },
      "chapter_draft_self_repair": {
        "model": "deepseek-v4-pro"
      }
    }
  }
}
```

## 变更范围

### 1. API 层

文件：

- `backend/api/orchestration.py`

新增：

- `runtime_overrides`
- `model_overrides`

让 `run-until-idle` 和 work order execute 都能接收临时模型覆盖。

需要同时区分：

- `runtime_overrides`：本次请求临时覆盖
- `runtime_settings_patch`：写入当前 graph run 的持久设置

如果只实现 `runtime_overrides`，调用方必须每次续跑都传同一份设置；如果要“临时改设置后持续续跑”，必须实现持久设置写入。

### 2. Graph runtime 层

文件：

- `backend/harness/graph/models.py`
- `backend/harness/graph/runtime.py`
- `backend/harness/graph/runner.py`
- `backend/harness/graph_harness.py`

新增图运行时设置承载和透传能力，保证断点续跑时仍能继承同一份 override。

需要明确存储位置：

- `GraphRun.diagnostics.runtime_settings`
- 或 `GraphLoopState.diagnostics.runtime_settings`

推荐以 `GraphLoopState.diagnostics.runtime_settings` 为主，因为 runner、dispatch 和 work order 都会读取 loop state。

### 3. Work order 装配层

文件：

- `backend/harness/graph/context_materializer.py`
- `backend/harness/graph/work_order_contract.py`
- `backend/harness/graph/work_order_executor.py`

在生成 `runtime_profile.model_requirement` 前合并 override，形成最终模型选择。

必须覆盖两种情况：

1. 尚未 dispatch 的节点：在 materializer 创建 work order 时合并。
2. 已经 active 的 work order：在 executor 接收 work order 后、创建 TaskRunContract 前再次合并。

第二点很重要。否则用户在节点已经处于 active work order 时切模型，续跑仍会使用旧 work order 中的旧模型需求。

### 4. 模型选择层

文件：

- `backend/harness/loop/task_executor.py`
- `backend/agent_system/models/model_profile_resolver.py`

保持主选择逻辑不变，只增强输入来源和诊断可见性。

注意：如果上游已经把 `TaskRun.diagnostics.model_selection` 写死，`task_executor._task_model_selection()` 会优先使用这个 snapshot。图层 override 必须在创建 task run 前完成，或在恢复旧 task run 前显式重建/更新该 snapshot。

### 5. 图配置脚本

文件：

- `scripts/configure_writing_modular_novel_graph.py`

将图配置里的模型写死项逐步收敛为能力需求，不再把具体模型作为唯一事实来源。

短期可以保留 `deepseek-v4-flash` 作为默认模型，但必须把它标记为 default，而不是不可覆盖的 contract authority。

长期目标：

- `model_requirement` 描述能力需求：长输出、流式、最小输出、credential role。
- 具体 `model` 从运行时策略解析。

## 需要保留不动的部分

- 单 agent 调用链
- `task_executor._task_model_selection()` 的主体结构
- 现有质量门
- 现有写作图节点拓扑

但需要允许图层在进入单 agent 前重建 `runtime_profile.model_requirement` 和 `model_selection`，这不属于改动单 agent 主链路。

## 诊断要求

任务诊断里要能看见：

- override 来源
- 匹配粒度
- 原始模型
- 生效模型

但不能写入 raw secret。

诊断字段建议：

```json
{
  "graph_model_override": {
    "matched_scope": "node_id",
    "matched_key": "chapter_draft",
    "source": "run_until_idle.runtime_overrides",
    "original": {
      "provider": "deepseek",
      "model": "deepseek-v4-flash",
      "credential_ref": "env:DEEPSEEK_WRITING_API_KEY"
    },
    "effective": {
      "provider": "deepseek",
      "model": "deepseek-v4-pro",
      "credential_ref": "env:DEEPSEEK_WRITING_API_KEY"
    }
  }
}
```

`credential_ref` 可以记录，实际 API key 不能记录。

## 验收标准

1. 不改图配置就能临时把写作节点切到 `deepseek-v4-pro`。
2. `chapter_draft` 和 `chapter_draft_self_repair` 可单独切模型。
3. `chapter_review`、`memory_commit` 等节点可维持原策略。
4. 单 agent 正常任务不受影响。
5. 断点续跑后，后续节点继续沿用同一份 override。
6. 覆盖前后均有回归测试。

补充验收：

7. 已经 active 的 work order 在执行前也能吃到本次 override。
8. 裸节点 ID 和模块化节点 ID 都能匹配。
9. role_group 覆盖只影响对应节点组。
10. override 不修改 `graph_harness_configs.json` 中的发布态配置。
11. raw secret 被拒绝，只允许 `credential_ref`。

## 实施顺序

1. 先加 override 数据结构和 API 入参。
2. 再把 override 透传到 graph runtime 和 work order。
3. 再合并到 node 级 model selection。
4. 最后补测试。

更细分的实施顺序：

1. 增加 override schema 和 secret 校验。
2. 增加节点匹配器，支持完整 ID、裸 ID、agent_id、role_group、global。
3. 在 `run-until-idle` 和 `work-orders/execute` 接收临时 override。
4. 增加 graph run 持久 runtime settings patch。
5. 在 work order materializer 合并未 dispatch 节点的模型需求。
6. 在 work order executor 合并 active work order 的模型需求。
7. 在 task run diagnostics 写入 override 诊断。
8. 增加回归测试。

## 已发现的冲突与处理

### 冲突 1：临时 override 与持久设置边界不清

如果只实现 `runtime_overrides`，下一次 API 调用不会自动继承。  
处理：同时设计 `runtime_settings_patch`，让用户可以选择“一次性覆盖”或“写入当前 graph run”。

### 冲突 2：active work order 可能已经带旧模型需求

如果只在 dispatch/materializer 阶段合并，已经 active 的节点不会生效。  
处理：executor 创建 `TaskRunContract` 前必须再次合并 override。

### 冲突 3：role_group 当前不是图节点字段

如果直接按 role_group 匹配，会没有稳定来源。  
处理：短期用 agent_id/node_type/node_id 派生；长期写入节点 metadata。

### 冲突 4：节点 ID 存在模块前缀

发布态节点 ID 可能是 `graph_module.chapter_cycle::chapter_draft`。  
处理：匹配器必须支持完整 ID 和 `::` 后缀裸 ID。

### 冲突 5：任务恢复时已有 model_selection snapshot

如果旧 task run 已经创建，`task_executor` 可能优先使用 diagnostics 里的 `model_selection`。  
处理：图层续跑应优先对新执行节点生效；如果要改已经创建的 waiting executor task run，需要显式更新或重建该 task run 的 model_selection。

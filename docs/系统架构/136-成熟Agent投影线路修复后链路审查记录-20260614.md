# 136-成熟Agent投影线路修复后链路审查记录

日期：2026-06-14

审查对象：当前前后端投影线路是否符合 `135-成熟Agent投影线路结构性修复计划书-20260614.md`。

审查结论：实时主链路已经基本按“正文区 + 工具窗口”双表面接通，但不能判定为完整验收。当前最大缺口是历史 replay 不是从完整 public projection ledger 恢复，而是从 task/turn 原始事件尾窗重建；长任务、刷新、切会话、非任务状态下会丢失旧工具窗口，且 live 与 history 的 frame identity/event_offset 可能漂移。

## Findings

### [P1] 历史 replay 只读尾部事件，长会话会丢旧工具窗口

证据：

- 计划要求：135 文档 L29、L39、L336、A6 要求刷新/切会话/非任务状态下，旧工具窗口必须从 public projection replay 恢复。
- `backend/harness/runtime/session_timeline.py:50-57` 和 `97-103` 为 task/turn attachment 只读取 `_recent_events(... limit=max_timeline_items * 8)`，默认 `24 * 8 = 192`。
- `backend/harness/runtime/session_timeline.py:376-380` 调用 `runtime_host.event_log.list_event_window(... include_payloads=True)`。
- `backend/runtime/shared/event_log.py:161-166` 的 `list_event_window(... include_payloads=True)` 实际调用 `read_event_tail_raw(... tail_limit=limit)`，是尾部窗口，不是完整事件账本。
- `backend/harness/runtime/projection/projector.py:31-141` 的 `ProjectionLifecycleState` 要求工具生命周期严格按 `tool_call_requested -> tool_permission_decided -> tool_item_started -> tool_item_completed` 进入；其中 completed 在 `114-126` 明确要求已有 started record，否则会落成隐藏诊断。

断线：

`session timeline` -> `tail raw runtime events` -> `history projector lifecycle state`

当 request/permission/start 已经掉出尾窗、completion 仍在尾窗内时，history projector 只看到 completion，无法恢复同一个工具窗口。

影响：

- 非任务状态下旧工具窗口可能消失。
- 刷新/切会话后，用户看不到 agent 曾经调动系统的完整控制轨迹。
- 这正好违反 135 的 L6/A6，是功能完整性缺口，不是 UI 小问题。

要求修复方向：

历史恢复不能依赖短尾窗。要么读取完整 public projection ledger，要么至少读取足够完整的 lifecycle 公开 frame；不能只拿最近 N 条 raw event 重建工具窗口。

### [P2] history replay 没有使用 live 已保存的 public stream ledger，live/history 身份可能漂移

证据：

- 实时路径在 `backend/api/chat.py:1009-1054` 先 `_attach_public_projection_frame()`，再 `replay.append_public_event()`。
- `backend/runtime/shared/stream_replay.py:38-57` 把 public event 追加到 `run.event_log_id`；`backend/runtime/shared/runtime_run_registry.py:69-74` 创建的 event log 是 `chatrun:*`。
- session timeline 当前从 `state_index.list_session_task_runs()` 和 `list_session_turn_runs()` 取 task/turn run：`backend/harness/runtime/session_timeline.py:17-31`。
- history frame 是在 `backend/harness/runtime/session_timeline.py:184-241` 里重新调用 `_project_public_stream_event()` 和 `project_public_projection_event()` 从 raw event 重建。
- history 重建还把 source offset 改写成 `source_offset * 10 + public_index`：`backend/harness/runtime/session_timeline.py:221`。实时路径使用 chat public event 的连续 offset：`backend/api/chat.py:1044-1054`。
- `RuntimeRunRegistry` 已有 `list_session_runs()` 能列出 `chatrun:*` run：`backend/runtime/shared/runtime_run_registry.py:100-103`，但 session timeline 没使用它。

错接：

135 的 Flow 要求：

`runtime event -> backend public event bridge -> PublicProjectionFrame -> persisted public projection frames / ledger -> live SSE reducer -> history hydrate replay`

当前实际是：

`live SSE`: `runtime event -> public bridge -> PublicProjectionFrame -> chatrun public event log -> reducer`

`history`: `task/turn raw event tail -> re-project -> new PublicProjectionFrame -> reducer`

这不是同一个 ledger replay。

影响：

- live 和 history 的 `frame_id/event_offset` 不是同源账本，刷新、重连、合并 volatile state 时存在重复投影或错序风险。
- 即使多数场景下因为整页刷新是从空状态重建而看不出重复，这条线路仍不符合成熟 agent 的 replay 设计。

要求修复方向：

history timeline 应优先读取 `chatrun:*` 中已经持久化的 public event/data.public_projection_frame，并按原始 public event offset replay；只有缺 ledger 的旧数据才允许受控迁移或一次性重建，不能把 raw-event reproject 当主路径。

### [P2] public bridge 对 `tool_calls[]` 的直接输入不完整，依赖上游拆分

证据：

- `backend/api/chat.py:1980-1989` 的模型反馈识别允许 `tool_call` 和 `tool_calls`。
- 但 `backend/api/chat.py:2025-2035` 的 `_tool_call_requested_data()` 只接受 `action_type == "tool_call"`，并只读取 `request.tool_call`。
- 当前主路径有拆分保护：single-agent turn 在 `backend/harness/loop/single_agent_turn.py:2638-2698` 把 native tool calls 解析成多个 `tool_actions`，并在 `840-917` 循环记录 admission；started 事件在 `4615-4674` 逐个写入。
- task executor 在 `backend/harness/loop/task_executor.py:3056-3088` 把 `tool_calls[]` 拆成 child request，并在 `1810-1895` 为每个 child request 写 admission。

风险：

当前主路径通常不会直接把未拆分的 `tool_calls[]` 喂给 `_tool_call_requested_data()`，所以不是马上可复现的主故障。但 bridge 函数本身不完整；只要未来有新的 producer 直接发 `action_type=tool_calls`，模型反馈会进入正文，工具 request 窗口会丢失。

要求修复方向：

public bridge 应明确支持数组输入并为每个 tool call 产出独立 `tool_call_requested`，或在 bridge 边界显式拒绝未拆分 `tool_calls[]` 并记录结构诊断。不能靠隐含“上游一定拆过”作为长期契约。

### [P3] 模型反馈去重偏文本级，不是严格 action/frame 级

证据：

- `backend/api/chat.py:2006-2017` 给模型反馈构造了带 action ref 的 `runtime_event_id`。
- 但 `backend/harness/runtime/projection/projector.py:703-708` 的 `_runtime_stage_status_item_id()` 只按 task/turn 生成 public stage status item id。
- `backend/harness/runtime/projection/projector.py:693-694` 的 status frame id 由 stage item id、title、detail 组成；同一 task/turn 内相同 title/detail 会合并。
- 前端 `frontend/src/components/chat/ChatMessage.tsx:323-365` 又用 `compactText(feedbackText)` 做显示层去重。

影响：

这能满足 135 的“同一句模型阶段反馈不要随多个工具重复出现”，但不是严格时序保真。如果同一模型公开反馈在不同 action 中合法重复出现，当前 UI 会倾向折叠掉后一次。

要求修复方向：

保留“同一 action 同一句反馈不重复”的目标，但去重 key 应优先用 frame/action/source identity；纯文本去重只能作为同一 source/action 下的辅助规则。

## 审查结论

当前线路不能标记为“逻辑完全精准、功能完整”。它已经完成了实时双表面投影的主体结构：

1. 正文进入正文区。
2. 模型公开反馈进入正文区。
3. 工具请求、开始、完成按 `tool_call_id` 合并为工具窗口。
4. final/commit 后不再清空工具轨迹。
5. 工具窗口具备折叠态和基本命令/目标/参数/输出信息。

但仍有两个结构性验收阻塞：

1. 历史 replay 只读尾部 raw events，长会话旧工具窗口会消失。
2. 历史 replay 不是 live public ledger replay，live/history 身份和 offset 可能漂移。

因此，下一轮修复应优先改 `session_timeline` 的数据源和 replay 权威：把 `chatrun:*` public stream ledger 作为历史投影主来源，然后再处理数组 bridge 和反馈去重精度。

本次审查未启动前后端固定端口实测；结论来自代码线路和 135 计划书逐行对照。

# 用户输入模型回应义务协议修复计划书

日期：2026-06-13

## 1. 问题定义

用户输入之后，agent 可以进入工具调用、任务执行、运行控制或等待模型输出，但只要用户当前输入包含提问、质疑、状态追问、纠错、补充要求或需要解释的意图，语义回应必须来自大模型本身。系统层可以显示空白思考占位，也可以在模型长时间无响应或服务异常时显示诊断解释，但系统不得生成“已收到”“等待中”“正在处理”这类替模型表达理解或承诺的公开语义文本。

当前失败模式是：运行时协议把 `tool_call` 缺少公开模型回应降级为 `contract_gaps`，然后仍然执行工具。结果是用户看到内部动作和等待状态，但没有看到 agent 对当前问题的模型回应。

## 2. Codex 源码参照

参照源码位置：`D:\AI应用\openai-codex`。

关键设计点：

- `codex-rs/protocol/src/prompts/base_instructions/default.md` 的 `Preamble messages` 要求模型在工具调用前发送简短说明。这是模型生成的语义文本，不是系统伪造状态。
- `codex-rs/core/src/stream_events_utils.rs` 中，工具调用和 `TurnItem::AgentMessage` 分离。工具调用执行后只把 observation 回灌给模型，并设置 follow-up；是否向用户解释由后续模型消息决定。
- `codex-rs/core/src/session/turn.rs` 的 turn loop 会把工具 observation、pending user input 继续送回模型，不把工具事件当成对用户的回答。
- `codex-rs/core/src/tasks/regular.rs` 在存在 pending input 时继续跑下一轮模型 turn，保证用户输入不会被工具执行悄悄吞掉。

本项目应借鉴的是这些不变量，而不是直接照搬 Rust 结构：

1. 用户语义文本只来自模型。
2. 工具事件是内部执行事件，不等于回应。
3. 工具 observation 回灌给模型，由模型决定是否解释。
4. 用户输入触发的首个 action 必须满足回应义务；纯内部 follow-up 可以保留工具槽。

## 3. 当前代码证据

- `backend/harness/loop/model_action_protocol.py`：`require_public_progress_note=True` 时，`tool_call` 缺少 `public_progress_note` 只进入 `contract_gaps`，仍然 accepted。
- `backend/harness/loop/single_agent_turn.py`：native tool call 会从参数中提取 `public_progress_note`，缺失时只记录 `public_progress_note_missing_for_native_tool_call`，仍然返回工具 action。
- `backend/harness/loop/task_executor.py`：task execution 调用 parser 时要求 public note/state，但由于 parser 对 tool_call 放宽，裸工具调用仍可执行。
- `backend/harness/runtime/public_progress.py` 和 run monitor/frontend 投影中仍存在“等待模型输出”“已收到补充要求”等系统语义文案，容易被误当成 agent 回应。

## 4. 目标架构

目标权威链：

```text
UserInput / PendingSteer
-> ModelTurnDecision
-> ModelActionProtocol(public_response_required)
-> ActionPermit
-> ToolExecution / TaskExecution
-> ToolObservation
-> ModelFollowupDecision
-> Projection
```

层职责：

- `ModelTurnDecision`：唯一可以生成用户语义回应的层。
- `ModelActionProtocol`：检查 action 是否满足本轮回应义务；不补写语义文本。
- `ActionPermit`：只授权或拒绝动作，不替模型解释。
- `ToolExecution`：只执行并记录 observation。
- `Projection`：只展示模型语义字段、工具状态、空白思考占位或故障诊断；不把系统状态改写成用户可读承诺。

## 5. 实施计划

### 阶段一：协议强约束

修改 `backend/harness/loop/model_action_protocol.py`：

- 新增 `public_response_required` 参数。
- 当 `public_response_required=True` 时，`respond` 必须有 `final_answer`，`ask_user` 必须有 `user_question`，`block` 必须有 `blocking_reason`。
- 当 `public_response_required=True` 时，`tool_call`、`request_task_run`、`active_work_control` 必须至少提供 `public_progress_note` 或 `public_action_state.current_judgment`，否则 invalid。
- 只有 `public_response_required=False` 的内部 follow-up 才允许裸工具调用继续作为低层工具槽。

### 阶段二：执行链路接入

修改 `backend/harness/loop/single_agent_turn.py`：

- 初始用户 turn 解析模型 action 时传入 `public_response_required=True`。
- 工具 observation 后 follow-up 解析时可以传 `False`，因为这属于内部工具循环；如果 observation 改变用户问题判断，prompt 仍要求模型解释。
- native tool call 在回应义务开启时，缺少 public note/current judgment 必须产生 repairable protocol error，不执行工具。

修改 `backend/harness/loop/task_executor.py`：

- 用户 steer、任务模型 action 入口保持 `public_response_required=True`。
- approved pending tool replay 属于已批准动作重放，可继续 `False`，避免要求用户重复确认旧工具动作。

### 阶段三：Prompt 合同校准

修改 `backend/harness/runtime/compiler.py`：

- 保留当前 `public_response_obligation`，但进一步强调：用户问题必须被回应，工具 observation 按触发条件解释，系统/工具槽不能替代回应。
- 协议修复提示要明确：修复输出仍必须满足 `public_response_required`，不得只补工具 JSON。

### 阶段四：投影边界清理

修改：

- `backend/harness/runtime/public_progress.py`
- `backend/harness/runtime/run_monitor/projector.py`
- `backend/harness/loop/task_steering.py`
- `backend/harness/entrypoint/runtime_facade.py`
- `frontend/src/lib/store/runtime.ts`

要求：

- “已收到/等待模型输出/正在处理”不能作为 `latest_public_progress_note`。
- 模型长时间无响应时，可以显示明确诊断：模型响应超时、服务或网络可能异常。
- 聊天窗口的思考占位可以保留为空白/非语义状态，不进入模型语义字段。

### 阶段五：验证

重点测试：

- 用户问题触发的裸 `tool_call` 被 rejected。
- native tool call 在回应义务开启时缺少 public note 被 rejected。
- 内部 follow-up 裸工具调用仍可通过。
- task execution 的裸 `tool_call` 被 rejected。
- 投影不把系统等待/已收到文案提升为公开回应。
- stale model wait 仍显示故障诊断。

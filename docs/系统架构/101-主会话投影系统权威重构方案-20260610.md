## 一、重构结论

主会话投影系统不能继续按单点问题修补。当前问题的本质不是文案、清洗或某个组件渲染异常，而是正文、工具活动、任务阶段、控制命令、持久化恢复和前端水合之间存在多重权威。

目标重构方向：

```text
用户输入
-> ActiveTurn 建立唯一当前轮次
-> 模型产生公开正文或动作决策
-> ActionPermit 授权动作
-> Runtime 执行动作并记录 observation
-> StageController 判断阶段边界
-> PublicProjection 生成唯一公开投影
-> SessionTimeline 按同一契约重建
-> Frontend 只按 envelope/anchor 渲染
```

核心原则：

1. assistant 正文只能来自模型公开自然语言。
2. 工具调用、等待、观察、执行状态属于动作层，不属于正文。
3. 控制命令、JSON action、工具协议、terminal reason 不允许进入正文或用户可复制内容。
4. 工具反馈必须短，只描述工具事实。
5. 阶段总结必须一等化，承载分析、归因、决策变化和下一阶段方向。
6. `task_projection` 不能替代 `public_timeline`。
7. live stream 与刷新后重建必须同构。
8. 找不到明确 anchor 时 fail closed，不挂到最新 assistant。
9. 新旧投影契约必须可区分；旧 envelope 宽读，新 envelope 严读。

## 二、本地成熟源码依据

### 1. Claude Code：任务对象只管生命周期，进度是结构事件

本地路径：`D:\AI应用\claude-code-nb-main`

关键依据：

- `Task.ts`：`TaskStateBase` 只包含 `id/type/status/description/toolUseId/startTime/endTime/outputFile/outputOffset/notified` 等生命周期字段。任务接口只暴露 `kill`，没有把显示、正文、工具结果混到任务对象里。
- `utils/task/framework.ts`：`registerTask()` 发送 `task_started` SDK 事件；`generateTaskAttachments()` 通过 `outputOffset` 读增量，并避免异步读后的旧状态覆盖新状态。
- `utils/task/sdkProgress.ts`：`task_progress` 是结构化 system event，包含 `task_id/tool_use_id/usage/last_tool_name/summary`。
- `tasks/LocalAgentTask/LocalAgentTask.tsx`：后台 agent 有 `pendingMessages/progress/summary/retain/diskLoaded/evictAfter`，进度摘要是 1-2 句结构化 summary，不是正文占位。

可借鉴规则：

```text
Task = 生命周期和执行容器
SDK progress = 结构化任务进度
Assistant body = 模型正文
```

不要借鉴的部分：

- Claude Code 面向 CLI/SDK 的 task notification XML 不适合直接作为本项目页面投影。
- 本项目应保留现有 `PublicProjectionEnvelope`，用它承载页面契约。

### 2. Codex：turn、正文流、工具事件、控制事件彻底分离

本地路径：`D:\AI应用\openai-codex`

关键依据：

- `codex-rs/protocol/src/protocol.rs`：`EventMsg` 明确拆分 `TurnStarted`、`TurnComplete`、`AgentMessage`、`AgentMessageContentDelta`、`ExecCommandBegin/OutputDelta/End`、`McpToolCallBegin/End`、`TurnAborted`、`PlanUpdate`。
- `codex-rs/protocol/src/models.rs`：`MessagePhase` 区分 `Commentary` 和 `FinalAnswer`；缺失 phase 时消费者必须保留 fallback，而不是随意改写正文。
- `codex-rs/protocol/src/items.rs`：`TurnItem` 把 `AgentMessage`、`Plan`、`Reasoning`、`WebSearch`、`FileChange`、`McpToolCall` 分成不同 item。
- `codex-rs/core/src/tools/events.rs`：工具 begin/end 是公开事件；工具结果另行格式化回模型。工具失败也返回给模型，不静默吞掉。
- `codex-rs/tui/src/chatwidget/streaming.rs`：正文流有 stream controller；工具/item 事件在正文流活跃时 `defer_or_handle`，避免顺序错乱。
- `codex-rs/tui/src/chatwidget/turn_runtime.rs`：turn complete 时先 flush answer stream，再 finalize plan stream，再处理下一条 queued input。
- `codex-rs/app-server/README.md`：`turn/steer` 追加到 active regular turn，不发新 `turn/started`；`turn/interrupt` 以 interrupted completion 收口。
- `codex-rs/core/src/agent/control.rs`：子 agent 由 `AgentControl` 管理 spawn/message/interrupt/status；父线程通过状态订阅和完成通知等待子线程。

可借鉴规则：

```text
AgentMessage = 正文
ToolBegin/ToolEnd = 工具活动
TurnStarted/TurnComplete/TurnAborted = 生命周期
Steer = active turn 内追加输入
Interrupt = 请求中断，并以 interrupted terminal 收口
```

## 三、当前链路冲突

### 1. 动作协议仍允许裸工具

文件：`backend/harness/loop/model_action_protocol.py`

当前问题：

```python
if require_public_progress_note and action_type != "tool_call" and not public_progress_note:
    errors.append("public_progress_note_required")
```

含义：`tool_call` 被排除在公开反馈要求外。模型可以不解释就调用工具。

目标：`tool_call` 不应被强行阻断，但必须支持并鼓励公开反馈；缺失时记录 `public_feedback_missing` contract gap。

### 2. Native tool call 固定空公开反馈

文件：`backend/harness/loop/single_agent_turn.py`

当前问题：

```python
public_progress_note=""
```

这会让原生工具调用天然没有开局反馈。前端只能看到工具，无法看到模型为什么要查。

目标：从模型安全 content、tool args 中提取公开反馈候选；没有候选时不伪造正文，只记录缺口。

### 3. `task_projection` 会吃掉 `public_timeline`

文件：`backend/harness/runtime/session_timeline.py`

当前问题：

```python
public_timeline = []
if not task_projection:
    public_timeline = project_public_timeline_from_events(...)
```

这直接导致刷新后只剩 task projection，看不到工具后的阶段反馈。

目标：`task_projection` 和 `public_timeline` 必须共存。前者负责结构态，后者负责公开反馈链。

### 4. 任务阶段不是一等语义

文件：`backend/harness/loop/work_rollout.py`

当前类型：

```text
progress
user_instruction
pause_boundary
interrupted_boundary
final_response
```

缺少：

```text
stage_summary
phase_boundary
tool_group_summary
```

结果：阶段总结只能混在普通 progress 里，前端无法可靠折叠被总结覆盖的工具记录。

### 5. `session_task_projection` 的 phase 是技术态

文件：`backend/harness/runtime/session_task_projection.py`

当前 phase 主要是：

```text
executing
tool_waiting
scheduled
handoff
blocked
completed
```

这些适合内部执行，不适合作为用户理解的任务阶段。

目标用户阶段：

```text
opening_judgment
investigation
plan_confirmation
implementation
verification
closeout
blocked
interrupted
```

### 6. 前端仍有旧 fallback 和正则清洗

相关文件：

- `frontend/src/lib/internalControlText.ts`
- `frontend/src/lib/store/events.ts`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/lib/runtimeVisibilityProjection.ts`
- `frontend/src/components/chat/PublicTimelineActivity.tsx`

当前风险：

- 依赖前端识别 `action_type/tool_call/model_action_request` 来遮挡泄露。
- 旧 fallback 仍会生成 `正在思考`、`处理进展`、`工具返回成功，正在根据结果继续。` 等泛化展示。
- 这些逻辑只能作为最后防线，不能作为控制系统安全边界。

目标：公开投影 envelope 成为唯一渲染事实；旧 fallback 只允许临时 UI chrome，不写入 message 或持久 activity。

## 五、公开投影契约

### 1. Envelope 不变，但收紧解释

继续使用：

```text
harness.public_projection.v1
```

不急于改 authority 版本号。先收紧 v1 的生产和消费规则，避免重写过大。

但必须增加可区分的新契约字段：

```json
{
  "contract_revision": "20260610-authority-refactor",
  "projection_mode": "shadow|authoritative",
  "producer": "harness.public_projection"
}
```

兼容规则：

1. 没有 `contract_revision` 的旧 envelope 按旧规则宽读，不能用新规则硬丢。
2. 有新 `contract_revision` 的 envelope 按本文严格校验。
3. shadow 期 `projection_mode=shadow` 只用于对照，不自动压制 legacy 显示。
4. cutover 后 `projection_mode=authoritative` 才允许前端用 envelope 抢占 legacy。

最小 envelope：

```json
{
  "authority": "harness.public_projection.v1",
  "contract_revision": "20260610-authority-refactor",
  "projection_mode": "authoritative",
  "projection_id": "publicproj:...",
  "sequence": 12,
  "session_id": "...",
  "anchor": {
    "turn_id": "turn:...",
    "message_id": "assistant:...",
    "task_run_id": "taskrun:...",
    "run_id": "turnrun:..."
  },
  "lifecycle": "running",
  "source_authority": "model|tool|system|runtime",
  "surface": "assistant_body|tool_window|status_bar|task_projection|control",
  "items": []
}
```

硬规则：

1. `slot=body` 只允许 `source_authority=model` 且 `surface=assistant_body`。
2. `tool_window` item 必须有稳定 `item_id` 或 `tool_call_id`。
3. `stage_summary` 必须带 `phase_ref` 和 `covers_tool_refs`。
4. `control` surface 不进入正文，也不进入可复制 assistant content。
5. `terminal.visible=false` 只关闭临时 turn，不产出正文。
6. 审批等待和安全边界等待必须表现为 `ActiveTurn.state`，不能只写在 `item.detail`、`metadata` 或前端局部状态里。

### 2. Public item 类型

保留和补充以下类型：

```text
opening_judgment       模型开局判断
work_action            工具动作开始/等待/完成/失败
tool_observation       工具观察摘要，可并入 work_action done
observation_report     模型基于观察的短判断
stage_summary          阶段总结
final_summary          最终正文摘要或最终回答辅助 item
pause_boundary         暂停边界
interrupted_boundary   中断边界
steer_ack              用户补充要求已纳入
approval_request       权限/审批请求
approval_decision      权限/审批结果
blocked                阻塞说明
```

工具反馈与阶段总结必须区分：

```text
工具反馈：读取到 session_timeline 在 task_projection 存在时跳过 public_timeline。
阶段总结：刷新后反馈丢失发生在持久化重建层；下一阶段先修重建，让 task_projection 与 public_timeline 共存。
```

### 3. ActiveTurnState 一等状态

`ActiveTurn.state` 是运行时控制权威，不是公开 item 的 detail 字段。detail 只能补充解释某个公开 item，不能承担状态机语义。

目标枚举：

```text
starting                turn 已建立，模型尚未开始稳定输出
model_turn              模型正在产生正文或动作决策
running_task            runtime 正在执行工具、工作流或子 agent
waiting_executor        已调度执行器，等待执行资源或结果
waiting_user            等待用户普通输入，例如 ask_user
waiting_approval        等待用户审批工具、写入、外部执行或高风险控制动作
waiting_safe_boundary   pause/interrupt/stop 已请求，等待模型流、工具或子 agent 到达可收口边界
interrupting            正在执行中断收口
terminal                当前 turn 已收口
```

硬规则：

1. `waiting_approval` 只能由 ActionPermit/approval flow 设置，必须绑定 `approval_request` control item。
2. `waiting_safe_boundary` 只能由 pause/interrupt/stop flow 设置，必须绑定 `pause_boundary` 或 `interrupted_boundary` control item。
3. 两个状态都必须进入后端序列化、恢复、水合、前端类型和测试。
4. 前端可以把这两个状态渲染成控制面提示，但不能写入 assistant 正文或 public body item。
5. 旧 `waiting_user` 不能继续混用审批等待；旧 `running_task` 不能继续混用安全边界等待。

### 4. 阶段边界与阶段总结分权

阶段系统必须拆成两个对象：

```text
phase_boundary     runtime/StageController 产出，说明阶段变更事实
stage_summary      model 产出，说明用户可读判断和下一阶段方向
```

`StageController` 只能决定“是否进入阶段边界、覆盖哪些工具、下一阶段是什么”，不能写用户可复制的模型正文。模型只能写 `stage_summary.text`，不能自己篡改 runtime 阶段事实。

固定链路：

```text
工具/观察/steer/pause 触发边界候选
-> StageController 产出 phase_boundary
-> prompt 要求模型基于 phase_boundary 和 observation 写 stage_summary
-> PublicProjection 将 phase_boundary 放 timeline/control，将 stage_summary 放 model-authored feedback
```

如果模型没有给出阶段总结：

1. 不伪造正文。
2. 保留 `phase_boundary` 作为 runtime timeline/control item。
3. 记录 `stage_summary_missing` diagnostics。

### 5. 阶段总结 item

建议结构：

```json
{
  "kind": "stage_summary",
  "slot": "body",
  "surface": "assistant_body",
  "source_authority": "model",
  "phase_ref": "phase:investigation:2",
  "phase_boundary_ref": "phase-boundary:...",
  "summary_source_ref": "assistant-message-item:...",
  "title": "阶段总结",
  "text": "问题定位在持久化重建层。",
  "next_step": "下一阶段修 session_timeline 并补刷新水合测试。",
  "covers_tool_refs": ["tool:read-session-timeline", "tool:read-runtime-store"],
  "collapse_after_body_feedback": true,
  "state": "done"
}
```

阶段总结出现后：

1. 被 `covers_tool_refs` 覆盖的成功工具默认折叠。
2. 失败工具、等待中工具、产物生成、测试失败保留一行摘要。
3. 折叠不删除 trace refs、artifact refs、observation refs。

## 六、固定时序

### 1. 开局

```text
用户提交
-> 前端创建 assistant turn 容器
-> UI chrome 可显示“正在思考”
-> 不写入正文，不持久化
-> 模型输出 opening_judgment 或直接 respond/ask_user/block
```

如果模型首个动作是工具：

```text
opening_judgment
-> work_action running
```

没有开局反馈时：

```text
工具仍允许执行
-> 记录 public_feedback_missing
-> 前端只显示工具动作或临时 UI chrome
-> 不合成“开始处理”
```

### 2. 工具调用

```text
model feedback
-> tool begin
-> tool waiting
-> tool observation
-> model observation_report 或 stage_summary
```

禁止：

```text
tool begin
-> generic 恢复/处理完成
```

### 3. 权限/审批

工具、文件写入、外部执行、子 agent 或高风险控制动作需要审批时，必须走 control surface，不进入正文。

固定时序：

```text
ActionPermit needs approval
-> approval_request control item
-> ActiveTurn state=waiting_approval
-> 用户 approve / deny
-> approval_decision control item
-> approve: tool begin
-> deny: tool_denied observation 回灌给模型
-> 模型说明下一步或收口
```

规则：

1. approval request 只展示用户需要知道的动作、目标、风险和选项。
2. permission profile、内部 policy、sandbox 细节不进入正文。
3. deny 不是静默失败，必须成为模型可见 observation。
4. approval 超时或取消必须产生结构化 error projection。
5. `waiting_approval` 是一等 `ActiveTurnState`，必须同步修改 `active_turn.py` enum、序列化恢复、前端类型和测试。

### 4. 工具组与阶段边界

阶段边界触发条件：

1. 从开局判断进入信息收集。
2. 从信息收集进入实施修改。
3. 从实施修改进入验证。
4. 从验证进入收口。
5. 工具失败导致改变方向。
6. 用户 steer 改变当前阶段目标。
7. 准备暂停、中断或阻塞收口。

不是每次工具返回都总结。连续工具调用时，工具反馈保持短，阶段总结在边界产生。

### 5. 子 agent

```text
模型说明为什么需要子 agent
-> subagent_spawn running
-> subagent_waiting
-> subagent_observation done/error
-> 主 agent observation_report/stage_summary
```

子 agent 不是普通工具一闪而过。父任务不能在 child still running 时显示完成。

### 6. pause / stop / interrupt

暂停按钮：

```text
UI 立即显示 pause requested
-> ActiveTurn state=waiting_safe_boundary
-> runtime 在安全边界停止下一轮模型或工具
-> 产生 pause_boundary / interrupted_boundary
```

规则：

1. UI 反馈要立即出现，但这是 control surface，不是正文。
2. 正在输出的模型正文是否截断由运行策略决定，但必须有明确 terminal event。
3. 工具无法立即停止时，显示“正在等待安全边界”，不能静默卡住。
4. 错误一定要返回结构化 error projection 和模型可见 observation。
5. `waiting_safe_boundary` 是一等 `ActiveTurnState`，必须同步修改 `active_turn.py` enum、序列化恢复、前端类型和测试。

### 7. steer

```text
用户追加指令
-> 必须携带 expected active turn id
-> ActiveTurn 校验 steerable
-> accepted: 追加到当前 task/turn
-> rejected: 返回结构化错误
```

steer 不发新 `turn_started`。后续工具、观察、阶段总结仍归属原 active turn。

拒绝条件：

1. 没有 active turn。
2. expected turn mismatch。
3. 当前 turn 非 steerable。
4. 输入为空或超限。
5. 当前处于 review/compact 等不可 steer 模式。

## 七、流式输出稳定方案

目标：不要再出现“卡在前两个字节”“工具挂到下个回答”“边流式边覆盖”的问题。

规则：

1. 正文流只处理 `assistant_text_delta` / `assistant_text_final`。
2. 正文流活跃时，工具 begin/end、stage summary、terminal 事件进入本地队列。
3. 正文流 flush 后按稳定排序键处理队列。
4. 工具 begin 前必须 flush 当前正文 controller。
5. turn complete 前必须 flush answer stream 和 stage stream。
6. 最终持久化以 source text consolidation 为准，不以零散 delta 为准。
7. 字节流只作为 transport chunk，不直接决定 public item 边界。

跨通道排序键：

```text
event_log.offset
-> public_projection.sequence
-> stream_chunk_index
-> created_at
```

排序规则：

1. 同一个 `tool_call_id` 的 begin/observation/end 必须合并为同一工具窗口状态机。
2. terminal/error 到达时必须先 flush 正文流和 deferred activity queue，再关闭 active turn。
3. 缺少排序键的事件不能抢占已有正文或工具窗口，只能进入 diagnostics 或等待重建。

前端对应：

```text
StreamController
-> active tail
-> source consolidation
-> deferred activity queue
-> final attachment merge
```

后端对应：

```text
assistant delta event
-> public_projection assistant_body only when model source
tool event
-> public_projection tool_window
terminal
-> flush and close active turn
```

## 八、终局正文权威

最终正文只能有一个 canonical source。

权威顺序：

```text
assistant_text_final / AgentMessage item
-> 已 consolidated 的 assistant source text
-> TurnComplete.last_agent_message 只补空，不覆盖已有正文
-> final_summary 只作为 timeline/summary，不改写 assistant content
```

规则：

1. `done`、`assistant_message`、terminal receipt 永远不能生成正文。
2. 如果本轮已有 item-level 正文，turn complete 只能写 terminal 状态和指标。
3. `final_summary` 不能和 `assistant_text_final` 同时写入 message content。
4. 阶段总结不覆盖最终回答；最终回答也不删除阶段总结。
5. 刷新重建时按同一权威顺序恢复，不使用“最后一个 assistant”兜底。

## 九、Prompt 配套

Prompt 必须写给 agent，而不是写给 runtime 节点。

### 1. 动作决策 prompt

推荐规则：

```text
你正在处理用户的当前请求。
你可以直接回答、请求用户补充信息、调用工具、启动子任务，或在无法继续时说明阻塞原因。

如果你需要调用工具，请先给用户一句真实、具体、简短的公开判断：
- 说明你现在已经知道什么，或需要确认什么边界；
- 不要预测工具结果；
- 不要写“开始处理”“正在处理”“处理完成”；
- 不要输出 JSON、工具协议、action_type、tool_call 或内部控制字段。

如果你只是还没有形成任何判断，不要编造开局反馈；系统会用非正文的临时状态显示等待。
```

### 2. 工具后 followup prompt

推荐规则：

```text
你已经收到工具观察。
先判断这个观察对任务意味着什么，再决定继续调用工具、进入下一阶段、请求用户补充信息或收口回答。

如果继续调用工具：
- 工具反馈只保留一句事实；
- 如果已经形成阶段结论，请输出阶段总结；
- 阶段总结要说明上一阶段结论和下一阶段方向。

不要连续裸调用工具。
不要把工具错误、控制命令或协议字段复制给用户。
```

### 3. 阶段总结 prompt

推荐规则：

```text
当你完成一个阶段、发现关键事实、改变方向、遇到失败恢复、收到用户 steer，或准备最终收口时，给出阶段总结。

阶段总结应包含：
- 本阶段结论；
- 结论依据的关键观察；
- 下一阶段要做什么，或是否准备收口；
- 仍然存在的风险或需要用户确认的边界。

阶段总结不复述完整工具日志。
```

### 4. 子 agent prompt

推荐规则：

```text
如果你启动子 agent，请说明为什么需要它处理这部分工作。
子 agent 运行期间你会收到完成通知或观察摘要。
不要假装已经知道子 agent 的结果。
不要轮询或重复启动相同子 agent。
```

## 十三、切换规则

### Shadow 阶段

允许同时产出 legacy timeline 和 envelope，但 UI 不自动优先 envelope。

要求：

- 比较 legacy 与 envelope 的 item 数、anchor、顺序。
- 发现差异只记录 diagnostics，不让 legacy 覆盖 envelope。
- `projection_mode=shadow` 时，envelope 只参与 diagnostics 和并行测试。
- envelope 没有 items、缺 anchor、或未标记 authoritative 时，不得压制 legacy display。

### Cutover 阶段

条件：

1. 后端 envelope 覆盖所有主会话 runtime event。
2. 后端明确发送 `projection_mode=authoritative`。
3. 前端 reducer 测试通过。
4. session timeline 重建测试通过。
5. E2E 固定端口验证通过。

切换：

```text
frontend display source = authoritative public_projection_envelope
legacy public_timeline_delta = debug/shadow only
```

### Rollback 阶段

只允许回滚到“envelope shadow + legacy display”，不允许恢复无 anchor 挂最新 assistant 的旧逻辑。

## 十四、删除标准

必须删除或降权的逻辑：

1. 有 `task_projection` 就跳过 `public_timeline`。
2. 找不到 anchor 时挂最近 assistant。
3. 找不到 assistant 时按 source index 向后找。
4. 把 `done/assistant_message/开始处理/处理完成` 变成公开反馈。
5. 用前端正则当主防线。
6. 工具 observation 只显示工具名，不显示结果摘要。
7. 子 agent completed 早于真实返回。
8. steer 不校验 expected active turn。
9. approval/permission 请求落入正文或普通工具状态。
10. turn complete 覆盖已有 item-level 最终正文。

允许保留的逻辑：

1. 前端 control text filter：最后防线。
2. legacy projection：shadow 对照期。
3. 临时“正在思考”：仅 UI chrome，不持久化，不写正文。

## 十五、完成定义

重构完成必须同时满足：

1. 开局没有正文时，页面只有临时 UI chrome 或工具动作，不造假。
2. 有模型公开反馈时，反馈显示在工具前。
3. 工具观察后有短反馈。
4. 阶段边界有阶段总结。
5. 阶段总结后已覆盖成功工具默认折叠。
6. task_projection 与 public_timeline 共存。
7. live 和刷新后重建顺序一致。
8. 控制命令、JSON action、工具协议不进入正文。
9. 上一轮工具不泄露到下一轮。
10. 子 agent 有等待态和返回态。
11. pause/interrupt/steer 有明确 active turn 归属。
12. 错误一定有结构化返回，不静默阻塞 agent。
13. approval/permission 有 control surface、用户决策和模型可见 observation。
14. 最终正文只有一个 canonical source，不被 terminal/done 覆盖。
15. shadow/cutover/rollback 不会因为空 envelope 丢失公开反馈。

## 十六、执行建议

建议按阶段 1 到阶段 7 一次性实施，不要再按单个症状补丁式修改。

如果实施中发现某阶段需要改变目标契约，必须暂停并更新本文，而不是在代码里临时兜底。

最优先落地顺序：

```text
ActiveTurn/Anchor 收紧
-> PublicProjectionEnvelope 单一入口
-> SessionTimeline 同构重建
-> StageSummary 一等化
-> Frontend envelope-first
-> Legacy cleanup
```

原因：

如果不先收紧 anchor 和 envelope，后续阶段总结、工具折叠、steer/pause 都会继续依赖模糊归属，最终还是会出现工具挂错回答、反馈刷新丢失、控制状态暴露等问题。

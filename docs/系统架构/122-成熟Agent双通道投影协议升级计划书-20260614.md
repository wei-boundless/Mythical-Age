## 1. 目的

本计划用于把当前 agent 投影系统从“能显示修复后的事件”升级为成熟 coding agent 风格的公开输出协议。目标不是再补一个工具卡片或正文 fallback，而是建立一条可长期维护的投影权威链：

```text
Runtime canonical event log
-> Public projection event families
-> PublicProjectionFrame
-> Frontend ordered projection ledger
-> Message surface
```

升级后的系统必须满足：

1. 正文是大模型返回给用户的 assistant body，只能来自 `assistant_text_delta` / `assistant_text_final`。
2. 工具调用是 agent 在模型决策和系统授权下产生的控制轨迹，只能来自工具生命周期事件。
3. 正文与工具轨迹在语义上分通道，在传输和展示时服从同一个公开事件时钟。
4. 前端只按 `event_offset` 合并展示，不再根据 terminal snapshot、工具 observation、中文文案或到达时间推断语义。
5. lossless 事件可追踪、可重放、不可被普通状态进度挤掉。

这份计划承接 `121-Agent投影实时输出与工具轨迹时序修复计划书-20260614.md`。121 解决的是三个直接故障；本计划解决的是协议长期稳定性，防止后续功能再次把正文、工具、commit、terminal 混成多套权威。

| 合约版本未表达双通道语义 | 现有 revision 是 `20260613-user-first`，更像 UI 主视图合约 | 升级为明确的 `dual-channel` 投影协议版本 |

| 前端 ledger 仍是 slot 聚合而非事件族模型 | body 单独保存，timeline 保存工具/status | 可以保留结构，但必须把排序、显示和测试都绑定同一事件时钟 |

## 3. Source Basis

### 3.1 本项目本地原则

本计划从现有计划和代码抽取以下约束：

- `docs/系统架构/121-Agent投影实时输出与工具轨迹时序修复计划书-20260614.md`
  - 正文来自模型可见 canonical final text。
  - 工具轨迹来自 action permit 和 execution loop 下的生命周期事件。
  - 前端按 projection frame offset 展示，不从 terminal snapshot 反推正文。
- `docs/系统架构/115-Codex参考的公开输出时序与桥接权威修复计划书-20260613.md`
  - 稳定 turn anchor 先建立。
  - completed item / commit ack 是收口权威。
  - terminal 只关闭 turn，不代替正文或提交。
  - 同一个 chat stream 只能有一个 public event clock。
- `backend/maintenance/codex_lossless_output_stream_replacement_plan_20260611.md`
  - transcript、completed item、turn completed 应归入 lossless tier。
  - presentation projection 不拥有正文生成权。
  - 工具 item lifecycle 应是 first-class，不靠语义 key 合并。
- `AGENTS.md`
  - agent 架构必须以 Claude Code、Codex 等成熟 agent 为标准。
  - 核心 runtime/workflow/tool calling/state/prompt 改动必须先写计划并等待确认。
  - 不允许以兼容或兜底为理由保留无用旧链路。

抽象成可执行设计原则：

1. 单一事实源：公开 event log 是投影唯一时序事实源。
2. 语义分权：assistant body、tool control、commit、terminal、status 各有独立事件族。
3. 传输单钟：语义可分通道，展示只能用一个 `event_offset`。
4. 输出不反推：前端不从 raw event、snapshot、terminal、observation 推断正文。
5. 失败显性化：乱序、缺 id、缺 body、缺 commit 必须变成 protocol diagnostic。
6. 清理旧链路：升级完成后删除旧 fallback 和旧测试断言，不长期双轨。

### 3.4 结论

Codex 和 Claude Code 都不是简单意义上的“两条物理通道”。成熟方案的本质是：

```text
单一有序事件体系
+ 多个语义事件族
+ lossless/best-effort 分层
+ item/tool/message id 稳定
+ presentation 被动消费
```

因此本项目采用一条公开投影流，用一个事件 offset 排序，用事件族表达 assistant body 与 tool control 的逻辑双通道。

## 4. 采用的设计方向

本计划采用：

- 所有公开事件进入同一个 chat public event clock。
- 每个事件带 `event_family/channel/source_authority/source_event_type/event_offset`。
- assistant body 与 tool control 语义分离。
- 前端以 `event_offset` 合并显示。
- lossless 事件支持 replay/gap recovery。

这样做的工程依据：

- 与 Codex / Claude Code 的成熟机制一致。
- 与本项目当前 `PublicProjectionFrame`、`ProjectionLedger`、task bridge 主干兼容。
- 能最小化协议面，最大化时序确定性。
- 便于测试：只要断言事件序列和 ledger 输出，不靠文案。

## 5. Recommended Design

### 5.1 协议版本

新增公开投影协议修订：

```text
PUBLIC_PROJECTION_CONTRACT_REVISION = "20260614-dual-channel-v1"
```

该修订表达的是协议语义升级，不只是 UI 文案升级。实施后所有新生成 `PublicProjectionFrame` 必须携带该 revision。

旧 revision 不应长期并行保留。允许在切换阶段读取历史记录，但不允许为了兼容旧 revision 新增另一套运行时主链路。

### 5.2 事件族与逻辑通道

公开事件分为五个家族：

| Event family | Channel | 事件 | 权威 | 投影职责 |
| --- | --- | --- | --- | --- |
| `assistant_body` | `body` | `assistant_text_delta`、`assistant_text_final`、`assistant_stream_repair` | `model` | 追加或定稿 assistant 正文 |
| `tool_control` | `control` | `tool_call_requested`、`tool_permission_decided`、`tool_item_started`、`tool_item_completed` | `model/runtime/tool` | 展示工具请求、授权、执行、完成 |
| `runtime_commit` | `commit` | `session_output_commit_checked/ack/failed/skipped` | `runtime` | 记录正文写回与收口状态 |
| `turn_anchor_terminal` | `terminal` | `chat_turn_bound`、`task_bridge_started`、`task_bridge_terminal`、`turn_completed` | `runtime` | 绑定 turn、bridge 和关闭本轮 |
| `status_trace` | `status` | `runtime_status`、`runtime_step_summary`、诊断类事件 | `runtime/system` | 运行状态和协议诊断，不产生正文 |

### 5.3 Lossless / Best-effort 分层

必须 lossless：

```text
assistant_text_delta
assistant_text_final
assistant_stream_repair
tool_call_requested
tool_permission_decided
tool_item_started
tool_item_completed
session_output_commit_checked
session_output_commit_ack
session_output_commit_failed
session_output_commit_skipped
chat_turn_bound
task_bridge_started
task_bridge_terminal
turn_completed
protocol_diagnostic
```

允许 best-effort：

```text
runtime_status
runtime_step_summary
ordinary progress note
non-terminal trace-only diagnostic
tool stdout/stderr chunk preview
```

注意：工具 completed 不能因为“只是控制轨迹”而 best-effort。completed 丢失会让 UI 永远显示运行中。

### 5.4 单一排序权威

统一规则：

```text
source task offset
  = 证据引用

chat public event_offset
  = 前端排序唯一权威

frontend arrival time / Date.now
  = 不参与正文与工具时序裁决
```

`source_task_event_offset`、`source_turn_event_offset`、`source_handoff_event_offset` 只能进入 trace refs 或调试字段，不能参与 UI 主排序。

### 5.5 身份与生命周期

工具生命周期必须固定四类 id：

| id | 来源 | 用途 | 禁止事项 |
| --- | --- | --- | --- |
| `tool_call_id` | 模型 `tool_call.id`，缺失时 runtime 生成一次 | 贯穿 requested/permission/started/completed | 不得与 `request_id` 随意互换 |
| `action_request_ref` | `TaskExecutionModelActionRequest.request_id` | 追踪模型动作请求 | 不得作为工具 item identity |
| `permission_decision_id` | admission/permission 层 | 证明 started/completed 已被允许 | 不得替换 `tool_call_id` |
| `tool_lifecycle_id` | runtime execution item | 表达单次执行实例 | 不得改变 requested item 的 tool identity |

assistant 正文身份：

| id | 来源 | 用途 |
| --- | --- | --- |
| `stream_ref` | assistant body stream | 聚合同一正文流 |
| `message_ref` | assistant message anchor | 绑定最终消息 |
| `body_segment_id` | 可选 | 支持未来多段正文 |

## 5A. Fixed Execution Flow

### 5A.1 普通 assistant 正文流

```text
Model gateway emits text delta
-> assistant_text_delta
-> PublicProjectionFrame(op=body_append, slot=body, source_authority=model)
-> Frontend ledger.body append
-> ChatMessage renders body block at delta event_offset

Model body finalized
-> assistant_text_final
-> PublicProjectionFrame(op=body_finalize, slot=body, source_authority=model)
-> Frontend ledger.body finalize
```

禁止：

- 从 `done.content`、`terminal.final_answer`、tool observation 中生成正文。
- 把 JSON action token 投影成用户正文。

### 5A.2 工具控制轨迹

```text
Model emits tool_call
-> tool_call_requested(tool_call_id)
-> permission/admission
-> tool_permission_decided(tool_call_id, permission_decision_id)
-> execution starts
-> tool_item_started(tool_call_id, permission_decision_id, tool_lifecycle_id)
-> execution completes
-> tool_item_completed(tool_call_id, permission_decision_id, tool_lifecycle_id)
-> ProjectionLifecycleState validates order
-> Frontend timeline renders tool window/control track by event_offset
```

禁止：

- completed 缺 started 时静默修复。
- started/completed 用 `request_id` 替换 `tool_call_id`。
- 工具 observation 原文进入 assistant body。

### 5A.3 Task bridge 到主会话

```text
chat turn starts
-> chat_turn_bound creates immutable public anchor
-> request_task_run schedules task
-> task_bridge_started binds task_run to turn/message
-> task events bridged into chat public event log
-> each bridged event receives chat event_offset
-> source task event id/offset recorded as trace evidence
-> task_bridge_terminal marks bridge closed
-> turn_completed closes chat stream
```

禁止：

- 每个 task event 重新猜 `turn_id/message_ref`。
- 用 task offset 替代 chat public offset。

### 5A.4 Commit 与 terminal

```text
assistant_text_final
-> session_output_commit_checked
-> session_output_commit_ack | failed | skipped
-> scope_retire transient status/tool progress
-> turn_completed
```

禁止：

- terminal 负责生成正文。
- commit ack 之前把正文当作已持久化。
- commit failed 被隐藏到 trace-only。

## 6. Data Model Changes

### 6.1 PublicProjectionFrame 增补字段

在保持现有 frame 主体不重写的前提下，新增或锁定以下字段：

```ts
type PublicProjectionFrame = {
  authority: "harness.public_projection";
  contract_revision: "20260614-dual-channel-v1";
  frame_id: string;
  source_event_id: string;
  source_event_type: string;
  event_offset: number;
  event_family:
    | "assistant_body"
    | "tool_control"
    | "runtime_commit"
    | "turn_anchor_terminal"
    | "status_trace";
  channel: "body" | "control" | "commit" | "terminal" | "status";
  lossless: boolean;
  sequence: number;
  anchor: {
    session_id: string;
    turn_id: string;
    message_id?: string;
    task_run_id?: string;
    run_id?: string;
    turn_run_id?: string;
  };
  op: string;
  slot: string;
  source_authority: "model" | "tool" | "runtime" | "system";
  main_visibility: "visible_live" | "visible_final" | "pinned" | "trace_only" | "hidden";
  retention: "transient" | "final" | "pinned_until_resolved" | "trace";
}
```

其中 `event_family/channel/lossless` 是协议可读性和测试稳定性的关键，不应只靠 `source_event_type` 隐式推断。

### 6.2 ProjectionLedger 目标结构

保留现有结构，但锁定语义：

```ts
type ProjectionLedger = {
  body: {
    text: string;
    stream_state: "streaming" | "finalized" | "committed";
    source_offsets: number[];
  };
  timeline: PublicProjectionItem[];
  currentAction?: PublicProjectionItem;
  pinned: PublicProjectionItem[];
  status: PublicProjectionItem[];
  trace: PublicProjectionItem[];
  commit: { state: "none" | "checked" | "committed" | "failed" | "skipped" };
}
```

约束：

- `timeline` 是正文以外可见控制轨迹的时间线。
- `bodyEventOffset` 是 body 块参与跨块排序的 offset。
- `timeline` 与 `bodyEventOffset` 使用同一 `event_offset` 域。
- 不允许 timeline 反向写 body。

### 6.3 需要移除或降级的旧数据入口

升级实施时必须审计并删除或降级：

- 从 terminal/snapshot/final_answer 合成 assistant body 的路径。
- `done.content` 直接进入可见正文的路径。
- `answer_candidate` 与 `assistant_text_final` 并列成为正文权威的路径。
- 工具 completed 阶段用 permission/admission id 静默修正 tool id 的路径。
- 前端按中文文案、工具名、semantic key 合并主视图 item 的路径。

## 6A. Early Fixed Decisions

1. 公开投影统一使用现有单一 SSE 事件流。
2. 升级目标协议名固定为 `20260614-dual-channel-v1`。
3. `event_offset` 的唯一排序权威来自 chat public event log。
4. `tool_call_id` 是工具身份权威；`request_id` 只保留为 action request trace。
5. `assistant_text_final` 必须早于 commit checked/ack/terminal。
6. task bridge 必须先创建不可变 public anchor，再桥接 task events。
7. 协议诊断必须进入投影 trace 或 pinned failure，不能静默吞掉。
8. 本计划确认后执行，不在计划阶段改业务代码。

## 10A. Migration and Cutover

### 10A.1 迁移方式

采用一次主链路切换，不长期双轨。

允许：

- 历史消息读取旧 revision。
- 测试中构造旧 fixture 用于证明不会污染新链路。

不允许：

- 新 runtime 同时生成旧 projection envelope 和新 frame 两套主输出。
- 前端为了旧 frame 继续保留 terminal/body fallback。

### 10A.2 Cutover 条件

满足以下条件后切换：

1. 后端 contract 测试通过。
2. task executor 正文和工具生命周期测试通过。
3. 前端 reducer/ChatMessage 排序测试通过。
4. 固定端口端到端场景通过。
5. `rg` 搜索确认旧正文入口和 id 兼容修复不在主链路。

### 10A.3 Rollback 规则

如果实施中发现新协议导致主会话完全不可用：

- 只能回滚本次未完成的代码改动。
- 不允许新增第三套 fallback。
- 回滚后保留协议计划，重新修正 Phase 1-2 的设计偏差。

如果只是某个边界测试失败：

- 暂停当前 phase。
- 先修结构性原因。
- 不通过降低断言、跳过测试、mock 核心链路制造通过。

### 10A.4 最终清理条件

以下内容必须清理：

- terminal/snapshot 合成 body。
- `done.content` 作为正文权威。
- `answer_candidate` 与 `assistant_text_final` 并列作为最终正文。
- tool lifecycle id 静默修复。
- 前端按文案或 semantic key 合并主轨迹。
- 旧测试中鼓励 fallback 的断言。

## 11. Prohibited Shortcuts

实施时禁止：

1. 排序问题必须在同一公开事件时钟内解决。
2. 用前端到达时间修正文/工具顺序。
3. 从 terminal snapshot 或 final_answer 反推正文。
4. 把工具 observation/stdout 当 assistant 正文显示。
5. 把 `request_id` 当 `tool_call_id` 的兼容替代长期保留。
6. 对缺 started 的 completed 做静默补链。
7. 用中文文案判断事件类型、显示层级或收口。
8. 为了过测试删除失败用例、降低断言或 mock 掉核心事件链。
9. 保留旧主链路并称为兼容。
10. 联调时临时切到 3001/8004 等随机端口。

## 12. Expected Outcome

升级完成后：

- 用户看到的 assistant 正文与工具轨迹会严格按真实 agent 时序展示。

# 图任务独立工作区与主会话隔离升级计划书

日期：2026-06-09
状态：待确认执行

## 1. 问题判断

当前图任务的项目、会话和运行监控会串入主会话页，本质不是单点 UI 泄漏，而是“图任务实例归属权”没有被单独建模。

已经确认的冲突点：

1. 前端主页面在多个工作区全局挂载 `TaskGraphRunInteractionDock`，主会话页也会显示图运行浮窗。
2. `TaskGraphPublishRunPage.startRun()` 仍强制读取 `graphConfig.task_environment_id`，再通过 `resolveTaskEnvironmentSession()` 创建或复用任务环境会话。
3. 后端 `backend/api/orchestration.py` 已经通过 `_graph_instance_scope()` 清空 `task_environment_id`，说明图运行实例已经在向“图独立、项目作用域、环境不绑定图”的方向迁移。
4. `backend/harness/graph/runtime.py` 的 runtime scope 会剥离 `task_environment_id/environment_id`，进一步证明环境只能作为节点 agent 装配信息，不能成为图实例作用域。
5. `RunMonitorController.navigateSignal()` 打开图任务信号时会调用 `applySelectedSessionShell()`，把 graph session 设为当前普通会话，这是图任务会话串入主会话页的直接入口。
6. 前端 `SessionPoolKey` 只有 `main-chat` 和 `task_environment:*`，没有 graph task session pool，导致 graph session 没有自己的前端归属池。
7. 图编辑保存链路仍把 `task_environment_id/environment_id` 写入 graph metadata/runtime/context，和“图不绑定环境，环境给 agent 装配”的目标冲突。

## 2. 目标架构

图任务需要一个专门的实例工作区，建议命名为 `graph_task` session scope：

```text
workspace_view: "graph_task"
task_environment_id: ""
project_id: "<project id, optional by graph contract>"
```

权威边界：

| 层级 | 当前问题 | 目标权威 |
| --- | --- | --- |
| 图定义 | 被环境字段污染 | 只描述拓扑、节点、边、资源、契约，不绑定环境 |
| 节点 agent | 部分从图环境继承 | 节点自己的 agent/runtime 装配决定环境 |
| 图运行实例 | 借用任务环境会话 | 由 graph start API 创建 `graph_task` session |
| 主会话页 | 可能接管 graph session | 只展示 `chat/project` 普通会话，不展示 graph session |
| 图任务工作区 | 现在只是 task-system 子页 | 负责打开、绑定、监控、续跑 graph session |
| 运行监控 | 图任务信号会切当前 session | 图任务信号只进入图任务工作区，不切主会话 |

## 3. 实施计划

### 阶段一：定义 graph task scope

1. 在前端 store 类型中增加 graph task session pool：
   - `SessionPoolKey = "main-chat" | "graph_task:<project_id>:<graph_run_id|graph_id>" | task_environment...`
2. 增加工具函数：
   - `isGraphTaskScope(scope)`
   - `graphTaskSessionScope(projectId)`
   - `sessionPoolKeyForScope()` 识别 `workspace_view === "graph_task"`。
3. `visibleMainChatSessions()` 排除：
   - `workspace_view === "graph_task"`
   - `task_binding.kind === "task_graph"`
   - 带 `graph_run_id/graph_harness_config_id` 的 session。

### 阶段二：后端 start API 统一图任务实例作用域

1. `backend/api/orchestration.py`
   - `_validated_graph_request_scope()` 接受 `graph_task`。
   - `_graph_instance_scope()` 输出 `workspace_view: "graph_task"`、`task_environment_id: ""`、`project_id`。
   - `TaskGraphRunStartRequest.session_id` 暂时保留为 launch session，但只作为发起来源，不绑定 graph run。
2. `GraphRun`、`TaskRun diagnostics`、`session_scope_key` 全部记录 graph task scope。
3. `run-until-idle/monitor/resume` 的 scope 校验接受 graph task scope，禁止前端继续传 `task_environment_id` 来匹配图实例。

### 阶段三：前端启动运行不再解析任务环境会话

1. `TaskGraphPublishRunPage.startRun()`：
   - 删除对 `graphConfig.task_environment_id` 的强制要求。
   - 删除 `resolveTaskEnvironmentSession()` 作为图运行会话创建入口。
   - 使用当前普通 session 作为 `launch_session_id`，由后端创建独立 graph session。
   - 发送 `session_scope: { workspace_view: "graph_task", task_environment_id: "", project_id }`。
2. 成功后绑定：
   - `session_id = result.graph_session_id`
   - `session_scope = graph_task scope`
   - `taskGraphMonitorBinding` 只进入图任务工作区状态，不写入 `currentSessionId`。

### 阶段四：运行监控导航隔离

1. `RunMonitorController.navigateSignal()`：
   - 图任务信号不再调用 `applySelectedSessionShell()`。
   - 只设置：
     - `activeWorkspaceView: "task-system"`
     - `taskGraphWorkspaceTarget`
     - `taskGraphMonitorBinding`
2. `loadSignalDetail()` 和 `continueBoundGraphRun()` 使用 binding 中的 graph task scope。
3. 主会话的 `messages/currentSessionId/conversationActiveEnvironment` 不因图任务信号变化。

### 阶段五：图编辑保存去除 graph 级环境绑定

1. `GraphTaskWorkspace.saveTaskGraphStack()` 不再要求 `draftEnvironmentId`。
2. 不再向 graph metadata/runtime_policy/context_policy 写入：
   - `task_environment_id`
   - `environment_id`
3. 保留节点 agent 的环境配置入口；如果某个节点需要环境，由节点配置系统负责。
4. 任务系统图列表从“环境下的图”迁移为“图任务库/项目图任务”视图，避免用户误以为图属于环境。

### 阶段六：UI 展示收口

1. `TaskGraphRunInteractionDock` 默认只在 `task-system` 的图任务工作区显示。
2. 主会话页不展示 graph run dock，不把 graph session 混进会话列表。
3. 图任务工作区显示当前 graph session、graph run、project scope 和节点运行状态。

   - graph run 的 `task_environment_id == ""`。
2. monitor/resume/run-until-idle 只接受 graph task scope。
3. project-scoped graph run 不再回填项目环境 id。

4. 主会话页不渲染图任务浮窗。
5. 图保存不写 graph 级环境字段。

## 5. 风险与处理

1. 旧测试仍断言 `workspace_view: "task_environment"`：按新架构更新测试，不保留旧兼容链路。
2. 旧 graph config 里可能还有环境字段：编译/保存时清理 graph 级字段，节点级环境配置保留。
3. 运行中的旧 graph run 可能仍带旧 scope：不在新代码中继续生产旧 scope；如需恢复旧运行，优先通过一次性迁移或重新创建 graph run 解决。
4. 当前 100 章写作任务的续跑：修改后只要 graph run/session binding 仍可由 `graph_run_id` 找到，就可以继续；若旧 scope 冲突，按图任务迁移脚本修正 scope，不破坏已生成产物。

## 6. 交付标准

完成后必须满足：

1. 图定义不绑定环境。
2. 图运行实例使用独立 `graph_task` 工作区 session。
3. 主会话页不展示、不恢复、不切换 graph session。
4. 图任务工作区可以启动、绑定、监控、续跑 graph run。
5. 节点 agent 仍可单独装配环境。
6. 相关后端与前端测试真实通过。

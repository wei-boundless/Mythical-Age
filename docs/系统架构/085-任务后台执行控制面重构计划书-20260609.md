# 任务后台执行控制面重构计划书

日期：2026-06-09

## 范围边界

本计划重构任务执行入口的运行形态：所有长任务、模型调用、工具执行、GraphRun 节点推进从 HTTP 同步请求中迁出，统一交给后台执行面。前台只负责调度、授权、控制和监控。

允许改动范围：

- `backend/api/orchestration.py`
- `backend/api/orchestration_harness.py`
- `backend/harness/graph/*`
- `backend/harness/graph_harness.py`
- `backend/harness/entrypoint/runtime_facade.py`
- `backend/harness/loop/task_executor_controller.py`
- `backend/harness/loop/task_executor.py`
- `backend/harness/runtime/run_monitor/*`
- `frontend/src/lib/api.ts`
- `frontend/src/lib/run-monitor/controller.ts`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/components/workspace/views/task-system/*`
- `backend/tests/*graph*`
- `backend/tests/*runtime_monitor*`
- `frontend/src/lib/store/runtime.test.ts`

禁止事项：

- 不把旧同步执行链路包装成新的“后台”外壳后继续作为主链路。
- 不让前端等待模型、工具、GraphRun 节点执行完成。
- 不让 `run-until-idle` 继续承担用户交互主入口。
- 不用随机端口绕过阻塞问题。
- 不通过降低测试断言、跳过测试或 mock 核心执行来制造通过。

## 当前问题

任务系统已经有部分后台能力，但不是所有入口都接入后台执行面。

当前事实：

- `TaskExecutorController.schedule(...)` 已经能通过 `spawn_background_task(...)` 调度单个 TaskRun 后台执行。
- Runtime monitor / SSE 已经能承担前端观测面。
- GraphRun 状态模型已经能保存 active work orders、ready nodes、completed nodes 和 checkpoints。

当前冲突：

- `/orchestration/harness/graph-runs/{graph_run_id}/run-until-idle` 在 HTTP 请求里直接 `await graph_harness.run_until_idle(...)`。
- `GraphRunRunner.run_until_idle(...)` 对 `active_orders` 使用串行 `for order in active_orders`。
- `RuntimeFacade.execute_graph_agent_work_order(...)` 创建 graph node TaskRun 后直接 `await execute_task_run(...)`，没有走 `TaskExecutorController.schedule(...)`。
- 用户或 CLI 一旦以 `max_runtime_seconds=240` 调用同步入口，单个请求可能长时间占住 `8003`，导致 catalog、monitor、页面请求排队。

所以问题不是“任务系统没有并发能力”，而是 GraphRun 主执行入口没有把执行权交给已有后台调度器。

## 目标架构

目标架构拆成三层：

```text
调度控制面
-> 后台执行面
-> 运行观测面
```

### 调度控制面

只负责：

- 校验 graph/task/session scope。
- 校验权限、人工确认、暂停/恢复/停止条件。
- 创建或恢复 TaskRun、GraphRun、GraphNodeWorkOrder。
- 将可执行工作提交给后台执行面。
- 立即返回 `accepted`、`run_id`、`monitor_url`、`submission_id`。

禁止：

- 等待模型响应。
- 执行工具。
- 循环推进整张图。
- 等待节点完成。

### 后台执行面

只负责：

- 接收 TaskRun 或 GraphNodeWorkOrder 调度。
- 执行模型轮次、工具调用、节点工作。
- 写事件日志和状态索引。
- 将 graph node 的结果提交给 GraphLoop。
- 根据 GraphLoop 新状态继续调度后续 ready nodes。
- 在失败、取消、重启后写入可恢复状态。

### 运行观测面

只负责：

- 从 event log、state index、GraphLoop checkpoint 投影 monitor。
- 通过 SSE 推送 snapshot、event delta、heartbeat。
- 为前端和 CLI 提供 watch/monitor 查询。

禁止：

- 观测面不得重新决定是否执行任务。
- 观测面不得暗中恢复、派发或跳过节点。

## 权责迁移图

```text
用户点击继续
-> API 校验 GraphRun + SessionScope
-> GraphLoop resume / dispatch ready nodes
-> GraphNodeWorkOrder 写入 active_work_orders
-> GraphBackgroundSupervisor 提交 node TaskRun
-> TaskExecutorController.schedule(...)
-> 后台 TaskRun executor 执行模型和工具
-> 生成 NodeResultEnvelope
-> GraphLoop.accept_node_result(...)
-> 产生新的 ready nodes
-> Supervisor 继续调度或进入 idle / waiting / terminal
-> RuntimeMonitor / SSE 向前端展示
```

## 设计准则

1. 前台 API 必须短事务化。
   - 目标响应时间是毫秒到低秒级。
   - 任何可能超过一次普通读接口耗时的执行，都必须提交后台。

2. 后台执行必须可去重。
   - 以 `task_run_id`、`graph_run_id`、`work_order_id` 作为 named background task 或 lease key。
   - 同一个 GraphRun 不允许多个 supervisor 竞争同一 work order。

3. 并发不是无限并发。
   - GraphRun 级别需要 max active node workers。
   - 工具层继续遵守 tool batch planner 的并行安全规则。
   - 写文件、破坏性操作、共享资源必须持有资源锁。

4. 监控是唯一前端反馈源。
   - 前端不从 submit API 等最终结果。
   - submit API 只返回提交回执。
   - 页面状态来自 monitor/SSE。

5. 同步执行入口不能继续作为主链路。
   - `run-until-idle` 保留为 debug/test-only 或短期迁移入口。
   - UI、CLI 长任务、自动推进必须使用 submit/watch 模式。

## 实施阶段

### 阶段一：入口收口

目标：所有前端和 CLI 用户交互入口从同步执行切到 submit/watch。

改动：

- `frontend/src/lib/api.ts`
  - 增加或统一 `submitGraphRunUntilIdle(...)` / `submitTaskRunExecution(...)`。
  - 返回类型统一为 `accepted`、`background_started`、`already_running`、`monitor_url`。

- `frontend/src/lib/run-monitor/controller.ts`
  - `continueBoundGraphRun`、自动推进入口只提交后台任务。
  - 不等待 GraphRun 完整完成。

- `frontend/src/lib/store/runtime.ts`
  - `resumeTaskGraphRun` 改为后台提交。

- `frontend/src/components/workspace/views/task-system/*`
  - 所有“继续运行/自动推进/运行到空闲”按钮改为 submit/watch。

验收：

- 前端点击继续后 API 立即返回。
- 页面通过 runtime monitor 更新状态。
- 不再出现用户交互调用同步 `/run-until-idle`。

### 阶段二：GraphRun 后台 supervisor

目标：GraphRun 由后台 supervisor 管理，而不是由 HTTP 请求调用 `GraphRunRunner.run_until_idle(...)`。

新增模块建议：

- `backend/harness/graph/background_supervisor.py`
  - `GraphRunBackgroundSupervisor`
  - 负责 resume、dispatch、schedule active work orders、处理节点完成后的下一轮调度。

- `backend/harness/graph/work_order_scheduler.py`
  - `GraphWorkOrderScheduler`
  - 将 `GraphNodeWorkOrder` 转换为 graph node TaskRun，并调用 `TaskExecutorController.schedule(...)`。

- `backend/harness/graph/node_result_bridge.py`
  - `GraphNodeResultBridge`
  - 从完成的 graph node TaskRun 里提取 `NodeResultEnvelope`，调用 `GraphLoop.accept_node_result(...)`。

改动：

- `backend/harness/entrypoint/runtime_facade.py`
  - `execute_graph_agent_work_order(...)` 不再作为主链路执行节点。
  - 新增 schedule graph node work order 的后台入口。

- `backend/harness/graph/runner.py`
  - 从主链路降级为 debug/test runner。
  - 或拆出纯 dispatch/reconcile 工具函数，避免继续拥有执行权。

- `backend/harness/graph_harness.py`
  - 暴露 `submit_run_until_idle(...)` 或 `schedule_graph_run(...)`。
  - 提交给 supervisor 后立即返回。

验收：

- GraphRun 有 active work orders 时，后台能调度对应 node TaskRun。
- node TaskRun 完成后 GraphLoop 状态能推进。
- 多个 ready nodes 在资源不冲突时可并发调度。
- 同一个 work order 重复提交不会重复执行。

### 阶段三：统一 TaskRun 后台执行入口

目标：所有单 TaskRun 执行入口也只提交后台，不在 HTTP 请求中执行。

改动：

- `backend/api/orchestration_harness.py`
  - `task-runs/{task_run_id}/execute` 保持短事务，返回 schedule receipt。
  - 审批、恢复、继续运行统一调用 schedule/recover。

- `backend/harness/loop/task_executor_controller.py`
  - 强化 named task 去重。
  - 明确 lease 状态：`scheduled`、`running`、`lost`、`waiting_executor`。
  - 增加后台任务异常统一落事件。

验收：

- TaskRun execute API 不等待模型和工具执行。
- 重复点击继续只返回 already_running，不产生第二个 executor。
- runtime restart 后 waiting_executor 可以恢复。

### 阶段四：阻塞执行隔离

目标：后台任务即使执行阻塞型模型 SDK、文件扫描或工具，也不能拖住 API event loop。

改动方向：

- 将已知同步阻塞模型调用迁移到 async 客户端，或通过 worker thread/process 执行。
- 对文件系统重扫描、重 catalog 构建、重 prompt 装配建立后台/缓存边界。
- 给后台执行器增加并发上限和资源锁。

验收：

- 后台长任务运行时，`/orchestration/agents?include_options=false`、`/runtime-monitor`、`/sessions/*/timeline` 仍能响应。
- 长任务不会让 `8003` 全局无首字节超时。

### 阶段五：旧同步链路清理

目标：旧同步执行入口不再被主链路引用。

处理：

- `run-until-idle` 标记为 debug/test-only。
- 前端、CLI、自动推进、健康恢复命令全部迁移到 submit/watch。
- 删除保护旧同步行为的测试，替换为后台提交和 monitor 推进测试。

验收：

- `rg "runGraphRunUntilIdle|run-until-idle"` 只剩 debug/test 或兼容边界。
- 主 UI 和 CLI 长任务路径不再直接调用同步执行。

## 风险与处理

### 风险一：后台任务仍在 event loop 内阻塞

处理：

- 第一阶段只解决 HTTP 同步等待。
- 第四阶段专门处理阻塞模型/工具隔离。

### 风险二：GraphLoop 与 TaskRun 状态双写不一致

处理：

- GraphLoop 只接受 `NodeResultEnvelope`。
- TaskRun executor 只产出节点结果，不直接改 GraphLoop 拓扑。
- node result bridge 是唯一回填入口。

### 风险三：重复提交导致重复节点执行

处理：

- work order scheduler 使用 `graph_run_id + work_order_id` 去重。
- TaskRun diagnostics 写入 `graph_work_order_id`。
- 已运行、已完成、已取消的 work order 不重复 schedule。

### 风险四：旧同步测试保护错误行为

处理：

- 保留必要 debug runner 测试。
- 主链路测试全部改为 submit/watch/monitor。
- 删除保护旧 UI 等待行为的测试。

## 完成标准

- 用户可启动多个长任务，前端不等待执行完成。
- 长任务运行时，`8003` 普通读接口不再整体被占住。
- GraphRun 节点执行通过后台 TaskRun 调度。
- runtime monitor 能完整展示 submitted、scheduled、running、waiting、completed、failed。
- 前端所有继续/自动推进入口都使用 submit/watch。
- 同步 `run-until-idle` 不再是 UI/CLI 主链路。

剩余边界：

- 当前已经移除了 HTTP 请求等待 GraphRun 完成的主链路，但后台 work-order 协程内部仍运行在同一个 runtime host event loop。若模型、工具或文件操作存在同步阻塞，仍可能拖慢同进程轻量读接口。
- 下一阶段应把阻塞型模型/工具执行隔离到 async client、thread pool、process worker 或独立 worker service，并让 runtime monitor 只观察 worker receipt。

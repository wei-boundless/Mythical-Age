# Agent 动作契约逐项对照审查报告

日期：2026-06-10

## 0. 审查结论

控制命令泄露绝对不允许。动作契约必须是闭合链路：

```text
prompt/contract 暴露动作
-> model 返回动作
-> protocol parser 识别动作
-> validator 校验必填字段
-> admission 准入
-> dispatcher 有唯一执行分支
-> executor 产出 observation/status/body
-> output boundary 裁决是否可作为正文
-> SSE/store/UI 按 surface 展示
```

本轮逐项对照后，发现三类高风险：

1. `active_work_control` 裸 JSON 泄露是 P0：控制形状不得落入正文 fallback。
2. `request_registered_engagement` 是 P0 契约不闭合：它出现在 schema/prompt/admission 中，但 single-agent final dispatch 没有执行分支。
3. `single_agent_turn` final dispatch 不是 exhaustive：任何已解析但未处理的 action type 都可能继续落入普通 content fallback。这个结构必须改成穷尽分发，未处理动作一律 protocol error。

## 1. 动作契约硬规则

1. `respond` 是唯一可以直接产生 assistant body 的模型动作。
2. `tool_call` 只能产生 tool window / observation / follow-up，不能产生 assistant body。
3. `request_task_run` 只能产生 task_control/status/task projection，不能产生 assistant body。
4. `request_registered_engagement` 如果保留，必须有明确执行分支；否则不得暴露给模型。
5. `active_work_control` 只能产生 active-work control observation/status，不能产生 assistant body。
6. `ask_user` 是控制/交互请求，问题应出现在 control surface，不应作为普通 assistant body 进入长期记忆。
7. `block` 是阻塞状态，说明可见，但 channel 必须是 blocked/progress-only，不是 stable answer。
8. parser 只要看到动作形状，就必须 normalize 或 invalid，不能退回 no-action。
9. dispatcher 必须穷尽处理所有 allowed action；未处理动作必须 fail-closed。
10. output boundary 和 frontend store 都必须按内容再防一次，不能只相信 metadata。

## 2. 动作总表

| Action | Contract 暴露 | Parser 校验 | Admission | Dispatcher | 可见 surface | 正文/记忆策略 | 风险结论 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `respond` | 支持 | 要求 `final_answer` | 默认 allow | 有分支 | assistant body | 需 canonical boundary | 基本闭合 |
| `ask_user` | 支持 | 要求 `user_question` | 默认 allow | 有分支 | control/waiting_user | progress_only/debug-only | 需要确保问题投影可见但不进 body |
| `block` | 支持 | 要求 `blocking_reason` | 默认 allow | 有分支 | blocked/status | progress_only/debug-only | 基本闭合 |
| `tool_call` | 支持 | 要求 tool name/args | 工具权限检查 | 有分支 | tool window/observation | 不写 body | 基本闭合 |
| `request_task_run` | 支持 | 要求 `task_contract_seed` | task lifecycle policy | 有分支 | task_control/task projection | progress_only/debug-only | 基本闭合，但开局文案需防 body 化 |
| `request_registered_engagement` | 多处暴露 | 要求 `plan_id` | 有 admission | **无 single-agent dispatch** | 不闭合 | 不闭合 | **P0，必须移除或实现** |
| `active_work_control` | 支持 | 只要求 action，relation 校验后置 | active-work policy | tool-loop 有分支；final 判 unreachable | control/status/observation | 禁止 body | **P0，需强 fail-closed** |

## 3. 逐项审查

### 3.1 `respond`

证据：

- `backend/harness/loop/model_action_protocol.py:143` 要求 `final_answer_required_for_respond`。
- `backend/harness/loop/single_agent_turn.py` final dispatch 中对 `respond` 有明确分支。
- `_commit_final_message` 会调用 `canonical_output_decision_for_final_text`。

正确契约：

- `respond.final_answer` 是模型可见正文候选。
- 但它不是自动稳定答案，必须经过 output boundary。

风险：

- 如果 `final_answer` 包含控制 JSON、tool protocol、DSML 或伪工具调用，必须被 `boundary.py` 拦截。

结论：基本闭合。需要保留输出边界测试。

### 3.2 `ask_user`

证据：

- `model_action_protocol.py:145` 要求 `user_question_required_for_ask_user`。
- single-agent final dispatch 对 `ask_user` 以 `answer_channel="ask_user"` commit。
- `frontend/src/lib/store/assistantContentVisibility.ts` 将 `ask_user` 归入控制 channel。
- public timeline 中 `waiting_user` 是 control item。

正确契约：

- 问题应可见，但属于 control surface，不是 assistant body。
- 不应被长期记忆当作 stable answer。

风险：

- 如果 UI 只隐藏 `ask_user` body，但 control projection 没渲染，会造成用户看不到问题。

结论：链路基本正确，但测试必须覆盖“ask_user 不进正文，但问题在 control surface 可见”。

### 3.3 `block`

证据：

- `model_action_protocol.py:147` 要求 `blocking_reason_required_for_block`。
- single-agent final dispatch 对 `block` 以 `answer_channel="blocked"` commit。
- output boundary 将 `blocked` 视为 debug-only/progress-only channel。

正确契约：

- 阻塞说明可以给用户看，但不是 stable answer。
- 不能写入长期记忆为任务结论。

风险：

- block 文案中如果含内部协议，应继续被 output boundary 清理。

结论：基本闭合。

### 3.4 `tool_call`

证据：

- `model_action_protocol.py:149` 起要求 tool name 和 args。
- `admission.py:63` 起检查工具是否在 runtime assembly、是否受 plan mode/权限限制。
- single-agent tool loop 有 tool execution / observation / follow-up。
- frontend tool window/public timeline 负责展示工具活动。

正确契约：

- 工具调用不是正文。
- 工具结果是 observation，必须回给模型或投影为 tool window。
- 工具 args/raw output 不应进入 assistant body。

风险：

- output boundary 已识别 `<tool_call>`、DSML、tool_calls 等协议，但前端 live store 仍应按内容再防一次。

结论：基本闭合。

### 3.5 `request_task_run`

证据：

- `model_action_protocol.py:156` 要求 `task_contract_seed`。
- `admission.py:125` 起检查 task lifecycle policy。
- single-agent final dispatch 对 `request_task_run` 调用 `start_task_from_action_request`。
- task lifecycle 以 `answer_channel="task_control"` 产出开局/调度反馈。
- frontend `assistantContentVisibility.ts` 将 `task_control` 归入控制 channel。

正确契约：

- 这是任务生命周期动作，不是 assistant body。
- “我会开始处理...” 这类开局反馈如果存在，也应是 task_control/status，不是 stable answer。
- UI 应显示任务投影、tool/activity，而不是把 handoff receipt 当正文。

风险：

- `backend/harness/loop/task_lifecycle.py:791` 仍存在“我会开始处理：{goal}”文案。只要 channel 是 `task_control` 且前端不作为正文显示，可以接受；但它绝不能成为 stable memory。

结论：主链基本闭合，文案 surface 需继续治理。

### 3.6 `request_registered_engagement`

证据：

- `ModelActionType` 包含 `request_registered_engagement`。
- `model_action_protocol.py:158` 起要求 `plan_id`。
- `admission.py:145` 起对该 action 有准入检查。
- `compiler.py:1422`、`:1462`、`:1751` 允许 action types 包含 `request_registered_engagement`。
- `compiler.py:1966` 的 model action schema 包含 `request_registered_engagement`。
- `compiler.py:3765` 还有 prompt 侧说明。
- 但 `backend/harness/loop/single_agent_turn.py` final dispatch 只处理 `respond/request_task_run/block/ask_user/active_work_control`，没有 `request_registered_engagement` 分支。

失败场景：

1. 某个 observation follow-up packet 允许 `request_registered_engagement`。
2. 模型按 schema 返回该 action。
3. parser 解析成功，admission 允许。
4. dispatcher 没有分支。
5. 代码继续走到后面的 `content = response.content` fallback。
6. action JSON 或相关控制内容可能成为 conversation 正文。

这是 P0。它和 active_work_control 泄露是同一类结构性问题：动作契约存在，但执行链不闭合，最后 fallback 兜成正文。

修复选择：

- 如果当前产品不需要它：从 `ModelActionType`、compiler allowed actions、schema、prompt rules、public progress、admission 中删除或禁用。
- 如果需要它：实现 single-agent dispatch，明确 engagement plan 如何转成 task lifecycle，不允许落入 fallback。

短期安全线：任何 parsed action type 未被 dispatcher 穷尽处理时，直接 protocol error。

### 3.7 `active_work_control`

证据：

- `compiler.py:2213` 暴露 active_work_control。
- `compiler.py:2216` 声明 required fields: `action`, `relation_to_current_work`。
- `model_action_protocol.py:162` 起仅校验 active_work action 是否存在。
- `active_work.py:189` 才校验 relation 必须是 `current_work`。
- `single_agent_turn.py` tool-loop active_work_control 分支会执行控制并生成 observation/status。
- final 阶段若 action_request 是 active_work_control，会报 `active_work_control_final_dispatch_unreachable`。
- 但裸 JSON 曾经没有被识别为 action_request，导致落入 fallback。

风险：

- 协议 schema 说 relation required，但 validator 没要求 relation 字段存在。
- 裸 active-work JSON 必须 normalize 或 invalid。
- action-only 控制 JSON 也应 invalid，不能 fallback。
- 前端 live stream 仍缺内容级过滤。

结论：当前修复方向正确，但还没达到成熟闭合。

## 4. 跨层不一致清单

### P0：dispatcher 非穷尽

`single_agent_turn` 已解析 action 后，必须穷尽所有 action types。当前 `request_registered_engagement` 没有分支，存在 fallback 风险。

目标：

```python
match action_request.action_type:
    case "respond": ...
    case "request_task_run": ...
    case "request_registered_engagement": ...
    case "ask_user": ...
    case "block": ...
    case "active_work_control": ...
    case "tool_call": ...
    case _:
        protocol_error(...)
```

没有实现的 action 不允许出现在 allowed actions。

### P0：控制形状 fallback 到正文

任何 JSON action-like/control-like 内容都不能进入：

- `answer_channel="conversation"`
- `assistant_text_delta`
- `assistant_text_final`
- `assistant_stream_repair`
- session public messages
- ChatMessage copy affordance

### P1：schema 与 validator 不一致

`compiler.py` 对 `active_work_control` 声明 required `relation_to_current_work`，但 `model_action_protocol.py` 只要求 action。应统一：

- 协议层要求字段存在。
- 语义层判断字段值是否允许。

### P1：旧 schema 与新 output contract 并存

`model_action_request_schema` 包含 `request_registered_engagement` 但不包含 `active_work_control`；新的 `_single_agent_turn_output_contract` 包含 `active_work_control` 但 control action list 不含 `request_registered_engagement`。

这说明动作契约有两套来源。必须统一为一个 authority，否则 prompt 和 parser 会继续漂移。

### P1：native control action 兼容路径应移除或显式禁用

`compiler.py` 明确 `control_actions_exposed_as_native_tools=False`，但 `single_agent_turn.py` 仍有 `_CONTROL_NATIVE_TOOL_NAMES` 和 native control parser，可把 native tool call 名称 `ask_user/block/request_task_run/active_work_control` 转成控制动作。

如果 provider 在未绑定工具时返回这些 tool calls，parser 仍可能接受。成熟方案应：

- 删除 native control action 支持；或
- 只有 packet 显式声明 `native_tool_transport_enabled=True` 才允许；默认 hard fail。

### P1：public_progress_note 命名和语义容易误导

schema 中 `public_progress_note` 被描述为“公开正文反馈”，但对 `ask_user/block/request_task_run/active_work_control` 来说，它不是 assistant body，而是 status/progress note。

建议改名或明确 prompt：

- `public_status_note`
- 或 prompt 中写明：该字段只能进入状态投影，不能成为最终正文。

## 5. 统一目标契约

### 5.1 Action schema authority

只能有一个动作契约来源：

```text
ActionContractRegistry
  action_type
  allowed_contexts
  required_fields
  forbidden_fields
  parser_shape
  admission_policy
  dispatch_handler
  output_channel
  persistence_policy
  public_surface
```

所有 prompt/schema/compiler/admission/dispatch/test 都从这里派生或被它校验。

### 5.2 Dispatch authority

`single_agent_turn` 不应靠“分支后自然落到 content fallback”。它必须：

- parsed action 一定进入 action dispatcher。
- dispatcher 返回 terminal/action outcome。
- unhandled action type 直接 protocol error。
- no-action fallback 只允许非 JSON、非协议、非控制形状的自然语言。

### 5.3 Presentation authority

输出面必须固定：

| Surface | 允许内容 | 禁止内容 |
| --- | --- | --- |
| assistant_body | `respond.final_answer` 的 canonical stable text | action JSON、control fields、tool args、status receipt |
| status_bar | 简短运行状态 | raw protocol、authority、packet refs |
| tool_window | 工具名、目标、状态、摘要 | 隐藏 prompt、完整工具协议 |
| control | ask_user、active_work_control、pause/stop/steer 状态 | control JSON 原文 |
| task_projection | todo、当前步骤、验收状态 | 模型 raw output |

## 6. 必须新增的契约测试

### 6.1 Action coverage test

新增一个测试遍历所有 `ModelActionType`：

- 如果 action 出现在 compiler allowed actions，则 single-agent dispatcher 必须有 handler。
- 如果没有 handler，该 action 不得被 compiler/prompt 暴露。

### 6.2 No control fallback test

对每个控制动作构造 raw JSON：

- `{"action_type":"ask_user",...}`
- `{"action_type":"block",...}`
- `{"action_type":"request_task_run",...}`
- `{"action_type":"request_registered_engagement",...}`
- `{"action_type":"active_work_control",...}`
- 裸 `{"action":"continue_active_work",...}`

断言：

- 不会以 `answer_channel=conversation` commit。
- 不会进入 assistant body。
- 不会显示复制按钮。

### 6.3 `request_registered_engagement` safety test

在未实现 dispatch 前：

- compiler 不得允许该 action；或
- parser/admission 后必须 protocol error，不能 fallback。

如果实现 dispatch：

- fake model 返回该 action，断言进入 engagement lifecycle，不写 body。

### 6.4 Frontend stream content guard

对 `assistant_text_delta/final/repair` 三类事件分别注入控制 JSON：

- metadata 缺失。
- metadata 错标 `conversation/stable_answer/persist_canonical`。

断言 message content 不变。

## 7. 修复优先级

1. P0：single-agent dispatcher 改为 exhaustive，未处理 action type protocol error。
2. P0：`request_registered_engagement` 立即从 allowed actions/schema/prompt 中移除，或实现完整 dispatch。
3. P0：active-work 裸 JSON/action-only JSON 全部进入 parser，不能 no-action。
4. P0：frontend live stream 对 `content/replacement_content` 做内部协议过滤。
5. P1：统一 action schema authority，删除旧 schema 漂移。
6. P1：禁用 native control action 兼容路径。
7. P1：把 public_progress_note 语义改成 status note，避免正文误导。

## 8. 最终标准

修复完成后，任何控制命令泄露测试都必须满足：

```text
raw model output contains control command
-> parser detects action/control shape
-> invalid or admitted action
-> execution/status/control surface only
-> output boundary blocks body persistence
-> frontend store drops polluted frames
-> UI never renders control JSON
-> copy button never appears for control JSON
```

只要有一处能把控制命令写入 `assistant.content`，就不能算修复完成。

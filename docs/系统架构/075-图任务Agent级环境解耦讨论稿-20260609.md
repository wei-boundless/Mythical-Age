# 075-图任务 Agent 级环境解耦讨论稿

日期：2026-06-09  
状态：讨论稿，不是最终实施方案  
目标：先把设计事实和关键分歧记录下来，讨论成熟后再写正式计划书并实施。

## 1. 背景

图任务系统的目标已经从“一个任务图绑定一个任务环境”升级为“一个图中可以有多个 agent，每个 agent 可以拥有自己的任务环境”。这意味着图任务不能继续把 `GraphHarnessConfig.task_environment_id` 当成所有节点执行、产物写入、记忆命名空间和 prompt 环境挂载的唯一来源。

新的目标不是取消环境，而是取消图级环境对所有节点的强绑定。图仍然需要一个控制面所属环境用于 session scope、监控归属、项目入口和审计；但每个可执行节点必须拥有自己的 effective environment，由编译期锁定，运行期只消费。

## 2. 当前代码事实

当前图任务链路已经具备可升级基础：

- `backend/task_system/graphs/task_graph_models.py` 已有节点、边、资源、`contract_bindings` 和 runtime contract。
- `backend/task_system/compiler/graph_harness_config_publisher.py` 已经负责发布 `GraphHarnessConfig`，并锁定 graph-level `task_environment_id` 与 `environment` payload。
- `backend/harness/graph/context_materializer.py` 当前在 build input package 时把 `graph_config.task_environment_id` 和 `graph_config.environment` 写入每个 work order。
- `backend/harness/graph/work_order_contract.py` 当前 `_graph_node_runtime_contract()` 优先读取 `graph_config.task_environment_id`，这会压过节点自己的环境。
- `backend/harness/graph/output_policy.py` 当前 `_environment_projection()` 只从 graph config 取环境。
- `backend/harness/graph/work_order_executor.py` 的 artifact / memory receipts 多处使用 `graph_config.task_environment_id`。
- `backend/api/orchestration.py` 当前要求 graph run 的 session scope 与 graph config 的 task environment 匹配。

这些事实说明：图系统不是没有环境能力，而是环境权威仍集中在 graph level，节点级环境尚未贯穿编译、work order、runtime contract、产物、记忆和审计。

## 3. 旧限制的本质

旧限制不是“环境字段太少”，而是“环境决策权放错层级”：

- graph config 现在同时承担控制面环境、节点执行环境、产物根目录、记忆空间和 runtime prompt mount 的来源。
- work order 没有稳定的 `effective_task_environment_id`，导致 executor 只能回退 graph config。
- runtime profile 已经可通过 metadata 承载扩展配置，但图编译器尚未把 agent/profile 级环境解析成节点级锁定配置。
- 如果运行时才根据 agent profile 动态取环境，断点续跑会受到 profile 后续修改影响，破坏 published graph config 的可恢复性。

因此，成熟方案必须在编译期解析并锁定节点环境，而不是在执行时临时查 agent profile。

## 4. 建议的目标模型

建议把环境分成两层：

1. `graph_control_environment_id`

用于 graph run 的 session scope、监控入口、项目归属、控制面显示和启动 API 校验。它不再代表所有节点的执行环境。

2. `node_effective_environment`

用于单个节点的 prompt mount、文件访问表、artifact root、memory namespace、sandbox policy、runtime scope、artifact/materialization receipt 和 memory receipt。

节点环境解析优先级建议为：

```text
node explicit environment
-> node executor/runtime contract environment
-> agent runtime profile environment
-> graph control environment
-> env.general.workspace
```

这里的 “agent runtime profile environment” 需要讨论是做成一等字段，还是先通过 `metadata.task_environment_id` / `metadata.runtime_contract.task_environment_id` 承载。

## 5. 编译期锁定原则

`GraphHarnessConfig` 发布时应生成节点环境索引，而不是只生成 graph-level environment：

```text
node_environment_index: {
  "<node_id>": {
    "task_environment_id": "...",
    "environment": {... locked runtime_environment_payload ...},
    "source": "node_explicit | agent_runtime_profile | graph_control_default",
    "agent_id": "...",
    "agent_profile_id": "...",
    "authority": "task_system.graph_harness_config_publisher.node_environment_lock"
  }
}
```

这样做的理由：

- 配置发布后，节点执行环境不受 agent profile 后续改动影响。
- 断点续跑可以用 structure hash 校验环境边界。
- artifact / memory / file access 的目标空间可审查、可恢复。
- 编译器成为唯一环境解析权威，runtime 不二次决策。

## 6. 运行时消费原则

运行时只消费已锁定的 node effective environment：

- `GraphContextMaterializer` 为每个 work order 写入 `input_package.task_environment_id`、`input_package.task_environment`、`runtime_scope.task_environment_id` 和 `environment_refs`。
- `GraphNodeExecutionSlot.output_contract.environment_projection` 使用 node environment，不再默认 graph environment。
- `work_order_contract` 构造 `TaskRunContract` 和 runtime contract 时，以 graph slot / input package 的 node environment 为第一来源。
- `work_order_executor` 的 artifact materialization、formal memory candidate、memory commit、quality receipt 都使用 work order 的 effective environment。
- monitor / event log 同时展示 graph control environment 和 node effective environment，避免审计时混淆。

## 7. 权限与安全原则

环境不能扩大 agent 权限。节点最终权限应是以下三者的交集或显式收窄结果：

```text
agent runtime profile allowed operations
∩ task environment execution/file/sandbox policy
∩ graph/node contract permission ceiling
```

runtime settings patch 不能切换节点环境，也不能通过环境切换扩大工具、文件、shell、browser、network 或 memory 写入权限。需要换环境时必须重新发布兼容的 graph config。

## 8. 需要讨论的设计问题

1. Agent runtime profile 的环境应不应该成为一等字段？

选项 A：新增 `AgentRuntimeProfile.task_environment_id`。优点是清晰、可校验、可被前端直接编辑。缺点是需要迁移模型、API、测试。

选项 B：先使用 `metadata.task_environment_id` 或 `metadata.runtime_contract.task_environment_id`。优点是改动小；缺点是长期语义不够硬，容易被别的 metadata 语义污染。

建议：正式方案采用一等字段；如果实施分两阶段，第一阶段可以读 metadata 作为迁移输入，但发布产物必须变成标准 `node_environment_index`。

2. graph 没有单一环境时，启动 API 应如何校验？

建议保留 `graph_control_environment_id`。用户从某个任务环境打开图任务时，该环境是控制面归属，不限制节点执行环境。API 校验应从“必须等于 graph_config.task_environment_id”改成“必须等于 graph control environment，或 graph config 明确声明允许多环境控制面”。

3. 节点环境变化是否进入 structure hash？

建议进入。原因是节点环境会改变 file access table、artifact root、memory namespace 和 sandbox policy，属于执行结构，不是纯 runtime model 参数。

4. 旧 graph-level environment 字段是否保留？

建议改名语义，不作为节点执行默认事实。短期可以保留 `task_environment_id` 字段作为 `graph_control_environment_id` 的兼容投影，但所有新代码必须读明确语义的 control / node environment index。

## 10. 验证方向

正式方案至少需要证明：

- 同一个 graph 中两个 agent 节点可以解析到不同 task environment。
- graph run 的 control environment 不会覆盖 node effective environment。
- work order input package、graph slot、TaskRunContract、runtime contract、artifact receipt、memory receipt 使用同一个 node environment。
- 改 agent profile 环境后，已发布 graph config 的断点续跑不受影响。
- runtime settings patch 不能切换环境或扩大权限。
- 普通 single-agent turn 仍保持现有显式环境选择规则，除非我们决定全局升级 agent profile environment。

## 11. 暂定结论

图任务升级的成熟方向是：图负责编排，agent 负责任务能力与默认环境，编译器负责把 agent/default/node override 解析成锁定的 node environment，runtime 只按 work order 执行。环境从“图级限制”变成“节点级执行边界”，这样才能支持真正的多 agent 连续任务、恢复、审查和环境隔离。

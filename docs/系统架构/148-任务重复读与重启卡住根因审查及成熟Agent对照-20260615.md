# 任务重复读与重启卡住根因审查及成熟 Agent 对照

日期：2026-06-15

## 1. 结论

这次问题由两个相关但不同的故障组成：

1. **重复读文件**：不是 `read_file` 工具自己重试，而是模型在第 9 次任务调用里主动请求了 3 个并行 `read_file`。根因不是“模型懒”或“工具慢”，而是本项目的文件证据投影同时给出了互相冲突的信号：一边说 `coverage.complete=true`，一边又生成 `read_missing_windows`。模型为了满足 `edit_file` 的精确 `old_text -> new_text` 合同，保守地再次读取关键窗口。
2. **重启前卡住**：第 10 次调用已经进入模型等待阶段，没有工具在跑。随后 runtime 启动恢复打断了正在等待的模型调用，而恢复控制器又明确禁止 `runtime_start_recovery` 自动重新调度，所以任务不会自动续跑。此前还发生过一次 Windows 文件替换 `WinError 5`，说明动态上下文 replacement store 在 Windows 上存在原子替换脆弱点。

最核心的代码级 bug 是：

`backend/harness/runtime/dynamic_context/task_state_projector.py` 的 `_read_missing_window_decisions()` 把 `coverage.missing_ranges == []` 当成“没有缺口字段，需要根据 active_windows 重新计算缺口”，而不是当成“确认没有缺口”。因此即使同一个 payload 中 `coverage.complete=true`，也可能重新造出 `read_missing_windows`。

这说明 2026-06-13 已实施的 File Evidence Authority 方向是对的，但落地没有闭环：项目已有 `FileStateAuthority`、`do_not_repeat_read_ranges`、`rehydration_plan` 和 read dedup 的片段，却还没有形成 Claude Code / Codex 一类成熟 agent 的硬权威链路。

## 3. 为什么它重复读

### 3.1 直接诱因：同一 payload 内证据自相矛盾

第 10 次 payload 中多处同时出现：

- `read_missing_windows`: `start_line=366`、`start_line=820`
- `do_not_repeat_read_ranges`: 已覆盖部分当前窗口
- `coverage.complete=true`
- `coverage.merged_ranges=[{"start_line":1,"end_line":1083}]`
- `task_progress_facts.file_evidence.missing_window_count=2`

对应文件位置：

- `event_payloads\57\57aeb...json:11779-11842`
- `event_payloads\57\57aeb...json:19466-19529`
- `event_payloads\57\57aeb...json:30886-30949`

这等于告诉模型两件互斥的事：

1. 当前文件 1-1083 已完整覆盖。
2. 仍有 366 和 820 起始的缺失窗口需要读取。

模型后续面对 `edit_file` 的要求时，会优先相信“old_text 必须来自当前有效读窗证据”，于是重复读关键窗口。

### 3.2 代码级根因：空缺口被当成未知缺口

文件：

`D:\AI应用\langchain-agent\backend\harness\runtime\dynamic_context\task_state_projector.py`

关键代码：

- `_file_evidence_decisions_projection()`：1235-1278
- `_read_missing_window_decisions()`：1323-1358
- `_missing_ranges_from_active_windows()`：1370-1398
- `_read_resource_state_projection()`：1431-1491

问题点：

```python
missing = [dict(segment) for segment in list(coverage.get("missing_ranges") or []) if isinstance(segment, dict)]
if not missing:
    missing = _missing_ranges_from_active_windows(active_ranges, total_lines=_safe_int(item.get("total_lines")))
```

当 `coverage.complete=true` 时，`missing_ranges` 正确结果就是空列表。这里却把空列表当成“没有可用缺口信息”，继续用 `active_ranges` 重新计算。

如果 `active_ranges` 没包含完整全文件窗口，或混入了编辑前后、stale 与非 stale 的窗口，重新计算就会制造出虚假的 `read_missing_windows`。这正好解释了为什么 payload 里同时出现完整覆盖和缺失窗口。

正确语义应该是：

- `coverage.complete=true`：直接返回无缺口。
- `coverage` 明确含有 `missing_ranges: []`：直接返回无缺口。
- 只有 `coverage` 缺失、`complete` 不为 true、且没有可信 merged coverage 时，才允许从 active ranges 推导缺口。

### 3.3 工具层没有挡住子窗口重复读

文件：

`D:\AI应用\langchain-agent\backend\runtime\tool_runtime\native_tools.py`

`_unchanged_previous_read_window()` 在 606-649 行只匹配：

- 同一路径
- 同一 `content_sha256`
- 同一 `start_line`
- 同一 `end_line`
- mtime 相同或兼容

因此第 8 次读了全文件 `1-1083`，第 9 次读 `270-329`、`340-399`、`770-819` 时，不会命中去重。工具层没有识别“子窗口已被全文件当前读覆盖”，于是仍返回新窗口正文。

这不是单纯实现小瑕疵。成熟 agent 的文件状态权威不能只做 exact-range dedup，还需要表达：

- 这个子窗口已经被当前完整读覆盖。
- 如果精确内容已被省略，应恢复旧工具输出或使用持久化内容。
- 如果文件变更，才允许重新读最小窗口。

### 3.4 编辑工具没有硬性消费读状态

文件：

`D:\AI应用\langchain-agent\backend\runtime\tool_runtime\native_tools.py`

`NativeEditFileTool` 在 817-873 行读取文件、执行 `files.edit_text(path, old_text, new_text)`，成功后返回 `Edit succeeded`。它没有像 Claude Code 那样在工具层强制校验：

- 文件是否已被当前任务显式读取。
- 读取是否是完整可编辑视图。
- 文件 mtime/hash 是否晚于上次读。
- `old_text` 是否来自当前有效读窗或某个 evidence ref。

所以模型承担了本该由工具层承担的证据判断。模型越想避免 `old_text not found`，越会重复读。

## 4. 为什么重启前卡住

第 10 次调用时任务状态是：

- offset 191：`runtime_invocation_packet_compiled`，payload size `1,215,635` bytes。
- offset 193：`task_model_action_invocation_started:10`。
- offset 194：`task_model_action_waiting:10`。
- offset 195：`task_run_executor_recovered_after_runtime_start`。

这说明重启前它在等模型响应，不是在执行文件工具，也不是 read_file 卡死。

恢复逻辑：

`D:\AI应用\langchain-agent\backend\harness\loop\task_executor_controller.py`

- `schedule()` 53-61 行：如果 scheduler 是 `runtime_start_recovery`，返回 `runtime_start_recovery_does_not_auto_schedule`。
- `recover_scheduled()` 131-139 行：同样禁止 runtime start recovery 自动 schedule。

因此 runtime 重启后，任务被记录为“恢复过”，但不会自动继续第 10 次调用。用户看到的就是任务像卡住或失联。

另一个恢复侧风险：

`D:\AI应用\langchain-agent\backend\harness\runtime\dynamic_context\replacement_store.py`

- `_write()` 130-132 行：固定 `.tmp` 后缀，然后 `tmp.replace(path)`。
- `_index_task_run_record()` 157-159 行：同样固定 `.tmp` 后缀，然后 replace。

本次 offset 90 已经出现：

```text
[WinError 5] 拒绝访问:
...taskrun_turn_session-9458b9376ed8437e_9_5c7c9c1a.tmp
-> ...taskrun_turn_session-9458b9376ed8437e_9_5c7c9c1a.json
```

Windows 下固定 tmp 文件名、无重试/backoff、无唯一临时文件名，会让并发投影或杀软/索引器占用时更容易失败。

## 7. 本项目与成熟实现的差距

### 7.1 本项目已有正确方向

`backend/maintenance/file_evidence_authority_prompt_runtime_refactor_plan_20260613.md` 已经写明：

- 问题不是模型笨，而是读取证据决策权分散。
- 目标是 `Tool Observation -> FileStateAuthority -> File Evidence Decision Projection -> Runtime Prompt`。
- 只有缺失、过期、变更、目标行未覆盖时才再次读取。

本次事故证明该方向正确，但实现仍有漏洞。

### 7.2 审查时未闭环的部分

1. `coverage.complete=true` 没有压倒 `read_missing_windows`。
2. `missing_ranges=[]` 和 `missing_ranges` 缺失没有区分。
3. read dedup 只支持 exact same window，不支持“已被更大当前窗口覆盖”的子窗口。
4. full-file read 的精确内容被投影/摘要/refs 分散后，模型无法稳定把它当成可编辑 exact content。
5. `edit_file` 不消费 FileStateAuthority，不强制 read-before-edit、mtime/hash freshness、partial view 拒绝。
6. `rehydration_plan` 是模型提示，不是编辑工具可直接消费的证据句柄。
7. recovery 记录了 runtime restart，但明确不自动恢复调度。
8. Windows replacement store 使用固定 `.tmp` + 无重试 replace，已经在本次任务中失败过。

本次修复后，1、2、3、5、8 的核心链路已落地；并额外补齐了空文件 `total_lines=0` 的 complete 语义，以及 `read_resource_state` 不把 stale 旧窗口冒充当前可用证据的投影边界。4、6、7 仍是后续架构项：精确内容 rehydration 还没有成为工具层可消费证据句柄，runtime restart 仍是显式“不自动续跑”的恢复语义。

## 8. 推荐修复方向

本节保留原审查计划，同时标注截至 2026-06-15 本次复查后的实施状态。已实施项以代码和回归测试为准；未实施项仍应按计划单独评审。

### Phase 1：修复文件证据投影的互斥裁决

状态：已实施核心修复。

目标：同一个文件同一轮上下文中不能同时出现 complete coverage 和 missing windows。

涉及文件：

- `backend/harness/runtime/dynamic_context/task_state_projector.py`
- `backend/runtime/memory/file_state_authority.py`
- `backend/tests/file_state_authority_regression.py`

关键修改：

- `_read_missing_window_decisions()` 先判断 `coverage.complete is True`，直接返回空。
- 区分 `missing_ranges` 字段存在且为空，与字段不存在。
- 如果 `coverage.merged_ranges` 已覆盖 total_lines，不允许再由 active_ranges 反推缺口。
- stale windows 只能出现在 `read_after_stale_windows`，不能参与当前缺口推导。
- 空文件 `total_lines=0` 且 `has_more=false` 视为 complete，不再生成下一读建议。
- `read_resource_state` 只统计非 stale read range；完整空文件允许作为当前资源事实出现，但 `available_range_count=0`。

验收标准：

- 回放本次 payload，`missing_window_count` 必须为 0。
- `file_evidence_decisions` 中不再同时出现 `coverage.complete=true` 和 `read_missing_windows`。

### Phase 2：让 read_file 支持覆盖窗口去重

状态：已实施核心修复。

目标：全文件当前读已覆盖子窗口时，子窗口 read 不再重复塞正文。

涉及文件：

- `backend/runtime/tool_runtime/native_tools.py`
- `backend/runtime/memory/file_state_store.py`
- `backend/runtime/memory/file_state_authority.py`
- `backend/tests/file_state_authority_regression.py`

关键修改：

- 在 `_unchanged_previous_read_window()` 增加 covered-by-current-range 判断。
- 若旧窗口 `start<=requested_start` 且 `end>=requested_end`，hash/mtime 当前，则返回 `file_unchanged` 或 `covered_by_previous_read` stub。
- stub 必须提供可复用 observation/ref，必要时指向 `read_persisted_tool_result`。

验收标准：

- 先读 `1-1083`，再读 `270-329`，工具返回省略正文的结构化 stub。
- 文件 edit/write 后旧覆盖窗口必须失效，不能错误复用。

### Phase 3：把 edit_file 改成消费 File Evidence Authority

状态：已实施核心修复。

目标：编辑工具承担读状态校验，模型只提供编辑意图和精确文本。

涉及文件：

- `backend/runtime/tool_runtime/native_tools.py`
- `backend/runtime/tool_runtime/tool_result_envelope.py`
- `backend/runtime/memory/file_state_authority.py`
- `backend/tests/workspace_file_tools_regression.py`
- `backend/tests/file_gateway_tool_runtime_regression.py`

关键修改：

- `edit_file` 调用前按操作语义检查证据：已存在文件的非空 `old_text` 局部替换必须有当前有效读证据；新文件创建、已确认为空文件的初始化、或明确全量写入不应被这一条误拒。
- 如果已存在文件未读、读证据 partial、mtime/hash 过期，且本次操作依赖旧内容匹配，返回结构化错误并要求最小 read。
- 对 `old_text == ""` 单独定义合同：目标不存在时允许作为新文件创建；目标存在且大小为 0 或已由 `stat_path` / `read_file` 确认为空时允许初始化；目标存在且非空但未读时拒绝，避免把空串插入/替换语义变成隐式覆盖。
- 全量写入应优先走 `write_file` 合同；`edit_file` 不承担未读非空文件的整文件覆盖。
- 编辑成功后，把新文件内容或 hash/mtime 写回 FileStateAuthority；旧窗口标记为 stale，只能作为历史证据，不能再被投影为当前可用读范围。
- 可选升级：`edit_file` 支持 `evidence_ref`，让 old_text 与某次读窗绑定。

验收标准：

- 未读且已存在的非空文件局部编辑被拒绝。
- 目标不存在时，`old_text == ""` 创建新文件成功。
- 目标已确认为空时，`old_text == ""` 初始化成功。
- 目标存在且非空但未读时，`old_text == ""` 被拒绝并要求 read/stat 后选择 `write_file` 或精确 edit。
- 已读后编辑成功。
- 文件外部变更后编辑被拒绝并要求重读。
- 编辑成功后不再默认要求重读确认。

### Phase 6：明确 runtime restart 恢复语义

状态：未实施，仍是后续架构项。

## 10. 不能走的捷径

- 不能只在 prompt 里追加“不要重复读”。
- 不能让模型自己判断 `coverage.complete` 与 `read_missing_windows` 谁更可信。
- 不能保留两套文件证据解释链路。
- 不能用“编辑后默认重读”作为稳定性兜底。
- 不能把 `old_text not found` 失败作为正常控制流。
- 不能让 rehydration 只是提示，不进入工具层证据消费。
- 不能用固定 `.tmp` 在 Windows 并发/索引环境里做 replacement store 写入。

## 11. 最终判断

这次重复读的根因可以精确落在三层：

1. **投影 bug**：`missing_ranges=[]` 被错误 fallback，导致 complete coverage 与 missing windows 同时出现。
2. **工具边界不足**：read_file 只 exact-range dedup，edit_file 不硬性消费读状态。
3. **上下文恢复不闭环**：full-file exact content 被摘要/refs 化后，没有形成可直接复用的当前证据句柄。

这次卡住的根因可以落在两层：

1. **运行时中断点**：第 10 次模型等待期间发生 runtime start recovery。
2. **恢复策略**：`runtime_start_recovery` 明确不自动 schedule，用户侧又没有足够清晰的恢复等待状态。

成熟 Agent 的标准不是“提示模型少读”，而是把读状态、编辑前置条件、压缩恢复、去重和重启恢复做成硬协议。Claude Code 的 `readFileState + Read dedup + Edit/Write hard guard + compact restore` 是本项目最应该借鉴的主线；Codex 的搜索/上下文 baseline 策略则适合用来约束搜索结果、避免 stale context diff 和大 payload 膨胀。

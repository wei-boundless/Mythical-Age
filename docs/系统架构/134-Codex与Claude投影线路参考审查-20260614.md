# Codex 与 Claude Code 投影线路参考审查

日期：2026-06-14

本记录回应当前争议：前端投影系统到底应该怎么处理“模型正文”和“工具调用窗口”，以及之前只靠前端过滤、去重、正文收口后隐藏工具的方向是否可靠。

结论先写清楚：之前把重点放在前端删重复、正文收口后压掉工具窗口，是不完整的，部分方向是错的。成熟 agent 的做法不是从最终正文里反推工具，也不是把工具生命周期挂进正文，而是：

```text
同一轮运行只有一条严格时序的事件/消息时间线；
模型正文和工具控制轨迹是 typed sibling items / content blocks；
前端按同一时序投影到不同表面；
工具生命周期用同一个 tool_call_id / tool_use_id 闭合；
历史回放从 typed timeline / transcript item 恢复，不能从最终正文猜。
```

## 4. 成熟 agent 的硬约束

| 约束 | 正确做法 |
| --- | --- |
| 单时序 | 所有正文、工具 request/start/progress/result、终止都属于同一 ordered timeline |
| 双表面 | 模型正文投正文区；工具控制轨迹投工具窗口 |
| 类型化 | 正文、命令、MCP、dynamic tool、hook、commit 都必须是 typed item/frame/block |
| 稳定身份 | 工具生命周期用 `tool_call_id` / `tool_use_id`，不能用 title/path/时间戳 |
| 原地闭合 | request/start/progress/result 更新同一个工具窗口 |
| 正文不吞工具 | final answer / body finalized 只代表正文收口，不代表删除工具轨迹 |
| 工具不吞正文 | 模型反馈是正文或 status/body frame，不能挂在工具窗口里导致正文缺失 |
| 历史可 replay | 非任务状态、刷新、恢复会话必须从 typed history/projection replay 工具窗口 |
| 用户视图简洁 | 默认不显示内部 ID/offset；工具窗口显示动作、目标、状态、命令/窗口信息、结果摘要 |
| trace 分层 | 完整 ID、offset、大 payload、raw output 进入 trace/debug，不进主聊天默认视图 |

## 5. 对当前项目的判断

### C1. 之前的前端过滤方向不可靠

当前 `frontend/src/components/chat/ChatMessage.tsx`：

| 行 | 现状 | 问题 |
| --- | --- | --- |
| 88-91 | `hasCommittedAssistantText` 为真时 `publicTimelineItems=[]` | 非流式历史正文存在就清空工具窗口 |
| 390-393 | `projectionHasClosedBody(projection)` 为真直接返回空 timeline | body finalized/committed 后工具轨迹消失 |
| 411-413 | `bodyState=finalized/committed` 且有正文即视为 closed body | 把“正文收口”和“工具历史删除”混成一个条件 |

这和 Codex/Claude 的做法相反。成熟 agent 可以在最终态弱化工具 UI、默认折叠工具窗口，但不能因为最终正文存在就删除工具记录。

所以“旧工具记录非任务状态消失”不是单个组件问题，是后端历史数据和前端 hydrate 缺线。

### C4. 后端仍有重复反馈风险和窗口信息不足风险

| 行 | 现状 | 问题 |
| --- | --- | --- |

2. 两个不同 `read_file` 读取同一文件不同区间，因为 UI 没显示 `start_line`，看起来像重复工具。

## 6. 修正后的目标线路

本项目应该对齐下面这条线路：

```text
runtime event
-> public projection event/frame
-> persisted public projection ledger
-> live SSE reducer
-> message publicProjection
-> ChatMessage 按同一 event_offset 投影
-> body surface + tool window surface
-> session timeline/history hydrate replay
```

### 6.1 正文线路

| 阶段 | 信号 | 前端表面 |
| --- | --- | --- |
| 模型正文增量 | `assistant_text_delta` / `body_append` | 正文区 |
| 模型阶段反馈 | `runtime_step_summary`，`source_authority=model`，`event_family=status_trace` | 正文区 |
| 正文收口 | `assistant_text_final` / `body_finalize` / commit | 正文区进入 finalized/committed |

规则：

- 同一句模型反馈只能由 producer 产生一次，或带稳定 body segment id 让前端按 identity 合并。
- 正文去重只能作为 replay 防护，不能成为主修复。
- 正文收口只影响正文状态，不允许清空工具 timeline。

### 6.2 工具线路

| 阶段 | 信号 | 前端行为 |
| --- | --- | --- |
| 请求工具 | `tool_call_requested` | 以 `tool_call_id` 创建工具窗口 |
| 权限/准入 | `tool_permission_decided` | 更新同一窗口；正常允许不需要单独噪声行 |
| 开始执行 | `tool_item_started` | 同一窗口进入 running，可补充命令/窗口信息 |
| 观察/完成 | `tool_item_completed` / error | 同一窗口完成/失败，显示摘要 |

规则：

- 合并键优先使用真实 provider/model `tool_call_id` / `tool_use_id`。
- 如果存在 runtime `tool_lifecycle_id`，它只能作为同一 call 下的 attempt/lifecycle 辅助身份。
- 不能用文件路径、工具名、title、offset 推断同一工具。
- 同一文件不同 `start_line` 是两个工具调用，应显示两个窗口，但窗口标题/展开命令必须能区分。

### 6.3 历史回放线路

| 阶段 | 正确要求 |
| --- | --- |
| 后端 timeline | 返回每个 anchor message/turn 对应的 public projection frames 或已归并 projection |
| 前端 hydrate | `toUiMessages()` 后按 attachment anchor 将 projection replay 到对应 assistant message |
| 最终态显示 | 正文为主，工具窗口默认折叠保留 |
| trace | 内部 ID、offset、raw output 保留在 trace/debug，不进默认主聊天 |

## 8. 对 129 文档的修正解释

129 里“final 后显示最终正文”不能解释成“final 后删除工具窗口”。正确解释应是：

```text
final 后，正文区只显示最终正文；
工具轨迹不再抢主视觉，但作为同一 turn 的控制轨迹保留在折叠工具窗口/历史 transcript 中；
trace/debug 仍可查看内部 ID 和完整事件。
```

也就是说：

- `bodyState=committed` 是正文状态，不是 timeline delete signal。
- `tool_call_requested -> tool_item_completed` 是工具窗口生命周期，不是正文的一部分。
- 历史态和运行态应使用同一 typed projection 合同，不能 live 有工具、history 只剩正文。

## 9. 本阶段裁决

本阶段裁决如下：

```text
当前方案必须从“前端过滤重复”升级为“typed public projection ledger + 历史 replay + 稳定 tool id 合并”。
只修 ChatMessage 的显示条件，不能完全解决问题。
只加 compactText 去重，不能保证时序正确。
正文和工具同时返回时，前端必须按同一 event_offset 排序，然后分表面投影。
正文不能挂在工具窗口里，工具也不能在正文收口后消失。
```

下一步执行应按本审查重写修复计划，核心目标是把 live projection 和 history projection 打通，而不是继续在 UI 层补丁式删重。

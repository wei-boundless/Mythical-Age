# 成熟 Agent 投影线路结构性修复计划书

日期：2026-06-14

依据：

- `docs/系统架构/129-agent显示流程与投影规则说明-20260614.md`
- `docs/系统架构/131-投影重复根因审查记录-20260614.md`
- `docs/系统架构/132-非任务状态旧工具记录消失审查记录-20260614.md`
- `docs/系统架构/134-Codex与Claude投影线路参考审查-20260614.md`

本计划替代前面偏 UI 层过滤/去重的修复思路。后续执行必须按本文落地，不再把“最终正文存在”当成删除工具轨迹的理由，不再从正文反推工具窗口，不再靠文本去重掩盖后端重复生产。

## 1. 修复目标

目标只有一条：

```text
把 agent 运行时的正文和工具调用接成一条可实时显示、可历史 replay、严格按时序归位的 typed public projection 线路。
```

最终用户应看到：

| 状态 | 正文区 | 工具窗口 |
| --- | --- | --- |
| 运行中 | 大模型自然语言反馈和正文实时出现 | agent 调动系统的工具窗口实时出现并更新 |
| 工具完成 | 正文继续按时序显示 | 同一工具窗口从运行中变成完成/失败 |
| 正文收口 | 正文区显示最终正文 | 旧工具窗口保留为折叠历史轨迹 |
| 刷新/切会话/非任务状态 | 历史正文恢复 | 历史工具窗口从 public projection replay 恢复 |

## 2. 必须锁死的工程原则

| 原则 | 约束 |
| --- | --- |
| 单时序 | 所有正文 frame、工具 frame、commit frame 都按 `event_offset` / source event 顺序进入同一 projection ledger |
| 双表面 | 正文只能进正文区，工具只能进工具窗口，状态栏只显示连接/运行状态 |
| 稳定身份 | 工具生命周期主键是 `tool_call_id`；缺失时才使用 `tool_lifecycle_id` |
| 原地闭合 | `tool_call_requested -> tool_permission_decided -> tool_item_started -> tool_item_completed` 更新同一工具窗口 |
| 历史 replay | 非任务状态不能只靠 `history.messages[].content`；必须 replay public projection |
| 正文收口不删工具 | `bodyState=finalized/committed` 只表示正文收口，不是 timeline 清空信号 |
| 信息足够区分 | 工具窗口必须显示 path/query/command/cwd/start_line/line_count 等能区分系统动作的信息 |
| trace 分层 | 内部 ID、offset、raw output 只进 trace/debug，不进默认主聊天 |
| 清理旧路 | 遇到旧语义测试、旧兜底、旧投影分支，按实际影响删除，不以兼容为理由保留 |

## 3. 目标线路

修复后的唯一主线路：

```text
runtime event
-> backend public event bridge
-> PublicProjectionFrame
-> persisted public projection frames / ledger
-> live SSE reducer
-> message.publicProjection
-> ChatMessage 分表面渲染
-> session timeline/history hydrate replay
```

其中：

| 层 | 责任 |
| --- | --- |
| runtime event | 产生真实模型正文、模型反馈、工具请求、工具执行、工具观察、commit |
| backend public event bridge | 把 runtime event 转成公开、可投影的 typed event |
| projector | 生成 `PublicProjectionFrame`，保留稳定 identity 和排序字段 |
| persistence/timeline | 保存并返回可 replay 的公开 frame，不返回默认 trace 大 payload |
| frontend reducer | 按 frame identity 和 tool identity 入账，保证 replay 幂等 |
| ChatMessage | 只做表面分发，不重新解释工具语义 |
| PublicTimelineActivity | 渲染成熟工具窗口，默认简洁、可折叠、展开为控制台窗口 |

## 4. 数据合同修复

### 4.1 `PublicProjectionFrame`

继续使用现有 `PublicProjectionFrame` 合同，但执行时必须保证以下字段在 live 和 history 中一致保留：

| 字段 | 要求 |
| --- | --- |
| `frame_id` | frame 去重主键 |
| `source_event_id` | source event 去重和追溯 |
| `source_event_type` | 保留事件来源，供 reducer 分类和 trace |
| `event_offset` | 主排序依据 |
| `event_family` | `assistant_body`、`status_trace`、`tool_control`、`runtime_commit` 等 |
| `channel` | `body`、`status`、`control`、`commit` |
| `slot` | `body`、`current_action`、`status`、`final_result`、`trace` |
| `source_authority` | `model`、`runtime`、`tool`、`system` |
| `tool_call_id` | 工具窗口主生命周期 ID |
| `tool_lifecycle_id` | 同一工具调用下的执行/attempt ID |
| `tool_name` | 工具名 |
| `target` | 用户能理解的目标，例如文件、query、url |
| `arguments_preview` | 可区分窗口的参数摘要 |
| `state` | `running`、`waiting`、`done`、`failed` |
| `text/detail` | 正文或工具摘要；不能混用表面 |

### 4.2 `runtime_attachments`

`/sessions/{session_id}/timeline` 的 `runtime_attachments[]` 增加：

```text
public_projection_frames: PublicProjectionFrame[]
projection_anchor: {
  session_id
  anchor_turn_id
  anchor_message_id
  run_id
}
```

返回范围只允许包含主聊天需要 replay 的公开 frame：

| 允许返回 | 用途 |
| --- | --- |
| `slot=body` | 恢复正文 projection |
| `event_family=status_trace` 且 `source_authority=model` | 恢复模型公开反馈 |
| `event_family=tool_control` | 恢复工具窗口 |
| `event_family=runtime_commit` | 恢复 commit 状态 |
| 必要 terminal/status frame | 恢复停止/失败的公开状态 |

禁止把完整 debug trace、完整工具 raw output、内部 prompt、内部 runtime 状态作为默认主聊天字段返回。

### 4.3 `Message.publicProjection`

前端历史 hydrate 后，每条 assistant message 可以拥有：

```text
publicProjection: MessagePublicProjection
projectionLedger: ProjectionLedger
```

这些 projection 只能来自 public frames replay，不能从正文内容猜。

## 5. 分阶段执行计划

### 阶段 1：后端历史 replay 数据补齐

目标：让非任务状态有真实 public projection 数据源。

改动文件：

| 文件 | 改动 |
| --- | --- |
| `backend/harness/runtime/session_timeline.py` | 为 task/turn runtime attachment 读取 recent events，提取或重建公开 `public_projection_frames` |
| `backend/api/sessions.py` | 确认 `/sessions/{session_id}/timeline` 返回新增字段，不做字段过滤丢失 |
| `backend/harness/runtime/projection/projector.py` | 如当前只能 live 投影，需要补一个可从历史 event 重建 frame 的公共函数 |
| `frontend/src/lib/api.ts` | 给 `runtime_attachments[].public_projection_frames` 和 `projection_anchor` 增加类型 |

执行要求：

1. 优先复用已有 projector，不复制一套投影规则。
2. 历史 replay 只返回公开 frame，不把 trace 大 payload 塞进主聊天。
3. attachment 必须能绑定到 `anchor_message_id` 或 `anchor_turn_id`。
4. 缺 anchor 时不乱挂到最新消息，可进入 trace/diagnostic，不进入默认主聊天。

完成标准：

| 标准 | 说明 |
| --- | --- |
| S1 | `/sessions/{id}/timeline` 返回至少含工具 frame 的 `public_projection_frames[]` |
| S2 | frame 保留 `tool_call_id`、`tool_lifecycle_id`、`event_offset`、`target`、`arguments_preview` |
| S3 | 返回数据不包含完整 raw output 或内部 prompt |

### 阶段 2：前端历史 projection hydrate

目标：刷新、切会话、运行结束后，旧工具窗口仍能恢复。

改动文件：

| 文件 | 改动 |
| --- | --- |
| `frontend/src/lib/store/runtime.ts` | `refreshSessionDetails()` 在 `toUiMessages()` 后 replay `runtime_attachments[].public_projection_frames` 到对应 assistant message |
| `frontend/src/lib/store/utils.ts` | 增加按 `anchor_message_id` / `sourceTurnId` 查找 message 的辅助函数，避免靠数组位置猜 |
| `frontend/src/lib/projection/reducer.ts` | 确保 replay frame 幂等；同一 frame 重放不产生重复 timeline item |
| `frontend/src/lib/store/types.ts` | 如缺字段，补齐 message 上的 `projectionLedger/publicProjection` 类型 |

执行要求：

1. hydrate 只能使用 `frame_id/source_event_id/tool_call_id` 等 canonical identity。
2. 不能从 `message.content` 解析工具窗口。
3. live SSE 和 history replay 进入同一个 reducer 路径。
4. replay 不创建新的语义事件，只恢复已有 projection。

完成标准：

| 标准 | 说明 |
| --- | --- |
| S1 | 非任务状态下历史 assistant message 带回 `publicProjection.timeline` |
| S2 | 同一历史 frame 多次 refresh 不重复显示 |
| S3 | 没有 anchor 的 frame 不错挂到其他消息 |

### 阶段 3：移除正文收口删除工具窗口的错误逻辑

目标：正文 finalized/committed 后，工具轨迹仍作为折叠历史轨迹保留。

改动文件：

| 文件 | 改动 |
| --- | --- |
| `frontend/src/components/chat/ChatMessage.tsx` | 删除 `hasCommittedAssistantText ? [] : publicTimelineItemsFromProjection(...)` 的硬清空 |
| `frontend/src/components/chat/ChatMessage.tsx` | 删除或改造 `projectionHasClosedBody()` 对工具 timeline 的整体拦截 |
| `frontend/src/components/chat/ChatMessage.tsx` | 明确正文区和工具 timeline 分别渲染；body closed 只影响正文显示优先级 |

执行要求：

1. `bodyState=finalized/committed` 不得清空工具 timeline。
2. final 后工具默认折叠，视觉权重低于正文。
3. 模型反馈仍进正文区，不挂到工具窗口。
4. 工具 frame 不因为正文存在被吞掉。

完成标准：

| 标准 | 说明 |
| --- | --- |
| S1 | final 后可见最终正文 |
| S2 | final 后工具窗口仍存在，默认折叠 |
| S3 | 工具窗口内不显示模型正文反馈 |

### 阶段 4：后端反馈和工具窗口信息修正

目标：解决“同一句反馈重复”和“两个不同读取看起来相同”。

改动文件：

| 文件 | 改动 |
| --- | --- |
| `backend/api/chat.py` | `_tool_action_public_events()` 不再对每个工具 action 重复生产同一句 `runtime_step_summary` |
| `backend/api/chat.py` | 给模型反馈使用稳定 identity，例如基于 source model action / source event / normalized public note |
| `backend/api/chat.py` | `_tool_arguments_preview()` 改为优先字段策略 |
| `backend/api/chat.py` | 对 read/search/run/edit 等工具生成清晰 `target` 和 `arguments_preview` |

`arguments_preview` 优先顺序：

```text
path/file/file_path
start_line
line_count
end_line/range
query/pattern
cwd
command
url
```

执行要求：

1. 同一模型动作的公开反馈只显示一次。
2. 多个不同 `tool_call_id` 可以各自显示工具窗口，但必须能通过窗口信息区分。
3. `tool_call_id` 不展示给普通用户，只作为内部合并键。
4. 不用前端删除正文来掩盖后端重复生产。

完成标准：

| 标准 | 说明 |
| --- | --- |
| S1 | 两个 read_file 同文件不同 `start_line` 的窗口显示不同读取区间 |
| S2 | 同一句模型阶段反馈不会随两个工具调用重复出现 |
| S3 | 工具 request/start/completed 仍能按同一 `tool_call_id` 合并 |

### 阶段 5：工具窗口 UI 重做为成熟 agent 控制台窗口

目标：工具窗口清晰说明系统在做什么，减少噪声，支持折叠。

改动文件：

| 文件 | 改动 |
| --- | --- |
| `frontend/src/components/chat/PublicTimelineActivity.tsx` | 工具窗口折叠态：动作、目标、状态 |
| `frontend/src/components/chat/PublicTimelineActivity.tsx` | 展开态：单个控制台窗口，第一行命令/调用摘要，后续为结果摘要 |
| `frontend/src/components/chat/ChatMessage.tsx` | 确保传入 tool window 的字段完整，不把内部 ID 当主展示内容 |
| 相关样式文件 | 调整间距、折叠态、运行态/完成态视觉 |

折叠态格式：


展开态格式：


执行要求：

1. 不展示 `tool_call_id`、`tool_lifecycle_id`、`event_offset`、`source_event_type`。
2. 不展示“请求 #11 -> #14”这类内部串。
3. 不把目标、参数、观察拆成字段表格。
4. 工具输出摘要控制长度，完整内容留 trace/debug。

完成标准：

| 标准 | 说明 |
| --- | --- |
| S1 | 工具窗口可折叠/展开 |
| S2 | 用户能看懂 agent 调动系统做了什么 |
| S3 | 主聊天没有内部 ID/offset 噪声 |

### 阶段 6：真实运行验收和旧路清理

目标：确认固定端口真实显示正确，并清理干扰新线路的旧残留。

改动/操作：

| 项目 | 要求 |
| --- | --- |
| 固定端口 | 前端 `127.0.0.1:3000`，后端 `127.0.0.1:8003` |
| 进程检查 | 如果端口被旧项目进程占用，关闭旧项目进程后重启；不换随机端口 |
| 真实运行 | 用 CLI 启动前后端，打开页面实测 |
| 旧测试清理 | 遇到语义文案测试、旧投影假设测试，删除或改成结构性线路验证 |
| 文档记录 | 执行后新增执行记录，写明实际修改和验收结果 |

验收场景：

| 场景 | 必须看到 |
| --- | --- |
| 模型先反馈再调用工具 | 反馈在正文区，工具在工具窗口 |

| 刷新/切会话 | 旧工具窗口从 history projection 恢复 |
| SSE 断开/重连 | 状态栏显示连接状态，不污染正文或工具窗口 |

## 6. 连接矩阵

| Line | 期望连接 | 生产者 | 运输/存储 | 消费者 | 用户表面 | 完成状态 |
| --- | --- | --- | --- | --- | --- | --- |
| L1 | 模型正文 -> 正文区 | `assistant_text_delta/final` | `PublicProjectionFrame(body)` + SSE/history | projection reducer + `ChatMessage` | 正文区 | 阶段 1-3 验收 |
| L2 | 模型反馈 -> 正文区 | `runtime_step_summary` | `status_trace` frame | reducer + `ChatMessage` | 正文区 | 阶段 4 验收 |
| L3 | 工具请求 -> 工具窗口 | `tool_call_requested` | `tool_control` frame | reducer timeline | 工具窗口 | 阶段 1-3 验收 |
| L4 | 工具完成 -> 同一窗口 | `tool_item_completed` | 同一 `tool_call_id` | reducer merge | 工具窗口状态更新 | 阶段 2-4 验收 |
| L5 | 正文收口 -> 不删工具 | `body_finalize/commit` | projection ledger | `ChatMessage` | 正文收口 + 工具保留 | 阶段 3 验收 |
| L6 | 历史 timeline -> replay | session timeline | `public_projection_frames[]` | `refreshSessionDetails()` | 历史正文 + 工具窗口 | 阶段 1-2 验收 |
| L7 | 工具窗口信息区分 | tool args | `target/arguments_preview` | `PublicTimelineActivity` | 可读工具窗口 | 阶段 4-5 验收 |
| L8 | 内部 trace 不进主聊天 | runtime trace | debug refs | trace/debug viewer | 默认不显示 | 阶段 1/5 验收 |

## 7. 禁止事项

执行时禁止：

- 禁止用 `message.content` 解析或复原工具窗口。
- 禁止因为 `bodyState=finalized/committed` 清空 `publicProjection.timeline`。
- 禁止用 path/title/文本内容合并工具生命周期。
- 禁止把模型反馈塞进工具窗口。
- 禁止把工具调用信息塞进正文区。
- 禁止把完整 raw tool output 默认展示到主聊天。
- 禁止新增语义文案测试来固定错误显示。
- 禁止为了避免端口冲突切换到随机端口。
- 禁止保留旧投影分支作为兼容兜底。

## 8. 执行顺序锁定

执行必须按以下顺序：

```text
1. 后端 session timeline 返回 public_projection_frames
2. 前端 refreshSessionDetails hydrate projection
3. ChatMessage 移除正文收口清空工具 timeline
4. 后端 feedback 去重与 arguments_preview 修正
5. PublicTimelineActivity 工具窗口 UI 重做
6. 固定端口真实运行验收
7. 清理旧语义测试和旧残留
8. 写执行记录
```

不能先做第 5 步 UI 美化来掩盖第 1-4 步线路问题。UI 只能消费已经接正确的信号。

## 10. 验收方式

验收不以“测试文案看起来通过”为准，以真实线路和真实前端显示为准。

必须执行：

1. 检查固定端口：

```text
前端：127.0.0.1:3000
后端：127.0.0.1:8003
API：127.0.0.1:8003/api
```

2. CLI 启动前后端，确认没有旧进程占用固定端口。
3. 在真实前端运行一次包含模型反馈、多个工具调用、工具完成、最终正文的 agent 会话。
4. 刷新页面或切换会话，确认工具窗口仍在。
5. 记录截图或运行结果到执行记录。

结构性验收标准：

| 编号 | 标准 |
| --- | --- |
| A1 | 正文反馈没有被工具窗口吞掉 |
| A2 | 工具窗口没有出现在正文区 |
| A3 | 同一 `tool_call_id` 只显示一个工具窗口 |
| A4 | 不同 `tool_call_id` 的同文件不同区间调用能区分 |
| A5 | final 后正文收口，工具窗口折叠保留 |
| A6 | 非任务状态/刷新后工具窗口恢复 |

## 12. 最终裁决

这次修复不是“工具窗口 UI 优化”，而是投影线路结构性修复。

必须以 Codex / Claude Code 的成熟模式为准：

```text
typed timeline
+ stable tool identity
+ body/tool dual surface
+ historical replay
+ concise collapsible tool window
```

只有这条线路完成，才能判断“agent 运作时的输出能实时投影，正文是大模型输出，工具调用是 agent 调动系统的控制轨迹，且严格按时序不错位”。

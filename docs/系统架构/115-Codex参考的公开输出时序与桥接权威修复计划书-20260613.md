# Codex 参考的公开输出时序与桥接权威修复计划书

日期：2026-06-13

状态：已按本计划完成首轮主链路实施与固定端口验证

适用范围：主会话 SSE、公开投影、assistant 正文流、工具生命周期投影、`request_task_run` 后台任务桥接、前端 Projection Ledger、输出提交回执。

明确不包含：图任务前台重做、重新设计模型决策权、增加语义 intent 层、前端按文案裁决显示、为了兼容保留旧主链路。

---

## 1. 结论

当前系统已经有一条接近目标的公开投影主干：

```text
Runtime Event
-> Canonical Public Event
-> PublicProjectionFrame
-> Frontend ProjectionLedger
-> MessagePublicProjection
-> Chat render
```

但“桥已经有了，每一轮时序仍接不上”的核心原因是：**桥接 task run 时没有创建一个不可变的公开输出上下文**。现在 `chat.py` 在看到 `task_executor_scheduled` 后会转去监听 task event log，但后续事件仍靠 `_task_bridge_refs()` 每次从 payload、refs、active turn 和 task snapshot 里猜 `turn_id / turn_run_id / task_run_id / message_ref`。这不是语义错误，而是时序和锚点权威缺失。

修复方向不是让系统接管模型，不是硬编码文本过滤，也不是前端继续猜显示。修复方向是借 Codex 的成熟不变量：

```text
稳定 turn 先建立
可见输出归属稳定 item/message
delta 是运行中投影
completed item / commit ack 是收口权威
terminal 只关闭 turn，不代替正文或提交
UI 消费结构化事件，不从 raw event 或中文文案推断语义
```

本计划只新增“引用和时序上下文”，不新增“语义裁决层”。模型仍负责回答、工具调用、任务交接等语义决策；系统只负责承接、授权、执行、记录、投影、诊断和安全边界。

## 2. 必须遵守的六条原则

### 2.1 运行中透明

每个可见输出必须能说明来自哪个 `stream_run_id / turn_id / turn_run_id / task_run_id / source_event_id / event_offset`。

后台任务接入主会话时必须产生结构化 `task_bridge_started` 或等价 public event，明确告诉投影层：这个 task run 已经被绑定到当前 chat turn。这个事件不是 assistant 正文，不替模型说话，只公开运行事实。

### 2.2 收口后清爽

普通工具、普通进度、普通 task bridge 状态在 `session_output_commit_ack` 后由后端 frame 明确 retire 到 trace。前端不得因为 final answer、turn completed 或文案相似自行清理。

### 2.3 异常不隐藏

缺失 commit、乱序工具生命周期、孤儿 task event、bridge context 缺字段、terminal 试图合成正文等情况必须变成 `protocol_diagnostic` 或 pinned failure。不能静默忽略，也不能用 fallback 正文掩盖。

### 2.4 证据可追溯

所有投影 frame 必须带 `source_event_id`、`event_offset`、`source_event_type` 和 anchor。task bridge 还必须带 `source_task_event_id/source_task_event_offset` 或 trace ref，让用户和开发者能从公开投影追到 runtime event log。

### 2.5 顺序严格稳定

同一个 chat stream 只能有一个 public event clock。task event 被桥接进 chat stream 后，以 chat public offset 为前端排序权威，以 source task offset 为证据引用。不能让前端混用 task offset、chat offset、到达顺序和本地 Date.now。

### 2.6 前端不裁决语义

前端只接受 `public_projection_frame`，只按 `op / slot / main_visibility / retention / pin_reason / commit.state` 更新账本。前端不得：

- 从 raw event 绑定 assistant message 主锚点。
- 按中文文案隐藏或折叠 projection item。
- 根据 final answer、turn terminal、工具名判断收口。
- 把 task handoff、tool started、transport done 转成用户可见语义。

## 3. Codex 参考依据

本计划参考的是 `D:\AI应用\openai-codex` 的输出和投影不变量，不照搬 Rust 类型或 JSON-RPC 名称。

### 3.1 稳定 turn anchor

Codex `turn/start` 返回 `TurnStartResponse { turn }`，而 `Turn` 本身有稳定 `id`、`status`、`items` 和时间信息。

参考文件：

- `D:\AI应用\openai-codex\codex-rs\app-server-protocol\src\protocol\v2\turn.rs`
- `D:\AI应用\openai-codex\codex-rs\app-server-protocol\src\protocol\v2\thread_data.rs`

对本项目的约束：

```text
chat_run_started 不能长期只代表 transport run。
一旦 runtime 创建 turn_run，就必须建立 PublicTurnOutputContext。
之后所有 body/tool/task/commit/terminal frame 都必须带同一个 turn anchor。
```

### 3.2 item lifecycle owns visible identity

Codex 的 `ItemStartedNotification`、`AgentMessageDeltaNotification`、`ItemCompletedNotification` 都带 `thread_id / turn_id`，delta 还带 `item_id`。

参考文件：

- `D:\AI应用\openai-codex\codex-rs\app-server-protocol\src\protocol\v2\item.rs`

对本项目的约束：

```text
assistant body 必须绑定 assistant_message_ref/message_id。
tool current_action 必须绑定模型工具请求 tool_call_id。
task bridge 必须绑定同一个 turn/message，而不是每个 task event 重新猜。
```

### 3.3 completed item 和 turn completed 分权

Codex 的 `ItemCompletedNotification` 携带完成后的 item，`TurnCompletedNotification` 携带完成后的 turn。turn completed 是终态信号，不替代 completed item。

对本项目的约束：

```text
assistant_text_final / session_output_commit_ack 才能决定正文完成和 hydrate。
turn_completed 只能关闭 stream，不生成正文，不触发 hydrate，不清理工具。
task terminal 不能从 snapshot 合成 final body。
```

### 3.4 lossless tier

Codex app-server client 明确把 transcript deltas、`ItemCompleted` 和 `TurnCompleted` 归为 lossless tier；best-effort progress 可以丢，但正文、completed item 和 terminal 不能丢。

参考文件：

- `D:\AI应用\openai-codex\codex-rs\app-server-client\src\lib.rs`

对本项目的约束：

```text
assistant_text_delta/final
tool_call_requested
tool_permission_decided
tool_item_started/completed
task_bridge_started
session_output_commit_checked/ack/failed/skipped
turn_completed
```

这些事件在 public stream 中必须可重放、可追踪、不可被 best-effort 进度挤掉。普通 runtime status、内部 step summary、监控心跳只能是 best-effort 或 trace。

## 4. 本项目当前链路核对

### 4.1 已经接近目标的部分

1. `backend/harness/runtime/projection/projector.py` 已经有 `ProjectionLifecycleState`，能校验工具链路：

```text
tool_call_requested
-> tool_permission_decided
-> tool_item_started
-> tool_item_completed
```

2. `backend/harness/runtime/projection/authority.py` 已经会把非法 frame spec 转成 `protocol_diagnostic`，不再只是静默 fallback。

3. `backend/harness/loop/single_agent_turn.py` 的 direct turn commit 已经写 `session_output_commit_checked/ack/failed/skipped`，直接单轮不再是本轮主要缺口。

4. `frontend/src/lib/projection/reducer.ts` 已经按 `PublicProjectionFrame` 更新 `ProjectionLedger`，并且 `current_action` 只接受 `source_event_type=tool_call_requested` 和 `source_authority=model`。

5. 旧的前端文案过滤函数在当前生产搜索里基本已经移除，主剩余风险转移到 raw anchor binding 和 legacy event fallback。

### 4.2 仍然破坏时序的部分

| 位置 | 当前行为 | 问题 | 目标 |
| --- | --- | --- | --- |
| `backend/api/chat.py::_run_chat_to_event_log()` | 发现 `task_executor_scheduled` 后记录 diagnostics 并跳到 `_bridge_task_run_to_chat_stream()` | handoff 事件被当作内部跳转，没有公开 bridge-start frame，也没有不可变 bridge context | 创建 `ChatTaskBridgeContext`，先发 public bridge frame，再桥接 task event |
| `backend/api/chat.py::_bridge_task_run_to_chat_stream()` | 订阅 task event log，按 task offset 拉事件 | 桥接时只有 task_run_id，没有固定 `turn_id/message_ref/sequence_base` | bridge 函数只接受 context，不再重新猜 anchor |

| `frontend/src/lib/store/events.ts::bindStreamSessionAnchor()` | 无 frame 时仍从 raw event 推断 turn/task/run anchor | 前端参与主锚点绑定，容易把旧任务或 task handoff 接错 assistant message | 主消息 anchor 只来自 frame；raw 只允许更新非主视图 diagnostics |
| `backend/runtime/output_boundary/classifier.py` 与 `boundary.py` | final text 仍使用 procedural promise / ack prefix 等语义分类 | 系统可能把模型合法自然语言当“过程承诺”降级 | final body 边界只做协议泄露清理、空输出处理和 commit policy，不做自然语言语义裁决 |

## 5. 目标权威链

### 5.1 控制权威

```text
UserInput
-> ModelTurnDecision
-> ModelOutputItem / ModelActionRequest
-> Runtime Normalizer
-> ActionPermit / ToolPermission
-> Execution
-> Runtime Event Log
-> Public Projection Authority
-> Frontend Ledger
-> Pure Render
```

模型拥有语义调度权。系统不能因为“安全”“权限”“流畅”之名改变模型合法请求。系统的安全边界只包括：

- 工具是否存在。
- 工具参数是否符合协议。
- 权限是否允许执行。
- 输出是否泄露内部协议。
- event 时序是否可证明。
- commit 是否成功。

这些边界都必须产出可追溯事件，不能静默重写。

### 5.2 输出时序权威

```text
ChatRun
-> PublicTurnOutputContext
-> assistant body/tool/task public events
-> optional ChatTaskBridgeContext
-> bridged task public events
-> session_output_commit_checked
-> session_output_commit_ack/failed/skipped
-> scope_retire
-> turn_completed
```

`turn_completed` 永远不能代替 `assistant_text_final` 或 `session_output_commit_ack`。

### 5.3 投影权威

```text
CanonicalPublicEvent
-> ProjectionLifecycleState
-> PublicProjectionFrame
-> ProjectionLedger
-> MessagePublicProjection
```

前端只按 frame 操作，不解释 event 语义。

## 6. 新增结构：只做 refs，不做语义

本计划只允许新增两个内部 context。它们不是 prompt，不暴露给模型，不表达意图，不判断答案，只固定时序和证据。

### 6.1 PublicTurnOutputContext

创建时机：首次拿到 `turn_id + turn_run_id`，通常来自 `harness_run_started` / `single_agent_turn_started`。

字段：

```text
context_id
stream_run_id
session_id
turn_id
turn_run_id
assistant_message_ref
source_turn_event_id
source_turn_event_offset
public_sequence_started_at
created_at
```

职责：

- 为本轮所有 public frame 提供 anchor。
- 创建 assistant projection message 的稳定 `message_id/message_ref`。
- 记录 source event 证据。

禁止：

- 不读取用户语义。
- 不判断 action 类型。
- 不生成 assistant 正文。
- 不决定工具是否执行。

### 6.2 ChatTaskBridgeContext

创建时机：模型合法 `request_task_run` 后，任务生命周期产生 `task_executor_scheduled` handoff 边界。

字段：

```text
bridge_id
stream_run_id
event_log_id
session_id
turn_id
turn_run_id
task_run_id
assistant_message_ref
source_handoff_event_id
source_handoff_event_offset
task_event_start_offset
public_sequence_base
created_at
```

职责：

- 固定 task run 到 chat turn/message 的公开锚点。
- 把 task event log 事件映射到 chat public event clock。
- 记录 task source offset，保证证据可追溯。
- 在 bridge start 和 bridge terminal 时产生结构化 frame。

禁止：

- 不改写 task goal。
- 不从 task snapshot 猜 final answer。
- 不让 frontend 根据 task_run_id 找 assistant message。
- 不保留 `_task_bridge_refs()` 作为运行时兜底权威。

## 7. 固定执行流

### 7.1 普通 direct turn

```text
chat_run_started
-> turn runtime started
-> PublicTurnOutputContext created
-> chat_turn_bound public event
-> assistant_text_delta/final frames
-> tool lifecycle frames, if any
-> session_output_commit_checked
-> session_output_commit_ack/failed/skipped
-> turn_completed
```

要求：

- `chat_turn_bound` 可进入 trace/status，但不能伪造 assistant 正文。
- direct turn 已有 commit ack 代码，实施时只做核验和统一 frame anchor。
- commit failed 必须 pinned，不能假 completed。

### 7.2 request_task_run handoff

```text
model action: request_task_run
-> task_run_lifecycle_started
-> task_executor_scheduled
-> ChatTaskBridgeContext created
-> task_bridge_started public event
-> bridge subscribes task event log from task_event_start_offset
-> each task event normalized with bridge context anchor
-> assistant_text_delta/final or commit events project body/commit
-> task terminal only emits turn_completed/bridge_terminal
```

要求：

- `task_bridge_started` 必须携带 `turn_id / turn_run_id / task_run_id / assistant_message_ref`。
- task bridge 后所有 public frame 的 anchor 都来自 context。
- source task event offset 只作为 trace evidence，不作为前端排序依据。
- 如果 task terminal 到达但没有 output/commit 证据，发 `task_completed_without_output_commit` diagnostic，不合成 final body。

### 7.3 工具生命周期

```text
tool_call_requested
-> tool_permission_decided
-> tool_item_started
-> tool_item_completed
```

要求：

- 已有 `ProjectionLifecycleState` 保留为 projection authority 的状态机。
- bridge context 只提供 anchor 和 public clock，不绕过工具状态机。
- `tool_item_started/completed` 没有合法上游时只能进 protocol diagnostic。

### 7.4 收口

```text
session_output_commit_checked
-> session_output_commit_ack
-> scope_retire transient items
-> turn_completed
```

要求：

- commit ack 是 hydrate 唯一权威。
- turn completed 不 hydrate、不清屏、不替换 body。
- 普通 bridge/status/tool transient item 在 ack 后 retire 到 trace。
- pinned failure、waiting、permission、commit_failed 保留。

## 8. 具体信号线路预规划

本节是实施时的接线图。后续代码只能按这里接，不允许在某个函数里临场猜“这个 event 大概应该显示在哪里”。

### 8.1 信号分级

| 等级 | 信号 | 处理要求 |
| --- | --- | --- |
| L0 锚点信号 | `harness_run_started(turn_run)`、`single_agent_turn_started`、`task_executor_scheduled`、新增 `chat_turn_bound`、新增 `task_bridge_started` | 必须先建立 context，再允许正文、工具、task bridge 进入主投影 |
| L1 无损正文/收口信号 | `assistant_text_delta`、`assistant_text_final`、`assistant_stream_repair`、`session_output_commit_checked/ack/failed/skipped`、`turn_completed` | 不得丢，不得 best-effort，不得由 terminal 替代 |
| L1 无损工具信号 | `tool_call_requested`、`tool_permission_decided`、`tool_item_started`、`tool_item_completed` | 必须按 request -> permission -> started -> completed 状态机接线 |
| L1 无损桥接信号 | `task_bridge_started`、bridged task 的 assistant/tool/commit/terminal public event、`task_bridge_terminal` | 必须使用同一个 `ChatTaskBridgeContext` anchor |
| L2 trace/diagnostic 信号 | `runtime_status`、`runtime_step_summary`、`active_task_steer_accepted`、`input_commit_gate`、`runtime_branch_decided`、`task_run_lifecycle_started` | 只进 trace/status/diagnostics，不创建 assistant body，不创建 current_action |
| 禁止主链路信号 | legacy `answer_candidate`、legacy `assistant_text`、raw `token`、raw `public_timeline`、`task_projection.activities`、terminal snapshot final_answer | 不得直接进入 body/current_action；迁移期最多转 diagnostic 或在唯一迁移点转 typed event，最终删除 |

### 8.2 锚点线路

| 输入 signal | 前置条件 | 后端动作 | public event / frame | 缺失处理 | 禁止接法 |
| --- | --- | --- | --- | --- | --- |
| `chat_run_started` | `stream_run_id/session_id` | 只记录 transport run started | `chat_run_started` -> trace/status | 不创建 assistant message | 禁止用它当 turn/message anchor |
| `harness_run_started` 且含 `turn_run` | `turn_id/turn_run_id/session_id` | 创建 `PublicTurnOutputContext`，生成 `assistant_message_ref` | `chat_turn_bound` -> trace/status，anchor 写全 | 缺 `turn_id/turn_run_id` 则 diagnostic，后续 body/tool fail-closed | 禁止继续靠 frontend raw event 绑定 |
| `single_agent_turn_started` | 已有或可创建 `PublicTurnOutputContext` | 补充 allowed action/type facts | trace/status | 缺 context 时 diagnostic | 禁止生成正文或 current_action |
| `task_executor_scheduled` | 已有 `PublicTurnOutputContext`，且能确定 `task_run_id` | 创建 `ChatTaskBridgeContext`，记录 source handoff event | `task_bridge_started` -> status/trace，anchor 写 `turn_id/message_id/task_run_id` | 缺 turn context 或 task_run_id，pinned diagnostic，不进入 bridge | 禁止直接 break 后无公开 bridge frame |
| task event replay | 已有 `ChatTaskBridgeContext` | 统一用 bridge context anchor，source task offset 进 trace | 对应 canonical public event + frame | context mismatch 或 task_run_id 不一致则 diagnostic | 禁止每个事件调用 `_task_bridge_refs()` 猜 anchor |

### 8.3 assistant body 线路

| 输入 signal | 前置条件 | 后端动作 | projection frame | 缺失处理 | 禁止接法 |
| --- | --- | --- | --- | --- | --- |
| `assistant_text_delta` | `PublicTurnOutputContext`；bridged 时还需 `ChatTaskBridgeContext` | 归一为 canonical body delta，写 source refs | `body_append / slot=body / source_authority=model` | 无 context -> protocol diagnostic，不进 body | 禁止前端直接把 raw token 拼到 message |
| `assistant_text_final` | 同上 | finalize 当前 body segment，不提交 hydrate | `body_finalize / slot=body` | 无 context -> diagnostic；空文本按 output boundary 处理 | 禁止触发 hydrate 或清理工具 |
| `assistant_stream_repair` | 已有 body segment 和 context | 修复当前 body segment | `body_finalize` 或 repair spec | 无可修复 segment -> diagnostic | 禁止覆盖 unrelated message |
| raw `token` | 仅迁移期，且来源明确为模型正文 | 唯一迁移点转 `assistant_text_delta`；Phase 4 后删除 | 同 delta | 无 context -> diagnostic | 禁止长期作为 public body 主入口 |
| legacy `answer_candidate/assistant_text` | 仅内部/旧 worker 字段 | 不进入 body；生产 emitter 必须改发 typed body | diagnostic/trace | 若仍是唯一输出，暴露缺 emitter 的错误 | 禁止转成 `assistant_text_final` 长期兜底 |
| terminal/snapshot `final_answer` | 无 | 不生成 body | 无 body frame；必要时 diagnostic | `task_terminal_final_without_output_event` | 禁止 `_append_task_bridge_terminal()` 合成 final body |

### 8.4 工具生命周期线路

| 输入 signal | 前置条件 | 后端动作 | projection frame | 缺失处理 | 禁止接法 |
| --- | --- | --- | --- | --- | --- |
| `model_action_admission_checked` 且 `action_type=tool_call` | context 存在；`tool_name/tool_call_id` 存在 | 生成 `tool_call_requested` | `item_upsert / slot=current_action / source_authority=model` | 缺字段 -> diagnostic | 禁止从 tool started 反推 current_action |
| 同一 admission 的 permission | 已有同 tool_call_id 的 request | 生成 `tool_permission_decided` | allow -> trace；ask/deny -> pinned | request 不存在或 offset 不晚于 request -> diagnostic | 禁止 permission 改写工具语义 |
| `tool_item_started` | 已有 allow/auto_allow permission，`permission_decision_id` 匹配 | 记录执行开始 | trace-only，更新证据 | 无 request/permission 或乱序 -> diagnostic | 禁止进入主视图生成工具卡 |
| `turn_tool_observation_recorded/task_tool_observation_recorded/tool_observation` | 已有 started lifecycle | 生成 `tool_item_completed` | success -> retire current_action 到 trace；failed -> pinned | 无 started 或乱序 -> diagnostic | 禁止工具结果写 assistant body |

### 8.5 task bridge 线路

| 输入 signal | 前置条件 | 后端动作 | projection frame | 缺失处理 | 禁止接法 |
| --- | --- | --- | --- | --- | --- |
| `task_run_lifecycle_started` | task run 已创建，可追到 action request | 只作 trace 证据 | trace/status | 缺 refs -> diagnostic | 禁止创建主消息 |
| `task_executor_scheduled` | `PublicTurnOutputContext + task_run_id` | 创建 `ChatTaskBridgeContext` | `task_bridge_started` | 缺 context -> pinned diagnostic | 禁止无 frame 直接转后台监听 |
| bridged task `model_action_admission_checked` | bridge context | 映射为 chat public tool request/permission，anchor 来自 bridge | 工具 frames | task_run_id 不匹配 -> diagnostic | 禁止使用 task 自己的 message_ref 覆盖 chat message |
| bridged task `tool_item_started/completed` | bridge context + tool lifecycle state | 映射为 chat public tool execution/result | trace/retire/pinned | lifecycle 缺上游 -> diagnostic | 禁止裸工具进主视图 |
| bridged task `session_output_commit_*` | bridge context | 映射为 chat public commit frame | checked/ack/failed/skipped | commit message_ref 与 context 冲突 -> diagnostic | 禁止 task completed 代替 commit ack |
| `task_run_lifecycle_finished/task_run_terminal_observed` | bridge context | 发 `task_bridge_terminal` + `turn_completed` | terminal/status，非 body | 无 commit/output 时 pinned diagnostic | 禁止 terminal 生成正文、hydrate、清屏 |

### 8.6 commit 与 terminal 线路

| 输入 signal | 前置条件 | 后端动作 | projection frame | 缺失处理 | 禁止接法 |
| --- | --- | --- | --- | --- | --- |
| `session_output_commit_checked` | context + content hash | 记录 commit gate | commit checked trace/status | 无 context -> diagnostic | 禁止 hydrate |
| `session_output_commit_ack` | checked 已出现或有可追踪 receipt | 标记 committed，生成 hydrate key | `commit_ack` + `scope_retire` | 无 checked -> diagnostic 但 ack 仍可 pinned/trace 暴露 | 禁止 task completed 伪装 ack |
| `session_output_commit_failed` | context | pinned commit failure，保留 live body | `commit_failed / pinned` | 无 body 也必须显示失败 | 禁止覆盖成 completed |
| `session_output_commit_skipped` | context | trace/status，按 reason 表示不持久化 | commit skipped | 无 reason -> diagnostic | 禁止假 hydrate |
| `turn_completed` | stream terminal | 只关闭 turn/run | `turn_terminal` | 若缺 commit 且有 body，显示 diagnostic/pending commit 状态 | 禁止生成正文、清工具、hydrate |
| raw `done/error/stopped` | transport terminal | 归一为 `turn_completed` | `turn_terminal` 或 pinned error | error/stopped 保留可见失败 | 禁止从 `done.content` 取正文 |

### 8.7 辅助信号线路

| 输入 signal | 线路 | 说明 |
| --- | --- | --- |
| `runtime_branch_decided` | trace/status only | 展示运行分支事实，不决定主消息内容 |
| `input_commit_gate` | trace only | 只记录用户输入提交状态，不创建 assistant 投影 |
| `runtime_status` | status/trace | 可显示运行状态，但必须带 frame；不能生成正文 |
| `runtime_step_summary` | trace only，除非后端明确发 pinned/final_result frame | 不允许前端从 summary 文案判断主视图 |
| `active_task_steer_accepted` | trace/status | 控制动作回执，不生成普通 assistant 正文 |
| `retrieval` | retrieval attachment 或 trace | 不直接写 assistant body；回答仍由模型 body event 输出 |
| `output_boundary` | diagnostics/trace | 只说明边界处理，不替换模型正文 |

### 8.8 接线完整性规则

1. 每个 L1 signal 必须有且只有一个 canonical public event 归宿。
2. 每个 main-visible frame 必须有完整 anchor；缺 anchor 不允许前端补。
3. 每个 bridged task event 必须同时有两个 offset：
   - `event_offset`：chat public stream 排序权威。
   - `source_task_event_offset`：证据追溯，不参与前端排序。
4. 每个 tool lifecycle frame 必须有同一个 `tool_call_id`；started/completed 还必须有匹配的 `permission_decision_id`。
5. 每个 commit frame 必须有 `message_ref/message_id` 或从 context 派生的稳定 assistant message ref。
6. 每个 terminal frame 必须只能终止 stream，不能替代 body、commit、hydrate、scope retire。
7. 每个 legacy signal 必须有明确迁移终点；不得以兼容为理由继续进入主视图。
8. 每个 diagnostic 必须进入 trace 或 pinned failure，不能只写后端日志。

```text
assistant_text_final / commit_ack / turn_completed 均使用同一 turn/message anchor。
```

仍需后续真实场景继续验证：

1. 模型实际触发 `request_task_run` 后的 `task_bridge_started -> bridged task events -> task_bridge_terminal -> turn_completed` 全链路。
2. task terminal 缺 output/commit evidence 时的 pinned commit failure UI 呈现。
3. 断线重连 replay 下同一 assistant message 不重复、不串到旧 task。

## 16. 工具生命周期 started 后卡住修复记录

实施时间：2026-06-13

触发现象：

```text
tool_call_requested
-> tool_permission_decided
-> tool_item_started
```

之后长时间没有 `tool_item_completed / assistant_text_final / turn_completed`。这说明工具生命周期投影状态机已经存在，但执行服务层在 started 后缺少足够及时的完成、失败或诊断收口。该问题不是模型决策问题，也不是 permission 拦截问题；模型已经发起合法工具，系统必须把执行服务的真实状态透明回传。

修复内容：

1. `search_files` 不再先收集完整项目文件列表再筛选，改为可剪枝、可早停的路径扫描；达到 `max_results` 后立即返回，避免 `roots=["."]` 时对整个工作区做无意义全量遍历。
2. native runtime 的 `search_files` 与非 native `SearchFilesTool` 统一到同一条成熟路径，避免一个入口已修、另一个入口继续全量扫描。
3. single-agent turn 工具 batch 默认超时从 300 秒收紧到 45 秒；task-run 工具 batch 同步收紧到 45 秒。显式 `execution_policy.tool_batch_timeout_seconds` 仍是权威配置。
4. 并行工具 batch 超时后不再长期等待不可取消的工具任务退出；取消后只做短 drain，然后立即生成 timeout observation，由现有 `tool_item_completed` failure/pinned 投影收口。
5. `tool_batch_group_started` 增加 runtime status projection，显示“正在执行 N 个工具”。`tool_item_started` 仍保持 trace-only，不绕过 `tool_call_requested -> permission -> started -> completed` 状态机。

权威边界：

```text
模型仍决定是否调用工具、调用哪个工具、如何基于 observation 继续。
系统只提供执行服务 SLA、可见运行态、超时 observation 和 projection evidence。
前端仍只消费 public_projection_frame，不猜测工具语义。
```

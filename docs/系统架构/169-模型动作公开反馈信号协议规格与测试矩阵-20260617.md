# 模型动作公开反馈信号协议规格与测试矩阵

日期：2026-06-17

## 1. 目标

本规格用于修正单轮 agent 在 `json_action_required` 与 provider-native tool call 同时存在时，公开正文信号不稳定的问题。

目标不是强制模型在每个低层工具前都输出正文，而是让每一种模型意图都有准确、单一、可追踪的信号通道：

- 控制动作必须走 JSON action。
- 普通工具可以走 provider-native tool call。
- 用户可见反馈必须来自模型产生的公开语义字段或 native tool preamble。
- 系统不得替模型伪造“已收到”“正在处理”“我会看看”等语义正文。
- 前端只展示 canonical public projection frame，不根据工具轨迹猜正文。

本规格修订 `119-用户输入模型回应义务协议修复计划书-20260613.md` 的强约束方向：旧计划要求在回应义务开启时裸工具直接 invalid；本规格改为将缺失公开反馈记录为 contract gap，并按场景决定是否要求后续修复，不用统一硬拦截普通工具。

## 2. 当前问题

当前链路中存在三个信号被同一个 `requires_json_action_protocol` 间接耦合：

1. 控制动作运输方式：`request_task_run`、`active_work_control`、`ask_user`、`block` 必须 JSON。
2. 普通工具运输方式：`tool_call` 可以是 JSON action，也可以是 provider-native tool call。
3. 正文发布方式：assistant text delta 在 JSON action required 时被关闭，只能通过 action 字段或最终提交发布。

关键证据：

- `backend/harness/runtime/compiler.py` 中 `_single_agent_turn_effective_control_capabilities()` 因控制动作存在而设置 `requires_json_action_protocol=True`。
- `_single_agent_turn_output_contract()` 同时输出 `format=json_action` 和 `native_tool_calls.enabled=true`。
- `backend/harness/loop/single_agent_turn.py` 调用模型时，当 `single_agent_requires_json_action=True`，传入 `allow_assistant_text_delta=False`。
- native tool response 的 assistant content preamble 可以被解析成 `public_progress_note`，但协议没有把它定义为正式公开反馈通道。
- 无 preamble 的 native tool call 只进入 `contract_gaps`，仍可执行，因此长工具链可以只剩工具轨迹。

## 3. 目标权威链

```text
UserInput / RuntimeState
-> RuntimeStartPacket
-> ModelTurnDecision
-> ModelResponseProtocol
-> ModelActionRequest / NativeToolCall
-> ActionPermit
-> ToolExecution
-> ToolObservation
-> RuntimePublicProjection
-> FrontendProjection
-> FinalCommit
```

职责边界：

| 层 | 职责 | 禁止 |
| --- | --- | --- |
| RuntimeStartPacket | 给模型说明可用动作、运输方式、公开反馈义务 | 重新判断用户语义意图 |
| ModelTurnDecision | 决定回答、询问、阻塞、控制动作或工具调用，并生成公开语义文本 | 自行授权工具或绕开协议 |
| ModelResponseProtocol | 解析 JSON action、native tool call、assistant preamble | 把工具轨迹改写成用户正文 |
| ActionPermit | 授权或拒绝动作 | 替模型补公开回应 |
| ToolExecution | 执行工具并返回 observation | 把 observation 当成最终答复 |
| RuntimePublicProjection | 投影模型公开反馈、工具生命周期、最终正文 | 根据工具名/状态猜正文 |
| FrontendProjection | 按 canonical frame 展示正文和工具窗口 | 过滤掉合法正文或伪造正文 |

## 4. 信号定义

### 4.1 动作运输信号

| 信号 | 含义 | 允许来源 | 消费方 |
| --- | --- | --- | --- |
| `json_action` | 结构化模型动作 | assistant content JSON object | `model_action_protocol` |
| `native_tool_call` | provider-native 普通工具调用 | provider tool call field | `single_agent_turn` native parser |
| `assistant_body` | 自然语言正文流或最终正文 | assistant text delta/final | output stream projection |

### 4.2 公开反馈信号

| 信号 | 用途 | 进入正文表面条件 |
| --- | --- | --- |
| `final_answer` | 最终回答 | respond action 通过 commit gate |
| `user_question` | 向用户询问 | ask_user action 通过 commit gate |
| `blocking_reason` | 阻塞说明 | block action 通过 commit gate |
| `public_progress_note` | 阶段反馈或工具前公开判断 | `presentation_source=model_action.public_progress_note` |
| `public_action_state.current_judgment` | 当前公开判断 | 作为 public_progress_note 的补充，不替代最终回答 |
| `native_tool_preamble` | native tool call 同时携带的 assistant content | 解析为 `public_progress_note` 并投影为 `runtime_step_summary` |

### 4.3 诊断信号

| 信号 | 含义 | 是否用户正文 |
| --- | --- | --- |
| `contract_gaps.public_response_missing_for_native_tool_call` | native 工具调用缺少公开反馈 | 否 |
| `contract_gaps.public_progress_note_missing_for_tool_call` | JSON tool_call 缺少进度说明 | 否 |
| `model_protocol_violation` | 输出无法被当前协议唯一解析 | 否 |
| `agent_contract_feedback_required` | 协议修复反馈，需要模型重新提交合法动作 | 否 |

诊断信号只能进入 trace、runtime control 或下一轮模型上下文，不得被前端当作正文展示。

## 5. 运输决策表

| 场景 | 允许动作 | 动作运输方式 | 公开反馈通道 | 缺失反馈处理 |
| --- | --- | --- | --- | --- |
| 最终答复 | `respond` | JSON action 或 final assistant body，按阶段决定 | `final_answer` / assistant final | 缺失则 invalid |
| 向用户提问 | `ask_user` | JSON action | `user_question` | 缺失则 invalid |
| 阻塞 | `block` | JSON action | `blocking_reason` | 缺失则 invalid |
| 新开持续任务 | `request_task_run` | JSON action | `public_progress_note` + `task_contract_seed` | 控制动作缺字段 invalid |
| 控制当前任务 | `active_work_control` | JSON action | `public_progress_note` 或 control response 字段 | 控制动作缺字段 invalid |
| 恢复可恢复任务 | `resume_recoverable_work` | JSON action | `public_progress_note` + `recovery_resume` | 控制动作缺字段 invalid |
| 用户输入触发的普通工具 | `tool_call` | native tool call 或 JSON action | native preamble 或 `public_progress_note` | 记录 gap；不伪造正文；必要时在后续模型调用中反馈 |
| 低层连续读取/搜索 | `tool_call` | native tool call | 可省略 | 只显示工具生命周期 |
| 工具失败、范围变化、阶段结束后继续工具 | `tool_call` | native tool call 或 JSON action | native preamble 或 `public_progress_note` | 记录 gap；下一次模型上下文必须带回 gap |
| 工具预算耗尽收口 | `respond` / `ask_user` / `block` | JSON action | `final_answer` / `user_question` / `blocking_reason` | 不允许继续工具 |
| 协议恢复 | `respond` / `ask_user` / `block` | JSON action | 对应字段 | 不允许 native tool call |

## 6. RuntimeStartPacket 合同形状

`output_contract.action_protocol` 应明确拆出以下字段，而不是只用 `format=json_action` 表达整轮规则：

```json
{
  "transport_decision_table": {
    "control_actions": {
      "transport": "json_action",
      "native_tool_transport_enabled": false
    },
    "ordinary_tool_calls": {
      "transport": "provider_native_tool_call_or_json_tool_call",
      "native_tool_transport_enabled": true
    },
    "assistant_body": {
      "enabled_when": "non_json_control_phase_or_final_commit",
      "disabled_reason_when_json_required": "avoid publishing invalid protocol text"
    }
  },
  "public_feedback_contract": {
    "feedback_must_be_model_authored": true,
    "system_must_not_synthesize_user_semantic_text": true,
    "json_action_feedback_fields": [
      "public_progress_note",
      "public_action_state.current_judgment",
      "final_answer",
      "user_question",
      "blocking_reason"
    ]
  },
  "native_tool_feedback_contract": {
    "assistant_content_preamble_is_public_feedback": true,
    "projection_target": "runtime_step_summary",
    "missing_preamble_policy": "record_contract_gap_without_synthesizing_body",
    "low_level_tool_calls_may_omit_feedback": true,
    "stage_change_or_failure_should_emit_feedback": true
  }
}
```

## 7. Prompt 规则

`_runtime_projection_instruction()` 不能只输出“当前运行事实”。它应该把动态 JSON 中最关键的运行事实改写成模型可执行的短规则：

```text
你本轮可以选择：respond、ask_user、block、request_task_run、active_work_control、tool_call。
控制动作必须输出 JSON action，不要用 provider-native tool call 表达控制决定。
普通工具可以使用 provider-native tool call。
如果使用 provider-native tool call 且需要回应用户或说明阶段判断，请在同一条 assistant content 中先写一句用户可见说明；系统会把这句 preamble 作为 public_progress_note 投影到正文。
低层连续读取、搜索、列目录可以不逐次解释，但当用户在追问、工具失败、范围变化、阶段结束或连续多批工具后继续执行时，必须给出公开判断。
不要在 JSON 外输出控制动作正文；不要把工具名、协议字段、内部事件号写给用户。
```

这段是 agent prompt，不是开发说明；它直接告诉模型身份内的行为边界和输出义务。

## 10. 实施顺序建议

这不是本规格阶段要执行的改动，只是后续实现顺序：

1. 先改合同文本和 prompt 渲染，不改变运行行为。
2. 补编译器合同测试，证明模型能看到准确协议。
3. 补 native preamble 到 `runtime_step_summary` 的端到端测试。
4. 补缺失 preamble 的 gap 测试，确认不伪造正文、不误拦截低层工具。
5. 再决定是否将重复 gap、失败后 gap、用户追问 gap 升级为协议反馈。
6. 最后清理 119 号计划中“裸工具一律 rejected”的过强测试或旧断言。

## 11. 明确不做

- 不打开 JSON required 阶段的 raw assistant delta 作为正文兜底。
- 不由系统生成“正在处理”“已收到”等语义回应。
- 不要求每个低层工具都 preamble。
- 不把 contract gap 直接展示给普通用户。
- 不把前端作为修复协议缺陷的主战场。

## 12. 验收标准

协议改动完成后，应满足：

1. 模型 packet 中能明确区分控制动作运输方式和普通工具运输方式。
2. native tool preamble 被正式定义为公开反馈信号。
3. 缺失 preamble 时不会伪造正文，也不会让前端误以为有正文。
4. 工具生命周期继续只作为工具窗口，不抢正文主位。
5. 用户最终看到的正文只来自模型公开语义字段、native preamble 或 final answer。
6. 测试覆盖成功、缺失反馈、协议失败、投影、最终提交五类链路。

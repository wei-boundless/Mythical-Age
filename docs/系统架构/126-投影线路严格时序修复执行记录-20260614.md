## 1. 已修复线路

| 线路 | 修复结果 |
| --- | --- |
| public frame offset | `PublicProjectionFrame.event_offset` 只使用 public sequence，不再读取 payload `event_offset/offset`。 |
| lifecycle offset 校验 | 工具 lifecycle 状态机只使用 public sequence 校验 request/permission/start/completed 顺序。 |
| active steer projection | `active_task_steer_accepted` 改为可见 `status` item，不进入 assistant body。 |
| tool identity | 新增 `runtime.shared.tool_identity`，集中生成/读取 canonical `tool_call_id` 和 fallback `permission_decision_id`。 |
| chat bridge | `_tool_call_requested_data()` 不再用 `request_id` 充当 `tool_call_id`。 |
| permission fallback | task executor、single agent turn、chat completion fallback 统一到 canonical helper；缺 admission id 时使用 `admission:{tool_call_id}`。 |
| frontend body ordering | `ProjectionLedger.body.blocks` 保存 body segment offset，ChatMessage 按 body block 与 timeline item 混排。 |
| frontend tool lifecycle | 工具 timeline display key 改为 `tool_call_id || tool_lifecycle_id`，completed 更新同一项，不再按 frame id 拆成第二项。`tool_lifecycle_id` 是缺失 `tool_call_id` 时的执行实例 fallback，不反过来覆盖模型工具调用身份。 |
| frontend block rendering | 分块正文使用各自 `block.text` 判断显示，不再依赖整条消息的全局正文状态；带图片消息不会把正文块错误渲染成图片。 |
| frontend timeline contract | `PublicChatTimelineItem` 补齐 `source_event_type/sourceEventType`，前端不再丢失事件来源类型。 |
| status timeline identity | 非工具 status timeline item 改用 `frame_id/source_event_id`，不再用稳定 `task-stage-status:*` 合并到旧位置；`projection.status` 仍保留最新状态。 |
| tool completion identity | `tool_item_completed` 从 observation invocation/admission ref 取 `tool_lifecycle_id` 和 `permission_decision_id`，不再把 `tool_call_id` 或 `admission:{tool_call_id}` 当完成身份。 |
| model progress body | `step_summary_recorded` 或 `model_action_admission/model_action_admission_checked` 中的公开反馈统一转成 `runtime_step_summary -> status_trace`；前端运行中按正文块显示，不混入工具窗口。 |
| tool window surface | 工具 timeline item 统一渲染为可折叠工具控制台窗口；主视图只显示一次系统动作、目标、状态，展开显示 `$ <tool> <target/args>` 和系统返回摘要。内部时序、offset 和调用号只进 trace/debug，不进主聊天窗口。 |
| final body surface | 正文 `finalized/committed` 后，主消息只显示正文；运行中才按 offset 混排正文反馈和工具窗口。 |

## 3.1 2026-06-14 晚间卡住问题复盘

截图中多个 `读取文件 mario.html 运行中` 没有返回，不是前端单纯不刷新。真实 SSE 证据显示：

```text
tool_call_requested: item_id=call_00_... / tool_call_id=call_00_...
tool_permission_decided: permission_decision_id=admission:model-action:...
tool_item_started: item_id=toolinv:... / tool_call_id=call_00_... / permission_decision_id=admission:model-action:...
tool_item_completed: item_id=toolinv:... / tool_call_id=call_00_... / permission_decision_id=admission:toolinv:...
public_projection_frame: hidden diagnostic tool_completed_without_started_lifecycle
```

根因：

- 公开工具窗口的身份没有锁死在 `tool_call_id`，started/completed 使用了内部执行实例 `toolinv:*` 作为 `item_id`。
- completion 桥接把 `toolinv:*` 伪造成 `admission:toolinv:*`，污染了 permission identity。
- public projection lifecycle 在 completed 阶段又做了一次 permission id 匹配，导致已经执行完成的工具无法按工具生命周期收口。
- `model_action_admission` 中的 `public_progress_note/current_judgment` 被挂在 `tool_call_requested` 数据里；前端为了让工具窗口降噪又不把这些字段当正文，所以用户看起来像 agent 直接无反馈调用工具。

修复后的硬规则：

| 信号 | 公开通道 | 公开主键 | 显示责任 |
| --- | --- | --- | --- |
| `public_progress_note/current_judgment/next_action` | `runtime_step_summary` / status_trace | `frame_id/source_event_id` | 作为模型正文反馈显示，排在工具窗口前。 |
| `tool_call_requested` | control | `item_id=tool_call_id` | 创建一个工具窗口。 |
| `tool_item_started` | control/trace | `item_id=tool_call_id` | 标记同一个工具窗口进入执行；`tool_lifecycle_id=toolinv:*` 只作内部执行实例引用。 |
| `tool_item_completed` | control/trace | `item_id=tool_call_id` | 收口同一个工具窗口；不得用 `toolinv:*` 生成新的公开主键或 permission id。 |

当前代码修复：

- `backend/api/chat.py`
  - `_tool_action_public_events()` 先派生 `runtime_step_summary`，再派生 `tool_call_requested/tool_permission_decided`，恢复正文和工具控制双通道。
  - `_tool_call_requested_data()` 和 `TOOL_CALL_REQUESTED_EVENT` allowlist 都移除 `public_progress_note/public_action_state`，工具事件不再夹带模型反馈。
  - `_tool_call_requested_data()`、`_tool_item_started_data()`、`_tool_item_completed_data()` 统一 `item_id=tool_call_id`。
  - `_permission_decision_id_from_observation()` 不再把 `toolinv:*` 或 `admission:toolinv:*` 当成公开 permission identity。
- `backend/harness/runtime/projection/projector.py`
  - `tool_item_started` 记录 `tool_lifecycle_id`，但公开 `item_id` 仍是 `tool_call_id`。
  - `tool_item_completed` 只要求同一 `tool_call_id` 已 started 且时序在 started 之后；completion 不再重新匹配 permission id。
  - completion frame 使用 started 记录中的 permission id，避免 observation 桥接漂移污染公开收口。
- `frontend/src/lib/projection/reducer.ts`
  - timeline item 显式透传 `toolCallId/toolLifecycleId/toolName`，避免 frame 级身份丢失后掉回 frame id 或 `toolinv` 合并。

## 4. 已清理的旧主权威

当前主显示权威：

- 正文显示主权威是 `MessagePublicProjection.bodyBlocks[]` 的 `firstOffset/lastOffset`。
- 工具/状态显示主权威是 `MessagePublicProjection.timeline[]` 的 `eventOffset/updatedEventOffset`。
- `bodyEventOffset` 只保留为兼容元数据，不再用于 ChatMessage 的正文/工具混排。
- `frame:tool-completed` 这类按完成帧拆工具行的旧显示方式已改为 lifecycle-keyed upsert。
- 后端 public frame offset 由 public sequence 产生，不从 payload `event_offset/offset` 反读。
- `model_action_admission/model_action_admission_checked` 可以拆出模型反馈，但只能生成 `runtime_step_summary`；工具请求事件不得携带正文反馈字段。
- `ChatMessage` 运行中把 `source_authority=model && status_trace/status` 的进展项转成正文块；工具项只能进入 `PublicTimelineActivity` 的 `tool_window` 表面。
- `ChatMessage` 在正文 finalized/committed 后隐藏工具轨迹，主消息最终态只保留模型正文。

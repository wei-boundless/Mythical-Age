# 工具生命周期投影修复记录 - 2026-06-15

## 问题

前端工具窗口能显示 `tool_call_requested`，但后续 `tool_item_started` / `tool_item_completed` 没有稳定接回同一个窗口，表现为工具长时间保持“运行中”。

## 根因

后端 `ProjectionLifecycleState` 把 `permission_decision_id` 当成 started 阶段的强匹配条件。实际运行里同一个工具调用的许可身份存在两种表现：

- permission frame 使用规范化后的 `admission...:<tool_call_id>`。
- started event 可能只带原始 `admission...`，或者不带完整规范化后缀。

两者语义上属于同一个已获准工具调用，但字符串不完全相等，导致投影权威把 started 降级为 hidden diagnostic：`tool_started_without_allowed_permission`。随后 completed 找不到已开始的生命周期，又被降级为 `tool_completed_without_started_lifecycle`。

## 修复

工具生命周期的公开投影主键统一以 `tool_call_id` 为准：

- `tool_call_requested` 记录 `tool_call_id`、scope、permission 状态。
- `tool_permission_decided` 只负责确认该 `tool_call_id` 已允许。
- `tool_item_started` 只要在同 scope 下能找到已允许的 `tool_call_id`，就接入生命周期，并使用记录中的 permission id 写回投影 spec。
- `tool_item_completed` 继续要求同一 `tool_call_id` 已 started，再退休同一个工具窗口。

这样 permission 仍是准入权威，但不再成为前端生命周期合并的第二主键。

## 验证

- `python -m py_compile backend/harness/runtime/projection/projector.py backend/api/chat.py backend/harness/runtime/session_timeline.py`
- 最小投影探针覆盖 permission id 不完全一致的场景，结果为 request → permission → started → retire，无 hidden diagnostic。
- `npx tsc --noEmit`

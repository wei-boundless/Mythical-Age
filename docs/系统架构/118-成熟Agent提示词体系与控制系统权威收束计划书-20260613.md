## 1. 目标结论

本计划的目标不是给现有系统再叠一层“更强 prompt”，而是把成熟 agent 的控制框架落实为明确的权威链：

```text
RequestFacts
-> BoundaryPolicy
-> ContextCandidates
-> ModelTurnDecision
-> ActionPermit
-> RuntimeStartPacket
-> ExecutionLoop
-> TurnStateAuthority
-> OutputBoundary
```

当前项目已经具备不少成熟 agent 的组件形态：`TurnInputFacts`、`ModelActionRequest`、`AdmissionDecision`、`ActionPermit`、`RuntimeInvocationPacket`、`PromptAssemblyResult`、`DynamicContextProjection`、`TaskGraphRuntimeEnvelope`、`ToolObservation`、`PublicProjection` 等。但核心问题是：**组件多，权威仍有重叠；诊断多，强制边界还不够硬；旧字段仍能参与新运行语义。**

本计划要求后续实施时优先修结构，而不是靠提示词补漏洞。最终状态应达到：

1. prompt precedence 从诊断信息升级为编译器强制规则。
2. task goal / intent 只有一个当前 turn canonical contract，旧 `goal_frame`、`task_goal_spec` 等不能继续裁决目标。
3. 权限只由 `ActionPermit` / permission policy 链授权，tool executor 只做参数、资源和执行边界校验，不重新裁决用户目标。
4. TaskGraph runtime semantics 不再把 legacy fields 作为运行 manifest 常驻产物。
5. turn / active work / pending approval / interrupt / resume / finalization 有单一状态权威。
6. 输出层只消费 canonical result / summary / refs，不从 raw tool logs 或运行噪声拼最终答案。

## 2. 当前设计差距

### 2.1 Prompt 体系：已有分层，缺少强制优先级

外部参考中，Claude Code 的 prompt 体系有几个关键机制：

- `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 用于区分稳定 prompt 前缀和动态 prompt 内容：`D:\AI应用\claude-code-nb-main\constants\prompts.ts:114`。
- `splitSysPromptPrefix()` 在 API 层把 system prompt 按 cache scope 分块：`D:\AI应用\claude-code-nb-main\utils\api.ts:321`。
- `buildEffectiveSystemPrompt()` 明确处理 override、coordinator、agent、custom、default prompt 的优先级：`D:\AI应用\claude-code-nb-main\utils\systemPrompt.ts:41`。
- 工具权限、hook、prompt injection 被写入基础系统约束，而不是散落在业务 prompt 中：`D:\AI应用\claude-code-nb-main\constants\prompts.ts:189`。

本项目已有 `PromptAssemblyService`、cache boundary、prompt precedence report：

- `backend/prompt_library/assembly.py:14`
- `backend/prompt_library/assembly.py:135`
- `backend/prompt_library/assembly.py:240`

但当前 `_prompt_precedence_report` 明确写着 `diagnostic_only_preserves_requested_order`。这意味着优先级只被报告，不被强制执行。成熟 agent 不应允许调用方用传入顺序破坏 system / agent / runtime / tool / skill / project / contract 的边界。

### 2.2 权限体系：ActionPermit 已成型，但 executor 仍有二级裁决

Codex 的权限协议有明确的授权作用域：

- `PermissionGrantScope` 支持 turn / session：`D:\AI应用\openai-codex\codex-rs\protocol\src\request_permissions.rs:12`。
- `RequestPermissionsResponse` 包含 permissions、scope、strict_auto_review：`D:\AI应用\openai-codex\codex-rs\protocol\src\request_permissions.rs:57`。
- thread settings 同时记录 approval policy 和 sandbox policy：`D:\AI应用\openai-codex\codex-rs\app-server\src\thread_state.rs:222`。

本项目已有：

- `ActionPermit`：`backend/harness/loop/action_permit.py:10`
- `admit_model_action()`：`backend/harness/loop/admission.py:44`

但 `ToolRuntimeExecutor` 仍在执行前调用 `runtime_tool.check_permissions()`：

- `backend/runtime/tool_runtime/tool_executor.py:542`
- `backend/runtime/tool_runtime/tool_executor.py:919`

这不一定全错，因为工具仍需要资源级、参数级、文件作用域级校验；但若它拥有“是否允许执行这个动作”的最终语义权，就会和 `ActionPermit` 形成双重权威。成熟方案应把权限裁决集中在 permit / operation gate，tool executor 只消费 permit receipt 并做低层边界校验。

### 2.3 Task goal / intent：旧字段仍参与新任务裁决

本项目在 `TaskRequirementContract` 中仍读取旧 goal 入口：

- `current_turn.get("task_goal_spec") or current_turn.get("goal_frame")`：`backend/task_system/contracts/task_requirement_contracts.py:76`
- `_resolve_task_goal_type()` 继续从 `task_goal_spec`、`semantic_task_type`、`task_goal_type`、`task_contract_seed` 多处取值：`backend/task_system/contracts/task_requirement_contracts.py:238`

这违反了成熟 agent 的一个关键原则：当前 turn 的语义决策必须来自当前模型动作合同或显式 runtime contract，旧 frame 只能作为历史证据或迁移诊断，不能继续参与裁决。

### 2.4 Graph runtime semantics：legacy fields 仍进入运行 manifest

`compile_runtime_semantics_manifest()` 会收集 legacy diagnostics 并生成 `legacy_fields`：

- `backend/task_system/runtime_semantics/compiler.py:17`
- `backend/task_system/runtime_semantics/compiler.py:27`
- `backend/task_system/runtime_semantics/compiler.py:46`

这比直接使用旧字段更安全，但仍把 legacy 作为运行 manifest 的常驻结构。成熟方案应将旧字段归入 publish/preflight 阶段的阻断或迁移报告，而不是 runtime execution 的输入资产。

### 2.5 RuntimeCompiler：装配层承担过多职责

`RuntimeCompiler` 同时处理 prompt assembly、dynamic context、project instructions、tool catalog、artifact scope、task contract、message rendering、sanitizer、cache manifest：

- `backend/harness/runtime/compiler.py:97`
- `backend/harness/runtime/compiler.py:341`
- `backend/harness/runtime/compiler.py:768`
- `backend/harness/runtime/compiler.py:1345`
- `backend/harness/runtime/compiler.py:1372`

成熟架构中，RuntimeStartPacket 层只应该组装已经决定的事实、上下文、权限和 prompt；不应该重新推断 intent、修复缺失决策或把旧字段合并成新语义。

### 2.6 Turn state：状态分散，缺少 Codex 式 turn 权威

Codex 的 `ThreadState` 集中管理 active turn、pending interrupts、rollback、terminal turn、listener generation：

- `D:\AI应用\openai-codex\codex-rs\app-server\src\thread_state.rs:74`
- `active_turn_snapshot()`：`D:\AI应用\openai-codex\codex-rs\app-server\src\thread_state.rs:136`
- `track_current_turn_event()`：`D:\AI应用\openai-codex\codex-rs\app-server\src\thread_state.rs:140`

本项目有 event log、runtime objects、state index、run monitor、graph loop、frontend projection，但缺少一个明确的 `TurnStateAuthority` 来统一：

- 当前 turn 是否仍 active。
- 哪个 approval / interrupt / resume 请求属于哪个 turn。
- active work 是候选、已绑定、已完成还是已失效。
- 哪个 final answer 是 canonical。
- 前端公开状态从哪个状态快照投影。

## 4. 目标权威链

### 4.1 固定执行流

```text
1. RequestFacts
   输入：用户消息、会话元数据、编辑器上下文、active work candidate
   输出：TurnInputFacts
   禁止：选择任务类型、选择工具、重写目标

2. BoundaryPolicy
   输入：TurnInputFacts、项目固定端口/安全约束、collaboration mode、permission mode
   输出：BoundaryDecision
   禁止：替模型选择具体任务方案

3. ContextCandidates
   输入：TurnInputFacts、BoundaryDecision、memory/file/tool/project instruction candidates
   输出：ContextCandidateSet
   禁止：把候选材料改写成隐藏指令

4. ModelTurnDecision
   输入：facts、boundary、candidate context、action schema
   输出：ModelActionRequest
   禁止：自授权限、绕过 action schema、依赖旧 deterministic intent

5. ActionPermit
   输入：ModelActionRequest、BoundaryDecision、operation/resource policy、approval state
   输出：ActionPermit / ApprovalRequest / Denial
   禁止：改变用户目标、自动换一个替代动作

6. RuntimeStartPacket
   输入：ModelActionRequest、ActionPermit、ContextCandidateSet、prompt manifest
   输出：RuntimeInvocationPacket
   禁止：重新计算 intent、加入改变角色的 fallback prompt

7. ExecutionLoop
   输入：RuntimeInvocationPacket
   输出：ToolObservation / ModelObservation / RuntimeEvent
   禁止：用 regex 或旧状态覆盖模型决策

8. TurnStateAuthority
   输入：runtime events、approval/interrupt/resume responses
   输出：TurnStateSnapshot
   禁止：让前端、恢复层或 raw event 各自解释 active 状态

9. OutputBoundary
   输入：TurnStateSnapshot、canonical result refs、public projection contract
   输出：用户可见正文和公开进度
   禁止：直接泄露 raw tool logs 或内部协议
```

### 4.2 权威表

| 文件/模块 | 当前职责 | 隐性决策风险 | 目标层 | 处理动作 |
| --- | --- | --- | --- | --- |
| `request_facts.py` | 收集 turn 输入事实 | 风险低 | RequestFacts | 保持，只增加不可回写 invariant |
| `model_action_protocol.py` | 验证模型动作 | 旧字段可能继续膨胀 | ModelTurnDecision contract | 收紧 schema，删除旧字段入口 |
| `admission.py` | 动作准入、服务面判断 | 与 tool executor / operation gate 重叠 | ActionPermit | 合并为 permit 生成入口 |
| `action_permit.py` | 执行许可 receipt | 缺 scope / risk lifecycle | ActionPermit | 扩展为带作用域的 permit |
| `tool_executor.py` | 执行工具、生成 observation | 可能重新裁决权限 | ExecutionLoop | 权限降级为 permit receipt 校验和资源边界校验 |
| `prompt_library/assembly.py` | prompt 资源装配 | precedence 只诊断 | RuntimeStartPacket / PromptAuthority | 改成强制排序和 fail-closed |
| `runtime/compiler.py` | runtime packet 大装配器 | assemble 层重算语义 | RuntimeStartPacket | 拆出 PromptAuthority、ContextAuthority、PermitBinding |

| run monitor / projection | 展示运行进度 | 前端可能从事件猜语义 | OutputBoundary | 只消费 TurnStateSnapshot 和 PublicProjectionFrame |

## 5. 推荐设计方向

### 5.1 借鉴 Claude Code，但不照搬

借鉴：

- prompt 片段稳定边界。
- prompt owner precedence。
- tool schema / prompt cache generation 的稳定性。
- prompt injection 和 tool result 不可信提示进入基础系统约束。

不照搬：

- Claude Code 是 CLI-first 单 agent 主循环，本项目有 TaskGraph、多环境、多 agent、持久任务，不能把所有规则塞进一个 system prompt。
- 本项目已有 prompt library 和 runtime manifest，应升级这些结构，而不是退回单字符串 prompt。

### 5.2 借鉴 Codex，但不照搬

借鉴：

- thread/turn 状态集中管理。
- permissions 有 scope、review、sandbox policy。
- AGENTS.md / project instructions 有层级来源和 precedence。
- app-server 事件和 active turn snapshot 分离。

不照搬：

- Codex 的 thread 主要围绕 coding turn，本项目有任务图和创作/知识/结构化能力，需要把 `TurnStateAuthority` 设计成 runtime-neutral。
- Codex 的 Rust protocol 不能直接映射为 Python dataclass 全量重写，应先以现有 event_log / state_index 增量建立 authority snapshot。

### 5.3 本项目目标设计

本项目应走“权威收束式重构”：

```text
PromptAuthority
TaskContractAuthority
PermissionAuthority
TurnStateAuthority
OutputAuthority
```

这五个 authority 不一定都是新大模块，但必须成为可测试的边界。每个 authority 只能做自己的事，不能互相替代。

作用：

- 统一恢复、前端、输出、监控的状态来源。
- 防止旧 event / snapshot 覆盖当前 turn。

涉及文件：

- `backend/task_system/contracts/task_requirement_contracts.py`

- `backend/task_system/runtime_semantics/compiler.py`
- `backend/prompt_library/assembly.py`

- `backend/tests/*contract*`
- `backend/tests/*permission*`

3. 给 prompt assembly、task contract、permission、turn state 各生成当前 authority map。

## 9. 迁移和切换规则

### 9.1 旧路径冻结

计划获批后，以下旧路径不得新增功能：

- `goal_frame` 目标裁决。
- `task_goal_spec` 作为运行 fallback。
- prompt precedence diagnostic-only 主路径。
- runtime semantics `legacy_fields` 常驻 manifest。
- executor 无 permit 执行副作用。

### 9.2 Shadow 模式

Phase 1 到 Phase 5 中，新 authority 可先 shadow 输出诊断，但不得改变主行为，直到对应 phase 的测试通过。

Shadow 输出必须包含：

```text
old_decision
new_decision
same_or_diff
diff_reason
source_refs
authority
```

### 9.3 Cutover 条件

每一阶段 cutover 必须满足：

1. 聚焦测试通过。
2. `rg` 搜索确认旧字段不再在主路径裁决。
3. runtime packet / event log / projection 中有新 authority refs。
4. 失败路径能显式 block / denial / diagnostic，而不是静默 fallback。

### 9.4 回滚规则

允许回滚到上一阶段的条件：

- 新 authority 输出和旧主路径存在未解释分歧。
- 真实运行链路出现无法恢复的任务丢失、权限误放行、输出泄露。
- TaskGraph 发布/运行出现系统性阻断，且不能通过迁移报告解释。

禁止回滚方式：

- 不允许恢复旧 fallback 作为长期兼容。
- 不允许在新旧两套裁决同时在线时让下游自行选择。

### 9.5 最终清理条件

完成 Phase 6 后必须删除：

- 已无权威的旧字段读取。
- 只保护旧内部结构的测试。
- deprecated prompt 资源和旧 pack 引用。
- legacy manifest 常驻字段。
- 无实际迁移端点的 compatibility shim。

## 11. 禁止的捷径

1. 禁止用更长 prompt 代替权威边界。
2. 禁止为了兼容长期保留旧 goal / intent / legacy semantic 主路径。
3. 禁止 executor 在 permit 缺失时自行判断“看起来安全”而执行。
4. 禁止前端通过文案、关键词、工具名裁决语义。
5. 禁止把 raw tool result 当最终答案来源。
6. 禁止 mock 掉 runtime / permission / tool execution 来制造测试通过。
7. 禁止删除失败用例而不建立新行为测试。
8. 禁止让 restore/session/history 覆盖当前 turn 的明确决策。
9. 禁止把开发说明当 agent prompt；prompt 必须是角色、职责、边界、输入、输出、裁决标准和失败处理。

## 12. 预期成果

完成后，本项目应从“多模块协作但权威重叠”的状态，收束为成熟 agent 架构：

- 模型知道自己能做什么、不能做什么、下一步动作必须满足什么协议。
- runtime 知道自己只能装配和执行，不能替模型重新裁决目标。
- permission 知道自己只授权或拒绝，不改写用户目标。
- tool executor 知道自己只执行已授权动作并返回 observation。
- graph runtime 知道自己只消费发布后的显式语义，不再携带旧投影。
- turn state 知道 active、pending、terminal 的唯一事实。
- output boundary 知道什么可以给用户看，什么只能留在 trace。

这会让后续长任务、写作团队、代码任务、浏览器任务、MCP 工具、TaskGraph 并发和恢复链路更稳定，也更接近 Claude Code / Codex 这类成熟 agent 的工程标准。

用户在 2026-06-13 已要求再次审查项目，确认没有遗漏或冲突后即可实施。本次实施按以下决策执行：

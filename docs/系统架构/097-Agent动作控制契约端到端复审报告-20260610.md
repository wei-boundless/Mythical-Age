# Agent 动作控制契约端到端复审报告

日期：2026-06-10

## 0. 结论

当前系统还没有达到“成熟稳定投影系统”的标准。095 和 096 的结论成立，但本轮复审发现更严重的闭环问题：控制命令不只是可能被渲染成正文，还可能在公共 SSE 网络层被原样投给前端；并且在某些路径上，落盘边界已经判定应阻断，实时 typed stream 仍会把原始内容作为 `stable_answer/persist_canonical` 发出去。

这不是 UI 文案问题，是控制面、输出边界、公共投影、前端合并层之间的权威链断裂。必须按 P0 安全事故处理，不能继续靠“某一层过滤一下”补洞。

## 1. 本轮审查范围

本轮重新对照了以下链路：

1. Prompt / schema 暴露：`backend/harness/runtime/compiler.py`、`backend/prompt_library/*`

8. 现有测试：`backend/tests/*`、`frontend/src/lib/store/*.test.ts`

## 2. 当前真实链路

```text
RuntimeCompiler
-> model output contract / allowed_action_types
-> model_response_protocol_from_response
-> _single_agent_action_request_from_response
-> model_action_request_from_payload
-> admit_model_action
-> single_agent final dispatch / tool loop dispatch
-> _commit_final_message / canonical_output_decision_for_final_text
-> emit_terminal_then_final / assistant_final_stream_events
-> api.chat _project_public_stream_event
-> stream_replay public SSE
-> frontend reduceStreamEvent
-> ChatMessage
```

目标链路应该是：

```text
控制动作只进入控制执行器
正文只来自 canonical model-visible answer
任何未知、未闭合、内部协议形状都 fail-closed
公共 SSE 只输出 public projection，不输出 raw control payload
前端只合并经过 public projection 和内容级检查的正文
```

当前没有做到。

## 3. P0 问题

### P0-1：实时 typed stream 绕过 canonical 输出边界

证据：

- `backend/harness/loop/single_agent_turn.py:3673-3705` 的 `_commit_final_message()` 会调用 `canonical_output_decision_for_final_text()`，并将 `decision.content` 落盘。
- 但 `backend/harness/loop/single_agent_turn.py:239-279` 的 `emit_terminal_then_final()` 随后调用 `assistant_final_stream_events()` 时使用的是原始 `content`，不是 `commit_decision.content`。
- 同一函数还在 `single_agent_turn.py:257-258` 把 typed final 硬编码为 `answer_canonical_state="stable_answer"`、`answer_persist_policy="persist_canonical"`。
- `single_agent_turn.py:1426-1435` 的 raw fallback 会把原始 `content` 传给 `emit_terminal_then_final()`。

真实后果：

即使落盘边界已经把控制 JSON 判为 `do_not_persist/missing_answer`，`assistant_text_final` 仍可能把原始控制 JSON 作为稳定正文发给前端。前端当前又会相信这组 metadata。

本轮探针确认：

```text
canonical_output_decision_for_final_text('{"action":"continue_active_work"}')
=> answer_canonical_state=missing_answer
=> answer_persist_policy=do_not_persist
```

但解析器在 `require_json_action=False` 时对同一 action-only 裸 JSON 返回：

```text
action_request=None
error=None
```

于是 final fallback 可继续进入 typed final 发送链。

必须整改：

- `emit_terminal_then_final()` 必须接收并使用同一个 `CanonicalFinalTextDecision`。
- typed `assistant_text_final/repair` 的 content、canonical_state、persist_policy、leak_flags 必须来自 canonical decision。
- 不允许 commit 用 sanitized content、SSE 用 raw content。

### P0-2：`request_registered_engagement` 允许、准入、解析都存在，但 final dispatch 不闭合

证据：

- `backend/harness/loop/model_action_protocol.py:6-14` 将 `request_registered_engagement` 放入 `ModelActionType`。
- `model_action_protocol.py:158-161` 校验其 `plan_id`。
- `backend/harness/loop/admission.py:145-165` 对它做准入，并可返回 allow。
- `backend/harness/runtime/compiler.py:1422`、`:1462`、`:1751`、`:1966` 暴露它。
- `backend/prompt_library/rules.py:21` 和 `backend/prompt_library/packs.py:13` 仍提示模型可使用它。
- 但 `backend/harness/loop/single_agent_turn.py:1244-1383` final dispatch 只处理 `respond/request_task_run/block/ask_user/active_work_control`，没有 `request_registered_engagement` 分支。
- 然后进入 `single_agent_turn.py:1385-1436` raw fallback。

本轮探针确认：

```text
model action JSON:
{"authority":"harness.loop.model_action_request","action_type":"request_registered_engagement","engagement_request":{"plan_id":"p1"}}

解析结果：action_request.action_type=request_registered_engagement
准入结果：allow
output boundary：contains_internal_protocol=False
canonical decision：stable_answer + persist_canonical
```

这意味着它不是“会被边界兜住”的问题，而是可持久正文泄露。

必须整改：

- 如果产品当前不使用 registered engagement，立即从 `ModelActionType`、schema、prompt、admission、progress、前端 mapping 中删除。
- 如果保留，必须实现完整 dispatch、公共投影、测试和失败收口。
- final dispatcher 必须穷尽所有 `ModelActionType`；未知或未处理 action 必须 protocol error，绝不能进入 raw fallback。

### P0-3：公共 SSE 暴露 raw control event

证据：

- `backend/api/chat.py:84-88` 允许 `model_action_request/model_action_admission/model_action_admission_checked/bounded_observation` 携带 `event`。
- `backend/harness/loop/single_agent_turn.py:3803-3826` 的 `_record_model_action_admission()` 将 `model_action_request: action_request.to_dict()` 放进 event payload。
- `backend/api/chat.py:658-684` 只做字段 allowlist 和内部 key 删除，没有语义脱敏。

真实后果：

即使 React 组件最终不渲染，前端网络层和 store 已经收到原始控制命令 payload。这违反“控制系统命令不能暴露到前端”的底线。

必须整改：

- 公共 SSE 不得传 raw `event.payload.model_action_request`。
- `model_action_*` 只能投影为 public timeline/status/tool window 的安全摘要。
- raw control payload 只能进入后端 event log、debug-only 开发通道或受权限保护的诊断接口，不能走普通 chat stream。

### P0-4：single-agent final dispatcher 不是 fail-closed

证据：

- `single_agent_turn.py:1138-1383` 在 `action_request is not None` 后用一串 `if` 处理已知 action。
- 处理完 `active_work_control` 后没有 `else` 或 exhaustive guard。
- 未处理 action 会落到 `single_agent_turn.py:1385-1436` 的 raw fallback。

真实后果：

任何新增到协议、准入或 prompt 的 action，只要 dispatch 没同步实现，就可能成为正文。

必须整改：

- 用显式 exhaustive dispatch。
- `ModelActionType` 与 dispatcher 建立测试级约束。
- `action_request is not None` 但未 dispatch 时，直接 `_emit_single_agent_protocol_error(code="single_agent_turn_unhandled_model_action")`。

## 4. P1 冲突与遗漏

### P1-1：旧 schema 与新 single-agent output contract 双权威

证据：

- `compiler.py:1962-2036` 的旧 `model_action_request_schema()` 包含 `request_registered_engagement`，不包含 `active_work_control`。
- `compiler.py:2127-2234` 的 `_single_agent_turn_output_contract()` 包含 `active_work_control`，不包含 `request_registered_engagement`。
- `compile_observation_followup_packet()` 仍使用旧 schema 和 `request_registered_engagement`。

必须统一为单一动作契约源。

### P1-2：active_work_control 必填字段冲突

证据：

- `compiler.py:2213-2222` 声明 `active_work_control.required_fields=["action","relation_to_current_work"]`。
- `model_action_protocol.py:162-170` 只要求能解析 action，不要求 relation。
- `active_work.py:189-204` 后置把非 `current_work` 判 deny。

必须统一：缺 relation 是 invalid protocol，还是 allow 后由 active-work decision deny。不能一层说必填，另一层放过。

### P1-3：裸 active-work action-only 没有在 parser 层 fail-closed

证据：

- `single_agent_turn.py:2200-2226` 的 `_is_bare_active_work_control_payload()` 要求除了 action 外，还出现 relation/response/appended_instruction 等字段。
- 因此 `{"action":"continue_active_work"}` 在 `require_json_action=False` 时解析为 `action_request=None,error=None`。
- output boundary 现在能识别该文本，但 P0-1 会让 typed final 绕过 canonical 决策。

必须整改：

- 只要 JSON object 内出现 active-work control action，就必须视为 action-like protocol。
- 缺字段应成为 invalid action 或 denied control observation，绝不能当普通正文。

### P1-4：native control compatibility 与契约冲突

证据：

- `compiler.py:2188-2195` 对模型声明 `native_tool_transport_enabled=False`、`control_actions_exposed_as_native_tools=False`。
- `single_agent_turn.py:86` 仍定义 `_CONTROL_NATIVE_TOOL_NAMES={"request_task_run","active_work_control","ask_user","block"}`。
- `single_agent_turn.py:2229-2580` 仍能把 native control tool call 转为控制 action。

必须二选一：

- 删除 native control parser；
- 或在契约中明确它只是 provider compatibility parser，不向模型暴露，并加测试保证不会经可见工具 catalog 暴露。

### P1-5：前端实时流缺内容级防线

证据：

- `frontend/src/lib/store/events.ts:249-252` 的 `shouldKeepStreamAssistantText()` 只调用 `shouldDisplayAssistantContent(metadata)`。
- `events.ts:254-427` 的 delta/final/repair 会写入 `message.content`。
- `assistantContentVisibility.ts:42-44` 已有 `shouldDisplayAssistantStreamContent(defaultVisible=false)`，但 `events.ts` 未使用。
- `internalControlText.ts` 已存在，但实时流合并层未导入检查 content。

必须整改：

- `assistant_text_delta.content`、`assistant_text_final.content`、`assistant_stream_repair.replacement_content` 都必须做内容级 fail-closed。
- stream 默认可见应为 false；只有 canonical metadata + 内容检查通过才合并。
- `ChatMessage.assistantDisplayContent()` 也应有最后一道内容级防线。

### P1-6：后端 SSE allowlist 不是语义边界

证据：

- `chat.py:101-152` 允许 typed stream 的 `content/replacement_content`。
- `_redact_public_stream_data()` 只删除内部 key，不检查文本语义。

必须整改：

- 后端 public projection 对 typed assistant stream 加内容级检查。
- 检查失败时应 drop event 或替换为 fail-closed public status，不能继续发送正文。

### P1-7：stage/status 与 assistant body 边界不一致

证据：

- `public_projection_envelope.py:10` 把 `progress_feedback/stage_feedback` 归为 `_MODEL_BODY_CHANNELS`。
- `assistantContentVisibility.ts:17-26` 又把 `stage_feedback` 放入控制/非公开通道。
- `events.ts:779-807` 仍有 `done: "处理完成"`、`tool_call: "正在调用工具"` 等中文状态标签。
- `events.ts:93-140` 对多种内部事件返回 `"正在思考"`。

判断：

这些不一定进入 assistant 正文，但当前契约没有清楚区分 body/status/timeline/tool window。用户看到“开始处理/处理完成”这类句子，根因就是状态文本和正文投影长期混在一起。

必须整改：

- `assistant body` 只接受模型正文。
- `status_bar/timeline/tool_window/control` 使用结构化 public projection，不进入 `message.content`。
- 通用状态文案需要统一来源和显示位置，不允许从 SSE event label 回填为正文。

### P1-8：active-turn steer fast path 只会追加要求，不会语义执行 pause/stop

证据：

- `frontend/src/lib/store/runtime.ts:2566-2582` active stream 期间会把输入送入 `submitActiveTurnSteerDuringActiveStream()`。
- `runtime.ts:1583-1594` 请求体固定 `active_turn_input_policy: "steer"`。
- `backend/harness/entrypoint/runtime_facade.py:776-908` 的 `_active_turn_steer_fast_path()` 在通过 guard 后直接 `append_user_work_instruction()`，并把动作写成 `append_instruction_to_active_work`。
- 真正的 `pause_active_work/stop_active_work` 在 `runtime_facade.py:1596-1608`，需要模型 active_work_control 决策，或走 monitor action。

真实后果：

如果用户在运行中输入“暂停/停止”，该自然语言可能被 fast path 当作补充要求入队，而不是立即暂停/停止。这与成熟 agent 的控制语义不一致。

必须整改：

- UI 的暂停/停止按钮必须走 monitor control action，立即登记控制请求。
- 自然语言输入是否 fast steer，需要先做安全级命令分类，或要求用户显式点击控制按钮。
- “暂停是马上暂停，还是等当前步骤收口暂停”必须在 public status 中明确：控制请求立即记录，执行器在可中断边界收口。

### P1-9：session append 不是安全边界

证据：

- `runtime_facade.py:2153-2178` 的 `_apply_assistant_message_commit()` 直接 append payload。
- 安全完全依赖调用方先 canonicalize。

必须整改：

- 保留 `_commit_final_message()` 作为唯一 assistant final commit 入口。
- 对绕过它的 direct append 做审计。
- `session_manager.append_messages()` 不应被当作语义安全边界。

## 5. 已经存在但不能算闭环的保护

1. `boundary.py:172-212` 能识别 active-work JSON，包括 action-only。
2. `utils.ts:202-204` 历史加载会过滤 active-work 控制文本。
3. `public_timeline_stream.py:24-45` 抑制了部分泛化状态文本。
4. `assistant_stream_normalizer.py:112-132` 对开头 JSON/action protocol 有 stream safety gate。

这些保护都不是完整闭环：

- boundary 决策没有被 typed final 复用。
- history 过滤挡不住实时流。
- timeline 抑制挡不住 raw SSE payload。
- stream normalizer 挡不住 final/repair 事件使用原始 content。

- `backend/tests/output_boundary_final_text_regression.py:35-44` 覆盖 active-work JSON final boundary。
- `backend/tests/harness_model_action_protocol_regression.py:479-526` 覆盖带 relation/response 的裸 active-work JSON。

2. dispatcher 穷尽所有 `ModelActionType`。
3. `{"action":"continue_active_work"}` action-only 裸 JSON 在 `require_json_action=False` 时不得成为正文。
4. `_commit_final_message()` 的 canonical decision 必须驱动 `assistant_text_final`。
5. 公共 SSE 不得包含 raw `event.payload.model_action_request`。
6. typed stream delta/final/repair 注入控制 JSON 时前端不得写入 `message.content`。

8. active stream 期间自然语言“暂停/停止”不会被误当作普通补充要求。

## 7. 目标动作契约

动作契约应收敛为以下规则：

1. `respond`：唯一可产生 assistant body 的普通动作。
2. `ask_user`：产生 control/timeline 问题，不直接作为普通 body；如产品需要显示问题，应通过 control item 显示。
3. `block`：产生 blocked/control status，不进入普通 body。
4. `request_task_run`：只产生 task lifecycle handoff/status，不产生 assistant body。
5. `active_work_control`：只产生 active-work observation/status/control，不产生 assistant body。
6. `tool_call`：只产生 tool window/observation；工具观察返回后由模型再决定是否 respond。
7. `request_registered_engagement`：当前建议删除；若保留，必须完整实现，不能只留 schema/prompt/admission。

任何不在上述闭环内的 action：

```text
parsed action but no executor
-> protocol error
-> blocked/control status
-> no raw content
-> no assistant body
-> no public raw payload
```

## 8. 修复优先级

### 第一阶段：立即阻断泄露

1. 修 `emit_terminal_then_final()`，让 typed final 使用 canonical decision。
2. final dispatch 增加 exhaustive guard。
3. 后端 SSE public projection 删除 raw `model_action_request/model_action_admission.event`，改安全摘要。
4. 前端 stream 合并层增加内容级 guard，并使用 `shouldDisplayAssistantStreamContent()`。

### 第二阶段：统一协议权威

1. 移除或完整实现 `request_registered_engagement`。
2. 合并旧 `model_action_request_schema()` 与 `_single_agent_turn_output_contract()`。
3. 统一 `active_work_control` required fields。
4. 决定 native control parser 去留。

### 第三阶段：成熟投影与控制体验

1. 重定义 body/status/timeline/tool/control 五类 surface。
2. 清理“开始处理/处理完成/正在处理”类泛化状态的来源。
3. 明确开局反馈：只有模型提供真实 `current_judgment` 时显示；否则显示非正文 loading，不写入 message body。
4. 明确暂停/停止/steer 时序：控制按钮立即记录；自然语言 steer 不得误执行 pause/stop；执行器在可中断边界收口并投影状态。
5. 增加端到端回放测试：SSE、store、history、refresh、resume。

## 9. 最终判断

现在的问题不是某个判断漏了一句正则，而是多层都在做“是否可展示”的决定：

- parser 决定一部分；
- output boundary 决定一部分；
- stream normalizer 决定一部分；
- SSE allowlist 决定一部分；
- frontend visibility 决定一部分；
- ChatMessage 再决定一部分。

成熟方案必须恢复单向权威链：

```text
ModelActionDecision
-> ActionAdmission
-> ActionExecutor
-> CanonicalOutputDecision
-> PublicProjection
-> FrontendRender
```

控制动作永远不能降级成正文。公共前端永远不能收到 raw control payload。任何协议不闭合，必须 fail-closed，而不是 fallback 到 `conversation`。

# 输出与投影稳定性优化升级计划书

日期：2026-06-13  
状态：待用户确认，未实施  
适用范围：主会话输出、后台任务运行反馈、公开投影、运行监控、session history 刷新、SSE/monitor 恢复链路

设计决策更新：采用“session final message commit 成功后，再将主会话可见任务标记为 completed”的生命周期顺序。允许改动 task lifecycle。该决策替代本文早期的“保持 completed 先发生但 hydrate 只认 ack”的过渡方案。

## 1. 目标

本计划解决当前主会话“运行内容在收口前被刷新掉”的问题。它不是单独设计一个算法框架，而是把输出、投影、监控、历史刷新和性能优化统一成一条稳定工程链路。

目标体验：

1. agent 正在运行时，用户能稳定看到 live 输出和工具进度。
2. 任务进入 closeout / completed 后，前端不会提前用旧 session history 覆盖 live 内容。
3. 最终 assistant 正文真正写入会话后，前端再进行历史收敛。
4. 刷新后可见 transcript 与运行时 live transcript 收敛为同一份内容，不重复、不丢失、不闪烁。
5. 监控系统继续快速反馈任务状态，但不能越权决定主会话输出是否已经持久化。
6. 不用任意延时、防抖、重复轮询、旧兼容分支来掩盖竞态。

## 2. 问题定义

### 2.1 当前故障

用户看到的问题是：后台任务还在收口或最终内容还没稳定出现，主会话区域已经被刷新，导致运行内容消失、被旧历史覆盖，或收口前输出断层。

当前触发链路：

```text
task_executor 标记 task completed
-> task_projection 把 completed 投成 closeout
-> run monitor SSE 推送 monitor snapshot
-> frontend 看到 closeout / terminal status
-> hydrateCurrentSessionFromStream()
-> refreshSessionDetails()
-> getSessionTimeline() 读取 session history
-> session history 里最终 assistant message 可能还未 commit
-> live 内容被旧历史替换
```

### 2.2 深层原因

这是输出权威、投影权威和刷新权威混用导致的结构性问题。

当前系统把以下几个事实混为一谈：

| 事实 | 正确含义 | 当前误用 |
| --- | --- | --- |
| `task status = completed` | 后台任务生命周期结束 | 被当成最终正文已写入 session |
| `task_projection.current_action.kind = closeout` | 投影层可以展示收口状态 | 被当成可以刷新当前会话历史 |
| `turn_completed` | 本轮终止信号 | 容易被当成正文兜底或 hydrate 依据 |
| `assistant_text_final` | 正文流最终帧 | 还没有成为 session history commit 的确认 |
| session history | 已持久化 transcript | 在 commit 前可能落后于 live stream |

正确系统属性应该是：任务完成、投影收口、正文完成、正文持久化、前端历史收敛是五个不同阶段，必须有明确顺序和不同权威。

## 4. 本项目当前证据

### 4.1 后端完成顺序存在竞态

`backend/harness/loop/task_executor.py` 中 `_finish_executor_success()` 当前顺序是：

```text
finish_task_lifecycle(status="completed")
-> _record_task_step_summary("task_run_completed")
-> append_work_rollout_item(final_response)
-> _commit_task_run_final_message()
```

这意味着 monitor 可以先看到 task completed / closeout，但 session 最终 assistant message 还没有写入。

### 4.2 投影层把 completed 变成 closeout

`backend/harness/runtime/projection/task_projection.py` 中：

```text
status == completed -> current_action.kind = closeout
```

这是合理的展示投影，但它只说明“可以展示收口状态”，不能证明“主会话最终正文已持久化”。

### 4.3 前端 monitor 越权触发 hydrate

`frontend/src/lib/run-monitor/controller.ts` 当前逻辑：

```text
applyStreamPayload()
-> hydrateCurrentSessionFromStream()
-> currentSessionHasFormalCloseout()
-> refreshSessionDetails()
```

`currentSessionHasFormalCloseout()` 会把以下信号视为可刷新依据：

- `task_projection.current_action.kind === "closeout"`
- `status/lifecycle/activity_state/state` 属于 completed / failed / stopped / cancelled 等终态

这就是 live 内容被提前刷新掉的直接触发点。

### 4.4 session refresh 会替换 messages

`frontend/src/lib/store/runtime.ts` 中 `refreshSessionDetails()` 读取 `getSessionTimeline()` 后会生成 refreshed messages。当前只有 `activeStreamSessionIds` 仍包含 session 时，`messagesForSessionDetailsRefresh()` 才保留当前 messages。

但收口阶段 active stream 可能已经移除，monitor closeout 又先于 final message commit 到达，于是 refreshed history 会覆盖 live state。

### 4.5 项目已有正确契约雏形

`backend/runtime/output_stream/public_contract.py` 已经定义：

```text
TRANSCRIPT_PUBLIC_EVENTS:
  assistant_text_delta
  assistant_text_final
  assistant_stream_repair

ITEM_LIFECYCLE_PUBLIC_EVENTS:
  tool_item_started
  tool_item_completed

TERMINAL_PUBLIC_EVENTS:
  turn_completed

LOSSLESS_PUBLIC_EVENTS = transcript + item lifecycle + terminal
```

这说明项目已有 Codex-style lossless public event 的基础。问题不是没有契约，而是 monitor hydrate 链路没有尊重这个契约。

## 5. 设计方向

### 5.1 核心原则

1. `task completed` 不能触发主会话 hydrate。
2. `task_projection.closeout` 不能触发主会话 hydrate。
3. `turn_completed` 只能代表 turn terminal，不代表 session history 已完成。
4. `assistant_text_final` 代表正文流完成，但仍不代表 session history 写入成功。
5. 只有 `session_output_commit_ack` 才能触发当前会话历史收敛。
6. monitor 只观察，不决定主 transcript。
7. frontend 必须维护 committed transcript 与 live active tail 的边界。
8. 刷新要按 ack 和 revision 精确触发，不能靠时间、防抖、轮询猜测。

### 5.2 目标权威链

```text
Runtime Event Log / Task Facts
-> Canonical Output Stream
   - assistant_text_delta
   - assistant_text_final
   - assistant_stream_repair
   - tool_item_started
   - tool_item_completed
   - turn_completed
-> Session Output Commit
   - session_output_commit_checked
   - session_output_commit_ack
   - session_output_commit_failed
-> Public Projection
   - public timeline
   - task projection
   - runtime attachment
-> Run Monitor
   - observe status/progress/diagnostics only
   - expose session_output_commit state but not infer it
-> Frontend Store
   - committed messages
   - live active tail
   - hydrate only on commit ack
-> Chat UI
   - render stable transcript + live tail
```

### 5.3 已锁定生命周期策略

主会话可见的成功任务必须遵守：

```text
execution result ready
-> final assistant message commit checked
-> final assistant message commit ack
-> task lifecycle completed
-> task projection closeout
-> monitor completed
-> frontend hydrate
```

这意味着：

1. `task_run_lifecycle_finished(status="completed")` 不能早于 final assistant message commit ack。
2. `task_projection.closeout` 只能在 commit ack 后出现，避免 closeout 先于最终正文。
3. 如果 final message commit 失败，任务不能标记为 `completed`；应标记为 `failed` 或 `blocked`，并保留执行结果诊断。
4. 如果某类任务明确不需要写主会话 final message，例如 `subagent_task`，必须先写 `session_output_commit_skipped(reason="not_main_session_visible")`，然后才允许 completed。
5. 失败、停止、用户中断、等待继续不套用“commit 成功后 completed”规则；它们可以终止生命周期，但若要写用户可见边界消息，应在 terminal 前尽力 commit，并把失败原因写入诊断。

## 6. 优化问题定义

本次升级不是单点修 bug，而是一个多目标优化问题：

```text
maximize:
  输出不丢失
  收口稳定性
  前端可见连续性
  session history 与 live transcript 最终一致性
  故障可诊断性

minimize:
  不必要 session timeline fetch
  messages 整体替换次数
  monitor snapshot 对主会话的副作用
  render churn
  backpressure 下关键事件丢失概率
  旧兼容分支和重复权威
```

### 6.1 稳定性优化

优化目标：

- closeout 前 live assistant text 不被旧 history 覆盖。
- commit ack 前 runtime attachments / draft timeline 不消失。
- commit failed 时显示明确状态，不静默刷新。
- monitor reconnect / initial snapshot 不触发错误 hydrate。

关键策略：

1. hydrate 条件从 terminal status 改为 commit ack。
2. live tail 与 committed history 分区存储。
3. task projection 只能更新 runtime attachment，不能替换 assistant body。
4. `refreshSessionDetails()` 增加 commit-aware merge，不再盲目整体替换当前消息。

### 6.2 延迟优化

优化目标：

- live 输出仍然低延迟显示。
- final commit ack 到达后快速收敛到 session history。
- 不通过固定 sleep 等待后端写入。

关键策略：

1. `assistant_text_delta` 继续直接更新 live message。
2. `assistant_text_final` 只完成 live body，不触发 history refresh。
3. `session_output_commit_ack` 到达后立即触发一次 hydrate。
4. 对同一 `commit_epoch` / `event_offset` 做去重，防止重复刷新。

### 6.3 I/O 优化

当前 monitor stream 每次 snapshot 都可能触发 `getSessionTimeline()`，这是错误且昂贵的。

目标行为：

- 普通 monitor snapshot：0 次 session timeline fetch。
- closeout snapshot 但无 commit ack：0 次 session timeline fetch。
- commit ack 首次出现：1 次 session timeline fetch。
- 同一 ack 重复出现：0 次额外 fetch。
- 用户切 session / 初始化恢复：保留现有必要 fetch。

建议指标：

| 场景 | 当前风险 | 目标 |
| --- | --- | --- |
| running monitor snapshot | 可能不刷新 | 不刷新 |
| completed closeout snapshot | 会刷新 | 不刷新，除非有 commit ack |
| commit ack snapshot | 无明确区分 | 精确刷新一次 |
| monitor reconnect initial snapshot | 可能刷新旧终态 | 不刷新旧终态 |
| session switch | 正常拉历史 | 保留 |

### 6.4 Render 优化

目标：

- 不因为 monitor 状态更新重建全部 messages。
- 只更新目标 assistant message 的 runtime attachment / draft timeline。
- 刷新后按 stable message id 合并，不造成 React key 抖动。

关键策略：

1. runtime attachment 使用 `task_run_id + anchor_turn_id + anchor_message_id` 稳定定位。
2. commit ack 前用 `anchor_turn_id` 临时定位 live tail。
3. commit ack 后使用 `anchor_message_id` 或 history message id 稳定定位。
4. `mergeVolatileMessageProgress()` 只保留未提交 live tail，不能把旧 terminal projection 当作正文。

### 6.5 Backpressure 优化

`backend/runtime/shared/event_log.py` 当前 live subscription queue 满时 `_put_event_drop_oldest()` 会丢旧事件。这对普通 progress 可以接受，但对 transcript / completion / commit ack 不安全。

目标：

- `assistant_text_delta/final/repair`、`tool_item_started/completed`、`turn_completed`、`session_output_commit_ack/failed` 必须不可丢或可按 offset 恢复。
- 普通 monitor progress、heartbeat、stdout tail 可以 best-effort。

可选实现：

1. 短期：commit ack 写入持久 event log，monitor snapshot 从 event log 派生；即使 SSE 丢帧，下次 snapshot 仍能恢复 ack。
2. 中期：runtime event subscription 引入 lossless tier，关键事件队列满时阻塞或触发 gap recovery。
3. 长期：前端 SSE 消费 event offset，发现 gap 后从 replay endpoint 补关键事件。

### 6.6 Cache 与 revision 优化

monitor 当前有 global cache。优化重点不是取消缓存，而是让 cache revision 能反映 commit ack。

目标：

- task lifecycle terminal 更新 monitor revision。
- session output commit ack 也更新 monitor revision。
- frontend 用 `session_id + task_run_id + turn_id + commit_epoch/event_offset` 去重 hydrate。
- 不用 `updated_at` 这种弱时间戳作为唯一刷新条件。

- 区分旧 terminal monitor snapshot 与新 commit ack。
- 为断线恢复提供明确断点。

## 8. 固定执行流

### 8.1 后台任务成功收口

```text
1. executor 得到 final_answer
2. 写 agent_run_result
3. 发 assistant_text_final / turn_completed 所需输出事件
4. 执行 session output commit gate
5. append session assistant message 成功
6. 写 session_output_commit_ack
7. 标记 task lifecycle completed
8. monitor snapshot 展示 completed + closeout + committed
9. frontend 看到 commit ack 后 hydrate 一次
10. live tail 与 persisted history 合并
```

该顺序是本计划的固定方案。`completed` 不再表示“executor 算完了”，而表示“用户可见结果已经成功持久化，任务可以对外宣告完成”。

如果 executor 已经得到 final_answer 但 commit 失败：

```text
execution result ready
-> final assistant message commit checked
-> final assistant message commit failed
-> task lifecycle failed
-> monitor 显示 output_commit_failed
-> frontend 保留 live tail，不用旧 history 覆盖
```

任务诊断中必须保留：

- `execution_result_status="completed"`
- `output_commit_status="failed"`
- `final_answer_ref` 或 final answer 摘要
- commit failure reason

这样可以区分“任务执行失败”和“输出持久化失败”，同时避免用户界面虚假 completed。

### 8.2 前端 monitor snapshot

```text
1. apply monitor snapshot
2. 更新 monitor UI / task lane
3. 检查当前 session 是否有新的 session_output_commit.state=committed
4. 如果没有：不刷新 session history
5. 如果有且 cursor 未处理过：refreshSessionDetails(sessionId)
6. refresh 时保留未被 commit 覆盖的 live tail
7. 记录 handled hydrate cursor
```

### 8.3 断线恢复

```text
1. SSE reconnect 后拿 initial monitor snapshot
2. snapshot 中 terminal task 不触发 hydrate
3. snapshot 中 committed ack 触发 hydrate，且按 cursor 去重
4. 如果 frontend 有 live tail 但 backend 无 commit ack，继续保留 live tail 或显示 commit pending
5. 如果 backend 有 commit failed，显示明确失败反馈，不用旧 history 覆盖 live tail
```

| `frontend/src/lib/store/utils.ts` | history -> UI messages / attachment anchor | 优先稳定 message id / anchor_message_id，turn anchor 只作临时 | 刷新前后 attachment 不跳位 |

## 10. 分阶段实施计划

### Phase 1：重排成功生命周期

目标：让主会话可见任务只有在 final assistant message commit ack 后才进入 completed。

改动：

1. 修改 `_finish_executor_success()`：
   - 先构造带 `final_answer`、`artifact_refs`、`final_action_diagnostics` 的 task_run candidate。
   - 先执行 `_commit_task_run_final_message()`。
   - commit ack 成功后再调用 `finish_task_lifecycle(status="completed")`。
   - `append_work_rollout_item(final_response)` 的 `event_offset` 使用 completed lifecycle event。
2. 修改 `_finish_specialist_runtime_execution()`：
   - 对 `completed` 且主会话可见的 specialist task，先 commit final message，再 lifecycle completed。
   - 对 `subagent_task` 写 explicit commit skipped receipt，再 lifecycle completed。
3. `_commit_task_run_final_message()` 从 `None` 返回值改为结构化 receipt：
   - `status: committed | skipped | failed`
   - `commit_allowed`
   - `session_id`
   - `turn_id`
   - `task_run_id`
   - `anchor_message_id`
   - `content_sha256`
   - `event_offset`
   - `reason`
4. commit 失败时不进入 completed：
   - 普通成功路径改为 lifecycle `failed`，`terminal_reason="session_output_commit_failed"`。
   - diagnostics 记录 execution result 已完成，但 output commit failed。

完成条件：

- event log 中成功顺序固定为 commit ack 在前，lifecycle completed 在后。
- monitor 不会看到 committed=false 的 completed 主会话任务。
- final message commit failed 时任务不会显示为 completed。

禁止：

- 不允许先 `finish_task_lifecycle(completed)` 再 commit final message。
- 不允许 commit failed 后仍把任务标为 completed。
- 不允许用 `subagent_task` 的 skipped 当作主会话 final message committed。

### Phase 2：后端 commit ack 一等化

目标：让 session history 写入成功成为明确 runtime fact。

改动：

1. `_commit_task_run_final_message()` 返回 commit receipt。
2. `_apply_assistant_message_commit()` 返回 message identity / appended info。
3. append 成功后写 `session_output_commit_ack`。
4. commit disallowed / exception 写 `session_output_commit_failed` 或 `session_output_commit_skipped`。
5. monitor projector 从 event log 派生 `session_output_commit`。
6. `finish_task_lifecycle(status="completed")` 入口增加成功前置检查：
   - 主会话可见任务缺少 commit ack 时拒绝 completed，或由调用层转为 failed。
   - 该检查可以先放在 task_executor 调用层，后续再沉入 lifecycle helper。

完成条件：

- event log 可看到 checked -> ack / failed。
- monitor signal 能区分 completed-but-not-committed 与 completed-and-committed。
- 正常成功任务不存在 completed-but-not-committed。

禁止：

- 不允许 task projection 根据 final_answer 文本推断 committed。
- 不允许 commit failed 时写假 ack。

### Phase 3：切断错误 hydrate 权威

目标：消除“closeout/terminal 导致当前会话刷新”的前端副作用，并只接受 commit ack hydrate。

改动：

1. 前端删除 `currentSessionHasFormalCloseout()` 作为 hydrate 条件。
2. 新增 `currentSessionHasOutputCommitAck()`，只读 monitor signal 上明确 commit state。
3. monitor snapshot 没有 commit ack 时不触发 `refreshSessionDetails()`。
4. 保留用户主动切换 session / 初始化加载历史的路径。

完成条件：

- completed/closeout monitor snapshot 不调用 `getSessionTimeline()`。
- running/live 内容不会被 monitor closeout 覆盖。

禁止：

- 不允许加固定 1s/3s/5s 延迟。
- 不允许用 status failed/stopped/completed 作为替代条件。

### Phase 4：前端 committed transcript + live tail 合并

目标：把 Codex-style UI 边界落到 store。

改动：

1. 增加 hydrate cursor，按 `sessionId + turnId + taskRunId + commitEventOffset` 去重。
2. `refreshSessionDetails()` 在 commit ack 下收敛 persisted history。
3. 无 ack 的 refresh 不替换 live active tail。
4. `mergeVolatileMessageProgress()` 只合并 runtime attachment / draft timeline，不让旧 projection 覆盖正文。

完成条件：

- commit ack 前 live tail 保留。
- commit ack 后 persisted assistant message 替换 live body，且内容不重复。
- attachment anchor 不跳位。

### Phase 5：lossless / backpressure 强化

目标：把关键输出事件从 best-effort 里剥离出来。

改动：

1. 将 commit ack/failed 纳入关键恢复路径。
2. 评估 runtime event subscription 是否需要 lossless queue 或 gap recovery。
3. monitor snapshot 保持 best-effort，但必须能从 event log 重建 commit state。

完成条件：

- 普通 monitor progress 可丢不影响最终 transcript。
- commit ack 丢 SSE 后仍能通过下一次 snapshot 或 replay 恢复。

### Phase 6：清理旧链路和旧测试

目标：删除会继续保护错误权威的旧逻辑。

改动：

1. 删除 formal closeout hydrate 测试。
2. 删除 terminal status hydrate 分支。
3. 删除 projection closeout 作为 refresh trigger 的任何残留。
4. 检索 `currentSessionHasFormalCloseout`、`closeout hydrate`、`formal closeout` 等旧名，确保不可达。

完成条件：

- 搜不到旧 hydrate 权威。
- 测试保护的是 commit ack 驱动刷新，而不是 closeout 驱动刷新。

## 13. 风险与处理

### 13.1 风险：commit ack 需要 message id，但 session message 当前无显式 id

处理：

- 第一阶段 `anchor_message_id` 可使用现有 `history-message:${turn_id}:assistant` 规则。
- 后续可给 session message 增加显式 `message_id`，但不作为本次第一阶段阻塞项。

### 13.2 风险：已有测试依赖 closeout hydrate

处理：

- 这些测试保护的是错误权威，应重写。
- 新测试保护 commit ack hydrate。
- 不保留 closeout hydrate 兼容分支。

### 13.3 风险：task lifecycle completed 仍早于 commit

处理：

- 第一阶段允许 lifecycle completed 早于 commit，但 hydrate 不认 lifecycle。
- 后续如果要更成熟，可把用户可见 completed 推迟到 commit ack 后，或拆成 `execution_completed` 与 `visible_completed`。

### 13.4 风险：commit failed 时用户看到任务 completed 但无最终消息

处理：

- monitor 显示 `completed / commit_failed` 的复合状态。
- 主会话保留 live tail，并显示“最终消息写入失败，可重试恢复”的用户级反馈。
- 不用旧 history 覆盖 live 内容。

## 14. 禁止的捷径

1. 禁止用 `setTimeout` 延迟刷新来赌后端 commit 完成。
2. 禁止继续用 `task_projection.closeout` 触发 hydrate。
3. 禁止把 `completed/failed/stopped` 终态当作 session history ready。
4. 禁止让 public projection 生成最终 assistant 正文。
5. 禁止 commit failed 时伪造 committed。
6. 禁止保留 closeout hydrate 兼容旧链路。
7. 禁止通过降低测试断言或删除失败测试制造通过。
8. 禁止把 raw runtime event / tool id / internal protocol 暴露为用户正文。

## 15. 预期结果

完成后，主会话输出链路会变成：

```text
live stream 负责即时可见
commit ack 负责持久化确认
history refresh 负责最终收敛
monitor 负责观察和控制
projection 负责展示，不负责改写正文权威
```

用户侧表现：

- 运行中不会突然空掉。
- 收口状态可以出现，但不会抢先刷新正文。
- 最终回答写入后再稳定收敛。
- 断线、重连、monitor snapshot、后台任务完成都不会破坏主 transcript。

工程侧收益：

- 输出权威清晰。
- session refresh 次数减少。
- 前端 render churn 降低。
- monitor 不再拥有主会话 hydrate 决策权。
- 后续做 lossless stream、replay、断线恢复时有明确事件边界。

## 16. 实施确认项

如果确认实施，建议按以下顺序一次性推进：

1. Phase 1 先重排成功生命周期：commit ack 在前，completed 在后。
2. Phase 2 增加后端 commit ack / failed / skipped 一等事件。
3. Phase 3 切断 closeout hydrate。
4. Phase 4 前端按 ack hydrate 并保护 live tail。
5. Phase 5 补关键事件恢复能力。
6. Phase 6 清理旧测试和旧链路。

这套方案的关键不是多加一个判断，而是把“任务结束”和“输出已持久化”彻底分开。只有这样，监控可以快，投影可以活，主会话 transcript 也能稳定。

# 主会话公开投影剩余问题修复计划

日期：2026-06-11

## 一、修复目标

本计划用于修复主会话公开投影在正文、工具、阶段总结、控制状态、持久化回放和收口上的剩余结构性问题。

目标不是继续增加清洗词或局部兜底，而是把公开显示链路收敛成一条稳定权威链：

```text
用户输入
-> Runtime/Prompt Packet
-> 模型正文流 assistant_text_delta / assistant_text_final
-> 前端 assistant body

模型工具动作
-> tool_window 工具投影
-> 工具结果观察
-> 观察进入下一轮模型输入
-> 模型给出阶段总结或最终正文

Runtime 控制状态
-> control surface
-> 不进入 assistant body

Terminal done/error/stopped
-> terminal/control closeout
-> 不制造正文
```

## 二、必须遵守的显示原则

1. assistant body 只允许模型正文进入。
2. 工具调用、工具结果、子 agent 生命周期只允许进入 tool window 或任务投影活动。
3. pause、stop、continue、approval、safe boundary、steer acknowledgement 只允许进入 control surface。
4. `done` 只表示运行终止，不能作为正文来源。
5. 阶段总结必须来自模型对观察的公开判断，不允许 runtime 把工具摘要改写成模型正文。
6. 没有正文时允许只显示工具窗口或控制状态，不允许补“正在思考”“开始处理”“处理完成”。
7. 错误必须返回给用户，但错误显示走 error/control，不伪装成模型正文。
8. 投影异常不得阻塞 agent 执行链路。显示失败只能降级为诊断或空投影，不能阻断工具调用、模型续跑、等待审批或收口。

## 三、当前关键问题

### 1. `done.summary` 会被错误提升为模型正文

位置：

- `backend/harness/runtime/projection/projector.py`
- `backend/harness/runtime/projection/items.py`

现状：

- `_done_item()` 把 `content/receipt_summary/summary` 包成 `observation_report_item()`。
- `observation_report_item()` 固定输出 `slot=body`、`surface=assistant_body`、`source_authority=model`。
- 结果是 runtime terminal summary 可以绕过前端保护，成为正文。

风险：

- 工具完成、运行收口、系统摘要会进入用户可复制正文。
- 前端“done 不覆盖正文”的测试不能防住 envelope body。

### 2. control item 可能被前端拼回 assistant body

位置：

- `frontend/src/components/chat/ChatMessage.tsx`

现状：

- `askUserQuestionFromPublicTimelineItems()` 会扫描 control item。
- 只要 control item 有 detail/text，就可能被拼进 assistant body。

风险：

- 等待确认、暂停、停止、steer ack 等控制消息可能变成正文。
- 即使后端正确投到 control surface，前端仍会破坏边界。

### 3. 阶段总结契约没有闭环

位置：

- `backend/harness/runtime/projection/items.py`
- `backend/harness/runtime/projection/projector.py`
- `frontend/src/components/chat/PublicTimelineActivity.tsx`

现状：

- `stage_summary_item()` 已定义 `covers_tool_refs` 和 `collapse_after_body_feedback`，但主链路没有稳定使用。
- 前端目前主要靠“后面出现 body 就折叠之前工具”的启发式。

风险：

- 工具记录折叠不精确。
- 无关正文也可能折叠工具。
- 用户需要的“简短工具反馈 + 阶段总结 + 之前工具折叠”没有成为协议。

### 4. 实时 SSE 与持久化回放不是同一条投影权威

位置：

- `backend/api/chat.py`
- `backend/harness/runtime/session_timeline.py`
- `backend/harness/runtime/projection/timeline_builder.py`

现状：

- 实时 SSE 使用 `public_projection_envelope`。
- 会话恢复时又从 runtime events 和 progress entries 合并 public timeline。

风险：

- 刷新后工具、反馈、正文可能重复或排序变化。
- 实时链路修好了，持久化回放仍可能漂移。

### 5. active turn 控制状态映射不严

位置：

- `backend/harness/runtime/projection/authority.py`
- `frontend/src/lib/projection/reducer.ts`
- `frontend/src/lib/store/types.ts`

现状：

- 前端 active turn 类型只允许 `starting/model_turn/running_task/waiting_* /terminal`。
- projection reducer 对 envelope 里的 state 没有严格白名单。
- 后端可能把 task status 直接当 active turn state 传出。

风险：

- steer、pause、continue、approval 判断依据不稳。
- 任务状态和交互控制状态混用。

### 6. 工具显示仍依赖启发式文案

位置：

- `backend/harness/runtime/projection/items.py`
- `frontend/src/components/chat/PublicTimelineActivity.tsx`

现状：

- 工具标题由工具名和 action_kind 推断。
- 子 agent 只靠工具名包含 `agent` 映射成“子任务”。
- 工具 waiting、done、failed、covered-by-summary 没有完整结构字段。

风险：

- 子 agent 等待不明显。
- 工具显示和折叠依赖文案，不够稳定。

## 四、分阶段修复方案

### 阶段 1：修复正文权威边界

目标：

- 清除 runtime summary、done summary、tool observation 进入 assistant body 的路径。

修改点：

1. 拆分 `items.py` 构造函数：
   - 保留 `model_body_item()`，只给模型正文使用。
   - 新增或改造 `model_stage_summary_item()`，只接收模型公开判断。
   - 新增 `runtime_timeline_item()`，用于 runtime/status 反馈。
   - 新增 `tool_observation_item()` 或强化 `work_action_item()`，确保工具观察只进 tool window。
2. 改造 `projector.py`：
   - `_done_item()` 不再返回 body item。
   - `runtime_step_summary` 根据 `presentation_source` 或明确 provenance 决定：
     - `model_action.public_action_state/current_judgment` 可以生成模型阶段反馈。
     - `tool_observation.summary` 只能生成 tool item。
     - `task_run_completed` 只能生成 control/status closeout，不能生成 body。
3. 修改 `authority.py`：
   - body item 必须同时满足 `slot=body`、`surface=assistant_body`、`source_authority=model`、`body_origin=model_output|model_public_judgment`。
   - 对没有 body_origin 的旧 body-looking item fail closed。

验收：

- `done.summary` 不生成 assistant body。
- 工具观察不生成 assistant body。
- runtime status 不生成 assistant body。
- 模型真实 `current_judgment` 可以生成 stage summary。

### 阶段 2：修复前端 body/control 混入

目标：

- control surface 永远不能被 ChatMessage 拼回正文。

修改点：

1. 删除或收窄 `askUserQuestionFromPublicTimelineItems()`：
   - control item 不允许生成 assistant body。
   - 真正的 ask_user 必须由后端投为模型 body item，或用专门 `kind=model_question`。
2. 强化 `isPublicTimelineBodyItem()`：
   - 不再只凭 kind 判断 body。
   - 必须有明确 slot/surface/source/body_origin。
3. `PublicTimelineActivity` 继续可显示 control activity，但不进入可复制正文。

验收：

- waiting approval detail 不出现在 assistant body。
- pause/stop/steer ack 不出现在 assistant body。
- ask_user 的真实模型问题仍能作为正文显示。

### 阶段 3：建立阶段总结和工具折叠协议

目标：

- 工具反馈简短，阶段总结负责收口，折叠由协议驱动，不由前端猜。

协议：

```json
{
  "kind": "stage_summary",
  "slot": "body",
  "surface": "assistant_body",
  "source_authority": "model",
  "body_origin": "model_public_judgment",
  "text": "已确认文件路径和前端流式状态，问题集中在正文投影入口。",
  "next_step": "继续修复 done/control 边界。",
  "covers_tool_refs": ["tool:read-events", "tool:read-chat-message"],
  "collapse_after_body_feedback": true
}
```

工具 item 必须有：

```json
{
  "kind": "work_action",
  "slot": "tool",
  "surface": "tool_window",
  "source_authority": "tool",
  "tool_ref": "tool:read-events",
  "state": "running|waiting|done|error",
  "stream_state": "streaming|done"
}
```

前端折叠规则：

1. running/error 工具永不自动折叠。
2. done 工具只有在被后续 stage summary 的 `covers_tool_refs` 覆盖时才折叠。
3. 没有 `covers_tool_refs` 时只按用户手动折叠，不自动折叠。

验收：

- 工具之后有阶段总结时，相关工具折叠。
- 无关正文不会折叠工具。
- 失败工具保持展开。
- 子 agent wait 状态明确显示为等待，不被当作普通工具完成。

### 阶段 4：统一实时投影和持久化回放

目标：

- 页面实时显示和刷新恢复显示完全同源。

修改点：

1. `session_timeline.py` 不再用 progress entries 生成第二套 body。
2. 持久化回放统一调用 `project_runtime_monitor_event_public_delta()` 或同级 projector。
3. progress entries 只能作为 task_projection 的补充数据，不再作为 public body 权威。
4. 合并逻辑只按稳定 `item_id/tool_ref/stage_ref` 去重，不按文案猜。

验收：

- 实时看到什么，刷新后仍是什么。
- 不出现重复工具。
- 不出现工具跑到下一条回答。
- 不出现恢复后正文丢失或多出系统状态词。

### 阶段 5：收紧 active turn/control 状态

目标：

- 控制状态精确、可 steer、可暂停、可等待审批，不被任务状态污染。

修改点：

1. 后端 `_active_turn_update()` 只输出合法 ActiveTurnState：
   - running task -> `running_task`
   - waiting executor -> `waiting_executor`
   - waiting approval -> `waiting_approval`
   - pause requested -> `waiting_safe_boundary`
   - terminal task -> `terminal`
2. 前端 projection reducer 对 active turn state 使用白名单；未知状态丢弃，不覆盖旧状态。
3. waiting_approval、waiting_safe_boundary 显示在 control/task activity，不作为正文。

验收：

- pause 后立刻进入可见控制状态。
- stop 后 active turn terminal，不再可 steer。
- waiting approval 不阻塞错误返回和页面显示。
- continue/steer 不写入上一条或下一条回答正文。

### 阶段 6：Prompt 配套修复

目标：

- 模型知道什么时候该说正文，什么时候只调用工具，什么时候给阶段总结。

修改点：

1. `compiler.py` 的 action schema 保留禁止泛化状态词规则。
2. 增加明确阶段反馈规则：
   - 首次工具前：只有已有真实判断或边界时才写 `current_judgment`。
   - 工具返回后：如果继续工具，必须基于观察写一句短阶段判断；不能写“工具返回成功，正在继续”。
   - 阶段完成：写阶段总结，并列出覆盖的 evidence/tool refs。
   - 收口：最终答案只写在 final answer/assistant_text_final，不依赖 done summary。
3. 子 agent prompt 规则：
   - spawn 后如果结果未返回，下一步应 wait_subagent 或说明等待，不得直接最终收口。
   - wait_subagent 返回后必须综合子 agent 观察再阶段总结或最终回答。

验收：

- 模型不再输出“正在思考/开始处理/处理完成”。
- 工具后有真实观察反馈。
- 阶段总结有覆盖 refs。
- 子 agent 有等待和结果整合。

### 阶段 7：测试和实测

后端测试：

- `done.summary` 不生成 assistant body。
- task control done hidden。
- runtime status/control 不生成 body。
- tool observation 只进 tool window。
- stage summary 只有模型公开判断能生成，并携带 `covers_tool_refs`。
- 持久化回放和实时 projector 结果一致。

前端测试：

- control detail 不拼入 assistant body。
- waiting approval/safe boundary 只显示控制状态。
- stage summary 才折叠 covered tool refs。
- running/error tool 不折叠。
- active turn state 拒绝非法状态。
- SSE CJK delta 继续稳定追加。

联调验证：

固定端口：

- 前端：`http://127.0.0.1:3000`
- 后端：`http://127.0.0.1:8003`

必须真实启动并验证：

1. 普通直接回答。
2. 首次直接工具调用。
3. 工具返回后模型阶段总结。
4. 多工具后阶段总结折叠。
5. 子 agent spawn/wait。
6. pause/continue/stop/steer。
7. waiting approval。
8. final answer + done closeout。
9. 刷新恢复同一条运行。
10. 错误返回。

## 五、禁止项

1. 不允许为了页面好看添加“正在思考”“开始处理”“处理完成”正文。
2. 不允许把 runtime/control/tool 的文本伪装成 `source_authority=model`。
3. 不允许用前端清洗列表替代后端权威分类。
4. 不允许让投影失败阻塞工具调用或模型续跑。
5. 不允许保留旧投影主链路作为兼容兜底。
6. 不允许通过降低测试断言制造通过。

## 六、推荐实施顺序

1. 先修 `done/control -> body` 的硬泄露。
2. 再修 stage summary 和工具折叠协议。
3. 再统一持久化回放。
4. 再收紧 active turn 控制状态。
5. 最后补 prompt 和端到端测试。

这个顺序保证：先清除泄露，再建立成熟展示，再收紧恢复和控制，不会因为投影问题堵塞 agent 执行链路。

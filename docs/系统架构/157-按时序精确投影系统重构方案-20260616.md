# 按时序精确投影系统重构方案

日期：2026-06-16  
状态：待确认后执行  
目标架构名：Chronological Projection  
范围：后端公开投影协议、session runtime projection、前端投影 reducer、store 可见刷新、ChatMessage / PublicTimelineActivity 渲染、相关结构性测试

## 1. 问题定义

这次不是继续修一个“重复显示”或“工具投影消失”的局部问题，而是重做主聊天投影系统。

用户提出的两个硬要求是：

1. **按时序精确投影**：模型正文、工具请求、权限、工具开始、工具完成、状态反馈、commit、terminal 必须按同一事件时钟投影；不能靠文本去重、组件猜测或刷新恢复时的附加规则拼出来。
2. **不要有之前的重度刷新问题**：流式期间不能每个小 frame 都造成全局消息数组、整页订阅、Markdown 和工具面板一起重算；已完成消息应成为稳定快照，只让正在运行的投影增量刷新。

当前真正损坏的系统属性是：

```text
单一事实源
单一投影折叠权威
单一显示视图模型
增量可见刷新
```

现在这四件事被分散到了后端 session attachment、前端 projection reducer、runtime hydration、ChatMessage 渲染启发式、runtimeDisplayState patch、以及 store 全局 messages 刷新里，所以继续修补会反复出现同类问题。

## 2. 实施前取证

以下取证描述的是重构前的旧主路径，用于解释为什么必须删除旧链路；它不是目标实现。

### 2.1 旧正文所有权重复

`frontend/src/lib/projection/reducer.ts:115-133` 的 `patchProjectionMessage()` 在折叠 frame 后写入：

```text
content: publicProjection.bodyText || message.content
```

这意味着 live projection body 会回写 `message.content`。

但 `frontend/src/components/chat/ChatMessage.tsx:90-139` 又同时读取：

```text
publicProjection.bodyText
assistantDisplayContent(content)
projectionBodyText || assistantContentText
```

所以同一段模型正文可以同时存在于：

- `message.content`
- `message.publicProjection.bodyText`
- `message.publicProjection.bodyBlocks`
- history hydrate 生成的 runtime attachment projection
- ChatMessage 的 fallback display content

用户截图里同一句 assistant 正文重复出现，符合这个结构性根因。

### 2.2 投影折叠和显示决策分裂

`frontend/src/lib/store/events.ts:1389-1419` 的 SSE reducer 流程是：

```text
bind stream anchor
-> patch assistant stream anchor
-> applyPublicProjectionFrame
-> patchAssistantRuntimeDisplaySurface
-> patchActiveTaskTurnGate
-> applyChatStreamConnectionStatus
```

问题不是这些步骤都不该存在，而是 `applyPublicProjectionFrame()` 和 `patchAssistantRuntimeDisplaySurface()` 分别修改同一条 assistant message：一个决定正文/工具投影，一个决定 main surface。这样 `ChatMessage` 必须再用启发式把二者合起来。

目标系统必须让“显示模式”成为投影视图模型的结果，而不是 reducer、runtime attachment、组件三方各自裁决。

### 2.3 旧 Hydration 会 replay projection 并再次 patch surface

旧版 `hydrateSessionRuntimeProjection()` 会从 runtime attachments 收集裸 frame 列表，排序后直接 replay 到 message projection。

随后 `frontend/src/lib/store/runtime.ts:166-202` 又用 `applyRuntimeAttachmentDisplayState()` 独立 patch：

- `runtimeDisplayState`
- `mainChatSurface`
- `closeoutSummary`
- `runtimeLogRef`
- source ids

这导致刷新恢复不是“读取同一个投影视图快照”，而是“先 replay frames，再 patch display state”。如果 history 里已有 committed assistant content，hydration 仍可能把 live projection body 重新塞回主消息，造成重复或表面错位。

### 2.4 旧后端已有 public ledger，但仍混有 raw event reproject

旧版 `backend/harness/runtime/session_timeline.py` 已能从 `chatrun:*` public ledger 读取 stream attachment，但返回的是裸 frame 列表和独立 display state。

但同文件 `backend/harness/runtime/session_timeline.py:463-555` 和 `backend/harness/runtime/session_timeline.py:798-902` 仍会从 task/turn raw events 重新生成 projection frames。

这造成两个问题：

- live path 和 restore path 可能不是同一批 frame identity。
- task/turn trace attachment 可能把诊断或恢复信号当成主聊天 projection 重新投出来。

目标系统必须规定：主聊天投影只消费 canonical public event ledger 或 chronological projection slice；raw runtime event tail 只能用于 debug trace / diagnostics，不能再作为主聊天投影重建源。

### 2.5 ChatMessage 承担了过多协议解释

`frontend/src/components/chat/ChatMessage.tsx:588-718` 会把 `projection.timeline`、`currentAction`、`status`、`pinned`、`finalResults`、`trace` 合并、coalesce、转换成 `PublicChatTimelineItem`。

这让渲染组件知道了太多协议细节：

- trace/timeline/status 如何降级
- 工具窗口如何合并
- 模型等待占位如何过滤
- tool window title/command/output 如何拼
- body offset 如何和 timeline offset 比较

目标系统里，ChatMessage 只消费 `ProjectionViewModel.blocks`。它不再读取 raw `trace/timeline/status`，不再做协议推理。

### 2.6 高频刷新范围过大

`frontend/src/lib/store/events.ts:1262-1275` 的 `patchAssistant()` 会对整个 `state.messages` 做 `map()`。

`frontend/src/lib/projection/reducer.ts:123-133` 每个投影 frame 都复制 `messages` 数组并替换消息对象。

已有 `156-前端流式输出流畅度优化计划书-20260616.md` 也确认：当前 `useAppStore()` 订阅完整 store snapshot，纯正文 delta 会放大全局 setState、ChatPanel 滚动、ChatMessage Markdown 和 timeline 计算成本。

新投影系统必须从数据结构上支持“只更新当前 active projection”，而不是继续在旧 `messages: Message[]` 上做高频全量替换。

## 3. 设计原则

### 3.1 单一事实源

事实源只有一个：

```text
Runtime/Public Event Log
```

`PublicProjectionFrame` 可以是 event log 的标准化视图，但不允许由不同路径重新发明 frame。session restore、SSE reconnect、live stream 都必须读取同一事件身份体系。

### 3.2 单一折叠权威

所有 frame 必须经过一个 projection reducer：

```text
ProjectionFrameNormalizer
-> ProjectionAccumulator
-> ProjectionViewModelBuilder
```

ChatMessage、runtime hydration、session attachment 都不允许自行解释 trace/timeline/status/body。

### 3.3 正文权威分离

`message.content` 只表示已提交、可持久化的最终 assistant 正文。

live projection body 只能存在于 projection ledger/view model 中，不能写回 `message.content`。

规则：

| 状态 | 正文来源 | 是否写 `message.content` |
| --- | --- | --- |
| live / streaming | `ProjectionViewModel.blocks[kind=body_segment]` | 否 |
| finalized but not committed | projection ledger | 否 |
| committed / history message | canonical session message content | 是 |
| failed / no commit | recovery/status block + log entry | 否 |

### 3.4 显示模式由 reducer 输出

显示模式不是 ChatMessage 猜出来的，也不是 runtime attachment 单独 patch 出来的。

目标模式：

```text
ProjectionDisplayMode =
  live
  committed
  closeout
  recovery
  log_only
```

`ProjectionViewModel` 必须带上 display mode。UI 只按 mode 渲染。

### 3.5 时序严格、幂等 replay

每个 frame 必须有稳定身份：

```text
FrameIdentity = {
  event_log_id,
  event_offset,
  frame_id
}
```

replay 时以 `FrameIdentity` 去重，以 `event_offset -> frame_id` 排序。同一个 frame 反复到达只能影响一次；out-of-order 到达允许被 accumulator 重排到正确位置。

### 3.6 运行态和完成态分离

已完成消息应冻结为 `MessageSnapshot`：

```text
MessageSnapshot = {
  messageId,
  role,
  content,
  projectionSummary?,
  logRef?,
  sourceRefs
}
```

正在运行的消息才持有 live `ProjectionLedger`。commit 后前端应冻结快照，释放或压缩 live ledger，只保留日志入口和必要 summary。

### 3.7 性能不是旁路

优化不能新增第二套正文拼接权威。

允许：

- reducer 每个 frame 更新 canonical projection store；
- 可见刷新按 rAF/33-50ms 合并 body frame；
- control/tool/error/terminal 立即 flush；
- 已完成消息不再随 active projection 重渲染。

禁止：

- 另建 `pendingTextByMessageId` 绕过 projection；
- 用 DOM append 伪造流式输出；
- 延迟工具、权限、错误、预算、terminal 信号。

## 4. 目标架构

### 4.1 总体链路

```text
Backend Runtime Event
-> Public Event Log / chatrun:* ledger
-> ProjectionFrameNormalizer
-> ProjectionSlice API / SSE frame
-> Frontend ProjectionAccumulator
-> ProjectionViewModel
-> ChatMessageView
```

目标是让 live stream 与 history restore 走同一个 fold 逻辑：

```text
SSE public_projection_frame       -> normalize -> fold -> view
session projection slice replay   -> normalize -> fold -> view
```

区别只在 frame 来源和 slice 边界，不在投影解释逻辑。

### 4.2 后端职责

后端只负责事实规范化和 bounded slice，不负责前端显示启发式。

后端输出：

```text
ProjectionSlice = {
  schema_version: "chronological_projection",
  projection_key,
  event_log_id,
  start_offset,
  end_offset,
  cursor,
  display_hint,
  commit,
  terminal,
  frames[]
}
```

其中 `display_hint` 只表达事实状态：

```text
display_hint = {
  lifecycle: running | committed | failed | stopped | log_only,
  main_surface_hint: live | committed | closeout | recovery | log_only,
  log_ref,
  tool_event_count
}
```

真正的 blocks 仍由前端 chronological reducer 从 frames 构建。

### 4.3 前端职责

前端新增 `chronological` 投影模块，建议目录：

```text
frontend/src/lib/projection/chronological/
  types.ts
  normalize.ts
  accumulator.ts
  viewModel.ts
  storeAdapter.ts
  __tests__/
```

职责分层：

| 模块 | 职责 | 禁止 |
| --- | --- | --- |
| `normalize.ts` | 校验 frame、生成 `ProjectionKey` / `FrameIdentity` | 不决定 UI 文案 |
| `accumulator.ts` | 幂等折叠 body/tool/status/commit/terminal | 不读 React / store |
| `viewModel.ts` | 输出 ordered render blocks 和 display mode | 不访问 raw store |
| `storeAdapter.ts` | 把 view model 绑定到 MessageSnapshot / active projection store | 不解释 frame 内容 |
| `ChatMessage.tsx` | 渲染 view blocks | 不读取 raw trace/timeline/status |

### 4.4 数据模型

#### ProjectionKey

```text
ProjectionKey = {
  sessionId: string
  turnId: string
  messageId?: string
  streamRunId?: string
  turnRunId?: string
  taskRunId?: string
}
```

匹配优先级：

1. `messageId`
2. `streamRunId`
3. `turnRunId`
4. `taskRunId`
5. `turnId`

如果一个 frame 无法形成 `ProjectionKey`，它只能进入 debug diagnostics，不能进入主聊天。

#### ProjectionLedger

```text
ProjectionLedger = {
  key: ProjectionKey
  seenFrames: Set<FrameIdentity>
  cursor: {
    minOffset,
    maxOffset,
    lastCommittedOffset?
  }
  bodySegments: OrderedMap<segmentId, BodySegment>
  toolLifecycles: OrderedMap<toolUseId, ToolLifecycle>
  statusEvents: OrderedMap<statusId, StatusEvent>
  pinnedEvents: OrderedMap<pinnedId, StatusEvent>
  commit?: CommitState
  terminal?: TerminalState
  diagnostics: ProjectionDiagnostic[]
}
```

注意：`seenFrames` 是 ledger 内部索引，不再把不断增长的 `sourceFrameIds` 暴露给每个 UI block。

#### ProjectionViewModel

```text
ProjectionViewModel = {
  key: ProjectionKey
  displayMode: live | committed | closeout | recovery | log_only
  canonicalContent?: string
  copyText?: string
  logRef?: string
  toolEventCount?: number
  blocks: ProjectionRenderBlock[]
}
```

#### ProjectionRenderBlock

```text
ProjectionRenderBlock =
  | { kind: "body_segment", id, text, firstOffset, lastOffset, state }
  | { kind: "tool_event", id, title, state, target, commandLine?, summary?, firstOffset, lastOffset }
  | { kind: "status_event", id, title, detail?, state, offset }
  | { kind: "commit_event", id, state, offset }
  | { kind: "terminal_event", id, state, reason?, offset }
  | { kind: "log_entry", id, logRef, toolEventCount }
```

UI 不再从 `trace`、`timeline`、`currentAction` 临时拼 blocks。

### 4.5 Store 结构

目标 store 从“所有消息都在 `messages: Message[]` 上高频变更”改为：

```text
messagesById: Record<MessageId, MessageSnapshot>
messageOrder: MessageId[]
activeProjectionsByKey: Record<ProjectionKeyString, ProjectionRuntimeState>
messageProjectionKeyById: Record<MessageId, ProjectionKeyString>
```

为了降低一次性风险，实施时可以先在现有 store 外挂 `activeProjectionsByKey`，但切换完成后必须删除旧 `message.publicProjection` 主路径。

主原则：

- 已完成消息只更新 `messagesById[messageId]` 一次。
- live projection 只更新 `activeProjectionsByKey[key]`。
- ChatPanel 通过 selector 订阅 message order + 当前 active projection view。
- 纯正文 frame 合并可见 flush，工具/control/terminal 立即 flush。

## 5. 固定执行流

### 5.1 Live streaming

```text
SSE event(public_projection_frame)
-> normalize frame
-> derive ProjectionKey + FrameIdentity
-> accumulator.fold(frame)
-> build view model
-> schedule visible update
```

刷新策略：

- `body_append`：允许 rAF / 33-50ms 合并可见刷新。
- `tool_*`、`permission_*`、`status`、`commit`、`terminal`、`error`：立即可见刷新。

### 5.2 Tool lifecycle

```text
tool_call_requested
-> tool_permission_decided
-> tool_item_started
-> tool_item_completed
```

这四类事件必须更新同一个 `tool_event` block。

合并键：

```text
tool_use_key = tool_call_id || tool_lifecycle_id
```

如果缺少稳定 tool id，frame 进入 projection diagnostic，不进入主聊天工具窗口。

### 5.3 Body / tool / body 精确混排

输入：

```text
body_append(offset=1, "先检查")
tool_call_requested(offset=2)
tool_item_completed(offset=4)
body_append(offset=5, "检查完成")
```

输出 blocks：

```text
body_segment(offset 1)
tool_event(offset 2-4)
body_segment(offset 5)
```

`body_finalize` 只更新 body state 和 canonical body，不允许把完整正文重放成最后一个大 block 覆盖前面的时序。

### 5.4 Commit / closeout

```text
commit_ack / assistant_text_final / turn_terminal
-> viewModel.displayMode = committed 或 closeout
-> freeze MessageSnapshot
-> show canonical content / closeout summary once
-> append log_entry when tool events exist
-> release or compress live ledger
```

commit 后主聊天不再渲染 live tool blocks。完整工具轨迹进入日志/trace。这样既保证主聊天干净，也不丢工具投影。

### 5.5 Hydration / reconnect

```text
session history messages
-> MessageSnapshot
session projection slices
-> normalize + replay frames
-> viewModel
-> bind snapshot to active/live/closeout display
```

规则：

- 如果 snapshot 已有 committed `content`，projection body 不得再写回 content。
- 如果 slice 有 `commit_ack`，显示 canonical content / closeout summary 一次。
- 如果 slice 仍 running，恢复 live timeline。
- 如果 slice 是 log_only，只产生日志入口或 trace ref，不进入主聊天 blocks。

### 5.6 Completed message persistence

完成后的会话：

```text
MessageSnapshot(content)
ProjectionSummary(logRef, toolEventCount, terminalState)
```

前端无需保存完整 live ledger 在主 store；需要查看日志时再按 `logRef` 加载完整 trace。

## 6. 与当前方案的关键区别

| 当前系统 | Chronological 目标 |
| --- | --- |
| projection body 回写 `message.content` | live body 永不写 content |
| ChatMessage 同时读 bodyText、content、trace、timeline | ChatMessage 只读 ViewModel blocks |
| runtime hydrate replay frames 后再 patch surface | projection slice replay 直接产出 displayMode |
| session_timeline 混用 public ledger 和 raw event reproject | 主聊天只消费 public ledger / chronological slice |
| 每 frame 复制 `messages[]` | active projection key 局部更新 |
| body block 带增长的 sourceFrameIds | ledger 内部 seen index，UI block bounded |
| 文本级去重兜底 | FrameIdentity / segmentId / toolUseId 幂等 |
| 旧 reducer 和 UI heuristics 长期共存 | shadow 验证后删除旧路径 |

6. ChatMessage 对非 active message 保持 memo 稳定。

完成标准：

- 纯正文流式期间不再每个 frame 触发整页 store snapshot 刷新。
- 工具观察、错误、预算、terminal 不被延迟。
- 长文本输出肉眼连续，Performance 中长任务显著减少。

## 11. 迁移与删除规则

本重构允许短期 shadow reducer，但不允许长期双轨。

实施期允许短期存在：

- 旧 reducer 与 Chronological reducer 同时在测试中对同一 frame 运行，用于对照。
- 后端可继续使用 `runtime_attachments` 作为外层恢复容器，但主投影内容必须放在 `projection_slices` 中，不再返回裸 frame 主路径。

必须在切换完成后删除：

- projection body 写入 `message.content`。
- ChatMessage raw trace/timeline/status 合并逻辑。
- session_timeline 主聊天 raw event reproject。
- 保护旧 `MessagePublicProjection` 内部形状的测试。
- 用文本相等作为主去重依据的投影逻辑。

不允许以“兼容”为理由保留两条主聊天投影链路。

## 12. 禁止事项

1. 禁止 live projection body 写入 `message.content`。
2. 禁止 ChatMessage 读取 raw `trace/timeline/status/currentAction` 做协议解释。
3. 禁止 session restore 从 raw task/turn event tail 重建主聊天 projection。
4. 禁止用文本相等修复重复正文。
5. 禁止用到达时间、`Date.now()` 或数组位置决定展示顺序。
6. 禁止把工具 observation 拼进 assistant body。
7. 禁止为流畅度新增第二套正文缓存权威。
8. 禁止延迟工具、权限、错误、预算、terminal 反馈。
9. 禁止保留旧投影主路径作为长期 fallback。
10. 禁止改固定端口或绕过真实前后端验证。

## 13. 预期结果

重构完成后，主聊天投影系统应变成：

```text
一个事件时钟
一个 projection key
一个 reducer
一个 view model
一个 UI 消费面
```

用户能看到：

- 模型正文不重复。
- 工具投影稳定出现，不会刷新后消失。
- 任务 live 期间按真实顺序展示正文和工具。
- 任务完成后主聊天收口干净，但日志仍保留完整工具轨迹。
- 长文本流式输出明显更顺，不再整页重刷。

工程上能得到：

- live / replay / hydration 同源。
- 正文、工具、状态、commit 权责清晰。
- 前端组件不再承担协议解释。
- 后续新增投影类型时只改 reducer/view model，不再到 ChatMessage 里补启发式。

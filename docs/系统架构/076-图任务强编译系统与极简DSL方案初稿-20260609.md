# 076-图任务强编译系统与极简 DSL 方案初稿

日期：2026-06-09  
状态：初稿，供讨论，不作为最终实施计划  
关联讨论稿：`075-图任务Agent级环境解耦讨论稿-20260609.md`

## 1. 一句话目标

把图任务系统从“用户手写大量底层契约的配置系统”升级为“用户写极简意图图，编译器生成完整运行契约的自动化系统”。

用户只描述：

- 要完成什么目标。
- 哪些 agent / 角色参与。
- 节点之间是什么语义关系。
- 哪些输出或人工门控是业务必须显式指定的。

系统自动生成：

- 节点运行契约。
- 边交接协议。
- 输入输出 schema。
- acceptance / review 规则。
- prompt contract。
- agent 级环境锁定。
- memory / artifact / file access 策略。
- 权限和工具天花板。
- scheduler view。
- checkpoint / resume / retry 策略。
- compile report。

## 2. 核心判断

现在图任务最大的瓶颈不是缺字段，而是配置层级错误。

当前许多底层运行结构暴露给用户，导致用户必须理解 `contract_bindings`、`runtime_policy`、`artifact_policy`、`memory_policy`、edge handoff、acceptance、环境、权限等大量细节。成熟系统不应该这样。成熟系统应该让用户表达业务意图，让编译器用模板、profile、规则和校验把意图编译成严谨的运行产物。

因此，升级重点应该是：

```text
少配置
+ 强编译
+ 可解释报告
+ fail-closed 校验
+ runtime 只执行编译产物
```

## 3. 目标架构

建议采用四层：

```text
Authoring DSL
-> GraphCompiler
-> GraphCompilationUnit
-> GraphHarnessConfig
-> GraphRuntime
```

### 3.1 Authoring DSL

这是用户主要编辑的层。它应该短、稳定、接近业务语言。

示例：

```yaml
graph_id: graph.novel.chapter_pipeline
title: 章节生产线
goal: 完成一章从起草、审核到修订的连续任务
control_environment: env.creation.writing

nodes:
  - id: draft
    agent: writer
    intent: chapter_draft

  - id: review
    agent: verifier
    intent: quality_review

  - id: revise
    agent: writer
    intent: chapter_revision

edges:
  - draft -> review: review_feedback
  - review -> revise: revision_request
  - revise -> review: review_feedback

outputs:
  - from: review
    kind: accepted_chapter
```

### 3.2 GraphCompiler

编译器是系统核心。它负责把 DSL 编成可运行、可恢复、可审查的 graph artifact。

它不应该靠运行时猜测，也不应该把缺失字段默默兜底成危险默认值。编译器可以自动补全，但必须能解释补全来源。

### 3.3 Node Instance Builder

节点配置不应该全部由用户手写。建议引入“节点实例构建器”：由系统编排一个单 agent，根据极简节点种子、图目标、上下游关系、agent profile、环境候选和模板库，生成节点配置实例草案。

基本链路：

```text
NodeSeed
-> NodeInstanceBuilderAgent
-> NodeConfigurationInstanceDraft
-> GraphCompiler validation
-> Compiled node contract
```

这里的单 agent 是配置实例构建者，不是图运行时的特殊分支。它负责把“这个节点想干什么”转成“这个节点应该如何运行”的草案。最终能不能发布，仍由确定性编译器裁决。

节点实例草案可以包含：

- node title / role。
- agent_id / agent_profile_id。
- task_environment_id 候选。
- prompt contract 草案。
- input / output contract 草案。
- edge input slot 建议。
- artifact / memory 需求。
- tool / permission 需求。
- acceptance criteria。
- compile rationale。

它不能直接发布，不能扩大权限，不能绕过模板库和 registry。它只能提出候选配置，GraphCompiler 负责校验、收敛、拒绝或发布。

### 3.4 GraphCompilationUnit

这是建议新增的中间产物。它用于记录编译过程中的结构化结果，不直接给 runtime 执行。

建议包含：

```text
normalized_graph
resolved_agents
resolved_node_intents
resolved_edge_relations
node_environment_index
node_contract_plan
edge_contract_plan
memory_plan
artifact_plan
permission_plan
scheduler_plan
compile_report
```

### 3.5 GraphHarnessConfig

这是最终发布物，runtime 只消费它。

`GraphHarnessConfig` 应保持不可变发布物语义。它不关心用户配置多简单，只关心编译后是否完整、可验证、可恢复。

## 4. 最小 DSL 边界

建议最小必填字段只有：

- `graph_id`
- `goal`
- `nodes[].id`
- `nodes[].agent` 或 `nodes[].intent`
- `edges`

可选字段：

- `title`
- `control_environment`
- `node.goal`
- `node.output`
- `node.environment`
- `node.model`
- `node.human_gate`
- `edge.relation`
- `edge.condition`
- `outputs`
- `constraints`

原则：用户可以覆盖，但不需要覆盖。

如果用户什么都不覆盖，编译器也应能根据 agent profile、intent template、edge relation template 生成完整运行图。

## 5. 编译流水线

建议编译器分七个 pass。

### Pass 1：Normalize

输入：Authoring DSL  
输出：NormalizedGraphSpec

职责：

- 标准化 node id、edge id、agent ref、intent ref。
- 解析 `draft -> review: review_feedback` 这类短语法。
- 补全 graph title、start nodes、terminal nodes。
- 拒绝重复节点、断边、未知语法。

### Pass 2：Resolve

输入：NormalizedGraphSpec  
输出：ResolvedGraphSpec

职责：

- 从 agent registry 解析 agent。
- 从 runtime profile 解析 agent_profile。
- 从 node intent template registry 解析节点意图。
- 从 edge relation template registry 解析边语义。
- 从 environment registry 解析环境。
- 从 task/domain template 解析领域默认输出、验收和记忆策略。

### Pass 2.5：Build Node Instances

输入：ResolvedGraphSpec  
输出：NodeConfigurationInstanceDrafts

职责：

- 对缺少详细配置的节点，调用单 agent 节点实例构建器。
- 为每个节点生成配置实例草案。
- 把上下游边、图目标、agent 能力、环境候选和模板约束提供给 builder agent。
- 要求 builder agent 输出结构化草案和理由。
- 禁止 builder agent 直接发布或绕过权限校验。

关键原则：

```text
builder agent 负责生成候选
compiler 负责校验和裁决
runtime 只消费发布产物
```

### Pass 3：Synthesize Contracts

输入：ResolvedGraphSpec + NodeConfigurationInstanceDrafts  
输出：ContractPlan

职责：

- 自动生成 node input contract。
- 自动生成 node output contract。
- 自动生成 prompt contract。
- 自动生成 acceptance contract。
- 自动生成 edge handoff contract。
- 自动生成 output / artifact contract。

关键规则：

- prompt contract 必须是 agent 能直接执行的角色指令。
- 不能把开发说明写进 prompt。
- 缺少必要业务语义时 fail closed，而不是生成空泛 prompt。

### Pass 4：Bind Environment And Resources

输入：ResolvedGraphSpec + ContractPlan  
输出：EnvironmentResourcePlan

职责：

- 解析 graph control environment。
- 解析每个 node effective environment。
- 锁定环境 payload。
- 锁定 file access table。
- 锁定 artifact root。
- 锁定 memory namespace。
- 解析 resource nodes。
- 生成 `node_environment_index`。

节点环境优先级建议：

```text
node.environment
-> node.runtime.environment
-> agent_runtime_profile.task_environment_id
-> graph.control_environment
-> env.general.workspace
```

### Pass 5：Constrain Permissions

输入：ResolvedGraphSpec + EnvironmentResourcePlan  
输出：PermissionPlan

职责：

- 合并 agent profile allowed operations。
- 合并 environment execution / sandbox / file policy。
- 合并 node/graph contract permission ceiling。
- 取交集或显式收窄。
- 拒绝 runtime patch 可以扩大权限的结构。

核心原则：

```text
环境不能授予 agent 原本没有的能力。
图不能授予环境禁止的能力。
runtime patch 不能扩大编译期权限天花板。
```

### Pass 6：Plan Runtime

输入：前面所有 plan  
输出：RuntimePlan

职责：

- 生成 scheduler view。
- 生成 ready / blocked / terminal 规则。
- 生成 parallel / background / human gate 行为。
- 生成 checkpoint / resume 策略。
- 生成 retry / requeue 策略。
- 生成 monitor projection。

### Pass 7：Verify And Publish

输入：GraphCompilationUnit  
输出：GraphHarnessConfig + CompileReport

职责：

- 结构校验。
- 合同校验。
- 环境校验。
- 权限校验。
- artifact / memory 路径校验。
- 断点续跑结构 hash 校验。
- 输出 compile report。
- 发布 `GraphHarnessConfig`。

## 6. 模板系统

强编译系统必须依赖模板库，而不是让用户手写所有契约。

建议至少有五类模板：

### 6.1 NodeIntentTemplate

描述一个节点 intent 如何变成运行契约。

示例：

```yaml
intent_id: chapter_draft
default_agent_kind: writer
input_slots:
  - outline
  - world_context
  - chapter_requirements
outputs:
  - chapter_text
acceptance:
  - must_match_outline
  - must_preserve_style
prompt_template: writer.chapter_draft
artifact_template: longform.chapter_text
```

### 6.2 EdgeRelationTemplate

描述边语义如何变成交接协议。

示例：

```yaml
relation_id: review_feedback
source_output: chapter_text
target_input: review_material
delivery_policy: contract_payload_and_refs
ack_required: true
failure_policy: isolate_failure
```

### 6.3 AgentRuntimeProfile

描述 agent 的能力、模型、工具、默认环境、上下文边界和 prompt refs。

这里建议正式加入一等字段：

```text
task_environment_id
```

短期可以兼容读取 metadata，但编译产物必须写成标准节点环境索引。

### 6.4 EnvironmentProfile

描述任务环境的文件、产物、记忆、sandbox、risk、prompt mount plan。

### 6.5 OutputArtifactTemplate

描述常见输出如何物化成 artifact。

例如：

- chapter_text
- review_report
- world_setting
- code_patch
- test_report
- research_brief

## 7. Agent 级环境设计

图任务需要摆脱 graph-level environment 对所有节点的限制。

建议区分：

```text
graph_control_environment
node_effective_environment
```

`graph_control_environment` 用于 session scope、监控归属、图任务入口、项目选择。

`node_effective_environment` 用于节点执行、prompt mount、artifact root、memory namespace、file access、sandbox。

编译器发布时生成：

```text
node_environment_index[node_id]
```

runtime 不再自己解析 agent 环境，只按 work order 中的 effective environment 执行。

## 8. Compile Report

自动化越强，报告越重要。

每次编译必须输出：

- 每个节点选了哪个 agent。
- 每个节点用了哪个 runtime profile。
- 每个节点用了哪个 task environment。
- 每个节点的 input/output contract 来源。
- 每条边的 handoff contract 来源。
- 每个权限来自 agent、environment 还是 graph contract。
- 哪些字段是用户显式配置。
- 哪些字段是模板推导。
- 哪些字段是默认值。
- 哪些风险被拒绝。
- 哪些地方需要用户补充。

示例：

```text
node draft:
  agent: writer_agent
  environment: env.creation.writing
  environment_source: agent_runtime_profile.task_environment_id
  output_contract: contract.chapter_text
  output_contract_source: intent_template.chapter_draft
  allowed_operations:
    source: agent_profile ∩ environment ∩ node_contract
```

## 9. Runtime 边界

runtime 不做以下事情：

- 不根据 agent profile 重新决定环境。
- 不临时生成合同。
- 不扩大权限。
- 不猜测边协议。
- 不把 graph control environment 当成所有节点环境。

runtime 只做：

- 启动 graph run。
- 读取 published config。
- 生成 work order。
- 执行节点。
- 接收 result envelope。
- 推进 loop state。
- checkpoint。
- 恢复。
- 输出 monitor / trace。

## 10. 迁移策略

建议分三步，不一口吃掉全部复杂度。

### 阶段一：编译器解释层

目标：不改 runtime 行为，先新增 `GraphCompilationUnit` 和 compile report。

完成标准：

- 现有 graph 能生成 compilation unit。
- compile report 能解释当前所有节点、边、环境、权限来源。
- 不改变现有 `GraphHarnessConfig` 行为。

### 阶段二：极简 DSL 编译

目标：支持最小 DSL 编译成现有 `GraphHarnessConfig`。

完成标准：

- 用户只写 nodes / edges / agent / intent。
- 编译器生成 contracts、edge handoff、prompt contract、scheduler view。
- 新增最小 DSL 回归测试。

### 阶段三：Agent 级环境与节点环境索引

目标：实现 node effective environment。

完成标准：

- 一个 graph 内不同节点可使用不同环境。
- work order、TaskRunContract、runtime contract、artifact receipt、memory receipt 使用节点环境。
- graph control environment 只作为控制面归属。
- structure hash 包含节点环境边界。

## 11. 需要继续讨论的问题

1. Agent runtime profile 是否立即新增一等字段 `task_environment_id`？

我的倾向：是。metadata 可作为迁移输入，但最终模型应有一等字段。

2. DSL 是否需要支持自然语言节点 intent？

例如：

```yaml
- id: review
  agent: verifier
  intent: 检查章节是否达标
```

我的倾向：可以让 UI/LLM 助手把自然语言转成标准 intent，但编译器发布时必须使用标准 intent id。

3. 编译器是否允许自动创建缺失模板？

我的倾向：不允许自动发布。可以生成建议，但必须人工确认后进入模板库。

4. graph control environment 缺省是什么？

我的倾向：如果用户从某个任务环境启动，则使用当前环境；否则用 `env.general.workspace`。但这只影响控制面，不覆盖节点环境。

5. 低风险默认和高风险默认如何区分？

我的倾向：prompt、schema、handoff 可以模板默认；文件写入、shell、browser、network、memory commit 必须经过权限收敛，不允许危险默认。

6. 节点实例构建 agent 的边界是什么？

我的倾向：它只生成 `NodeConfigurationInstanceDraft`，不直接改 registry，不直接发布 graph config，不拥有权限裁决权。它的输出必须有 schema，并被 GraphCompiler fail-closed 校验。

## 12. 第一版实施边界建议

第一版不要做全量 UI，不要做 LLM 自动生成模板，不要改普通 single-agent turn 的环境规则。

第一版只做：

- 强编译器骨架。
- 节点实例构建 agent 的结构化输出协议。
- 极简 DSL 到 `GraphHarnessConfig`。
- compile report。
- agent 级环境锁定。
- 图任务 runtime 消费节点环境。
- 回归测试。

## 13. 验收标准

正式落地后至少要通过这些行为：

- 极简节点配置可以通过单 agent builder 生成 `NodeConfigurationInstanceDraft`。
- builder 输出不合法时，GraphCompiler 拒绝发布并给出 compile report。
- 一个 2 节点最小 DSL graph 能编译、发布、启动。
- 用户没有手写 input/output/handoff contract，编译器自动生成。
- compile report 能解释所有自动生成内容来源。
- writer 节点使用 `env.creation.writing`，code 节点使用 `env.coding.vibe_workspace`，互不覆盖。
- 修改 agent profile 环境后，已发布 graph config 的断点续跑仍使用旧锁定环境。
- runtime settings patch 不能切换节点环境。
- artifact 和 memory receipt 记录节点 effective environment。
- monitor 同时显示 graph control environment 和 node effective environment。

## 14. 暂定结论

图任务系统要达到成熟 agent 产品级别，不能继续让用户堆配置、手写合同。正确方向是：用户表达业务拓扑和 agent 意图，强编译器生成完整可运行图，runtime 只执行发布产物。这样才能同时做到低配置、高自动化、可恢复、可审查、可控权限和多 agent 多环境协作。

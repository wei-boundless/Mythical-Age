## 1. 一句话目标

把图任务的边从“节点之间的连线”升级成“可编译、可审查、可恢复的交互协议实例”。

用户仍然只写极简关系：

```yaml
edges:
  - draft -> review: submit_for_review
  - reviewer <-> writer: discuss_revision
  - review -> revise: when verdict == revise
```

编译器把它们展开成：

```text
CompiledEdgeContract
edge protocol kind
scheduler role
payload / message / artifact contract
source output selector
target input slot
ack / timeout / retry
conversation session policy
failure propagation
checkpoint / replay policy
trace and audit policy
```

核心原则：

```text
用户写关系。
编译器选协议。
runtime 只执行发布后的边契约。
```

## 3. 外部参考带来的设计约束

### 3.1 A2A 的可借鉴点

A2A 适合表达 agent 之间的状态化通信：发送消息、生成 task、流式返回状态和 artifact、通过 `contextId` / `taskId` 保持多轮连续性。它还明确区分 Message 和 Artifact：Message 用于通信、澄清、状态更新和继续任务；任务结果应该通过 Artifact 返回，而不是把聊天消息当成可靠结果。

对我们的启发：

- A2A 边不能只是“把上游文本塞给下游”。它应该有会话、任务、消息、artifact、状态、终止条件和错误处理。
- A2A 的最终产物必须是结构化 artifact / edge result，而不是一段聊天历史。
- A2A 边必须校验对端能力，例如是否支持 streaming、push、extended card 或指定内容类型。
- A2A 边需要 `context_id` / `task_id` 映射到我们的 `graph_run_id`、`edge_run_id`、`work_order_id` 和 trace。

### 3.2 LangGraph 的可借鉴点

LangGraph 对边的区分很清楚：普通边负责固定流转，条件边负责动态路由，多出边可在下一 superstep 并行执行；如果节点用 `Command` 动态路由，就不能同时混用静态出边，否则多个路径会一起执行，图行为会变难推理。

对我们的启发：

- 静态拓扑边和动态路由边要有明确互斥规则。
- 条件路由应该是可编译的 route contract，而不是运行时自由跳转。
- fan-out、map-reduce、并行边要生成独立 packet / branch identity，不能复用同一个隐式状态。
- checkpoint 边界应清楚，边协议执行失败后能够知道重放哪个 packet、哪个 conversation turn、哪个 artifact commit。

### 3.3 Temporal 的可借鉴点

Temporal 的核心价值是 durable execution：失败、重启或网络中断后，工作流能从确定的历史状态恢复。我们不需要复制 Temporal，但图任务边协议必须学习它的工程要求：长任务、异步交互、外部信号、人工门控、重试和恢复都必须进入事件历史，而不是留在内存里的临时状态。

对我们的启发：

- 每条边都要有可重放的 edge run history。
- A2A / human gate / event edge 不能依赖临时连接是否还活着。
- retry、timeout、ack、delivery receipt、artifact receipt、memory receipt 都必须成为边协议历史的一部分。

## 4. 核心设计判断

边应拆成四层语义，不要混在一个 `edge_type` 字符串里：

```text
EdgeRelation
业务关系：submit_for_review、revision_request、discuss_design

EdgeProtocol
交互协议：node_handoff、a2a_session、conditional_route、resource_read、resource_commit、event_signal

SchedulerSemantics
调度语义：dependency、conditional_dependency、context、commit、event、audit、none

PacketContract
数据语义：payload schema、source selector、target slot、visibility、artifact refs、memory refs
```

这四层分别回答不同问题：

- `EdgeRelation` 回答“业务上是什么关系”。
- `EdgeProtocol` 回答“怎么交互”。
- `SchedulerSemantics` 回答“是否阻塞、何时激活下游、是否进入并行/条件/提交队列”。
- `PacketContract` 回答“传什么、可见什么、写到哪里、如何审查”。

## 5. 边协议族

### 5.1 NodeHandoffContract：单向节点交接

适用场景：

- 起草到审核。
- 规划到执行。
- 上游产物进入下游处理。
- 一个节点的结构化输出成为另一个节点输入。

特点：

- 单向。
- 以 FlowPacket / artifact refs / bounded outputs 为核心。
- 默认 `scheduler_role=dependency`。
- 下游只看到 contract 允许暴露的内容。
- 需要 ack 时，ack 是“交接包被接收并符合输入合同”，不是“业务一定通过”。

建议最小 DSL：

```yaml
- draft -> review: submit_for_review
```

编译产物：

```yaml
protocol_kind: node_handoff
direction: unidirectional
scheduler_role: dependency
payload_contract_id: contract.writing.draft.review_payload
source_output_selector: outputs.chapter_text
target_input_slot: review_material
result_delivery_policy: contract_payload_and_refs
ack_required: true
timeout_policy: fail_closed
```

### 5.2 A2ASessionContract：双向 agent 通信讨论

适用场景：

- 两个 agent 需要协商方案。
- reviewer 和 writer 需要围绕返修意见讨论。
- planner 和 implementer 需要澄清任务边界。
- 多 agent 对一个争议点做有限轮次的结论生成。

不适用场景：

- 简单单向传递。
- 已经有明确 schema 的审核输入。
- 可以由一个 review gate 节点完成的裁决。
- 需要严格因果链但不需要对话的普通依赖。

关键判断：

```text
A2A 边是“会话协议”，不是普通 handoff 的高级名称。
```

A2A 边应该生成一个边拥有的 conversation session。session 负责消息往返、状态记录、turn policy、终止条件和产物归档。最终进入图状态的不是完整聊天记录，而是一个结构化 `conversation_artifact`，例如：

- `consensus_decision`
- `revision_plan`
- `open_questions`
- `resolved_constraints`
- `handoff_summary`
- `artifact_refs`

建议最小 DSL：

```yaml
- reviewer <-> writer: discuss_revision
```

可选增强：

```yaml
- participants: [reviewer, writer]
  relation: discuss_revision
  protocol: a2a_session
  agenda: 讨论审核问题并形成可执行返修计划
  max_turns: 4
  output: revision_plan
```

编译产物：

```yaml
protocol_kind: a2a_session
direction: bidirectional
scheduler_role: conditional_dependency
a2a:
  session_scope: edge_run
  context_id_strategy: graph_run_edge_id
  task_id_strategy: per_participant_or_external_agent_task
  message_types:
    - message/send
    - message/stream
  turn_policy:
    mode: bounded_round_robin
    max_turns: 4
    stop_when:
      - consensus_reached
      - unresolved_requires_human
      - max_turns_reached
  output_artifact_contract_id: contract.a2a.revision_plan
  transcript_policy:
    persist_full_trace: true
    expose_to_downstream: summary_and_refs
  failure_policy:
    timeout: escalate_or_fail_closed
    invalid_agent_response: fail_closed
```

运行时原则：

- A2A 消息历史是 trace，不是默认上下文。
- 下游拿到的是编译器定义的 A2A 输出 artifact。
- 如果参与 agent 属于不同 task environment，边只能传递 contract 允许的 packet / refs，不能直接共享文件系统或记忆空间。
- 如果对接外部 A2A agent，必须先通过 Agent Card / capability registry 校验能力和安全策略。

### 5.3 ReviewFeedbackContract：审核反馈与返修边

适用场景：

- 审核不通过后返回作者返修。
- 质量门给出问题清单和修改要求。
- 自动重试或人工转介。

特点：

- 通常是条件边，默认 `scheduler_role=conditional_dependency`。
- 需要读取上游 review verdict。
- 必须有最大返修次数或 loop frame 边界。
- 反馈必须结构化，不允许只传自然语言“改一下”。

建议 DSL：

```yaml
- review -> revise: revision_request when verdict == revise
```

编译产物：

```yaml
protocol_kind: review_feedback
scheduler_role: conditional_dependency
trigger:
  verdict: revise
payload_contract_id: contract.writing.revision_request
carry:
  - artifact_ref
  - issues
  - revision_request
loop_policy:
  max_revision_attempts: 3
failure_propagation_policy: coordinator_decides
```

### 5.4 ConditionalRouteContract：条件路由边

适用场景：

- 根据节点输出选择下一阶段。
- pass / revise / reject 分支。
- router 节点按任务类型分发。

特点：

- 边只表达路由条件，不承担业务裁决。
- 业务裁决必须由源节点或 review gate 节点产出结构化 verdict。
- route expression 必须可审查、可重放。

建议 DSL：

```yaml
- review -> publish: when verdict == pass
- review -> revise: when verdict == revise
- review -> human_gate: when verdict == reject
```

编译产物：

```yaml
protocol_kind: conditional_route
scheduler_role: conditional_dependency
route_expression:
  source: review.outputs.verdict
  operator: eq
  value: pass
static_dynamic_rule: no_mixed_unbounded_dynamic_routes
```

### 5.5 ResourceReadContract：资源读取边

适用场景：

- memory repository 向 agent 节点提供正式记忆。
- artifact repository 向节点提供上下文 artifact。
- file resource 向节点提供只读文件材料。

特点：

- 默认 `scheduler_role=context`。
- 不能把资源节点误当成调度前置任务。
- 读取结果进入节点 input package 的受控上下文区。
- 读取策略必须包含 selector、visibility、missing policy 和 limit。

建议 DSL：

```yaml
- memory.world -> draft: memory.read_required
- artifact.outline -> draft: artifact.context
```

编译产物：

```yaml
protocol_kind: resource_read
scheduler_role: context
resource_kind: memory
selector:
  collection_id: world
  status_filter: [committed]
  version_selector: latest_committed_before_stage_start
on_missing: block
visibility:
  model_visible_label: 世界观资料
  max_records: 50
```

### 5.6 ResourceCommitContract：资源写入与提交边

适用场景：

- 审核通过后提交候选记忆。
- 生成 artifact 并提交到 artifact repository。
- 写入 progress / issue ledger。

特点：

- 默认 `scheduler_role=commit`。
- 必须有 approval source 或 commit policy。
- commit receipt 是正式运行结果的一部分。
- 不允许 agent 自己把候选记忆当成已提交事实。

建议 DSL：

```yaml
- review -> memory.world: memory.commit_after_review
```

编译产物：

```yaml
protocol_kind: resource_commit
scheduler_role: commit
resource_kind: memory
approval_policy: approved_upstream_review_gate
required_verdict: pass
candidate_ref_key: memory_candidate_ref
commit_visibility_policy:
  visible_after: next_clock
receipt_required: true
```

### 5.7 EventSignalContract：事件与通知边

适用场景：

- 非阻塞通知 monitor。
- 后台任务状态更新。
- 外部 webhook / signal 唤醒图任务。

特点：

- 默认 `scheduler_role=event` 或 `none`。
- 不应作为普通数据交接。
- 必须有 idempotency key、ordering policy 和 replay strategy。

建议 DSL：

```yaml
- draft -> monitor: event.progress_update
- external.webhook -> payment_gate: event.signal
```

编译产物：

```yaml
protocol_kind: event_signal
scheduler_role: event
delivery_policy: at_least_once
idempotency_key: graph_run_id.edge_id.event_id
replay_policy: append_event_history
```

### 5.8 HumanGateContract：人工门控边

适用场景：

- 审核驳回转人工。
- 高风险工具调用前人工批准。
- 预算、权限、发布动作确认。

特点：

- human gate 不是普通 agent 节点的补丁。
- 必须可暂停、可恢复、可审计。
- 人工输入必须结构化进入 graph history。

建议 DSL：

```yaml
- review -> human_gate: when verdict == reject
```

编译产物：

```yaml
protocol_kind: human_gate
scheduler_role: conditional_dependency
interrupt_policy:
  mode: wait_for_human_decision
decision_schema:
  allowed_values: [approve, revise, reject]
resume_policy: decision_packet
```

### 5.9 BarrierJoinContract：汇合与聚合边

适用场景：

- 多个并行分支完成后汇总。
- map-reduce。
- 多 reviewer 投票。

特点：

- edge 需要表达 join policy：all_success、any_success、quorum、allow_partial_with_issues。
- 每个 branch packet 必须有 branch identity。
- 聚合逻辑应在 aggregator node 或 join contract 中明确。

建议 DSL：

```yaml
- [review_a, review_b, review_c] -> aggregate: quorum 2
```

编译产物：

```yaml
protocol_kind: barrier_join
scheduler_role: dependency
join_policy: quorum
quorum: 2
branch_packet_policy:
  identity: source_node_id + result_id
  dedupe: true
```

### 5.10 GraphModulePortContract：子图模块接口边

适用场景：

- 一个图作为另一个图的节点模块。
- 父图把输入交给子图。
- 子图把 output port 返回父图。

特点：

- 需要 port contract，而不是直接连内部节点。
- 子图内部边不能泄露成父图调度事实。
- 作用域、node id、edge id、artifact refs 需要 scope prefix。

建议 DSL：

```yaml
- outline -> module.chapter_pipeline.input
- module.chapter_pipeline.output -> publish
```

编译产物：

```yaml
protocol_kind: graph_module_port
scheduler_role: dependency
source_port_id: outline.output
target_port_id: chapter_pipeline.input
scope_policy: scoped_graph_module
```

## 8. 编译器边处理流程

建议在 076 的 `Pass 3：Synthesize Contracts` 中拆出明确的边编译子流程。

### Pass E1：Resolve Edge Relation

输入：极简 DSL edge  
输出：`ResolvedEdgeRelation`

职责：

- 解析 `draft -> review: submit_for_review`。
- 解析 `<->`、`?`、`when`、`quorum` 等短语法。
- 从模板库解析 relation。
- 根据 source / target node intent 校验关系是否合理。

### Pass E2：Infer Edge Protocol

输入：`ResolvedEdgeRelation` + node contracts  
输出：`EdgeProtocolPlan`

职责：

- 如果是 `<->`、`discussion`、`negotiate`、`clarify`，优先推导为 `a2a_session`。
- 如果是普通 `A -> B` 且有明确 payload，推导为 `node_handoff`。
- 如果带 `when` 或 verdict trigger，推导为 `conditional_route` 或 `review_feedback`。
- 如果 source / target 是 resource node，推导为 `resource_read` 或 `resource_commit`。
- 如果 target 是 human gate，推导为 `human_gate`。
- 如果 source 是多个节点或 join policy，推导为 `barrier_join`。

### Pass E3：Synthesize Packet Contract

输入：`EdgeProtocolPlan` + node output/input contracts  
输出：`EdgePacketContractDraft`

职责：

- 绑定 `source_output_selector`。
- 绑定 `target_input_slot`。
- 生成 `payload_contract_id` / `packet_contract_id`。
- 生成 context filter、artifact ref policy、memory handoff policy。
- 校验源输出和目标输入 schema 是否兼容。

### Pass E4：Synthesize Interaction Contract

输入：`EdgeProtocolPlan`  
输出：`EdgeInteractionContractDraft`

职责：

- 为 A2A 生成 participants、turn policy、message type、context id strategy、task id mapping、transcript policy、output artifact contract。
- 为 human gate 生成 interrupt/resume contract。
- 为 event edge 生成 signal schema、idempotency key 和 ordering policy。
- 为 barrier join 生成 branch identity、quorum 和 aggregation policy。

### Pass E5：Bind Scheduler Semantics

输入：前面所有 edge draft  
输出：`EdgeSchedulerPlan`

职责：

- 生成 `semantic_role`。
- 生成 `scheduler_role`。
- 生成 wait / join / ack / timeout。
- 确认是否阻塞下游。
- 确认是否进入 checkpoint boundary。

### Pass E6：Verify Edge Safety

输入：完整 edge plan  
输出：`CompiledEdgeContract` 或 compile error

必须 fail closed 的情况：

- A2A 边没有终止条件或最大轮次。
- A2A 边把 Message 当成可靠结果传给下游。
- source output 与 target input schema 不兼容。
- 条件边引用不存在的 verdict / field。
- 同一 source node 同时存在不受控静态边和动态路由边。
- 跨环境边试图直接共享文件路径或未授权 memory refs。
- resource commit 没有 approval policy。
- human gate 没有 resume schema。
- event edge 没有 idempotency key。
- barrier join 没有 branch identity。

## 9. DSL 设计建议

### 9.1 最短语法

```yaml
edges:
  - draft -> review: submit_for_review
  - review -> revise: revision_request when verdict == revise
  - review -> publish: pass_to_publish when verdict == pass
  - reviewer <-> writer: discuss_revision
```

### 9.2 结构化语法

```yaml
edges:
  - id: edge.draft.review
    from: draft
    to: review
    relation: writing.submit_for_review

  - id: edge.review.writer.discuss
    participants: [review, writer]
    relation: writing.discuss_revision
    protocol: a2a_session
    agenda: 讨论审核问题并形成可执行返修计划
    max_turns: 4
    output: revision_plan

  - id: edge.review.revise
    from: review
    to: revise
    relation: writing.revision_request
    when:
      field: verdict
      equals: revise
```

### 9.3 高风险覆盖

高风险字段允许覆盖，但必须解释来源并进入 compile report：

```yaml
edges:
  - from: reviewer
    to: writer
    relation: discuss_revision
    protocol: a2a_session
    max_turns: 6
    transcript:
      persist_full_trace: true
      expose_to_downstream: summary_only
    timeout:
      seconds: 600
      on_timeout: escalate_to_human
```

## 10. Runtime 执行模型

建议 runtime 把边协议分成三类执行器：

### 10.1 FlowPacketEdgeExecutor

执行：

- node_handoff
- review_feedback
- resource_read
- artifact_context
- file_context

职责：

- 根据 source result 生成 FlowPacket。
- 过滤 visible payload。
- 绑定 artifact / memory refs。
- 写入 inbound context。
- 生成 receipt。

### 10.2 SessionEdgeExecutor

执行：

- a2a_session
- human_gate
- event_signal

职责：

- 创建 edge session。
- 写入 session event history。
- 管理 turn / interrupt / resume / stream。
- 产出结构化 conversation artifact 或 decision packet。
- 支持断点恢复。

### 10.3 CommitEdgeExecutor

执行：

- memory_commit
- artifact_commit
- progress_commit
- issue_ledger_write

职责：

- 校验 approval policy。
- 物化 commit。
- 写入 receipt。
- 控制可见性时钟。

## 11. A2A 边的关键边界

### 11.1 A2A 边不等于两个节点互相直接调用

如果让两个 agent 在运行时随意互调，会破坏图的可审查性。A2A 边必须由 graph runtime 作为协议控制器托管：

```text
GraphRuntime
-> A2ASessionEdgeExecutor
-> participant agent turns
-> conversation artifact
-> downstream packet
```

参与 agent 可以发消息，但不能自行绕过边契约改图、开新边、扩大权限或提交资源。

### 11.2 A2A 输出必须 artifact 化

A2A 会话的聊天过程是 trace，最终交付给下游的是 artifact：

```yaml
conversation_artifact:
  verdict: revise
  agreed_revision_plan:
    - 增强冲突动机
    - 修正世界观术语
  unresolved_questions: []
  participant_positions:
    reviewer: ...
    writer: ...
  transcript_ref: trace://...
```

### 11.3 A2A 必须有停止条件

必须至少有一个硬停止条件：

- `max_turns`
- `max_duration_seconds`
- `consensus_reached`
- `decision_schema_filled`
- `human_escalation_required`
- `terminal_task_state`

没有停止条件不允许发布。

### 11.4 A2A 必须区分内部 agent 和外部 agent

内部 agent：

- 由我们的 runtime profile、环境、权限、prompt contract 约束。
- 可以使用内部 session executor。

外部 A2A agent：

- 必须通过 Agent Card / capability registry 发现能力。
- 必须校验认证、授权、内容类型、streaming / push 能力。
- 必须把外部 task / message / artifact 映射到我们的 trace 和 edge artifact。

## 12. 与节点实例构建器的关系

节点实例构建器只负责生成节点配置草案。边协议也可以有一个 `EdgeContractBuilderAgent`，但边的裁决权仍归确定性编译器。

建议链路：

```text
EdgeSeed
-> EdgeContractBuilderAgent
-> EdgeContractDraft
-> GraphCompiler validation
-> CompiledEdgeContract
```

`EdgeContractBuilderAgent` 可以建议：

- 这条边应使用什么 protocol kind。
- 应传哪些 output key。
- target input slot 应是什么。
- A2A agenda 和最大轮次。
- failure / timeout 建议。
- compile rationale。

它不能：

- 发布 graph config。
- 自动创建 registry 模板。
- 扩大 agent 权限。
- 绕过环境隔离。
- 把 A2A transcript 暴露给所有下游。
- 自行决定 resource commit。

## 13. Compile Report 必须解释边

每条边的报告至少包含：

```yaml
edge_id: edge.review.writer.discuss
relation: writing.discuss_revision
relation_source: user_dsl
protocol_kind: a2a_session
protocol_source: inferred_from_bidirectional_relation
scheduler_role: conditional_dependency
payload_contract: contract.a2a.revision_plan
source_visible_outputs:
  - review.verdict
  - review.issues
target_input_slots:
  - writer.revision_context
a2a:
  participants: [reviewer, writer]
  max_turns: 4
  stop_conditions: [consensus_reached, max_turns_reached]
security:
  cross_environment_policy: packet_only
failure_policy:
  timeout: escalate_or_fail_closed
explanation: 双向讨论关系需要状态化会话，但下游只接收 revision_plan artifact。
```

自动化越强，边报告越重要。用户要能审查：

- 为什么用了 A2A 而不是 handoff。
- 传给下游的具体内容是什么。
- 哪些内容被隐藏。
- 边是否阻塞调度。
- 出错会影响哪些节点。
- 断点恢复时从哪里继续。

## 14. 第一版实施边界建议

第一版不要一次性实现完整外部 A2A 网络协议。建议先做内部协议族：

1. 标准化 `CompiledEdgeContract`。
2. 把现有 edge fields 投影成明确的 relation / protocol / scheduler / packet / reliability / trace。
3. 新增 `EdgeRelationTemplate` 与 `EdgeProtocolTemplate` 的最小 registry。
4. 强化 `node_handoff`、`review_feedback`、`resource_read`、`resource_commit`。
5. 实现内部 `a2a_session` 的编译产物和 session history，不急着接外部 Agent Card。
6. Compile report 解释每条边。
7. Runtime 先消费编译产物，不再根据 `edge_type` 零散猜测。

外部 A2A adapter 可以作为第二阶段：

- Agent Card discovery / cache。
- capability validation。
- `contextId` / `taskId` 映射。
- streaming / polling / push 接入。
- 外部 artifact 映射成内部 artifact ref。

## 15. 验收标准

正式落地后至少要证明：

- 用户写最短边语法，编译器能生成完整 `CompiledEdgeContract`。
- `draft -> review` 编译为 `node_handoff`，下游拿到受控 FlowPacket。
- `reviewer <-> writer` 编译为 `a2a_session`，有 max turns、session history 和结构化 conversation artifact。
- A2A transcript 不会默认污染下游上下文。
- 条件边引用不存在字段时 fail closed。
- 跨环境边只能传 packet / refs，不能直接共享未授权路径。
- resource read 不阻塞为普通 dependency，resource commit 必须有 approval policy。
- 同一 source node 不能混用不受控静态边和动态路由边。
- edge compile report 能解释协议选择、payload 选择、权限、失败策略和恢复策略。
- checkpoint / resume 后，边 packet、A2A turn、commit receipt 不丢失、不重复提交。

## 16. 暂定结论

边契约应该成为图任务系统的第二个核心抽象，仅次于节点契约。节点决定“谁做什么”，边决定“他们如何交互、传递什么、何时继续、失败怎么传播、审计看什么、恢复从哪里接上”。

成熟方向不是把所有边升级成 A2A，而是建立边协议族：

```text
node_handoff
a2a_session
review_feedback
conditional_route
resource_read
resource_commit
event_signal
human_gate
barrier_join
graph_module_port
```

A2A 是其中最强的交互协议，专门用于需要多轮通信和协商的 agent 关系；普通单向传递继续用 NodeHandoffContract。这样既能支持复杂多 agent 协作，又不会把简单图任务拖进过度对话化和不可审查的运行状态。

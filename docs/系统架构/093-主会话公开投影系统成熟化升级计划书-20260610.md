# 主会话公开投影系统成熟化升级计划书

日期：2026-06-10

状态：计划书，待审阅后实施

## 1. 目的

本计划用于把主会话公开投影从当前“可用的过渡态”升级为成熟稳定的单一权威系统。

当前已经完成一轮关键收口：

- single-agent task history hydrate 在存在 `task_projection` 时不再同时携带完整旧 `public_timeline`。
- `ChatMessage` 不再把稳定模型正文搬进 timeline。
- `PublicTimelineActivity` 不再把 `kind=observation/action` 强行改成工具窗口，而是尊重后端 `display_surface`。
- 停止、暂停等生命周期投影会压掉内部工具调用 detail。

这些修复解决了最明显的混合显示问题，但仍不足以称为成熟系统。成熟投影系统必须满足：

```text
Raw Runtime / Chat / Monitor Event
-> PublicProjectionProjector
-> PublicProjectionEnvelope
-> MessagePublicProjection reducer
-> Pure React render
```

任何用户可见正文、工具窗口、任务进展、系统状态都必须从同一个投影合同产生。前端只能渲染、折叠、去重和做安全清洗，不能重新判断语义归属。

## 2. 当前设计缺口

### 2.1 Envelope 仍是包装器，不是唯一 projector

代码事实：

- `backend/api/chat.py::_project_public_stream_event()` 先调用 `project_public_timeline_delta()` 生成旧 `public_timeline_delta`。
- `backend/api/chat.py::_attach_public_projection_envelope()` 再把 `public_timeline_delta` 包进 envelope。
- `backend/harness/runtime/public_projection_envelope.py` 当前从已有 items 推导 source/surface/slot。

问题：

`PublicProjectionEnvelope` 目前还没有拥有最终投影权威。它依赖旧 delta 先做一次判断，再做包装和过滤。这样 live SSE、runtime monitor、session hydrate 仍可能在不同路径中出现不同语义。

目标：

`PublicProjectionProjector` 直接接收 raw event，输出 envelope。`public_timeline_stream.py` 降级为内部 item builder，不再决定最终 `surface`、`slot`、`source_authority`。

### 2.2 前端 raw event fallback 仍能绕过 envelope

代码事实：

- `frontend/src/lib/store/events.ts` 仍导入 `projectRuntimeStreamEvent()`。
- `stageStatusForEvent()`、`activityDetailForEvent()`、`assistantDoneContentFromEvent()` 仍会从 raw event 生成状态、receipt 或正文。
- `reduceStreamEvent()` 在 envelope 缺失时继续使用 legacy `public_timeline_delta` 和 `task_projection_delta`。

问题：

只要某个后端事件没有 envelope，前端就会退回旧逻辑。这个 fallback 对短期兼容有帮助，但长期会让系统继续“修好一条入口，另一条入口又漏出来”。

目标：

切到 envelope-only 后，runtime/control/system 类事件缺 envelope 必须 fail closed：只记录诊断，不生成 assistant 正文、工具窗口或用户可见状态。

### 2.3 Legacy timeline 仍按 kind 反推 body/tool/status

代码事实：

- `frontend/src/lib/store/publicTimeline.ts` 的 `BODY_TIMELINE_KINDS` 仍把 `observation_report`、`stage_summary`、`final_summary` 等 kind 当作 body 候选。
- `frontend/src/components/chat/PublicTimelineActivity.tsx::activityLineKind()` 仍对 legacy item 用 `kind/surface` 推导 body/tool/status。

问题：

即使 task projection 已经尊重 `display_surface`，旧 turn-run 或 monitor public timeline 仍可能靠 kind 被前端重新分类。成熟系统中，slot 必须是后端投影合同的一部分，不能由前端猜。

目标：

所有进入主聊天区的公开 item 必须带 `slot`。没有 `slot` 的 legacy item 只能由明确的 legacy adapter 转换，且 adapter 有删除条件。

### 2.4 ChatPanel 仍通过 timeline body 猜测状态栏压制

代码事实：

- `ChatPanel.shouldSuppressSessionActivityBar()` 合并 public timeline 后判断是否隐藏底部活动条。
- 它调用 `hasDisplayablePublicTimelineBody()` 来判断消息级反馈是否已经展示。

问题：

这让 footer 状态栏继续受 public timeline body 推断影响，而不是受统一 `MessagePublicProjection` 控制。后续可能出现正文正常、活动条异常隐藏或重复显示。

目标：

`ChatPanel` 只读取 `MessagePublicProjection.sessionActivity` 或明确 `status/control` slot，不再从 timeline body 判断状态栏。

### 2.5 工具观察语义存在丢失风险

代码事实：

- 当前 `session_task_projection.py` 已过滤低信号和内部 activity。
- 但部分有意义的工具失败、观察判断、下一步动作仍可能因为 work unit kind/state 被压到 diagnostics。
- 移除旧 `public_timeline` 后，如果 task projection 没有稳定 tool observation slot，用户会看到 current action，却看不到关键失败观察。

问题：

成熟系统不能靠旧 public timeline 补齐 task projection 的语义缺口。工具观察必须进入统一 projection 合同，并同时保留给模型执行循环。

目标：

失败、超时、产物、验证结果等工具观察必须形成明确 `tool` 或 `timeline` slot。隐藏 UI 不得影响模型可见 observation 记录。

### 3.2 本地设计原则

本计划继承以下本地设计约束：

- `docs/系统架构/090-统一公开投影协议与投屏时序设计-20260610.md`
  - 三类输出不可混权：模型正文、工具观察、系统状态必须分开。
  - 前端只渲染投影，不重新裁决。
  - handoff done 必须 `terminal.visible=false`，不产生可见文案。
- `docs/系统架构/082-流式输出权威链路升级技术书-20260609.md`
  - ChatRun 是传输和重放真相，不拥有语义完成权威。
  - OutputBoundary 是最终 assistant 正文真相。
  - 正文流和进度流必须分离。
- `docs/系统框架/077-TaskRun状态权威收口与监控投影修复方案-20260609.md`
  - `TaskRunStateView` 是 TaskRun 状态和控制能力唯一解释器。
  - Session projection、monitor、前端都应消费 canonical view。
- `docs/系统架构/092-主会话公开投影单一权威重构计划书-20260610.md`
  - 已明确目标链路：raw event -> envelope -> reducer -> pure render。
  - 已列出 `public_timeline + task_projection` 混合 attachment 的删除方向。

### 3.3 参考架构原则

采用成熟 agent 系统的单向权威链原则：

```text
Observe raw facts
-> Normalize public facts
-> Decide projection authority
-> Assemble stable envelope
-> Reduce canonical UI state
-> Present without re-deciding
```

借用的机制：

- 事件、状态和输出分层。
- 输出边界由后端统一确定。
- 前端缺少权威字段时 fail closed，而不是自行补语义。
- 工具观察保留给 runtime/model，公开 UI 只消费投影后的摘要。

拒绝的机制：

- 不让前端继续通过 raw event name、tool name、answer_channel、terminal_reason 自行生成用户语义。
- 不用兼容字段长期驱动 UI。
- 不靠新增过滤词解决结构性混权。

## 4. 推荐目标设计

### 4.1 唯一权威链

```text
RuntimeEvent / ChatEvent / MonitorEvent / SessionHydrateRecord
-> PublicProjectionProjector
-> PublicProjectionEnvelope
-> MessagePublicProjectionReducer
-> ChatMessage / PublicTimelineActivity / SessionActivityBar
```

### 4.2 层级职责

| 层 | 允许做什么 | 禁止做什么 |
| --- | --- | --- |
| Raw event | 记录事实、refs、payload、状态 | 直接决定用户可见正文 |
| PublicProjectionProjector | 决定 source/surface/slot/terminal/anchor | 依赖前端兜底修语义 |
| TaskProjectionAdapter | 将 TaskRun state view 和 progress presentation 转成 task projection | 重新解释控制能力，泄漏内部工具 |
| MessagePublicProjectionReducer | 合并 envelope，去重，维护 body/timeline/tool/status/control | 从 raw event name 生成语义 |
| React render | 布局、折叠、Markdown、复制、视觉状态 | 改变 source/surface/slot |

### 4.3 MessagePublicProjection

前端应收敛到一个稳定结构：

```ts
type MessagePublicProjection = {
  body: {
    text: string;
    source: "model";
    canonicalState?: string;
  };
  timeline: PublicProjectionItem[];
  toolWindows: PublicProjectionItem[];
  taskProjection?: SingleAgentTaskProjection;
  controls: PublicProjectionItem[];
  status: PublicProjectionItem[];
  terminal?: {
    event: "done" | "error" | "stopped";
    visible: boolean;
    reason?: string;
  };
  tone: "running" | "waiting" | "done" | "error" | "stopped";
};
```

说明：

- `body.text` 只来自 `source_authority=model` 且 `slot=body`。
- `task_projection.final_answer` 只作为 TaskRun projection 的收口材料；若与 canonical body 相同，reducer 去重。
- `toolWindows` 只来自 `slot=tool`。
- `status` 和 `controls` 不允许变成 assistant body。

### 4.4 PublicProjectionEnvelope 合同

后端 envelope 必须稳定包含：

```ts
type PublicProjectionEnvelope = {
  authority: "harness.public_projection.v1";
  projection_id: string;
  sequence: number;
  session_id: string;
  created_at: number;
  anchor: {
    turn_id?: string;
    message_id?: string;
    task_run_id?: string;
    run_id?: string;
    turn_run_id?: string;
    anchor_role: "assistant" | "status" | "monitor";
  };
  source_authority: "model" | "tool" | "system" | "runtime";
  surface: "assistant_body" | "timeline" | "tool_window" | "status_bar" | "task_projection" | "control" | "diagnostics";
  lifecycle: "running" | "waiting" | "done" | "error" | "stopped";
  terminal?: {
    event: "done" | "error" | "stopped";
    visible: boolean;
    reason?: string;
  };
  items: PublicProjectionItem[];
  task_projection?: SingleAgentTaskProjection;
  active_turn_update?: {
    turn_id?: string;
    task_run_id?: string;
    state?: string;
  };
};
```

每个 `PublicProjectionItem` 必须含有：

```ts
{
  item_id: string;
  slot: "body" | "timeline" | "tool" | "status" | "task" | "control" | "diagnostics";
  source_authority: "model" | "tool" | "system" | "runtime";
  surface: string;
  kind: string;
}
```

缺少 `slot/source_authority/surface` 的 item 不进入主聊天渲染，只能进入 legacy adapter 或 diagnostics。

## 5. 固定执行流

### 5.1 普通模型回答

```text
OutputBoundary final answer
-> PublicProjectionProjector(source=model, surface=assistant_body, slot=body)
-> MessagePublicProjection.body
-> ChatMessage body
```

禁止：

- `done.summary` 伪造成正文。
- `receipt_summary` 作为正文 fallback。
- frontend 从 `answer_channel` 决定正文是否生成。

### 5.2 工具调用与观察

```text
ModelAction(tool_call)
-> Tool execution
-> ToolObservation
-> PublicProjectionProjector(source=tool/runtime, slot=tool or timeline)
-> MessagePublicProjection.toolWindows / timeline
-> PublicTimelineActivity render
```

禁止：

- 工具 stdout、JSON、路径列表进入 assistant body。
- UI 隐藏某条观察导致 runtime/model 观察也丢失。
- 前端按 `kind=work_action` 自行决定 tool window。

### 5.3 后台任务 handoff

```text
request_task_run
-> task_run created
-> task_projection generated
-> done(terminal_reason=task_executor_scheduled)
-> envelope(terminal.visible=false, surface=task_projection, active_turn_update)
-> stream closes
-> monitor envelope continues updating same task attachment
```

禁止：

- handoff done 生成 assistant body。
- handoff done 生成“后台任务已接管”类可见正文。
- session refresh 恢复旧 public timeline。

### 5.4 Session hydrate

```text
Persisted task run / turn run
-> session_timeline
-> task_projection or persisted envelope projection
-> runtime attachment
-> MessagePublicProjectionReducer
```

规则：

- single-agent task attachment 有 `task_projection` 时，`public_timeline` 只能为空或缺省。
- turn-run legacy attachment 允许短期保留 `public_timeline`，但必须带 legacy 标记并有删除条件。

## 6. 阶段计划

### Phase 0：合同冻结与测试前置

目标：先锁住成熟投影合同，避免后续实现继续漂移。

影响文件：

- `frontend/src/lib/api.ts`
- `backend/harness/runtime/public_projection_envelope.py`
- `backend/tests/public_projection_envelope_test.py`
- `frontend/src/lib/store/publicProjectionReducer.test.ts`

动作：

1. 明确定义 `PublicProjectionEnvelope`、`PublicProjectionItem`、`MessagePublicProjection`。
2. 后端补合同测试：
   - model body 只能进入 `slot=body`。
   - tool observation 只能进入 `slot=tool/timeline`。
   - system/runtime/control 不能进入 body。
   - handoff done 必须 `terminal.visible=false`。
   - 缺 anchor 的 event 不静默投主消息。
3. 前端补 reducer 测试：
   - envelope -> body/timeline/tool/status/control。
   - envelope 缺 slot 时 fail closed。
   - stable body 与 task final answer 去重。

完成条件：

- 没有实现生产切换前，合同测试已经能明确描述目标结构。
- 所有新合同测试先红后绿，不通过放宽断言制造绿。

禁止：

- 继续新增前端过滤词。
- 继续让 legacy `kind` 成为正式 slot。

### Phase 1：后端 PublicProjectionProjector 单一化

目标：chat stream 和 runtime monitor 统一调用同一个 projector。

影响文件：

- `backend/harness/runtime/public_projection_projector.py` 新增
- `backend/harness/runtime/public_projection_envelope.py`
- `backend/harness/runtime/public_timeline_stream.py`
- `backend/harness/runtime/runtime_monitor_public_projection.py`
- `backend/api/chat.py`

动作：

1. 新增 `PublicProjectionProjector.project_event(event, context)`。
2. 把 `_project_public_stream_event()` 从“生成 data + public_timeline_delta”改为“生成 raw public fact + projector envelope”。
3. `runtime_monitor_public_projection.py` 改为调用同一 projector。
4. `public_timeline_stream.py` 改名或降级为 `public_projection_items.py`，只提供内部 item builder。
5. 所有 envelope item 由 projector 赋予 `slot/source_authority/surface`。

完成条件：

- chat SSE 和 monitor SSE 对同一 runtime event 产生同构 envelope。
- `api/chat.py` 不再直接向 UI 决定 `public_timeline_delta`。
- legacy fields 只允许 shadow 输出，不驱动 UI。

回滚条件：

- 若 projector 生成 envelope 丢失 anchor 或 terminal，回滚到 Phase 0 合同，不恢复前端 raw event 推断。

### Phase 2：TaskProjection 工具观察补齐

目标：移除旧 `public_timeline` 后，task projection 仍能稳定展示关键工具动作和观察。

影响文件：

- `backend/harness/runtime/session_task_projection.py`
- `backend/harness/runtime/progress_presenter.py`
- `backend/harness/runtime/public_projection_filters.py`
- `backend/tests/session_task_projection_test.py`

动作：

1. 为 tool observation 生成明确 activity：
   - success -> `display_surface=tool_window/timeline`, `visibility_level=primary/secondary`
   - error/timeout -> `display_surface=tool_window`, `state=failed`
   - debug read/search/path -> diagnostics
2. `stage` work unit 不再吞掉 primary tool observation。
3. `current_action` 不承载完整工具失败语义，只描述当前动作。
4. `activities` 承载关键动作和观察。
5. 增加测试：
   - image timeout 显示为 tool/timeline activity。
   - write/edit artifact 显示为 tool window。
   - read/search 低信号默认不进聊天。
   - agent_todo/subagent 内部工具不进聊天。

完成条件：

- 删除 single-agent task legacy public timeline 后，工具失败、产物、验证结果仍可投屏。
- UI 隐藏不影响 runtime observation 保存和模型后续输入。

## 8. 迁移、切换和回滚规则

### 8.1 Shadow 规则

允许短期同时输出：

- `public_projection_envelope`
- legacy `public_timeline_delta`
- legacy `task_projection_delta`

但 UI 主链路只能使用 envelope。legacy fields 只用于测试对比和旧日志 adapter。

### 8.2 Cutover 条件

满足以下条件后切到 envelope-only：

1. chat stream、runtime monitor、session hydrate 都能产出 envelope 或 task projection。
2. `MessagePublicProjection` 覆盖正文、工具、状态、控制、任务投影。
3. 固定端口 3000/8003 实测通过。
4. `rg` 确认旧权威没有生产导入。
5. 所有缺 envelope 的 runtime/control/system event 都 fail closed。

### 8.3 回滚规则

允许回滚到上一阶段的 envelope adapter，但不允许重新启用多个 UI 决策源。

触发回滚：

- envelope anchor 丢失导致消息绑定错位。
- terminal.visible 误判导致普通回复不显示或 handoff 变正文。
- 工具观察大面积丢失。
- session refresh 覆盖 live projection。

回滚后必须补合同测试，再继续实施。

## 13. 边界、时序、提示词与流式稳定细化

本章把“用户真正看到什么、什么时候看到、谁决定、如何收口”写成硬规则。后续实现必须以这些边界为准，不允许再用某个组件、某个事件名或某句过滤词单独补洞。

### 13.1 总体权威链

成熟主会话投影必须同时收口四条链：

```text
用户输入事实
-> TurnInputFacts / ActiveWorkContext
-> 模型动作裁决或确定性系统控制
-> Runtime / Tool / TaskRun 事实事件
-> PublicProjectionProjector
-> PublicProjectionEnvelope
-> MessagePublicProjectionReducer
-> React slot renderer
```

职责边界：

| 层 | 权威 | 禁止 |
| --- | --- | --- |
| `TurnInputFacts` | 记录用户输入、expected active turn、当前工作候选 | 根据关键词直接决定 steer、pause、stop |
| 模型动作裁决 | 决定 respond、ask_user、tool_call、request_task_run、active_work_control、block | 同一轮混合多个互斥控制动作 |
| Runtime control | 执行 pause、stop、resume、replan，记录真实 observation | 把系统控制结果写成 assistant 正文 |
| Tool executor | 执行工具并产生 observation | 决定工具观察是否是正文 |
| `PublicProjectionProjector` | 决定 `source_authority/surface/slot/lifecycle/terminal` | 依赖前端按 kind 或关键词补判 |
| Reducer | 幂等合并 envelope，维护 `MessagePublicProjection` | 从 raw event 推导用户语义 |
| React | 渲染 body/tool/status/task/control 槽位 | 改变槽位含义或生成新文案冒充模型 |

关键原则：

1. 主聊天正文只接受 `source_authority=model` 且 `slot=body`。
2. 工具窗口、任务进度、系统状态、控制结果都不能升级为正文。
3. `done/stopped/error` 只收口传输和生命周期，不产出正文。
4. 对缺 anchor、缺 slot、缺 source authority 的事件 fail closed：进入 diagnostics 或 monitor-only，不进入主聊天消息。

### 13.2 开局显示规则

用户发送消息后，前端可以立即创建 assistant 投影壳，但壳本身不是正文。

开局时序：

```text
user message committed
-> assistant projection shell created
-> input_commit_gate / runtime_assembly_compiled 只进 diagnostics 或低调 status
-> 等待第一个 public envelope
-> 根据 envelope slot 决定显示正文、工具窗口、任务投影或状态
```

显示规则：

| 场景 | 第一屏显示 | 不允许 |
| --- | --- | --- |
| 普通模型思考 | 轻量 running 状态或空 assistant 壳 | 生成“我正在思考/我会处理”正文 |
| 模型开始正文流 | 显示 assistant body streaming | 同时显示 task/timeline 噪声 |
| 模型先调用工具 | 显示 tool window running | 用正文描述“我正在调用某工具” |
| request_task_run | 显示 task projection skeleton | 显示“后台任务已接管”正文 |
| active work steer fast path | 显示 status/control receipt | 显示 assistant body |
| stale expected active turn | 显示 blocked control/status | 把刷新提示写成模型回答 |

assistant 壳的生命周期：

1. `starting`：只有消息容器和轻量运行态，不显示正文。
2. `body_streaming`：收到 `assistant_text_delta` 或 `slot=body` 后显示正文。
3. `tool_running`：收到 tool running item 后显示工具窗口。
4. `task_handoff`：收到 task projection 后绑定到同一个 assistant message。
5. `waiting_user`：模型 `ask_user` 进入 control/body 受控槽位。
6. `terminal`：收到 terminal 后清理 running 状态，按 terminal.visible 决定是否显示收口 item。

为了避免闪烁，前端可以设置最小可见延迟：

- 150ms 内没有 public envelope 时，不显示“正在准备”文字，只保留输入后的运行态。
- 超过 150ms 仍未收到正文、工具或任务投影，可以显示 `status_bar` 的中性状态。
- 一旦收到 body/tool/task/control，开局状态必须被对应槽位替换，不得并存成两套提示。

### 13.3 工具调用显示规则

工具调用显示由后端确定性投影决定，不由模型提示词或前端关键词决定。

工具时序：

```text
model_action(tool_call)
-> tool_call_admitted / tool_invocation_started
-> envelope(slot=tool, state=running)
-> tool_observation_recorded
-> envelope(slot=tool or timeline, state=done/error)
-> observation enters runtime/model context
-> model continues or finalizes
```

工具窗口状态：

| 状态 | 显示 | 数据来源 |
| --- | --- | --- |
| `queued` | 等待执行 | tool admission / scheduler |
| `running` | 正在执行公开动作 | invocation started |
| `done` | 结果摘要、产物引用、关键观察 | tool observation projection |
| `error` | 用户需要知道的失败、超时、拒绝 | tool observation projection |
| `diagnostics` | 默认不进聊天，只进监控 | low-signal/internal observation |

工具可见性分级：

| 类别 | 默认槽位 | 说明 |
| --- | --- | --- |
| 写文件、生成产物、运行测试、浏览器验证 | `tool` 或 `timeline` | 用户需要看到 |
| 工具失败、超时、权限拒绝 | `tool` | 必须可见，但不变成正文 |
| 读取文件、搜索、路径检查 | `diagnostics` 或折叠 timeline | 只有与结果判断强相关时升为 secondary |
| todo、subagent 生命周期、内部恢复 | `diagnostics` | 不进入主聊天 |
| 大 stdout、JSON、traceback | `tool` 摘要 + artifact/trace refs | 不直接塞正文，不整段灌 UI |

工具窗口内容要求：

1. title 使用工具 manifest 或 projector 的公开名称，避免泄漏内部函数名。
2. detail 是面向用户的结果摘要，不是 raw JSON。
3. artifact refs、file refs、trace refs 结构化保存。
4. stdout/stderr 只保留安全摘要；完整内容进入 trace 或 artifact，不直接进入 assistant body。
5. 同一 tool invocation 的 running/done/error 用同一个 `item_id` 幂等覆盖，不能出现两条重复窗口。

### 13.4 工具与正文的交错时序

普通工具循环的屏幕顺序应是：

```text
用户消息
-> assistant shell
-> tool window running
-> tool window done/error
-> assistant body delta/final
-> terminal done
```

如果模型在工具前已经输出了公开正文：

```text
assistant body partial
-> tool window
-> assistant body continues
-> assistant_text_final reconciles body
```

约束：

1. 工具观察不能插入正文字符串中间。
2. 正文流和工具窗口是不同槽位，可以视觉上相邻，但不能共享数据源。
3. `assistant_text_final` 到达后，正文以 final 为 canonical；工具窗口保留为 timeline/tool，不与 final 合并。
4. 如果工具失败导致模型最终 block 或 ask_user，失败观察留在 tool slot，模型解释进入 body/control。

### 13.5 后台任务开启与 handoff 流畅度

任务开启必须让用户感觉“当前工作被接住了”，但不能伪造 assistant 正文。

推荐时序：

```text
model action: request_task_run
-> task_run created
-> initial task_projection generated:
   goal/title/status/current_action/todo skeleton
-> envelope(surface=task_projection, lifecycle=running)
-> active_turn_update binds task_run to original assistant message
-> done(terminal_reason=task_executor_scheduled, terminal.visible=false)
-> chat stream closes
-> runtime monitor continues same anchor
```

UI 显示：

1. 首次 task projection 应在 `done(task_executor_scheduled)` 前到达。
2. assistant message 上显示任务卡片或任务投影，不显示“后台任务已接管”正文。
3. 当前 chat stream 关闭时，任务卡片继续保持 running，由 monitor envelope 更新。
4. session refresh 后必须恢复同一 task projection anchor，不重新生成一条 assistant 正文。

任务 projection 初始字段最低要求：

```ts
{
  task_run_id: string;
  status: "running" | "waiting_executor";
  goal: string;
  current_action: {
    title: string;
    detail?: string;
    display_surface: "timeline";
    visibility_level: "secondary";
  };
  activities: [];
  artifact_refs: [];
}
```

开局流畅度的底线：

- 如果 task_run 创建成功但 executor 尚未启动，显示“准备执行”类 `status/task`，不显示正文。
- 如果 task_run 创建失败，模型或系统必须产出 `block/error`，但错误说明仍按 envelope 进入 control/status/body 中正确槽位。
- 如果 monitor 首帧延迟，前端保持 initial task_projection，不回退到旧 public timeline。

### 13.6 暂停、停止与恢复边界

暂停不应等模型完整输出完才反馈，也不能粗暴破坏当前工具或 transcript。成熟策略是 cooperative interrupt。

暂停语义分两层：

| 层 | 行为 |
| --- | --- |
| 请求层 | 用户点击/输入暂停后，立即记录 `pause_requested`，投影为 `status/control`：正在暂停 |
| 执行层 | 执行器取消可取消的模型流或工具；不可取消工具等安全边界到达后落为 `paused` |

暂停时序：

```text
user pause
-> request_task_run_pause
-> task_run control_state=pause_requested
-> envelope(slot=status/control, state=waiting, text=正在暂停)
-> cancel model task if cancellable
-> cancel tool invocation if tool runtime supports caller cancellation
-> safe boundary reached
-> task_run_paused
-> envelope(task_projection.status=paused, can_resume=true)
```

停止时序：

```text
user stop
-> stop_task_run
-> control_state=stop_requested or stopped
-> envelope(slot=status/control, state=stopped or waiting)
-> cancel model/tool when possible
-> terminal stopped/user_aborted
-> task projection terminal
```

恢复时序：

```text
user resume
-> resume_paused_task_run
-> status=waiting_executor / resume_requested
-> runtime packet rebuilt from TaskRunStateView + pending steers
-> task executor continues
-> monitor envelope updates same anchor
```

硬规则：

1. pause 请求到达后 UI 立即显示“正在暂停”，但只有 `task_run_paused` 后才显示“已暂停”。
2. 如果当前模型正在流式输出且可取消，不等待自然语言输出完；已收到的正文 frame 保留为 partial canonical，terminal 标记 stopped/paused，不补写 final。
3. 如果工具不可中断，不显示“已暂停”，只显示“正在暂停，等待当前工具返回到安全边界”。
4. stop 比 pause 更强，可以取消模型/工具；但仍要保留 observation：用户停止、取消结果、已完成产物、失败原因。
5. pause/stop 的系统 receipt 不进入 assistant body。
6. resume 后不重放已完成工具，不重新消费 consumed steer，不复活已 terminal 的 task。

### 13.7 Steer 实现边界

steer 是用户对当前运行中工作的补充、纠偏、优先级变化或进展询问，不是普通新消息，也不是入口层关键词判断。

输入边界：

```text
frontend sends:
  expected_active_turn_id
  active_turn_input_policy: "steer" | "auto"
  editor_context
```

两种路径：

| 路径 | 使用条件 | 行为 |
| --- | --- | --- |
| explicit steer fast path | `active_turn_input_policy="steer"` 且 expected active turn 匹配、任务 running | 只记录 `ActiveTaskSteer`，必要时 `replan_requested`，返回 status/control envelope |
| model-decided active_work_control | `auto` 或语义不确定 | 入口只提供 `TurnInputFacts` 和 `ActiveWorkContext`，模型决定是否输出 `active_work_control` |

explicit steer fast path 的限制：

1. 可以低延迟记录用户补充。
2. 可以触发 executor replan signal。
3. 不能生成 assistant body。
4. 不能回答用户提出的复杂问题。
5. stale expected active turn 必须 blocked + status/control，不准猜测接入哪个任务。

steer lifecycle：

```text
user submission recorded
-> active_task_steer_recorded(pending)
-> if executor running: task_run_replan_requested
-> next runtime packet includes pending steer
-> active_task_steer_included
-> model action consumes or rejects steer
-> active_task_steer_consumed/rejected/superseded
```

steer 分类：

| 用户输入 | steer_kind | priority | 处理 |
| --- | --- | --- | --- |
| “按这个方向改” | instruction | high | pending + replan |
| “刚才方向错了” | correction | blocking | interrupt/replan |
| “先别做 X” | priority_change / correction | blocking | replan，必要时 pause |
| “现在做到哪了” | status_question | normal | 模型或 status reply，不必 replan |
| 独立新需求 | none | none | 不接入当前任务 |

### 13.8 active_work_control 提示词边界

提示词必须让 agent 做语义裁决，而不是把开发说明塞给它。投影系统本身不应依赖 LLM prompt 分类 UI 槽位；prompt 只负责让模型正确选择动作。

主会话动作选择提示词建议：

```text
你是主会话中的执行型 agent。
你需要根据用户最新输入、当前工作状态、可用工具和运行边界，选择本轮唯一正确动作。

你可以选择：
- respond：直接回答用户。
- ask_user：需要用户补充关键信息。
- tool_call：需要在当前轮使用工具获得事实或执行动作。
- request_task_run：需要启动可持续执行、可追踪、可恢复的后台任务。
- active_work_control：用户明确在控制当前正在运行或可恢复的工作。
- block：当前请求不能安全推进或缺少必要条件。

你必须先判断用户输入和当前工作的关系：
- current_work：用户在继续、暂停、停止、补充、纠正或询问当前工作。
- independent_turn：用户提出了新的独立请求或普通聊天。
- ambiguous：关系不明确。

只有 relation_to_current_work 为 current_work 时，才允许 active_work_control。
不要因为系统提供了 active_work_context，就自动把独立请求接到旧任务上。
不要同时输出 respond、request_task_run、tool_call、ask_user 或 active_work_control。
系统控制结果只是观察，不是你的最终表达；如果需要向用户解释，必须基于观察事实再输出。
```

active_work_control 动作提示词建议：

```text
当用户明确指向当前工作时，你可以输出 active_work_control。

可用控制动作：
- continue_active_work：继续当前已暂停或可恢复工作。
- pause_active_work：请求暂停当前工作。
- stop_active_work：请求停止当前工作。
- append_instruction_to_active_work：把用户补充要求加入当前工作。
- answer_about_active_work：回答用户关于当前工作状态的问题。
- answer_then_continue_active_work：先回答用户问题，再继续当前工作。

输出 active_work_control 时，必须包含：
- action
- relation_to_current_work
- evidence：你为什么认为用户在指向当前工作
- user_turn_kind：question、command、complaint、mixed 或 statement
- answer_obligation：direct_answer_required、acknowledgement_only 或 none
- continuation_strategy：same_run_resume、already_running、defer 或 none

如果用户输入是独立新任务，选择 respond、tool_call 或 request_task_run，不要控制当前工作。
如果关系不明确，优先 ask_user 或普通回答，不要擅自 steer。
```

任务执行器消费 steer 的提示词建议：

```text
你正在继续一个已有任务。
系统会提供当前任务目标、已完成事实、待办状态、最新工具观察，以及用户在运行中插入的 pending steering。

你必须逐条检查 pending steering：
- 如果 steering 改变目标、优先级、禁止项或验收标准，先调整计划再继续。
- 如果 steering 与当前任务冲突，说明冲突并请求用户裁决，或按更高优先级阻塞。
- 如果 steering 已被你纳入本次动作，标记为 consumed。
- 如果 steering 已被更新要求覆盖，标记为 superseded。
- 不要把旧 steering 当成长期偏好；只在当前任务上下文内使用。

你不能伪造工具结果或完成状态。
你不能为了继续任务而忽略用户刚刚插入的纠偏。
```

工具使用提示词建议：

```text
如果需要事实、文件、命令、浏览器或外部能力，直接使用工具。
不要把“我正在调用某工具”写成正文来替代工具调用。
工具结果会作为 observation 回到你；你需要基于 observation 再决定下一步。
工具失败、权限拒绝、超时或输出不足都是事实，不要假设成功。
最终回复只总结对用户有价值的结论、产物、风险和下一步，不要倾倒工具 stdout、JSON 或内部 trace。
```

### 13.9 control / status / body 收口规则

每次 turn 必须在这几类终点之一收口：

| 终点 | body | status/control | terminal |
| --- | --- | --- | --- |
| 普通回答完成 | `assistant_text_final` 或 `slot=body` | 清理 running | `done.visible=true` |
| 工具后回答完成 | final body | tool windows 保留 | `done.visible=true` |
| request_task_run handoff | 无 body | task projection running | `done.visible=false` |
| explicit steer accepted | 无 body | “补充已接入” control/status | `done.visible=false` 或 control-only |
| pause requested | 无 body | `pause_requested` | stream/control terminal 不代表任务 paused |
| paused at boundary | 无 body | task projection paused | task lifecycle waiting |
| stopped | 无新 body | stopped status/task terminal | `stopped` |
| ask_user | 模型问题可见 | control waiting user | waiting terminal |
| error/block | 模型 block 可进 body；系统错误进 status | error status | `error.visible` 按 source 决定 |

收口硬约束：

1. 如果 `terminal_reason=task_executor_scheduled`，terminal.visible 必须为 false。
2. 如果 `answer_channel` 是 `runtime_control/task_control/active_work_control`，不能因为有 `content` 就进入 body。
3. `done.content` 不再是正文来源。正文只来自 assistant stream final 或 model body item。
4. `status/control` receipt 只更新运行态，不创建新的 assistant 正文。
5. session hydrate 只能重建已确认投影，不根据 terminal reason 补写旧文案。

### 13.10 流式输出稳定方案

流式正文使用 `assistant_stream_frame.v1`，公开投影负责把它归到 body 槽位；前端 display scheduler 只控制视觉节奏。

固定时序：

```text
ProviderRawDelta
-> AssistantStreamNormalizer
-> assistant_text_delta*
-> OutputBoundary canonical final
-> assistant_text_final
-> optional assistant_stream_repair
-> public done/stopped/error
-> SSE close
```

稳定性规则：

1. `assistant_text_final` 必须早于 public `done` 写入 ChatRun event log。
2. reducer 按 `(stream_ref, message_ref, sequence)` 去重，不按字符数去重。
3. UTF-8 byte offset 和 sha256 是跨端 replay/reconcile 契约。
4. provider chunk 不能直接改写 `Message.content`。
5. display scheduler 的 `displayedContent/timer/pending` 不进入全局 store，不参与 hydrate。
6. final hash 不匹配时发 repair 或进入 error state，不能静默用 `done.content` 覆盖。
7. tool_call、JSON action、内部协议前缀必须被 safety gate 阻止，不产生 public text delta。
8. stop/pause/error 到达时，normalizer 先 flush 已确认 public-safe 的 frame；不可确认内容丢弃或 diagnostics，不进入 body。

前端 canonical/display 双状态：

```text
canonicalContent: reducer 中的真实正文，可复制、可持久化、可 hydrate
displayedContent: ChatMessage 运行态动画文本，可丢弃、可重新派生
```

边界情况：

| 情况 | 处理 |
| --- | --- |
| SSE 重连重复 delta | sequence/hash 相同直接丢弃 |
| SSE 乱序 | 缓存缺口或 request replay，不直接追加 |
| done 先于 final | repair/error，不用 done.content 当正文 |
| pause/stop 中断流 | 保留已确认 canonical，terminal 标记 stopped/paused，不补写 |
| final 与 delta 不一致 | repair replacement，session history 保存 repair 后 final |
| 禁用流式显示 | canonical 仍合并，display final 时一次性显示 |
| reduced motion | 直接显示 canonical，不影响语义 |

### 13.11 缺 anchor、刷新和 monitor 接续

anchor 是投影能否进主聊天消息的前提。

anchor 解析优先级：

```text
public_anchor.anchor_turn_id
-> refs.turn_ref
-> payload.turn_id
-> model_action_request.turn_id
-> task_run.diagnostics.latest_interaction_turn_id
-> active_turn_registry(session_id)
-> monitor active_turn_snapshot
```

缺 anchor 处理：

1. `source_authority=model` 且可能是 body：不进入主消息，进入 pending + diagnostics，等待补 anchor。
2. `source_authority=tool/runtime/system`：可以进入 monitor-only，不进入主聊天。
3. 后续 active turn snapshot 补齐时，按 projection_id 合并。
4. 超时仍无法补齐，不显示为聊天正文。

刷新恢复：

```text
session hydrate
-> persisted envelope / task_projection
-> reducer rebuild MessagePublicProjection
-> live monitor envelope 覆盖 transient items
```

禁止：

- refresh 后把旧 `public_timeline` body fallback 重新塞进正文。
- monitor 事件因为找不到 message 就新建一条 assistant body。
- 同一 task_run 的 task projection 在刷新前后换位置。

### 13.12 迁移边界

实施时允许短期 shadow，但不能保留双主链路：

| 阶段 | 允许 | 禁止 |
| --- | --- | --- |
| Shadow | 同时生成 envelope 和 legacy delta，对比结果 | 前端继续新增 legacy 语义判断 |
| Dual public | 前端优先 envelope，legacy 只在无 envelope 的旧日志 adapter 使用 | 新功能依赖 `public_timeline_delta` |
| Cutover | envelope-only，legacy adapter 限定旧日志 | raw event -> body/status/tool fallback |

删除条件：

1. chat stream、monitor、hydrate 都能产出 envelope 或 task_projection。
2. `projectRuntimeStreamEvent`、`assistantDoneContentFromEvent` 不再被生产 reducer 调用。
3. `PublicTimelineActivity` 不再按 kind 推 body/tool/status。
4. `runtimeVisibilityProjection.ts` 无生产主链路导入。
5. fixed-port 长场景验证通过。

### 13.13 新增验收场景

除第 9 章验证矩阵外，实施时必须补充以下场景：

| 场景 | 预期 |
| --- | --- |
| 用户发送普通问题，模型 500ms 后才开始输出 | 只显示轻量 running，不出现伪正文 |
| 模型第一步调用工具 | 工具窗口先出现，正文为空 |
| 工具失败后模型解释 | 失败在 tool slot，解释在 body |
| request_task_run 后 stream 关闭 | task projection 保持运行，handoff done 不可见 |
| explicit steer accepted | 用户消息下只显示 control/status receipt，不显示 assistant body |
| stale expected_active_turn_id | fail closed，提示刷新，不接入错误任务 |
| running 任务 pause | 立即显示 pause_requested，安全边界后显示 paused |
| pause 时模型正在流式输出 | 不等待模型自然结束，不用 partial 伪 final |
| 不可取消工具 pause | 显示正在暂停，工具返回后落 paused |
| running 任务 stop | stop_requested/stopped 清楚，不复活 running spinner |
| resume paused task | 同一 task attachment 继续，不新建 assistant body |
| SSE reconnect | body/tool/task/status 均不重复 |
| final hash mismatch | repair 或 error，不静默覆盖 |
| session refresh | 正文、工具窗口、任务投影位置不变 |

### 13.14 实施中禁止再次出现的补丁行为

1. 不允许用前端关键词判断“后台任务”“工具调用”“停止”等文案归属。
2. 不允许让 `done.content`、`terminal_reason`、`answer_channel` 在前端直接生成正文。
3. 不允许把 `active_task_steer_accepted` 显示为模型回答。
4. 不允许暂停时等模型输出完整自然语言后才反馈用户。
5. 不允许暂停时直接丢弃 tool observation 或 transcript 事实。
6. 不允许把 prompt 写成“这是 runtime 节点/这是投影节点”这类开发说明；写给 agent 的 prompt 必须是可执行角色、边界和裁决标准。
7. 不允许用 UI 隐藏替代 runtime observation 保存。
8. 不允许用 legacy adapter 作为新功能入口。

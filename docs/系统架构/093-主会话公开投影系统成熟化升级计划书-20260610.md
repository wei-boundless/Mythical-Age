# 主会话公开投影系统成熟化升级计划书

日期：2026-06-10

状态：计划书，待审阅后实施

## 1. 目的

本计划用于把主会话公开投影从当前“可用的过渡态”升级为成熟稳定的单一权威系统。

当前已经完成一轮关键收口：

- single-agent task history hydrate 在存在 `task_projection` 时不再同时携带完整旧 `public_timeline`。
- `ChatMessage` 不再把稳定模型正文搬进 timeline。
- `PublicTimelineActivity` 不再把 `kind=observation/action` 强行改成工具窗口，而是尊重后端 `display_surface`。
- 停止、暂停等生命周期投影会压掉内部工具调用 detail。

这些修复解决了最明显的混合显示问题，但仍不足以称为成熟系统。成熟投影系统必须满足：

```text
Raw Runtime / Chat / Monitor Event
-> PublicProjectionProjector
-> PublicProjectionEnvelope
-> MessagePublicProjection reducer
-> Pure React render
```

任何用户可见正文、工具窗口、任务进展、系统状态都必须从同一个投影合同产生。前端只能渲染、折叠、去重和做安全清洗，不能重新判断语义归属。

## 2. 当前设计缺口

### 2.1 Envelope 仍是包装器，不是唯一 projector

代码事实：

- `backend/api/chat.py::_project_public_stream_event()` 先调用 `project_public_timeline_delta()` 生成旧 `public_timeline_delta`。
- `backend/api/chat.py::_attach_public_projection_envelope()` 再把 `public_timeline_delta` 包进 envelope。
- `backend/harness/runtime/public_projection_envelope.py` 当前从已有 items 推导 source/surface/slot。

问题：

`PublicProjectionEnvelope` 目前还没有拥有最终投影权威。它依赖旧 delta 先做一次判断，再做包装和过滤。这样 live SSE、runtime monitor、session hydrate 仍可能在不同路径中出现不同语义。

目标：

`PublicProjectionProjector` 直接接收 raw event，输出 envelope。`public_timeline_stream.py` 降级为内部 item builder，不再决定最终 `surface`、`slot`、`source_authority`。

### 2.2 前端 raw event fallback 仍能绕过 envelope

代码事实：

- `frontend/src/lib/store/events.ts` 仍导入 `projectRuntimeStreamEvent()`。
- `stageStatusForEvent()`、`activityDetailForEvent()`、`assistantDoneContentFromEvent()` 仍会从 raw event 生成状态、receipt 或正文。
- `reduceStreamEvent()` 在 envelope 缺失时继续使用 legacy `public_timeline_delta` 和 `task_projection_delta`。

问题：

只要某个后端事件没有 envelope，前端就会退回旧逻辑。这个 fallback 对短期兼容有帮助，但长期会让系统继续“修好一条入口，另一条入口又漏出来”。

目标：

切到 envelope-only 后，runtime/control/system 类事件缺 envelope 必须 fail closed：只记录诊断，不生成 assistant 正文、工具窗口或用户可见状态。

### 2.3 Legacy timeline 仍按 kind 反推 body/tool/status

代码事实：

- `frontend/src/lib/store/publicTimeline.ts` 的 `BODY_TIMELINE_KINDS` 仍把 `observation_report`、`stage_summary`、`final_summary` 等 kind 当作 body 候选。
- `frontend/src/components/chat/PublicTimelineActivity.tsx::activityLineKind()` 仍对 legacy item 用 `kind/surface` 推导 body/tool/status。

问题：

即使 task projection 已经尊重 `display_surface`，旧 turn-run 或 monitor public timeline 仍可能靠 kind 被前端重新分类。成熟系统中，slot 必须是后端投影合同的一部分，不能由前端猜。

目标：

所有进入主聊天区的公开 item 必须带 `slot`。没有 `slot` 的 legacy item 只能由明确的 legacy adapter 转换，且 adapter 有删除条件。

### 2.4 ChatPanel 仍通过 timeline body 猜测状态栏压制

代码事实：

- `ChatPanel.shouldSuppressSessionActivityBar()` 合并 public timeline 后判断是否隐藏底部活动条。
- 它调用 `hasDisplayablePublicTimelineBody()` 来判断消息级反馈是否已经展示。

问题：

这让 footer 状态栏继续受 public timeline body 推断影响，而不是受统一 `MessagePublicProjection` 控制。后续可能出现正文正常、活动条异常隐藏或重复显示。

目标：

`ChatPanel` 只读取 `MessagePublicProjection.sessionActivity` 或明确 `status/control` slot，不再从 timeline body 判断状态栏。

### 2.5 工具观察语义存在丢失风险

代码事实：

- 当前 `session_task_projection.py` 已过滤低信号和内部 activity。
- 但部分有意义的工具失败、观察判断、下一步动作仍可能因为 work unit kind/state 被压到 diagnostics。
- 移除旧 `public_timeline` 后，如果 task projection 没有稳定 tool observation slot，用户会看到 current action，却看不到关键失败观察。

问题：

成熟系统不能靠旧 public timeline 补齐 task projection 的语义缺口。工具观察必须进入统一 projection 合同，并同时保留给模型执行循环。

目标：

失败、超时、产物、验证结果等工具观察必须形成明确 `tool` 或 `timeline` slot。隐藏 UI 不得影响模型可见 observation 记录。

### 3.2 本地设计原则

本计划继承以下本地设计约束：

- `docs/系统架构/090-统一公开投影协议与投屏时序设计-20260610.md`
  - 三类输出不可混权：模型正文、工具观察、系统状态必须分开。
  - 前端只渲染投影，不重新裁决。
  - handoff done 必须 `terminal.visible=false`，不产生可见文案。
- `docs/系统架构/082-流式输出权威链路升级技术书-20260609.md`
  - ChatRun 是传输和重放真相，不拥有语义完成权威。
  - OutputBoundary 是最终 assistant 正文真相。
  - 正文流和进度流必须分离。
- `docs/系统框架/077-TaskRun状态权威收口与监控投影修复方案-20260609.md`
  - `TaskRunStateView` 是 TaskRun 状态和控制能力唯一解释器。
  - Session projection、monitor、前端都应消费 canonical view。
- `docs/系统架构/092-主会话公开投影单一权威重构计划书-20260610.md`
  - 已明确目标链路：raw event -> envelope -> reducer -> pure render。
  - 已列出 `public_timeline + task_projection` 混合 attachment 的删除方向。

### 3.3 参考架构原则

采用成熟 agent 系统的单向权威链原则：

```text
Observe raw facts
-> Normalize public facts
-> Decide projection authority
-> Assemble stable envelope
-> Reduce canonical UI state
-> Present without re-deciding
```

借用的机制：

- 事件、状态和输出分层。
- 输出边界由后端统一确定。
- 前端缺少权威字段时 fail closed，而不是自行补语义。
- 工具观察保留给 runtime/model，公开 UI 只消费投影后的摘要。

拒绝的机制：

- 不让前端继续通过 raw event name、tool name、answer_channel、terminal_reason 自行生成用户语义。
- 不用兼容字段长期驱动 UI。
- 不靠新增过滤词解决结构性混权。

## 4. 推荐目标设计

### 4.1 唯一权威链

```text
RuntimeEvent / ChatEvent / MonitorEvent / SessionHydrateRecord
-> PublicProjectionProjector
-> PublicProjectionEnvelope
-> MessagePublicProjectionReducer
-> ChatMessage / PublicTimelineActivity / SessionActivityBar
```

### 4.2 层级职责

| 层 | 允许做什么 | 禁止做什么 |
| --- | --- | --- |
| Raw event | 记录事实、refs、payload、状态 | 直接决定用户可见正文 |
| PublicProjectionProjector | 决定 source/surface/slot/terminal/anchor | 依赖前端兜底修语义 |
| TaskProjectionAdapter | 将 TaskRun state view 和 progress presentation 转成 task projection | 重新解释控制能力，泄漏内部工具 |
| MessagePublicProjectionReducer | 合并 envelope，去重，维护 body/timeline/tool/status/control | 从 raw event name 生成语义 |
| React render | 布局、折叠、Markdown、复制、视觉状态 | 改变 source/surface/slot |

### 4.3 MessagePublicProjection

前端应收敛到一个稳定结构：

```ts
type MessagePublicProjection = {
  body: {
    text: string;
    source: "model";
    canonicalState?: string;
  };
  timeline: PublicProjectionItem[];
  toolWindows: PublicProjectionItem[];
  taskProjection?: SingleAgentTaskProjection;
  controls: PublicProjectionItem[];
  status: PublicProjectionItem[];
  terminal?: {
    event: "done" | "error" | "stopped";
    visible: boolean;
    reason?: string;
  };
  tone: "running" | "waiting" | "done" | "error" | "stopped";
};
```

说明：

- `body.text` 只来自 `source_authority=model` 且 `slot=body`。
- `task_projection.final_answer` 只作为 TaskRun projection 的收口材料；若与 canonical body 相同，reducer 去重。
- `toolWindows` 只来自 `slot=tool`。
- `status` 和 `controls` 不允许变成 assistant body。

### 4.4 PublicProjectionEnvelope 合同

后端 envelope 必须稳定包含：

```ts
type PublicProjectionEnvelope = {
  authority: "harness.public_projection.v1";
  projection_id: string;
  sequence: number;
  session_id: string;
  created_at: number;
  anchor: {
    turn_id?: string;
    message_id?: string;
    task_run_id?: string;
    run_id?: string;
    turn_run_id?: string;
    anchor_role: "assistant" | "status" | "monitor";
  };
  source_authority: "model" | "tool" | "system" | "runtime";
  surface: "assistant_body" | "timeline" | "tool_window" | "status_bar" | "task_projection" | "control" | "diagnostics";
  lifecycle: "running" | "waiting" | "done" | "error" | "stopped";
  terminal?: {
    event: "done" | "error" | "stopped";
    visible: boolean;
    reason?: string;
  };
  items: PublicProjectionItem[];
  task_projection?: SingleAgentTaskProjection;
  active_turn_update?: {
    turn_id?: string;
    task_run_id?: string;
    state?: string;
  };
};
```

每个 `PublicProjectionItem` 必须含有：

```ts
{
  item_id: string;
  slot: "body" | "timeline" | "tool" | "status" | "task" | "control" | "diagnostics";
  source_authority: "model" | "tool" | "system" | "runtime";
  surface: string;
  kind: string;
}
```

缺少 `slot/source_authority/surface` 的 item 不进入主聊天渲染，只能进入 legacy adapter 或 diagnostics。

## 5. 固定执行流

### 5.1 普通模型回答

```text
OutputBoundary final answer
-> PublicProjectionProjector(source=model, surface=assistant_body, slot=body)
-> MessagePublicProjection.body
-> ChatMessage body
```

禁止：

- `done.summary` 伪造成正文。
- `receipt_summary` 作为正文 fallback。
- frontend 从 `answer_channel` 决定正文是否生成。

### 5.2 工具调用与观察

```text
ModelAction(tool_call)
-> Tool execution
-> ToolObservation
-> PublicProjectionProjector(source=tool/runtime, slot=tool or timeline)
-> MessagePublicProjection.toolWindows / timeline
-> PublicTimelineActivity render
```

禁止：

- 工具 stdout、JSON、路径列表进入 assistant body。
- UI 隐藏某条观察导致 runtime/model 观察也丢失。
- 前端按 `kind=work_action` 自行决定 tool window。

### 5.3 后台任务 handoff

```text
request_task_run
-> task_run created
-> task_projection generated
-> done(terminal_reason=task_executor_scheduled)
-> envelope(terminal.visible=false, surface=task_projection, active_turn_update)
-> stream closes
-> monitor envelope continues updating same task attachment
```

禁止：

- handoff done 生成 assistant body。
- handoff done 生成“后台任务已接管”类可见正文。
- session refresh 恢复旧 public timeline。

### 5.4 Session hydrate

```text
Persisted task run / turn run
-> session_timeline
-> task_projection or persisted envelope projection
-> runtime attachment
-> MessagePublicProjectionReducer
```

规则：

- single-agent task attachment 有 `task_projection` 时，`public_timeline` 只能为空或缺省。
- turn-run legacy attachment 允许短期保留 `public_timeline`，但必须带 legacy 标记并有删除条件。

## 6. 阶段计划

### Phase 0：合同冻结与测试前置

目标：先锁住成熟投影合同，避免后续实现继续漂移。

影响文件：

- `frontend/src/lib/api.ts`
- `backend/harness/runtime/public_projection_envelope.py`
- `backend/tests/public_projection_envelope_test.py`
- `frontend/src/lib/store/publicProjectionReducer.test.ts`

动作：

1. 明确定义 `PublicProjectionEnvelope`、`PublicProjectionItem`、`MessagePublicProjection`。
2. 后端补合同测试：
   - model body 只能进入 `slot=body`。
   - tool observation 只能进入 `slot=tool/timeline`。
   - system/runtime/control 不能进入 body。
   - handoff done 必须 `terminal.visible=false`。
   - 缺 anchor 的 event 不静默投主消息。
3. 前端补 reducer 测试：
   - envelope -> body/timeline/tool/status/control。
   - envelope 缺 slot 时 fail closed。
   - stable body 与 task final answer 去重。

完成条件：

- 没有实现生产切换前，合同测试已经能明确描述目标结构。
- 所有新合同测试先红后绿，不通过放宽断言制造绿。

禁止：

- 继续新增前端过滤词。
- 继续让 legacy `kind` 成为正式 slot。

### Phase 1：后端 PublicProjectionProjector 单一化

目标：chat stream 和 runtime monitor 统一调用同一个 projector。

影响文件：

- `backend/harness/runtime/public_projection_projector.py` 新增
- `backend/harness/runtime/public_projection_envelope.py`
- `backend/harness/runtime/public_timeline_stream.py`
- `backend/harness/runtime/runtime_monitor_public_projection.py`
- `backend/api/chat.py`

动作：

1. 新增 `PublicProjectionProjector.project_event(event, context)`。
2. 把 `_project_public_stream_event()` 从“生成 data + public_timeline_delta”改为“生成 raw public fact + projector envelope”。
3. `runtime_monitor_public_projection.py` 改为调用同一 projector。
4. `public_timeline_stream.py` 改名或降级为 `public_projection_items.py`，只提供内部 item builder。
5. 所有 envelope item 由 projector 赋予 `slot/source_authority/surface`。

完成条件：

- chat SSE 和 monitor SSE 对同一 runtime event 产生同构 envelope。
- `api/chat.py` 不再直接向 UI 决定 `public_timeline_delta`。
- legacy fields 只允许 shadow 输出，不驱动 UI。

回滚条件：

- 若 projector 生成 envelope 丢失 anchor 或 terminal，回滚到 Phase 0 合同，不恢复前端 raw event 推断。

### Phase 2：TaskProjection 工具观察补齐

目标：移除旧 `public_timeline` 后，task projection 仍能稳定展示关键工具动作和观察。

影响文件：

- `backend/harness/runtime/session_task_projection.py`
- `backend/harness/runtime/progress_presenter.py`
- `backend/harness/runtime/public_projection_filters.py`
- `backend/tests/session_task_projection_test.py`

动作：

1. 为 tool observation 生成明确 activity：
   - success -> `display_surface=tool_window/timeline`, `visibility_level=primary/secondary`
   - error/timeout -> `display_surface=tool_window`, `state=failed`
   - debug read/search/path -> diagnostics
2. `stage` work unit 不再吞掉 primary tool observation。
3. `current_action` 不承载完整工具失败语义，只描述当前动作。
4. `activities` 承载关键动作和观察。
5. 增加测试：
   - image timeout 显示为 tool/timeline activity。
   - write/edit artifact 显示为 tool window。
   - read/search 低信号默认不进聊天。
   - agent_todo/subagent 内部工具不进聊天。

完成条件：

- 删除 single-agent task legacy public timeline 后，工具失败、产物、验证结果仍可投屏。
- UI 隐藏不影响 runtime observation 保存和模型后续输入。

## 8. 迁移、切换和回滚规则

### 8.1 Shadow 规则

允许短期同时输出：

- `public_projection_envelope`
- legacy `public_timeline_delta`
- legacy `task_projection_delta`

但 UI 主链路只能使用 envelope。legacy fields 只用于测试对比和旧日志 adapter。

### 8.2 Cutover 条件

满足以下条件后切到 envelope-only：

1. chat stream、runtime monitor、session hydrate 都能产出 envelope 或 task projection。
2. `MessagePublicProjection` 覆盖正文、工具、状态、控制、任务投影。
3. 固定端口 3000/8003 实测通过。
4. `rg` 确认旧权威没有生产导入。
5. 所有缺 envelope 的 runtime/control/system event 都 fail closed。

### 8.3 回滚规则

允许回滚到上一阶段的 envelope adapter，但不允许重新启用多个 UI 决策源。

触发回滚：

- envelope anchor 丢失导致消息绑定错位。
- terminal.visible 误判导致普通回复不显示或 handoff 变正文。
- 工具观察大面积丢失。
- session refresh 覆盖 live projection。

回滚后必须补合同测试，再继续实施。

## 10. 禁止捷径

1. 禁止继续添加前端关键词过滤来掩盖混权。
2. 禁止保留两个正式 UI 决策源。
3. 禁止用兼容理由长期保留 raw event -> body fallback。
4. 禁止把 `answer_channel`、`terminal_reason`、`tool_name` 当作前端正文决策依据。
5. 禁止通过删除失败测试或放宽断言制造通过。
6. 禁止让系统状态伪装成“我会/我已”的模型正文。
7. 禁止让 UI 投屏过滤影响 runtime/model 的真实 observation 记录。
8. 禁止遇到端口问题临时换端口；仍使用固定 `3000` / `8003`。

## 11. 预期结果

完成后，主会话公开投影具备以下系统属性：

1. **单一权威**：所有公开显示都来自 `PublicProjectionEnvelope`。
2. **刷新一致**：live、monitor、session hydrate 三条路径显示语义一致。
3. **正文干净**：只有模型正文进入 assistant body。
4. **工具清楚**：工具动作和观察进入 tool/timeline slot，不污染正文。
5. **系统有边界**：handoff、pause、stop、runtime status 只进入 status/control/task projection。
6. **可验证**：每条投影能追溯 source authority、anchor、sequence、terminal。
7. **可维护**：前端组件不再维护大堆语义兜底，后续问题可在 projector 合同层修复。

# 写作图自审 Loop 结构化改造方案

日期：2026-06-03

## 1. 背景

当前写作图已经能跑通主链路、质量门同节点回灌和章节/批次/分卷循环，但“自审”这件事的落点仍然不对。

现状不是“图里存在一个可复用的自审 loop 模块”，而是：

- 在 [scripts/configure_writing_modular_novel_graph.py](/D:/AI应用/langchain-agent/scripts/configure_writing_modular_novel_graph.py) 里用 `_with_writer_self_repair_nodes(...)` 直接展开 `volume_plan_self_repair`、`chapter_outline_self_repair`、`chapter_draft_self_repair`
- 再由 `CHAPTER_BUSINESS_EDGES` 手工拼接自修边
- 同时把 `*_self_repair` 节点名写入 `memory_topics`、`scope_node_ids`、`unit_repair_node_id`、`OUTLINE_THREAD_INDEX_NODE_IDS`

这说明当前“是否启用自审、如何回路、回到哪里、哪些节点属于自审作用域”都由配置脚本硬编码决定，而不是由图结构自己表达。

用户要求是正确的：自审应该是独立 loop 结构，而不是一个节点的硬编码块。

## 2. 当前代码源报告

### 2.1 自审的当前实现位置

主要集中在 [scripts/configure_writing_modular_novel_graph.py](/D:/AI应用/langchain-agent/scripts/configure_writing_modular_novel_graph.py)：

- `CHAPTER_BUSINESS_EDGES`
- `_chapter_loop_frames()`

结论：

- 自审节点是“脚本注入”的，不是图编辑器显式表达的结构
- runtime 看到的是一堆普通节点和边，并不知道这是一个通用的 repair loop
- 图的业务结构和自审策略装配耦合在一起

### 2.2 runtime 当前已经具备的通用能力

现有 runtime 并不缺“循环基础能力”，关键文件有：

- [backend/task_system/compiler/layered_graph_normalizer.py](/D:/AI应用/langchain-agent/backend/task_system/compiler/layered_graph_normalizer.py)

现有通用能力包括：

- `loop_frames`

- `scope_node_ids`
- loop route 决策
- continue / exit / blocked
- revision requeue
- quality gate 失败后的 same-node recoverable requeue

结论：

- runtime 已经有能力承载“自审 loop”
- 问题不在 runtime 没有 loop，而在写作图没有把自审建模成 loop 结构
- 因此后续优先方向应当是复用现有 `loop_frames + revision/repair edges`，而不是立刻再发明一层新的底层 loop schema

### 2.3 现有正式返修链路

当前系统并不是只有 `*_self_repair` 这一套返修手段。

在 [scripts/configure_writing_modular_novel_graph.py](/D:/AI应用/langchain-agent/scripts/configure_writing_modular_novel_graph.py) 中，多个审核节点已经通过 `review_revision_stage_id` 声明返修目标，例如：

- `chapter_review -> chapter_draft`
- `outline_review -> outline_design`
- `final_review -> final_assemble`

同时，脚本会自动生成 `revision_request` 边，相关语义已经存在于：

- [backend/task_system/contracts/writing_contract_families.py](/D:/AI应用/langchain-agent/backend/task_system/contracts/writing_contract_families.py)
- [backend/harness/graph/language.py](/D:/AI应用/langchain-agent/backend/harness/graph/language.py)
- [backend/task_system/compiler/layered_graph_normalizer.py](/D:/AI应用/langchain-agent/backend/task_system/compiler/layered_graph_normalizer.py)

结论：

- 审核返修链路已经是现有正式机制的一部分
- 后续不能把 repair loop 设计成一套与 `revision_request` 平行的新体系
- 必须统一三类机制：审核返修、质量门回灌、独立 self_repair 节点链
### 2.4 目前已经暴露出的设计问题

1. 结构权和运行权混淆  
自审是否存在，不是图定义说了算，而是配置脚本说了算。

2. 节点名污染运行时语义  
`chapter_draft_self_repair_ref`、`unit_repair_node_id` 这类字段把“某个具体节点名”写进了通用 contract/memory 元数据。

3. 复用性差  
换一个任务图、换一个阶段、换一个写手节点，都得继续扩展 `*_self_repair` 的脚本分支。

4. 图编辑器无表达力  
如果自审是脚本注入的，编辑器看到的并不是“一个结构模块”，而是一堆散边和散节点。

5. 质量门与自审语义交叉  
现在既有独立自修节点，又有 quality gate same-node 回灌，同时还有审核节点生成的 `revision_request` 链路，实际上已经形成三套 repair 机制并存。

6. runtime 里仍有基于节点名/字符串的 repair 猜测  
当前 [backend/harness/graph/work_order_executor.py](/D:/AI应用/langchain-agent/backend/harness/graph/work_order_executor.py) 的 `_has_quality_repair_route(...)` 仍会通过 `"self_repair" in target` 之类的字符串规则推断 repair 语义，这与“runtime 只处理通用结构”的目标矛盾。

## 3. 目标设计

目标不是“保留 `*_self_repair` 但把提示词写好”，而是把自审收回到图结构层，形成可复用 loop。

终态设计：

- 图编辑器负责表达 repair loop 结构
- graph compiler 负责把 repair loop 编译成标准 `loop_frames + nodes + edges`
- graph runtime 只负责通用 loop 调度，不认识“写作自审”这种业务词
- 普通写手节点不再硬编码 `*_self_repair_ref`
- 质量门只负责提供失败反馈和重试信号，不负责定义自审拓扑
- 审核返修、质量门回灌、自审修复都收敛到一套统一 repair 语义，而不是三套并存

## 4. 目标架构

### 4.1 分层职责

图结构层：

- 决定某个阶段是否启用 repair loop
- 决定 repair loop 的入口、修复节点、裁决节点、回跳条件、退出条件、上限次数

编译层：

- 把 repair loop 转成标准 graph 结构
- 统一生成 `loop_frames`、作用域、修复边、退出边、反馈注入边

runtime 层：

- 只处理通用 loop frame 的进入、继续、退出、阻塞、恢复
- 不关心节点是不是“自审写手”

节点层：

- 写手只负责产出候选稿
- repair writer 只负责在 repair loop 内根据反馈改稿
- reviewer/quality gate 只负责裁决与反馈

### 4.2 标准 repair loop 结构

建议抽象成一个通用子结构：

1. `producer`  
产出候选稿

2. `repair_gate`  
收集质量门、审核、连续性检查返回的失败反馈

3. `repair_worker`  
基于反馈重写当前候选稿

4. `repair_router`  
判断继续 repair、退出成功、达到上限后阻塞

这个 loop 应当是结构模块，不应该要求名字必须叫 `chapter_draft_self_repair`。

但这里的“结构模块”优先应由现有图结构表达，不应先急着新增一层底层 schema。
在当前系统里，优先路线是：

- 用现有 `loop_frames`
- 用现有 `revision_request / repair_feedback / repair_route` 语义边
- 用现有节点与边组合出 repair loop

只有当编辑器层确实无法稳定表达该结构时，才考虑新增 authoring sugar；不应先改 runtime 底层模型。

### 4.3 对写作图的具体落点

写作图里有三类可选 repair loop：

1. 卷计划 repair loop  
`volume_plan -> repair_loop -> chapter_outline`

2. 批次细纲 repair loop  
`chapter_outline -> repair_loop -> chapter_draft`

3. 单章正文 repair loop  
`chapter_draft -> repair_loop -> chapter_unit_router`

其中“单章正文 repair loop”与“质量门 same-node 回灌”不能长期并存为两套主机制，必须二选一；同时也必须和现有 `chapter_review -> chapter_draft` 的 `revision_request` 返修链统一。

建议：

- 短期：保留 same-node 回灌，关闭独立 `*_self_repair` 节点，先保证效率
- 中期：把 same-node 回灌和 review revision 一起升级为图结构化 repair loop
- 长期：所有 repair 都走统一 loop 模块，不再依赖节点名分支

## 5. 对现有代码的具体修正建议

### 5.1 配置脚本层

需要删除或降级的写法：

- `_with_writer_self_repair_nodes(...)`
- `SELF_REPAIR_WRITER_NODE_IDS`
- `SELF_REPAIR_SOURCE_BY_NODE_ID`
- `SELF_REPAIR_ARTIFACT_PATHS_BY_SOURCE`
- `CHAPTER_BUSINESS_EDGES` 中硬编码的 `*_self_repair` 链路
- `_chapter_loop_frames()` 中把 `*_self_repair` 写死到 `scope_node_ids`
- `memory_topics` 中硬编码 `chapter_draft_self_repair_ref`
- `_unit_batch_binding(...)` 里的 `unit_repair_node_id`

替代方式：

- 图定义直接用现有节点、边、`loop_frames` 表达 repair loop
- 配置脚本只负责装配结构，不直接注入具体“self_repair 节点族”
- 若未来编辑器 authoring 层需要更高层封装，也应在编辑层展开到现有图结构，而不是先改 runtime 底层 schema

### 5.2 图模型层

当前阶段不建议先修改底层 `TaskGraphDefinition` schema 去新增 `repair_loops` / `subloops`。

原因：

- 现有系统已经有 `loop_frames`
- 现有系统已经有 `revision_request / repair_feedback / repair_route`
- 现有系统已经能表达节点、边、作用域和路由

因此本期目标应当是：

- 先把 repair loop 收敛成“现有图结构的标准组合”
- 先证明编辑器和编译层能稳定画出、发布、运行这类结构
- 只有在 authoring 体验确实不足时，再考虑增加“编辑层快捷结构”，并在发布前展开成现有图模型

也就是说：

- repair loop 应当先成为“结构约定”
- 不应在本期直接成为新的 runtime 底层 schema

### 5.3 编译层

在 [backend/task_system/compiler/graph_harness_config_publisher.py](/D:/AI应用/langchain-agent/backend/task_system/compiler/graph_harness_config_publisher.py) 与 [backend/task_system/compiler/layered_graph_normalizer.py](/D:/AI应用/langchain-agent/backend/task_system/compiler/layered_graph_normalizer.py) 中：

- 新增 repair loop 结构到标准 `loop_frames` 的编译
- 自动生成 repair 作用域边
- 自动把 feedback packet 注入 repair worker 输入
- 自动生成次数上限与退出策略

编译层原则：

- repair loop 是通用 loop 的业务投影
- 编译后 runtime 只看到标准 loop，不看到“写作专用自审”
- 编译层还需要把现有 `review_revision_stage_id` 返修链与 quality gate repair 信号统一映射到同一套 repair 结构，而不是让两边各跑一套

### 5.4 runtime 层

runtime 不需要再加“写作自审”特判。

只需要保证：

- loop exit 节点在 frame 未退出前不能被调度
- recoverable error 能触发 repair loop 继续
- loop router 可以读取反馈并决定 continue / exit / blocked

当前已有的通用修复已证明方向正确：

- [backend/harness/graph/state_machine.py](/D:/AI应用/langchain-agent/backend/harness/graph/state_machine.py)
- [backend/harness/graph/loop.py](/D:/AI应用/langchain-agent/backend/harness/graph/loop.py)

这部分应该继续保持“通用调度语义”，不要再塞写作专用节点名。

另外必须补一项清理：

- 把 [backend/harness/graph/work_order_executor.py](/D:/AI应用/langchain-agent/backend/harness/graph/work_order_executor.py) 中 `_has_quality_repair_route(...)` 的字符串猜测逻辑收掉
- repair 是否存在，必须来自显式 edge semantic、retry policy、loop frame metadata 或明确的结构标记
- runtime 不能继续通过 `"self_repair" in target` 这类规则推断业务语义

## 6. 迁移策略

### 阶段一：止血

目标：先把效率拉回来。

- 关闭独立 `volume_plan_self_repair / chapter_outline_self_repair / chapter_draft_self_repair`
- 保留质量门 same-node 回灌
- 保留审核节点
- 保证 0-10 章能稳定推进

### 阶段二：结构抽象

目标：把 repair 从节点硬编码抽到图结构。

- 用现有 `loop_frames + revision/repair edges` 组合出 repair loop
- 调整 graph normalizer / publisher，让这类组合结构发布稳定
- 让写作图通过结构装配启用 repair loop，而不是注入节点族
- 把 `review_revision_stage_id` 与 quality repair 信号纳入统一 repair 结构

### 阶段三：统一 repair 机制

目标：消除两套修复机制并存。

- 让质量门失败、审核返修、连续性返修都走统一 repair loop
- same-node requeue 只作为 runtime 恢复手段，不再作为最终业务拓扑
- 删除独立 `*_self_repair` 节点链
- 删除 runtime 中基于节点名/字符串猜测 repair 的逻辑

## 7. 风险与边界

1. 不能把“关闭独立自审节点”误当成最终方案  
这只是止血，不是结构终态。

2. 不能让 runtime 知道太多 repair 业务词  
例如 `unit_repair_node_id`、`chapter_draft_self_repair_ref` 这类字段不应继续扩散。

3. 不能把 repair loop 做成写作专用  
它必须是图系统通用能力，否则后续别的任务图还会重复犯同样问题。

4. 不能把 review loop 和 repair loop 混成一个东西  
review 是裁决，repair 是修复，它们可以串联，但职责必须分开。

5. 不能忽略现有正式 revision 链  
如果 repair loop 方案不吸收 `review_revision_stage_id -> revision_request` 这条现有链，最后一定会继续留下多套返修机制并存。

6. 不能只改图脚本不改 runtime 猜测逻辑  
否则图结构虽然变干净了，runtime 仍会通过节点名字符串携带旧业务假设。

## 8. 推荐实施顺序

1. 回退当前未完成的“配置脚本开关式禁用自审”半成品，保持代码整洁
2. 单独做一期“止血版”：在写作图里关闭独立 `*_self_repair` 节点，仅保留 same-node 质量回灌
3. 清理 runtime 中基于节点名/字符串猜 repair 的逻辑，改为显式结构判断
4. 用现有 `loop_frames + revision/repair edges` 先落“单章正文 repair loop”最小闭环
5. 把 `chapter_review -> chapter_draft` 返修链纳入统一 repair 结构
6. 再把卷计划和批次细纲 repair loop 迁移过去
7. 最后删除旧的 `_with_writer_self_repair_nodes(...)` 体系

## 9. 结论

基于现有代码，问题已经很明确：

- runtime 的 loop 能力基本够用
- 真正不成熟的是写作图自审的结构表达方式
- 当前 `*_self_repair` 体系属于“配置脚本注入节点”的过渡实现，不适合作为长期架构

因此后续正确路线不是继续修补 `chapter_draft_self_repair`，而是：

- 短期把独立自审节点关掉，保效率
- 中期把自审重建成图结构层的 repair loop
- 长期让 runtime 只处理通用 loop，彻底移除写作专用硬编码节点链

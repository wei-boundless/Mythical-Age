# Agent 投影实时输出与工具轨迹时序修复计划书

日期：2026-06-14

## 目标

修复当前 agent 投影系统的三个结构性问题：

1. 长任务完成时，正文只存在于 `final_answer` / terminal snapshot，没有作为公开正文事件实时投影。
2. 工具调用轨迹已经进入 projection reducer，但前端消息面只展示正文和少量 pinned 错误，正常工具生命周期不可见。
3. `tool_call.id` 和 `action_request.request_id` 混用，导致 requested / started / completed 生命周期 id 漂移。

修复后的公开投影必须满足：

```text
model visible body
-> assistant_text_delta / assistant_text_final

agent-controlled tool trajectory
-> tool_call_requested
-> tool_permission_decided
-> tool_item_started
-> tool_item_completed

single ordering authority
-> runtime event offset
-> chat_stream_event offset
-> projection frame event_offset
-> frontend ordered timeline
```

正文只能来自模型可见输出的 canonical final text；工具轨迹只能来自 runtime 在 action permit 和 execution loop 下产生的生命周期事件；前端只负责按投影帧顺序呈现，不允许根据 terminal snapshot 反推正文，不允许把工具输出伪装成 assistant 正文。

## 当前问题证据

### 1. 正文投影缺失

- `backend/harness/loop/task_executor.py` 的长任务模型输出按 JSON action 解析，不能直接把流式 token 当用户正文。
- `_finish_executor_success` 只提交 session final message 并 finish lifecycle，没有在 commit 前写入 `assistant_text_final`。
- `backend/api/chat.py` 已经把“terminal 带 final_answer 但前面没有 assistant output / commit”视为协议失败，这个诊断应该保留。

### 2. 工具轨迹前端不可见

- `frontend/src/lib/projection/reducer.ts` 已经保存 `currentAction`、`status`、`pinned`、`trace`、`body`。
- `frontend/src/components/chat/ChatMessage.tsx` 主要只把 `pinned` 系统错误变成可见 timeline activity。
- 正常的 requested / permission / started / completed 生命周期没有进入 assistant message surface。

### 3. tool id 漂移

- 模型 action 中 `tool_call.id` 已经保留。
- `_execute_task_tool_call` 构造 `ToolInvocationRequest` 时又使用 `action_request.request_id` 作为 `tool_call_id`。
- completed 事件从 observation / result envelope 读取到的 id 可能与 requested / started 不一致，触发 `tool_completed_without_started_lifecycle`。
- projector 里存在 completed 阶段的 permission id 兼容修正，这会掩盖源头错误，应在源头修复后移除。

## 目标权威链

| 层级 | 权威 | 责任 |
| --- | --- | --- |
| ModelTurnDecision | `TaskExecutionModelActionRequest` | 决定 respond / ask_user / tool_call / block，并提供模型可见正文或工具调用请求 |
| ActionPermit | `admit_model_action` + `action_permit_from_admission` | 决定工具是否允许执行 |
| ExecutionLoop | `task_executor` | 执行工具、记录 observation、记录正文投影和 commit |
| Record | runtime event log | 用单调 offset 记录正文、工具生命周期、commit、terminal |
| Projection | `api/chat.py` + `ProjectionLifecycleState` | 把 runtime event 转为公开 projection frame，并校验生命周期 |
| Present | frontend reducer / ChatMessage | 按 `event_offset` 合并正文和工具轨迹，不重写语义 |

## 实施计划

### Slice 1：后端 tool id 权威收束

文件：

- `backend/harness/loop/task_executor.py`
- `backend/harness/runtime/projection/projector.py`
- `backend/api/chat.py`

改动：

1. 增加 canonical tool call id 读取逻辑：优先 `action_request.tool_call.id`，缺失时才使用 `action_request.request_id`。
2. `build_tool_invocation_id`、`ToolInvocationRequest.tool_call_id`、`tool_item_started`、executor error observation、tool control plane observation 全部使用同一个 canonical id。
3. admission fallback id 保持与 action request ref 分离；permission id 可以继续使用 admission id，但不得用来替换 tool call id。
4. 删除 projector completed 阶段“把 `admission:{tool_call_id}` 悄悄改回 recorded permission id”的兼容修正。

测试：

- 模型自带 `tool_call.id` 时，requested / started / completed 的 `tool_call_id` 一致。
- 模型未带 id 时，runtime 只生成一次 id 并贯穿生命周期。
- 故意错 id 时仍产生 protocol diagnostic，不做静默修复。

### Slice 2：后端 assistant body projection

文件：

- `backend/harness/loop/task_executor.py`
- `backend/runtime/shared/events.py`
- `backend/api/chat.py`

改动：

1. 在 `_commit_task_run_final_message` 中，以同一个 `canonical_output_decision_for_final_text` 结果先写入 `assistant_text_final`，再写入 `session_output_commit_checked` 和 commit terminal event。
2. `assistant_text_final` 包含 `session_id`、`turn_id`、`task_run_id`、`message_ref`、canonical answer metadata、content hash。
3. commit 使用同一份 canonical content，避免正文投影和历史提交不同步。
4. 保留 `api/chat.py` 的 terminal diagnostic；它继续作为缺正文/缺 commit 的协议守卫。

边界：

- 当前 task executor 的模型输出协议是 JSON action，因此本次不把 JSON stream token 投影为正文 delta。
- 本次保证 final body 进入公开投影且顺序正确；真正 token/chunk 级正文实时化需要后续升级模型输出协议为“控制流 + 正文流”双通道。

测试：

- task executor 成功完成时，event log 中 `assistant_text_final` 必须早于 `session_output_commit_checked` / `session_output_commit_ack` / lifecycle terminal。
- chat bridge 不再因为成功 task terminal final_answer 缺正文而产生 commit failed diagnostic。

### Slice 3：前端按 offset 展示工具轨迹

文件：

- `frontend/src/lib/projection/reducer.ts`
- `frontend/src/lib/store/events.ts`
- `frontend/src/components/chat/ChatMessage.tsx`
- 相关测试文件

改动：

1. 生成统一 timeline item：
   - `assistant_text`
   - `tool_requested`
   - `tool_permission`
   - `tool_started`
   - `tool_completed`
   - `tool_failed`
   - `diagnostic`
2. `body`、`currentAction`、`status`、`pinned`、`trace` 都带 projection frame offset 进入同一个排序列表。
3. ChatMessage 渲染时按 offset 展示正文和工具活动；工具活动用控制轨迹样式展示，不并入 assistant 正文。
4. 删除或改写“工具投影不进入 assistant surface”的旧测试断言。

测试：

- 无正文但有工具调用时，消息区展示工具活动，不只显示“正在思考”。
- 正文和工具生命周期交错时，展示顺序严格等于 projection frame offset。
- successful completed tool 能在轨迹中可见，失败工具仍 pinned。

## 不做的事

- 不从 terminal snapshot 在前端反推正文。
- 不把工具 observation 原文当 assistant 正文渲染。
- 不保留 id 漂移兼容修正。
- 不把 JSON action token 误投影成用户正文。
- 不切换本地固定端口。

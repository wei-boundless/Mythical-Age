# Turn 总结收口与过程反馈线路修复记录

日期：2026-06-15

## 问题

截图中的普通 turn 已经完成多次工具调用，但页面没有最终总结收口，且工具后的自然过程反馈没有继续出现。

## 真实事件证据

目标 run：`strun:f687d8b7e0214ab4871f471e347437e0`

关键事件：

| turn event offset | 信号 | 证据 |
| --- | --- | --- |
| 1 | `model_action_admission_checked/tool_call` | 首条 `public_progress_note` 被投影 |
| 31 | `model_action_admission_checked/tool_call` | “现在更新游戏说明书...” 被模型返回 |
| 37 | `model_action_admission_checked/tool_call` | “让我验证一下文件是否完整...” 被模型返回 |
| 43 | `model_action_admission_checked/respond` | 模型返回 `final_answer`：“优化完成。以下是本次改动总结...” |
| 44 | `session_output_commit_checked` | 提交闸门收到空正文，原因 `empty_assistant_message_blocked` |
| 46 | `agent_turn_terminal` | turn 被错误标记为 failed |

结论：不是模型没有给总结，而是后端输出边界把模型总结清空；不是模型没有过程反馈，而是投影身份去重把同一 turn 内后续反馈吞掉。

## 根因

### 1. 最终总结被误判为伪工具调用

`backend/runtime/output_boundary/boundary.py` 原先用过宽正则识别伪工具调用：

```text
任意函数名(...)
```

因此总结里的普通代码函数名 `moveAndCollide()` 被当成工具调用协议泄漏，触发 `inline_pseudo_tool_call_final_text`，最终正文被清空并导致 commit 被拦截。

### 2. respond 反馈没有进入公开投影

`backend/api/chat.py` 的 `_model_action_feedback_step_data` 只允许 `tool_call/tool_calls` 产生 `runtime_step_summary`。模型 `respond` 里的 `public_progress_note` 没有被投影。

### 3. 同一 turn 内多次反馈被错误去重

`_model_action_feedback_identity` 优先使用 `runtime_invocation_packet_ref`。同一 turn 的多次模型动作共享同一个 packet ref，导致 offset 31、37、43 的反馈被当成重复反馈吞掉。

## 修复

1. `backend/runtime/output_boundary/boundary.py`
   - 伪工具调用识别改为只匹配独立行上的已知工具名调用，例如 `read_file(...)`。
   - 普通代码函数名如 `moveAndCollide()` 不再触发工具协议泄漏。
   - 独立伪工具调用行只作为可清洗内容处理；真正 `<tool_call>`、DSML、JSON action 等内部协议仍然硬拦。

2. `backend/api/chat.py`
   - 只要模型动作包含 `public_progress_note/current_judgment`，无论 action 是 `tool_call` 还是 `respond`，都生成 `runtime_step_summary`。
   - 反馈 identity 改为 `batch_action_request_ref/action_request_ref/request_id` 优先，`runtime_invocation_packet_ref` 只做兜底。

## 修复后的线路

```text
model_action_admission_checked
  -> public_progress_note/current_judgment
  -> runtime_step_summary
  -> public_projection_frame(body_append)
  -> 前端按 event_offset 插入过程反馈

model_action_admission_checked/respond
  -> final_answer
  -> canonical_output_decision_for_final_text
  -> session_output_commit_checked/ack
  -> assistant_text_final(body_finalize)
  -> 前端收口为最终正文
```

# 任务入口控制与 Prompt 协议泄露修复计划书

日期：2026-06-13
状态：待实施计划书
适用范围：主会话单轮决策、`request_task_run` 任务入口、普通工具可见性、`agent_todo` 任务态绑定、单轮公开反馈、Prompt cache 动态/稳定分区
明确不包含：权限模式重设、用户权限配置调整、旧链路兼容保留

---

## 1. 结论

当前问题不是单纯模型不听提示，也不是权限设计问题，而是任务入口链路中存在三处权威错位：

1. Prompt 已要求任务承接时选择 `request_task_run`，但 output contract 仍允许模型把控制判断写成普通正文。
2. 单轮任务入口前仍暴露普通工具，模型会先调用 `agent_todo`、`search_files`、`glob_paths` 等工具，导致任务入口被工具调用消耗。
3. `request_task_run` 示例和公开反馈字段含有协议化句子，模型容易把“应进入任务”这种内部控制判断展示给用户。

成熟 agent 的目标行为应是：

```text
用户已经说“开始实施 / 启动 / 继续吧”
-> 模型选择一个结构化控制动作 request_task_run
-> 系统创建 TaskRun 并返回任务启动观察
-> 任务执行循环内再读取文件、维护 todo、调用工具、验证和收口
```

不允许出现：

```text
用户说启动
-> 模型先普通 tool_call
-> 模型再解释“按规则应进入 request_task_run”
-> 用户看到协议词或被要求二次确认
```

本修复应把 `request_task_run` 从“提示词建议”升级为“动作面约束”，并把任务态工具从单轮普通工具面移出。

## 2. 事故证据

### 2.1 二次确认问题

会话文件：`storage/sessions/session-a33ea744f5b34bd1.json`

- 用户在 `turn:session-a33ea744f5b34bd1:16` 说：`开始实施`。
- 模型已经判断这是明确任务承接请求，并在推理中承认应选择 `request_task_run`。
- 模型随后先调用了 `agent_todo`。
- `agent_todo` 返回后，模型意识到普通工具和控制动作不能混合，于是生成“请确认”话术。

这说明二次确认不是系统要求，而是模型在错误动作序列后自我修补出来的聊天内容。

### 2.2 协议泄露问题

截图中出现：

截图中出现了把内部动作选择和任务态判断直接写给用户的句子。

问题有两层：

- 第一层：模型把“应该提交控制动作”的判断写给了用户，而没有提交控制动作。
- 第二层：公开反馈字段过于协议化，用户看到的是系统判断，不是任务运行状态。

这类内容不应出现在用户正文中。用户需要看到的是任务已进入、当前阶段、下一步和可验收进展，而不是 action type、协议判断或内部规则。

### 2.3 第二次启动仍未正确进入任务

用户在后续 `turn:session-a33ea744f5b34bd1:18` 说：`启动`。

模型仍先调用 `search_files` 和 `glob_paths` 定位文件，随后把 `request_task_run` JSON 放进 Markdown 代码块正文，而不是作为系统可解析的控制动作提交。

这说明问题不止在 `agent_todo`，而在单轮动作面：任务承接请求出现时，普通工具和普通正文仍是可选出口。

## 3. 当前链路问题定位

### 3.1 Prompt 规则已经存在但权威不足

相关文件：`backend/harness/runtime/compiler.py`

当前规则已经表达：

```text
任务承接条件成立时必须选择 request_task_run；
不要用 respond 给计划替代启动任务；
也不要先用普通 tool_call 消耗任务。
```

问题是这仍然只是模型可读说明。只要 output contract 和 provider action surface 允许普通正文或普通工具，模型仍可能走错出口。

### 3.2 Output contract 允许普通正文

相关文件：`backend/harness/runtime/compiler.py`

当前 output contract 在 JSON action 非强制时存在“自由正文、native 工具、JSON action”并列出口。

这会让模型把控制动作解释成自然语言正文。对于有控制动作的单轮主会话，这个格式过宽。

目标行为：

```text
只要本轮允许 request_task_run / active_work_control / ask_user / block 等控制裁决，
模型必须输出一个结构化 action。
respond 也应是结构化 action，由运行时投影 final_answer 给用户。
```

### 3.3 单轮仍暴露任务态工具

相关文件：

- `backend/permissions/model_visible_operations.py`
- `backend/permissions/operations.py`
- `backend/harness/runtime/tool_plan.py`
- `backend/harness/loop/admission.py`

`agent_todo` 当前是 model-visible state operation，并且是非只读状态写入工具。它的描述是维护当前任务可见执行 todo 状态。

这意味着它天然属于任务运行态，不应在单轮未进入 TaskRun 时作为普通工具暴露。

目标行为：

```text
single_agent_turn 未进入 TaskRun：
  不暴露 agent_todo
  不允许模型用 agent_todo 建立计划来替代 request_task_run

task_execution 已进入 TaskRun：
  暴露 agent_todo
  自动绑定 session_id / task_run_id
```

### 3.4 示例文本污染公开反馈

相关文件：`backend/harness/loop/single_agent_turn.py`

当前 request_task_run 示例包含低信号的系统调度句式，模型容易照抄到用户正文。

目标行为：

```text
示例只展示字段结构，不提供可照抄的协议化正文。
公开反馈必须来自用户目标和当前阶段，例如：
  “已进入任务，我会先读取 mario.html 的当前结构，再实施第 1 批功能。”
```

但更重要的是：这些文本只能作为 `request_task_run` action 的字段，由运行时投影，不应变成普通 respond 正文。

## 4. 成熟链路标准

目标权威链路如下：

```text
RequestFacts
-> RuntimeBoundary
-> ContextCandidates
-> ModelTurnDecision
-> ActionPermit
-> RuntimeStartPacket
-> ExecutionLoop
-> PublicProjection
```

各层职责：

| 层级 | 当前责任 | 不允许做的事 |
| --- | --- | --- |
| RequestFacts | 记录用户话语、附件、当前会话事实 | 不判断是否启动任务，不选择工具 |
| RuntimeBoundary | 给出本轮允许的动作类型和工具边界 | 不替模型改写目标，不用旧链路兜底 |
| ContextCandidates | 提供历史、记忆候选、文件候选、工具目录 | 不把候选上下文写成任务裁决 |
| ModelTurnDecision | 模型选择 `respond / ask_user / tool_call / request_task_run / active_work_control / block` | 不把 action type 写给用户，不调用未授权工具 |
| ActionPermit | 验证动作是否在本轮边界内 | 不把被拒动作改写成聊天确认 |
| RuntimeStartPacket | 组装进入 TaskRun 的合同、工具、技能和动态上下文 | 不重新推断用户意图 |
| ExecutionLoop | 执行工具、接收观察、推进任务 | 不在未收口时刷新成最终显示 |
| PublicProjection | 投影用户需要知道的进展、结果、风险 | 不展示协议字段、内部状态词、工具噪声 |

本次修复重点是 `ModelTurnDecision -> ActionPermit -> RuntimeStartPacket` 的入口收束。

## 5. Prompt 与缓存分区规则

### 5.1 稳定区只能放长期规则

稳定 prompt cache 可包含：

- agent 角色和沟通原则
- 动作协议的抽象说明
- 工具调用基本规则
- `request_task_run` 字段结构
- 用户可见反馈质量标准

稳定区不能包含：

- 本轮用户是否说了“启动”
- 当前 active work 是否存在
- 本轮哪些工具被过滤
- 当前 session/task id
- 当前文件候选、工具观察、审批状态
- 本轮是否应优先 `request_task_run`

### 5.2 动态区必须承载 active 内容

动态 runtime projection 必须包含：

- latest_user_request facts
- task entry intent facts
- `request_task_run_allowed`
- active_work_context presence
- visible tool count
- filtered task-scoped tool routes
- tool boundary reason
- current turn output contract

特别要求：

```text
“用户已经说开始/启动/继续吧，因此不需要二次确认”
```

这是当前轮 active 判断，只能放动态区或由模型根据当前用户话语判断，不能写入稳定 prompt cache。

### 5.3 Prompt 文案调整原则

需要删除或替换的表达：低信号的系统调度句式、要求用户二次确认的任务入口话术、以及把内部 action 名直接暴露给用户的句子。

目标表达：

```text
当用户已经明确要求开始、启动、继续实施时，这已经是任务启动确认。
如果目标和验收标准足够，直接提交 request_task_run。
不要用普通正文解释你应该提交 request_task_run。
不要把 action_type、协议字段或内部规则写给用户。
```

## 6. 目标行为

### 6.1 用户明确启动任务

输入：

```text
开始实施
启动
继续吧
按这个计划做
```

当已有计划、目标和验收标准足以形成任务合同时：

```text
模型输出一个 request_task_run action
系统创建 TaskRun
用户看到自然语言 opening judgment
任务流显示继续保持执行态
```

不得：

- 先调用 `agent_todo`
- 先调用搜索工具
- 用 Markdown 输出 JSON
- 询问“请确认”
- 输出“按规则应进入 request_task_run”

### 6.2 信息不足

当目标、范围或验收标准不足：

```text
模型输出 ask_user action
```

不是：

```text
先用工具猜范围
先用 todo 填计划
把缺口包装成 request_task_run
```

### 6.3 只是一次性查询

如果用户只是要求一次性检查、解释、状态说明，且不要求持续交付：

```text
允许 respond 或只读 tool_call
```

但如果用户要求修改、落地、验证、生成真实产物，则必须进入任务入口。

## 7. 文件级修复计划

文件：

## 12. 成功标准

修复完成后，以下链路必须成立：

```text
用户：开始实施
模型：request_task_run action
系统：创建 TaskRun
界面：显示任务流启动和第一阶段进展
工具：在 TaskRun 内读取、编辑、验证
收口：只有正式 final_answer / closeout_summary 才刷新为最终显示
```

同时以下链路必须不可达：

```text
用户：启动
模型：先 agent_todo
模型：先 search_files / glob_paths
模型：把 request_task_run JSON 写进 Markdown
模型：说“按规则应进入 request_task_run”
模型：要求用户二次确认
```

这才符合成熟 agent 的操作模式：模型负责裁决，系统负责承接和执行；prompt 指导模型操作系统，但不能让协议说明替代真实动作。

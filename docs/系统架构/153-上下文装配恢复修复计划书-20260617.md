# 上下文装配恢复修复计划书

日期：2026-06-17  
状态：待用户审阅，未实施  
范围：Graph / task execution 上下文装配、DynamicContextManager、TaskStateProjector、RuntimeCompiler task_execution packet、file_state 运行中证据来源  
不在范围：Chronological Projection / 公开投影恢复、前端 UI 重构、模型供应商替换、任务图语义重写、权限系统整体重做、通过增加 fallback 或兼容旧链路掩盖装配问题

> 2026-06-17 补充：本计划中的“投影”只指模型上下文装配里的 `DynamicContextProjection` / `TaskStateProjection`。回档前的 Chronological Projection / 公开投影链路不是坏链路，不能在本计划实施时被删除、降级或回滚。公开投影恢复应按独立计划处理。

## 1. 目的

当前系统在图写作任务和长任务执行中暴露出上下文装配不稳定问题。用户要求对照 2026-06-05 中午档案，恢复当时更稳的上下文装配方式。该恢复只处理模型上下文装配，不处理公开投影。公开投影已在回档前形成更成熟的 Chronological Projection 主线，应优先恢复并保护。

本计划的目标不是全量回滚仓库到 2026-06-05，而是恢复该时间点的上下文装配语义：

```text
task_executor
-> assemble_runtime
-> RuntimeCompiler.compile_task_execution_packet
-> DynamicContextManager.project
-> TaskStateProjector.project
-> _model_messages_and_segment_plan
-> model_messages
```

恢复后的主原则：

- 运行边界由 `assemble_runtime` 与 compiler 明确装配。
- 动态上下文只做投影，不重新决定任务语义。
- task execution 当前状态由一份完整 `volatile_task_state` 承载。
- file_state 的运行中事实必须能从当前 task observations 恢复，不能只依赖持久 store。
- 删除或降级当前引入的重复上下文权威，不保留无用途兼容旧链路。


当时的 task execution message 顺序：

- `tool_result_refs` 会保留 `tool_result_ref` 或 `replacement_ref`，便于长工具结果压缩后追踪。
- `_model_messages_and_segment_plan(..., enforce_dynamic_context_reports=True)` 会检查动态段 metadata，保证 volatile 段可审计。

## 3. 当前偏离与风险

### 3.1 file_state 权威偏移

当前版本中，`DynamicContextManager._with_file_state_authority_projection` 不再从 observations 重建 file_state，而是合并 execution projection 中的持久 file_state 与 editor context。

风险：

- 工具执行层写入 `FileStateAuthorityStore` 失败时，模型看不到读窗状态。
- `runtime_host.file_state_store` 与 `dynamic_context_storage_root(runtime_assembly)` 不一致时，store 和 compiler 读取不同根目录。
- editor context 属于用户界面快照，只能作为线索，不能替代工具 observation 形成的运行中事实。
- 当前 task observations 已有读写证据时，持久 store 不应覆盖它。

恢复原则：

```text
当前 task observations
-> 运行中 file_state 权威
-> 持久 file_state_store 作为补充
-> editor_context 作为提示线索，不授予事实权威
```

### 3.2 task_state 被拆成 replay + cursor

当前版本新增：

```text
TaskStateProjector.split_for_prompt_cache
-> task_state_replay_entries
-> task_state_cursor
-> compiler 插入多条 task_state_replay_entry message
-> volatile_task_state 只保留 cursor
```

风险：

- 上下文状态不再是一份完整事实，而是拆成历史 replay 和当前 cursor。
- replay entry、cursor、latest_tool_results 可能表达同一件事的不同版本。
- prompt cache 优化侵入了语义装配层，把“缓存形状”变成了“上下文语义”。
- 模型可能看到历史证据在 replay 中、当前状态在 cursor 中，两者不同步。

恢复原则：

```text
prompt cache 只能影响段落 cache 策略
不能改变 task_state 的语义结构
```

### 3.3 bound_task_context 二次绑定

当前 compiler 在 task_execution packet 中新增：

```text
bound_task_context_stable
bound_task_runtime_context
```

该层从 contract、planning_protocol、dynamic_context、task_state_projection 中重新抽取 plan_refs、known_task_files、edit_targets、artifact_refs、rehydration_refs。

风险：

- 它不是单纯 assemble，而是在 compiler 内再次整理任务文件、编辑目标和恢复策略。
- 它可能成为第二个上下文事实来源，与 `volatile_task_state`、tool observations、file_state_store 并行。
- 对模型可见时，会让上下文装配链路从“一份 task_state”变成“两份语义投影”。

恢复原则：

```text
compiler 只装配，不二次解释任务状态
```

如果确需保留该结构，只允许作为 diagnostics / manifest，不作为 task_execution 常规模型可见上下文。

### 3.4 read_evidence_injection 权威重叠

当前 compiler 新增 `read_evidence_injection`，从 file_state 和 read observation artifact store 注入精确读窗证据。

风险：

- 如果 file_state 来源已不稳定，该层会把不稳定状态进一步放大。
- 如果它与 tool observation 中的 read result 重叠，模型会收到两套读证据。
- 如果 storage root 不一致，它可能读不到真实 artifact，导致模型误判自己没有读过。

恢复原则：

```text
精确读证据应首先由 tool observation / latest_tool_results 进入 task_state
额外 read evidence injection 只能在证明无重复、无 root 分裂后保留
```

### 3.5 context_refs / replacement_refs 弱化

当前版本移除了 `replacement_history_ref`，并且 `tool_result_refs` 不再保留 `replacement_ref`。

风险：

- 长历史被 omitted 后，packet 报告缺少可追踪替换引用。
- 长工具结果被 replacement store 压缩后，模型可见报告和 runtime trace 的连接变弱。

恢复原则：

```text
被压缩或 omitted 的上下文必须有 ref
不能只靠可见摘要维持追踪性
```

## 4. 目标权威链

恢复后的目标链路：

```text
TaskRun / Contract / Observations
-> RuntimeAssembly
-> ToolPlan / Permission Boundary
-> DynamicContextProjection
-> TaskStateProjection
-> RuntimeCompilationPacket
-> ExecutionLoop
```

权责表：

| 层 | 目标职责 | 禁止事项 |
| --- | --- | --- |
| `task_executor` | 观察 task_run、contract、observations，调度 compiler 与执行循环 | 不重写 task 语义，不在执行循环中拼接第二套 prompt |
| `assemble_runtime` | 决定 runtime 边界、profile、environment、permission、tool visibility | 不读取动态任务事实，不决定当前步骤语义 |
| `DynamicContextManager` | 投影 runtime delta、history、observations、execution state、work history | 不做 prompt cache 语义拆分，不发明任务状态 |
| `TaskStateProjector` | 从 execution/observation/work history 白名单投影当前 task_state | 不做 replay/cursor 拆分，不二次裁决任务目标 |
| `RuntimeCompiler` | 把稳定 prompt、runtime boundary、dynamic context、volatile task_state 装成 packet | 不重新绑定 edit targets、known files、restore policy 作为第二上下文 |
| `ExecutionLoop` | 执行模型请求、记录 observation、进入下一轮 | 不靠旧状态猜测缺失上下文 |

## 5. 修复切片

### 5.1 切片一：恢复 DynamicContextManager 装配语义

涉及文件：

- `backend/harness/runtime/dynamic_context/manager.py`
- `backend/harness/runtime/dynamic_context/models.py`

修改内容：

1. 恢复 `build_file_state_projection_from_observations` 导入和使用。
2. 恢复 `replacement_history_ref` 导入和 `context_refs` 注入。
3. `_with_file_state_authority_projection` 改回以 observations 为运行中事实来源。
4. 如果 persisted file_state 仍保留，必须作为补充合并，不能覆盖当前 observations。
5. `_volatile_state_projection` 恢复返回完整 `{"task_state": task_state}`，不返回 replay entries。
6. 删除或停止填充 `task_state_replay_entries`。
7. `tool_result_refs` 恢复 `tool_result_ref or replacement_ref`。
8. `_section_reports` 删除 `task_state_replay_prefix` 报告。

验收标准：

- task execution projection 中有完整 `volatile_state_projection.task_state`。
- observations 中有 file read/write event 时，即使 file_state_store 为空，也能生成 file_state。
- omitted history 有 replacement ref。
- projection report 不再暴露 replay prefix 作为上下文语义。

### 5.2 切片二：收敛 TaskStateProjector

2. 保留白名单 task_state 字段：
   - `runtime_status`

验收标准：

- TaskStateProjector 不再包含 prompt-cache replay 逻辑。
- task_state 是当前状态投影，不是缓存优化产物。
- 同一工具结果不会同时以 replay entry 和 cursor result 两种语义进入模型。

### 5.3 切片三：恢复 RuntimeCompiler task_execution packet 顺序

涉及文件：

### 5.4 切片四：file_state store 降级为持久补充

修改内容：

3. `task_executor._observations_for_packet` 可继续读取 store，但 DynamicContextManager 必须能从 observations 重建。
4. 如果 observation file_state 与 store 冲突，以当前 observations 形成的运行中事实优先。
5. 审查 `runtime_state_root` 来源，确保 runtime_host root、runtime_assembly storage root、compiler read evidence root 不分裂。

验收标准：

## 8. 风险与停止条件

### 8.1 风险

1. 当前某些新测试可能只保护 replay/cursor 或 bound_task_context 形态，恢复时需要删除或重写。
2. 当前 read evidence injection 可能被某些 edit/read guard 依赖，删除前必须确认真实读写权限链路仍由 tool validation 和 observations 保护。
3. file_state_store 已被多个模块使用，不能直接删除；应降级为补充来源。
4. 如果 graph writing task 的 prompt 依赖当前新增字段，需要把真实需要的内容迁回 `task_prompt_contract` 或 `volatile_task_state`，不能保留双链路。

### 8.2 停止条件

实施中遇到以下情况必须暂停并重新确认：

- 发现 `bound_task_context` 已成为任务图节点执行的唯一目标来源。
- 发现 `read_evidence_injection` 承担了权限授权，而不是证据展示。
- 删除 replay/cursor 后，compiler 无法形成稳定 segment plan。
- 测试暴露出当前任务恢复、写作图执行、工具授权三者存在更大的结构冲突。

## 9. 预期结果

修复完成后，task execution 的上下文装配应回到成熟单向链路：

```text
runtime boundary 是边界
dynamic projection 是投影
volatile_task_state 是当前事实
tool observations 是证据
compiler 只装配 packet
executor 只执行与记录
```

模型不再同时面对 replay entry、cursor、bound runtime context、read evidence injection 多套上下文事实。上下文装配会更接近 2026-06-05 中午档案的稳定形态，同时保留当前系统中确实有价值的持久 file_state store，但不让它成为唯一事实来源。

本计划经用户确认后方可实施。

# 105-主会话投影前端彻底重构设计书

日期：2026-06-11

## 1. 问题定性

当前问题不是单个 UI bug，而是主会话公开投影链路没有形成“每一轮 turn 的唯一投影权威”。

成熟 agent 的显示链路应当很简单：

```text
用户输入
-> 创建本轮 user message 和本轮 assistant projection container
-> 模型输出正文或结构化动作
-> 正文进入 assistant body
-> 工具动作进入 tool window
-> 工具结果更新同一个 tool window，并作为 observation 回灌模型
-> 模型基于 observation 继续输出正文或下一步动作
-> done 只收口，不制造正文
```

现在错位和泄露的根因是：前端没有把后端状态机输出当作唯一权威，而是在消息、monitor、runtime attachment、stageStatus、public timeline、done summary 之间多次猜测归属和语义。

必须按 P0 结构问题处理，不再靠“再加一个清洗词”修补。

## 2. 当前故障链路

### 2.1 工具为什么会飞到用户消息上面

当前活跃任务 steer 路径会出现：

```text
startQueuedActiveTurn
-> 只创建 user message
-> assistantId 为空
-> 后端 tool/status projection 到达
-> 前端按 turn_id / task_run_id 找旧 assistant
-> 旧 assistant 在新 user 上方
-> 工具卡片显示到用户消息上方
```

涉及文件：

- `frontend/src/lib/store/events.ts`
  - `startQueuedActiveTurn()` 当前只有 user message，没有本轮 assistant projection container。
  - `reduceStreamEvent()` 接到 projection 后依赖 `assistantId` 和 anchor 归属。
- `frontend/src/lib/projection/reducer.ts`
  - `projectionMessageIndex()` 会先找已有匹配 assistant；同一个后台 task 跨多次 steer 时，旧 assistant 可能抢走新事件。
- `frontend/src/lib/store/utils.ts`
  - `syntheticAssistantMessagesForRuntimeAttachments()` 在找不到明确 user/assistant 时，会按 turn id 尾号猜 `sourceIndex`，这是错位来源。

### 2.2 内部状态为什么会泄露

当前存在多条把内部状态变成公开文本的路径：

- `backend/harness/runtime/projection/projector.py`
  - `done` 事件曾把 `summary / receipt_summary` 转成 `observation_report`，等价于把系统 done 摘要升格为模型正文。
- `backend/harness/runtime/public_progress.py`
  - 曾把 `single_turn_tool_iteration_limit` 改写成“工具检查次数达到边界”这种用户可见文案。内部 reason/code 不应公开。
- `frontend/src/components/chat/ChatMessage.tsx`
  - control item 的 detail/text 曾被拼进 assistant display content。
  - `stageStatus` 可在空正文时直接显示，形成 legacy 状态泄露。
- `frontend/src/components/chat/PublicTimelineActivity.tsx`
  - 既从 public timeline 读 body，又从 task projection final answer、activity、status 推断展示；同一组件承担了正文、工具、状态、折叠和过滤多重职责。

### 2.3 实时和回放为什么不一致

当前实时 SSE 和持久化回放不是同一权威：

- 实时：`public_projection_envelope` 进入 `applyPublicProjectionEnvelope()`。
- monitor：`patchRuntimeAttachmentFromMonitor()` 可直接改 assistant runtime attachment。
- runtime event：`patchRuntimeAttachmentFromRuntimeEvent()` 可绕过同一 reducer。
- 历史回放：`toUiMessages()` 会用 runtime attachments 合成 assistant。

这导致同一工具事件在实时、刷新、恢复后三种路径可能挂到不同消息上。

## 3. 成熟 Agent 的用户显示标准

用户应该看到三类内容，且永不混用。

| 类型 | 来源 | 用户看到 | 绝不允许 |
|---|---|---|---|
| 模型正文 | `source_authority=model` + `surface=assistant_body` + `slot=body` | 开局判断、阶段总结、最终回答 | 系统状态、工具结果、terminal_reason 冒充正文 |
| 工具窗口 | `slot=tool` 或 `surface=tool_window` | 工具名、目标、运行中/完成/失败、简短结果 | 原始 JSON、内部工具协议、控制命令 |
| 控制/状态 | `slot=control/status` | 等待确认、暂停请求、失败提示等 UI 状态 | 拼进 assistant 正文；显示内部 code/reason |

关键原则：

1. 没有模型正文时，可以只显示工具窗口，不补“正在思考/开始处理/处理完成”。
2. 工具调用由模型动作触发，但工具执行是系统行为；显示为工具窗口。
3. 工具结果先回灌模型；模型之后的自然语言反馈才是正文。
4. `done` 只结束本轮，不生成正文。
5. 内部 reason/code/prompt/control 字段只给模型和诊断链，不给用户。

## 4. 目标架构

### 4.1 单一权威链

```text
Backend Runtime State Machine
-> PublicProjectionEnvelope Event Log
-> Frontend ProjectionReducer
-> TurnProjectionStore
-> ProjectionTurn Renderer
```

允许做决定的层：

| 层 | 允许决定 | 不允许决定 |
|---|---|---|
| Backend state machine | turn 状态、工具执行、pause/stop/approval、terminal | 用户正文 |
| Model | assistant body、阶段判断、最终回答、下一步动作 | 工具执行结果、系统状态 |
| PublicProjectionProjector | 把 runtime/model/tool 事件转成 typed public envelope | 翻译内部 code 给用户；把 done summary 造正文 |
| Frontend ProjectionReducer | 按 anchor 把 envelope 放进本轮 assistant container | 猜 turn、猜正文、猜 assistant |
| Projection Renderer | 按 slot/surface/source 展示 | 根据中文文案判断控制/正文 |

### 4.2 TurnProjectionSession

每一轮用户输入必须有明确 projection session：

```ts
type TurnProjectionSession = {
  sessionId: string;
  turnId?: string;
  turnRunId?: string;
  taskRunId?: string;
  userMessageId: string;
  assistantMessageId: string;
  state: "opening" | "model_streaming" | "tool_running" | "waiting_model" | "waiting_user" | "paused" | "stopped" | "done" | "error";
  latestSequence: number;
};
```

硬规则：

1. 每个 user message 后必须同步创建一个 assistant projection container，即使当前模型还没有正文。
2. queued/steer 输入也必须创建 assistant container，不能只放 user。
3. projection 只能挂到 `assistantMessageId`；没有明确映射就不公开显示。
4. 同一 task_run 跨多次 steer 时，当前 turn 的 assistant 优先；旧 assistant 不得抢占。

### 4.3 PublicProjectionEnvelope 合同

后端必须输出稳定 envelope：

```ts
type PublicProjectionEnvelope = {
  authority: "harness.public_projection";
  contract_revision: string;
  projection_id: string;
  sequence: number;
  event_offset: number;
  session_id: string;
  anchor: {
    turn_id: string;
    message_id?: string;
    task_run_id?: string;
    run_id?: string;
    turn_run_id?: string;
    anchor_role: "assistant";
  };
  lifecycle: "running" | "waiting" | "done" | "error" | "stopped";
  source_authority: "model" | "tool" | "runtime" | "system";
  surface: "assistant_body" | "tool_window" | "control" | "timeline" | "diagnostics";
  items: PublicProjectionItem[];
};
```

Envelope fail-closed 规则：

1. 没有 anchor 的 authoritative envelope 不进入公开消息。
2. `slot=body` 必须同时满足：
   - `source_authority=model`
   - `surface=assistant_body`
   - 有明确文本字段
3. `done/error/stopped` 不能自动生成 body。
4. `terminal_reason/code/internal_status` 只能进入 diagnostics 或模型 observation，不进入 public item 文案。
5. `control/status` item 必须有明确 public label；没有 public label 就只更新状态机，不显示。

## 5. 固定执行流

### 5.1 普通模型正文

```text
user submitted
-> frontend creates user + assistant container
-> backend creates turn/run anchors
-> model streams body
-> backend emits assistant_text_delta/final with body authority
-> frontend appends body into assistant container
```

显示：

```text
User
Assistant body streamed...
```

### 5.2 模型先调用工具

```text
user submitted
-> assistant container exists
-> model emits tool_call action
-> runtime admits action
-> backend emits tool_window item
-> frontend shows tool card under current assistant
-> tool result updates same card
-> observation goes back to model
-> model emits body
```

显示：

```text
User
Tool card: 读取文件 running -> done
Assistant body: 我定位到问题在...
```

### 5.3 模型先正文后工具

```text
model body delta
-> tool_call action
-> tool window
-> observation
-> next model body
```

显示顺序必须按 event sequence：

```text
Assistant body: 我先检查...
Tool card
Assistant body: 检查结果是...
```

### 5.4 多工具和阶段总结

工具反馈保持短。

阶段总结只由模型 body 或明确 `stage_summary` 产生。

工具折叠只允许由 `covers_tool_refs` 驱动：

```text
Tool A
Tool B
Stage summary body covers [Tool A, Tool B]
-> Tool A/B collapsed
```

不允许前端因为“后面出现了正文”就猜测折叠。

### 5.5 Pause / Stop / Approval

```text
user presses pause
-> control recorded immediately
-> frontend control lane shows request accepted if public
-> executor stops at safe boundary
-> backend emits paused/stopped state
-> no assistant body unless model later explains
```

暂停文案不能声称“已暂停”，除非 executor 已进入 paused/stopped。

### 5.6 Error / Boundary

内部错误 code 不能直接公开。

正确路径：

```text
runtime error code
-> diagnostics/internal observation
-> model receives observation
-> model produces user-facing explanation
```

如果必须由系统直接公开错误，必须是显式 public error item：

```ts
{
  slot: "control",
  surface: "control",
  source_authority: "runtime",
  public_title: "当前步骤没有完成",
  public_detail: "可以重试，或补充新的核查方向。"
}
```

不允许显示：

```text
single_turn_tool_iteration_limit
工具检查次数达到边界
terminal_reason
runtime_control_boundary
```

## 6. 前端重构设计

### 6.1 删除旧判断权威

必须移除或降权：

| 旧结构 | 问题 | 处理 |
|---|---|---|
| `stageStatus` 作为空正文 fallback | 状态词泄露 | 不再渲染为 assistant 内容 |
| `askUserQuestionFromPublicTimelineItems()` 拼 control detail | control 进正文 | 删除 |
| `taskProjection.final_answer` 直接当 body | runtime projection 冒充模型正文 | 只显示显式 model body |
| `latestModelBodyIndex` 驱动工具折叠 | 前端猜阶段边界 | 改为 `covers_tool_refs` |
| `syntheticAssistantMessagesForRuntimeAttachments()` 猜 sourceIndex | 历史错位 | 无明确 user anchor 不合成 |
| `patchRuntimeAttachmentFromMonitor()` 直接挂消息 | monitor 绕过 projection reducer | 改为 envelope path |
| 文案清洗词表当主边界 | 只能挡已知词 | 主边界改为结构字段 |

### 6.2 新前端模块

目标文件结构：

```text
frontend/src/lib/projection/
  contract.ts              # typed slot/surface/source guard
  turnReducer.ts           # 唯一 projection reducer
  turnStore.ts             # assistantId/turnId/taskRunId 映射
  viewModel.ts             # body/tool/control 分类

frontend/src/components/chat/
  ChatMessage.tsx          # 只渲染 user/assistant shell
  ProjectionTurn.tsx       # 本轮 assistant 投影
  ProjectionBody.tsx       # 模型正文
  ProjectionActivity.tsx   # 工具/控制/状态
  ProjectionToolWindow.tsx # 工具窗口
```

不新增 V2 文件夹；旧文件直接替换或删除。

### 6.3 ProjectionReducer 规则

Reducer 输入只有 `PublicProjectionEnvelope`。

```text
apply(envelope):
  validate authority
  validate anchor
  resolve assistantMessageId
  validate item slots
  order by sequence/event_offset
  merge by item_id/tool_call_id
  store under assistantMessageId
```

归属优先级：

1. `anchor.message_id` 精确匹配 assistant。
2. 当前 stream session 的 assistantId，且 envelope anchor 与当前 stream anchor 匹配。
3. `turn_id / turn_run_id / task_run_id` 反向查找最新 assistant。
4. 无匹配则丢到 diagnostics，不公开显示。

禁止：

1. 用第一个匹配 assistant。
2. 用 task_run_id 跨 turn 抢占当前 turn。
3. 无 anchor 时挂到最新 assistant。
4. 历史恢复时凭 turn id 尾号猜 sourceIndex。

### 6.4 Projection Renderer 规则

渲染层不做语义推断，只按结构显示。

```text
body items:
  slot=body
  surface=assistant_body
  source_authority=model

tool items:
  slot=tool or surface=tool_window
  source_authority=tool/runtime

control items:
  slot=control/status
  surface=control/timeline/status_bar
  source_authority=runtime/system
  visible only if explicit public label exists

diagnostics:
  never shown in main chat
```

工具窗口显示：

- 标题：动作 + 目标。
- 状态：running/done/error/waiting/stopped。
- 简短结果：只显示 public summary / observation summary。
- 原始输出：默认隐藏；没有明确用户价值时不展示。
- 失败：显示用户可理解失败摘要，不显示内部 code。

### 6.5 每轮消息容器

`startStreamingTurn()` 和 `startQueuedActiveTurn()` 都必须创建：

```text
user message
assistant projection container
```

assistant container 初始：

```ts
{
  role: "assistant",
  content: "",
  runtimeAttachments: [],
  runtimePublicTimelineDraft: [],
  sourceIndex: user.sourceIndex + 1,
  sourceTurnId?: currentTurnId
}
```

这个 container 可以视觉上为空，但必须存在，保证后续工具投影有归属。

## 7. 后端重构设计

### 7.1 ProjectionProjector

后端 projector 必须只做结构映射：

| 输入事件 | 输出 |
|---|---|
| assistant text | body item |
| model tool action | tool item |
| tool observation | tool item update |
| model stage summary | body item with `covers_tool_refs` |
| runtime control | control item |
| done | terminal only，无 body |
| error/stopped | control/terminal only，无 body，除非有显式 public_error |

### 7.2 public_progress

`public_progress.py` 不再把内部 code 翻译成公开话术。

允许：

- 清理低价值泛化状态。
- 清理 raw tool output。
- 只保留已经明确是 public text 的内容。

禁止：

- `single_turn_tool_iteration_limit -> 工具检查次数达到边界`
- `terminal_reason -> 用户可见句子`
- `runtime_control_boundary -> 状态文案`

### 7.3 Prompt 配套

模型 prompt 需要明确：

```text
你输出给用户看的自然语言只能放在 assistant body。
当你决定调用工具时，不要把“正在处理/开始处理/工具调用完成”写成正文。
工具结果会作为 observation 回到你这里。
你需要基于 observation 给用户写真实判断、阶段总结或最终回答。
如果只是继续调用工具，不要产出空泛状态句。
阶段总结必须说明已经确认的事实、剩余问题和下一步。
```

工具观察回灌 prompt 需要明确：

```text
你刚收到工具观察。
请判断是否已经到阶段边界。
如果还需要工具，输出下一步 action。
如果已经形成判断，输出一段用户可见反馈。
不要复述工具原始输出，不要暴露内部字段或 terminal reason。
```

Action schema 需要保留：

- `public_action_state.current_judgment`
- `public_action_state.next_action`
- `stage_summary.text`
- `stage_summary.covers_tool_refs`

但这些字段只有在模型明确表达给用户时才能进入 body。

## 8. 持久化和恢复

正确恢复链：

```text
stored public_projection_events
-> rebuild TurnProjectionStore
-> rebuild messages by stored user/assistant ids
-> render same ProjectionTurn
```

禁止恢复链：

```text
runtime_attachments
-> guess sourceIndex
-> synthetic assistant
-> attach tool to guessed message
```

恢复规则：

1. 有 `anchor.message_id`：挂到该 assistant。
2. 有 `anchor.turn_id` 且 history 中存在该 turn 的 user/assistant pair：挂到 pair 的 assistant。
3. 只有 task_run_id：不能公开合成消息；放 diagnostics。
4. monitor 只可补充 envelope event，不可直接 patch chat message。

## 11. Cutover 规则

1. 不建 `V2` 文件夹。
2. 旧组件原地替换；无用旧文件删除。
3. 不保留旧 fallback 作为兼容。
4. 测试只保护目标行为，不保护旧内部形状。
5. 若出现不可恢复偏差，回滚整个阶段，不留下半套并行逻辑。

## 12. 为什么这个方案不会再错位

因为它去掉了错位的三个根源：

1. 每轮都有 assistant projection container，工具永远有当前归属。
2. projection 只按 explicit anchor/assistantId 归属，不按旧消息猜。
3. 实时、恢复、monitor 都收敛到同一个 projection event/reducer。

只要执行这三条，工具就没有机会“飞到上面”；只要正文只认 model body，系统状态就没有机会泄露成 assistant 回复。

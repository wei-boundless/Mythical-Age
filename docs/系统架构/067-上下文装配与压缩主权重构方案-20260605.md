# 067-上下文装配与压缩主权重构方案

日期：2026-06-05

## 1. 审查结论

当前上下文压力显示异常的根因不是前端百分比，而是后端上下文装配主权错位。

当前代码在压缩触发前已经静默裁剪历史：

- `backend/runtime/shared/history_assembler.py` 默认只保留最近 12 条 session message。
- `backend/harness/runtime/dynamic_context/history_projector.py` 默认只投影最近 6 条 history item。
- `backend/harness/runtime/compiler.py` 的 provider protocol replay 默认只保留 12 条协议消息，并且混入普通会话消息。
- `backend/runtime/context_management/session_compaction.py` 的自动 compact 判断主要依赖 `ContextUsageMeter.auto_replacement_allowed`，而 `ContextUsageMeter` 的 provider anchor 来自已经裁剪后的最终 prompt。

这会造成一个结构性错误：旧历史先被移出 prompt，随后压力计看到的自然一直很低，自动压缩无法按真实 active history 触发。

成熟 agent 的不变量是：

```text
active history 是会话连续性的权威
prompt projection 不能静默删除 active history
compact 是 active history replacement，不是最近 N 条裁剪
provider protocol tail 只服务工具协议配对和热工具观察，不服务会话记忆
```

## 2. 当前代码主链路

### 2.1 Session 账本

`backend/sessions/__init__.py` 当前有三类状态：

- `messages`：UI 可见用户/助手消息，也是 runtime 当前实际使用的 session history 来源。
- `api_transcript`：provider protocol / 工具协议 transcript。
- `compressed_context`：compact 后的会话 checkpoint summary。

问题：

- `messages` 具备 active history 权威条件，但进入 prompt 前被 `history_assembler` 和 `history_projector` 裁剪。
- `api_transcript` 应作为协议恢复来源，但当前会被 compiler 作为 provider protocol history tail 注入，容易和 active history 重叠。
- `append_messages()` 没有稳定写入 `created_at`，导致基于 anchor 时间的 pending 估算没有可靠消息时间。

### 2.2 Turn 前处理

`backend/harness/entrypoint/runtime_facade.py` 在模型调用前执行：

```text
auto_compact_session_if_needed
-> load_session_for_agent
-> assemble_runtime_history
-> RuntimeCompiler.compile_single_agent_turn_packet
```

问题：

- `auto_compact_session_if_needed` 使用的 context snapshot 基于最近一次模型 prompt usage。
- 最近一次模型 prompt usage 已经受历史裁剪影响。
- 后续 `assemble_runtime_history` 继续裁剪，导致 compact gate 和 prompt assembly 都看不到完整 active history。

### 2.3 Prompt 编译

`backend/harness/runtime/compiler.py` 的 single-agent turn 主要结构是：

```text
global_static
turn_stable
turn_context
provider_protocol_history
dynamic_projection
volatile_user
```

问题：

- 会话连续性主要藏在 `volatile_user` 的 dynamic history projection 中，但该 projection 会按 recent limit 裁剪。
- provider protocol replay 会把 `api_transcript` 的尾部作为消息 replay。它适合保留热 tool pair，不适合作为普通对话记忆。

### 2.4 Token 与压缩

`backend/runtime/prompt_accounting/context_usage_meter.py` 的职责应是解释最终模型请求的 token/cache 状态。

它不应单独决定 active session 是否需要 compact，因为它无法从已经裁剪的 provider prompt 反推出完整 active history。

`backend/context_system/compaction/compactor.py` 已具备可复用基础：

- token-aware `pressure_level`
- microcompact
- semantic compactor 注册校验
- full compact summary + recent real messages replacement
- compact invariant receipt

需要调整的是触发与装配权威，不是另起一套 compactor。

## 3. 目标设计

目标主链路：

```text
SessionManager.messages
-> ActiveHistoryAssembly
-> ActiveContextSnapshot
-> ContextCompactionGate
-> ContextCompactor replacement
-> RuntimeCompiler
-> ModelRuntime
-> PromptAccountingLedger
-> ContextUsageMeter
```

职责边界：

| 模块 | 目标职责 | 不允许做 |
| --- | --- | --- |
| `SessionManager` | 持久化 active messages、api transcript、compressed context | 不决定压缩 |
| `ActiveHistoryAssembly` | 规范化 active history，不裁剪 | 不按 recent N 丢弃历史 |
| `ContextCompactionGate` | 根据 active history 和 provider prompt 判断 compact | 不修改历史 |
| `ContextCompactor` | 执行 compact / microcompact / full replacement | 不读取 runtime 当前意图 |
| `RuntimeCompiler` | 装配模型输入 | 不静默隐藏 active history |
| `ProviderProtocolReplay` | 保留热工具协议和 reasoning tail | 不承担普通会话记忆 |
| `ContextUsageMeter` | 显示最终模型 prompt 压力和 cache | 不作为唯一 compact gate |

## 4. 代码级改造方案

### 4.1 恢复 active history 主权

修改：

- `backend/runtime/shared/history_assembler.py`

要求：

- 移除 `DEFAULT_RECENT_MESSAGE_LIMIT` 和 `recent_message_limit` 主链路语义。
- `assemble_runtime_history()` 返回完整规范化 active history。
- 诊断字段改为 active history 语义，不再报告正常路径的 dropped history。

### 4.2 修正 dynamic history projection

修改：

- `backend/harness/runtime/dynamic_context/history_projector.py`

要求：

- 不再默认 `recent_limit=6`。
- `recent_turns` 字段暂时保留为兼容字段名，但内容应是 compact 后 active history，而不是最近 N 条。
- 如果未来出现超预算，应由 compact gate 处理；projector 不能静默丢弃。

### 4.3 修正 auto compact gate

修改：

- `backend/runtime/context_management/session_compaction.py`

要求：

- `auto_compact_session_if_needed()` 同时检查：
  - provider prompt pressure；
  - active history token pressure。
- 如果 provider prompt 低但 active history 达到 `microcompact/full_compact`，仍应触发 compact。
- `ContextUsageMeter` 继续保留为 prompt meter，但不再独占 auto replacement 判断。

### 4.4 修正 provider protocol replay

修改：

- `backend/harness/runtime/compiler.py`

要求：

- provider protocol replay 只选择工具协议相关消息：
  - assistant tool_calls
  - tool result
  - 必要 reasoning_content
- 普通 user/assistant 对话由 active history projection 提供。
- 保留 tool call/result pairing 修复逻辑。

### 4.5 修正 session message 时间

修改：

- `backend/sessions/__init__.py`

要求：

- `append_messages()` 和 `append_api_messages()` 在消息缺少 `created_at` 时写入当前时间。
- 这样 `ContextUsageMeter` 的 pending 估算有真实时间依据。

## 5. 测试计划

必须修改或新增：

- `backend/tests/history_assembly_regression.py`
  - 删除最近 N 条裁剪断言。
  - 新增完整 active history 断言。

- `backend/tests/harness_runtime_facade_regression.py`
  - 新增：provider prompt usage 很低但 active history 很大时，auto compact 会触发。
  - 新增：compact 后 prompt 含 `compressed_context`，旧原文不进入 active prompt。

- `backend/tests/dynamic_prompt_context_projection_test.py`
  - 新增：history projector 不默认丢弃 active history。
  - 调整 provider protocol replay 期望：普通对话不再通过 provider protocol tail 进入。

- `backend/tests/prompt_context_usage_meter_regression.py`
  - 保留“latest provider usage 不是累计账单”的测试。
  - 新增 fingerprint/pending 场景可选，但不作为第一阶段必需。

- `frontend/src/components/chat/ChatPanel.test.ts`
  - 只保留当前展示测试；不扩大 UI。

## 6. 自检：遗漏与冲突

已检查的冲突点：

- 不删除 prompt accounting；它仍是 cache 和 provider usage 事实账本。
- 不新增硬编码 compactor subagent；现有 semantic compactor 仍要求外部注册。
- 不让 provider protocol replay 承担会话记忆；它只保留热工具协议。
- 不修改前端展示目标；前端继续显示当前上下文 token、压缩阈值、百分比。
- 不把累计 billing tokens 当上下文压力。
- 不保留 recent crop 作为兼容兜底；否则会继续掩盖真实上下文压力。

第一阶段实施完成标准：

- active history 不再被 `history_assembler` 或 `history_projector` 默认裁剪。
- auto compact 可以被 active history token pressure 触发。
- provider protocol replay 不再重复普通会话消息。
- focused backend tests 通过。

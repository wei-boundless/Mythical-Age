# 139-Turn 边界公开投影路由修复记录

日期：2026-06-15

## 1. 本次修复的问题

同一轮 turn 的 `public_projection_frame` 已经由后端标记为 `visible_live / visible_final`，但前端把 `mainChatSurface=body_only` 理解成“只显示正文，不显示投影时间线”，导致工具调用、读取、搜索、模型阶段反馈没有进入主聊天可见投影，只剩“正在思考”或只有正文。

这是错接线：`body_only` 只能说明该回合不是任务 live/closed 流程，不能作为 public projection 的过滤权威。

更正：turn 不是显示模式。turn 是用户输入到 assistant 输出收口的一轮运行边界。正文、工具调用、工具返回、最终回答都必须按同一个 `turn_id/message_id` 归到同一条 assistant 消息；不能因为 surface 名称再复制一条消息。

## 2. 当前信号接线

| 信号/字段 | 权威含义 | 前端线路 |
| --- | --- | --- |
| `public_projection_frame.main_visibility=visible_live/visible_final/pinned` | 允许进入主聊天公开投影 | 进入 assistant message 的 `publicProjection` |
| `public_projection_frame.main_visibility=hidden` | 内部诊断或不可见事件 | 只进 trace，不进主窗口 |
| `public_projection_frame.main_visibility=trace_only` | 生命周期证据/日志 | 只能更新已存在工具窗口或进入 trace，不能凭空创建主窗口 |
| `mainChatSurface=body_only` | 该 turn 无任务收口表面 | 不再过滤投影时间线，也不能创建第二条 assistant |
| `mainChatSurface=live_timeline` | 任务运行中 | 显示正文 + 工具窗口时间线 |
| `mainChatSurface=closeout_summary` | 任务已收口 | 主聊天只显示正文/收口总结 + 日志入口，不显示工具窗口 |
| `mainChatSurface=log_only` | 仅日志/trace | 不进主聊天 |

## 2.1 消息身份规则

| 阶段 | 允许做什么 | 禁止 |
| --- | --- | --- |
| 实时 SSE | 可把投影挂到当前 live assistant；当前 live assistant 不存在时才可创建临时 assistant | 禁止用工具生命周期另开 assistant |
| 历史 hydrate | 只能把投影补到已有 assistant message | 禁止根据 projection frame 创建第二条 assistant |
| 收口后 | 主聊天只显示正文/收口摘要和日志入口 | 禁止把任务工具窗口留在主聊天 |

## 3. 已改代码

| 文件 | 修改 |
| --- | --- |
| `frontend/src/components/chat/ChatMessage.tsx` | `body_only` 不再关闭投影时间线；只有 `closeout_summary/log_only` 关闭主聊天工具窗口。 |
| `frontend/src/lib/store/runtime.ts` | 历史 hydrate 接受该 turn 的 public frames；只跳过 `log_only/closeout_summary`。 |
| `frontend/src/lib/projection/reducer.ts` | `hidden` 工具诊断禁止进主 timeline；`trace_only` 工具生命周期只允许更新已有工具窗口；历史重放禁止创建第二条 assistant。 |

## 4. 目标效果

同一 turn 内都按同一个 public projection 时序显示：

```text
model_action.* runtime_step_summary -> 后端直接投成 body_append -> 正文区
普通 runtime_status / 非模型 runtime_step_summary -> status/trace，不让前端猜正文
tool_call_requested -> 创建一个工具窗口
tool_permission_decided -> 更新同一窗口或 trace
tool_item_started -> 更新同一窗口运行态
tool_item_completed -> 更新同一窗口完成/失败态
assistant_text_final/body_finalize -> 正文收口
```

任务 `closeout_summary` 后主聊天不保留工具窗口，工具轨迹去日志/trace。

## 5. 2026-06-15 追加修正：反馈信号不得双路投影

截图复现了同一句模型公开反馈显示两次：

```text
正在读取 projection 系统核心文件，补齐之前未覆盖的窗口。
补读 dynamic_context/manager.py 241 行之后，以及确认 projection 目录下的完整文件列表。
```

根因不是“前端少做去重”，而是信号权威层不该让同一语义反馈走两条可见线路。

新的接线规则：

| 信号 | 后端投影 | 前端职责 |
| --- | --- | --- |
| `runtime_step_summary` 且 `presentation_source=model_action.*` | `op=body_append`、`slot=body`、`event_family=assistant_body`、`channel=body` | 直接按正文 block 显示 |
| 同一模型反馈再次到达 | `ProjectionLifecycleState` 按 `feedback_identity`，缺失时按同一 turn/task 的规范化正文拦截 | 不收到第二条可见 public event |
| 工具请求/权限/开始/完成 | `tool_control` frame，按 `tool_call_id/tool_lifecycle_id` 更新同一工具窗口 | 折叠窗口显示动作、目标、状态；展开显示命令行和系统返回 |
| 普通 runtime/status 诊断 | `status/trace` | 状态或日志，不提升成正文 |

前端保留一层旧历史回放保护：当旧的 `runtime_step_summary/status_trace` 被提升为正文时，如果同文本已经存在于 body block，则不再重复显示。这不是主路由，只是旧事件回放的防重复保护。

## 6. 2026-06-15 追加修正：断线不是收口

`background_executor_missing_after_restart`、`runtime_process_restarted`、`stream_cancelled` 这类信号属于系统运行/连接故障，不是模型完成任务。

新的权威边界：

| 事件 | 归属 | 主聊天显示 |
| --- | --- | --- |
| `session_output_commit_ack` / `commit_ack` | 模型最终正文已被系统成功提交 | `task_closed / closeout_summary` |
| `turn_completed` | turn 生命周期结束或中断记录 | 不触发收口 |
| `task_bridge_terminal` | task bridge 关闭记录 | 不触发收口 |
| `failed/stopped/orphaned` run status | 系统运行状态 | 不触发收口 |
| `background_executor_missing_after_restart` | 后端重启后执行器丢失 | 保持 `task_live / live_timeline`，由断线/日志/状态显示问题 |

前端 chat stream 重连也不再使用 5 次上限。连接断开后持续重连，使用指数退避，最大间隔 30 秒；UI 只显示“重连中第 N 次”，不把重连失败伪装成任务结束。

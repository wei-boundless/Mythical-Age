- Task run: `taskrun:turn:session-9458b9376ed8437e:24:72de6060`

## 账本结论

这条 task run 的 DeepSeek provider prompt cache 不是完全失效，而是被稳定前缀切裂：

- provider usage records: `7`
- cached tokens: `193536`
- prompt tokens: `348909`
- deepseek cache hit rate: `0.5547`
- post-warm hit rate: `0.5631`

同一 session 的 `tool-followup` 链路能达到约 `0.96` 到 `0.99` 的单次 provider cache hit rate，说明 provider 自动 prefix cache 本身可用。低命中主要集中在 `task_execution` 路径。

## 主要错接线一：动态 lifecycle prompt 被放进 session_stable

代码位置：

- `backend/harness/runtime/environment_prompt_controller.py`
  - `single_agent_turn`: `has_memory` 时才追加 `memory_read_context`
  - `tool_observation_followup`: `has_memory` 时才追加 `memory_read_context`
  - `task_execution`: `has_memory` 时才追加 `memory_read_context`
- `backend/harness/runtime/compiler.py`
  - `prompt_mount_plan.lifecycle_prompt_refs` 被合入 `environment_stable`
  - `environment_stable` 被标为 `cache_scope="session"`、`cache_role="session_stable"`

账本证据：

同一个 task run 连续调用中，`environment_stable` 在两种内容之间来回切换。

| invocation | environment hash | bytes | memory_read_context |
| --- | --- | ---: | --- |
| `1:1` | `f9906f49ec60` | `23020` | false |
| `2:2` | `f9906f49ec60` | `23020` | false |
| `3:3` | `cc4f449e0017` | `23440` | true |
| `4:4` | `f9906f49ec60` | `23020` | false |
| `5:5` | `cc4f449e0017` | `23440` | true |
| `6:6` | `cc4f449e0017` | `23440` | true |
| `7:7` | `f9906f49ec60` | `23020` | false |

`memory_read_context` 的 prompt 内容是静态规则，但它的挂载由动态 memory 可见性决定。结果是一个本应稳定的 prompt 段变成了运行时开关，导致 provider prefix hash 反复变化。

## 主要错接线二：bound_task_context_stable 实际包含运行时变化

代码位置：

- `backend/harness/runtime/bound_task_context.py`
  - `build_bound_task_context()` 从 `dynamic_context` 和 `task_state_projection` 读取 `context_refs`、`known_task_files` 等内容。
  - `_stable_payload_body()` 把这些内容作为 stable body 输出。
- `backend/harness/runtime/compiler.py`
  - `bound_task_context_payload` 被挂为 `kind="bound_task_context_stable"`。
  - 该段被标为 `cache_role="session_stable"`。

账本证据：

同一个 task run 中：

- `1:1` 的 `bound_task_context_stable` hash 是 `cb6a4aeff588`
- 后续 `4:4` / `7:7` 变为 `55ea8c96c35b`

这说明该段不是稳定输入，而是会随任务文件状态、上下文引用或恢复提示变化。它不应该整体作为 stable prefix 的一部分。

## 次要问题：agent closeout 有未规划模型调用

账本记录了 `3` 次 `unplanned_model_call`：

- source: `harness.single_agent_turn.agent_closeout`
- call kind: `invoke_messages`
- reason: `missing_segment_plan`

对应代码：

- `backend/harness/loop/single_agent_turn.py`
  - `emit_agent_authored_closeout()` 调用 `_invoke_single_turn_model()` 时没有传 `segment_plan`。
  - `tool-limit-closeout` 已经构造了 `closeout_segment_plan`，这说明 closeout 链路并非不能规划，而是 agent-authored closeout 这条分支漏接了。

这个问题不是刚才 task run 低命中的主因，但会污染全局 prompt cache 统计，也违反“所有主链路模型调用必须由 RuntimeCompiler / segment_plan 管控”的要求。

## 根因判断

当前 prompt cache 偏低不是单一 provider 问题，而是 prompt 分层错误：

1. 静态规则 prompt 的挂载受动态状态影响。
2. 运行时变化的 bound task context 被命名并标注为 stable。
3. 少数 closeout / repair 分支没有统一进入 segment plan 账本。

成熟 agent 的 prompt cache 分层应按“内容跨调用是否不变”决定，而不是按段名或语义边界决定。

## 修复方向

### 第一阶段：稳定 environment prefix

- 将 lifecycle prompt 的选择从“动态信号触发挂载”改为“按 invocation kind 固定挂载”。
- `memory_read_context` 这类静态规则应稳定存在；真正的 `memory_context` 内容继续留在 dynamic / volatile payload。
- 修复目标：同一个 invocation kind 内，`environment_stable` hash 不再因 memory 是否可见而切换。

### 第二阶段：拆分 bound task context

- 把 `BoundTaskContext` 拆成两层：
  - `bound_task_contract_stable`: 只包含 plan refs、task contract refs、固定恢复策略模板。
  - `bound_task_runtime_context`: 包含 known files、file evidence、context refs、rehydration refs 等会随执行变化的内容，标为 volatile 或 task replay。
- 禁止把由 `task_state_projection` 或 `dynamic_context` 派生的变化内容标为 `session_stable`。

### 第三阶段：补齐 closeout segment plan

- `agent-authored closeout` 必须构造 closeout segment plan，并写入 accounting context。
- 所有主链路 model call 都必须明确 `cache_metric_scope` 和 `segment_plan`。
- 目标是清零 `unplanned_model_call`。

### 第四阶段：验证口径

- 对刚才同类 task run 重新跑诊断：
  - `environment_stable` hash 在同一 invocation kind 内保持不变。
  - `bound_task_runtime_context` 不再出现在 stable prefix。
  - `unplanned_model_call_breaks` 为 `0`。
  - task_execution post-warm hit rate 不再被 early stable section 切裂。

## 实施记录

### 已完成修复

1. 固定 lifecycle prompt 挂载：
   - 文件：`backend/harness/runtime/environment_prompt_controller.py`
   - 取消用 `has_memory`、`has_observations`、`has_pending_steers`、`has_memory_write`、`has_compaction` 等运行态信号决定 stable lifecycle prompt 是否挂载。

   - 删除失去职责的动态判断 helper，避免后续重新把运行态信号接回 stable prompt 挂载层。

3. 调整 task_execution prompt 组装：

   - `bound_task_context_stable` 继续位于 task stable prefix。
   - 新增 `bound_task_runtime_context` 段，标为 `cache_scope="none"`、`cache_role="volatile"`。
   - 将 `active_skills` 从 task stable 段之前移到 stable prefix 之后，避免 active skill 作为 volatile 内容提前截断稳定前缀。

4. 补齐 agent closeout segment plan：
   - 文件：`backend/harness/loop/single_agent_turn.py`
   - `emit_agent_authored_closeout()` 现在复用 `_single_agent_turn_followup_segment_plan()`，并把 `segment_plan` 与 `segment_plan_ref` 写入 accounting context。

5. 明确 runtime context 来源：
   - 文件：`backend/prompt_composition/message_specs.py`
   - `bound_task_runtime_context` 显式归类为 `runtime.dynamic_context_fragment`。

# 后台并发任务隔离成熟架构设计方案

日期：2026-06-13

状态：计划书，待确认后实施

## 1. 结论

本项目后台并发任务不应设计成“每个任务直接开一个线程”。

成熟方案应采用：

```text
TaskRun / Graph WorkOrder
-> ExecutionIsolationService 申请执行隔离胶囊
-> TaskExecutionCapsule 持有权限、资源锁、工作区视图、产物命名空间
-> BackgroundWorkerSlot 选择 async/thread/process/sandbox 执行载体
-> RuntimeAssembly 只消费已解析 capsule
-> ToolRuntimeExecutor 执行已授权工具
-> staged output
-> CommitGate 提升到 canonical session / workspace / memory / artifact
```

线程、async task、进程、容器都只是 worker 载体，不是隔离边界。真正的隔离边界是 `TaskExecutionCapsule`。

目标架构名称：

```text
TaskRun 级后台 Actor
+ Execution Capsule
+ Resource Lock Manager
+ Commit Promotion Gate
```

核心原则：

1. 默认隔离，显式共享。
2. 后台任务可以并发运行，但共享写入必须受资源锁和提交门约束。
3. 后台任务不能直接污染主 session、真实 workspace、canonical memory。
4. 任务完成只产生可追踪的结果引用和通知；用户接受或系统提交后才进入 canonical 状态。
5. 无法证明 sandbox / permission / lock 生效时，失败闭合，不降级为裸执行。

## 2. 当前问题定义

### 2.1 用户症状

用户希望：

- 每个任务都可以分配到独立后台执行单元。
- 后台任务互不干扰。
- 多个任务可以同时跑。
- 任务结果可以回到主会话或项目中。

当前系统已有后台执行能力，但没有任务级隔离权威。结果是：

- executor 是否运行有记录，但不代表它拥有独立资源边界。
- 工具层有 sandbox，但不是 TaskRun / work order 生命周期边界。
- graph work order 可以并发调度，但资源冲突没有统一仲裁。
- session-scoped storage 会让并发任务天然共享写入面。
- commit gate 已存在，但还不是 isolated output 的唯一提升边界。

### 2.2 系统性质缺口

真正缺失的是：

```text
后台任务启动前，必须有一个可持久化、可观察、可恢复、可释放的隔离所有权对象。
```

这个对象必须回答：

- 这个任务是谁？
- 它能读哪里？
- 它能写哪里？
- 它占用哪些共享资源？
- 它的工具权限是什么？
- 它的产物先存哪里？
- 它什么时候可以把结果提交到主状态？
- 它失败或取消后如何释放资源？

如果这些问题不由单一权威回答，并发任务就会继续依靠隐式状态、线程时序、临时 sandbox root 或工具自己的权限判断运行。这不是成熟 agent runtime。

## 3. 技术源报告

### 3.1 本项目当前实现证据

#### TaskRun executor 启动权威

`backend/harness/loop/task_executor_controller.py`

当前 `TaskExecutorController.schedule()` 负责把 `TaskRun` 调度到后台，并通过 `_is_task_run_executor_claimed()` 判断是否已有 executor。

已有能力：

- TaskRun 生命周期边界。
- `waiting_executor` / `scheduled` / recovery 状态。
- `runtime_host.spawn_background_task(...)` 后台启动。
- scheduled executor 恢复。

缺口：

- `executor claimed` 只说明“有执行器在跑”，不说明“执行器拥有独立资源”。
- 没有 task-level workspace view。
- 没有 task-level artifact namespace。
- 没有 task-level memory write buffer。
- 没有资源锁。
- 没有隔离提交策略。

#### 通用后台任务管理

`backend/orchestration/execution_scheduler.py`

当前 `BackgroundTaskManager` 有 `queued/running/succeeded/failed/skipped` 记录和 coalesce key。

已有能力：

- 后台任务记录。
- handler 注册。
- coalesce 避免部分重复任务。
- event loop 不存在时 fallback thread。

缺口：

- coalesce key 不是 resource lock。
- 不知道任务读写哪些资源。
- 不知道任务能否并发。
- 不知道任务是否拥有 workspace / artifact / memory 边界。
- fallback thread 只是执行载体，不是隔离。

#### Graph work order 调度

`backend/harness/graph/background_supervisor.py`

当前 `GraphRunBackgroundSupervisor` 根据 active work orders 调度后台任务，使用 task name 判断是否 already running。

已有能力：

- work order 后台执行。
- graph_run / work_order id 命名。
- active work order 调度。

缺口：

- `graph-work-order:{graph_run_id}:{work_order_id}` 只防同一 work order 重复执行。
- 不判断两个 work order 是否写同一文件、同一 memory scope、同一 artifact space。
- 不区分 read/read 可并发和 write/write 冲突。
- graph node 缺 resource key 时没有保守排他默认。

#### Tool sandbox

`backend/runtime/tool_runtime/sandbox_backend.py`

当前 `LocalOverlaySandboxBackend` 能把工具执行引导到 overlay sandbox，读可从真实 workspace materialize，写落到 sandbox root。

已有能力：

- local overlay sandbox。
- workspace_root / sandbox_root 边界。
- 文件 materialization。
- tool_workspace_root 对读工具和写工具区分。

缺口：

- sandbox policy 从工具调用链传入，不是 TaskRun capsule 的唯一产物。
- sandbox root 缺失时仍可能由调用链临时生成。
- sandbox 不知道资源锁，不知道 graph work order 并发冲突。
- sandbox 本身不负责 staged output 的提交提升。

#### Tool execution context

`backend/runtime/tool_runtime/tool_use_context.py`

当前 `ToolUseContext` 已包含：

- `workspace_root`
- `sandbox_root`
- `task_run_id`
- `artifact_root`
- `permission_mode`
- `sandbox_policy`

这是很好的承载点。

缺口：

- `ToolUseContext` 目前只承载工具上下文，不证明该上下文来自有效 lease。
- 没有 `capsule_id` / `lease_id`。
- 没有 permission fingerprint。
- 没有 resource lock proof。
- 后台写工具不能通过 context 证明“自己被隔离授权”。

#### Environment storage

`backend/harness/runtime/environment_storage.py`

当前 storage 以 session 为主：

```text
mythical-agent/sessions/{session}/environments/{namespace}
```

缺口：

- 并发后台任务默认共享 session storage。
- work order / attempt 级 storage 不明确。
- staged output 和 canonical output 容易混在一起。

#### Sandbox execution scope

`backend/harness/runtime/sandbox_execution_scope.py`

当前已有：

- artifact root
- scratch roots
- task write roots
- workspace write roots
- write roots

缺口：

- 这些是写入范围描述，不是并发资源锁。
- 多个后台任务可获得同样 write root，但没有冲突仲裁。
- workspace write allowed 只说明权限，不说明是否可与其他任务同时写。

#### Commit gate

`backend/orchestration/commit_gate.py`

当前已有：

- user message commit decision
- task run final commit decision
- assistant session message commit decision
- blocked commit gate

缺口：

- 没有 isolated file/artifact/memory output promotion。
- 没有把 task staged output 提升为 canonical workspace / memory 的统一入口。
- 后台任务完成和主会话消息提交之间还缺明确边界。

## 4. 目标设计

### 4.1 核心概念

#### ExecutionIsolationRequest

后台执行入口向隔离服务提交的请求。

字段：

```text
request_id
owner_kind: task_run | graph_work_order | subagent | maintenance
session_id
task_run_id
graph_run_id
work_order_id
parent_task_run_id
attempt_id
workspace_root
runtime_contract_ref
requested_resources
requested_permissions
execution_mode
commit_policy_ref
deadline
```

职责：

- 描述任务想要的执行边界。
- 不直接授权。
- 不创建 sandbox。
- 不写 state_index。

#### TaskExecutionCapsule

后台任务的唯一隔离所有权对象。

字段：

```text
capsule_id
lease_id
owner
executor_epoch
attempt_id
workspace_view
sandbox_root
artifact_namespace
memory_namespace
resource_locks
permission_profile
approval_policy
commit_policy
worker_policy
heartbeat
cleanup_policy
diagnostics
authority = "orchestration.execution_isolation"
```

职责：

- 持有任务级 sandbox / storage / permission / lock / commit policy。
- 作为 RuntimeAssembly 和 ToolRuntimeExecutor 的唯一隔离输入。
- 作为 monitor/frontend 展示后台任务隔离状态的事实源。

禁止：

- 不做模型语义判断。
- 不决定任务目标。
- 不生成 prompt。
- 不执行工具。

#### ResourceLock

共享资源锁。

字段：

```text
lock_id
resource_key
mode: read | write | exclusive
owner_capsule_id
owner_task_run_id
acquired_at
expires_at
heartbeat_at
reentrant_group
diagnostics
```

资源 key：

```text
workspace:file:<normalized_path>
workspace:tree:<normalized_root>
git:index:<repo_root>
shell:<cwd>
browser:<session_id>
memory:<scope>
artifact:<namespace>
task:<task_run_id>
graph:<graph_run_id>
project:<project_id>
credential:<provider>:<identity_hash>
```

锁规则：

- read/read 兼容。
- write/exclusive 与同资源任何锁冲突。
- unknown resource 默认 exclusive。
- 无法规范化路径时，默认 project exclusive。
- 同 capsule 内部允许 reentrant。
- 过期锁需要 stale recovery，而不是直接忽略。

#### WorkspaceView

任务看到的工作区视图。

类型：

```text
read_only_workspace
overlay_sandbox
git_worktree
external_sandbox
direct_workspace_write
```

默认：

- coding / file mutation task：`overlay_sandbox`。
- graph independent coding branch：可升级为 `git_worktree`。
- pure research / review：`read_only_workspace`。
- direct workspace write：只允许在用户明确授权、资源锁持有、commit policy 允许时使用。

#### CommitPolicy

控制 staged output 如何进入 canonical。

字段：

```text
policy_id
file_outputs: draft | changeset_required | direct_commit_allowed
artifact_outputs: staged | repository_materialize
memory_outputs: candidate_only | commit_allowed
session_outputs: notification_only | assistant_message_allowed
requires_user_accept
requires_verification
allowed_targets
blocked_targets
```

默认：

- 文件：`changeset_required`。
- memory：`candidate_only`。
- session：`notification_only`。
- artifact：`staged`，需要 materialization receipt。

### 4.2 后台 Actor 模型

每个可执行 TaskRun 或 graph work order 对应一个逻辑 Actor。

```text
TaskRunActor
  state
  inbox
  capsule
  worker_slot
  output_ref
  cancellation_token
```

Actor 状态：

```text
queued
waiting_resource
starting
running
waiting_approval
paused
cancelling
committing
completed
failed
cancelled
stale_recoverable
```

Actor inbox：

```text
UserSteer
Pause
Resume
Cancel
ApprovalDecision
ResourceGranted
WorkerHeartbeat
WorkerCompleted
WorkerFailed
CommitAccepted

```

规则：

- 用户追加要求进入对应 actor inbox。
- actor 在工具轮次边界消费 steer。
- actor 不直接修改其他 actor 的 inbox。
- graph supervisor 可以观察 actor 状态，但不越权修改工具执行上下文。

### 4.3 Worker 载体选择

| 任务类型 | 推荐 worker | 原因 |
| --- | --- | --- |
| LLM 调用、SSE、等待工具结果 | `asyncio.Task` | IO 密集，适合 async |
| 阻塞文件操作、同步 SDK、轻量清理 | `ThreadPoolExecutor` | 避免阻塞 event loop |
| CPU 重任务、长脚本、不可信运行 | process / external sandbox | 避免 GIL 和进程内污染 |
| coding 文件修改 | overlay sandbox / git worktree | 输出可 review，可回滚 |
| graph 并行节点 | actor + lock + bounded pool | 需要资源仲裁 |

限制：

- 不允许无限创建线程。
- worker pool 必须有全局上限、每 session 上限、每 graph 上限。
- 同一 task actor 不允许重复 executor。
- worker 崩溃必须能从 capsule/lease 恢复或显式失败。

推荐初始配额：

```text
global_task_workers: 6
per_session_task_workers: 3
per_graph_work_order_workers: 4
blocking_thread_workers: 4
process_workers: 2
subagent_depth_limit: 2
subagent_children_limit: 3
```

这些值应进入配置，而不是硬编码在 executor 内部。

## 5. 固定执行流

### 5.1 TaskRun 后台启动

```text
TaskExecutorController.schedule(task_run_id)
-> load TaskRun
-> build ExecutionIsolationRequest
-> ExecutionIsolationService.acquire(request)
   -> normalize workspace root
   -> compile permission profile
   -> compile resource locks
   -> allocate sandbox/artifact/memory namespace
   -> persist capsule
   -> mark TaskRun waiting_resource or scheduled
-> spawn TaskRunActor with capsule_id
-> actor starts worker
-> RuntimeAssembly loads capsule
-> TaskExecutor step loop
-> ToolRuntimeExecutor executes tools with capsule sandbox policy
-> staged output
-> CommitGate promotion
-> release capsule
```

### 5.2 Graph work order 后台启动

```text
GraphRunBackgroundSupervisor.dispatch_ready()
-> read active/ready work orders
-> for each work order:
     build IsolationRequest from graph node contract
     acquire resource locks
     if conflict:
        mark waiting_resource with blocker refs
     else:
        spawn WorkOrderActor
-> join policy decides downstream readiness
```

规则：

- graph node 声明 resource keys 时按声明调度。
- graph node 未声明资源时默认 `exclusive:project:<project_id>`。
- read-only research/review node 可以声明 read locks。
- memory_commit / artifact_materialization / workspace_write 默认 exclusive。

### 5.3 Tool execution

```text
ToolRuntimeExecutor.execute(tool_call, sandbox_policy)
-> require capsule_id for background task writes
-> validate capsule exists and active
-> validate tool permission profile
-> validate resource lock proof
-> build ToolUseContext
-> execute tool
-> emit invocation receipt with capsule_id
```

后台写工具无 capsule 时：

```text
error = background_tool_requires_execution_capsule
```

后台工具声明的 resource key 与 capsule lock 不匹配时：

```text
error = tool_resource_lock_not_held
```

### 5.4 Commit promotion

```text
TaskRun completed
-> collect staged outputs
-> build CommitCandidate(s)
-> CommitGate evaluates policy
-> if requires user accept:
     expose pending ChangeSet / artifact / memory candidate
-> if allowed:
     promote to canonical
-> record receipts
-> notify session
```

提升目标：

- `session_message`
- `task_result`
- `workspace_changeset`
- `artifact_repository_materialization`
- `memory_candidate`
- `memory_commit`

禁止：

- worker 直接写 canonical memory。
- work order postprocess 绕过 CommitGate。
- 后台任务直接把 raw transcript 注入主会话正文。

## 6. 权责链

| 模块 | 当前职责 | 目标职责 | 禁止事项 |
| --- | --- | --- | --- |
| `TaskExecutorController` | 调度/恢复 TaskRun executor | TaskRun 启动权威，必须先申请 capsule | 不自行生成 sandbox root；不决定资源冲突 |
| `GraphRunBackgroundSupervisor` | 调度 ready work order | Graph work order 启动权威，按 resource locks 分发 | 不用 task name 代替资源锁 |
| `BackgroundTaskManager` | 通用后台队列 | 非 runtime-critical 后台任务队列，或承载 worker slot | 不作为 TaskRun 隔离权威 |

| `RuntimeAssembly` | 装配 runtime inputs | 消费 capsule 解析后的 workspace/sandbox/artifact/memory | 不重算权限，不猜资源 |
| `ToolRuntimeExecutor` | 执行工具和权限检查 | 执行已授权工具，校验 capsule/lock proof | 不授予权限，不创建 lease |
| `LocalOverlaySandboxBackend` | overlay sandbox | 执行 workspace view 的底层 backend | 不决定提交，不决定并发 |
| `CommitGate` | session/task result commit | staged output 到 canonical 的唯一提升门 | 不读取 prompt，不执行工具 |
| `StateIndex/EventLog` | 持久记录 | 记录 capsule lifecycle / locks / receipts | 不做资源仲裁 |

| 前端 store | 展示运行状态 | 展示后台任务、锁等待、approval、commit 状态 | 不决定任务是否可并发 |

## 11. 迁移与切换规则

### 11.1 Shadow 阶段

先启用 shadow isolation：

- acquire capsule 但不阻塞现有执行。
- 记录 would_block / would_allow。
- 对比当前调度行为。

适用范围：

- 只读任务。
- 非关键 graph work order。
- 测试环境。

退出条件：

- shadow 记录证明资源冲突检测可靠。
- 无 false positive 阻塞关键只读任务。

### 11.2 Enforce 阶段

对后台写任务强制 capsule：

- TaskRun 后台写工具必须有 capsule。
- graph write work order 必须有 resource keys 或默认 exclusive。
- staged output 必须过 CommitGate。

### 11.3 Cutover 阶段

删除旧路径：

- 无 capsule 后台写工具路径。
- session-scoped task write 默认路径。
- graph name-only 并发判断作为授权依据。
- 临时 sandbox root fallback。

### 11.4 Rollback 规则

如果隔离服务出现严重问题：

- 可临时关闭 enforce，回到 shadow。
- 不能恢复无 capsule 的后台写入。
- 对写任务应降级为串行 exclusive，而不是降级为共享写。

## 12. 风险与控制

### 风险 1：过度串行导致并发收益下降

控制：

- read/read 明确可并发。
- graph node 支持精确 resource key。
- unknown 才 exclusive。

### 风险 2：资源 key 推导不准

控制：

- 文件路径统一 normalize 到 workspace absolute path。
- shell 默认 cwd exclusive。
- memory/artifact 使用显式 scope。
- 无法规范化默认 exclusive。

### 风险 3：任务卡在 waiting_resource

控制：

- lease heartbeat。
- stale lock recovery。
- 前端展示 blocker。
- 支持用户 cancel / retry。

### 风险 4：staged output 与 canonical output 混乱

控制：

- staged namespace 必须包含 capsule_id/attempt_id。
- CommitGate 产生 receipt。
- canonical 只接受 receipt。

### 风险 5：旧代码绕过新权威

控制：

- 搜索旧 fallback。
- 后台写工具无 capsule 直接拒绝。
- 测试覆盖旧路径不可达。

## 13. 不允许的实现方式

1. 不允许把 OS thread 当隔离边界。
2. 不允许无限制“一任务一线程”。
3. 不允许后台任务直接写真实 workspace。
4. 不允许 graph 节点无资源声明还共享写并发。
5. 不允许工具执行器自行发明权限。
6. 不允许 RuntimeAssembly 重算用户意图。
7. 不允许 commit gate 之外提交 memory / artifact / workspace。
8. 不允许用前端状态决定后端控制能力。
9. 不允许为了测试通过 mock 掉核心隔离逻辑。
10. 不允许保留旧链路作为“兼容兜底”继续运行。

## 14. Agent Prompt 约束

本方案主要是 runtime 结构修复，不应靠 prompt 补丁实现隔离。

如果需要给后台 worker agent 提示，应写成角色和边界，不写成开发说明。

正确方向：

```text
你是一名后台任务执行员。
你只处理当前 TaskRun 胶囊授权范围内的工作。
你可以读取胶囊允许的上下文和文件。
你只能把文件变更写入当前任务的草稿工作区。
你不能直接修改主会话、真实项目工作区或长期记忆。
当你完成后，输出可验证的结果摘要、产物引用和需要提交的变更说明。
如果你发现目标资源不在授权范围内，停止并报告需要新的授权。
```

错误方向：

```text
这是后台任务节点。
根据 capsule 执行 work_order。
这个节点用于隔离任务。
```

## 15. 最终目标状态

完成后，系统应具备以下能力：

1. 每个 TaskRun 可以后台运行。
2. 每个后台任务都有独立 capsule。
3. 多个只读任务可以并发。
4. 多个写任务按资源锁安全串行或并行。
5. graph work order 并发由 resource key 和 join policy 决定。
6. 后台任务产物先 staged，再由 CommitGate 提升。
7. 主会话只接收通知和结果引用，不被 raw transcript 污染。
8. 用户能看到任务为何等待、为何阻塞、是否需要审批。
9. 任务失败、取消、进程重启后可以恢复或清理。
10. 旧的隐式共享写链路被删除。

这才是接近 Codex、Claude Code、Hermes 这类成熟 agent 的后台任务隔离方式：后台运行是用户体验，执行胶囊才是系统安全边界。

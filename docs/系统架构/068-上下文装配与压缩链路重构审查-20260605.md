# 上下文装配与压缩链路重构审查

日期：2026-06-05

## 目标口径

聊天页展示的上下文压力只回答一个问题：当前 session 可压缩上下文状态有多少 token，距离自动压缩阈值还差多少。它不是累计账单、不是累计会话原文、不是最后一次 provider prompt，也不是 prompt cache 命中率。

系统必须避免“每回合先把历史静默截短，再声称上下文压力很低”的假象。普通用户/助手对话应以压缩后的活动历史全量进入模型输入；当活动历史接近阈值时，由会话压缩链路显式改写历史并留下摘要检查点。只有工具协议热尾和大工具输出可以走投影、预览和可恢复原文链路。

## 目标权威链

1. `sessions.SessionManager` 保存两条历史：
   - `messages`：运行时活动历史，会被显式压缩改写。
   - `api_transcript`：provider 协议原始轨迹，只服务工具调用协议恢复，不承载普通对话记忆。
2. `runtime.context_management.session_compaction` 在模型回合前检查压力：
   - `session_compaction` 负责计算 deterministic session pressure：压缩摘要、活动公开历史、压缩边界之后的工具/协议轨迹。
   - `ContextUsageMeter` 只负责套用模型窗口、压缩阈值、剩余额度和压力等级，并保留 provider/cache 诊断。
   - `ContextCompactor` 负责活动历史自身是否需要显式压缩。
   - 自动压缩只由同一个 `context_meter.auto_replacement_allowed` 授权；history 压缩预览不得作为第二套自动触发口径。
3. `runtime.shared.history_assembler` 只装配压缩后的活动历史全量，不做 recent-N 截断。
4. `harness.runtime.dynamic_context.HistoryProjector` 只把活动历史结构化给模型，不再对普通历史做硬截断。
5. `harness.runtime.compiler` 只把 provider protocol 的工具热尾作为真实 chat/tool 消息重放；普通旧对话不进入 provider replay。
6. `runtime.model_gateway.ModelRuntime` 在每次请求前写入 local prediction，请求后写入 provider usage。
7. `ContextUsageMeter` 在收到 `session_pressure_tokens` 时必须以它作为 `current_context_tokens` 的唯一压力权威；最新 agent runtime provider/local 记录只保留为 cache 和请求诊断。
8. 聊天页只显示 `context_meter.current_context_tokens`、`context_meter.replacement_threshold_tokens` 和 `context_meter.compaction_pressure_ratio`。累计账单、累计 transcript、压缩节省和缓存命中保留在管理/监控诊断中。

## 审查发现

### 1. 活动历史仍存在被动硬截断

`HistoryProjector` 已经不再 recent-N 截断，但仍读取 `history_message_chars` 并对每条普通历史调用 `compact_text`。这会让模型看到的“当前上下文”不是活动历史全量，而是每回合被动压薄的历史，压缩阈值自然很难触发。

修正：普通对话历史不走 per-message hard cap。显式压缩由 `ContextCompactor` 负责。

### 2. 模型可见字段仍叫 `recent_turns`

字段名和 section report 仍表达旧的 recent-window 语义，容易让后续维护者继续往旧链路上补逻辑。

修正：模型可见 payload 改为 `active_history`，诊断改为 `active_history_message_count`。

### 3. provider protocol 测试仍把普通旧对话当作协议截断

新链路中 provider replay 只保留 assistant tool_calls、tool 结果和 reasoning_content。普通 user/assistant 旧对话被过滤是正确行为，应记录为 `non_protocol_message_count`，不应算 `protocol_truncated_count`。

修正：测试改为验证冷普通对话没有进入 provider replay，并通过 `non_protocol_message_count` 可观测。

### 4. 聊天页上下文条仍有历史预算回退

`ChatPanel` 在缺少 `context_meter` 字段时回退到 `history_usage_ratio`、`history_tokens`、`history_budget_tokens`。这会把上下文压力和历史压缩预览混成一个 UI 口径。

修正：聊天页上下文条只读 `context_meter`。没有 `context_meter` 时显示等待/未知，不用累计或历史预算代替。

## 不改的边界

- 不删除监控系统的 token 账本统计。累计账单和缓存命中仍属于 `PromptAccountingLedger`、health 页面和记忆管理页诊断。
- 不把普通会话历史塞进 `api_transcript`。`api_transcript` 只用于 provider 工具协议闭环。
- 不绕过语义压缩 agent 注册链路。语义压缩 worker 继续由 runtime 编排注册后接入 `ContextCompactor`，不是在压缩器内部硬编码子 agent。

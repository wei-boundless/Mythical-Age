## 1. 一句话目标

交接核心是边契约，不是场景标签。我们要做的是建立一批“边契约交接原型”，让用户通过可理解的工作关系选择原型，再由编译器生成完整 `CompiledEdgeContract`。

用户应该面对的是：

```text
交付审核
返修闭环
协商定案
澄清问答
并行会审
汇总裁决
记忆候选提交
高风险授权
```

而不是：

```text
edge_type
a2a_message_type
ack_policy
result_delivery_policy
context_filter_policy
artifact_ref_policy
working_memory_handoff_policy
scheduler_role
failure_propagation_policy
```

强语义的核心是：

```text
用户选择工作关系。
编译器选择边契约原型。
边契约原型生成 CompiledEdgeContract。
runtime 只执行发布后的边契约。
```

## 2. 问题定义

当前边系统已经具备协议字段和语义关系预设，但用户仍然容易被拉到低层配置里。用户真正想表达的是：

- 写手完成草稿后交给审核员。
- 审核员不通过时把结构化问题返回给写手。
- 两个 agent 围绕一个争议点讨论并形成结论。
- 多个专家并行评审后由汇总员给出裁决。
- 节点产生候选记忆，经过审核后再提交为正式记忆。

这些不是“通信协议配置”，而是“agent 协作模式”。如果系统要求用户自己判断 A2A、handoff、context、commit、ack、timeout、packet visibility，用户就会回到繁琐配置，强编译器的价值会被削弱。

正确方向不是让“场景”成为新权威，而是让场景成为用户侧入口，核心权威仍然是边契约：

```text
Authoring DSL
-> EdgeContractPrototype
-> EdgeRelationTemplate
-> EdgeProtocolTemplate
-> CompiledEdgeContract
-> Runtime Executor
```

## 3. 当前代码基础

现有代码已有可承接的低层结构：

- `TaskGraphSemanticRelationPreset` 已经能表达 `writing.draft_to_review`、`writing.review_revise_to_writer`、`memory.read_required`、`memory.commit_after_review`。
- `TaskGraphEdgeDefinition` 已经有 payload、ack、timeout、delivery、artifact、memory、A2A message type。
- `harness.graph.language` 已经分出 edge semantic role 和 scheduler role。
- `FlowPacket` 已经能表达受控交接包。

但还缺一层用户可理解的强语义目录。`semantic_relation` 目前更像“边关系预设”，不是完整“边契约交接原型”。新的 `EdgeContractPrototype` 应该把节点角色、输入输出、边协议、验收、失败恢复和 prompt contract 一起编译出来，最终落到 `CompiledEdgeContract`。

## 4. 目标模型

建议新增一等编译期概念：

```text
EdgeContractPrototype
```

它不是 runtime 执行对象，而是编译器模板对象。

建议结构：

```yaml
prototype_id: edge_contract.review_submission
title_zh: 交付审核
summary: 上游生产者把产物交给审核者，审核者按质量门输出裁决。

user_phrases:
  - 交付审核
  - 提交审核
  - draft_to_review

source_role_kinds:
  - producer
  - writer
  - implementer

target_role_kinds:
  - reviewer
  - verifier
  - quality_gate

edge_contract:
  default_relation_id: writing.draft_to_review
  default_protocol_kind: node_handoff
  default_scheduler_role: dependency

compiled_outputs:
  - edge_relation
  - edge_protocol
  - packet_contract
  - target_input_contract_patch
  - prompt_contract_patch
  - acceptance_contract
  - compile_report_entry

required_source_outputs:
  - deliverable_ref
  - handoff_summary

required_target_outputs:
  - verdict
  - issues
  - acceptance_notes

failure_policy:
  missing_deliverable: fail_closed
  invalid_review_verdict: fail_closed
```

## 5. 场景原型选择原则

### 5.1 用户只选场景，不选协议

用户可以写：

```yaml
edges:
  - draft -> review: 交付审核
  - review -> revise: 返修
  - reviewer <-> writer: 协商定案
```

系统可以展示：

```text
交付审核：把上游产物作为审核材料交给审核节点。
返修闭环：审核不通过时把问题和返修要求交回修订节点。
协商定案：两个 agent 进行有限轮讨论并产出结构化结论。
```

但不要求用户知道：

- `node_handoff`
- `a2a_session`
- `conditional_dependency`
- `contract_payload_and_refs`
- `wait_required_contracts`

### 5.2 编译器必须解释协议选择

compile report 需要写清楚：

```yaml
edge: reviewer <-> writer
scenario: 协商定案
protocol_kind: a2a_session
reason: 双向参与者 + discussion relation + 需要共识产物
output_artifact: contract.a2a.consensus_decision
transcript_visibility: summary_and_refs
```

### 5.3 低置信度时问场景，不问字段

如果编译器无法判断，应该问：

```text
你希望 reviewer 和 writer 的关系是哪一种？
1. 返修闭环：reviewer 单向给 writer 修改要求。
2. 协商定案：reviewer 和 writer 讨论后形成共同返修计划。
3. 人工裁决：reviewer 把争议交给人工确认。
```

而不是问：

```text
请选择 edge_type、ack_policy、scheduler_role。
```

### 5.4 Agent 协作不等于 Agent 对话

需要明确一条原则：

```text
A2A 只是边协议的一种。
Agent 之间可以交流，也可以完全不交流。
```

很多成熟的 agent 协作并不需要两个 agent 互相聊天：

- 写手把草稿交给审核员：这是交付包，不是对话。
- 审核员输出返修要求：这是结构化反馈，不是协商。
- 执行者把结果交给验证者：这是验收输入，不是讨论。
- 节点把候选记忆提交给记忆库：这是资源提交，不是 agent 交流。
- router 根据 verdict 选择分支：这是条件路由，不是通信。
- monitor 接收状态更新：这是事件通知，不是任务交接。

因此，原型目录的重点不是推广 A2A，而是把多种边协议集成到可理解的工作关系中：

```text
工作关系原型
-> 选择一种或多种边协议
-> 合成 payload / artifact / memory / event / session / gate 契约
-> 交给 runtime 执行
```

只有当场景确实需要多轮澄清、协商、争议解决或共同定案时，才应选择 `a2a_session`。默认应优先使用更简单、更可审查的非对话协议。

## 6. 第一批场景原型目录

下面是建议第一批内置原型。它们覆盖最常见的多 agent 连续任务。

## 6.1 交付审核 Review Submission

场景 ID：

```text
scenario.review_submission
```

用户意图：

```text
上游 agent 完成一个产物，下游 agent 负责质量审核或验收。
```

典型角色：

```text
writer -> reviewer
implementer -> verifier
planner -> critic
researcher -> fact_checker
```

用户 DSL：

```yaml
edges:
  - draft -> review: 交付审核
```

默认编译：

```yaml
default_protocol_kind: node_handoff
default_scheduler_role: dependency
default_relation_id: writing.draft_to_review
payload_contract_family: deliverable.review_packet
target_input_slot: review_material
result_delivery_policy: contract_payload_and_refs
ack_required: true
```

默认生成的下游 prompt contract 片段：

```text
你是一名质量审核员。
你收到的是上游 agent 提交的正式交付包。
你只负责根据审核标准判断交付物是否达标。
你必须输出明确 verdict，并列出问题、证据和是否允许进入下一阶段。
你不得替上游重写交付物，除非任务明确要求你同时承担修订职责。
```

fail closed：

- 上游没有交付物 ref。
- 下游没有 verdict schema。
- payload 超过可见性限制且没有 artifact ref。
- 审核节点缺少质量标准。

## 6.2 返修闭环 Revision Loop

场景 ID：

```text
scenario.revision_loop
```

用户意图：

```text
审核不通过时，把结构化问题和返修要求交给修订 agent，修订后再回到审核。
```

典型角色：

```text
reviewer -> reviser
verifier -> implementer
critic -> writer
```

用户 DSL：

```yaml
edges:
  - review -> revise: 返修 when verdict == revise
  - revise -> review: 复审
```

默认编译：

```yaml
review_to_revise:
  default_protocol_kind: review_feedback
  default_scheduler_role: conditional_dependency
  trigger:
    verdict: revise
  payload_contract_family: revision_request
  carry:
    - artifact_ref
    - issues
    - revision_request
    - quality_bar
  loop_policy:
    max_revision_attempts: 3

revise_to_review:
  default_protocol_kind: node_handoff
  default_scheduler_role: dependency
  payload_contract_family: revised_deliverable.review_packet
```

默认生成的修订 agent prompt contract 片段：

```text
你是一名修订执行者。
你收到的是审核节点给出的结构化问题清单和返修要求。
你只负责按要求修订当前交付物，不得忽略审核裁决，也不得把未解决问题标记为已解决。
你需要输出修订后的交付物、修改说明、未解决问题和再次提交审核的摘要。
```

fail closed：

- 返修边没有最大轮次。
- review verdict 不存在或不是枚举值。
- revision request 没有问题清单。
- 修订节点没有被允许访问原交付物 ref。

```yaml
  - plan.constraints
  - plan.acceptance_criteria
```

场景 ID：

```text
scenario.execution_verification
```

用户意图：

```text
执行 agent 完成任务后，由验证 agent 检查是否满足验收标准。
```

典型角色：

```text
implementer -> verifier
operator -> auditor
coder -> test_runner
writer -> editor
```

用户 DSL：

```yaml
edges:
  - execute -> verify: 执行验收
```

默认编译：

```yaml
default_protocol_kind: node_handoff
default_scheduler_role: dependency
payload_contract_family: execution_result.verification_packet
source_output_selector:
  - result_summary
  - artifact_refs
  - test_or_evidence_refs
  - known_issues
target_input_slot: verification_material
required_target_outputs:
  - verdict
  - evidence
  - blocking_issues
```

默认生成的验证 agent prompt contract 片段：

```text
你是一名验收验证员。
你只负责检查执行结果是否满足给定验收标准。
你必须基于交付物、证据和测试结果给出 verdict。
你不能因为交付物看起来合理就直接通过；缺少证据时必须指出缺口。
```

fail closed：

- 没有验收标准。
- 没有 artifact 或 evidence ref。
- verifier 试图直接改交付物。
- verdict 没有证据引用。

场景 ID：

```text
scenario.evidence_handoff
```

```text
研究、检索或分析 agent 把证据包交给写作、决策或审核 agent。
```

典型角色：

```text
researcher -> writer
researcher -> analyst

```

用户 DSL：

```yaml
edges:
  - research -> write: 证据移交
```

默认编译：

```yaml
default_protocol_kind: node_handoff
default_scheduler_role: dependency
payload_contract_family: evidence_bundle
source_output_selector:
  - claims
  - source_refs
  - confidence
  - limitations
target_input_slot: evidence_bundle
visibility:
  require_refs: true
  expose_raw_sources: bounded_by_policy
```

默认生成的接收 agent prompt contract 片段：

```text
你收到的是证据包，而不是最终结论。
你必须区分已证实事实、推断、争议点和证据缺口。

```

fail closed：

- 没有 source refs。
- claims 没有 confidence 或 limitation。
- 下游任务要求事实准确但证据包缺少来源。

## 6.6 协商定案 Bounded Consensus

场景 ID：

```text
scenario.bounded_consensus
```

用户意图：

```text
两个或多个 agent 围绕一个争议点、设计选择或返修方案进行有限轮讨论，并形成结构化结论。
```

典型角色：

```text
reviewer <-> writer
architect <-> implementer
planner <-> verifier
domain_expert <-> editor
```

用户 DSL：

```yaml
edges:
  - reviewer <-> writer: 协商定案
```

默认编译：

```yaml
default_protocol_kind: a2a_session
default_scheduler_role: conditional_dependency
a2a:
  session_scope: edge_run
  turn_policy:
    mode: bounded_round_robin
    max_turns: 4
  stop_when:
    - consensus_reached
    - unresolved_requires_human
    - max_turns_reached
  output_artifact_contract_family: consensus_decision
transcript_policy:
  persist_full_trace: true
  expose_to_downstream: summary_and_refs
```

默认生成的参与 agent prompt contract 片段：

```text
你是一名协商参与者。
你的目标是在自己的职责边界内提出理由、回应对方意见，并帮助形成可执行结论。
你不能无限延长讨论；无法达成一致时需要明确分歧点和建议的下一步裁决方式。
会话结束时必须产出结构化结论，而不是只保留聊天记录。
```

fail closed：

- 没有最大轮次或停止条件。
- 没有 output artifact contract。
- transcript 被默认暴露给所有下游。
- 参与 agent 权限边界不清。

```yaml
edges:
```

```yaml
edges:
  - draft -> [style_review, logic_review, market_review]: 并行会审
```

```yaml
default_protocol_kind: fanout_handoff

branch_packet_policy:
```


```yaml
candidate_edge:
```

```text
你只负责提出候选记忆。
候选记忆不是已提交事实，不能在后续节点中当作正式资料使用。
你需要说明候选记忆的来源、适用范围和不确定性。
```

fail closed：

- 候选记忆直接对下游可见。
- commit 没有 approval source。
- commit 后没有 receipt。
- 记忆命名空间没有绑定节点 effective environment。

## 6.11 高风险授权 Risk Approval

场景 ID：

```text
scenario.risk_approval
```

用户意图：

```text
执行高风险动作前，必须经过安全审核、权限确认或人工批准。
```

典型角色：

```text
tool_operator -> safety_reviewer
executor -> human_gate
publisher -> approval_gate
```

用户 DSL：

```yaml
edges:
  - plan_tool_action -> safety_review: 高风险授权
```

默认编译：

```yaml
default_protocol_kind: human_or_review_gate
default_scheduler_role: conditional_dependency
risk_policy:
  require_approval_for:
    - file_write
    - shell
    - network
    - publish
decision_schema:
  allowed_values: [approve, reject, request_revision]
resume_policy: decision_packet
```

默认生成的安全审核 prompt contract 片段：

```text
你是一名高风险动作审核员。
你只负责判断拟执行动作是否被授权、是否必要、是否有可接受风险。
你不能替执行 agent 执行动作。
你必须输出 approve、reject 或 request_revision，并说明依据。
```

fail closed：

- 风险动作没有权限 ceiling。
- 批准结果不是结构化 decision。
- 审核 agent 直接执行了动作。
- 人工批准没有进入可恢复 history。

## 6.12 状态通知 Progress Notification

场景 ID：

```text
scenario.progress_notification
```

用户意图：

```text
节点把进度、状态或观察结果通知 monitor / ledger，不阻塞主流程。
```

典型角色：

```text
agent -> monitor
agent -> progress_ledger
reviewer -> issue_ledger
```

用户 DSL：

```yaml
edges:
  - draft -> monitor: 状态通知
```

默认编译：

```yaml
default_protocol_kind: event_signal
default_scheduler_role: event
delivery_policy: at_least_once
idempotency_key: graph_run_id.edge_id.event_id
blocks_downstream: false
```

默认生成的通知 contract 片段：

```text
该通知只用于状态记录和监控展示。
它不能作为下游任务的事实输入，也不能改变主调度链路。
```

fail closed：

- event 没有 idempotency key。
- 通知边被误用为主数据输入。
- monitor 事件没有来源节点和 graph run lineage。

## 7. MVP 优先级

第一版不需要把 12 个场景全部工程化。建议 MVP 先做 8 个：

```text
1. 交付审核
2. 返修闭环
3. 规划交付
4. 执行验收
5. 证据移交
6. 协商定案
7. 并行会审
8. 汇总裁决
```

第二批再做：

```text
9. 澄清问答
10. 记忆候选提交
11. 高风险授权
12. 状态通知
```

理由：

- 第一批覆盖大多数多 agent 连续任务的基本生产链。
- 第二批涉及 session、memory commit、human gate、event history 等更强 runtime 语义，可以在边协议执行器稳定后扩展。

## 8. 用户界面建议

用户配置图时，不展示协议字段。展示“关系原型”：

```text
从 draft 到 review
关系：交付审核
含义：把 draft 的交付物作为审核材料交给 review，由 review 输出 verdict。
自动生成：交接包、审核输入、verdict schema、失败策略。
```

用户点开高级视图时，可以看到 compile report，而不是直接编辑底层字段：

```yaml
scenario: 交付审核
compiled_protocol: node_handoff
scheduler_role: dependency
payload_contract: contract.deliverable.review_packet
target_input_slot: review_material
visibility: bounded_payload_and_refs
ack: explicit_ack
```

真正的底层覆盖只给高级用户，并且必须触发重新编译和风险报告。

## 9. 编译器匹配流程

建议编译器按以下顺序选择场景原型：

```text
EdgeSeed
-> source node intent
-> target node intent
-> role kind match
-> user phrase / relation alias
-> graph goal context
-> HandoffScenarioPrototype candidates
-> deterministic validation
-> selected scenario
-> edge relation / protocol / packet contracts
```

匹配结果分三档：

### 9.1 High Confidence

直接编译，并在报告里解释：

```yaml
selected_scenario: 交付审核
confidence: high
reason:
  - source intent produces deliverable
  - target intent is quality_review
  - user phrase is submit_for_review
```

### 9.2 Medium Confidence

可以编译，但提示用户审查：

```yaml
selected_scenario: 执行验收
confidence: medium
reason:
  - target agent has verifier role
  - source output has artifact refs
warning:
  - missing explicit acceptance criteria; using verifier intent default
```

### 9.3 Low Confidence

不问协议，问场景：

```text
execute 和 review 之间可能是：
1. 执行验收
2. 交付审核
3. 协商定案
请选择一个工作关系。
```

## 10. 与 Builder Agent 的关系

可以让 `EdgeContractBuilderAgent` 辅助选择场景，但最终仍由确定性编译器裁决。

Builder agent 输入：

```yaml
graph_goal: ...
source_node:
  id: draft
  intent: chapter_draft
  agent_role: writer
target_node:
  id: review
  intent: quality_review
  agent_role: reviewer
user_edge_phrase: 交付审核
candidate_scenarios:
  - scenario.review_submission
  - scenario.execution_verification
  - scenario.bounded_consensus
```

Builder agent 输出：

```yaml
recommended_scenario_id: scenario.review_submission
confidence: high
reason: source produces draft artifact and target role is reviewer
required_user_clarification: ""
```

它不能：

- 自己发布边契约。
- 自己创建新场景原型。
- 跳过 compiler validation。
- 直接扩大协议权限。
- 自行决定 A2A transcript 可见性。

## 11. Prompt Contract 生成原则

场景原型必须能生成真正给 agent 执行的 prompt contract，而不是开发说明。

错误风格：

```text
这是交付审核节点。
根据边契约处理上游 payload。
```

正确风格：

```text
你是一名质量审核员。
你收到的是上游 agent 提交的正式交付包。
你需要根据审核标准给出 verdict、问题清单和是否允许进入下一阶段。
你不负责替上游重写交付物。
如果交付包缺少必要材料，你必须报告阻塞，而不是自行补全。
```

每个场景原型至少要提供：

- source agent prompt patch。
- target agent prompt patch。
- output schema instruction。
- forbidden actions。
- failure handling instruction。

## 12. 场景原型与边协议的映射

建议第一版映射如下：

| 场景原型 | 默认协议 | 调度语义 | 备注 |
| --- | --- | --- | --- |
| 交付审核 | `node_handoff` | `dependency` | 最常见的产物到审核 |
| 返修闭环 | `review_feedback` + `node_handoff` | `conditional_dependency` | 必须有 max attempts |
| 规划交付 | `node_handoff` | `dependency` | plan 必须有 acceptance |
| 执行验收 | `node_handoff` | `dependency` | verifier 必须输出 evidence |
| 证据移交 | `node_handoff` | `dependency` | source refs 必需 |
| 协商定案 | `a2a_session` | `conditional_dependency` | 必须 artifact 化 |
| 澄清问答 | `a2a_session` | `context` | ask-answer，短轮次 |
| 并行会审 | `fanout_handoff` | `dependency` | 每个 branch 独立 packet |
| 汇总裁决 | `barrier_join` | `dependency` | 必须 join policy |
| 记忆候选提交 | `resource_write_candidate` + `resource_commit` | `commit` | 候选和正式分离 |
| 高风险授权 | `human_or_review_gate` | `conditional_dependency` | 必须 resume schema |
| 状态通知 | `event_signal` | `event` | 非阻塞 |

## 13. 需要讨论的问题

1. 场景原型 ID 是用中文语义还是英文稳定 ID？

建议：内部使用英文稳定 ID，例如 `scenario.review_submission`；UI 展示中文名称和说明。

2. `semantic_relation` 是否升级为 `HandoffScenarioPrototype`？

建议：不要直接替换。`semantic_relation` 保持作为边关系模板，`HandoffScenarioPrototype` 作为更上层的组合模板，引用一个或多个 relation。

3. A2A 场景是否第一版就实现？

建议：第一版可以实现内部 `a2a_session` 的编译产物和 session history，但不急着接外部 A2A Agent Card。

4. 用户是否允许自定义场景原型？

建议：允许创建草案，但不能直接发布为系统模板。必须经过模板校验、样例图编译、回归测试和人工确认。

5. 场景原型是否跨领域复用？

建议：分两层：通用原型 + 领域参数。例如 `scenario.review_submission` 通用，写作领域绑定 `writing.draft_to_review`，代码领域绑定 `coding.patch_to_review`。

## 14. 验收标准

讨论成熟后，正式实现至少要证明：

- 用户可以只选择“交付审核”，无需配置协议字段。
- 编译器能把“交付审核”生成 node_handoff、payload contract、target input slot、review prompt patch 和 compile report。
- 用户可以只选择“协商定案”，编译器生成 bounded A2A session，而不是暴露底层 A2A message 配置。
- 低置信度时，系统询问场景选择，不询问低层协议字段。
- 每个场景原型都有 prompt contract patch，且是 agent 可执行的角色指令。
- 每个场景原型都有 fail-closed 规则。
- compile report 能解释场景选择、协议映射、payload 可见性、失败策略和恢复策略。
- 场景原型不会绕过节点环境、权限 ceiling、resource commit approval 或 runtime checkpoint。

## 15. 暂定结论

强语义的产品形态应该是“场景原型目录”，不是“协议参数表”。用户要配置的是 agent 之间的工作关系，系统再把关系编译成边协议。

第一批原型建议以生产链为核心：

```text
交付审核
返修闭环
规划交付
执行验收
证据移交
协商定案
并行会审
汇总裁决
```

这批原型足以覆盖写作、代码、研究、审核、规划和多专家评审的大多数图任务。后续再扩展记忆、授权、事件、外部 A2A adapter，会更稳。

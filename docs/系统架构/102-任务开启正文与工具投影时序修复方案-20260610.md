## 一、事故结论

这次“只有工具投影、看不到助手正文”的问题不是前端样式问题，也不是应该靠清洗把内容压掉的问题。根因是任务开启链路把一次正常的模型反馈与工具执行交接拆错了层：

```text
模型/动作契约给出开局反馈
-> task_lifecycle 只把它作为 transient assistant_text 事件发出
-> 随后发送空 content 的 task_control done
-> 当前 assistant turn 被视为控制交接完成
-> 可持久化 assistant content 为空
-> 执行器继续跑，工具和任务投影附着到 runtime attachment
-> 页面刷新或重建后只剩工具投影，看不到正文
```

这说明当前链路存在三个严重错层：

1. `opening_judgment` 是模型面向用户的正文，但没有成为稳定、可重建的 assistant body。
2. `task_control` 是控制事件，却以 `final_answer_event(content="")` 参与 turn 收口，容易覆盖或终止正文通道。
3. 工具观察、执行器状态、任务恢复状态被投影层二次解释，部分场景会把工具事实冒充成模型反馈，或把原始值如 `true` 回退到公开摘要。

控制命令、协议字段、系统状态绝对不能暴露到正文；但修复不能靠“把东西不显示”。正确做法是先完整记录 raw trace，再按权威分层生成公开投影。

## 二、目标时序

修复后的单轮链路必须稳定为：

```text
UserInput
-> PromptPacket
-> ModelAction
-> PublicModelBody(if any)
-> ToolProjection
-> ToolObservation
-> ObservationToModel
-> ModelStageFeedback or NextAction
-> PersistedTimeline
-> FrontendRender
```

展开为用户可理解的页面顺序：

```text
用户输入
-> 当前轮容器建立
-> 临时 UI chrome：正在思考，不持久化
-> 模型开局反馈或直接正文
-> 工具动作开始
-> 工具等待/运行
-> 工具观察摘要
-> 模型基于观察给出阶段反馈或下一步
-> 最终回答或等待用户/审批/安全边界
-> 刷新后按同一顺序重建
```

禁止出现：

```text
工具先显示
-> 之后才恢复出开局反馈
```

禁止出现：

```text
task_control done
-> 空正文收口
-> 工具投影挂在空 assistant 下
```

## 三、Prompt 输入与投影时序

这一段是修复的核心。模型输入和页面投影必须是两条受同一事件链约束、但权限不同的链路：

```text
Prompt 输入链：给模型看的事实、上下文、工具观察、动作契约
公开投影链：给用户看的模型正文、工具摘要、控制状态
```

两条链不能互相偷用：

1. PromptPacket 不能直接投影到页面。
2. public projection 不能反过来当作模型事实来源，除非它指向原始 observation refs。
3. 控制命令可以进入模型输入作为运行状态，但不能进入 assistant body。
4. 工具原始结果可以进入 raw trace，也可以被压缩后进入模型输入，但不能原样进入公开正文。
5. 模型自然语言输出可以进入正文，也可以成为后续上下文；但工具窗口摘要不能冒充模型正文。

### 1. PromptPacket 输入组成

每次调用模型前必须显式生成 `PromptPacket`，并保存 `prompt_packet_ref` 供 trace 使用。它是模型输入权威，不是页面显示权威。

PromptPacket 必须包含：

| 输入块 | 内容 | 是否公开投影 |
| --- | --- | --- |
| Agent Role Prompt | agent 身份、职责、边界、输出要求 | 不直接投影 |
| User Request | 用户当前输入、附件、显式约束 | 用户消息本身公开 |
| Conversation Context | 已确认的历史正文、必要摘要、当前任务目标 | 不直接投影 |
| Active Turn State | `turn_id`、`turn_run_id`、`task_run_id`、当前 phase、是否可 steer | control surface 可投影，不能进正文 |
| Tool Catalog | 可用工具、参数 schema、权限边界 | 不公开 |
| Action Contract | 允许 `respond/ask_user/tool_call/request_task_run/block`，以及公开反馈要求 | 不公开 |
| Task Contract | `user_visible_goal`、`task_run_goal`、完成标准、产物要求 | 可公开其中用户目标，不能公开内部控制字段 |
| Runtime Observations | 工具观察摘要、错误、子 agent 返回、审批结果 | 工具摘要可公开，原始结果不直接公开 |
| Stage Boundary | 阶段边界、covered tool refs、下一阶段候选 | 阶段事实可进 control/timeline，模型总结才进 body |
| Output Schema | 模型本轮应返回正文、动作或阻塞说明的格式 | 不公开 |

PromptPacket 禁止包含：

1. 面向开发者的节点说明，例如“这是 runtime 节点，根据任务图执行”。
2. 用来补页面的假正文，例如“开始处理”“处理完成”“任务执行器已接管”。
3. 要求模型输出 `task_control`、`terminal_reason`、内部 JSON action 字段给用户的指令。
4. 把 public projection 文案当作事实重新喂给模型，而不带原始 observation refs。
5. 未标明来源和时序的旧工具结果。

PromptPacket 推荐结构：

```text
PromptPacket
├─ role_prompt
├─ user_request
├─ explicit_user_constraints
├─ visible_conversation_context
├─ active_turn_state
├─ task_contract_if_any
├─ tool_catalog
├─ permission_contract
├─ observation_context
├─ stage_boundary_context
├─ action_output_contract
└─ trace_refs
```

### 2. 首轮 prompt 输入时序

用户提交消息后，后端必须按固定顺序推进：

```text
1. user_message_persisted
2. active_turn_created
3. request_facts_normalized
4. context_candidates_selected
5. permission/tool boundary assembled
6. prompt_packet_created
7. model_call_started
8. model_delta / model_action_received
```

页面在第 2 步之后可以创建当前轮容器，并显示非正文 UI chrome：

```text
正在思考
```

但第 1 到第 7 步都不允许生成 assistant body。只有第 8 步里的模型自然语言输出，才允许成为正文或 body timeline item。

### 3. 首轮投影时序：直接回答

当模型选择直接回答：

```text
model_call_started
-> assistant_text_delta
-> assistant_text_final
-> assistant_message_persisted
-> turn_complete
```

投影规则：

1. `assistant_text_delta` 只进入正文流。
2. `assistant_text_final` 是 canonical assistant content。
3. `turn_complete` 只关闭 turn，不生成正文。
4. 如果已有 final text，terminal event 不能覆盖正文。

### 4. 首轮投影时序：工具调用

当模型选择调用工具或启动任务：

```text
model_action_received(tool_call/request_task_run)
-> public_model_body(opening_judgment, if model provided)
-> action_admitted
-> tool_or_task_projection(running)
-> active_turn_state=running_task
```

这里的顺序不能反：

```text
opening_judgment sequence N
tool projection sequence N+1
control state sequence N+2
```

如果模型没有给出真实开局反馈：

```text
model_action_received(tool_call)
-> diagnostics: public_feedback_missing
-> tool_or_task_projection(running)
-> active_turn_state=running_task
```

前端可以继续显示临时 `正在思考` 或工具动作，但不生成假正文。

### 5. 工具结果回灌 prompt 时序

工具完成后，系统必须先形成两份不同产物：

```text
RawToolResult       完整工具返回，进 trace
ToolObservation     面向模型和用户的结构化观察摘要
```

然后按顺序推进：

```text
tool_result_recorded
-> tool_observation_projected(tool_window/timeline)
-> observation_prompt_packet_created
-> model_call_started_with_observation
-> model_observation_report_or_next_action
```

工具观察给模型的 prompt 不能只说“工具完成”，必须包含：

```text
工具名称
调用目标
成功/失败
关键观察
错误摘要
artifact refs / trace refs
是否需要用户或权限介入
```

工具观察给用户的投影必须短：

```text
读取到 task_lifecycle.py 中 task_control 空收口在调度执行器后发出。
```

模型收到观察后的输出才是阶段反馈：

```text
这说明正文丢失发生在任务启动控制交接处；下一步需要把 task_control 从 assistant final answer 中拆出去。
```

### 6. 多工具与阶段总结时序

连续工具调用不能变成“裸工具链”。合法时序：

```text
opening_judgment
-> tool A running
-> tool A observation
-> tool B running
-> tool B observation
-> model stage_summary
-> covered tools folded
```

如果模型在 tool A 后继续 tool B，可以不长篇分析，但必须至少满足其一：

1. tool A 有短观察摘要。
2. tool B 的调用附带基于观察的下一步说明。
3. 到阶段边界时产出 `stage_summary`。

阶段总结出现后，前面的成功工具可以折叠，但不能删除：

```text
stage_summary.covers_tool_refs = [toolA, toolB]
```

### 7. `request_task_run` 专用时序

`request_task_run` 不能再被当成“模型把任务交给系统，所以正文结束”。它仍然属于当前 assistant turn 的模型动作。

固定时序：

```text
model_action_received(request_task_run)
-> opening_judgment persisted as assistant body or body timeline
-> task_run_contract_created
-> task_run_lifecycle_event(created/waiting_executor)
-> task_projection(running/waiting)
-> executor_scheduled control event
-> executor tool events
-> executor observations
-> observation_prompt_packet_created
-> model stage_feedback/final/next_action
```

禁止：

```text
opening_judgment transient only
-> task_control final_answer(content="")
-> turn closed
-> executor events attach to empty assistant
```

`task_executor_scheduled` 的正确归属：

```text
raw trace: yes
control surface: yes
task projection: yes
assistant body: no
final answer: no
```

### 8. Live 投影与持久化重建时序

live 期间每个公开 item 必须带稳定排序键：

```text
event_log.offset
-> public_projection.sequence
-> stream_chunk_index
-> created_at
```

持久化重建必须从同一事实源重建：

```text
assistant_messages
runtime_events
task_runs
agent_runs
work_rollout
public_projection_envelopes
```

重建规则：

1. 先恢复 assistant canonical content。
2. 再按 anchor 合并 `public_timeline`。
3. 再合并 `task_projection`。
4. 再合并 control state。
5. 最后按排序键渲染。

禁止：

```text
有 task_projection 就跳过 public_timeline
没有 anchor 就挂最近 assistant
terminal event 覆盖 assistant_text_final
空 task_control 生成 assistant message
```

### 9. 输入与投影的边界表

| 事实 | 进 PromptPacket | 进公开投影 | 进 assistant body |
| --- | --- | --- | --- |
| 用户当前输入 | 是 | 用户消息 | 否 |
| 模型自然语言正文 | 后续上下文可用 | 是 | 是 |
| 模型 action JSON | 结构化解析后可用 | 否 | 否 |
| 工具调用参数 | 必要时作为 trace refs | 工具目标摘要 | 否 |
| 工具原始结果 | 压缩或引用后可用 | 否，除非用户展开 debug | 否 |
| 工具观察摘要 | 是 | tool window/timeline | 否 |
| 模型基于观察的判断 | 后续上下文可用 | 是 | 是 |
| task_control | 运行状态可用 | control surface | 否 |
| terminal_reason | 运行状态可用 | control/error surface | 否 |
| prompt_packet | trace refs only | 否 | 否 |

## 四、权威分层

| 层 | 允许内容 | 禁止内容 | 当前问题 | 修复目标 |
| --- | --- | --- | --- | --- |
| Raw Trace | 所有模型、工具、控制、错误、stdout、协议事件 | 删除事实 | 部分事实只在 live 中可见 | 原始事实完整保留，不承担公开显示 |
| Normalized Event | typed event、refs、offset、anchor | 生成用户正文 | 工具/控制事件被二次解释 | 只做结构化归一 |
| Assistant Body | 模型自然语言反馈 | 工具 JSON、控制命令、terminal reason、空状态 | `opening_judgment` 未稳定持久化 | 只有模型内容能进入 body |
| Tool Window | 工具开始、等待、完成、失败、观察摘要 | 模型阶段结论 | 工具观察可能被冒充为 body | 工具事实只在 tool surface |
| Control Surface | pause、resume、approval、waiting、recoverable、error | assistant 正文 | `task_control` 参与 final answer | 控制只改状态，不写正文 |
| Public Projection | 可展示 timeline item | 原始协议、debug trace | 清洗失败后回退到 raw string | 公共摘要必须是显式可见文本 |
| Session Timeline | live 等价重建 | 用最近 assistant 猜归属 | 空正文 + attachment 导致重建缺 body | 按 anchor 和 event offset 重建 |
| Frontend Render | 展示正文、工具、阶段、控制 | 改写事实或补假正文 | fallback 容易制造状态文案 | envelope/anchor first，UI chrome 不持久化 |

硬规则：

1. `slot=body` 只允许 `source_authority=model` 且 `surface=assistant_body`。
2. 工具观察默认不是模型正文；只有模型收到观察后产生的自然语言判断才是正文。
3. `task_control`、`terminal_reason`、`done`、`assistant_message` 不得生成 assistant content。
4. 空 content 的控制收口不能覆盖已有正文，也不能作为“处理完成”展示。
5. 清洗函数返回空时，不能回退到原始布尔值、JSON、协议文本或内部状态词。

## 五、当前代码证据

### 1. 任务开启正文未持久化

文件：`backend/harness/loop/task_lifecycle.py`

当前 `start_task_lifecycle_from_contract()` 先发出开局反馈：

```python
if opening_content:
    yield assistant_text_event(
        content=opening_content,
        answer_channel="opening_judgment",
        answer_source=f"{answer_source}.opening_judgment",
    )
```

随后调度执行器成功后又发出空正文控制收口：

```python
yield final_answer_event(
    content="",
    answer_channel="task_control",
    answer_source=answer_source,
    terminal_reason="task_executor_scheduled",
    extra={...},
)
```

这里的问题不是“空内容显示出来”，而是空的 `task_control` 收口占用了 turn 生命周期。它表达的是运行时交接，不是模型对用户的最终回答，也不能关闭正文通道。

### 2. 系统状态被写成公开进度

同一文件中还写入了泛化状态：

```python
started_summary = "已接手任务，正在整理执行步骤。"
scheduled_summary = "任务执行器已接管，正在推进第一步。"
```

这些不是模型正文，也不是用户需要的任务分析。它们最多是内部 lifecycle trace 或 control surface 状态，不应作为主会话公开反馈，更不应替代模型开局反馈。

### 3. 清洗失败后回退到原始值

文件：`backend/harness/loop/work_rollout.py`

当前逻辑：

```python
visible_summary = public_runtime_progress_summary(summary) or str(summary or "").strip()
brief = public_runtime_progress_summary(agent_brief_output) or str(agent_brief_output or "").strip()
```

如果 `summary` 是 `true`、原始 JSON、协议片段或内部字段，清洗函数返回空后又被 `str(summary)` 放回公开链路。这会直接制造“看起来像被清洗了，实际又泄露回来”的问题。

### 4. 工具观察可能被标成模型正文

文件：`backend/harness/runtime/public_timeline_stream.py`

当前 `_runtime_step_summary_items()` 在非工具、非状态步骤里会强制生成：

```python
kind="observation_report"
slot="body"
surface="assistant_body"
source_authority="model"
```

如果这个 step 的来源是工具结果、执行器诊断或系统构造摘要，就会把非模型内容提升成模型正文。`observation_report` 必须区分两类：

```text
tool_observation     工具事实，tool_window
observation_report   模型基于工具观察后的判断，assistant_body
```

### 5. `current_action` 仍可能使用泛化状态

文件：`backend/harness/runtime/session_task_projection.py`

当前 fallback 仍会使用：

```python
monitor.latest_step_summary
latest_step.summary
monitor.summary
"正在处理"
```

这类 fallback 只能作为非正文 UI chrome 或内部状态，不能作为持久公开反馈。尤其 `latest_step_summary=true` 时，当前动作标题会变成 `true`，这是公开投影契约失败。

## 六、修复原则

### 1. 先记录，再分层，不靠前端遮挡

Raw trace 必须完整：

```text
模型原始输出
工具调用参数
工具返回
控制状态
错误
恢复
调度
```

公开投影必须严格：

```text
模型正文 -> assistant body
工具事实 -> tool window
控制状态 -> control surface
错误摘要 -> error/control + model-visible observation
debug trace -> diagnostics
```

### 2. 开局反馈必须是真实模型反馈

合法开局：

```text
我先检查任务开启链路里正文在哪里丢失，再对照工具投影和持久化重建。
```

非法开局：

```text
开始处理
正在处理
处理完成
任务执行器已接管
```

如果模型没有给开局反馈：

```text
页面可显示临时“正在思考”或工具动作
后端记录 public_feedback_missing
不伪造正文
不阻塞工具执行
```

### 3. 工具反馈短，阶段总结承载判断

工具反馈只回答：

```text
这个工具做了什么，观察到了什么，是否失败。
```

阶段总结回答：

```text
这些观察意味着什么，当前阶段是否结束，下一阶段做什么。
```

推荐页面顺序：

```text
开局判断：我先追查任务开启和投影归属。
工具：读取 task_lifecycle.py。
工具反馈：看到 task_control 空收口在调度执行器后发出。
工具：读取 work_rollout.py。
工具反馈：发现清洗失败后会回退到原始 summary。
阶段总结：正文丢失发生在 task start 控制交接和持久化正文权威之间；下一步修正文持久化与控制收口分离。
```

### 4. 错误必须返回，不允许阻碍 agent

任何工具失败、审批拒绝、执行器 lease 丢失、安全边界等待都必须产生：

```text
结构化 runtime/control event
用户可理解的公开状态
模型可见 observation
可恢复/不可恢复标记
```

不能静默等待，不能只在前端显示一个空状态，更不能把内部错误栈或控制命令塞进正文。

## 七、Prompt 配套

### 1. 任务开启 prompt

应写给 agent：

```text
你正在处理用户的当前任务。
你可以直接回答、请求补充信息、调用工具、启动子任务，或在无法继续时说明阻塞原因。

如果你需要调用工具或启动任务，请先给用户一句真实、具体、简短的公开反馈：
- 说明你已经判断到什么，或你接下来需要确认什么；
- 不要预测工具结果；
- 不要写“开始处理”“正在处理”“处理完成”；
- 不要输出 JSON、工具协议、action_type、tool_call、task_control 或内部控制字段。

如果你还没有形成任何判断，不要编造开局反馈；系统会使用非正文 UI 状态表示等待。
```

### 2. 工具观察后 prompt

```text
你已经收到工具观察。
请先判断这个观察对用户任务意味着什么，再决定继续调用工具、进入下一阶段、请求用户补充信息或收口回答。

如果继续调用工具：
- 工具反馈只保留一句事实；
- 如果已经形成阶段结论，请输出阶段总结；
- 阶段总结需要说明上一阶段结论、关键依据和下一阶段方向。

不要连续裸调用工具。
不要把工具错误、控制命令、协议字段或 JSON action 复制给用户。
```

### 3. 子 agent prompt

```text
如果你需要启动子 agent，请先说明为什么需要它处理这部分工作。
子 agent 运行期间你会收到等待状态、完成通知或观察摘要。
不要假装已经知道子 agent 的结果。
不要在没有完成通知时把子 agent 投影为 completed。
```

## 十、完成定义

这次修复完成必须同时满足：

1. 任务开启不再出现空白正文只剩工具。
2. 开局反馈来自模型，不来自系统状态文案。
3. `task_control` 不再作为 assistant final answer 收口正文。
4. 工具观察不会冒充模型正文。
5. 工具后的模型阶段反馈可见且可持久化。
6. `true/false`、JSON、工具协议、控制命令不会进入公开摘要。
7. `开始处理/处理完成/任务执行器已接管` 不进入正文或持久活动。
8. `task_projection` 与 `public_timeline` 共存，互不覆盖。
9. live stream 与刷新重建顺序一致。
10. 等待、审批、暂停、恢复、错误都有结构化返回，不阻碍 agent，也不泄露控制系统。
11. 工具、观察、阶段总结、最终回答都能追溯到明确 `turn_id/run_id/task_run_id/anchor_message_id`。
12. 每次模型调用都有可追溯 `PromptPacket`，且 PromptPacket 不直接投影到页面。
13. 工具结果必须通过 observation prompt 回灌模型，不能由投影层代替模型写阶段判断。

## 十一、实施顺序

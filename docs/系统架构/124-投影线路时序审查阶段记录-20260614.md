## 3. 已确认的线路问题

### P1：前端把 assistant body 压成单块，只用最后一个 body offset 参与混排

位置：

- `frontend/src/lib/projection/reducer.ts`
- `frontend/src/components/chat/ChatMessage.tsx`

当前逻辑：

- `body_append` 只把文本累加到 `ledger.body.text`。
- `messagePublicProjectionFromLedger()` 只输出一个 `bodyEventOffset`，取 `body.source_offsets` 的最后一个 offset。
- `ChatMessage` 把整个正文作为一个 block，用这个最后 offset 和工具 timeline item 排序。

风险：

如果事件真实顺序是：

```text
1 assistant_text_delta: "我先检查"
2 tool_call_requested
3 tool_item_started
4 tool_item_completed
5 assistant_text_delta: "检查完成"
```

前端会把完整正文块放在 offset 5 的位置，显示成：

```text
工具轨迹
我先检查检查完成
```

而不是严格体现 offset 1 的正文已经先出现。

这违反 123 表：

```text
body block 与工具/status item 严格按 eventOffset 混排
```

待判断：

- 如果产品接受“正文整体块只按最新正文位置展示”，则 123 表需要修订。
- 如果坚持成熟 agent 实时投影，ledger 需要保留 body segment/block 级 offset，而不是只保留合并后的 body text。

### P1：工具生命周期在前端 timeline 中按 frame_id 拆成多条可见工具项

位置：

- `frontend/src/lib/projection/reducer.ts`
- `frontend/src/components/chat/ChatMessage.tsx`
- `frontend/src/components/chat/PublicTimelineActivity.tsx`
- `frontend/src/lib/projection/reducer.test.ts`
- `frontend/src/components/chat/ChatMessage.test.ts`

当前逻辑：

- `timelineItemFromFrame()` 使用 `frame_id/projection_id` 作为 timeline item id。
- `tool_call_requested`、`tool_item_completed` 会成为不同 timeline item。
- 测试已固化：

```text
projection.timeline.map(sourceEventType)
== ["tool_call_requested", "tool_item_completed"]
```

这会导致同一工具生命周期显示为多条轨迹，而不是同一个工具项被 terminalize。

违反 123 表：

```text
工具轨迹必须是一条稳定生命周期，不允许 start/return 拆成两条互不相干的 UI 记录。
completed 更新同一 tool_call_id/tool_lifecycle_id，不生成重复返回项。
```

建议方向：

- timeline upsert key 应以 `tool_lifecycle_id || tool_call_id` 为工具生命周期主键。
- frame id 仍可保留为 trace/replay 去重证据，但不应作为工具窗口主显示项 id。
- 测试需要从“保留多帧 timeline”改为“同一 lifecycle item 状态更新”。

### P1：`active_task_steer_accepted` 被接到 hidden trace，没有接到控制/状态显示面

位置：

- `backend/harness/runtime/projection/projector.py`

当前逻辑：

```text
active_task_steer_accepted -> _hidden_trace_spec()
```

风险：

用户在 active task 期间发送补充时，系统内部可能已正确 append steer，但公开投影没有给用户可见回执，用户会误以为补充没有被接收，甚至继续重复输入。

违反 123 表：

```text
active_task_steer_accepted -> status_trace/status
UI 表现：控制/任务状态显示“补充已纳入”
```

建议方向：

- 将 `active_task_steer_accepted` 投为 `status` 或 `control` surface 的 visible live item。
- 文案要是面向用户的控制事实，不要写开发式说明。

### P1：工具 permission fallback id 可能在不同层不一致

位置：

- `backend/api/chat.py`
- `backend/harness/loop/task_executor.py`
- `backend/harness/loop/single_agent_turn.py`

观察到的差异：

- `chat.py` 的 permission event fallback 倾向 `admission:{tool_call_id}`。
- `task_executor.py` 的 started event fallback 使用 `admission:{action_request.request_id}`。
- `single_agent_turn.py` 的 started event fallback 使用 `admission:{tool_call_id}`。

风险：

当 admission 缺少真实 `admission_id`，且 `request_id != tool_call_id` 时，投影生命周期校验会认为：

```text
tool_permission_decided.permission_decision_id != tool_item_started.permission_decision_id
```

结果是工具 started/completed 被打成 protocol diagnostic，正常工具轨迹断裂。

违反 123 表：

```text
tool_item_started 必须绑定已允许的 permission_decision_id
permission_decision_id 不能替代 tool_call_id，但必须作为同一授权证明稳定传递
```

建议方向：

- admission fallback id 统一使用同一个函数。
- 更成熟的做法是：admission 层保证总是生成 admission_id；投影层不临时发明多个 fallback。

### P2：chat bridge 中 `tool_call_id` 仍有 `request_id` fallback

位置：

- `backend/api/chat.py`

当前逻辑：

```text
tool_call_id = tool.id or request.tool_call_id or request.request_id
```

风险：

123 表允许“模型缺失 tool_call.id 时 runtime 生成一次”，但这里使用 action `request_id` 直接充当工具身份，会重新模糊：

- `tool_call_id`
- `request_id`
- `action_request_ref`

这会让 trace/action ref 和工具 item identity 再次混在一起。

建议方向：

- 缺模型 `tool_call.id` 时，应由模型动作标准化层生成 canonical tool id，并写回 action request。
- chat bridge 只读取 canonical `tool_call_id`，不要自己用 `request_id` 发明身份。

## 4. 需要继续核对的项目

### 4.1 public event_offset 是否完全不受 source offset 污染

已看到：

- `_append_chat_public_event()` 用 `next_sequence = current.latest_event_offset + 1` 构建 projection frame。
- task bridge 会把 task event offset 放进 `source_task_event_offset`。
- `build_public_projection_frame()` 优先读取 payload 中的 `event_offset/offset`，再使用 sequence。

补充审查结论：

- 规范化的 assistant/tool/commit/terminal 事件没有明显把 task source offset 写成 `event_offset`。
- `_apply_bridge_context_to_public_data()` 只写 `source_task_event_offset`，没有写 `event_offset`。
- SSE replay/cursor 的 `event_offset` 来自 public event log offset，符合单一 public clock。
- 仍有一个隐性错接：`build_public_projection_frame()` 优先读取 payload 中的 `event_offset/offset`，再使用 `_append_chat_public_event()` 传入的 public sequence。如果未来某个旁路事件或未知事件携带 raw `event_offset`，frame 内部排序 offset 会偏离 SSE/public event log offset。

风险：

一旦 task source offset 被当成 public frame event_offset，前端排序会跨 task/turn 错位。

建议：

- `PublicProjectionFrame.event_offset` 应只由 public append sequence 写入。
- source offset 必须在 `source_*_event_offset` 或 `trace_refs` 中表达。
- `build_public_projection_frame()` 不应默认接受 payload 的 `event_offset/offset` 作为 public frame offset。

### 4.2 replay/hydrate 是否按 frame id 去重，同时不破坏 lifecycle display key

当前问题不是 frame id 去重本身，而是 display timeline 也用了 frame id。后续需要拆分：

```text
frame_id = replay/gap recovery id
tool_lifecycle_id = 工具窗口显示 id
```

补充审查结论：

- `RuntimeStreamReplayService.to_public_sse()` 使用 event log offset 输出 SSE `event_offset`。
- 前端 `streamChatRunEvents()` 用 SSE data `event_offset` 维护 cursor，能避免 replay 重复消费旧 SSE event。
- 前端 projection ledger 的工具 timeline display key 使用 frame id，导致 replay 去重 id 与工具显示 id 混在一起。

结论：

replay cursor 方向是对的；重复显示风险来自 projection ledger 显示层，而不是 SSE cursor 本身。

## 5. 最终审查结论

当前线路不是完全错接，后端协议和正文边界已经明显接近 123 表。但按 123 新时序表严格审查，至少有三条主线仍不符合：

1. 前端正文以“整块最后 offset”参与混排，不能严格还原正文 delta 与工具轨迹交错时序。
2. 工具生命周期在前端 timeline 中按 frame 拆成多条显示项，没有真正 terminalize 同一 lifecycle item。
3. active steer accepted 进入 hidden trace，缺少用户可见控制/任务状态回执。

另外，permission fallback id 和 chat bridge 的 `request_id` fallback 需要继续收束，避免在边界场景下打断工具生命周期。

补充后的完整问题列表：

| 优先级 | 问题 | 影响 |
| --- | --- | --- |
| P1 | body 只保存合并文本和最后 offset | 正文 delta 与工具轨迹交错时会显示错位 |
| P1 | 工具 timeline 以 frame id 为显示 id | 同一工具生命周期拆成多条工具窗口 |
| P1 | `active_task_steer_accepted` 投为 hidden trace | 用户补充已接收但没有可见回执 |
| P1 | permission fallback id 不统一 | admission id 缺失时工具 lifecycle 可能被打断 |
| P2 | chat bridge 仍用 `request_id` fallback 成 `tool_call_id` | 工具身份和 action request trace 再次混权 |
| P2 | `PublicProjectionFrame.event_offset` 可被 payload `event_offset/offset` 覆盖 | 旁路事件可能污染 public frame 排序 |
| P2 | 前端测试保护旧工具多项 timeline | 修复后必须同步改测试，否则测试会把旧结构拉回来 |

已确认基本符合：

| 线路 | 结论 |
| --- | --- |
| 双通道协议字段 | 已接上 |
| legacy body signal 阻断 | 已接上 |
| terminal final_answer 不合成正文 | 已接上 |
| runtime-private workspace 边界 | 已接上且有 focused regression |
| SSE replay cursor | 以 public event log offset 为准，方向正确 |

下一步不建议继续只审查，应进入修复计划：

1. 重做前端 projection ledger 的 body segment 与 tool lifecycle display key。
2. 收束后端 permission/tool id fallback。
3. 将 active steer/control 投到可见 status/control surface。
4. 锁死 `PublicProjectionFrame.event_offset` 只来自 public event clock。
5. 更新前端测试，从保护旧 timeline frame 形状改为保护新 lifecycle 时序。

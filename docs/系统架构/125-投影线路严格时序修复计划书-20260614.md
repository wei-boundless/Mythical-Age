# 投影线路严格时序修复计划书

日期：2026-06-14

状态：待确认后执行

依据：

- `docs/系统架构/123-246与双通道投影新路线时序表-20260614.md`
- `docs/系统架构/124-投影线路时序审查阶段记录-20260614.md`

目标：把当前公开投影线路修到严格符合 123 时序表。修复重点不是补 UI 文案，而是收束投影时序权威：

```text
Runtime canonical event log
-> PublicProjectionFrame(public event_offset)
-> Frontend ProjectionLedger(block/lifecycle display model)
-> ChatMessage ordered surface
```

---

## 1. 必须修复的问题

本轮只修当前审查确认的问题：

| 优先级 | 问题 | 修复目标 |
| --- | --- | --- |
| P1 | assistant body 被合并成单块，只用最后 offset 排序 | 按 body segment/block 保留 offset，正文与工具严格混排 |
| P1 | 工具生命周期按 frame id 拆成多条 timeline item | 工具显示项以 `tool_lifecycle_id || tool_call_id` 为 key，completed terminalize 同一项 |
| P1 | `active_task_steer_accepted` 只进 hidden trace | 投到可见 status/control surface，给用户当前任务补充回执 |
| P1 | permission fallback id 多层不一致 | 统一 permission decision id 生成/读取，不让 started/completed 因 fallback 断链 |
| P2 | chat bridge 用 `request_id` fallback 成 `tool_call_id` | chat bridge 只消费 canonical tool id，不自行混用 action request id |
| P2 | frame `event_offset` 可被 payload 覆盖 | `PublicProjectionFrame.event_offset` 只来自 public event clock |
| P2 | 前端测试保护旧 timeline 形状 | 改成保护 body segment 和单 lifecycle item |

---

## 2. 目标时序模型

### 2.1 正文 block 时序

当前：

```text
ledger.body.text = all body text
ledger.body.source_offsets = [1, 5]
body block offset = 5
```

目标：

```text
ledger.body.text = canonical full assistant body
ledger.body.blocks = [
  { kind: "body", text: "我先检查", firstOffset: 1, lastOffset: 1 },
  { kind: "body", text: "检查完成", firstOffset: 5, lastOffset: 5 }
]
```

规则：

1. `body_append` 到达时，如果上一条公开显示 block 也是 body，则追加到同一 body block。
2. 如果上一条公开显示 block 是 tool/status/pinned/control，则新建 body block。
3. `body_finalize` 更新 canonical `body.text` 和 body state；如果此前没有 body block，则创建 final body block。
4. `body_finalize` 不得把整个正文重放成一个最后 offset block，避免覆盖前面已按时序显示的正文。
5. `ChatMessage` 只消费 ordered blocks，不再用单个 `bodyEventOffset` 与 timeline 临时拼接。

### 2.2 工具 lifecycle 时序

当前：

```text
timeline item id = frame_id
tool_call_requested -> frame:tool-requested
tool_item_completed -> frame:tool-completed
```

目标：

```text
timeline item id = tool_lifecycle_id || tool_call_id
tool_call_requested -> create running item
tool_permission_decided -> update same lifecycle or pinned permission item
tool_item_started -> update same lifecycle as running
tool_item_completed -> terminalize same lifecycle as done/error
```

规则：

1. frame id 只用于 replay/gap recovery 和 trace，不作为工具窗口显示 id。
2. 工具 display item key 固定为 `tool_lifecycle_id || tool_call_id`。
3. 成功 completed 更新同一工具项为 `done`，可 collapsed，但不能新增第二条“返回”工具项。
4. 失败 completed 更新同一工具项为 `failed`，并可进入 pinned，但不能丢失生命周期。
5. permission allowed 可以 trace-only；permission waiting/blocked 进入 pinned/control，但仍绑定同一 `tool_call_id`。

### 2.3 Active steer/control 显示时序

目标：

```text
active_task_steer_accepted
-> status_trace/status
-> visible_live
-> 用户看到“补充要求已纳入当前任务”
```

规则：

1. steer/control 回执不是 assistant body。
2. 回执必须可见于状态/控制面。
3. 文案必须是用户可理解的事实说明，不出现开发式字段名、内部 action name 或协议名。
4. turn mismatch、terminal task、stopped task 的 blocked receipt 也要进入可见状态或 pinned block。

### 2.4 Tool id / permission id 时序

目标身份链：

```text
tool_call_id
  = model tool_call.id
  or model action normalization generated canonical id

request_id
  = action request trace ref

permission_decision_id
  = admission/admission_id
  or one centralized fallback derived from canonical tool_call_id
```

规则：

1. chat bridge 不再用 `request_id` 自行填充 `tool_call_id`。
2. task executor、single agent turn、tool runtime、chat bridge 使用同一个 canonical id helper。
3. admission 层应优先保证 `admission_id` 存在；fallback 只能集中在一个 helper 中。
4. started/completed 的 `permission_decision_id` 必须与 permission event 完全一致。

### 2.5 Public frame offset 时序

目标：

```text
PublicProjectionFrame.event_offset = chat public event log offset
source_task_event_offset = task source evidence only
```

规则：

1. `build_public_projection_frame()` 不再从 payload 的 `event_offset/offset` 推断 public frame offset。
2. frame offset 由 `_append_chat_public_event()` 传入的 public sequence 决定。
3. source offsets 全部保留在 `source_*_event_offset` 或 `trace_refs`。
4. 生命周期校验使用 public frame offset/sequence，不使用 task source offset。

## 5. 验收标准

修复完成必须满足：

1. 正文和工具轨迹严格按 public `event_offset` 展示。
2. assistant body 不再只有单个最后 offset block。
3. 工具生命周期显示项不再按 frame id 拆分。
4. active steer/control 有可见状态回执，但不进入 assistant body。
5. `tool_call_id`、`request_id`、`permission_decision_id` 分权清晰。
6. `PublicProjectionFrame.event_offset` 不受 payload/source offset 污染。
7. replay/hydrate 不重复显示工具项。
8. 所有相关后端/前端 focused tests 通过。
9. 固定端口 3000/8003 真实启动检查通过。

---

## 6. 禁止事项

1. 禁止用到达时间或 `Date.now()` 修正正文/工具顺序。
2. 禁止继续用 frame id 作为工具窗口显示主键。
3. 禁止把 tool observation 拼接进 assistant body。
4. 禁止保留旧 `bodyEventOffset + timelineItems` 作为并行主权威。
5. 禁止用 `request_id` 冒充 `tool_call_id`。
6. 禁止靠中文文案判断工具生命周期是否合并成功。
7. 禁止通过删除失败断言或降低断言制造测试通过。
8. 禁止临时切换到非固定端口验证。

---

## 7. 执行确认点

确认本计划后，按 Phase 1 到 Phase 5 执行。实施中如果发现现有数据模型无法支持 body segment 或 lifecycle display key，需要暂停并提交偏差说明，不直接用旧显示路径兜底。

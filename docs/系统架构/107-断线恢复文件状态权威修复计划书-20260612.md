# 107-断线恢复文件状态权威修复计划书-20260612

## 目标

修复 agent 在操作文件期间断线、恢复或继续任务后“看起来文件消失”的问题。

成熟目标不是让模型多搜索几遍，而是让运行时恢复链路保留明确的文件身份、工作区根、文件状态和工具观察事实。恢复后如果文件仍在项目中，agent 必须能直接继续读取该文件；如果恢复上下文真的丢失，系统必须给出结构化恢复错误，而不是让模型误判“文件不存在”。

## 架构校正

`task_environment` 不是 file_state 存储权威。

它只负责：
- 环境提示词与工作方式说明。
- 工具能力、sandbox、file policy 的默认边界。
- 资源空间声明和 UI/编排层的环境分类。

它不负责：
- 决定 runtime state 存储根。
- 保存断线恢复状态。
- 作为已读文件、已写文件、工具观察、执行记录的权威来源。

成熟路径应为：

```text
RuntimeHost.root_dir
-> RuntimeExecutionStore
-> RuntimeEventLog
-> FileStateAuthorityStore
-> RuntimeStartPacket / DynamicContext
-> Model visible recovery context
```

`task_environment.storage_space.runtime_state_root` 在 file_state 恢复链路中属于不成熟路径，应从主链路移除。环境可以声明策略，但不能成为恢复状态的来源。

## 当前证据

- 截图中 agent 报告：`search_files("mario.html")`、`search_files("mario")` 未找到，并推断 `mario.html` 不在项目根目录。
- 磁盘事实相反：`D:\AI应用\langchain-agent\mario.html` 存在，大小约 28000 字节，且 `git status -- mario.html` 显示为已修改。
- 当前 session 的动态上下文 replacement 中多次出现成功的 `read_file` 结果，路径为 `mario.html`，包含 `total_lines: 967`、`content_sha256` 和多个读取窗口。
- `WorkspaceFileService.safe_roots()` 默认搜索根为 `docs/backend/frontend/knowledge`。当这些目录存在时，默认 `search_files` 不会扫描项目根 `.`，因此根目录文件 `mario.html` 会被漏掉。
- `BoundTaskContext.to_model_visible_payload()` 当前只暴露 `plan_refs/context_refs/restore_policy`，不暴露已知任务文件；这些信息主要停留在 manifest/volatile state，恢复或压缩后模型可能丢掉“已知目标文件就是 mario.html”的稳定事实。

## 当前主链路

- 文件读取执行：
  - `backend/harness/loop/task_executor.py` 调用 `_execute_task_tool_call()`。
  - `backend/runtime/tool_runtime/tool_executor.py` 执行 native tool。
  - `backend/runtime/tool_runtime/native_tools.py` 的 `read_file` 产生 `file_state_events`。
  - `ToolRuntimeExecutor._commit_file_state_events()` 将事件写入 `FileStateAuthorityStore(execution_store.root_dir)`。

- 文件状态进入 packet：
  - `backend/harness/loop/task_executor.py::_observations_for_packet()` 从 `runtime_host.file_state_store` 读取 file_state 并注入 `execution_state.system_projection.file_state`。
  - `backend/harness/runtime/dynamic_context/manager.py` 当前还会在 projection 为空时，从 `task_environment.storage_space.runtime_state_root` 再读一次，这是要删除的次级权威。
  - `backend/harness/runtime/bound_task_context.py` 从 volatile task state 提取 `task_files` 并写入 manifest，但稳定模型可见 payload 不包含已知文件身份。

- 搜索执行：
  - `NativeSearchFilesTool` 使用 `WorkspaceFileService.safe_roots(args.roots)`。
  - 默认 roots 存在时不会 fallback 到 workspace root。
  - 根目录文件必须显式 roots `["."]` 或通过更精确的已知路径读取才能找到。

## Authority Map

| 模块 | 当前职责 | 问题 | 目标职责 | 动作 |
| --- | --- | --- | --- | --- |
| `SingleAgentRuntimeHost` | 持有 `root_dir`、event log、execution store、file_state store | 职责正确，但 packet 装配没有把它作为唯一恢复状态权威表达清楚 | runtime state 单一权威 | 保留 |
| `task_executor.py` | 执行循环、收集 observations、装配 execution_state | 已从 `runtime_host.file_state_store` 注入 file_state，这是成熟路径 | packet 前唯一 retrieve 层 | 保留并补测试 |
| `tool_executor.py` | 执行工具并提交 file_state events | 使用 `execution_store.root_dir`，与 runtime_host root 同源，职责合理 | record 层 | 保留 |
| `dynamic_context/manager.py` | 投影动态上下文 | 从 `task_environment.storage_space.runtime_state_root` 读取 file_state，是次级权威 | 只投影上游注入的 runtime state，不自己猜环境存储 | 删除 storage-root file_state fallback |
| `bound_task_context.py` | 构建稳定任务上下文 | 已知文件只进入 manifest，模型稳定恢复包不可见 | 以 runtime file_state 为来源暴露精简 `known_task_files` | 增加模型可见精简路径，不暴露内容 |
| `native_tools.py::search_files` | 定位未知文件 | 无结果时不说明搜索根限制，模型容易误判文件不存在 | 搜索工具只报告搜索事实与边界 | 增加 searched_roots/used_default_roots/omitted_workspace_root |
| legacy `SearchFilesTool` | 旧 LangChain 搜索工具 | 仍可能输出缺少边界的“没有找到匹配项” | 与 native 工具保持观察边界一致 | 同步无结果提示 |

## 目标链路

```text
Tool observation
-> FileStateAuthorityStore(runtime_host.root_dir)
-> task_executor retrieves file_state
-> DynamicContext projects retrieved runtime state
-> BoundTaskContext exposes known_task_files
-> Model uses read_file/path_exists on known paths
-> search_files only for unknown paths and reports searched roots
```

## 修复 Slice

### Slice 1：删除 environment storage 的 file_state 次级权威

文件：
- `backend/harness/runtime/dynamic_context/manager.py`
- `backend/tests/dynamic_prompt_context_projection_test.py`
- `backend/tests/editor_context_file_state_bridge_test.py`

改动：
- `DynamicContextManager` 不再通过 `dynamic_context_storage_root()` 读取 `FileStateAuthorityStore`。
- file_state 只接受上游 `execution_state.system_projection.file_state` 和 editor context。
- 删除或改写依赖 `runtime_assembly.task_environment.storage_space.runtime_state_root` 自动恢复 file_state 的测试。

验证：
- 当 execution_state 提供 file_state，packet 仍投影文件状态。
- 当只有 task_environment storage root 而 execution_state 无 file_state，不再读取环境路径。

### Slice 2：从 runtime_host file_state 形成稳定恢复文件身份

文件：
- `backend/harness/runtime/bound_task_context.py`
- `backend/tests/dynamic_prompt_context_projection_test.py`

改动：
- `BoundTaskContext.to_model_visible_payload()` 增加 `known_task_files`。
- 来源仅限 volatile task state 中的 `file_state`，也就是 task_executor 从 `runtime_host.file_state_store` 注入的状态或 editor context。
- 精简字段只包含 `path/status/total_lines/content_sha256/next_suggested_read/last_observation_ref`，不放代码内容。
- restore policy 明确：已知路径应直接 `read_file` 或 `path_exists`，不要先把已知路径降级成 broad `search_files`。

验证：
- 编译 packet 后稳定段包含 `known_task_files`。
- 第二次读取窗口增长时，稳定段 hash 可以变化，因为文件身份恢复比缓存稳定更重要；但不包含大内容。

### Slice 3：搜索工具补边界诊断

文件：
- `backend/runtime/tool_runtime/native_tools.py`
- `backend/capability_system/tools/tool_units/search_files_tool.py`
- `backend/tests/workspace_file_tools_regression.py`

改动：
- native `search_files` structured payload 加入 `searched_roots`、`used_default_roots`、`workspace_root`、`omitted_workspace_root`。
- 默认 roots 未包含 `.` 且无结果时，文本提示未搜索项目根，并建议已知路径用 `read_file/path_exists`，未知根目录文件用 `roots=["."]`。
- legacy `SearchFilesTool` 同步提示。
- 不把默认搜索改成全项目深扫，避免卡顿。

验证：
- 根目录有 `mario.html`，默认 search no-match payload 必须说明 omitted workspace root。
- `roots=["."]` 可以找到根目录文件。

### Slice 4：运行时恢复回归测试

文件：
- `backend/tests/read_file_authority_chain_regression.py`
- `backend/tests/dynamic_prompt_context_projection_test.py`

改动：
- 补测试：工具 read_file 写入 `execution_store.root_dir` 后，`runtime_host.file_state_store` 路径可被 task_executor 注入 execution_state。
- 补测试：没有 runtime_host 注入 file_state 时，DynamicContext 不从 task_environment storage 偷读。

## 不做的事

- 不把 file_state 绑定到 `task_environment.storage_space.runtime_state_root`。
- 不把 `task_environment` 写成运行状态存储权威。
- 不删除或回滚 `mario.html`。
- 不把 `search_files` 默认改成全仓库深扫。
- 不用 prompt 安慰模型“继续找”，而是让 RuntimeStartPacket 给出事实。

## 审核结论

之前把 `task_environment.storage_space.runtime_state_root` 当作 file_state 权威是不成熟路径。正确修复是：运行时状态由 `RuntimeHost` 和 task run execution state 管理；环境只描述提示词和边界。断线恢复必须从 runtime state 恢复已知文件身份，搜索工具只报告它实际搜过哪里。

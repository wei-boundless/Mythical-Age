# 081-图任务节点交互与内置配置监管节点设计草案

日期：2026-06-09  
状态：设计草案，供讨论，不作为最终实施计划  
关联计划书：`080-图任务系统强语义升级计划书-20260609.md`

## 1. 一句话目标

让每个图节点都具备可审查、可控的单独交互入口，并内置两个系统节点：

```text
__configurator__：配置节点，帮助用户配置图任务。
__supervisor__：监管节点，自动检查和维护图任务运行。
```

但这两个节点都不能绕过系统权威：

```text
配置节点不能直接发布。
监管节点不能直接改发布契约。
所有变更必须经过 compiler 或 runtime contract。
```

## 2. 为什么需要节点单独交流

图任务不是只要点“运行”就结束。用户需要在不同阶段和节点交互：

- 配置阶段：问某个节点应该做什么、需要哪些输入、用哪个 agent。
- 编译阶段：解释这个节点的 contract、环境、权限和边关系。
- 运行前：给某个节点补充材料或澄清要求。
- 运行中：查看节点为什么阻塞、是否需要人工输入。
- 运行后：询问节点结果、审查输出、请求重跑或返修。

如果用户只能和整个 graph 对话，图会变成黑盒。节点级交流能让用户更容易部署、审查和维护图任务。

## 3. 核心概念：NodeInteractionContract

建议给每个节点编译出：

```text
NodeInteractionContract
```

它不是普通边交接，也不是 A2A。它是用户和单个节点之间的控制面交互契约。

建议结构：

```yaml
node_interaction_contract:
  node_id: review
  allowed_modes:
    - explain_config
    - inspect_context
    - clarify_before_run
    - provide_user_input
    - review_result
    - request_retry
  forbidden_modes:
    - mutate_published_contract
    - expand_permissions
    - switch_environment
  context_projection:
    include:
      - node_contract
      - inbound_packet_summaries
      - artifact_refs
      - latest_result_summary
      - compile_report_entry
    exclude:
      - raw_credentials
      - hidden_prompt_material
      - unauthorized_memory
  action_policy:
    safe_actions:
      - explain
      - create_user_input_packet
      - create_retry_request
    requires_compile:
      - change_node_agent
      - change_prompt_contract
      - change_edge_contract
      - change_environment
    requires_human_approval:
      - skip_node
      - override_verdict
      - force_commit_resource
```

## 4. 节点单独交流的模式

### 4.1 配置解释 explain_config

用户问：

```text
这个 review 节点负责什么？
它为什么需要 artifact refs？
它会写记忆吗？
```

系统回答来自：

- node contract。
- edge contract。
- resource contract。
- compile report。

不能让节点自由编造自己的职责。

### 4.2 上下文审查 inspect_context

用户查看节点将收到什么：

- inbound FlowPacket。
- artifact refs。
- memory projection。
- loop context。
- human input packet。

必须显示可见性来源和过滤规则。

### 4.3 运行前澄清 clarify_before_run

用户对某个 pending 节点补充要求。

补充不是直接写进 prompt，而是生成：

```text
UserInputPacket
```

然后由 compiler/runtime 判断是否允许进入该节点的 input package。

### 4.4 运行中介入 provide_user_input

如果节点 blocked 或 waiting_human_gate，用户可以提供输入。

输入进入：

```text
HumanInterventionPacket
```

不能直接改 GraphLoopState。

### 4.5 结果审查 review_result

节点完成后，用户可以询问：

- 结果来自哪些输入。
- 用了哪些 artifact。
- 输出是否满足 contract。
- 为什么失败。
- 是否可以重试。

这类交流读取 trace 和 receipts，不重写历史。

### 4.6 请求重试 request_retry

用户可以请求重跑节点，但必须生成：

```text
RetryRequest
```

并由 retry policy、permission ceiling、loop contract 判断能否执行。

## 5. 节点交互不是绕过边契约

节点交流只属于控制面，不属于普通图数据面。

错误做法：

```text
用户直接给节点塞新 prompt。
用户直接改节点环境。
用户直接让节点跳过上游边。
用户直接把对话内容写入下游上下文。
```

正确做法：

```text
用户消息
-> NodeInteractionController
-> NodeInteractionContract validation
-> UserInputPacket / HumanInterventionPacket / RetryRequest
-> GraphRuntime 按契约处理
```

## 6. 内置配置节点：__configurator__

配置节点用于帮助用户建立图任务配置。它属于 authoring / compiler control plane，不是普通运行节点。

职责：

- 询问用户目标。
- 识别需要哪些节点。
- 推荐 agent role。
- 推荐资源。
- 推荐边契约原型。
- 生成 AuthoringSpec 草案。
- 解释 compile report。
- 帮用户修复 compile errors。

输入：

- 用户目标。
- 当前项目环境。
- agent profile registry。
- resource registry。
- EdgeContractPrototype registry。
- NodeIntentTemplate registry。
- compile report。

输出：

```yaml
AuthoringSpecDraft
NodeSeedDrafts
ResourceBindingDrafts
EdgePrototypeSelections
RequiredCredentialRequests
CompileFixSuggestions
```

边界：

- 不能直接发布 graph config。
- 不能直接创建权限。
- 不能直接写 credential。
- 不能绕过 compile validation。
- 不能把用户自然语言直接当 runtime prompt。

推荐 prompt contract：

```text
你是图任务配置助手。
你负责把用户的目标整理成可编译的图任务草案。
你需要识别需要哪些节点、资源和边契约原型。
你不能发布任务图，也不能绕过编译器校验。
如果缺少关键环境、凭证、资源或验收标准，你必须指出缺口并请求用户补充。
你的输出必须是结构化配置草案和解释，不是运行时任务结果。
```

## 7. 内置监管节点：__supervisor__

监管节点用于运行期自动检查和维护任务运行。它属于 runtime control plane，可以表现为系统内置节点或 sidecar monitor。

职责：

- 观察 graph run 状态。
- 检查节点阻塞、失败、超时、缺 receipt。
- 检查 edge packet 是否丢失或不符合 contract。
- 检查 loop 是否超过上限或停不下来。
- 检查 memory candidate 是否错误可见。
- 检查 external endpoint 是否失败。
- 推荐或触发受控维护动作。

输入：

- GraphLoopState。
- GraphHarnessConfig。
- compile report。
- work order index。
- result history。
- edge states。
- packet refs。
- receipts。
- runtime monitor events。

输出：

```yaml
SupervisorObservation
MaintenanceActionCandidate
HealthStatus
RiskAlert
HumanApprovalRequest
AutoRecoveryReceipt
```

## 8. 监管节点可自动做什么

可以自动做的动作必须写进 `MaintenanceContract`。

低风险自动动作：

- 标记可恢复 blocked node。
- 在 retry policy 允许范围内 requeue。
- 重新投递丢失但可重放的 FlowPacket。
- 生成缺失诊断报告。
- 通知 monitor。
- 暂停继续调度并请求人工确认。

必须人工批准的动作：

- 跳过节点。
- 修改 graph contract。
- 修改 node prompt。
- 扩大权限。
- 切换环境。
- 强制提交 memory。
- 强制通过 review。
- 删除或覆盖 artifact。

推荐 `MaintenanceContract`：

```yaml
maintenance_contract:
  auto_actions:
    - requeue_recoverable_failed_node
    - reset_source_failed_edge
    - emit_health_alert
    - request_missing_human_input
  require_human_approval:
    - skip_node
    - override_result
    - force_commit_resource
    - mutate_contract
  max_auto_recovery_attempts: 2
  audit_required: true
  receipt_required: true
```

## 9. 监管节点不是总控 agent

监管节点不能变成新的“上帝 agent”。

它不能：

- 自己决定业务结果。
- 替审核节点通过产物。
- 替用户发布新 graph。
- 自行扩大权限。
- 自行修改 node effective environment。
- 自行改变 edge contract。

它只能：

```text
观察
诊断
建议
执行已授权维护动作
请求人工确认
写入审计 receipt
```

## 10. 系统节点在图里的呈现

建议把系统节点放在 system control lane：

```text
System Control Lane
  __configurator__
  __supervisor__

User Graph Lane
  draft -> review -> revise
  memory.world
  artifact.chapter
```

它们可以在 UI 中显示，但不应混入用户业务拓扑，避免用户误以为配置节点和监管节点也是业务 agent。

在发布物中可以表示为：

```yaml
system_nodes:
  configurator:
    node_id: __configurator__
    role: authoring_assistant
    lifecycle: draft_compile
  supervisor:
    node_id: __supervisor__
    role: runtime_supervisor
    lifecycle: graph_run
```

## 11. 与 GraphDeploymentPackage 的关系

配置节点生成或修复：

```text
AuthoringSpecDraft
```

编译器生成：

```text
GraphCompilationUnit
GraphDeploymentPackage
```

监管节点维护：

```text
GraphRun
GraphLoopState
Receipts
HealthStatus
```

链路：

```text
用户
-> __configurator__
-> AuthoringSpecDraft
-> GraphCompiler
-> GraphDeploymentPackage
-> GraphRuntime
-> __supervisor__
-> Health / Recovery / Human Approval
```

## 12. 最小落地建议

第一版先做轻量版本：

### 12.1 NodeInteractionContract

支持：

- explain_config。
- inspect_context。
- review_result。
- request_retry。

暂不支持：

- 直接修改节点配置。
- 运行中自由聊天注入上下文。

### 12.2 __configurator__

支持：

- 从用户目标生成 AuthoringSpecDraft。
- 推荐节点和边契约原型。
- 解释 compile report。

### 12.3 __supervisor__

支持：

- graph run health check。
- blocked / failed / timeout 诊断。
- retry policy 范围内的 requeue 建议。
- 人工确认请求。

## 13. 验收标准

正式实现后至少证明：

- 每个节点都有可打开的交互面板。
- 用户能问某节点“你负责什么、收到什么、为什么阻塞”。
- 用户给 pending 节点补充输入时，系统生成 UserInputPacket，而不是直接改 prompt。
- `__configurator__` 能根据目标生成可编译草案。
- `__configurator__` 不能绕过编译器发布。
- `__supervisor__` 能发现 blocked / failed / timeout。
- `__supervisor__` 的自动维护动作受 MaintenanceContract 限制。
- `__supervisor__` 不能强行通过审核或扩大权限。
- 所有节点交互、配置建议、维护动作都有 trace 和 receipt。

## 14. 暂定结论

可以实现“每个节点都可以单独交流”，但必须做成控制面交互契约，而不是让用户直接改运行上下文。

内置配置节点和监管节点也应该做，但它们是系统节点：

```text
__configurator__ 帮用户配置，不发布。
__supervisor__ 帮系统监管，不越权。
```

这样才能同时做到：

- 用户部署更轻松。
- 每个节点可解释、可审查、可沟通。
- 运行中自动维护。
- 编译器仍是发布权威。
- runtime 仍只执行契约。

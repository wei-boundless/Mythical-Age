## 1. 总原则

图任务系统不是“画几个节点再用线连起来”的流程图系统，而是一个以契约为权威的多 agent 编排系统。

建议先固定五个核心对象：

```text
GraphTask
Node
Resource
Edge
Contract
```

其中：

- 节点定义“谁或什么参与图”。
- 资源定义“可被读取、写入、引用、提交、审计的对象”。
- 边定义“节点之间如何发生关系”。
- 契约定义“关系、输入、输出、权限、环境、失败和恢复的可执行规则”。
- 编译器负责把用户意图和原型编译成完整契约。

一句话：

```text
图是拓扑。
节点是参与单元。
资源是受治理对象。
边是交接核心。
契约是执行权威。
```

## 2. 权威链路

需要严格区分用户配置、编译模板、发布契约和运行历史。

建议权威链路：

```text
AuthoringSpec
-> GraphCompiler
-> GraphCompilationUnit
-> PublishedGraphContract / GraphHarnessConfig
-> GraphRuntime
-> Trace / Receipts / Checkpoints
```

各层职责：

| 层 | 职责 | 是否 runtime 权威 |
| --- | --- | --- |
| AuthoringSpec | 用户表达目标、节点、资源、关系原型 | 否 |
| Template / Prototype Library | 提供节点、边、资源、协议的可复用模板 | 否 |
| GraphCompilationUnit | 编译过程产物和解释报告 | 否 |
| PublishedGraphContract | 已发布的完整执行契约 | 是 |
| GraphRuntime | 只消费发布契约并执行 | 是，但不能改契约 |
| Trace / Receipts | 记录执行历史、恢复依据、审计证据 | 是历史事实，不是新决策权威 |

核心约束：

```text
用户配置不直接运行。
原型不直接运行。
builder agent 不直接发布。
runtime 不猜测缺失契约。
发布契约才是执行权威。
```

## 3. GraphTask 是什么

`GraphTask` 是一个可编译、可恢复、可审查的任务图定义。

它至少包含：

- 图目标。
- 控制面环境。
- 节点集合。
- 资源集合。
- 边集合。
- 图级约束。
- 输出目标。
- 编译报告。
- 发布版本。

`GraphTask` 不是单纯 DAG。它可以包含：

- 顺序依赖。
- 条件路由。
- 循环返修。
- 并行 fan-out。
- barrier / join。
- A2A 会话边。
- 人工门控。
- 资源读写。
- 子图模块。
- checkpoint / resume。

建议定义：

```text
GraphTask = Topology + Contracts + Resources + Runtime Policy + Trace Policy
```

## 4. Node 是什么

节点是图中的可寻址参与单元。它可以是 agent，也可以是资源库、工具、门控、路由器、子图模块或监控单元。

节点不是 prompt，也不是单纯函数调用。节点必须能被编译成一个或多个节点契约。

建议定义：

```text
Node = Addressable graph unit with identity, kind, ports, contract, environment boundary, executor policy, and runtime state.
```

节点至少需要：

- `node_id`
- `node_kind`
- `title`
- `role_or_resource_kind`
- `input_ports`
- `output_ports`
- `node_contract`
- `effective_environment`
- `permission_ceiling`
- `executor_policy`
- `state_policy`
- `trace_policy`

## 5. 节点分类

建议把节点分为三大类：

```text
ExecutableNode
ResourceNode
ControlNode
```

### 5.1 ExecutableNode

会执行任务、产出结果、消耗输入。

包括：

- `agent_node`
- `tool_node`
- `review_node`
- `builder_node`
- `aggregator_node`
- `router_node`

典型例子：

```text
writer
reviewer
planner
implementer
verifier
researcher
summarizer
```

ExecutableNode 的核心是执行契约：

```text
input contract
prompt contract
runtime contract
output contract
acceptance contract
environment lock
permission ceiling
```

### 5.2 ResourceNode

代表可被读取、写入、提交或引用的资源。

包括：

- `memory_repository`
- `memory_collection`
- `artifact_repository`
- `file_workspace`
- `thread_ledger`
- `progress_ledger`
- `issue_ledger`
- `runtime_state_store`
- `working_memory_store`
- `model_profile_ref`
- `tool_capability_ref`
- `external_agent_endpoint`

ResourceNode 不一定执行模型推理。它可以只是被边协议读取、写入或提交的目标。

### 5.3 ControlNode

负责控制图的流转、裁决或暂停恢复。

包括：

- `router`
- `barrier`
- `join`
- `manual_gate`
- `human_gate`
- `safety_gate`
- `runtime_monitor`
- `graph_module`

ControlNode 的重点不是产出内容，而是改变调度状态、等待状态或恢复状态。

## 6. AgentNode 是什么

`AgentNode` 是一种 ExecutableNode。它代表一个由 agent profile、任务意图、环境、工具权限和 prompt contract 组成的执行实例。

AgentNode 不应该由用户手写所有底层字段。建议链路：

```text
NodeSeed
-> NodeInstanceBuilderAgent
-> NodeConfigurationInstanceDraft
-> GraphCompiler validation
-> CompiledNodeContract
```

关键边界：

- builder agent 只生成候选实例。
- compiler 负责校验和裁决。
- runtime 只消费发布契约。

AgentNode 的 prompt 不是开发说明，必须是 agent 可执行的角色指令。

错误：

```text
这是 review 节点。
根据图执行审核。
```

正确：

```text
你是一名质量审核员。
你只负责判断当前交付物是否满足审核标准。
你必须输出 verdict、问题证据、阻塞项和是否允许进入下一阶段。
你不负责替作者重写交付物。
```

## 7. Resource 是什么

资源是被节点和边引用的受治理对象。资源可以表现为 ResourceNode，也可以作为环境、权限、文件、记忆、artifact 或外部服务绑定在契约里。

建议定义：

```text
Resource = Governed object that can be read, written, referenced, committed, mounted, or audited by graph contracts.
```

资源至少要定义：

- resource id。
- resource kind。
- owner / namespace。
- access policy。
- visibility policy。
- lifecycle policy。
- environment binding。
- receipt policy。
- audit policy。

## 8. 资源类型

第一版建议至少支持这些资源：

### 8.1 TaskEnvironment

任务环境。用于文件访问、artifact root、memory namespace、sandbox、prompt mount。

需要区分：

```text
graph_control_environment
node_effective_environment
```

### 8.2 MemoryRepository

正式记忆库。只读、候选写入、审核提交必须分离。

典型操作：

```text
read_committed
write_candidate
commit_after_review
finalize
```

### 8.3 ArtifactRepository

产物库。用于交付物、报告、patch、章节、研究包、会话结论。

典型操作：

```text
materialize
reference
read_context
commit
publish
```

### 8.4 FileWorkspace

文件工作区。必须受环境和权限约束。

典型操作：

```text
read
write
diff
patch
rollback
```

### 8.5 ToolCapability

工具能力。不是资源本身，但必须被权限 ceiling 管理。

典型能力：

```text
shell
browser
web_search
file_edit
test_runner
mcp_tool
model_call
```

### 8.6 Ledger

账本资源。记录线程、进度、问题、审计、决策。

典型类型：

```text
thread_ledger
progress_ledger
issue_ledger
audit_ledger
decision_ledger
```

### 8.7 RuntimeStateStore

运行状态资源。用于 checkpoint、resume、edge session history、work order state。

### 8.8 ExternalEndpoint

外部 agent 或服务端点，例如外部 A2A agent、webhook、API 服务。

必须有：

- capability discovery。
- authentication / authorization。
- content type policy。
- timeout / retry。
- trace mapping。

### 8.9 HumanActor

人工参与者。用于审批、裁决、补充输入和风险确认。

必须通过 human gate contract 进入图，而不是作为隐式旁路。

## 9. Edge 是什么

边是节点之间关系和交接的核心权威。边不是“线”，也不是“场景标签”。边必须编译成边契约。

建议定义：

```text
Edge = Contracted relation between graph units that defines activation, data exchange, protocol, visibility, reliability, failure propagation, security, and trace.
```

边至少回答：

- 源是谁。
- 目标是谁。
- 业务关系是什么。
- 使用什么边协议。
- 是否阻塞调度。
- 传什么 payload。
- 目标输入槽是什么。
- 是否需要 ack。
- 超时和重试怎么处理。
- 失败怎么传播。
- 是否跨环境。
- 哪些内容可见。
- 如何 checkpoint / resume。
- 如何审计。

边的核心产物是：

```text
CompiledEdgeContract
```

## 10. EdgeContract 是什么

`EdgeContract` 是边的可执行契约。

建议结构：

```yaml
edge_id: edge.draft.review
source_node_id: draft
target_node_id: review

relation:
  relation_id: writing.draft_to_review
  meaning: 草稿进入审核

protocol:
  kind: node_handoff
  direction: unidirectional

scheduler:
  semantic_role: control
  scheduler_role: dependency
  wait_policy: wait_required_contracts

packet:
  payload_contract_id: contract.writing.draft_review_packet
  source_output_selector: outputs.chapter_text
  target_input_slot: review_material
  visibility: bounded_payload_and_refs

reliability:
  ack_required: true
  timeout_policy: fail_closed
  retry_policy: bounded

security:
  source_environment_id: env.writer
  target_environment_id: env.review
  cross_environment_policy: packet_only

failure:
  propagation_policy: fail_downstream

trace:
  persist_packet: true
  receipt_required: true
  checkpoint_policy: edge_packet_replayable
```

## 11. 边协议是什么

边协议是 EdgeContract 的交互实现方式。A2A 只是其中一种。

建议第一版边协议族：

| 协议 | 作用 | 是否 agent 对话 | 默认调度语义 |
| --- | --- | --- | --- |
| `control_dependency` | 只表达激活依赖，不传主要数据 | 否 | `dependency` |
| `node_handoff` | 单向交付包传递 | 否 | `dependency` |
| `conditional_route` | 根据结构化输出选择分支 | 否 | `conditional_dependency` |
| `review_feedback` | 审核反馈和返修要求 | 否 | `conditional_dependency` |
| `resource_read` | 读取 memory / artifact / file 上下文 | 否 | `context` |
| `resource_write_candidate` | 写入候选资源 | 否 | `commit` |
| `resource_commit` | 审核后正式提交资源 | 否 | `commit` |
| `event_signal` | 非阻塞状态、事件、通知 | 否 | `event` |
| `a2a_session` | agent 间有限轮通信、澄清、协商 | 是 | `conditional_dependency` 或 `context` |
| `human_gate` | 人工批准、裁决、补充输入 | 人参与 | `conditional_dependency` |
| `barrier_join` | 多分支汇合、投票、聚合 | 否 | `dependency` |
| `graph_module_port` | 父图与子图模块接口 | 否 | `dependency` |
| `audit_observation` | 审计观察，不改变主链路 | 否 | `audit` |

判断原则：

```text
能用非对话协议表达，就不要用 A2A。
只有需要多轮澄清、协商、争议解决或共同定案时，才使用 a2a_session。
```

## 12. 边协议执行器

runtime 可以按协议族拆执行器：

### 12.1 FlowPacketEdgeExecutor

执行：

- `control_dependency`
- `node_handoff`
- `conditional_route`
- `review_feedback`

核心产物：

- FlowPacket。
- inbound context。
- delivery receipt。

### 12.2 ResourceEdgeExecutor

执行：

- `resource_read`
- `resource_write_candidate`
- `resource_commit`

核心产物：

- resource projection。
- candidate receipt。
- commit receipt。
- visibility clock。

### 12.3 SessionEdgeExecutor

执行：

- `a2a_session`
- `human_gate`

核心产物：

- session history。
- conversation artifact。
- human decision packet。
- resume token。

### 12.4 EventAuditEdgeExecutor

执行：

- `event_signal`
- `audit_observation`

核心产物：

- event record。
- audit record。
- idempotency receipt。

### 12.5 ModulePortEdgeExecutor

执行：

- `graph_module_port`

核心产物：

- scoped input packet。
- scoped output packet。
- parent-child graph trace mapping。

## 13. 原型是什么

原型不是执行权威。原型是用户可选择的边契约模板。

建议定义：

```text
EdgeContractPrototype = User-facing semantic preset that compiles into one or more EdgeContracts.
```

原型负责降低配置复杂度：

```text
用户选择：交付审核
编译器生成：node_handoff + packet contract + review input slot + verdict schema + prompt patch + fail-closed rules
```

原型不能绕过：

- node effective environment。
- permission ceiling。
- payload schema。
- artifact / memory policy。
- runtime checkpoint。
- compile report。

## 14. 第一批边契约原型

建议先提供 12 个原型，其中 MVP 优先 8 个。

| 原型 | 核心边协议 | 作用 |
| --- | --- | --- |
| `prototype.review_submission` 交付审核 | `node_handoff` | 上游交付物进入审核 |
| `prototype.revision_cycle` 返修闭环 | `review_feedback` + `node_handoff` | 审核不通过后返修再复审 |
| `prototype.plan_to_execution` 规划交付 | `node_handoff` | 计划交给执行者 |
| `prototype.execution_verification` 执行验收 | `node_handoff` | 执行结果交给验证者 |
| `prototype.evidence_handoff` 证据移交 | `node_handoff` | 研究证据交给写作、决策或审核 |
| `prototype.bounded_consensus` 协商定案 | `a2a_session` | 多 agent 有限轮讨论并产出结论 |
| `prototype.clarification` 澄清问答 | `a2a_session` | 执行前短轮次澄清关键阻塞 |
| `prototype.parallel_specialist_review` 并行会审 | `node_handoff` + `barrier_join` | 多专家并行审查同一交付物 |
| `prototype.aggregated_decision` 汇总裁决 | `barrier_join` | 多分支结果汇总成统一裁决 |
| `prototype.memory_candidate_commit` 记忆候选提交 | `resource_write_candidate` + `resource_commit` | 候选记忆经审核后提交 |
| `prototype.risk_approval` 高风险授权 | `human_gate` 或 `review_feedback` | 高风险动作前审批 |
| `prototype.progress_notification` 状态通知 | `event_signal` | 非阻塞进度和状态记录 |

MVP 建议：

```text
交付审核
返修闭环
规划交付
执行验收
证据移交
协商定案
并行会审
汇总裁决
```

## 15. 原型示例

### 15.1 交付审核

用户写：

```yaml

  - draft -> review: 交付审核
```

编译器生成：

```yaml
prototype: prototype.review_submission
edge_contract:
  protocol: node_handoff
  scheduler_role: dependency
  payload_contract: contract.deliverable.review_packet
  target_input_slot: review_material
  required_target_outputs:
    - verdict
    - issues
    - evidence
    - next_action
prompt_patch:
  target_agent: quality_reviewer
  instruction: 你是一名质量审核员。你需要根据审核标准判断交付物是否达标，并输出 verdict、问题证据和是否允许进入下一阶段。
fail_closed:
  - missing_deliverable_ref
  - missing_review_verdict_schema
```

### 15.2 协商定案

用户写：

```yaml
edges:
  - reviewer <-> writer: 协商定案
```

编译器生成：

```yaml
prototype: prototype.bounded_consensus
edge_contract:
  protocol: a2a_session
  scheduler_role: conditional_dependency
  turn_policy:
    mode: bounded_round_robin
    max_turns: 4
  output_artifact_contract: contract.consensus_decision
  transcript_visibility: summary_and_refs

  - missing_stop_condition
  - missing_output_artifact_contract
  - transcript_exposed_as_default_context
```

### 15.3 记忆候选提交

用户写：

```yaml
edges:
  - draft -> memory.world: 候选记忆
  - review -> memory.world: 审核后提交
```

编译器生成：

```yaml

  - protocol: resource_write_candidate
    visibility: candidate_not_committed
  - protocol: resource_commit
    approval_policy: approved_upstream_review_gate
    required_verdict: pass
    visible_after: next_clock
fail_closed:
  - candidate_visible_as_committed
  - missing_approval_source
  - missing_commit_receipt_policy
```

## 16. 编译器职责

强编译器需要把干净定义落实成发布契约。

建议编译流程：

```text
Normalize AuthoringSpec
-> Resolve Nodes
-> Build Node Instances
-> Resolve Resources
-> Resolve Edge Prototypes
-> Synthesize Node Contracts
-> Synthesize Edge Contracts
-> Bind Environments
-> Constrain Permissions
-> Verify Runtime Semantics
-> Publish Graph Contract
```

关键原则：

- 缺节点契约，不发布。
- 缺边契约，不发布。
- 缺资源权限，不发布。
- 缺环境绑定，不发布。
- 缺恢复策略，不发布。
- A2A 没有停止条件，不发布。
- resource commit 没有 approval policy，不发布。

## 17. Runtime 职责

runtime 不负责解释用户意图。runtime 只执行发布契约。

runtime 做：

- 创建 graph run。
- 读取 published graph contract。
- 生成 work order。
- 执行节点。
- 执行边协议。
- 写入 packet、receipt、trace、checkpoint。
- 根据契约推进调度。
- 从 checkpoint 恢复。

runtime 不做：

- 不重新选择 agent。
- 不重新推断环境。
- 不临时生成边协议。
- 不扩大权限。
- 不把场景标签当执行逻辑。
- 不把 A2A transcript 默认注入下游。

## 18. UI 层应该展示什么

UI 不应该暴露低层协议字段作为默认配置。

用户默认看到：

- 节点角色。
- 资源节点。
- 工作关系原型。
- 输出目标。
- 风险提示。
- compile report。

高级视图可以展示：

- CompiledNodeContract。
- CompiledEdgeContract。
- node environment lock。
- permission ceiling。
- packet visibility。
- failure policy。
- checkpoint / resume policy。

但高级视图也不应该让用户绕过编译器直接改 runtime 发布物。

## 19. 干净 DSL 示例

用户写：

```yaml
graph_id: graph.novel.chapter_pipeline
goal: 完成章节从起草到审核、返修、提交记忆的连续任务
control_environment: env.creation.novel

nodes:
  - id: draft
    agent: writer
    intent: chapter_draft

  - id: review
    agent: reviewer
    intent: quality_review

  - id: revise
    agent: writer
    intent: chapter_revision

resources:
  - id: memory.world
    kind: memory_repository
    collection: world

edges:
  - draft -> review: 交付审核
  - review -> revise: 返修闭环 when verdict == revise
  - revise -> review: 交付审核
  - draft -> memory.world: 候选记忆
  - review -> memory.world: 审核后提交 when verdict == pass

outputs:
  - from: review
    kind: accepted_chapter
```

```text

scheduler_plan
checkpoint_policy
compile_report

```

## 20. 需要继续讨论的问题

1. ResourceNode 和 Resource 是否要严格分开？

建议：概念上分开。Resource 是受治理对象，ResourceNode 是资源在图拓扑里的表现形式。

2. EdgeContractPrototype 是否要作为一等 registry？

建议：是。它是用户可选原型和底层边契约之间的关键桥梁。

3. A2A 是否第一版实现？

建议：第一版实现内部 `a2a_session` 的编译契约和 session history；外部 A2A adapter 后置。

4. 节点环境是否进入 node contract 还是 resource binding？

建议：进入 node effective environment lock，并被 edge security 引用。

5. 原型是否允许用户自定义？

建议：允许草案，不允许直接发布。自定义原型必须通过 compiler validation、样例图编译和回归测试。

## 21. 暂定结论

图任务系统的干净定义应是：

```text
Node 表达参与单元。
Resource 表达受治理对象。
Edge 表达交接核心。
Contract 表达执行权威。
Prototype 表达用户可选模板。
Compiler 负责把模板和意图编译成契约。
Runtime 只执行契约。
```

这样可以同时满足：

- 用户低配置。
- 编译器强语义。
- 边契约是交接核心。
- A2A 只是协议族之一。
- 资源和 agent 都能作为图节点参与拓扑。
- 多 agent 连续任务可控、可恢复、可审查。

# 主会话公开投影单一权威重构计划书

日期：2026-06-10

## 1. 审查结论

当前主会话投影问题不是“工具窗口、正文、状态文案难以区分”，而是公开投影权威分裂。模型正文、工具动作、系统状态、TaskRun 生命周期、session refresh hydrate、runtime monitor 回补分别在不同模块里被重新解释，导致每次修补只能压住一个入口，换成 SSE、monitor、刷新或完成态后又从另一条旧链路漏回来。

目标重构不是继续增加过滤词，而是把公开投影收口为唯一协议、唯一 reducer、唯一聊天渲染模型：

```text
Runtime/Chat Raw Event
-> PublicProjectionEnvelope
-> MessagePublicProjection reducer
-> ChatMessage / PublicTimelineActivity pure render
```

本计划以现有代码和本地架构文档为依据，尤其对齐：

- `docs/系统架构/090-统一公开投影协议与投屏时序设计-20260610.md`
- `docs/系统架构/082-流式输出权威链路升级技术书-20260609.md`
- `docs/系统框架/077-TaskRun状态权威收口与监控投影修复方案-20260609.md`
- `backend/maintenance/agent_run_projection_single_authority_refactor_plan_20260605.md`

## 2. 当前失败模式

### 2.1 正文权威分裂

同一段最终回答可能来自以下路径：

- assistant history message 的 `content`
- `task_projection.final_answer`
- `public_timeline` / `public_timeline_delta` 中的 `final_summary`
- `done.content` 或 `receipt_summary` fallback

结果是最终正文可能重复显示、被搬进 timeline、刷新后显示位置变化，或者因为 metadata 不完整而被误藏。

### 2.2 动作展示权威分裂

后端 `session_task_projection.py` 已经给 activity 标记 `display_surface` 和 `visibility_level`，但前端 `PublicTimelineActivity.tsx` 又按 `kind` 再判断一次，把 `observation` / `action` 强行变成 tool window。这会覆盖后端投影决定。

### 2.3 实时链路与历史链路契约不一致

实时 reducer 多处执行“有 `task_projection` 就清空 `public_timeline`”，但 `session_timeline.py` 历史重建仍然同时返回 `public_timeline` 和 `task_projection`。现有 `session_runtime_timeline_contract_test.py` 还断言这种混合结构必须存在，测试保护了旧形态。

### 2.4 系统状态伪装成模型正文

`task_control`、`runtime_control`、`active_work_control`、`task_executor_scheduled` 这类系统生命周期事件仍可能经由前端 fallback 或 public timeline body 规则变成“我会开始处理”“后台任务已接管”等类似 assistant 正文的内容。

### 2.5 终态清理仍有 detail 泄漏

`session_task_projection._lifecycle_current_action()` 会在停止/暂停时替换低信号 title，但 detail 仍可能保留“执行 2 个工具调用”“agent_todo”“工具调用失败”等内部执行细节。

## 3. 当前权威地图

| 模块 | 当前角色 | 隐藏决策 | 目标角色 | 处理 |
| --- | --- | --- | --- | --- |

| `backend/harness/runtime/public_projection_envelope.py` | envelope 雏形 | source/surface 判定仍依赖已有 payload，过滤不完整 | 唯一公开投影信封构建器 | 扩展为正式 authority |

| `backend/harness/runtime/session_task_projection.py` | TaskRun 投屏生成 | 状态、活动、可见性、终态 detail 自行裁决 | Task projection adapter，消费 canonical state view | 保留但收口 |
| `backend/harness/runtime/session_timeline.py` | history hydrate | 同时返回 public timeline 与 task projection | 从持久 envelope / task projection 派生唯一 attachment | 重写 |
| `frontend/src/lib/store/events.ts` | chat stream reducer | stage/status/receipt/done content fallback | 只接收 envelope，更新 canonical stream state | 大幅清理 |
| `frontend/src/lib/store/runtime.ts` | runtime monitor store | 从 raw monitor event 生成 attachment 和 timeline | monitor envelope 调用方 | 清理 |
| `frontend/src/lib/store/publicProjectionReducer.ts` | envelope reducer 雏形 | body/timeline/task attachment 仍部分自行解释 | 唯一 MessagePublicProjection reducer | 扩展 |
| `frontend/src/lib/store/publicTimeline.ts` | timeline normalize/merge | `kind` fallback 会把无 surface 的 final_summary 视为 body | 只做格式清洗与去重 | 降权 |
| `frontend/src/components/chat/ChatMessage.tsx` | 消息渲染 | 把正文搬进 timeline、根据 activity 决定正文隐藏 | 只渲染 `MessagePublicProjection.body` | 重写 |
| `frontend/src/components/chat/PublicTimelineActivity.tsx` | 活动渲染 | 根据 `kind/surface/state` 再判 tool/body/status | 纯渲染 projection slots | 重写 |
| 相关测试 | 回归保护 | 部分断言保护混合旧结构 | 保护目标行为合同 | 更新/删除旧断言 |

## 4. 目标设计

### 4.1 三类输出不可混权

| 类别 | 唯一语义来源 | 可见位置 | 禁止 |
| --- | --- | --- | --- |
| 模型正文 | model / OutputBoundary | assistant body | runtime/status/tool fallback 伪造正文 |
| 工具和观察 | tool / runtime observation | tool window / timeline observation | 写进 assistant body |
| 系统生命周期 | system / runtime | status bar / task projection lifecycle | 冒充模型回答 |

### 4.2 统一数据结构

后端统一输出 `PublicProjectionEnvelope`。前端统一归并为 `MessagePublicProjection`。

```ts
type MessagePublicProjection = {
  body: string;
  timeline: PublicProjectionItem[];
  toolWindows: PublicProjectionItem[];
  taskProjection?: SingleAgentTaskProjection;
  controls: PublicProjectionItem[];
  terminal?: "done" | "error" | "stopped";
  tone: "running" | "waiting" | "done" | "error" | "stopped";
};
```

### 4.3 固定权威链

```text
TaskRun / TurnRun / ToolObservation / OutputBoundary
-> PublicProjectionEnvelope
-> MessagePublicProjection
-> Pure React render
```

任何模块不得从 raw event name、raw status、tool name、answer_channel、terminal_reason 重新生成用户语义正文。缺字段只能导致“不可投影/进入诊断”，不能静默 fallback 成用户可见正文。

## 5. 阶段实施计划

### Phase 0：冻结目标合同与基线

目标：先锁定合同和测试，不再继续加 UI 过滤补丁。

文件：

- `frontend/src/lib/api.ts`
- `backend/harness/runtime/public_projection_envelope.py`
- `backend/tests/public_projection_envelope_test.py`
- `frontend/src/lib/store/publicProjectionReducer.test.ts`

动作：

1. 补齐 `PublicProjectionEnvelope` / `PublicProjectionItem` / `MessagePublicProjection` 类型。
2. 增加合同测试，覆盖 source authority、surface、slot、anchor、terminal visibility。
3. 增加禁止规则测试：system/runtime/control 不得进入 assistant body。

完成条件：

- 后端 envelope test 能明确失败暴露 source/surface 混权。
- 前端 reducer test 能从 envelope 生成唯一 message projection。

### Phase 1：后端统一 envelope projector

目标：chat stream、runtime monitor、session hydrate 不再各自投影。

文件：

- `backend/api/chat.py`
- `backend/harness/runtime/public_projection_envelope.py`
- `backend/harness/runtime/runtime_monitor_public_projection.py`
- `backend/harness/runtime/public_timeline_stream.py`
- `backend/harness/runtime/session_task_projection.py`

动作：

1. 将 `_project_public_stream_event()` 改为输出 envelope。
2. `runtime_monitor_public_projection.py` 复用同一个 envelope projector。
3. `public_timeline_stream.py` 降级为 item builder，不再决定最终 `surface`。
4. `session_task_projection.py` 保留 TaskRun 投屏，但修正：
   - `current_action.detail` 同步清洗内部工具文本。
   - `activities` 严格遵守 `display_surface` / `visibility_level`。
   - `final_answer` 明确标记来源，不和 assistant body 重复。

完成条件：

- chat SSE 和 runtime monitor 对同一个 runtime event 生成同一种 envelope。
- `task_executor_scheduled` 只产生 `terminal.visible=false` 和 active turn update，不产生正文/status 文案。

### Phase 2：历史刷新链路收口

目标：session refresh 不再复活旧 timeline 或重复 final summary。

文件：

- `backend/harness/runtime/session_timeline.py`
- `backend/tests/session_runtime_timeline_contract_test.py`
- `frontend/src/lib/store/utils.ts`
- `frontend/src/lib/store/runtime.ts`

动作：

1. `_runtime_attachment()` 对 single-agent task 只返回 task projection 或 derived envelope projection，不再同时返回完整 `public_timeline`。
2. 删除测试中要求 `attachment["public_timeline"]` 与 `task_projection` 同时存在的断言。
3. 对 turn-run legacy attachment 保留 `public_timeline`，但必须带明确 `authority` 和无 task projection。
4. session refresh 后用 persisted projection 覆盖 transient projection，不追加第二套公开时间线。

完成条件：

- 刷新前后同一任务的公开显示语义一致。
- 已完成任务不会出现“message content + task_projection.final_answer + public_timeline.final_summary”三重正文。

### Phase 3：前端唯一 reducer

目标：前端 store 不再从 raw event 推断用户语义。

文件：

- `frontend/src/lib/store/publicProjectionReducer.ts`
- `frontend/src/lib/store/events.ts`
- `frontend/src/lib/store/runtime.ts`
- `frontend/src/lib/store/publicTimeline.ts`
- `frontend/src/lib/runtimeVisibilityProjection.ts`

动作：

1. `reduceStreamEvent()` 优先只处理 envelope；legacy raw path 只能作为临时 adapter。
2. `patchRuntimeAttachmentFromRuntimeEvent()` / `patchRuntimeAttachmentFromMonitor()` 改为 envelope 调用方。
3. 删除或降级以下前端决策：
   - `stageStatusForEvent`
   - `activityDetailForEvent`
   - `userReceiptForEvent`
   - runtime/system channel 的 `assistantDoneContentFromEvent` 正文兜底
   - `runtimeVisibilityProjection.ts` 对 runtime event 的文案解释
4. `publicTimeline.ts` 只做 sanitize/merge，不再通过 `kind` 反推 body。

完成条件：

- 前端没有生产代码从 raw runtime event name 生成聊天正文。
- `rg "runtimeVisibilityProjection" frontend/src` 无生产导入。

### Phase 4：聊天渲染纯化

目标：React 组件只渲染 reducer 输出，不做语义裁决。

文件：

- `frontend/src/components/chat/ChatMessage.tsx`
- `frontend/src/components/chat/PublicTimelineActivity.tsx`
- `frontend/src/components/chat/ChatPanel.tsx`
- `frontend/src/components/chat/agentRunProjection.ts`
- `frontend/src/components/chat/PublicTimelineActivity.test.ts`
- `frontend/src/components/chat/ChatMessage.test.ts`

动作：

1. `ChatMessage` 移除 `contentProjectedIntoTimeline` 这类正文搬运逻辑。
2. `PublicTimelineActivity` 不再按 `kind === "observation" || kind === "action"` 重判 tool window。
3. `agentRunProjection.ts` 升级为唯一 message projection 选择器，或并入 `publicProjectionReducer` 后只保留纯 helper。
4. `ChatPanel.shouldSuppressSessionActivityBar()` 不再通过 public timeline body 猜测消息级反馈。

完成条件：

- 稳定正文与 `task_projection.final_answer` 相同只显示一次。
- `display_surface=timeline` 的 observation 不会被前端改成 tool window。
- 停止/暂停后没有 running spinner、运行中文案或内部工具 detail。

### Phase 5：删除旧权威与测试清理

目标：不保留旧链路，不用兼容作为长期兜底。

删除/降级候选：

- `backend/harness/runtime/public_chat_timeline.py`
- `frontend/src/lib/runtimeVisibilityProjection.ts`
- 旧的 public timeline body fallback 测试
- 保护 `public_timeline + task_projection` 混合 attachment 的测试断言

保留条件：

- 若某 legacy adapter 必须临时保留，必须写明删除条件：
  - 所有 chat stream event 已有 envelope。
  - runtime monitor SSE 已有 envelope。
  - session hydrate 已能从 envelope/task projection 重建。

完成条件：

- `rg "public_chat_timeline" backend` 无生产导入。
- `rg "runtimeVisibilityProjection" frontend/src` 无生产导入。
- 旧测试不再保护内部形状，只保护用户可见行为。

## 7. 迁移与回滚规则

### 7.1 Shadow 期

允许同时生成 legacy fields 和 envelope，但前端以 envelope 为准。Shadow 期只用于比较，不允许两个结果同时驱动 UI。

### 7.2 Cutover 条件

满足以下条件后切到 envelope-only：

1. chat stream、runtime monitor、session timeline 都能输出 envelope。
2. 前端所有主会话渲染都通过 `MessagePublicProjection`。
3. 固定端口 3000/8003 实测通过。
4. `rg` 确认旧权威无生产导入。

### 7.3 回滚规则

回滚只允许回滚到上一阶段入口，不允许重新开启多个 UI 决策源。每次回滚必须记录失败场景，并补充合同测试后再继续。

## 8. 风险与防线

| 风险 | 防线 |
| --- | --- |
| 修改面跨后端、store、组件，容易局部通过整体漂移 | 每阶段固定输入/输出和 Done 条件 |
| session refresh 覆盖 live projection | refresh 必须替换 transient，不追加第二套 timeline |
| 工具观察被 UI 隐藏后模型也拿不到 | UI 隐藏只影响投屏，不影响 runtime observation 记录与模型后续输入 |
| 旧测试保护旧结构 | 同阶段更新测试，测试用户行为而不是旧内部形状 |
| 为了快而保留兼容链路 | 兼容只能作为 shadow adapter，并写删除条件 |

## 9. 明确不做

1. 不改 agent prompt 主体，不用 prompt 文案掩盖投影混权。
2. 不用更多正则过滤替代权威收口。
3. 不重写无关图任务前台、内存系统、VSCode 绑定。
4. 不跳过测试、不降低断言、不 mock 核心投影链路来制造通过。

## 10. 执行前确认点

这次重构会跨后端 runtime 投影、session hydrate、前端 store、聊天组件和测试体系，属于项目规则里的大改动。计划确认后，应按阶段完整实施；若实施中发现目标合同错误、旧链路仍有真实外部契约或测试暴露架构偏差，需要暂停并重新确认，而不是继续叠补丁。

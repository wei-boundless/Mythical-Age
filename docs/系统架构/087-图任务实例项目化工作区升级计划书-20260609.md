# 图任务实例项目化工作区升级计划书

日期：2026-06-09
状态：待确认执行

## 1. 核心结论

图任务工作区的主对象不应该是 `graph_run`，也不应该是普通作品页里的“当前作品”。新的主对象应为：

```text
图任务实例 = 一个可运行、可恢复、可审查、可管理文件的多 agent 项目会话
```

也就是说：

```text
图任务定义
  ├─ 图任务实例 A = 项目 A
  │   ├─ 项目文件
  │   ├─ 运行轮次 / 当前运行状态
  │   ├─ 多个 agent 节点
  │   ├─ 每个 agent 节点自己的会话
  │   └─ 产物与审查记录
  └─ 图任务实例 B = 项目 B
```

这个模型和主会话页的关系是：

```text
普通主会话页：一个会话 = 一个单 agent 对话
图任务工作区：一个图任务实例 = 一个多 agent 对话集合
```

因此图任务系统必须独立成一个项目化工作区：左侧列出同一图任务定义下的多个实例，进入实例后才能管理文件、启动运行、监控节点、查看节点会话和管理产物。

## 2. 当前结构问题

### 2.1 graph_run 被误当成实例

当前前端已有：

- `frontend/src/components/workspace/views/task-graph-workbench/GraphTaskWorkspace.tsx`
- `frontend/src/components/workspace/views/task-system/TaskGraphRunControlPanel.tsx`
- `frontend/src/components/workspace/views/task-system/TaskGraphRunInteractionDock.tsx`
- `frontend/src/lib/store/types.ts` 中的 `TaskGraphMonitorBinding`
- `frontend/src/lib/api.ts` 中的 `startTaskGraphHarnessRun()`、`getGraphRunMonitor()`

这些能力能启动和监控 `graph_run`，但还缺少“图任务实例/项目”这一层。结果是用户看到的是运行绑定，而不是项目化实例；页面也会出现“未选择作品”这种和图任务无关的空状态。

### 2.2 旧 ProjectInstance 仍强制绑定环境

当前后端 `backend/task_system/repositories/project_instance_repository.py` 的 `ProjectInstanceRepository.validate()` 要求：

```text
project_id
environment_id
library_id
```

这和新的图任务实例模型冲突。图任务实例是项目，但图本身不能绑定环境；环境只能给节点 agent 装配。

因此不能直接把现有 `ProjectInstance` 原封不动当作图任务实例使用。需要二选一：

1. 新建 `GraphTaskInstance` 模型和仓库。
2. 泛化 `ProjectInstance`，允许 `project_kind = graph_task_instance` 时不要求 `environment_id`。

建议采用第 1 种：新建图任务实例模型，避免旧项目环境语义继续污染图任务。

### 2.3 前端工作区层级混杂

当前页面把这些层级混在一起：

- 图编辑
- 图运行监控
- 作品/项目空状态
- 主会话/图会话
- 文件浏览

目标结构必须把层级拆开：

```text
图任务定义列表
  -> 图任务实例列表
    -> 图任务实例详情
      -> 运行控制台 / 节点会话 / 项目文件 / 产物中心 / 配置与编译报告
```

### 2.4 监控同步不是单个刷新问题

截图中看不到新启动的图任务，是因为前端没有以“图任务实例”为订阅目标。现在更多是全局 `taskGraphMonitorBinding` 或 `graph_run_id` 绑定，无法自然表达：

- 一个图任务有多个实例。
- 一个实例有当前运行和历史运行。
- 一个实例下有多个节点会话。
- 一个实例有自己的项目文件和产物库。

## 3. 目标概念模型

### 3.1 GraphTaskDefinition

图任务定义，描述可复用的任务图模板。

示例：

- `graph.writing.modular_novel.master`
- 长篇写作图任务
- 研究报告图任务

职责：

- 节点定义
- 边契约
- 资源库声明
- 编译配置
- 默认节点 agent 装配

不负责：

- 项目文件
- 运行状态
- 具体产物
- 用户实例会话
- 环境绑定

### 3.2 GraphTaskInstance

图任务实例，也就是用户理解中的项目。

示例：

- `仙侠小说 100 章项目`
- `AI Agent 架构研究报告`

建议字段：

```text
graph_task_instance_id
graph_id
title
description
status
created_at
updated_at
active_graph_run_id
graph_run_ids
root_session_id
artifact_index_id
file_space_id
metadata
```

实例拥有：

- 项目文件
- 当前运行状态
- 历史运行记录
- 节点会话集合
- 产物索引
- 风险和监管记录

实例不拥有：

- 环境装配权
- 节点 prompt 编译权
- graph runtime 的调度权

### 3.3 GraphRun

`GraphRun` 不再是前端主对象，而是图任务实例的一次运行轮次或当前执行状态。

职责：

- 调度节点
- 维护 checkpoint
- 维护 graph_loop_state
- 记录 work_order
- 触发后台 runner

归属：

```text
GraphRun belongs to GraphTaskInstance
```

### 3.4 NodeAgentSession

节点会话是图任务实例内部的子会话。

建议归属键：

```text
graph_task_instance_id
graph_run_id
node_id
work_order_id
session_id
```

一个节点可以有多个会话历史：

- 初次执行会话
- 重试会话
- 人工接管会话
- 审查会话

但 UI 默认只展示当前 active work order 对应的节点会话。

### 3.5 InstanceFileSpace

每个图任务实例必须有项目文件区。

建议目录抽象：

```text
input/
  用户资料、设定草案、需求文件

working/
  节点中间产物、临时草稿

artifacts/
  可验收产物

memory/
  实例级记忆快照与索引

logs/
  运行日志、监管记录、错误报告
```

前端不应直接展示裸目录为唯一入口，而应提供“项目文件”和“产物中心”两个视角：

- 项目文件：用户可直接管理。
- 产物中心：按契约、节点、章节、状态聚合。

## 4. 权威链路

| 层级 | 权威 | 说明 |
| --- | --- | --- |
| 图任务定义 | TaskGraphRepository / GraphHarnessConfigRepository | 只负责图结构、节点、边、契约、编译结果 |
| 图任务实例 | 新增 GraphTaskInstanceRepository | 负责实例项目、文件空间、实例状态、当前运行引用 |
| 运行状态 | GraphHarness / GraphLoop | 负责 graph_run、checkpoint、work_order、节点状态 |
| 节点会话 | SessionManager | 负责每个 agent 节点会话的消息、scope、binding |
| 文件与产物 | GraphTaskInstanceFileService / ArtifactIndex | 负责实例文件、产物版本、产物归档 |
| 前端展示 | GraphTaskWorkspace | 只展示后端投影，不重新判断运行状态 |

关键原则：

1. 图定义不绑定环境。
2. 图任务实例是项目。
3. 图运行是实例的一次运行状态，不是项目本身。
4. 节点会话是实例下的子会话。
5. 文件和产物归属实例，不归属主会话。
6. 前端只订阅实例和运行投影，不自己发明状态。

## 5. API 规划

### 5.1 图任务实例 API

新增建议：

```text
GET    /api/orchestration/graph-tasks
GET    /api/orchestration/graph-tasks/{graph_id}/instances
POST   /api/orchestration/graph-tasks/{graph_id}/instances
GET    /api/orchestration/graph-task-instances/{instance_id}
PATCH  /api/orchestration/graph-task-instances/{instance_id}
DELETE /api/orchestration/graph-task-instances/{instance_id}
```

实例创建请求：

```json
{
  "title": "仙侠小说 100 章项目",
  "description": "",
  "initial_files": [],
  "metadata": {}
}
```

实例返回：

```json
{
  "graph_task_instance_id": "gti:writing:...",
  "graph_id": "graph.writing.modular_novel.master",
  "title": "仙侠小说 100 章项目",
  "status": "idle",
  "active_graph_run_id": "",
  "graph_run_ids": [],
  "file_space": {},
  "artifact_summary": {},
  "runtime_summary": {}
}
```

### 5.2 实例运行 API

新增实例级运行入口：

```text
POST /api/orchestration/graph-task-instances/{instance_id}/runs
GET  /api/orchestration/graph-task-instances/{instance_id}/monitor
POST /api/orchestration/graph-task-instances/{instance_id}/pause
POST /api/orchestration/graph-task-instances/{instance_id}/resume
POST /api/orchestration/graph-task-instances/{instance_id}/stop
```

底层仍可复用现有：

```text
POST /api/orchestration/harness/task-graphs/{graph_id}/start
GET  /api/orchestration/harness/graph-runs/{graph_run_id}/monitor
```

但前端主链路应调用实例级 API。这样用户不用理解 `graph_run_id`、`graph_harness_config_id`。

### 5.3 实例文件 API

建议新增或包装现有项目文件服务：

```text
GET  /api/orchestration/graph-task-instances/{instance_id}/files/tree
GET  /api/orchestration/graph-task-instances/{instance_id}/files
PUT  /api/orchestration/graph-task-instances/{instance_id}/files
POST /api/orchestration/graph-task-instances/{instance_id}/files/import
```

如果复用 `ProjectFileService`，必须先解除旧 `ProjectInstance` 对 `environment_id` 的强制依赖，或者为图任务实例单独提供 `GraphTaskInstanceFileService`。

### 5.4 节点会话 API

建议新增实例视角：

```text
GET /api/orchestration/graph-task-instances/{instance_id}/node-sessions
GET /api/orchestration/graph-task-instances/{instance_id}/nodes/{node_id}/sessions
GET /api/orchestration/graph-task-instances/{instance_id}/nodes/{node_id}/messages
POST /api/orchestration/graph-task-instances/{instance_id}/nodes/{node_id}/messages
```

底层仍复用现有 sessions：

```text
GET /api/sessions/{session_id}
GET /api/sessions/{session_id}/messages
```

但前端不应让用户在主会话页里找这些节点会话。

节点直聊必须区分三种模式：

1. 观察模式：只读查看节点会话，不改变运行。
2. 协助模式：向节点追加用户补充说明，作为下一轮 node turn 的 `human_instruction`。
3. 接管模式：暂停当前节点或图运行，由用户和该节点 agent 交互，最终提交一个受契约校验的节点结果。

直聊消息必须记录为实例事件：

```text
graph_task_instance_id
graph_run_id
node_id
work_order_id
session_id
operator_message_id
intervention_mode
```

直聊不能绕过边契约和节点输出契约。用户和节点交流后，节点产物仍必须经过：

- output contract 校验
- progress receipt
- supervisor observation
- 下游边协议传递

如果节点正在进行一次不可中断的模型调用，用户消息不能强塞进已发出的请求。系统应将消息标记为 pending intervention，并提供：

- 等当前模型调用结束后进入下一轮。
- 请求暂停当前节点。
- 停止并重跑当前节点。

### 5.5 产物 API

建议新增：

```text
GET  /api/orchestration/graph-task-instances/{instance_id}/artifacts
GET  /api/orchestration/graph-task-instances/{instance_id}/artifacts/{artifact_id}
PATCH /api/orchestration/graph-task-instances/{instance_id}/artifacts/{artifact_id}
```

产物必须支持按以下维度过滤：

```text
node_id
graph_run_id
contract_id
artifact_type
status
chapter_index
version
```

## 6. 前端工作区设计

### 6.1 顶层布局

图任务工作区应采用操作型控制台布局，而不是作品页布局。

```text
左侧：图任务定义与实例列表
中间：实例工作台
右侧：文件 / 节点 / 产物详情
```

左侧示例：

```text
图任务工作区

长篇写作
  仙侠小说 100 章项目
  科幻小说短篇项目

研究报告
  Agent 架构报告
```

中间默认展示当前实例，而不是“未选择作品”。

### 6.2 实例主页

进入实例后，顶部显示：

```text
实例标题
图任务定义
状态
当前运行
活跃节点
产物数量
最近更新时间
```

核心操作：

- 启动运行
- 暂停
- 继续
- 停止
- 查看编译报告
- 打开项目文件

### 6.3 实例内视图

建议分为 5 个主视图：

```text
概览
运行图
节点会话
项目文件
产物中心
```

说明：

- 概览：项目状态、当前节点、最近产物、风险。
- 运行图：拓扑、节点状态、边传递、work order。
- 节点会话：每个 agent 的对话列表和实时消息。
- 项目文件：输入、工作区、产物文件。
- 产物中心：按契约和状态管理可验收结果。

### 6.4 运行图视图

运行图视图不只是画拓扑，应展示执行状态：

- pending
- ready
- running
- completed
- failed
- blocked
- waiting_human

点击节点打开右侧详情：

```text
节点详情
  - 当前状态
  - 节点角色
  - 当前 work order
  - 节点会话
  - 输入包
  - 输出产物
  - 错误与恢复
```

点击边打开：

```text
边详情
  - 边协议
  - payload contract
  - source result
  - target input
  - ack 状态
```

### 6.5 节点会话视图

节点会话应像多 agent 对话集合：

```text
左：节点列表
中：选中节点会话消息
右：节点输入/输出/产物
```

默认只实时订阅：

- 当前 running 节点
- 用户选中的节点
- 用户 pin 的节点

不要一次性打开所有节点消息流，避免前端和后端都被压垮。

节点会话视图必须支持用户直接和节点交流，但需要明确模式：

```text
只读观察
发送补充说明
暂停并接管
提交节点结果
```

UI 上建议把直聊能力放在节点详情的“节点会话”标签内，而不是放到全局聊天框里。这样用户会清楚自己正在和哪个节点 agent 交流，也不会误把节点消息发到主会话。

### 6.6 项目文件视图

实例文件视图应直接服务项目管理：

- 上传资料
- 查看输入文件
- 查看节点生成文件
- 管理正文、设定、审稿意见
- 打开文件预览
- 和产物中心互相跳转

文件归属实例，不归属主会话，不归属环境。

### 6.7 产物中心

产物中心需要比文件树更结构化。

对于写作图，建议显示：

```text
世界观设定
人物设定
大纲
章节正文
章节审核
记忆提交
问题清单
```

每个产物显示：

- 产物名称
- 产物类型
- 来源节点
- 所属 graph run
- 契约
- 版本
- 状态
- 是否被下游消费

## 7. 前端状态规划

### 7.1 新增 store 状态

建议新增：

```ts
type GraphTaskInstanceSummary = {
  graph_task_instance_id: string;
  graph_id: string;
  title: string;
  status: string;
  active_graph_run_id?: string;
  updated_at: number;
};

type GraphTaskWorkspaceState = {
  selectedGraphId: string;
  selectedInstanceId: string;
  instancesByGraphId: Record<string, GraphTaskInstanceSummary[]>;
  activeInstance: GraphTaskInstanceDetail | null;
  activeInstanceMonitor: GraphTaskInstanceMonitor | null;
  selectedNodeId: string;
  selectedArtifactId: string;
};
```

### 7.2 降级现有全局 monitor binding

当前 `TaskGraphMonitorBinding` 是全局绑定，适合临时浮窗，不适合实例项目化工作台。

目标：

1. 实例详情页以 `selectedInstanceId` 为主。
2. `graph_run_id` 只是实例 monitor 返回的内部字段。
3. 全局浮窗如果保留，只作为“最近活动实例提醒”，不作为主交互入口。

### 7.3 SessionScope 策略

建议第一阶段沿用现有 `SessionScope` 字段，但语义明确为：

```json
{
  "workspace_view": "graph_task",
  "task_environment_id": "",
  "project_id": "<graph_task_instance_id>"
}
```

这里的 `project_id` 是兼容现有 scope 查询结构的实例隔离键，不代表环境项目。

后续可以考虑将 `SessionScope` 扩展为：

```json
{
  "workspace_view": "graph_task",
  "graph_task_instance_id": "...",
  "project_id": "",
  "task_environment_id": ""
}
```

但第一阶段不建议为了命名洁癖扩大协议变更。

## 8. 后端实施计划

### 阶段一：实例模型

新增：

```text
backend/task_system/graph_instances/
  models.py
  repository.py
  file_service.py
  artifact_index.py
```

最小模型：

```text
GraphTaskInstance
GraphTaskInstanceRepository
GraphTaskInstanceFileService
GraphTaskArtifactIndex
```

存储位置建议：

```text
runtime_state/graph_task_instances/
storage/task_system/graph_task_instances.json
output/graph_task_instances/{safe_instance_id}/
```

具体落点可实施前根据现有 `TaskSystemStorage` 和 runtime object 目录统一选择。

### 阶段二：实例级 API

新增实例 API facade，并让前端使用实例 API：

```text
backend/api/graph_task_instances.py
```

职责：

- 创建实例。
- 列出某图任务定义下的实例。
- 打开实例详情。
- 启动实例运行。
- 聚合实例 monitor。
- 暴露实例文件和产物。

### 阶段三：启动运行绑定实例

实例启动运行时，内部调用现有 graph start API 逻辑，但必须写入：

```text
graph_task_instance_id
session_scope.workspace_view = graph_task
session_scope.task_environment_id = ""
session_scope.project_id = graph_task_instance_id
```

`GraphRun.diagnostics`、`TaskRun.diagnostics`、graph session `task_binding` 都应记录实例 id。

### 阶段四：节点会话归属实例

节点 work order 创建会话时，必须继承实例 scope：

```text
workspace_view = graph_task
project_id = graph_task_instance_id
task_environment_id = ""
```

同时保留：

```text
graph_run_id
node_id
work_order_id
```

用于前端在实例内聚合节点会话。

### 阶段五：文件与产物

最小可用版本：

1. 实例创建时初始化文件空间。
2. graph runtime 的 `artifact_root` 指向实例下的运行产物目录。
3. 节点产物写入 artifact index。
4. 前端能按实例列出产物。

### 阶段六：清理旧入口

完成实例化后，清理或降级：

- 直接从普通作品页打开 graph monitor。
- 主会话页显示 graph session。
- 以 `graph_run_id` 作为用户主入口的 UI。
- 图任务工作区里的“未选择作品”空态。
- graph 级 `task_environment_id` 保存逻辑。

### 阶段四：实例实时监控

前端订阅策略：

1. 进入实例详情后定时拉取实例 monitor。
2. 如果后端提供 SSE，则订阅实例事件流。
3. 正在运行的节点单独刷新节点会话。
4. 文件和产物用显式刷新，不和节点消息共用高频轮询。

第一阶段可以用 polling，后续再升级实例级 SSE。

### 阶段五：节点会话 UI

节点会话面板必须支持：

- 查看所有节点。
- 标记 running/failed/blocked。
- 打开当前节点对话。
- 跳转到产物。
- 显示输入包和输出包。

不在第一阶段做：

- 多节点同时全量实时展开。
- 节点之间自由聊天 UI。
- 节点节流调度。

### 阶段六：项目文件 UI

项目文件视图必须支持：

- 目录树。
- 文件预览。
- 上传或新建资料。
- 打开产物文件。
- 和产物中心互跳。

第一阶段可复用现有文件浏览组件，但必须改成实例 scope。

## 10. 迁移策略

### 10.1 旧 graph_run

旧 graph run 不作为新工作区主线兼容对象。

可选处理：

1. 隐藏旧运行，只保留后端可查。
2. 提供一次性“导入为图任务实例”动作。
3. 对当前重要写作运行手动绑定到一个新实例。

不建议保留长期双入口，否则又会变成重复权威。

### 10.2 旧 ProjectInstance

旧 `ProjectInstance` 继续服务任务环境/普通项目。

图任务实例不要继承其 `environment_id` 强制规则。

如果后续要统一项目系统，应先把 `ProjectInstance` 泛化为：

```text
ProjectInstanceBase
EnvironmentProjectInstance
GraphTaskProjectInstance
```

但这不是第一阶段必需。

### 10.3 当前写作图

当前写作图可以按新模型创建：

```text
graph_id = graph.writing.modular_novel.master
instance = 长篇写作项目
```

启动后：

```text
graph_run belongs to instance
node sessions belong to instance
artifacts belong to instance
files belong to instance
```

## 12. 实施顺序

建议一次按以下阶段完成，不在旧壳上继续堆 UI：

1. 后端新增 `GraphTaskInstance` 模型和仓库。
2. 后端新增实例级 API。
3. 实例启动运行打通现有 GraphHarness。
4. 节点会话和产物写入实例归属。
5. 前端新增实例 API client 和 store。
6. 重构 GraphTaskWorkspace 为实例工作区。
7. 加入节点会话、项目文件、产物中心。
8. 清理旧 graph_run-only UI 入口。
9. 跑后端、前端和固定端口联调测试。

## 13. 非目标

第一阶段不做：

1. 节点节流。
2. agent 之间自由聊天面板。
3. 跨图任务实例的全局项目组合管理。
4. 多实例资源争抢调度。
5. 复杂权限协作。

这些后续可以在实例模型稳定后继续加。

## 14. 验收标准

完成后必须满足：

1. 一个图任务定义可以创建多个实例。
2. 一个实例就是一个项目。
3. 实例能直接管理项目文件。
4. 实例能启动、暂停、继续、停止图运行。
5. 实例能实时看到当前运行状态。
6. 每个节点 agent 的会话能在实例内查看。
7. 节点产物能进入实例产物中心。
8. 图任务实例不绑定环境。
9. 环境只属于节点 agent 装配。
10. 主会话页不混入图任务实例、节点会话或产物。

# 137-成熟Agent投影线路 Ledger 权威修复方案

日期：2026-06-14

修订：2026-06-15

本次修订纠正两个混淆：

1. **非任务普通对话**不是任务状态，不进入任务 timeline，也不需要任务收口刷新。
2. **任务收口**不是隐藏工具窗口就完事，而是主聊天表面从 live timeline 切换为任务收口总结；完整工具 timeline 只进入日志 / trace。

依据：

- `135-成熟Agent投影线路结构性修复计划书-20260614.md`
- `136-成熟Agent投影线路修复后链路审查记录-20260614.md`

## 1. 修复目标

把投影事实源收束到 live 阶段已经持久化的 public projection ledger，同时把主聊天显示策略按运行状态分层。

修复后必须满足：

1. public ledger 永久保存正文、模型反馈、工具调用、工具观察、commit/terminal 的完整时序。
2. 非任务普通对话：主聊天只按普通聊天显示模型正文；刷新只从 session history 恢复正文。
3. 任务运行中：主聊天显示任务 live timeline，包括阶段反馈、工具窗口、运行状态。
4. 任务收口后：主聊天显示任务收口总结，不继续显示工具窗口。
5. 完整工具轨迹只能在 run log / trace 中查看。
6. 断线重连只在 run 仍处于 running/active 状态时恢复 live timeline。
7. session timeline 不再用 raw runtime event 尾窗重建主聊天投影。

## 2. 三种主显示状态

| 状态 | 判定 | 主聊天显示 | 刷新后 | 工具轨迹位置 |
| --- | --- | --- | --- | --- |
| 非任务普通对话 | 没有进入 task run / active work run；只是普通 assistant turn | 用户消息 + assistant 正文 | 从 session history 恢复正文 | 不恢复到主聊天；如有轨迹，只在日志/trace |
| 任务运行中 | task/run 未 terminal，仍 active/running | live timeline：阶段反馈 + 工具窗口 + 正文片段 | 可从 public ledger replay 当前 live timeline | public ledger + live 工具窗口 |
| 任务已收口 | task/run terminal 或 commit 完成 | 任务收口总结 + 日志入口 | 恢复收口总结，不恢复工具窗口 | run log / trace |

关键规则：

- `tool_call_requested/start/completed` 可以驱动任务运行中的 live 工具窗口。
- `TASK_BRIDGE_TERMINAL_EVENT`、`TURN_COMPLETED_EVENT`、`session_output_commit_*`、assistant final body 是切换到收口展示的依据。
- 收口后的主聊天不展示工具窗口；不是折叠保留，也不是刷新恢复。

## 3. 当前断线

当前 live 路径大体是：

```text
runtime event
-> backend public event bridge
-> PublicProjectionFrame
-> chatrun:* public event log
-> SSE reducer
-> message.publicProjection
-> ChatMessage / PublicTimelineActivity
```

当前 history 路径存在问题：

```text
task/turn raw event tail
-> session_timeline 重新 project
-> public_projection_frames
-> refreshSessionDetails hydrate
```

断线点：

1. `session_timeline` 只读 task/turn raw event tail，长会话会丢 request/start。
2. history 不是 replay live 已保存的 `chatrun:*` ledger，frame identity 和 event_offset 可能漂移。
3. history 没区分非任务普通对话、任务运行中、任务已收口，容易把工具窗口恢复到错误表面。
4. `_tool_call_requested_data()` 只支持单个 `tool_call`，public bridge 对直接输入的 `tool_calls[]` 不完整。
5. 模型反馈去重偏文本级，可能误折叠不同 action 的相同反馈。

## 4. 目标权威链

| 层 | 唯一职责 | 不允许做的事 |
| --- | --- | --- |
| runtime event | 产生真实模型正文、工具请求、工具执行、工具观察、terminal、commit | 不直接决定主聊天最终表面 |
| backend public event bridge | 把 runtime event 规范化为 public event，并在 live 阶段生成 `PublicProjectionFrame` | 不把正文塞进工具，不把工具塞进正文 |
| `RuntimeStreamReplayService` / `chatrun:*` | 持久化完整 public event ledger | 不丢 frame，不重排 frame |
| `session_timeline` | 读取 public ledger，返回 run attachment、display state、日志所需 frame | 不从 raw event 尾窗重建主聊天工具窗口 |
| display state resolver | 判定 `normal_turn` / `task_live` / `task_closed` | 不根据 UI 当前有没有工具窗口倒推状态 |
| frontend reducer | 幂等保存 projection ledger，按 `tool_call_id` 合并工具生命周期 | 不按标题、数组位置或时间猜测工具身份 |
| `ChatMessage` | 根据 display state 渲染主聊天：普通正文、任务 live timeline、任务收口总结 | 不在 task closed 后显示工具窗口 |
| `PublicTimelineActivity` | 只渲染任务运行中的 live 工具窗口 | 不承担历史日志职责 |
| run log / trace viewer | 展示完整工具轨迹 | 不反向污染主聊天 |

## 5. 固定执行流

### 5.1 非任务普通对话

```text
user message
-> assistant body stream / final body
-> session history
-> refresh restores normal assistant message
```

显示规则：

- 主聊天正常显示正文。
- 不需要任务收口。
- 不从 history public projection 恢复工具窗口。

### 5.2 任务运行中

```text
runtime task events
-> backend public event bridge
-> PublicProjectionFrame
-> chatrun:* public ledger
-> SSE reducer
-> task_live display state
-> ChatMessage renders live timeline / tool windows
```

显示规则：

- 工具 request/start/completed 按 `tool_call_id` 合并为运行中的一个工具窗口。
- 模型公开反馈显示为任务阶段反馈或正文片段。
- 断线重连时，只要 run 仍 active/running，可以 replay ledger 恢复当前 live timeline。

### 5.3 任务收口

```text
TASK_BRIDGE_TERMINAL_EVENT / TURN_COMPLETED_EVENT / session_output_commit_* / assistant final body
-> display state becomes task_closed
-> main chat replaces live timeline with task closeout summary
-> full tool timeline remains available in run log / trace
```

显示规则：

- 主聊天不再显示工具窗口。
- 主聊天显示任务收口总结。收口总结优先使用 assistant final body / committed output；如果没有可用正文，显示明确的 terminal 状态摘要，不用工具轨迹代替正文。
- 主聊天可以显示一个轻量日志入口，例如“查看执行日志 / N 次工具调用”，但不展开工具窗口。

### 5.4 历史刷新

```text
/api/sessions/{session_id}/timeline
-> session_timeline reads chatrun:* public ledger
-> resolve display state
-> normal_turn: use session history body only
-> task_live: replay live projection
-> task_closed: restore closeout summary + log ref
```

History replay 的目标不是把完成后的工具窗口塞回主聊天，而是：

1. 支持 active/running run 的 live timeline 恢复。
2. 支持 run log / trace 查看完整工具轨迹。
3. 支持 task closed 状态下恢复收口总结。

## 6. 数据模型调整

### 6.1 Runtime attachment 增加显示状态

`runtime_attachments[]` 需要明确表达展示状态：

```text
display_state: normal_turn | task_live | task_closed | log_only
main_chat_surface: body_only | live_timeline | closeout_summary
log_ref: string
tool_event_count: number
closeout_summary: string
public_projection_frames: PublicProjectionFrame[]
```

规则：

- `normal_turn`：不把 `public_projection_frames` 应用到主聊天工具窗口；主聊天走 session history。
- `task_live`：允许前端把 `public_projection_frames` replay 到主聊天 live timeline。
- `task_closed`：只把正文/closeout summary 应用到主聊天；工具 frames 进入日志表面。
- `log_only`：只供日志/trace 使用，不进入主聊天。

### 6.2 PublicProjectionFrame 保持完整

frame 仍需要保留：

- `frame_id`
- `source_event_id`
- `source_event_type`
- `event_offset`
- `event_family`
- `channel`
- `slot`
- `source_authority`
- `tool_call_id`
- `tool_lifecycle_id`
- `tool_name`
- `target`
- `arguments_preview`
- `state`

注意：frame 完整保存，不等于主聊天完整展示。

### 6.3 Closeout Summary

任务收口总结来源优先级：

1. assistant final body / committed output
2. `session_output_commit_*` 中的 summary/reason/error
3. task terminal context 中的 status/terminal_reason/error_summary

如果缺少模型正文，必须显示“任务已结束但没有可提交正文”的明确状态，不能把工具结果拼成正文。

## 7. 代码修改计划

### Phase 1：后端改为 public ledger replay，并输出 display state

目标：`/sessions/{id}/timeline` 读取 `chatrun:*` public ledger，并明确 run 的主聊天展示状态。

改动文件：

- `backend/runtime/shared/stream_replay.py`
- `backend/harness/runtime/session_timeline.py`
- `backend/api/sessions.py`

具体改动：

1. 在 `RuntimeStreamReplayService` 增加 public ledger 读取方法：
   - 输入：`RuntimeRun`
   - 读取：`run.event_log_id`
   - 只接受 `event_type == chat_stream_event`
   - 输出：按 public event offset 排序的 public event/data/frame
2. 在 `session_timeline.py` 增加 stream-run attachment：
   - 从 `runtime_host.run_registry.list_session_runs(session_id)` 枚举 run。
   - 只处理 `event_log_id.startswith("chatrun:")` 的 run。
   - 从 public event payload 读取 `data.public_projection_frame`。
   - 从 run status、task bridge、terminal/commit event 推导 `display_state` 和 `main_chat_surface`。
3. `session_timeline.py` 删除主路径上的 `_public_projection_frames_from_events()`：
   - task/turn raw events 只用于 trace/metadata。
   - 不再重建主聊天工具窗口。
4. `api/sessions.py` 继续透传 attachment 字段。

完成标准：

- `public_projection_frames` 来源是 `chatrun:*` ledger。
- attachment 明确给出 `display_state/main_chat_surface`。
- task closed 不再被当成可恢复 live timeline。

### Phase 2：前端按 display state 分发主聊天表面

目标：普通对话、任务运行、任务收口三种主聊天显示不串线。

改动文件：

- `frontend/src/lib/store/runtime.ts`
- `frontend/src/lib/projection/reducer.ts`
- `frontend/src/components/chat/ChatMessage.tsx`
- run log / trace 相关 store 或组件

具体改动：

1. `hydrateSessionRuntimeProjection()` 根据 attachment 的 `main_chat_surface` 分发：
   - `body_only`：不 replay 工具 timeline 到主聊天。
   - `live_timeline`：replay body/status/tool frames 到主聊天。
   - `closeout_summary`：只写入 closeout summary / final body / log ref。
2. reducer 继续完整保留 projection ledger：
   - 同一 `frame_id` 幂等。
   - 同一 `tool_call_id` 合并为同一 ledger item。
3. `ChatMessage` 根据 message/display state 渲染：
   - 普通对话：正文。
   - 任务运行中：正文片段 + live timeline。
   - 任务已收口：收口总结 + 日志入口。
4. `PublicTimelineActivity` 只在 `task_live` 状态出现。

完成标准：

- 普通对话刷新后只显示 assistant 正文。
- task live 刷新/重连能恢复 live timeline。
- task closed 刷新后显示收口总结，不显示工具窗口。

### Phase 3：任务收口 summary 写入和恢复

目标：任务结束时主聊天有明确收口总结，而不是工具窗口残留。

改动文件：

- `backend/api/chat.py`
- `backend/harness/runtime/projection/projector.py`
- `backend/harness/runtime/session_timeline.py`
- `frontend/src/lib/api.ts`
- `frontend/src/components/chat/ChatMessage.tsx`

具体改动：

1. 在 task terminal / commit 事件转 public projection 时，确保有可恢复的 closeout summary 字段：
   - assistant final body 优先。
   - commit summary / terminal reason 作为状态摘要。
2. projector 为收口摘要生成稳定 frame 或 attachment 字段：
   - `event_family=runtime_commit` 或明确的 closeout family。
   - `main_chat_surface=closeout_summary`。
3. session timeline 在 task closed attachment 中返回：
   - `closeout_summary`
   - `log_ref`
   - `tool_event_count`
4. 前端 ChatMessage 渲染 closeout summary，不渲染工具窗口。

完成标准：

- 任务 terminal 后主聊天 live timeline 被收口总结替换。
- 没有 assistant final body 时，用户看到明确 terminal 状态，不看到工具窗口伪装成总结。
- 日志入口能指向完整工具轨迹。

### Phase 4：修正 `tool_calls[]` public bridge

目标：public bridge 自身支持数组工具调用，不再依赖上游永远拆分。

改动文件：

- `backend/api/chat.py`
- `backend/harness/loop/single_agent_turn.py`
- `backend/harness/loop/task_executor.py`

具体改动：

1. 把 `_tool_call_requested_data()` 改为数组规范化入口：
   - 单个 `tool_call` 输出一个 request frame data。
   - `tool_calls[]` 输出多个 request frame data。
   - 每个 tool call 必须有独立 `tool_call_id`。
2. `_tool_action_public_events()` 输出顺序固定：
   - 同一模型 action 的 `runtime_step_summary` 只输出一次。
   - 然后按 tool_calls 顺序输出每个 `tool_call_requested`。
   - 每个 tool_call 后输出对应 `tool_permission_decided`。
3. admission permission id 必须按 tool_call 生成。

完成标准：

- task live 中，多个工具调用能形成多个运行中工具窗口。
- task closed 后，这些工具窗口不再常驻主聊天，只在日志/trace 可查。

### Phase 5：模型反馈去重改为 identity 级

目标：去重只去同一 action/frame 的重复反馈，不误删不同 action 的相同正文反馈。

改动文件：

- `backend/api/chat.py`
- `backend/harness/runtime/projection/projector.py`
- `frontend/src/components/chat/ChatMessage.tsx`

具体改动：

1. `_model_action_feedback_step_data()` 的 feedback identity 优先级：
   - `batch_action_request_ref`
   - `runtime_invocation_packet_ref`
   - `action_request_ref`
   - `request_id`
   - source runtime event id
2. `_runtime_step_summary_spec()` 对 `presentation_source=model_action.*` 使用 feedback identity 生成 `item_id/frame_id`。
3. `ChatMessage` 的模型反馈去重从纯文本改为 source identity。

完成标准：

- 同一模型 action 的同一句反馈不会重复。
- 不同模型 action 的相同反馈不会被误删。
- 模型反馈不会进入工具窗口。

### Phase 6：清理旧投影主路径和旧语义测试

目标：删除会重新引导错误的旧线路。

改动文件：

- `backend/harness/runtime/session_timeline.py`
- 相关旧测试文件，只处理直接保护旧 raw-event history projection 或语义文案的测试

具体改动：

1. 删除或停用 `session_timeline` 主路径中的 raw event history projection。
2. trace 读取能力只服务日志/trace，不进入主聊天 projection。
3. 删除保护旧语义文案的测试。
4. 保留或新增结构性线路验证：
   - ledger replay 来源验证
   - display state 分发验证
   - task live 工具窗口验证
   - task closed 收口总结验证
   - no-anchor 不错挂验证

完成标准：

- 搜索不到 session timeline 主路径调用 `_project_public_stream_event()` 重建主聊天 projection。
- raw event tail 只用于 trace/metadata。
- 测试不再断言语义文案作为行为证明。

### Phase 7：固定端口真实验收

固定节点：

- 前端：`http://127.0.0.1:3000`
- 后端：`http://127.0.0.1:8003`
- API Base：`http://127.0.0.1:8003/api`

验收场景：

1. 非任务普通对话：
   - 运行中显示模型正文。
   - 完成后是一条 assistant 正文。
   - 刷新后仍只显示正文。
2. 任务运行中：
   - 显示任务 live timeline。
   - 工具 request/start/completed 合并为同一个运行中窗口。
   - 多工具调用能区分目标和参数。
3. 任务收口：
   - live timeline 被收口总结替换。
   - 主聊天不保留工具窗口。
   - 显示日志入口。
4. 日志/trace：
   - 能看到完整工具调用轨迹。
   - 不污染主聊天正文。
5. SSE 断开/重连：
   - running task 恢复 live timeline。
   - closed task 恢复收口总结。

## 9. 禁止事项

1. 禁止继续把 raw runtime event tail 当 history projection 主数据源。
2. 禁止为了“兼容”同时保留 ledger replay 和 raw reproject 两条主线路。
3. 禁止把非任务普通对话纳入任务收口逻辑。
4. 禁止在 task closed 后把工具窗口继续常驻主聊天。
5. 禁止把任务工具 timeline 简单隐藏但不生成收口总结。
6. 禁止用标题、工具名、数组位置合并工具生命周期。
7. 禁止用纯文本相等删除模型反馈。
8. 禁止把内部 ID、offset、raw output 放进默认主聊天。
9. 禁止新增语义文案测试来证明投影正确。
10. 禁止改端口或绕过固定前后端实测。

## 11. 执行顺序锁定

执行时必须按以下顺序：

1. 先改 backend ledger replay 数据源，并输出 display state。
2. 再改前端 hydrate/display policy，分清普通对话、task live、task closed。
3. 再补任务收口 summary 的生成和恢复。

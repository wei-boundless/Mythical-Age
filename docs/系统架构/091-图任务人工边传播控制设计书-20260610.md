# 091-图任务人工边传播控制设计书

日期：2026-06-10  
状态：设计审查稿，待确认后执行

## 1. 核心结论

这个功能可以设计，而且应该进入通用图任务控制体系，而不是写作图任务的专用审核按钮。

目标模型是：

```text
用户人工介入
  -> 生成 HumanEdgeDecision
  -> 按图任务实例、图运行、边契约验证
  -> 可选写入项目正式库 / 产物库
  -> 生成 HumanEdgePropagationPacket
  -> 交给 GraphLoop 推进边传播和下游调度
```

关键判断：

- 人工介入的核心对象是边，不是节点会话，也不是普通聊天消息。
- `pass` 是沿当前传播方向把已接受内容送给下游。
- `revise` 是沿返修/反馈边把意见回传给上游或修订节点。
- `replace` 是用户提交正式产物，成为 human-authored artifact，再按边契约传播。
- 人工输入必须可审计、可恢复、可重放，不能直接改 `GraphLoopState`。

写作图只是这个机制的一个场景。研究、代码生成、审核发布、资料整理也可以使用同一套边传播控制。

## 2. 问题定义

当前前台已经有图任务实例、监控、节点会话、项目文件区，但用户只能“看”和“编辑文件”，不能把人工判断作为图运行的一部分送回 runtime。

缺失的系统属性是：

```text
用户对图任务传播过程的结构化控制权
```

现在已有能力：

- 图任务实例拥有项目文件、根会话、当前运行和节点会话。
- GraphLoop 可以接收节点结果并生成 edge state / flow packet。
- Human gate 可以让节点进入 `waiting_human_gate`。
- 前端可以显示监控、节点输出、项目文件和产物。

现在缺失：

- 没有实例级人工决策账本。
- 没有 `HumanEdgeDecision` API。
- 没有把用户“通过/退稿/替写”编译成边传播包的后端服务。
- 编译器没有在边契约里声明哪些人工动作被允许。
- 前端没有把人工动作绑定到具体边、具体产物、具体传播方向。

正确终态：

```text
GraphTaskInstance 是项目和人工控制入口。
GraphLoop 是唯一运行推进者。
EdgeContract 定义人工动作是否允许、如何路由、如何生成包。
FileService / ArtifactService 承接正式内容写入。
Frontend 只提交结构化决策并展示结果，不直接改运行状态。
```

### 3.3 本地架构文档约束

已有文档已经给出三个重要原则：

- `077-图任务边契约协议族规划初稿`：HumanGateContract 必须可暂停、可恢复、可审计，人工输入结构化进入 graph history。
- `081-图任务节点交互与内置配置监管节点设计草案`：用户输入应进入 `HumanInterventionPacket`，不能直接改 `GraphLoopState`。
- `087-图任务实例项目化工作区升级计划书`：图任务实例是多 agent 项目会话，拥有项目文件、节点会话、产物和当前运行状态。

本设计把这些原则收口为：

```text
HumanGate = 等待机制
HumanEdgeDecision = 传播控制机制
HumanArtifactSubmission = 用户正式产物提交机制
```

### 3.4 外部成熟模式参考

LangGraph 官方文档的 human-in-the-loop interrupt 模式强调：图执行可以暂停，状态由 checkpoint 保存，然后通过 resume 输入继续；它也把审核、编辑、批准作为常见模式。参考：[LangGraph Interrupts](https://docs.langchain.com/oss/python/langgraph/interrupts)。

Temporal 的 Workflow message passing 把外部交互分成 Query、Signal、Update：Query 只读，Signal 异步控制流程，Update 是可验证、可追踪、可改变状态的同步请求。参考：[Temporal Workflow message passing](https://docs.temporal.io/develop/go/workflows/message-passing)。

借鉴结论：

- 监控接口必须保持只读，不能顺手改变图状态。
- 人工决策应类似 Temporal Update：先验证，再写历史，再推进。
- 等待人工输入应类似 LangGraph interrupt：checkpoint 后恢复，而不是前端临时 patch 状态。
- 人工输入必须 JSON 结构化、幂等、可审计。

| 层级 | 当前职责 | 新增职责 | 禁止事项 |

| HumanEdgeDecisionRepository | 无 | 持久化人工决策账本、幂等键、状态流转 | 不执行运行调度 |
| HumanEdgeDecisionService | 无 | 验证边契约、写入用户产物、构造运行控制包 | 不自行判断 scheduler ready |
| GraphLoop | 运行推进、checkpoint、edge state | 消费已验证的 `HumanEdgeDecision` 并生成/更新 flow packet | 不接受未验证的前端 payload |
| FileService / ArtifactService | 项目文件和产物 | 写入 human-authored artifact，返回 artifact ref | 不负责调度 |
| Compiler | 编译节点/边契约 | 编译 `human_control_policy` 和 compile report | 不把写作特例硬编码进 runtime |

```json
{
  "decision_id": "hedge.gti_xxx.0001",
  "graph_task_instance_id": "gti.xxx",
  "graph_run_id": "grun.xxx",
  "edge_id": "edge.writer.reviewer",

  "instruction": "",

    }
  ],
```

字段含义：

- `instruction`：返修意见、传播说明、下游要求。
- `idempotency_key`：防重复提交。
- `status`：`draft | submitted | accepted | applied | rejected | failed | superseded`。

## 6. 边控制语义

### 6.1 pass

含义：当前产物被人工接受，沿选中的 forward edge 传播。

运行效果：

```text
source node result / human artifact
  -> selected forward edge ready
  -> flow packet persisted
  -> target node becomes schedulable when dependency satisfied
```

写作例子：

```text
用户看完第 12 章
  -> 点击“通过并交给下游”
  -> 第 12 章 artifact_ref 进入 reviewer / next_writer 的 inbound context
```

### 6.2 revise

含义：当前产物不通过，反馈沿返修边回传。

运行效果：

```text
human feedback
  -> revision / repair edge ready
  -> upstream writer or revise node ready
  -> downstream affected scope reset according to edge policy
```

写作例子：

```text
用户退稿
  -> 输入问题和修改方向
  -> 反馈包回传 writer
  -> writer 根据反馈重写这一章
```

### 6.3 replace

含义：用户直接提交正式内容，替代某个节点产物。

运行效果：

```text
human content
  -> write project file / artifact
  -> create human-authored node result summary
  -> update result history with provenance
  -> propagate selected edge
```

写作例子：

```text
用户自己写好第 12 章
  -> 保存到 chapters/chapter-012.md
  -> 作为 human-authored artifact
  -> 继续交给 reviewer 或下一章 writer
```

### 6.4 comment

非 MVP。只追加审查意见，不改变调度。

### 6.5 hold / abort

非 MVP。`hold` 只 checkpoint，不推进。`abort` 终止当前运行，应继续使用 GraphLoop 权威入口。

### 7.3 Runtime 接入

新增 GraphHarness 方法：

```python
def apply_human_edge_decision(
    self,
    *,
    graph_config: GraphHarnessConfig,
    graph_run_id: str,
    decision: HumanEdgeDecision | dict[str, Any],
    max_requests: int | None = None,
) -> GraphLoopAdvance:
    ...
```

新增 GraphLoop 方法：

```python
def apply_human_edge_decision_and_checkpoint(...):
    ...
```

GraphLoop 方法是唯一允许更新 edge state、ready nodes、result history、checkpoint 的地方。

实现细则：

- `pass`：读取 source node 最新 result 或 decision artifact，生成目标 edge 的 `FlowPacket`，更新该 edge 为 `ready`，随后走 state machine 计算 ready nodes。
- `revise`：校验目标 edge 是否为 `revision_request | repair_feedback | repair_route` 或 compiled `human_control_policy.revise_edge_id`；写反馈 packet；重置受影响下游 scope；让返修 target ready。
- `replace`：由 FileService 写入内容，构造 human-authored `NodeResultEnvelope` 或 result summary，写入 `result_history`，再按 selected edge 生成 packet。
- 所有动作必须写 event log，例如 `graph_human_edge_decision_applied`。
- 所有动作必须 checkpoint。

注意：不要让 `edge_control_service` 自己拼 `GraphLoopState`。它只能验证和提交。

### 7.4 与现有 human gate 的关系

保留 `apply_human_gate_decision()`，但语义收窄：

```text
HumanGateDecision = 解开等待点
HumanEdgeDecision = 指定内容如何沿边传播
```

兼容策略：

- 如果节点处于 `waiting_human_gate`，前端仍展示“需要人工处理”。
- 用户点击通过/退稿/替写时，后端生成 `HumanEdgeDecision`。
- `HumanEdgeDecisionService` 可以在 apply 前调用 gate resolve，或 GraphLoop 在 apply 时把 gate 状态一并清理。
- 不新增另一套“审核人工介入”，避免概念重复。

## 8. 编译器和边契约设计

边契约新增：

```yaml
human_control_policy:
  enabled: true
  allowed_decisions:
    - pass
    - revise
    - replace
  pass:
    route: current_edge
    required_artifact_kinds: [chapter]
  revise:
    route_edge_id: edge.review.writer_revision
    reset_scope: chapter_iteration
    required_instruction: true
  replace:
    write_policy:
      repository_id: instance
      path_template: chapters/chapter-{chapter_index:03d}.md
      content_kind: chapter
    propagate_after_write: true
```

compile report 必须解释：

```text
用户可以在 edge.writer.review 上执行 pass，因为该边是交付审核边。
用户可以执行 revise，因为图中存在 review -> writer 的返修边。
用户可以执行 replace，因为 source node 允许 human-authored artifact 替代产出。
```

推荐原型：

| 原型 | 决策 | 说明 |
| --- | --- | --- |
| `human.forward_accept` | pass | 接受上游结果并传给下游 |
| `human.revision_return` | revise | 把反馈回传给修订/写手节点 |
| `human.replace_output` | replace | 用户提交正式产物并替代节点输出 |
| `human.annotation_only` | comment | 只记录意见，不推进 |
| `human.stop_checkpoint` | hold | 暂停并保存检查点 |

MVP 只实现前三个。

## 9. 前端设计

### 9.1 图监控屏

图监控屏增加一个“待人工处理”控制区：

```text
待人工处理
  - 当前节点 / 边
  - 产物
  - 可执行动作：通过、退稿、替写
  - 决策状态：已提交 / 已应用 / 失败
```

节点卡上显示：

- 当前是否等待人工。
- 可控制的边。
- 最近一次人工决策。
- 被决策影响的下游节点。

### 9.2 会话章节屏

章节/文件视图增加边传播动作条：

```text
[通过并传给下游] [退稿并回传写手] [我来替写并继续]
```

点击后打开决策抽屉：

- 当前文件 / artifact。
- source node、edge、target node。
- 决策类型。
- 返修意见或下游说明。
- 替写内容保存路径。
- apply now 开关。

交互原则：

- 用户不需要选择协议字段。
- 如果只有一条合法边，系统自动选择。
- 如果存在多个合法边，前端用业务名展示，例如“交给审校”“交给下一章写手”“退回正文写手”。
- 没有合法边时按钮禁用，并显示编译器解释。

### 9.3 项目文件和产物

文件管理要区分：

- 正式库：用户可编辑，图任务可引用。
- 运行产物：节点输出，默认只读。
- 人工提交：作为正式库写入或候选产物，有 provenance。

`replace` 默认写正式库，不直接写运行产物目录。

### 9.4 决策历史

实例详情增加“人工决策历史”：

```text
时间 / 用户 / 决策 / 边 / 产物 / 状态 / 应用结果
```

这不是聊天记录，而是运行控制日志。

## 10. 实施计划

### Phase 1：后端契约与账本

目标：

- 新增 `HumanEdgeDecision` / `HumanArtifactSubmission` 模型。
- 新增实例级决策 repository。
- 新增 API 类型和 list/submit 入口。

完成标准：

- 可以提交决策并持久化。
- 幂等键重复提交不会产生重复记录。
- 决策尚不推进 runtime 时也能被监控查看。

### Phase 2：GraphLoop 人工边传播

目标：

- 新增 `GraphHarness.apply_human_edge_decision()`。
- 新增 `GraphLoop.apply_human_edge_decision_and_checkpoint()`。
- `pass/revise/replace` 能更新 edge state、flow packet、ready nodes、checkpoint。

完成标准：

- 人工 pass 能让下游节点 ready。
- 人工 revise 能让返修节点 ready，并重置受影响下游。
- 人工 replace 能写项目文件并继续传播。
- 所有动作都有 event 和 checkpoint。

### Phase 3：编译器协议

## 12. 验证矩阵

| 场景 | 验证点 |
| --- | --- |
| pass | source result 通过 selected edge 生成 packet，下游 ready |
| revise | feedback packet 进入返修边，上游/修订节点 ready |
| replace | 文件写入正式库，human artifact ref 被传播 |
| 重复提交 | idempotency key 返回已有 decision，不重复推进 |
| 非法边 | API 返回 409，不写 checkpoint |
| 非法状态 | graph_run / config hash 不匹配时拒绝 |

| 恢复 | 后端重启后 decision ledger 和 checkpoint 可恢复 |
| 前端 | 按钮来自 `human_controls`，无能力不展示 |

## 13. 反模式

必须避免：

- 把人工退稿写成普通聊天消息，让 agent 自己猜下一步。
- 前端直接设置 node status / edge status。
- 后端 API 绕过 GraphLoop 写 checkpoint。
- `pass` 只标记节点完成但不生成边传播包。
- `replace` 把长正文直接塞进全局 memory。
- 编译器不解释人工动作，让用户自己选协议字段。
- 保留两套人工介入概念：审核人工介入一套、写手建议一套。

## 14. 建议锁定的 MVP

第一版只做：

```text
pass      通过并传给下游
revise    退稿并回传上游/返修节点
replace   用户替写并作为正式产物继续传播
```

不做：

- 节点节流。
- 任意节点 A2A 交流。
- 复杂人工投票。
- 多人权限流。
- 任意边重路由。

这样足够支撑写作图任务“一章一章推进到 100 章”的体验，也能成为通用图任务控制系统的基础。

### 15.3 EdgeControlService

新增文件：

```text

```

职责：

```text
  -> require instance
  -> resolve graph_run_id
  -> load published GraphHarnessConfig
  -> load GraphLoopState
```

伪代码：

```python
def submit_decision(instance_id, request, *, apply_now=True):
    instance = instance_repo.require(instance_id)
    graph_run_id = request.graph_run_id or instance.active_graph_run_id
    graph_config = registry.get_published_graph_harness_config(instance.graph_id)
    state = graph_harness.graph_loop.get_state(graph_run_id)
    edge = require_edge(graph_config, request.edge_id)
    policy = require_human_control_policy(graph_config, edge, request.decision)
    decision = normalize_decision(instance, graph_config, state, edge, policy, request)
    existing = repo.find_by_idempotency_key(instance_id, decision.idempotency_key)
    if existing:
        return existing
    if decision.content_submission:
        decision = materialize_submission(decision)
    decision = repo.create(instance_id, decision)
    if not apply_now:
        return decision
    try:
        advance = graph_harness.apply_human_edge_decision(
            graph_config=graph_config,
            graph_run_id=graph_run_id,
            decision=decision.to_dict(),
        )
    except Exception as exc:
        repo.transition(instance_id, decision.decision_id, "failed", {"apply_error": str(exc)})
        raise
    repo.transition(instance_id, decision.decision_id, "applied", {"apply_result_ref": advance.checkpoint.get("checkpoint_id", "")})
    return decision, advance
```

服务层不能直接写 `GraphLoopState`。它只负责验证、写文件、写账本、调用 GraphHarness。

### 16.2 关键内部 helper

现有 `_edge_states_after_node_result()` 会处理一个节点的所有出边。人工控制必须只处理用户选择的边，因此需要新增单边 helper：

```python
def _edge_state_after_result_for_edge(
    *,
    graph_config: GraphHarnessConfig,
    state: GraphLoopState,
    edge: dict[str, Any],
    result: NodeResultEnvelope,
    result_ref: str,
    services: Any | None,
) -> dict[str, Any]:
    ...
```

它只更新 `decision.edge_id` 对应的 edge state，并复用 `build_flow_packet()` / `store_flow_packet()`。

这是本设计的核心点：人工传播必须以 selected edge 为单位，不能退回“节点完成后所有出边一起 ready”的旧逻辑。

### 16.3 pass 算法

输入：

```text
decision.decision == "pass"
edge = selected forward edge
source node has completed result or human gate source result
```

步骤：

```text
1. assert graph_config compatible with state
2. require edge exists and human_control_policy allows pass
3. require source_node_id == edge.source_node_id
4. require target_node_id == edge.target_node_id
5. load source result from node_state.result_ref or decision.source_result_ref
6. if source node is waiting_human_gate, clear gate and mark completed
7. generate packet only for selected edge
8. set selected edge status ready
9. recompute ready/running/completed/blocked through state machine
10. dispatch ready nodes unless graph is blocked/terminal
11. write checkpoint and event
```

注意：

- 不修改 source result 内容。
- 不清空无关 active work orders。
- 不让其他 outbound edges 自动 ready。

### 16.4 revise 算法

输入：

```text
decision.decision == "revise"
edge = selected revision edge
instruction is required
```

步骤：

```text
1. require edge protocol kind in review_feedback / conditional_route, or human_control_policy.revise.route_edge_id matches
2. create human-authored NodeResultEnvelope:
   node_id = edge.source_node_id
   executor_type = "human"
   work_order_id = "human-edge-decision:{decision_id}"
   outputs.human_feedback = instruction
   decisions.review_verdict = "revise"
   handoff_summary = instruction
3. store node result
4. append result_history for source node
5. generate packet for selected revision edge
6. reset affected downstream scope according to policy
7. mark revision target node ready
8. checkpoint and dispatch
```

`result_index` 是否替换：

- 如果 source node 是 review/human gate 节点，允许把 `result_index[source_node_id]` 更新为人工裁决。
- 如果 source node 是普通 writer，不应把 writer 产物覆盖为反馈；只写 result_history，并通过 revision edge 传 feedback。

这个判断由 `human_control_policy.revise.result_index_policy` 控制：

```yaml
result_index_policy: replace_if_control_source
```

### 16.5 replace 算法

输入：

```text
decision.decision == "replace"
content_submission or artifact_refs required
```

步骤：

```text
1. materialize content_submission through GraphTaskInstanceFileService
2. create artifact_ref pointing to project file
3. create human-authored NodeResultEnvelope:
   node_id = edge.source_node_id
   executor_type = "human"
   outputs.human_artifact_submission = {...}
   artifact_refs = [project_file_ref]
   handoff_summary = "用户提交正式产物并替代节点输出"
4. store node result
5. update result_index[source_node_id]
6. append result_history[source_node_id]
7. clear waiting gate if present
8. generate packet for selected edge
9. checkpoint and dispatch
```

replace 的默认政策：

- 写项目正式库，不写运行产物目录。
- 不自动写全局 memory。
- memory 写入必须另由 `memory_commit` 边或资源契约完成。

### 19.4 提交行为

`pass`：

```text
submitGraphTaskInstanceHumanEdgeDecision({
  graph_run_id,
  edge_id,
  decision: "pass",
  artifact_refs: selected file/artifact refs,
  apply_now: true
})
```

`revise`：

```text
submitGraphTaskInstanceHumanEdgeDecision({
  graph_run_id,
  edge_id,
  decision: "revise",
  instruction,
  artifact_refs,
  apply_now: true
})
```

`replace`：

```text
submitGraphTaskInstanceHumanEdgeDecision({
  graph_run_id,
  edge_id,
  decision: "replace",
  instruction,
  content_submission: {
    path,
    content,
    content_kind: "chapter",
    commit_policy: "project_file"
  },
  apply_now: true
})
```

提交成功后：

```text
refresh monitor
refresh file tree if replace
keep user on current screen
show applied decision notice
```

## 20. 端到端固定流程

### 20.1 用户通过章节

```mermaid
sequenceDiagram
  participant User as 用户
  participant FE as 图任务前台
  participant API as GraphTaskInstance API
  participant SVC as EdgeControlService
  participant Loop as GraphLoop
  participant Store as Decision/File Store

  User->>FE: 点击“通过并传给下游”
  FE->>API: POST human-edge-decisions(pass)
  API->>SVC: submit_decision(apply_now=true)
  SVC->>Store: 写入 submitted decision
  SVC->>Loop: apply_human_edge_decision
  Loop->>Loop: selected edge 生成 FlowPacket
  Loop->>Loop: checkpoint + dispatch ready
  Loop-->>SVC: GraphLoopAdvance
  SVC->>Store: decision -> applied
  API-->>FE: decision + apply_result
  FE->>API: refresh monitor
```

### 20.2 用户退稿

```text
章节内容
  -> 用户输入退稿意见
  -> HumanEdgeDecision(revise)
  -> revision edge packet
  -> writer/revise node ready
  -> 下游章节相关 scope reset
```

### 20.3 用户替写

```text
用户输入正文
  -> FileService 写 chapters/chapter-012.md
  -> human-authored NodeResultEnvelope
  -> selected edge FlowPacket
  -> 下游继续
```

## 21. 这次设计锁定的实现边界

第一轮实现必须完成：

- `HumanEdgeDecision` 模型和 repository。
- `submit human edge decision` API。
- `GraphLoop.apply_human_edge_decision_and_checkpoint()`。
- `pass/revise/replace` 三种动作。
- `human_control_policy` 编译与 compile report。
- `GraphTaskInstanceMonitor.human_controls` 投影。
- 前端动作条、决策抽屉、历史展示。
- 后端和前端测试。

第一轮不做：

- 多人审批。
- 任意重路由。
- A2A 讨论控制。
- 节点节流。
- 复杂权限组织。
- 独立 memory 人工提交 UI。

## 22. 设计自检

本设计满足用户提出的关键要求：

- 不是两种人工介入，而是一种通用边传播控制。
- 通过是向下游传播。
- 退稿是向上游或返修边传播。
- 用户替写是正式产物提交，不是聊天消息。
- 前端和后端都围绕图任务实例工作区。
- GraphLoop 仍是唯一运行推进权威。
- 环境仍只属于节点装配，不绑定图任务。
- 写作图可以用，其他通用图任务也可以用。

# 非图任务体系审计报告

审计日期：2026-06-15

审计范围：普通 single-agent turn、由普通对话升级出的 `single_agent_task`、subagent task、ActiveTurn、TaskRun/AgentRun 状态存储、暂停/继续/停止控制链路。

不在本报告范围：graph runtime、graph node task、graph work order、graph background supervisor。

## 结论

项目的 task/turn 核心分层方向是正确的：普通 turn 可以直接调用工具，不需要创建 TaskRun；TaskRun 承担持久、可暂停、可恢复、可监督的长任务生命周期。这与 Codex / Claude Code 这类成熟 coding agent 的公开行为一致：工具调用属于普通 agent loop 能力，task 不是工具访问前提，而是后台或持久工作单元。

当前主要缺陷不在“普通 turn 是否能调工具”，而在非图任务体系的状态规范和 AgentRun/TaskRun 投影一致性。最需要优先处理的是 AgentRun 状态词表漂移，其次是 TaskRun 终态别名进入多套判断逻辑后导致恢复、监控、active work 判断分裂。

## 已达标的结构

### 普通 turn 和 TaskRun 没有混线

`TurnRun` 明确声明自己只是对话 turn 的可追踪运行记录，不是 orchestrated TaskRun。位置：

- `backend/runtime/shared/models.py:107`

普通 turn 的工具调用走 `single_agent_turn` 内部工具批处理和 tool control plane，工具观测的 `caller_kind` 是 `agent_turn`，`caller_ref` 指向 `turnrun:`。位置：

- `backend/harness/loop/single_agent_turn.py:953`
- `backend/harness/loop/single_agent_turn.py:1033`
- `backend/harness/loop/single_agent_turn.py:3171`
- `backend/harness/loop/single_agent_turn.py:3194`

测试明确保证普通 turn 调工具不产生 TaskRun：

- `backend/tests/harness_single_agent_tool_runtime_regression.py:94`
- `backend/tests/harness_single_agent_tool_runtime_regression.py:173`
- `backend/tests/harness_single_agent_tool_runtime_regression.py:186`

这点是成熟架构，不能倒退成“只有 TaskRun 才能调工具”。

### 普通 turn 升级为长任务时有合同门槛

模型请求创建 TaskRun 时必须经过 `contract_from_action_request`，要求任务目标、完成标准或产物/验证要求、工作范围、能力意图、skill 意图、观察契约。位置：

- `backend/harness/loop/task_lifecycle.py:85`
- `backend/harness/loop/task_lifecycle.py:602`

这比“模型说要 task 就创建 task”的做法成熟。TaskRun 有明确的 `TaskRunContract`，并在启动时写入 runtime object 与 TaskRun diagnostics：

- `backend/harness/loop/task_lifecycle.py:308`
- `backend/harness/loop/task_lifecycle.py:334`
- `backend/harness/loop/task_lifecycle.py:413`

### ActiveTurn 没有从历史 TaskRun 反推当前工作

`ActiveTurnRegistry` 明确是 current-turn authority，和 TaskRun、RuntimeRun 分离。位置：

- `backend/harness/runtime/active_turn.py:43`

测试也覆盖了历史 `waiting_executor` TaskRun 不会自动成为当前 active turn：

- `backend/tests/active_turn_authority_regression.py:37`

这点很重要。成熟 agent 必须区分“历史任务状态”和“当前用户输入绑定到哪个可控工作”。

## 主要缺陷

### P1：AgentRun 状态词表与真实写入不一致

核心定义中 `AgentRunStatus` 只允许：

```text
pending / running / completed / failed / killed
```

位置：

- `backend/runtime/shared/models.py:53`
- `backend/runtime/shared/models.py:154`

但非图任务链路实际写入了不属于该枚举的状态：

- `created`：`backend/harness/loop/task_lifecycle.py:395`
- `created`：`backend/harness/loop/task_lifecycle.py:404`
- `waiting_executor`：`backend/harness/agent_control/controller.py:431`
- `waiting_executor`：`backend/harness/loop/task_executor.py:5586`
- `waiting_executor`：`backend/harness/loop/task_executor.py:5721`
- `waiting_executor`：`backend/harness/loop/task_executor.py:5966`
- `blocked`：`backend/harness/loop/task_executor.py:5602`
- `blocked`：`backend/harness/loop/task_executor.py:5984`
- `blocked`：`backend/harness/loop/task_executor.py:6041`

Python 的 `Literal` 不会运行时校验，所以这些值会正常进入状态存储。反序列化也没有 canonicalize：

- `backend/runtime/memory/state_index.py:1654`

风险场景：

1. 主任务启动时 AgentRun 以 `created` 入库，但 AgentRunStatus 没有 `created`。
2. 执行器开始时 `_ensure_agent_run` 会把已有 AgentRun 改成 `running`，但暂停、等待、阻塞时又可能把 AgentRun 改成 `waiting_executor` 或 `blocked`。
3. subagent 控制器用 `pending/running` 判断 active subagent：
   - `backend/harness/agent_control/controller.py:302`
4. 如果某个子 agent run 被写成 `waiting_executor`，它既不是 terminal，也不在 active 集合里，可能绕过 `max_active_subagents` 或在列表/等待语义中变成“不活跃但未结束”的灰态。

成熟标准：

AgentRun 应该表达 agent execution instance 的生命周期，建议只保留：

```text
pending / running / completed / failed / killed
```

等待执行器、暂停、等待用户、等待审批、可恢复等状态应放在 TaskRun 的 `status`、`diagnostics.executor_status`、`diagnostics.runtime_control.state` 或专门的 execution lease 字段中，不应该污染 AgentRun.status。

修复方向：

1. 清理所有 `AgentRun.status="created"`，改成 `pending`。
2. 清理所有 `AgentRun.status="waiting_executor"`，根据场景改成：
   - 任务未执行：`pending`
   - 当前执行中但暂停到安全边界：`running` 或 `failed`，具体等待态放到 diagnostics
   - 子任务被父任务关闭：`killed`
3. 清理所有 `AgentRun.status="blocked"`，改成 `failed`，阻塞原因放 `diagnostics.terminal_reason` 或 result。
4. 在 `AgentRun.__post_init__` 或 state_index 写入边界加入运行时校验，拒绝非规范状态再次进入。

### P1：TaskRun 状态规范和投影别名分裂

核心 TaskRunStatus 定义为：

```text
created / running / waiting_executor / waiting_approval / blocked / completed / failed / aborted
```

位置：

- `backend/runtime/shared/models.py:7`
- `backend/runtime/shared/models.py:83`

但终态判断和状态投影接受了大量别名：

- `success/error/cancelled/canceled/stopped/user_aborted`：`backend/harness/task_run_status.py:6`
- `done/succeeded`：`backend/harness/task_run_state_view.py:6`
- `blocked_expired/runtime_retention_expired/approval_expired`：`backend/harness/task_run_state_view.py:8`
- `success`：`backend/harness/loop/task_run_recovery_state.py:14`

同时 state_index 从存储读取 TaskRun 时直接保留 payload status：

- `backend/runtime/memory/state_index.py:1435`

风险场景：

1. 某条历史或测试链路写入 `status="success"`。
2. `task_run_state_view` 会把它当 completed。
3. 另一个只按核心 TaskRunStatus 或只按 `completed/failed/aborted` 判断的模块可能不认它。
4. ActiveTurn、run monitor、active executor index、recent work outcome 可能对同一条 TaskRun 得出不同结论。

已有迹象：

- `ActiveTurnRegistry.resolve_current` 依赖 `is_stopped_or_terminal_task_run`，它接受别名。
- `_release_transient_active_turn` 又内联了一套终态集合，包含 `success/error/cancelled/canceled/stopped/user_aborted`。
- `task_run_state_view` 还有自己的 `COMPLETED_STATUSES/FAILED_STATUSES/STOPPED_STATUSES`。

位置：

- `backend/harness/runtime/active_turn.py:100`
- `backend/harness/entrypoint/runtime_facade.py:1128`
- `backend/harness/task_run_state_view.py:6`

成熟标准：

状态别名只能存在于输入兼容边界，不能在核心状态机内部长期流动。核心持久状态必须 canonical。

修复方向：

1. 定义唯一 canonical TaskRun status 表：
   - active：`created/running`
   - waiting：`waiting_executor/waiting_approval`
   - blocked：`blocked`
   - terminal：`completed/failed/aborted`
2. 定义唯一 canonical terminal_reason 表，过期、取消、用户停止等放 terminal_reason，不扩展 status。
3. 建一个 `normalize_task_run_status_for_storage`，只在 API/旧存储读取边界接受别名，写回状态时统一 canonical。
4. 删除各模块重复的终态集合，统一调用同一个状态库。

### P2：ActiveWorkContext 对运行态的判断过度依赖原始 status

Active work context 判断：

```python
running = status in {"created", "running"}
continuation_kind = "paused" if control_state == "paused" else ("active" if running else "waiting")
```

位置：

- `backend/harness/entrypoint/runtime_facade.py:1096`
- `backend/harness/entrypoint/runtime_facade.py:1114`

这在状态词表干净时可用，但当前 TaskRun/AgentRun 状态别名较多，会带来投影偏差。

风险场景：

1. TaskRun 被标为 `waiting_executor`，但 diagnostics 中 `executor_status="scheduled"`。
2. `task_run_state_view` 可能认为它 running_claimed。
3. `_active_work_context_from_task_run` 却显示 `running=False`，`continuation_kind="waiting"`。
4. 用户继续输入时，decision 层可能把它当等待用户/可继续任务，而不是已经有执行器 claim 的任务。

成熟标准：

ActiveWorkContext 不应直接用裸 status 判断 running，而应使用统一的 TaskRunStateView：

- `task_work_state`
- `running_claimed`
- `can_resume`
- `control_state`
- `executor_lease_state`

修复方向：

1. `_active_work_context_from_task_run` 直接消费 `task_run_state_view(task_run)`。
2. `running` 取 `view["running_claimed"] or view["task_work_state"] == "active"`。
3. `resumable` 取 `view["can_resume"]`。
4. `continuation_kind` 从 `task_work_state/control_state` 映射，而不是裸 status。

### P2：subagent task 的 AgentRun 与 TaskRun 状态边界不清

subagent 创建时：

- child AgentRun：`status="pending"`，合理。
- child TaskRun：`status="waiting_executor"`，合理。

位置：

- `backend/harness/agent_control/controller.py:108`
- `backend/harness/agent_control/controller.py:422`

问题在后续状态：

1. 子任务 TaskRun 的等待/执行状态会被同步或间接投射到 AgentRun.status。
2. subagent active 数量只把 `pending/running` 当 active：
   - `backend/harness/agent_control/controller.py:302`
3. 子 agent run 如果被写成 `waiting_executor` 或 `blocked`，就不再被 active 限制统计，但也不是 `completed/failed/killed`。

风险场景：

父 agent 连续 spawn subagent，已有子任务还处于 TaskRun waiting_executor，但 AgentRun.status 不在 active 集合，导致 active subagent 限制失效或显示不一致。

修复方向：

1. AgentRun.status 不表达等待执行器。
2. subagent active 判断应基于 AgentRun.status 的规范集合：
   - active：`pending/running`
   - terminal：`completed/failed/killed`
3. 子 TaskRun 的 `waiting_executor/waiting_approval/blocked` 只能影响子 TaskRun 控制面，不改变 AgentRun.status 到非枚举值。

### P3：TaskRun 状态写入缺少运行时校验

`TaskRun.__post_init__` 校验了 authority、id、session_id、task_id，但没有校验 status 和 terminal_reason 是否在规范集合内。

位置：

- `backend/runtime/shared/models.py:72`

state_index 反序列化也没有校验：

- `backend/runtime/memory/state_index.py:1435`

风险场景：

任何旧链路、测试、手工数据、未来代码都可以写入 `status="stopped"`、`status="success"`、`status="done"`。由于当前多个模块接受这些别名，问题短期不会爆炸，但会让状态机越来越难收敛。

修复方向：

1. 在 `TaskRun.__post_init__` 校验 `status` 属于 canonical set。
2. 在 state_index 读取旧数据时做一次 legacy alias canonicalization，并写回清理后的状态。
3. 测试里不要再构造 `status="stopped"` 这种非规范 TaskRun；改用 `status="aborted", terminal_reason="user_aborted"`。

### P3：TaskRun terminal_reason 混入等待态和流程态

当前有不少代码会设置：

- `terminal_reason="waiting_executor"`
- `terminal_reason="waiting_approval"`
- `terminal_reason="task_executor_scheduled"`
- `terminal_reason="task_launch_supervision"`

位置示例：

- `backend/harness/loop/task_executor.py:348`
- `backend/harness/loop/task_executor.py:460`
- `backend/harness/loop/task_executor.py:5855`
- `backend/harness/loop/task_lifecycle.py:540`
- `backend/harness/loop/task_lifecycle.py:758`

这些值不是 terminal reason，而是等待原因、流程阶段或调度结果。这样会让“terminal_reason 是否为空”失去语义。

成熟标准：

`terminal_reason` 只在 terminal 或 interrupted boundary 上有值。等待继续、等待审批、已调度应放：

- `status`
- `diagnostics.executor_status`
- `diagnostics.latest_step`
- `diagnostics.pending_approval`
- `diagnostics.recovery_action`

修复方向：

1. 对非 terminal TaskRun，默认 `terminal_reason=""`。
2. 如果确实需要表达等待原因，新增 `diagnostics.wait_reason` 或复用 `latest_step`。
3. `waiting_executor` 作为 status，不应同时作为 terminal_reason。

- 非 terminal 状态下 `terminal_reason` 为空，等待原因进入 diagnostics。

## 不建议的修复方式

不要通过继续扩充状态别名解决问题。例如继续把 `created/waiting_executor/blocked` 加进 `AgentRunStatus`，虽然短期能让类型看起来匹配，但会把 TaskRun 的控制状态永久污染到 AgentRun 生命周期里。

不要在 UI 或 API 层单独兜底每种状态。问题根源在持久状态规范没有收敛，应该从模型层和写入边界解决。

不要把普通 turn 的工具调用改成 TaskRun。当前普通 turn 可直接调工具是正确设计，改成所有工具调用都生成 TaskRun 会降低交互效率，也会让短 turn 和长任务生命周期混线。

## 最小验证清单

修复后建议保留这些结构性测试：

1. 普通 turn 调 `read_file`，`task_run_count == 0`，tool observation `caller_kind == "agent_turn"`。
2. 模型请求 `request_task_run` 时，缺少目标或完成证据会拒绝创建 TaskRun。
3. `AgentRun.status` 非规范值写入失败或读取时被规范化。
4. `TaskRun.status` 非规范别名读取后 canonicalize。
5. `waiting_executor + recovery_action=resume_task_run` 的 TaskRun 在 active work 中显示可继续。
6. `waiting_executor + executor_status=scheduled` 的 TaskRun 在 active work 中显示 already running。
7. subagent active 限制只根据规范 AgentRun 状态判断，不受 TaskRun waiting_executor 污染。

## 确认后的执行计划

本节为 2026-06-15 确认后的落地计划。图任务仍不在本次实施范围内。

### 目标架构

普通 turn 继续直接调用工具。TaskRun 只负责长任务生命周期。AgentRun 只表示 agent execution instance 生命周期。等待、暂停、审批、执行器租约状态统一放到 TaskRun status 与 diagnostics，不再污染 AgentRun.status。

### 第一阶段：收敛 AgentRun 状态

规范状态只保留：

```text
pending / running / completed / failed / killed
```

实施动作：

1. 将 `AgentRun(status="created")` 改为 `pending`。
2. 将 AgentRun 的 `waiting_executor` 改为 `pending` 或 `running`，等待语义放到 TaskRun/diagnostics。
3. 将 AgentRun 的 `blocked` 改为 `failed`，阻塞原因写入 diagnostics。
4. 在模型或 state index 边界加入校验或规范化，防止非规范 AgentRun.status 再进入存储。

涉及文件：

- `backend/runtime/shared/models.py`
- `backend/harness/loop/task_lifecycle.py`
- `backend/harness/loop/task_executor.py`
- `backend/harness/agent_control/controller.py`
- `backend/runtime/memory/state_index.py`

### 第二阶段：统一 TaskRun 状态规范

`TaskRun.status` 只允许 canonical set：

```text
created / running / waiting_executor / waiting_approval / blocked / completed / failed / aborted
```

实施动作：

1. `success/error/cancelled/canceled/stopped/done/succeeded/user_aborted` 只允许在 legacy normalization 边界出现。
2. 删除或收敛重复状态集合，让 `task_run_status.py` 成为唯一状态权威。
3. state index 读取旧状态时 canonicalize，写入新状态时拒绝非规范值。

涉及文件：

- `backend/harness/task_run_status.py`
- `backend/harness/task_run_state_view.py`
- `backend/harness/loop/task_run_recovery_state.py`
- `backend/harness/runtime/active_turn.py`
- `backend/harness/entrypoint/runtime_facade.py`
- `backend/runtime/memory/state_index.py`

### 第三阶段：清理 terminal_reason 语义

非终态 TaskRun 不再写：

```text
waiting_executor / waiting_approval / task_executor_scheduled / task_launch_supervision
```

等待原因放入 diagnostics，例如 `latest_step`、`executor_status`、`wait_reason`、`pending_approval`。真正 terminal 才写 terminal_reason。

### 第四阶段：ActiveWorkContext 改用 TaskRunStateView

不再用裸 `status in {"created", "running"}` 判断 running。改用 `task_run_state_view(task_run)` 的：

- `task_work_state`
- `running_claimed`
- `can_resume`
- `executor_lease_state`
- `control_state`

确保：

- `waiting_executor + executor_status=scheduled` 显示 already running。
- `waiting_executor + recovery_action=resume_task_run` 显示可继续。
- `waiting_approval` 显示 needs approval。

### 第五阶段：测试与验证

聚焦测试：


结构性验收：

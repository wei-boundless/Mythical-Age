# 长期记忆重构收口清单

> 目的：把 `docs/29-长期记忆重构方案.md` 从“方向正确的蓝图”收口成“当前实现到底完成到哪一步、哪些旧路径必须下线、下一轮改造先切哪里”的执行清单。

---

## 1. 结论

当前问题的核心不是 `29` 方案方向错了，而是实现仍停留在**新旧双路径并存**的过渡态。

换句话说：

- 方案里已经指出了大部分真正的问题
- 但代码还没有把这些问题从主路径中彻底移除
- 一些关键约束只停留在原则层，没有变成代码级退出条件

因此，这一轮的目标不是再补一份泛化方案，而是把旧主路径真正下线。

---

## 2. 已完成

这些部分已经形成了新架构的骨架，可以保留并继续往前推进。

### 2.1 长期记忆的文件型分层

已具备：

- `context_profile/` 作为静态长期上下文
- `durable_memory/index/MEMORY.md` 作为 manifest 入口
- `durable_memory/notes/*.md` 作为 durable note 真相源
- `session-memory/` 作为 working memory 存储

判断：

- 这部分方向符合 `29` 的 Claude Code 风格分层
- 问题不在“有没有层”，而在“边界还不够硬”

### 2.2 session/context/task 第一轮隔离骨架

已具备：

- `MainContextState`
- `TaskSummaryRef`
- `AnswerAssembler`
- `FollowupResolver`
- `TaskContextRef / result_ref / summary` 的第一轮落地

判断：

- 上下文隔离的主骨架已经有了
- 但 durable memory 仍然还没完全接上这套新边界

### 2.3 durable schema 与 note 治理字段的第一轮升级

已具备：

- `schema_version`
- `canonical_statement`
- `retrieval_hints`
- `status`
- `eligible_for_injection`
- `supersedes`
- `last_confirmed_at`

判断：

- note 生命周期治理已经开始
- 但治理字段存在，不等于旧脏数据已经退出主流程

## 3. 半完成

这些部分已经开始改，但仍然被旧路径掺回主流程，是当前最危险的区域。

### 3.1 extractor 已加入 projection/state 输入，但 message-first 旧路径仍在主流程

当前状态：

- `backend/structured_memory/extractor.py`
  - 已支持 projection message
  - 已支持 session state durable candidates
  - 但 `extract()` 末尾仍直接遍历最近消息里的显式写入句

问题本质：

- durable extraction 仍然没有真正切换到 `summary-first / projection-first`
- message transcript 只是从唯一输入，降成了并列输入
- 这仍不符合 `29` 的“messages-only extract 退为兼容模式”

必须收口为：

- 主路径：`projection -> session state candidates`
- fallback：只有前两者都没有候选时，才允许 explicit message fallback

### 3.2 memory_intent 已有 read/write/signal 字段，但仍是强路由器

当前状态：

- `backend/understanding/memory_intent.py`
  - 已区分 session continuity / durable read / durable write
  - 但语义 recall 仍会直接返回 `durable_memory_query + should_skip_rag=True`

问题本质：

- durable read 仍在抢主路由
- “项目主线是什么”“你知道我喜欢你怎么回答吗”这类语义 recall，本应是弱信号，不应直接把 route 切成 memory-only

必须收口为：

- 显式问“长期记忆里有什么 / 你都记了什么 / what do you remember”才是强 memory route
- 语义 recall 只返回 `memory_read_signal`
- `should_skip_rag=False`

### 3.3 runtime 已接入 context/task 分层，但 durable read 时机仍过早

当前状态：

- `backend/query/runtime.py`
  - retrieval 前就创建 `prefetch_relevant_notes` task
  - 后续 prompt 装配默认仍会经过 durable block builder

问题本质：

- durable 仍然像“预加载大背景”
- 不符合 `29` 里“durable read 必须排在 task/session/evidence 之后”的要求

必须收口为：

- 先跑 task/session/retrieval 判断
- 仅当存在显式 memory read 或弱 durable signal 时才读 durable
- generic turn 不再预取 durable notes

### 3.4 session memory 新增了 context-state 入口，但内部仍回投成伪 messages

当前状态：

- `backend/structured_memory/session_memory.py`
  - 已有 `update_from_context_state(...)`
  - 但内部仍调用 `_project_context_state_to_messages(...)`
  - 再交给 message-centric processor

问题本质：

- 这说明 session memory 还不是纯 projection/reducer
- 只是把 state 重新包成 transcript，再让旧逻辑消费

结论：

- 这是第二波收口项
- 不是本轮第一刀，但必须在 `29` 的最终验收前完成

## 4. 未完成

这些是方案里明确提出，但代码尚未真正落地的点。

### 4.1 durable read 的“manifest fallback 只在显式 memory inventory 查询时触发”

当前缺口：

- `backend/memory/durable.py` 在没有 exact/relevant 命中时，仍然较宽地回退到 manifest

目标：

- manifest fallback 只服务于“你都记了什么 / 长期记忆里有什么”这类显式元记忆查询
- generic query 不再因为 durable block builder 默认存在，就平白多出 memory 噪声

### 4.2 session memory 还没有完全成为恢复索引

当前缺口：

- 仍在复用旧 `SessionUnderstandingProcessor`
- 仍保留强 transcript 处理思路

目标：

- 直接由 `main_context + task_summaries + corrections + restore_hints` 构建 process state
- 不再要求先翻译成 user/assistant message

### 4.3 durable 候选生成器还没有彻底从 task-local/transient 内容中独立出来

当前缺口：

- 现有 candidate 过滤已经比以前强
- 但仍有一部分逻辑挂在 extractor 侧，而不是单独稳定的 projection pipeline

目标：

- durable candidates 由显式 projection 来源提供
- extractor 负责归一化与保存，不再承担“从任何输入里猜 durable memory”

---

## 5. 必须先下线的旧路径

这一节不是建议，而是本轮必须切掉的旧主路径。

### 5.1 `extractor.extract()` 里的 transcript explicit-write 主循环

位置：

- `backend/structured_memory/extractor.py`

原因：

- 它让 transcript 仍然是 durable write 的主来源之一
- 与 `29` 的 projection-first 原则冲突

本轮处理：

- 降级成 fallback only

### 5.2 `memory_intent` 把语义 recall 直接升级为 `durable_memory_query`

位置：

- `backend/understanding/memory_intent.py`

原因：

- 这是 durable 抢主路由的核心来源

本轮处理：

- 改成 `memory_read_signal`
- 仅显式元记忆问法保留强 route

### 5.3 `runtime` 每轮预取 durable notes

位置：

- `backend/query/runtime.py`

原因：

- 这会把 durable 重新变成 prompt 常驻背景噪声

本轮处理：

- 改成条件触发
- 并放到 retrieval 判断之后

### 5.4 `build_persistent_memory_block()` 的泛化 manifest 回退

位置：

- `backend/memory/durable.py`

原因：

- 它会让没有 durable 需要的 turn 也平白多一段 memory 提示

本轮处理：

- 仅保留在显式 memory inventory 查询场景

---

## 6. 本轮执行顺序

本轮按下面顺序推进，不再回到大而空的方案讨论：

1. 收口清单落档
2. 下线 extractor 的 message-first 主路径
3. 把 `memory_intent` 从强路由改成弱信号优先
4. 收紧 runtime 的 durable read 时机
5. 收紧 durable manifest fallback
6. 更新回归测试
7. 跑 targeted regressions

---

## 7. 本轮完成标准

本轮不是要一次性做完整个 `29`，而是要先让主路径不再违背 `29`。

达到下面条件，就算本轮收口成功：

1. transcript 不再是 durable extraction 的主输入
2. 语义 recall 不再直接抢 memory-only route
3. durable notes 不再每轮预取
4. generic turn 不再默认出现 durable manifest fallback
5. 现有回归测试更新后通过

---

## 8. 仍留到下一轮的事项

这些不是放弃，而是明确排到下一轮：

- `session_memory.py` 去消息桥接
- process-state 彻底 projection-first
- durable candidates 独立成稳定 projection pipeline
- 旧 note 批量治理与长场景再验证

结论：

- `29` 方案没有根本遗漏
- 真正的问题是实现没有收口
- 本轮就是把最关键的旧主路径切干净，然后再继续第二波重构

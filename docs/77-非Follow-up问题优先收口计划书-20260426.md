## 1. 这份计划书解决什么问题

当前长场景里的残留问题很多都“看起来像 follow-up 问题”，但如果现在直接进入 follow-up 主链重构，会有三个现实风险：

1. 问题耦合太高，修一处会影响 PDF、结构化数据、记忆、RAG 多条链路。
2. 还有一批外围问题本身就会制造噪音，导致我们很难判断 follow-up 修复是否真的有效。
3. 如果不先把外围问题收口，后续对 follow-up 的验证会一直被“伪失败”和“弱质量问题”污染。

因此本计划书的原则是：

> **先把非 follow-up 问题逐项追踪、逐项修掉，再集中研究 follow-up 主链。**

这里的“非 follow-up 问题”不是指完全与上下文无关，而是指：

1. 它们的主问题不在 follow-up 解析边界本身。
2. 即使 follow-up 暂时不动，也可以先推进定位、修复、回归验证。

---

## 2. 当前阶段划分

### 阶段 A：先收口非 follow-up 问题

本阶段只处理以下四组问题：

1. 长期记忆召回稳定性
2. PDF 降级输出质量
3. 结构化数据执行协议一致性
4. 工具失败兜底与错误可解释性

### 阶段 B：集中研究 follow-up 主链

本阶段暂缓，不在本计划书内直接施工：

1. follow-up 解析
2. active_goal / flow_state 漂移
3. risk flags 到 runtime gate 的执行门控
4. bounded-agent 在高风险上下文中的工具暴露问题

本计划书的交付目标不是“最终全绿”，而是：

> **先把外围独立问题清掉，让后续 follow-up 研究面对的是更干净的系统。**

---

## 3. 问题分层与优先级

## 3.1 第一优先级：长期记忆召回稳定性

### 问题表现

在长场景中，用户已经明确写入过稳定偏好：

- `回答我时可以直接称呼我岩。`

但后续在问：

- `你之后应该怎么称呼我？`

时，系统仍然没有稳定召回“岩”。

对应失败 turn：

1. `turn-25-recall.json`
2. `turn-52-recall2.json`

对应运行：

- `output/test_runs/20260426-054022-long/artifacts/sixty-turn-real-user-marathon/turn-25-recall.json`
- `output/test_runs/20260426-054022-long/artifacts/sixty-turn-real-user-marathon/turn-52-recall2.json`

### 为什么它适合先修

这类问题边界清楚：

1. 写入对象明确
2. 召回问题明确
3. 不依赖 follow-up 主链重构
4. 修完之后收益直接可见

### 当前怀疑点

需要逐项排查：

1. durable memory 写入是否真的成功
2. 写入后的存储类型是否正确
3. 召回排序是否把这条偏好排掉了
4. prompt 组装时是否把相关记忆块正确注入
5. session memory / durable memory 是否存在覆盖或冲突

### 追踪目标

本问题必须给出以下明确结论：

1. “岩”这条偏好到底有没有被写入 durable memory
2. 如果写入了，为什么 recall 轮没有召回
3. 如果召回了，为什么最终回答没有使用

### 交付要求

1. 给出完整写入链路和读取链路的证据
2. 给出最小复现
3. 补回归测试，覆盖：
   - 稳定称呼偏好写入
   - 跨会话召回
   - 回答落地使用

---

## 3.2 第二优先级：PDF 降级输出质量

### 问题表现

当前 PDF 跟读链已经不再大量崩溃，但很多失败 turn 仍退化成：

- `已尝试读取这份 PDF 的 P3，但没有形成稳定摘要。原因：target_page_text_quality_low。`

对应失败 turn：

1. `turn-06-main.json`
2. `turn-07-main.json`
3. `turn-43-doc.json`

### 为什么它适合先修

这里先不碰 follow-up 续接逻辑本身，只处理“失败时怎么解释”和“降级时给用户多少可用信息”。

这块可以独立推进，因为：

1. 即使 section/page 续接逻辑暂时不变
2. 我们也可以先提升失败输出的质量
3. 避免现在这种“知道失败，但用户拿不到有效下一步”

### 当前问题不是哪里

当前主问题不是：

1. provider 崩了
2. 工具完全不可用

而是：

1. 已经感知到证据不足
2. 但没有把这种不足转成足够可用的用户解释

### 当前怀疑点

需要追踪：

1. degraded reason 的 typed 信息是否已经足够丰富
2. output policy 是否把 typed degraded reason 压扁成了统一模板
3. 在页面正文不足时，是否还能返回更细粒度的“能确认/不能确认”信息
4. 是否应该在失败输出中附带更强的下一步建议

### 交付要求

1. 梳理 PDF degraded reason 的完整类型表
2. 梳理每一种 degraded reason 目前的用户可见文案
3. 找出哪些 degraded reason 被错误合并
4. 补充更可操作的失败输出策略

---

## 3.3 第三优先级：结构化数据执行协议一致性

### 问题表现

当前结构化数据类 turn 经常出现一种很典型的状态：

1. 内容看起来答得基本对
2. 但 route / worker / tool 协议形态与预期不一致
3. 最终在长场景里仍被判失败

对应失败 turn：

1. `turn-10-main.json`
2. `turn-11-main.json`
3. `turn-13-main.json`
4. `turn-44-ops.json`
5. `turn-47-ops.json`
6. `turn-57-main.json`

### 为什么它适合先修

因为这里的核心问题不是 follow-up 语义边界，而是：

1. 同一类请求有时走 worker
2. 有时走 direct tool
3. 有时走 bounded-agent

也就是说，能力已有，但协议不稳。

这类问题如果先收口，后面 follow-up 研究会更轻松，因为：

1. 可以先把结构化数据这条链收成一个稳定主协议
2. 后续如果还有 follow-up 问题，就更容易判断是“续接错了”，还是“执行协议本来就不稳”

### 当前怀疑点

需要排查：

1. 为什么同类请求没有稳定走同一执行路线
2. current structured worker 与 direct tool facade 的边界是否清楚
3. route / worker / answer_source 的收口是否统一
4. subset / result handle 的使用是否稳定

### 交付要求

1. 把所有结构化数据失败 turn 分成“内容错”和“协议错”两类
2. 优先处理“内容对但协议错”的项
3. 形成一个统一的结构化数据主执行协议说明
4. 补 route/worker/tool 级别回归测试

---

## 3.4 第四优先级：工具失败兜底与错误可解释性

### 问题表现

某些失败或退化场景下，最终输出仍然过弱，例如：

- `调用工具失败`

对应典型案例：

1. `turn-55-main.json`

### 为什么它适合先修

这类问题相对独立，而且收益很稳定：

1. 不需要大规模改 runtime 主链
2. 不依赖 follow-up 语义重构
3. 但能明显改善用户感知和测试可诊断性

### 当前怀疑点

需要排查：

1. 哪些工具失败被统一压缩成了空泛兜底
2. 当前 output policy 是否丢失了足够有用的失败上下文
3. 是否存在已经拿到了失败原因，但没有正确透出给用户的情况

### 交付要求

1. 给出一张“工具失败 -> 当前文案 -> 目标文案”的映射表
2. 优先修最常见、最容易误导用户的失败类型
3. 补至少一组针对工具失败解释的回归测试

---

## 4. 暂缓处理但必须保留观察的 follow-up 主问题

本计划书虽然不优先处理 follow-up，但并不忽略它。

本阶段需要做的是：

1. 把 follow-up 相关问题单独留档
2. 在修外围问题时避免误伤它
3. 给后续集中研究 follow-up 留出更干净的验证环境

当前需要保留观察但暂缓施工的主问题包括：

1. `active_goal` 与 `flow_state` 漂移
2. `risk_flags` 已识别但未转成 execution gate
3. `bounded_agent` 在高风险 turn 中仍可自由试工具
4. 多线程总结、跨主题切换、模糊追问中的续接误判

也就是说：

> follow-up 不是不修，而是先不把它当作当前阶段的第一施工面。  

---

## 5. 本阶段的追踪方式

为了避免“问题很多，但最后都停留在口头判断”，本阶段要求每个问题都必须走完以下追踪路径：

1. 问题分类
2. 失败 turn 列表
3. 最小复现
4. 代码入口文件
5. 当前链路证据
6. 根因假设
7. 预期修复边界
8. 回归测试清单

每个问题都必须最终落成一张可执行表：

| 问题 | 失败 turn | 当前证据 | 主入口文件 | 是否依赖 follow-up | 是否可独立修复 | 当前状态 |
| --- | --- | --- | --- | --- | --- | --- |

## 6. 推荐施工顺序

### 第一步：先追长期记忆召回

目标：

1. 确认 durable 写入是否真实成功
2. 确认 recall 注入链是否正确
3. 确认回答落地是否生效

为什么先做：

1. 范围最清晰
2. 依赖最少
3. 容易形成确定结论

### 第二步：追 PDF 降级输出

目标：

1. 把 degraded reason 和用户可见回答对齐
2. 提升失败 turn 的可用性

### 第三步：追结构化数据协议一致性

目标：

1. 让 route / worker / tool 形态稳定
2. 降低“内容答对但结构不稳”的长场景噪音

### 第四步：追工具失败兜底

目标：

1. 提升诊断性
2. 提升用户可恢复性

### 第五步：再进入 follow-up 主研究

到这个阶段，外围噪音明显下降后，再集中处理：

1. follow-up 解析
2. session-memory 漂移
3. risk gate
4. bounded-agent 高风险工具暴露

---

## 7. 本阶段的验收标准

这份计划书对应的阶段性验收，不要求一次性把 60 turn 全绿。

本阶段更合理的验收标准是：

1. 长期记忆召回类问题得到明确根因和回归覆盖
2. PDF 降级输出从“统一空泛退化”变成“有类型区分的可用解释”
3. 结构化数据类失败 turn 明显减少“协议形态不一致”的问题
4. 工具失败类回答不再只有“调用工具失败”这种弱兜底
5. follow-up 主链问题被压缩到更纯粹的边界层，而不再混杂外围噪音

也就是说，本阶段要达到的状态是：

> **后续当我们进入 follow-up 研究时，面对的是一组更干净、更聚焦的问题。**

---

## 8. 最终判断

当前最稳的推进方式不是马上冲进 follow-up 主链，而是：

1. 先把非 follow-up 问题拆清楚
2. 逐个给出证据和根因
3. 把可独立修的外围问题先修掉
4. 再把 follow-up 作为一个更高纯度、更高耦合的专项来处理

如果这一步做对，后面 follow-up 的研究会更像“主问题精修”，而不是“在噪音里盲修”。

# 测试系统使用说明书

日期：2026-04-26

## 1. 测试系统是什么

测试系统是工作台里的前端测试控制台。它的目标不是单纯展示报告，而是让你可以从前端直接发起测试、查看运行状态、读取测试产物，并把失败问题定位到系统框架图里的具体节点和链路。

当前测试系统已经接入：

- 冒烟测试
- 稳定门禁
- 长场景核心
- 长场景批量
- 六十轮长跑
- 最近测试运行列表
- 测试日志尾部
- 失败问题列表
- 失败问题到系统框架图的高亮跳转
- prompt 装配透明分析
- turn 级记忆链路跳转

后续会继续接入：

- trace 到系统图的动态映射
- 节点/连线级 debug 面板

## 2. 入口位置

在前端右侧工作台中，模块顺序是：

1. 会话
2. 系统框架
3. 编排系统
4. 记忆系统
5. 测试系统
6. 操作系统
7. agent 系统
8. 灵魂系统

点击“测试系统”即可进入测试控制台。

## 3. 当前支持的测试类型

### 冒烟测试

用途：快速确认基础链路是否可用。

对应命令：

```bash
python -m harness.run --profile smoke
```

主要检查：

- 后端聊天接口是否可用
- SSE 流式事件是否正常
- 前端事件 reducer 是否能正确处理流式消息

适合场景：

- 小改动后快速确认项目还活着
- 前端/后端接口联调后快速验证
- 正式跑长测试前先做低成本检查

### 稳定门禁

用途：确认主执行链和核心回归测试没有明显断裂。

对应命令：

```bash
python -m harness.run --profile stable
```

主要检查：

- 冒烟测试
- core regression gate
- 前端生产构建
- 查询执行核心
- 任务路由
- 基础记忆与工具合同

适合场景：

- 修完核心逻辑后
- 调整运行时、query runtime、memory、retrieval 后
- 准备进入长场景测试前

### 长场景核心

用途：运行三条核心长场景，检查长链上下文、follow-up、记忆、检索、工具和状态漂移。

对应命令：

```bash
python -m harness.run --profile long --scenario-set core
```

主要检查：

- 真实多轮会话链路
- follow-up 解析
- session memory
- durable memory
- RAG / PDF / structured data
- tool route
- topic switch
- context compaction
- 长链状态漂移

适合场景：

- 大改动后做系统级验证
- 修复 follow-up、记忆、状态链条问题后
- 需要看真实长场景残留问题时

### 长场景批量

用途：在核心长场景基础上追加复合任务、权限边界和多会话隔离，适合较大改动后的深度验证。

对应命令：

```bash
python -m harness.run --profile long --scenario-set batches
```

适合场景：

- 改动影响多个子系统时
- 修复一批长链问题后
- 需要比核心长场景更高覆盖，但暂时不想跑六十轮时

### 六十轮长跑

用途：运行 60 turn 真实用户马拉松，专门压测状态漂移、follow-up、记忆召回、工具续写和恢复能力。

对应命令：

```bash
python -m harness.run --profile long --scenario-set mega
```

适合场景：

- 版本冻结前
- 长场景核心和批量都稳定后
- 需要复现真实工作日级别的系统漂移问题时

注意：所有长场景档位都会调用真实后端执行链。`长场景核心 / 长场景批量 / 六十轮长跑` 都会做二次确认。

## 4. 如何运行测试

### 运行冒烟测试或稳定门禁

1. 进入“测试系统”。
2. 在测试系统图中点击“冒烟测试”或“稳定门禁”节点。
3. 点击右上角“运行所选测试”。
4. 等待运行状态从“运行中”变成“通过”或“失败”。
5. 查看日志尾部、最近运行列表和失败问题。

### 运行长场景测试

1. 进入“测试系统”。
2. 点击“长场景核心”“长场景批量”或“六十轮长跑”节点。
3. 确认弹窗。
4. 等待测试完成。
5. 如果失败，查看“失败问题”列表，并点击问题定位到系统框架图。

右下角的“直接运行六十轮”会跳过节点选择，直接运行最高强度的六十轮长跑。

## 5. 测试运行状态说明

测试运行可能出现以下状态：

- `running`：测试正在执行。
- `passed`：测试执行完成，所有检查通过。
- `failed`：测试执行完成，但存在失败项。
- `cancelled`：测试被取消。
- `unknown`：前端能看到测试目录，但缺少完整运行状态。

如果测试长时间停留在 `running`，可以先刷新测试系统。后续会补更完整的进程健康检查和超时提示。

## 6. 测试产物位置

每次测试都会写入：

```text
output/test_runs/<run_id>/
```

常见文件：

- `run_state.json`
  测试系统自己的运行状态记录。

- `runner.log`
  测试进程输出日志。

- `run_result.json`
  结构化测试结果，包含 profile、运行环境、结果列表和统计信息。

- `issues.json`
  失败问题列表。

- `trace.jsonl`
  测试级 trace 摘要。

- `report.md`
  人类可读的测试报告。

长场景还会在 `artifacts/` 目录下写入每一轮对话的 turn 文件。

## 7. 如何从失败问题定位系统链路

如果测试失败，测试系统会显示“失败问题”区域。

每个失败问题都可以点击。点击后会发生两件事：

1. 自动切换到“系统框架”页面。
2. 高亮该问题对应的系统节点和连线。

例如：

- 长场景或 follow-up 问题通常会高亮：
  `测试与观测`、`对话执行核心`、`任务理解与规划`、`记忆门面`、`会话存储`

- 记忆问题通常会高亮：
  `记忆门面`、`项目持久层`、`检索服务`

- 检索问题通常会高亮：
  `证据编排`、`检索服务`、`项目持久层`

- 工具问题通常会高亮：
  `对话执行核心`、`工具与技能运行`

- 模型或 reasoning 续写问题通常会高亮：
  `模型流式输出`、`对话执行核心`

系统框架图右上角会出现“测试定位”说明，解释这次高亮的原因。点击“清除定位”可以取消高亮。

## 8. 运行链路图的后续使用方式

后续测试系统会把每轮对话转换成运行链路图。

目标效果：

```text
接口路由层
-> 运行时装配中心
-> 对话执行核心
-> 提示词组装
-> 记忆门面
-> 模型流式输出
-> 会话存储
```

如果调用工具，会展开：

```text
对话执行核心
-> 工具与技能运行
-> 工具结果
-> 模型续写
```

如果走 RAG、PDF 或结构化数据，会展开：

```text
对话执行核心
-> 证据编排
-> 检索服务
-> 项目持久层
-> 模型流式输出
```

未来你可以在测试报告里选择某个 turn，然后系统框架图会直接显示这一轮对话实际经过了哪些节点、哪条边耗时最长、哪里失败。

## 9. Prompt 装配透明分析

测试系统现在已经接入“Prompt 装配”节点。

这个节点会显示本轮系统 prompt 是怎么组成的，而不是只展示最终回答。

计划拆成三层：

### 静态层

包含：

- 当前灵魂 seed
- CORE 固有准则
- 技能摘要
- RAG grounding guard
- 隐藏实现细节约束

### 会话层

包含：

- 当前工作指引
- active skill prompt
- session memory
- active process context
- hot truth window

### 当前轮层

包含：

- relevant durable memory
- exact durable context
- retrieval evidence
- 当前轮相关记忆

默认展示结构、来源、长度和是否进入模型。调试模式下可以展开具体内容。

在前端使用时，有两个入口：

- 在“测试系统”的轮次链路回放里，每张 turn 卡片会显示 `Prompt 已记录` 或 `Prompt 缺失`。
- 点击某一轮进入“系统框架图”后，提示词组装节点会显示 `Prompt 已记录` 徽标。右侧详情面板可以展开三层装配摘要。

这样可以分析：

- 当前回答是否受 seed 风格影响
- CORE 是否压过了 seed
- 记忆是否召回错了
- 检索证据是否进入了 prompt
- skill prompt 是否改变了执行策略

## 10. 安全边界

测试系统有几个固定边界：

- 前端不能传任意 shell 命令。
- 前端只能运行白名单 profile。
- 当前白名单是 `smoke / stable / long`。
- 长场景测试需要二次确认。
- 测试产物只写入 `output/test_runs/<run_id>`。
- 系统图高亮是派生视图，不会修改原始 `issues.json`。

这些边界是为了避免误操作，也方便后续把测试系统作为长期工具保留下来。

## 11. 推荐使用流程

日常小改：

```text
运行冒烟测试
-> 如果失败，点击 issue 定位系统图
-> 修复后再跑冒烟测试
```

核心链路改动：

```text
运行稳定门禁
-> 查看失败问题
-> 跳转系统框架定位
-> 修复后再跑稳定门禁
```

长场景问题修复：

```text
先跑冒烟测试
-> 再跑稳定门禁
-> 最后跑长场景测试
-> 从失败 turn 进入运行链路图
-> 检查 prompt 装配、记忆召回、工具续写和状态链条
```

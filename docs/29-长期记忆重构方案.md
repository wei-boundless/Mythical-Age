# 长期记忆重构方案

> 目标：严格对照 `docs/23-Memory系统.md`、`docs/06-上下文管理.md`、`docs/07-Prompt-Cache.md`、`docs/12-Agent-系统.md`、`docs/14-任务系统.md`、`docs/25-架构模式总结.md`、`docs/26-OpenClaw-架构改造计划.md`、`docs/28-上下文隔离逐文件改造清单.md`，把当前 OpenClaw 的长期记忆系统从“已分层但边界仍松散”的状态，重构为“静态层稳定、动态层可治理、注入层按需召回、写入层策略收口、会话层与任务层严格隔离”的 Claude Code 风格文件型记忆系统。

---

## 1. 这份文档的定位

这不是一份泛化建议，也不是只讲理念的技术随笔，而是一份从概念到可实施的正式蓝图。

它同时承担三类职责：

1. 定义长期记忆系统的目标边界  
说明什么属于长期记忆，什么不属于长期记忆。

2. 给出 Claude Code 风格方案在本项目中的落地选型  
说明借鉴哪些原则，不借鉴哪些形式。

3. 展开为逐文件实施清单  
说明每个文件新增什么、替换什么、兼容什么、每一阶段如何验证。

因此，后续所有长期记忆相关改动，都必须同时满足两类约束：

- 对照本方案检查“记忆边界是否正确”
- 对照实施清单检查“代码修改是否落到正确模块和正确阶段”

---

## 2. 问题判断：当前长期记忆系统哪里不够 Claude Code

当前项目并不是“没有长期记忆”，而是已经有以下骨架：

- `soul/` 承载静态长期上下文
- `durable_memory/notes/*.md` 承载动态长期记忆
- `durable_memory/index/MEMORY.md` 承载索引视图
- `session-memory/` 承载当前会话 working memory
- `memory/facade.py` 承载统一入口

方向是对的，但还存在 6 个关键问题：

1. durable write 仍然偏触发词驱动  
`记住`、`长期记忆`、`偏好`、`项目` 等 marker 仍然是强判据，导致正常工作句子容易误入 durable memory。

2. durable extract 仍然残留 message-centric 逻辑  
提取仍会从用户原句直接抽 durable fact，容易把临时任务句子、测试句子、工具回显带入长期存储。

3. durable read 仍然偏“大块 memory block 注入”  
当前系统虽然有 exact/relevant 概念，但仍然容易把 durable memory 当一块统一 prompt 文本，而不是轻索引 + 按需 topic file。

4. session memory 与 durable memory 的边界还不够硬  
一些原本应由 session/task-local 层解决的问题，会通过 durable route 被错误接管。

5. durable note 生命周期治理不足  
旧 note 缺少 supersede、deprecate、review、confirm 等治理字段和流程，导致历史坏数据长期参与排序。

6. 持久化内容仍不够“只存稳定事实”  
当前写入层已经在收紧，但还没有形成 Claude Code 那种系统性 “What NOT to save” 纪律。

---

## 3. 设计目标

本轮重构的最终目标不是“让更多东西能被记住”，而是让长期记忆系统满足下面 8 条约束：

1. 长期记忆只保存跨会话稳定且值得保存的信息。
2. 当前任务状态、当前工具结果、当前工作流绑定不得进入 durable memory。
3. session memory 只负责恢复当前工作，不负责充当 durable memory 的预备垃圾桶。
4. `MEMORY.md` 必须是轻量索引，不是正文仓库。
5. 详细 durable note 只按需召回，不默认全量注入。
6. durable write 必须经过策略判断，不再由显性触发词直接主导。
7. durable read 必须排在 task-local、session-local 恢复之后，而不是抢主路由。
8. durable memory 的每次写入、命中、降级、失效都必须可观察、可测试、可审计。

## 4. Claude Code 方案里要借鉴什么

### 4.1 要采用的原则

#### 原则 1：记忆分层，而不是单一 memory bucket

借鉴来源：

- `docs/23-Memory系统.md`
- Claude Code 的 `CLAUDE.md / Auto Memory / Session Memory / Relevant Memories`

本项目落地：

- `Static Memory`：`soul/`
- `Durable Index`：`durable_memory/index/MEMORY.md`
- `Durable Topic Notes`：`durable_memory/notes/*.md`
- `Relevant Memory Selection`：按 query 召回少量 note
- `Session Working Memory`：`session-memory/`

#### 原则 2：轻量索引常驻，详细 note 按需读取

借鉴来源：

- Claude Code 的 `MEMORY.md` 截断入口 + topic files

本项目落地：

- `MEMORY.md` 只保留 note hook、类型、状态、摘要
- 详细 durable note 只有命中时才进入 prompt

#### 原则 3：长期记忆写入必须有严格的 “What NOT to save”

借鉴来源：

- `docs/23-Memory系统.md` 中对 memory types 和不应保存内容的总结

本项目落地：

- 明确禁止保存：
  - 当前任务步骤
  - 当前工具结果
  - 当前检索片段
  - 当前情绪状态
  - 测试台词
  - 可从 repo、文件、知识源直接推导的内容

#### 原则 4：Relevant Memories 是读取策略，不是存储策略

借鉴来源：

- Claude Code 的 Relevant Memories prefetch

本项目落地：

- durable store 不等于默认注入
- 注入必须走 relevant selector

#### 原则 5：session memory 是 working memory，不是 transcript archive

借鉴来源：

- `docs/06-上下文管理.md`
- `docs/28-上下文隔离逐文件改造清单.md`

本项目落地：

- session 只保留当前目标、活跃绑定、最近纠正、task summaries、warm snapshots
- durable 抽取改成从这些投影层中选候选，而不是从原始 transcript 生吞

### 4.2 明确不采用的形式

#### 不采用整套新框架替换

不迁移到 LangGraph / AutoGen / Letta / MemGPT 作为主 runtime。

原因：

- 你现在最缺的是边界纪律，不是框架能力。
- 当前项目已经有文件型 durable store、session working state、统一 facade，不应推倒重来。

#### 不采用“显性触发词驱动 durable write”

原因：

- 它与 Claude Code 的 memory type / worth-saving 判断相违背。
- 它会把当前任务话语、测试语句、否定句、情绪句错误晋升为 durable fact。

#### 不采用“默认全量 durable 注入”

原因：

- 这会让 durable memory 重新变成 prompt 污染源。
- 这与 Claude Code 的轻索引 + 按需 topic file 思想相反。

## 5. 目标架构

本轮长期记忆重构的正式结构分为 5 层：

1. `Static Memory Layer`
2. `Durable Manifest Layer`
3. `Durable Topic Notes Layer`
4. `Relevant Memory Selection Layer`
5. `Session-to-Durable Projection Layer`

外加一层不进模型的：

- `Memory Governance / Trace Layer`

### 5.1 Static Memory Layer

内容：

- constitution
- user profile
- agent guide
- project profile

职责：

- 承载高稳定、人工维护、长期有效的静态原则

禁止项：

- 不承载自动抽取结果
- 不承载当前任务状态

### 5.2 Durable Manifest Layer

内容：

- `durable_memory/index/MEMORY.md`

职责：

- 作为 durable store 的轻量入口
- 让系统知道“有哪些 durable note 值得按需读取”

禁止项：

- 不承载完整正文
- 不承载长段历史

### 5.3 Durable Topic Notes Layer

内容：

- `durable_memory/notes/*.md`

职责：

- 作为 durable memory 的真相源
- 每个 note 表达一条治理后的稳定事实

禁止项：

- 不直接保存原始消息
- 不保存任务局部绑定

### 5.4 Relevant Memory Selection Layer

内容：

- exact hits
- relevant hits
- manifest fallback

职责：

- 每轮只选极少量 durable notes 注入模型

禁止项：

- 不默认注入全库
- 不让 durable selection 抢占 task-local / session-local 恢复

### 5.5 Session-to-Durable Projection Layer

内容：

- main context
- task summaries
- explicit corrections
- stable user statements

职责：

- 生成 durable candidate，而不是直接保存 transcript

禁止项：

- 不把 session raw transcript 直接晋升 durable

### 5.6 Memory Governance / Trace Layer

内容：

- why stored
- why rejected
- why superseded
- why not injected
- exact/relevant hit source

职责：

- 支持调试、回归、脏数据清理、长期演化

禁止项：

- 不进入模型 prompt

## 6. 长期记忆数据模型

建议把当前 durable schema 升级为 `durable-memory.v3`。

### 6.1 Note 顶层字段

- `schema_version`
- `slug`
- `title`
- `summary`
- `canonical_statement`
- `body`
- `memory_type`
- `memory_class`
- `scope`
- `stability`
- `retrieval_hints`
- `tags`
- `created_at`
- `updated_at`
- `created_by`
- `source_kind`
- `source_session_id`
- `source_role`
- `source_message_excerpt`
- `confidence`
- `status`
- `eligible_for_injection`
- `last_confirmed_at`
- `review_after`
- `supersedes`
- `invalidation_reason`

### 6.2 字段语义

- `memory_type`
  - `user | feedback | project | reference`

- `memory_class`
  - `preference | work`

- `scope`
  - `user | project | agent | shared`

- `stability`
  - `stable | provisional`

- `status`
  - `active | inactive | archived | deprecated`

- `eligible_for_injection`
  - `true | false`
  - 表示运行时是否允许继续参与 relevant selection

- `source_kind`
  - `preference`
  - `correction`
  - `policy`
  - `reference`
  - `derived_summary`

### 6.3 为什么要补这些字段

因为 durable memory 不是“存下来就完了”，而是需要支持：

- 旧事实失效
- note 合并
- note 替代
- 暂时不注入
- 到期复核

## 7. 新的写入模型

### 7.1 写入原则

durable write 不再由“用户说了记住”触发，而由下面这条逻辑触发：

> 这是经过治理判断后，确认值得跨会话保存的稳定事实。

### 7.2 新的写入流水线

#### Step 1：Candidate Generation

输入来源：

- `main_context`
- `task_summaries`
- `explicit corrections`
- `stable user preference statements`
- 少量需要保留的规范化 evidence summary

不再直接以最近 N 条 messages 作为主输入。

#### Step 2：Policy Filter

候选进入 durable 之前必须先过 reject/session-only gate。

必须拒绝的内容：

- 当前任务步骤
- 当前文件绑定
- 当前页码绑定
- 当前天气/当前时间等即时状态
- 情绪宣泄和依恋表达
- 测试用示例语句
- 工具 raw output
- 检索原文
- 可以从 repo、知识文件、固定 profile 直接推导的内容

#### Step 3：Normalization

把候选统一整理成：

- `title`
- `summary`
- `canonical_statement`
- `why_stored`
- `retrieval_hints`
- `memory_type`
- `memory_class`
- `scope`
- `stability`

#### Step 4：Merge / Confirm / Supersede

写入前先在 durable store 中查重：

- 如果等价：刷新 `last_confirmed_at`
- 如果近似但更完整：merge
- 如果新事实替代旧事实：写 `supersedes`
- 如果旧事实失效：标记 `deprecated`

#### Step 5：Manifest Rebuild

重建 `durable_memory/index/MEMORY.md`：

- 只保留轻量 hook
- 严格控制总长度
- 不让 manifest 变成第二份全文

### 7.3 明确的 “What NOT to save”

这一节必须写成代码级规则，而不是只停留在文档里。

不得保存：

- raw transcript
- raw tool output
- raw retrieval chunk
- 单轮任务目标
- 单轮任务恢复提示
- 当前上下文槽位
- 当前 session summary
- “用户今天心情如何”之类瞬时状态
- “回到这个文件继续看”之类 task-local 绑定
- “当前天气”“当前库存结果”之类即时查询答案

## 8. 新的读取模型

### 8.1 读取顺序

读取 durable memory 必须排在以下层之后：

1. task-local context
2. main working context
3. session memory restore hints
4. 当前 retrieval / tool evidence

只有这些层不能回答时，才进入 durable memory。

### 8.2 durable selection 的三段式策略

#### 第一段：Exact Match

适用：

- 明确在问既有偏好、项目约定、长期规则、外部引用

输出：

- 最多 1-3 条 exact durable notes

#### 第二段：Relevant Match

适用：

- 问法不是精确匹配，但 durable facts 对当前问题有帮助

输出：

- 最多 3-5 条 relevant notes

#### 第三段：Manifest Fallback

适用：

- 没有 exact/relevant note，但需要给模型“durable store 里大概有什么”这一层弱提示

输出：

- 轻量 manifest 摘要

### 8.3 注入模型时的可见内容

主模型不应看到整份 note 原文，只应看到 note projection：

- `title`
- `kind`
- `canonical`
- `summary`
- `recall hints`
- 极短的 `details excerpt`

### 8.4 为什么这样读

因为 Claude Code 风格的关键不是“有记忆库”，而是：

- 模型只看必要的记忆
- 轻索引常驻
- 详细内容按需读
- 记忆不是 prompt 污染源

## 9. 与上下文隔离蓝图的衔接

本方案必须与 `docs/28-上下文隔离逐文件改造清单.md` 形成一套一致系统。

### 9.1 边界对齐

- `Static Prompt Layer` 对应 `soul`
- `Session-Stable Context Layer` 不直接承载 durable topic 正文
- `Main Working Context Layer` 不直接引用 durable raw note
- `Task Local Context Layer` 不能晋升为 durable memory
- `Evidence Layer` 不能直接进 durable write
- `Session Memory Layer` 只做 projection 来源
- `Durable Memory Layer` 只存 stable facts
- `Debug / Trace Layer` 只做观测

### 9.2 关键纪律

长期记忆系统绝不能反过来接管以下职责：

- 当前任务恢复
- 当前文件恢复
- 当前 evidence 真相层
- 编号 follow-up 解析

这些职责都属于 task/session/main context，不属于 durable memory。

## 13. 旧数据迁移方案

### 13.1 迁移原则

- 不直接删除旧 note
- 先迁 schema，再做治理
- 旧 note 默认保守，不默认继续注入

### 13.2 迁移步骤

1. 读取旧 note frontmatter
2. 补齐缺失字段
3. 根据 hygiene/policy 评估：
   - active
   - inactive
   - deprecated
4. 重建 manifest
5. 重建 retrieval index

### 13.3 风险控制

- 对历史 note 的自动治理必须保守
- 无法确定的 note 宁可降级为 `inactive`
- 不做不可逆 destructive rewrite

## 15. 明确的技术决策

本轮长期记忆重构固定采用下面这组技术决策：

- 存储形态：本地 markdown note + manifest
- 真相源：`durable_memory/notes/*.md`
- 轻量入口：`durable_memory/index/MEMORY.md`
- 静态层：`soul/`
- working memory：`session-memory/`
- 写入策略：post-turn, summary-first, policy-gated
- 读取策略：exact -> relevant -> manifest fallback
- 注入策略：selected note projection，不注入 raw note body
- 生命周期：confirm / merge / supersede / deprecate
- 观测方式：trace + regression + long scenario semantic assertions

本轮明确不采用：

- 触发词主导 durable write
- transcript 直抽 durable fact
- durable store 全量注入
- 用 durable memory 替代 task/session 恢复
- 引入外部 memory framework 重写底座

## 16. 最终验收标准

当下面这些条件同时满足时，本轮长期记忆重构才算完成：

1. durable write 不再靠显性 trigger words 主导
2. transient / task-local / runtime noise 不再进入 durable store
3. `MEMORY.md` 成为轻量 manifest，而不是第二份正文
4. 主 prompt 中 durable 注入显著减少且更稳定
5. session/task 恢复不再被 durable route 抢走
6. 历史坏 note 能被治理和降权
7. 长场景里 durable memory 不再稳定制造漂移

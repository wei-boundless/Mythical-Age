<h2>Writing better release notes</h2>
<p class="mobile-date">31st January 2022</p>



<p>Release notes are an important part of the open source process. I’ve been thinking about these a lot recently, and I’ve assembled some thoughts on how to do a better job with them.</p>
<ol>
<li>
<p><strong>Write release notes</strong>. Seriously—if you want people to take advantage of the work you have been doing to improve your projects, you need to tell them about it!</p>
</li>
<li>
<p><strong>Include the date</strong>. The date matters a lot, because I want to be able to determine how old a release is—especially important for the dependencies I am using.</p>
<p>It’s much more reasonable to assume people will be running as a minimum a version from 3 years ago than one that came out last week.</p>
</li>
<li>
<p><strong>Make sure people can link to the release notes</strong> for a version. This can be a page-per-release or an anchor target on the releases page, but it needs to be possible and it needs to be discoverable.</p>
<p>There are some projects for which I have to view source on the HTML page to find the anchor links for the version headers—don’t make me do that!</p>
</li>
<li>
<p>For larger releases, <strong>break them up with headers</strong>. “New features” v.s. “Bug fixes” is a useful distinction.</p>
</li>
<li>
<p><strong>Emphasize the highlights</strong>. It can be easy for the highlights of a larger release to get lost in a sea of bullet points. I’ll sometimes include an introductory paragraph highlighting the major themes.</p>
</li>
<li>
<p><strong>Link to relevant documentation</strong> from the release notes. If I want to know more about a new feature that should be the best place for me to start.</p>
</li>
<li>
<p>Flesh them out with <strong>examples and screenshots</strong>. Most release notes are pretty dry—there’s no space limit on these things, so feel free to use all of the tools at your disposal to best answer the question “what has changed and why does this matter to me?”</p>
</li>
<li>
<p><strong>Link to the associated issue thread</strong>. If I want to know more about a feature, and you’ve been using issues effectively, the issue link can tell me the entire story of the new feature in more detail than anything else.</p>
</li>
<li>
<p><strong>Credit your contributors!</strong> If someone helped build a feature the release notes are a great place to give them a shout out.</p>
